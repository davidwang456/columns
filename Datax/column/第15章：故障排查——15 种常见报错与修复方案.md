# 第15章：故障排查——15 种常见报错与修复方案

## 1. 项目背景

某金融公司数据平台维护了 200+ 个 DataX 同步任务，每天凌晨批量执行。运维团队每周平均接到 3-5 个任务失败告警，原因五花八门：连接超时、OOM、脏数据超限、JSON 格式错误、ClassLoader 冲突……运维小陈的微信收藏里存了 30 多条"DataX 报错排查记录"，每次出问题都要翻聊天记录找相似的 Case。

TL 让他把排障经验系统化整理成一份 SOP。小陈花了两周时间分析了 200+ 个工单，按错误类型归纳出 15 种高频故障，并为每种故障编写了"现象-根因-验证-修复"四步排查法。本章就是这份排障手册的精华——掌握这 15 种故障模式的排查方法，能覆盖生产环境中 95% 的 DataX 任务故障。

## 2. 项目设计——剧本式交锋对话

**（运维监控室，凌晨 3:15，告警声此起彼伏）**

**小陈**：（盯着满屏红色告警）又来了……任务 ID 1024 失败：`OutOfMemoryError: GC overhead limit`。这已经是本周第三次 OOM 了。

**小胖**：（端着一杯咖啡走过来）你这每次都是重启重跑，怎么不找根因啊？

**小陈**：怎么找？日志里就一句 OOM，连哪个 Task 内存爆了都不知道。

**大师**：（半夜被 call 醒，远程连进服务器）来，跟着我的思路走。DataX 的 OOM 有 6 种根因，你遇到的是第 3 种——GC overhead limit。看日志里有没有 `GC overhead limit exceeded` 这个字眼。

**技术映射**：GC overhead limit = 垃圾回收员太累了。JVM 花了 98% 的时间收垃圾，却只回收了不到 2% 的堆空间，JVM 崩溃了。

**小陈**：（找到了关键日志）有！就在 Task 统计摘要前面一行：
```
Exception in thread "taskGroup-0" java.lang.OutOfMemoryError: GC overhead limit exceeded
```

**大师**：好，现在查 3 件事：
1. JVM 启动参数是多少？（查 datax.py 或 Engine 日志开头的 -Xms、-Xmx）
2. Channel 数是多少？（查 speed.channel）
3. 单条 Record 的大小？（查平均流量 ÷ 记录写入速度）

**小陈**：（快速计算）Xmx=2G，channel=20，单条 Record 约 5KB。

**大师**：（秒答）根因确认了。20 个 Channel × 128(Capacity) × 2(Reader+Writer buffer) × 5KB = 25.6MB Channel 内存 + 20 × 2048(batchSize) × 5KB = 205MB Writer batch = 总计约 500MB 理论数据内存。但加上 JVM 对象头和 GC 开销，2G 的 heap 可能 80% 被占。而且你 20 个 GC 线程竞争，GC overhead 就超了。

解决方案：Xmx 调到 4G，channel 降到 10。

**小胖**：（佩服）我靠，这排查逻辑太清晰了。怎么把这种经验教给新人？

**大师**：（打开一张思维导图）我整理了一张 DataX 故障拓扑图——15 种高频故障，每种按"现象→根因→验证→修复"四步走。新人照着排查，新人也能独立解决问题。

## 3. 项目实战——15 种常见报错修复

### 故障 1：OOM——GC overhead limit exceeded

**现象**：
```
Exception in thread "taskGroup-0" java.lang.OutOfMemoryError: GC overhead limit exceeded
```

**根因**：堆内存不足以同时容纳 20 个 Channel 的缓冲 + batch buffer。

**验证**：
```bash
# 查 Xmx
grep "Xmx" datax.py
# 查 Channel 数
grep "channel" job.json
```

**修复**：
1. Xmx 调大到原来的 2~3 倍
2. Channel 数减半
3. 降低 batchSize（从 2048 → 1024）
4. 排除大字段列（TEXT/BLOB）

**紧急措施**：运行时指定 Xmx：
```bash
python bin/datax.py job.json -p "-Ddatax.jvm.xms=1g -Ddatax.jvm.xmx=4g"
```

---

### 故障 2：OOM——Direct buffer memory

**现象**：
```
java.lang.OutOfMemoryError: Direct buffer memory
```

**根因**：DirectBuffer（堆外内存）不足。常见于 ORC/Parquet 文件读写（大量 NIO 操作使用 DirectBuffer）。

**验证**：
```bash
# 检查系统内存
free -h
# 确认 DirectBuffer 使用量（借助 Arthas）
java -jar arthas-boot.jar
> dashboard
# 查看 direct 列
```

**修复**：增加 `-XX:MaxDirectMemorySize`：
```bash
java -XX:MaxDirectMemorySize=512m -Xmx4g -jar datax.jar ...
```

或减少 ORC 文件读取的并发度。

---

### 故障 3：Communications link failure

**现象**：
```
Caused by: com.mysql.cj.jdbc.exceptions.CommunicationsException: 
Communications link failure. The last packet sent successfully to the server was 0 milliseconds ago.
```

**根因**：有三种可能：
1. MySQL 的 `wait_timeout` 到期（默认 8 小时，但防火墙可能在 1 小时就断开）
2. 防火墙/安全组规则拦截了 3306 端口
3. JDBC URL 中 IP 或端口拼写错误

**验证**：
```bash
# 1. 测试 MySQL 连接
mysql -h 10.0.1.100 -P 3306 -u root -p -e "SELECT 1"

# 2. 查看 MySQL wait_timeout
mysql -e "SHOW VARIABLES LIKE 'wait_timeout';"

# 3. 检查防火墙（以 iptables 为例）
iptables -L -n | grep 3306
```

**修复**：
- JDBC URL 中增加 `autoReconnect=true` 和长超时
- 在数据同步任务中设置 session 参数：`SET SESSION wait_timeout = 86400`
- 如果防火墙主动断开，在 JDBC URL 中加上 `socketTimeout=0`

---

### 故障 4：Access denied for user

**现象**：
```
java.sql.SQLException: Access denied for user 'datax_reader'@'10.0.1.50' (using password: YES)
```

**根因**：MySQL 用户权限问题——`datax_reader` 没有从 IP `10.0.1.50` 连接的权限。

**验证**：
```sql
SELECT user, host FROM mysql.user WHERE user = 'datax_reader';
-- 可能只授权了 'datax_reader'@'localhost'，没授权 '10.0.1.50'
```

**修复**：
```sql
GRANT SELECT ON trade_db.* TO 'datax_reader'@'10.0.1.%' IDENTIFIED BY 'password';
FLUSH PRIVILEGES;
```

---

### 故障 5：Unknown database / Table doesn't exist

**现象**：
```
java.sql.SQLSyntaxErrorException: Unknown database 'trade_db'
或: Table 'trade_db.orders' doesn't exist
```

**根因**：JDBC URL 中的 database 名拼写错误，或表名拼写错误，或表确实不存在。

**验证**：
```bash
mysql -h host -u user -p -e "SHOW DATABASES;" | grep trade_db
mysql -h host -u user -p -e "SHOW TABLES FROM trade_db;" | grep orders
```

**修复**：修正 JSON 配置中的 database 名或 table 名。

---

### 故障 6：JSON 格式错误

**现象**：
```
com.alibaba.fastjson.JSONException: syntax error, expect {, actual }
或: com.alibaba.fastjson.JSONException: not close json text, token : ,
```

**根因**：JSON 语法错误——多余的逗号、缺少引号、括号不匹配。

**验证**：
```bash
# 使用 jq 校验 JSON
cat job.json | jq . > /dev/null
# 如果有错误会提示行号
```

**修复工具**：VS Code 安装 Prettier 插件，`Shift+Alt+F` 自动格式化并高亮语法错误。

---

### 故障 7：Required parameter missing

**现象**：
```
DataXException: 您提供配置文件有误，[job.content[0].reader.parameter.column]是必填参数，但不允许为空。
```

**根因**：JSON 配置缺少必填参数。

**验证**：查看官方文档中该 Reader/Writer 的必填参数列表，对照 JSON 配置检查。

**修复**：补全缺失参数。

---

### 故障 8：ClassNotFoundException——插件类找不到

**现象**：
```
java.lang.ClassNotFoundException: com.alibaba.datax.plugin.reader.mysqlreader.MysqlReader
```

**根因**：plugin.json 中声明的类名与实际 JAR 中的类不匹配，或 plugin 目录下缺少对应 JAR。

**验证**：
```bash
# 检查 plugin.json
cat plugin/reader/mysqlreader/plugin.json
# 检查 JAR 是否存在
ls plugin/reader/mysqlreader/mysqlreader-*.jar
# 检查 JAR 中是否有声明的类
jar tf mysqlreader-0.0.1.jar | grep MysqlReader
```

**修复**：重新编译插件，或修正 plugin.json 中的 `class` 字段。

---

### 故障 9：脏数据超限

**现象**：
```
ERROR JobContainer - 脏数据记录数[1500]超过限制[1000]
或: ERROR JobContainer - 脏数据百分比[0.0250]超过限制[0.0100]
```

**根因**：数据质量问题超出容忍阈值。常见于类型转换失败（VARCHAR 转 DECIMAL）、日期格式不匹配。

**验证**：查看脏数据日志中的前 10 条详情，统计失败字段的分布。

**修复**（按优先级）：
1. 修复源端数据（最根本）
2. 在 Reader 的 querySql 中做 SQL 级清洗：`CAST(NULLIF(amount, 'N/A') AS DECIMAL)`
3. 在 Transformer 中做 Groovy 脚本级清洗
4. 提高 errorLimit 阈值（临时方案）

---

### 故障 10：NumberFormatException——类型转换失败

**现象**：
```
java.lang.NumberFormatException: For input string: "N/A"
at sun.misc.FloatingDecimal.readJavaFormatString(FloatingDecimal.java:2043)
```

**根因**：源端字段值是字符串 `"N/A"`，MySQL Reader 尝试转为 `Long`/`Double` 时失败。

**验证**：检查源端数据分布：
```sql
SELECT amount, COUNT(*) FROM orders 
WHERE amount REGEXP '[^0-9.]' GROUP BY amount;
-- 找出所有非法的 amount 值
```

**修复**：
```sql
-- Reader querySql 中用 REGEXP 过滤
SELECT * FROM orders WHERE amount REGEXP '^[0-9]+(\\.[0-9]{1,2})?$' AND $CONDITIONS
```

---

### 故障 11：Data too long for column

**现象**：
```
java.sql.BatchUpdateException: Data too long for column 'remark' at row 1
```

**根因**：目标表 `remark VARCHAR(200)`，但源端某行的 remark 有 300 个字符。

**验证**：
```sql
SELECT MAX(LENGTH(remark)) FROM source_table;
-- 如果 > 200，就是这里的问题
```

**修复**：
1. 扩大目标表字段：`ALTER TABLE target MODIFY remark VARCHAR(500)`
2. 在 Transformer 中截断：`dx_substr [0, 200]`
3. 在 querySql 中截断：`SELECT LEFT(remark, 200) as remark FROM ...`

---

### 故障 12：Too many connections

**现象**：
```
Data source rejected establishment of connection, message from server: "Too many connections"
```

**根因**：DataX 的并发连接数超过 MySQL `max_connections`。

```
channel × 2(Reader+Writer) + Druid预留池 > max_connections
```

**验证**：
```sql
SHOW VARIABLES LIKE 'max_connections';  -- 假设 200
SHOW PROCESSLIST;  -- 当前连接数
```

**修复**：
1. 降低 channel 数
2. 调大 MySQL `max_connections`
3. 每个 JDBC URL 只配置一个实例（避免多 IP 轮询增加连接数）

---

### 故障 13：Connection reset / Connection timed out

**现象**：
```
java.net.SocketTimeoutException: Read timed out
或: java.net.SocketException: Connection reset
```

**根因**：网络中断或 socket 超时。长时间运行的任务在中间阶段出现网络抖动。

**修复**：JDBC URL 中增加超时参数：
```
jdbc:mysql://host:3306/db?connectTimeout=60000&socketTimeout=300000
```

---

### 故障 14：splitPk 分片失败——主键类型不支持

**现象**：
```
DataXException: splitPk [id] 的类型 [VARCHAR] 暂不支持切分
```

**根因**：`splitPk` 指定的列不是数值类型（如 VARCHAR UUID）。

**修复**（两种方案）：
1. channel=1，放弃分片
2. 用 querySql + 哈希取模手动切分：
```sql
SELECT * FROM orders WHERE MOD(CRC32(id), 10) = 0 AND $CONDITIONS  -- Task 0
SELECT * FROM orders WHERE MOD(CRC32(id), 10) = 1 AND $CONDITIONS  -- Task 1
```

---

### 故障 15：JAR 冲突——NoSuchMethodError / ClassCastException

**现象**：
```
java.lang.NoSuchMethodError: com.google.common.base.Preconditions.checkArgument(ZLjava/lang/String;)
或: java.lang.ClassCastException: com.alibaba.fastjson.JSONObject cannot be cast to ...
```

**根因**：不同插件的第三方依赖版本冲突（如 Guava 版本不一致）。ClassLoader 虽然隔离了插件，但如果两个插件通过核心框架间接传递对象，仍可能冲突。

**修复**：
1. 在 pom.xml 中统一依赖版本
2. 使用 `<exclusions>` 排除冲突的传递依赖
3. 检查 `plugin/reader/xxx/libs/` 目录下是否有重复的 JAR

## 4. 项目总结

### 4.1 故障排障 SOP（四步法）

```
第1步：看现象
  └─ 在日志中搜索 ERROR 关键字，截取完整的异常堆栈

第2步：定根因
  └─ 按本节的 15 种故障模式匹配，缩小排查范围

第3步：验证
  └─ 用 SQL/命令行/HDFS 命令复现确认

第4步：修复
  └─ 按优先级：改动最少 > 改动最快 > 改动最彻底
```

```json
// 一份"抗揍"的 Job 配置模板
{
    "setting": {

        },
```

JDBC URL 防御参数：
```
?useSSL=false
&autoReconnect=true
&connectTimeout=60000
&socketTimeout=300000
&serverTimezone=Asia/Shanghai
&tinyInt1isBit=false
```

### 4.4 注意事项

1. 任务失败后不要盲目重跑——先分析根因，避免同样错误反复出现
2. 脏数据日志默认只存 10 条详情——排查问题时可能信息不足，改为 DEBUG 级别日志
3. OOM 后不要立即调大 Xmx——先降 channel 和 batchSize，找到最小可用配置再做增量优化
4. 防火墙问题容易误判为 MySQL 配置问题——先 telnet 测试端口连通性
5. Jar 冲突通常在新插件上线时出现——建议灰度发布新插件
