# 第26章：数据倾斜处理与自适应切分策略

## 1. 项目背景

某支付平台每日凌晨执行用户交易记录的全量同步任务：`orders` 表 2 亿行，`channel=20`，`splitPk=order_id`（自增主键），配置看起来很标准。但运维团队发现一个怪现象——任务日志里 20 个 Task 的耗时严重不均：`taskId=0` 2 分钟跑完 50 万行，`taskId=17` 竟然跑了 47 分钟，处理了 4200 万行！

排查后发现，`order_id` 虽然是自增主键，但业务上存在大量"补单"操作——运营团队在活动期间集中补录历史订单，这些补单的 `order_id` 集中在 800000000~950000000 区段，占全表数据的 65%。而早期 ID 区段（1~300000000）虽然跨度大，但实际行数只占 15%。

默认的等距切分算法 `step = (MAX - MIN) / channel` 只看着"值范围"切分，完全不关心"数据量分布"。当数据倾斜时，木桶效应的后果就是：整个 Job 的耗时 = 最慢 Task 的耗时，其余 19 个 Task 虽然早已完成，但资源（数据库连接、JVM 线程）一直空转等待。更严重的是，倾斜 Task 长时间持有源表锁（如有），可能导致线上业务查询超时。

架构组要求数据组设计一套"自适应切分策略"：在 `split()` 阶段根据数据的实际分布动态调整切分边界，让每个 Task 处理的数据量尽可能接近，从而消除长尾。本章从木桶效应原理讲起，深入 `JobAssignUtil.assignFairly()` 与 `SingleTableSplitUtil` 源码，手把手实现一套基于采样的自适应分片算法。

## 2. 项目设计——剧本式交锋对话

**（数据组例会，投影仪上显示 Task 耗时柱状图——最高的一根柱子是其他柱子的 20 倍）**

**小胖**：（啃着蛋挞）这台服务器是不是在摸鱼？同样的 CPU、同样的内存，凭啥 task-17 要跑 47 分钟？

**小白**：（推了推眼镜）不是硬件的问题，是数据分布的问题。你看这张图——task-17 分配到的 `order_id` 范围是 [850000000, 950000000)，这个区段聚集了 4200 万行补单数据。而 task-0 分配到的 [1, 5000000) 区段只有 50 万行。等距切分的刀法是均匀的，但肉不均匀。

**小胖**：那不就是"别人碗里有肉、我碗里全是汤"吗？

**大师**：（在白板上画了一条不均匀分布曲线）准确地说，是你碗里全是肉、别人碗里只有汤。数据倾斜有三种典型场景：

**场景一：热点值聚集**。比如订单表 `status=1`（已支付）占 90%，`status=0/2/3` 合计占 10%。如果按 status 做分片键，一个 Task 拿到 90% 的数据。

**场景二：自增 ID 空洞**。早期的数据 ID 分布稀疏（测试数据被清理后留下空洞），近期 ID 分布密集。等距切分不了解这种"前松后紧"的分布。

**场景三：分区表数据不均**。分区表 `p202401` 每月 5000 万行，但 `p202312` 因为有活动冲到了 2 亿行。按分区分片——有的 Task 是别人 4 倍工作量。

**技术映射**：等距切分 = 按面积切披萨。如果披萨上的配料不均匀（一半堆满火腿、一半只有面饼），面积切得再均匀也没用。我们需要的是"按配料量切披萨"——即按数据量切分。

**小胖**：那 DataX 源码里就没有考虑这个问题吗？

**小白**：看过源码了。`SingleTableSplitUtil.genSplitSql()` 的核心就三行：

```java
long min = getMin(conn, table, splitPk);
long max = getMax(conn, table, splitPk);
long step = (max - min) / splitNum;
// 生成 WHERE splitPk >= ? AND splitPk < ?
```

它只查了 MIN 和 MAX，中间的分布完全不管。真正的问题是 `split()` 返回的 `List<Configuration>` 中，每个 Configuration 的 `where` 条件是根据步长硬算的，不是根据数据量算的。

**大师**：改进思路也很直接——用行数反推边界。具体来说：

**方案 A：ORDER BY + LIMIT 采样**

```sql
SELECT splitPk FROM table ORDER BY splitPk 
LIMIT 1 OFFSET 50000000;  -- 第 5000 万行的 splitPk 值
LIMIT 1 OFFSET 100000000; -- 第 1 亿行的 splitPk 值
```

用这些采样点的 splitPk 值作为分段边界，保证每个分段的数据量接近。

**方案 B：桶统计 + 动态调整**

```sql
SELECT FLOOR(splitPk / 10000000) as bucket, COUNT(*) as cnt
FROM table GROUP BY bucket ORDER BY bucket;
```

先按粗粒度桶统计数据量，再根据每个桶的数据量动态决定切分边界。

**方案 C：TaskGroup 负载不均的二次兜底——`assignFairly`**

即使分片已经相对均匀，在分配 Task 到 TaskGroup 时也可能出现不均。`JobAssignUtil.assignFairly()` 使用贪心算法——每次将下一个 Task 分配给当前总工作量最小的 TaskGroup，保证各 TaskGroup 的 Task 数尽量接近。

**小胖**：（举手）但我还有个问题——如果表没有 splitPk 怎么办？或者 splitPk 是 UUID 这种？

**大师**：好问题。没有数值型 splitPk 时，有两种替代方案：

1. **哈希取模分片**：`WHERE MOD(CRC32(uuid), channel) = ${taskIndex}`。但缺点是全表扫描，没有索引可用。
2. **字符前缀分片**：`WHERE uuid >= 'a0000000' AND uuid < 'd0000000'`。前提是 UUID 的前几位分布相对均匀（MD5/SHA 生成的 UUID 满足这个条件）。

不过这些方案的倾斜修复难度更大——哈希取模天然均匀但性能差，字符前缀需要先做分布分析。

**大师总结**：解决数据倾斜不需要玄学，只需要两步——第一步，在 `split()` 里用行数代替步长来决定切分边界；第二步，让 `assignFairly` 做最后的 TaskGroup 级兜底。下面我们就来动手实现。

```sql

    order_id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,

CREATE PROCEDURE gen_skewed_data()
BEGIN
    DECLARE i BIGINT DEFAULT 1;
    
    -- 前 50 万行均匀分布在 ID 1~400000000（稀疏区段）
    WHILE i <= 500000 DO
        INSERT INTO orders VALUES (
            FLOOR(1 + RAND() * 400000000),
            FLOOR(1000 + RAND() * 500000),
            ROUND(RAND() * 10000, 2),

            DATE_ADD('2023-01-01', INTERVAL FLOOR(RAND() * 1000) DAY)

        SET i = i + 1;

    SET i = 1;
    WHILE i <= 4500000 DO
        INSERT INTO orders VALUES (
            800000000 + i,
            FLOOR(1000 + RAND() * 500000),
            ROUND(RAND() * 10000, 2),
            1,
            DATE_ADD('2025-01-01', INTERVAL FLOOR(RAND() * 365) DAY)
        );
        SET i = i + 1;
    END WHILE;
END$$
DELIMITER ;

CALL gen_skewed_data();
```

### 3.3 步骤三：实现基于行数采样的自适应切分算法

**目标**：覆盖 `Reader.Job.split()`，通过 ORDER BY + 采样查询确定每个 Task 的切分边界，使每个 Task 的数据量接近。

**核心源码**（`AdaptiveSplitUtil.java`）：

```java
package com.alibaba.datax.plugin.reader.mysqlreader.util;

import java.sql.*;
import java.util.ArrayList;
import java.util.List;

public class AdaptiveSplitUtil {

    /**
     * 基于行数采样的自适应切分
     * 
     * @param conn        数据库连接
     * @param table       表名
     * @param splitPk     分片键（必须有索引！）
     * @param adviceNum   期望的 Task 数量（= channel 数）
     * @param totalRows   总行数
     * @param whereClause 额外的 WHERE 条件
     * @return 每个 Task 的 WHERE 边界列表 ["1,50000000", "50000000,120000000", ...]
     */
    public static List<String> adaptiveSplit(
            Connection conn, String table, String splitPk,
            int adviceNum, long totalRows, String whereClause) {
        
        long rowsPerTask = totalRows / adviceNum;
        if (rowsPerTask < 10000) {
            // 数据量太小，不用自适应，直接等距切分
            return equalDistanceSplit(conn, table, splitPk, adviceNum, whereClause);
        }
        
        long minPK = queryMin(conn, table, splitPk, whereClause);
        long maxPK = queryMax(conn, table, splitPk, whereClause);
        
        List<Long> boundaries = new ArrayList<>();
        boundaries.add(minPK);
        
        // 为每个 Task 找一个边界值（通过 COUNT 二分查找）
        for (int i = 1; i < adviceNum; i++) {
            long targetRow = i * rowsPerTask;
            long boundary = findBoundaryByBinarySearch(
                conn, table, splitPk, minPK, maxPK, targetRow, whereClause);
            boundaries.add(boundary);
        }
        boundaries.add(maxPK + 1);
        
        // 生成切分边界字符串
        List<String> splits = new ArrayList<>();
        for (int i = 0; i < adviceNum; i++) {
            splits.add(boundaries.get(i) + "," + boundaries.get(i + 1));
        }
        return splits;
    }
    
    /**
     * 二分查找第 targetRow 行的 splitPk 值
     * 
     * 原理：在 [min, max] 范围内二分，每次猜一个 mid 值，
     * 统计 splitPk < mid 的行数，根据行数与 targetRow 的关系缩小范围。
     */
    private static long findBoundaryByBinarySearch(
            Connection conn, String table, String splitPk,
            long lo, long hi, long targetRow, String whereClause) {
        
        while (lo < hi) {
            long mid = lo + (hi - lo) / 2;
            
            long countLeft = queryCountBefore(conn, table, splitPk, mid, whereClause);
            
            if (countLeft < targetRow) {
                lo = mid + 1;
            } else if (countLeft > targetRow) {
                hi = mid - 1;
            } else {
                return mid;
            }
        }
        return lo;
    }
    
    /**
     * 统计 splitPk < threshold 的行数
     * 优化点：利用 splitPk 索引走 range scan + COUNT
     */
    private static long queryCountBefore(
            Connection conn, String table, String splitPk,
            long threshold, String whereClause) {
        
        StringBuilder sql = new StringBuilder();
        sql.append("SELECT COUNT(*) FROM ").append(table)
           .append(" WHERE ").append(splitPk).append(" < ").append(threshold);
        
        if (whereClause != null && !whereClause.isEmpty()) {
            sql.append(" AND ").append(whereClause);
        }
        
        try (Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery(sql.toString())) {
            if (rs.next()) {
                return rs.getLong(1);
            }
        } catch (SQLException e) {
            throw new RuntimeException("Count query failed", e);
        }
        return 0;
    }
    
    private static long queryMin(Connection conn, String table, 
            String splitPk, String whereClause) {
        String sql = "SELECT MIN(" + splitPk + ") FROM " + table;
        if (whereClause != null) sql += " WHERE " + whereClause;
        return querySingleLong(conn, sql);
    }
    
    private static long queryMax(Connection conn, String table,
            String splitPk, String whereClause) {
        String sql = "SELECT MAX(" + splitPk + ") FROM " + table;
        if (whereClause != null) sql += " WHERE " + whereClause;
        return querySingleLong(conn, sql);
    }
    
    private static long querySingleLong(Connection conn, String sql) {
        try (Statement stmt = conn.createStatement();
             ResultSet rs = stmt.executeQuery(sql)) {
            if (rs.next()) return rs.getLong(1);
        } catch (SQLException e) {
            throw new RuntimeException("Query failed", e);
        }
        return 0;
    }
    
    /**
     * 后备方案：等距切分
     */
    private static List<String> equalDistanceSplit(
            Connection conn, String table, String splitPk,
            int splitNum, String whereClause) {
        long min = queryMin(conn, table, splitPk, whereClause);
        long max = queryMax(conn, table, splitPk, whereClause);
        long step = (max - min) / splitNum;
        
        List<String> splits = new ArrayList<>();
        for (int i = 0; i < splitNum; i++) {
            long lower = min + i * step;
            long upper = (i == splitNum - 1) ? max + 1 : min + (i + 1) * step;
            splits.add(lower + "," + upper);
        }
        return splits;
    }
}
```

**在 Reader.Job.split() 中集成**：

```java
@Override
public List<Configuration> split(int adviceNumber) {
    List<Configuration> configurations = new ArrayList<>();
    
    // 注入键: adaptSplit
    boolean useAdaptive = this.originalConfig.getBool("adaptSplit", false);
    
    Connection conn = DBUtil.getConnection(this.originalConfig);
    long totalRows = DBUtil.queryTotalRows(conn, table, whereClause);
    
    List<String> splitBoundaries;
    if (useAdaptive) {
        // 使用自适应切分
        splitBoundaries = AdaptiveSplitUtil.adaptiveSplit(
            conn, table, splitPk, adviceNumber, totalRows, whereClause);
    } else {
        // 使用默认等距切分
        splitBoundaries = SingleTableSplitUtil.split(
            adviceNumber, minPK, maxPK);
    }
    
    for (String boundary : splitBoundaries) {
        String[] parts = boundary.split(",");
        Configuration conf = this.originalConfig.clone();
        conf.set("lowerBound", Long.parseLong(parts[0]));
        conf.set("upperBound", Long.parseLong(parts[1]));
        configurations.add(conf);
    }
    
    DBUtil.closeConnection(conn);
    return configurations;
}
```

### 3.4 步骤四：TaskGroup 负载均衡——`assignFairly` 源码剖析

**目标**：理解即使分片已经均匀，Task 分配到 TaskGroup 时如何保证公平。

**源码关键位置**：`core/src/main/java/com/alibaba/datax/core/util/JobAssignUtil.java`

```java
public static List<Configuration> assignFairly(
        List<Configuration> taskConfigs, int taskGroupNum) {
    
    // 每个 TaskGroup 维护自己的任务数（工作量计数器）
    int[] taskCounters = new int[taskGroupNum];
    List<List<Configuration>> taskGroups = new ArrayList<>();
    for (int i = 0; i < taskGroupNum; i++) {
        taskGroups.add(new ArrayList<>());
    }
    
    // 贪心算法：每次找当前工作量最小的 TaskGroup
    for (int i = 0; i < taskConfigs.size(); i++) {
        int minIndex = 0;
        for (int j = 1; j < taskGroupNum; j++) {
            if (taskCounters[j] < taskCounters[minIndex]) {
                minIndex = j;
            }
        }
        taskGroups.get(minIndex).add(taskConfigs.get(i));
        taskCounters[minIndex]++;
    }
    
    // 展平为结果列表（按 taskGroupIndex 标记）
    List<Configuration> result = new ArrayList<>();
    for (int i = 0; i < taskGroupNum; i++) {
        for (Configuration conf : taskGroups.get(i)) {
            conf.set("taskGroupIndex", i);
            result.add(conf);
        }
    }
    return result;
}
```

**算法特点**：
- 时间复杂度：O(N × M)，N 为 Task 数，M 为 TaskGroup 数
- 保证任意两个 TaskGroup 的 Task 数量 **最大差 ≤ 1**
- 但不考虑每个 Task 的数据量差异——如果 `split()` 没做好均分，assignFairly 也无能为力

### 3.6 可能遇到的坑及解决方法

**坑1：二分查找的 COUNT 查询太慢**

如果表有 5 亿行，每次 COUNT 都要扫描索引。用 `EXPLAIN` 确认走的是：

```sql
EXPLAIN SELECT COUNT(*) FROM orders WHERE order_id < 500000000;
-- type: range, key: PRIMARY → 走索引 range scan，没问题
```

如果没有索引：先建索引，否则每次 COUNT 都是全表扫描，10 次 COUNT 可能需要几分钟。

**坑2：splitPk 有重复值导致的边界误差**

如果 splitPk 不是主键、存在重复值，二分查找的"第 targetRow 行的 splitPk 值"不唯一。此时改用 `COUNT(DISTINCT splitPk)` 或改用 `ROW_NUMBER() OVER (ORDER BY splitPk)` 窗口函数。

**坑3：`adviceNumber` 大于实际数据分布粒度**

1 万行数据配 channel=100 → 每个 Task 只有 100 行，不需要自适应切分。在 `split()` 中判断：如果 `totalRows / adviceNumber < 10000`，降级为等距切分或减少 Task 数。

## 4. 项目总结

### 4.1 数据倾斜解决方案对比

| 方案 | 原理 | 倾斜修复效果 | 性能开销 | 适用场景 |
|------|------|-------------|---------|---------|
| 等距切分 | (MAX-MIN)/N | 无 | 极低（2 次查询） | 数据均匀分布 |
| 哈希取模 | MOD(CRC32(col), N) | 好 | 高（全表扫描） | UUID 等非数值键 |
| 字符前缀 | col >= 'prefix' | 中 | 低（索引扫描） | 均匀字符串键 |
| 自适应采样 | 二分查找行数边界 | 好 | 中（N 次 COUNT） | 数值键分布不均 |
| querySql 手动 | 用户手写 WHERE | 最好 | 无额外开销 | 用户最了解数据 |

### 4.2 优点

1. **自适应切分消除长尾**：每个 Task 的数据量基本持平，CV 从 1.33 降到 0.02
2. **不影响原有接口**：通过 `adaptSplit` 配置开关，默认关闭，向后兼容
3. **与 `assignFairly` 双重保障**：split 层消除数据量不均 + assignFairly 消除 Task 数不均
4. **二分查找利用索引**：`WHERE splitPk < X` 走 range scan，性能可接受

### 4.3 缺点

1. **二分查找的 COUNT 次数 = adviceNumber × log₂(totalRows)**：大表可能需要几十次 COUNT
2. **依赖 splitPk 索引**：没有索引时退化严重
3. **非数值 splitPk 不适用**：二分查找需要数值比较，字符串、UUID 需另外处理
4. **不解决写入倾斜**：只解决读取倾斜，如果 Writer 端也有热点（如按 hash key 写分区），需要额外处理

### 4.4 注意事项

1. `adaptSplit` 仅在数据量 > 100 万行且倾斜系数 > 0.5 时开启
2. 二分查找的终止条件设为 `lo < hi`（不要在 10 亿行表上精确到 1 行）
3. `assignFairly` 是贪心算法，保证 Task 数均衡但不保证数据量均衡
4. QuerySql 模式不支持自适应切分——因为用户已手写 WHERE 条件
5. 如果倾斜是由于数据分布随时间变化（如分区表），建议每周重新执行一次自适应采样
