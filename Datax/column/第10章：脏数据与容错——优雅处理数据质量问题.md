# 第10章：脏数据与容错——优雅处理数据质量问题

## 1. 项目背景

某广告平台每天将广告投放日志（MongoDB）通过 DataX 同步到 MySQL 数据仓库。日志数据由广告 SDK 自动采集，字段质量参差不齐：有的 `click_time` 字段是时间戳字符串，有的是 Unix 毫秒数；有的 `device_id` 为空字符串，有的是 null；有的 `ad_price` 填写了 "N/A" 而非数字。

第一次执行同步任务时，团队配置了默认的 `errorLimit.record=0`（零容忍）。结果任务跑到 30% 时报错退出——一条数据 `ad_price="N/A"` 尝试转为 DoubleColumn 时抛了 `NumberFormatException`，被当作脏数据，累积到 1 条超过了 limit=0。

运维排查发现——原来 MongoDB 中约 0.3%（3000 条 / 100 万条）的数据存在字段质量问题。团队改配置：`errorLimit.percentage=0.02`（容忍 2%），任务顺利完成——3000 条脏数据被收集到脏数据日志，950000 条正常数据成功写入 MySQL。但这次经历暴露了一个重大设计问题：**3000 条脏数据中，有 1000 条是核心付费数据（只是 click_time 格式不标准），丢失它们导致财务对账差了几万元。**

DataX 的脏数据机制是"粗粒度选择题"——要么全部接受（含脏数据），要么按阈值整批丢弃。生产环境中，脏数据不是"非黑即白"的问题，而是需要分级处理、逐条修复的精细化管理。本章带你掌握脏数据的收集、分析和修复策略。

## 2. 项目设计——剧本式交锋对话

**（周一晨会，数据质量看板上红灯闪烁——"昨日同步数据缺失率 0.3%"）**

**小胖**：（看着报表，很困惑）怎么少了 3000 条数据？任务不是显示"读写失败总数: 3000, 任务成功"吗？

**小白**：（表情严肃）"读写失败总数"就是脏数据！这些数据被写到脏数据日志里，但没有进目标库。成功是指任务没被脏数据超限中断，不代表数据全进去了。

**小胖**：（急了）那岂不是我的核心 KPI 数据都丢了？这 3000 条里有多少是付费数据啊！

**大师**：（冷静地打开脏数据日志）问题在于我们把脏数据的阈值设得太松了。`errorLimit.percentage=0.02` 意味着 100 万条里允许 2 万条垃圾。但真正的垃圾只有 3000 条。如果设为 `errorLimit.record=0`，任务会失败，你至少知道出问题了。现在任务成功了，反而没人注意到数据丢了。

**技术映射**：errorLimit = 安检标准。设太松（percentage=10%）→ 炸弹（数据异常）也能过。设太严（record=0）→ 带瓶矿泉水（小异常）就过不了。

**小白**：（追问）那脏数据日志里到底能看到什么？3000 条数据都有详细记录吗？

**大师**：（指着日志）你看这段：

```
2026-05-06 02:30:15 [taskGroup-0] WARN  TaskPluginCollector - 脏数据:
{"record":"[StringColumn:user_12345, DoubleColumn:N/A←这里! , DateColumn:2026-05-06]",
 "exception":"java.lang.NumberFormatException: For input string: \"N/A\"",
 "message":"无法将值[N/A]转换为DoubleColumn"}
```

脏数据日志包含三个关键信息：**哪条 Record**（列值明细）、**什么异常**（NumberFormatException）、**错误信息**（无法转换）。但——只打印默认的 10 条脏数据详情，其余 2990 条只存计数。

**小胖**：那我们怎么修复这些脏数据？重新从 MongoDB 拉到本地，手工改完再写进去？

**大师**：（摇头）3000 条可以手工修，30 万条呢？正确的做法分三步：
1. **分类**：看脏数据都是什么原因造成的——类型转换错误？空值？超长字段？
2. **修复**：在 Reader 阶段用 Transformer 或 querySql 做预处理：
   - `ad_price` 不是数字 → `CAST(ad_price AS DECIMAL(10,2))` 自动转为 null + 脏数据
   - 或使用 `dx_groovy` Transformer：`if (record.getColumn(3).asString().equals("N/A")) record.setColumn(3, new LongColumn(0L));`
3. **兜底**：在 Writer 阶段用 `session` 参数设置容错模式（如 `SET sql_mode=''` 允许宽松写入）

**小白**：（快速记笔记）那如果脏数据实在无法修复呢？比如 click_time 格式完全乱码，无法匹配任何日期格式。

**大师**：那就接受它的"丢失"，但要做好记录——在业务方允许的数据质量 SLA 范围内执行。比如广告平台允许丢 0.5% 的数据，那你的 `errorLimit.percentage=0.005`。严谨的做法是：执行两次任务——第一次 `errorLimit.record=0` 严格模式，发现脏数据来源；第二次 `errorLimit.percentage=0.01` 宽松模式，完成同步，同时把脏数据单独导出为 CSV 交给运营手工核对。

## 3. 项目实战

### 3.1 步骤一：errorLimit 参数详解

```json
{
    "setting": {
        "errorLimit": {
            "record": 0,
            "percentage": 0.02
        }
    }
}
```

| 参数 | 类型 | 含义 | 示例 | 推荐值 |
|------|------|------|------|--------|
| `record` | int | 最大允许脏记录数 | 1000 | 生产环境建议 500~5000 |
| `percentage` | double | 最大允许脏记录百分比（0~1） | 0.02（2%） | 取决于 SLA（0.001~0.1） |

**触发逻辑（源码）**：

```java
// ErrorRecordChecker.java
public static void checkErrorRecord(Communication communication, Configuration configuration) {
    long totalRecords = communication.getLongCounter(READ_SUCCEED_RECORDS) 
                       + communication.getLongCounter(READ_FAILED_RECORDS);
    
    long failedRecords = communication.getLongCounter(READ_FAILED_RECORDS);
    long maxFailedRecords = configuration.getLong("errorLimit.record", 0);
    double maxFailedPercentage = configuration.getDouble("errorLimit.percentage", 0);
    
    // 检查 1：脏记录数是否超过阈值
    if (maxFailedRecords > 0 && failedRecords > maxFailedRecords) {
        throw DataXException.asDataXException(FrameworkErrorCode.PLUGIN_DIRTY_DATA_LIMIT_EXCEED,
            String.format("脏数据记录数[%d]超过限制[%d]", failedRecords, maxFailedRecords));
    }
    
    // 检查 2：脏数据百分比是否超过阈值
    if (maxFailedPercentage > 0 && totalRecords > 0) {
        double actualPercentage = (double) failedRecords / totalRecords;
        if (actualPercentage > maxFailedPercentage) {
            throw DataXException.asDataXException(FrameworkErrorCode.PLUGIN_DIRTY_DATA_LIMIT_EXCEED,
                String.format("脏数据百分比[%.4f]超过限制[%.4f]", actualPercentage, maxFailedPercentage));
        }
    }
}
```

**两个条件是 OR 关系**——触发任意一个就抛异常终止 Job。

### 3.2 步骤二：查看脏数据收集器源码

**目标**：理解脏数据从产生到收集的完整链路。

```java
// AbstractTaskPluginCollector.java
public class AbstractTaskPluginCollector implements TaskPluginCollector {
    
    // 每条 Task 独立维护脏数据列表（线程安全）
    private final List<String> dirtyRecords = Collections.synchronizedList(new ArrayList<>());
    private static final int MAX_DIRTY_RECORD_LOG = 10; // 最多打印10条详情
    
    @Override
    public void collectDirtyRecord(Record dirtyRecord, Throwable t, String errorMessage) {
        // 1. 构建脏数据详情（JSON格式）
        String recordStr = buildDirtyRecordString(dirtyRecord);
        
        // 2. 记录到Task级脏数据列表
        if (dirtyRecords.size() < MAX_DIRTY_RECORD_LOG) {
            dirtyRecords.add(recordStr + " | " + t.getMessage());
        }
        
        // 3. 打印到日志
        LOG.warn("脏数据: {\"record\":\"" + recordStr 
            + "\",\"exception\":\"" + t.getClass().getSimpleName() 
            + "\",\"message\":\"" + errorMessage + "\"}");
        
        // 4. 递增统计计数器
        this.communication.increaseCounter(READ_FAILED_RECORDS, 1);
    }
}
```

**关键限制**：`MAX_DIRTY_RECORD_LOG = 10`。即使产生了 10 万条脏数据，日志中只有前 10 条有详情，其余只有计数。如果需要完整脏数据，建议在生产环境修改此阈值为更大的数，或自行扩展收集器将脏数据写入文件/数据库。

### 3.3 步骤三：实战——构造脏数据场景

**目标**：在测试环境中模拟字段质量问题，验证脏数据机制的行为。

**第 1 步**：在源端 MySQL 建表并插入混合质量数据。

```sql
CREATE TABLE order_dirty_test (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,

    amount VARCHAR(20),      -- 故意用VARCHAR模拟劣质数据
    create_time VARCHAR(30), -- 同上
    status INT

-- 正常数据 × 950
INSERT INTO order_dirty_test (user_name, amount, create_time, status)
SELECT 
    CONCAT('user_', n),
    ROUND(RAND()*1000, 2),     -- 正常的数字
    DATE_FORMAT(DATE_ADD('2026-01-01', INTERVAL n DAY), '%Y-%m-%d'),  -- 正常的日期
    n % 5
FROM (SELECT @rownum:=@rownum+1 AS n FROM (SELECT 0 UNION ALL SELECT 1 UNION ALL ...) t1
      CROSS JOIN (SELECT @rownum:=0) r) nums

-- 脏数据 × 50

('bad_user_1', 'N/A', '2026-13-01', 1),          -- amount错误 + 月份13
('bad_user_2', '-999', 'not_a_date', 2),          -- create_time乱码
('bad_user_3', '0.001', '2026-05-06', -1),        -- 正常（-1也是合法int）
```

**第 3 步**：分别测试三种 errorLimit 配置。

| 配置 | 结果 | 脏数据计数 | 任务状态 |
|------|------|-----------|---------|
| `record=0, percentage=0` | 第1条脏数据出现即中止 | 1 | 失败 |
| `record=100, percentage=0` | 50条脏数据 < 100，任务完成 | 50 | 成功 |
| `record=10, percentage=0` | 第11条脏数据触发超限 | 11 | 失败 |
| `record=0, percentage=0.1` | 50/1000=5% > 10% → 不超限 | 50 | 成功 |

### 3.4 步骤四：脏数据的修复方案

**目标**：用 Transformer + querySql 实现脏数据的自动清洗。

**方案 A：在 Reader 端用 querySql 预清洗**

```sql
SELECT 
    id,
    user_name,
    -- 尝试将 amount 转为数字，失败返回 null
    CAST(NULLIF(amount, 'N/A') AS DECIMAL(10,2)) AS amount,
    -- 尝试解析日期，失败返回 null
    STR_TO_DATE(NULLIF(create_time, 'not_a_date'), '%Y-%m-%d') AS create_time,
    status
FROM order_dirty_test
WHERE $CONDITIONS
```

`NULLIF(amount, 'N/A')` 将 `'N/A'` 转为 NULL → CAST(NULL AS DECIMAL) → NULL → MySQL Reader 读出为 null → DoubleColumn(null)，不会报错。但 null 值最终会带来 Writer 写入的问题（取决于 `setWithNull` 参数）。

**方案 B：用 Transformer + Groovy 脚本智能处理**

```json
"transformer": [{
    "name": "dx_groovy",
    "parameter": {
        "code": "import com.alibaba.datax.common.element.*;\n"
                + "if (record.getColumn(2).asString() == 'N/A') {\n"
                + "    record.setColumn(2, new LongColumn(0L));\n"
                + "}\n"
                + "try {\n"
                + "    java.text.SimpleDateFormat sdf = new java.text.SimpleDateFormat('yyyy-MM-dd');\n"
                + "    sdf.parse(record.getColumn(3).asString());\n"
                + "} catch (Exception e) {\n"
                + "    record.setColumn(3, new DateColumn(new java.util.Date(0L)));\n"  // 设为1970-01-01
                + "}\n"
                + "return record;"
    }
}]
```

**方案 C：Writer 端用 `setWithNull` 处理 null 值**

```json
"writer": {
    "parameter": {
        "nullFormat": "\\N",
        "writeMode": "insert",
        "connection": [...]
    }
}
```

当 Reader 传来的 Record 中某列是 null，Writer 会按 `nullFormat` 的值写入目标库。MySQL Writer 默认用 `"null"` 字符串，可改为 `\N`（与 Hive 文本格式兼容）。

### 3.5 可能遇到的坑及解决方法

**坑1：脏数据日志打印不完整**

只打印前 10 条详情，后续只有计数。

解决：重写 `TaskPluginCollector.collectDirtyRecord()`，将脏数据写入独立文件或数据库表：

```java
// 扩展方案：将脏数据写入文件
public class FileDirtyCollector extends AbstractTaskPluginCollector {
    private PrintWriter writer;
    
    @Override
    public void init() {
        writer = new PrintWriter(new FileWriter("dirty_data_" + taskId + ".json"));
    }
    
    @Override
    public void collectDirtyRecord(Record dirtyRecord, Throwable t, String msg) {
        String json = buildDirtyRecordJson(dirtyRecord, t, msg);
        writer.println(json);
        super.collectDirtyRecord(dirtyRecord, t, msg); // 保留原有计数逻辑
    }
    
    @Override
    public void destroy() {
        writer.close();
    }
}
```

**坑2：partial update 导致脏数据超限**

Writer 用 update 模式时，如果某条 UPDATE 影响了 0 行（WHERE 条件匹配不到），某些 Writer 插件会把它当作脏数据。但实际上"没匹配到"不是数据质量问题。

解决：查看具体 Writer 插件的实现。MySQL Writer 的 update 模式在受影响行数为 0 时**不会**收集脏数据。

**坑3：脏数据阈值在分阶段任务中的累积**

如果 Job 包含 3 组 content（3 组 Reader-Writer），errorLimit 是针对 Job 整体的。第 1 组产生 1000 条脏数据，第 2 组产生 500 条，合计 1500 条——如果 record=1000，任务会在第 2 组执行时报超限。

## 4. 项目总结

### 4.1 脏数据处理决策树

```
数据质量是否可接受？
├─ 是 → 设 errorLimit.percentage = SLA允许的最大丢失率
│       └─ 定期review脏数据日志，评估是否需要修复
└─ 否 → 第一步：errorLimit.record = 0 严格模式 → 发现脏数据来源
        └─ 第二步：修复源数据 or 用Transformer自动清洗
            └─ 第三步：重新执行，逐步放宽 errorLimit 直到通过
```

### 4.2 优点

1. **双重检查**：record 绝对值和 percentage 百分比互为补充
2. **自动收集**：每条脏数据自动记录到 Task 级收集器，无需手动写 try-catch
3. **提前终止**：脏数据超限立即终止 Job，避免大量错误数据污染目标库
4. **日志可追溯**：脏数据详情含列值、异常类型、错误信息三维信息
5. **零侵入**：脏数据机制对正常数据流零影响（只统计不阻塞）

### 4.3 缺点

1. **粗粒度**：无法区分"重要脏数据"和"可丢弃脏数据"，一刀切策略
2. **详情截断**：默认只打印 10 条脏数据详情，海量脏数据时信息丢失
3. **无自动修复**：脏数据只能被记录，不会触发自动修复流程
4. **阈值无动态调整**：不会根据脏数据趋势自动修改 errorLimit
5. **脏数据不落盘**：收集器只在内存中保留前 10 条，任务结束后无法回溯

### 4.4 适用场景

1. 产线环境全量同步（设 percentage=0.001，99.9% 数据质量要求）
2. 数据迁移前的质量评估（设 record=0 严格模式，找出所有脏数据）
3. 实时日志采集同步（大量脏数据，设 percentage=0.1 放宽容错）
4. 测试环境数据灌入（设 record=-1，不管脏数据全力灌入）

### 4.5 注意事项

1. `record=0` 和 `percentage=0` 同时设 0 结果一样——零容忍
2. 计算 percentage 的分母**包含脏数据条目**（总记录数 = 成功 + 失败）
3. 脏数据计数是累加的，不会因为"某条脏数据后来修复了"而回溯
4. 不要在生产环境同时设 `record=0` 和 `percentage=0`（任何一条脏数据都会终止）
5. 脏数据日志中的列顺序和 JSON 配置的 column 数组顺序一致

### 4.6 思考题

1. 如果一个 Job 的 `content` 数组有 3 组 Reader-Writer，第 1 组产生 500 条脏数据，第 2 组产生 400 条，第 3 组产生 200 条。`errorLimit.record=1000` 时任务会成功吗？如果第 2 组和第 3 组调换执行顺序呢？
2. DataX 的脏数据机制能否和 Transformer 协同工作？比如——用 Transformer 检测字段是否为 "N/A"，是则当作脏数据标记但不丢弃（留给 Writer 处理），而不是直接在 Transformer 中修改值？
