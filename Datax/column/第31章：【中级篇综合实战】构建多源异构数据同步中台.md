# 第31章：【中级篇综合实战】构建多源异构数据同步中台

## 1. 项目背景

某大型电商平台的数据架构经过五年的野蛮生长，沉淀出三套异构数据体系：MySQL 集群承担订单与交易核心链路（日均 3 亿行写入），MongoDB 存储用户行为日志与商品评论（日均 5 亿条 JSON 文档），HDFS 数据湖存放历史归档与推荐算法离线特征（PB 级 Parquet 文件）。三套体系的共同问题是——**数据孤岛**。

分析师想做"用户浏览→下单→支付转化漏斗"分析，需要同时关联 MySQL 订单表、MongoDB 行为日志和 HDFS 上的用户画像宽表。当前做法是：每天凌晨由三个独立脚本分别从三个源抽取数据、写成 CSV 放在 FTP 上、再由第四个脚本导入分析库 ClickHouse。这一链路有四个致命缺陷：

1. **时效性差**：四个脚本串行执行，整个链路跑完 5~7 小时，分析师上午 10 点才能看到昨天的数据
2. **一致性无保证**：如果 Mongo→FTP 脚本报错但 MySQL→FTP 成功了，分析库里的订单数据就比行为数据多——数据"断裂"
3. **运维成本高**：四套脚本分别由三个团队维护，用 Python/Shell/Java 写成，任何数据源 Schema 变更都要改四份代码
4. **扩展性差**：新增一个数据源（比如 Kafka 实时流）需要新增一个 ETL 脚本、一个 FTP 落地点、一个导入脚本

架构组下了决心：**用 DataX + 配置管理平台 + 调度器 + 监控告警，构建统一的"多源异构数据同步中台"**。所有异构数据源的读取统一由 DataX 的 Reader 插件封装，写入统一由 Writer 插件封装，配置集中管理，调度统一编排，监控统一采集。

目标指标：日同步 100 亿条记录、任务成功率 99.9%、P99 同步延迟 < 5 分钟。

## 2. 项目设计——剧本式交锋对话

**（周一上午，架构组周会，白板上画满了三个数据源的拓扑图）**

**小胖**：（端着一杯奶茶走进来）我听说要搞"数据同步中台"？这不就是把之前的三个脚本改成 DataX 配置吗？一个月够了吧。

**小白**：（指着白板上的箭头）你仔细看——MySQL 要做增量同步，MongoDB 有嵌套文档要展平，HDFS 上的结果还要回流到 MySQL 给业务系统查询。这哪是改配置的事？

**大师**：（在白板上写下四个关键词）这个项目的核心挑战不是"同步"本身，而是四个关键词——**异构、增量、回流、管控**。我们先拆解这几个点。

**异构问题**：MySQL 是关系型（二维表、有 schema）、MongoDB 是文档型（嵌套 JSON、无 schema）、HDFS 是文件型（Parquet/ORC、列存）。DataX 的架构优势就体现在这里——Reader/Writer 插件体系天然屏蔽了异构差异。上层的 Job 逻辑不关心数据来源是什么，只看到统一的 Record 流。

**技术映射**：DataX 的 Reader/Writer 插件 = 万能电源适配器。不同国家的插座（异构数据源）插上对应的转换头（插件），出来的都是 5V USB 电（统一 Record 流）。

```
MySQL → mysqlreader → Record → mysqlwriter → 数据仓库
MongoDB → mongodbreader → Record → hdfswriter → HDFS
HDFS   → hdfsreader  → Record → mysqlwriter → MySQL(报表)
```

**小胖**：等等，增量同步怎么做？MySQL 每天 3 亿行，总不能全量重跑吧？

**大师**：关键参数：`splitPk` + `where` 条件。MySQL Reader 支持在 `where` 里写 `update_time >= '${yesterday}' AND update_time < '${today}'`，配合调度器传入前一天的时间参数。同时设 `splitPk=id`，DataX 会按 ID 范围等距切分 20 个 Task，每个 Task 只拉一个 ID 区间内的增量数据。

**小白**：（追问）那 MongoDB 呢？它的增量条件怎么写？MongoDB 的查询语法可不是 SQL。

**大师**：（翻到下一页 PPT）MongoDB Reader 的 `query` 参数支持原生 MongoDB 查询语法：

```json
{
    "query": "{ \"ts\": { \"$gte\": { \"$date\": \"2026-05-05T00:00:00Z\" }, \"$lt\": { \"$date\": \"2026-05-06T00:00:00Z\" } } }"
}
```

这等价于 `db.collection.find({"ts": {"$gte": ISODate(...), "$lt": ISODate(...)}})`。MongoDB Reader 把这个 query 传给 MongoDB 服务端执行，利用 MongoDB 的 `ts` 字段索引做范围扫描，效率很高。

**小胖**：（举手）那嵌套文档展平呢？MongoDB 一条文档里有 `user.id`、`metadata.os` 这种嵌套字段，怎么变成 MySQL 的二维表？

**大师**：MongoDB Reader 的 `column` 配置支持**点号路径**：

```json
{
    "column": [
        {"name": "_id", "type": "string"},
        {"name": "user.id", "type": "int"},
        {"name": "user.name", "type": "string"},
        {"name": "metadata.device", "type": "string"},
        {"name": "metadata.os", "type": "string"}
    ]
}
```

Reader 内部会把 `user.id` 解析为 JSON 路径 `$.user.id`，自动提取展平。但如果遇到数组字段（如 `tags: ["promo", "new_user"]`），点号路径就不好使了——这时候需要 Groovy Transformer：

```groovy
// 将 tags 数组转为逗号分隔字符串
import com.alibaba.datax.common.element.*;
import com.alibaba.fastjson.JSON;

// tagsJson 是从原始文档中解析的 JSON 数组字符串
String tagsJson = record.getColumn(5).asString();
if (tagsJson != null && tagsJson.startsWith("[")) {
    List<String> tags = JSON.parseArray(tagsJson, String.class);
    record.setColumn(5, new StringColumn(String.join(",", tags)));
}
return record;
```

**小白**：（追问）那 HDFS 结果回流到 MySQL 的意义是什么？数据湖不是已经有 Hive/Spark 了吗？

**大师**：业务系统（订单管理后台、客服查询系统）需要毫秒级的查询响应。Hive 做不到——一个 MapReduce 查询 30 秒起步。所以要"数据湖算一遍，结果集回流 MySQL"。比如"每个用户过去 30 天的购买力评分"，在 HDFS 上跑完 Spark ML 模型后，将打分结果通过 DataX 回流到 MySQL 的 `user_score` 表，业务系统用 SQL 直接查，延迟 < 10ms。

**小胖**：那这么多任务怎么调度？不会还是要手动一个个跑吧？

**大师**：用 Apache DolphinScheduler 或 Azkaban 做 DAG 编排。调度器的职责是：
1. 每天早上 6:00 触发 MySQL 增量同步（依赖前置检查通过）
2. MySQL 完成后，触发 MongoDB → HDFS 任务
3. HDFS 离线计算完成后，触发 HDFS → MySQL 回流
4. 每个节点有重试策略（失败重试 3 次，间隔 5 分钟）
5. 最终节点发出飞书/钉钉通知

再加上 Prometheus + Grafana 监控每个 Task 的 QPS、延迟、脏数据量——这就是一个完整的同步中台。

### 3.2 步骤二：MySQL 增量同步——核心链路

**目标**：实现 MySQL orders 表的每日增量同步，利用 `splitPk` + `where` 条件将前一天变更的数据同步到数据仓库。

**DataX 配置文件**（`jobs/mysql_orders_inc.json`）：

```json
{
    "job": {
        "content": [{
            "reader": {
                "name": "mysqlreader",
                "parameter": {
                    "username": "etl_user",
                    "password": "etl_pass_2026",
                    "column": ["order_id", "user_id", "product_id", "amount", "status", "channel", "create_time", "update_time"],
                    "splitPk": "order_id",
                    "where": "update_time >= '${yesterday}' AND update_time < '${today}'",
                    "connection": [{
                        "table": ["orders"],
                        "jdbcUrl": ["jdbc:mysql://10.0.1.100:3306/ecommerce?useSSL=false&useCursorFetch=true&defaultFetchSize=-2147483648"]
                    }]
                }
            },
            "writer": {
                "name": "mysqlwriter",
                "parameter": {
                    "username": "etl_user",
                    "password": "etl_pass_2026",
                    "writeMode": "replace",
                    "column": ["order_id", "user_id", "product_id", "amount", "status", "channel", "create_time", "update_time", "sync_batch"],
                    "preSql": ["ALTER TABLE orders_dw ADD COLUMN IF NOT EXISTS sync_batch VARCHAR(20)"],
                    "session": [
                        "SET unique_checks = 0",
                        "SET foreign_key_checks = 0"
                    ],
                    "batchSize": 4096,
                    "connection": [{
                        "table": ["orders_dw"],
                        "jdbcUrl": ["jdbc:mysql://10.0.2.100:3306/data_warehouse?useSSL=false&rewriteBatchedStatements=true"]
                    }]
                }
            },
            "transformer": [{
                "name": "dx_groovy",
                "parameter": {
                    "code": "import com.alibaba.datax.common.element.*;\n"
                            + "record.addColumn(new StringColumn('${batch_id}'));\n"
                            + "return record;"
                }
            }]
        }],
        "setting": {
            "speed": {
                "channel": 12,
                "byte": 104857600
            },
            "errorLimit": {
                "record": 0,
                "percentage": 0
            }
        }
    }
}
```

**参数说明**：
- `where` 中的 `${yesterday}` 和 `${today}` 是占位符，由调度器在运行时替换为实际日期值
- `splitPk=order_id`：增量数据也按 order_id 分片，避免单个 Task 拉取全量增量
- `writeMode=replace`：用 `REPLACE INTO` 实现幂等写入——同 order_id 的数据重复同步也不会出错
- `batchSize=4096`：增量数据量小，batchSize 适度即可

**调度器传参脚本**（`run_inc_job.sh`）：

```bash
#!/bin/bash
# 增量同步调度入口
YESTERDAY=$(date -d "yesterday" +"%Y-%m-%d")
TODAY=$(date +"%Y-%m-%d")
BATCH_ID="INC_${YESTERDAY}"

# 用 sed 替换占位符
sed -e "s/\${yesterday}/${YESTERDAY}/g" \
    -e "s/\${today}/${TODAY}/g" \
    -e "s/\${batch_id}/${BATCH_ID}/g" \
    jobs/mysql_orders_inc.template.json > jobs/mysql_orders_inc.json

python $DATAX_HOME/bin/datax.py jobs/mysql_orders_inc.json

if [ $? -eq 0 ]; then
    echo "MySQL orders incremental sync SUCCESS at $(date)"
else
    echo "MySQL orders incremental sync FAILED at $(date)"
    exit 1
fi
```

### 3.5 步骤五：DAG 编排与监控

**目标**：用 Shell 脚本模拟 DolphinScheduler 的 DAG 编排逻辑。

```bash
#!/bin/bash
# dag_orchestrator.sh — 三个同步链路的 DAG 编排
set -euo pipefail

YESTERDAY=$(date -d "yesterday" +"%Y-%m-%d")
LOG_BASE="logs/${YESTERDAY}"
mkdir -p "$LOG_BASE"

# 阶段 1：MySQL 增量（无依赖，最先启动）
echo "[DAG] Stage 1: MySQL incremental sync"
bash run_mysql_inc.sh > "${LOG_BASE}/mysql_inc.log" 2>&1
if [ $? -ne 0 ]; then
    echo "[DAG] MySQL incremental sync FAILED, abort pipeline"
    exit 1
fi

# 阶段 2：MongoDB → HDFS（依赖阶段 1 成功）
echo "[DAG] Stage 2: MongoDB → HDFS sync"
bash run_mongo_to_hdfs.sh > "${LOG_BASE}/mongo_to_hdfs.log" 2>&1
if [ $? -ne 0 ]; then
    # MongoDB 失败不阻塞后续流程（数据可容忍延迟一天）
    echo "[DAG] MongoDB sync FAILED, but continuing pipeline"
    echo "WARN: MongoDB sync failed at $(date)" >> "${LOG_BASE}/alerts.log"
fi

# 阶段 3：HDFS → MySQL 回流（依赖离线计算完成）
echo "[DAG] Stage 3: HDFS → MySQL report backfill"
bash run_hdfs_to_mysql.sh > "${LOG_BASE}/hdfs_to_mysql.log" 2>&1

# 阶段 4：数据校验
echo "[DAG] Stage 4: Data validation"
bash validate_all.sh > "${LOG_BASE}/validation.log" 2>&1

# 阶段 5：发送通知
echo "[DAG] Stage 5: Notify"
python send_notify.py --status "SUCCESS" --yesterday "$YESTERDAY"

echo "[DAG] All stages completed at $(date)"
```

### 3.6 可能遇到的坑及解决方法

**坑1：MongoDB splitter 分片失效**

MongoDB Reader 默认用 `_id` 的 ObjectId 做 splitter 分片。但如果 `_id` 不是 ObjectId 类型（比如自定义字符串），splitter 会因为无法按范围等分而退化为单 Task——所有数据堆积在一个 Task 上，性能极差。

```
报错: WARN - splitter failed, fallback to single task
解决: 在 query 中增加过滤条件缩小范围，或用 MongoDB 的 $bucketAuto 做服务端预分片
```

**坑2：MySQL 增量 where 条件中的 SQL 注入风险**

`${yesterday}` 占位符由 Shell 脚本的 `sed` 替换，如果调度器传入的日期格式错误（如 `2026-05-06'; DROP TABLE orders; --`），会导致 SQL 注入。

```
解决:
1. 在 Shell 中校验日期格式: date -d "$YESTERDAY" +"%Y-%m-%d" > /dev/null 2>&1
2. 使用 DataX 的参数化查询而非字符串拼接
```

**坑3：HDFS writer 的路径权限**

DataX 以当前用户的身份向 HDFS 写文件时，可能遇到 `Permission denied: user=datax, access=WRITE`。

```
解决:
1. 在 HDFS 上预先创建目录并授权: hadoop fs -chmod -R 777 /data_lake/behavior_log
2. 或设置 DataX 启动用户为 Hadoop 代理用户: export HADOOP_USER_NAME=hdfs
```

**坑4：MongoDB 连接池耗尽**

channel=20 时，每个 Task 需要 1 个 MongoDB 连接 = 20 个连接。如果 MongoDB 服务端 `maxIncomingConnections` 默认 65536 没问题，但中间如果有防火墙限制连接数，会导致部分 Task 卡住。

```
解决: 检查 MongoDB 连接数并调高
db.serverStatus().connections  # 查看当前连接数
```

## 4. 项目总结

### 4.1 中台架构总览

```
┌──────────────────────────────────────────────────────┐
│                    调度层 (DolphinScheduler)            │
│  ┌─────────┐    ┌─────────────┐    ┌──────────────┐  │
│  │ MySQL增量│───▶│ MongoDB→HDFS│───▶│ HDFS→MySQL回流│  │
│  └─────────┘    └─────────────┘    └──────────────┘  │
└──────────────────────────────────────────────────────┘
                         │
┌──────────────────────────────────────────────────────┐
│                  监控层 (Prometheus + Grafana)          │
│   QPS | 延迟 | 脏数据 | GC | 内存 | CPU | 网络IO      │
└──────────────────────────────────────────────────────┘
                         │
┌──────────────────────────────────────────────────────┐
│                   配置管理平台 (Git + 模板引擎)           │
│   JSON模板 → 占位符替换 → 运行时配置 → 归档 → 版本管理  │
└──────────────────────────────────────────────────────┘
```

### 4.2 优点

1. **统一抽象**：MySQL、MongoDB、HDFS 三种异构数据源全部收敛到 DataX 的 Record 流——新增数据源只需开发一个 Reader/Writer 插件
2. **增量可复用**：MySQL 的 `where + splitPk` 模式可复用到 PostgreSQL、Oracle 等所有 RDBMS Reader
3. **幂等保证**：`writeMode=replace` 确保重复执行不会产生重复数据
4. **DAG 编排灵活**：三个链路独立可维护，任意节点失败可选择重试、跳过或中断
5. **监控可观测**：每个 Task 的 QPS 和延迟都可在 Grafana 上实时查看

### 4.3 缺点

1. **调度器自建成本高**：DolphinScheduler 需要额外部署维护，轻量场景可用 Shell + cron 替代
2. **跨数据类型映射复杂**：MongoDB 嵌套文档展平到 MySQL 二维表时，列数会膨胀——一个含 10 个嵌套字段的文档展平后可能有 40+ 列
3. **MongoDB 增量非精确**：MongoDB Reader 基于 `query` 过滤时间字段，如果写入延迟 5 分钟，增量窗口边界的数据可能丢失
4. **HDFS 回流依赖 T+1**：如果 HDFS 离线计算在 8:00 完成，但 DataX 回流任务 7:00 就启动了——需要依赖检测机制

### 4.4 适用场景与不适用场景

**适用场景**：
1. 多数据源融合到统一数据仓库（数据中台核心链路）
2. Hadoop 离线计算结果回流到在线业务库
3. 跨数据中心数据同步（利用 DataX 的限速保护带宽）
4. 历史数据归档（MySQL → HDFS → Parquet 压缩）
5. 异构数据库间的定期对账

**不适用场景**：
1. 实时同步（毫秒级延迟）——改用 Kafka + Flink CDC
2. 双向同步（主主互备）——需处理冲突检测
