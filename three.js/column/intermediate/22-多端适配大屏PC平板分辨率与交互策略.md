# 第22章：多端适配大屏PC平板分辨率与交互策略

## 元信息
- 学习目标：实现同一套三维应用在大屏、PC、平板上的稳定展示与交互体验。
- 先修要求：完成实时同步章节，具备状态驱动架构。
- 预估时长：5-7 小时。
- 最终产出：多端适配策略（分辨率、交互、性能档位）及落地代码模板。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

项目上线后最常见投诉之一是“同一页面在不同设备表现完全不同”：会议室大屏看起来正常，PC 上 UI 挤压，平板上点选偏移。三维项目里这个问题更突出，因为不仅有 DOM 布局，还有相机视角、像素比、触控交互等多维变量。若无系统适配策略，每加一个端都像重做一版。

业务侧常见场景包括：运维值班看大屏、工程师在 PC 深度操作、领导在平板巡检。三种角色关注点不同：大屏要总览稳定，PC 要交互精细，平板要触控容错。技术上如果只做“CSS 自适应”，远远不够。

本章目标是建立多端统一方案：定义端能力模型、分辨率策略、交互模式切换、性能档位降级，并把端差异控制在配置层，避免业务代码分叉。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：只靠响应式够吗

小胖：  
“把 UI 响应式做好不就行了？”

小白：  
“三维还涉及相机 FOV、像素比、触摸与鼠标事件，不只是 CSS。”

大师：  
“多端适配是全链路问题：渲染、交互、布局都要有策略。”  

技术映射：  
“全链路适配” = `render + interaction + layout adaptation`。

### 第二轮：端差异如何管理

小胖：  
“每个端写一套 if/else 最直接。”

小白：  
“if/else 会失控，建议抽成 profile 配置。”

大师：  
“用配置描述端能力，代码消费配置，不要把端逻辑写死。”  

技术映射：  
“能力配置化” = `device profile driven design`。

### 第三轮：触控和鼠标冲突

小胖：  
“平板点一下经常触发拖拽，体验很差。”

小白：  
“需要区分 tap、pan、pinch，并设置阈值。”

大师：  
“交互协议要按输入设备重定义，而不是照搬 PC。”  

技术映射：  
“输入协议” = `pointer gesture abstraction`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
```

### 3.2 分步实现

#### 步骤1：定义端能力 Profile

```ts
type DeviceProfile = {
  id: "wall" | "pc" | "tablet";
  pixelRatioCap: number;
  fov: number;
  interaction: "mouse" | "touch";
  quality: "high" | "medium" | "low";
};

const profiles: Record<string, DeviceProfile> = {
  wall: { id: "wall", pixelRatioCap: 1.5, fov: 58, interaction: "mouse", quality: "high" },
  pc: { id: "pc", pixelRatioCap: 1.5, fov: 60, interaction: "mouse", quality: "medium" },
  tablet: { id: "tablet", pixelRatioCap: 1.0, fov: 65, interaction: "touch", quality: "low" }
};
```

结果：端差异被配置化。  
坑：profile 仅影响 UI，不影响渲染参数。

#### 步骤2：按 profile 应用渲染策略

```ts
function applyProfile(p: DeviceProfile) {
  renderer.setPixelRatio(Math.min(window.devicePixelRatio, p.pixelRatioCap));
  camera.fov = p.fov;
  camera.updateProjectionMatrix();
  applyQualityLevel(p.quality);
}
```

结果：各端画面稳定且性能可控。  
坑：切 profile 后未重设后处理尺寸。

#### 步骤3：统一指针手势层

```ts
function bindInputMode(mode: "mouse" | "touch") {
  if (mode === "touch") {
    enableTapSelect({ tapMoveThreshold: 8 });
    enablePinchZoom();
  } else {
    enableMousePick();
    enableWheelZoom();
  }
}
```

结果：触控端误触率下降。  
坑：未阻止默认行为导致页面滚动冲突。

## 4. 项目总结（约500-800字）

### 优点
1. 单代码基线支持多端。  
2. 端差异可配置，迭代成本低。  
3. 交互体验更符合设备特性。

### 缺点
1. 需要维护 profile 组合。  
2. 跨端调试与测试成本上升。  
3. 极端分辨率仍需专项适配。

### 常见故障案例
1. 平板点选偏移：DPR 与 canvas 尺寸不一致。  
2. 大屏 UI 拉伸：布局基准设计不合理。  
3. 触控卡顿：高频手势未节流。

### 思考题
1. 如何做“运行时设备能力探测”替代 UA 判断？  
2. 如何让运营人员可视化配置不同端的 profile？
