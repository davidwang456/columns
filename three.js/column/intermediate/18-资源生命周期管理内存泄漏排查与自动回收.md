# 第18章：资源生命周期管理内存泄漏排查与自动回收

## 元信息
- 学习目标：掌握 three.js 资源生命周期治理，建立“创建-使用-释放”闭环。
- 先修要求：完成分块加载章节。
- 预估时长：5-7 小时。
- 最终产出：可观测、可回收、可审计的资源管理器。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

项目在长时间运行场景下出现典型“慢性崩溃”：前 10 分钟流畅，30 分钟后帧率下降，1 小时后浏览器内存占用明显上涨。业务反馈是“大屏值班久了就卡”。排查后发现并非单点 bug，而是系统性资源泄漏：卸载场景时只移除对象，不释放几何体、材质、纹理和事件订阅。

中级规模项目中，资源管理不再是可选项。因为分块加载、模型切换、后处理开关都会频繁创建 GPU 资源，如果缺少生命周期机制，内存问题必然累积。更现实的是，这类问题很隐蔽，测试短流程不一定复现，上线后才暴露。

本章目标是建立资源治理框架：统一注册资源、统一销毁入口、定期采样指标、自动发现异常增长。把“靠经验排查”升级为“有数据、有规则”的治理体系。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：remove 就够了吗

小胖：  
“对象从 scene 删了，不就释放了吗？”

小白：  
“JS 引用可能还在，GPU 资源也要显式 dispose。”

大师：  
“`remove` 只断开场景树，不等于释放内存。要把资源生命周期看成两条线：CPU 引用和 GPU 句柄。”  

技术映射：  
“两条线” = `GC references + GPU dispose`。

### 第二轮：谁负责回收

小胖：  
“每个模块自己回收就行？”

小白：  
“靠自觉不可靠，建议统一资源注册中心。”

大师：  
“对。创建即注册，卸载即回收，责任集中可追踪。”  

技术映射：  
“集中回收” = `resource registry manager`。

### 第三轮：如何证明没泄漏

小胖：  
“感觉没问题，能不能就算过？”

小白：  
“需要基线指标和长稳测试。”

大师：  
“没有度量就没有治理。至少记录 texture、geometry、program 数量趋势。”  

技术映射：  
“有度量” = `memory telemetry baseline`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
```

### 3.2 分步实现

#### 步骤1：资源注册中心

```ts
const disposableSet = new Set<{ dispose: () => void }>();

export function registerDisposable<T extends { dispose: () => void }>(res: T): T {
  disposableSet.add(res);
  return res;
}

export function disposeAll() {
  disposableSet.forEach((d) => d.dispose());
  disposableSet.clear();
}
```

结果：回收入口统一。  
坑：只注册纹理，不注册材质和几何体。

#### 步骤2：场景对象递归释放

```ts
export function disposeObjectTree(root: THREE.Object3D) {
  root.traverse((obj) => {
    const mesh = obj as THREE.Mesh;
    if (mesh.geometry) mesh.geometry.dispose();
    const mats = Array.isArray(mesh.material) ? mesh.material : [mesh.material];
    mats.filter(Boolean).forEach((m: any) => m.dispose?.());
  });
}
```

结果：批量回收场景资源。  
坑：忽略 render target、postprocess pass 的释放。

#### 步骤3：采集内存与渲染指标

```ts
function snapshot(renderer: THREE.WebGLRenderer) {
  const info = renderer.info;
  console.debug("[mem]", {
    geometries: info.memory.geometries,
    textures: info.memory.textures,
    calls: info.render.calls
  });
}
```

结果：可观察资源变化趋势。  
坑：只看一次快照不能发现泄漏趋势。

## 4. 项目总结（约500-800字）

### 优点
1. 显著降低长稳运行风险。  
2. 排障从猜测变为数据驱动。  
3. 与分块加载天然配套。

### 缺点
1. 需要额外维护生命周期代码。  
2. 第三方库资源释放需额外适配。  
3. 指标采集本身有轻微开销。

### 常见故障案例
1. 只 remove 不 dispose 导致泄漏。  
2. 事件监听未释放造成隐性内存增长。  
3. 离屏渲染目标未回收导致 GPU 占用高。

### 思考题
1. 如何把资源回收接入路由切换钩子？  
2. 如何自动检测“异常增长斜率”并告警？
