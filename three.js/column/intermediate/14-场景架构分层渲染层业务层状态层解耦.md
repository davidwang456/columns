# 第14章：场景架构分层渲染层业务层状态层解耦

## 元信息
- 学习目标：建立中级篇统一架构，把渲染、业务、状态拆分为可独立演进的模块。
- 先修要求：完成基础篇，能独立搭建场景与交互。
- 预估时长：5-7 小时。
- 最终产出：一个具备清晰边界和可测试接口的三层架构骨架。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

项目进入多团队并行开发后，原先“单文件+快速迭代”模式开始失效：渲染代码里混业务判断，业务回调里直接操作 Mesh，状态变更靠事件串来串去。短期看是灵活，长期看是高耦合。一次简单需求（新增告警等级）可能同时改 5 个模块，还容易引入回归。

业务规模扩大后，问题进一步放大：开发抱怨不敢重构，测试抱怨同一功能不同页面行为不一致，运维抱怨线上故障难定位。根因是架构缺少“责任边界”。三维项目本身复杂度就高，如果不分层，技术债会在中级阶段爆发。

本章目标是建立“渲染层(Render) + 业务层(Domain) + 状态层(Store)”三层架构，定义清晰数据流：业务事件进入状态层，状态层发布快照，渲染层仅消费快照并做表现。通过这一步，中级篇后续的性能优化、可观测和权限控制才有稳定落点。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：分层是不是“过度设计”

小胖：  
“功能能跑就行，分这么多层会不会写起来更慢？”

小白：  
“短期可能慢一点，但需求一复杂，不分层会指数级返工。”

大师：  
“分层不是为了漂亮，而是为了可变更。你把它当作城市道路规划，先画主干道，后面扩建才不堵。”  

技术映射：  
“先画主干道” = `architecture for change`。

### 第二轮：状态到底放哪里

小胖：  
“设备状态直接挂 mesh.userData 不就好了？”

小白：  
“那业务状态和渲染状态耦死了，测试也很难 mock。”

大师：  
“业务真值放 Store，渲染对象只是投影。渲染可重建，业务真值不能丢。”  

技术映射：  
“业务真值” = `single source of truth store`。

### 第三轮：如何防止跨层污染

小胖：  
“业务层里偶尔改一下相机，应该也没事吧？”

小白：  
“这就是污染入口。边界一破，后面全破。”

大师：  
“跨层调用必须通过接口，禁止越层写入。框架约束比人治更可靠。”  

技术映射：  
“禁止越层” = `layer contract + dependency rule`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm create vite@latest chapter-14-architecture -- --template vanilla-ts
cd chapter-14-architecture
npm install three
```

### 3.2 分步实现

#### 步骤1：定义三层目录与接口

```ts
// src/domain/types.ts
export type DeviceState = { id: string; level: "normal" | "warning" | "critical" };

// src/store/store.ts
export interface AppState {
  devices: DeviceState[];
  selectedId: string | null;
}
```

结果：领域模型先于实现，边界清晰。  
坑：直接复用渲染对象作为业务类型。

#### 步骤2：实现状态层（只管状态）

```ts
type Listener = (s: AppState) => void;
const listeners = new Set<Listener>();
let state: AppState = { devices: [], selectedId: null };

export const store = {
  getState: () => state,
  patch(next: Partial<AppState>) {
    state = { ...state, ...next };
    listeners.forEach((l) => l(state));
  },
  subscribe(fn: Listener) {
    listeners.add(fn);
    return () => listeners.delete(fn);
  }
};
```

结果：状态变化集中可追踪。  
坑：在订阅里再 patch 导致循环更新。

#### 步骤3：渲染层仅消费快照

```ts
store.subscribe((s) => {
  s.devices.forEach((d) => {
    const mesh = deviceMap.get(d.id);
    if (!mesh) return;
    const mat = mesh.material as THREE.MeshStandardMaterial;
    mat.color.set(d.level === "critical" ? "#dc2626" : d.level === "warning" ? "#f59e0b" : "#22c55e");
  });
});
```

结果：渲染逻辑纯函数化，便于测试。  
坑：渲染层反向修改 store 会破坏单向数据流。

### 3.3 完整代码清单

- `src/domain/*`：业务类型与规则
- `src/store/*`：状态容器与事件
- `src/render/*`：场景与可视投影
- `src/app.ts`：装配入口

### 3.4 测试验证

```bash
npm run build
```

验证：
1. 修改 `store` 可驱动渲染更新；
2. 不允许业务层直接引用 three 实例；
3. 断开渲染层后，状态层仍可单测通过。

## 4. 项目总结（约500-800字）

### 优点
1. 模块边界明确，迭代风险降低。  
2. 测试更容易隔离，回归效率提升。  
3. 为性能与可观测改造提供稳定结构。

### 缺点
1. 初期搭建成本高于“直接写”。  
2. 团队需要纪律执行依赖规则。  
3. 小型 demo 可能显得偏重。

### 常见故障案例
1. 越层调用导致状态错乱。  
2. 订阅未释放造成内存增长。  
3. 类型定义漂移引发联调冲突。

### 思考题
1. 如何在不引入重型框架的前提下增强 store 能力？  
2. 分层后如何做跨层链路追踪？

## 跨部门推广提示
- 开发：先落架构守则，再扩业务功能。  
- 测试：按层写测试（store 单测、render 集成测）。  
- 运维：从架构层面建立故障定位路径。
