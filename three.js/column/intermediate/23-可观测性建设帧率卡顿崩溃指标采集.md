# 第23章：可观测性建设帧率卡顿崩溃指标采集

## 元信息
- 学习目标：从“能跑”升级到“可观测”，建立三维应用在线诊断体系。
- 先修要求：完成性能定位与多端适配章节。
- 预估时长：5-7 小时。
- 最终产出：覆盖性能、稳定性、交互质量的指标采集框架。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

当系统进入生产环境后，最棘手的问题是“线下没问题，线上偶发卡顿或崩溃”。没有观测数据时，团队只能靠用户描述和猜测排查，定位效率极低。三维应用尤其如此：帧率下降可能来自模型、后处理、脚本计算、浏览器差异等多因素，缺少指标就无法判断根因。

业务侧要求通常是明确的：关键大屏不能卡、不能黑、不能频繁异常刷新。要满足这些要求，必须把运行状态数字化：FPS 分布、长帧率、JS 错误、资源加载失败、交互耗时等。中级阶段需要从“开发调试指标”升级到“线上运营指标”。

本章目标是建立“采集-上报-告警-排障”闭环：前端埋点标准化、批量上报防抖、异常采样策略、可视化面板和阈值告警。让运维和开发共享同一事实来源。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：本地看 FPS 够吗

小胖：  
“开发机上看着 60 帧，不就证明性能没问题？”

小白：  
“线上设备复杂，开发机结论不可外推。”

大师：  
“本地指标是调试，线上指标才是运营。两者都要有。”  

技术映射：  
“线上事实” = `production telemetry`。

### 第二轮：采集越多越好？

小胖：  
“那就把所有数据都上报，最保险。”

小白：  
“全量上报会增加网络与存储成本，还可能影响性能。”

大师：  
“可观测不是堆数据，而是采关键指标、做分级采样。”  

技术映射：  
“分级采样” = `sampling + event priority`。

### 第三轮：怎么让告警有用

小胖：  
“报了很多警，最后没人看。”

小白：  
“要定义 SLO 和告警阈值，减少噪音告警。”

大师：  
“告警要可行动，不可行动的告警就是噪音。”  

技术映射：  
“可行动告警” = `actionable alerting`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
```

### 3.2 分步实现

#### 步骤1：定义指标模型

```ts
type MetricEvent =
  | { type: "fps"; value: number; ts: number }
  | { type: "long_frame"; value: number; ts: number }
  | { type: "js_error"; message: string; ts: number }
  | { type: "asset_fail"; url: string; ts: number };
```

结果：采集口径统一。  
坑：不同模块各自定义字段导致数据不可聚合。

#### 步骤2：实现采集与批量上报

```ts
const queue: MetricEvent[] = [];

export function pushMetric(e: MetricEvent) {
  queue.push(e);
  if (queue.length >= 20) flush();
}

function flush() {
  if (!queue.length) return;
  const batch = queue.splice(0, queue.length);
  navigator.sendBeacon?.("/api/metrics", JSON.stringify(batch));
}
```

结果：上报成本可控。  
坑：每条实时上报导致网络风暴。

#### 步骤3：接入帧率与错误监听

```ts
let last = performance.now();
function frameProbe(now: number) {
  const delta = now - last;
  last = now;
  pushMetric({ type: "fps", value: 1000 / delta, ts: now });
  if (delta > 50) pushMetric({ type: "long_frame", value: delta, ts: now });
  requestAnimationFrame(frameProbe);
}
requestAnimationFrame(frameProbe);

window.addEventListener("error", (e) => {
  pushMetric({ type: "js_error", message: e.message, ts: Date.now() });
});
```

结果：性能与稳定性可持续追踪。  
坑：采样频率太高反而影响性能。

## 4. 项目总结（约500-800字）

### 优点
1. 线上问题可量化、可定位。  
2. 开发与运维协作效率提升。  
3. 为 SRE 体系建立数据基础。

### 缺点
1. 采集体系建设成本较高。  
2. 指标口径需要持续治理。  
3. 告警阈值需按场景动态调整。

### 常见故障案例
1. 指标缺失：关键路径未埋点。  
2. 告警泛滥：阈值配置过敏。  
3. 上报阻塞：同步请求影响主线程。

### 思考题
1. 如何将指标与用户会话关联，提升排障复现率？  
2. 如何为“卡顿”定义更贴近业务体验的 SLO？
