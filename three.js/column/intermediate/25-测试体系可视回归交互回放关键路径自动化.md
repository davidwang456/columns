# 第25章：测试体系可视回归交互回放关键路径自动化

## 元信息
- 学习目标：搭建适配三维系统的自动化测试体系，覆盖视觉、交互、关键流程回放。
- 先修要求：完成可观测与权限章节。
- 预估时长：5-7 小时。
- 最终产出：可在 CI 中稳定运行的三维回归测试方案。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

三维项目最痛的不是“不会写功能”，而是“改一个功能，别处悄悄坏了”。普通单元测试难覆盖视觉变化，人工点点点成本又高，导致回归依赖经验。随着中级阶段功能增多，回归风险直线上升：材质改了影响告警识别、相机改了导致点选偏移、优化改了导致帧率掉线。

很多团队把三维测试简化为“能跑起来就算过”，这是典型隐患。真正可交付的系统需要三层测试：可视回归（截图差异）、交互回放（点击/拖拽流程）、关键路径自动化（从数据到展示全链路）。没有这套体系，发布质量无法稳定。

本章目标是建立中级篇质量基线：定义关键场景、固定测试相机、统一截图阈值、录制交互脚本、接入 CI 门禁，让“视觉变化可发现、交互回归可复现、质量风险可阻断”。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：人工回归够不够

小胖：  
“上线前我手动点一遍，不就能发现问题？”

小白：  
“功能一多就漏测，而且人工结果不可重复。”

大师：  
“人工回归是补充，不是主力。关键路径必须自动化。”  

技术映射：  
“关键路径自动化” = `automated regression gates`。

### 第二轮：三维怎么做快照

小胖：  
“三维画面每次都可能有微差，截图对比靠谱吗？”

小白：  
“需要固定相机、固定时间点、固定随机种子。”

大师：  
“可视回归的核心是控制变量，不是追求像素绝对一致。”  

技术映射：  
“控制变量” = `deterministic rendering test`。

### 第三轮：交互回放值不值

小胖：  
“录制交互脚本很麻烦。”

小白：  
“但关键链路（点选、联动、巡航）最容易回归，最值得自动化。”

大师：  
“高价值路径优先自动化，ROI 最高。”  

技术映射：  
“高价值优先” = `risk-based test prioritization`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install -D playwright
```

### 3.2 分步实现

#### 步骤1：定义可视回归基线

```ts
// tests/visual.spec.ts (示意)
import { test, expect } from "@playwright/test";

test("dashboard baseline", async ({ page }) => {
  await page.goto("http://localhost:5173");
  await page.waitForTimeout(1000); // 等场景稳定
  expect(await page.screenshot()).toMatchSnapshot("dashboard.png");
});
```

结果：视觉变化可被自动检测。  
坑：未固定相机和动画时间，快照波动大。

#### 步骤2：录制关键交互回放

```ts
test("pick and panel linkage", async ({ page }) => {
  await page.goto("http://localhost:5173");
  await page.mouse.click(820, 420); // 点选设备
  await expect(page.locator("[data-testid=device-panel]")).toBeVisible();
});
```

结果：交互链路可重复验证。  
坑：坐标依赖分辨率，需统一测试视口。

#### 步骤3：CI 门禁策略

```json
{
  "scripts": {
    "test:visual": "playwright test tests/visual.spec.ts",
    "test:e2e": "playwright test tests/e2e.spec.ts"
  }
}
```

结果：PR 阶段自动阻断高风险回归。  
坑：无分级策略会导致轻微差异频繁阻塞。

### 3.3 完整代码清单

- `tests/visual.spec.ts`
- `tests/e2e.spec.ts`
- `playwright.config.ts`
- `scripts/test-ci.sh`

### 3.4 测试验证

```bash
npm run test:visual
npm run test:e2e
```

验证：
1. 视觉快照异常可被识别；  
2. 核心交互链路自动通过；  
3. CI 可输出失败截图与日志。

## 4. 项目总结（约500-800字）

### 优点
1. 发布质量可量化、可门禁。  
2. 回归效率显著提升。  
3. 问题定位更快（失败截图+步骤）。

### 缺点
1. 初期建设投入较大。  
2. 三维快照稳定性需要持续调优。  
3. 用例维护成本随功能增长上升。

### 常见故障案例
1. 快照抖动：测试环境不确定因素未控制。  
2. 回放失败：视口和渲染比例不一致。  
3. CI 超时：场景初始化等待策略不合理。

### 思考题
1. 如何将性能断言（FPS、长帧率）纳入自动化测试？  
2. 如何设计“分级门禁”（阻断/告警）策略？

## 跨部门推广提示
- 开发：新功能必须补关键路径用例。  
- 测试：维护快照基线和回放脚本库。  
- 运维：把测试结果与发布审批联动。
