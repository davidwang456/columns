# 第20章：WebGL性能调优CPUGPU瓶颈定位方法论

## 元信息
- 学习目标：建立 WebGL 性能排障路径，能快速判断 CPU 瓶颈还是 GPU 瓶颈。
- 先修要求：完成中级篇 14-19 章。
- 预估时长：5-7 小时。
- 最终产出：一套可执行的性能诊断清单和优化闭环。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

当业务方反馈“场景卡”，团队最容易陷入无效优化：有人改材质，有人改动画，有人删特效，最后效果有限。原因是没有先定位瓶颈。WebGL 性能问题并不神秘，但必须区分 CPU 与 GPU：CPU 常见于脚本计算和提交开销，GPU 常见于像素填充和着色复杂度。

中级阶段的难点不在“知道一些优化技巧”，而在“建立稳定方法论”。如果没有诊断路径，优化就是试错；如果没有基线数据，优化成果无法证明；如果没有回归机制，下一次需求迭代会把优化成果冲掉。本章就是要解决这三个问题。

我们将提供一条落地路径：先测指标，再做分类定位，再做小步实验，再做回归固化。目标是让性能优化从“个人经验”升级为“团队流程”。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：卡顿先改哪里

小胖：  
“卡了先把阴影关掉，通常有用。”

小白：  
“可能有用，但不一定对因。先定位再改更稳。”

大师：  
“优化第一原则：先测量，后行动。盲改容易引入新问题。”  

技术映射：  
“先测量” = `measure-before-optimize`。

### 第二轮：怎么区分 CPU/GPU

小胖：  
“肉眼看都卡，怎么判断是谁慢？”

小白：  
“看主线程时间、draw calls、shader 成本、分辨率敏感性。”

大师：  
“如果降分辨率帧率明显提升，多半是 GPU；如果不敏感，多半是 CPU 或提交瓶颈。”  

技术映射：  
“降分辨率试验” = `resolution sensitivity test`。

### 第三轮：优化怎么闭环

小胖：  
“我优化完感觉快了，就算完工？”

小白：  
“要有前后对比数据和回归脚本。”

大师：  
“没有基线的优化不可复用。把结果写进预算和门禁。”  

技术映射：  
“可复用优化” = `baseline + regression gate`。

### 3.2 分步实现

#### 步骤1：建立性能采样

```ts
function collectFrameMetrics(renderer: THREE.WebGLRenderer) {
  const info = renderer.info;
  return {
    calls: info.render.calls,
    triangles: info.render.triangles,
    textures: info.memory.textures
  };
}
```

结果：具备基础观察能力。  
坑：只看平均值，忽略抖动峰值。

#### 步骤2：执行定位实验

```ts
function toggleLowResolution(on: boolean) {
  const scale = on ? 0.5 : 1;
  renderer.setPixelRatio(window.devicePixelRatio * scale);
}
```

结果：快速判断 GPU 敏感度。  
坑：切分辨率后未重算后处理尺寸。

#### 步骤3：按瓶颈应用优化

```ts
function applyCpuOptimizations() {
  // 降低每帧 JS 计算、减少对象创建、分帧更新
}

function applyGpuOptimizations() {
  // 降低后处理强度、减少像素填充、压缩纹理

```

结果：优化动作有针对性。

## 4. 项目总结（约500-800字）

### 优点
1. 性能优化有路径、可复现。  
2. 避免盲目试错，节省迭代成本。  
3. 便于团队协同与知识沉淀。

### 缺点
1. 需要持续维护指标体系。  
2. 诊断过程有一定学习成本。  
3. 不同硬件差异需额外校准。

### 常见故障案例
1. 优化后反而更慢：误判瓶颈类型。  
2. 数据不稳定：采样窗口过短。  
3. 回归反复：没有性能门禁。

### 思考题
1. 如何把性能预算纳入 CI？  
2. 如何设计“性能告警分级”机制？

## 跨部门推广提示
- 开发：每次优化必须附前后指标对比。  
- 测试：建立性能回归场景集。  
- 运维：接入线上端性能监控与告警阈值。
