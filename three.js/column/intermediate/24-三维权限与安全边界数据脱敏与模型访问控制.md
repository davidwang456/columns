# 第24章：三维权限与安全边界数据脱敏与模型访问控制

## 元信息
- 学习目标：为三维系统建立“看什么、点什么、导出什么”的权限边界与数据脱敏策略。
- 先修要求：完成实时同步与可观测章节。
- 预估时长：5-7 小时。
- 最终产出：具备 RBAC 权限校验、字段脱敏、模型访问控制的安全中间层。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

随着系统从内部试用走向多角色协作，权限问题变得敏感：运维可看全部设备详情，外包人员只能看公开区域，领导可看汇总但不看敏感字段。如果前端只靠“隐藏按钮”做权限，风险极高：接口数据仍可能在网络响应中暴露，模型资源 URL 也可能被直接访问。

三维系统的安全边界比传统后台更复杂，因为权限不仅体现在 API，还体现在空间对象：某些楼层不可见、某些设备不可点、某些字段要脱敏展示。若无统一策略，会出现“UI 看不见但 API 还能拿到”“对象隐藏了但模型仍可下载”等漏洞。

本章目标是建立端到端权限闭环：后端令牌声明能力范围，前端基于策略渲染与交互，数据层做字段脱敏，资源层做签名访问。最终实现“最小权限原则”。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：前端隐藏够安全吗

小胖：  
“按钮藏起来了，别人就点不到，应该安全吧？”

小白：  
“前端隐藏只是体验层控制，不是安全控制。数据如果已下发就可能泄漏。”

大师：  
“权限必须后端兜底，前端负责执行策略和减少误用。”  

技术映射：  
“后端兜底” = `server-side authorization`。

### 第二轮：三维对象怎么控权

小胖：  
“权限不是接口层问题吗？跟场景对象有什么关系？”

小白：  
“对象可见性、可交互性、字段展示都属于权限范围。”

大师：  
“三维权限是 API 权限 + 场景权限 + 资源权限三层叠加。”  

技术映射：  
“三层叠加” = `api + scene + asset permissions`。

### 第三轮：脱敏放哪层

小胖：  
“前端直接把手机号中间打星就行。”

小白：  
“前端脱敏只能防误展示，不能防数据泄露，服务端也要脱敏。”

大师：  
“敏感字段应默认不下发，前端脱敏是最后一道显示防线。”  

技术映射：  
“双层脱敏” = `backend minimization + frontend masking`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
```

### 3.2 分步实现

#### 步骤1：定义权限模型

```ts
type Permission = {
  canViewZones: string[];
  canInspectDevice: boolean;
  canExportData: boolean;
  sensitiveLevel: "full" | "masked";
};
```

结果：权限语义可读可测。  
坑：权限项粒度过粗导致误放权。

#### 步骤2：场景级访问控制

```ts
function applyScenePermission(root: THREE.Object3D, perm: Permission) {
  root.traverse((obj) => {
    const zone = obj.userData.zoneId as string | undefined;
    if (zone && !perm.canViewZones.includes(zone)) {
      obj.visible = false;
    }
    if (obj.userData.pickable && !perm.canInspectDevice) {
      obj.userData.pickable = false;
    }
  });
}
```

结果：不可见区域无法显示，不可点对象无法交互。  
坑：仅隐藏顶层对象，子节点仍可能被拾取。

#### 步骤3：字段脱敏与资源签名

```ts
function maskPhone(phone: string) {
  return phone.replace(/^(\d{3})\d{4}(\d{4})$/, "$1****$2");
}

function secureDeviceInfo(raw: any, perm: Permission) {
  if (perm.sensitiveLevel === "masked") {
    return { ...raw, ownerPhone: maskPhone(raw.ownerPhone) };
  }
  return raw;
}
```

结果：敏感信息按权限展示。  
坑：只做前端脱敏，网络层仍有明文字段。

## 4. 项目总结（约500-800字）

### 优点
1. 明确安全边界，降低数据泄漏风险。  
2. 权限策略可复用、可审计。  
3. 与业务角色模型天然对齐。

### 缺点
1. 实现与测试复杂度提高。  
2. 需要后端、前端、运维协同一致。  
3. 权限矩阵过细会增加配置负担。

### 常见故障案例
1. 前端隐藏不等于后端授权。  
2. 场景隐藏后仍可通过接口拿到详情。  
3. 资源 URL 无签名导致模型泄露。

### 思考题
1. 如何实现“动态权限切换”且不中断场景状态？  
2. 如何对权限策略变更做灰度发布？
