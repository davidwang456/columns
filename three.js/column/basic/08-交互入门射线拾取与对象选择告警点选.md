# 第08章：交互入门射线拾取与对象选择告警点选

## 元信息
- 学习目标：掌握 Raycaster 拾取流程，完成“点击设备 -> 展示详情 -> 高亮反馈”闭环。
- 先修要求：完成前面章节的对象命名与状态管理规范。
- 预估时长：4-6 小时。
- 最终产出：可扩展的场景交互模块。
- 适用角色：开发、测试。

## 1. 项目背景（约500字）

看板上线试运行后，值班同学提出关键诉求：不能只“看”，要能“点”。他们希望点击告警设备后直接看到编号、责任人、最近工单和处理建议。但当前页面只有自动轮播，没有交互入口，导致信息链路断裂：看见问题却无法深入问题。

很多团队第一次做 3D 交互时，会把点击当成普通 DOM 事件处理，忽略了“屏幕坐标 -> 三维空间”的映射问题。结果常见 bug 包括：点不中、点偏移、点到遮挡物、缩放后命中错位。随着对象增多，交互性能也会下降。用户体验上就会出现“点了没反应”“点一次选中两个”等挫败感。

本章将交付一个标准交互链路：鼠标坐标归一化、射线检测、命中过滤、对象高亮、面板联动。目标不是写一个 demo，而是建立可用于后续告警联动和巡检流程的交互基础设施。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：为什么点不到

小胖：  
“我明明点在设备上，结果选中旁边地板。”

小白：  
“是不是坐标没归一化？或者 canvas 不是全屏导致偏移？”

大师：  
“对。拾取第一步永远是坐标校准。不要假设 `window.innerWidth` 就等于渲染区域。”  

技术映射：  
“坐标校准” = `NDC(normalized device coordinates)`。

### 第二轮：命中对象太多怎么办

小胖：  
“我点一次返回一堆对象，哪个才是真的设备？”

小白：  
“是不是需要给可交互对象打标签，只在白名单里选？”

大师：  
“非常重要。把‘可见对象’和‘可交互对象’分离。渲染是全量，拾取是白名单。”  

技术映射：  
“白名单拾取” = `pickableObjects[]`。

### 第三轮：高亮如何不污染原状态

小胖：  
“点中后我改了颜色，取消选择后颜色回不去了。”

小白：  
“要不要缓存原始材质状态，再做临时高亮层？”

大师：  
“对。高亮是临时态，不应该破坏业务态。两层状态分离。”  

技术映射：  
“临时态分离” = `selection overlay state`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
```

### 3.2 分步实现

#### 步骤1：实现坐标归一化

```ts
function toNdc(ev: MouseEvent, canvas: HTMLCanvasElement) {
  const rect = canvas.getBoundingClientRect();
  return {
    x: ((ev.clientX - rect.left) / rect.width) * 2 - 1,
    y: -((ev.clientY - rect.top) / rect.height) * 2 + 1
  };
}
```

结果：不同分辨率命中一致。  
坑：直接用 `window` 尺寸会偏移。

#### 步骤2：射线拾取与白名单过滤

```ts
const raycaster = new THREE.Raycaster();
const pointer = new THREE.Vector2();
const pickables: THREE.Object3D[] = [];

canvas.addEventListener("click", (ev) => {
  const p = toNdc(ev, canvas);
  pointer.set(p.x, p.y);
  raycaster.setFromCamera(pointer, camera);
  const hits = raycaster.intersectObjects(pickables, true);
  if (!hits.length) return;
  const target = hits[0].object;
  onSelect(target);
});
```

结果：点选稳定，只选可交互对象。  
坑：`intersectObjects` 第二个参数未设 `true` 时子节点无法命中。

#### 步骤3：高亮与详情联动

```ts
let selected: THREE.Mesh | null = null;
let selectedOriginalColor: string | null = null;

function onSelect(obj: THREE.Object3D) {
  const mesh = obj as THREE.Mesh;
  if (selected && selectedOriginalColor) {
    (selected.material as THREE.MeshStandardMaterial).color.set(selectedOriginalColor);
  }
  selected = mesh;
  const mat = mesh.material as THREE.MeshStandardMaterial;
  selectedOriginalColor = `#${mat.color.getHexString()}`;
  mat.color.set("#fde047");
  showPanel(mesh.userData); // 同步业务详情面板
}
```

结果：选中反馈明显，取消后颜色恢复。  
坑：共享材质对象时，高亮会影响其他设备。

## 4. 项目总结（约500-800字）

### 优点
1. 形成业务闭环，从“看见”升级为“可操作”。  
2. 交互链路可复用到告警、工单、巡检。  
3. 可测试性强，命中规则可自动化验证。

### 缺点
1. 对对象命名和数据绑定依赖高。  
2. 对性能敏感，复杂场景需做空间索引优化。  
3. 高亮策略需要与材质体系统一。

### 常见故障案例
1. 命中偏移：NDC 转换未基于 canvas rect。  
2. 点选穿透：遮挡优先级处理不当。  
3. 高亮串色：共享材质未 clone。

### 思考题
1. 如何支持“框选多个设备”并批量操作？  
2. 如何在移动端做点选容错（扩大点击热区）？
