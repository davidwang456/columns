# 第05章：光照系统与阴影让产线场景可信而非玩具

## 元信息
- 学习目标：理解环境光、平行光、点光源、阴影参数，并形成生产可用的光照配置基线。
- 先修要求：完成材质章节，理解 PBR 参数。
- 预估时长：4-6 小时。
- 最终产出：一个具备方向感、层次感、可读性的产线光照方案。
- 适用角色：开发、测试。

## 1. 项目背景（约500字）

项目进入业务演示阶段后，出现“看起来像游戏玩具”的评价。明明模型和数据都在，场景却缺乏真实空间感。管理层提出两个关键诉求：第一，画面要一眼看出设备高低、前后关系；第二，告警区域需要被自然地“突出”，而不是突兀叠加 UI。团队复盘后发现核心问题不是模型精度，而是光照和阴影设计缺失。

没有合理光照时，用户只能靠轮廓猜位置；没有阴影时，设备像“漂浮在空中”；阴影配置错误时，又会引入锯齿、闪烁、性能暴涨。很多项目会走极端：要么全开阴影导致帧率骤降，要么完全关闭阴影导致场景塑料感严重。本章要解决的是“真实感与性能平衡”。

我们将采用“主光 + 辅光 + 受控阴影”的工程策略：平行光负责方向感，环境光负责底色，关键对象启用阴影，非关键对象关闭阴影，配合阴影贴图分辨率和裁剪范围优化。最终目标是：画面可信、可控、可维护。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：光越多越好？

小胖：  
“我试了下加 5 个灯，画面亮堂多了，是不是灯越多越高级？”

小白：  
“灯多不等于好。多光源会让阴影复杂度飙升，而且方向信息可能互相抵消。”

大师：  
“正确思路是‘主次分明’，先确定主光方向，再用环境光补底。像拍照一样，先有主灯，再谈补光。”  

技术映射：  
“主次分明” = `key light + fill light`。

### 第二轮：阴影为什么糊

小胖：  
“开了阴影但边缘像马赛克，还闪来闪去。”

小白：  
“可能是 shadow map 分辨率太低，或者相机 frustum 太大？”

大师：  
“对。阴影相机覆盖范围越大，单位像素精度越低。不要贪大，按场景关键区域精确裁剪。”  

技术映射：  
“阴影精度” = `shadow.camera bounds + mapSize`。

### 第三轮：性能怎么兜底

小胖：  
“要真实又要流畅，会不会两头都做不到？”

小白：  
“是不是可以只给关键设备投射阴影，其他对象接收阴影但不投射？”

大师：  
“这就是工程实践。阴影不是全开全关，而是按业务优先级分层开启。”  

技术映射：  
“按需阴影” = `castShadow / receiveShadow 分级控制`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
```

### 3.2 分步实现

#### 步骤1：启用渲染器阴影

```ts
renderer.shadowMap.enabled = true;
renderer.shadowMap.type = THREE.PCFSoftShadowMap;
```

结果：全局阴影能力打开。  
坑：忘记开启 `shadowMap.enabled` 导致所有阴影无效。

#### 步骤2：配置主光与环境光

```ts
const ambientLight = new THREE.AmbientLight("#ffffff", 0.35);
scene.add(ambientLight);

const keyLight = new THREE.DirectionalLight("#ffffff", 1.2);
keyLight.position.set(8, 12, 6);
keyLight.castShadow = true;
keyLight.shadow.mapSize.set(2048, 2048);
keyLight.shadow.camera.left = -20;
keyLight.shadow.camera.right = 20;
keyLight.shadow.camera.top = 20;
keyLight.shadow.camera.bottom = -20;
scene.add(keyLight);
```

结果：场景有方向感，阴影初步可见。  
坑：阴影范围过大导致糊；范围过小导致裁切。

#### 步骤3：分级配置对象阴影

```ts
const floor = new THREE.Mesh(
  new THREE.PlaneGeometry(60, 40),
  new THREE.MeshStandardMaterial({ color: "#0f172a" })
);
floor.rotation.x = -Math.PI / 2;
floor.receiveShadow = true;
scene.add(floor);

criticalDevices.forEach((mesh) => {
  mesh.castShadow = true;
  mesh.receiveShadow = true;
});

normalDevices.forEach((mesh) => {
  mesh.castShadow = false;
  mesh.receiveShadow = true;
});
```

结果：关键设备阴影明显，性能可控。  
坑：全部对象开投射阴影会明显降帧。

```bash
npm run build
```

验收：
1. 主方向光下设备层次清晰；  
2. 告警区域阴影稳定无大面积闪烁；  
3. 关闭阴影与开启阴影帧率差异在可接受范围内。

## 4. 项目总结（约500-800字）

### 优点
1. 显著提升空间可信度。  
2. 有利于识别关键设备与告警区域。  
3. 可通过分级策略平衡性能。

### 缺点
1. 参数较多，调试成本高。  
2. 复杂场景容易出现阴影伪影。  
3. 不同终端表现存在差异。

### 常见故障案例
1. 阴影锯齿严重：`mapSize` 过低。  
2. 阴影抖动：阴影相机范围不稳定。  
3. 帧率突降：过多对象开启 `castShadow`。

### 思考题
1. 在固定预算下，优先提升阴影精度还是覆盖范围？为什么？  
2. 如果场景有昼夜切换，光照策略应如何演进？
