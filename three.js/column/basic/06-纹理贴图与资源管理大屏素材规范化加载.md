# 第06章：纹理贴图与资源管理大屏素材规范化加载

## 元信息
- 学习目标：掌握纹理加载、缓存、销毁与失败兜底策略，避免首屏慢和内存泄漏。
- 先修要求：完成材质与光照章节。
- 预估时长：4-6 小时。
- 最终产出：一套可复用的纹理资源管理模块。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

项目接入真实贴图后，问题集中爆发：首屏从 2 秒涨到 9 秒、切换场景后内存不回收、弱网环境出现“灰模”闪烁。业务侧最直观的反馈是“演示时老卡壳”。技术侧根因并不复杂：贴图没有分级加载、没有缓存策略、没有失败兜底。

在大屏项目里，纹理通常是体量最大的资源。若把所有素材一次性加载，不仅首屏慢，还会触发浏览器内存压力；若每次切页面都重复加载，又会浪费带宽并引发卡顿。更糟糕的是，很多团队只会“加载”，不会“释放”，导致运行越久越慢。

本章目标是把贴图管理工程化：建立资源清单、按优先级加载、统一缓存复用、离场时显式销毁，并加上加载进度与错误兜底。你会得到一个“可上线”的纹理资源体系，而不是零散示例。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：一次全加载最省事？

小胖：  
“我把贴图全 preload 了，省得后面卡。”

小白：  
“全 preload 会拖慢首屏，用户还没看到页面就先等资源。是不是应该分首屏必需和延迟加载？”

大师：  
“没错。首屏只加载关键纹理，次要资源在空闲时加载。先开门营业，再慢慢补货。”  

技术映射：  
“先营业再补货” = `critical path loading + lazy loading`。

### 第二轮：缓存要不要做

小胖：  
“一个 logo 在十个模型里用，重复加载也能跑啊。”

小白：  
“能跑不代表合理。重复请求会拖网络，重复创建会占内存。”

大师：  
“统一用 key 做纹理缓存，同 URL 只加载一次。资源管理器是基础设施，不是锦上添花。”  

技术映射：  
“同 URL 一次加载” = `texture cache map`。

### 第三轮：销毁什么时候做

小胖：  
“我页面切走了，浏览器会帮我回收吧？”

小白：  
“WebGL 资源很多是显式生命周期，需要主动 `dispose`。”

大师：  
“对。离场时要释放纹理、材质、几何体。否则就是慢性内存泄漏。”  

技术映射：  
“主动释放” = `dispose lifecycle`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
```

### 3.2 分步实现

#### 步骤1：实现纹理管理器

```ts
import * as THREE from "three";

const loader = new THREE.TextureLoader();
const cache = new Map<string, THREE.Texture>();

export async function loadTexture(url: string) {
  if (cache.has(url)) return cache.get(url)!;
  const texture = await loader.loadAsync(url);
  cache.set(url, texture);
  return texture;
}
```

结果：同资源复用，减少重复请求。  
坑：未做异常处理会导致整体流程中断。

#### 步骤2：实现分级加载和兜底

```ts
const fallback = new THREE.DataTexture(new Uint8Array([200, 200, 200, 255]), 1, 1);
fallback.needsUpdate = true;

export async function safeLoadTexture(url: string) {
  try {
    return await loadTexture(url);
  } catch (e) {
    console.warn("texture load failed:", url, e);
    return fallback;
  }
}
```

结果：弱网或缺失文件时场景不崩。  
坑：兜底纹理尺寸过大反而浪费内存。

#### 步骤3：离场销毁资源

```ts
export function disposeAllTextures() {
  cache.forEach((t) => t.dispose());
  cache.clear();
}
```

结果：场景切换后内存回落。  
坑：只释放纹理不释放材质，仍会残留引用。

## 4. 项目总结（约500-800字）

### 优点
1. 首屏加载更可控。  
2. 缓存复用降低网络和内存压力。  
3. 错误兜底提升稳定性。

### 缺点
1. 需要维护资源清单与生命周期。  
2. 过度缓存可能引发内存占用。  
3. 跨域与 CDN 配置需额外协调。

### 常见故障案例
1. 首屏白屏：关键纹理阻塞渲染。  
2. 内存持续上涨：未执行 `dispose`。  
3. 部分终端贴图发黑：色彩空间配置不一致。

### 思考题
1. 如何设计“纹理热更新”而不刷新整页？  
2. 大屏离线场景下，资源预打包策略如何设计？

## 跨部门推广提示
- 开发：统一接入资源管理器，不要直接散调 `TextureLoader`。  
- 测试：增加弱网、丢包、404 贴图回归。  
- 运维：配置 CDN 缓存策略与静态资源版本化。
