# 第13章：基础综合项目单机场景版智慧工厂驾驶舱

## 元信息
- 学习目标：综合应用基础篇能力，完成一个可演示、可验证、可扩展的单机版驾驶舱。
- 先修要求：完成基础篇 01-12 章。
- 预估时长：6-10 小时。
- 最终产出：智慧工厂驾驶舱 MVP（场景、交互、联动、导览、调试）。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

经过前 12 章，团队已掌握单点能力：场景、材质、光照、交互、联动、调试。业务方希望看到“能交付”的整体成果：一个单机运行的智慧工厂驾驶舱，至少包含厂区总览、设备状态、告警点选、图表联动、巡检导览。目标不是全功能平台，而是可供评审、培训、立项的 MVP。

历史项目常见失败点是“章节都懂，工程不会拼”。每一块能力都能跑，合在一起就冲突：状态重复管理、事件循环触发、资源释放遗漏、性能快速下滑。为了避免“拼装灾难”，综合项目必须先定架构边界，再逐步集成模块并做回归。

本章采用“骨架先行，能力逐步挂载”的策略：先搭核心容器（scene/store/bus），再依次接入模型、材质状态、交互、2D 联动、巡航导览、调试面板，最后做验收脚本。你将得到一个真实可演示的基础版产品。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：先拼功能还是先搭架构

小胖：  
“把前面代码复制进来，能跑就是胜利。”

小白：  
“复制拼接很快会失控。要先明确模块边界和依赖方向。”

大师：  
“先搭骨架：`core` 管生命周期，`scene` 管渲染，`feature` 管业务。先定规矩再上功能。”  

技术映射：  
“先定规矩” = `modular architecture skeleton`。

### 第二轮：如何保证集成稳定

小胖：  
“模块一个个加，出了问题再修不行吗？”

小白：  
“要有集成顺序和验收清单，不然难定位回归。”

大师：  
“每加一个能力就做一次小验收。像搭积木，每层都要稳。”  

技术映射：  
“小步集成” = `incremental integration + checkpoint`。

### 第三轮：MVP 如何定义完成

小胖：  
“只要画面酷就算完成吧？”

小白：  
“还要可操作、可验证、可维护。至少要有测试脚本和故障清单。”

大师：  
“MVP 不是 demo，而是‘可交付最小系统’。”  

技术映射：  
“可交付” = `feature complete + testable + maintainable`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm create vite@latest chapter-13-cockpit -- --template vanilla-ts
cd chapter-13-cockpit
npm install three stats.js
npm run dev
```

### 3.2 分步实现

#### 步骤1：搭建工程骨架

```ts
// src/core/bootstrap.ts
export function bootstrap() {
  initStore();
  initBus();
  initScene();
  mountFeatures();
}
```

结果：主流程清晰，便于扩展。  
坑：初始化顺序错误会导致空引用。

#### 步骤2：接入核心能力模块

```ts
mountSceneObjects();      // 几何体/模型
mountMaterialStates();    // 状态材质
mountInteraction();       // 点选拾取
mountUiLinkage();         // 2D/3D 联动
mountTourSystem();        // 相机巡航
mountDebugTools();        // 调试和性能面板
```

结果：基础能力全部整合。  
坑：模块间直接互调引发循环依赖。

#### 步骤3：集成验收脚本

```ts
function runSmokeChecks() {
  assertSceneReady();
  assertPickWorks();
  assertUiSync();
  assertTourWorks();
}

runSmokeChecks();
```

结果：每次启动都有最小健康检查。  
坑：缺少自动检查导致回归问题滞后发现。

### 3.3 完整代码清单

- `src/core/*`：生命周期与依赖注入
- `src/scene/*`：场景与渲染
- `src/feature/*`：交互、联动、导览
- `src/debug/*`：调试与性能

### 3.4 测试验证

```bash
npm run build
```

手工验收：
1. 页面加载后可见厂区场景；  
2. 点击设备有高亮与详情面板；  
3. 图表点击反向定位对象；  
4. 自动巡航可启停并支持人工接管；  
5. 调试面板在开发可见、生产关闭。

## 4. 项目总结（约500-800字）

### 优点
1. 基础能力形成闭环，具备演示与立项价值。  
2. 模块边界清晰，便于后续升级中级篇。  
3. 测试与调试体系初步完整。

### 缺点
1. 仍是单机场景，未覆盖分布式数据与高并发。  
2. 复杂特效和性能极限未深入。  
3. 资产流程与权限体系仍待强化。

### 常见故障案例
1. 集成后点选失效：事件总线重复绑定。  
2. 长时间运行卡顿：资源未释放。  
3. 巡航中断异常：状态机分支不完整。

### 思考题
1. 如何把本章单机项目平滑升级到多场景联动架构？  
2. 如何为驾驶舱增加基础可观测性（FPS、错误率、交互耗时）？
