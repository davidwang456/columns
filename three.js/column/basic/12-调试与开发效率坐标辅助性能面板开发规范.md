# 第12章：调试与开发效率坐标辅助性能面板开发规范

## 元信息
- 学习目标：建立统一调试工具链，提升排障效率并减少“玄学问题”。
- 先修要求：完成前置基础篇章节。
- 预估时长：4-6 小时。
- 最终产出：调试开关、性能面板、日志规范三件套。
- 适用角色：开发、测试、运维。

## 1. 项目背景（约500字）

随着功能增多，团队进入“问题很多但难复现”的阶段：某些设备偶发看不见、某些交互偶发点不中、某些终端偶发掉帧。开发说本地正常，测试说线上不稳，运维说指标不足。问题并不神秘，只是缺少可观测调试体系：没有坐标辅助、没有性能面板、没有统一日志标签。

3D 项目相比普通前端更依赖可视化调试。没有 `AxesHelper` 你很难判断对象方向；没有 `GridHelper` 很难判断高度漂移；没有 FPS 面板就无法确认卡顿来源。很多团队把这些工具当“开发阶段临时脚本”，导致上线前无法系统排查。

本章目标是搭建可开关的调试基础设施：一键打开坐标辅助、性能统计、事件追踪；发布构建自动关闭调试功能；问题复现时可以快速定位是“坐标错误、资源问题、交互问题还是性能瓶颈”。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：调试工具会污染生产吗

小胖：  
“加这么多 helper，不会把线上画面搞花吗？”

小白：  
“需要环境开关，开发开启、生产关闭。”

大师：  
“对。调试能力要内建，但默认关。就像消防通道，平时不用，关键时刻必须有。”  

技术映射：  
“可控调试” = `debug feature flag`。

### 第二轮：掉帧怎么定位

小胖：  
“卡了就是卡了，感觉很难查。”

小白：  
“至少要区分 CPU 瓶颈、GPU 压力、资源阻塞。”

大师：  
“先看基础指标：FPS、draw calls、triangles、纹理数量，再决定优化方向。”  

技术映射：  
“先测再调” = `measure-first optimization`。

### 第三轮：团队协作如何统一

小胖：  
“每个人日志格式都不一样，看着头大。”

小白：  
“应统一日志前缀和事件字段，便于筛选检索。”

大师：  
“是的，调试规范本质是团队协议，不是个人习惯。”  

技术映射：  
“统一协议” = `structured debug logging`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
npm install stats.js
```

### 3.2 分步实现

#### 步骤1：添加调试开关

```ts
const DEBUG = import.meta.env.DEV;

if (DEBUG) {
  scene.add(new THREE.AxesHelper(5));
  scene.add(new THREE.GridHelper(50, 50));
}
```

结果：开发环境可见辅助线。  
坑：忘记环境判断导致线上泄露调试元素。

#### 步骤2：接入性能面板

```ts
import Stats from "stats.js";

let stats: Stats | null = null;
if (DEBUG) {
  stats = new Stats();
  document.body.appendChild(stats.dom);
}

function loop() {
  stats?.begin();
  renderer.render(scene, camera);
  stats?.end();
  requestAnimationFrame(loop);
}
```

结果：实时观察 FPS。  
坑：多次初始化 `Stats` 造成重复面板。

#### 步骤3：统一调试日志

```ts
function logDebug(scope: string, msg: string, payload?: Record<string, unknown>) {
  if (!DEBUG) return;
  console.debug(`[3D:${scope}] ${msg}`, payload ?? {});
}

logDebug("pick", "device selected", { id: "device-101" });
```

结果：日志可按模块快速检索。  
坑：日志过多影响性能，需按级别控制。

### 3.3 完整代码清单

```bash
npm run build
```

验证：
1. 开发环境显示调试面板；  
2. 生产构建不包含调试 UI；  
3. 日志可按模块筛选定位问题。

## 4. 项目总结（约500-800字）

### 优点
1. 排障效率显著提升。  
2. 团队协作更规范。  
3. 为中级篇可观测性打基础。

### 缺点
1. 需要维护调试工具代码。  
2. 面板和日志可能引入轻微开销。  
3. 规范落地依赖团队纪律。

### 常见故障案例
1. 线上出现辅助线：环境变量配置错误。  
2. 误判性能瓶颈：只看 FPS 不看 draw calls。  
3. 日志淹没关键信息：无级别控制。

### 思考题
1. 如何将调试日志接入远程收集系统？  
2. 如何在不影响帧率的前提下增加更多性能指标？

## 跨部门推广提示
- 开发：强制 PR 包含调试截图和性能数据。  
- 测试：建立标准复现模板（环境、步骤、指标）。  
- 运维：关联前端日志与监控告警。
