# 第03章：几何体与网格搭建仓储数字孪生基础积木

## 元信息
- 学习目标：掌握 Geometry、Material、Mesh 的关系，完成仓储场景基础建模。
- 先修要求：完成前两章的场景和相机基础。
- 预估时长：5-7 小时。
- 最终产出：一个包含货架、托盘、通道与状态色的仓储基础场景。
- 适用角色：开发、测试。

## 1. 项目背景（约500字）

仓储业务团队希望在大屏中直观看到库位占用率、堵塞通道和异常托盘。传统列表只能看“数字”，无法看“空间关系”：某区域明明占用率不高，但通道被临时堆放阻断，导致叉车效率下降。管理者希望通过三维场景实现“秒级定位”：哪个货架高占用、哪个通道拥塞、哪个托盘异常。

第一反应往往是导入完整高精度模型，但这样会遇到两个现实问题：一是美术资产准备周期长，业务验证慢；二是模型面数高，前期就引入性能风险。更务实的做法是先用几何体快速构建“结构正确”的数字孪生骨架，再逐步替换为真实模型。这样既能让业务快速评审，也方便后续工程优化。

本章将用 Box、Plane、Cylinder 等基础几何体，快速搭建一个“可运营”的仓储场景：货架、托盘、地面网格、通道标识，并通过网格命名规则将其绑定业务 ID。重点在于建模规范：如何把“看起来像仓库”的图形，变成“可交互、可追踪、可扩展”的业务对象集合。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：先做精模还是先做骨架

小胖：  
“要不直接上美术做好的仓库模型，一步到位，省得自己搭积木。”

小白：  
“精模当然好看，但我们现在要验证的是业务流程，不是宣传片。高精模会不会把排障和调试成本抬高？”

大师：  
“先骨架后精模。先确保库位关系、交互语义、数据绑定跑通，再换模型。就像先画户型图再做精装修。”  

技术映射：  
“先骨架后精模” = `Geometry MVP + 资源渐进替换`。

### 第二轮：如何让模型可追踪

小胖：  
“我能画出货架，但怎么知道这个是 A-03-12 库位？”

小白：  
“是不是每个 Mesh 要绑定业务主键？例如 `mesh.userData` 存储库位 ID 和状态。”

大师：  
“没错，并且命名规则要统一，后续拾取、告警、回放都依赖这个标识。”  

技术映射：  
“可追踪对象” = `mesh.name + mesh.userData + 业务主键`。

### 第三轮：大批量对象会不会卡

小胖：  
“仓库几千个托盘都建 Mesh，会不会还没上线就掉帧？”

小白：  
“是不是需要先按区域分组，再为后续 Instancing 留接口？”

大师：  
“对。基础篇先建立分组层级：仓库 -> 区域 -> 货架 -> 库位。中级篇再升级批量渲染策略。”  

技术映射：  
“分组层级” = `Scene Graph 结构化组织`。

### 3.2 分步实现

#### 步骤1：搭建地面和通道

```ts
const ground = new THREE.Mesh(
  new THREE.PlaneGeometry(80, 40),
  new THREE.MeshStandardMaterial({ color: "#0f172a" })
);
ground.rotation.x = -Math.PI / 2;
scene.add(ground);

const lane = new THREE.Mesh(
  new THREE.PlaneGeometry(80, 4),
  new THREE.MeshStandardMaterial({ color: "#1d4ed8" })
);
lane.rotation.x = -Math.PI / 2;
lane.position.y = 0.01;
scene.add(lane);
```

结果描述：仓库地面与主通道可视化完成。  
常见坑：通道与地面 z-fighting，需微抬高 `y`。

#### 步骤2：批量创建货架与库位

```ts
function createShelf(id: string, x: number, z: number) {
  const shelf = new THREE.Group();
  shelf.name = `shelf-${id}`;

  const frame = new THREE.Mesh(
    new THREE.BoxGeometry(2, 4, 1),
    new THREE.MeshStandardMaterial({ color: "#475569" })
  );
  frame.position.y = 2;
  shelf.add(frame);

  for (let i = 0; i < 4; i++) {
    const slot = new THREE.Mesh(
      new THREE.BoxGeometry(1.6, 0.2, 0.8),
      new THREE.MeshStandardMaterial({ color: "#22c55e" })
    );
    slot.position.set(0, 0.6 + i * 0.9, 0);
    slot.name = `${id}-slot-${i + 1}`;
    slot.userData = { slotId: slot.name, occupied: false };
    shelf.add(slot);
  }

  shelf.position.set(x, 0, z);
  return shelf;
}
```

结果描述：货架结构完整且每个库位可追踪。  
常见坑：未绑定业务 ID，后续无法做精准高亮。

#### 步骤3：根据业务状态更新颜色

```ts
function setSlotOccupied(slot: THREE.Mesh, occupied: boolean) {
  const mat = slot.material as THREE.MeshStandardMaterial;
  mat.color.set(occupied ? "#ef4444" : "#22c55e");
  slot.userData.occupied = occupied;
}

// 模拟实时状态推送
setInterval(() => {
  scene.traverse((obj) => {
    if (obj instanceof THREE.Mesh && obj.userData.slotId) {
      setSlotOccupied(obj, Math.random() > 0.7);
    }
  });
}, 2000);
```

结果描述：库位颜色周期变化，可表达占用状态。  
常见坑：每次更新创建新材质导致内存上涨，建议复用材质。

## 4. 项目总结（约500-800字）

### 优点
1. 建模成本低，业务反馈周期短。  
2. 对象天然可绑定业务 ID，便于交互与告警。  
3. 结构化分组为性能优化留出空间。

### 缺点
1. 视觉精细度有限，需要后续模型替换。  
2. 命名与分组规范一旦混乱，维护成本高。  
3. 状态频繁更新时需要更精细的性能策略。

### 常见故障案例
1. 大批量对象闪烁：地面重叠造成深度冲突。  
2. 颜色更新无效：材质共享导致全局串色。  
3. 拾取命中错误：未区分 Group 与 Mesh 目标对象。

### 思考题
1. 如果货架数量翻 10 倍，如何改造成 InstancedMesh 方案？  
2. 如何在不改变业务层接口的情况下替换成 glTF 精模？
