# 第02章：坐标系相机与视角控制从看得见到看得准

## 元信息
- 学习目标：掌握世界坐标、局部坐标、透视相机与轨道控制，避免常见“看得到但看不准”问题。
- 先修要求：完成第 01 章的最小场景搭建。
- 预估时长：4-6 小时。
- 最终产出：可在固定点位和自由漫游之间切换的业务视角系统。
- 适用角色：开发、测试。

## 1. 项目背景（约500字）

第一章上线后，业务方提出更具体诉求：值班人员希望一键切到“总览视角”“车间视角”“设备视角”。同时，测试同学反馈：同样的数据在不同分辨率下，画面聚焦点不一致；开发同学反馈：某些模型导入后方向反了，定位坐标总是对不上。这些问题本质都指向一个主题：坐标系和相机控制没有体系化设计。

在 3D 业务项目里，很多 bug 看上去像“渲染问题”，实际上是空间定义不一致。比如业务系统返回的是厂区平面坐标，前端却按模型局部坐标去移动对象；再比如相机 `lookAt` 每帧被覆盖，导致 OrbitControls 失效。结果就是：画面会动，但不可信。对运营场景来说，不可信比不好看更严重，因为可能误导值班决策。

本章核心不是讲公式，而是建立“空间一致性约定”：业务坐标如何映射到 three.js 世界坐标、相机如何提供可复现视角、交互控制如何保证易用不失真。最终目标是让每个角色都能在相同视角语义下协作：开发可实现、测试可验证、业务可理解。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：为什么每次打开视角都不一样

小胖：  
“我点同一个设备，今天看着在中间，明天就偏左，这跟食堂打饭窗口每天换位置一样离谱。”

小白：  
“是不是相机初始参数和容器尺寸耦合了？还有 aspect 更新是不是漏了？”

大师：  
“对。视角不稳定通常有三类原因：相机参数漂移、容器尺寸变化未同步、控制器状态未重置。先建立‘命名视角’，每个视角是一个可复用配置，而不是临时算一组值。”  

技术映射：  
“命名视角” = `camera preset + deterministic view`。

### 第二轮：坐标到底以谁为准

小胖：  
“我把设备往右挪 10 米，结果它跑到后面去了。右边到底是哪边？”

小白：  
“是不是业务系统用的是二维平面坐标（X/Y），three.js 场景用的是（X/Z）地面？”

大师：  
“完全正确。要先定义坐标转换函数，禁止在业务代码里手写换轴逻辑。约定一次，处处复用。”  

技术映射：  
“统一换轴” = `businessToWorldTransform()`。

### 第三轮：自由漫游和固定镜头怎么兼容

小胖：  
“领导演示要自动飞，值班员操作要手动拖。两套系统会不会打架？”

小白：  
“OrbitControls 和 tween 动画同时控制相机时会冲突，是否需要状态机切换？”

大师：  
“用相机控制状态机。‘自动巡航态’禁用手动输入，‘人工分析态’开启控制器。切换时同步目标点和阻尼参数，避免跳变。”  

技术映射：  
“两套控制不打架” = `cameraControlStateMachine`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
npm install three-stdlib
```

### 3.2 分步实现

#### 步骤1：定义业务坐标到世界坐标映射

```ts
type BizPoint = { x: number; y: number; h?: number };

function bizToWorld(p: BizPoint): [number, number, number] {
  // 约定：业务 y 映射到世界 z，业务 h 映射到世界 y
  return [p.x, p.h ?? 0, p.y];
}
```

结果描述：模型定位与业务平面图一致。  
常见坑：团队成员在多个文件重复手写换轴逻辑，导致漂移。

#### 步骤2：建立命名视角系统

```ts
import * as THREE from "three";

const viewPresets = {
  overview: { pos: new THREE.Vector3(20, 16, 20), target: new THREE.Vector3(0, 0, 0) },
  workshopA: { pos: new THREE.Vector3(8, 6, 10), target: new THREE.Vector3(2, 0, 0) },
  deviceFocus: { pos: new THREE.Vector3(3, 2, 3), target: new THREE.Vector3(2, 1, 0) }
};

function applyPreset(camera: THREE.PerspectiveCamera, key: keyof typeof viewPresets) {
  const preset = viewPresets[key];
  camera.position.copy(preset.pos);
  camera.lookAt(preset.target);
}
```

结果描述：切换视角稳定，可重复。  
常见坑：只改 `position` 不改 `lookAt`，导致焦点偏移。

#### 步骤3：接入 OrbitControls 与状态切换

```ts
import { OrbitControls } from "three/examples/jsm/controls/OrbitControls.js";

const controls = new OrbitControls(camera, renderer.domElement);
controls.enableDamping = true;

let mode: "auto" | "manual" = "manual";

function setMode(next: "auto" | "manual") {
  mode = next;
  controls.enabled = next === "manual";
}

function loop() {
  if (mode === "auto") {
    camera.position.x = Math.cos(Date.now() * 0.0002) * 20;
    camera.position.z = Math.sin(Date.now() * 0.0002) * 20;
    camera.lookAt(0, 0, 0);
  }
  controls.update();
  renderer.render(scene, camera);
  requestAnimationFrame(loop);
}
```

结果描述：自动巡航与手动拖拽可切换，画面稳定。  
常见坑：自动模式下仍调用用户输入导致抖动。

### 3.3 完整代码清单

- `src/camera/mode.ts`：控制状态机

## 4. 项目总结（约500-800字）

### 优点
1. 视角语义清晰，方便业务沟通。  
2. 坐标转换标准化，降低定位类 bug。  
3. 便于后续接入拾取、路径漫游与告警聚焦。

### 缺点
1. 前期需要建立规范，初始投入略高。  
2. 多模型来源时仍需额外做坐标对齐工具。  
3. 自动与手动控制切换逻辑要持续维护。

### 典型故障案例
1. 缩放后设备点选偏移：未同步相机 aspect。  
2. 漫游晕眩：相机移动速度和阻尼参数配置不当。  
3. 某批模型朝向反转：导入坐标系约定与引擎不一致。

### 思考题
1. 如何为“告警设备自动聚焦”设计平滑过渡，不打断用户手动操作？  
2. 当地图坐标改为经纬度时，坐标转换层应如何改造？
