# 第09章：UI叠加与2D3D联动图表与三维场景协同

## 元信息
- 学习目标：实现 2D 面板与 3D 场景双向联动，建立统一事件总线。
- 先修要求：完成第 08 章交互基础。
- 预估时长：4-6 小时。
- 最终产出：点击场景联动图表、点击图表反向定位场景对象。
- 适用角色：开发、测试。

## 1. 项目背景（约500字）

在业务评审中，管理层提到一个高频动作：先看总体图表，再定位具体设备。当前系统里，图表和 3D 场景是两张皮：图表显示告警数，场景显示设备位置，但两者互不驱动。用户需要在不同区域反复切换，造成认知断裂。产品要求：图表点击要能直接定位场景，场景选中也要同步刷新图表与详情。

这类问题本质不是 UI 组件，而是状态一致性。很多项目会在图表组件里直接调用 three.js 对象，或在 3D 逻辑里直接改 DOM，短期可用，长期不可维护。随着页面复杂度上升，很快出现“场景选中了但图表没更新”“图表筛选后场景残留旧高亮”等一致性问题。

本章的目标是建立一个轻量联动架构：统一事件、统一状态、统一渲染边界。你会得到一个可扩展模式，后续可接入 Vue/React 图表库而不破坏三维核心逻辑。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：直接互调行不行

小胖：  
“图表点了就直接 `scene.select(id)`，场景点了就直接 `chart.update()`，多快。”

小白：  
“直接互调会循环依赖，后期改任何一边都容易牵一发而动全身。”

大师：  
“用事件总线中转。UI 和 3D 都只依赖总线，不互相引用实现细节。”  

技术映射：  
“中转层” = `pub/sub event bus`。

### 第二轮：状态归谁管

小胖：  
“那选中状态放在图表还是场景？”

小白：  
“应该有一个全局 selection store，双方订阅同一数据源。”

大师：  
“对，状态单一来源。避免‘你以为选中 A，我以为选中 B’。”  

技术映射：  
“单一来源” = `single source of truth`。

### 第三轮：性能会不会被拖垮

小胖：  
“如果每次鼠标移动都发事件，图表和场景都刷，不会卡吗？”

小白：  
“高频事件要节流，且区分 hover 与 click 等级。”

大师：  
“事件分级处理：hover 轻量，click 完整联动，筛选批量更新做防抖。”  

技术映射：  
“事件分级” = `throttle + debounce + priority event`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install three
```

### 3.2 分步实现

#### 步骤1：定义事件总线

```ts
type AppEvent =
  | { type: "device:selected"; payload: { id: string } }
  | { type: "chart:filterChanged"; payload: { level: string } };

type Handler = (e: AppEvent) => void;
const handlers = new Set<Handler>();

export const bus = {
  emit(e: AppEvent) {
    handlers.forEach((h) => h(e));
  },
  on(h: Handler) {
    handlers.add(h);
    return () => handlers.delete(h);
  }
};
```

结果：UI 与 3D 可解耦通信。  
坑：事件名随意命名，后期难维护。

#### 步骤2：场景侧订阅与发布

```ts
bus.on((e) => {
  if (e.type === "chart:filterChanged") {
    applyFilterToScene(e.payload.level);
  }
});

function onPickDevice(id: string) {
  highlightDevice(id);
  bus.emit({ type: "device:selected", payload: { id } });
}
```

结果：图表筛选可驱动场景变更。  
坑：遗漏取消订阅导致页面切换后重复触发。

#### 步骤3：UI 侧联动

```ts
bus.on((e) => {
  if (e.type === "device:selected") {
    updateInfoPanel(e.payload.id);
    focusChartSeries(e.payload.id);
  }
});

function onChartClick(id: string) {
  bus.emit({ type: "device:selected", payload: { id } });
}
```

结果：双向联动闭环形成。  
坑：UI 和场景都触发同一事件时要防止重复处理。

## 4. 项目总结（约500-800字）

### 优点
1. 联动逻辑清晰，扩展成本低。  
2. 状态一致性好，减少“显示不同步”。  
3. 易接入多种 UI 框架。

### 缺点
1. 需要维护事件协议文档。  
2. 设计不当会出现事件风暴。  
3. 复杂筛选场景要做性能治理。

### 常见故障案例
1. 面板不刷新：事件被覆盖或监听丢失。  
2. 重复高亮：双向触发未去重。  
3. 内存泄漏：页面销毁未解除订阅。

### 思考题
1. 如何为事件总线增加可观测日志？  
2. 如何做“联动事务”，保证多组件原子更新？

- 测试：建立联动回放脚本。
