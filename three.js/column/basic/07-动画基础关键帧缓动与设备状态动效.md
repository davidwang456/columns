# 第07章：动画基础关键帧缓动与设备状态动效

## 元信息
- 学习目标：掌握基于时间轴的动画组织，避免“硬切状态”造成的信息噪音。
- 先修要求：完成材质、光照、资源章节。
- 预估时长：4-6 小时。
- 最终产出：具备平滑过渡和状态节奏感的设备动效系统。
- 适用角色：开发、测试。

## 1. 项目背景（约500字）

业务看板在联调时常见一个问题：状态更新是准的，但视觉反馈“太吵”。设备一旦告警就瞬间变红，恢复时又瞬间变绿，多个设备同时变化时像在闪烁噪点。值班人员反馈“看着累，重点不突出”。产品希望改成“有节奏、有层次”的动效：正常状态稳态呼吸、告警状态脉冲强化、恢复状态平滑回落。

如果动画没有统一机制，项目很快会变成“每个模块一套写法”：有人直接改 `position`，有人在 `setInterval` 里暴力刷新，有人每帧创建对象。短期能跑，长期会出现卡顿、抖动、内存抖升和不可维护。本章核心是把动画从“临时效果”升级为“工程系统”。

我们将从三个层面建立能力：统一时间驱动（delta time）、统一缓动函数（ease）、统一状态机（normal/warning/recover）。目标是在保证性能的前提下，让状态变化更可感知、更可验证。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：直接改值不就行

小胖：  
“告警就红、恢复就绿，瞬间切最干脆。”

小白：  
“瞬间切换对机器友好，对人不友好。用户需要过渡来理解变化方向。”

大师：  
“动画本质是信息过渡。让用户看懂‘正在发生变化’，而不是只看到变化结果。”  

技术映射：  
“信息过渡” = `state transition animation`。

### 第二轮：动画写在哪

小胖：  
“那我在业务回调里直接写动画代码？”

小白：  
“业务回调里混动画会耦合。是否应由动画系统消费状态事件？”

大师：  
“对。业务层只发状态，动画层只做表现。这样测试可以单独验证动画行为。”  

技术映射：  
“解耦” = `event-driven animation layer`。

### 第三轮：性能怎么控

小胖：  
“如果 500 个设备都做脉冲，会不会卡？”

小白：  
“每个设备都起独立定时器肯定不行，应该统一在主循环按时间计算。”

大师：  
“统一 `tick`，按公式计算，不开海量定时器。这是高并发动效的基本功。”  

技术映射：  
“统一 tick” = `single render loop update`。

### 3.2 分步实现

#### 步骤1：建立缓动函数

```ts
export function easeInOutCubic(t: number) {
  return t < 0.5 ? 4 * t * t * t : 1 - Math.pow(-2 * t + 2, 3) / 2;
}
```

结果：过渡自然，不突兀。  
坑：`t` 未限制在 `[0,1]` 导致曲线异常。

#### 步骤2：定义设备动画状态机

```ts
type DeviceAnimState = "normal" | "warning" | "recover";

interface DeviceAnim {
  mesh: THREE.Mesh;
  state: DeviceAnimState;
  startTime: number;
}

const animMap = new Map<string, DeviceAnim>();
```

结果：动效逻辑可管理。  
坑：状态切换未重置起始时间导致跳变。

#### 步骤3：在主循环中统一更新

```ts
function updateDeviceAnim(item: DeviceAnim, now: number) {
  const elapsed = (now - item.startTime) / 1000;
  const mat = item.mesh.material as THREE.MeshStandardMaterial;

  if (item.state === "warning") {
    const pulse = (Math.sin(elapsed * 6) + 1) * 0.5;
    mat.emissive.setRGB(0.4 * pulse, 0, 0);
    item.mesh.scale.y = 1 + pulse * 0.08;
  } else if (item.state === "recover") {
    const t = Math.min(elapsed / 1.2, 1);
    const k = easeInOutCubic(t);
    mat.emissive.setRGB((1 - k) * 0.3, 0, 0);
    item.mesh.scale.y = 1 + (1 - k) * 0.06;
    if (t === 1) item.state = "normal";
  } else {
    item.mesh.scale.y = 1 + Math.sin(elapsed * 2) * 0.02;
  }
}

function loop(now: number) {
  animMap.forEach((item) => updateDeviceAnim(item, now));
  renderer.render(scene, camera);
  requestAnimationFrame(loop);
}
```

结果：状态变化有节奏且稳定。  
坑：更新函数内创建临时对象会带来 GC 抖动。

## 4. 项目总结（约500-800字）

### 优点
1. 动效可读性高，告警重点突出。  
2. 动画系统化，便于维护和测试。  
3. 单循环驱动，性能更稳定。

### 缺点
1. 状态机设计有学习成本。  
2. 动效过多仍可能造成视觉噪音。  
3. 需建立统一节奏规范。

### 常见故障案例
1. 动画卡顿：使用多个 `setInterval` 竞争主线程。  
2. 恢复抖动：状态切换时未清理上一段动画。  
3. 偶发掉帧：每帧创建临时 `Color` 对象。

### 思考题
1. 如何将动画节奏与告警等级绑定（轻微/严重）？  
2. 如何为动画系统增加“全局降级开关”？

## 跨部门推广提示
- 开发：统一动画事件协议，避免各模块自定义。  
- 测试：对关键动效做录屏回归和阈值断言。  
- 运维：在低配终端启用简化动画策略。
