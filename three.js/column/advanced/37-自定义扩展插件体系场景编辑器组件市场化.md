# 第37章：自定义扩展——插件体系、场景编辑器与组件市场化

## 元信息
- 学习目标：构建可扩展平台能力，把“项目制开发”升级为“产品化生态”。
- 先修要求：完成高级篇核心技术与 SRE 章节。
- 预估时长：6-8 小时。
- 最终产出：插件协议、编辑器扩展点、组件发布规范三件套。
- 适用角色：架构师、平台开发、产品技术负责人。

## 1. 项目背景（约500字）

当三维平台进入规模化交付后，最大瓶颈常常不是性能，而是复用效率。每个项目都在重复开发：新场景加一套交互、新行业改一套面板、新团队复制一套特效。长期下去，维护成本线性增长，质量反而下降。要走出“项目泥潭”，必须产品化：插件化扩展、编辑器化编排、组件化分发。

插件体系的价值在于把变化隔离在边缘，把稳定沉淀在核心。核心平台只提供运行时、协议、权限、生命周期；行业能力通过插件接入。场景编辑器则让非核心开发也能编排场景，减少硬编码成本。组件市场化进一步让能力可共享、可版本化、可审计。

本章目标是设计一套可落地扩展生态：定义插件 API 合同、权限沙箱、版本兼容策略，接入编辑器扩展点，并建立组件发布与审核流程，支撑多团队并行创新。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：为什么要插件化

小胖：  
“直接在主仓库加功能，最快啊。”

小白：  
“短期快，长期会让核心仓库膨胀、发布风险上升。”

大师：  
“插件化是把变化外置，把核心稳定化。”  

技术映射：  
“变化外置” = `core-platform + extension modules`。

### 第二轮：插件会不会不安全

小胖：  
“第三方插件接进来，万一乱改核心状态怎么办？”

小白：  
“需要沙箱权限和 API 白名单。”

大师：  
“开放不等于失控，扩展必须在可审计边界内运行。”  

技术映射：  
“可审计开放” = `sandboxed extension runtime`。

### 第三轮：组件市场怎么保证质量

小胖：  
“大家都能上传组件，会不会鱼龙混杂？”

小白：  
“要有版本规范、质量门槛和审核流程。”

大师：  
“市场化核心是治理，不是仓库列表。”  

技术映射：  
“市场治理” = `registry + review + semver policy`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install
```

### 3.2 分步实现

#### 步骤1：定义插件协议

```ts
export interface ScenePlugin {
  meta: { name: string; version: string; permissions: string[] };
  setup(ctx: PluginContext): void;
  mount(): void;
  unmount(): void;
}
```

运行结果：插件具备统一生命周期。  
坑：协议无版本字段，升级时难兼容。

#### 步骤2：实现插件管理器与沙箱权限

```ts
class PluginManager {
  register(plugin: ScenePlugin) { /* 校验 meta + 权限 */ }
  activate(name: string) { /* mount */ }
  deactivate(name: string) { /* unmount */ }
}
```

运行结果：插件可动态启停。  
坑：未隔离全局对象导致插件互相污染。

#### 步骤3：编辑器扩展点与组件发布

```txt
编辑器扩展点：
- 属性面板扩展
- 工具栏扩展
- 场景节点上下文菜单

组件发布规范：
- 包含 manifest.json
- 遵循 semver
- 附带可视回归截图和性能报告
```

运行结果：能力可被复用和分发。  
坑：无审核门禁导致低质量组件进入市场。

## 4. 项目总结（约500-800字）

### 优点
1. 显著提升复用与交付速度。  
2. 核心平台更稳定。  
3. 支持生态化协作与能力沉淀。

### 缺点
1. 设计与治理成本较高。  
2. 协议演进需要兼容策略。  
3. 插件质量参差需持续审核。

### 常见故障案例
1. 插件冲突：命名空间未隔离。  
2. 版本断裂：核心 API 变更无兼容层。  
3. 组件失控：无审查流程引入风险代码。

### 思考题
1. 如何设计插件灰度发布与快速回滚机制？  
2. 如何把组件市场接入计费与授权体系？
