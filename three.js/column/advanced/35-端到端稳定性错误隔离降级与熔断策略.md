# 第35章：端到端稳定性——错误隔离、降级与熔断策略

## 元信息
- 学习目标：建立生产级稳定性防线，确保局部故障不拖垮整个平台。
- 先修要求：完成可观测、并行渲染与性能章节。
- 预估时长：6-8 小时。
- 最终产出：覆盖前端渲染、数据链路、服务依赖的稳定性治理框架。
- 适用角色：架构师、资深开发、SRE。

## 1. 项目背景（约500字）

高级阶段上线后，真正难的是“持续稳定运行”。系统往往不是一次大故障，而是局部故障连锁：某个数据源超时导致状态更新堆积，渲染线程被异常数据拖慢，交互模块卡住，最终用户看到整屏冻结。没有隔离和降级机制时，一个小问题会变成全局事故。

传统“报错就重试”在三维系统里不够用。因为这里有实时链路、GPU 资源、跨线程通信、后处理管线，多链路叠加后容错复杂度高。稳定性必须体系化设计：错误要被捕获、影响范围要被隔离、超时要可熔断、核心功能要可降级保底。

本章目标是构建端到端稳定性体系：错误分级（可恢复/不可恢复）、隔离域（模块沙箱）、降级策略（特效/精度/频率）、熔断策略（外部依赖保护），让系统“在异常中继续服务”。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：报错后重刷页面行不行

小胖：  
“崩了就刷新，最直接。”

小白：  
“刷新会中断用户操作，且根因可能持续存在。”

大师：  
“稳定性目标是‘不中断’，不是‘重来一次’。”  

技术映射：  
“不中断” = `graceful degradation`。

### 第二轮：隔离有必要吗

小胖：  
“模块都在一个进程，隔离是不是多余？”

小白：  
“没有隔离，任何模块异常都可能扩散。”

大师：  
“隔离域是防火墙，不是装饰品。”  

技术映射：  
“防火墙” = `fault containment boundary`。

### 第三轮：熔断会不会影响功能

小胖：  
“熔断等于主动关功能，会不会太激进？”

小白：  
“不熔断可能把主流程也拖死。”

大师：  
“熔断是保护核心链路的最后手段，短痛换长稳。”  

技术映射：  
“保护核心链路” = `circuit breaker for critical path`。

### 3.2 分步实现

#### 步骤1：错误分级与隔离包装

```ts
type FaultLevel = "recoverable" | "critical";

function runIsolated(name: string, fn: () => void) {
  try {
    fn();
  } catch (e) {
    reportFault(name, e);
    markModuleDegraded(name);
  }
}
```

运行结果：局部错误不再直接炸全局。  
坑：吞错不告警会掩盖真实问题。

#### 步骤2：功能降级开关

```ts
function applyDegradePolicy(score: number) {
  if (score < 40) {
    disablePostEffects();
    reduceUpdateFrequency();
  } else if (score < 60) {
    disableHeavyShaders();
  }
}
```

运行结果：负载异常时系统仍可用。  
坑：降级后无恢复策略，长期低质量。

#### 步骤3：外部依赖熔断器

```ts
class CircuitBreaker {
  private fails = 0;
  private openUntil = 0;
  call<T>(fn: () => Promise<T>) {
    if (Date.now() < this.openUntil) throw new Error("circuit-open");
    return fn().catch((e) => {
      this.fails++;
      if (this.fails >= 5) this.openUntil = Date.now() + 15000;
      throw e;
    });
  }
}
```

运行结果：故障依赖不会持续拖垮主链路。  
坑：阈值和恢复窗口配置不合理会误伤。

## 4. 项目总结（约500-800字）

### 优点
1. 显著提升生产稳定性。  
2. 故障影响可控，避免级联事故。  
3. 具备可恢复机制，支持长期运行。

### 缺点
1. 策略复杂，需持续演练。  
2. 阈值调优需要线上数据支撑。  
3. 过度降级可能影响用户体验。

### 常见故障案例
1. 错误扩散：未设置隔离域。  
2. 降级失效：策略触发条件不完整。  
3. 熔断误开：失败统计口径错误。

### 思考题
1. 如何把稳定性策略配置化并支持灰度发布？  
2. 如何设计“故障演练脚本”验证熔断与恢复链路？
