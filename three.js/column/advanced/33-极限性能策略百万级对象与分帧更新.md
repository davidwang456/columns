# 第33章：极限性能策略——百万级对象与分帧更新

## 元信息
- 学习目标：掌握百万级对象场景的极限优化方法，建立“预算驱动 + 分帧调度”体系。
- 先修要求：完成高级篇前置章节（源码、Shader、Pass、GIS）。
- 预估时长：6-8 小时。
- 最终产出：可在高对象规模下保持交互可用性的分帧更新框架。
- 适用角色：架构师、资深开发。

## 1. 项目背景（约500字）

当项目从“可展示”走向“超大规模运营”时，百万级对象是绕不过去的门槛：管线节点、传感器、路网点位、热区网格叠加后，数量很容易爆炸。常规优化（实例化、LOD、合批）在 10 万级仍有效，但到百万级，更新成本本身成为瓶颈。即使不渲染全部对象，只要每帧扫描和更新策略不对，主线程就会被压垮。

真实生产场景中，团队常见误区是“把所有计算塞进一帧”。结果是每隔几秒出现长帧卡顿，交互断续，告警响应滞后。用户感知上就是“画面会抽搐”。高级阶段需要转变思路：不是追求每帧做完所有事情，而是追求整体连续性和关键路径优先级。

本章目标是构建极限性能策略：把任务拆成可调度单元，按优先级和预算分帧执行，保证关键对象实时、次要对象延后，形成“稳定 30/60 FPS 优先”的工程模式。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第二轮：哪些对象先更新

技术映射：

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

### 3.2 分步实现

#### 步骤1：定义分帧任务模型

```ts
type UpdateTask = {
  id: string;
  priority: number; // 数值越小优先级越高
  run: (dt: number) => void;
  costHint: number; // 预估耗时 ms
};

const taskQueue: UpdateTask[] = [];
```

运行结果：更新逻辑可调度化。  
坑：任务粒度过大，无法细粒度调度。

#### 步骤2：实现帧预算调度器

```ts
const FRAME_BUDGET_MS = 6.0; // 除渲染外给逻辑的预算

function runScheduledTasks(dt: number) {
  const start = performance.now();
  taskQueue.sort((a, b) => a.priority - b.priority);
  for (const task of taskQueue) {
    if (performance.now() - start > FRAME_BUDGET_MS) break;
    task.run(dt);
  }
}
```

运行结果：长帧峰值明显降低。  
坑：每帧排序开销大，可改为分桶队列。

#### 步骤3：关键对象实时、普通对象轮询

```ts
function buildTasks(critical: string[], normal: string[]) {
  critical.forEach((id) => enqueueCriticalTask(id)); // 每帧
  normal.forEach((id, idx) => enqueueRoundRobinTask(id, idx % 8)); // 每 8 帧轮转
}
```

运行结果：关键区域响应稳定，整体负载可控。  
坑：轮转周期过长会造成可见延迟。

## 4. 项目总结（约500-800字）

### 优点
1. 把极限场景从“不可用”变为“可运营”。  
2. 关键路径可保障，体验更稳定。  
3. 预算驱动策略便于持续调优。

### 缺点
1. 调度系统复杂度高。  
2. 任务优先级配置需要持续维护。  
3. 分帧策略不当会带来一致性风险。

### 常见故障案例
1. 调度饥饿：低优任务长期得不到执行。  
2. 状态跳变：延迟更新对象未做插值。  
3. 预算失效：隐藏的重任务未拆分。

### 思考题
1. 如何结合用户视线预测动态调高任务优先级？  
2. 如何把调度指标纳入线上告警（如 budget overrun rate）？
