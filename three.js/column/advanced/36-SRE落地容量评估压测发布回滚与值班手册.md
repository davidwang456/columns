# 第36章：SRE 落地——容量评估、压测、发布回滚与值班手册

## 元信息
- 学习目标：把三维系统从“可开发”升级到“可运营”，形成 SRE 交付闭环。
- 先修要求：完成高级篇稳定性与可观测章节。
- 预估时长：6-8 小时。
- 最终产出：容量模型、压测基线、发布回滚剧本、值班 runbook。
- 适用角色：SRE、运维、架构师、核心开发。

## 1. 项目背景（约500字）

很多数字孪生项目在功能上很强，但上线后不稳定：活动日流量突增导致实时链路堆积、版本发布后回滚慢、夜间值班遇到故障无标准手册。问题不是某个模块写得差，而是缺少 SRE 体系：容量未建模、压测不完整、发布无守门、值班无剧本。

三维系统的运维复杂度更高，因为它既有前端渲染性能约束，也有实时数据、资产服务、权限链路等后端依赖。单看 API 指标无法代表真实可用性，必须把端侧帧率、卡顿率、错误率、实时延迟一起纳入 SLO。

本章目标是把 SRE 方法落到具体工程动作：定义容量公式、设计分层压测、实施灰度发布与自动回滚、沉淀值班手册和演练流程。最终达成“故障可预测、发布可控、恢复可执行”。

## 2. 项目设计（剧本式交锋对话，约1200字）

### 第一轮：上线后再扩容可以吗

小胖：  
“先上线看情况，不够再扩机器。”

小白：  
“突发流量不等人，被动扩容可能来不及。”

大师：  
“容量规划不是保守，是防止事故型成本。”  

技术映射：  
“防事故容量” = `capacity planning before launch`。

### 第二轮：压测只测后端接口够不够

小胖：  
“接口 QPS 很高，说明系统扛得住。”

小白：  
“三维系统还要测端侧渲染和实时推送，单测接口结论不完整。”

大师：  
“必须端到端压测：服务端吞吐 + 客户端可用性。”  

技术映射：  
“端到端压测” = `server+client full-path stress test`。

### 第三轮：回滚为啥要演练

小胖：  
“真出问题再回滚也来得及吧？”

小白：  
“事故时决策压力最大，临场操作最容易出错。”

大师：  
“回滚流程必须像消防演练一样，平时反复练。”  

技术映射：  
“可执行回滚” = `rehearsed rollback playbook`。

## 3. 项目实战（约1500-2000字）

### 3.1 环境准备

```bash
npm install
```

### 3.2 分步实现

#### 步骤1：定义容量模型与 SLO

```ts
type SLO = {
  p95RealtimeDelayMs: number;
  clientCrashRate: number;
  fpsP50: number;
};

const targetSLO: SLO = {
  p95RealtimeDelayMs: 1500,
  clientCrashRate: 0.001,
  fpsP50: 30
};
```

运行结果：稳定性目标可量化。  
坑：没有业务峰值场景，模型偏乐观。

#### 步骤2：分层压测与报告

```txt
压测层级：
1) API 压测：鉴权、查询、写入
2) 实时链路压测：WS 消息风暴
3) 端侧压测：多场景渲染 + 交互回放
```

```bash
curl -X POST http://localhost:8080/ops/loadtest/start -d '{"scenario":"peak_alarm"}'
```

运行结果：能识别瓶颈在服务、网络或端侧。  
坑：压测环境与生产差异过大导致结论偏差。

#### 步骤3：发布回滚与值班手册

```txt
灰度策略：
- 5% -> 20% -> 50% -> 100%
回滚触发：
- 错误率 > 1%
- p95 延迟 > 3s 持续 5 分钟
- 客户端 crash rate 超过基线 2 倍
```

运行结果：发布决策有硬门槛，回滚动作可快速执行。  
坑：只定义触发条件，不定义值班角色和响应时限。

## 4. 项目总结（约500-800字）

### 优点
1. 从“上线”升级到“稳定运营”。  
2. 事故响应速度和确定性提升。  
3. 跨部门协作链路更清晰。

### 缺点
1. 前期流程和演练投入较高。  
2. 阈值需要持续调优。  
3. 文档维护成本不可忽略。

### 常见故障案例
1. 容量误判：仅按日均流量估算。  
2. 压测失真：未覆盖实时风暴与端侧场景。  
3. 回滚失败：流程未自动化、职责不清。

### 思考题
1. 如何把错误预算（Error Budget）纳入发布节奏？  
2. 如何实现一键发布与一键回滚并保留审计链路？

## 跨部门推广提示
- 开发：重大改动必须附容量影响评估。  
- 测试：压测与回滚演练纳入发布门禁。  
- 运维：值班交接必须过 runbook 演练清单。
