# 第25章：模型服务化进阶：异步队列、批量合并与限流

## 1 项目背景

### 业务场景

客服工单分类服务上线第三个月，随着业务扩展，日均工单量从 8000 涨到 3 万，高峰期 QPS 突破 50。第 14 章搭建的简单 FastAPI 服务开始暴露出问题：

某个周一早上，大量用户集中提交工单（早高峰效应），瞬间 QPS 冲到 80。FastAPI 单 worker 同步推理模式下，请求开始排队——第 50 个请求要等前面 49 个都处理完才轮到自己，等待时间超过 3 秒。部分请求超时返回 504，客服系统把这批工单标记为"系统错误"，转给了人工——结果早高峰 8 个客服面对了平时 3 倍的工单量。

更糟糕的是，同一个 IP 在 1 秒内提交了 200 条工单——经查是测试脚本失控。这 200 条请求挤爆了推理队列，正常用户的请求被延迟到 5 秒以上。

运维团队紧急扩容了 GPU 实例，但新实例从冷启动到模型加载完成需要 15 秒——这 15 秒内所有请求都失败了。

### 痛点放大

简单 API 到生产级推理服务，中间需要补齐三个工程能力：

1. **请求队列与流量整形**：突发流量不能直接打到模型上，需要排队缓冲 + 削峰填谷
2. **限流与熔断**：保护后端模型不被打垮，防止某个用户的异常行为影响全体
3. **优雅启动与健康检查**：新实例就绪前自动切流量，就绪后无缝接入

```
请求 → 限流检查 → 请求队列 → 动态Batching → 模型推理 → 返回
         ↓                    ↓
      429限流拒绝          超时降级/熔断
```

## 2 项目设计

### 剧本式交锋对话

**场景**：周一上午 9:00，War Room。大屏上推理服务的延迟曲线像过山车，P99 突破 5 秒。运维小王满头大汗。

---

**小胖**:"这延迟曲线比我过山车还刺激。咋回事？模型又不是突然变慢了。"

**小陈**:"不是模型变慢，是请求太多了。50 个请求同时进来，单 worker 一次只能处理一个，后面的全在排队。"

**小胖**:"那你多开几个 worker 不就行了？就像食堂多开几个窗口。"

**小白**:"多 worker 方案对于 GPU 推理不适用——每个 worker 都要加载一份模型到显存，两张卡就 OOM 了。真正需要的是**请求层面的削峰填谷**——先让请求进队列，后台慢慢消费。"

**大师**:"对，这正是生产级推理服务的核心——流控和队列设计。让我讲清楚三个关键组件。

**组件一：异步任务队列。**

最简单的实现是用 Python 的 `asyncio.Queue`。请求进来时不是直接调用模型，而是：
1. 把请求封装为一个 task，放入队列
2. 返回一个 `Future` 对象给客户端（客户端 await 结果）
3. 后台 worker 从队列中取出请求、批量推理、设置 Future 的结果

这种模式天然地削峰填谷——即使瞬间来 100 个请求，它们都在队列里排队，不会直接压垮模型。

**组件二：动态批量合并（Dynamic Batching）。**

队列模式的一个关键优化是：worker 取任务时不是一条一条取，而是把队列中短时间内到达的所有请求打包成一个 batch 一起推理。这既利用了 GPU 的并行能力，又不显著增加单个请求的等待时间。

关键技术参数：
- `max_batch_size`：batch 上限（GPU 显存的硬约束）
- `max_wait_time`：最大等待时间（用户体验的软约束）
- worker 每隔 `max_wait_time`（或队列满 `max_batch_size`）触发一次批量推理

**组件三：限流与熔断。**

三层防护：
- **用户级限流**：同一用户/IP 每秒最多 N 次请求（防止脚本失控）
- **服务级限流**：推理服务总体 QPS 上限（保护 GPU）
- **熔断降级**：当模型响应超时或错误率过高时，自动返回降级结果（如'系统繁忙，已转人工处理'），而不是无限等待

另外还有**预热（warmup）**机制：服务启动时，在接入流量前先跑一批假请求让模型"热身"，避免首批真实请求遭遇冷启动延迟。"

**技术映射总结**：
- 异步队列 = 银行取号排队，人再多也不会一起挤到柜台
- Dynamic Batching = 拼车出行，顺路的乘客凑一车再出发
- 限流熔断 = 保险丝，问题请求被拦截不传导到后端
- 预热 = 运动员热身，上场前先活动开

### 3.2 生产级推理引擎

```python
# production_inference.py
"""生产级推理引擎: 异步队列 + 动态批处理 + 限流"""

import asyncio
import time
import logging
from typing import List, Dict, Optional, Tuple
from collections import deque
from dataclasses import dataclass

import torch
from transformers import AutoTokenizer, AutoModelForSequenceClassification

logging.basicConfig(level=logging.INFO, format="%(asctime)s [%(levelname)s] %(message)s")
logger = logging.getLogger(__name__)


@dataclass
class InferenceTask:
    text: str
    future: asyncio.Future
    arrive_time: float
    priority: int = 0  # 数值越低优先级越高


class ProductionInferenceEngine:
    """生产级推理引擎"""

    def __init__(
        self,
        model_path: str,
        max_batch_size: int = 32,
        max_wait_ms: float = 30.0,
        max_queue_size: int = 200,
        warmup_rounds: int = 3,
    ):
        self.tokenizer = AutoTokenizer.from_pretrained(model_path)
        self.model = AutoModelForSequenceClassification.from_pretrained(model_path)
        self.device = "cuda" if torch.cuda.is_available() else "cpu"
        self.model.to(self.device)
        self.model.eval()

        self.max_batch_size = max_batch_size
        self.max_wait_sec = max_wait_ms / 1000
        self.max_queue_size = max_queue_size

        # 请求队列（优先级队列）
        self.queue = deque()
        self.queue_lock = asyncio.Lock()

        # 统计
        self.stats = {
            "total_requests": 0,
            "total_batches": 0,
            "queue_timeouts": 0,
            "avg_latency_ms": 0,
        }

        # 预热
        self._warmup(warmup_rounds)
        logger.info(f"推理引擎就绪: device={self.device}, max_batch={max_batch_size}")

    def _warmup(self, rounds: int):
        """模型预热"""
        logger.info(f"预热中 ({rounds} 轮)...")
        warmup_texts = ["预热文本"] * min(4, self.max_batch_size)
        for _ in range(rounds):
            inputs = self.tokenizer(
                warmup_texts, truncation=True, max_length=128,
                padding=True, return_tensors="pt"
            ).to(self.device)
            with torch.no_grad():
                _ = self.model(**inputs)
        logger.info("预热完成")

    async def submit(self, text: str, priority: int = 0,
                     timeout_ms: float = 5000) -> Dict:
        """
        提交推理请求

        Args:
            text: 输入文本
            priority: 优先级（0最高）
            timeout_ms: 超时（毫秒）
        """
        # 队列满检查
        async with self.queue_lock:
            if len(self.queue) >= self.max_queue_size:
                raise RuntimeError(f"请求队列已满 ({self.max_queue_size})，请稍后重试")

            future = asyncio.Future()
            task = InferenceTask(
                text=text,
                future=future,
                arrive_time=time.time(),
                priority=priority,
            )
            self.queue.append(task)
            self.stats["total_requests"] += 1

        # 等待结果（带超时）
        try:
            result = await asyncio.wait_for(future, timeout=timeout_ms / 1000)
            return result
        except asyncio.TimeoutError:
            self.stats["queue_timeouts"] += 1
            return {
                "label": "unknown",
                "confidence": 0,
                "error": "请求超时，请稍后重试",
                "status": "timeout",
            }

    async def batch_worker(self):
        """后台批量推理 worker"""
        logger.info("Batch worker 已启动")

        while True:
            # 收集 batch
            batch_tasks = []
            batch_start = time.time()

            async with self.queue_lock:
                # 按优先级排序
                sorted_queue = sorted(self.queue, key=lambda t: t.priority)
                self.queue.clear()

                while sorted_queue and len(batch_tasks) < self.max_batch_size:
                    task = sorted_queue.pop(0)
                    wait_time = time.time() - task.arrive_time
                    if wait_time > 5.0:  # 超过5秒的丢弃
                        task.future.set_exception(
                            TimeoutError("请求在队列中等待过久")
                        )
                    else:
                        batch_tasks.append(task)

                # 剩余放回队列
                if sorted_queue:
                    self.queue.extendleft(reversed(sorted_queue))

            # 如果没有任务，等待一小段时间再检查
            if not batch_tasks:
                await asyncio.sleep(self.max_wait_sec / 2)
                continue

            # 批量推理
            if len(batch_tasks) < self.max_batch_size:
                elapsed = time.time() - batch_start
                if elapsed < self.max_wait_sec:
                    # 给更多请求到达的时间（但不强制等待）
                    await asyncio.sleep(0.005)

            texts = [t.text for t in batch_tasks]
            try:
                results = self._batch_inference(texts)
                self.stats["total_batches"] += 1

                for task, result in zip(batch_tasks, results):
                    wait_ms = (time.time() - task.arrive_time) * 1000
                    result["wait_ms"] = round(wait_ms, 1)
                    task.future.set_result(result)

            except Exception as e:
                logger.error(f"批量推理失败: {e}")
                for task in batch_tasks:
                    task.future.set_exception(e)

    def _batch_inference(self, texts: List[str]) -> List[Dict]:
        """批量推理"""
        inputs = self.tokenizer(
            texts, truncation=True, max_length=128,
            padding=True, return_tensors="pt"
        ).to(self.device)

        start = time.time()
        with torch.no_grad():
            outputs = self.model(**inputs)
            probs = torch.softmax(outputs.logits, dim=-1)
            top_probs, top_labels = torch.max(probs, dim=-1)
        infer_ms = (time.time() - start) * 1000

        id2label = self.model.config.id2label
        results = []
        for i in range(len(texts)):
            results.append({
                "label": id2label.get(str(top_labels[i].item()), "unknown"),
                "confidence": round(top_probs[i].item(), 4),
                "batch_size": len(texts),
                "infer_ms": round(infer_ms / len(texts), 1),
                "status": "success",
            })

        return results

    def get_stats(self) -> Dict:
        return self.stats


class RateLimiter:
    """简单的令牌桶限流器"""

    def __init__(self, rate: float, burst: int):
        """
        Args:
            rate: 每秒允许的请求数
            burst: 突发允许的最大请求数
        """
        self.rate = rate
        self.burst = burst
        self.tokens = burst
        self.last_refill = time.time()
        self.lock = asyncio.Lock()

    async def acquire(self) -> bool:
        """尝试获取令牌"""
        async with self.lock:
            now = time.time()
            elapsed = now - self.last_refill
            self.tokens = min(self.burst, self.tokens + elapsed * self.rate)
            self.last_refill = now

            if self.tokens >= 1:
                self.tokens -= 1
                return True
            return False
```

### 3.3 FastAPI 服务集成

```python
# server_advanced.py
"""生产级推理服务 —— 队列 + 限流 + 健康检查"""

import asyncio
import time
from fastapi import FastAPI, HTTPException, Request
from pydantic import BaseModel, Field
from production_inference import ProductionInferenceEngine, RateLimiter

# 初始化引擎
engine = ProductionInferenceEngine(
    model_path="./models/sentiment_minimal",
    max_batch_size=16,
    max_wait_ms=30,
    max_queue_size=100,
)

# 限流器（每IP每秒10次，突发20次）
rate_limiters = {}
GLOBAL_RATE = 50  # 全局 QPS 上限
global_limiter = RateLimiter(GLOBAL_RATE, burst=100)

app = FastAPI(title="生产级推理服务 v3.0")

@app.on_event("startup")
async def startup():
    asyncio.create_task(engine.batch_worker())
    app.state.start_time = time.time()
    # 等待队列就绪
    await asyncio.sleep(0.5)

class PredictRequest(BaseModel):
    text: str = Field(..., min_length=1, max_length=5000)
    priority: int = Field(10, ge=0, le=100, description="0=最高")

class PredictResponse(BaseModel):
    label: str
    confidence: float
    wait_ms: float
    status: str

@app.post("/predict", response_model=PredictResponse)
async def predict(request: PredictRequest, req: Request):
    # 全局限流
    if not await global_limiter.acquire():
        raise HTTPException(429, "服务繁忙，请稍后重试")

    # 用户级限流
    client_ip = req.client.host
    if client_ip not in rate_limiters:
        rate_limiters[client_ip] = RateLimiter(10, burst=20)
    if not await rate_limiters[client_ip].acquire():
        raise HTTPException(429, "请求过于频繁，请稍后重试")

    try:
        result = await engine.submit(
            text=request.text,
            priority=request.priority,
            timeout_ms=5000,
        )
        return PredictResponse(**result)
    except RuntimeError as e:
        raise HTTPException(503, str(e))

@app.get("/health")
async def health():
    stats = engine.get_stats()
    uptime = time.time() - app.state.start_time if hasattr(app.state, "start_time") else 0
    return {
        "status": "healthy",
        "queue_depth": len(engine.queue),
        "uptime_seconds": round(uptime, 1),
        "total_requests": stats["total_requests"],
        "total_batches": stats["total_batches"],
        "queue_timeouts": stats["queue_timeouts"],
    }

@app.get("/stats")
async def stats():
    return {
        "engine_stats": engine.get_stats(),
        "limiter_tokens": round(global_limiter.tokens, 1),
    }

if __name__ == "__main__":
    import uvicorn
    uvicorn.run(app, host="0.0.0.0", port=8000, workers=1)
```

## 4 项目总结

### 4.1 优点与缺点

| 方案 | 优点 | 缺点 |
|------|------|------|
| **异步队列** | 削峰填谷，突发流量不丢请求 | 增加等待延迟，队列过长时用户体验差 |
| **动态批处理** | GPU利用率高，吞吐提升3-5倍 | 增加代码复杂度，max_wait参数需要调优 |
| **令牌桶限流** | 平滑流量，保护后端 | 突发流量时部分请求被拒绝 |
| **优先级队列** | VIP请求优先处理 | 低优先级请求可能"饿死" |

### 4.2 适用场景

| 场景 | 推荐方案 |
|------|---------|
| 高并发在线推理（>50 QPS） | 异步队列 + 动态批处理 |
| 多租户SaaS | 每租户限流 + 优先级队列 |
| 大促流量洪峰 | 队列缓冲 + 弹性扩容 + 降级策略 |
| 实时性要求<50ms | 跳过队列直接推理 |

**不适用场景**：
- 严格单条实时推理（如对话流式输出），不适合批量合并
- GPU资源极端稀缺（需排队等待分钟级），考虑异步回调模式

### 4.3 注意事项

1. **队列深度监控**：queue_depth > max_queue_size × 80% 时告警，准备扩容
2. **优雅关闭**：收到 SIGTERM 后停止接新请求、排空队列、保存状态、再退出
3. **future 内存泄漏**：超时的 future 必须 cancel，否则引用无法释放

### 4.4 常见踩坑经验

| 故障现象 | 根因 | 解决方案 |
|---------|------|---------|
| 队列越来越长最终OOM | worker消费速度 < 入队速度 | 扩容或限流，设置 max_queue_size 硬上限 |
| 批量推理结果张冠李戴 | batch 顺序与 future 对应错误 | 用有序列表确保 texts 与 futures 一一对应 |
| 限流器时间漂移 | `time.time()` 可能回调 | 生产环境用 `time.monotonic()` |

### 4.6 第24章思考题答案

**第24章思考题1**：
- num_gpus=8 时 ZeRO-3 的 per_gpu 显存不会继续线性下降到 4GPU 的一半，因为激活值（activations）不参与拆分——激活值取决于 batch_size 和 seq_len，与 GPU 数量无关。随着 GPU 数增加，激活值占总显存的比例越来越高，边际收益递减。

**第24章思考题2**：
- all_gather 发生在前向传播每层开始前（收集参数）和反向传播每层结束后（收集梯度）。PCIe 32GB/s vs NVLink 600GB/s，通信带宽差约 20 倍。在 PCIe 环境下 ZeRO-3 的通信时间占训练总时间的 40-60%，而在 NVLink 下仅占 5-10%。这就是为什么大模型训练通常需要 NVLink GPU。
