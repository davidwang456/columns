# 第5章：文本分类实战：客服工单自动分派

## 1 项目背景

### 业务场景

某电商公司的客服中心日处理工单量已突破 8000 条。目前的分派方式是：客服人员接到工单后，手动阅读内容，根据经验判断属于"投诉""咨询""退款""物流""售后"中的哪一类，然后转发给对应的处理小组。

运营总监李总在季度复盘会上看到了三个触目惊心的数据：
- **平均分派耗时**：人工阅读+判断平均 45 秒/条，8000 条 × 45 秒 = 100 人时/天
- **分派错误率**：新手客服错误率 22%，老手也有 8%，错误分派导致工单在处理小组之间反复转发，平均延迟 4.2 小时
- **垃圾工单占比**：约 15% 的工单是无效内容（测试工单、空白内容、广告骚扰），占用了人工审核时间

技术总监拍板：用 Transformers 训练一个自动分类模型，三个月内实现 90%+ 的自动分派准确率，人工只需复核模型不确定的边界工单。

### 痛点放大

从"能跑 Demo"到"能用的业务系统"，中间还有一座大山：

```
┌─────────────┐    ┌──────────┐    ┌──────────┐    ┌────────────┐
│ 原始工单数据  │───→│ 数据清洗  │───→│ 模型训练  │───→│ 线上推理    │
│ (CSV/Excel) │    │ 标签映射  │    │ 评估指标  │    │ 业务校验    │
└─────────────┘    └──────────┘    └──────────┘    └────────────┘
                         ↓               ↓               ↓
                    脏数据怎么办？    loss不下降了？  线上准确率掉了？
```

三个核心难题：
1. **数据质量问题**：真实工单中存在大量噪声——测试工单（内容为 "1111"）、客服手动关闭的空工单、第三方系统同步的半结构化 JSON 片段。
2. **标签分布不均**：80% 工单集中在"咨询"和"物流"，"投诉"和"售后"仅占 5%。模型会倾向于预测多数类。
3. **业务验收标准**：开发人员关心 F1 Score，但运营关心的是"自动分派后，有多少工单不需要人工干预"。这两个指标之间需要翻译。

**小白**:"这就是分类任务的核心难点——**边界样本**。单标签分类假设每条工单只属于一个类别，但现实中很多工单同时涉及多个问题。我们需要明确标签的定义边界和优先级。"

**小陈**:"对。我们的标注规范可以这样定：'退款退货'类优先级最高，只要提到退款意图就归这一类。然后是'投诉'——表达强烈不满但未提具体的退款要求。以此类推。"

**大师**:"很好，这就是**业务规则优先于模型规则**。在做技术选型和数据标注之前，先要跟运营团队对齐标签定义。我来画出完整的流程。"

大师在白板上画出技术方案：

```
阶段 1: 数据准备
├── 导出 6 个月历史工单（CSV 格式，约 5 万条）
├── 数据清洗：去空、去重、去测试数据、去系统自动回复
├── 标签映射：原有 15 个细分类 → 5 个大类
└── 训练/验证/测试集划分：80% / 10% / 10%

阶段 2: 模型训练
├── 模型选择：bert-base-chinese（4层分类头）
├── 训练框架：Trainer API

├── 验收 F1（宏平均）> 0.85

```

**小胖**:"等一下，5 万条数据够吗？我听说大模型都是几十亿条数据训练的。"

**大师**:"5 万条是微调（fine-tuning），不是从零预训练。BERT 已经在海量中文语料上预训练过，学会了中文语法和常识。我们要做的只是给它看我们自己的工单数据，让它学会'什么是投诉、什么是咨询'。就像你雇了一个大学中文系毕业生，现在培训他三天熟悉客服业务——不需要从拼音开始教。"

**小白**:"那模型的选择呢？为什么要用 BERT 而不是直接上大模型？现在 GPT-4 一条 prompt 也能分类啊。"

**大师**:"成本和场景决定：
- BERT-base-chinese：110M 参数，单条推理 CPU 上 50ms，无可变成本。一次训练 20 分钟。
- GPT-4 API：按 token 计费，5 万条分类转人工审核后再调 API？没必要。再考虑到数据安全和离线可用，自建小模型是我们的最优解。
- 为什么不是更大的模型？分类任务不需要生成能力，Encoder 足够。BERT-base 的参数量和速度是性价比最优的点。"

**小陈**:"我还有一个问题。如果线上出现新类型的工单怎么办？比如公司推出了'以旧换新'业务，这种工单不在我们的 5 个类别里。"

**大师**:"两个层面处理：第一，在模型的置信度阈值之上加一道规则——如果 top-1 置信度 < 0.5，转人工处理并记录新类型。第二，每月 review 转人工的工单，发现新类别后重新标注、增量训练。这就是 ML 系统的日常运维闭环，不是训完就完了。"

**技术映射总结**：
- Fine-tuning = 用预训练大模型的骨架 + 自己的业务数据微调，非从零训练
- BERT (Encoder-only) 是分类任务的第一选择，非 GPT (Decoder-only)
- 不确定工单转人工是生产系统必备的"降级策略"
- 业务标签定义应由运营牵头、算法配合，而非算法自行决定

def clean_data(df: pd.DataFrame) -> pd.DataFrame:
    """清洗数据"""
    initial_count = len(df)

    # 2.1 去除空文本
    df = df[df["text"].str.strip() != ""].copy()
    print(f"去除空文本: {initial_count} → {len(df)}")

    # 2.2 去除测试数据（长度 < 5 或只包含数字/英文/标点）
    count_before = len(df)
    df = df[df["text"].apply(lambda x: len(x.strip()) >= 5)]
    print(f"去除过短文本: {count_before} → {len(df)}")

    # 2.3 去除重复工单
    count_before = len(df)
    df = df.drop_duplicates(subset=["text"])
    print(f"去除重复: {count_before} → {len(df)}")

    # 2.4 映射标签为数字 ID
    label2id = {
        "退款退货": 0, "投诉": 1, "咨询": 2,
        "物流问题": 3, "售后": 4, "其他": 5, "无效": 6
    }
    id2label = {v: k for k, v in label2id.items()}

    # 过滤无效标签
    valid_labels = set(label2id.keys())
    count_before = len(df)
    df = df[df["label"].isin(valid_labels)]
    print(f"过滤无效标签: {count_before} → {len(df)}")

    df["label_id"] = df["label"].map(label2id)

    # 打印分布
    print("\n标签分布:")
    print(df["label"].value_counts())

    return df, label2id, id2label

        examples["text"],

        truncation=True,
        max_length=128,

tokenized_dataset = dataset.map(tokenize_function, batched=True)

#### 目标

```python
# train_classifier.py
"""客服工单分类模型训练"""

import os
import numpy as np
import torch
from datasets import load_from_disk
from transformers import (
    AutoModelForSequenceClassification,
    AutoTokenizer,
    Trainer,
    TrainingArguments,
    EarlyStoppingCallback,
)
from sklearn.metrics import accuracy_score, precision_recall_fscore_support, confusion_matrix
import evaluate

# ============ 1. 加载数据与配置 ============
dataset = load_from_disk("./data/ticket_dataset")
num_labels = 7  # 与我们数据清洗时的 7 个类别对应
model_name = "bert-base-chinese"

# 判断设备
device = "cuda" if torch.cuda.is_available() else "cpu"
print(f"使用设备: {device}")

tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForSequenceClassification.from_pretrained(
    model_name,
    num_labels=num_labels,
    id2label={i: f"class_{i}" for i in range(num_labels)},  # 实际应用中用 id2label
    label2id={f"class_{i}": i for i in range(num_labels)},
)

# ============ 2. 定义评估指标 ============
metric = evaluate.load("accuracy")

def compute_metrics(eval_pred):
    """计算分类指标"""
    logits, labels = eval_pred
    predictions = np.argmax(logits, axis=-1)

    accuracy = accuracy_score(labels, predictions)
    precision, recall, f1, _ = precision_recall_fscore_support(
        labels, predictions, average="macro", zero_division=0
    )

    return {
        "accuracy": accuracy,
        "precision": precision,
        "recall": recall,
        "f1": f1,
    }


# ============ 3. 配置训练参数 ============
training_args = TrainingArguments(
    output_dir="./output/ticket_classifier",
    evaluation_strategy="epoch",       # 每个 epoch 结束后评估
    save_strategy="epoch",             # 每个 epoch 结束后保存
    logging_strategy="steps",
    logging_steps=10,
    learning_rate=2e-5,
    per_device_train_batch_size=8,
    per_device_eval_batch_size=16,
    num_train_epochs=3,
    weight_decay=0.01,
    load_best_model_at_end=True,       # 训练结束后加载最佳模型
    metric_for_best_model="f1",        # 以 F1 作为最佳模型选择标准
    greater_is_better=True,
    save_total_limit=2,                # 最多保留 2 个 checkpoint
    push_to_hub=False,
    report_to="none",                  # 不上报 wandb/tensorboard（可改为 "tensorboard"）
    fp16=torch.cuda.is_available(),    # GPU 启用混合精度
    dataloader_num_workers=2,
)

# ============ 4. 训练 ============
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=dataset["train"],
    eval_dataset=dataset["validation"],
    tokenizer=tokenizer,
    compute_metrics=compute_metrics,
    callbacks=[EarlyStoppingCallback(early_stopping_patience=2)],
)

print("开始训练...")
trainer.train()

# ============ 5. 评估 ============
print("\n在测试集上评估...")
test_results = trainer.evaluate(dataset["test"])
print(f"测试集结果: {test_results}")

# ============ 6. 保存最终模型 ============
save_path = "./models/ticket_classifier_final"
os.makedirs(save_path, exist_ok=True)
trainer.save_model(save_path)
tokenizer.save_pretrained(save_path)
print(f"模型已保存至 {save_path}")
```

### 3.4 模型推理与置信度策略

```python
# inference.py
"""客服工单分类推理脚本 —— 带置信度阈值"""

import torch
from transformers import AutoTokenizer, AutoModelForSequenceClassification
import time


class TicketClassifier:
    def __init__(self, model_path: str, confidence_threshold: float = 0.7):
        self.tokenizer = AutoTokenizer.from_pretrained(model_path)
        self.model = AutoModelForSequenceClassification.from_pretrained(model_path)
        self.model.eval()
        self.threshold = confidence_threshold
        self.id2label = self.model.config.id2label

        # 获取设备
        self.device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
        self.model.to(self.device)

    def predict(self, text: str) -> dict:
        """单条预测（带置信度）"""
        start = time.time()

        inputs = self.tokenizer(
            text,
            max_length=128,
            truncation=True,
            padding="max_length",
            return_tensors="pt"
        ).to(self.device)

        with torch.no_grad():
            outputs = self.model(**inputs)
            probs = torch.softmax(outputs.logits, dim=-1)
            top_prob, top_label = torch.max(probs, dim=-1)

        confidence = top_prob.item()
        label = self.id2label.get(top_label.item(), "未知")

        latency = (time.time() - start) * 1000

        # 置信度策略
        if confidence < self.threshold:
            action = "转人工 (置信度不足)"
            label = f"{label} (需人工复核)"
        else:
            action = "自动分派"

        return {
            "label": label,
            "confidence": round(confidence, 4),
            "latency_ms": round(latency, 1),
            "action": action,
        }

    def batch_predict(self, texts: list) -> list:
        """批量预测"""
        inputs = self.tokenizer(
            texts,
            max_length=128,
            truncation=True,
            padding="max_length",
            return_tensors="pt"
        ).to(self.device)

        with torch.no_grad():
            outputs = self.model(**inputs)
            probs = torch.softmax(outputs.logits, dim=-1)
            top_probs, top_labels = torch.max(probs, dim=-1)

        results = []
        for prob, label_id in zip(top_probs, top_labels):
            confidence = prob.item()
            label = self.id2label.get(label_id.item(), "未知")
            results.append({
                "label": label,
                "confidence": round(confidence, 4),
                "action": "自动分派" if confidence >= self.threshold else "转人工",
            })
        return results


# ===== 使用示例 =====
if __name__ == "__main__":
    classifier = TicketClassifier(
        model_path="./models/ticket_classifier_final",
        confidence_threshold=0.7
    )

    test_texts = [
        "订单XH20240501，买的手机收到是坏的，要退货退款！",
        "你好，请问这个产品包邮吗？",
        "快递太慢了，投诉！",
    ]

    for text in test_texts:
        result = classifier.predict(text)
        print(f"📝 {text[:50]}...")
        print(f"   📊 结果: {result['label']} ({result['confidence']:.1%}) → {result['action']}")
        print(f"   ⏱️ 耗时: {result['latency_ms']:.1f}ms\n")
```

## 4 项目总结

### 4.1 优点与缺点

| 方面 | 优点 | 缺点 |
|------|------|------|
| **Trainer API** | 一行 `.train()` 完成训练，自动处理 checkpoint 保存、日志、评估 | 高度封装，难以自定义训练循环细节（如梯度操作、动态 batch） |
| **BERT 微调** | 小数据（数千条）即可收敛，训练耗时短（GPU 10-20 分钟） | 长文本（>512 tokens）需截断，超长工单可能丢信息 |
| **置信度阈值** | 降低误分派成本，不确定的工单自动转人工 | 阈值设定依赖人工经验，过高增加人力成本，过低增加出错率 |
| **AutoClass 模型** | `from_pretrained` 自动选择正确模型类 | 自定义分类头结构时需手动构造 model 对象 |

### 4.2 适用场景

| 场景 | 推荐方案 |
|------|---------|
| 客服工单自动分派 | BERT + 单标签分类 + 置信度阈值 |
| 包含多个意图的工单 | 多标签分类（sigmoid 替代 softmax） |
| 层级类别（大类→小类） | 分层分类：先大类模型，再小类子模型 |
| 实时流式分派 | 模型预加载单例 + 异步批量推理 |

**不适用场景**：
- 工单类别频繁新增/删除（需要重新标注和训练，运维成本高）
- 完全无标注数据（需要使用零样本分类或少样本学习方法，后续章节会介绍）

### 4.3 注意事项

1. **训练/推理 Tokenizer 配置一致**：`max_length`、`truncation`、`padding` 参数必须完全一致，否则输入维度不匹配
2. **label 顺序与 id2label 一致**：保存模型时将 `label2id`/`id2label` 写入 config.json，推理时从 config 读取，防止 id 错位
3. **不平衡数据处理**：如果使用 `class_weight` 或 `WeightedRandomSampler`，需在 `Trainer` 的 `compute_loss` 中自行实现（Trainer 默认不支持自动 class_weight）

### 4.4 常见踩坑经验

| 故障现象 | 根因 | 解决方案 |
|---------|------|---------|
| 预测全是一个类别 | 标签分布极度不均 + 模型坍塌 | 检查标签分布，加入 `class_weight` 或对少数类过采样 |
| 训练 loss 不下降 | 学习率过大导致震荡 | 降低 lr（1e-5），增加 warmup_ratio |
| 验证集 F1 虚高 | 数据泄漏：训练集和验证集有重复文本 | `df.drop_duplicates(subset=['text'])` 后重新划分 |
| 线上准确率显著低于测试 | 线上文本分布与训练集不同（数据漂移） | 增加线上数据回标流程，每月增量训练 |

### 4.6 第4章思考题答案

**第4章思考题1答案**：
- 大小通常不一致。`pytorch_model.bin` 使用 pickle 序列化，包含 Python 对象元数据和压缩后的张量数据。`model.safetensors` 仅存储纯张量数据（dtype + shape + raw bytes），不含 Python 对象序列化开销。对于 BERT-base（110M 参数），safetensors 通常比 pickle 约小 0-2%，差异在张量对齐和元数据上。对于大模型，safetensors 支持内存映射（mmap），加载时内存占用更少。

**第4章思考题2答案**：
- 可以复用，步骤如下：
  1. 加载原模型的 config，将 `num_labels` 改为 3
  2. 加载原模型权重，但跳过分类头（`classifier.weight` 和 `classifier.bias`）
  3. 新建一个 3 分类的分类头（`nn.Linear(768, 3)`）
  4. 用少量标注数据仅训练分类头参数（冻结 BERT 骨架），收敛速度极快
- 如果不想写额外代码，最简方案：用 `AutoModelForSequenceClassification.from_pretrained(model_dir, num_labels=3, ignore_mismatched_sizes=True)`，新的分类头会随机初始化但 BERT 骨架权重保留。
