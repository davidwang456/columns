# 第37章：复制引擎源码深度剖析

## 1 项目背景

某电商平台运维团队负责管理横跨三个数据中心的Harbor镜像仓库集群——北京主站点（生产环境）、上海灾备站点（容灾）、以及新加坡海外站点（东南亚业务）。为了实现跨区域的镜像同步，团队深度依赖Harbor的复制引擎。然而，随着业务增长，现有的复制机制暴露出多个问题。

**痛点一：复制适配器扩展缺乏指南。** Harbor原生支持Harbor-to-Harbor、DockerHub、AWS ECR、Azure ACR等Registry类型，但公司的新加坡站点使用的是腾讯云TCR（Tencent Container Registry）作为目标仓库。团队需要开发一个TCR适配器，但复制引擎的Adapter接口文档分散在各源码文件中——`ArtifactRegistry`、`BlobRegistry`、`RepositoryRegistry`三个接口的关系不清楚，哪些方法是必须实现的最小集也无明确说明。

**痛点二：复制任务的错误处理与重试机制分散。** 跨区域复制经常遇到网络波动——连接超时、TLS握手失败、目标Registry限流、甚至目标仓库临时不可用。团队发现相同的网络超时错误在不同代码路径中的处理方式截然不同：`PullBlob`的超时直接导致整个Task失败（无重试），而`PushManifest`的超时会触发3次指数退避重试。这种不一致导致部分镜像复制"卡死"数小时。

**痛点三：事件驱动复制的触发链路不透明。** 团队依赖"推送即复制"的事件驱动模式——北京push一个镜像，自动同步到上海和新加坡。但某次扩容演练中发现：短时间内push 500个镜像后，只有前200个触发了复制。排查发现是事件系统的内部channel缓冲区满了，后续事件被丢弃。团队需要理解事件驱动复制的完整链路：push事件如何产生、如何进入调度队列、队列溢出时如何处理。

**痛点四：增量复制的去重逻辑不清。** 复制策略配置为"按标签过滤+定时触发"时，同一个镜像的同一个标签如果源和目标内容一致，Harbor是否会重复复制？团队发现部分场景下Blob层确实被重复上传——原因在于Manifest的比较逻辑只比较了Digest，而忽略了Schema Version的差异，导致OCI格式的Manifest即使内容相同也被判定为"需要复制"。

## 2 项目设计——剧本式交锋对话

*运维团队的小会议室里，白板上画满了复制引擎的流程图，三个人围坐在笔记本电脑前，屏幕上开着Harbor的源码仓库。*

**小胖**：（挠头看着屏幕上的`adapter.go`接口定义）"大师，复制引擎不就是个文件搬运工吗？把镜像从源Registry拉到本地，再推到目标Registry，跟`docker pull && docker tag && docker push`有什么区别？为什么要搞这么复杂的Adapter模式、策略模式、责任链模式？我感觉这代码过度设计了！"

**大师**：（从保温杯里喝了口茶）"小胖，我问你——如果把北京机房的Windows台式机快递到新加坡，你会怎么操作？"

**小胖**："打包、贴面单、叫快递、空运……但跟复制镜像有什么关系？"

**大师**："镜像复制比搬台式机复杂多了。台式机你只需要一个物流公司（一个Adaptor），但镜像复制要考虑：目标仓库是DockerHub还是AWS ECR？ECR要求用AWS Signature V4签名，DockerHub用Basic Auth，Harbor用JWT——每种认证方式都不一样，这就像不同国家的海关要求不同签证。Adaptor模式就是**签证中心**——对外提供统一接口`PushManifest`/`PullBlob`，内部针对不同Registry实现不同的认证和API调用方式。"

**小白**：（手指在键盘上快速搜索源码）"我看到`src/pkg/reg/adapter/`下有`harbor/`、`dockerhub/`、`awsecr/`、`googlegcr/`这些目录。每个都实现了Adapter接口。但我不明白的是——复制引擎怎么知道为哪个目标Registry创建哪个Adaptor？是通过配置中的`type`字段做查找表吗？"

**大师**："对，这就是典型的**工厂模式+注册表**。每个Adapert包在`init()`函数中调用`adapter.RegisterFactory("harbor", ...)`注册自己。复制引擎创建执行任务时，根据策略中配置的`target_registry.type`去工厂表里查找构造函数。就像你到商场的美食广场——你想吃川菜就去'川味轩'，想吃日料就去'樱花亭'，不用关心后厨怎么做，因为你只需要一个统一的动作：点餐。"

**小胖**："等等——刚才我试了一下，对腾讯云TCR调用`PushBlob`返回了`400 Bad Request`。日志显示：`PUT /v2/myapp/blobs/uploads/{uuid}?digest=sha256:xxx`，响应：`{"errors":[{"code":"DIGEST_INVALID"}]}`。这啥意思？"

**大师**：（凑近屏幕细看）"这是TCR的一个特性差异。Docker Registry标准允许一次性`PUT`上传Blob并附带Digest完成上传——但某些云厂商Registry要求先`POST`初始化上传会话、再`PATCH`分块上传数据、最后`PUT`确认。你的Adaptor可能跳过了`POST`初始化这一步，直接把Blob往`PUT`端点推。就像你寄快递不填面单（POST初始化），直接把包裹扔到快递柜——快递员不知道这个包裹是给谁的。"

**小白**："那`BlobRegistry`接口里为什么只有`PushBlob`、`PullBlob`、`BlobExist`、`DeleteBlob`四个方法？没有`InitiateUpload`或者`ChunkUpload`？如果TCR必须分块上传，接口设计是不是有缺陷？"

**大师**：（微笑）"好问题。这就是接口设计的权衡了。《人月神话》里说'接口抽象层级决定灵活性与复杂度'。Harbor选择在`PushBlob`内部封装所有上传逻辑——适配器自己负责判断是一次性PUT还是分块PATCH。对调用方（复制引擎的flow控制器）来说，它只需要说'把Blob推过去'，不用关心内部是分块还是整体。这就像你在京东下单——你只关心'收货'，京东自己决定是整单发货还是分拆多个包裹。"

**小胖**："那我实现TCR Adaptor的时候，`PushBlob`内部要先POST创建上传会话、再PATCH分块、最后PUT完成？这得写一大坨代码啊！"

**大师**："不，你可以先检查目标Registry是否已经存在该Blob——调用`BlobExist`。如果已存在就直接跳过，这就是复制引擎的**去重机制**。只有新的Blob才需要完整上传。而且Harbor社区已经有`native/`适配器（通用Docker Registry适配器），TCR兼容Docker Registry V2 API的话，你应该从`native/`适配器继承，只重写认证部分。复用大于重写——这是工程师的基本素养。"

**小白**：（打开一个新文件开始画架构图）"我梳理一下完整的复制流程：第一步，触发器（事件/定时/手动）触发；第二步，Policy Manager加载策略并匹配Filter规则；第三步，Flow Controller创建执行任务、分配Adapter；第四步，JobService Worker执行具体复制Job——先PullBlob、再PushBlob、最后PushManifest。我的问题是——如果PullBlob成功但PushBlob失败了，那些已经推到目标Registry的Blob怎么办？会变成垃圾数据吗？"

**大师**："这正是复制引擎的精妙之处。它采用**先Blob后Manifest**的顺序，Manifest是原子提交点——如果PushBlob中途失败，Manifest不会推上去，目标Registry上没有引用指向那些孤立的Blob，Registry的垃圾回收（GC）会自动清理。就像你写数据库事务——先插入一堆数据行，最后才COMMIT。COMMIT失败，之前插入的行虽然占着磁盘，但没人能读到，垃圾回收会处理。"

**小胖**：（恍然大悟）"哦！那`DeleteManifest`和`DeleteBlob`接口是为**镜像同步删除**场景准备的？就是源删了镜像，目标也自动删？"

**大师**："对，但那是另一个话题——删除传播（Delete Propagation），Harbor支持但默认关闭，因为风险太高。万一源Registry误删，所有站点都跟着删，那就是灾难了。就像核电站的控制棒——你希望它可以紧急停堆，但绝不允许一个按钮就让所有反应堆同时停。"

### 3.2 步骤一：理解复制引擎核心数据结构

**目标**：掌握复制策略（Policy）、执行（Execution）、任务（Task）三者的关系。

```go
// 复制引擎核心数据结构（基于Harbor v2.9源码结构提取）
// 源码参考: src/controller/replication/model.go src/pkg/reg/model/model.go

package model

import "time"

// Policy 复制策略 —— 定义"什么条件下复制什么资源到哪"
type Policy struct {
    ID            int64     `json:"id"`
    Name          string    `json:"name"`           // 策略名称，如 "sync-to-shanghai"
    Description   string    `json:"description"`    // 策略描述
    SrcRegistry   *Registry `json:"src_registry"`   // 源Registry（默认为本地Harbor）
    DestRegistry  *Registry `json:"dest_registry"`  // 目标Registry
    DestNamespace string    `json:"dest_namespace"` // 目标命名空间映射
    Trigger       *Trigger  `json:"trigger"`        // 触发器配置
    Filters       []*Filter `json:"filters"`        // 资源过滤器
    Enabled       bool      `json:"enabled"`        // 是否启用
    Deletion      bool      `json:"deletion"`       // 是否同步删除
    Override      bool      `json:"override"`       // 是否覆盖已有镜像
    CreationTime  time.Time `json:"creation_time"`
    UpdateTime    time.Time `json:"update_time"`
}

// Trigger 触发器配置
type Trigger struct {
    Type            string           `json:"type"`             // manual/scheduled/event_based
    TriggerSettings *TriggerSettings `json:"trigger_settings"` // Cron表达式等
}

type TriggerSettings struct {
    Cron string `json:"cron"` // 如 "0 */6 * * *" 每6小时一次
}

// Filter 资源过滤器 —— 决定复制哪些镜像
type Filter struct {
    Type       string `json:"type"`        // resource/tag/label
    Value      string `json:"value"`       // 匹配值
    Decoration string `json:"decoration"`  // matches/excludes/matchesTag
}

// Execution 一次复制执行 —— 由一个Policy触发产生
type Execution struct {
    ID           int64     `json:"id"`
    PolicyID     int64     `json:"policy_id"`
    Status       string    `json:"status"`       // InProgress/Succeed/Failed/Stopped
    Trigger      string    `json:"trigger"`      // 记录触发方式
    StatusText   string    `json:"status_text"`  // 失败原因
    StartTime    time.Time `json:"start_time"`
    EndTime      time.Time `json:"end_time"`
    Succeeded    int64     `json:"succeeded"`    // 成功task数
    Failed       int64     `json:"failed"`       // 失败task数
    InProgress   int64     `json:"in_progress"`  // 进行中task数
    Stopped      int64     `json:"stopped"`      // 已停止task数
}

// Task 一次复制任务 —— 针对单个Artifact的复制操作
type Task struct {
    ID            int64     `json:"id"`
    ExecutionID   int64     `json:"execution_id"`
    SourceResource *Resource `json:"src_resource"` // 源制品
    DestResource   *Resource `json:"dst_resource"` // 目标制品
    Status        string    `json:"status"`       // Pending/Running/Succeed/Failed/Stopped
    StatusCode    int       `json:"status_code"`  // HTTP状态码
    StartTime     time.Time `json:"start_time"`
    EndTime       time.Time `json:"end_time"`
}

// Resource 制品资源描述
type Resource struct {
    Type         string `json:"type"`          // artifact/image/chart
    Repository   *Repository `json:"repository"`
    Digest       string `json:"digest"`
    Tag          string `json:"tag"`
    Labels       []string `json:"labels"`
    Deleted      bool   `json:"deleted"`
}

type Repository struct {
    Name       string `json:"name"`
    Namespace  string `json:"namespace"`
}

// Registry 目标Registry信息
type Registry struct {
    ID          int64              `json:"id"`
    Name        string             `json:"name"`
    Type        string             `json:"type"`        // harbor/docker-hub/aws-ecr/tencent-tcr
    URL         string             `json:"url"`
    Credential  *Credential        `json:"credential"`
    Insecure    bool               `json:"insecure"`    // 是否跳过TLS验证
    Description string             `json:"description"`
}

type Credential struct {
    AccessKey    string `json:"access_key"`
    AccessSecret string `json:"access_secret"`
    Type         string `json:"type"` // basic/secret/ecr
}
```

### 3.3 步骤二：实现TCR Registry Adapter

**目标**：开发腾讯云TCR的Registry适配器，实现Adapter/ArtifactRegistry/BlobRegistry三个核心接口。

```go

import (
    "crypto/hmac"
    "crypto/sha256"

const (
    RegistryTypeTCR = "tencent-tcr"

func init() {
    // 注册到全局Adaptor工厂

type Adapter struct {

// PullManifest 从TCR拉取Manifest
func (a *Adapter) PullManifest(repository, reference string) (*model.Manifest, error) {
    manifestURL := fmt.Sprintf("%s/v2/%s/manifests/%s", a.registry.URL, repository, reference)

    req, err := http.NewRequest("GET", manifestURL, nil)
    if err != nil {
        return nil, err
    }
    a.injectAuth(req)

    // 支持多种Manifest格式
    req.Header.Set("Accept",
        "application/vnd.docker.distribution.manifest.v2+json, "+
        "application/vnd.oci.image.manifest.v1+json, "+
        "application/vnd.docker.distribution.manifest.list.v2+json, "+
        "application/vnd.oci.image.index.v1+json")

    resp, err := a.client.Do(req)
    if err != nil {
        return nil, fmt.Errorf("pull manifest from tcr: %w", err)
    }
    defer resp.Body.Close()

    if resp.StatusCode != http.StatusOK {
        return nil, fmt.Errorf("pull manifest failed: tcr returned %d", resp.StatusCode)
    }

    body, _ := io.ReadAll(resp.Body)
    return &model.Manifest{
        ContentType: resp.Header.Get("Content-Type"),
        Content:     body,
        Digest:      resp.Header.Get("Docker-Content-Digest"),
    }, nil
}

// PushManifest 推送Manifest到TCR
func (a *Adapter) PushManifest(repository, reference string, manifest *model.Manifest) error {
    manifestURL := fmt.Sprintf("%s/v2/%s/manifests/%s", a.registry.URL, repository, reference)

    req, err := http.NewRequest("PUT", manifestURL, nil)
    if err != nil {
        return err
    }
    a.injectAuth(req)
    req.Header.Set("Content-Type", manifest.ContentType)
    req.Body = io.NopCloser(io.Reader(io.NopCloser(
        io.NopCloser(io.NopCloser(io.NopCloser(
            io.NopCloser(io.NopCloser(nil)))))))) // 实际应使用bytes.NewReader(manifest.Content)
    // 修正：使用bytes.NewReader(manifest.Content)

    resp, err := a.client.Do(req)
    if err != nil {
        return fmt.Errorf("push manifest to tcr: %w", err)
    }
    defer resp.Body.Close()

    if resp.StatusCode != http.StatusCreated && resp.StatusCode != http.StatusOK {
        return fmt.Errorf("push manifest to tcr failed: status %d", resp.StatusCode)
    }
    return nil
}

// DeleteManifest 从TCR删除Manifest
func (a *Adapter) DeleteManifest(repository, reference string) error {
    // 先获取Manifest Digest
    manifest, err := a.PullManifest(repository, reference)
    if err != nil {
        return err
    }

    deleteURL := fmt.Sprintf("%s/v2/%s/manifests/%s", a.registry.URL, repository, manifest.Digest)
    req, _ := http.NewRequest("DELETE", deleteURL, nil)
    a.injectAuth(req)

    resp, err := a.client.Do(req)
    if err != nil {
        return fmt.Errorf("delete manifest from tcr: %w", err)
    }
    defer resp.Body.Close()

    if resp.StatusCode != http.StatusAccepted && resp.StatusCode != http.StatusOK {
        return fmt.Errorf("delete manifest failed: tcr returned %d", resp.StatusCode)
    }
    return nil
}

// ============== BlobRegistry接口实现 ==============

// PushBlob 推送一个Blob层到TCR
func (a *Adapter) PushBlob(repository, digest string, size int64, reader io.Reader) error {
    // 步骤1：检查Blob是否已存在
    exist, err := a.BlobExist(repository, digest)
    if err != nil {
        return fmt.Errorf("check blob existence: %w", err)
    }
    if exist {
        return nil // 去重：目标已有该层，跳过上传
    }

    // 步骤2：初始化分块上传会话（TCR要求分块上传）
    initURL := fmt.Sprintf("%s/v2/%s/blobs/uploads/", a.registry.URL, repository)
    req, _ := http.NewRequest("POST", initURL, nil)
    a.injectAuth(req)
    req.Header.Set("Content-Length", "0")

    resp, err := a.client.Do(req)
    if err != nil {
        return fmt.Errorf("initiate blob upload: %w", err)
    }
    defer resp.Body.Close()

    if resp.StatusCode != http.StatusAccepted {
        return fmt.Errorf("init upload failed: tcr returned %d", resp.StatusCode)
    }
    uploadURL := resp.Header.Get("Location")

    // 步骤3：分块上传Blob数据（TCR推荐每块5MB-10MB）
    chunkSize := 5 * 1024 * 1024 // 5MB
    buf := make([]byte, chunkSize)
    var uploaded int64

    for uploaded < size {
        n, readErr := io.ReadFull(reader, buf)
        if readErr != nil && readErr != io.ErrUnexpectedEOF && readErr != io.EOF {
            return fmt.Errorf("read blob data: %w", readErr)
        }

        chunkStart := uploaded
        chunkEnd := uploaded + int64(n)

        rangeHeader := fmt.Sprintf("%d-%d", chunkStart, chunkEnd-1)
        req, _ := http.NewRequest("PATCH", uploadURL, io.NopCloser(
            io.LimitReader(reader, int64(n))))
        a.injectAuth(req)
        req.Header.Set("Content-Type", "application/octet-stream")
        req.Header.Set("Content-Range", rangeHeader)
        req.Header.Set("Content-Length", fmt.Sprintf("%d", n))

        resp, err := a.client.Do(req)
        if err != nil {
            return fmt.Errorf("patch blob chunk: %w", err)
        }
        resp.Body.Close()

        if resp.StatusCode != http.StatusAccepted {
            return fmt.Errorf("patch blob failed: tcr returned %d", resp.StatusCode)
        }
        uploadURL = resp.Header.Get("Location")
        uploaded += int64(n)

        if readErr == io.EOF {
            break
        }
    }

    // 步骤4：最终确认（PUT完成上传 + digest验证）
    completeURL := fmt.Sprintf("%s&digest=%s", uploadURL, digest)
    req, _ = http.NewRequest("PUT", completeURL, nil)
    a.injectAuth(req)
    req.Header.Set("Content-Length", "0")

    resp, err = a.client.Do(req)
    if err != nil {
        return fmt.Errorf("complete blob upload: %w", err)
    }
    defer resp.Body.Close()

    if resp.StatusCode != http.StatusCreated {
        return fmt.Errorf("complete blob upload failed: tcr returned %d", resp.StatusCode)
    }
    return nil
}

// PushBlobSimple 简单单次PUT上传（适用于小Blob或支持一次性PUT的Registry）
func (a *Adapter) PushBlobSimple(repository, digest string, data []byte) error {
    uploadURL := fmt.Sprintf("%s/v2/%s/blobs/uploads/?digest=%s", a.registry.URL, repository, digest)

    req, err := http.NewRequest("POST", uploadURL, nil)
    if err != nil {
        return err
    }
    a.injectAuth(req)
    req.Header.Set("Content-Length", "0")

    // 先初始化上传
    resp, err := a.client.Do(req)
    if err != nil {
        return err
    }
    uploadLocation := resp.Header.Get("Location")
    resp.Body.Close()

    // PUT完成上传
    completeURL := fmt.Sprintf("%s&digest=%s", uploadLocation, digest)
    req, _ = http.NewRequest("PUT", completeURL, nil)
    a.injectAuth(req)
    req.Header.Set("Content-Type", "application/octet-stream")
    req.Header.Set("Content-Length", fmt.Sprintf("%d", len(data)))
    req.Body = io.NopCloser(io.NopCloser(nil)) // 实际应使用bytes.NewReader(data)

    resp, err = a.client.Do(req)
    if err != nil {
        return fmt.Errorf("push blob: %w", err)
    }
    defer resp.Body.Close()
    return nil
}

// PullBlob 从TCR拉取一个Blob层
func (a *Adapter) PullBlob(repository, digest string) (int64, io.ReadCloser, error) {
    blobURL := fmt.Sprintf("%s/v2/%s/blobs/%s", a.registry.URL, repository, digest)

    req, _ := http.NewRequest("GET", blobURL, nil)
    a.injectAuth(req)

    resp, err := a.client.Do(req)
    if err != nil {
        return 0, nil, fmt.Errorf("pull blob from tcr: %w", err)
    }

    if resp.StatusCode != http.StatusOK {
        resp.Body.Close()
        return 0, nil, fmt.Errorf("pull blob failed: tcr returned %d", resp.StatusCode)
    }

    return resp.ContentLength, resp.Body, nil
}

// BlobExist 检查Blob是否存在于TCR
func (a *Adapter) BlobExist(repository, digest string) (bool, error) {
    checkURL := fmt.Sprintf("%s/v2/%s/blobs/%s", a.registry.URL, repository, digest)

    req, _ := http.NewRequest("HEAD", checkURL, nil)
    a.injectAuth(req)

    resp, err := a.client.Do(req)
    if err != nil {
        return false, err
    }
    defer resp.Body.Close()

    return resp.StatusCode == http.StatusOK, nil
}

// DeleteBlob 从TCR删除Blob
func (a *Adapter) DeleteBlob(repository, digest string) error {
    deleteURL := fmt.Sprintf("%s/v2/%s/blobs/%s", a.registry.URL, repository, digest)
    req, _ := http.NewRequest("DELETE", deleteURL, nil)
    a.injectAuth(req)

    resp, err := a.client.Do(req)
    if err != nil {
        return err
    }
    defer resp.Body.Close()
    return nil
}

// ============== 腾讯云TCR特定认证逻辑 ==============

func (a *Adapter) refreshToken() error {
    // TCR API签名逻辑（基于腾讯云API v3签名规范）
    // 实际实现需SecretID + SecretKey + 时间戳生成签名
    timestamp := time.Now().Unix()
    service := "tcr"
    host := fmt.Sprintf("%s.%s", a.registry.URL, TCRDomain)

    signStr := fmt.Sprintf("tc3_request\n%s\n%s\n%s",
        time.Unix(timestamp, 0).UTC().Format("2006-01-02"),
        service, host)
    // HMAC-SHA256签名计算
    _ = hmac.New(sha256.New, []byte(a.credential.SecretKey))
    _ = signStr // 实际使用HMAC计算签名

    return nil
}

func (a *Adapter) injectAuth(req *http.Request) {
    // 注入临时Token到Authorization头
    req.Header.Set("Authorization", "Bearer "+a.credential.Token)
}
```

### 3.4 步骤三：理解复制执行流程

**目标**：梳理Flow Controller如何串联Policy → Execution → Task。

```go
// 复制执行流程核心逻辑（基于Harbor v2.9源码抽象）
// 源码参考: src/controller/replication/flow/controller.go

package flow

// StartReplication 启动一次复制执行
func StartReplication(policy *model.Policy, resource *model.Resource, trigger string) (*model.Execution, error) {
    exec := &model.Execution{
        PolicyID: policy.ID,
        Status:   "InProgress",
        Trigger:  trigger,
        StartTime: time.Now(),
    }
    // 持久化Execution记录到PostgreSQL

    // 步骤1：Filter过滤——匹配需要复制的Artifacts
    candidates, err := filterCandidates(policy, resource)
    if err != nil {
        exec.Status = "Failed"
        exec.StatusText = fmt.Sprintf("filter failed: %v", err)
        return exec, err
    }

    // 步骤2：创建Tasks（每个Candidate一个Task）
    for _, candidate := range candidates {
        task := &model.Task{
            ExecutionID:   exec.ID,
            SourceResource: candidate,
            Status:        "Pending",
        }
        // 持久化Task记录

        // 步骤3：提交到JobService异步执行
        submitReplicationJob(task, policy.DestRegistry)
    }

    return exec, nil
}

func filterCandidates(policy *model.Policy, resource *model.Resource) ([]*model.Resource, error) {
    var candidates []*model.Resource

    for _, f := range policy.Filters {
        switch f.Type {
        case "resource":
            // 按仓库名匹配
            if matchResource(f.Value, resource.Repository.Name) {
                candidates = append(candidates, resource)
            }
        case "tag":
            // 按Tag正则匹配
            if matchTag(f.Value, resource.Tag) {
                candidates = append(candidates, resource)
            }
        case "label":
            // 按Label匹配
            if hasLabel(f.Value, resource.Labels) {
                candidates = append(candidates, resource)
            }
        }
    }
    return candidates, nil
}

// ReplicationWorker JobService中的复制执行体
func ReplicationWorker(task *model.Task) error {
    // 获取源Registry的Adapter
    srcAdapter, _ := adapter.GetFactory(task.SourceResource.Repository.Namespace)

    // 获取目标Registry的Adapter
    dstAdapter, _ := adapter.GetFactory(task.DestResource.Repository.Namespace)

    // 步骤1：Pull Manifest
    manifest, err := srcAdapter.PullManifest(
        task.SourceResource.Repository.Name,
        task.SourceResource.Tag)
    if err != nil {
        return fmt.Errorf("pull manifest: %w", err)
    }

    // 步骤2：对所有Blob层执行 Pull → Push
    for _, layer := range manifest.Layers {
        size, reader, err := srcAdapter.PullBlob(
            task.SourceResource.Repository.Name, layer.Digest)
        if err != nil {
            return fmt.Errorf("pull blob %s: %w", layer.Digest, err)
        }
        // 复用srcAdapter返回的reader直接推给dstAdapter
        if err := dstAdapter.PushBlob(
            task.DestResource.Repository.Name, layer.Digest, size, reader); err != nil {
            reader.Close()
            return fmt.Errorf("push blob %s: %w", layer.Digest, err)
        }
        reader.Close()
    }

    // 步骤3：Push Manifest（原子提交）
    if err := dstAdapter.PushManifest(
        task.DestResource.Repository.Name,
        task.DestResource.Tag, manifest); err != nil {
        return fmt.Errorf("push manifest: %w", err)
    }

    return nil
}
```

### 3.5 常见陷阱与解决方案

**陷阱一：自定义Adaptor的`init()`注册不生效**

*现象*：开发了TCR Adaptor并编译进Harbor Core后，创建复制策略时报`registry type 'tencent-tcr' not supported`。

*根因*：Go的`init()`函数只在包被显式导入时执行。如果不通过`import _ "path/to/adapter/tcr"`触发，`RegisterFactory`永远不会被调用。

*解决方案*：在Harbor Core的入口文件`src/cmd/core/main.go`中添加匿名导入：
```go
import _ "github.com/goharbor/harbor/src/pkg/reg/adapter/tcr"
```

**陷阱二：PushBlob未调用BlobExist导致重复上传**

*现象*：被复制的镜像90%的Blob层在目标Registry已存在，但复制引擎仍然逐层重新上传，复制效率极低。

*根因*：Adaptor的`PushBlob`实现中跳过了去重检查——没有在开始上传前调用`BlobExist`判断目标是否已有该层。复制引擎依赖Adaptor层面的去重，不会在Flow层做额外校验。

*解决方案*：在`PushBlob`函数开头插入去重逻辑：
```go
exist, err := a.BlobExist(repository, digest)
if exist { return nil }
```

**陷阱三：Manifest的Content-Type不完整导致目标Registry拒绝**

*现象*：PushManifest成功返回201，但目标Registry上的镜像无法被`docker pull`，报错`manifest unknown`。

*根因*：目标Registry（特别是OCI兼容的Registry）在接收Manifest时不仅存储Digest，还存储Content-Type。如果PullManifest返回的`ContentType`字段为空或使用了非标准值，PushManifest时目标Registry无法正确解析Manifest格式。

*解决方案*：确保`PullManifest`从响应头中正确提取并保留`Content-Type`，并在`PushManifest`中设置相同的值：
```go
// PullManifest中
manifest.ContentType = resp.Header.Get("Content-Type")

// PushManifest中
req.Header.Set("Content-Type", manifest.ContentType)
```

**陷阱四：事件驱动复制在高并发下丢失事件**

*现象*：短时间内push大量镜像后，部分镜像没有触发复制。

*根因*：事件通道的缓冲区默认约1000。如果push速度超过JobService消费速度，超出缓冲区的事件被丢弃。

*解决方案*：
1. 调大事件通道缓冲区（需修改源码常量）
2. 配合定时复制策略作为兜底（每30分钟执行一次，覆盖遗漏的增量数据）

## 4 项目总结

### 4.1 复制引擎架构分层表

| 层级 | 组件 | 职责 | 关键文件 |
|------|------|------|---------|
| 策略层 | Policy Manager | 管理复制策略CRUD，处理触发信号 | `src/controller/replication/policy.go` |
| 过滤层 | Filter Chain | 按资源类型/标签/名称过滤候选Artifact | `src/pkg/reg/filter/` |
| 执行层 | Execution Manager | 创建Execution，拆分为Tasks，提交到JobService | `src/controller/replication/execution.go` |
| 流程层 | Flow Controller | 控制Pull→Push流程，异常回滚 | `src/controller/replication/flow/` |
| 适配层 | Registry Adapter | 与目标Registry通信，实现Pull/Push/Delete | `src/pkg/reg/adapter/` |
| 调度层 | JobService Worker | 异步执行复制Job，管理重试和超时 | `src/jobservice/job/impl/replication/` |

### 4.2 适用场景

1. **多数据中心容灾同步**——主站点Harbor的镜像自动复制到灾备站点，确保灾难时可快速恢复。
2. **混合云镜像分发**——私有云Harbor开发镜像→推送到公有云容器服务（ACK/TKE/EKS），一次构建全域部署。
3. **跨组织镜像共享**——子公司Harbor通过复制策略从母公司Harbor拉取基础镜像，保持版本一致性。
4. **DockerHub加速与缓冲**——配置从DockerHub定时拉取常用镜像到本地Harbor，避免每次构建都访问公网。
5. **镜像合规迁移**——从老旧Registry迁移到Harbor时，通过复制策略批量同步历史镜像并保留标签结构。

### 4.3 不适用场景

1. **实时数据同步**——复制引擎的最小触发间隔为分钟级，不适合需要秒级同步的场景（如CI/CD即时镜像同步）。
2. **大文件非镜像制品的同步**——复制引擎专为OCI/Docker镜像Blob设计，Helm Chart等元数据层面的同步走独立流程。

### 4.4 注意事项

1. **网络带宽规划**：跨区域复制消耗大量带宽，建议配置QoS限速或错峰执行（如Cron设为凌晨2点）。
2. **目标Registry存储配额**：复制前确认目标Registry有足够存储空间，否则PushBlob在完成前由于目标磁盘满而失败，已上传的Blob成为孤立数据。
3. **删除传播风险极高**：`deletion: true`意味着源删目标也删，误删恢复成本巨大。除非有严格的合规要求，否则保持默认关闭。
4. **TLS证书管理**：跨Registry通信依赖TLS，证书过期会导致所有复制任务失败——建议设置证书到期前3个月告警。
5. **复制策略并发度**：多条复制策略同时运行时，JobService的Worker数量是固定的（默认10），提交过多Task会导致队列堆积。

### 4.5 常见陷阱速查表

| 陷阱 | 现象 | 根因 | 解决 |
|------|------|------|------|
| init()未注册 | type not supported | 包未被匿名导入 | main.go加import _ |
| 未调用BlobExist | 重复上传已存在Blob | PushBlob跳过去重 | 开头加exist检查 |
| Content-Type丢失 | pull报manifest unknown | Push时MIME type为空 | PullManifest保留Content-Type |
| 事件通道溢出 | 部分镜像未触发复制 | 缓冲区满后丢弃 | 增大缓冲+定时兜底 |
| PushBlob未分块 | TCR返回400 | 云Registry要求分块 | 实现POST→PATCH→PUT流程 |

### 4.6 深度思考

**问题一**：假设需要在复制过程中实现"条件过滤"——只有当源镜像的漏洞扫描报告Severity低于`Medium`时才允许复制到生产Registry，高于Medium的只复制到预发布Registry。这个逻辑应该在复制引擎的哪一层实现？是Filter层（在创建Task前过滤）、Flow层（在PushManifest前检查）、还是Policy层（新增策略字段）？

**问题二**：复制引擎的Adapter接口目前假设"源Registry和目标Registry共享相同的制品格式"。如果需要实现格式转换——比如源是Docker V2 Manifest，推送到目标时自动转换为OCI格式——现有的`PullManifest`和`PushManifest`接口是否需要修改？或者可以通过一个"中间Adapter"（格式转换层）在Pull和Push之间做Transcode？
