# 第12章：垃圾回收（GC）机制实战

## 1 项目背景

2024年6月的一个午夜，晓数科技（虚构）的SRE工程师陈磊被PagerDuty的告警电话炸醒——Harbor服务器磁盘使用率飙到了97%，仅剩12GB可用空间。而Harbor所在的挂载卷总容量是1TB——也就是说，976GB的空间已被占满。按照当前的增长速度，预计4小时后磁盘将100%写满，届时所有正在push的CI任务将全部失败，正在上线的发布流程也会被阻断。

陈磊连夜赶到数据中心，开始了漫长而痛苦的问题排查。

**痛点一：删除标签不等于释放空间——存储账目"对不上"。** 陈磊首先想到的清理手段很简单：删除旧标签。流水线上的CI每周产生约200个构建标签，其中大部分是中间测试版本——累计起来，过去半年应该产生了5000+个废弃标签。他在Harbor Portal中花了2小时手动删除了约800个最早期的 `ci-build-*` 标签，每个都确认"删除成功"。然后他打开终端，敲下 `du -sh /data/registry/`——结果让他愣住了：空间使用量从976GB变成了976GB——**纹丝不动**。"我明明删了800个标签，空间没释放？"陈磊反复确认了三次 `du` 的输出，结果都一样。这时候他才隐约意识到：Harbor的"删除"和他理解的"删除"不是一回事。

**痛点二：GC等于"停机维护"——业务中断代价高昂。** 在翻阅Harbor文档后，陈磊终于理解了：他需要执行"垃圾回收"（Garbage Collection, GC）来真正释放那些被删除标签但仍在磁盘上的Blob文件。于是在工作日下午3:00——业务高峰期——他手动触发了GC。1分钟后，Jenkins群里开始炸锅：50+条构建流水线同时报错 `Error: Harbor is in read-only mode`。原来GC执行期间Harbor会自动切换到只读模式——所有push/delete操作被全局锁定。陈磊惊恐地发现：他触发的GC预计要运行2-3小时，而这3小时内整个公司的CI/CD全部瘫痪。他拼命搜索"如何取消Harbor GC"，得到的答案只有冰冷的一句：**GC一旦开始，无法中途取消**。

**痛点三：长期未GC导致执行耗时失控。** 第一次GC整整执行了**5小时47分钟**。在这将近6个小时里，Harbor处于只读模式，研发部门的47条CI流水线全部停顿。凌晨1点开始，凌晨6点47分才结束。陈磊在凌晨3点的时候收到CTO的微信："为什么今晚的上线失败了？"他只能回复："Harbor在执行维护……预计还有几小时……"——然后整个人陷入崩溃。

**痛点四：GC释放的空间远超预期——隐藏的"垃圾债"。** GC完成后的结果显示：释放了**523GB**空间——976GB → 453GB，直接释放了超过一半的存储。陈磊非常困惑："我们才删了800个标签，怎么释放了523GB？"经过排查发现：过去6个月内，虽然没有人执行过GC，但**各种操作一直在产生"孤儿Blob"**——包括：（1）被覆盖的标签（push同标签的不同版本时，旧版本的Blob失去了引用）；（2）失败的push（镜像上传了一半后网络中断，部分Blob已被写入磁盘）；（3）标签保留策略删除的标签（策略自动删除了旧标签的数据库记录，但Blob文件留在磁盘上）。这些"垃圾"日积月累，6个月没有GC的结果就是：453GB的数据中，有523GB是垃圾——**实际有效数据只有430GB，而垃圾比有效数据还多**。

**痛点五：GC不可中断——但可以"意外死亡"。** 后续调查还揭示了一个更惊险的细节：GC执行到第4小时时，因为磁盘写满了GC过程中的临时标记文件（Mark Set），GC进程自己崩溃了——留下了一个"半清理"的状态：部分垃圾Blob已被物理删除，但Mark Set文件残留在磁盘上占了额外的37GB空间，且Harbor被"卡"在了只读模式。运维不得不在凌晨5点拨通了Harbor社区专家的电话，通过手动删除Mark Set文件和重启Registryctl才恢复读写模式。

本章将从原理到实践，深入剖析Harbor垃圾回收的完整机制——包括Registry存储模型、Mark & Sweep两阶段原理、定时GC配置策略、以及生产环境的避坑指南。

## 2 项目设计——剧本式交锋对话

**场景：晓数科技"GC事故"事后复盘会议——运维陈磊、平台架构师大师、安全工程师小白、开发小胖围坐着，陈磊的脸上还挂着凌晨通宵加班后没能洗掉的黑眼圈。**

**小胖**（端着一杯超大杯的冰美式，一脸困惑）："我有个小白问题——GC不就是'垃圾回收'吗？我在Windows上删了文件，清空回收站，空间就回来了。为啥Harbor的GC这么折腾？删个标签还得专门执行一次GC才行？这不是脱裤子放屁吗？"

**大师**（站起身，在白板上画了两层结构）："小胖，你的困惑来自一个认知误区——你把'Harbor的数据库'和'磁盘上的文件'当成同一个东西了。实际上它们是两个完全独立的世界。"

他在白板上画了一个巨大的分层图：

```
  ┌──────────────────────────────────────────┐
  │          PostgreSQL 数据库（元数据层）        │
  │  ┌───────────┐  ┌──────────┐  ┌────────┐ │
  │  │ 标签表      │  │ Artifact │  │ Blob表 │ │
  │  │tag: v1.0   │  │ 引用关系  │  │ sha256 │ │
  │  │ deleted    │  │          │  │ :abc.. │ │
  │  └───────────┘  └──────────┘  └────────┘ │
  │  当你删除标签时 → 只在这里标记 "deleted"     │
  └──────────────────────────────────────────┘
                        ↓
         【删除标签 ≠ 删除文件！】
                        ↓
  ┌──────────────────────────────────────────┐
  │      物理存储（文件系统层）                    │
  │  /data/registry/docker/registry/v2/       │
  │    └── blobs/sha256/                      │
  │         ├── abc123... (2.3 MB) ← 仍然在这！  │
  │         ├── def456... (1.8 MB) ← 仍然在这！  │
  │         └── ghi789... (0.5 MB) ← 仍然在这！  │
  │                                            │
  │  GC的作用 = 扫描文件，对比数据库，发现      │
  │             "数据库里没有引用的文件" → 删除   │
  └──────────────────────────────────────────┘
```

"当你删除一个标签时，Harbor只做了一件事：在PostgreSQL的 `artifact_tag` 表中把这条标签记录的 `deleted` 字段从 `false` 改为 `true`。这叫做**元数据删除（Logical Delete）**。那些构成镜像的实际文件——被称作Blob（Binary Large Object），以SHA256摘要命名的压缩层文件——仍然完整地存放在 `/data/registry/docker/registry/v2/blobs/sha256/` 目录下。"

"GC的工作是：扫描磁盘上**所有的**Blob文件，逐一去数据库中查询——'这个Blob还有没有标签在引用它？'如果有引用 → 保留；没有引用 → 它就是垃圾 → 删除。这个过程叫做**物理删除（Physical Delete）**。"

**技术映射**：Harbor的GC机制继承自Docker Distribution（Registry v2）的垃圾回收规范。GC由两个组件协作完成：`registryctl`（控制器）负责调度GC流程和调用Mark/Sweep API；`registry`（Registry核心）负责实际的Blob扫描和物理删除。整个流程分为**Mark阶段**（标记可删除的Blob）和**Sweep阶段**（物理删除被标记的Blob）。

**小白**（若有所思地转着笔）："那GC期间为什么要设置为只读模式？这个只读锁具体锁的是什么？"

**大师**（拿起另一支马克笔，在分层图的中间加了一个大锁图标）："只读锁保护的是**数据一致性**。我来用一个生活场景解释——"

"想象你是一个图书馆管理员。你的工作是整理书架——检查每本书是否还有人借阅（数据引用）。如果没有人借阅了，你就把书下架回收（物理删除）。但是——你正在检查的时候，有读者过来还了一本刚下架的书，或者借走了一本你正要检查的书——那你的整个清查过程就会乱套。"

"GC的只读锁就是'整理期间暂时关闭图书馆'——确保在Mark阶段扫描的Blob列表和Sweep阶段删除的Blob之间，不会有新的Blob写入，也不会有Blob的引用关系发生变化。这样GC拿到的是一个**一致性快照**——从开始到结束，数据世界是静止的。"

**技术映射**：只读模式通过Harbor Core的 `PUT /api/v2.0/system/readonly` 接口控制。GC开始时，JobService调用此接口将 `read_only` 设置为 `true`，Core的所有写操作（push、delete、修改配置等）都会被拦截并返回503错误。GC结束后，JobService再次调用此接口将 `read_only` 设为 `false`。如果GC进程崩溃而未正常恢复读写模式，管理员需要通过API手动恢复：`curl -X PUT ... -d '{"read_only":false}'`。

**小胖**（举手）："那我就不明白了——为什么一定要在白天不能执行GC？你看我们凌晨3点执行不就好了？周末凌晨也行啊——正好没人上班。"

**大师**（点头）："方向很对——但你遗漏了一个问题：GC需要多长时间？"

他翻出数据：

| 存储量 | Blob总数 | 距上次GC时间 | Mark阶段耗时 | Sweep阶段耗时 | 总计 |
|--------|---------|------------|------------|------------|------|
| 200GB | 3万 | 1周 | 8分钟 | 5分钟 | **13分钟** |
| 500GB | 8万 | 1个月 | 25分钟 | 18分钟 | **43分钟** |
| 1TB | 20万 | 6个月 | 2小时10分钟 | 1小时35分钟 | **3小时45分钟** |
| 2TB | 50万 | 12个月 | 6小时 | 4小时 | **10小时+** |

"看到了吗？GC的执行时间不取决于你要删除多少垃圾——而取决于**磁盘上有多少Blob需要扫描**。如果你每周执行一次GC，每次只要15-30分钟——放在凌晨3:00的维护窗口完全没问题。但如果你半年才执行一次，GC要跑5-6小时——从凌晨3:00跑到早上8:30，直接跑进上班时间。"

**技术映射**：GC的Mark阶段需要遍历Registry存储中所有Blob目录——对于每个Blob，判断其在数据库中是否存在活跃引用。这个过程的时间复杂度是O(n)，n是Blob总数。Sweep阶段只处理被标记的Blob，时间复杂度是O(m)，m是垃圾Blob数。因此GC的总耗时主要由Mark阶段（O(n)）决定——**半年不GC意味着n膨胀了6倍**。

**小白**（翻着Harbor的源码注释）："我读了Registryctl的代码，GC似乎不支持'选择性清理'——不能只清理某个项目的垃圾。这有什么考量吗？"

**大师**："这恰恰是GC设计中最容易被人误解的地方。确实——Harbor的GC总是全量的（Full GC），它扫描的是整个Registry存储，而不是某个特定项目的Blob。这背后的原因在于：**Blob是可以跨项目共享的**。"

他在白板上画了一个实际的例子：

```
项目 A: order-platform/java-service:v1.0
        Layer sha256:abc ← 这个Blob被引用1次

项目 B: payment-platform/java-service:v1.0
        Layer sha256:abc ← 同样的Blob！因为用了同样的基础镜像层！

如果GC"只清理项目A"：
  GC看到项目A中sha256:abc没有引用 → 删除sha256:abc
  → 项目B的镜像立即损坏！（因为项目B也在用这个Blob）
```

"这就是为什么GC必须全局扫描——它要汇总**跨所有项目**的引用关系，才能判断一个Blob是否真的是'孤儿'。如果允许局部GC，可能导致'幽灵Blob'——A项目删了，B项目的镜像拉取失败。"

**技术映射**：Docker镜像的分层存储模型天然鼓励Blob共享——两个项目如果使用同一个基础镜像层（如 `FROM openjdk:17`），它们引用的Blob在磁盘上只有一份。GC必须通过整个Registry的全局引用计数来判断Blob是否可以被安全删除。这个引用计数存储在PostgreSQL的 `blob` 表中。

**小胖**（突然拍桌子）："那如果我磁盘真的满了怎么办？！比如现在——97%了——GC还要跑6个小时——那4小时后磁盘100%写满，所有push失败，业务全挂！GC又不能中途取消——这不是死循环吗？"

**大师**（严肃地点点头）："你问到了GC在生产环境中最致命的场景——**"磁盘满了 → 触发GC → GC期间需要写临时文件 → 磁盘更满 → GC崩溃 → Harbor卡在只读模式"**的死循环。"

他走到白板前，画了一条时间线：

```
紧急处理流程（磁盘97%）：

Step 1 [立即]: 通过手动清理非核心文件争取时间
    → 删除 /data/registry/docker/registry/v2/blobs/ 下已被数据库标记为 deleted 但仍存留的Blob
    → 这个操作有风险，但可以快速释放20-30%空间

Step 2 [30分钟内]: 扩容存储
    → 挂载一个临时NFS/新磁盘到 /data 目录
    → 或者使用LVM在线扩容

Step 3 [存储充足后]: 执行GC
    → 确保有至少20%的磁盘余量用于GC临时文件
```

**技术映射**：GC执行过程中需要在Registry存储目录下生成一个 `_gc_mark_set` 的临时标记文件（记录Mark阶段已扫描的Blob列表）。这个文件的大小与Blob总数成正比——通常占总存储量的2-5%。如果磁盘没有足够余量容纳这个临时文件，GC会在Mark阶段失败。

**小白**："那从运维的最佳实践来看——GC到底应该怎么配置？手动触发还是定时自动？频率多高合适？"

**大师**："三个原则——"

他在白板上写下：

```
GC配置三原则：

1. 定时自动 + 手动应急 = 双保险
   定时： 每周日凌晨 3:00 自动执行（Cron: 0 3 * * 0）
   手动： 磁盘使用率超过 80% 时手动触发

2. GC 频率 = f(镜像产生速度)
   高强度CI（每天 > 500次push）: 每天 3:00 AM
   中等强度CI（每天 50-500次push）: 每周日 3:00 AM
   低强度（每天 < 50次push）: 每两周 3:00 AM

3. GC 前检查清单：
   □ 确认当前没有正在 Running 的 GC（API查询）
   □ 确认磁盘剩余空间 > 总Blob大小的 5%
   □ 确认凌晨维护窗口有足够的时长（预估 GC 耗时）
   □ 通知相关团队（CI构建可能在维护窗口内失败）
```

### 3.3 步骤二：模拟产生垃圾——删除标签但不执行GC

**目标：** 通过实际操作理解"删除标签≠释放空间"。

```bash
# 1. 推送10个测试镜像（产生新的Blob）
for i in $(seq 1 10); do
  echo "gc-test v1.$i.0" > test.txt
  docker build -t harbor.xiaoshukeji.com/order-platform/gc-test:v1.$i.0 .
  docker push harbor.xiaoshukeji.com/order-platform/gc-test:v1.$i.0
done

# 记录GC前的存储大小
echo "=== GC前存储状态 ==="
sudo du -sh /data/registry/docker/registry/v2/blobs/
BEFORE_GC_SIZE=$(sudo du -sb /data/registry/docker/registry/v2/blobs/ | awk '{print $1}')
echo "Blobs总大小: $(( BEFORE_GC_SIZE / 1024 / 1024 )) MB"
```

```bash
# 2. 删除所有10个标签（只做元数据删除）
for i in $(seq 1 10); do
  curl -X DELETE -u admin:Harbor12345 \
    "https://harbor.xiaoshukeji.com/api/v2.0/projects/order-platform/repositories/gc-test/artifacts/v1.$i.0"
done

echo "=== 删除标签后（GC前）存储状态 ==="
sudo du -sh /data/registry/docker/registry/v2/blobs/
AFTER_DELETE_SIZE=$(sudo du -sb /data/registry/docker/registry/v2/blobs/ | awk '{print $1}')
echo "Blobs总大小: $(( AFTER_DELETE_SIZE / 1024 / 1024 )) MB"
echo "释放空间: $(( (BEFORE_GC_SIZE - AFTER_DELETE_SIZE) / 1024 / 1024 )) MB"
# 预期：释放空间 ≈ 0 MB（标签删了，Blob文件还在！）

# 3. 验证数据库中的状态变化
docker exec harbor-db psql -U postgres -d registry -c \
  "SELECT status, COUNT(*) FROM blob WHERE status = 'delete' GROUP BY status;"
# 预期：delete 状态的Blob数量增加
```

### 3.4 步骤三：配置定时GC

### 3.9 排坑指南

#### 坑1：GC执行中想紧急取消——无法取消

**现象：** GC在白天被意外触发（或定时GC预估错误），导致Harbor进入只读模式且CI全部阻塞。运维尝试各种方式想要中断GC，但发现**GC一旦开始无法中途取消**。

**根因分析：** Harbor GC的JobService设计为"不可取消"——一旦GC Job被提交到JobService队列并开始执行，没有暴露的API可以中断它。这是由GC的两阶段设计决定的：如果在Sweep阶段中途取消，将导致部分Blob被删除、部分未被删除的不一致状态——可能损坏镜像。

**应对策略（预防为主）：**
```bash
# 预防1：手动触发GC前务必确认没有Running状态的GC
RUNNING=$(curl -s -u admin:Harbor12345 \
  "https://harbor.xiaoshukeji.com/api/v2.0/system/gc" | \
  jq '[.[] | select(.job_status=="running")] | length')
echo "当前运行的GC数: $RUNNING"

# 预防2：在非业务高峰期触发（检查当前时间）
HOUR=$(date +%H)
if [ "$HOUR" -ge 9 ] && [ "$HOUR" -le 18 ]; then
  echo "⚠️  当前是工作时间（9:00-18:00），不建议触发GC"
fi

# 如果真的已经触发了GC且必须中断（最后手段——有数据损坏风险）：
# docker restart registryctl  # 重启控制器进程
# 然后立即通过API恢复读写模式（见步骤六）
# ⚠️ 警告：此操作可能导致部分Blob处于不一致状态，重启后需重新执行GC
```

#### 坑2：GC执行后存储空间反而变大了

**现象：** 执行GC后，`du -sh /data/` 显示的空间使用量比GC前**更多**了——从800GB变成了815GB。

**根因分析：** 三个可能原因：
1. **GC的Mark阶段会写入临时标记文件**（`_gc_mark_set`），这些文件在Sweep阶段才被清理。在Mark完成后、Sweep完成前的短暂窗口期，空间会临时增加。
2. **文件系统的延迟回收**——某些文件系统（如XFS）在文件被删除后不会立即更新 `df`/`du` 的统计信息，而是在后台异步回收。
3. **新的镜像在GC完成后立即被push进了Harbor**——GC释放了空间，但新的push又占用了空间。

**解决方法：**
```bash
# 1. 等待5分钟后重新检查空间（允许文件系统异步回收完成）
sleep 300
sudo du -sh /data/registry/

# 2. 使用df查看文件系统级别的使用情况（而非du的目录统计）
df -h /data

# 3. 检查GC后是否有新的push
docker logs harbor-core --since 10m 2>&1 | grep -c "push blob"
```

#### 坑3：GC报告显示释放空间为0

**现象：** GC执行日志显示 `"GC completed: 0 GB freed"`，但数据库中有大量 `status='delete'` 的Blob记录。

**根因分析：** 有四种常见原因：

1. **标签保留策略未执行。** 标签保留了策略设置（如"只保留最近10个标签"），但策略只是标记了超标标签——实际的删除操作需要由定时任务触发。如果定时任务还没执行，被策略标记的标签的引用关系仍然存在。

2. **`delete` 状态的Blob仍然被其他Artifact引用。** Blob的引用计数可能不是0——另一个项目或另一个标签还在使用同一个Blob。

3. **GC未能正确连接到Registry API。** Registryctl调用Registry的 `/admin/gc` API时可能返回错误（如网络超时、认证失败）。

4. **Blob文件权限问题。** Registry进程没有删除Blob文件的权限。

**排查方法：**
```bash
# 1. 检查Blob引用计数
docker exec harbor-db psql -U postgres -d registry -c \
  "SELECT b.digest, b.status, COUNT(ab.artifact_id) as ref_count
   FROM blob b
   LEFT JOIN artifact_blob ab ON b.id = ab.blob_id
   WHERE b.status = 'delete'
   GROUP BY b.digest, b.status
   HAVING COUNT(ab.artifact_id) > 0
   LIMIT 10;"
# 如果返回结果——说明这些"delete"的Blob仍被引用，GC不会删除它们

# 2. 检查Registryctl是否成功调用了Registry API
docker logs registryctl 2>&1 | grep -i "admin/gc\|error.*gc\|failed.*gc" | tail -20

# 3. 触发标签保留策略的立即执行（如果有配置保留策略）
# 如果上一步确认是标签保留策略导致的，手动触发策略执行
```

#### 坑4：GC后某些镜像拉取失败——"manifest unknown"

**现象：** GC完成后几个小时，某个服务上线时拉取镜像失败，提示 `manifest unknown` 或 `BLOB_UNKNOWN`。检查发现该镜像的某些层文件在磁盘上已经不存在了。

**根因分析（罕见但严重）：** GC的Mark阶段在扫描Blob时，由于某种原因（如并发竞态条件、数据库事务隔离导致的一致性偏差、或Registryctl和Core之间的时钟偏差），遗漏了某个仍有活跃引用的Blob——导致该Blob被错误地标记为"可删除"并在Sweep阶段被物理删除了。

**排查和恢复步骤：**
```bash
# 1. 确认受影响的镜像
AFFECTED_REPO="order-platform/java-service"
AFFECTED_TAG="v1.0.0"

# 2. 查看该Artifact的Blob引用列表
curl -s -u admin:Harbor12345 \
  "https://harbor.xiaoshukeji.com/api/v2.0/projects/order-platform/repositories/java-service/artifacts/$AFFECTED_TAG" | \
  jq '{digest: .digest, references: [.references[]?.child_digest]}'

# 3. 检查这些digest对应的Blob文件是否存在
# 对于每个child_digest：
DIGEST="sha256:abc123def456..."
PREFIX=$(echo "$DIGEST" | sed 's/sha256://' | head -c 2)
DIGEST_ID=$(echo "$DIGEST" | sed 's/sha256://')
BLOB_PATH="/data/registry/docker/registry/v2/blobs/sha256/$PREFIX/$DIGEST_ID/data"

if [ ! -f "$BLOB_PATH" ]; then
  echo "缺失Blob: $DIGEST ← GC错误删除了这个文件"
  echo "恢复方法：重新构建并push该镜像标签"
fi

# 4. 恢复：重新build并push该镜像
# 这是唯一的恢复方式——因为GC已经物理删除了文件，无法恢复
```

## 4 项目总结

### 4.1 Docker Registry存储模型与GC原理总结

| 概念 | 层次 | 说明 | 删除操作的影响 |
|------|------|------|-------------|
| **标签（Tag）** | 最上层 | 人类可读的镜像标识（如 `v1.0.0`、`latest`） | 删除标签 → 只移除该标签指向Artifact的引用 |
| **Artifact（制品）** | 中间层 | 一个镜像的具体版本（由SHA256 digest唯一标识） | 删除Artifact → 移除该版本对Blob的引用（如果该Artifact是最后一个引用某Blob的，则该Blob成为"孤儿"） |
| **Blob（二进制大对象）** | 最底层 | 镜像的实际文件层（压缩后的文件系统层），以SHA256摘要命名存储在磁盘上 | 删除Blob → 物理删除磁盘上的文件——由GC执行 |
| **Manifest（清单）** | 元数据 | 描述一个Artifact的Blob组成、平台信息、配置等 | 删除Manifest → 同时删除关联的Artifact和标签 |
| **元数据删除（Logical Delete）** | 数据库操作 | 在 `artifact_tag` 表中将 `deleted` 字段设为 `true` | 仅影响数据库记录，磁盘文件不变 |
| **物理删除（Physical Delete）** | 文件系统操作 | GC的Sweep阶段删除磁盘上的Blob文件 | 释放磁盘空间（不可逆） |
| **引用计数（Reference Count）** | 数据库逻辑 | Blob表通过 `artifact_blob` 关联表维护引用关系 | 引用计数为0的Blob是GC的候选删除目标 |
| **Mark阶段** | GC第一阶段 | 扫描磁盘上所有Blob，查询数据库引用关系，标记无引用的Blob | 生成 `_gc_mark_set` 临时文件，记录可删除Blob列表 |
| **Sweep阶段** | GC第二阶段 | 物理删除Mark阶段标记的Blob文件 | 真正释放磁盘空间 |
| **只读模式（Read-Only）** | GC保护机制 | GC期间Harbor禁止所有写操作（push/delete/配置修改），确保数据一致性 | 读操作（pull/查询）不受影响 |

### 4.2 GC调度策略对比

| 调度策略 | Cron表达式 | GC耗时（典型） | 对业务的影响 | 存储效率 | 推荐场景 |
|---------|-----------|-------------|------------|---------|---------|
| **每天凌晨** | `0 3 * * *` | 10-20分钟 | 凌晨3:00-3:20 CI可能失败 | ⭐⭐⭐⭐⭐ 极优 | 高频CI环境（>500 push/天） |
| **每周日凌晨** | `0 3 * * 0` | 20-40分钟 | 周日凌晨3:00-3:40 CI可能失败 | ⭐⭐⭐⭐ 优 | 中等强度生产环境 |
| **每两周凌晨** | `0 3 1,15 * *` | 40-90分钟 | 受影响时段较长 | ⭐⭐⭐ 中等 | 低强度使用（<50 push/天） |
| **仅手动触发** | 无定时 | 1-6小时（取决于间隔） | 不可预测 | ⭐ 差（垃圾累积） | ❌ 不推荐——易导致磁盘写满 |
| **从不执行** | — | — | — | ❌ 极差（持续膨胀） | ❌ 绝对不推荐——最终磁盘写满 |

### 4.3 适用场景

- **定期存储空间维护：** 配合标签保留策略（Tag Retention Policy），定时GC维持Harbor存储的健康水平。标签保留策略负责"标记哪些标签该删除"，GC负责"真正回收被删除标签的磁盘空间"——两者配合形成完整的存储生命周期管理。
- **紧急空间释放：** 当监控告警显示磁盘使用率达到85%+时，手动触发GC。但需先确认：磁盘有足够的余量容纳GC临时文件（建议预留5%以上），且有足够的维护窗口覆盖GC耗时。
- **CI/CD高吞吐环境优化：** 在每天产生大量中间构建标签的CI环境中，GC是控制存储成本的关键。配合短周期的标签保留策略（如"仅保留最近10个构建标签"），GC可以实现存储使用的稳态平衡。
- **Harbor迁移前瘦身：** 在Harbor数据迁移（如从一台服务器迁移到另一台）之前，先执行一次全量GC——减少需要传输的物理数据量，缩短迁移窗口。
- **镜像层去重后的空间稽核：** 在多个项目共享基础镜像层的环境中，GC可以帮助稽核——"理论上去重后的存储量是多少？实际磁盘使用量是多少？"——发现磁盘膨胀问题。

### 4.4 不适用场景

- **需要秒级恢复push能力的紧急故障场景：** GC的最小耗时通常在10分钟以上（即使在小型环境中），且GC期间Harbor处于只读模式。不适用于需要"立即恢复push能力"的生产紧急情况。
- **需要选择性清理指定项目或仓库的场景：** GC是全量扫描和清理——无法指定"只清理order-platform项目的垃圾"或"只清理ci-snapshots仓库的垃圾"。选择性清理必须通过其他机制（如流水线中对接单独的清理脚本）。
- **极大规模（PB级）Registry的实时空间回收：** 当Blob总数达到百万级时，GC的Mark阶段可能需要数小时——在这种情况下，传统的"定时全量GC"模式可能不适合，需要考虑企业级的存储分层管理方案。

### 4.5 五项关键注意事项

1. **GC"不可取消"是最重要的心智模型。** 一旦开始，只能等待完成（或冒险重启Registryctl）。每次手动触发GC前，务必确认：（1）当前没有Running状态的GC；（2）当前不在业务高峰期；（3）磁盘剩余空间 > 最大Blob的2倍 + 总Blob的5%（用于GC临时文件）。
2. **GC频率与GC耗时的反比关系。** 每周GC一次耗时15分钟；每月GC一次耗时45分钟；每季度GC一次耗时2小时+；半年GC一次耗时4-6小时。**越拖越慢，越慢越不敢做，形成恶性循环。** 建立雷打不动的周期性GC机制是防止存储失控的核心。
3. **标签保留策略 + GC = 完整的存储生命周期。** 标签保留策略控制"应该有多少标签"（上层元数据管理），GC控制"应该释放多少磁盘空间"（下层物理存储管理）。两者缺一不可——仅有标签保留策略没有GC，磁盘空间永不释放；仅有GC没有标签保留策略，所有的标签都会永久存在，GC无事可做。
4. **GC后的存储报告不等同于实际的磁盘可用空间。** GC的日志报告（如"2.3 GB freed"）是GC期间物理删除的Blob大小总和。但由于文件系统的Block Size、Inode开销、以及Docker Registry的存储碎片等因素，`df -h` 显示的可用空间增加量可能小于GC报告的释放量。
5. **GC的定时调度依赖JobService正常运行。** 如果JobService容器意外停止，定时GC将不会触发——而你可能直到磁盘满了才发现。建议：（1）为JobService配置健康检查和自动重启；（2）在外部监控系统（如Prometheus + Alertmanager）中添加"最近N天内是否有成功的GC执行记录"的监控规则。

### 4.6 生产环境典型故障案例

| 故障案例 | 现象描述 | 根因分析 | 解决方法 |
|---------|---------|---------|---------|
| **"雪崩之夜"——定时GC恰逢大促导致CI全面瘫痪** | 公司定于每周日凌晨3:00执行GC。但2024年双11大促恰好是周一，运维团队在周日晚上10:00启动了预热部署——CI开始大量push镜像。凌晨3:00定时GC准时触发，Harbor进入只读模式。所有push操作返回503。CI流水线全线崩溃，预热部署被迫中止。GC执行了38分钟后恢复，但预热窗口已被浪费 | 定时GC的Cron表达式未考虑特殊业务时期（大促、年终结算、季度发布）。GC在业务需要的"非常规工作时间"触发 | 1. 在大促/重要发布前一周，临时暂停定时GC（通过API DELETE schedule）；2. 在运维日历中标注GC时间，避免与特殊业务活动冲突；3. 增加GC前的人工确认环节（如：预发布前由值班运维确认GC不会在关键窗口期触发） |
| **"空间黑洞"——GC执行了一半因磁盘满而崩溃** | GC开始前磁盘使用率93%（剩余70GB）。GC的Mark阶段扫描了30万Blob，生成了约50GB的临时标记文件。磁盘使用率达到98%时，Sweep阶段开始——需要先读取临时标记文件再删除——此时操作系统因磁盘满拒绝了所有写操作和部分读操作。GC进程因无法写入日志而崩溃。Harbor被卡在只读模式 | GC在执行过程中自身会产生临时文件（`_gc_mark_set`），如果磁盘在GC开始前就已接近满额，GC本身的临时文件会加速磁盘耗尽，导致GC"自杀" | 1. 设置磁盘使用率告警阈值（如80%时告警，立即触发手动GC）；2. 在GC前确保磁盘至少剩余15-20%的空间；3. 如果磁盘已接近满额，先执行应急清理（本章步骤七）释放空间后再GC |
| **"幽灵Blob"——GC完成后发现有些Blob文件永远删不掉** | 运维发现一个奇怪的现象：每次GC都报告"0 bytes freed"，但 `du` 显示有大量文件存在于 `/data/registry/` 下某子目录中。排查后发现这些文件不在Registry的标准Blob路径下——它们是被一个错误的自动化脚本通过 `docker save` 命令写到Registry存储目录中的 `.tar` 文件，总计占了180GB | 非标准的文件（不属于Registry Blob格式的文件）不会被GC扫描到——GC只识别符合Registry Blob存储格式（`sha256/<prefix>/<digest>/data`）的文件 | 1. 定期审计 `/data/registry/` 目录，查找非标准文件和超常规大小的文件；2. 禁止任何自动化脚本或人工向Registry存储目录写入非Blob格式的文件；3. 对找到的非标准文件，在确认安全后手动删除 |

### 4.7 思考题

1. **晓数科技的Harbor存储量已达到2TB，Blob总数约50万个。当前GC策略是"每季度执行一次"，每次耗时约4小时。业务方希望在Q4大促期间（共30天）取消所有GC以保障100%的CI可用性。但取消意味着未来90天+30天=120天没有GC——按当前增长速率，存储将从2TB膨胀到约3.2TB，而挂载卷总容量是3TB。设计一个折中方案：既能保障大促期间的CI可用性（不执行GC），又能避免磁盘写满。请写出具体的实施步骤和风险评估。**

2. **Harbor的原生GC只支持"全量GC"。在大规模场景（百万级Blob、PB级存储）下，全量GC可能耗时超过12小时——这在任何维护窗口内都无法接受。能否设计一个"增量GC"方案？增量GC的核心挑战在于——如何避免清理"在增量窗口之前就已存在、但在增量窗口内才被删除标签的Blob"，同时不遗漏任何真正的垃圾？如果需要修改Harbor/Registry源码，请指出需要修改的核心模块和关键逻辑。如果不能修改源码，请提出在不执行GC的情况下管理存储增长的替代方案。**
