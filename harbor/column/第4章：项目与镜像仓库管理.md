# 第4章：项目与镜像仓库管理

## 1 项目背景

"迅捷电商"容器化推进三个月后，运维组长陈工在周一例会上拍了一张监控截图投屏——Harbor 的 `library` 项目下有 347 个仓库、4200+ 个镜像标签，存储消耗从 50GB 暴增至 523GB。更可怕的是，这个数字还在以每周 35GB 的速度增长。IT 预算组的邮件已经来了——"本月云盘扩容费用 ¥8,400，请说明增长原因并提供长期优化方案。"

**痛点一：命名混乱导致的"幽灵部署"事故。** 2024 年 3 月 15 日下午，运维执行例行发布——`kubectl set image deployment/order-service order-service=library/order-service:latest`。部署完成后，线上用户的订单页面多出了一个"邀请好友砍一刀"的悬浮按钮（产品经理明确说过这个功能因合规问题暂不上线）。紧急排查发现：推送到 `latest` 标签的镜像不是本次发布的 v1.3.2，而是三周前开发自测的 v1.3.2-dev-abc123 版本——因为 CI 流水线每次构建都覆盖 `latest` 标签，而测试团队提测时没有打固定版本标签。受影响订单 327 笔，直接退款金额 ¥4620，客服团队花费 1.5 个工作日逐一联系用户解释。事后，运维组长在群里发了一条血红色的群公告："从今天起，谁再推送 `latest` 标签到生产仓库，200 块罚款，直接扣绩效。"

**痛点二：权限管理失控与安全边界模糊。** `library` 项目被设为"公开"（为了让所有团队都能拉取基础镜像），但这个"方便"的决定带来了连锁问题——外包团队居然能拉取到核心支付模块 `payment-gateway` 的镜像（因为支付镜像也堆在 library 下）。安全团队紧急排查发现：library 下有 347 个仓库混合了 6 个业务线的全部镜像——从 openjdk 基础镜像到支付核心、从日志收集 Sidecar 到数据中台 Spark 镜像，全都混在一个项目里。权限控制的粒度只有"公开"和"私有"这一级——无法做到"基础镜像大家都能拉，但支付镜像只有支付团队能拉"。

**痛点三：跨团队镜像发现困难。"前端一个同事想在本地跑后端最新的认证 SDK 镜像进行联调，但不知道镜像存放在哪个仓库、最新稳定版本是哪个标签。在 IM 群里问了三次，三个同事给了三个不同的标签——v1.0、release-1.0.1、stable-auth——他一怒之下自己写了认证模块，复用了 2300 行代码。这正是镜像管理混乱的次生灾害——可复用的团队资产因为无法发现而无法复用。

**痛点四：存储成本失控——无人清理历史镜像。** Harbor 存储从 50GB 到 523GB 的过程并非线性的——前两个月增长到 200GB，第三个月因某个业务的 CI 流水线配置错误，每次构建都生成新的镜像标签而非覆盖已有标签，一个月内产生了 2300 个未使用的 CI 快照镜像。运维发现同一个 `order-service` 仓库下有 47 个标签，但只有最近 3 个版本在生产环境中运行。其他 44 个标签（从 v0.9.0-beta 到 v1.2.8-rc）已经半年无人问津，累计占用 187GB。

**痛点五：多环境隔离的额外复杂度。** 公司有三个隔离环境（dev、staging、prod），当前的做法是在 Harbor 中建三个项目——`library-dev`、`library-staging`、`library-prod`——每个项目下复制了相同的仓库结构。这意味着一个微服务有三个镜像副本，存储占用是去重前的 3 倍。更糟的是，每次需要在环境间"晋级"镜像时——从 dev 晋级到 staging、从 staging 晋级到 prod——运维需要手动打标签、手动推送、手动验证，过程冗长且易错。2024 年 4 月的一次晋级中，运维误将 dev 版本的 `inventory-service`（含未测试的批量操作功能）晋级到了生产，导致凌晨 3:00 的定时批量任务错误更新了 15 万条库存记录——花了整整 8 个小时回滚。

这些问题的根源归结为一个判断：**团队没有建立 Harbor 的项目层级结构、仓库命名规范、标签语义化版本策略和镜像生命周期管理机制。** 本章将系统讲解 Harbor 中"项目→仓库→标签"三级模型的管理最佳实践、命名规范设计和自动化清理策略。

## 2 项目设计——剧本式交锋对话

**场景：周四下午 14:00，凌云计划架构评审会。会议室里坐着技术负责人老张、架构师陈工、运维组长冯姐、三个业务线开发代表。投影仪上展示着 Harbor 当前的项目结构——347 个仓库全部堆在 `library` 项目下，标签名五花八门。老张把激光笔点在投影上。**

**小胖**（嘴里含着一块凤梨酥，含糊不清地说）："老张，不就是个项目名吗？我觉得现在挺好的啊——所有镜像都在 library 下面，找起来多方便，不用点来点去。我们组就叫 `order-team`，他们组叫 `payment-team`，各自建个项目，每个项目下面塞各自的服务镜像。多简单的事儿，讨论半天干嘛？"

**老张**（叹了口气）："小胖，你这个想法就是我们现在问题的根源。我问你几个问题——你们订单组下面现在有多少个微服务？"
**小胖**（掰手指数）："订单服务、库存服务、物流服务……还有啥来着？哦对，发票服务、物流追踪、还有订单定时任务——一共 6 个。"
**老张**："那六个月后会变成多少个？"
**小胖**："嗯……产品经理们天天提需求拆服务，按这趋势估计得 15 个以上。"
**老张**："那就对了。每个服务一个仓库的话，你们订单组一个项目下面就要塞 15 个仓库。一年后呢？30 个？新来的同事看到 `order-team` 这个项目名，知道里面有哪些东西吗？如果再混进了一个测试用的临时仓库（比如 `order-service-test-bak`），谁能区分哪些是生产服务、哪些是临时测试？"

**大师**（站起身走到白板前，拿起记号笔）："小胖，我给你换个例子你就懂了。想象你家里的书柜——你书柜上摆了 300 本书，但你从不分类——小说、工具书、菜谱、漫画全混在一起。有一天你想找一本《Docker 从入门到入土》，你得把所有 300 本书全部翻一遍。而如果你按'文学→中国文学→鲁迅全集→《呐喊》第一版'分类，你一眼就能定位到那本书在哪个位置。Harbor 的项目结构就是书柜的分类系统——项目是柜子，仓库是格子，标签是书的版本号。"

**技术映射**：Harbor 的层级结构为 `Registry → Project → Repository → Artifact → Tag`。Project 是 Harbor 中的顶级命名空间，所有镜像操作（Push/Pull/Scan/Replicate）都限定在项目范围内。一个 Project 可以包含多个 Repository，一个 Repository 包含多个 Artifact（每个 Artifact 由唯一的 digest 标识），一个 Artifact 可以有多个 Tag。

**小胖**："那按你这个比喻，library 就是我们现在的书柜——300 本书全塞在唯一的一个格子里，不乱才怪。"

**大师**："对，所以现在我们得把书分类放到不同的柜子里。这就是项目命名的设计。"

**小白**（从白板旁边站起来，拿了一支红色记号笔）："大师，我昨天调查了一下我们公司现在的微服务结构——共 6 个业务线（订单、支付、用户、商品、物流、数据），每条业务线 5-15 个微服务。我算了三种项目划分方案——但不知道哪种最好。"

1. "**按业务线建项目**：`order-platform`、`payment-platform`、`user-platform`……"
2. "**按环境建项目**：`dev`、`staging`、`prod`……"
3. "**按技术栈建项目**：`java-services`、`python-services`、`go-services`、`node-services`……"

"另外，我们还有公共基础镜像（openjdk、nginx、python）、第三方镜像缓存（从 Docker Hub 代理的）、以及 CI 构建的快照镜像——这些怎么归类？"

**大师**："小白你把问题拆得很到位。三种方案我都见过实际案例，我给你分析每个方案的利弊——"

"**方案一：按业务线建项目（推荐，80% 的团队应该选这个）**——"

```
harbor.xunjie.com/
├── order-platform/           # 项目名 = 业务线
│   ├── order-service         # 仓库名 = 微服务名
│   ├── inventory-service
│   ├── logistics-service
│   ├── invoice-service
│   ├── order-scheduler
│   └── logistics-tracker
├── payment-platform/
│   ├── payment-gateway
│   ├── billing-service
│   └── reconciliation-service
├── user-platform/
│   ├── user-auth-api
│   ├── user-profile
│   └── member-center
├── shared-base/              # 公共基础镜像（公开只读）
│   ├── java-base
│   ├── nginx-base
│   ├── python-base
│   └── node-base
├── proxy-cache/              # Docker Hub 代理缓存（公开只读）
│   ├── library/ubuntu
│   ├── library/alpine
│   └── bitnami/redis
└── ci-snapshots/             # CI 构建的快照（私有，自动清理）
    ├── order-service-snapshots
    ├── payment-gateway-snapshots
    └── user-auth-snapshots
```

"为什么推荐方案一？理由有三个——"

1. "**权限边界天然清晰**——订单组只需要 `order-platform` 项目的开发者权限，不需要访问支付组的项目。如果在方案一中，你想控制'张三只能 Push 到 order-service 但不能 Push 到 inventory-service'——这做不到，因为项目是最小权限边界。如果你的团队需要仓库级别的权限隔离，那需要把粒度提升——每个仓库单独建项目（如 `order-order-service`、`order-inventory-service`），但这样会导致项目数量爆炸。"

2. "**配额管理直观**——按业务线设置存储配额（如 order-platform 设 200GB，payment-platform 设 100GB），哪个团队的镜像膨胀了一目了然。在方案二（按环境）中，同一个服务的镜像分布在三个项目里，你无法直观统计'订单服务线的总镜像存储消耗'。"

3. "**环境区分用标签，不用项目**——同一仓库下，`order-service:v1.2.3-staging` 和 `order-service:v1.2.3-prod` 是不同的标签。方案二会把同一个服务拆到三个项目里（dev/staging/prod），带来两个问题：一是权限管理复杂（开发需要同时访问 dev 和 staging 项目），二是复制同步策略混乱（要把 dev 的镜像同步到 staging 和 prod 项目，需要配置多条复制规则）。"

"**方案二：按环境建项目（不推荐，除非有极强的合规需求）**——"

"只有一种场景适合按环境建项目——**强合规隔离**。比如金融行业要求生产环境的镜像仓库与开发环境物理隔离（不同机柜、不同服务器），这时你可能需要两套独立的 Harbor 实例而非仅按项目隔离。如果在同一个 Harbor 内按环境建项目，不解决任何核心问题，反而增加管理复杂度。"

"**方案三：按技术栈建项目（完全不推荐）**——"

"`java-services` 这个项目名下能看出业务含义吗？完全不能。新同事看到一个错误，想定位是哪个服务抛出的——他看到 `java-services` 项目，里面 30 个 Java 仓库，根本不知道哪个是哪个。"

**小胖**（把凤梨酥咽下去）："那公共基础镜像呢？比如我们所有 Java 服务都依赖同一个 `java-base` 镜像。这个镜像应该放在哪个项目？如果放在每个业务线项目里（复制三份），那存储不就三倍了吗？"

**大师**："好问题。公共基础镜像的正确存放方式是——**单独建一个 `shared-base` 项目，设为公开只读**。所有业务线在自己的 Dockerfile 中 `FROM harbor.xunjie.com/shared-base/java-base:v11` 引用，而不是把 `java-base` 复制到每个业务线项目下。这样 6 条业务线、30 个微服务，全靠同一个基础镜像层——磁盘上只存一份，去重后存储消耗极小。"

"但需要注意一点——`shared-base` 的 Push 权限应该只授予平台组或基础架构组的 2-3 个人。其他所有人只有 Pull 权限。这是因为基础镜像的任何变更都会影响所有上游服务——如果某个开发者'顺手优化'了 java-base 然后 Push 了上去，可能导致 30 个服务在下一次构建时全炸。"

**技术映射**：Harbor 的公开项目（Public）允许任何人（包括未登录的匿名用户）通过 Docker 客户端执行 Pull 操作。但 Push 操作仍然需要项目至少为"开发者"角色的成员。这就实现了"全公司可读、仅少数人可写"的共享模式。私有项目（Private）则要求 Pull 也需要项目成员身份。

**小胖**："等下等下——如果 `shared-base` 设为公开，未登录的人也能拉取我们的基础镜像？那岂不是安全风险？"

**大师**："这取决于你的安全策略。公开项目的好处是方便——你不需要给每个新入职的同事分别授权到 `shared-base` 项目。但如果你公司的安全策略要求'镜像不出公司网络'（如金融行业），那你应该把 `shared-base` 设为私有，并创建一个'全公司开发组'的用户组，自动给所有开发者分配 `shared-base` 的访客角色。在 Harbor 中可以通过 LDAP/AD 集成来实现——将 LDAP 中的开发组 OU 映射到 Harbor 的用户组，再给用户组分配项目角色。"

**小白**（放下记号笔，拿起笔记本）："大师，项目结构定下来了，那仓库命名规范和标签规范呢？我们现在的标签名完全是灾难——有 `latest`、`v1`、`v1.0`、`v1.0.0`、`20240101-dev`、`feature-login-v2`、`hotfix-20240301`、甚至还有中文标签——`不要删-备份`。有没有一套可以强推的标准？"

**大师**（拿起另一支黑色记号笔）:"这是 Harbor 治理中最容易被忽视但影响最深远的环节。我给你三套规范——"

"**第一套：仓库命名规范**——"

```
┌─── Harbor 仓库命名规范 v1.0 ──────────────────────────────────┐
│                                                                │
│  格式：<微服务名称>[-<子模块>]                                   │
│                                                                │
│  规则：                                                        │
│  1. 全部小写字母                                                │
│  2. 单词之间用连字符 - 分隔（禁止下划线 _）                      │
│  3. 名称必须具有业务含义（禁止 test、tmp、bak 等无意义词）      │
│  4. 与 Git 仓库名保持一致（降低认知负担）                       │
│  5. 长度不超过 255 个字符                                       │
│                                                                │
│  正确示例：                                                     │
│    order-service             ✓ 订单服务主仓库                  │
│    order-service-worker      ✓ 订单服务的 Worker 子模块        │
│    payment-gateway           ✓ 支付网关                        │
│    user-auth-api             ✓ 用户认证 API                    │
│    logistics-tracker         ✓ 物流追踪                        │
│    spark-data-cleaner        ✓ Spark 数据清洗任务               │
│                                                                │
│  错误示例：                                                     │
│    OrderService              ✗ 包含大写字母                    │
│    order_service             ✗ 使用下划线                      │
│    my-test-image             ✗ 不具业务描述性                  │
│    test123                   ✗ 完全无意义                      │
│    order-service-不要删-备份  ✗ 中文标签（绝对禁止）            │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

"**第二套：标签（Tag）语义化版本规范**——"

```
标签格式：<语义化版本>[-<环境后缀>]

生产版本：
  order-service:v1.2.3                    ← 语义化版本
  order-service:v2.0.0                    ← 大版本

预发布版本：
  order-service:v1.2.3-staging            ← 预发布环境
  order-service:v1.2.3-dev                ← 开发环境
  order-service:v1.2.3-rc.1               ← Release Candidate

CI 构建快照（与 Git 关联）：
  order-service:v1.2.3-dev-abc123         ← git short hash
  order-service:build-20240416-001        ← 构建编号

稳定发布（推荐打 release 前缀）：
  order-service:release-v1.2.3            ← 稳定发布版（不可变）
  order-service:release-v2.0.0            ← 大版本稳定发布

严格禁止：
  order-service:latest                    ✗ 禁止使用 latest
  order-service:stable                    ✗ 无版本号，无法追溯
  order-service:test                      ✗ 无版本号
  order-service:tmp-20240416              ✗ 无版本号
```

"**第三套：'禁止 latest 标签'的强硬规则**——"

"'latest 标签'这个 Docker 惯例是容器生态中最危险的反模式之一。Docker 官方用 `latest` 是因为他们想要一个'新手友好'的默认值，但对企业生产环境来说，`latest` 本质上是'我不知道这个标签对应哪个版本'的代名词。我们一刀切的策略是——"

1. "通过 Harbor 的标签不可变性规则，禁止覆盖 `latest` 标签——你推可以，但一旦推了一次，就不能再覆盖。这会迫使开发者不能再依赖 `latest` 作为'最新发布版'。"
2. "CI 流水线中禁止使用 `latest` 标签——在 Jenkinsfile/GitLab CI 配置中推送固定版本标签。"
3. "K8s Deployment 中禁止使用 `latest` 作为镜像标签——改为 `release-v1.2.3` 或 `v1.2.3`。"

**技术映射**：Harbor 的标签不可变性（Tag Immutability）规则限制了特定标签的覆盖操作。规则匹配后，Docker 客户端推送同名标签时会返回 `denied: The tag is immutable and cannot be overwritten`。标签保留策略（Tag Retention）则是在标签数量超过阈值时自动清理旧标签，两者互补——不可变性管"你不许改这个标签"，保留策略管"我不需要这么多标签"。

**小胖**（突然站起来）："等等！我想到一个问题——如果 K8s Deployment 不能用 `latest`，那每次发布新版本，K8s YAML 文件也得跟着改镜像标签？那运维不得疯了？"

**大师**："这个问题恰好引出了一个关键最佳实践——K8s Deployment 中的镜像标签应该由 CI/CD 流水线自动更新，而不是手工改。"

"具体工作流如下：①开发者提交代码并创建 Git Tag（如 `v1.2.3`）；②CI 流水线触发构建，使用 Git Tag 作为镜像标签推送到 Harbor；③CI 流水线自动执行 `kubectl set image deployment/order-service order-service=harbor.xunjie.com/order-platform/order-service:v1.2.3` 或使用 Helm/Kustomize 更新 values；④K8s 执行滚动更新。整个流程中，运维不需要手动修改任何 YAML 文件。"

"但如果你的团队还没有到这个自动化水平，一个过渡方案是——使用 GitOps 工具（如 ArgoCD/FluxCD），它们会自动监听 Harbor 中的镜像标签变更并同步到 K8s。具体来说——配置 ArgoCD 监听 `order-service:release-v1.*`，当 CI 推送了 `release-v1.2.3` 标签后，ArgoCD 自动更新 K8s Deployment。"

**小白**（表情严肃）："大师，我还有两个深入的场景想问。第一个：我们现在的项目结构是按业务线建的——每个业务线一个项目。但如果一个业务线需要对外提供一个公共 SDK 镜像——比如支付组的 `payment-sdk` 需要被订单组、用户组引用——这个 SDK 镜像应该放在支付组的私有项目里（需要给其他组开权限），还是应该单独建一个如 `shared-sdks` 公开项目？"

**大师**："这个场景恰好暴露了'按业务线建项目'的一个边界问题——跨业务线共享。我的建议是——**单独建一个 `shared-sdks` 项目，设为公开只读**。理由与 `shared-base` 类似——共享资源应该集中管理、统一版本。支付团队通过某种流程（如 Merge Request 审批）将 `payment-sdk:v2.1.0` 推送到 `shared-sdks` 项目，其他团队从 `shared-sdks` 拉取使用。不要给每个业务线单独开权限到支付项目的 SDK 仓库——这样做权限管理会越来越碎片化。"

**小白**："第二个场景——关于多环境的隔离策略。我们公司有三个环境（dev/staging/prod），现在镜像晋级流程是手动打标签——dev 测试通过后，运维从 `order-service:v1.3.0-dev` 打一个新标签 `order-service:v1.3.0-staging` 推送到 staging 环境用的项目。这个过程每次都担心'打错标签'或'推错项目'。有没有更可靠的方式？"

**大师**："这正好引出了 Harbor 进阶功能——**复制规则（Replication Rules）**。你的镜像晋级流程不需要手动打标签——"

"推荐方案：**每个环境一个 Harbor 实例（或至少独立项目）**，通过 Harbor 的复制规则自动化镜像晋级。"

```
开发环境 Harbor (dev-harbor)                  生产环境 Harbor (prod-harbor)
┌──────────────────────────┐    Push-based     ┌──────────────────────────┐
│ order-platform/          │     Replication   │ order-platform/          │
│  order-service:v1.3.0    │ ─────────────────→│  order-service:v1.3.0    │
│                          │     晋级规则      │  (不可变标签)            │
│  order-service:v1.3.1    │   触发条件：       │                          │
│  order-service:v1.4.0    │   标签匹配         │                          │
│                          │   release-v*       │                          │
└──────────────────────────┘                   └──────────────────────────┘
```

"具体配置方式——在生产 Harbor 中创建一条 Pull-based 复制规则：源 Registry = 开发 Harbor，源项目 = `order-platform`，过滤器 = `release-v*` 标签，触发模式 = 手动触发或定时触发。每次需要晋级时，运维在 Portal 中点击'复制'按钮，Harbor 自动把符合条件的镜像从开发环境拉到生产环境，无需手动打标签。"

"如果你使用的是 Harbor 内按项目隔离（而非独立实例），可以简化为——每个月一个项目（如 `order-platform-dev`、`order-platform-prod`），复制规则在项目间同步。但我个人更推荐独立实例的方式——生产 Harbor 在物理/网络上与开发环境隔离，符合安全合规。"

**技术映射**：Harbor 支持两种复制模式：①Push-based（源 Harbor 推送到目标）；②Pull-based（目标 Harbor 从源拉取）。两种模式各有适用场景——Push-based 适合源端可控（你知道何时触发复制），Pull-based 适合目标端有自主权（比如生产 Harbor 从多个上游拉取镜像）。复制通过 JobService 的异步任务执行，支持 Harbor↔Harbor、Harbor↔Docker Hub、Harbor↔AWS ECR 等多种目标类型。

### 3.2 规划企业级项目结构

> **步骤目标**：基于"按业务线建项目"的方案，创建 6 个业务项目 + 3 个辅助项目。

```bash
#!/bin/bash
# ==========================================
# 批量创建 Harbor 项目结构脚本
# ==========================================

HARBOR_URL="https://harbor.xunjie.com"
ADMIN_AUTH="admin:Harbor12345"

# 定义项目配置：项目名:公开/私有:存储配额(GB):说明
declare -A PROJECTS=(
  # === 业务线项目（私有） ===
  ["order-platform"]="false:200:订单业务线所有微服务"
  ["payment-platform"]="false:100:支付业务线所有微服务"
  ["user-platform"]="false:100:用户业务线所有微服务"
  ["product-platform"]="false:150:商品与库存业务线所有微服务"
  ["logistics-platform"]="false:100:物流与配送业务线所有微服务"
  ["data-platform"]="false:300:数据中台与 Spark/Flink 任务镜像"
  
  # === 辅助项目 ===
  ["shared-base"]="true:500:公共基础镜像（JDK/Python/Nginx 等，公开只读）"
  ["shared-sdks"]="true:200:跨团队共享 SDK 镜像（公开只读）"
  ["proxy-cache"]="true:1000:Docker Hub 代理缓存（公开只读）"
  ["ci-snapshots"]="false:500:CI 构建快照（私有，自动清理策略）"
)

echo "开始创建 Harbor 项目结构..."
for project_name in "${!PROJECTS[@]}"; do
  IFS=':' read -r public storage_gb description <<< "${PROJECTS[$project_name]}"
  storage_bytes=$((storage_gb * 1024 * 1024 * 1024))
  
  response=$(curl -k -s -u "$ADMIN_AUTH" -X POST \
    -H "Content-Type: application/json" \
    -d "{
      \"project_name\": \"$project_name\",
      \"public\": $public,
      \"storage_limit\": $storage_bytes
    }" \
    "$HARBOR_URL/api/v2.0/projects")

  if echo "$response" | grep -q '"project_id"'; then
    echo "✓ 项目 $project_name 创建成功（配额: ${storage_gb}GB, 公开: $public）"
  elif echo "$response" | grep -q 'already exists'; then
    echo "○ 项目 $project_name 已存在，跳过"
  else
    echo "✗ 项目 $project_name 创建失败: $response"
  fi
done

echo ""
echo "项目创建完成！请登录 Portal 确认项目结构："
echo "  $HARBOR_URL"
```

### 3.5 镜像推送与标签规范实战

> **步骤目标**：按照命名规范和标签规范，完成一次完整的镜像构建、打标签、推送流程。

```bash
# ==========================================
# 规范的镜像 Push 流程（从构建到推送）
# ==========================================

# 1. 在 CI 环境中（模拟 Jenkins），获取 Git Tag 作为版本号
GIT_TAG="v1.3.2"
GIT_HASH=$(git rev-parse --short HEAD)    # 如：abc123

# 2. 构建镜像
docker build -t order-service:${GIT_TAG} .

# 3. 打标签规范流程
# 3a. 生产版本（语义化版本标签）
docker tag order-service:${GIT_TAG} \
  harbor.xunjie.com/order-platform/order-service:${GIT_TAG}

# 3b. 稳定发布标签（release-* 前缀，设置为不可变）
docker tag order-service:${GIT_TAG} \
  harbor.xunjie.com/order-platform/order-service:release-${GIT_TAG}

# 3c. 开发快照标签（Git hash 可追溯）
docker tag order-service:${GIT_TAG} \
  harbor.xunjie.com/order-platform/order-service:${GIT_TAG}-dev-${GIT_HASH}

# 3d. 环境标签（staging 环境的预发布版本）
docker tag order-service:${GIT_TAG} \
  harbor.xunjie.com/order-platform/order-service:${GIT_TAG}-staging

# 4. 推送到 Harbor（一次性推送所有标签）
docker push harbor.xunjie.com/order-platform/order-service:${GIT_TAG}
docker push harbor.xunjie.com/order-platform/order-service:release-${GIT_TAG}
docker push harbor.xunjie.com/order-platform/order-service:${GIT_TAG}-dev-${GIT_HASH}
docker push harbor.xunjie.com/order-platform/order-service:${GIT_TAG}-staging

# 5. 验证推送结果（通过 API 查看该仓库的所有标签）
curl -k -s -u admin:password \
  "$HARBOR_URL/api/v2.0/projects/order-platform/repositories/order-service/artifacts?page_size=5" | \
  jq -r '.[] | "Tags: \(.tags[].name)\tDigest: \(.digest[0:15])\tSize: \(.size)"'
```

**预期输出**：

```
Tags: v1.3.2            Digest: sha256:a1b2c3d4e5f6...    Size: 218MB
Tags: release-v1.3.2    Digest: sha256:a1b2c3d4e5f6...    Size: 218MB
Tags: v1.3.2-dev-abc123 Digest: sha256:a1b2c3d4e5f6...    Size: 218MB
Tags: v1.3.2-staging    Digest: sha256:a1b2c3d4e5f6...    Size: 218MB
```

注意：四个标签指向**同一个 Digest**（sha256:a1b2c3...），表明它们指向同一个镜像制品。磁盘上只消耗 218MB，而非 218MB × 4 = 872MB。

### 3.6 配置复制规则实现镜像环境晋级

> **步骤目标**：配置从开发 Harbor 到生产 Harbor 的 Pull-based 复制规则，自动化镜像晋级。

```bash
# ==========================================
# 配置复制规则（自动化镜像环境晋级）
# ==========================================

# 前提条件：
# 1. 已部署两套 Harbor 实例：dev-harbor (10.0.1.100) 和 prod-harbor (10.0.3.100)
# 2. 在 prod-harbor 中已经创建了 order-platform 项目
# 3. prod-harbor 可以网络访问 dev-harbor（至少允许 443 端口）

PROD_HARBOR="https://harbor-prod.xunjie.com"
ADMIN_AUTH="admin:ProdAdminPass2024"

# 1. 先在 prod-harbor 中注册 dev-harbor 为远程 Registry 目标
curl -k -s -u "$ADMIN_AUTH" -X POST \
  -H "Content-Type: application/json" \
  -d '{
    "name": "dev-harbor",
    "type": "harbor",
    "url": "https://harbor-dev.xunjie.com",
    "access_key": "admin",
    "access_secret": "DevAdminPass2024",
    "insecure": false
  }' \
  "$PROD_HARBOR/api/v2.0/registries"

echo "✓ dev-harbor 已注册为远程 Registry"

# 2. 创建复制策略：从 dev-harbor 的 order-platform 拉取 release-v* 标签
curl -k -s -u "$ADMIN_AUTH" -X POST \
  -H "Content-Type: application/json" \
  -d '{
    "name": "order-platform-prod-sync",
    "description": "从开发 Harbor 同步 order-platform 的 release 镜像到生产",
    "src_registry": {
      "id": 1
    },
    "dest_namespace": "order-platform",
    "trigger": {
      "type": "manual"
    },
    "filters": [
      {"type": "name", "value": "**"},
      {"type": "tag", "value": "release-v**"}
    ],
    "enabled": true,
    "deletion": false
  }' \
  "$PROD_HARBOR/api/v2.0/replication/policies"

echo "✓ 复制策略已创建（手动触发，筛选 release-v* 标签）"
```

# 3. 对于确认不用的僵尸项目，删除前先归档（导出镜像列表）

echo ""
echo "=== 归档项目 $ZOMBIE_PROJECT 的镜像列表 ==="
curl -k -s -u "$ADMIN_AUTH" \
  "$HARBOR_URL/api/v2.0/projects/$ZOMBIE_PROJECT/repositories?page_size=100" | \
  jq '.[] | {repo: .name, artifact_count: .artifact_count, pull_count: .pull_count}' \
  > ./harbor-archive-$ZOMBIE_PROJECT-$(date +%Y%m%d).json

echo "镜像列表已导出到: ./harbor-archive-$ZOMBIE_PROJECT-*.json"

# 4. 删除僵尸项目（软删除，72 小时内可恢复）
# 注意：删除项目前必须确认无业务依赖！
read -p "确认要删除项目 '$ZOMBIE_PROJECT' 吗？(输入 yes 确认): " confirm
if [ "$confirm" = "yes" ]; then
  curl -k -s -u "$ADMIN_AUTH" -X DELETE \
    "$HARBOR_URL/api/v2.0/projects/$ZOMBIE_PROJECT"
  echo "✓ 项目 $ZOMBIE_PROJECT 已软删除（72 小时内可恢复）"

  echo "已取消删除操作"

### 3.9 可能遇到的坑

**坑1：删除项目后无法立即创建同名项目（软删除机制）**

- **现象**：运维删除了一个项目 `test-temp`，紧接着想创建一个新的同名项目 `test-temp`（为了重建项目结构），Portal 提示"项目名称已存在"或 API 返回 `409 Conflict`。
- **根因**：Harbor 的删除操作是软删除（Soft Delete），删除时只在数据库中将 `deleted` 字段标记为 `true`，实际的数据库记录保留 72 小时。这是出于数据安全考虑——防止误删后数据无法恢复。在这 72 小时内，同名项目无法创建。
- **解决方法**：
  1. 方案一（推荐）：等待 72 小时，系统自动清理后即可创建同名项目。
  2. 方案二（紧急）：联系 DBA 手动操作 PostgreSQL 清除 `deleted=true` 的记录：
     ```sql
     -- 谨慎操作！确认项目已备份后再执行
     SELECT project_id, name, deleted FROM project WHERE name = 'test-temp';
     -- 如果确认要物理删除：
     DELETE FROM project WHERE name = 'test-temp' AND deleted = true;
     ```
  3. 方案三（临时）：使用不同的项目名称（如 `test-temp-v2`），避免等待。

**坑2：标签保留策略执行后，磁盘空间释放很少**

- **现象**：运维在 `ci-snapshots` 项目配置了"保留最近 10 个制品"的策略，策略执行完成后（凌晨 3:00），Dashboard 显示标签数量减少了（从 2300 个减到 180 个），但"总存储消耗"数字几乎没有下降（仅从 523GB 降到 518GB）。
- **根因**：标签保留策略只清理标签元数据（Tag → Artifact 的映射关系），**不会直接删除镜像的 blob 层数据**。也就是说，标签被删了但底层的镜像层文件还在磁盘上。真正的空间释放需要两个步骤：①标签保留策略执行（删除标签）→②垃圾回收 GC 执行（删除未被任何标签引用的孤儿 blob）。如果只做了第①步不做第②步，磁盘空间不会释放。
- **解决方法**：
  ```bash
  # 1. 确认标签保留策略已执行完成（Portal → ci-snapshots → 仓库页签确认标签减少）
  # 2. 手动触发 GC 释放孤儿 blob：
  #    a) 先设只读模式：curl -X PUT -d '{"read_only":true}' .../api/v2.0/configurations
  #    b) Portal → 垃圾回收 → 立即执行 GC
  #    c) 等 GC 完成后取消只读模式
  ```
- **教训**：标签保留策略 + GC 是一个两阶段清理流程。仅配置标签保留策略而不配置 GC 定时执行，就等于"只把垃圾打包好堆在门口，但没有垃圾车来运走"。

**坑3：同时配置配额限制和标签保留策略后，保留策略可能无法执行**

- **现象**：项目 `order-platform` 配额设为 100GB，标签保留策略设为"保留最近 10 个制品"。当存储达到 99GB 时，新的 Push 被配额阻止（返回 413 Request Entity Too Large），但标签保留策略也没有自动执行来清理空间。
- **根因**：配额检查发生在 Push 请求阶段——当 Docker Client 尝试推送一个新的 blob 时，Core 先检查项目当前存储 + 即将推送的 blob 大小是否超过配额。如果超出，直接在 API 层面拒绝，不给 Registry 写入的机会。而标签保留策略是定时执行的（如每天 3:00），在它执行之前，配额就已经阻止了任何新的 Push。这形成了一个死锁——清理需要 Push 新标签覆盖旧标签，但 Push 被配额阻止。
- **解决方法**：
  1. 运维手动增加配额（临时）→ 触发标签保留策略 → 等待策略执行 → 恢复配额
     ```bash
     curl -k -u admin:pass -X PUT \
       -d '{"storage_limit": 214748364800}' \
       https://harbor.xunjie.com/api/v2.0/projects/1  # 临时扩到 200GB
     ```
  2. 或者：手动删除一些不需要的旧标签 → 触发 GC → 配额自动释放
  3. 长期方案：将配额预警线从 100% 改为 80%——当存储达到 80GB 时发送告警，运维手动触发标签保留策略和 GC，在配额满之前释放空间

**坑4：复制规则同步后，目标 Harbor 的标签被意外覆盖**

- **现象**：开发 Harbor 中 `order-service:release-v1.3.2` 的 digest 是 sha256:abc123，生产 Harbor 中相同标签的 digest 已经变成了 sha256:def456（因为架构组手工重新构建并推送了一次）。配置复制规则后，开发 Harbor 的 `release-v1.3.2` 再次同步时，**覆盖了**生产 Harbor 上的版本。
- **根因**：复制规则默认行为——如果目标 Harbor 上已存在同名标签，复制任务会用源 Harbor 的镜像覆盖目标 Harbor 的标签。在 Harbor 的复制策略中，没有"检测到目标已存在同名标签时跳过"的默认选项（v2.12 中仍如此）。这是因为复制策略的设计初衷是"保持源和目标一致性"，而不是"增量追加"。
- **解决方法**：
  1. 生产环境的关键标签（如 `release-*`）配置标签不可变性，即使复制规则尝试覆盖也会被阻止：`denied: The tag is immutable and cannot be overwritten`
  2. 在配置复制规则时，使用"事件驱动"的触发模式而非"定时"模式——只在开发 Harbor 有新标签推送时才触发同步到生产 Harbor，避免重复同步覆盖
  3. 如果已经发生覆盖，从目标 Harbor 的镜像日志中查找被覆盖版本的 digest，联系开发使用新 digest 重新推送

### 4.4 注意事项（项目与仓库管理的核心原则）

1. **绝不在 Harbor 中混用 `latest` 标签**：这是容器镜像管理中最危险的反模式。通过标签不可变性规则强制禁止覆盖，通过 CI 流水线规范禁止推送。
2. **公开项目的 Pull 是全透明的**：公开项目中任何人（包括未登录的 Docker Client）都可以拉取镜像。如果将内部工具镜像放到公开项目，需要评估是否有安全合规风险。
3. **标签不可变性和保留策略可以共存**：不可变性优先级高于保留策略——如果一个标签匹配了不可变性规则，即使它也在保留策略的清理范围内，也不会被删除。生产环境应该对 `release-*` 标签同时设置不可变性和排除于保留策略。
4. **配额只限制镜像层，不限制元数据**：配额限制的是 `blob` 表的存储消耗，不包括 Helm Chart、CVE 扫描报告、审计日志等元数据。实际磁盘占用会比配额显示的数字大 10-20%。
5. **机器人账户 Secret 在创建后只展示一次**：务必在创建后立即复制并存入密钥管理工具（HashiCorp Vault / 1Password / 公司内部密钥平台）。如果丢失，只能删除重建机器人账户。
6. **复制规则的"覆盖"行为需要警惕**：如果目标 Harbor 上已存在同名标签，复制任务会默认覆盖。生产环境的关键标签应设置不可变性作为最后一道防线。
7. **项目删除后 72 小时内无法创建同名项目**：这是软删除机制的保护窗口。如果紧急需要重建，需要直接操作 PostgreSQL 清理 `deleted` 标记。

### 4.5 常见踩坑经验（生产项目管理的血泪教训）

| 故障案例 | 故障现象 | 根因诊断 | 修复措施 | 教训总结 |
|---------|---------|---------|---------|---------|
| **"标签不可变性规则配置太宽导致 CI 构建失败"** | 运维为了"保险"，给项目 `order-platform` 配置了一条全局不可变性规则——仓库匹配 `**`、标签匹配 `**`。这意味着**所有标签都不可覆盖**。CI 流水线每次构建推送 `v1.3.2-dev-abc123` 标签（固定标签，每次构建都推同一个标签名）时，Harbor 返回 `denied: The tag is immutable`，CI 构建全部失败 | 标签不可变性规则的 `tag_selectors` 设成了 `**`（匹配所有标签），导致 CI 的快照标签（`*-dev-*`）也被阻止覆盖。这是配置粒度过于粗糙的典型错误 | ①修改不可变性规则：`tag_selectors` 从 `**` 改为 `latest,release-**`，只锁定生产关键标签；②CI 构建标签使用动态标签（含时间戳或 Git hash），如 `v1.3.2-dev-20240416-001`，每次都不同，不需要覆盖 | 标签不可变性是双刃剑——配置得当可以防止生产事故，配置过度会导致 CI 完全瘫痪。正确做法：只对明确需要锁定的标签模式（`latest`、`release-*`、`v*.*.*` 生产版本）设置不可变性，对 CI 快照标签（`*-dev*`、`*-snapshot*`）不要设置 |
| **"项目存储配额 200GB 填满后，所有服务的 Push 全部卡死"** | 周一早上 9:30，Jenkins 流水线集中触发构建（周末积压的 8 个特性分支全部合入主干）。上午 10:15，所有服务的构建开始失败——Harbor 返回 `413 Request Entity Too Large`。排查发现：`order-platform` 项目存储已用 199.8GB（配额 200GB），而周末的 CI 构建产生了约 8GB 的快照镜像。配额满了之后，连新建仓库的第一次 Push 也被阻止 | 周末的 CI 构建产生了大量快照镜像，加上标签保留策略只清理"标签"而不触发 GC，实际 blob 数据一直累积。周一的集中构建成为"压死骆驼的最后一根稻草" | ①紧急扩容配额到 300GB（重置 CI 构建）；②手动触发 GC 释放约 35GB 的孤儿 blob；③事后配置：配额告警从 100% 改为 80%——当存储达到 160GB 时 Prometheus 发送告警到运维群，避免配额打满 | 存储配额是一个硬限制——一旦到达，整个项目的所有 Push 全部阻塞。需要建立一个三级防护：①配额设为业务估算值的 1.5 倍（给突发增长留缓冲）；②在 80% 配额时自动告警；③标签保留策略 + 定时 GC 构成自动清理管线，避免配额被无用的旧镜像消耗 |
| **"按环境建项目导致镜像晋级时标签不一致"** | 团队按环境建了 `dev-images`、`staging-images`、`prod-images` 三个项目。开发在 `dev-images/order-service:v1.3.0` 测试通过后，运维手动执行 `docker pull dev-harbor/dev-images/order-service:v1.3.0` → `docker tag ... prod-harbor/prod-images/order-service:v1.3.0` → `docker push`。镜像 digest 在这一过程中保持不变（因为是同一个制品）。但运维有一次手误，在 Pull 之前没有确认本地 Docker 是否已缓存了另一个 `v1.3.0` 的标签——本地恰好有一个旧的 `v1.3.0-dev` 标签指向了一个更早的镜像。运维用 `docker tag v1.3.0-dev` 错误地覆盖了生产标签，导致生产环境部署了 v1.3.0-dev 版本 | 手动晋级流程的致命缺陷——依赖运维的"记忆力"和"操作习惯"。三个项目名让运维在操作时需要频繁切换上下文，增加了出错概率 | ①紧急回滚——用 `kubectl rollout undo deployment/order-service` 恢复到上一个版本；②与架构组讨论后，重构镜像晋级流程——改为单项目 + 标签区分环境（`v1.3.0-staging`、`v1.3.0-prod`），运维不再需要跨项目操作；③在 CI 流水线中加入"环境晋级"自动化步骤——开发验证通过后，流水线自动给镜像打上 `-staging` 标签，无需运维手动操作 | "按环境建项目"的最大弊端不是技术问题，而是操作复杂性导致的人为错误。如果在单项目 + 标签区分环境的方案中，配合复制规则自动化同步（而非手动打标签），出错的概率远低于"多项目 + 手动晋级"方案 |

### 4.6 思考题

**问题 1**：公司有开发（dev）、预发布（staging）、生产（prod）三个环境。方案 A：在 Harbor 中建三个项目（`order-platform-dev`、`order-platform-staging`、`order-platform-prod`）；方案 B：只建一个项目（`order-platform`），用不同标签区分环境（`v1.3.0-dev`、`v1.3.0-staging`、`v1.3.0-prod`）。请从以下三个维度分析两种方案的优劣并给出推荐：①运维成本（镜像晋级的操作步骤和出错概率）；②存储消耗（同一镜像在三环境中的副本数）；③安全性（生产环境镜像被开发意外访问/推送的风险）。（提示：考虑 Harbor 的 RBAC 是否能实现"同一个项目中的不同标签有不同的权限"？如果不支持，方案 B 在生产安全上是否有缺陷？）

**问题 2**：如果某个项目 `order-platform` 的配额满了（200GB），GC 也释放不出空间（因为全部 blob 都被至少一个标签引用），还有哪些手段可以快速恢复 Push 能力？请至少列出三种方案，并分析每种方案的适用场景和副作用。（提示：考虑临时扩配额、手动删除不用的旧标签、将部分不常用的历史镜像迁移到冷存储项目、拆分项目等策略。）
