# 第21章：复制引擎深度实战

## 1 项目背景

跨国云计算服务商"CloudNova"的Harbor部署在全球5个Region——北美（us-east-1）、欧洲（eu-west-1）、亚太（ap-southeast-1）、南美（sa-east-1）、中东（me-central-1）。核心基础镜像由北美CI/CD构建后推送到本地Harbor，然后需要分发到其他4个Region。日均镜像复制量约2000个制品、总数据量约300GB。

**痛点一：复制任务大量失败但缺乏分类处理。** 某次北美CI/CD系统批量构建了500个新版本镜像并触发复制——388个成功，112个失败。失败原因分布：52个超时（跨太平洋网络波动）、38个目标Registry不可达（中东Region在做维护）、22个权限不足（目标项目配置错误）。运维逐个手动重试失败任务耗费了3个多小时，期间亚太团队只能用旧版本镜像进行测试。

**痛点二：复制延迟影响多Region业务协作。** 北美DevOps团队凌晨2:00推送了一个修复关键CVE的hotfix镜像，但亚太团队早上9:00（UTC+8）尝试拉取时发现还是旧版本。从北美push到亚太可pull——中间有2分30秒的延迟被放大为7小时的"有效延迟"（因为时区差异，北美push时亚太在睡觉）。业务方质疑"你们的跨Region同步为什么这么慢？"

**痛点三：复制拓扑从星型改为网状后出现"环路"。** 最初部署时采用星型拓扑（北美Hub → 4个Spoke），复制规则只有4条。但欧洲团队反馈从北美pull基础镜像跨大西洋太慢，要求改为网状拓扑（双向复制）。改为网状后（北美⇄欧洲、欧洲⇄亚太、北美⇄亚太...），复制规则从4条暴涨到20条（n×(n-1)=5×4=20），且出现了"A同步镜像到B，B又把它同步回A"的环路问题，导致同一个镜像反复复制。

**痛点四：删除同步的策略困惑。** 北美团队因合规要求需要清理带高危CVE的旧镜像——在北美Harbor删除了这些镜像。但因为复制规则配置了`deletion: false`（默认值，只增不删），其他4个Region中的高危镜像原封不动。安全部门在审计中问："这个漏洞镜像到底在哪些Region还存在？为什么删了北美版本，欧洲的版本还在？"

本章将深度解析Harbor复制引擎的架构设计、适配器模式、复杂复制拓扑配置以及生产环境的监控和排障方案。

## 2 项目设计——剧本式交锋对话

**场景：全球基础设施SRE团队视频会议，讨论多Region镜像复制策略。背景是Harbor复制任务的SLA告警。**

**小胖**（拿着咖啡，语气随意）："复制有啥难的？不就是把文件从A服务器拷贝到B服务器吗？rsync能干的事Harbor复制引擎能高级到哪去？我写个Shell脚本都能干。"

**大师**："小胖，rsync拷贝的是文件系统上的文件，Harbor复制引擎拷贝的是**OCI标准制品**——这不是简单拷贝Blob。复制引擎需要处理：Manifest List多架构同步（amd64+arm64同步但保持各自digest关联）、标签冲突（目标已有同名标签时是覆盖还是跳过）、元数据关联（扫描报告、签名、SBOM是否跟镜像一起同步）、增量复制（只同步差异Blob）——这些rsync一个都搞不定。"

大师在白板上画出复制引擎的架构：

```

│  │ • 仓库名 │    │   映射   │    │ • 事件驱动 │                │
```

"这个架构的精妙之处在于**适配器模式**——复制引擎核心逻辑不变，通过不同的Adapter适配不同的目标Registry类型。理论上你可以把一个Docker Registry的镜像复制到任何兼容OCI的Registry——Docker Hub、Harbor、AWS ECR、GCR、ACR等。"

**技术映射**：复制引擎的源码结构：
- 适配器接口定义：`src/controller/replication/adapter.go`——定义了`Adapter`接口，包含`PrepareForPush(artifacts)`、`PushManifest()`、`PushBlob()`等方法
- 具体适配器实现：`src/controller/replication/docker_registry_adapter.go`支持Docker Registry V2 API
- 过滤器逻辑：`src/controller/replication/flow/`——实现Filter→Mapper→Scheduler管道
- 任务调度：`src/jobservice/job/impl/replication/transfer.go`——执行实际的镜像传输

**小胖**："那那个'环路问题'——A同步到B，B同步回A——怎么解决？我们之前就碰到过——同一个镜像被复制了两三遍。"

**大师**："Harbor内置了**环路检测（Loop Detection）**机制，利用OCI Annotation实现。每个复制任务在目标Registry写入制品的Manifest时，会自动添加一个特殊的annotation："

```
OCI Manifest中自动添加的annotation：
{
  "annotations": {
    "io.goharbor.replication.source": "https://harbor-us.company.com",
    "io.goharbor.replication.source_artifact": "sha256:abc123...",
    "io.goharbor.replication.time": "2025-12-22T08:00:00Z"
  }
}
```

"当这个制品再次被作为复制源时，Harbor的Filter检查该annotation——发现`io.goharbor.replication.source`的值等于自己的Registry URL，说明这个镜像就是从自己这里复制出去的——跳过，不再复制回去。这样就避免了无限循环。"

```
具体流程：
A push镜像（annotation: {"source": "harbor-us"}）
  → 复制规则触发 → 复制到B（B上的Manifest含 annotation: {"source": "harbor-us"}）
    → B也有一条复制规则（源过滤匹配该仓库）
    → 但Filter检查 annotation: source == "harbor-us"
    → 发现这镜像就是从A来的 → 跳过 → 不会复制回A
```

**技术映射**：环路检测在`src/controller/replication/transfer/checker.go`中实现，核心函数`isFromSourceRegistry()`通过读取制品的Manifest annotations来判断来源。annotation常量定义在`src/controller/replication/model.go`: `const annotationReplicationSource = "io.goharbor.replication.source"`。

**小白**："那复制任务的失败重试机制是怎样的？你们之前说112个失败任务手动重试了3小时——能不能区分'可重试的错误'和'不可重试的错误'？Harbor自动重试的边界在哪里？"

**大师**："这是生产环境最关键的知识点——对待失败任务要做'分诊'。Harbor内置了重试机制，但默认次数有限，且只有部分错误类型会重试："

| 错误类型 | 典型HTTP状态 | 是否自动重试 | 重试策略 | 处理建议 |
|---------|------------|------------|---------|---------|
| 网络超时 | dial tcp timeout | ✅ 是 | 3次 / 间隔30s/1min/2min | 网络恢复后自动成功 |
| DNS解析失败 | no such host | ✅ 是 | 3次 | DNS恢复后自动成功 |
| 503 Service Unavailable | 503 | ✅ 是 | 3次 | 目标Registry过载——降速或等待 |
| 401 Unauthorized | 401 | ❌ 否 | 立即失败 | 凭证错误——重试也没用 |
| 403 Forbidden | 403 | ❌ 否 | 立即失败 | 权限配置错误——需人工修复 |
| 目标命名空间不存在 | 404 | ❌ 否 | 立即失败 | 需先创建目标项目 |
| 目标配额已满 | 413 | ❌ 否 | 立即失败 | 需扩容目标存储 |
| Manifest冲突 | 409 | ❌ 否 | 立即失败 | 目标已有不可覆盖的制品 |

"理解这个分诊逻辑后，运维的正确处理流程是：先批量查询所有失败任务的错误信息，按错误类型分组——网络类错误等自动重试，权限/配置类错误手动修复后再重试。不要让权限错误反复重试——浪费JobService Worker时间。"

**技术映射**：重试逻辑在`src/jobservice/job/impl/replication/transfer.go`的`MaxRetry()`方法中控制。`ShouldRetry()`函数根据适配器返回的error类型判断是否可重试——HTTP错误通过`statusCode`分类，not found/unauthorized直接标记为不可重试。

### 3.2 创建复杂复制规则（多维度过滤器）

**目标**：创建精确过滤的复制规则——仅同步特定项目的特定标签到特定目标Region。

```bash
# 复制规则：同步 order-platform 和 payment-platform 的 release-* 和 latest 标签
# 目标：亚太Region（registry_id=3）
TARGET_REGISTRY_ID=3

cat <<EOF > replication-policy-ap.json
{
  "name": "core-services-to-asia",
  "description": "Sync core service release images to APAC region",
  "dest_registry": {"id": $TARGET_REGISTRY_ID},
  "dest_namespace": "order-platform",
  "dest_namespace_replace_count": 0,
  "trigger": {
    "type": "event_based",
    "trigger_settings": {"cron": ""}
  },
  "filters": [
    {
      "type": "resource",
      "value": "{order-platform,pymt-platform}/**"
    },
    {
      "type": "tag",
      "value": "{release-*,latest}"
    }
  ],
  "enabled": true,
  "override": true,
  "deletion": false
}
EOF

curl -X POST -u "$ADMIN_AUTH" \
  -H "Content-Type: application/json" \
  -d @replication-policy-ap.json \
  "$SOURCE_HARBOR/api/v2.0/replication/policies" | jq '.'

# 预期输出：返回创建的Policy ID和状态
# {
#   "id": 7,
#   "name": "core-services-to-asia",
#   "enabled": true,
#   "trigger": {"type": "event_based"}
# }
```

**过滤器语法说明**：
- `{order-platform,pymt-platform}/**`：匹配这两个项目下的所有仓库（花括号内用逗号分隔，**无空格**）
- `{release-*,latest}`：匹配以`release-`开头的标签和`latest`标签
- `**`：通配所有（仓库名过滤器中的`**`匹配任意子路径，标签过滤器中的`**`匹配所有标签）
- 注意：过滤器之间是AND关系——必须同时满足所有过滤器条件

### 3.4 批量分类重试失败任务

**目标**：不是盲目全部重试——先按错误分类，只重试可恢复的错误类型。

```bash
#!/bin/bash
# classify-and-retry.sh — 智能重试：区分可恢复和不可恢复的失败

EXECUTION_ID=$1
ADMIN_AUTH="admin:Str0ng@Admin2024"
HARBOR_URL="https://harbor-us.company.com"

echo "=== Analyzing Failed Tasks for Execution #$EXECUTION_ID ==="

# 获取所有失败任务
FAILED_TASKS=$(curl -s -u "$ADMIN_AUTH" \
  "$HARBOR_URL/api/v2.0/replication/executions/$EXECUTION_ID/tasks?page=1&page_size=500" | \
  jq -r '.[] | select(.status=="Failed") | 
    {id: .id, resource: .src_resource, error: (.status_text // "unknown")}')

# 分类统计
echo "=== Failure Classification ==="
echo "$FAILED_TASKS" | jq -r '.error' | sort | uniq -c | sort -rn

# 重试网络类错误（timeout/connection refused/DNS）
echo "=== Retrying Recoverable Failures ==="
RECOVERABLE_COUNT=0
UNRECOVERABLE_COUNT=0

while IFS= read -r task_json; do
  task_id=$(echo "$task_json" | jq -r '.id')
  error=$(echo "$task_json" | jq -r '.error')
  resource=$(echo "$task_json" | jq -r '.resource')
  
  # 判断是否可恢复
  if echo "$error" | grep -qiE "timeout|connection refused|dns|temporarily unavailable|503"; then
    echo "  [RETRY] Task $task_id: $resource (error: $error)"
    # Harbor API暂不支持单任务重试，需要通过更新任务状态触发
    curl -s -X PUT -u "$ADMIN_AUTH" \
      "$HARBOR_URL/api/v2.0/replication/executions/$EXECUTION_ID/tasks/$task_id" \
      -H "Content-Type: application/json" \
      -d '{"status": "Pending"}' > /dev/null
    ((RECOVERABLE_COUNT++))
  else
    echo "  [SKIP] Task $task_id: $resource — unrecoverable (error: $error)"
    ((UNRECOVERABLE_COUNT++))
  fi
done < <(echo "$FAILED_TASKS" | jq -c '.')

echo "=== Summary ==="
echo "Auto-retried: $RECOVERABLE_COUNT tasks"
echo "Skipped (unrecoverable): $UNRECOVERABLE_COUNT tasks — manual fix required"
```

### 3.6 监控特定镜像的复制延迟

**目标**：自动检测关键镜像从源到目标的复制完成时间，延迟超阈值则告警。

```bash
#!/bin/bash
# monitor-replication-lag.sh — 持续监控关键镜像的复制延迟

IMAGE="order-platform/order-service:release-2.5"
SRC_HARBOR="https://harbor-us.company.com"
DST_HARBOR="https://harbor-ap.company.com"
ADMIN_AUTH="admin:Str0ng@Admin2024"
LAG_THRESHOLD=120  # 延迟阈值（秒）：超过2分钟触发告警

echo "=== Checking replication lag for $IMAGE ==="

# 获取源镜像digest
SRC_DIGEST=$(curl -s -u "$ADMIN_AUTH" \
  "$SRC_HARBOR/api/v2.0/projects/order-platform/repositories/order-service/artifacts/release-2.5" | \
  jq -r '.digest')

if [ -z "$SRC_DIGEST" ] || [ "$SRC_DIGEST" = "null" ]; then
  echo "ERROR: Cannot resolve source artifact. Does the tag exist?"
  exit 1
fi

# 获取目标镜像digest
DST_DIGEST=$(curl -s -u "$ADMIN_AUTH" \
  "$DST_HARBOR/api/v2.0/projects/order-platform/repositories/order-service/artifacts/release-2.5" | \
  jq -r '.digest')

if [ "$SRC_DIGEST" = "$DST_DIGEST" ] && [ -n "$SRC_DIGEST" ] && [ "$SRC_DIGEST" != "null" ]; then
  echo "✅ SYNC OK: $IMAGE"
  echo "   Source digest: ${SRC_DIGEST:0:19}..."
  echo "   Target digest: ${DST_DIGEST:0:19}..."
else
  echo "❌ SYNC DELAYED: $IMAGE"
  echo "   Source digest: ${SRC_DIGEST:0:19}..."
  echo "   Target digest: ${DST_DIGEST:0:19}..."

  # 检查最近一次复制任务的完成时间
  LAST_SYNC=$(curl -s -u "$ADMIN_AUTH" \
    "$SRC_HARBOR/api/v2.0/replication/executions?page=1&page_size=5" | \
    jq -r '.[] | select(.status=="Succeed") | .end_time' | head -1)

  echo "   Last successful sync: $LAST_SYNC"
  
  # 如果LAST_SYNC超过阈值，触发告警
  if [ -n "$LAST_SYNC" ]; then
    NOW=$(date -u +%s)
    LAST=$(date -u -d "$LAST_SYNC" +%s 2>/dev/null || date -u -j -f "%Y-%m-%dT%H:%M:%SZ" "$LAST_SYNC" +%s)
    LAG=$((NOW - LAST))
    if [ $LAG -gt $LAG_THRESHOLD ]; then
      echo "   🚨 ALERT: Replication lag exceeds ${LAG_THRESHOLD}s (current: ${LAG}s)"
      # 集成告警系统（PagerDuty/Slack/邮件）
    fi
  fi
fi
```

### 3.7 可能遇到的坑

**坑1：复制过滤器写了正则但所有镜像都被同步了**

| 项目 | 内容 |
|------|------|
| **症状** | 在过滤器资源中写了`{order-platform, payment-platform}/**`，期望只同步这两个项目。但实际所有项目的镜像都被复制了。 |
| **根因** | filter的JSON格式写错了——在花括号内的逗号后多加了一个空格：`{order-platform, payment-platform}`。Harbor的过滤器解析器会把空格也当作匹配模式的一部分，导致花括号展开失败，退化为通配。 |
| **解决** | 去掉花括号内的所有空格。正确写法：`{order-platform,pymt-platform}/**`。测试方法：在Portal的复制规则编辑页面的"测试过滤器"功能验证匹配结果。 |

**坑2：复制后目标标签digest不一致（但功能正常）**

| 项目 | 内容 |
|------|------|
| **症状** | 源Harbor上`order-service:release-2.5`的digest是`sha256:a1b2...`，目标Harbor上同一个标签的digest是`sha256:c3d4...`。但`docker pull`能成功且运行正确。 |
| **根因** | 这是Manifest List（多架构索引）的正常现象。源Harbor的Manifest List是平台无关的索引digest，目标Harbor在复制时可能只复制了特定平台的子Manifest（取决于`override`选项和复制策略）。子架构digest仍然一致。 |
| **解决** | 确认不影响业务。如果需要完全一致：复制时设置`"override": true`，并确保源和目标都支持Manifest List。用`docker buildx imagetools inspect`查看多架构索引。 |

**坑3：事件驱动复制不触发**

| 项目 | 内容 |
|------|------|
| **症状** | push了一个release-2.6镜像，Portal中"事件驱动"的复制规则状态为Enabled，但并没有自动触发复制。 |
| **根因** | （1）过滤器未匹配：release-2.6不符合`{release-*,latest}`以外的其他标签。（2）JobService Worker没有空闲——所有Worker都在处理其他复制/扫描任务，任务在Redis队列中等待。（3）目标Registry不可达——Core先验证目标健康状态，不健康则不入队。 |
| **解决** | 手动触发一次复制规则：`curl -X POST -u admin:pass "$HARBOR/api/v2.0/replication/policies/{id}/replication"`。如果手动触发成功但事件驱动不生效，检查JobService Worker状态和Redis队列。 |

**坑4：删除同步开启后目标Harbor标签被意外删除**

| 项目 | 内容 |
|------|------|
| **症状** | 团队开启了`"deletion": true`，期望源删了目标也删。但某天CI流水线误删了源上的`latest`标签并重新推送了新版本——目标Harbor上的`latest`也被删，导致100+个K8s Pod触发`ImagePullBackOff`。 |
| **根因** | `deletion: true`是精确的"镜像危险"——源上任何标签/Artifact的删除都会忠实地同步到目标。没有"安全网"——没有"至少保留一个标签"或"延迟删除"的保护机制。 |
| **解决** | （1）灾备场景永远设置`deletion: false`。（2）即使需要开启，应该在目标Harbor上额外配置Tag保留策略（Retention Policy）作为安全网。（3）用定时全量补偿代替删除同步——源删了，目标不删，靠审计发现差异。 |

## 4 项目总结

### 4.1 四种复制触发模式完整对比

| 维度 | 事件驱动（Push） | 事件驱动（Pull） | 定时全量（Scheduled） | 手动触发 |
|------|----------------|-----------------|---------------------|---------|
| 触发条件 | 源Harbor有push事件 | 定时轮询源Registry | Cron表达式指定时刻 | 人工在Portal或API操作 |
| 延迟 | 10-60秒（含JobService调度延迟） | 视Cron间隔（最少1分钟） | 按Cron计划执行 | 几乎立即 |
| 网络要求 | 源Harbor可出网到目标 | 目标Harbor可出网到源 | 同Push | 同Push |
| 增量/全量 | 增量（仅push的制品） | 全量（每次轮询对比差异） | 全量（每次全量对比） | 全量 |
| 资源消耗 | 低（只传diff） | 高（每次都做diff对比） | 高 | 高 |
| 适用场景 | 主要同步方式——实时性高 | 源不可出网（如生产网仅入站） | 补偿机制——防止遗漏 | 应急/测试 |
| JobService Worker占用 | 1个Worker/trigger | 1个Worker/poll_interval | 1个Worker/Cron触发 | 1个Worker/手动触发 |

### 4.2 复制拓扑对比

| 拓扑模式 | 结构 | 规则数量 | 延迟 | 优点 | 缺点 |
|---------|------|---------|------|------|------|
| Hub-Spoke（星型） | Hub→N个Spoke | N条 | Hub到Spoke单向 | 简单、管理成本低、无环路 | Hub单点、Hub带宽瓶颈 |
| Full Mesh（全互联） | 每个节点双向互联 | N×(N-1)条 | 任意两节点间最低 | 无中心瓶颈、任一Region自治 | 规则多(20条)、需环路检测 |
| Ring（环形） | A→B→C→D→A | N条 | 累积延迟（最长N-1跳） | 规则少、带宽均匀 | 单点故障影响下游、延迟高 |
| Tree（树形） | 层级结构 | 取决于层级数 | 取决于层级深度 | 适合地理分层（大洲→国家→城市） | 上层故障影响下层 |

### 4.3 适用场景与不适用场景

**适用场景：**
- **多Region镜像分发**：核心镜像由中心CI/CD构建，自动分发到全球Region——降低跨洲pull延迟
- **混合云灾备**：主Harbor在私有云，备份Harbor在公有云——私有云故障时快速切换
- **离线环境同步**：源Harbor在公网，目标Harbor在隔离网络——通过Pull模式拉取
- **大促前预热**：双11/618前手动触发全量同步，确保所有Region镜像就绪
- **跨团队镜像共享**：Team A的Harbor复制关键镜像到Team B的Harbor，实现镜像级协作

**不适场景：**
- **实时同步要求（<1秒）**：复制引擎本质是异步的——从push到目标可pull中间有10-60秒延迟（JobService调度+网络传输）。证券交易等毫秒级同步场景不适用
- **跨平台类型转换**：Harbor复制引擎不支持"Docker Image→OCI Bundle"或"Chart→Image"的类型转换——只做同类型制品的迁移

### 4.4 注意事项

1. **复制规则过滤器中的花括号内绝不能有空格**——`{a,b}/**`正确，`{a, b}/**`错误。空格会被当作字面量匹配，导致花括号展开失效
2. **`deletion: true`是一把双刃剑**——源上误删的标签会立即同步到目标。建议灾备场景始终设为`false`，配合定时补偿同步保持数据完整
3. **复制任务失败默认只自动重试3次**——且仅限网络类错误。权限/配置类错误不会自动重试——运维需要手动分类处理失败任务
4. **全Mesh拓扑下规则数量 = n×(n-1)**——5个Region就是20条规则。务必用脚本/代码管理而非Portal手动配置，否则易出错且不易维护
5. **事件驱动复制依赖JobService的Worker数**——如果所有Worker都在处理慢任务（如GB级镜像复制），新push镜像的复制任务会排队等待。生产环境建议至少为复制Worker预留2个专用Worker

### 4.5 常见故障速查表

| 故障现象 | 根因 | 快速解决 |
|---------|------|---------|
| push后复制不触发 | JobService Worker全忙或过滤器不匹配 | `docker exec redis redis-cli LLEN job:replication:queue` 查看队列 |
| 复制任务大量超时 | 跨Region网络波动 | 检查网络延迟 + 考虑专线/加速 |
| 环路：同一镜像被反复复制 | 环路检测失效（旧版Harbor） | 升级到v2.5+ 或手动检查annotation |
| 目标命名空间不存在导致失败 | 目标项目未创建 | 先用API在目标Harbor创建同名项目 |
| 目标Harbor存储满导致复制失败 | 目标Registry的存储容量不足 | GC释放空间 + 扩容存储 |
| 过滤器测试通过但实际不生效 | 测试只测字符串匹配，不测权限/网络 | 参数确认后手动触发一次复制规则 |
