# 第23章：漏洞扫描系统架构与自定义策略

## 1 项目背景

某大型金融科技企业的安全部门在年度审计中发现了Harbor默认扫描配置的严重不足。该公司管理着15000+容器镜像、日均推送上百个新版本，漏洞扫描是CI/CD流水线中的强制门禁。然而，默认配置远无法满足企业级安全需求。

**痛点一：单一扫描器无法覆盖全栈安全。** Trivy对操作系统层软件包（apt/rpm/apk）的CVE检测非常出色，扫描速度快，但对Java应用层的深层依赖（如传递依赖中的log4j、fastjson等）检测不够深入。安全团队要求同时运行Trivy（OS层快速扫描）和Snyk（应用层深度分析），任何一方发现Critical级别CVE就阻止部署。但Harbor的默认配置只支持选择一个"默认扫描器"——如何并跑多扫描器成为难题。

**痛点二：CVE策略过于粗糙。** Harbor的Prevent Vulnerability策略只支持按CVE级别（Critical/High/Medium/Low）阻止部署——这是一个粗粒度的"全或无"策略。真实业务场景远更复杂：基础镜像`openjdk:17`自带的2个Low级别CVE不应该阻止所有Java服务部署；某个CVE-2023-4586虽然在Trivy中被标记为"High"，但安全团队评估后确认"在当前业务场景下不存在利用路径"，需要单独豁免此CVE而不豁免所有High级别。更棘手的是——一个新发布的CVE尚未被NVD评分，Severity为"Unknown"，应该放行还是阻止？

**痛点三：扫描超时导致CI流水线大面积卡死。** 一个1.5GB的AI训练镜像，Trivy扫描需要8分钟。Harbor默认的JobService超时只有5分钟——超过时间扫描任务被标记为"Error"，但Worker进程仍然被占用没有释放。结果：并发扫描数被快速耗尽，后续的扫描请求全部进入Pending队列，CI流水线中等待Scan Gate通过的步骤大面积超时失败。运维临时将超时调到15分钟，但这只是治标不治本。

**痛点四：私有CVE数据库需求。** 公司采购了多个商业中间件（Oracle WebLogic、IBM MQ、TIBCO EMS），这些商业产品有自己发布的漏洞公告，但不在Trivy的公共CVE数据库（NVD/GHSA/GLAD）中。安全部门维护了一份内部CVE清单（约200条），希望能注入到Harbor的扫描结果中，实现统一展示和统一门禁。

本章将深入Harbor漏洞扫描系统的可插拔架构，从Scanner Adapter接口规范出发，逐步实现自定义扫描器开发、多扫描器策略编排、CVE白名单精细化管理，以及在CI/CD中的完整集成。

## 2 项目设计——剧本式交锋对话

**场景：安全部门会议室，安全负责人老王与技术团队讨论扫描策略的精细化配置。白板上画满了扫描器架构图。**

**小胖**（运维工程师）："CVE扫描不都一个样吗？扫出来有高危就阻止，没高危就放行——还要什么自定义策略？跟我们小区门禁一样，有门禁卡就进，没有就拦。"

**老王**（安全负责人，CISSP持证）："小胖，你这个门禁卡的比喻太简化了。真实的安全策略比你想的复杂得多。我给你讲三个真实场景：

场景一：基础镜像`openjdk:17`有2个Low级别的CVE——你阻止还是不阻止？阻止了，全公司300个Java服务全部不能部署，业务停摆；不阻止，你那份'零CVE目标'的合规报告怎么写？

场景二：一个High级别的CVE是某个CLI工具的漏洞——但你的容器里根本没有任何调用shell的地方，这个漏洞在你的业务场景下风险为零。你阻止还是不阻止？

场景三：一个新CVE刚被公布，NVD还没来得及给它打分——默认Severity是'Unknown'。你的策略是放行还是阻止？放行有风险，阻止会误伤大量合法镜像。"

**小白**（高级开发，喜欢深挖原理）："老王，我对Scanner Adapter的架构很好奇。Harbor是怎么做到'可插拔'的？任意一个扫描器，只要实现了几个API就能接入？这个接口契约是什么？"

**老王**："这正是Harbor扫描框架的精妙之处。它定义了一套标准化的HTTP API契约——任何扫描器只需要实现这个契约，就能注册为Harbor的扫描器。这就像USB接口：不管你是什么设备（U盘、键盘、摄像头），只要实现了USB协议就能插上去。"

"Scanner Adapter需要实现4个核心端点："

| API端点 | HTTP方法 | 用途 | 关键要求 |
|---------|---------|------|---------|
| `/api/v1/metadata` | GET | 返回扫描器的能力描述（名称、版本、支持的文件类型） | 必须包含`properties.harbor.scanner-adapter/scanner-type`字段 |
| `/api/v1/scan` | POST | 接受扫描请求（Artifact信息+Registry临时凭证） | 必须支持Registry认证（从`authorization`字段提取Bearer Token） |
| `/api/v1/scan/{scan_request_id}/report` | GET | 返回标准化Harbor格式的漏洞报告 | 报告格式必须严格遵循Harbor的Report Schema |
| `/api/v1/metrics` | GET | （可选）返回扫描器运行指标 | 用于Harbor监控面板展示 |

**技术映射**：Scanner Adapter接口定义在Harbor源码的`src/pkg/scan/rest/v1/spec.go`中。任何符合这个HTTP API规范的外部服务都可以通过`POST /api/v2.0/scanners`注册为Harbor的扫描器。Harbor Core通过HTTP与Adapter通信，不关心Adapter后端用的是什么语言或什么扫描引擎。

**小胖**："那多扫描器同时用怎么搞？一个镜像扫两次？不会冲突吗？你们这个设计比我家的万能遥控器还复杂啊。"

**老王**（笑）："万能遥控器的比喻不错——但关键是，Harbor不是'同时用多个扫描器'，而是"每个项目选一个默认扫描器+按需触发其他扫描器"。每个项目可以设置一个默认扫描器——每次push自动触发。但你可以通过API对同一个Artifact手动触发多次扫描（用不同的扫描器），结果独立存储。"

"多扫描器策略最佳实践是这样编排的：

- **第一层：快速门禁（每次Push）** → Trivy做OS包扫描（30秒-2分钟），阻止Critical/High CVE
- **第二层：深度分析（定时任务）** → Snyk做应用依赖分析（5-15分钟），每日凌晨扫描所有生产镜像
- **第三层：合规审计（每周）** → 自研扫描器注入私有CVE数据，生成合规报告

浅层门禁用Trivy结果（速度快），深度报告用Snyk结果（覆盖全），私有CVE用自研扫描器注入。"

**小白**："我注意到扫描器适配器返回的报告必须遵循Harbor的标准化格式。这个格式的Schema在哪里定义？有哪些必填字段？如果我写的Adapter返回了一个格式不对的报告，Harbor会怎么处理？"

**老王**："好问题。Harbor的扫描报告Schema定义在`src/pkg/scan/v1/api.go`中。核心字段如下："

```json
{
  "generated_at": "2024-01-16T10:30:00Z",
  "artifact": {
    "repository": "order-platform/order-service",
    "digest": "sha256:abc123..."
  },
  "scanner": {
    "name": "Trivy",
    "vendor": "Aqua Security",
    "version": "0.50.0"
  },
  "severity": "High",
  "vulnerabilities": [
    {
      "id": "CVE-2023-4586",
      "package": "glibc",
      "version": "2.31-0ubuntu9.12",
      "fix_version": "2.31-0ubuntu9.14",
      "severity": "High",
      "description": "A vulnerability was found in glibc...",
      "links": ["https://nvd.nist.gov/vuln/detail/CVE-2023-4586"],
      "cvss_score_v3": 7.5,
      "cvss_vector_v3": "CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N",
      "preferred_cvss": {
        "score_v3": 7.5,
        "vector_v3": "CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N"
      },
      "cwe_ids": ["CWE-20"]
    }
  ]
}
```

"如果Adapter返回的报告格式不符合Schema，Harbor Core在解析时会返回错误，扫描状态标记为`Error`。Core对报告的解析是严格的——extra fields会被忽略但不会报错，missing required fields会直接导致失败。"

**小胖**："我还有个问题——如果我们公司的业务场景是'CVE数目不能增加'？比如镜像v2.3.0有10个CVE，v2.4.0的CVE不能超过10个——扫描本身不支持这种增量对比吧？"

**老王**："对，这个是业务逻辑，不在扫描器层面实现。你需要自己开发一个'扫描结果对比器'——从Harbor API获取v2.3.0和v2.4.0的扫描报告，对比CVE列表，判断是否新增。这个对比器可以作为一个独立的CI步骤或Harbor的Webhook消费者。Harbor提供了完善的API来获取历史扫描报告，你基于这些API做二次开发就行。"

## 3 项目实战

### 3.1 环境要求

| 组件 | 版本要求 | 说明 |

### 3.2 第一步：开发自定义扫描器适配器

**目标**：编写一个最小可用的Scanner Adapter，实现4个核心API端点，能够成功注册到Harbor。

```python
#!/usr/bin/env python3
# ══════════════════════════════════════════════
# custom_scanner_adapter.py
# 最小化 Scanner Adapter 实现
# 支持：metadata / scan / report / metrics
# ══════════════════════════════════════════════

from flask import Flask, request, jsonify
import uuid
import time
from datetime import datetime, timezone

app = Flask(__name__)

# 内存存储（生产环境应使用数据库或Redis）
scan_results = {}

@app.route('/api/v1/metadata', methods=['GET'])
def metadata():
    """返回扫描器能力描述 — Harbor注册扫描器时首先调用此端点"""
    return jsonify({
        "scanner": {
            "name": "CustomEnterpriseScanner",
            "vendor": "MyFintechCompany",
            "version": "1.2.0"
        },
        "capabilities": [{
            "consumes_mime_types": [
                "application/vnd.docker.distribution.manifest.v2+json",
                "application/vnd.oci.image.manifest.v1+json",
                "application/vnd.oci.image.index.v1+json"
            ],
            "produces_mime_types": [
                "application/vnd.scanner.adapter.vuln.report.harbor+json; version=1.0",
                "application/vnd.security.vulnerability.report; version=1.1"
            ]
        }],
        "properties": {
            # ⚠️ 关键字段：缺失此字段将导致无法设为默认扫描器
            "harbor.scanner-adapter/scanner-type": "os",
            "harbor.scanner-adapter/vulnerability-database-update-time": 
                datetime.now(timezone.utc).isoformat(),
            # 可选：声明支持的额外功能
            "harbor.scanner-adapter/sbom": "true",
            "harbor.scanner-adapter/fixable": "true"
        }
    })

@app.route('/api/v1/scan', methods=['POST'])
def scan():
    """接受扫描请求 — Harbor每次触发扫描时调用"""
    scan_request = request.json
    
    # Harbor传递的关键信息：
    # - registry.url: Registry地址
    # - registry.authorization: 临时Bearer Token（用于拉取镜像）
    # - artifact.repository: 仓库名
    # - artifact.digest: 内容摘要
    registry_url = scan_request.get('registry', {}).get('url')
    registry_token = scan_request.get('registry', {}).get('authorization')
    artifact_repo = scan_request.get('artifact', {}).get('repository')
    artifact_digest = scan_request.get('artifact', {}).get('digest')
    
    scan_id = str(uuid.uuid4())
    
    # 记录扫描请求（生产环境：使用Registry Token拉取镜像 → 解压文件系统 → 执行扫描引擎）
    scan_results[scan_id] = {
        "status": "pending",
        "artifact": {"repository": artifact_repo, "digest": artifact_digest},
        "created_at": datetime.now(timezone.utc).isoformat()
    }
    
    # ⚠️ 简化示例：直接返回结果（生产环境应异步扫描 + 轮询report端点）
    # 异步模式：Adapter返回 "accepted"，后台扫描，客户端轮询 /report
    scan_results[scan_id]["status"] = "finished"
    scan_results[scan_id]["report"] = generate_mock_report(artifact_repo, artifact_digest)
    
    return jsonify({"id": scan_id}), 202  # 202 Accepted（异步模式）

@app.route('/api/v1/scan/<scan_request_id>/report', methods=['GET'])
def report(scan_request_id):
    """返回标准化扫描报告 — Harbor获取扫描结果时调用"""
    result = scan_results.get(scan_request_id)
    
    if not result:
        return jsonify({"error": "scan request not found"}), 404
    
    if result["status"] == "pending":
        # 扫描尚未完成 — Harbor会定期重试此端点
        return jsonify({}), 302  # 临时重定向，Harbor会稍后重试
    
    return jsonify(result["report"])

@app.route('/api/v1/metrics', methods=['GET'])
def metrics():
    """返回扫描器运行指标（可选端点）"""
    return jsonify({
        "scans_completed": len([r for r in scan_results.values() if r["status"] == "finished"]),
        "scans_pending": len([r for r in scan_results.values() if r["status"] == "pending"]),
        "uptime_seconds": time.time() - start_time
    })

def generate_mock_report(repo, digest):
    """生成符合Harbor Schema的标准化扫描报告"""
    return {
        "generated_at": datetime.now(timezone.utc).strftime("%Y-%m-%dT%H:%M:%SZ"),
        "artifact": {
            "repository": repo,
            "digest": digest
        },
        "scanner": {
            "name": "CustomEnterpriseScanner",
            "vendor": "MyFintechCompany",
            "version": "1.2.0"
        },
        "severity": "Medium",
        "vulnerabilities": [
            {
                "id": "CVE-2024-0001",
                "package": "openssl",
                "version": "3.0.7-0ubuntu1",
                "fix_version": "3.0.7-0ubuntu2",
                "severity": "Medium",
                "description": "OpenSSL: Excessive time spent checking DH keys",
                "links": ["https://nvd.nist.gov/vuln/detail/CVE-2024-0001"],
                "cvss_score_v3": 5.3,
                "cvss_vector_v3": "CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:L",
                "preferred_cvss": {
                    "score_v3": 5.3,
                    "vector_v3": "CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:L"
                },
                "cwe_ids": ["CWE-327"],
                "fixable": True
            },
            {
                "id": "COMPANY-PRIVATE-2024-001",  # 私有CVE编号
                "package": "oracle-weblogic",
                "version": "14.1.1.0",
                "fix_version": "14.1.1.1",
                "severity": "Critical",
                "description": "[Private] WebLogic T3 Protocol Deserialization RCE",
                "links": [],
                "cvss_score_v3": 9.8,
                "cwe_ids": ["CWE-502"],
                "fixable": True
            }
        ]
    }

if __name__ == '__main__':
    start_time = time.time()
    app.run(host='0.0.0.0', port=8080, debug=False)
```

### 3.3 第二步：注册自定义扫描器到Harbor

**目标**：将自定义扫描器注册到Harbor，并在项目中使用它。

```bash
# ══════════════════════════════════════════════
# 启动自定义扫描器
# ══════════════════════════════════════════════
python3 custom_scanner_adapter.py &
# 验证启动
curl -s http://localhost:8080/api/v1/metadata | jq '.scanner.name'
# 预期输出："CustomEnterpriseScanner"

# ══════════════════════════════════════════════
# 注册扫描器到Harbor
# ══════════════════════════════════════════════
HARBOR_URL="https://harbor.company.com"

# 1. 先查看已有扫描器列表
curl -s -u admin:Str0ng@Admin2024 \
  "${HARBOR_URL}/api/v2.0/scanners" | jq '.[].name'
# 预期输出：
# "Trivy"
# (可能还有其他已注册的扫描器)

# 2. 注册自定义扫描器
curl -X POST -u admin:Str0ng@Admin2024 \
  -H "Content-Type: application/json" \
  -d '{
    "name": "CustomEnterpriseScanner",
    "description": "Company private CVE scanner with WebLogic/IBM MQ support",
    "url": "http://192.168.1.50:8080",
    "type": "os",
    "auth": "",
    "access_credential": "",
    "skip_cert_verify": false,
    "use_internal_addr": false,
    "disabled": false
  }' \
  "${HARBOR_URL}/api/v2.0/scanners"

# 预期输出（HTTP 201 Created）：
# {"uuid": "scanner-uuid-xxxx"}

# 3. 验证注册成功
curl -s -u admin:Str0ng@Admin2024 \
  "${HARBOR_URL}/api/v2.0/scanners" | jq '.[] | {name: .name, healthy: .health}'
# 预期输出：
# {"name": "Trivy", "healthy": "healthy"}
# {"name": "CustomEnterpriseScanner", "healthy": "healthy"}

# ══════════════════════════════════════════════
# 将自定义扫描器设为项目默认扫描器
# ══════════════════════════════════════════════
PROJECT_ID=1

# 获取扫描器UUID
SCANNER_UUID=$(curl -s -u admin:Str0ng@Admin2024 \
  "${HARBOR_URL}/api/v2.0/scanners" | \
  jq -r '.[] | select(.name=="CustomEnterpriseScanner") | .uuid')

# 设为项目默认扫描器
curl -X PUT -u admin:Str0ng@Admin2024 \
  -H "Content-Type: application/json" \
  -d "{\"scanner_uuid\": \"${SCANNER_UUID}\"}" \
  "${HARBOR_URL}/api/v2.0/projects/${PROJECT_ID}/scanner"
# 预期输出：HTTP 200 OK
```

### 3.4 第三步：配置多扫描器策略

**目标**：编排Trivy（OS层快速扫描）+ Snyk（应用层深度扫描）的双扫描器策略。

```bash
# ══════════════════════════════════════════════
# 策略一：Trivy作为默认扫描器（每次push自动触发）
# ══════════════════════════════════════════════
# Harbor安装时Trivy已默认注册，直接设为项目默认

TRIVY_UUID=$(curl -s -u admin:Str0ng@Admin2024 \
  "${HARBOR_URL}/api/v2.0/scanners" | \
  jq -r '.[] | select(.name=="Trivy") | .uuid')

curl -X PUT -u admin:Str0ng@Admin2024 \
  -H "Content-Type: application/json" \
  -d "{\"scanner_uuid\": \"${TRIVY_UUID}\"}" \
  "${HARBOR_URL}/api/v2.0/projects/${PROJECT_ID}/scanner"

# ══════════════════════════════════════════════
# 策略二：定时触发Snyk深度扫描（每周日凌晨）
# ══════════════════════════════════════════════
# 创建定时扫描任务 — 使用Harbor的Schedule API
curl -X POST -u admin:Str0ng@Admin2024 \
  -H "Content-Type: application/json" \
  -d '{
    "schedule": {
      "type": "Weekly",
      "cron": "0 2 * * 0"
    },
    "parameters": {
      "scanner_uuid": "'${SNYK_UUID}'",
      "scan_all": true
    }
  }' \
  "${HARBOR_URL}/api/v2.0/projects/${PROJECT_ID}/scanner/schedule"

# ══════════════════════════════════════════════
# 策略三：关键版本额外触发Snyk扫描（手动/CI API触发）
# ══════════════════════════════════════════════
REPO="order-platform/order-service"
TAG="v2.4.0"

# 重要：必须先获取Artifact的引用方式
# Harbor中Artifact通过 project_name/repo_name + tag 或 digest 标识
curl -X POST -u admin:Str0ng@Admin2024 \
  -H "Content-Type: application/json" \
  -d "{\"scanner_uuid\": \"${SNYK_UUID}\"}" \
  "${HARBOR_URL}/api/v2.0/projects/${PROJECT_ID}/repositories/${REPO}/artifacts/${TAG}/scan"
# 预期输出：HTTP 202 Accepted — 扫描已提交到JobService队列

# 轮询扫描状态
SCAN_STATUS="Pending"
while [ "$SCAN_STATUS" == "Pending" ] || [ "$SCAN_STATUS" == "Scanning" ]; do
  SCAN_STATUS=$(curl -s -u admin:Str0ng@Admin2024 \
    "${HARBOR_URL}/api/v2.0/projects/${PROJECT_ID}/repositories/${REPO}/artifacts/${TAG}" \
    | jq -r '.scan_overview["application/vnd.scanner.adapter.vuln.report.harbor+json; version=1.0"].scan_status')
  echo "Scan status: $SCAN_STATUS"
  sleep 10
done

echo "Final scan status: $SCAN_STATUS"
# 预期输出：Success / Error
```

### 3.6 第五步：在CI/CD中集成扫描门禁

**目标**：在GitLab CI流水线中等待Harbor扫描完成，根据扫描结果决定是否继续部署。

```yaml
# ══════════════════════════════════════════════
# .gitlab-ci.yml — Harbor 扫描门禁集成
# ══════════════════════════════════════════════
stages:
  - build
  - scan-gate
  - deploy

variables:
  HARBOR_URL: "https://harbor.company.com"
  HARBOR_PROJECT: "order-platform"
  HARBOR_REPO: "order-service"

build-and-push:
  stage: build
  image: docker:24
  script:
    - docker build -t ${HARBOR_URL}/${HARBOR_PROJECT}/${HARBOR_REPO}:${CI_COMMIT_SHORT_SHA} .
    - docker push ${HARBOR_URL}/${HARBOR_PROJECT}/${HARBOR_REPO}:${CI_COMMIT_SHORT_SHA}
  # Harbor自动触发Trivy扫描（项目默认扫描器）

scan-gate:
  stage: scan-gate
  image: alpine:3.19
  before_script:
    - apk add --no-cache curl jq bash
  script:
    - |
      IMAGE_TAG="${CI_COMMIT_SHORT_SHA}"
      MAX_WAIT=600  # 最多等10分钟
      WAITED=0
      POLL_INTERVAL=15

      echo "Waiting for Harbor scan to complete..."

      while [ $WAITED -lt $MAX_WAIT ]; do
        # 调用Harbor API获取扫描结果
        RESPONSE=$(curl -s -u ${HARBOR_USER}:${HARBOR_PASSWORD} \
          "${HARBOR_URL}/api/v2.0/projects/${HARBOR_PROJECT}/repositories/${HARBOR_REPO}/artifacts/${IMAGE_TAG}")

        SCAN_STATUS=$(echo "$RESPONSE" | jq -r '
          .scan_overview["application/vnd.scanner.adapter.vuln.report.harbor+json; version=1.0"]
          .scan_status // "Pending"')

        echo "[${WAITED}s] Scan status: ${SCAN_STATUS}"

        case "${SCAN_STATUS}" in
          "Success")
            echo "✅ Scan completed successfully!"
            # 检查CVE级别
            SEVERITY=$(echo "$RESPONSE" | jq -r '
              .scan_overview["application/vnd.scanner.adapter.vuln.report.harbor+json; version=1.0"]
              .severity')
            echo "Highest severity: ${SEVERITY}"

            # 获取详细CVE报告
            curl -s -u ${HARBOR_USER}:${HARBOR_PASSWORD} \
              "${HARBOR_URL}/api/v2.0/projects/${HARBOR_PROJECT}/repositories/${HARBOR_REPO}/artifacts/${IMAGE_TAG}/additions/vulnerabilities" \
              | jq '.["application/vnd.scanner.adapter.vuln.report.harbor+json; version=1.0"]' \
              > scan-report.json

            # 检查Critical CVE数量
            CRITICAL_COUNT=$(jq '[.vulnerabilities[] | select(.severity=="Critical")] | length' scan-report.json)
            echo "Critical CVEs: ${CRITICAL_COUNT}"

            if [ "${CRITICAL_COUNT}" -gt 0 ]; then
              echo "❌ ${CRITICAL_COUNT} Critical CVE(s) found! Blocking deployment."
              jq '.vulnerabilities[] | select(.severity=="Critical") | {id, package, fix_version}' scan-report.json
              exit 1
            fi
            exit 0
            ;;
          "Error")
            echo "❌ Scan failed! Check Harbor JobService logs."
            exit 1
            ;;
          *)
            sleep $POLL_INTERVAL
            WAITED=$((WAITED + POLL_INTERVAL))
            ;;
        esac
      done

      echo "❌ Scan timed out after ${MAX_WAIT}s!"
      exit 1

deploy:
  stage: deploy
  script:
    - kubectl set image deployment/order-service \
      order-service=${HARBOR_URL}/${HARBOR_PROJECT}/${HARBOR_REPO}:${CI_COMMIT_SHORT_SHA}
  only:
    - main
```

### 3.7 可能遇到的坑

**坑1：自定义扫描器注册后无法设为项目默认扫描器**

现象：扫描器注册成功、health状态为"healthy"，但在项目设置中无法将其选为默认扫描器，下拉列表中看不到它。

根因：扫描器的`/api/v1/metadata`返回值中缺少`properties.harbor.scanner-adapter/scanner-type`字段。Harbor依赖此字段判断扫描器类型（os/image），如果缺失则不显示在可用扫描器列表中。

解决方案：
```python
# 确保metadata返回中包含此字段
@app.route('/api/v1/metadata')
def metadata():
    return jsonify({
        # ... 其他字段 ...
        "properties": {
            "harbor.scanner-adapter/scanner-type": "os"  # 或 "image"
        }
    })
```

**坑2：扫描超时但JobService Worker没有被释放**

现象：大镜像（>3GB）扫描超过5分钟后状态变为"Error"，但JobService的Worker数仍然显示被占满，新扫描请求全部Pending。

根因：Harbor JobService的Worker执行扫描任务是同步阻塞模式。当HTTP请求到Scanner Adapter超时时，Worker内的事务回滚，但Worker本身不会立即释放——它会等待HTTP连接完全超时（默认约30秒的TCP超时）。

解决方案：
```yaml
# harbor.yml — 增大JobService配置
jobservice:
  max_job_workers: 50          # 增加Worker池大小（默认10）
  job_loggers:
    - name: FILE
      level: DEBUG
  
# 或者：开发异步扫描器（推荐）
# Scanner Adapter的/scan端点返回202 Accepted（而非200 OK）
# Adapter后台执行扫描，Harbor定期轮询/report端点
# 这样Worker立即释放，不占用JobService资源
```

**坑3：CVE白名单配置后不生效 — "Unknown"级别的CVE没有被豁免**

现象：在CVE白名单中添加了某个CVE ID，但Harbor Portal的扫描结果中该CVE仍显示为"未豁免"状态，阻止策略仍然生效。

根因分析：
1. **CVE ID大小写不匹配**：Harbor的CVE白名单匹配是精确大小写匹配。Trivy报告中的CVE ID格式为`CVE-YYYY-NNNNN`（全大写），如果你的白名单中写的是`cve-2023-4586`（小写），匹配会失败。
2. **扫描器报告格式差异**：不同扫描器返回的CVE ID字段名不同——Trivy使用`id`，Clair使用`Name`。确保白名单中的CVE ID与扫描器报告中的字段值完全一致。
3. **白名单过期**：如果设置了`expires_at`或项目级`expires_at`，超过过期时间后白名单自动失效。

解决方案：
```bash
# 1. 获取实际扫描报告中的CVE ID格式
curl -s -u admin:Str0ng@Admin2024 \
  "${HARBOR_URL}/api/v2.0/projects/${PROJECT_ID}/repositories/${REPO}/artifacts/${TAG}/additions/vulnerabilities" \
  | jq '.["application/vnd.scanner.adapter.vuln.report.harbor+json; version=1.0"].vulnerabilities[].id'

# 2. 确保白名单中的CVE ID与上面输出的完全一致（包括大小写）
# 3. 重新扫描镜像（白名单不回溯应用到历史扫描结果）
curl -X POST -u admin:Str0ng@Admin2024 \
  "${HARBOR_URL}/api/v2.0/projects/${PROJECT_ID}/repositories/${REPO}/artifacts/${TAG}/scan"
```

**坑4：Trivy扫描失败报"too many open files"**

现象：Trivy Adapter日志显示`failed to scan image: open /path/to/layer.tar: too many open files`。

根因：大型镜像有50+层时，Trivy需要同时打开多个layer文件进行分析，超过了Linux默认的文件句柄限制（ulimit -n 1024）。

解决方案：
```bash
# 在Trivy Adapter的docker-compose配置中增大ulimit
trivy-adapter:
  ulimits:
    nofile:
      soft: 65536
      hard: 65536
```

## 4 项目总结

### 4.1 主流扫描器对比

| 扫描器 | 擅长领域 | 不擅长领域 | 扫描速度 | 内存占用 | 适用场景 |
|--------|---------|-----------|---------|---------|---------|
| Trivy | OS包(apt/rpm/apk)、语言依赖(Java/Go/Node) | .NET/Windows容器、商业中间件 | ⭐⭐⭐⭐⭐ (30s-2min) | 低 (~500MB) | 通用CI门禁、快速扫描 |
| Clair | Alpine/Debian/Red Hat生态系统 | 非主流Linux发行版 | ⭐⭐⭐ (2-8min) | 中 (~1GB) | Red Hat生态、企业级CVE库 |
| Snyk | 应用依赖深度分析、许可证合规 | OS层包CVE | ⭐⭐⭐⭐ (1-5min) | 中 (~800MB) | 商业版全栈安全、合规审计 |
| Grype | SBOM集成、快速上手 | 无自有CVE数据库（依赖上游） | ⭐⭐⭐⭐⭐ (20s-1min) | 低 (~300MB) | SBOM场景、轻量级扫描 |
| Aqua CSP | 运行时安全、合规策略 | 开源场景成本高 | ⭐⭐⭐⭐ | 高 (>1GB) | 商业全栈安全平台 |

### 4.2 自定义策略最佳实践

| 策略目标 | 配置方式 | 粒度 | 适用场景 |
|---------|---------|------|---------|
| 阻止含Critical CVE的镜像部署 | `prevent_vul` + `severity=critical` | 项目级 | 生产环境基础门禁 |
| 豁免特定CVE | `cve-allowlist` 添加CVE ID | CVE级 | 已评估无风险的单个CVE |
| 多扫描器分层扫描 | 项目默认扫描器 + API手动触发 | 项目/Artifact级 | 深度+广度覆盖 |
| 全量镜像周期扫描 | `scanner/schedule` 定时任务 | 项目级 | 检测新发现的CVE（已有镜像） |
| 私有CVE注入 | 开发自定义Scanner Adapter | 全局 | 商业中间件漏洞管理 |
| CVE增量对比 | 自定义CI步骤（API获取历史报告） | Artifact级 | 版本间CVE不增加的合规要求 |

### 4.3 适用场景

1. **金融/政务/医疗等强合规行业**：需要多扫描器分层、私有CVE注入、详细的审计报告。
2. **大规模微服务集群**：500+微服务，需要快速扫描门禁（<2分钟），不能阻塞CI/CD流水线。
3. **多语言技术栈**：Java/Go/Python/Node.js混合，单一扫描器无法覆盖所有语言的依赖分析。
4. **商业软件使用方**：使用WebLogic/IBM MQ等商业中间件，需要管理厂商私有漏洞公告。
5. **安全左移实践**：将安全扫描前置到CI/CD流水线中，实现"每次push必扫、每次部署必验"。

### 4.4 不适用场景

1. **纯Windows容器环境**：当前主流扫描器对Windows容器的支持都有限（Windows容器文件系统结构不同），可能扫描结果不准确。
2. **无互联网访问的内网环境**：Trivy/Clair等需要定期从互联网更新CVE数据库。完全隔离的内网环境需要搭建数据库镜像服务，运维成本高。

### 4.5 注意事项

1. **Scanner Adapter的`/scan`端点必须支持Registry认证**：Harbor通过`registry.authorization`字段传递临时Bearer Token，Adapter需要使用此Token从Registry拉取镜像进行分析。如果Token过期（默认30分钟），大镜像扫描可能中途失败。
2. **扫描报告大小限制**：Harbor对单次扫描报告有大小限制（约10MB）。如果镜像包含数千个CVE，报告可能超过限制导致存储失败。此时考虑分页或精简报告内容。
3. **CVE白名单有有效期且不回溯**：白名单到期后自动失效（即使`expires_at=0`也会受Harbor全局配置影响）。修改白名单后，需要**重新触发扫描**才能对新结果生效——历史扫描结果不受白名单影响。
4. **多扫描器结果独立存储**：Trivy和Snyk的扫描结果不会合并，Harbor Portal上只显示最新一次扫描的结果。如果需要聚合展示，需要自行开发聚合器。
5. **JobService Worker池大小设计**：`max_job_workers`并非越大越好——每个Worker在启动时会创建数据库连接，过多的Worker会耗尽PostgreSQL连接池。建议`Worker数 × 2 < PG max_connections × 0.3`。

### 4.6 常见故障排查表

| 故障现象 | 根因 | 排查命令 | 解决方案 |
|---------|------|---------|---------|
| Scanner注册后健康检查失败 | Adapter URL不可达或返回格式错误 | `curl http://adapter:8080/api/v1/metadata` | 检查Adapter网络和防火墙，验证返回JSON格式 |
| 扫描永远停留在"Pending"状态 | JobService无可用Worker | `curl harbor/api/v2.0/jobs/queues` | 增加Worker数或排查是否有卡死的扫描任务 |
| Trivy扫描不识别私有CVE | 私有CVE数据源未配置 | 检查Trivy的`trivy.db`路径 | 将私有CVE数据注入到Trivy的自定义数据源中 |
| 扫描结果中出现Severity为"Unknown"的CVE | CVE数据库中该漏洞未评分 | `curl harbor/api/.../vulnerabilities \| jq '.vulnerabilities[] \| select(.severity=="Unknown")'` | 在Prevent Policy中决定Unknown的处理方式，或通过白名单管理 |
| CVE白名单中的CVE仍被阻止 | 白名单修改后未重新扫描 | 检查扫描时间戳与白名单修改时间 | 重新触发扫描，等待结果更新 |

### 4.7 深度思考

1. **设计一个"扫描结果聚合器"——同时触发Trivy和Grype扫描同一个Artifact，将两者的CVE结果合并去重（基于CVE ID）、取最高Severity、生成统一的扫描报告。这个聚合器应该注册为Harbor的Scanner Adapter，还是作为一个独立的外部服务？两者的优劣势分别是什么？**

2. **公司合规要求"所有生产镜像的CVE数量与上一版本相比不能增加"。设计一个CI步骤：从Harbor API获取当前版本和上一版本的扫描报告，对比CVE列表，如果新增CVE则阻止部署。考虑场景：如果新增的CVE是因为CVE数据库更新（而非代码变更）导致的，这种情况应该阻止部署吗？如何区分"新发现但早已存在的漏洞"和"新引入代码导致的漏洞"？**
