# 第14章：日志系统与审计追踪

## 1 项目背景

某 800 人规模的金融科技公司的安全部门正在经历季度 SOX 合规审计的"至暗时刻"。

审计师提出了一个看似简单的要求："请提供过去六个月，`order-service` 镜像仓库的完整操作轨迹——包括谁、在什么时间、从哪个 IP、执行了什么操作、操作结果如何。"这个需求背后隐藏着对 Harbor 日志体系的严峻考验。

**痛点一：Portal 日志查询能力极其有限。** 运维团队在 Harbor Portal 的"日志"页面翻了整整 3 个小时后发现：Portal 只展示最近 1000 条记录，而六个月来 `order-service` 项目累积了 27 万条审计日志。更致命的是，Portal 的过滤功能只能按时间范围和项目筛选，不能按"特定用户"或"特定操作类型"过滤。审计师追问："张三是负责 `order-service` 的开发主管，过去三个月他有过哪些 DELETE 操作？"——运维人员只能把 Portal 上的 1000 条记录导出为 CSV，然后打开 Excel 手动筛选。一个本该 2 分钟的查询，活生生变成了 45 分钟的"Excel 考古"。

**痛点二：日志保留时间与合规要求脱节。** Harbor 默认审计日志保留 90 天——这是出于数据库空间考虑。但 SOX 合规要求关键系统的操作日志保留 1-3 年，等保 2.0 要求不少于 6 个月。当审计师要求查看 6 个月前的那次关键镜像删除操作时，运维尴尬地发现——数据库中的 `audit_log` 表已被 Harbor 的自动清理 Job 清空，只剩下容器中早已被 rotate 覆盖的 Nginx 日志。安全部门主管当场暴怒："日志就是证据——证据都没了，你怎么证明那次操作是授权操作而非恶意删除？"

**痛点三：多种日志源分散、格式不统一、无法关联。** Harbor 的一个 `docker push` 操作会同时产生三条日志记录——在三个不同容器中：
- Nginx 容器记录了一条 HTTP 请求日志：`192.168.1.50 - - [15/Mar/2024:08:22:10] "PUT /v2/order-platform/order-service/blobs/uploads/..." 202 0`
- Core 容器记录了认证和授权决策：`[2024-03-15T08:22:10Z] [INFO] token granted for user 'dev01' to 'order-platform/order-service'`
- 审计日志表记录了一行操作轨迹：`username=dev01, operation=PUSH, resource=order-service:v2.4.1, op_time=2024-03-15T08:22:10Z`

三条日志分散在三个位置，没有统一的 Trace ID 关联。运维人员在排查一个"Push 明明成功了但镜像列表看不到"的问题时，需要先查审计日志确认操作发生 → 再查 Core 日志确认认证过程 → 最后查 Nginx 日志确认 HTTP 层面的细节——三个容器之间切来切去，定位一个问题的平均时间是 20-35 分钟。

**痛点四：缺少实时告警，安全事件响应滞后。** 某周五晚上 23:15，离职员工"李四"的 Harbor 账号未被及时回收。李四用该账号登录 Harbor，在 23:17 删除了 `payment-platform` 项目——所有支付微服务的 120+ 个镜像瞬间消失，导致周一早高峰支付业务全部中断。运维和安全团队直到周一早上才发现问题。如果 Harbor 有实时告警机制——"有人在非工作时间删除了整个项目"——当晚就能触发电话告警、即时止损。事后复盘时，CTO 写了一行批注："运维监控不能只靠'人肉巡检'，必须建立自动化告警规则。"

**痛点五：日志量大但洞察力低。** Harbor 每天产生约 3.5GB 的各类日志（Nginx 1.2GB + Core 800MB + 审计日志 500MB + Registry 1GB），一个月就是 100GB。但这些日志的利用率极低——运维团队只在"出了问题"时才去看日志，平时这些数据躺在磁盘上默默消耗存储空间。安全团队多次提议"从日志中提取镜像拉取热度、识别异常访问模式、提前发现潜在攻击行为"，但一直没人知道怎么从海量非结构化日志中提取这些洞察。

本章将系统讲解 Harbor 的三层日志架构、审计日志的结构化查询与导出、基于 Webhook 和日志采集的实时告警方案，以及如何从日志数据中提取运维和安全的可操作洞察。

## 2 项目设计——剧本式交锋对话

**场景：审计师离开后，安全部门召集运维组开紧急复盘会议。会议室气氛凝重。**

---

**小胖**（把厚厚一叠审计需求摔在桌上，叹气）："又来了又来了！每次审计都让我们去翻日志——上次翻了一整天，这次还要翻六个不同的项目！我就想知道：Harbor 的日志到底有多少种？每种存在哪？能不能一次性全部导出来给审计师自己看去？"

**大师**（在白板上画出三层结构，平静地）："小胖你急躁了。你之所以觉得翻日志痛苦，是因为没搞清楚 Harbor 日志的三层模型——每一层的职责、格式、存储位置完全不同。搞清楚这个模型，你就会知道'查什么去第几层'。"

```
┌─────────────────────────────────────────────────────────────┐
│                Harbor 三层日志架构                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第 1 层：Nginx Access Log（反向代理层）                       │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ 记录：HTTP 请求的完整信息（来源 IP、URL、响应码、耗时）     ││
│  │ 格式：Apache Combined / JSON（可配置）                    ││
│  │ 存储：/var/log/nginx/access.log                         ││
│  │ 容器：harbor-nginx                                      ││
│  │ 粒度：每个 HTTP 请求一条（含 Pull Registry API 请求）      ││
│  │ 适用：流量分析、性能排查、IP 来源追溯                     ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
│  第 2 层：Core 应用日志（业务逻辑层）                           │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ 记录：认证/授权决策、API 调用详情、ERROR/WARNING/INFO     ││
│  │ 格式：文本格式（带时间戳和日志级别）                        ││
│  │ 存储：/var/log/harbor/core.log                          ││
│  │ 容器：harbor-core                                       ││
│  │ 粒度：业务操作级别（Token 签发、权限检查、DB 操作）        ││
│  │ 适用：故障排查、认证问题定位、性能瓶颈分析                ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
│  第 3 层：审计日志 / Audit Log（数据库持久化层）               │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ 记录：完整的操作轨迹（谁、什么操作、什么资源、什么时间）    ││
│  │ 格式：结构化数据库表（PostgreSQL audit_log 表）            ││
│  │ 存储：PostgreSQL 数据库                                  ││
│  │ 容器：harbor-db                                         ││
│  │ 粒度：业务操作级别（CREATE/DELETE/PUSH/PULL/UPDATE）       ││
│  │ 适用：合规审计、安全事件溯源、操作统计、用户行为分析        ││
│  └─────────────────────────────────────────────────────────┘│
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**技术映射**：第 1 层和第 2 层是容器级别的文件系统日志——由 Docker 的 logging driver（默认 `json-file`，也可配置为 `syslog`、`fluentd`、`journald`）管理。第 3 层是数据库级别的结构化记录——Harbor Core 在处理每个 API 请求时，在 handler 返回前向 `audit_log` 表插入一条记录。这意味着即使 harbor-core 容器被删除重建，审计日志也不会丢失（只要数据库存活）。但反过来也意味着——如果数据库损坏，审计日志就永久丢失了。

---

**小白**（笔记本上飞速记录，然后抬头）："那审计日志具体记录了哪些字段？是所有操作都记录，还是只记录'敏感操作'？Pull 操作每天几千次——如果全记录了，数据库表不会爆炸吗？"

**大师**（赞许地点头）："小白问到点子上了。审计日志的表结构是这样的——"

```sql
-- Harbor audit_log 表结构（简化）
CREATE TABLE audit_log (
    id          SERIAL PRIMARY KEY,
    project_id  INTEGER NOT NULL,
    operation   VARCHAR(32) NOT NULL,    -- CREATE/DELETE/PULL/PUSH/UPDATE/PURGE
    resource    VARCHAR(256),            -- 操作目标资源
    resource_type VARCHAR(64),          -- project/repository/artifact/tag/member
    username    VARCHAR(255) NOT NULL,   -- 操作者用户名
    op_time     TIMESTAMP DEFAULT NOW(), -- 操作时间（UTC）
    repo_name   VARCHAR(256),            -- 关联仓库全名（如 order-platform/order-service）
    repo_tag    VARCHAR(128),            -- 关联标签
    comment     TEXT                     -- 额外备注
);

-- Harbor 记录的 6 种操作类型
-- CREATE  : 创建项目、创建用户、创建复制规则、创建 Webhook ...
-- DELETE  : 删除镜像、删除仓库、删除项目、删除成员 ...
-- PULL    : docker pull / helm pull / 拉取制品（默认关闭！）
-- PUSH    : docker push / helm push / 推送制品
-- UPDATE  : 修改项目配置、修改 CVE 策略、修改成员角色 ...
-- PURGE   : 清理审计日志本身（管理员操作）
```

**小胖**（打断）："等等——Pull 审计日志默认关闭？那如果审计师要查'谁拉取了这个敏感镜像'，岂不是查不到？"

**大师**："这就涉及一个重要的权衡——**全量审计 vs 性能开销**。Pull 操作的特征是：频率极高（每个部署 Pod 启动、每个 CI 构建都会 Pull）、并且 99% 是合法的。一个 50 人的团队，每天 pull 次数在 5000-15000 次左右。如果全量记录 pull 审计日志——一个月就是 45 万条，一年超过 500 万条——`audit_log` 表会迅速膨胀，查询性能严重下降。Harbor 默认关闭 Pull 日志是基于这个现实考量。"

"但如果你确实需要——可以在 `harbor.yml` 中启用："

```yaml
# harbor.yml
audit_log:
  include_pull_operations: true   # 启用 Pull 审计日志
  retention_period: 365           # 保留天数（等保要求至少 180 天）
  rotate_period: daily            # 日志轮转周期
```

"启用之前，请务必评估：(1) 数据库存储是否足够——500 万条日志约占 2-3GB；(2) 查询性能是否可接受——必要时给 `audit_log` 表加索引。"

**技术映射**：Pull 审计日志的原理是：Docker 客户端在 `docker pull` 时，先通过 Registry API 向 Core 申请 Bearer Token。Core 的 Token Service 在签发 Token 时，如果 `include_pull_operations=true`，就会在这次 Token 签发过程中向 `audit_log` 表插入一条 PULL 记录。这意味着每次 Pull 都会增加一次数据库写入操作——在高并发环境下，这会给 PostgreSQL 带来显著的 I/O 压力。

**小白**（追问）："那如果我想实现实时告警呢？比如'有人在非工作时间删除了项目'，立刻给安全团队发通知。Harbor 有没有内置的告警机制？"

**大师**："Harbor 本身没有内置的告警引擎——但是提供三种方式可以实现实时告警，按需求复杂度从简到繁排列："

**方案一：Harbor Webhook（内置，最简单）**

"Harbor 支持在项目级别配置 Webhook。当特定事件发生时（如 DELETE_ARTIFACT、DELETE_REPOSITORY、SCANNING_COMPLETED），Harbor 会向指定的 URL 发送 HTTP POST 请求——请求体包含完整的事件信息。你只需要写一个接收端服务，解析事件并发送告警即可。"

```yaml
# Webhook 支持的事件类型
event_types:
  - PUSH_ARTIFACT          # 推送制品
  - PULL_ARTIFACT          # 拉取制品
  - DELETE_ARTIFACT        # 删除制品 ⚠️ 重点关注
  - DELETE_REPOSITORY      # 删除仓库 ⚠️ 重点关注
  - DELETE_TAG             # 删除标签
  - SCANNING_COMPLETED     # 扫描完成
  - SCANNING_FAILED        # 扫描失败
  - QUOTA_EXCEED           # 配额超限 ⚠️ 容量告警
  - REPLICATION_COMPLETED  # 复制完成
```

**方案二：日志采集 + Elasticsearch Alerting / Grafana Alerting（推荐，最灵活）**

"将 Nginx 日志、Core 日志和审计日志通过 Filebeat / Fluentd 采集到 Elasticsearch 或 Loki 中，然后配置告警规则。例如：Kibana Watcher 检测到过去 5 分钟内有 DELETE 操作 × 3 次 = 触发 PagerDuty 告警。"

**方案三：CronJob 轮询 API（自建，最可控但有时延）**

"写一个 CronJob（K8s CronJob 或 Linux crontab），每分钟调用一次 `GET /api/v2.0/audit-logs` API，查询最近 1 分钟内的 DELETE 操作。如果发现异常（如非工作时间删除项目），发送告警。缺点是：有 1 分钟的检测延迟。"

**大师**："三种方案不是互斥的——我建议：**Webhook 做近实时告警（几秒内响应）、日志采集做深度分析和长期留存、CronJob 轮询做兜底检查（防止 Webhook 遗漏）。**"

---

**小胖**（突然想起一个关键问题）："大师，我还有一个担心——审计日志会不会被管理员偷偷删除？比如某个管理员做了违规操作，然后登录数据库把 audit_log 里的相关记录删掉——这不就'毁尸灭迹'了？"

**大师**（神情严肃）："这是个深刻的问题——涉及到**审计日志的防篡改性**。Harbor 在设计上做了几层防护："

"第一层——**Harbor API 层面**：管理员通过 Harbor Portal 或 API **无法删除审计日志**。API 中删除审计日志的唯一方式是调用 `POST /system/purgeaudit`，但这个操作本身会被记录为一条 audit_log（operation=PURGE），并且只有系统管理员可以执行。"

"第二层——**数据库层面**：如果管理员直接登录到 PostgreSQL 执行 `DELETE FROM audit_log`——Harbor 确实无法阻止。这就需要配合更底层的安全措施：(a) 数据库操作日志（PostgreSQL `pg_stat_statements`）会记录 SQL 执行；(b) 操作系统 auditd 记录 `psql` 进程的启动；(c) 数据库权限最小化——不给管理员 `DELETE` 权限。"

"第三层——**合规级防护**：如果你们需要满足严格的合规要求（如 SOX、PCI DSS），建议将审计日志实时转发到外部 syslog 服务器或 ELK 集群。这样即使 Harbor 数据库被入侵，外部日志副本仍然完整。"

**技术映射**：Harbor Core 中的审计日志写入代码位于 `src/common/dao/audit.go`。每次 API handler 返回前，Core 调用 `dao.AddAuditLog()` 向数据库插入记录。这个插入操作是**同步阻塞**的——也就是说，如果数据库写入失败，API 请求也会失败。这保证了"操作成功 = 日志已记录"的语义一致性，但也意味着数据库故障会阻塞所有 API 调用。

### 3.2 步骤一：审计日志结构查询（API + SQL 双通道）

**场景**：安全部门要求导出过去 90 天内所有 DELETE 操作的审计日志。

#### 方式 A：通过 API 查询

```bash
#!/bin/bash
# audit-query-api.sh — 通过 API 查询审计日志

HARBOR="https://harbor.company.com"
AUTH="admin:Harbor12345"

# 时间范围：过去 90 天
START_TS=$(date -d '90 days ago' +%s)
END_TS=$(date +%s)

echo "=== Audit Log: DELETE operations (past 90 days) ==="

PAGE=1
TOTAL=0
while true; do
  RESPONSE=$(curl -sf -u "$AUTH" \
    "$HARBOR/api/v2.0/audit-logs?page=$PAGE&page_size=50&\
operation=DELETE&begin_timestamp=$START_TS&end_timestamp=$END_TS")
  
  COUNT=$(echo "$RESPONSE" | jq '. | length')
  if [ "$COUNT" -eq 0 ]; then
    break
  fi
  
  echo "$RESPONSE" | jq -r '.[] | 
    [.op_time, .username, .resource, .repo_name // "N/A", .resource_type] | @tsv'
  
  TOTAL=$((TOTAL + COUNT))
  PAGE=$((PAGE + 1))
  
  if [ "$COUNT" -lt 50 ]; then
    break
  fi
done

echo ""
echo "Total DELETE operations: $TOTAL"
```

**输出示例**：
```
=== Audit Log: DELETE operations (past 90 days) ===
2024-03-14T23:17:05Z  lisi  payment-platform  payment-platform  project
2024-03-10T09:30:12Z  dev01  order-service:v1.0.3  order-platform/order-service  artifact
2024-03-08T14:45:33Z  dev02  user-service:dev-abc123  user-platform/user-service  tag
2024-03-05T16:22:08Z  admin  ci-snapshots:test-xyz   ci-snapshots/test-app   tag

Total DELETE operations: 47
```

#### 方式 B：通过 PostgreSQL 直接查询

```sql
-- 登录 Harbor 数据库
docker exec -it harbor-db psql -U postgres -d registry

-- 查询 1：过去 90 天的 DELETE 操作 TOP 10 用户
SELECT 
    username,
    COUNT(*) as delete_count
FROM audit_log
WHERE 
    operation = 'DELETE'
    AND op_time > NOW() - INTERVAL '90 days'
GROUP BY username
ORDER BY delete_count DESC
LIMIT 10;

-- 输出示例：
--  username  | delete_count
-- -----------+-------------
--  lisi      |     23
--  dev01     |     15
--  admin     |      8

-- 查询 2：按小时统计操作量分布（识别异常时段）
SELECT 
    EXTRACT(HOUR FROM op_time) as hour,
    COUNT(*) as operation_count
FROM audit_log
WHERE op_time > NOW() - INTERVAL '30 days'
GROUP BY EXTRACT(HOUR FROM op_time)
ORDER BY hour;

-- 输出示例：
--  hour | operation_count
-- ------+----------------
--     0 |            156   ← 凌晨操作？需要关注
--     1 |             42
--     9 |           2341   ← 上班高峰
--    10 |           3120
--    14 |           2890   ← 下午高峰

-- 查询 3：按项目统计操作密度（识别热点项目）
SELECT 
    p.name as project_name,
    COUNT(al.id) as operation_count,
    COUNT(DISTINCT al.username) as unique_users
FROM audit_log al
JOIN project p ON al.project_id = p.project_id
WHERE al.op_time > NOW() - INTERVAL '90 days'
GROUP BY p.name
```

```python
def verify_signature(payload: bytes, signature: str) -> bool:
```

### 3.8 步骤七：构建日志分析仪表盘（SQL + Grafana）

```sql
-- 创建 Grafana 可用的 PostgreSQL 数据视图

-- 视图 1：每日操作统计（用于 Grafana 折线图）
CREATE OR REPLACE VIEW v_harbor_daily_operations AS
SELECT 
    DATE(op_time) as operation_date,
    operation,
    COUNT(*) as operation_count,
    COUNT(DISTINCT username) as unique_users,
    COUNT(DISTINCT project_id) as affected_projects
FROM audit_log
WHERE op_time > NOW() - INTERVAL '90 days'
GROUP BY DATE(op_time), operation
ORDER BY operation_date DESC;

-- 视图 2：异常用户行为（短时间内大量操作）
CREATE OR REPLACE VIEW v_harbor_unusual_activity AS
SELECT 
    username,
    operation,
    COUNT(*) as ops_in_5min,
    MIN(op_time) as first_op,
    MAX(op_time) as last_op
FROM audit_log
WHERE op_time > NOW() - INTERVAL '1 day'
GROUP BY username, operation, 
    DATE_TRUNC('minute', op_time) - 
    (EXTRACT(MINUTE FROM op_time)::int % 5) * INTERVAL '1 minute'
HAVING COUNT(*) > 10  -- 5 分钟内超过 10 次操作视为异常
ORDER BY ops_in_5min DESC;

-- 视图 3：项目风险评分（删除操作占比）
CREATE OR REPLACE VIEW v_harbor_project_risk AS
SELECT 
    p.name as project_name,
    COUNT(CASE WHEN al.operation = 'DELETE' THEN 1 END) as delete_count,
    COUNT(*) as total_operations,
    ROUND(
        100.0 * COUNT(CASE WHEN al.operation = 'DELETE' THEN 1 END) / 
        GREATEST(COUNT(*), 1), 1
    ) as delete_ratio_pct
FROM audit_log al
JOIN project p ON al.project_id = p.project_id
WHERE al.op_time > NOW() - INTERVAL '30 days'
GROUP BY p.name
HAVING COUNT(CASE WHEN al.operation = 'DELETE' THEN 1 END) > 0
ORDER BY delete_ratio_pct DESC;
```

### 3.9 可能遇到的坑

**坑1：审计日志表膨胀导致 API 查询超时 30 秒+**

现象：审计日志 API 查询耗时超过 30 秒，Portal 日志页面加载缓慢。

根因：`audit_log` 表累积了数千万条记录（如未开启自动清理），缺少合适的数据库索引。

解决：
```sql
-- 在 PostgreSQL 中创建关键索引
CREATE INDEX CONCURRENTLY idx_audit_log_op_time 
    ON audit_log (op_time DESC);
CREATE INDEX CONCURRENTLY idx_audit_log_username 
    ON audit_log (username);
CREATE INDEX CONCURRENTLY idx_audit_log_project_id 
    ON audit_log (project_id);
CREATE INDEX CONCURRENTLY idx_audit_log_operation 
    ON audit_log (operation);
CREATE INDEX CONCURRENTLY idx_audit_log_composite 
    ON audit_log (project_id, operation, op_time);

-- 查看索引使用情况
EXPLAIN ANALYZE SELECT * FROM audit_log 
WHERE project_id = 1 AND operation = 'DELETE' 
ORDER BY op_time DESC LIMIT 50;

-- 手动清理过期日志（如果自动清理未生效）
DELETE FROM audit_log 
WHERE op_time < NOW() - INTERVAL '90 days'
  AND id NOT IN (
    SELECT id FROM audit_log 
    WHERE op_time < NOW() - INTERVAL '90 days' 
    LIMIT 10000  -- 分批删除，每次 1 万条
  );
-- 重复执行直到删完
-- 然后执行 VACUUM ANALYZE audit_log; 回收空间
```

```bash
# 3. 如果执行历史显示 "status: error" 或 "status: 400/500"
# 检查 Core 容器是否能连接到 Webhook URL
docker exec harbor-core curl -v https://alert-webhook.company.com/harbor-events

# 4. 检查 Core 日志中是否有 Webhook 错误
docker exec harbor-core grep -i "webhook" /var/log/harbor/core.log | tail -20
```

**坑3：日志文件过大导致磁盘写满，Harbor 宕机**

现象：`/var/log/harbor/core.log` 涨到 20GB+，磁盘使用率 100%，所有容器不可用。

根因：日志级别设为 DEBUG（开发环境设置），且 `rotate_size` 设为 0（不轮转）。

解决：
```yaml
# harbor.yml 中设置合理的日志轮转
log:
  level: info          # 生产环境务必设置 info，不要 debug
  local:
    rotate_count: 30   # 最多保留 30 个日志文件
    rotate_size: 200M  # 单个日志文件最大 200MB
    location: /var/log/harbor

# 紧急清理（立即释放磁盘空间）
docker exec harbor-core bash -c "> /var/log/harbor/core.log"
docker exec nginx bash -c "> /var/log/nginx/access.log"
```

**坑4：审计日志时间戳时区混乱**

现象：API 返回的 `op_time` 是 UTC 时间，但 Portal 显示的是本地时间——同一个操作在 API 和 Portal 上显示的时间相差 8 小时。

解决：
```bash
# API 返回的时间戳全部是 UTC（RFC3339 格式）
# 查询时使用 Unix 时间戳可以避免时区歧义

# ✅ 推荐：使用 Unix 时间戳查询
START_TS=$(date -u -d '7 days ago' +%s)  # 使用 -u 确保是 UTC
END_TS=$(date -u +%s)

curl -u admin:pass \
  "https://harbor.company.com/api/v2.0/audit-logs?\
begin_timestamp=$START_TS&end_timestamp=$END_TS"

# 在 jq 中将 UTC 时间转为北京时间
jq '.[] | {time_utc: .op_time, 
           time_cn: (.op_time | strptime("%Y-%m-%dT%H:%M:%SZ") | 
                     mktime + 28800 | strftime("%Y-%m-%d %H:%M:%S CST"))}'
```

## 4 项目总结

### 4.1 Harbor 三层日志深度对比

| 维度 | Nginx Access Log | Core 应用日志 | 审计日志（PostgreSQL） | Registry 日志 |
|------|-----------------|-------------|----------------------|--------------|
| 格式 | Apache Combined / JSON | 文本行（`[时间][级别] 消息`） | 结构化数据库表（15 列） | Docker Registry 标准日志 |
| 查询方式 | `grep` / `awk` / ELK | `grep` / `awk` / ELK | SQL / Harbor API | `grep` / `journalctl` |
| 包含 Pull 操作 | ✅（所有 HTTP 请求） | ✅（Token 分配日志） | ❌（默认关闭，可选启用） | ✅（Registry API 请求） |
| 包含认证细节 | ❌（仅 HTTP 状态码） | ✅（Token 签发/拒绝原因） | ✅（用户名、操作类型） | ❌（仅操作类型） |
| 包含操作结果 | ✅（HTTP 状态码） | ✅（业务层成功/失败） | ✅（记录了即表示执行了） | ✅（HTTP 状态码） |
| 可查询时段 | 依赖日志 rotate | 依赖日志 rotate | 依赖 `retention_period` 配置 | 依赖日志 rotate |
| 保留建议 | 30 天 | 90 天 | 365 天（合规）| 30 天 |
| 防篡改性 | 低（root 可修改文件） | 低（root 可修改文件） | 中（API 不可删，DB 可删） | 低（root 可修改文件） |
| 最佳用途 | 流量分析、IP 追溯、性能排查 | 认证问题排查、错误定位、调试 | 合规审计、安全溯源、操作统计 | Registry 层故障排查 |
| 关联方式 | 时间戳 + HTTP 方法 + URL | 时间戳 + 请求 ID | 时间戳 + 用户名 + 操作类型 | 时间戳 + HTTP 方法 |

### 4.2 实时告警方案选型对比

| 方案 | 实时性 | 可靠性 | 复杂度 | 适用场景 | 局限性 |
|------|--------|--------|--------|---------|--------|
| Webhook（Harbor 内置） | ★★★★★（秒级） | ★★★☆（无重试机制） | ★☆☆☆（低） | 单项目实时告警 | 事件丢失不重试，需每个项目单独配置 |
| 日志采集 + ELK Alerting | ★★★★☆（分钟级） | ★★★★★（高） | ★★★★☆（高） | 集中式日志监控告警 | 需要 ES + Filebeat 基础设施 |
| CronJob 轮询 API | ★★☆☆☆（分钟级延迟） | ★★★☆☆（中） | ★★☆☆☆（中） | 简易兜底检查 | 有检测延迟，频繁轮询增加 API 负载 |
| syslog 转发 + SIEM | ★★★★☆ | ★★★★★ | ★★★★☆（高） | 企业级安全监控 | 需要 SIEM 系统（如 Splunk） |

### 4.3 日志管理最佳实践

| 实践 | 说明 | 反模式 |
|------|------|--------|
| 生产环境日志级别设为 `info` | DEBUG 级别会产生海量日志 | 部署后忘记改回 info |
| 配置日志轮转（rotate） | 防止单文件过大或磁盘写满 | `rotate_size: 0`（不轮转） |
| 审计日志保留期 ≥ 180 天 | 满足等保 2.0 最低要求 | 默认 90 天不改 |
| Pull 审计日志按需启用 | 权衡合规需求与数据库压力 | 盲目启用导致 DB 膨胀 |
| Webhook 配置幂等接收 | 同一事件可能重复发送 | 服务端不做去重导致重复告警 |
| 关键操作告警分等级 | Critical / Warning / Info 三级 | Push 和 Delete 同一告警级别 |
| 定期演练审计日志恢复 | 证明备份可恢复、日志可查询 | 备份了但从没验证过 |

### 4.4 常见踩坑经验（故障排查表）

| 故障现象 | 根因分析 | 排查步骤 | 解决方案 |
|---------|---------|---------|---------|
| 审计日志 API 返回 400 Bad Request | timestamp 格式错误——需 Unix 秒级时间戳，传入了毫秒或日期字符串 | 检查传入的 `begin_timestamp` 值 | 使用 `date +%s` 获取正确的秒级 Unix 时间戳 |
| Nginx 日志全是 200 但操作实际失败 | Nginx 只记录 HTTP 层状态码；Core 返回的业务错误被封装在 Response Body 中，HTTP 依然是 200 | 查看 Core 日志中对应时间段的 ERROR 记录 | 在 Core 日志中搜索 `"error"` 或 `"failed"` 关键字 |
| Webhook 发送了但服务端未处理 | Webhook 请求体结构变化（Harbor 版本升级后字段名可能不同）| 抓包对比 Webhook 请求体与预期结构 | 适配新版本字段名，或在服务端做多版本兼容 |
| 数据库 audit_log 表大小超过 50GB | 开启了 Pull 审计日志 + 保留期过长 + 未创建分区表 | `SELECT pg_size_pretty(pg_total_relation_size('audit_log'));` | 清理过期日志 + 创建按月分区表 + 定时归档 |
| `pg_dump` 失败因 audit_log 表过大 | 备份超时或磁盘空间不足 | 检查 `/tmp` 空间（pg_dump 可能用到临时文件） | 将 audit_log 排除出日常备份，单独使用 `pg_dump --exclude-table=audit_log` |
