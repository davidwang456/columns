# 第10章：用户管理与认证体系

## 1 项目背景

2024年4月的一个周一早上，启明星辰（虚构）的平台运维总监张磊被一封邮件惊出一身冷汗。邮件来自公司安全审计委员会，标题是《关于Harbor仓库异常访问事件的通报》。通报显示：上周六凌晨3:17，有人使用 `admin` 账号从境外IP（`185.220.x.x`）成功拉取了 `payment-platform` 项目的所有镜像——总计187个标签，包含完整的数据库连接配置、内部API密钥、甚至第三方支付渠道的签名证书私钥。而这些敏感信息，被打包进了早期为了方便调试而直接 `COPY` 进镜像的 `.env` 文件中。

安全部门追查了整整三天，得出一个令人沮丧的结论：**无法定位这次泄露是由谁造成的**。原因是——从2022年Harbor上线至今，全公司500+名开发者和运维人员一直共用一个 `admin` 账号，密码就是 `Harbor12345`。审计日志显示近两个月内，`admin` 账号从217个不同的IP登录过，分布在北京、上海、深圳、成都四个研发中心——任何一个离职员工、任何一个被植入木马的开发机，都可以在任意时间拉走任意项目的镜像。

**痛点一：共享账号导致零审计追溯能力。** 启明星辰的Harbor中共有38个项目、2300+个镜像，所有操作都由 `admin` 完成。去年Q4发生了一次严重事故——`payment-gateway:release-4.2` 生产标签被意外覆盖为一个测试版本，导致支付网关全部宕机37分钟，直接损失交易额约260万元。安全部门翻遍了Harbor的审计日志，只能看到 "操作者: admin, 操作: 删除Artifact, 时间: 2023-11-15 14:23:07"。而当天下午用 `admin` 账号登录过的运维和开发人员多达23人——其中有人在清理测试镜像，有人在做正式发布。最终花了整整两周排查，始终无法确定是谁做了误操作，更无法防止类似事件再次发生。

**痛点二：多套账号体系导致运维成本倍增。** 启明星辰雇佣了3200名员工，组织架构由Windows AD统一管理。但Harbor有一套独立的本地用户数据库——每来一个新员工，IT运维需要：（1）在AD中创建账号；（2）在Harbor中创建同名本地账号；（3）设置初始密码；（4）将账号加入相应的项目并分配角色。这四个步骤平均耗时15分钟/人，按每月入职约25人计算，仅Harbor账号创建一项就消耗掉运维团队6.25小时/月。更糟糕的是离职——IT在AD中禁用账号后，经常忘记同步禁用Harbor中的本地账号。去年安全审计发现：17个已离职员工的Harbor本地账号仍然处于活跃状态，其中3个账号在离职后的30天内仍有登录记录。

**痛点三：SSO缺失导致的密码管理混乱。** 公司早在2021年就部署了基于Keycloak的统一SSO平台，员工只需一次登录即可访问Confluence、Jira、GitLab、Jenkins等12个内部系统。唯独Harbor不在这个SSO体系中——员工需要通过Portal使用另一套独立的用户名和密码登录。开发者们普遍使用两种"捷径"：（1）在 `~/.docker/config.json` 中直接写入Base64编码的密码（极易在代码仓库中意外提交）；（2）使用极其简单的密码（如 `Abc123456`），因为没人能记住第13套系统密码。安全部门在2023年12月的一次巡检中，从GitLab仓库中搜索到了44个包含 `harbor.*password` 的提交记录。

**痛点四：CI/CD流水线的权限失控。** 由于为每个项目创建专用凭证太"麻烦"，DevOps团队直接在Jenkins和GitLab CI的全局环境变量中配置了 `admin:Harbor12345`。这意味着：任何一个开发者只要在他的 `Jenkinsfile` 中添加一行 `docker push`，就能以 `admin` 身份向任意项目推送任意镜像——包括覆盖生产标签。更可怕的是，这个 `admin` Token也被硬编码在了3个K8s集群的 `imagePullSecret` 中——攻击者只要攻破集群中任意一个Pod（哪怕是测试环境的），就能获取 `admin` 凭证，然后拉取所有私有镜像并反编译其中的业务逻辑。

**痛点五：外包与合作伙伴权限隔离有漏洞。** 公司引入了两个外包团队（合计约40人），分别负责前端UI开发和测试自动化。按合同规定，外包人员只能访问 `frontend-services` 和 `testing-tools` 两个项目。但由于Harbor没有细粒度的"仓库级"访问控制，运维给他们开了这两个项目的"访客"角色——隐患在于：`frontend-services` 项目中除了前端镜像，还存放着3个包含后端接口mock的服务镜像（早期为了方便前后端联调），里面嵌入了内部API的请求逻辑和数据格式。外包人员理论上可以拉取这些镜像并反推业务架构。

本章将系统性地解决上述五层问题——从Harbor的三级用户体系（本地用户、LDAP/AD、OIDC SSO）到机器人账户的精细权限控制，构建一个可审计、可追溯、可自动化的企业级认证方案。

## 2 项目设计——剧本式交锋对话

**场景：启明星辰技术管理委员会月度例会——CTO赵岩、平台架构师大师、安全工程师小白、开发代表小胖、运维老李围坐在长条形会议桌旁。投屏显示着安全审计委员会那封"通报"邮件的全文。**

**老李**（拍桌子）："受不了了！又有人用 `admin` 删了生产镜像！这次是 `risk-engine:release-5.1`——风控系统挂了整整22分钟，风控未命中导致两笔欺诈交易通过了！赵总，今天我必须在会上把话说清楚——要么给我配2个专职做Harbor权限管理的人，要么这个黑锅我不背了！"

**小胖**（低头嘀咕）："我密码不是早就跟老李说过了吗——Harbor12345——大家都知道的……为啥不直接统一用一个密码呢，简单方便……"

**大师**（站起身，在白板上写下一行字）："小胖，你这句话说出了问题的核心——'简单方便'——但方便了谁？方便了使用者，也方便了攻击者。"

他在白板上画了一个大大的门，门上贴满了钥匙：

"目前我们的Harbor就像这扇门——门上的锁只有一把，钥匙却复制了500把。每个人进出都方便。但问题是：哪天钥匙丢了一把，你根本不知道丢的是谁的——你也换不了锁，因为其他499把也作废了。这就是共享 `admin` 的致命缺陷——**无法隔离风险**。"

**技术映射**：Harbor的RBAC（基于角色的访问控制）模型是解决这个问题的基石。RBAC不是"给每个人配一把不同的钥匙"那么简单，而是把整个系统分成了两个权限层：**系统级**（决定你能不能开Harbor这个"大楼"的门）和**项目级**（决定你进了大楼后能进哪些"房间"、能在房间里做什么）。一个人在不同项目里可以有不同角色——就像你在家是主人（管理员），去朋友家是客人（访客）。

**小白**（从笔记本电脑旁抬起头，推了推眼镜）："我把Harbor的认证架构梳理了一下，发现有四个认证入口：本地数据库、LDAP/AD、OIDC、和机器人账户Token。这四者之间是什么关系？能同时启用吗？还是只能四选一？"

**大师**："好问题。答案是——**可以同时启用，但优先级有明确的顺序**。想象一下机场的安检流程——你先在柜台人工值机（OIDC），如果排队太长可以走自助机（本地登录），机组人员有专门的快速通道（LDAP），机器人行李车有自己的通道（Robot Token）。四条通道互不干扰，但每条通道的乘客都要被验证身份。"

他在白板上画出了完整的认证流程：

```
用户/系统请求访问Harbor
          │
          ▼
    ┌─────────────┐
    │ 1. OIDC     │ ◄── 人类用户通过Web Portal：检测OIDC Cookie
    │   单点登录    │     如果存在且有效 → 自动登录，首次自动创建Harbor账号
    └──────┬──────┘
           │ 未命中
           ▼
    ┌─────────────┐
    │ 2. LDAP/AD  │ ◄── 用户输入AD用户名+密码 → Harbor验证LDAP Bind
    │   目录认证    │     LDAP用户需手动导入或按需验证
    └──────┬──────┘
           │ 未命中
           ▼
    ┌─────────────┐
    │ 3. 本地用户   │ ◄── Harbor内置数据库中的用户（包括admin）
    │   数据库认证   │     适用于小团队、临时账号、管理员应急后门
    └──────┬──────┘
           │ 未命中
           ▼
    ┌─────────────┐
    │ 4. 机器人账户  │ ◄── CI/CD系统使用Token认证（不能登录Portal）
    │   Token认证   │     典型的非交互认证（API、Docker CLI、K8s ImagePullSecret）
    └─────────────┘
```

**技术映射**：Harbor的认证架构遵循"认证链"模式——每种认证源都是一个独立的Authenticator，依次尝试。但实际生产环境中通常只启用一种主认证模式（`auth_mode` 字段）：`db_auth`（纯本地）、`ldap_auth`（LDAP/AD）、`oidc_auth`（OIDC SSO）。`auth_mode` 的值决定了Portal的登录页面显示哪种登录方式。但无论选择哪种 `auth_mode`，Robot Account的Token认证始终可用（因为它是API/CLI层面的认证，不走Portal登录页）。

**小胖**（挠头）："等等，我有点乱。LDAP和OIDC到底有什么区别？听起来不都是用同一个公司账号登录吗？为什么需要两个不同的协议？"

**大师**："这个区分确实容易混淆。一句话说清楚——"

他从白板上取下两张图：

| 维度 | LDAP/AD 认证 | OIDC 认证 |
|------|------------|----------|
| **底层协议** | LDAP(S) — 轻量目录访问协议，第一版发布于1993年 | OAuth 2.0 + OpenID Connect — 现代身份认证协议，2014年发布 |
| **认证交互方式** | 用户在Harbor Portal输入AD用户名和密码 → Harbor作为LDAP客户端，发送Bind请求到AD服务器验证 | 用户点击"OIDC登录" → 浏览器重定向到公司SSO页面（Keycloak/Okta/Azure AD）→ 用户登录 → SSO页面回调Harbor，携带授权码 |
| **用户体验** | 和本地登录一样的体验（Harbor Portal内输入密码） | 跳转到公司统一的SSO登录页——如果浏览器已有SSO的Cookie，**自动登录无感** |
| **账号创建方式** | 需管理员手动在Portal中导入LDAP用户（或使用LDAP组同步） | 首次通过OIDC登录时，Harbor**自动创建**该用户的账号 |
| **密码管理位置** | 密码存储在AD/LDAP服务器，Harbor不存储LDAP用户的密码 | 密码存储在IDP（Identity Provider），Harbor完全不接触密码 |
| **Token过期与刷新** | 无Token概念——每次API调用都需要重新验证LDAP凭证 | OIDC使用短期Access Token（默认1小时）+ Refresh Token（可配置） |
| **适用组织类型** | 使用传统AD域（Windows Active Directory）的企业 | 已建立现代SSO体系（Keycloak、Okta、Azure AD等）的企业 |
| **管理员应急后门** | 本地 `admin` 账号不走LDAP——即使在LDAP模式下，`admin` 仍可用本地密码登录 | 同左——本地 `admin` 始终可用 |

**小胖**（似懂非懂）："所以你意思是——如果公司已经有Keycloak SSO了，就用OIDC模式？那LDAP就完全没用了吗？"

**大师**："不尽然。有一个非常常见的混合场景：Keycloak背后对接的**就是**公司的AD。术语叫'LDAP Federation'——Keycloak作为IDP，后端连接到AD来验证用户名和密码。这样就实现了：用户看到的SSO页面（OIDC），但底层验证逻辑仍然是AD（LDAP）。这就像是——"

他拿起桌上的咖啡杯："你喝到的咖啡（SSO体验）是现磨的，但咖啡豆（认证数据）来自同一个仓库（AD）。这就是现代企业的标准配置：AD ∩ Keycloak ∩ OIDC → Harbor。"

**技术映射**：Harbor的 `oidc.group_claim` 配置是实现"通过OIDC同步AD群组"的关键。Keycloak可以将用户的AD群组成员信息映射到OIDC Token的 `groups` Claim中。Harbor读取这个Claim后，可以自动将用户分配到对应的项目中。`admin_group` 配置项允许将指定AD群组的成员自动提升为Harbor系统管理员。

**小白**（若有所思）："我想深挖一下机器人账户。从安全角度看，它和普通用户的核心区别是什么？为什么它会更安全？"

**大师**（拿起白板笔，画了一张对比表）：

"机器人账户和普通用户，从Harbor的角度看是两种完全不同的安全模型："

```
              普通用户                    机器人账户
           ┌──────────┐              ┌──────────────┐
登录Portal  │    ✅    │              │     ❌       │ ← 最大差异！
           ├──────────┤              ├──────────────┤
密码        │  用户设置  │              │ 系统自动生成  │
           │  可复用    │              │ Token格式     │
           ├──────────┤              ├──────────────┤
密码过期    │  30天强制  │              │  永不过期     │ ← CI不需要半夜登录失败
           │  修改     │              │  (可选到期时间) │
           ├──────────┤              ├──────────────┤
命名格式    │  alice   │              │ robot$project│ ← 审计日志一目了然
           │          │              │ +ci-builder  │
           ├──────────┤              ├──────────────┤
权限范围    │  全局     │              │  项目级      │ ← 最小权限原则
           ├──────────┤              ├──────────────┤
典型用例    │  开发者   │              │  Jenkins CI  │
           │  日常登录  │              │  K8s ImagePull│
           │          │              │  Ansible部署  │
           └──────────┘              └──────────────┘
```

"机器人账户的'无Portal登录'特性是它的最大安全优势——而且这是'不可逆'的。即使有人拿到了Robot Token，他也只能通过 `docker login` 或API调用使用它——他无法登录Portal查看项目成员列表、项目配置、或审计日志。这极大地限制了Token泄露后的影响范围。"

```bash
# 机器人账户创建示例
curl -X POST -u admin:Harbor12345 \
  -H "Content-Type: application/json" \
  -d '{
    "name": "ci-builder",
    "description": "Jenkins CI pipeline for order-platform",
    "expires_at": -1,
    "access": [
      {
        "resource": "repository",
        "action": "pull",
        "effect": "allow"
      },
      {
        "resource": "repository",
        "action": "push",
        "effect": "allow"
      }
    ]
  }' \
  https://harbor.qimingxingchen.com/api/v2.0/projects/order-platform/robots

# 返回的Token仅显示一次，务必立即保存！
# {"name":"robot$order-platform+ci-builder","token":"eyJhbGciOiJSUzI1NiIs..."}
```

**技术映射**：机器人账户的底层实现利用了Harbor的Secret管理机制。Token以JWT（JSON Web Token）格式签发，Payload中包含 `project_id`、`access` 权限列表和 `exp` 过期时间。Harbor Core在每个API请求中验证JWT的签名和Payload中的权限声明——这与普通用户的Session认证是两套不同的中间件。

**小胖**（突然兴奋）："那我有个大胆的想法——我们能不能完全废除普通用户？每个人一个机器人账户？这样就没人能登录Portal乱改配置了！"

**大师**（笑着摇头）："想法很好，但过于极端。首先，机器人账户不能登录Portal——这就意味着所有操作都要通过API/CLI完成，包括查看扫描报告、管理CVE白名单、查看镜像列表——对普通开发者来说操作成本太高。其次，机器人账户不能成为项目管理员（没有 `manage_member` 权限）——你还是需要有人来管理项目成员。最后，审计日志中需要区分'人类操作'和'机器操作'——机器人账户天然标明这是自动化操作，如果用机器人账户替代人类，审计反而更困难。"

**小白**："说到审计，我们总要有个兜底方案吧？万一主认证模式故障了——比如Keycloak服务器或AD域控挂了——我们怎么办？"

**大师**："这是每个生产运维人员必须回答的问题——**认证系统的容灾设计**。Harbor提供了一个至关重要的后门——**本地 `admin` 账号永远不会被禁用**。"

他在白板上写下：

```
认证容灾的三层防线：

第一层：IDP高可用（Keycloak集群 / AD多域控）
         ↓ 故障
第二层：本地admin账号（手动切换，紧急操作入口）
         ↓ 故障
第三层：Harbor数据库直接修改（sql修改 auth_mode=db_auth）
         终极恢复手段——需要直接操作PostgreSQL
```

"核心规则：**永远不要删除admin账号，永远不要忘记admin密码，永远把admin密码存储在只有2-3个高级运维知道的保险柜中**。admin就是你最后的'破门锤'——平时不用，但必须随时可用。"

## 3 项目实战

### 3.1 环境要求

| 组件 | 版本/配置要求 | 说明 |
|------|-------------|------|
| Harbor | v2.12 | 完整安装（非离线版），需JobService和Core正常运行 |
| AD/LDAP服务器 | Windows Server 2019+ 或 OpenLDAP 2.6+ | 可选——如果只测试本地用户则不需要 |
| Keycloak | 22.0+ | 可选——测试OIDC模式需要 |
| Docker客户端 | 20.10+ | 用于测试 `docker login` |
| curl + jq | 任意版本 | API测试 |
| admin密码 | 已知并可登录 | 所有操作起点 |

```bash
# 确认Harbor各组件的运行状态
docker compose -f /opt/harbor/docker-compose.yml ps
# 预期：所有服务状态为 Up
```

### 3.2 步骤一：创建本地用户并分配系统角色

**目标：** 创建一批测试用户，掌握通过API批量管理用户的方法。

```bash
# 1. 创建四个不同角色的本地用户
curl -X POST -u admin:Harbor12345 \
  -H "Content-Type: application/json" \
  -d '{"username":"zhangsan","email":"zhangsan@qimingxingchen.com","password":"ZhangSan@2024!","realname":"张三 (运维负责人)"}' \
  https://harbor.qimingxingchen.com/api/v2.0/users

curl -X POST -u admin:Harbor12345 \
  -H "Content-Type: application/json" \
  -d '{"username":"lisi","email":"lisi@qimingxingchen.com","password":"LiSi@2024!","realname":"李四 (后端开发)"}' \
  https://harbor.qimingxingchen.com/api/v2.0/users

curl -X POST -u admin:Harbor12345 \
  -H "Content-Type: application/json" \
```

### 3.9 排坑指南

#### 坑1：LDAP配置后admin无法登录

**现象：** 修改 `harbor.yml` 中的 `auth_mode: ldap_auth` 并执行 `./prepare && docker compose up -d` 后，Portal登录页面变为LDAP登录模式，但用 `admin` + 原密码登录失败。

**根因分析：** 切换到LDAP认证模式后，Portal默认只显示LDAP登录框。但Harbor设计了一个安全后门：即使 `auth_mode` 为 `ldap_auth`，本地 `admin` 账号仍然可以通过特定的后端路由登录。问题在于：用户需要知道如何访问本地登录入口。

**解决方法：**
```bash
# 方案A：直接访问本地登录的API端点
# 本地admin永远可以通过 /c/login 的后端路由登录（绕过LDAP）
# 在浏览器中访问：
# https://harbor.qimingxingchen.com/c/login

# 方案B：如果方案A不可用，通过API重置admin密码并验证
curl -X PUT -u admin:Harbor12345 \
  -H "Content-Type: application/json" \
  -d '{"old_password":"Harbor12345","new_password":"NewAdmin@2024!"}' \
  https://harbor.qimingxingchen.com/api/v2.0/users/1/password
```

#### 坑2：OIDC登录后重定向到空白页面（500 Internal Server Error）

**现象：** 用户在Keycloak页面登录成功后，浏览器重定向回Harbor，但显示"500 Internal Server Error"或完全空白页面。

**根因分析：** 这是Harbor OIDC集成中最常见的错误，通常由三个原因之一引起：

1. **Redirect URI 不匹配。** Keycloak客户端配置中的 `Valid Redirect URIs` 必须与Harbor的 `external_url` + `/c/oidc/callback` 完全一致（含协议和端口）。例如Harbor的 `external_url` 是 `https://harbor.qimingxingchen.com`，则Keycloak中应填写 `https://harbor.qimingxingchen.com/*`。

2. **OIDC endpoint与Keycloak的issuer不一致。** `harbor.yml` 中的 `oidc.endpoint` 必须精确等于Keycloak的 `issuer` 值（可通过 `/.well-known/openid-configuration` 端点验证）。

3. **Client Secret 错误。** 在Keycloak中重新生成了Client Secret但未更新 `harbor.yml`。

**排查步骤：**
```bash
# 1. 验证OIDC endpoint配置
ENDPOINT="https://sso.qimingxingchen.com/realms/qimingxingchen"
curl -s "$ENDPOINT/.well-known/openid-configuration" | jq '.issuer'
# 确保返回的issuer值与 harbor.yml 中的 oidc.endpoint 完全一致

# 2. 验证Client Secret
curl -s -u "harbor:<client_secret>" \
  "$ENDPOINT/protocol/openid-connect/token" || echo "Credentials invalid"

# 3. 检查Harbor Core日志
docker logs harbor-core 2>&1 | grep -i "oidc\|callback\|500\|error" | tail -30
```

#### 坑3：LDAP搜索不到任何用户

**现象：** 在Portal中点击"导入LDAP用户"后搜索，无论输入什么关键字都返回"没有找到匹配的用户"。

**根因分析：** 有四个常见原因：

1. **`ldap.uid` 字段名写错。** Windows AD中用户登录名字段是 `sAMAccountName`（注意大小写），而OpenLDAP一般用 `uid`。如果是在AD环境中写成了 `uid`，搜索将永远无结果。

2. **`ldap.filter` 过滤条件过窄。** 例如 `"(objectClass=person)"` 在某些AD环境中不匹配——AD用户对象可能使用的是 `"(objectClass=user)"`。

3. **`ldap.base_dn` 不包含目标用户的OU。** 如果用户都在 `OU=Engineering,OU=Shanghai,DC=...` 下面，但 `base_dn` 只写到 `OU=Shanghai,DC=...`，确认搜索范围 `scope` 是否为 `2`（SubTree）。

4. **LDAP搜索账号（`search_dn`）没有读取权限。**

**排查方法：**
```bash
# 使用ldapsearch手动测试
ldapsearch -H ldaps://ad.qimingxingchen.com:636 \
  -D "CN=harbor-svc,OU=Service Accounts,DC=qimingxingchen,DC=com" \
  -w "SvcAccount@2024!" \
  -b "DC=qimingxingchen,DC=com" \
  "(sAMAccountName=zhangsan)" \
  sAMAccountName mail displayName memberOf

# 如果返回结果为空，尝试用更宽松的过滤条件：
ldapsearch ... "(objectClass=user)" sAMAccountName  | head -20
```

#### 坑4：机器人账户在K8s中pull镜像失败："pull access denied"

**现象：** K8s Pod创建时 `ImagePullBackOff`，`kubectl describe pod` 显示：
```
Failed to pull image "harbor.qimingxingchen.com/order-platform/my-service:v1.0": 
rpc error: code = Unknown desc = Error response from daemon: 
pull access denied for harbor.qimingxingchen.com/order-platform/my-service, 
repository does not exist or may require 'docker login': denied: 
requested access to the resource is denied
```

**根因分析：** 三个典型原因：

1. **机器人账户Token中包含特殊字符没有正确转义。** `robot$project+name` 中的 `$` 和 `+` 在Shell和YAML中需要特殊处理。

2. **机器人账户只有Push权限，没有Pull权限。** 创建时选择了错误的 `access` 配置。

3. **ImagePullSecret的namespace与Pod不一致。** Secret必须在Pod所在的namespace中创建。

**解决方法：**
```bash
# 1. 确认机器人账户名称的正确格式（Shell转义）
# 在bash中使用单引号防止$展开：
docker login harbor.qimingxingchen.com \
  -u 'robot$order-platform+k8s-puller' \
  -p "eyJhbGciOiJSUzI1NiIs..."

# 2. 在K8s YAML中正确的Secret创建方式
kubectl create secret docker-registry harbor-secret \
  --docker-server=harbor.qimingxingchen.com \
  --docker-username='robot$order-platform+k8s-puller' \
  --docker-password='eyJhbGciOiJSUzI1NiIs...' \
  --namespace=order-platform

# 3. 验证Secret中的凭证可用
kubectl get secret harbor-secret -n order-platform -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d
```

### 4.1 Harbor三种认证模式全景对比

| 维度 | 本地用户 (db_auth) | LDAP/AD (ldap_auth) | OIDC SSO (oidc_auth) | 机器人账户 (Token) |
|------|-------------------|---------------------|----------------------|-------------------|
| **账号存储位置** | Harbor PostgreSQL数据库 | 企业AD/LDAP服务器 | Identity Provider (IDP) | Harbor PostgreSQL（Token） |
| **密码管理方** | Harbor独立管理 | AD域控制器 | IDP统一管理 | Harbor自动生成Token |
| **密码修改后的影响** | 仅Harbor内生效 | AD修改后Harbor同步生效 | IDP修改后Harbor同步生效 | Token不关联用户密码 |
| **登录Portal** | ✅ | ✅ | ✅（通过SSO重定向） | ❌（仅API/CLI） |
| **登录Docker CLI** | ✅ | ✅ | ✅ | ✅ |
| **首次登录时自动创建账号** | ❌（需管理员手动创建） | ❌（需管理员手动导入） | ✅（oidc_auto_onboard） | ❌（需管理员手动创建） |
| **离职后自动失效** | ❌（需管理员手动禁用） | ✅（AD中禁用/删除即无法登录） | ✅（IDP中禁用/删除即无法登录） | —（Token是独立的） |
| **Session/Token过期时间** | 默认30分钟 | 默认30分钟 | Access Token默认1小时 | 永久有效（或自定义到期） |
| **推荐团队规模** | < 30人小团队 | 中型企业（已有AD） | 现代企业（已有SSO） | 任意规模（CI/CD） |

### 4.2 机器人账户 vs 普通用户详细对比

| 维度 | 普通用户 | 机器人账户 | 安全影响分析 |
|------|---------|-----------|------------|
| **Web Portal访问** | ✅ 可以登录、浏览、操作 | ❌ 完全不能登录Portal | 攻击者即使获取Token也无法查看项目结构/成员 |
| **密码/Token遗忘** | 可自助重置（通过邮箱） | Token丢失只能重新生成（旧Token立即失效） | Token泄露的响应速度更快（重新生成=即刻废止） |
| **密码强度要求** | 强制复杂度+定期更换 | 自动生成的JWT Token（256位） | Token比人工密码更难暴力破解 |
| **在多项目间复用** | ✅ 一个用户可加入多个项目 | ❌ 一个机器人账户绑定一个项目 | 最小化权限爆炸半径——一个Token泄露只影响一个项目 |
| **审计日志中的身份** | `username` 例如 "zhangsan" | `robot$project+name` 例如 "robot$order-platform+jenkins-ci" | 机器人操作在审计日志中一目了然 |
| **权限范围** | 系统级权限 + 多个项目角色 | 仅项目级，仅创建时指定的 `access` 权限列表 | 无法通过机器人账户进行系统级操作 |

### 4.3 适用场景

- **企业LDAP/AD统一认证：** 已有Windows AD域环境的公司，将Harbor接入AD后——员工的入职/离职/转岗通过AD统一管理，Harbor权限随之自动调整。离职时AD账号一禁用，该员工立即无法登录Harbor。
- **现代SSO单点登录体系：** 已部署Keycloak/Okta/Azure AD/OneLogin等IDP的企业，配置OIDC集成后——员工访问Harbor就像访问GitLab和Jira一样，浏览器自动携带SSO Cookie，无需额外输入密码。`admin_group` 可实现：公司中"平台架构组"的成员自动获得Harbor系统管理员权限。
- **多团队权限精细隔离：** 在大型组织中，每个业务团队拥有1-2个Harbor项目。通过项目角色（项目管理员/开发者/访客/受限访客），精确控制每个成员能看什么、能做什么——外包团队只能pull指定仓库，无法发现其他敏感项目的存在。
- **CI/CD流水线安全凭证管理：** Jenkins、GitLab CI、GitHub Actions等工具通过机器人账户访问Harbor——每个流水线使用单独的、仅有必要权限的Token。即使某个CI系统的配置仓库被泄露，攻击者拿到的Token也只能操作那一个项目。
- **安全审计与合规追溯：** 每个用户（含机器人账户）的每次操作都被记录在审计日志中，关联到唯一的用户ID。当发生安全事故时，可以在5分钟内定位到"谁、什么时候、从哪个IP、做了什么操作"。

### 4.4 不适用场景

- **极小型团队（< 5人）：** 部署LDAP/OIDC的配置和维护成本（Keycloak部署、证书管理、定期运维）高于直接用Harbor本地用户。5人以下团队建议直接使用本地用户模式，但务必做到一人一账号。
- **完全公开的匿名镜像分发：** Harbor的所有操作（包括pull公开项目的镜像）都需要客户端认证——`docker login` 是必需的。如果需要完全匿名、无需登录的镜像分发，应使用Docker Hub或其他公共Registry。

### 4.5 五项关键注意事项

1. **`admin` 账号是最终的救命稻草——永远不要删除或遗忘。** 无论选择哪种 `auth_mode`，本地 `admin` 账号都不会被禁用。当LDAP服务器或OIDC IDP全部故障时，`admin` 是唯一的登录入口。建议将 `admin` 密码加密后存储在公司的密码保险柜（如HashiCorp Vault或1Password）中，授权不超过3人查看。
2. **切换 `auth_mode` 前必须备份数据库。** 从 `db_auth` 切换到 `ldap_auth` 或 `oidc_auth` 是一个不可逆的操作——如果配置有误（如LDAP Base DN写错、OIDC Client Secret不匹配），可能导致所有用户被锁定。备份命令：`docker exec harbor-db pg_dump -U postgres registry > /backup/harbor-backup-$(date +%Y%m%d).sql`。
3. **OIDC `group_claim` 字段大小写敏感且因IDP而异。** Keycloak默认返回 `groups`（全小写），Azure AD返回 `groups`（全小写），Okta返回 `groups`（全小写），但Ping Identity可能返回 `Groups`（首字母大写）。务必在IDP的Token Response中确认实际的Claim名称。
4. **机器人账户Token仅显示一次——丢失后只能重新生成。** 创建机器人账户时返回的Token是完整的JWT，之后无法再次查看明文。如果Token丢失，只能通过 `PATCH /robots/{id}` 重新生成（旧Token立即失效）。建议在安全的位置（如HashiCorp Vault或K8s Secret）存储Token，不要仅放在Shell的Command History中。
5. **OIDC自动创建账号（`oidc_auto_onboard`）是一把双刃剑。** 启用后，任何在IDP中存在且能通过SSO登录的人都会自动在Harbor中创建账号——这包括外包人员、合作伙伴、甚至临时访问的实习生。建议同时配置 `oidc.admin_group` 来限制谁能成为管理员，避免自动创建账号获得不期望的权限。

### 4.6 生产环境典型故障案例

| 故障案例 | 现象描述 | 根因分析 | 解决方法 |
|---------|---------|---------|---------|
| **"认证地震"——Keycloak集群宕机导致全员无法登录Harbor** | 周一上午Keycloak的三节点集群因内存溢出同时重启。在此期间，Harbor Portal的OIDC登录页面报错"Unable to connect to OIDC Provider"，所有依赖OIDC的开发者和运维无法登录 | `auth_mode: oidc_auth` 模式下，如果IDP不可达，OIDC认证会失败。Portal上没有"本地用户登录"的降级选项 | 1. 确保运维团队知晓本地admin登录的URL（`/c/login`）；2. Keycloak部署确保3节点以上且分散在不同AZ；3. 考虑在Harbor前面加反向代理（Nginx），配置OIDC认证的fallback路由 |
| **"幽灵用户"——离职员工账号未被禁用** | 季度安全审计发现17个已离职员工的Harbor账号仍处于活跃状态，其中3个在离职后30天内仍有Docker CLI登录记录 | LDAP/OIDC集成只解决了"认证"问题（离职=无法登录），但没有解决"授权"问题（已分配的项目角色仍在数据库中）。管理员没有同步清理项目成员列表 | 1. 定期执行本课程"步骤六"中的用户活跃度审计脚本；2. 在HR系统中集成Webhook——员工离职时自动触发Harbor API禁用账号；3. 将LDAP用户导入与HR系统的入离职流程集成 |
| **"Token泄露链"——Jenkins全局凭证导致Harbor仓库被恶意篡改** | 安全部门接到匿名报告：一个名为 `backdoor:latest` 的标签出现在 `payment-core` 项目中，内含挖矿程序。经查，Jenkins的全局凭证中保存的机器人Token在3周前的一次Git仓库公开泄露事件中被获取 | 机器人Token被配置为Jenkins的全局凭证（对所有Pipeline可见），某个开发者在公开GitHub仓库的README中无意粘贴了Jenkins的构建脚本（含Token）。CI平台权限过大 | 1. 机器人Token不使用Jenkins全局凭证，改为使用Pipeline级别的绑定凭证；2. Token 30天轮换一次（通过API自动更新）；3. 在代码仓库中配置Secret Scanning（GitHub Secret Scanning或GitLab Secret Detection） |

### 4.7 思考题

1. **启明星辰有两套独立的目录服务：中国总部使用Microsoft AD（域：`cn.qimingxingchen.com`），新加坡分公司使用OpenLDAP（域：`dc=sg,dc=qimingxingchen,dc=com`）。Harbor的 `auth_mode` 只能设置为一种——`ldap_auth` 模式下只能配置一个LDAP服务器。在不部署两个Harbor实例的前提下，设计一种方案让两地的员工都能用各自的LDAP账号登录同一个Harbor实例。请说明你选择的中间件（如Keycloak LDAP Federation）和具体的配置策略。**

2. **Harbor的审计日志记录了每个用户的操作（登录、push、pull、delete等）。设计一个自动化审计脚本——每天凌晨分析前24小时的审计日志，检测并报警以下异常行为：（1）同一用户从超过3个不同地域IP登录；（2）push操作量超过日常平均值的3倍（可能是恶意灌入）；（3）数据库中没有记录的未知用户名尝试登录超过50次（疑似暴力破解）。输出一份Markdown格式的《Harbor安全日报》，发送到安全部门的企业微信/钉钉群。**
