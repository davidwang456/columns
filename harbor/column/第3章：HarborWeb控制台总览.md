# 第3章：Harbor Web 控制台总览

## 1 项目背景

运维新人李明入职"迅捷电商"的第二天，Leader 陈工给他分配了第一项任务："明天下班前熟悉 Harbor 控制台，周四下午带研发团队做一次镜像管理流程演示。演示内容包括——如何创建项目、如何分配权限、如何看懂 Dashboard、以及怎么找到一个特定镜像的最新版本。"

李明信心满满地打开 Harbor Portal（`https://harbor.xunjie.com`），在管理员账号登录的一瞬间，他愣住了——左侧菜单树有 6 个一级目录、30+ 个子页面，Dashboard 首页显示着"公共项目 5 / 私有项目 23""总存储消耗 1.27 TB""漏洞扫描总数 8742""待处理高危漏洞 156"。这么多数字，哪些是正常的、哪些是告警信号、哪些需要立刻处理？李明完全没有头绪。

**痛点一：Dashboard 指标的数字迷雾。** 1.27TB 的存储消耗是正常还是已经踩了红线？8742 个 CVE 漏洞是不是说明公司的镜像安全状况很差？156 个高危漏洞应该优先修哪些？李明打开 Grafana 监控面板做对比，发现 Harbor Dashboard 显示的存储消耗（1.27TB）和 Linux `du -sh /data/harbor` 的结果（1.83TB）差了 560GB。这 560GB 到底去哪了？后来排查才知道——Dashboard 只统计"被至少一个标签引用的镜像层"的大小（即"可达"数据），而磁盘上还有 560GB 是 GC 待回收的孤立 blob、镜像层去重后的元数据开销、PostgreSQL WAL 日志等"不可见"消耗。

**痛点二：功能入口的隐性层级。** Harbor Portal 采用了"6 个一级菜单 + 多层子页签"的导航结构。一些高频操作被藏在了意想不到的位置：垃圾回收（GC）藏在"系统管理"的三级子菜单里而非独立的"垃圾清理"菜单；项目配额设置在项目→"配置"页签中而非项目列表页；CVE 白名单在系统管理→"拦截器"中而非项目的"策略"页签。李明在"系统管理"里翻了三个子菜单才找到 GC 入口，期间还不小心点进了 P2P 分发配置页面——一个公司从未使用过的 Dragonfly 集成功能，差点改错了配置。

**痛点三：多团队协作的权限分配困惑。** 研发团队要把镜像推到哪个项目？测试团队从哪个仓库拉镜像？运维如何给不同业务线分配最小权限？李明不清楚 Harbor 中"项目-仓库-镜像"的层级关系如何映射到真实的团队组织结构。更麻烦的是——Harbor 的项目角色有 5 种（访客、开发者、维护者、项目管理员、系统管理员），每种角色的权限矩阵（Pull？Push？Delete？Manage Members？）只有查阅官方文档才能搞清楚。李明在准备演示时，误把一个需要 push 权限的研发同事只给了"访客"角色，导致该同事推镜像时反复报 401，一度怀疑 Harbor 出了问题。

**痛点四：生产环境操作的安全焦虑。** 李明发现"系统管理"菜单下有"仓库只读"开关——只要勾选，整个 Harbor 所有项目的 Push 操作全部被阻止。他还发现"项目"→"配置"页签中的"删除项目"按钮没有任何二次确认弹窗（v2.8 之后已加上）。更令他不寒而栗的是——"垃圾回收"菜单里的"立即执行 GC"按钮，点击后直接开始清理不可达 blob，而 GC 期间如果有人在 push 镜像，可能导致镜像层损坏。这些"一键毁灭"操作没有任何操作提示和风险说明。

**痛点五：API 和 Portal 的不对称能力。** 李明在准备演示时发现——批量创建 10 个项目、批量给 20 个用户分配权限、批量修改所有项目的 CVE 阻止策略——这些在 Portal 上只能一个一个手动操作，而在 Harbor REST API 中一行 curl 就能搞定。但问题是：API 文档分散在 Swagger UI（`/devcenter`），且部分 API 端点（如 v1.0 vs v2.0）的请求格式不同。李明想用 API 批量操作，但不知道从哪个端点开始。

本章将完成一次 Harbor Portal 的完整功能巡览——从 Dashboard 的每一个数字的底层业务含义，到 6 个一级菜单的全部子页签操作详解，建立 Harbor 控制台的全景认知地图。

## 2 项目设计——剧本式交锋对话

**场景：周三下午 15:30，李明对着 Harbor Portal 的 Dashboard 页面发呆。屏幕左侧是密密麻麻的菜单树，右侧 Dashboard 上四个彩色数字方块——"1.27TB""5/23""8742""156"。大师从前台打完电话回来，路过李明工位。**

**小胖**（端着奶茶凑过来）："哇明哥，这界面花花绿绿的——比我玩的手游还复杂！你看这几个数字——1.27TB、8742、156，感觉像游戏里的伤害数值。8724 个漏洞是不是说明咱们的镜像千疮百孔啊？我都不敢把我写的代码打包成镜像了！"

**李**（苦笑）："别说了，Leader 让我明天给研发团队做 Harbor 演示，我现在连这几个数字啥意思都没搞懂。1.27TB 到底算正常还是快撑爆了？8742 个漏洞跟我们的 47 个微服务怎么对应的？156 个高危到底先修哪个？"

**大师**（放下手中的咖啡杯，拉过一把椅子坐下）："别慌，Dashboard 上的每个数字都不是为了吓唬你，它们每个背后都有明确的业务含义。我把它们分成'看趋势'和'看告警'两类——"

"**第一类：看趋势（需要历史对比才有意义）**——"

"- **总存储消耗（1.27TB）**：这是 Harbor 中所有**被至少一个标签引用**的镜像层的去重后大小。注意三个关键词——'去重后'意味着同一个 Ubuntu:22.04 基础层被 100 个业务镜像引用，只算一份；'趋势'意味着你应该关注的是——上周是 1.1TB、这周 1.27TB，周增长 170GB，按这个速度一个月后就需要扩容；'不是全部'意味着磁盘上还有 GC 待回收的孤立 blob、PostgreSQL WAL 日志、Trivy 漏洞数据库缓存等，实际的 `du -sh /data` 会比这个数字大 30-50%。"

"- **公共/私有项目数（5/23）**：公开项目（5 个）应该放基础镜像（openjdk、nginx、python 等）和代理缓存，任何人都能拉取但只有管理员能推送；私有项目（23 个）放业务镜像，需要授权才能访问。如果你发现私有项目数和业务线数量不匹配——比如只有 6 条业务线但有 23 个私有项目——说明有过多的临时测试项目或者离职同事的遗留项目，需要清理。"

"- **漏洞总数（8742）**：这是所有镜像最近一次扫描结果中 CVE 数量的简单求和（不是去重求和）。举个例子——如果同一个 CVE-2024-1234 存在于 50 个镜像中，它会被计算 50 次。所以 8742 这个数字不要直接用来评估风险——要看的是**严重级别的分布**。如果 8742 中 80% 是 Low/Medium，且大部分在基础镜像层里（你无法修改 openjdk 的 CVE），那你的安全状况可能并不差。"

"**第二类：看告警（数字异常需要立刻行动）**——"

"- **严重/高危漏洞数（156 High / 23 Critical）**：这才是需要你紧盯的数字。Critical（严重）CVE 意味着攻击者可以远程执行代码或提权——比如 Log4Shell（CVE-2021-44228）这种级别。如果你发现 Critical 数量突然从 2 跳变到 23，说明最近推送的某个镜像引入了严重的已知漏洞，需要立刻定位是哪个镜像并通知对应的开发团队。"

"- **存储配额告警**：Dashboard 不会直接显示配额告警，但如果你进入项目列表，会看到某个项目的存储消耗条变成了红色——说明该项目已经达到或超过配额的 90%。这时候需要扩容配额或通知项目负责人清理旧镜像。"

**技术映射**：Dashboard 的数据源是 Core API 的 `/api/v2.0/statistics` 接口，底层从 PostgreSQL 的多个表（`project`、`repository`、`blob`、`artifact`、`scan_report`）聚合而来。存储消耗通过 `blob.size` 表汇总计算（sum of sizes of all blobs referenced by at least one tag）。漏洞数从 Trivy 的 `vulnerability_record` 表按严重级别（Critical/High/Medium/Low/None）聚合。

**小胖**（把奶茶一饮而尽）："那是不是我每天早上打开 Dashboard 看一眼这几个数字，就能高枕无忧了？"

**大师**："小胖你这个思路很危险。Dashboard 是'仪表盘'——你开车时看仪表盘（时速、油量、水温）很重要，但如果你只看仪表盘不看路（具体项目、具体仓库、具体镜像），你照样会撞车。Dashboard 告诉你'当前系统是否健康'，但你还需要**下钻**到具体项目、具体仓库、具体镜像去定位问题的根源。"

"比如——Dashboard 显示存储消耗 1.27TB，环比上周增长 17%。这时候你应该下钻到'项目→存储消耗排名'，看看到底是哪个项目在疯狂增长。可能是 `ci-snapshots` 项目里 Jenkins 每天构建产生的 43 个快照镜像没人清理。如果你只看 Dashboard，你只会知道'存储空间在减少'，但不会知道是谁造成的。"

**小白**（一直在安静地做笔记，此时合上笔记本抬头）："大师，左侧菜单树一共 6 个一级菜单，每个下面还有一堆子页签。我想给它们排一个'使用频率排行榜'——哪些是我每天都要用的，哪些是一个月才用一次甚至根本用不到的？"

**大师**："好，我按频率从高到低排——"

| 优先级 | 菜单/页签 | 使用角色 | 使用频率 | 核心操作 |
|--------|---------|---------|---------|---------|
| P0 | 项目→仓库列表 | 所有人 | 每天 10+ 次 | 浏览镜像、查看版本、拉取命令复制 |
| P0 | 项目→成员管理 | 项目管理员 | 每周 3-5 次 | 添加/移除成员、调整角色 |
| P1 | 仪表盘 | 运维/管理员 | 每天 1-2 次 | 检查系统健康、存储趋势 |
| P1 | 项目→配置 | 项目管理员 | 每周 1-2 次 | 修改配额、CVE 策略、标签规则 |
| P1 | 系统管理→用户管理 | 管理员 | 每周 1-2 次 | 创建/禁用用户、重置密码 |
| P2 | 项目→日志 | 运维/安全 | 按需查询 | 审计追溯——谁在什么时间做了什么 |
| P2 | 垃圾回收 | 运维 | 每月 1-2 次 | 手动触发 GC、查看历史释放空间量 |
| P3 | 项目→Webhook | 架构师 | 按需配置 | 镜像 Push/Delete 事件通知 |
| P3 | 系统管理→配置 | 管理员 | 按需修改 | 外部 URL、会话超时、CVE 白名单 |
| P4 | 项目→复制 | 运维 | 按需配置 | 跨 Harbor 同步规则 |
| P4 | 项目→扫描器 | 运维 | 初始配置一次 | 选择扫描器（Trivy/Clair） |
| PX | 分发 (P2P) | 几乎不用 | 几乎从不 | Dragonfly P2P 分发（需额外部署） |

**小胖**（惊讶）："'分发'这个菜单我从来没点过——它是干啥的？"

**大师**："'分发'菜单是 Harbor 集成的 Dragonfly P2P 镜像分发功能。在超大规模 Kubernetes 集群中——比如你有 500 个 Node 同时需要拉取同一个镜像——传统的 Registry 模式会成为瓶颈。Dragonfly 利用 BitTorrent 协议的思路，让 Node 之间互相传输镜像层，Registry 只充当源头种子。但这个功能需要额外部署 Dragonfly 集群，普通公司（<50 Node）完全用不上。如果你没有部署 Dragonfly，这个菜单就等于摆设。"

**小白**（翻开笔记本新的一页）："大师，我注意到项目详情页有 10+ 个子页签（仓库、成员、标签、复制、扫描器、策略、日志、配置、机器人账户、P2P 策略、Webhook），能不能用一句话概括每个页签是干嘛的？以及——哪些是'设一次就忘'的配置，哪些是'每天都要操作'的功能？"

**大师**："这样，我把项目内 11 个子页签分成'高频操作'和'一次性配置'两类——"

"**高频操作（每周都在用）**："

1. "**仓库（Repositories）**：这就是你的镜像浏览器。相当于文件夹里的文件列表——你能看到该项目下有哪些镜像仓库、每个仓库有多少个标签、最新推送时间。点进具体仓库可以看到所有标签列表、每个标签的漏洞扫描结果、镜像大小、拉取命令。**核心能力：找到你要用的镜像，复制命令给 CI 流水线。**"

2. "**成员（Members）**：项目的权限控制中心。相当于房间的'门禁卡管理'——你能添加/移除用户或用户组，并分配 5 种角色之一。项目管理员和系统管理员才能操作这个页签，普通开发者和访客只能看到成员列表但不能修改。"

3. "**日志（Logs）**：该项目的操作审计追踪。相当于房间门口的'摄像头回放'——记录了这个项目内所有的操作：谁（用户名）、什么时候（精确到秒）、从哪个 IP、做了什么（push/pull/delete/create tag）。出安全问题时，这里是第一手的调查入口。"

"**一次性配置（设完基本不用动）**："

4. "**配置（Configuration）**：项目的全局开关——公开/私有、存储配额、CVE 阻止策略、自动扫描开关。一次性配置好，后续基本不用改，除非业务需求变化（如配额扩容）。"

5. "**标签（Labels）**：自定义标签系统。你可以给某些仓库打上 `production-ready` 或 `deprecated` 的标签，方便分类管理。但这个功能用得极少——大部分团队直接用项目名和仓库名来区分，不需要额外的标签。"

6. "**机器人账户（Robot Accounts）**：给 CI/CD 系统（Jenkins、GitLab CI、GitHub Actions）创建的自动化账户。相比于给 Jenkins 分配一个完整的人类用户账号（有密码、会过期、能登录 Portal），机器人账户更安全——它只能走 API 访问，有独立的 Secret，可以限定到项目、仓库级别。**CI/CD 集成时必须用机器人账户。**"

7. "**Webhook**：镜像事件通知。当有人推送或删除了镜像，Harbor 可以通过 Webhook 向你的 CI/CD 系统或告警平台（钉钉/飞书/Slack）发送通知。比如——有人向生产项目推送了一个新镜像，自动触发一条通知到运维群。"

8. "**复制（Replications）**：配置该项目的跨 Harbor 复制规则。生产 Harbor 的镜像自动同步到灾备 Harbor——全靠这个页签。不过现在推荐用系统管理级的复制规则（可以批量管理多个项目），项目级的复制规则只适用于特殊场景。"

9. "**扫描器（Scanner）**：选择该项目使用的漏洞扫描器。默认只有一个 Trivy 适配器，不需要修改。除非你部署了额外的扫描器（如 Aqua、Anchore），在这里切换。"

10. "**策略（Policy）**：仅对 Proxy Cache 类型的项目生效——配置从上游 Registry 拉取镜像的缓存策略。普通项目这个页签是空的。"

11. "**P2P 策略（P2P Policy）**：仅 P2P 分发场景使用，普通团队永远用不到。"

**技术映射**：Project → Repository → Artifact → Tag 的四级层级是 Harbor 的核心数据模型。项目（Project）是顶级命名空间，所有镜像的入口。仓库（Repository）对应一个微服务或组件。制品（Artifact）是同一仓库下不同 digest 的镜像版本。标签（Tag）是指向特定 Artifact 的"指针"——一个 Artifact 可以有多个标签（如 `v1.2.3` 和 `latest` 指向同一个 digest）。

**小胖**："那系统管理菜单呢？我点进去看到里面有——用户管理、垃圾回收、复制管理、系统设置、拦截器——这些是运维的专属领地吗？"

**大师**："基本是运维和管理员的领地。但开发也需要了解其中的三个——"

"**用户管理**：创建/禁用/删除用户。开发不需要操作这个，但如果你的账号登录不了、密码忘了、状态是'未激活'，就要找管理员在这里处理。用户管理还有一个容易被忽略的功能——'分配系统角色'。系统管理员可以把另一个用户提升为管理员。如果一个公司只有一个 admin 账号，万一 admin 的密码丢了……你们懂的。"

"**垃圾回收（GC）**：你会看到——上次 GC 时间（如 2024-04-05 02:30:15）、释放了多少空间（如 230GB）、是否成功。GC 需要手动点击'立即执行 GC'按钮——点击前一定记得先把 Harbor 设为只读模式。这个按钮不是'先确认再执行'的，是点了就立刻开始——GC 的执行时间取决于孤立 blob 的数量，可能 5 分钟也可能 2 小时。"

"**系统设置（Configuration）**：这里是最容易被忽视但也是最重要的运维配置区域。几个关键配置项：①'外部 URL'——如果配错，Docker 客户端的 token 校验会 401；②'仓库只读模式'——紧急故障时防止任何人推送镜像，相当于给整个 Harbor 按下暂停键；③'机器人账户 Token 过期时间'——默认 30 天，如果你的 CI/CD 流水线要长期运行，设成 365 天；④'CVE 白名单（拦截器）'——允许特定的 CVE 绕过扫描门禁。"

**小白**（突然放下笔）："等一下大师，你提到了'机器人账户 Token 过期时间'。我们今天早上刚好遇到一个问题——Jenkins 流水线突然报 401，排查发现是三个月前创建的机器人账户 Token 过期了。有没有办法让机器人账户的 Token 自动续期或者永不过期？"

**大师**："好问题，这牵涉到 Harbor 的安全模型。机器人账户的设计哲学是——宁可让 CI/CD 偶尔中断（Token 过期），也不能留下一个永不过期的后门。但实际操作中，如果你明确知道某些机器人账户的用途（比如生产 CI 流水线），可以在创建或修改机器人账户时手动设置较长的过期时间（最长 365 天）。"

"另外还有一个最佳实践——**创建多个机器人账户，每个用于不同的 CI/CD 流水线**。

比如：
- `robot$order-service-ci` —— 仅限 `order-platform` 项目的 Push 权限，Token 365 天
- `robot$payment-service-ci` —— 仅限 `payment-platform` 项目的 Push 权限，Token 365 天
- `robot$base-image-sync` —— 仅限 `shared-base` 和 `proxy-cache` 项目的 Pull+Push 权限，Token 180 天

这样即使某个机器人账户泄露或过期，影响面也被控制在单个项目内。"

**技术映射**：机器人账户的名称格式为 `robot$<account_name>`。在 API 层面，机器人账户和普通用户账户是两个独立的实体——机器人账户不能登录 Portal（没有 Web 交互界面），只能通过 CLI/API 使用。其 Secret 生成后只展示一次——如果你忘了复制 Secret，只能删除重建。

**小胖**（拿过一个笔记本开始画）："我总结一下——Harbor Portal 的核心使用路径是——每天早上打开 Dashboard 看一眼健康状态，然后进'项目'找到你的业务项目，点'仓库'找镜像、复制拉取命令；如果需要加新人，去'成员'页签；如果系统整体配置出问题了，才需要去'系统管理'。对吧？"

**大师**："小胖你这次总结得非常精准。我再补充一条——如果你发现 Portal 做不到的操作（如批量创建 10 个项目、批量修改 50 个仓库的标签策略），不要硬着头皮一个一个在 Portal 上点——去用 Harbor REST API。Portal 是为'人的手动操作'设计的，API 是为'机器的自动化操作'设计的。下一章讲到具体的镜像 Push/Pull 时，我会详细展开 API 的使用。"

### 3.5 可能遇到的坑

**坑1：Dashboard 数据显示不刷新（数据显示滞后）**

- **现象**：刚 push 了一个 500MB 的镜像，Dashboard 上的"总存储消耗"数字没有变化，等待 2 分钟后仍无变化。
- **根因**：Dashboard 数据不是实时刷新的。Harbor 使用缓存机制（默认来自 PostgreSQL 的物化视图或 Core 的内存缓存），刷新间隔约 5 分钟。此外，如果刚 push 的镜像层与已有镜像有大量共享层（去重），实际新增的存储可能很小（仅几十 MB），Dashboard 数字变化不明显。
- **解决方法**：
  1. 等待 5 分钟后手动刷新页面（F5）
  2. 点击 Dashboard 页面右上角的手动刷新按钮（如果有）
  3. 确认不是"去重导致的数字变化不明显"——进入项目 → 仓库列表，查看该仓库的存储大小

**坑2：新创建的用户登录 Portal 后看不到任何项目**

- **现象**：管理员在"系统管理→用户管理"创建了用户 `zhangwei`，该用户在登录 Portal 后，Dashboard 上的"项目"列表一片空白。
- **根因**：Harbor 的用户管理和项目成员管理是两个独立的操作。创建用户只是注册了一个"账号"，但这个账号尚未被添加到任何一个项目的"成员"列表中。默认情况下，新用户没有任何项目的访问权限（即使是公开项目，也只能 Pull 不能查看项目详情）。
- **解决方法**：
  1. 管理员登录 Portal → 进入目标项目 → 成员页签 → "添加成员" → 搜索用户名 → 选择角色 → 确认
  2. 或者通过 API 批量添加：
     ```bash
     curl -k -u admin:Harbor12345 -X POST \
       -H "Content-Type: application/json" \
       -d '{"role_id": 2, "member_user": {"username": "zhangwei"}}' \
       https://harbor.yourcompany.com/api/v2.0/projects/1/members
     # role_id: 1=项目管理员, 2=开发者, 3=访客, 4=维护者
     ```

**坑3：修改了"外部 URL"后 Portal 页面仍然跳转旧地址**

- **现象**：修改了系统管理→配置中的"外部 URL"从 `http://10.0.1.50` 改为 `https://harbor.xunjie.com`，但 Portal 内的链接（如通知邮件中的链接、页面跳转）仍使用旧地址。
- **根因**：修改"外部 URL"后，Harbor Core 和 Portal 容器内部已经加载了旧配置。需要重启容器才能使新配置生效。
- **解决方法**：
  ```bash
  cd /opt/harbor
  docker compose restart harbor-core harbor-portal
  # 等待约 30 秒后刷新 Portal 页面
  ```

**坑4：CVE 白名单设置后已经存在的扫描结果不更新**

- **现象**：在系统管理→配置→"CVE 白名单"中添加了一个 CVE-2024-1234，但之前扫描出该 CVE 的镜像仍然显示"Critical"，且在部署时仍被阻止。
- **根因**：CVE 白名单只对**新执行的扫描结果**生效。已存在的扫描报告不会因为白名单的新增而自动重新评估。需要手动重新触发扫描。
- **解决方法**：
  1. 进入项目 → 仓库 → 选择受影响的仓库 → 选择制品 → 点击"重新扫描"
  2. 或通过 API 触发批量重新扫描：
     ```bash
     curl -k -u admin:Harbor12345 -X POST \
       https://harbor.yourcompany.com/api/v2.0/projects/order-platform/repositories/order-service/artifacts/sha256:abc123.../scan
     ```

### 4.3 适用场景

- **新人 Onboarding**：通过 Portal 直观理解 Harbor 的三级镜像管理模型（项目→仓库→制品），无需阅读 API 文档
- **日常运维监控**：Dashboard + 健康状态面板快速判断系统是否正常，发现异常时下钻到项目/仓库级定位
- **权限快速分配**：通过项目→成员页面可视化分配角色，比 API 更直观
- **安全事件溯源**：日志页面的关键词搜索（用户名/IP/操作类型）快速定位"谁删了镜像""谁修改了策略"
- **系统维护窗口**：在 GC、升级等维护操作前，通过"仓库只读"开关一键暂停所有 Push 操作
- **镜像版本查看与拉取命令复制**：仓库列表页一键复制 `docker pull` 命令，提升研发效率

### 4.4 不适用场景

- **批量操作（>5 个对象）**：创建 10 个项目、添加 20 个用户到项目、修改 15 个仓库的标签策略——Portal 逐个操作效率极低，请使用 API 脚本
- **CI/CD 自动化集成**：流水线中的镜像 Push/Pull 应调用 Harbor REST API + 机器人账户，绝不依赖 Portal
- **实时数据监控告警**：Dashboard 有 5 分钟缓存延迟，不适用于实时告警。请将 Harbor 指标接入 Prometheus + Grafana + AlertManager
- **多 Harbor 实例统一管理**：如果有 3 套 Harbor（测试/预发布/生产），Portal 只能逐个登录管理，无法跨实例批量操作

### 4.5 注意事项（Portal 操作陷阱）

1. **"立即执行 GC"按钮无二次确认**：点击后立即开始清理操作，且执行期间可能影响 Push 操作。务必先通过 API 将 Harbor 设为只读模式。
2. **修改"外部 URL"后必须重启 Core 和 Portal 容器**，否则 Portal 链接和 Docker Token 签发地址不一致。
3. **CVE 白名单仅对新扫描生效**：添加白名单后，已有的扫描报告不会自动更新，需要手动触发重新扫描。
4. **机器人账户 Secret 只展示一次**：创建后如果忘记复制 Secret，只能删除重建（无法恢复）。建议创建后立即复制到密钥管理工具（如 Vault）。
5. **项目删除有 72 小时软删除期**：删除项目后无法立即创建同名项目。数据库记录标记 `deleted=true` 保留 72 小时，超时后物理删除。
6. **Portal 用户 Token（CLI Secret）和机器人账户 Secret 是不同的**：CLI Secret 替代用户密码用于 API 请求（可手动撤销）；机器人账户 Secret 用于 CI/CD 自动化（有过期时间）。
7. **"仓库只读"模式影响所有项目**：这是一个全局开关，启用后所有项目（包括管理员）都无法 Push。仅用于紧急故障场景。

### 4.6 常见踩坑经验（生产操作故障案例）

| 故障案例 | 故障现象 | 根因诊断 | 修复措施 | 教训总结 |
|---------|---------|---------|---------|---------|
| **"误触 GC 按钮导致 Push 镜像损坏"** | 运维在上午 10:30 高峰期点击了"立即执行 GC"按钮（以为只是查看历史记录）。GC 执行 12 分钟后，Jenkins 正在 Push 的 3 个镜像全部损坏——Push 完成后 Registry 报 `MANIFEST_UNKNOWN` | GC 在标记可达 blob 和删除不可达 blob 之间存在时间窗口。此时有 3 个 Push 操作正在写入新 blob 但 manifest 尚未关联——GC 将这些正在写入的 blob 标记为不可达并删除 | ①立即停止 GC（无法中途停止，只能等完成）；②从备份恢复损坏的镜像；③请求开发重新推送受影响版本 | Harbor 的 GC 按钮设计存在 UX 隐患——"查看历史"和"执行 GC"在同一个页面上。建议：①通过 Harbor API 权限限制，仅特定管理员账号可见 GC 按钮；②在 GC 按钮旁增加醒目文字"⚠ 点击后立即开始清理，请设置只读模式" |
| **"用户被提升为系统管理员后获取了所有项目权限"** | 运维为方便排障，将开发者 `wangwu` 提升为系统管理员。`wangwu` 随后发现自己能看到、拉取甚至删除所有业务项目的镜像——包括之前被拒绝访问的支付模块 | 系统管理员拥有 Harbor 内所有资源的完全访问权限——所有项目的 Pull/Push/Delete/Manage Members 权限。这是设计如此，而非 Bug | ①紧急撤销 `wangwu` 的管理员角色，降级回开发者；②审计 `wangwu` 在管理员期间的所有操作日志 | 系统管理员角色的权限边界需要明确——这是一个"上帝模式"角色，不是"高级运维"角色。日常排障应该使用项目级权限（项目管理员）而非系统管理员。分配系统管理员角色前必须走审批流程 |
| **"标签保留策略清理了 K8s 生产环境正在使用的镜像"** | 运维在 `order-platform` 项目配置了"保留最近 5 个制品"的标签保留策略，每日凌晨 3:00 执行。三周后，K8s 集群中运行了 21 天的 `order-service:v1.1.0` 的标签被保留策略删除。扩容新 Node 时 Kubelet 报 `Failed to pull image: not found` | v1.1.0 是 21 天前推送的第 8 个旧制品，不在"最近 5 个"范围内，被保留策略标记为可清理。GC 随后清理了该标签对应的 blob。虽然已有 Node 上本地缓存该镜像（`imagePullPolicy: IfNotPresent`），但新 Node 没有缓存 | ①对生产环境镜像使用**标签不可变性**而非依赖保留策略；②给生产版本的镜像打上 `release-*` 标签并设置不可变；③确保生产环境 Deployment 的 `imagePullPolicy: Always` 或用固定 digest | 标签保留策略和标签不可变性是两个互补的机制——前者管"自动清理多少版本"，后者管"永远不要清理这些版本"。生产镜像应同时配置两者：①`release-*` 标签设为不可变；②开发/CI 快照标签用保留策略控制数量 |

### 4.7 思考题

**问题 1**：Dashboard 上显示的"总存储消耗"数字（比如 1.27TB）和 Linux 命令 `du -sh /data/harbor/registry/` 的结果通常不一致——后者比前者大 30-50%。请分析这 30-50% 的"差距"由哪些数据构成？如果要准确获取 Harbor 实际占用的全部磁盘空间，应该统计哪些目录和文件？（提示：考虑 GC 待回收的孤立 blob、PostgreSQL WAL 日志、Trivy 缓存、Docker 容器日志、Docker 镜像层。Dashboard 只统计 Core API 返回的统计值，而非文件系统级别的磁盘占用。）

**问题 2**：Portal 中项目设置"公开"后，任何未登录的用户都能通过 `docker pull` 拉取该项目的镜像。但是，如果一个非登录用户试图通过 Portal 浏览器页面访问公开项目下的"仓库列表"或"成员列表"页面，能否看到内容？Docker Client 的 Pull 和 Portal 的页面访问在认证鉴权上有什么本质区别？（提示：Docker Client 走的是 OCI Distribution 协议的匿名拉取（anonymous pull），Portal 页面走的是 Core API 的 token-based 认证。公开项目在 Registry 层面允许匿名 Pull，但在 Core API 层面仍然要求登录态才能查看详细列表。）
