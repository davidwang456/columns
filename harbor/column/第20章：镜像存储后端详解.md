# 第20章：镜像存储后端详解

## 1 项目背景

某跨境零售企业"环球优选"将Harbor从单机部署迁移到K8s集群后，存储策略面临全面重新评估。Harbor管理着12TB镜像数据，每天新增约50GB，同时有5个区域的K8s集群需要拉取镜像。旧方案的问题在迁移后集中暴露。

**痛点一：本地存储无法跨节点共享。** Docker Compose时代，Harbor用本地SSD存储镜像（`/data/registry/`），Registry只有一个实例。迁移到K8s后，Pod可能被调度到任意Node——如果之前某个Node上的本地磁盘中存有Blob，而Registry Pod被重新调度到了另一个Node，该Blob就找不到了，`docker pull`直接报`MANIFEST_UNKNOWN`。团队这才意识到本地存储是"有状态"的，和K8s的"无状态Pod"理念冲突。

**痛点二：存储成本失控。** 本地SSD存储12TB镜像数据，使用高性能NVMe SSD——每月存储成本约7200元（按600元/TB/月计算）。而实际上，12TB数据中有70%是超过3个月未被拉取的"冷镜像"——这些旧版本一直占着昂贵的SSD。如果能把冷数据迁移到对象存储（S3约120元/TB/月），存储成本可降低80%以上。

**痛点三：高峰期磁盘I/O瓶颈。** 双11大促前夕，180个K8s Node同时拉取基础镜像（如`java-base:11`、`node:18-alpine`），Registry的本地SSD磁盘I/O达到极限——`iostat`显示`await`超过200ms，磁盘队列深度（`avgqu-sz`）峰值超500，导致`docker pull`频繁超时（10分钟超时阈值），部分Node的Pod原地`ImagePullBackOff`。

**痛点四：备份与灾备流程复杂且耗时。** 本地存储的定期备份需要`rsync`完整同步`/data/registry/`到备机房的NAS。12TB数据每次全量备份需要6小时（跨专线网络带宽限制），备份窗口挤占了业务I/O带宽，导致备份期间的镜像推送速度下降60%。更糟糕的是，如果源机房故障，需要手动从备机房的NAS恢复——整个RTO（恢复时间目标）至少4小时。

Harbor通过Docker Distribution的存储驱动抽象层支持7种存储后端——Local FS、AWS S3、Azure Blob、Google GCS、OpenStack Swift、阿里云OSS、华为云OBS。本章将详解各存储后端的架构、性能特征、选型决策树和实际迁移方案。

## 2 项目设计——剧本式交锋对话

**场景：基础设施技术选型会，讨论了1小时还没有结论，白板上贴满了各种存储产品的logo。**

**小胖**（不耐烦地转着笔）："存储不就是个存文件的地方吗？本地磁盘、S3、OSS——不都一样吗？文件放进去、读出来——有什么好纠结的？选最便宜的不就完了？"

**大师**（在白板上画了一个两轴坐标图）："小胖，同一个文件放在不同的存储系统上，读写性能可能差10到50倍。关键区别在三个维度——**延迟、吞吐、成本**。我给你一个直观对比："

| 存储类型 | 首字节延迟 | 吞吐量 | 月成本/TB | 代表 |
|---------|----------|-------|----------|------|
| 本地NVMe SSD | < 0.5ms | 3.5 GB/s | ~600元 | `/data/registry/` |
| 本地SATA SSD | < 1ms | 550 MB/s | ~300元 | - |
| MinIO (同机房) | 3-8ms | 1-2 GB/s | ~150元 | 自建S3兼容 |
| AWS S3 (同Region) | 10-50ms | 数GB/s | ~120元 | 公有云对象存储 |
| 阿里云OSS (内网) | 5-20ms | 数GB/s | ~100元 | 阿里云对象存储 |

"本地SSD延迟最低但最贵，适合热数据——频繁被拉取的最新几个版本。对象存储延迟高一到两个数量级，但便宜5-6倍——适合冷数据，比如三个月前的旧版本。这是经典的**热-冷存储分层**思路。"

**技术映射**：Harbor通过Docker Distribution的存储驱动（Storage Driver）抽象层实现多后端支持。每个后端驱动实现相同的`storagedriver.StorageDriver` Go接口，包含`GetContent()`、`PutContent()`、`Stat()`、`Delete()`、`List()`、`Move()`、`URLFor()`等方法。切换存储后端只需修改`harbor.yml`中的`storage_service`配置块，不会影响镜像内容本身——因为OCI镜像的内容可寻址（Content-Addressable），digest由内容决定而非存储位置。

**小白**："我要搞清楚每种存储后端在Harbor场景下的真实表现。特别是S3的'最终一致性'对镜像push/pull有什么影响？我听说S3写入后立即读取可能读不到？"

**大师**："这个问题问到点上了。Harbor的镜像上传使用分块上传（Chunked Upload）模式——Docker客户端把一个大Blob切成多块逐一上传，最后发一个'Complete Multipart Upload'请求来合并它们。如果存储后端是'最终一致性'的，可能出现以下致命问题："

```
时间线：S3最终一致性场景
T1: Docker上传Chunk 1 → S3返回200 OK（但后端节点间尚未完全复制）
T2: Docker上传Chunk 2 → S3返回200 OK
T3: Docker上传Chunk 3 → S3返回200 OK
T4: Docker发起 Complete Multipart Upload
    → S3需要读取所有Chunk做合并
    → Chunk 1 读取失败！（因为它所在的节点还没来得及同步副本）
    → 合并失败 → push报错 → Docker重试整个push
```

"**好消息是**——AWS S3自2020年底起，所有区域都支持了'写后读一致'（Read-after-Write Consistency）——你在US-East-1写入后立即读取，一定是最新版本的。MinIO从RELEASE.2020版本起也默认强一致了。但如果你用的是自建的S3兼容存储（如Ceph RGW的早期版本），一定要确认其一致性模型——否则会随机出现push失败。"

**技术映射**：Registry的S3驱动通过`aws-sdk-go`库调用S3 API。关键操作序列为：`CreateMultipartUpload` → `UploadPart`（多次）→ `CompleteMultipartUpload`。如果`CompleteMultipartUpload`返回`NoSuchKey`或`NoSuchUpload`错误，说明存在一致性问题。错误处理逻辑在Docker Distribution的`registry/storage/driver/s3-aws/s3.go`中。

**小胖**："那到底选哪种啊？给个痛快话——我们公司有AWS、阿里云、还有自建IDC的K8s。这么多场景怎么选？"

**大师**（笑着拿出一张准备好的决策表）："我就知道你要问这个。看场景，不是一刀切："

| 部署场景 | 推荐存储 | 理由 |
|---------|---------|------|
| 单机Docker Compose / 小团队(<20人) | Local FS | 简单、低延迟、零外部依赖 |
| K8s集群（自建IDC） | MinIO (S3兼容) | K8s原生Operator部署、可扩展、比SSD便宜 |
| K8s集群（AWS上） | AWS S3 | 免运维、与EKS/ECR深度集成、弹性无限扩展 |
| K8s集群（阿里云上） | 阿里云OSS | 内网Endpoint免流量费、延迟5-20ms |
| 混合云 / 多Region分发 | S3 + Cross-Region Replication | 全球化分发 + 异地灾备自动同步 |
| 金融等高合规行业 | 本地FS + Ceph RBD | 数据不出IDC、可控IOPS、自主加密 |

**小白**（若有所思）："我还有一个深层问题——如果Harbor配置了S3存储后端，`docker pull`时数据流是怎样的？镜像的Blob是Registry从S3下载然后转发给Docker Client——还是能直接让Client从S3读？如果是前者，Registry岂不是变成了代理，消耗了大量带宽？"

**大师**："非常好的性能问题。答案取决于是否启用**Redirect模式**。默认情况下："

```
默认模式（Proxy模式）：
Docker Client → Registry Pod → S3（下载Blob到Registry内存）
                              → Registry转发给Docker Client
带宽消耗：S3→Registry + Registry→Client（双倍）
Registry内存需要能容纳一个Blob的所有分块

Redirect模式（S3签名URL）：
Docker Client → Registry Pod → Registry生成S3预签名URL返回
                              → 302重定向
Docker Client → 直接用预签名URL从S3下载
带宽消耗：仅S3→Client（Registry不消耗带宽）
但Docker Client必须能直连S3（内网可达）
```

"Redirect模式大幅减轻Registry的带宽压力，但要求Client网络能直连S3。在K8s环境中，如果Pod运行在VPC内且S3 Endpoint可达——强烈推荐开启Redirect模式。配置项：`storage_service.s3.redirect.disable: false`。"

**技术映射**：Redirect模式通过`storagedriver.StorageDriver`接口的`URLFor(ctx, path, options)`方法实现——生成一个有时效的签名URL（通常5-15分钟有效），直接返回给Client。源码在`registry/handlers/blob.go`的`blobHandler.ServeBlob()`函数中——检测`storageDriver`是否实现了`URLFor`方法，如果实现了且配置允许，则返回302重定向。

### 3.2 部署MinIO并配置Harbor使用S3存储

**目标**：从Local FS迁移到S3兼容存储，学习完整配置流程。

```bash
# Step 1: 部署MinIO测试实例
docker run -d --name minio \
  --network harbor_harbor \
  -p 9000:9000 -p 9001:9001 \
  -e MINIO_ROOT_USER=minioadmin \
  -e MINIO_ROOT_PASSWORD=minioadmin \
  -v minio_data:/data \
  minio/minio server /data --console-address ":9001"

# Step 2: 创建Bucket
# 方法A: 通过MinIO Console
# 浏览器打开 http://192.168.1.100:9001 → 登录 → Buckets → Create Bucket
# Bucket Name: harbor-registry

# 方法B: 通过mc命令行
docker run --rm --network harbor_harbor \
  -e MC_HOST_minio=http://minioadmin:minioadmin@minio:9000 \
  minio/mc mb minio/harbor-registry
```

```yaml
# harbor.yml — S3/MinIO存储配置
storage_service:
  s3:
    accesskey: minioadmin
    secretkey: minioadmin
    region: us-east-1
    regionendpoint: http://minio:9000       # 使用Docker网络中的服务名
    bucket: harbor-registry
    encrypt: false
    secure: false                           # MinIO默认HTTP（测试）
    skipverify: true                        # 跳过证书验证
    v4auth: true
    rootdirectory: /registry                # Bucket内的前缀路径
    # 分块上传参数
    chunksize: 5242880                      # 分块大小 5MB（适合小文件）
    multipartcopychunksize: 33554432        # 复制时分块大小 32MB
    multipartcopymaxconcurrency: 100        # 最大并发复制数
    multipartcopythresholdsize: 33554432    # 超过此大小使用分块复制
    # Redirect模式（可选）
    # redirect:
    #   disable: false                      # 启用重定向（要求Client能直连MinIO）
```

```bash

cd /opt/harbor
```

### 3.4 存储后端迁移实战（零停机方案）

**目标**：从Local FS迁移到S3，同时保持Harbor在线。

```bash
# 方案一：使用rclone（需停机约30分钟）
# Step 1: 安装rclone
curl -O https://downloads.rclone.org/v1.65.0/rclone-v1.65.0-linux-amd64.deb
sudo dpkg -i rclone-v1.65.0-linux-amd64.deb

# Step 2: 配置MinIO remote
rclone config create minio-harbor s3 \
  provider Minio \
  endpoint http://minio:9000 \
  access_key_id minioadmin \
  secret_access_key minioadmin

# Step 3: 停止Harbor并同步数据
docker compose down
echo "Harbor stopped. Starting data migration..."

rclone sync /data/registry/docker/ minio-harbor:harbor-registry/registry/docker/ \
  --progress \
  --transfers 16 \
  --checkers 32 \
  --retries 3 \
  --log-file /tmp/harbor-migration.log

# Step 4: 修改harbor.yml → 执行prepare → 重启
./prepare && docker compose up -d

# Step 5: 验证数据完整性
# 抽样检查：拉取几个不同大小的镜像，验证digest一致
for img in "order-service:v1.0.0" "java-base:11" "node:18-alpine"; do
  docker pull harbor.company.com/test/$img
  echo "Verified: $img"
done
```

```bash
# 方案二：使用Harbor复制功能（零停机迁移）
# Step 1: 在另一个K8s Namespace或独立机器部署New Harbor（使用S3存储）
# Step 2: 在New Harbor上创建与Old Harbor相同的所有项目
# Step 3: 为每个项目创建复制规则（Old → New，全量同步）
for project in $(curl -s -u admin:pass https://old-harbor/api/v2.0/projects | jq -r '.[].name'); do
  curl -X POST -u admin:pass \
    -H "Content-Type: application/json" \
    -d "{
      \"name\": \"migrate-$project\",
      \"dest_registry\": {\"id\": $(NEW_HARBOR_REGISTRY_ID)},
      \"dest_namespace\": \"$project\",
      \"trigger\": {\"type\": \"manual\"},
      \"filters\": [{\"type\": \"resource\", \"value\": \"$project/**\"}],
      \"enabled\": true,
      \"override\": true,
      \"deletion\": false
    }" \
    "https://old-harbor/api/v2.0/replication/policies"
done
# Step 4: 手动触发所有复制规则
# Step 5: 验证完整性后切换DNS → New Harbor
```

### 3.6 可能遇到的坑

**坑1：MinIO配置后docker push报NoSuchKey**

| 项目 | 内容 |
|------|------|
| **症状** | Harbor切换到MinIO后，push镜像时偶尔报错`NoSuchKey: The specified key does not exist`。重试后有时成功，有时失败。 |
| **根因** | MinIO版本太低（< RELEASE.2020-10-18）不支持AWS Signature V4的某些路径风格请求。或者MinIO单机模式下重启后数据路径发生变化。 |
| **解决** | （1）升级MinIO到最新版本。（2）确认`harbor.yml`中`v4auth: true`。（3）如果使用Docker部署，确保MinIO的`/data`目录有持久卷挂载，不会因容器重建而丢失。 |

**坑2：OSS内网Endpoint在容器内无法解析**

| 项目 | 内容 |
|------|------|
| **症状** | 配置了`endpoint: oss-cn-beijing-internal.aliyuncs.com`，但Harbor Core/Registry容器仍报`dial tcp: lookup oss-cn-beijing-internal.aliyuncs.com: no such host`。 |
| **根因** | 容器使用Docker内置DNS（127.0.0.11），可能无法解析阿里云内网域名（需要VPC内的DNS支持）。尤其是在非阿里云ECS上部署的Docker中。 |
| **解决** | （1）`docker exec harbor-core nslookup oss-cn-beijing-internal.aliyuncs.com` 确认解析情况。（2）在docker-compose.yml中为Core和Registry配置`dns: 100.100.2.136`（阿里云VPC DNS地址）。（3）或暂时用公网endpoint测试（会产生流量费）。 |

**坑3：迁移后镜像digest不一致**

| 项目 | 内容 |
|------|------|
| **症状** | 从Local FS迁移到S3后，同一个镜像标签在新旧Harbor上的digest不同。 |
| **根因** | （1）OCI Spec规定digest由内容SHA256决定——如果内容一致，digest必然相同。digest不同说明迁移过程中数据损坏或不完整。（2）也可能是Manifest的JSON序列化格式不同（空格、换行差异），但正常工具不会改变Manifest的字节序列。 |
| **解决** | 在源和目标上分别`docker pull`同一个标签，对比`docker inspect`的输出（特别是RepoDigests）。如果不同，用`rclone check`（checksum模式）对比源和目标存储的每个文件，定位损坏的Blob。 |

**坑4：S3 Redirect模式下docker pull失败（Client无法直连S3）**

| 项目 | 内容 |
|------|------|
| **症状** | 开启`redirect.disable: false`后，`docker pull`报`error fetching blob: connection refused`。Registry日志显示返回了302，但Client似乎在尝试连接S3地址时失败。 |
| **根因** | （1）Docker Client所在网络（如办公网）无法直连S3 Endpoint——Redirect URL是S3的内网域名，仅VPC内可访问。（2）如果部署MinIO，`docker-compose`只暴露了MinIO的9000端口给Harbor内部网络，但外部Client访问不到这个端口。 |
| **解决** | （1）如果Client在VPC外（如办公网），禁用Redirect模式：`redirect.disable: true`——让Registry代理转发。（2）如果MinIO需要外部Client访问，额外暴露9000端口到宿主机：`ports: - "9000:9000"`。 |

## 4 项目总结

### 4.2 选型决策树

```
是否在K8s中运行？
├─ 否 → 是否团队<20人，存储<500GB？

      ├─ 腾讯云TKE → COS（与阿里云OSS类似）
      ├─ GCP GKE → GCS（与GKE深度集成）
      └─ 自建IDC → MinIO分布式集群（Erasure Code 4+4节点）
```

### 4.3 适用场景与不适用场景

**适用场景：**
- **K8s生产环境**：MinIO或云对象存储——避免Local PV引起的Pod-Node绑定问题，支持Registry多副本
- **全球化镜像分发**：S3 + Cross-Region Replication + CloudFront CDN——就近拉取，延迟从跨洲300ms降到同城10ms
- **成本敏感型企业**：对象存储 + 生命周期策略（热→冷→归档）——存储成本可降低70-80%
- **合规要求数据不出IDC**：自建MinIO集群或Ceph RGW——数据100%在内部可控
- **突发大流量场景**：云对象存储的弹性吞吐——无需为峰值预置SSD容量，按实际使用付费

**不适场景：**
- **极致低延迟需求（<1ms）**：只有Local NVMe SSD能满足——HFT（高频交易）等对延迟极度敏感的场景不适用对象存储
- **大量小文件高频随机读写**：对象存储本质上是HTTP REST API，不适合每秒数千次的小文件随机I/O——这类场景本地文件系统更高效（如Harbor自身的Blob GC操作）

### 4.4 注意事项

1. **对象存储的`endpoint`配置不要自带Bucket名**——Harbor会自动在URL中拼接`/<bucket>/<rootdirectory>/...`。如果endpoint写好完整URL，会导致路径重复
2. **分块大小（chunksize）建议5MB-10MB**——过小（<1MB）导致大量HTTP请求（S3 API有QPS限制和请求费用）；过大（>100MB）导致Registry内存占用高且网络中断后重传代价大
3. **MinIO单机模式仅用于测试**——单机MinIO没有Erasure Code保护，硬盘损坏=数据丢失。生产环境至少4节点（EC 2+2），每个节点独立故障域
4. **开启S3 Redirect模式前确认Client网络可直连S3**——如果Client在VPC外（如本地开发机），Redirect会导致pull/push失败。生产环境建议在VPC内做Redirect，VPC外用Proxy模式
5. **迁移数据后用`docker pull`比对digest**——同一个镜像标签迁移前后的digest必须完全一致（SHA256）。不一致意味着数据损坏——立即停止使用目标Harbor，排查`rclone`或复制工具的错误日志

### 4.5 常见故障速查表

| 故障现象 | 根因 | 快速解决 |
|---------|------|---------|
| push报`NoSuchKey` | MinIO版本低或不支持V4签名 | 升级MinIO + 确认`v4auth: true` |
| pull报`BLOB_UNKNOWN` | 存储后端配置错误或Blob损坏 | 检查`storage_service`配置 + 验证Blob存在 |
| OSS内网地址无法解析 | 容器DNS无法解析VPC内网域名 | 配置容器DNS为VPC DNS地址 |
| push/pull速度慢 | 对象存储跨Region或公网访问 | 确认使用内网Endpoint + 就近Region |
| Registry重启后存储数据丢失 | 本地FS的卷未持久化挂载 | 配置docker volume或hostPath持久化 |
| 大镜像push中途失败 | Token超时或chunksize过大 | 增大`token_expiration` + 减小`chunksize` |
