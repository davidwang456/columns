# 第8章：复制规则与镜像同步

## 1 项目背景

**环球商旅**（一家跨境商旅管理 SaaS 平台，服务全球 3200 家企业客户）在 2024-2025 年完成了全球多区域基础设施的布局：北京（主数据中心 + CI/CD 构建中心）、法兰克福（欧洲客户数据合规节点）、新加坡（亚太节点）、圣保罗（拉美灾备节点）。四个区域的 Kubernetes 集群统一从各自本地的 Harbor 实例拉取镜像。然而，随着业务规模从"单区域"向"全球多区域"演进，镜像分发的复杂度呈指数级增长。

**痛点一：跨洲际镜像拉取的"龟速传输"。** 北京 CI 构建完成一个 920MB 的 `booking-engine:v4.2.1` Java 镜像后，法兰克福集群的 45 个 K8s Node 需要从北京 Harbor 拉取该镜像。由于中欧之间的公网链路经过 16 跳路由，平均 RTT 为 285ms，单次拉取耗时 5-12 分钟。更致命的是——法兰克福集群配置了 HPA（弹性伸缩），在业务高峰期（欧洲时间 9:00 AM）突发扩容 30 个 Pod 时，30 个 Node 同时从北京 Harbor 拉取同一个 920MB 镜像——相当于 30 × 920MB = 27.6GB 数据同时在一条 200Mbps 的跨洲专线上传输，理论传输时间 18 分钟，加上 TCP 拥塞控制和并发竞争，实际耗时超过 35 分钟。用户投诉"酒店搜索结果页面加载超过 10 秒"——但根因不是应用性能，而是镜像还没拉下来，Pod 还处于 `ContainerCreating` 状态。

**痛点二：跨区域单点故障的多米诺效应。** 2024 年 11 月，北京数据中心因为核心交换机固件升级意外触发了一个 BGP 路由广播故障，导致北京 Harbor 对外中断 3 小时 47 分钟。在这近 4 小时内，法兰克福、新加坡、圣保罗三个区域的所有 CI 构建全部失败——因为 Dockerfile 中的 `FROM harbor-bj.hq-globaltrip.com/shared-base/java:17` 基础镜像无法拉取。更糟的是，新加坡团队恰好在此期间需要紧急热修复一个 P0 级别的支付 Bug——但因为基础镜像不可用，无法构建任何新镜像。损失：支付服务降级 4 小时，涉及交易金额约 210 万美元。

**痛点三：选择性同步的"人工噩梦"。** 环球商旅在 Harbor 中管理着 4800+ 个镜像仓库，但并非所有镜像都需要全球同步——仅有 65 个核心业务镜像（booking、payment、user-auth 等）需要同步到所有区域，另有 120 个区域专属镜像仅需要在北京和特定区域之间同步（如 GDPR 合规相关的 `data-residency-eu` 仅同步到法兰克福）。运维最初的做法：每次发布时，手动 SSH 到北京服务器，执行 `docker pull` → `docker tag` → `docker push` 到目标区域 Harbor。一张发布 Checklist 上列着 65 个镜像 × 3 个区域 = 195 次手动操作。某次深夜发布中，运维漏掉了 `booking-engine:release-4.2.1` 对新加坡的同步，导致第二天新加坡集群使用旧版本镜像，用户在酒店搜索时遇到了已修复的 Bug——被客户投诉了三次。

**痛点四：合规审计要求——"镜像跨区域备份的证明"。** 环球商旅的金融行业客户（如跨国银行的企业差旅系统）在合同中规定了严格的合规要求：所有生产镜像必须在至少两个地理区域冗余备份，备份延迟不得超过 1 小时，且必须提供可审计的同步日志。CIO 要求运维提供"过去 90 天所有镜像同步的完整审计记录"。运维翻遍了 Harbor 的日志文件，发现——之前手动 `docker push` 的同步方式根本没有结构化日志，无法证明"镜像 A 在北京 push 后是否在 1 小时内到达了法兰克福"。

**痛点五——新增：同步冲突与覆盖风险。** 2025 年 2 月，法兰克福团队为适配欧盟的 GDPR 数据本地化要求，修改了 `user-profile-service` 的 Dockerfile（添加了数据脱敏模块），并以 `release-3.5.0` 标签推送到了法兰克福 Harbor。几乎同时（时间差 3 分钟），北京团队以同一标签推送了另一个版本的 `user-profile-service:release-3.5.0`（没有 GDPR 模块）到北京 Harbor。此时如果开启双向同步，两个 Harbor 之间的复制规则会产生"覆盖竞赛"——最后一个完成同步的版本会覆盖另一个。最终结果是：法兰克福 Harbor 中的版本包含 GDPR 模块，而北京 Harbor 中的版本不包含——但运维直到用户投诉"欧盟用户数据未脱敏"才发现这个问题，此时已过去 5 天。

Harbor 的 **Replication（复制同步）** 机制正是为解决以上所有问题而设计的——支持 Push 模式和 Pull 模式的灵活组合、基于仓库名/标签/标签正则的精细化过滤器、事件驱动 + 定时补偿的双重触发机制、以及完整的同步审计日志。

## 2 项目设计——剧本式交锋对话

**场景：环球商旅运维部 F4 会议室，白板上画满了北京、法兰克福、新加坡、圣保罗的拓扑图。运维小刘正在展示手动同步镜像的 195 步 Checklist，会议室里充满了无奈的笑声。**

**小胖**（大口吃着外卖送来的北京烤鸭卷饼）："大师，我实在不理解——跨机房同步镜像不就是 docker pull + docker tag + docker push 三部曲吗？写个 Shell 脚本，配合 cron 定时跑一下，两小时搞定的事。为啥非要搞什么'复制规则引擎'这么复杂的玩意？这不就是用挖掘机炒菜——大材小用吗？"

**大师**（放下手中的咖啡杯）："小胖，你现在只有 65 个镜像要同步，每个镜像平均 500MB。你算算——你的 Shell 脚本一次全量同步，需要传输多少数据？"

**小胖**（心算）："65 × 500MB = 32.5GB……好像确实有点大。"

**大师**："32.5GB 只是理论值。你的 Shell 脚本不会判断'哪些镜像已经同步过了'——它要么全量同步 65 个镜像，要么一个都不同步。而且——如果传输到第 58 个镜像时网络断开，你的脚本从哪里恢复？从头开始？还是从第 58 个继续？"

"Harbor 的复制引擎不只是'帮你传文件'——它解决了 Shell 脚本无法解决的四个核心问题："

```
┌──────────────────────────────────────────────────────────┐
│        Harbor Replication 引擎 vs Shell 脚本              │
│                                                          │
│ ① 增量同步：只同步新的/变更的制品，不重复传输已有的        │
│ ② 事件驱动：镜像 Push 后自动触发同步（不是定时轮询）       │
│ ③ 状态追踪：每个复制任务有独立状态（Pending→Running→       │
│    Succeed/Failed），可查询、可重试、可审计               │
│ ④ 任务队列：当大量同步任务同时触发时，排队执行而非         │
│    并发抢占带宽                                          │
└──────────────────────────────────────────────────────────┘
```

**技术映射**：Harbor Replication 引擎的核心架构是一个**异步任务队列**。当镜像被推送至源 Harbor 时，Core 组件向 JobService 提交一个 Replication Job。JobService 的任务队列基于 Redis 实现（生产环境可配置为 Redis Cluster），支持 Worker 并发处理。每个 Job 有完整的生命周期（Pending → Scheduled → Running → Succeeded/Failed），且失败后支持自动重试（默认 3 次，指数退避）。

**小白**（在白板上画了一个双箭头图示）："我看了 Harbor 文档，复制有两种模式——Push 和 Pull。但什么时候该用哪种？我怎么跟领导解释这个技术选型？"

**大师**："这个问题问得非常精准。Push 和 Pull 的选择本质上是在回答——**'谁有出网能力，谁来发起连接'**。"

`Push 模式 vs Pull 模式的本质区别：`

```
┌─ Push 模式 ────────────────────────────────────────────┐
│                                                        │
│   北京 Harbor ───(主动推送)──► 法兰克福 Harbor           │
│   (源 Registry)              (目标 Registry)            │
│                                                        │
│   ★ 发起方: 北京                                         │
│   ★ 网络要求: 北京 → 法兰克福 方向可达                     │
│   ★ 触发方式: 事件驱动（push 即触发） + 定时补偿           │
│   ★ 权限要求: 北京持有法兰克福 Harbor 的写入凭证            │
│   ★ 最佳场景: 集中式构建中心 → 多地分发                     │
│   ★ 反面: 如果法兰克福在内网无法被北京访问 → Push 不可用    │
│                                                        │
└────────────────────────────────────────────────────────┘

┌─ Pull 模式 ────────────────────────────────────────────┐
│                                                        │
│   北京 Harbor ───(被拉取)──► 法兰克福 Harbor              │
│   (源 Registry)              (目标 Registry)            │
│                                                        │
│   ★ 发起方: 法兰克福                                     │
│   ★ 网络要求: 法兰克福 → 北京 方向可达                     │
│   ★ 触发方式: 仅定时轮询（Pull 不支持事件驱动！）          │
│   ★ 权限要求: 法兰克福持有北京 Harbor 的读取凭证            │
│   ★ 最佳场景: 源 Harbor 在内网隔离 → 目标主动拉取          │
│   ★ 反面: 延迟大——最快也依赖于轮询间隔                     │
│                                                        │
└────────────────────────────────────────────────────────┘
```

"环球商旅的场景很典型——北京是唯一的 CI/CD 构建中心，生产镜像全部从北京产生。所以**推荐 Push 模式**：北京主动推到各区域。如果某个区域（如圣保罗）网络不稳定，可以为它额外添加一个 Pull 模式的定时补偿规则——两种模式可以组合使用。"

**小胖**："等一下——Pull 模式不支持事件驱动？那法兰克福拉了新镜像，我怎么知道什么时候拉完了？万一我这边急着用呢？"

**大师**："Pull 模式的局限性就在这里——它必须通过定时轮询（Cron）或手动触发。这就像你每隔 1 小时去快递柜看看有没有你的包裹，而不是快递员打电话通知你。所以 Pull 模式常用于以下特殊场景："

1. "**源 Harbor 在内网隔离环境**：比如你的北京 Harbor 在金融专网中，不能主动访问公网上的法兰克福 Harbor。这时候法兰克福用 Pull 模式从北京拉取。"
2. "**多源聚合**：法兰克福需要从北京拉核心业务镜像、从阿里云 ACR 拉基础镜像、从 Docker Hub 拉第三方镜像——Pull 模式可以把多个源的镜像聚合到一个 Harbor 中。"
3. "**作为'补偿同步'**：平时用 Push 事件驱动保证实时性，再配一个 Pull 定时任务作为兜底——如果某次 Push 因网络瞬断失败了，定时 Pull 会补上。"

**小白**："过滤器的语法我一直觉得不够直观——`{order-*, payment-*}` 这个花括号语法到底对标的是正则表达式还是 Glob？很多 CI 同事分不清。"

**大师**："这是 Harbor 的一个设计选择——它使用的是 **doublestar（双星号 Glob）** 语法，不是正则表达式。这个选型有它的道理——Glob 语法的学习曲线比正则低。"

`Harbor 复制过滤器语法速查：`

| 过滤器类型 | 示例 | 含义 | 是否匹配 `order-platform/billing` | 是否匹配 `payment/gateway` |
|-----------|------|------|------|------|
| 全匹配 | `**` | 匹配所有仓库/标签 | ✅ | ✅ |
| 精确项目匹配 | `order-platform/**` | 只匹配该项目下所有仓库 | ✅ | ❌ |
| 前缀匹配 | `order-*/**` | 匹配以 `order-` 开头的项目 | ✅ | ❌ |
| 多值匹配 | `{order-platform, payment-platform}/**` | 匹配花括号内任一项目 | ✅ | ✅（如果 payment-platform 存在） |
| 标签前缀 | `release-*` | 匹配以 `release-` 开头的标签 | — | — |
| 复杂标签 | `{release-*, latest, v1.*}` | 匹配多种标签模式 | — | — |

**小胖**："那 '删除远程资源时本地同步删除' 这个开关——我看起来就像个定时炸弹。万一北京这边因为存储满了执行了 GC，法兰克福的灾备镜像也被同步删了——那灾备还有什么意义？"

**大师**："小胖你抓住了灾备场景中最关键的配置。这个开关有三种策略——你**绝对不能在灾备场景中开启同步删除**："

| 删除策略 | 行为 | 中文直译 | 推荐场景 | 风险 |
|---------|------|---------|---------|------|
| `deletion: false` | 源删除 → 目标保留 | 只增不删 | **灾备/合规备份** | 低（目标会累积，但不会丢数据） |
| `deletion: true` | 源删除 → 目标同步删除 | 同步删除 | 镜像迁移（新 Harbor 替代旧 Harbor） | 高（误删不可逆） |
| `override: true` | 同名标签存在时覆盖 | 覆盖模式 | 镜像统一更新（确保所有区域同一个标签指向同一个 digest） | 中（覆盖会丢失旧版本） |

"**环球商旅的灾备场景必须选 `deletion: false`**——绝对不要因为北京执行了一次 GC，法兰克福的灾备副本也跟着没了。这不是'同步'，这是'同步自杀'。"

**技术映射**：复制规则的 `deletion` 和 `override` 参数在 Harbor Core 的 Replication 引擎中分别控制两类行为：`deletion` 控制源删除时是否在目标执行 Step 1（发送 DELETE 请求到目标 Registry 删除对应标签）；`override` 控制目标已存在同名且同 digest 的标签时是否跳过（skip）还是覆盖（overwrite）。默认值：`deletion: false`, `override: true`。

**小白**："还有多架构镜像的问题——如果北京推送了一个包含 AMD64 + ARM64 的 Manifest List 到 Harbor，复制引擎会同步子 Manifest 吗？还是只同步 Manifest List 本身？"

**大师**："复制引擎在同步一个 Manifest List（也称为 Image Index）时，会**递归处理**——先同步 Manifest List 本身，然后遍历 `manifests` 数组中的每一个子 Manifest，递归同步每个子 Manifest 及其引用的 Blob 层。"

"这意味着如果你在北京 Harbor 推了一个 Manifest List（包含 AMD64 + ARM64 两个架构），复制引擎会：
1. 检查法兰克福 Harbor 是否已有这两个子 Manifest（通过 digest 去重检查）
2. 只推送法兰克福没有的 Blob 层
3. 推送 Manifest List 索引文件

整个过程是对等的——法兰克福 Harbor 最终也会拥有完全一致的 Manifest List 结构。"

"但这里有一个坑：**复制后 Manifest List 的 digest 可能会变**——因为子 Manifest 在 Manifest List JSON 中的排列顺序可能不同。这不影响功能（子 Manifest 的 digest 不变，就意味着镜像内容完全一致），但不建议你在两个 Harbor 之间通过 Manifest List digest 做精确比对——应该比对子 Manifest 的 digest。"

### 3.1 环境准备

| 组件 | 说明 |
|------|------|
| Harbor 实例 A（北京 - 源） | `harbor-bj.hq-globaltrip.com`，v2.12，HTTPS+自签名证书 |
| Harbor 实例 B（法兰克福 - 目标） | `harbor-fra.hq-globaltrip.com`，v2.12，HTTPS+自签名证书 |
| Harbor 实例 C（新加坡 - 目标） | `harbor-sg.hq-globaltrip.com`，v2.12 |
| 网络连通性 | 北京↔法兰克福互访（Push 模式要求北京能访问法兰克福） |
| 项目 | `booking-platform`（已在北京创建，需在法兰克福手动创建同名项目） |
| 测试镜像 | `hello-app`（Docker 镜像，用于验证同步功能） |

```bash
# ================================================================
# 前置条件验证——确保跨区域网络可达
# ================================================================
# 从北京 Harbor 机器验证能访问法兰克福 Harbor 的 API
curl -s -k "https://harbor-fra.hq-globaltrip.com/api/v2.0/health"
```

### 3.2 在源 Harbor 创建目标 Registry 端点

**步骤一：通过 Portal 创建法兰克福 Registry 端点**

> **步骤目标**：在北京 Harbor 中注册法兰克福 Harbor 作为复制目标，建立信任关系和连接通道。

```
Portal 操作路径（北京 Harbor）：
系统管理 → 仓库管理 → 新建目标

填写表单：
  ┌──────────────────────────────────────────────┐
  │ 提供者:          Harbor                        │
  │ 名称:            frankfurt-dr                  │
  │ 描述:            法兰克福区域灾备 Harbor 实例      │
  │ 端点 URL:        https://harbor-fra.hq-globaltrip.com  │
  │ 访问 ID:         admin                         │
  │ 访问密码:        Harbor12345                   │
  │ 验证远程证书:    ☑ (勾选，使用受信证书)           │
  │                 如果使用自签名证书，不勾选       │
  └──────────────────────────────────────────────┘
```

**步骤二：通过 API 批量创建多个 Registry 端点**

> **步骤目标**：使用 Harbor API 以声明式方式批量创建多个区域的 Registry 端点，便于版本化管理。

```bash

# 为法兰克福创建 Registry 端点
# ================================================================

    "description": "法兰克福区域灾备 Harbor（Push 模式目标）",
    "url": "https://harbor-fra.hq-globaltrip.com",
    "credential": {
      "access_key": "admin",
      "access_secret": "Harbor12345",
      "type": "basic"
    },
    "insecure": false,
    "type": "harbor"

  "https://harbor-bj.hq-globaltrip.com/api/v2.0/registries"

# 预期输出：HTTP 201 Created

# ================================================================
# 为新加坡创建 Registry 端点
# ================================================================
curl -X POST -u admin:Harbor12345 \
  -H "Content-Type: application/json" \
  -d '{
    "name": "singapore-dr",
    "description": "新加坡区域灾备 Harbor",
```

### 3.7 可能遇到的坑

**坑1：复制任务一直 Pending（排队中，永不执行）**

| 维度 | 详情 |
|------|------|
| **现象** | 在 Harbor Portal 的"复制"页面看到任务状态为 `Pending`，持续数小时不变化。执行 `curl .../replication/executions` 返回 `status: "InProgress"` 且 `in_progress` 计数一直为 0 |
| **根因** | 三种可能：① JobService 的 Worker 数量不足以处理积压的复制任务队列；② JobService 与 Redis 之间的连接出了问题（任务状态更新的消息队列堵塞）；③ 目标 Registry 端点 Ping 失败，但 Harbor 不会自动跳过——所有后续任务都在排队等待第一个失败任务重试超时 |
| **解决方法** | ① 查看 JobService 的队列深度：`curl -s -u admin:Harbor12345 "https://harbor-bj.hq-globaltrip.com/api/v2.0/jobs/replication/queue" \| jq '.total'`——如果 > 100，在 `harbor.yml` 中增加 `jobservice.max_job_workers: 50` 并重启 JobService；② 查看 JobService 的日志：`docker logs harbor-jobservice --tail 200 \| grep -i "error\|panic"`；③ 清理积压队列：取消所有 Pending 状态的任务（通过 API 或 Portal）并重新触发 |

**坑2：同步后 Manifest List digest 不一致**

| 维度 | 详情 |
|------|------|
| **现象** | 北京 Harbor 和法兰克福 Harbor 中同一个标签 `release-4.2.1` 的 Manifest List digest 不同——北京是 `sha256:abc...`，法兰克福是 `sha256:xyz...`。运维担心"镜像内容被篡改" |
| **根因** | 这不是数据损坏。Manifest List 是一个 JSON 文档，其中 `manifests` 数组是一个列表。在复制过程中，子 Manifest 的引用可能因为 JSON 序列化顺序不同而导致 Manifest List 本身的 digest 变化。但每个子 Manifest 的 digest（即实际镜像内容的 digest）是完全一致的——功能上无任何差异 |
| **解决方法** | ① 验证子 Manifest 的 digest 一致性：`docker manifest inspect harbor-bj.hq-globaltrip.com/booking-platform/booking-engine:release-4.2.1 \| jq '.manifests[].digest' \| sort` 和法兰克福的相同命令对比；② 如果排序后子 Manifest digest 完全一致——功能无差异，可以忽略；③ 如果子 Manifest digest 也不同——说明复制过程中某个步骤出现数据损坏，需要排查网络传输完整性 |

**坑3：远程 Harbor 证书验证失败——Registry 端点 Ping 失败**

| 维度 | 详情 |
|------|------|
| **现象** | 创建 Registry 端点时，点击"测试连接"或 API Ping 返回 `x509: certificate signed by unknown authority`。北京 Harbor（源）无法与法兰克福 Harbor（目标）建立信任的 HTTPS 连接 |
| **根因** | 北京 Harbor 的 Core 容器内的操作系统不信任法兰克福 Harbor 的 TLS 证书。这通常发生在：① 法兰克福 Harbor 使用自签名证书；② 北京 Harbor 的 Core 容器内没有安装法兰克福 Harbor 的 CA 根证书。注意——这不是 Docker 客户端的问题，而是 Harbor Core（Go 程序）的 TLS 验证问题 |
| **解决方法** | 方案 A（生产推荐）：将法兰克福 Harbor 的 CA 根证书复制到北京 Harbor 的 `harbor-core` 容器中：`docker cp frankfurt-ca.crt harbor-core:/etc/ssl/certs/ && docker exec harbor-core update-ca-certificates && docker compose restart harbor-core`；方案 B（仅测试）：在创建 Registry 端点时设置 `"insecure": true`——`curl -X POST ... -d '{"insecure":true,...}'`——这样 Harbor Core 会跳过证书验证 |

**坑4：复制成功但目标缺少部分标签**

| 维度 | 详情 |
|------|------|
| **现象** | 复制规则的日志显示 `status: "Succeed"`，但在法兰克福 Harbor 中检查时，只有部分期望的标签被同步了。例如，北京有 `release-4.2.1`, `release-4.2.0`, `release-4.1.9`，但法兰克福只有 `release-4.2.1` |
| **根因** | ① 标签过滤器太窄——检查过滤器语法是否有误（如 `{release-*}` 少写了花括号变成字符串字面量 `"release-*"` 而非模式匹配）；② 某些标签在过滤器创建之前就已经存在——事件驱动只对**规则创建之后**的新 Push 生效，历史的旧标签不会自动触发事件驱动同步；③ `override: false` 且目标已存在同名标签（可能是之前的同步遗留）——复制引擎跳过了已存在的标签 |
| **解决方法** | ① 手动触发一次补偿同步（定时全量同步会补充历史遗漏）；② 验证过滤器语法：使用 `jq` 测试模式是否匹配；③ 如果目标已存在同名但不同 digest 的标签，且你确定目标端的旧版本可以覆盖，将 `override` 改为 `true` 后再触发一次全量同步 |

## 4 项目总结

### 4.1 Push vs Pull 复制模式全维度对比

| 维度 | Push 模式 | Pull 模式 | 推荐场景 |
|------|----------|----------|---------|
| **数据流向** | 源 → 目标（源主动推送） | 源 ← 目标（目标主动拉取） | — |
| **发起方** | 源 Harbor | 目标 Harbor | — |
| **网络要求** | 源必须能访问目标（出网） | 目标必须能访问源（入方向） | 根据网络策略选择 |
| **触发机制** | **事件驱动**（镜像 Push 即触发）+ 定时补偿 | **仅定时**（Cron 轮询） + 手动触发 | Push 适合实时同步 |
| **同步延迟** | < 10 秒（事件驱动） | 取决于 Cron 间隔（通常 ≥ 1 分钟） | Push 低延迟，Pull 有窗口延迟 |
| **权限要求** | 目标 Harbor 的写入凭证 | 源 Harbor 的读取凭证 | — |
| **安全风险** | 目标暴露写入凭证给源（如果源被入侵） | 源暴露读取凭证给目标 | 双向认证优先 |
| **去重逻辑** | 比较目标已有制品的 digest——无变化则跳过 | 同 Push | 避免重复传输 |
| **多架构支持** | 递归同步 Manifest List + 子 Manifest | 同 Push | 完整支持 |
| **Helm Chart 支持** | 是（Chart 属于 Artifact 子类） | 是 | 同 Push |
| **经典用例** | 集中式构建中心 → 多地分发 | 多源聚合、内网隔离源 | — |

### 4.2 复制规则过滤器组合最佳实践

| 业务场景 | 仓库过滤器 | 标签过滤器 | 触发模式 | 是否覆盖 | 是否同步删除 | 补偿 Cron |
|---------|-----------|-----------|---------|---------|------------|----------|
| **核心业务生产镜像** | `{booking,pymt,auth}-platform/**` | `{release-*, latest}` | 事件驱动 | false（防止覆盖目标的本地修改） | **false**（灾备必选） | `0 2 * * *` |
| **基础/共享镜像全量** | `shared-base/**` | `**`（所有标签） | 事件驱动 | true（确保基础层一致） | false | `0 1 * * *` |
| **CI 快照选择性同步** | `ci-snapshots/**` | `{main-*, tagged-*}` | 事件驱动 | false | true（源 CI 清理了旧构建，目标跟着清理） | — |
| **全量灾备（合规）** | `{booking,pymt,data}-platform/**` | `**`（所有标签） | 定时（每天 2 次） | false | **false**（死也不能删） | `0 2,14 * * *` |
| **跨云迁移（一次性）** | `**`（所有仓库） | `**` | 手动触发（一次） | true | true（源 Harbor 下架后目标成为唯一） | 无 |

### 4.3 常见复制拓扑架构

| 拓扑 | 图示 | 适用规模 | 故障接管能力 | 配置复杂度 | 延迟 |
|------|------|---------|------------|----------|------|
| **中心辐射（Hub-Spoke）** | 北京→法兰克福、北京→新加坡、北京→圣保罗 | 3-10 区域 | 单中心（北京故障→全停） | 低（N-1 条规则） | 取决于北京→各区域 |
| **级联（Cascade）** | 北京→新加坡→悉尼→奥克兰 | 4+ 区域、深度地域 | 中（中间节点故障影响下游） | 中 | 累积（每级 +2-5 min） |
| **全网状（Full Mesh）** | 北京↔法兰克福、北京↔新加坡、法兰克福↔新加坡 | 2-3 区域、高可用 | 高（任一点故障其余仍工作）| 高（M×(M-1) 条规则，M=3 时为 6 条） | 低（最短路径） |
| **混合（Hybrid）** | 北京 Push→新加坡 + 新加坡 Pull←法兰克福 + 北京 Push→圣保罗 | 3+ 区域、混合网络 | 中 | 中 | 取决于组合 |

### 4.4 适用场景

- **跨洲际/跨地域镜像分发**：北京 CI/CD 构建 → Push 到各区域 Harbor → 各区域 K8s 集群从本地 Harbor 拉取，将跨洲拉取延迟从 5-12 分钟降低到 < 1 分钟（本地局域网拉取）
- **混合云/多云镜像统一分发**：自建 IDC Harbor → 阿里云 ACR / 腾讯云 CCR / AWS ECR / GCP Artifact Registry（通过 Harbor 的 Docker Hub/ACR/ECR 等目标适配器）
- **灾备合规（同城/异地）**：主 Harbor 通过 `deletion: false` + 定时全量复制，实现 RPO < 1 小时的灾备能力，满足金融行业等保三级合规要求
- **镜像迁移/升级**：从旧 Harbor v1.x 迁移到新 Harbor v2.x——通过 `override: true` + `deletion: true` 全量复制，逐步引流后下线旧 Harbor
- **边缘计算镜像预热**：中心 Harbor → 边缘节点 Harbor（工厂、门店、基站等）按需同步——仅同步边缘业务需要的 `release-*` 标签

### 4.5 不适用场景

- **实时性要求 < 5 秒的金融交易场景**：事件驱动同步也有 2-10 秒的调度延迟（Core 提交 Job → JobService 调度 → 网络传输），不适合股票高频交易等需要毫秒级一致性的场景
- **超大镜像（> 5GB）+ 高频推送（> 10 次/天）**：每次事件驱动同步都会触发一次完整的跨洲传输。建议调整为定时批量同步（如每天凌晨 3:00 全量同步一次），避免白天业务高峰期占用跨洲带宽
- **纯内网隔离且无公网访问的源 Harbor**：如果北京 Harbor 完全在内网隔离环境，且不能主动连接法兰克福，Push 模式不可用——必须切换到 Pull 模式（由有出网能力的法兰克福主动拉取，或者通过中间 DMZ 区的跳板 Harbor 做两次复制）

### 4.6 注意事项

1. **目标项目必须提前创建**：复制规则不会自动在目标 Harbor 创建项目（这是设计如此——避免自动化脚本在目标 Harbor 中随意创建资源）。如果目标缺少同名项目，复制任务会失败并在 JobService 日志中记录 `project not found`。建议使用 IaC 工具（Terraform Harbor Provider）在创建复制规则前自动创建所有目标项目
2. **同步方向（Push/Pull）一旦创建不可修改**：要改变方向必须删除规则重建——这意味着旧的执行日志和统计会被清空。如果需要在审计表中保留历史，在删除前先导出执行记录
3. **目标配额限制**：如果目标项目的存储配额满了，复制任务会失败。建议为目标项目配置比源项目大 20% 的配额（因为目标保留了更多历史版本，镜像总大小会更大）
4. **跨 Harbor 版本兼容性**：强烈建议源 Harbor 和目标 Harbor 使用相同的主版本号（如都是 v2.10+）。跨大版本复制（如 v2.1 → v2.12）虽然 API 层面向后兼容，但在 Manifest List 同步、Helm Chart 同步等高级特性上可能存在不兼容
5. **JobService Worker 数量的调优**：`max_job_workers` 的默认值为 10——这对 3 个复制目标 + 1 个事件触发源的场景足够。但如果有 30 个区域、每个区域有 10+ 条规则，需要将 Worker 数增加到 50-100。Worker 数增加会线性增加内存消耗——`harbor.yml` 中 `jobservice.max_job_workers` 每增加 10 个 Worker，约增加 200MB 内存消耗

### 4.7 常见踩坑经验（真实生产故障案例）

| 故障案例 | 故障时间线 | 根因分析 | 解决方案 | 避免措施 |
|---------|----------|---------|---------|---------|
| **BGP 路由故障导致全区域 CI 中断** | 11:03 北京核心交换机固件升级 → BGP 路由广播失败 → 11:07 所有区域 CI 失败（基础镜像 `FROM harbor-bj.../java:17` 拉取不到）→ 11:10 运维得知 → 11:30 手动回退交换机固件 → 11:45 BGP 恢复 → 持续 42 分钟 | 所有区域的 CI Dockerfile 中 `FROM` 指令硬编码了北京 Harbor 地址——没有在本地 Harbor 中缓存基础镜像 | 为基础镜像项目（shared-base）创建高频率复制规则（每天 2 次全量 + 每次基础镜像变更时事件驱动），确保所有区域 Harbor 都有基础镜像的副本 | ① 所有 CI Dockerfile 中的 `FROM` 应指向本地 Harbor（如法兰克福 CI 使用 `FROM harbor-fra.../java:17`）；② 基础镜像变更后自动触发全区域同步 |
| **手动同步遗漏导致生产使用旧版本镜像** | 深夜发布 checklist 195 项 → 运维遗漏了 `booking-engine:release-4.2.1` 对新加坡的同步 → 第二天新加坡集群使用 `release-4.2.0`（旧版本）→ 用户投诉已修复 Bug 重现 → 排查 2 小时后发现同步遗漏 | 手动同步清单冗长（195 个镜像 × 3 区域），人类执行出错概率 ~1-3%（符合认知科学中的自动化缺失导致的疏忽错误） | 立即创建事件驱动复制规则——确保每次 push 自动同步，不再依赖手动清单 | 复制规则创建走 IaC（Terraform Harbor Provider + PR Review），发布 checklist 中删除手动同步步骤 |
| **双向同步的"覆盖竞赛"** | 法兰克福团队推送 `release-3.5.0`（含 GDPR 模块）到法兰克福 Harbor → 3 分钟后北京团队推送同名标签（不含 GDPR 模块）到北京 Harbor → 双向同步规则彼此覆盖 → 最终结果不确定（取决于哪一次同步任务最后完成）→ 直到 5 天后用户投诉才发现 | 两个团队并行工作，同一标签在两地同时更新——双向 Push 同步产生竞态条件（race condition） | ① 严格规定"每个标签只有一个推送源头"（如生产标签只能在北京 CI 中心推送，区域团队使用不同标签前缀如 `release-3.5.0-fra-gdpr`）；② 启用标签不可变性规则防止覆盖 | ① 在推送规范中写入"单一源原则"（Single Source of Truth for Tags）；② CI Pipeline 中加入标签命名校验——非合规标签拒绝推送 |

### 4.8 思考题

1. **环球商旅的运维团队收到 CTO 的要求——"北京到法兰克福的复制同步 RPO 必须从当前的 < 1 小时提升到 < 5 分钟，且同步的完整性必须可审计。"设计一个满足该要求的方案：① 当前是定时全量同步（Cron 每天 2 次），如何提升到 < 5 分钟？② 如何为每次同步生成可审计的日志（包含同步了什么镜像、从哪个 digest 到哪个 digest、传输了多少字节、耗时多少、是增量还是全量）？③ 审计日志的保留周期应为多久？如何防止日志被篡改（WORM）？**

2. **假设环球商旅在 2026 年将北京数据中心迁至新的云平台（AWS 北京区域），需要在一个周末窗口内将现有 Harbor（3.5TB 镜像数据）从自建 IDC 完全迁移到 AWS。设计一个零停机迁移方案：① 如何在迁移期间保证业务正常运行（所有区域 K8s 集群正常拉取镜像）？② 如何验证迁移后 AWS 上的 Harbor 与源 Harbor 数据完全一致？③ 迁移完成后，如何无损地将所有区域的复制规则从旧的北京 Harbor 切换到新的 AWS Harbor？**
