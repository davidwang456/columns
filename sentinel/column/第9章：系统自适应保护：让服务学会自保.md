# 第9章：系统自适应保护：让服务学会自保

## 1 项目背景

第 8 章的热点参数限流解决了"单个参数被打爆"的问题。但接下来在预发布环境的压测中，运维团队发现了一个更加棘手的问题：

压测团队将订单服务的入口 QPS 从 300 逐步加压到 800。在 QPS=800 时，服务的响应时间仍然稳定在 100ms 左右，一切正常。但当 QPS 加到 900 时，服务突然崩溃——CPU 瞬间飙到 95%，GC 频繁到每秒 3 次 Full GC，平均 RT 从 100ms 暴涨到 5000ms，最终所有请求超时。

复盘发现，单台服务器的 CPU 极限大约在 QPS=850。在此之前，限流规则配置的阈值是 1000——开发团队拍脑袋定的数字，根本没有对标服务器的实际容量。如果没有 Sentinel 系统保护，服务会在达到 CPU 极限后进入死亡螺旋（请求慢 → 线程堆积 → CPU 竞争更激烈 → 更慢）。

更隐蔽的问题是：Kubernetes 中 Pod 的 CPU Limit 是 2 核，但开发配的流控阈值为 800 QPS——这是按照 4 核物理机测出来的数据。容器限制下服务实际只能跑 400 QPS，但 Sentinel 的 QPS 限流根本感知不到容器的真实资源限制。

Sentinel 的**系统自适应保护（SystemRule）**正是解决这类问题的利器：它不依赖人工拍脑袋的 QPS 阈值，而是直接监控服务器的 Load、CPU 使用率、平均 RT、入口 QPS 和并发线程数等系统指标，在系统陷入过载之前自动触发保护。

## 2 项目设计

**小胖**（看到压测报告跳起来）："什么鬼！QPS=800 的时候还正常，900 就直接崩？这也太陡了吧！"

**大师**："这就是系统的'临界点'。就像一个水库，水位 8 米时好好的，8.5 米还能撑，9 米直接决堤。你不会等到了 8.9 米才开始泄洪，对吧？"

**小白**："所以 Sentinel 的系统保护就是在业务指标（QPS/RT）之外，再加一层系统级别的保护？"

**大师**："对。系统规则和普通流控最大的区别是：**它只对入口流量生效**。什么意思呢？系统保护看的是'有没有新的请求要进来'，但已经进来的请求（内部调用、子资源调用）不受系统规则限制。"

**小胖**："为什么只限制入口？内部调用超了也会拖垮系统啊？"

**大师**："因为入口是你能控制的唯一关卡。一个请求进来后可能在内部调 10 个服务，你没法在每个内部调用点都配系统保护。系统规则是'当整台机器已经很忙了，就先别放新请求进来了'。"

**小白**："那 Sentinel 提供了哪些系统指标？我在文档里看到了 Load、CPU、RT、入口 QPS、并发线程数。"

**大师**："五种指标，我一个个说："

- **Load**：系统负载（仅 Linux/Unix）。`SystemRule.setHighestSystemLoad(3.0)` 表示 Load 超过 3.0 时触发保护。注意：Load 是 1/5/15 分钟平均值，有滞后性。
- **CPU 使用率**：`setHighestCpuUsage(0.8)` 表示 CPU 使用率超过 80% 时触发。这个指标比较直接，推荐优先使用。
- **平均 RT**：`setAvgRt(100)` 表示入口请求的平均响应时间超过 100ms 时触发。适合 IO 密集但 CPU 不高的场景。
- **入口 QPS**：`setQps(500)` 表示入口 QPS 超过 500 时触发。这是"最大承载能力"的硬限制。
- **并发线程数**：`setMaxThread(100)` 表示并发入口线程数超过 100 时触发。

**小胖**："这么多指标，我该用哪个？总不能全配上吧？"

**大师**："建议是：**CPU + Load 二选一（看操作系统），RT + 并发线程数做补充**。入口 QPS 是兜底的。配置太多指标容易误触发。"

**小白**："那系统保护和 Kubernetes HPA 怎么配合？HPA 看到 CPU 高了会自动扩容，Sentinel 看到 CPU 高了会拒绝新请求——两者不会冲突吗？"

**大师**："这是个非常好的问题。正确的关系是：**Sentinel 系统保护是'保命'，HPA 扩容是'健身'。** Sentinel 应该在 HPA 来不及扩容时兜底。HPA 缩容阈值设为 50%，Sentinel 系统保护阈值设为 80%——让 HPA 在系统压力中等时就开始扩容，Sentinel 只在压力濒临崩溃时才出手。"

**小胖**："大师，我在 Windows 开发机上配置了 `highestSystemLoad=3.0`，死活不生效——是不是我写错了？"

**大师**："不是你写错了，是 Windows 不支持。`OperatingSystemMXBean.getSystemLoadAverage()` 在 Windows 上返回 -1，Sentinel 拿到负数会自动跳过 Load 检查。开发环境用 CPU 使用率替代。"

**小白**："那系统保护有没有'冷却时间'？比如 CPU 从 90% 降到 50%，限流是立即解除还是等一会儿？"

**大师**："立即解除。系统保护是实时判断的——当前 CPU 不超过阈值就放行。这也是它的一个潜在问题：如果流量有个脉冲，CPU 从 90% 骤降到 30%，所有排队的请求瞬间涌入，CPU 又飙到 90%——形成振荡。解决方案是配合流控规则做'软着陆'——系统保护解除后，流控规则仍然生效，防止二次过载。"

**小胖**（突然想到）："那如果我的服务混部——同一台机器上跑了订单服务和库存服务，系统保护是分别判断还是全局判断？"

**大师**："全局判断——系统规则看的是整台机器的 CPU/Load，不区分是哪个服务的流量。所以混部场景下，一个服务的流量可能触发系统保护，导致另一个完全正常的服务也被拒绝入口请求。如果你需要按服务隔离系统保护，建议每个服务独立部署或用容器做资源隔离。"

**技术映射**：
- 系统保护的 unique 特性：**只对入口流量（EntryType.IN）生效**。这是因为内部调用（EntryType.OUT 或默认）无法判断"系统整体忙不忙"，只有入口才能做系统级判断。
- SystemRule 的判断逻辑在 `SystemSlot` 中执行。它会在 `entry()` 调用时检查当前系统指标，如果任意一个指标超出阈值则抛出 `SystemBlockException`。
- 系统指标的采集是有间隔的：CPU 和 Load 通过 `OperatingSystemMXBean` 获取（Java 9+），默认每 1 秒更新一次。

## 3 项目实战

### 3.1 环境准备

无需额外依赖，系统规则是 sentinel-core 内置能力。准备一个简单的压测环境。

### 3.2 分步实现

**步骤一：配置系统保护规则**

```java
@Component
public class SystemRuleConfig {

    @PostConstruct
    public void initSystemRules() {
        List<SystemRule> rules = new ArrayList<>();

        SystemRule cpuRule = new SystemRule();
        // CPU 使用率超过 80% 时触发系统保护
        cpuRule.setHighestCpuUsage(0.8);

        SystemRule rtRule = new SystemRule();
        // 入口平均 RT 超过 100ms 时触发
        rtRule.setAvgRt(100);

        SystemRule threadRule = new SystemRule();
        // 入口并发线程数超过 200 时触发
        rtRule.setMaxThread(200);

        SystemRule qpsRule = new SystemRule();
        // 入口 QPS 超过 500 时触发（兜底）
        qpsRule.setQps(500);

        rules.add(cpuRule);
        rules.add(rtRule);
        rules.add(threadRule);
        rules.add(qpsRule);

        SystemRuleManager.loadRules(rules);
    }
}
```

**步骤二：验证只有入口流量被限制**

创建入口 Controller 和内部 Service：

```java
@RestController
public class SystemProtectController {

    @Autowired
    private ComputeService computeService;

    // 入口方法 —— 受系统保护
    @GetMapping("/system/entry")
    public String entryMethod() throws InterruptedException {
        // 内部调用了 computeService
        return computeService.doCompute();
    }
}

@Service
public class ComputeService {

    // 内部方法 —— 不受系统保护！
    @SentinelResource(value = "internalCompute", blockHandler = "computeBlock")
    public String doCompute() throws InterruptedException {
        // 模拟 CPU 密集型计算
        long start = System.currentTimeMillis();
        while (System.currentTimeMillis() - start < 200) {
            Math.pow(Math.random(), Math.random());  // 纯 CPU 消耗
        }
        return "计算完成";
    }

    public String computeBlock(BlockException e) {
        return "内部计算被限流";
    }
}
```

**验证**：高并发压测 `/system/entry`，当 CPU 超过 80% 时，`/system/entry` 入口请求会被 `SystemBlockException` 拦截，但内部 `doCompute()` 调用不受影响（内部调用不会触发 SystemSlot 判断）。

**步骤三：模拟 CPU 过载触发系统保护**

编写一个可以精确控制 CPU 负载的测试接口：

```java
@GetMapping("/system/cpu-burn")
public String cpuBurn(@RequestParam(defaultValue = "70") int targetCpuPercent)
        throws InterruptedException {
    
    int cores = Runtime.getRuntime().availableProcessors();
    int targetThreads = (int) Math.ceil(cores * targetCpuPercent / 100.0);
    
    ExecutorService executor = Executors.newFixedThreadPool(targetThreads);
    for (int i = 0; i < targetThreads; i++) {
        executor.submit(() -> {
            while (!Thread.currentThread().isInterrupted()) {
                // 死循环消耗 CPU
                Math.sin(Math.random());
            }
        });
    }
    
    // 维持 30 秒
    Thread.sleep(30000);
    executor.shutdownNow();
    
    return "CPU 烧鸡完成，目标 " + targetCpuPercent + "%, 使用 " + targetThreads + " 个线程";
}
```

测试步骤：

1. 先确认系统保护规则生效（CPU > 80% 触发）。
2. 用一个终端启动 CPU 负载：`curl http://localhost:8090/system/cpu-burn?targetCpuPercent=85`。
3. 另一个终端快速请求入口接口：`ab -n 100 -c 10 http://localhost:8090/system/entry`。
4. 观察结果：大量请求被 `SystemBlockException` 拦截，返回限流提示。

**步骤五：系统保护与 Kubernetes 配合**

在 Kubernetes 环境中，Sentinel 无法直接获取宿主机的 CPU 信息（容器内通过 cgroup 限制）。从 Java 9+ 开始，`OperatingSystemMXBean.getSystemCpuLoad()` 会返回容器级别的 CPU 使用率，可以直接用于 Sentinel 系统保护。

对于 Java 8，需要添加启动参数：

```bash
java -XX:+UseContainerSupport -XX:ActiveProcessorCount=2 -jar app.jar
```

**踩坑记录**：

1. **系统规则对内部调用无效**：这是 Sentinel 的设计特性，不是 Bug。如果需要对内部调用也做保护，需要用普通流控规则。
2. **Load 指标在 Windows 上不可用**：Windows 的 `OperatingSystemMXBean` 不支持获取系统 Load，`highestSystemLoad` 配置在 Windows 下不生效。建议 Windows 开发环境用 CPU 使用率替代。
3. **系统保护阈值不宜设太低**：CPU 使用率阈值 50% 会导致 HPA 扩容和 Sentinel 限流交替生效，系统在"扩缩容-限流-扩容"的循环中震荡。建议设为 75-85%。
4. **多个 SystemRule 是 OR 关系**：任一指标触发都会导致保护，不要把几个临界值同时设得刚好，可能互相影响。

**步骤六：系统保护与普通流控的协作模式**

系统保护不应该独立工作——它应该作为"最后防线"。推荐的三层保护模型：

```java
@Component
public class LayeredProtectionConfig {

    @PostConstruct
    public void init() {
        // 第一层：普通流控（预防性）
        // QPS=400 —— 在系统进入高负载前就开始限流
        List<FlowRule> flowRules = Collections.singletonList(
            new FlowRule("createOrder")
                .setGrade(RuleConstant.FLOW_GRADE_QPS)
                .setCount(400)
        );
        FlowRuleManager.loadRules(flowRules);

        // 第二层：系统保护（兜底性）
        // CPU > 85% —— 流控阈值设低了？还是突然流量脉冲？
        List<SystemRule> systemRules = Collections.singletonList(
            new SystemRule()
                .setHighestCpuUsage(0.85)
                .setAvgRt(200)
        );
        SystemRuleManager.loadRules(systemRules);

        // 第三层：服务降级（极端情况）
        // 如果系统保护持续触发超过 30 秒，触发告警通知运维
    }
}
```

**步骤七：系统保护触发后的自动恢复验证**

编写一个自动化测试脚本验证保护→恢复的完整生命周期：

```bash
#!/bin/bash
# system-protect-lifecycle-test.sh
echo "=== 阶段 1: 正常流量 ==="
ab -n 50 -c 5 http://localhost:8090/system/entry 2>&1 | grep "Failed requests"

echo "=== 阶段 2: 触发 CPU 负载 ==="
curl -s "http://localhost:8090/system/cpu-burn?targetCpuPercent=90" &
sleep 3

echo "=== 阶段 3: 验证系统保护生效 ==="
BLOCKED=$(ab -n 50 -c 10 http://localhost:8090/system/entry 2>&1 | grep -c "non-2xx")
echo "被系统保护拦截的请求数: $BLOCKED"

echo "=== 阶段 4: 恢复验证（等待 CPU 降低）==="
sleep 35  # 等待 CPU burn 结束
PASSED=$(ab -n 20 -c 5 http://localhost:8090/system/entry 2>&1 | grep -c "2xx")
echo "恢复后通过的请求: $PASSED / 20"
```

## 4 项目总结

### 4.1 优点与缺点

| 维度 | Sentinel 系统保护 | Kubernetes HPA | 传统阈值告警 + 人工介入 |
|------|-----------------|---------------|----------------------|
| 响应速度 | 毫秒级（本地判断） | 分钟级（指标采集 + 扩缩容） | 分钟-小时级 |
| 误杀风险 | 中（阈值判断） | 低 | 低 |
| 精细度 | 低（只看入口） | 中（副本级别） | 高（人工判断） |
| 自动化程度 | 高 | 高 | 低 |
| 对业务影响 | 直接拒绝请求 | 扩容期间不中断 | 严重时需停服 |

### 4.2 适用场景

- **物理机/固定资源部署**：单机容量明确，直接用系统保护兜底
- **容器弹性环境**：作为 HPA 来不及扩容时的最后防线
- **突发流量保护**：流量突然涨 5-10 倍，限流规则阈值来不及调整
- **GC 抖动保护**：Full GC 导致 CPU 飙升时自动限流，防止雪崩
- **不适用**：纯 IO 等待型服务（CPU 不高但 RT 高，用 RT 指标即可）

### 4.3 注意事项

1. 系统保护只对入口流量（EntryType.IN）生效。Web 场景下 Controller 层的请求是入口流量，Feign/Dubbo 的内部调用不是。
2. CPU 使用率指标在容器化场景中可能不准确，需要验证 Java 版本和容器运行时对 cgroup 的支持。
3. 不要将系统保护阈值和 HPA 缩容阈值设得太近，建议间隔 20% 以上。
4. 系统规则的统计窗口是全局的，不是按资源的——因为它保护的是"整台机器"。

### 4.4 五种系统指标选用决策表

| 指标 | 适用场景 | 不适用场景 | 推荐阈值 |
|------|---------|-----------|---------|
| CPU 使用率 | CPU 密集型服务（计算、编解码） | 纯 IO 等待型服务 | 0.75-0.85 |
| 系统 Load | Linux 物理机/虚拟机 | Windows 和部分容器环境 | CPU 核数 × 1.5 |
| 平均 RT | IO 密集型服务（数据库查询） | CPU 密集型（不反映真实压力） | P99 RT × 2 |
| 入口 QPS | 所有场景的兜底 | 弹性伸缩（阈值会随副本变化） | 拐点 QPS × 0.9 |
| 并发线程数 | 同步阻塞模型（Tomcat） | 异步非阻塞模型（WebFlux/Netty） | Tomcat max-threads × 0.8 |

### 4.5 系统保护 vs 普通流控的差异要点

| 维度 | 系统保护 (SystemRule) | 普通流控 (FlowRule) |
|------|---------------------|-------------------|
| 作用范围 | 全局（入口流量） | 按资源（任意流量） |
| 判断依据 | 系统指标（CPU/Load/RT） | 资源指标（每个资源的 QPS/线程） |
| 触发速度 | 毫秒级（本地指标） | 毫秒级（本地滑动窗口） |
| 配置难度 | 低（1-2 个指标） | 中（每个资源单独配置） |
| 运维介入 | 几乎不需要 | 需要定期调整阈值 |
| 组合建议 | 兜底保护 | 主要保护手段 |

### 4.4 常见踩坑经验

| 故障现象 | 根因 | 解决方案 |
|---------|------|---------|
| 配置了系统保护但从不触发 | 阈值设太高（CPU 99%） | 降低到 75-85%，模拟高负载验证 |
| 系统保护触发后所有请求全部被拒绝 | 入口流量短时间内没有丝毫下降 | 配合 QPS 限流做"软着陆" |
| 系统保护误触发 | 多个规则 OR 关系，RT 波动导致误判 | 只用 1-2 个最相关的系统指标 |
| 容器中 CPU 使用率不准 | Java 8 不支持 cgroup v2 | 升级 JDK 或添加容器感知参数 |

### 4.5 思考题

1. 如果一台服务器上部署了多个服务，Sentinel 的系统保护是针对"整个 JVM"还是"单个服务"？如果多个服务同时触发系统保护，会不会互相影响？
2. 系统保护配置了 `highestCpuUsage=0.8` 和 `highestSystemLoad=3.0`，如果 CPU 使用率是 50% 但 Load 是 5.0（CPU 在等 IO），系统保护会触发吗？这是正确的行为吗？

### 4.6 推广计划

- **开发团队**：在服务上线前必须做一次"过载压测"，找出实际的系统临界点，并据此配置系统保护阈值。
- **测试团队**：设计系统过载测试用例——模拟 CPU 满载、线程池耗尽、GC 频繁等场景，验证系统保护是否生效。
- **运维/SRE 团队**：将系统保护触发事件接入告警（如 Prometheus 指标），配置"系统保护触发超过 1 分钟"的紧急告警。
