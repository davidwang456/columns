# 第29章：生产事故复盘：限流、熔断与降级的误用

## 1 项目背景

有了规则治理平台，团队的 Sentinel 规则管理终于有了章法。但规则管理"规范化"不等于"不出错"——历史经验表明，50% 的线上故障不是没有规则，而是**规则配错了**。

本章将通过 5 个真实化的生产事故案例（脱敏后），复盘 Sentinel 规则误用的典型场景。每个事故还原完整的时间线、根因分析、修复方案和预防清单。这些案例不是虚构的——它们来自社区 issue、企业实战和公开的故障复盘报告。

## 2 项目设计

**大师**："学 Sentinel 最好的方式之一，就是看别人怎么把它用坏的。今天我跟你们复盘 5 个真实的 Sentinel 误用事故。"

**小胖**："事故？我最爱听事故了！别人的血泪教训，我们的经验积累！"

**大师**："听完你会对 Sentinel 的边界条件有更深刻的理解。就拿事故 1 来说——一个运维手误改了阈值，业务损失了 90% 的交易量，将近 10 分钟。这个事故告诉我们：Sentinel 是一把双刃剑，配对了保护系统，配错了就是自己给自己'拔网线'。"

**小白**："那有没有办法让 Sentinel 自动检测这种'阈值异常变更'？比如 createOrder 本来就是 500 QPS，突然改成 50——这个变化太大了，系统应该能识别出来。"

**大师**："好思路。这其实就是规则治理平台应该实现的**变更合理性校验**。规则变更时计算历史阈值的变化幅度，超过 50% 的变动需要二次确认。还有更进一步的——对比当前实际 QPS：如果实际 QPS 是 450，你把阈值改成 50，系统应该弹窗警告'阈值远低于实际流量，确认后将拦截 90% 请求'。"

**小胖**："事故 2 里熔断窗口 300 秒确实太夸张了。但说实话，我之前也不知道应该设多少秒，就是随便抄的。"

**大师**："熔断窗口的设置有一个黄金规则：**timeWindow ≥ 你期望的下游恢复时间**。如果你知道 Redis 故障后运维能在 30 秒内切到从库，timeWindow 就设 30 秒。同时结合半开状态：每 10 秒放一个请求探测下游是否恢复，恢复了就立即关闭熔断。"

**小白**："事故 3 的 fallback 吞异常问题，我遇到过一个类似的。当时 fallback 返回 null，上游 NPE，排查了半天才发现是 fallback 把真正的异常吃掉了。后来我总结了一个原则：**fallback 只做兜底，不做业务逻辑；真正的业务异常应该向上抛，让全局异常处理器处理**。"

**大师**："对。区分 blockHandler 和 fallback 是老生常谈，但真正理解的人不多。记住：`blockHandler` 处理 Sentinel 阻断（BlockException），`fallback` 处理业务异常（Throwable）。很多人的 fallback 写成 `catch (Throwable e) { return "error"; }`——这等于把所有异常都吞了，包括 NullPointerException。"

**小胖**："事故 5 的热点参数问题，50 万 SKU 填满 LRU 缓存——这个坑我踩过类似的。我们当时是用户 ID 维度，2000 万用户，Sentinel 内存直接炸了。"

**大师**："热点参数限流有一条铁律：**只为 TopN 热点参数配置规则**。怎么知道 TopN？从监控中分析。绝大多数接口的流量集中在极少数参数值上——比如电商平台 80% 的流量集中在 Top 20 个 SKU 上。你只需要为这 20 个热点 SKU 配置规则，其余用默认阈值覆盖。"

**小胖**："事故 1 里运维手误改了规则——但我们小团队没有治理平台，有没有更简单的防护措施？"

**大师**："最简单的方案：在 Nacos 配置中设定一个'最低安全阈值'。比如 createOrder 的 QPS 最低不能低于 100——低于这个值要么拒绝修改，要么弹出二次确认。在 Nacos 配置内容里加注释 `# SAFETY_MIN=100`，配合一个前置检查脚本。"

**小白**："那事故 3 的 fallback 吃异常，除了'不要吞异常'之外，还有什么防御方案？"

**大师**："三个防御层。第一层：fallback 中永远不要返回 `null`，至少返回一个有意义的兜底响应。第二层：在 fallback 的 `create()` 方法中，记录异常堆栈到日志系统（不是 `e.printStackTrace()`），方便事后审计。第三层：关键操作（扣库存、扣钱）需要独立的补偿机制——即使 fallback 漏了异常，补偿服务也能发现数据不一致并修复。"

**小胖**："大师，这 6 大事故有没有一个统一的'安全基线'？我能不能给我们团队一条一条检查？"

**大师**："有。我把 6 个事故总结成 8 条 Sentinel 安全配置基线，你贴在 wiki 上：
1. 所有生产规则发布须经审批流（事故 1）
2. 熔断窗口 timeWindow ≤ 30 秒（事故 2）
3. fallback 中永不禁用异常日志（事故 3）
4. 网关限流 ≥ 服务层限流 × 1.2（事故 4）
5. 热点参数缓存上限 ≤ 10000（事故 5）
6. 系统保护规则必须在所有服务启用（事故 6 兜底）
7. 核心链路须配置关联流控/链路流控做隔离
8. 每个服务至少保留最近 10 个版本的规则快照用于回滚"

**小白**："第 6 条特别关键——很多人把系统保护当可选项，其实它应该是最后一道防线，不论有没有限流规则都应该开着。"

## 3 项目实战：五大事故

### 事故 1：阈值过低导致核心链路被误杀

**时间线**：
- 14:00：运维在上线一个新版本时，同步更新了 Nacos 中的流控规则，将 `createOrder` 的 QPS 从 500 改为 50——因为运维记错了服务，以为这个接口是内部管理接口
- 14:02：订单服务对所有超过 50 QPS 的 `createOrder` 请求返回 `FlowException`
- 14:05：监控显示订单成交量骤降 90%，客服收到大量投诉
- 14:15：运维通过查看监控还原到规则变更记录，紧急回滚规则
- 14:16：服务恢复正常

**根因**：人工操作失误 + 缺少变更审批 + 没有变更窗口确认机制

**修复**：
1. 所有生产环境 Sentinel 规则变更必须走审批流（参考第 28 章）
2. 规则变更后 5 分钟内的核心指标（订单量/支付量）与历史同期对比，异常自动告警
3. 规则治理平台增加"一键回滚"功能

### 事故 2：熔断窗口过长导致服务不可用

**时间线**：
- 某次压测中，库存服务因为 Redis 抖动出现 2 秒延迟。订单服务的熔断规则（`timeWindow=300`/5 分钟）触发
- 库存服务在 10 秒后恢复，但订单服务在接下来的 4 分 50 秒内对所有 Feign 调用返回降级结果
- 300 秒中订单服务积累了 5 万个降级订单，全部进入异步处理队列，导致 MQ 积压

**根因**：熔断窗口 `timeWindow=300` 过长。下游服务故障持续 10 秒，但熔断持续了 300 秒

**修复**：将熔断窗口从 300 秒改为 15 秒，增加半开探测频率

### 事故 3：Fallback 吞掉业务异常造成数据不一致

**时间线**：
- 订单服务的 `createOrder` 有一个 fallback 方法：当任何异常时返回"下单失败，请重试"
- 用户下单 → 库存扣减成功 → 创建订单记录时数据库写入异常 → fallback 返回"下单失败"
- 用户重试 → 库存再次扣减 → 但第一次的库存实际上已经被扣了，导致库存被重复扣减

**根因**：fallback 掩盖了业务异常，且没有做幂等或补偿

**修复**：区分 blockHandler（处理 Sentinel 阻断）和 fallback（处理业务异常）。fallback 中不要简单返回"请重试"，应该记录异常详情并通过异步补偿机制修复数据

### 事故 4：网关层和服务层重复限流导致通过量异常

**时间线**：
- 网关层配了 QPS=200，服务层配了 QPS=100
- 实际打到服务的流量是 150 QPS → 网关放行了 150，但服务层只收了 100，50 个请求在服务层被拦截
- 用户看到的错误信息是网关返回的 429，但运维在服务层看到的 BlockException 却不明白为什么

**根因**：两层限流叠加，阈值不匹配

**修复**：网关和服务层的限流应有明确分工——网关管"总量"，服务管"自身保护"。建议网关阈值 = 服务层阈值 × 1.2-1.5

### 事故 5：热点参数维度过多导致内存和 CPU 飙升

**时间线**：
- 开发为商品详情接口配置了热点参数限流，参数维度是 skuId。由于该服务有 50 万个 SKU，Sentinel 内部 LRU 缓存被迅速填满
- 内存从 512MB 飙升到 2.5GB，频繁 Full GC
- 服务 P99 RT 从 50ms 飙升到 3000ms

**根因**：热点参数限流不适用于高基数（High Cardinality）场景

**修复**：为热点参数限流配置合理缓存上限（默认 4000），并通过日志分析确认"真正的热点 TopN"只有 20 个 SKU

### 事故 6：规则生效延迟导致流控滞后

**时间线**：
- 某次大促开始后，流量 3 秒内从 500 QPS 飙升至 2000 QPS
- 运维在 Nacos 中更新了流控阈值（从 300 改为 600），但客户端加载规则有 5 秒延迟
- 在这 5 秒内，服务已经被 2000 QPS 冲垮，所有请求超时
- 等 Nacos 规则推送到客户端时，服务已经 OOM 重启了

**根因**：Nacos 客户端默认轮询间隔 30 秒，即使改成 1 秒，在流量突变时仍然不够快

**修复**：结合 Sentinel 的系统保护规则（CPU 80%）作为硬兜底，不依赖动态规则的热更新速度

### 3.6 事故预防自动化

基于五大事故的教训，实现自动化保护护盾：

```java
@Component
public class SentinelGuardian {

    @EventListener
    public void onRuleChanged(RuleChangeEvent event) {
        // 事故 1 防御：阈值变更幅度超过 50% 时告警
        FlowRule oldRule = event.getOldRule();
        FlowRule newRule = event.getNewRule();
        double changeRatio = Math.abs(newRule.getCount() - oldRule.getCount()) / oldRule.getCount();

        if (changeRatio > 0.5) {
            log.warn("[Guardian] 阈值大幅变更: {} {} -> {}, 变化幅度 {:.0f}%",
                oldRule.getResource(),
                oldRule.getCount(), newRule.getCount(), changeRatio * 100);
            // 发送告警到企微/钉钉
            alertService.send("阈值异常变更",
                String.format("资源 %s 阈值从 %.0f 变更为 %.0f（变化 %.0f%%），请确认",
                    oldRule.getResource(), oldRule.getCount(),
                    newRule.getCount(), changeRatio * 100));
        }
    }

    @Scheduled(fixedDelay = 30000)
    public void checkDegradeWindow() {
        // 事故 2 防御：检查熔断窗口是否过长
        List<DegradeRule> rules = DegradeRuleManager.getRules();
        for (DegradeRule rule : rules) {
            if (rule.getTimeWindow() > 30) {
                log.warn("[Guardian] 熔断窗口过长: {} timeWindow={}s, 建议 ≤30s",
                    rule.getResource(), rule.getTimeWindow());
            }
        }
    }

    @Scheduled(fixedDelay = 60000)
    public void checkParamFlowCache() {
        // 事故 5 防御：检查热点参数缓存使用量
        int ruleCount = ParamFlowRuleManager.getRules().size();
        if (ruleCount > 50) {
            log.warn("[Guardian] 热点参数规则过多: {} 条, 建议控制在 TopN 范围内", ruleCount);
        }
    }
}
```

### 3.7 规则变更审计钩子

```java
@Component
public class SentinelAuditHook {

    @EventListener
    public void onRuleLoad(RuleLoadEvent event) {
        for (Object rule : event.getRules()) {
            String ruleJson = JSON.toJSONString(rule);
            String resource = extractResource(rule);
            // 写入 MySQL 审计表
            auditMapper.insert(new AuditRecord(
                System.currentTimeMillis(),
                getCurrentOperator(),
                resource,
                rule.getClass().getSimpleName(),
                ruleJson,
                "LOAD"
            ));
            log.info("[Audit] 规则加载: {} -> {}", resource, ruleJson);
        }
    }

    private String extractResource(Object rule) {
        if (rule instanceof FlowRule) return ((FlowRule) rule).getResource();
        if (rule instanceof DegradeRule) return ((DegradeRule) rule).getResource();
        if (rule instanceof AuthorityRule) return ((AuthorityRule) rule).getResource();
        return "unknown";
    }
}
```

4. **规则回滚不能只看时间**：事故 1 中运维"紧急回滚"花费了 10 分钟——因为回滚脚本不熟练。需要定期演练。

### 4.1 六大事故总结

|------|------|------|---------|

| 2 | 参数不当 | 熔断窗口过长 | 根据故障恢复时间设定 timeWindow |
| 3 | 逻辑缺陷 | Fallback 掩盖异常 | 区分 blockHandler/fallback，做好补偿 |

### 4.4 Sentinel 安全配置基线

- [ ] 所有生产规则变更须经审批
- [ ] 熔断窗口 ≤ 30 秒
- [ ] Fallback 不吞掉业务异常，异常信息须记录
- [ ] 网关和服务层的阈值成比例（网关 ≥ 服务层 × 1.2）
- [ ] 热点参数限流缓存上限 ≤ 10000
