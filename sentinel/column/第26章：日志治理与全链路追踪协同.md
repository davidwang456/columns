# 第26章：日志治理与全链路追踪协同

## 1 项目背景

第 25 章我们搭建了 Prometheus + Grafana 监控体系，能实时看到 QPS 和 RT 曲线。但当客服反馈"某用户下单失败"时，运维仍然需要排查 15 分钟：先在 ELK 中找到用户请求的 traceId → 找到关联的订单服务日志 → 发现是 Sentinel 限流 → 再去 Sentinel block 日志中找对应时间的拦截记录 → 确认是哪个规则。

问题是：**Sentinel 的 block 日志和业务日志没有打通**。业务日志里可能记录了"下单失败"但没有写明是 Sentinel 拦截；Sentinel block 日志中记录了"createOrder 被限流"但没有 traceId 或 userId，无法关联到具体用户。

生产排障的核心需求是：**通过一个 traceId/requestId，能看到这个请求经过的所有服务、执行了哪些 Sentinel 规则、是否被拦截、为什么被拦截**。这就是全链路追踪的价值所在。

## 2 项目设计

**小胖**："我在 Kibana 里搜 traceId 'abc123'，能看到订单服务、库存服务的日志，但就是看不到 Sentinel 的 block 记录。两个日志系统没打通。"

**大师**："因为 Sentinel 的 block 日志是框架自动打的，没有业务上下文。你需要自定义 BlockException 处理逻辑，在拦截时主动打印包含 traceId 的结构化日志。"

**小白**："OpenTelemetry 呢？能把 Sentinel 事件作为 Span 的一环吗？"

**大师**："可以。你可以给 Sentinel 的阻断事件创建一个 OpenTelemetry Event 附加到当前 Span 上。这样在 Jaeger/Zipkin 中就能看到某个请求在哪个 Span 被 Sentinel 拦截了。"

**小胖**："日志采样那里，100 条才打一条会不会漏掉重要信息？比如有个用户被限流了，客服查不到他的 traceId 对应的 block 日志。"

**大师**："好问题。采样策略需要按场景设计——全量采样（100%）适合低流量场景或 P0 级别的告警；按比例采样适合高流量。更好的做法是按 resource 维度做精细控制：核心接口采样 100%，非核心接口采样 10%。甚至可以用'基于用户维度的蓄水池采样'——保证同一个 userId 的请求中至少有一条被采样。"

**小白**："除了 ELK，还有什么其他方案？比如在 Grafana Loki 里直接查 block 日志？"

**大师**："Loki 是更好的轻量方案——它不需要 Logstash 做 JSON 解析，直接按 label 索引。你只需要在 Promtail 配置中把 `event=sentinel_block` 作为 label，就能在 Grafana 中直接关联 Metric（Prometheus）和 Log（Loki）。但 Loki 的全文搜索性能不如 ES，适合已知 label 的查询场景。"

**小胖**："那时间线怎么对齐？Prometheus 的 metric 每 15s 一个点，Loki 的日志是实时推送，Jaeger 的 Span 是请求粒度的。这三者的时间线各不相同。"

**大师**："通过 traceId 做纽带。在 Grafana 面板中可以用 Jaeger 数据源看完整的 Span 时间线，点击某个 Span 可以看到'关联的 Loki 日志'（通过 traceId label），同时在同一个 Dashboard 的 Prometheus 面板上看该 resource 的 QPS/RT 曲线。Grafana 8+ 支持 Loki 和 Jaeger 之间的 traceId 关联跳转。"

**小白**："MDC 传播在 Spring Cloud Gateway / Zuul 这种网关层应该怎么做？网关的请求线程和下游调用线程不一样。"

**大师**："网关层需要在 Filter 中把 traceId 通过 HTTP Header（如 `X-B3-TraceId`）透传给下游。Spring Cloud Sleuth 基于 Brave 会自动做这件事。如果不用 Sleuth，手动实现的话：在网关的 Pre Filter 中生成/提取 traceId → 写入 MDC → 通过 `RequestTemplate.header()` 传到下游 → 下游的 Filter 中从 Header 读取后写入自己的 MDC。"

**小胖**："如果请求量特别大，比如双十一 QPS=50000，block 日志每秒可能几万条——这磁盘扛不住啊。"

**大师**："这就需要在日志输出层做聚合。一个思路是：不在每个 block 事件时直接写日志，而是用内存中的环形缓冲区（RingBuffer）汇总——每 5 秒统计一次 `resource X 在过去 5 秒内被 block 了 N 次，sample traceId=[id1, id2, ...]`——然后每 5 秒输出一条聚合日志。即把数千条日志压缩成一条。"

## 3 项目实战

### 3.1 步骤一：结构化 Block 日志

```java
@Component
public class StructuredBlockHandler implements BlockExceptionHandler {

    @Override
    public void handle(HttpServletRequest request, HttpServletResponse response,
                       BlockException e) throws Exception {

        // 提取 TraceId（假设已通过 MDC 传播）
        String traceId = MDC.get("traceId");
        String requestId = request.getHeader("X-Request-Id");
        String userId = request.getHeader("X-User-Id");
        String resourceName = e.getRule().getResource();

        // 结构化日志
        Map<String, Object> blockLog = new LinkedHashMap<>();
        blockLog.put("event", "sentinel_block");
        blockLog.put("traceId", traceId);
        blockLog.put("requestId", requestId);
        blockLog.put("userId", userId);
        blockLog.put("resource", resourceName);
        blockLog.put("exception", e.getClass().getSimpleName());
        blockLog.put("rule", e.getRule().toString());
        blockLog.put("timestamp", System.currentTimeMillis());
        blockLog.put("uri", request.getRequestURI());
        blockLog.put("clientIp", request.getRemoteAddr());

        // 输出 JSON 格式日志（可被 ELK/Loki 解析）
        log.warn("SENTINEL_BLOCK {}", JSON.toJSONString(blockLog));

        response.setStatus(429);
        response.getWriter().write(JSON.toJSONString(
            Map.of("code", "BLOCK", "msg", "请求被拦截", "requestId", requestId)
        ));
    }
}
```

### 3.3 步骤三：OpenTelemetry 集成

```java
@Configuration
public class OpenTelemetrySentinelIntegration {

    @PostConstruct
    public void init() {
        // 注册 Sentinel 事件监听器
        SentinelEventBus.subscribe(new SentinelEventSubscriber() {
            @Override
            public void onBlock(String resource, BlockException ex) {
                // 获取当前 OpenTelemetry Span
                Span currentSpan = Span.current();
                if (currentSpan != null && currentSpan.isRecording()) {
                    // 在 Span 上添加 Sentinel 阻断事件
                    currentSpan.addEvent("sentinel.blocked",
                        Attributes.of(
                            AttributeKey.stringKey("resource"), resource,
                            AttributeKey.stringKey("exception"),
                                ex.getClass().getSimpleName(),
                            AttributeKey.stringKey("rule"),
                                ex.getRule().toString()
                        )
                    );
                    currentSpan.setStatus(StatusCode.ERROR,
                        "Blocked by Sentinel: " + resource);
                }
            }
        });
    }
}
```

在 Jaeger 中可以看到请求的 Span 上有一个 `sentinel.blocked` 事件。

### 3.4 步骤四：限流事件过多时的采样

```java
@Component
public class BlockEventSampler {

    private final ConcurrentHashMap<String, AtomicLong> counter = new ConcurrentHashMap<>();
    private static final int SAMPLE_EVERY_N = 100; // 每 100 次才打一条日志

    public boolean shouldLog(String resource) {
        long count = counter.computeIfAbsent(resource,
            k -> new AtomicLong(0)).incrementAndGet();
        return count % SAMPLE_EVERY_N == 1; // 第 1, 101, 201... 次打日志
    }

    public void logBlockWithSampling(String traceId, String resource, BlockException e) {
        if (shouldLog(resource)) {
            log.warn("SENTINEL_BLOCK_SAMPLED resource={}, exception={}, traceId={}",
                     resource, e.getClass().getSimpleName(), traceId);
        }
    }
}
```

### 3.5 步骤五：Kibana 完整故障链路查询

通过 Kibana Discover，输入 Lucene 查询语法：

```
traceId: "abc123" AND (event: "sentinel_block" OR level: "ERROR")
```

结果一目了然：

| 时间 | 服务 | traceId | 事件 |
|------|------|---------|------|
| 10:23:00.100 | order-service | abc123 | POST /order/create 收到请求 |
| 10:23:00.150 | order-service | abc123 | SENTINEL_BLOCK: createOrder FlowException |
| 10:23:00.151 | order-service | abc123 | 返回 429 Too Many Requests |

**踩坑记录**：

1. **MDC 传播**：在异步调用和线程池中，MDC 不会自动传播。需在 `ContextUtil.runOnContext()` 中手动传递 MDC。
2. **日志量爆炸**：限流高峰时每秒可能有数千条 block 日志，必须加采样机制。
3. **OpenTelemetry Agent**：如果用 Java Agent 自动注入，Sentinel 事件可能在 Agent 之前，导致 Span 为空。确保 Span 传播在 Sentinel Slot 执行之前。

### 3.8 聚合 Block 日志的 RingBuffer 实现

```java
@Component
public class AggregatedBlockLogger {

    private final ConcurrentHashMap<String, BlockAggregation> aggregations = new ConcurrentHashMap<>();
    private final ScheduledExecutorService scheduler = Executors.newSingleThreadScheduledExecutor();

    @PostConstruct
    public void init() {
        // 每 5 秒输出一次聚合日志
        scheduler.scheduleAtFixedRate(this::flush, 5, 5, TimeUnit.SECONDS);
    }

    public void record(String resource, String traceId, String userId, String exception) {
        aggregations.compute(resource, (k, agg) -> {
            if (agg == null) agg = new BlockAggregation(resource);
            agg.count++;
            if (agg.sampleTraceIds.size() < 5) {
                agg.sampleTraceIds.add(traceId);
            }
            return agg;
        });
    }

    private void flush() {
        aggregations.forEach((resource, agg) -> {
            if (agg.count > 0) {
                Map<String, Object> aggregated = new LinkedHashMap<>();
                aggregated.put("event", "sentinel_block_aggregated");
                aggregated.put("resource", resource);
                aggregated.put("count", agg.count);
                aggregated.put("sampleTraceIds", agg.sampleTraceIds);
                aggregated.put("window", "5s");
                log.warn("SENTINEL_BLOCK_AGG {}", JSON.toJSONString(aggregated));
                agg.reset();
            }
        });
    }

    static class BlockAggregation {
        String resource;
        AtomicLong count = new AtomicLong(0);
        Set<String> sampleTraceIds = ConcurrentHashMap.newKeySet();
        BlockAggregation(String r) { this.resource = r; }
        void reset() { count.set(0); sampleTraceIds.clear(); }
    }
}
```

### 3.9 MDC 跨线程传播工具类

```java
public class MdcPropagationUtil {

    @FunctionalInterface
    public interface MdcAwareRunnable {
        void run(Map<String, String> parentMdc);
    }

    public static void runWithMdc(MdcAwareRunnable task) {
        Map<String, String> parentMdc = MDC.getCopyOfContextMap();
        new Thread(() -> task.run(parentMdc)).start();
    }

    // 配合 Sentinel ContextUtil 使用
    public static void runInSentinelContext(Context context, Runnable task) {
        Map<String, String> mdcContext = MDC.getCopyOfContextMap();
        ContextUtil.runOnContext(context, () -> {
            if (mdcContext != null) MDC.setContextMap(mdcContext);
            try { task.run(); } finally { MDC.clear(); }
        });
    }
}
```

## 4 项目总结

### 4.1 日志治理清单

| 治理项 | 方案 | 优先级 | 成熟度 |
|-------|------|-------|-------|
| Block 日志结构化 | JSON 格式 + traceId + userId + resource | P0 | 生产验证 |
| 日志采集 | Filebeat/Promtail → Elasticsearch/Loki | P0 | 生产验证 |
| 日志查询 | Kibana Discover + Grafana Loki | P1 | 生产验证 |
| 链路关联 | traceId 关联 block 日志和业务日志 | P1 | 生产验证 |
| 采样机制 | 限流高峰期日志采样/聚合 | P2 | 需定制 |
| MDC 跨线程传播 | ContextUtil + MDC 工具类 | P1 | 生产验证 |
| 聚合输出 | RingBuffer 聚合 → 每 5s 一条聚合日志 | P2 | 需定制 |

### 4.2 ELK vs Loki 技术选型对比

| 维度 | ELK (Elasticsearch) | Loki + Promtail |
|------|--------------------|--------------------|
| 部署复杂度 | 高（需要 ES + Logstash + Kibana） | 低（Grafana 内置 + Promtail DaemonSet） |
| 全文搜索 | 极强，支持复杂 Lucene 查询 | 弱，仅支持 LogQL label 过滤 |
| 存储成本 | 高（索引体积约为原始日志的 2-3x） | 低（仅索引 label，原始日志压缩存储） |
| 与 Prometheus 联动 | 需额外配置 | 原生支持（Grafana 同一数据源面板） |
| 适合场景 | 频繁全文搜索、复杂日志分析 | 按 label 查询、与 Metric 联动查看 |

### 4.3 日志采样策略决策表

| 采样策略 | 实现方式 | 适用场景 | 采样率 |
|---------|---------|---------|-------|
| 全量 | 不采样 | P0 核心接口，低流量 | 100% |
| 固定比例 | `count % N == 0` | 一般业务接口 | 1-10% |
| 按用户维度 | `hash(userId) % N == 0` | 需要保证同一用户可追溯 | 10-20% |
| 时间窗口聚合 | RingBuffer 聚合 | 超高流量（> 10000 QPS） | 1 条/5s |
| 按资源分级 | 不同 resource 不同采样率 | 混合核心/非核心接口 | 动态 |

### 4.5 推广计划

- **开发团队**：在统一的日志框架中封装 Sentinel 结构化日志模板，所有服务统一使用。
- **运维团队**：在 Kibana/Grafana 中建立 Sentinel 专用查询视图，减少排障时间。
- **SRE**：将 Sentinel block 日志接入告警系统（如 ElastAlert/Prometheus AlertManager）。
