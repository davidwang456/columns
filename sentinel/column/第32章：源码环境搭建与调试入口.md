# 第32章：源码环境搭建与调试入口

## 1 项目背景

前三篇（基础/中级）我们都是"使用" Sentinel，知道怎么配规则、怎么看 Dashboard、怎么接 Nacos。但从这一章开始，我们要进入 Sentinel 的内部世界——理解 `SphU.entry()` 到底做了什么、Slot Chain 如何工作、滑动窗口怎么统计。

为什么要读源码？三个场景驱动：

- 你在排查"为什么规则不生效"，Dashboard 和日志都看不出问题，必须断点进入 Sentinel 内部查看
- 你需要自定义一个 Slot（比如增加业务标签、修改限流判断逻辑），必须理解 Slot Chain 的扩展机制
- 你在 10 万 QPS 场景下发现 Sentinel 有性能瓶颈，需要精确定位热点代码并优化

本章将搭建 Sentinel 源码调试环境，带你从 `SphU.entry()` 入口一路跟到 Slot Chain 的完整调用链路。

## 2 项目设计

**小胖**："源码？我最怕看源码了。光 Sentinel 的 core 模块就有几百个类。"

**大师**："不用全看。Sentinel 源码阅读有主线：`SphU.entry()` → `CtSph.entry()` → `lookupProcessChain()` → SlotChain 遍历。你只需要跟这条线走一遍，80% 的核心逻辑就掌握了。"

**小白**："我 fork 了 Sentinel 的 GitHub 仓库，但模块太多了——sentinel-core、sentinel-adapter、sentinel-dashboard、sentinel-extension...从哪里开始？"

**大师**："从 `sentinel-core` 开始。它是核心，其他模块都依赖它。核心类就几个：`SphU`（入口 API）、`CtSph`（实现）、`ProcessorSlotChain`（责任链）、`DefaultSlotChainBuilder`（链构建器）。"

**小胖**："大师，我 clone 了源码，但 Maven 下载依赖要很久，而且 `sentinel-dashboard` 模块依赖前端 npm，总是构建失败。"

**大师**："看源码不需要构建全部模块。你只需要 `sentinel-core`、`sentinel-transport-common`、`sentinel-extension` 这三个模块。在 IDE 中把其他模块标记为 `Excluded`，Maven 就不会下载它们的依赖。Dashboard 前端代码和源码调试无关，直接用 Docker 镜像运行 Dashboard 即可。"

**小白**："断点调试 Slot Chain 时，我发现 `chainMap` 是一个 `ConcurrentHashMap`，每个资源对应一个 Slot Chain。这意味着资源名写错了（比如 `orderCreate` vs `createOrder`）会创建两个独立的 Slot Chain，而且不会报错。这个有什么办法预防？"

**大师**："没有。资源名就是自由字符串，Sentinel 不做校验。但你可以利用 `chainMap.size()` 来监控异常——如果资源数量突然从预期的 20 个变成 50 个，说明有拼写错误或动态资源名。另一个好习惯是：所有 `@SentinelResource` 注解的资源名统一维护在一个常量类中。"

**小胖**："那 `ContextUtil.getContext()` 返回的 Context 是怎么管理的？每个请求一个 Context 还是全局共享？"

**大师**："Context 是基于 **ThreadLocal** 的，所以每个线程一个。框架集成（如 Spring MVC Interceptor、Dubbo Filter）会自动帮你创建 Context。入口在 `WebServletConfig`（`CommonFilter`），每次请求进入时调用 `ContextUtil.enter()`，请求结束时调用 `ContextUtil.exit()`。理解这一点很重要——如果你在异步线程中调用 `SphU.entry()`，一定要手动传递 Context。"

**小胖**："那如果我在 `@Async` 方法里调 `SphU.entry()` 会怎样？Context 会丢吗？"

**大师**："会的。`@Async` 使用新的线程池线程，ThreadLocal 中的 Context 不会自动传递。这就是为什么 Sentinel 提供了 `SphU.asyncEntry()` 和 `AsyncEntry` 机制。在异步场景下，你需要手动把父线程的 Context 传到子线程：

```java
Context context = ContextUtil.getContext();
CompletableFuture.supplyAsync(() -> {
    ContextUtil.runOnContext(context, () -> {
        try (Entry entry = SphU.entry("asyncResource")) {
            // 异步业务逻辑
        } catch (BlockException e) {
            // 处理限流
        }
    });
});
```

如果调试时发现异步线程中 `ContextUtil.getContext()` 返回 null，就是这个原因。"

**小白**："那 Sentinel 用 ThreadLocal 存 Context，在高并发下不会有内存泄漏吗？"

**大师**："理论上不会，因为 `ContextUtil.exit()` 会在请求结束时清理 ThreadLocal。但如果框架集成有 bug——比如 `CommonFilter` 的 `finally` 块没有执行 `exit()`——就会泄漏。调试时可以用 `ContextUtil.getContextSize()` 查看当前活跃的 Context 数量，如果持续增长不下降，说明存在 Context 泄漏。"

### 3.2 分步实现

**步骤一：从 SphU.entry() 入口开始**

打开 `sentinel-core/src/main/java/com/alibaba/csp/sentinel/SphU.java`：

```java
public static Entry entry(String name) throws BlockException {
    return Env.sph.entry(name, EntryType.OUT, 1, OBJECTS0);
}
```

跟踪 `Env.sph` → 在 `Env.java` 中发现通过 SPI 加载 `Sph` 接口的实现类 → `CtSph`。

**步骤二：CtSph.entry() — 核心入口**

```java
// CtSph.java 简化版
public Entry entry(String name, EntryType type, int count, Object... args)
        throws BlockException {
    // 1. 创建 Context
    Context context = ContextUtil.getContext();
    if (context == null) {
        context = InternalContextUtil.internalEnter(Constants.CONTEXT_DEFAULT_NAME);
    }

    // 2. 创建资源包装器
    ResourceWrapper resourceWrapper = new StringResourceWrapper(name, type);

    // 3. 获取该资源的 Slot Chain
    ProcessorSlotChain chain = lookProcessChain(resourceWrapper);

    // 4. 创建 Entry
    Entry e = new CtEntry(resourceWrapper, chain, context);

    // 5. 执行 Slot Chain
    chain.entry(context, resourceWrapper, null, count, false, args);
    return e;
}
```

**关键断点**：
1. `lookProcessChain()` — 观察 Slot Chain 如何创建和缓存
2. `chain.entry()` — 观察每个 Slot 的执行过程

**步骤三：Slot Chain 构建过程**

```java
// CtSph.java
private ProcessorSlotChain lookProcessChain(ResourceWrapper resourceWrapper) {
    ProcessorSlotChain chain = chainMap.get(resourceWrapper);
    if (chain == null) {
        // 获取 SlotChainBuilder，默认是 DefaultSlotChainBuilder
        chain = SlotChainProvider.newSlotChain();
        chainMap.put(resourceWrapper, chain);
    }
    return chain;
}
```

`DefaultSlotChainBuilder` 通过 SPI 加载所有 `ProcessorSlot` 的实现，按 `@SpiOrder` 注解的顺序构建链。

### 3.6 调试 StatisticSlot：理解滑动窗口统计

`StatisticSlot` 是 Sentinel 中最复杂的 Slot——它负责统计 QPS、RT、线程数。

```java
// StatisticSlot.entry() 简化核心逻辑
public void entry(Context context, ResourceWrapper resourceWrapper,
                  DefaultNode node, int count, boolean prioritized, Object... args) {
    // 1. 增加线程数（并发度统计）
    node.increaseThreadNum();

    // 2. 记录开始时间（后续 exit 中计算 RT）
    context.getCurEntry().setStartTime(System.currentTimeMillis());

    // 3. 调用后续 Slot（规则检查）
    fireEntry(context, resourceWrapper, node, count, prioritized, args);

    // 4. 如果到这里，说明通过了所有规则 → 增加通过 QPS
    node.addPassRequest(count);
}
```

断点设置在 `node.addPassRequest(count)` 处，然后单步进入 `StatisticNode`：
- 观察 `LeapArray<MetricBucket>` 如何获取当前滑动窗口
- 观察 `MetricBucket.add(MetricEvent.PASS, count)` 如何用 `LongAdder` 累加计数
- 观察窗口过期后如何通过 CAS 重置

**关键发现**：`MetricBucket` 使用 `LongAdder` 而非 `AtomicLong`——在高并发下，`LongAdder` 的写入性能更好（分散热点到多个 Cell）。

### 3.7 调试 FlowSlot：理解限流判断逻辑

```java
// FlowSlot.entry() 核心调用链
public void entry(Context context, ResourceWrapper resourceWrapper,
                  DefaultNode node, int count, boolean prioritized, Object... args) {
    // 1. 获取该资源的所有流控规则
    List<FlowRule> rules = FlowRuleManager.getRules(resourceWrapper.getName());

    // 2. 逐条规则检查
    for (FlowRule rule : rules) {
        if (!FlowRuleChecker.checkFlow(rule, context, node, count, prioritized)) {
            throw new FlowException(rule.getLimitApp(), rule);
        }
    }
}
```

断点调试技巧：
1. 在 `FlowRuleChecker.checkFlow()` 中断点，观察它如何根据 `rule.getGrade()` 选择限流算法
2. `FLOW_GRADE_QPS` → `DefaultController.canPass()` → 基于滑动窗口的 QPS 判断
3. `FLOW_GRADE_THREAD` → 直接比较 `node.curThreadNum()` 和阈值

特别观察 `DefaultController.canPass()` 中的 `Warm Up` 和 `匀速排队` 两种控制行为：
- Warm Up：令牌桶的初始容量较小，逐渐增加
- 匀速排队（Rate Limiter）：漏桶算法，严格控制通过间隔

### 3.8 调试 ClusterBuilderSlot：理解 Node 层级

```java
// ClusterBuilderSlot.entry() — 构建 ClusterNode 树
public void entry(Context context, ResourceWrapper resourceWrapper,
                  DefaultNode node, int count, boolean prioritized, Object... args) {
    // ClusterNode 是全局共享的（所有 Context 共用）
    // DefaultNode 是 Context 隔离的（每个 Context 有自己的调用链树）
    ClusterNode clusterNode = ClusterBuilderSlot.getClusterNode(resourceWrapper.getName());

    // 关键：将 DefaultNode 挂到 Context 的 Node Tree 上
    // 这一步建立了调用链的树形结构
    context.getCurEntry().setCurNode(node);
    node.setClusterNode(clusterNode);

    fireEntry(context, resourceWrapper, node, count, prioritized, args);
}
```

在 IDE 的 Variables 窗口观察：
- `node`（DefaultNode）：与当前 Context 绑定，记录了到当前资源为止的调用链统计
- `clusterNode`（ClusterNode）：全局唯一，记录该资源在所有 Context 中的总统计

```java
public class AsyncContextDebugTest {

        initFlowRules();

        // 模拟父线程请求
        ContextUtil.enter("async-context-debug", "app-web");
        Context ctx = ContextUtil.getContext();

        CompletableFuture<Void> future = CompletableFuture.runAsync(() -> {

                try (Entry entry = SphU.entry("asyncResource")) {
                    System.out.println("[子线程] Context=" +
                        ContextUtil.getContext().getName() +
                        " 通过");

                    System.out.println("[子线程] 被限流");
                }
            });
        });

        future.get();
        ContextUtil.exit();
        System.out.println("[主线程] 退出 Context, 活跃数=" + ContextUtil.getContextSize());
    }
}
```

### 4.5 思考题

1. 在 `CtSph.entry()` 中，如果 `chainMap` 已经缓存了某个资源的 Slot Chain，后续请求会复用还是重新构建？这会不会导致内存泄漏？
