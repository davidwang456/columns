# 第28章：规则治理平台设计

## 1 项目背景

随着微服务数量从 5 个增长到 30 个，每个服务有 5-15 条 Sentinel 规则，Nacos 中累积了超过 300 条规则配置。问题也随之而来：

- 某个周五，一个开发在 Nacos 中"手滑"修改了生产环境的流控阈值，导致订单服务被限流 2 小时才被发现。因为没有变更审批，没有操作审计，没有人知道是谁改的
- 测试团队反馈："我想知道有哪些规则改变了，但 Nacos 的历史版本只能逐条查看，30 个服务的规则太多了"
- 运维团队反馈："线上紧急需要把 5 个服务的限流阈值翻倍——但我得在 Nacos 控制台里逐个配置、逐个发布，至少 15 分钟"

这些痛点说明：当 Sentinel 规则数量超过一定规模后，**需要一个专门的规则治理平台**——而不是直接裸操作 Nacos。这个平台需要提供：规则概览、变更审批、一键回滚、定时生效、变更审计、冲突检测。

## 2 项目设计

**大师**："规则治理平台的核心是'把人对规则的修改变成可追溯、可审批、可回滚的流程'。技术上可以基于 Nacos 的 API 做二次封装，或者直接用 Dashboard 二次开发。"

**小白**："Dashboard 二次开发可行吗？我看 Dashboard 源码架构还算清晰。"

**大师**："可行但成本不低。Dashboard 源码是前后端分离的 Spring Boot 项目，你需要在上面加审批流、审计日志、权限控制等功能。对小团队来说，性价比更高的方案是：Nacos 存规则 + 自建轻量级治理平台（只做审批和回滚）+ Grafana 做监控。"

**小胖**："那核心功能包括哪些？"

**大师**：
1. **规则元数据管理**：每条规则关联服务名、负责人、创建时间、生效范围
2. **审批流**：规则变更 → 发起审批 → Leader 审批通过 → 自动写入 Nacos
3. **定时生效**：大促规则可以设置"5 月 1 日 00:00 生效"
4. **紧急回滚**：一键恢复到指定时间点的规则快照
5. **冲突检测**：新增规则时检测是否与已有规则冲突

**小白**："那规则的'灰度发布'怎么实现？大促规则不能一刀切地全量上线。"

**大师**："灰度规则在 Sentinel 原生的规则模型中没有直接支持，但可以通过 Nacos 的 namespace 分组来实现。比如，创建 `prod-gray` 命名空间，先让 10% 的实例订阅灰度规则。验证通过后，再把灰度规则合并到 `prod` 命名空间。Sentinel 客户端配置：`spring.cloud.nacos.discovery.namespace=prod-gray`。"

**小胖**："审批流用哪个工作流引擎？Camunda？Flowable？还是自己写？"

**大师**："小团队不要引入工作流引擎——那又是一套需要维护的系统。推荐两种轻量方案：一是基于钉钉/企微的 OA 审批（审批通过后回调你的治理平台 API）；二是基于 Git PR——把规则配置写成 YAML 文件放在 Git 仓库里，修改规则 = 提 PR，Leader Approve = 合并 PR，CI Pipeline 合并后自动推送到 Nacos。第二种方案天然有版本历史和审计。"

**小白**："那冲突检测除了'同资源同类型规则重复'，还有哪些需要检测的？"

**大师**："至少四种。第一，**阈值冲突**：上层限流阈值 < 下层限流阈值（网关 100，服务 200 → 网关先挡，服务限流永不会触发）。第二，**资源名冲突**：两个服务用了相同的资源名。第三，**熔断与限流交叉**：同一个资源既配了限流又配了严格的熔断，可能导致'双重拦截'。第四，**全局限流与单机限流不一致**：集群流控阈值 / 实例数 ≠ 单机阈值。"

**小胖**："大师，规则治理平台的用户角色怎么设计？总不能所有人都能改生产规则吧？"

**大师**："最少三种角色。**规则查看者**（只读，测试和运维值班）、**规则编辑者**（可修改规则但需审批，核心开发）、**规则管理员**（可审批和回滚，Tech Lead/架构师）。如果需要跨团队，再加一个**超级管理员**（负责命名空间和环境权限分配）。权限验证可以对接企业 SSO 或 LDAP。"

**小白**："规则治理平台的告警应该怎么设计？我发现有些问题是'规则变了但没告警'导致的。"

**大师**："告警分三级。**info 级**：规则变更通知（谁改了哪条规则），发到团队 IM 群。**warn 级**：规则变更后 5 分钟内核心指标异常（如拒绝率突增 > 30%），@owner 和值班人员。**critical 级**：系统保护触发超过 1 分钟，或熔断 OPEN 超过 5 分钟——电话告警。"

```sql
CREATE TABLE sentinel_rule_metadata (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    service_name VARCHAR(128) NOT NULL,

    created_at DATETIME DEFAULT NOW(),
    updated_at DATETIME DEFAULT NOW(),

    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    rule_id BIGINT NOT NULL,
    action VARCHAR(16) NOT NULL,           -- CREATE/UPDATE/DELETE/ROLLBACK
    operator VARCHAR(64) NOT NULL,
    old_content TEXT,
    new_content TEXT,
    created_at DATETIME DEFAULT NOW()
);
```

### 3.2 审批流实现

```java
@RestController
@RequestMapping("/governance")
public class RuleGovernanceController {

    @PostMapping("/rule/propose")
    public Result<String> proposeRuleChange(@RequestBody RuleChangeRequest req) {
        // 1. 记录变更请求
        Long ruleId = ruleMetadataService.saveProposed(req);

        // 2. 触发审批（发企微/钉钉通知给 owner）
        approvalService.submitApproval(ruleId, req.getOwner());

        return Result.success("规则变更已提交审批, ID=" + ruleId);
    }

    @PostMapping("/rule/approve/{ruleId}")
    public Result<String> approveRule(@PathVariable Long ruleId) {
        // 1. 更新审批状态
        ruleMetadataService.approve(ruleId);

        // 2. 检查是否有定时生效配置
        RuleMetadata meta = ruleMetadataService.get(ruleId);
        if (meta.getEffectiveTime() != null) {
            // 延迟生效：存入定时任务
            scheduledRuleService.schedule(ruleId, meta.getEffectiveTime());
        } else {
            // 立即生效：写入 Nacos
            nacosService.pushRule(meta);
        }

        // 3. 记录审计日志
        auditService.log(ruleId, "APPROVED", meta.getOwner());

        return Result.success("规则审批通过并已部署");
    }
}
```

### 3.3 一键回滚

```java
@Service
public class RuleRollbackService {

    public void rollback(String serviceName, String environment, Date targetTime) {
        // 1. 从审计表获取目标时间点的规则快照
        List<RuleMetadata> snapshot = auditService.getSnapshot(
            serviceName, environment, targetTime);

        // 2. 批量写入 Nacos
        for (RuleMetadata rule : snapshot) {
            nacosService.pushRule(rule);
        }

        // 3. 记录回滚审计
        auditService.logRollback(serviceName, environment, targetTime);

        // 4. 通知相关人
        notificationService.notify(
            "规则回滚通知",
            String.format("%s/%s 的规则已回滚至 %s", serviceName, environment, targetTime)
        );
    }
}
```

### 3.4 冲突检测

```java
public List<RuleConflict> detectConflicts(FlowRule newRule, String serviceName) {
    List<RuleConflict> conflicts = new ArrayList<>();

    // 加载该服务的所有已有流控规则
    List<FlowRule> existingRules = nacosService.loadFlowRules(serviceName);

    for (FlowRule existing : existingRules) {
        // 检测：同一个资源有两条 QPS 流控规则
        if (existing.getResource().equals(newRule.getResource())
            && existing.getGrade() == newRule.getGrade()
            && existing.getStrategy() == newRule.getStrategy()) {
            conflicts.add(new RuleConflict(
                "DUPLICATE",
                "资源 " + newRule.getResource() + " 已有同类流控规则",
                existing
            ));
        }
    }
    return conflicts;
}
```

### 3.6 规则版本管理与快照

```java
@Service
public class RuleSnapshotService {

    // 每日自动快照 + 手动快照
    @Scheduled(cron = "0 0 2 * * ?") // 每天凌晨 2 点
    public void dailySnapshot() {
        List<String> services = registryService.getAllServiceNames();
        for (String service : services) {
            for (String env : Arrays.asList("prod", "staging")) {
                List<RuleMetadata> rules = nacosService.loadAllRules(service, env);
                RuleSnapshot snapshot = new RuleSnapshot();
                snapshot.setServiceName(service);
                snapshot.setEnvironment(env);
                snapshot.setSnapshotTime(new Date());
                snapshot.setRules(JSON.toJSONString(rules));
                snapshotRepository.save(snapshot);
            }
        }
        log.info("Daily snapshot completed for {} services", services.size());
    }

    // 对比两个快照的差异（用于变更审计）
    public List<RuleDiff> compareSnapshot(Long snapshotId1, Long snapshotId2) {
        List<RuleMetadata> rules1 = loadSnapshot(snapshotId1);
        List<RuleMetadata> rules2 = loadSnapshot(snapshotId2);

        List<RuleDiff> diffs = new ArrayList<>();
        for (RuleMetadata r2 : rules2) {
            RuleMetadata r1 = findMatching(rules1, r2);
            if (r1 == null) {
                diffs.add(new RuleDiff("ADDED", null, r2));
            } else if (!r1.getRuleContent().equals(r2.getRuleContent())) {
                diffs.add(new RuleDiff("MODIFIED", r1, r2));
            }
        }
        return diffs;
    }
}
```

### 3.7 规则推荐引擎（基于历史数据）

```java
@Service
public class RuleRecommendationService {

    // 根据过去 7 天的 QPS 数据推荐流控阈值
    public FlowRule recommendFlowRule(String resourceName) {
        // 查询 Prometheus 过去 7 天的峰值 QPS
        double peakQps = prometheusQuery(
            "max_over_time(sentinel_resource_pass_qps{resource=\"" + resourceName + "\"}[7d])");

        // 推荐阈值 = 峰值 × 1.3（留 30% 余量）
        double recommended = peakQps * 1.3;

        FlowRule rule = new FlowRule(resourceName);
        rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
        rule.setCount(recommended);
        rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_WARM_UP);
        rule.setWarmUpPeriodSec(60);

        log.info("推荐阈值: {} → 峰值 QPS={}, 建议={}",
            resourceName, peakQps, recommended);
        return rule;
    }
}
```

### 3.9 Git PR 模式规则变更流程

这是一个完全不需要治理平台的轻量级方案：

```
开发修改规则 → Git 提交 YAML 规则文件 → 提交 PR
→ CI 自动校验（格式 + 冲突检测）
→ Tech Lead 审批 PR
→ 合并到主分支
→ CI Pipeline 读取 YAML → 调用 Nacos API 推送规则
```

### 3.10 规则冲突自动修复策略

```java
public enum ConflictResolutionStrategy {
    KEEP_EXISTING,      // 保留已有规则，拒绝新规则
    OVERWRITE,           // 新规则覆盖已有规则（需审批）
    MERGE_HIGHER,        // 取两者中更高的阈值（保守）
    MERGE_LOWER,         // 取两者中更低的阈值（激进）
    CREATE_ALIAS,        // 创建新资源名（如 createOrder-v2）
    REQUIRE_MANUAL       // 标记为需人工处理
}

public ConflictResolutionResult autoResolve(RuleConflict conflict) {
    return switch (conflict.getType()) {
        case "DUPLICATE" -> new ConflictResolutionResult(
            ConflictResolutionStrategy.KEEP_EXISTING,
            "已有同类规则，拒绝自动合并"
        );
        case "THRESHOLD_MISMATCH" -> new ConflictResolutionResult(
            ConflictResolutionStrategy.MERGE_LOWER,
            "网关阈值低于服务阈值，自动取较低值作为统一阈值"
        );
        default -> new ConflictResolutionResult(
            ConflictResolutionStrategy.REQUIRE_MANUAL,
            "需人工确认"
        );
    };
}
```

### 4.1 治理平台功能优先级矩阵

| 功能 | 重要性 | 实现难度 | 建议优先级 |
|------|-------|---------|----------|
| 规则 CRUD + 概览 | ★★★★★ | ★ | P0 第一期 |
| 变更审批流 | ★★★★★ | ★★ | P0 第一期 |
| 一键回滚 | ★★★★★ | ★★ | P0 第一期 |
| 操作审计日志 | ★★★★ | ★ | P1 第二期 |
| 冲突检测 | ★★★★ | ★★★ | P1 第二期 |

### 4.2 自研 vs 二次开发 vs 商业产品

| 方案 | 成本 | 适用团队规模 | 推荐度 |
|------|------|------------|-------|

### 4.2 思考题

1. 如果你的规则治理平台需要支持"灰度发布"——新规则先对 10% 的流量生效，验证通过后再全量生效。在 Sentinel 架构中，技术上怎么实现？
2. 如何设计一个"规则推荐引擎"——根据服务的历史流量模式，自动推荐合理的限流阈值？

### 4.3 推广计划
