# 第7章：熔断降级规则：慢调用、异常比例与异常数

## 1 项目背景

第 6 章的关联流控和链路流控让订单服务在"资源竞争"场景下有了更精细的控制能力。但接下来发生的一件事，让团队意识到限流远远不够。

凌晨 3 点，运维收到告警：订单服务整体 RT 从 50ms 飙升到 3000ms，CPU 使用率反而从 60% 降到了 15%。这在直觉上说不通——RT 涨了，CPU 应该更忙才对。排查发现，库存服务的数据库在做主从切换，导致库存查询从 10ms 变成 2000ms。订单服务有 200 个线程等待库存响应，线程池被耗尽，但 CPU 确实空闲——因为线程全在 IO 等待。

更重要的是，订单服务的 QPS 限流规则一直在正常生效（QPS=500），但因为每个请求的 RT 变长了，500 个线程全部卡在库存查询上，新的请求虽然被限流拦截（BlockException），但已经在处理的 500 个线程全部超时返回 500 错误给用户。

这就是典型的**级联故障**：一个下游服务变慢 → 上游线程堆积 → 上游服务资源耗尽 → 整个调用链不可用。限流只能控制"进多少"，不能隔离"已经进来但卡住的"。这时候需要的是**熔断降级**：检测到下游异常 → 主动切断调用 → 快速失败 → 给上游一个兜底方案。

本章将掌握 Sentinel 的三种熔断策略（慢调用比例、异常比例、异常数）以及熔断状态机（Closed → Open → Half-Open）的完整工作流程。

## 2 项目设计

**小胖**（看着监控大盘）："大师！限流不是已经把 QPS 压住了吗？为什么 RT 还是飙到 3 秒？"

**大师**："因为限流是'门禁'——它决定你能进多少人。但熔断是'巡逻员'——它决定进去的人有没有出问题。库存服务变慢了，你限流再严，已经进来的 500 个请求还是在等库存返回。"

**小白**："所以熔断是调用方的自我保护？当发现下游不可用时，主动切断调用，快速失败？"

**大师**："对。而且 Sentinel 的熔断比 Hystrix 更灵活。它支持三种策略：慢调用比例（响应时间超过阈值）、异常比例（调用异常超过比例）、异常数（调用异常超过数量）。"

**小胖**："那熔断器怎么知道什么时候该'恢复'呢？总不能断了就永远不用了吧？"

**大师**："这就涉及熔断状态机了。有三种状态：Closed（正常，全量调用）→ Open（熔断，全部快速失败）→ Half-Open（半开，放少量请求试探）。打个比方：你跟一家外卖店点了三次都送晚了（慢调用），你就把它拉黑了（Open→熔断）。一小时后你想'要不要再给它一次机会'（Half-Open），点了一份试探。如果还是慢，继续拉黑；如果正常了，把它从黑名单里放出来（Closed）。"

**小白**："那 Sentinel 的半开探测是怎么做的？放一个请求还是放一定比例？"

**大师**："放一个请求。探测请求通过后，熔断器从 Open 转 Closed，恢复正常调用。如果探测也超时/异常，重新计时熔断时长。"

**小胖**："那如果多个服务同时从熔断恢复怎么办？比如库存服务挂了，订单服务熔断、支付服务熔断——等库存恢复后，订单和支付同时发探测请求，会不会又把库存打挂？"

**大师**："这就是**雪崩恢复**问题。Sentinel 本身不支持跨资源的协调探测。你需要在上层做两个防护：第一，恢复顺序——先恢复非核心链路（如查询），再恢复核心链路（如支付）；第二，引入探测抖动（Jitter）——不要所有探测请求在同一秒发出，在 timeWindow 基础上加上一个随机延迟（如 +0~5 秒）。"

**小白**："那如果我想人为控制恢复顺序，比如手动触发 Half-Open，Sentinel 支持吗？"

**大师**："Sentinel 1.8 不提供 API 直接控制系统状态。但可以通过规则管理间接实现——当你准备好恢复时，把熔断时间 `timeWindow` 临时设为 1 秒，让熔断器自然进入 Half-Open。或者在自定义 DegradeSlot 中增加一个'半开信号'机制，由外部服务（监控平台）发出恢复指令。"

**小胖**："那我怎么知道选哪种策略？慢调用、异常比例、异常数，三个看起来差不多。"

**小白**："不一样。慢调用比例关注的是延迟（RT），适用于下游服务变慢但没挂的情况。异常比例关注的是错误率，适用于下游服务报错的情况。异常数适合调用量不大的场景，避免比例统计在低基数下不准确。"

**大师**："补充一个关键点：熔断的判断是基于统计窗口的，不是实时判断每个请求。比如你配了'10 秒内慢调用比例 > 50%'，Sentinel 不是看某个请求慢了就立即熔断，而是等统计窗口结束后看整体比例。"

**技术映射**：
- **熔断状态机**：Closed（正常调用）→ Open（拒绝调用，快速失败）→ Half-Open（放行一个探测请求）→ Closed（探测通过）/ Open（探测失败，重新计时）。
- **慢调用比例**（`GRADE_RT`）：统计时长内，慢调用（RT > maxAllowedRt）占比超过阈值时触发熔断。`slowRatioThreshold` 是比例阈值（0.0~1.0），`statIntervalMs` 是统计窗口（默认 1000ms），`minRequestAmount` 是最小请求数（防止低流量误熔断）。
- **异常比例**（`GRADE_EXCEPTION_RATIO`）：统计时长内，异常占比超过阈值（count 字段，0.0~1.0）触发熔断。
- **异常数**（`GRADE_EXCEPTION_COUNT`）：统计时长内，异常数量超过 count 值触发熔断。
- **timeWindow**：熔断时长（秒），从 Open 到 Half-Open 的等待时间。

### 3.1 环境准备

编写一个模拟库存服务——支持人为注入延迟和异常，用于触发熔断：

```java
@RestController
@RequestMapping("/inventory")
public class MockInventoryController {

    // 延迟开关：true 时模拟 2 秒延迟
    private volatile boolean slowMode = false;
    // 异常开关：true 时模拟 50% 概率抛异常
    private volatile boolean errorMode = false;

    @GetMapping("/query")
    public String queryStock(@RequestParam String skuId) throws InterruptedException {
        if (slowMode) {
            Thread.sleep(2000);  // 模拟 2 秒延迟
        }
        if (errorMode && Math.random() > 0.5) {
            throw new RuntimeException("库存服务暂时不可用");
        }
        return "库存(skuId=" + skuId + "): 剩余 100 件";
    }

    @PostMapping("/toggle-slow")
    public String toggleSlow(@RequestParam boolean on) {
        this.slowMode = on;
        return "slowMode = " + on;
    }

    @PostMapping("/toggle-error")
    public String toggleError(@RequestParam boolean on) {
        this.errorMode = on;
        return "errorMode = " + on;
    }
}
```

### 3.2 分步实现

**步骤一：配置慢调用比例熔断**

订单服务中调用库存的接口：

```java
@RestController
public class OrderWithCircuitBreaker {

    @Autowired
    private RestTemplate restTemplate;

    @GetMapping("/order/seckill")
    @SentinelResource(value = "seckillOrder", fallback = "seckillFallback")
    public String seckillOrder(@RequestParam String skuId) {
        // 调用库存服务
        String stock = restTemplate.getForObject(
            "http://localhost:8082/inventory/query?skuId=" + skuId, String.class);
        return "秒杀下单成功, " + stock;
    }

    public String seckillFallback(String skuId, Throwable t) {
        if (t instanceof DegradeException) {
            return "秒杀活动太火爆，请稍后再试（已熔断）";
        }
        return "下单失败: " + t.getMessage();
    }
}
```

配置慢调用熔断规则：

```java
@PostConstruct
public void initDegradeRules() {
    List<DegradeRule> rules = new ArrayList<>();

    // 慢调用比例熔断：
    // 统计窗口 10 秒内，如果请求数 >= 5 且慢调用(RT > 200ms) 占比 > 50%，
    // 触发熔断，持续 10 秒
    DegradeRule slowRule = new DegradeRule("seckillOrder")
            .setGrade(RuleConstant.DEGRADE_GRADE_RT)       // 慢调用比例
            .setCount(200)                                  // maxAllowedRt = 200ms
            .setSlowRatioThreshold(0.5)                     // 慢调用比例 > 50%
            .setMinRequestAmount(5)                         // 最小请求数
            .setStatIntervalMs(10000)                       // 统计窗口 10 秒
            .setTimeWindow(10);                             // 熔断时长 10 秒
    rules.add(slowRule);

    DegradeRuleManager.loadRules(rules);
}
```

**验证步骤**：

1. 正常调用：`curl http://localhost:8090/order/seckill?skuId=1001` → 快速返回。
2. 开启慢模式：`curl -X POST http://localhost:8082/inventory/toggle-slow?on=true`。
3. 用 JMeter 以 10 QPS 压测 `/order/seckill`，每个请求耗时约 2 秒 → 10 秒后（统计窗口结束），慢调用比例 100% > 50% → 触发熔断。
4. 熔断期间（10 秒内）继续请求 → 直接返回 fallback 结果，不再实际调用库存服务。
5. 熔断窗口结束 → 进入 Half-Open → 下一个请求尝试调用库存 → 如果仍然慢 → 重新熔断 10 秒。
6. 关闭慢模式后，半开探测通过 → 熔断关闭，恢复正常。

**步骤二：配置异常比例熔断**

```java
// 异常比例熔断：
// 统计窗口 10 秒内，如果请求数 >= 5 且异常比例 > 30%，触发熔断
DegradeRule exceptionRule = new DegradeRule("seckillOrder")
        .setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_RATIO)  // 异常比例
        .setCount(0.3)                                          // 异常比例 > 30%
        .setMinRequestAmount(5)
        .setStatIntervalMs(10000)

```

**验证**：关闭慢模式，开启异常模式（`errorMode=true`），压测 10 秒 → 异常比例约 50% > 30% → 触发熔断。

**步骤三：配置异常数熔断**

```java
// 异常数熔断：
// 统计窗口 10 秒内，异常数 > 5 即触发熔断
DegradeRule countRule = new DegradeRule("seckillOrder")
        .setGrade(RuleConstant.DEGRADE_GRADE_EXCEPTION_COUNT)   // 异常数
        .setCount(5)                                             // 异常数 > 5
        .setStatIntervalMs(10000)
        .setTimeWindow(10);
```

**验证**：低流量场景（QPS=2），异常比例可能误判（基数太小）。异常数模式更合适——只要有 5 次异常就熔断。

**步骤四：观察熔断日志**

在 `~/logs/csp/sentinel-record.log` 中可以看到熔断状态变化：

```log
2024-06-01 10:30:15 | [seckillOrder] degrade from CLOSED to OPEN, 
    rule: DegradeRule{grade=1, count=200.0, timeWindow=10}
2024-06-01 10:30:25 | [seckillOrder] degrade from OPEN to HALF_OPEN
2024-06-01 10:30:25 | [seckillOrder] HALF_OPEN probe passed, 
    degrade from HALF_OPEN to CLOSED
```

**踩坑记录**：

1. **熔断窗口内所有请求都被拦截**：包括正常请求。Half-Open 阶段只释放一个探测请求。
2. **统计窗口和熔断窗口是两个概念**：`statIntervalMs` 是统计窗口（收集数据），`timeWindow` 是熔断窗口（拒绝请求的时长）。如果设置 `statIntervalMs=10000` 和 `timeWindow=5`，可能导致统计还没出结果熔断已经结束了。
3. **MinRequestAmount 的重要性**：如果流量很低（如 QPS=1），设置异常比例 > 50%，可能一个异常就触发熔断（100% 异常比例）。设置合适的 minRequestAmount 可以避免这种误熔断。
4. **慢调用比例的 count 字段**：在慢调用模式下，`count` 的含义是 `maxAllowedRt`（最大允许的响应时间，毫秒），而不是比例值。比例值在 `slowRatioThreshold` 中。

**步骤五：编程方式观察熔断状态变化**

```java
@RestController
public class DegradeMonitorController {

    @GetMapping("/debug/degrade-status")
    public Map<String, Object> checkDegradeStatus(@RequestParam String resource) {
        Map<String, Object> result = new LinkedHashMap<>();

        List<DegradeRule> rules = DegradeRuleManager.getRulesOfResource(resource);
        for (DegradeRule rule : rules) {
            Map<String, Object> info = new LinkedHashMap<>();
            info.put("grade", rule.getGrade());           // 熔断策略
            info.put("count", rule.getCount());           // 阈值
            info.put("timeWindow", rule.getTimeWindow()); // 熔断时长
            info.put("minRequestAmount", rule.getMinRequestAmount());
            info.put("statIntervalMs", rule.getStatIntervalMs());

            // 获取当前熔断状态
            Optional<CircuitBreaker> cb = CircuitBreakerManager.getCircuitBreakers(resource)
                .stream().findFirst();
            cb.ifPresent(c -> {
                info.put("state", c.currentState().name());
                info.put("halfOpenProbeCount", c.getHalfOpenProbeCount());
                // 检查是否需要人工介入恢复
                if (c.currentState() == CircuitBreaker.State.OPEN && rule.getTimeWindow() > 60) {
                    info.put("suggestion", "熔断窗口过长(" + rule.getTimeWindow() + "s), 建议人工检查下游服务状态");
                }
            });

            result.put(rule.getResource(), info);
        }
        return result;
    }
}
```

## 4 项目总结

### 4.1 优点与缺点

| 熔断策略 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| 慢调用比例 | 下游服务 RT 变慢 | 早于异常触发，更快隔离故障 | 需要合理的 maxAllowedRt 基线 |
| 异常比例 | 下游服务报错 | 精准识别故障 | 低流量场景下统计不准确 |
| 异常数 | 调用量低但异常致命 | 不受基数影响 | 阈值依赖经验值 |

### 4.2 适用场景

- **慢调用比例**：下游数据库慢查询、Redis 大 Key 操作、第三方 API 抖动
- **异常比例**：下游服务发布中频繁重启、网络闪断导致偶发超时
- **异常数**：对异常零容忍的金融/支付场景，哪怕 QPS 很低，一个问题就应触发熔断
- **不适用**：业务本身的降级逻辑（如查询不到数据返回空列表不算异常）；对延迟波动不敏感但需要 100% 可靠性的场景（熔断期本身不可用）

### 4.3 注意事项

1. 熔断器与限流器是独立的：一个资源可以同时有限流规则和熔断规则。
2. `fallback` 可以同时捕获 BlockException 和业务异常，但建议用 blockHandler 专门处理限流。
3. 熔断恢复不是瞬时完成的——需要经过 Half-Open 探测，总恢复时间 = timeWindow + 探测请求耗时。
4. 警惕"雪崩恢复"——多个服务同时从熔断恢复可能导致二次雪崩。建议在集群层面协调半开时机。

### 4.4 熔断策略选型决策表

| 判断条件 | 推荐策略 | 原因 |
|---------|---------|------|
| 下游服务 RT 波动大，但很少直接报错 | 慢调用比例 | 在真正报错前隔离慢调用，减少级联影响 |
| 下游服务偶尔返回 5xx 错误 | 异常比例 | 基于错误率做判断，容忍偶发异常 |
| 调用量很低（< 5 QPS）但每次调用很关键 | 异常数 | 低基数下比例统计不可靠，异常数更精确 |
| 调用量很高（> 100 QPS） | 慢调用比例或异常比例 | 高基数下比例统计稳定 |
| 需要尽可能不丢请求（如支付确认） | 不适用熔断 | 改用线程数限流 + 重试 + 补偿 |
| 下游是缓存（Redis）且有过期机制 | 异常数 | Redis 偶发连接超时可快速熔断，避免请求堆积 |

### 4.5 常见踩坑经验

| 故障现象 | 根因 | 解决方案 |
|---------|------|---------|
| 熔断后一直不恢复 | Half-Open 探测请求也被拦截 | 确保探测请求计数在统计窗口之外 |
| 正常流量也被熔断 | minRequestAmount 太低，偶发慢调用误触发 | 调高 minRequestAmount，使用更长的统计窗口 |
| 熔断窗口结束后没有探测请求 | 没有新流量进来 | 手动发一个测试请求触发 Half-Open |
| 多个服务同时熔断恢复导致二次雪崩 | 恢复时机同步 | 引入随机抖动（Jitter）或手动控制恢复顺序 |

### 4.6 思考题

1. 如果慢调用比例熔断规则设置了 `statIntervalMs=5000` 且 `timeWindow=10`，在触发熔断后的第 3 秒（还在 Open 状态），Sentinel 怎么判断是否应该进入 Half-Open？
2. 一个资源同时配置了慢调用比例熔断和异常比例熔断。如果在同一时间段内，慢调用比例为 60%（触发熔断），异常比例为 10%（未触发熔断），最终状态是什么？哪个规则优先生效？

### 4.7 推广计划

- **开发团队**：为每个关键的远程调用（Feign/Dubbo/RestTemplate）配置熔断规则，并设定合理的 maxAllowedRt（基于接口 P95 延迟）。
- **测试团队**：设计熔断恢复测试用例——注入故障 → 验证熔断触发 → 等待恢复 → 验证探测定时 → 验证恢复后流量。
- **运维团队**：在监控大盘中加入熔断状态面板（Open/Half-Open/Closed），配置"熔断持续超过 5 分钟"告警。
