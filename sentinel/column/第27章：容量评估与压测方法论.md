# 第27章：容量评估与压测方法论

## 1 项目背景

前面各章我们从"拍脑袋阈值"过渡到了"Nacos 动态规则"，但还有一个核心问题没有解决：**阈值的数字到底是怎么算出来的？**

团队中发生过这样一次对话——运维问开发："createOrder 的流控阈值为什么是 500？"，开发回答："经验值，上次压测时 600 就崩了，所以我设了 500。"运维追问："那次压测是单机 4C8G 的物理机，现在线上是 2C4G 的容器——你的 500 还适用吗？"开发沉默了。

这种"经验主义"的阈值设定在微服务环境下非常危险。服务规模扩张后（从 3 个到 30 个服务），没有人能凭经验给所有接口配置合理的阈值。生产环境的配置漂移（硬件不同、网络延迟不同、依赖链路不同）使得"一个阈值到处抄"成为故障的定时炸弹。

本章将建立一套系统化的 Sentinel 容量评估方法论：从**基准压测**到**阶梯压测**，从**容量建模**到**阈值推导**，让 Sentinel 的规则从"拍脑袋"走向"数据驱动"。

## 2 项目设计

**小胖**："大师，上次压测是在 4C8G 的机器上，现在生产是 2C4G 的容器——阈值要不要减半？"

**大师**："不能简单线性减半。CPU 减半了，但内存、网络带宽、IOPS 都变了。你需要重新做一次阶梯压测来找出新的拐点。"

**小白**："阶梯压测是怎么做的？从低 QPS 开始，逐步加压直到崩溃？"

**大师**："对。典型的流程是：从 50 QPS 开始，每 30 秒增加 50 QPS，观察 RT、CPU、错误率的变化。找到 RT 开始明显上升的点（如从 50ms 升到 80ms），那就是**性能拐点**。流控阈值取拐点 QPS 的 70-85%。"

**小胖**："那混合压测怎么搞？实际线上业务可不是只有一个接口在被调用。我之前只压了 createOrder，结果 queryOrder 和 createOrder 一起跑的时候，数据库连接池被打满了，createOrder 的拐点从 400 降到了 200。"

**大师**："这就是为什么要有**混合压测场景**。你需要先梳理核心调用链路，把同一链路上的接口按真实比例混合。比如订单链路上：createOrder 占 30%、queryOrder 占 50%、cancelOrder 占 20%。混合压测的拐点才是你配阈值的真实依据。"

**小白**："那稳定性压测的作用是什么？阶梯压测不是已经找到拐点了吗？"

**大师**："阶梯压测找的是'短时间内的极限'，但生产上很多问题是'慢性病'——比如内存泄漏、线程池逐渐堆积、GC 逐渐恶化。稳定性压测就是用拐点的 80% 持续跑 30 分钟以上，看性能指标是否漂移。如果 30 分钟内 RT 从 60ms 慢慢爬到 200ms，说明系统有隐藏问题。"

**小胖**："大师，我们线上用的是 K8s 容器，每次部署 CPU Limit 都可能被运维调整。压测一次的成本太高了，有没有快速估算的方法？"

**大师**："有近似公式，但只能作为参考。对于 CPU 密集型接口：`QPS ≈ (CPU核数 × 1000) / 单次请求CPU耗时(ms)`。对于 IO 密集型接口：`QPS ≈ 数据库连接池大小 / (平均RT/1000)`。记住，这只是估算，上线前必须做至少一轮基准压测验证。"

**小胖**："那混合压测的真实流量比例怎么拿？运维能提供 Nginx 日志分析吗？"

**大师**："可以。从 Nginx 日志或网关日志中，按 URI 分组统计过去 7 天的请求比例。比如 `/order/create` 30%、`/order/query` 50%、`/order/cancel` 20%。JMeter 的 Throughput Controller 可以按这个比例分配到不同线程组。还有一个更简单的方法：录线上流量 → 回放压测。"

**小白**："流量回放比混合压测更真实，但回放工具怎么选？GoReplay、tcpcopy、还是自研？"

**大师**："GoReplay 最简单，一条命令 `gor --input-raw :8090 --output-http http://test-env:8090` 就能回放。但注意：回放流量里的 userId、skuId 等参数可能和测试环境的数据不匹配，需要做好数据适配和幂等性处理。而且回放不能放大流量——如果线上是 100 QPS，你想测 500 QPS，光回放不够，还需要 JMeter 补充。"

## 3 项目实战

### 3.1 压测类型与方法

| 压测类型 | 目的 | 方法 | 时长 |
|---------|------|------|------|
| 基准压测 | 找出单接口性能天花板 | 固定并发，逐步增加 | 5-10 分钟 |
| 阶梯压测 | 找出性能拐点 | QPS 从低到高阶梯递增 | 10-20 分钟 |
| 稳定性压测 | 验证长期运行稳定性 | 固定拐点 80% 的 QPS | 30-60 分钟 |
| 突刺压测 | 验证突发流量处理 | 瞬间 3-5 倍峰值的流量脉冲 | 1-2 分钟 |

### 3.2 阶梯压测实战

JMeter 阶梯压测配置：

```text
Thread Group: Stepping Thread Group
  - Start threads: 10
  - Initial delay: 0 sec
  - Start threads count: 50
  - Next, add: 50 threads every 30 seconds
  - Hold load for: 60 seconds after peak
  - Total duration: 300 seconds
```

压测结果整理：

| QPS | CPU% | RT(avg) | RT(P99) | 错误率 | 备注 |
|-----|------|---------|---------|-------|------|
| 100 | 15 | 45ms | 80ms | 0% | 正常 |
| 200 | 25 | 48ms | 85ms | 0% | 正常 |
| 300 | 40 | 52ms | 90ms | 0% | 正常 |
| 400 | 55 | 60ms | 110ms | 0% | ⚠️ 拐点 |
| 500 | 75 | 85ms | 180ms | 0.1% | 🔴 接近极限 |
| 600 | 90 | 150ms | 500ms | 2% | ❌ 超载 |
| 700 | 95 | 2000ms | 5000ms | 15% | ❌ 崩溃 |

结论：拐点 QPS=400，流控阈值建议=320（80%）

### 3.3 三类规则的阈值推导公式

```
流控阈值 QPS = 拐点 QPS × 安全系数（0.7-0.85）
线程数阈值 = 拐点 QPS × 平均 RT(秒) × 安全系数
熔断慢调用阈值 = P99 RT × 1.5-2.0
系统保护 CPU = 容器 CPU Limit × 0.8
```

生产校验：压测环境到生产环境换算系数：

```
生产阈值 = 压测阈值 × (生产 CPU 核数 / 压测 CPU 核数) × (生产内存 / 压测内存) × 环境系数(0.8)
```

### 3.4 压测工具脚本

`wrk` 快速压测脚本：

```bash
#!/bin/bash
# quick-bench.sh — 快速性能基准测试

URL="http://localhost:8090/order/create?skuId=TEST&userId=BENCH"

for qps in 50 100 200 300 400 500 600; do
  echo "=== 压测 QPS=$qps ==="
  wrk -t4 -c$((qps/10)) -d30s --latency "$URL" \
    | grep -E "Requests/sec|Latency.*99%"
  sleep 5
done
```

### 3.5 容量建模与阈值推导实战

通过阶梯压测数据，建立接口的"容量画像"：

```java
@Service
public class CapacityProfileService {

    @Data
    public static class CapacityProfile {
        private String resourceName;
        private int kneePointQps;      // 拐点 QPS
        private double safeThreshold;  // 安全阈值（拐点 × 80%）
        private double avgRtAtKnee;    // 拐点时的平均 RT
        private double p99RtAtKnee;    // 拐点时的 P99 RT
        private double cpuAtKnee;      // 拐点时的 CPU 使用率
        private boolean isCpuBound;    // 是否 CPU 密集型
    }

    public CapacityProfile buildProfile(String resourceName) {
        // 从压测报告中提取数据
        CapacityProfile profile = new CapacityProfile();
        profile.setResourceName(resourceName);
        profile.setKneePointQps(400);
        profile.setSafeThreshold(400 * 0.8);   // 320
        profile.setAvgRtAtKnee(60.0);
        profile.setP99RtAtKnee(110.0);
        profile.setCpuAtKnee(55.0);
        profile.setCpuBound(cpuAtKnee > 70.0);
        return profile;
    }

    // 根据容量画像自动生成 Sentinel 规则
    public FlowRule generateFlowRule(CapacityProfile profile) {
        FlowRule rule = new FlowRule(profile.getResourceName());
        if (profile.isCpuBound()) {
            rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
            rule.setCount(profile.getSafeThreshold());
        } else {
            // IO 密集型用线程数限流
            rule.setGrade(RuleConstant.FLOW_GRADE_THREAD);
            double threads = profile.getSafeThreshold() * (profile.getAvgRtAtKnee() / 1000.0);
            rule.setCount(threads);
        }
        return rule;
    }
}
```

### 3.6 生产环境与压测环境的换算验证

在实际部署中，压测环境与生产环境的性能差异必须量化：

```bash
#!/bin/bash
# calibrate-env.sh — 环境性能差异标定脚本

# 在同一接口、相同 QPS（如 200）下，对比压测环境和生产环境的关键指标
# 输出环境换算系数

PRESSURE_RT=$(curl -s "http://pressure-env:8090/bench/rt?qps=200")
PROD_RT=$(curl -s "http://prod-env:8090/bench/rt?qps=200")

echo "压测环境 RT: ${PRESSURE_RT}ms"
echo "生产环境 RT: ${PROD_RT}ms"
echo "环境系数: $(echo "scale=2; $PROD_RT / $PRESSURE_RT" | bc)"

# 应用：生产阈值 = 压测阈值 × 环境系数 × 0.8(安全系数)
# 例如：压测拐点 400 × 1.2(环境系数) × 0.8(安全系数) = 384 QPS
```

### 3.7 自动化压测流水线

将压测集成到 CI/CD 流程中：

```yaml
# .github/workflows/benchmark.yml
name: Sentinel Benchmark

on:
  push:
    branches: [main]
    paths: ['order-service/**']

jobs:
  benchmark:
    runs-on: self-hosted
    steps:
      - name: Deploy to staging
        run: kubectl rollout restart deployment/order-service -n staging
      - name: Wait for ready
        run: kubectl wait --for=condition=ready pod -l app=order-service -n staging --timeout=120s
      - name: Run stair-step benchmark
        run: |
          ./scripts/stair-bench.sh --url http://staging:8090/order/create --start 50 --step 50 --duration 30
      - name: Generate report
        run: ./scripts/capacity-report.py --output capacity-report.md
      - name: Update rules if degraded
        run: |
          if grep -q "DEGRADED" capacity-report.md; then
            echo "性能劣化，阻止上线" && exit 1
          fi
```

**踩坑记录**：

1. **压测环境 ≠ 生产环境**：压测环境的数据库、缓存、网络延迟与生产不同，阈值需要乘以环境系数。
2. **冷启动效应**：服务刚启动时的 JIT 未编译完成，前 1-2 分钟的压测数据不可用。
3. **单接口压测 vs 混合压测**：单独压一个接口的结果不代表混合场景下的能力。混合压测时各接口会竞争资源。
4. **压测数据隔离**：压测请求必须携带特殊 Header（如 `X-Benchmark: true`），确保压测数据不污染生产监控指标。
5. **GC 干扰**：G1GC 的 Mixed GC 可能导致压测期间的 RT 突刺。建议在压测前手动触发一次 Full GC，并记录 GC 日志与压测数据的时序对应关系。

**步骤三.8：压测报告自动化生成**

通过压测结果数据自动生成 Sentinel 规则建议：

```python
#!/usr/bin/env python3
# generate-rule-from-bench.py

import json
import sys

def generate_flow_rule(resource, knee_qps, avg_rt, cpu_percent, is_cpu_bound):
    rule = {
        "resource": resource,
        "limitApp": "default",
        "grade": 1 if is_cpu_bound else 0,  # QPS or Thread
        "count": round(knee_qps * 0.75),
        "strategy": 0,
        "controlBehavior": 0
    }
    return rule

def generate_degrade_rule(resource, p99_rt, knee_qps):
    return {
        "resource": resource,
        "grade": 0,  # RT
        "count": round(p99_rt * 1.5),
        "timeWindow": 10,
        "minRequestAmount": max(5, round(knee_qps * 0.1)),
        "slowRatioThreshold": 0.5,
        "statIntervalMs": 10000
    }

# 输入: JSON 格式的压测结果
# 输出: Nacos 可用的 Sentinel 规则 JSON
bench_data = json.load(sys.stdin)
flow_rules = []
degrade_rules = []

for result in bench_data["results"]:
    flow_rules.append(generate_flow_rule(**result))
    degrade_rules.append(generate_degrade_rule(**result))

print("=== Flot Rules (Nacos) ===")
print(json.dumps(flow_rules, indent=2, ensure_ascii=False))
print("\n=== Degrade Rules (Nacos) ===")
print(json.dumps(degrade_rules, indent=2, ensure_ascii=False))
```

## 4 项目总结

### 4.1 阈值设定决策树

```
你的接口是 CPU 密集型还是 IO 密集型？
├── CPU 密集型 → QPS 限流
│   └── 阈值 = 阶梯压测拐点 × 75%
└── IO 密集型 → 线程数限流
    └── 阈值 = 数据库连接池大小 × 80%

你的下游依赖稳定吗？
├── 稳定 → 不配熔断，或配很宽松的熔断（慢调用比例 80%）
└── 不稳定 → 配严格的熔断（慢调用比例 30% + 异常比例 20%）

你的部署方式是怎样的？
├── 单机 → 单机限流
├── 多副本固定 → 集群流控（全局阈值）
└── 弹性伸缩 → 系统保护 + HPA（阈值随副本数动态调整）
```

### 4.2 压测执行检查清单

- [ ] 确认压测环境硬件规格（CPU/内存/磁盘）与生产对齐或标定换算系数
- [ ] 压测前预热服务 2 分钟（触发 JIT 编译）
- [ ] 阶梯压测从 50 QPS 起步，每 30 秒增加 50 QPS
- [ ] 同时监控：QPS、RT(P50/P99)、CPU、内存、GC、错误率
- [ ] 找到 RT 明显上升的拐点，记录拐点 QPS
- [ ] 混合压测按真实流量比例（createOrder 30% + queryOrder 50% + cancelOrder 20%）
- [ ] 稳定性压测 ≥ 30 分钟，观察指标是否漂移
- [ ] 突刺压测：瞬间 3 倍峰值脉冲，验证 Sentinel 是否及时拦截
- [ ] 输出容量画像报告，自动生成 Sentinel 规则建议

### 4.3 各压测类型产出物

| 压测类型 | 产出物 | 用途 |
|---------|-------|------|
| 基准压测 | 单接口拐点 QPS | 确定流控阈值基准 |
| 阶梯压测 | 性能拐点 + RT 曲线 | 推导安全阈值 |
| 混合压测 | 真实容量上限 | 修正单接口阈值的偏差 |
| 稳定性压测 | 长时间 RT 漂移曲线 | 发现隐藏的内存/连接池泄漏 |
| 突刺压测 | 限流响应时效 | 验证 Sentinel 规则是否够快 |

### 4.4 压测环境与生产环境的差异补偿

| 差异维度 | 典型差距 | 补偿方法 |
|---------|---------|---------|
| CPU 核数 | 压测 4C vs 生产 2C | 阈值 × 0.5 |
| 内存 | 压测 8G vs 生产 4G | 观察 GC 频率差异 |
| 数据库连接池 | 压测 50 vs 生产 200 | IO 密集型不能线性推导 |
| 网络延迟 | 压测 0.1ms vs 生产 1ms | RT 差值加到 maxAllowedRt |
| 数据量 | 压测 10 万条 vs 生产 1000 万条 | 影响 DB 索引效率 |

### 4.5 压测常见错误及纠正

| 错误做法 | 为什么错 | 正确做法 |
|---------|---------|---------|
| 只用 1 个线程压测 | 无法测试并发竞争 | 从 10 线程起步，逐步增加 |
| 压测 10 秒就得出结论 | JIT 未完全编译 | 至少跑 2 分钟后取数据 |
| 只看平均 RT | 隐藏了长尾延迟 | 必看 P99 / P999 |
| 压测机和被压测服务在同一台机器 | 资源竞争导致数据失真 | 独立压测节点 |
| 不看 GC 日志 | GC 暂停影响 RT 分布 | 压测期间同时采集 GC 日志 |

### 4.4 思考题

1. 你对订单服务做了阶梯压测，发现拐点是 400 QPS。但你只压测了下单接口（createOrder），没有同时压测查询接口（queryOrder）。如果两个接口同时被压测，拐点会怎样变化？如何修正阈值？
2. 生产环境的流量模型不是恒定的——有的接口白天 500 QPS、晚上 50 QPS。你如何为这种"双峰"流量设计 Sentinel 规则？

### 4.3 推广计划

- **测试团队**：将阶梯压测流程标准化，输出每个服务的拐点报告和阈值建议。
- **开发团队**：在服务上线前完成基准压测，并更新 Sentinel 规则。
- **运维团队**：建立生产环境与压测环境的性能转换系数，在监控中设置基于实际容量的告警。
