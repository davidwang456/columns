# 第40章：【高级篇综合实战】从源码扩展到生产级流量防护平台

## 1 项目背景

高级篇（第 32-39 章）我们深入了 Sentinel 源码，掌握了 Slot Chain、滑动窗口、熔断状态机、SPI 扩展、高并发调优、SRE 落地。现在是时候把所有能力整合起来，交付一个**从源码扩展起的生产级流量防护平台**。

本章以一个金融支付系统为场景，要求实现：自定义业务标签 Slot 用于风控标记、基于数据库的规则数据源、自定义 JSON 异常输出、规则审批与一键回滚、完整的可观测性体系、SRE 应急预案。

## 2 项目设计

**大师**："高级篇的综合实战是你们整个专栏学习的最终交付。验收标准我定了 6 条：
1. 核心链路（支付 → 风控 → 清算）P99 RT 稳定在 100ms 以内
2. 所有 Sentinel 规则变更可审计（谁、什么时间、改了什么）
3. 故障 5 分钟内定位（通过 traceId 关联到 Sentinel 事件）
4. 规则 1 分钟内回滚（通过治理平台一键操作）
5. 自定义 Slot 扩展正常运行，不影响核心链路
6. 完整监控面板 + 告警 + 值班手册"

**小胖**："这个要求...还挺高。不过前 39 章学下来，应该都能搞定。"

**小胖**："大师，生产级平台的性能基准是多少？Sentinel 全链路（9 个 Slot + 自定义 Slot）的 overhead 大概多少？"

**大师**："根据 Sentinel 社区的基准测试和阿里内部实践，9 个内置 Slot 的纯遍历开销约 1-2 μs。加上一个轻量的自定义 Slot（无 IO 操作），总开销控制在 3 μs 以内。对于 P99 RT 在 10ms 以上的接口，这个开销可以忽略。但如果你的接口 P99 RT 在 1ms 以内——比如高性能网关、金融交易核心链路——那就要评估是否每个请求都走完整 Slot Chain。"

**小白**："那监控面板这六大模块有没有推荐的 Grafana Dashboard 模板？从零开始画太费时间了。"

**大师**："Sentinel 社区有一些现成的模板可以参考。流量总览用 Graph 面板 + QPS/RT 双 Y 轴，熔断状态用 State Timeline 面板，热点 TopN 用 Bar Gauge。关键指标：
- `sentinel_pass_qps_total{resource="*"}` 
- `sentinel_block_qps_total{resource="*"}`
- `sentinel_degrade_open_count{resource="*"}`
- `sentinel_param_flow_cache_size`
如果你用了 Prometheus exporter（第 12 章讲过），这些指标会自动暴露。"

**小胖**："最后一个问题——这套平台上线后，最可能出问题的环节是什么？我好提前准备应急预案。"

**大师**："最容易出问题的是三个环节：
1. **MySQL 规则存储**——如果数据库慢查询导致规则加载超时。预防：客户端必须有本地规则快照兜底。
2. **Nacos 推送延迟**——大流量突发时，规则更新不够快。预防：系统保护规则作为硬兜底。
3. **Token Server 单点**——如果 Token Server 挂了，集群流控退化到单机模式。预防：部署 3 节点 Raft 集群，配合 `fallbackToLocalWhenFail`。"

**大师**："对了，小胖——SRE 值班这块你们打算怎么排？"

**小胖**："还没想好... 一周一轮？"

**大师**："建议按核心链路分责任田。支付链路一个 Owner，风控链路一个 Owner，清算链路一个 Owner。每个 Owner 对自己负责的链路做巡检：每天早上 9 点核对前一天的 block 率、熔断次数、规则变更记录。SRE 的巡检不是等告警，而是**主动发现异常趋势**。"

**小白**："那如果负责人休假怎么办？"

**大师**："建立 Backup 机制。每个链路至少两个 Owner（主+备），值班表提前两周排定。哨兵监控面板给所有 Owner 开放推送通知——Grafana Alert 同时发给主备和值班经理。核心原则：告警 3 分钟内有人响应，5 分钟内人到位。"

### 3.2 核心交付物清单

#### 1. 源码扩展

| 扩展组件 | 功能 | 验收标准 |
|---------|------|---------|
| `BizTagSlot` | 业务标签传播与风控标记 | 与 FlowSlot 协同，不影响 P99 RT |
| `JdbcDataSource` | MySQL 规则持久化 | 支持定时拉取 + 推送更新 |
| `UnifiedBlockExceptionHandler` | 统一 JSON 错误码输出 | 含 suggestion 字段 |
| `OtelSentinelExporter` | OpenTelemetry 事件导出 | Span 上可看到 sentinel.blocked 事件 |

#### 2. 规则治理平台

| 功能 | 实现方式 | 验收标准 |
|------|---------|---------|
| 规则审批 | Git PR → 审批 → CI/CD 推送到 Nacos | 审批记录可查询 |
| 一键回滚 | 治理平台调用 Nacos API 回滚到指定版本 | < 5 秒完成 |
| 审计日志 | MySQL 记录所有规则变更 | 包含操作人、时间、新旧值对比 |
| 冲突检测 | 新规则提交时自动检测与已有规则的冲突 | 重复资源名 + 相同策略告警 |

#### 3. 可观测性

| 面板 | 指标 | 告警 |
|------|------|------|
| 流量总览 | pass QPS, block QPS, RT | block 率 > 50% 告警 |
| 熔断状态 | 各资源熔断状态 (OPEN/CLOSED) | OPEN > 5min 告警 |
| 热点 TopN | Top 10 热点参数及其 QPS | 热点 QPS 突增告警 |
| Token Server | 令牌发放率、延迟 | Server 离线告警 |
| 规则变更 | 变更时间线 | 变更通知（钉钉/企微） |

#### 4. SRE 预案

```markdown
## 支付系统 Sentinel 应急预案 v2.0

### 故障场景 1: 风控服务响应变慢
1. 现象：支付服务 RT 从 50ms 升到 2000ms
2. 判断：Grafana → Degrade State OPEN
3. 操作：确认风控团队在处理 → 如 2 分钟未恢复，临时调低熔断阈值
4. 回滚：风控服务恢复后，恢复熔断规则

### 故障场景 2: 清算服务全熔断
1. 现象：所有清算请求被 Sentinel 熔断
2. 判断：Grafana → Degrade State OPEN × 3
3. 操作：启动清算异步队列，通知业务方交易将延迟入账
4. 回滚：清算服务恢复后，消费异步队列中的积压订单

### 故障场景 3: 数据库规则源不可用
1. 现象：Sentinel 客户端日志报告 JDBC 连接失败
2. 判断：检查 MySQL 连接池状态
3. 操作：自动降级到本地缓存的规则快照（< 1 分钟前）
4. 回滚：MySQL 恢复后自动切回
```

### 3.4 生产级健康检查端点

```java
@RestController
@RequestMapping("/health/sentinel")
public class SentinelHealthController {

    @GetMapping
    public Map<String, Object> healthCheck() {
        Map<String, Object> result = new LinkedHashMap<>();

        // 1. Slot Chain 完整性检查
        int slotCount = SlotChainProvider.getSlotChain("__health__").size();
        result.put("slotChain", slotCount >= 9 ? "OK (" + slotCount + " slots)" : "DEGRADED");

        // 2. 规则数据源连通性检查
        try {
            List<FlowRule> rules = FlowRuleManager.getRules();
            result.put("flowRules", "OK (" + rules.size() + " rules)");
        } catch (Exception e) {
            result.put("flowRules", "FAIL: " + e.getMessage());
        }

        // 3. 客户端心跳状态检查
        result.put("dashboardHeartbeat", TransportConfig.isHeartbeatRunning() ? "OK" : "STALE");

        // 4. 本地规则快照存在性检查
        File snapshot = new File(System.getProperty("user.home"), "sentinel/rules/snapshot.json");
        result.put("ruleSnapshot", snapshot.exists() ? "OK" : "WARN: no snapshot");

        // 5. 集群 Token Server 连通性检查（如有）
        result.put("tokenServer", ClusterStateManager.isAvailable() ? "OK" : "STANDALONE");

        boolean healthy = result.values().stream().noneMatch(v -> v.toString().contains("FAIL"));
        result.put("overall", healthy ? "HEALTHY" : "DEGRADED");
        return result;
    }
}
```
