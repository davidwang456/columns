# 第38章：极端高并发调优与低延迟实践

## 1 项目背景

前 37 章我们搭建了完整的 Sentinel 防护体系，但在某次大促压测中，订单服务的目标是 10 万 QPS。当 QPS 超过 5 万时，Sentinel 自身的统计开销开始显现——P99 RT 中出现了一个 1-3ms 的"Sentinel 开销峰"。虽然绝对值不大，但对于延迟敏感的支付链路来说，这是不可接受的。

更严重的是，随着热点参数增多（500 个 SKU 各配了独立的 ParamFlowRule），`ParamFlowSlot` 的 `entry()` 耗时从 0.01ms 增加到了 0.5ms——500 条规则逐条匹配，时间复杂度从 O(1) 变成了 O(n)。

高并发场景下，Sentinel 的性能瓶颈主要在三处：**Slot Chain 的遍历开销**、**LeapArray 的 CAS 竞争**、**规则数量的线性匹配**。本章将从这三个维度切入，给出可落地的调优方案。

## 2 项目设计

**小白**："10 万 QPS 下，Sentinel 自身的开销大概多少？"

**大师**："根据 Sentinel 官方 benchmark，一个完整的 Slot Chain 调用（9 个 Slot）的开销在 1-3 微秒级别。在这个量级下，性能瓶颈往往不是 Sentinel 本身，而是规则太多、Node 太多、热点参数太多。"

**小白**："那如何在不停机的情况下评估当前 Sentinel 的开销？生产环境不敢随便上 profiler。"

**大师**："可以通过 JMX 或 Micrometer 暴露 Sentinel 的内部耗时指标。你也可以用一个更轻量的方式：在生产流量中抽样 1%，在 entry 前后打时间戳，统计 Sentinel 自身开销的 P99。如果 P99 超过 100 微秒，就需要排查了。"

**小胖**："大师，你说的 LeapArray 的 CAS 竞争是什么？我理解 LeapArray 是 Sentinel 滑动窗口的数据结构。"

**大师**："LeapArray 内部使用 `AtomicReferenceArray<WindowWrap<T>>` 来存储窗口数据。在高并发下，多个线程同时调用 `currentWindow()` 获取当前时间窗口时，会通过 CAS 循环竞争更新窗口引用。QPS 超过 5 万后，这个 CAS 自旋的 CPU 开销就不可忽视了。优化方向有两个：一是扩大窗口长度（默认 500ms 改为 1000ms），减少窗口切换频率；二是升级到 jdk21 的虚拟线程，降低上下文切换成本。"

**小白**："那 ParamFlowSlot 的规则匹配 O(n) 问题怎么优化？500 条规则逐条匹配确实太慢了。"

**大师**："有几个优化思路。第一，规则分组——把同一资源的规则放在一起，先根据资源名做 O(1) 的 Map 查找，再在小组内匹配。第二，规则排序——把命中率高的规则排在前面，利用短路原则提前返回。第三，最根本的——削减规则数量。500 条 ParamFlowRule 说明你的规则治理出了问题，应该合并同类项。"

**小胖**："我们压测发现一个问题：高 QPS 下 Sentinel 的 metric 日志写入成为瓶颈。每秒几百条日志写磁盘，IO wait 明显上升。"

**大师**："Metrics 日志是 Sentinel 的'阿喀琉斯之踵'。10 万 QPS、1000 个资源时，metric 日志每秒约 1000 行。解决方案：把 metric 日志目录放到 tmpfs（内存文件系统），或者增加 `metric.flush.interval` 到 5 秒以降低写入频率。终极方案是用 Micrometer 替代 Sentinel 内置的 metric 日志，直接写 Prometheus Remote Write。"

**小白**："JDK 21 的虚拟线程出来了，Sentinel 在虚拟线程环境下表现如何？会省掉线程数的开销吗？"

**大师**："虚拟线程确实降低了线程资源开销，但对 Sentinel 的影响是双面的。好的一面是：Tomcat/Dubbo 线程池不再那么紧张，线程数限流的必要性降低了。坏的一面是：Sentinel 内部的 `ThreadLocal` 存储上下文（`ContextUtil.getContext()`），虚拟线程的"挂载/卸载"会导致 ThreadLocal 上下文丢失。解决方案：使用 Sentinel 1.8.6+ 的 `TransmittableThreadLocal` 支持，或者改用 `SphU.asyncEntry()` 显式传递上下文。"

**小胖**："我们准备上 GraalVM Native Image 来降低启动时间和内存占用。Sentinel 支持吗？"

**大师**："Sentinel 1.8.7 开始提供 GraalVM 支持，但有几个限制要注意：1) 动态反射的类需要提前在 `reflect-config.json` 中注册；2) Nacos DataSource 的动态加载需要额外处理；3) Sentinel Dashboard 通信的心跳机制不受影响。内存从 512MB 降到 64MB——这对 Serverless/K8s 场景诱惑很大。但代价是规则不能动态加载（AOT 编译），需要通过配置文件预定义。"

**小白**："还有一个场景：突发流量下'冷启动'问题。每天早上服务刚启动，Sentinel 的滑动窗口统计还是空的，此时如果有流量突增，规则来不及生效怎么办？"

**大师**："这就是'统计冷启动'问题。两种对策：一是设置 `warmUpPeriodSec`（预热时间），让 Sentinel 在预热期间用更低的阈值逐步放量。二是通过 `SentinelConfig` 设置初始统计窗口的默认值——但这在社区版还不支持，需要自定义实现一个 `StatisticNode` 的预热策略。最务实的方案是：在 K8s Pod 启动后、流量接入前，先发 10-20 秒的低流量预热请求，让统计窗口建立起来。"

**小胖**："如果在线程数限流中，Sentinel 把 RPC 调用线程和 Health Check 线程混在一起限流了怎么办？"

**大师**："Sentinel 的线程数限流默认统计的是 `Thread.currentThread()` 数量。但 Sentinel 的 `ContextUtil.enter()` 会绑定入口上下文，你可以自定义 `RequestOriginParser` 来区分"业务请求"和"健康检查"——不同来源走不同的限流规则。具体做法是：在健康检查接口上设置特殊的 `origin`（比如 `health-check`），配一条不限流的规则。"

### 3.8 规则缓存预热与批量匹配优化

```java
// 优化前：O(n) 线性匹配 500 条规则
// ParamFlowSlot.checkFlow() → 遍历所有 ParamFlowRule

// 优化后：按资源名分组，O(1) 查找 + O(m) 小范围匹配
public class OptimizedParamFlowChecker {

    private final Map<String, List<ParamFlowRule>> ruleIndex = new ConcurrentHashMap<>();

    public void rebuildIndex() {
        Map<String, List<ParamFlowRule>> newIndex = new HashMap<>();
        for (ParamFlowRule rule : ParamFlowRuleManager.getRules()) {
            newIndex.computeIfAbsent(rule.getResource(), k -> new ArrayList<>())
                    .add(rule);
        }
        // 每组内按命中率排序：热点规则在前
        newIndex.values().forEach(list ->
            list.sort((a, b) -> Long.compare(
                b.getMetrics().getPassQps(),  // 降序：高 QPS 在前
                a.getMetrics().getPassQps())));
        ruleIndex.clear();
        ruleIndex.putAll(newIndex);
    }

    public ParamFlowRule matchRule(String resource, Object param) {
        List<ParamFlowRule> candidates = ruleIndex.get(resource);
        if (candidates == null) return null;
        for (ParamFlowRule rule : candidates) {
            if (matches(rule, param)) return rule; // 短路返回
        }
        return null;
    }
}
```

**踩坑记录**：

1. **async-profiler 本身有开销**：生产环境建议用 `-d 10`（10 秒采样）而非长时间运行。
2. **指标日志磁盘 IO**：10 万 QPS、1000 个资源时，metric 日志每秒写入多达 1000 行。确保日志磁盘是 SSD。
3. **Dashboard 拉取开销**：Dashboard 每秒拉取一次客户端指标，如果客户端有 1000 个资源，每次拉取的数据量可达 50KB+。
4. **虚拟线程 + ThreadLocal 丢失**：JDK 21 虚拟线程环境下，Sentinel 的 `ContextUtil.getContext()` 在 carrier 线程切换时会丢失上下文。解决方案：升级到 Sentinel 1.8.6+，或使用 `SphU.asyncEntry()` 显式管理上下文。
5. **LeapArray 窗口长度调大的副作用**：窗口从 500ms 改为 2000ms 后，QPS 统计精度从"每秒"变成"每 2 秒"，告警规则的 `for: 2m` 可能需要调整为 `for: 3m`，否则可能出现基于不精确数据的误告警。

**步骤十：避免 GC 抖动——Sentinel 对象池化**

```java
// Sentinel 在高 QPS 下大量创建短生命周期对象（WindowWrap，MetricNode 等）
// 通过 JVM 参数减少 GC 频率

// 1. 适当增加年轻代大小（Sentinel 对象大多是朝生夕死）
//    -XX:NewRatio=2  ← 年轻代 : 老年代 = 1:2
//    -XX:SurvivorRatio=8

// 2. 监控 GC 日志中 Sentinel 相关类的分配速率
//    使用 JFR (JDK Flight Recorder) 记录：
//    -XX:StartFlightRecording=filename=sentinel.jfr,duration=60s

// 3. 严重情况下可为 Sentinel 内部热点对象实现对象池
public class WindowWrapPool {
    private static final ThreadLocal<Queue<WindowWrap<MetricBucket>>> pool =
        ThreadLocal.withInitial(ArrayDeque::new);
    
    public static WindowWrap<MetricBucket> acquire() {
        WindowWrap<MetricBucket> w = pool.get().poll();
        return w != null ? w.reset() : new WindowWrap<>();
    }
    
    public static void release(WindowWrap<MetricBucket> w) {
        Queue<WindowWrap<MetricBucket>> q = pool.get();
        if (q.size() < 64) q.offer(w);
    }
}
```

**步骤十一：自定义 Slot Chain——按需裁剪减少遍历开销**

```java
@Configuration
public class MinimalSlotChainBuilder {

    @PostConstruct
    public void buildMinimalChain() {
        // Sentinel 默认 Slot Chain 有 9 个 Slot
        // 如果你的服务不需要某些功能，可以裁剪 Slot 减少遍历开销
        
        // 精简链路（仅保留流控 + 熔断 + 统计）：
        // 默认: NodeSelectorSlot → ClusterBuilderSlot → LogSlot 
        //       → StatisticSlot → AuthoritySlot → SystemSlot 
        //       → ParamFlowSlot → FlowSlot → DegradeSlot
        //
        // 如果不需要热点参数限流、授权、系统保护：
        // 精简: NodeSelectorSlot → ClusterBuilderSlot → LogSlot
        //       → StatisticSlot → FlowSlot → DegradeSlot
        // 省去 3 个 Slot 的 entry() 调用，降延迟 ~0.3μs
        
        SlotChainBuilder builder = new DefaultSlotChainBuilder() {
            @Override
            public ProcessorSlotChain build() {
                ProcessorSlotChain chain = new DefaultProcessorSlotChain();
                chain.addLast(new NodeSelectorSlot());
                chain.addLast(new ClusterBuilderSlot());
                chain.addLast(new StatisticSlot());
                chain.addLast(new FlowSlot());
                chain.addLast(new DegradeSlot());
                chain.addLast(new LogSlot());
                return chain;
            }
        };
        
        // 注册自定义 Slot Chain Builder
        // (注：Sentinel 原生不支持在运行时替换 SlotChainBuilder，
        //  此示例演示思路——实际需要通过 SPI 或 fork 源码实现)
    }
}
```

**踩坑记录（续）**：

4. **虚拟线程 + ThreadLocal 丢失**：JDK 21 虚拟线程环境下，Sentinel 的 `ContextUtil.getContext()` 在 carrier 线程切换时会丢失上下文。解决方案：升级到 Sentinel 1.8.6+，或使用 `SphU.asyncEntry()` 显式管理上下文。
5. **LeapArray 窗口长度调大的副作用**：窗口从 500ms 改为 2000ms 后，QPS 统计精度从"每秒"变成"每 2 秒"，告警规则的 `for: 2m` 可能需要调整为 `for: 3m`，否则可能出现基于不精确数据的误告警。
6. **应用启动时 Sentinel 初始化竞态**：Spring Boot 启动期间，业务请求可能在 Sentinel 初始化完成前到达。使用 `ApplicationRunner` 确保 Sentinel 规则在 `WebServer` 启动前就绪，并配置 `readinessProbe` 在规则加载后才让 Pod 接收流量。

## 4 项目总结

### 4.1 调优检查清单

- [ ] 资源数 < 5000
- [ ] FlowRule < 500，DegradeRule < 100
- [ ] 热点参数缓存上限 < 2000
- [ ] 使用 G1GC，配置合理的 pause target
- [ ] metric 日志目录使用 SSD
- [ ] 定期清理无流量资源的规则
- [ ] LeapArray 窗口长度根据 QPS 调整（高 QPS → 扩大窗口）
- [ ] ParamFlowRule 按资源名建立索引，规则组内按命中率排序
- [ ] metric.flush.interval 根据磁盘 IO 能力调整（SSD: 1s, HDD: 5s）
- [ ] JDK 21 虚拟线程场景已配置 TransmittableThreadLocal
- [ ] 服务启动后有 10-30 秒预热期，逐步放开阈值
- [ ] JFR/JMC 录制 60 秒，确认 Sentinel 相关方法不在热点 Top 20
- [ ] 非必要 Slot 已从 Chain 中裁剪
- [ ] GC 日志中 Sentinel 对象分配速率 < 50 MB/s

### 4.2 不同 QPS 量级的优化策略

| QPS 量级 | 主要瓶颈 | 优化方向 | 效果预估 | 投入成本 |
|---------|---------|---------|---------|---------|
| < 1 万 | 无需优化 | 默认配置即可 | — | 零 |
| 1-5 万 | 规则线性匹配 | 合并同类规则 + 分组索引 | RT 降低 20-30% | 低（规则整理） |
| 5-10 万 | CAS 竞争 + 规则匹配 | 扩大窗口长度 + 规则排序 | RT 降低 40-50% | 中（需要压测验证） |
| 10 万+ | 磁盘 IO + GC | 内存文件系统 + 指标异步上报 | RT 降低 50-70% | 高（架构改造） |
| 20 万+ | 多维度瓶颈 | 自定义 SlotChain + 对象池 + GraalVM | RT 降低 70-80% | 极高（需要 fork 或定制） |

### 4.3 优化方案决策矩阵

| 优化方案 | 复杂度 | 风险 | 适用阶段 | 副作用 |
|---------|-------|------|---------|-------|
| 合并同类规则 | 低 | 低 | 任何阶段 | 管理粒度降低 |
| 扩大统计窗口 | 低 | 中 | QPS > 5 万 | 统计精度下降 |
| 规则索引分组 | 中 | 低 | QPS > 1 万 | 需自定义实现 |
| 裁剪 Slot Chain | 中 | 中 | 不需要被裁剪的功能 | 功能缺失 |
| 自定义 Metric 上报 | 高 | 中 | Disk IO 成瓶颈 | 依赖外部存储 |
| 对象池化 | 高 | 高 | GC 成瓶颈 | 内存泄漏风险 |
| GraalVM Native | 极高 | 极高 | Serverless 场景 | 动态加载受限 |

### 4.4 Sentinel 自身开销基准

| 操作 | 默认开销 | 优化后开销 | 备注 |
|------|---------|-----------|------|
| Slot Chain 完整遍历（9 Slot） | 1-3 μs | 0.5-1.5 μs (裁剪后) | 取决于 Slot 数量 |
| LeapArray.currentWindow() | 0.1-0.5 μs | 0.05-0.2 μs (调整窗口) | QPS 越高 CAS 竞争越重 |
| FlowSlot.entry() (100条规则) | 0.01-0.5 μs | 0.01-0.1 μs (索引后) | 线性增长 O(n) |
| Metric 日志写入（每资源） | 1-5 μs | 0 (异步上报) | 阻塞 IO 的影响 |
| ParamFlowSlot（500条规则） | 0.1-2 μs | 0.05-0.3 μs (分组后) | 热点参数匹配 |
