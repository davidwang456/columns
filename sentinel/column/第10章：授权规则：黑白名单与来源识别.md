# 第10章：授权规则：黑白名单与来源识别

## 1 项目背景

第 9 章的系统自适应保护让订单服务能在系统级过载时自动止损。但在一次安全审计中，安全团队发现了一个严重的漏洞：

某个内部管理接口 `/order/admin/batch-cancel` 直接暴露在网关后面，没有任何访问控制。通过查看 Sentinel 的日志，发现凌晨 3 点有大量来自 `unknown` 来源的请求调用了这个接口——显然是外部攻击者在尝试批量取消订单。

运维团队紧急排查后发现：原来这个接口的 Sentinel 限流规则（QPS=10）确实在生效，但攻击者用低频率（1 QPS）持续调用，成功取消了 37 个正常用户的订单！问题在于，限流只能控制"量"，控制不了"谁"可以调用。对于管理类接口，即使 QPS 很低，也不应该允许非授权来源访问。

此外，还有另一个需求：App 端和 Web 端的流量应该有不同的优先级。Web 端在搜索引擎爬虫的冲击下经常被打爆，影响 App 端用户的体验。理想情况是：当 Web 端流量异常时，不影响 App 端的调用。

这就是**授权规则（AuthorityRule）**的用武之地。它提供了白名单和黑名单两种模式，让你可以根据调用来源（origin）来控制谁能访问、谁不能访问。配合自定义的 `RequestOriginParser`，可以从请求 Header、Token、AppId 中提取来源信息，实现轻量级的访问控制。

## 2 项目设计

**小胖**（惊恐地指着告警）："大师不好了！有人半夜偷偷调我们取消订单的接口！明明配了限流 QPS=10，他怎么还是成功了？"

**大师**："因为攻击者很聪明，他知道你限流是 10，他就用 1 QPS 慢慢刷。10 个并发限流拦不住慢速攻击。"

**小白**："这就是典型的'限流不治本'。限流是量的问题，但授权是身份的问题。管理接口不应该对所有人开放。"

**大师**："对。Sentinel 的授权规则就是做这件事的。白名单模式：只有在名单里的来源才能访问。黑名单模式：名单里的来源不能访问。"

**小胖**："那'来源'是什么？IP 地址？用户 ID？"

**大师**："是由你定义的。Sentinel 通过 `RequestOriginParser` 接口让你自定义怎么提取来源。你可以从 Header 里的 `X-App-Id` 提取，可以从 JWT Token 里提取，也可以从 IP 里提取——完全由你决定。"

**小白**："那如果我配了白名单 `web-app` 和 `admin-app`，但有人在 Header 里伪造了 `X-App-Id: admin-app` 怎么办？"

**大师**："这就是授权规则的关键局限性了——它是'声明式'的，不是'鉴权式'的。Header 确实可以被伪造。所以 Sentinel 的授权规则不是安全机制，而是**流量路由控制**。真正的安全鉴权应该由网关或 Spring Security 来做。授权规则的应用场景是：你信任上游已经做了鉴权，你只需要根据鉴权结果做差异化控制。"

**小胖**："那它主要用来干嘛？防止内部服务之间的误调用？"

**大师**："几个典型场景：一是防止测试环境的流量打到生产服务；二是根据 App/Web/H5 不同端做流量隔离；三是内部管理接口只允许通过网关（网关会注入来源标识）访问，防止绕过网关直接调。"

**小白**："那 RequestOriginParser 是怎么被调用的？"

**大师**："在 Sentinel 的 `AuthoritySlot` 中执行。当一个请求进入时，AuthoritySlot 会调用你注册的 `RequestOriginParser.parseOrigin(request)` 获取来源，然后和 AuthorityRule 中的黑白名单比对。如果不匹配，抛出 `AuthorityException`。"

**小胖**："那如果我想在限流之前先做来源控制，比如'只允许 VIP 用户访问，且 VIP 用户 QPS 限制为 100'——授权规则和流控规则能协同工作吗？"

**大师**："可以。授权规则在 AuthoritySlot 中，流控规则在 FlowSlot 中。两个 Slot 顺序执行——先 Authority，后 Flow。所以请求先经过'你是谁'的检查，再经过'你多不多'的检查。被 Authority 拒绝的请求不会计入 Flow 的统计，所以不影响流量指标。"

**小白**："那多套来源标识怎么办？比如我需要同时按 AppId 和 UserType（VIP/普通）做控制。"

**大师**："RequestOriginParser 只能返回一个字符串。如果需要多维度，你有两个方案：一是把多个维度编码到 origin 字符串中（如 `app-web:vip`），规则中按前缀匹配（但 Sentinel 原生不支持通配符匹配，需要自定义 AuthoritySlot）；二是用自定义 Slot 替代 AuthoritySlot，实现多维度鉴权。不过对于大多数场景，单维度 + 分端规则已经够用了。"

**技术映射**：
- `AuthorityRule`：`strategy` 字段决定模式，`RULE_AUTHORITY_WHITE`（白名单）或 `RULE_AUTHORITY_BLACK`（黑名单）；`limitApp` 字段存放来源名列表（逗号分隔）。
- `RequestOriginParser`：接口，实现 `parseOrigin(request)` 方法，从请求中提取来源标识字符串。
- 调用链：`AuthoritySlot.entry()` → 获取 `Context.getOrigin()` → 如果没有则调用 `RequestOriginParser` → 和 `AuthorityRule` 名单比对 → 决定放行或拦截。
- 授权规则的执行在限流规则之前？不，AuthoritySlot 在 Sentinel 默认的 Slot Chain 中位于 FlowSlot 之前（默认顺序：AuthoritySlot → SystemSlot → FlowSlot → DegradeSlot），也就是说先检查"你是谁"，再检查"你多不多"。

## 3 项目实战

### 3.1 环境准备

定制 `RequestOriginParser`，从请求 Header 中提取调用方来源：

```java
@Component
public class CustomOriginParser implements RequestOriginParser {

    @Override
    public String parseOrigin(HttpServletRequest request) {
        // 1. 优先从 Header 取 AppId
        String appId = request.getHeader("X-App-Id");
        if (StringUtils.isNotBlank(appId)) {
            return appId;
        }
        // 2. 其次从请求参数取
        String origin = request.getParameter("origin");
        if (StringUtils.isNotBlank(origin)) {
            return origin;
        }
        // 3. 默认来源
        return "unknown";
    }
}
```

在 Spring Boot 中自动注册：

```java
@Configuration
public class SentinelConfig {

    @Bean
    public RequestOriginParser requestOriginParser() {
        return new CustomOriginParser();
    }
}
```

### 3.2 分步实现

**步骤一：管理接口白名单保护**

```java
@RestController
@RequestMapping("/order/admin")
public class OrderAdminController {

    /**
     * 批量取消订单 —— 只允许 admin-app 和内部网关调用
     */
    @PostMapping("/batch-cancel")
    @SentinelResource(value = "adminBatchCancel", blockHandler = "adminBlock")
    public String batchCancel(@RequestBody List<String> orderIds) {
        return "已取消 " + orderIds.size() + " 个订单";
    }

    public String adminBlock(List<String> orderIds, BlockException e) {
        if (e instanceof AuthorityException) {
            return "Access Denied: 该接口仅限管理端调用";
        }
        return "系统繁忙";
    }
}
```

配置授权规则：

```java
@PostConstruct
public void initAuthorityRules() {
    List<AuthorityRule> rules = new ArrayList<>();

    AuthorityRule adminRule = new AuthorityRule();
    adminRule.setResource("adminBatchCancel");
    adminRule.setStrategy(RuleConstant.AUTHORITY_WHITE);  // 白名单模式
    // 只允许 admin-app 和 internal-gateway 调用
    adminRule.setLimitApp("admin-app,internal-gateway");

    rules.add(adminRule);
    AuthorityRuleManager.loadRules(rules);
}
```

**验证**：

```bash
# 无 X-App-Id → 被拒绝
curl -X POST -H "Content-Type: application/json" \
  -d '["ORD001","ORD002"]' \
  http://localhost:8090/order/admin/batch-cancel
# 输出: Access Denied: 该接口仅限管理端调用

# 正确的 AppId → 允许
curl -X POST -H "Content-Type: application/json" \
  -H "X-App-Id: admin-app" \
  -d '["ORD001","ORD002"]' \
  http://localhost:8090/order/admin/batch-cancel
# 输出: 已取消 2 个订单

# 错误的 AppId → 被拒绝
curl -X POST -H "Content-Type: application/json" \
  -H "X-App-Id: evil-app" \
  -d '["ORD001","ORD002"]' \
  http://localhost:8090/order/admin/batch-cancel
# 输出: Access Denied
```

**步骤三：基于 IP 的来源识别**

如果需要在网关层按 IP 限流（注意：公网环境下 IP 不可信）：

```java
@Component
public class IpBasedOriginParser implements RequestOriginParser {

    @Override
    public String parseOrigin(HttpServletRequest request) {
        // 先从 X-Forwarded-For 取真实 IP（网关代理后）
        String ip = request.getHeader("X-Forwarded-For");
        if (ip == null) {
            ip = request.getRemoteAddr();
        }
        // 内部 IP 段标记为 internal
        if (ip.startsWith("10.") || ip.startsWith("192.168.") || ip.startsWith("172.")) {
            return "internal";
        }
        // 外部 IP 按段分类
        return "external:" + ip;
    }
}
```

**步骤四：网关注入来源标识**

在 Spring Cloud Gateway 中统一注入来源标识，服务端信任即可：

```java
@Component
public class OriginHeaderFilter implements GlobalFilter, Ordered {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        // 从 JWT 或 API Key 中提取 AppId
        String appId = extractAppId(request);

        ServerHttpRequest modifiedRequest = request.mutate()
                .header("X-App-Id", appId)
                .build();

        return chain.filter(exchange.mutate().request(modifiedRequest).build());
    }

    @Override
    public int getOrder() {
        return -100;  // 在 Sentinel Gateway Filter 之前执行
    }
}
```

**踩坑记录**：

1. **来源为空时的行为**：如果 `RequestOriginParser.parseOrigin()` 返回 null 或空字符串，Sentinel 会使用默认来源 `"default"`。如果不希望空来源绕过白名单，务必在 parser 中返回一个非白名单的值（如 `"unknown"`）。
2. **授权规则不支持动态参数**：AuthorityRule 的 `limitApp` 是在规则加载时就确定的，不能在运行时动态添加。需要动态变更的话，得配合 Nacos 数据源。
3. **白名单和黑名单可以共存**：对同一资源配一个白名单和一个黑名单是先校验黑名单还是白名单，取决于 Slot 中的遍历顺序（两个规则都会检查）。
4. **Header 伪造**：`X-Forwarded-For` 和自定义 Header 在公网环境下可以被伪造。生产环境中应只在信任的内网链路（网关 → 服务）使用。敏感操作必须依赖业务鉴权（JWT/OAuth）。

## 4 项目总结

### 4.1 优点与缺点

| 维度 | Sentinel 授权规则 | Spring Security | 网关层 IP 白名单 |
|------|-----------------|----------------|----------------|
| 接入成本 | 低（一个接口+规则） | 高（需深入理解安全框架） | 低（Nginx/网关配置） |
| 灵活性 | 高（可根据业务自定义来源） | 极高（RBAC/ABAC） | 低（只能按 IP） |
| 安全性 | 低（Header 可伪造） | 高（加密 Token） | 中（内网可信） |
| 适用层级 | 服务层 | 服务层/方法层 | 网关层 |

### 4.2 适用场景

- **环境隔离**：防止测试环境/灰度环境的流量误打到生产服务
- **多端流量路由**：App、Web、H5、小程序等不同端走不同的限流/熔断策略
- **内部接口保护**：管理接口、回调接口只允许网关/内部服务调用
- **爬虫/恶意来源拦截**：黑名单模式拒绝已知的恶意来源
- **不适用**：需要强安全鉴权（如支付确认、个人信息查询）——授权规则不替代安全框架

### 4.3 注意事项

1. 授权规则基于 `origin`，不是 `userId` 或 `role`。不要用它做用户级别的权限控制。
2. `RequestOriginParser` 的实现应尽可能地快（毫秒级），因为它在每个请求的热路径上。
3. 来源名中不要包含逗号——`limitApp` 用逗号分隔多个来源。
4. Dashboard 支持配置授权规则（流控规则页面 → 选择"授权"）。

### 4.4 来源识别方案对比表

| 方案 | 安全性 | 实现复杂度 | 适用场景 |
|------|-------|----------|---------|
| 网关 JWT 解析后注入 Header | 中（内网可信） | 低 | 网关 + 服务之间，有认证体系 |
| RequestOriginParser 解析 IP | 低（IP 可伪造） | 低 | 内网环境，IP 段区分 |
| API Key 校验 | 高（加密验证） | 中 | 对外开放的 API |
| 请求参数 origin=xxx | 极低 | 极低 | 仅测试/开发环境 |
| 自定义 AuthoritySlot 多维度 | 高（自定义逻辑） | 高 | 复杂租户体系 |

### 4.5 常见踩坑经验

| 故障现象 | 根因 | 解决方案 |
|---------|------|---------|
| 配了白名单但所有请求都被拒绝 | RequestOriginParser 返回了非预期的值 | 打印 origin 日志确认实际返回值 |
| 来源为空的请求绕过了白名单 | parseOrigin 返回 null 时 Sentinel 用 "default" | 确保返回非白名单值如 "unknown" |
| 网关注入的 Header 到了服务层丢失 | 服务间调用（RestTemplate/Feign）未透传 Header | 配置拦截器自动透传 `X-App-Id` 等关键 Header |
| 多个 AuthorityRule 互相覆盖 | 规则加载方式有问题 | 确认 `AuthorityRuleManager.loadRules()` 只调用一次 |

### 4.6 思考题

1. Sentinel 的授权规则和 Spring Security 的权限控制在架构上有什么区别？在什么场景下两者应该同时使用？
2. 如果攻击者伪造了 `X-App-Id: admin-app`，Sentinel 授权规则会放行。但如果在代码中同时使用 Spring Security 做业务鉴权，攻击者绕过了 Sentinel 能调接口吗？

### 4.7 推广计划

- **开发团队**：将管理类接口和回调类接口全部配置授权白名单，防止绕过网关直接调用。
- **测试团队**：为每个授权规则设计正向和反向测试用例（合法来源、非法来源、空来源）。
- **运维团队**：在网关层统一规划 `X-App-Id` 的取值标准和注册机制，避免来源标识混乱。
