# 第17章：动态规则持久化：从内存态到配置中心

## 1 项目背景

基础篇的 16 章实战让我们搭建了一个完整的秒杀流量防护系统，但第 16 章的复盘报告明确指出了三个遗留问题，其中第一个——**Dashboard 重启后规则丢失**——是最致命的。

在我们公司的测试环境中，Sentinel Dashboard 部署在一台共享服务器上。某个周五下午，服务器因为内存不足被运维重启了。周一早上测试团队开始压测时发现——所有流量防护规则全部失效，订单服务在毫无保护的情况下裸奔了整整一个周末。幸运的是测试环境没有真实用户，但如果是生产环境，这就是一个 P0 级事故。

另一个更大的痛点是：团队有 5 个微服务、合计 120+ 条 Sentinel 规则。每次发布新版本时，运维需要先在 Dashboard 上逐条导出规则、再导入到新环境中。这 120 条规则的维护变成了"人肉运维"——经常出现"测试环境配了但生产环境漏配"的线上事故。

这些问题的本质是：**Sentinel 默认的"内存态"规则管理模式无法满足生产环境对持久化、多环境管理和版本控制的需求**。本章将引入 Sentinel 的 DataSource 扩展机制，以及最常用的方案——将规则持久化到 Nacos 配置中心。

## 2 项目设计

**小胖**（看着空空如也的 Dashboard 规则列表）："又丢了！这是我第三次重新配置这 70 多条规则了。大师，能不能让规则像 application.yml 一样存文件里？"

**大师**："存文件里确实可以，Sentinel 支持本地文件数据源。但多环境怎么搞？开发环境 30 条规则、测试环境 50 条、生产环境 70 条，难道每次部署前手动改文件？"

**小白**："那就上配置中心。Nacos、Apollo、ZooKeeper 都可以。把规则当配置管理，环境隔离、灰度发布、版本回滚全都有了。"

**大师**："对。这就是 Sentinel DataSource 扩展机制的核心思路。Sentinel 定义了 `ReadableDataSource` 和 `WritableDataSource` 两个接口。前者负责从外部数据源拉取规则，后者负责将 Dashboard 的修改推送回数据源。"

**小胖**："那 push 模式和 pull 模式有什么区别？"

**大师**："这个问题问得好。pull 模式是客户端定期去配置中心拉取规则（比如每 10 秒拉一次），简单但有时延。push 模式是配置中心主动推送变更通知给客户端，实时性强但需要配置中心支持回调。"

**小白**："Nacos 支持 push。在 Spring Cloud Alibaba 生态中，Sentinel 和 Nacos 的集成是最完善的——配置变更 → Nacos 通知 → Sentinel 客户端热更新规则，全过程不需要重启。"

**大师**："除了 Nacos，Apollo 和 ZooKeeper 也各有优势。Apollo 的权限管理和审批流更完善，ZooKeeper 在非 Spring 生态中更常见。但考虑到我们用的是 Spring Cloud Alibaba 全家桶，Nacos 是最自然的选型。"

**小胖**："大师，如果 Nacos 挂了，客户端已有的规则会丢吗？还是用最后一次拉取的缓存？"

**大师**："不会丢。Sentinel 客户端在内存中保留的是最后一次成功加载的规则。Nacos 挂了只是不能更新规则，已加载的规则照常工作。但如果客户端也重启了——那就没地方加载了。这正是为什么需要'本地文件快照'做最后的兜底。"

**小白**："那多环境隔离除了 Nacos namespace，还有没有更细的粒度？比如同一个 namespace 下，我想让 10% 的实例用灰度规则。"

**大师**："有。可以用 Nacos 的 group 做二级分组。比如 group=`DEFAULT_GROUP` 是基线规则，group=`GRAY_GROUP` 是灰度规则。10% 的实例配置灰度 group，验证通过后再把灰度规则合并到默认 group。另一种方式是同一 dataId 下用不同的配置内容 ID，在客户端通过启动参数区分。"

**小胖**："还有个实操问题——我在 Nacos 里改了规则，但 Dashboard 上显示的规则没变化，怎么回事？"

**大师**："这是 Dashboard 的已知限制。Dashboard 从 Nacos 加载规则依赖 `writableDataSource` 的回写逻辑。如果只配了 `ReadableDataSource`，Dashboard 是读不到 Nacos 规则的——它仍然显示自己内存中的规则。解决方案是：Dashboard 也配置 Nacos 数据源（参考 Sentinel Dashboard 的 Nacos 改造方案），或者干脆用 Nacos 控制台替代 Dashboard 管理规则。"

**技术映射**：
- `ReadableDataSource<S, T>`：泛型 S 是数据源读取的原始类型（如 String），T 是转换后的 Sentinel 规则类型。核心方法 `readSource()` 从数据源读取，`loadConfig()` 反序列化为规则对象并写入 RuleManager。
- `WritableDataSource<T>`：核心方法 `write(T value)` 将 Dashboard 修改的规则写回数据源。
- **pull 模式**：通过定时任务（如 `ScheduledExecutorService`）定期调用 `readSource()`。
- **push 模式**：配置中心监听器回调触发 `readSource()` 重新读取，调用 `loadConfig()` 热更新。
- Spring Cloud Alibaba Sentinel 的自动配置会根据 classpath 自动选择数据源类型。

## 3 项目实战

### 3.1 环境准备

确保 Nacos 服务可用：

```bash
# Docker 启动 Nacos（单机模式）
docker run -d --name nacos \
  -e MODE=standalone \
  -p 8848:8848 \
  -p 9848:9848 \
  nacos/nacos-server:2.3.2
```

访问 `http://localhost:8848/nacos`，用户名密码默认 `nacos/nacos`。

### 3.2 分步实现

**步骤一：Sentinel 规则数据模型与 Nacos 配置格式**

Sentinel 的流控规则 JSON 格式：

```json
[
  {
    "resource": "createOrder",
    "grade": 1,
    "count": 10.0,
    "limitApp": "default",
    "strategy": 0,
    "controlBehavior": 0
  }
]
```

熔断规则：

```json
[
  {
    "resource": "queryStock",
    "grade": 0,
    "count": 50.0,
    "timeWindow": 10,
    "minRequestAmount": 5,
    "statIntervalMs": 10000,
    "slowRatioThreshold": 0.5
  }
]
```

**步骤二：Nacos 数据源依赖配置**

`pom.xml` 添加：

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
    <version>2023.0.1.0</version>
</dependency>
<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-datasource-nacos</artifactId>
    <version>1.8.6</version>
</dependency>
```

`application.yml` 配置 Nacos 数据源：

```yaml
spring:
  application:
    name: order-service
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8080
        port: 8719
      eager: true
      datasource:
        # 流控规则数据源
        flow:
          nacos:
            server-addr: localhost:8848
            namespace: ${NACOS_NAMESPACE:dev}     # dev/test/prod 隔离
            group-id: DEFAULT_GROUP
            data-id: ${spring.application.name}-flow-rules
            rule-type: flow                        # 规则类型：flow/degrade/param-flow/system/authority
            data-type: json
        # 熔断规则数据源
        degrade:
          nacos:
            server-addr: localhost:8848
            namespace: ${NACOS_NAMESPACE:dev}
            group-id: DEFAULT_GROUP
            data-id: ${spring.application.name}-degrade-rules
            rule-type: degrade
            data-type: json
        # 热点参数规则
        param-flow:
          nacos:
            server-addr: localhost:8848
            namespace: ${NACOS_NAMESPACE:dev}
            group-id: DEFAULT_GROUP
            data-id: ${spring.application.name}-param-flow-rules
            rule-type: param-flow
            data-type: json
        # 系统规则
        system:
          nacos:
            server-addr: localhost:8848
            namespace: ${NACOS_NAMESPACE:dev}
            group-id: DEFAULT_GROUP
            data-id: ${spring.application.name}-system-rules
            rule-type: system
            data-type: json
        # 授权规则
        authority:
          nacos:
            server-addr: localhost:8848
            namespace: ${NACOS_NAMESPACE:dev}
            group-id: DEFAULT_GROUP
            data-id: ${spring.application.name}-authority-rules
            rule-type: authority
            data-type: json
```

**步骤三：在 Nacos 中创建规则**

1. 登录 Nacos 控制台（`http://localhost:8848/nacos`）
2. 进入"配置管理" → "配置列表" → 新建配置
3. Data ID：`order-service-flow-rules`
4. Group：`DEFAULT_GROUP`
5. 配置内容：

```json
[
  {
    "resource": "createOrder",
    "limitApp": "default",
    "grade": 1,
    "count": 50,
    "strategy": 0,
    "controlBehavior": 0
  },
  {
    "resource": "queryOrder",
    "limitApp": "default",
    "grade": 1,
    "count": 100,
    "strategy": 0,
    "controlBehavior": 0
  }
]
```

6. 发布配置。

**步骤四：验证动态规则持久化**

1. 启动订单服务，查看日志：

```log
[Sentinel Nacos DataSource] Loaded 2 flow rules from Nacos:
    DataId=order-service-flow-rules, Group=DEFAULT_GROUP
```

2. 访问 Dashboard，流控规则页面自动显示这 2 条规则（注意：规则来源标记为 Nacos）。

3. 在 Nacos 中修改配置，将 `createOrder` 的 `count` 从 50 改为 80：

```json
[
  {
    "resource": "createOrder",
    "grade": 1,
    "count": 80,
    ...
  }
]
```

4. 观察客户端日志，几秒内出现：

```log
[Sentinel Nacos DataSource] Rules updated from Nacos: 2 flow rules reloaded
```

5. 用 JMeter 验证新阈值生效（QPS=80 而非旧的 50）。

6. **重启验证**：重启订单服务 → 规则自动从 Nacos 加载 → Dashboard 依然显示这 2 条规则。

**步骤五：多环境隔离**

在 Nacos 中创建三个命名空间：

| 命名空间 | Namespace ID | 用途 |
|---------|-------------|------|
| dev | dev-xxxx | 开发环境规则 |
| test | test-xxxx | 测试环境规则 |
| prod | prod-xxxx | 生产环境规则 |

每个环境启动时通过环境变量指定：

```bash
# 开发环境
NACOS_NAMESPACE=dev java -jar order-service.jar

# 生产环境
NACOS_NAMESPACE=prod java -jar order-service.jar
```

对应 Nacos 中 `order-service-flow-rules` 在每个 namespace 下可以有不同的阈值配置。

**步骤六：push 模式 vs pull 模式对比实施**

```java
// Pull 模式示例（不推荐生产使用）
@Configuration
public class PullModeConfig {

    @PostConstruct
    public void init() {
        ReadableDataSource<String, List<FlowRule>> dataSource =
            new NacosDataSource<>(
                "localhost:8848", "dev", "order-service-flow-rules",
                source -> JSON.parseObject(source,
                    new TypeReference<List<FlowRule>>() {})
            );

        // 每 10 秒拉取一次
        FlowRuleManager.register2Property(dataSource.getProperty());
    }
}

// Push 模式（Spring Cloud Alibaba 自动配置默认就是 push 模式）
// 无需额外代码，Nacos 配置变更后自动推送通知
```

**踩坑记录**：

1. **Nacos 版本兼容**：Sentinel 1.8.6 + Nacos Client 2.x 与 Nacos Server 1.x/2.x 的兼容矩阵需确认。
2. **规则 JSON 格式错误**：Nacos 中的 JSON 配置如果格式不对，Sentinel 会解析失败（静默回退到空规则列表）。务必在 Nacos 的"配置校验"功能或本地 JSON 验证工具确认格式。
3. **多数据源覆盖**：如果同时存在 Nacos 数据源、Dashboard 推送、本地代码加载三种方式，最后生效的取决于加载顺序。
4. **namespace 区分大小写**：Nacos 的 namespace ID 是 UUID，但 `spring.cloud.sentinel.datasource.flow.nacos.namespace` 配置的必须是 namespace ID，不是 namespace 名称。

**步骤七：本地文件快照兜底——防止 Nacos 不可用时规则丢失**

```java
@Configuration
public class FileSnapshotFallbackConfig {

    private static final String SNAPSHOT_DIR = "/var/sentinel/snapshots/";

    @PostConstruct
    public void initLocalSnapshot() {
        // 1. 尝试从 Nacos 加载规则（正常路径）
        // 2. 如果 Nacos 不可用，回退到本地文件快照
        List<FlowRule> rules = loadRulesFromNacos();
        if (rules.isEmpty()) {
            log.warn("Nacos 不可达，从本地快照加载规则");
            rules = loadRulesFromLocalSnapshot();
        }
        if (!rules.isEmpty()) {
            FlowRuleManager.loadRules(rules);
        }

        // 3. 注册规则变更监听器：规则变更时自动写入本地快照
        FlowRuleManager.getProperty().addListener(updatedRules -> {
            saveRulesToLocalSnapshot(updatedRules);
        });
    }

    private void saveRulesToLocalSnapshot(List<FlowRule> rules) {
        try {
            File file = new File(SNAPSHOT_DIR + "flow-rules-snapshot.json");
            file.getParentFile().mkdirs();
            Files.writeString(file.toPath(), JSON.toJSONString(rules));
        } catch (IOException e) {
            log.error("保存规则快照失败", e);
        }
    }
}
```

**步骤八：多环境规则一键同步工具**

```bash
#!/bin/bash
# sync-rules.sh — 从 dev 环境同步规则到 test 环境

SRC_NAMESPACE="dev"
DST_NAMESPACE="test"
NACOS_URL="http://nacos:8848"
SERVICE="order-service"

for RULE_TYPE in flow degrade param-flow system authority; do
    DATA_ID="${SERVICE}-${RULE_TYPE}-rules"

    echo "同步规则: ${DATA_ID} (${SRC_NAMESPACE} → ${DST_NAMESPACE})"

    # 1. 从源环境导出
    CONTENT=$(curl -s "${NACOS_URL}/nacos/v1/cs/configs?tenant=${SRC_NAMESPACE}&dataId=${DATA_ID}&group=DEFAULT_GROUP")

    # 2. 写入目标环境
    curl -X POST "${NACOS_URL}/nacos/v1/cs/configs" \
      -d "tenant=${DST_NAMESPACE}&dataId=${DATA_ID}&group=DEFAULT_GROUP&content=${CONTENT}"
done

echo "同步完成: ${SRC_NAMESPACE} → ${DST_NAMESPACE}"

## 4 项目总结

### 4.1 优点与缺点

| 持久化方案 | 优点 | 缺点 | 适用环境 |
|-----------|------|------|---------|
| Nacos 数据源 | 与 Spring Cloud Alibaba 原生集成，实时推送 | 需部署 Nacos | 开发/测试/生产 |
| Apollo 数据源 | 审批流完善，灰度发布 | 社区版功能受限 | 大型企业 |
| ZooKeeper 数据源 | 可靠的分布式协调 | 学习成本高，配置管理功能弱 | 已有 ZK 基础设施 |
| 本地文件数据源 | 极简，零依赖 | 无法动态变更，不支持集群 | 开发/演示 |
| Redis 数据源 | 高性能 | 需自行处理序列化和监听 | 高并发场景 |

### 4.2 适用场景

- Nacos 数据源是 Spring Cloud Alibaba 生态的首选方案
- Apollo 适用于有严格审批流程的金融/银行场景
- 本地文件适用于本地开发和快速 Demo

### 4.3 注意事项

1. 规则 JSON 格式在 Nacos 中务必做格式校验，生产环境配置修改建议走审批 + 灰度流程。
2. Nacos 的配置变更通知是非幂等的——多次推送同一条规则会重复加载，虽然不影响最终结果但会增加 CPU 开销。
3. 不要同时在 Nacos 和本地代码中为同一个资源加载规则，避免互相覆盖。
4. Dashboard 的规则推送逻辑在 Nacos 模式下会变——Dashboard 修改规则后应该写回 Nacos，而非推送到客户端。这需要配置 `writableDataSource`。

### 4.4 规则持久化方案的成本对比

| 方案 | 部署成本 | 开发成本 | 运维成本 | 可靠性 | 推荐指数 |
|------|---------|---------|---------|-------|---------|
| 代码硬编码 | 零 | 零 | 高（重启才生效） | 低 | ⭐ |
| 本地文件 | 零 | 低 | 中（手动管理文件） | 中 | ⭐⭐ |
| Nacos 数据源 | 中（需搭建 Nacos） | 低 | 低 | 高 | ⭐⭐⭐⭐⭐ |
| Apollo 数据源 | 高（需搭建 Apollo） | 中 | 低 | 高 | ⭐⭐⭐⭐ |
| 自建治理平台 | 高 | 高（4-8 人月） | 中 | 高 | ⭐⭐⭐ |

### 4.5 规则持久化的反模式

| 反模式 | 后果 | 正确做法 |
|-------|------|---------|
| 同一个资源在代码和 Nacos 各配一次 | 加载顺序不确定，实际生效的规则不可预知 | 全部迁移到 Nacos |
| Nacos namespace 使用名称而非 UUID | 规则加载失败 | 使用 Nacos 控制台查到的 UUID |
| 所有环境的规则放在同一个 group | 测试改规则可能影响生产 | 不同环境用不同 namespace |
| 未配置本地文件快照兜底 | Nacos 不可用时规则全部丢失 | 配置步骤七的快照机制 |

### 4.4 常见踩坑经验

| 故障现象 | 根因 | 解决方案 |
|---------|------|---------|
| 启动后规则为空 | Nacos 连接失败或配置不存在 | 检查 Nacos 地址和 namespace ID |
| Nacos 修改后规则不更新 | namespace ID 写成了名称 | 用 Nacos 控制台查看正确的 namespace ID |
| 规则加载成功但类型不对 | `rule-type` 配置错误（flow 写成了 degrade） | 检查 dataId 对应的 rule-type |
| 多环境规则混乱 | namespace 配置被环境变量覆盖 | 确认启动脚本中 NACOS_NAMESPACE 是否正确 |

### 4.5 思考题

1. Sentinel 的 Nacos 数据源默认是 push 模式还是 pull 模式？如果 Nacos 服务暂时不可达，客户端已有的规则会丢失吗？
2. 如果一个资源在 Nacos 中配置了 3 条流控规则，但生产环境临时需要通过 Dashboard 紧急修改其中 1 条（绕过 Nacos 审批），技术上可行吗？会有什么副作用？

### 4.6 推广计划

- **开发团队**：所有环境的 Sentinel 规则统一迁移到 Nacos，禁止代码硬编码规则。
- **运维团队**：负责 Nacos 权限管理和 namespace 规划，制定规则变更的审批和灰度发布流程。
- **测试团队**：增加"规则持久化"验收测试——重启服务后验证规则自动加载。
