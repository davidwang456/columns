# 第6章：关联流控与链路流控

## 1 项目背景

第 5 章的 QPS 和线程数限流让订单服务有了基础保护能力，但测试团队在一次全链路压测中发现了两个"单点保护"解决不了的问题：

**场景一**：订单服务的"支付回调"接口平时流量很低，但每当有促销活动，支付回调流量会在某个时间点集中爆发（用户赶在活动结束前支付）。开发团队发现，支付回调接口需要查询订单、更新订单状态、发放积分，内部依赖了订单查询服务（`queryOrder`）和积分服务（`addPoints`）。如果支付回调流量突然暴增，`queryOrder` 和 `addPoints` 会被打爆，进而影响其他业务（如用户自己查询订单也会变慢）。

理想情况是：当支付回调流量大时，自动限制"用户查询订单"的流量，把更多资源留给支付回调处理。这就是**关联流控**。

**场景二**：同一个"查询库存"资源（`/inventory/query`）同时被"商品详情页"和"下单页面"两个入口调用。产品经理要求：商品详情页浏览量大，QPS 限制可以放松到 200；但下单页面是核心交易链路，库存查询的可靠性要求更高，QPS 限制应该收紧到 50。如果只在 `queryInventory` 资源上配一条 QPS=50 的规则，详情页的体验就会受影响。这就是**链路流控**。

但测试团队在实际配置中发现，链路流控怎么都配不生效！查了半天才发现是 Sentinel 的 Web 场景下链路收敛问题：不同的 URL 入口在 Sentinel 内部被合并成了同一个调用入口，链路规则形同虚设。

## 2 项目设计

**小胖**（一脸困惑）："大师大师，我配了一个链路规则——同一个资源`queryInventory`从`/goods/detail`入口进来限流 200，从`/order/create`入口进来限流 50。但测试发现全部按 50 限了！"

**大师**："你先告诉我，你的 Web 入口 URL 是什么？"

**小胖**："`/goods/detail` 和 `/order/create` 啊，两个不同的 Controller。"

**大师**："问题出在 `web-context-unify` 上。Sentinel 为了减少 Context 开销，默认会把所有 Web 请求的入口统一为一个叫 `sentinel_web_servlet_context` 的名字。换句话说，不管你是从 `/goods/detail` 进还是从 `/order/create` 进，Sentinel 认为都是从'同一个门'进来的。"

**小白**："这就是链路收敛（Context Unify）？那岂不是链路流控在 Spring MVC 下默认不生效？"

**大师**："对，这是个经典坑。你需要在配置中设置 `spring.cloud.sentinel.web-context-unify=false`，让 Sentinel 用真实的 URL 作为入口资源名。"

**小胖**："那关联流控呢？我刚配了一个'当支付接口 QPS 超过 100 时，限制订单查询到 50'的规则，但感觉没什么用啊，支付接口爆了跟订单查询有什么关系？"

**小白**："有用。如果支付服务和订单查询服务共享了一个数据库连接池或 Redis 集群，支付接口的高流量会耗尽连接，订单查询就查不出来了。关联流控就是在这种资源竞争场景下用的。"

**大师**："小白理解对了。我再补充一个真实案例：某电商的'商品详情'和'加入购物车'都依赖同一个推荐引擎。双十一时，商品详情页流量暴增（4000 QPS），推荐引擎响应变慢，导致加入购物车也超时，影响了转化率。如果当时配了关联流控——当详情页 QPS 超过 3000 时，限制加入购物车的并发——就能保护核心交易链路。"

**小胖**："那链路流控和关联流控的区别到底是什么？我有点晕。"

**大师**："画个图你就懂了。关联流控是两个**不同资源**之间的关系（资源 A 流量大 → 限制资源 B）。链路流控是**同一个资源**在不同**入口**下的差异化控制（从入口 X 进来的资源 A 限制 50，从入口 Y 进来的资源 A 限制 200）。"

**小胖**："那能不能把关联流控和链路流控串在一起用？比如'从 App 入口进来的订单查询，当支付回调流量大时，限制到 30 QPS'？"

**大师**："这就涉及规则叠加了。关联流控和链路流控的 `strategy` 字段只能选一个值——`STRATEGY_RELATE` 和 `STRATEGY_CHAIN` 是互斥的。但你可以用两条规则叠加：一条链路流控（从 App 入口 → queryOrder 限制 100），一条关联流控（payCallback 流量大 → 限制 queryOrder 到 30）。两条规则同时生效，第一个不通过的拦截请求。"

**小白**："那如果再加上授权规则呢？比如'只允许 App 端调用，且从 App 入口进入时限制 50 QPS'——这需要链路流控 + 授权规则同时配？"

**大师**："对。授权规则在 AuthoritySlot 中执行，链路流控在 FlowSlot 中执行，两个 Slot 是按顺序遍历的。AuthoritySlot 先判断'你是谁'（来源是否合法），FlowSlot 再判断'你多不多'（是否超阈值）。两者互补，不冲突。"

**技术映射**：
- **关联流控**：`FlowRule.strategy = STRATEGY_RELATE`，`refResource` 指定关联资源名。当关联资源的 QPS/线程数达到阈值时，限制本资源的流量。
- **链路流控**：`FlowRule.strategy = STRATEGY_CHAIN`，`refResource` 指定入口资源名（即 Context 的名字）。只有当请求从该入口进入时，限制才生效。
- **入口资源**：通过 `ContextUtil.enter(contextName, origin)` 中的 `contextName` 定义，Web 场景下默认是 URL 路径（需关闭 context-unify）。
- **调用树**：Sentinel 内部用 `DefaultNode` 构建调用树，每个 Context 下有自己的 Node 链。链路流控就是在这一层做统计和判断。

## 3 项目实战

### 3.1 环境准备

配置 `application.yml` 关闭 Web 上下文收敛：

```yaml
spring:
  cloud:
    sentinel:
      web-context-unify: false  # 关闭链路收敛，让真实 URL 作为入口
```

### 3.2 分步实现

**步骤一：关联流控实战——支付回调影响订单查询**

创建两个受保护资源：

```java
@RestController
public class RelateFlowController {

    // 用于测试的共享 Service
    @Autowired
    private InventoryService inventoryService;

    /**
     * 支付回调接口 —— 流量大时会影响订单查询
     */
    @GetMapping("/pay/callback")
    @SentinelResource(value = "payCallback", blockHandler = "blockHandler")
    public String payCallback(@RequestParam String orderId) throws InterruptedException {
        // 模拟支付回调的复杂处理
        inventoryService.queryAndUpdateStock(orderId);
        Thread.sleep(100);
        return "支付回调处理完成: " + orderId;
    }

    /**
     * 订单查询接口 —— 受关联流控保护
     */
    @GetMapping("/order/query")
    @SentinelResource(value = "queryOrderDetail", blockHandler = "blockHandler")
    public String queryOrder(@RequestParam String orderId) throws InterruptedException {
        Thread.sleep(50);
        return "订单详情: " + orderId;
    }

    public String blockHandler(BlockException e) {
        return "资源受限: " + e.getClass().getSimpleName();
    }
}
```

配置关联流控规则：

```java
@PostConstruct
public void initRelateRule() {
    List<FlowRule> rules = new ArrayList<>();

    // 关联流控：当 payCallback 的 QPS 超过 10 时，限制 queryOrderDetail 的 QPS 到 5
    FlowRule relateRule = new FlowRule("queryOrderDetail")
            .setGrade(RuleConstant.FLOW_GRADE_QPS)
            .setCount(5)                                    // 本资源的阈值
            .setStrategy(RuleConstant.STRATEGY_RELATE)      // 关联流控模式
            .setRefResource("payCallback");                 // 关联资源
    rules.add(relateRule);

    FlowRuleManager.loadRules(rules);
}
```

**验证**：

1. 先用 JMeter 单独压测 `/order/query`，发现不受限制（因为 `payCallback` 流量低）。
2. 同时用两个线程组压测：一组对 `/pay/callback` 打 20 QPS，另一组对 `/order/query` 打 15 QPS。
3. 观察结果：当 `payCallback` 的 QPS 超过 10 后，`order/query` 开始被限流到 5 QPS。
4. 停止对 `payCallback` 的压测，`order/query` 又恢复正常。

**步骤二：链路流控实战——同一资源不同入口不同阈值**

创建两个入口方法，都调用同一个 `queryInventory` 资源：

```java
@RestController
public class LinkFlowController {

    @Autowired
    private InventoryService inventoryService;

    /**
     * 入口 1：商品详情页 —— 浏览量大，限制宽松
     */
    @GetMapping("/goods/detail")
    public String goodsDetail(@RequestParam String skuId) {
        // 内部调用 queryInventory 资源
        return "商品详情: " + inventoryService.queryInventory(skuId);
    }

    /**
     * 入口 2：下单页面 —— 核心链路，限制严格
     */
    @GetMapping("/order/create-page")
    public String orderCreatePage(@RequestParam String skuId) {
        // 内部也调用同一个 queryInventory 资源
        return "下单页面: " + inventoryService.queryInventory(skuId);
    }
}

@Service
public class InventoryService {

    @SentinelResource(value = "queryInventory", blockHandler = "queryBlock")
    public String queryInventory(String skuId) {
        return "库存信息(skuId=" + skuId + "): 剩余 100 件";
    }

    public String queryBlock(String skuId, BlockException e) {
        return "查询库存繁忙，请稍后重试";
    }
}
```

配置链路流控规则：

```java
@PostConstruct
public void initLinkRule() {
    List<FlowRule> rules = new ArrayList<>();

    // 链路流控：从 /goods/detail 入口进入的 queryInventory，QPS 限制 200
    FlowRule linkRuleGoods = new FlowRule("queryInventory")
            .setGrade(RuleConstant.FLOW_GRADE_QPS)
            .setCount(200)
            .setStrategy(RuleConstant.STRATEGY_CHAIN)       // 链路流控模式
            .setRefResource("/goods/detail");               // 入口资源名（即 Context 名）
    rules.add(linkRuleGoods);

    // 链路流控：从 /order/create-page 入口进入的 queryInventory，QPS 限制 50
    FlowRule linkRuleOrder = new FlowRule("queryInventory")
            .setGrade(RuleConstant.FLOW_GRADE_QPS)
            .setCount(50)
            .setStrategy(RuleConstant.STRATEGY_CHAIN)
            .setRefResource("/order/create-page");
    rules.add(linkRuleOrder);

    FlowRuleManager.loadRules(rules);
}
```

**验证**：

1. 用 JMeter 对 `/goods/detail` 压测 250 QPS：约 200 个通过，50 个被链路流控拦截。
2. 用 JMeter 对 `/order/create-page` 压测 100 QPS：约 50 个通过，50 个被链路流控拦截。
3. 两个接口返回的拦截结果都指向 `queryInventory` 资源，但阈值不同。

**步骤三：调试入口资源名**

如果链路流控不生效，用以下方式查看实际的入口资源名：

```java
@GetMapping("/debug/context")
public String debugContext() {
    Context context = ContextUtil.getContext();
    return "Context Name: " + context.getName()
         + ", Origin: " + context.getOrigin()
         + ", EntranceNode: " + context.getEntranceNode();
}
```

访问 `http://localhost:8090/debug/context`，确认 `web-context-unify=false` 生效后 Context Name 是真实 URL。

**步骤四：关联流控叠加链路流控验证**

```java
@PostConstruct
public void initCombinedRules() {
    List<FlowRule> rules = new ArrayList<>();

    // 规则 A：链路流控——从 /order/create-page 入口进入的 queryInventory，限制 50
    FlowRule chainRule = new FlowRule("queryInventory")
            .setGrade(RuleConstant.FLOW_GRADE_QPS)
            .setCount(50)
            .setStrategy(RuleConstant.STRATEGY_CHAIN)
            .setRefResource("/order/create-page");
    rules.add(chainRule);

    // 规则 B：关联流控——当 payCallback 的 QPS 超过 20 时，queryInventory 限 10
    FlowRule relateRule = new FlowRule("queryInventory")
            .setGrade(RuleConstant.FLOW_GRADE_QPS)
            .setCount(10)
            .setStrategy(RuleConstant.STRATEGY_RELATE)
            .setRefResource("payCallback");
    rules.add(relateRule);

    FlowRuleManager.loadRules(rules);
}
```

验证：同时压测 `/order/create-page`（触发链路流控规则 A）和 `/pay/callback`（触发关联流控规则 B）。当 payCallback QPS < 20 时，queryInventory 受规则 A 限制（≤ 50）；当 payCallback QPS ≥ 20 时，规则 B 激活，queryInventory 被进一步限制到 ≤ 10。两条规则取最严格的值生效。

**踩坑记录**：

1. **链路流控不生效**：最常见的原因是 `web-context-unify` 默认为 true，导致所有 Context 收敛为同一个名字。必须设置为 false。
2. **链路流控的 refResource 写错**：refResource 是 Context 的名字（入口资源名），不是 Controller 的方法名或接口 URL 参数。可以用上面的 debug 接口确认。
3. **关联流控的关联资源不需要配规则**：关联流控只看关联资源的实时流量，不看关联资源有没有流控规则。很多人误以为需要先给关联资源配置 QPS 限制。
4. **关联流控和直接流控的优先级**：如果一个资源同时有直接流控和关联流控，两者都会生效（先到先判断，任一不满足即 Block）。实际上 Sentinel 的 FlowSlot 会遍历所有 FlowRule，按配置顺序判断，第一个不通过的规则触发 BlockException。

## 4 项目总结

### 4.1 优点与缺点

| 流控模式 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| 直接流控 | 单接口保护 | 简单直接 | 无法感知资源间关系 |
| 关联流控 | 共享下游资源的多个接口 | 自动协调资源分配 | 依赖关联资源的实时指标 |
| 链路流控 | 同一个资源被多个入口调用 | 不同入口差异化控制 | 需理解 Context 机制，Web 场景需额外配置 |

### 4.2 适用场景

- **关联流控**：支付回调影响订单查询、搜索服务影响推荐服务、日志写入影响业务接口——本质上都是共享了同一个瓶颈资源。
- **链路流控**：同一接口在 App 和 Web 端不同限制；免费用户和付费用户共享接口但不同阈值；不同业务线共享同一中台服务。
- **不适用**：关联资源间没有真正的资源竞争关系（关联限流反而浪费了本资源的容量）；链路维度过多导致规则膨胀。

### 4.3 注意事项

1. 关联流控的 `count` 是指**本资源**的阈值，`refResource` 时才检查关联资源的流量。
2. 链路流控的 `refResource` 必须与 `ContextUtil.enter()` 时的 contextName 一致。Web 场景下需要关闭 context-unify。
3. 关联流控和链路流控不能在同一规则上同时设置（strategy 只能选一个值）。
4. 链路流控在非 Web 场景（如 Dubbo、纯 Java 应用）下可以直接用，只要手动调用 `ContextUtil.enter()` 指定入口名。

### 4.4 流控模式选型决策树

```
需要控制的是同一个资源的流量吗？
├── 是 → 不同入口需要不同阈值吗？
│   ├── 是 → 链路流控 (STRATEGY_CHAIN)
│   └── 否 → 直接流控 (STRATEGY_DIRECT)
└── 否 → 两个资源共享瓶颈资源吗？
    ├── 是 → 关联流控 (STRATEGY_RELATE)
    └── 否 → 各自配置直接流控
```

### 4.5 常见踩坑经验

| 故障现象 | 根因 | 解决方案 |
|---------|------|---------|
| 链路流控配置后完全不生效 | web-context-unify=true（默认） | 设置为 false |
| 链路流控限流结果不符合预期 | refResource 与实际 Context 名不一致 | 用 debug 接口确认实际 Context 名 |
| 关联流控把两个接口都限住了 | refResource 写反了（限制了关联资源而非本资源） | 确认规则配置的角色：count 是本资源的阈值 |
| 压测时关联流控反应"慢" | 滑动窗口统计延迟（最长 1 秒） | 正常现象，等待 1-2 秒观察指标即可 |

### 4.6 思考题

1. 如果同一个资源 `queryInventory` 同时配置了直接流控（QPS=100）和链路流控（从入口 A 进入 QPS=50），当从入口 A 进入且 QPS=80 时，请求会被拦截吗？为什么？
2. 关联流控的语义是"当关联资源流量大时限制本资源"，如果关联资源本身也被限流了（比如关联资源 QPS=100，但被限到 50），关联流控的判断基准是被限后 50 还是原始请求 100？

### 4.7 推广计划

- **开发团队**：梳理系统中共享同一瓶颈资源（数据库、缓存、MQ、下游服务）的接口，评估是否需要关联流控。
- **测试团队**：为链路流控设计多入口压测方案，验证不同入口的限流阈值是否正确生效。
- **运维团队**：理解 web-context-unify 配置的作用，在链路流控上线前确认该配置已生效。
