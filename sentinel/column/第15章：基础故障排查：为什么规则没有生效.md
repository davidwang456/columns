# 第15章：基础故障排查：为什么规则没有生效

## 1 项目背景

前 14 章我们学会了配置各种 Sentinel 规则。但现实总是残酷的——"明明配了规则，为什么没生效？"这是社区 issue 区出现频率最高的一类问题。

某公司在生产环境配置了一条流控规则（QPS=100），Dashboard 显示已推送成功，但实际流量达到了 300 QPS 仍然没有任何限流。排查了整整一个下午，最后发现是 `@SentinelResource` 注解的 `value` 写成了 `"createOrder"`，而流控规则的 `resource` 写成了 `"CreateOrder"`——一个字母大小写的差异。

另一次事故中，链路流控规则配置了"从 `/app/detail` 入口进入 `queryStock` 资源时限流 50"，但测试发现毫无效果。最终根因是 `web-context-unify` 默认为 true，所有 Web 请求的入口名被收敛成了同一个名字。

更隐蔽的是：某个服务在上线一周后，监控发现大量的 `BlockException` 被记录为 500 错误而非友好提示。排查发现是 `@ControllerAdvice` 全局异常处理器在 `BlockException` 之前就捕获了所有异常，Sentinel 的限流变成了对用户完全透明的 500 错误。

本章将集中解决初学者最常遇到的 10+ 个"规则不生效"问题，每个问题都会给出根因分析、排查方法和修复方案，帮你形成一套标准的 Sentinel 故障排查 SOP。

## 2 项目设计

**小胖**（崩溃地抓头发）："大师！我搞了三个小时了！流控规则明明配了，但就是不生效！Dashboard 里也看不到拦截日志！"

**大师**："别急，先告诉我你排查了哪几步？"

**小胖**："我就看了 Dashboard，显示规则已推送，客户端在线。"

**大师**："这只是第一步。'规则不生效'可能有几十种原因。我给你一个排查顺序，从上往下查："

| 排查步骤 | 检查点 | 常见原因 |
|---------|-------|---------|
| 1 | 资源是否被 Sentinel 保护 | 未加 @SentinelResource 或未调 SphU.entry() |
| 2 | 资源名是否一致 | 规则中的 resource 和代码中的不一致（大小写、空格） |
| 3 | 规则是否成功加载 | 规则加载时机晚于首次请求，或加载失败 |
| 4 | BlockHandler 是否被拦截 | 全局异常处理器先于 Sentinel 捕获了异常 |
| 5 | Dashboard 是否真正推送 | Dashboard 推送失败但未报错（网络不通/端口不可达） |
| 6 | 链路流控特殊检查 | web-context-unify=true 导致链路收敛 |

**小白**："我觉得第 4 点特别容易被忽略。之前我们有个项目，全局 `@ControllerAdvice` 把 `BlockException` 当普通 Runtime 异常处理了，返回了 500。但用户看到的是 500，不是友好提示。"

**大师**："对。而且更隐蔽的是：如果你没有自定义 BlockExceptionHandler，Spring Cloud Alibaba Sentinel 会自动注册一个默认的 Handler。它会返回 `429 Too Many Requests`，但如果你有自己的 `@ControllerAdvice`，可能把 429 改成 500。"

**小胖**："那 BlockHandler 写了但没执行呢？我在方法上写了 blockHandler='myBlock'，但打死都不走我的 myBlock 方法。"

**大师**："90% 的情况是方法签名写错了。blockHandler 方法的参数必须和原方法一致 + 最后加一个 BlockException。哪怕你多一个空格、少一个参数、参数类型不匹配，都不会报错——它就静默失效。"

**小白**："这确实是个坑。Java 反射不会告诉你'找不到匹配的 blockHandler 方法'，而是在运行时默默跳过。"

**大师**："所以我建议团队用 `@SentinelResource` 时开启一个简单的验证测试：故意触发一次 BlockException，断言返回的消息是你定义的 blockHandler 消息。如果消息不对，说明 blockHandler 没绑定上。"

**大师**："另外，当 Dashboard 和日志都看不出问题时，最后的手段是用 JVM 诊断工具直接看内存中的规则。Arthas 是最常用的——用 `vmtool` 或 `ognl` 直接调用 `FlowRuleManager.getRules()`，看规则内存里有几条、资源名对不对。"

**小白**："我试过 Arthas，但每次定位到 `FlowSlot` 时请求已经过去了。Arthas 能动态打断点吗？"

**大师**："Arthas 的 `tt`（TimeTunnel）可以记录方法入参和返回值。你可以 `tt -t com.alibaba.csp.sentinel.slots.block.flow.FlowSlot entry`，然后触发一次请求，看 `tt -i <index>` 回放的完整参数——包括 `resourceWrapper.getName()` 和 `FlowRuleChecker` 的判断结果。这比打断点侵入性低，适合生产排查。"

**小胖**："那如果用 Arthas 发现规则在内存里是对的，但还是不生效呢？"

**大师**："那就检查 Slot Chain 的执行顺序。用 Arthas 的 `stack` 命令跟踪 `BlockException` 的堆栈：`stack com.alibaba.csp.sentinel.slots.block.flow.FlowException <init>`。如果触发限流时没有这个堆栈，说明 FlowSlot 根本没被调用——可能是 AuthoritySlot 在前面就拦住了，或者 Context 创建失败。"

**技术映射**：
- 规则不生效的最常见根因分类：**资源定义问题**（40%）、**规则加载问题**（20%）、**异常处理链问题**（15%）、**Dashboard 通信问题**（10%）、**版本/配置兼容性问题**（10%）、**其他**（5%）。
- 排查优先级：先确认规则在内存中（`FlowRuleManager.getRules()` 打印），再确认资源在调用链中（Dashboard 簇点链路），最后确认异常处理链。

## 3 项目实战

### 3.1 预备：故障案例环境

我们将构造 6 个真实的"规则不生效"故障案例，逐一排查和修复。

### 3.2 分步实现

**故障案例 1：资源名大小写不一致**

**现象**：`@SentinelResource(value = "createOrder")` 配置了流控规则 QPS=2，但 10 个并发请求全部通过。

**根因**：流控规则中的 `resource` 字段写成了 `"CreateOrder"`。

**排查**：

```java
// 在 Controller 中添加 debug 接口
@GetMapping("/debug/rules")
public String debugRules() {
    List<FlowRule> rules = FlowRuleManager.getRules();
    StringBuilder sb = new StringBuilder("Loaded Flow Rules:\n");
    for (FlowRule rule : rules) {
        sb.append("  resource=").append(rule.getResource())
          .append(", qps=").append(rule.getCount()).append("\n");
    }
    return sb.toString();
}
```

访问 `/debug/rules` 会发现规则加载了，但资源名不匹配。

**修复**：统一资源名为常量。

```java
public static final String RESOURCE_CREATE_ORDER = "createOrder";

@SentinelResource(value = RESOURCE_CREATE_ORDER, ...)
// 规则配置中也使用同一个常量
new FlowRule(RESOURCE_CREATE_ORDER).setCount(10);
```

**故障案例 2：Dashboard 看不到应用**

**现象**：应用已启动，控制台看不到任何应用。

**排查清单**：

1. 确认客户端配置：
```yaml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: localhost:8080  # 检查地址是否正确
        port: 8719                  # 检查端口是否被占用
      eager: true                   # 提前初始化
```

2. 检查心跳日志（客户端日志）：
```bash
grep "register to dashboard" ~/logs/csp/sentinel-record.log
# 期望：Register to dashboard succeeded. server address: localhost:8080
# 失败：Register to dashboard failed. connection refused
```

3. 检查端口可达性：
```bash
# 从 Dashboard 所在服务器测试
curl http://<client-ip>:8719/api/machine
```

**常见根因**：
- Dashboard 地址配错（用了容器内的 localhost，实际应配宿主机 IP）
- 客户端 8719 端口被防火墙拦截
- `project.name` 未配置，Dashboard 无法识别应用

**修复**：Docker 环境中应用连接 Dashboard，如果用 `docker-compose`，应用应用服务名而非 localhost。

**故障案例 3：BlockHandler 静默失效**

**现象**：限流触发了，但返回的不是 blockHandler 定义的消息，而是 "500 Internal Server Error"。

**错误代码**：
```java
// 原方法
@SentinelResource(value = "createOrder", blockHandler = "myBlock")
public String createOrder(String skuId, Integer quantity) {
    return "订单创建成功";
}

// ❌ 错误：缺少 BlockException 参数
public String myBlock(String skuId, Integer quantity) {
    return "系统繁忙";
}
```

**排查**：`@SentinelResource` 找不到匹配的 blockHandler 方法时不会报错，而是返回 Sentinel 默认的错误页面或交给全局异常处理器。

**修复**：
```java
// ✅ 正确：参数包含 BlockException（必须放在最后）
public String myBlock(String skuId, Integer quantity, BlockException e) {
    return "系统繁忙，请稍后再试";
}
```

**故障案例 4：@ControllerAdvice 拦截了 BlockException**

**现象**：限流触发后业务接口返回 500，而非 blockHandler 定义的友好提示。

**错误代码**：
```java
@ControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(Exception.class)
    public ResponseEntity<?> handleException(Exception e) {
        log.error("全局异常", e);
        return ResponseEntity.status(500).body("系统错误");
    }
}
```

**排查**：`BlockException` 继承自 `RuntimeException`，被 `@ExceptionHandler(Exception.class)` 捕获。

**修复**：方案 1——用 `BlockExceptionHandler` 替代 `@ControllerAdvice`，方案 2——在 `@ControllerAdvice` 中单独处理。

```java
// 方案 1：自定义 BlockExceptionHandler（推荐）
@Component
public class SentinelBlockHandler implements BlockExceptionHandler {
    @Override
    public void handle(HttpServletRequest request, HttpServletResponse response,
                       BlockException e) throws Exception {
        response.setStatus(429);
        response.setContentType("application/json;charset=UTF-8");
        response.getWriter().write("{\"code\":\"BLOCK\",\"msg\":\"系统繁忙\"}");
    }
}

// 方案 2：@ControllerAdvice 中单独捕获
@ExceptionHandler(BlockException.class)
public ResponseEntity<?> handleBlock(BlockException e) {
    return ResponseEntity.status(429).body("系统繁忙");
}
```

**故障案例 5：Web 链路收敛导致链路流控不生效**

**现象**：配置了链路流控规则，不同入口对同一资源有不同阈值，但表现一致。

**排查**：

```java
@GetMapping("/debug/context")
public String debugContext(HttpServletRequest request) {
    Context context = ContextUtil.getContext();
    return String.format(
        "Context Name: %s\nOrigin: %s\nRequest URI: %s",
        context != null ? context.getName() : "NULL",
        context != null ? context.getOrigin() : "NULL",
        request.getRequestURI()
    );
}
```

访问 `/goods/detail` 和 `/order/create` 两个 URL，如果 `Context Name` 都显示 `sentinel_web_servlet_context`，说明 Context 被收敛了。

**修复**：
```yaml
spring:
  cloud:
    sentinel:
      web-context-unify: false  # 关闭 Web 上下文收敛
```

重启后再次检查 Context Name 应分别为 `/goods/detail` 和 `/order/create`。

**故障案例 6：规则加载晚于首次请求**

**现象**：应用刚启动时，前几个请求没有被限流，过了一会才生效。

**排查**：规则在 `@PostConstruct` 中加载，但第一个请求在 `@PostConstruct` 完成之前就到达了。

**修复**：
```yaml
spring:
  cloud:
    sentinel:
      eager: true  # 立即初始化 Sentinel Transport，不等首次请求
```

或确保规则在所有 Bean 初始化前完成：

```java
@Configuration
public class SentinelEarlyInit {

    static {
        // 静态初始化块在类加载时执行，早于任何 Bean
        initRules();
    }

    private static void initRules() {
        List<FlowRule> rules = new ArrayList<>();
        rules.add(new FlowRule("createOrder").setGrade(FLOW_GRADE_QPS).setCount(10));
        FlowRuleManager.loadRules(rules);
    }
}
```

```bash
arthas@pid$ ognl '@com.alibaba.csp.sentinel.adapter.spring.webmvc.SentinelWebInterceptor@registrySize()'

arthas@pid$ stack com.alibaba.csp.sentinel.context.ContextUtil enter -n 3
```

```markdown
## Sentinel 规则不生效排查 SOP

### 第一步：确认资源被保护（30 秒）
- [ ] 访问 Dashboard → 簇点链路 → 是否看到该资源？
- [ ] 如果没有：检查是否加了 @SentinelResource 或 SphU.entry()

### 第二步：确认规则已加载（30 秒）
- [ ] Dashboard → 流控规则 → 是否有对应规则？
- [ ] 代码调试：打印 `FlowRuleManager.getRules().size()`
- [ ] 检查规则加载日志：grep "Flow rules loaded" sentinel-record.log

### 第三步：确认资源名一致（1 分钟）
- [ ] 规则中的 resource 字段是否与代码中完全一致（大小写、空格）？
- [ ] 建议用常量定义资源名

### 第四步：确认 BlockHandler 正确绑定（2 分钟）
- [ ] blockHandler 方法签名是否包含 BlockException 参数（最后一个）？
- [ ] 返回类型是否与原方法一致？
- [ ] blockHandler 是否在同一个类中（或指定了 blockHandlerClass）？

### 第五步：确认异常处理链（2 分钟）
- [ ] 是否有 @ControllerAdvice 捕获了 BlockException？
- [ ] 是否自定义了 BlockExceptionHandler？

### 第六步：链路流控专项检查（2 分钟）
- [ ] web-context-unify 是否为 false？
- [ ] 链路流控的 refResource 是否与实际 Context Name 一致？

### 第七步：Dashboard 通信检查（3 分钟）
- [ ] 客户端是否在线（Dashboard 机器列表）？
- [ ] 8719 端口是否可达？
- [ ] 规则推送日志是否有错误？

### 第八步：版本兼容检查（2 分钟）
- [ ] Sentinel 版本与 Spring Cloud Alibaba 版本是否兼容？
- [ ] JDK 版本是否有反射/模块化兼容问题？
```

**踩坑记录总结**：

| 故障 | 一句话根因 | 一句话修复 |
|------|----------|----------|
| 资源名不一致 | 大小写/空格 | 用常量 |
| Dashboard 无应用 | 网络不通 | 检查地址和端口 |
| blockHandler 不执行 | 方法签名错 | 加 BlockException 参数 |

## 4 项目总结

### 4.1 排查原则

1. **由近及远**：先查代码（资源定义、规则配置），再查基础设施（Dashboard、网络）。
2. **证据驱动**：不要在头脑中猜测，用日志、Dashboard、JVM 运行时信息来验证。
3. **二分排除**：如果多种规则同时不生效，先只保留一条最简单规则来缩小范围。
4. **最小复现**：在本地用最简单的 Demo 复现问题，避免生产环境的复杂度干扰排查。

### 4.3 排查工具速查表

| 工具 | 适用场景 | 核心命令/操作 | 特点 |
|------|---------|-------------|------|
| Dashboard 簇点链路 | 确认资源是否被保护 | 查看资源列表 | 直观，但需要网络和心跳正常 |
| `FlowRuleManager.getRules()` | 确认规则是否在内存中 | 暴露 Debug endpoint | 最可靠，不依赖 Dashboard |
| sentinel-record.log | 确认规则加载/心跳 | `grep "Flow rules loaded"` | 历史可追溯 |
| Arthas vmtool | 在线诊断，不重启服务 | `vmtool --action getInstances` | 适合生产环境 |
| Arthas tt | 记录方法调用参数 | `tt -t FlowSlot entry` | 可回放，侵入性低 |
| Arthas ognl | 直接调用静态方法 | `ognl '@FlowRuleManager@getRules()'` | 即时查看运行时状态 |
| JMX / JConsole | 查看 JVM 指标 | MBean `sentinel.*` | 标准协议，跨平台 |
| `/debug/context` endpoint | 确认 Context 名 | 自定义 debug 接口 | 代码级验证 |

### 4.4 思考题

1. 如果一个资源同时配置了 `@SentinelResource` 和 Web MVC 自动保护（通过 `SentinelWebInterceptor`），哪个优先生效？为什么？
2. 在排查"规则不生效"时，你明明看到 Dashboard 显示规则已推送且客户端在线，但实际没有拦截效果。你怀疑是客户端收到了规则但忽略了。你如何通过 JVM 诊断（如 Arthas 或 JMX）来验证客户端内存中是否真的有这条规则？
