# 第24章：链路统计与调用树设计

## 1 项目背景

前 23 章我们都在"使用" Sentinel 的规则，但从未深入理解 Sentinel 是如何统计资源调用关系的。运维团队在一个深夜排查中发现了一个诡异现象：

Dashboard 显示 `createOrder` 资源的 QPS 是 500，但同时显示它的"上游入口"有 5 个不同的 Context Name，但下游内部调用的 `queryInventory` 资源的 QPS 却只有 200。为什么入口 500 QPS 但内部调用只有 200？是不是有 300 QPS 在内部被"吞掉"了？

排查后发现，300 个请求实际上是被 Controller 层的一个 if 条件提前返回了（不需要库存查询的场景），并没有走到 `queryInventory` 的内部调用。但从 Dashboard 的"簇点链路"来看，这两个资源没有显示的调用关系——因为 Sentinel 的链路树不是自动关联的，而是依赖 Context 的传播。

另一个问题是：为什么 Dashboard 上有些资源永远看不到"下游调用"？为什么 `@SentinelResource` 注解在 Service 层的方法调用不会被 Dashboard 展示为链路关系？

这些问题的答案都在 Sentinel 的链路统计与调用树设计中。

## 2 项目设计

**小胖**（点开 Dashboard 的簇点链路）："为什么 `createOrder` 的 QPS 是 500，但 `queryInventory` 只有 200？另外 300 个请求去哪了？"

**大师**："并不是所有 `createOrder` 请求都会调用 `queryInventory`。你的代码里可能有条件分支。Dashboard 显示的是独立资源的统计，不反映真实的调用关系。"

**小白**："那 Sentinel 内部的调用树是怎么建的？我翻源码看到了 EntranceNode、DefaultNode、ClusterNode 三个类。"

**大师**："这三个 Node 是理解 Sentinel 链路统计的关键。我给你画个树："

```
Context Tree (每个 Context 一棵树)
└── EntranceNode: sentinel_web_servlet_context (入口)
    └── DefaultNode: createOrder (资源)
        ├── ClusterNode: createOrder (该资源的集群统计)
        └── DefaultNode: queryInventory (子资源调用)
            └── ClusterNode: queryInventory (该资源的集群统计)
```

**小胖**："所以 DefaultNode 是'某个上游入口下该资源的统计'，ClusterNode 是'该资源所有入口的汇总统计'？"

**大师**："完全正确！DefaultNode 是链路限流的基础——你需要限制'从入口 A 进来的资源 X'时，看的其实就是 DefaultNode。ClusterNode 是全局汇总——配置普通限流时看的就是它。"

**小白**："那 `EntranceNode` 是怎么创建的？我翻代码看到了 `ContextUtil.enter()` 但没看到 EntranceNode 的构造函数。"

**大师**："EntranceNode 在 `ContextUtil.trueEnter()` 中被隐式创建——它通过 `Constants.ROOT` 作为 parent，第一次进入某个 Context Name 时创建。调用链是这样的：`ContextUtil.enter(name, origin)` → `new Context(node, name)` → `node` 就是 EntranceNode。如果同一个 name 已经存在，会复用已有 EntranceNode。"

**小胖**："Dashboard 上的'簇点链路'图是怎么画出来的？它是基于 DefaultNode 的父子关系还是基于 Entry 的嵌套关系？"

**大师**："基于 `DefaultNode.getChildList()` 的树形结构。每次 `SphU.entry()` 创建一个 Entry 时，Sentinel 会把当前 Entry 的 Node 作为子节点挂到上一个 Entry 的 Node 上。所以你看到的链路图实际上是'在当前调用上下文中的嵌套 Entry 关系'的投影。"

**小白**："如果同一个资源在同一个 Context 中被调用了两次（比如 A() → B() → B()，B 被递归调用），调用树会怎样？"

**大师**："DefaultNode 是唯一的——每个 Context 下每个 Resource 只有一个 DefaultNode。所以递归调用 B 时，不会创建新的 DefaultNode，只是在已有的 B 的 DefaultNode 上累加统计。但 Entry 是嵌套的——每次 `SphU.entry("B")` 都创建一个新 Entry，退出顺序是后进先出。"

**小胖**："那 Controller 层和 Service 层如果用了不同的 Context 会怎样？比如 /order/create 入口创建一个 Context，但 Service 方法内部又 `ContextUtil.enter("customContext")` 创建了新 Context？"

**大师**："这会创建两个独立的调用树——两者之间没有父子关系。Dashboard 上看不到 /order/create → orderService 的链路关系。这就是为什么生产环境推荐统一使用一个 Context（通过 `web-context-unify=true`），避免人为创建 Context 导致链路断裂。"



**技术映射**：
- `EntranceNode`：每个 Context（入口）的根节点，描述"请求从哪个门进来的"。在 Web 场景中如果 `web-context-unify=false`，每个 URL 是一个入口。
- `DefaultNode`：资源在特定调用路径下的节点，每个资源在每个 Context 下最多有一个 DefaultNode。
- `ClusterNode`：资源的全局统计节点，不区分入口，一个资源只有一个 ClusterNode。
- 调用树的生命周期：Context 创建时建树 → 请求经过各资源时在树上添加/更新 Node → Context exit 时清理（但 Node 对象不立即回收，而是复用）。

## 3 项目实战

### 3.1 步骤一：打印调用树结构

```java
@RestController
public class CallTreeDebugController {

    @GetMapping("/debug/tree")
    public String debugCallTree() {
        Context context = ContextUtil.getContext();
        if (context == null) {
            return "Context 为空（非 Sentinel 管理的线程）";
        }

        StringBuilder sb = new StringBuilder();
        sb.append("=== 当前 Context 调用树 ===\n");
        sb.append("Context Name: ").append(context.getName()).append("\n");
        sb.append("Origin: ").append(context.getOrigin()).append("\n");
        sb.append("EntranceNode: ").append(context.getEntranceNode()).append("\n");

        // 遍历当前 Context 下的所有 Node
        DefaultNode entranceNode = (DefaultNode) context.getEntranceNode();
        printNodeTree(entranceNode, sb, 0);

        return sb.toString();
    }

    private void printNodeTree(DefaultNode node, StringBuilder sb, int depth) {
        String indent = "  ".repeat(depth);
        sb.append(indent).append("├── Resource: ").append(node.getId().getName())
          .append(", Type: ").append(node.getId().getType())
          .append(", Pass QPS: ").append(node.passQps())
          .append(", RT: ").append(node.avgRt()).append("ms")
          .append("\n");

        // 子节点
        for (DefaultNode child : node.getChildList()) {
            printNodeTree(child, sb, depth + 1);
        }
    }
}
```

访问 `/debug/tree` 输出示例：

```
=== 当前 Context 调用树 ===
Context Name: sentinel_web_servlet_context
Origin: 
EntranceNode: entranceNode[sentinel_web_servlet_context]
├── Resource: GET:/debug/tree, Type: IN, Pass QPS: 1, RT: 2ms
```

### 3.2 步骤二：多层级调用链演示

```java
@RestController
public class ChainDemoController {

    @Autowired
    private ChainServiceA serviceA;

    @GetMapping("/chain/level1")
    public String level1() {
        return serviceA.doLevel2();
    }
}

@Service
public class ChainServiceA {

    @SentinelResource("level2Resource")
    public String doLevel2() {
        return "Level 2 → " + doLevel3();
    }

    @SentinelResource("level3Resource")
    public String doLevel3() {
        return "Level 3";
    }
}
```

访问 `/chain/level1` 后，调用树变化：

```
EntranceNode: /chain/level1
├── DefaultNode: level2Resource
│   └── DefaultNode: level3Resource
```

Dashboard 中可以在"簇点链路"看到这三个资源的调用关系。

**注意**：如果 `level3Resource` 是通过 `this.doLevel3()` 调用的（同类调用），则 Spring AOP 不会生效，链路会断裂。必须通过注入的 Bean 调用才能被 Sentinel 感知。

### 3.3 步骤三：Context 传播注意事项

```java
// 线程池中异步任务 —— Context 不会自动传播
@GetMapping("/chain/async")
public String asyncChain() {
    // 主线程中获取当前 Context
    Context mainContext = ContextUtil.getContext();

    new Thread(() -> {
        // ❌ 错误：新线程中 Context 为空
        // ContextUtil.getContext() == null

        // ✅ 正确：手动传递 Context
        ContextUtil.runOnContext(mainContext, () -> {
            try (Entry entry = SphU.entry("asyncResource")) {
                return "异步任务完成";
            } catch (BlockException e) {
                return "异步任务被限流";
            }
        });
    }).start();

    return "异步任务已提交";
}
```

### 3.5 步骤五：验证 DefaultNode vs ClusterNode 统计差异

```java
@RestController
public class NodeDiffController {

    @GetMapping("/debug/node-diff")
    public String diff() {
        // 获取 createOrder 的 ClusterNode（全局汇总）
        ClusterNode clusterNode = ClusterBuilderSlot
            .getClusterNode("createOrder");

        // 获取当前 Context 下的 DefaultNode（链路统计）
        Context context = ContextUtil.getContext();
        DefaultNode defaultNode = ((DefaultNode) context.getEntranceNode())
            .getChildList().stream()
            .filter(n -> n.getId().getName().equals("createOrder"))
            .findFirst().orElse(null);

        StringBuilder sb = new StringBuilder();
        sb.append("=== ClusterNode (全局) ===\n");
        sb.append("Pass QPS: ").append(clusterNode.passQps()).append("\n");
        sb.append("Avg RT: ").append(clusterNode.avgRt()).append("ms\n");
        sb.append("Block QPS: ").append(clusterNode.blockQps()).append("\n\n");

        if (defaultNode != null) {
            sb.append("=== DefaultNode (当前入口下) ===\n");
            sb.append("Pass QPS: ").append(defaultNode.passQps()).append("\n");
            sb.append("Avg RT: ").append(defaultNode.avgRt()).append("ms\n");
            sb.append("Block QPS: ").append(defaultNode.blockQps()).append("\n");
        }

        // 关键差异：
        // - 如果从多个入口调用 createOrder，ClusterNode 的 QPS = 所有入口之和
        // - DefaultNode 的 QPS = 仅当前入口的统计
        return sb.toString();
    }
}
```

### 3.6 步骤六：Context 命名策略验证

```java
@Configuration
public class SentinelContextConfig implements WebContextUnifyHandler {

    // web-context-unify=false 时，每个 URL 是一个独立 Context
    // 如果设为 true，所有 Web 请求共用同一个 Context（推荐）

    @Override
    public boolean shouldUnify(String servletPath) {
        // 返回 true → 所有 Web 请求统一 Context
        // 返回 false → 每个 URL 独立 Context
        return true;
    }
}

// 验证不同 Context 策略下的调用树差异
@GetMapping("/context-demo/a")
public String contextA() {
    // 如果 web-context-unify=true：
    //   Context Name = "sentinel_web_servlet_context"（统一）
    // 如果 web-context-unify=false：
    //   Context Name = "/context-demo/a"（独立）
    return "Context: " + ContextUtil.getContext().getName();
}
```

### 3.7 步骤七：Node 内存占用精确测算

```java
@GetMapping("/debug/node-memory")
public String nodeMemory() {
    Map<ResourceWrapper, ClusterNode> map =
        ClusterBuilderSlot.getClusterNodeMap();

    // 遍历所有 ClusterNode，统计每个 Node 的 child DefaultNode 数量
    int totalDefaultNodes = 0;
    long totalMetricBuckets = 0;

    for (ClusterNode cn : map.values()) {
        // 估算 MetricBucket 占用（每个 LongAdder 数组 ~ 7 个元素 × 16 字节 ≈ 112 字节）
        totalMetricBuckets += cn.getLeapArray().list().size();
        totalDefaultNodes += cn.getOriginCountMap().size();
    }

    return String.format(
        "ClusterNode: %d\nDefaultNode: %d\nMetricBucket: %d\n估算内存: %.2f MB",
        map.size(), totalDefaultNodes, totalMetricBuckets,
        (map.size() * 500L + totalDefaultNodes * 300L + totalMetricBuckets * 150L) / 1048576.0
    );
}
```

**踩坑记录**：

1. **资源太多导致 Node 膨胀**：如果资源按 `userId` 维度定义（每个用户一个资源），ClusterNode 数量会爆炸（百万级别）。资源粒度应该控制在 1000 以内。
2. **Context 泄露**：在线程池中使用了 `ContextUtil.enter()` 但没 `exit()` 会导致 Context 链表无限增长。
3. **异步调用链路断裂**：默认情况下新线程不会自动继承父线程的 Context，需要用 `ContextUtil.runOnContext()` 手动传递。

## 4 项目总结

### 4.1 核心概念对照表

| 概念 | 数量 | 生命周期 | 用途 |
|------|------|---------|------|
| EntranceNode | 每个 Context 一个 | Context 创建/销毁 | 标识入口 |
| DefaultNode | 每个 Context 下每个资源一个 | 随 Context 复用 | 链路限流统计 |
| ClusterNode | 每个资源全局一个 | 应用生命周期 | 普通限流统计 |

### 4.2 Context 统一模式 vs 独立模式

| 配置 | web-context-unify=true | web-context-unify=false |
|------|----------------------|------------------------|
| Context 数量 | 1 个 | N 个（每个 URL 一个） |
| EntranceNode 数量 | 1 个 | N 个 |
| 链路限流能力 | 弱（所有 URL 共用入口） | 强（可按 URL 区分入口） |
| Dashboard 链路图 | 扁平的星型图 | 多棵树 |
| 内存开销 | 低 | 较高（N 倍 EntranceNode） |
| 推荐场景 | 通用服务 | 需要按入口 URL 限流的网关 |

### 4.3 Node 对象内存估算

| 对象 | 估算大小 | 包含 |
|------|---------|------|
| ClusterNode | ~500 字节 | MetricBucket + LeapArray + child map |
| DefaultNode | ~300 字节 | MetricBucket + LeapArray + ClusterNode 引用 |
| EntranceNode | ~200 字节 | 基础 Node + Context 引用 |
| MetricBucket | ~150 字节 | 7 个 LongAdder 实例 |

### 4.4 链路断裂常见原因排查清单

- [ ] 同类方法调用（`this.method()`）—— Spring AOP 无法拦截，`@SentinelResource` 不生效
- [ ] 异步线程没有传递 Context —— 需手动 `ContextUtil.runOnContext()`
- [ ] 手动创建了新 Context —— 调用树脱离主 Context，Dashboard 看不到关系
- [ ] web-context-unify=true 掩盖了链路关系 —— 所有入口共用 Context
- [ ] 使用了 `SphO.entry()` 而非 `SphU.entry()` —— 前者不参与统计

### 4.5 思考题

1. 为什么 Dashboard 在"链路"模式下限流时需要指定 `refResource`？这个 refResource 对应的是 EntranceNode、DefaultNode 还是 ClusterNode？
2. 如果你有 100 个 Controller 方法，每个方法都通过 `@SentinelResource` 定义了不同的资源名，Sentinel 内部会创建多少个 EntranceNode？多少个 DefaultNode？多少个 ClusterNode？

### 4.6 推广计划

- **开发团队**：在设计 Sentinel 资源命名时，理解 DefaultNode 和 ClusterNode 的区别，避免命名过细导致资源膨胀。
- **测试团队**：验证异步调用链中 Context 是否正确传播，防止链路限流失效。
