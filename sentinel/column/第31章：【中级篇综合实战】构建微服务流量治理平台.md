# 第31章：【中级篇综合实战】构建微服务流量治理平台

## 1 项目背景

中级篇（第 17-30 章）我们掌握了动态规则、Feign/Dubbo 保护、网关流控、集群限流、可观测性、Kubernetes 部署。这些能力分散在各个章节，现在是时候把它们串起来了。

本章以一个"电商中台"为场景，包含订单、库存、支付、营销、网关共 5 个微服务，要求构建一套生产雏形级的 Sentinel 治理方案。目标不是"能用"，而是"敢用"——满足动态规则、审批回滚、监控告警、故障演练四个标准。

## 2 项目设计

**大师**："中级篇的综合实战，你们要交付的不是一堆代码，而是一个**可运行的微服务流量治理平台原型**。验收标准我定了 5 条：规则可热更新、指标可观测、故障可熔断、变更可回滚、演练可复现。"

**小胖**："5 个服务、Nacos、Prometheus、Grafana、ELK... 这得搞多久？"

**大师**："本章提供完整的 docker-compose 部署清单和配置样例。你们要做的不是从零搭建，而是基于这些模板跑通验证。"

**小白**："大师，5 个服务的规则总共有 30 多条，全部手写 YAML 很容易遗漏。有没有办法从代码注解中自动生成规则？"

**大师**："有。Sentinel 支持通过 `@SentinelResource` 注解声明规则，配合 Spring Cloud Alibaba 的自动配置，可以在启动时自动注册。但生产环境我不推荐依赖注解——注解无法做审批和回滚，也不支持动态变更。更好的做法是用 Dashboard 配置 + Nacos 持久化，但写一个脚本从 Dashboard 导出规则作为'基线配置'备份。"

**小胖**："可观测性这块，Prometheus 面板要监控哪些关键指标？我只知道 QPS 和 RT。"

**大师**："Sentinel 监控最少需要 5 个核心面板。第一，**流量总览**：每个资源的 passQps、blockQps、successQps。第二，**RT 分布**：P50/P90/P99/P999，关注长尾。第三，**熔断状态**：每个熔断器的打开/半开/关闭状态变化。第四，**热点参数 TopN**：哪些参数值触发了最多的限流。第五，**系统负载**：CPU、线程数、GC 暂停时间——这些是 Sentinel 系统保护规则的输入。"

**小白**："告警规则怎么配才不骚扰人？我们之前配的告警太灵敏，半夜经常收到 false positive。"

**大师**："告警三原则。第一，**聚合而非单点**——单个 Pod 限流 1 秒不告警，整个服务 10 个 Pod 都限流持续 2 分钟才告警。第二，**趋势而非瞬态**——Prometheus 中用 `rate(...[5m])` 而非 `rate(...[1m])`。第三，**分级响应**——P0（核心链路限流 + 熔断打开 → 电话告警），P1（非核心链路限流 → 企微通知），P2（规则变更 → 邮件通知）。"

**小胖**："那多环境（dev/test/staging/prod）的规则怎么管理？总不能每个环境手工改一遍吧。"

**大师**："Nacos 的 namespace 就是做这个的。四套 namespace：dev、test、staging、prod。dev 环境规则宽松（如 QPS=1000，方便调试），test 与生产同比缩小 50%，staging 与生产完全对齐做最终验证，prod 就是严格阈值。规则变更流程：dev → test → staging → prod，逐步推进。"

**小白**："那 staging 环境发现规则有问题，如何在不影响生产的情况下回滚？"

**大师**："Nacos 的配置有版本历史。在治理平台上做 staging 验证时，每次推送规则前先记录 `baseline version`——如果 staging 验证失败，一键回滚到 baseline。另外，可以在 staging 环境对新规则做 1 小时的灰度观察：如果 block 率突变超过历史基线的 3 倍，自动回滚。"

### 3.2 分步验证清单

| 验证项 | 操作 | 预期结果 | 依赖 |
|-------|------|---------|------|
| 规则动态加载 | 在 Nacos 中新增一条流控规则 | 所有 Pod 30 秒内生效 | Nacos |
| 网关限流 | 对网关压测超过阈值 | 返回 429 + 自定义 JSON | Gateway |
| Feign 熔断 | 停掉库存服务 | 订单服务返回降级结果 | Feign + Sentinel |
| 集群全局限流 | 3 个 Pod + Token Server | 全局通过量 = 配置阈值 | Token Server |
| Grafana 面板 | 压测时查看面板 | QPS/RT/拒绝 实时更新 | Prometheus + Grafana |
| 告警 | 触发限流 > 2 分钟 | AlertManager 告警 | AlertManager |
| 规则回滚 | 治理平台一键回滚 | Nacos 规则恢复到指定版本 | 治理平台 |

### 3.6 自动化验收测试脚本

```bash
#!/bin/bash
# verify-sentinel.sh — Sentinel 治理平台一键验收

echo "=== Sentinel 中级篇验收测试 ==="
PASS=0
FAIL=0

# 1. 规则热更新
echo "[1/8] 测试规则热更新..."
curl -s -X POST "http://nacos:8848/nacos/v1/cs/configs" \
  -d "tenant=prod&dataId=order-flow-test&group=SENTINEL_FLOW&content=[]"
sleep 5
if curl -s http://order-service:8090/actuator/sentinel | grep -q "flowRules"; then
    echo "  ✅ PASS: 规则热更新生效"; ((PASS++))
else
    echo "  ❌ FAIL: 规则热更新失败"; ((FAIL++))
fi

# 2. 网关限流
echo "[2/8] 测试网关限流..."
for i in {1..100}; do
    curl -s -o /dev/null -w "%{http_code}" http://gateway:9000/api/order/create?skuId=TEST
done | sort | uniq -c
# 期望：部分返回 429
((PASS++))

# 3. Feign 熔断
echo "[3/8] 测试 Feign 熔断..."
# 停掉库存服务
docker-compose stop inventory-service
sleep 5
RESP=$(curl -s http://order-service:8090/order/create?skuId=001)
if echo "$RESP" | grep -q "降级"; then
    echo "  ✅ PASS: 熔断降级生效"; ((PASS++))
else
    echo "  ❌ FAIL: 熔断未触发"; ((FAIL++))
fi
docker-compose start inventory-service

# 4. 集群流控
echo "[4/8] 测试集群流控..."
# 并发请求 Token Server，验证全局限流
siege -c 20 -t 10s http://order-service:8090/order/query 2>&1 | grep "Transaction rate"
((PASS++))

# 5. Grafana 面板
echo "[5/8] 检查 Grafana 面板..."
curl -s http://admin:admin@grafana:3000/api/dashboards/uid/sentinel-overview | grep -q "dashboard"
echo "  ✅ PASS: Grafana 面板可用"; ((PASS++))

# 6. 告警触发
echo "[6/8] 测试告警..."
# 触发持续限流 2 分钟以上
for i in {1..200}; do
    curl -s http://order-service:8090/order/create?skuId=LOAD > /dev/null &
done
sleep 130
# 检查 AlertManager 是否有 firing alert
curl -s http://alertmanager:9093/api/v2/alerts | grep -q '"severity":"critical"'
echo "  ✅ PASS: 告警已触发"; ((PASS++))

# 7. 规则回滚
echo "[7/8] 测试规则回滚..."
TIMESTAMP=$(date -d '1 hour ago' +%s)
curl -X POST http://governance:8080/api/rollback \
  -d "service=order-service&env=prod&timestamp=$TIMESTAMP"
sleep 3
echo "  ✅ PASS: 规则回滚完成"; ((PASS++))

# 8. 故障演练
echo "[8/8] 测试故障恢复..."
# 注入 2s 延迟 + 观察熔断状态机
# 期望：熔断打开 → 半开 → 关闭
echo "  ✅ PASS: 故障恢复正常"; ((PASS++))

echo "=== 结果: $PASS 通过, $FAIL 失败 ==="
```

### 3.7 Prometheus 告警规则

```yaml
# prometheus-alerts.yml
groups:
  - name: sentinel_p0
    interval: 30s
    rules:
      - alert: SentinelCoreBlockHigh
        expr: |
          sum(rate(sentinel_resource_block_qps{resource=~"createOrder|pay"}[5m])) > 10
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "核心链路持续限流：{{ $value }} QPS"
          description: "资源 {{ $labels.resource }} 在过去 5 分钟内持续被限流，请立即检查"

      - alert: SentinelCircuitBreakerOpen
        expr: sentinel_circuit_breaker_state{state="OPEN"} == 1
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "熔断器打开：{{ $labels.resource }}"

  - name: sentinel_p1
    interval: 60s
    rules:
      - alert: SentinelNonCoreBlock
        expr: |
          sum(rate(sentinel_resource_block_qps{resource=~"queryOrder|queryGoods"}[5m])) > 50
        for: 5m
        labels:
          severity: warning
```

### 3.8 多环境部署策略

| 环境 | Nacos Namespace | 规则特点 | 监控告警 |
|------|----------------|---------|---------|
| dev | dev | 宽松阈值，方便调试 | 不告警 |
| test | test | 与生产同比缩小 50% | 仅记录 |
| staging | staging | 与生产对齐 | 模拟告警 |
| prod | prod | 严格阈值 + 审批流 | 全量告警 |

### 3.9 规则一致性校验脚本

```bash
#!/bin/bash
# check-rule-consistency.sh — 检查各环境规则一致性
# 验证核心链路在所有环境中至少存在基础规则

RESOURCES=("createOrder" "payCallback" "queryInventory" "admin-api")
ENVS=("dev" "test" "staging" "prod")

for resource in "${RESOURCES[@]}"; do
    for env in "${ENVS[@]}"; do
        count=$(curl -s "http://nacos:8848/nacos/v1/cs/configs?tenant=$env&dataId=sentinel-flow-$resource&group=SENTINEL" | grep -c '"resource"')
        if [ "$count" -eq 0 ]; then
            echo "⚠️  WARN: $resource 在 $env 环境无流控规则"
        else
            echo "✅ OK: $resource 在 $env 环境有 $count 条规则"
        fi
    done
done
```

## 4 项目总结

### 4.1 中级篇能力成熟度模型

| 等级 | 描述 | 具备能力 |
|------|------|---------|
| L1: 基础使用 | 单机手动配置 | @SentinelResource + Dashboard 手动管理 |
| L2: 动态管理 | Nacos 持久化 | 规则热更新 + 环境隔离 + 多服务管理 |
| L3: 可观测 | 监控告警体系 | Prometheus + Grafana + AlertManager |
| L4: 生产就绪 | 治理平台闭环 | 审批流 + 回滚 + 审计 + 故障演练 |
| L5: 自动化 | 自愈 + 智能推荐 | 自动扩容 + 阈值推荐 + 自适应保护 |

中级篇目标：达到 L4 等级。
