# 第23章：集群热点参数限流与大促保护

## 1 项目背景

第 22 章的集群流控解决了"多实例整体 QPS 不准"的问题，但还有一个场景没有覆盖：**热点参数的集群级别控制**。

以去年的"双十一"为例：某电商平台有 5 个商品服务实例，每个实例配置了热点参数限流（`skuId` 维度，默认 200 QPS）。活动开始后，爆款商品 `SKU_HOT_001` 的流量达到了 1000 QPS，但 5 个实例的流量分布不均匀——实例 1 收到了 300 QPS，实例 2 收到 250 QPS，实例 3 收到 200 QPS，实例 4 收到 150 QPS，实例 5 收到 100 QPS。

结果：实例 1 的 `SKU_HOT_001` 被限流了 100 QPS（200 阈值），但实例 4 和 5 还有大量余量。**从全局看，这个 SKU 实际上还有 150 QPS 的可用容量，但已经有 100 QPS 的请求被拒绝了**。

这就是**集群热点参数限流**的价值：为热点参数（如爆款 SKU、明星用户 ID）在集群层面统一分配配额，避免局部限流浪费全局容量。

除此之外，大促场景还有特殊的运维需求：活动开始前需要预热规则、活动进行中可能需要动态调整阈值、活动结束后需要快速回滚规则。

## 2 项目设计

**小胖**："大师，集群流控是管全局 QPS，那热点参数呢？我配了 skuId 限流 200，但 5 个实例各自为政。"

**大师**："集群热点参数限流的原理和普通集群流控类似——需要部署一个 ParamFlowTokenServer 来统一管理每个参数值的令牌。所有 Client 同一个 hotspot 参数的请求都向 Token Server 申请令牌。"

**小白**："热点参数的 Cardinality 问题呢？如果有 10 万个热门 skuId，Token Server 要维护 10 万个计数器？"

**大师**："这就是关键风险。单机热点参数限流有 LRU 缓存上限（默认 4000），集群模式同样需要限制。所以集群热点参数限流只适用于 **TopN 热点**——需要先通过日志分析或监控识别出哪些参数值是真的热点，只为这些热点值单独配置集群规则。"

**小胖**："大促前我们一般怎么准备 Sentinel 规则？"

**大师**："三步走——预案制定、压测验证、实时监控。大促的规则不是平时规则的简单放大，而是有针对性的调整：热点阈值上调、无关紧要的接口降级、关键链路保障。"

**小白**："那大促结束后的规则回收也很重要吧？我见过有团队大促结束后忘了把阈值改回来，结果一个多月后才发现日常流量早已超过大促阈值——但因为日常流量还没到那个量级，所以一直没有触发限流，等于规则形同虚设。"

**大师**："所以大促规则一定要带'有效期'。在规则元数据中标记 `effective_time` 和 `expire_time`，治理平台定时检查，到期自动回滚。还有一种做法是用 Crontab——大促开始时自动推送大促规则，结束时自动切回基线规则，全程不需要人工参与。"

**小胖**："大师，集群热点参数限流的 Token Server 挂了怎么办？我配了 `fallbackToLocalWhenFail=true`，但本地单机阈值只有 200，全局阈值是 500。Token Server 挂了之后，5 个实例每个都限 200，实际全局变成了 1000——反而放大了。"

**大师**："这是集群模式的经典困境。对 `fallbackToLocalWhenFail`，你有三个选择：一是设置本地退回阈值 = 全局阈值 / 实例数（保守方案，保证不超）；二是高可用 Token Server——部署 2-3 个实例，用 Raft 协议保证一致性；三是容忍短暂的超额——Token Server 挂了后，过 30 秒切换到本地模式，同时触发告警让运维 5 分钟内修复。"

**小白**："热点参数限流的例外项配置感觉很容易过时——大促前配了 Top 20 热点 SKU，活动开始后发现实际上 Top 5 变了，新增了一个社交传播的爆款。"

**大师**："这就是**动态热点发现**的价值。你可以写一个定时任务，每分钟从 Prometheus 查询 `topk(10, sum(rate(sentinel_param_flow_qps[1m])) by (param_value))`，自动对比当前配置的例外项，如果发现 Top 3 中有未配置的 SKU，自动添加例外项或发送告警通知运维。"

**小胖**："大师，Token Server 在集群热点模式下处理一个参数值请求大概要多久？会影响整体 RT 吗？"

**大师**："Token Server 的请求是异步的，默认通过 Netty 客户端发送，P99 延迟通常在 1-3ms。对于 RT 在 50ms+ 的业务接口，3ms 的额外开销可以忽略。但如果你有 1000 个热点参数值、每个每秒都在请求 Token Server，那 Token Server 本身的 CPU 可能成为瓶颈——需要监控 Token Server 的线程池和队列积压。"

**小白**："大促场景下，有没有比集群热点参数限流更好的替代方案？比如直接用 Redis 做个全局计数器？"

**大师**："Redis 计数器方案确实更灵活——可以精确到任意维度、不受缓存上限限制、天然集群化。但代价是：每次请求都要一次 Redis 网络 IO，P99 延迟至少增加 2-5ms。Sentinel 的集群热点参数限流的优势在于：Token Server 在同一个机房内网络延迟极低（< 1ms），且 LRU 缓存命中后本地判断，不产生网络开销。所以适用场景不同：高频热点用 Sentinel，低频全量用 Redis。"

### 3.2 分步实现

**步骤一：配置集群热点参数规则**

```java
@Component
public class ClusterParamFlowConfig {

    @PostConstruct
    public void init() {
        List<ParamFlowRule> rules = new ArrayList<>();

        ParamFlowRule rule = new ParamFlowRule("goodsDetail")
                .setParamIdx(0)           // 参数索引 0 = skuId
                .setCount(500)            // 全局热点参数阈值（集群级别）
                .setClusterMode(true)     // 开启集群模式
                .setClusterConfig(
                    new ParamFlowClusterConfig()
                        .setFlowId(100L)
                        .setThresholdType(ClusterRuleConstant.FLOW_THRESHOLD_GLOBAL)
                        .setFallbackToLocalWhenFail(true)
                );

        // 例外项：超级爆款可以到 1000 QPS
        ParamFlowItem hotItem = new ParamFlowItem()
                .setObject("SUPER_HOT_SKU")
                .setClassType(String.class.getName())
                .setCount(1000);
        rule.setParamFlowItemList(Collections.singletonList(hotItem));

        rules.add(rule);
        ParamFlowRuleManager.loadRules(rules);
    }
}
```

**步骤二：大促前的规则准备清单**

| 阶段 | 操作 | 规则变更 |
|------|------|---------|
| 预热期（活动前 2 小时） | 将所有流控规则的 Warm Up 效果开启 | `controlBehavior: 1, warmUpPeriodSec: 120` |
| 活动开始 | 将热点商品阈值上调 2-3 倍 | `count: 200 → 500` |
| 流量高峰 | 关闭非核心接口的限流，保障核心链路 | 删除或调高无关规则 |
| 活动结束 | 逐步恢复日常规则 | 分批回滚到基线 |

**步骤三：大促实时预案脚本**

```bash
#!/bin/bash
# sentinel-plan.sh — 大促 Sentinel 规则切换

ACTION=$1  # start / peak / end

case $ACTION in
  start)
    echo "=== 大促开始：提高限流阈值 ==="
    # 通过 Nacos API 更新规则
    curl -X POST "http://nacos:8848/nacos/v1/cs/configs" \
      -d "tenant=prod&dataId=order-service-flow-rules&group=SENTINEL_FLOW&content=$(cat rules/peak-flow.json)"
    ;;
  peak)
    echo "=== 峰值预案：关闭非核心接口保护 ==="
    curl -X POST "http://nacos:8848/nacos/v1/cs/configs" \
      -d "tenant=prod&dataId=order-service-flow-rules&group=SENTINEL_FLOW&content=$(cat rules/peak-flow.json)"
    ;;
  end)
    echo "=== 活动结束：恢复默认规则 ==="
    curl -X POST "http://nacos:8848/nacos/v1/cs/configs" \
      -d "tenant=prod&dataId=order-service-flow-rules&group=SENTINEL_FLOW&content=$(cat rules/default-flow.json)"
    ;;
  *)
    echo "Usage: $0 {start|peak|end}"
    ;;
esac
```

// 监控热点参数的统计总量，防止内存爆炸
@Scheduled(fixedDelay = 60000)
public void monitorParamCacheSize() {
    // 通过 JMX 或反射获取 ParamMapBucket 大小
    int cacheSize = ParamFlowRuleManager.getRuleMap().size();
    log.info("热点参数规则数: {}, 估计内存: {} KB",
             cacheSize, cacheSize * 2);  // 每项约 2KB
    if (cacheSize > 3000) {
        log.warn("热点参数缓存接近上限 4000，建议确认是否需要扩展");
    }
}

### 3.4 动态热点发现与自动例外项

```java
@Component
public class DynamicHotspotDetector {

    @Scheduled(fixedDelay = 60000)
    public void detectAndUpdateHotspots() {
        // 1. 从 Prometheus 查询 Top 10 热点参数
        List<HotspotParam> top10 = prometheusQuery(
            "topk(10, sum(rate(sentinel_param_flow_block_qps[5m])) by (param_value))");

        // 2. 获取当前已配置的例外项
        Set<String> existing = ParamFlowRuleManager.getRules().stream()
            .flatMap(r -> r.getParamFlowItemList().stream())
            .map(ParamFlowItem::getObject)
            .collect(Collectors.toSet());

        // 3. 检测新热点
        for (HotspotParam hp : top10) {
            if (!existing.contains(hp.getParamValue())) {
                log.warn("[HotspotDetector] 发现新热点: {} (拒绝 QPS={}), 建议添加例外项",
                    hp.getParamValue(), hp.getBlockQps());

                // 自动或半自动添加例外项
                addExceptionItem(hp.getParamValue(), 300); // 默认给 300 QPS 配额
            }
        }

        // 4. 检测已冷却的热点（过去 10 分钟内无流量）
        for (String existingParam : existing) {
            double recentQps = prometheusQuery(
                "rate(sentinel_param_flow_pass_qps{param_value=\"" + existingParam + "\"}[10m])");
            if (recentQps < 1.0) {
                log.info("[HotspotDetector] 热点冷却: {}, 可考虑移除例外项", existingParam);
            }
        }
    }

    private void addExceptionItem(String paramValue, int count) {
        // 更新 Nacos 中的规则，添加新的例外项
        ParamFlowRuleManager.getRules().stream()
            .filter(r -> r.getResource().equals("goodsDetail"))
            .findFirst()
            .ifPresent(rule -> {
                ParamFlowItem item = new ParamFlowItem()
                    .setObject(paramValue)
                    .setClassType(String.class.getName())
                    .setCount(count);
                rule.getParamFlowItemList().add(item);
                // 推送到 Nacos
                nacosService.pushParamFlowRules(ParamFlowRuleManager.getRules());
            });
    }
}
```

### 3.5 Token Server 高可用部署

```yaml
# Token Server StatefulSet — 多实例 + Raft 一致性
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: sentinel-token-server
spec:
  serviceName: sentinel-token-server
  replicas: 3
  selector:
    matchLabels:
      app: sentinel-token-server
  template:
    metadata:
      labels:
        app: sentinel-token-server
    spec:
      containers:
      - name: token-server
        image: token-server:1.8.6
        env:
        - name: SERVER_PORT
          value: "18730"
        - name: EMBEDDED_SERVER_MODE
          value: "raft"  # Raft 协议保证一致性
        ports:
        - containerPort: 18730
---
apiVersion: v1
kind: Service
metadata:
  name: sentinel-token-server
spec:
  clusterIP: None  # Headless Service，用于 StatefulSet
  selector:
    app: sentinel-token-server
  ports:
  - port: 18730
```

**踩坑记录**：

1. **集群热点参数的性能开销大于普通集群流控**：因为 Token Server 需要为每个参数值维护计数器，热点越多、内存和 CPU 开销越大。
2. **活动期间的规则变更需要灰度**：不要在活动开始瞬间全量切换规则——先在一个实例验证，再逐步扩大。
3. **热点参数的变化速度快**：大促中可能某个商品突然变成热点（如社交传播），但你的规则还没有为它配例外项。需要监控 + 快速响应。

## 4 项目总结

### 4.1 集群热点参数限流 vs 其他限流策略

| 对比维度 | 单机限流 | 集群全局限流 | 集群热点参数限流 |
|---------|---------|------------|---------------|
| 控制粒度 | 实例级别 | 全局 QPS | 全局 + 参数维度 |
| 部署复杂度 | 零 | Token Server | ParamFlowTokenServer |
| 内存开销 | 低（单实例统计） | 中（全局计数） | 高（每个参数值一个计数） |
| 适用场景 | 固定实例数 | 弹性伸缩 + 全局控制 | 大促热点商品保护 |
| 故障容错 | — | fallbackToLocalWhenFail | fallbackToLocalWhenFail |

### 4.2 大促 Sentinel 预案总结

| 预案项 | 日常 | 预热期 | 高峰期 | 恢复期 |
|-------|------|-------|-------|-------|
| 流控阈值 | 基线 | 上调 30% | 上调 100-200% | 逐步恢复 |
| 熔断窗口 | 10s | 10s | 5s（快速恢复） | 10s |
| 热点例外项 | Top10 | Top20 | Top50 | Top10 |
| 系统保护 CPU | 80% | 85% | 90% | 80% |

### 4.3 Token Server 高可用方案对比

| 方案 | 实现复杂度 | 故障恢复时间 | 数据一致性 | 推荐场景 |
|------|----------|------------|-----------|---------|
| 单节点 Token Server | 低 | 人工重启（分钟级） | N/A | 开发/测试 |
| Raft 集群（3 节点） | 高 | 自动选主（秒级） | 强一致 | 生产核心链路 |
| fallbackToLocalWhenFail | 低 | 自动降级（秒级） | 可能超额 | 非核心链路 |
| Redis 替代 Token Server | 中 | 秒级切换 | 最终一致 | 高基数参数 |

### 4.4 大促 Sentinel 规则变更倒计时

| 时间节点 | 操作 | 验证 |
|---------|------|------|
| T-7 天 | 输出大促压测报告，确定各接口拐点 | 压测报告通过 Review |
| T-2 小时 | 预热规则生效（Warm Up） | Dashboard 观察无异常 |
| T-0 分钟 | 活动规则生效（阈值上调） | 核心接口 QPS 和 RT 正常 |
| T+10 分钟 | 第一个流量高峰 | 监控热点参数变化，必要时调整例外项 |
| T+30 分钟 | 活动高峰 | 观察 Token Server 延迟和队列 |
| T+活动结束 | 逐步恢复基线规则 | 分 3 批回滚，每批间隔 5 分钟 |

### 4.2 思考题

1. 集群热点参数限流的 Token Server 维护 500 个参数值、每个每秒更新一次。估算 Token Server 的内存和 CPU 开销（每个参数值的计数器约 200 字节）。
2. 大促期间某 SKU 在 30 秒内从第 20 名热点飙升至第 1 名。你的 Sentinel 规则没来得及更新，会发生什么？如何设计自动化的热点检测与规则下发系统？
