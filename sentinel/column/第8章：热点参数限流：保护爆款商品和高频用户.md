# 第8章：热点参数限流：保护爆款商品和高频用户

## 1 项目背景

第 7 章的熔断降级让订单服务在"下游挂了"的时候有了自我保护能力。但在一次"华为新机首发"活动中，一个意想不到的问题出现了：

秒杀开始 5 秒后，监控显示订单服务整体 QPS 仅为 300，远未达到配置的流控阈值 1000。但大量用户反馈"页面卡死""下单报错"。排查发现，300 个 QPS 中有 280 个都在请求同一个 `skuId="HUAWEI-MATE70"` 的库存查询接口。库存服务的 Redis 单 Key 热点被打爆（单 Key QPS 超过 10 万），导致这个 Key 所在的 Redis 分片响应超时。但因为 Sentinel 配置的是"整个库存查询接口 QPS=1000"，整体流量确实没超，规则没有触发任何限流。

这就是**热点参数问题**：整体流量不高，但某个参数值（如爆款商品 ID、明星用户 ID、热门 IP）被集中访问，导致底层资源（缓存分片、数据库行锁）被打爆。普通的 QPS 流控对此无能为力——它按"整个资源"统计，看不到具体某个参数有多少流量。

此外，运营团队提出新需求：对于 VIP 用户和普通用户，同一个下单接口应该有不同的限流阈值——VIP 用户 QPS 可以到 50，普通用户限制在 10。这种"按用户维度差异化"也是普通流控做不到的。

Sentinel 的热点参数限流（ParamFlowRule）正是为这类场景设计的：它可以针对某个参数的具体值做精确限流，支持设置例外项（VIP 用户高阈值），让你从"保护整个接口"升级为"保护具体的热点数据"。

## 2 项目设计

**小胖**（抓耳挠腮）："大师，这不对啊！我们配了 QPS=1000 的流控，但实际才 300 QPS 就把 Redis 打爆了。Sentinel 是不是出 Bug 了？"

**大师**："不是 Bug。你想想，300 个 QPS 都落在同一个 Redis Key 上。普通流控看的是'整个接口'，但热点在'某个 skuId'。就像一个食堂有 10 个打饭窗口，1000 个人进来没问题，但如果 800 个人全挤在红烧肉窗口，那个窗口的师傅就崩溃了。"

**小白**："这就是热点参数限流要解决的问题？针对某个参数值的流量做控制？"

**大师**："没错。热点参数限流本质上是在 FlowSlot 之后又加了一层更细粒度的控制。它不仅看'资源总流量'，还看'参数 X 的流量'。比如你可以配置：`资源=queryStock`，`参数索引=0`（第一个参数 skuId），`阈值=50`。这样每个 skuId 最多 50 QPS。"

**小胖**："那如果有个 VIP 商品确实需要更高 QPS 呢？总不能一刀切吧。"

**大师**："可以用例外项（Exception Item）。比如华为新机 skuId='HW-MATE70'，单独给它配 200 QPS。例外项优先级高于默认阈值。"

**小白**："参数类型呢？除了 String，能限流 Long、int 吗？还有 IP 地址怎么办？"

**大师**："支持七种参数类型：int、long、double、float、String、byte、char。IP 的话，你可以先用 `RequestOriginParser` 把 IP 提取出来，然后用 `SphU.entry()` 时通过 `EntryType` 的 `args` 参数传入。"

**小胖**："那参数是对象怎么办？比如 `createOrder(OrderRequest req)`，我想按 `req.getUserId()` 限流？"

**小胖**："我也想到了这个问题！@SentinelResource 是不是支持指定参数索引？"

**大师**："@SentinelResource 不直接支持对象内部字段。但你可以两种办法解决：一是把 userId 单独作为方法参数传进来；二是在调用 `SphU.entry()` 时手动构建参数数组。第二种更灵活但代码侵入性强。"

**小白**："热点参数限流的内存开销呢？如果参数值很多（比如百万级 userId），Sentinel 会爆内存吗？"

**大师**："这是个非常好的问题。热点参数限流内部使用 LRU 缓存来统计每个参数值的流量。缓存大小默认是 4000（可通过 `ParamFlowRuleManager.setMaxCacheSize()` 调整）。超出上限的参数值会走默认阈值限制。所以热点参数限流不适用于'对所有用户做精确限流'的场景——它是为'Top 热点'设计的。"

**小胖**："大师，那例外项能动态增删吗？比如大促期间 Top 20 的热点商品随时在变，我总不能每次变了就重启服务吧？"

**大师**："例外项可以动态更新。如果你用了 Nacos 数据源（第 17 章），修改 Nacos 中的 ParamFlowRule JSON，客户端会自动热加载。而且你可以配合一个定时任务，每分钟从监控中拉取 Top 10 被限流的参数值，自动添加到例外项中——这就是动态热点发现。"

**小白**："我注意到热点参数限流只能按 QPS 限，不能按线程数或者预热模式。为什么有这个限制？"

**大师**："因为热点参数限流的核心数据结构是 ParamMapBucket——一个参数值对应一个滑动窗口计数器。如果你每个参数值还要维护排队队列（匀速模式）或预热曲线，内存和 CPU 开销会指数级增长。Sentinel 在这里做了一个权衡：热点参数限流追求的是'简单高效'，而不是'功能完备'。"

**小胖**（追问）："那如果我想实现'每个用户 QPS 不超过 5，且超过后自动冷却 10 秒'——这个用 ParamFlowRule 能做吗？"

**大师**："纯粹的 ParamFlowRule 做不到——它没有'冷却时间'概念。但你可以组合使用：热点参数限流 + 熔断规则。热点限控制了每个用户的 QPS，熔断规则在某个用户频繁达到限流后自动熔断该用户 10 秒。不过这种组合需要自定义规则，不在 Sentinel 的内置能力中。"

**技术映射**：
- `ParamFlowRule` 核心字段：`resource`（资源名）、`paramIdx`（参数索引，从 0 开始）、`grade`（限流模式，只支持 QPS）、`count`（阈值）、`paramFlowItemList`（例外项列表）。
- 例外项 `ParamFlowItem`：`object`（参数值）、`classType`（参数类型，如 `java.lang.String`）、`count`（针对该值的阈值）。
- 统计方式：热点参数限流内部使用 `ParamMapBucket` 做滑动窗口统计，每个参数值有独立的计数器。
- 与普通流控的地关系：如果同时配置了普通流控和热点参数限流，两者都会判断。普通流控管"总量"，热点参数管"单个值"。

## 3 项目实战

### 3.1 环境准备

使用 `@SentinelResource` 注解的热点参数限流支持，确保依赖中有 `sentinel-parameter-flow-control`：

```xml
<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-parameter-flow-control</artifactId>
    <version>1.8.6</version>
</dependency>
```

### 3.2 分步实现

**步骤一：商品详情接口——按 skuId 热点限流**

```java
@RestController
public class HotParamController {

    /**
     * 商品详情接口
     * @SentinelResource 中的 value 是资源名，
     * blockHandler 处理限流后的兜底
     * 热点参数限流要求在 @SentinelResource 标注的方法中才能生效
     */
    @GetMapping("/goods/detail")
    @SentinelResource(value = "goodsDetail", blockHandler = "goodsDetailBlock")
    public String goodsDetail(@RequestParam String skuId) {
        // 模拟查询商品详情，耗时 20ms
        return "商品详情: skuId=" + skuId + ", 名称=华为Mate70, 库存=50件";
    }

    public String goodsDetailBlock(String skuId, BlockException e) {
        return "该商品访问过于火爆，请稍后再试! (skuId=" + skuId + ")";
    }
}
```

配置热点参数规则：

```java
@PostConstruct
public void initHotParamRules() {
    List<ParamFlowRule> rules = new ArrayList<>();

    ParamFlowRule rule = new ParamFlowRule("goodsDetail")
            .setParamIdx(0)                                 // 参数索引 0 = skuId
            .setGrade(RuleConstant.FLOW_GRADE_QPS)
            .setCount(10);                                  // 每个 skuId 默认最多 10 QPS

    // 例外项：华为新机 skuId 可以到 50 QPS
    ParamFlowItem vipItem = new ParamFlowItem()
            .setObject("HW-MATE70")
            .setClassType(String.class.getName())
            .setCount(50);

    // 例外项：iPhone 也可以到 30 QPS
    ParamFlowItem iphoneItem = new ParamFlowItem()
            .setObject("IPHONE-16")
            .setClassType(String.class.getName())
            .setCount(30);

    rule.setParamFlowItemList(Arrays.asList(vipItem, iphoneItem));
    rules.add(rule);

    ParamFlowRuleManager.loadRules(rules);
}
```

**验证**：

```bash

# IPHONE-16, 500 行
# XIAOMI-15, 200 行

# 期望结果：
# HW-MATE70: QPS 限制约 50（例外项）
# IPHONE-16: QPS 限制约 30（例外项）

**步骤二：下单接口——按 userId 热点限流**

```java
@PostMapping("/order/create")
@SentinelResource(value = "hotCreateOrder", blockHandler = "createOrderBlock")
public String createOrder(@RequestParam String skuId,
                           @RequestParam String userId) {
    // userId 在参数索引 1
    return "下单成功, orderId=ORD_" + UUID.randomUUID();
}

public String createOrderBlock(String skuId, String userId, BlockException e) {
    if (e instanceof ParamFlowException) {
        return "用户 " + userId + " 操作过于频繁，请稍后再试";
    }
    return "系统繁忙";
}
```

```java
// 按 userId（索引 1）做热点限流
ParamFlowRule userRule = new ParamFlowRule("hotCreateOrder")
        .setParamIdx(1)           // 参数索引 1 = userId
        .setCount(5);             // 每个用户默认 5 QPS

// VIP 用户例外项
ParamFlowItem vipUser = new ParamFlowItem()
        .setObject("VIP-888")
        .setClassType(String.class.getName())
        .setCount(20);
userRule.setParamFlowItemList(Collections.singletonList(vipUser));
```

**步骤三：自定义参数提取**

如果参数是复杂对象，需要手动提取：

```java
// 复杂对象的参数提取
@PostMapping("/order/create-v2")
public String createOrderV2(@RequestBody CreateOrderRequest request) {
    // 手动调用 SphU.entry，同时传入参数数组
    try (Entry entry = SphU.entry("hotCreateOrder",
             EntryType.OUT, 1, request.getUserId())) {
        return "下单成功";
    } catch (BlockException e) {
        return "系统繁忙";
    }
}
```

注意：`SphU.entry("resourceName", EntryType.OUT, 1, args...)` 中的可变参数 args 就是传给 ParamFlowSlot 的参数数组。

**步骤四：观察热点参数统计**

在 `~/logs/csp/sentinel-record.log` 中可看到热点参数日志：

```log
2024-06-01 14:30:15 | ParamFlowChecker: resource=goodsDetail, 
    paramIdx=0, paramValue=HW-MATE70, 
    currentQps=55, threshold=50 -> blocked
```

**踩坑记录**：

1. **@SentinelResource 不支持复杂对象参数提取**：如果参数是 `@RequestBody OrderRequest req`，Sentinel 不会自动解析 req 内部的字段。需要编程式 API 手动传入参数数组。
2. **热点参数缓存上限**：默认 4000 个参数值统计。超出的参数值统一按默认阈值处理（如果不配默认阈值则不限流）。对于百万级用户场景，需要自行扩展或改用网关层 IP 限流。
3. **参数索引从 0 开始**：很多人写 `paramIdx=1` 以为是"第一个参数"，实际是第二个参数。务必确认方法签名中的参数顺序。
4. **例外项的 classType 必须写全限定名**：`String.class.getName()` 返回 `java.lang.String`，而不是 `String`。

**步骤四：通过 Dashboard/Nacos 动态更新热点参数例外项**

如果规则已持久化到 Nacos，可以在 Nacos 控制台直接修改例外项而不重启服务：

```json
// Nacos 中的 order-service-param-flow-rules 配置
[
  {
    "resource": "goodsDetail",
    "grade": 1,
    "paramIdx": 0,
    "count": 10.0,
    "paramFlowItemList": [
      {"object": "HW-MATE70", "count": 50, "classType": "java.lang.String"},
      {"object": "IPHONE-16", "count": 30, "classType": "java.lang.String"},
      {"object": "XIAOMI-15-ULTRA", "count": 25, "classType": "java.lang.String"}
    ],
    "clusterMode": false
  }
]
```

修改后 Nacos 会自动推送变更，客户端日志输出：
```log
[Sentinel Nacos DataSource] ParamFlow rules updated: 1 rules reloaded
```

**步骤五：热点参数限流的组合策略**

实际生产中，热点参数限流通常不是独立使用的。以一个电商场景为例：

| 策略层 | 规则 | 目的 |
|-------|------|------|
| 网关层 | IP 限流 QPS=20 | 防单一 IP 高频刷接口 |
| 网关层 | 用户 ID 热点 QPS=10 | 防脚本用户批量操作 |
| 服务层 | 接口全局限流 QPS=500 | 保护服务整体容量 |
| 服务层 | SKU 热点限流 QPS=50 | 保护爆款商品背后的 Redis/DB |
| 服务层 | VIP 例外项 SKU QPS=200 | 为重点商品保留高配额 |

```java
// 组合策略的规则加载顺序建议
@PostConstruct
public void initAllParamRules() {
    // 1. 先配全局流控（兜底）
    FlowRuleManager.loadRules(flowRules);
    // 2. 再配热点参数（精细化）
    ParamFlowRuleManager.loadRules(paramFlowRules);
    // 这样全局流控保证不会被打爆，热点参数限流在全局阈值内做分配
}

## 4 项目总结

### 4.1 优点与缺点

| 维度 | 热点参数限流 | 普通 QPS 限流 | Redis 计数限流 |
|------|------------|-------------|-------------|
| 精确度 | 按参数值统计 | 按资源总量 | 可精确到参数值 |
| 性能开销 | 极低（本地 LRU） | 极低 | 高（网络 IO） |
| 扩展性 | 参数值上限 ~4000 | 无限 | 无限 |
| 接入复杂度 | 低（注解+规则） | 低 | 中 |
| 精确到对象字段 | 需手动提取 | 不支持 | 可支持 |
| 例外项灵活性 | 高 | 无 | 需额外逻辑 |

### 4.2 适用场景

- **热点商品**：秒杀 skuId、爆款商品详情、限时优惠活动
- **高频用户**：黄牛/脚本用户识别和限流、VIP 用户高阈值
- **热门 IP**：防刷、DDoS 初步防御（配合网关层使用）
- **热点直播/话题**：直播间 ID、短视频 ID、热搜话题 ID
- **不适用**：全量用户精确限流（超出缓存上限）；参数值变化极快（每秒新增上万个新参数值）

### 4.3 注意事项

1. 热点参数限流只支持 QPS 模式，不支持线程数、预热、排队模式。
2. 参数缓存上限默认 4000，可通过 `-Dcsp.sentinel.param.max.cache.size=10000` 调整，但会增加内存开销。
3. 热点参数规则目前不支持 Dashboard 直接配置（1.8.6 版本），需要用代码或 Nacos 持久化方式配置。
4. 不需要为每个接口都配热点参数限流——只对已识别出的"热点维度"配置。

### 4.4 热点参数限流 vs 其他精细化限流方案

| 方案 | 实现方式 | 性能 | 灵活性 | 推荐场景 |
|------|---------|------|-------|---------|
| ParamFlowRule | Sentinel 内置 | 极高 | 中（例外项可配） | TopN 热点商品/用户 |
| Redis 滑动窗口计数 | 自行实现 + Redis | 中 | 高（可精确到任意维度） | 全量用户限流 |
| 网关层 IP 限流 | Sentinel Gateway Adapter | 高 | 低（仅 IP 维度） | 防爬虫/防刷 |
| 自定义 Slot | SPI 扩展 | 高 | 极高 | 复杂业务逻辑限流 |

### 4.5 热点参数限流内存估算公式

- 每个参数值计数器约占用 **150-200 字节**（LRU Entry + 滑动窗口 bucket）
- 默认缓存 4000 个参数值 → 约 0.8 MB
- 如果调整到 10000 → 约 2 MB
- 每个资源独立缓存，如果一个服务有 5 个接口配了热点参数限流 → 10 MB
- 生产环境预留 **50 MB** 给热点参数统计足够

### 4.4 常见踩坑经验

| 故障现象 | 根因 | 解决方案 |
|---------|------|---------|
| 热点参数限流不生效 | 参数索引写错；或 ParamFlowRuleManager 未初始化 | 检查方法签名参数顺序；确保规则在首次请求前加载 |
| 热点限流粒度不够 | 对象参数内部字段无法自动提取 | 用编程式 API 手动传参数数组 |
| 某些参数值始终不限流 | 超出了 LRU 缓存上限，该参数值未被统计 | 调整 maxCacheSize 或确认该参数是否真的是热点 |
| 例外项被忽略 | classType 不匹配 | 确保例外项的 object 类型与运行时参数类型一致 |

### 4.5 思考题

1. Sentinel 的热点参数限流内部使用什么数据结构维护参数值计数器？当参数值超出 LRU 缓存上限时，漏掉的参数值会怎样？
2. 如果你需要对 100 万个用户各设置不同的 QPS 限制（每个用户不同阈值），Sentinel 的热点参数限流能做吗？如果不行，你有什么替代方案？

### 4.6 推广计划

- **开发团队**：梳理所有"按参数维度访问"的接口（商品详情、用户主页、直播间），评估是否需要热点参数限流。
- **测试团队**：设计多参数值压测方案（JMeter CSV 参数化），验证不同参数值的限流行为。
- **运维团队**：关注热点参数缓存大小（`maxCacheSize`）和内存开销，在监控面板中加入热点参数相关指标。
