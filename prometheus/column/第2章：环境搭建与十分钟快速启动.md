# 第2章：环境搭建与十分钟快速启动

## 一、项目背景

新入职的小李接到了一个紧急任务：今晚新服务上线，必须在两小时内完成监控接入。组里的老员工丢下一句"用Prometheus就行"便匆匆赶去开会，留下小李面对搜索引擎上眼花缭乱的教程发呆。官网文档是全英文的，博客文章有的讲v1.x版本、有的讲v2.x版本，配置项的写法还互不兼容。小李越看越焦虑——明明只是一个监控工具，为什么入门门槛这么高？

小李的困惑恐怕是每一位Prometheus新手的共同记忆。打开官方下载页，光二进制包就分linux-amd64、linux-arm64、darwin-amd64、darwin-arm64等十几个变体，稍有不慎下错版本，启动就报"cannot execute binary file"。好不容易装对了，写好配置文件一启动，又因为YAML缩进混用了Tab和空格，迎来一长串"couldn't load configuration"的红字。更有甚者，端口9090被本地其他进程占用，Prometheus无声无息地退出了，查日志查到头秃。如果你用的是Docker，还可能在权限问题上绊一跤——容器里的Prometheus以nobody用户运行，宿主机上的配置文件死活读不到。还有人跟着教程一路走到Grafana可视化那一步，才发现Node Exporter压根没装，数据源空空如也。

这些坑，本章带你全部跳过。我们不追求一次讲透所有配置参数，而是以"先跑起来再说"为原则，用Docker Compose三分钟编排好Prometheus + Node Exporter + Grafana全家桶，十分钟内看到第一张监控图表。跑通之后，再回头理解每项配置的含义，你会发现这些东西远没有想象中那么可怕。

## 二、剧本式交锋对话

**场景**：运维组的茶水间，三台显示器并排亮着。小胖正往docker-compose.yml里疯狂敲字，小白端着咖啡凑过来。

---

**小胖**（头也不抬）："搞定了！一个docker-compose up -d，Prometheus直接跑起来了。我之前折腾二进制安装搞了半小时，又是wget又是tar又是写systemd service，早知道用Docker这么省事。"

**小白**："等等，你刚才说的二进制部署很麻烦吗？我听说生产环境都是直接用二进制跑的，Docker会不会有性能损耗？"

**大师**（从工位转过来，手里拿着保温杯）："这个问题好。二进制部署的优势在于零虚拟化开销、直接访问宿主机文件系统和网络栈，在大规模采集场景下确实更优。但Docker的优势也很明显：一键启停、环境隔离、版本切换方便。你想想，如果Prometheus要升级，二进制部署你得停服务、下载新包、替换二进制、重启，每一步都可能出错；Docker呢？改个镜像tag重新up就行了。所以我的建议是——**本地开发和学习用Docker Compose，生产环境按团队习惯选二进制或K8s**。"

**小白**（指着小胖的配置文件）："prometheus.yml里这个global是什么意思？还有scrape_interval，为什么是15秒？"

**大师**："global块是全局默认配置，它下面定义的参数会自动应用到所有采集任务上。比如scrape_interval: 15s表示每15秒去目标拉一次数据，evaluation_interval: 15s表示每15秒计算一次告警规则。15秒是一个经验值，既能保证数据实时性，又不至于把目标服务压垮。注意这两个值可以在单个scrape_config里覆盖，灵活性很高。"

**小胖**（滚动着鼠标轮）："那你再看看这个scrape_configs，里面有个job_name: 'prometheus'，这个job_name是随便起的吗？"

**大师**："绝对不是随便起的。job_name是这个采集任务的身份标识，它会被附加到每一条指标数据上，变成标签`{job="prometheus"}`。你在PromQL里查数据、在Grafana里分组展示、在告警规则里精确定位，全靠这个job标签。如果你把两个完全不同的服务配成同一个job_name，Prometheus会把它们的指标混在一起，统计出来的数据毫无意义。所以job_name要起得有意义，比如'node_exporter'、'nginx'、'redis'这种一眼知道是谁的名字。"

**小白**："我还想问一个细节——Prometheus默认启动就监听9090端口，这个端口能改吗？万一我的机器上9090被其他服务占了呢？"

**大师**："当然能改。启动时加`--web.listen-address=:9091`就行，但更优雅的做法是在docker-compose.yml里做端口映射，把宿主机的9091映射到容器的9090，这样容器内部不用改任何配置。不过话说回来，9090这个端口号在监控圈几乎成了Prometheus的'专属门牌号'，就像3306之于MySQL、6379之于Redis，建议不冲突就别改，团队沟通成本会低很多。"

**小胖**："那prometheus.yml里的alerting和rule_files这两个配置块怎么都是空的？它们不写不行吗？"

**大师**："问得好。这两个块是为未来的告警能力预留的接口。alerting指向Alertmanager的地址，rule_files引用你的告警规则文件。现在先空着完全没问题，Prometheus的核心能力——指标采集和查询——不依赖它们。等第5章我们讲告警的时候，这两个空槽就会填上实实在在的内容。这其实体现了Prometheus设计上的一个理念：**按需组装，不强制绑定**。你可以只用它的采集和查询能力，告警用别的系统来做，互不干扰。"

**小胖**："行了行了，Web UI打开了，这个Graph和Table有什么区别？"

**大师**（指着屏幕）："Graph是折线图视图，适合看指标随时间变化的趋势，比如CPU使用率是不是在爬坡、内存有没有缓慢泄漏。Table是表格视图，展示当前时刻的精确值，适合排查'此时此刻哪个节点异常'。你用的时候记住一句话：**趋势看Graph，快照看Table**。来，在Expression Browser里输入`up`试一下，点Execute。"

**小白**："出来了！一条值为1的线……这是什么意思？"

**大师**："`up`是Prometheus里最重要的指标之一。它只有两个值：1表示目标能正常采集到数据，0表示目标挂了或者网络不通。你现在看到的1，说明Prometheus自己采集自己成功了。如果哪天这条线从1变成0，你就知道——Prometheus它自己先崩了。顺便输入`prometheus_http_requests_total`看看，这是Prometheus暴露的自身HTTP请求计数器，也是我们下一章要深入讲的指标类型。"

## 三、项目实战

### 环境准备

- 操作系统：Linux (CentOS 7+ / Ubuntu 20.04+) 或 macOS 或 Windows WSL2
- Docker 20.10+ 及 Docker Compose v2+
- 或 Prometheus 二进制包 2.55+（本章以Docker Compose为主进行演示）

执行以下命令确认环境就绪：

```bash
docker --version
# Docker version 24.0.7, build afdd53b

docker compose version
# Docker Compose version v2.21.0
```

### 步骤1：一键启动Prometheus

创建项目目录并编写配置文件：

```bash
mkdir -p ~/prometheus-quickstart && cd ~/prometheus-quickstart
```

编写 `docker-compose.yml`：

```yaml
version: '3.8'

services:
  prometheus:
    image: prom/prometheus:v2.55.0
    container_name: prometheus
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - prometheus_data:/prometheus
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--web.enable-lifecycle'
    restart: unless-stopped

volumes:
  prometheus_data:
```

编写 `prometheus.yml`：

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

alerting:
  alertmanagers:
    - static_configs:
        - targets: []

rule_files: []

scrape_configs:
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]
```

启动：

```bash
docker compose up -d
```

输出：

```
[+] Running 1/1
 ✔ Container prometheus  Started
```

### 步骤2：验证Prometheus启动成功

健康检查端点：


输出：


### 步骤3：Web UI初体验

打开浏览器访问 `http://localhost:9090`，进入Prometheus Web UI。这个界面默认呈现深色主题，顶部是Prometheus的经典橙色导航栏，中央最醒目的位置就是Expression Browser（表达式浏览器）——我们查询一切指标数据的入口。

在Expression Browser输入框中输入 `up`，点击Execute，切换到Graph视图。你将看到一条值为1的绿色水平直线——这就是Prometheus自己监控自己的结果。之所以是一条直线，是因为`up`的值在我们启动Prometheus后就一直是1（健康状态），没有发生过变化。

再输入 `prometheus_http_requests_total`，点击Execute。这次你会看到多条不同颜色的折线，每条折线代表一种HTTP handler的请求累计量（如`/api/v1/query`、`/metrics`）。这个指标是典型的Counter类型——只增不减，所以折线整体呈上升趋势。切换到Table视图，可以看到每个handler当前的精确请求次数，适合做数值对比而非趋势分析。

观察要点：
- Graph视图的折线是单调递增的（counter类型指标的典型特征）
- 右上角的时间范围选择器可以调整查询区间，默认展示最近1小时
- 点击每个指标名旁的"stacked"图标可切换堆叠模式
- 将鼠标悬停在折线上可查看具体时间点的数值

### 步骤4：添加Node Exporter采集任务

修改 `prometheus.yml`，在 `scrape_configs` 下新增一个job：

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

alerting:
  alertmanagers:
    - static_configs:
        - targets: []

rule_files: []

scrape_configs:
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]

  - job_name: "node_exporter"
    static_configs:
      - targets: ["localhost:9100"]
```

无需重启Prometheus，热加载配置即可生效：

```bash
curl -X POST http://localhost:9090/-/reload
```

现在打开Web UI，进入 Status → Targets 页面。你会看到两个采集目标：

| 状态 | job_name        | instance          |
|------|-----------------|-------------------|
| UP   | prometheus      | localhost:9090    |
| DOWN | node_exporter   | localhost:9100    |

Node Exporter显示为DOWN是因为我们还没有在9100端口启动它。注意Last Error栏会显示类似"connection refused"的信息——这恰恰说明Prometheus的采集逻辑是正常的，只是目标不可达。

### 步骤5：三件套整合部署

现在把Prometheus、Node Exporter、Grafana整合到同一个编排文件。

更新 `docker-compose.yml`：

```yaml
version: '3.8'

services:
  prometheus:
    image: prom/prometheus:v2.55.0
    container_name: prometheus
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - prometheus_data:/prometheus
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--web.enable-lifecycle'
    restart: unless-stopped

  node_exporter:
    image: prom/node-exporter:v1.8.2
    container_name: node_exporter
    ports:
      - "9100:9100"
    volumes:
      - /proc:/host/proc:ro
      - /sys:/host/sys:ro
      - /:/host/rootfs:ro
    command:
      - '--path.procfs=/host/proc'
      - '--path.sysfs=/host/sys'
      - '--path.rootfs=/host/rootfs'
    restart: unless-stopped

  grafana:
    image: grafana/grafana:11.2.0
    container_name: grafana
    ports:
      - "3000:3000"
    volumes:
      - grafana_data:/var/lib/grafana
    environment:
      - GF_SECURITY_ADMIN_PASSWORD=admin
    restart: unless-stopped

volumes:
  prometheus_data:
  grafana_data:
```

同时确保 `prometheus.yml` 已包含node_exporter的采集配置（如步骤4所示）。

重新启动全家桶：

```bash
docker compose down && docker compose up -d
```

验证三个服务全部正常：

```bash
# Prometheus
curl -s http://localhost:9090/-/healthy

# Node Exporter（前5行）
curl -s http://localhost:9100/metrics | head -5

# Grafana
curl -s http://localhost:3000/api/health
```

三次验证的输出分别为：

```
Prometheus Server is Healthy.
```



三个服务全部正常响应，全家桶部署完成。

### 可能遇到的坑及解决方法

**1. YAML缩进用空格不能用Tab**

YAML规范要求缩进必须使用空格而非Tab字符。如果混用了Tab，Prometheus启动时会报错：

```
yaml: line X: found character that cannot start any token
```

解决办法：在VS Code中点击右下角"Tab Size"切换到"Indent Using Spaces"，或使用在线YAML校验工具。建议在编辑器中设置`"editor.renderWhitespace": "all"`来可视化空格和Tab的区别。

**2. 端口被占用**

```bash
# 查占用9090的进程
sudo lsof -i:9090

# macOS也可用
lsof -nP -iTCP:9090 | grep LISTEN

# 杀掉占用进程
kill -9 <PID>
```

**3. Docker Volume挂载路径错误**

容器内路径必须是绝对路径。常见错误：在`docker-compose.yml`中写了相对路径，或宿主机上的配置路径与volume映射不匹配。始终使用`$(pwd)`或绝对路径指定配置文件位置，并确认文件的读写权限。

## 四、项目总结

### Docker部署 vs 二进制部署对比

| 维度       | Docker Compose部署                          | 二进制部署                                |
|------------|---------------------------------------------|-------------------------------------------|
| 安装速度   | ⭐⭐⭐ 一条命令启动全家桶                    | ⭐ 下载、解压、写systemd、设置开机自启     |
| 版本管理   | ⭐⭐⭐ 改镜像tag重新up                      | ⭐⭐ 手动下载替换二进制，版本回滚麻烦       |
| 性能开销   | ⭐⭐ 轻微（AUFS/OverlayFS + 网络NAT）       | ⭐⭐⭐ 零虚拟化，直接使用宿主机资源         |
| 配置管理   | ⭐⭐ 通过volume挂载，需要理解容器文件系统   | ⭐⭐⭐ 直接编辑宿主机文件，所见即所得       |
| 迁移性     | ⭐⭐⭐ 一个compose文件到处跑                | ⭐ 依赖操作系统、库版本，迁移需重新适配    |

### 适用场景

- **本地开发测试**：Docker Compose是首选。五分钟拉起环境，用完即毁，零残留。特别适合微服务开发阶段需要快速验证指标暴露是否正常的场景，一条`docker compose down`就能彻底清理，不用操心systemd残留进程和/var/lib下的遗留文件。
- **生产环境（传统服务器）**：建议二进制部署，配合systemd管理进程，配合Ansible/SaltStack统一配置下发。原因有三：不需要额外的Docker守护进程减少了系统复杂度、避免容器网络NAT在高并发采集场景下成为瓶颈、查看Prometheus自身日志和资源消耗的路径更短——直接`journalctl -u prometheus`比`docker logs`快得多。此外二进制部署可以让Prometheus直接绑定到宿主机的网络接口上，采集局域网内其他机器时不会遇到Docker网络隔离的麻烦。
- **K8s环境**：使用Prometheus Operator或kube-prometheus-stack Helm Chart进行云原生部署，享受自动服务发现（ServiceMonitor/PodMonitor）、自动伸缩（Thanos/Cortex远程存储）等K8s生态能力。此时不建议在K8s中用Docker Compose那一套，因为K8s本身就是容器编排平台，重复套一层Compose毫无意义。

### 注意事项

- **数据持久化**：本次演示中我们为Prometheus和Grafana配置了命名卷（`prometheus_data`、`grafana_data`），确保容器重启后历史数据不丢失。生产环境中建议挂载到宿主机的SSD目录以获得更好的IO性能。
- **版本选择**：演示中使用了固定版本号（如`v2.55.0`），而非`latest`标签。这是生产环境的基本素养——latest在不同时间拉取到的镜像内容不同，可能导致"昨天好好的今天怎么崩了"的灵异现象。
- **安全配置**：`--web.enable-lifecycle`允许通过HTTP接口热加载配置和优雅退出，这在开发阶段很方便，但生产环境中如果Prometheus的9090端口暴露在公网，就等于把开关交给了攻击者。同样的，`--web.enable-admin-api`用于管理TSDB（如数据清理），如非必要切勿开启。

### 常见踩坑经验

**踩坑1：Windows换行符导致配置文件报错**

在Windows上编辑的YAML文件，换行符默认为CRLF（`\r\n`），而Prometheus容器运行在Linux上，只认LF（`\n`）。表现在日志中就是各种奇怪的解析错误，而肉眼检查完全看不出问题。解决办法：在VS Code右下角切换到LF换行符，或使用`dos2unix`工具转换。

**踩坑2：Docker容器时区导致图表时间偏移**

默认Docker容器使用UTC时区，而你的浏览器使用东八区时间。这会导致Grafana面板中的峰值出现在"错误"的时间点上，排查问题时容易误判。解决办法：在docker-compose.yml中为各服务添加环境变量`TZ=Asia/Shanghai`即可。

**踩坑3：Node Exporter版本与Prometheus版本的兼容性**

Node Exporter 1.8.0+重命名了部分CPU和内存相关指标（例如`node_cpu`从`cpu`模式改为`seconds`模式），如果你参考一份老教程的Grafana Dashboard JSON导入，会发现部分面板显示"No data"。遇到这种情况请核对Node Exporter的CHANGELOG，或直接使用对应版本的Dashboard模板。

### 思考题

1. **Prometheus自身也暴露了/metrics端点，那么由谁来监控Prometheus本身呢？** 提示：一种方案是搭建第二个Prometheus实例专门监控主Prometheus（即"监控的监控"，meta-monitoring）。另一种方案是借助Grafana的Alerting功能对Prometheus自身的指标（如`prometheus_tsdb_head_series`、`up`）设置告警。还有一种思路是利用云厂商的拨测服务或外部黑盒监控系统（如Blackbox Exporter）来探测Prometheus的HTTP端点可达性。你还能想到其他办法吗？

2. **如果把global.scrape_interval从默认的15秒改成1秒，会有什么影响？** 提示：数据精度确实提高了，但代价是什么？从三个维度思考——采集目标的CPU负载（每次HTTP GET /metrics都会消耗目标进程的计算资源）、Prometheus自身的内存消耗（TSDB中每个时间序列会记录更多采样点，默认2小时的内存窗口会容纳7200个采样点而非480个）、以及存储空间占用量（磁盘消耗同样暴涨15倍）。试试在你的环境中实际操作一把，用`docker stats`观察资源变化，并尝试回答：在你的业务场景下，1秒和15秒的精度差异真的有意义吗？
