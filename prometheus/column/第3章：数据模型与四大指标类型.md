# 第3章：数据模型与四大指标类型

## 一、项目背景

某电商公司订单服务团队最近接入了Prometheus。开发小王接到任务：给订单服务加上监控指标。他三下五除二写了几个指标上报，信心满满地提交了代码。

几天后，运维同事盯着Grafana大盘发懵——"订单数"曲线上周还有数据，今天怎么变成了0？数据库里明明每天都有新订单。排查之后，原因令人哭笑不得：

**第一刀**：小王用Gauge（可增可减的指标）来记录订单总数。程序重启时Gauge归零，累积的订单计数一夜回到解放前。订单数是一个只会增长的量，从来不会减少，应该用Counter（只增不减）。更坑的是，小王还把每个用户的订单数用`user_id`作为label值——`orders_total{user_id="10001"}`、`orders_total{user_id="10002"}`……每个用户一条时间序列，几百万用户就是几百万条线，Prometheus直接OOM（Out of Memory）。

**第二刀**：小王想监控API延迟的P99，拍脑袋选了个Summary。等运维想看"整个集群所有实例的P99延迟"时才发现——Summary的分位数是每个实例在客户端独立计算的，无法跨实例聚合。三个实例的P99分别是100ms、200ms、150ms，集群级别的P99到底是多少？算不出来。

**第三刀**：Histogram的bucket设置成了默认值。业务高峰期大部分请求耗时在200ms左右，但默认bucket的第一个值是0.005秒，最后一个是10秒。结果就是：0.005到0.01之间的bucket永远是空的（浪费内存），超过10秒的请求全被丢进`+Inf`桶里（丢失了长尾分布细节）。

这三个问题指向同一个根因：**不理解Prometheus四种指标类型的语义差异，不了解label设计原则，不明Histogram和Summary的核心区别。**

本章从数据模型出发，把四种指标类型的"脾气"彻底讲透。

## 二、剧本式交锋对话

> 场景：午休时间，办公室茶水间。小胖端着一杯拿铁走过来，满脸写着"我被Prometheus坑了"。

**小胖**："兄弟们，我心态崩了。我写的订单指标，重启一下服务就清零，运维说Prometheus都没报错，但数据就是不对。这玩意儿也太玄学了吧？"

**小白**："你这么一说，我也有个问题。昨天我想看API的P99延迟，在Grafana上写了个PromQL查Histogram，算出来的值是300ms，但用Summary直接查quantile标签，显示的是280ms。同一个接口，两个P99差了20ms，到底哪个准？"

**大师**（放下手中的《SRE：Google运维解密》）："你俩的问题正好是一对儿。小胖的问题出在**选错了指标类型**——该用Counter的你用了Gauge。小白的问题出在**没搞懂Histogram和Summary的本质区别**。来，咱们一个一个说。"

---

**小胖**："Counter就Counter呗，不就是个计数器吗？为啥不用Gauge？都是数字，我重启的时候把Gauge读数据库重新设置回去不就行了？"

**大师**："你开车吗？"

**小胖**："开啊。"

**大师**："你仪表盘上的里程表，是每次熄火归零，还是从出厂开始一直往上走？"

**小胖**："那肯定是一直往上走啊，归零那不成了调表车了。"

**大师**："对。Counter就是里程表——只增不减。你今天开了100公里，明天又开了50公里，总里程就是150，它只会越来越大。如果你用Gauge记录总里程——不好意思，熄火重开它就给你算0。Prometheus的设计哲学是：**"偷懒"把复杂性交给服务端**。客户端只管上报当前值，不要做任何复杂计算。Counter只增不减，客户端连"减"这个操作都不暴露给你——`inc()`和`add()`，没有`dec()`。"

**小胖**："那Gauge又是什么意思？是不是Counter反过来，只减不增？"

**大师**："错了。Gauge是'可增可减'——像温度计。内存使用率、CPU温度、队列长度，这些值是上上下下的，没有固定方向。你把内存从4GB升到8GB，值就变大；垃圾回收了，值就变小。Gauge对应的就是这类'瞬时快照'。"

**小白**："我懂了——Counter记总数，Gauge记状态。那Histogram和Summary又有什么区别？我在/metrics里看它们生成的指标都差不多，一堆`_bucket`和`_sum`。"

**大师**："好问题。先说Histogram。你上大学的时候，学校食堂统计打饭量，是怎么做的？"

**小白**："分档呗。'二两的站左边，三两的站右边，四两的站前面，半斤以上的站我对面'。"

**大师**："没错，这就是bucket——分箱。Histogram预先把数值范围切成几个区间（bucket），每次请求来了，把耗时往对应的bucket里一扔：小于0.01秒的算一个，0.01到0.05的算一个……每个bucket记录有多少请求落在了这个区间里，同时记录一个`_sum`（总耗时）和一个`_count`（总请求数）。分位数是你**查的时候在服务端用PromQL算出来的**。"

**小胖**："那Summary呢？"

**大师**："Summary是反过来的——分位数在**客户端就算好了**。客户端每次收到请求，除了更新`_sum`和`_count`，还维护一个滑动窗口，直接算出P50、P90、P99，然后把结果写在label里暴露出来：`{quantile="0.99"}`。这就是小白刚才看到的现象——Histogram和Summary算出来的P99不一样，因为一个在服务端算，一个在客户端算，算法和窗口都不同。"

**小白**："也就是说，Summary拿到的P99是'客户端本地视角'，Histogram的P99是'全局视角'？"

**大师**："精准。所以Summary有一个致命弱点：**不可聚合**。三个实例各自算了一个P99——100ms、200ms、150ms——你没法得出'集群层面的P99'。Histogram的bucket是原始数据，多实例的bucket可以加起来再算分位数，所以能跨实例聚合。"

---

**小胖**："等一下，我突然想起来一个问题。我之前把user_id当label用，Prometheus直接挂了。为啥user_id不能做label？"

**大师**："你班上同学有多少人？"

**小胖**："50个。"

**大师**："全校呢？"

**小胖**："几万吧……哦，我懂了。label的每个不同取值都是一条时间序列，user_id有几百万个就是几百万条线。用**班级号**做label就50条线，用**学号**就是几万条。这个叫**label cardinality（标签基数）**——label的取值数量直接决定了时间序列的数量。Prometheus官方建议：单个label的取值控制在100以内，单个指标的总时间序列不超过10万。user_id属于'无量级增长'的label，绝对不能做。"

**小白**："那metric name的命名有没有什么讲究？我见过`http_requests_total`、`http_request_duration_seconds`，这些后缀是约定俗成的吗？"

**大师**："是的，Prometheus有严格的命名规范。metric name用下划线`_`分隔，全部小写，必须有意义后缀：Counter必须用`_total`结尾（如`http_requests_total`），时间类用`_seconds`结尾（不是`_sec`也不是`_ms`），字节类用`_bytes`结尾。label也用下划线分隔，不能包含`-`或`.`。另外，`_bucket`和`_sum`是Histogram/Summary自动生成的后缀，不要手动创建同名的指标。"

**小胖**："那/metrics端点暴露出来的文本是什么样子的？我还从来没仔细看过。"

**大师**："来，咱们去工位上写一段代码看看。"

## 三、项目实战

### 环境准备

```bash
pip install prometheus-client==0.21.0
```

确保第2章的Prometheus实例正在运行（默认`http://localhost:9090`）。

### 步骤1：编写指标暴露服务

创建 `app.py`，同时暴露Counter、Gauge、Histogram、Summary四种指标：

```python
import time
import random
import math
from http.server import HTTPServer, BaseHTTPRequestHandler
from prometheus_client import (
    Counter, Gauge, Histogram, Summary,
    generate_latest, REGISTRY,
    CONTENT_TYPE_LATEST,
)

# ===== Counter：只增不减，记录HTTP请求总数（必须_total结尾） =====
http_requests_total = Counter(
    'http_requests_total',
    'Total number of HTTP requests',
    ['method', 'endpoint']
)

# ===== Gauge：可增可减，模拟当前内存使用量 =====
current_memory_usage_bytes = Gauge(
    'current_memory_usage_bytes',
    'Current memory usage in bytes'
)

# ===== Histogram：服务端计算分位数，预先定义bucket区间 =====
http_request_duration_seconds = Histogram(
    'http_request_duration_seconds',
    'HTTP request duration in seconds (histogram)',
    ['endpoint'],
    buckets=[0.01, 0.05, 0.1, 0.5, 1, 5]  # 6个bucket + 1个+Inf = 7个
)

# ===== Summary：客户端计算分位数，直接暴露quantile标签 =====
http_request_duration_summary_seconds = Summary(
    'http_request_duration_summary_seconds',
    'HTTP request duration in seconds (summary)',
    ['endpoint'],
)


class MetricsHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        # 模拟请求耗时：在100ms~600ms之间随机波动
        duration = random.uniform(0.1, 0.6)

        if self.path == '/metrics':
            self.send_response(200)
            self.send_header('Content-Type', CONTENT_TYPE_LATEST)
            self.end_headers()
            self.wfile.write(generate_latest(REGISTRY))
            return

        # 匹配 / 或 /api/orders 等路径
        endpoint = self.path if self.path != '/' else '/'
        method = self.command

        # Counter inc：每次请求+1
        http_requests_total.labels(method=method, endpoint=endpoint).inc()

        # Gauge：模拟内存使用量在 128MB 到 512MB 之间浮动
        memory_mb = 128 + 384 * (0.5 + 0.5 * math.sin(time.time() / 10))
        current_memory_usage_bytes.set(memory_mb * 1024 * 1024)

        # Histogram：记录本次请求耗时，自动落入对应bucket
        http_request_duration_seconds.labels(endpoint=endpoint).observe(duration)

        # Summary：记录本次请求耗时，客户端直接计算分位数
        http_request_duration_summary_seconds.labels(endpoint=endpoint).observe(duration)

        self.send_response(200)
        self.send_header('Content-Type', 'text/plain')
        self.end_headers()
        self.wfile.write(b'Hello Prometheus!\n')


if __name__ == '__main__':
    server = HTTPServer(('0.0.0.0', 8000), MetricsHandler)
    print('Metrics server running on http://localhost:8000')
    print('  /         — 模拟业务接口')
    print('  /metrics  — Prometheus指标暴露端点')
    server.serve_forever()
```

关键观察：
- **Counter**：只有一个值，每次请求单调递增。

- **Summary**：自动生成`_count`、`_sum`和带`quantile`标签的分位数。分位数是客户端实时算的。

| 指标类型 | PromQL | 含义 |
|---------|--------|------|
| Counter | `rate(http_requests_total[1m])` | 过去1分钟内每秒请求速率 |
| Gauge | `current_memory_usage_bytes` | 当前内存使用量瞬时值 |
| Histogram | `histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[1m]))` | 基于bucket计算的P99延迟 |
| Summary | `http_request_duration_summary_seconds{quantile="0.99"}` | 客户端直接暴露的P99延迟 |

### 步骤4：对比Histogram和Summary的P99差异

两条查询返回的P99大概率不相等（可能差10~50ms）。原因有三：

1. **计算位置不同**：Histogram在服务端用PromQL的`histogram_quantile()`计算，Summary在客户端Python进程中用滑动窗口算法计算。
2. **精度不同**：Histogram的分位数精度受bucket粒度影响——如果真实P99在0.3~0.5之间，而bucket设了`0.1, 0.5`，那Prometheus只能线性插值，存在误差。Summary在客户端维护的是精确样本，分位数精度更高（但窗口大小有限）。
3. **窗口不同**：`rate(…[1m])`取的是1分钟数据，而Summary的滑动窗口是客户端内部的时间窗口，两者覆盖的时间范围不一致。

### 可能遇到的坑与排查方法

**坑1：Label基数爆炸**

如果你不小心把user_id写进了label（别真这么干），可以这样排查：

```promql
topk(5, count by (__name__)({__name__=~".+"}))
```

这条PromQL列出时间序列数量最多的Top 5指标。如果某个指标的序列数远超预期，就得检查label取值是否"无量级增长"。

**坑2：Histogram bucket设置不合理**

查看bucket分布是否均匀：

```promql
rate(http_request_duration_seconds_bucket[1m])
```

如果`le="0.01"`和`le="0.05"`两条线始终贴在一起（值相同），说明这两个bucket之间没有数据，中间bucket浪费了。如果`le="+Inf"`持续增长但`le="5.0"`不动，说明超过5秒的请求都被丢进了兜底桶，最大bucket设小了。

改进方法：根据线上实际耗时分布调整bucket。假设P50约200ms、P99约800ms，合理的bucket设置应该是：

```python
buckets=[0.05, 0.1, 0.2, 0.3, 0.5, 0.8, 1.0, 2.0, 5.0]
```

**坑3：Counter归零（重启）的处理**

Counter在程序重启后会变成0。如果直接看原始值会看到一个"断崖"，但`rate()`函数内置了Counter Reset检测逻辑：

```python
# rate() 伪逻辑：
# 如果 当前值 < 上一个采样值 → 判定为Reset，用当前值直接作为增量
# 如果 当前值 >= 上一个采样值 → 正常计算差值
```

所以在Prometheus里查`rate(http_requests_total[1m])`，即使服务重启过，速率曲线也不会出现"负数"或"跳变"——`rate()`自动处理了。

## 四、项目总结

### Histogram vs Summary 对比

| 维度 | Histogram | Summary |
|------|-----------|---------|
| 灵活性 | 高。可以在服务端按任意百分位查询（P50/P90/P95/P99/P999都行） | 低。客户端预先算好了quantile，服务端不能改 |
| 准确性 | 受bucket粒度影响，bucket不够密时分位数有插值误差 | 客户端精确计算，但窗口大小有限 |
| 跨实例聚合 | 支持。多实例bucket可累加后统一算分位数 | 不支持。各实例分位数独立计算，无法算术合并 |
| 资源消耗 | bucket越多消耗越大（每个bucket都是一条时间序列） | 消耗小（只暴露预定义的几个分位数） |
| 适用场景 | 需要跨实例看整体延迟分布、或灵活切换百分位 | 不关心跨实例聚合的单实例场景 |

### 四大指标适用场景速查

- **Counter**：请求计数、任务完成数、错误总数——任何"只增不减"的累积量。典型命名：`*_total`。
- **Gauge**：内存使用率、CPU温度、连接池大小、队列长度——任何"有涨有落"的瞬时状态。
- **Histogram**：API响应延迟、请求体大小——需要查看分布、灵活计算多种百分位、且要跨实例聚合。
- **Summary**：黑盒探测延迟、单实例性能分析——不需要聚合，但需要客户端精确分位数的场景。

### 命名与Label规范要点

1. **metric name**：全小写，下划线分隔，Counter必须`_total`结尾，时间单位用`_seconds`，字节单位用`_bytes`。label名同样全小写下划线分隔，不能包含`-`。
2. **label value永远可控**：用枚举值（`method="GET"`、`status="200"`）没问题；用`user_id`、`request_id`这类每请求一变的值是自爆。
3. **`_bucket`和`_sum`是自动生成的**：不要手动创建同名的指标，会导致类型冲突。
4. **每个label的不同取值 = 一条独立时间序列**。新增label前问自己：这个label的取值数量是常数级的还是线性/指数级增长的？

### 常见踩坑经验

**案例一：metric name与label重名冲突。** 你创建了`requests_total{path="/api"}`，然后又创建了`requests_total{total="100"}`——label名叫`total`跟Counter的后缀`_total`没关系，但如果你取了一个跟其他metric名一样的label值，Prometheus会报错：`collected metric "requests_total" { label:<name:"total" value:"100" …> was collected before with the same name and label values`。label名本身不要用跟指标后缀相同的词以避免混淆。

**案例二：Summary分位数不可聚合。** 小王用Summary记录了三个实例的P99延迟，想在Grafana上展示"集群P99"。他把三个`{quantile="0.99"}`的值用`avg()`一包——算出来的值毫无意义，因为分位数不可线性相加。正确做法：换成Histogram，用`histogram_quantile(0.99, sum(rate(…_bucket[5m])) by (le))`。

**案例三：Go Prometheus client默认collector造成基数过高。** Go的`promhttp.Handler()`默认注册了Go runtime的collector（goroutine数量、GC统计等）。每个Go版本、每个实例都有一组默认指标。如果部署了100个pod，每个pod有50个go_开头的指标，总共就是5000条序列——还没开始写业务指标就已经占了大量基数。建议评估是否需要这些默认指标，不需要就用`promhttp.HandlerFor(registry, promhttp.HandlerOpts{})`传入自定义Registry。

### 思考题

1. **如果一个Counter因为程序重启归零了，`rate()`函数如何知道并给出正确结果？**（提示：Prometheus存储了每个采样点的原始值和时间戳，当检测到`当前值 < 上一个值`时，视为发生了一次Reset事件。`rate()`会将Reset点前后的差值重新计算，避免出现负数。如果你手动把Counter`set(0)`——这不是Counter的标准用法——会怎样？）

2. **Histogram的bucket应该如何选择才能既准确又省内存？**（提示：考虑"P50附近密、P99附近稀"的分布策略，以及如何通过分析线上流量来确定bucket边界。每个bucket都是一条时间序列，bucket数量 x label基数 x 实例数 = 总序列数。）扫描线上流量日志、用P50/P90/P99反推bucket边界、定期review bucket使用率——这三个步骤缺一不可。
