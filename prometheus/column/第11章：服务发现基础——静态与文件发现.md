# 第11章：服务发现基础——静态与文件发现

## 一、项目背景

运维老周的Prometheus监控着30台服务器，每台都跑着node_exporter。这个规模不大不小，但问题出在"变化"上——每次新上线一台机器或扩容，老周都要打开`prometheus.yml`，在`static_configs`下添加新的target，然后执行`kill -HUP`让Prometheus重新加载配置。要是只加一两台还好说，某次大促前批量扩容了20台服务器，老周对着表格一行一行地改配置文件，折腾了快半小时。更糟糕的是，他漏掉了3台——大促当天，这3台服务器完全处于监控盲区，CPU被打满、内存OOM都没人知道，最终导致部分服务雪崩。

如果说新增漏掉还算"不小心"，那下线机器就更像一个慢性病。老周经常在下线旧机器后忘记从配置中删掉对应的target，久而久之，Prometheus的Targets页面常年飘着一片红色的"DOWN"，页面加载越来越慢，告警规则中不得不加上`up == 1`来屏蔽已下线的机器——治标不治本。CTO在回顾会上直接问："能不能自动发现新机器？新机器一上线就自动接入监控，下线了就自动剔除，别让人工维护了。"

这个问题的本质是监控目标的"生命周期管理"。`static_configs`是最简单的目标配置方式——把target的地址硬编码写在YAML中，清晰直观，但在动态环境中维护成本极高，任何变更都需要手动修改配置并reload。而文件服务发现（`file_sd_configs`）恰好是一个承上启下的过渡方案：它依然通过文件来维护target列表，但这个文件可以由外部系统（CMDB、配置管理工具、自研脚本）自动生成和更新，Prometheus会定期读取文件，一旦发现变化就自动加载——无需手动reload。配合relabeling机制，我们还可以在target被抓取之前对其标签做增删改查，实现按角色过滤、自动打标签等高级功能。

本章从`static_configs`的局限出发，引入`file_sd_configs`的解决方案，再深入relabeling的基础用法，帮你建立起从"手工维护"到"半自动发现"的技术认知。

## 二、剧本式交锋对话

> 场景：监控值班室，小胖又接到告警——"target down"刷屏。

**小胖**（看着Prometheus Targets页面上一排红色）："大师，又来了！运维那边昨晚下线了5台测试机，但我们的Prometheus配置没更新，现在'DOWN'告警一直在叫。static_configs太烦了，每次都要手动改配置文件。"

**大师**："static_configs本来就是给静态环境用的。你看，它的定义方式就是'我给你一个列表，你就只监控这些'。在机器不怎么变的小环境里没问题，但一旦频繁扩缩容，就成了噩梦。你想想，static_configs的变更流程是什么？"

**小白**："改prometheus.yml → reload → 等生效。最关键的是要reload，万一不小心敲了个`kill -9`，Prometheus就挂了。"

**大师**："对。文件服务发现（file_sd_configs）能解决一半的问题。它的思路很简单：你把target列表写在一个独立的JSON或YAML文件里，Prometheus定期去读这个文件，发现内容变化了就自动更新target列表——不需要你手动reload。"

**小胖**："等等，这听起来和static_configs区别不大啊？只是把target从配置里抽出去了？"

**大师**："区别在于——这个文件可以不由人写，而是由程序自动生成。比如你们的CMDB系统里记录了所有服务器信息，可以写一个脚本每5分钟从CMDB API拉取最新列表，生成JSON文件覆盖掉旧的。Prometheus检测到文件变化，自动就把新机器加进来了，下线机器也自动剔除了。整个过程人零参与。"

**小胖**："有意思！那relabeling又是干什么的？刚才在配置文件里看到好多relabel_configs的配置，看得我头晕。"

**大师**："relabeling简单理解就是'在抓取数据之前，对target的标签做一波增删改查'。它有四个核心字段：`source_labels`指定你要从哪些现有标签取值，`regex`用正则匹配这些值，`replacement`是替换后的新值（可以用`$1`、`$2`引用捕获组），`target_label`指定把结果写入哪个标签，`action`决定做什么操作——默认是`replace`，还有`keep`、`drop`等。"

**小白**："所以replace就是把一个标签转换成一个新的？"

**大师**："没错。比如CMDB提供的主机列表里address字段是`10.0.1.1:9100`，你可以用relabeling从`__address__`中提取出IP部分存到`ip`标签里，方便后续查询时按IP过滤。更狠的是keep和drop——"source_labels里指定的标签值匹配regex就保留，不匹配就丢弃"。你可以靠这个只监控role=web的机器，把role=db的target全部过滤掉。"

**小胖**（若有所思）："明白了！文件SD负责'发现target'，relabeling负责'给target做标记和筛选'。那这两个搭配起来，是不是就能做到：CMDB脚本自动生成文件 → Prometheus自动加载 → relabeling自动过滤和打标签？"

**大师**："完全正确。你现在这套static_configs，上线新机器要三步：改配置、reload、还要手工确认。换成文件SD+relabeling之后，一切自动化——脚本跑一次，30秒内Prometheus就知道了。而且relabeling打上的标签（比如env=prod、region=beijing）可以在PromQL里直接用，告警规则也能按标签分组，这才是现代Prometheus的正确用法。"

## 三、项目实战

### 环境准备

确保Prometheus已在docker-compose环境中运行。创建文件SD目录：

```bash
mkdir -p /etc/prometheus/file_sd
```

### 步骤1：从static_configs改造为file_sd_configs

原始static_configs配置（每次增减target都要编辑prometheus.yml并reload）：

```yaml
scrape_configs:
  - job_name: 'node'
    static_configs:
      - targets: ['node1:9100', 'node2:9100', 'node3:9100']
        labels:
          env: 'prod'
```

改造为file_sd_configs后，主配置文件不再包含具体的target列表：

```yaml
scrape_configs:
  - job_name: 'node'
    file_sd_configs:
      - files:
          - '/etc/prometheus/file_sd/node_targets.json'
        refresh_interval: 30s
```

关键参数说明：
- `files`：支持glob模式（如`*.json`），但不递归子目录
- `refresh_interval`：Prometheus重新读取文件的间隔，默认5分钟，这里设为30秒便于验证

创建node_targets.json文件（JSON格式，也支持YAML）：

```json
[
  {"targets": ["node1:9100", "node2:9100"], "labels": {"env": "prod", "role": "web"}},
  {"targets": ["node3:9100"], "labels": {"env": "prod", "role": "db"}}
]
```

JSON和YAML格式的选择：JSON解析速度快于YAML，适合大文件场景；YAML可读性更好，适合小文件人工维护。格式修改后Prometheus会自动识别，无需在配置中声明格式。

### 步骤2：编写自动发现脚本模拟CMDB驱动

模拟从CMDB API获取主机列表并更新SD文件的脚本`update_targets.sh`：

```bash
#!/bin/bash
# 模拟从CMDB API获取主机列表
# 实际场景替换为：curl http://cmdb.internal/api/servers?env=prod
cat > /etc/prometheus/file_sd/node_targets.json << 'EOF'
[
  {"targets": ["10.0.1.1:9100", "10.0.1.2:9100"], "labels": {"env": "prod", "region": "beijing"}},
  {"targets": ["10.0.2.1:9100"], "labels": {"env": "prod", "region": "shanghai"}}
]
EOF
```

设置crontab每5分钟执行一次：

```bash
*/5 * * * * /usr/local/bin/update_targets.sh
```

验证：修改JSON文件内容（模拟新增或删除target），观察Prometheus Targets页面，在30秒内自动更新，全程无需手动reload。

### 步骤3：Relabeling实战——自动打标签和过滤

在file_sd_configs基础上叠加relabel_configs：

```yaml
scrape_configs:
  - job_name: 'node'
    file_sd_configs:
      - files: ['/etc/prometheus/file_sd/node_targets.json']
    relabel_configs:
      # 1. 从__address__中提取IP（10.0.1.1:9100 → ip: 10.0.1.1）
      - source_labels: [__address__]
        regex: '(.*):\d+'
        target_label: 'ip'
        replacement: '$1'

      # 2. 给所有target打上datacenter标签
      - target_label: 'datacenter'
        replacement: 'beijing'

      # 3. 只监控role=web的target（其他角色的target丢弃）
      - source_labels: [role]
        regex: 'web'
        action: keep

      # 4. 丢弃带env=test的target（防止测试环境混入）
      - source_labels: [env]
        regex: 'test'
        action: drop
```

每条规则的解读：

- **规则1**：从内置标签`__address__`（值为`10.0.1.1:9100`）中用正则捕获IP部分，写入自定义标签`ip`。`replacement`中的`$1`对应第一个捕获组。
- **规则2**：没有`source_labels`时，`replacement`就是静态值，直接写入`target_label`。这条规则给所有target打上`datacenter=beijing`。
- **规则3**：取`role`标签的值与`regex`匹配，匹配成功则保留target，否则丢弃。这里只有`role=web`的target能通过。
- **规则4**：取`env`标签的值与`regex`匹配，匹配成功则丢弃。防止测试环境的target误入生产监控。

关键提醒：relabel_configs按顺序执行，规则2的静态标签写入发生在规则1之后，但规则3的keep在规则4的drop之前判断——如果顺序写反，过滤逻辑就会出错。

### 步骤4：使用Prometheus UI调试Relabeling

打开Prometheus Web UI → Status → Service Discovery，展开node job：

- **Discovered Labels**：SD发现的原始标签（file_sd_configs从JSON文件中读取的标签以及Prometheus自动添加的`__address__`、`__meta_*`等内置标签）
- **Target Labels**：经过relabel_configs处理后的最终标签（这是真正被抓取时使用的标签）

对比两者的差异：
- `__address__`在Discovered Labels中存在，经过规则1提取出`ip`标签
- `role`标签通过规则3过滤，role=db的target在Target Labels中直接消失
- `datacenter=beijing`是完全由relabeling新增的标签

调试技巧：故意写一条错误的keep规则，比如`regex: 'nonexist'`，然后观察Service Discovery页面——所有target都会被标记为"dropped"，帮你直观理解relabeling的过滤效果。

### 测试验证

查询当前所有活跃target的标签和健康状态：

```bash
curl -s http://localhost:9090/api/v1/targets | jq '[.data.activeTargets[] | {labels: .labels, health: .health}]'
```

修改JSON文件后，等待30秒再次执行上述命令，确认新target自动加入、旧target自动消失。在Web UI的Service Discovery页面确认relabel_configs的转换效果——关键是Discovered Labels与Target Labels的差异。

### 常见坑点

1. **file_sd文件格式错误**：JSON文件如果少了引号或多了逗号，Prometheus不会在Targets页面显示任何错误，只会忽略整个文件内容，真正的错误信息需要看Prometheus日志（`level=error`，关键词"error parsing file_sd"）。踩过这个坑的人都会养成先用`jq`或`python -m json.tool`校验JSON格式的习惯。

2. **relabel_configs的执行顺序**：rules是顺序执行的。如果你在keep规则中引用了某个标签，这个标签必须是前面已经通过replace规则创建好的，否则keep无法匹配。同样，drop/keep的判断基于当前标签状态，顺序颠倒会导致逻辑完全相反。

3. **`__address__`标签的特殊性**：`__address__`可以在relabel_configs中被读取，但它的值最终决定Prometheus向哪个地址抓取数据。你可以用replace将其改写（修改抓取地址），但不能用labelmap之类的操作来处理它。

## 四、项目总结

### 服务发现方式对比

| 方式 | 特点 | 适用场景 | 自动感知变更 |
|------|------|----------|-------------|
| static_configs | 硬编码target，配置即文档 | 静态环境、测试环境、少于10台机器 | 不支持，需要reload |
| file_sd_configs | 文件驱动，外部系统可写入 | 中小规模动态环境、CMDB驱动 | 支持，定期读取自动更新 |
| K8s/Consul SD（后续章节） | 原生集成，实时监听 | 大规模云原生环境 | 支持watch实时更新 |

### Relabeling Actions速查

| Action | 用途 | 典型场景 |
|--------|------|----------|
| replace | 对标签值做替换或新增标签 | 从address提取IP、统一添加env标签 |
| keep | 仅保留匹配正则的target | 按角色/集群过滤采集目标 |
| drop | 丢弃匹配正则的target | 排除测试环境、排除特定节点 |
| labelmap | 批量重命名标签 | 将`__meta_xxx`开头的标签转为普通标签 |
| labelkeep/labeldrop | 按名称保留/删除标签集 | 精简标签，减少时间序列基数 |
| hashmod | 对标签值取模 | 多Prometheus实例分片采集 |

### 适用场景与注意事项

**适用场景**：静态环境首选static_configs（最直观）；中小规模的动态环境通过CMDB/脚本+file_sd_configs实现半自动化（本章推荐方案）；在全面迁移到K8s/Consul SD之前，file_sd也是验证relabel规则的理想沙盒。

**注意事项**：
- `refresh_interval`不宜设得过短（默认5分钟已足够），每次刷新都需要读取磁盘文件，间隔过短会增加I/O负担且没有实际收益
- JSON格式文件的解析速度优于YAML，大规模target列表建议用JSON
- 文件写入必须是原子操作——先写入临时文件，再`mv`覆盖目标文件，避免Prometheus在高并发场景下读到不完整的文件内容

### 常见踩坑经验

1. **文件权限问题**：Prometheus进程（通常以非root用户运行）无法读取file_sd目录或文件，导致target列表为空。检查方法：用Prometheus运行用户身份执行`cat`测试权限。

2. **JSON格式错误静默失败**：如前所述，JSON格式错误不会在UI显示，只能通过日志排查。建议在更新脚本中加入格式校验步骤——`jq . node_targets.json > /dev/null`。

3. **relabel_configs中source_labels写错**：某个标签名拼写错误或不存在时，relabeling不会报错，只是该条规则静默跳过。如果恰好是keep/drop规则，会导致target被意外丢弃或保留，排查时务必对照Service Discovery页面的Discovered Labels逐一核对。

### 思考题

1. **如果要实现"新机器加入后自动根据IP段打上region标签（10.0.1.x → beijing，10.0.2.x → shanghai）"，relabel_configs能实现吗？为什么不行？**

   *提示：relabel中的regex只能做简单的正则匹配和替换，无法实现"if IP属于某网段 then 打某标签"的条件分支逻辑。这种场景需要借助relabel的`source_labels`组合多个标签，或者将逻辑前置到CMDB脚本中处理。*

2. **file_sd_configs的refresh_interval设成1秒会有什么问题？**

   *提示：每秒读取一次磁盘文件会产生持续的I/O请求，在大规模target列表（数千条）下CPU和磁盘开销显著；同时Prometheus的SD模块更新频繁会导致内部状态不断重建，影响稳定性。实际上5分钟已经能满足绝大多数场景——秒级变更的需求应该考虑Consul/ZK等真正的事件驱动SD。*
