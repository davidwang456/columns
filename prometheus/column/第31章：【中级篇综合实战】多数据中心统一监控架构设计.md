# 第31章：【中级篇综合实战】多数据中心统一监控架构设计

## 一、项目背景

GlobalMart是一家总部位于北京的跨国零售巨头，业务遍布亚欧美三大洲。其IT基础设施跨越3个物理数据中心——北京（亚洲核心）、法兰克福（欧洲核心）、圣保罗（南美核心），以及2个公有云Region——AWS新加坡和Azure美东。每个DC和云Region各自独立搭建了Prometheus + Node Exporter监控，看似"每个地方都有监控"，实则问题严重。

**全球视图缺失**：北京GOC想看圣保罗的集群状态，必须VPN登录当地Grafana，忍受300ms+延迟和葡萄牙语界面。圣保罗凌晨3点磁盘满，北京团队第二天早上才知道——告警只推送到当地Slack。

**数据留存太短**：各DC的Prometheus仅保留7天本地数据。运维团队做月度容量规划，只能靠每周手动导出snapshot拼凑。季度趋势分析？根本不存在。

**告警规则混乱**：北京的`node_cpu_utilization`告警阈值80%，法兰克福定义90%，圣保罗干脆没设CPU告警。同一个坑被三个团队重复踩。CTO下了死命令：两周内搭建全球统一平台，要求三条硬指标——全球任意Grafana看全球视图、所有指标保留3年、告警规则统一管理且就近通知。

本章作为中级篇收官之战，综合运用第17-30章知识，设计并部署基于**Thanos（长期存储+全局查询）+ Prometheus Operator + Alertmanager集群 + Grafana多DC大屏**的完整架构。

## 二、剧本式交锋对话

**小胖**（运维新人）冲进会议室，挥舞三张架构图："大师！CTO让两周内搞定全球监控，我把Federation、Thanos、VictoriaMetrics方案都画了一遍——到底用哪个？"

**大师**转向**小白**："你先说说Federation有什么问题？"

**小白**放下杯子："第23章我们学过，Federation是主Prometheus去抓从Prometheus的`/federate`端点，但它只能拿聚合后的指标。Boss要看法兰克福某台具体机器的CPU曲线，Federation过来的数据早已丢掉了`instance`标签，根本没法钻取到单机粒度。"

**小胖**接话："Thanos就没有这问题——Sidecar直接读Prometheus TSDB block上传对象存储，Querier查询时通过Store Gateway拉回原始数据，包括`instance`、`pod`这些细粒度标签全部保留。而且Thanos用`--query.replica-label`按replica去重，比Federation的`honor_labels`方案严谨得多。"

**大师**追问："那为什么不直接全用VictoriaMetrics？第29章不是测过它查询比Thanos快40%吗？"

**小白**解释道："VM确实快，但它的集群模式运维门槛高。而且法兰克福已经有6个月的Prometheus历史数据了——用Thanos Sidecar直接就能把存量block推上MinIO，VM还得用`vmctl`做离线导入。最关键的是，VM生态里没有Compactor这种降采样工具，3年数据存储成本压不下来。"

**小胖**皱眉："告警怎么管？三个DC、上百条规则，靠手动同步YAML文件？"

**大师**在白板上画起Git分支："第24章学过Alertmanager的gossip协议，第26章学过PrometheusRule CRD。告警规则统一放一个Git仓库——`global/`目录放所有DC通用的基础设施告警（`node_filesystem_avail_bytes < 10%`之类），`dc-beijing/`、`dc-frankfurt/`子目录放各自特有规则。每个DC的ArgoCD自动sync对应目录加global目录到本地K8s集群。Alertmanager再通过`--cluster.peer`互相组gossip集群，同一条告警只会被一个实例发送通知，route规则按`datacenter`标签分流——北京告警走企业微信、法兰克福走Slack、critical级别走全球PagerDuty。"

**小白**补充："别忘了第21章的Recording Rules——'全球总QPS'这种跨DC聚合指标，如果Grafana每次刷新都实时`sum(rate(...))`跨三大洲算，延迟至少2秒。用Recording Rules预聚合到5分钟粒度的`global:qps:5m`指标，大屏秒出图。"

## 三、项目实战

### 环境准备

用3个Docker Compose栈模拟三个DC，或3个K8s namespace。每个DC栈包含Prometheus + Thanos Sidecar + Node Exporter + Alertmanager + 业务Exporter。中央组件：MinIO、Thanos Store Gateway、Thanos Compactor、Thanos Querier、VictoriaMetrics（Remote Write接收端）、全局Grafana。

### 步骤1：架构设计

```
[DC-Beijing]              [DC-Frankfurt]           [DC-SaoPaulo]
Prometheus+Sidecar        Prometheus+Sidecar        Prometheus+Sidecar
    |  Remote Write            |  Remote Write           |  Remote Write
    +----------->  VictoriaMetrics (Central)  <-----------+
                          |
                  Thanos Store GW ← Thanos Compactor ← MinIO
                          |
                     Thanos Querier
                     /             \
                Grafana       Alertmanager Cluster(gossip)
```

- **Sidecar**：持续将TSDB block上传MinIO，同时为Querier提供实时查询gRPC接口。
- **Store Gateway**：桥接MinIO中的历史block与Querier。`--index-cache-size=250MB`缓存索引以降低跨DC查询延迟。
- **Compactor**：对block降采样（原始→5m→1h），压缩合并——3年数据存储成本控制的核心。
- **Querier**：全局查询入口，连接所有Sidecar和Store Gateway，提供统一PromQL API。
- **VictoriaMetrics**：作为Thanos之外的冗余通道，单机版压缩率优于Prometheus TSDB约40%。
- **Alertmanager集群**：三实例gossip组网，按`datacenter`标签路由告警。
- **Grafana**：对接Querier为统一数据源，变量切换DC视角。

### 步骤2：配置Thanos实现全球视图

每个DC的Prometheus配置`external_labels`区分来源：

```yaml
# dc-beijing/prometheus.yml
global:
  external_labels:
    datacenter: beijing
    region: asia
    replica: beijing-01

# dc-frankfurt: {datacenter: frankfurt, region: europe}
# dc-saopaulo:  {datacenter: saopaulo, region: south-america}
```

Thanos Querier连接所有端点：

```yaml
thanos-querier:
  command:
    - 'query'
    - '--query.replica-label=replica'
    - '--query.replica-label=datacenter'
    - '--endpoint=thanos-sidecar-beijing:10901'
    - '--endpoint=thanos-sidecar-frankfurt:10901'
    - '--endpoint=thanos-sidecar-saopaulo:10901'
    - '--endpoint=thanos-store:10901'
```

`--query.replica-label`中加入`datacenter`确保按DC维度去重。

### 步骤3：全球告警管理

告警规则Git仓库结构：

```
monitoring-rules/
├── global/
│   ├── infrastructure.yml    # 通用基础设施告警
│   └── slo.yml              # 全球SLO告警
├── dc-beijing/
│   └── local-app.yml        # 北京特有应用告警
├── dc-frankfurt/
│   └── local-app.yml
└── dc-saopaulo/
    └── local-app.yml
```

Alertmanager跨DC集群配置（gossip协议）：

```yaml
# DC-Beijing Alertmanager
alertmanager-beijing:
  command:
    - '--config.file=/etc/alertmanager/alertmanager.yml'
    - '--cluster.listen-address=0.0.0.0:9094'
    - '--cluster.peer=alertmanager-frankfurt:9094'
    - '--cluster.peer=alertmanager-saopaulo:9094'

# 告警路由——按datacenter标签就近通知
route:
  group_by: ['alertname', 'datacenter']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  receiver: 'global-fallback'
  routes:
    - match:
        datacenter: beijing
      receiver: 'beijing-ops'
      routes:
        - match:
            severity: critical
          receiver: 'beijing-oncall'
    - match:
        datacenter: frankfurt
      receiver: 'frankfurt-ops'
```

`group_by`含`datacenter`确保不同DC告警不合到同一通知。各DC用各自习惯的通知渠道——北京走企业微信、法兰克福走Slack、圣保罗走Telegram。

### 步骤4：Grafana全球大屏设计

**3行×4列布局**（回顾第25章Grafana进阶）：

**第一行——全球总览**：全球总QPS(Stat)、活跃用户数(Stat)、DC健康状态(Status，3个红绿灯)、全球SLO合规率(Gauge，99.9%目标)。

**第二行——DC对比**：各DC CPU利用率Bar chart、内存使用率Bar chart、网络流量Time series（3条线分色）、活跃告警数量Table（按DC汇总）。

**第三行——业务指标**：全球订单成功率Time series、各DC支付成功率Bar chart、世界地图面板（Worldmap插件按DC坐标和SLO着色）。

**变量设计**：`$datacenter`（All/beijing/frankfurt/saopaulo）联动所有面板，`$time_range`（1h/6h/24h/7d/30d）切换时间窗口。面板使用`datacenter=~"$datacenter"`标签过滤。Annotations关联CI/CD部署事件——大屏QPS波动时，鼠标划过Annotation线即显示"法兰克福v2.3.1已发布"。

关键PromQL（回顾第19章PromQL进阶）：

```promql
# 全球总QPS
sum(rate(http_requests_total[5m]))

# 各DC可用性
avg(probe_success) by (datacenter) * 100

# SLO合规率——30天窗口子查询（第19章核心技巧）
sum_over_time((sum(rate(http_requests_total{status!~"5.."}[5m])))[30d:5m])
/
sum_over_time((sum(rate(http_requests_total[5m])))[30d:5m])
```

### 步骤5：自定义业务Exporter

GlobalMart需监控订单量和支付成功率（回顾第27章Exporter开发）：

```python
from prometheus_client import start_http_server, Counter, Gauge
import random, time

ORDERS_TOTAL = Counter('globalmart_orders_total', 'Total orders',
                       ['datacenter', 'status'])
PAYMENT_SUCCESS_RATE = Gauge('globalmart_payment_success_rate',
                              'Payment success rate', ['datacenter'])

def collect_business_metrics():
    dcs = ['beijing', 'frankfurt', 'saopaulo']
    for dc in dcs:
        ORDERS_TOTAL.labels(datacenter=dc, status='success').inc(random.randint(1,100))
        ORDERSTOTAL.labels(datacenter=dc, status='failed').inc(random.randint(0,5))
        PAYMENT_SUCCESS_RATE.labels(datacenter=dc).set(random.uniform(0.95, 0.999))

if __name__ == '__main__':
    start_http_server(8000)
    while True:
        collect_business_metrics()
        time.sleep(60)
```

生产中将`random`替换为数据库查询。指标带`datacenter`标签，无缝融入Thanos全局查询——在Grafana大屏上与`node_cpu_utilization`并列展示。

### 可能遇到的坑

1. **跨DC查询延迟**：北京Querier查法兰克福Sidecar，跨洲RTT 300ms+致超时。解决：Store Gateway配置`--index-cache-size=250MB`，Compactor启用降采样——查30天趋势自动命中5m降采样数据，扫描量减少80%。

2. **gossip端口**：Alertmanager集群需开放9094/tcp和9094/udp端口穿越防火墙。

3. **Grafana大屏慢**：太多数据源导致加载超时。用Recording Rules预聚合全球指标（第21章），Grafana查预聚合结果而非实时跨DC聚合。

4. **对象存储地域延迟**：跨区访问S3延迟高，考虑用各区域本地S3 bucket + Compactor在各区独立运行。

### 测试验证

在Thanos Querier UI执行`count(up) by (datacenter)`——验证看到3个DC的上报节点。手动停掉法兰克福一个Node Exporter——验证告警路由到`frankfurt-ops`而非`beijing-ops`。在Grafana大屏切换`$datacenter`——验证全部面板联动。

## 四、项目总结

### 架构选型决策

| 维度 | Federation | Thanos | VM+vmagent |
|------|-----------|--------|------------|
| 原始标签保留 | ✗ 丢失instance | ✓ 原样保留 | ✓ 完整保留 |
| 历史数据导入 | ✗ 需手动backfill | ✓ Sidecar自动上传 | △ 需vmctl |
| 存储成本 | 高（本地存储） | 中（对象存储+降采样） | 低（压缩率优） |
| 运维复杂度 | 低 | 中 | 中 |
| 社区成熟度 | Prometheus原生 | CNCF毕业 | 独立开源 |

选择Thanos核心理由：**存量历史数据零成本接入 + query.replica-label去重成熟 + Compactor降采样控制3年存储成本**。VictoriaMetrics作为Remote Write冗余通道，当Thanos链路故障时Grafana可切到VM应急。

### 适用场景与注意事项

**适用**：跨国公司多DC、混合云、需全球统一视图且3+年数据留存。**注意事项**：对象存储成本——务必启用Compactor降采样（5m保留30天，1h保留3年），存储费用可降70%；跨DC带宽——Sidecar每2h上传block（50-200MB），确保带宽充足；Grafana全球大屏配CDN加速静态资源。

### 常见踩坑案例

**案例1**：Thanos Querier跨DC查询超时。北京→法兰克福RTT 350ms，默认gRPC超时2分钟不够。解决：调高`--query.timeout`到5分钟，Store Gateway增大`--index-cache-size`。

**案例2**：gossip集群脑裂致告警重复。圣保罗网络闪断5秒，Alertmanager分裂成两个子集群各自独立发告警。解决：每实例配置完整N-1个peer（不只一个），PagerDuty dedup_key设为`{{ .Labels.alertname }}-{{ .Labels.datacenter }}`。

**案例3**：Recording Rule label不当。定义了`sum(rate(...)) by (status)`却漏了`by (datacenter)`，导致Grafana按DC筛选返回空。修正：Recording Rules必须保留所有下游查询标签维度。

### 思考题

1. **新增悉尼DC需要变更哪些？** 提示：external_labels添加悉尼标识、Querier endpoint注册、对象存储bucket、告警规则目录、Alertmanager gossip peer更新、Grafana变量添加选项、防火墙规则——共7个维度。

2. **如何实现中国DC数据不出境？** 提示：北京Sidecar上传到北京本地MinIO（境内），法兰克福和圣保罗上传到境外MinIO。部署两个Querier——境内Querier只查境内数据，境外Querier查全球。Compactor独立运行不跨区合并block。北京Grafana以境内Querier为数据源。
