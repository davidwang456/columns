# 第37章：高基数问题——根源、检测与治理

## 一、项目背景

某电商公司的Prometheus已经平稳运行了6个月，监控着2000+微服务和5万台容器。运维团队对这套体系颇有信心——直到某天凌晨两点，值班运维被P0告警电话叫醒。Prometheus Pod被OOM Kill了。

运维火速登录集群，发现Prometheus重启后内存从正常的32GB迅速飙到了48GB。还没来得及抓取heap profile，Pod再次被Kill——"重启→OOM→重启"的死循环形成，监控系统彻底瘫痪。紧急切走40%的抓取目标后，Prometheus终于勉强活了下来。排查发现：一个当天下午上线的新版API Gateway在处理每个HTTP请求时，错误地把`request_id`（每个请求生成一个UUID）作为label值写入了Histogram指标。短短几个小时，这个指标的label组合数爆炸到2000万条unique time series，TSDB的Head Block被直接撑爆。

这个事故引出了Prometheus运维领域公认的"头号杀手"——高基数问题。Prometheus的数据模型本质是：每条指标的每个唯一label组合 = 一条独立的time series = 占用一份独立的内存和磁盘。内存方面，每个series在Head Block中的memSeries结构体约占用1.2KB（包括约800B的labelset和约400B的chunk meta），2000万条series意味着至少24GB的纯内存开销。更糟糕的是，这些series一旦写入即"常驻"——只要在retention窗口内，即使数据已不再更新，内存也不会释放，直到TSDB的compaction周期将该block从Head中刷盘并最终被GC回收。这本质上是一个"只进不出"的系统：写入容易，清退难。

高基数治理也因此成为Prometheus生产化过程中绕不开的核心课题。本章将从根源剖析高基数的形成机制，介绍通过PromQL和promtool检测高基数源头的实用方法，系统讲解metric_relabel_configs裁剪、Recording Rules降基数、Exporter侧聚合三大治理策略，并探讨TSDB在高基数压力下的性能退化曲线及对应的监控告警方案。

## 二、剧本式交锋对话

**小胖**：（抱着笔记本冲进茶水间）大师救命！我们集群的Prometheus昨天刚升到48GB内存又OOM了。我查了`prometheus_tsdb_head_series`，数字飙到了600万！我就纳闷了，我们一共才不到2000个微服务，怎么搞出600万条series的？

**大师**：（放下咖啡杯）你先别急，说说最近上线了什么新服务？

**小胖**：两周前上了个新的用户行为追踪服务，工程师在每个指标上都加了`user_id`和`session_id`标签，说方便按用户维度分析。

**大师**：（倒吸一口气）这两个标签马上删掉。`user_id`和`session_id`是典型的高基数标签——你们有多少用户？

**小胖**：大概300万月活用户……

**大师**：那好，算一笔账。哪怕你这服务只暴露一个指标`user_action_total`，每个用户一个`user_id`值 = 300万条series。再乘上`session_id`的数量——每个用户可能有多条session——这个组合就直接起飞了。高基数的本质不是标签种类多，而是标签的取值多。`request_id`、`user_id`、`session_id`、`trace_id`这类"每个请求/每个用户一个唯一值"的标签，本质上就是无限基数。

**小白**：（探过头来）大师，那怎么判断一个label是不是安全的？我们不可能都不用label吧？

**大师**：问得好。安全的label有三类：第一类是枚举值，比如`status=success/failure/unknown`，取值就两三个，完全安全；第二类是有限值，比如`HTTP method=GET/POST/PUT/DELETE`，数量固定且小；第三类是范围值——不要把原始值直接做label，而是用bucket切割。比如响应时间，你不需要把每个毫秒值都做label，而是用`le="0.1"`、`le="0.5"`、`le="1"`这样的bucket边界。

**小胖**：我还有个问题。就算我删了`user_id`，那些已经写入的600万series能自动消失吗？

**大师**：不能。这就是最棘手的地方——series一旦在Head Block中注册，就会一直占用内存，直到对应的block被compaction赶出Head、落到磁盘、然后等retention到期被GC。中间整个过程可能长达数天甚至数周。所以治理高基数必须在数据写入前拦截，写入后再救火就很被动了。

**小白**：那TSDB到底能扛多少series？有没有一个参考线？

**大师**：（在白板上画了一条曲线）经验上分三个区域：100万series以下，查询和compaction都正常，这是安全区；100万到500万之间，查询开始明显变慢，compaction的压力增大，这是黄区；超过500万，compaction严重卡顿、WAL回放时间激增、OOM风险骤升，这是红区。你们600万已经踩在红区里了。每个series ~1.2KB，600万就是7GB+的纯memSeries开销，还不算chunk数据本身的内存。

**小胖**：那我现在能做些什么？

**大师**：三步走。第一步，在`metric_relabel_configs`里用`labeldrop`直接把`user_id`和`session_id`这两个标签摘掉——这是在采集后、存储前执行的，可以防止新的series继续涌入。第二步，如果你还需要按用户维度做分析，把原始数据导到其他地方（比如ClickHouse或Loki）去做OLAP，Prometheus只保留聚合后的结果。第三步，用Recording Rules把按用户维度拆分的指标聚合成总量指标，把基数降下来。

**小白**：（若有所思）所以`metric_relabel_configs`的drop和labeldrop还是不一样的？

**大师**：对，很多人搞混。`action: drop`是当标签值匹配某个正则时，丢弃整条series；`action: labeldrop`是删除匹配的标签名本身，无论它的值是什么。如果你只想干掉`user_id="12345"`这条series，用drop；如果你想删除`user_id`这个标签（不管值是什么），用labeldrop。在高基数场景下，通常用labeldrop一次性摘掉整个高基数标签。

## 三、项目实战

### 环境准备

- Prometheus运行中（建议已有5万+series便于观察）
- `promtool`命令行工具（通常随Prometheus发行版附带）
- 可选：avalanche工具（用于生成高基数指标进行模拟测试）

### 步骤1：检测当前的高基数指标

首先使用PromQL快速定位高基数源头：

```promql
# Top10 最高基数的指标名（按unique label组合排序）
topk(10, count by (__name__)({__name__!=""}))

# 找出哪个label的值最多——针对可疑指标
count by (user_id)(http_requests_total)

# 查看特定指标的总体基数
count(http_requests_total)
```

然后利用Prometheus自监控指标判断系统整体状况：

```promql
# 当前总活跃series数（核心指标）
prometheus_tsdb_head_series

# Series创建速率——如果突然飙升说明有新指标/新label值涌入
rate(prometheus_tsdb_head_series_created_total[5m])

# Head中chunk数量，高基数下会异常膨胀
prometheus_tsdb_head_chunks
```

使用promtool对TSDB数据目录做离线分析：

```bash
promtool tsdb analyze /prometheus/data/
```

注意输出中的两项关键信息："Label names most involved in churning"揭示了哪些label正频繁产生新series；"Most common label pairs"则展示按出现频次排序的label组合，帮助快速发现异常的高频组合。

### 步骤2：使用metric_relabel_configs裁剪高基数标签

**场景A**：API Gateway误带了`request_id`标签。

```yaml
scrape_configs:
  - job_name: 'api-gateway'
    static_configs:
      - targets: ['gateway:8080']
    metric_relabel_configs:
      # 直接删除整个request_id标签
      - source_labels: [request_id]
        regex: '.+'
        action: labeldrop

      # 如果request_id有前缀模式（如 "svc-a-xxxx"），可提取前缀降基数
      - source_labels: [request_id]
        regex: '([a-z]+-[a-z]+)-.*'
        target_label: 'service_id'
        replacement: '$1'
      - source_labels: [request_id]
        regex: '.+'
        action: labeldrop
```

验证效果：

```promql
# 配置前：count(http_requests_total) = 20000000
# 配置后：count(http_requests_total) ≈ 200（仅由status等枚举label组合决定）
```

**场景B**：`user_id`标签覆盖100万用户，需要降基数而非完全删除。

```yaml
metric_relabel_configs:
  # 将VIP用户映射到user_tier=vip
  - source_labels: [user_id]
    regex: 'VIP_\d+'
    target_label: 'user_tier'
    replacement: 'vip'
  # 其余用户映射到user_tier=normal
  - source_labels: [user_id]
    regex: '.*'
    target_label: 'user_tier'
    replacement: 'normal'
  # 最后删除原始user_id标签
  - source_labels: [user_id]
    regex: '.+'
    action: labeldrop
```

效果：100万条series压缩为2条（`user_tier=vip`和`user_tier=normal`）。

### 步骤3：Exporter侧聚合——从源头控制基数

在应用代码中，采集指标之前先对label值做规范化。

**Python（prometheus_client库）**：

```python
from prometheus_client import Counter
import re

http_requests = Counter('http_requests_total', 'Total HTTP requests', ['endpoint'])

def normalize_path(path):
    # /users/123/orders → /users/{id}/orders
    path = re.sub(r'/users/\d+', '/users/{id}', path)
    # /orders/abc-def-456 → /orders/{order_id}
    path = re.sub(r'/orders/[a-f0-9-]+', '/orders/{order_id}', path)
    return path

# 正确做法：先规范化再打点
http_requests.labels(endpoint=normalize_path(request.path)).inc()

# 错误做法（会导致高基数）：
# http_requests.labels(endpoint=request.path).inc()
```

效果：200万个各不相同的URL路径被压缩为约100个模式化endpoint。

**Java（Micrometer）**：

```java
// 错误做法：路径参数会变成tag值，产生高基数
@GetMapping("/users/{id}/orders")
public Order getOrders(@PathVariable String id) {
    // id会作为tag被记录 → 一个用户一条series
}

// 正确做法：使用固定tag值，在业务层屏蔽动态参数
@Timer(value = "http.requests", extraTags = {"endpoint", "users_orders"})
@GetMapping("/users/{id}/orders")
public Order getOrders(@PathVariable String id) {
    // tag值固定为 "endpoint=users_orders"，不随id变化
}
```

### 步骤4：使用Recording Rules降基数

当原始数据天然带有合理的高基数（如Pod监控中每个Pod一条series），而长期存储不需要保留最细粒度时，使用Recording Rules预聚合。

```yaml
groups:
  - name: pod_cpu_aggregation
    rules:
      # 最细粒度：per pod（基数高，仅保留7天原始数据）
      - record: pod:container_cpu_usage:rate5m
        expr: rate(container_cpu_usage_seconds_total[5m])

      # 中等粒度：per namespace（基数适中，保留30天）
      - record: namespace:container_cpu_usage:sum_rate5m
        expr: sum(rate(container_cpu_usage_seconds_total[5m])) by (namespace)

      # 粗粒度：per cluster（基数极低，适合长期存储）
      - record: cluster:container_cpu_usage:sum_rate5m
        expr: sum(rate(container_cpu_usage_seconds_total[5m])) by (cluster)
```

查询策略配合时间分层：最近7天查Pod级`pod:container_cpu_usage:rate5m`；7～30天查namespace级；30天以上查cluster级。结合递归联邦（federation），可以让主Prometheus只拉取粗粒度Recording Rules的结果，细粒度数据留在各机房Prometheus本地。

### 步骤5：TSDB性能退化曲线监控

部署以下告警规则持续监控自身健康度：

```yaml
groups:
  - name: cardinality_alerts
    rules:
      - alert: HighCardinalityDetected
        expr: prometheus_tsdb_head_series > 1000000
        for: 10m
        labels:
          severity: critical
        annotations:
          summary: "Prometheus series数超过100万，当前值: {{ $value }}"

      - alert: CardinalityGrowing
        expr: deriv(prometheus_tsdb_head_series[1h]) > 10000
        for: 30m
        labels:
          severity: warning
        annotations:
          summary: "Series数正以每小时1万+的速度增长，请检查是否有高基数label泄露"

      - alert: CompactionSlowing
        expr: rate(prometheus_tsdb_compactions_total[1h]) < 0.1
        for: 1h
        labels:
          severity: warning
        annotations:
          summary: "Compaction速率异常降低，可能由高基数导致Block过重"
```

### 常见踩坑与测试验证

**坑1**：`labeldrop`误删关键标签。`action: labeldrop`会删除标签名，不加区分地作用于该job下的所有指标。如果不小心drop掉了`instance`或`job`，会导致无法区分数据来源，所有target的数据混在一起。务必仔细审查drop的范围。

**坑2**：混淆`drop`与`labeldrop`。`drop`的语义是"如果匹配，丢弃整条series"；`labeldrop`是"只删标签名，保留series"。如果想针对特定的`user_id`值（如测试用户）丢弃对应series，用`drop`；想在所有series中移除`user_id`这个维度，用`labeldrop`。

**坑3**：Recording Rules执行后历史细粒度数据不可恢复。一旦决定只保留粗粒度Recording Rule结果并缩短原始数据retention，意味着无法回溯查看7天前的Pod级CPU详情。必须在设计阶段就做好分层存储方案。

**坑4**：高基数指标的WAL膨胀。大量stale series在WAL中会留下墓碑标记，占用WAL空间。应及时使用`promtool tsdb clean`清理已失效的block。

**验证方法**：

| 测试场景 | 预期结果 | 验证手段 |
|---------|---------|---------|
| 使用avalanche生成10万/50万/100万series | 内存随series数线性增长（~1.2KB/series） | `process_resident_memory_bytes` |
| 配置metric_relabel_configs前后对比 | `prometheus_tsdb_head_series`显著下降 | PromQL查询 |
| 模拟每次请求不同label值写入 | TSDB compaction延迟增加、OOM风险上升 | 观察`prometheus_tsdb_compactions_total`和内存曲线 |

## 四、项目总结

高基数问题是Prometheus运维中的核心挑战，但其治理并非无章可循。首先需要建立清晰的基数分类体系：低基数（枚举值<100个，如`status`、`method`）完全安全；中基数（100～10000个，如K8s的`node`、`pod`标签）需要关注增长趋势并配合Recording Rules做降级；高基数（>10000个，如`user_id`、`request_id`、`trace_id`）则必须在上游拦截，绝不允许进入TSDB。

治理策略存在清晰的决策树：**源头控制**是第一优先级——在Exporter/应用代码侧就做好label规范化，用模式匹配将动态值替换为静态模板（如`/users/123`规范化为`/users/{id}`），这是成本最低、效果最好的方式。**裁剪**是第二道防线——在`metric_relabel_configs`中通过`labeldrop`摘除高基数标签，或通过正则提取将高基数标签映射为枚举值。**降级**是第三道防线——用Recording Rules生成低基数聚合指标，原始高基数数据设置较短的retention，聚合后数据保留更长时间。**接受**是最后的选项——对于确实有短期分析需求的高基数指标（如灰度发布期间的`canary_version`），可以仅在有限窗口内保留。

关于检测工具的对比：`promtool tsdb analyze`适合离线深度分析，能揭示label churning和top label pairs；`count by (__name__)({__name__!=""})`的PromQL实时查询适合快速排查当前最高基数指标；`prometheus_tsdb_head_series`适合作为持续监控的基础指标。三者配合使用效果最佳。

高基数治理是一个持续过程，而非一次性任务。新服务的上线、业务增长带来的用户量攀升、K8s Pod频繁扩缩容、IoT设备数量的增加，都会持续引入新的基数压力。一个可行的工程实践是：将"Review Top 10 metrics by cardinality"纳入团队月度的可观测性巡检；在code review阶段对标label使用规范（禁止携带request_id/user_id/session_id/trace_id等唯一标识作为label）；对`prometheus_tsdb_head_series`设置分级告警。

最后分享三个经典踩坑案例，引以为戒：（1）某团队在A/B测试中将`experiment_id`（每次实验生成一个UUID）作为label，一次全量实验导致series翻倍，Prometheus直接OOM；（2）K8s监控中误将Pod UID作为label，每次Pod重建（UID变化）创建全新series，滚动更新一次产生数千条废弃series；（3）某网关将完整URL Query String作为label，导致`/search?q=xxx`中每个不同query参数值创建一条独立series，一天积累数百万条。

**思考题**：

1. 如果要监控100万个用户各自的行为指标，但Prometheus不接受100万series级别的高基数，有哪些替代方案？提示：考虑将用户维度数据推送到ClickHouse/Elasticsearch等OLAP系统，Prometheus仅保留聚合后的总量指标；或者使用分层采样（只监控Top 1%用户）+ k6/locust等工具做合成监控。

2. `metric_relabel_configs`和`write_relabel_configs`都能过滤指标，在高基数场景下哪个更适合？为什么？提示：`metric_relabel_configs`在采集后、写入本地TSDB前执行，直接从源头阻止高基数数据进入存储，是最高效的拦截点；`write_relabel_configs`在本地TSDB向外remote write时执行——如果数据已经写入本地TSDB，内存已经被占用，remote write端的过滤只是"亡羊补牢"。因此高基数治理必须先做好`metric_relabel_configs`。
