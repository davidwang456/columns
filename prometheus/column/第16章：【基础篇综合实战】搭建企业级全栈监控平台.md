# 第16章：【基础篇综合实战】搭建企业级全栈监控平台

## 1. 项目背景

速购商城是一家成立三年的中型电商公司，日均订单量约5000单，技术团队15人。公司目前正处于从"粗放运维"向"精细化运营"转型的关键期。CTO老张在最近一次故障复盘会上拍桌子："上个月线上故障4次，没有一次是我们主动发现的，全是用户投诉才知道！平均故障发现时间（MTTD）超过30分钟，这像话吗？"

现状令人揪心。5台Linux服务器支撑着整个业务：2台跑Nginx + 前端页面，2台跑Spring Boot API服务，1台MySQL数据库，外加一个独立的Redis缓存实例。运维小王用一套祖传的shell脚本做巡检——每5分钟跑一次，检查进程是否存活、端口是否监听、磁盘是否快满。但这套脚本的短板在一次次故障中暴露无遗：问题只能看"死活"，看不到"趋势"。CPU打到90%持续了40分钟但没触发阈值？脚本不管。API响应时间从50ms涨到800ms？脚本看不见。凌晨2点Redis内存打满导致服务雪崩？脚本跑了但没人看——告警只发到邮箱，谁会半夜盯着邮箱？

更让老张头疼的是汇报。每次周会老板要看"系统健康度"，小王只能手工登录各台服务器截图，拼成PPT，数据还是滞后的。老板问他"这个QPS曲线为什么有个坑"，他答不上来——因为根本就没有QPS曲线。

本章正是要解决这个困局。我们将综合运用基础篇前15章学到的所有知识——Prometheus Server部署与配置（第1-3章）、PromQL查询语言（第4-6章）、Node Exporter主机监控（第7-8章）、Grafana可视化（第9-10章）、告警规则编写（第11-13章）、Alertmanager告警管理（第14-15章）——搭建一套开箱即用的企业级全栈监控平台。技术选型覆盖六层监控体系：基础设施层（Node Exporter × 5）、中间件层（MySQL Exporter + Redis Exporter）、应用层（Nginx指标 + JVM指标）、外部可达性层（Blackbox Exporter）、可视化层（Grafana）、告警层（Alertmanager），使用Docker Compose一键编排，让速购商城从"盲飞"状态进入"仪表飞行"时代。

## 2. 剧本式交锋对话

**小胖**（挠头）：大师，前面15章我们学了一堆东西——Prometheus怎么装、PromQL怎么写、Node Exporter怎么用、Grafana怎么画图、Alertmanager怎么发告警……但说实话，真要让我从零搭一套监控，我还是有点虚。速购商城5台服务器，MySQL、Redis、Nginx、Spring Boot全要监控，我该从哪儿下手？

**大师**（抿了一口茶）：小胖，你这种感觉很正常。知识点就像乐高积木，散落一地的时候看不出名堂，但拼起来就很清晰了。你先把速购商城的监控需求按层次列出来。

**小白**（举手）：我试着分层！第一层是基础设施——5台服务器的CPU、内存、磁盘、网络，用Node Exporter搞定。第二层是中间件——MySQL和Redis得用专用的Exporter。第三层是应用层——Nginx的请求状态、Spring Boot的JVM指标。第四层……哦对了，还得监控网站本身能不能访问，这个得用Blackbox Exporter做HTTP探针。

**大师**：分得不错。这就是监控体系最经典的四层模型。那这么多Exporter，Prometheus怎么统一管理？它可是pull模型。

**小胖**：就是在一份prometheus.yml里配多个job！每个job指定要抓取的目标列表，Prometheus定时去拉。我统计了一下，一共有5个scrape job：prometheus自身、node_exporter（5台）、mysql_exporter、redis_exporter、blackbox_exporter。总共8个target，每15秒抓一次。

**小白**：但我有个疑问，5台node_exporter实例，它们的job值都一样，到了Grafana里我怎么区分哪台是Web服务器、哪台是API服务器、哪台是数据库？

**大师**：好问题。在prometheus.yml里给每个target配置label就行。比如Web服务器就加`role="web"`，API服务器加`role="api"`，DB服务器加`role="db"`。这个label会自动附着在每次采集的时序数据上，查询的时候用`node_cpu_seconds_total{role="web"}`就能精准定位了。

**小胖**：明白了！那告警规则我该怎么做？总不能所有告警都发给同一个人吧？

**大师**：告警要分级处理。我们设计两级：infra级别用critical和warning两个severity，比如InstanceDown、DiskFull这种算critical，发给运维群，通知渠道用企业微信+PagerDuty；应用级别的告警如HighErrorRate、SlowResponse算warning，发给开发群，通知渠道用企业微信就够了。Alertmanager里用route配置按severity标签分支路由。

**小白**：那Recording Rules呢？之前大师说它能预计算、加速查询，这个项目里适合用在哪儿？

**大师**：Recording Rules在这个项目的用法很巧妙。第一，Grafana大屏上要展示"QPS"这种聚合指标，如果每次刷新都用`rate(http_requests_total[5m])`实时算，数据量大时慢得要死。用Recording Rule预计算成`job:http_requests_per_second:rate5m`，Grafana直接读这个预聚合值，秒出图。第二，跨多个target做聚合时，Recording Rule相当于做了数据裁切，比如`instance:node_memory_used_percent:avg`预先把5台服务器的内存使用率算好，后面告警规则引用它就行。

**小胖**：那Grafana大屏怎么布局？老板说他要在会议室电视上看到"监控大屏"。

**大师**：从上到下四行设计。第一行Stat面板——业务总览，展示当日UV、API总QPS、订单总数、支付成功率。第二行状态面板+曲线——应用健康度，每个服务的UP/DOWN状态一目了然，下面配QPS、延迟P99、错误率的时序图。第三行Time Series面板——基础设施综合视图，CPU、内存、磁盘利用率用`$instance`变量联动，点击某台机器只看它的数据。第四行Table面板——当前活跃告警，列出告警名称、级别、持续时间。这样一张大屏，从业务到应用再到基础设施，逐层下钻，老板一眼就能判断整体健康状态。

**小白**：还有一个关键问题——如果监控系统本身把5台服务器都监控了，但它自己跑在哪？要是它自己挂了谁通知我们？

**大师**（笑）：这叫"监控的监控"，是进阶话题。简单说就是在办公网络另起一个Prometheus实例，专门监控生产环境的Prometheus和Alertmanager是否健康。不过这个留给后续"集群与高可用"章节展开。

## 3. 项目实战

### 环境准备

确保本地已安装Docker和Docker Compose（v2.0+）。所有组件通过docker-compose统一编排，模拟完整的5台服务器+中间件环境。

### 步骤1：整体架构设计

项目目录结构如下：

```
speedbuy-monitor/
├── docker-compose.yml

├── rules/
```

核心的`docker-compose.yml`编排文件，将所有组件纳入统一网络`monitor-net`：

```yaml
version: '3.8'

services:
  # ============ Prometheus 核心 ============
  prometheus:
    image: prom/prometheus:v2.50.0
    container_name: prometheus
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml:ro
      - ./rules:/etc/prometheus/rules:ro
      - prometheus_data:/prometheus
    command:
      - '--config.file=/etc/prometheus/prometheus.yml'
      - '--storage.tsdb.path=/prometheus'
      - '--web.enable-lifecycle'
    networks:
      - monitor-net
    restart: always

  # ============ Node Exporter（5台模拟服务器） ============
  node-exporter-web1:
    image: prom/node-exporter:v1.7.0
    container_name: node-web1
    ports: ["9100:9100"]
    networks: [monitor-net]

  node-exporter-web2:
    image: prom/node-exporter:v1.7.0
    container_name: node-web2
    ports: ["9101:9100"]
    networks: [monitor-net]

  node-exporter-api1:
    image: prom/node-exporter:v1.7.0
    container_name: node-api1
    ports: ["9102:9100"]
    networks: [monitor-net]

  node-exporter-api2:
    image: prom/node-exporter:v1.7.0
    container_name: node-api2
    ports: ["9103:9100"]
    networks: [monitor-net]

  node-exporter-db:
    image: prom/node-exporter:v1.7.0
    container_name: node-db
    ports: ["9104:9100"]
    networks: [monitor-net]

  # ============ 中间件 Exporter ============
  mysql:
    image: mysql:8.0
    container_name: mysql
    ports: ["3306:3306"]
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: speedbuy
    networks: [monitor-net]

  mysql-exporter:
    image: prom/mysqld-exporter:v0.15.1
    container_name: mysql-exporter
    ports: ["9105:9104"]
    environment:
      DATA_SOURCE_NAME: "root:root123@(mysql:3306)/"
    networks: [monitor-net]
    depends_on: [mysql]

  redis:
    image: redis:7-alpine
    container_name: redis
    ports: ["6379:6379"]
    networks: [monitor-net]

  redis-exporter:
    image: oliver006/redis_exporter:v1.58.0
    container_name: redis-exporter
    ports: ["9121:9121"]
    command: ['--redis.addr=redis://redis:6379']
    networks: [monitor-net]
    depends_on: [redis]

  # ============ Blackbox Exporter ============
  blackbox-exporter:
    image: prom/blackbox-exporter:v0.24.0
    container_name: blackbox-exporter
    ports: ["9115:9115"]
    volumes:
      - ./blackbox.yml:/etc/blackbox_exporter/config.yml:ro
    networks: [monitor-net]

  # ============ Alertmanager ============
  alertmanager:
    image: prom/alertmanager:v0.27.0
    container_name: alertmanager
    ports: ["9093:9093"]
    volumes:
      - ./alertmanager.yml:/etc/alertmanager/config.yml:ro
    networks: [monitor-net]
    restart: always

  # ============ Grafana ============
  grafana:
    image: grafana/grafana:10.3.0
    container_name: grafana
    ports: ["3000:3000"]
    environment:
      GF_SECURITY_ADMIN_USER: admin
      GF_SECURITY_ADMIN_PASSWORD: admin
    volumes:
      - grafana_data:/var/lib/grafana
      - ./grafana/provisioning:/etc/grafana/provisioning:ro
    networks: [monitor-net]
    restart: always

  # ============ 模拟Web应用 ============
  nginx:
    image: nginx:alpine
    container_name: nginx
    ports: ["8080:80"]
    networks: [monitor-net]

networks:
  monitor-net:
    driver: bridge

volumes:
  prometheus_data:
  grafana_data:
```

端口规划说明：宿主机上，Prometheus占9090、Grafana占3000、Alertmanager占9093、Blackbox占9115；5个Node Exporter依次映射到9100-9104端口，互不冲突；MySQL Exporter单独用9105。所有容器通过`monitor-net`桥接网络互通，Prometheus通过容器名（如`mysql-exporter:9104`）抓取数据，无需关注宿主机端口。

### 步骤2：编写完整的prometheus.yml

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s
  external_labels:
    datacenter: beijing

alerting:
  alertmanagers:
    - static_configs:
        - targets:
            - 'alertmanager:9093'

rule_files:
  - '/etc/prometheus/rules/infra_critical.yml'
  - '/etc/prometheus/rules/infra_warning.yml'
  - '/etc/prometheus/rules/middleware_alerts.yml'
  - '/etc/prometheus/rules/app_alerts.yml'
  - '/etc/prometheus/rules/recording_rules.yml'

scrape_configs:
  # Job 1: Prometheus 自身监控
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # Job 2: Node Exporter（5台服务器，通过relabel打角色标签）
  - job_name: 'node_exporter'
    static_configs:
      - targets: ['node-exporter-web1:9100']
        labels: { role: 'web',   instance: 'web-server-01' }
      - targets: ['node-exporter-web2:9100']
        labels: { role: 'web',   instance: 'web-server-02' }
      - targets: ['node-exporter-api1:9100']
        labels: { role: 'api',   instance: 'api-server-01' }
      - targets: ['node-exporter-api2:9100']
        labels: { role: 'api',   instance: 'api-server-02' }
      - targets: ['node-exporter-db:9100']
        labels: { role: 'db',    instance: 'db-server-01' }

  # Job 3: MySQL Exporter
  - job_name: 'mysql'
    static_configs:
      - targets: ['mysql-exporter:9104']
        labels: { role: 'mysql' }

  # Job 4: Redis Exporter
  - job_name: 'redis'
    static_configs:
      - targets: ['redis-exporter:9121']
        labels: { role: 'redis' }

  # Job 5: Blackbox Exporter — HTTP探测
  - job_name: 'blackbox_http'
    metrics_path: /probe
    params:
      module: [http_2xx]
    static_configs:
      - targets:
          - 'http://nginx:80'
        labels: { service: 'speedbuy-web' }
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: 'blackbox-exporter:9115'
```

关键设计点：node_exporter的每个target都显式指定了`instance`和`role`标签，避免5个容器的默认instance（IP:端口）难以识别。Blackbox Exporter通过relabel_configs实现"地址重写"——Prometheus把探测目标传给Blackbox，Blackbox发起实际HTTP请求，结果返给Prometheus。

### 步骤3：告警规则体系设计

**`rules/infra_critical.yml`** — 基础设施严重告警：

```yaml
groups:
  - name: infra_critical
    rules:
      - alert: InstanceDown
        expr: up{job="node_exporter"} == 0
        for: 1m
        labels: { severity: critical, team: ops }
        annotations:
          summary: "服务器 {{ $labels.instance }} 已下线"
          description: "{{ $labels.role }} 角色服务器超过1分钟无响应"

      - alert: DiskFull
        expr: (1 - node_filesystem_avail_bytes{fstype!="tmpfs"} /
              node_filesystem_size_bytes) * 100 > 90
        for: 5m
        labels: { severity: critical, team: ops }
        annotations:
          summary: "磁盘使用率超过90%"

      - alert: OutOfMemory
        expr: node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes < 0.05
        for: 1m
        labels: { severity: critical, team: ops }
        annotations:
          summary: "内存不足，可用低于5%"
```

**`rules/infra_warning.yml`** — 基础设施预警：

```yaml
groups:
  - name: infra_warning
    rules:
      - alert: HighCPUUsage
        expr: 100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
        for: 10m
        labels: { severity: warning, team: ops }
        annotations:
          summary: "CPU使用率持续高于80%"

      - alert: HighLoad
        expr: node_load15 / count by(instance) (node_cpu_seconds_total{mode="idle"}) > 2
        for: 5m
        labels: { severity: warning }
        annotations:
          summary: "15分钟负载超过CPU核数2倍"

      - alert: DiskInodeUsage
        expr: node_filesystem_files_free{fstype!="tmpfs"} /
              node_filesystem_files > 0.85
        for: 5m
        labels: { severity: warning }
        annotations:
          summary: "磁盘inode使用率超过85%"
```

**`rules/middleware_alerts.yml`** — 中间件告警：

```yaml
groups:
  - name: middleware
    rules:
      - alert: MySQLDown
        expr: mysql_up == 0
        for: 1m
        labels: { severity: critical, team: dba }
        annotations:
          summary: "MySQL不可用"

      - alert: RedisDown
        expr: redis_up == 0
        for: 1m
        labels: { severity: critical, team: dba }
        annotations:
          summary: "Redis不可用"

      - alert: TooManyMySQLConnections
        expr: mysql_global_status_threads_connected /
              mysql_global_variables_max_connections * 100 > 80
        for: 5m
        labels: { severity: warning, team: dba }
        annotations:
          summary: "MySQL连接数超过最大连接数80%"

      - alert: RedisMemoryHigh
        expr: redis_memory_used_bytes / redis_memory_max_bytes > 0.8
        for: 5m
        labels: { severity: warning }
        annotations:
          summary: "Redis内存使用率超过80%"
```

**`rules/app_alerts.yml`** — 应用层告警：

```yaml
groups:
  - name: app
    rules:
      - alert: ServiceDown
        expr: probe_success{job="blackbox_http"} == 0
        for: 1m
        labels: { severity: critical, team: dev }
        annotations:
          summary: "{{ $labels.service }} 服务探测失败"

      - alert: SlowResponse
        expr: probe_duration_seconds{job="blackbox_http"} > 2
        for: 5m
        labels: { severity: warning, team: dev }
        annotations:
          summary: "{{ $labels.service }} 响应时间超过2秒"
```

### 步骤4：Grafana Dashboard设计

Grafana数据源预配置（`grafana/provisioning/datasources/prometheus.yml`）：

```yaml
apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    url: http://prometheus:9090
    access: proxy
    isDefault: true
```

"速购商城-全局监控大屏"四行布局及核心PromQL：

**第一行 — 业务总览（4个Stat面板）**：

| 面板 | PromQL |
|------|--------|
| 网站健康 | `probe_success{job="blackbox_http"}` |
| 服务器在线数 | `count(up{job="node_exporter"} == 1)` |
| MySQL状态 | `mysql_up` |
| Redis状态 | `redis_up` |

**第二行 — 应用健康（Status面板 + 时序图）**：

| 面板 | PromQL |
|------|--------|
| 各服务UP状态 | `up{job=~"node_exporter|mysql|redis"}` |
| HTTP探测延迟P99 | `probe_duration_seconds{job="blackbox_http"}` |
| HTTP探测成功率 | `probe_success{job="blackbox_http"} * 100` |

**第三行 — 基础设施（`$instance`变量联动）**：

设置变量`$instance`：`label_values(node_uname_info, instance)`

| 面板 | PromQL |
|------|--------|
| CPU使用率 | `100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle",instance=~"$instance"}[5m])) * 100)` |
| 内存使用率 | `(1 - node_memory_MemAvailable_bytes{instance=~"$instance"} / node_memory_MemTotal_bytes{instance=~"$instance"}) * 100` |
| 磁盘使用率 | `(1 - node_filesystem_avail_bytes{fstype!="tmpfs",instance=~"$instance"} / node_filesystem_size_bytes{fstype!="tmpfs",instance=~"$instance"}) * 100` |

**第四行 — 活跃告警列表（Table面板）**：

数据源选Prometheus，Query填`ALERTS{alertstate="firing"}`，Columns展示`alertname`、`severity`、`instance`、`summary`。

```bash
# 等待1分钟，确认告警状态变为resolved
```

### 可能遇到的坑

**坑1：Docker网络中hostname解析**。docker-compose内部通过服务名做DNS解析，所以prometheus.yml中用`mysql-exporter:9104`而非`localhost:9105`。如果宿主机测试某个Exporter的`/metrics`端点，要用`localhost:9105`。

**坑2：MySQL Exporter权限**。mysqld-exporter需要MySQL的连接权限。如果MySQL没有提前创建exporter用户，可在mysql容器启动后执行：

```sql
CREATE USER 'exporter'@'%' IDENTIFIED BY 'password';
GRANT PROCESS, REPLICATION CLIENT, SELECT ON *.* TO 'exporter'@'%';
```

在本项目的docker-compose中直接使用root账号简化了此步骤。

**坑3：多Node Exporter的instance标签**。如果不指定instance标签，5个容器默认instance都是`容器名:9100`，Grafana中难以区分。通过prometheus.yml中每个target显式指定`instance`标签解决。

**坑4：Grafana数据源URL**。Grafana和Prometheus都在docker-compose的同一网络内，数据源URL必须用容器名`http://prometheus:9090`，不能用`localhost`，否则Grafana容器内部解析不到宿主机。

### 自建监控 vs 云厂商监控

| 维度 | 自建Prometheus | 云监控（如阿里云） | DataDog/New Relic |
|------|---------------|-------------------|--------------------|
| 成本 | 服务器成本+人力 | 按量计费，大规模后昂贵 | 按host计费，$15/host起 |
| 定制能力 | 极高，任意Exporter扩展 | 受限于厂商SDK | 中等，Agent能力有限 |
| 数据私有化 | 完全私有，数据不出机房 | 数据在云上 | 数据在SaaS端 |
| 维护负担 | 需要专人维护 | 免运维 | 免运维 |
| 适用场景 | 中小规模、合规要求高 | 云上业务 | 海外业务、SaaS友好 |

### 适用场景与注意事项

本章方案最适合以下场景：中小规模企业（10-50台服务器）、私有化部署要求、混合云环境、金融/政府等监管合规行业。对于纯公有云且规模在5台以下的小团队，直接使用云厂商监控更省心。

注意事项：第一，数据持久化务必挂载Volume（`prometheus_data`、`grafana_data`），否则容器重启后历史数据丢失；第二，当前架构Prometheus和Alertmanager都是单点，生产环境需升级到Thanos/Cortex或Prometheus联邦（留待进阶篇）；第三，容量规划——5台Node Exporter每15秒采集约2000个series，加上其他Exporter共计约5000个series，日均数据量约1.5GB，保留30天约需45GB。

### 常见踩坑经验

**案例1：Exporter端口管理混乱**。团队开发时在宿主机直接启动各种Exporter，端口冲突频发。统一采用docker-compose管理，每个Exporter容器内部用标准端口（9100），对外映射到不冲突的宿主机端口（9100-9105）。

**案例2：Blackbox Exporter无法探测外网**。Prometheus通过`relabel_configs`将探测目标传给Blackbox，但Blackbox模块配置了DNS解析走容器网络，外部域名无法解析。解决方式是在blackbox.yml中配置DNS服务器或使用IP直连。

**案例3：Exporter版本不兼容**。mysqld-exporter 0.14.x版本采集的指标名与0.15.x有差异（如`mysql_global_status_*`前缀变更），导致Grafana面板突然无数据。建议在docker-compose中锁定镜像版本号（`:v0.15.1`而非`:latest`）。

### 思考题

1. **如果要把这套监控系统扩展到50台服务器，哪些地方需要调整？** 提示：Prometheus单节点TSDB的series上限（约1000万），scrape_interval是否需要放宽，Recording Rules如何减轻查询压力，Grafana变量联动在大instance数量下的性能问题。

2. **监控体系本身如何实现高可用？哪些组件存在单点故障？** 提示：Prometheus Server、Alertmanager、Grafana都是单点。业界方案包括Prometheus联邦+多副本、Alertmanager Gossip集群、Grafana背后挂数据库实现会话共享。
