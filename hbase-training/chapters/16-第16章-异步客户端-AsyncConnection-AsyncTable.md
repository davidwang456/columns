### 开场一分钟（趣味钩子）

同步 API 像**在餐厅一道菜一道菜等上齐**；异步像**同时下单多道菜，上齐再开吃**。菜越多，你越需要**托盘大小（线程池队列）**和**最长等待时间（超时）**——否则餐厅门口会挤爆。

### 1）项目背景

- **开发**：高并发扇出读、与异步框架（WebFlux、Vert.x 等）集成；必须规范超时与异常链。
- **测试**：`CompletableFuture` 组合、超时取消、部分失败；压测线程池耗尽场景。
- **运维**：异步不等于省 RS 压力；扇出过高会放大集群负载。
- **若跳过本章**：网关层用线程池硬扛 I/O，扩展性遇到天花板。

### 2）项目设计（大师 × 小白）

- **小白**：「和同步比啥好处？」
- **大师**：「**线程阻塞少**；适合 I/O 密集、多行并行 Get。」
- **小白**：「更难调试？」
- **大师**：「是，需要规范 **超时、异常链、背压**。」
- **小白**：「能无限 allOf 吗？」
- **大师**：「不行；要有**最大并行度**与**熔断**。」
- **小白**：「异步还需要 Connection 复用吗？」
- **大师**：「要；资源模型与同步类似，别另起炉灶泄漏。」
- **段子**：小白说「异步了所以不用限流。」大师：「RS 正在写遗书。」

### 3）项目实战

使用 `AsyncConnection` 获取 `AsyncTable`，对多个 RowKey 发起 `get`，`CompletableFuture.allOf` 等待；设置统一超时。具体 API 以当前 HBase 版本为准，参阅 [Client API](https://hbase.apache.org/book.html#client) 与源码 `AsyncConnection`。

**建议实验步骤**

1. 选 20 个 RowKey，串行同步 Get 记时。
2. 改异步并行（限制并发 8），对比 P50 / P99（若工具支持）或总耗时。
3. 人为缩短超时，观察异常类型与业务降级路径。

**验收**：两段代码（同步 / 异步）+ 一张对比表 + 「最大并行度」取值理由。

### 4）项目总结

- **优点**：高并发友好；适合 I/O 密集组合读。
- **缺点**：编程与排查复杂度高；易踩 Future 泄漏与背压失控。
- **适用**：网关聚合、并行补全、批量详情拼装。
- **注意**：线程池与队列长度；与同步客户端超时配置一致化。
- **踩坑**：无限制 `allOf` 千万 Future；吞掉异常导致「静默空数据」。
- **测试检查项**：超时风暴、线程池耗尽、部分失败降级。
