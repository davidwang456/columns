### 开场一分钟（趣味钩子）

把数据栈想成「城市基础设施」：**MySQL** 像社区银行柜台（规则多、对账强）；**HBase** 像按门牌号取件的巨型蜂巢仓（单行/前缀范围极快，但不帮你全城联表）；**ES** 像黄页搜索；**Kafka** 像地铁（运人运货，但到站还要有人卸货入库）；**HDFS** 像地底下的管道——**仓库存地上，但顾客从不直接钻管道拿货**。第一课：**先选对「建筑类型」，再谈装修（RowKey）**。

### 1）项目背景

- **业务 / 开发**：避免把 HBase 当「分布式 MySQL」设计复杂 SQL 式查询，导致上线后延迟与成本失控；需求阶段就要写下**主访问路径**（点查 / 前缀 Scan / 批量导入），否则后期改 RowKey 成本极高。
- **运维 / SRE**：明确 HBase 依赖 HDFS（或兼容文件系统）与协调组件，容量与故障域与对象存储、消息队列不同；选型错误会直接传导为「扩容也救不了的热点」。
- **测试 / 质量**：选型错误会导致用例模型整个推翻，需在需求阶段对齐「无跨行事务、无丰富 SQL」等边界，避免用 OLTP 用例模板验收宽表存储。
- **若跳过本章**：团队会在错误场景里堆机器、调参数，最后发现瓶颈在**数据模型与访问模式**，返工成本由全员买单。

### 2）项目设计（大师 × 小白）

- **小白**：「我们有个订单表，能不能上 HBase 替代 MySQL？」
- **大师**：「先问**访问模式**：是不是主键点查 + 按 RowKey 范围扫？如果要复杂 JOIN、强 ACID 跨表，HBase 不是第一选择。」
- **小白**：「那和 ES 呢？」
- **大师**：「ES 偏**检索与聚合**；HBase 偏**海量行存与低延迟点查 / 顺序扫**。常组合：ES 查 ID，HBase 取明细。」
- **小白**：「Kafka 呢？」
- **大师**：「Kafka 是**日志流**；HBase 是**在线存储**。可以从 Kafka 消费后写入 HBase，职责不同。」
- **小白**：「底层不是 HDFS 吗？」
- **大师**：「数据文件在 HDFS 上，但**读写路径**由 RegionServer 服务，不是直接当文件系统用。」
- **小白**：「那我们全放 HDFS 文件里不行吗？」
- **大师**：「可以存文件，但没有 HBase 这层，你就得自己解决**索引、范围路由、一致性、Compaction**——相当于自建一套简化版 HBase。」
- **段子**：小白说「HBase + Phoenix 不就是 MySQL 了吗？」大师叹气：「那是**给会看执行计划的人**用的 SQL，不是给『随便 JOIN』的人用的。」

### 3）项目实战

**实战 A：10 分钟圆桌——场景分拣**

1. 每组从真实业务里选 **3 个场景**（如：用户画像、订单详情、运营多维筛选、消息投递状态、审计日志检索）。
2. 填表（可抄到白板 / 文档）：

| 场景 | 主访问路径（点查 / 前缀 / 全表？） | 事务需求 | 量级（行数 / QPS 量级即可） | 适合 HBase？ | 备注（组合架构？） |
|------|-----------------------------------|----------|----------------------------|--------------|-------------------|
|  |  |  |  | 是 / 否 / 需改造 |  |

3. **验收**：每个「适合」场景必须写出**拟定 RowKey 一句话**；每个「不适合」必须写出**替代技术**（如数仓、ES、RDBMS）。

**实战 B：选型表（讲师示例，可对答案）**

| 场景 | 适合 HBase？ | 备注 |
|------|--------------|------|
| 用户画像宽表，按 userId 读写 | 通常适合 | RowKey=userId 或带短散列前缀 |
| 报表多表 JOIN | 不适合为主存 | 用数仓或联邦查询 |
| 日志检索 + 全文 | 需配合 ES / Solr | HBase 存原文或摘要 |

**环境**：无需集群；有白板即可。有测试集群时可追加讨论：该场景是否会触发**热点**。

### 4）项目总结

- **优点**：水平扩展、稀疏宽表、与 Hadoop 生态一体。
- **缺点**：无通用 SQL、无跨行事务、严重依赖 RowKey 设计。
- **适用**：海量键值 / 宽表、时序与日志索引、特征与画像。
- **注意**：先定访问模式再定存储；组合架构要写清**主存储与投影层**谁为准。
- **踩坑**：用业务方熟悉的「表」字误导为 RDBMS；把 ES 的「慢查询能跑完」预期搬到 HBase。
- **测试检查项**：需求评审中是否记录「不支持的能力」与验收口径。
- **运维检查项**：是否与业务对齐 SLA 与依赖栈（HDFS、ZK 等）。

### 5）课堂自测与作业（讲师可选用）
