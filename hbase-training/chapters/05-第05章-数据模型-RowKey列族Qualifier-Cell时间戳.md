### 开场一分钟（趣味钩子）

HBase 的表像**巨型 Excel**，但规则很怪：**行号（RowKey）**决定你坐在哪一排；**列族**是「这一排只能设几个大类目」，类目数要少；**Qualifier** 是便利贴，可以随手贴新字段。最魔性的是：**同一格可以贴很多张时间旅行便利贴**（多版本），读的时候要想好要几张。

### 1）项目背景

- **开发**：列族在表级固定，Qualifier 可动态增加；设计评审要写清**热点与列族数量**。
- **运维**：列族数量影响 flush / compaction 行为与 RS 压力；多列族表调优更复杂。
- **测试**：验证多版本、TTL 需理解 **Cell 粒度**；断言时要指定版本或「最新」。
- **若跳过本章**：后面 RowKey、Filter、压缩全都会变成「死记硬背」。

### 2）项目设计（大师 × 小白）

- **小白**：「表要先建 100 个列吗？」
- **大师**：「**Qualifier 不必预建**；**列族**要规划，不宜过多。」
- **小白**：「同一格写两次呢？」
- **大师**：「每次是一个带 **timestamp** 的 Cell；可保留多 **VERSIONS** 或用最新覆盖策略。」
- **小白**：「时间戳谁生成？」
- **大师**：「默认服务端时间；也可客户端指定——**高级玩法，踩坑多**（见第 32 章）。」
- **小白**：「列族和 Qualifier 名字长短要紧吗？」
- **大师**：「**都要紧**：名字越长，KeyValue 元数据越多，缓存越亏。」
- **小白**：「那我一个业务模块一个列族？」
- **大师**：「常见错误；列族应按**访问模式与 IO 隔离**切，不是按组织架构建。」
- **段子**：小白说「我们把 JSON 塞一个 Qualifier 里就省列了。」大师：「省的是**设计脑力**，费的是**解析 CPU 与部分更新**。」

### 3）项目实战

**环境**：`training:orders` 或自建测试表。

```text
put 'training:orders', 'rk002', 'd:amt', '100'
put 'training:orders', 'rk002', 'd:amt', '200'
get 'training:orders', 'rk002', { COLUMN => 'd:amt', VERSIONS => 2 }
```

**延伸实验（选做）**

1. 设置列族 `VERSIONS => 3` 再写入 4 个版本，观察 `get` 行为。
2. 与讲师讨论：若业务要「审计不可改」，应多版本还是 WORM 存储？

**验收**：能口述「RowKey 不变、Qualifier 相同、时间戳不同」三者的关系。

### 4）项目总结

- **优点**：稀疏、灵活；适合半结构化快速演进。
- **缺点**：滥用列族与版本会放大 IO 与存储；设计失误难以后改。
- **适用**：半结构化、宽表、时序与画像。
- **注意**：列族名、Qualifier 不宜过长；TTL 与 VERSIONS 要与合规对齐。
- **踩坑**：以为像 MySQL 改列名很轻量；自定义时间戳导致「读不到刚写的」。
- **开发检查项**：VERSIONS、TTL 是否与业务一致；是否需 MinVersions。
- **运维检查项**：表描述是否审计过 CF 数量与压缩。
