### 0）本章路线图（由浅入深）

| 层次 | 你要达成的状态 | 建议用时 |
|------|----------------|----------|
| L1 听懂 | 知道「CPU 低也可能慢」：队列、IO、GC、cache | 20 min |
| L2 能做 | 从 UI / JMX 抄 5 个延迟相关指标并解释 | 45 min |
| L3 能讲 | 能把指标分层：入口 SLA → RS → 磁盘 / 网络 | on-call 级 |

### 开场一分钟（趣味钩子）

只看 CPU 像**只看心率不量血压**：运动员心率低但可能**低血压晕倒**。HBase 慢常在 **flush/compaction 队列、block cache miss、handler 堵塞**——它们不一定先把 CPU 打满。

### 1）项目背景

- **运维**：建立最小可观测性；告警阈值结合业务低峰校准；与 HDFS、ZK 指标联动。
- **开发**：理解客户端重试在指标上的反映；慢查询要带回 **Region、RowKey 范围、Scan 参数**。
- **测试**：压测报告附监控截图；混沌演练记录**告警是否触发、是否误报**。
- **若跳过本章**：排障靠猜，扩容靠运气。

### 2）项目设计（大师 × 小白）

- **小白**：「看 CPU 够吗？」
- **大师**：「还要看 **队列、flush、compaction、block cache 命中率、RPC 延迟**。」
- **小白**：「Master UI 和 RS UI 先看谁？」
- **大师**：「用户读写多在 **RS**；**表 / Region / 均衡**看 Master。」
- **小白**：「日志太多怎么办？」
- **大师**：「先建立**关键字手册**（slow、flush、compaction、handler、FSHLog…），再谈采样。」
- **小白**：「Prometheus 拉不到指标？」
- **大师**：「核对 **JMX exporter**、认证、网络策略；指标名随版本会变。」
- **段子**：小白说「磁盘还有 50% 空间。」大师：「**IOPS** 听了笑出声。」

### 3）项目实战

**步骤 1：抄指标（必做）**

从 Master / RS UI 或 JMX 记录至少 **5 个**指标：**名称 + 一句话含义**。建议覆盖：请求延迟类、memstore / flush、block cache、compaction 队列（按你集群可见项选）。

**步骤 2：对照官方**

阅读 [Metrics](https://hbase.apache.org/book.html#metrics)，把抄来的名字与文档对上号；对不上的标记「待确认版本差异」。

**步骤 3：慢请求回放（讲师演示 + 学员跟做）**

- 人为制造轻量慢 Scan（无 LIMIT）后立刻中止；在日志中找对应线索（安全环境下）。
- **验收**：每人提交「5 指标表」+ 1 段日志摘录（脱敏）。

### 4）项目总结

- **优点**：提前发现热点与慢节点；容量与版本升级有据。
- **缺点**：指标过多需分层告警；否则「狼来了」。
- **适用**：生产必备；培训环境可简化为「最小集」。
- **注意**：阈值与业务低峰校准；冷热数据切换季节要重调。
- **踩坑**：仅磁盘满才看集群；忽略 meta / NameNode 症状。
- **开发检查项**：业务埋点与 trace id；慢查询日志字段规范。
- **测试检查项**：压测是否导出监控截图；告警是否可回归验证。
