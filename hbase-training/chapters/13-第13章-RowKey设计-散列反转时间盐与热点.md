### 开场一分钟（趣味钩子）

顺序 RowKey 像**全班按学号顺序交作业**：老师（单个 RegionServer）改到腱鞘炎，其他同学（其他 RS）吃瓜。散列前缀像**抽签分队**：交作业快了，但**按学号找齐全班**就费劲——**查询模式要写进设计**。

### 1）项目背景

- **开发**：RowKey 决定延迟与均衡；顺序键易热点；定长字段避免 Scan 边界错误。
- **运维**：热点直接表现为单 RS CPU / 请求堆积；扩机器未必救得了错误 RowKey。
- **测试**：压测需覆盖「打散后」与「恶意顺序」对比；skew 场景进回归。
- **若跳过本章**：集群再扩也是「一台干活，多台围观」。

### 2）项目设计（大师 × 小白）

- **小白**：「用自增 long 当 RowKey 最省事。」
- **大师**：「**字典序连续**会把写入打到少数 Region；用 **hash 前缀、盐、反转时间戳** 等打散。」
- **小白**：「打散后怎么按用户查？」
- **大师**：「**查询模式要写在设计里**：若主要按 userId 查，把 **userId（定长）** 放进 RowKey，前面再加短散列。」
- **小白**：「盐（salt）放几位？」
- **大师**：「看租户数与热点容忍度；太短没用，太长 Scan 要合并多次。」
- **小白**：「反转时间戳干什么？」
- **大师**：「让**新数据在字典序上靠前**，方便『最近 N 条』前缀 Scan（常配合 userId）。」
- **小白**：「我能中途改 RowKey 吗？」
- **大师**：「相当于**搬家**；要双写、对账、切换读，成本极高。」
- **段子**：小白说「我们 RowKey 用 UUID。」大师：「那你的**范围查询**是用爱发电吗？」

### 3）项目实战

**设计对比**

- **方案 A**：`seq` 或时间戳单调递增前缀。
- **方案 B**：`hash4(userId) + pad(userId) + reverse(ts) + orderId`（字段定长规则团队自定）。

**动手**

```java
try (Connection conn = ConnectionFactory.createConnection(conf);
     RegionLocator locator = conn.getRegionLocator(TableName.valueOf("training", "orders"))) {
  HRegionLocation loc = locator.getRegionLocation(Bytes.toBytes("your_rowkey"));
  // 记录 region 名与 host
}
```

**压测思辨（可无集群，纸上推演）**

- 10 倍写入下哪种先触顶单 RS？若业务必须「按 seq 范围扫」，热点如何缓解（预分区、异步批量、多表）？

**验收**：提交两行 RowKey 样例 + 各自主要读写路径 + 热点风险评估（低/中/高）。

### 4）项目总结

- **优点**：合理设计可同时服务点查与范围扫。
- **缺点**：难以事后「改 RowKey」；需迁移或双写。
- **适用**：高写入、多租户、时序。
- **注意**：定长字段避免 Scan 边界错误；盐策略与运维监控联动。
- **踩坑**：只 hash 不保留业务前缀导致无法用户维 Scan；盲目 UUID 失去范围查询。
- **运维检查项**：是否有热点 Region 告警；请求热力图（若工具支持）。
- **测试检查项**：skew 压测场景是否入回归。
