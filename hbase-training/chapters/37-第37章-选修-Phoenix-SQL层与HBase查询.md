### 开场一分钟（趣味钩子）

Phoenix 像**给蜂巢仓贴了一个 SQL 对讲机**：你能用熟悉的 `SELECT`，但**仓库仍然是按 RowKey 取货**。如果你对讲机里喊「把全仓库蓝色箱子找出来」，对讲机不会变成**自动传送带**——它只会老实地**走遍每一排**（Scan）。

### 1）项目背景

- **开发**：团队已熟悉原生 API，需要 **SQL**、二级索引、联合查询以降低接入成本；必须会看 **执行计划**。
- **运维**：多一层 Phoenix 服务（或嵌入式）与 schema 工具，版本需与 HBase 对齐；JVM、连接池、查询超时纳入 SRE 面板。
- **测试**：SQL 语义与 HBase 底层仍受 RowKey、热点、Scan 代价约束，用例不能按「传统 OLTP」照搬；必须有 **EXPLAIN + 压测** 门禁。
- **若跳过本章**：团队要么拒绝一切 SQL，要么把 Phoenix 当 MySQL 用炸集群。

### 2）项目设计（大师 × 小白）

- **小白**：「上了 Phoenix 就能随便 JOIN 了吧？」
- **大师**：「Phoenix 把 SQL **编译成 Scan/协处理器**；大表 JOIN、无索引字段过滤仍会 **扫很多数据**，要会看 **执行计划**。」
- **小白**：「和 Hive 呢？」
- **大师**：「Phoenix 偏 **低延迟在线查询**；Hive 偏 **批分析**。选型看延迟与数据新鲜度。」
- **小白**：「salt buckets 干啥？」
- **大师**：「打散热点；但会影响**范围查询形态**，要读文档理解副作用。」
- **小白**：「二级索引爽不爽？」
- **大师**：「爽，但要付 **额外表、写入放大、构建延迟** 的账单。」
- **小白**：「能 `SELECT *` 吗？」
- **大师**：「技术上可以，职业上**看工资够不够赔**。」
- **段子**：小白在生产跑七表 JOIN。大师：「Phoenix 没哭，RS 哭了。」

### 3）项目实战

1. 阅读 [Apache Phoenix](https://phoenix.apache.org/) 官方文档中与 **HBase 版本兼容矩阵**。
2. 在测试环境：创建一张映射表，执行 `UPSERT` / `SELECT` / `CREATE INDEX`（若环境允许）；用 `EXPLAIN` 观察是否退化为全表 Scan。
3. 记录：盐（salt buckets）、主键与底层 RowKey 的对应关系；把**最坏 SQL** 截图存档（脱敏）。

**验收**：提交 `EXPLAIN` 输出 + 一句「若退化全表扫的替代方案」（改表、改索引、改应用）。

### 4）项目总结

- **优点**：SQL 接入、二级索引能力（视版本与配置）、生态工具多。
- **缺点**：增加组件与运维面；错误 SQL 仍可压垮集群。
- **适用**：需 SQL 的运营后台、中等复杂度查询、与现有 HBase 表共存。
- **注意**：索引表额外存储与写入放大；预分区与 RowKey 仍要规划。
- **踩坑**：把 Phoenix 当无限扩展的 MySQL；忽略 `EXPLAIN`。
- **运维检查项**：Phoenix 与 HBase、JDK 版本锁表；查询超时与并发限制。
- **测试检查项**：核心查询必须有 EXPLAIN 与压测基线。
