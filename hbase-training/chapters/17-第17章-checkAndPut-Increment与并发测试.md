### 开场一分钟（趣味钩子）

先 Get 再 Put 像**看一眼余额再刷卡**：你低头看手机的两秒里，别人可能已经刷走了。**checkAndPut** 像**银行柜员同时核对纸条和账本再盖章**——仍是单行世界里的「防插队」，不是分布式锁万能药。

### 1）项目背景

- **开发**：乐观锁、状态机迁移；计数用 `Increment`（注意边界与语义）；避免误用为全局锁。
- **测试**：并发冲突与重试路径；断言最终状态与冲突率；探索 ABA 是否可接受。
- **运维**：高冲突业务会放大写放大与热点；需监控单行更新频率。
- **若跳过本章**：库存、状态机容易出现「覆盖写」类隐蔽 bug。

### 2）项目设计（大师 × 小白）

- **小白**：「先 get 再 put 安全吗？」
- **大师**：「**不安全**；中间可能被改。用 **`checkAndPut`** 做 CAS。」
- **小白**：「Increment 呢？」
- **大师**：「适合**原子计数**；仍要理解边界、是否允许负值、与业务幂等关系。」
- **小白**：「ABA 问题？」
- **大师**：「值从 A→B→A，CAS 可能误判；关键业务要**版本列或业务令牌**。」
- **小白**：「能当分布式锁吗？」
- **大师**：「不推荐；TTL、会话、 fencing token 都要设计，通常用专用协调组件。」
- **段子**：小白说「我 checkAndPut 比较整行。」大师：「API 比较的是**单元格**，不是「整行对象」。」

### 3）项目实战

```java
Put put = new Put(row);
put.addColumn(fam, qual, Bytes.toBytes("PAID"));
boolean ok = table.checkAndPut(row, fam, qual, Bytes.toBytes("NEW"), put);
```

**并发测试（必做）**

- 多线程同时尝试 `NEW → PAID`；统计 `ok==true` 次数、最终单元格值是否唯一为 `PAID`。
- 记录：**冲突重试策略**（退避、最大次数）是否写进代码。

**Increment 实验（选做）**

- 对计数列并发 `increment` 1000 次，验证结果是否为期望值。

**验收**：测试输出（截图 / 日志）+ 一段文字说明失败时业务如何处理。

### 4）项目总结

- **优点**：单行原子 CAS；`Increment` 简化计数。
- **缺点**：冲突时需业务重试；误用场景多。
- **适用**：库存、状态机（谨慎跨系统）、计数器。
- **注意**：比较的是**单元格值**；与多列状态机组合时要格外小心。
- **踩坑**：误用存在性检查当分布式锁；忽略 ABA。
- **测试检查项**：并发冲突率、最终一致性、重试可终止性。

### 5）课堂自测与作业（讲师可选用）

2. `checkAndPut` 返回 false 时常见正确处理方式？
3. 何种业务会特别在意 ABA？

**作业**
