# HBase 培训讲义：统一模板与受众标签

本文档为《HBase 多部门培训计划（36 章）》的**写作规范**，供讲师与学员统一使用。请勿与单章技术细节混淆；具体章节见 `part1-basic.md`～`part4-capstone.md`。

---

## 一、统一四章结构（每章必含）

| 序号 | 章节名 | 写作要点 |
|------|--------|----------|
| 1 | **项目背景** | 分别从**业务/开发**、**运维/SRE**、**测试/质量**各写 1～2 句话：本章解决什么问题、不学会有什么风险。 |
| 2 | **项目设计（大师 × 小白）** | 3～6 轮对话；小白可代表「MySQL 思维」「怕麻烦」「想一把梭」；大师纠正误区并给出类比（Bigtable、字典序、Region 等）。可穿插 1 个小段子，控制篇幅。 |
| 3 | **项目实战** | 给出**可执行**步骤：Shell / Java / 运维命令 / 测试用例择一为主；附**主代码或命令片段**；注明前置环境（如单机 HBase、JDK 版本）。 |
| 4 | **项目总结** | 表格或列表：**优点、缺点、适用场景、注意事项**；**常见踩坑**单独成段；**测试检查项**、**运维检查项**（若本章相关）各 2～5 条。 |

---

## 二、受众标签（Dev / Ops / QA）

每章标题下增加一行：

```text
受众：主【Dev|Ops|QA】 辅【…】  难度：【基础|中级|高级】
```

说明：

- **Dev**：设计与实现、客户端 API、RowKey、一致性语义、源码阅读。
- **Ops**：部署、参数、容量、备份、监控、Compaction/Split、故障与 RIT、安全。
- **QA**：用例、数据、自动化、性能与混沌、验收标准。

**辅受众**表示应旁听或阅读「总结」中的检查项，不必全程深挖。

## 三、作业示例（讲师可选用）

### 示例 A：开发向（第 6～7 章类）

1. 使用 `try-with-resources` 完成：建连、对 `training:demo` 表执行一次 Put + Get，RowKey 自定。
2. 在代码中打印 `Result` 中每个 Cell 的 family、qualifier、timestamp。
3. 提交：源码 + 运行截图或日志（脱敏）。

### 示例 B：运维向（第 11、19 章类）

1. 列出当前集群 Master / RegionServer 的 JMX 或 UI 中至少 5 个与请求延迟相关的指标名称及含义。
2. 在测试环境完成一次 snapshot `snapshot_training_日期`，并写出恢复到新表名的步骤（不执行生产）。

### 示例 C：测试向（第 8、10、12 章类）

1. 设计 5 条用例覆盖：`Scan` 边界（含空结果）、并发写入与 Scan 交错、模拟客户端超时后的重试策略（预期行为说明）。
2. 输出：用例表（前置、步骤、预期、优先级）。

## 四、检查表示例

### 开发评审检查表（节选）

- [ ] RowKey 是否与主要查询路径一致？是否存在顺序写入热点？
- [ ] 是否明确单行原子与多行 batch 非原子？
- [ ] 超时与重试是否幂等？是否避免「超时当失败」重复写？
- [ ] `Scan` 使用场景是否接受「非表级快照」语义？
- [ ] `Connection`/`Table` 是否及时关闭或使用 try-with-resources？

### 运维变更检查表（节选）

- [ ] 表 DDL 是否含预分区或分裂策略评估？
- [ ] 重大参数变更是否有回滚方案与观察窗口？
- [ ] 备份 / snapshot 是否在变更前完成？
- [ ] 是否关注 Compaction、Split 窗口对 SLA 的影响？

### 测试验收检查表（节选）

- [ ] 是否覆盖 RowKey 边界与非法输入？
- [ ] 是否覆盖部分失败 batch 的断言？
- [ ] 性能用例是否给出基线与环境说明？
- [ ] 是否与运维对齐告警阈值与日志关键字？

## 六、大师 × 小白人设建议（保持趣味一致）

- **小白**：喜欢用关系型数据库、Redis、ES 类比；总想「一条 SQL 搞定」或「先上线再改 RowKey」。
- **大师**：耐心、少训人；多用「字典序」「Region 边界」「行级原子」等精准词；复杂处一句「详见官方文档第 × 节」。
