# 第07章 相关性与评分：从 BM25 到可解释搜索（增强版）

## 1. BM25 的直觉理解

### 1.1 核心评分机制

BM25（Best Match 25）是 Elasticsearch 默认采用的相关性评分算法，它代表了信息检索领域从经典 TF-IDF 向概率检索模型的重要演进。与早期简单的词频统计不同，BM25 通过引入**饱和度控制**和**文档长度归一化**，使得评分结果更符合人类对"相关性"的直觉判断。理解 BM25 的三核心要素——词频（TF）、逆文档频率（IDF）和字段长度归一化——是掌握搜索排序调优的基础。

#### 1.1.1 词频（TF）：查询词在文档中出现频率越高，相关性得分越高

词频（Term Frequency, TF）衡量查询词在目标文档中的出现次数。直观上，若"轻薄笔记本"在某商品标题中出现 3 次，显然比仅出现 1 次的商品更相关。但 BM25 并非线性叠加，而是引入**饱和度机制**：当词频达到一定阈值后，继续增加的边际贡献迅速递减。

BM25 的词频计算公式为：

$$\text{TF}_{\text{component}} = \frac{f \times (k_1 + 1)}{f + k_1 \times (1 - b + b \times \frac{dl}{avgdl})}$$

其中 $f$ 为词频，$k_1$ 为饱和度控制参数（默认 1.2），$dl$ 为文档长度，$avgdl$ 为平均文档长度，$b$ 为长度归一化参数。

| 词频 $f$ | 简化得分（$k_1$=1.2, $b$=0, $dl$=$avgdl$） | 边际增益           |
|:------ |:------------------------------------ |:-------------- |
| 1      | 1.00                                 | —              |
| 2      | 1.43                                 | +43%           |
| 5      | 1.88                                 | +31%（相对 $f$=2） |
| 10     | 2.08                                 | +11%（相对 $f$=5） |
| 50     | 2.38                                 | +4%（相对 $f$=10） |

这一特性对电商搜索尤为重要——它防止了关键词堆砌（如标题重复"笔记本笔记本笔记本"）获得不正当高分，同时保证内容丰富的商品获得合理评分。

#### 1.1.2 逆文档频率（IDF）：稀有词权重更高，常见词权重更低

逆文档频率（IDF）解决"常见词稀释相关性"问题。在电商语料中，"手机"可能出现在 30% 的商品中，而"折叠屏"仅出现在 2% 的商品中——后者更能表征用户特定意图，应获得更高权重。

BM25 的 IDF 计算公式为：

$$\text{IDF}(q) = \ln\left(1 + \frac{N - n(q) + 0.5}{n(q) + 0.5}\right)$$

其中 $N$ 为总文档数，$n(q)$ 为包含查询词 $q$ 的文档数。

| 场景             | $N$=1,000,000 | $n(q)$  | IDF 值    | 解读             |
|:-------------- |:------------- |:------- |:-------- |:-------------- |
| 稀有技术词"RTX4090" | 1,000,000     | 500     | **7.60** | 极高区分度，匹配即高度相关  |
| 品牌词"索尼"        | 1,000,000     | 50,000  | **2.94** | 中等区分度，品牌筛选作用明显 |
| 类目词"笔记本"       | 1,000,000     | 200,000 | **1.39** | 较低区分度，基础类目过滤   |
| 停用词"的"         | 1,000,000     | 950,000 | **0.05** | 几乎无区分度，权重极低    |

IDF 机制解释了为何搜索"索尼降噪耳机"时，精确匹配"索尼"品牌的商品会显著靠前——品牌词的文档频率远低于通用词，IDF 权重形成天然壁垒。

#### 1.1.3 字段长度归一化：短字段中的匹配词比长字段中的更有意义

字段长度归一化是 BM25 相较于 TF-IDF 的关键改进。其核心直觉：**同样的词频，出现在短字段中更能体现相关性**。例如，"iPhone"出现在标题"iPhone 15 Pro"（5 词）与描述"Apple 公司推出的全新一代智能手机..."（50+ 词）中，前者显然更能代表商品核心属性。

归一化因子通过参数 $b$ 控制：

$$\text{length\_norm} = 1 - b + b \times \frac{dl}{avgdl}$$

| $b$ 值        | 场景       | 效果                    |
|:------------ |:-------- |:--------------------- |
| **0.0**      | 完全忽略长度差异 | 长描述商品可能因词频高而不当占优      |
| **0.5**      | 温和归一化    | 平衡短标题与长描述的影响          |
| **0.75（默认）** | 标准电商场景   | 标题匹配显著优先于描述匹配         |
| **1.0**      | 严格归一化    | 极短字段获得极大优势，可能过度惩罚丰富描述 |

电商典型配置：标题字段 `b=0.75`（保护核心地位），描述字段 `b=0.3-0.5`（避免过度惩罚），确保信息密度与内容深度的平衡。

### 1.2 参数调优空间

BM25 的 $k_1$ 和 $b$ 参数提供了针对特定数据特征的调优空间，但调整必须基于量化分析而非经验猜测。

#### 1.2.1 k1 参数：控制词频饱和度上升速度（默认1.2）

| $k_1$ 取值    | 适用场景         | 典型业务            |
|:----------- |:------------ |:--------------- |
| **0.5-1.0** | 快速饱和，抑制关键词堆砌 | UGC 内容搜索（论坛、评论） |
| **1.2（默认）** | 平衡词频与饱和度     | 通用电商商品搜索        |
| **1.5-2.0** | 较慢饱和，重视内容深度  | 技术文档、知识库搜索      |
| **2.0+**    | 近似线性词频加权     | 需要精确词频区分的科研文献   |

**调优原则**：分析索引的词频分布——若核心词频集中在 1-2 次，降低 $k_1$ 避免过度饱和；若词频分布分散（1-10+ 次），提高 $k_1$ 保留区分度。

#### 1.2.2 b 参数：控制文档长度归一化作用（默认0.75，0.0禁用，1.0完全启用）

$b$ 参数的调整需结合字段语义角色：

| 字段类型            | 建议 $b$    | 理由            |
|:--------------- |:--------- |:------------- |
| 商品标题（10-30 词）   | 0.75-0.90 | 强化短标题的精确匹配优势  |
| 品牌字段（1-5 词）     | 0.00-0.30 | 长度方差极小，归一化需求弱 |
| 商品描述（100-500 词） | 0.30-0.50 | 避免冗长描述带来的噪声干扰 |
| 规格参数（固定长度）      | 0.00      | 结构化数据，无需长度校正  |

Elasticsearch 支持字段级相似度配置：

```json
PUT /products
{
  "mappings": {
    "properties": {
      "title": {
        "type": "text",
        "similarity": "BM25",
        "k1": 1.2,
        "b": 0.75
      },
      "description": {
        "type": "text", 
        "similarity": "BM25",
        "k1": 1.2,
        "b": 0.3
      }
    }
  }
}
```

## 2. 影响排序的常见因素

### 2.1 字段权重（boost）

字段权重是搜索调优中最直接、最灵活的工具，它将业务优先级转化为数学上的得分放大。

#### 2.1.1 查询时动态设置：通过 `boost` 参数调整字段权重

查询时 boost 通过 `^` 语法或显式 `boost` 参数指定，具有场景自适应的灵活性：

```json
{
  "query": {
    "multi_match": {
      "query": "iPhone 15",
      "fields": ["title^3", "brand^2", "category^1", "description^0.5"]
    }
  }
}
```

同一字段在不同场景下可动态调整：品牌查询强化 `brand^4`，功能查询强化 `description^2`，类目浏览保持 `category^1.5`。

#### 2.1.2 映射时静态设置：在 mappings 中预设字段权重

```json
PUT /products
{
  "mappings": {
    "properties": {
      "title": { "type": "text", "boost": 3.0 },
      "brand": { "type": "text", "boost": 2.0 }
    }
  }
}
```

**注意**：Elasticsearch 7.x+ 已废弃映射级 `boost`，推荐统一使用查询时 boost，便于版本控制和 A/B 测试。

#### 2.1.3 电商典型配置：title^3、brand^2、category^1 的层级权重设计

| 字段                  | 典型权重    | 业务 rationale | 调优注意事项                    |
|:------------------- |:------- |:------------ |:------------------------- |
| **title（商品标题）**     | 3.0-5.0 | 最直接表达商品核心属性  | 监控关键词堆砌导致的异常高分            |
| **brand（品牌名）**      | 2.0-3.0 | 强用户意图信号，高转化率 | 注意品牌别名、中英文差异              |
| **category（类目）**    | 1.0-1.5 | 基础相关性保障      | 避免权重过高导致误匹配               |
| **attribute（规格属性）** | 1.5-2.5 | 精准参数匹配，高购买意向 | 规范化属性值（"15.6寸"vs"15.6英寸"） |
| **description（描述）** | 0.3-0.8 | 补充召回，低精确匹配价值 | 长文本需配合低 $b$ 参数            |
| **tag（标签/卖点）**      | 1.0-2.0 | 运营可控的优化空间    | 建立标签质量审核机制                |

### 2.2 精确匹配优先（phrase）

#### 2.2.1 `match_phrase` 与 `match` 的核心差异：顺序敏感 vs 无序匹配

| 维度       | `match` 查询   | `match_phrase` 查询 |
|:-------- |:------------ |:----------------- |
| **召回机制** | 词项布尔 OR，词序无关 | 位置约束交集，顺序敏感       |
| **得分基础** | 最佳匹配字段的 BM25 | 短语匹配的 TF/IDF 变体   |
| **计算开销** | 标准倒排扫描       | 需加载位置信息，额外 20-50% |
| **典型应用** | 宽泛探索、长尾查询    | 品牌型号、功能短语、精确意图    |

`match_phrase` 的 `slop` 参数提供灵活性：`slop=0` 要求严格相邻，`slop=1` 允许插入一个词，中文搜索建议 `slop=1-2` 应对分词颗粒度差异。

#### 2.2.2 短语匹配权重应高于普通 match，避免得分被冲淡

生产环境推荐组合策略：

```json
{
  "query": {
    "bool": {
      "must": {
        "match": { "title": { "query": "轻薄笔记本", "boost": 1.0 }}
      },
      "should": {
        "match_phrase": { 
          "title": { "query": "轻薄笔记本", "boost": 3.0, "slop": 1 }
        }
      }
    }
  }
}
```

`must` 保证基础召回，`should` 的短语匹配以 **3 倍权重**叠加，确保精确短语匹配在排序中的优先地位。

### 2.3 多字段匹配（multi_match）

#### 2.3.1 跨字段查询的得分合并机制

| `type` 类型             | 得分计算方式      | 适用场景                             |
|:--------------------- |:----------- |:-------------------------------- |
| **`best_fields`**（默认） | 取单字段最高得分    | 字段间互斥，如 title vs description     |
| **`most_fields`**     | 累加所有字段得分    | 字段互补，如 title + tags + attributes |
| **`cross_fields`**    | 多字段视为单一虚拟字段 | 同一概念分散多字段，如 brand + model        |
| **`phrase`**          | 各字段短语匹配后取最高 | 需要短语精确性                          |
| **`bool_prefix`**     | 前缀匹配组合      | 搜索建议、自动补全                        |

`cross_fields` 解决电商数据建模的经典困境：品牌存 `brand` 字段、型号存 `model` 字段，查询"Apple iPhone"时，`cross_fields` 使分字段存储的商品能够像单字段一样被完整短语匹配。

#### 2.3.2 字段权重在 multi_match 中的叠加效果

`best_fields` 模式下，高权重字段的"垄断效应"需警惕：若 `title^3` 和 `description^1` 同时匹配，title 的得分通常是 description 的 3 倍以上，可能导致描述匹配的优质商品被压制。解决方案：采用 `most_fields` 累加描述贡献，或针对长尾查询动态降低 title boost。

## 3. 可解释性工具详解

### 3.1 `_explain` API 核心用法

`explain` API 是诊断搜索排序问题的手术刀，它将评分"黑箱"转化为可逐层拆解的透明计算树。

#### 3.1.1 单文档评分诊断：`GET /{index}/_explain/{doc_id}`

```bash
GET /products/_explain/SONY-WH1000XM5
{
  "query": {
    "multi_match": {
      "query": "索尼 耳机",
      "fields": ["title^3", "brand^2", "category^1"]
    }
  }
}
```

返回的层级结构：

```json
{
  "_index": "products",
  "_id": "SONY-WH1000XM5",
  "matched": true,
  "explanation": {
    "value": 15.234,
    "description": "max of:",
    "details": [
      {
        "value": 15.234,
        "description": "sum of:",
        "details": [
          {
            "value": 12.456,
            "description": "weight(title:索尼 in 1234) [BM25], result of:",
            "details": [
              { "value": 3.892, "description": "idf, computed as log(...)" },
              { "value": 2.96, "description": "tf, computed as freq / (...)" }
            ]
          }
        ]
      }
    ]
  }
}
```

**解读路径**：根节点总分 → 字段贡献分解 → BM25 三要素（IDF/TF/fieldNorm）→ 原始统计量验证。

#### 3.1.2 搜索请求全局解释：`"explain": true` 参数

```bash
POST /products/_search
{
  "explain": true,
  "query": { ... },
  "size": 10
}
```

全局 explain 为每个 `hit` 附加 `_explanation`，适用于批量诊断排序异常。注意性能开销——explain 计算增加约 **20-50% 查询耗时**，生产环境建议仅对抽样结果启用。

#### 3.1.3 返回结构解析：description、value、details 三层嵌套

| 字段            | 含义       | 诊断价值                                     |
|:------------- |:-------- |:---------------------------------------- |
| `description` | 计算类型说明   | 识别当前处理的是哪个字段、哪个词项、哪种运算                   |
| `value`       | 当前节点数值结果 | 量化该步骤的实际贡献，定位异常高/低值                      |
| `details`     | 子节点递归数组  | 下钻追踪，直至原始统计量（freq, docFreq, fieldLength） |

关键诊断信号：`description` 中出现 **"NO MATCH"** 或贡献为 0，提示字段匹配失败；`value` 异常高而 `details` 显示 `tf` 或 `idf` 不合理，提示参数调优或数据质量问题。

### 3.2 电商场景诊断实例

#### 3.2.1 场景：搜索"索尼耳机"时某商品排名异常

**现象**：搜索"索尼耳机"，索尼官方旗舰店 WH-1000XM5 排名第 15，第三方兼容配件排名第 3。

**初步假设**：品牌匹配失效、标题权重不足、或业务因子干扰。

#### 3.2.2 诊断步骤：执行 explain 查询 → 拆解 TF/IDF/FieldNorm 贡献 → 识别 boost 失效点

**步骤一：获取异常商品与预期商品的 explain 输出**

| 得分组件            | 索尼官方店（第15位）                           | 第三方配件（第3位）                           | 差异分析            |
|:--------------- |:------------------------------------- |:------------------------------------ |:--------------- |
| **总得分**         | 10.56                                 | 14.23                                | 差距 3.67 分       |
| title: "索尼"     | 3.45（tf=1, idf=3.89, fieldNorm=0.625） | 4.12（tf=1, idf=2.45, fieldNorm=0.89） | 官方店标题长，长度惩罚严重   |
| title: "耳机"     | 2.85（tf=1, fieldNorm=0.625）           | 6.78（tf=2, fieldNorm=0.89）           | 第三方标题堆砌"耳机"关键词  |
| **brand: "索尼"** | **0.00** [NO MATCH]                   | 0.00 [NO MATCH]                      | **双方均未匹配，异常！**  |
| category: "耳机"  | 4.26                                  | 3.33                                 | 类目权重过高，部分抵消品牌缺失 |

**步骤二：根因定位**

深入 `brand` 字段匹配失败原因：

- 官方店文档：`brand` 字段值为 **"SONY"**（英文大写）
- 查询分词结果：**"索尼"**（中文）
- 同义词词典：未配置 **"SONY" ↔ "索尼"** 映射

**核心问题**：品牌字段的中英文差异导致精确匹配失效，title 字段因长度惩罚（fieldNorm=0.625）无法弥补品牌 boost 2.0 的缺失。

#### 3.2.3 典型发现：品牌字段未匹配、类目权重过高、文档长度惩罚过度

| 问题类别        | 具体表现               | 修复策略                       |
|:----------- |:------------------ |:-------------------------- |
| **字段匹配失效**  | brand 字段中英文未统一     | 扩展同义词词典，或索引时标准化为中文         |
| **权重配置失衡**  | category^1 贡献 40%+ | 降至 0.8，或改用 filter 纯过滤      |
| **长度归一化过度** | 合理长标题被惩罚 37.5%     | title 字段 $b$ 从 0.75 降至 0.5 |
| **关键词堆砌套利** | 第三方 tf=2 来自"耳机耳机"  | 降低 $k_1$ 至 0.8，加速饱和        |

修复后验证：索尼官方店排名提升至第 2 位，explain 显示 brand 匹配贡献 4.68 分（占比 31%），各因子分布趋于合理。

### 3.3 对比验证方法

#### 3.3.1 控制变量法：固定查询词，逐次调整单一参数

| 实验组  | title boost | 索尼官方店排名 | 首屏品牌命中率 | 决策        |
|:---- |:----------- |:------- |:------- |:--------- |
| 基准组  | 3.0         | 15      | 35%     | —         |
| 实验 A | 4.0         | 8       | 52%     | 正向，继续     |
| 实验 B | 5.0         | 3       | 71%     | **最优**    |
| 实验 C | 6.0         | 2       | 68%     | 过度优化，长尾受损 |

每组实验保存完整 explain 快照，量化参数-效果因果关系。

#### 3.3.2 A/B 查询对比：并行执行两种查询结构

对比 `multi_match best_fields` vs 显式 `bool` 组合：

| 查询结构          | 索尼官方店得分 | 得分构成                                 | 长尾召回           |
|:------------- |:------- |:------------------------------------ |:-------------- |
| `multi_match` | 12.34   | title 垄断 89%                         | 1,247 条        |
| `bool` 显式     | 14.56   | title 62% + brand 24% + category 14% | 1,189 条（-4.6%） |

`bool` 显式结构使多字段贡献可见，品牌匹配得以体现，代价是轻微召回损失。

#### 3.3.3 业务目标校验：将 explain 结果映射到"相关性+转化率"双指标

| explain 特征  | 业务指标     | 关联假设               | 验证方法   |
|:----------- |:-------- |:------------------ |:------ |
| brand 字段匹配率 | 品牌商品点击率  | 品牌匹配 → 用户信任 → 点击提升 | 分组回归分析 |
| phrase 匹配占比 | 搜索转化率    | 精确需求满足 → 购买意愿增强    | A/B 测试 |
| 首屏得分离散度     | 结果多样性满意度 | 适度离散 → 避免同质化疲劳     | 用户调研   |
| 长尾商品得分      | 长尾商品曝光率  | 公平性算法生效 indicator  | 基尼系数监控 |

## 4. 调优原则与最佳实践

### 4.1 分层调优策略

#### 4.1.1 第一层：结果正确性——确保无漏匹配、无错匹配

**验收标准**：

- 导航型查询（品牌名、型号名）：目标商品 **100% 召回**
- 排除条件（"非卖品"、"下架"）：不符合条件商品 **100% 过滤**

**核心工具**：`_validate/query`、`_analyze`、简单 `match` 查询验证。

**典型问题**：分词器过度拆分、同义词缺失、字段映射错误。

#### 4.1.2 第二层：排序合理性——核心商品位于首屏

**验收标准**：

- 品牌官方店在品牌查询中进入 **Top 3**
- 完全匹配商品优先于部分匹配
- 价格、评分等辅助信息排序符合用户预期

**核心工具**：explain 驱动诊断、人工标注 NDCG、A/B 测试验证。

#### 4.1.3 第三层：体验最优化——个性化与业务目标融合

**技术方案**：`function_score` 乘法融合，确保可解释性：

```json
{
  "function_score": {
    "query": {"multi_match": {...}},
    "functions": [
      {"field_value_factor": {"field": "sales", "factor": 0.1, "modifier": "log1p"}},
      {"gauss": {"publish_date": {"origin": "now", "scale": "30d", "decay": 0.5}}}
    ],
    "score_mode": "multiply",
    "boost_mode": "multiply"
  }
}
```

**关键约束**：个性化提升幅度控制在基础相关分的 **±20%** 范围内，确保相关性始终是主导因素。

### 4.2 可解释性优先原则

#### 4.2.1 避免黑箱规则：禁用无法 explain 的自定义脚本

**高风险反例**：

```json
"script_score": {
  "script": {
    "source": "Math.pow(doc['popularity'].value, 0.7) * (1 + Math.sin(doc['timestamp'].value / 86400000))"
  }
}
```

非线性变换、随机扰动、外部 API 调用——explain 无法拆解，维护者无法回答"为什么排第 3"。

**替代方案**：`function_score` 标准函数组合，每个因子独立配置、独立解释。

#### 4.2.2 保留调优路径：所有权重参数需文档化、可回滚

| 要素        | 具体要求                       | 工具支持             |
|:--------- |:-------------------------- |:---------------- |
| **参数版本化** | 所有 boost、weight 值纳入 Git 管理 | 配置文件 + CI/CD 流水线 |
| **变更审批**  | 核心参数调整需双人 review           | 代码审查机制           |
| **灰度发布**  | 新参数先 5% 流量验证，再全量           | 流量分割 + 实时监控      |
| **快速回滚**  | 参数异常时 5 分钟内恢复上一版本          | 配置中心热更新          |
| **效果记录**  | 每次变更前后的 NDCG、CTR 对比        | 实验平台自动报表         |

#### 4.2.3 乘法提升优于加法：function_score 的乘法模式更易解释

| 模式                | 公式                                   | 可解释性                         | 适用场景   |
|:----------------- |:------------------------------------ |:---------------------------- |:------ |
| **乘法** `multiply` | `score = BM25 × factor_A × factor_B` | 各因子贡献以 **比例** 体现，与 BM25 尺度无关 | 业务因子融合 |
| **加法** `sum`      | `score = BM25 + boost_A + boost_B`   | 绝对贡献，受 BM25 尺度影响大            | 独立维度叠加 |

乘法模式的核心优势：无论 BM25 基础分是 0.1 还是 100，**10% 的提升始终产生 1.1 倍乘数**，效果稳定可预测。

### 4.3 常见误区规避

#### 4.3.1 误区一：`_score` 越高越好——忽视业务指标验证

**灾难案例**：提升 `k1` 使整体 `_score` 提高 15%，但 CTR 下降 3%——高 `k_1` 放大了关键词堆砌商品的 TF 优势，损害结果质量。

**纠正方法**：建立"技术指标 + 业务指标"双轨监控，任何调优必须两者共同验证。

#### 4.3.2 误区二：BM25 参数随便调——未基于数据特征分析

**科学流程**：

1. 采样代表性文档，统计字段长度分布、词频分布
2. 建立参数-效果仿真模型
3. 小流量实验验证
4. 交叉验证防止过拟合

**常见错误**：盲目采用"业界最佳实践"（`k1=1.2, b=0.75`）而忽视自身数据特征。

#### 4.3.3 误区三：不用 explain 猜排序——依赖经验而非数据驱动

**经验的价值**：快速定位可能的问题方向。

**经验的局限**：难以处理多因子交互效应、容易过度拟合个人熟悉的查询模式。

**数据驱动的建立**：强制要求"无 explain 不调优"，建立 explain 案例库和诊断 SOP。

### 4.4 生产环境 checklist

#### 4.4.1 查询前：使用 `_analyze` 验证分词效果

```bash
POST /products/_analyze
{
  "field": "title",
  "text": "索尼WH-1000XM5无线降噪耳机"
```

**验证清单**：

- [ ] 是否存在过度切分（如"无线"被拆为"无"+"线"）？

#### 4.4.2 查询中：对异常结果立即执行 explain 诊断

**异常信号**：

- 用户投诉特定查询结果不合理
- 监控显示某查询 CTR/转化率偏离历史基线
- 人工质检发现首屏质量下降

## 5. 实战代码示例

### 5.1 基础 explain 查询（curl）

```bash
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{
    "explain": true,
    "query":{
      "multi_match":{
        "query":"轻薄 笔记本",
        "fields":["name^3","category"]
      }
    }
  }'
```

**响应解读要点**：

- `name^3` 的 boost 是否在 `weight(name:轻薄 in ...)` 中体现为乘法因子？
- `category` 字段的 IDF 值是否与预期一致（通常低于 name 字段）？
- 各字段贡献占比是否符合设计意图（name 应占 60%+）？

### 5.2 Java SDK 实现

```java
var resp = client.search(s -> s.index("products_v1")
    .explain(true)
    .query(q -> q.multiMatch(mm -> mm.query("轻薄 笔记本").fields("name^3","category"))),
    Map.class);

// 提取并打印完整 explain 树
for (var hit : resp.hits().hits()) {
    System.out.printf("Doc: %s, Score: %.3f%n", hit.id(), hit.score());
    printExplanation(hit.explanation(), 0);
}

void printExplanation(Explanation exp, int depth) {
    String indent = "  ".repeat(depth);
    System.out.printf("%s[%.3f] %s%n", indent, exp.value(), exp.description());
    for (var detail : exp.details()) {
        printExplanation(detail, depth + 1);
    }
}
```

### 5.3 进阶：function_score 融合业务指标

#### 5.3.1 销量对数衰减：`modifier: LOG1P` 避免高销量垄断

```json
{
  "field_value_factor": {
    "field": "sales_count",
    "factor": 0.1,
    "modifier": "log1p",
    "missing": 1
  }
}
```

| 销量         | 计算过程                                     | 乘数因子     |
|:---------- |:---------------------------------------- |:-------- |
| 0（missing） | log1p(1) = 0.693 → 取 missing=1           | **1.0**  |
| 100        | 1 + 0.1 × log1p(100) = 1 + 0.1 × 4.615   | **1.46** |
| 10,000     | 1 + 0.1 × log1p(10000) = 1 + 0.1 × 9.210 | **1.92** |

1000 倍销量差距压缩为 **1.3 倍乘数差距**，有效防止销量垄断。

#### 5.3.2 多因子加权：相关性 × 销量因子 × 新品因子

```json
{
  "function_score": {
    "query": {"multi_match": {"query": "耳机", "fields": ["name^3", "brand^2"]}},
    "functions": [
      {"field_value_factor": {"field": "sales", "factor": 0.1, "modifier": "log1p"}},
      {"gauss": {"publish_date": {"origin": "now", "scale": "30d", "decay": 0.5}}}
    ],
    "score_mode": "sum",
    "boost_mode": "multiply",
    "max_boost": 3
  }
}
```

**结构解析**：

- `score_mode: "sum"`：新品因子（1.0-1.5）+ 销量因子（1.0-1.3）= 综合业务因子（1.0-2.0）
- `boost_mode: "multiply"`：BM25 × 业务因子，保持相关性主导
- `max_boost: 3`：防止极端情况下的得分爆炸

#### 5.3.3 分组个性化：cohort 标签的 term 过滤 + 乘法提升

```json
{
  "function_score": {
    "query": {"match": {"name": "口红"}},
    "functions": [
      {"filter": {"term": {"cohorts": "female"}}, "weight": 0.1},
      {"filter": {"term": {"cohorts": "beauty"}}, "weight": 0.1},
      {"filter": {"term": {"cohorts": "luxury"}}, "weight": 0.2}
    ],
    "score_mode": "sum",
    "boost_mode": "multiply"
  }
}
```

对于用户标签 `["female", "beauty", "budget"]`：

- 匹配 2 个标签 → 因子 = 1 + 0.1 + 0.1 = **1.2**（20% 提升）
- 奢侈品偏好用户 `["female", "beauty", "luxury"]` → 因子 = **1.4**（40% 提升）

Explain 输出清晰展示每个 cohort 标签的匹配状态和独立贡献，完全透明可调试。

### 6.3 match vs match_phrase 对比

#### 6.3.1 查询构造：同一关键词的两种查询形式

**Query A（match）**：

```json
{
  "multi_match": {
    "query": "无线降噪耳机",
    "fields": ["name^3", "feature^2"],
    "type": "best_fields"
  }
}
```

**Query B（match_phrase 融合）**：

```json
{
  "bool": {
    "should": [
      {"match_phrase": {"name": {"query": "无线降噪耳机", "boost": 5, "slop": 1}}},
      {"multi_match": {"query": "无线降噪耳机", "fields": ["name^2","feature"], "boost": 1}}
    ]
  }
}
```

#### 6.3.2 结果对比：召回数量、排序差异、explain 得分拆解

| 对比维度 | match (A) | match_phrase 融合 (B) | 差异分析 |
|:---|:---|:---|
| 总召回量 | 1,247 | 1,089 | phrase 要求顺序匹配，召回下降 12.7% |
| 完全匹配商品 Top 3 占比 | 33% | **67%** | phrase boost 显著提升精确匹配 |
| 平均查询耗时 | 45ms | 52ms | phrase 查询增加约 15% 开销 |
| 用户满意度评分 | 3.8/5 | **4.3/5** | 精确匹配提升 perceived quality |

#### 6.3.3 场景选择：何时用 match 保证召回，何时用 match_phrase 强调精确

| 场景特征             | 推荐查询类型                | 理由                   |
|:---------------- |:--------------------- |:-------------------- |
| 长尾关键词、探索性搜索      | **match**             | 保证召回广度，避免漏掉相关变体      |
| 品牌+型号精确查询        | **match_phrase**      | 用户意图明确，顺序匹配提升精准度     |
| 属性组合筛选（"红色 连衣裙"） | **match + phrase 融合** | 颜色等属性顺序不敏感，核心名词短语需精确 |
| 语音/手写输入（含噪声）     | **match + 高 slop**    | 容错性强，允许词序偏差和额外词汇     |

## 7. 总结

### 7.1 核心要点

#### 7.1.1 排序质量 = BM25 算法基础 + 业务规则加权

BM25 提供了相关性计算的数学基础，但其默认参数和纯文本特征无法满足商业搜索的复杂需求。高质量的排序系统必须在 BM25 之上，通过**字段权重、短语匹配、业务因子**等机制注入领域知识。关键认知：**BM25 不是终点，而是起点**——它确保了"相关"与"不相关"的基本区分，而"更相关"的精细排序需要业务规则的补充。

#### 7.1.2 可解释性是搜索系统长期维护的质量基石

搜索系统的复杂性随时间指数增长：新商品类型、新查询模式、新业务目标持续涌现。唯有建立在**可解释性**基础上的系统，才能在这种演化中保持可控。Explain API 不仅是调试工具，更是**知识沉淀的载体**——每次诊断记录的根因与解决方案，构成组织的搜索优化知识库，使经验转化为可复用的工程实践。

### 7.2 持续优化方向

#### 7.2.1 建立 explain 驱动的调优流程

将 explain 分析纳入标准开发流程：feature 上线前的 explain 验证、线上异常的 explain 诊断、季度性的 explain 审计。目标是使 **80% 以上的排序问题可在 30 分钟内定位根因**。

#### 7.2.2 结合 A/B 测试验证调优效果

技术参数的优化必须经过用户行为的检验。建立分层实验平台：参数级实验（单个 boost 调整）、查询级实验（新查询结构）、流量级实验（全系统改版）。核心原则：**任何未通过 A/B 测试验证的调优，不得全量上线**。

#### 7.2.3 探索向量搜索与 BM25 的混合评分可解释性

随着语义搜索的普及，稠密向量（dense vector）与 BM25 的混合评分成为新挑战。当前方向包括：**学习稀疏检索**（learning sparse retrieval）通过可解释的 token 扩展保持透明度，以及 **RRF（Reciprocal Rank Fusion）**等无需分数校准的融合方法。下一代可解释性工具需同时覆盖词汇匹配与语义匹配的联合评分机制。
