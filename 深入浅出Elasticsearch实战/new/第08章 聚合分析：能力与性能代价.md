# 第08章 聚合分析：能力与性能代价

## 1. 聚合的核心能力与定位

### 1.1 从搜索引擎到分析引擎的演进

Elasticsearch 的架构演进体现了从纯文本搜索引擎向通用实时分析引擎的范式转变。这一转型的技术根基在于其**双重索引结构**：**倒排索引（Inverted Index）**支撑高效的全文检索能力，而 **Doc Values 列式存储**则支撑大规模聚合分析 。倒排索引以词项为中心组织文档列表，适用于快速定位包含特定关键词的文档；Doc Values 则以文档为中心、按列存储字段值，使得聚合操作可以顺序读取、高效缓存，避免了运行时解析原始 JSON 的开销 。

这种双轨设计让 ES 能够同时服务于两类截然不同的工作负载：搜索场景追求低延迟的文档召回，分析场景追求高吞吐的统计计算。在实际生产环境中，许多团队将 ES 直接用作"实时分析引擎"，用于日志监控、业务指标看板、用户行为分析等场景——数据写入后秒级即可查询，无需等待传统数仓的 T+1 延迟 。然而，这种能力并非无代价：聚合操作涉及全量数据扫描、中间状态维护和跨节点结果归并，其资源消耗与数据规模、查询复杂度呈**非线性增长关系**，尤其在缺乏优化意识时，极易引发集群性能劣化甚至服务不可用 。

实时分析场景的典型应用可归纳为三大类：

| 场景类型       | 核心需求                    | 典型聚合组合                                 | 性能敏感点        |
|:---------- |:----------------------- |:-------------------------------------- |:------------ |
| **时序指标分析** | 服务器监控、APM 性能追踪、IoT 数据采集 | `date_histogram` + `avg`/`percentiles` | 时间粒度与窗口长度的权衡 |
| **业务统计报表** | 电商 GMV 拆解、转化率漏斗、留存曲线    | `terms` 嵌套 + `sum`/`cardinality`       | 高基数字段的分桶策略   |
| **用户画像计算** | 标签分布、行为路径、人群圈选          | `composite` 多维度组合                      | 流式分页与内存控制    |

### 1.2 聚合解决的统计类问题

聚合框架的设计目标是回答"统计类问题"，即对数据集进行分组、过滤、计算后得到可解释的汇总信息。从业务维度划分，可分为三大类别：

**时间维度聚合**是时序数据分析的基础形态。日活跃用户（DAU）、周活跃用户（WAU）、用户留存率等指标，本质上都是对事件时间戳进行分桶后的计数或去重统计。`date_histogram` 聚合支持**日历间隔（`calendar_interval`）**——如 1d、1w、1M，考虑时区和闰年规则，适合业务报表；以及**固定间隔（`fixed_interval`）**——如 1h、5m，保证严格的等长区间，适合技术指标 。时间聚合的关键挑战在于**桶数控制**：查询一年数据按分钟聚合将产生 525,600 个桶，若再嵌套其他聚合，总桶数可能指数级膨胀至千万级，远超默认的 `max_buckets` 限制（65536）。

**分类维度聚合**用于理解离散属性的分布特征。品类销量分布、地域订单占比、渠道转化效果等场景，主要依赖 `terms` 聚合对字符串字段进行分桶计数。`terms` 聚合的实现深度依赖 **Global Ordinals 机制**——将字符串值映射为紧凑的整型序号，以加速比较和减少内存占用 。然而，这一优化在**高基数字段**（如用户 ID、订单号、UUID）上会适得其反：Global Ordinals 的构建时间与字段唯一值数量呈**非线性增长**，且每次段变动后需重建，导致查询延迟剧烈波动 。

**数值维度聚合**则关注连续变量的统计特征。客单价分位（P50、P90、P99）、API 响应时间分布、库存周转率方差等场景，需要 `percentiles`、`avg`、`sum` 等 Metric 聚合的配合。`percentiles` 聚合基于 **TDigest 算法**提供近似分位计算，内存占用与数据量无关，仅取决于压缩参数；`cardinality` 聚合则采用 **HyperLogLog++ 算法**实现近似去重，以可配置的精度换取固定内存 。这些近似算法体现了 ES 的设计哲学：**在速度与精度之间优先保障前者**，用户可通过参数调整在二者间取得平衡。

## 2. 聚合类型与实现机制

### 2.1 Bucket 聚合：数据分桶

Bucket 聚合是 ES 分析能力的核心抽象，其功能类似于 SQL 中的 `GROUP BY`，将文档集合划分为多个子集（桶），后续可在每个桶内独立计算指标。

**`terms` 聚合**是最常用的字符串分桶方式，其实现机制与性能特征需要深入理解：

| 执行模式                  | 触发条件                  | 核心机制                | 适用场景        | 主要风险                      |
|:--------------------- |:--------------------- |:------------------- |:----------- |:------------------------- |
| `global_ordinals`（默认） | 首次查询或段变动后             | 预构建字符串→整型映射表，后续查询复用 | 重复查询、中低基数字段 | 高基数场景构建耗时数秒至数分钟           |
| `map`                 | 显式指定 `execution_hint` | 直接遍历原始值，哈希分桶，无预构建   | 强过滤条件、结果集极小 | 单次查询内存与结果集大小成正比，大结果集易 OOM |

Global Ordinals 的构建成本是 `terms` 聚合的核心性能变量。对于包含 **5000 万唯一值**的字段，首次查询的构建耗时可达 **15 秒以上**，堆内存增长 **10GB**，且这一过程**不受断路器实时监控**，可能在触发保护前已导致节点 OOM 。`shard_size` 参数控制每个分片返回的候选桶数，默认公式 `size × 1.5 + 10` 在高基数场景下往往不足，需手动提升至 `size` 的 5-10 倍以改善精度，但代价是线性增长的内存和网络开销 。

**`date_histogram` 聚合**针对时间字段优化，其实现直接操作数值型时间戳，**无需 Global Ordinals**，因此即使在高基数时间字段上也能保持高效。关键配置参数包括：

- `calendar_interval` vs `fixed_interval`：前者处理时区、夏令时、月份天数差异，计算复杂但结果符合业务直觉；后者保证严格等长，性能更优
- `offset`：调整桶起始边界，如将自然日对齐调整为业务日（从早上8点开始）
- `min_doc_count`：过滤空桶，减少响应体积和序列化开销

**`range` 聚合**支持自定义数值或日期区间，允许为非连续区间指定业务语义化的 `key`（如 "0-100:低价", "100-500:中价"）。与 `histogram` 的自动等分不同，`range` 适合非均匀分布场景，其实现基于 Doc Values 的顺序扫描和二分查找，区间数量建议控制在 10-20 个以内。

**`composite` 聚合**是 ES 5.0 引入的**流式分桶机制**，专为高基数场景的分页遍历设计 。其核心优势在于：

| 特性    | `terms` 聚合                       | `composite` 聚合     |
|:----- |:-------------------------------- |:------------------ |
| 结果排序  | 按 `doc_count` 或子聚合值排序            | 仅按组合键字典序           |
| 分页机制  | 无原生分页，`from` 不可靠                 | `after_key` 精确游标分页 |
| 内存保证  | 与 `size` 和 `shard_size` 相关，无严格上限 | 与 `size` 成正比，严格有界  |
| 结果完整性 | 近似 Top N，可能遗漏低频桶                 | 精确遍历，无遗漏           |
| 适用场景  | 快速概览、排行榜                         | 全量导出、对账、深度分析       |

`composite` 聚合通过 `after` 参数实现状态传递，每次查询仅处理当前页面对应的数据范围，**内存占用与总桶数解耦**，使其成为处理千万级唯一值组合的唯一可靠方案 。

### 2.2 Metric 聚合：指标计算

Metric 聚合在 Bucket 聚合建立的子集上计算统计指标，其算法选择直接影响精度、性能和资源占用的三角权衡。

| 聚合类型                    | 算法/实现         | 内存特征                | 精度特征           | 典型应用场景       |
|:----------------------- |:------------- |:------------------- |:-------------- |:------------ |
| `sum`、`avg`、`max`、`min` | Kahan 求和、直接比较 | O(1)，与文档数无关         | 精确（浮点误差可控）     | 基础统计、财务汇总    |
| `value_count`           | 计数器递增         | O(1)                | 精确             | 文档计数、非空字段统计  |
| `cardinality`           | HyperLogLog++ | O(1)，固定 ~12KB（默认阈值） | 近似，误差 1-6%（可调） | UV 统计、唯一设备数  |
| `percentiles`           | TDigest       | O(1)，与压缩参数相关        | 近似，尾部精度较低      | P99 延迟、客单价分位 |

**`cardinality` 聚合的精度-内存权衡**需要特别关注。`precision_threshold` 参数控制算法行为：当实际唯一值数量**低于阈值**时，自动切换为线性计数，返回**精确结果**；超过阈值后，采用 HLL++ 概率估计，标准误差约为 `1.04/√m`。默认阈值 3000 对应内存约 24KB，最大有效阈值 40000 对应约 320KB，误差可降至 0.5% 以下 。关键认知：**`cardinality` 从设计上就是近似算法**，追求精确结果应改用 `terms` 聚合配合 `size: 0`（全量返回），但这在高基数字段上不可行。

### 2.3 Pipeline 聚合：结果二次加工

Pipeline 聚合对其他聚合的结果进行后处理，实现复杂的派生计算和趋势分析。其核心特征是**仅在协调节点执行**，不涉及数据节点的文档扫描，因此不增加 I/O 开销，但可能因桶数量过多导致协调节点内存压力。

**桶级运算聚合**：

- `bucket_script`：跨桶算术运算，如 `转化率 = 订单数 / 访问数 × 100`
- `bucket_selector`：条件过滤，如仅保留 `转化率 > 5%` 的渠道
- `bucket_sort`：按子聚合值对桶重新排序

**移动窗口聚合**：

- `moving_avg`：滑动平均，支持简单、线性加权、指数平滑等模型
- `derivative`：一阶差分，计算相邻桶的变化率
- `cumulative_sum`：累积求和

Pipeline 聚合的执行时机在父聚合完成之后，因此其输入是已归并的全局桶结果。深层嵌套时（如 5 层以上），中间结果需在协调节点内存中保留，可能成为性能瓶颈，建议控制前置桶数量在 **10 万以内** 。

## 3. 聚合执行原理与资源模型

### 3.1 Scatter-Gather 执行流程

Elasticsearch 聚合查询遵循经典的 **Scatter-Gather 分布式计算模式**，分为三个阶段协同完成：

**协调节点阶段（Scatter）**接收客户端请求，解析查询 DSL，确定目标索引的分片分布，将子请求并行路由至各数据节点。关键决策包括：副本分片选择策略（轮询、自适应、本地优先）、是否启用跨集群搜索、以及查询缓存的利用。协调节点本身不处理文档数据，但其网络线程和内存缓冲区可能成为瓶颈——当聚合结果集庞大时，序列化和反序列化的开销显著 。

**数据节点阶段（Map）**在分片级别执行实际的聚合计算。每个分片独立遍历本地匹配的文档，根据聚合类型维护中间状态：

- 对于 `terms` 聚合，构建分片级的术语频率表，涉及 Global Ordinals 的构建或加载
- 对于 `date_histogram`，按时间槽累加计数，直接基于数值范围划分
- 对于嵌套聚合，支持深度优先（DFS）和广度优先（BFS）两种执行模式，内存特征各异

分片级结果序列化为压缩的二进制格式返回协调节点，序列化优化包括：term 的序数编码替代原始字符串、指标值的紧凑表示等。

**全局归并阶段（Reduce）**在协调节点收集各分片结果，执行最终的桶合并、排序和子聚合递归。这是**精度损失的主要发生环节**：`terms` 聚合的各分片独立计算 Top N，协调节点归并时可能遗漏跨分片的高频项，误差大小由 `doc_count_error_upper_bound` 字段量化 。Reduce 阶段的内存占用与 `shard_size` 和分片数量成正比，是 OOM 的高发区域。

### 3.2 核心数据结构开销

聚合性能深受三种底层数据结构的影响，理解其资源特征是优化的基础：

| 数据结构                | 存储位置         | 构建时机     | 主要开销                      | 优化杠杆                         |
|:------------------- |:------------ |:-------- |:------------------------- |:---------------------------- |
| **Doc Values**      | 磁盘 + 操作系统页缓存 | 索引时      | 磁盘空间（原始大小的 10-30%）、解压 CPU | 禁用无需聚合字段的 `doc_values`       |
| **Global Ordinals** | JVM 堆（按段维护）  | 首次查询或刷新时 | 构建时间 O(N log N)、内存与基数正比   | `eager_global_ordinals`、强制合并 |
| **Fielddata**       | JVM 堆（已废弃）   | 查询时动态加载  | 无界增长、OOM 风险               | 彻底禁用，改用 `keyword` 子字段        |

**Doc Values** 的列式布局使得聚合查询能够利用操作系统的页缓存机制，热数据集的访问接近内存速度。但需注意，过度增大 JVM 堆（如超过 32GB）会挤压页缓存空间，反而降低性能——最佳实践是 **JVM 堆设为物理内存的 50% 且不超过 30GB** 。

**Global Ordinals** 的段级隔离设计加剧了碎片问题。每个 Lucene 段维护独立的 ordinal 空间，分片级的 Global Ordinals 需要建立"段 ordinal → 分片 ordinal"的二次映射。未合并的索引可能包含数十至上百个段，每个段的映射表叠加后内存占用惊人，且**未被纳入标准监控指标**，用户只能通过 JVM 堆的"无法解释"增长间接推断 。

### 3.3 资源消耗的三重维度

聚合查询的资源模型可从三个正交维度量化分析：

**内存消耗**的核心公式为：**总内存 ≈ 桶数量 × 单桶大小 × 字段基数系数**。桶数量由聚合层次结构决定，嵌套聚合产生**笛卡尔积效应**——外层 1000 桶 × 内层 100 桶 = 10 万桶，三层嵌套即可达千万级。单桶大小取决于子聚合复杂度，纯计数桶仅需 8 字节，包含 `percentiles` 的复合桶可能达数 KB。字段基数影响 Global Ordinals 映射表大小，高基数字段的段级映射表可能占用数百 MB。

**CPU 消耗**与遍历文档数、聚合算法复杂度、分片数成正比。简单 Metric 聚合（`sum`、`avg`）的每文档成本极低；`cardinality` 的 HLL++ 哈希计算、`percentiles` 的 TDigest 聚类更新，以及嵌套 `terms` 的哈希表操作，是主要热点。分片并行度的权衡：更多分片提升 Map 阶段并行效率，但增加 Reduce 阶段的归并成本——建议单个查询涉及的分片数控制在 **1000 以内** 。

**网络消耗**来自分片结果的序列化与传输。`terms` 聚合的 `shard_size` 参数直接决定传输数据量：设置过小导致精度损失，过大则浪费带宽。`composite` 聚合通过流式分页将单次传输控制在 O(页大小)，是大结果集场景的网络优化方案。

## 4. 高基数字段：性能瓶颈深度解析

### 4.1 Global Ordinals 构建机制

Global Ordinals 是理解高基数聚合性能问题的关键概念，其工作机制决定了优化策略的选择空间。

**构建时机的双重模式**构成首要权衡：

| 模式           | 配置方式                            | 构建触发时机  | 查询延迟特征            | 适用场景           |
|:------------ |:------------------------------- |:------- |:----------------- |:-------------- |
| **延迟构建（默认）** | 无额外配置                           | 首次查询需要时 | 首次极高（数秒至数分钟），后续极低 | 查询模式不可预测、写入频繁  |
| **预构建**      | `"eager_global_ordinals": true` | 每次索引刷新时 | 稳定低延迟，无冷查询惩罚      | 查询频繁、写入较少、延迟敏感 |

延迟构建模式将成本推迟到首次查询，导致明显的**"冷查询"问题**——同一查询的首次执行可能比后续慢 1000 倍以上 。预构建模式将查询延迟转化为写入延迟，但需注意：**每次 refresh 都触发重建**，若 `refresh_interval` 为 1 秒，则每秒都需遍历全部分段，严重拖慢写入吞吐。优化组合是 **预构建 + 延长刷新间隔**（如 30 秒），摊平构建成本 。

**基数敏感的非线性增长**是性能恶化的核心机制。Global Ordinals 构建需要遍历段的 term 字典，执行排序和去重，时间复杂度接近 **O(N log N)**。实测数据显示：100 万唯一值约 200ms，1000 万唯一值增至 3-5 秒，5000 万唯一值可能超过 30 秒 。更严峻的是，这一构建过程**阻塞查询线程**，期间节点近乎无响应。

**内存占用的段级叠加**使得问题复杂化。每个段的 Global Ordinals 映射表独立维护，未强制合并的索引可能包含大量小段。一个包含 100 个段、每段 50 万唯一值的索引，理论映射表内存可达 **100 × 50万 × 30字节 ≈ 1.5GB**，且这部分内存**不受 Fielddata 断路器监控**，成为"隐形"的内存泄漏源 。

### 4.2 优化策略矩阵

针对高基数字段的 `terms` 聚合，需根据场景特征选择差异化策略：

| 场景特征             | 推荐策略               | 核心配置                                                     | 收益           | 代价与风险                    |
|:---------------- |:------------------ |:-------------------------------------------------------- |:------------ |:------------------------ |
| 查询频繁、基数极高、写入持续   | 预构建 + 延长刷新间隔       | `eager_global_ordinals: true`, `refresh_interval: "30s"` | 消除冷查询，延迟稳定   | 写入延迟增加 10-50%，索引膨胀 5-15% |
| 查询偶发、结果集小、过滤条件强  | 绕过 Global Ordinals | `"execution_hint": "map"`                                | 避免构建开销，冷查询快速 | 单次查询内存升高，大结果集易熔断         |
| 只读历史索引、查询性能极致    | 强制合并为单段            | `POST /index/_forcemerge?max_num_segments=1`             | 彻底消除重建，查询最优  | 操作重量级，期间只读，存储临时翻倍        |
| 超高基数（>1亿）、仅需近似统计 | 哈希降维               | `murmur3` 子字段，对哈希值聚合                                     | 基数降至可控范围     | 结果需反查，存在哈希冲突             |
| 超大规模、实时性可妥协      | 预聚合架构              | Transform/Rollup 物化结果                                    | 查询毫秒级，与基数解耦  | 架构复杂，结果延迟分钟级             |

`execution_hint: "map"` 是关键的应急手段，但其适用条件严格：**仅当匹配文档数远小于总文档数时性能优于 `global_ordinals`**。该模式直接遍历匹配文档的原始 term 值，在内存中构建临时的哈希映射——若过滤条件失效或基数极高，临时映射可能撑爆内存，触发 `CircuitBreakingException`  。

### 4.3 典型反模式

生产环境中反复出现的高基数反模式，其根源往往是对 ES 聚合机制的理解偏差：

**反模式一：直接对 UUID 字段执行 `terms` 聚合**

`user_id`、`order_id`、`device_fingerprint` 等字段的唯一值数量与文档数接近 1:1，`terms` 聚合将产生与文档数相当的桶数，完全丧失聚合的压缩意义。某案例显示，对 4000 万 ASIN 的聚合查询响应时间超过 30 秒，内存占用 46GB，触发父级断路器，集群陷入 5-10 分钟不可用  。**此类需求应改用 `cardinality` 估算总数，或通过业务维度降维后再聚合**。

**反模式二：嵌套多层高基数 `terms` 聚合**

第一层按 `user_id`（1000万基数）、第二层按 `product_id`（100万基数）聚合，理论桶数达万亿级，即使实际匹配文档有限，协调节点的桶合并开销也可能导致超时。**解决方案**：将高基数字段移至 `composite` 聚合的最后一个维度，利用其流式特性控制内存；或预构建实体中心索引，将关系物化为独立文档  。

**反模式三：忽视 `shard_size` 的精度-性能权衡**

默认 `shard_size = size × 1.5 + 10` 在均匀分布场景有效，但数据倾斜严重时，全局 Top 10 的术语可能在某分片排名第 25 而被截断。增大 `shard_size` 改善精度，但需监控网络传输和 Reduce 内存的增长，建议通过 `show_term_doc_count_error` 量化误差后逐步调优  。

## 5. 大时间窗口与大数据量场景

### 5.1 时间范围聚合的陷阱

`date_histogram` 聚合虽无 Global Ordinals 问题，但时间范围与粒度的组合仍可能产生性能灾难：

**桶数爆炸的量化公式**：`桶数 = 时间跨度 / 间隔精度`。常见陷阱包括：

| 查询场景 | 时间跨度 | 间隔粒度 | 桶数         | 风险等级   |
|:---- |:---- |:---- |:---------- |:------ |
| 实时大盘 | 1 小时 | 1 分钟 | 60         | 低      |
| 日报趋势 | 30 天 | 1 小时 | 720        | 中      |
| 年度分析 | 1 年  | 1 分钟 | 525,600    | **高**  |
| 多年对比 | 3 年  | 1 秒  | 94,608,000 | **极高** |

嵌套聚合会进一步放大桶数：上述年度分析若再嵌套 1000 个类目，理论桶数达 **5.26 亿**，远超任何合理配置的内存容量。

**时区处理的隐藏成本**：`calendar_interval` 的夏令时、闰秒处理增加 10-20% CPU 开销，且跨时区查询的边界对齐可能导致"重复小时"或"缺失小时"。建议：时区一致性要求高的场景，**摄入时预计算本地时间字段**，聚合时直接使用。

**冷数据扫描的 I/O 放大**：热温冷架构下，大时间窗口查询可能触及数月甚至数年的冷数据，I/O 延迟从毫秒级升至秒级。冻结索引（frozen index）机制可缓解——冷索引卸载至持久化存储，查询时按需加载，但首次查询延迟极高（数十秒）。

### 5.2 流式处理方案

当聚合结果集超出单机内存或单次响应限制时，ES 提供多种流式处理机制：

| 机制                     | 核心原理             | 内存特征        | 适用场景        | 关键限制          |
|:---------------------- |:---------------- |:----------- |:----------- |:------------- |
| **`composite` 分页**     | `after_key` 游标导航 | O(页大小)，严格有界 | 全量导出、精确遍历   | 仅按键排序，无 Top N |
| **异步搜索**               | 后台任务 + 轮询结果      | 结果临时存储      | 超长时间查询、报表生成 | 额外存储开销，延迟分钟级  |
| **PIT + search_after** | 快照一致性 + 深度分页     | 上下文持有段引用    | 流式导出、外部处理   | 长期持有阻止段合并     |

`composite` 聚合是官方推荐的大数据量遍历方案。与 `terms` 的近似 Top N 不同，它保证**无遗漏、无重复**，但代价是**无法按子聚合值排序**——结果严格按组合键字典序返回，"最热门"的桶可能分散在任意位置  。

### 5.3 预聚合架构

对于重复执行的固定模式聚合，预计算是根治性能问题的方案：

| 方案                         | 执行模式          | 数据延迟    | 查询灵活性     | 适用场景           |
|:-------------------------- |:------------- |:------- |:--------- |:-------------- |
| **Rollup**（7.x，8.x 逐步淘汰）   | 定时批处理降采样      | 小时级     | 低（固定维度）   | 历史趋势报表         |
| **Transform**（7.3+ 推荐）     | 持续 CDC 物化     | 秒级-分钟级  | 中（预定义聚合）  | 实时仪表盘、指标看板     |
| **外部数仓**（ClickHouse/Doris） | 批量导入 + MPP 查询 | 分钟级-小时级 | 高（任意 SQL） | 复杂 OLAP、PB 级数据 |

Transform 的核心优势在于**近实时性**和**查询透明性**——配置 `sync` 模式后，目标索引与源数据保持秒级同步，查询时直接访问物化结果，延迟从秒级降至毫秒级。代价是存储冗余（原始数据 + 聚合结果）和架构复杂度（需监控转换延迟、管理目标索引生命周期）。

## 6. 工程优化实践

### 6.1 查询侧优化

**前置过滤的杠杆效应**是聚合优化的首要原则。通过 `query` 子句的 `filter` 上下文将参与聚合的文档数从亿级降至百万级，性能提升可达 **两个数量级** 。关键技巧：

- 时间范围过滤使用 `range` 查询，利用 BKD 树的快速剪枝
- 等值过滤使用 `term` 而非 `match`，避免分词开销
- 多条件组合使用 `bool.filter` 嵌套，结果可缓存复用

**桶数控制的三参数联合调优**：

| 参数              | 作用     | 默认值               | 调优建议                   |
|:--------------- |:------ |:----------------- |:---------------------- |
| `size`          | 返回桶数上限 | 10                | 设为 UI 展示所需，如 20-50     |
| `shard_size`    | 分片候选桶数 | `size × 1.5 + 10` | 高基数场景提升至 `size × 5-10` |
| `min_doc_count` | 最小文档阈值 | 1                 | 长尾场景提升至 10-100，过滤噪声    |

**精度权衡的参数化配置**：

- `cardinality.precision_threshold`：默认 3000，提升至 10000 可将误差从 ~2% 降至 ~0.5%，内存增至 80KB
- `percentiles.compression`：默认 100，降至 50 减少内存但增大尾部误差

### 6.2 建模侧优化

**字段类型的精准选择**是基础规范：

| 场景     | 推荐类型                              | 关键配置                | 禁止事项                      |
|:------ |:--------------------------------- |:------------------- |:------------------------- |
| 聚合分桶   | `keyword`                         | 启用 `doc_values`（默认） | 对 `text` 字段启用 `fielddata` |
| 数值计算   | `integer`/`long`/`float`/`double` | 选择满足范围的最小类型         | 字符串存储数值后运行时转换             |
| 时间戳    | `date`                            | 统一格式，预计算本地时间        | 字符串比较导致的词典序错误             |
| 高基数 ID | `keyword` + `murmur3` 子字段         | 对哈希值聚合              | 直接对原始 ID `terms` 聚合       |

**多字段策略实现维度降级**：对 `user_id` 等字段，同时存储原始值和派生的低基数版本（如 `user_id_hash = hash(user_id) % 1000`），聚合时使用派生字段，需要精确值时反查原始字段。存储成本增加 10-20%，查询性能提升 10-100 倍  。

**索引分层与 ILM 策略**：

| 层级         | 数据年龄    | 硬件配置          | 聚合策略       | 典型设置                         |
|:---------- |:------- |:------------- |:---------- |:---------------------------- |
| 热（Hot）     | 0-7 天   | SSD，高 CPU，大内存 | 实时聚合，激进缓存  | `refresh_interval: 1s`，完整副本  |
| 温（Warm）    | 7-30 天  | 平衡配置          | 延迟可接受，降低频率 | `refresh_interval: 30s`，缩减副本 |
| 冷（Cold）    | 30-90 天 | HDD，低成本       | 冻结索引，按需加载  | `frozen: true`，快照备份          |
| 删除（Delete） | > 90 天  | —             | 归档或删除      | ILM 自动执行                     |

### 6.3 运维侧保障

**断路器的多级防护**：

| 断路器类型                             | 默认限制                  | 监控重点             | 典型触发场景         |
|:--------------------------------- |:--------------------- |:---------------- |:-------------- |
| `indices.breaker.fielddata.limit` | 60% JVM 堆             | Fielddata 缓存异常增长 | 误对 `text` 字段聚合 |
| `indices.breaker.request.limit`   | 40% JVM 堆             | 单请求内存超限          | 大 `size` 或深层嵌套 |
| `indices.breaker.total.limit`     | 70% JVM 堆（或 95% 实际内存） | 总体内存压力           | 多查询并发叠加        |

**关键监控指标与告警阈值**：

| 指标                               | 来源                     | 告警条件     | 处置建议           |
|:-------------------------------- |:---------------------- |:-------- |:-------------- |
| `aggregation_time_in_millis`     | `_nodes/stats`         | P99 > 5s | 优化查询结构或扩容      |
| `heap_used_percent`              | `_nodes/stats/jvm`     | 持续 > 85% | 增大堆或优化 GC      |
| `fielddata.memory_size_in_bytes` | `_nodes/stats/indices` | 非零即异常    | 检查 `text` 字段聚合 |
| `breakers.*.tripped`             | `_nodes/stats/breaker` | 任意增长     | 分析触发查询，优化或限流   |

**Profile API 的瓶颈定位**：启用 `"profile": true`，关注各阶段耗时占比：

- `build_aggregation` 过高 → Global Ordinals 问题，考虑预构建或 `execution_hint: "map"`
- `collect` 过高 → 扫描文档数过多，加强前置过滤
- `reduce` 过高 → 分片结果过大，降低 `shard_size` 或优化协调节点

### 7.1 基础聚合（问题版本）

```json
{

  "aggs": {
    "by_user": {
      "terms": {
        "field": "user_id",
```

- 嵌套结构理论桶数 100 × 365 = 36,500，内存占用 500MB+

## 8. 边界意识与决策框架

### 8.1 聚合适用性评估

在采用 ES 聚合前，需系统评估三个维度：

| 维度       | 适合 ES 聚合            | 需考虑替代方案          |
|:-------- |:------------------- |:---------------- |
| **数据规模** | 百万级文档实时，千万级配合优化     | 亿级以上实时，十亿级预聚合或分流 |
| **延迟要求** | 秒级可接受，P99 < 5s      | 毫秒级硬性要求，需缓存或预计算  |
| **精度要求** | 近似估计可接受，Top N 有误差容忍 | 精确去重必需，财务级对账     |

**关键认知**：分布式环境下，即使"精确"聚合也存在分片级精度损失（`doc_count_error_upper_bound`），绝对精确需单分片或自定义路由 。

### 8.2 替代方案选型

| 场景特征               | ES 聚合能力                 | 推荐替代方案                    | 迁移触发条件                   |
|:------------------ |:----------------------- |:------------------------- |:------------------------ |
| 超大规模日志分析（PB 级）     | 受限：JVM 堆瓶颈、索引成本高        | ClickHouse、Apache Doris   | 单日索引 > 50TB，聚合 P99 > 10s |
| 复杂 OLAP 多维分析（任意下钻） | 受限：无预计算立方体              | Apache Druid、Apache Pinot | 维度组合 > 1000，需亚秒级响应       |
| 预计算指标查询（固定维度）      | 中等：Transform/Rollup 可满足 | Prometheus、InfluxDB       | 时间线 > 100 万，需降采样策略       |
| 交互式探索分析（ad-hoc）    | 良好：配合优化                 | 保持 ES，或评估 StarRocks       | 低-中                      |

混合架构（ES 负责搜索和轻量分析，ClickHouse/Druid 负责重度分析）是大型组织的常见演进路径。选型决策应基于 **POC 验证**——选取代表性查询，在同等硬件上对比延迟、吞吐、资源占用，避免"纸上选型"。

## 9. 本章知识检验

### 9.1 基础练习

**练习 1：构建带过滤条件的 `date_histogram` + `avg` 嵌套聚合**

目标：统计最近 30 天，各天的"已完成订单"平均客单价，排除测试用户（`is_test: true`）。

关键检查点：

- `bool.filter` 上下文的正确使用（`range` + `term` + `must_not`）
- `date_histogram` 使用 `fixed_interval: "1d"` 而非 `calendar_interval`
- `avg` 聚合的字段类型验证（数值型，非 `keyword`）

**练习 2：对比 `terms` 与 `composite` 的分页语义差异**

| 特性    | `terms` 聚合              | `composite` 聚合    |
|:----- |:----------------------- |:----------------- |
| 结果排序  | 按 `doc_count` 或子聚合值     | 仅按组合键字典序          |
| 分页机制  | 无原生分页，`from` 不可靠        | `after_key` 精确游标  |
| 内存特征  | 与 `shard_size` 相关，无严格上限 | 与 `size` 成正比，严格有界 |
| 结果完整性 | 近似 Top N，可能遗漏           | 精确遍历，无遗漏          |
| 适用场景  | 快速概览、排行榜                | 全量导出、对账、深度分析      |

### 9.2 深度分析

**分析 1：高基数字段 Global Ordinals 成本测算**

选定字段 `device_fingerprint`（`keyword` 类型，基数 5000 万，索引 200GB，5 分片）：

| 测量项  | 方法                                    | 预期结果                |
|:---- |:------------------------------------- |:------------------- |
| 构建耗时 | Profile API `build_aggregation` 阶段    | 5-15 秒              |
| 内存占用 | `_nodes/stats/jvm` 堆增长                | 2-4 GB              |
| 优化对比 | `eager_global_ordinals` 启用前后          | 查询延迟方差降低 90%+       |
| 替代方案 | `execution_hint: "map"` 1% vs 50% 匹配率 | 小结果集快 5-10 倍，大结果集更慢 |

**分析 2：预聚合方案设计**

场景：支持"任意 90 天内，按小时、品类、省份、渠道四维分析 GMV、订单量、退款率"，数据 10 亿订单/年。

Transform 配置要点：

- 分组：`date_histogram`（hour）+ `terms`（category, province, channel）
- 指标：`sum`（gmv）、`value_count`（orders）、`bucket_script`（refund_rate）
- 频率：`"1m"` 同步模式，目标索引 `order_summary_hourly`
- 查询路由：近 24h 走实时聚合，历史走 Transform 索引

### 9.3 故障排查

**场景：解读 `CircuitBreakingException`**

异常信息：

```json
{
  "error": {
    "type": "circuit_breaking_exception",
    "reason": "[parent] Data too large, data for [<aggregation>] would be [12.3gb/11.4gb]"
  }
}
```

排查步骤：

1. **识别断路器类型**：`parent` 表示总体内存压力，`request` 表示单请求超限
2. **定位触发查询**：`_tasks?detailed=true&actions=*search*` 获取运行中查询
3. **分析聚合结构**：检查 `size`、`shard_size`、嵌套深度、高基数字段使用
4. **临时缓解**：减小 `size`、增加 `min_doc_count`、添加严格过滤
5. **长期修复**：建模优化（降基数字段）、架构调整（Transform 预聚合）、扩容

**场景：Profile API 慢查询分析**

典型瓶颈模式：

- `build_aggregation` > 80% → Global Ordinals 问题，启用预构建或 `execution_hint: "map"`
- `collect` > 80% → 扫描文档数过多，加强前置过滤
- `reduce` > 30% → 分片结果过大，降低 `shard_size` 或优化协调节点
