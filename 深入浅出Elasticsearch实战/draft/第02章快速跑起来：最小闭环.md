

# 第02章 快速跑起来：最小闭环

# 背景

团队开始用 ES 时，最常见的问题不是"性能不够"，而是"大家本地验证步骤不一致"——有人装的 7.x，有人装的 8.x；有人带安全，有人裸跑；bulk 写完不知道成功了没有。因此需要一套标准化最小闭环：同样的命令、同样的预期结果、同样的排查方式。

## 本章目标

- 跑通 Elasticsearch 最小闭环：建索引 -> 写文档 -> 查文档 -> 更新 -> 删除 -> 清理。
- 理解 ES 的 RESTful API 设计哲学及其与 SQL 概念的映射关系。
- 用一套可复制脚本，帮助团队统一本地验证方式。
- 识别近实时（NRT）陷阱，建立"先正确、再优化"的工程习惯。

---

## 1. 什么是最小闭环

最小闭环就是用最少步骤确认系统可用：

1. 创建索引（带显式 mapping）
2. 写入样例数据（单条 + 批量）
3. 执行查询（全文检索 + 过滤 + 排序 + 聚合）
4. 单文档操作（获取 + 更新 + 删除）
5. 清理索引（保证测试可重复）

它的目标不是"高性能"，而是**确定性可运行**。每次新环境上线、每次版本升级，先做这一圈验证，确认基本链路没问题，再谈优化。

## 2. 环境准备

### 2.1 环境变量设置

后续所有命令都依赖以下变量，先统一设好：

**bash / macOS / Linux：**

```bash
export ES_URL="http://localhost:9200"
export ES_USER="elastic-admin"
export ES_PASS="elastic-password"
export IDX="products_v1"
```

**Windows PowerShell：**

```powershell
$env:ES_URL="http://localhost:9200"
$env:ES_USER="elastic-admin"
$env:ES_PASS="elastic-password"
$env:IDX="products_v1"
```

如果本地未开启安全认证，去掉命令中的 `-u "$ES_USER:$ES_PASS"` 即可。

### 2.2 Docker 单节点快速启动

将下面的内容保存为 `docker-compose.yml`，然后执行 `docker-compose up -d`：

```yaml
version: '3.8'
services:
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.17.0
    container_name: es-node01
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false
      - bootstrap.memory_lock=true
      - "ES_JAVA_OPTS=-Xms512m -Xmx512m"
    ulimits:
      memlock:
        soft: -1
        hard: -1
    ports:
      - 9200:9200
  kibana:
    image: docker.elastic.co/kibana/kibana:8.17.0
    container_name: kibana
    ports:
      - 5601:5601
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
```

启动后访问 `http://localhost:9200` 确认 ES 可用，访问 `http://localhost:5601` 进入 Kibana Dev Tools。

### 2.3 安全配置说明

ES 8.x 默认开启安全（TLS + 用户认证）。上面的 compose 文件通过 `xpack.security.enabled=false` 将其关闭，以降低初次上手的心智负担。

如果你的环境已开启安全，使用默认超级用户账号，或根据实际情况替换环境变量中的 `ES_USER` 和 `ES_PASS`。生产环境**务必保留安全配置**，启用 TLS 并使用强密码。

启动后可用以下命令验证集群是否就绪：

```bash
curl -sS -u "$ES_USER:$ES_PASS" "$ES_URL/_cluster/health?pretty"
```

预期 `status` 为 `green`（或 `yellow`，取决于是否有未分配的副本分片）。

## 3. ES 的 RESTful API 设计哲学

### 3.1 HTTP 方法与资源的映射

ES 的所有操作都通过 HTTP RESTful API 完成。HTTP 动词直接映射到资源操作语义：

| HTTP 方法   | 语义         | 典型用法                     |
| --------- | ---------- | ------------------------ |
| `PUT`     | 创建或替换      | 创建索引、写入/替换指定 ID 的文档      |
| `POST`    | 创建或执行操作    | 自动生成 ID 写入、搜索、批量操作、更新文档  |
| `GET`     | 读取         | 获取文档、查看索引设置、查看集群状态       |
| `DELETE`  | 删除         | 删除文档、删除索引                |

### 3.2 URL 结构

```
/<index>/_doc/<id>       单文档操作（CRUD）
/<index>/_search         搜索
/<index>/_bulk           批量操作
/<index>/_update/<id>    部分更新
/<index>/_mapping        查看映射
/_cat/indices            查看所有索引
```

### 3.3 与 SQL 概念的映射

| SQL 概念           | Elasticsearch 概念      | 说明                     |
| ----------------- | ---------------------- | ---------------------- |
| Database / Table  | **Index**（索引）          | 数据存储的逻辑容器              |
| Row               | **Document**（文档）       | 最小的数据单元，JSON 格式        |
| Column            | **Field**（字段）          | 文档中的键值对                |
| Schema            | **Mapping**（映射）        | 定义字段类型和搜索行为            |
| SQL 语句            | **Query DSL**          | 基于 JSON 的查询语言          |

这个映射不完全精确（例如 Index 同时包含了 Database 和 Table 的概念），但作为入门心智模型足够用。

## 4. 完整闭环：10 步验证

以下描述整个闭环中的每一步以及关键知识点。完整的可执行 curl 命令和 Java 代码分别见文末"实战"部分。

### 4.1 创建索引（Step 1-2）

使用 `PUT /<index>` 创建索引并显式定义 mapping。本例的 mapping 覆盖了最常用的字段类型组合：

```json
{
  "properties": {
    "id":         { "type": "keyword" },
    "name":       { "type": "text", "fields": { "keyword": { "type": "keyword" } } },
    "category":   { "type": "keyword" },
    "price":      { "type": "double" },
    "in_stock":   { "type": "boolean" },
    "created_at": { "type": "date" }
  }
}
```

- `name` 字段同时配置 `text`（分词后全文检索）和 `keyword` 子字段（精确匹配、排序、聚合），这是字符串字段最常见的组合方式。
- `category` 用 `keyword` 而不是 `text`，因为类目名称需要精确匹配而不是模糊搜索。
- 本地测试设置 `number_of_shards: 1, number_of_replicas: 0`，单分片零副本即可。

创建后用 `_cat/indices/<index>?v` 验证，预期 health 为 green、docs.count 为 0。

### 4.2 写入文档（Step 3-4）

**单条写入**使用 `PUT /<index>/_doc/<id>`，指定文档 ID。加 `?refresh=wait_for` 确保写入后等待 Refresh 完成再返回，后续查询一定能查到（详见第 5 节 NRT 陷阱）。

**批量写入**使用 `POST /<index>/_bulk`，请求体为 NDJSON（Newline Delimited JSON）格式：

```
{"index":{"_id":"p2"}}          ← 操作元数据行
{"id":"p2","name":"...","price":8999}  ← 文档数据行
{"index":{"_id":"p3"}}          ← 下一组操作元数据
{"id":"p3","name":"...","price":599}   ← 下一组文档数据
                                ← 必须以空行（换行符）结尾
```

每两行为一组，第一行声明操作类型和文档 ID，第二行是文档内容。请求体**必须以换行符结尾**，否则最后一条操作会被忽略。

写入后必须检查响应中的 `errors` 字段，确认是否全部成功（详见第 6.2 节）。

### 4.3 查询与聚合（Step 5-6）

**全文检索 + 过滤 + 排序**使用 `bool` 查询：

- `must`：放全文检索条件（如 `match`），参与相关性评分。
- `filter`：放精确过滤条件（如 `term`、`range`），不评分，性能更好且结果可被缓存。
- `sort`：按指定字段排序，使用后默认按排序字段而非评分返回。

**聚合**使用 `terms` 聚合按 `category` 统计文档数量。设置 `size: 0` 表示不返回命中文档，只返回聚合结果——这在只需要统计数据、不需要文档内容时能显著减少网络传输。

### 4.4 单文档操作（Step 7-9）

- **获取**：`GET /<index>/_doc/<id>` 按 ID 获取完整文档，返回包含 `_source`、`_version`、`_seq_no` 等元数据。
- **更新**：`POST /<index>/_update/<id>` 做部分更新（Partial Update），只修改 `doc` 中指定的字段，其余字段保持不变。ES 内部实际是"读旧文档 -> 合并变更 -> 写入新版本"。
- **删除**：`DELETE /<index>/_doc/<id>` 删除单条文档。文档并非立即物理删除，而是被标记为已删除，后续 Segment Merge 时才真正回收空间。

### 4.5 清理索引（Step 10）

使用 `DELETE /<index>` 删除整个索引及其全部数据。这一步确保测试可以重复执行——每次重跑闭环不会因为索引已存在而报错。

## 5. 近实时（NRT）陷阱

### 5.1 现象

代码中刚写入一条文档，紧接着执行 `_search` 查询，结果返回为空。

### 5.2 原因

ES 是近实时（Near Real-Time）系统，写入到可搜索之间存在一个时间窗口：

```
写入请求
  │
  ▼
Memory Buffer（内存缓冲区）── 数据已持久化到 Translog，但不可搜索
  │
  ▼  Refresh（默认每 1 秒）
Lucene Segment（打开新 Searcher）── 数据变为可搜索
```

文档写入后先进入内存缓冲区。默认每隔 1 秒，ES 执行一次 Refresh 操作，将缓冲区中的数据生成新的 Lucene Segment 并打开新的 Searcher。只有 Refresh 之后，文档才能被 `_search` API 查到。

这个设计是有意为之的：频繁 Refresh 会消耗大量 CPU 和 IO 资源，1 秒间隔是 ES 在搜索实时性与写入吞吐之间的默认平衡点。

### 5.3 解决方案

在写入请求的 URL 中加 `?refresh=wait_for`，让请求阻塞直到下一次 Refresh 完成后再返回：

```
PUT /<index>/_doc/<id>?refresh=wait_for
POST /<index>/_bulk?refresh=wait_for
```

`wait_for` 不会额外触发 Refresh，而是等待定时 Refresh 自然发生。多个 `wait_for` 请求会合并等待同一次 Refresh，因此比 `?refresh=true`（立即强制 Refresh）更温和。但无论哪种，都会降低写入吞吐。

### 5.4 最佳实践

| 场景           | 建议                        | 原因                     |
| ------------ | ------------------------- | ---------------------- |
| 测试/验证        | `?refresh=wait_for`       | 确保写入后立刻可查              |
| 生产搜索业务       | 不加 refresh 参数（默认 1s 延迟）   | 99% 的场景完全可以接受          |
| 初始数据导入       | `refresh_interval: "-1"` | 关闭定时 Refresh，导完后手动刷新   |

注意：`GET /<index>/_doc/<id>` 是实时的，不依赖 Refresh——它直接从 Translog 或 Segment 中读取。如果 Get 能拿到但 Search 查不到，说明就是 Refresh 延迟。

## 6. 常见问题排查

### 6.1 写入后查不到

最常见的原因是 NRT 延迟。排查步骤：

1. 用 `GET /<index>/_doc/<id>` 确认文档是否已写入（Get API 实时，不依赖 Refresh）。
2. 用 `GET /<index>/_count` 确认文档总数。
3. 如果 Get 能拿到但 Search 查不到，等 1 秒后重试或加 `?refresh=wait_for`。

### 6.2 Bulk 返回 200 但数据不全

Bulk API 是**部分成功**语义——即使部分操作失败，HTTP 状态码仍然是 200。这与关系数据库的事务行为完全不同：Bulk 中的 10 条操作如果第 3 条因为 mapping 冲突失败，其余 9 条仍然会成功写入。

正确的检查方式：

1. 先看顶层 `errors` 字段：`false` 表示全部成功，`true` 表示至少有一条失败。
2. 遍历 `items` 数组，逐条检查每个操作的 `status` 和 `error` 字段。
3. 对失败的操作记录原因并根据错误类型决定是否重试（如 429 Rejected 应重试，mapping 冲突不应重试）。

常见失败原因：mapping 类型冲突（往 `double` 字段写入字符串）、文档过大、版本冲突等。

### 6.3 字段类型不对导致查询异常

没有显式定义 mapping 时，ES 会根据第一条写入的文档自动推断字段类型（Dynamic Mapping）。自动推断有时会出错：

- 数字字符串 `"123"` 被推断为 `text` 而不是 `integer`，导致无法做 `range` 查询。
- 日期字符串 `"2026/03/01"` 不符合默认的日期格式 `strict_date_optional_time`，被推断为 `text`，导致日期相关查询和聚合全部失效。

排查步骤：

1. `GET /<index>/_mapping?pretty` 查看实际映射类型。
2. 如果类型不对，mapping 中已存在字段的类型无法修改，只能重建索引。

**建议始终显式定义 mapping**，这是避免此类问题的根本方法。

---

# 总结

- 最小闭环是 ES 项目启动阶段最重要的"共识资产"：创建索引 -> 写入 -> 查询 -> 更新 -> 删除 -> 清理。
- ES 的 API 是纯 RESTful 设计，HTTP 动词直接映射 CRUD 语义，URL 路径即资源定位。
- 先用 curl 手动跑通每一步，再用 Java SDK 固化成可重复的自动化测试。
- NRT 不是 bug 而是设计——1 秒的 Refresh 间隔是吞吐与实时性的最佳平衡点。
- Bulk 的部分失败语义是最容易被忽略的坑——必须检查 `errors` 字段。
- 显式 mapping 优于 Dynamic Mapping，避免类型推断错误带来的后续排障成本。

## 实战（curl）

> 完整的 10 步闭环。所有命令可直接复制执行，未开启安全时去掉 `-u "$ES_USER:$ES_PASS"` 即可。

### Step 1：创建索引

```bash
curl -sS -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/$IDX" \
  -H "Content-Type: application/json" \
  -d '{
    "settings": { "number_of_shards": 1, "number_of_replicas": 0 },
    "mappings": { "properties": {
      "id":         { "type": "keyword" },
      "name":       { "type": "text", "fields": { "keyword": { "type": "keyword" } } },
      "category":   { "type": "keyword" },
      "price":      { "type": "double" },
      "in_stock":   { "type": "boolean" },
      "created_at": { "type": "date" }
    }}
  }'
```

### Step 2：验证索引

```bash
curl -sS -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/indices/$IDX?v"
```

### Step 3：单条写入

```bash
curl -sS -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/$IDX/_doc/p1?refresh=wait_for" \
  -H "Content-Type: application/json" \
  -d '{
    "id":"p1","name":"轻薄笔记本 Pro 14","category":"laptop",
    "price":6999,"in_stock":true,"created_at":"2026-03-01T10:00:00Z"
  }'
```

### Step 4：批量写入

```bash
curl -sS -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_bulk?refresh=wait_for" \
  -H "Content-Type: application/x-ndjson" \
  -d '{"index":{"_id":"p2"}}
{"id":"p2","name":"游戏笔记本 Max","category":"laptop","price":8999,"in_stock":false,"created_at":"2026-03-02T08:00:00Z"}
{"index":{"_id":"p3"}}
{"id":"p3","name":"无线机械键盘","category":"keyboard","price":599,"in_stock":true,"created_at":"2026-03-03T12:30:00Z"}
{"index":{"_id":"p4"}}
{"id":"p4","name":"降噪头戴耳机","category":"audio","price":1299,"in_stock":true,"created_at":"2026-03-04T09:00:00Z"}
{"index":{"_id":"p5"}}
{"id":"p5","name":"便携显示器 15.6 寸","category":"monitor","price":1599,"in_stock":true,"created_at":"2026-03-05T14:00:00Z"}
'
```

### Step 5：确认文档数

```bash
curl -sS -u "$ES_USER:$ES_PASS" "$ES_URL/$IDX/_count?pretty"
```

### Step 6：全文检索 + 过滤 + 排序

```bash
curl -sS -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{
    "query": { "bool": {
      "must":   [{ "match": { "name": "笔记本" } }],
      "filter": [
        { "term":  { "category": "laptop" } },
        { "range": { "price": { "lte": 8000 } } }
      ]
    }},
    "sort": [{ "price": "asc" }],
    "size": 10
  }'
```

### Step 7：聚合统计

```bash
curl -sS -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{
    "size": 0,
    "aggs": { "by_category": { "terms": { "field": "category" } } }
  }'
```

### Step 8：获取单文档

```bash
curl -sS -u "$ES_USER:$ES_PASS" "$ES_URL/$IDX/_doc/p1?pretty"
```

### Step 9：更新文档

```bash
curl -sS -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_update/p1?refresh=wait_for&pretty" \
  -H "Content-Type: application/json" \
  -d '{ "doc": { "price": 6499, "in_stock": false } }'
```

### Step 10：删除文档

```bash
curl -sS -u "$ES_USER:$ES_PASS" -X DELETE "$ES_URL/$IDX/_doc/p5?refresh=wait_for&pretty"
```

### 清理索引

```bash
curl -sS -u "$ES_USER:$ES_PASS" -X DELETE "$ES_URL/$IDX?pretty"
```

## 实战（Java SDK）

### Maven 依赖

```xml
<dependency>
  <groupId>co.elastic.clients</groupId>
  <artifactId>elasticsearch-java</artifactId>
  <version>8.17.0</version>
</dependency>
<dependency>
  <groupId>com.fasterxml.jackson.core</groupId>
  <artifactId>jackson-databind</artifactId>
  <version>2.17.0</version>
</dependency>
```

### 完整示例

```java
import co.elastic.clients.elasticsearch.ElasticsearchClient;
import co.elastic.clients.elasticsearch._types.Refresh;
import co.elastic.clients.elasticsearch._types.SortOrder;
import co.elastic.clients.elasticsearch.core.BulkRequest;
import co.elastic.clients.elasticsearch.core.SearchResponse;
import co.elastic.clients.elasticsearch.core.bulk.BulkOperation;
import co.elastic.clients.elasticsearch.indices.CreateIndexResponse;
import co.elastic.clients.json.jackson.JacksonJsonpMapper;
import co.elastic.clients.transport.rest_client.RestClientTransport;
import org.apache.http.HttpHost;
import org.apache.http.auth.AuthScope;
import org.apache.http.auth.UsernamePasswordCredentials;
import org.apache.http.impl.client.BasicCredentialsProvider;
import org.elasticsearch.client.RestClient;

import java.util.List;
import java.util.Map;

public class EsChapter02Demo {
    public static void main(String[] args) throws Exception {
        String index = "products_v1";

        BasicCredentialsProvider cp = new BasicCredentialsProvider();
        cp.setCredentials(AuthScope.ANY,
            new UsernamePasswordCredentials("elastic-admin", "elastic-password"));

        RestClient restClient = RestClient
            .builder(new HttpHost("localhost", 9200, "http"))
            .setHttpClientConfigCallback(hc -> hc.setDefaultCredentialsProvider(cp))
            .build();
        ElasticsearchClient client = new ElasticsearchClient(
            new RestClientTransport(restClient, new JacksonJsonpMapper()));

        // 1) 创建索引
        CreateIndexResponse createResp = client.indices().create(c -> c
            .index(index)
            .settings(s -> s.numberOfShards("1").numberOfReplicas("0"))
            .mappings(m -> m
                .properties("id", p -> p.keyword(k -> k))
                .properties("name", p -> p.text(t -> t
                    .fields("keyword", sf -> sf.keyword(k -> k))))
                .properties("category", p -> p.keyword(k -> k))
                .properties("price", p -> p.double_(d -> d))
                .properties("in_stock", p -> p.boolean_(b -> b))
                .properties("created_at", p -> p.date(d -> d)))
        );
        System.out.println("create acknowledged=" + createResp.acknowledged());

        // 2) 批量写入
        List<BulkOperation> ops = List.of(
            BulkOperation.of(o -> o.index(i -> i.index(index).id("p1").document(
                Map.of("id", "p1", "name", "轻薄笔记本 Pro 14", "category", "laptop",
                    "price", 6999, "in_stock", true, "created_at", "2026-03-01T10:00:00Z")))),
            BulkOperation.of(o -> o.index(i -> i.index(index).id("p2").document(
                Map.of("id", "p2", "name", "游戏笔记本 Max", "category", "laptop",
                    "price", 8999, "in_stock", false, "created_at", "2026-03-02T08:00:00Z")))),
            BulkOperation.of(o -> o.index(i -> i.index(index).id("p3").document(
                Map.of("id", "p3", "name", "无线机械键盘", "category", "keyboard",
                    "price", 599, "in_stock", true, "created_at", "2026-03-03T12:30:00Z"))))
        );
        var bulkResp = client.bulk(BulkRequest.of(b -> b
            .index(index).operations(ops).refresh(Refresh.WaitFor)));
        if (bulkResp.errors()) {
            bulkResp.items().stream()
                .filter(item -> item.error() != null)
                .forEach(item -> System.err.println(
                    "Failed: " + item.id() + " -> " + item.error().reason()));
        } else {
            System.out.println("Bulk wrote " + bulkResp.items().size() + " docs");
        }

        // 3) 搜索
        SearchResponse<Map> searchResp = client.search(s -> s
                .index(index)
                .query(q -> q.bool(b -> b
                    .must(m -> m.match(mm -> mm.field("name").query("笔记本")))
                    .filter(f -> f.term(t -> t.field("category").value("laptop")))
                    .filter(f -> f.range(r -> r.number(n -> n.field("price").lte(8000d))))))
                .sort(so -> so.field(f -> f.field("price").order(SortOrder.Asc)))
                .size(10),
            Map.class);
        System.out.println("hits=" + searchResp.hits().total().value());
        searchResp.hits().hits().forEach(hit ->
            System.out.println("  " + hit.id() + " -> " + hit.source()));

        // 4) 更新文档
        client.update(u -> u.index(index).id("p1")
            .doc(Map.of("price", 6499, "in_stock", false))
            .refresh(Refresh.WaitFor), Map.class);

        // 5) 删除文档
        client.delete(d -> d.index(index).id("p3").refresh(Refresh.WaitFor));

        // 6) 删除索引
        client.indices().delete(d -> d.index(index));
        System.out.println("Index deleted");

        restClient.close();
    }
```
