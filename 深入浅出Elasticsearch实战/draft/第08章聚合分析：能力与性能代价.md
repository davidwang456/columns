# 第08章 聚合分析：能力与性能代价

## 背景

搜索引擎解决的核心问题是"哪些文档匹配"，但业务上还有大量"统计类问题"：每天新增多少用户、各类目销量如何、平均客单价是多少。在关系型数据库中，这些问题通过 `GROUP BY` 配合聚合函数来回答。Elasticsearch 提供了功能极其丰富的聚合框架（Aggregations），能够在海量数据上实时完成分桶、指标计算和管道分析。然而，聚合的灵活性伴随着显著的性能代价——高基数字段爆桶、堆内存膨胀、GC 停顿等问题在生产环境中频繁出现。理解聚合的能力边界与性能陷阱，是用好 Elasticsearch 的关键一课。

## 本章目标

1. 掌握 Bucket、Metric、Pipeline 三大聚合类型的用法与适用场景
2. 理解聚合嵌套（分桶内嵌指标）的组合模式
3. 识别高基数 terms、大范围 date_histogram、text 字段聚合等性能风险
4. 掌握 Composite 聚合的分页遍历模式
5. 掌握聚合与查询组合、global 聚合的使用方式
6. 通过 curl 和 Java SDK 完成典型聚合实战

---

## 1. 聚合解决什么问题

搜索与聚合是 Elasticsearch 的两大核心能力，但回答的问题截然不同：

| 能力 | 回答的问题 | 类比 SQL |
|------|-----------|----------|
| 搜索 | 哪些文档匹配条件 | `SELECT * FROM ... WHERE ...` |
| 聚合 | 满足条件的数据呈现怎样的统计特征 | `SELECT COUNT(*), AVG(price) ... GROUP BY category` |

典型的聚合场景包括：

- 电商：各类目销量 Top10、日均订单量趋势、客单价分布
- 日志：每小时错误数、按状态码分组的请求量、P99 延迟
- 用户画像：各地区注册用户数、年龄段分布

聚合请求通过 `aggs`（或 `aggregations`）字段嵌入搜索请求体中，可以与查询条件组合使用。

## 2. 聚合类型体系

Elasticsearch 的聚合分为三大类：Bucket（分桶）、Metric（指标）和 Pipeline（管道）。

### 2.1 Bucket 聚合：分桶

Bucket 聚合将文档划分为若干"桶"，每个桶内的文档满足某种条件。它不直接计算指标，而是为后续的 Metric 聚合提供分组基础。

#### 2.1.1 terms 聚合

按字段值分组，类似 SQL 的 `GROUP BY`。

关键参数：

- `size`：返回的桶数量，默认 10。注意这不是精确的 Top-N，而是近似结果
- `shard_size`：每个分片返回的候选桶数量，默认 `size * 1.5 + 10`，增大可提高精度但增加开销
- `order`：排序方式，默认按 `_count` 降序，可按子聚合指标排序

#### 2.1.2 date_histogram 聚合

按时间分桶，是时序分析的基础。

- `fixed_interval`：固定间隔，如 `"1d"`、`"1h"`、`"30m"`
- `calendar_interval`：日历间隔，如 `"1M"`（考虑月份天数差异）、`"1q"`（季度）
- `min_doc_count`：最小文档数，设为 0 可显示空桶
- `extended_bounds`：强制覆盖的时间范围，即使没有数据也生成桶

#### 2.1.3 histogram 聚合

按数值等宽分桶，适合价格区间、年龄段等分析。

- `interval`：桶宽度，如 100 表示 0-99、100-199...

#### 2.1.4 range 聚合

自定义区间分桶，区间可以不等宽，适合业务自定义分段。

#### 2.1.5 filter / filters 聚合

按查询条件分桶。`filter` 只有一个条件（一个桶），`filters` 支持多个命名条件（多个桶），适合将不同业务规则映射为桶。

### 2.2 Metric 聚合：指标

Metric 聚合在一组文档上计算统计指标。

#### 2.2.1 基础指标

- `sum`：求和
- `avg`：平均值
- `max` / `min`：最大 / 最小值
- `stats`：一次返回 count、min、max、avg、sum 五个指标，减少请求次数
- `value_count`：计数（包含该字段的文档数）

#### 2.2.2 cardinality 近似去重计数

基于 HyperLogLog++ 算法，返回字段的近似唯一值数量。

- `precision_threshold`：精度阈值，默认 3000，最大 40000。低于阈值的基数几乎精确，高于阈值则误差率约 1-6%
- 内存占用与 precision_threshold 正相关，生产中建议根据实际基数选择合适值

#### 2.2.3 percentiles 分位数

计算数据分布的百分位值，常用于延迟分析（P50、P95、P99）。

- 基于 TDigest 算法，极端分位（P99、P99.9）的精度高于中间分位
- `percents` 参数指定要计算的百分位列表

### 2.3 Pipeline 聚合：管道

Pipeline 聚合不直接作用于文档，而是基于其他聚合的输出做二次计算。

- `avg_bucket`：对多个桶的某个指标求平均
- `max_bucket`：找到指标最大的那个桶
- `derivative`：计算相邻桶之间的差值（环比）
- `cumulative_sum`：计算累计和

Pipeline 聚合通过 `buckets_path` 引用父级聚合的输出路径。

## 3. 聚合嵌套

Bucket 聚合内可以嵌套 Metric 聚合，形成"分桶 + 统计"的组合模式。例如按类目分桶后，在每个桶内计算平均价格：

```json
{
  "aggs": {
    "by_category": {
      "terms": { "field": "category.keyword", "size": 10 },
      "aggs": {
        "avg_price": { "avg": { "field": "price" } }
      }
    }
  }
}
```

嵌套可以多层：Bucket 内再嵌 Bucket，再嵌 Metric。但层数越深，计算量指数增长，需要控制嵌套深度。

---

## 4. 性能风险

### 4.1 高基数 terms 聚合

当字段的唯一值（cardinality）非常高时（如用户 ID、IP 地址），terms 聚合会生成海量桶，导致：

- 每个分片构建大量桶结构，内存急剧膨胀
- 协调节点合并各分片的桶列表，GC 压力增大
- 极端情况下触发 CircuitBreakingException

应对策略：控制 `size` 参数、使用 Composite 聚合分页遍历、或先用 filter 缩小数据范围。

### 4.2 大时间范围 date_histogram

如果时间范围跨数年且 interval 设为 `1h`，桶数量将达到数万，同样引发内存问题。应确保 interval 与时间范围匹配：范围越大，interval 越粗。

### 4.3 text 字段聚合

对 text 字段做聚合需要加载 fielddata 到 JVM 堆内存，这会迅速耗尽堆空间。默认情况下 Elasticsearch 禁止 text 字段的 fielddata。正确做法是使用 keyword 子字段（如 `title.keyword`）进行聚合。

---

## 5. Composite 聚合

### 5.1 解决的问题

terms 聚合的 `size` 有上限，无法遍历所有桶。Composite 聚合提供了基于游标的分页机制，允许逐页获取所有桶，且内存消耗可控。

### 5.2 工作原理

- 第一次请求不带 `after` 参数，返回第一页桶和一个 `after_key`
- 后续请求将上一次的 `after_key` 作为 `after` 参数传入，获取下一页
- 当返回的桶数量少于 `size` 时，遍历结束

### 5.3 与 terms 的对比

| 维度 | terms | composite |
|------|-------|-----------|
| 结果集 | 近似 Top-N | 精确遍历全部桶 |
| 内存 | 随 size 增大线性增长 | 恒定（每页固定 size） |
| 分页 | 不支持 | 支持（after 游标） |

---

## 6. 聚合 + 查询组合

聚合默认作用于查询命中的文档集合。先通过 filter 缩小范围再做聚合，是提升性能的核心手段：

```json
{
  "query": { "bool": { "filter": [{ "range": { "created_at": { "gte": "2024-01-01" } } }] } },
  "aggs": { "by_status": { "terms": { "field": "status.keyword" } } }
}
```

上例先过滤出 2024 年之后的数据，再按状态分桶，避免对全量数据聚合。

---

## 7. global 聚合

global 聚合会跳过查询的 filter 条件，在索引的全量文档上进行聚合。典型场景：查询某品牌的商品，同时需要展示所有品牌的总销量作为对比。

```json
{
  "query": { "term": { "brand.keyword": "Apple" } },
  "aggs": {
    "apple_avg_price": { "avg": { "field": "price" } },
    "all_brands": {
      "global": {},
      "aggs": { "total_avg_price": { "avg": { "field": "price" } } }
    }
  }
}
```

`apple_avg_price` 只计算 Apple 商品的均价，而 `all_brands.total_avg_price` 计算所有商品的均价。

---

## 8. 总结

- 聚合解决的是"统计类问题"，与搜索（"匹配类问题"）互补
- 三大聚合类型：Bucket（分桶）提供分组、Metric（指标）计算统计值、Pipeline（管道）做二次计算
- 聚合嵌套是核心使用模式：Bucket 内嵌 Metric
- 高基数 terms、大范围 date_histogram、text 字段 fielddata 是三大性能杀手
- Composite 聚合通过分页遍历解决高基数问题，内存开销可控
- 先 filter 再聚合是性能优化的第一原则
- global 聚合提供"跳出当前查询范围"的全局视角

## 11. 实战（Java SDK）

```java
import co.elastic.clients.elasticsearch.ElasticsearchClient;
import co.elastic.clients.elasticsearch._types.aggregations.*;
import co.elastic.clients.elasticsearch.core.SearchRequest;
import co.elastic.clients.elasticsearch.core.SearchResponse;

import java.util.List;
import java.util.Map;

public class AggregationExamples {

    private final ElasticsearchClient client;

    public AggregationExamples(ElasticsearchClient client) {
        this.client = client;
    }

    /**
     * terms + metric 嵌套：各类目平均价格
     */
    public void termWithMetric() throws Exception {
        SearchResponse<Void> resp = client.search(s -> s
            .index("orders")
            .size(0)
            .aggregations("by_category", a -> a
                .terms(t -> t.field("category").size(10))
                .aggregations("avg_price", sub -> sub
                    .avg(av -> av.field("price"))
                )
            ),
            Void.class
        );

        StringTermsAggregate byCategory = resp.aggregations()
            .get("by_category").sterms();
        for (StringTermsBucket bucket : byCategory.buckets().array()) {
            double avg = bucket.aggregations().get("avg_price").avg().value();
            System.out.printf("类目: %s, 订单数: %d, 平均价格: %.2f%n",
                bucket.key().stringValue(), bucket.docCount(), avg);
        }
    }

    /**
     * date_histogram：按天统计营收
     */
    public void dateHistogram() throws Exception {
        SearchResponse<Void> resp = client.search(s -> s
            .index("orders")
            .size(0)
            .aggregations("daily_orders", a -> a
                .dateHistogram(dh -> dh
                    .field("created_at")
                    .fixedInterval(fi -> fi.time("1d"))
                    .minDocCount(0)
                )
                .aggregations("total_revenue", sub -> sub
                    .sum(sm -> sm.field("price"))
                )
            ),
            Void.class
        );

        DateHistogramAggregate daily = resp.aggregations()
            .get("daily_orders").dateHistogram();
        for (DateHistogramBucket bucket : daily.buckets().array()) {
            double revenue = bucket.aggregations()
                .get("total_revenue").sum().value();
            System.out.printf("日期: %s, 订单数: %d, 总营收: %.2f%n",
                bucket.keyAsString(), bucket.docCount(), revenue);
        }
    }

    /**
     * Composite 分页遍历所有类目
     */
    public void compositePagination() throws Exception {
        Map<String, String> afterKey = null;

        while (true) {
            final Map<String, String> currentAfter = afterKey;
            SearchResponse<Void> resp = client.search(s -> {
                s.index("orders").size(0);
                s.aggregations("all_categories", a -> a
                    .composite(c -> {
                        c.size(2);
                        c.sources(List.of(
                            Map.of("category",
                                CompositeAggregationSource.of(
                                    cs -> cs.terms(t -> t.field("category"))
                                ))
                        ));
                        if (currentAfter != null) {
                            c.after(Map.of("category", FieldValue.of(
                                currentAfter.get("category"))));
                        }
                        return c;
                    })
                    .aggregations("avg_price", sub -> sub
                        .avg(av -> av.field("price"))
                    )
                );
                return s;
            }, Void.class);

            CompositeAggregate composite = resp.aggregations()
                .get("all_categories").composite();
            List<CompositeBucket> buckets = composite.buckets().array();
            if (buckets.isEmpty()) break;

            for (CompositeBucket bucket : buckets) {
                double avg = bucket.aggregations().get("avg_price").avg().value();
                System.out.printf("类目: %s, 平均价格: %.2f%n",
                    bucket.key().get("category").stringValue(), avg);
            }

            if (composite.afterKey() == null || buckets.size() < 2) break;
            afterKey = Map.of("category",
                composite.afterKey().get("category").stringValue());
        }
    }

    /**
     * cardinality：独立用户数
     */
    public void cardinality() throws Exception {
        SearchResponse<Void> resp = client.search(s -> s
            .index("orders")
            .size(0)
            .aggregations("unique_users", a -> a
                .cardinality(c -> c.field("user_id").precisionThreshold(1000))
            ),
            Void.class
        );
        long uniqueUsers = (long) resp.aggregations()
            .get("unique_users").cardinality().value();
        System.out.println("独立用户数: " + uniqueUsers);
    }

    /**
     * percentiles：价格分位数
     */
    public void percentiles() throws Exception {
        SearchResponse<Void> resp = client.search(s -> s
            .index("orders")
            .size(0)
            .aggregations("price_pct", a -> a
                .percentiles(p -> p
                    .field("price")
                    .percents(50.0, 75.0, 90.0, 95.0, 99.0)
                )
            ),
            Void.class
        );
        Percentiles pcts = resp.aggregations()
            .get("price_pct").tdigestPercentiles();
        pcts.values().keyed().forEach((k, v) ->
            System.out.printf("P%s: %.2f%n", k, v));
    }
}
```
