# 第08章 聚合分析：能力与性能代价

ES 的聚合能力很强，很多团队把它当“实时分析引擎”使用。但聚合并非零成本，尤其在高基数字段和大时间窗口场景中，资源消耗会迅速放大。

## 本章目标
- 会用聚合做统计分析。
- 理解聚合对性能和内存的影响。

## 1. 聚合解决什么问题
聚合适合回答“统计类问题”，例如：
- 每天新增用户数
- 每个类目销量
- 平均客单价

## 2. 常见聚合类型
- Bucket：`terms`、`date_histogram`、`range`
- Metric：`sum`、`avg`、`max`、`cardinality`

## 3. 性能风险点
- 高基数字段做 `terms` 聚合开销大。
- 大范围时间聚合会拉高内存与延迟。

## 4. 工程建议
- 先限制时间窗口与筛选条件。
- 大结果分页聚合优先 `composite`。
- 指标字段类型要提前建好，避免运行时转换。

# 总结
- 聚合是 ES 的核心优势之一，但必须有边界意识。
- 先控制数据范围，再提升统计复杂度。

2. 比较 `terms` 与 `composite` 在分页场景的差异。  
3. 选一个高基数字段，分析聚合成本。  

## 实战（curl）

```bash
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{
    "size":0,
    "aggs":{
      "by_category":{"terms":{"field":"category"}},
      "avg_price":{"avg":{"field":"price"}}
    }
  }'
```

## 实战（Java SDK）

```java
var aggResp = client.search(s -> s.index("products_v1")
    .size(0)
    .aggregations("by_category", a -> a.terms(t -> t.field("category"))), Map.class);
System.out.println(aggResp.aggregations().keySet());
```
