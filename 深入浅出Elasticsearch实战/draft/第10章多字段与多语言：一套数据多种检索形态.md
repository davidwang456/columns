

# 第10章 多字段与多语言：一套数据多种检索形态

# 背景

同一个业务字段往往同时承担"搜索、过滤、排序、聚合"多种职责。`text` 能全文检索但不能排序，`keyword` 能精确过滤但不能分词搜索——单一字段类型无法满足所有需求。如果建模阶段没有一次性设计好，后续只能通过重建索引来补救，成本极高。Multi-fields、normalizer、copy_to 三个机制正是为了在一次写入中同时构建多种检索形态。当数据涉及多语言时，问题进一步放大：不同语言需要不同的分词器，索引结构和查询策略都必须相应调整。

## 本章目标

- 理解 multi-fields 的工作原理，掌握常见组合模式。
- 学会使用 normalizer 解决 keyword 字段的大小写归一化问题。
- 掌握 copy_to 实现"统一搜索入口"的建模技巧。
- 对比三种多语言建模方案，能根据业务特征做出选择。
- 通过电商商品名称的完整示例，串联本章所有知识点。

---

## 1. 为什么需要多字段

以"商品名称"字段为例，它至少要满足四种用途：

| 用途 | 需要的字段类型 | 原因 |
| --- | --- | --- |
| 全文搜索 | `text` | 需要分词、计算相关性评分 |
| 精确过滤 | `keyword` | `term` 查询要求未分词的完整值 |
| 排序 | `keyword` | `text` 字段无法直接排序 |
| 聚合 | `keyword` | `terms` 聚合需要未分词的原始值 |

如果只声明为 `text`，排序和聚合无法工作；如果只声明为 `keyword`，搜索"无线鼠标"无法匹配"罗技无线蓝牙鼠标"。Multi-fields 允许同一个字段同时以多种类型和分析器建立索引，一次写入、多份倒排索引、各取所需。

---

## 2. Multi-fields 详解

### 2.1 基本语法

在 mapping 中通过 `fields` 子属性为主字段声明多个子字段：

```json
{
  "mappings": { "properties": {
    "name": {
      "type": "text",
      "fields": {
        "keyword":  { "type": "keyword", "ignore_above": 256 },
        "standard": { "type": "text", "analyzer": "standard" }
      }
    }
  }}
}
```

### 2.2 最常用组合：text + keyword

这是 Elasticsearch 动态映射的默认行为——不指定 mapping 直接写入字符串字段时，ES 自动生成 `text` 主字段 + `keyword` 子字段。手动建模时建议显式声明，以便控制 `ignore_above` 等参数。查询时的字段路径：

| 操作 | 字段 | 查询示例 |
| --- | --- | --- |
| 全文搜索 | `name` | `{ "match": { "name": "无线鼠标" } }` |
| 精确过滤 | `name.keyword` | `{ "term": { "name.keyword": "罗技无线蓝牙鼠标" } }` |
| 排序 | `name.keyword` | `{ "sort": [{ "name.keyword": "asc" }] }` |
| 聚合 | `name.keyword` | `{ "terms": { "field": "name.keyword" } }` |

### 2.3 多分词器 Multi-fields

当业务需要支持多种检索方式时，可以为同一字段配置多个分词器：

```json
{
  "name": {
    "type": "text", "analyzer": "standard",
    "fields": {
      "cn":      { "type": "text", "analyzer": "ik_smart" },
      "pinyin":  { "type": "text", "analyzer": "pinyin" },
      "keyword": { "type": "keyword" }
    }
  }
}
```

- `name`：standard 分词器，适合英文和通用场景。
- `name.cn`：IK 中文分词器，"罗技无线蓝牙鼠标"被切分为"罗技/无线/蓝牙/鼠标"。
- `name.pinyin`：拼音分词器，支持用户输入拼音搜索。
- `name.keyword`：保留原始值用于过滤、排序和聚合。

### 2.4 存储原理

每个子字段独立维护自己的倒排索引。写入一条文档时，`name` 字段的值被分别交给每个子字段的分析器处理，生成各自的 term 并写入对应的倒排索引。存储开销与子字段数量成正比，建模时需要在检索灵活性和存储成本之间权衡。

## 3. Normalizer：keyword 的归一化处理

### 3.1 问题场景

keyword 字段不分词，意味着大小写敏感。同一个品牌名 `Apple`、`apple`、`APPLE` 在 `term` 查询和 `terms` 聚合中会被视为三个不同的值，导致过滤遗漏和聚合碎片化。

### 3.2 定义与使用

Normalizer 是 keyword 字段专属的"分析器"——不做分词（token 数量始终为 1），但可以对整个值做归一化处理。定义在 index settings 的 `analysis` 节点中：

```json
{
  "settings": { "analysis": { "normalizer": {
    "lowercase_normalizer": {
      "type": "custom",
      "filter": ["lowercase", "asciifolding"]
    }
  }}},
  "mappings": { "properties": {
    "brand": { "type": "keyword", "normalizer": "lowercase_normalizer" }
  }}
}
```

写入 `"Apple"` 后，实际索引的值为 `"apple"`。查询时输入 `"APPLE"` 同样会被归一化为 `"apple"`，确保匹配。支持的常用 filter 包括 `lowercase`（转小写）、`asciifolding`（去除变音符号，如 `cafe` 匹配 `cafe`）。

## 4. Copy_to：统一搜索入口

### 4.1 使用场景

用户在搜索框中输入关键词时，往往不区分搜的是"标题"还是"描述"还是"标签"。`copy_to` 在写入时将多个字段的值复制到一个"超级搜索字段"，查询时只搜这个字段即可：

```json
{
  "mappings": { "properties": {
    "title":       { "type": "text", "copy_to": "search_all" },
    "description": { "type": "text", "copy_to": "search_all" },
    "tags":        { "type": "keyword", "copy_to": "search_all" },
    "search_all":  { "type": "text", "analyzer": "ik_smart" }
  }}
}
```

### 4.2 关键特性

- **不出现在 `_source` 中**：`copy_to` 目标字段只存在于倒排索引中，`_source` 里看不到它。无法用它返回内容或高亮，但不影响搜索。
- **支持多目标**：`"copy_to": ["search_all", "search_title"]`，一个字段可以复制到多个目标。
- **分析器独立**：目标字段使用自己声明的分析器处理复制过来的值，与源字段的分析器无关。

## 5. 多语言建模策略

不同语言的词法规则差异巨大——英文靠空格分词，中文需要词典切分，日文需要形态素分析。以下对比三种常见方案。

### 5.1 方案一：同一索引、多字段

为每种语言创建独立的子字段，各自配置对应的分析器：

```json
{
  "title": {
    "type": "text", "analyzer": "standard",
    "fields": {
      "en": { "type": "text", "analyzer": "english" },
      "cn": { "type": "text", "analyzer": "ik_smart" },
      "ja": { "type": "text", "analyzer": "kuromoji" }
    }
  }
}
```

查询时使用 `multi_match` 跨子字段搜索。优点是管理简单，单次查询可跨语言匹配；缺点是不同语言的数据量差异会导致 IDF 偏差——中文文档占 90% 时，同一个词在日文子字段中的 IDF 值会异常高，影响评分公平性。

### 5.2 方案二：按语言拆索引

每种语言独立索引（`products_en`、`products_cn`、`products_ja`），各自配置最优的分析器。跨语言查询时使用 multi-index 或别名。优点是隔离性好，IDF 在各语言内部计算，评分更准确；缺点是维护成本高，需要管理多个索引的生命周期和模板。

### 5.3 方案三：语言检测 + 路由

先用语言检测（如 langdetect 插件或应用层检测）识别文档语言，再路由到对应索引。查询时同样先检测用户输入语言再路由。此方案是方案二的增强版，适合语言种类多且无法预先确定的场景。

### 5.4 选择建议

| 维度 | 同一索引多字段 | 按语言拆索引 | 语言检测+路由 |
| --- | --- | --- | --- |
| 语言数量 | 2~3 种 | 3 种以上 | 不确定/动态增长 |
| 数据量差异 | 各语言接近 | 差异大 | 差异大 |
| IDF 准确性 | 可能偏差 | 准确 | 准确 |
| 管理复杂度 | 低 | 中 | 高 |
| 推荐场景 | 内部系统、中小规模 | 电商国际化 | UGC 平台 |

## 6. 实战场景：电商商品名称的多字段建模

综合运用本章知识，为电商商品设计完整的多字段方案：

| 字段 | 类型 | 用途 |
| --- | --- | --- |
| `name` | text (standard) | 通用全文搜索 |
| `name.cn` | text (ik_smart) | 中文分词搜索 |
| `name.keyword` | keyword | 精确匹配、排序、聚合 |
| `name.suggest` | completion | 自动补全（搜索建议） |
| `brand` | keyword + normalizer | 品牌过滤，忽略大小写 |
| `search_all` | text (ik_smart) | copy_to 目标，统一搜索入口 |

这种设计确保了一次写入即可满足搜索、过滤、排序、聚合和自动补全五种需求。`search_all` 接收 `name`、`brand`、`description` 的内容，用户在搜索框中输入任意关键词都可以命中。

---

# 总结

- Multi-fields 让同一个字段在一次写入时生成多份倒排索引，满足不同检索形态的需求。
- `text + keyword` 是最基本的组合；多分词器 multi-fields 可同时支持标准分词、中文分词、拼音分词等。
- Normalizer 解决 keyword 字段的大小写和变音符号归一化问题，避免聚合碎片化和过滤遗漏。
- `copy_to` 将多个字段的值汇聚到统一搜索字段，简化查询逻辑；注意目标字段不出现在 `_source` 中。
- 多语言建模有三种主流方案，核心取舍在于 IDF 准确性与管理复杂度之间的平衡。
- 建模先于查询——在设计阶段就要明确每个字段"谁来搜、怎么搜、如何排序、是否聚合"。

## 实战（Java SDK）

```java
// ---------- 创建索引：multi-fields + normalizer + copy_to ----------
client.indices().create(c -> c.index("product_mf")
    .settings(s -> s.analysis(a -> a
        .normalizer("lowercase_norm", n -> n.custom(cn -> cn.filter("lowercase")))))
    .mappings(m -> m
        .properties("name", p -> p.text(t -> t
            .analyzer("standard").copyTo("search_all")
            .fields("cn", f -> f.text(tx -> tx.analyzer("ik_smart")))
            .fields("keyword", f -> f.keyword(k -> k.ignoreAbove(256)))))
        .properties("brand", p -> p.keyword(k -> k
            .normalizer("lowercase_norm").copyTo("search_all")))
        .properties("description", p -> p.text(t -> t
            .analyzer("ik_smart").copyTo("search_all")))
        .properties("search_all", p -> p.text(t -> t.analyzer("ik_smart")))));

// ---------- 批量写入 ----------
client.bulk(b -> b
    .operations(op -> op.index(i -> i.index("product_mf").id("1")
        .document(Map.of("name", "罗技无线蓝牙鼠标", "brand", "Logitech",
            "description", "静音办公无线鼠标，蓝牙5.0连接"))))
    .operations(op -> op.index(i -> i.index("product_mf").id("2")
        .document(Map.of("name", "Apple Magic Mouse", "brand", "APPLE",
            "description", "苹果妙控鼠标，多点触控表面"))))
    .operations(op -> op.index(i -> i.index("product_mf").id("3")
        .document(Map.of("name", "雷蛇电竞游戏鼠标", "brand", "Razer",
            "description", "高精度光学传感器，RGB灯效"))))
    .refresh(Refresh.WaitFor));

// ---------- 全文搜索（text 主字段）----------
var matchResp = client.search(s -> s.index("product_mf")
        .query(q -> q.match(m -> m.field("name").query("无线鼠标"))),
    Map.class);
System.out.println("全文搜索命中: " + matchResp.hits().total().value());

// ---------- 精确过滤（keyword 子字段）----------
var termResp = client.search(s -> s.index("product_mf")
        .query(q -> q.term(t -> t.field("name.keyword").value("罗技无线蓝牙鼠标"))),
    Map.class);
System.out.println("精确过滤命中: " + termResp.hits().total().value());

// ---------- Normalizer：大小写不敏感的品牌查询 ----------
var brandResp = client.search(s -> s.index("product_mf")
        .query(q -> q.term(t -> t.field("brand").value("apple"))),
    Map.class);
System.out.println("品牌查询命中: " + brandResp.hits().total().value());

// ---------- copy_to 统一搜索 ----------
var searchAllResp = client.search(s -> s.index("product_mf")
        .query(q -> q.match(m -> m.field("search_all").query("蓝牙鼠标"))),
    Map.class);
System.out.println("统一搜索命中: " + searchAllResp.hits().total().value());

// ---------- 品牌聚合 ----------
var aggResp = client.search(s -> s.index("product_mf").size(0)
        .aggregations("by_brand", a -> a.terms(t -> t.field("brand"))),
    Map.class);
aggResp.aggregations().get("by_brand").sterms().buckets().array().forEach(bucket ->
    System.out.println("品牌: " + bucket.key().stringValue() + ", 数量: " + bucket.docCount()));

// ---------- 排序 ----------
var sortResp = client.search(s -> s.index("product_mf")
        .sort(so -> so.field(f -> f.field("name.keyword").order(SortOrder.Asc))),
    Map.class);
sortResp.hits().hits().forEach(hit -> System.out.println("排序结果: " + hit.source()));

// ---------- 清理 ----------
client.indices().delete(d -> d.index("product_mf"));
```
