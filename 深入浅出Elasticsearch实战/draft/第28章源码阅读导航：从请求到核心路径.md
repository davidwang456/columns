# 背景

# 第28章 源码阅读导航：从请求到核心路径

当线上问题进入“API 层无法解释”阶段，就需要进入源码。关键不是读完整个仓库，而是建立一条可复用的定位路径。

## 本章目标
- 为高级工程师提供一条高效源码学习路线。
- 建立“从现象到代码”的定位能力。

## 1. 从哪里开始读
推荐从一个具体 API 入手：
- 搜索请求
- 创建索引请求

先跑通一条路径，再扩展到其他模块。

## 2. 典型阅读路径
- REST 层入口
- Transport Action 执行层
- 分片级执行与结果汇总

## 3. 阅读方法建议
- 先看主流程，再看异常分支。
- 用断点和日志验证调用链假设。
- 每次只解决一个问题，避免过度发散。

## 4. 团队落地方式
- 建立“源码导读会”，每次讲一条链路。
- 把定位经验沉淀成排障手册。

# 总结
- 源码学习的目标是提升问题解决效率，而不是记住所有类名。
- 以问题驱动阅读，收益最高。

## 练习题
1. 选一个搜索请求，画出你理解的调用路径。  
2. 在关键节点打断点验证执行顺序。  
3. 把本次阅读结果整理为团队 wiki。  

## 实战（curl）

```bash
# 用 profile + explain 观察执行行为
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{"profile":true,"query":{"match":{"name":"笔记本"}}}'

curl -u "$ES_USER:$ES_PASS" -X GET "$ES_URL/$IDX/_explain/p1?pretty" \
  -H "Content-Type: application/json" \
  -d '{"query":{"match":{"name":"笔记本"}}}'
```

## 实战（Java SDK）

```java
// 建议把断点下在 search 调用处，然后沿客户端到服务端链路追踪
var resp = client.search(s -> s.index("products_v1")
    .query(q -> q.match(m -> m.field("name").query("笔记本")))
    .profile(true), Map.class);
System.out.println(resp.took());
```
