

# 第28章 源码阅读导航：从请求到核心路径

# 背景

当线上问题进入"API 文档无法解释"的阶段——比如某个查询的行为与预期不符、某个配置项的实际效果不明确、或者你需要理解 ES 内部的某个决策逻辑——就需要进入源码。

但 Elasticsearch 的代码仓库有数千个 Java 文件、数百万行代码。漫无目的地阅读是低效的。关键在于建立一条"从现象到代码"的定位路径：以一个具体的 API 请求为线索，从 REST 入口开始，沿着 Transport 层、分片级执行、Lucene 底层一路追踪下去。

本章会给出搜索请求和索引请求的完整源码路径（包含实际类名和文件路径），一份关键类速查表，以及实用的源码阅读方法论和调试技巧。

## 本章目标

- 掌握 ES 源码的模块结构，知道去哪里找什么。
- 跟踪一条搜索请求从 REST 层到 Lucene 的完整源码路径。
- 跟踪一条索引请求从 REST 层到 Lucene IndexWriter 的完整源码路径。
- 建立关键类速查表（20 个最重要的类），快速定位代码。
- 掌握源码阅读方法论和调试技巧，形成可复用的定位能力。

## 1. ES 源码的模块结构

```
elasticsearch/
├── server/                    # 核心模块——绝大多数核心逻辑在这里
│   └── src/main/java/org/elasticsearch/
│       ├── rest/              # REST 层（HTTP 请求入口）
│       │   └── action/        # 各 API 的 REST Handler
│       │       ├── search/    # RestSearchAction 等
│       │       ├── document/  # RestIndexAction, RestGetAction 等
│       │       └── admin/     # 集群/索引管理 API
│       ├── action/            # Transport Action（业务逻辑层）
│       │   ├── search/        # TransportSearchAction 等搜索逻辑
│       │   ├── bulk/          # TransportBulkAction 等写入逻辑
│       │   ├── get/           # 实时 Get 逻辑
│       │   └── admin/         # 集群/索引管理逻辑
│       ├── search/            # 搜索引擎核心
│       │   ├── SearchService.java
│       │   ├── fetch/         # Fetch 阶段
│       │   └── builder/       # SearchSourceBuilder 等
│       ├── index/             # 索引引擎核心
│       │   ├── engine/        # Engine, InternalEngine（Lucene 交互层）
│       │   ├── shard/         # IndexShard（分片管理）
│       │   ├── translog/      # Translog 实现
│       │   └── mapper/        # Mapping 和字段类型
│       ├── cluster/           # 集群管理
│       │   ├── service/       # ClusterService, MasterService
│       │   ├── routing/       # 分片路由和分配
│       │   └── coordination/  # 选举和协调
│       ├── transport/         # 节点间通信（Transport 协议）
│       └── common/            # 公共工具类
│
├── modules/                   # 默认随 ES 发布的模块
│   ├── aggregations/          # 聚合实现
│   ├── analysis-common/       # 常用分析器
│   ├── lang-painless/         # Painless 脚本语言
│   ├── reindex/               # Reindex API
│   └── ...
│
├── plugins/                   # 可选插件
│   ├── analysis-icu/          # ICU 分析器
│   ├── repository-s3/         # S3 快照仓库
│   └── ...
│
├── x-pack/                    # 商业功能（安全、ML、监控等）
│   └── plugin/
│       ├── security/          # 安全认证和授权
│       ├── ml/                # 机器学习
│       ├── watcher/           # 告警
│       └── ...
│
├── libs/                      # 内部库（不依赖 ES 核心）
│   ├── core/                  # 核心工具
│   ├── x-content/             # JSON/YAML 解析
│   └── ...
│
├── distribution/              # 发布包构建
└── qa/                        # 集成测试和多版本测试
```

**核心原则**：90% 的核心逻辑在 `server/` 模块中。阅读源码时，优先关注这个模块。

## 2. 一条搜索请求的完整源码路径

以 `POST /my_index/_search` 为例，追踪从 HTTP 请求到 Lucene 查询的完整路径。

### 2.1 HTTP 层入口

```
客户端发送 POST /my_index/_search
  │
  ▼
RestSearchAction
  路径: server/src/main/java/org/elasticsearch/rest/action/search/RestSearchAction.java
  职责:
    - 注册 HTTP 路由（GET/POST /{index}/_search）
    - 解析 HTTP 请求参数（from, size, search_type, preference 等）
    - 构建 SearchRequest 对象
    - 将 HTTP 请求体解析为 SearchSourceBuilder
    - 委托给 Transport 层执行
```

`RestSearchAction` 继承 `BaseRestHandler`，核心方法是 `prepareRequest()`，它将 HTTP 请求转换为内部的 `SearchRequest` 对象。

### 2.2 请求对象构建

```
SearchRequest
  路径: server/src/main/java/org/elasticsearch/action/search/SearchRequest.java
  职责:
    - 封装搜索请求的所有参数
    - 目标索引、search_type、preference、routing 等
    - 内含 SearchSourceBuilder

SearchSourceBuilder
  路径: server/src/main/java/org/elasticsearch/search/builder/SearchSourceBuilder.java
  职责:
    - 封装搜索请求体（query, aggs, sort, from, size, highlight 等）
    - 从 JSON 解析为内部数据结构
    - 负责 toXContent 序列化
```

### 2.3 Transport 层分发

```
TransportSearchAction
  路径: server/src/main/java/org/elasticsearch/action/search/TransportSearchAction.java
  职责:
    - 解析目标索引，展开别名和数据流
    - 从 Cluster State 获取分片路由信息
    - 决定搜索类型（query_then_fetch / dfs_query_then_fetch）
    - 创建对应的异步执行器（AbstractSearchAsyncAction 的子类）
    - 发起分布式搜索
```

这是搜索链路中最关键的"枢纽"类——它连接了 REST 层和分布式执行层。

### 2.4 分布式查询协调

```
AbstractSearchAsyncAction
  路径: server/src/main/java/org/elasticsearch/action/search/AbstractSearchAsyncAction.java
  职责:
    - 定义分布式搜索的通用框架
    - 管理分片级请求的发送和响应收集
    - 处理分片失败和超时
    - 子类实现具体的搜索策略

SearchQueryThenFetchAsyncAction（默认搜索类型）
  路径: server/src/main/java/org/elasticsearch/action/search/SearchQueryThenFetchAsyncAction.java
  职责:
    - 实现 Query Then Fetch 两阶段搜索
    - Query Phase: 向所有分片发送 query 请求，收集 doc_id + score
    - Fetch Phase: 全局排序后，向目标分片发送 fetch 请求取回文档

SearchDfsQueryThenFetchAsyncAction（DFS 模式）
  路径: server/src/main/java/org/elasticsearch/action/search/SearchDfsQueryThenFetchAsyncAction.java
  职责:
    - 在 Query Phase 之前增加 DFS 阶段
    - 收集全局 term statistics
    - 使用全局统计信息重新评分
```

### 2.5 分片级查询执行

```
SearchService
  路径: server/src/main/java/org/elasticsearch/search/SearchService.java
  职责:
    - 分片级搜索的入口
    - executeQueryPhase(): 在本地分片上执行 Query Phase
    - executeFetchPhase(): 在本地分片上执行 Fetch Phase
    - 管理搜索上下文（Search Context）的生命周期
    - 处理 scroll、PIT 等状态管理
```

`SearchService.executeQueryPhase()` 是"协调节点发来的 Query 请求在数据节点上的着陆点"。

### 2.6 Lucene 层查询

```
IndexSearcher.search()
  Lucene 的核心搜索方法
  职责:
    - 在所有 Segment 上执行查询
    - 通过 Weight → Scorer → DocIdSetIterator 遍历匹配文档
    - 使用 Collector 收集和排序结果
```

ES 的 `SearchService` 通过 `Engine.Searcher`（封装了 Lucene 的 `IndexSearcher`）执行实际查询。

### 2.7 结果合并

```
SearchPhaseController
  路径: server/src/main/java/org/elasticsearch/action/search/SearchPhaseController.java
  职责:
    - 合并所有分片的 Query Phase 结果
    - 全局排序，选出最终 top-N
    - 合并聚合结果
    - 构建最终的 SearchResponse
```

### 2.8 Fetch 阶段

```
FetchPhase
  路径: server/src/main/java/org/elasticsearch/search/fetch/FetchPhase.java
  职责:
    - 根据 doc_id 加载文档完整内容
    - 加载 _source、stored_fields
    - 执行 highlight
    - 执行 FetchSourcePhase、FetchFieldsPhase 等子阶段
```

### 2.9 完整调用链图

```
RestSearchAction.prepareRequest()
  │  构建 SearchRequest + SearchSourceBuilder
  ▼
TransportSearchAction.doExecute()
  │  解析索引 → 获取分片路由 → 创建异步执行器
  ▼
SearchQueryThenFetchAsyncAction.start()
  │
  ├── Query Phase ──→ 各数据节点的 SearchService.executeQueryPhase()
  │                      │
  │                      └── Engine.Searcher → IndexSearcher.search()
  │                              └── Weight → Scorer → Collector
  │
  ├── SearchPhaseController.merge()
  │      全局排序 → 选出 top-N → 确定需要 fetch 的分片
  │
  └── Fetch Phase ──→ 目标分片的 SearchService.executeFetchPhase()
                         │
                         └── FetchPhase.execute()
                                 └── 加载 _source, highlight, fields
```

## 3. 一条索引请求的完整源码路径

以 `PUT /my_index/_doc/1` 为例，追踪从 HTTP 请求到 Lucene IndexWriter 的完整路径。

### 3.1 HTTP 层入口

```
RestIndexAction
  路径: server/src/main/java/org/elasticsearch/rest/action/document/RestIndexAction.java
  职责:
    - 注册 HTTP 路由（PUT/POST /{index}/_doc/{id}）
    - 解析请求体和参数
    - 构建 IndexRequest
    - 委托给 Transport 层
```

### 3.2 Transport 层（单条也走 Bulk）

```
TransportBulkAction
  路径: server/src/main/java/org/elasticsearch/action/bulk/TransportBulkAction.java
  职责:
    - 所有写入操作（index, update, delete）都经过 Bulk 路径
    - 单条 IndexRequest 会被包装为只有一个操作的 BulkRequest
    - 按目标 ShardId 将操作分组
    - 每组构建 BulkShardRequest 发送到对应分片

TransportShardBulkAction
  路径: server/src/main/java/org/elasticsearch/action/bulk/TransportShardBulkAction.java
  职责:
    - 在主分片上逐条执行操作
    - 处理版本冲突（乐观锁）
    - 执行完主分片后，复制到副本分片
```

**重要设计**：即使只写一条文档，也走 `TransportBulkAction`。这简化了写入链路——只有一条路径需要维护。

### 3.3 分片级执行

```
IndexShard
  路径: server/src/main/java/org/elasticsearch/index/shard/IndexShard.java
  职责:
    - 分片管理的核心类
    - applyIndexOperationOnPrimary(): 主分片上的索引操作入口
    - 协调 Engine、Translog、Mapping 更新等
    - 管理分片的生命周期（创建、恢复、关闭等）
```

### 3.4 Engine 层

```
Engine（抽象类）
  路径: server/src/main/java/org/elasticsearch/index/engine/Engine.java
  职责:
    - 定义索引引擎的抽象接口
    - index(), delete(), get() 等核心操作
    - 管理 Searcher、Translog、版本映射

InternalEngine（默认实现）
  路径: server/src/main/java/org/elasticsearch/index/engine/InternalEngine.java
  职责:
    - Engine 的标准实现
    - index(): 执行索引操作
      1. 版本检查（乐观并发控制）
      2. 调用 Lucene IndexWriter 写入文档
      3. 写入 Translog
    - 管理 Lucene IndexWriter 的生命周期
    - refresh(): 打开新的 Searcher
    - flush(): 执行 Lucene Commit + 清理 Translog
```

### 3.5 Lucene 层

```
IndexWriter.addDocument() / updateDocument()
  Lucene 的核心写入方法
  职责:
    - 将文档转为倒排索引、doc_values、stored_fields 等数据结构
    - 写入内存中的 IndexWriter buffer
    - 由 ES 的 refresh 和 flush 机制控制持久化时机
```

### 3.6 Translog 记录

```
Translog
  路径: server/src/main/java/org/elasticsearch/index/translog/Translog.java
  职责:
    - WAL（Write-Ahead Log）实现
    - 每次写入操作同时记录到 Translog
    - durability=request 时每次操作后 fsync
    - 节点恢复时重放未 commit 的操作
```

### 3.7 完整调用链图

```
RestIndexAction.prepareRequest()
  │  构建 IndexRequest
  ▼
TransportBulkAction.doExecute()
  │  将 IndexRequest 包装为 BulkRequest
  │  按 ShardId 分组 → 构建 BulkShardRequest
  ▼
TransportShardBulkAction.shardOperationOnPrimary()
  │  逐条执行操作
  ▼
IndexShard.applyIndexOperationOnPrimary()
  │
  ▼
InternalEngine.index()
  │  1. 版本检查
  │  2. IndexWriter.addDocument() / updateDocument()
  │  3. Translog.add()
  ▼
返回结果 → 复制到副本分片 → 返回客户端
```

## 5. 源码阅读方法论

### 5.1 从 REST 层入口开始

所有 REST Handler 都在 `server/src/main/java/org/elasticsearch/rest/action/` 目录下，按功能域组织：

```
rest/action/
├── search/      # 搜索相关：RestSearchAction, RestMultiSearchAction, ...
├── document/    # 文档 CRUD：RestIndexAction, RestGetAction, RestDeleteAction, ...
├── admin/       # 管理 API
│   ├── cluster/ # 集群管理：RestClusterHealthAction, ...
│   └── indices/ # 索引管理：RestCreateIndexAction, RestPutMappingAction, ...
└── cat/         # _cat API：RestCatAction 的各子类
```

想了解一个 API 的行为，第一步就是找到对应的 `Rest*Action` 类。

### 5.2 找到对应的 TransportAction

每个 `Rest*Action` 都会将请求委托给一个 `TransportAction`。找到映射关系的方法：

1. 在 `Rest*Action` 中找到 `routes()` 方法——确认 HTTP 路由。
2. 找到 `prepareRequest()` 中构建的 `ActionRequest` 类型。
3. 搜索该 `ActionRequest` 对应的 `ActionType` 常量——它通常定义在请求类或专门的常量类中。
4. `ActionType` 的字符串名（如 `indices:data/read/search`）对应一个 `TransportAction` 实现。

### 5.3 跟踪到分片级执行

对于数据操作（搜索、写入），`TransportAction` 最终会将请求路由到具体分片：

- **搜索**：`TransportSearchAction` → `AbstractSearchAsyncAction` → `SearchService.executeQueryPhase()`。
- **写入**：`TransportBulkAction` → `TransportShardBulkAction` → `IndexShard.applyIndexOperationOnPrimary()`。

### 5.4 理解 Engine 接口

`Engine` 是 ES 与 Lucene 之间的抽象层。所有的索引操作（index, delete, get, refresh, flush）都通过 Engine 接口定义，由 `InternalEngine` 实现。

阅读 `InternalEngine` 的关键方法：

| 方法 | 职责 |
| --- | --- |
| `index()` | 索引文档：版本检查 → IndexWriter 写入 → Translog 记录 |
| `delete()` | 删除文档：版本检查 → IndexWriter 删除 → Translog 记录 |
| `get()` | 实时 Get：先查 Translog（最新版本），再查 Lucene |
| `refresh()` | 打开新的 IndexSearcher，让新文档可搜索 |
| `flush()` | Lucene Commit + 清理 Translog |

## 6. 调试技巧

### 6.1 IDE 中运行单节点

```bash
./gradlew :server:run
```

这会启动一个单节点 ES 实例。启动完成后，可以直接用 curl 或 Kibana 连接 `localhost:9200` 进行操作。

### 6.2 远程调试

```bash
./gradlew :server:run --debug-jvm
```

加上 `--debug-jvm` 后，ES 会在启动时等待调试器连接（默认端口 5005）。在 IDE 中配置 Remote Debug：

- Host: `localhost`
- Port: `5005`
- 连接后在感兴趣的类上设置断点

### 6.3 推荐断点位置

| 目的 | 断点位置 | 类名 |
| --- | --- | --- |
| 追踪搜索请求入口 | `RestSearchAction.prepareRequest()` | `RestSearchAction` |
| 追踪搜索分发逻辑 | `TransportSearchAction.doExecute()` | `TransportSearchAction` |
| 追踪分片级查询执行 | `SearchService.executeQueryPhase()` | `SearchService` |
| 追踪索引请求入口 | `RestIndexAction.prepareRequest()` | `RestIndexAction` |
| 追踪写入分片执行 | `InternalEngine.index()` | `InternalEngine` |
| 追踪 Translog 写入 | `Translog.add()` | `Translog` |
| 追踪 Refresh 过程 | `InternalEngine.refresh()` | `InternalEngine` |
| 追踪集群状态变更 | `MasterService.submitStateUpdateTask()` | `MasterService` |

### 6.4 运行单个测试

```bash
# 运行某个测试类
./gradlew :server:test --tests org.elasticsearch.action.search.TransportSearchActionTests

# 运行某个测试方法
./gradlew :server:test --tests org.elasticsearch.action.search.TransportSearchActionTests.testMethodName

# 附加调试器运行测试
./gradlew :server:test --tests org.elasticsearch.action.search.TransportSearchActionTests --debug-jvm
```

### 6.5 使用日志辅助

在 `log4j2.properties` 中临时调高特定包的日志级别：

```properties
logger.search.name = org.elasticsearch.action.search
logger.search.level = trace

logger.engine.name = org.elasticsearch.index.engine
logger.engine.level = debug
```

或者通过 API 动态调整（无需重启）：

```bash
PUT /_cluster/settings
{
  "transient": {
    "logger.org.elasticsearch.action.search": "TRACE"
  }
}
```

## 7. 推荐阅读顺序

### 7.1 第一轮：搜索路径（最直观）

搜索路径是最适合入门的路径——链路清晰，两阶段设计容易理解：

```
RestSearchAction → TransportSearchAction
  → AbstractSearchAsyncAction / SearchQueryThenFetchAsyncAction
  → SearchService.executeQueryPhase()
  → SearchPhaseController.merge()
  → FetchPhase
```

**阅读要点**：
- `TransportSearchAction` 如何决定 search_type。
- `AbstractSearchAsyncAction` 如何管理分片级请求的扇出和收集。
- `SearchService.executeQueryPhase()` 如何调用 Lucene IndexSearcher。

### 7.2 第二轮：写入路径

写入路径涉及更多的持久性和一致性逻辑：

```
RestIndexAction → TransportBulkAction
  → TransportShardBulkAction
  → IndexShard → InternalEngine.index()
  → Lucene IndexWriter + Translog
```

**阅读要点**：
- `TransportBulkAction` 如何按分片分组。
- `InternalEngine.index()` 的版本检查逻辑（乐观锁）。
- Translog 的写入时机和 durability 语义。

### 7.3 第三轮：集群状态管理

当你对搜索和写入路径都熟悉后，可以进入更底层的集群管理：

```
ClusterService → MasterService
  → ClusterState 更新 → 分片分配（AllocationService）
```

**阅读要点**：
- `ClusterState` 的结构：包含节点列表、路由表、metadata 等。
- `MasterService` 如何调度和应用集群状态变更。
- `AllocationService` 如何决定分片的分配和迁移。

### 7.4 第四轮：按兴趣深入

| 感兴趣的领域 | 起点类 | 所在模块 |
| --- | --- | --- |
| 聚合实现 | `AggregatorFactory` | `modules/aggregations/` |
| Mapping 和字段类型 | `MapperService`, `FieldMapper` | `server/.../index/mapper/` |
| 分词器 | `AnalysisRegistry` | `server/.../index/analysis/` |

| 安全认证 | `AuthenticationService` | `x-pack/.../security/` |

## 8. 高频问题的源码定位技巧

### 8.1 "这个 API 参数在哪里处理？"

1. 找到 `Rest*Action` 类。
2. 在 `prepareRequest()` 方法中搜索参数名（如 `"routing"`、`"preference"`）。
3. 追踪参数被传入的 `Request` 对象的哪个字段。

### 8.2 "这个错误信息从哪里抛出？"

1. 搜索错误信息的文本（如 `"Data too large"`）。
2. 找到抛出异常的位置。
3. 向上追踪调用栈，理解触发条件。

### 8.3 "这个配置项的实际效果是什么？"

1. 搜索配置项名称（如 `"index.refresh_interval"`）。
2. 找到 `Setting` 对象的定义——它通常包含默认值和验证逻辑。
3. 搜索 `Setting.get()` 的调用位置，理解配置值在哪里被使用。

### 8.4 "这个 Transport Action 的名字是什么？"

Transport Action 的名字编码在 `ActionType` 常量中，命名规则：

```
scope:category/operation

示例：
  indices:data/read/search        → TransportSearchAction
  indices:data/write/bulk          → TransportBulkAction
  cluster:admin/settings/update   → TransportClusterUpdateSettingsAction
  indices:admin/create            → TransportCreateIndexAction
```

搜索 `ActionType` 常量可以快速找到对应的 Transport Action。

# 总结

- ES 源码的核心在 `server/` 模块，按照 `rest/` → `action/` → `search/` / `index/` 的层次组织。90% 的常见问题都能在这个模块中找到答案。
- 搜索路径：`RestSearchAction` → `TransportSearchAction` → `SearchQueryThenFetchAsyncAction` → `SearchService.executeQueryPhase()` → Lucene `IndexSearcher.search()` → `SearchPhaseController` 合并 → `FetchPhase` 取回文档。
- 写入路径：`RestIndexAction` → `TransportBulkAction`（单条也走 Bulk）→ `TransportShardBulkAction` → `IndexShard` → `InternalEngine.index()` → Lucene `IndexWriter` + `Translog`。
- 源码阅读的方法论是**以问题驱动、以请求路径为线索**——从 REST 入口开始，找到 Transport Action，跟踪到分片级执行，理解 Engine 和 Lucene 的交互。
- 调试技巧：`./gradlew :server:run --debug-jvm` 启动单节点并附加调试器，在关键类（如 `SearchService.executeQueryPhase()`、`InternalEngine.index()`）设置断点。
- 推荐阅读顺序：**搜索路径** → **写入路径** → **集群状态管理** → 按兴趣深入。每次只解决一个问题，避免过度发散。

## 实战（Java SDK）

```java
// ---------- Profile 查询：观察源码中 Query Phase 的执行结果 ----------
var profileResp = client.search(s -> s.index("source_demo")
    .profile(true)
    .query(q -> q.bool(b -> b
        .must(m -> m.match(mt -> mt.field("title").query("Elasticsearch")))
        .filter(f -> f.term(t -> t.field("category").value("book"))))),
    Map.class);

System.out.println("=== Profile 结果（对应源码中 SearchService.executeQueryPhase 的输出）===");
System.out.println("took: " + profileResp.took() + "ms");
if (profileResp.profile() != null) {
    profileResp.profile().shards().forEach(shard -> {
        System.out.println("分片: " + shard.id());
        shard.searches().forEach(search -> {
            System.out.println("  Query 树（对应 Lucene IndexSearcher.search 中的查询分解）:");
            printQueryTree(search.query(), "    ");
            System.out.println("  Collector（对应 Lucene Collector 层级）:");
            search.collector().forEach(col ->
                System.out.println("    " + col.name() + " -> " + col.reason()
                    + " time=" + col.timeInNanos() + "ns"));
        });
        if (shard.fetch() != null) {
            System.out.println("  Fetch（对应源码中 FetchPhase.execute）:");
            System.out.println("    time=" + shard.fetch().timeInNanos() + "ns");
        }
    });
}

// ---------- Explain：观察 BM25 评分的内部计算 ----------
var explainResp = client.explain(e -> e
        .index("source_demo")
        .id("1")
        .query(q -> q.match(m -> m.field("title").query("Elasticsearch"))),
    Map.class);
System.out.println("\n=== Explain 结果（对应源码中 Weight.explain 的输出）===");
System.out.println("matched: " + explainResp.matched());
if (explainResp.explanation() != null) {
    printExplanation(explainResp.explanation(), "  ");
}

// ---------- _search_shards：观察分片路由（对应 TransportSearchAction 中的分片解析）----------
var shards = client.searchShards(s -> s.index("source_demo"));
System.out.println("\n=== _search_shards（对应 TransportSearchAction 的分片路由逻辑）===");
System.out.println("节点数: " + shards.nodes().size());
shards.shards().forEach(group -> {
    group.forEach(shard ->
        System.out.println("  分片 " + shard.shard()
            + " primary=" + shard.primary()
            + " node=" + shard.node()));
});

// ---------- 辅助方法 ----------
static void printQueryTree(List<QueryProfile> queries, String indent) {
    queries.forEach(pq -> {
        System.out.println(indent + pq.type()
            + " desc=\"" + pq.description() + "\""
            + " time=" + pq.timeInNanos() + "ns");
        if (pq.children() != null && !pq.children().isEmpty()) {
            printQueryTree(pq.children(), indent + "  ");
        }
    });
}

static void printExplanation(ExplanationDetail detail, String indent) {
    System.out.println(indent + detail.value() + " - " + detail.description());
    if (detail.details() != null) {
        detail.details().forEach(d -> printExplanation(d, indent + "  "));
    }
}

// ---------- 清理 ----------
client.indices().delete(d -> d.index("source_demo"));
```
