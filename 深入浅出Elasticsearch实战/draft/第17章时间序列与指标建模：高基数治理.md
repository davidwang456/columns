# 背景

# 第17章 时间序列与指标建模：高基数治理

监控与指标场景经常因为“标签太自由”导致高基数爆炸。等到查询变慢再治理，成本通常远高于建模阶段的约束。

## 本章目标
- 掌握时序与指标数据的建模重点。
- 避免高基数字段导致的性能失控。

## 1. 指标数据的两类字段
- 维度（dimensions）：用于分组过滤
- 指标（metrics）：用于统计计算

## 2. 高基数的风险
高基数字段会抬高聚合成本，  
带来更高内存占用和更长响应时间。

## 3. 治理策略
- 收敛维度数量与粒度
- 控制标签自由度
- 用预聚合或降采样降低查询成本

## 4. 工程建议
- 先定义指标字典，再落地写入规范。
- 查询侧限制默认时间范围，避免全量扫库。

# 总结
- 时序系统稳定性的核心是“维度治理”。
- 模型设计越前置，后期成本越低。

## 练习题
1. 给一个监控场景列出维度与指标清单。  
2. 找出可能高基数的字段并提出限制策略。  
3. 设计一版“实时 + 历史”分层查询方案。  

## 实战（curl）

```bash
# 时间窗口聚合
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/metrics-*/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{
    "size":0,
    "query":{"range":{"@timestamp":{"gte":"now-1h"}}},
    "aggs":{
      "by_host":{"terms":{"field":"host.keyword","size":20}},
      "avg_cpu":{"avg":{"field":"cpu"}}
    }
  }'
```

## 实战（Java SDK）

```java
client.search(s -> s.index("metrics-*").size(0)
    .query(q -> q.range(r -> r.field("@timestamp").gte("now-1h")))
    .aggregations("avg_cpu", a -> a.avg(v -> v.field("cpu"))), Map.class);
```
