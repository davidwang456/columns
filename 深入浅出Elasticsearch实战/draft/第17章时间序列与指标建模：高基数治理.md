

# 第17章 时间序列与指标建模：高基数治理

# 背景

监控指标、IoT 传感器数据、应用 APM 链路——这些时序数据天然具备"追加写入、按时间查询、聚合密集"的特征。Elasticsearch 从 8.x 开始提供 TSDB（Time Series Data Stream）模式，通过维度/指标字段分离、自动降采样、合成 `_source` 等手段，在存储效率和查询性能上实现了针对性优化。

然而，时序场景最难对付的问题不是写入量，而是**高基数**（High Cardinality）。当 `terms` 聚合的 unique values 达到百万甚至千万级别时，bucket 数爆炸会导致内存膨胀、GC 压力飙升、查询超时。本章从时序数据建模出发，讲透 TSDB 模式的底层优化，再深入高基数的识别与治理策略。

## 本章目标

- 理解 Data Stream 的设计原理：backing indices、自动 rollover、只追加语义。
- 掌握 TSDB 模式的配置与优化：维度字段、指标字段、合成 `_source`。
- 识别高基数问题的典型症状，建立系统化的治理策略。
- 学会使用 Downsampling 和 Transform 进行数据降采样与预聚合。
- 在日期直方图 + 多维聚合的组合中平衡精度与性能。

## 1. 时序数据的特点

### 1.1 写入模型

时序数据与业务数据的写入模型截然不同：

| 特征 | 业务数据（如订单） | 时序数据（如指标） |
| --- | --- | --- |
| 写入模式 | 随机插入/更新/删除 | 只追加（append-only） |
| 时间分布 | 无固定模式 | 严格按时间递增 |
| 查询模式 | 按 ID、按条件 | 按时间范围 + 维度过滤 |
| 聚合需求 | 偶尔聚合 | 核心操作，高频执行 |
| 生命周期 | 长期保留 | 有限保留，冷热分层 |

### 1.2 时序数据的三要素

每条时序数据由三部分组成：

- **时间戳**（Timestamp）：数据的采集时刻。
- **维度**（Dimensions）：标识数据来源的标签，如 `host`、`service`、`region`。
- **指标**（Metrics）：具体的度量值，如 `cpu_usage`、`memory_used`、`request_latency`。

```json
{
  "@timestamp": "2025-06-01T10:00:00Z",
  "host": "web-01",
  "service": "api-gateway",
  "region": "us-east-1",
  "cpu_usage": 72.5,
  "memory_used_bytes": 3221225472,
  "request_latency_ms": 45
}
```

## 2. Data Stream：时序数据的基础抽象

### 2.1 什么是 Data Stream

Data Stream 是 Elasticsearch 为时序数据设计的逻辑抽象，具有以下特性：

- **Backing Indices**：Data Stream 由一组隐藏的 backing indices 组成（命名格式 `.ds-<name>-<generation>`）。
- **自动 Rollover**：配合 ILM 策略自动创建新的 backing index。
- **只追加语义**：只支持 `create`（`op_type=create`）操作，不支持直接 `index`/`update`/`delete`。
- **内置 `@timestamp` 字段**：所有 Data Stream 必须包含 `@timestamp` 字段。

### 2.2 Data Stream 的内部结构

```
Data Stream: metrics-system
  │
  ├── .ds-metrics-system-000001 (只读, 已 rollover)
  ├── .ds-metrics-system-000002 (只读, 已 rollover)
  └── .ds-metrics-system-000003 (当前写入目标)
       │
       写入 ──────────────┘
       查询 ──────────── 所有 backing indices
```

写入只发往最新的 backing index，查询自动扇出到所有 backing indices。

### 2.3 创建 Data Stream

Data Stream 不需要显式创建——定义好 Index Template 后，第一次写入会自动创建：

```json
PUT _index_template/metrics_template
{
  "index_patterns": ["metrics-*"],
  "data_stream": {},
  "template": {
    "settings": {
      "number_of_shards": 1,
      "number_of_replicas": 1
    },
    "mappings": {
      "properties": {
        "@timestamp":         { "type": "date" },
        "host":               { "type": "keyword" },
        "service":            { "type": "keyword" },
        "cpu_usage":          { "type": "float" },
        "memory_used_bytes":  { "type": "long" },
        "request_latency_ms": { "type": "float" }
      }
    }
  }
}
```

## 3. TSDB 模式

### 3.1 启用 TSDB 模式

TSDB（Time Series Data Stream）是 Elasticsearch 8.x 引入的专用时序优化模式，通过 `index.mode: time_series` 启用：

```json
PUT _index_template/metrics_tsdb_template
{
  "index_patterns": ["metrics-tsdb-*"],
  "data_stream": {},
  "template": {
    "settings": {
      "index.mode": "time_series",
      "index.routing_path": ["host", "service"]
    },
    "mappings": {
      "properties": {
        "@timestamp":         { "type": "date" },
        "host":               { "type": "keyword", "time_series_dimension": true },
        "service":            { "type": "keyword", "time_series_dimension": true },
        "cpu_usage":          { "type": "float",   "time_series_metric": "gauge" },
        "memory_used_bytes":  { "type": "long",    "time_series_metric": "gauge" },
        "request_count":      { "type": "long",    "time_series_metric": "counter" }
      }
    }
  }
}
```

### 3.2 维度字段（time_series_dimension）

维度字段用于标识时间序列的唯一身份。同一组维度值定义了一条时间序列：

- 必须是 `keyword`、`ip`、`byte`、`short`、`integer`、`long` 或 `unsigned_long` 类型。
- 所有维度字段的组合构成时间序列的唯一标识（TSID）。
- 维度字段必须出现在 `index.routing_path` 中。

### 3.3 指标字段（time_series_metric）

指标字段声明度量值的语义类型：

| 类型 | 语义 | 降采样聚合方式 | 典型字段 |
| --- | --- | --- | --- |
| `gauge` | 可上可下的瞬时值 | min/max/sum/count/last_value | cpu_usage, temperature |
| `counter` | 单调递增的累计值 | last_value | request_count, bytes_sent |
| `position` | 地理位置 | 不聚合 | lat/lon |

### 3.4 TSDB 的底层优化

TSDB 模式启用后，Elasticsearch 自动应用以下优化：

| 优化项 | 说明 |
| --- | --- |
| TSID 路由 | 同一时间序列的文档路由到同一分片，提升聚合局部性 |
| 合成 `_source` | 不存储原始 JSON，从 doc_values 重建，节省 30~50% 存储 |
| 时间排序 | 同一 TSID 内按时间排序存储，提升范围查询效率 |
| 索引排序 | 自动启用 index sorting（按 TSID + @timestamp） |
| 降采样支持 | 可通过 ILM 自动将细粒度数据聚合为粗粒度 |

### 3.5 合成 _source（Synthetic Source）

TSDB 模式默认启用合成 `_source`——不存储原始 JSON 文档，而是在需要时从 doc_values 和其他列式存储中重建 `_source`。

优势：存储空间节省 30~50%。

限制：
- 重建的字段顺序可能与原始文档不同。
- 不支持 `_source` filtering 中的 `includes`/`excludes`。
- 某些字段类型（如 `text` 且不带 `keyword` 子字段）不支持合成。

## 4. 高基数问题深度分析

### 4.1 什么是高基数

高基数（High Cardinality）指某个字段的唯一值数量极大。例如：

| 字段 | 唯一值估计 | 基数级别 |
| --- | --- | --- |
| `status` | 5 | 极低 |
| `region` | 20 | 低 |
| `service` | 200 | 中等 |
| `user_id` | 10,000,000 | 高 |
| `trace_id` | 100,000,000 | 极高 |
| `client_ip` | 数千万 | 高 |

### 4.2 高基数的影响

当对高基数字段执行 `terms` 聚合时：

```
terms 聚合（size=10, shard_size=默认 1500）
  │
  每个分片构建 shard_size 个 bucket（消耗内存）
  │
  协调节点合并所有分片的 bucket
  │
  结果：
  - 内存占用 ∝ shard_size × 分片数 × bucket 大小
  - 网络传输 ∝ shard_size × 分片数
  - 如果 shard_size 不足以覆盖长尾分布 → 结果不准确
```

具体影响链：

1. **内存膨胀**：每个 bucket 在堆内存中维护，百万级 bucket 可能消耗数 GB heap。
2. **GC 压力**：大量短生命周期 bucket 对象触发频繁 GC。
3. **查询超时**：聚合计算耗时超过 `timeout` 设置。
4. **结果不准确**：`shard_size` 不够大时，长尾 bucket 被截断。
5. **断路器熔断**：超出 `indices.breaker.request.limit`（默认 heap 的 60%）触发 CircuitBreakingException。

### 4.3 典型高基数场景

| 场景 | 高基数字段 | 风险 |
| --- | --- | --- |
| 用户行为分析 | `user_id`, `session_id` | 用户量级直接决定基数 |
| 分布式链路追踪 | `trace_id`, `span_id` | 每个请求唯一 |
| 网络流量监控 | `src_ip`, `dst_ip`, `port` | IP 组合爆炸 |
| 自由标签系统 | `tags.*`（用户自定义键值） | 无法预测基数上限 |
| 日志分析 | `url.full`（含查询参数） | 参数变化导致唯一值爆炸 |

## 5. 高基数治理策略

### 5.1 建模阶段治理

**策略一：限制标签自由度**

不允许用户传入任意键值对作为标签。定义枚举白名单：

```json
{
  "mappings": {
    "properties": {
      "env":     { "type": "keyword" },
      "region":  { "type": "keyword" },
      "service": { "type": "keyword" }
    }
  }
}
```

**策略二：枚举化高基数字段**

将连续值分桶，降低基数：

```json
{
  "script": {
    "source": """
      double latency = ctx._source.latency_ms;
      if (latency < 100) ctx._source.latency_bucket = 'fast';
      else if (latency < 500) ctx._source.latency_bucket = 'normal';
      else ctx._source.latency_bucket = 'slow';
    """
  }
}
```

**策略三：分层建模**

将高基数字段拆分为多个低基数维度：

```
原始：url = "/api/v1/users/12345/orders/67890"

拆分后：
  url_path_template = "/api/v1/users/{id}/orders/{id}"
  url_resource = "orders"
  url_version = "v1"
```

**策略四：控制 Dynamic Mapping**

禁止自动创建字段，防止字段爆炸：

```json
{
  "mappings": {
    "dynamic": "strict"
  }
}
```

### 5.2 查询阶段治理

**策略一：限制时间范围**

永远带上时间过滤，减少扫描的分片和文档数：

```json
{
  "query": {
    "bool": {
      "filter": [
        { "range": { "@timestamp": { "gte": "now-1h", "lt": "now" } } },
        { "term":  { "service": "api-gateway" } }
      ]
    }
  },
  "aggs": {
    "top_hosts": { "terms": { "field": "host", "size": 20 } }
  }
}
```

**策略二：控制 shard_size 和 size**

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `size` | `10` | 返回的 top N 个 bucket |
| `shard_size` | `size * 1.5 + 10` | 每个分片返回的候选 bucket 数 |

减小 `shard_size` 可以降低内存和网络开销，但可能牺牲准确性：

```json
{
  "aggs": {
    "top_hosts": {
      "terms": {
        "field": "host",
        "size": 10,
        "shard_size": 50
      }
    }
  }
}
```

**策略三：用 composite 聚合分批获取**

`composite` 聚合支持翻页，不会一次性把所有 bucket 加载到内存：

```json
{
  "size": 0,
  "aggs": {
    "by_host_service": {
      "composite": {
        "size": 100,
        "sources": [
          { "host":    { "terms": { "field": "host" } } },
          { "service": { "terms": { "field": "service" } } }
        ]
      }
    }
  }
}
```

下一页使用 `after` 参数：

```json
{
  "size": 0,
  "aggs": {
    "by_host_service": {
      "composite": {
        "size": 100,
        "sources": [
          { "host":    { "terms": { "field": "host" } } },
          { "service": { "terms": { "field": "service" } } }
        ],
        "after": { "host": "web-99", "service": "api-gateway" }
      }
    }
  }
}
```

**策略四：使用 Cardinality 聚合预判基数**

在执行高开销的 `terms` 聚合前，先用 `cardinality` 估算唯一值数量：

```json
{
  "size": 0,
  "aggs": {
    "host_cardinality": {
      "cardinality": {
        "field": "host",
        "precision_threshold": 3000
      }
    }
  }
}
```

`precision_threshold`（默认 `3000`，最大 `40000`）：低于此值时精度接近 100%，超过时误差率约 1~6%。

### 5.3 存储阶段治理

**策略一：降采样（Downsampling）**

将细粒度数据聚合为粗粒度，如 1 分钟 → 1 小时：

- 原始数据：每分钟每个 host 一条记录 → 1440 条/天/host
- 降采样后：每小时每个 host 一条记录 → 24 条/天/host
- 压缩比：60:1

**策略二：预聚合（Transform）**

使用 Transform 将明细数据持续聚合到汇总索引：

```json
PUT _transform/service_hourly_stats
{
  "source": {
    "index": "metrics-*",
    "query": { "match_all": {} }
  },
  "dest": { "index": "metrics-hourly-summary" },
  "pivot": {
    "group_by": {
      "date_hour": { "date_histogram": { "field": "@timestamp", "fixed_interval": "1h" } },
      "service":   { "terms": { "field": "service" } },
      "host":      { "terms": { "field": "host" } }
    },
    "aggregations": {
      "avg_cpu":       { "avg": { "field": "cpu_usage" } },
      "max_cpu":       { "max": { "field": "cpu_usage" } },
      "p99_latency":   { "percentiles": { "field": "request_latency_ms", "percents": [99] } },
      "request_count": { "value_count": { "field": "@timestamp" } }
    }
  },
  "frequency": "1h",
  "sync": {
    "time": {
      "field": "@timestamp",
      "delay": "5m"
    }
  }
}
```

## 6. Downsampling 详解

### 6.1 Downsampling 的原理

Downsampling 将 TSDB 索引中的细粒度时序数据聚合为粗粒度。对于每个时间序列（由 TSID 标识），在每个降采样时间窗口内：

- `gauge` 类型指标：计算 `min`、`max`、`sum`、`value_count`。
- `counter` 类型指标：保留 `last_value`。
- 维度字段：保持不变。
- `@timestamp`：设为时间窗口的起始时间。

### 6.2 手动执行 Downsampling

```json
POST metrics-tsdb-prod/_downsample/metrics-tsdb-prod-1h
{
  "fixed_interval": "1h"
}
```

前置条件：源索引必须是只读的（已 rollover 或手动设为 readonly）。

### 6.3 配合 ILM 自动降采样

在 ILM 策略的 warm 或 cold phase 中配置 `downsample` action：

```json
PUT _ilm/policy/metrics_tsdb_policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": { "max_age": "1d" },
          "set_priority": { "priority": 100 }
        }
      },
      "warm": {
        "min_age": "7d",
        "actions": {
          "downsample": { "fixed_interval": "1h" },
          "set_priority": { "priority": 50 }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "downsample": { "fixed_interval": "1d" },
          "set_priority": { "priority": 0 }
        }
      },
      "delete": {
        "min_age": "180d",
        "actions": { "delete": {} }
      }
    }
  }
}
```

### 6.4 Downsampling 的存储收益

| 原始间隔 | 降采样间隔 | 压缩比（数据量） | 适用查询范围 |
| --- | --- | --- | --- |
| 10s | 1min | ~6:1 | 最近 1 小时 |
| 10s | 1h | ~360:1 | 最近 7 天 |
| 10s | 1d | ~8640:1 | 最近 30 天以上 |

## 7. Transform：持续预聚合

### 7.1 Transform 的工作模式

| 模式 | 说明 |
| --- | --- |
| `batch` | 一次性执行，适合历史数据回填 |
| `continuous` | 持续运行，增量处理新数据 |

### 7.2 Transform 的核心参数

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `frequency` | `1m` | 检查新数据的间隔 |
| `sync.time.delay` | `60s` | 等待迟到数据的延迟窗口 |
| `max_page_search_size` | `500` | 每次聚合的 composite bucket 数 |
| `docs_per_second` | 无限制 | 处理速率限制 |

### 7.3 Transform vs Downsampling

| 维度 | Downsampling | Transform |
| --- | --- | --- |
| 适用对象 | TSDB 模式索引 | 任意索引 |
| 操作类型 | 时间维度压缩 | 自由定义聚合 |
| 输出位置 | 替换原索引 | 独立目标索引 |
| 执行模式 | 一次性（配合 ILM） | 可持续运行 |
| 灵活性 | 低（仅时间维度） | 高（任意 pivot） |

## 8. 查询优化：date_histogram + terms 组合

### 8.1 性能考量

`date_histogram` + `terms` 的嵌套聚合是时序查询的最常见模式，也是最容易出性能问题的地方：

```json
{
  "size": 0,
  "query": { "range": { "@timestamp": { "gte": "now-7d" } } },
  "aggs": {
    "per_hour": {
      "date_histogram": {
        "field": "@timestamp",
        "fixed_interval": "1h"
      },
      "aggs": {
        "by_service": {
          "terms": {
            "field": "service",
            "size": 20
          },
          "aggs": {
            "avg_cpu": { "avg": { "field": "cpu_usage" } }
          }
        }
      }
    }
  }
}
```

**bucket 数估算**：时间 bucket 数 × terms bucket 数 = 168 × 20 = 3360 个 bucket。如果 terms 的基数更高或时间范围更大，bucket 数将指数级增长。

### 8.2 优化策略

| 优化手段 | 做法 | 效果 |
| --- | --- | --- |
| 缩小时间范围 | `now-1h` 替代 `now-7d` | 减少时间 bucket 数 |
| 增大时间间隔 | `1d` 替代 `1h` | 减少时间 bucket 数 |
| 减小 terms size | `size: 10` 替代 `size: 100` | 减少每个时间 bucket 的子 bucket |
| 前置 filter | 先过滤再聚合 | 减少参与聚合的文档数 |
| 查询预聚合索引 | 查 Transform 输出的汇总索引 | 大幅减少文档量 |
| 使用 sampler | 在聚合前采样 | 牺牲精度换速度 |

### 8.3 使用 filter 聚合收窄范围

```json
{
  "size": 0,
  "aggs": {
    "recent": {
      "filter": {
        "range": { "@timestamp": { "gte": "now-1h" } }
      },
      "aggs": {
        "by_service": {
          "terms": { "field": "service", "size": 10 },
          "aggs": {
            "avg_cpu": { "avg": { "field": "cpu_usage" } }
          }
        }
      }
    }
  }
}
```

## 9. 实战：监控指标场景设计

### 9.1 数据模型

设计一个主机级别的系统监控场景：

| 字段 | 类型 | 维度/指标 | 说明 |
| --- | --- | --- | --- |
| `@timestamp` | `date` | — | 采集时间 |
| `host` | `keyword` | 维度 | 主机名 |
| `service` | `keyword` | 维度 | 服务名 |
| `region` | `keyword` | 维度 | 区域 |
| `cpu_usage` | `float` | gauge | CPU 使用率 |
| `memory_used_bytes` | `long` | gauge | 内存使用量 |
| `disk_io_ops` | `long` | counter | 磁盘 IO 操作累计 |
| `request_latency_ms` | `float` | gauge | 请求延迟 |

### 9.2 基数评估

| 维度 | 预估基数 | 风险 |
| --- | --- | --- |
| `host` | 500 | 低 |
| `service` | 50 | 极低 |
| `region` | 5 | 极低 |
| `host × service` | 2,500 | 低 |
| 全维度组合 | ~12,500 | 可接受 |

# 总结

- **Data Stream** 是时序数据的基础管理单元——自动 rollover、只追加、内置 `@timestamp`，消除手工索引管理的负担。
- **TSDB 模式** 通过维度/指标字段分离、TSID 路由、合成 `_source` 和索引排序，实现 30~50% 的存储压缩和显著的聚合性能提升。
- **高基数** 是时序聚合的头号杀手——terms 聚合的 bucket 数 × 分片数决定了内存消耗，超过阈值会触发断路器熔断。
- 高基数治理需要**全链路思维**：建模阶段限制自由度、查询阶段控制 size/shard_size、存储阶段降采样/预聚合。
- **Downsampling** 适合 TSDB 索引的时间维度压缩，**Transform** 适合任意维度的灵活预聚合——两者互补而非替代。
- `date_histogram + terms` 组合查询的 bucket 数是性能瓶颈的核心指标——缩小时间范围、增大间隔、前置过滤是最有效的优化手段。

---

## 练习题

1. 创建一个 TSDB 模式的 Data Stream，包含 `host` 和 `service` 两个维度字段和 `cpu_usage`（gauge）、`request_count`（counter）两个指标字段。
2. 写入 100 条模拟时序数据，然后使用 `cardinality` 聚合估算 `host` 字段的基数。
3. 对比 `terms` 聚合在 `size=10` 和 `size=1000` 时的响应时间和内存消耗。
4. 使用 `composite` 聚合分批获取所有 `host × service` 的组合及其平均 CPU 使用率。
5. 创建一个 continuous Transform，将分钟级指标数据汇总为小时级，验证目标索引的数据量压缩比。

## 实战（Java SDK）

```java
// ---------- 创建 TSDB Index Template ----------
client.indices().putIndexTemplate(t -> t
    .name("metrics_tsdb_template")
    .indexPatterns("metrics-tsdb-*")
    .dataStream(ds -> ds)
    .template(tmpl -> tmpl
        .settings(s -> s
            .mode(IndexMode.TimeSeries)
            .routingPath("host", "service")
            .numberOfShards("1")
            .numberOfReplicas("0"))
        .mappings(m -> m
            .properties("@timestamp", p -> p.date(d -> d))
            .properties("host", p -> p.keyword(k -> k.timeSeriesDimension(true)))
            .properties("service", p -> p.keyword(k -> k.timeSeriesDimension(true)))
            .properties("cpu_usage", p -> p.float_(f -> f.timeSeriesMetric(TimeSeriesMetricType.Gauge)))
            .properties("memory_used_bytes", p -> p.long_(l -> l.timeSeriesMetric(TimeSeriesMetricType.Gauge)))
            .properties("request_count", p -> p.long_(l -> l.timeSeriesMetric(TimeSeriesMetricType.Counter)))
            .properties("request_latency_ms", p -> p.float_(f -> f.timeSeriesMetric(TimeSeriesMetricType.Gauge))))));

// ---------- 写入时序数据 ----------
var bulkResp = client.bulk(b -> b
    .operations(op -> op.create(c -> c
        .index("metrics-tsdb-system")
        .document(Map.of(
            "@timestamp", "2025-06-01T10:00:00Z",
            "host", "web-01", "service", "api-gw",
            "cpu_usage", 72.5, "memory_used_bytes", 3221225472L,
            "request_count", 15000, "request_latency_ms", 45.2))))
    .operations(op -> op.create(c -> c
        .index("metrics-tsdb-system")
        .document(Map.of(
            "@timestamp", "2025-06-01T10:01:00Z",
            "host", "web-01", "service", "api-gw",
            "cpu_usage", 68.3, "memory_used_bytes", 3200000000L,
            "request_count", 15100, "request_latency_ms", 42.1))))
    .operations(op -> op.create(c -> c
        .index("metrics-tsdb-system")
        .document(Map.of(
            "@timestamp", "2025-06-01T10:00:00Z",
            "host", "web-02", "service", "order-svc",
            "cpu_usage", 55.1, "memory_used_bytes", 2147483648L,
            "request_count", 12000, "request_latency_ms", 38.7)))));

if (bulkResp.errors()) {
    bulkResp.items().stream()
        .filter(item -> item.error() != null)
        .forEach(item -> System.err.println("Failed: " + item.error().reason()));
}

// ---------- 基数预估 ----------
var cardResp = client.search(s -> s
    .index("metrics-tsdb-system")
    .size(0)
    .aggregations("host_cardinality", a -> a
        .cardinality(c -> c.field("host").precisionThreshold(3000))),
    Void.class);

System.out.println("Host cardinality: "
    + cardResp.aggregations().get("host_cardinality").cardinality().value());

// ---------- terms 聚合 ----------
var termsResp = client.search(s -> s
    .index("metrics-tsdb-system")
    .size(0)
    .query(q -> q.range(r -> r
        .field("@timestamp")
        .gte(JsonData.of("2025-06-01T10:00:00Z"))
        .lt(JsonData.of("2025-06-01T11:00:00Z"))))
    .aggregations("by_host", a -> a
        .terms(t -> t.field("host").size(10))
        .aggregations("avg_cpu", sub -> sub
            .avg(avg -> avg.field("cpu_usage")))),
    Void.class);

termsResp.aggregations().get("by_host").sterms().buckets().array().forEach(bucket ->
    System.out.println("Host: " + bucket.key().stringValue()
        + ", avg_cpu: " + bucket.aggregations().get("avg_cpu").avg().value()));

// ---------- composite 聚合分批获取 ----------
Map<String, String> afterKey = null;
do {
    final Map<String, String> currentAfter = afterKey;
    var compResp = client.search(s -> {
        var comp = s.index("metrics-tsdb-system")
            .size(0)
            .aggregations("by_dims", a -> a
                .composite(c -> {
                    var builder = c.size(2)
                        .sources(
                            src -> src.put("host", cs -> cs.terms(t -> t.field("host"))),
                            src -> src.put("service", cs -> cs.terms(t -> t.field("service"))));
                    if (currentAfter != null) {
                        builder.after(currentAfter.entrySet().stream()
                            .collect(java.util.stream.Collectors.toMap(
                                Map.Entry::getKey,
                                e -> FieldValue.of(e.getValue()))));
                    }
                    return builder;
                })
                .aggregations("avg_cpu", sub -> sub.avg(avg -> avg.field("cpu_usage"))));
        return comp;
    }, Void.class);

    var compAgg = compResp.aggregations().get("by_dims").composite();
    compAgg.buckets().array().forEach(bucket ->
        System.out.println("Dims: " + bucket.key()
            + ", avg_cpu: " + bucket.aggregations().get("avg_cpu").avg().value()));

    afterKey = compAgg.afterKey() != null
        ? compAgg.afterKey().entrySet().stream()
            .collect(java.util.stream.Collectors.toMap(
                Map.Entry::getKey, e -> e.getValue().stringValue()))
        : null;
} while (afterKey != null);

// ---------- 查看 Data Stream ----------
var dsResp = client.indices().getDataStream(g -> g.name("metrics-tsdb-system"));
dsResp.dataStreams().forEach(ds ->
    System.out.println("Data stream: " + ds.name()
        + ", backing indices: " + ds.indices().size()));

// ---------- 清理 ----------
client.indices().deleteDataStream(d -> d.name("metrics-tsdb-system"));
client.indices().deleteIndexTemplate(d -> d.name("metrics_tsdb_template"));
```
