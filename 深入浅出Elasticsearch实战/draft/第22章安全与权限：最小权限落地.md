

# 第22章 安全与权限：最小权限落地

# 背景

Elasticsearch 默认在 8.x 版本中开启安全功能，但"开启安全"只是起点。生产环境中，不同团队、不同应用对集群的访问范围差异巨大——开发人员只需读写特定索引，运维人员需要监控集群但不应修改业务数据，分析师只需查询特定字段。如果所有人都用 `elastic` 超级管理员账号，一次误操作就可能删除整个生产索引。本章从 ES 安全体系的三层架构出发，讲透角色设计、API Key 管理、字段/文档级安全，以及生产环境的安全加固清单。

## 本章目标

- 理解 ES 安全体系的三层架构：认证、授权、传输加密。
- 掌握 Role-Based Access Control（RBAC）的角色定义，包括 cluster privileges、index privileges、field_security 和 document-level security。
- 设计三类典型角色（开发、运维、分析），并落地到 API 调用。
- 掌握 API Key 的创建、管理与最佳实践。
- 建立生产环境安全配置的标准 checklist。

## 1. ES 安全体系三层架构

ES 的安全模型可以分为三层，每层解决一个核心问题：

```
第一层：认证（Authentication）
  │  "你是谁？" — 验证用户身份
  │  支持：内置用户、Native Realm、LDAP/AD、SAML、OIDC
  ▼
第二层：授权（Authorization）
  │  "你能做什么？" — 基于角色控制权限
  │  支持：RBAC、字段级安全、文档级安全
  ▼
第三层：传输加密（Encryption in Transit）
  │  "通信是否安全？" — TLS/SSL 加密
  │  覆盖：HTTP 层（客户端↔ES）+ Transport 层（节点↔节点）
```

### 1.1 认证（Authentication）

认证回答"你是谁"。ES 支持多种认证方式（称为 Realm），按优先级链式匹配：

| Realm 类型 | 说明 | 适用场景 |
| --- | --- | --- |
| `native` | ES 内置用户存储（存在 `.security` 索引中） | 小团队、开发/测试环境 |
| `file` | 本地文件 `users`/`users_roles` | 紧急恢复、bootstrap |
| `ldap` | 对接 LDAP/Active Directory | 企业统一身份管理 |
| `saml` | SAML 2.0 单点登录 | Kibana + 企业 SSO |
| `oidc` | OpenID Connect | 对接 Keycloak、Okta 等 |
| `pki` | 客户端证书认证 | 节点间通信、高安全场景 |

内置用户（安装时自动创建）：

| 用户名 | 用途 | 备注 |
| --- | --- | --- |
| `elastic` | 超级管理员 | 拥有 `superuser` 角色，仅用于初始设置 |
| `kibana_system` | Kibana 服务账户 | 用于 Kibana 连接 ES |
| `beats_system` | Beats 服务账户 | 用于 Filebeat/Metricbeat 等 |
| `logstash_system` | Logstash 监控账户 | 用于 Logstash 上报监控数据 |
| `remote_monitoring_user` | 远程监控 | 跨集群监控场景 |

**关键原则**：生产环境绝不应该用 `elastic` 账号做日常操作。`elastic` 只用于初始配置和紧急恢复。

### 1.2 授权（Authorization）

授权回答"你能做什么"。ES 采用 Role-Based Access Control（RBAC），核心概念：

- **Role**：定义一组权限（cluster 级 + index 级）。
- **User**：一个用户可以拥有多个 Role，权限取并集。
- **Role Mapping**：将外部身份（LDAP 组、SAML 属性）映射到 ES Role。

### 1.3 传输加密（TLS/SSL）

ES 8.x 默认启用 TLS，覆盖两个层面：

| 层面 | 配置前缀 | 保护内容 |
| --- | --- | --- |
| Transport 层 | `xpack.security.transport.ssl.*` | 节点间通信（分片复制、集群状态同步） |
| HTTP 层 | `xpack.security.http.ssl.*` | 客户端与 ES 之间的 REST 通信 |

Transport 层 TLS 是**强制要求**——没有它，节点间通信明文传输，存在中间人攻击风险。HTTP 层 TLS 在生产环境中也应始终开启。

## 2. 角色（Role）详细结构

一个 Role 的完整结构如下：

```json
{
  "cluster": ["monitor"],
  "indices": [
    {
      "names": ["logs-*"],
      "privileges": ["read", "view_index_metadata"],
      "field_security": {
        "grant": ["timestamp", "message", "level", "service"]
      },
      "query": "{\"term\": {\"service\": \"payment\"}}"
    }
  ],
  "applications": [],
  "run_as": [],
  "metadata": {}
}
```

### 2.1 Cluster Privileges

控制集群级操作的权限：

| Privilege | 允许的操作 |
| --- | --- |
| `monitor` | 只读集群状态（`_cluster/health`、`_cat/*`、`_nodes/stats`） |
| `manage` | 集群管理操作（settings、templates、pipeline 等） |
| `manage_security` | 管理用户、角色、API Key |
| `manage_index_templates` | 管理索引模板 |
| `manage_ilm` | 管理 ILM 策略 |
| `manage_pipeline` | 管理 Ingest Pipeline |
| `all` | 所有集群操作（等同于超级管理员的集群权限） |

### 2.2 Index Privileges

控制索引级操作的权限：

| Privilege | 允许的操作 |
| --- | --- |
| `read` | 搜索、获取文档、聚合 |
| `write` | 写入文档（index、bulk、update、delete） |
| `create_index` | 创建索引 |
| `delete_index` | 删除索引 |
| `manage` | 索引管理（settings、mapping、aliases、force_merge） |
| `view_index_metadata` | 查看索引 metadata（`_mapping`、`_settings`，但不能读数据） |
| `create` | 仅创建文档（不能更新已有文档） |
| `index` | 创建和更新文档 |
| `monitor` | 查看索引统计（`_stats`、`_segments`） |
| `all` | 所有索引操作 |

### 2.3 Field Security（字段级安全）

控制用户能看到哪些字段。有两种模式：

**白名单模式（grant）**：只允许看到指定字段：

```json
"field_security": {
  "grant": ["timestamp", "message", "level"]
}
```

**黑名单模式（except）**：排除敏感字段（需配合 grant 使用）：

```json
"field_security": {
  "grant": ["*"],
  "except": ["credit_card", "ssn", "password_hash"]
}
```

### 2.4 Document-Level Security（文档级安全）

通过 `query` 字段限制用户只能看到满足条件的文档：

```json
{
  "names": ["orders-*"],
  "privileges": ["read"],
  "query": "{\"term\": {\"region\": \"asia\"}}"
}
```

该用户通过 `orders-*` 索引搜索时，ES 会自动附加这个 filter，只返回 `region=asia` 的订单。

## 3. 三类典型角色设计

### 3.1 开发角色：特定索引的读写

开发人员需要读写自己负责的业务索引，但不应有集群管理权限，也不能访问其他团队的索引。

```json
{
  "cluster": [],
  "indices": [
    {
      "names": ["products-*", "orders-*"],
      "privileges": ["read", "write", "view_index_metadata"]
    }
  ]
}
```

特点：
- 无 cluster privileges——不能查看集群状态或修改集群配置。
- 只能操作 `products-*` 和 `orders-*` 索引。
- 不能创建/删除索引（无 `create_index` / `delete_index`）。

### 3.2 运维角色：监控 + 管理

运维人员需要监控集群健康、管理索引生命周期、调整 settings，但不应读写业务数据。

```json
{
  "cluster": ["monitor", "manage", "manage_ilm", "manage_index_templates"],
  "indices": [
    {
      "names": ["*"],
      "privileges": ["monitor", "manage", "view_index_metadata"]
    }
  ]
}
```

特点：
- 有 cluster 级 `monitor` 和 `manage` 权限。
- 所有索引的 `manage` 权限（可以调 settings、执行 force_merge 等）。
- 没有 `read` / `write`——不能查看或修改业务文档内容。

### 3.3 分析角色：只读 + 字段限制

数据分析师只需查询特定索引，且不应看到敏感字段（如用户手机号、身份证号）。

```json
{
  "cluster": ["monitor"],
  "indices": [
    {
      "names": ["orders-*"],
      "privileges": ["read"],
      "field_security": {
        "grant": ["*"],
        "except": ["customer_phone", "customer_id_card"]
      },
      "query": "{\"range\": {\"created_at\": {\"gte\": \"now-90d\"}}}"
    }
  ]
}
```

特点：
- 只读权限，无写入能力。
- 通过 `field_security` 屏蔽敏感字段。
- 通过 document-level security 限制只能查询最近 90 天的数据。

## 4. API Key 详解

API Key 是一种轻量级的认证凭证，适合应用程序接入 ES。与用户名密码相比，API Key 可以精细控制权限范围和有效期。

### 4.1 创建 API Key

```json
POST /_security/api_key
{
  "name": "order-service-key",
  "expiration": "30d",
  "role_descriptors": {
    "order_writer": {
      "cluster": [],
      "indices": [
        {
          "names": ["orders-*"],
          "privileges": ["read", "write"]
        }
      ]
    }
  }
}
```

响应：

```json
{
  "id": "VuaCfGcBCdbkQm-e5aOx",
  "name": "order-service-key",
  "api_key": "ui2lp2axTNmsyakw9tvNnw",
  "encoded": "VnVhQ2ZHY0JDZGJrUW0tZTVhT3g6dWkybHAyYXhUTm1zeWFrdzl0dk5udw==",
  "expiration": 1735689600000
}
```

使用方式——在 HTTP 请求头中传入：

```
Authorization: ApiKey VnVhQ2ZHY0JDZGJrUW0tZTVhT3g6dWkybHAyYXhUTm1zeWFrdzl0dk5udw==
```

### 4.2 API Key vs Service Account Token

| 维度 | API Key | Service Account Token |
| --- | --- | --- |
| 用途 | 应用程序接入 ES | Elastic 内部服务通信（如 Kibana、Fleet） |
| 创建方式 | `_security/api_key` | `_security/service/<ns>/<service>/credential/token` |
| 权限控制 | 通过 `role_descriptors` 自定义 | 由服务账户预定义 |
| 可过期 | 支持 `expiration` | 不过期（需手动删除） |
| 适用场景 | 微服务、ETL 管道、CI/CD | Kibana 连接 ES、Fleet Server |

### 4.3 API Key 最佳实践

1. **每个应用一个 API Key**——不共享，出问题时可以单独吊销。
2. **设置过期时间**——推荐 30~90 天，到期前轮换。
3. **最小权限**——`role_descriptors` 中只授予应用实际需要的权限。
4. **定期清理**——通过 `_security/api_key?owner=true` 查询并清理过期 Key。
5. **审计日志**——开启审计日志追踪 API Key 的使用情况。

### 4.4 查询和吊销 API Key

```bash
# 查询当前用户创建的所有 API Key
GET /_security/api_key?owner=true

# 按名称查询
GET /_security/api_key?name=order-service-key

# 吊销指定 API Key
DELETE /_security/api_key
{
  "ids": ["VuaCfGcBCdbkQm-e5aOx"]
}
```

## 5. 安全配置 Checklist

生产环境上线前的安全检查清单：

| # | 检查项 | 说明 |
| --- | --- | --- |
| 1 | 启用 `xpack.security.enabled: true` | 8.x 默认开启，确认未被手动关闭 |
| 2 | 配置 Transport TLS | 节点间通信加密，防中间人攻击 |
| 3 | 配置 HTTP TLS | 客户端通信加密，防凭证泄露 |
| 4 | 修改内置用户密码 | `elasticsearch-reset-password` 工具重置 |
| 5 | 创建应用专用用户/API Key | 禁止应用使用 `elastic` 账号 |
| 6 | 禁用匿名访问 | 确认 `xpack.security.authc.anonymous` 未开启 |
| 7 | 网络隔离 | ES 端口不暴露到公网，通过网关/VPN 访问 |
| 8 | 开启审计日志 | 追踪登录、权限拒绝、敏感操作 |
| 9 | 定期轮换凭证 | API Key 设过期时间，密码定期更换 |
| 10 | 最小权限原则 | 每个角色只授予必要的最小权限集 |

### 5.1 配置 TLS 的最小 elasticsearch.yml

```yaml
xpack.security.enabled: true

xpack.security.transport.ssl:
  enabled: true
  verification_mode: certificate
  keystore.path: elastic-certificates.p12
  truststore.path: elastic-certificates.p12

xpack.security.http.ssl:
  enabled: true
  keystore.path: http.p12
```

### 5.2 生成证书

```bash
# 生成 CA
bin/elasticsearch-certutil ca

# 生成节点证书
bin/elasticsearch-certutil cert --ca elastic-stack-ca.p12

# 生成 HTTP 证书
bin/elasticsearch-certutil http
```

## 6. 审计日志（Audit Logging）

审计日志记录所有安全相关事件，用于合规审查和安全事件追溯。

### 6.1 启用审计日志

在 `elasticsearch.yml` 中配置：

```yaml
xpack.security.audit.enabled: true
```

审计日志默认写入 `<ES_HOME>/logs/<cluster_name>_audit.json`。

### 6.2 审计事件类型

| 事件 | 说明 |
| --- | --- |
| `authentication_success` | 认证成功 |
| `authentication_failed` | 认证失败（错误密码/不存在的用户） |
| `access_denied` | 授权失败（无权限） |
| `access_granted` | 授权成功 |
| `anonymous_access_denied` | 匿名访问被拒绝 |
| `connection_granted` | IP 过滤允许连接 |
| `connection_denied` | IP 过滤拒绝连接 |

### 6.3 过滤审计事件

默认记录所有事件会产生大量日志。推荐只记录关键事件：

```yaml
xpack.security.audit.enabled: true
xpack.security.audit.logfile.events.include:
  - access_denied
  - authentication_failed
  - connection_denied
xpack.security.audit.logfile.events.exclude:
  - access_granted
```

### 6.4 审计日志示例

```json
{
  "type": "audit",
  "timestamp": "2025-06-15T10:30:00.000Z",
  "event.action": "access_denied",
  "user.name": "dev_user",
  "user.roles": ["developer"],
  "request.name": "DeleteIndexAction",
  "indices": ["orders-2025"],
  "opaque_id": "abc123"
}
```

这条日志清楚记录了 `dev_user` 试图删除 `orders-2025` 索引但权限不足被拒绝。

## 7. 常见安全反模式

| 反模式 | 正确做法 |
| --- | --- |
| 所有应用共用 `elastic` 账号 | 每个应用创建专用用户或 API Key |
| 角色使用 `all` privilege | 按需精确授权（read、write、monitor 等） |
| 索引名用 `*` 通配 | 限定到具体索引模式（如 `orders-*`） |
| API Key 永不过期 | 设置 30~90 天过期，定期轮换 |
| Transport 层不加密 | 必须启用 Transport TLS |
| 忽略审计日志 | 至少记录 `access_denied` 和 `authentication_failed` |
| 把密码硬编码在应用代码中 | 使用 ES Keystore 或外部密钥管理服务 |

# 总结

- ES 安全体系分为三层：**认证（谁）→ 授权（权限）→ 加密（通道）**，缺一不可。
- 角色设计遵循最小权限原则——开发只读写自己的索引，运维只管理不读数据，分析师只读且屏蔽敏感字段。
- API Key 是应用接入 ES 的推荐方式——每个应用一个 Key，设过期时间，权限精确到索引和操作类型。
- Field Security 和 Document-Level Security 是数据隔离的两把利器——前者控制"看到哪些列"，后者控制"看到哪些行"。
- 生产上线前必须走完安全 checklist——TLS 加密、密码重置、专用账号、审计日志缺一不可。

## 实战（curl）

### 创建角色

```bash
# 开发角色：只能读写 products-* 和 orders-* 索引
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/_security/role/developer" \
  -H "Content-Type: application/json" \
  -d '{
    "cluster": [],
    "indices": [
      {
        "names": ["products-*", "orders-*"],
        "privileges": ["read", "write", "view_index_metadata"]
      }
    ]
  }'

# 运维角色：集群监控 + 管理，所有索引的管理权限，但不能读写数据
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/_security/role/ops_admin" \
  -H "Content-Type: application/json" \
  -d '{
    "cluster": ["monitor", "manage", "manage_ilm", "manage_index_templates"],
    "indices": [
      {
        "names": ["*"],
        "privileges": ["monitor", "manage", "view_index_metadata"]
      }
    ]
  }'

# 分析角色：只读 orders-*，屏蔽敏感字段，限制只能看最近 90 天数据
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/_security/role/analyst" \
  -H "Content-Type: application/json" \
  -d '{
    "cluster": ["monitor"],
    "indices": [
      {
        "names": ["orders-*"],
        "privileges": ["read"],
        "field_security": {
          "grant": ["*"],
          "except": ["customer_phone", "customer_id_card"]
        },
        "query": "{\"range\": {\"created_at\": {\"gte\": \"now-90d\"}}}"
      }
    ]
  }'

# 查看所有自定义角色
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_security/role?pretty" | head -60
```

```bash
# 开发用户：可以读写 orders-*
echo "=== dev_user: read orders ==="
curl -u "dev_user:Dev@12345678" "$ES_URL/orders-2025/_search?pretty&size=1"

# 开发用户：不能查看集群状态（无 cluster 权限）
echo "=== dev_user: cluster health (should fail) ==="
curl -u "dev_user:Dev@12345678" "$ES_URL/_cluster/health?pretty"

# 运维用户：可以查看集群状态
echo "=== ops_user: cluster health ==="
curl -u "ops_user:Ops@12345678" "$ES_URL/_cluster/health?pretty"

# 运维用户：不能读取文档（无 read 权限）
echo "=== ops_user: read orders (should fail) ==="
curl -u "ops_user:Ops@12345678" "$ES_URL/orders-2025/_search?pretty&size=1"

# 分析用户：可以读取但看不到敏感字段
echo "=== analyst_user: read orders (no sensitive fields) ==="
curl -u "analyst_user:Ana@12345678" -X POST "$ES_URL/orders-2025/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{ "size": 5, "_source": true }'
```

## 实战（Java SDK）

```java
// ---------- 创建角色 ----------
client.security().putRole(r -> r
    .name("developer")
    .indices(i -> i
        .names("products-*", "orders-*")
        .privileges("read", "write", "view_index_metadata")));

client.security().putRole(r -> r
    .name("analyst")
    .cluster("monitor")
    .indices(i -> i
        .names("orders-*")
        .privileges("read")
        .fieldSecurity(fs -> fs
            .grant("*")
            .except_("customer_phone", "customer_id_card"))
        .query("{\"range\": {\"created_at\": {\"gte\": \"now-90d\"}}}")));

// ---------- 创建用户 ----------
client.security().putUser(u -> u
    .username("dev_user")
    .password("Dev@12345678")
    .roles("developer")
    .fullName("Developer User")
    .email("dev@example.com"));

// ---------- 创建 API Key ----------
var apiKeyResp = client.security().createApiKey(k -> k
    .name("order-service-key")
    .expiration(t -> t.time("30d"))
    .roleDescriptors("order_rw", rd -> rd
        .indices(i -> i
            .names("orders-*")
            .privileges("read", "write"))));

System.out.println("API Key ID: " + apiKeyResp.id());
System.out.println("API Key encoded: " + apiKeyResp.encoded());

// ---------- 查询 API Key ----------
var apiKeys = client.security().getApiKey(g -> g.owner(true));
apiKeys.apiKeys().forEach(key ->
    System.out.println("Key: " + key.name() + ", expires: " + key.expiration()));

// ---------- 吊销 API Key ----------
client.security().invalidateApiKey(inv -> inv.ids(apiKeyResp.id()));

// ---------- 查看角色 ----------
var role = client.security().getRole(g -> g.name("developer"));
System.out.println("Developer role indices: " + role.get("developer").indices());

// ---------- 清理 ----------
client.security().deleteRole(d -> d.name("developer"));
client.security().deleteRole(d -> d.name("analyst"));
client.security().deleteUser(d -> d.username("dev_user"));
```
