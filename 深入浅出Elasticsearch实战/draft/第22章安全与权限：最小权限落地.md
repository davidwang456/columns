# 背景

# 第22章 安全与权限：最小权限落地

ES 上线后最容易被忽视的是权限治理。权限过大有风险，权限过细又难维护，本章给出可执行的最小权限实践路径。

## 本章目标
- 建立 ES 权限模型的基础认知。
- 在不牺牲效率的前提下保障数据安全。

## 1. 安全三件套
- 认证：你是谁
- 授权：你能做什么
- 传输加密：数据如何安全传输

## 2. 角色设计原则
- 按职责划分角色（开发、运维、分析）。
- 只给最小可用权限，避免“全开”。

## 3. 实践建议
- 应用侧优先 API Key。
- 管理账号与业务账号分离。
- 敏感索引做额外访问控制与审计。

## 4. 常见问题
- 权限过大导致误删风险。
- 权限过细导致维护成本高。

需要在“安全性”和“可维护性”之间找平衡。

# 总结
- 安全配置要工程化，不靠人工记忆。
- 最小权限是成本最低的长期安全策略。

3. 列出你们当前权限模型的 3 个风险点。  

## 实战（curl）

```bash
# 创建只读角色
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/_security/role/reader_products" \
  -H "Content-Type: application/json" \
  -d '{"indices":[{"names":["products_*"],"privileges":["read","view_index_metadata"]}]}'

# 创建 API Key
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/_security/api_key" \
  -H "Content-Type: application/json" \
  -d '{"name":"products-reader-key","role_descriptors":{"limited":{"indices":[{"names":["products_*"],"privileges":["read"]}]}}}'
```

## 实战（Java SDK）

```java
var key = client.security().createApiKey(c -> c.name("products-reader-key"));
System.out.println(key.id());
```
