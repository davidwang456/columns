


# 第18章 部署拓扑与容量规划：从数据量到节点数

# 背景

Elasticsearch 部署中最常见的翻车场景不是配置写错，而是"拍脑袋定集群规模"。节点太少——磁盘水位报警、分片无法分配、查询超时；节点太多——资源闲置、运维复杂度陡增。容量规划的核心是把业务数据量翻译成具体的磁盘、内存、节点数，再结合节点角色分离和分片策略，构建出可预测、可扩展的集群拓扑。

## 本章目标

- 掌握从日增量到总存储、从总存储到节点数的完整容量计算公式。
- 理解分片大小与分片数的规划原则，避免分片过多或过大。
- 掌握节点角色分离策略：master、data_hot、data_warm、data_cold、ingest、coordinating 的职责与硬件配置。
- 理解 JVM Heap 配置的关键约束：50% 物理内存上限与 compressed oops 阈值。
- 掌握磁盘水位线（watermark）的三级机制与集群行为。
- 能根据业务场景选择小型、中型、大型三种拓扑模式。
- 通过 `_cat` API 和 `_cluster/settings` 进行集群容量监控与水位调整。

## 1. 容量规划公式

### 1.1 总存储量计算

容量规划的起点是业务数据量，核心公式：

```
总存储 = 日增量 × 保留天数 × (1 + 副本数) × 膨胀系数
```

各参数说明：

| 参数 | 含义 | 典型值 |
|------|------|--------|
| 日增量 | 原始数据每天写入量 | 根据业务实测 |
| 保留天数 | 数据在集群中保留的天数 | 7/30/90/180 |
| 副本数 | 通常为 1（生产环境至少 1 副本） | 1~2 |
| 膨胀系数 | 原始数据写入 ES 后因索引结构产生的膨胀 | 1.1~1.5 |

膨胀系数取决于 Mapping 设计：纯 keyword 字段膨胀低（~1.1），大量 text 字段 + 多 analyzer 膨胀高（~1.5）。建议先写入一天样本数据，对比原始大小与 `_cat/indices` 中 `store.size` 来实测。

### 1.2 磁盘总需求

磁盘不能用满——Elasticsearch 在磁盘占用超过水位线后会触发分片迁移甚至禁止写入。需要预留 15% 缓冲：

```
磁盘总需求 = 总存储 / 0.85
```

### 1.3 节点数计算

```
节点数 = 磁盘总需求 / 单节点可用磁盘
```

向上取整，且数据节点数不少于副本数 + 1（否则副本无法分配）。

### 1.4 完整计算示例

场景：日增 500GB 日志，保留 30 天，1 副本，膨胀系数 1.2，单节点 4TB SSD。

```
总存储     = 500GB × 30 × (1+1) × 1.2 = 36,000GB = 36TB
磁盘总需求 = 36TB / 0.85 ≈ 42.4TB
数据节点数 = 42.4TB / 4TB ≈ 11 → 取 12 节点
```

加上 3 台专用 master 节点和 2 台协调节点，集群共 17 个节点。

## 2. 分片规划

### 2.1 单分片大小

单个分片推荐大小为 **10~50GB**：

- **过小（<5GB）**：分片元数据占比高，master 节点内存压力大，段合并频繁。
- **过大（>50GB）**：恢复时间长，relocate 成本高，单个慢查询影响面大。
- **日志场景**：建议 30~50GB，充分利用段合并优势。
- **搜索场景**：建议 10~30GB，控制单分片查询延迟。

### 2.2 分片数计算

```
主分片总数 = 总存储 / 目标单分片大小
```

沿用上例（总存储 36TB，目标单分片 40GB）：

```
主分片总数 = 36,000GB / 40GB = 900
总分片数（含副本）= 900 × (1+1) = 1800
```

### 2.3 分片与 Heap 的关系

经验法则：**每 GB JVM Heap 约支撑 20 个分片**。一个 30GB Heap 的节点最多承载约 600 个分片。

```
单节点分片上限 ≈ Heap(GB) × 20
```

沿用上例：12 个数据节点，每节点 30GB Heap → 单节点上限 600 分片，总容量 7200 分片 >> 实际 1800 分片，余量充裕。

### 2.4 分片过多的危害

- master 节点维护的 ClusterState 膨胀，集群状态更新变慢。
- 每次搜索请求扇出到所有分片，分片越多协调开销越大。
- 段合并的并发数受限，过多小分片导致合并队列积压。

监控分片数的方法：

```bash
curl -u $ES_USER:$ES_PASS "$ES_URL/_cluster/health?pretty" | grep "active_shards"
```

## 3. 节点角色分离

### 3.1 角色总览

| 角色 | `node.roles` 配置 | 职责 | 推荐配置 |
|------|-------------------|------|----------|
| Master | `master` | 管理集群状态、分片分配决策 | 3 节点，4C/8G/100GB SSD |
| Data Hot | `data_hot` | 存储最新数据，承担写入+查询 | 16C/64G/4TB NVMe SSD |
| Data Warm | `data_warm` | 存储较旧数据，以查询为主 | 8C/32G/8TB HDD |
| Data Cold | `data_cold` | 存储极少访问数据 | 4C/16G/12TB HDD |
| Ingest | `ingest` | 执行 Pipeline 预处理 | 8C/16G/200GB SSD |
| Coordinating | 空（`[]`） | 请求路由、结果聚合 | 8C/32G/200GB SSD |

### 3.2 Master 节点

专用 master 节点是生产集群的基本要求。混用 master 与 data 角色时，数据节点的 GC 停顿可能导致 master 失联，触发不必要的选举。

关键配置：

```yaml
# elasticsearch.yml — master 节点
node.roles: [ master ]
cluster.initial_master_nodes: ["master-1", "master-2", "master-3"]
```

- 必须奇数个（3 或 5），满足多数派选举要求。
- 配置无需大磁盘、大内存，4~8GB Heap 足够。

### 3.3 Data 节点分层

Data 节点通过 `data_hot`、`data_warm`、`data_cold` 角色实现分层，配合 ILM 自动迁移数据：

```yaml
# Hot 节点
node.roles: [ data_hot, data_content ]

# Warm 节点
node.roles: [ data_warm ]

# Cold 节点
node.roles: [ data_cold ]
```

Hot 节点用 SSD 保证写入吞吐与最新数据查询性能；Warm 节点用大容量 HDD 降低成本；Cold 节点进一步压缩成本，可挂载廉价存储。

### 3.4 Ingest 与 Coordinating 节点

- **Ingest 节点**：当 Pipeline 逻辑复杂（Grok 解析、GeoIP 查询、脚本转换）时，独立 ingest 节点可避免预处理抢占数据节点的 CPU。
- **Coordinating 节点**：不承担任何角色（`node.roles: []`），纯粹做请求路由与结果聚合。在大量并发查询且涉及重度聚合时，独立协调节点能显著降低数据节点内存压力。

```yaml
# Coordinating 节点
node.roles: [ ]
```

## 4. JVM Heap 配置

### 4.1 核心约束

JVM Heap 配置有两条红线：

1. **不超过物理内存的 50%**：剩余内存留给操作系统文件缓存（Lucene 段文件高度依赖 page cache）。
2. **不超过 30~31GB**：超过此阈值 JVM 无法使用 compressed ordinary object pointers（compressed oops），指针从 4 字节膨胀到 8 字节，实际可用堆反而减少。

```
# jvm.options
-Xms30g
-Xmx30g
```

Xms 与 Xmx 必须相等，避免运行时 Heap 扩缩引发 GC 抖动。

### 4.2 验证 compressed oops

启动后检查日志是否包含：

```
[INFO ] heap size [30gb], compressed ordinary object pointers [true]
```

若显示 `[false]`，说明 Heap 设置过大，需降低至 31GB 以下。也可通过 API 检查：

```bash
curl -u $ES_USER:$ES_PASS "$ES_URL/_nodes/jvm?pretty" \
  | grep "using_compressed"
```

### 4.3 不同内存规格的推荐配置

| 物理内存 | 推荐 Heap | 文件缓存 | 适用场景 |
|---------|-----------|---------|---------|
| 16GB | 8GB | 8GB | 小型/开发环境 |
| 32GB | 16GB | 16GB | 中型数据节点 |
| 64GB | 30~31GB | 33~34GB | 大型数据节点（最优甜点） |
| 128GB | 30~31GB | 97~98GB | 大量段文件场景，额外内存全归 page cache |

> 即使物理内存超过 64GB，Heap 也不要超过 31GB。多出的内存全部给文件缓存，对查询性能的收益远大于增加 Heap。

## 5. 磁盘水位线

### 5.1 三级水位机制

Elasticsearch 通过三级磁盘水位线保护集群：

| 水位 | 默认值 | 触发行为 |
|------|--------|---------|
| `disk.watermark.low` | 85% | 不再向该节点分配新分片 |
| `disk.watermark.high` | 90% | 尝试将分片从该节点迁移走 |
| `disk.watermark.flood_stage` | 95% | 将该节点上的索引设为 `read_only_allow_delete` |

### 5.2 水位的实际影响

- **Low（85%）**：新创建的索引或 rebalance 不会选中该节点，但已有分片不受影响。
- **High（90%）**：集群主动搬迁分片到其他节点，如果所有节点都超 high 水位则无处可搬——集群变黄。
- **Flood Stage（95%）**：索引被强制只读，写入请求报 `FORBIDDEN/12/index read-only / allow delete (api)`。需要先释放磁盘空间，然后手动解除只读状态。

解除 flood stage 只读：

```bash
curl -X PUT -u $ES_USER:$ES_PASS "$ES_URL/_all/_settings" \
  -H 'Content-Type: application/json' -d '{
  "index.blocks.read_only_allow_delete": null
}'
```

### 5.3 自定义水位

对于大容量磁盘（如 10TB），85% 仍有 1.5TB 空闲，可以适当提高水位：

```bash
curl -X PUT -u $ES_USER:$ES_PASS "$ES_URL/_cluster/settings" \
  -H 'Content-Type: application/json' -d '{
  "persistent": {
    "cluster.routing.allocation.disk.watermark.low": "88%",
    "cluster.routing.allocation.disk.watermark.high": "92%",
    "cluster.routing.allocation.disk.watermark.flood_stage": "96%"
  }
}'
```

也可以用绝对值（如 `500gb`）代替百分比，表示"剩余空间低于此值时触发"。

## 6. 三种典型拓扑

### 6.1 小型拓扑（3 节点混合）

适用于日增量 <50GB、总存储 <2TB 的场景。

```
┌──────────────────────────────────────┐
│  Node-1 (master + data + ingest)     │
│  Node-2 (master + data + ingest)     │
│  Node-3 (master + data + ingest)     │
└──────────────────────────────────────┘
```

配置：每节点 8C/32G/2TB SSD，Heap 16GB。

优点：运维简单，成本低。缺点：角色耦合，单节点故障同时影响 master 选举和数据可用性。

### 6.2 中型拓扑（3 master + 6 data）

适用于日增量 50~500GB、总存储 2~30TB。

```
┌─────────────────────────────────────────────────────────┐
│  Master-1    Master-2    Master-3     (专用 master)     │
│  Data-1      Data-2      Data-3       (data_hot)        │
│  Data-4      Data-5      Data-6       (data_hot)        │
└─────────────────────────────────────────────────────────┘
```

Master 节点：4C/8G/100GB SSD，Heap 4GB。Data 节点：16C/64G/4TB SSD，Heap 30GB。

此拓扑已将 master 职责隔离，数据节点可安全重启而不影响集群管理。

### 6.3 大型拓扑（全角色分离）

适用于日增量 >500GB 的场景。

```
┌────────────────────────────────────────────────────────────────┐
│  Master-1    Master-2    Master-3         (专用 master)        │
│  Hot-1 ~ Hot-N                            (data_hot, NVMe)    │
│  Warm-1 ~ Warm-M                          (data_warm, HDD)    │
│  Cold-1 ~ Cold-K                          (data_cold, HDD)    │
│  Ingest-1    Ingest-2                     (ingest)            │
│  Coord-1     Coord-2                      (coordinating)      │
└────────────────────────────────────────────────────────────────┘
```

特点：

- Hot/Warm/Cold 分层配合 ILM 自动迁移（参见第 16 章）。
- Ingest 节点隔离复杂的 Pipeline 预处理。
- Coordinating 节点承接外部查询流量，保护数据节点。
- 各层可独立扩缩容，如写入量增大只需扩 Hot 节点。

## 7. 弹性扩容策略

### 7.1 水平扩容

Elasticsearch 天然支持水平扩容：新数据节点加入集群后，master 自动将分片 rebalance 到新节点。

扩容步骤：

1. 新节点配置相同 `cluster.name`，加入集群。
2. 等待分片 rebalance 完成：`_cat/recovery` 监控进度。
3. 确认 `_cat/allocation` 中各节点数据分布均匀。

### 7.2 扩容时机判断

| 指标 | 阈值 | 行动 |
|------|------|------|
| 磁盘使用率 | >75% | 计划扩容 |
| CPU 持续使用率 | >70% | 加数据节点或协调节点 |
| Heap 使用率 | >75% | 检查分片数、fielddata，可能需加节点 |
| 搜索延迟 P99 | >业务 SLA | 加协调节点或拆分索引 |
| 分片数/节点 | >500 | 减少分片或加节点 |

### 7.3 缩容注意事项

缩容比扩容风险更高：

1. 先通过 allocation exclude 将目标节点上的分片迁走。
2. 等待所有分片迁移完成（`_cat/shards` 中不再有该节点）。
3. 确认集群 green 后再下线节点。

```bash
curl -X PUT -u $ES_USER:$ES_PASS "$ES_URL/_cluster/settings" \
  -H 'Content-Type: application/json' -d '{
  "persistent": {
    "cluster.routing.allocation.exclude._name": "data-6"
  }
}'
```

## 8. 总结

| 知识点 | 核心要点 |
|--------|---------|
| 容量公式 | 日增 × 天数 × (1+副本) × 膨胀系数 → 磁盘/0.85 → 节点数 |
| 分片大小 | 10~50GB/分片，每 GB Heap 约 20 分片 |
| 角色分离 | master 必须专用，data 按温度分层，大集群加 ingest/coord |
| JVM Heap | ≤50% 物理内存且 ≤31GB，剩余给 page cache |
| 磁盘水位 | low=85% 停分配，high=90% 迁分片，flood=95% 只读 |
| 拓扑选择 | 小型 3 混合，中型 3+6，大型全分离 |
| 扩容 | 水平扩容自动 rebalance，缩容先 exclude 再下线 |

## 10. 实战（curl）

以下命令均使用 `$ES_USER`、`$ES_PASS`、`$ES_URL` 环境变量。

```bash
# ========== 1. 查看各节点磁盘分配 ==========
# 显示每个节点的分片数、磁盘使用量、磁盘可用量、磁盘使用率
curl -u $ES_USER:$ES_PASS "$ES_URL/_cat/allocation?v&s=disk.percent:desc"

# ========== 2. 查看节点角色与资源概况 ==========
# ip、heap、ram、cpu、load、角色、节点名
curl -u $ES_USER:$ES_PASS "$ES_URL/_cat/nodes?v&h=ip,name,node.role,heap.percent,ram.percent,cpu,load_1m,disk.used_percent"

# ========== 3. 查看节点详细统计 ==========
# JVM Heap、OS 内存、文件系统、线程池
curl -u $ES_USER:$ES_PASS "$ES_URL/_nodes/stats/jvm,os,fs,thread_pool?pretty"

# ========== 4. 查看当前磁盘水位配置 ==========
curl -u $ES_USER:$ES_PASS "$ES_URL/_cluster/settings?include_defaults=true&flat_settings=true&pretty" \
  | grep "disk.watermark"

# ========== 5. 设置自定义磁盘水位 ==========
curl -X PUT -u $ES_USER:$ES_PASS "$ES_URL/_cluster/settings" \
  -H 'Content-Type: application/json' -d '{
  "persistent": {
    "cluster.routing.allocation.disk.watermark.low": "85%",
    "cluster.routing.allocation.disk.watermark.high": "90%",
    "cluster.routing.allocation.disk.watermark.flood_stage": "95%"
  }
}'

# ========== 6. 查看集群健康与分片统计 ==========
curl -u $ES_USER:$ES_PASS "$ES_URL/_cluster/health?pretty"

# ========== 7. 查看分片分布详情 ==========
# 按索引统计分片数和大小
curl -u $ES_USER:$ES_PASS "$ES_URL/_cat/shards?v&h=index,shard,prirep,state,docs,store,node&s=store:desc"

# ========== 8. 查看大索引的分片大小（评估是否需要调整分片数） ==========
curl -u $ES_USER:$ES_PASS "$ES_URL/_cat/indices?v&h=index,pri,rep,docs.count,store.size,pri.store.size&s=pri.store.size:desc"

# ========== 9. 查看节点 JVM 信息（验证 compressed oops） ==========
curl -u $ES_USER:$ES_PASS "$ES_URL/_nodes/jvm?pretty" \
  | grep -E "heap_max|using_compressed"

# ========== 10. 排除节点用于缩容 ==========
curl -X PUT -u $ES_USER:$ES_PASS "$ES_URL/_cluster/settings" \
  -H 'Content-Type: application/json' -d '{
  "persistent": {
    "cluster.routing.allocation.exclude._name": "data-node-to-remove"
  }
}'

# ========== 11. 清除排除规则（缩容完成后） ==========
curl -X PUT -u $ES_USER:$ES_PASS "$ES_URL/_cluster/settings" \
  -H 'Content-Type: application/json' -d '{
  "persistent": {
    "cluster.routing.allocation.exclude._name": null
  }
}'

# ========== 12. 查看分片恢复进度（扩缩容监控） ==========
curl -u $ES_USER:$ES_PASS "$ES_URL/_cat/recovery?v&active_only=true&h=index,shard,stage,source_node,target_node,bytes_percent"

# ========== 13. 解除 flood_stage 只读限制 ==========
curl -X PUT -u $ES_USER:$ES_PASS "$ES_URL/_all/_settings" \
  -H 'Content-Type: application/json' -d '{
  "index.blocks.read_only_allow_delete": null
}'

# ========== 14. 查看集群分片分配决策解释 ==========
# 当分片 unassigned 时诊断原因
curl -u $ES_USER:$ES_PASS "$ES_URL/_cluster/allocation/explain?pretty" \
  -H 'Content-Type: application/json' -d '{
  "index": "my-index",
  "shard": 0,
  "primary": true
}'

# ========== 15. 查看热点线程（排查 CPU 瓶颈） ==========
curl -u $ES_USER:$ES_PASS "$ES_URL/_nodes/hot_threads"
```

import co.elastic.clients.elasticsearch.cat.*;

var credsProv = new BasicCredentialsProvider();

    .build();

var client = new ElasticsearchClient(
    new RestClientTransport(restClient, new JacksonJsonpMapper()));

// ---------- 1. 查看节点磁盘分配 ----------
var allocationRecords = client.cat().allocation(a -> a.v(true));
allocationRecords.valueBody().forEach(r ->
    System.out.printf("Node: %-20s Shards: %s  Disk.Used: %s  Disk.Avail: %s  Disk.Percent: %s%n",
        r.node(), r.shards(), r.diskUsed(), r.diskAvail(), r.diskPercent()));

// ---------- 2. 查看节点角色与资源 ----------
var nodeRecords = client.cat().nodes(n -> n.v(true)
    .h("ip", "name", "node.role", "heap.percent", "ram.percent", "cpu", "load_1m", "disk.used_percent"));
nodeRecords.valueBody().forEach(n ->
    System.out.printf("%-15s %-20s roles=%-10s heap=%s%% ram=%s%% cpu=%s%n",
        n.ip(), n.name(), n.nodeRole(), n.heapPercent(), n.ramPercent(), n.cpu()));

// ---------- 3. 查看节点 JVM/OS/FS 统计 ----------
var nodesStats = client.nodes().stats(s -> s
    .metric("jvm", "os", "fs"));
nodesStats.nodes().forEach((nodeId, stats) -> {
    var jvm = stats.jvm();
    var fs = stats.fs();
    System.out.printf("Node [%s]: heap_used=%s, heap_max=%s, fs_total=%s, fs_available=%s%n",
        stats.name(),
        jvm.mem().heapUsedInBytes(),
        jvm.mem().heapMaxInBytes(),
        fs.total().totalInBytes(),
        fs.total().availableInBytes());
});

// ---------- 4. 获取当前集群设置（含水位） ----------
var clusterSettings = client.cluster().getSettings(g -> g.includeDefaults(true).flatSettings(true));
System.out.println("Persistent settings: " + clusterSettings.persistent());
System.out.println("Defaults: " + clusterSettings.defaults());

// ---------- 5. 设置磁盘水位 ----------
client.cluster().putSettings(p -> p
    .persistent(m -> m
        .put("cluster.routing.allocation.disk.watermark.low", "85%")
        .put("cluster.routing.allocation.disk.watermark.high", "90%")
        .put("cluster.routing.allocation.disk.watermark.flood_stage", "95%")));
System.out.println("Disk watermarks updated.");

// ---------- 6. 查看集群健康 ----------
var health = client.cluster().health();
System.out.printf("Cluster: %s, Status: %s, Nodes: %d, Shards: %d (active) / %d (primary)%n",
    health.clusterName(), health.status(),
    health.numberOfNodes(), health.activeShards(), health.activePrimaryShards());

// ---------- 7. 查看分片分布 ----------
var shardRecords = client.cat().shards(s -> s.v(true)
    .h("index", "shard", "prirep", "state", "docs", "store", "node"));
shardRecords.valueBody().forEach(s ->
    System.out.printf("%-30s shard=%s %s state=%s docs=%s store=%s node=%s%n",
        s.index(), s.shard(), s.prirep(), s.state(), s.docs(), s.store(), s.node()));

// ---------- 8. 查看索引大小排名 ----------
var indexRecords = client.cat().indices(i -> i.v(true)
    .h("index", "pri", "rep", "docs.count", "store.size", "pri.store.size")
    .s("pri.store.size:desc"));
indexRecords.valueBody().forEach(i ->
    System.out.printf("%-40s pri=%s rep=%s docs=%s store=%s pri_store=%s%n",
        i.index(), i.pri(), i.rep(), i.docsCount(), i.storeSize(), i.priStoreSize()));

// ---------- 9. 排除节点（缩容准备） ----------
client.cluster().putSettings(p -> p
    .persistent(m -> m
        .put("cluster.routing.allocation.exclude._name", "data-node-to-remove")));
System.out.println("Node exclusion set. Waiting for shard relocation...");

// ---------- 10. 监控分片恢复进度 ----------
var recoveryRecords = client.cat().recovery(r -> r.v(true).activeOnly(true)
    .h("index", "shard", "stage", "source_node", "target_node", "bytes_percent"));
recoveryRecords.valueBody().forEach(r ->
    System.out.printf("%-30s shard=%s stage=%s %s -> %s (%s)%n",
        r.index(), r.shard(), r.stage(), r.sourceNode(), r.targetNode(), r.bytesPercent()));

// ---------- 11. 清除节点排除规则 ----------
client.cluster().putSettings(p -> p
    .persistent(m -> m
        .putNull("cluster.routing.allocation.exclude._name")));
System.out.println("Node exclusion cleared.");

// ---------- 12. 解除 flood_stage 只读 ----------
client.indices().putSettings(s -> s
    .index("_all")
    .settings(st -> st.blocksReadOnlyAllowDelete(false)));
System.out.println("Read-only-allow-delete block removed from all indices.");

// ---------- 清理连接 ----------
restClient.close();
