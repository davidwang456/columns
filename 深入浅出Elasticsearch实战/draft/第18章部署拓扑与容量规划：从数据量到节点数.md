# 背景

# 第18章 部署拓扑与容量规划：从数据量到节点数

很多 ES 故障看似是“配置问题”，根源其实是容量规划不足。先建立可解释的容量模型，才能减少后续扩容与迁移风险。

## 本章目标
- 学会做基础容量估算。
- 形成可执行的节点与分片规划方法。

## 1. 容量规划输入项
- 日增数据量
- 保留周期
- 副本数
- 查询与写入峰值

## 2. 估算思路
先算总存储，再考虑索引放大和预留空间，  
最后反推节点数量与硬件配置。

## 3. 拓扑建议
- 小规模：混部节点可快速起步
- 中大规模：分离 master/data/ingest/coord 更稳

## 4. 常见误区
- 只看磁盘不看 CPU 与内存
- 分片数量一次性拍脑袋决定

# 总结
- 容量规划是持续动作，不是一次性文档。
- 规划要和实际监控数据闭环迭代。

## 练习题
1. 为“日增 300GB，保留 30 天”做容量估算。  
2. 输出一版 3 节点和 6 节点拓扑对比。  
3. 说明你的分片大小目标及理由。  

## 实战（curl）

```bash
# 快速拉取容量规划相关信息
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/indices?v&bytes=gb"
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/allocation?v&bytes=gb"
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_nodes/stats/fs,jvm,os?pretty"
```

