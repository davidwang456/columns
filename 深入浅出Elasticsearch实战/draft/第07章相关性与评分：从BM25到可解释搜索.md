

# 第07章 相关性与评分：从 BM25 到可解释搜索

# 背景

数据库回答"有或没有"，搜索引擎回答"谁更相关"。这个差异的核心就是评分。Elasticsearch 默认使用 BM25 算法为每条命中文档计算 `_score`，按分数从高到低排序。理解评分机制，才能在"搜索结果不符合预期"时知道从何下手。

## 本章目标

- 理解搜索评分的本质意义以及它与数据库过滤的区别。
- 掌握 BM25 算法的直觉原理与三大核心因子：TF、IDF、文档长度归一化。
- 学会使用 boost、dis_max、constant_score 等手段影响评分。
- 熟练使用 Explain API 解读评分拆解树。
- 掌握 function_score 查询的完整结构与常见调分模式。
- 能够在电商搜索等实际场景中综合运用上述技能优化排序。

---

## 1. 为什么搜索需要评分

传统数据库的 `WHERE` 子句只做布尔判断：满足条件则返回，不满足则丢弃，所有返回行地位平等。但在搜索场景中，用户输入"笔记本电脑 轻薄"可能命中几千条文档，用户期望最匹配的排在最前面。评分的目标就是给每条命中文档一个数值分数，量化"这条文档和用户查询有多相关"。

Elasticsearch 中，`query` 上下文计算评分，`filter` 上下文只做布尔过滤不评分（分数恒为 0）。`bool` 查询中 `must` 和 `should` 参与评分，`filter` 和 `must_not` 不参与。

---

## 2. BM25 算法深入

BM25（Best Matching 25）是 Elasticsearch 5.0 起的默认评分算法，取代了 TF-IDF。它由三个核心因子决定。

### 2.1 TF：词频

TF（Term Frequency）衡量查询词在文档中出现的次数。BM25 中的 TF 是**非线性、有上限**的：第一次出现贡献最大，后续出现贡献递减并趋于饱和。公式形如 `tf / (tf + k1)`，当 tf 趋向无穷时整个值趋向 1。

直觉：一篇文章提到"Elasticsearch"3 次比 1 次更相关，但提到 100 次不会比 50 次显著更相关。

### 2.2 IDF：逆文档频率

IDF（Inverse Document Frequency）衡量查询词在整个索引中的稀有程度。越稀有的词越有区分度。"的"出现在几乎每个文档中，IDF 极低；"Elasticsearch"只出现在少数文档中，IDF 很高。公式形如 `log(1 + (N - n + 0.5) / (n + 0.5))`，其中 N 是文档总数，n 是包含该词的文档数。

### 2.3 文档长度归一化

长文档天然包含更多词汇，出现某个查询词的概率更高。BM25 通过 `dl/avgdl`（文档长度 / 平均文档长度）进行归一化，文档越长于平均值，TF 的贡献被压缩得越多。

### 2.4 参数 k1 与 b

- **k1**（默认 1.2）：控制 TF 饱和速度。k1 越大，需要更多次出现才饱和；k1 越小，TF 很快饱和。
- **b**（默认 0.75）：控制文档长度归一化强度。b=0 完全忽略文档长度；b=1 完全按长度比例归一化。

绝大多数场景下默认值已经足够好。需要调整时可通过 `PUT /<index>/_settings` 修改 similarity 配置。

### 2.5 直觉总结

```
BM25 得分 ≈ IDF(词的全局稀有度) × TF(词在文档中的频率, 有上限) × 长度修正(dl/avgdl)
```

多词查询（如 "笔记本 轻薄"）的最终分数是每个词的 BM25 得分之和。

## 3. 影响评分的手段

### 3.1 boost：字段级权重

在 `multi_match` 中用 `^` 语法为字段指定权重倍数：`"fields": ["name^3", "description"]`，表示 name 字段上的匹配分数乘以 3。在 `bool.should` 子句中也可单独设置 `"boost": 3`。

### 3.2 match_phrase 与 match

`match` 只要求词出现在文档中（无序），`match_phrase` 要求词按顺序连续出现。因为约束更强，`match_phrase` 通常在 `should` 中与 `match` 搭配，为短语精确匹配的文档加分。

### 3.3 dis_max 查询

`multi_match` 默认将各字段分数求和，可能导致"多字段勉强匹配"的文档分数高于"单字段完美匹配"的文档。`dis_max` 取最佳匹配字段的分数作为主分数，`tie_breaker`（0~1）决定非最佳字段的贡献比例，0.3 是常用值。`multi_match` 的 `type: "best_fields"` 底层就是 `dis_max`。

### 3.4 constant_score

当只需要布尔过滤不需要评分时，用 `constant_score` 包装，所有匹配文档获得相同分数（默认 1.0）。等价于 `bool.filter`，但语义更显式。

## 4. Explain API 详解

### 4.1 单文档解释

端点 `GET /<index>/_explain/<id>`，对指定文档解释为何得到某个分数（或为何未命中）。响应包含 `matched`（布尔）和 `_explanation` 对象。`_explanation` 是一棵递归树：每个节点包含 `value`（分值）、`description`（计算说明）和 `details`（子节点）。

### 4.2 读懂 BM25 拆解

典型的拆解路径：

1. **最外层**：`sum of:` — 多词查询的各词分数之和。
2. **每个词**：`weight(field:term in docId)` — 该词在该文档中的 BM25 得分。
3. **内部**：`idf(docFreq=N, docCount=M)` — IDF 值及其参数。
4. **内部**：`tf(freq=F, k1=1.2, b=0.75, dl=L, avgdl=A)` — TF 值及文档长度参数。
5. **boost**：如果指定了 boost，会出现乘法节点。

### 4.3 搜索请求中的 explain

在搜索请求中加 `"explain": true`，每个命中文档都附带 `_explanation`。注意 explain 会显著增加响应体积和计算开销，仅在调试时使用。

## 5. function_score 查询

当 BM25 的文本相关性不足以满足排序需求时（还需考虑销量、新鲜度、地理距离），使用 `function_score` 在 BM25 分数基础上叠加业务因子。

### 5.1 整体结构

```json
{
  "function_score": {
    "query": { "match": { "name": "手机" } },
    "functions": [ ... ],
    "score_mode": "sum",
    "boost_mode": "multiply"
  }
}
```

- `query`：基础查询，产生原始 BM25 分数。
- `functions`：评分函数数组，每个元素可附带 `filter` 限定作用范围。
- `score_mode`：多个 function 结果的合并方式（multiply / sum / avg / first / max / min）。
- `boost_mode`：function 结果与 query 原始分数的合并方式（multiply / replace / sum / avg / max / min）。

### 5.2 weight

最简单的评分函数，将满足 filter 条件的文档分数乘以固定值。

### 5.3 field_value_factor

基于文档某个数值字段的值调整分数。`modifier` 控制变换方式，常用 `log1p`（log(1+x)，平滑大值差异）、`sqrt`、`none`。`factor` 是乘法因子。`missing` 是字段缺失时的默认值。

### 5.4 script_score

用 Painless 脚本实现任意评分逻辑，如 `"source": "_score * Math.log(2 + doc['sales'].value)"`。脚本在每个匹配文档上执行，大规模数据下需关注性能。

### 5.5 衰减函数：gauss / exp / linear

对数值、日期、地理坐标字段，基于距离原点的远近给分。四个参数：

- `origin`：最佳值（得分最高点）。
- `scale`：从 origin 偏移多远时，得分衰减到 `decay`。
- `offset`：在此范围内不衰减（保持满分）。
- `decay`：在 `origin ± (offset + scale)` 处的得分比例（默认 0.5）。

三种曲线：`gauss`（钟形，平滑）、`exp`（指数，快速下降）、`linear`（线性，匀速下降）。

## 6. 实战场景：电商搜索排序优化

假设需要优化电商搜索排序，综合考虑文本相关性、销量和新鲜度：

- 标题匹配权重是描述的 3 倍。
- 销量越高排越靠前（用 `log1p` 平滑，避免爆款过度霸榜）。
- 最近 30 天上架的商品通过高斯衰减加分，30 天外逐渐失去新鲜度加成。

最终排序逻辑：`BM25(name^3 + description) × (log1p(0.1 * sales) + gauss_decay(created_at))`。完整查询见实战部分 function_score 示例。

---

## 7. 调优原则

1. **先保证结果正确，再追求排序更优**。分词器不对或字段映射错误，再怎么调 function_score 也无济于事。
2. **保持可解释路径**。每次调整后用 Explain API 验证典型文档的得分拆解。避免叠加过多 function 导致分数来源无法追溯。
3. **避免过度依赖 boost**。boost 是乘法因子，多层叠加容易产生意料之外的分数膨胀。
4. **善用 A/B 测试**。排序优化的效果最终要靠用户行为数据（点击率、转化率）验证。

---

## 总结

| 概念 | 核心要点 |
|------|----------|
| BM25 | TF(有上限) x IDF(词的稀有度) x 长度修正，默认 k1=1.2, b=0.75 |
| boost | 字段级权重倍数，name^3 表示分数乘以 3 |
| dis_max | 取最佳匹配字段分数，tie_breaker 控制次要字段贡献 |
| constant_score | 忽略评分，所有匹配文档得相同分数 |
| Explain API | 递归树结构拆解评分来源，调试利器 |
| function_score | 在 BM25 基础上叠加 weight / field_value_factor / script_score / 衰减函数 |
| score_mode | 多个 function 之间的合并方式 |
| boost_mode | function 结果与 query 原始分数的合并方式 |

## 实战（Java SDK）

```java
// ========== 创建索引 ==========
client.indices().create(c -> c.index("score_demo")
    .mappings(m -> m
        .properties("name", p -> p.text(t -> t.analyzer("standard")))
        .properties("description", p -> p.text(t -> t.analyzer("standard")))
        .properties("sales", p -> p.integer(i -> i))
        .properties("rating", p -> p.float_(f -> f))
        .properties("is_promoted", p -> p.boolean_(b -> b))
        .properties("created_at", p -> p.date(d -> d))));

// ========== 批量写入 ==========
client.bulk(b -> b.refresh(Refresh.WaitFor)
    .operations(op -> op.index(i -> i.index("score_demo").id("1")
        .document(Map.of("name", "lightweight laptop ultrabook",
            "description", "thin and light laptop for business",
            "sales", 5000, "rating", 4.8, "is_promoted", true, "created_at", "2026-03-25"))))
    .operations(op -> op.index(i -> i.index("score_demo").id("2")
        .document(Map.of("name", "laptop stand holder",
            "description", "adjustable lightweight aluminum laptop stand",
            "sales", 800, "rating", 4.2, "is_promoted", false, "created_at", "2026-02-10"))))
    .operations(op -> op.index(i -> i.index("score_demo").id("3")
        .document(Map.of("name", "laptop bag lightweight",
            "description", "waterproof laptop bag",
            "sales", 3000, "rating", 4.6, "is_promoted", false, "created_at", "2026-03-30"))))
    .operations(op -> op.index(i -> i.index("score_demo").id("4")
        .document(Map.of("name", "gaming laptop",
            "description", "high performance gaming laptop with lightweight design",
            "sales", 1200, "rating", 4.5, "is_promoted", true, "created_at", "2026-01-15"))))
    .operations(op -> op.index(i -> i.index("score_demo").id("5")
        .document(Map.of("name", "lightweight keyboard",
            "description", "portable lightweight bluetooth keyboard",
            "sales", 600, "rating", 4.0, "is_promoted", false, "created_at", "2025-12-01")))));

// ========== Explain API ==========
var explainResp = client.explain(e -> e.index("score_demo").id("1")
    .query(q -> q.match(m -> m.field("name").query("lightweight laptop"))), Map.class);
System.out.println("matched=" + explainResp.matched() + " explanation=" + explainResp.explanation());

// ========== multi_match with boost ==========
var mmResp = client.search(s -> s.index("score_demo")
    .query(q -> q.multiMatch(mm -> mm
        .query("lightweight laptop")
        .fields("name^3", "description")
        .type(TextQueryType.BestFields)
        .tieBreaker(0.3))), Map.class);
mmResp.hits().hits().forEach(h -> System.out.println("id=" + h.id() + " score=" + h.score()));

// ========== dis_max ==========
var dmResp = client.search(s -> s.index("score_demo")
    .query(q -> q.disMax(dm -> dm
        .queries(
            q1 -> q1.match(m -> m.field("name").query("lightweight laptop").boost(3.0f)),
            q2 -> q2.match(m -> m.field("description").query("lightweight laptop")))
        .tieBreaker(0.3))), Map.class);
dmResp.hits().hits().forEach(h -> System.out.println("id=" + h.id() + " score=" + h.score()));

// ========== constant_score ==========
var csResp = client.search(s -> s.index("score_demo")
    .query(q -> q.constantScore(cs -> cs
        .filter(f -> f.term(t -> t.field("is_promoted").value(true)))
        .boost(1.0f))), Map.class);
csResp.hits().hits().forEach(h -> System.out.println("id=" + h.id() + " score=" + h.score()));

// ========== function_score ==========
var fsResp = client.search(s -> s.index("score_demo")
    .query(q -> q.functionScore(fs -> fs
        .query(fq -> fq.multiMatch(mm -> mm
            .query("lightweight laptop")
            .fields("name^3", "description")
            .type(TextQueryType.BestFields)
            .tieBreaker(0.3)))
        .functions(
            fn -> fn.fieldValueFactor(fvf -> fvf
                .field("sales").modifier(FieldValueFactorModifier.Log1p).factor(0.1).missing(0.0)),
            fn -> fn.gauss(dec -> dec.dateDecayFunction("created_at",
                df -> df.origin("now").scale("15d").offset("3d").decay(0.5))),
            fn -> fn.filter(f -> f.term(t -> t.field("is_promoted").value(true))).weight(2.0))
        .scoreMode(FunctionScoreMode.Sum)
        .boostMode(FunctionBoostMode.Multiply))), Map.class);
fsResp.hits().hits().forEach(h -> System.out.println("id=" + h.id() + " score=" + h.score()));

// ========== 清理 ==========
client.indices().delete(d -> d.index("score_demo"));
```
