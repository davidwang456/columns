

# 第19章 可观测性体系：指标、告警与诊断地图

# 背景

集群运行平稳时，运维人员最容易忽略可观测性建设。然而当第一次线上告警响起时，缺少指标体系和诊断流程的团队往往手足无措——"慢了但不知道慢在哪""红了但不知道哪个分片出了问题"。可观测性不是"等出了问题再建"的东西，而是上线前就应该就位的基础设施。

本章将建立从集群到查询的四层指标体系，设计 5 个核心告警，并画出按问题类型索引的诊断地图——让排障从"靠经验猜"转为"按图索骥"。

## 本章目标

- 掌握四层可观测指标体系（集群 → 节点 → 索引 → 查询），知道每层该看哪些 API 和哪些字段。
- 设计 5 个核心告警，每个告警包含指标、阈值、首查项、处置动作。
- 建立按问题类型索引的诊断地图，面对常见故障能快速定位入口。
- 熟练使用 `_cat` API 进行日常巡检。
- 了解与外部监控系统（Prometheus、Metricbeat + Kibana）的集成思路。

## 1. 四层可观测指标体系

### 1.1 总览

```
第四层：查询层（慢日志 / Profile API）
  │  "这条查询为什么慢？"
  │
第三层：索引层（_stats API）
  │  "哪个索引的写入/查询/merge 有异常？"
  │
第二层：节点层（_nodes/stats API）
  │  "哪个节点的 JVM / CPU / 磁盘 / 线程池有压力？"
  │
第一层：集群层（_cluster/health, _cluster/stats）
  │  "集群整体是否健康？"
```

由上往下，粒度越来越细；排障时通常**自底向上**——先看集群状态判断严重程度，再逐层下钻定位根因。

### 1.2 集群层

**核心 API：** `_cluster/health`

| 字段                          | 含义                               | 关注点            |
| --------------------------- | -------------------------------- | -------------- |
| `status`                    | green / yellow / red             | 非 green 即需排查   |
| `number_of_nodes`           | 当前集群节点数                          | 低于预期说明有节点离线    |
| `number_of_data_nodes`      | 数据节点数                            | 同上             |
| `active_primary_shards`     | 活跃主分片数                           | 低于预期说明有主分片未分配  |
| `active_shards_percent_as_number` | 活跃分片百分比                    | < 100% 说明有未分配分片 |
| `unassigned_shards`         | 未分配分片数                           | > 0 即需排查       |
| `relocating_shards`         | 正在迁移的分片数                         | 持续非 0 说明在做再平衡  |
| `initializing_shards`       | 正在初始化的分片数                        | 节点刚加入时为正常值     |
| `delayed_unassigned_shards` | 延迟分配的分片数                         | 节点短暂离线时的缓冲分片   |

```bash
GET _cluster/health?pretty
```

**补充 API：** `_cluster/stats` 提供集群级聚合统计——总文档数、总存储大小、节点分布、JVM 版本分布等。适合做全局容量看板。

```bash
GET _cluster/stats?pretty&filter_path=indices.count,indices.docs,indices.store,nodes.count
```

### 1.3 节点层

**核心 API：** `_nodes/stats`

| 指标路径                                    | 含义                | 告警阈值参考          |
| --------------------------------------- | ----------------- | --------------- |
| `jvm.mem.heap_used_percent`             | JVM 堆内存使用率        | > 85% 告警        |
| `jvm.gc.collectors.old.collection_time_in_millis` | Old GC 累计耗时 | 单次 > 5s 告警      |
| `jvm.gc.collectors.old.collection_count` | Old GC 次数         | 短时间频繁增长需关注      |
| `os.cpu.percent`                        | 节点 CPU 使用率        | 持续 > 90% 告警     |
| `fs.total.available_in_bytes`           | 可用磁盘空间            | 低于 15% 告警       |
| `thread_pool.write.rejected`            | write 线程池拒绝数      | > 0 即告警         |
| `thread_pool.search.rejected`           | search 线程池拒绝数     | > 0 即告警         |
| `thread_pool.write.queue`               | write 队列长度        | 持续 > 200 告警     |
| `thread_pool.search.queue`              | search 队列长度       | 持续 > 1000 告警    |
| `http.current_open`                     | 当前 HTTP 连接数       | 异常飙升需关注         |
| `breakers.parent.estimated_size_in_bytes` | 断路器估算内存占用       | 接近 limit 需关注    |

```bash
GET _nodes/stats/jvm,os,fs,thread_pool?pretty&filter_path=nodes.*.name,nodes.*.jvm.mem.heap_used_percent,nodes.*.os.cpu.percent,nodes.*.fs.total,nodes.*.thread_pool.write,nodes.*.thread_pool.search
```

获取特定节点的统计：

```bash
GET _nodes/<node_name>/stats/jvm,os,fs?pretty
```

### 1.4 索引层

**核心 API：** `_stats`

| 指标路径                        | 含义              | 关注点                |
| --------------------------- | --------------- | ------------------ |
| `indexing.index_total`      | 索引操作总数（累计）      | 计算写入速率：delta / 时间间隔 |
| `indexing.index_time_in_millis` | 索引操作累计耗时    | 与 total 结合算平均耗时    |
| `search.query_total`        | 查询总数            | 计算 QPS             |
| `search.query_time_in_millis` | 查询累计耗时        | 与 total 结合算平均延迟    |
| `refresh.total`             | Refresh 次数      | 异常高说明 refresh 过频   |
| `refresh.total_time_in_millis` | Refresh 累计耗时 | 单次 > 500ms 需关注     |
| `merges.total`              | Merge 次数        | 突增说明写入量大           |
| `merges.total_size_in_bytes` | Merge 累计处理数据量  | 可计算 merge 吞吐       |
| `flush.total`               | Flush 次数        | 异常频繁说明 translog 过小 |
| `store.size_in_bytes`       | 索引存储大小          | 容量规划依据             |
| `docs.count`                | 文档总数            | 增长趋势监控             |
| `segments.count`            | Segment 数       | 单分片 > 50 个需关注     |

```bash
GET /my_index/_stats?pretty&filter_path=_all.primaries
```

多索引对比：

```bash
GET _stats/indexing,search,merge,refresh,flush?pretty&filter_path=indices.*.primaries
```

### 1.5 查询层

查询层可观测由第 12 章详细介绍的三个工具覆盖：

- **慢日志（Slow Log）**：持续运行，发现慢查询。
- **Profile API**：按需使用，定位查询中的耗时瓶颈。
- **Task API**：查看当前正在执行的长任务。

```bash
GET _tasks?actions=*search&detailed=true&pretty
```

## 2. 五个核心告警设计

### 2.1 告警 1：集群状态非 green

| 项目     | 内容                                            |
| ------ | --------------------------------------------- |
| **指标** | `_cluster/health` → `status`                  |
| **阈值** | `yellow` 告警，`red` 严重告警                        |
| **首查项** | `_cluster/health?level=indices` 找到非 green 的索引  |
| **处置** | yellow: 检查 unassigned 副本原因；red: 立即定位 unassigned 主分片 |

```bash
GET _cluster/health?level=indices&pretty&filter_path=status,indices.*.status
```

### 2.2 告警 2：节点 heap_used_percent > 85%

| 项目     | 内容                                                      |
| ------ | ------------------------------------------------------- |
| **指标** | `_nodes/stats` → `jvm.mem.heap_used_percent`            |
| **阈值** | > 75% 预警，> 85% 告警                                      |
| **首查项** | 检查 Old GC 频率和耗时、是否有大聚合/大 `_source` 查询、fielddata 使用     |
| **处置** | 短期：减少并发大查询；长期：增加节点或优化查询、检查 Circuit Breaker 配置           |

```bash
GET _nodes/stats/jvm?pretty&filter_path=nodes.*.name,nodes.*.jvm.mem.heap_used_percent,nodes.*.jvm.gc
```

### 2.3 告警 3：write 线程池 rejected > 0

| 项目     | 内容                                                         |
| ------ | ---------------------------------------------------------- |
| **指标** | `_nodes/stats` → `thread_pool.write.rejected`              |
| **阈值** | > 0 即告警（rejected 是累计值，需监控增量）                               |
| **首查项** | 检查写入并发度、Bulk 批大小、是否有 merge 风暴（merge 限速导致 write 堆积）         |
| **处置** | 降低写入并发、增大 Bulk 批大小、检查 `index.refresh_interval` 是否过低         |

```bash
GET _cat/thread_pool/write?v&h=name,node_name,active,queue,rejected
```

### 2.4 告警 4：磁盘使用率 > 85%

| 项目     | 内容                                                        |
| ------ | --------------------------------------------------------- |
| **指标** | `_nodes/stats` → `fs.total.available_in_bytes` vs `fs.total.total_in_bytes` |
| **阈值** | 使用率 > 85% 告警（对应 ES 默认 high watermark）                     |
| **首查项** | `_cat/allocation` 查看各节点磁盘分布，`_cat/indices` 找出最大的索引         |
| **处置** | 清理过期索引 / 执行 ILM 策略 / 扩容磁盘 / 清理 snapshot 仓库               |

```bash
GET _cat/allocation?v&h=node,disk.used,disk.avail,disk.total,disk.percent
```

### 2.5 告警 5：search 延迟 P99 > 阈值

| 项目     | 内容                                                           |
| ------ | ------------------------------------------------------------ |
| **指标** | 通过慢日志或外部监控（如 Prometheus）采集的 search latency P99               |
| **阈值** | 根据 SLA 定义，如 P99 > 500ms 告警                                   |
| **首查项** | 慢日志中找出高频慢查询 → Profile API 分析 → 检查是否有热点分片                     |
| **处置** | 优化查询（filter 替代 must、避免深分页）、检查是否有节点 GC 导致延迟抖动                |

### 2.6 告警设计总表

| #  | 告警名称              | 指标来源               | 阈值                 | 首查项                      |
| -- | ----------------- | ------------------ | ------------------ | ------------------------ |
| 1  | 集群状态非 green       | `_cluster/health`  | yellow / red       | `?level=indices`         |
| 2  | Heap > 85%        | `_nodes/stats`     | > 85%              | GC 统计 + 断路器              |
| 3  | Write rejected    | `_nodes/stats`     | 增量 > 0             | 写入并发 + merge 状态          |
| 4  | 磁盘 > 85%          | `_nodes/stats`     | > 85%              | `_cat/allocation`        |
| 5  | Search P99 > 阈值   | 慢日志 / Prometheus  | 按 SLA 定            | 慢日志 → Profile            |

## 3. 诊断地图

### 3.1 慢查询排查路径

```
问题：查询响应慢
  │
  ├─ 1. 慢日志确认具体查询
  │     GET <index>/_settings (确认慢日志已开启)
  │
  ├─ 2. Profile 定位瓶颈
  │     POST <index>/_search { "profile": true, ... }
  │     ├── query 树耗时 → 优化查询结构（filter/must 拆分）
  │     ├── collect 耗时 → 聚合范围过大，加 filter
  │     └── fetch 耗时 → _source 过大，用 source filtering
  │
  ├─ 3. 检查热点分片
  │     GET _cat/shards/<index>?v&s=docs:desc
  │     分片大小严重不均 → 检查 routing 设计
  │
  └─ 4. 检查节点资源
        GET _nodes/stats/jvm,os?filter_path=nodes.*.name,nodes.*.jvm.mem,nodes.*.os.cpu
        CPU / GC 异常 → 节点级问题
```

### 3.2 写入拒绝排查路径

```
问题：写入被拒绝（429 / rejected）
  │
  ├─ 1. 确认 rejected 指标
  │     GET _cat/thread_pool/write?v&h=name,node_name,active,queue,rejected
  │
  ├─ 2. 检查 merge 压力
  │     GET <index>/_stats/merge?filter_path=_all.primaries.merges
  │     merge 积压 → 检查 refresh_interval / merge 线程数
  │
  ├─ 3. 检查 Bulk 参数
  │     批大小是否过大？并发是否过高？
  │
  └─ 4. 检查磁盘 IO
        GET _nodes/stats/fs
        磁盘 IO 饱和 → 扩容或换 SSD
```

### 3.3 集群红色排查路径

```
问题：集群状态 red
  │
  ├─ 1. 找到 red 索引
  │     GET _cluster/health?level=indices&filter_path=indices.*.status
  │
  ├─ 2. 找到 unassigned 主分片
  │     GET _cat/shards/<red_index>?v&h=index,shard,prirep,state,unassigned.reason
  │
  ├─ 3. 查看分配解释
  │     POST _cluster/allocation/explain { "index": "<red_index>", "shard": 0, "primary": true }
  │
  └─ 4. 根据原因处置
        ├── NODE_LEFT → 等待节点恢复或手动 reroute
        ├── ALLOCATION_FAILED → 检查失败原因并重试
        └── 分片损坏 → 从 snapshot 恢复或 allocate_empty_primary（丢数据）
```

### 3.4 GC 频繁排查路径

```
问题：Old GC 频繁 / 长停顿
  │
  ├─ 1. 确认 GC 统计
  │     GET _nodes/stats/jvm?filter_path=nodes.*.name,nodes.*.jvm.gc,nodes.*.jvm.mem
  │
  ├─ 2. 检查断路器
  │     GET _nodes/stats/breaker?pretty
  │     parent breaker 接近限制 → 大查询占用内存
  │
  ├─ 3. 检查 fielddata
  │     GET _nodes/stats/indices/fielddata?pretty
  │     fielddata 过大 → 避免对 text 字段做聚合/排序
  │
  └─ 4. 检查大查询
        GET _tasks?actions=*search&detailed=true&pretty
        取消超长时间的查询任务
```

### 3.5 磁盘满排查路径

```
问题：磁盘空间不足
  │
  ├─ 1. 检查各节点磁盘
  │     GET _cat/allocation?v&h=node,disk.used,disk.avail,disk.percent
  │
  ├─ 2. 找到最大的索引
  │     GET _cat/indices?v&s=store.size:desc&h=index,store.size,docs.count
  │
  ├─ 3. 检查是否触发 flood_stage
  │     GET _cat/indices?v&h=index,status&s=status
  │     索引变为 read_only → 需手动解除
  │
  ├─ 4. 清理空间
  │     ├── 删除过期索引
  │     ├── 清理 snapshot 仓库中的旧快照
  │     └── 对停写索引做 force merge 回收删除文档空间
  │
  └─ 5. 解除只读块
        PUT <index>/_settings { "index.blocks.read_only_allow_delete": null }
        PUT _cluster/settings { "persistent": { "cluster.routing.allocation.disk.watermark.flood_stage": "95%" } }
```

## 4. `_cat` API 速查表

`_cat` API 返回简洁的文本表格，适合终端快速查看。添加 `?v` 显示表头，`?help` 查看可用列，`&s=column:desc` 排序，`&h=col1,col2` 选择列。

### 4.1 常用 `_cat` 端点

| 端点                    | 用途               | 典型用法                                            |
| --------------------- | ---------------- | ----------------------------------------------- |
| `_cat/health`         | 集群健康状态           | `_cat/health?v`                                 |
| `_cat/nodes`          | 节点列表和资源概况        | `_cat/nodes?v&h=name,heap.percent,cpu,disk.avail` |
| `_cat/indices`        | 索引列表和基本统计        | `_cat/indices?v&s=store.size:desc`              |
| `_cat/shards`         | 分片分布和状态          | `_cat/shards/<index>?v&s=docs:desc`             |
| `_cat/thread_pool`    | 线程池状态            | `_cat/thread_pool/write,search?v`               |
| `_cat/allocation`     | 节点磁盘分配           | `_cat/allocation?v`                             |
| `_cat/segments`       | Segment 详情       | `_cat/segments/<index>?v`                       |
| `_cat/recovery`       | 分片恢复进度           | `_cat/recovery?v&active_only=true`              |
| `_cat/pending_tasks`  | 待处理的集群状态更新       | `_cat/pending_tasks?v`                          |
| `_cat/tasks`          | 当前运行的任务          | `_cat/tasks?v&actions=*search`                  |
| `_cat/aliases`        | 别名列表             | `_cat/aliases?v`                                |
| `_cat/templates`      | 索引模板列表           | `_cat/templates?v`                              |
| `_cat/nodeattrs`      | 节点自定义属性          | `_cat/nodeattrs?v`                              |

### 4.2 实用组合

```bash
# 找到最大的 10 个索引
GET _cat/indices?v&s=store.size:desc&h=index,health,status,docs.count,store.size&format=text

# 找到所有 UNASSIGNED 分片
GET _cat/shards?v&h=index,shard,prirep,state,unassigned.reason&s=state

# 节点资源一览
GET _cat/nodes?v&h=name,ip,heap.percent,ram.percent,cpu,load_1m,disk.avail,node.role

# 线程池拒绝统计
GET _cat/thread_pool/write,search,get?v&h=name,node_name,active,queue,rejected
```

## 5. 运维脚本：一键健康巡检

以下 bash 脚本覆盖日常巡检的核心检查项，可加入 crontab 定期执行或集成到运维平台。

```bash
#!/bin/bash
# es-health-check.sh — Elasticsearch 集群健康巡检脚本
# 用法: ES_USER=elastic ES_PASS=xxx ES_URL=https://localhost:9200 bash es-health-check.sh

set -euo pipefail

echo "=========================================="
echo " ES 集群健康巡检 - $(date '+%Y-%m-%d %H:%M:%S')"
echo "=========================================="

echo ""
echo ">>> 1. 集群状态"
curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cluster/health?pretty"

echo ""
echo ">>> 2. 节点概况"
curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/nodes?v&h=name,ip,heap.percent,ram.percent,cpu,load_1m,disk.avail,node.role"

echo ""
echo ">>> 3. 磁盘分配"
curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/allocation?v"

echo ""
echo ">>> 4. Unassigned 分片"
UNASSIGNED=$(curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/shards?h=state" | grep -c UNASSIGNED || true)
if [ "$UNASSIGNED" -gt 0 ]; then
  echo "  [WARN] 存在 $UNASSIGNED 个未分配分片！"
  curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/shards?v&h=index,shard,prirep,state,unassigned.reason&s=state" | grep UNASSIGNED
else
  echo "  [OK] 无未分配分片"
fi

echo ""
echo ">>> 5. 线程池拒绝统计"
curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/thread_pool/write,search,get?v&h=name,node_name,active,queue,rejected"

echo ""
echo ">>> 6. 最大的 10 个索引"
curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/indices?v&s=store.size:desc&h=index,health,docs.count,store.size" | head -11

echo ""
echo ">>> 7. Pending Tasks"
curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/pending_tasks?v"

echo ""
echo ">>> 8. 活跃的恢复任务"
curl -s -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/recovery?v&active_only=true" | head -20

echo ""
echo "=========================================="
echo " 巡检完成"
echo "=========================================="
```

## 6. 与外部监控集成

### 6.1 Prometheus + Elasticsearch Exporter

`elasticsearch-exporter`（开源项目）将 ES 指标暴露为 Prometheus 格式：

```
# 部署 exporter（示意）
docker run -d --name es-exporter \
  -p 9114:9114 \
  quay.io/prometheuscommunity/elasticsearch-exporter:latest \
  --es.uri=https://elastic:password@es-node:9200 \
  --es.all --es.indices --es.shards

# Prometheus scrape 配置
scrape_configs:
  - job_name: 'elasticsearch'
    static_configs:
      - targets: ['es-exporter:9114']
```

常用 Prometheus 指标：

| Prometheus 指标名                                      | 对应 ES 指标                        |
| --------------------------------------------------- | ------------------------------- |
| `elasticsearch_cluster_health_status`               | `_cluster/health` → `status`   |
| `elasticsearch_jvm_memory_used_bytes`               | `jvm.mem.heap_used_in_bytes`    |
| `elasticsearch_thread_pool_rejected_count`          | `thread_pool.*.rejected`        |
| `elasticsearch_filesystem_data_available_bytes`     | `fs.total.available_in_bytes`   |
| `elasticsearch_indices_indexing_index_total`        | `indexing.index_total`          |
| `elasticsearch_indices_search_query_time_seconds`   | `search.query_time_in_millis`   |

### 6.2 Metricbeat → Elasticsearch → Kibana Dashboard

Elastic Stack 自带的监控方案：

```yaml
# metricbeat.yml
metricbeat.modules:
  - module: elasticsearch
    metricsets:
      - node
      - node_stats
      - index
      - index_summary
      - shard
    period: 10s
    hosts: ["https://es-node:9200"]
    username: "${ES_USER}"
    password: "${ES_PASS}"
    ssl.verification_mode: none

output.elasticsearch:
  hosts: ["https://monitoring-es:9200"]
  username: "${MON_USER}"
  password: "${MON_PASS}"
```

Kibana 中导入 Metricbeat 的 Elasticsearch 仪表板后，可以直接获得：

- 集群概览：状态、节点数、分片数
- JVM 内存与 GC 趋势图
- 搜索/索引速率与延迟曲线
- 线程池队列和拒绝趋势
- 磁盘使用率趋势
- 每索引的统计详情

### 6.3 监控架构选择

| 方案                          | 优点             | 缺点                       | 适用场景         |
| --------------------------- | -------------- | ------------------------ | ------------ |
| Elastic Stack 自监控           | 原生集成，开箱即用      | 被监控集群故障时监控也受影响           | 小规模集群、快速起步   |
| 独立监控集群 + Metricbeat         | 监控与业务解耦        | 需要额外集群资源                 | 中大规模生产环境     |
| Prometheus + Grafana        | 统一监控技术栈        | 需要额外 exporter            | 已有 Prometheus 体系 |
| 云托管监控（Elastic Cloud 等）      | 零运维            | 成本高                      | 云上部署         |

# 总结

- **四层指标体系**（集群 → 节点 → 索引 → 查询）是可观测性的骨架——每一层都有对应的 API 和关键字段。
- **5 个核心告警**覆盖了最常见的生产故障信号：集群状态、heap 使用率、write rejected、磁盘水位、search 延迟。
- **诊断地图**让排障有据可依——按问题类型索引，每条路径从现象到根因层层下钻。
- `_cat` API 是终端快速诊断的利器，但不适合做自动化监控采集（格式不稳定）。
- 生产环境**必须**将 ES 监控与业务集群解耦——独立监控集群或外部 Prometheus/Grafana。
- 可观测性建设的核心原则：**上线前就位，而非出事后补建**。

## 实战（Java SDK）

```java
// ---------- 集群健康 ----------
var health = client.cluster().health();
System.out.println("Status: " + health.status());
System.out.println("Nodes: " + health.numberOfNodes());
System.out.println("Active shards: " + health.activeShards());
System.out.println("Unassigned: " + health.unassignedShards());
System.out.println("Active shards %: " + health.activeShardsPercentAsNumber());

if (!"green".equals(health.status().jsonValue())) {
    System.err.println("[WARN] 集群状态非 green: " + health.status());
}

// ---------- 节点统计 ----------
var nodesStats = client.nodes().stats(s -> s.metric("jvm", "os", "fs", "thread_pool"));
nodesStats.nodes().forEach((nodeId, stats) -> {
    System.out.println("Node: " + stats.name());
    System.out.println("  Heap used: " + stats.jvm().mem().heapUsedPercent() + "%");
    System.out.println("  CPU: " + stats.os().cpu().percent() + "%");

    var writePool = stats.threadPool().get("write");
    if (writePool != null) {
        System.out.println("  Write rejected: " + writePool.rejected());
        System.out.println("  Write queue: " + writePool.queue());
    }

    var searchPool = stats.threadPool().get("search");
    if (searchPool != null) {
        System.out.println("  Search rejected: " + searchPool.rejected());
    }

    System.out.println("  Disk available: " + stats.fs().total().availableInBytes() + " bytes");
});

// ---------- 索引统计 ----------
var indexStats = client.indices().stats(s -> s.index("my_index")
    .metric("indexing", "search", "merge", "refresh", "flush", "segments"));

var primaries = indexStats.primaries();
System.out.println("Indexing total: " + primaries.indexing().indexTotal());
System.out.println("Search query total: " + primaries.search().queryTotal());
System.out.println("Segments count: " + primaries.segments().count());
System.out.println("Merges total: " + primaries.merges().total());
System.out.println("Refresh total: " + primaries.refresh().total());

// ---------- 分配解释 ----------
var allocExplain = client.cluster().allocationExplain(e -> e
    .index("my_index")
    .shard(0)
    .primary(true));
System.out.println("Can allocate: " + allocExplain.canAllocate());
System.out.println("Allocate explanation: " + allocExplain.allocateExplanation());

// ---------- 任务列表 ----------
var tasks = client.tasks().list(l -> l.actions("*search").detailed(true));
tasks.nodes().forEach((nodeId, nodeTask) ->
    nodeTask.tasks().forEach((taskId, taskInfo) -> {
        System.out.println("Task: " + taskInfo.action());
        System.out.println("  Running for: " + taskInfo.runningTimeInNanos() + " ns");
    }));
```
