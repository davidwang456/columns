# 背景

# 第09章 排序与分页：深分页治理

分页是最常见需求，却也是最容易被忽略的性能雷区。很多系统上线后才发现深分页把集群拖慢，本章通过可执行示例把这一问题前置解决。

## 本章目标
- 理解深分页为什么慢。
- 掌握 `search_after + PIT` 的稳定分页方案。

## 1. 深分页慢在哪里
`from + size` 在深分页时会让每个分片都保留大量候选结果，  
数据越大，协调开销越高。

## 2. 推荐方案：search_after
- 基于“上页最后一条排序值”继续翻页。
- 不需要扫描并丢弃大量中间结果。

## 3. 一致性问题与 PIT
- 翻页期间数据可能变化，导致重复或漏数。
- PIT 可固定查询快照，保证分页一致性。

## 4. 实践建议
- 排序字段要稳定，最好加唯一字段兜底。
- 控制 PIT 生命周期，避免资源泄漏。

# 总结
- 浅分页用 from/size，深分页用 search_after。
- 稳定分页的关键是“稳定排序 + 稳定快照”。

## 练习题
1. 实现一个基于 `search_after` 的翻页查询。  
2. 在数据持续写入下测试有无 PIT 的分页差异。  
3. 解释为什么深分页会拖慢集群。  

## 实战（curl）

```bash
# 第1页
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{"size":2,"sort":[{"price":"asc"},{"id":"asc"}],"query":{"match_all":{}}}'

# 拿到上一页最后一条 sort 值后，执行 search_after（示例值）
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{"size":2,"sort":[{"price":"asc"},{"id":"asc"}],"search_after":[6999,"p1"],"query":{"match_all":{}}}'
```

## 实战（Java SDK）

```java
var page2 = client.search(s -> s.index("products_v1")
    .size(2)
    .sort(so -> so.field(f -> f.field("price")))
    .searchAfter(List.of(co.elastic.clients.json.JsonData.of(6999), co.elastic.clients.json.JsonData.of("p1"))),
    Map.class);
System.out.println(page2.hits().hits().size());
```
