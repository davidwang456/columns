
# 第09章 排序与分页：深分页治理

# 背景

分页是搜索系统最基础的交互之一，但 Elasticsearch 的分布式架构使得"翻到第 10000 页"的代价远超直觉。从 `from+size` 到 `search_after` 再到 Point in Time，三种分页机制各有取舍。理解它们的内部工作原理，才能在"用户体验"与"集群稳定性"之间找到正确的平衡点。

## 本章目标

- 掌握 ES 排序的底层要求，理解 `_score` 排序与字段排序的场景选择。
- 透彻理解 `from+size` 深分页的性能放大问题。
- 掌握 `search_after` 的工作原理和 tiebreaker 设计。
- 理解 Point in Time（PIT）如何提供快照一致性。
- 了解 Scroll API 的历史定位以及为何不再推荐。
- 能够根据业务场景选择最合适的分页策略。

---

## 1. 排序机制

### 1.1 默认排序：_score 降序

不指定 `sort` 时，ES 按 `_score`（BM25 相关性评分）从高到低返回结果。这是全文搜索场景的默认行为——最相关的文档排在最前面。

```json
{ "query": { "match": { "title": "elasticsearch 入门" } } }
```

此时返回结果中每条文档都包含 `_score` 字段，按降序排列。

### 1.2 自定义排序：sort 参数

当业务需要按价格、时间、库存等字段排序时，使用 `sort` 参数：

```json
{
  "sort": [
    { "created_at": "desc" },
    { "price": "asc" },
    "_score"
  ],
  "query": { "match_all": {} }
}
```

`sort` 是一个数组，支持多字段排序。ES 按数组顺序依次比较：先按 `created_at` 降序，相同时按 `price` 升序，再相同时按 `_score` 降序。

指定 `sort` 后，ES 不再计算 `_score`（除非 `sort` 中显式包含 `_score` 或设置 `track_scores: true`），这能节省 CPU 开销。

### 1.3 排序字段的数据要求

排序依赖 doc_values（列式存储），因此只有开启了 doc_values 的字段类型才可排序：

- `keyword`、`numeric`（`integer`/`long`/`float`/`double`）、`date`、`boolean`、`ip` 等——默认开启 doc_values，可直接排序。
- `text` 类型——默认不开启 doc_values，不能直接排序。常见做法是为 `text` 字段添加 `keyword` 子字段，排序时使用 `title.keyword`。

```json
{
  "mappings": {
    "properties": {
      "title": { "type": "text", "fields": { "keyword": { "type": "keyword" } } }
    }
  }
}
```

### 1.4 数组字段排序：mode 参数

当排序字段是数组（如 `prices: [10, 20, 30]`）时，ES 需要从多个值中选一个用于比较。通过 `mode` 参数指定选取策略：

| mode | 行为 |
|------|------|
| `min` | 取最小值（numeric/date 默认 asc 时） |
| `max` | 取最大值（numeric/date 默认 desc 时） |
| `sum` | 取总和 |
| `avg` | 取平均值 |
| `median` | 取中位数 |

```json
{ "sort": [{ "prices": { "order": "asc", "mode": "min" } }] }
```

### 1.5 场景选择：_score vs 字段排序

- **全文搜索**（如商品搜索框）：使用 `_score` 排序，让最相关的结果靠前。
- **列表浏览**（如"最新发布"、"价格最低"）：使用字段排序，关闭评分计算。
- **混合场景**：先按字段排序，将 `_score` 作为 tiebreaker；或使用 `function_score` 将业务权重融入评分。

## 2. from+size 分页

### 2.1 工作原理

`from` 表示跳过的文档数，`size` 表示返回的文档数。看起来像 SQL 的 `LIMIT 10 OFFSET 20`，但在分布式环境下代价完全不同。

ES 的执行流程：

1. 协调节点将请求广播到所有相关分片。
2. **每个分片**返回按排序规则排好的前 `from + size` 条文档（的排序值和文档 ID）。
3. 协调节点收集所有分片的结果，全局排序后取第 `from` 到 `from + size - 1` 条。
4. 协调节点根据文档 ID 回源获取 `_source`（fetch 阶段）。

### 2.2 深分页的代价：数字说话

假设索引有 5 个主分片，请求 `from=10000, size=10`：

- 每个分片需要排序并返回前 `10000 + 10 = 10010` 条文档的排序值。
- 协调节点收到 `5 x 10010 = 50050` 条结果。
- 全局排序后，只取其中 10 条。
- **50040 条数据的排序和传输完全浪费。**

如果 `from=100000`，每个分片返回 100010 条，协调节点合并 500050 条——内存和网络开销线性增长，集群随时可能 OOM。

### 2.3 max_result_window 保护机制

ES 通过 `index.max_result_window`（默认值 10000）限制 `from + size` 的上限。超过此值会返回错误：

```
Result window is too large, from + size must be less than or equal to: [10000]
```

这个限制不是 bug，而是刻意的设计保护。虽然可以通过 `PUT /my_index/_settings` 调大，但不建议这样做——调大只是推迟了问题爆发的时间点，没有解决根本的性能放大问题。

### 2.4 适用场景

`from+size` 适合浅分页——用户只浏览前几页的场景（`from < 1000`）。对于搜索引擎 UI、后台管理列表的前几页，它简单直接、支持跳页，是最佳选择。

## 3. search_after 分页

### 3.1 工作原理

`search_after` 基于上一页最后一条文档的 `sort` 值作为下一页的起点。每个分片只需返回排序值大于 `search_after` 的前 `size` 条文档，而不是前 `from + size` 条。无论翻到第几页，每个分片始终只返回 `size` 条数据，性能恒定。

### 3.2 完整使用流程

**第 1 页**（不带 `search_after`）：

```json
{
  "size": 3,
  "sort": [{ "created_at": "desc" }, { "_shard_doc": "asc" }],
  "query": { "match_all": {} }
}
```

返回结果中每条文档包含 `sort` 数组，例如最后一条的 `sort` 为 `[1700000000000, 42]`。

**第 2 页**（将上一页最后一条的 `sort` 值传入）：

```json
{
  "size": 3,
  "sort": [{ "created_at": "desc" }, { "_shard_doc": "asc" }],
  "search_after": [1700000000000, 42],
  "query": { "match_all": {} }
}
```

**第 3 页**：取第 2 页最后一条的 `sort` 值，继续传入 `search_after`，以此类推。

### 3.3 Tiebreaker：唯一值字段

`sort` 必须包含一个能唯一标识文档的字段作为 tiebreaker，否则排序值相同的文档可能在翻页时被重复或遗漏。常用选择：

- `_shard_doc`：分片内部文档序号，轻量高效，搭配 PIT 使用时自动隐式添加。
- `_id`：文档 ID，全局唯一但排序开销较大（需要 fielddata）。
- 业务唯一字段：如 `order_id.keyword`，类型为 `keyword` 且值唯一。

### 3.4 使用限制

- **不能跳页**：只能顺序向后翻，无法直接跳到第 N 页。
- **不保证一致性**：翻页期间如果有数据写入或删除，可能出现重复或遗漏。这个问题由 PIT 解决。

## 4. Point in Time（PIT）

### 4.1 问题：翻页期间的数据变化

`search_after` 在翻页过程中，索引可能发生变化：新文档插入到已翻过的区间导致后续页重复；已翻过的文档被删除导致后续页遗漏；Segment merge 导致 `_shard_doc` 值变化使排序不稳定。

### 4.2 PIT 的解决方案

PIT 在创建时冻结一个查询时间点快照（类似数据库的 MVCC 快照）。后续基于该 PIT 的所有搜索都在同一个快照上执行，不受索引实时变化的影响。

### 4.3 使用流程

**步骤 1：创建 PIT**——`POST /products/_pit?keep_alive=5m`，返回一个 `id`（即 pit_id）。

**步骤 2：使用 PIT + search_after 翻页**——URL 中不再指定索引名，索引信息已包含在 pit_id 中：

```json
POST /_search
{
  "size": 3,
  "sort": [{ "created_at": "desc" }],
  "pit": { "id": "<pit_id>", "keep_alive": "5m" },
  "query": { "match_all": {} }
}
```

使用 PIT 时，ES 自动在 `sort` 末尾添加 `_shard_doc` 作为隐式 tiebreaker，无需手动指定。后续页面使用 `search_after` 传入上一页最后一条的完整 `sort` 数组即可。

**步骤 3：关闭 PIT**——`DELETE /_pit`，传入 `id`。翻页完成后必须释放 PIT，否则它会持有 Segment 引用，阻止 merge 清理旧 Segment。

### 4.4 keep_alive 与资源注意

`keep_alive` 不是绝对过期时间，而是"距离上次使用的最大空闲时间"。每次搜索请求中携带 `keep_alive` 都会刷新这个计时器。

PIT 持有创建时刻所有 Segment 的引用。即使后台 merge 产生了新 Segment，旧 Segment 也不会被删除，直到 PIT 关闭。因此：

- 设置合理的 `keep_alive`（分钟级，不要设置为小时级）。
- 翻页结束后立即调用 `DELETE /_pit`。
- 监控 `_nodes/stats` 中的 `open_contexts` 指标。

---

## 5. Scroll API（已不推荐）

Scroll API 是 ES 早期提供的深分页方案，最初设计用于批量数据导出（如 reindex、ETL）。使用方式：首次请求带 `scroll=5m` 参数返回 `_scroll_id`，后续通过 `POST /_search/scroll` 获取下一批。

Scroll 的问题：占用服务端资源（维护搜索上下文）；不适合实时用户请求（快照在首次请求时创建）；并行 Scroll 过多会耗尽节点资源。从 ES 7.x 开始，官方推荐使用 `search_after` + PIT 替代。Scroll API 仍保留以保证兼容，但新项目不应使用。

---

## 6. 三种分页方式对比

| 维度 | from+size | search_after | search_after + PIT |
|------|-----------|--------------|-------------------|
| 跳页支持 | 支持 | 不支持 | 不支持 |
| 深分页性能 | 差（线性膨胀） | 好（恒定开销） | 好（恒定开销） |
| 数据一致性 | 无保证 | 无保证 | 快照一致 |
| 资源占用 | 无状态 | 无状态 | 持有 Segment 引用 |
| 典型场景 | 前几页浅分页 | 大量顺序翻页 | 需要一致性的翻页 |
| 排序要求 | 无特殊要求 | 需要 tiebreaker | 自动添加 tiebreaker |

选择建议：

- 搜索结果前几页（用户 UI）-> `from+size`。
- 后台批量遍历、无限滚动加载 -> `search_after`。
- 数据导出、报表生成等需要一致快照 -> `search_after` + PIT。

---

# 总结

- ES 默认按 `_score` 降序排序；自定义排序需要字段开启 doc_values。`text` 字段不能直接排序，需使用 `keyword` 子字段。
- `from+size` 简单但有深分页陷阱：每个分片返回 `from+size` 条数据，协调节点汇总后丢弃绝大部分，`max_result_window` 默认 10000 是刻意的保护。
- `search_after` 基于上一页的 sort 值定位下一页，每个分片只返回 `size` 条，性能恒定，但不能跳页。
- PIT 提供查询时间点快照，配合 `search_after` 实现一致性翻页。用完必须关闭，避免 Segment 泄漏。
- Scroll API 已不推荐，新项目应使用 `search_after` + PIT。

## 实战（Java SDK）

```java
import co.elastic.clients.elasticsearch.ElasticsearchClient;
import co.elastic.clients.elasticsearch._types.Refresh;
import co.elastic.clients.elasticsearch._types.SortOrder;
import co.elastic.clients.elasticsearch.core.*;
import co.elastic.clients.elasticsearch.core.search.Hit;
import java.util.List;
import java.util.Map;

// ---------- 准备索引与数据 ----------
client.indices().create(c -> c.index("sort_demo")
    .settings(s -> s.numberOfShards("2"))
    .mappings(m -> m
        .properties("title", p -> p.text(t -> t.fields("keyword", f -> f.keyword(k -> k))))
        .properties("price", p -> p.integer(i -> i))
        .properties("sales", p -> p.integer(i -> i))
        .properties("created_at", p -> p.date(d -> d))));

client.bulk(b -> b.index("sort_demo").refresh(Refresh.WaitFor)
    .operations(op -> op.index(i -> i.id("1").document(Map.of(
        "title", "Elasticsearch 实战", "price", 89, "sales", 500, "created_at", "2024-01-15"))))
    .operations(op -> op.index(i -> i.id("2").document(Map.of(
        "title", "Lucene 原理", "price", 75, "sales", 320, "created_at", "2024-03-20"))))
    .operations(op -> op.index(i -> i.id("3").document(Map.of(
        "title", "数据密集型应用", "price", 120, "sales", 800, "created_at", "2023-11-01"))))
    .operations(op -> op.index(i -> i.id("4").document(Map.of(
        "title", "深入理解 JVM", "price", 95, "sales", 650, "created_at", "2024-06-10"))))
    .operations(op -> op.index(i -> i.id("5").document(Map.of(
        "title", "Kafka 权威指南", "price", 88, "sales", 420, "created_at", "2024-02-28")))));

// ---------- PIT + search_after 完整翻页 ----------
OpenPointInTimeResponse pitResp = client.openPointInTime(o -> o
    .index("sort_demo").keepAlive(a -> a.time("5m")));
String pitId = pitResp.id();

List<String> searchAfter = null;
int page = 0;

while (true) {
    final List<String> sa = searchAfter;
    SearchRequest.Builder rb = new SearchRequest.Builder()
        .size(2)
        .sort(s -> s.field(f -> f.field("sales").order(SortOrder.Desc)))
        .pit(p -> p.id(pitId).keepAlive(a -> a.time("5m")))
        .query(q -> q.matchAll(m -> m));
    if (sa != null) { rb.searchAfter(sa); }

    SearchResponse<Map> resp = client.search(rb.build(), Map.class);
    List<Hit<Map>> hits = resp.hits().hits();
    if (hits.isEmpty()) break;

    page++;
    System.out.println("===== 第 " + page + " 页 =====");
    for (Hit<Map> hit : hits) {
        System.out.println(hit.id() + " -> " + hit.source());
    }
    searchAfter = hits.get(hits.size() - 1).sort();
}

// ---------- 关闭 PIT 并清理 ----------
client.closePointInTime(c -> c.id(pitId));
client.indices().delete(d -> d.index("sort_demo"));
```
