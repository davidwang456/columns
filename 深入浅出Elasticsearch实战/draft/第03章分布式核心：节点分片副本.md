# 第03章 分布式核心：节点、分片、副本

# 背景

单机检索在数据量增长后会迅速触顶：磁盘装不下、CPU 打满、单点故障即全站不可用。Elasticsearch 从诞生之初就以分布式为核心设计目标，通过节点（Node）、分片（Shard）、副本（Replica）三个概念实现横向扩展与高可用。然而这三个概念若理解不清，就会出现"分片乱配、性能反降、故障难恢复"等棘手问题。本章将带你深入 ES 分布式架构的内脏，从原理到规划，再到实战操作，把这三个核心概念彻底弄透。

## 本章目标

- 深入理解节点、分片、副本的含义与协作关系。
- 掌握 ES 节点角色体系及其配置方式。
- 理解写入路径与查询路径的完整流程。
- 学会科学规划分片数量，避免过多或过少的陷阱。
- 理解集群状态（green/yellow/red）与自愈机制。
- 掌握路由策略与热点分片的排查与解决。

---

## 1. 三个核心概念深入

### 1.1 节点（Node）

节点是一个运行中的 ES 实例，本质上就是一个 JVM 进程。多个节点通过相同的 `cluster.name` 配置值自动发现彼此，组成一个集群。在同一台物理机上可以启动多个节点（用于开发测试），但生产环境中推荐每台机器仅运行一个节点，以避免资源竞争。

每个节点在启动时会被分配一个全局唯一的 UUID，并可在 `elasticsearch.yml` 中通过 `node.name` 指定一个可读名称，方便运维时快速辨认。

### 1.2 分片（Shard）

当你创建一个索引时，ES 会将索引的数据拆分成若干个逻辑分块，每一块就是一个主分片（Primary Shard）。每个主分片在底层就是一个完整的 Lucene 索引，拥有独立的倒排索引、存储文件和段（Segment）。

分片是 ES 实现横向扩展的基本单位：

- 一个 10TB 的索引可以拆成 10 个 1TB 的分片，分布在 10 台机器上。
- 查询请求可以在多个分片上并行执行，充分利用集群算力。

关键约束：`number_of_shards` 在索引创建后不可更改。这一限制源于路由公式对主分片数的依赖（详见第 3 节）。如果必须调整，只能通过 Reindex、Split API 或 Shrink API 实现。

### 1.3 副本（Replica）

副本分片是主分片的完整拷贝。每个主分片可以配置零到多个副本。副本承担两大职责：

- **高可用**：当主分片所在节点宕机时，对应的副本分片会被提升为新的主分片，保证数据不丢失、服务不中断。
- **读扩展**：搜索请求可以在主分片或任意副本上执行。读压力大时，增加 `number_of_replicas` 是最直接的扩容手段。

ES 有一个铁律：同一分片的主本和副本绝不会被分配到同一个节点上。如果集群只有一个节点，副本将处于 `UNASSIGNED` 状态，集群状态为 yellow。

## 2. 节点角色详解

ES 支持为每个节点配置一个或多个角色，通过 `elasticsearch.yml` 中的 `node.roles` 参数指定。不同角色承担不同的职责，生产环境推荐进行角色分离。

### 2.1 master-eligible（候选主节点）

负责集群层面的管理操作：创建与删除索引、追踪节点上下线、做出分片分配决策、维护集群元数据。集群中有且仅有一个节点在任意时刻担任 Active Master，由所有 master-eligible 节点通过选举产生。

配置方式：

```yaml
node.roles: [master]
```

生产建议：部署 3 个专用的 master-eligible 节点，配置小规格（2 核 4GB 即可），不承担数据存储，确保集群管理操作不受数据负载影响。

### 2.2 data（数据节点）

真正的"苦力"角色，负责保存分片数据，执行文档的增删改查及聚合分析。数据节点的 CPU、内存、磁盘 IO 是集群性能的决定性因素。

```yaml
node.roles: [data]
```

### 2.3 分层数据节点

ES 支持将数据节点按存储介质和数据温度进一步细分，与 ILM（Index Lifecycle Management）配合实现成本治理：

| 角色 | 存储介质 | 数据特征 |
|------|----------|----------|
| `data_hot` | NVMe SSD | 最近写入、频繁查询的数据 |
| `data_warm` | SAS SSD / HDD | 查询频次降低，仍需在线 |
| `data_cold` | 大容量 HDD | 偶尔查询，可接受较高延迟 |
| `data_frozen` | 对象存储（S3 等）| 极少访问，searchable snapshot |

配置示例：

```yaml
node.roles: [data_hot]
```

### 2.4 ingest（预处理节点）

在文档写入索引前执行 Ingest Pipeline 中定义的处理器（如 grok 解析日志、日期格式化、字段重命名等）。如果 pipeline 处理逻辑较重，建议使用专用 ingest 节点，避免影响数据节点性能。

```yaml
node.roles: [ingest]
```

### 2.5 coordinating（协调节点）

当 `node.roles` 设置为空列表时，该节点不承担任何特定角色，仅作为协调节点。它接收客户端请求，将请求路由到正确的分片，收集各分片返回的结果并汇总后返回给客户端。

```yaml
node.roles: []
```

在大规模集群中，专用协调节点可以有效分担结果汇总和排序的 CPU 与内存开销。

### 2.6 混合角色配置

小规模集群中，一个节点可以同时承担多个角色：

```yaml
node.roles: [master, data, ingest]
```

但随着集群规模增长，角色分离是提升稳定性的关键实践。

## 3. 写入路径详解

一条文档从客户端发出到写入完成，经历以下完整流程：

```
Client
  |
  v
Coordinating Node（接收请求）
  |
  |  shard = hash(_routing) % number_of_shards
  v
Primary Shard（所在的 Data Node）
  |
  |  同步写入
  v
Replica Shard(s)（所在的 Data Node(s)）
  |
  v
Response -> Client
```

### 3.1 路由计算

ES 使用一个确定性的哈希公式来决定文档该落入哪个分片：

```
shard_num = hash(_routing) % number_of_shards
```

默认情况下 `_routing` 的值等于文档的 `_id`。这意味着相同 `_id` 的文档永远会被路由到同一个分片，保证了更新和获取操作的一致性。

### 3.2 为什么主分片数不可更改

路由公式中的 `number_of_shards` 是除数。如果索引创建后修改了主分片数，已有文档的路由结果就会改变，导致按 `_id` 获取文档时找不到正确的分片。这就是 ES 将 `number_of_shards` 设为创建后不可变的根本原因。

### 3.3 写入确认机制

默认情况下，写入操作在主分片和所有 in-sync 副本都写入成功后才返回确认。可以通过 `wait_for_active_shards` 参数调整这一行为，但降低确认级别意味着牺牲数据安全性。

## 4. 查询路径详解

ES 的分布式查询采用经典的 scatter-gather 模型，分为两个阶段。

### 4.1 Query Phase（查询阶段）

1. 协调节点接收到搜索请求后，将请求广播到索引的所有相关分片（每个分片组选主本或副本之一）。
2. 每个分片在本地执行查询，根据 `from + size` 构建一个本地优先队列，返回轻量级结果：仅包含 `_id` 和 `_score`（以及排序值）。
3. 协调节点收集所有分片的结果，进行全局排序，选出最终的 top-N 文档。

### 4.2 Fetch Phase（获取阶段）

1. 协调节点根据 Query Phase 确定的 top-N 文档列表，向持有这些文档的分片发送 multi-get 请求。
2. 各分片返回完整的 `_source` 文档内容。
3. 协调节点将完整文档组装后返回给客户端。

### 4.3 深度分页的代价

如果请求 `from=10000, size=10`，每个分片需要在本地准备 10010 条结果返回给协调节点。假设有 5 个分片，协调节点需要汇总排序 50050 条记录。这就是深度分页性能急剧下降的原因。生产中推荐使用 `search_after` 或 Scroll API 来替代深度分页。

## 5. 分片规划最佳实践

分片数量的选择直接影响集群的性能、稳定性和扩展能力。

### 5.1 单分片建议大小

官方推荐每个分片的大小控制在 10-50GB 之间。这个范围在检索性能、段合并效率和恢复速度之间取得了较好的平衡。

### 5.2 分片太少的问题

- 无法充分利用多节点的并行能力。
- 单个分片承受的写入和查询压力过大。
- 数据增长后无法通过增加节点来分散负载（因为分片数是固定的）。

### 5.3 分片太多的问题

- 每个分片都是一个 Lucene 索引，占用文件句柄、内存和 CPU 资源。
- Master 节点需要维护所有分片的元数据，分片过多会加重集群状态管理的负担。
- 查询时协调节点需要合并更多分片的结果，增加网络和计算开销。
- 经验公式：每 GB heap 内存约承载 20 个分片。一个 30GB heap 的节点建议管理不超过 600 个分片。

### 5.4 规划要点

| 参数 | 特性 | 建议 |
|------|------|------|
| `number_of_shards` | 创建后不可变 | 根据预估数据量提前规划 |
| `number_of_replicas` | 可随时动态调整 | 默认 1，读压力大时增加 |

规划时的思考路径：预估索引的总数据量，除以单分片目标大小（如 30GB），得到主分片数。再根据可用性需求确定副本数。确保总分片数（主分片 x (1 + 副本数)）不超过集群的承载能力。

## 6. 集群状态与自愈机制

### 6.1 三种集群状态

| 状态 | 含义 |
|------|------|
| **green** | 所有主分片和副本分片均已分配，集群完全健康 |
| **yellow** | 所有主分片已分配，但存在未分配的副本分片。数据完整可用，但高可用性受损 |
| **red** | 存在未分配的主分片，部分数据不可用 |

### 6.2 节点宕机后的自动恢复过程

假设一个 3 节点集群，索引配置为 3 主分片 1 副本（共 6 个分片）：

```
正常状态 (green):
  Node-1: [P0] [R2]
  Node-2: [P1] [R0]
  Node-3: [P2] [R1]

Node-2 宕机:
  Node-1: [P0] [R2]
  Node-2: --- offline ---
  Node-3: [P2] [R1]

  -> P1 丢失，R0 在 Node-2 上也丢失
  -> 但 R1 在 Node-3 上，Master 将 R1 提升为新的 P1
  -> 集群状态变为 yellow（P1 和 P2 缺少副本）

自愈完成 (green):
  Node-1: [P0] [R1] [R2]
  Node-3: [P2] [P1] [R0]

  -> Master 在剩余节点上创建缺失的副本
  -> 数据通过网络从主分片复制到新副本
  -> 集群恢复 green
```

### 6.3 关键约束

- 主分片和它的副本绝对不会被分配到同一个节点。如果集群节点数少于 `1 + number_of_replicas`，部分副本将无法分配，集群状态为 yellow。
- 恢复过程会消耗网络带宽和磁盘 IO。可以通过 `cluster.routing.allocation.node_concurrent_recoveries` 控制并发恢复数量，避免恢复风暴影响在线服务。

## 7. 路由策略与热点问题

### 7.1 默认路由

默认情况下，ES 使用文档 `_id` 的哈希值进行路由：

```
shard_num = hash(_id) % number_of_shards
```

由于 `_id` 通常是随机的 UUID 或递增 ID，文档会比较均匀地分布在各个分片上。

### 7.2 自定义 routing

在某些业务场景下，你可能希望将同一用户或同一租户的数据集中到同一个分片上，以提高查询效率（避免广播到所有分片）。此时可以使用 `_routing` 参数：

```bash
# 写入时指定 routing
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/orders/_doc/1?routing=user_100" \
  -H "Content-Type: application/json" \
  -d '{"user_id": "user_100", "amount": 99.9}'
```

查询时也必须带上相同的 routing 值，否则 ES 仍会广播到所有分片：

```bash
curl -u "$ES_USER:$ES_PASS" "$ES_URL/orders/_search?routing=user_100&pretty" \
  -H "Content-Type: application/json" \
  -d '{"query": {"term": {"user_id": "user_100"}}}'
```

### 7.3 热点分片问题

自定义 routing 的风险在于数据倾斜。如果某些 routing 值对应的数据量远大于其他值，就会产生热点分片。表现为：

- 某些分片的磁盘使用量远超平均值。
- 某些节点的 CPU 和 IO 持续偏高。
- 查询延迟不均匀。

解决方案：

- 使用 `index.routing_partition_size` 将同一 routing 值的数据分散到多个分片中（而非仅一个），公式变为：`shard = (hash(_routing) + hash(_id) % routing_partition_size) % number_of_shards`。
- 在业务侧设计更细粒度的 routing 键，避免出现超级大户。
- 通过 `_cat/shards` 和 `_cat/allocation` 监控分片大小分布。

## 总结

- 节点通过 `cluster.name` 组成集群，通过角色分离实现职责隔离。
- 分片是数据分布与并行计算的基本单位，每个分片就是一个独立的 Lucene 索引。
- 副本既提供高可用保障，又扩展读吞吐量。
- 写入通过路由公式定位主分片，再同步到副本；查询通过 Query-Fetch 两阶段完成。
- 分片规划需要在扩展性和管理开销之间找到平衡，单分片 10-50GB 是推荐范围。
- 集群具备自愈能力，节点宕机后会自动提升副本并重建缺失的拷贝。
- 自定义路由能提升特定查询效率，但需警惕数据倾斜引发热点。

## 实战（curl）

以下所有命令使用环境变量 `$ES_USER`、`$ES_PASS`、`$ES_URL`，请根据实际环境提前设置。

```bash
# --------------------------------------------------
# 1. 查看集群健康状态
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cluster/health?pretty"

# --------------------------------------------------
# 2. 查看集群中所有节点及其角色
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/nodes?v&h=name,ip,role,master,heap.percent,disk.used_percent"

# --------------------------------------------------
# 3. 创建一个指定分片和副本数的索引
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/shard_demo" \
  -H "Content-Type: application/json" \
  -d '{
    "settings": {
      "number_of_shards": 3,
      "number_of_replicas": 1
    }
  }'

# --------------------------------------------------
# 4. 查看该索引的分片分布
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/shards/shard_demo?v&s=shard,prirep"

# --------------------------------------------------
# 5. 查看各节点的磁盘分配情况
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/allocation?v"

# --------------------------------------------------
# 6. 动态调整副本数（应对突发读流量）
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/shard_demo/_settings" \
  -H "Content-Type: application/json" \
  -d '{"index": {"number_of_replicas": 2}}'

# 验证：总分片数应变为 3 主 + 6 副 = 9
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/shards/shard_demo?v&s=shard,prirep"

# --------------------------------------------------
# 7. 使用自定义 routing 写入文档
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/shard_demo/_doc/1?routing=tenant_A&refresh=wait_for" \
  -H "Content-Type: application/json" \
  -d '{"tenant": "tenant_A", "message": "hello from tenant A"}'

curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/shard_demo/_doc/2?routing=tenant_A&refresh=wait_for" \
  -H "Content-Type: application/json" \
  -d '{"tenant": "tenant_A", "message": "another doc from tenant A"}'

curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/shard_demo/_doc/3?routing=tenant_B&refresh=wait_for" \
  -H "Content-Type: application/json" \
  -d '{"tenant": "tenant_B", "message": "hello from tenant B"}'

# --------------------------------------------------
# 8. 查看 routing 会命中哪些分片
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" "$ES_URL/shard_demo/_search_shards?routing=tenant_A&pretty"

# --------------------------------------------------
# 9. 带 routing 的定向查询（仅命中 tenant_A 所在分片）
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" "$ES_URL/shard_demo/_search?routing=tenant_A&pretty" \
  -H "Content-Type: application/json" \
  -d '{
    "query": {
      "term": { "tenant": "tenant_A" }
    }
  }'

# --------------------------------------------------
# 10. 恢复副本数为 1 并清理
# --------------------------------------------------
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/shard_demo/_settings" \
  -H "Content-Type: application/json" \
  -d '{"index": {"number_of_replicas": 1}}'
```

## 实战（Java SDK）

以下代码使用 Elasticsearch Java Client（`co.elastic.clients:elasticsearch-java`）。

```java
import co.elastic.clients.elasticsearch.ElasticsearchClient;
import co.elastic.clients.elasticsearch.cluster.HealthResponse;
import co.elastic.clients.elasticsearch.core.*;
import co.elastic.clients.elasticsearch.indices.*;
import co.elastic.clients.json.JsonData;
import java.util.Map;

public class Chapter03ShardDemo {

    private final ElasticsearchClient client;

    public Chapter03ShardDemo(ElasticsearchClient client) {
        this.client = client;
    }

    /** 查看集群健康状态 */
    public void clusterHealth() throws Exception {
        HealthResponse h = client.cluster().health();
        System.out.printf("Cluster: %s, Status: %s, Nodes: %d, Active shards: %d, Unassigned: %d%n",
            h.clusterName(), h.status(), h.numberOfNodes(), h.activeShards(), h.unassignedShards());
    }

    /** 创建指定分片数的索引 */
    public void createIndex() throws Exception {
        if (client.indices().exists(e -> e.index("shard_demo")).value()) {
            client.indices().delete(d -> d.index("shard_demo"));
        }
        CreateIndexResponse resp = client.indices().create(c -> c
            .index("shard_demo")
            .settings(s -> s.numberOfShards("3").numberOfReplicas("1"))
        );
        System.out.println("Index created: " + resp.acknowledged());
    }

    /** 查看分片分布 */
    public void catShards() throws Exception {
        client.cat().shards(s -> s.index("shard_demo")).valueBody().forEach(s ->
            System.out.printf("Shard %-3s | %-7s | %-10s | Node: %s%n",
                s.shard(), s.prirep(), s.state(), s.node())
        );
    }

    /** 动态调整副本数 */
    public void updateReplicas(int replicas) throws Exception {
        client.indices().putSettings(p -> p
            .index("shard_demo")
            .settings(s -> s.numberOfReplicas(String.valueOf(replicas)))
        );
    }

    /** 使用自定义 routing 写入文档 */
    public void indexWithRouting() throws Exception {
        client.index(i -> i.index("shard_demo").id("1").routing("tenant_A")
            .document(Map.of("tenant", "tenant_A", "message", "hello from tenant A"))
            .refresh(co.elastic.clients.elasticsearch._types.Refresh.WaitFor));

        client.index(i -> i.index("shard_demo").id("2").routing("tenant_A")
            .document(Map.of("tenant", "tenant_A", "message", "another doc"))
            .refresh(co.elastic.clients.elasticsearch._types.Refresh.WaitFor));
    }

    /** 查看 routing 命中的分片 */
    public void searchShards() throws Exception {
        SearchShardsResponse resp = client.searchShards(s -> s
            .index("shard_demo").routing("tenant_A"));
        resp.shards().forEach(group -> group.forEach(shard ->
            System.out.printf("  -> node: %s, shard: %s, primary: %s%n",
                shard.node(), shard.shard(), shard.primary())
        ));
    }

    /** 带 routing 的定向查询 */
    public void searchWithRouting() throws Exception {
        SearchResponse<JsonData> resp = client.search(s -> s
            .index("shard_demo").routing("tenant_A")
            .query(q -> q.term(t -> t.field("tenant").value("tenant_A"))),
            JsonData.class);
        System.out.println("Hits: " + resp.hits().total().value());
        resp.hits().hits().forEach(hit ->
            System.out.println("  " + hit.id() + " -> " + hit.source()));
    }
```
