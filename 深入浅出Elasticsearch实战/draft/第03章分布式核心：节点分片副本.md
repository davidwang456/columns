# 背景

# 第03章 分布式核心：节点分片副本

单机检索在数据规模上涨后会很快触顶。ES 通过节点、分片、副本实现横向扩展与高可用，但这三个概念若理解不清，就会出现“分片乱配、性能反降、故障难恢复”的问题。

## 本章目标
- 理解 ES 横向扩展与高可用的基本机制。
- 明确分片与副本如何影响性能和稳定性。

## 1. 三个核心概念
- **节点（Node）**：运行 ES 的实例。
- **分片（Shard）**：索引数据的逻辑分块。
- **副本（Replica）**：主分片的复制副本，用于容灾与读扩展。

## 2. 写入和查询是怎么走的
### 2.1 写入路径
- 文档根据路由规则进入某个主分片。
- 主分片成功后再同步到副本分片。

### 2.2 查询路径
- 协调节点把请求发到相关分片。
- 各分片返回结果后聚合输出。

## 3. 分片不是越多越好
- 分片太少：扩展性不足，单分片压力大。
- 分片太多：管理开销高，查询协调成本上升。

经验上应按数据量、节点数、查询模式综合规划。

## 4. 热点问题与路由策略
- 如果某些路由键过热，会导致单分片被打爆。
- 需要在业务侧设计更均衡的路由策略，避免倾斜。

# 总结
- 分片决定扩展能力，副本决定可用性与读能力。
- 合理分片规划是 ES 成败的第一步。

## 练习题
1. 设计一个 3 节点集群的分片与副本方案。  
2. 模拟热点路由，观察查询延迟变化。  
3. 解释“为什么同样数据量，分片过多反而更慢”。  

## 实战（curl）

```bash
# 1) 查看集群与节点
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cluster/health?pretty"
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/nodes?v"

# 2) 查看分片分布
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/shards/$IDX?v"

# 3) 用自定义 routing 写入并验证热点
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/$IDX/_doc/r1?routing=hot_key&refresh=wait_for" \
  -H "Content-Type: application/json" -d '{"name":"hot-doc-1"}'
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_search_shards/$IDX?routing=hot_key&pretty"
```

## 实战（Java SDK）

```java
// 查看分片路由信息（简化示例）
var shards = client.searchShards(s -> s.index("products_v1").routing("hot_key"));
System.out.println(shards.nodes().size());
```
