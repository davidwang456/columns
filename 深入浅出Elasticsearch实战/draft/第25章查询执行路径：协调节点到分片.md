

# 第25章 查询执行路径：协调节点到分片

# 背景

同一条查询在不同集群上表现差异巨大，根因往往不在查询本身，而在执行链路。一条看似简单的 `match` 查询，从客户端发出到最终响应返回，在 ES 内部要经历 HTTP 解析、分片路由、分布式扇出、本地 Lucene 搜索、全局排序、文档回取等多个阶段。任何一个环节出现瓶颈，都会拖慢整体延迟。

理解查询的完整生命周期，是性能调优从"凭感觉"走向"证据驱动"的前提。

## 本章目标

- 掌握一条搜索请求从客户端到返回结果的完整生命周期。
- 深入理解 Query Phase 和 Fetch Phase 的职责、执行细节与性能特征。
- 理解 DFS Query Then Fetch 模式解决的评分偏差问题。
- 掌握分片数、preference 参数、Adaptive Replica Selection 对查询路由和性能的影响。
- 学会使用 `_search_shards` API 和 Profile API 定位查询瓶颈。

---

## 1. 查询的完整生命周期

一条搜索请求从客户端到最终响应，经历以下阶段：

```
客户端（curl / Java SDK / Kibana）
  │
  │  HTTP 请求（POST /my_index/_search）
  ▼
接收节点（任意节点均可接收 HTTP 请求）
  │
  │  该节点自动成为本次请求的"协调节点"（Coordinating Node）
  │  解析请求体 → 构建 SearchRequest + SearchSourceBuilder
  │  根据索引名解析目标索引（含别名展开、数据流解析）
  │  根据 Cluster State 确定目标分片列表（primary 或 replica 均可）
  ▼
===== Query Phase =====
  │
  │  协调节点向每个目标分片发送 Query 请求（并行扇出）
  │  每个分片在本地 Lucene Index 上执行查询
  │  返回 top-N 的 {doc_id, score, sort_values}（不含文档内容）
  │
  ▼
协调节点收集所有分片的局部 top-N
  │  全局排序 → 选出最终 top-N（from + size 条）
  ▼
===== Fetch Phase =====
  │
  │  协调节点只向持有最终 top-N 文档的分片发送 Fetch 请求
  │  （不是所有分片，只是持有目标文档的那几个分片）
  │  分片返回完整文档内容（_source、highlight、stored_fields 等）
  │
  ▼
协调节点组装最终响应
  │  合并 hits、aggregations、suggest 等
  │  序列化为 JSON
  ▼
客户端收到响应
```

**关键洞察**：Query Phase 是"广播式"（所有相关分片都参与），Fetch Phase 是"定点式"（只有持有目标文档的分片参与）。这就是 ES 两阶段设计的核心——先用轻量的 doc_id+score 做全局排序，再用定向取回避免传输大量无用文档。

## 2. Query Phase 详解

### 2.1 协调节点解析请求

协调节点收到 HTTP 请求后，执行以下步骤：

1. **解析请求体**：将 JSON 转为 `SearchSourceBuilder` 对象（包含 query、aggs、sort、from、size 等）。
2. **索引解析**：展开别名（alias）、解析数据流（data stream）、应用索引通配符。
3. **确定目标分片**：从 Cluster State 的 routing table 中，找出目标索引的所有分片。对于每个分片，可以选择 primary 或任一 replica——具体选哪个由 Adaptive Replica Selection 决定。
4. **分组与扇出**：将分片按所在节点分组，批量发送 Query 请求到各数据节点。

可以使用 `_search_shards` API 预览这个过程：

```bash
GET /my_index/_search_shards
```

该 API 返回查询将落到哪些分片、在哪些节点上。

### 2.2 分片本地执行

每个分片收到 Query 请求后，在本地 Lucene Index 上执行：

1. **查询重写（Rewrite）**：将高层查询（如 `match`）重写为 Lucene 原语（如 `TermQuery`、`BooleanQuery`）。
2. **创建 Weight**：为查询准备统计信息（term frequency、document frequency 等）。
3. **遍历 Segment**：在每个 Segment 上执行查询，通过倒排索引定位匹配文档。
4. **评分与排序**：使用 BM25（或自定义 Similarity）计算每个匹配文档的得分。
5. **收集 top-N**：通过优先队列（Priority Queue）维护本分片的局部 top-N 结果。

每个分片返回给协调节点的数据非常轻量：

```json
{
  "doc_id": 42,
  "score": 3.14,
  "sort_values": [...]
}
```

**不包含 `_source`**——这是两阶段设计的关键优化点。如果索引有 5 个分片、每个分片返回 top-10，那么 Query Phase 只传输 50 个 `{doc_id, score}` 组合，而非 50 条完整文档。

### 2.3 协调节点全局排序

协调节点收到所有分片的局部 top-N 后：

1. 将所有结果放入一个全局优先队列。
2. 按 score（或自定义 sort）全局排序。
3. 取出 `from` 到 `from + size` 范围内的文档——这就是最终要返回的文档。
4. 记录这些文档分别在哪些分片上。

**性能关键**：每个分片返回的数量不是 `size`，而是 `from + size`。这就是深分页性能差的根因——如果 `from=10000, size=10`，每个分片都要返回 10010 条，协调节点要排序 `分片数 × 10010` 条记录。

## 3. Fetch Phase 详解

### 3.1 定点取回

Query Phase 完成后，协调节点已经知道最终 top-N 文档的 `{shard_id, doc_id}`。Fetch Phase 只需要：

1. **按分片分组**：将最终 top-N 文档按所在分片分组。
2. **发送 Fetch 请求**：只向持有目标文档的分片发送请求（Multi Get 语义）。
3. **分片返回完整内容**：`_source`、`highlight`、`stored_fields`、`fields` 等。

假设最终 top-10 文档分布在 3 个分片上（分片A有4条、分片B有3条、分片C有3条），那么只有这 3 个分片会收到 Fetch 请求，其余分片不参与。

### 3.2 Fetch 阶段的开销来源

Fetch 阶段的耗时通常远小于 Query 阶段，但以下情况可能成为瓶颈：

| 场景 | 原因 | 优化方向 |
| --- | --- | --- |
| `_source` 过大 | 文档体积大，序列化和网络传输慢 | `_source` filtering 或 `stored_fields` |
| `highlight` 复杂 | 高亮需要重新分析文本 | 使用 `fvh`（Fast Vector Highlighter）或 `unified` |
| `script_fields` | 每条文档执行脚本 | 预计算字段或使用 `runtime_fields` |
| 返回条数过多 | `size` 设置过大 | 减小 size，使用 `search_after` 分批 |

### 3.3 组装最终响应

协调节点将 Fetch 结果与 Query Phase 的排序信息合并，构建最终的 `SearchResponse`：

```json
{
  "took": 15,
  "timed_out": false,
  "_shards": { "total": 5, "successful": 5, "skipped": 0, "failed": 0 },
  "hits": {
    "total": { "value": 1234, "relation": "eq" },
    "max_score": 3.14,
    "hits": [
      { "_index": "my_index", "_id": "42", "_score": 3.14, "_source": {...} },
      ...
    ]
  }
}
```

`took` 时间包含 Query Phase + Fetch Phase + 协调节点处理时间，但**不包含**网络排队和 HTTP 层开销。

## 4. DFS Query Then Fetch

### 4.1 问题：分片间评分偏差

BM25 评分依赖 term statistics（如 document frequency、average field length）。默认模式下，每个分片使用**本地统计信息**计算评分。当数据分布不均匀时，同一个 term 在不同分片上的 IDF 值可能差异显著，导致评分不一致。

**典型场景**：索引只有 2 个分片，分片 A 有 100 万文档（其中 1000 条包含"elasticsearch"），分片 B 只有 100 条文档（其中 50 条包含"elasticsearch"）。"elasticsearch" 在分片 A 中的 IDF 很高（稀有），在分片 B 中的 IDF 很低（常见），导致相同内容的文档在不同分片上获得差距很大的分数。

### 4.2 解决：DFS 阶段

`search_type=dfs_query_then_fetch` 在 Query Phase 之前增加一个 **DFS（Distributed Frequency Scatter）** 阶段：

```
===== DFS Phase（额外阶段） =====
  │
  │  协调节点向所有目标分片请求 term statistics
  │  收集每个分片的 document frequency、total term frequency 等
  │  计算全局统计信息（Global Statistics）
  │  将全局统计信息随 Query 请求一起下发给各分片
  ▼
===== Query Phase（使用全局统计信息评分） =====
  │  ...
  ▼
===== Fetch Phase =====
```

### 4.3 何时使用 DFS

| 场景 | 建议 |
| --- | --- |
| 索引数据量大、分布均匀 | 不需要 DFS，默认 `query_then_fetch` 即可 |
| 索引数据量小（< 10 万条） | 考虑使用 DFS，因为小数据量下分片间统计偏差明显 |
| 单分片索引 | 不需要 DFS（只有一个分片，不存在偏差） |
| 多索引联合搜索、数据分布极不均匀 | 考虑使用 DFS |

DFS 的代价是多一轮网络往返，对高吞吐场景影响较大。**大多数生产场景下不需要 DFS**。

```bash
POST /my_index/_search?search_type=dfs_query_then_fetch
{
  "query": { "match": { "title": "elasticsearch" } }
}
```

## 5. 分片数对查询性能的影响

### 5.1 分片越多，协调开销越大

Query Phase 中，**每个分片都必须参与**。分片数量直接决定了：

- **网络扇出数**：协调节点要向 N 个分片发送请求，等待 N 个响应。
- **合并开销**：协调节点要排序 `N × (from + size)` 条记录。
- **线程竞争**：每个节点的 search 线程池需要处理更多的分片级请求。

### 5.2 分片过少也有问题

单分片数据量过大（如 200GB）会导致：

- 单次查询在大 Segment 上遍历时间长。
- 无法利用多节点并行——该分片只在一个节点上。
- Recovery 和 Rebalance 耗时极长。

### 5.3 最佳实践

| 指标 | 建议范围 | 说明 |
| --- | --- | --- |
| 单分片大小 | 10~50 GB | 过小浪费资源，过大影响查询和恢复速度 |
| 每个节点的分片数 | 每 GB heap 不超过 20 个分片 | 避免 Cluster State 和 Segment 元数据过大 |
| 总分片数 | 按数据量倒推 | `总数据量 / 单分片大小 = 分片数` |

**示例**：索引总数据量 200GB，选择单分片 40GB，则 5 个主分片。加上 1 个副本，总共 10 个分片。

## 6. Preference 参数：控制查询路由

`preference` 参数控制协调节点将请求路由到哪些分片副本：

| 值 | 行为 |
| --- | --- |
| `_local` | 优先选择协调节点本地的分片副本（减少网络跳转） |
| `_prefer_nodes:node1,node2` | 优先选择指定节点上的分片副本 |
| `_shards:0,1` | 只在指定分片上执行（可与其他 preference 组合） |
| `_only_local` | 只在本地分片上执行（如果本地没有则返回空） |
| 自定义字符串（如 `user_123`） | 基于字符串哈希选择分片副本，保证同一字符串总是路由到同一副本（Session Affinity） |

### 6.1 Session Affinity（会话亲和）

自定义 preference 值的典型用途是解决"刷新可见性不一致"问题：

如果用户刚写入一条文档，紧接着搜索，可能因为请求被路由到不同的分片副本（primary vs. replica），而 replica 的 refresh 稍有延迟，导致用户"看不到刚写的数据"。

使用 `preference=user_123`，可以确保同一用户的连续请求总是落在同一个分片副本上，避免这种不一致体验。

```bash
POST /my_index/_search?preference=session_abc123
{
  "query": { "match": { "name": "笔记本" } }
}
```

### 6.2 _shards 限定范围

当你知道数据在哪些分片上时（例如通过 routing），可以用 `_shards` 跳过无关分片：

```bash
POST /my_index/_search?preference=_shards:0,1
{
  "query": { "match_all": {} }
}
```

这避免了对其他分片的无谓扇出，对大索引的点查询场景很有用。

## 7. Adaptive Replica Selection（自适应副本选择）

### 7.1 工作原理

ES 默认启用 Adaptive Replica Selection（ARS）。当一个分片有多个副本可选时，协调节点不是简单轮询，而是综合考虑以下因素选择"最优"副本：

- **响应时间**：节点历史响应时间的 EWMA（指数加权移动平均）。
- **队列大小**：目标节点的 search 线程池队列长度。
- **服务时间**：请求在节点上的实际处理时间。

ARS 的目标是避免将请求发送到负载较重或响应较慢的节点上，实现自动化的负载均衡。

### 7.2 效果

ARS 对以下场景效果显著：

- **异构集群**：节点硬件配置不同（如部分 SSD、部分 HDD）。
- **热点分片**：某些分片因为数据特征或缓存状态导致响应时间波动。
- **节点临时性能下降**：GC 暂停、磁盘 IO 毛刺等。

ARS 通过 `cluster.routing.use_adaptive_replica_selection` 控制（默认 `true`）。一般无需关闭。

## 8. _search_shards API：预览查询路由

`_search_shards` API 不执行查询，只返回查询将落到哪些分片和节点上：

```bash
GET /my_index/_search_shards
```

响应示例：

```json
{
  "nodes": {
    "node_id_1": { "name": "node-1", "transport_address": "10.0.0.1:9300" },
    "node_id_2": { "name": "node-2", "transport_address": "10.0.0.2:9300" }
  },
  "shards": [
    [
      { "index": "my_index", "shard": 0, "primary": true, "node": "node_id_1" },
      { "index": "my_index", "shard": 0, "primary": false, "node": "node_id_2" }
    ],
    [
      { "index": "my_index", "shard": 1, "primary": true, "node": "node_id_2" },
      { "index": "my_index", "shard": 1, "primary": false, "node": "node_id_1" }
    ]
  ]
}
```

### 8.1 配合 routing 使用

如果查询指定了 routing 值，`_search_shards` 也会反映：

```bash
GET /my_index/_search_shards?routing=user_123
```

此时返回的 `shards` 只包含 routing 值映射到的那一个分片，而不是所有分片。这可以用来验证 routing 设计是否正确减少了分片扇出。

### 8.2 配合 preference 使用

```bash
GET /my_index/_search_shards?preference=_local
```

可以预览在当前节点上执行查询时，会选择哪些分片副本。

## 9. 性能瓶颈定位：结合 Profile API

Profile API 的输出按照查询执行路径组织，包含 query、collector、fetch 三部分。将执行路径知识与 Profile 输出结合，可以精准定位瓶颈。

### 9.1 Query Phase 瓶颈

在 Profile 输出的 `profile.shards[].searches[].query[]` 中：

- `create_weight` 耗时高：查询初始化昂贵（如展开大量 term 的 fuzzy 查询）。
- `next_doc` / `advance` 耗时高：匹配文档量太大，遍历开销大。
- `score` 耗时高：评分计算复杂（如 `script_score`）。

### 9.2 Collector 瓶颈

在 `profile.shards[].searches[].collector[]` 中：

- `search_top_hits` 耗时高：排序开销大（特别是 `from` 很大时）。
- `aggregation` 耗时高：聚合计算是主要开销。

### 9.3 Fetch Phase 瓶颈

在 `profile.shards[].fetch` 中：

- `load_stored_fields` 耗时高：文档 `_source` 过大。
- `FetchSourcePhase` 耗时高：`_source` filtering 或序列化慢。

### 9.4 协调节点瓶颈

Profile API **不度量**协调节点的合并耗时和网络延迟。如果 Profile 显示每个分片的执行时间都很短，但整体 `took` 很高，说明瓶颈在：

- **网络延迟**：分片分布在远距离节点上。
- **协调节点排序开销**：`from` 过大导致全局排序数据量大。
- **队列等待**：search 线程池队列拥堵。

此时应检查 `_cat/thread_pool/search` 和网络延迟。

## 10. 查询路径优化清单

| 优化方向 | 具体措施 | 收益 |
| --- | --- | --- |
| 减少分片扇出 | 使用 routing 让相关数据落在同一分片 | 减少 Query Phase 参与分片数 |
| 控制返回数量 | 使用 `search_after` 替代深分页 | 避免 `from + size` 膨胀 |
| 减轻 Fetch 开销 | `_source` filtering，只返回必要字段 | 减少网络传输和序列化 |
| 利用 filter 缓存 | 结构化条件放在 `bool.filter` 中 | 跳过评分 + Node Query Cache |
| 合理设置分片数 | 单分片 10-50GB | 平衡并行度与协调开销 |
| 使用 preference | Session Affinity 或 `_local` | 减少网络跳转、一致性体验 |
| 限制并发分片请求 | `max_concurrent_shard_requests` | 控制协调节点压力 |

---

# 总结

- 查询执行分为 Query Phase（广播式扇出，返回轻量的 doc_id + score）和 Fetch Phase（定点式取回，只取最终 top-N 的完整文档）。两阶段设计是 ES 分布式搜索的核心架构。
- 协调节点是查询链路的"中枢"——它负责分片路由、全局排序、Fetch 分发和响应组装。深分页的性能问题本质上就是协调节点的排序压力。
- DFS Query Then Fetch 通过额外一轮网络往返收集全局 term statistics，解决分片间评分偏差问题，适用于小数据量或数据分布极不均匀的场景。
- 分片数是性能的放大器——多了增加协调开销，少了限制并行度。单分片 10-50GB 是平衡点。
- Adaptive Replica Selection 自动选择响应最快的分片副本，无需手动配置。
- `_search_shards` 是诊断查询路由问题的利器，Profile API 的 query/collector/fetch 三部分对应查询执行的三个阶段。

## 练习题

1. 对一个 5 分片索引执行 `_search_shards`，画出分片在各节点的分布图。
2. 分别使用 `from/size` 深分页（from=10000）和 `search_after` 查询，通过 Profile API 对比两者的协调节点开销。
3. 设置 `preference=my_session_id` 连续执行 3 次查询，验证每次是否落在同一分片副本上。
4. 对一个小索引（< 1000 条文档、2 个分片），分别使用 `query_then_fetch` 和 `dfs_query_then_fetch`，对比同一文档的评分差异。
5. 使用 Profile API 分析一条包含聚合的查询，分别找出 query、collector、fetch 三部分的耗时占比。

      "user_id":  { "type": "keyword" }
    }}
  }'

curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/query_path_demo/_bulk?refresh=wait_for" \

## 实战（Java SDK）

```java
// ---------- 查看 _search_shards ----------
var searchShards = client.searchShards(s -> s.index("query_path_demo"));
System.out.println("节点数: " + searchShards.nodes().size());
System.out.println("分片组数: " + searchShards.shards().size());
searchShards.shards().forEach(group ->
    group.forEach(shard ->
        System.out.println("  分片 " + shard.shard() + " primary=" + shard.primary()
            + " 节点=" + shard.node())));

// ---------- 带 routing 的 _search_shards ----------
var routedShards = client.searchShards(s -> s.index("query_path_demo").routing("u001"));
System.out.println("routing=u001 命中分片组数: " + routedShards.shards().size());

// ---------- Profile 查询（观察 query/collector/fetch） ----------
var profileResp = client.search(s -> s.index("query_path_demo")
    .profile(true)
    .query(q -> q.bool(b -> b
        .must(m -> m.match(mt -> mt.field("title").query("Elasticsearch")))
        .filter(f -> f.term(t -> t.field("category").value("book"))))),
    Map.class);

System.out.println("took: " + profileResp.took() + "ms");
if (profileResp.profile() != null) {
    profileResp.profile().shards().forEach(shard -> {
        System.out.println("=== 分片: " + shard.id() + " ===");
        shard.searches().forEach(search -> {
            search.query().forEach(pq -> {
                System.out.println("  Query: " + pq.type()
                    + " time=" + pq.timeInNanos() + "ns");
            });
            search.collector().forEach(col ->
                System.out.println("  Collector: " + col.name()
                    + " time=" + col.timeInNanos() + "ns"));
        });
        if (shard.fetch() != null) {
            System.out.println("  Fetch time: " + shard.fetch().timeInNanos() + "ns");
        }
    });
}

// ---------- DFS Query Then Fetch ----------
var dfsResp = client.search(s -> s.index("query_path_demo")
    .searchType(SearchType.DfsQueryThenFetch)
    .query(q -> q.match(m -> m.field("title").query("Elasticsearch 实战"))),
    Map.class);
System.out.println("DFS 模式 took: " + dfsResp.took() + "ms, hits: "
    + dfsResp.hits().total().value());

// ---------- Preference 参数 ----------
var prefResp = client.search(s -> s.index("query_path_demo")
    .preference("my_session_abc")
    .query(q -> q.match(m -> m.field("title").query("Elasticsearch"))),
    Map.class);
System.out.println("Preference 查询 hits: " + prefResp.hits().total().value());

// ---------- search_after 分页 ----------
var page1 = client.search(s -> s.index("query_path_demo")
    .size(2)
    .sort(so -> so.field(f -> f.field("price").order(SortOrder.Desc)))
    .sort(so -> so.field(f -> f.field("_id").order(SortOrder.Asc)))
    .query(q -> q.matchAll(m -> m)),
    Map.class);

System.out.println("第一页:");
page1.hits().hits().forEach(h ->
    System.out.println("  id=" + h.id() + " sort=" + h.sort()));

if (!page1.hits().hits().isEmpty()) {
    var lastHit = page1.hits().hits().get(page1.hits().hits().size() - 1);
    var page2 = client.search(s -> s.index("query_path_demo")
        .size(2)
        .sort(so -> so.field(f -> f.field("price").order(SortOrder.Desc)))
        .sort(so -> so.field(f -> f.field("_id").order(SortOrder.Asc)))
        .query(q -> q.matchAll(m -> m))
        .searchAfter(lastHit.sort()),
        Map.class);

    System.out.println("第二页:");
    page2.hits().hits().forEach(h ->
        System.out.println("  id=" + h.id() + " sort=" + h.sort()));
}

// ---------- 清理 ----------
client.indices().delete(d -> d.index("query_path_demo"));
```
