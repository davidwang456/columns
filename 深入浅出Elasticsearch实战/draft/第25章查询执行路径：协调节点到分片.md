# 背景

# 第25章 查询执行路径：协调节点到分片

同一条查询在不同集群表现差异很大，核心原因通常在执行链路。理解协调节点与分片执行关系，才能精准定位瓶颈。

## 本章目标
- 了解一次查询在 ES 内部的主要阶段。
- 用执行路径思维定位性能瓶颈。

## 1. 查询的两个主要阶段
- Query Phase：各分片计算候选结果。
- Fetch Phase：拉取文档详情并组装响应。

## 2. 协调节点的角色
协调节点负责：
- 分发请求到相关分片
- 汇总并排序分片结果
- 返回最终响应

## 3. 性能影响点
- 分片过多会增加协调成本。
- 复杂排序和深分页会放大开销。

## 4. 实践建议
- 按查询特征规划分片数量。
- 用 profile 识别 query/fetch 各自耗时占比。

# 总结
- 查询性能不是单点问题，而是链路问题。
- 先定位“慢在哪个阶段”，再做针对性优化。

## 练习题
1. 对一条慢查询拆分 query/fetch 耗时。  
2. 比较不同分片数对查询延迟的影响。  
3. 给出一条“协调成本过高”的优化建议。  

## 实战（curl）

```bash
# 查看查询 profile（观察 query/fetch）
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/$IDX/_search?pretty" \
  -H "Content-Type: application/json" \
  -d '{"profile":true,"query":{"match":{"name":"笔记本"}}}'

# 查看 search_shards，理解请求将落到哪些分片
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_search_shards/$IDX?pretty"
```
