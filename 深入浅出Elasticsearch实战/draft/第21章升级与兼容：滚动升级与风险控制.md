# 第21章 升级与兼容：滚动升级与风险控制

# 背景

版本升级是 Elasticsearch 运维中风险最高的常规操作之一。真正困难的不只是"怎么升"，而是"怎么在可回滚、可验证的前提下安全地升"。滚动升级（Rolling Upgrade）允许逐节点升级而不中断服务，是 Minor / Patch 版本升级的首选方案；但它有严格的前提条件，并且每一步都需要对应的验证动作。本章将版本策略、升级流程和风险控制组合成一套可落地的操作手册。

## 本章目标

- 理解 Elasticsearch 版本号语义（Major / Minor / Patch）及其对兼容性的影响。
- 掌握升级前 Checklist 的完整内容：Release Notes、Breaking Changes、Deprecation API、备份验证。
- 手把手走通 Rolling Upgrade 8 步流程，每一步配有对应的 curl 命令。
- 建立升级风险控制体系：窗口选择、每批验证项、暂停与回滚触发条件。
- 熟悉跨大版本升级路径（7 → 8 → 9）及 Reindex 要求。

---

## 1. ES 版本策略

### 1.1 语义化版本号

Elasticsearch 遵循 `Major.Minor.Patch` 的版本号规范：

```
8.15.2
│  │  └── Patch：Bug 修复，无行为变更
│  └───── Minor：新功能、新 API，向后兼容
└──────── Major：可能包含破坏性变更（Breaking Changes）
```

| 升级类型 | 示例 | 兼容性 | 推荐方式 |
|---------|------|--------|---------|
| Patch | 8.15.1 → 8.15.2 | 完全兼容 | Rolling Upgrade |
| Minor | 8.14.x → 8.15.x | 向后兼容，可能废弃部分 API | Rolling Upgrade |
| Major | 7.x → 8.x | 可能有破坏性变更 | Rolling Upgrade（相邻大版本） |

### 1.2 兼容性矩阵

滚动升级要求集群中所有节点的版本差距不超过一个 Minor。跨大版本滚动升级只支持**相邻大版本的最后一个 Minor**到下一个大版本：

```
7.17.x → 8.x.x    ✅ 支持 Rolling Upgrade
7.16.x → 8.x.x    ❌ 必须先升到 7.17.x
6.x.x  → 8.x.x    ❌ 必须先经过 7.x
```

### 1.3 索引兼容性

Elasticsearch 可以读取**前一个大版本**创建的索引。更早版本创建的索引必须在升级前 Reindex：

```
8.x 能读取 7.x 创建的索引   ✅
8.x 不能读取 6.x 创建的索引  ❌  → 必须在 7.x 中 Reindex
```

## 2. 升级前 Checklist

### 2.1 文档与变更审查

1. **阅读 Release Notes**：目标版本的所有 Release Notes，关注 Bug Fixes 和已知问题。
2. **阅读 Breaking Changes**：Major 升级时重点审查，Minor 升级也应浏览 Deprecation 列表。
3. **阅读 Migration Guide**：官方每个 Major 版本都提供完整的迁移指南。

### 2.2 Deprecation API 检查

```bash
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_migration/deprecations?pretty"
```

返回四类废弃警告：`cluster_settings`、`node_settings`、`index_settings`、`ml_settings`。`critical` 级别必须在升级前修复。

### 2.3 插件与客户端兼容

- **插件**：确认所有已安装插件在目标版本可用，执行 `elasticsearch-plugin list` 逐一核对。
- **Java SDK**：elasticsearch-java 客户端 Minor 版本应 ≥ 集群版本。
- **Logstash / Beats / Kibana**：版本必须与 ES 保持一致。
- **Painless 脚本**：审查是否使用了废弃的 API 或语法。

### 2.4 Snapshot 备份

```bash
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/_snapshot/my_backup/pre_upgrade_snapshot?wait_for_completion=true&pretty"
```

**Snapshot 必须验证可恢复**——仅创建成功不够，必须在测试环境恢复验证。

### 2.5 测试环境预演

在与生产配置一致的测试环境中完整走一遍升级流程，验证核心查询结果不变、写入链路正常、自定义插件无报错。

## 3. 滚动升级 8 步流程

核心原则：**每次只下线一个节点，升级后重新加入集群，确认集群恢复 green 后再升级下一个**。

### 步骤 1：检查集群健康

```bash
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cluster/health?pretty"
```

确认 `status` 为 `green`，`unassigned_shards` 为 0。集群不是 green 则先解决问题。

### 步骤 2：禁用分片分配

```bash
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/_cluster/settings?pretty" \
  -H 'Content-Type: application/json' -d'
{
  "persistent": {
    "cluster.routing.allocation.enable": "primaries"
  }
}'
```

设为 `primaries` 后，节点下线时集群不会将该节点上的副本分片搬迁到其他节点——避免大量不必要的数据拷贝。

### 步骤 3：停止非必要操作并执行 Flush

```bash
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/_ml/set_upgrade_mode?enabled=true&pretty"
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/_flush?pretty"
```

Flush 确保 Translog 中的数据持久化到 Lucene Segment，减少节点重启后的恢复时间。

### 步骤 4：停止节点、升级、启动

```bash
sudo systemctl stop elasticsearch
sudo rpm -Uvh elasticsearch-8.16.0-x86_64.rpm    # 以 RPM 为例
sudo systemctl start elasticsearch
```

升级期间不要同时变更 `elasticsearch.yml` 中的配置。检查配置文件是否被包管理器覆盖。

### 步骤 5：等待节点加入集群

```bash
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/nodes?v&h=name,version,master,node.role"
```

确认升级的节点出现在列表中且版本号已更新。

### 步骤 6：恢复分片分配

```bash
curl -u "$ES_USER:$ES_PASS" -X PUT "$ES_URL/_cluster/settings?pretty" \
  -H 'Content-Type: application/json' -d'
{
  "persistent": {
    "cluster.routing.allocation.enable": null
  }
}'
```

设为 `null` 恢复默认行为（`all`），集群开始重新分配副本分片。

### 步骤 7：等待集群恢复 Green

```bash
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cluster/health?wait_for_status=green&timeout=5m&pretty"
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/recovery?v&active_only=true&h=index,shard,stage,bytes_percent"
```

**只有集群恢复到 green 状态后，才能对下一个节点重复步骤 2–7**。

### 步骤 8：对下一个节点重复

推荐升级顺序：Data-only 节点 → Ingest 节点 → Coordinating-only 节点 → Master-eligible 节点（最后升级当前 master）。全部完成后关闭 ML 升级模式：

```bash
curl -u "$ES_USER:$ES_PASS" -X POST "$ES_URL/_ml/set_upgrade_mode?enabled=false&pretty"
```

## 4. 全集群重启升级

### 4.1 适用场景

- 跨两个 Major 版本的中间过渡（如 6.x → 7.x 的部分场景）。
- 重大配置变更需要同时生效。

### 4.2 流程概览

```
1. 禁用分片分配 → 2. Flush 所有索引 → 3. 停止所有节点
→ 4. 升级所有节点 → 5. 先启动 master-eligible 节点
→ 6. 等待集群形成 → 7. 恢复分片分配 → 8. 等待 green
```

**全集群重启意味着停机**——窗口期内所有读写请求都会失败，必须提前与业务方协调。

---

## 5. 升级风险控制

### 5.1 窗口选择

- **避开高峰**：选择写入和查询量最低的时段。
- **预留回滚时间**：窗口时长应至少是预估升级时间的 2 倍。
- **通知上下游**：通知所有依赖 ES 的服务方，约定降级策略。

### 5.2 每批节点升级后的验证项

| 验证项 | 命令 | 通过标准 |
|-------|------|---------|
| 集群状态 | `_cluster/health` | status = green |
| 节点版本 | `_cat/nodes?h=name,version` | 升级节点版本正确 |
| 分片分布 | `_cat/shards?v` | 无 UNASSIGNED 分片 |
| 核心查询 | Smoke Test 查询 | 结果与基线一致 |
| 写入验证 | 写入测试文档并读回 | 成功 |

### 5.3 暂停条件

遇到以下任一情况时，**立即暂停升级**：

- 集群状态变为 `red`。
- 升级节点超过 10 分钟无法加入集群。
- 分片恢复超过 30 分钟未恢复 green。
- Smoke Test 查询结果与基线不一致。
- 日志中出现 `CircuitBreakingException` 或 `OutOfMemoryError`。

### 5.4 回滚触发条件

暂停后问题无法在窗口内解决，执行回滚：停止已升级节点 → 重装旧版本 → 从 Snapshot 恢复（如有必要）→ 启动并验证。**没有备份的升级等于裸奔。**

---

## 6. 跨大版本升级路径

### 6.1 官方推荐路径

```
6.8.x → 7.17.x → 8.x.x → 9.x.x
```

不允许跳版本：`6.x → 8.x` 不可行，`7.x → 9.x` 也不可行。每次只能跨一个 Major。

### 6.2 跨大版本的额外工作

1. **Reindex 旧索引**：8.x 不能读取 6.x 创建的索引，须在 7.x 中先 Reindex。
2. **处理 Breaking Changes**：每个 Major 版本有数十项 Breaking Changes，必须逐项审查。
3. **更新客户端代码**：8.x 中 `RestHighLevelClient` 迁移到 `ElasticsearchClient`。
4. **安全配置变更**：8.x 默认开启安全特性（TLS + 认证），7.x 默认关闭。

---

## 7. 升级后验证清单

全部节点升级完成后，执行 5 项核心验证：

1. **集群健康与版本一致性**：所有节点版本一致、集群 green、master 已选举。
2. **索引与分片完整性**：无 red / yellow 索引，无 UNASSIGNED 分片。
3. **核心查询回归**：对预定义 Smoke Test 查询集执行回归测试，比对排序结果和聚合数值。
4. **写入链路验证**：写入测试文档并读回，确认读写正常。
5. **Deprecation 警告清理**：记录新版本产生的 Deprecation 警告并制定处理计划。

---

## 8. Deprecation API 详解

### 8.1 响应结构

```json
{
  "cluster_settings": [
    { "level": "critical", "message": "Setting [xxx] is deprecated", "url": "https://...", "details": "..." }
  ],
  "node_settings": [...],
  "index_settings": { "my_index": [...] },
  "ml_settings": [...]
}
```

### 8.2 Level 含义

| Level | 含义 | 是否阻塞升级 |
|-------|------|-------------|
| `critical` | 必须在升级前修复，否则升级后功能异常 | 是 |
| `warning` | 建议修复，当前仍可工作但未来版本将移除 | 否 |
| `info` | 信息提示 | 否 |

### 8.3 常见 Deprecation 示例

- `_type` 字段的使用（7.x → 8.x 已移除）。
- 旧版 Mapping 参数（如 `_all`、`include_in_all`）。
- 废弃的集群设置（如 `discovery.zen.minimum_master_nodes` → `cluster.initial_master_nodes`）。

---

# 总结

- 版本号语义决定升级策略——Patch / Minor 用 Rolling Upgrade，跨 Major 需提前评估 Breaking Changes 和 Reindex 需求。
- 升级的核心原则是**可验证、可暂停、可回滚**——Snapshot 是回滚的基石。
- Rolling Upgrade 8 步中，**禁用分片分配**和**等待 green**是最关键的两步——前者避免不必要的数据搬迁，后者确保集群完全恢复后再继续。
- 每批节点升级后的 Smoke Test 不是可选步骤——它是及早发现问题、控制爆炸半径的最有效手段。
- `_migration/deprecations` API 是升级前后最重要的诊断工具——`critical` 级别必须在升级前处理。

## 实战（Java SDK）

```java
// ---------- 集群版本与健康 ----------
var info = client.info();
System.out.println("Cluster: " + info.clusterName()
    + ", version: " + info.version().number());

var health = client.cluster().health();
System.out.println("Status: " + health.status()
    + ", nodes: " + health.numberOfNodes()
    + ", unassigned: " + health.unassignedShards());

// ---------- 节点信息（版本、角色）----------
var nodesInfo = client.nodes().info();
nodesInfo.nodes().forEach((nodeId, nodeInfo) ->
    System.out.println("Node: " + nodeInfo.name()
        + ", version: " + nodeInfo.version()
        + ", roles: " + nodeInfo.roles()));

// ---------- 禁用分片分配 ----------
var disableAlloc = client.cluster().putSettings(s -> s
    .persistent("cluster.routing.allocation.enable",
        co.elastic.clients.json.JsonData.of("primaries")));
System.out.println("Disable allocation: " + disableAlloc.acknowledged());

// ---------- Flush ----------
client.indices().flush(f -> f);

// ---------- 恢复分片分配 ----------
var enableAlloc = client.cluster().putSettings(s -> s
    .persistent("cluster.routing.allocation.enable",
        co.elastic.clients.json.JsonData.of("all")));
System.out.println("Enable allocation: " + enableAlloc.acknowledged());

// ---------- 等待集群 green ----------
var waitHealth = client.cluster().health(h -> h
    .waitForStatus(co.elastic.clients.elasticsearch._types.HealthStatus.Green)
    .timeout(t -> t.time("5m")));
System.out.println("Status: " + waitHealth.status() + ", timed_out: " + waitHealth.timedOut());

// ---------- 升级后写入验证 ----------
client.index(i -> i.index("upgrade_verify")
    .document(Map.of("test", "post_upgrade", "ts", "2025-06-01T00:00:00Z"))
    .refresh(co.elastic.clients.elasticsearch._types.Refresh.WaitFor));

var searchResp = client.search(s -> s.index("upgrade_verify")
    .query(q -> q.match(m -> m.field("test").query("post_upgrade"))),
    Map.class);
System.out.println("Write verify: " + searchResp.hits().total().value() + " hits");

// ---------- 升级后版本确认 ----------
client.nodes().info().nodes().forEach((id, n) ->
    System.out.println("Node: " + n.name() + ", version: " + n.version()));

// ---------- 清理 ----------
client.indices().delete(d -> d.index("upgrade_verify"));
```
