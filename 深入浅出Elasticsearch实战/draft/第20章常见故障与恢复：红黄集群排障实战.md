

# 第20章 常见故障与恢复：红黄集群排障实战

# 背景

集群变红或变黄是 Elasticsearch 运维最常遇到的紧急事件。变红意味着部分数据不可用，变黄意味着高可用性降级——两者都不能被忽视。然而很多团队在面对 unassigned 分片时只会反复重启节点，缺少系统化的排查流程和恢复手段。

本章将建立一套标准化的故障排查流程——从判断严重程度到定位根因、从临时止血到彻底恢复，覆盖 10 种以上常见的 unassigned 原因和对应处置方案。

## 本章目标

- 深入理解 green / yellow / red 三种集群状态的含义和影响。
- 掌握 `_cluster/allocation/explain` API 的完整输出解读。
- 建立 unassigned 分片的系统化排查流程，覆盖 10+ 种常见原因。
- 掌握磁盘水位故障的诊断与恢复步骤。
- 了解节点离线和分片损坏的应急处置手段。
- 建立标准应急 Runbook 和故障复盘模板。

## 1. 集群状态详解

### 1.1 三种状态的含义

| 状态     | 含义                     | 影响                        | 紧急程度 |
| ------ | ---------------------- | ------------------------- | ---- |
| green  | 所有主分片和副本分片都已分配         | 系统正常，具备完整的高可用能力           | 无    |
| yellow | 所有主分片已分配，**部分副本未分配**    | 数据完整可用，但高可用降级——再失去一个节点可能丢数据 | 中    |
| red    | **部分主分片未分配**，数据可能不可用   | 涉及该分片的数据无法读取或写入           | 高    |

### 1.2 状态判断的层级

集群状态是所有索引状态中**最差的那个**——如果有一个索引是 red，集群就是 red。

```bash
# 集群级状态
GET _cluster/health?pretty

# 索引级状态——找出哪些索引不健康
GET _cluster/health?level=indices&pretty&filter_path=status,indices.*.status

# 分片级状态——精确到每个分片
GET _cluster/health?level=shards&pretty
```

### 1.3 常见的 yellow 场景

- **单节点集群**：副本无法分配到同一节点（same_shard decider 拒绝）——这是开发环境最常见的 yellow 原因。
- **节点短暂离线**：副本所在节点宕机，等待延迟分配。
- **新增副本数**：动态增加了 `number_of_replicas`，但节点数不足。
- **磁盘水位超 high watermark**：节点磁盘不足，副本无法分配到该节点。

### 1.4 常见的 red 场景

- **承载主分片的节点永久离线**，且没有可用的副本晋升。
- **分片数据损坏**（corrupted shard），Lucene index 文件损坏。
- **磁盘满导致主分片写入失败**。
- **集群刚启动恢复中**（`gateway.expected_nodes` 未满足，分片分配尚未开始）。

## 2. Unassigned 分片排查完整流程

### 2.1 第一步：确认 unassigned 分片

```bash
# 快速查看 unassigned 分片数
GET _cluster/health?pretty&filter_path=status,unassigned_shards

# 列出所有 unassigned 分片
GET _cat/shards?v&h=index,shard,prirep,state,unassigned.reason,unassigned.at,node&s=state
```

### 2.2 第二步：分配解释 API

`_cluster/allocation/explain` 是排查 unassigned 分片最重要的工具——它不仅告诉你分片为什么没被分配，还会逐个检查每个节点并列出拒绝原因。

```bash
# 自动解释第一个 unassigned 分片
POST _cluster/allocation/explain?pretty
{}

# 指定具体分片
POST _cluster/allocation/explain?pretty
{
  "index": "my_index",
  "shard": 0,
  "primary": true
}
```

响应结构关键字段：

| 字段                           | 含义                             |
| ---------------------------- | ------------------------------ |
| `index` / `shard` / `primary` | 分片标识                         |
| `current_state`              | 当前状态（`unassigned`）             |
| `unassigned_info.reason`     | 变为 unassigned 的原因（见下表）         |
| `unassigned_info.at`         | 变为 unassigned 的时间              |
| `unassigned_info.details`    | 附加详情（如失败原因）                    |
| `can_allocate`               | 是否可以分配（`yes` / `no` / `worse_balance`） |
| `allocate_explanation`       | 人类可读的分配解释                      |
| `node_allocation_decisions`  | 每个节点的分配决策及拒绝原因                 |

### 2.3 unassigned_info.reason 全解

| reason                      | 含义                     | 常见场景                     | 处置                        |
| --------------------------- | ---------------------- | ------------------------ | ------------------------- |
| `INDEX_CREATED`             | 新建索引，分片等待初始分配          | 创建索引后短暂出现                | 通常自动解决；检查是否有节点可分配         |
| `CLUSTER_RECOVERED`         | 集群完全重启恢复中              | 集群冷启动后                   | 等待恢复完成                    |
| `INDEX_REOPENED`            | 索引被重新打开                | `POST <index>/_open` 后   | 等待自动分配                    |
| `DANGLING_INDEX_IMPORTED`   | 悬挂索引被导入                | 节点加入带来了未知索引              | 确认索引是否需要，不需要则删除           |
| `NEW_INDEX_RESTORED`        | 从 snapshot 恢复的新索引      | snapshot restore 后       | 等待恢复完成                    |
| `EXISTING_INDEX_RESTORED`   | 从 snapshot 恢复到已有索引     | snapshot restore 后       | 等待恢复完成                    |
| `REPLICA_ADDED`             | 新增副本                   | 增加 `number_of_replicas`  | 检查是否有足够节点容纳副本             |
| `ALLOCATION_FAILED`         | 分配尝试失败                 | 目标节点磁盘满、连接超时等            | 检查 details 中的失败原因         |
| `NODE_LEFT`                 | 承载该分片的节点离线             | 节点宕机或网络中断                | 判断节点是临时还是永久离线             |
| `REROUTE_CANCELLED`         | reroute 被取消             | 手动 reroute 冲突            | 重新执行 reroute              |
| `REINITIALIZED`             | 分片重新初始化（因 recovery 失败）  | 恢复过程出错                   | 检查源分片健康状态                 |
| `REALLOCATED_REPLICA`       | 副本被重新分配（更好的均衡策略）       | 再平衡过程中                   | 通常自动解决                    |
| `PRIMARY_FAILED`            | 主分片失败                  | 分片损坏或节点崩溃                | 检查是否有可晋升的副本               |
| `FORCED_EMPTY_PRIMARY`      | 手动分配了空主分片              | `allocate_empty_primary` | 数据已丢失，需从备份恢复              |
| `INDEX_CLOSED`              | 索引被关闭                  | `POST <index>/_close`    | 如需分片分配，先 open 索引          |

### 2.4 Decider（决策器）拒绝原因

`node_allocation_decisions` 中列出了每个候选节点上各个 decider 的判断结果。常见的拒绝 decider：

| Decider                    | 拒绝原因                          | 处置                               |
| -------------------------- | ----------------------------- | -------------------------------- |
| `same_shard`               | 同一分片的主副本不能在同一节点               | 增加节点数（单节点集群的 yellow 根因）            |
| `disk_threshold`           | 节点磁盘使用超过 watermark            | 清理磁盘或调整水位线                       |
| `max_retry`                | 分配重试次数超限（默认 5 次）              | 手动 `POST _cluster/reroute?retry_failed=true` |
| `awareness`                | 违反分配感知规则（如 zone awareness）    | 检查 zone 分布和 awareness 配置          |
| `filter`                   | 索引级分配过滤规则阻止                   | 检查 `index.routing.allocation.*` 设置  |
| `throttling`               | 并发恢复数达到上限                     | 等待或调大 `cluster.routing.allocation.node_concurrent_recoveries` |
| `shards_limit`             | 节点分片数达到限制                     | 检查 `index.routing.allocation.total_shards_per_node` |
| `node_version`             | 目标节点版本低于源节点                   | 升级节点版本                           |
| `resize`                   | resize 操作要求分片在特定节点            | 等待 resize 完成                     |

### 2.5 常见排查决策树

```
Unassigned 分片
  │
  ├── primary = true（主分片未分配 → red）
  │   ├── reason = NODE_LEFT
  │   │   ├── 节点可恢复 → 等待节点重新加入
  │   │   └── 节点不可恢复 → 有副本？晋升副本 : 从 snapshot 恢复 / allocate_stale_primary
  │   ├── reason = ALLOCATION_FAILED
  │   │   └── 检查 details → 修复根因 → retry_failed
  │   └── reason = PRIMARY_FAILED
  │       └── 分片损坏 → 从 snapshot 恢复 / allocate_empty_primary（丢数据）
  │
  └── primary = false（副本未分配 → yellow）
      ├── decider = same_shard → 增加节点
      ├── decider = disk_threshold → 清理磁盘
      ├── decider = max_retry → retry_failed
      ├── decider = filter → 检查分配过滤规则
      └── reason = REPLICA_ADDED → 检查节点数是否足够
```

## 3. 磁盘水位故障排查与恢复

### 3.1 三级水位机制

ES 通过三级磁盘水位控制分片分配行为：

| 水位级别               | 默认值    | 行为                                            |
| ------------------ | ------ | --------------------------------------------- |
| `low` watermark    | 85%    | **不再向该节点分配新的副本分片**                             |
| `high` watermark   | 90%    | ES **尝试将分片从该节点迁移走**                            |
| `flood_stage`      | 95%    | 将该节点上的所有索引设为**只读**（`index.blocks.read_only_allow_delete`） |

水位设置的两种方式（取更宽松的一个生效）：

```bash
# 百分比方式（默认）
PUT _cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.disk.watermark.low": "85%",
    "cluster.routing.allocation.disk.watermark.high": "90%",
    "cluster.routing.allocation.disk.watermark.flood_stage": "95%"
  }
}

# 绝对值方式
PUT _cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.disk.watermark.low": "50gb",
    "cluster.routing.allocation.disk.watermark.high": "20gb",
    "cluster.routing.allocation.disk.watermark.flood_stage": "10gb"
  }
}
```

### 3.2 flood_stage 触发后的症状

当磁盘使用率超过 `flood_stage`，ES 会自动给该节点上的所有索引添加 `index.blocks.read_only_allow_delete: true`——索引变为只读，只允许删除操作。

此时写入会收到如下错误：

```json
{
  "error": {
    "type": "cluster_block_exception",
    "reason": "index [my_index] blocked by: [TOO_MANY_REQUESTS/12/disk usage exceeded flood-stage watermark...]"
  }
}
```

### 3.3 恢复步骤

```bash
# 1. 确认磁盘使用情况
GET _cat/allocation?v&h=node,disk.used,disk.avail,disk.total,disk.percent

# 2. 清理空间（选择适用的操作）
#    - 删除不需要的索引
DELETE /old_logs_2023*
#    - 清理旧 snapshot
DELETE /_snapshot/my_backup/snapshot_old
#    - 对停写索引做 force merge 回收空间
POST /closed_index/_forcemerge?max_num_segments=1

# 3. 磁盘空间恢复到水位以下后，解除只读块
PUT /my_index/_settings
{
  "index.blocks.read_only_allow_delete": null
}

# 4. 批量解除所有索引的只读块
PUT _all/_settings
{
  "index.blocks.read_only_allow_delete": null
}

# 5. 验证写入恢复
POST /my_index/_doc
{ "test": "write_recovered" }
```

### 3.4 预防措施

- 设置磁盘使用率告警，在达到 `low` watermark（85%）前就处理。
- 配合 ILM 策略自动删除过期索引。
- 为集群预留足够的磁盘余量（建议不超过 80% 常态使用率）。

## 4. 节点离线故障

### 4.1 判断节点状态

```bash
# 当前节点列表
GET _cat/nodes?v&h=name,ip,node.role,master,heap.percent,cpu

# 对比预期节点数
GET _cluster/health?filter_path=number_of_nodes,number_of_data_nodes
```

如果节点数低于预期，需要判断离线是临时的还是永久的：

- **临时离线**：网络抖动、短暂重启、GC 停顿导致 master 认为节点失联。
- **永久离线**：硬件故障、磁盘损坏、被手动下线。

### 4.2 延迟分配机制

当节点离线时，ES 不会立即重新分配该节点上的分片，而是等待一个 `delayed_timeout`：

| 参数                                              | 默认值  | 说明                  |
| ----------------------------------------------- | ---- | ------------------- |
| `index.unassigned.node_left.delayed_timeout`    | `1m` | 节点离线后延迟分配的等待时间      |

这个设计是为了避免"节点短暂重启就触发大量分片迁移"的不必要开销。

```bash
# 查看延迟分配状态
GET _cluster/health?pretty&filter_path=delayed_unassigned_shards

# 调整延迟时间（如预知节点需要更长时间恢复）
PUT /my_index/_settings
{
  "index.unassigned.node_left.delayed_timeout": "5m"
}

# 取消延迟立即分配（确认节点不会回来）
PUT /my_index/_settings
{
  "index.unassigned.node_left.delayed_timeout": "0"
}
```

### 4.3 手动 Reroute

当自动分配无法解决问题时，可以使用手动 reroute 作为最后手段。

**重试失败的分配：**

```bash
POST _cluster/reroute?retry_failed=true
```

**分配 stale primary（可能丢失最近的写入）：**

```bash
POST _cluster/reroute
{
  "commands": [{
    "allocate_stale_primary": {
      "index": "my_index",
      "shard": 0,
      "node": "node-1",
      "accept_data_loss": true
    }
  }]
}
```

`allocate_stale_primary` 会使用节点上的旧副本数据作为主分片。这意味着**最后一批写入可能丢失**，只在确认数据丢失可接受时使用。

**分配空主分片（丢失该分片全部数据）：**

```bash
POST _cluster/reroute
{
  "commands": [{
    "allocate_empty_primary": {
      "index": "my_index",
      "shard": 0,
      "node": "node-1",
      "accept_data_loss": true
    }
  }]
}
```

这是最后手段——**该分片的所有数据将丢失**，索引恢复为 green 但分片为空。

## 5. 分片损坏恢复

### 5.1 如何发现分片损坏

分片损坏通常表现为：

- 分片反复分配失败（`ALLOCATION_FAILED`）
- 节点日志中出现 `CorruptIndexException` 或 `ChecksumException`
- `_cluster/allocation/explain` 的 details 提示索引文件损坏

### 5.2 恢复方案对比

| 方案                       | 数据损失       | 前提条件                     | 适用场景            |
| ------------------------ | ---------- | ------------------------ | --------------- |
| 从 snapshot 恢复            | 无（恢复到快照时间点）| 有可用 snapshot             | 首选方案            |
| `allocate_stale_primary` | 可能丢失最新写入   | 其他节点有该分片的旧副本             | 没有 snapshot 时   |
| `allocate_empty_primary` | 该分片全部数据丢失  | 无                        | 最后手段            |
| 从上游重放数据                  | 无          | 数据源支持重放（如 Kafka）         | 有数据重放能力时        |

### 5.3 从 Snapshot 恢复分片

```bash
# 1. 查看可用 snapshot
GET _snapshot/my_backup/_all?pretty

# 2. 关闭受损索引
POST /damaged_index/_close

# 3. 从 snapshot 恢复
POST _snapshot/my_backup/latest_snapshot/_restore
{
  "indices": "damaged_index",
  "index_settings": {
    "index.number_of_replicas": 0
  }
}

# 4. 等待恢复完成
GET _cat/recovery/damaged_index?v&active_only=true

# 5. 恢复副本数
PUT /damaged_index/_settings
{
  "index.number_of_replicas": 1
}

# 6. 验证索引状态
GET _cluster/health/damaged_index?pretty
```

## 6. 标准应急 Runbook

### 6.1 10 分钟应急流程

```
T+0min  收到告警
  │
  ├── 1. 判断严重程度（30s）
  │     GET _cluster/health
  │     status = red → 进入紧急流程
  │     status = yellow → 评估影响范围
  │
  ├── 2. 定位问题索引（1min）
  │     GET _cluster/health?level=indices
  │     GET _cat/shards?v&h=index,shard,prirep,state,unassigned.reason&s=state
  │
  ├── 3. 分配解释（2min）
  │     POST _cluster/allocation/explain { "index": "xxx", "shard": 0, "primary": true }
  │     记录 reason 和 decider 拒绝原因
  │
  ├── 4. 检查节点状态（1min）
  │     GET _cat/nodes?v
  │     GET _cat/allocation?v
  │     确认是否有节点离线 / 磁盘满
  │
  ├── 5. 止血操作（3min）
  │     ├── 磁盘满 → 清理空间 + 解除只读块
  │     ├── 节点离线 → 尝试恢复节点 / 调整 delayed_timeout
  │     ├── 分配失败 → retry_failed
  │     └── 分片损坏 → 从 snapshot 恢复
  │
  └── 6. 验证恢复（2min）
        GET _cluster/health
        确认 status 恢复为 green
        确认 unassigned_shards = 0
```

### 6.2 常见故障速查表

| 现象                     | 首查 API                           | 常见根因           | 处置                       |
| ---------------------- | -------------------------------- | -------------- | ------------------------ |
| 集群 red                | `_cluster/allocation/explain`    | 主分片所在节点离线      | 恢复节点 / stale primary     |
| 集群 yellow              | `_cat/shards?v&s=state`         | 副本无节点可分配       | 增加节点 / 降低副本数             |
| 写入报 429                | `_cat/thread_pool/write?v`       | write 队列满       | 降低并发 / 增大 bulk 批         |
| 写入报 read_only           | `_cat/allocation?v`              | 磁盘 flood_stage | 清理磁盘 + 解除只读             |
| 查询超时                   | `_tasks?actions=*search`         | 大查询 / GC       | 优化查询 / 增加超时              |
| 节点频繁退出                 | 节点日志                            | OOM / GC 停顿    | 检查 heap 大小 / 查询优化        |
| recovery 卡住             | `_cat/recovery?v&active_only`    | 带宽限制 / 源节点慢    | 调大 `indices.recovery.max_bytes_per_sec` |

## 7. 故障复盘模板

每次故障恢复后，建议用以下模板做复盘记录，推动系统性改进：

```
# 故障复盘报告

## 基本信息
- 故障时间：2025-06-15 14:30 ~ 15:10 (40min)
- 影响范围：orders 索引不可写入
- 严重程度：P1（数据写入中断）

## 时间线
| 时间     | 事件                                   |
| ------ | ------------------------------------ |
| 14:30  | 监控告警：集群状态变为 red                      |
| 14:32  | 运维确认：node-3 离线                       |
| 14:35  | 定位：node-3 磁盘故障，无法恢复                  |
| 14:40  | 操作：设置 delayed_timeout=0 触发分片重分配       |
| 14:50  | 分片恢复中：使用副本晋升主分片                      |
| 15:05  | 集群状态恢复 yellow                         |
| 15:10  | 添加新节点，集群恢复 green                     |

## 根因分析
node-3 的磁盘出现物理故障，导致节点进程崩溃。
该节点承载了 orders 索引 shard-2 的主分片。

## 恢复措施
1. 取消延迟分配，使副本晋升为主分片
2. 添加新节点替代 node-3
3. 等待副本完成同步

## 改进项
```

# 总结

- **集群状态 = 所有索引中最差的那个**——排障时先用 `?level=indices` 找到问题索引。
- `_cluster/allocation/explain` 是排查 unassigned 分片的**唯一正确入口**——它不仅给出原因，还逐节点列出拒绝理由。
- **磁盘 flood_stage 是最常见的"写入突然中断"原因**——`index.blocks.read_only_allow_delete` 需要手动解除。
- **`delayed_timeout` 是节点短暂离线的安全网**——不要设为 0，除非确认节点不会回来。
- 手动 reroute 是最后手段——`allocate_stale_primary` 可能丢最新数据，`allocate_empty_primary` 丢全部数据。
- **每次故障都值得复盘**——不是为了追责，而是为了系统性地减少下一次故障的影响。

2. 模拟磁盘 flood_stage 场景：将 `flood_stage` 临时调低到当前磁盘使用率以下，观察索引变为只读，然后恢复。

5. 使用本章的应急 Runbook 模板，对一次模拟故障做完整的排查和记录。

## 实战（Java SDK）

```java
// ---------- 集群健康检查 ----------
var health = client.cluster().health(h -> h.level(Level.Indices));
System.out.println("Status: " + health.status());
System.out.println("Unassigned shards: " + health.unassignedShards());

health.indices().forEach((indexName, indexHealth) -> {
    if (!"green".equals(indexHealth.status().jsonValue())) {
        System.err.println("[WARN] Index " + indexName + " is " + indexHealth.status());
    }
});

// ---------- 分配解释 ----------
try {
    var explain = client.cluster().allocationExplain(e -> e
        .index("yellow_demo")
        .shard(0)
        .primary(false));
    System.out.println("Can allocate: " + explain.canAllocate());
    System.out.println("Explanation: " + explain.allocateExplanation());
    if (explain.nodeAllocationDecisions() != null) {
        explain.nodeAllocationDecisions().forEach(decision -> {
            System.out.println("  Node: " + decision.nodeName());
            if (decision.deciders() != null) {
                decision.deciders().forEach(decider ->
                    System.out.println("    " + decider.decider() + ": " + decider.decision() + " - " + decider.explanation()));
            }
        });
    }
} catch (Exception e) {
    System.out.println("No unassigned shards to explain: " + e.getMessage());
}

// ---------- 模拟 yellow → 修复 ----------
client.indices().create(c -> c.index("yellow_demo")
    .settings(s -> s.numberOfShards("1").numberOfReplicas("1"))
    .mappings(m -> m.properties("msg", p -> p.text(t -> t))));

var statusBefore = client.cluster().health(h -> h.index("yellow_demo")).status();
System.out.println("Before fix: " + statusBefore);

client.indices().putSettings(p -> p.index("yellow_demo")
    .settings(s -> s.numberOfReplicas("0")));

var statusAfter = client.cluster().health(h -> h.index("yellow_demo")).status();
System.out.println("After fix: " + statusAfter);

// ---------- 解除只读块 ----------
client.indices().putSettings(p -> p.index("yellow_demo")
    .settings(s -> s.otherSettings("index.blocks.read_only_allow_delete", JsonData.of((Object) null))));

// ---------- 重试失败的分配 ----------
client.cluster().reroute(r -> r.retryFailed(true));

// ---------- 清理 ----------
client.indices().delete(d -> d.index("yellow_demo"));
```
