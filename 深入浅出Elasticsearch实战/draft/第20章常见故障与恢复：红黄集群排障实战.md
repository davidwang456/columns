# 背景

# 第20章 常见故障与恢复：红黄集群排障实战

红黄集群是 ES 运维最常见告警。能否快速判断“是容量、分配还是节点异常”，决定了故障恢复速度和业务损失大小。

## 本章目标
- 掌握红黄集群的快速排障流程。
- 形成可复用的恢复与复盘方法。

## 1. 红黄状态怎么理解
- Yellow：主分片正常，部分副本不可分配。
- Red：存在主分片不可用，数据服务受影响。

## 2. 排查优先级
1) 看 unassigned shards  
2) 看磁盘水位与节点状态  
3) 看分片分配解释与限制条件

## 3. 常见根因
- 磁盘水位触发分配限制
- 节点离线或网络异常
- 分片策略不合理导致无法分配

## 4. 恢复与止血
- 先恢复可用性，再做结构优化。
- 恢复后要做“防复发”改进。

# 总结
- 故障处理中“顺序”很关键：先定位、再止血、后优化。
- 标准化 Runbook 可显著提升响应效率。

## 练习题
1. 设计一份红黄集群 10 分钟应急流程。  
2. 模拟一次磁盘水位故障并完成恢复。  
3. 输出一次故障复盘模板。  

## 实战（curl）

```bash
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cluster/health?pretty"
curl -u "$ES_USER:$ES_PASS" "$ES_URL/_cat/shards?v"
curl -u "$ES_USER:$ES_PASS" -X GET "$ES_URL/_cluster/allocation/explain?pretty" \
  -H "Content-Type: application/json" -d '{}'
```

## 实战（Java SDK）

```java
var explain = client.cluster().allocationExplain(a -> a);
System.out.println(explain.currentState());
```
