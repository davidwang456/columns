# 第22章：依赖与许可证合规扫描

## 1. 项目背景

> **业务场景**：一家 SaaS 公司被竞争对手举报——"你们的产品使用了 GPL 协议的代码库，但没有开源你的代码"。法务团队紧急审查后发现：前端项目依赖了 200+ 个 npm 包，其中 3 个是 GPL 许可证，5 个是 AGPL（更严格的"网络感染"协议）。按照 GPL 条款，公司可能需要公开全部源代码。

更糟的是依赖漏洞：某次安全扫描发现项目使用了 `lodash@4.17.20`（有已知原型污染漏洞 CVE-2021-23337），但因为团队没有依赖扫描机制，这个漏洞在生产环境存在了 8 个月。攻击者利用此漏洞获取了部分用户数据。事后分析——如果当时有 Dependency Scanning 和自动告警，这个漏洞在 CVE 公布当天就能被发现。

法务和 CTO 联合下令："所有项目都必须启用 Dependency Scanning 和 License Compliance，不符合许可证政策的依赖不能进入生产，有高危 CVE 的依赖必须在 24 小时内升级。"

**痛点放大**：供应链安全是近年最热门的安全话题——你的代码只占应用的 10%，剩下 90% 都是第三方依赖。你不了解这些依赖的许可证条款（MIT？GPL？商用需付费？），也不知道它们是否包含已知漏洞（CVE）。GitLab 的依赖扫描可以自动识别这些问题并在 MR 中阻断——让安全合规从"事后救火"变成"事前预防"。

## 2. 项目设计——剧本式交锋对话

**场景**：法务部门发来的紧急邮件——"请立即审查所有项目的开源许可证合规性"。

---

**小胖**："GPL 和 MIT 不就是两个开源协议吗？我能商用有什么区别？"

**大师**："区别大了。MIT 是最宽松的——你怎么用都行，只需保留版权声明。GPL 是'传染性'最强的——如果你用了 GPL 协议的代码，你自己的代码也必须开源。AGPL 更严格——即使你没有分发代码，只是通过网络提供服务，也得开源。LGPL 介于两者之间——动态链接的不需要开源，静态链接的需要。"

**小胖**："那如果项目依赖了 200 个 npm 包，里面有 1 个 GPL 的，我整个项目就得开源？"

**大师**："不完全是这样——npm 的 GPL 包通常是你通过 require/import 的方式引用的，这通常被视为'动态链接'，不一定会触发 GPL 的传染条款。但 AGPL 就不同了——它是专门为 SaaS 场景设计的，你通过网络提供服务就算'分发'。所以法务团队最怕的就是 AGPL。"

**小白**："GitLab 的 License Compliance 是怎么工作的？"

**大师**："GitLab 会在 CI 中运行 `license_finder` 工具，分析你的依赖声明文件（package.json、pom.xml、requirements.txt 等），找出每个依赖的许可证类型。然后根据你配置的许可证政策——比如'Accepted: MIT, Apache-2.0, BSD'——标记出不合规的依赖。在 MR 中会显示'此 MR 引入了 2 个新增的不合规许可证依赖'。"

**大师**："Dependency Scanning 则是另一个维度——它检查你的每个第三方依赖是否有已知的 CVE 漏洞。比如你用了 `lodash@4.17.20`，GitLab 会告诉你这个版本有 CVE-2021-23337（原型污染），并建议你升级到 4.17.21。"

**小胖**："那这两个扫描能自动化阻断吗？"

**大师**："可以。在 Merge Checks 中勾选'License Compliance must be met'和'Security approvals required'（EE），不符合条件的 MR 无法合并。技术映射——这就像出入境检查：签证（许可证）和健康证明（漏洞检查），缺一不可。"

---

## 3. 项目实战

### 环境准备

> **目标**：为一个 Node.js 项目配置 Dependency Scanning 和 License Compliance，体验从发现到修复的全流程。

**前置条件**：GitLab CE/EE 17.x。

### 分步实现

#### 步骤1：配置 Dependency Scanning

**目标**：在 CI 中集成依赖扫描，自动检测第三方库的 CVE 漏洞。

```yaml
# .gitlab-ci.yml - Dependency Scanning
stages:
  - test
  - security

# 引用 GitLab 官方依赖扫描模板
include:
  - template: Jobs/Dependency-Scanning.gitlab-ci.yml

variables:
  DEPENDENCY_SCANNING_DISABLED: "false"
  # 排除开发依赖（默认只扫描生产依赖）
  DS_EXCLUDED_PATHS: "spec, test, tests, tmp"
  # 设置漏洞严重级别阈值（只报告 >= Medium）
  DS_JAVA_GRADLE: "false"   # 非 Gradle 项目关闭

# 也可以自定义扫描参数
gemnasium-dependency-scanning:
  variables:
    DS_REMEDIATE: "true"     # 生成修复建议
    DS_MAX_DEPTH: "10"       # 依赖树最大深度
```

**查看依赖扫描结果**：

```bash
# 1. MR 页面 → Security tab → Dependency Scanning
# 2. 查看新增漏洞列表：
#    - 漏洞名称、CVE 编号
#    - 严重级别（Critical/High/Medium/Low）
#    - 位于哪个依赖、哪个版本
#    - 修复方案（升级到哪个版本）

# 3. 点击某个漏洞 → 查看详情：
#    - CVE 描述
#    - CVSS 评分
#    - 受影响版本范围
#    - 修复版本
#    - 相关链接（NVD、GitHub Advisory）
```

#### 步骤2：配置 License Compliance（许可证合规）

**目标**：在 CI 中集成许可证扫描，制定企业许可证政策。

```yaml
# .gitlab-ci.yml - License Compliance
include:
  - template: Jobs/License-Scanning.gitlab-ci.yml

variables:
  LICENSE_SCANNING_DISABLED: "false"
  # 指定许可证政策（白名单/黑名单）
  LICENSE_MANAGEMENT_SETUP_CMD: ""
```

**创建许可证政策文件**（需 EE）：

```yaml
# .gitlab/license-policy.yml（或通过 GitLab UI 配置）
# Settings → CI/CD → License Compliance → Policies

# 白名单模式：只允许这些许可证
policy:
  approved_licenses:
    - MIT
    - Apache-2.0
    - BSD-2-Clause
    - BSD-3-Clause
    - ISC
    - Unlicense
    - CC0-1.0
    - Python-2.0
    - Zlib

  # 黑名单模式：明确禁止这些许可证
  denied_licenses:
    - GPL-2.0
    - GPL-3.0
    - AGPL-1.0
    - AGPL-3.0
    - LGPL-2.1     # 如果静态链接则需要禁止
    - MPL-2.0      # 视公司政策
    - WTFPL        # 法律效力存疑
```

**通过 API 管理许可证政策**（EE）：

```bash
# 创建许可证政策
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "name": "MIT",
    "approval_status": "approved"
  }' \
  "$GITLAB_URL/api/v4/projects/$PROJECT_ID/approvals/license_policy"

# 查看当前许可证政策
curl --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  "$GITLAB_URL/api/v4/projects/$PROJECT_ID/approvals/license_policy"
```

#### 步骤3：处理不合规依赖

**目标**：当扫描发现不合规许可证或有漏洞依赖时，学习如何处理。

**场景A：许可证不合规**：

```bash
# 发现某个依赖使用了 GPL-3.0 许可证

# 处理选项：
# 1. 替换为 MIT 许可的同等功能库
npm uninstall gpl-library
npm install mit-alternative

# 2. 如果无法替换，联系法务评估风险
#    - 你的产品是否分发这个库的二进制？
#    - 使用方式是否为动态链接？
#    - 是否有购买商业许可的选项？

# 3. 如果是误报（某些库的双许可证），提交 issue 到 GitLab
#    - 或在 MR 中标记为 "False positive"
```

**场景B：依赖有高危漏洞**：

```bash
# 发现 lodash@4.17.20 有 CVE-2021-23337

# 1. 查看修复建议（GitLab 自动提示）
#    → "Upgrade lodash to >= 4.17.21"

# 2. 升级依赖
npm install lodash@4.17.21
npm audit fix          # 自动修复已知漏洞（如果 semver 兼容）

# 3. 如果 semver 不兼容，评估影响范围
npm ls lodash           # 查看哪些包依赖了 lodash
npm audit               # 查看完整漏洞报告

# 4. 提交修复
git add package.json package-lock.json
git commit -m "fix: upgrade lodash to 4.17.21 (CVE-2021-23337)"
```

#### 步骤4：定期全量扫描

**目标**：配置定时 Pipeline，对已有项目做全量依赖审计。

```yaml
# .gitlab-ci.yml - 定时全量扫描
# 在 GitLab UI 中创建 Pipeline Schedule：
# CI/CD → Schedules → New schedule
# Cron: 0 3 * * 1 (每周一凌晨 3 点)

nightly-security-audit:
  stage: security
  image: node:20-alpine
  script:
    - npm ci
    - npm audit --audit-level=high
    - npm outdated --long
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule"'
  artifacts:
    paths:
      - audit-report.json
    expire_in: 30 days

# 生成 SBOM（软件物料清单）
generate-sbom:
  stage: security
  image: cyclonedx/cyclonedx-node:latest
  script:
    - cyclonedx-bom -o sbom.json
  artifacts:
    paths:
      - sbom.json
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule"'
```

### 完整代码清单

- `.gitlab-ci.yml`：Dependency Scanning + License Compliance + 定时审计
- `.gitlab/license-policy.yml`：许可证政策文件
- `sbom.json`：软件物料清单输出

### 测试验证

```bash
# 验证1：依赖扫描在 MR 中显示
# 创建一个 MR，新增一个引入已知漏洞的依赖（如 lodash@4.17.20）

# 提交 MR → Security widget → 应显示 Dependency Scanning 发现新漏洞

# 验证2：许可证扫描显示不合规

# 重新 push → 前述漏洞应标记为 "Fixed" 或消失

# 验证4：SBOM 文件生成

# 确认 CycloneDX 格式的 SBOM 生成正确
```

## 4. 项目总结

### 主要开源许可证对比

| 许可证 | 传染性 | 可商用 | 需要开源 | 典型场景 |
|--------|--------|--------|---------|---------|
| MIT | 无 | ✅ | 不需要 | 最宽松，npm 包常用 |
| Apache-2.0 | 无 | ✅ | 不需要 | 大公司项目首选 |
| BSD | 无 | ✅ | 不需要 | 类似 MIT |
| GPL-2.0/3.0 | 强传染 | ✅ | 需要 | Linux 内核等 |
| AGPL-3.0 | 网络传染 | ✅ | 需要 | MongoDB（已改为SSPL）|
| LGPL | 弱传染 | ✅ | 视情况 | 动态链接无需开源 |
| MPL-2.0 | 文件级传染 | ✅ | 部分 | Mozilla 项目 |
| CC-NC | 不可商用 | ❌ | - | 个人作品 |

### 适用场景

- **所有有第三方依赖的项目**：都应启用 Dependency Scanning
- **对外分发/售卖的软件产品**：必须启用 License Compliance
- **合规审计场景**：SBOM 是 SOC2 和供应链安全的基本交付物

**不适用场景**：
- 完全自研、零第三方依赖的内部工具（极少见）
- 完全隔离的内网系统且不需合规审计

### 注意事项

- **Dependency Scanning 默认只扫描生产依赖**：开发依赖（devDependencies）不扫描，除非配置
- **许可证检测基于 SPDX 标识符**：某些双许可证库（如 MIT/GPL 双许可）可能显示为 "Unknown"
- **npm audit 不是 GitLab Dependency Scanning**：前者是 npm 内置的快速检查，后者更全面且集成到 GitLab UI
- **漏洞修复不是终点**：升级大版本可能引入 breaking change，需要全面测试

### 常见踩坑经验

1. **许可证扫描显示 Unknown**：依赖没有在 package.json 中声明 `license` 字段。根因：某些非常小的 npm 包没有声明许可证。解决：到包的 GitHub 仓库查看 LICENSE 文件，手动确认后可以在 GitLab 中标记为 "Approved"。
2. **Dependency Scanning 耗时过长**：大型 Monorepo 有 1000+ 依赖，扫描一次 20 分钟。根因：每次 CI 都重新下载所有依赖的元数据。解决：GitLab 会自动缓存扫描结果，后续扫描只检查变更部分。
3. **SBOM 格式不兼容**：用 cyclonedx 生成了 SBOM，但客户要求 SPDX 格式。根因：SBOM 有多种标准格式。解决：使用 `syft` 或 `cdxgen` 工具，它们支持输出多种 SBOM 格式（CycloneDX、SPDX、Syft JSON）。

### 思考题

1. 你的公司开发了一款 SaaS 产品，后端用了 GNU AGPL-3.0 的库（通过网络提供功能）。根据 AGPL 条款，你的产品是否需要开源？如果这个库是 GPL-2.0 而不是 AGPL-3.0，答案会变吗？
2. 某依赖的 CVE 修复版本是 5.0，但你的代码用的是 3.x——跨了 2 个大版本。如何评估升级的可行性和风险？什么情况下可以接受"不升级，但接受风险"？

### 推广计划提示

- **开发**：每次 `npm install` 时多看一眼 `npm audit` 的输出，养成定期升级依赖的习惯
- **法务/安全**：建立企业许可证白名单，并通过 License Compliance 自动执行
- **运维**：定期扫描的 SBOM 应归档保存，作为合规审计证据
