# 第29章：备份恢复与灾难演练

## 1. 项目背景

> **业务场景**：某公司运行 GitLab 两年，每周定时备份。上个月运维在清理磁盘时误操作删除了 `/var/opt/gitlab` 下的数据目录。团队信心满满地执行恢复操作——结果发现最近一次备份是 13 天前的，因为自动备份脚本在 12 天前因为磁盘空间不足而静默失败，没有人发现。更糟的是，恢复后发现 GitLab 无法启动——备份时忘了备份 `gitlab-secrets.json`，导致数据库加密字段无法解密。

3 天数据永久丢失，业务影响评估结果显示直接损失超过 50 万。CTO 震怒："备份不是做了就行！必须定期验证恢复流程！"

另一个惊险时刻：数据库的主库因为硬件故障完全损坏，但团队没有做过 PostgreSQL 流复制的切换演练，在主库挂掉后手忙脚乱地切备库，花了 4 个小时才恢复服务。

**痛点放大**：备份恢复不是"配个 crontab 就完事"——你需要理解 GitLab 备份的范围（什么会被备份、什么不会）、知道如何验证备份的完整性、定期在备用环境演练恢复流程、了解 Geo 灾备和 Backup Restore 的 RTO/RPO 差异。

## 2. 项目设计——剧本式交锋对话

**场景**：数据丢失事故的复盘会，气氛凝重。

---

**CTO**："你说你每周都做备份，为什么最新的备份是 13 天前的？"

**运维小王**："自动备份脚本因为磁盘满了，cron job 静默失败了……我没有设置备份失败告警。"

**大师**："备份系统必须有三层保障：第一，备份任务本身有告警——cron job 失败后发通知。第二，备份文件有监控——定期检查备份的年龄和大小，异常时告警。第三，恢复流程定期演练——至少每季度一次，在备用服务器上做完整的恢复测试。"

**小胖**："恢复演练真的有必要吗？每次都搭一套新环境太麻烦了吧？"

**大师**："恢复演练就像消防演习——你不希望第一次用灭火器是在真实火灾中。我见过太多团队备份做得很勤快，但从来没恢复过。等到真的需要恢复时，发现备份文件损坏、版本不匹配、秘钥文件丢失——这些都是可以提前发现的问题。用 Docker Compose 搭建一套专用恢复验证环境，每次备份完成后自动触发恢复测试。"

**小白**："那 Geo 灾备和 Backup Restore 有什么区别？我们该用哪个？"

**大师**："Backup Restore（备份恢复）是冷备——你有一个时间点的快照，恢复需要停机，RTO 可能在 1-4 小时，RPO 取决于备份频率（通常每天一次）。Geo 灾备是热备——有一个实时同步的副本，故障时可以切换，RTO 可能是几分钟，RPO 接近实时。技术映射——Backup Restore 就像把重要文件复印一份放保险柜（需要时去取），Geo 就像在隔壁办公室有个实时同步的备份工位（随时可以用）。"

**自动备份脚本 + 失败告警**：

```bash
# /usr/local/bin/gitlab-backup-with-alert.sh
#!/bin/bash
set -e

BACKUP_LOG="/var/log/gitlab-backup.log"
SLACK_WEBHOOK="https://hooks.slack.com/services/xxx"

# 执行备份
echo "[$(date)] Starting backup..." >> $BACKUP_LOG
if sudo gitlab-backup create CRON=1 >> $BACKUP_LOG 2>&1; then
    # 备份成功
    BACKUP_SIZE=$(sudo du -sh /data/gitlab-backups/*$(date +%Y_%m_%d)* | awk '{print $1}')
    echo "[$(date)] Backup completed successfully. Size: $BACKUP_SIZE" >> $BACKUP_LOG

    # 同时备份配置文件
    sudo tar -czf /data/gitlab-backups/gitlab-config-$(date +%s).tar.gz \
      /etc/gitlab/gitlab.rb /etc/gitlab/gitlab-secrets.json

    # 清理旧备份
    sudo find /data/gitlab-backups -name "*_gitlab_backup.tar" -mtime +7 -delete
else
    # 备份失败
    echo "[$(date)] Backup FAILED!" >> $BACKUP_LOG

    # 发送 Slack 告警
    curl -X POST "$SLACK_WEBHOOK" \
      -H "Content-Type: application/json" \
      -d "{\"text\": \"❌ GitLab backup FAILED! Check $BACKUP_LOG\"}"
fi
```

**配置 Crontab**：

```bash
# 每天凌晨 2 点执行备份
```

#### 步骤2：备份监控

**目标**：监控备份文件的年龄和数量，过期告警。

```python
#!/usr/bin/env python3
"""backup_monitor.py - 监控备份健康状态"""
import os
import time
from datetime import datetime

BACKUP_DIR = "/data/gitlab-backups"
MAX_AGE_HOURS = 26  # 备份不能超过 26 小时（留 2 小时缓冲）
SLACK_WEBHOOK = "https://hooks.slack.com/services/xxx"

def check_latest_backup():
    """检查最新备份的年龄"""
    backup_files = [
        f for f in os.listdir(BACKUP_DIR)
        if f.endswith('_gitlab_backup.tar')
    ]

    if not backup_files:
        send_alert("❌ No backup files found!")
        return

    # 找最新的备份
    latest = max(backup_files, key=lambda f: os.path.getmtime(os.path.join(BACKUP_DIR, f)))
    latest_path = os.path.join(BACKUP_DIR, latest)
    age_seconds = time.time() - os.path.getmtime(latest_path)
    age_hours = age_seconds / 3600

    size_gb = os.path.getsize(latest_path) / (1024**3)

    status = "✅" if age_hours < MAX_AGE_HOURS else "❌"
    print(f"{status} Latest backup: {latest} ({age_hours:.1f}h ago, {size_gb:.1f}GB)")

    if age_hours > MAX_AGE_HOURS:
        send_alert(f"❌ Latest GitLab backup is {age_hours:.1f}h old (threshold: {MAX_AGE_HOURS}h)")

    if size_gb < 1:
        send_alert(f"⚠️ Backup size is suspiciously small: {size_gb:.1f}GB")

def send_alert(message):
    """发送告警"""
    print(f"ALERT: {message}")
    # requests.post(SLACK_WEBHOOK, json={"text": message})

if __name__ == '__main__':
    check_latest_backup()
```

#### 步骤3：恢复演练

**目标**：在 Docker 备用环境中执行完整的恢复验证流程。

```bash
# 恢复演练脚本（在备用服务器上执行）
#!/bin/bash
# restore-drill.sh

set -e

GITLAB_VERSION="17.0.0-ce.0"
BACKUP_FILE="1715472000_2026_05_12_17.0.0_gitlab_backup.tar"

echo "=== GitLab 恢复演练 ==="
echo "1. 准备恢复环境..."

# 创建测试目录
mkdir -p /tmp/gitlab-restore/{config,logs,data}
chmod 777 /tmp/gitlab-restore/data

# 复制备份文件
cp /data/gitlab-backups/$BACKUP_FILE /tmp/gitlab-restore/data/
cp /data/gitlab-backups/gitlab-config-*.tar.gz /tmp/gitlab-restore/

# 恢复配置文件
cd /tmp/gitlab-restore
tar xzf gitlab-config-*.tar.gz
cp etc/gitlab/gitlab.rb config/
cp etc/gitlab/gitlab-secrets.json config/

# 启动 GitLab 容器（使用与生产相同的版本）
docker run -d --name gitlab-restore-test \
  --hostname gitlab-test.local \
  -p 9999:80 \
  -v /tmp/gitlab-restore/config:/etc/gitlab \
  -v /tmp/gitlab-restore/logs:/var/log/gitlab \
  -v /tmp/gitlab-restore/data:/var/opt/gitlab \
  -e GITLAB_OMNIBUS_CONFIG="external_url 'http://localhost:9999'" \
  gitlab/gitlab-ce:$GITLAB_VERSION

echo "2. 等待 GitLab 启动（约 3-5 分钟）..."
sleep 180

echo "3. 执行恢复..."
docker exec gitlab-restore-test bash -c "
  gitlab-ctl stop puma sidekiq
  # 恢复秘钥文件后恢复数据
  gitlab-backup restore BACKUP=$(echo $BACKUP_FILE | sed 's/_gitlab_backup.tar//')
"

echo "4. 重配置并重启..."
docker exec gitlab-restore-test gitlab-ctl reconfigure
docker exec gitlab-restore-test gitlab-ctl restart

sleep 60

echo "5. 验证恢复结果..."
# 健康检查
HEALTH=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:9999/-/health)
if [ "$HEALTH" = "200" ]; then
    echo "✅ GitLab 恢复成功，健康检查通过"
else
    echo "❌ GitLab 恢复失败，健康检查返回 $HEALTH"
fi

# 验证数据
echo "6. 数据完整性检查..."
docker exec gitlab-restore-test gitlab-rake gitlab:check SANITIZE=true

echo "7. Git 操作验证..."
docker exec gitlab-restore-test gitlab-rails runner "
  project = Project.first
  puts \"Project: #{project&.name}, Repository size: #{project&.statistics&.repository_size}\"
"

echo "=== 恢复演练完成 ==="
echo "清理：docker stop gitlab-restore-test && docker rm gitlab-restore-test"
```

## 4. 项目总结

### 备份恢复 vs Geo 灾备

| 维度 | Backup Restore | Geo 灾备 |
|------|---------------|---------|
| RPO（数据丢失） | 备份间隔（通常 24h） | 接近实时（秒/分钟级）|
| RTO（恢复时间） | 1-4 小时 | 5-15 分钟 |
| 成本 | 低（只需存储空间） | 高（需要完整的备用基础设施）|
| 复杂度 | 低 | 高（需要维护同步链路）|
| 适用团队 | < 500 人 | 500+ 人，或 7x24 业务 |

### 适用场景

- **每日备份**：所有生产环境都必须有
- **恢复演练**：任何依赖 GitLab 的团队都应至少每季度做一次
- **Geo 灾备**：7x24 在线业务、金融/医疗行业、多地域团队

**不适用场景**：
- 备份完全不做（即使测试环境也应该有基本备份）
- 有 Geo 就不做恢复演练（Geo 需要同样定期演练切换）

### 注意事项

- **必须同时备份 `gitlab-secrets.json`**：这个文件包含数据库加密密钥，没有它恢复后数据无法解密
- **备份和 GitLab 版本必须匹配**：跨大版本恢复需要按升级路径先升级备份
- **恢复后必须执行 `reconfigure`**：否则组件配置文件不匹配
- **备份时的数据库一致性**：`gitlab-backup create` 使用 `pg_dump` 确保一致性

### 常见踩坑经验

1. **恢复后 GitLab 无法启动**：忘记恢复 `gitlab-secrets.json`。根因：数据库中的 CI 变量、Webhook Secret 等被加密存储，恢复时缺少解密密钥。解决：始终备份和恢复 secrets 文件。
2. **备份文件巨大（50GB+）但恢复后发现数据丢失**：备份时加了 `SKIP=artifacts`（跳过制品），但团队以为制品也被备份。根因：不了解 `SKIP` 参数的含义。解决：明确备份范围，文档化哪些内容不备份。
3. **Cron 备份静默失败**：磁盘满了，`gitlab-backup create` 执行到一半失败但不报错。根因：cron 的错误输出没有被捕获。解决：备份脚本中加磁盘空间检查（`df -h | grep backup-path`）和多渠道告警。
