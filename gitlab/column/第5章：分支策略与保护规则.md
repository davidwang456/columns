# 第5章：分支策略与保护规则

## 1. 项目背景

> **业务场景**：某电商公司年中大促前夕，一个实习生误操作把测试环境的 feature 分支直接 force push 到了 main 分支，导致线上支付服务的中断回滚版本被覆盖，生产环境宕机 4 小时。事后复盘发现：main 分支没有任何保护措施，任何人、任何分支都可以直接 force push。

这次事故触发了公司全员的"分支安全运动"。CTO 下令："所有仓库的 main 分支必须强制保护，至少需要 2 人 Code Review 才能合并，禁止 force push，禁止直接 push。"但当运维团队开始配置保护规则时，发现真相比想象中复杂：

- 代码仓库有 5 种角色权限，保护规则里的 "Allowed to merge" 和 "Allowed to push" 怎么配？
- 有人说主线分支不应该有任何人可以 push，那 hotfix 怎么办？
- Code Owner 审批规则怎么设置才能覆盖所有关键模块？
- 保护规则和 CI/CD Pipeline 的触发条件怎么联动？

**痛点放大**：分支保护不是简单的"开启/关闭"，而是一套与团队流程、代码架构、CI/CD 管道深度耦合的安全体系。配置不当会有两种极端结果：要么安全过严阻碍开发效率（一个 hotfix 要等 5 个人审批），要么形同虚设（规则设置了但没有真正拦截到错误操作）。

## 2. 项目设计——剧本式交锋对话

**场景**：事故复盘的会议室，白板上写满了分支图。

---

**小胖**："这次事故不就是因为没开分支保护吗？直接在 GitLab 设置里点几下不就好了，至于开个会？"

**大师**："那你来说说，Protected Branches 的'Allowed to merge'和'Allowed to push'分别对应什么操作？如果只配置了 merge 保护但没配置 push 保护，会发生什么？"

**小胖**："呃……merge 就是合并 MR 嘛，push 就是直接推送代码？"

**小白**："让我来补充。'Allowed to merge'控制谁能把 MR 合并进分支——这是通过 GitLab UI 的 Merge 按钮操作的。'Allowed to push'控制谁能通过 `git push` 直接推送代码到分支——这是绕过 MR 的路径。如果只保护 merge 不保护 push，那攻击者可以直接 `git push origin main` 绕过 Code Review。"

**大师**："精确。技术映射——merge 保护是'正门守卫'，push 保护是'后门锁'。你必须两扇门都锁好。生产环境的最佳实践是：Allowed to merge = Maintainers only，Allowed to push = No one。"

**小胖**："那 hotfix 怎么办？线上出 bug 必须马上修，等 Maintainer 审批黄花菜都凉了。"

**大师**："hotfix 不等于可以跳过保护。正确的做法是：hotfix 也需要创建分支 → 发起 MR → 但可以标记为'紧急'，触发单独的审批通道（比如只需要 1 个 Maintainer 审批，而不是常规的 2 个）。技术映射——保护规则不是铁板一块，你可以为不同分支、不同紧急程度设置不同的规则组合。"

**小白**："Code Owner 又是什么？跟普通审批人有什么区别？"

**大师**："Code Owner 是模块级别的审批人。比如你的仓库里 `payment/` 目录下的代码涉及资金安全，你可以指定支付团队的 Tech Lead 为这些文件的 Code Owner——任何修改 payment 目录的 MR，必须经过他的审批才能合并。普通审批人只要求 N 个人 Approve，不管审批人是谁。"

**小胖**："那如果一个 MR 同时改了 payment 和 product 两个目录，是不是需要两个 Code Owner 都审批？"

**大师**："对，这就是 Code Owner 的叠加效应。技术映射——Code Owner 不是替代普通审批人，而是额外加了一层'领域专家把关'。你可以要求：至少有 1 个普通审批人 + 所有涉及文件的 Code Owner 全部 Approve。"

**小白**："我还有一个问题：分支保护规则和 CI Pipeline 怎么联动？比如我希望所有 MR 必须通过 CI 测试才能合并。"

**大师**："这就是 Merge Checks 的范畴。你可以在分支保护规则中勾选'Pipelines must succeed'——这意味着如果 MR 关联的 Pipeline 有任何一个 job 失败（或者没跑完），合并按钮就会变灰。这是最后一道防线：即使审批通过了，代码质量不达标也不能合并。"

## 3. 项目实战

### 环境准备

> **目标**：为一个多模块仓库配置完整的保护规则体系，包含分支保护、Code Owner 审批、流水线检查、推送规则。

**前置条件**：已创建项目（参考第4章），有 Maintainer 以上权限。

### 分步实现

#### 步骤1：配置 main 分支的完整保护规则

**目标**：将 main 分支设为"铜墙铁壁"——任何人不能直接 push，合并需要审批 + CI 通过 + 线性历史。

**通过 GitLab UI 配置**：

```
项目 → Settings → Repository → Protected branches
→ 展开 main 分支（如果尚未保护，点击 Protect）
```

**核心配置项**：

| 配置项 | 推荐值 | 说明 |
|--------|--------|------|
| Allowed to merge | Maintainers | 只有 Maintainer 及以上可以点击 Merge 按钮 |
| Allowed to push and merge | No one | 禁止任何人直接 push 到 main |
| Allowed to force push | 不勾选 | 禁止 force push |
| Code owner approval | 勾选（见步骤3） | 需要 Code Owner 审批 |

**通过 API 配置**：

```bash
export GITLAB_URL="http://gitlab.local:8929"
export GITLAB_TOKEN="glpat-xxxx"
export PROJECT_ID="<your-project-id>"

# 保护 main 分支
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "name": "main",
    "push_access_level": 0,
    "merge_access_level": 40,
    "unprotect_access_level": 40,
    "allow_force_push": false,
    "code_owner_approval_required": true
  }' \
  "$GITLAB_URL/api/v4/projects/$PROJECT_ID/protected_branches"

# push_access_level 说明：
# 0 = No one
# 30 = Developers + Maintainers
# 40 = Maintainers
# 60 = Instance admins
```

#### 步骤2：配置 Merge Checks（合并检查）

**目标**：要求所有 MR 必须通过 CI Pipeline 且所有讨论已解决。

**通过 UI 配置**：

```
项目 → Settings → Merge requests → Merge checks
```

**必须开启的选项**：

```
✅ Pipelines must succeed
   → "所有 Pipeline 必须成功"（阻止有失败 Job 的 MR 合并）

✅ All threads must be resolved
   → "所有讨论必须解决"（阻止有未解决评论的 MR 合并）

✅ All discussions must be resolved
   → 同上，更严格的版本

□ Must be associated with a milestone（可选）
□ Must have a merge request template（可选）
```

**通过 API 配置合并检查**（GitLab API v4）：

```bash
# 配置 MR 合并检查
curl --request PUT \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "only_allow_merge_if_pipeline_succeeds": true,
    "only_allow_merge_if_all_discussions_are_resolved": true
  }' \
  "$GITLAB_URL/api/v4/projects/$PROJECT_ID"
```

#### 步骤3：配置 Code Owner 审批规则

**目标**：按文件路径指定 Code Owner，确保关键模块必须经过指定人审批。

**创建 `.gitlab/CODEOWNERS` 文件**：

```bash
mkdir -p .gitlab
cat > .gitlab/CODEOWNERS << 'EOF'
# GitLab Code Owners 文件
# 格式：<文件路径模式> <@username 或 @group>

# 根级别配置文件——需要架构组审批
*.md            @tech-lead
.gitlab-ci.yml  @devops-lead
Dockerfile      @devops-lead

# 支付模块——需要支付团队审批（必须有支付相关的 Code Owner approve）
payment/        @payment-lead @payment-senior

# 订单模块——需要订单团队审批
order/          @order-lead @order-senior

# 基础设施/配置——需要 DevOps 审批
terraform/      @devops-lead
k8s/            @devops-lead

# 数据库迁移——需要 DBA 审批
**/migrations/  @dba-lead

# 安全敏感文件
**/secrets/**   @security-lead
**/*.pem        @security-lead

# 所有 Go 文件——需要架构组审批
# *.go          @arch-lead
EOF

git add .gitlab/CODEOWNERS
git commit -m "chore: add CODEOWNERS for review enforcement"
git push origin main
```

**验证 Code Owner 规则生效**：

```bash
# 创建一个修改 payment 目录的 MR
git checkout -b feature/payment-fix
echo "// fix payment rounding bug" >> payment/process.js
git add payment/process.js
git commit -m "fix: payment rounding issue"
git push origin feature/payment-fix

# 在 GitLab 上创建 MR 后，查看 MR 页面
# 应该显示：
# - "Approval is required from @payment-lead or @payment-senior"
# - 合并按钮为灰色，直到指定 Code Owner 审批
```

#### 步骤4：配置推送规则（Push Rules）

**目标**：在 Git 推送层面拦截不合规的提交，减少 MR 阶段的返工。

**通过 API 配置 Push Rules**（部分功能需 EE）：

```bash
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "commit_message_regex": "^(feat|fix|docs|style|refactor|perf|test|chore|ci)(\\(.+\\))?: .{10,}",
    "commit_message_negative_regex": "WIP|TODO|FIXME",
    "branch_name_regex": "^(feature|bugfix|hotfix|release|chore)/[a-z0-9._-]+$",
    "deny_delete_tag": true,
    "member_check": true,
    "prevent_secrets": true,
    "max_file_size": 10
  }' \
  "$GITLAB_URL/api/v4/projects/$PROJECT_ID/push_rules"

# 参数说明：
# commit_message_regex: 强制使用 Conventional Commits 规范
# branch_name_regex: 强制分支命名规范（feature/、bugfix/、hotfix/）
# deny_delete_tag: 禁止删除 tag
# member_check: 检查提交者必须是项目成员
# prevent_secrets: 扫描是否包含密钥（AWS/GCP/SSH 私钥等）
# max_file_size: 单个文件最大 10MB
```

#### 步骤5：验证保护规则是否真正生效

**目标**：用实际 Git 操作验证保护规则——"把规则验证一遍，而不是看一眼就当配置好了"。

```bash
# 测试1：验证任何人都不能直接 push 到 main
git checkout main
echo "test" >> README.md
git add README.md && git commit -m "test direct push"
git push origin main
# 预期结果：Remote rejected → "You are not allowed to push code to protected branches"

# 测试2：验证 Developer 不能合并未经审批的 MR
# （用 Developer 账号）创建 MR → 尝试 merge
# 预期结果：Merge 按钮灰色 → "Requires approval"

# 测试3：验证 CI 不通过则不能合并
# 创建一个 MR，故意写一个会 CI 失败的改动
# 预期结果：Merge 按钮灰色 → "Pipeline must succeed"

# 测试4：验证 Code Owner 未审批则不能合并
# 修改 payment/ 目录文件 → 提交 MR
# 预期结果：即使有其他 Maintainer 审批，Code Owner 未审批前也不能合并

# 测试5：验证 force push 到 main 被拒绝
git push --force origin main
# 预期结果：Remote rejected

# 测试6：验证分支命名规范
git checkout -b wrong-name
git push origin wrong-name
# 预期结果：如果配置了 branch_name_regex，推送被拒绝
```

- Push Rules API 配置：分支/提交规范校验（步骤4）
- Protected Branch API 配置：分支保护规则（步骤1）

## 4. 项目总结

### 优点 & 缺点

| 特性 | 优点 | 缺点 |
|------|------|------|
| Protected Branches | 阻止误操作 force push，强制走 MR 流程 | 配置不当可能导致 hotfix 无法及时合并 |
| Code Owners | 领域专家强制审核，防止不熟悉代码的人乱审批 | Code Owner 请假/离职时需要及时更新文件 |
| Merge Checks | 自动化门禁（CI 通过 + 讨论解决），减少人为疏漏 | 所有检查都通过才能合并，紧急情况可能被阻塞 |
| Push Rules | 在推送阶段拦截问题，减少 MR 阶段返工 | 过于严格的规则可能阻碍开发（如 max_file_size 过小） |

### 适用场景

- **金融/支付系统**：必须开启所有安全规则（保护分支 + Code Owner + CI 检查）
- **多人协作的开源项目**：Code Owner + 多审批人确保代码质量
- **CI/CD 严格的项目**：Pipelines must succeed 是底线

**不适用场景**：
- 个人实验/原型项目（保护规则反而增加操作成本）
- 快速迭代的早期创业团队（可以先只保护 main 分支，逐步增加规则）

### 注意事项

- **保护分支不意味着不能删除**：删除保护分支的权限由 "Allowed to force push" 和 "unprotect_access_level" 控制
- **Code Owner 文件在每个分支独立生效**：如果在 feature 分支上修改了 CODEOWNERS 文件，该分支的 MR 审批规则会按修改后的文件执行
- **Merge Checks 中 CI 检查依赖 Runner**：确保至少有 1 个 Shared Runner 在线，否则所有 MR 都会因为 Pipeline 未运行而无法合并
- **Push Rules 只拦截客户端 push 操作**：通过 GitLab UI 创建的文件不会触发 Push Rules 检查

### 常见踩坑经验

1. **Code Owner 离职导致所有 MR 卡住**：某 Code Owner 请假两周，期间所有涉及他负责模块的 MR 都无法合并。根因：Code Owner 只设置了一个人。解决：每个模块至少设置 2 个 Code Owner（或使用 Group 替代个人）。
2. **Pipelines must succeed 导致 CI 故障时全公司瘫痪**：Shared Runner 全部宕机，所有 MR 的 Pipeline 无法运行，导致无人能合并代码。根因：没有 Runner 高可用和 CI 跳过机制。解决：配置 "Allow pipelines in the parent project to control whether they can skip" + Runner 集群化部署。
3. **Push Rule 的 prevent_secrets 误报**：开发者代码中包含示例密钥（`EXAMPLE_API_KEY = "sk-test-xxxx"`），被 Push Rule 拦截无法推送。根因：`prevent_secrets` 的正则无法区分示例和真实密钥。解决：在 `.gitlab/secret-detection.yml` 中配置排除规则，或用注释标记 `// gitlab-secret-detection:ignore`。

### 思考题

1. 如果 main 分支的 Allowed to push 设置为 "No one"，但紧急情况下需要直接修改 main（比如修复 CI 配置错误），你应该如何操作？有几种可行方案？
2. 假设你的团队有 3 个模块（支付、订单、用户），Code Owner 分别由 3 个不同的 Tech Lead 负责。如果一个 MR 同时修改了支付和订单模块的文件，合并条件是什么？

### 推广计划提示

- **开发**：理解保护分支的约束边界，学会在约束下高效协作（如提前准备 MR、及时更新 Code Owner）
- **运维**：Push Rules 和 Merge Checks 是运维团队治理代码质量的重要抓手
- **测试**：了解 Merged Checks 中 CI 检查的触发机制，有助于设计合理的测试流水线
