# 第9章：.gitlab-ci.yml 语法基础

## 1. 项目背景

> **业务场景**：一家公司刚搭建好 GitLab，项目经理兴冲冲地要求"所有项目都要配 CI/CD"，但开发团队对 `.gitlab-ci.yml` 一窍不通。有人把网上搜来的配置直接复制粘贴，结果 Pipeline 跑一次 40 分钟——因为它同时 build 了 3 个无关的环境。还有人把数据库密码硬编码在 ci 文件里，被安全审计抓了个现行。

最惨的一次：某开发写了一个 deploy job，脚本里有一行 `rm -rf /${DEPLOY_PATH}`，结果 `DEPLOY_PATH` 变量为空，脚本变成了 `rm -rf /`——幸好 Runner 是在 Docker 容器里执行的，只炸了容器本身。但如果 Runner 用的是 Shell Executor……

事后团队痛定思痛，决定系统性地学好 `.gitlab-ci.yml` 的语法规则——从 stages、jobs、variables 到 rules、needs、include，理解每一段配置的含义，而不是无脑复制。

**痛点放大**：`.gitlab-ci.yml` 是 GitLab CI/CD 的入口配置，语法看似简单但陷阱密布——YAML 缩进错误导致整个文件被忽略，variables 优先级搞混导致生产环境密码泄露，stages 顺序定义不当导致部署跑在测试之前。没有系统性地理解这些概念，CI/CD 不仅不能提升效率，反而会成为事故的温床。

## 2. 项目设计——剧本式交锋对话

**场景**：开发组在调试一个总是失败的 Pipeline，已经折腾了两个小时。

---

**小胖**："这个 Pipeline 又红了！我明明在 `.gitlab-ci.yml` 里定义了 build、test、deploy 三个 job，为什么它只执行了 build？"

**大师**："让我看看你的文件……果然，你没定义 `stages`。GitLab 默认的 stages 顺序是 `[.pre, build, test, deploy, .post]`，但如果你没有显式声明 stages，job 的 stage 默认是 `test`。你的 build job 写了 `stage: build`，但因为没有定义 stages 列表，GitLab 不知道这些 stage 的顺序关系。"

**小白**："那 before_script 和 script 有什么区别？我看有些配置里两个都有。"

**大师**："before_script 在每个 job 的 script 之前运行，用于做共性的准备工作——比如安装依赖。after_script 在每个 job 之后运行，无论 job 成功还是失败——比如清理临时文件。技术映射——before_script 就像去健身房前的热身，script 是你实际的锻炼，after_script 是锻炼后的拉伸——不管你今天练得好不好，拉伸都要做。"

**小胖**："那 variables 呢？我定义在文件顶部的变量和定义在 job 里的变量，哪个优先级高？"

**大师**："变量优先级从低到高是：GitLab 预定义变量 < .gitlab-ci.yml 全局 variables < job 级别 variables < 触发器传入变量 < Project/Group CI Variables。同级时，后定义的覆盖先定义的。"

**小白**："我还有一个问题：only/except 和 rules 都是控制 job 执行条件的，有什么区别？"

**大师**："only/except 是旧语法，简单但不够灵活——比如 'only: main' 表示只在 main 分支执行。rules 是新语法，更强大——你可以组合多个条件（if + changes + exists + when）。而且 rules 支持更细粒度的控制：你可以分别控制 job 是否创建、是否允许手动触发、失败时是否阻塞后续 stage。官方推荐用 rules 替代 only/except。"

**小胖**："那 when 又是什么？我看到有 when: manual、when: delayed 之类的。"

**大师**："when 控制 job 的执行时机：on_success（默认，前面的 job 都成功才执行）、on_failure（前面有失败时执行，用于通知）、always（无论如何都执行）、manual（需要手动点击触发）、delayed（等待指定时间后自动执行）。技术映射——when 就像高速公路上的关卡：有的关卡自动放行（on_success），有的需要停车检查（manual），有的让你等一段时间（delayed）。"

## 3. 项目实战

### 环境准备

> **目标**：为一个 Node.js 项目编写完整的 `.gitlab-ci.yml`，覆盖 stages、jobs、variables、rules、cache、artifacts 等核心语法。

**前置条件**：GitLab 项目 + Docker Runner（参考第10章）。

### 分步实现

#### 步骤1：编写最小化的 CI 配置文件

**目标**：从最简单的 2 个 job 开始，理解 stages 和 script 的基本用法。

```bash
# 在项目根目录创建 .gitlab-ci.yml
cat > .gitlab-ci.yml << 'EOF'
# ============================================
# 最小化 CI 配置：理解 stages 和 job
# ============================================

# 定义流水线阶段（按顺序执行）
stages:
  - build
  - test

# 定义全局变量
variables:
  NODE_VERSION: "20-alpine"
  APP_NAME: "shop-api"

# Job 1: 构建
build-job:
  stage: build
  image: node:${NODE_VERSION}
  script:
    - echo "开始构建 ${APP_NAME}..."
    - npm ci
    - npm run build
    - echo "构建完成，产物在 dist/ 目录"

# Job 2: 测试
test-job:
  stage: test
  image: node:${NODE_VERSION}
  script:
    - echo "开始测试 ${APP_NAME}..."
    - npm ci
    - npm run test
  # 仅在 build-job 成功后运行（默认行为）
EOF

# 提交并观察 Pipeline 运行
git add .gitlab-ci.yml
git commit -m "ci: add minimal pipeline with build and test stages"
git push origin main

# 在 GitLab UI 中：
# CI/CD → Pipelines → 点击最新的 Pipeline
# 查看 job 日志，确认 stages 按顺序执行
```

#### 步骤2：添加 before_script、after_script、cache 和 artifacts

**目标**：优化 CI 配置——减少重复安装、缓存依赖、传递构建产物。

```bash
cat > .gitlab-ci.yml << 'EOF'
stages:
  - install
  - build
  - test
  - deploy

variables:
  NODE_VERSION: "20-alpine"

# ============================================
# 阶段 1: 安装依赖（独立出来以便缓存）
# ============================================
install-deps:
  stage: install
  image: node:${NODE_VERSION}
  script:
    - npm ci --prefer-offline
  cache:
    key:
      files:
        - package-lock.json    # 按 lock 文件生成缓存 key
    paths:
      - node_modules/          # 缓存 node_modules
    policy: pull-push          # 读写缓存（默认值）
  artifacts:
    paths:
      - node_modules/          # 传递给后续 job 的制品
    expire_in: 1 hour          # 制品 1 小时后过期

# ============================================
# 阶段 2: 构建
# ============================================
build-job:
  stage: build
  image: node:${NODE_VERSION}
  before_script:
    - echo "=== 构建开始 [$(date)] ==="
  script:
    - npm run build
    - ls -la dist/
  after_script:
    - echo "=== 构建结束 [$(date)] ==="
  cache:
    key:
      files:
        - package-lock.json
    paths:
      - node_modules/
    policy: pull               # 只读缓存（不需要写入）
  artifacts:
    paths:
      - dist/                  # 构建产物传递给后续 job
    expire_in: 1 week
  needs:
    - install-deps             # 指定依赖关系（比 stage 顺序更灵活）

# ============================================
# 阶段 3: 测试（包含代码质量检查）
# ============================================
lint-job:
  stage: test
  image: node:${NODE_VERSION}
  script:
    - npm run lint
  cache:
    key:
      files:
        - package-lock.json
    paths:
      - node_modules/
    policy: pull
  needs:
    - install-deps

unit-test-job:
  stage: test
  image: node:${NODE_VERSION}
  script:
    - npm run test
  cache:
    key:
      files:
        - package-lock.json
    paths:
      - node_modules/
    policy: pull
  needs:
    - install-deps
  artifacts:
    when: always               # 无论测试通过与否都保存报告
    reports:
      junit: junit.xml         # GitLab 自动解析 JUnit 格式

# ============================================
# 阶段 4: 部署（手动触发）
# ============================================
deploy-staging:
  stage: deploy
  image: alpine:latest
  before_script:
    - apk add --no-cache openssh-client
  script:
    - echo "Deploying to staging server..."
    - echo "APP: ${APP_NAME}"
    - echo "VERSION: ${CI_COMMIT_SHORT_SHA}"
  environment:
    name: staging
    url: https://staging.shop.example.com
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'  # 仅 main 分支
  needs:
    - build-job
    - unit-test-job
  when: manual                  # 需要手动点击触发
EOF
```

#### 步骤3：使用 rules 实现精确的条件控制

**目标**：根据分支类型、文件变更、MR 事件等条件，精确控制 job 是否执行。

```bash
cat >> .gitlab-ci.yml << 'EOF'

# ============================================
# 高级 rules 示例
# ============================================

# 仅在 MR 中运行的 job
review-app:
  stage: deploy
  image: alpine:latest
  script:
    - echo "Deploying review app for MR !${CI_MERGE_REQUEST_IID}"
    - echo "URL: https://${CI_COMMIT_REF_SLUG}.review.shop.example.com"
  environment:
    name: review/${CI_COMMIT_REF_SLUG}
    url: https://${CI_COMMIT_REF_SLUG}.review.shop.example.com
    on_stop: stop-review-app     # MR 合并后自动停止环境
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
      when: manual
      allow_failure: true        # 允许失败，不阻塞后续 job

# 停止 review 环境的 job
stop-review-app:
  stage: deploy
  image: alpine:latest
  script:
    - echo "Stopping review app..."
  environment:
    name: review/${CI_COMMIT_REF_SLUG}
    action: stop
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
      when: manual
  variables:
    GIT_STRATEGY: none           # 不需要拉取代码

# 仅当 package.json 变更时才运行
dependency-audit:
  stage: test
  image: node:${NODE_VERSION}
  script:
    - npm audit --audit-level=high
  rules:
    - changes:
        - package.json
        - package-lock.json
      when: always
    - when: never

# 定时任务（每天晚上 2 点跑全面测试）
nightly-full-test:
  stage: test
  image: node:${NODE_VERSION}
  script:
    - npm run test:full
    - npm run test:e2e
  rules:
    - if: '$CI_PIPELINE_SOURCE == "schedule"'
EOF
```

#### 步骤4：使用 include 拆分 CI 配置

**目标**：将复杂 CI 配置拆分为多个文件，提高可维护性。

```bash
# 创建 CI 模板目录
mkdir -p .gitlab/ci

# 将基础配置拆分出去
cat > .gitlab/ci/base.yml << 'EOF'
# 基础构建模板
.build-template:
  image: node:${NODE_VERSION}
  before_script:
    - echo "Job started at $(date)"
    - npm ci --prefer-offline
  cache:
    key:
      files:
        - package-lock.json
    paths:
      - node_modules/
    policy: pull

# 基础测试模板
.test-template:
  extends: .build-template
  coverage: '/All files[^|]*\|[^|]*\s+([\d\.]+)/'
EOF

cat > .gitlab/ci/jobs.yml << 'EOF'
# Job 定义文件
build-job:
  extends: .build-template
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/

unit-test:
  extends: .test-template
  stage: test
  script:
    - npm run test
  artifacts:
    reports:
      junit: junit.xml

lint:
  extends: .test-template
  stage: test
  script:
    - npm run lint
EOF

cat > .gitlab/ci/deploy.yml << 'EOF'
# 部署 Job 定义
.deploy-template:
  stage: deploy
  image: alpine:latest
  before_script:
    - apk add --no-cache openssh-client curl

deploy-staging:
  extends: .deploy-template
  script:
    - echo "Deploying ${CI_COMMIT_SHORT_SHA} to staging"
  environment:
    name: staging
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
EOF

# 修改主 .gitlab-ci.yml 使用 include
cat > .gitlab-ci.yml << 'EOF'
# ============================================
# 主 CI 配置文件（使用 include 拆分）
# ============================================

stages:
  - build
  - test
  - deploy

variables:
  NODE_VERSION: "20-alpine"

# 引用外部 CI 文件
include:
  - local: .gitlab/ci/base.yml
  - local: .gitlab/ci/jobs.yml
  - local: .gitlab/ci/deploy.yml
EOF
```

### 完整代码清单

- `.gitlab-ci.yml`：主 CI 配置文件
- `.gitlab/ci/base.yml`：基础模板
- `.gitlab/ci/jobs.yml`：构建和测试 job
- `.gitlab/ci/deploy.yml`：部署 job

## 4. 项目总结

### 优点 & 缺点

| 特性 | 优点 | 缺点 |
|------|------|------|
| YAML 配置 | 人类可读，版本可追踪，与代码同仓 | 缩进敏感，容易写错 |
| stages | 清晰的流水线阶段划分 | 线性模型对复杂 DAG 不够灵活 |
| rules | 灵活的条件控制，替代 only/except | 语法比 only/except 复杂，学习曲线 |
| include | 支持拆分文件、跨项目引用 | 多层 include 嵌套时难以调试 |
| extends | 复用 job 定义，减少重复 | 多级继承时字段合并规则不直观 |

### 适用场景

- **持续集成**：每次 push 自动构建 + 测试 + 代码检查
- **持续部署**：main 分支自动部署到 staging，打 tag 自动部署到 production
- **多环境管理**：为每个 MR 自动创建 review 环境
- **定时任务**：夜间跑全量测试、数据备份等

**不适用场景**：
- 纯静态网站（GitHub Pages + Actions 更简单）
- 不需要 CI/CD 的文档/配置仓库

### 注意事项

- **YAML 缩进必须用空格**：不能用 Tab，否则文件解析失败
- **变量中不要硬编码敏感信息**：数据库密码、API Key 等请使用 Project/Group CI Variables
- **cache 不是 artifacts**：cache 可能不命中（缓存被清理），不要依赖 cache 传递构建结果
- **artifact:expire_in 要合理设置**：过长的保存时间会耗尽存储配额

### 常见踩坑经验

1. **YAML 语法错误导致整个文件被忽略**：忘了 `:` 后面的空格，`script:echo hello` 写成 `script:echo hello`（缺少空格）。根因：YAML 要求 `key: value` 之间必须有空格。解决：使用 CI Lint 工具预先检查。
2. **Cache 不命中**：cache key 设为 `${CI_COMMIT_REF_SLUG}`，但在 MR 中分支名变了。根因：cache key 的选择不合理。解决：cache key 应该基于 `package-lock.json` 的内容 hash，这样只要依赖没变就会命中缓存。
3. **when: manual 的 job 被跳过**：配置了 `when: manual`，但是在 MR 的 Pipeline 视图中找不到手动触发的按钮。根因：该 job 被 rules 排除（例如 `if: '$CI_COMMIT_BRANCH == "main"'` 在 MR 中不满足）。解决：检查 rules 条件是否覆盖了目标场景。

### 思考题

### 推广计划提示

- **开发**：每个项目都必须有 `.gitlab-ci.yml`，CI Lint 应该在提交前就验证通过
- **运维**：通过 include 引用运维团队的 CI 模板（安全扫描、合规检查），实现统一治理
- **测试**：利用 artifacts:reports 的 junit 格式，GitLab 会自动生成测试报告和覆盖率图表
