# 第14章：容器镜像仓库基础

## 1. 项目背景

> **业务场景**：某公司全面容器化后，Docker 镜像一直用的是 Docker Hub 免费版。但随着镜像数量增加到 200+，Docker Hub 开始限速——每次 CI 拉取基础镜像都超时，Pipeline 成功率从 95% 暴跌到 30%。更糟的是，有一次生产环境的关键镜像被团队误删，而 Docker Hub 上只有一个 `latest` 标签——无法回滚到上一个版本。

CTO 拍板："启用 GitLab Container Registry，镜像和代码放在同一个平台管理。"但团队很快又遇到了新问题：
- CI 构建的镜像推送到内网 Registry 时端口映射不对，怎么都连不上
- 镜像标签管理混乱——有的项目用 `latest`，有的用 commit SHA，有的完全没标签策略
- 镜像存储持续膨胀，一个月积累了 300GB 垃圾镜像（旧版本无人清理）
- 镜像构建太慢——每次 CI 都从零构建，没有利用 Docker layer caching

**痛点放大**：GitLab Container Registry 不是"打开开关就能用好"的功能。你需要在 CI 中配置安全的认证登录、设计合理的标签策略、配置自动清理规则、利用多阶段构建和 layer caching 加速构建。如果这些工作没做好，Registry 不仅不能解决 Docker Hub 的问题，反而会变成新的性能瓶颈和存储黑洞。

## 2. 项目设计——剧本式交锋对话

**场景**：DevOps 团队在讨论容器化 CI/CD 的优化方案。

---

**小胖**："Docker Hub 限速真的太恶心了，我们一小时内拉了 100 次 `node:20-alpine` 就被限制了。难道每次 CI 都要拉一遍基础镜像吗？"

**大师**："这就是 GitLab Dependency Proxy 的用武之地。它本质上是一个镜像缓存代理——你第一次拉 `node:20-alpine` 时，Dependency Proxy 会从 Docker Hub 拉取并缓存到你的 GitLab 实例上。后续所有 CI job 直接从 GitLab 拉取，完全不走 Docker Hub 的限速。除了加速，它还能审计所有镜像拉取记录。"

**小胖**："那 Container Registry 和 Dependency Proxy 有什么区别？"

**大师**："Container Registry 存的是你'自己构建'的镜像——你的应用、你的服务。Dependency Proxy 存的是'别人构建'的镜像——node、nginx、alpine 等基础镜像。技术映射：Registry 是你家的仓库，存你自己生产的东西；Dependency Proxy 是你家门口的代收点，帮你收快递（Docker Hub 的镜像）。"

**小白**："镜像标签策略呢？我们用 `latest` 标签部署，结果有一次 `latest` 指向了一个有 bug 的版本，回滚时发现上一个 `latest` 已经没了。"

**大师**："`latest` 是最危险的标签——它不携带任何版本信息，你不知道它指向哪个构建。一个好的标签策略应该包含三层信息：版本语义（`v1.2.3`）、Git 信息（`commit-abc1234`）、构建元数据（`20260512-build42`）。结合 GitLab CI 的预定义变量，你可以自动生成有意义的标签。"

**小胖**："那镜像清理呢？我看 Registry 存储一天天涨。"

**大师**："GitLab 提供了 tag cleanup policy 来自动清理旧标签。你可以按规则清理：保留最近 10 个标签、删除 30 天前的标签、匹配特定正则（如 `feature-*`）的标签到期清理。技术映射：Tag cleanup 就像手机照片的自动清理——不是一次性删除，而是按规则保留重要的。"

## 3. 项目实战

### 环境准备

> **目标**：在 CI 中构建 Docker 镜像并推送到 GitLab Container Registry，配置完整的标签策略和自动清理。

**前置条件**：GitLab 项目 + Docker Runner（privileged 模式），Container Registry 已启用（默认启用）。

### 分步实现

#### 步骤1：启用和验证 Container Registry

**目标**：确认 GitLab Container Registry 正常工作，配置 CI 认证。

```bash
# 查看 Registry 状态（管理员）
gitlab-rails runner "puts Gitlab.config.registry.enabled"

# Registry URL 格式：
# <gitlab-url>:<registry-port>/<namespace>/<project>
# 例如：gitlab.local:5050/acme-corp/ecommerce/shop-api

# 在 GitLab UI 中查看 Registry：
# Project → Packages and Registries → Container Registry

# 测试 Registry 登录
docker login gitlab.local:5050
# Username: <your-gitlab-username>
# Password: <personal-access-token-with-read_registry+write_registry-scope>

# 或使用 CI Job Token（仅 CI 内可用）
# docker login -u gitlab-ci-token -p $CI_JOB_TOKEN $CI_REGISTRY
```

#### 步骤2：编写多阶段 Dockerfile

**目标**：用多阶段构建优化镜像大小，分离编译环境和运行环境。

```bash
# Dockerfile - Node.js 多阶段构建示例
cat > Dockerfile << 'EOF'
# ===== 阶段 1: 构建 =====
FROM node:20-alpine AS builder

WORKDIR /app

# 先复制依赖文件（利用 Docker layer cache）
COPY package.json package-lock.json ./
RUN npm ci --only=production --no-audit

# 复制源代码
COPY . .

# 构建
RUN npm run build

# ===== 阶段 2: 运行 =====
FROM node:20-alpine AS runner

# 安全：以非 root 用户运行
RUN addgroup -g 1001 -S nodejs && \
    adduser -S nodejs -u 1001

WORKDIR /app

# 只复制生产依赖和构建产物（不复制源码和 devDependencies）
COPY --from=builder --chown=nodejs:nodejs /app/node_modules ./node_modules
COPY --from=builder --chown=nodejs:nodejs /app/dist ./dist
COPY --from=builder --chown=nodejs:nodejs /app/package.json ./

# 切换到非 root 用户
USER nodejs

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget --no-verbose --tries=1 --spider http://localhost:3000/health || exit 1

EXPOSE 3000

CMD ["node", "dist/main.js"]
EOF

# 查看优化效果
# 构建后对比：
# - 未优化的单阶段镜像：~500MB
# - 多阶段构建后：~80MB
# 减少约 84%！
```

#### 步骤3：在 CI 中构建并推送镜像

**目标**：完整的 CI job——构建镜像、打标签、推送到 Registry。

```yaml
# .gitlab-ci.yml
stages:
  - build
  - push

variables:
  DOCKER_DRIVER: overlay2
  DOCKER_TLS_CERTDIR: ""        # 禁用 TLS（内网 Registry 不需要）
  IMAGE_TAG: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

# 构建 Docker 镜像
docker-build:
  stage: build
  image: docker:24-dind
  services:
    - docker:24-dind
  before_script:
    # 登录到 GitLab Container Registry
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    # 构建镜像（利用 layer cache）
    - docker build
        --cache-from $CI_REGISTRY_IMAGE:latest
        --build-arg BUILDKIT_INLINE_CACHE=1
        -t $IMAGE_TAG
        -t $CI_REGISTRY_IMAGE:latest
        .
    # 推送到 Registry
    - docker push $IMAGE_TAG
    - docker push $CI_REGISTRY_IMAGE:latest
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
    - if: '$CI_COMMIT_TAG'

# 为每个 MR 构建预览镜像（不推 latest）
docker-build-review:
  stage: build
  image: docker:24-dind
  services:
    - docker:24-dind
  before_script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
  script:
    - docker build
        -t $CI_REGISTRY_IMAGE:review-$CI_COMMIT_REF_SLUG
        .
    - docker push $CI_REGISTRY_IMAGE:review-$CI_COMMIT_REF_SLUG
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
  environment:
    name: review/$CI_COMMIT_REF_SLUG
```

#### 步骤4：使用 Dependency Proxy 加速基础镜像拉取

**目标**：配置 Dependency Proxy，避免 Docker Hub 限速。

```bash
# Dependency Proxy URL 格式：
# $CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX/<image>
# 例如：gitlab.local:443/acme-corp/dependency_proxy/containers/node:20-alpine

# 在 Dockerfile 中使用 Dependency Proxy
cat > Dockerfile.deploy << 'EOF'
# 使用 GitLab Dependency Proxy 拉取基础镜像
ARG DEPENDENCY_PROXY=${CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX}
FROM ${DEPENDENCY_PROXY}/node:20-alpine AS builder

# ... 其余不变 ...
EOF

# 在 CI 中使用 Dependency Proxy 拉取镜像
# .gitlab-ci.yml
dependency-proxy-demo:
  stage: build
  image: ${CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX}/node:20-alpine
  script:
    - echo "Using Dependency Proxy for base image"
    - node -v
```

#### 步骤5：配置 Tag Cleanup Policy

**目标**：自动清理旧标签，防止存储膨胀。

**通过 GitLab UI 配置**：

```
Project → Settings → Packages and Registries → Container Registry cleanup policy

配置：
  ✅ Enabled
  保留最近的标签数量: 10         # 保留最新的 10 个标签
  保留匹配标签: .*               # 所有标签参与清理
  删除旧于: 30 天                # 30 天前的标签删除
  清理计划: 每天凌晨执行
  
排除规则（可选）：
  保留匹配: ^v\d+\.\d+\.\d+$     # 保留语义版本标签（v1.2.3）
  保留匹配: ^latest$             # 保留 latest 标签
```

**通过 API 配置**：

```bash
curl --request PUT \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "container_expiration_policy_attributes": {
      "cadence": "1d",
      "enabled": true,
      "keep_n": 10,
      "older_than": "30d",
      "name_regex": ".*",
      "name_regex_keep": "^(latest|v\\d+\\.\\d+\\.\\d+)$"
    }
  }' \
  "$GITLAB_URL/api/v4/projects/$PROJECT_ID"
```

## 4. 项目总结

### 优点 & 缺点

| 特性 | 优点 | 缺点 |
|------|------|------|
| Container Registry | 与代码同平台，一次认证；自动关联项目 | 大镜像（>5GB）推送可能超时 |
| Dependency Proxy | 避免 Docker Hub 限速；内网缓存加速 | 仅缓存 Docker Hub 的镜像 |
| Tag Cleanup | 自动化管理存储，防止爆炸 | 清理不是实时的，有延迟 |
| 多阶段构建 | 镜像大小减少 80%+ | 构建时间略有增加 |

### 镜像标签策略对比

| 策略 | 标签示例 | 优点 | 缺点 |
|------|---------|------|------|
| Git SHA | `abc1234` | 唯一、可追溯 | 不包含版本语义 |
| 语义版本 | `v1.2.3` | 清晰表达变更 | 需要手动打 tag |
| 分支名 | `feature-login` | 区分功能 | 分支删除后标签无用 |
| latest | `latest` | 简单 | 不可追溯，回滚困难 |
| **推荐组合** | `v1.2.3-abc1234` | 唯一 + 可读 | 标签稍长 |

### 适用场景

- **内部服务容器镜像**：最标准的用法，与 CI/CD 深度集成
- **多团队共享基础镜像**：通过 CI 自动构建和发布共享的基础镜像（如 Node 环境镜像）
- **私有化部署的镜像分发**：不需要外部 Registry，内网传输速度更快

**不适用场景**：
- 需要全球 CDN 分发的公开镜像（Docker Hub 的 CDN 更有优势）
- 需要 Helm Chart 完整分发（GitLab 的 Helm Registry 单独配置，见第27章）

### 注意事项

- **Registry 口令安全**：不要在脚本中硬编码 `docker login` 的密码，使用 `$CI_REGISTRY_PASSWORD` 自动变量
- **Docker layer cache**：Layer cache 基于 Dockerfile 指令顺序，把不常变的指令（如 `COPY package.json`）放在前面
- **多层构建的中间层**：只有最后一个 `FROM` 的镜像会被推送到 Registry，中间层不占用 Registry 空间
- **Registry 清理的延迟**：Tag cleanup policy 的执行是渐进的——不是设置后立即清理

### 常见踩坑经验

1. **Registry 端口映射问题**：Omnibus 安装默认 Registry 用 5005 端口，但有些环境被防火墙拦截。根因：Registry 默认监听 localhost:5000，Nginx 代理到 5005。解决：检查 `/etc/gitlab/gitlab.rb` 中 `registry_external_url` 的端口配置。
2. **docker push 提示 authentication required**：使用了正确用户名但密码错误。根因：`$CI_REGISTRY_PASSWORD` 是 CI Job Token，不是 Personal Access Token。解决：CI 内用 `-u $CI_REGISTRY_USER -p $CI_JOB_TOKEN`，本地用 Personal Access Token。
3. **Dependency Proxy 拉取失败**：环境变量 `CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX` 为空。根因：Dependency Proxy 需要在 Group 级别启用。解决：Group → Settings → Packages & Registries → Dependency Proxy → Enable。

### 推广计划提示

- **开发**：多阶段 Dockerfile 是镜像优化的基本功，务必掌握 layer cache 优化技巧
- **运维**：Dependency Proxy + Tag Cleanup Policy 的组合可以大幅降低外部依赖和存储成本
- **安全**：Registry 中的镜像也需要做安全扫描（SAST），在 push 前或定期扫描镜像中的 CVE 漏洞
