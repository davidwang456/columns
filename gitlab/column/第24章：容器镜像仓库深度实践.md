# 第24章：容器镜像仓库深度实践

## 1. 项目背景

> **业务场景**：一家公司使用 GitLab Container Registry 半年后，镜像数量从最初的 20 个飙升到 500+ 个。运维发现几个严重问题：CI 构建的镜像越来越大（从 80MB 膨胀到 1.2GB），开发经常抱怨"pull 镜像要 5 分钟"；有一个生产环境的镜像被打上了 `latest` 标签，被某个 CI 任务意外覆盖，导致线上故障；镜像没有签名验证机制，理论上任何人都可以推送一个同名镜像到 Registry。

更糟的是：公司为了省钱，Docker 镜像的基础镜像（node、nginx、alpine）每次都是从 Docker Hub 实时拉取。Docker Hub 限速后，每天的 CI 构建有 30% 因为镜像拉取超时而失败。运维团队紧急启用了 Dependency Proxy，但配置不当，缓存很快就满了——3 天就积累了 200GB。

**痛点放大**：容器镜像仓库的管理是一个持续优化过程——不是"搭好就能用"。镜像瘦身（多阶段构建、layer 优化）、标签治理（语义化标签 + 自动清理）、安全加固（镜像签名、漏洞扫描）、缓存加速（Dependency Proxy、layer cache）——每个维度都会影响 CI 速度、存储成本和安全性。

## 2. 项目设计——剧本式交锋对话

**场景**：运维工位，小王正在清理 Registry 存储空间，一脸无奈。

---

**小王**："Registry 存储又满了！500 个镜像标签，有些是 6 个月前的 feature 分支的预览镜像，有些是 daily build 的临时镜像。怎么清理都清不完。"

**大师**："清理 Registry 不能靠手动删除。你需要配置 tag cleanup policy——根据规则自动清理：保留最近 N 个标签、删除超过 X 天的标签、用正则匹配保留重要标签（如 v* 版本标签）。这样那些 feature 分支的临时标签 7 天后自动消失，不需要人操心。"

**小胖**："镜像太大怎么办？我们的 Java 服务镜像从最初的 200MB 变成了 1.5GB。"

**大师**："1.5GB 的镜像通常不是代码本身大，而是包含了不该包含的东西。检查三点：第一，是不是在 Dockerfile 中 `COPY . .` 把所有文件（包括 node_modules、target、.git）都打包了？第二，有没有用多阶段构建——在编译器阶段用完整 JDK，在运行阶段只复制 JRE？第三，基础镜像是不是用的 `ubuntu:latest`（72MB）而不是 `alpine:3.19`（7MB）？"

**小白**："镜像签名是什么？有必要吗？"

**大师**："镜像签名（Cosign/Notary）可以验证镜像的来源和完整性。举个例子——如果有人获得了你的 Registry 推送权限，推送了一个同名的恶意镜像，你怎么知道拉取的是你构建的？镜像签名可以在部署时验证'这个镜像确实是我们 CI 构建的，中间没有被篡改过'。技术映射——镜像签名就像你网购电子产品时盒子上的一次性封条，打开过就能看出来。"

## 3. 项目实战

### 环境准备

> **目标**：深度优化容器镜像的构建、标签、缓存、签名和清理全流程。

**前置条件**：GitLab CE 17.x，Container Registry 已启用，Docker Runner。

### 分步实现

#### 步骤1：镜像瘦身——多阶段构建 + Layer 优化

**目标**：将 Java 或 Node.js 镜像从 1GB+ 优化到 100MB 以内。

**Node.js 优化前后对比**：

```dockerfile
# ===== 优化前（约 350MB）=====
FROM node:20
WORKDIR /app
COPY . .
RUN npm ci
RUN npm run build
CMD ["node", "dist/main.js"]

# ===== 优化后（约 80MB）=====
# Stage 1: 构建
FROM node:20-alpine AS builder
WORKDIR /app
# 分开 COPY 以利用 Docker layer cache
COPY package.json package-lock.json ./
RUN npm ci --only=production --no-audit
COPY . .
RUN npm run build

# Stage 2: 运行（极简）
FROM node:20-alpine AS runner
# 安全：创建非 root 用户
RUN addgroup -g 1001 -S appuser && adduser -S appuser -u 1001
WORKDIR /app
# 只复制生产依赖和构建产物
COPY --from=builder --chown=appuser:appuser /app/node_modules ./node_modules
COPY --from=builder --chown=appuser:appuser /app/dist ./dist
COPY --from=builder --chown=appuser:appuser /app/package.json ./
USER appuser
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget -q --spider http://localhost:3000/health || exit 1
CMD ["node", "dist/main.js"]
```

**.dockerignore 文件**（减少构建上下文）：

```dockerfile
node_modules
dist
.git
.gitlab-ci.yml
Dockerfile
docker-compose*.yml
*.md
.env*
coverage
.nyc_output
.vscode
.idea
```

**Java 镜像优化示例**：

```dockerfile
# Stage 1: Maven 编译（大镜像）
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
# 先下载依赖（利用 layer cache）
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests -B

# Stage 2: 运行（只用 JRE，不用完整 JDK）
FROM eclipse-temurin:17-jre-alpine
RUN addgroup -S app && adduser -S app -G app
WORKDIR /app
COPY --from=builder --chown=app:app /app/target/*.jar app.jar
USER app
EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -q --spider http://localhost:8080/actuator/health || exit 1
ENTRYPOINT ["java", "-jar", "app.jar"]
```

#### 步骤2：镜像标签治理

**目标**：设计生产级标签策略，告别 `latest` 滥用。

```yaml
# .gitlab-ci.yml - 标签策略
stages:
  - build-image

variables:
  IMAGE_BASE: $CI_REGISTRY_IMAGE

docker-build:
  stage: build-image
  image: gcr.io/kaniko-project/executor:debug
  script:
    - |
      # 根据分支/Pipeline 来源生成标签
      if [ -n "$CI_COMMIT_TAG" ]; then
        # Tag 触发 → 版本标签
        TAG_SEMVER="$CI_COMMIT_TAG"
        TAG_MAJOR=$(echo $CI_COMMIT_TAG | cut -d. -f1)
        TAG_MINOR=$(echo $CI_COMMIT_TAG | cut -d. -f1-2)
        DESTINATIONS="--destination $IMAGE_BASE:$TAG_SEMVER \
                      --destination $IMAGE_BASE:$TAG_MAJOR \
                      --destination $IMAGE_BASE:$TAG_MINOR"

      elif [ "$CI_COMMIT_BRANCH" = "main" ]; then
        # Main 分支 → stable 标签
        TIMESTAMP=$(date +%Y%m%d-%H%M%S)
        DESTINATIONS="--destination $IMAGE_BASE:stable \
                      --destination $IMAGE_BASE:$CI_COMMIT_SHORT_SHA \
                      --destination $IMAGE_BASE:main-$TIMESTAMP"

      elif [ "$CI_PIPELINE_SOURCE" = "merge_request_event" ]; then
        # MR → 预览标签
        DESTINATIONS="--destination $IMAGE_BASE:mr-$CI_MERGE_REQUEST_IID"

      else
        # Feature 分支
        DESTINATIONS="--destination $IMAGE_BASE:branch-$CI_COMMIT_REF_SLUG"
      fi

      /kaniko/executor \
        --context $CI_PROJECT_DIR \
        --dockerfile Dockerfile \
        $DESTINATIONS \
        --cache=true --cache-ttl=168h
```

**标签策略设计表**：

| 触发场景 | 标签示例 | 生命周期 | 说明 |
|---------|---------|---------|------|
| Tag `v1.2.3` | `v1.2.3`, `v1`, `v1.2` | 永久保留 | 正式发布版本 |
| main 分支 | `stable`, `abc1234`, `main-20260512` | 30 天 | 最新稳定版 |
| Feature 分支 | `branch-feat-login` | 7 天 | 开发中版本 |
| MR | `mr-42` | MR 合并后删除 | 预览/Review |
| 每日构建 | `nightly-20260512` | 7 天 | 每日自动化构建 |

#### 步骤3：Dependency Proxy 深度配置

**目标**：配置 Dependency Proxy 缓存，彻底解决 Docker Hub 限速。

```bash
# 1. 在 Group 级别启用 Dependency Proxy
# Group → Settings → Packages and Registries → Dependency Proxy
# ✅ Enable

# 2. Dependency Proxy URL 格式：
# $CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX/<image>:<tag>
# 实际地址：gitlab.local:443/acme-corp/dependency_proxy/containers/

# 3. 在 Dockerfile 中使用 Dependency Proxy
cat > Dockerfile << 'DOCKERFILE'
# 使用 ARG 在 CI 中注入 Dependency Proxy 前缀
ARG DEPENDENCY_PROXY=${CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX}

FROM ${DEPENDENCY_PROXY}/node:20-alpine AS builder
# ... 后续不变
DOCKERFILE

# 4. 在 CI 中配置
cat >> .gitlab-ci.yml << 'EOF'
variables:
  # 启用 Dependency Proxy
  DOCKER_BUILDKIT: 1
  BUILDKIT_INLINE_CACHE: 1

docker-build:
  stage: build-image
  image: gcr.io/kaniko-project/executor:debug
  script:
    - |
      /kaniko/executor \
        --context $CI_PROJECT_DIR \
        --dockerfile Dockerfile \
        --build-arg DEPENDENCY_PROXY=${CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX} \
        --destination $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA \
        --cache=true \
        --cache-repo=${CI_DEPENDENCY_PROXY_GROUP_IMAGE_PREFIX}/cache
EOF

# 5. 查看 Dependency Proxy 使用情况和存储
# Group → Packages & Registries → Dependency Proxy
# 显示缓存的镜像列表和大小
```

#### 步骤4：镜像签名（Cosign）

**目标**：使用 Cosign 对镜像签名，在部署时验证镜像完整性。

```yaml
# .gitlab-ci.yml - Cosign 镜像签名
stages:
  - build-image
  - sign-image

docker-build:
  stage: build-image
  image: gcr.io/kaniko-project/executor:debug
  script:
    - /kaniko/executor --destination $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA ...

# 用 Cosign 对镜像签名
sign-image:
  stage: sign-image
  image: bitnami/cosign:2.2
  script:
    # 生成无密钥签名（keyless signing）
    - cosign sign --yes $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

    # 或用私钥签名
    # - echo "$COSIGN_PRIVATE_KEY" > cosign.key
    # - cosign sign --key cosign.key $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
    - if: '$CI_COMMIT_TAG'
  variables:
    COSIGN_EXPERIMENTAL: 1    # 启用 keyless signing
  needs:
    - docker-build
```

**部署时验证镜像签名**：

```bash
# 在部署脚本中验证签名
cosign verify \
  --certificate-identity https://gitlab.local/acme-corp/project//.gitlab-ci.yml@refs/heads/main \
  --certificate-oidc-issuer https://gitlab.local \
  $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

if [ $? -ne 0 ]; then
  echo "❌ Image signature verification failed! Aborting deployment."
  exit 1
fi
echo "✅ Image signature verified. Proceeding with deployment."
```

## 4. 项目总结

### 镜像优化效果一览

| 优化项 | 效果 | 典型节省 |
|--------|------|---------|
| 多阶段构建 | 分离编译和运行环境 | 60-80% 体积减少 |
| alpine 基础镜像 | 替代 ubuntu/debian | 70MB → 7MB |
| .dockerignore | 减少构建上下文 | 构建时间减少 30% |
| Layer cache | 利用 Docker 缓存 | 重复构建 < 1 分钟 |
| Dependency Proxy | 缓存外部镜像 | 消除 Docker Hub 限速 |

### 适用场景

- **镜像瘦身**：任何项目都应该做，最简单的 ROI 最高的优化
- **标签策略**：团队 > 2 人的项目
- **Dependency Proxy**：CI 中频繁拉取 dockerhub 镜像的项目
- **镜像签名**：金融、医疗等有安全合规要求的行业

**不适用场景**：
- 临时原型/实验项目（不需要签名和标签策略）

### 注意事项

- **Tag cleanup policy 的执行是渐进的**：不是设置后立即清理，每天执行一次
- **Cosign keyless signing 依赖 OIDC**：需要 GitLab 作为 OIDC provider，确保时间同步
- **Dependency Proxy 的存储也需要清理**：定期清理不常用的缓存镜像

### 常见踩坑经验

1. **多阶段构建后镜像仍然很大**：在第二阶段用了 `FROM node:20` 而非 `FROM node:20-alpine`。根因：基础镜像本身大。解决：统一使用 alpine 版本的基础镜像。
2. **Tag cleanup 没生效**：设置了 cleanup policy 但标签还在。根因：cleanup 周期是每天一次，且匹配 `name_regex_keep` 的标签不会被删除。解决：检查 `name_regex_keep` 是否过于宽松，手动触发 cleanup。
3. **Dependency Proxy "Image not found"**：首次拉取时报错。根因：Dependency Proxy 需要 Docker 客户端先登录。解决：CI 中增加 `docker login $CI_DEPENDENCY_PROXY_SERVER` 步骤。

2. 镜像签名（Cosign）和镜像内容信任（Docker Content Trust）有什么区别？在生产环境中应该选择哪一个？

- **运维**：Dependency Proxy + Tag Cleanup Policy 可以大幅降低外网流量成本和存储成本
