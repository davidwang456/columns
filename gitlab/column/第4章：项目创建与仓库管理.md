# 第4章：项目创建与仓库管理

## 1. 项目背景

> **业务场景**：一家快速扩张的 SaaS 公司，半年内从 1 个代码仓库膨胀到 50+ 个仓库，项目权限混乱——实习生可以看到核心支付代码，离职员工的账号还保留着 Owner 权限，一个仓库的 CI Runner 被另一个项目盗用……CTO 说："我们需要重新梳理项目结构和权限体系。"

安全团队在一次例行审计中发现了几条触目惊心的问题：
- 某离职前端的个人账号在离职 3 个月后仍能通过 API Token 读取核心仓库代码
- 测试环境的数据库密码明文写在项目 CI 变量中，且该变量被设置在 Group 级别，导致该 Group 下的 15 个项目全部暴露
- 3 个不同团队使用相同的项目模板但各自维护不同的 CI/CD 流程，重复工作严重
- 没有项目归档机制，4 年前废弃的项目依然占用存储和 Runner 配额

**痛点放大**：GitLab 的项目管理远不止"建个仓库"那么简单。项目可见性设置、成员权限模型、Group 与 Subgroup 的组织结构、项目模板与导入导出、CI 变量的作用域——这些如果一开始不规划好，后续的安全审计、权限梳理和资源优化会让你痛不欲生。

## 2. 项目设计——剧本式交锋对话

**场景**：安全审计结果出来后，CTO 召集架构组讨论项目结构重构方案。

---

**小胖**："我一直觉得 GitLab 的权限很简单啊——要么私有要么公开，再加个 Maintainer 和 Developer 不就完了？"

**大师**："那你解释一下：Guest、Reporter、Developer、Maintainer、Owner 这 5 个角色有什么区别？Reporter 能看代码吗？Guest 能提 Issue 吗？"

**小胖**："呃……Guest 大概只能看？Reporter 能看代码？不对……"

**小白**："我来总结一下。Guest 只能看 Issue 和 MR，不能看代码；Reporter 可以 clone 代码但不能 push；Developer 可以 push 到非保护分支；Maintainer 可以管理保护分支和 CI 设置；Owner 可以删项目。"

**大师**："基本正确，少了一点——Guest 在私有项目中连 Issue 列表都看不到，只能看被明确邀请参与的 Issue。技术映射：GitLab 的权限模型是'最小可见 + 递增操作'，每提升一级，新增的不仅是写权限，还有更大的可见范围。"

**小胖**："那 Group 和 Subgroup 又是干嘛的？我们公司有前端组、后端组、数据组，是不是该每个组建一个 Group？"

**大师**："Group 是 GitLab 里最强大的组织单元。我们可以这样设计：公司顶层 Group → 部门 Subgroup → 团队 Subgroup。比如 `company/frontend/h5` 和 `company/backend/payment`。Group 级别的设置（CI 变量、Runner、权限）会自动继承到下属的所有项目。"

**小白**："继承？那如果我在 `company/backend` 设置了一个 CI 变量 `DB_PASSWORD`，`company/backend/payment` 和 `company/backend/order` 都能用？"

**大师**："对，这就是 CI 变量级联继承：Instance → Group → Subgroup → Project。但注意优先级——子级的同名变量会覆盖父级的。这就是为什么测试环境的数据库密码不应该放在顶层 Group——它会泄露给所有子项目。"

**小胖**："那我们公司 50 个仓库是不是太分散了？有没有办法统一管理 CI/CD？"

**大师**："这就用到项目模板和 CI/CD 模板了。如果你有 10 个 Node.js 项目，它们的 `.gitlab-ci.yml` 有 80% 是相同的——你可以提取一个 `ci-templates/nodejs.yml` 放在公共仓库，其他项目通过 `include` 引用。再配合 Group 级别的变量管理，一次配置，全部复用。"

---

## 3. 项目实战

### 环境准备

> **目标**：在 GitLab 上搭建一个企业级项目结构，包含 Group 层级、权限分配、项目模板和导入导出。

**前置条件**：已部署 GitLab CE 17.x（参考第2章），有管理员账号。

### 分步实现

#### 步骤1：创建企业级 Group 层级结构

**目标**：创建 "公司 → 业务线 → 团队" 的三层 Group 结构。

```bash
# 通过 GitLab API 批量创建 Group 结构
# 需要先创建 Personal Access Token（Settings → Access Tokens → 勾选 api 权限）

export GITLAB_URL="http://gitlab.local:8929"
export GITLAB_TOKEN="glpat-xxxx"

# 创建顶层 Group（公司）
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "name": "Acme Corp",
    "path": "acme-corp",
    "description": "公司顶层组织",
    "visibility": "private"
  }' \
  "$GITLAB_URL/api/v4/groups"

# 创建业务线 Subgroup
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "name": "电商平台",
    "path": "ecommerce",
    "description": "电商业务线",
    "parent_id": <acme-corp-group-id>,
    "visibility": "private"
  }' \
  "$GITLAB_URL/api/v4/groups"

# 创建团队 Subgroup
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "name": "订单服务",
    "path": "order-service",
    "description": "订单团队",
    "parent_id": <ecommerce-group-id>,
    "visibility": "private"
  }' \
  "$GITLAB_URL/api/v4/groups"
```

**最终 Group 结构**：

```
Acme Corp (顶层)
├── 电商平台 (Subgroup)
│   ├── 订单服务 (Subgroup)
│   ├── 商品服务 (Subgroup)
│   └── 支付服务 (Subgroup)
├── 基础架构 (Subgroup)
│   ├── CI 模板 (Subgroup)
│   └── 运维 (Subgroup)
└── 数据平台 (Subgroup)
```

#### 步骤2：配置成员与权限

**目标**：为不同 Group 层级配置角色权限，验证继承与隔离。

```bash
# 先创建测试用户（需要 API）
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "name": "Zhang San",
    "username": "zhangsan",
    "email": "zhangsan@acme.com",
    "password": "Test123456!",
    "skip_confirmation": true
  }' \
  "$GITLAB_URL/api/v4/users"

# 添加用户到 Group（不同角色）
# 张三：电商平台 - Developer（继承到所有子项目）
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --data "user_id=<zhangsan-id>&access_level=30" \
  "$GITLAB_URL/api/v4/groups/<ecommerce-group-id>/members"
# access_level: 10=Guest, 20=Reporter, 30=Developer, 40=Maintainer, 50=Owner

# 查看 Group 成员权限
curl --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  "$GITLAB_URL/api/v4/groups/<ecommerce-group-id>/members" | python3 -m json.tool
```

**权限验证实验**：

```bash
# 场景1：Guest 能否 clone 私有项目？
# 创建一个 Guest 用户，仅添加到项目 → 尝试 clone → 应该被拒绝（403）
# Guest 在私有项目中默认不可见代码

# 场景2：Developer 能否 push 到保护分支？
# Developer push 到 main → 应该被拒绝
# 但 push 到 feature 分支 → 应该成功

# 场景3：父 Group 的 Maintainer 能否修改子 Group 的设置？
# 父 Group Maintainer → 查看子 Group Settings → 可以看到部分设置但不能修改
# Owner 才能修改所有层级

# 验证 Group 变量继承
# 1. 在父 Group 设置变量：DB_HOST=prod.db.com
# 2. 在子 Group 设置变量：DB_HOST=staging.db.com
# 3. 子 Group 的项目中查看 CI 变量 → 应该是 staging.db.com（子 Group 覆盖）
```

#### 步骤3：配置项目模板与 CI/CD 模板继承

**目标**：创建项目模板仓库，让所有 Node.js 项目继承统一的 CI/CD 配置。

```bash
# 1. 在 Group 中创建 CI 模板仓库
# Group: acme-corp/infra/ci-templates
# Project: nodejs-template

# 2. 编写模板文件 .gitlab-ci-template.yml
cat > .gitlab-ci-template.yml << 'EOF'
# Node.js 项目 CI/CD 通用模板
# 其他项目通过 include 引用此文件

.node_cache: &node_cache
  cache:
    key: ${CI_COMMIT_REF_SLUG}
    paths:
      - node_modules/

.build: &build
  stage: build
  image: node:20-alpine
  <<: *node_cache
  before_script:
    - npm ci
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 week

.test: &test
  stage: test
  image: node:20-alpine
  <<: *node_cache
  before_script:
    - npm ci
  script:
    - npm run test
    - npm run lint
  coverage: /All files[^|]*\|[^|]*\s+([\d\.]+)/
  artifacts:
    reports:
      junit: junit.xml
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml

.deploy_staging: &deploy_staging
  stage: deploy
  image: alpine:latest
  script:
    - echo "Deploying to staging..."
    - echo "App URL: https://$CI_PROJECT_PATH_SLUG.staging.example.com"
  environment:
    name: staging
    url: https://$CI_PROJECT_PATH_SLUG.staging.example.com
  only:
    - main
EOF

# 3. 在具体项目中引用模板
# 在任意 Node.js 项目的 .gitlab-ci.yml 中：
cat > .gitlab-ci.yml << 'EOF'
# 引用 Group 级别的 CI 模板
include:
  - project: 'acme-corp/infra/ci-templates/nodejs-template'
    file: '.gitlab-ci-template.yml'

stages:
  - build
  - test
  - deploy

build:
  <<: *build

test:
  <<: *test

deploy_staging:
  <<: *deploy_staging
EOF
```

#### 步骤4：项目导入导出实战

**目标**：将一个 GitHub 仓库迁移到 GitLab，保留完整的 Issue 和 MR 历史。

```bash
# 方法A：通过 GitLab UI 导入
# Projects → New Project → Import project → GitHub
# 需要 GitHub Personal Access Token（repo 权限）

# 方法B：通过 API 导入
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --header "Content-Type: application/json" \
  --data '{
    "personal_access_token": "ghp_xxxx",
    "repo_id": 123456,
    "target_namespace": "acme-corp/ecommerce",
    "new_name": "imported-project"
  }' \
  "$GITLAB_URL/api/v4/import/github"

# 方法C：手动迁移（仅迁移代码）
# 1. 从 GitHub 克隆（mirror 模式保留所有分支和标签）
git clone --mirror https://github.com/user/old-repo.git
cd old-repo.git

# 2. 推送到 GitLab
git remote add gitlab http://gitlab.local:8929/acme-corp/ecommerce/new-repo.git
git push --mirror gitlab

# 3. 清理本地
cd .. && rm -rf old-repo.git

# 方法D：GitLab 项目导出导入（用于迁移或备份）
# 导出项目
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  "$GITLAB_URL/api/v4/projects/<project-id>/export"

# 等待导出完成后下载
curl --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --output project.tar.gz \
  "$GITLAB_URL/api/v4/projects/<project-id>/export/download"

# 导入到新 GitLab 实例
curl --request POST \
  --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
  --form "path=imported-project" \
  --form "file=@project.tar.gz" \
  "$GITLAB_URL/api/v4/projects/import"
```

## 4. 项目总结

### 优点 & 缺点

| 特性 | 优点 | 缺点 |
|------|------|------|
| Group/Subgroup | 层级清晰，权限继承，CI 变量统一管理 | 层级过多时路径变长（最多 20 级） |
| 权限模型 | 5 级权限粒度足够，满足企业级需求 | Guest 在私有项目的可见性过于严格 |
| 项目模板 | 一键生成标准化项目结构，减少重复工作 | 模板更新后不会自动同步到已建项目 |
| CI 模板引用 | 一处修改全局生效，版本可追踪 | YAML 锚点和 `include` 的调试比较困难 |
| 导入导出 | 支持 GitHub/Bitbucket/URL 等多种源 | 大项目（>10GB）导出可能超时 |

### 适用场景

- **多团队协作**：Group/Subgroup 结构天然适合大型组织的部门划分
- **微服务架构**：每个微服务一个 Project，通过 Group 共享 CI 配置
- **安全合规**：通过私有项目 + 精确权限控制代码访问范围
- **统一 CI/CD 治理**：通过父 Group 的 CI 变量控制基础设置

**不适用场景**：
- 个人开发者的单仓库项目（GitHub 免费版更合适）
- 完全扁平化的团队（不需要 Group 层级）

### 注意事项

- **Owner 权限慎给**：Owner 可以删除整个项目/Group，无法恢复（GitLab 17.x 支持延迟删除但默认 7 天后永久删除）
- **CI 变量继承可能泄露秘钥**：不要在父 Group 设置敏感的 production 变量——所有子项目可见
- **项目归档优于删除**：废弃的项目先 Archive（Settings → General → Archive），保留完整数据但设为只读
- **Fork 关系需维护**：Group 内项目的 Fork 关系会在原项目删除时断开

### 常见踩坑经验

1. **顶层 Group 被误删**：某 Owner 在清理 Group 时没注意勾选了子 Group，导致整个业务线的仓库全部删除。根因：GitLab 删除操作会级联删除所有子群组和项目。解决：启用延迟删除策略，设置 7 天确认期。
2. **Reporter 权限泄露代码**：以为 Reporter 不能看代码，结果发现 Reporter 可以 clone。根因：Reporter = Guest + Read Code。解决：对敏感代码，使用 Guest + 单独邀请可见 Issue/MR。
3. **CI 变量作用域混乱**：多个团队共享一个 Group，production 密钥被设在 Group 级别，导致测试环境的 Job 误读到生产数据库密码。根因：CI 变量设计时忽略继承传导。解决：敏感变量放在 Project 级别，或使用 Protected/Masked 属性限制。
