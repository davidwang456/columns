# 第15章：基础故障排查与调试技巧

## 1. 项目背景

算法团队每周二的例会上都有一个固定环节——"本周踩坑分享"。过去 3 个月里，团队踩过的典型问题包括：

- 新人 A 训练了一个图像分类模型，运行 2 小时后 loss 突然变成 NaN，GPU 显存占用 95%，排查了半天发现是学习率设成了 0.1 而不是 0.001
- 开发 B 用 `model.fit` 训练，第一个 epoch 跑了 10 分钟，第二个 epoch 跑了 8 分钟，但第三个 epoch 突然变成了 1 分钟——val_loss 也同时飙到了 1000+。原来是数据管道里 `repeat()` 放错了位置
- 工程师 C 在一个 4 卡 GPU 机器上训练，nvidia-smi 显示 GPU 利用率一直在 10%-30% 波动，但 CPU 却是 100%。后来发现是 `num_parallel_calls` 忘记设了，数据加载全在单线程上
- 算法 D 把训练好的模型交给后端部署，但 SavedModel 推理结果和训练时的 `model.predict` 结果相差 30%。原因——Dropout 层在导出时没有设 `training=False`

技术 Lead 老陈意识到：与其让大家各自踩坑、各自 google，不如系统化地整理一套"TensorFlow 故障排查 SOP"。本章模拟 5 个最常见的训练故障，从症状到根因到修复，形成团队的排查方法论。

**痛点放大**：TensorFlow 的训练故障通常表现为"loss NaN""loss 不降""OOM""速度慢""结果不一致"五种典型症状。但每种症状可能有 3-5 种不同的根因——缺乏系统化的排查方法论会导致反复试错，浪费大量 GPU 时间和人力。

## 2. 项目设计

**小胖**（瘫在椅子上）：我训练的模型 loss 突然变成 NaN 了！这已经是今天的第三次了！我改了学习率、换了优化器、加了梯度裁剪……全都不管用！

**大师**：别急。把问题拆开来看。Loss NaN 只有三种可能：(1) 数据里有 NaN/Inf；(2) 计算过程产生了 NaN（比如 log(0)、除以 0、梯度爆炸）；(3) 参数数值溢出。我们来逐个排查，而不是盲目改参数。

**大师**（拿出一张打印好的排查清单）：这是我们的调试 SOP，记住五个关键词：**看数据 → 看梯度 → 看数值 → 看形状 → 看速度**。

**技术映射**：TensorFlow 故障排查的核心方法论是"缩小范围"——从数据到计算到参数的逐层检查，每次排除一类可能性。

**小白**：具体怎么操作？我该先检查什么？

**大师**：第一步——**加断言，而不是盯着屏幕看 loss**。用 `tf.debugging.assert_all_finite` 在关键位置插入检查：

```python
# 在每个 epoch 结束后检查
tf.debugging.assert_all_finite(loss, "Loss contains NaN or Inf!")
tf.debugging.assert_all_finite(grads[0], "Gradient contains NaN!")
```

这比盯着终端看 loss 数字可靠得多。一旦发现 NaN，它会立刻抛异常并告诉你哪个变量出了问题。排查链是：先看数据里有没有 NaN（`tf.reduce_any(tf.math.is_nan(data))`），再看 forward 过程中哪个位置开始出现 NaN（逐层加断言），最后看梯度是否爆炸（`tf.norm(grad)` 是否超过 1000）。

**小白**：那 loss 一直在 0.69 附近震荡（二分类），死活不降是怎么回事？

**大师**：0.69 基本等于 `-log(0.5)`，意味着模型输出恒为 0.5——它在"随机猜"。这个问题排查链是：
1. **学习率对不对**？太大会震荡（loss 上下乱跳），太小会不动（loss 纹丝不动）。先用 `1e-3 → 1e-4 → 1e-5` 快速扫一遍。
2. **数据标签对不对**？检查 `np.unique(y_train)` 是不是只有 {0,1}，有没有全 0 或全 1。
3. **模型输出层对不对**？二分类最后一层应该是 `Dense(1, activation="sigmoid")`，loss 用 `binary_crossentropy`。如果错用了 `softmax` 或 `categorical_crossentropy`，loss 会混乱但不会报语法错误。
4. **数据有没有 shuffle**？如果没有 shuffle，每个 batch 可能全是同一个类别的样本，模型在一个 batch 内"暴力纠正"，下一个 batch 又推翻。

**技术映射**：loss=0.69 的二分类"不动"通常是因为模型输出退化为常数 0.5，根因依次排查：学习率 → 标签正确性 → 输出层/loss 匹配 → 数据顺序。

**小胖**：那 OOM（Out of Memory）呢？我的模型 GPU 显存 24GB 全用满了还是爆，但参数量总共才 500 万。

**大师**：显存并非只存模型参数——还有激活值（前向传播的中间结果）、梯度、优化器状态。一个参数在训练时大概占用：(1 份参数) + (1 份梯度) + (2 份 Adam 的 m/v 状态) + (N 份激活值) = 约 4+N 倍参数量的显存。激活值通常是大头——batch_size=64 的 ResNet 激活值可能占到 70% 的显存。

快速排查 OOM 的步骤：
1. `batch_size` 减半试试，如果 OOM 消失，说明瓶颈在激活值
2. 检查是否在 tape 外调用了不必要的 `model(x, training=True)` 产生额外中间结果
3. 检查 `tf.data` 的 `prefetch` 缓存了多少 batch（`prefetch(1)` vs `prefetch(10)`）
4. 如果必须用大 batch_size，使用梯度累积——每 4 个小 batch 累加梯度后统一更新

**技术映射**：显存 = 参数 + 梯度 + 优化器状态 + 激活值。激活值占比通常最大且与 batch_size 线性相关。梯度累积是"用时间换空间"——多步小 batch 累加梯度模拟大 batch 效果。

**小白**：训练速度特别慢怎么办？GPU 利用率才 15%。

**大师**：速度问题排查三问——(1) 瓶颈在 CPU 还是 GPU？(2) 瓶颈在数据加载还是计算？(3) 瓶颈在所有 step 还是某些 step？

```bash
# 快速诊断
nvidia-smi dmon -s u -d 1  # 看 GPU 利用率
top -p $(pgrep python)       # 看 CPU 使用率
```

如果 GPU 利用率低但 CPU 高——数据管道是瓶颈，加 `num_parallel_calls=AUTOTUNE`，检查 `prefetch` 和 `cache` 的位置。
如果 GPU 利用率高但训练慢——模型太大或 Keras 图模式下 Python 开销大，考虑 `@tf.function` 编译或 `jit_compile=True`。
如果某些 step 突然变慢——检查数据管道中是否有"慢样本"（超大图片或超长文本），导致个别 batch 拖累整体。

**技术映射**：训练速度优化的核心是定位瓶颈在"数据供给"还是"GPU 计算"。前者用 `prefetch/parallel_map/cache` 解决；后者用 `@tf.function/XLA/mixed_precision` 解决。

### 3.2 五种常见故障模拟与修复

**故障 1：Loss NaN 排查**

目标：定位 NaN 的来源（数据 → 前向 → 梯度 → 参数）。

```python
import tensorflow as tf
import numpy as np

# === 模拟故障：log(0) 导致的 NaN ===
def scenario_log_zero():
    """模型输出 log(0) 导致 NaN"""
    tf.random.set_seed(42)
    x = tf.constant([[0.1], [0.2], [-0.1]], dtype=tf.float32)

    # 错误做法：二分类 sigmoid + binary_crossentropy 自动防 log(0)
    # 但自定义 loss 如果没有 clip 就危险

    # 模拟一个输出 0 的情况
    y_pred = tf.constant([0.0, 0.5, 0.9], dtype=tf.float32)
    y_true = tf.constant([1.0, 0.0, 1.0], dtype=tf.float32)

    # 错误 loss（无 clip）
    loss_bad = -tf.reduce_mean(y_true * tf.math.log(y_pred) + (1-y_true) * tf.math.log(1-y_pred))
    print(f"无 clip loss: {loss_bad.numpy()}")  # NaN!

    # 修复：对预测值做个 clip
    eps = 1e-7
    y_pred_clipped = tf.clip_by_value(y_pred, eps, 1 - eps)
    loss_fixed = -tf.reduce_mean(y_true * tf.math.log(y_pred_clipped) + (1-y_true) * tf.math.log(1-y_pred_clipped))
    print(f"clip 后 loss: {loss_fixed.numpy():.4f}")

scenario_log_zero()
```

运行输出：
```
无 clip loss: nan
clip 后 loss: 3.9120
```

**故障 2：Loss 不下降（0.69 震荡）**

目标：检查学习率、标签、输出层匹配、shuffle。

```python
def scenario_loss_stuck():
    """模拟 loss 卡在 0.69 的场景排查"""
    np.random.seed(42)
    N = 1000
    X = np.random.randn(N, 10).astype(np.float32)
    y = np.random.randint(0, 2, N).astype(np.float32)

    # --- Bug 场景 A：学习率太大 ---
    model_a = tf.keras.Sequential([
        tf.keras.layers.Dense(32, activation="relu", input_shape=(10,)),
        tf.keras.layers.Dense(1, activation="sigmoid"),
    ])
    model_a.compile(optimizer=tf.keras.optimizers.Adam(1.0),  # ❌ lr=1.0 太大
                    loss="binary_crossentropy", metrics=["accuracy"])
    h_a = model_a.fit(X, y, epochs=5, batch_size=64, verbose=0)
    print(f"场景A (lr=1.0): loss {h_a.history['loss'][0]:.4f} → {h_a.history['loss'][-1]:.4f}")

    # --- Bug 场景 B：输出层用的 softmax ---
    model_b = tf.keras.Sequential([
        tf.keras.layers.Dense(32, activation="relu", input_shape=(10,)),
        tf.keras.layers.Dense(2, activation="softmax"),  # ❌ 二分类用了 softmax+2 类
    ])
    model_b.compile(optimizer="adam",
                    loss="sparse_categorical_crossentropy",
                    metrics=["accuracy"])
    h_b = model_b.fit(X, y.astype(int), epochs=5, batch_size=64, verbose=0)
    print(f"场景B (softmax+2): loss {h_b.history['loss'][0]:.4f} → {h_b.history['loss'][-1]:.4f}")

    # --- 正常场景 ---
    model_ok = tf.keras.Sequential([
        tf.keras.layers.Dense(32, activation="relu", input_shape=(10,)),
        tf.keras.layers.Dense(1, activation="sigmoid"),
    ])
    model_ok.compile(optimizer=tf.keras.optimizers.Adam(1e-3),
                     loss="binary_crossentropy", metrics=["accuracy"])
    h_ok = model_ok.fit(X, y, epochs=5, batch_size=64, verbose=0)
    print(f"场景OK (正常):   loss {h_ok.history['loss'][0]:.4f} → {h_ok.history['loss'][-1]:.4f}")

scenario_loss_stuck()
```

**故障 3：OOM 排查与梯度累积修复**

目标：定位显存瓶颈，用梯度累积模拟大 batch_size。

```python
def scenario_oom_fix():
    """模拟超大模型 OOM，用梯度累积解决"""
    import tensorflow as tf

    # 构造一个小型"大模型"示例（假设显存只够 batch_size=4）
    X = tf.random.normal([100, 256])
    y = tf.random.normal([100, 1])

    model = tf.keras.Sequential([
        tf.keras.layers.Dense(1024, activation="relu", input_shape=(256,)),
        tf.keras.layers.Dense(1024, activation="relu"),
        tf.keras.layers.Dense(1),
    ])
    optimizer = tf.keras.optimizers.Adam(1e-3)

    # === 梯度累积：每 4 个小 batch 累积后再更新 ===
    accumulation_steps = 4
    target_batch_size = 64   # 想要的大 batch
    micro_batch_size = target_batch_size // accumulation_steps  # 每次只处理 16 个

    ds = tf.data.Dataset.from_tensor_slices((X, y)).batch(micro_batch_size)

    for epoch in range(3):
        epoch_loss = 0
        for step, (xb, yb) in enumerate(ds):
            with tf.GradientTape() as tape:
                loss = tf.reduce_mean(tf.square(model(xb, training=True) - yb))
                loss = loss / accumulation_steps  # 除以累积步数

            grads = tape.gradient(loss, model.trainable_variables)
            if not hasattr(optimizer, "_accumulated_grads"):
                optimizer._accumulated_grads = [tf.zeros_like(v) for v in model.trainable_variables]

            for i in range(len(optimizer._accumulated_grads)):
                optimizer._accumulated_grads[i].assign_add(grads[i])

            if (step + 1) % accumulation_steps == 0:
                optimizer.apply_gradients(zip(optimizer._accumulated_grads, model.trainable_variables))
                optimizer._accumulated_grads = [tf.zeros_like(v) for v in model.trainable_variables]

            epoch_loss += loss.numpy() * accumulation_steps

        print(f"Epoch {epoch+1}: avg_loss = {epoch_loss / (step+1):.4f}")

scenario_oom_fix()
```

**故障 4：训练速度慢——数据管道瓶颈**

目标：对比有无 prefetch/parallel_map/cache 的性能差异。

```python
import time

def benchmark_pipeline():
    """对比不同 pipeline 配置的训练速度"""
    # 模拟数据加载（decode 很慢的图片）
    def slow_load(i):
        tf.py_function(lambda: time.sleep(0.01), [], tf.float32)  # 模拟 10ms 延迟
        return tf.ones([224, 224, 3], dtype=tf.float32), tf.constant(0)

    N = 500
    indices = tf.data.Dataset.range(N)

    # 配置 1: 无优化
    ds_slow = indices.map(lambda i: tf.py_function(
        lambda x: (tf.ones([224,224,3]), tf.constant(0)), [i],
        (tf.float32, tf.int32))).batch(32)

    # 配置 2: 加 prefetch + parallel_map
    ds_fast = indices.map(lambda i: tf.py_function(
        lambda x: (tf.ones([224,224,3]), tf.constant(0)), [i],
        (tf.float32, tf.int32)),
        num_parallel_calls=tf.data.AUTOTUNE).batch(32).prefetch(tf.data.AUTOTUNE)

    print("=== Pipeline 性能对比 ===")
    for name, ds in [("无优化", ds_slow), ("prefetch+parallel", ds_fast)]:
        start = time.time()
        count = 0
        for img, label in ds:
            count += 1
        elapsed = time.time() - start
        print(f"  {name:<20}: {count} batches in {elapsed:.1f}s ({count/elapsed:.1f} batch/s)")

benchmark_pipeline()
```

**故障 5：训练/推理结果不一致**

目标：定位 Dropout 导致的训练态 vs 推理态差异。

```python
def scenario_inference_mismatch():
    """验证 training=True/False 对 Dropout 输出的影响"""
    tf.random.set_seed(42)
    model = tf.keras.Sequential([
        tf.keras.layers.Dense(64, activation="relu", input_shape=(20,)),
        tf.keras.layers.Dropout(0.5),
        tf.keras.layers.Dense(1),
    ])

    fixed_input = tf.ones([1, 20])

    # 训练态：每次结果不同
    print("=== training=True (5次推理) ===")
    for i in range(5):
        out = model(fixed_input, training=True)
        print(f"  run {i+1}: {out.numpy().flatten()[0]:.6f}")

    # 推理态：每次结果一致
    print("\n=== training=False (5次推理) ===")
    for i in range(5):
        out = model(fixed_input, training=False)
        print(f"  run {i+1}: {out.numpy().flatten()[0]:.6f}")

    # model.predict() 默认 training=False
    print("\n=== model.predict (应等于 training=False) ===")
    p1 = model.predict(fixed_input, verbose=0)

    # 导出 SavedModel 后推理
    import tempfile, os
    path = os.path.join(tempfile.mkdtemp(), "test_dropout")
    tf.saved_model.save(model, path)
    loaded = tf.saved_model.load(path)
    p2 = loaded.signatures["serving_default"](fixed_input)
    p2_val = list(p2.values())[0].numpy().flatten()[0]
    print(f"model.predict:  {p1.flatten()[0]:.6f}")
    print(f"SavedModel:     {p2_val:.6f}")

scenario_inference_mismatch()
```

### 3.3 故障排查 SOP 速查表

| 症状 | 优先排查项 | 诊断命令 |
|------|-----------|----------|
| **Loss NaN** | 1.数据 NaN → 2.前向 NaN → 3.梯度爆炸 | `tf.debugging.assert_all_finite(loss)` |
| **Loss 不降** | 1.学习率 → 2.标签 → 3.输出层匹配 → 4.shuffle | `print(model.optimizer.lr.numpy())` |
| **OOM** | 1.batch_size → 2.激活值占比 → 3.gradient_tape 范围 | `nvidia-smi; print(model.count_params())` |
| **速度慢** | 1.数据管道 → 2.Python 开销 → 3.Kernel 效率 | `nvidia-smi dmon; tf.data.experimental.Stats` |
| **结果不一致** | 1.training 参数 → 2.Dropout/BN → 3.浮点精度 | `model(x, training=True/False)` |
| **Shape Mismatch** | 1.打印每层输出 shape → 2.检查输入 reshape | `model.summary(); print(tensor.shape)` |

### 3.4 最小可复现样例（MRE）模板

```python
# === TensorFlow 最小可复现样例模板 ===
import tensorflow as tf
import numpy as np

# 1. 固定随机种子
tf.random.set_seed(42); np.random.seed(42)

# 2. 最小化数据（10 条样本）
X = np.random.randn(10, 5).astype(np.float32)
y = np.random.randn(10, 1).astype(np.float32)

# 3. 最简模型
model = tf.keras.Sequential([
    tf.keras.layers.Dense(4, activation="relu", input_shape=(5,)),
    tf.keras.layers.Dense(1),
])
model.compile(optimizer="adam", loss="mse")

# 4. 单步训练/推理
model.fit(X, y, epochs=1, verbose=1)  # 如果这里就报错，说明问题在模型/数据本身
```

> **原则**：MRE 的三个"最小"——最少样本数（10-20）、最简模型（2-3 层）、最短训练（1 epoch）。如果 MRE 能复现，问题通常很快定位。

## 4. 项目总结

### 4.1 故障排查工具箱

| 工具 | 用途 | 用法 |
|------|------|------|
| `tf.debugging.assert_all_finite` | 检测 NaN/Inf | 插入到训练循环关键位置 |
| `model.summary()` | 查看每层输入输出 | 训练前必调 |
| `tf.print()` | 图模式下的调试打印 | 比 Python `print` 在图模式下可靠 |
| `run_eagerly=True` | 关闭图模式，逐行执行 | `model.compile(..., run_eagerly=True)` |
| `nvidia-smi dmon` | GPU 利用率实时监控 | 终端开第二个窗口持续观察 |
| `tf.config.list_physical_devices()` | 确认 GPU/CPU 可见 | 训练脚本开头打印 |
| `tf.data.Dataset.take(2)` | 检查数据管道输出 | 正式训练前验证 2 个 batch |

### 4.2 适用场景

1. **训练启动阶段**：验证数据管道 + 单步前向 + loss 计算无误
2. **训练中断排查**：Loss NaN / OOM / 速度骤降时的快速定位
3. **模型上线前**：训练 vs 推理一致性验证
4. **新成员 Onboarding**：提供 SOP 作为团队的知识库

**不适用场景**：
1. 极复杂的分布式训练故障（需第 37/39 章的性能剖析工具）
2. 底层 Runtime/CUDA 级别的 bug（需 GitHub Issues + 源码阅读）

### 4.3 注意事项

- **`tf.print` vs `print`**：`@tf.function` 或 `model.fit` 内部，Python `print` 只在 tracing 阶段执行一次。任何时候调试都用 `tf.print()`
- **`run_eagerly` 的性能代价**：开启后训练速度下降 10-30 倍，仅调试时使用，确认问题后关闭
- **`nvidia-smi` 的精度**：`nvidia-smi` 显示的 GPU 利用率是采样值（通常每 1-2 秒采样一次），短时尖峰可能捕捉不到。精细分析用 `TensorBoard Profiler`

### 4.4 常见踩坑经验

1. **坑**：`model.predict()` 返回的结果 batch 间不一致，但检查了 training=False。
   **根因**：自定义层中的随机逻辑没有用 `training` 参数控制。例如有人在 `call()` 里写了 `tf.random.uniform()`，但没有包裹在 `if training:` 中。
   **解决**：所有随机操作必须在 `if training:` 内，并确保 `call(self, inputs, training=None)` 参数正确传递。

2. **坑**：`model.fit` 卡住不动，但 GPU 和 CPU 利用率都很低。
   **根因**：数据管道死锁——`tf.data` 迭代器因为某些 `tf.py_function` 中的 Python 阻塞操作而卡住（如网络请求超时）。
   **解决**：先用 `.take(2).as_numpy_iterator()` 测试数据管道是否正常产出，再开训练；数据管道中使用 `timeout` 参数。

3. **坑**：`model.fit` 在 epoch 1 正常，epoch 2 突然报 `InvalidArgumentError: Incompatible shapes`。
   **根因**：数据管道中 `repeat()` 后的元素个数不是 `batch_size` 的整数倍，最后一个不完整的 batch 被丢弃（`drop_remainder=False`）后形状不一致。
   **解决**：`batch(batch_size, drop_remainder=True)` 确保所有 batch 形状一致。
