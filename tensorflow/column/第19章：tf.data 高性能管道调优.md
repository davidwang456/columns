# 第19章：tf.data 高性能管道调优

## 1. 项目背景

算法工程师阿伟负责一个"工业缺陷检测"图像分类项目——训练集有 50 万张 1024×1024 的高清 PCB 板图片，存储在远程 NFS 上（网络延迟约 5ms）。他用第 6 章学到的 tf.data 管道搭好了训练流程，但发现一个严重问题——单张 V100 GPU 利用率始终在 25% 上下徘徊，训练一个 epoch 要 3 小时。

阿伟做了 first check——`nvidia-smi dmon` 显示 GPU SM 利用率（计算核心占用率）只有 25%，但 `top` 显示 CPU 使用率高达 180%（Python 进程占了近 2 个核全力跑）。很明显，瓶颈在数据加载上——CPU 拼命读图解码，GPU 却在"等饭喂"。

他尝试加了 `num_parallel_calls=AUTOTUNE`，GPU 利用率从 25% 升到了 35%——有一点改善，但远不够。他又加了 `prefetch(AUTOTUNE)`，升到 40%。加了 `cache()`——内存炸了（50 万张 1024 图片不可能全 cache 进 32GB 内存）。最后他意识到：高分辨率图片的 JPEG 解码是 CPU 密集型操作，单靠 parallel_map 不够——需要把数据转为 TFRecord 格式（二进制存储 + 零解码开销），并且设计好分片策略。

**痛点放大**：当训练规模从"小 demo"（几千张图、本地 SSD）升级到"生产环境"（几十万张图、远程存储、高分辨率），tf.data 管道的默认配置会撞上多个性能瓶颈——I/O 带宽、CPU 解码、内存分配、数据倾斜。需要 caches/parallel/interleave/TFRecord 的组合拳来解决。

## 2. 项目设计

**小胖**（看着 nvidia-smi 上可怜的 25% 利用率）：这不科学啊！我明明加了 `prefetch`，也用了 `parallel_map`，GPU 为啥还在摸鱼？

**大师**：因为你只解决了一部分问题。数据管道有三个阶段可能成为瓶颈：

```
[磁盘/网络 I/O] → [CPU 解码/预处理] → [GPU 训练]
    阶段 1              阶段 2              阶段 3
```

你的 `parallel_map` 只加速了阶段 2（多线程解码），但阶段 1（从 NFS 读 1024×1024 的大文件）仍然是瓶颈——网络延迟 5ms + 文件 I/O 10ms = 每张图 15ms，单线程每秒只能读 66 张图。batch_size=64 的话，刚好够填 1 秒的 GPU 计算——一旦 GPU 快一点点，数据供应就跟不上了。

**技术映射**：管道性能瓶颈分三级——I/O 瓶颈（存储/网络带宽）→ CPU 瓶颈（解码/预处理计算）→ GPU 瓶颈（训练计算）。优化需要定位瓶颈阶段，对症下药。

**小白**：那怎么判断瓶颈在哪个阶段？

**大师**：用三个工具：
1. **`nvidia-smi dmon -s u`**：看 GPU SM 利用率。低 → GPU 在等数据（瓶颈在阶段 1/2）。
2. **`top -H`**：看 CPU 线程。如果单个 CPU 核 100% 而其他空闲 → 阶段 2 的单线程瓶颈。
3. **TensorBoard Profiler**（最精准）：能直接显示每个数据加载 step 的耗时分布——`IteratorGetNext` 耗时就是你等数据的时间。

**小胖**：那这些优化手段——`cache`、`interleave`、`TFRecord`——到底什么时候用哪个？

**大师**：来，我们画个决策树：

```
数据量 < 内存？
├─ 是 → cache() 把数据全缓存到内存，只读一次
│       （适合 < 10GB 的数据集）
└─ 否 → 数据在哪里？
        ├─ 本地 SSD → 用 TFRecord + parallel_map + prefetch
        └─ 远程存储 → 用 TFRecord + interleave(并行读取多个文件) + prefetch + cache(部分)
```

**大师**：`interleave` 是最容易被忽略的"大招"——它不是按顺序读文件，而是同时打开 N 个文件，交替从里面取数据。这样即使某个文件加载慢，其他文件的数据先到，不会 block 整个流水线。

```python
# 顺序读取：文件 1 → 文件 2 → 文件 3 （某一个慢就全堵）
ds = tf.data.Dataset.from_tensor_slices(file_paths)
ds = ds.map(read_file)

# 并行交错读取：同时打开 4 个文件，交错取数据
ds = tf.data.Dataset.from_tensor_slices(file_paths)
ds = ds.interleave(
    lambda path: tf.data.TFRecordDataset(path),
    cycle_length=4,            # 同时打开 4 个文件
    num_parallel_calls=AUTOTUNE,
)
```

**技术映射**：`interleave` 实现了"无阻塞文件交错读取"——多个文件并行打开，谁先读出数据谁先进入管道，不因单个慢文件阻塞整体。

**小白**：那 TFRecord 到底是什么？比普通 JPEG 好在哪里？

**大师**：TFRecord 是 TensorFlow 的二进制序列化格式，相当于"预烹饪的料理包"——JPEG 需要现场解码（CPU 密集型），TFRecord 里存储的是已解码的 raw bytes 或序列化的 `tf.train.Example`，读取时直接 bytes→tensor，省掉了解码开销。一个 TFRecord 文件通常存 1000-5000 条记录，配合分片（shard）和压缩（GZIP/Snappy），是大规模训练的标配。

### 3.2 分步实现

**步骤一：对比不同优化手段的性能提升**

目标：量化 cache/prefetch/interleave/parallel_map 对吞吐的提升效果。

```python
import tensorflow as tf
import numpy as np
import time
import tempfile
import os

# 创建 200 个模拟"大图片文件"（每个文件模拟 100ms 的读取延迟）
tmpdir = tempfile.mkdtemp()
n_files = 200
file_paths = []
for i in range(n_files):
    fpath = os.path.join(tmpdir, f"data_{i:04d}.npy")
    np.save(fpath, np.random.rand(224, 224, 3).astype(np.float32))
    file_paths.append(fpath)

AUTOTUNE = tf.data.AUTOTUNE

def load_cpu_heavy(path):
    """模拟 CPU 密集的读取+解码操作"""
    data = np.load(path.numpy().decode())
    # 模拟 5ms 预处理延迟
    time.sleep(0.005)
    return data.astype(np.float32), 0

def benchmark(pipeline_fn, name, n_batches=50):
    """测试管道吞吐"""
    ds = pipeline_fn()
    start = time.time()
    count = 0
    for img, label in ds.take(n_batches):
        count += 1
    elapsed = time.time() - start
    throughput = n_batches * 32 / elapsed
    print(f"  {name:<30}: {elapsed:.1f}s ({throughput:.0f} samples/s)")
    return throughput

print("=== Pipeline 性能对比 (batch_size=32) ===")

# 配置 1: 原始（无优化）
def baseline():
    return (tf.data.Dataset.from_tensor_slices(file_paths)
            .map(lambda p: tf.py_function(load_cpu_heavy, [p], (tf.float32, tf.int32)))
            .batch(32))

# 配置 2: + parallel_map
def with_parallel():
    return (tf.data.Dataset.from_tensor_slices(file_paths)
            .map(lambda p: tf.py_function(load_cpu_heavy, [p], (tf.float32, tf.int32)),
                 num_parallel_calls=AUTOTUNE)
            .batch(32))

# 配置 3: + parallel_map + prefetch
def with_prefetch():
    return (tf.data.Dataset.from_tensor_slices(file_paths)
            .map(lambda p: tf.py_function(load_cpu_heavy, [p], (tf.float32, tf.int32)),
                 num_parallel_calls=AUTOTUNE)
            .batch(32)
            .prefetch(AUTOTUNE))

# 配置 4: interleave 代替 map（适合多文件场景）
def with_interleave():
    def file_reader(path):
        return tf.data.Dataset.from_tensor_slices([path])
    return (tf.data.Dataset.from_tensor_slices(file_paths)
            .interleave(
                lambda p: tf.data.Dataset.from_tensor_slices([p]).map(
                    lambda x: tf.py_function(load_cpu_heavy, [x], (tf.float32, tf.int32)),
                    num_parallel_calls=1),
                cycle_length=4,
                num_parallel_calls=AUTOTUNE,
            )
            .batch(32)
            .prefetch(AUTOTUNE))

throughputs = {}
for fn, name in [(baseline, "无优化"), (with_parallel, "+ parallel_map"),
                 (with_prefetch, "+ prefetch"), (with_interleave, "+ interleave")]:
    throughputs[name] = benchmark(fn, name)

print(f"\n总提升: {throughputs['+ interleave'] / throughputs['无优化']:.1f}x")

# 清理
import shutil
shutil.rmtree(tmpdir, ignore_errors=True)
```

**步骤二：TFRecord 写入与读取**

目标：掌握 TFRecord 的生产和消费全流程。

```python
import tensorflow as tf
import numpy as np
import os
import tempfile

tmpdir = tempfile.mkdtemp()

# === 1. 写入 TFRecord ===
def serialize_example(features, label):
    """构建 tf.train.Example"""
    feature_dict = {
        "pixel_values": tf.train.Feature(
            float_list=tf.train.FloatList(value=features.flatten().tolist())),
        "label": tf.train.Feature(
            int64_list=tf.train.Int64List(value=[label])),
    }
    return tf.train.Example(features=tf.train.Features(feature=feature_dict)).SerializeToString()

# 写入 3 个分片（shard），每个 50 条记录
n_records = 150
n_shards = 3
records_per_shard = n_records // n_shards

for shard_id in range(n_shards):
    fname = os.path.join(tmpdir, f"data_shard_{shard_id:04d}.tfrecord")
    with tf.io.TFRecordWriter(fname) as writer:
        for i in range(records_per_shard):
            img = np.random.rand(32, 32, 3).astype(np.float32)
            label = np.random.randint(0, 10)
            writer.write(serialize_example(img, label))

print(f"已写入 {n_shards} 个 TFRecord 分片到 {tmpdir}")

# === 2. 读取 TFRecord ===
def parse_example(proto):
    """解析 TFRecord → (image, label)"""
    feature_desc = {
        "pixel_values": tf.io.FixedLenFeature([32 * 32 * 3], tf.float32),
        "label": tf.io.FixedLenFeature([], tf.int64),
    }
    parsed = tf.io.parse_single_example(proto, feature_desc)
    img = tf.reshape(parsed["pixel_values"], [32, 32, 3])
    return img, parsed["label"]

# 读取单个分片
file_pattern = os.path.join(tmpdir, "data_shard_*.tfrecord")
file_ds = tf.data.Dataset.list_files(file_pattern, shuffle=True)

# interleave 读取多个分片
ds = file_ds.interleave(
    lambda f: tf.data.TFRecordDataset(f).map(parse_example),
    cycle_length=n_shards,           # 同时打开 N 个分片
    num_parallel_calls=tf.data.AUTOTUNE,
)
ds = ds.batch(16).prefetch(tf.data.AUTOTUNE)

for batch_imgs, batch_labels in ds.take(2):
    print(f"Batch images shape: {batch_imgs.shape}")
    print(f"Batch labels: {batch_labels.numpy()}")
```

**步骤三：TFRecord 压缩与分片策略**

目标：对比无压缩/GZIP/Snappy 的文件大小和读取速度。

```python
import tensorflow as tf
import numpy as np
import time
import os
import tempfile

tmpdir = tempfile.mkdtemp()

def write_tfrecords_with_compression(compression_type=None):
    fname = os.path.join(tmpdir, f"compressed_{compression_type or 'none'}.tfrecord")
    options = tf.io.TFRecordOptions(compression_type=compression_type) if compression_type else None
    with tf.io.TFRecordWriter(fname, options=options) as writer:
        for _ in range(1000):
            img = np.random.rand(64, 64, 3).astype(np.float32).flatten()
            writer.write(
                tf.train.Example(features=tf.train.Features(feature={
                    "pixels": tf.train.Feature(float_list=tf.train.FloatList(value=img.tolist())),
                })).SerializeToString()
            )

    file_size = os.path.getsize(fname)
    # 测读取时间
    start = time.time()
    for _ in tf.data.TFRecordDataset(fname, compression_type=compression_type or "").take(500):
        pass
    read_time = time.time() - start
    return file_size, read_time

print("=== TFRecord 压缩方案对比 ===")
print(f"{'压缩':<12} {'文件大小':>10} {'读取时间':>10}")
print("-" * 34)
for comp in [None, "GZIP", "ZLIB"]:
    size, read_time = write_tfrecords_with_compression(comp)
    print(f"{str(comp):<12} {size/1024:>8.0f} KB {read_time:>8.3f}s")

import shutil
shutil.rmtree(tmpdir, ignore_errors=True)
```

### 3.3 性能优化速查表

| 优化手段 | 适用场景 | 效果 | 代价 |
|----------|----------|------|------|
| `num_parallel_calls=AUTOTUNE` | 所有 `map` 操作 | CPU 利用率提升 2-4x | 增加内存占用 |
| `prefetch(AUTOTUNE)` | 所有管道 | 隐藏 GPU 等待时间 | 增加少量内存 |
| `cache()` | 数据量 < 内存 | 第一个 epoch 后 I/O 零开销 | 内存/磁盘占用 |
| `interleave` | 多文件读取 | 避免单文件 I/O 阻塞 | 文件句柄数增加 |
| TFRecord | 数据量 > 10GB | 减少解码开销 30-50% | 需预处理转换 |
| `batch` 的 `drop_remainder=True` | 确保 batch 形状一致 | 避免动态形状导致 retracing | 丢弃不足一个 batch 的数据 |
| 分片 (shard) | 数据量 > 100GB | 并行读取、分布式分发 | 增加管理复杂度 |

## 4. 项目总结

### 4.1 管道优化决策

| 瓶颈类型 | 诊断信号 | 优化手段 |
|----------|----------|----------|
| I/O 瓶颈 | GPU 利用率低 + `iostat` 磁盘利用率高 | TFRecord + interleave + SSD 本地缓存 |
| CPU 解码瓶颈 | GPU 利用率低 + CPU 单核 100% | `parallel_map` + `AUTOTUNE` |
| 内存瓶颈 | OOM 或 swap 频繁 | 减小 `prefetch` buffer、分片读取 |
| 数据倾斜 | 某些 step 突然变慢 | 检查数据分布，对慢文件单独优化 |

### 4.2 适用场景

1. **大规模图像数据集**（> 10 万张）→ TFRecord + interleave + prefetch
2. **远程存储训练**（NFS/HDFS/S3）→ 本地 SSD 缓存 + 预取策略
3. **分布式训练** → 每个 worker 读取不同的 TFRecord 分片，避免重复读取
4. **多模态数据** → 不同模态用不同的 TFRecord 文件，通过 interleave 并行加载

**不适用场景**：
1. 小数据集（< 1000 条）——直接用 `from_tensor_slices` 最简单，过度优化反而不值
2. 数据是实时生成的（如在线学习）——用 `from_generator` + `py_function`

### 4.3 注意事项

- **`cache()` 的位置**必须放在 `map`（解码）之后、`shuffle` 之前——缓存解码后的数据，shuffle 的内存开销很小
- **`AUTOTUNE` 不是万能的**：它根据 CPU 核数自动分配并行度，但不会替你判断瓶颈在哪。如果瓶颈在远程 I/O，`AUTOTUNE` 的并行只会让网络更挤
- **TFRecord 的 schema 演进**：如果后续要加新特征，TFRecord 的 proto 定义需要向后兼容（新增 optional 字段），否则旧数据全部不可读
- **压缩格式选择**：GZIP 压缩率高但读取慢（单线程解压），Snappy 压缩率低但读取快——实时训练用 Snappy，归档存储用 GZIP

### 4.4 常见踩坑经验

1. **坑**：加了 `cache()` 后第一个 epoch 正常，第二个 epoch OOM。
   **根因**：`cache()` 把所有解码后的数据全存进内存/磁盘。数据量 > 可用内存时，cache 写磁盘变得极慢且可能爆盘。
   **解决**：用 `cache(filename_local)` 指定磁盘路径；或用局部 cache——只 cache 常用数据的子集。

2. **坑**：`interleave(cycle_length=4)` 后 batch 里的数据顺序完全随机了。
   **根因**：`interleave` 的默认行为是"哪个文件先读完就从哪个取"，天然具有随机性。如果 pipeline 后又加了 `shuffle`，不会出错。但如果没加 `shuffle` 且需要保持某种顺序（如时序数据），会导致问题。
   **解决**：对时序数据用 `interleave(..., deterministic=True)` 强制确定顺序。

3. **坑**：TFRecord 文件写到一半中断，训练读到损坏文件时报 `DataLossError`。
   **根因**：TFRecord 文件末尾不完整时，读取器无法判断是文件结束还是数据损坏。
   **解决**：写入时先写临时文件，写完做完整性校验，校验通过后 `rename` 到正式文件名（原子的）。

### 4.6 推广计划提示

- **新人开发**：默认在所有 `map` 和 `prefetch` 加 `AUTOTUNE`，这是零成本的性能提升
- **算法工程师**：数据管道是模型训练的第一道门槛，建议为每个数据集写专门的 `data_pipeline.py`，在训练前做一次"管道吞吐 benchmark"
- **平台工程师**：在训练平台的标准化 Docker 镜像中预装 `libsnappy`（Snappy 压缩库），并将常用数据集预转为 TFRecord 格式放到共享存储
