# 第30章：安全、合规与模型治理

## 1. 项目背景

某金融科技公司的风控模型刚刚通过了监管审查——但过程极其痛苦。合规部门要求提供：(1) 训练数据的来源和授权证明；(2) 模型从哪些数据源提取了特征；(3) 上线前的质量验证报告；(4) 如果模型出错，回滚方案是什么。算法团队花了整整两周翻 Git 历史和 Slack 聊天记录来拼凑这些材料。

更糟糕的是，安全团队在一次例行的渗透测试中发现：模型文件（SavedModel）可以被任意下载，推理 API 没有做输入校验——测试人员发了一个 10MB 的 JSON body，Serving 进程直接 OOM 了。还有一次，一个离职员工的账号仍然能访问训练数据服务器——他在离职后还登录了一次。

合规总监拍了桌子："模型上线不是技术问题——是治理问题。我们需要数据权限、模型审批、灰度发布、审计日志、回滚预案——这些不是训完模型顺手做的，是需要系统化设计的。"

**痛点放大**：机器学习系统的安全和治理问题常被忽略——(1) 数据隐私和脱敏不充分，训练数据可能含个人身份信息（PII）；(2) 模型文件缺少访问控制和版本管理，权重泄露 = 商业模式泄露；(3) 对抗样本和投毒攻击可能导致模型做出危险决策；(4) 审计和合规要求模型全生命周期可追溯，缺乏 Metadata 记录 = 无法提交监管审查。

## 2. 项目设计

**小胖**（不以为然地抽了一口气）：安全和合规？那都是运维和法务的事吧？我们模型团队只管把准确率训上去就行了。

**大师**：那我问你——你训练风控模型用的用户数据里，有没有身份证号、手机号、银行卡号？你保存的 `.h5` 模型文件，谁都能从共享文件夹下载——万一竞争对手拿到了，直接用你的模型做反欺诈，你的技术壁垒在哪？如果明天监管机构来问你要"模型上线审批记录"，你有吗？

**小胖**（声音越来越小）……数据是脱敏过的……吧？模型文件确实每个人都能访问……

**大师**："吧"——问题就出在这个"吧"上面。安全和治理不是"事后补材料"，而是从数据第一天开始就要嵌入流程中的。

**技术映射**：ML 安全治理 = 数据安全（脱敏+权限） + 模型安全（文件保护+反序列化防御） + 流程治理（审批+灰度+回滚+审计）。三者缺一不可。

**小白**：数据脱敏具体怎么做？训练数据里的姓名和身份证不能直接删——有些情况下"年龄段"和"地域"对风控模型很重要。

**大师**：脱敏不是"全删"，而是"去标识化（De-identification）"。几种常用手段：

- **直接删除**：姓名、电话号码、精确地址 → 直接删
- **泛化**：年龄 36 → "30-40"；收入 18500 → "15000-20000"；地址精确到门牌号 → 只保留城市
- **哈希/Token化**：用户 ID 用 HMAC 加盐哈希——同一个人每次的哈希值一样，但不可逆推原始 ID
- **差分隐私**：在训练数据梯度中添加噪声——即使别人拿到了模型权重，也无法反推训练数据中的个体信息

```python
# 简单脱敏示例
def pseudonymize(user_id, secret_key):
    import hashlib, hmac
    return hmac.new(secret_key.encode(), user_id.encode(), hashlib.sha256).hexdigest()[:16]

# user_123 → pseudonymize("user_123", "SECRET") → "a1b2c3d4e5f6..."
```

**技术映射**：数据脱敏的目标是在保护个人隐私的前提下保留数据的统计价值。泛化和哈希是不可逆手段；差分隐私是可证明的数学隐私保障。

**小白**：模型文件安全呢？SavedModel 不就是一个文件目录吗？怎么保护？

**大师**：SavedModel 包含了完整的模型架构和训练好的权重——这可能是公司最有价值的 IP。安全措施分三层：

1. **存储层**：模型文件放在 ACL 控制的私密存储（如 S3 + IAM 或 GCS + ServiceAccount），只有 Serving 服务账号有读取权限
2. **传输层**：模型下载通过 TLS 加密 + 签名校验（`saved_model_cli` 支持 `--signature` 校验）
3. **运行时**：TensorFlow Serving 可以配置请求大小限制、速率限制、超时——防御 DoS 攻击

```bash
# Serving 的输入防护配置
tensorflow_model_server \
  --rest_api_port=8501 \
  --grpc_max_message_length=10485760 \  # 限制 10MB
  --tensorflow_session_parallelism=8 \   # CPU 核限制
  --file_system_poll_wait_seconds=5
```

**技术映射**：模型安全的三层防御——存储层 ACL、传输层 TLS+签名、运行时输入校验+速率限制。

**小白**：对抗样本攻击呢？给模型投喂恶意构造的输入？

**大师**：对抗样本是模型安全的"终极 boss"。比如：一张正常的猫图片，人眼看是人类只看是猫，但添加了人眼不可见的微小噪声后，模型以 99% 的置信度判定为"狗"。在风控场景——攻击者可能通过逐步调整交易金额和频率，找到模型的"盲区"，从而绕过检测。

防御手段：
- **对抗训练（Adversarial Training）**：在训练时加入对抗样本，增强鲁棒性
- **输入平滑**：对输入做轻微的高斯模糊/降噪，减少对抗噪声的影响
- **集成模型**：多个不同结构的模型投票，单个模型被攻破不影响整体结果

**小胖**：模型卡和数据卡又是什么？

**大师**：模型卡（Model Card）是一份"模型说明书"——记录模型的基本信息、训练数据、性能指标、使用限制、道德审查。数据卡（Data Card）类似，但针对数据集。Google 几年前就推广这个概念，现在逐渐成为行业标准。它们解决的就是"模型从哪来、用了什么数据、有什么局限"的问题——监管机构问你时，直接甩一份 Model Card 就行。

**步骤一：训练数据脱敏与权限控制**

目标：实现用户 ID 的哈希匿名化和数据访问审计。

```python
import hashlib
import hmac
import numpy as np
import pandas as pd
import logging

class DataAnonymizer:
    """训练数据脱敏工具"""

    def __init__(self, secret_key=None):
        self.secret_key = secret_key or "CHANGE_ME_IN_PRODUCTION"
        self.audit_log = []

    def pseudonymize(self, identifier):
        """不可逆伪匿名化（加盐哈希）"""
        hashed = hmac.new(
            self.secret_key.encode(),
            str(identifier).encode(),
            hashlib.sha256
        ).hexdigest()[:16]
        self.audit_log.append(f"anonymized: id={identifier[:4]}... → {hashed}")
        return hashed

    def generalize_age(self, age, bucket_size=5):
        """年龄泛化到区间"""
        bucket = (age // bucket_size) * bucket_size
        return f"{bucket}-{bucket+bucket_size-1}"

    def mask_card_number(self, card_num):
        """银行卡号掩码"""
        s = str(card_num)
        return s[:4] + "****" + s[-4:]  # 6222****1234

    def report(self):
        """审计摘要"""
        return {
            "total_anonymized": len(self.audit_log),
            "operations": self.audit_log[-5:],  # 最近 5 条
        }

# 示例
anonymizer = DataAnonymizer()
print("原始用户 ID: user_8888_zhangsan")
print(f"脱敏后:      {anonymizer.pseudonymize('user_8888_zhangsan')}")
print(f"年龄 37 → {anonymizer.generalize_age(37)}")
print(f"卡号 6222021234567890 → {anonymizer.mask_card_number('6222021234567890')}")
```

**步骤二：模型文件签名与校验**

目标：模型保存时自动加哈希签名，加载时验证完整性。

```python
import tensorflow as tf
from tensorflow import keras
import hashlib
import json
import os
import tempfile

class SignedModelManager:
    """模型签名管理——防止模型文件被篡改"""

    @staticmethod
    def save_with_signature(model, save_path, metadata=None):
        """保存模型并生成签名文件"""
        os.makedirs(save_path, exist_ok=True)

        # 保存模型
        model.save(os.path.join(save_path, "model.keras"))

        # 计算哈希
        model_bytes = open(os.path.join(save_path, "model.keras"), "rb").read()
        sha256_hash = hashlib.sha256(model_bytes).hexdigest()

        # 生成签名文件
        signature = {
            "model_hash": sha256_hash,
            "timestamp": str(pd.Timestamp.now()),
            "tf_version": tf.__version__,
            "metadata": metadata or {},
        }
        with open(os.path.join(save_path, "signature.json"), "w") as f:
            json.dump(signature, f, indent=2)

        return sha256_hash

    @staticmethod
    def load_with_verification(save_path):
        """加载模型并验证签名"""
        sig_path = os.path.join(save_path, "signature.json")
        model_path = os.path.join(save_path, "model.keras")

        if not os.path.exists(sig_path):
            raise SecurityError(f"签名文件缺失: {sig_path}")

        with open(sig_path) as f:
            sig = json.load(f)

        # 验证哈希
        model_bytes = open(model_path, "rb").read()
        current_hash = hashlib.sha256(model_bytes).hexdigest()

        if current_hash != sig["model_hash"]:
            raise SecurityError(
                f"模型完整性校验失败!\n"
                f"  预期: {sig['model_hash']}\n"
                f"  实际: {current_hash}"
            )

        model = keras.models.load_model(model_path)
        return model, sig

class SecurityError(Exception):
    pass

# 演示
import pandas as pd

tmpdir = tempfile.mkdtemp()
model = keras.Sequential([
    keras.layers.Dense(8, activation="relu", input_shape=(4,)),
    keras.layers.Dense(1, activation="sigmoid"),
])

hash_val = SignedModelManager.save_with_signature(
    model, os.path.join(tmpdir, "secure_model"),
    metadata={"author": "algo_team", "version": "1.2.3"}
)
print(f"模型已保存，签名哈希: {hash_val[:16]}...")

loaded_model, sig_info = SignedModelManager.load_with_verification(
    os.path.join(tmpdir, "secure_model")
)
print(f"模型加载成功 ✅")
print(f"签名信息: v{sig_info['metadata']['version']} by {sig_info['metadata']['author']}")

import shutil
shutil.rmtree(tmpdir, ignore_errors=True)
```

**步骤三：输入安全校验（防御恶意请求）**

目标：在推理 API 前增加输入防护层。

```python
class InferenceGuard:
    """推理请求安全防护"""

    def __init__(self, max_input_size_mb=5, input_dim=15, input_dtype="float32"):
        self.max_input_size_mb = max_input_size_mb
        self.input_dim = input_dim
        self.input_dtype = input_dtype

    def validate(self, instances):
        """验证推理请求的合法性和安全性"""

        # 1. 大小检查（防御 DoS）
        import sys
        size_mb = sys.getsizeof(instances) / 1024 / 1024
        if size_mb > self.max_input_size_mb:
            return False, f"请求过大: {size_mb:.1f}MB > {self.max_input_size_mb}MB"

        # 2. 类型检查
        if not isinstance(instances, list):
            return False, "instances 必须是 list"

        # 3. 维度检查
        for i, sample in enumerate(instances):
            if not isinstance(sample, list):
                return False, f"instances[{i}] 必须是 list"
            if len(sample) != self.input_dim:
                return False, f"instances[{i}] 维度 {len(sample)} != {self.input_dim}"

        # 4. 数值安全检查
        import numpy as np
        data = np.array(instances, dtype=np.float32)
        if np.any(np.isnan(data)) or np.any(np.isinf(data)):
            return False, "输入含 NaN 或 Inf"

        # 5. 范围检查（防御异常大/小的值）
        if np.max(np.abs(data)) > 1e8:
            return False, f"输入值过大: max_abs={np.max(np.abs(data)):.1e}"

        return True, "OK"

    def log_anomaly(self, client_ip, reason):
        """记录异常请求（用于后续审计和封禁）"""
        print(f"[SECURITY] 异常请求 from {client_ip}: {reason}")
        # 实际系统中写入数据库或安全日志

# 测试
guard = InferenceGuard(max_input_size_mb=5, input_dim=15)

test_cases = [
    ([[1.0]*15], "正常请求"),
    ([[1.0]*10], "维度错误"),
    ([[float("nan")]*15], "NaN 输入"),
    ([list(range(1000))]*10, "异常大请求"),
]

for data, desc in test_cases:
    ok, msg = guard.validate(data)
    print(f"  {desc:<12}: {'PASS' if ok else 'REJECT'} — {msg}")
```

### 3.3 模型安全治理 Checklist

| # | 检查项 | 责任人 | 完成标准 |
|---|--------|--------|----------|
| 1 | 训练数据已脱敏（PII 去标识化） | 算法工程师 | 数据传输前完成 |
| 2 | 训练数据有访问权限控制 | 安全团队 | 仅授权账号可读 |
| 3 | 模型文件有签名校验 | 平台工程师 | `load_with_verification` 通过 |
| 4 | 推理 API 有输入校验 | 算法/后端 | 异常输入返回 400 |
| 5 | 模型卡已填写并归档 | 算法工程师 | 包含 6 要素（见下） |
| 6 | 灰度发布流程已审批 | Tech Lead | 审批单 + 灰度计划 |
| 7 | 回滚预案已测试 | 运维工程师 | 3 分钟内可回滚 |
| 8 | 审计日志开启（训练+推理） | 平台工程师 | 日志保留 ≥ 90 天 |

**Model Card 6 要素**：(1) 模型基本信息（名称/版本/训练日期）；(2) 训练数据说明（来源/规模/代表性问题）；(3) 性能指标（全量 + 关键分片）；(4) 使用限制（什么场景下不能用）；(5) 公平性审查（对不同群体的效果差异）；(6) 道德/隐私审查结果。

## 4. 项目总结

### 4.1 安全措施分级

| 级别 | 措施 | 防御什么 |
|------|------|---------|
| **基础 (必须)** | 数据脱敏、模型 ACL、输入校验 | PII 泄露、未授权访问、服务崩溃 |
| **标准 (推荐)** | 模型签名、数据/模型权限审批、灰度发布 | 模型篡改、权限滥用、上线事故 |
| **增强 (高风险)** | 对抗训练、差分隐私、模型水印、联邦学习 | 对抗攻击、隐私反推、IP 窃取 |

### 4.2 适用场景

1. **金融风控**：强监管行业，需要完整的审批→灰度→回滚流程 + 数据脱敏
2. **医疗诊断**：患者隐私保护（HIPAA/GDPR）+ 模型决策的可解释审计
3. **内容推荐**：用户行为数据脱敏 + 模型公平性审查（避免对某群体歧视）
4. **自动驾驶**：模型安全性验证（对抗样本防御）+ 事故追溯的审计日志

**不适用场景**：
1. 纯内部使用的实验模型——完整治理流程的成本可能高于其价值
2. 开源研究模型——研究阶段的模型不需要生产级治理，但 Model Card 仍推荐

### 4.3 注意事项

- **数据脱敏与模型效用的平衡**：过度脱敏（如所有数值泛化到大区间）→ 特征区分度丧失 → 模型性能暴跌
- **模型签名的密钥管理**：用于 HMAC/数字签名的密钥不能硬编码在代码中——需用密钥管理服务（如 AWS KMS/HashiCorp Vault）
- **灰度发布的时间窗口**：灰度观察时间太短（< 30 分钟）→ 流量不够，指标信噪比低；太长 → 影响迭代速度。建议：10% 灰度 1 小时 → 50% 灰度 2 小时 → 全量
- **回滚不是终点**：每次回滚后必须做复盘（Post-mortem），更新 Model Card 中的已知问题、测试用例和监控阈值，防止同类问题再次流出

### 4.4 常见踩坑经验

1. **坑**：脱敏后的用户 ID（加盐哈希）做 groupby 聚合——不同日期的哈希值不同（因为盐变了）。
   **根因**：用了随机盐而非固定密钥——同一用户不同时间的哈希值不一致，聚合失效。
   **解决**：用 HMAC + 固定密钥（而不是随机盐），确保同一用户在任何时间的伪匿名化结果一致。

2. **坑**：模型签名校验在生产环境频繁失败——但模型文件没被改。
   **根因**：Docker 镜像更新了 TensorFlow 版本，`model.save()` 的输出格式有微调（protobuf 序列化顺序变化），导致哈希变化。
   **解决**：模型签名只校验模型权重文件（`variables/` 下的 checkpoint），忽略图文件的微小变化；或固定 TensorFlow 版本。

3. **坑**：灰度发布期间，新模型的业务指标（CTR）明显提升——但全量后反而下跌。
   **根因**：灰度流量不是随机的——灰度用户是新注册的活跃用户（天然 CTR 高），不能代表全量用户。
   **解决**：灰度流量必须用随机分流（如 `hash(user_id) % 100 < 10`），确保灰度组和全量组的用户分布一致。

### 4.5 思考题

1. 你需要在保护用户隐私的前提下，将一家银行的交易数据共享给第三方建模团队。请设计一套基于差分隐私的数据共享方案，并估算 ε（隐私预算）的合理取值及其对模型 AUC 的影响。

2. 你的模型被人恶意复制（逆向工程）并部署到了竞品服务上。除了法律手段外，你能在模型训练时做哪些"水印"处理，使得你可以证明这个模型是从你的模型复制而来的？请设计至少两种模型水印方案。

### 4.6 推广计划提示

- **算法工程师**：每个模型上线前必须填写 Model Card（用模板，10 分钟完成），这是团队的硬性流程
- **安全团队**：每季度对模型服务做一次渗透测试（输入校验 bypass、权限越权、模型文件访问）
- **合规/法务团队**：建立"模型风险分级"机制——高风险模型（涉及金融决策/健康/法律）需要完整审批链，低风险模型简化流程
