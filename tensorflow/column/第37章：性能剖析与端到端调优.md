# 第37章：性能剖析与端到端调优

## 1. 项目背景

算法工程师阿磊的推荐模型训练速度慢得令人发指——单 epoch 需要 4 小时，数据集才 100 万条。他已加满了"标准优化套餐"：`@tf.function`（第 33 章）、混合精度（第 35 章）、XLA（第 36 章）——但训练速度只提升了 40%，还是需要 2.4 小时/epoch。

团队 Leader 老陈让他用 TensorBoard Profiler 做一次全链路剖析。Profiler 的结果让阿磊大吃一惊：

- **Trace Viewer** 显示：每个训练 step 中，`IteratorGetNext`（数据加载）耗时占比 45%——原来数据管道里有一个 `tf.py_function` 在做文本正则化，这个 Python 调用没法被图模式加速
- **Kernel Stats** 显示：GPU Kernel 执行时间只占了 35%，大量时间花在了"GPU→CPU 拷贝"上——原来代码里每隔 50 个 step 调用一次 `tensor.numpy()` 来记录日志，导致了隐式跨设备拷贝
- **Memory Profile** 显示：显存使用呈现"锯齿状"——每 100 步显存猛增 2GB 然后释放，循环往复。这个周期性的峰值对应着 `model.save_weights()` 的调用

**痛点放大**：当"标准优化套餐"都用了但性能还不达标时，你需要系统化的性能剖析（Profiling）——不是猜瓶颈在哪，而是用工具精确度量每个环节的耗时，找到最大的优化杠杆。

## 2. 项目设计

**小胖**（瘫在椅子上）：我都加了 `@tf.function`、开了混合精度、启用了 XLA——还慢？这三板斧不是号称"通用加速"吗？还要怎么优化？

**大师**："通用加速"是指它们通常能带来提升——但提升多少取决于你的瓶颈在哪。如果你的训练瓶颈是数据加载（CPU 忙、GPU 在等），`@tf.function` 和 XLA 优化的是 GPU 计算——它们根本碰不到真正的瓶颈。你需要一个"诊断工具"来告诉你：时间到底花在哪了？

**技术映射**：性能优化三步法——(1) Profile（找到瓶颈在哪）；(2) Optimize（针对瓶颈做优化）；(3) Verify（验证优化效果并检测有无引入副作用）。

**小白**：TensorBoard Profiler 具体怎么用？它和 `nvidia-smi` 有什么不同？

**大师**：`nvidia-smi` 是"宏观仪表盘"——告诉你 GPU 利用率是 25% 还是 95%。Profiler 是"显微镜"——告诉你 GPU 时间花在了哪个具体的 Kernel 上，CPU 时间花在了哪个 Python 函数上，数据加载每步花了多少毫秒。

TensorBoard Profiler 启动方式：

```python
# 方法 1: 训练脚本中加 Callback
callbacks = [
    tf.keras.callbacks.TensorBoard(
        log_dir=log_dir,
        profile_batch='500,520',  # 只在 batch 500-520 之间 profile
    ),
]

# 方法 2: 手动录制
tf.profiler.experimental.start(log_dir)
# ... 训练几步 ...
tf.profiler.experimental.stop()
```

然后在 TensorBoard 的 PROFILE 标签页查看。

**小白**：那 Profiler 提供了哪些分析视图？

**大师**：四个核心视图：

| 视图 | 回答什么问题 | 怎么看 |
|------|------------|--------|
| **Overview Page** | 整体瓶颈在 Input / Compute / Output 哪一步？ | Step-time 饼图：红色（Input）→ 数据瓶颈，蓝色（Compute）→ 计算瓶颈 |
| **Trace Viewer** | 每个 Op 和 Kernel 的精确时间线 | 横向时间轴，找"空隙"和"长条" |
| **Kernel Stats** | 哪些 GPU Kernel 耗时最多 | 按耗时排序的 Kernel Top-K |
| **Memory Profile** | 显存在哪些时间点被什么操作分配/释放 | 时间-显存曲线，找"尖峰"和"锯齿" |

**技术映射**：Overview Page 是宏观定位（Input vs Compute vs Overhead）；Trace Viewer 是微观时间线（每个 Op/Kernel 的开始-结束时间）；Kernel Stats 是 Kernel 级聚合（哪种 Kernel 最耗时）；Memory Profile 是显存 timeline（分配/释放的时间模式）。

**小胖**：找到瓶颈后怎么优化？万一瓶颈在数据管道，但 `tf.py_function` 绕不过去怎么办？

**大师**：这就是优化要靠"分析→决策→验证"的原因。假设分析发现瓶颈在 `tf.py_function` 的 45% 耗时——你的决策应该是"把它改写为纯 TensorFlow 操作"。如果改写不了（比如依赖一个外部 C 库），那就评估代价——这个 Python 操作是数据加载的瓶颈，是否可以提前离线做（预处理到 TFRecord）？还是可以通过增加 `num_parallel_calls` 来分摊？

关键原则是——**只优化占时最多的那个瓶颈，而不是到处撒胡椒面**。Profiler 会告诉你 Top-1 瓶颈是什么，你的优化就直奔它去。

**步骤一：手动 Profiling——录制并分析训练性能**

目标：用 `tf.profiler` 录制一段训练，导出分析报告。

```python
import tensorflow as tf
import numpy as np
import os
import tempfile

# 模拟一个"慢训练"场景
log_dir = os.path.join(tempfile.mkdtemp(), "profile_logs")

# 构建模型
model = tf.keras.Sequential([
    tf.keras.layers.Dense(1024, activation="relu", input_shape=(512,)),
    tf.keras.layers.Dense(1024, activation="relu"),
    tf.keras.layers.Dense(1024, activation="relu"),
    tf.keras.layers.Dense(10, activation="softmax"),
])

model.compile(optimizer="adam", loss="sparse_categorical_crossentropy")

# 数据（模拟一些处理开销）
def slow_preprocess(x, y):
    # 模拟一个慢的 Python 预处理（类似 tf.py_function 的开销）
    x = x * 1.0001  # 微小计算
    return x, y

X = np.random.randn(500, 512).astype(np.float32)
y = np.random.randint(0, 10, 500)

ds = tf.data.Dataset.from_tensor_slices((X, y))
ds = ds.map(lambda x, y: tf.py_function(
    slow_preprocess, [x, y], [tf.float32, tf.int64]
), num_parallel_calls=tf.data.AUTOTUNE)
ds = ds.batch(32).prefetch(tf.data.AUTOTUNE)

# 手动 Profiling
print("开始 Profiling 录制...")
tf.profiler.experimental.start(log_dir)

# 训练一小段（约 10 个 step）
model.fit(ds, epochs=1, steps_per_epoch=10, verbose=0)

tf.profiler.experimental.stop()
print(f"Profiling 完成，日志保存在: {log_dir}")
print(f"在 TensorBoard 中查看: tensorboard --logdir={os.path.dirname(log_dir)}")
print(f"然后点击 PROFILE 标签页")
```

**步骤三：端到端调优实战——模拟完整的优化闭环**

目标：从 Profile → 找瓶颈 → 优化 → 再 Profile 验证。

```python
import tensorflow as tf
import numpy as np
import time

# === 场景: 模拟一个"慢训练"并逐步优化 ===

# 1. 构建相同的模型和数据
model = tf.keras.Sequential([
    tf.keras.layers.Dense(512, activation="relu", input_shape=(256,)),
    tf.keras.layers.Dense(256, activation="relu"),
    tf.keras.layers.Dense(10, activation="softmax"),
])

X = np.random.randn(2000, 256).astype(np.float32)
y = np.random.randint(0, 10, 2000)

# === Baseline: 最慢版本（模拟各种"反模式"） ===
@tf.function
def bad_train_step(x, y):
    with tf.GradientTape() as tape:
        logits = model(x, training=True)
        loss = tf.reduce_mean(
            tf.keras.losses.sparse_categorical_crossentropy(y, logits, from_logits=True)
        )
    grads = tape.gradient(loss, model.trainable_variables)
    model.optimizer.apply_gradients(zip(grads, model.trainable_variables))
    # 反模式1: 在训练循环中调用 .numpy()（触发 GPU→CPU 拷贝）
    loss_np = loss.numpy()
    return loss_np

model.compile(optimizer="adam", loss="sparse_categorical_crossentropy")

# 测 Baseline
start = time.time()
for i in range(0, 200, 32):
    xb = X[i:i+32]
    yb = y[i:i+32]
    bad_train_step(xb, yb)
bad_time = time.time() - start
print(f"Baseline (bad): {bad_time:.3f}s")

# === 优化 1: 去掉 .numpy()（消除跨设备拷贝） ===
@tf.function
def better_train_step(x, y):
    with tf.GradientTape() as tape:
        logits = model(x, training=True)
        loss = tf.reduce_mean(
            tf.keras.losses.sparse_categorical_crossentropy(y, logits, from_logits=True)
        )
    grads = tape.gradient(loss, model.trainable_variables)
    model.optimizer.apply_gradients(zip(grads, model.trainable_variables))
    return loss  # 返回 Tensor 而非 NumPy

start = time.time()
for i in range(0, 200, 32):
    xb = X[i:i+32]
    yb = y[i:i+32]
    better_train_step(xb, yb)
better_time = time.time() - start
print(f"Optimized (no .numpy): {better_time:.3f}s ({bad_time/better_time:.1f}x)")

# === 优化 2: 加 XLA ===
@tf.function(jit_compile=True)
def best_train_step(x, y):
    with tf.GradientTape() as tape:
        logits = model(x, training=True)
        loss = tf.reduce_mean(
            tf.keras.losses.sparse_categorical_crossentropy(y, logits, from_logits=True)
        )
    grads = tape.gradient(loss, model.trainable_variables)
    model.optimizer.apply_gradients(zip(grads, model.trainable_variables))
    return loss

# 预热触发 XLA 编译
_ = best_train_step(X[:32], y[:32])

start = time.time()
for i in range(0, 200, 32):
    xb = X[i:i+32]
    yb = y[i:i+32]
    best_train_step(xb, yb)
best_time = time.time() - start
print(f"Optimized (+XLA): {best_time:.3f}s ({bad_time/best_time:.1f}x)")

# === 优化 3: 加混合精度 (FP16) ===
from tensorflow.keras import mixed_precision
mixed_precision.set_global_policy("mixed_float16")
# (实际效果需在 GPU 上验证)

print(f"\n=== 优化总览 ===")
print(f"Baseline:     {bad_time:.3f}s (包含反模式)")
print(f"+no .numpy:   {better_time:.3f}s ({bad_time/better_time:.1f}x)")
print(f"+XLA:         {best_time:.3f}s ({bad_time/best_time:.1f}x)")
print(f"+mixed_prec:  (需 GPU 验证, 预期 +20-40%)")
```

### 3.3 性能瓶颈定位速查表

| 症状 | Profile 表现 | 根因 | 优化手段（对应章节） |
|------|------------|------|-------------------|
| GPU 利用率低、CPU 高 | Input Pipeline 占比 > 40% | 数据加载慢 | TFRecord + interleave + AUTOTUNE（第 19 章） |
| step 之间有空隙 | Trace Viewer 中 Kernel 间有空白 | Python 开销 | `@tf.function`（第 33 章） |
| 训练 loop 中周期性慢 | 某些 step 耗时翻倍 | 数据倾斜 | 检查数据集分布（第 19 章） |
| GPU 计算占比高但慢 | Kernel Stats 中有大量 MatMul | 密集型计算 | XLA + 混合精度（第 36 章 + 第 35 章） |
| 显存"锯齿状" | Memory Profile 有周期尖峰 | Checkpoint 保存开销 | 降低 save_freq / 异步保存 |

## 4. 项目总结

### 4.1 Profiler 工具箱

| 工具 | 精度 | 适用场景 | 特点 |
|------|------|---------|------|
| **TensorBoard Profiler** | Kernel 级 | 端到端训练 | 最全面，支持 Trace / Kernel / Memory |
| **nvidia-smi dmon** | 秒级 | 实时 GPU 利用率 | 简单快速，无侵入 |
| **Nsight Systems** | ns 级 | GPU Kernel 深度分析 | NVIDIA 专用，精度最高 |
| **py-spy** | 函数级 | Python 开销分析 | 采样型，低开销 |
| **`tf.print`** | Op 级 | 调试单个 Op 值 | 侵入式，手动放置 |

### 4.2 适用场景

1. **训练速度不达标**：用 Profiler 定位 Input / Compute / Overhead 各自占比
2. **GPU 利用率低**：分析是等数据还是等通信
3. **推理延迟 P99 抖动**：Trace Viewer 找耗时最长的请求类型
4. **显存不足**：Memory Profile 找显存尖峰的触发源
5. **团队性能标准建设**：为每个模型类型建立 Baseline Profile

### 4.3 注意事项

- **Profiling 本身有开销**：不建议对整个训练做 profile——选 10-20 个 step 的区间录制，减少开销
- **Production 环境不要开启 profiling**：profile 会显著拖慢训练（10-30% overhead），仅在调试时使用
- **Profile 结果的时效性**：优化完数据管道后 GPU 计算占比会上升——这是好事，说明瓶颈转移了，需要重新 profile 找下一瓶颈
- **不同 GPU 型号的 Profile 行为不同**：A100 的 TF32 模式和 V100 的 FP16 行为不同，profile 分析要和实际硬件对应

### 4.4 常见踩坑经验

1. **坑**：Profile 显示 GPU 利用率 85%，但训练还是慢。
   **根因**：85% 是"SM 占用率"，不是"有效计算"。可能有大量 Kernel 在等数据、等同步、或执行很小规模的并行度不足的 Kernel。
   **解决**：看 Trace Viewer 中 GPU Stream 的时间线——如果有"竖向白线"（idle），说明该时间段 GPU 在空转。

2. **坑**：优化 Input Pipeline 后，GPU 利用率反而下降了。
   **根因**：数据变快后每 step 时间缩短，但 step 数量没变——GPU 的 idle 时间百分比上升了，但绝对计算时间不变。
   **解决**：这是正常现象——总训练时间变短了，"利用率"这个相对指标不重要。关注绝对时间。

3. **坑**：`profile_batch` 指定的 batch 范围不是刚好在感兴趣的 interval 内。
   **根因**：`profile_batch='500,520'` 是 batch index 而非 step index——如果使用了 `validation_freq` 或 `shuffle`，batch 编号可能偏移。
   **解决**：先用 `profile_batch=0` 关闭自动 profiling，手动用 `tf.profiler.experimental.start()`/`stop()` 包裹感兴趣的代码段。
