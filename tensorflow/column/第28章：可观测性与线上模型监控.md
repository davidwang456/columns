# 第28章：可观测性与线上模型监控

## 1. 项目背景

某电商平台的推荐模型已上线运行 3 个月。运维团队某天早上发现——告警系统静悄悄，但业务方反馈"最近推荐的东西越来越不准了"。排查后发现：

- 模型推理延迟 P50 正常（45ms），但 P99 从 80ms 飙升到了 950ms——只有少数慢请求没触发平均延迟告警
- 输入特征的分布已经偏移——用户群体的年龄结构从年轻人为主逐渐变成中老年人为主，但模型是基于 3 个月前年轻用户数据训练的
- 推荐 API 错误率从 0.1% 涨到了 1.5%（每天约 15 万次失败），但因为没有错误率监控，没人注意到

运维负责人老周痛定思痛："我们监控了服务器 CPU、内存、磁盘——但这些'机器指标'正常不代表'模型正常'。模型有它自己的生命体征——数据分布、推理延迟、业务指标——没有这些，上线只是盲飞。"

**痛点放大**：模型上线不等于运维结束。机器学习系统需要三项专属监控：(1) 服务层监控（QPS、延迟、错误率）；(2) 数据质量监控（特征缺失率、分布漂移、异常值）；(3) 业务指标监控（线上 CTR、转化率、误杀率）。没有这些，模型可能悄无声息地退化，直到用户投诉才发现。

## 2. 项目设计

**小胖**（盯着 Grafana 大盘上密密麻麻的 CPU 曲线）：我们不是已经有监控了吗？CPU < 80%、内存 < 90%、磁盘 < 85%——三个绿灯，有啥好担心的？

**大师**：那你解释一下——CPU 50%、内存 60% 的情况下，你的推荐模型突然给所有男性用户推了卫生巾——这算不算"系统正常"？传统运维监控的是机器的"心跳"，模型监控监控的是模型的"智力"。心跳正常 ≠ 脑子清醒。

**技术映射**：基础设施监控（CPU/内存/磁盘） = 机器的生命体征；模型监控（数据漂移/业务指标/推理质量）= 模型的健康度。两者互补，不可替代。

**小白**：数据漂移（Data Drift）到底是什么？怎么检测？

**大师**：数据漂移就是模型看到的输入数据，和它训练时看到的分布不一样了。比如你的模型训练时用户年龄均值是 25 岁，上线 3 个月后线上用户年龄均值变成了 35 岁——这就是特征分布漂移。

检测方法用 **PSI（Population Stability Index）** 或直接对比分布：

```python
# 检测特征漂移
def detect_drift(train_feature, online_feature, threshold=0.2):
    """简单版的分布漂移检测：KS 统计量"""
    from scipy.stats import ks_2samp
    statistic, p_value = ks_2samp(train_feature, online_feature)
    drifted = p_value < 0.01  # p < 0.01 → 分布显著不同
    return {"drifted": drifted, "ks_stat": statistic, "p_value": p_value}
```

如果漂移了——说明线上数据已经和训练时不一样了，模型的效果可能不再可靠，需要重新训练。

**技术映射**：数据漂移 = 线上特征分布与训练时的差异。常用检测方法有 KS 检验（连续特征）、PSI（分箱特征）、卡方检验（离散特征）。检测到漂移后应触发"重新训练"或"人工审核"。

**小白**：那 Prometheus + Grafana 怎么和 TensorFlow Serving 集成？

**大师**：很简单——TensorFlow Serving 自带 Prometheus 指标暴露端口。你只需要加一行启动参数：

```bash
tensorflow_model_server \
  --rest_api_port=8501 \
  --model_name=my_model \
  --model_base_path=/models/my_model \
  --monitoring_config_file=/prometheus/monitoring_config.txt
```

Serving 会暴露 Prometheus 的 `/monitoring/prometheus/metrics` 端点，包含：
- `:tensorflow:serving:request_count` — 请求总数
- `:tensorflow:serving:request_latency` — 请求延迟（可配置分位数）
- `:tensorflow:serving:runtime_memory_bytes` — 内存使用

然后在 Prometheus 中配置 scrape，Grafana 通过 Prometheus 数据源展示大盘。

**技术映射**：TensorFlow Serving 原生支持 Prometheus metrics 导出。通过 `--monitoring_config_file` 配置暴露哪些分位数的延迟。Grafana 通过 PromQL 查询这些指标构建可视化大盘。

**小胖**：告警怎么设？延迟超过多少该告警？

**大师**：告警需要分级：

| 级别 | 条件 | 动作 | 通知对象 |
|------|------|------|---------|
| **P3 (Info)** | P95 延迟 > 基线 × 1.5 | 记录日志 | 无 |
| **P2 (Warning)** | P99 延迟 > 基线 × 2 或 错误率 > 1% | 发群消息 | 算法+运维 |
| **P1 (Critical)** | 错误率 > 5% 或 模型服务不可用 | 电话 + 自动回滚 | 全团队 |

关键是——**告警不只是通知，还要有回滚预案**。如果新模型上线的延迟异常，应该能在 30 秒内回滚到上一个版本（TensorFlow Serving 的模型版本目录天然支持）。

**步骤二：数据漂移检测实现**

目标：实现一个可定时运行的漂移检测脚本。

```python
import numpy as np
from scipy import stats

class DataDriftDetector:
    """数据漂移检测器"""

    def __init__(self, reference_data, feature_names=None):
        """
        reference_data: dict of feature_name → numpy array (训练集基线)
        """
        self.reference = reference_data
        self.feature_names = feature_names or list(reference_data.keys())

    def detect_ks_drift(self, online_data, threshold=0.01):
        """
        用 KS 检验检测每个特征的分布漂移
        online_data: dict of feature_name → numpy array (线上采样)
        返回: 漂移的特征列表
        """
        drifted_features = []
        report = {}

        for feat in self.feature_names:
            ref = self.reference[feat]
            onl = online_data[feat]

            stat, p_value = stats.ks_2samp(ref, onl)
            is_drifted = p_value < threshold

            report[feat] = {
                "ks_statistic": stat,
                "p_value": p_value,
                "drifted": is_drifted,
                "ref_mean": np.mean(ref),
                "online_mean": np.mean(onl),
            }

            if is_drifted:
                drifted_features.append(feat)

        return drifted_features, report

    def detect_missing_rate(self, online_data, max_increase=0.05):
        """
        检测缺失率是否显著上升
        返回: 缺失率异常的特征列表
        """
        missing_alerts = []

        for feat in self.feature_names:
            ref_missing = np.mean(np.isnan(self.reference[feat]))
            onl_missing = np.mean(np.isnan(online_data[feat]))

            if onl_missing - ref_missing > max_increase:
                missing_alerts.append({
                    "feature": feat,
                    "ref_missing_rate": ref_missing,
                    "online_missing_rate": onl_missing,
                })

        return missing_alerts

# === 模拟漂移检测 ===
np.random.seed(42)

# 训练集（年轻用户为主，年龄 mean=25）
ref_data = {
    "age": np.random.normal(25, 5, 5000),
    "income": np.random.normal(8000, 2000, 5000),
    "purchase_count": np.random.poisson(3, 5000).astype(float),
}

# 线上数据（中老年用户增多，年龄 mean=38 → 漂移！）
online_data = {
    "age": np.random.normal(38, 7, 1000),
    "income": np.random.normal(8200, 2500, 1000),  # 收入轻微变化
    "purchase_count": np.random.poisson(3, 1000).astype(float),  # 无变化
}

detector = DataDriftDetector(ref_data)

print("=== 数据漂移检测报告 ===\n")
drifted, report = detector.detect_ks_drift(online_data)

for feat, r in report.items():
    status = "⚠️ 漂移!" if r["drifted"] else "✅ 正常"
    print(f"{feat:<20} {status:<10} "
          f"KS={r['ks_statistic']:.4f}, p={r['p_value']:.4f}, "
          f"ref_mean={r['ref_mean']:.1f}, online_mean={r['online_mean']:.1f}")

# 缺失率检测
missing = detector.detect_missing_rate(online_data)
if missing:
    print(f"\n缺失率告警: {len(missing)} 个特征")
else:
    print(f"\n缺失率正常 ✅")
```

### 3.3 监控指标速查表

| 层级 | 指标 | 采集方式 | 告警阈值建议 |
|------|------|---------|-------------|
| **服务层** | QPS | Serving Prometheus endpoint | 下降 50% → P2 |
| **服务层** | P50/P99 延迟 | Serving Prometheus endpoint | P99 > 基线×2 → P2 |
| **服务层** | 错误率 | Serving Prometheus endpoint | > 1% → P2, > 5% → P1 |
| **数据层** | 特征缺失率 | 定时采样脚本 | 上升 5% → P2 |
| **数据层** | 分布漂移 (KS/PSI) | 定时采样脚本 | KS > 0.2 → P2 |
| **数据层** | 异常值比例 | 定时采样脚本 | > 10% → P3 |
| **业务层** | 线上 CTR/转化率 | 业务数据库 | 下降 10% → P2 |
| **业务层** | 模型版本 | Serving API | 非预期版本 → P1 |

## 4. 项目总结

### 4.1 模型监控 vs 传统运维监控

| 方面 | 传统运维监控 | 模型监控 |
|------|------------|---------|
| 监控对象 | 服务器/进程/网络 | 模型推理/数据/业务 |
| 核心指标 | CPU/内存/磁盘/带宽 | QPS/延迟/错误率/漂移/CTR |
| 异常表现 | 机器宕机、资源耗尽 | 模型退化（静悄悄的） |
| 告警响应 | 重启/扩容 | 回滚/重训练/数据修复 |
| 工具 | Prometheus + Node Exporter | Prometheus + TF Serving metrics + 自研采样 |

### 4.2 适用场景

1. **在线推理服务**：TensorFlow Serving 的实时 QPS/延迟/错误率监控
2. **模型效果退化告警**：数据漂移检测 + 业务指标波动 → 自动触发重训练
3. **多模型 A/B 测试**：对比不同版本的 CTR/延迟，辅助灰度决策
4. **合规审计**：记录每次模型调用的输入 hash（脱敏后），用于事后追溯

**不适用场景**：
1. 离线批量推理——使用 Spark/TFX 的批处理监控体系
2. 极低频调用（< 1 QPS）——统计样本太少，延迟和错误率不稳定，用日志分析替代

### 4.3 注意事项

- **Prometheus 的 `histogram_quantile` 在小流量下不准确**：QPS < 10 时，P99 的分位数估计误差很大。需等流量稳定后再判断
- **数据漂移检测的采样频率**：太频繁（每分钟）→ 计算开销大 + 误报多；太稀疏（每周）→ 发现太晚。建议每小时采样 1000 条做检测
- **业务指标有自然周期**：电商的 CTR 在工作日和周末不同，凌晨和白天不同。告警阈值应考虑周期性（用同周期对比而非绝对值）
- **监控数据存储成本**：Prometheus 的 TSDB 存储 15 天，如果需要长期趋势分析（数月），需配置 Remote Write 到 InfluxDB/VictoriaMetrics

### 4.4 常见踩坑经验

1. **坑**：特征漂移检测全部飘红，KS > 0.5——但模型效果没降。
   **根因**：KS 检验对样本量敏感（样本量大 → 微小差异也显著）。用效应量（Cohen's d）代替 p-value 判断实际影响。
   **解决**：结合 PSI（分箱稳定性指数）——PSI < 0.1 无需关注，0.1-0.25 告警，> 0.25 严重。

2. **坑**：Prometheus 的 `request_latency_bucket` 分位数都是 0。
   **根因**：Serving 的 histogram bucket 配置的上界太小（如最大 100ms），所有请求延迟都超过了最大 bucket。
   **解决**：通过 `--monitoring_config_file` 自定义 bucket 范围，确保覆盖 P99 延迟。

3. **坑**：半夜模型自动回滚了，但第二天早上发现回滚错了——回了 3 个版本前的，准确率下降了 5%。
   **根因**：自动回滚只检查了"延迟正常"，没检查"模型指标正常"。回滚到的旧版本可能模型效果更差。
   **解决**：回滚逻辑中加上一个最低指标门槛——回滚到的版本必须满足最新的 Golden Dataset 评估阈值。

### 4.5 思考题

1. 你的推荐模型 QPS 在每天 12:00 和 20:00 有两个高峰（午饭和晚饭时间刷手机）。P99 延迟在高峰期从 50ms 升到 200ms，但平均延迟从 45ms 只涨到 55ms。你的告警应该基于 P99 还是平均延迟？为什么？如何配置 Prometheus 的告警规则避免高峰期误报？

2. 数据漂移检测到"用户年龄分布变了"——但这不是 Bug，而是你的产品成功拓展了中老年市场。这种情况下，你应该"重新训练模型"还是"接受漂移"？如何设计一个"有意义的漂移 vs 无害的漂移"的分类标准？
