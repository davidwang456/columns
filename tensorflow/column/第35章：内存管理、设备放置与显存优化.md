# 第35章：内存管理、设备放置与显存优化

## 1. 项目背景

算法工程师阿磊正在训练一个 12 层的 Transformer 文本生成模型。他在 V100 (32GB) 上训练，batch_size=16 时显存用满 31GB——模型常驻 8GB，剩余 23GB 用于存储中间激活值（activations）和梯度。他试图把 batch_size 加到 32——OOM 了。

排查发现：激活值占用了他 23GB 显存中的 18GB！原因是——在前向传播过程中，每一层的输出都被保留下来（因为反向传播时需要它们计算梯度）。层数越深、batch_size 越大，激活值占的显存就越恐怖。

阿磊尝试了三种方案：(1) 减小 batch_size（损害训练稳定性）；(2) 开启 Gradient Checkpointing（用时间换空间——只保留部分中间激活值，丢掉的需要在反向传播时重新计算）；(3) 开启混合精度训练（FP16，显存减半）。最终，梯度检查点 + 混合精度将显存从 31GB 降到 18GB，batch_size 成功翻倍。

**痛点放大**：显存不只是存模型参数——参数 + 梯度 + 优化器状态 + 激活值，四者的总和决定了显存占用。理解 TensorFlow 的内存分配器（BFC Allocator）、设备放置（Placer）、以及梯度检查点/混合精度的原理，是训练大模型和解决 OOM 问题的必修课。

## 2. 项目设计

**小胖**（看着 `nvidia-smi` 上 31GB/32GB 的红线）：我模型总共就 1000 万参数，float32 也就 40MB——怎么占了我 31GB 显存？其他 30GB 去哪了？

**大师**：你这 1000 万参数只是"冰山一角"。完整的显存占用是：

```
总显存 = 参数 + 梯度 + 优化器状态 + 激活值

参数:          40 MB  (你看到的)
梯度:          40 MB  (与参数同大小)
优化器状态:    80 MB  (Adam 需要存 m 和 v，各 40MB)
激活值:     ~18,000 MB  (每一层的中间输出！)
其他:         ~3,000 MB  (cuDNN workspace, 临时 buffer)
────────────────────────
总计:       ~21,160 MB → 20.7 GB (这只是估算)
```

> 实际：12 层 Transformer, batch=16, seq_len=512, hidden=768 → 激活值约 18GB

**技术映射**：模型训练显存 = 参数 + 梯度 + 优化器状态（通常 2-4x 参数）+ 激活值（通常 5-20x 参数，与网络深度和 batch_size 正相关）。

**小白**：每个层的激活值是"怎么保留的"？为什么反向传播需要它们？

**大师**：这是自动微分的"隐性成本"。当我们用 `GradientTape` 追踪前向计算时，它默认会保留所有被 `watch` 的 Tensor 和中间结果。比如 ReLU 的梯度需要知道"前向时哪些值 > 0"，Dropout 的梯度需要知道"前向时哪些神经元被关了"。这些信息都在激活值里——TensorFlow 把它们存在显存中直到反向传播完成。

**技术映射**：GradientTape 的默认行为（`persistent=False, watch_accessed_variables=True`）会保留前向传播中所有中间 Tensor——这是自动微分的便利性代价。

**小白**：那梯度检查点（Gradient Checkpointing）怎么"用时间换空间"？

**大师**：原理很简单——前向时不存所有激活值，只存"检查点"（checkpoint，如每 2 层存一个）。反向传播时，从最近的检查点重新计算缺失的激活值。比如一个 12 层的网络——不存第 3-10 层的激活值，只存第 2、5、8、11 层。反向传播从最后一层往前时，如果需要第 7 层的激活值，就从第 5 层的检查点重新算到第 7 层——多花一点计算时间，但省下了存 7 层激活值的显存。

**技术映射**：Gradient Checkpointing 是一个时空权衡——以重新计算（约 20% 额外计算）为代价，将激活值显存从 O(L) 降到 O(√L) 或 O(log L)。

**小胖**：混合精度训练又是什么？FP16 不就是 float16 吗？为什么能省显存又能加速？

**大师**：混合精度训练 = FP16 计算 + FP32 参数。前向和反向传播用 FP16（速度快 2-3 倍，显存减半），但参数和优化器状态保持 FP32（防止精度损失累积）。此外，FP16 的数值范围是 [-65504, 65504]，某些大梯度可能溢出——所以使用**损失缩放**（Loss Scaling）：前向时把 loss 乘以一个大数（如 1024），让小的梯度落在 FP16 的可表达范围内，反向后再除以这个数还原。

```python
# 开启混合精度
from tensorflow.keras import mixed_precision
mixed_precision.set_global_policy("mixed_float16")
# FP16 计算 + FP32 参数存储，显存约省 40%，速度提升 2-3x
```

**步骤一：显存占用分析与诊断**

目标：量化各部分的显存占用，定位大头。

```python
import tensorflow as tf
import numpy as np

# === 诊断模型显存占用 ===
def analyze_memory(model, sample_input):
    """粗略估算模型的显存占用"""
    # 1. 参数量
    total_params = model.count_params()
    param_mb = total_params * 4 / 1024 / 1024  # float32 = 4 bytes

    print(f"=== 显存占用分析 ===")
    print(f"参数量: {total_params:,}")
    print(f"参数显存: {param_mb:.1f} MB")
    print(f"梯度显存: {param_mb:.1f} MB (与参数同大小)")
    print(f"优化器状态: {param_mb * 2:.1f} MB (Adam 需 m + v, 各与参数同大小)")

    # 2. 估算激活值 (大致的)
    # 简化估算：每层输出 = batch × hidden × seq_len × bytes
    print(f"\n激活值显存: 取决于 batch_size / 层数 / hidden_dim")
    print(f"  经验公式: 激活值 ≈ O(batch × layers × hidden²)")
    print(f"  12 层 Transformer, batch=16, hidden=768:")
    print(f"    约 12 × 16 × 768² × 4 bytes ≈ 450 MB (粗略, 实际更大)")

    # 3. cuDNN workspace
    print(f"\n其他显存:")
    print(f"  cuDNN workspace: 通常 500MB-2GB")
    print(f"  TF runtime overhead: 通常 200-500MB")

# 构造一个"超大"模型来演示
model = tf.keras.Sequential([
    tf.keras.layers.Dense(4096, activation="relu", input_shape=(1024,)),
    tf.keras.layers.Dense(4096, activation="relu"),
    tf.keras.layers.Dense(4096, activation="relu"),
    tf.keras.layers.Dense(1000, activation="softmax"),
])

sample_input = tf.ones([1, 1024])
analyze_memory(model, sample_input)

# === GPU 内存诊断 ===
gpus = tf.config.list_physical_devices("GPU")
if gpus:
    try:
        for gpu in gpus:
            details = tf.config.experimental.get_device_details(gpu)
            print(f"\nGPU: {details.get('device_name', 'Unknown')}")
            # 显存增长模式
            tf.config.experimental.set_memory_growth(gpu, True)
            print("显存增长模式: 已开启 (仅用才申请)")
    except RuntimeError:
        print("GPU 已初始化，无法更改显存配置")
else:
    print("\n(无 GPU 可用，跳过显存诊断)")
```

**步骤二：梯度检查点（Gradient Checkpointing）实现**

目标：演示"有/无 checkpointing"时显存和速度的差异。

```python
import tensorflow as tf
from tensorflow import keras
import time

# 构建一个深层 MLP（模拟深层网络的激活值累积）
def build_deep_model(layers=12, units=512):
    model = keras.Sequential()
    model.add(keras.layers.Dense(units, activation="relu", input_shape=(units,)))
    for _ in range(layers - 1):
        model.add(keras.layers.Dense(units, activation="relu"))
    model.add(keras.layers.Dense(10, activation="softmax"))
    return model

# === 无 Checkpointing ===
model_no_ckpt = build_deep_model(12, 512)
model_no_ckpt.compile(optimizer="adam", loss="sparse_categorical_crossentropy")

# 用 %timeit 模拟（实际应在有 GPU 环境下运行）
X = tf.random.normal([64, 512])
y = tf.random.uniform([64], 0, 10, dtype=tf.int32)

print("训练一步（无 checkpointing）：")
start = time.time()
model_no_ckpt.fit(X, y, batch_size=64, epochs=1, verbose=0)
print(f"  耗时: {time.time()-start:.3f}s")

# === 如何开启 Gradient Checkpointing ===
print("""
在 TensorFlow 中开启梯度检查点:
  目前 TF 2.x 的推荐方式是在自定义训练循环中手动实现:

  # 无需 checkpointing 的前向（存所有激活值）
  with tf.GradientTape() as tape:
      output = model(x, training=True)
      loss = loss_fn(y_true, output)

  # 使用 checkpointing（不存 tape 中间结果, 需要时重新计算）
  # tf.recompute_grad 或 gradient_checkpointing 库

  方案 A: 使用 tf.recompute_grad (TF Addons)
    pip install tensorflow-addons
    from tensorflow_addons.layers import wrappers
    # 或用 recompute_grad 装饰自定义 Layer

  方案 B: 手动实现——只对需要检查点的部分开 tape
    def forward_and_loss(x, y_true):
        # 分两段: checkpoint1 → 不需要 tape → checkpoint2 → tape
        h1 = model.layer1(x)
        # 第二段需要梯度
        with tf.GradientTape() as tape:
            output = model.rest(h1)
            loss = loss_fn(y_true, output)
        return loss, tape, h1

  方案 C: 使用官方 recompute_grad (TF 2.16+ 实验性):
    tf.recompute_grad(model.call)(x)  # 标记不存激活值

  效果: 显存约减少 40-60%, 训练时间增加约 20-30%
""")
```

**步骤三：混合精度训练实战**

目标：开启混合精度，验证显存下降和精度保持。

```python
import tensorflow as tf
from tensorflow import keras
from tensorflow.keras import mixed_precision
import numpy as np

# === 1. 开启混合精度 ===
# 注意: 需要 NVIDIA GPU (Volta 及以上, compute capability ≥ 7.0)
policy = mixed_precision.Policy("mixed_float16")
mixed_precision.set_global_policy(policy)
print(f"计算精度: {policy.compute_dtype}")    # float16
print(f"变量精度: {policy.variable_dtype}")   # float32
print(f"损失缩放: 自动 (dynamic)")

# === 2. 构建模型（混合精度兼容） ===
model = keras.Sequential([
    keras.layers.Dense(512, activation="relu", input_shape=(256,), dtype="float32"),
    # 第一层保持 float32（输入通常是 float32）
    keras.layers.Dense(512),
    keras.layers.Dense(10, dtype="float32"),
    # 输出层保持 float32（softmax 在 float16 下可能不稳定）
])

# 优化器使用损失缩放
optimizer = keras.optimizers.Adam(1e-3)
optimizer = mixed_precision.LossScaleOptimizer(optimizer)
# LossScaleOptimizer 自动对 loss 做缩放/还原

model.compile(
    optimizer=optimizer,
    loss=keras.losses.SparseCategoricalCrossentropy(from_logits=True),
    metrics=["accuracy"],
)

# 模型参数统计
for layer in model.layers:
    for w in layer.weights:
        print(f"{w.name}: shape={w.shape}, dtype={w.dtype}")
    print()

# === 3. 训练验证 ===
X = np.random.randn(1000, 256).astype(np.float32)
y = np.random.randint(0, 10, 1000)

history = model.fit(X, y, batch_size=32, epochs=3, validation_split=0.2, verbose=1)

print(f"\n最终验证准确率: {history.history['val_accuracy'][-1]:.4f}")
print(f"混合精度模式验证完成")

# 恢复默认精度（避免影响后续实验）
mixed_precision.set_global_policy("float32")
```

**步骤四：设备放置策略与跨设备拷贝优化**

目标：手动控制 Tensor 放在哪个设备上，避免隐式拷贝。

```python
import tensorflow as tf

# 检查可用设备
print("=== 可用设备 ===")
for device in tf.config.list_logical_devices():
    print(f"  {device.device_type}: {device.name}")

# === 手动设备放置 ===
print("\n=== 设备放置示例 ===")

# 默认: TensorFlow 自动选择设备
a = tf.constant([1.0, 2.0])
print(f"默认放置: {a.device}")

# 手动指定 CPU

    b = tf.constant([3.0, 4.0])
    print(f"手动 CPU: {b.device}")

# 如果有 GPU, 指定 GPU
gpus = tf.config.list_physical_devices("GPU")
if gpus:
    with tf.device("/GPU:0"):
        c = tf.constant([5.0, 6.0])
        print(f"手动 GPU: {c.device}")

# === 跨设备拷贝的性能陷阱 ===
print("\n=== 跨设备拷贝分析 ===")
print("""
常见隐式拷贝 (性能杀手):
  1. CPU Tensor + GPU Tensor → 隐式拷贝到 GPU
  2. numpy_array + GPU Tensor → 隐式上传到 GPU
  3. model.predict(cpu_data) → 自动分批上传

优化策略:
  1. tf.data 管道的 prefetch 放在 GPU 上:
     dataset.prefetch(tf.data.AUTOTUNE)
  2. 避免在训练循环中 .numpy() (强制拷贝回 CPU)
  3. 用 tf.device 确保数据在正确的设备上

示例: 如何在 Serving 中避免 CPU↔GPU 拷贝
  - 预处理放在 CPU (image decode/resize)
  - 模型推理放在 GPU (matmul/conv)
  - 用 tf.data pipeline 的 prefetch 做 pipeline 化
""")
```

### 3.3 显存优化手段速查

| 优化手段 | 显存节省 | 训练速度影响 | 适用条件 |
|---------|---------|------------|---------|
| **减小 batch_size** | 线性（减半省半） | 可能降低收敛速度 | 任何场景 |
| **梯度累积** | 同减小 batch_size | 增加训练时间 | 显存有限但需要大 batch |
| **混合精度 (FP16)** | 40-50% | 加速 2-3x | Volta+ GPU (V100/A100) |
| **梯度检查点** | 40-60% | 慢 20-30% | 深层网络、大 batch_size |
| **减小 hidden_dim** | 线性 | 可能降低模型能力 | 模型设计阶段 |
| **梯度裁剪** | 无 | 无 | 防止梯度爆炸,非直接省显存 |
| **删除中间 Tensor (del)** | 少量 | 极小 | 手动优化,谨慎使用 |

## 4. 项目总结

### 4.1 BFC Allocator vs 简化理解

| 概念 | 简化理解 | 源码位置 |
|------|---------|---------|
| **BFC Allocator** | GPU 显存的"管家"——分配/合并/释放显存块 | `tensorflow/core/common_runtime/bfc_allocator.cc` |
| **显存碎片** | 管家分配的块太多小块，大块申请时"有空间但没连片" | — |
| **memory_growth** | 管家说"用了再要，别一开始全占着" | `tf.config.experimental.set_memory_growth(True)` |
| **Placer** | "决策中心"——这个 Op 该放 CPU 还是 GPU | `tensorflow/core/common_runtime/placer.cc` |

### 4.2 适用场景

1. **大模型训练（> 1GB 参数）**：混合精度 + 梯度检查点的组合拳
2. **显存受限训练**：梯度累积用时间换空间
3. **多任务模型**：手动设备放置——backbone 放 GPU，分支放不同 GPU
4. **推理服务优化**：固定 batch_size + 预分配显存，避免动态分配碎片化

**不适用场景**：
1. 模型极小（< 10MB 参数）——优化手段的开销可能大于收益
2. 模型只能用 CPU 训练——显存管理针对 GPU，CPU 内存管理不同

### 4.3 注意事项

- **FP16 的数值稳定性**：Softmax/LayerNorm/CrossEntropy 在 FP16 下容易溢出——这些层应该强制设为 float32
- **Gradient Checkpointing 与 BN 的互斥**：BN 在重计算时的 batch 统计量可能和首次前向不同——梯度检查点需要特殊处理 BN 层
- **`memory_growth=True` 不是万能的**：它只是推迟了显存分配的时间（第一次用到才分配），但不能减少峰值显存
- **显存碎片与重启**：长期训练（数天）可能导致 BFC Allocator 的显存碎片化——定期重启训练（如每 24 小时从 checkpoint 恢复）清理碎片

### 4.4 常见踩坑经验

1. **坑**：开启混合精度后 loss 暴增到 NaN。
   **根因**：网络的最后一层（Softmax）或 loss 函数（CrossEntropy）直接在 FP16 下计算，导致下溢（underflow）。
   **解决**：强制输出层和 loss 用 float32：`output_layer(..., dtype="float32")`；loss 使用 `from_logits=True`（内部做 FP32 的 softmax）。

2. **坑**：`model.fit` 时前 3 个 batch 正常，第 4 个 batch OOM。
   **根因**：`memory_growth=True` 下显存在前几个 batch 逐步增长，第 4 个 batch 触发了额外的 `cuDNN workspace` 分配导致爆显存。
   **解决**：降低 batch_size；或在第一个 batch 后手动触发一次完整的显存分配以测试峰值。

3. **坑**：梯度检查点后模型训练不收敛，loss 反复震荡。
   **根因**：Dropout 在重计算时的随机 mask 和第一次前向不同——导致梯度计算基于不同的"dropout pattern"。
   **解决**：在重计算时设置相同的随机种子；或使用 `tf.random.experimental.Generator` 为每个 checkpoint 固定种子。
