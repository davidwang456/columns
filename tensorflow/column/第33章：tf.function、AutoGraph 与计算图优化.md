# 第33章：tf.function、AutoGraph 与计算图优化

## 1. 项目背景

算法工程师大刚正在优化一个推理服务。他的模型是一个带有自定义前处理逻辑的 NLP 模型——输入一个字符串，输出分类结果。他用 Python 写了一个 `infer(text)` 函数，里面包含了字符串处理、Token 化、查词典、Padding、模型推理。这个函数的 QPS 只有 15，离业务要求的 200 差了 10 倍以上。

大刚在函数上加了一个 `@tf.function`，QPS 飙升到 180——但他发现了一个严重问题：当输入文本长度从 50 字变成 100 字时，函数触发了 retracing——整个图的构建和优化被重新执行了一遍，这 100ms 的额外开销把 P99 延迟从理想情况推到了不可接受的水平。更糟糕的是，他的代码里有一段 `if len(text) > 200: text = text[:200]`——AutoGraph 试图把这个 Python `if` 转换成图模式的条件分支，但 `len(text)` 是一个 Python string 操作（不可图化），导致转换失败，整个函数退出了图模式。

**痛点放大**：`@tf.function` 是 TensorFlow 2.x 中连接 Eager 开发体验和图模式性能的桥梁——但它不是"加一行就万事大吉"。tracing/retracing、AutoGraph 的 Python→图转换、`input_signature` 的设计、图模式下的 Python 副作用——这四件事如果理解不到位，`@tf.function` 带来的不是加速而是 bug。

## 2. 项目设计

**小胖**（困惑地看着代码）：`@tf.function` 到底做了什么魔法？同样的代码，加一行 QPS 从 15 变成 180？

**大师**：解释 `@tf.function` 之前，你先理解 Eager 模式的"隐形税"。你写的每行 `tf.add(a, b)` ——Python 解释器要花时间做函数调用、参数解析、返回值包装。如果你的计算图有 500 个 Operation，每个 Operation 都要过一次 Python 解释器——这 500 次"过路费"加起来可能比真正计算的时间还长。

`@tf.function` 做的第一件事就是**把 Python 代码编译成计算图（Graph）**——编译过程叫 Tracing。一旦图编译好了，后续调用就直接执行图——500 个 Operation 变成一次 `Session::Run(graph)`。Python 解释器只参与了第一次编译，后续调用完全不经过 Python——这就是 QPS 从 15 飙到 180 的原因。

**技术映射**：`@tf.function` = Tracing（编译） + Caching（缓存） + Execution（直接执行图）。首次调用触发 tracing 生成图，后续同参数调用直接执行缓存的图，跳过 Python 解释器。

**小白**：那 AutoGraph 是什么？和 `@tf.function` 是包含关系还是并列关系？

**大师**：`@tf.function` 内部依赖 AutoGraph。AutoGraph 的职责是把 Python 的控制流（`if`/`for`/`while`）转换成 TensorFlow 的图操作（`tf.cond`/`tf.while_loop`）。比如：

```python
@tf.function
def my_fn(x):
    if x > 0:       # Python if → AutoGraph 转换为 tf.cond
        return x * 2
    else:
        return x * 3
```

AutoGraph 会把这段代码分析 AST（抽象语法树），把 `if x > 0` 替换为 `tf.cond(x > 0, lambda: x*2, lambda: x*3)`。这样整个函数就变成了一个纯 TensorFlow 操作组成的图。

**小胖**：那为什么有时候 `if` 转换会失败？

**大师**：因为 AutoGraph 只能转换"依赖 Tensor 的条件"。如果 `x` 是一个 Python 的 int，`if x > 0` 在 tracing 阶段就确定了——它只会走一个分支，另一个分支根本不进入图。如果 `x` 是一个 `tf.Tensor`，AutoGraph 才能正确转换为 `tf.cond`。更关键的坑是——如果你的 `if` 条件混杂了 Python 逻辑（如 `len(string)`），AutoGraph 会放弃转换，直接执行 Python 分支（导致图不完整，性能退化）。

**技术映射**：AutoGraph 的转换限制——只转换 Tensor 依赖的控制流。Python 类型的条件在 tracing 阶段确定，不可后续改变。

**小白**：频繁 retracing 怎么解决？每次输入 shape 变了就重新 trace？

**大师**：三个手段：

1. **`input_signature` 锁定**：`@tf.function(input_signature=[tf.TensorSpec(shape=[None, 256], dtype=tf.float32)])`——告诉 TF "只要是一维不定长 + 256 列的 float32 矩阵，都不要重新 trace"。

2. **保持 shape 稳定**：在数据管道中统一 padding 长度、用 `drop_remainder=True` 确保 batch 大小一致。

3. **避免在函数内部引用 Python 全局变量**：`@tf.function` 在 tracing 时会把 Python 对象"烘焙"进图——之后改全局变量不会生效，而且不同类型的全局变量会导致额外 retracing。

### 3.2 分步实现

**步骤一：`@tf.function` 性能对比——量化 Python 开销**

目标：对比有/无 `@tf.function` 的加速效果，理解 Python 开销。

```python
import tensorflow as tf
import time
import numpy as np

# 一个小型 Transformer 推理模拟（多个 matmul + softmax + add）
def compute_eager(x, w1, w2):
    """Eager 模式：每步都回 Python"""
    h = tf.matmul(x, w1)
    h = tf.nn.relu(h)
    h = h + tf.random.normal(tf.shape(h)) * 0.01
    h = tf.matmul(h, w2)
    return tf.nn.softmax(h)

@tf.function
def compute_graph(x, w1, w2):
    """图模式：tracing 一次，后续直接执行"""
    h = tf.matmul(x, w1)
    h = tf.nn.relu(h)
    h = h + tf.random.normal(tf.shape(h)) * 0.01
    h = tf.matmul(h, w2)
    return tf.nn.softmax(h)

# 准备数据
d, k = 256, 128
x = tf.random.normal([64, d])
w1 = tf.random.normal([d, k])
w2 = tf.random.normal([k, 10])

# === Benchmark ===
# Eager 预热 + 测试
compute_eager(x, w1, w2)
start = time.time()
for _ in range(200):
    compute_eager(x, w1, w2)
eager_time = time.time() - start

# Graph 预热（第一次触发 tracing）+ 测试
compute_graph(x, w1, w2)
start = time.time()
for _ in range(200):
    compute_graph(x, w1, w2)
graph_time = time.time() - start

print(f"Eager 模式: {eager_time:.3f}s ({200/eager_time:.0f} calls/s)")
print(f"Graph 模式: {graph_time:.3f}s ({200/graph_time:.0f} calls/s)")
print(f"加速比:    {eager_time/graph_time:.1f}x")
```

运行输出：
```
Eager 模式: 2.345s (85 calls/s)
Graph 模式: 0.156s (1282 calls/s)
加速比:    15.0x
```

**步骤二：`input_signature` 防止 retracing**

目标：理解 retracing 的触发条件，用 `input_signature` 锁定。

```python
import tensorflow as tf

# === 问题版本：频繁 retracing ===
@tf.function
def problem_fn(x):
    return tf.reduce_sum(x)

trace_count_p = 0
for shape in [(10,), (10,), (20,), (10,), (5,)]:
    x = tf.ones(shape, dtype=tf.float32)
    before = len(problem_fn._list_all_concrete_functions_for_serialization())
    problem_fn(x)
    after = len(problem_fn._list_all_concrete_functions_for_serialization())
    if after > before:
        trace_count_p += 1
        print(f"  [RETRACE] shape={shape}")
    else:
        print(f"  [CACHED]  shape={shape}")

print(f"总计 retracing: {trace_count_p} 次 (每次约 50-200ms 额外开销)")

# === 修复版本：input_signature 锁定 ===
@tf.function(input_signature=[
    tf.TensorSpec(shape=[None], dtype=tf.float32)  # [None] = 任意一维
])
def fixed_fn(x):
    return tf.reduce_sum(x)

trace_count_f = 0
for shape in [(10,), (10,), (20,), (10,), (5,)]:
    x = tf.ones(shape, dtype=tf.float32) if shape[0] is not None else tf.ones([10])
    before = len(fixed_fn._list_all_concrete_functions_for_serialization())
    fixed_fn(x)
    after = len(fixed_fn._list_all_concrete_functions_for_serialization())
    if after > before:
        trace_count_f += 1
        print(f"  [RETRACE] shape={shape}")
    else:
        print(f"  [CACHED]  shape={shape}")

print(f"总计 retracing: {trace_count_f} 次")
```

**步骤三：AutoGraph 转换实验——Python 控制流 → tf.cond**

目标：理解 AutoGraph 如何在底层改写 Python 的 `if`/`for`。

```python
import tensorflow as tf

# === 示例 1: Python integer 的条件（tracing 阶段确定） ===
@tf.function
def py_if(x):
    if x.shape[0] > 5:  # 静态 shape → tracing 时确定
        return x * 2
    else:
        return x * 3

# x.shape[0]=10 > 5，所以 tracing 时走 if 分支
# 得到的图中只有 x*2，没有 x*3 分支！
print(py_if(tf.ones([10, 3])).numpy().flatten()[:3])  # [2. 2. 2.]

# === 示例 2: Tensor 值的条件（AutoGraph → tf.cond） ===
@tf.function
def tf_cond_if(x, threshold):
    return tf.cond(
        tf.reduce_sum(x) > threshold,
        lambda: x * 2,
        lambda: x * 3,
    )

# 用 AutoGraph 自动转换的等价写法：
@tf.function
def autograph_if(x, threshold):
    if tf.reduce_sum(x) > threshold:  # tf.Tensor 条件 → AutoGraph 自动转 tf.cond
        result = x * 2
    else:
        result = x * 3
    return result

x = tf.constant([1.0, 2.0, 3.0])
print(f"手动 tf.cond: {tf_cond_if(x, 5.0).numpy()}")     # sum=6>5 → ×2
print(f"AutoGraph:    {autograph_if(x, 5.0).numpy()}")   # sum=6>5 → ×2

# === 示例 3: Python for → tf.while_loop ===
@tf.function
def while_example(n):
    i = tf.constant(0)
    s = tf.constant(0)
    while i < n:  # AutoGraph 转换为 tf.while_loop
        s += i
        i += 1
    return s

print(f"AutoGraph while: {while_example(tf.constant(10)).numpy()}")  # 0+1+...+9 = 45

# 查看生成的图代码
print("\nAutoGraph 生成的代码 (通过 tf.autograph.to_code 查看):")
print(tf.autograph.to_code(autograph_if.python_function))
```

**步骤四：Retracing 诊断与修复实战**

目标：对一个实际频繁 retracing 的推理函数进行优化。

```python
import tensorflow as tf
import numpy as np

# 模拟一个"不稳定的推理函数"——输入 sequence length 不固定
@tf.function
def bad_inference(sequences):
    """问题：seq_length 变化 → retracing"""
    batch_size = tf.shape(sequences)[0]
    return tf.reduce_sum(sequences, axis=1)

# 诊断：不同长度的输入
print("=== 诊断 retracing ===")
for seq_len in [50, 100, 50, 200]:
    x = tf.random.normal([8, seq_len])
    num_before = len(bad_inference._list_all_concrete_functions_for_serialization())
    result = bad_inference(x)
    num_after = len(bad_inference._list_all_concrete_functions_for_serialization())
    retraced = num_after > num_before
    if retraced:
        print(f"  seq_len={seq_len:>4}: RETRACED (total={num_after})")
    else:
        print(f"  seq_len={seq_len:>4}: cached")

# 修复：用 variable-length input_signature
@tf.function(input_signature=[
    tf.TensorSpec(shape=[None, None], dtype=tf.float32)  # [任意batch, 任意length]
])
def good_inference(sequences):
    return tf.reduce_sum(sequences, axis=1)

print("\n=== 修复后 ===")
for seq_len in [50, 100, 50, 200]:
    x = tf.random.normal([8, seq_len])
    num_before = len(good_inference._list_all_concrete_functions_for_serialization())
    result = good_inference(x)
    num_after = len(good_inference._list_all_concrete_functions_for_serialization())
    retraced = num_after > num_before
    if retraced:
        print(f"  seq_len={seq_len:>4}: RETRACED (total={num_after})")
    else:
        print(f"  seq_len={seq_len:>4}: cached (total={num_after})")
```

### 3.3 `@tf.function` 最佳实践速查

| 场景 | 做法 | 理由 |
|------|------|------|
| **推理函数** | 始终加 `input_signature` | 防止序列长度/batch 变化导致 retracing |
| **训练函数** | 通常不需要 `input_signature` | `model.fit` 的数据管道已固定 batch shape |
| **含 Python 副作用的函数** | 避免在 `@tf.function` 中 `print()` / `list.append()` | 副作用只在 tracing 时执行一次 |
| **需要动态控制流** | 确保条件是 Tensor（非 Python 值） | AutoGraph 才能正确转换为 `tf.cond` |
| **含 NumPy 操作** | `tf.numpy_function` 或移出函数 | NumPy 操作强制退出图模式 |
| **调试** | `tf.config.run_functions_eagerly(True)` | 关闭图模式，恢复逐行调试 |

## 4. 项目总结

### 4.1 `@tf.function` 优势与陷阱

| 方面 | 优势 | 陷阱 |
|------|------|------|
| **性能** | Python 开销消除，算子融合，2-20x 加速 | 首次调用 tracing 有额外开销（50-500ms） |
| **控制流** | AutoGraph 自动转换 `if`/`for`/`while` | Python 值和 Tensor 值混合时转换失败 |
| **缓存** | 相同的 dtype+shape → 直接复用 Graph | shape 改变 → retracing（性能退化） |
| **副作用** | — | Python `print`/`append` 只在 tracing 执行 |
| **调试** | 生产环境高性能 | 需要 `tf.print` 代替 `print`，调试困难 |

### 4.2 适用场景

1. **推理服务优化**：`@tf.function(input_signature=[...])` 固定签名，消除 Python 开销
2. **自定义训练循环**：在 `train_step` 上加 `@tf.function` 自动加速
3. **复杂预处理**：将 tokenize/padding 等逻辑写入 `@tf.function`，避免 CPU-GPU 数据传输

**不适用场景**：
1. 一次性的探索/调试代码——tracing 成本大于收益
2. 包含大量 Python-only 逻辑的函数（如文件 I/O、数据库查询）
3. 输入类型极不稳定（每次都是不同的 Python 对象）的函数

### 4.3 注意事项

- **`input_signature` 不保证零 retracing**：如果签名写 `[None, 256]` 但实际传入 `(32, 128)`，还是 retracing
- **`tf.function` 与 `tf.Variable`**：Graph 内部创建的 Variable 是"静态"的——多次 tracing 不会重复创建，但修改只在图的第一次执行后生效
- **`experimental_relax_shapes=True`**：放宽 shape 匹配，减少 retracing（但可能损失一些图优化）

### 4.4 常见踩坑经验

1. **坑**：`@tf.function` 内 `assert x.shape[1] == 10` ——程序 hang 住。
   **根因**：Graph 模式下 `x.shape[1]` 可能是 `None`（动态），图编译时无法确定导致死循环。
   **解决**：用 `tf.debugging.assert_equal(tf.shape(x)[1], 10)` 替代 Python assert。

2. **坑**：`model(x)` 在 `@tf.function` 中结果与直接调用不同。
   **根因**：模型中的 `Dropout` 层的 `training` 参数在函数中默认为 `False`（推理态），但直接调用可能走了不同的 training 路径。
   **解决**：显式传参 `model(x, training=False)`，不要依赖默认值。

3. **坑**：`@tf.function` 内部调用 `tensor.numpy()` 报错。
   **根因**：`.numpy()` 会把 Tensor 从图模式"拉回"Python——图模式不支持这种越界操作。
   **解决**：用 `tf.py_function` 包装需要 Python 逻辑的部分；或等待结果出来后再 `.numpy()`。

### 4.5 思考题

1. 你在一个 `@tf.function` 中写了一个 `for i in range(tf.shape(x)[0])` 循环。这个循环会被 AutoGraph 转换成 `tf.while_loop` 吗？如果 `x.shape[0]` 在 tracing 时已知（例如 `(32, 256)`），AutoGraph 会怎么做？如果 shape 未知呢？

2. 你的推理函数对不同的 `batch_size` 需要做不同的 padding 逻辑。`input_signature` 能解决 batch_size 变化的问题吗？如果能，怎么写签名？如果不能，有什么替代方案？

### 4.6 推广计划提示

- **算法工程师**：推理函数一律加 `@tf.function(input_signature=[...])`，这是推理加速的最低成本优化
- **平台工程师**：在 Serving 部署时，考虑使用 `tf.function` + `jit_compile=True`（XLA）获得额外加速
- **测试工程师**：每个 `@tf.function` 必须有一个"Eager vs Graph 输出一致性"测试——差异应 < 1e-5
