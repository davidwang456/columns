## 1. 项目背景

容器平台通常采集 **stdout/stderr**。JSON 一行一条日志便于 **Loki / ELK** 解析，并与 **traceId** 关联。本章区分 **dev 人类可读** 与 **prod 机器可读**，避免「开发爽、运维哭」。

---

## 2. 项目设计：大师与小白的对话

**运维**：「我要能按 `pod`、`namespace`、`level` 过滤。」

**大师**：「JSON 里要有稳定字段；业务自定义键要**登记字典**，别每人发明一个 `msg2`。」

**小白**：「我用 `System.out.println` 最快。」

**大师**：「最快往往最贵：没有级别、没有上下文、无法统一脱敏。」

**安全**：「日志里能打 JWT 吗？」

**大师**：「**禁止**。只打 subject id 或 hash，且要经过评审。」

**测试**：「集成测试里日志级别怎么定？」

**大师**：「CI 用 `WARN` 降噪；排障临时开 `DEBUG` 类别。」

---

## 3. 知识要点

- 使用 `org.jboss.logging.Logger`，勿混用多套 facade。  
- `%prod.quarkus.log.console.json=true` 常见（以当前版本文档为准）。  
- MDC 与 OpenTelemetry 日志关联见第 30 章。

---

## 4. 项目实战

### 4.1 `pom.xml`

默认 `quarkus-arc` 已带日志；可加 `quarkus-logging-json`（若单独扩展，以官方 guide 为准——3.x 常通过 `quarkus-logging-json` 扩展启用 JSON）。

```xml
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-logging-json</artifactId>
</dependency>
```

### 4.2 `application.properties`

```properties
%dev.quarkus.log.console.json=false
%dev.quarkus.log.console.enable=true
%dev.quarkus.log.console.format=%d{yyyy-MM-dd HH:mm:ss,SSS} %-5p [%c{2.}] (%t) %s%e%n

%prod.quarkus.log.console.json=true
%prod.quarkus.log.console.enable=true
%prod.quarkus.log.level=INFO
```

### 4.3 代码

```java
package org.acme;

import jakarta.ws.rs.GET;
import jakarta.ws.rs.Path;
import org.jboss.logging.Logger;

@Path("/log-demo")
public class LogDemoResource {

    private static final Logger LOG = Logger.getLogger(LogDemoResource.class);

    @GET
    public String demo() {
        LOG.info("demo invoked");
        return "ok";
    }
}
```

### 4.4 Kubernetes：`Pod` 日志相关（无额外 YAML 必填）

确保容器进程以前台运行、日志走 stdout；示例 `Deployment` 片段仅标注：

```yaml
containers:
  - name: app
    image: registry.example.com/acme/app:1.0.0
    # 日志默认 stdout，无需 volume 除非落文件
```

## 5. 课堂实验

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | `%dev` 启动，访问 `/log-demo`，观察人类可读格式 | 一行一条、含级别 |
| 2 | `export QUARKUS_PROFILE=prod` 或使用 `-Dquarkus.profile=prod` 启动 | 输出为 JSON 行 |
| 3 | 用 `jq` 或在线 JSON 格式化解析一行 | 字段可被脚本提取 |
| 4 | （讨论）若集群用 Fluent Bit，需要额外 label 吗？ | 对齐平台规范 |

## 6. 项目总结

| 维度 | 说明 |
|------|------|
| **优点** | 开关化；与容器采集一致。 |
| **缺点** | 本地 JSON 难读；字段爆炸需治理。 |
| **适用场景** | 生产、共享集群。 |
| **注意事项** | PII 脱敏；日志量与费用。 |
| **常见踩坑** | 打印密钥；debug 忘记关。 |
