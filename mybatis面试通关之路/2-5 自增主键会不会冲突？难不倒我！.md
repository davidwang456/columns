## 好玩的谜底

> 小白：![为什么呢](img\为什么呢.gif)
>
> 扫地僧：自增主键取的地方不对，该是从插入的对象中读取，而不是返回值。
>
> ![关爱的眼神](img\关爱的眼神.jpg)
>
> ​	  返回值代表影响的行数，程序改成这样就好了！
>
> ```JAVA
> System.out.println("auto increment id:"+dto.getId());
> ```
>
> 小白：可以dto对象的id并没有被我赋值呀？
>
> 扫地僧：当你使用useGeneratedKeys属性时，Jdbc3KeyGenerator实例来给dto对象赋值。这个过程可以通过单步调试一步步进去查看.

其中，插入执行逻辑和自增ID在PreparedStatementHandler.java的update()方法：

```java
  @Override
  public int update(Statement statement) throws SQLException {
    PreparedStatement ps = (PreparedStatement) statement;
    ps.execute();
    int rows = ps.getUpdateCount();
    Object parameterObject = boundSql.getParameterObject();
    KeyGenerator keyGenerator = mappedStatement.getKeyGenerator();
    keyGenerator.processAfter(executor, mappedStatement, ps, parameterObject);
    return rows;
  }
```

ps.execute()执行插入操作，rows是update受影响的行数，返回值即是此值。然后调用Jdbc3KeyGenerator.java的processAfter()方法：

```java
  @Override
  public void processAfter(Executor executor, MappedStatement ms, Statement stmt, Object parameter) {
	  StackUtils.getStack();
    processBatch(ms, stmt, parameter);
  }

  public void processBatch(MappedStatement ms, Statement stmt, Object parameter) {
    final String[] keyProperties = ms.getKeyProperties();
    if (keyProperties == null || keyProperties.length == 0) {
      return;
    }
    try (ResultSet rs = stmt.getGeneratedKeys()) {
      final ResultSetMetaData rsmd = rs.getMetaData();
      final Configuration configuration = ms.getConfiguration();
      if (rsmd.getColumnCount() < keyProperties.length) {
        // Error?
      } else {
        assignKeys(configuration, rs, rsmd, keyProperties, parameter);
      }
    } catch (Exception e) {
      throw new ExecutorException("Error getting generated key or setting result to parameter object. Cause: " + e, e);
    }
  }
```

至此，谜底最终揭晓！自增键是从ResultSet中取出的：

```java
    protected void assign(ResultSet rs, Object param) {
      if (paramName != null) {
        // If paramName is set, param is ParamMap
        param = ((ParamMap<?>) param).get(paramName);
      }
      MetaObject metaParam = configuration.newMetaObject(param);
      try {
        if (typeHandler == null) {
          if (metaParam.hasSetter(propertyName)) {
            Class<?> propertyType = metaParam.getSetterType(propertyName);
            typeHandler = typeHandlerRegistry.getTypeHandler(propertyType,
                JdbcType.forCode(rsmd.getColumnType(columnPosition)));
          } else {
            throw new ExecutorException("No setter found for the keyProperty '" + propertyName + "' in '"
                + metaParam.getOriginalObject().getClass().getName() + "'.");
          }
        }
        if (typeHandler == null) {
          // Error?
        } else {
          Object value = typeHandler.getResult(rs, columnPosition);
          metaParam.setValue(propertyName, value);
        }
      } catch (SQLException e) {
        throw new ExecutorException("Error getting generated key or setting result to parameter object. Cause: " + e,
            e);
      }
    }
```

> 小白：我的理解是Jdbc3KeyGenerator负责给实体赋值，但为什么叫Jdbc3KeyGenerator？
>
> 扫地僧：说到这里，就能体现jdbc规范的重要性了。jdbc规范 3.0 (jsr-54)发布于2002年，从这个版本开始支持自增主键的获取：
>
> ![jdbc3](img\jdbc3.png)
>
> 最新的jdbc规范4.3.0（jsr-221）发布于2017年，每一个版本的变化都有功能的变化，jdbc-connector也会遵循jdbc的规范进行版本更新。故熟悉jdbc规范对于开发者也很重要。
>
> 小白：是不是意味着对要维护的古老程序来说，自增主键的获取要通过别的方式如SELECT LAST_INSERT_ID()，这种方式和通过getGeneratedKeys()这种方式有什么差别？
>
> 扫地僧：使用SELECT LAST_INSERT_ID()并发可能会出现问题，因为该方法值的作用域是一个连接Connection。因此，如果在同一连接Conection上发生其他Insert，则会覆盖该值。而getGeneratedKeys()方法的作用域是语句Statement实例，因此即使其他查询发生在同一连接上Conection，只要不是在同一语句实例上，也可以使用它。
>
> 小白：那并发情况下，自增主键会不会冲突呀？
>
> 扫地僧：插入时自增主键不会重复，但可能会出现所谓的MySQL auto_increment 空洞问题，空说无凭，咱们来演练一番！

## MySQL auto_increment 空洞问题

**1. 产生空洞问题最简单的方式--不提交**

```java

import java.io.IOException;
import java.io.Reader;

import org.apache.ibatis.session.SqlSessionFactoryBuilder;

public class AutoIncrementIdTest {

	public static void main(String[] args) throws IOException {
		testBatch1();
	   }
	
	public static void testBatch1() throws IOException {
	      Reader reader = Resources.getResourceAsReader("SqlMapConfig.xml");
	      SqlSessionFactory sqlSessionFactory = new SqlSessionFactoryBuilder().build(reader);		
	      SqlSession session = sqlSessionFactory.openSession();      
	      StudentMapper studentMapper=session.getMapper(StudentMapper.class);
	      StudentDTO dto=new StudentDTO();
	      dto.setFirstName("www");
	      dto.setLastName("david");
	      dto.setAge(20);

```

在session.commit(true);处打断点。

第一次全量执行

第二次执行到断点处，停止程序运行

第三次 全量执行

数据库信息student表信息如下：

```java
1	www	david	20
3	www	david	20
```

出现自增黑洞。

**2. 插入操作的同时，进行删除操作**

```java
package com.davidwang456.mybatis.autoincrement;

import java.io.IOException;
import java.io.Reader;

import org.apache.ibatis.io.Resources;
import org.apache.ibatis.session.SqlSession;
import org.apache.ibatis.session.SqlSessionFactory;
import org.apache.ibatis.session.SqlSessionFactoryBuilder;

public class BatchTest {

	public static void main(String[] args) throws IOException {
		testBatch();
	   }
	
	public static void testBatch() throws IOException {
	      Reader reader = Resources.getResourceAsReader("SqlMapConfig.xml");
	      SqlSessionFactory sqlSessionFactory = new SqlSessionFactoryBuilder().build(reader);		
	      SqlSession session = sqlSessionFactory.openSession(true);      
	      StudentMapper studentMapper=session.getMapper(StudentMapper.class);
	      Long start=System.currentTimeMillis();
	      int seq=0;
	      for(int i=0;i<10000;i++) {
		      StudentDTO dto=new StudentDTO();
		      seq=i+1;
		      dto.setFirstName("www"+seq);
		      dto.setLastName("david"+seq);
		      dto.setAge(20+seq%10);		      
		      if(i>1&&i%100==0) {
		    	  studentMapper.deleteById(i-1);
		      }else {
		    	  studentMapper.insertStudentInfo(dto);
		      }
	      }
	      long end = System.currentTimeMillis();
	      System.out.println("------BatchTest-------" + (start - end) + "ms---------------");
	      session.close();	
	}
}
```

测试结果

```java
select count(1) from student; //9899
select MAX(id)from student; //9901
```

说明空洞现象是存在的。

**3.并发插入并没有产生主键冲突**

这个是mysql的锁机制决定的，有兴趣的可以参考[官方文档](https://dev.mysql.com/doc/refman/8.0/en/innodb-auto-increment-handling.html)，实验可以通过同时运行多个批量插入操作查看。

在本机验证了 1w，10w，100w级别的插入，没有发现主键冲突。

## 总结

**1.学习JDBC规范的重要性**
	开发时总觉得规范这种东西离我们很远，但这些规范恰恰从根本上解决了我们的问题，熟悉这些规范可以让我们在开发的路上走的更远，效率更高。JDBC规范中可以得知，不仅仅时mysql，只要实现了jdbc3.0及以后规范的jdbc实现（包括但不仅限于oracle，sqlserver，postgresql等）都可以通过相同的形式获取到自增主键而不需要我们使用底层的方式，屏蔽了数据库底层的不一致性。

**2.学习Mysql官方文档的重要性**

​	[官方文档](https://dev.mysql.com/doc/connector-j/8.0/en/connector-j-usagenotes-last-insert-id.html)提供了获取自增主键的三种方式：

- ​    Statement.getGeneratedKeys()
- ​    SELECT LAST_INSERT_ID()
- ​	 Updatable ResultSets

并推荐最优的选项为Statement.getGeneratedKeys()。有兴趣的读者可以详细看看官方给的示例。
