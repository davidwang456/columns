# 第3章：配置文件语法与指令系统

> 源码关联：src/core/ngx_conf_file.c、src/core/ngx_conf_file.h、src/http/ngx_http_core_module.c

---

## 1. 项目背景

鲜果园的 Nginx 编译完成后，小李信心满满地打开了 `/usr/local/nginx/conf/nginx.conf`，准备配置第一个虚拟主机。然而，配置文件里的内容让他傻了眼：`worker_processes auto;`、`events { worker_connections 1024; }`、`http { server { location / { ... } } }`……这些花括号层层叠叠，有的指令在括号外面，有的在括号里面，还有的指令名字前面带个 `set`、`map`、`if`，简直像一门全新的编程语言。

更麻烦的是测试工程师小白。她想让 Nginx 根据用户的 Cookie 值把流量分发到不同的后端版本，于是在配置文件里写了一段 `if ($http_cookie ~* "version=A")`，结果 Nginx 启动时报错：`"if" directive is not allowed here`。小白气得直跺脚："这破配置文件，连个 if 语句都不让用！"

事实上，Nginx 的配置文件并非随意堆砌的文本，而是一门有着严格语法规则的**领域特定语言（DSL）**。它有自己的上下文（context）、指令类型、变量系统和继承规则。理解这套语法，是驾驭 Nginx 的必经之路。本章将带领鲜果园团队深入 Nginx 配置的底层世界，从源码层面理解配置文件是如何被解析、验证和生效的。

## 2. 项目设计

**场景**：鲜果园的会议室，投影仪上显示着一份报错信息，小白、小胖和大师正在讨论。

---

**小白**：（指着屏幕）大师，你看看，我就想在 server 里加个 if 判断，Nginx 非说不让。这 if 到底能在哪用？

**大师**：（喝了口茶）Nginx 的配置不是自由文本，它有严格的**上下文层级**。你直接把 if 写在了 server 的根级别，但 if 只能出现在 location 内部或者 server 的某些特定阶段。

**小胖**：（举手）等等，什么是"上下文"？

**大师**：上下文就是配置指令的"作用域"。Nginx 的配置大致分为这几个层级：

1. **main** —— 全局上下文，在最外层，影响整个 Nginx 实例
2. **events** —— 连接处理上下文，在 main 里面，控制事件模型
3. **http** —— HTTP 协议上下文，在 main 里面，包含所有 Web 相关配置
4. **server** —— 虚拟主机上下文，在 http 里面，一个 server 对应一个站点
5. **location** —— 请求路由上下文，在 server 里面，匹配具体的 URI

**小胖**：这不就是俄罗斯套娃吗？一个套一个！

**大师**：（笑）对。而且每个指令都有它的"合法居住地"。比如 `worker_processes` 只能住在 main，`listen` 只能住在 server，`proxy_pass` 只能住在 location。你把 `listen 80;` 写到 location 里，Nginx 启动就会报错。

**小白**：那指令分哪些类型呢？

**大师**：大致三种。第一种叫**简单指令**，一行一个，以分号结尾，比如 `worker_processes 4;`。第二种叫**块指令**，有花括号包裹一个上下文，比如 `server { ... }`。第三种叫**控制指令**，像 `if`、`return`、`rewrite`，它们会改变请求的处理流程。

**小胖**：那变量呢？我看配置里好多 `$` 开头的，像 `$uri`、`$args`。

**大师**：Nginx 的变量是一种**弱类型**的字符串，可以在配置运行时被求值。变量分两种：一种是**内置变量**，比如 `$uri`（当前请求路径）、`$host`（请求头 Host）、`$remote_addr`（客户端 IP）；另一种是**自定义变量**，通过 `set`、`map`、`geo` 指令创建。

**小白**：`set` 我懂，就是赋值。`map` 和 `geo` 是干嘛的？

**大师**：`map` 是**映射表**，适合做批量转换。比如你把用户代理字符串映射到设备类型：`map $http_user_agent $device { ~*iphone mobile; ~*android mobile; default desktop; }`。`geo` 是**地理映射**，根据客户端 IP 返回不同的值，常用于按地区分流。

**小胖**：那配置的继承规则呢？我在 http 里设了 `root /var/www;`，server 里还要再设一遍吗？

**大师**：这就要看**指令的继承规则**了。有些指令会向下继承，比如 `root`、`index`、`error_log`；有些不会继承，必须在每个上下文里重新声明，比如 `listen`、`server_name`。更复杂的是，有些指令在子上下文中声明后会**完全覆盖**父上下文（不是合并），比如 `access_log`；有些则是**追加**，比如 `location` 的匹配规则。

**小白**：这也太容易踩坑了吧……

**大师**：所以 Nginx 在启动时会做**配置合并（merge）**。每个模块在创建配置结构时，会分别生成 main、server、location 三个层级的配置块，然后通过 `ngx_conf_merge_*` 宏按规则合并。如果合并出错，启动就直接失败，绝不在运行时爆炸。

**小胖**：那配置文件是怎么被解析的呢？是一行一行读的吗？

**大师**：`ngx_conf_file.c` 里有一套完整的解析器。流程大概是：**词法分析（tokenize）** -> **语法分析（parse）** -> **配置合并（merge）** -> **有效性校验（init）**。tokenize 阶段把配置文件切成一个个 token（指令名、参数、分号、花括号）；parse 阶段根据模块注册的 `ngx_command_t` 表，验证指令的上下文、参数个数和类型；merge 阶段把父子配置合并；最后 init 阶段调用模块的初始化回调。

**小白**：原来启动时 `nginx -t` 做了这么多事……

**大师**：`nginx -t` 就是执行完整的解析和校验，但不真正启动进程。所以生产环境改配置前，一定要先用 `nginx -t` 验证！

## 3. 项目实战

### 环境准备

- **Nginx 版本**：1.31.0（已按第 2 章编译安装）
- **工作目录**：`/usr/local/nginx/conf/`
- **备份原配置**：`sudo cp /usr/local/nginx/conf/nginx.conf /usr/local/nginx/conf/nginx.conf.bak`

### 步骤一：编写一份多层嵌套配置

创建 `/usr/local/nginx/conf/nginx.conf`：

```nginx
# ========== main 上下文 ==========
user www-data;
worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

# ========== events 上下文 ==========
events {
    use epoll;
    worker_connections 4096;
    multi_accept on;
}

# ========== http 上下文 ==========
http {
    include       /usr/local/nginx/conf/mime.types;
    default_type  application/octet-stream;

    # 日志格式定义
    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';

    access_log /var/log/nginx/access.log main;

    # 全局 keepalive 设置
    keepalive_timeout 65;
    keepalive_requests 1000;

    # 使用 map 定义设备类型变量
    map $http_user_agent $device_type {
        ~*iphone        mobile;
        ~*android       mobile;
        ~*ipad          tablet;
        default         desktop;
    }

    # 使用 geo 按 IP 段定义区域变量
    geo $client_region {
        default          unknown;
        192.168.1.0/24   office;
        10.0.0.0/8       internal;
        127.0.0.1        local;
    }

    # ========== server 上下文：主站 ==========
    server {
        listen       80;
        server_name  www.xianguoyuan.com;
        root         /var/www/xianguoyuan;
        index        index.html index.htm;

        # 根据设备类型返回不同首页
        location = / {
            if ($device_type = mobile) {
                rewrite ^ /mobile/index.html last;
            }
            if ($device_type = tablet) {
                rewrite ^ /tablet/index.html last;
            }
            try_files /desktop/index.html =404;
        }

        # 静态资源（带版本号）
        location ~ ^/assets/v\d+/ {
            expires 1y;
            add_header Cache-Control "public, immutable";
            access_log off;
        }

        # API 代理
        location /api/ {
            proxy_pass         http://api_backend/;
            proxy_set_header   Host $host;
            proxy_set_header   X-Real-IP $remote_addr;
            proxy_set_header   X-Region $client_region;
            proxy_connect_timeout 5s;
            proxy_read_timeout    30s;
        }

        # 内部健康检查端点（不记录访问日志）
        location /health {
            access_log off;
            return 200 "OK\n";
            add_header Content-Type text/plain;
        }
    }

    # ========== server 上下文：管理后台 ==========
    server {
        listen       8080;
        server_name  admin.xianguoyuan.com;
        root         /var/www/admin;
        index        index.html;

        # 只允许内网访问
        location / {
            allow 10.0.0.0/8;
            allow 192.168.0.0/16;
            allow 127.0.0.1;
            deny  all;

            try_files $uri $uri/ /index.html;
        }
    }

    # ========== upstream 上下文 ==========
    upstream api_backend {
        server 10.0.1.10:8080 weight=5;
        server 10.0.1.11:8080 weight=5;
        server 10.0.1.12:8080 backup;
        keepalive 32;
    }
}
```

### 步骤二：语法验证

```bash
# 测试配置语法
sudo /usr/local/nginx/sbin/nginx -t -c /usr/local/nginx/conf/nginx.conf
```

预期输出：

```
nginx: the configuration file /usr/local/nginx/conf/nginx.conf syntax is ok
nginx: configuration file /usr/local/nginx/conf/nginx.conf test is successful
```

如果报错，根据错误信息定位行号并修正。常见错误：

| 错误信息 | 原因 | 解决 |
|---------|------|------|
| `unknown directive "xxx"` | 指令名拼写错误或模块未编译 | 检查拼写，确认模块已启用 |
| `"xxx" directive is not allowed here` | 指令出现在错误的上下文 | 将该指令移动到合法上下文 |
| `invalid number of arguments` | 参数个数不符合指令要求 | 查阅文档，补充或删减参数 |
| `no "events" section` | 缺少 events 上下文 | 添加 `events { ... }` |

### 步骤五：源码速览——配置解析器

打开 `src/core/ngx_conf_file.c`，查看核心解析函数：

```c
// src/core/ngx_conf_file.c
char *
ngx_conf_parse(ngx_conf_t *cf, ngx_str_t *filename)
{
    // 1. 打开配置文件
    // 2. 进入 tokenize 循环
    // 3. 对每个 token，查找对应的 ngx_command_t
    // 4. 调用 command->set 回调函数处理参数
    // 5. 如果遇到 {，递归进入子上下文
    // 6. 如果遇到 }，返回上一级上下文
}
```

查看 `ngx_command_t` 结构体：

```c
// src/core/ngx_conf_file.h
typedef struct ngx_command_s {
    ngx_str_t             name;       // 指令名称，如 "worker_processes"
    ngx_uint_t            type;       // 指令类型和允许出现的上下文
    char               *(*set)(ngx_conf_t *cf, ngx_command_t *cmd, void *conf);  // 解析回调
    ngx_uint_t            conf;       // 配置结构体偏移量
    ngx_uint_t            offset;     // 字段偏移量
    void                 *post;       // 后置处理指针
} ngx_command_t;
```

**代码注释**：
- `type` 字段通过位运算组合，如 `NGX_MAIN_CONF|NGX_DIRECT_CONF|NGX_CONF_TAKE1` 表示该指令只能在 main 上下文出现，且必须带 1 个参数
- `set` 回调是解析器的"执行引擎"，负责将配置文本转换为内存中的数据结构
- `conf` 和 `offset` 用于自动将解析结果写入模块的配置结构体，避免了大量手动的字段映射

## 4. 项目总结

### 优点与缺点

| 维度 | Nginx 配置 DSL | 传统 XML/JSON 配置 |
|------|--------------|-------------------|
| 可读性 | 高（层级清晰，接近自然语言） | 中（XML 冗长，JSON 无注释） |
| 校验时机 | 启动时完整校验，运行时不报错 | 部分在启动时，部分在运行时 |
| 灵活性 | 高（变量、map、geo、if、rewrite） | 低（通常是静态键值） |
| 学习曲线 | 较陡（上下文规则、继承规则复杂） | 平缓 |
| 动态更新 | 支持热重载（reload） | 通常需重启服务 |
| 错误提示 | 行号精确，但描述有时模糊 | 依赖具体实现 |

### 适用场景

1. **多站点托管**：通过 server 块轻松管理数百个虚拟主机
2. **A/B 测试与灰度发布**：利用 map、split_clients 实现动态分流
3. **多环境配置管理**：通过 include 和变量，一套配置适配开发/测试/生产
4. **复杂路由规则**：location 的正则匹配和 rewrite 规则满足绝大多数 Web 路由需求
5. **安全访问控制**：allow/deny、geo、auth_request 构建多层防线

**不适用场景**：
1. **需要复杂逻辑判断的配置**：Nginx 的 `if` 有诸多陷阱，复杂逻辑建议用 Lua/NJS
2. **运行时频繁变更的配置**：每次 reload 都有成本，高频变更建议用外部配置中心
3. **跨上下文共享数据**：Nginx 配置上下文之间数据隔离，需借助共享内存模块

### 注意事项

1. **if 指令是邪恶的**：Nginx 的 `if` 不是真正的条件语句，它在 rewrite 阶段执行，且有很多隐式行为。能用 `map` 替代的场景，绝不用 `if`。
2. **location 匹配顺序有陷阱**：`=` 精确匹配优先，`^~` 前缀匹配次之，正则 `~`/`~*` 再次之，普通前缀匹配最后。多个正则 location 时，按配置文件中的先后顺序匹配第一个。
3. **配置继承不是合并**：子上下文的某些指令会完全覆盖父上下文，而不是智能合并。比如 server 中的 `location` 如果声明了新的 `proxy_set_header`，会覆盖而不是追加父级的 header。

### 常见踩坑经验

**案例一：if 与 try_files 共用导致 404**
- **现象**：`location / { if ($uri ~* \.php$) { ... } try_files $uri =404; }`，PHP 请求全部 404
- **根因**：`if` 会创建隐式的内部 location，破坏 `try_files` 的执行流程
- **解决**：用 `location ~* \.php$` 替代 `if`，避免在 location 内混用 `if` 和 `try_files`

**案例二：map 变量在日志中为空**
- **现象**：`log_format` 中引用了 `$device_type`，但日志中始终显示空字符串
- **根因**：`map` 在 HTTP 上下文中定义，但日志格式在 `log_format` 中引用的变量如果未定义，默认输出空字符串
- **解决**：确保 `map` 定义在 `http` 上下文中，且变量名拼写正确；或者在 `map` 的 `default` 分支给出默认值

**案例三：配置 reload 后部分 location 不生效**
- **现象**：新增了一个 location，reload 后访问报错 404
- **根因**：新 location 被前面的正则 location 拦截了（正则 location 按配置文件顺序匹配）
- **解决**：调整 location 的顺序，或使用 `^~` 前缀匹配提高优先级
