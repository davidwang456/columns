# 第13章：日志系统与访问控制

> 源码关联：`src/http/modules/ngx_http_log_module.c`、`src/http/modules/ngx_http_access_module.c`

---

## 1. 项目背景

"鲜果园"的技术平台在经历了一年的高速迭代后，积累了越来越多的运维难题。最让团队头疼的是两个问题：日志混乱和访问失控。

先说日志。鲜果园目前有四十多台Nginx服务器分布在三个机房，每天产生的访问日志超过500GB。这些日志格式五花八门——有的服务器用的是默认的combined格式，有的运维同学随手改了格式，还有的为了排查问题临时加了一行调试日志却忘了删。当线上出现问题时，排查团队需要把多台机器的日志聚合到一起分析，但格式不统一导致日志清洗脚本频繁报错。更糟的是，安全团队需要对接SIEM（安全信息和事件管理）系统，要求日志必须是结构化的JSON格式，而现有的文本格式日志无法直接入库。

再说访问控制。上个月的运营活动中，鲜果园推出了"凌晨0点抢优惠券"的活动，结果引发了羊毛党的疯狂围攻。大量脚本在几秒内刷光了全部优惠券库存，正常用户根本抢不到。事后分析发现，Nginx层没有任何IP级别的限频和访问控制策略，所有请求长驱直入打到后端API。虽然后端有接口限流，但Nginx作为第一道防线却形同虚设。

小胖接到两个任务：一是统一全站Nginx日志格式，输出JSON结构化日志，方便ELK（Elasticsearch + Logstash + Kibana）系统采集和分析；二是在Nginx层建立访问控制体系，对内部管理后台做基础认证，对可疑IP段进行封禁，对敏感接口增加额外的认证校验。

小白在调研方案时发现了三个可用的Nginx模块：`ngx_http_log_module`负责日志记录，`ngx_http_access_module`提供IP级别的allow/deny，`ngx_http_auth_basic_module`提供HTTP Basic认证。但她对这几个模块的协作关系和边界条件还有疑问：auth_basic的密码文件怎么生成最安全？allow/deny和auth_basic在同一个location中的执行顺序是什么？自定义log_format时，如果变量在请求的某个阶段不存在，会输出什么？

大师在一次技术分享中指出："日志和访问控制是Nginx最基础也最容易被低估的能力。很多团队只把Nginx当作反向代理，却忽略了它在可观测性和安全边界上的价值。日志不仅仅是'记录下来'，它需要是可解析的、可聚合的、可告警的；访问控制也不仅仅是'允许或拒绝'，它需要是分层的、可审计的、与业务联动的。"

于是，本章的任务是：为鲜果园建立一套标准化的日志体系，以及一套多层次的访问控制策略。

## 2. 项目设计

周三下午，运维中心的监控大屏上滚动着各色曲线。小胖坐在工位上，面前开着三个终端窗口，分别ssh到不同的Nginx服务器上。小白端着一杯热茶走过来，笔记本上是ELK架构的部署文档。大师从会议室走出来，手里拿了一沓纸，上面画着访问控制流程图。

**小胖**（揉了揉眼睛）："大师，我把几台服务器的日志格式统一改成了combined，但安全团队说还是不够，他们要把日志直接送进Elasticsearch，必须是JSON格式。我手动拼了一个JSON字符串当log_format，结果双引号老是冲突，愁死我了。"

**大师**（把流程图放在桌上）："手动拼JSON字符串确实容易出问题，特别是变量值本身含有引号、反斜杠、换行符的时候。Nginx从1.11.8版本开始原生支持JSON格式的日志输出，你可以用`escape=json`参数让Nginx自动处理特殊字符的转义。"

**小白**（放下茶杯）："我查了一下，`escape`参数有`default`、`json`、`none`三种模式。默认模式对双引号和非ASCII字符做转义，json模式会额外处理控制字符和反斜杠，确保输出是合法的JSON。"

**大师**（点点头）："没错。而且JSON日志格式有一个巨大的优势：它是自描述的。传统格式如combined，解析时需要依赖正则表达式或grok模式来提取字段；而JSON日志中每个字段都有明确的名字，Logstash可以直接用`json`过滤器解析，不需要复杂的模式匹配。"

**小胖**（来了精神）："那是不是我可以这样写："

```nginx
log_format json_analytics escape=json
    '{"time":"$time_iso8601","remote_addr":"$remote_addr",'
    '"request":"$request","status":$status,'
    '"request_time":$request_time,'
    '"upstream_response_time":"$upstream_response_time"}';
```

**大师**（看了一眼）："方向对了，但还有优化空间。第一，建议把`$http_host`和`$server_name`都加上，方便区分请求是通过哪个域名进来的；第二，`$upstream_response_time`如果请求没有走到上游（比如直接返回404），这个变量是空的，在JSON中会变成空字符串，如果你的分析系统期望它是数字类型，后续需要做类型转换；第三，可以加一个`$request_id`，用Nginx 1.11.0引入的`$request_id`变量，给每个请求一个唯一的UUID，这对于全链路追踪非常关键。"

**小白**（补充道）："另外，日志中还可以加入用户身份相关的变量，比如`$http_x_user_id`，这样可以把访问日志和用户行为关联起来。但要注意，这些自定义头部变量需要前端或网关提前注入。"

**小胖**（快速记下）："好，我明白了。那访问控制这块，我有几个场景不太确定怎么做。"

**大师**："你说说看。"

**小胖**："第一个场景是内部管理后台，只有公司内网IP能访问。这个是不是直接用`allow`和`deny`就行？"

**大师**："对，这是最简单的场景。`ngx_http_access_module`提供了`allow`和`deny`两个指令，按顺序匹配。注意它们的执行顺序是从上到下的，一旦匹配到`allow`或`deny`，就会立即生效。"

**小胖**："第二个场景是管理后台除了IP限制，还需要用户名密码登录。我听说`auth_basic`可以做到，但我不清楚它和`allow/deny`在一起怎么协作。"

**大师**："这是典型的**多层次访问控制**场景。Nginx的访问控制是分阶段的：`allow/deny`在`NGX_HTTP_ACCESS_PHASE`执行，`auth_basic`也在同一个阶段，但模块之间有固定的执行顺序。`ngx_http_access_module`通常排在`auth_basic`前面，这意味着IP白名单会先被检查——如果IP不在白名单内，直接返回403，根本不会走到密码验证。"

**小白**（追问）："那如果我想让'公司内网IP免密码访问，外网IP需要密码'呢？"

**大师**（赞许地看了小白一眼）："这需要用到`satisfy`指令。`satisfy any`表示多个访问控制条件满足任意一个即可通过；`satisfy all`表示必须满足所有条件。你的场景可以这样配："

```nginx
location /admin/ {
    satisfy any;           # 满足任意一个条件即可
    
    allow 192.168.0.0/16;  # 公司内网免密
    deny all;              # 其他IP需要继续检查
    
    auth_basic "Admin Area";
    auth_basic_user_file /etc/nginx/.htpasswd;
}
```

"在这个配置中，内网IP匹配`allow`，直接通过；外网IP被`deny all`拦截，但由于`satisfy any`，Nginx会继续检查`auth_basic`。外网用户如果提供了正确的用户名密码，也能访问。"

**小胖**（恍然大悟）："原来`satisfy`是控制多个条件之间'与/或'关系的开关！"

**小白**（又抛出一个问题）："大师，我们还有一个更复杂的场景。有些敏感接口（比如发放优惠券、查询用户隐私数据），不仅要认证，还要调用后端的统一认证服务做校验。这个用`auth_basic`就不够了吧？"

**大师**："这种场景应该使用`auth_request`模块。它的工作方式是：当请求到达时，Nginx先向指定的内部location发一个子请求，由这个子请求去调用后端认证服务。如果子请求返回2xx，Nginx允许原请求继续；如果返回401/403，Nginx把子请求的响应返回给客户端。"

"`auth_request`的强大之处在于，认证逻辑完全由后端控制。后端可以检查Token有效性、用户权限、请求签名、甚至风控规则，然后把结果通过HTTP状态码告诉Nginx。这比在Nginx层写死密码文件灵活得多。"

**小胖**（挠挠头）："这好像门口保安通过对讲机问物业中心'这个人能不能进'，而不是自己翻名册查。"

**大师**（微笑）："非常形象的比喻。`auth_basic`是翻名册，`auth_request`是打电话问物业。"

**小白**（最后问了一个问题）："日志的写入性能呢？我听说磁盘IO是Nginx的瓶颈之一，如果日志量很大，会不会影响请求处理？"

**大师**："Nginx的日志写入是异步缓冲的，`access_log`指令支持`buffer=`和`flush=`参数。默认情况下，日志先写入内存缓冲区，满了或到了刷新时间才批量刷盘。如果日志量极大，还可以考虑关闭磁盘日志，直接通过`syslog`发送到远程日志服务器，或者使用`ngx_http_lua_module`把日志异步发送到Kafka。"

**小胖**（看着满桌子的笔记）："日志和访问控制看起来是小事，但要做成生产级的方案，里面门道真不少。"

**大师**（收起流程图）："基础不牢，地动山摇。现在我们把所有方案落地。"

### 步骤二：编写后端认证服务

创建一个Flask应用，模拟统一认证服务：

```python
# auth_service.py
from flask import Flask, request, jsonify
import base64
import hmac
import hashlib

app = Flask(__name__)

# 模拟Token黑名单和权限数据库
TOKEN_BLACKLIST = set()
ADMIN_TOKENS = {"sk-xianguoyuan-2024-admin"}
USER_TOKENS = {"sk-xianguoyuan-2024-user"}

@app.route('/auth/verify')
def verify():
    """Nginx auth_request子请求目标"""
    auth_header = request.headers.get('Authorization', '')
    
    # 检查Bearer Token
    if auth_header.startswith('Bearer '):
        token = auth_header[7:]
        if token in TOKEN_BLACKLIST:
            return jsonify({"error": "Token revoked"}), 403
        if token in ADMIN_TOKENS:
            # 在响应头中附加用户信息，供上游使用
            response = jsonify({"user": "admin", "role": "administrator"})
            response.headers['X-Auth-User'] = 'admin'
            response.headers['X-Auth-Role'] = 'administrator'
            return response, 200
        if token in USER_TOKENS:
            response = jsonify({"user": "user", "role": "customer"})
            response.headers['X-Auth-User'] = 'user'
            response.headers['X-Auth-Role'] = 'customer'
            return response, 200
    
    return jsonify({"error": "Unauthorized"}), 401

@app.route('/api/coupon')
def coupon():
    """模拟敏感接口：发放优惠券"""
    return jsonify({
        "message": "优惠券发放成功",
        "coupon_id": "CPN-2024-8888",
        "amount": 50
    })

@app.route('/api/user/profile')
def user_profile():
    """模拟用户隐私接口"""
    return jsonify({
        "user_id": "U12345",
        "name": "张三",
        "phone": "138****8888",
        "address": "上海市浦东新区"
    })

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080)
```

启动服务：

```bash
python auth_service.py &
```

### 步骤三：编写Nginx配置文件

```nginx
user  nginx;
worker_processes  auto;

events {
    worker_connections  1024;
}

http {
    include       mime.types;
    default_type  application/octet-stream;

    # === JSON结构化日志格式 ===
    log_format json_combined escape=json
        '{"timestamp":"$time_iso8601",'
        '"request_id":"$request_id",'
        '"remote_addr":"$remote_addr",'
        '"remote_user":"$remote_user",'
        '"host":"$http_host",'
        '"method":"$request_method",'
        '"uri":"$uri",'
        '"query":"$args",'
        '"status":$status,'
        '"bytes_sent":$bytes_sent,'
        '"request_time":$request_time,'
        '"upstream_response_time":"$upstream_response_time",'
        '"upstream_addr":"$upstream_addr",'
        '"http_referer":"$http_referer",'
        '"http_user_agent":"$http_user_agent",'
        '"ssl_protocol":"$ssl_protocol",'
        '"ssl_cipher":"$ssl_cipher",'
        '"http_x_forwarded_for":"$http_x_forwarded_for",'
        '"http_x_user_id":"$http_x_user_id"}';

    # === 传统格式日志（用于兼容旧系统） ===
    log_format traditional '$remote_addr - $remote_user [$time_local] '
                           '"$request" $status $body_bytes_sent '
                           '"$http_referer" "$http_user_agent" '
                           'rt=$request_time uct="$upstream_connect_time"';

    # 主访问日志使用JSON格式
    access_log  logs/access.json.log  json_combined buffer=32k flush=5s;
    
    # 错误日志
    error_log   logs/error.log  warn;

    # === 定义认证子请求目标 ===
    upstream auth_backend {
        server 127.0.0.1:8080;
        keepalive 32;
    }

    server {
        listen       80;
        server_name  localhost;
        root         /data/xian果园/static;
        index        index.html;

        # === 场景1：静态资源，记录日志但无访问限制 ===
        location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
            expires 30d;
            add_header Cache-Control "public, immutable";
            # 继承server级别的access_log配置
        }

        # === 场景2：内部管理后台 - IP白名单 + Basic认证 ===
        location /admin/ {
            satisfy any;                        # 满足任一条件即可
            
            allow 192.168.0.0/16;              # 公司内网免密访问
            allow 10.0.0.0/8;
            deny all;                           # 非白名单IP继续检查
            
            auth_basic "鲜果园管理后台";
            auth_basic_user_file /etc/nginx/.htpasswd;
            
            # 自定义日志：单独记录管理后台访问
            access_log logs/admin.access.json.log json_combined;
            
            proxy_pass http://127.0.0.1:8080;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }

        # === 场景3：敏感接口 - auth_request后端认证 ===
        location /api/ {
            # 先通过子请求调用认证服务
            auth_request /auth_verify;
            
            # 认证成功后，把认证响应头透传给后端
            auth_request_set $auth_user $upstream_http_x_auth_user;
            auth_request_set $auth_role $upstream_http_x_auth_role;
            
            proxy_pass http://127.0.0.1:8080;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Auth-User $auth_user;
            proxy_set_header X-Auth-Role $auth_role;
            
            # 认证失败时的自定义错误页
            error_page 401 = @auth_required;
            error_page 403 = @auth_forbidden;
        }

        # 认证子请求的内部location
        location = /auth_verify {
            internal;                          # 禁止外部直接访问
            proxy_pass http://auth_backend/auth/verify;
            proxy_pass_request_body off;       # 不转发请求体
            proxy_set_header Content-Length "";
            proxy_set_header X-Original-URI $request_uri;
            proxy_set_header X-Original-Method $request_method;
        }

        # 自定义错误响应
        location @auth_required {
            return 401 '{"error":"认证失败，请提供有效的Token"}\n';
            add_header Content-Type application/json;
        }

        location @auth_forbidden {
            return 403 '{"error":"权限不足，禁止访问"}\n';
            add_header Content-Type application/json;
        }

        # === 场景4：IP黑名单 - 封禁已知恶意IP ===
        location /coupon/ {
            # 封禁已知羊毛党IP段
            deny 103.21.244.0/22;    # 示例：某IDC段
            deny 104.16.0.0/12;      # 示例：某云厂商段
            allow all;               # 其他IP允许
            
            proxy_pass http://127.0.0.1:8080;
            proxy_set_header Host $host;
        }

        # === 场景5：健康检查接口 - 关闭日志 ===
        location /health {
            access_log off;          # 健康检查不产生日志，避免刷屏
            return 200 '{"status":"healthy","server":"nginx"}\n';
            add_header Content-Type application/json;
        }

        # 默认location
        location / {
            try_files $uri $uri/ =404;
        }
    }
}
```

**配置要点说明**：

1. **`escape=json`**：确保所有变量值中的特殊字符（引号、反斜杠、换行、控制字符）都被正确转义，输出合法的JSON。

2. **`buffer=32k flush=5s`**：日志缓冲区32KB，或每5秒强制刷新一次。减少磁盘IO，提高吞吐量。

3. **`satisfy any`**：实现"IP白名单免密，外网需要密码"的分层访问控制。

4. **`auth_request` + `auth_request_set`**：子请求认证完成后，将认证响应中的头部变量（如`X-Auth-User`）捕获到Nginx变量，再透传给上游后端。

5. **`error_page`自定义错误响应**：把Nginx默认的HTML错误页替换为JSON格式，保持API接口的一致性。

6. **`internal`**：`/auth_verify` location标记为内部，禁止客户端直接访问。

### 步骤六：源码级理解

打开`src/http/modules/ngx_http_log_module.c`，查看日志写入的核心逻辑：

```c
static ngx_int_t
ngx_http_log_handler(ngx_http_request_t *r)
{
    u_char                   *line, *p;
    size_t                    len;
    ngx_uint_t                i, l;
    ngx_http_log_t           *log;
    ngx_http_log_op_t        *op;
    ngx_http_log_buf_t       *buffer;
    ngx_http_log_loc_conf_t  *lcf;

    lcf = ngx_http_get_module_loc_conf(r, ngx_http_log_module);
    log = lcf->logs->elts;

    for (l = 0; l < lcf->logs->nelts; l++) {
        if (ngx_time() == log[l].disk_full_time) {
            continue;  // 磁盘已满，跳过写入
        }

        len = 0;
        op = log[l].format->ops->elts;
        for (i = 0; i < log[l].format->ops->nelts; i++) {
            if (op[i].len == 0) {
                len += op[i].getlen(r, op[i].data);
            } else {
                len += op[i].len;
            }
        }

        // 分配缓冲区并组装日志行
        line = ngx_pnalloc(r->pool, len);
        if (line == NULL) {
            return NGX_ERROR;
        }

        p = line;
        for (i = 0; i < log[l].format->ops->nelts; i++) {
            p = op[i].run(r, p, &op[i]);  // 执行每个变量的run函数
        }

        // 写入文件或缓冲区
        ngx_http_log_write(r, &log[l], line, p - line);
    }

    return NGX_OK;
}
```

这段代码揭示了Nginx日志模块的核心设计：`log_format`中的每个变量被编译成`ngx_http_log_op_t`操作数组。在请求结束时，`run`函数依次执行，把变量值写入缓冲区。这种设计使得日志格式化非常高效——无需解析字符串模板，直接调用预编译的操作函数。

再看`src/http/modules/ngx_http_access_module.c`中的访问控制逻辑：

```c
static ngx_int_t
ngx_http_access_handler(ngx_http_request_t *r)
{
    ngx_int_t                rc;
    ngx_uint_t               i;
    ngx_http_access_loc_conf_t  *alcf;

    alcf = ngx_http_get_module_loc_conf(r, ngx_http_access_module);

    if (alcf->rules == NULL) {
        return NGX_DECLINED;  // 没有配置规则，跳过
    }

    // 获取客户端真实IP（考虑X-Forwarded-For等）
    rc = ngx_http_access_real_ip(r, alcf->rules);

    if (rc == NGX_DECLINED) {
        return NGX_DECLINED;  // 没有匹配到规则
    }

    if (rc == NGX_OK) {
        return NGX_OK;        // allow匹配成功
    }

    return NGX_HTTP_FORBIDDEN;  // deny匹配成功
}
```

注意`NGX_HTTP_ACCESS_PHASE`阶段的handler返回值处理逻辑：返回`NGX_OK`表示允许访问，`NGX_HTTP_FORBIDDEN`表示拒绝，`NGX_DECLINED`表示本模块不做决策，交给下一个handler。这与`auth_basic`和`auth_request`的返回值在`satisfy`指令下被综合判断，形成了Nginx灵活的多层访问控制体系。

### 优劣对比：Nginx层访问控制 vs 应用层访问控制

| 维度 | Nginx层访问控制 | 应用层访问控制 |
|------|----------------|---------------|
| **执行时机** | 请求最早阶段，无效请求不进入后端 | 请求已进入后端，消耗应用资源 |
| **性能开销** | 极低，C语言实现 | 较高，需启动应用进程处理 |
| **灵活程度** | 受限于Nginx指令，规则较固定 | 图灵完备，任意业务逻辑 |
| **维护成本** | 配置化管理，变更秒级生效 | 需修改代码、测试、部署 |
| **审计能力** | 可记录到access_log | 可记录更细粒度的业务上下文 |
| **Token验证** | 需借助auth_request或Lua | 原生支持，灵活解析JWT等 |
| **适用场景** | IP封禁、基础认证、简单规则 | 复杂权限、RBAC、业务风控 |

### 适用场景

1. **统一日志网关**：所有请求经过Nginx，天然适合作为全站日志的统一采集点。
2. **IP级别安全防护**：基于地理位置、IP信誉库、已知攻击源进行快速封禁。
3. **内部系统隔离**：管理后台、监控面板、API文档等仅对内网或授权人员开放。
4. **静态资源缓存**：结合日志分析热点资源，优化CDN缓存策略。
5. **多租户流量隔离**：基于Host或IP区分不同租户，输出独立的访问日志。

### 不适用场景

1. **复杂RBAC权限控制**：涉及角色、资源、操作的多维权限判断，应在应用层实现。
2. **细粒度API限流**：基于用户ID、接口路径、时间窗口的复杂限流，建议使用专用网关（如Kong、Envoy）。
3. **实时风控拦截**：需要分析用户行为序列、设备指纹、关联图谱的场景，Nginx层无法胜任。

### 注意事项与警告

- **变量空值处理**：Nginx变量未定义时为空字符串`""`。在JSON日志中，`"upstream_response_time":""`虽然合法，但分析系统可能需要处理空字符串到NULL的转换。
- **`auth_request`超时**：子请求如果长时间无响应，会阻塞原请求。建议配置`proxy_connect_timeout`和`proxy_read_timeout`在子请求location中。
- **密码文件权限**：`.htpasswd`文件必须设置为`640`权限，且属主为nginx运行用户，避免密码泄露。
- **日志磁盘空间**：JSON日志比传统格式大约膨胀30%（因为字段名重复），需预留足够磁盘空间，并配置logrotate轮转。
- **`satisfy`的隐性风险**：`satisfy any`配合多个控制条件时，如果某个条件配置错误导致"永远通过"，安全策略会被绕过。建议定期审计访问控制配置。

### 生产环境三大踩坑实录

**案例一：JSON日志中的换行符导致Logstash解析失败**

某公司在日志格式中包含了`$request_body`，用于记录POST请求的表单数据。当用户上传包含换行符的内容时，Nginx默认会把换行符原样输出到日志，导致一行JSON被拆成多行，Logstash的JSON过滤器解析报错。解决方案：使用`escape=json`参数，让Nginx自动把换行符转义为`\n`。

**案例二：`auth_request`子请求阻塞导致雪崩**

某公司在高峰期部署了`auth_request`对接统一认证服务，但认证服务响应变慢后，Nginx的worker进程大量阻塞在子请求上，导致正常请求也无法处理，形成级联故障。根因是子请求没有配置独立的超时和连接池限制。解决方案：为auth子请求配置短超时（如`proxy_read_timeout 2s`），并配置`limit_req`保护认证接口。

**案例三：allow/deny顺序错误导致内网被误封**

某运维同学复制粘贴了一段deny规则到location顶部，把`deny all`写在了`allow 192.168.0.0/16`之前。结果Nginx按顺序匹配，第一条`deny all`就直接拒绝了所有请求，包括内网。服务中断20分钟后才定位到问题。解决方案：严格遵循`allow`在前、`deny all`在后的顺序，配置变更前先用`nginx -t`检查语法，并在测试环境验证。

### 进阶思考题

**思考题1**：在`ngx_http_log_module.c`中，每个`log_format`变量被编译为`ngx_http_log_op_t`结构体数组，包含`getlen`、`run`两个函数指针。请分析：为什么Nginx在日志记录时需要先遍历一遍所有op计算总长度`len`，再分配内存、第二次遍历执行`run`写入数据？为什么不能像`sprintf`那样直接写入固定大小的缓冲区？这种设计的优势和代价分别是什么？

*提示*：考虑Nginx变量值的长度在运行时才确定（如`$request_uri`的长度随请求变化），以及单次内存分配vs多次追加realloc的性能权衡。
