# 第15章：Nginx 日常运维与故障排查

> 源码关联：`src/os/unix/ngx_process_cycle.c`、`src/http/ngx_http_upstream.c`

---

## 1. 项目背景

"星海教育"平台每天有三段明显流量高峰：早上上课前、午休刷题时、晚上直播课开始前。业务团队已经把Nginx部署成标准入口层，但最近一个月线上故障频繁：有时是`502 Bad Gateway`，有时是`504 Gateway Timeout`，还有一批`499`让产品经理误以为服务器不稳定。更麻烦的是，故障发生时值班同学经常靠经验乱试，恢复时间长、复盘资料也不完整。

公司CTO提出了明确要求：建立一套可执行、可复用、可交接的Nginx故障排查SOP。不是"谁懂谁来救火"，而是任何值班工程师拿到文档都能按步骤定位问题。

小胖负责值班，他最痛的点是"看到报错但不知道先查哪儿"；小白负责稳定性建设，担心"排障步骤太散，交接后会走样"；大师的目标是把常见故障变成标准化流程，把个人经验沉淀成团队能力。

本章就以生产常见五类问题为主线：

- 502、504、499、413、444；
- 配置热重载失败；
- worker CPU 100%或连接耗尽；
- 日志与系统命令联动排查；
- 输出一份值班可直接执行的SOP模板。

---

## 2. 项目设计

晚高峰前，三人在值班室做演练。

**小胖**："我每次看到`502`都先重启Nginx，能好一阵，但总感觉治标不治本。"

**小白**："`502`只是结果，不是原因。可能是后端挂了，也可能是端口不通、协议不匹配、超时配置不合理。"

**大师**："排障要像急诊分诊，先判断'是否可用'，再定位'哪一层坏了'。"

**技术映射**：错误码是症状，分层排查才是诊断。

---

**小胖**："那`502`和`504`怎么快速区分？"

**大师**："一句话：`502`更像'连接或协议失败'，`504`更像'等待上游超时'。看`error.log`关键字最直接。"

**小白**："比如`connect() failed (111: Connection refused)`多半是`502`；`upstream timed out`通常是`504`。"

**大师**："对，别上来就改超时。先确认上游是否真的慢，还是根本没起来。"

**技术映射**：日志关键字优先于直觉操作。

---

**小白**："`499`经常很多，这是不是服务端问题？"

**大师**："`499`是客户端主动断开连接，常见于移动网络抖动、前端超时过短、用户手动取消。它不一定代表Nginx异常，但数量突增要关注：可能是接口太慢，用户等不及。"

**小胖**："也就是说，`499`要联动看`request_time`和前端超时配置。"

**大师**："没错。排障不是看单指标，是看因果链。"

**技术映射**：`499`是体验信号，常用于发现慢请求和链路超时不匹配。

---

**小胖**："还有个问题，配置改完`reload`报错，业务又催上线，最慌。"

**大师**："任何变更都要遵循三步：`nginx -t`语法检查、灰度机`reload`验证、再全量发布。`-t`不过，绝不进下一步。"

**小白**："那worker CPU 100%怎么查？"

**大师**："先`top`定位PID，再`strace -p`看系统调用模式；如果一直在`epoll_wait`说明空转少，若频繁文件IO或网络重试，再结合日志定位热点URI。"

**技术映射**：系统工具负责定位"资源维度"，业务日志负责定位"请求维度"。

## 3. 项目实战

### 环境准备

- Nginx 1.24+
- 示例上游：`127.0.0.1:9000`
- Linux工具：`ss`、`lsof`、`strace`、`top`、`awk`
- 日志路径：`/var/log/nginx/access.log`、`/var/log/nginx/error.log`

---

### 步骤一：建立日常巡检命令基线

步骤目标：形成值班首屏信息。

```bash
# 1) 查看编译参数与版本
nginx -V

# 2) 配置语法检查
nginx -t

# 3) 当前连接状态
ss -s
ss -lntp | rg nginx

# 4) 最近错误日志
tail -n 50 /var/log/nginx/error.log
```

判断标准：

- `nginx -t`必须`syntax is ok`且`test is successful`；
- 监听端口存在且进程正常；
- 错误日志无持续性致命报错。

---

### 步骤二：502故障SOP（连接失败类）

步骤目标：10分钟内定位`502`主要原因。

典型日志：

```text
connect() failed (111: Connection refused) while connecting to upstream
no live upstreams while connecting to upstream
```

排查流程：

```bash
# A. 上游服务是否存活
curl -i http://127.0.0.1:9000/health

# B. 端口监听是否存在
ss -lntp | rg 9000

# C. Nginx到上游网络是否可达
nc -zv 127.0.0.1 9000

# D. 检查upstream配置拼写与端口
rg "upstream|proxy_pass" /etc/nginx -n
```

常见根因：

- 上游服务宕机或未监听目标端口；
- `proxy_pass`写错协议或端口；
- 上游被防火墙/安全组拦截。

---

### 步骤三：504故障SOP（超时类）

步骤目标：区分"上游慢"与"超时配置过紧"。

典型日志：

```text
upstream timed out (110: Connection timed out) while reading response header
```

排查流程：

```bash
# A. 统计慢请求URI
awk '$9 ~ /504/ {print $7}' /var/log/nginx/access.log | sort | uniq -c | sort -nr | head

# B. 对热点URI直接打上游
time curl -i "http://127.0.0.1:9000/api/slow-endpoint"

# C. 检查超时参数
rg "proxy_connect_timeout|proxy_read_timeout|proxy_send_timeout" /etc/nginx -n
```

调优建议：

- 先优化上游SQL/依赖调用，再考虑放宽`proxy_read_timeout`；
- 对特定慢接口单独配置超时，不要全局无限放大。

---

### 步骤四：499故障SOP（客户端取消类）

步骤目标：判断是否由服务慢引发客户端放弃。

```bash
# 最近1小时499比例（示例，按日志时间格式自行调整）
awk '$9==499 {c++} END {print c}' /var/log/nginx/access.log

# 关联请求耗时分布（需要日志中含$request_time）
awk '$9==499 {print $NF}' /var/log/nginx/access.log | sort -n | tail
```

分析方向：

- 若499集中在长耗时请求，优先优化接口性能；
- 若499集中在弱网地区，评估CDN与前端超时重试策略；
- 若499突然与某次发版同步，排查前端请求超时参数变化。

---

### 步骤五：413与444故障SOP（请求大小/主动拒绝）

`413 Request Entity Too Large`常见于上传接口：

```nginx
location /upload/ {
    client_max_body_size 20m;
    proxy_pass http://app_backend;
}
```

验证：

```bash
curl -F "file=@big.zip" http://example.com/upload/
```

`444`是Nginx主动断连，常用于黑名单策略：

```nginx
location / {
    if ($http_user_agent ~* "malicious-bot") {
        return 444;
    }
}
```

注意：`444`对客户端表现为连接被直接关闭，业务方要知道这是安全策略不是服务崩溃。

---

### 步骤六：配置错误与热重载SOP

步骤目标：保障变更安全发布。

```bash
# 1. 语法检查
nginx -t

# 2. 平滑重载
nginx -s reload

# 3. 复核master/worker状态
ps -ef | rg "nginx: master|nginx: worker"
```

建议流程：

1. 先在预发环境`-t`和功能验证；  
2. 生产先灰度1台，再全量；  
3. 保留上个版本配置快照，便于快速回滚。

---

### 步骤七：CPU 100%与连接耗尽SOP

步骤目标：定位资源异常来源。

```bash
# A. 找高CPU worker
top -H -p $(pgrep -d',' -f "nginx: worker")

# B. 看系统调用行为
strace -p <worker_pid> -tt -f

# C. 查看连接数
ss -ant | awk 'NR>1{print $1}' | sort | uniq -c
```

排查思路：

- CPU高且日志集中某URI：多半是热点接口或恶意流量；
- `worker_connections`逼近上限：调整连接数并优化keepalive；
- TIME_WAIT激增：评估上游短连接过多。

---

### 步骤八：一份可执行值班SOP模板

可落地模板（建议贴到团队Wiki）：

1. **确认影响面**：错误码比例、受影响URI、时间窗口；  
2. **判断层级**：Nginx层、网络层、上游层、客户端层；  
3. **执行分支SOP**：502/504/499/413/444对应脚本；  
4. **临时止血**：限流、熔断、降级页、扩容；  
5. **恢复验证**：核心接口成功率与延迟回归；  
6. **复盘沉淀**：根因、改进项、演练计划。

---

### 测试验证

```bash
# 基础连通性验证（示例）
curl -i http://127.0.0.1:8080/health
```

验证要点：
- 关键接口返回码符合预期（2xx/4xx/5xx与设计一致）；
- 关键日志字段完整（如 request_id、upstream 耗时、状态码）；
- 在小流量压测下无异常错误峰值。

## 4. 项目总结

本章的目标不是背命令，而是建立排障方法论：先分类、再定位、后止血，最后复盘。把`502/504/499`这些高频故障流程化后，团队可以显著缩短MTTR（平均恢复时间），并降低值班对个人经验的依赖。

### 优点与缺点（SOP化运维）

| 维度 | 优点 | 缺点 |
|------|------|------|
| 效率 | 排障路径固定，响应更快 | 早期需要投入整理成本 |
| 质量 | 降低误操作，减少二次事故 | 规则过死会限制灵活性 |
| 交接 | 新人更快接手值班 | 需持续维护文档新鲜度 |
| 复盘 | 数据与步骤可追溯 | 无监控支撑时效果受限 |

### 适用场景

- 7x24值班团队；
- 多环境、多服务的网关层运维；
- 频繁发版、故障敏感业务。

### 不适用场景

- 单机实验环境、低频访问内部系统；
- 无日志、无监控、无变更流程的临时系统。

### 三个踩坑经验

1. **只盯Nginx不看上游**：误把数据库慢查询当网关故障。  
2. **未先`nginx -t`就reload**：配置错误导致服务不可用。  
3. **盲目调大超时**：掩盖上游性能问题，延迟雪崩更严重。  

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
