# 第36章：Upstream 源码——连接池与负载均衡

> 源码关联：src/http/ngx_http_upstream.c、src/http/ngx_http_upstream_round_robin.c
> 批次说明：第 4 批（精修版）

**关键词摘要**：upstream peer、动态权重、连接池管理、负载均衡、慢节点治理

---

## 1. 项目背景

后端节点性能差异持续扩大，固定权重导致慢节点长期拖累整体。团队尝试基于实时延迟做动态权重实验。

进入这一阶段后，Nginx 不再只是“转发工具”，而是稳定性、性能与工程效率的共同支点。仅靠参数经验已经不足以支撑复杂系统，必须把“设计假设、实验数据、故障演练、回滚策略”绑定成一个闭环。
本章沿着“问题定义 -> 方案取舍 -> 实验验证 -> 上线守护”的路径展开，确保方案可复制、可复盘、可演进。

---

从知识路径上看，本章与上一章《HTTP 请求生命周期完整源码链路》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：性能分析会上，慢节点把整条链路 P99 拖到 5s，小白展示了 upstream peer 的延迟分布直方图，三个颜色分别对应三个后端。

---

**小胖**：（嘬着奶茶）这不就跟外卖平台派单一样嘛——有的骑手电动车没电了还不停给他派单，整个区域的配送时间都被拉高了。

**小白**：目前 upstream 用的 round-robin，每个后端权重固定。但实际容器的 CPU 分配不一样，有的节点已经在 thrashing 了还被源源不断地发请求。

**大师**：我们拆解 upstream 的工作流：①ngx_http_upstream_init 初始化上下文；②peer.init 选择后端节点（默认 ngx_http_upstream_init_round_robin）；③ngx_http_upstream_connect 发起连接；④ngx_http_upstream_send_request 发送请求头；⑤ngx_http_upstream_process_header 处理响应头；⑥ngx_http_upstream_process_upstream 读取响应体。这就像外卖调度：接单→分派→骑手取餐→配送→确认送达。

**小胖**：那能不能让系统自动给骑手打分——送得慢的少派单？

**大师**：这就是**动态权重**的思路。我们可以基于 peer->current_weight 做动态衰减——每次选择节点后降低其权重，周期恢复。进一步可以采集实时延迟（avg_latency_ms），延迟过高时临时降低 effective_weight。

---

**小白**：那如果选中的节点连接不上呢？只是降权还是立刻剔除？

**大师**：upstream 的 failover 机制通过 max_fails 和 fail_timeout 控制。好比骑手连续 3 次配送超时就暂时冻结 30 秒——这就是 max_fails=3、fail_timeout=30s 的含义。ngx_http_upstream_next 根据上游返回的状态码或连接错误，决定是否将 peer 标记为不可用，在 fail_timeout 后尝试恢复。

**小胖**：那就是"三振出局"规则——连着三单出问题，先凉一会儿再说。

**小白**：连接池呢？每次请求都新建 TCP 连接太浪费。

**大师**：keepalive 连接池机制让 upstream 在请求结束后不关闭连接，放回池中（ngx_http_upstream_keepalive 模块）。下次请求同一 upstream 时优先复用已有连接，省去三次握手和慢启动。这就像给好评骑手安排专属停车位——送完一单不用回去排队抢位置，直接接下一单。

**技术映射**：**peer.init** 决定负载均衡策略（round-robin/least_conn/ip_hash），**ngx_http_upstream_connect** 发起连接复用 keepalive 池，**ngx_http_upstream_next** 实现故障转移，**动态权重**可基于实时延迟修正选择概率。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等级 Linux 发行版）
- Nginx：1.31.0（建议保留 debug 能力）
- 工具：`curl`、`wrk`、`ss`、`jq`
- 按章补充：数据库/DNS/WebSocket/K8s/profiling 工具

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：构建可运行、可验证、可回滚的最小基线。

```nginx
# 伪代码
# if peer->avg_latency_ms > threshold:
#     peer->effective_weight = max(1, peer->effective_weight - 1);
# else:
#     peer->effective_weight = min(peer->weight, peer->effective_weight + 1);
```

**运行结果（预期）**：
- 配置检查通过；
- 关键路径连通；
- `error.log` 无持续异常。

**可能遇到的坑**：仅做功能验证，不做压力与异常验证，导致上线后才暴露边界问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用数据而非主观感受评估改动价值。

```bash
wrk -t8 -c1500 -d60s http://127.0.0.1:8080/pay
watch -n1 "curl -s http://127.0.0.1:8080/upstream_stats"
```

**建议观测指标**：
- 吞吐：QPS/TPS
- 时延：P95/P99
- 可靠性：5xx、超时率、重试率
- 资源：CPU、内存、活动连接

**可能遇到的坑**：压测时间太短、场景太单一。建议至少覆盖平峰/高峰/故障三种情形。

### 步骤三：故障注入与回滚演练

**步骤目标**：确保故障发生时可快速止损。

```bash
# 应用变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线单个上游节点；
2. 注入后端延迟或错误码；
3. 叠加突发流量验证尾延迟。

**本章实战目标**：理解 upstream peer 选择机制并完成动态权重实验验证。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 稳定性 | 有明确的演练与回滚机制 | 前期建设成本增加 |
| 性能 | 可持续、可量化优化 | 依赖监控与压测体系 |
| 团队协作 | 结论标准化、可传承 | 对执行纪律要求高 |

### 适用场景

1. 高峰明显、故障成本高的核心业务。
2. 网关承载治理能力的平台化系统。
3. 希望把经验运维升级为工程化运维的团队。

**不适用场景**：
- 低流量 demo 或短生命周期项目。
- 缺少基础观测能力、无法形成验证闭环的团队。

### 注意事项

1. 无基线不优化：先量化现状再改动。
2. 单变量优先：保障结论可归因。
3. 回滚先行：上线前先验证止损路径。

### 常见踩坑经验

- 只看平均值，不看尾延迟。
- 只做成功路径，不做故障演练。
- 变更记录缺失，导致后续无法复盘。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
