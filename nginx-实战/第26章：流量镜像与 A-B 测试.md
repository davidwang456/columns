## 1. 项目背景

推荐模型准备灰度，新模型离线评估很好，但团队担心线上真实流量分布与离线样本差异巨大。为了降低发布风险，架构组决定先镜像再分桶，做到“先看效果，再放流量”。

这类问题在中级阶段非常典型：指标表面好看，但业务体验在关键节点失真。团队过去依赖“经验调参”，现在必须切换为“指标驱动 + 故障演练 + 可回滚”的交付模式。
本章会沿着“真实业务痛点 -> 方案博弈 -> 逐步落地 -> 验证闭环”来推进，确保这套方案不仅能跑，还能在高峰与异常中站得住。

---

从知识路径上看，本章与上一章《WebSocket 与长连接管理》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：推荐模型灰度上线评审会，离线 AUC 曲线很漂亮，但团队对线上真实效果心里完全没底。

---

**小胖**：（吃着薯片）流量镜像不就是在主路旁边修条辅路，把车流复制一份过去跑跑看吗？这有啥难的？

**小白**：但镜像流量影响到主链路怎么办？镜像请求的延迟、错误、资源消耗会不会拖慢正常请求？

**大师**：（在白板上画了两条并行的箭头）小胖的修路比喻很直观。mirror 模块就是在主请求处理完后，“异步”复制一份同样的请求发到镜像后端。注意，这个复制是 fire-and-forget——主请求不会等镜像请求返回，镜像端超时或报错也不影响主链路。所以小白担心的“镜像拖慢主路”在正常情况下不会发生，但如果镜像端响应太慢，会占用 worker 连接池，极端情况下可能间接影响主链路。建议给镜像后端单独部署，别和生产环境混用。

**小白**：那 split_clients 呢？它怎么保证同一个用户始终落入同一实验组？

**大师**：split_clients 用的是**一致性哈希**——以 `$request_id` 或 `$remote_addr` 为输入，算出一个 0 到 2^32-1 的哈希值，再按配置的比例映射到不同分组。只要输入不变，哈希值就不变，同一个用户始终落同一个桶。这就好比食堂按学号尾数分配窗口——1-3 号去川菜窗口，4-6 号去粤菜窗口，7-9 号去湘菜窗口，不管你来几次，尾数不变，窗口就不变。

**大师**：

**技术映射**：`mirror` 模块实现请求复制（影子流量），`mirror_request_body` 控制是否复制请求体；`split_clients` 通过一致性哈希实现请求级分组路由，支持按比例分配流量到不同上游，是 A/B 测试的基础设施。

---

**小胖**：那灰度上线呢？先让 1% 的用户尝新，没问题再 5%，再 20%，这不跟新菜品先让几个老顾客试吃一样？

**大师**：（点头）A/B 测试和灰度发布的区别就在这儿。A/B 测试是“同时对比”——两个版本同时在线上运行，看哪个指标好。灰度发布是“逐步放量”——先小范围验证，再逐步扩大。两者可以配合使用：先用 split_clients 分 5% 流量到新版本做 A/B 对比，确认指标正向后再逐步调高比例直到全量。

**小白**：但步长和间隔怎么定？从 5% 直接跳到 50% 风险还是太大。

**大师**：灰度步长没有一个固定值，但有一个通用原则：**每次放量不超过当前已验证规模的 3-5 倍**。比如 5% 验证 24 小时无异常，下一步可以放到 20%；20% 验证通过后再放到 50%、100%。同时要设好“一键切回”——在 Nginx 里就是改一下 split_clients 的比例，reload 即可回滚，整个过程秒级完成。

**大师**：**技术映射**：灰度发布的 Nginx 方案——`split_clients` + 多 upstream 组 + 监控看板。步长遵循“3-5 倍放大”原则，每一步留足观察窗口。回滚策略：修改 split 比例后 `nginx -s reload`，秒级生效无需重启。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等发行版）
- Nginx：1.31.0
- 工具：`curl`、`wrk`、`ss`、`jq`
- 附加工具：第 24 章建议 `mysql` / `dig`，第 25 章建议 `websocat`

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：先建立可运行基线，避免“边改边猜”。

```nginx
split_clients "$request_id$remote_addr" $ab_group {
    5%  canary;
    *   stable;
}
server {
    listen 8080;
    location /recommend {
        mirror /_mirror_recommend;
        proxy_set_header X-Ab-Group $ab_group;
        proxy_pass http://rec_$ab_group;
    }
    location = /_mirror_recommend {
        internal;
        proxy_method POST;
        proxy_set_header X-Mirror-Request 1;
        proxy_pass http://rec_shadow;
    }
}
```

**运行结果（预期）**：
- 配置校验通过；
- 关键接口连通；
- `error.log` 无持续性异常。

**可能遇到的坑**：仅用功能联调代替压测与异常演练，导致上线后才暴露问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用可比较的数据判断方案优劣。

```bash
for i in {1..50}; do curl -s http://127.0.0.1:8080/recommend > /dev/null; done
rg "X-Ab-Group" /var/log/nginx/access.log
```

**建议关注指标**：
- 吞吐：QPS / TPS
- 延迟：P95、P99
- 稳定性：错误率、超时率、重试率
- 资源：活动连接、CPU、内存

**可能遇到的坑**：压测时间太短，掩盖连接池、缓存、GC 等“慢热型问题”。建议最少跑 5~10 分钟稳定段。

### 步骤三：故障注入与回滚演练

**步骤目标**：验证系统在异常下仍可控。

```bash
# 触发变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线一个后端节点；
2. 人工增加后端延迟（如 tc 注入）；
3. 叠加突发流量观察尾延迟。

**本章实战目标**：落地镜像 + A/B 双轨发布，形成低风险上线流程。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 可用性 | 有明确保护与回滚机制，异常可控 | 建设初期流程更重 |
| 性能 | 能在真实指标上做持续优化 | 需要持续压测资源 |
| 协作 | 跨团队共享同一评估标准 | 对执行纪律要求更高 |

### 适用场景

1. 高峰波动明显、对可用性要求高的业务。
2. 网关承担统一流量治理职责的平台。
3. 希望建立“可复盘、可复制”优化流程的团队。

**不适用场景**：
- 单机低流量且无明显增长预期的项目。
- 缺少监控与压测基础设施、无法闭环验证的团队。

### 注意事项

1. 指标先行：没有基线就没有优化结论。
2. 变更最小化：一次只改一个关键变量。
3. 回滚可执行：任何优化都必须自带止损路径。

### 常见踩坑经验

- 只追求命中率/QPS，不看业务正确性。
- 缺少异常注入，线上首遇故障才发现边界。
- 变更记录缺失，后续团队无法继承经验。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
