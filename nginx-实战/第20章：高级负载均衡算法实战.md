## 1. 项目背景

图片与推荐两类业务共用同一网关。图片请求强缓存，适合稳定落点；推荐请求耗时波动大，适合动态分配。团队长期使用 round robin，结果是缓存命中差、热点节点过载、平均成本上升。

在复盘会上，运维团队给出一个关键结论：以前“能跑就行”的配置策略，在中级阶段会迅速失效。因为系统面对的已经不是单点问题，而是“流量规模 + 协议复杂度 + 组织协作成本”的复合压力。
因此本章不只讲参数含义，还会给出一套可复用的方法：**先建基线，再做单变量实验，最后用故障注入验证边界**。这套方法会贯穿中级篇后续章节。

---

从知识路径上看，本章与上一章《Upstream 高级配置与故障转移》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：监控大屏上，图片服务和推荐服务的流量曲线像两条打架的蛇——一条请求全扎到同一台机器，另一条来回跳导致缓存命中率惨不忍睹。

---

**小胖**：（吃着薯片）不就是轮着来嘛，一人一个号，排到谁是谁。round robin 都用了这么多年了，有啥问题？

**小白**：问题有两类。一是像图片这种强缓存的业务，每次请求落到不同节点，缓存就白建了——好比每次去图书馆都坐不同位置，你的书永远不能放在桌上。二是像推荐这种计算密集的业务，每个请求耗时差异很大，round robin 均匀分配请求数但不均匀分配负载，导致有的节点忙死、有的闲死。

**大师**：（在白板上画了三张示意图）所以需要“因业务施策”。图片要的是**一致性哈希**——同一个 URL 永远落到同一个节点，缓存命中率直线上升。推荐要的是**最小连接数**——哪个节点当前处理的请求最少就扔给谁，实现真正的负载均衡。

**小胖**：哦，那哈希不就是把菜按种类分到不同窗口——凉菜窗口永远卖凉菜，热菜窗口永远卖热菜，我去固定的窗口就不用重新排队了呗？

---

**小白**：一致性哈希我理解，但如果加节点或者缩节点，大部分 key 的映射会不会也发生变化？那可得大量缓存失效了。

**大师**：这就是一致性哈希比普通哈希聪明的地方。它把节点和 key 都映射到一个环上，增删节点时只影响环上相邻的一小段 key，而不是全量 rehash。

**小胖**：就像食堂新开一个窗口，只把隔壁窗口的一部分客人分过去，其他窗口的人完全不受影响。

**技术映射**：Nginx 的 `hash $request_uri consistent` 使用一致性哈希环，默认每个节点在环上有 160 个虚拟节点（可通过 hash_again 调节），分散更均匀。增删节点时，只有环上相邻区间的 key 重新映射——缓存失效比例从普通 hash 的 1/N 降为 1/(N²)。

---

**小白**：那 least_conn 有没有坑？如果所有请求处理时间差不多，它跟 round robin 区别大吗？

**大师**：区别不大——好比所有菜都是 3 分钟出锅，哪个窗口空闲都差不多。但如果有慢请求卡住一个 worker，least_conn 会少往那里派活，而 round robin 照派不误。

**小胖**：那 sticky cookie 又是什么？跟食堂发会员卡一样，你刷脸就知道你爱吃什么菜？

**大师**：（笑）差不多。sticky cookie 是 Nginx 给客户端种一个 cookie，后续请求带着这个 cookie 就直接路由到同一台后端。适用于那些不能做全局 session 共享的老系统——你第一次去哪个窗口，以后就固定去那个窗口。

**技术映射**：负载均衡选型口诀——**有状态用 hash，有波动用 lc，有 session 用 sticky，没要求用 rr**。实际项目中常按 URI 路径分流：静态资源用 hash 做缓存亲和，动态计算用 least_conn，老旧单体用 sticky 保兼容。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04 / Rocky Linux 9（任选其一）
- Nginx：1.31.0（建议开启 debug）
- 工具：`wrk`、`curl`、`ss`、`jq`、`grpcurl`（第21章需要）

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：建立最小可运行配置

**步骤目标**：让核心路径先稳定跑通，保证后续实验有可信起点。

```nginx
upstream img_hash {
    hash $request_uri consistent;
    server 10.0.2.11:7001;
    server 10.0.2.12:7001;
    server 10.0.2.13:7001;
}
upstream rec_lc {
    least_conn;
    server 10.0.2.21:8001;
    server 10.0.2.22:8001;
    server 10.0.2.23:8001;
}
server {
    listen 8080;
    location /img/ { proxy_pass http://img_hash; }
    location /rec/ { proxy_pass http://rec_lc; }
}
```

**运行结果（预期）**：
- `nginx -t` 输出 syntax is ok / test is successful
- 健康检查接口返回 200
- `error.log` 无持续性报错

**可能遇到的坑**：配置能通过但链路没通，多数是上游地址、证书路径或防火墙规则未对齐。

### 步骤二：做基线压测与指标采集

**步骤目标**：获得“优化前”与“优化后”的可比数据。

```bash
wrk -t4 -c1000 -d60s http://127.0.0.1:8080/img/banner.png
wrk -t6 -c1200 -d60s http://127.0.0.1:8080/rec/list
```

**建议记录指标**：
- 吞吐：QPS / RPS
- 延迟：P50、P95、P99
- 稳定性：5xx 比例、连接失败率、重试比例
- 资源：CPU、内存、活动连接数

**可能遇到的坑**：压测机成为瓶颈，导致测试结果误判。建议压测端与被测端分离，且至少跑 2-3 轮取中位值。

### 步骤三：故障注入与回滚演练

**步骤目标**：确认异常场景下系统仍可控，发布后具备止损能力。

```bash
# 触发配置变更
sudo nginx -t && sudo nginx -s reload
# 回滚（示例）
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 人工下线一个上游节点。
2. 将一个上游延迟注入到 1~2s。
3. 模拟突发流量（瞬时并发提升 2~3 倍）。

**本章实战目标**：按业务特征匹配负载均衡算法，产出“场景-算法-指标”的决策表。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 方案代价 |
|---|---|---|
| 性能收益 | 能针对性改善吞吐与尾延迟 | 需要压测与监控支撑 |
| 稳定性 | 有故障注入与回滚兜底 | 发布流程更严格 |
| 协作效率 | 开发/测试/运维可共享同一结论 | 对变更纪律要求更高 |

### 适用场景

1. 日常有流量峰谷，且峰值抖动明显的系统。
2. 网关承担统一入口、对稳定性要求高的平台。
3. 需要沉淀标准化运维流程的中大型团队。

**不适用场景**：
- 单机低流量 demo 项目。
- 暂无监控与压测能力、难以形成验证闭环的团队。

### 注意事项

1. 先统一指标口径，再讨论优化效果。
2. 单次只改一个关键变量，方便归因。
3. 每次上线都保留可在 1 分钟内执行的回滚路径。

### 常见踩坑经验

- 只看平均延迟，忽略 P99 与错误率。
- 基线记录不完整，导致结论不可复验。
- “一把梭”改太多参数，上线后无法定位问题来源。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
