## 1. 项目背景

数据库迁移窗口只有 20 分钟，应用团队不能改任何连接串。为了实现无感切流，平台团队选择 Nginx stream 作为四层中转。但首次演练失败：TCP 配得通，UDP 超时不断。

这类问题在中级阶段非常典型：指标表面好看，但业务体验在关键节点失真。团队过去依赖“经验调参”，现在必须切换为“指标驱动 + 故障演练 + 可回滚”的交付模式。
本章会沿着“真实业务痛点 -> 方案博弈 -> 逐步落地 -> 验证闭环”来推进，确保这套方案不仅能跑，还能在高峰与异常中站得住。

---

从知识路径上看，本章与上一章《限流熔断——limit_req 与 limit_conn》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：数据库迁移方案评审会，白板上画着迁移拓扑，20 分钟窗口倒计时让所有人神经紧绷。

---

**小胖**：（嘬着酸奶）四层代理不就转发个数据包吗，跟网线延长线似的，一头插上去另一头就通了，能有什么花样？

**小白**：但上次演练 UDP 代理一直超时，TCP 却没问题。同样都是 stream 代理，为什么一个行一个不行？

**大师**：（在白板上画了 OSI 七层模型，圈出 4 和 7）小胖说对了一半——四层代理确实是“透明的数据搬运工”。但你先得分清：七层代理（HTTP）是在看信里写了什么，读完内容再决定寄给谁；四层代理（TCP/UDP）只看信封上的地址，不拆信。所以四层更快、资源消耗更少，但也做不了内容路由。

**小白**：那 UDP 为什么比 TCP 难配？

**大师**：（画了两个箭头）TCP 有连接概念——三次握手、状态维护、按序传输，Nginx 知道一个连接什么时候开始、什么时候结束。UDP 是无状态的，发完一个包就完了，Nginx 要额外通过 `proxy_responses` 告诉它“等待几个响应包就算一次会话结束”。上次演练失败就是因为没配 `proxy_responses`，Nginx 一直在等，直到超时。

**大师**：

**技术映射**：`stream` 上下文配置四层代理，`proxy_pass` 指向上游。TCP 代理靠连接状态管理，UDP 代理需额外配置 `proxy_responses` 标识会话结束条件，`proxy_timeout` 控制空闲连接超时。

---

**小胖**：那 SSL 终结呢？四层也能做 HTTPS 吗？是不是跟把快递外包的透明胶带撕开再封上一样？

**大师**：（笑）你这比喻很传神。SSL termination 就是在 Nginx 这一层把加密包拆开（解密），再决定往哪转——可以转发明文到后端（减轻后端 SSL 计算压力），也可以重新加密再转发。四层做 SSL 终结跟七层不同：七层能看到 HTTP 请求里的 Host 头来做路由，四层只能看 IP 和端口。但 Nginx 有一个杀手特性叫 `ssl_preread`——它在不解密完整握手的情况下，通过 Client Hello 里的 SNI 字段提前判断客户端要访问哪个域名，然后路由到不同的后端。

**小白**：这倒适合数据库和 Redis 这种不能用 HTTP 协议的中间件的灰度迁移场景。

**大师**：没错。MySQL、Redis、MongoDB 都走 TCP，没法用七层代理做精细路由。但配合 `ssl_preread` 和 `map` 指令，你甚至能做到“根据域名分流到不同数据库集群”，而客户端零感知。

**大师**：

**技术映射**：`ssl_preread` + `map` 实现四层 SNI 路由；`proxy_ssl` 系列指令配置四层 SSL/TLS 终结。四层代理性能优于七层但路由能力受限，适用于数据库、DNS、LDAP 等非 HTTP 协议的无感迁移。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等发行版）
- Nginx：1.31.0
- 工具：`curl`、`wrk`、`ss`、`jq`
- 附加工具：第 24 章建议 `mysql` / `dig`，第 25 章建议 `websocat`

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：先建立可运行基线，避免“边改边猜”。

```nginx
stream {
    upstream mysql_cluster {
        server 10.0.4.11:3306 max_fails=2 fail_timeout=5s;
        server 10.0.4.12:3306 max_fails=2 fail_timeout=5s;
    }
    server {
        listen 3307;
        proxy_connect_timeout 1s;
        proxy_timeout 30s;
        proxy_pass mysql_cluster;
    }
    upstream dns_cluster {
        server 10.0.4.21:53;
        server 10.0.4.22:53;
    }
    server {
        listen 5353 udp;
        proxy_timeout 5s;
        proxy_responses 1;
        proxy_pass dns_cluster;
    }
}
```

**运行结果（预期）**：
- 配置校验通过；
- 关键接口连通；
- `error.log` 无持续性异常。

**可能遇到的坑**：仅用功能联调代替压测与异常演练，导致上线后才暴露问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用可比较的数据判断方案优劣。

```bash
mysql -h127.0.0.1 -P3307 -uroot -p -e "select 1"
dig @127.0.0.1 -p 5353 example.com
```

**建议关注指标**：
- 吞吐：QPS / TPS
- 延迟：P95、P99
- 稳定性：错误率、超时率、重试率
- 资源：活动连接、CPU、内存

**可能遇到的坑**：压测时间太短，掩盖连接池、缓存、GC 等“慢热型问题”。建议最少跑 5~10 分钟稳定段。

### 步骤三：故障注入与回滚演练

**步骤目标**：验证系统在异常下仍可控。

```bash
# 触发变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线一个后端节点；
2. 人工增加后端延迟（如 tc 注入）；
3. 叠加突发流量观察尾延迟。

**本章实战目标**：完成 TCP/UDP 双协议代理并形成可复用排障清单。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 可用性 | 有明确保护与回滚机制，异常可控 | 建设初期流程更重 |
| 性能 | 能在真实指标上做持续优化 | 需要持续压测资源 |
| 协作 | 跨团队共享同一评估标准 | 对执行纪律要求更高 |

### 适用场景

1. 高峰波动明显、对可用性要求高的业务。
2. 网关承担统一流量治理职责的平台。
3. 希望建立“可复盘、可复制”优化流程的团队。

**不适用场景**：
- 单机低流量且无明显增长预期的项目。
- 缺少监控与压测基础设施、无法闭环验证的团队。

### 注意事项

1. 指标先行：没有基线就没有优化结论。
2. 变更最小化：一次只改一个关键变量。
3. 回滚可执行：任何优化都必须自带止损路径。

### 常见踩坑经验

- 只追求命中率/QPS，不看业务正确性。
- 缺少异常注入，线上首遇故障才发现边界。
- 变更记录缺失，后续团队无法继承经验。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
