# 第2章：源码目录解析与编译安装

> 源码关联：auto/configure、auto/options、auto/sources、auto/modules、src/core/、src/event/、src/http/、src/stream/、src/mail/、src/os/

---

## 1. 项目背景

鲜果园的架构师小李在上一章向团队讲解了 Nginx 的 master-worker 模型，CTO 老张听后拍了板："就用 Nginx 了！"但问题接踵而至——团队里没有人能看懂 Nginx 的源码。测试工程师小白在 GitHub 上 clone 了代码，面对密密麻麻的 C 文件一脸茫然；后端新人小胖直接双击了 nginx.c，结果被满屏的宏定义吓退；运维老王更关心的是：这玩意儿怎么编译？能不能只开我需要的模块，把用不上的统统关掉？

事实上，Nginx 的源码结构并非无迹可寻。Igor Sysoev 在设计之初就遵循了清晰的模块化原则：`src/core/` 存放基础设施，`src/event/` 处理网络事件，`src/http/` 负责七层协议，`src/stream/` 负责四层代理……而 `auto/` 目录下的 Shell 脚本则构成了 Nginx 独特的编译系统。理解这套目录结构和编译机制，是深入 Nginx 源码的第一步——就像医生手术前必须先熟悉人体解剖图。

本章将带着鲜果园团队一起"解剖"Nginx 源码。我们会从业务需求出发，通过一场对话厘清目录职责，再亲手执行 configure、make、make install 的完整流程，最终编译出一个精简高效的 Nginx 二进制文件。

---

## 2. 项目设计

**场景**：鲜果园技术部的工位区，小胖、小白和大师围在一台显示器前，屏幕上打开着 Nginx 源码目录。

---

**小胖**：（指着 src/ 目录）大师，这源码目录跟迷宫似的，我从哪开始看啊？

**小白**：我数了一下，src/ 下面有六个子目录，每个里面又有几十个 .c 文件。有没有一张"地图"能告诉我：我想看反向代理的实现，该点哪个文件？

**大师**：（打开终端，tree 了一下 src/）当然有。你们看，Nginx 的源码目录就是一张分层地图。

**大师**：最底层是 `src/core/`，这是 Nginx 的"基建队"。内存池（ngx_palloc）、字符串（ngx_string）、数组（ngx_array）、哈希表（ngx_hash）、红黑树（ngx_rbtree）、配置文件解析器（ngx_conf_file）……这些基础工具都在这里。不管上层是 HTTP 还是 Stream，都要调用 core 里的基础设施。

**小胖**：哦，就像公司的行政部和财务部，所有人都要用。

**大师**：对。再往上是 `src/event/`，这是"交通指挥中心"。epoll、kqueue、select 这些 IO 多路复用技术的实现都在这里。Nginx 的高并发能力，核心就靠这一层的事件循环。

**小白**：那 `src/http/` 呢？

**大师**：这是"业务大厅"，负责处理 HTTP 协议。你们常用的反向代理（ngx_http_proxy_module）、负载均衡（ngx_http_upstream_round_robin）、Rewrite（ngx_http_rewrite_module）、Gzip（ngx_http_gzip_filter_module）……所有 HTTP 相关的模块都在这里。`src/http/modules/` 子目录里就有五六十个模块文件。

**小胖**：我懂了！那 `src/stream/` 呢？

**大师**：`src/stream/` 是"物流通道"，负责 TCP/UDP 四层代理。比如你们以后要代理 MySQL、Redis 这种非 HTTP 协议，就得靠它。`src/mail/` 是"邮局"，处理 IMAP/POP3/SMTP 邮件代理——这个现在用得少了，但架构上很经典。

**小白**：还有 `src/os/` 呢？

**大师**：这是"本地化适配层"。里面有 `unix/` 和 `win32/` 两个子目录。Nginx 原本是 Unix 世界的产物，后来移植到 Windows 时，就把平台相关的代码（进程创建、信号处理、文件 IO）隔离在这里。

**小胖**：（挠头）那 `auto/` 目录是干嘛的？我看里面全是 .sh 脚本。

**大师**：这是 Nginx 最精妙的设计之一——**自动构建系统**。你们编译软件时，是不是经常遇到"缺少依赖""头文件找不到"的问题？Nginx 的 configure 脚本会主动探测系统环境：你的系统支持 epoll 吗？有 OpenSSL 吗？Pcre 版本是多少？然后自动生成适合你的 Makefile。

**小白**：所以 configure 不是 GNU Autotools 生成的，而是手写的？

**大师**：完全手写！`auto/options` 定义了所有编译选项，`auto/sources` 管理源文件列表，`auto/modules` 处理模块的启用/禁用，`auto/feature` 负责环境探测。这套系统虽然古老，但极其高效——整个 configure 执行时间不到 10 秒。

**小胖**：那我要是想只编译我需要的模块，把不需要的关掉，该改哪？

**大师**：`auto/options` 里定义了 `--with-*` 和 `--without-*` 参数。比如你们不需要 mail 模块，就加 `--without-mail`；需要 HTTP/3，就加 `--with-http_v3_module`。configure 会根据这些参数，在 `ngx_modules.c` 中生成最终的模块列表。

**小白**：明白了。core 是地基，event 是交通，http/stream/mail 是业务楼，os 是地基适配，auto 是施工图纸。

**大师**：（竖起大拇指）总结得很到位。现在，咱们就按这张图纸，亲手"施工"一次。

## 3. 项目实战

### 环境准备

- **操作系统**：Ubuntu 22.04 LTS / CentOS 8 / macOS（需 Xcode Command Line Tools）
- **源码版本**：Nginx 1.31.0
- **依赖包**：gcc、make、pcre、pcre-devel、zlib、zlib-devel、openssl、openssl-devel

```bash
# Ubuntu/Debian
sudo apt update
sudo apt install -y build-essential libpcre3-dev zlib1g-dev libssl-dev

# CentOS/RHEL
sudo yum groupinstall -y "Development Tools"
sudo yum install -y pcre pcre-devel zlib zlib-devel openssl openssl-devel
```

### 步骤一：获取源码并初识目录

```bash
# 进入项目目录（如果已经在 nginx 源码根目录则跳过）
cd /path/to/nginx

# 查看根目录
ls -la
```

预期输出：

```
auto/   conf/   contrib/  docs/   misc/   src/
LICENSE  Makefile  README.md  auto/    configure
```

关键目录说明：

| 目录 | 职责 | 关键文件 |
|------|------|---------|
| `auto/` | 编译配置脚本 | `configure`、`options`、`sources`、`modules`、`feature` |
| `src/core/` | 核心基础库 | `ngx_palloc.c`、`ngx_string.c`、`ngx_array.c`、`ngx_hash.c`、`ngx_rbtree.c` |
| `src/event/` | 事件驱动层 | `ngx_event.c`、`ngx_event_timer.c`、`modules/ngx_epoll_module.c` |
| `src/http/` | HTTP 协议层 | `ngx_http.c`、`ngx_http_request.c`、`ngx_http_upstream.c`、`modules/` |
| `src/stream/` | TCP/UDP 四层代理 | `ngx_stream.c`、`ngx_stream_proxy_module.c` |
| `src/mail/` | 邮件代理 | `ngx_mail.c`、`ngx_mail_proxy_module.c` |
| `src/os/` | 操作系统适配 | `unix/ngx_process_cycle.c`、`unix/ngx_socket.c` |
| `conf/` | 默认配置文件 | `nginx.conf`、`mime.types`、`fastcgi_params` |

### 步骤二：分析 configure 的工作原理

```bash
# 查看 configure 脚本的前 50 行
head -n 50 auto/configure
```

输出核心逻辑：

```sh
#!/bin/sh

# auto/configure 是一个入口脚本
# 它依次调用 auto/options、auto/init、auto/sources、auto/modules 等子脚本

. auto/options    # 解析命令行参数
. auto/init       # 初始化环境变量
. auto/sources    # 收集源文件列表
. auto/modules    # 处理模块依赖
. auto/feature    # 探测系统特性
. auto/make       # 生成 Makefile
```

查看 `auto/options` 中的关键编译选项：

```bash
grep -E "^--(with|without)-" auto/options | head -n 30
```

常用编译选项分类：

| 类别 | 选项示例 | 说明 |
|------|---------|------|
| 路径类 | `--prefix=/usr/local/nginx` | 安装根目录 |
| 模块类 | `--with-http_ssl_module` | 启用 SSL 模块 |
| 模块类 | `--without-http_proxy_module` | 禁用代理模块 |
| 模块类 | `--add-module=/path/to/module` | 添加第三方模块 |
| 调试类 | `--with-debug` | 开启 debug 日志 |
| 优化类 | `--with-cc-opt="-O2"` | 编译器优化选项 |
| 系统类 | `--with-file-aio` | 启用异步文件 IO |
| 系统类 | `--with-threads` | 启用线程池 |

### 步骤三：自定义编译 Nginx

鲜果园的需求是：只需要 HTTP 代理和静态文件服务，不需要邮件代理，需要 HTTPS 支持，未来可能要加自定义模块。

```bash
# 进入源码根目录
cd /path/to/nginx

# 执行 configure，按需启用/禁用模块
./auto/configure \
    --prefix=/usr/local/nginx \
    --with-http_ssl_module \
    --with-http_v2_module \
    --with-http_realip_module \
    --with-http_stub_status_module \
    --with-http_gzip_static_module \
    --with-file-aio \
    --with-threads \
    --with-debug \
    --without-mail_pop3_module \
    --without-mail_imap_module \
    --without-mail_smtp_module \
    --without-http_uwsgi_module \
    --without-http_scgi_module \
    --with-cc-opt="-O2 -g" \
    --with-ld-opt="-Wl,-rpath,/usr/local/lib"

# 查看生成的 objs/ngx_modules.c，确认模块列表
cat objs/ngx_modules.c | grep "ngx_module_t" | head -n 20
```

configure 执行成功后，会输出摘要信息：

```
Configuration summary
  + using threads
  + using file AIO
  + using system PCRE library
  + using system OpenSSL library
  + using zlib library: /usr/lib/x86_64-linux-gnu/libz.so
  + nginx path prefix: "/usr/local/nginx"
  + nginx binary file: "/usr/local/nginx/sbin/nginx"
  + nginx configuration prefix: "/usr/local/nginx/conf"
```

### 步骤四：编译与安装

```bash
# 编译（-j 指定并行任务数，加快编译速度）
make -j$(nproc)

# 安装
sudo make install
```

编译完成后，验证目录结构：

```bash
ls -la /usr/local/nginx/
# 预期输出：conf/  html/  logs/  sbin/
```

### 步骤五：验证编译结果

```bash
# 查看版本和编译参数
/usr/local/nginx/sbin/nginx -V
```

预期输出（类似）：

```
nginx version: nginx/1.31.0
built by gcc 11.4.0 (Ubuntu 11.4.0-1ubuntu1~22.04)
built with OpenSSL 3.0.2 15 Mar 2022
TLS SNI support enabled
configure arguments: --prefix=/usr/local/nginx --with-http_ssl_module --with-http_v2_module ...
```

可以看到所有启用的模块和编译参数。

### 步骤六：精简二进制体积（进阶）

如果需要在嵌入式设备或容器中使用，可以进一步精简：

```bash
# 使用 strip 去除调试符号
sudo strip /usr/local/nginx/sbin/nginx

# 查看文件大小
ls -lh /usr/local/nginx/sbin/nginx
# 精简前约 15MB，strip 后约 3-5MB
```

**可能遇到的坑**：

1. **缺少 PCRE 库**：configure 报错 "the HTTP rewrite module requires the PCRE library"。
   - 解决：`sudo apt install libpcre3-dev`（Ubuntu）或 `sudo yum install pcre-devel`（CentOS）

2. **OpenSSL 版本不兼容**：`--with-http_v3_module` 需要 OpenSSL 3.0+ 或 BoringSSL。
   - 解决：安装 OpenSSL 3.0，或手动指定 `--with-openssl=/path/to/openssl-3.0`

3. **第三方模块编译失败**：`--add-module` 引入的模块使用了不兼容的 Nginx API。
   - 解决：检查第三方模块是否支持 Nginx 1.31.0，必要时升级模块版本

4. **Windows 下编译**：Nginx 官方不支持 `make install` 到 Windows，需使用 MSYS2 或 Visual Studio 项目。
   - 解决：参考 `src/os/win32/` 下的 README 和 Visual Studio 项目文件

## 4. 项目总结

### 优点与缺点

| 维度 | Nginx 源码编译 | 包管理器安装（apt/yum） |
|------|--------------|----------------------|
| 模块控制 | 完全自定义，按需开关 | 固定模块集，无法灵活调整 |
| 版本选择 | 可装任意版本，包括主线/mainline | 通常滞后数月 |
| 调试能力 | 支持 --with-debug，便于源码追踪 | 通常已 strip，无 debug 符号 |
| 第三方模块 | 可通过 --add-module 集成 | 难以集成非官方模块 |
| 编译时间 | 1-5 分钟（取决于机器性能） | 秒级安装 |
| 维护成本 | 需自行管理更新和补丁 | 系统自动更新 |

### 适用场景

1. **生产环境定制**：只编译需要的模块，减少攻击面和二进制体积
2. **源码学习与研究**：带 debug 符号的编译便于 GDB 调试和源码阅读
3. **第三方模块集成**：如 OpenResty、Lua 模块、WAF 模块等
4. **嵌入式/容器化**：精简二进制，构建最小镜像
5. **新特性尝鲜**：主线版本往往比发行版仓库更新

**不适用场景**：
1. **快速原型验证**：包管理器安装更省时
2. **无编译环境的生产服务器**：预编译二进制包更稳妥
3. **需要长期安全补丁维护**：发行版仓库会自动推送安全更新

### 注意事项

1. **--prefix 要合理设置**：默认 /usr/local/nginx 是独立目录，不会影响系统 Nginx。如果要替换系统 Nginx，需谨慎操作。
2. **升级时保留旧二进制**：执行 make install 前，建议备份 sbin/nginx，以便回滚。
3. **编译依赖要记录**：团队内应维护一份 compile.sh 脚本，记录所有 configure 参数，确保环境一致性。

### 常见踩坑经验

**案例一：模块遗漏导致功能缺失**
- **现象**：编译完成后发现 HTTPS 无法使用，报错 "unknown directive ssl"
- **根因**：configure 时忘记加 `--with-http_ssl_module`
- **解决**：重新编译，确保 SSL 模块在 `objs/ngx_modules.c` 中

**案例二：动态模块 vs 静态模块混淆**
- **现象**：使用了 `--add-dynamic-module` 编译的模块，启动时报 "module is not binary compatible"
- **根因**：动态模块编译时使用的 Nginx 源码版本与运行时版本不一致
- **解决**：动态模块必须与运行时的 Nginx 版本完全匹配，建议用同一套源码编译

**案例三：configure 缓存导致配置不生效**
- **现象**：修改了 configure 参数后重新执行，但生成的 Makefile 未更新
- **根因**：objs/ 目录下的缓存文件未清理
- **解决**：执行 `make clean` 或 `rm -rf objs/` 后再重新 configure

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。

### 思考题

1. **进阶题**：`auto/modules` 脚本是如何根据 `--with-*` 和 `--without-*` 参数，在 `objs/ngx_modules.c` 中生成最终的 `ngx_modules[]` 数组的？请阅读 `auto/modules` 的关键逻辑。

2. **进阶题**：如果你需要在 Nginx 中集成一个自定义的 C 模块，需要编写哪些文件？`config` 文件的语法和作用是什么？

> 答案提示：第 1 题关注 `auto/modules` 中的 `ngx_module_type` 和 `ngx_module_name` 变量；第 2 题需要了解 `ngx_addon_name`、`HTTP_MODULES`、`NGX_ADDON_SRCS` 等变量。
