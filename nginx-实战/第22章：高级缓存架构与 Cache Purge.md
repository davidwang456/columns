## 1. 项目背景

视频业务在一次热点片单更新后，用户投诉“新封面一直刷不出来”。监控显示缓存命中率高达 96%，看起来像优化成功，实际上命中了错误内容。团队最终确认根因：cache key 设计不完整，且缺少可控 purge 机制。

这类问题在中级阶段非常典型：指标表面好看，但业务体验在关键节点失真。团队过去依赖“经验调参”，现在必须切换为“指标驱动 + 故障演练 + 可回滚”的交付模式。
本章会沿着“真实业务痛点 -> 方案博弈 -> 逐步落地 -> 验证闭环”来推进，确保这套方案不仅能跑，还能在高峰与异常中站得住。

---

从知识路径上看，本章与上一章《HTTP/2 与 gRPC 代理》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：视频业务事故复盘会，大屏上展示着“命中率 96%”和“用户投诉新封面刷不出来”的反差数据。

---

**小胖**：（嚼着口香糖）缓存命中率都 96% 了还出问题？那不跟我冰箱里塞满了菜，结果拿出来全馊了一样吗——命中是命中了，但是命中的是过期内容。

**小白**：（若有所思）所以问题不在命中率，而在“命中的内容对不对”。cache key 设计、失效策略、purge 机制，但凡一个没对齐，命中率再高也是虚假繁荣。

**大师**：（在白板上画出三个圈）小胖的冰箱比喻很形象。没设计好的 cache key，就像把所有剩菜都倒进一个没贴标签的保鲜盒——回锅肉和糖醋排骨混一起，根本分不清。加了 `$uri$args` 当 key 还不够——不同用户看到的视频封面可能不同，你还得把版本号、语言等维度都加进去。

**小白**：那 slice 分片缓存呢？我看配置里有个 `slice 1m`。

**大师**：（画了一条长线，折成几段）大文件缓存就像抱着一整个西瓜存冰箱，太占地方也不好取。slice 就是切成片，每片 1MB 单独缓存。好处有三：一是支持断点续传，客户端请求 Range 片段时直接命中缓存，不用全量回源；二是降低写放大——热点文件只更新变化的“切片”；三是内存利用更高效，大文件不用一次性加载。

**大师**：**技术映射**：`proxy_cache_key` 决定缓存标识维度，`slice` 指令将大文件切块存储，`proxy_cache_purge` 提供按 key 精确失效的能力。三者配合构成“可治理缓存”体系——存得对、找得到、删得掉。

---

**小胖**：那缓存惊群呢？就是所有缓存同时过期，请求全打到后端——这不跟公司食堂中午 12 点所有人同时涌进去一样？

**大师**：这就是 Cache Stampede。一个热点 key 过期后，成百上千个请求同时回源，后端瞬间被打垮。常见解法有三种：一是锁机制，只让一个请求回源，其他等待；二是用 Nginx 的 `proxy_cache_lock` 开启缓存锁；三是主动刷新，在缓存过期前提前预热。

**小白**：那 purge 接口是不是只能在测试环境用？线上谁敢动缓存啊。

**大师**：恰恰相反，越是线上越需要 purge。业务方更新封面后，你不 purge 旧缓存，用户看到的永远是老图。关键是权限控制——`allow/deny` 限定内网 IP 才能执行 purge，同时记录操作日志，出问题可追溯。这就像冰箱里专设一个“过期食品丢弃区”，不是不让扔，而是规定只有管理员能碰。

**大师**：

**技术映射**：`proxy_cache_lock` + `proxy_cache_lock_timeout` 应对缓存惊群；Cache Purge 通过 `proxy_cache_purge` 指令实现精确失效，配合 ACL 和审计日志做到可控运维。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等发行版）
- Nginx：1.31.0
- 工具：`curl`、`wrk`、`ss`、`jq`
- 附加工具：第 24 章建议 `mysql` / `dig`，第 25 章建议 `websocat`

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：先建立可运行基线，避免“边改边猜”。

```nginx
proxy_cache_path /data/nginx/cache levels=1:2 keys_zone=vod:200m max_size=50g inactive=30m use_temp_path=off;
server {
    listen 8080;
    location /vod/ {
        slice 1m;
        proxy_cache vod;
        proxy_cache_key "$scheme$host$uri$is_args$args$slice_range";
        proxy_set_header Range $slice_range;
        proxy_cache_valid 200 206 10m;
        add_header X-Cache-Status $upstream_cache_status always;
        proxy_pass http://vod_backend;
    }
    location ~ /purge(/.*) {
        allow 10.0.0.0/8;
        deny all;
        proxy_cache_purge vod "$scheme$host$1";
    }
}
```

**运行结果（预期）**：
- 配置校验通过；
- 关键接口连通；
- `error.log` 无持续性异常。

**可能遇到的坑**：仅用功能联调代替压测与异常演练，导致上线后才暴露问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用可比较的数据判断方案优劣。

```bash
curl -I http://127.0.0.1:8080/vod/a.mp4
curl -I http://127.0.0.1:8080/vod/a.mp4
curl -X PURGE http://127.0.0.1:8080/purge/vod/a.mp4
```

**建议关注指标**：
- 吞吐：QPS / TPS
- 延迟：P95、P99
- 稳定性：错误率、超时率、重试率
- 资源：活动连接、CPU、内存

**可能遇到的坑**：压测时间太短，掩盖连接池、缓存、GC 等“慢热型问题”。建议最少跑 5~10 分钟稳定段。

### 步骤三：故障注入与回滚演练

**步骤目标**：验证系统在异常下仍可控。

```bash
# 触发变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线一个后端节点；
2. 人工增加后端延迟（如 tc 注入）；
3. 叠加突发流量观察尾延迟。

**本章实战目标**：搭建可治理缓存体系：高命中、可失效、可追踪。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 可用性 | 有明确保护与回滚机制，异常可控 | 建设初期流程更重 |
| 性能 | 能在真实指标上做持续优化 | 需要持续压测资源 |
| 协作 | 跨团队共享同一评估标准 | 对执行纪律要求更高 |

### 适用场景

1. 高峰波动明显、对可用性要求高的业务。
2. 网关承担统一流量治理职责的平台。
3. 希望建立“可复盘、可复制”优化流程的团队。

**不适用场景**：
- 单机低流量且无明显增长预期的项目。
- 缺少监控与压测基础设施、无法闭环验证的团队。

### 注意事项

1. 指标先行：没有基线就没有优化结论。
2. 变更最小化：一次只改一个关键变量。
3. 回滚可执行：任何优化都必须自带止损路径。

### 常见踩坑经验

- 只追求命中率/QPS，不看业务正确性。
- 缺少异常注入，线上首遇故障才发现边界。
- 变更记录缺失，后续团队无法继承经验。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
