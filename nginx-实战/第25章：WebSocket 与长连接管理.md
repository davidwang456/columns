## 1. 项目背景

实时聊天室上线后连接数迅速突破 4 万，网关内存随连接线性增长，凌晨出现批量断连。排查发现：心跳 90s、proxy_read_timeout 60s，网关将大量正常连接当成空闲连接回收。

这类问题在中级阶段非常典型：指标表面好看，但业务体验在关键节点失真。团队过去依赖“经验调参”，现在必须切换为“指标驱动 + 故障演练 + 可回滚”的交付模式。
本章会沿着“真实业务痛点 -> 方案博弈 -> 逐步落地 -> 验证闭环”来推进，确保这套方案不仅能跑，还能在高峰与异常中站得住。

---

从知识路径上看，本章与上一章《Stream 四层代理——TCP/UDP》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：实时聊天室事故复盘会，大屏显示凌晨 4 万连接批量断开的时间线，运维顶着一对黑眼圈。

---

**小胖**：（泡着方便面）WebSocket 不就是从 HTTP 升级一下嘛，跟打电话一样，接通了就一直聊呗。为啥咱们的网关还把正常连接给掐了？

**小白**：问题在心跳和超时的剪刀差——心跳间隔 90s，但 proxy_read_timeout 才 60s。网关 60s 没读到数据就砍连接，客户端第 90s 才发心跳，中间 30s 窗口期全被误杀了。

**大师**：（在白板上画了一条时间线）小胖说的“打电话”很对，但我们得弄懂这个电话是怎么打通的。WebSocket 的握手不是魔法——就是一次 HTTP Upgrade 请求，客户端说“我不想用 HTTP 了，改用 WebSocket 协议吧”，服务端回复“同意，升级！状态码 101”。这个握手过程在 Nginx 里就是 `proxy_set_header Upgrade $http_upgrade` 和 `proxy_set_header Connection "upgrade"` 这两行配置。

**小白**：那握完手之后呢？Nginx 怎么知道这个连接是长连接，不能按 HTTP 的短连接逻辑回收？

**大师**：握手成功后，连接协议从 HTTP 切换为 WebSocket，Nginx 的角色从“HTTP 代理”退化为“纯 TCP 隧道”——它不再解析协议内容，只负责在两端之间双向转发二进制帧。这时影响连接寿命的关键参数变了：不再是 `keepalive_timeout`，而是 `proxy_read_timeout` 和 `proxy_send_timeout`。这两个超时的时钟，是从最后一次读写数据开始计时的。

**大师**：

**技术映射**：WebSocket 通过 HTTP Upgrade 机制（101 Switching Protocols）完成协议协商；Nginx 用 `proxy_set_header Upgrade $http_upgrade` 和 `Connection "upgrade"` 中继握手。连接建立后，`proxy_read_timeout` 控制空闲超时，必须大于客户端心跳间隔。

---

**小胖**：那为什么不把 proxy_read_timeout 设个 24 小时？省得半夜被叫醒。

**小白**：（摇头）连接不释放，内存和文件描述符持续占用。4 万连接不回收，过几天内存就炸了。

**大师**：小白说得对——无限超时等于内存泄漏。正确的做法是“心跳间隔 + 容忍窗口”策略：比如客户端每 30s 发一次 ping，服务端等 45s，超时未收到就回收。另外 Nginx 的 `keepalive` 指令可以在上游连接池复用连接，减少频繁建连的开销。这里还有个选择问题：SSE（Server-Sent Events）是单向的，走普通 HTTP 长连接，自动重连，适合推送场景；WebSocket 是双向的，适合聊天室、实时协作。选哪个取决于业务是否需要上行通道。

**大师**：

**技术映射**：长连接管理三要素——心跳间隔（ping/pong）、空闲超时（`proxy_read_timeout`）、连接池复用（`keepalive`）。SSE（`text/event-stream`）是 WebSocket 的轻量替代，基于 HTTP 长轮询，无需 Upgrade 握手，适合服务端到客户端的单向推送。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等发行版）
- Nginx：1.31.0
- 工具：`curl`、`wrk`、`ss`、`jq`
- 附加工具：第 24 章建议 `mysql` / `dig`，第 25 章建议 `websocat`

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：先建立可运行基线，避免“边改边猜”。

```nginx
upstream ws_backend {
    server 10.0.5.11:9001;
    server 10.0.5.12:9001;
    keepalive 128;
}
server {
    listen 8080;
    location /ws/ {
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 120s;
        proxy_send_timeout 120s;
        proxy_pass http://ws_backend;
    }
}
```

**运行结果（预期）**：
- 配置校验通过；
- 关键接口连通；
- `error.log` 无持续性异常。

**可能遇到的坑**：仅用功能联调代替压测与异常演练，导致上线后才暴露问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用可比较的数据判断方案优劣。

```bash
websocat ws://127.0.0.1:8080/ws/room
ss -ant | rg ESTAB | wc -l
curl -s http://127.0.0.1:8080/nginx_status
```

**建议关注指标**：
- 吞吐：QPS / TPS
- 延迟：P95、P99
- 稳定性：错误率、超时率、重试率
- 资源：活动连接、CPU、内存

**可能遇到的坑**：压测时间太短，掩盖连接池、缓存、GC 等“慢热型问题”。建议最少跑 5~10 分钟稳定段。

### 步骤三：故障注入与回滚演练

**步骤目标**：验证系统在异常下仍可控。

```bash
# 触发变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线一个后端节点；
2. 人工增加后端延迟（如 tc 注入）；
3. 叠加突发流量观察尾延迟。

**本章实战目标**：建立 WebSocket 长连接治理基线，稳定承载高并发在线连接。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 可用性 | 有明确保护与回滚机制，异常可控 | 建设初期流程更重 |
| 性能 | 能在真实指标上做持续优化 | 需要持续压测资源 |
| 协作 | 跨团队共享同一评估标准 | 对执行纪律要求更高 |

### 适用场景

1. 高峰波动明显、对可用性要求高的业务。
2. 网关承担统一流量治理职责的平台。
3. 希望建立“可复盘、可复制”优化流程的团队。

**不适用场景**：
- 单机低流量且无明显增长预期的项目。
- 缺少监控与压测基础设施、无法闭环验证的团队。

### 注意事项

1. 指标先行：没有基线就没有优化结论。
2. 变更最小化：一次只改一个关键变量。
3. 回滚可执行：任何优化都必须自带止损路径。

### 常见踩坑经验

- 只追求命中率/QPS，不看业务正确性。
- 缺少异常注入，线上首遇故障才发现边界。
- 变更记录缺失，后续团队无法继承经验。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
