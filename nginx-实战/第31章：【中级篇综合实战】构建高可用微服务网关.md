## 1. 项目背景

服务数突破 50 后，认证、限流、灰度、监控分散在不同网关和中间件里，排障路径极长。团队决定交付统一微服务网关平台。

进入这一阶段后，Nginx 不再只是“转发工具”，而是稳定性、性能与工程效率的共同支点。仅靠参数经验已经不足以支撑复杂系统，必须把“设计假设、实验数据、故障演练、回滚策略”绑定成一个闭环。
本章沿着“问题定义 -> 方案取舍 -> 实验验证 -> 上线守护”的路径展开，确保方案可复制、可复盘、可演进。

---

## 2. 项目设计

**场景**：微服务网关平台架构评审会，白板左边写着"50+ 微服务"，右边写着"7 套网关"，中间一个大大的问号。

---

**小胖**：（边剥橘子边说）现在每个微服务都自己搞认证、限流、日志、监控——50 个服务就是 50 套"小摊子"，比我老家庙会一条街还乱！统一一个网关入口不就完事了吗？

**小白**：统一网关的好处我认同，但网关变成了中心节点后，它本身就是单点瓶颈——网关挂了所有服务全挂。这个风险怎么控制？而且 50 多个服务的协议不一样，有 HTTP、gRPC、WebSocket，网关能一把抓吗？

**大师**：（在白板上画了一个安检口示意图）小胖说的"庙会一条街"很形象——每个摊位都自己搞安检，游客得排 50 次队。统一网关就是在庙会入口设一个中央安检区（网关集群），一次安检过完，后面随便逛。但这个安检口不能只有一个窗口——必须是一排窗口（多副本）。网关承担五大核心职能：认证鉴权、限流降级、灰度路由、协议转换、可观测采集。高可用靠三层屏障：多副本 + 健康检查 + 限流自我保护。

**技术映射**：统一网关设计：`auth_request /auth` 做统一鉴权；`limit_req zone=api_rate burst=50 nodelay` 做速率限制；`split_clients $remote_addr $backend` 做灰度分流。upstream 配置 `max_fails=3 fail_timeout=30s` 自动摘除故障节点。

---

**小胖**：协议转换我真搞不懂，HTTP 请求进来，到后端变成 gRPC，这中间还得"翻译"一遍，不会慢得像蜗牛吧？

**小白**：而且 WebSocket 是长连接，跟 HTTP 短连接模式完全不同，代理层处理不好就会断连。

**大师**：协议转换就像国际机场的"货币兑换窗口"——一个窗口能把人民币换成美元、欧元、日元。Nginx 从 1.13 起原生支持 gRPC 代理，配置 `grpc_pass` 就行，底层走 HTTP/2 多路复用，连接复用率比 HTTP/1.1 还高。WebSocket 通过 `Upgrade` 和 `Connection` 头透传，Nginx 握手阶段升级到 WebSocket 协议后，后续完全透明转发。唯一要注意的是超时：WebSocket 要把 `proxy_read_timeout` 调到 3600s 以上，否则 nginx 会主动断开空闲连接。

**技术映射**：多协议路由配置：gRPC `location /grpc/ { grpc_pass grpc://backend_grpc; }`、WebSocket `proxy_set_header Upgrade $http_upgrade; proxy_set_header Connection "upgrade"; proxy_read_timeout 3600s;`、HTTP `proxy_pass http://backend_http;`。通过 `map $http_upgrade $connection_upgrade { default "upgrade"; "" "close"; }` 统一管理。

---

**小白**：网关上线后怎么验证是稳定的？总得有具体的指标决定是继续放量还是立刻回滚吧？

**大师**：网关上线守护看三个关键信号，任何一个超标立即触发自动回滚：① 全局 5xx 错误率超过基线 2 倍；② P99 延迟增长超过 50%；③ 活跃连接数异常陡降（说明连接被意外丢弃）。这三个指标在 Prometheus 里配成告警，联动 CI/CD 管线自动执行回滚。灰度放量策略是阶梯式的——先 1% 内部流量 → 10% 外部 → 50% → 全量，每阶段稳定观察 15 分钟。像煮饺子——先放几个试试水温，不破皮再全下锅。

**技术映射**：回滚触发器 PromQL：`rate(nginx_http_requests_total{status=~"5.."}[5m]) / rate(nginx_http_requests_total[5m]) > 0.1`。结合 `kubectl rollout undo deployment/gateway` 或 Ansible 切 upstream 配置实现自动化回滚。灰度通过 `split_clients $remote_addr $backend` 按客户端 IP 一致性哈希分流。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等级 Linux 发行版）
- Nginx：1.31.0（建议保留 debug 能力）
- 工具：`curl`、`wrk`、`ss`、`jq`
- 按章补充：数据库/DNS/WebSocket/K8s/profiling 工具

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：构建可运行、可验证、可回滚的最小基线。

```nginx
map $http_authorization $auth_ok {
    default 0;
    ~^Bearer\s+.+ 1;
}
upstream core_api {
    server 10.1.0.11:8080;
    server 10.1.0.12:8080;
}
server {
    listen 8080;
    location / {
        if ($auth_ok = 0) { return 401; }
        limit_req zone=api_rate burst=50 nodelay;
        proxy_set_header X-Trace-Id $request_id;
        proxy_pass http://core_api;
    }
}
```

**运行结果（预期）**：
- 配置检查通过；
- 关键路径连通；
- `error.log` 无持续异常。

**可能遇到的坑**：仅做功能验证，不做压力与异常验证，导致上线后才暴露边界问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用数据而非主观感受评估改动价值。

```bash
curl -i http://127.0.0.1:8080/order
curl -i -H "Authorization: Bearer demo" http://127.0.0.1:8080/order
```

**建议观测指标**：
- 吞吐：QPS/TPS
- 时延：P95/P99
- 可靠性：5xx、超时率、重试率
- 资源：CPU、内存、活动连接

**可能遇到的坑**：压测时间太短、场景太单一。建议至少覆盖平峰/高峰/故障三种情形。

### 步骤三：故障注入与回滚演练

**步骤目标**：确保故障发生时可快速止损。

```bash
# 应用变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线单个上游节点；
2. 注入后端延迟或错误码；
3. 叠加突发流量验证尾延迟。

**本章实战目标**：交付可上线的网关基线：统一入口、鉴权、限流、灰度、可观测。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 稳定性 | 有明确的演练与回滚机制 | 前期建设成本增加 |
| 性能 | 可持续、可量化优化 | 依赖监控与压测体系 |
| 团队协作 | 结论标准化、可传承 | 对执行纪律要求高 |

### 适用场景

1. 高峰明显、故障成本高的核心业务。
2. 网关承载治理能力的平台化系统。
3. 希望把经验运维升级为工程化运维的团队。

**不适用场景**：
- 低流量 demo 或短生命周期项目。
- 缺少基础观测能力、无法形成验证闭环的团队。

### 注意事项

1. 无基线不优化：先量化现状再改动。
2. 单变量优先：保障结论可归因。
3. 回滚先行：上线前先验证止损路径。

### 常见踩坑经验

- 只看平均值，不看尾延迟。
- 只做成功路径，不做故障演练。
- 变更记录缺失，导致后续无法复盘。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
