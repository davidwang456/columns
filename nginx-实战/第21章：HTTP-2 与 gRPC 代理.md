## 1. 项目背景

客户端团队反馈：同样的网络环境下，REST 接口首包时间在高峰期显著恶化，移动端电量消耗也偏高。进一步分析发现 HTTP/1.1 短连接过多，重复握手和头部冗余是主要开销。与此同时，内部服务准备切 gRPC，却缺统一入口。

在复盘会上，运维团队给出一个关键结论：以前“能跑就行”的配置策略，在中级阶段会迅速失效。因为系统面对的已经不是单点问题，而是“流量规模 + 协议复杂度 + 组织协作成本”的复合压力。
因此本章不只讲参数含义，还会给出一套可复用的方法：**先建基线，再做单变量实验，最后用故障注入验证边界**。这套方法会贯穿中级篇后续章节。

---

从知识路径上看，本章与上一章《高级负载均衡算法实战》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：故障演练日复盘会，白板上画满了 HTTP/1.1 和 HTTP/2 的对比图，三人对着压测报告争论不休。

---

**小胖**：（啃着苹果）HTTP/2 不就是把多个请求塞一个连接里吗？跟食堂打饭一个窗口排队，只不过队伍只有一条，不像以前每个窗口都得排一次队。

**小白**：（皱眉）但 HTTP/2 有队头阻塞问题——TCP 丢包时所有流都得等这一个包重传，这不比 HTTP/1.1 的独立连接更糟糕？

**大师**：（在白板上画了两张图）你们说的都对了一半。先看 HTTP/1.1，每个请求建一个 TCP 连接，就像你每次去食堂打饭都要重新拿个餐盘。HTTP/2 一个连接上跑多个流，就像一个大传送带上同时传不同人的餐盘。小胖，你说对了“一条队伍”，但小白的问题更关键——如果传送带卡住了（TCP 丢包），所有餐盘都动不了，这就是 HTTP/2 的 TCP 级队头阻塞。

**大师**：

**技术映射**：HTTP/2 的核心是**二进制分帧（Binary Framing Layer）**，把请求和响应拆成 HEADERS、DATA、SETTINGS 等小帧，通过多路复用（Stream Multiplexing）在一个 TCP 连接上交错传输。但 TCP 层面的队头阻塞确实是 HTTP/2 的固有弱点，这也是 HTTP/3（QUIC）要解决的问题。

---

**小胖**：那 gRPC 呢？我听说它用 HTTP/2 但又能像本地函数一样调用，这不就跟在食堂跟师傅说“老样子”一样，不用每次都把菜名报一遍？

**大师**：（点头）这个比喻很到位。HTTP/1.1 每次调用都要带一堆重复的 Header，就像每次点菜都从菜单第一行念到最后一行。HTTP/2 的 HPACK 压缩把常见 Header 存成索引表，后面只用传索引号——这就是你说的“老样子”。gRPC 在此基础上用 Protocol Buffers 做序列化，数据更小、解析更快，天然支持流式调用。

**小白**：那 gRPC 的流式调用跟普通 HTTP 的 Server Push 有什么区别？

**大师**：Server Push 是服务端“猜测”客户端需要什么并推送额外资源；而 gRPC 的流式是建立了一个长寿命双向数据通道——就像在食堂点了个“流水席”，后厨做好一道上一道，不用等全部做完才端上来。Nginx 通过 `grpc_pass` 代理 gRPC 流量，需要配置 `grpc_read_timeout` 和 `grpc_send_timeout` 管理流式连接的保活。

**大师**：

**技术映射**：gRPC 基于 HTTP/2 实现四种调用模式：一元调用（Unary）、服务端流（Server Streaming）、客户端流（Client Streaming）、双向流（Bidirectional Streaming）。Nginx 的 grpc 模块负责将 HTTP/2 帧路由到上游 gRPC 服务，需在 SSL 和 HTTP/2 基础上启用。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04 / Rocky Linux 9（任选其一）
- Nginx：1.31.0（建议开启 debug）
- 工具：`wrk`、`curl`、`ss`、`jq`、`grpcurl`（第21章需要）

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：建立最小可运行配置

**步骤目标**：让核心路径先稳定跑通，保证后续实验有可信起点。

```nginx
server {
    listen 443 ssl http2;
    server_name api.demo.local;
    ssl_certificate     /etc/nginx/cert/fullchain.pem;
    ssl_certificate_key /etc/nginx/cert/privkey.pem;
    location /rest/ {
        proxy_pass http://rest_backend;
    }
    location /grpc.OrderService/ {
        grpc_pass grpc://grpc_backend;
        grpc_read_timeout 30s;
        grpc_send_timeout 30s;
    }
}
```

**运行结果（预期）**：
- `nginx -t` 输出 syntax is ok / test is successful
- 健康检查接口返回 200
- `error.log` 无持续性报错

**可能遇到的坑**：配置能通过但链路没通，多数是上游地址、证书路径或防火墙规则未对齐。

### 步骤二：做基线压测与指标采集

**步骤目标**：获得“优化前”与“优化后”的可比数据。

```bash
curl --http2 -I https://api.demo.local/rest/health
grpcurl -insecure api.demo.local:443 list
grpcurl -insecure api.demo.local:443 grpc.OrderService/Health
```

**建议记录指标**：
- 吞吐：QPS / RPS
- 延迟：P50、P95、P99
- 稳定性：5xx 比例、连接失败率、重试比例
- 资源：CPU、内存、活动连接数

**可能遇到的坑**：压测机成为瓶颈，导致测试结果误判。建议压测端与被测端分离，且至少跑 2-3 轮取中位值。

### 步骤三：故障注入与回滚演练

**步骤目标**：确认异常场景下系统仍可控，发布后具备止损能力。

```bash
# 触发配置变更
sudo nginx -t && sudo nginx -s reload
# 回滚（示例）
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 人工下线一个上游节点。
2. 将一个上游延迟注入到 1~2s。
3. 模拟突发流量（瞬时并发提升 2~3 倍）。

**本章实战目标**：完成同域名双协议网关（HTTP/2 + gRPC）并验证流式调用稳定性。

### 完整代码清单

- 建议仓库目录：`ops/nginx/ch21/`
- 包含文件：`nginx.conf`、`upstream.conf`、`bench.sh`、`rollback.sh`、`README.md`
- 推荐把实验记录沉淀为 `result.md`，保留每次变更的指标截图与结论

### 测试验证

```bash
# 基础连通性验证（示例）
curl -i http://127.0.0.1:8080/health
```

验证要点：
- 关键接口返回码符合预期（2xx/4xx/5xx与设计一致）；
- 关键日志字段完整（如 request_id、upstream 耗时、状态码）；
- 在小流量压测下无异常错误峰值。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 方案代价 |
|---|---|---|
| 性能收益 | 能针对性改善吞吐与尾延迟 | 需要压测与监控支撑 |
| 稳定性 | 有故障注入与回滚兜底 | 发布流程更严格 |
| 协作效率 | 开发/测试/运维可共享同一结论 | 对变更纪律要求更高 |

### 适用场景

1. 日常有流量峰谷，且峰值抖动明显的系统。
2. 网关承担统一入口、对稳定性要求高的平台。
3. 需要沉淀标准化运维流程的中大型团队。

**不适用场景**：
- 单机低流量 demo 项目。
- 暂无监控与压测能力、难以形成验证闭环的团队。

### 注意事项

1. 先统一指标口径，再讨论优化效果。
2. 单次只改一个关键变量，方便归因。
3. 每次上线都保留可在 1 分钟内执行的回滚路径。

### 常见踩坑经验

- 只看平均延迟，忽略 P99 与错误率。
- 基线记录不完整，导致结论不可复验。
- “一把梭”改太多参数，上线后无法定位问题来源。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
