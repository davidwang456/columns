# 第4章：进程模型——信号处理与进程间通信

> 源码关联：src/os/unix/ngx_process_cycle.c、src/os/unix/ngx_process.c、src/core/ngx_cycle.c

---

## 1. 项目背景

鲜果园的 Nginx 服务已经稳定运行了两周，团队对它的配置也越来越熟悉。但一个周五的深夜，监控系统突然告警：线上 Nginx 的 access_log 文件膨胀到了 50GB，磁盘空间即将耗尽。运维工程师老王紧急登录服务器，本能地敲下了 `rm -f /var/log/nginx/access.log`，然后长舒一口气。然而，恐怖的事情发生了——`df -h` 显示磁盘占用纹丝未动！

老王懵了。文件明明删了，空间怎么没释放？他赶紧打电话给架构师小李。小李在电话那头苦笑："你删了文件，但 Nginx 的 worker 进程还握着文件句柄呢。得先让 Nginx 重新打开日志文件。"

`sudo kill -USR1 $(cat /var/run/nginx.pid)`

命令执行后，磁盘空间终于释放。老王心有余悸地问："这 USR1 到底是什么魔法？"

这背后正是 Nginx 进程模型的精髓。master 进程作为"指挥官"，通过 Unix 信号与 worker 进程通信；worker 进程作为"士兵"，在事件循环中处理请求。理解这套信号机制和进程间通信（IPC）模型，是运维 Nginx 的必备技能。本章将从业务事故出发，深入源码剖析 Nginx 的进程管理哲学。

## 2. 项目设计

**场景**：鲜果园的运维值班室，凌晨两点，老王、小胖（被电话叫醒的后备）和大师（远程语音）正在复盘事故。

---

**老王**：（擦着汗）大师，我算是服了。删了日志文件，空间居然不释放？这跟 Nginx 的进程模型有什么关系？

**大师**：（电话那头传来键盘声）这是 Linux 文件系统的特性：文件被删除只是删除了目录项，但只要还有进程持有文件描述符（fd），磁盘空间就不会释放。Nginx 的 worker 进程在启动时打开了 access_log，整个生命周期都不会主动关闭它。

**小胖**：（打着哈欠）那为什么不能直接重启 Nginx？重启了不就释放了吗？

**大师**：重启当然能释放，但会中断所有正在处理的连接。鲜果园半夜还有用户在下单，你重启一下，损失的可能就是几千块钱。而 `kill -USR1` 的作用，是让 master 进程通知所有 worker：**重新打开日志文件**——worker 会关闭旧的 fd，创建新的 fd，整个过程用户无感知。

**老王**：哦！这就是 Nginx 所谓的"热操作"？

**大师**：对。Nginx 的信号系统就是一套"热操作指令集"。我给你列个表：

| 信号 | 发送方式 | 作用 | 影响范围 |
|------|---------|------|---------|
| TERM/INT | kill -TERM | 快速关闭 | 立即停止所有 worker |
| QUIT | kill -QUIT | 优雅关闭 | 等待请求处理完再关闭 |
| HUP | kill -HUP | 重新加载配置 | 启动新 worker，优雅关闭旧 worker |
| USR1 | kill -USR1 | 重新打开日志 | 所有 worker 切换日志 fd |
| USR2 | kill -USR2 | 热升级二进制 | 启动新 master，新旧共存 |
| WINCH | kill -WINCH | 优雅关闭 worker | 配合 USR2 实现热升级收尾 |

**小胖**：HUP 是重新加载配置？我 reload 时一直用的 `nginx -s reload`，原来底层就是发 HUP？

**大师**：没错。`nginx -s reload` 的本质就是找到 master 进程的 pid，然后向它发送 SIGHUP。同理，`nginx -s reopen` 发送的是 SIGUSR1，`nginx -s stop` 发送的是 SIGTERM，`nginx -s quit` 发送的是 SIGQUIT。

**老王**：那 USR2 和 WINCH 的热升级是怎么回事？上次我升级 Nginx 版本， downtime 居然为零。

**大师**：这是 Nginx 进程模型最精妙的设计。当你发送 USR2 时，当前 master 进程会执行 `execve()` 启动新版本的 Nginx 二进制文件，同时把 listen socket 的 fd 通过环境变量 `NGINX` 传递给新 master。新 master 启动自己的 worker 开始接客，旧 master 的 worker 继续处理已有连接。等旧 worker 全部退出后，旧 master 自动消失。

**小胖**：这就像是……老店新开？老店还在招待老客人，新店已经开始接待新客人了。

**大师**：正是。而且如果新版本有问题，你还可以随时回滚——给旧 master 发 HUP，让它重新启动 worker，然后关闭新 master。

**老王**：master 和 worker 之间具体是怎么通信的？除了信号，还有别的机制吗？

**大师**：信号是操作系统层面的"广播"，只能传递简单指令。如果需要传递复杂数据，比如"新配置已就绪"或者"请切换日志"，Nginx 使用两种 IPC 机制：

1. **socketpair**：master 和每个 worker 之间建立一对 Unix 域套接字，用于双向数据传输。master 通过 socketpair 向 worker 发送命令（如关闭连接、重新初始化）。

2. **共享内存**：Nginx 的 `ngx_shm_zone_t` 结构用于在多个 worker 之间共享数据，比如 limit_req 的计数器、缓存的元数据、自定义模块的状态等。

**小白**：（不知何时也到了值班室）那 worker 之间能直接通信吗？

**大师**：不能。Nginx 的 worker 进程设计为**完全独立、互不干扰**。它们不共享内存（除了显式分配的共享内存区），不直接传递消息。这种设计虽然牺牲了一些灵活性，但带来了巨大的可靠性优势：一个 worker 崩溃或被阻塞，其他 worker 不受影响，master 会立刻 fork 一个新的 worker 补上。

**老王**：这就好比每个 worker 都是独立的工位，一个工位着火了，不影响其他工位干活。

**大师**：（笑）你这比喻比小胖的海底捞还贴切。

## 3. 项目实战

### 环境准备

- **Nginx 版本**：1.31.0（已编译安装到 /usr/local/nginx）
- **操作系统**：Linux（信号机制在 Windows 下不同）
- **工具**：ps、kill、strace、lsof

### 步骤一：启动 Nginx 并观察进程关系

```bash
# 启动 Nginx
sudo /usr/local/nginx/sbin/nginx

# 查看进程树
ps auxf | grep nginx
```

预期输出：

```
root      12345  0.0  0.1  12345  2345 ?        Ss   10:00   0:00 nginx: master process /usr/local/nginx/sbin/nginx
www-data  12346  0.0  0.2  13456  3456 ?        S    10:00   0:00  \_ nginx: worker process
www-data  12347  0.0  0.2  13456  3456 ?        S    10:00   0:00  \_ nginx: worker process
```

注意 worker 进程是 master 的子进程（通过 fork 创建）。

### 步骤二：信号操作实战

**2.1 优雅关闭（SIGQUIT）**

```bash
# 先产生一些并发请求（另开终端）
ab -n 10000 -c 100 http://localhost/ &

# 发送优雅关闭信号
sudo kill -QUIT $(cat /usr/local/nginx/logs/nginx.pid)

# 观察：worker 进程不会立刻消失，而是等当前请求处理完才退出
watch -n 1 'ps aux | grep "nginx:"'
```

**2.2 热重载配置（SIGHUP）**

```bash
# 修改配置文件（比如改 worker_connections）
sudo sed -i 's/worker_connections 1024/worker_connections 2048/' /usr/local/nginx/conf/nginx.conf

# 发送 HUP 信号
sudo kill -HUP $(cat /usr/local/nginx/logs/nginx.pid)

# 观察：master 启动新的 worker，旧 worker 优雅退出
ps auxf | grep nginx
```

**2.3 日志重开（SIGUSR1）**

```bash
# 模拟日志切割场景
sudo mv /usr/local/nginx/logs/access.log /usr/local/nginx/logs/access.log.$(date +%Y%m%d)

# 发送 USR1 信号
sudo kill -USR1 $(cat /usr/local/nginx/logs/nginx.pid)

# 验证：Nginx 会创建新的 access.log
ls -lh /usr/local/nginx/logs/access.log*
```

**2.4 热升级（SIGUSR2 + SIGWINCH）**

```bash
# 备份旧二进制
sudo cp /usr/local/nginx/sbin/nginx /usr/local/nginx/sbin/nginx.old

# 编译新版本（假设已完成）
# sudo cp /opt/nginx-1.32.0/objs/nginx /usr/local/nginx/sbin/nginx

# 发送 USR2 启动新 master
sudo kill -USR2 $(cat /usr/local/nginx/logs/nginx.pid)

# 观察：现在有两个 master 进程
ps auxf | grep nginx
# root  12345 ... nginx: master process /usr/local/nginx/sbin/nginx
# root  12400 ... nginx: master process /usr/local/nginx/sbin/nginx

# 验证新版本正常工作后，关闭旧 master 的 worker
sudo kill -WINCH $(cat /usr/local/nginx/logs/nginx.pid.oldbin)

# 旧 worker 全部退出后，旧 master 只剩一个进程
# 如果确认新版本稳定，关闭旧 master
sudo kill -QUIT $(cat /usr/local/nginx/logs/nginx.pid.oldbin)

# 如果新版本有问题，回滚：
# sudo kill -HUP $(cat /usr/local/nginx/logs/nginx.pid.oldbin)  # 重启旧 worker
# sudo kill -QUIT $(cat /usr/local/nginx/logs/nginx.pid)        # 关闭新 master
```

### 步骤三：用 lsof 观察日志文件句柄

```bash
# 查看 worker 进程持有的日志文件句柄
sudo lsof -p $(pgrep -f "nginx: worker" | head -n 1) | grep log
```

执行 USR1 前后分别运行此命令，可以观察到 access_log 的 fd 编号变化（旧 fd 关闭，新 fd 创建）。

### 步骤四：源码速览——信号处理与进程管理

打开 `src/os/unix/ngx_process_cycle.c`，查看 master 进程的信号处理逻辑：

```c
// src/os/unix/ngx_process_cycle.c
void ngx_master_process_cycle(ngx_cycle_t *cycle)
{
    ngx_uint_t  i;
    sigset_t    set;
    ngx_pid_t   pid;

    // 初始化信号集
    sigemptyset(&set);
    sigaddset(&set, SIGCHLD);   // 子进程状态变化
    sigaddset(&set, SIGALRM);   // 定时器
    sigaddset(&set, SIGIO);     // 异步 IO
    sigaddset(&set, SIGINT);    // Ctrl+C
    sigaddset(&set, SIGHUP);    // 终端挂起 / reload
    sigaddset(&set, SIGUSR1);   // 重新打开日志
    sigaddset(&set, SIGUSR2);   // 热升级
    sigaddset(&set, SIGWINCH);  // 窗口变化 / 优雅关闭 worker
    sigaddset(&set, SIGTERM);   // kill 默认信号
    sigaddset(&set, SIGQUIT);   // 优雅退出

    // 阻塞上述信号，直到进入 sigsuspend
    if (sigprocmask(SIG_BLOCK, &set, NULL) == -1) {
        ngx_log_error(NGX_LOG_ALERT, cycle->log, ngx_errno,
                      "sigprocmask() failed");
    }

    // 启动 worker 进程
    ngx_start_worker_processes(cycle, ccf->worker_processes, NGX_PROCESS_RESPAWN);

    // master 主循环
    for ( ;; ) {
        sigsuspend(&set);  // 阻塞等待信号

        // 根据 ngx_process 标志判断收到了什么信号
        if (ngx_reap) {           // SIGCHLD：有 worker 退出
            ngx_reap = 0;
            ngx_reap_children(cycle);
        }

        if (ngx_terminate) {      // SIGTERM/SIGINT：强制终止
            ngx_signal_worker_processes(cycle, ngx_signal_value(NGX_TERMINATE_SIGNAL));
        }

        if (ngx_quit) {           // SIGQUIT：优雅退出
            ngx_signal_worker_processes(cycle, ngx_signal_value(NGX_SHUTDOWN_SIGNAL));
        }

        if (ngx_reconfigure) {    // SIGHUP：重新加载配置
            ngx_reconfigure = 0;
            cycle = ngx_init_cycle(cycle);
            ngx_start_worker_processes(cycle, ccf->worker_processes, NGX_PROCESS_RESPAWN);
            ngx_signal_worker_processes(cycle, ngx_signal_value(NGX_SHUTDOWN_SIGNAL));
        }

        if (ngx_reopen) {         // SIGUSR1：重新打开日志
            ngx_reopen = 0;
            ngx_reopen_files(cycle, ccf->user);
            ngx_signal_worker_processes(cycle, ngx_signal_value(NGX_REOPEN_SIGNAL));
        }

        if (ngx_change_binary) {  // SIGUSR2：热升级
            ngx_change_binary = 0;
            ngx_exec_new_binary(cycle, ngx_argv);
        }
    }
}
```

**代码注释**：
- `sigprocmask(SIG_BLOCK, &set, NULL)` 在 master 初始化时阻塞所有工作信号，防止信号在关键操作期间到达
- `sigsuspend(&set)` 是 master 的"心跳停止点"，master 绝大部分时间都在这里休眠，直到信号到达
- `ngx_reap_children()` 负责回收退出的 worker 进程（waitpid），并根据需要重新 fork 新的 worker
- `ngx_signal_worker_processes()` 通过 socketpair 或 kill 向所有 worker 广播信号

查看 worker 进程的信号处理：

```c
// src/os/unix/ngx_process_cycle.c
static void ngx_worker_process_cycle(ngx_cycle_t *cycle, void *data)
{
    ngx_uint_t         i;
    ngx_connection_t  *c;

    // worker 初始化：设置进程标题、初始化模块、打开监听 socket
    ngx_worker_process_init(cycle, -1);

    // 进入事件循环
    for ( ;; ) {
        // 如果收到退出信号，且没有活跃连接，则退出
        if (ngx_exiting) {
            c = cycle->connections;
            for (i = 0; i < cycle->connection_n; i++) {
                if (c[i].fd != -1 && c[i].idle) {
                    c[i].close = 1;
                    c[i].read->handler(c[i].read);
                }
            }

            if (ngx_event_timer_rbtree.root == ngx_event_timer_rbtree.sentinel) {
                ngx_worker_process_exit(cycle);
            }
        }

        // 核心：处理事件（epoll_wait）
        ngx_process_events_and_timers(cycle);

        // 如果收到终止信号，直接退出
        if (ngx_terminate) {
            ngx_worker_process_exit(cycle);
        }

        // 如果收到优雅退出信号，标记 exiting
        if (ngx_quit) {
            ngx_quit = 0;
            ngx_setproctitle("worker process is shutting down");
            ngx_exiting = 1;
        }

        // 如果收到重新打开日志信号
        if (ngx_reopen) {
            ngx_reopen = 0;
            ngx_reopen_files(cycle, -1);
        }
    }
}
```

**代码注释**：
- worker 的主循环是**事件驱动**的：`ngx_process_events_and_timers()` 内部调用 epoll_wait
- 收到 `SIGQUIT`（优雅退出）时，worker 不会立刻退出，而是标记 `ngx_exiting = 1`，然后等待所有活跃连接关闭
- 收到 `SIGUSR1`（重新打开日志）时，worker 直接调用 `ngx_reopen_files()` 关闭旧 fd、打开新 fd

## 4. 项目总结

### 优点与缺点

| 维度 | Nginx 信号驱动进程模型 | 传统 systemd/service 重启 |
|------|----------------------|-------------------------|
| 配置更新 | 零停机（HUP 优雅切换） | 通常需重启服务 |
| 日志切割 | 零停机（USR1 重开 fd） | 通常需重启或 logrotate + copytruncate |
| 版本升级 | 零停机（USR2 热升级） | 必须停机替换 |
| 故障恢复 | 自动（master 监控 worker） | 依赖外部守护进程 |
| 信号复杂度 | 较高（需熟记信号语义） | 低（封装为命令） |
| 调试难度 | 较高（多进程、信号异步） | 低 |

### 适用场景

1. **7x24 在线服务**：电商、金融、游戏等对可用性要求极高的场景
2. **高频配置变更**：A/B 测试、灰度发布需要频繁调整路由规则
3. **日志密集型应用**：需要按小时/天切割日志，且不能中断服务
4. **持续交付流水线**：CI/CD 频繁部署新版本，要求零停机发布
5. **高可用集群**：单节点热升级配合负载均衡，实现集群级零停机

**不适用场景**：
1. **开发调试环境**：单进程模式（`master_process off;`）更简单直接
2. **短生命周期容器**：Serverless 或一次性任务，不需要热升级能力
3. **Windows 生产环境**：Nginx on Windows 使用多线程而非多进程模型，信号机制不同

### 注意事项

1. **不要用 kill -9 杀 master**：`SIGKILL` 无法被捕获，master 来不及通知 worker 优雅退出，会导致连接直接断开
2. **热升级后保留旧二进制**：USR2 升级后，旧 master 进程不会立刻退出，保留旧二进制便于回滚
3. **worker 退出后 master 会自动 fork**：如果 worker 异常崩溃（SIGSEGV），master 会立刻 fork 新 worker，但应排查崩溃原因

### 常见踩坑经验

**案例一：USR1 后日志不写入**
- **现象**：执行 `kill -USR1` 后，新的 access.log 大小始终为 0
- **根因**：worker 进程对新日志文件没有写权限（日志文件被 root 创建，worker 以 www-data 运行）
- **解决**：确保日志目录对 worker 用户可写，或使用 `chown www-data:www-data /var/log/nginx`

**案例二：HUP 后配置不生效**
- **现象**：修改 nginx.conf 后 reload，但新配置似乎没有生效
- **根因**：修改的是错误的配置文件路径，或 `nginx -t` 通过了但 `include` 的文件未更新
- **解决**：使用 `nginx -T` 查看最终合并后的完整配置，确认修改确实被加载

**案例三：USR2 热升级后旧 master 不退出**
- **现象**：新 master 已经稳定运行，但旧 master 和旧 worker 仍然存在
- **根因**：旧 worker 上还有长连接（如 WebSocket、大文件下载）未断开
- **解决**：设置 `worker_shutdown_timeout 30s;`，或等待长连接自然结束，或强制 `kill -TERM` 旧 master
