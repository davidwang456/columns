## 1. 项目背景

集团计划替换商业 API 网关，目标是性能可控、成本可控、能力可扩展。项目要求在一个版本内交付动态路由、鉴权、限流、追踪与灰度发布。

进入这一阶段后，Nginx 不再只是“转发工具”，而是稳定性、性能与工程效率的共同支点。仅靠参数经验已经不足以支撑复杂系统，必须把“设计假设、实验数据、故障演练、回滚策略”绑定成一个闭环。
本章沿着“问题定义 -> 方案取舍 -> 实验验证 -> 上线守护”的路径展开，确保方案可复制、可复盘、可演进。

## 2. 项目设计

**场景**：集团技术评审会，各部门 leader 要求在一个版本内交付能替代商业网关的方案。小胖被任命为 prototype 负责人，白板上画满了架构草图。

---

**小胖**：（激动地比划）这不就是盖一座"一站式购物中心"嘛——入口闸机验票（鉴权）、每家店限流排队（限流）、商场地图指路（路由）、还得记录每个顾客的行踪（trace）！

**小白**：比喻很形象——但最大挑战是流量像春节庙会一样波动，10 万人同时涌进来，闸机不能崩、地图不能乱、每家店不能挤爆。

**大师**：好，先拆解 API 网关的核心功能层：
1. **动态路由**：基于 Redis 或 etcd 动态更新 location 映射，不 reload。
2. **分布式限流**：基于共享内存 + 令牌桶算法，跨 worker 协同。
3. **JWT 鉴权**：auth_request 模块调用内部鉴权服务，或 access_by_lua 直接校验。
4. **Trace 注入**：请求进入时生成 $request_id，通过 proxy_set_header 透传上游。

**小胖**：这跟我去万达广场一样——门口扫一次码（鉴权），进去后每家店自己想进就进（路由），但喜茶排队太长会限流，商场广播全程跟着你播报（trace）。

---

**小白**：动态路由不 reload 怎么实现？Nginx 的 location 不是静态的吗？

**大师**：好眼力。纯 Nginx location 是静态的，但我们可以用**两级代理**架构——入口 Nginx 用 lua_shared_dict 存储路由表，接到请求后查询 dict 获取目标 upstream，通过 ngx.exec 动态转发。路由表变更时通过 REST API 直接写入共享内存，不需要 reload。就像购物中心的电子导览屏——不用拆了重建，后台更新数据，屏幕实时刷新。

**小胖**：那限流怎么保证每个 worker 看到的"剩余令牌"是一样的？

**大师**：用 ngx_http_limit_req_module 配合共享内存 zone 实现。共享内存中的计数通过自旋锁互斥访问。如果想做更灵活的全局总限流（比如 10 万 QPS 上限），需要引入 Redis 做第二级"总闸口"——本地共享内存做第一级缓冲，Redis 做全局协调。

---

**小白**：JWT 校验的性能呢？每次请求都解密 RSA 扛得住 10 万 QPS 吗？

**大师**：第一，JWT 校验用 HMAC-SHA256（对称签名）替代 RSA，性能高一个数量级。第二，在 auth_request 阶段做校验，通过后把 payload 通过 header 传给上游，上游不用再验。第三，配合 token 缓存——相同 token 在一段时间内不做二次验签。这些组合下来，单机 10 万 QPS 完全可支撑。

**小胖**：就是进商场扫脸一次（auth_request），发你个"已验票"手环（payload header），然后在手环有效期内随便逛，不用再验。

**技术映射**：**两级路由**（静态 location + lua 动态转发）实现无 reload 动态路由；**双层限流**（共享内存 + Redis）保障全局限流一致性；**JWT 校验**通过 auth_request + HMAC + token 缓存实现高性能鉴权；**请求全链路追踪**通过 $request_id 透传实现。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等级 Linux 发行版）
- Nginx：1.31.0（建议保留 debug 能力）
- 工具：`curl`、`wrk`、`ss`、`jq`
- 按章补充：数据库/DNS/WebSocket/K8s/profiling 工具

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：构建可运行、可验证、可回滚的最小基线。

```nginx
limit_req_zone $binary_remote_addr zone=api_rate:50m rate=200r/s;
upstream api_cluster {
    server 10.2.0.11:8080;
    server 10.2.0.12:8080;
}
server {
    listen 8080;
    location /api/ {
        auth_request /_auth;
        limit_req zone=api_rate burst=100 nodelay;
        proxy_set_header X-Trace-Id $request_id;
        proxy_pass http://api_cluster;
    }
}
```

**运行结果（预期）**：
- 配置检查通过；
- 关键路径连通；
- `error.log` 无持续异常。

**可能遇到的坑**：仅做功能验证，不做压力与异常验证，导致上线后才暴露边界问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用数据而非主观感受评估改动价值。

```bash
curl -i -H "Authorization: Bearer demo" http://127.0.0.1:8080/api/order
wrk -t12 -c4000 -d90s http://127.0.0.1:8080/api/order
```

**建议观测指标**：
- 吞吐：QPS/TPS
- 时延：P95/P99
- 可靠性：5xx、超时率、重试率
- 资源：CPU、内存、活动连接

**可能遇到的坑**：压测时间太短、场景太单一。建议至少覆盖平峰/高峰/故障三种情形。

### 步骤三：故障注入与回滚演练

**步骤目标**：确保故障发生时可快速止损。

```bash
# 应用变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线单个上游节点；
2. 注入后端延迟或错误码；
3. 叠加突发流量验证尾延迟。

**本章实战目标**：交付生产级 API 网关蓝图与可执行落地方案。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 稳定性 | 有明确的演练与回滚机制 | 前期建设成本增加 |
| 性能 | 可持续、可量化优化 | 依赖监控与压测体系 |
| 团队协作 | 结论标准化、可传承 | 对执行纪律要求高 |

### 适用场景

1. 高峰明显、故障成本高的核心业务。
2. 网关承载治理能力的平台化系统。
3. 希望把经验运维升级为工程化运维的团队。

**不适用场景**：
- 低流量 demo 或短生命周期项目。
- 缺少基础观测能力、无法形成验证闭环的团队。

### 注意事项

1. 无基线不优化：先量化现状再改动。
2. 单变量优先：保障结论可归因。
3. 回滚先行：上线前先验证止损路径。

### 常见踩坑经验

- 只看平均值，不看尾延迟。
- 只做成功路径，不做故障演练。
- 变更记录缺失，导致后续无法复盘。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
