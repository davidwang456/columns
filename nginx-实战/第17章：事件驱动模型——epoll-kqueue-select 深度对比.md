# 第17章：事件驱动模型——epoll/kqueue/select 深度对比

> 源码关联：src/event/modules/ngx_epoll_module.c、src/event/modules/ngx_kqueue_module.c、src/event/ngx_event.c
> 批次说明：第 1 批（精修版 1/5）

**关键词摘要**：事件驱动、epoll vs select、高并发连接、I/O 多路复用、性能基线

---

## 1. 项目背景

鲜果园在一场“限时半价”活动中，API 网关连接数在 90 秒内从 6k 涨到 11w。监控显示 CPU 使用率仅 62%，但 P99 延迟从 38ms 飙到 690ms。最开始大家怀疑是 Java 服务 Full GC，结果排查后端发现核心接口耗时稳定，真正的瓶颈出现在网关事件模型：测试环境误用了 select，导致高连接数下 fd 扫描成本线性增长。

在复盘会上，运维团队给出一个关键结论：以前“能跑就行”的配置策略，在中级阶段会迅速失效。因为系统面对的已经不是单点问题，而是“流量规模 + 协议复杂度 + 组织协作成本”的复合压力。
因此本章不只讲参数含义，还会给出一套可复用的方法：**先建基线，再做单变量实验，最后用故障注入验证边界**。这套方法会贯穿中级篇后续章节。

---

从知识路径上看，本章与上一章《前一章》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：凌晨应急会，值班群还在刷屏，白板上画着连接数从 6k 飙到 11w 的折线图，三人大眼瞪小眼。

---

**小胖**：（啃着面包）我看这 cpu 才 62%，内存也够，怎么延迟就上天了呢？就跟食堂打饭似的，明明窗口还有空位，队伍却走得贼慢。

**小白**：问题很可能出在 IO 模型上。测试环境默认用的是 select，生产环境是 epoll，但没人在意这个差异。select 的 fd 集合是线性扫描——连接数越多，每轮扫描越慢。

**大师**：（在白板上画了几张图）小白说得对。小胖，你想想你们家大促排队：select 是“保安挨个问每桌客人吃完了没有”，哪怕只有一桌要结账，他也得把整层楼走一遍。epoll 呢？是“客人按铃，服务员才过来”——有事件才通知，不会空转。

**小胖**：哦，那 epoll 就是装了个呼叫铃呗。那 ET 和 LT 又是什么？一个铃一直响和按一次的区别？

---

**小白**：EPOLLET（边缘触发）是文件描述符状态变化时才通知一次，你一次没读完就不再提醒了，得自己循环读完。EPOLLLT（水平触发）是只要还有数据没读完，每次 epoll_wait 都通知你。

**大师**：类比一下——LT 是你家水龙头没关紧，一直在滴水，物业就一直在敲门提醒你。ET 是水龙头突然爆了，物业只喊你一次，你爱修不修，不修下次漏水也不管了。

**小胖**：那 ET 不是很容易丢事件？

**大师**：所以用 ET 必须搭配非阻塞 IO 和循环读写，保证一次事件把数据全部处理完。Nginx 的 epoll 模块就用的 ET 模式——这是它能用少量 worker 撑起百万连接的关键。

**技术映射**：ET 模式下，Nginx 的事件循环一次性收完所有就绪 fd，然后逐个处理。每次 epoll_wait 返回的只是“有变化的连接”，而不是全量 fd 集合——复杂度从 select 的 O(n) 降为 O(1)。

---

**小白**：那为什么 select 和 poll 在小连接数下也没问题？阈值在哪？

**大师**：经验上看，连接数 < 1000 时 select 还能凑合——食堂几个窗口，保安巡逻一圈不费事。到了 1w+ 连接，每轮扫描几万个 fd，再加上 fd 集合在用户态和内核态之间来回拷贝，开销直接炸了。

**小胖**：这就跟双十一食堂开 100 个窗口，保安还一个一个窗口去问“你吃完了吧”一样傻。

**技术映射**：select 受限于 FD_SETSIZE（默认 1024）和位图全量拷贝，poll 虽然解除了 1024 限制但仍是链式扫描。epoll 通过红黑树管理注册 fd、就绪链表返回活动 fd，只在有事件时才触发拷贝——这才是大规模并发连接的基石。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04 / Rocky Linux 9（任选其一）
- Nginx：1.31.0（建议开启 debug）
- 工具：`wrk`、`curl`、`ss`、`jq`、`grpcurl`（第21章需要）

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：建立最小可运行配置

**步骤目标**：让核心路径先稳定跑通，保证后续实验有可信起点。

```nginx
events {
    worker_processes auto;
    worker_connections 16384;
    use epoll;
    multi_accept on;
}
http {
    server {
        listen 8080;
        location /ping { return 200 "ok\n"; }
    }
}
```

**运行结果（预期）**：
- `nginx -t` 输出 syntax is ok / test is successful
- 健康检查接口返回 200
- `error.log` 无持续性报错

**可能遇到的坑**：配置能通过但链路没通，多数是上游地址、证书路径或防火墙规则未对齐。

### 步骤二：做基线压测与指标采集

**步骤目标**：获得“优化前”与“优化后”的可比数据。

```bash
wrk -t8 -c10000 -d60s http://127.0.0.1:8080/ping
ss -s
curl -s http://127.0.0.1:8080/nginx_status
```

**建议记录指标**：
- 吞吐：QPS / RPS
- 延迟：P50、P95、P99
- 稳定性：5xx 比例、连接失败率、重试比例
- 资源：CPU、内存、活动连接数

**可能遇到的坑**：压测机成为瓶颈，导致测试结果误判。建议压测端与被测端分离，且至少跑 2-3 轮取中位值。

### 步骤三：故障注入与回滚演练

**步骤目标**：确认异常场景下系统仍可控，发布后具备止损能力。

```bash
# 触发配置变更
sudo nginx -t && sudo nginx -s reload
# 回滚（示例）
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 人工下线一个上游节点。
2. 将一个上游延迟注入到 1~2s。
3. 模拟突发流量（瞬时并发提升 2~3 倍）。

**本章实战目标**：在相同硬件上完成 select 与 epoll 对照压测，形成“连接规模 -> 事件模型”选型基线。

### 完整代码清单

- 建议仓库目录：`ops/nginx/ch17/`
- 包含文件：`nginx.conf`、`upstream.conf`、`bench.sh`、`rollback.sh`、`README.md`
- 推荐把实验记录沉淀为 `result.md`，保留每次变更的指标截图与结论

### 测试验证

```bash
# 基础连通性验证（示例）
curl -i http://127.0.0.1:8080/health
```

验证要点：
- 关键接口返回码符合预期（2xx/4xx/5xx与设计一致）；
- 关键日志字段完整（如 request_id、upstream 耗时、状态码）；
- 在小流量压测下无异常错误峰值。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 方案代价 |
|---|---|---|
| 性能收益 | 能针对性改善吞吐与尾延迟 | 需要压测与监控支撑 |
| 稳定性 | 有故障注入与回滚兜底 | 发布流程更严格 |
| 协作效率 | 开发/测试/运维可共享同一结论 | 对变更纪律要求更高 |

### 适用场景

1. 日常有流量峰谷，且峰值抖动明显的系统。
2. 网关承担统一入口、对稳定性要求高的平台。
3. 需要沉淀标准化运维流程的中大型团队。

**不适用场景**：
- 单机低流量 demo 项目。
- 暂无监控与压测能力、难以形成验证闭环的团队。

### 注意事项

1. 先统一指标口径，再讨论优化效果。
2. 单次只改一个关键变量，方便归因。
3. 每次上线都保留可在 1 分钟内执行的回滚路径。

### 常见踩坑经验

- 只看平均延迟，忽略 P99 与错误率。
- 基线记录不完整，导致结论不可复验。
- “一把梭”改太多参数，上线后无法定位问题来源。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
