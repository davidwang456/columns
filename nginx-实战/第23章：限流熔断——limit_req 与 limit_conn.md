## 1. 项目背景

登录接口被脚本流量突刺后，认证服务线程池在 30 秒内耗尽，正常用户也被拖垮。事后复盘发现团队只做了“接口优化”，没有做入口防护，导致网关成为放大器。

这类问题在中级阶段非常典型：指标表面好看，但业务体验在关键节点失真。团队过去依赖“经验调参”，现在必须切换为“指标驱动 + 故障演练 + 可回滚”的交付模式。
本章会沿着“真实业务痛点 -> 方案博弈 -> 逐步落地 -> 验证闭环”来推进，确保这套方案不仅能跑，还能在高峰与异常中站得住。

---

从知识路径上看，本章与上一章《高级缓存架构与 Cache Purge》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：凌晨应急复盘会，值班群刚消停，三人对着登录接口的流量曲线图，峰值像刀切一样陡峭。

---

**小胖**：（喝着红牛）限流嘛，不就是门口保安大哥数人头，一分钟进 100 个，多的挡外面呗。这 limit_req 有啥好研究的？

**小白**：但如果前 10 秒来了 200 个请求，后 50 秒只有 50 个，平均下来 5r/s 没超，可前 10 秒后端已经跪了。限流的粒度怎么定？按 IP、按用户、还是按接口？

**大师**：（在纸上画了一个漏斗）小胖说的保安只数人，但限流的核心是“水龙头”问题。你们见过厨房里那种细颈油瓶吗？倒油的时候，瓶口再大也只能匀速流出一股细油。这就是漏桶算法——先来后到，大家排队，超出桶深的请求直接拒绝或排队等候。

**小胖**：哦，那 burst 不就是把瓶子稍微倾斜一下，让油在桶里先蓄一蓄，再一股一股倒出去吗？

**大师**：（笑着点头）你这比喻比我讲的还形象。没错，`burst` 就是桶的容量，允许短时间内超过 `rate` 的请求暂存在桶里，再按配置的速率慢慢处理。而 `nodelay` 就是“既然桶能存住，队列里的请求不要额外延迟”，直接以最快速度放行桶里已存的数量，后续请求再按 rate 限流。但注意——burst + nodelay 不是不限制，总量还是被 `rate` 兜住的。

**大师**：**技术映射**：`limit_req_zone` 定义漏桶参数（`rate`），`limit_req` 设置 `burst`（桶深）和 `nodelay`（是否排队等待）。漏桶算法的特点是“输出速率恒定”，适合保护后端不被突发打满。

---

**小白**：那 token bucket 又是什么？跟漏桶有什么区别？

**大师**：漏桶是“水多了倒掉”，token bucket 是“有票才能进”。想象一家自助餐厅，门口有个票箱，每分钟往里面放 10 张餐票。客人来了，有票就进，没票就等。客人如果攒了一叠票，可以连续进去吃——这允许短时间突发。漏桶不管你手里有没有票，反正出口就那么大，匀速放行。

**小白**：所以 token bucket 适合允许短期突发的场景，漏桶适合需要严格平滑的场景？

**大师**：对。Nginx 的 `limit_req` 实现的是漏桶。选哪个取决于业务——支付接口用漏桶，平滑最重要；下载服务用 token bucket，允许用户短时间加速。而分布式场景下限流还需要共享状态，单机漏桶不够用，得用 Redis + Lua 实现中心化的令牌桶。

**大师**：

**技术映射**：漏桶（Leaky Bucket）保证输出速率恒定，消除突发；令牌桶（Token Bucket）允许一定程度的突发，适合带宽突发场景。Nginx 内置漏桶，分布式限流需要借助 Redis + Lua 扩展。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等发行版）
- Nginx：1.31.0
- 工具：`curl`、`wrk`、`ss`、`jq`
- 附加工具：第 24 章建议 `mysql` / `dig`，第 25 章建议 `websocat`

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：先建立可运行基线，避免“边改边猜”。

```nginx
limit_req_zone $binary_remote_addr zone=ip_rate:20m rate=10r/s;
limit_req_zone $http_x_user_id zone=user_rate:20m rate=5r/s;
limit_conn_zone $binary_remote_addr zone=ip_conn:20m;
server {
    listen 8080;
    location /login {
        limit_req zone=ip_rate burst=20 nodelay;
        limit_req zone=user_rate burst=10;
        limit_conn ip_conn 20;
        limit_req_status 429;
        proxy_pass http://auth_backend;
    }
}
```

**运行结果（预期）**：
- 配置校验通过；
- 关键接口连通；
- `error.log` 无持续性异常。

**可能遇到的坑**：仅用功能联调代替压测与异常演练，导致上线后才暴露问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用可比较的数据判断方案优劣。

```bash
wrk -t8 -c500 -d30s http://127.0.0.1:8080/login
awk '$9==429 || $9==503 {print}' /var/log/nginx/access.log | wc -l
```

**建议关注指标**：
- 吞吐：QPS / TPS
- 延迟：P95、P99
- 稳定性：错误率、超时率、重试率
- 资源：活动连接、CPU、内存

**可能遇到的坑**：压测时间太短，掩盖连接池、缓存、GC 等“慢热型问题”。建议最少跑 5~10 分钟稳定段。

### 步骤三：故障注入与回滚演练

**步骤目标**：验证系统在异常下仍可控。

```bash
# 触发变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线一个后端节点；
2. 人工增加后端延迟（如 tc 注入）；
3. 叠加突发流量观察尾延迟。

**本章实战目标**：实现按维度限流与连接保护，确保突发流量下核心接口可用。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 可用性 | 有明确保护与回滚机制，异常可控 | 建设初期流程更重 |
| 性能 | 能在真实指标上做持续优化 | 需要持续压测资源 |
| 协作 | 跨团队共享同一评估标准 | 对执行纪律要求更高 |

### 适用场景

1. 高峰波动明显、对可用性要求高的业务。
2. 网关承担统一流量治理职责的平台。
3. 希望建立“可复盘、可复制”优化流程的团队。

**不适用场景**：
- 单机低流量且无明显增长预期的项目。
- 缺少监控与压测基础设施、无法闭环验证的团队。

### 注意事项

1. 指标先行：没有基线就没有优化结论。
2. 变更最小化：一次只改一个关键变量。
3. 回滚可执行：任何优化都必须自带止损路径。

### 常见踩坑经验

- 只追求命中率/QPS，不看业务正确性。
- 缺少异常注入，线上首遇故障才发现边界。
- 变更记录缺失，后续团队无法继承经验。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
