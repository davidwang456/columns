# 第18章：连接池管理与 Keepalive 优化

> 源码关联：src/http/modules/ngx_http_upstream_keepalive_module.c、src/http/ngx_http_upstream.c
> 批次说明：第 1 批（精修版 1/5）

**关键词摘要**：upstream keepalive、连接复用、握手开销、连接池上限、P99 优化

---

## 1. 项目背景

订单查询接口在早高峰频繁出现 connect timeout。应用日志显示业务处理仅 12ms，但网关到后端平均建连耗时 45ms，TLS 协商更是放大抖动。核心原因是 Nginx 每次请求都新建 upstream 连接，未启用 keepalive 复用。

在复盘会上，运维团队给出一个关键结论：以前“能跑就行”的配置策略，在中级阶段会迅速失效。因为系统面对的已经不是单点问题，而是“流量规模 + 协议复杂度 + 组织协作成本”的复合压力。
因此本章不只讲参数含义，还会给出一套可复用的方法：**先建基线，再做单变量实验，最后用故障注入验证边界**。这套方法会贯穿中级篇后续章节。

---

从知识路径上看，本章与上一章《事件驱动模型——epoll/kqueue/select 深度对比》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：发布评审会，运维把一张“建连耗时分布图”投在大屏上——早高峰的尾巴拖得老长，产品经理在旁边一直叹气。

---

**小胖**：（嚼着饼干）不就是每次请求都重新连一次后端吗？多连几次又不会少块肉，建个 TCP 连接才多大点事？

**小白**：一个 TCP 连接三次握手最少 1 个 RTT，算上 TLS 协商还要再加 2 个 RTT。网关每次请求都重建连接，早高峰上万 QPS，光握手就吃掉几十毫秒。我们后端业务处理才 12ms，一半时间都花在“打招呼”上了。

**大师**：（在白板上画了一条时间线）小胖，你去食堂打饭——每次打饭都要重新排队、重新认识打菜阿姨、重新报一遍菜名，累不累？

**小胖**：那也太蠢了……我一般都认准一个窗口，阿姨认识我了，直接说“老样子”就行。

**大师**：对，这就叫“连接复用”。Nginx 的 upstream keepalive 就是把用过的 TCP 连接放回池子里，下个请求来了直接拿现成的，省掉三次握手和慢启动。

---

**小白**：那 keepalive 池设多大合适？设大了会占用 fd 资源，设小了复用率不够。

**大师**：好问题。keepalive 池的大小决定了能缓存多少空闲连接回收到池里。好比食堂高峰期，你保留 10 个“老顾客窗口”——太少的话新请求还是要排新队，太多的话窗口空着也浪费服务员。

**小胖**：那要是池子满了咋办？新来的请求还能用吗？

**大师**：满了就用 LRU 策略淘汰最久未用的连接，关闭它再建新连接。不会阻塞请求，只是复用率下降。

**技术映射**：keepalive 指令的值是每个 worker 进程中缓存的最大空闲连接数。计算公式参考：`keepalive = 后端节点数 × 每个节点预期并发 / worker_processes`。过小则复用率低，过大则占用过多 fd。

---

**小白**：还有个问题——keepalive_timeout 设太长会不会拖死后端？后端如果有连接超时限制，Nginx 这边还挂着不关，就会产生上下游不一致。

**大师**：（点头）你点到了关键。keepalive_timeout 是网关侧的空闲连接存活时间，必须和后端服务器的 keepalive 配置对齐。好比食堂规定“老顾客窗口占座不能超过 5 分钟”，后端也说“5 分钟没动静我就收桌了”——两边得对表。

**小胖**：这我懂，就跟外卖柜一样——柜子说能放 30 分钟，你 30 分钟不来取就回收了，超时被拿走别怪柜子。

**技术映射**：HTTP 协议层面，Nginx 通过 `proxy_http_version 1.1` 和 `proxy_set_header Connection ""` 启用上游 keepalive。网关和后端的 keepalive_timeout 应一致，避免 half-open 连接浪费资源。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04 / Rocky Linux 9（任选其一）
- Nginx：1.31.0（建议开启 debug）
- 工具：`wrk`、`curl`、`ss`、`jq`、`grpcurl`（第21章需要）

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：建立最小可运行配置

**步骤目标**：让核心路径先稳定跑通，保证后续实验有可信起点。

```nginx
upstream order_api {
    server 10.0.0.11:9000 max_fails=2 fail_timeout=10s;
    server 10.0.0.12:9000 max_fails=2 fail_timeout=10s;
    keepalive 256;
}
server {
    listen 8080;
    location /order/ {
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_connect_timeout 500ms;
        proxy_read_timeout 2s;
        proxy_pass http://order_api;
    }
}
```

**运行结果（预期）**：
- `nginx -t` 输出 syntax is ok / test is successful
- 健康检查接口返回 200
- `error.log` 无持续性报错

**可能遇到的坑**：配置能通过但链路没通，多数是上游地址、证书路径或防火墙规则未对齐。

### 步骤二：做基线压测与指标采集

**步骤目标**：获得“优化前”与“优化后”的可比数据。

```bash
wrk -t8 -c1200 -d90s http://127.0.0.1:8080/order/query
watch -n 1 "ss -ant | rg ':9000' | wc -l"
```

**建议记录指标**：
- 吞吐：QPS / RPS
- 延迟：P50、P95、P99
- 稳定性：5xx 比例、连接失败率、重试比例
- 资源：CPU、内存、活动连接数

**可能遇到的坑**：压测机成为瓶颈，导致测试结果误判。建议压测端与被测端分离，且至少跑 2-3 轮取中位值。

### 步骤三：故障注入与回滚演练

**步骤目标**：确认异常场景下系统仍可控，发布后具备止损能力。

```bash
# 触发配置变更
sudo nginx -t && sudo nginx -s reload
# 回滚（示例）
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 人工下线一个上游节点。
2. 将一个上游延迟注入到 1~2s。
3. 模拟突发流量（瞬时并发提升 2~3 倍）。

**本章实战目标**：建立 upstream keepalive 的参数基线，验证连接复用对 QPS、P99、后端连接数的综合影响。

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 方案代价 |
|---|---|---|
| 性能收益 | 能针对性改善吞吐与尾延迟 | 需要压测与监控支撑 |
| 稳定性 | 有故障注入与回滚兜底 | 发布流程更严格 |
| 协作效率 | 开发/测试/运维可共享同一结论 | 对变更纪律要求更高 |

### 适用场景

1. 日常有流量峰谷，且峰值抖动明显的系统。
2. 网关承担统一入口、对稳定性要求高的平台。
3. 需要沉淀标准化运维流程的中大型团队。

**不适用场景**：
- 单机低流量 demo 项目。
- 暂无监控与压测能力、难以形成验证闭环的团队。

### 注意事项

1. 先统一指标口径，再讨论优化效果。
2. 单次只改一个关键变量，方便归因。
3. 每次上线都保留可在 1 分钟内执行的回滚路径。

### 常见踩坑经验

- 只看平均延迟，忽略 P99 与错误率。
- 基线记录不完整，导致结论不可复验。
- “一把梭”改太多参数，上线后无法定位问题来源。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
