## 1. 项目背景

业务反馈偶发 499，但网关和应用互相“甩锅”。要终结争议，必须还原请求从 accept 到 finalize 的完整链路。

进入这一阶段后，Nginx 不再只是“转发工具”，而是稳定性、性能与工程效率的共同支点。仅靠参数经验已经不足以支撑复杂系统，必须把“设计假设、实验数据、故障演练、回滚策略”绑定成一个闭环。
本章沿着“问题定义 -> 方案取舍 -> 实验验证 -> 上线守护”的路径展开，确保方案可复制、可复盘、可演进。

---

从知识路径上看，本章与上一章《事件循环源码——epoll 实现原理》形成前后呼应：上一章解决的是“局部能力成立”，本章要进一步回答“在真实流量和复杂协作下如何稳定成立”。

## 2. 项目设计

**场景**：晚上十一点，值班群炸了——客户端报 499 但 access.log 显示 200。小白要求三方会审，逐行过请求生命周期源码。

---

**小胖**：（揉着眼睛）这不就跟外卖骑手取餐一样嘛——商家说做好了（200），但骑手迟迟没拿到（499），到底是商家没真做完，还是骑手没等？

**小白**：问题的核心是：Nginx 什么时候把请求标记为"完成"的？499 是客户端断开触发的，但日志里那个 200 是谁在什么时候写入的？

**大师**：（打开 src/http/ngx_http_request.c）让我们把一次 HTTP 请求走一遍：①accept 建立连接调用 ngx_http_init_connection；②读取请求行 ngx_http_process_request_line；③解析请求头 ngx_http_process_request_headers；④进入 11 个 phase handler 按顺序执行；⑤调用 ngx_http_output_filter 发送响应；⑥请求完毕执行 ngx_http_finalize_request。

**小胖**：就跟去餐厅吃饭一样——进门领位（accept）、看菜单点菜（parse request line）、加饮料（parse headers）、厨房按顺序做菜（phase handlers）、服务员端上来（output filter）、吃完结账走人（finalize）。

---

**小白**：那 499 和 200 同时出现是怎么回事？客户端都断开了怎么还能写出 200？

**大师**：这就是**时序问题**。ngx_http_finalize_request 调用 output_filter 把响应发出去后检查返回值——如果发送成功但客户端已断开，Nginx 在 flush 时得到 EPIPE 错误。此时它依然把 200 写入 access log（因为业务处理成功了），然后才把这个请求标记为 done。客户端侧的 499 发生在断开那瞬间。两条日志记录的是同一时间窗口里不同角色的视角：服务端说"我答完了"，客户端说"我没等到"。

**小胖**：所以不是谁撒谎了，而是摄像头没对准同一个人？

**大师**：精确。要根治这类争议，需要在 11 个 phase 的关键节点注入 trace ID 和耗时埋点——从 ngx_http_init_request 到 ngx_http_core_run_phases 再到 ngx_http_request_handler，把请求的每一步都打上时间戳。

---

**小白**：phase handlers 的执行顺序是写死在代码里的吗？能不能自定义？

**大师**：11 个 phase 的顺序是固定枚举（NGX_HTTP_POST_READ_PHASE 到 NGX_HTTP_LOG_PHASE），但每个 phase 内的 handler 是链表结构，模块可以通过 postconfiguration 钩子往指定 phase 里插入自己的 handler。这就像餐厅的菜品顺序——前菜（rewrite）、主菜（content）、甜点（log）的顺序不变，但你可以决定今天的前菜是沙拉还是汤。

**技术映射**：**accept → read → phase handlers → filter → finalize** 构成请求完整生命周期；**phase handlers** 是责任链模式，按固定阶段挂载自定义逻辑；**finalize_request** 触发资源释放和日志写入。

## 3. 项目实战

### 环境准备

- 系统：Ubuntu 22.04（或同等级 Linux 发行版）
- Nginx：1.31.0（建议保留 debug 能力）
- 工具：`curl`、`wrk`、`ss`、`jq`
- 按章补充：数据库/DNS/WebSocket/K8s/profiling 工具

```bash
sudo nginx -V
sudo nginx -t
```

### 步骤一：最小配置落地

**步骤目标**：构建可运行、可验证、可回滚的最小基线。

```nginx
log_format trace '$request_id $status $request_time $upstream_response_time $upstream_status';
access_log logs/trace.log trace;
```

**运行结果（预期）**：
- 配置检查通过；
- 关键路径连通；
- `error.log` 无持续异常。

**可能遇到的坑**：仅做功能验证，不做压力与异常验证，导致上线后才暴露边界问题。

### 步骤二：基线压测与指标采集

**步骤目标**：用数据而非主观感受评估改动价值。

```bash
curl -H "X-Request-Id: trace-001" http://127.0.0.1:8080/api
rg 'trace-001' logs/trace.log
```

**建议观测指标**：
- 吞吐：QPS/TPS
- 时延：P95/P99
- 可靠性：5xx、超时率、重试率
- 资源：CPU、内存、活动连接

**可能遇到的坑**：压测时间太短、场景太单一。建议至少覆盖平峰/高峰/故障三种情形。

### 步骤三：故障注入与回滚演练

**步骤目标**：确保故障发生时可快速止损。

```bash
# 应用变更
sudo nginx -t && sudo nginx -s reload
# 回滚示例
sudo cp /etc/nginx/nginx.conf.bak /etc/nginx/nginx.conf
sudo nginx -t && sudo nginx -s reload
```

**故障注入建议**：
1. 下线单个上游节点；
2. 注入后端延迟或错误码；
3. 叠加突发流量验证尾延迟。

**本章实战目标**：建立请求全链路追踪视角，定位阶段性超时与异常。

### 完整代码清单

- 目录建议：`ops/nginx/ch35/`
- 文件建议：`nginx.conf`、`upstream.conf`、`bench.sh`、`fault_inject.sh`、`rollback.sh`
- 记录建议：`result.md`（参数、指标、结论、回滚点）

### 测试验证

验证要点：
- 关键接口返回码符合预期（2xx/4xx/5xx与设计一致）；

## 4. 项目总结

### 优点与缺点

| 维度 | 方案优势 | 潜在代价 |
|---|---|---|
| 稳定性 | 有明确的演练与回滚机制 | 前期建设成本增加 |
| 性能 | 可持续、可量化优化 | 依赖监控与压测体系 |
| 团队协作 | 结论标准化、可传承 | 对执行纪律要求高 |

### 适用场景

1. 高峰明显、故障成本高的核心业务。
2. 网关承载治理能力的平台化系统。
3. 希望把经验运维升级为工程化运维的团队。

**不适用场景**：
- 低流量 demo 或短生命周期项目。
- 缺少基础观测能力、无法形成验证闭环的团队。

### 注意事项

1. 无基线不优化：先量化现状再改动。
2. 单变量优先：保障结论可归因。
3. 回滚先行：上线前先验证止损路径。

### 常见踩坑经验

- 只看平均值，不看尾延迟。
- 只做成功路径，不做故障演练。
- 变更记录缺失，导致后续无法复盘。

### 推广计划提示

- **开发团队**：先复用本章最小配置与脚本，按“单变量”方式做参数实验并沉淀变更记录。
- **测试团队**：优先补齐异常路径用例（超时、重试、限流、故障转移），并固化回归清单。
- **运维团队**：将监控阈值、告警策略与回滚脚本纳入发布流程，确保高峰期可快速止损。
- **协作顺序建议**：开发先完成方案基线 -> 测试做功能/压力/故障验证 -> 运维执行灰度与上线守护。
