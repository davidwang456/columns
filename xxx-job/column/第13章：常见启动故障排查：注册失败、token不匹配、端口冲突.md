# 第13章：启动排障——Admin 与 Executor 起不来怎么办

## 1. 项目背景

XXL-JOB 的入门阻塞点通常不是写 JobHandler，而是“服务起不来、起了也不通、通了也跑不动”。在真实团队中，这类问题高频出现在三种场景：第一次搭建环境、版本升级后重启、新增节点扩容。常见现象包括 Admin 页面无法访问、Executor 在管理台始终离线、手动触发提示地址不可达、看似成功启动但任务无任何执行日志。很多同学会习惯性反复重启，短期可能碰巧恢复，但问题并没有被解释，后续仍会反复出现。

启动排障的核心不是“猜配置”，而是建立一条标准化诊断路径：先确认进程和端口，再确认依赖和配置，最后做端到端功能验证。只有把每一步都变成可验证动作，团队才能减少经验依赖。你会发现，大部分启动故障都不是复杂 bug，而是配置漂移、依赖未就绪、鉴权不一致、网络不可达这四类基础问题。本章将把这些问题沉淀为一套可以执行、可以复用、可以交接的排障清单，让新人也能在半小时内从“服务起不来”走到“任务闭环成功”。

## 2. 项目设计（剧本式交锋对话）

**小胖**：Admin 报错了，我一般先重启几次。  
**小白**：重启是动作，不是结论。你连报错层级都没分清。  
**大师**：排障先分层：进程层、依赖层、配置层、链路层。每层都要有“最小验证动作”。  
**技术映射**：分层排障 = 把不可控问题变成可控步骤。

**小胖**：Executor 明明启动成功，为啥控制台显示离线？  
**小白**：有可能 admin 地址错、token 不一致、端口被 NAT 映射错。  
**大师**：还有 `appname` 冲突与注册地址不可回连。不要只看“本机能启动”，要看“Admin 能否回调”。  
**技术映射**：启动成功不等于可调度，回调可达才算联通。

**小胖**：页面能登录，说明系统没问题吧？  
**小白**：页面可用只说明 Web 层基本正常。  
**大师**：必须做端到端：建任务 -> 触发 -> 查看执行日志 -> 核对业务副作用。四步少一步都不算恢复。  
**技术映射**：系统可用性以业务闭环定义，而不是以页面状态定义。

**小胖**：那排障文档要写到多细？  
**小白**：细到新人按文档能独立定位。  
**大师**：每个故障都沉淀“症状关键词、定位命令、修复动作、回归验证”。这样才能从个体经验升级为团队能力。  
**技术映射**：排障文档不是备忘录，而是可执行剧本。

## 3. 项目实战

### 3.1 四层排障法（可直接执行）

1. **进程层**：确认 Admin/Executor 进程已启动且监听预期端口；  
2. **依赖层**：确认 MySQL 可连接、表结构完整、账号权限正常；  
3. **配置层**：核对 `xxl.job.admin.addresses`、`accessToken`、`executor.appname`、`executor.address`；  
4. **链路层**：创建最小任务并手动触发，验证触发与执行日志都落地。

### 3.2 源码路径映射表

| 问题环节 | 核心源码路径 | 检查重点 |
| --- | --- | --- |
| Admin 启动入口 | `xxl-job-admin/src/main/java/com/xxl/job/admin/XxlJobAdminApplication.java` | 应用是否正常完成 Spring 启动 |
| Executor 启动入口 | `xxl-job-core/src/main/java/com/xxl/job/core/executor/XxlJobExecutor.java` | 执行器初始化、注册、销毁流程 |
| 执行器注册心跳 | `xxl-job-admin/src/main/java/com/xxl/job/admin/core/thread/JobRegistryHelper.java` | 心跳入库与离线摘除逻辑 |
| 登录与首页访问 | `xxl-job-admin/src/main/java/com/xxl/job/admin/controller/base/LoginController.java` | 管理台可访问性与基础鉴权 |
| 调度触发入口 | `xxl-job-admin/src/main/java/com/xxl/job/admin/core/trigger/XxlJobTrigger.java` | 触发请求是否真正下发到执行器 |

### 3.3 验证命令块（最小可验证）

```bash
# 启动 admin（命令成功返回且日志无致命异常）
mvn -pl xxl-job-admin spring-boot:run
```

```bash
# 启动 executor 样例
mvn -pl xxl-job-executor-samples/xxl-job-executor-sample-springboot spring-boot:run
```

```bash
# 验证调度库可访问（按你的账号调整）
mysql -h127.0.0.1 -uroot -p -e "show tables like 'xxl_job_%';" xxl_job
```

```bash
# 验证执行器心跳是否入库
mysql -h127.0.0.1 -uroot -p -e "select registry_key,registry_value,update_time from xxl_job_registry order by id desc limit 5;" xxl_job
```

### 3.4 故障复盘表格

| 故障现象 | 快速判定 | 根因 | 修复动作 | 回归验证 |
| --- | --- | --- | --- | --- |
| Admin 页面打不开 | 端口无监听 | 启动参数错误或端口冲突 | 修正端口并重启 | 浏览器可访问登录页 |
| Executor 一直离线 | 注册表无心跳 | admin 地址错误或网络不通 | 修正 `admin.addresses`/网络策略 | `xxl_job_registry` 有持续更新 |
| 手动触发失败 401 | 两端 token 不一致 | 配置漂移 | 对齐 `accessToken` 并重启 | 同任务再次触发成功 |
| 页面可用但任务不执行 | 有触发日志无执行日志 | handler 名称错误或执行器组错误 | 修正任务配置并重新触发 | 执行日志出现 handler 输出 |

## 4. 项目总结

启动排障真正要解决的是“不确定性”。只要你把问题固定到四层模型，并为每一层设计最小验证动作，排障就会从“凭经验猜”变成“按证据查”。本章的价值不在于记住多少参数，而在于建立方法：先看系统是否活着，再看依赖是否可达，再看配置是否一致，最后看业务闭环是否成立。

建议把本章输出直接纳入团队值班手册：每次线上故障都强制回填“症状、命令、根因、修复、验证”，每月复盘一次高频问题。经过几个迭代后，启动故障会明显下降，且新人能快速接管排障工作。下一章将把“能启动”进一步升级为“能证明任务结果正确”，进入任务正确性测试体系。

补充一个实战建议：在团队内部建立“启动排障决策树”。例如，第一问是“Admin 是否可访问”，第二问是“执行器是否在线”，第三问是“手动触发是否有执行日志”，每个节点都附带命令和预期输出。这样做的好处是，排障从“开放式问题”变成“封闭式判断”，值班同学不需要具备全部源码知识，也能快速走到可执行结论。同时，决策树要与版本绑定，版本升级后立即更新，否则文档会比故障更快过期。

再强调一条经常被忽略的经验：排障过程必须保留时间线。记录“何时发现、何时定位、何时修复、何时验证”，可以让复盘不仅回答“哪里错了”，还能回答“为什么耗时这么长”。一旦你把时间线与命令证据结合，就能量化团队改进效果，比如平均恢复时长（MTTR）是否下降、重复故障是否减少。这是把运维经验产品化的关键一步。

## 本章一页纸速记

**3条结论**
- 启动排障必须按“进程层-依赖层-配置层-链路层”分层推进，禁止跳步猜测。
- `Admin` 可访问不等于任务可用，必须以 `Executor` 在线且可回调为准。
- 排障完成标准是业务闭环成功，而不是页面恢复可登录。

**5条落地清单**
- 统一术语：统一使用 `Admin`、`Executor`、`accessToken`、`appname`、`admin.addresses`。
- 统一参数：地址参数统一写 `http://127.0.0.1:8080/xxl-job-admin`，避免主机名漂移。
- 统一命令：固定执行“启动 Admin -> 启动 Executor -> 查注册表 -> 手动触发”四步。
- 统一注释：命令块以“步骤编号 + 预期结果”注释，如“日志无致命异常”。
- 统一复盘：故障记录必须包含症状关键词、定位命令、修复动作、回归证据。

**3条误区**
- 误区一：反复重启服务代替定位，导致问题暂时隐藏后反复出现。
- 误区二：只看本机启动成功，不验证 Admin 回调链路是否可达。
- 误区三：排障后不沉淀决策树，团队恢复能力无法复用。
