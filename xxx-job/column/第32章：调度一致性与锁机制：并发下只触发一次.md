# 第32章：调度一致性与锁——在并发与故障下守住“只该执行一次”

> 统一约定：术语统一写作 `XXL-JOB`；命令基线统一为 `Admin 8080`、`Executor 9999`、`MySQL Schema xxl_job`、`AppName xxl-job-executor-sample`。

## 1. 项目背景

调度一致性并不是“高级优化”，而是分布式任务平台的生存底线。只要系统进入多节点、多线程、多重试的运行状态，就必然面对三类冲突：并发冲突（同一任务被多个节点同时触发）、时序冲突（重试与补偿重叠执行）、语义冲突（平台认为可重试、业务认为不可重入）。这些冲突在低流量阶段可能只是少量脏数据，在高峰期会演化为批量资金异常、库存错账、对账失败和合规风险。  

团队常犯的错误是把一致性等同于“锁”。锁确实重要，但只能解决“同时做”的问题，不能解决“重复做”“错序做”“回放做”的问题。要构建可演进的一致性架构，至少要有四层设计：  
1) 调度入口互斥（避免同窗口双触发）；  
2) 业务副作用幂等（避免多次落地）；  
3) 失败补偿可重入（避免补偿二次污染）；  
4) 审计可追踪（可还原谁在何时以何参数执行了什么）。  

本章以“高并发订单超时关闭”类任务为案例，给出从锁模型、幂等模型到回放模型的一体化方案，并强调高级篇必须具备的演进能力：从单库锁平滑演进到跨机房一致性控制、从静态阈值演进到数据驱动的动态策略。

## 2. 项目设计（剧本式交锋对话）

### 第一轮：为什么“偶发重复”最难治理

**小胖**：偶尔重复一次影响不大吧，后面修数据就行。  
**小白**：一旦涉及资金和库存，修复成本很高，还可能无法完全回滚。  
**大师**：对，调度一致性要前置治理。事后修复不是能力，只是止损。  
**技术映射**：一致性问题越晚发现，代价呈指数上升。

### 第二轮：锁到底锁什么

**小胖**：那我给整个任务加全局锁，最稳。  
**小白**：全局锁会拖垮吞吐，长任务还可能阻塞后续窗口。  
**大师**：锁要分层：  
1) 调度层锁：防止同一触发窗口被多个调度线程重复分发；  
2) 业务层锁/幂等键：防止同一业务实体重复处理；  
3) 补偿层去重：防止人工补跑与自动重试冲突。  
**技术映射**：锁保护的是“关键临界区”，不是整个系统。

### 第三轮：锁失效怎么办

**小胖**：拿到锁就万事大吉。  
**小白**：如果节点宕机，锁可能悬挂；如果时钟漂移，可能提前过期。  
**大师**：要准备失效策略：锁超时、续租、幂等兜底、死锁清理。并以“最终可恢复”为设计底线。  
**技术映射**：分布式锁只能降低冲突概率，不能替代业务幂等。

### 第四轮：一致性和性能如何平衡

**小胖**：那我们就全部串行，绝对不冲突。  
**小白**：串行会让任务窗口超时，业务照样出问题。  
**大师**：平衡方法是“关键路径强一致、非关键路径最终一致”。对核心写操作加严格幂等，对统计类任务允许可回放。  
**技术映射**：一致性是业务分级策略，不是单一技术开关。

### 第五轮：如何设计可演进的一致性架构

**小胖**：我们先把当前问题修掉，跨机房以后再说。  
**小白**：但现在不留扩展位，后面会整体推翻。  
**大师**：一致性架构要按阶段设计：  
- 阶段A：单库锁 + 幂等键，先守住核心副作用；  
- 阶段B：引入租约与续约，解决长任务误抢锁；  
- 阶段C：多活场景下按业务域分片一致性，减少全局锁冲突；  
- 阶段D：通过事件日志构建可回放能力，实现“故障可恢复”。  
**技术映射**：可演进架构不是一开始做最复杂，而是每一步都能向下一步兼容。

## 3. 项目实战

### 3.1 环境准备

- 至少 2 个 Admin 节点 + 2 个 Executor 节点；  
- MySQL 开启慢日志，便于定位锁竞争；  
- 准备业务幂等表、任务执行审计表；  
- 压测环境可模拟并发触发、节点重启、网络抖动。

### 3.2 步骤一：调度层互斥控制

**步骤目标**：避免同一窗口被重复调度。

核心原则：锁只覆盖“判定并触发”临界区，不覆盖整个执行生命周期，避免长事务压垮吞吐。

### 3.3 步骤二：业务幂等键兜底

**运行结果（文字）**：同一调度窗口只有一个节点成功抢到锁。  
**常见坑**：锁过期时间固定，长任务期间锁被误抢。  
**解决**：引入续租机制并监控锁持有时长。

**步骤目标**：即使重复触发也不产生重复副作用。

幂等键建议构成：`业务实体ID + 时间窗口 + 动作类型 + 版本号`。这样在补偿、重试、人工回放同时存在时，仍能准确区分“一次业务动作”。

### 3.4 步骤三：回放与补偿统一去重

**运行结果（文字）**：重复执行同一 `bizKey` 仅首次生效。  
**常见坑**：幂等键粒度过粗，误伤正常处理。  
**解决**：按“业务实体 + 时间窗 + 动作类型”定义键。

**步骤目标**：自动重试与人工补跑不冲突。

实现建议：
- 回放请求必须带 `replayNo`，避免多次手工点击产生重复副作用；
- 主流程和回放流程共用幂等组件，避免“主流程有保护、回放无保护”；
- 对每次补偿落审计日志，支持按 `bizKey` 追溯完整生命周期。

**运行结果（文字）**：重复点击补跑或并发补跑不会产生多次副作用。  
**常见坑**：只对主流程做幂等，补偿链路缺失保护。  
**解决**：主流程与补偿流程共用幂等组件与审计日志。

### 3.5 验证命令块与验收

```bash
# 统一基线：Admin=8080，Executor=9999，DB=xxl_job，AppName=xxl-job-executor-sample
# 步骤1：确认执行器样例存活
curl "http://localhost:9999/actuator/health"

# 步骤2：触发并发任务，观察锁冲突与幂等命中
for /l %i in (1,1,50) do curl -X POST "http://localhost:8080/xxl-job-admin/api/trigger?id=1001"

# 步骤3：检查锁表状态与更新时间
mysql -uroot -p -D xxl_job -e "select * from xxl_job_lock;"

# 步骤4：检查业务幂等表命中
mysql -uroot -p -D xxl_job -e "select biz_key,count(*) from t_job_idempotent group by biz_key having count(*)>1;"

# 步骤5：检查执行器分组与回放审计记录
mysql -uroot -p -D xxl_job -e "select app_name,update_time from xxl_job_group where app_name='xxl-job-executor-sample';"
mysql -uroot -p -D xxl_job -e "select replay_no,status,update_time from t_job_replay_audit order by update_time desc limit 20;"
```

验收标准：
- 并发触发下，同一业务键只落地一次副作用；  
- 回放与自动重试并行时，不出现重复扣减或重复通知；  
- 锁竞争升高时，系统吞吐下降可控且无大面积超时。

## 4. 项目总结

调度一致性不是一个“加锁技巧”，而是一套业务与平台协同策略。锁解决的是并发入口竞争，幂等解决的是副作用重复，补偿解决的是失败后的可恢复性。三者缺一不可。只有把一致性目标前置到任务设计阶段，系统才能在并发和故障下保持可预测行为。

#### 故障复盘表格

| 故障编号 | 现象 | 根因 | 影响 | 修复动作 | 长期改进 |
| --- | --- | --- | --- | --- | --- |
| CONS-01 | 同一订单被重复关闭 | 调度抢占冲突 + 幂等键缺失 | 用户状态异常 | 回滚并修复订单状态 | 统一幂等 SDK 与审计 |
| CONS-02 | 高峰时调度窗口明显漂移 | 锁粒度过大导致串行化 | 任务超时率升高 | 缩小临界区并拆分锁域 | 建立锁竞争监控 |
| CONS-03 | 人工补偿后再次自动重试 | 回放链路未去重 | 重复通知与资金风险 | 临时停自动重试并清洗 | 主/补偿共用幂等组件 |
| CONS-04 | 节点宕机后长时间无调度 | 锁租约过长且无续租策略 | SLA 失约 | 强制释放锁并人工接管 | 租约 + 续租 + 过期巡检 |

### 优点与代价

- 优点：重复执行风险可控、故障恢复路径明确、审计更清晰  
- 代价：实现复杂度提高，需要维护锁与幂等元数据

### 适用与不适用

- 适用：资金、库存、营销发放等高副作用任务场景  
- 不适用：纯统计、可随时重算且无副作用的轻量任务

- [ ] 锁策略是否经过故障演练并具备可观测性  
- [ ] 幂等键是否覆盖主流程、重试、补偿三路径

### 架构延展：一致性体系分阶段升级

建议按“幂等兜底 -> 锁租约续租 -> 补偿平台化 -> 多活域切分”四步演进，并将一致性设计卡纳入发布评审，确保目标、幂等键、重试和补偿策略可审计、可复核。

## 本章一页纸速记

### 3 个结论
- 一致性治理是“锁 + 幂等 + 补偿 + 审计”的组合能力，不是单点技术。
- 锁只能减少并发冲突概率，不能替代业务副作用幂等。
- 可恢复能力比“绝不失败”更现实，也更关键。

### 3 个误区
- 误区1：加了分布式锁就等于严格一次。
- 误区2：补偿链路可以不做幂等。
- 误区3：一致性和性能只能二选一。
