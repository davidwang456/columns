# 第38章：综合实战 III —— 企业级调度平台转型收官

> 统一约定：术语统一写作 `XXL-JOB`；命令基线统一为 `Admin 8080`、`Executor 9999`、`MySQL Schema xxl_job`、`AppName xxl-job-executor-sample`。

## 1. 项目背景

某中大型企业过去三年建设了十余条业务线，定时任务由各团队分散维护（`crontab`、`@Scheduled`、数据库轮询并存）。当任务总量突破 1.5 万后，问题集中暴露：状态不可见、告警滞后、链路追踪缺失、灰度发布困难、批量变更高风险。

在高峰演练中，团队确认三大矛盾：系统能跑但稳定性边界不清、部门有分工但缺少统一治理语言、存量任务多但迁移风险高。

## 2. 项目设计（剧本式交锋对话）

**小胖**：我们现在已经把 XXL-JOB 跑起来了，页面也能看任务也能触发，为啥还要搞“转型收官”？

**小白**：能跑不等于能稳。现在每个部门用法不一样，SLA 出问题时也互相甩锅。到底谁负责最终交付质量？

**大师**：你们说到了核心。平台化不是“统一工具”，而是“统一责任模型”。我们要把“任务生命周期”拆成可观测、可追责、可恢复三条主线：  
1) 可观测：任务触发、执行、重试、告警全链路标准化；  
2) 可追责：按域划分 Owner、审批人、值班人；  
3) 可恢复：故障可定位、可演练、可降级。  

---

**小胖**：那多部门怎么分工才不打架？

**小白**：还得考虑测试介入时机。

**大师**：落地要按“部门责任矩阵 + 工作流门禁”来设计。建议采用 RACI 模型简化执行：  
- 开发（Responsible）：任务实现、幂等、重试语义、回滚脚本。  
- 测试（Accountable for quality gate）：契约测试、失败注入、回放验证。  
- 运维/SRE（Consulted + Gatekeeper）：容量评估、告警规则、演练编排。  
- 业务方（Informed + Sign-off）：业务窗口期确认、影响范围验收。  

再配三条门禁：  
1) **准入门禁**：新任务必须填写“超时阈值、重试策略、补偿策略、Owner”；  
2) **发布门禁**：灰度任务必须过回放用例和压测阈值；  
3) **运行门禁**：连续失败达到阈值自动降级并通知值班群。  

---

**小胖**：迁移最怕“一刀切”。老系统几千个任务，怎么搬？总不能停服一周吧？

**小白**：而且任务有强依赖关系，A 失败会导致 B、C 连锁失败。迁移期间如果双跑不一致，账务就会出问题。

**大师**：迁移采用“四阶段路线图”，每阶段可回滚：  
- **阶段1：资产盘点**：按任务类型（实时、准实时、离线）与风险等级（高/中/低）分层建档。  
- **阶段2：双轨运行**：核心任务影子流量验证，比较老平台与新平台执行结果。  
- **阶段3：分批切流**：先低风险、后高风险；先无状态、后有状态；每批次设置回退开关。  
- **阶段4：治理收口**：下线旧入口、统一告警、统一审计与看板。  

每批迁移必须满足“三一致”验收：触发一致、结果一致、补偿一致。  

## 3. 项目实战

### 1）迁移路线图（90天收官版）

- **第1-2周：基线建立**  
  - 输出任务资产清单（任务数、依赖、峰值、失败率、平均耗时）。  
  - 设立迁移指挥群（开发、测试、运维、业务代表）。

- **第3-6周：双轨验证**  
  - 低风险任务先双跑：只比对结果不对外生效。  
  - 每周固定复盘：失败任务 TopN、重试放大效应、告警误报率。

- **第7-10周：高风险切换**  
  - 账务、履约、库存类任务采用“窗口期切换 + 人工观察 + 自动回退”。  
  - 引入保底策略：关键任务失败后自动触发补偿任务链。

- **第11-13周：治理收口**  
  - 停用旧调度入口，统一变更审批模板。  
  - 发布平台操作手册与 oncall runbook。  
  - 完成“周报自动化”：SLO达成、故障次数、恢复时长、演练达成率。

### 2）故障演练（3个典型场景）

**故障A：注册中心抖动导致执行器离线误判**  
- 现象：任务短时大量失败，控制台显示执行器离线。  
- 根因：网络抖动叠加心跳阈值过紧，误触发离线判定。  
- 演练动作：注入 1-3 秒网络抖动，观察心跳恢复与重试行为。  
- 修复策略：放宽心跳阈值 + 增加连续判定次数 + 区分“真离线/假离线”告警等级。

**故障B：任务重试引发下游雪崩**  
- 现象：失败任务自动重试后，下游数据库连接池被打满。  
- 根因：重试策略未设置指数退避，瞬时放大请求流量。  
- 演练动作：模拟下游 50% 错误率，验证重试风暴是否触发。  
- 修复策略：指数退避 + 重试上限 + 重试隔离队列 + 下游熔断。

### 3）SLO治理建议（从“报错率”到“业务可用性”）

建议企业建立三级 SLO：

- **平台层 SLO（调度系统本身）**  
  - 指标：任务准时触发率 >= 99.95%，调度 API 可用性 >= 99.9%。  
  - 预算：月度错误预算 0.05%，超预算即冻结高风险变更。

- **任务层 SLO（单任务质量）**  
  - 指标：成功率、P95 执行时长、重试后恢复率。  
  - 机制：按任务等级（P0/P1/P2）设不同阈值，P0 必须有补偿链。

- **业务层 SLO（最终业务结果）**  
  - 指标：如“订单超时关闭延迟 < 5 分钟占比 99.9%”。  
  - 治理：SLO 必须绑定业务 Owner，避免只有技术指标没有业务含义。

配套治理动作：  
1) 每周 SLO Review，关注“预算消耗速度”而非单点故障；  
2) 每月故障演练日，覆盖“网络、容量、时钟”三大类；  
3) 每季度规则清理，淘汰低价值任务与冗余告警。

## 4. 项目总结

### 1）优缺点清单

**优点**  
- 调度入口统一，任务状态可视化，跨团队协同成本显著下降。  
- 支持分层治理（平台/任务/业务），便于建立标准化 SLO 体系。  
- 迁移路线可灰度可回滚，降低存量系统改造风险。  
- 故障演练机制可复用，提升团队应急响应与恢复能力。

**缺点**  
- 初期治理成本高，需要投入平台工程与流程改造资源。  
- 老任务历史包袱重，语义不清导致迁移分析耗时。  
- 指标体系若设计不当，可能出现“追指标不追业务价值”。

### 2）适用场景与不适用场景

**适用场景**  
- 多业务线共用调度平台，任务量大且跨部门协作频繁。  
- 对稳定性、审计与可观测性有明确要求的企业系统。  
- 存在批处理与实时任务混合，需要统一治理模型的团队。

**不适用场景**  
- 仅有少量轻量级定时任务、团队规模很小的项目。  
- 业务变化极快且缺乏最基本流程纪律，短期难以执行治理门禁。

### 3）注意事项

- 不要只迁代码不迁责任：Owner、值班、审批要同步落地。  
- 不要只看成功率：要同时看时效性与业务结果一致性。  
- 不要忽视时钟与时区：跨机房任务必须统一时间基线。

## 本章一页纸速记

### 3 个结论
- 企业转型成败关键在“统一责任模型”，不只在“统一工具”。
- 存量迁移必须分阶段、可回滚，并坚持“三一致”验收标准。
- 平台层、任务层、业务层 SLO 联动后，治理才能长期稳定运行。

### 5 项清单
- [ ] 是否统一使用 `XXL-JOB` 术语与参数基线（8080/9999/xxl_job/xxl-job-executor-sample）
- [ ] 是否完成任务资产分层与风险评分建档
- [ ] 是否按批次执行双轨验证与“三一致”验收
- [ ] 是否将错误预算与发布门禁接入日常变更流程
- [ ] 是否完成收官后的统一告警、审计与运行手册发布

### 3 个误区
- 误区1：先把任务迁完，再补治理机制也来得及。
- 误区2：迁移只要成功率高，就说明业务结果一致。
- 误区3：流程门禁会拖慢业务，因此应尽量弱化。
