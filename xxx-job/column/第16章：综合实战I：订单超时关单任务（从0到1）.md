# 第16章：综合实战 I——订单超时关单任务（从 0 到 1）

## 1. 项目背景

在电商业务里，“下单未支付超时自动关单”是最典型的定时任务场景之一。它看似简单：超过 30 分钟未支付就把订单状态改成“已关闭”，并回收库存、释放优惠、记录审计日志。但真正落地到线上时，问题并不简单。订单量大时，任务需要分钟级稳定触发；系统偶发抖动时，任务不能漏关单；执行失败时，必须有重试和告警；重复执行时，必须保证幂等；出了故障，还要能快速排查“哪一批单没处理、为什么没处理、是否补偿成功”。

如果没有统一调度平台，团队通常会陷入三个困境：开发侧逻辑分散、运维侧缺乏集中可观测、测试侧难以复现复杂故障。xxl-job 在这个业务里不是“锦上添花”，而是基础设施，它把超时关单从脆弱脚本升级为可治理能力。

本章将产出一个可交付的最小实践：跑得通、跑得稳、可验证。

## 2. 项目设计（剧本式交锋对话）

### 第一轮：需求拆解与边界定义

**小胖**：业务说“30 分钟未支付自动关单”，我直接写个 SQL 定时跑不就完了？  
**小白**：先别急，这个 SQL 跑在哪？失败了怎么办？重复执行会不会把已支付订单关掉？  
**大师**：先拆边界。我们不做“粗暴更新”，做“安全关单任务”：  
1) 只处理 `待支付` 且创建时间超过阈值的订单；  
2) 关单前二次校验支付状态；  
3) 每次处理可追踪（批次号、数量、成功/失败明细）；  
4) 失败可重试、异常可告警。

**技术映射**：  
业务需求不是“定时 SQL”，而是“任务语义 + 状态机 + 可治理链路”。

### 第二轮：任务模型选择

**小胖**：那我们用 BEAN 还是 GLUE？  
**小白**：如果后续逻辑会复杂，应该用代码仓库管理。  
**大师**：这类核心业务任务首选 **BEAN 模式**，原因：  
- 版本可追踪（随应用发布）；  
- 便于单元测试与代码评审；  
- 可复用业务服务与事务能力。  
GLUE 更适合临时修复、轻量脚本，不适合长期核心链路。

**技术映射**：  
长期核心任务选 BEAN，临时运维任务可考虑 GLUE。

### 第三轮：并发与幂等如何兜底

**小胖**：我担心任务重试会不会重复关单。  
**小白**：如果两台执行器同时处理同一批订单，冲突怎么避免？  
**大师**：我们用“三层幂等”：
- SQL 条件幂等：只更新 `status=WAIT_PAY` 的订单；
- 业务幂等：按订单状态机检查，只允许 `WAIT_PAY -> CLOSED`；
- 任务幂等：每次执行生成批次号，处理记录可追溯可重放。  
并发层面使用“串行阻塞策略 + 单任务单 handler”，先保证正确性，再做吞吐优化。

**技术映射**：  
重试不可怕，不幂等才可怕。

### 第四轮：失败怎么处理才可运营

**小胖**：报错了我在控制台看日志就行吧？  
**小白**：线上凌晨报错谁看控制台？  
**大师**：必须把“开发可读日志”升级成“运维可运营日志”：
- 每批次输出处理总数、成功数、失败数；
- 失败样本打印订单号和异常摘要；
- 设置失败重试与超时阈值；
- 配置告警，保证值班可感知。

**技术映射**：  
能打印日志不等于可观测，结构化上下文才是可运营。

### 第五轮：测试覆盖风险

**小胖**：这任务是不是测通正常路径就行？  
**小白**：至少要补并发重复触发和执行器短时不可用。  
**大师**：对，任务测试不能只看“功能对”，还要确认“故障下不失控”。

## 3. 项目实战

### 3.1 环境准备

**依赖与版本**
- JDK 17+
- Maven 3.8+
- MySQL 8.x
- xxl-job admin + xxl-job-executor-sample-springboot

**最小配置**
1. 执行 `doc/db/tables_xxl_job.sql` 初始化调度库；  
2. 配置 Admin 数据源与 `xxl.job.accessToken`；  
3. 配置 Executor 的 `xxl.job.admin.addresses`、`appname`、`accessToken`；  
4. 启动 Admin 与 Executor，确认执行器在线。

### 3.2 领域建模（最小订单表）

建议本地准备简化订单表（示例）：

```sql
create table if not exists t_order (
  id bigint primary key auto_increment,
  order_no varchar(64) not null unique,
  status varchar(32) not null,
  amount decimal(10,2) not null default 0,
  create_time datetime not null,
  pay_time datetime null,
  close_time datetime null,
  update_time datetime not null
);
```

状态约定：
- `WAIT_PAY`：待支付
- `PAID`：已支付
- `CLOSED`：已关闭

### 3.3 代码实现（关键片段说明）

#### 步骤一：定义任务处理器

**步骤目标**：创建 `closeTimeoutOrderJobHandler`，接收 `timeoutMinutes`、`batchSize` 参数，生成批次号并输出统一日志。

#### 步骤二：实现业务服务（幂等 + 批处理）

**步骤目标**：分页查找待支付超时订单，逐条执行状态迁移并统计成功失败数量；失败时抛出异常触发平台重试。

#### 步骤三：SQL 条件更新

**步骤目标**：用 `where status='WAIT_PAY'` 保证状态机安全，避免已支付订单被误关。

#### 步骤四：管理台任务配置

**步骤目标**：将代码能力转换为可运营任务。

推荐配置：
- 执行器：`xxl-job-executor-sample`
- JobHandler：`closeTimeoutOrderJobHandler`
- 调度类型：Cron（如每分钟）
- 阻塞策略：串行
- 超时：60 秒（按实际调整）
- 失败重试：2~3 次
- 任务参数：`timeoutMinutes=30,batchSize=200`

### 3.4 运行结果（文字描述）

成功路径下应看到：
- 调度日志显示触发成功；
- 执行日志打印批次号、处理总数、成功数、失败数；
- 数据库中符合超时条件的 `WAIT_PAY` 订单变为 `CLOSED`；
- 已支付订单保持 `PAID` 不变。

失败路径下应看到：
- 某些订单处理异常触发任务失败；
- 平台按配置自动重试；
- 多次失败后触发告警通知。

### 3.5 常见坑与解决方法

1. **坑：把支付状态校验放在任务外层，导致并发竞态。**  
   解决：更新语句带状态条件，确保 SQL 层幂等。  

2. **坑：一次处理量过大，任务超时。**  
   解决：批处理分页 + 限流，必要时拆任务并使用分片。  

3. **坑：日志只打印“成功/失败”，缺少上下文。**  
   解决：统一带 `batchNo/orderNo/timeout/batchSize` 关键字段。  

4. **坑：重试后重复发送业务通知。**  
   解决：通知逻辑单独幂等，使用事件去重键。  

### 3.6 测试验证清单

#### 单元测试建议
- `closeIfWaitPay` 在 `WAIT_PAY` 时更新成功；
- `closeIfWaitPay` 在 `PAID/CLOSED` 时更新 0 行；
- 参数解析容错（缺省参数、非法参数）。

#### 集成验证命令（示意）
- 手动触发任务 1 次，检查日志与库数据；
- 人工制造异常触发重试；
- 将一个执行器实例下线，验证任务仍可执行。

#### 回归场景
- 高峰时段 1 分钟触发一次连续 30 分钟；
- 支付与关单并发冲突测试；
- 数据库短时抖动后的补偿与恢复。

### 3.9 故障复盘表格

| 故障现象 | 根因 | 应急动作 | 根治方案 |
| --- | --- | --- | --- |
| 已支付订单被误关 | SQL 未限制状态条件 | 立即回滚受影响订单状态 | 统一 `where status='WAIT_PAY'` 并加回归测试 |
| 活动高峰任务超时 | 单批次量过大+索引缺失 | 降低批次并临时扩容 | 分页批处理+补索引+拆分任务 |
| 重试后重复通知 | 通知逻辑无幂等键 | 临时关闭重复通知 | 引入业务去重键与幂等表 |

## 4. 项目总结

### 优点（本方案）
1. 业务语义清晰：关单条件、状态流转、异常处理可复述；  
2. 工程可维护：BEAN 模式便于版本管理和代码审计；  
3. 平台可治理：重试、超时、告警、日志形成闭环；  
4. 跨角色协同顺畅：开发/测试/运维围绕同一任务配置协作。  

### 缺点与权衡
1. 初始建设成本较“单 SQL 脚本”更高；  
2. 需要团队具备幂等与状态机设计能力；  
3. 运维体系未建立前，告警与日志仍可能噪声较大。  

### 适用场景
- 订单超时关单、库存回补、优惠券过期处理；
- 账务日终补偿、状态对账；
- 需要任务审计、重试与告警的后台批处理。

### 不适用场景
- 毫秒级实时处理链路；
- 极轻量单实例应用且几乎无运维诉求。

### 注意事项
- 任务参数必须可审计可回放；
- 任务逻辑必须天然幂等；
- 告警要分级，避免“狼来了”效应；
- 定期回顾失败日志并沉淀故障手册。

## 本章一页纸速记

**3条结论**
- 订单超时关单不是定时 SQL，而是“状态机 + 幂等 + 可恢复”的任务系统工程。
- `WAIT_PAY -> CLOSED` 必须通过 SQL 条件与业务校验双重保障，防止误关已支付订单。
- 任务稳定性要以“触发、执行、回写、告警”闭环证据判定，而非单点日志成功。

**3条落地清单**
- 统一术语：统一使用 `Admin`、`Executor`、`JobHandler`、`batchNo`、`bizDate`。
- 统一参数：任务参数固定为 `timeoutMinutes=30,batchSize=200` 这种 `k=v` 形式。
- 统一命令：固定“启动 Admin、启动 Executor、查 `xxl_job_log`、查 `t_order` 状态分布”。

**3条误区**
- 误区一：只靠应用层判断幂等，不在 SQL 层做状态条件限制。
- 误区二：把批次放大当作性能优化，忽略超时与数据库压力。
