# 第14章：任务正确性测试——不只测“跑了”，还要测“跑对了”

## 1. 项目背景

在日常开发里，接口测试通常聚焦“请求-响应”是否符合预期；但调度任务不是一次性动作，它会周期触发、异步执行、带重试、可能并发，还会跨越多个系统写入状态。因此，任务测试如果只看“执行成功”会产生严重错觉：日志里是 success，业务表却可能出现重复更新、漏处理、错误状态迁移。生产事故里最常见的三类问题——重复扣减、超时判断错误、失败重试后脏数据——几乎都能在上线前通过正确的测试模型提前发现。

本章目标是搭建任务正确性测试的四维框架：功能正确（结果对不对）、幂等正确（重复执行是否安全）、时序正确（时间边界是否准确）、恢复正确（故障后是否可回归）。你将把“任务测试”从手工点按钮升级为有断言、有数据集、有故障注入、有回归基线的工程流程。换句话说，我们不是验证“任务跑过”，而是证明“任务在复杂条件下仍然可信”。

## 2. 项目设计（剧本式交锋对话）

**小胖**：任务日志是成功，我觉得就可以提测了。  
**小白**：日志成功只能证明代码没有显式抛错，不能证明业务状态正确。  
**大师**：任务测试要有“状态断言”，比如订单只能从 `WAIT_PAY` 迁移到 `CLOSED`，不能跳级、不能逆转。  
**技术映射**：正确性 = 状态机约束被满足。

**小胖**：我准备只测一条标准数据。  
**小白**：标准数据最容易过，风险都在边界。  
**大师**：最小用例集合必须包含：正常、空集、边界时间、非法参数、部分失败、并发重复触发。  
**技术映射**：覆盖边界比覆盖路径数量更关键。

**小胖**：并发重复触发怎么判定通过？  
**小白**：看是否发生重复副作用。  
**大师**：用“最终一致断言”：执行 1 次与执行 N 次，最终状态一致；允许日志重复，不允许业务结果重复。  
**技术映射**：幂等是任务系统的生命线。

**小胖**：故障注入是不是太重了？  
**小白**：不上线前不测故障，上线后就会在生产测。  
**大师**：至少要做三类故障注入：数据库短暂不可用、下游接口超时、执行器重启。验证恢复过程而不是只看失败瞬间。  
**技术映射**：可恢复性是正确性的组成部分。

## 3. 项目实战

### 3.1 正确性断言模板

每个任务先写断言再写用例，建议固定四类断言：
1. 结果断言：目标状态是否达成；
2. 幂等断言：重复执行不产生重复副作用；
3. 时序断言：临界时间点行为精确；
4. 恢复断言：故障恢复后系统回到正确状态。

## 4. 项目总结

任务正确性测试的难点，不在于写多少测试代码，而在于是否建立了可证明的断言体系。只测成功路径会得到脆弱信心，覆盖幂等、边界和恢复后，你才拥有可上线信心。本章给出的四维断言、源码映射、最小命令验证和复盘模板，可以直接作为团队测试规范的基础版本。

落地建议是“先小后全”：先选一个高价值任务（如超时关单）做完整四维测试，再把模板复制到其他任务。每次线上事故都要反向补充一个自动化回归用例，让经验不断进入基线。这样做两三个迭代后，你会看到显著效果：线上故障更少，定位更快，发布信心更强。

在执行层面，建议把任务测试分成三个发布门禁：开发自测门禁、测试回归门禁、发布前抽检门禁。开发自测至少覆盖正常+幂等两类；测试回归覆盖边界+故障恢复；发布前抽检覆盖生产参数下的一次最小实跑验证。门禁越前置，线上风险越小。此外，测试报告不要只写“通过/失败”，要附带关键断言结果与样本数据，这样复盘时可直接复用，不需要重新造测试场景。

如果你的任务跨多个服务，建议引入“端到端业务主键”作为统一追踪标识，把调度日志、业务日志、数据库状态串成同一证据链。这样当任务结果异常时，你可以在 5~10 分钟内完成跨服务定位，而不是在多个系统里盲搜关键词。这也是把正确性测试从单任务思维升级为系统思维的分水岭。

最后补充一个容易被忽略的治理点：把“错误样本”沉淀为长期测试资产。很多团队每次都重新造测试数据，结果关键缺陷不断重复。正确做法是维护一份任务异常样本库，包含临界时间数据、历史事故数据、脏数据变体、并发冲突数据，并在每次版本发布前自动回放。这样你不仅能验证当前修改，还能持续守住历史稳定性。对调度系统而言，历史缺陷复发往往比新缺陷更昂贵，因此“样本库+自动回放”应当成为正确性测试的标准配置。

如果你希望更进一步，可以把任务测试结果接入 CI 门禁：当幂等断言或恢复断言失败时，流水线直接阻断发布。这样“测试结论”就不再停留在文档里，而是成为真正的工程约束。长期来看，这种机制会显著降低“带病上线”的概率。

对关键任务，还建议设置“发布后回归再验证”步骤，确保线上参数与测试参数一致时结果仍稳定。

此外，建议在团队周会中固定一个“任务正确性十分钟”环节：只讨论最近一次失败用例与新补充的回归断言，不讨论实现细节。这个机制看似简单，但能持续把注意力从“功能交付”拉回“结果可信”，避免测试工作在迭代高压下被动缩水。长期执行后，团队会形成共同语言：先看断言是否完备，再看实现是否优雅。

## 本章一页纸速记

**3条结论**
- 任务测试必须证明“结果正确、幂等正确、时序正确、恢复正确”，单一 success 日志不构成通过证据。
- 并发与重试场景是任务缺陷高发区，测试优先级必须高于普通功能路径。
- 正确性测试的终点是门禁化：断言失败要能阻断发布，而不是仅写在报告里。

**5条落地清单**
- 统一术语：全员使用“四维断言”与 `WAIT_PAY -> CLOSED` 这类状态机表达。
- 统一参数：测试任务参数统一写成 `timeoutMinutes=30,batchSize=200` 形式。
- 统一命令：验证命令固定为“启动 Admin、启动 Executor、查 `xxl_job_log`、查业务表状态”。
- 统一注释：每条命令注释都写清“验证目标”，避免只写“执行命令”。
- 统一门禁：CI 至少接入幂等断言与恢复断言两类失败阻断。

**3条误区**
- 误区一：只测标准数据，不测边界和异常样本。
- 误区二：把“重复日志”误判为“重复业务副作用”。
- 误区三：线上事故不反补为自动化回归用例。
