# 第33章：路由算法深潜——从“能分发”到“分得稳、分得准”

> 统一约定：术语统一写作 `XXL-JOB`；命令基线统一为 `Admin 8080`、`Executor 9999`、`MySQL Schema xxl_job`、`AppName xxl-job-executor-sample`。

## 1. 项目背景

进入高级阶段后，团队通常会发现一个“隐形瓶颈”：任务本身代码没问题，但同一类任务在多执行器集群中表现不稳定。有的节点经常被打满，有的节点长时间空闲；某些任务总在同一台机器执行，导致局部热点；某台机器短时抖动后，调度系统恢复缓慢，失败率上升。表面看是“执行器偶发不稳定”，本质却是“路由策略与任务特征不匹配”。

在中小规模场景下，轮询或随机策略往往“够用”；但在高并发、异构节点、跨机房和混部场景中，仅靠默认路由很容易触发放大效应：慢节点被持续分配、重试流量叠加、尾延迟抬升、局部故障扩散。最终影响的不只是任务成功率，还会拖垮下游数据库与业务服务。

本章目标是把“路由”从配置项提升为工程能力。你将建立一套路由选型框架：根据任务性质（是否有数据局部性、是否要求稳定命中）选择策略，并结合阻塞策略、超时与重试构建稳定性。章节结束后，你应能回答三个关键问题：
- 为什么同样是多节点，某些任务适合轮询，某些要哈希？
- 路由策略如何与“失败重试”联动，避免雪崩式重放？
- 如何在不改业务代码的情况下，通过策略组合降低 P99 延迟？

## 2. 项目设计（剧本式交锋对话）

### 第一轮：路由到底在解决什么

**小胖**：我理解路由就是“随便挑一台机器跑”，差别真有那么大吗？  
**小白**：如果某台机器 CPU 已经 90%，再把任务丢过去不是更慢？  
**大师**：路由解决的不是“能不能发出去”，而是“在不确定系统里，持续做出较优选择”。路由的核心是三件事：  
1) 分配公平性；  
2) 故障规避能力；  
3) 数据与执行局部性。  

**技术映射**：  
路由不是随机动作，而是调度系统的在线决策器。

### 第二轮：常见策略怎么选

**小胖**：轮询看起来最公平，那是不是万能？  
**小白**：一致性哈希是不是只适合缓存，不适合任务调度？  
**大师**：没有万能策略，只有“匹配任务画像”的策略：  
- **轮询/随机**：适合无状态、短任务，目标是吞吐均衡；  
- **故障转移**：适合高可用优先场景，优先可达节点；  
- **一致性哈希**：适合需要“稳定命中”的任务，如按租户、门店、分片键处理；  
- **最不经常使用/最近最少使用**：适合节点负载差异明显场景。  

**技术映射**：  
路由选型的第一原则：先看任务状态模型，再看集群健康模型。

### 第三轮：路由和重试会“打架”吗

**小胖**：失败重试不是救命吗？为什么线上经常越重试越糟？  
**小白**：是不是因为重试还打到原来的坏节点？  
**大师**：对。若路由与重试策略不协同，重试会成为放大器。改进思路：  
- 首次执行可按局部性路由（例如一致性哈希）；  
- 重试阶段切换故障转移策略，优先健康节点；  
- 设置退避窗口，避免瞬时重压。  

**技术映射**：  
重试不是“再来一次”，而是“带策略的二次调度”。

### 第四轮：如何评估路由效果

**小胖**：我看任务成功率 99% 了，应该没问题吧？  
**小白**：成功率高，但 P99 很差也会拖业务。  
**大师**：评估至少看四组指标：  
- 节点分配偏斜度（是否长期热点）；  
- 任务尾延迟（P95/P99）；  
- 重试放大倍数；  
- 故障恢复时间（MTTR）。  

**技术映射**：  
路由优化目标不是“平均值漂亮”，而是“尾部风险可控”。

## 3. 项目实战

### 3.1 环境准备

- JDK 17+，Maven 3.8+，MySQL 8.x；
- `xxl-job-admin` 与 3 个执行器实例（同 `appname`）；
- 压测脚本用于连续触发任务；
- 观测指标：任务成功率、执行耗时分位数、节点命中分布。

建议准备两类任务样本：
- **样本 A**：无状态短任务（<1s）；
- **样本 B**：按业务键处理（要求同键稳定命中）。

| 触发主链 | `JobTrigger` | `xxl-job-admin/src/main/java/com/xxl/job/admin/core/trigger/JobTrigger.java` | 触发时执行路由决策 | 输出决策原因用于回放 |

### 3.2 步骤一：建立基线（轮询策略）

**步骤目标**：得到默认策略下的分配与延迟基线。  
将两个样本任务都设置为轮询，持续运行 30 分钟，记录：
- 每节点任务数量；
- 平均耗时与 P99；
- 失败与重试次数。

**运行结果（文字描述）**：  
样本 A 通常表现稳定；样本 B 在跨节点命中下可能出现缓存冷启动、连接预热重复，导致尾延迟波动。

**常见坑**：
- 仅看平均耗时，忽略尾延迟；
- 测试窗口过短，未覆盖节点抖动周期。

### 3.3 步骤二：一致性哈希用于稳定命中

**步骤目标**：验证“同键任务”在一致性哈希下的稳定性收益。  
为样本 B 增加业务键参数（如 `tenantId`），路由切换为一致性哈希。

示例参数约定：

```text
tenantId=shop_1024,batchSize=200
```

对比观察：
- 同一 `tenantId` 在多次触发中的节点稳定度；
- 缓存命中率与 P99 变化；
- 节点上下线时重映射比例。

**运行结果（文字描述）**：  
正常情况下，同键稳定命中，P99 降低；少量节点变更时，仅部分键迁移，扰动小于普通哈希。

### 3.4 步骤三：故障注入 + 重试协同

**步骤目标**：验证“首次局部性路由 + 重试故障转移”的抗抖能力。  
操作流程：
1. 首次触发使用一致性哈希；
2. 人为下线一个高命中节点；
3. 失败重试阶段切换为故障转移；
4. 观察恢复时间与重试成功率。

建议配置：
- 超时阈值：30~60 秒；
- 重试次数：2~3；
- 重试间隔：指数退避（如 5s、15s、30s）。

**运行结果（文字描述）**：  
故障窗口内任务失败短时上升，但因重试路由切换，成功率快速回升，未出现长时间热点积压。

### 3.5 步骤四：策略决策表落地

**步骤目标**：把经验固化为可复用规则。  
建议形成团队决策表：
- 无状态短任务：轮询/随机；
- 同键局部性任务：一致性哈希；
- 高可用强依赖任务：故障转移优先；
- 混部且负载不均场景：结合 LRU/LFU 与健康探测。

并在变更流程中要求：
- 新任务上线前必须给出路由理由；
- 发布后 1 周回收指标，必要时复盘调参。

### 3.6 高级演进：从静态路由到自适应路由

在企业级场景中，纯静态策略会逐步失效，建议按三阶段演进：
1. **阶段一：规则驱动**。按任务类型选固定策略（轮询/哈希/故障转移）；  
2. **阶段二：指标驱动**。把节点延迟、失败率、队列深度作为路由评分因子；  
3. **阶段三：反馈驱动**。根据过去 N 分钟执行结果自动调权，形成“在线学习式”自适应分流。  

该演进可在不改任务代码的前提下完成，核心是把“路由策略”从配置项升级为“可观测决策系统”。

## 4. 项目总结

### 优点
- 将“路由”从经验配置升级为数据驱动决策；
- 通过策略协同降低重试放大和尾延迟；
- 可在不侵入业务代码情况下获得显著稳定性收益；
- 形成标准化评估指标，便于跨团队协作。

### 缺点
- 需要持续观测数据支撑，维护成本高于默认配置；
- 一致性哈希对参数规范要求高，传参不稳定会失效；
- 多策略组合后配置复杂度上升，误配风险增加。

### 适用场景
- 多节点执行器集群；
- 对 P99、SLA 敏感的核心任务；
- 存在业务键局部性（租户、门店、区域）任务；
- 节点异构明显、资源波动频繁场景。  
不适用：单机轻量任务、低频离线脚本。

### 故障复盘表格

| 复盘编号 | 故障现象 | 根因分析 | 即时止血 | 长期方案 |
| --- | --- | --- | --- | --- |
| ROUTE-01 | 单节点负载长期 90%+ | 一致性哈希键设计过粗 | 临时切轮询并扩容 | 引入热点键拆分规则 |
| ROUTE-02 | 重试后成功率反降 | 重试仍沿用故障节点路由 | 强制重试走 failover | 路由与重试策略解耦 |
| ROUTE-03 | 机房波动引发全局抖动 | 未做本地优先策略 | 临时限流 | 增加机房亲和路由 |
| ROUTE-04 | P99 长尾持续恶化 | 仅关注平均值，无尾延迟治理 | 关闭高风险批任务 | 建立尾延迟门禁与看板 |

### 架构延展：路由治理平台化

当任务量达到万级后，建议建设路由治理平台，提供策略目录化、决策可解释化与效果可回放，并支持“首次偏局部性、重试偏可用性”的阶段化编排。

## 本章一页纸速记

### 3 个结论
- 路由优化目标是“尾部风险可控”，不只是平均值好看。
- 路由策略必须和重试策略协同，否则重试会放大故障。
- 任务画像决定路由策略，不能迷信单一“万能算法”。

- [ ] 是否统一使用 `XXL-JOB` 术语与参数基线（8080/9999/xxl_job/xxl-job-executor-sample）
- [ ] 是否按任务类型沉淀路由策略决策表
- [ ] 是否对策略变更执行灰度与回滚预案

### 3 个误区
- 误区1：轮询最公平，所以总是最优。
- 误区3：成功率高就说明路由健康。
