# 第 06 章：探针机制：活着、能接流量、刚启动，三件事不是一回事

> 对应学习计划第 1 周探针配置与故障模拟。

很多团队第一次给 Kubernetes 工作负载加探针时，都会犯一个很典型的错误：拿一套配置，试图同时解决“服务是不是活着”“服务能不能接流量”“服务是不是刚刚还在冷启动”这三类完全不同的问题。结果往往是应用不是被探针误杀，就是还没准备好就被流量打爆。要学懂探针，关键不是背字段，而是先弄清楚：**活着、可用、启动完成，本来就不是一回事。**

## 1）项目背景

在没有 Kubernetes 探针之前，很多服务的健康判断其实非常粗糙。最常见的方式是“进程还在就算正常”，更进一步一点的会做一个简单的 HTTP `/healthz` 接口，只要能返回 200，就认定服务没问题。表面上看，这似乎已经够用了，但一旦进入真实生产环境，你很快会发现这里面至少有三类不同语义：

- 进程有没有彻底死掉，是否需要重启。
- 服务是不是已经准备好接收外部流量。
- 服务是不是只是启动慢，还没来得及完成预热。

这三件事混在一起，会产生很多诡异问题。比如：

- Java 应用启动要 40 秒，但 liveness 探针 10 秒就开始检查，结果容器反复重启。
- Web 服务进程已经起来了，但数据库连接池还没建立好，readiness 没配，流量提前打进来导致大量 500。
- 某个依赖偶尔抖动 2 秒，liveness 配得太激进，结果业务本来能自愈，却被平台直接杀容器。

Kubernetes 用三种探针分别处理这三类语义：

- `livenessProbe`：判断容器是否还活着。如果失败，Kubernetes 会重启容器。
- `readinessProbe`：判断容器是否已经准备好接流量。如果失败，容器不会被重启，但会从 Service 后端摘除。
- `startupProbe`：给启动慢的应用留出保护窗口，在启动完成前屏蔽 liveness/readiness 的过早干预。

这三者的价值，不只是“多了几个字段”，而是把原本隐含在应用启动脚本、运维经验和值班手册里的运行语义，正式交给平台管理。换句话说，你不再只是告诉 Kubernetes“这个容器启动了”，而是在告诉它“什么时候应该重启我，什么时候应该先别给我流量，什么时候应该多给我一点启动耐心”。

探针设计得好，发布稳定性、故障恢复和流量质量都会明显提升；设计得不好，它反而会成为制造故障的来源。所以本章真正要建立的，是一种非常重要的工程意识：**健康检查不是“有没有接口”这么简单，而是“平台如何理解你的运行状态”。**

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我看文档里有 liveness、readiness、startup 三种探针，感觉名字都差不多。我能不能只配一个 `/healthz`，三种都复用？

**大师**：能配，但大概率会给自己埋坑。因为这三种探针名字像，职责却完全不同。

**小白**：不都是在问“服务是不是正常”吗？

**大师**：不是。`liveness` 问的是“你是不是已经坏到必须重启”；`readiness` 问的是“你现在能不能安全接流量”；`startup` 问的是“你是不是还在合理地慢启动”。

**小白**：原来平台关心的是三种不同的状态语义。

**大师**：对。举个例子，一个服务刚启动时，进程可能已经活着，但缓存还没预热、数据库连接池还没建好。这个时候它适合被判定为“活着但未就绪”，而不是“死了”。

**小白**：那如果我把 liveness 配得太严格，会怎样？

**大师**：应用本来只是启动慢，或者短暂卡顿，结果被探针当成死亡反复重启。最终不是应用自己坏了，而是探针把它搞坏了。

**小白**：readiness 失败就不会重启，是吗？

**大师**：对。它的作用更像“摘流”，而不是“杀进程”。这也是为什么它特别适合处理依赖抖动、预热未完成、临时降级等情况。

**小白**：那 startupProbe 是给哪些应用准备的？

**大师**：凡是启动时间明显长于正常运行检查周期的应用，都应该认真考虑它。像 JVM、大型框架、模型加载、冷启动预热这种场景，startupProbe 往往能救命。

**小白**：我好像明白了。探针不是机械配置，而是在向平台描述应用的生命周期阶段。

**大师**：这句话非常到位。探针不是“监控接口”，而是“运行语义声明”。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们构造一个最小 Deployment，并同时配置 `livenessProbe`、`readinessProbe` 和 `startupProbe`，观察三者如何影响容器重启与流量接入。

### 第一步：准备一个带探针的 Deployment

新建 `probe-demo.yaml`：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: probe-demo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: probe-demo
  template:
    metadata:
      labels:
        app: probe-demo
    spec:
      containers:
      - name: app
        image: nginx:1.27
        ports:
        - containerPort: 80
        startupProbe:
          httpGet:
            path: /
            port: 80
          failureThreshold: 30
          periodSeconds: 2
        readinessProbe:
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 3
          periodSeconds: 5
        livenessProbe:
          httpGet:
            path: /
            port: 80
          initialDelaySeconds: 10
          periodSeconds: 10
```

这份配置的思路是：

- 启动阶段先由 `startupProbe` 托底，允许应用有一段预热时间。
- 真正接流量之前，由 `readinessProbe` 决定是否就绪。
- 长期运行期间，由 `livenessProbe` 判断容器是否失活到需要重启。

### 第二步：应用资源并观察状态

```bash
kubectl apply -f probe-demo.yaml
kubectl get pods -l app=probe-demo -w
kubectl describe pod -l app=probe-demo
```

第一次练习时，建议重点盯住以下几个点：

- Pod 是否进入 `Running`。
- `Ready` 列什么时候从 `0/1` 变成 `1/1`。
- Events 里探针检查的成功与失败信息。
- Pod 是否发生了重启。

### 第三步：理解 readiness 的“摘流不杀进程”

如果你把 readiness 检查路径故意改成不存在的地址，例如 `/not-ready`，重新 `apply` 后继续观察：

```bash
kubectl get pods -l app=probe-demo
kubectl describe pod -l app=probe-demo
kubectl get endpoints
```

你会发现容器可能仍然在运行，但 Pod 不会进入 Ready 状态，并且不会被加入到 Service 的可用后端中。这一步很能帮助你建立 readiness 的正确心智：**它控制的是“能否接流量”，不是“是否继续活着”。**

### 第四步：理解 liveness 的“失败就重启”

如果把 liveness 检查路径改成必定失败的地址，例如 `/dead-check`，你就可能在 `describe pod` 中看到重启次数增加。这说明 liveness 的判断结果会直接影响容器生命周期。

但请一定注意，这并不意味着“liveness 越严格越好”。在很多生产事故中，真正的问题不是应用挂了，而是探针过于激进，导致平台把本来还能缓过来的实例一轮轮重启。

### 第五步：什么时候必须考虑 startupProbe

如果你的应用存在下面这些情况，就不要只靠 `initialDelaySeconds` 硬扛：

- JVM 启动很慢。
- 容器启动时要加载大量模型或缓存。
- 有明显的数据库迁移或预热流程。
- 首次启动耗时和稳定运行耗时差距很大。

这时 `startupProbe` 的价值就体现出来了。它不是简单增加等待时间，而是把“启动阶段”和“稳定运行阶段”明确区分开来。

### 这一节应该带走什么

- `livenessProbe` 用来决定“是否重启容器”。
- `readinessProbe` 用来决定“是否接入流量”。
- `startupProbe` 用来保护慢启动应用，避免被过早干预。
- 探针不是越严格越好，而是越贴近应用真实生命周期越好。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把应用运行状态正式声明给平台，提升了恢复和流量治理的确定性。
- 探针设计合理时，可以显著降低故障扩散概率。
- 它让“应用启动”“应用可用”“应用存活”这三件事被明确分层处理。

### 缺点

- 参数理解门槛不低，稍有不慎就会把探针变成故障制造者。
- 过度依赖探针而不改善应用本身的启动与健康设计，会掩盖真正的问题。
- 探针路径、超时和周期需要根据应用特性调优，无法照抄通用模板。

### 使用场景

- 需要让平台自动判断容器重启时机的业务。
- 需要在预热完成前防止流量接入的服务。
- 启动较慢、生命周期阶段明显的应用。

### 注意事项

- 不要把三个探针混成一个语义。
- 优先让应用提供真实反映状态的健康接口，而不是永远返回 200 的假接口。
- 先从温和参数开始，基于实际启动时长和运行曲线逐步调整。

### 常见踩坑经验

- 最常见的坑，是用同一套探针参数去解决所有应用的启动和运行问题。
- 第二个坑，是 readiness 没配或配得过于宽松，结果应用还没准备好就被接流量。
- 第三个坑，是把 liveness 配得过于激进，导致容器因为短暂抖动被频繁重启。

这一章真正想让你带走的，不是几个 YAML 字段，而是一条运行期设计原则：**平台对应用的判断，必须尊重应用真实的生命周期阶段。**
