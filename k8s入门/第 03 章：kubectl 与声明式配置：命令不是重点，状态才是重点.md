# 第 03 章：kubectl 与声明式配置：命令不是重点，状态才是重点

> 对应学习计划第 1 周对 kubectl 命令与 YAML 工作流的要求。

很多人学习 Kubernetes 的第一反应，是先记一份 `kubectl` 命令清单：`get`、`describe`、`apply`、`logs`、`exec`、`delete`。这些命令当然重要，但如果只停留在“会敲命令”这一层，你会很快遇到瓶颈。因为 `kubectl` 的真正价值，不只是命令入口，而是你和 Kubernetes API 交互时的**思维接口**。这一章的重点，就是把“会用命令”升级为“理解声明式交付”。

## 1）项目背景

在传统运维思路中，我们习惯通过脚本或手动命令改变系统状态：启动一个进程、复制一个配置文件、重启一个服务、查看日志、回滚一个版本。这种方式的问题不在于它不能工作，而在于它很难成为可复用、可审计、可协作的团队流程。

Kubernetes 提供了一套完全不同的范式。你不只是“执行命令”，而是在修改集群中对象的期望状态。而 `kubectl` 正是你最常用的客户端工具，它负责把你的意图翻译成对 API Server 的请求。

这意味着两件重要的事：

- `kubectl` 不是 Kubernetes 本身，它只是客户端入口。
- 真正决定系统行为的，不是你敲过什么命令，而是 API Server 里最终保存了什么对象状态。

初学者最容易忽略这一点。比如，很多人会记住：

```bash
kubectl scale deployment web --replicas=3
```

但没有进一步意识到，这条命令的本质是：**把 `Deployment` 对象的副本期望值更新为 3**。命令只是形式，期望状态才是核心。你换成 YAML、换成 CI/CD 流水线、换成 GitOps 工具，本质做的仍然是同一件事。

所以，当我们说 Kubernetes 是“声明式系统”时，真正的意思是：**你描述最终目标，平台负责推进实现过程。** `kubectl` 只是这套系统里最直接、最常用、也最适合学习的入口。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我最近一直在背 `kubectl` 命令，感觉命令特别多，怎么都记不完。

**大师**：那你大概率把重点放错了。命令需要会，但不需要死记硬背到百科全书的程度。

**小白**：可我不用命令，怎么操作集群？

**大师**：命令只是“入口”。真正重要的是你是否明白：这些命令在修改什么对象，提交了什么期望状态，最终由哪个控制器接管。

**小白**：也就是说，我现在看到的是表层，真正的核心在对象模型？

**大师**：没错。比如 `kubectl apply -f deployment.yaml`，本质上不是“执行了一份脚本”，而是把一份期望状态提交给 API Server。后续 Deployment 控制器再根据这个期望去维护 ReplicaSet 和 Pod。

**小白**：那为什么很多人都强调要用 YAML，而不是一直敲命令？

**大师**：因为 YAML 更适合协作。它能进 Git、能走代码评审、能审计、能回滚、能复用。临时命令适合探索和排障，但正式交付必须尽量回到声明式配置。

**小白**：所以命令式更像临场操作，声明式更像团队约定的事实来源？

**大师**：说得很好。你可以用命令探索对象、生成模板、临时诊断；但真正长期维护的，应该是一份能够被团队共同理解和追踪的配置。

**小白**：那我现在最该养成什么习惯？

**大师**：三个习惯。第一，能用 YAML 表达的内容尽量落文件；第二，改完资源要看状态和事件，而不是只看命令有没有报错；第三，把 `kubectl` 当成“读写集群状态”的工具，而不是“背命令比赛”的工具。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们做一个最常见的练习：**先用命令生成资源模板，再回到 YAML，通过 `apply` 走完整个声明式工作流。** 这样做的好处是，你既不会被一开始就手写大量字段吓到，又能逐步建立配置管理习惯。

### 第一步：先用命令生成一个 Deployment 模板

```bash
kubectl create deployment web --image=nginx --dry-run=client -o yaml > web.yaml
```

这条命令非常适合新手。它不是直接创建资源，而是让客户端帮你生成一个最小可用 YAML。你可以把它理解成“脚手架”，后续再继续补字段。

打开 `web.yaml`，你会看到如下结构要点：

- `apiVersion` 和 `kind`：说明资源类型。
- `metadata`：资源名称、标签等元数据。
- `spec`：你真正想要的行为描述。

### 第二步：把模板交给集群维护

```bash
kubectl apply -f web.yaml
kubectl get deployment web
kubectl get pods -l app=web
```

为什么这里推荐用 `apply`，而不是 `create`？因为 `apply` 更符合后续的声明式维护心智。第一次是创建，后面再改 YAML 继续 `apply`，本质上是在持续同步“我现在认可的期望状态”。

### 第三步：增加副本数和标签，感受配置变更

修改 `web.yaml`，把 `replicas` 改成 `3`，并为 Pod 模板补一个更明确的标签：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: nginx
        image: nginx:1.27
        ports:
        - containerPort: 80
```

然后重新应用：

```bash
kubectl apply -f web.yaml
kubectl get deployment web
kubectl get pods -l app=web
```

你会看到 Deployment 的副本数变化生效，新的 Pod 会被创建出来。这里一定要刻意体会一个事实：不是命令“帮你扩容”，而是你修改了对象的期望状态，控制器据此进行了扩容。

### 第四步：学会最常用的观察命令

```bash
kubectl describe deployment web
kubectl describe pod <pod-name>
kubectl logs <pod-name>
kubectl get events --sort-by=.lastTimestamp
```

初学者最容易犯的错误，是只关注创建命令有没有报错，而忽略了对象后续状态。实际上，很多问题都不是创建时发生的，而是在调度、镜像拉取、探针检查、依赖连接时暴露出来。`describe` 和 `events` 是你排障的主武器。

### 第五步：理解命令式和声明式各自的位置

在学习和排障时，命令式很有价值。例如：

```bash
kubectl scale deployment web --replicas=5
kubectl set image deployment/web nginx=nginx:1.28
```

这类命令很适合临时实验、快速验证和演示。**但如果这些改动会长期保留，就应该回写到 YAML 文件。** 否则你迟早会遇到“线上状态变了，但代码库里没体现”的配置漂移问题。

### 这一节应该带走什么

- `kubectl` 是客户端，不是控制器本身。
- `apply` 更适合长期维护，`create` 和临时命令更适合快速探索。
- 真正需要管理的是对象状态，而不是零散命令历史。
- 观察状态、事件和日志，是声明式交付的一部分，不是附属动作。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 通过 `kubectl`，你可以快速读写集群状态，是学习 Kubernetes 的最佳入口。
- 声明式配置天然适合 Git 管理、代码评审、审计和回滚。
- 命令生成模板 + YAML 持续维护，是非常适合初学者的过渡路径。

### 缺点

- 命令很多、参数很多，新手容易把注意力放在表层语法上。
- 如果不理解对象模型，只会机械执行命令，学习会非常碎片化。
- 临时命令太方便，容易养成“线上手改”的坏习惯。

### 使用场景

- 日常查看集群资源与状态。
- 使用 YAML 进行工作负载、服务、配置等资源交付。
- 快速探索、排障和模板生成。

### 注意事项

- 正式配置尽量落到文件，不要长期依赖命令式修改。
- 每次修改后都观察状态，而不是只看命令成功返回。
- 记住 `kubectl` 只是入口，真正被维护的是对象状态。

### 常见踩坑经验

- 最常见的坑，是在生产环境大量使用临时命令修改资源，却不回写到配置文件中。
- 第二个坑，是把 `kubectl apply` 理解成“执行脚本”，而不是“同步期望状态”。
- 第三个坑，是出问题时只反复重试命令，不去看 `describe`、事件和对象字段本身。

这一章的核心收获应该是：**学会把 Kubernetes 当成状态系统来理解，而不是把 `kubectl` 当成一套高级命令行技巧。**
