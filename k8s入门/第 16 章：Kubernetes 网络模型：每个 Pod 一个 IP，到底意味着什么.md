# 第 16 章：Kubernetes 网络模型：每个 Pod 一个 IP，到底意味着什么

> 对应学习计划第 2 周 Day 8-9 的网络模型理解。

一旦进入 Kubernetes 网络，你会发现很多传统单机容器时代的直觉都会开始失效。最典型的一条，就是“容器网络不就是 bridge 加端口映射吗”。在 Kubernetes 里，这种理解只够你起步，远远不够你排障。因为 Kubernetes 网络模型的核心前提非常激进也非常优雅：**每个 Pod 都应该拥有独立 IP，并且 Pod 之间默认可以直接互通。**

## 1）项目背景

为什么 Kubernetes 网络模型这么重要？因为几乎所有上层能力都建立在它之上：

- Service 能不能正确转发到 Pod。
- 跨节点 Pod 是否能直接访问。
- DNS 是否能正确解析服务名。
- NetworkPolicy 是否能真正生效。
- Ingress 和网关流量最终能否到达目标实例。

如果不理解网络模型，一旦出现“Service 不通”“跨节点访问失败”“Pod IP 可达但域名不可达”“探针偶发失败”这类问题，排障就很容易退化成靠猜。

Kubernetes 的网络设计有几条非常关键的基本假设：

- 每个 Pod 拥有自己的 IP。
- Pod 之间通信不依赖 NAT，理论上可以直接互通。
- 节点上的 Agent 和网络插件要共同保证跨节点路由成立。
- Service 再在 Pod 网络之上，提供稳定抽象和流量分发。

这意味着，Kubernetes 不是简单把单机 Docker 网络放大，而是建立了一层“面向 Pod 的统一网络平面”。Pod 不再是隐藏在节点端口背后的附属对象，而是网络中的一等公民。

你可以把它和传统部署方式对比来看：

- 在单机 Docker 里，你更常感知的是端口映射和宿主机地址。
- 在 Kubernetes 里，你更应该感知的是 Pod IP、Service IP、节点 IP 各自扮演什么角色。

如果这三者在脑子里混成一团，后面学 kube-proxy、CNI、Ingress、NetworkPolicy 都会变得很痛苦。因此，本章真正要建立的是一张“网络基础地图”：**Pod 在哪里、流量怎么走、地址分别代表什么。**

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我有点懵。Pod 有 IP，Service 也有 IP，节点也有 IP，这三个到底谁是拿来干什么的？

**大师**：这正是 Kubernetes 网络最容易把人绕进去的地方。你得先分清对象层次：Pod IP 是实例地址，Service IP 是服务抽象入口，Node IP 是宿主机地址。

**小白**：那为什么 Kubernetes 要强调“每个 Pod 一个 IP”？直接像 Docker 一样做端口映射不行吗？

**大师**：可以做，但那样上层的服务发现、负载均衡、网络策略和跨节点通信都会复杂很多。Pod 有独立 IP，平台就能把实例网络统一抽象出来。

**小白**：是不是可以理解成，Kubernetes 想把 Pod 变成网络里的基本单位？

**大师**：对，而且这会极大简化上层抽象。Service 不再需要关心某个容器绑定了哪个宿主机端口，只需要关心“后面有哪些 Pod”。

**小白**：那跨节点通信为什么还要特地强调？

**大师**：因为这是 Kubernetes 网络模型真正有挑战的地方。同一节点上 Pod 互通相对简单，难的是 Pod A 在节点 1，Pod B 在节点 2，它们还得像在同一个大平面里一样互相访问。这就需要网络插件和节点路由配合。

**小白**：所以一旦跨节点不通，我就该想到是网络模型底层没打通，而不是先怪 Service？

**大师**：非常对。先区分是 Pod 到 Pod 不通，还是 Service 到 Pod 不通，再顺着链路排查。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们不急着讲太多原理图，而是先做一个“地址观察实验”，把 Pod IP、Service IP、节点 IP 三层关系看清楚。

### 第一步：观察 Pod 与节点地址

```bash
kubectl get pods -o wide
kubectl get nodes -o wide
```

你会在 `kubectl get pods -o wide` 里看到：

- Pod 自己的 IP。
- Pod 当前所在节点。

而在节点列表里，你会看到节点本身的内网地址。这一步要先建立一个最基本认识：**Pod IP 和节点 IP 不是一回事。** Pod 只是运行在节点上，不代表它就等于节点网络地址。

### 第二步：观察 Service 地址

```bash
kubectl get svc
kubectl describe svc web
```

你会看到 Service 又拿到了另一个 `CLUSTER-IP`。这个地址不是某个具体 Pod 的 IP，而是平台给这组后端实例提供的稳定入口。它的目的不是替代 Pod，而是把“实例会变”抽象成“服务地址不变”。

### 第三步：从容器里看 Pod 自己的网络栈

建议起一个调试 Pod，例如 `nicolaka/netshoot` 或 `busybox`：

```bash
kubectl run netshoot --image=nicolaka/netshoot -it --rm --restart=Never -- bash
```

进入容器后执行：

```bash
ip addr
ip route
nslookup web
curl http://web
```

这一步非常关键，因为它能帮助你从 Pod 内部视角理解网络：

- Pod 看到的是自己独立的 IP 和网卡。
- Pod 能通过服务名访问 Service。
- Pod 的默认路由并不是你单机 Docker 时代熟悉的那一套简单模型。

### 第四步：理解一条典型请求路径

现在你可以试着在脑中还原一条最典型的链路：

1. 一个 Pod 访问 `http://web`。
2. DNS 把 `web` 解析成 Service 的 ClusterIP。
3. 节点上的 kube-proxy/网络规则把流量转到某个后端 Pod。
4. 如果目标 Pod 在别的节点上，流量还要经过跨节点网络。

这条链路一旦想明白，后面很多“怎么又不通了”的问题就会更容易拆解。

### 第五步：形成一条网络排障顺序

建议固定采用这样的顺序：

- 先看 Pod 自己有没有 IP。
- 再看 Pod 到 Pod 是否互通。
- 再看服务名解析是否正常。
- 再看 Service 和 Endpoints 是否匹配。
- 最后再看节点转发和网络插件。

这条顺序之所以有效，就是因为它遵循了 Kubernetes 网络模型的层次结构。

### 这一节应该带走什么

- Pod IP、Service IP、Node IP 分别代表不同层次的地址语义。
- Kubernetes 网络不是单机 bridge 的简单放大版。
- “每个 Pod 一个 IP”是很多上层能力成立的基础前提。
- 排障要先分层，再定位，不要一上来混着看。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它为 Pod 提供统一网络身份，极大简化了服务发现和后端抽象。
- 它让平台更容易围绕 Pod 建立 Service、策略和观测体系。
- 一旦理解这套模型，后续学 Service、Ingress、CNI 和 NetworkPolicy 会顺很多。

### 缺点

- 对初学者来说，地址层次较多，直觉容易混乱。
- 跨节点通信链路较长，出问题时涉及路由、转发、插件等多个层面。
- 不同环境和网络插件的实现细节不同，学习初期容易把“模型”和“实现”混淆。

### 使用场景

- 理解集群内服务发现和流量转发基础。
- 为后续学习 kube-proxy、CNI、Service、NetworkPolicy 建立前置心智。
- 处理 Pod 跨节点访问、DNS 解析和 Service 链路排障。

### 注意事项

- 不要把 Pod IP、Service IP、Node IP 混为一谈。
- 先理解网络模型，再深入某个插件或实现细节。
- 排障时优先按链路分层，而不是上来就全局乱查。

### 常见踩坑经验

- 最常见的坑，是把 Kubernetes 容器网络理解成单机 Docker bridge 的简单放大版。
- 第二个坑，是一看到服务不通就只盯 Service，却不先验证 Pod 到 Pod 的基础连通性。
- 第三个坑，是看到 Pod 有 IP 就默认网络已经通，忽略了 DNS、转发和跨节点路由仍可能有问题。

这一章真正想帮你建立的，是一张网络地图：**只有先分清地址和链路，你才有资格谈后面的流量治理与网络排障。**
