# 第 30 章：ServiceAccount 与 API 访问：Pod 也需要身份，不是只有人需要

> 对应学习计划第 4-5 周 ServiceAccount、Token 与 API Server 访问。

很多开发者对 Kubernetes 权限的第一反应，都是“人怎么登录、谁能操作集群”。但一旦业务程序自己要访问 API Server，比如 Operator、控制器、自动化任务、集群内运维脚本，你就会发现：**不仅人需要身份，Pod 里的程序同样需要身份。** ServiceAccount 正是 Kubernetes 为工作负载提供的原生身份机制。

## 1）项目背景

为什么工作负载身份这么重要？因为集群里的程序并不只是“跑业务”，它们还可能需要：

- 读取自身所在命名空间的资源。
- watch 某类对象变化。
- 调用 API Server 实现控制逻辑。
- 配合 CI/CD 或平台自动化做集群操作。

如果没有 ServiceAccount，很多团队很容易走向一些危险做法：

- 把高权限 Token 直接打进镜像。
- 在代码仓库里硬编码 kubeconfig 或 Bearer Token。
- 让所有程序共用一个超级权限身份。

这些做法的共同问题是：身份不可控、轮换困难、泄露风险大，而且几乎无法做到最小权限。

ServiceAccount 的价值，就是把“Pod 里的程序是谁”这件事正式对象化。它通常会和 RBAC 结合，形成一条清晰链路：

- ServiceAccount 表示“这个工作负载以谁的身份运行”。
- RBAC 决定“这个身份能做什么”。
- Token 或挂载机制让应用在运行期能够以该身份调用 API。

这和上一章 RBAC 是天然衔接的。RBAC 解决“授权给谁”，ServiceAccount 解决“Pod 到底是谁”。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我有个程序要在集群里 watch Pod 变化，是不是直接把管理员 kubeconfig 塞进容器就行？

**大师**：这是非常典型、也非常危险的第一反应。你这样做虽然快，但等于把高权限身份直接焊进了镜像。

**小白**：那 ServiceAccount 是干什么的？

**大师**：它就是 Pod 的身份。你可以把它理解成“工作负载登录集群时用的账号”。

**小白**：所以它和用户账号不是一回事？

**大师**：对。它不是给人交互式登录用的，而是给集群里运行的程序用的。

**小白**：那它本身就带权限吗？

**大师**：不带。它只是身份载体，真正的权限要靠 RBAC 绑定上去。

**小白**：那是不是可以理解成：ServiceAccount 回答“我是谁”，RBAC 回答“我能干什么”？

**大师**：这就是最核心的理解。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们创建一个最小 ServiceAccount，并把它和只读权限绑定起来，形成一个完整的“工作负载身份 + 授权”闭环。

### 第一步：创建一个 ServiceAccount

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-reader
  namespace: team-a
```

这一步表达的是：在 `team-a` 命名空间里，我们要定义一个名为 `app-reader` 的工作负载身份。

### 第二步：把 ServiceAccount 绑定给 Pod

在 Pod 或 Deployment 中，你通常会这样引用它：

```yaml
spec:
  serviceAccountName: app-reader
```

这时，Pod 运行时就不再只是“一个匿名程序”，而是“以 `app-reader` 这个身份在集群里活动”。

### 第三步：把身份和 RBAC 关联起来

如果你把上一章创建的只读 Role 通过 RoleBinding 绑定给 `app-reader`，那么这个 Pod 里的程序就具备了读取指定资源的能力。

这里最重要的认知是：

- ServiceAccount 本身不等于权限。
- Role / Binding 本身不等于运行身份。
- 两者结合后，才形成真正能工作的访问模型。

### 第四步：在容器内理解“它为什么能访问 API”

在很多 Kubernetes 版本和默认配置下，Pod 运行时会拿到与 ServiceAccount 相关的访问凭据，供程序访问 API Server。这也正是为什么你不该自己再塞一份硬编码 Token 进去：平台已经有更规范的方式来交付身份。

### 第五步：形成工作负载身份设计思路

以后只要遇到“集群里的程序需要访问 Kubernetes API”这个需求，都先问自己：

1. 它应该使用哪个 ServiceAccount。
2. 它真正需要哪些最小权限。
3. 它是否在错误地共享了高权限身份。

这三问会帮你避免绝大多数“图方便先跑起来”的危险做法。

### 这一节应该带走什么

- ServiceAccount 是工作负载身份，不是交互式用户账号。
- 它要和 RBAC 结合，才能形成真正可控的 API 访问能力。
- Pod 不应依赖硬编码凭证访问集群。
- 身份和权限分离，是 Kubernetes 安全模型里非常关键的一层。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让工作负载身份从“临时凭证”升级为平台原生对象。
- 它与 RBAC 天然配合，便于实现最小权限访问。
- 它是 Operator、自动化任务、控制器等程序访问 Kubernetes API 的标准路径。

### 缺点

- 初学者容易把它和普通用户身份混淆。
- 如果权限绑定和命名规范不清晰，也可能形成新的治理混乱。
- 一旦团队习惯了“先塞管理员 Token 再说”，回到正规模式会有改造成本。

### 使用场景

- 集群内程序需要访问 Kubernetes API 的场景。
- 控制器、Operator、自动化脚本、平台组件身份管理。
- 想把程序访问集群能力纳入最小权限治理的团队。

### 注意事项

- 不要把高权限 Token 打包进镜像或提交进代码库。
- ServiceAccount 与 RBAC 总是要结合看，不要割裂理解。
- 不同工作负载最好使用不同身份，避免权限共享过大。

### 常见踩坑经验

- 最常见的坑，是把高权限 Token 打包进镜像或写入代码库。
- 第二个坑，是给所有工作负载共用一个高权限 ServiceAccount，导致权限面过大。
- 第三个坑，是只创建了 ServiceAccount，却没有绑定所需权限，最终程序访问 API 仍然失败。

这一章真正想帮你建立的，是工作负载身份视角：**不仅人要有身份，集群里的程序也必须以可治理的身份访问平台。**
