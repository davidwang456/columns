# 第 23 章：CSI：存储插件化之后，Kubernetes 才真正长出了手脚

> 对应学习计划第 2 周存储插件与 CSI。

当你真正开始在 Kubernetes 里接入云盘、NFS、Ceph、快照、扩容等高级存储能力时，很快就会发现，PV/PVC/StorageClass 只是存储抽象层，而真正让这些抽象“落地到不同后端”的，是 CSI 驱动。也就是说，**CSI 不是另一个附属名词，而是 Kubernetes 存储生态真正能扩展起来的关键接口。**

## 1）项目背景

早期 Kubernetes 对很多存储类型采用的是内置实现，也就是所谓 in-tree volume 插件。这种方式能工作，但很快暴露出问题：

- 存储后端一多，Kubernetes 核心代码负担越来越重。
- 每次存储能力更新，都可能牵动 Kubernetes 本身发布。
- 云厂商、开源存储和第三方生态难以高效接入。

CSI（Container Storage Interface）出现后，情况发生了根本变化。它做了两件非常重要的事：

- 把存储接入标准接口化。
- 把具体存储实现从 Kubernetes 核心中解耦出去。

这意味着，无论后端是云盘、NFS、Ceph、SAN，还是别的分布式存储，只要实现了 CSI 接口，就能比较标准地接入 Kubernetes 的存储抽象体系。

从平台视角看，CSI 的价值主要体现在：

- 动态供应更容易标准化。
- 快照、扩容、挂载等能力有统一入口。
- 存储厂商或开源社区可以独立演进驱动能力。
- Kubernetes 本身不必为每种存储实现背负全部复杂性。

所以，学习 CSI 不能只停在“装个驱动”。更重要的是理解：**Kubernetes 通过 CSI，把“声明式存储需求”和“具体存储后端能力”连接了起来。**

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我已经知道 PVC 和 StorageClass 了，那 CSI 又是在解决什么问题？

**大师**：PVC 和 StorageClass 描述的是“平台怎么表达存储需求”，CSI 解决的是“这些需求到底由谁、以什么方式去实现”。

**小白**：也就是说，没有 CSI，很多高级存储能力其实没法真正落地？

**大师**：对。你可以把 CSI 看成 Kubernetes 和具体存储系统之间的标准适配层。

**小白**：那它和网络里的 CNI 有点像？

**大师**：非常像。一个是网络接口标准，一个是存储接口标准。它们都在做同一件事：把平台模型和底层实现解耦。

**小白**：那我作为开发者，为什么要懂它？直接用 PVC 不就行了吗？

**大师**：因为一旦出现卷挂载失败、动态供应异常、快照不可用、扩容不生效，你就不能只停留在 PVC 这一层。你得知道背后有驱动、有控制器、有节点插件在协同工作。

**小白**：原来 CSI 是“存储生态如何接进来”的答案。

**大师**：没错，而且理解这件事，会让你对 Kubernetes 的扩展哲学更有感觉。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们以 NFS CSI 驱动为例，感受一个具体驱动如何进入集群，并与 PVC/StorageClass 配合工作。

### 第一步：安装一个 CSI 驱动

```bash
helm repo add csi-driver-nfs https://raw.githubusercontent.com/kubernetes-csi/csi-driver-nfs/master/charts
helm install csi-nfs csi-driver-nfs/csi-driver-nfs -n kube-system
```

安装完成后，先观察新增了哪些组件：

```bash
kubectl get pods -n kube-system
```

你通常会看到：

- 控制器侧组件。
- 节点侧插件。
- 与 CSI 驱动配套的若干 sidecar。

这一步非常重要，因为它能帮你从“我装了个驱动”升级成“我看到了一整套存储接入协作组件”。

### 第二步：理解 CSI 的协作分层

你不需要第一天就记住所有 sidecar 名字，但至少要知道大概分工：

- 控制器侧：处理卷创建、删除、扩容、快照等控制平面动作。
- 节点侧：处理具体节点上的挂载、卸载和设备准备。
- StorageClass / PVC：业务侧仍然只通过声明式需求发起申请。

这个分层和前面很多 Kubernetes 设计是同一套哲学：上层声明、控制器协调、节点执行。

### 第三步：把 CSI 驱动与 StorageClass 串起来

很多 CSI 驱动会配套自己的 `StorageClass`。一旦配置好了，PVC 提交后就不再只是“理论上的申请”，而是可以真正触发对应后端卷的创建或子目录分配。

也就是说，从这一步开始，PVC 的“背后执行者”终于有了现实抓手。

### 第四步：为什么 CSI 让存储生态真正可扩展

学到这里，你就会明白：

- 没有 CSI，Kubernetes 很难优雅支持越来越多的存储后端。
- 有了 CSI，存储能力不再强耦合在 Kubernetes 核心里。
- 平台团队也终于能把卷快照、在线扩容、动态供应这些能力纳入标准路径。

### 第五步：学习阶段最容易忽略的边界

不要因为“装上了 CSI 驱动”就默认这些高级能力全可用。不同驱动对以下能力支持差异很大：

- 快照
- 在线扩容
- 访问模式
- 多节点挂载
- 故障恢复体验

所以，CSI 学习中非常重要的一点，是把“有接口”与“驱动真的支持”区分开。

### 这一节应该带走什么

- CSI 是 Kubernetes 存储插件化和生态扩展的关键接口。
- 它让存储后端能力能够标准化接入平台。
- 理解 CSI 有助于你把 PVC 问题继续往后端能力和驱动层追。
- “装了驱动”不等于“所有高级存储能力都自动拥有”。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把存储能力从 Kubernetes 核心中解耦出来，极大提升了生态扩展性。
- 它让快照、扩容、动态供应等能力更容易标准化。
- 它为云盘、NFS、Ceph 等不同后端接入提供了统一接口。

### 缺点

- 理解门槛高，涉及控制器、节点插件和后端系统多个层级。
- 不同驱动实现差异较大，学习和排障复杂度明显上升。
- 表面对象一致，不代表底层能力完全一致，容易产生错误期待。

### 使用场景

- 接入多种外部存储系统。
- 需要动态供应、快照、在线扩容等高级能力的场景。
- 平台团队建设统一存储接入能力时。

### 注意事项

- 先理解 CSI 的角色，再去背具体驱动安装步骤。
- 不要在不了解驱动能力的情况下，默认快照和扩容一定可用。
- 出现存储异常时，要意识到问题可能已经超出 PVC 层，进入驱动层了。

### 常见踩坑经验

- 最常见的坑，是在不了解驱动能力的情况下盲目依赖卷快照和在线扩容。
- 第二个坑，是把 PVC 绑定成功误认为所有挂载和运行期行为一定没问题。
- 第三个坑，是只会看业务 Pod 状态，不去看 CSI 控制器和节点插件日志。

这一章真正想帮你建立的，是存储扩展视角：**PVC 是声明入口，CSI 才是很多真实存储能力的落地桥梁。**
