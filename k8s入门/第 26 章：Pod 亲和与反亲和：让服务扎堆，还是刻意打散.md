# 第 26 章：Pod 亲和与反亲和：让服务扎堆，还是刻意打散

> 对应学习计划第 3 周 Pod 亲和与反亲和。

上一章的节点标签与节点亲和，解决的是“Pod 适合哪类机器”；而这一章的 Pod 亲和与反亲和，则把调度思路再往前推一步：**有时候，Pod 的落点不仅取决于节点本身，还取决于别的 Pod 在哪。** 同一服务是应该尽量挨着缓存、减少跨节点调用延迟，还是应该故意打散，避免单点故障，这背后没有唯一标准，只有取舍。Pod 亲和与反亲和，正是把这种取舍正式表达给调度器的工具。

## 1）项目背景

在真实系统里，工作负载的关系往往不仅仅是“我需要 CPU/内存”，还包括“我希望离谁近”或者“我最好别跟谁挤在一起”。比如：

- Web 服务如果和本地缓存、侧车代理、同类依赖靠得更近，可能减少网络延迟。
- 核心副本如果都挤在一个节点，一旦节点故障，等于服务被一锅端。
- 某些任务天然适合与数据服务靠近，减少跨区域通信。
- 某些高可用组件则必须跨节点甚至跨可用区分散。

如果没有 Pod 亲和与反亲和，你只能依赖默认调度器“尽量均匀分布”，或者靠人工猜测副本会不会被打散。这显然不可靠，因为默认均匀只是一般策略，不代表一定符合你的业务语义。

Kubernetes 提供的两种主要能力是：

- `podAffinity`：告诉调度器，我希望和某些 Pod 靠近。
- `podAntiAffinity`：告诉调度器，我希望和某些 Pod 分开。

这类规则最关键的地方不在字段本身，而在 `topologyKey`。它决定你所谓的“靠近”或“分开”到底是以什么拓扑为单位：

- 同一节点 `kubernetes.io/hostname`
- 同一可用区 / 机架 / 拓扑域
- 或者平台自定义的其他拓扑标签

一旦理解这一点，你就会意识到：Pod 亲和与反亲和，本质上不是“玄学调度”，而是在用拓扑语言描述性能与可用性的平衡。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我知道可以用节点标签控制调度了，那 Pod 亲和性又是在解决什么问题？

**大师**：节点标签回答的是“机器合不合适”，Pod 亲和回答的是“和谁在一起、和谁分开更合适”。

**小白**：这听起来像是从“机器视角”切到了“应用关系视角”。

**大师**：对。比如两个副本如果都跑在同一节点上，通信可能快一点，但故障风险也更集中；如果刻意分散，可用性更好，但网络路径可能更长。

**小白**：那是不是高可用场景都应该用反亲和？

**大师**：大多数时候是这样，但也不能一刀切。集群节点少、资源紧张时，过于严格的反亲和可能会把 Pod 全卡成 Pending。

**小白**：那亲和是不是更偏性能优化，反亲和更偏高可用？

**大师**：这是一个很好的初学理解。虽然现实会更复杂，但这个方向基本对。

**小白**：所以关键不是“有没有写亲和性”，而是“我到底在优化延迟还是在降低故障集中度”。

**大师**：完全正确。调度字段从来不是为了写而写，它是在表达业务的真实运行偏好。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们用最常见的反亲和场景入手：让 Web 副本尽量不要落在同一节点，减少单节点故障带来的集中风险。

### 第一步：给 Deployment 增加 `podAntiAffinity`

```yaml
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchExpressions:
        - key: app
          operator: In
          values:
          - web
      topologyKey: kubernetes.io/hostname
```

这段配置的意思是：

- 找到标签 `app=web` 的 Pod。
- 不要把新的同类 Pod 放到和它们同一个 `hostname` 拓扑域里。

直白一点说，就是“同一个节点上尽量别堆多个同类副本”。

### 第二步：观察副本是否被打散

```bash
kubectl get pods -l app=web -o wide
kubectl describe pod <pod-name>
```

如果你的集群节点足够多，你通常会看到副本被分散到不同节点上。这里要养成一个习惯：不要只看 Pod 有没有 Running，更要看它到底跑在哪。

### 第三步：理解为什么会 Pending

如果你的集群只有 2 个节点，却要求 3 个副本都必须互相反亲和，而且是 `requiredDuringSchedulingIgnoredDuringExecution`，那第三个副本很可能一直 Pending。这不是 Kubernetes 出错，而是你把规则写得比集群容量更严格了。

这一步非常重要，因为它能帮助你理解：**调度规则不是白给的，每写一条硬约束，都是在消耗调度器的回旋空间。**

### 第四步：把“必须”改成“尽量”

在很多真实场景中，更合理的写法反而是偏好型：

- `preferredDuringSchedulingIgnoredDuringExecution`

它的语义不是“必须分开”，而是“能分开最好，分不开也别卡住服务”。这就是典型的平台权衡：有时候 100% 理想分布，不如 80% 分布 + 100% 可调度来得实用。

### 第五步：什么时候考虑 `podAffinity`

如果你有这些场景，就会更偏向亲和：

- 与本地缓存或数据代理同节点协作。
- 某些高频调用组件希望缩短网络路径。
- 特定业务链路想在同一拓扑域内运行。

不过请注意，亲和不是“越靠近越好”，它也可能带来热点和故障集中问题。

### 这一节应该带走什么

- Pod 亲和与反亲和是在表达 Pod 与 Pod 的拓扑关系，而不只是节点条件。
- 反亲和常用于高可用分散，亲和常用于局部性优化。
- `topologyKey` 决定了“靠近/分开”的实际范围。
- 硬约束和软偏好之间，永远存在可用性与理想布局的取舍。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让调度能够表达应用间关系，而不仅是机器条件。
- 它为高可用部署、热点隔离和局部性优化提供了标准化手段。
- 它和节点亲和、污点容忍组合使用时，能构成更完整的调度策略。

### 缺点

- 学习和排障复杂度明显高于普通节点选择。
- 一旦规则写得过严，Pod 很容易 Pending。
- 在大规模集群中，复杂亲和性规则也会增加调度开销和理解成本。

### 使用场景

- 需要同类副本分散部署的高可用服务。
- 需要和特定 Pod 靠近部署、优化延迟的链路。
- 需要按节点、可用区等拓扑域表达部署偏好的系统。

### 注意事项

- 写亲和性前先明确目标是“优化性能”还是“提升可用性”。
- 小集群里尽量优先用偏好型规则，避免把自己卡死。
- 一定要理解 `topologyKey`，否则很容易以为自己在“分散”，实际上分散范围根本不对。

### 常见踩坑经验

- 最常见的坑，是在小集群里写过于严格的反亲和，结果大量副本 Pending。
- 第二个坑，是看到了亲和性字段，却没想清楚它到底要优化什么，最后写成装饰配置。
- 第三个坑，是忽略 `topologyKey`，导致规则和预期拓扑粒度完全不一致。

这一章真正想帮你建立的，是一种拓扑调度视角：**不是所有副本都应该均匀，也不是所有副本都应该靠近，关键在于你到底想优化什么。**
