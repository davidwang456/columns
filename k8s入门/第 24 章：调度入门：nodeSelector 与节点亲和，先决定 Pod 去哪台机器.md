# 第 24 章：调度入门：nodeSelector 与节点亲和，先决定 Pod 去哪台机器

> 对应学习计划第 3 周 Day 15-16 的节点选择与亲和性。

默认调度很好用，但它并不是万能的。只要你的业务开始对节点有明确要求，比如 GPU、SSD、高内存、指定机房、专属节点池，你就会发现“随便调度到任意节点”已经不够用了。`nodeSelector` 和 `nodeAffinity` 的价值，就是让开发者把这些基础设施约束正式表达给调度器，而不是靠人工挑节点或者临时写死节点名。

## 1）项目背景

为什么节点选择会成为一个独立主题？因为真实集群里的节点往往并不一样：

- 有的节点带 GPU。
- 有的节点是 SSD，本地盘性能更好。
- 有的节点属于特定可用区或地域。
- 有的节点专门留给某类业务使用。

如果不把这些差异表达给调度器，常见结果会是：

- 需要 GPU 的任务被调度到普通节点，根本跑不起来。
- 对磁盘性能敏感的服务落在慢盘节点上，延迟飙升。
- 跨可用区部署策略没有被表达，导致流量和存储成本异常。
- 平台团队精心划好的节点池，被默认调度策略打乱。

Kubernetes 的思路不是让你直接指定“就上 node-3”，而是通过标签和约束表达“我需要怎样的节点条件”。最常见的两种工具就是：

- `nodeSelector`：最简单直接的硬匹配。
- `nodeAffinity`：更灵活，可以表达硬约束和软偏好。

这套设计非常符合 Kubernetes 一贯的风格：不要直接控制具体实例，而是声明约束，让调度器去找满足条件的地方。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我的模型服务必须跑在 GPU 节点上，那我是不是直接把 Pod 指定到某个节点名最省事？

**大师**：看起来省事，实际上很脆弱。节点名是具体实例，集群一变、节点一替换，你的配置就失效了。

**小白**：那我该怎么告诉调度器“我要 GPU 节点”？

**大师**：给节点打标签，然后让 Pod 声明它需要哪些标签。这样你表达的是“条件”，不是“某一台机器”。

**小白**：`nodeSelector` 和 `nodeAffinity` 的区别是什么？

**大师**：`nodeSelector` 更简单，适合直接匹配；`nodeAffinity` 更强大，可以区分硬约束和软偏好，还能写更复杂的表达式。

**小白**：所以调度其实不是“点名某台机器”，而是“描述我适合什么机器”。

**大师**：这就是声明式调度思维。你给条件，调度器做决定。

**小白**：那如果我条件写太死，会怎样？

**大师**：Pod 很可能一直 Pending。越是强约束，越要意识到你是在和集群容量做交易。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先给节点打标签，再分别用 `nodeSelector` 和 `nodeAffinity` 体验硬约束与偏好表达。

### 第一步：给节点打标签

```bash
kubectl label nodes worker-1 disktype=ssd
kubectl label nodes worker-1 zone=az1
kubectl get nodes --show-labels
```

你要先建立一个非常重要的前提：调度约束不是凭空成立的，它依赖标签体系。标签如果没有规范，调度规则也会很快失控。

### 第二步：使用 `nodeSelector`

```yaml
nodeSelector:
  disktype: ssd
```

这表示 Pod 只能被调度到 `disktype=ssd` 的节点上。它简单、直接、易读，非常适合初学阶段和规则明确的场景。

### 第三步：使用 `nodeAffinity` 表达更复杂条件

```yaml
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: zone
          operator: In
          values:
          - az1
```

这段配置表达的是一个硬约束：调度时必须在 `zone=az1` 的节点上。相比 `nodeSelector`，这里的优势在于可以组合更多表达式，也可以进一步写“偏好但不强制”的规则。

### 第四步：理解硬约束和软约束的差别

这是学习亲和性时必须抓住的核心：

- `requiredDuringSchedulingIgnoredDuringExecution`：硬条件，不满足就不调度。
- `preferredDuringSchedulingIgnoredDuringExecution`：软偏好，尽量满足，但必要时可以不满足。

一旦理解这点，你就不会把所有约束都写成“硬死规则”，把自己逼进 Pending 死胡同。

### 第五步：形成一个调度设计原则

在真实环境里，优先顺序通常是：

1. 先定义清晰的节点标签体系。
2. 再用 `nodeSelector` 解决最简单确定的需求。
3. 当约束变复杂时，升级到 `nodeAffinity`。
4. 尽量表达“条件”，而不是绑死某台机器。

### 这一节应该带走什么

- 节点选择本质上是在把基础设施约束声明给调度器。
- `nodeSelector` 简单直接，`nodeAffinity` 更灵活更强大。
- 硬约束和软偏好是调度表达里最关键的区别。
- 不要把调度约束退化成写死节点名。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让调度结果更符合底层基础设施差异和业务需求。
- 它把“机器适配条件”从人工经验变成平台规则。
- 它为 GPU、SSD、专用节点池、多可用区等场景提供了标准表达方式。

### 缺点

- 标签体系一旦混乱，调度规则会迅速失去可维护性。
- 约束越多越硬，越容易把 Pod 卡成 Pending。
- 学习者很容易从“声明条件”滑向“写死节点”。

### 使用场景

- 需要指定硬件能力或节点池的业务。
- 对地域、可用区、磁盘类型有要求的服务。
- 平台团队需要把节点资源分层治理时。

### 注意事项

- 先设计标签体系，再设计调度规则。
- 优先表达“需求条件”，不要表达“具体实例”。
- 写硬约束前先确认集群里确实有足够容量满足它。

### 常见踩坑经验

- 最常见的坑，是直接把 Pod 绑死到具体节点名，结果节点一换配置就废了。
- 第二个坑，是标签随便命名、随便使用，最后没人说得清哪些规则还有效。
- 第三个坑，是所有要求都写成硬约束，导致调度器几乎没有回旋空间。

这一章真正想帮你建立的，是一种声明式调度视角：**不要告诉平台“去哪台机器”，而要告诉它“什么机器才适合我”。**
