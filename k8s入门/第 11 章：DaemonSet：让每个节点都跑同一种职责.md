# 第 11 章：DaemonSet：让每个节点都跑同一种职责

> 对应学习计划第 1 周 DaemonSet 日志采集实践。

当你开始接触 Kubernetes 的系统组件时，很快会发现有一类程序和普通业务服务完全不一样。它们不是“我需要 3 个副本”或者“我需要弹性扩缩容”，而是“只要集群里有一个节点，我就希望这个节点上一定有我”。日志采集器、监控 Agent、CNI 组件、节点安全代理，几乎都属于这一类。DaemonSet 存在的意义，就是把这种“按节点覆盖”的需求正式交给平台处理。

## 1）项目背景

如果没有 DaemonSet，团队通常会怎么做节点级组件部署？最原始的方式，是手工登录每台机器安装 Agent；稍微现代一点的方式，是用一套运维脚本给每个节点下发同样的容器；再进一步，有些人会试图用 Deployment 设置一个差不多的副本数，然后“希望”它能覆盖所有节点。

这几种方式都存在明显问题：

- 手工安装和脚本分发难以随着节点数量变化自动扩展。
- 节点新增后，需要额外补装组件，流程不统一。
- Deployment 只能保证总副本数，不能保证“每个节点都有一个”。
- 节点下线、替换、扩容时，节点级组件很容易遗漏。

而节点级组件往往又特别关键。比如：

- 日志采集器要贴着节点读取容器日志。
- 监控 Agent 要采集节点级指标。
- 网络插件要在每个节点上接管网络路径。
- 安全 Agent 要监听节点上的系统行为和容器事件。

这些组件有一个共同特点：**它们的部署维度不是服务，而是节点。**

DaemonSet 的模型正好匹配这个诉求。它会保证：在满足调度条件的节点上，每个节点都运行一个指定 Pod。当有新节点加入集群时，DaemonSet 会自动在新节点上拉起对应 Pod；当某个节点被移除时，相应 Pod 也会自动消失。这样一来，开发者和平台团队不再需要自己维护“节点覆盖率”这件事。

你也可以把 DaemonSet 理解成“节点级工作负载控制器”。Deployment 关心的是副本数，DaemonSet 关心的是节点覆盖面。这个区别非常重要，因为它决定了你应该如何设计日志、监控、网络和安全组件的交付方式。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我想在每个节点上都跑一个日志采集器，是不是直接建一个 5 副本 Deployment 就差不多了？

**大师**：这正是很多人刚入门时最容易犯的错误。Deployment 只能保证有 5 个副本，却不保证这 5 个副本分布在 5 个不同节点上。

**小白**：可我不是也可以配反亲和，把它们尽量打散吗？

**大师**：那只能“尽量”，而不是“必须每个节点一个”。而且一旦节点数变化，你还得自己重新估算副本数，整个模型天生就不稳。

**小白**：所以 DaemonSet 的本质不是“多副本”，而是“按节点覆盖”？

**大师**：对。它表达的是“每个符合条件的节点都要有一个”。节点多一个，它就多一个；节点少一个，它就少一个。

**小白**：那它适合拿来部署普通业务吗？

**大师**：通常不适合。普通业务看的是服务容量和请求量，应该关心副本数、扩缩容和滚动发布；DaemonSet 更适合系统组件、Agent 和贴近节点运行的工具。

**小白**：那是不是所有节点都会跑它？控制平面节点也会吗？

**大师**：不一定，取决于你的 `nodeSelector`、`affinity` 和 `tolerations`。你可以控制它跑哪些节点，也可以允许它跑进带污点的专用节点。

**小白**：明白了，Deployment 管“服务规模”，DaemonSet 管“节点覆盖”。

**大师**：这句话已经抓住核心了。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们用一个最小 DaemonSet 模拟节点级 Agent 的部署，重点观察“每个节点一个 Pod”的行为特征。

### 第一步：编写最小 DaemonSet

新建 `node-agent-daemonset.yaml`：

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: node-agent
spec:
  selector:
    matchLabels:
      app: node-agent
  template:
    metadata:
      labels:
        app: node-agent
    spec:
      containers:
      - name: agent
        image: fluent/fluent-bit:3.1
        resources:
          requests:
            cpu: "50m"
            memory: "64Mi"
```

这里先不加复杂的宿主机目录挂载、容忍度和节点选择器，只保留最小闭环，让你先看清对象行为。

### 第二步：创建并观察节点覆盖

```bash
kubectl apply -f node-agent-daemonset.yaml
kubectl get daemonset node-agent
kubectl get pods -l app=node-agent -o wide
kubectl get nodes
```

你应该重点观察两个数字：

- 集群里有多少个可调度节点。
- `node-agent` 实际运行了多少个 Pod。

在没有额外限制的情况下，这两个数字通常会高度对应。你也可以通过 `-o wide` 看每个 Pod 分别落在哪个节点上。

### 第三步：理解 DaemonSet 的自动补齐能力

如果你的实验环境支持新增节点，可以在 `kind` 或其他本地集群中模拟增加节点后重新执行：

```bash
kubectl get pods -l app=node-agent -o wide
```

你会看到新节点出现后，DaemonSet 会自动把 Pod 拉起来。即便你不能现场新增节点，也要记住它的语义：**它不是在维护一个固定副本数，而是在维护一个动态节点覆盖面。**

### 第四步：加入节点过滤和容忍逻辑

真实生产环境里，DaemonSet 往往不会无差别跑在所有节点上，而是会配合：

- `nodeSelector` 或 `nodeAffinity`：只跑到指定节点池。
- `tolerations`：允许跑进带污点的专用节点。
- `hostPath` / `hostNetwork` / `hostPID`：访问宿主机资源。

比如一个只部署在 Linux 工作节点上的 Agent：

```yaml
nodeSelector:
  kubernetes.io/os: linux
```

这能帮助你理解，DaemonSet 的能力不是“强行 everywhere”，而是“在符合条件的每个节点上都有一份”。

### 第五步：理解它与普通服务的边界

不要因为 DaemonSet 看起来“覆盖更全”，就误把它拿来当普通业务控制器。大部分业务服务应该随着流量扩缩容，而不是随着节点数量扩缩容。这个边界一定要分清。

### 这一节应该带走什么

- DaemonSet 维护的是节点覆盖面，而不是固定副本数。
- 它非常适合日志、监控、网络和安全等节点级组件。
- 它通常会和 `nodeSelector`、`tolerations` 等调度能力一起使用。
- 它不是普通业务发布的默认答案。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让节点级组件的部署从人工维护变成平台自动覆盖。
- 节点新增或替换时，节点级 Pod 能自动补齐，运维成本大幅降低。
- 它和宿主机资源访问、节点标签、污点容忍等机制天然适配。

### 缺点

- 学习者容易把它和 Deployment 的多副本能力混淆。
- 一旦节点很多，DaemonSet 的资源消耗会线性放大。
- 节点级组件常常需要更高权限或更多宿主机能力，安全风险更高。

### 使用场景

- 日志采集器、监控 Agent、CNI 网络组件、安全探针等系统级组件。
- 需要在每个节点本地采集或处理数据的场景。
- 需要随着节点生命周期自动扩展和收缩的 Agent 型程序。

### 注意事项

- 优先明确它的“节点级”语义，不要拿它替代 Deployment。
- 控制好资源请求，避免 Agent 抢占太多业务资源。
- 如果涉及宿主机挂载和高权限运行，要同时评估安全边界。

### 常见踩坑经验

- 最常见的坑，是用 Deployment 人工估算副本数来覆盖节点，结果新增节点后组件没跟上。
- 第二个坑，是 DaemonSet 没加过滤和容忍策略，导致跑进不该跑的节点池。
- 第三个坑，是只顾覆盖每个节点，却忽略单节点 Agent 本身的资源消耗和权限风险。

这一章真正要建立的，不只是对 DaemonSet 的记忆，而是一种很重要的平台视角：**有些工作负载的维度不是“服务有多少副本”，而是“集群里有多少节点”。**
