# 第 12 章：Job 与 CronJob：一次性任务和周期任务的正确打开方式

> 对应学习计划中批处理与定时任务的延伸补充。

大多数人刚开始把业务迁到 Kubernetes 时，只会想到 Deployment 和 Service，于是顺手把数据库备份、离线修复、账单对账、定时清理这些任务，也硬塞进在线服务容器里。短期看似少了一个对象，长期却会让任务不可观察、不可重试、不可审计。Job 与 CronJob 的意义，就是把“跑完就结束”和“按时间触发”这两类任务，重新拉回到平台可管理的轨道上。

## 1）项目背景

在线服务和离线任务虽然都可能运行在容器里，但它们的生命周期完全不同：

- 在线服务希望一直活着，随时接请求。
- 批处理任务希望成功完成，然后退出。
- 定时任务希望按计划触发，不需要一直常驻。

如果把这些任务塞进 Deployment，会出现很多问题：

- 任务成功结束后，Deployment 会把它当异常退出继续拉起。
- 任务失败重试逻辑不清晰，只能靠脚本自己兜底。
- 定时任务常常被做成容器内 crond，日志分散、执行历史模糊。
- 一次性任务和在线服务混在一起，职责边界被破坏。

Kubernetes 提供了两类对象专门处理这类场景：

- `Job`：适合一次性任务，重点是“跑完并成功完成”。
- `CronJob`：适合周期任务，重点是“按计划创建 Job”。

这个设计非常符合工程实践。平台不再把“任务”当作一个长期在线服务看待，而是把它当作一个有开始、有结束、有成功或失败结果的工作单元。于是你可以更清楚地表达：

- 这个任务失败后是否需要重试。
- 这个任务要跑几个并行实例。
- 这个定时任务多久执行一次。
- 任务历史该保留多少。

这会让备份、报表、补偿、清理、迁移、离线计算这类逻辑变得更清晰，也更容易纳入平台治理。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我有个凌晨两点跑一次的数据库备份脚本，最省事的办法是不是把 `crond` 装进业务容器？

**大师**：这就是很多团队一开始会走的弯路。它看上去省事，实际上把在线服务和定时任务绑死了。

**小白**：可我只是想定时执行一条命令而已。

**大师**：问题不在命令，而在生命周期。你的备份任务不是“永远活着”，而是“按计划启动一次、执行、结束、记录结果”。Deployment 并不擅长表达这种语义。

**小白**：那 CronJob 就是在平台层面帮我做调度？

**大师**：对，而且它不是直接跑容器，而是按时间创建 Job。Job 再负责真正执行任务并记录完成状态。

**小白**：那 Job 和普通 Pod 的区别是什么？

**大师**：Pod 更像运行实例，Job 更像“我要把这件事做完”。如果 Pod 失败，Job 会根据策略补新实例继续做，直到成功或达到失败阈值。

**小白**：明白了。在线服务强调持续可用，任务强调最终完成。

**大师**：非常准确。对象之所以分开，就是因为平台要理解你到底在维护“在线能力”还是“完成结果”。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先用 `Job` 跑一个一次性任务，再用 `CronJob` 做一个周期任务，感受两者在运行语义上的差别。

### 第一步：创建一个最小 Job

新建 `backup-job.yaml`：

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: one-time-backup
spec:
  template:
    spec:
      restartPolicy: OnFailure
      containers:
      - name: backup
        image: busybox:1.36
        command:
        - sh
        - -c
        - echo "run backup once" && sleep 5
```

应用并观察：

```bash
kubectl apply -f backup-job.yaml
kubectl get jobs
kubectl get pods
kubectl logs job/one-time-backup
```

你应该观察到：

- Job 创建后会拉起 Pod 执行任务。
- 任务成功完成后，Pod 不会像 Deployment 那样被长期维持运行。
- Job 会记录完成状态，告诉你这件事是否做完。

### 第二步：理解 Job 的完成语义

如果你把命令改成必定失败，例如 `exit 1`，再观察 Job 状态，就能看到平台会根据策略重试。这一步能帮你建立一个关键认知：**Job 关心的不是“容器一直活着”，而是“工作有没有最终完成”。**

### 第三步：创建一个最小 CronJob

新建 `db-backup-cronjob.yaml`：

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: db-backup
spec:
  schedule: "*/5 * * * *"
  successfulJobsHistoryLimit: 2
  failedJobsHistoryLimit: 1
  jobTemplate:
    spec:
      template:
        spec:
          restartPolicy: OnFailure
          containers:
          - name: backup
            image: busybox:1.36
            command:
            - sh
            - -c
            - date && echo "backup task"
```

这里我把时间调成每 5 分钟一次，方便本地实验。

### 第四步：观察 CronJob 如何定时创建 Job

```bash
kubectl apply -f db-backup-cronjob.yaml
kubectl get cronjob db-backup
kubectl get jobs --watch
```

你会看到：

- CronJob 本身不会直接执行业务逻辑。
- 到时间后，它会创建一个新的 Job。
- 每个 Job 再各自拥有执行 Pod 和结果状态。

这层嵌套结构正好体现了平台的抽象设计：**CronJob 管时间，Job 管完成，Pod 管运行。**

### 第五步：几个很重要的工程注意点

- 定时任务如果涉及数据库写入、账务补偿、批量清理，必须考虑幂等性。
- 周期任务不要默认假设“绝不会重复执行”，要预留补偿思路。
- 如果任务耗时较长，还要考虑并发策略，例如是否允许上一次没跑完时下一次继续开始。

这些并不是 CronJob 自动替你解决的，但对象模型能让你把这些规则明确表达出来，而不是藏在脚本里。

### 这一节应该带走什么

- Job 适合一次性任务，关注的是“最终完成”。
- CronJob 适合周期任务，关注的是“按计划创建 Job”。
- 在线服务和离线任务是两类完全不同的生命周期模型。
- 幂等性、重试和并发策略是任务设计的关键补充。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让任务从“脚本行为”升级为“平台对象”，可观察、可重试、可记录。
- 它把在线服务与批处理职责清晰拆分，避免生命周期混乱。
- 它非常适合做备份、报表、补偿、清理和迁移等标准化任务管理。

### 缺点

- 定时任务和批处理的复杂性并不会因为用了对象而自动消失。
- 一旦幂等性和并发策略没设计好，平台只会更稳定地重复错误。
- 任务失败排查有时比在线服务更隐蔽，因为它不是一直活着的实例。

### 使用场景

- 一次性数据修复、迁移、导入导出。
- 数据库备份、报表生成、定时清理等周期任务。
- 需要有明确完成状态和历史记录的离线工作单元。

### 注意事项

- 用 Job/CronJob 表达任务，不要把任务塞进 Deployment 容器里长期运行。
- 关注任务幂等性，尤其是定时补偿和写入型任务。
- 合理设置任务历史保留、失败重试和并发策略。

### 常见踩坑经验

- 最常见的坑，是在 Deployment 容器里加 `crond` 解决定时任务，结果职责混乱、日志分散、排障困难。
- 第二个坑，是把任务是否成功完全交给脚本自觉处理，而不借助 Job 的完成状态和重试能力。
- 第三个坑，是没考虑幂等性，任务偶发重复执行时直接造成数据副作用。

这一章真正要建立的是一种任务心智：**不是所有容器都应该长期在线，有些工作负载的意义就在于“做完并留下结果”。**
