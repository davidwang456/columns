# 第 17 章：kube-proxy：看不见的转发规则，决定了 Service 是否可用

> 对应学习计划第 2 周 Service 负载均衡与 iptables/IPVS 理解。

很多人第一次理解 Service 时，会把注意力放在 `ClusterIP` 这个地址本身，好像集群里真的有一台神秘机器在监听这个 IP。可当你再往下追一步就会发现，真正让 Service 可访问的，并不是一个“神奇的服务进程”，而是各节点上被动态维护的一套转发规则。kube-proxy 的存在，就是为了把 Service 的抽象，变成节点上真实可执行的流量路径。

## 1）项目背景

上一章我们讲了 Service IP 是一种稳定抽象，但“抽象”不能自己工作，必须有某种实现机制把访问 ClusterIP 的流量导向真实 Pod。这个机制在很多集群里就是 kube-proxy。

kube-proxy 最重要的工作不是“监听端口”，而是：

- 观察 Service 和 Endpoints/EndpointSlice 的变化。
- 在节点上生成或更新转发规则。
- 让访问 Service IP 的请求被重定向到某个后端 Pod。

换句话说，它干的是“把声明式服务抽象翻译成内核层或网络层规则”的工作。

初学者为什么容易误解它？因为日常使用时，你只看到：

- Service 建好了。
- ClusterIP 能访问。
- 后端 Pod 扩容了也能自动分流。

整个转发过程在节点上悄悄完成，看不到也摸不着。于是很多人一遇到 Service 不通，就只盯应用日志或者 Service YAML，却忽略了节点规则这层实现机制。

kube-proxy 常见有两种主要工作模式：

- `iptables`
- `IPVS`

它们都能完成转发，但在规则组织方式、性能表现和大规模场景下的行为上会有所不同。你在学习早期不必一开始就钻得太深，但至少要明白：**Service 可达，不是因为 ClusterIP 自带魔法，而是因为节点规则在替你工作。**

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我有点想不通。Service 的 ClusterIP 到底是谁在响应？为什么它看起来像一个虚拟地址，却又真的能访问？

**大师**：因为它本来就是虚拟地址。真正完成请求转发的，不是某个业务 Pod，而是节点上的转发规则。

**小白**：那是谁负责把这些规则配出来的？

**大师**：通常就是 kube-proxy。它会监听 Service 和后端变化，然后在节点上维护 iptables 或 IPVS 规则。

**小白**：所以 kube-proxy 不是普通意义上的“代理进程”？

**大师**：很多时候不是你想象中那种应用层代理。它更多是在帮你编排和同步内核层的转发规则。

**小白**：那如果后端 Pod 变了，为什么 Service 还能自动跟上？

**大师**：因为 kube-proxy 会持续 watching 对象变化。后端一变，规则就会被更新。这就是声明式系统的典型体现。

**小白**：那 Service 不通的时候，我是不是不能只看应用日志？

**大师**：对。你至少还得看 Service、Endpoints，以及节点上的转发实现有没有问题。

**小白**：原来 ClusterIP 背后真正的关键是“节点规则同步是否正确”。

**大师**：这就是 kube-proxy 的价值所在。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们不去陷入太多内核细节，而是通过“对象变化”和“节点规则”的对应关系，理解 kube-proxy 的工作方式。

### 第一步：先确认 Service 和后端

```bash
kubectl get svc web
kubectl get endpoints web
```

这一步先确认两件事：

- Service 对象是否存在。
- 它当前有哪些后端实例。

因为 kube-proxy 的规则就是围绕这两个对象同步出来的。

### 第二步：观察后端变化如何影响转发基础

你可以先扩容或替换后端 Deployment，再重新看 Endpoints：

```bash
kubectl scale deployment web --replicas=5
kubectl get endpoints web -w
```

当后端 Pod 数量变化时，Endpoints 会更新。虽然你没直接去碰节点规则，但 kube-proxy 正是在背后响应这类变化。

### 第三步：在实验节点上查看规则

如果你的环境允许进入节点，可以查看规则，例如：

```bash
iptables -t nat -L -n
```

在使用 IPVS 的环境中，也可能会通过 `ipvsadm -ln` 观察虚拟服务和后端列表。你第一次看这些输出时不需要全部看懂，只要先建立一个意识：**Service 的可达性最终落在节点规则上，而不是停留在 YAML 上。**

### 第四步：理解从 Service 到 Pod 的中间层

可以把这条链路简化理解为：

1. 客户端请求 ClusterIP。
2. 节点根据 kube-proxy 维护的规则捕获流量。
3. 规则把流量导向某个后端 Pod。
4. 如果后端 Pod 在别的节点上，再经过跨节点网络转发过去。

有了这条链路，你在排障时就不会把所有问题都推给应用本身。

### 第五步：什么时候该想到 kube-proxy 这一层

以下场景尤其要想到它：

- Service 对象存在，但访问超时或转发异常。
- Pod 互通正常，但走 Service 就不正常。
- Endpoints 已更新，但请求仍像打到旧实例。
- 节点间、节点内表现不一致。

这时候就该意识到，问题可能不只在应用，而在 Service 抽象落地的规则层。

### 这一节应该带走什么

- Service 可访问并不是因为“有个神秘进程监听 ClusterIP”，而是因为节点规则在工作。
- kube-proxy 持续把 Service 和后端变化翻译为转发实现。
- Service 排障不能只看 YAML 和应用，也要看 Endpoints 与规则层。
- 理解 kube-proxy，等于把 Service 从“会用”推进到“懂实现”。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把 Service 的抽象真正落地为节点可执行的转发能力。
- 它与 Service/Endpoints 联动，能够自动适配后端实例变化。
- 理解它之后，Service 排障和性能分析都会更扎实。

### 缺点

- 对新手来说，它处在一个“看不见却很关键”的层级，理解门槛较高。
- 节点规则层的问题往往比应用日志更隐蔽，排查需要更多系统视角。
- 不同模式和不同环境的实现差异，会提高学习复杂度。

### 使用场景

- 理解 Service 流量转发和负载分发机制。
- 排查 Service 可达性问题和后端转发异常。
- 为后续学习 IPVS、eBPF、CNI 实现打基础。

### 注意事项

- 不要把 Service 的可达性完全归咎于应用本身。
- 排障时要同时看 Service、Endpoints 和节点规则层。
- 先理解 kube-proxy 的职责，再深入具体实现模式。

### 常见踩坑经验

- 最常见的坑，是一看到 Service 不通，就只查应用日志和容器状态，却不去看 Endpoints 和节点转发规则。
- 第二个坑，是把 ClusterIP 想成真实网卡地址，从而误解它的工作方式。
- 第三个坑，是只会“用 Service”，却完全不知道后端变化为什么能自动跟上，导致一出异常就无从下手。

这一章真正想完成的，是从抽象到实现的一次下潜：**当你知道 Service 背后是谁在干活，很多“为什么不通”的问题就不再神秘。**
