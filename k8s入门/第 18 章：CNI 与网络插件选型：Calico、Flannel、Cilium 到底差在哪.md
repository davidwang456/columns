# 第 18 章：CNI 与网络插件选型：Calico、Flannel、Cilium 到底差在哪

> 对应学习计划第 2 周 Day 10-11 的网络插件深入。

学完 Kubernetes 网络模型之后，很多人会顺势问一句：既然平台要求每个 Pod 一个 IP、跨节点还要互通，那这些能力到底是谁实现的？答案通常不是 Kubernetes 核心组件单独完成，而是依赖 CNI 网络插件。也正因为如此，网络插件从来不只是“装一下让集群能跑起来”的基础配件，它其实直接决定了你的网络策略能力、可观测性、性能特征和后续运维复杂度。

## 1）项目背景

Kubernetes 本身定义了网络模型，但并不亲自实现所有底层细节。它把“如何给 Pod 配网、如何打通跨节点网络、如何提供网络策略能力”这部分能力交给插件体系处理，这就是 CNI（Container Network Interface）存在的意义。

从学习视角看，CNI 至少要分两层理解：

- **CNI 规范**：约定容器运行时和网络插件之间如何交互。
- **具体插件实现**：例如 Calico、Flannel、Cilium，各自提供不同能力和取舍。

为什么这件事不能被简单理解成“装哪个都差不多”？因为不同插件的侧重点真的不一样：

- 有的更轻量，适合快速打通网络。
- 有的更强调 NetworkPolicy 和路由能力。
- 有的更强调 eBPF、性能和可观测性。
- 有的更适合小团队快速起步，有的更适合平台化深度治理。

这意味着，插件选型最终不是一个“抄教程”的问题，而是一个平台决策问题。你要考虑的不只是现在能不能起集群，还包括：

- 后面要不要做网络策略隔离。
- 集群规模会不会持续增长。
- 团队对网络内核和 eBPF 的掌握深度如何。
- 排障能力和维护成本是否能承受。

对初学者来说，第一步当然还是把一个插件装起来并观察它在集群中的表现；但更进一步，你应该开始建立“插件选型框架”，而不是只会背几个名字。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我看到网上教程有人装 Calico，有人装 Flannel，还有人推荐 Cilium。它们不都是让 Pod 互通吗？

**大师**：表面目标类似，但侧重点差很多。网络插件不是“能通就一样”，它会深刻影响你的策略能力、性能表现和后续治理方式。

**小白**：那我是不是应该选功能最全的那个？

**大师**：不一定。功能越多，往往也意味着理解和维护成本更高。选型不是堆参数，而是看团队场景和能力边界。

**小白**：能不能先给我一个简单理解？

**大师**：可以。Flannel 更像“先把网络打通”；Calico 在路由和网络策略上很成熟；Cilium 则把 eBPF、可观测性和高级网络能力带得更深入。

**小白**：那我作为初学者，是不是先把它们都装一遍？

**大师**：装一遍有帮助，但更重要的是观察：装完后，Pod 网络怎么变了、策略能不能生效、排障时你能看到什么。理解“实现差异”比记名字更重要。

**小白**：所以 CNI 学习重点不是安装命令，而是“谁在实现网络模型，以及实现方式有什么代价”。

**大师**：完全正确。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们以 Calico 为例，先体验“网络插件进入集群后，平台有哪些组件变化”，再补上选型视角。

### 第一步：安装一个典型插件

```bash
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.29.0/manifests/calico.yaml
kubectl get pods -n kube-system
```

第一次安装时，先不要急着看所有字段，而是重点观察：

- `kube-system` 里新增了哪些组件。
- 网络相关 Pod 是否成功 Running。
- 节点是否恢复到 Ready。

这一步会让你意识到：网络插件不是“某个抽象选项”，而是真正运行在集群里的系统组件。

### 第二步：观察 Pod 配网结果

插件安装并稳定后，再去看业务 Pod 的地址与通信表现：

```bash
kubectl get pods -o wide
kubectl exec -it netshoot -- ping <another-pod-ip>
```

你不必一开始就深入路由细节，但要知道：Pod 拿到 IP、跨节点互通、Service 背后的后端连通性，都是这类插件在背后提供支撑。

### 第三步：用“能力矩阵”建立选型视角

可以先用一个朴素的对比框架：

- Flannel：简单、轻量，适合先打通网络。
- Calico：网络策略成熟，很多团队的主流选择。
- Cilium：eBPF 能力强，可观测性和高级功能丰富。

请注意，这不是绝对优劣排序，而是帮助你建立“按场景看差异”的思路。

### 第四步：把选型问题具体化

在真实环境里，你至少该问：

- 我需不需要严格的 NetworkPolicy？
- 团队是否具备处理更复杂网络栈的能力？
- 集群规模和性能诉求高不高？
- 出问题时我希望看到什么级别的诊断信息？

只要这些问题一提出，插件选型就不再是“跟着教程装哪个”，而是“团队想走哪条平台路线”。

### 第五步：学习阶段怎么做最有效

学习阶段最推荐的方式是：

1. 至少成功装过一种插件。
2. 理解它如何让节点恢复 Ready、Pod 获得 IP。
3. 再对比另外两种插件的核心差异与适用场景。

这样既不会陷入安装细节泥潭，也能建立真正有用的选型视角。

### 这一节应该带走什么

- CNI 是网络模型落地的重要插件接口，不是边缘组件。
- 不同插件差异不仅在“能不能通”，更在策略、性能和维护成本。
- 插件选型是平台决策，不是教程复制。
- 学习插件要同时看“安装结果”“系统行为”和“团队适配度”。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把 Kubernetes 网络模型真正落到了可运行实现上。
- 它让网络策略、跨节点通信和高级网络能力成为可能。
- 通过比较不同插件，你会逐渐建立平台选型能力，而不是停留在“装通就行”。

### 缺点

- 学习门槛高，尤其涉及内核网络、路由和 eBPF 时复杂度明显上升。
- 不同插件差异大，迁移和替换成本不低。
- 一旦插件本身异常，集群很多看似“不相关”的网络故障都会同时出现。

### 使用场景

- 集群基础网络能力构建。
- 需要 NetworkPolicy、可观测性和高级流量能力的场景。
- 平台化团队在网络层做深入治理与选型时。

### 注意事项

- 不要只看教程热度选插件，要看团队能力和未来治理需求。
- 学习时先理解“模型与职责”，再研究具体安装细节。
- 出问题时要意识到，网络插件异常会影响整个集群的基础连通性。

### 常见踩坑经验

- 最常见的坑，是只看“别人都用哪个”，不看自己的团队场景和维护能力。
- 第二个坑，是把网络插件当成一次性安装项，后续完全不理解它在集群中做了什么。
- 第三个坑，是遇到网络问题只查业务服务，不去看插件组件和节点网络状态。

这一章真正想帮你建立的，是网络选型框架：**网络插件不是配角，而是 Kubernetes 网络能力的真正落地点。**
