# 第 01 章：云原生与 Kubernetes：为什么不是多装几台虚拟机

> 对应学习计划第 1 周 Day 1-2 的架构理解部分。

这一章不是为了急着教你写 YAML，而是先回答一个更重要的问题：**为什么今天的后端开发、测试交付和基础设施协作，越来越绕不开 Kubernetes**。如果这个问题没有想清楚，后面你学到的 Pod、Deployment、Service、Ingress 都会变成散落的知识点；如果这个问题想清楚了，后面的对象模型几乎都会顺理成章。

## 1）项目背景

很多开发者第一次接触 Kubernetes 时，心里都会有一个很真实的疑问：我们已经有 Docker 了，也有虚拟机、Jenkins、Nginx、脚本发布，甚至团队里已经有一套不算差的部署流程，为什么还要再引入一个看起来更复杂、名词更多、学习成本更高的平台？

这个疑问非常正常，而且必须正面回答。因为 Kubernetes 从来不是为了“把容器跑起来”而生的。单机运行容器，Docker 已经足够优秀；多台机器上手工拉起若干容器，配合脚本和反向代理，也不是做不到。真正让团队开始痛苦的，是**规模、协作、变更频率和环境复杂度**同时上升之后，原有做法开始变得不可预测。

想象一个典型团队的成长过程。最开始，系统只有一个后端服务、一个数据库、一个前端静态站点，部署靠 SSH 上机器跑脚本就够了。后来服务越来越多，环境从开发、测试、预发到生产分出好几套；每次上线都要先拉镜像、改配置、重启进程、切流量、看日志、回滚；再后来，又出现了弹性扩缩容、灰度发布、跨机器调度、故障自愈、权限隔离、配置统一管理等需求。这个阶段，你会发现问题已经不再是“能不能部署成功”，而是：

- 这次发布是不是可重复的。
- 副本数是不是一致的。
- 某台机器挂了之后，系统会不会自己恢复。
- 团队成员改动配置时，是否可追踪、可审计、可回滚。
- 新服务接入平台时，是否要重复发明一套运维流程。

Kubernetes 真正解决的，就是这类问题。它把过去分散在脚本、人工经验、运维口口相传中的流程，抽象为一套**声明式对象 + 持续控制循环**。开发者不再说“去某台机器执行某条命令”，而是说“我希望系统最终呈现为这个状态”；控制平面则不断把现实状态拉回这个期望状态。

这就是它和传统脚本式运维最大的区别。脚本的思路是一次性动作，Kubernetes 的思路是持续维护。脚本擅长“执行”，Kubernetes 擅长“维持”。脚本告诉机器“做什么”，Kubernetes 告诉平台“最后应该是什么样”。

你也可以把它理解成一次职责上移。过去，很多“把系统保持在线”的责任，落在开发、运维、测试和值班同学头上；有了 Kubernetes 之后，平台开始接手这部分基础工作，比如副本自动补齐、节点故障迁移、服务发现、滚动更新和配置注入。于是团队可以把精力从“重复劳动”转向“业务设计”和“平台治理”。

当然，这并不意味着 Kubernetes 适合一切场景。一个单体内部系统，半年上线一次，只有一台服务器，团队也只有两三个人，这种情况下你完全可以不用 Kubernetes。真正适合它的，是以下几类团队：

- 服务数量持续增长，人工维护已经开始吃力。
- 环境数量较多，测试和生产之间存在明显差异。
- 发布频率较高，需要标准化回滚、灰度和扩缩容。
- 多团队共享基础设施，需要统一权限、资源和网络治理。
- 希望把交付过程纳入 Git、审计和自动化流水线。

所以，本章真正想解决的，不是“你会不会安装 Kubernetes”，而是帮你建立一个判断框架：**什么时候它值得引入，为什么值得引入，引入之后团队到底获得了什么。** 这套框架会成为你后续理解所有 Kubernetes 对象的出发点。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我看网上很多文章都说 Kubernetes 是容器编排平台。可我已经会 Docker 了，为什么还得学一个更重的东西？

**大师**：因为 Docker 解决的是“怎么跑一个容器”，而 Kubernetes 解决的是“怎么持续管理一群容器，以及这些容器背后的发布、调度、网络、配置和故障恢复”。

**小白**：可我现在也能用脚本批量拉起容器，甚至配上 Nginx 转发，看起来也能用。

**大师**：能用，不代表能持续维护。你现在脚本里写的是动作：拉镜像、启动、改端口、重启。等机器多了、服务多了、团队多了，这些动作会迅速膨胀，最后没人说得清某个服务“现在应该是什么状态”。

**小白**：那 Kubernetes 是怎么解决这个问题的？

**大师**：它把“动作”换成“状态”。你不再说“帮我在 3 台机器上各起一个 nginx”，而是说“我希望系统里有 3 个 nginx 副本，并且对外提供稳定访问入口”。至于其中一个 Pod 挂了、某个节点不可用、某次升级要不要回滚，这些都由控制平面持续调节。

**小白**：听起来像一个超级自动化系统。

**大师**：可以这么理解，但更准确地说，它是一个有强约束的声明式系统。自动化脚本只是帮你执行任务，Kubernetes 会持续比较“现在”和“期望”之间的差异，然后想办法把它们重新拉齐。

**小白**：所以它最大的价值不是让容器跑起来，而是让系统以一种统一方式被管理？

**大师**：对。你可以把 Kubernetes 看成一个“操作系统级的平台接口”。开发者向它提交工作负载，平台负责调度、恢复、暴露、配置和治理。于是不同团队虽然写的是不同语言、不同框架，但交付方式可以统一。

**小白**：那是不是只要上了 Kubernetes，所有问题都解决了？

**大师**：恰恰相反。它解决的是一类问题，不是所有问题。它不会自动帮你写出高质量业务代码，也不会替你做架构设计。它只是把很多重复、脆弱、难以协作的基础工作抽象成平台能力。你仍然要做资源规划、服务拆分、监控告警、安全治理这些事，只是做法变得标准化了。

**小白**：那我应该怎么学，才不会只记一堆概念？

**大师**：每学一个对象，都问自己四件事：它在解决什么痛点；没有它会怎样；它如何和其他对象配合；它最常见的误用是什么。沿着这四个问题学，你就会越来越像在“设计系统”，而不是“背命令”。

## 3）项目实战：通过主代码片段使用该主题的对象

这一章的实战不追求把一个复杂系统完整部署出来，而是先让你**亲身感受“命令式管理”和“声明式维护”之间的差异**。我们用一个最小的 Web 服务来对比两种思路。

### 第一步：用 Docker 命令式地启动服务

```bash
# 先感受命令式方式
docker run -d --name web -p 8080:80 nginx
```

这条命令的特点是：**它描述的是一个动作**。执行成功后，容器会起来；但如果容器进程异常退出，你需要自己检查并处理。如果要扩成 3 个副本，你要继续自己命名、自己分配端口、自己做负载均衡。如果机器挂了，这条命令并不会帮你在另一台机器上恢复服务。

### 第二步：切换到 Kubernetes 的声明式视角

```bash
kubectl create deployment web --image=nginx
kubectl scale deployment web --replicas=3
kubectl get deployment web
kubectl get pods -l app=web
```

这里最关键的变化，不是命令写法不同，而是你提交给平台的含义变了。`Deployment` 代表的是“我希望始终有这些副本存在”。于是当某个 Pod 被删掉时，控制器会自动补回来。你可以手动试一次：

```bash
kubectl delete pod -l app=web
kubectl get pods -w
```

你会看到旧 Pod 被删掉之后，新 Pod 会重新创建。这个现象极具代表性，因为它揭示了 Kubernetes 的本质：**对象不是为了记录一次历史动作，而是为了持续约束未来状态。**

### 第三步：观察声明式系统的几个核心信号

```bash
kubectl describe deployment web
kubectl describe pod <pod-name>
kubectl logs <pod-name>
```

建议你第一次练习时，不要急着读所有字段，而是只盯住下面几类信息：

- Deployment 想要几个副本。
- 当前实际有几个可用副本。
- Pod 是否成功被调度到某个节点。
- 镜像有没有拉取成功。
- 事件里是否出现失败和重试。

你会发现，Kubernetes 的学习非常像“读系统状态”。同样是一次部署，过去我们更关注脚本有没有执行完；现在则更关注控制器是否把对象推进到了目标状态。

### 第四步：做一个最小故障实验

把镜像改成一个不存在的版本：

```bash
kubectl set image deployment/web nginx=nginx:not-exist
kubectl rollout status deployment/web
kubectl describe pod -l app=web
```

这时你很可能会看到 `ImagePullBackOff` 或 `ErrImagePull`。这一步非常重要，因为它会让你意识到：Kubernetes 学习不能只练“成功路径”，更要练“失败路径”。失败时，你不是去猜，而是沿着对象状态、事件、日志和控制器行为往下查。

### 这一节应该带走什么

- Docker 更像容器运行时入口，Kubernetes 更像持续维持目标状态的平台。
- 命令式告诉系统“做一次”，声明式告诉系统“保持成这样”。
- 控制器是 Kubernetes 真正的灵魂，它让对象拥有“持续生效”的能力。
- 学会观察状态与事件，是后续所有章节的基础能力。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它帮开发者建立了正确的学习起点，不再把 Kubernetes 误解为“更复杂的容器命令集合”。
- 它把发布、恢复、扩缩容、服务发现等重复劳动抽象为平台能力，适合长期演进。
- 它天然适合与 Git、CI/CD、权限管理、资源治理结合，形成规范化交付流程。

### 缺点

- 学习曲线比单机容器工具陡很多，新手容易被概念密度吓退。
- 引入平台之后，团队需要接受更多约束，例如声明式配置、统一规范、可观测性要求。
- 如果团队规模和业务复杂度还很低，Kubernetes 可能会带来不必要的工程成本。

### 使用场景

- 微服务逐步变多，需要统一交付和治理方式的团队。
- 发布频繁，希望平台自动处理副本管理、恢复与滚动更新的系统。
- 多环境、多团队共享基础设施，需要标准化资源、权限和网络边界的组织。

### 注意事项

- 不要把“是否使用 Kubernetes”变成信仰问题，要看团队规模、交付频率和治理需求。
- 不要跳过基础概念直接背命令，否则后面会越来越碎片化。
- 学习初期先抓主线：工作负载、服务发现、配置管理、调度和排障，再逐步深入安全和可观测性。

### 常见踩坑经验

- 最大的坑是把 Kubernetes 只当成“更复杂的 docker run”。一旦这么理解，后面所有对象都会学得很别扭。
- 第二个坑是过早追求生产级复杂架构，结果环境装了三天，Pod 还没创建过一个。
- 第三个坑是只练成功路径，不练失败路径，导致一旦遇到 `CrashLoopBackOff`、调度失败、镜像拉取失败就束手无策。

本章的真正价值，不是让你背下“云原生”“编排”“声明式”这些词，而是完成一次思维切换：**以后再看 Kubernetes 里的任何对象，你都先问它在维护什么状态、解决什么协作问题、替团队承担了什么重复劳动。**
