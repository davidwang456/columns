# 第 25 章：污点与容忍：不是所有节点，都欢迎所有 Pod

> 对应学习计划第 3 周污点、容忍度与节点管理。

如果说上一章的节点标签和亲和性，是从 Pod 的角度表达“我想去哪里”，那么这一章的污点和容忍，就是从节点的角度表达“我不想让谁随便来”。这两种机制一正一反，合在一起，才组成了完整的调度治理手段。污点与容忍之所以重要，是因为很多集群治理需求，本质上都来自节点侧的约束，而不是业务侧的偏好。

## 1）项目背景

为什么节点侧约束会成为一个独立主题？因为真实集群里，经常会有一些节点不适合承接普通工作负载：

- GPU 节点成本高，只想让特定任务进去。
- 某些节点正处于维护窗口，不希望新的业务上来。
- 专用中间件节点只服务特定系统。
- 控制平面节点默认就不希望普通 Pod 随意调度进去。

如果没有节点侧限制，只靠 Pod 自己说“我想去哪”，就会出现很多问题：

- 普通业务占用稀缺专用节点。
- 维护中的节点仍不断接收新调度。
- 平台团队无法从节点角度建立准入边界。

Kubernetes 的解决方案就是：

- 节点打上 `taint`（污点），表示“我这里有条件，不是谁都能来”。
- Pod 通过 `tolerations` 声明“我可以容忍这些限制，因此允许进来”。

这里最重要的心智是：**污点不是“指定谁必须来”，而是“拒绝默认流量”；容忍不是“强制调度”，而是“获得资格”。**

这点如果没理解透，后面会频繁误用。很多人以为“加了容忍就一定会调度到这台节点”，其实不是；它只是让 Pod 不再因为这个污点被拒绝，至于是否真的去那，还要看其他调度条件。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我有一批 GPU 节点不想让普通服务乱用，是不是只能靠大家自觉？

**大师**：如果靠自觉，早晚会出事故。平台治理不能建立在口头承诺上。

**小白**：那是不是给 Pod 配亲和性，让 GPU 任务去 GPU 节点就行？

**大师**：还不够。亲和性是从 Pod 侧表达“我想去哪”；但你现在的问题是从节点侧表达“我不欢迎谁”。这正是污点的职责。

**小白**：所以污点更像节点自己挂出的“谢绝入内”牌子？

**大师**：非常贴切。默认情况下，看到这个牌子的 Pod 就别进来；只有显式声明了容忍的 Pod，才有资格继续考虑调度。

**小白**：那容忍是不是等于“保证一定调度进去”？

**大师**：不是。容忍只是拿到了入场券，不代表一定选中这台节点。真正能不能调度成功，还要看资源、亲和性、选择器等其他因素。

**小白**：原来污点和容忍是在做“准入控制”，不是在做“精确落点”。

**大师**：对，这个区别一定要记住。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先给节点打污点，再让一个 Pod 声明容忍度，观察它从“被拒绝”到“有资格调度”的过程。

### 第一步：给节点添加污点

```bash
kubectl taint nodes worker-1 dedicated=gpu:NoSchedule
```

这条命令表达的是：`worker-1` 节点带有一个 `dedicated=gpu` 的污点，效果是 `NoSchedule`。默认情况下，不容忍这个污点的 Pod 将不会被调度到该节点。

### 第二步：观察未容忍 Pod 的行为

如果你创建一个没有容忍度的普通 Pod，并且它本来可能被调度到这台节点，就可能在事件里看到与污点相关的调度失败信息。这一步很有教学意义，因为它能让你直观看到：**污点是在节点侧主动建立门槛。**

### 第三步：给 Pod 增加容忍度

```yaml
tolerations:
- key: dedicated
  operator: Equal
  value: gpu
  effect: NoSchedule
```

有了这段配置，Pod 的意思就变成了：“我可以容忍 `dedicated=gpu:NoSchedule` 这个限制，因此不再因为这个污点被排除。”

### 第四步：理解几种 effect 的差别

学习阶段至少要先建立三个效果的直觉：

- `NoSchedule`：不允许新的不容忍 Pod 调度进来。
- `PreferNoSchedule`：尽量别来，但不是绝对禁止。
- `NoExecute`：不仅不让新 Pod 来，还可能把现有不容忍的 Pod 驱逐出去。

这三个效果很像“强拒绝”“软拒绝”“拒绝并清场”的层层递进。只要这个心智建立了，后面再看复杂场景就不会太混乱。

### 第五步：把它放回平台治理视角

污点与容忍最常出现的场景通常包括：

- GPU / 高性能 / 稀缺专用节点保护。
- 维护窗口中的节点隔离。
- 控制平面节点限制普通业务进入。
- DaemonSet 和系统组件容忍某些专用节点污点。

你会发现，它几乎都是在做“节点侧的默认拒绝”。这正是它和上一章亲和性最大的不同。

### 这一节应该带走什么

- 污点是节点侧的准入限制，不是 Pod 侧偏好表达。
- 容忍代表“有资格不被这个污点拒绝”，不代表“一定调度过去”。
- `NoSchedule`、`PreferNoSchedule`、`NoExecute` 分别代表不同强度的限制。
- 污点与容忍非常适合做节点池治理和稀缺资源保护。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让平台团队可以从节点侧建立清晰的准入边界。
- 它非常适合保护稀缺资源节点和特殊用途节点池。
- 它与亲和性、选择器一起使用时，能形成更完整的调度治理方案。

### 缺点

- 初学者很容易把“容忍”误解成“强制调度”。
- 一旦污点策略设计混乱，排障复杂度会明显上升。
- 如果不给系统组件和关键工作负载正确补容忍，平台本身也可能被影响。

### 使用场景

- GPU 节点、专用节点池、维护节点隔离。
- 控制平面与普通工作负载分层治理。
- 需要从节点侧限制默认调度进入的场景。

### 注意事项

- 先明确目标是“节点侧限制”，而不是“指定落点”。
- 加污点后，要同步检查系统组件和关键工作负载是否需要容忍。
- 不要只写容忍而不看其他调度条件，否则容易误判为什么没调度上去。

### 常见踩坑经验

- 最常见的坑，是给节点加污点后忘记给系统组件补容忍，结果平台本身先出问题。
- 第二个坑，是把容忍理解成“有了就一定调度进去”，排障方向完全错位。
- 第三个坑，是污点加得越来越多，却没有统一治理规则，最终谁都说不清为什么 Pod 调度不上。

这一章真正想帮你建立的，是节点治理视角：**不是所有节点都默认欢迎所有工作负载，平台必须有办法从节点侧先把门守住。**
