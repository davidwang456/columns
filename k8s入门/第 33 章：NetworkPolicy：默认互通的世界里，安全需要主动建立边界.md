# 第 33 章：NetworkPolicy：默认互通的世界里，安全需要主动建立边界

> 对应学习计划第 5 周 NetworkPolicy 与网络隔离。

很多开发者第一次把一组服务部署进 Kubernetes 后，默认都会有一个危险的乐观假设：既然已经做了命名空间划分、Service 拆分和 RBAC 管理，网络上应该也差不多隔开了吧。现实往往并非如此。在很多集群里，**Pod 之间默认是能互相访问的。** 从安全视角看，这其实意味着：只要你不主动建立边界，默认世界就是互通的。NetworkPolicy 存在的意义，就是把“谁能访问谁”这件事，从模糊共识变成平台可执行的网络规则。

## 1）项目背景

为什么 NetworkPolicy 这么重要？因为安全问题里有一个非常朴素但经常被忽视的原则：**最小暴露。** 一个后端服务应该只接受它真正需要的来源访问，而不是“集群里谁都能试一下”。

如果没有网络策略，典型风险包括：

- 某个被入侵的 Pod 横向扫描整个集群内部服务。
- 本来只该被前端访问的后端，被其他无关工作负载直接连通。
- 不同团队、不同命名空间之间默认互通，隔离形同虚设。
- 故障排查时，网络流向完全不透明。

NetworkPolicy 的核心思想很简单：

- 用标签和命名空间来描述允许的来源和去向。
- 按 Ingress / Egress 两个方向限制流量。
- 让默认互通的世界，逐步变成“只允许需要的连接存在”。

它的价值不只是安全，还包括清晰的系统边界。因为一旦你开始写网络策略，你会被迫面对一个问题：**哪些调用是系统真实依赖，哪些调用只是历史偶然和无意识互通。** 这对平台治理非常有价值。

不过要注意，NetworkPolicy 并不是“写了对象就一定生效”的纯抽象能力，它依赖底层网络插件支持。也就是说，网络模型、CNI 能力和策略对象之间是强耦合的，这也是为什么本章放在网络和安全交汇位置上。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我已经把服务分了 Namespace，还做了 RBAC，应该已经挺安全了吧？

**大师**：还不够。Namespace 和 RBAC 解决的是组织和权限，不等于网络访问自动隔离。

**小白**：也就是说，不同命名空间的 Pod 默认也可能互通？

**大师**：是的，这正是很多人第一次接触 NetworkPolicy 时最震惊的地方。

**小白**：那 NetworkPolicy 的作用是不是有点像集群内部防火墙？

**大师**：这个比喻很贴切。你可以把它理解成面向 Pod 标签和命名空间的微分段规则。

**小白**：那是不是上来就应该先全拒绝？

**大师**：原则上“默认拒绝，再按需放行”是更安全的思路。但前提是你得知道系统真正依赖哪些流量，否则很容易把 DNS、健康检查和关键依赖一起拦死。

**小白**：所以 NetworkPolicy 不只是安全配置，也是系统依赖关系的显式化过程。

**大师**：这就是它最值钱的地方之一。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先做一个最经典的策略：在某个命名空间里下发默认拒绝，然后只放行前端到后端的必要流量。

### 第一步：建立默认拒绝策略

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
```

这段配置的关键意义在于：

- `podSelector: {}` 表示作用于该命名空间中所有 Pod。
- 同时声明 Ingress 和 Egress，意味着默认入站和出站都需要显式放行。

这是很多团队建立零信任式内部网络的第一步。

### 第二步：有默认拒绝，不等于系统能工作

一旦下发这类策略，你要立刻想到一些基础依赖：

- DNS 查询是否还通。
- 前端到后端是否还通。
- 后端到数据库是否还通。
- 应用探针、指标抓取、日志链路是否受影响。

这就是为什么 NetworkPolicy 学习不能停在“会写 YAML”，而要结合真实依赖链观察。

### 第三步：按需放行必要访问

比如，只允许带 `role=frontend` 标签的 Pod 访问带 `role=backend` 的 Pod：

```yaml
ingress:
- from:
  - podSelector:
      matchLabels:
        role: frontend
```

这一步的真正价值不在于字段本身，而在于你开始把“谁应该访问谁”正式写成平台规则，而不是留在脑子里。

### 第四步：记住 DNS 往往是第一类被误伤的依赖

这是 NetworkPolicy 最高频的生产踩坑之一。很多团队一上来就做默认拒绝，结果域名解析全挂，因为 DNS 请求同样属于网络流量，也需要显式考虑。这个细节特别适合在专栏里反复强调，因为它极具实战价值。

### 第五步：从“规则”走向“边界建模”

写到最后，你会发现 NetworkPolicy 最重要的不是对象语法，而是系统建模：

- 哪些服务本来就不该被横向访问。
- 哪些依赖必须明确存在。
- 哪些流量是平台层默认应该放行的。

这时它已经不只是安全工具，而是在逼团队把系统边界画清楚。

### 这一节应该带走什么

- 默认互通不等于默认安全，边界必须主动建立。
- NetworkPolicy 通过标签和命名空间表达允许的网络关系。
- “默认拒绝，再按需放行”是更成熟的安全治理思路。
- DNS 和基础依赖往往是策略上线时最容易被误伤的部分。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把网络隔离从口头规范落成了平台可执行能力。
- 它有助于建立最小暴露和微分段式内部安全边界。
- 它会倒逼团队把系统调用关系显式化，提升整体治理清晰度。

### 缺点

- 一旦依赖关系没梳理清楚，策略上线很容易误伤业务。
- 不同 CNI 对 NetworkPolicy 支持能力不同，不能脱离插件谈生效。
- 随着策略增多，规则管理和排障复杂度也会快速上升。

### 使用场景

- 多团队共享集群的命名空间级网络隔离。
- 前后端、中间件、数据库之间的最小暴露控制。
- 平台需要加强东西向流量治理时。

### 注意事项

- 默认拒绝策略上线前，先确认依赖链和必要放行项。
- 一定要明确底层 CNI 是否支持并正确实现 NetworkPolicy。
- 网络策略和 RBAC、Namespace 是互补关系，不是相互替代。

### 常见踩坑经验

- 最常见的坑，是下发默认拒绝后忘记放行 DNS 和必要依赖，导致系统整体雪崩。
- 第二个坑，是写了策略就以为一定生效，却忽略底层网络插件是否真正支持。
- 第三个坑，是只写少量策略满足眼前需求，却没有长期维护网络边界模型的意识。

这一章真正想帮你建立的，是内部安全边界视角：**在默认互通的世界里，安全从来不是默认给你的，而是你主动建立出来的。**
