# 第 02 章：实验集群搭建：minikube、kind、k3s 到底怎么选

> 对应学习计划第 1 周 Day 1-2 的环境搭建任务。

学习 Kubernetes 最容易出现的第一个挫败感，不是看不懂 Pod，也不是记不住命令，而是环境迟迟搭不起来。于是很多人刚开始学，就把大量精力耗在驱动、镜像、网络和系统兼容性上，最后还没真正进入 Kubernetes 的对象世界，就已经失去耐心了。这一章的目标，就是帮你用“工程化选择”的方式，选出一个**足够稳定、足够轻量、足够适合学习**的本地实验环境。

## 1）项目背景

对初学者来说，本地集群环境既是起点，也是分水岭。环境选得好，后面的学习会很顺；环境选得太重、太杂、太追求一步到位，往往会让学习过程被安装细节拖垮。

当前最常见的几类学习环境大致如下：

- `minikube`：偏“单机完整体验”，插件多，适合快速体验完整功能。
- `kind`：本质上是“跑在 Docker 容器里的 Kubernetes 节点”，很适合本地多节点实验和 CI 场景。
- `k3s`：轻量级发行版，资源占用小，适合边缘环境和轻量实验。
- 云厂商托管集群：最接近生产，但学习早期成本高、依赖多、也更容易被账单和权限问题打断。

如果你的目标是**从零开始理解对象模型、调度、网络和服务发现**，那么本地环境的优先级通常应是：稳定、便于销毁重建、能快速观察现象，而不是“无限接近生产”。很多人一开始就想搭一个三控制平面、多 worker、带 LB、带存储插件、带监控栈的复杂实验室，结果花了几天还在处理宿主机依赖问题。对学习来说，这种做法得不偿失。

为什么本地环境会这么关键？因为 Kubernetes 的很多知识，单靠读文字是很难真正理解的。比如：

- `kubectl get pods -o wide` 里 Pod IP、节点 IP 的差异。
- 一个 Service 创建后，流量为什么能分发到多个 Pod。
- Pod 删除后为什么会被自动补回来。
- 当节点资源不足时，调度失败的事件长什么样。

这些现象都需要一个你能反复折腾、随时销毁重建的实验环境来支撑。

因此，本章最重要的结论其实很简单：**学习环境不是越像生产越好，而是越适合验证概念越好。** 在这个前提下，`kind` 往往是非常合适的第一站，因为它启动快、重建快、多节点配置清晰，而且与后面网络、调度、Service 等实验兼容性很好。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我只是想学 Kubernetes，结果一上来就在选择 minikube、kind、k3s，我已经有点乱了。

**大师**：说明你已经遇到初学者的第一个真实问题了。技术学习不是只学概念，还要先选一块合适的练兵场。

**小白**：那我是不是应该一步到位，直接装一个最像生产的集群？

**大师**：不建议。学习环境的目标是“快速验证”，不是“完美复刻生产”。如果你把 80% 的时间花在环境搭建上，后面对象模型反而学不进去。

**小白**：那这几个工具分别适合什么场景？

**大师**：`minikube` 适合快速体验单机完整功能；`kind` 适合本地多节点实验和 CI；`k3s` 适合轻量环境或边缘场景。你现在要学的是 Kubernetes 本身，而不是安装发行版，所以优先选最不折腾、最容易重建的。

**小白**：听起来 `kind` 比较适合我。

**大师**：对，尤其是你后面还要学多节点调度、Service、网络插件这些内容，`kind` 的上手体验很好。你要记住一条原则：学习集群最好“坏了就能重建”，这样你才敢大胆实验。

**小白**：那我还需要一开始就装各种插件、监控、存储吗？

**大师**：不需要。第一阶段只做最小可用集群，确认节点正常、kubectl 正常、能创建 Pod。等你学到对应章节，再按主题加网络、存储、监控组件。这样你的学习路径才是分层推进，而不是一口吃成胖子。

**小白**：原来环境搭建本身，也是一种架构取舍。

**大师**：没错。越是初学者，越应该刻意避免“环境过度设计”。

## 3）项目实战：通过主代码片段使用该主题的对象

这一章我们用 `kind` 来创建一个两工作节点的本地集群。原因很简单：后面你会频繁做调度、Service、跨节点通信等实验，多节点环境能提前给你更多真实感。

### 第一步：准备 `kind` 集群配置

先创建一个 `kind-config.yaml`：

```yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
name: column
nodes:
- role: control-plane
- role: worker
- role: worker
```

这份配置刻意保持简单。它只表达一件事：我们要一个 1 控制平面、2 工作节点的学习型集群。现在你先不要急着去加端口映射、镜像仓库代理、额外挂载和各种高级配置，先让集群成功跑起来。

### 第二步：创建集群并验证上下文

```bash
kind create cluster --name column --config kind-config.yaml
kubectl cluster-info --context kind-column
kubectl get nodes -o wide
kubectl get all -A
```

如果一切正常，你应该能看到三个节点，并且 `kube-system` 命名空间下已经有 CoreDNS、kube-proxy 等基础组件。

建议你第一次搭建成功之后，立刻做三件事：

1. 看节点数量和角色是否符合预期。
2. 看 `kube-system` 组件是否都处于 Running。
3. 记住当前上下文名字，避免后续命令打到错误集群。

### 第三步：创建一个最小 Pod 验证集群可用

```bash
kubectl run demo --image=nginx:1.27 --restart=Never
kubectl get pod demo -o wide
kubectl describe pod demo
```

这一小步非常关键，因为它能帮你分清“集群已经创建成功”和“集群已经具备基本调度能力”不是一回事。只有 Pod 成功从创建到 Running，这个环境才算真正可以进入后续学习。

### 第四步：练习销毁与重建

```bash
kind delete cluster --name column
kind create cluster --name column --config kind-config.yaml
```

很多人学环境搭建时，只追求“搭起来一次”，却不敢重建。实际上，对学习来说，**可重建性比一次成功更重要**。你以后做网络、存储和权限实验时，随时可能把环境折腾坏。一个“敢删敢重建”的环境，才是合格的学习环境。

### 环境选择建议

如果你现在还没有确定到底选哪个工具，可以按下面的思路判断：

- 想要快速体验，单机即可：先选 `minikube`。
- 想练多节点、调度和网络：优先 `kind`。
- 机器资源紧张，或想体验轻量发行版：试 `k3s`。
- 想学云厂商托管产品：等基础概念稳了再上云，不要一开始就把学习复杂度拉满。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- `kind`、`minikube`、`k3s` 都能帮助你以较低成本建立本地实验场，适合边学边做。
- 本地环境可重建、可反复试错，非常适合练习对象状态观察和排障。
- 多节点本地集群能让后续的调度、Service、网络实验更接近真实场景。

### 缺点

- 与生产环境仍有差异，某些网络、存储、LB 现象无法完全复刻。
- 宿主机环境差异较大，Windows、macOS、Linux 的体验不完全相同。
- 如果一开始追求过度复杂配置，环境本身会反过来阻碍学习。

### 使用场景

- 入门学习 Kubernetes 的对象模型和工作流。
- 本地验证 YAML、脚本和交付过程。
- 做调度、网络、资源限制等可重复实验。

### 注意事项

- 学习初期优先使用最小配置，确认基础能力可用后再逐步扩展。
- 养成“坏了就重建”的心态，不要在一个脏环境里不断叠补丁。
- 记录好当前 `kubectl` 上下文，避免后面误操作其他集群。

### 常见踩坑经验

- 最常见的坑，是把学习环境做得像生产环境一样复杂，结果环境成了主角，学习反而变成配角。
- 第二个坑，是集群创建成功就以为万事大吉，没有继续验证 Pod 调度、DNS、节点状态。
- 第三个坑，是每次环境坏了都试图“修到完美”，却不敢直接删掉重建。对学习来说，这种心态非常拖进度。

本章你真正应该带走的，是一条非常务实的原则：**为学习选环境，不是选最强，而是选最稳、最省心、最容易验证概念。**
