# 第 32 章：准入控制与 Webhook：请求进入集群前，先过一遍安检

> 对应学习计划第 4 周准入控制器与 Webhook。

Kubernetes 很强大的一点，是它允许很多团队、很多系统、很多自动化工具同时往集群里提交资源。但这也意味着一个现实问题：**如果所有对象都能毫无约束地进来，平台迟早会被各种不合规配置、临时例外和低质量资源慢慢污染。** 准入控制与 Webhook 的意义，就是把“进入集群前的最后一道关”交给平台，而不是留给人肉 review 和口头规范。

## 1）项目背景

很多团队的治理能力之所以始终做不稳，问题不在于没有规范，而在于规范停留在文档里。比如：

- 镜像不能用 `latest`。
- 所有容器都必须带 requests/limits。
- 生产命名空间禁止特权容器。
- 某类资源必须自动补默认标签或注解。

这些要求如果只靠代码评审或自觉遵守，很快就会被现实打败。因为总会出现：

- 临时加急改动。
- 多团队风格不一致。
- 自动化脚本和人工操作混用。
- 老项目历史包袱太多。

Kubernetes 的准入链正是为此设计的。一个请求在真正写入 etcd 之前，通常还会经过：

- 认证：你是谁。
- 授权：你能不能做这件事。
- 准入：这件事本身合不合规，是否需要被修改或拒绝。

其中，Admission Webhook 最值得平台团队重视，因为它让你能把治理逻辑程序化。大致分两类：

- `Mutating`：请求进来后，先帮你改一改。
- `Validating`：请求进来后，检查不合规就拒绝。

所以，Webhook 的核心价值不是“又多一个 YAML”，而是：**把平台规则从“人要记住”升级成“集群会执行”。**

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我们团队明明约定了不准用 `latest` 镜像，为什么线上还是有人把它发进去了？

**大师**：因为约定不是执行。只要规则还停留在文档里，它迟早会被绕过去。

**小白**：那 Admission Webhook 就是帮我把规则强制化？

**大师**：对。你可以把它理解成 API Server 门口的安检机。资源请求要进集群，先过这一关。

**小白**：Mutating 和 Validating 的区别是什么？

**大师**：Mutating 像“帮你补齐表格”，比如自动加标签；Validating 像“安检发现违规就拦下”，比如拒绝特权容器或 `latest` 镜像。

**小白**：那是不是只要有 RBAC 就够了？

**大师**：不够。RBAC 解决“谁能做”，准入控制解决“你做的这件事本身是否合规”。这两者不是一层问题。

**小白**：原来平台治理真正成熟，靠的是把规范变成入口规则。

**大师**：这就是准入控制最值钱的地方。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们用一个最小 `ValidatingWebhookConfiguration` 结构来理解“如何把危险配置挡在门外”。

### 第一步：先建立一个治理目标

假设我们想做一条很典型的规则：**拒绝使用 `latest` 标签的镜像。** 这类规则特别适合当教学例子，因为：

- 它贴近真实平台规范。
- 它很好理解为什么要拦。
- 它很容易看出“入口治理”的价值。

### 第二步：看一个最小 Webhook 配置骨架

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: deny-latest-tag
webhooks:
- name: deny-latest.demo.io
  admissionReviewVersions: ["v1"]
  sideEffects: None
  failurePolicy: Fail
  rules:
  - apiGroups: ["apps"]
    apiVersions: ["v1"]
    operations: ["CREATE", "UPDATE"]
    resources: ["deployments"]
```

这段配置还不是完整可运行方案，但已经足够你理解 Webhook 的几个关键维度：

- 拦截哪些资源。
- 拦截哪些操作。
- 请求失败时怎么处理。
- 这是一个“验证型”还是“修改型”入口。

### 第三步：理解它在请求链中的位置

Webhook 之所以重要，是因为它发生在资源真正落库之前。也就是说：

- 不是“先放进去再慢慢改”，而是“进门前先检查”。
- 一旦验证不通过，请求可以直接被拒绝。
- 一旦需要统一补默认值，也可以在进入系统前就补上。

这和后置巡检完全不是一个治理级别。后置巡检能发现问题，准入控制能阻止问题进入。

### 第四步：Mutating 和 Validating 应该怎么理解

初学阶段你可以这样分：

- `MutatingWebhook`：偏“自动修正”和“标准化补齐”。
- `ValidatingWebhook`：偏“最终把关”和“强约束拒绝”。

真实生产里，两者经常会配合使用：先补，再查，最后放行或拒绝。

### 第五步：注意 Webhook 本身也会带来平台责任

准入控制虽然强大，但不是“多写几条规则就完事”。它本身会引入新的平台责任：

- Webhook 服务自己必须高可用。
- 失败策略必须设计合理。
- 性能和超时要考虑好。
- 错误规则可能直接挡住整个集群的正常发布。

所以，Webhook 是平台治理的利器，也是高风险能力，必须谨慎上线。

### 这一节应该带走什么

- 准入控制解决的是“请求是否合规”，而不是“谁能发请求”。
- Mutating 负责修改补齐，Validating 负责最终拦截。
- Admission Webhook 能把平台规范从文档变成集群入口规则。
- 越强大的治理能力，越需要严谨设计自己的可用性与回滚方案。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把平台规则自动化为集群入口守卫，而不是靠人工检查。
- 它特别适合标准化、合规性、默认值补齐和禁止危险配置。
- 它能显著提升平台一致性和治理执行力。

### 缺点

- 理解与实现门槛较高，牵涉 API Server 准入链和服务可用性。
- 配置错误时影响面很大，可能直接阻断正常发布。
- 平台团队需要同时承担规则正确性和 Webhook 自身高可用责任。

### 使用场景

- 强制镜像规范、资源声明、标签要求、安全约束。
- 自动补默认标签、注解、sidecar 注入等标准化场景。
- 需要把治理要求提升到集群入口层的团队。

### 注意事项

- 先明确规则是否适合在“入口阶段”执行，再决定是否做 Admission。
- Mutating 和 Validating 不要职责混乱。
- Webhook 服务自身必须被当成关键平台组件来设计。

### 常见踩坑经验

- 最常见的坑，是把所有治理逻辑都堆到应用侧人工检查，最终规则执行质量极不稳定。
- 第二个坑，是只关注规则本身，不关注 Webhook 服务的高可用与超时策略。
- 第三个坑，是 failure policy 设计不当，导致 Webhook 一出问题，整个发布链路都被卡住。

这一章真正想帮你建立的，是入口治理视角：**平台成熟的标志，不只是有规范，而是规范已经变成了集群在门口就会执行的规则。**
