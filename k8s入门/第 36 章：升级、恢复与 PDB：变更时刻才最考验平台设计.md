# 第 36 章：升级、恢复与 PDB：变更时刻才最考验平台设计

> 对应学习计划第 6 周升级、备份恢复和 PodDisruptionBudget。

很多 Kubernetes 学习者在前半程都会陷入一个错觉：只要第一次把系统部署起来，后面无非就是多写几个 YAML。但真正的生产难题往往恰恰不在“第一次成功”，而在“系统已经跑着时，如何在不停机或少停机前提下做升级、维护、节点腾挪、故障恢复”。换句话说，**平台最经受考验的时刻，不是静态运行，而是变更时刻。** `PodDisruptionBudget`、升级流程和恢复策略，就是在告诉你：Kubernetes 真正成熟的一面，体现在变更与恢复能力上。

## 1）项目背景

为什么变更时刻这么危险？因为生产系统的问题往往不是“不会跑”，而是“本来跑得好好的，被一轮升级、驱逐或恢复动作自己打坏了”。典型场景包括：

- 节点维护时批量 `drain`，把关键副本一口气赶没。
- 升级控制平面或节点时，没有明确先后顺序。
- 备份存在，但恢复从未演练，真出问题时没人知道怎么回。
- 多副本业务本来健康，却因为人为驱逐策略失控导致可用副本瞬间不足。

这说明：生产稳定性从来不只是“能否自动重启”，还包括“平台和人类在做计划内变更时，系统能否继续维持可用”。

本章有三条主线需要你建立起来：

- **升级**：版本演进不是一次命令，而是一套计划、顺序和回滚意识。
- **恢复**：备份不是做完就行，恢复路径必须被理解和验证。
- **PDB**：PodDisruptionBudget 帮你约束“计划内中断”不能一下子把业务打穿。

尤其是 PDB，初学者非常容易误解。它不是拿来防业务 bug 的，也不是拿来防节点宕机的，它主要作用于**主动中断场景**，例如驱逐、滚动维护、集群运维动作。它的本质是在告诉平台：“你可以动我，但至少要给我留下这么多可用副本。”

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我已经有 3 个副本了，是不是节点维护时随便 `drain` 一个节点也不会有问题？

**大师**：不一定。多副本是基础，但不是保证。你还得看副本当前分布、探针状态、实际 Ready 数量，以及运维动作会不会一下子驱逐过多实例。

**小白**：那 PDB 是不是就是“告诉系统别一下子赶太多人走”？

**大师**：这个理解很接近本质。它不是帮你修故障，而是约束计划内中断的冲击面。

**小白**：那为什么很多团队升级一出事，就不是版本问题，而是维护窗口问题？

**大师**：因为他们只设计了“如何运行”，没设计“如何改变”。真正成熟的平台，不只是会启动，还要会升级、会恢复、会优雅腾挪。

**小白**：所以变更策略本身也是架构的一部分？

**大师**：完全正确。你没有为变更设计系统，系统就会在变更时暴露真实脆弱性。

**小白**：那备份是不是也是同样道理？

**大师**：对。没有经过恢复演练的备份，只能叫心理安慰，不能叫恢复方案。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节先从最实用也最容易落地的对象入手：`PodDisruptionBudget`。它会帮你建立对“计划内中断保护”的基本认知。

### 第一步：为关键服务创建 PDB

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: web
```

这段配置表达的是：

- 针对标签 `app=web` 的这组 Pod。
- 在主动中断场景下，至少要保持 2 个可用副本。

如果你的 `web` 总共只有 3 个副本，这基本上就是在告诉平台：“一次最多只能安全打掉 1 个。”

### 第二步：理解它和 `drain` 的关系

在节点维护时，`kubectl drain` 会触发驱逐逻辑。如果某个工作负载受 PDB 保护，驱逐动作就不会无限制地继续，而是要先满足预算条件。

这一步最想帮你建立的认知是：**PDB 不是业务配置，它是运维动作和业务可用性之间的防撞栏。**

### 第三步：理解 `minAvailable` 和 `maxUnavailable`

学习阶段可以先这样记：

- `minAvailable`：至少要留多少个可用副本。
- `maxUnavailable`：最多允许多少个不可用副本。

两者表达的是同一类约束，只是视角不同。真正写哪种，取决于你更习惯从“保底值”还是“中断额度”思考。

### 第四步：把它放回升级与恢复主线里看

PDB 只是变更保护的一环，不是全部。真正完整的升级和恢复能力，通常还要同时关注：

- 升级顺序是否明确。
- 控制平面和节点升级是否分阶段。
- 关键数据是否有备份。
- 回滚和恢复流程是否真的演练过。

所以，别把 PDB 看成“安全感对象”，它只是把计划内中断这一个环节控制住。

### 第五步：形成一个生产心智

一个成熟的平台，在做任何升级、维护、恢复动作前，至少应该先问：

1. 当前最小可用副本数是多少。
2. 本次变更会不会突破这个底线。
3. 如果失败，回滚和恢复路径在哪里。

只要这三个问题开始成为习惯，你对平台稳定性的理解就已经比“会部署”高出很多层了。

### 这一节应该带走什么

- 升级、维护、恢复是生产稳定性设计的一部分，不是附属流程。
- PDB 的核心价值在于保护计划内中断时的最小可用性。
- 备份的价值必须通过恢复路径和演练来兑现。
- 变更能力是架构能力的一部分，不只是运维手册的附录。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让升级和维护窗口正式进入系统设计范围，而不是临场拍脑袋。
- PDB 能有效约束主动中断对业务可用性的冲击。
- 它把“恢复能力”从口头方案拉回到平台治理主线。

### 缺点

- 学习者容易把 PDB 误解成“万能高可用对象”。
- 变更策略一旦设计不完整，PDB 也无法挽救所有问题。
- 如果没有备份和演练，恢复能力仍然是纸面能力。

### 使用场景

- 节点维护、集群升级、滚动腾挪等主动中断场景。
- 对最小可用副本数有明确要求的核心业务。
- 需要把升级和恢复纳入常规平台治理的团队。

### 注意事项

- PDB 主要作用于计划内中断，不是解决所有故障的银弹。
- 设置预算前先确认真实副本数和就绪状态，不要闭眼写值。
- 备份一定要配合恢复验证，而不是只满足“做过备份”。

### 常见踩坑经验

- 最常见的坑，是没有 PDB 就直接批量驱逐节点上的关键服务，结果维护窗口演变成生产事故。
- 第二个坑，是 PDB 配得过严，导致集群维护和升级动作根本无法推进。
- 第三个坑，是只谈升级步骤，不谈失败后的回滚与恢复路径。

这一章真正想帮你建立的，是变更治理视角：**系统真正成熟，不是因为它能稳定运行，而是因为它在被改变时仍然可控。**
