# 第 29 章：RBAC：细粒度授权不是锦上添花，而是共享集群的底线

> 对应学习计划第 5 周 RBAC 与权限隔离。

在很多团队里，Kubernetes 权限问题往往要到出过一次事故之后，大家才开始认真对待。有人误删生产资源，有人拿着管理员权限做调试，有人为了省事直接给整个团队发 `cluster-admin`。这些做法前期看似高效，后期几乎注定出事。RBAC 的意义就在于：**让“谁能对什么资源做什么事”成为平台可声明、可审计、可收敛的规则，而不是靠口头约定和人情管理。**

## 1）项目背景

为什么 RBAC 是共享集群的底线，而不是可选增强？因为只要集群不再是一个人独占，就必然要面对授权问题：

- 开发者是否能看所有命名空间？
- 应用是否能创建或删除某类资源？
- 某个 SA 是否能读 Secret？
- 平台团队与业务团队的权限边界如何划分？

如果这些问题没有制度化，通常就会演变成两种极端：

- 全都不给，人人都很难用，平台效率极低。
- 全都给，人人都像管理员，事故迟早发生。

RBAC（Role-Based Access Control）试图在这两者之间建立可治理的中间状态。核心对象通常包括：

- `Role`：命名空间内的一组权限规则。
- `ClusterRole`：集群级范围的权限规则，或可复用到多个命名空间。
- `RoleBinding` / `ClusterRoleBinding`：把权限规则绑定给用户、组或 ServiceAccount。

你可以把它看成一句非常直接的话：

- Role/ClusterRole 定义“能做什么”。
- Binding 定义“把这些能力给谁”。

RBAC 真正的价值，不只是安全，还包括协作效率。因为只有当权限边界被清楚定义后，团队才能既高效工作，又不至于把整个集群暴露在过大的风险面前。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我觉得权限管理挺麻烦的。直接把 `cluster-admin` 发给大家，不是最省事吗？

**大师**：这是最省事，也是最危险的做法。省的是今天的沟通成本，埋的是明天的事故成本。

**小白**：可如果不给高权限，很多操作做不了，大家会抱怨平台卡流程。

**大师**：这正是 RBAC 的价值。它不是让大家“都没有权限”，而是让大家“有刚刚好的权限”。

**小白**：那 `Role` 和 `ClusterRole` 的区别是什么？

**大师**：最直观的区别是作用范围。`Role` 通常只在命名空间内有效；`ClusterRole` 面向集群范围，也可以被复用。

**小白**：那 Binding 就是把权限发给具体身份？

**大师**：对。权限规则定义“能干啥”，Binding 定义“给谁用”。两者缺一不可。

**小白**：那是不是所有人都应该从最小权限开始？

**大师**：这就是最小权限原则。你先给够完成工作所需的最小权限，再按需求加，不要反过来从管理员往下减。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们从最经典的例子开始：给某个命名空间的身份授予“只读 Pod”的权限。

### 第一步：创建一个最小 Role

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: team-a
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
```

这段配置表达得非常明确：

- 只允许访问核心 API 组中的 `pods`。
- 只能做 `get/list/watch` 这三种读取动作。
- 不允许创建、删除、更新。

### 第二步：绑定给一个主体

接下来可以通过 `RoleBinding` 把这组权限绑定给某个用户或 ServiceAccount。这里先记住一个最核心的关系：

- Role 定义权限集合。
- RoleBinding 才让某个主体真正获得这些权限。

没有 Binding，再完整的 Role 也只是“摆在架子上的制度”。

### 第三步：用 `kubectl auth can-i` 验证

```bash
kubectl auth can-i list pods -n team-a --as=system:serviceaccount:team-a:app-reader
```

这一步很重要，因为 RBAC 最怕停留在“我以为配对了”。真正好的习惯是：配置完权限，立刻验证这个主体到底能不能执行你预期的动作。

### 第四步：理解为什么不该滥发集群管理员权限

如果你总是用 `cluster-admin`，你得到的不是便利，而是：

- 权限边界模糊。
- 问题排查困难。
- 审计价值下降。
- 一次误操作影响面极大。

RBAC 的目标从来不是“尽量复杂”，而是“让最小权限成为默认”。

### 第五步：建立一条授权设计思路

学习阶段可以先记这样一条路径：

1. 先明确主体是谁。
2. 再明确它需要操作哪些资源。
3. 只给完成任务所需的 verbs。
4. 最后再用 `can-i` 验证。

这个顺序看起来简单，但它几乎就是最小权限设计的入门套路。

### 这一节应该带走什么

- RBAC 回答的是“谁能对什么资源做什么事”。
- Role/ClusterRole 管权限内容，Binding 管授予对象。
- 最小权限原则是共享集群最基本的安全底线。
- RBAC 不只是安全机制，也是团队协作效率的基础设施。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让权限边界从模糊习惯变成可声明、可审计、可验证的规则。
- 它非常适合共享集群中的多团队协作与职责分层。
- 它与 Namespace、ServiceAccount、Secret 等能力天然联动，构成基础安全模型。

### 缺点

- 初学者容易觉得规则多、对象多，理解成本较高。
- 权限一旦设计不清晰，排障和授权调整会变复杂。
- 如果团队缺少统一规范，RBAC 配置也可能逐渐膨胀失控。

### 使用场景

- 多团队共享集群的权限分层。
- 平台团队与业务团队职责边界划分。
- 应用、CI/CD、自动化系统对 Kubernetes API 的访问控制。

### 注意事项

- 先从最小权限开始，不要默认发管理员。
- Role 与 Binding 总是成套理解，不要只写一半。
- 配好权限后一定要验证，不要停留在“我觉得应该行”。

### 常见踩坑经验

- 最常见的坑，是图省事直接把 `cluster-admin` 发给所有人。
- 第二个坑，是 Role 写得很细，但忘了 Binding，结果主体其实什么都做不了。
- 第三个坑，是只凭感觉配权限，不做 `kubectl auth can-i` 验证，导致问题长期隐性存在。

这一章真正想帮你建立的，是授权治理视角：**共享集群里，权限不是附属配置，而是平台秩序本身。**
