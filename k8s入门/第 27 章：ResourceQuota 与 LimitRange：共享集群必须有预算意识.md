# 第 27 章：ResourceQuota 与 LimitRange：共享集群必须有预算意识

> 对应学习计划第 3 周资源治理与命名空间限制。

当集群开始被多个团队、多个项目共享时，光有 requests/limits 还不够。因为那更多是在约束“单个工作负载怎么花钱”，而不是回答“这个团队总共能花多少、默认应该怎么花”。如果没有命名空间级资源治理，集群迟早会从“大家都能用”滑向“谁手快谁占满”。`ResourceQuota` 和 `LimitRange` 存在的意义，就是把资源管理从单服务层，提升到团队和命名空间层面。

## 1）项目背景

共享集群里最常见的几类资源问题通常不是代码 bug，而是治理缺失：

- 某个团队一下子起了太多 Pod，把调度器和节点资源压满。
- 有的服务完全不写 requests/limits，调度和运行期都变得不可预测。
- 有的命名空间无上限增长，最终拖垮整个集群。
- 不同团队之间没有清晰预算边界，资源使用全靠自觉。

这类问题的本质是：**集群资源是一种共享资产，不能只靠应用开发者逐个自律。** 平台必须有更上层的治理工具。

Kubernetes 给出的两个关键对象是：

- `ResourceQuota`：限制一个命名空间里总共最多能用多少资源、建多少对象。
- `LimitRange`：给命名空间中的单个 Pod / Container 设默认值和上下限约束。

它们的关系可以这样理解：

- `ResourceQuota` 管总预算。
- `LimitRange` 管单项默认和边界。

这两个对象配合之后，平台团队终于可以说清楚：

- 某个团队命名空间最多能建多少 Pod。
- 某个命名空间默认每个容器至少/至多能申请多少资源。
- 不写资源声明的工作负载是否应该被拒绝或自动补默认值。

从平台治理角度看，这一步非常关键，因为它把资源竞争从“野蛮生长”拉回到了“可预期边界”。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我已经给服务都加 requests/limits 了，为什么平台同学还老说要做配额管理？

**大师**：因为 requests/limits 解决的是“一个服务怎么花”，而平台还得知道“一个团队总共能花多少”。

**小白**：也就是说，单应用约束和命名空间预算是两回事？

**大师**：对。共享集群里，真正危险的往往不是一个 Pod 配错，而是一个命名空间整体失控。

**小白**：那 `ResourceQuota` 就是在给团队做预算？

**大师**：可以这么理解。它像部门预算表，限制命名空间里总资源、对象数、PVC 数量等上限。

**小白**：那 `LimitRange` 呢？

**大师**：它更像默认规则和单体边界。比如不给你随便起一个没有 requests 的容器，或者默认帮你补上资源值。

**小白**：原来一个管“总量”，一个管“单体默认值和上下限”。

**大师**：没错。没有这两层治理，共享集群迟早会从“公共资源池”退化成“资源抢占现场”。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先用 `ResourceQuota` 建一个命名空间预算，再用 `LimitRange` 补上默认资源规则。

### 第一步：创建命名空间预算

新建 `team-a-quota.yaml`：

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-a-quota
  namespace: team-a
spec:
  hard:
    pods: "20"
    requests.cpu: "4"
    limits.memory: 8Gi
```

这段配置表达的是：

- `team-a` 命名空间最多只能有 20 个 Pod。
- 总 requests CPU 不能超过 4 核。
- 总 limits memory 不能超过 8Gi。

### 第二步：观察命名空间层面的约束生效

```bash
kubectl apply -f team-a-quota.yaml
kubectl describe quota -n team-a
```

这一步最重要的不是命令本身，而是你要开始用“命名空间预算”视角看资源，而不是只盯单个 Deployment。

### 第三步：用 `LimitRange` 设置默认规则

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: team-a-limits
  namespace: team-a
spec:
  limits:
  - type: Container
    defaultRequest:
      cpu: 100m
      memory: 128Mi
    default:
      cpu: 500m
      memory: 512Mi
```

这个对象的价值在于：即使业务方忘了写资源字段，平台也能给出一套合理默认值，避免“完全裸奔”的工作负载进入共享集群。

### 第四步：理解两者组合后的治理效果

把它们放在一起看，会形成这样的效果：

- `LimitRange` 让单个工作负载的默认行为更可控。
- `ResourceQuota` 让整个命名空间的总资源消耗有边界。

这和公司财务非常像：一边规定“单笔报销标准”，一边规定“部门全年预算”。两边都没有，系统一定失控。

### 第五步：什么时候特别需要它们

只要符合这些条件，就要认真考虑它们：

- 多团队共享一个集群。
- 平台已经开始出现资源争抢或误操作。
- 希望把成本和资源责任落实到命名空间层。
- 不希望未声明资源的工作负载随便进入生产集群。

### 这一节应该带走什么

- `ResourceQuota` 管命名空间总预算，`LimitRange` 管单体默认值和上下限。
- requests/limits 解决单服务问题，Quota/LimitRange 解决共享治理问题。
- 共享集群必须把资源问题制度化，而不能只靠口头约定。
- 资源治理不是“限制开发者”，而是保护平台整体稳定性。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把资源竞争从野蛮增长变成了可治理行为。
- 它让共享集群具备预算意识和默认约束能力。
- 它能显著降低“单团队误操作拖垮全局”的风险。

### 缺点

- 如果阈值配置不合理，可能让正常发布和扩容受阻。
- 初学者容易只看到“被限制”，看不到平台稳定性收益。
- 治理规则越多，命名空间配套管理就越需要标准化。

### 使用场景

- 多团队共享生产或准生产集群。
- 希望对不同项目做资源预算和成本边界管理。
- 想为命名空间建立默认资源治理规则的平台团队。

### 注意事项

- 先理解业务规模，再设置预算，不要拍脑袋给死值。
- `LimitRange` 和 `ResourceQuota` 最好成套设计，而不是单独上一个。
- 要结合监控持续调整，而不是一次配置永远不动。

### 常见踩坑经验

- 最常见的坑，是只靠口头约定来约束团队资源使用，最后集群被一次误操作拖垮。
- 第二个坑，是 Quota 设得过紧，结果发布和扩容频繁失败，大家开始绕规则。
- 第三个坑，是只上 ResourceQuota 不上 LimitRange，导致没写资源字段的工作负载仍然很混乱。

这一章真正想帮你建立的，是共享平台视角：**单个服务可以讲优化，多个团队共享集群时必须先讲预算和边界。**
