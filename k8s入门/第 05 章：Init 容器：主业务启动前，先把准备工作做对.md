# 第 05 章：Init 容器：主业务启动前，先把准备工作做对

> 对应学习计划第 1 周 Day 3-4 的 Init 容器实践。

很多应用在真正开始提供服务之前，都要先做一堆“准备动作”：检查依赖是否可用、拉取初始化数据、渲染配置模板、等待数据库或缓存就绪、执行一次迁移脚本。如果这些动作没有被好好设计，最终往往会被硬塞进主容器入口脚本，导致镜像职责混乱、启动过程不可观测、失败时也很难定位。Init 容器的意义，就是把这部分准备工作从主业务容器中清晰地拆出来。

## 1）项目背景

初学者刚开始写容器镜像时，常见做法是把所有启动前逻辑都丢进 `entrypoint.sh`。比如：

- while 循环等待 MySQL 就绪。
- 拉取远程配置文件。
- 初始化目录结构和权限。
- 执行数据库 migration。
- 最后再 `exec` 真正的业务进程。

这种做法短期看似方便，因为“都在一个脚本里”，部署时也只要起一个容器。但问题很快会暴露出来：

- 主镜像被迫包含很多与业务无关的工具，例如 `curl`、`nslookup`、`bash`。
- 启动过程混杂在一起，很难判断失败发生在准备阶段还是业务阶段。
- 日志混乱，责任边界不清。
- 一些初始化动作本来只需要执行一次，却跟着主容器重启反复执行。

Kubernetes 提供 `initContainers`，本质上就是把“准备阶段”和“运行阶段”拆开。Init 容器会在普通业务容器启动前依次执行，只有前一个成功退出，后一个才会开始。等所有 Init 容器都成功完成之后，主容器才会真正启动。

这个模型非常优雅，因为它符合真实工程流程：先准备环境，再启动服务；先校验依赖，再接流量；先做一次性动作，再进入长期运行状态。你可以把 Init 容器理解成 Pod 的“启动前检查与准备流水线”。

更重要的是，Init 容器让职责变得清晰：

- 主容器负责“长期运行的业务进程”。
- Init 容器负责“必须在业务启动前完成的准备工作”。

这种拆分会直接提升镜像设计质量、排障效率和团队协作清晰度。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我现在的应用启动前要等数据库和 Redis 都起来，所以我在主容器脚本里写了个 while 循环，一直等到它们可用再启动服务。

**大师**：能工作，但这不是最清晰的方式。你把“初始化逻辑”和“业务逻辑”搅在一起了。

**小白**：可它们不是都属于启动流程吗？放一起好像也没什么问题。

**大师**：问题在于职责。主容器应该代表“业务已经开始运行”。如果它还在忙着等待依赖、创建目录、拉配置，那运行语义就被污染了。

**小白**：那 Init 容器就是专门用来做这些事的？

**大师**：对。你可以把它看成正式开业前的准备班子。门还没开，服务还没接流量，但卫生、备货、检查、调试这些动作已经开始做了。

**小白**：它和 Sidecar 有什么区别？

**大师**：Init 容器是“先执行完再退出”；Sidecar 是“和主容器一起长期运行”。一个是启动前准备，一个是运行期协作，它们完全不是一类角色。

**小白**：如果 Init 容器失败了会怎样？

**大师**：Pod 会停在初始化阶段，主容器不会启动。也正因为这样，它非常适合做强前置校验。与其让业务启动后半死不活，不如在门口就拦下来。

**小白**：听起来它其实是在提高系统的“启动确定性”。

**大师**：说得非常好。它让你把“什么必须先满足”这件事，写成平台可理解、可观察、可重试的声明。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们做一个简单但非常典型的实验：让主容器在启动前先等待数据库服务名能够被解析。这个案例虽然简单，却几乎覆盖了 Init 容器最常见的思路。

### 第一步：编写带 Init 容器的 Pod

新建 `pod-with-init.yaml`：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-init
spec:
  initContainers:
  - name: wait-db
    image: busybox:1.36
    command:
    - sh
    - -c
    - |
      until nslookup mysql.default.svc.cluster.local; do
        echo "waiting for mysql service"
        sleep 2
      done
  containers:
  - name: app
    image: nginx:1.27
    ports:
    - containerPort: 80
```

这段配置表达得非常清楚：在 `app` 容器启动前，先执行 `wait-db`。只有当 `mysql.default.svc.cluster.local` 能被解析时，Init 容器才会退出成功，主容器随后启动。

### 第二步：应用配置并观察初始化状态

```bash
kubectl apply -f pod-with-init.yaml
kubectl get pod app-with-init -w
```

如果目标服务不存在，Pod 会停在 `Init:0/1` 或类似状态；如果目标服务存在并能被解析，Pod 最终会进入 `Running`。这个现象很有教学意义，因为它让你看到：**Pod 的“启动”并不是一个瞬间，而是一个有阶段的过程。**

### 第三步：查看 Init 容器日志

```bash
kubectl logs app-with-init -c wait-db
kubectl describe pod app-with-init
```

这一小步非常关键。很多人知道 Init 容器会先执行，但不知道它的日志也是独立可观测的。通过 `-c wait-db`，你可以只看初始化阶段输出什么，这比在一个超长入口脚本里翻日志清晰得多。

### 第四步：理解它与主容器的责任边界

你可以继续思考下面这些动作，到底应该放在哪：

- 等待数据库、缓存、配置中心可达：通常适合 Init 容器。
- 创建运行时目录和权限：通常适合 Init 容器。
- 一次性渲染模板文件：通常适合 Init 容器。
- 主业务服务启动并长期提供请求：属于主容器。
- 持续日志转发、代理、服务网格 sidecar：属于运行期容器，而不是 Init 容器。

### 第五步：一个非常重要的工程判断

并不是所有初始化动作都适合放在 Init 容器里。比如数据库 migration，如果它可能被多个副本同时执行，就要特别谨慎。Kubernetes 给了你工具，但不替你决定业务逻辑是否幂等。Init 容器负责“先做完再启动”，不负责替你解决并发冲突。

### 这一节应该带走什么

- Init 容器专门负责启动前准备，不应该和主业务生命周期混在一起。
- 它按顺序执行，失败会阻止主容器启动。
- 它非常适合做前置校验、目录初始化、模板渲染和依赖等待。
- 它不是 Sidecar，也不是所有启动脚本问题的万能解法。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把初始化阶段和主业务运行阶段清晰分离，职责边界更明确。
- 它让启动前逻辑具备可观察性，日志、状态和失败原因都更容易定位。
- 它能减少主镜像中不必要的工具依赖，使业务镜像更干净、更聚焦。

### 缺点

- 如果设计不当，可能把不该串行的逻辑硬塞进启动前流程，拖慢整体启动时间。
- 一些初始化动作涉及并发和幂等性问题，不能仅靠 Init 容器“自动解决”。
- 新手容易把所有前置逻辑都塞进去，导致 Init 容器本身变成新的复杂脚本堆积点。

### 使用场景

- 等待依赖服务可达或 DNS 可解析。
- 生成配置文件、准备目录、设置权限。
- 下载或初始化业务启动前必须具备的资源。

### 注意事项

- Init 容器适合做“必须先完成”的动作，不适合长期运行任务。
- 尽量让初始化逻辑保持简单、可重试、可观测。
- 涉及数据库变更、共享资源写入等操作时，要额外考虑幂等性和并发冲突。

### 常见踩坑经验

- 最常见的坑，是把所有依赖等待都写进主容器启动脚本，导致业务镜像职责失焦、排障困难。
- 第二个坑，是把持续运行的辅助逻辑误放到 Init 容器里，结果初始化阶段永远结束不了。
- 第三个坑，是没有看 Init 容器自己的日志，出了问题还在主容器里找原因，方向从一开始就错了。

学完这一章后，你应该把 Init 容器理解成一句很朴素的话：**先把开门前的事情做对，再让业务正式开门。** 这个思想看似简单，却能显著提升 Kubernetes 工作负载设计的清晰度和可维护性。
