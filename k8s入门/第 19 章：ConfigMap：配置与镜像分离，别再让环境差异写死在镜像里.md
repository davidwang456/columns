# 第 19 章：ConfigMap：配置与镜像分离，别再让环境差异写死在镜像里

> 对应学习计划中应用配置管理与综合项目配置拆分。

很多团队刚开始容器化时，都会下意识把配置一起打进镜像：测试环境一个镜像，预发环境一个镜像，生产环境又一个镜像。短期看似简单，长期却会把构建、发布和配置管理全都搅在一起。ConfigMap 的价值，就是把“应用是什么”和“应用在哪个环境怎么跑”拆开，让配置回到配置本身，而不是焊死在镜像里。

## 1）项目背景

为什么“配置与镜像分离”这么重要？因为镜像本质上应该描述的是一份相对稳定的应用交付物，而环境配置却天然是变化频繁的：

- 测试环境和生产环境地址不同。
- 日志级别可能会变化。
- 功能开关会随版本逐步启用。
- Nginx、应用参数、外部服务地址都可能独立调整。

如果所有这些内容都直接打进镜像，会带来很多问题：

- 仅仅改一个配置，也得重新构建镜像。
- 环境差异不再体现在部署层，而被藏进镜像版本里。
- 回滚时很难说清是“代码变了”还是“配置变了”。
- 团队协作时，代码发布和配置变更耦合得过紧。

ConfigMap 就是 Kubernetes 给出的标准答案之一。它允许你把非敏感配置以独立对象形式交给平台，再通过几种常见方式注入到容器：

- 环境变量。
- 文件挂载。
- 命令行参数拼接。

它的价值不只是“单独存一份配置”，而是让配置成为可版本化、可观察、可复用的交付元素。这样一来，镜像更纯粹，部署更灵活，环境差异也更容易被管理。

当然，ConfigMap 也有边界。它适合非敏感配置，不适合密码、证书、Token 等真正敏感信息；它支持更新，但“是否热生效”还取决于应用读取配置的方式。这些边界如果不提前讲清楚，学习者很容易误用。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我现在不同环境的配置都写在不同镜像里，感觉也能跑。为什么还要多搞一个 ConfigMap？

**大师**：因为你现在把“代码交付”和“环境差异”绑死了。以后只改一行配置，也得重新构建和发布镜像。

**小白**：可这样不是更省对象吗？

**大师**：对象少不代表管理简单。真正简单的是职责清晰。镜像应该描述应用本身，配置应该描述环境和运行参数。

**小白**：那 ConfigMap 就是把配置独立成 Kubernetes 对象？

**大师**：对。这样你既可以通过环境变量注入，也可以挂成文件，还能单独审查和管理变更。

**小白**：那我改了 ConfigMap，应用会自动生效吗？

**大师**：这正是边界问题。对象更新了，不等于应用一定热加载。平台负责把配置交给你，应用是否实时读取，是应用自己的实现问题。

**小白**：原来 ConfigMap 解决的是“配置交付”，不是“配置热更新自动魔法”。

**大师**：说得很对。理解边界，比记字段更重要。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们用 Nginx 配置示例，体验 ConfigMap 作为文件注入和环境变量注入的两种典型方式。

### 第一步：创建一个 ConfigMap

新建 `web-config.yaml`：

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: web-config
data:
  APP_ENV: test
  nginx.conf: |
    server {
      listen 80;
      location / {
        return 200 "config from configmap
";
      }
    }
```

这里同时放了两类配置：

- `APP_ENV`：适合环境变量使用。
- `nginx.conf`：适合以文件方式挂载。

### 第二步：把 ConfigMap 作为环境变量注入

```yaml
env:
- name: APP_ENV
  valueFrom:
    configMapKeyRef:
      name: web-config
      key: APP_ENV
```

这种方式很适合简单开关、地址、模式之类的单值配置。优点是应用读取简单，缺点是更新后往往不会自动反映到进程中，通常要靠重启或重新发布让应用重新读取。

### 第三步：把 ConfigMap 作为文件挂载

```yaml
volumeMounts:
- name: nginx-config
  mountPath: /etc/nginx/conf.d/default.conf
  subPath: nginx.conf
volumes:
- name: nginx-config
  configMap:
    name: web-config
```

这种方式更适合完整配置文件。但要注意，`subPath` 会影响热更新行为，很多初学者就在这里踩坑：对象改了，容器里的文件却没如预期变化。

### 第四步：观察配置修改后的行为

```bash
kubectl apply -f web-config.yaml
kubectl apply -f web-deployment.yaml
kubectl exec -it <pod-name> -- printenv APP_ENV
kubectl exec -it <pod-name> -- cat /etc/nginx/conf.d/default.conf
```

这里你要刻意理解：平台负责把配置注入进去，但应用是否会自动重载，是另一回事。对于像 Nginx 这样的程序，很多时候你还需要 reload；对于某些业务应用，则可能要重启 Pod。

### 第五步：建立 ConfigMap 的边界感

学习到这里，你应该牢牢记住：

- 非敏感配置：优先考虑 ConfigMap。
- 敏感信息：转到 Secret。
- “对象更新了”不等于“业务自动热更新成功”。

这三个判断，会帮你避免后面很多配置设计误区。

### 这一节应该带走什么

- ConfigMap 的核心价值是配置与镜像解耦。
- 它适合以环境变量和文件两类方式注入应用。
- 它解决配置交付问题，不保证业务一定自动热加载。
- ConfigMap 是非敏感配置对象，不应承担密钥管理职责。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让配置从镜像中独立出来，显著降低环境切换成本。
- 它便于审查、版本化和组合复用，是工程化交付的重要基础。
- 它和 Deployment、滚动更新、模板化管理天然配合。

### 缺点

- 只能用于非敏感配置，安全边界很明确。
- 热更新行为并不总是符合直觉，容易被误解。
- 一旦配置设计混乱，也可能演变成新的“配置垃圾堆”。

### 使用场景

- 环境变量、功能开关、服务地址、普通应用参数。
- Nginx、应用配置文件、模板化配置注入。
- 多环境部署中的配置解耦与管理。

### 注意事项

- 不要把敏感信息放进 ConfigMap。
- 先明确应用如何读取配置，再决定用环境变量还是文件挂载。
- 更新 ConfigMap 后要确认应用是否真的重新读取到了配置。

### 常见踩坑经验

- 最常见的坑，是把所有环境差异都打进镜像，导致配置变更必须伴随镜像重建。
- 第二个坑，是把敏感信息和普通配置混在同一个 ConfigMap 里。
- 第三个坑，是误以为 ConfigMap 改了业务一定自动热更新，结果线上变更根本没生效。

这一章真正要建立的，是一条很重要的交付原则：**应用镜像应该尽量稳定，环境差异应该尽量外置。**
