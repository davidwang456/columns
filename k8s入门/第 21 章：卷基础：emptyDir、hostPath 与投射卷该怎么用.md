# 第 21 章：卷基础：emptyDir、hostPath 与投射卷该怎么用

> 对应学习计划第 2 周卷类型实验。

很多开发者第一次把应用放进容器之后，都会经历一个典型的认知冲击：我明明把文件写进去了，为什么容器一重建数据就没了？这时你才会真正意识到，**容器文件系统默认不是一个可靠的持久化边界。** Kubernetes 的卷系统存在的意义，就是把“哪些数据跟着容器走，哪些数据跟着 Pod 走，哪些数据要独立于实例存在”这件事讲清楚。

## 1）项目背景

在单机开发时，我们往往不会太敏感地区分“写到容器里”和“写到持久磁盘上”之间的区别，因为场景简单、生命周期短、排障也近在眼前。但到了 Kubernetes 里，这个问题会迅速变严重：

- Pod 重建后，容器可写层数据通常不会保留。
- 多容器协作时，大家可能需要共享临时文件。
- 应用有时需要访问节点上的某些目录。
- 配置、证书、Secret 可能要以文件方式注入。

如果不区分卷类型，常见后果就会是：

- 把临时缓存错误地当成持久数据。
- 为了图方便长期使用 `hostPath`，结果把应用和节点路径强耦合。
- 以为“挂了个卷”就等于有了生产级持久化。
- 把配置和数据混在同一套卷语义里，导致后续治理混乱。

Kubernetes 卷的世界，最好先从三类最典型场景入手理解：

- `emptyDir`：Pod 生命周期内的临时共享空间。
- `hostPath`：直接映射节点目录，能力强但风险大。
- 投射类卷（如 `configMap`、`secret`、`projected`）：把配置和凭证以文件方式交给应用。

它们共同说明了一件事：**卷不只是“把目录挂进去”，而是在表达数据与实例之间的生命周期关系。** 这句话如果理解透了，后面学 PV、PVC、CSI 会轻松很多。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我的程序把文件写进 `/tmp`，结果 Pod 一重建就没了。我以为容器里的文件应该和机器上一样能留着。

**大师**：这正是容器化之后最容易踩的第一类坑。容器可写层从来不是你该默认依赖的持久化边界。

**小白**：那我只要加个 volume 就一定安全了吗？

**大师**：不一定。你得先回答一个问题：这个数据到底应该活多久？跟着容器活、跟着 Pod 活，还是要独立于实例存在？

**小白**：原来卷类型其实是在表达生命周期？

**大师**：对。比如 `emptyDir` 适合 Pod 内临时共享，`hostPath` 是直接拿节点目录，`configMap` 和 `secret` 更像把配置文件投射进去。它们不是一类问题的不同写法，而是在解决不同类型的数据关系。

**小白**：那为什么很多教程都喜欢用 `hostPath`？

**大师**：因为它最直观、最方便，尤其本地实验一眼就能看到文件。但方便往往意味着耦合。你一旦长期依赖它，应用就和某台节点、某个目录结构强绑定了。

**小白**：那是不是可以理解成，卷的第一原则不是“怎么挂”，而是“数据到底属于谁、跟谁一起生灭”？

**大师**：这句话已经非常接近生产思维了。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们用最小示例同时感受三种卷语义：`emptyDir`、`hostPath` 和配置投射卷。

### 第一步：用 `emptyDir` 在 Pod 内共享临时数据

新建 `emptydir-demo.yaml`：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
spec:
  containers:
  - name: writer
    image: busybox:1.36
    command: ["sh", "-c", "echo hello > /cache/data.txt && sleep 3600"]
    volumeMounts:
    - name: cache
      mountPath: /cache
  - name: reader
    image: busybox:1.36
    command: ["sh", "-c", "sleep 3600"]
    volumeMounts:
    - name: cache
      mountPath: /cache
  volumes:
  - name: cache
    emptyDir: {}
```

应用后你可以进入第二个容器查看：

```bash
kubectl apply -f emptydir-demo.yaml
kubectl exec -it emptydir-demo -c reader -- cat /cache/data.txt
```

这个实验最想说明的是：`emptyDir` 很适合 Pod 内多个容器共享临时文件，但它的生命周期只跟 Pod 绑定。Pod 没了，数据也就没了。

### 第二步：用 `hostPath` 体验“方便但危险”的节点目录映射

```yaml
volumes:
- name: host-log
  hostPath:
    path: /var/log
    type: Directory
```

`hostPath` 最大的好处，就是你能直接看到宿主机目录，很多调试型任务、节点 Agent、日志采集器都可能依赖它。但它的问题也同样明显：

- 路径是否存在依赖节点环境。
- 不同节点路径结构可能不同。
- 应用与节点强耦合，可迁移性下降。
- 一旦权限放大，安全边界也会明显变差。

所以，对普通业务应用来说，`hostPath` 不应该成为默认方案；它更适合系统级组件或实验型场景。

### 第三步：理解投射卷不是“存储”，而是“交付方式”

像 `ConfigMap`、`Secret` 这类对象通常也会以卷方式出现在 Pod 中。比如：

```yaml
volumes:
- name: app-config
  configMap:
    name: web-config
```

它们虽然也是卷，但语义和 `emptyDir`、`hostPath` 完全不同。这里重点不是“把数据写进去”，而是“把配置文件交给应用读取”。这一步很重要，因为它能帮你建立一个边界感：**不是所有 volume 都在解决持久化，很多时候它们只是数据交付方式。**

### 第四步：形成一个简单判断模型

学习阶段可以先这样记：

- Pod 内临时共享：优先想 `emptyDir`
- 访问节点真实目录：谨慎使用 `hostPath`
- 注入配置和凭证：考虑 `configMap` / `secret` / `projected`
- 真正持久化数据：继续看下一章的 PV / PVC / StorageClass

### 这一节应该带走什么

- 卷的本质不是“挂目录”，而是表达数据生命周期和归属关系。
- `emptyDir` 适合临时共享，不适合持久化。
- `hostPath` 强大但高耦合，只应在合适场景下使用。
- 投射类卷更像配置交付能力，而不是持久存储能力。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让 Pod 内的数据共享、配置注入和宿主机资源访问变得可声明、可管理。
- 不同卷类型明确表达了不同生命周期和使用边界。
- 学会这些基础卷语义后，后续学习持久化存储会更顺畅。

### 缺点

- 新手容易把所有 volume 混成同一个概念，忽略它们实际解决的问题差异。
- `hostPath` 虽然简单，但带来的耦合和安全风险很容易被低估。
- 如果没有明确生命周期意识，卷设计会很快变得混乱。

### 使用场景

- `emptyDir`：Pod 内临时缓存、共享文件、运行期临时目录。
- `hostPath`：系统组件、节点日志采集、调试和宿主机目录访问。
- 投射类卷：配置、Secret、证书等文件注入。

### 注意事项

- 先判断数据应该跟谁一起生灭，再决定卷类型。
- 不要因为 `hostPath` 简单就把它当成默认方案。
- 看到 volume 时，先问自己它解决的是“存储”还是“交付”。

### 常见踩坑经验

- 最常见的坑，是把容器写层或 `emptyDir` 当成真正持久化存储。
- 第二个坑，是因为调试方便长期依赖 `hostPath`，导致业务和节点环境绑死。
- 第三个坑，是把配置卷和业务数据卷混着设计，后面越用越乱。

这一章真正想帮你建立的是一个底层判断：**数据从来不是简单“放哪儿”，而是“它该跟哪个生命周期绑定”。**
