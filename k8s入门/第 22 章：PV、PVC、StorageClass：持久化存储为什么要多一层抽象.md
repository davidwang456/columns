# 第 22 章：PV、PVC、StorageClass：持久化存储为什么要多一层抽象

> 对应学习计划第 2 周 PV/PVC/StorageClass 主线。

如果说上一章是在帮你建立“卷有不同生命周期”的概念，那么这一章就是在告诉你：**真正的持久化存储，不能让业务直接依赖底层磁盘细节。** 否则，环境一变、存储一换、团队一扩，交付和迁移都会变得非常痛苦。PV、PVC、StorageClass 这一整套抽象，就是 Kubernetes 为持久化存储提供的声明式接口。

## 1）项目背景

为什么要多出 PV、PVC、StorageClass 这几层？因为真正生产里的存储，从来都不是“挂块盘”这么简单。你至少会遇到这些问题：

- 存储到底是谁来准备，是平台管理员先建好，还是应用自己声明就自动分配？
- 业务只关心要多少空间、什么访问模式，还是要知道底层具体是哪块盘、哪个卷、哪种后端？
- 应用迁移环境时，是否要跟着改底层存储细节？

Kubernetes 的回答是把存储也做成一套分层抽象：

- `PersistentVolume`（PV）：平台里可用的一块持久化存储资源。
- `PersistentVolumeClaim`（PVC）：业务对存储的声明式申请。
- `StorageClass`：动态供应规则，告诉平台“这类请求应该怎么创建对应存储”。

这套设计最核心的价值，不在于对象变多，而在于它把“平台提供什么”和“应用需要什么”解耦了。业务不再直接写死某块磁盘路径，而是说：

- 我要 5Gi。
- 我要 `ReadWriteOnce`。
- 我要某类存储策略。

至于具体底层是 NFS、云盘、本地卷、还是 Ceph，以及由谁创建，这些实现细节由平台层处理。

这和计算资源里 requests/limits 的思路非常像。开发者表达需求，平台负责兑现。也正因为如此，持久化存储终于能够纳入统一的声明式交付流程，而不再是“上线前单独找运维开个盘”。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我有个数据库应用需要持久化，那我是不是直接在 YAML 里写死某块磁盘路径就行？

**大师**：本地实验也许能这么干，但一旦换环境、换节点、换存储后端，这种方式马上就会变成负担。

**小白**：那 PV 和 PVC 是干什么的？

**大师**：你可以把 PV 理解成“平台可提供的一块持久存储资源”，PVC 理解成“业务提交的一张存储申请单”。

**小白**：所以应用不需要知道底层到底是哪块盘？

**大师**：没错。应用说“我要什么”，平台说“我怎么给”。这就是抽象层的价值。

**小白**：那 StorageClass 又是什么？

**大师**：它是动态供应规则。没有它，很多时候你得先手工准备 PV；有了它，PVC 一提交，平台就能按规则自动创建底层存储。

**小白**：这样看起来，StorageClass 有点像“存储模板”或“存储工单规则”。

**大师**：这个理解很不错。它把“如何供应这类盘”也抽象成了平台配置。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先从 PVC 视角出发，因为这才是业务开发者最常接触的入口。

### 第一步：声明一个最小 PVC

新建 `data-pvc.yaml`：

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
```

这段配置表达的是：

- 我要一个持久化存储。
- 大小至少 5Gi。
- 访问模式是 `ReadWriteOnce`。

这就是典型的“业务只提需求，不关心底层盘细节”。

### 第二步：观察 PVC 是否被绑定

```bash
kubectl apply -f data-pvc.yaml
kubectl get pvc
kubectl describe pvc data-pvc
kubectl get pv
```

如果集群里已经有可匹配的 PV，或者有默认 StorageClass 支持动态供应，你会看到 PVC 状态变成 `Bound`。这时就意味着：业务的申请已经和底层存储资源完成了对接。

### 第三步：把 PVC 挂进 Pod

```yaml
volumes:
- name: app-data
  persistentVolumeClaim:
    claimName: data-pvc
containers:
- name: app
  image: nginx:1.27
  volumeMounts:
  - name: app-data
    mountPath: /usr/share/nginx/html/data
```

这里最重要的不是挂载语法本身，而是你要意识到：Pod 现在引用的是“申请单”PVC，而不是具体某块底层卷。抽象层依然成立。

### 第四步：理解静态供应与动态供应

你可以把两种模式这样区分：

- 静态供应：管理员先创建 PV，业务提交 PVC 去匹配。
- 动态供应：业务提交 PVC，StorageClass 自动触发底层创建。

学习阶段先理解这两种模式的差异，比死记对象定义更重要。因为在真实团队里，存储交付方式就是平台成熟度的重要标志之一。

### 第五步：建立对“声明式存储”的正确心智

从这章开始，你要逐渐习惯一种新的思路：

- 业务不直接绑某块盘。
- 平台不要求每次上线都人工开存储。
- 持久化能力本身也应该是声明式交付的一部分。

### 这一节应该带走什么

- PV 表示平台可提供的存储资源，PVC 表示业务提出的存储需求。
- StorageClass 负责把“如何供应这类存储”也对象化。
- 业务优先面向 PVC，而不是直接面向底层存储实现。
- 持久化存储抽象的核心价值，是解耦与自动化。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让持久化存储也进入声明式交付体系，显著提高了自动化程度。
- 它把应用需求和底层存储实现解耦，便于迁移和平台治理。
- 它为数据库、中间件和有状态业务提供了标准化的存储接入方式。

### 缺点

- 对初学者来说，对象分层较多，理解门槛明显高于普通卷。
- 存储后端能力差异很大，不同环境表现可能完全不同。
- 一旦底层存储没设计好，表面抽象再漂亮也会在运行时暴露问题。

### 使用场景

- 数据库、中间件、需要跨 Pod 保留数据的业务。
- 需要动态申请存储、跨环境复用配置的团队。
- 想把持久化能力纳入标准平台交付流程的组织。

### 注意事项

- 先分清静态供应和动态供应，再看对象细节。
- 业务层尽量只依赖 PVC，不直接写死底层实现。
- 关注访问模式、回收策略和默认 StorageClass 是否符合预期。

### 常见踩坑经验

- 最常见的坑，是让业务配置直接写死某块磁盘路径，结果一迁移环境就彻底失控。
- 第二个坑，是看到 PVC 创建成功就放心了，却没继续确认它是否真正 `Bound`。
- 第三个坑，是把不同存储后端的能力想得完全一样，导致扩容、快照或访问模式和预期不符。

这一章真正想帮你建立的，是一种平台存储视角：**业务应该声明“我要什么存储”，而不是直接规定“底层必须是哪块盘”。**
