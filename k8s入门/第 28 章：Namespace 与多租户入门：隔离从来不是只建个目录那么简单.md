# 第 28 章：Namespace 与多租户入门：隔离从来不是只建个目录那么简单

> 对应学习计划第 5 周多租户基础。

很多团队第一次接手 Kubernetes 时，喜欢把所有资源全堆在 `default` 命名空间里。刚开始看似省事，过一阵子就会发现：权限不好分、资源不好看、配额不好控、服务名还容易撞。Namespace 的意义，远不只是“多建几个目录”。它是 Kubernetes 里最基础、也最容易被低估的组织与隔离单元。

## 1）项目背景

为什么 Namespace 如此重要？因为集群一旦被多个团队、多个环境、多个系统共享，你立刻就会遇到这些问题：

- 同名资源怎么区分。
- 权限到底该按什么边界分配。
- 配额和默认限制应该落在哪。
- 日志、监控、审计、发布责任怎么归组。

如果所有资源都塞在 `default`，短期看似简单，长期通常意味着：

- 没有清晰的责任边界。
- 配额和限额无法细分。
- 权限控制很难最小化。
- 故障排查时信息混成一锅粥。

Namespace 的核心价值主要体现在三个方面：

- **组织**：把资源按团队、项目、环境分组。
- **隔离**：为配额、RBAC、默认策略提供边界。
- **命名**：允许不同命名空间里存在同名资源。

但一定要注意，Namespace 不是完整安全边界。它提供的是“基础隔离单位”，而不是自动帮你搞定所有网络、权限和秘密隔离。如果把它误解成“建个命名空间就万事大吉”，后面一定会踩坑。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我们团队现在所有资源都放在 `default` 里，好像也还能用。为什么大家都说这样不规范？

**大师**：因为 `default` 只适合最初期实验，不适合长期治理。资源一多，你就会发现什么都缠在一起。

**小白**：Namespace 不就是分个目录吗？

**大师**：如果你把它只看成目录，那就太低估它了。它是很多治理能力的边界：RBAC、配额、默认限制、网络策略，很多都要围着 Namespace 转。

**小白**：所以它其实更像“治理边界单位”？

**大师**：对。它帮你先把“谁和谁是一组”这个问题说清楚。

**小白**：那是不是每个服务都要独立一个 Namespace？

**大师**：也不是。划分粒度要看团队、环境、责任边界和治理复杂度。Namespace 不是越多越好，而是越符合管理边界越好。

**小白**：那它是不是完整安全隔离？

**大师**：不是。它只是隔离的起点，不是终点。网络隔离还要 NetworkPolicy，权限隔离还要 RBAC，敏感信息还要 Secret 与外部管理能力。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们用最简单的方式感受 Namespace 在组织和命名上的价值。

### 第一步：创建多个命名空间

```bash
kubectl create namespace team-a
kubectl create namespace team-b
kubectl get namespaces
```

这一步的意义不是“多几个对象”，而是从此以后你开始以命名空间为单位思考：

- 哪些资源属于哪个团队。
- 哪些规则应该落到哪个范围。
- 哪些权限只给哪个命名空间。

### 第二步：在不同命名空间创建同名资源

比如你可以在 `team-a` 和 `team-b` 里都创建一个名为 `web` 的 Deployment。这样你会马上体会到：资源名称不再必须在整个集群全局唯一，而是可以在隔离边界内独立存在。

### 第三步：理解跨命名空间访问

Service 的 FQDN 常见形式是：

```text
<service>.<namespace>.svc.cluster.local
```

这意味着命名空间也参与了服务寻址。它不仅影响“资源归属”，还影响“服务名字在集群里怎么被解析”。

### 第四步：把 Namespace 放回治理主线

一旦你用了 Namespace，后面很多对象就有了明确落点：

- `ResourceQuota` 和 `LimitRange` 按命名空间生效。
- `Role` 和 `RoleBinding` 主要围绕命名空间授权。
- 多团队发布和审计也更容易分组。

你会发现，Namespace 就像是 Kubernetes 治理的第一层地图，没有它，后面的很多规范都无处安放。

### 第五步：理解它的边界

Namespace 很重要，但不是万能。它不能单独解决：

- 网络访问控制。
- 集群级资源授权。
- 敏感信息外泄。
- 物理级或真正强隔离需求。

因此，它更像“多租户与治理的基础单元”，而不是完整的安全解决方案。

### 这一节应该带走什么

- Namespace 是资源组织、命名和治理的基础边界。
- 它为配额、权限和发布流程提供了天然分组单位。
- 它不是完整安全边界，但它是很多隔离能力的起点。
- `default` 只适合早期实验，不适合长期共享治理。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让多团队、多环境、多项目共享集群时更有秩序。
- 它为资源配额、RBAC、默认规则等治理能力提供了清晰边界。
- 它让资源命名冲突得到天然缓解。

### 缺点

- 过度细分 Namespace 会带来新的管理复杂度。
- 初学者很容易把它误解成“目录”或“完整安全边界”。
- 仅靠 Namespace 本身无法完成所有隔离与治理目标。

### 使用场景

- 团队级、项目级、环境级资源分组。
- 共享集群中的权限、配额和审计边界划分。
- 多租户治理的基础组织模型。

### 注意事项

- Namespace 的划分应围绕责任边界，而不是单纯追求整齐。
- 不要把所有东西长期堆在 `default`。
- 不要误把 Namespace 当成完整的安全隔离方案。

### 常见踩坑经验

- 最常见的坑，是把所有应用长期堆在 `default`，结果后续权限、安全和治理都无从谈起。
- 第二个坑，是把 Namespace 误当成完整的安全边界，忽略了 NetworkPolicy 和 RBAC 等配套能力。
- 第三个坑，是命名空间划分过细或过乱，反而让治理成本失控。

这一章真正想帮你建立的，是组织治理视角：**先把边界划清楚，后面的权限、配额和隔离才有地方落。**
