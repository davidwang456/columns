# 第 35 章：高可用控制平面与 etcd：大脑不能只有一份备份

> 对应学习计划第 6 周高可用控制平面与 etcd。

很多团队第一次谈 Kubernetes 高可用时，注意力几乎都放在业务服务副本上：Deployment 多副本、Service 负载均衡、Pod 故障自动恢复。这些当然重要，但如果控制平面本身是单点，整个集群依然很脆弱。因为业务副本再多，真正调度、存储状态、处理 API 请求的“大脑”一旦失效，平台层照样会陷入瘫痪。高可用控制平面与 etcd 的意义，就是让你从“应用层高可用”进阶到“平台层高可用”的视角。

## 1）项目背景

Kubernetes 控制平面通常包括：

- API Server
- Scheduler
- Controller Manager
- etcd

其中，etcd 特别关键，因为它保存着整个集群的声明状态和关键数据。你可以把它理解成 Kubernetes 的状态数据库；而 API Server 则像整个控制平面的统一入口。只要这两层单点明显，平台风险就会非常集中。

为什么很多团队会误以为“业务多副本 = 集群高可用”？因为业务层的故障更容易被感知，而控制平面问题往往只有在：

- API 请求失败。
- 节点异常后无法重新调度。
- 状态更新无法持久化。
- etcd 损坏或不可用。

这些时刻才会集中爆发。一旦发生，就会让大家意识到：**原来集群本身也需要像业务一样被设计成高可用系统。**

控制平面高可用通常要关注几件事：

- 多控制平面节点，而不是单 Master。
- API Server 前面有稳定入口或负载均衡。
- etcd 至少具备合理仲裁与备份策略。
- 故障时平台能继续提供核心控制能力。

其中，etcd 是最不能掉以轻心的一层。因为哪怕控制平面组件还能重启，只要 etcd 数据损坏或不可用，整个平台状态都会陷入严重问题。

所以，本章真正想建立的是一个平台层判断：**高可用不只是业务副本数，更是控制平面和状态存储的持续可用能力。**

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我的应用都做成 3 副本了，是不是已经算高可用了？

**大师**：只能说业务层更抗打了，不代表集群本身就高可用。平台层如果还是单点，很多故障仍然扛不住。

**小白**：可控制平面挂了，已经跑着的应用不是还在吗？

**大师**：短时间也许还在，但你会失去很多关键能力：新的调度、状态更新、故障恢复、滚动发布、节点控制，这些都会受影响。

**小白**：那 etcd 为什么总被单独强调？

**大师**：因为它存的是整个集群的大脑记忆。API Server 可以重启，控制器可以恢复，但如果状态数据库出问题，平台就不是简单“重启一下”能解决的事了。

**小白**：所以高可用控制平面的重点，不只是多几个组件副本，而是要保证状态和入口都稳？

**大师**：完全正确。控制平面高可用，本质上是入口高可用 + 状态高可用 + 组件冗余一起成立。

**小白**：那是不是可以理解成，业务高可用是“服务不挂”，平台高可用是“系统还能继续治理服务”？

**大师**：这就是最关键的区别。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们不做超大规模实验，而是通过 etcd 健康检查和控制平面视角，建立最基本的平台高可用认知。

### 第一步：先理解 etcd 健康检查命令

```bash
ETCDCTL_API=3 etcdctl endpoint health   --endpoints=https://127.0.0.1:2379   --cacert=/etc/kubernetes/pki/etcd/ca.crt   --cert=/etc/kubernetes/pki/etcd/server.crt   --key=/etc/kubernetes/pki/etcd/server.key
```

第一次看到这类命令时，不用急着背参数，先理解它在回答什么问题：**控制平面的状态存储，现在到底健不健康。**

### 第二步：把控制平面高可用想成三层

学习阶段可以先把它拆成三层：

1. **入口层**：API Server 是否有稳定入口，是否单点。
2. **组件层**：控制器和调度器是否有冗余运行能力。
3. **状态层**：etcd 是否有健康仲裁、备份与恢复能力。

只要这三层里任何一层单点明显，高可用就不完整。

### 第三步：理解 etcd 的特殊性

etcd 不是“又一个普通组件”，它是集群状态的核心事实来源。因此，对 etcd 来说，最重要的通常不是“多副本启动了没”，而是：

- 健康是否正常。
- 仲裁是否成立。
- 备份是否可靠。
- 故障恢复路径是否被演练过。

很多团队把 etcd 备份当成运维附属项，直到第一次真实故障才发现“有无备份”根本不是细节，而是平台生死线。

### 第四步：建立对“平台高可用”和“业务高可用”的区别

这是本章最想让你彻底建立的视角差异：

- 业务高可用：流量还能被服务、实例还能扛故障。
- 平台高可用：即使控制平面组件局部故障，集群仍能继续管理、调度、存储和恢复状态。

这两者当然有关联，但绝不是同一件事。

### 第五步：学习阶段最该带走的实践习惯

先养成两个习惯，比一开始就搭完整 HA 环境更重要：

- 看到“高可用”三个字时，先问控制平面是不是单点。
- 看到 etcd 时，先问有没有健康检查、备份和恢复方案。

只要这两个问题开始成为习惯，你对 Kubernetes 平台层的理解就已经迈过了一个很大的门槛。

### 这一节应该带走什么

- 业务多副本不等于集群高可用。
- 控制平面高可用至少包含入口、组件和状态三层。
- etcd 是整个平台状态的核心，不是普通附属组件。
- 高可用的关键不是“看起来冗余”，而是故障时平台是否还能继续工作。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它帮你把高可用视角从业务层提升到平台层。
- 它能显著提高集群在控制平面故障时的持续治理能力。
- 它让 etcd、API Server 和控制平面入口的风险不再被忽视。

### 缺点

- 架构复杂度和运维成本明显高于单控制平面集群。
- 学习者很容易停留在“组件多副本”表面，而忽略真正的状态与恢复问题。
- 如果没有备份和演练，高可用设计很容易流于纸面。

### 使用场景

- 生产级 Kubernetes 平台建设。
- 多团队依赖、业务持续运行要求高的集群。
- 平台团队需要建立控制平面故障恢复能力的场景。

### 注意事项

- 不要把高可用仅理解成“应用副本数变多”。
- etcd 的健康、备份、恢复必须被单独认真对待。
- 控制平面设计要同时关注冗余、入口和状态一致性。

### 常见踩坑经验

- 最常见的坑，是只做应用多副本，不做控制平面冗余与备份。
- 第二个坑，是把 etcd 当成“默认就没事”的后台组件，从不检查也不演练恢复。
- 第三个坑，是高可用方案只停留在架构图上，真正出故障时没有任何经过验证的恢复路径。

这一章真正想帮你建立的，是平台高可用视角：**高可用不是副本数游戏，而是平台在故障面前是否还能持续维护系统状态。**
