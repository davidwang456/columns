# 第 07 章：资源请求与限制：调度器看的是请求，内核执行的是限制

> 对应学习计划第 1 周资源限制与第 3 周 QoS 主题。

很多人第一次给 Pod 配资源时，会直觉地觉得“限制越严越安全”。于是 CPU、内存上来就随便填一个值，或者干脆只配 `limits` 不配 `requests`。结果不是调度混乱、资源浪费，就是线上服务被 OOM Kill。这一章要解决的，就是 Kubernetes 资源模型里最容易被误解的一句话：**调度器看的是请求，运行时真正执行的是限制。**

## 1）项目背景

如果没有资源声明，Kubernetes 就像一个“大家都说自己只占一点，但没人真正报预算”的共享空间。调度器不知道一个 Pod 需要多少 CPU 和内存，只能在节点上盲目放置；业务之间也无法形成明确的资源边界。规模一小还看不出问题，一旦服务增多，资源竞争、节点挤爆、应用被杀、性能抖动就会接连出现。

Kubernetes 通过 `resources.requests` 和 `resources.limits` 来解决这个问题：

- `requests`：告诉调度器“我至少需要这么多资源”，也是调度是否能放得下的依据。
- `limits`：告诉运行时“我最多只能用到这么多”，超过之后会触发限制或被系统杀掉。

这两个字段看似只是数字，实际分别作用于两套完全不同的阶段：

- 调度阶段：决定 Pod 能否被放到某台节点上。
- 运行阶段：决定容器运行时是否会被限流或被 OOM 杀死。

为什么这点这么重要？因为很多事故都来自错误心智。比如：

- 只写 `limits` 不写 `requests`，结果调度器以为 Pod 几乎不占资源，节点被塞爆。
- `requests` 写得远高于真实需求，结果资源长期浪费，扩容成本被抬高。
- `limits.memory` 写得太低，业务稍一波动就被 OOM Kill。
- `limits.cpu` 写得太低，应用不是挂，而是变得非常慢，开发者误以为是代码问题。

更进一步，Kubernetes 还会根据 `requests` 和 `limits` 的配置组合，把 Pod 分成不同的 QoS（Quality of Service）等级：

- `Guaranteed`
- `Burstable`
- `BestEffort`

这决定了在资源紧张、驱逐发生时，谁更容易被保下来，谁更容易被先清出去。也就是说，资源声明不仅影响性能，还影响“谁能活到最后”。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我理解资源限制就是给容器设个上限，防止它吃太多，是这样吗？

**大师**：只说对了一半。`limits` 确实是上限，但 Kubernetes 的资源模型不是只看上限，它还有 `requests`。

**小白**：那 `requests` 是干什么的？

**大师**：给调度器看的。它告诉调度器：“如果你想把我放到某台节点上，至少要先给我留出这么多资源。”

**小白**：那如果我只写 `limits`，不写 `requests` 呢？

**大师**：通常会带来很糟糕的结果。因为调度器会低估你的真实需求，节点上可能被塞进太多 Pod。运行时再叠加资源竞争，问题就爆发了。

**小白**：原来调度阶段和运行阶段是分开的。

**大师**：对。调度器做的是“账面分配”，内核和容器运行时做的是“实际执行”。你得同时照顾这两层逻辑。

**小白**：那 QoS 又是什么？

**大师**：你可以把它理解成共享集群里的资源舱位等级。资源压力来了，不同等级的 Pod 命运不一样。配置更明确、约束更清晰的工作负载，通常更容易被平台优先保护。

**小白**：那是不是把 requests 和 limits 都写成一样，就最好？

**大师**：也不一定。这样会进入 `Guaranteed`，稳定性更高，但资源弹性更低。要不要这么做，得看业务对抖动和成本的敏感度。

**小白**：听起来资源配置其实是性能、稳定性和成本之间的平衡。

**大师**：没错。资源字段从来不是“顺手填一下”，而是非常重要的架构表达。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们创建一个带资源声明的 Pod，并观察它的调度、运行和 QoS 分类。

### 第一步：编写带资源配置的 Pod

新建 `resource-demo.yaml`：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: resource-demo
spec:
  containers:
  - name: app
    image: nginx:1.27
    resources:
      requests:
        cpu: "250m"
        memory: "256Mi"
      limits:
        cpu: "500m"
        memory: "512Mi"
```

这份配置表达的是：

- 调度器至少为这个容器预留 `250m CPU` 和 `256Mi` 内存。
- 运行时允许它最多使用 `500m CPU` 和 `512Mi` 内存。

### 第二步：创建并观察资源信息

```bash
kubectl apply -f resource-demo.yaml
kubectl describe pod resource-demo
kubectl get pod resource-demo -o yaml
```

在 `describe` 输出里，重点关注：

- Requests 和 Limits 是否被正确识别。
- Pod 是否顺利被调度到某台节点。
- 是否有资源不足相关事件。

### 第三步：查看 QoS 分类

```bash
kubectl get pod resource-demo -o jsonpath='{.status.qosClass}'
```

你大概率会看到 `Burstable`。原因是：requests 和 limits 都配了，但不完全相等。你可以继续对比三种情况：

- 什么都不配：`BestEffort`
- requests 和 limits 都配，但不相等：`Burstable`
- requests 和 limits 完全相等：`Guaranteed`

通过这个实验，你会直观理解 QoS 不是额外配置项，而是资源声明组合后自然得出的结果。

### 第四步：理解 CPU 和内存限制的差异

这是新手很容易忽略的一点：

- CPU 超限通常表现为“被限速、变慢”，不一定直接崩。
- 内存超限往往更直接，可能触发 OOM Kill。

这意味着，CPU limits 配得太小，应用可能不是报错，而是吞吐下降、延迟升高；内存 limits 配得太小，应用可能直接被杀。两者带来的故障现象完全不同。

### 第五步：形成一条资源调优思路

学习阶段可以先这样做：

1. 根据应用压测或历史监控，估出一个保守的 requests。
2. limits 不要离真实峰值太近，给业务留出短暂波动空间。
3. 配完之后再结合 `kubectl top pod`、监控图和 OOM 事件逐步调优。

资源配置不是一次性写完就结束，而是一个随着业务观察不断修正的过程。

### 这一节应该带走什么

- `requests` 主要影响调度，`limits` 主要影响运行时约束。
- QoS 是资源配置组合后的结果，会影响资源紧张时的存活优先级。
- CPU 和内存超限的表现不同，排障思路也不同。
- 资源配置不是装饰字段，而是稳定性和成本管理的核心手段。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让平台能更准确地调度工作负载，提高资源利用率和可预测性。
- 它帮助团队为不同业务建立明确的性能和稳定性边界。
- 它与 QoS、配额、弹性伸缩等机制天然联动，是资源治理的基础。

### 缺点

- 配置门槛高，既要理解调度逻辑，也要理解运行时限制机制。
- 估值不准时，容易在浪费资源和引发故障之间反复摇摆。
- 没有监控和压测支撑时，很多值只能靠经验猜，容易失真。

### 使用场景

- 共享集群中的常规业务工作负载。
- 对性能和稳定性有明确要求的服务。
- 需要做资源治理、成本控制和优先级区分的团队。

### 注意事项

- 不要只配 `limits` 不配 `requests`。
- 不要把线上故障全都归咎于代码，也要检查 CPU throttle 和 OOM。
- 配置资源要基于观察与监控，而不是想当然。

### 常见踩坑经验

- 最常见的坑，是只配置 `limits` 不配置 `requests`，导致调度阶段失真。
- 第二个坑，是把内存 limits 压得过低，结果服务一有波动就 OOM。
- 第三个坑，是看见 CPU 利用率不高就盲目降低请求值，结果高峰期调度和性能一起出问题。

这一章真正要建立的，不只是几个资源字段的认知，而是一种平台视角：**资源不是“剩多少就用多少”，而是必须被明确预算、调度和约束的共享资产。**
