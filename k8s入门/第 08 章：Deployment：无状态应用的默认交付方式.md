# 第 08 章：Deployment：无状态应用的默认交付方式

> 对应学习计划第 1 周 Day 5-6 的 Deployment、滚动更新与回滚。

学完 Pod 之后，很多新手都会很自然地问一句：既然 Pod 已经能跑业务了，为什么还要 Deployment？答案很简单，因为**Pod 只会代表一个实例，而不会替你长期维护业务的发布、扩容、自愈和版本演进。** 只要业务进入“多副本、持续上线、需要自动恢复”的阶段，直接管理 Pod 就会迅速失控。Deployment 正是为此而生。

## 1）项目背景

在真实生产环境里，无状态服务最常见的诉求通常有四个：

- 我希望某个服务始终保持若干个副本在线。
- 我希望发布新版本时可以逐步替换旧实例，而不是全部停掉重来。
- 我希望出问题时能够快速回滚。
- 我希望某个 Pod 挂了之后，平台自动补回来，而不是靠人盯着重启。

如果只用裸 Pod，这些需求都很难优雅解决。因为 Pod 本质上只是一个实例对象：

- Pod 被删了就没了。
- Pod 不知道“我应该有几个兄弟”。
- Pod 不负责版本变更与滚动替换。
- Pod 也不擅长表达“我想要的长期状态”。

Deployment 则不一样。它不是在描述“某个具体容器现在怎么跑”，而是在描述“这类 Pod 应该长期维持成什么样子”。它把副本数、标签选择器、Pod 模板、更新策略都统一纳入一个对象里，让控制器持续去维护这些目标。

也正因为这样，Deployment 成了 Kubernetes 里最经典、最常用、最适合无状态业务的工作负载控制器。你可以把它理解成“无状态服务的标准发布接口”。只要服务不依赖稳定身份、不依赖固定存储卷、不要求严格有序启动，大多数场景都优先从 Deployment 开始考虑。

Deployment 的学习重点不是把 YAML 字段背下来，而是搞清楚它背后的平台价值：**它把发布这件原本脆弱、重复、依赖人工经验的事情，变成了平台可持续维护的对象状态。**

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我已经会创建 Pod 了，那我是不是多复制几个 Pod YAML，就等于完成多副本部署了？

**大师**：你可以这么做一次，但你不会想长期这么维护。因为你管理的将变成一堆散落实例，而不是一个受控的服务状态。

**小白**：可我只要保证有 3 个 Pod 不就行了？

**大师**：问题在于“谁来保证”。你手工建 3 个，只是在某个时刻凑出了 3 个；Deployment 则是在持续保证“始终应该有 3 个”。

**小白**：那它和 Pod 最大的区别就是“持续维护”？

**大师**：没错。它不仅维护副本数，还负责版本升级、回滚和滚动替换。你可以把它看成应用发布的总负责人。

**小白**：那是不是所有服务都应该用 Deployment？

**大师**：不是。它适合无状态服务。如果你的业务依赖固定名字、固定卷、固定顺序，那应该考虑 StatefulSet。Deployment 的强项，是“可替换实例的大规模管理”。

**小白**：那我是不是可以理解成：Pod 更像士兵，Deployment 更像连长？

**大师**：这个比喻很不错。Pod 负责真正执行工作，Deployment 负责人数、替换、版本和编组策略。

**小白**：那我学 Deployment 时最重要的是看什么？

**大师**：三件事：副本如何维护、版本如何更新、失败如何回滚。只要抓住这三条主线，你对无状态工作负载就入门了。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们从一个最小 Deployment 开始，完成三件事：创建三副本、升级镜像、观察滚动更新。

### 第一步：编写最小 Deployment

新建 `web-deployment.yaml`：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: nginx
        image: nginx:1.27
        ports:
        - containerPort: 80
```

这里最值得关注的是两部分：

- `replicas: 3`：表示我期望长期有 3 个副本。
- `template`：表示未来所有 Pod 都应该按照这个模板创建。

### 第二步：应用并观察副本维护

```bash
kubectl apply -f web-deployment.yaml
kubectl get deployment web
kubectl get pods -l app=web -w
```

你会看到 3 个 Pod 被逐步创建出来。此时可以再做一个非常经典的实验：删除其中一个 Pod。

```bash
kubectl delete pod <pod-name>
kubectl get pods -l app=web -w
```

你会发现 Pod 被删掉之后，很快会有新 Pod 被补回来。这正是 Deployment 的核心价值之一：**业务副本数不再依赖人工维持，而是交给控制器持续纠偏。**

### 第三步：升级镜像，观察滚动更新

```bash
kubectl set image deployment/web nginx=nginx:1.28
kubectl rollout status deployment/web
kubectl get pods -l app=web -w
```

此时你应该观察到：

- 新 Pod 逐渐创建。
- 旧 Pod 逐渐终止。
- 整个过程不是“一刀切”，而是渐进替换。

这就是滚动更新的直观表现。它的目标是在尽量不影响服务可用性的前提下完成版本切换。

### 第四步：查看历史与回滚

```bash
kubectl rollout history deployment/web
kubectl rollout undo deployment/web
```

这一步很重要，因为它说明 Deployment 不是“只会向前”的发布工具，它同样重视失败后的恢复能力。一个真正可用的平台，不是从不出错，而是**出错之后能快速、清晰、低风险地回到稳定状态。**

### 第五步：形成一个无状态发布心智

看到这里，你可以开始总结 Deployment 的适用边界：

- 适合：Web 服务、API 服务、无状态后台任务执行器。
- 不适合：依赖固定身份、固定卷和严格顺序的数据库、注册中心、消息队列节点。

不要因为 Deployment 最常见，就把所有服务都往它上面套。对象之所以存在，是因为它们各自解决不同问题。

### 这一节应该带走什么

- Deployment 维护的是“无状态服务的长期目标状态”。
- 它负责副本维持、滚动更新和回滚。
- 删除一个 Pod，不等于服务实例永久减少，因为控制器会自动补齐。
- Deployment 是无状态应用的默认交付方式，但不是一切工作负载的答案。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它把无状态应用的发布、扩缩容和恢复统一抽象成标准对象。
- 它支持滚动更新与回滚，是现代交付流程的核心基础。
- 它和 Service、HPA、监控告警等机制天然适配，便于组合成完整发布链路。

### 缺点

- 只适用于无状态服务，边界一旦理解错，很容易误用。
- 新手容易只会创建，却不理解滚动更新、历史版本和底层控制关系。
- 如果探针、资源、标签设计不当，再好的 Deployment 也无法稳定运行。

### 使用场景

- 前端静态服务、网关、API 服务等典型无状态应用。
- 需要多副本、滚动更新和快速回滚的业务。
- 希望接入自动扩缩容和统一服务暴露能力的团队。

### 注意事项

- `selector` 和 `template.labels` 必须一致，否则控制器无法正确管理 Pod。
- Deployment 擅长管理“可替换实例”，不要强行拿它管理稳定身份需求。
- 发布时不仅要看对象创建成功，还要观察 rollout 是否完成、Pod 是否 Ready。

### 常见踩坑经验

- 最常见的坑，是把有状态服务按 Deployment 的心智部署，结果升级和恢复都很痛苦。
- 第二个坑，是标签写错，导致 Deployment 找不到或误选了 Pod。
- 第三个坑，是只会 `apply`，却不会看 `rollout status` 和历史版本，一出问题就只能手忙脚乱地猜。

这一章真正建立的，是你对无状态交付的默认心智：**如果服务实例可以被替换、可以横向扩展、版本需要平滑演进，那么先从 Deployment 想起。**
