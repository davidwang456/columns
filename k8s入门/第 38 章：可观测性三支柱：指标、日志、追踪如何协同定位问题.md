# 第 38 章：可观测性三支柱：指标、日志、追踪如何协同定位问题

> 对应学习计划第 7 周监控、日志与追踪体系。

如果说 Kubernetes 前面的章节都在回答“系统怎么部署、怎么运行、怎么治理”，那么可观测性真正回答的则是另一个更现实的问题：**一旦系统出问题，你靠什么知道它到底哪里出了问题。** 没有可观测性，再复杂的平台能力都像在黑箱里碰运气；而只有单一观测信号，又很难在复杂故障中快速逼近根因。指标、日志、追踪之所以被称为三支柱，正是因为它们分别回答了不同层次的问题。

## 1）项目背景

很多团队对可观测性的理解一开始都过于单一：

- 有图表就算监控。
- 能看日志就算排障。
- 引了链路追踪就算先进。

真实问题是，这三者各自解决的问题不同：

- **指标** 更像“生命体征”，告诉你哪里在异常、容量是否健康。
- **日志** 更像“细节记录”，告诉你具体发生了什么。
- **追踪** 更像“请求轨迹”，告诉你一个调用链路到底卡在哪。

如果只盯其中一种信号，常见问题会非常明显：

- 只有指标：你知道延迟升高，但不知道哪条请求出了问题。
- 只有日志：你能看到报错文本，但不知道它在整个链路中的位置和影响范围。
- 只有追踪：你看到请求慢，但不知道是不是整体容量已经逼近极限。

这就是为什么 Kubernetes 平台化之后，可观测性必须作为体系来看，而不是单个工具安装任务。真正成熟的平台观测体系，至少要解决这些问题：

- 集群和节点是否健康。
- 工作负载资源使用是否异常。
- 业务请求路径卡在哪里。
- 故障发生时，能不能从症状快速收缩到根因。

因此，本章最重要的不是背 Prometheus、ELK、Jaeger 的工具名字，而是建立一种分层感：**指标负责看整体，日志负责看细节，追踪负责看路径。**

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我已经接了日志采集，是不是就算有可观测性了？

**大师**：还差得远。日志只能回答一部分问题，它更像事后记录，不等于你对系统整体运行有实时感知。

**小白**：那监控图表呢？我有 CPU、内存和延迟图。

**大师**：这属于指标层，很重要，但它更多是在告诉你“哪里可能出问题了”，不一定能直接给你根因。

**小白**：那追踪就是用来找根因的？

**大师**：它更擅长看请求怎么流过系统、卡在了哪一跳。尤其微服务一多，没有追踪就很容易只看到局部。

**小白**：所以三支柱不是为了看起来高级，而是因为每个都在解决不同层次的问题。

**大师**：完全正确。只装工具不理解分工，最后往往只是多了几套界面，而不是多了真正的排障能力。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先从最主流、最容易起步的指标体系开始，用 `kube-prometheus-stack` 建一个最小监控闭环。

### 第一步：安装指标监控栈

```bash
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm install monitor prometheus-community/kube-prometheus-stack -n monitoring --create-namespace
```

这一步的价值不是“又装一套工具”，而是让你第一次把以下能力统一接进平台：

- 集群级指标采集
- 节点与工作负载资源观测
- 告警能力的基础骨架
- 仪表板可视化入口

### 第二步：理解“指标层”的位置

安装好之后，你应该优先关注的是：

- 节点 CPU / 内存使用率
- Pod 资源使用
- Restart、Pending、异常副本变化
- API Server / 核心组件状态

这会帮助你建立一个“先看生命体征”的排障习惯。很多时候，故障还没到业务报错那一步，指标层已经先给出信号了。

### 第三步：理解日志和追踪为什么不能缺席

即使指标告诉你某个服务延迟飙升，它仍然回答不了：

- 具体是哪条请求在报错。
- 是代码异常、依赖异常，还是某个参数场景导致。
- 一个请求在多个服务之间到底耗时分布如何。

这时就轮到：

- 日志去看细节。
- 追踪去看路径。

所以，可观测性真正的威力不是单点工具，而是“先用指标发现，再用日志细看，再用追踪串联”。

### 第四步：为什么“只堆工具”是很危险的假象

很多平台团队会掉进一个很典型的坑：工具全装了，界面很多，但没有统一口径，没有告警策略，没有关键指标定义。结果就是：

- 图很多，但没人知道该看哪张。
- 日志很多，但没有结构化字段。
- 追踪很多，但没有采样策略和关键链路。

这说明可观测性不是“工具采购”，而是“观测设计”。

### 第五步：形成一条最实用的排障路径

学习阶段最推荐固定练习这条顺序：

1. 先看指标，确认问题范围和严重度。
2. 再看日志，确认具体错误与上下文。
3. 最后看追踪，确认链路耗时和依赖位置。

只要这条路径开始成为习惯，你对复杂系统故障的把握能力就会明显提升。

### 这一节应该带走什么

- 指标、日志、追踪分别回答整体、细节和路径三个层次的问题。
- 观测体系必须作为整体设计，而不是单工具安装。
- 指标往往是第一信号，日志和追踪负责把问题继续收窄。
- 可观测性的价值不在“工具多”，而在“问题能否被系统性定位”。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让排障、容量管理和用户体验分析有了系统化基础。
- 它帮助团队从“靠经验猜问题”升级到“基于证据定位问题”。
- 它是平台稳定性、发布安全性和运维效率的重要支撑。

### 缺点

- 工具链多、概念多、成本高，初学阶段容易被界面和名词淹没。
- 如果没有统一口径和治理，工具越多反而越乱。
- 数据量、存储成本和告警噪音都需要长期管理。

### 使用场景

- 生产和准生产集群的集群/工作负载观测。
- 微服务系统的故障排查、容量分析和性能追踪。
- 平台团队建立统一监控告警与排障路径时。

### 注意事项

- 先定义关键指标和关键链路，再选工具和面板。
- 不要把“装上了工具”误当成“拥有了可观测性”。
- 日志、指标、追踪要按分工协同设计，而不是孤立堆叠。

### 常见踩坑经验

- 最常见的坑，是只堆工具，不定义指标口径和告警策略。
- 第二个坑，是只有日志没有指标，或者只有指标没有日志，导致排障总断在半路。
- 第三个坑，是追踪接入了，但没有关键链路和采样策略设计，最后几乎没人真正使用。

这一章真正想帮你建立的，是观测分层视角：**系统可观测，不是因为你装了多少工具，而是因为你能沿着信号把问题从症状一路追到根因。**
