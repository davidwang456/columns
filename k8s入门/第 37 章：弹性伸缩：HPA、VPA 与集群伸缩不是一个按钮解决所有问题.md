# 第 37 章：弹性伸缩：HPA、VPA 与集群伸缩不是一个按钮解决所有问题

> 对应学习计划第 7 周性能优化与自动扩缩容。

一旦业务开始出现明显的流量波峰波谷，平台团队很快就会面对一个现实问题：副本数到底该手工调，还是交给系统动态调？手工调当然能工作，但它通常意味着两个坏结果同时出现: 平峰期浪费资源，高峰期又来不及扩。弹性伸缩的意义，就是让容量不再只依赖人工判断，而是逐步根据指标、资源和集群能力动态变化。可惜的是，很多人一学到 HPA，就误以为“弹性”这件事已经结束了。实际上，**HPA、VPA、集群伸缩解决的是三个不同层面的问题。**

## 1）项目背景

为什么“弹性”不是一个按钮能解决？因为现实中的伸缩至少涉及三层：

- **副本层**：我需不需要更多 Pod 来承接流量。
- **资源层**：单个 Pod 的 CPU/内存配置是不是合理。
- **集群层**：节点本身够不够，能不能继续承载新增 Pod。

这三层如果混在一起，就会出现很多常见误解：

- 只上 HPA，却不看 requests/limits 配得是否合理。
- CPU 指标一高就扩 Pod，但真正瓶颈可能是数据库或锁竞争。
- 副本数已经能扩，可节点不够，Pod 全在 Pending。
- 把 HPA 当成性能优化替代品，应用本身性能问题却一直没解决。

因此，本章真正想建立的不是“怎么写一个 HPA”，而是对弹性体系的分层认识：

- `HPA`（Horizontal Pod Autoscaler）主要解决副本横向伸缩。
- `VPA`（Vertical Pod Autoscaler）主要解决单实例资源建议或调整。
- 集群伸缩（如 Cluster Autoscaler）主要解决节点容量随调度压力变化的问题。

只有把这三层区别开，弹性体系才不会变成“一遇到波动就盲目扩容”的机械反应。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我的服务一到高峰期就顶不住，是不是直接上 HPA 就行？

**大师**：HPA 很重要，但它不是万能按钮。你得先搞清楚，问题到底是副本不够、单 Pod 资源太小，还是节点根本不够。

**小白**：这三个不是一回事吗？

**大师**：不是。副本不够是横向问题，资源太小是纵向问题，节点不够是集群容量问题。三者看起来都叫“扩容”，本质却不同。

**小白**：那 HPA 主要看什么？

**大师**：最常见的是 CPU 或内存指标，但你不能因为看得到 CPU 就默认它一定是最好的伸缩依据。

**小白**：那 VPA 呢？

**大师**：它更像在回答“这个 Pod 本身应该配多大资源”，而不是“要不要多来几个副本”。

**小白**：所以 HPA 更像加人，VPA 更像给人换更大的工位和设备？

**大师**：这个比喻挺好。再往外一层，集群伸缩就是“办公楼要不要多开一层”。

**小白**：明白了。弹性不是一个对象，而是一整套容量调整体系。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先用最经典的 HPA 场景入手，因为它是大多数团队接触弹性的第一站。

### 第一步：创建一个基于 CPU 的 HPA

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

这段配置表达的是：

- 目标对象是 `Deployment/web`
- 副本数最少 2，最多 10
- 当平均 CPU 使用率超过 70% 时，倾向于扩容

### 第二步：理解它为什么依赖资源声明

HPA 之所以不能孤零零理解，是因为 CPU 利用率的判断往往依赖于 requests。也就是说，如果你的资源请求本身写得非常离谱，那么 HPA 看到的“利用率”也会失真。这里正好把前面资源管理章节和弹性伸缩连接起来了。

### 第三步：理解 HPA 不是性能优化替代品

很多人一看到 CPU 高，就第一反应扩副本。但你必须问自己：

- 这是 CPU 真正瓶颈，还是因为代码效率太差？
- 扩 Pod 之后，下游数据库顶得住吗？
- 请求是否天然可以横向拆散？

如果答案没想清楚，HPA 很容易把成本放大，却并没有真正解决性能瓶颈。

### 第四步：建立对 VPA 和集群伸缩的层次感

初学阶段不用一头扎进所有细节，但至少要建立一个清晰区分：

- HPA：副本数变多或变少。
- VPA：单 Pod 资源建议或调整。
- 集群伸缩：节点层面增减容量。

这三者的组合，才构成比较完整的弹性体系。只学 HPA 而不理解其他两层，后面很容易碰到“想扩但扩不上去”或“越扩越贵还没效果”的问题。

### 第五步：形成一条弹性设计原则

一个成熟的弹性设计，通常至少会考虑：

1. 指标是否真的反映业务压力。
2. 应用是否适合横向扩容。
3. 资源配置是否合理。
4. 节点容量是否能跟上。

只要这四个问题开始成为思考习惯，你就已经不再是“会写 HPA”，而是在真正做容量治理。

### 这一节应该带走什么

- 弹性至少分副本层、资源层、集群层三层。
- HPA 解决横向伸缩，不是万能弹性总开关。
- 资源配置不合理时，HPA 的判断也会失真。
- 扩容从来不是性能优化替代品，而是容量治理手段。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让容量规划从静态配置升级为动态调节，能更好匹配波动业务。
- 它帮助团队把人工经验逐步沉淀为平台化伸缩能力。
- 理解弹性分层后，后续的性能、成本和容量治理都会更清晰。

### 缺点

- 初学者很容易把 HPA、VPA、集群伸缩混成一个概念。
- 指标设计错误时，自动伸缩会把错误放大得更快。
- 如果应用本身不适合横向扩容，HPA 也帮不上忙。

### 使用场景

- 流量波峰波谷明显的无状态服务。
- 对资源成本和容量弹性有明确要求的系统。
- 希望减少人工调副本操作的团队。

### 注意事项

- 先确认应用是否真的适合横向扩容。
- 不要把 HPA 当成性能优化替代品。
- 伸缩指标、资源声明和节点容量必须一起看。

### 常见踩坑经验

- 最常见的坑，是把 HPA 当成性能优化的替代品，应用本身效率问题却一直没解决。
- 第二个坑，是 requests/limits 配置失真，导致 HPA 看到的利用率判断不可信。
- 第三个坑，是副本数能扩，但节点容量跟不上，结果大量 Pod Pending。

这一章真正想帮你建立的，是容量治理视角：**弹性不是“会自动扩”，而是系统在负载变化面前是否能有节制地调整容量。**
