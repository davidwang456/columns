# 第 04 章：Pod：Kubernetes 中最小但不简单的调度单元

> 对应学习计划第 1 周 Day 3-4 的 Pod 深入理解。

如果你只能记住 Kubernetes 的一个最核心对象，那一定是 `Pod`。因为几乎所有工作负载控制器，最终都是在管理 Pod；几乎所有服务发现、配置注入、资源限制、探针检查，也最终都落在 Pod 这个运行边界上。可是，Pod 又恰恰是最容易被误解的对象。很多初学者会直接把它等同于“一个容器”，这会让后面几乎所有知识都学偏。

## 1）项目背景

在 Docker 的世界里，我们习惯直接把“运行中的容器”当成最小单位：一个容器就是一个进程实例，容器里有镜像、端口、环境变量、挂载卷，已经够我们描述很多应用场景。但 Kubernetes 并没有把“容器”作为直接调度单位，而是专门引入了 Pod。为什么？

答案很简单：**平台需要一个更完整的运行边界**。一个真实的应用实例，除了业务进程本身，往往还需要共享网络、共享存储、共享生命周期管理，甚至还会有多个紧密协作的容器共同组成一个运行单元。Kubernetes 希望调度的不是孤零零的单个容器，而是一个“可以整体迁移、整体重建、整体管理”的最小单元，这个单元就是 Pod。

理解 Pod 时，最重要的是下面几个事实：

- 一个 Pod 里可以有一个或多个容器。
- 同一个 Pod 内的容器共享网络命名空间，也就是共享 IP 和端口空间。
- 同一个 Pod 内的容器可以共享卷。
- Pod 被调度时，是整体调度到某个节点上，不会拆分。

这意味着，Pod 不是“容器的别名”，而是**容器协作与资源调度的包裹层**。这层抽象非常重要，因为它既让 Kubernetes 能管理多容器协作，又避免开发者直接和底层容器运行时耦合得太深。

你还需要认识到：Pod 往往不是长期稳定身份的载体。大多数情况下，Pod 更像“可替换的实例”，真正长期稳定的是更上层的控制器，比如 Deployment 或 StatefulSet。也正因为如此，我们在学习 Pod 的时候，既要理解它的运行边界，也要知道为什么裸 Pod 很少直接作为正式业务交付方式。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我看 Pod 里不就是装容器吗？那我是不是可以把 Pod 理解成“容器换了个名字”？

**大师**：这正是最常见的误解。Pod 里确实装容器，但它不是容器的简单别名，而是 Kubernetes 选择的最小调度单位。

**小白**：为什么不能直接调度容器？

**大师**：因为平台管理的不是单个进程本身，而是一个运行边界。比如两个容器要共享网络、共享卷、共享生命周期，那调度时就应该一起处理，而不是拆开。

**小白**：也就是说，Pod 是“容器组合的运行外壳”？

**大师**：可以这么理解。你可以把它看成一个逻辑上的小机器。这个“小机器”有自己的 IP，有自己的卷，有自己的生命周期。里面可以放一个主业务容器，也可以放配套容器。

**小白**：那为什么很多文章又说一个 Pod 通常只放一个主容器？

**大师**：因为多容器并不代表越多越好。Pod 内多个容器必须强耦合，强到值得共享同一个运行边界。如果只是普通服务之间调用，完全没必要塞到同一个 Pod。

**小白**：那我能不能直接长期维护一个裸 Pod？

**大师**：实验可以，生产通常不建议。因为 Pod 天生是容易被替换的实例，不擅长承担版本发布、滚动升级、自动补副本这些职责。那是更上层控制器的工作。

**小白**：所以学习 Pod，既要理解它是什么，也要理解它不负责什么。

**大师**：对。学对象最怕只看“能干什么”，不看“边界在哪”。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们创建一个最基础的 Pod，然后沿着它的生命周期走一遍：创建、调度、运行、观察、删除。你会发现，Pod 是后面所有对象的共同基础。

### 第一步：创建一个最小 Pod

新建 `basic-pod.yaml`：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: basic-pod
  labels:
    app: basic-pod
spec:
  containers:
  - name: app
    image: nginx:1.27
    ports:
    - containerPort: 80
```

应用配置：

```bash
kubectl apply -f basic-pod.yaml
kubectl get pod basic-pod -w
```

这里建议你盯着 `-w` 的输出看几分钟，观察 Pod 从 `Pending` 变成 `ContainerCreating` 再进入 `Running`。这个过程比死记生命周期定义更有帮助，因为它让你直接看到了“对象状态是如何流动的”。

### 第二步：查看 Pod 被调度到了哪台节点

```bash
kubectl get pod basic-pod -o wide
kubectl describe pod basic-pod
```

你应该重点看下面几类信息：

- `Node`：Pod 最终落在哪台节点。
- `IP`：Pod 获得的集群内地址。
- `Containers`：容器镜像和启动信息。
- `Events`：调度、拉镜像、启动失败等事件。

通过这一步，你会开始体会“Pod 是调度边界”这句话的实际含义。容器不是自己决定去哪台机器，而是 Pod 整体被调度到某个节点上。

### 第三步：进入容器和查看日志

```bash
kubectl logs basic-pod
kubectl exec -it basic-pod -- /bin/sh
```

如果镜像支持 shell，你可以进入容器查看文件和进程。这一步的意义不只是“看一眼容器内容”，更重要的是你会知道：**Pod 的排障入口，通常从日志、事件和运行环境三件套开始。**

### 第四步：验证 Pod 是可替换实例

```bash
kubectl delete pod basic-pod
```

此时因为它是一个裸 Pod，被删掉之后不会自动补回来。这恰好说明：Pod 本身只代表一个运行实例，并不负责“维持一直存在”。后面你学 Deployment 时，就会看到更上层控制器如何接管这件事。

### 第五步：扩展理解多容器 Pod

虽然本章先从单容器 Pod 入门，但你要知道同一个 Pod 里也可以放多个容器。例如一个主应用容器，配一个日志转发或代理容器。只不过前提是它们必须足够紧密耦合，共享同一个生命周期才划算。否则，强行放进同一个 Pod，反而会让职责边界变模糊。

### 这一节应该带走什么

- Pod 是 Kubernetes 的最小调度单位，不等于容器本身。
- Pod 内容器共享网络和卷，因此具备强协作能力。
- 裸 Pod 适合学习和临时实验，不适合作为常规业务交付方式。
- 观察 Pod 状态与事件，是后面排障和发布能力的基础。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- Pod 为 Kubernetes 提供了统一的最小运行边界，便于调度、管理和扩展。
- 它支持多容器协作模型，适合处理共享网络和共享存储的场景。
- 几乎所有工作负载对象都建立在 Pod 之上，学会它就相当于拿到了 Kubernetes 的主入口。

### 缺点

- 抽象层比单容器更复杂，新手容易与 Docker 的容器概念混淆。
- 裸 Pod 本身不适合长期维护业务生命周期，需要上层控制器配合。
- 多容器 Pod 虽然灵活，但如果职责边界不清，很容易导致设计混乱。

### 使用场景

- 单个应用实例的运行载体。
- 强耦合多容器协作场景。
- 学习和验证容器运行、网络、卷、探针等基础机制。

### 注意事项

- 不要把 Pod 简化理解为“一个容器”。
- 不要轻易把不相关服务塞进同一个 Pod。
- 正式业务通常应由 Deployment、StatefulSet 等控制器来管理 Pod。

### 常见踩坑经验

- 最常见的坑，是直接长期维护裸 Pod，结果一删就没、也没有版本管理和自动恢复能力。
- 第二个坑，是看到 Pod 里能放多个容器，就把不相关功能都塞进去，导致耦合过重。
- 第三个坑，是只会 `kubectl get pod`，但不会看 `describe` 里的事件和状态细节，遇到异常就不知道从哪里排查。

本章学完后，你应该对 Pod 建立一个清晰认知：**它既是 Kubernetes 的起点，也是很多高级特性的落点。学会 Pod，不是学会一个对象，而是学会看懂整个工作负载体系的基础语言。**
