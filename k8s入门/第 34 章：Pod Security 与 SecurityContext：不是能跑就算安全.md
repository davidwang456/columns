# 第 34 章：Pod Security 与 SecurityContext：不是能跑就算安全

> 对应学习计划第 5 周 Pod 安全标准与 SecurityContext。

很多团队刚开始上 Kubernetes 时，对安全的判断往往非常朴素：容器能跑、服务可用、探针通过，就算上线成功了。但真正的安全问题通常不在“能不能跑”，而在“这个容器是不是被赋予了远超需求的权限”。容器默认权限一旦过宽，供应链风险、横向移动、容器逃逸和宿主机暴露面都会被放大。Pod Security 与 SecurityContext 的价值，就是帮助你把“默认宽松”收敛成“最小可用”。

## 1）项目背景

为什么容器安全会成为独立主题？因为 Kubernetes 默认给了你足够多的自由，而自由往往意味着风险。如果没有明确约束，常见现象包括：

- 容器以 root 身份运行。
- 根文件系统默认可写。
- 特权模式被随手打开。
- 容器具备不必要的 capability。
- 命名空间级别对危险 Pod 没有统一门槛。

这些做法在“先跑起来再说”的阶段看似高效，但长期几乎一定会留下安全债务。尤其当平台进入多团队、多环境、供应链复杂化阶段后，这些默认宽松配置就会成为高风险入口。

Kubernetes 在容器运行安全上主要有两层常见能力：

- `SecurityContext`：在 Pod/Container 级别收紧运行权限。
- Pod Security Admission / Pod Security Standards：在命名空间级别设安全基线。

这两者的关系可以这样理解：

- `SecurityContext` 回答“这个具体 Pod/容器怎么收敛权限”。
- Pod Security 标准回答“这个命名空间允许什么级别的安全姿态进入”。

也就是说，一个负责实例级细化，一个负责命名空间级守门。两者结合后，平台才开始真正拥有“默认安全姿态”的能力。

## 2）项目设计：通过大师和小白的对话引出该主题

**小白**：大师，我的容器只要能跑起来就行了吧？安全是不是等后面再补？

**大师**：这就是最典型的安全债起点。很多权限一旦默认开着，后面不是“补配置”那么简单，而是会渗透到镜像、应用和运维流程里。

**小白**：那最先该收紧什么？

**大师**：最常见的就是：不要 root、不要特权、不要可写根文件系统、不要无意义的能力提升。

**小白**：这些都能通过 SecurityContext 配？

**大师**：对。SecurityContext 是最直接的运行时权限收敛手段。

**小白**：那 Pod Security 又是干什么的？

**大师**：它更像命名空间门口的安全基线。不是每个团队都靠自觉写好 SecurityContext，而是平台先规定：这个命名空间里哪些危险姿态根本不许进。

**小白**：所以一个是“单个容器怎么收紧”，一个是“整个命名空间允许多宽松”。

**大师**：完全正确。

## 3）项目实战：通过主代码片段使用该主题的对象

这一节我们先从一个最小的容器级安全收敛配置开始，再回到命名空间级安全标准看整体策略。

### 第一步：给容器设置最小运行约束

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 10001
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
```

这段配置表达的是：

- 容器不要以 root 身份运行。
- 使用明确的非 root UID。
- 禁止权限提升。
- 根文件系统尽量只读。

这几项虽然看起来简单，但已经能砍掉一大批“默认宽松”带来的风险面。

### 第二步：理解为什么这些限制有时会让应用启动失败

这是学习 SecurityContext 时一个非常好的认知时刻：很多应用一旦被收紧权限就启动不了，不是 Kubernetes 配错了，而是应用或镜像本身早就依赖了“默认高权限”。

比如：

- 镜像里默认用户就是 root。
- 应用启动时会写根目录。
- 某些脚本依赖额外 capability。

所以，安全收敛不是“多写几个字段”，而是平台、镜像和应用设计之间的一次真实对账。

### 第三步：命名空间级安全基线怎么理解

当团队规模变大后，不能指望每个服务都靠自觉写对 SecurityContext。这时命名空间级 Pod Security 标准就会很有价值。你可以把它理解成：

- 某些命名空间允许更宽松策略。
- 某些命名空间只允许受限配置进入。

这样平台就能把“安全要求”前移到入口，而不是等资源进来后再靠人工发现问题。

### 第四步：为什么“先跑起来再说”在安全上代价很大

因为一旦团队长期默认 root、默认特权、默认可写根文件系统，后面你再收紧时，就不是改一两个 YAML，而是可能要回头修改：

- 镜像构建方式。
- 运行时目录设计。
- 文件权限管理。
- 应用启动脚本。

所以，安全收敛越早开始，长期成本越低。

### 第五步：建立一条最小安全姿态思路

学习阶段可以先固定问自己几件事：

- 这个容器一定要 root 吗？
- 一定要能写根文件系统吗？
- 一定要开特权吗？
- 一定要允许权限提升吗？

只要这些问题问起来，很多默认危险配置就已经开始被自然收敛了。

### 这一节应该带走什么

- 容器能跑，不代表它的权限姿态已经合理。
- `SecurityContext` 是容器级收敛，Pod Security 标准是命名空间级守门。
- 最小权限原则同样适用于容器运行时安全。
- 安全收敛越早做，后面治理成本越低。

## 4）项目总结：总结该主题对象的优点和缺点，使用场景，注意事项和常见踩坑经验

### 优点

- 它让容器从默认宽松姿态逐步收敛到最小可用权限。
- 它能显著降低供应链、逃逸和横向风险暴露面。
- 它为平台建立命名空间级安全基线提供了标准路径。

### 缺点

- 一旦历史镜像和应用设计过于依赖高权限，改造成本会比较高。
- 权限收紧后，很多兼容性问题会在短期集中暴露。
- 安全规则一多，团队需要更清晰的例外流程和治理规范。

### 使用场景

- 生产工作负载的权限收敛。
- 多团队共享平台的默认安全基线建设。
- 希望减少 root、特权和过度写权限的容器场景。

### 注意事项

- 不要把 SecurityContext 当成“上线前顺手填一下”的字段。
- 收紧安全姿态前，要评估镜像和应用本身是否支持。
- 命名空间级安全基线和实例级安全收敛最好结合使用。

### 常见踩坑经验

- 最常见的坑，是为了图快直接开 `privileged` 绕过所有问题。
- 第二个坑，是要求容器非 root，但镜像本身根本没为此准备好，结果启动失败后又一把放开。
- 第三个坑，是只写少量 SecurityContext 字段，却没有建立命名空间级基线，最后规范执行全靠自觉。

这一章真正想帮你建立的，是运行时安全视角：**默认宽松不是中性状态，而是一种尚未被收敛的风险。**
