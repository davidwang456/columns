# 第16章：基础篇综合实战——搭建单机版安全扫描工作台

> 版本：Trivy v0.50+
> 面向人群：开发、运维、测试、安全工程师

---

## 1. 项目背景

### 业务场景

云帆科技的基础篇培训已经进行了两个月。开发小王学会了用 `trivy image` 扫镜像，运维老李掌握了离线数据库部署，安全工程师小白能写自定义的 Secret 规则，测试组也用上了 pre-commit hook。但当 CTO 问「我们团队的安全扫描能力到底怎么样」时，所有人面面相觑——每个人手里都有一把「工具」，但没有一个「工作台」把这些工具串联起来。

更具体地说，团队面临以下混乱局面：

- 小王在 MacBook 上扫出的报告存在本地，老李在 Linux 服务器上的扫描结果存在另一个目录，两人互相不知道对方扫出了什么。
- 小白的 `.trivyignore` 文件散落在 20 个项目的仓库里，格式五花八门，有的写原因有的不写。
- 测试组发现每次安装 Trivy 都要重新配置一遍，新入职的同学经常因为漏配 `--cache-dir` 导致扫描超时。
- 最尴尬的是一次客户突击检查，CTO 想展示公司的安全扫描能力，结果现场花了 20 分钟找配置文件、下载数据库、调试权限——客户脸上的表情从期待变成了疑惑。

CTO 拍板：「基础篇学了 15 章，今天我要看到一个『开箱即用』的安全扫描工作台。一个人、一台电脑、一条命令，就能完成从镜像扫描到报告生成的全流程。这个工作台要成为团队的安全基础设施，新人入职第一天就能跑起来。」

### 痛点放大

**第一，知识碎片化。** 前 15 章每章都讲了一个专项能力，但缺少一个「总装车间」把这些能力组装成可用的产品。团队掌握了零件加工，但不会造整车。

**第二，环境配置成本高。** 每次换电脑、重装系统、新同事入职，都要重复一遍「安装 Trivy → 配缓存目录 → 下数据库 → 写配置文件 → 测 pre-commit hook」的冗长流程。没有标准化，每个人都在重复造轮子。

**第三，报告不可追溯。** 扫描结果分散在各个开发者的本地磁盘里，没有统一存储、没有版本管理、没有历史对比。上个月扫出 50 个漏洞，这个月扫出 30 个——是真的修复了，还是扫描参数变了？

**第四，缺少「黄金路径」。** 团队没有一套被验证过的、可复制的最佳实践。有人用 `--severity HIGH,CRITICAL`，有人用默认；有人开 secret 扫描，有人不开；有人在 CI 里跑，有人只在本地跑。行为的离散导致结果不可比。

**本章的核心目标是：将基础篇所学的全部能力整合为一个「单机版安全扫描工作台」，实现一键安装、统一配置、标准化扫描、结构化报告、基线管理。让任何人在任何一台机器上，5 分钟内就能开始专业的安全扫描。**

## 2. 项目设计

**场景**：云帆科技的「安全工作台」验收会，CTO 坐在旁边，小胖、小白和大师正在演示工作台的搭建过程。

---

**小胖**：（打开一台全新的虚拟机）「CTO 说要在『干净环境』里验证，所以我这是一台刚装好的 Ubuntu 22.04，除了 Docker 什么都没装。我要在 5 分钟内让这台机器具备完整的安全扫描能力。」

**小白**：「5 分钟？光下载 trivy-db 就要 2 分钟了吧？」

**小胖**：「这就是工作台的价值——数据库是预置的。你看，我运行一条命令：`

```bash
curl -fsSL https://cloud-sail.internal/install-workbench.sh | bash
```

这条脚本会自动完成：安装 Trivy → 配置缓存目录 → 导入预置数据库 → 安装 pre-commit 框架 → 复制团队标准配置 → 验证安装。」

**大师**：「技术映射：这个工作台就像『安全扫描的 IDE』。以前你写代码用记事本，要手动配编译器、调试器、版本控制；现在用 VS Code，一键安装插件，所有工具开箱即用。工作台就是 Trivy 的 VS Code。」

**小白**：「那工作台具体包含哪些组件？」

**小胖**：「我画个架构图：

```
┌─────────────────────────────────────────────────────────────┐
│                    云帆安全扫描工作台                         │
├─────────────────────────────────────────────────────────────┤
│  工具层                                                       │
│    ├── Trivy 二进制（固定版本 v0.50.0）                       │
│    ├── pre-commit 框架（Git Hook 管理）                       │
│    └── jq / Python3（报告处理依赖）                           │
├─────────────────────────────────────────────────────────────┤
│  配置层                                                       │
│    ├── ~/.config/trivy/trivy.yaml（全局标准配置）             │
│    ├── ~/.trivyignore-global（全局漏洞基线）                  │
│    └── ~/.cache/trivy/（共享缓存目录）                        │
├─────────────────────────────────────────────────────────────┤
│  脚本层                                                       │
│    ├── cs-scan（一键扫描镜像/目录/仓库）                      │
│    ├── cs-report（生成标准化报告）                            │
│    ├── cs-audit（审计 .trivyignore）                          │
│    └── cs-health（健康检查）                                  │
├─────────────────────────────────────────────────────────────┤
│  模板层                                                       │
│    ├── report-template.tpl（管理层摘要模板）                  │
│    └── report-detail.tpl（技术团队详细模板）                  │
└─────────────────────────────────────────────────────────────┘
```

**小白**：「那扫描后的报告存在哪里？」

**小胖**：「默认存在 `~/.cloud-sail/reports/YYYY-MM/` 目录下，按日期归档。每条报告都附带 `metadata.json`，记录扫描时间、Trivy 版本、数据库版本、扫描参数——这样三个月后的审计可以精确复现。」

**大师**：「还有一个关键点：工作台不是『安装完就完』，而是『持续更新』。团队的安全规范、数据库版本、配置文件都可能变化。工作台应该有一个 `cs-update` 命令，定期从团队的 Git 仓库拉取最新配置，就像 `apt update` 一样。」

**小胖**：「对，`cs-update` 会检查：

1. Trivy 版本是否需要升级。
2. `trivy.yaml` 是否有更新。
3. 全局 `.trivyignore` 是否有变更。
4. 报告模板是否有新版本。

更新前会自动备份旧配置，更新后运行 Sanity Check 确保没有坏掉。」

**CTO**：（点头）「那验收标准是什么？」

**小白**：「我定了几条：

1. **安装时效**：从干净系统到可扫描状态 < 5 分钟。
2. **扫描覆盖**：支持 image / fs / repo / config 四种目标的一键扫描。
3. **报告输出**：自动生成 table/json/html 三种格式报告。
4. **基线管理**：自动加载团队全局 `.trivyignore`，支持项目级叠加。
5. **健康监控**：内置 Sanity Check，扫描异常时自动告警。
6. **新人友好**：新入职开发运行 `cs-scan --tutorial` 可以交互式学习。」

## 3. 项目实战

### 环境准备

- **操作系统**：Ubuntu 22.04 LTS（或 macOS / WSL2）
- **Docker**：已安装并运行
- **网络**：可访问内部 Git 仓库（或本地文件）
- **权限**：当前用户有 sudo 权限（安装用）

### 步骤一：创建工作台安装脚本

**目标**：编写 `install-workbench.sh`，实现一键安装。

```bash
#!/bin/bash
# 云帆安全扫描工作台 - 安装脚本
# 用法: curl -fsSL https://.../install-workbench.sh | bash

set -e

WORKBENCH_VERSION="1.0.0"
TRIVY_VERSION="0.50.0"
INSTALL_DIR="${HOME}/.cloud-sail"
CACHE_DIR="${INSTALL_DIR}/cache"
CONFIG_DIR="${HOME}/.config/trivy"

echo "=== Cloud Sail Security Workbench v${WORKBENCH_VERSION} ==="

# 1. 创建目录结构
mkdir -p "${INSTALL_DIR}"/{bin,scripts,templates,reports,docs}
mkdir -p "${CACHE_DIR}/trivy/db"
mkdir -p "${CONFIG_DIR}"

# 2. 安装 Trivy
echo "[1/6] Installing Trivy v${TRIVY_VERSION}..."
if ! command -v trivy &> /dev/null; then
    curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s -- -b "${INSTALL_DIR}/bin" v${TRIVY_VERSION}
else
    echo "Trivy already installed: $(trivy version | head -1)"
fi

# 3. 复制标准配置
echo "[2/6] Setting up standard configuration..."
cat > "${CONFIG_DIR}/trivy.yaml" << 'EOF'
severity:
  - HIGH
  - CRITICAL
format: table
exit-code: 1
cache-dir: ~/.cloud-sail/cache/trivy
scanners:
  - vuln
  - secret
scan:
  skip-dirs:
    - ./node_modules
    - ./vendor
    - ./.git
timeout: 10m
EOF

# 4. 创建全局 .trivyignore
cat > "${INSTALL_DIR}/trivyignore-global" << 'EOF'
# 云帆科技 - 全局漏洞基线
# 适用于所有项目的通用忽略规则

# OS Vendor Backport 已知误报（Debian 系列）
# [REASON] Debian backport 版本号差异
# [EXPIRY] 2024-12-31
# [OWNER] ops@cloud-sail.com
CVE-2023-0464
EOF

# 5. 创建核心命令脚本
echo "[3/6] Creating command scripts..."

# cs-scan: 一键扫描
cat > "${INSTALL_DIR}/bin/cs-scan" << 'SCRIPT'
#!/bin/bash
# cs-scan: 云帆安全扫描工作台 - 主扫描命令

TARGET="${1:-.}"
SCAN_TYPE="${2:-auto}"
REPORT_DIR="${HOME}/.cloud-sail/reports/$(date +%Y-%m)"
TIMESTAMP=$(date +%Y%m%d-%H%M%S)
mkdir -p "${REPORT_DIR}"

# 自动检测扫描类型
if [ "$SCAN_TYPE" == "auto" ]; then
    if docker images --format "{{.Repository}}:{{.Tag}}" | grep -q "^${TARGET}$"; then
        SCAN_TYPE="image"
    elif [ -d "$TARGET/.git" ]; then
        SCAN_TYPE="fs"
    else
        SCAN_TYPE="fs"
    fi
fi

echo "=== Cloud Sail Scan ==="
echo "Target: $TARGET"
echo "Type: $SCAN_TYPE"
echo "Time: $(date -Iseconds)"
echo ""

# 执行扫描
case "$SCAN_TYPE" in
    image)
        trivy image --config "${HOME}/.config/trivy/trivy.yaml" \
            --format json --output "${REPORT_DIR}/scan-${TIMESTAMP}.json" \
            "$TARGET"
        ;;
    fs)
        trivy fs --config "${HOME}/.config/trivy/trivy.yaml" \
            --ignorefile "${HOME}/.cloud-sail/trivyignore-global" \
            --format json --output "${REPORT_DIR}/scan-${TIMESTAMP}.json" \
            "$TARGET"
        ;;
    repo)
        trivy repo --config "${HOME}/.config/trivy/trivy.yaml" \
            --format json --output "${REPORT_DIR}/scan-${TIMESTAMP}.json" \
            "$TARGET"
        ;;
    config)
        trivy config --severity HIGH,CRITICAL \
            --format json --output "${REPORT_DIR}/scan-${TIMESTAMP}.json" \
            "$TARGET"
        ;;
esac

# 生成人类可读报告
trivy convert --format table "${REPORT_DIR}/scan-${TIMESTAMP}.json" > "${REPORT_DIR}/scan-${TIMESTAMP}.txt"

# 生成元数据
cat > "${REPORT_DIR}/scan-${TIMESTAMP}-meta.json" << EOF
{
    "scan_time": "$(date -Iseconds)",
    "target": "$TARGET",
    "scan_type": "$SCAN_TYPE",
    "trivy_version": "$(trivy version | head -1)",
    "db_version": "$(cat ~/.cloud-sail/cache/trivy/db/metadata.json 2>/dev/null | jq -r '.Version // "unknown"')",
    "report_json": "scan-${TIMESTAMP}.json",
    "report_txt": "scan-${TIMESTAMP}.txt"
}
EOF

echo ""
echo "Scan complete. Reports saved to: ${REPORT_DIR}/"
echo "  - JSON: scan-${TIMESTAMP}.json"
echo "  - Table: scan-${TIMESTAMP}.txt"
SCRIPT
chmod +x "${INSTALL_DIR}/bin/cs-scan"

# cs-health: 健康检查
cat > "${INSTALL_DIR}/bin/cs-health" << 'SCRIPT'
#!/bin/bash
echo "=== Cloud Sail Health Check ==="

# 检查 Trivy
if command -v trivy &> /dev/null; then
    echo "[OK] Trivy: $(trivy version | head -1)"
else
    echo "[FAIL] Trivy not found"
    exit 1
fi

# 检查数据库
DB_FILE="${HOME}/.cloud-sail/cache/trivy/db/trivy.db"
if [ -f "$DB_FILE" ]; then
    DB_SIZE=$(du -m "$DB_FILE" | awk '{print $1}')
    echo "[OK] Database: ${DB_SIZE}MB"
else
    echo "[WARN] Database not found, will download on first scan"
fi

# Sanity Check
echo "[INFO] Running sanity check..."
trivy image --skip-db-update --format json python:3.4-alpine > /tmp/sanity.json 2>/dev/null
VULN_COUNT=$(cat /tmp/sanity.json | jq '[.Results[]?.Vulnerabilities?[]?] | length')
if [ "$VULN_COUNT" -gt 0 ]; then
    echo "[OK] Sanity check passed (${VULN_COUNT} vulnerabilities found in test image)"
else
    echo "[FAIL] Sanity check failed - Trivy may be malfunctioning"
    exit 1
fi

echo "=== Health Check Complete ==="
SCRIPT
chmod +x "${INSTALL_DIR}/bin/cs-health"

# 6. 添加到 PATH
echo "[4/6] Updating PATH..."
SHELL_RC=""
if [ -f "${HOME}/.bashrc" ]; then SHELL_RC="${HOME}/.bashrc"; fi
if [ -f "${HOME}/.zshrc" ]; then SHELL_RC="${HOME}/.zshrc"; fi

if [ -n "$SHELL_RC" ] && ! grep -q "cloud-sail/bin" "$SHELL_RC"; then
    echo 'export PATH="${HOME}/.cloud-sail/bin:${PATH}"' >> "$SHELL_RC"
    echo "Added to PATH in ${SHELL_RC}"
fi

# 7. 安装 pre-commit（可选）
echo "[5/6] Installing pre-commit..."
if command -v pip3 &> /dev/null && ! command -v pre-commit &> /dev/null; then
    pip3 install --user pre-commit > /dev/null 2>&1
    echo "[OK] pre-commit installed"
fi

# 8. 运行健康检查
echo "[6/6] Running health check..."
export PATH="${INSTALL_DIR}/bin:${PATH}"
if cs-health; then
    echo ""
    echo "=== Installation Complete ==="
    echo "Please run: source ${SHELL_RC}"
    echo "Then try: cs-scan python:3.4-alpine image"
    echo "Or: cs-scan /path/to/project fs"
else
    echo "Installation completed with warnings. Please check the output above."
fi
```

运行安装：

```bash
chmod +x install-workbench.sh
./install-workbench.sh
source ~/.bashrc
```

**目标**：用统一模板将 JSON 扫描结果转换为管理层和技术层两种报告。

创建 `~/.cloud-sail/templates/exec-summary.tpl`：

```
**扫描目标**: {{ .ArtifactName }}  
**扫描时间**: {{ now }}

| 严重级别 | 数量 | 状态 |
|----------|------|------|
| CRITICAL | {{ len (where .Results "Vulnerabilities" "severity" "CRITICAL") }} | 需立即处理 |
| HIGH | {{ len (where .Results "Vulnerabilities" "severity" "HIGH") }} | 7天内修复 |
| MEDIUM | {{ len (where .Results "Vulnerabilities" "severity" "MEDIUM") }} | 下次迭代 |
| LOW | {{ len (where .Results "Vulnerabilities" "severity" "LOW") }} | 酌情处理 |

## 建议

{{ if gt (len (where .Results "Vulnerabilities" "severity" "CRITICAL")) 0 }}

```

生成报告：

```bash

  --output report-summary.md \
  ~/.cloud-sail/reports/$(date +%Y-%m)/scan-*.json
```

### 步骤四：配置项目级叠加基线

**目标**：全局基线 + 项目级 `.trivyignore` 叠加使用。

在项目根目录创建 `.trivyignore`：

```
# 项目级漏洞基线（叠加在全局基线之上）
# [REASON] 测试专用依赖，不进入生产
# [EXPIRY] 2024-12-31
CVE-2021-23337
```

修改 `cs-scan` 支持叠加：

```bash
# 在 cs-scan 的 fs 分支中，合并全局和本地 ignore
trivy fs --config "${HOME}/.config/trivy/trivy.yaml" \
    --ignorefile "${HOME}/.cloud-sail/trivyignore-global" \
    --ignorefile "./.trivyignore" \
    ...
```

### 步骤五：搭建工作台更新机制

**目标**：实现配置的持续同步。

创建 `cs-update` 脚本：

```bash
#!/bin/bash
# cs-update: 更新工作台配置

WORKBENCH_REPO="https://github.com/cloud-sail/security-workbench.git"
TEMP_DIR="/tmp/cs-workbench-update"

echo "=== Checking for updates ==="

# 克隆最新配置
git clone --depth 1 "$WORKBENCH_REPO" "$TEMP_DIR" 2>/dev/null || true

# 备份旧配置
BACKUP_DIR="${HOME}/.cloud-sail/backups/$(date +%Y%m%d-%H%M%S)"
mkdir -p "$BACKUP_DIR"
cp -r "${HOME}/.config/trivy" "$BACKUP_DIR/" 2>/dev/null || true
cp -r "${HOME}/.cloud-sail/trivyignore-global" "$BACKUP_DIR/" 2>/dev/null || true

# 应用新配置
cp "$TEMP_DIR/configs/trivy.yaml" "${HOME}/.config/trivy/trivy.yaml" 2>/dev/null || true
cp "$TEMP_DIR/configs/trivyignore-global" "${HOME}/.cloud-sail/trivyignore-global" 2>/dev/null || true

# 清理
rm -rf "$TEMP_DIR"

# 验证
echo "=== Running post-update health check ==="
cs-health

echo "Update complete. Old config backed up to: $BACKUP_DIR"
```

### 测试验证

1. 在干净 Ubuntu 虚拟机上运行 `install-workbench.sh`，确认 5 分钟内完成安装。
2. 执行 `cs-health`，确认所有检查项通过。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 标准化 | 统一配置、统一命令、统一报告 | 灵活性降低，不适合需要高度自定义的场景 |
| 易用性 | 新人 5 分钟上手，无需理解 Trivy 全部参数 | 高级用户可能觉得封装过度 |
| 可追溯 | 报告按日期归档，附带元数据 | 长期存储需要磁盘管理策略 |
| 可维护 | 配置集中管理，更新一键完成 | 需要维护 Git 仓库和更新服务 |
| 覆盖度 | 整合 image/fs/repo/config 扫描 | 不包含 K8s/云环境等高级扫描 |

### 适用场景

1. **团队安全基础设施初始化**：新团队、新项目快速建立标准化扫描能力。
2. **新人入职培训**：「第一天就能跑扫描」，降低安全工具的入门门槛。
3. **客户现场演示/审计**：一键展示团队的安全治理能力。
4. **离线环境部署**：预置数据库 + 本地脚本，无外网也能工作。
5. **多项目统一管理**：全局基线 + 项目级叠加，保持整体一致性的同时允许局部差异。

**不适用场景**：
1. 需要与企业现有安全平台（如 DefectDojo、Splunk）深度集成的场景——工作台是单机版，不具备平台级联动能力。
2. 大规模分布式团队（100+ 人）——需要中央化管理而非单机工作台。

### 注意事项

- **数据库更新**：预置数据库会随时间过期，建议每月执行一次 `trivy image --download-db-only` 更新。
- **配置备份**：`cs-update` 会自动备份旧配置，但备份目录也会增长，建议定期清理。
- **权限管理**：`~/.cloud-sail/` 目录包含缓存和报告，确保运行用户对目录有读写权限。
- **模板语法**：Go Template 的 `where` 函数在 Trivy 不同版本中可能有差异，升级 Trivy 后需验证模板兼容性。

### 常见踩坑经验

**踩坑案例 1：安装后命令找不到**
- **现象**：运行 `cs-scan` 报 `command not found`。
- **根因**：`~/.cloud-sail/bin` 未加入 PATH，或 shell 未重新加载。
- **解法**：执行 `source ~/.bashrc`（或对应的 shell 配置文件），或直接使用完整路径 `~/.cloud-sail/bin/cs-scan`。

**踩坑案例 2：报告目录磁盘占满**
- **现象**：长期运行后 `~/.cloud-sail/reports/` 达到数十 GB。
- **根因**：每次扫描都生成新报告，没有自动清理机制。
- **解法**：添加 crontab 定期清理旧报告：`find ~/.cloud-sail/reports/ -type f -mtime +90 -delete`。

**踩坑案例 3：项目级 `.trivyignore` 格式不规范导致扫描失败**
- **现象**：`cs-scan` 报错 `invalid ignore file`。
- **根因**：项目级 `.trivyignore` 使用了旧格式或包含非法字符。
- **解法**：在 `cs-scan` 中增加 `.trivyignore` 格式预检，或统一使用本章的注释规范。

### 思考题

1. 假设你的团队从 20 人扩张到 100 人，单机版工作台无法满足协作需求。请设计一个「工作台 2.0」的架构升级方案：如何在保留现有命令行体验的同时，增加中央报告存储、团队级基线管理和跨项目漏洞趋势分析？
2. 工作台的 `cs-scan` 命令封装了 Trivy 的参数，但这也导致用户无法使用 Trivy 的新特性（如新的 Scanner 类型）。请设计一个「扩展机制」，让高级用户既能享受工作台的便利，又能透传自定义参数给底层 Trivy。

> **答案提示**：中级篇的 DevSecOps 网关（第 31 章）和高级篇的企业级安全平台（第 40 章）将揭晓大规模团队的安全基础设施方案。
