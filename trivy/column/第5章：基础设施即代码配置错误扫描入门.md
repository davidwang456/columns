# 第5章：基础设施即代码（IaC）配置错误扫描入门

> 版本：Trivy v0.50+
> 面向人群：DevOps、SRE、安全工程师、后端开发

---

## 1. 项目背景

### 业务场景

云帆科技的上云之路走得很快。运维组用 Terraform 管理了 200 多台云服务器，开发组用 Helm Chart 部署了 50 多个微服务，还有一个 rookie 工程师为了方便调试，在 Kubernetes Deployment 里加了个 `privileged: true`。

三个月后，公司参加了一次行业安全评测。评测机构用自动化工具扫描了云帆的 K8s 集群配置，结果让人触目惊心：

- 12 个 Pod 以 root 用户运行
- 8 个 Deployment 没有配置资源限制（ResourceQuota），导致某次流量高峰时节点被 OOM Kill
- 3 个 Service 的 LoadBalancer 暴露了 22 端口到公网
- 1 个 ConfigMap 里明文写了数据库密码
- 最离谱的是，某个 Terraform 模块创建了一个完全开放的 S3 Bucket，任何人都能读写

CTO 看着评测报告，问了老李一个灵魂问题：「这些配置在代码里都存在多久了？为什么发版前没人检查？」

老李无言以对。团队的 CI/CD 流水线已经集成了 Trivy 镜像扫描，但所有人都默认「没有漏洞的镜像就是安全的镜像」，完全忽略了 Kubernetes YAML、Terraform、Dockerfile 本身的配置风险。

### 痛点放大

**第一，「镜像安全 ≠ 运行安全」。** 一个镜像本身可能没有 CVE，但如果 Kubernetes 配置里允许它以 root 身份运行、挂载了宿主机的 `/etc/shadow`、或者没有设置只读文件系统，攻击者拿到容器后几乎可以畅通无阻。

**第二，配置错误具有「放大效应」。** 代码里一个 `privileged: true` 的字段，比 10 个 Medium 级别的 CVE 更危险——它直接让容器突破了隔离边界，拿到了宿主机的 root 权限。但传统的漏洞扫描器对此视而不见。

**第三，多 IaC 工具的管理混乱。** 团队同时用 Terraform 管云资源、用 Helm 管 K8s、用 Docker Compose 管本地开发。每种工具都有自己的配置语法和风险点，没有统一的检查标准，安全策略无法落地。

**第四，修复成本随时间指数增长。** 一个错误的 S3 Bucket 策略如果在创建当天发现，改一行代码、执行一次 `terraform apply` 就能修复。如果三个月后才发现，里面可能已经存了几 TB 的数据，还有上下游系统依赖这个 Bucket 的 ARN——改动牵一发而动全身。

**本章的核心目标是：用 Trivy 的 Misconfiguration Scanner，在代码提交阶段就发现并修复危险的 IaC 配置，把运行态风险消灭在萌芽期。**

## 2. 项目设计

**场景**：云帆科技的架构评审会，老李、DevOps 工程师小白和刚入职的小胖正在讨论如何把配置安全加入流水线。

---

**小胖**：（看着评测报告）「这报告也太吓人了……`privileged: true` 是啥意思？我 K8s 教程里看到过，说加了就能拥有宿主机的所有权限，方便调试啊。」

**小白**：（扶额）「方便调试，也方便黑客。你把 `privileged` 比作什么？」

**小胖**：「嗯……就像租了公寓，本来只能用自己的房间。加了 privileged，等于房东把整栋楼的钥匙都给你了，包括其他租客的房门。」

**大师**：「技术映射：非常准确。在 K8s 里，普通容器是被关在『房间』（cgroups + namespace）里的。`privileged: true` 等于拆掉了所有的墙，容器里的 root 就是宿主机的 root，可以访问任何设备、修改任何内核参数。这是配置扫描中最危险的标记之一。」

**小白**：「而且不只是 K8s。你看这个 Terraform 配置，S3 Bucket 的 ACL 是 `public-read-write`，等于把公司数据放在广场上任人取阅。」

**小胖**：「那 Terraform 和 K8s 的配置语法完全不一样，我们怎么统一检查？难道要学两套工具？」

**大师**：「这就是 Trivy `config` 子命令的价值。它内置了一个 Checks Bundle，里面包含了几百条针对各种 IaC 的检查规则：

- **Dockerfile**：禁止以 root 运行、强制使用特定基础镜像、限制暴露端口范围
- **Kubernetes YAML**：禁止 privileged、必须设置 resource limits、必须配置 securityContext
- **Terraform / CloudFormation**：禁止公开 S3 Bucket、必须启用 RDS 加密、必须配置安全组白名单
- **Helm Chart**：渲染后的 K8s 资源同样受检

你只需要一条命令：`trivy config /path/to/iac`，所有配置文件的潜在风险都会被列出来。」

**小胖**：「这些规则是谁写的？可靠吗？万一误报怎么办？」

**小白**：「规则主要来自两个来源：一是 Aqua 安全团队维护的内置 Checks，二是 Open Policy Agent（OPA）的 Rego 语言编写的自定义策略。内置规则覆盖的是业界公认的高危配置，比如 CIS Benchmark 的对应项。误报率比漏洞扫描低得多，因为配置检查是『白名单思维』——要么符合安全基线，要么不符合，没有太多灰色地带。」

**大师**：「技术映射：你可以把内置 Checks 理解为『交通法规』。红灯停绿灯行，没有『可能是红灯也可能不是』的模糊状态。配置错误扫描就是给你的 IaC 代码做『科目一考试』，每条规则都是一道必答题。」

**小胖**：「那如果我们的业务场景确实需要 `privileged` 呢？比如要加载内核模块的监控 Agent。」

**小白**：「Trivy 支持通过 `.trivyignore` 或内联注释跳过特定检查。但必须有明确的理由和审批记录。比如：

```yaml
# trivy:ignore:AVD-KSV-0017
securityContext:
  privileged: true
```

这样 Trivy 会跳过这条规则的告警，但代码审查时 reviewer 仍然能看到这个注释，要求你解释原因。」

**大师**：「更重要的不是『怎么跳过』，而是『怎么建立基线』。我建议团队先定义一份《IaC 安全基线文档》，列出必须遵守的 20 条规则。然后把这些规则对应的 Trivy Check ID 整理出来，在 CI 中配置为『不可跳过』的强制项。剩下的规则设为『建议修复』，允许延期处理。」

**小胖**：「听起来像给配置文件装了个安检仪。那扫描速度怎么样？我们的 Terraform 模块有几百个 `.tf` 文件，K8s 仓库有上千个 YAML。」

**小白**：「Trivy 的 `config` 扫描是纯静态分析，不需要拉取镜像或下载数据库（除了 checks bundle，可以离线缓存）。扫描一千个 YAML 文件通常只需要十几秒。而且可以通过 `--skip-dirs` 排除第三方 Chart 目录，进一步加速。」

**大师**：「最后提醒一点：配置扫描一定要放在『Plan 阶段』，而不是『Apply 阶段』。Terraform 的 `terraform plan` 输出的是即将创建的资源清单，Trivy 可以扫描这个 plan 文件，在资源真正创建之前发现风险。等 `terraform apply` 执行后再扫描，云资源已经上线了，修复成本倍增。」

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+，已安装
- **测试文件**：我们将创建 Dockerfile、Kubernetes YAML、Terraform 配置文件
- **编辑器**：任意文本编辑器

### 步骤一：扫描存在高危配置的 Dockerfile

**目标**：识别 Dockerfile 中的不安全指令。

创建 `Dockerfile.unsafe`：

```dockerfile
FROM ubuntu:latest

# 以 root 运行（高危）
USER root

# 安装 SSH 服务（扩大攻击面）
RUN apt-get update && apt-get install -y openssh-server

# 将密码硬编码在镜像中（极高危）
ENV DB_PASSWORD=supersecret123

# 暴露过多端口
EXPOSE 22 80 3306 6379

# 没有 healthcheck
# 没有使用多阶段构建

COPY app.py /app/
WORKDIR /app
CMD ["python", "app.py"]
```

执行扫描：

```bash
trivy config Dockerfile.unsafe
```

**预期输出**（截取关键部分）：
```
Dockerfile.unsafe (dockerfile)
==============================
Tests: 25 (SUCCESSES: 20, FAILURES: 5)

AVD-DS-0002: "Image user should not be 'root'"
═══════════════════════════════════════════════
Dockerfile.unsafe:4
─────────────────────────────────────────────────
   4 [ USER root
─────────────────────────────────────────────────

AVD-DS-0013: "Use secrets manager or environment variables"
═════════════════════════════════════════════════════════════
Dockerfile.unsafe:7
─────────────────────────────────────────────────
   7 [ ENV DB_PASSWORD=supersecret123
─────────────────────────────────────────────────
```

**分析要点**：
- `AVD-DS-0002`：Trivy 检测到 `USER root`，建议改用非 root 用户运行容器。
- `AVD-DS-0013`：硬编码密码被识别，建议使用环境变量或 Secrets Manager。
- 每条规则都有唯一的 AVD 编号，便于在 `.trivyignore` 或文档中引用。

### 步骤二：扫描存在高危配置的 Kubernetes YAML

**目标**：识别 K8s 资源定义中的安全配置错误。

创建 `deployment-unsafe.yaml`：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      containers:
      - name: app
        image: myapp:v1.0
        securityContext:
          privileged: true
          runAsRoot: true
          allowPrivilegeEscalation: true
          readOnlyRootFilesystem: false
        resources: {}
        volumeMounts:
        - name: host-etc
          mountPath: /host-etc
      volumes:
      - name: host-etc
        hostPath:
          path: /etc
```

执行扫描：

```bash
trivy config deployment-unsafe.yaml
```

**预期输出**（截取关键部分）：
```
deployment-unsafe.yaml (kubernetes)
====================================
Tests: 30 (SUCCESSES: 22, FAILURES: 8)

AVD-KSV-0017: "Privileged container"
═══════════════════════════════════════
deployment-unsafe.yaml:19
─────────────────────────────────────────
  19 [ privileged: true
─────────────────────────────────────────

AVD-KSV-0010: "Container should not mount sensitive host paths"
════════════════════════════════════════════════════════════════
deployment-unsafe.yaml:27
─────────────────────────────────────────
  27 [ path: /etc
─────────────────────────────────────────

AVD-KSV-0011: "Resource limits should be set"
═══════════════════════════════════════════════
deployment-unsafe.yaml:22
─────────────────────────────────────────
  22 [ resources: {}
─────────────────────────────────────────
```

**修复后的安全配置**：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
  template:
    metadata:
      labels:
        app: web-app
    spec:
      securityContext:
        runAsNonRoot: true
        seccompProfile:
          type: RuntimeDefault
      containers:
      - name: app
        image: myapp:v1.0
        securityContext:
          privileged: false
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          capabilities:
            drop:
              - ALL
        resources:
          limits:
            memory: "512Mi"
            cpu: "500m"
          requests:
            memory: "256Mi"
            cpu: "250m"
```

重新扫描修复后的 YAML，确认所有 `FAILURES` 清零。

### 步骤三：扫描 Terraform 配置

**目标**：识别云资源配置中的安全风险。

创建 `main.tf`：

```hcl
provider "aws" {
  region = "us-east-1"
}

resource "aws_s3_bucket" "data" {
  bucket = "cloud-sail-data-bucket"
  acl    = "public-read-write"
}

resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Allow all inbound traffic"

  ingress {
    from_port   = 0
    to_port     = 65535
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

resource "aws_db_instance" "postgres" {
  identifier        = "cloud-sail-db"
  engine            = "postgres"
  instance_class    = "db.t3.micro"
  allocated_storage = 20
  username          = "admin"
  password          = "password123"
  storage_encrypted = false
  publicly_accessible = true
}
```

执行扫描：

```bash
trivy config main.tf
```

**预期输出**（截取关键部分）：
```
main.tf (terraform)
===================
Tests: 15 (SUCCESSES: 9, FAILURES: 6)

AVD-AWS-0089: "S3 Bucket does not have logging enabled"
AVD-AWS-0090: "S3 Bucket has public access acl"
AVD-AWS-0107: "Security group rule allows unrestricted access"
AVD-AWS-0132: "RDS instance does not have encryption enabled"
AVD-AWS-0176: "RDS instance publicly accessible"
```

**关键修复**：
- S3 Bucket 将 `acl` 改为 `private`，并配置 `block_public_access`。
- Security Group 将 `cidr_blocks` 限制为内网网段（如 `10.0.0.0/8`）。
- RDS 设置 `storage_encrypted = true` 和 `publicly_accessible = false`。
- 密码使用 `aws_secretsmanager_secret` 或变量传入，禁止硬编码。

### 步骤四：Terraform Plan 扫描（事前拦截）

**目标**：在资源创建前扫描 `terraform plan` 的输出，实现真正的「左移」。

```bash
# 初始化并生成 plan
cd terraform-project
terraform init
terraform plan -out=tfplan
terraform show -json tfplan > tfplan.json

# 扫描 plan 文件
trivy config tfplan.json
```

**优势**：
- 在 `terraform apply` 之前发现风险，避免云资源上线后再回滚。
- 可以捕获动态计算后的最终配置（如变量、模块、条件逻辑展开后的结果）。

> **可能遇到的坑**：Terraform Plan 的 JSON 输出格式较大，Trivy 解析时可能需要较多内存。对于超大型基础设施（数千资源），建议分模块扫描。

### 步骤五：批量扫描与 CI 集成

**目标**：对包含多种 IaC 文件的仓库进行批量扫描，并配置 CI 门禁。

假设仓库结构如下：

```
infra-repo/
├── docker/
│   └── Dockerfile
├── k8s/
│   ├── deployment.yaml
│   └── service.yaml
├── terraform/
│   └── main.tf
└── helm/
    └── values.yaml
```

执行批量扫描：

```bash
# 扫描整个目录，Trivy 会自动识别文件类型
trivy config infra-repo/
```

**GitLab CI 集成示例**：

```yaml
stages:
  - validate
  - security

terraform_validate:
  stage: validate
  script:
    - cd terraform && terraform validate

iac_security_scan:
  stage: security
  image: aquasec/trivy:0.50.0
  script:
    - trivy config --severity HIGH,CRITICAL --exit-code 1 .
  allow_failure: false
```

**GitHub Actions 集成示例**：

```yaml
name: IaC Security Scan
on: [pull_request]
jobs:
  trivy-config:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Trivy Config Scanner
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'config'
          severity: 'HIGH,CRITICAL'
          exit-code: '1'
```

### 步骤六：自定义基线与分级策略

**目标**：根据团队实际情况，建立「强制遵守」和「建议遵守」两级规则。

创建 `trivy-config.yaml`：

```yaml
severity:
  - HIGH
  - CRITICAL
misconfiguration:
  terraform:
    exclude-downloaded-modules: true
  kubernetes:
    exclude-owned: true
scanners:
  - misconfig
skip-checks:
  # 跳过团队评估后接受的低风险规则
  - AVD-KSV-0001
```

**团队基线分级表示例**：

| 级别 | 规则示例 | 处理方式 | CI 行为 |
|------|----------|----------|---------|
| P0（强制）| privileged=true、root 运行、公开 S3 | 必须修复，不可跳过 | exit-code=1，阻断合并 |
| P1（重要）| 无 resource limits、无 healthcheck | 7 天内修复，需审批可跳过 | exit-code=1，可人工绕过 |
| P2（建议）| 标签不完整、注释缺失 | 建议修复，不阻断 | exit-code=0，仅告警 |

### 测试验证

1. 扫描 `Dockerfile.unsafe`，确认报出 `USER root` 和硬编码密码。
2. 扫描 `deployment-unsafe.yaml`，确认报出 `privileged`、`hostPath`、`resources` 缺失。
3. 修复所有配置后重新扫描，确认 `FAILURES` 为 0。
4. 扫描 `main.tf`，确认报出公开 S3、开放安全组、RDS 未加密。
5. 生成 `tfplan.json` 并扫描，确认能识别 Plan 中的风险配置。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 覆盖范围 | 支持 Dockerfile/K8s/Terraform/CloudFormation/Helm | 对自定义 DSL 或小众 IaC 工具不支持 |
| 检查深度 | 基于 CIS Benchmark 和最佳实践，规则成熟 | 复杂条件逻辑（如变量交叉引用）可能分析不全 |
| 扫描速度 | 纯静态分析，无需网络，秒级完成 | 超大型 Terraform 状态文件解析较慢 |
| CI 集成 | exit-code 机制天然支持门禁 | 规则升级后可能引入新的阻断项，需提前测试 |
| 误报率 | 配置检查逻辑清晰，误报率低于漏洞扫描 | 某些业务场景确实需要例外，需配合跳过机制 |

### 适用场景

1. **K8s 安全基线检查**：确保所有 Deployment 都配置了非 root、只读根文件系统、资源限制。
2. **Terraform 事前拦截**：在 `terraform apply` 之前扫描 plan 输出，防止高危云资源配置上线。
3. **Dockerfile 安全规范**：强制使用非 root 用户、禁止硬编码密钥、限制暴露端口。
4. **Helm Chart 发布前检查**：渲染后的 K8s 资源在发布到集群前进行安全审查。
5. **合规审计**：对照 CIS Docker Benchmark、CIS Kubernetes Benchmark 进行自动化合规检查。

**不适用场景**：
1. 需要运行时行为分析（如检测容器逃逸的实际利用链）——Trivy 是静态检查器，不运行目标配置。
2. 需要网络拓扑分析（如检测 K8s 集群内部的异常流量）——需配合 Falco、Cilium 等运行时工具。

### 注意事项

- **Plan 扫描的时效性**：`terraform plan` 的结果是临时的，如果变量或模块发生变化，需重新生成 plan 文件。
- **Helm Chart 的渲染**：Trivy 可以扫描 Helm Chart 的 values 文件，但更推荐先执行 `helm template` 渲染出完整 YAML，再扫描渲染结果，以捕获模板逻辑中的隐藏风险。
- **跳过规则的风险**：`skip-checks` 和 `# trivy:ignore` 应该被代码审查严格把关，避免成为「安全债务」的避风港。

### 常见踩坑经验

**踩坑案例 1：Terraform 模块下载超时**
- **现象**：扫描包含远程模块的 Terraform 配置时，卡在 `Downloading modules...`。
- **根因**：Trivy 默认会下载 Terraform 模块以解析完整配置，内网或模块源不可达时会超时。
- **解法**：在 `trivy.yaml` 中设置 `misconfiguration.terraform.exclude-downloaded-modules: true`，或预先用 `terraform init` 下载模块。

**踩坑案例 2：K8s YAML 扫描报「找不到镜像」**
- **现象**：Trivy 对 K8s YAML 扫描时输出 `unable to resolve image`。
- **根因**：Trivy 尝试解析容器镜像以获取额外的元数据，但镜像在本地不存在或无法拉取。
- **解法**：加上 `--skip-policy-update` 或确保镜像可访问；对于纯配置检查，该警告不影响 misconfiguration 结果。

**踩坑案例 3：修复了配置但 CI 仍然失败**
- **现象**：本地 `trivy config` 通过，但 CI 中报错。
- **根因**：CI 中的 Trivy 版本与本地不一致，新版本增加了规则。
- **解法**：团队统一 Trivy 版本（通过 Docker 镜像固定版本号），并在升级前先在 staging 环境验证。
