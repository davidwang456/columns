## 1. 项目背景

### 业务场景

经过前面几章的源码剖析，架构师小白已经理解了 Trivy 从入口到分析到检测的完整链路。但在实际交付场景中，他发现了一个最后一公里问题：如何让扫描结果以最佳格式呈现给不同的消费者？

云帆科技的客户服务团队需要给每个企业客户交付季度安全报告——一种包含客户 Logo、定制化表格、趋势图、修复建议的综合 PDF 文档。Deploy 团队希望在 CI/CD Pipeline 日志中看到简洁的红绿表。JIRA 集成团队需要 custom-jira.json 格式。合规团队需要 SARIF 格式上传到 GitHub Security Advisories。

目前，Trivy 内置了 10 种输出格式（table、json、sarif、cyclonedx、spdx、spdx-json、github、cosign-vuln、template、html）。但如果这 10 种都不满足需求呢？小白需要理解 pkg/report/ 的 Writer 接口，然后编写第 11 种输出格式。

### 痛点放大

**第一，Writer 接口看似简单实则关键。** 只有 Write(context.Context, types.Report) error 一个方法，但 types.Report 是一个包含嵌套数组、可选字段、多态类型（OS 包 vs 语言包 vs 配置 vs Secret vs 许可证）的复杂结构体。如果 Writer 实现不好（如未处理 nil 字段），可能 panic。

**第二，Go Template 引擎的功能边界。** 第 23 章介绍了 Go Template 的基本用法，但引擎实际的能力要大得多——它支持自定义函数注册、管道操作、子模板调用、条件编译。如果你不理解 template.go 的实现，你就无法写出能处理空 Results 时不渲染表格的高级模板。

**第三，SARIF 格式的复杂性。** SARIF（Static Analysis Results Interchange Format）不是简单的把 JSON 换了个 schema。它有自己的规则索引、结果级别映射、物理位置追踪等复杂语义。Trivy 的 SARIF Writer（pkg/report/sarif.go）是理解如何将领域模型映射到标准格式的绝佳案例。

**第四，Table Writer 的精巧设计。** pkg/report/table/ 下的子渲染器（vulnerability.go、misconfig.go、secret.go）各自处理不同类型的结果，但又共享同一个表格基础设施。这种组合模式是理解如何渲染复杂嵌套数据的绝佳案例。

**本章的核心目标是：深入 pkg/report/ 源码，理解 Writer 接口、Marshaler 注册机制、Table Renderer 的分发模式、Go Template 引擎实现，并实战编写一个自定义 Writer 实现企业内部格式输出。**

## 2. 项目设计

**场景**：云帆科技的报告即服务需求会，小白、老李和大师在讨论如何让 Trivy 的输出适配所有下游系统。

---

**老李**：「我们有 5 个下游系统都来消费 Trivy 的扫描结果——JIRA、DefectDojo、企业微信、自研 Dashboard、合规归档系统。每个系统的数据格式都不一样。现在团队方案是 Trivy 输出 JSON，然后 Python 脚本转换。5 个系统 = 5 个转换脚本 = 5 个维护噩梦。」

**大师**：「技术映射：这就像出口商品。Trivy 扫描结果是原材料（types.Report），不同的输出格式是加工后的产品（JSON、SARIF、CycloneDX 等）。如果每次出口一个新市场都要重新建一个工厂（转换脚本），成本太高。Trivy 的做法是统一工厂接口——Writer 接口。」

**小白**：「核心就在 pkg/report/writer.go 的 Write() 函数里：

```go
// pkg/report/writer.go (简化)
func Write(ctx context.Context, report types.Report, option flag.Options) error {
    // Step 1: 调用扩展钩子
    extension.PreReport(ctx, &report)

    // Step 2: 创建输出 writer
    writer, err := option.OutputWriter(ctx)

    // Step 3: 按 --format 参数选择 Writer
    switch option.Format {
    case table.FormatTable:
        writer = table.NewWriter(output, options)
    case json.FormatJSON:
        writer = json.NewWriter(output, options)
    case sarif.FormatSarif:
        writer = sarif.NewWriter(output, version)
    case cyclonedx.FormatCycloneDX:
        writer = cyclonedx.NewWriter(output, options)
    case spdx.FormatSPDXJSON:
        writer = spdx.NewWriter(output, options)
    case template.FormatTemplate:
        writer = template.NewWriter(output, options)
    default:
        return xerrors.Errorf("unknown format: %s", option.Format)
    }

    // Step 4: 输出
    return writer.Write(ctx, report)
}
```

这个 switch-case 隐藏了一个强大的扩展点——如果你需要第 11 种格式，只需要实现 Writer 接口并在这加一个 case。」

**小胖**：「那 Go Template 引擎怎么实现的？我写自定义模板时总觉得有些字段访问不了。」

**大师**：「pkg/report/template.go 的核心是 Go 标准库的 text/template 加自定义函数注册。Trivy 注册了三十多个自定义函数，让你在模板里做过滤、转换、格式化。包括类型转换（toJson, fromJson）、过滤函数（filterBySeverity, where）、格式化函数（severityColor, formatTime）、数学函数（add, sub, mul, div）和字符串函数（trim, contains）。这些函数让你在模板里可以做很多高级操作，比如 where 函数类似 SQL WHERE 来筛选特定严重级别。」

**老李**：「那 Table Writer 的分发模式是什么意思？」

**小白**：「Table Writer 不自己渲染所有内容——它根据 Result.Class 做分发：

- ClassOSPkg / ClassLangPkg → vulnerabilityRenderer 渲染（漏洞表格）
- ClassConfig → misconfigRenderer 渲染（配置错误表格）
- ClassSecret → secretRenderer 渲染（密钥泄露表格）
- ClassLicense → pkgLicenseRenderer 渲染（许可证表格）

每个子 Renderer 只关心自己的数据格式，输出的是各自的表格片段。Table Writer 最后把它们拼接在一起，加上 Summary 标题。这就是组合模式——每个类型独立渲染，最终统一拼接。」

**大师**：「再看 SARIF Writer（pkg/report/sarif.go）。SARIF 标准规定了一个 run 对象包含 tool（扫描工具信息）、results（发现列表）、rules（规则定义）、artifacts（扫描对象）。Trivy 的 SarifWriter 需要把内部 types.Report 的每种结果类型映射到对应的 SARIF rule index。这个映射逻辑就是理解和实现标准格式互通的关键。」

### 步骤一：深入 types.Report 结构体

**目标**：理解 pkg/types/report.go 中的数据结构。

```go
// pkg/types/report.go (简化结构)
type Report struct {
    SchemaVersion int          `json:"SchemaVersion"`
    ArtifactName  string       `json:"ArtifactName"`
    ArtifactType  string       `json:"ArtifactType"`
    Metadata      Metadata     `json:"Metadata"`
    Results       Results      `json:"Results"`    // []Result
}

type Result struct {
    Target            string                      `json:"Target"`
    Class             ResultClass                 `json:"Class"`
    Type              string                      `json:"Type"`
    Vulnerabilities   []DetectedVulnerability    `json:"Vulnerabilities,omitempty"`
    Misconfigurations []DetectedMisconfiguration `json:"Misconfigurations,omitempty"`
    Secrets           []DetectedSecret           `json:"Secrets,omitempty"`
    Licenses          []DetectedLicense          `json:"Licenses,omitempty"`
    Packages          []Package                  `json:"Packages,omitempty"`
}

type DetectedVulnerability struct {
    VulnerabilityID  string   `json:"VulnerabilityID"`  // CVE ID
    PkgName          string   `json:"PkgName"`
    InstalledVersion string   `json:"InstalledVersion"`
    FixedVersion     string   `json:"FixedVersion"`
    Severity         string   `json:"Severity"`          // CRITICAL/HIGH/MEDIUM/LOW
    Title            string   `json:"Title"`
    Description      string   `json:"Description"`
    PublishedDate    string   `json:"PublishedDate"`
    References       []string `json:"References"`
    CVSS             map[string]CVSSInfo `json:"CVSS"`
    CweIDs           []string `json:"CweIDs"`
}
```

> **关键设计**：注意 omitempty 标签——许多字段为 nil 时不出现在 JSON 中。在 Go Template 里访问这些字段前，必须用 if 做 nil 检查。

### 步骤二：实现自定义 CSV Writer

**目标**：编写一个输出 CSV 格式的 Writer，方便导入 Excel。

创建 `custom-writer/csv/csv_writer.go`：

```go
package csv

import (
    "context"
    "encoding/csv"
    "fmt"
    "io"

    "github.com/aquasecurity/trivy/pkg/types"
)

type CsvWriter struct {
    Output io.Writer
}

func NewWriter(output io.Writer) *CsvWriter {
    return &CsvWriter{Output: output}
}

func (w *CsvWriter) Write(ctx context.Context, report types.Report) error {
    writer := csv.NewWriter(w.Output)
    defer writer.Flush()

    // 写入 CSV 头部
    if err := writer.Write([]string{
        "CVE ID", "Severity", "Package", "Installed Version",
        "Fixed Version", "Title", "Artifact",
    }); err != nil {
        return err
    }

    // 遍历所有 Result 中的漏洞
    for _, result := range report.Results {
        for _, vuln := range result.Vulnerabilities {
            if err := writer.Write([]string{
                vuln.VulnerabilityID,
                vuln.Severity,
                vuln.PkgName,
                vuln.InstalledVersion,
                vuln.FixedVersion,
                vuln.Title,
                report.ArtifactName,
            }); err != nil {
                return err
            }
        }
    }

    // 写入 Summary 行
    total := 0
    for _, r := range report.Results {
        total += len(r.Vulnerabilities)
    }
    writer.Write([]string{fmt.Sprintf("TOTAL: %d vulnerabilities", total), "", "", "", "", "", ""})

    return nil
}
```

### 步骤三：解读 Table Renderer 的分发模式

**目标**：理解 pkg/report/table/table.go 的渲染分发机制。

源码结构（简化自 pkg/report/table/）：

```text
pkg/report/table/
├── table.go          # Table Writer 主逻辑 + Renderer 接口定义
├── vulnerability.go  # 漏洞渲染器
├── misconfig.go      # 配置错误渲染器
├── secret.go         # 密钥泄露渲染器
├── licensing.go      # 许可证渲染器
└── summary.go        # 摘要信息渲染器
```

Renderer 接口：

```go
// pkg/report/table/table.go
type Renderer interface {
    Render(result types.Result, options Options) string
}
```

Table Writer 的分发逻辑：

```go
func (w *Writer) render() string {
    var sections []string

    for _, result := range report.Results {
        var rendered string
        switch result.Class {
        case types.ClassOSPkg, types.ClassLangPkg:
            rendered = w.vulnerabilityRenderer.Render(result, w.options)
        case types.ClassConfig:
            rendered = w.misconfigRenderer.Render(result, w.options)
        case types.ClassSecret:
            rendered = w.secretRenderer.Render(result, w.options)
        case types.ClassLicense:
            rendered = w.pkgLicenseRenderer.Render(result, w.options)
        case types.ClassLicenseFile:
            rendered = w.fileLicenseRenderer.Render(result, w.options)
        default:
            continue
        }

        if rendered != "" {
            sections = append(sections, rendered)
        }
    }

    // 拼接 Summary + 各类型渲染结果
    summary := w.summaryRenderer.Render()
    return strings.Join(append([]string{summary}, sections...), "\n\n")
}
```

> **设计洞察**：新增一种 Result Class（如自定义资源），只需要新增一个 Renderer 实现并在 switch 中加一个 case。这是开闭原则的经典实践——对扩展开放，对修改封闭。

### 步骤四：解读 SARIF Writer 的格式映射

**目标**：理解 pkg/report/sarif.go 如何将内部结构映射到 SARIF 标准。

核心映射逻辑：

```go
// pkg/report/sarif.go (简化)
func (w *SarifWriter) Write(ctx context.Context, report types.Report) error {
    run := sarif.NewRunWithInformationURI(
        "Trivy",
        "https://github.com/aquasecurity/trivy",
    )

    for _, result := range report.Results {
        for _, vuln := range result.Vulnerabilities {
            // 将 DetectedVulnerability 映射为 SARIF Result
            run.CreateResultForRule(vuln.VulnerabilityID).
                WithLevel(sarifLevel(vuln.Severity)).
                WithMessage(sarif.NewTextMessage(vuln.Title)).
                WithLocations([]*sarif.Location{
                    {
                        PhysicalLocation: &sarif.PhysicalLocation{
                            ArtifactLocation: &sarif.ArtifactLocation{
                                URI: &report.ArtifactName,
                            },
                        },
                    },
                })
        }

        for _, misconf := range result.Misconfigurations {
            run.CreateResultForRule(misconf.ID).
                WithLevel(sarifLevel(misconf.Severity)).
                WithMessage(sarif.NewTextMessage(misconf.Description))
        }
    }

    return sarif.WriteReport(w.Output, run)
}
```

SARIF Level 映射：

```go
func sarifLevel(severity string) string {
    switch strings.ToUpper(severity) {
    case "CRITICAL", "HIGH":
        return "error"
    case "MEDIUM":
        return "warning"
    case "LOW":
        return "note"
    default:
        return "none"
    }
}
```

> **关键**：SARIF 标准中的 Level（error/warning/note/none）不等于 Trivy 的 Severity。必须做显式映射。漏掉这个映射就会导致 SARIF 报告中所有漏洞都是默认 Level。

### 步骤五：扩展报告系统的注册机制

**目标**：实现一个可注册的 Writer 工厂模式。

创建 `custom-writer/registry.go`：

```go
package customwriter

import (
    "context"
    "fmt"
    "io"
    "sync"

    "github.com/aquasecurity/trivy/pkg/types"
)

// CustomWriterFactory 自定义 Writer 的工厂函数
type CustomWriterFactory func(output io.Writer) Writer

// Writer 接口（与 Trivy 的 report.Writer 一致）
type Writer interface {
    Write(context.Context, types.Report) error
}

var (
    registry = make(map[string]CustomWriterFactory)
    mu       sync.RWMutex
)

// Register 注册一个新的 Writer 格式
func Register(format string, factory CustomWriterFactory) {
    mu.Lock()
    defer mu.Unlock()
    registry[format] = factory
}

// Get 获取已注册的 Writer 工厂
func Get(format string) (CustomWriterFactory, error) {
    mu.RLock()
    defer mu.RUnlock()

    factory, ok := registry[format]
    if !ok {
        return nil, fmt.Errorf("unknown custom format: %s", factory)
    }
    return factory, nil
}

// ListFormats 列出所有已注册的自定义格式
func ListFormats() []string {
    mu.RLock()
    defer mu.RUnlock()

    formats := make([]string, 0, len(registry))
    for f := range registry {
        formats = append(formats, f)
    }
    return formats
}
```

使用注册机制：

```go
// main.go
package main

import (
    "os"

    "github.com/cloud-sail/trivy-customwriter/csvwriter"
    customwriter "github.com/cloud-sail/trivy-customwriter/registry"
)

func init() {
    // 注册 CSV Writer
    customwriter.Register("csv", func(w io.Writer) customwriter.Writer {
        return csvwriter.NewWriter(w)
    })
}

func main() {
    formats := customwriter.ListFormats()
    // ["csv"]

    factory, _ := customwriter.Get("csv")
    writer := factory(os.Stdout)
    writer.Write(ctx, report)
}
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| Writer 接口 | 统一抽象；新增格式只需一个实现 | 缺少流式输出支持（大数据量时内存消耗大） |
| Renderer 分发 | 各类型独立渲染；开闭原则 | 新类型需要改 switch-case |
| Go Template | 强大灵活；零代码即可定制报告 | 学习曲线陡；nil 安全需手动处理 |
| SARIF 映射 | 符合行业标准；GitHub 原生支持 | 标准本身复杂；字段遗漏容易出错 |
| 插件式注册 | 可动态添加格式；无侵入扩展 | 全局注册表存在并发冲突的隐患 |

### 适用场景

1. **企业报告定制**：编写自定义 Writer 适配内部系统的数据格式。
2. **合规审计**：SARIF 是 GitHub/GitLab/Azure DevOps 的标准格式。
3. **数据导出与分析**：CSV Writer 打通 Excel/BI 工具。
4. **构建安全 Dashboard 后端**：从 types.Report 提取指标写入时序数据库。
5. **多格式报告自动化**：一条扫描流水线自动输出 Table + JSON + SARIF + CSV。

**不适用场景**：
1. 简单场景下，Trivy 内置格式已足够——不需要自定义 Writer。
2. 实时流式输出（如 WebSocket 推送）——现有 Writer 是同步的、批量的。

### 注意事项

- **types.Report 结构体有大变化**。Trivy 版本升级时，Report 的字段可能增减或重命名。自定义 Writer 需要做向后兼容测试。
- **Go Template 的 nil 安全**。务必在使用嵌套字段前做 nil 检查（if .CVSS then .CVSS.nvd.V3Score）。
- **Writer 的并发安全**。如果多个 goroutine 调用同一个 Writer.Write()，需要确保内部无共享可变状态。
- **大报告的内存问题**。对于包含 10000+ 漏洞的 Report，JSON Marshal 可能消耗大量内存。考虑分页或流式输出。

### 常见踩坑经验

**踩坑案例 1：模板中访问 nil map 导致 panic**
- **现象**：trivy convert --template 报错 nil pointer evaluating。
- **根因**：模板中直接访问了 CVSS.nvd.V3Score，但某些漏洞的 CVSS 字段为 nil map。
- **解法**：加多重 nil 保护：if and .CVSS .CVSS.nvd .CVSS.nvd.V3Score。

**踩坑案例 2：SARIF 报告在 GitHub UI 中不显示**
- **现象**：上传 SARIF 到 GitHub 后，Security Tab 中没有显示。
- **根因**：SARIF 的 run.tool.driver.name 为空或格式不符合规范。
- **解法**：确保 tool 对象正确填充；用 GitHub 的 sarif-tools 验证格式。

**踩坑案例 3：CSV Writer 中字段包含逗号导致列错位**
- **现象**：漏洞 Title 为 Multiple CVEs found in package A, B, C，导致 CSV 多出一列。
- **根因**：CSV 字段未加引号包裹。
- **解法**：使用 encoding/csv 标准库的 Write() 方法——它会自动处理字段的引号转义。

### 思考题

1. Trivy 的 Table Writer 使用 strings.Builder 在内存中构建完整输出，然后一次性写入 io.Writer。如果要输出一个包含 10 万条漏洞的巨大表格，这种设计会导致什么性能瓶颈？请设计一个基于流式输出的方案。

2. 如果要在不修改 Trivy 源码的前提下，实现一个新的输出格式（如 Excel .xlsx），有哪些可行方案？比较通过 Trivy Plugin 机制、通过报告后处理脚本、通过 Library 模式重新包装这三种方案的优劣。

> **推广计划**：本章建议在团队内部分享 pkg/report/ 的 Writer 接口设计——这是一个优秀的接口隔离和开闭原则实例。对于需要自研安全平台的团队，建议从编写一个 CSV Writer 开始（最简单），然后逐步升级到 JIRA Custom Fields Writer 和 Excel Writer。所有自定义 Writer 使用统一的 registry 机制管理，避免散落各地的转换脚本。
