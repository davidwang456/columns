# 第9章：漏洞评估与优先级排序实战

> 版本：Trivy v0.50+
> 面向人群：安全工程师、运维、开发负责人

---

## 1. 项目背景

### 业务场景

云帆科技的安全团队最近陷入了「漏洞泥潭」。运维老李用 Trivy 扫描了公司最核心的支付网关镜像，结果弹出了 312 个漏洞。报告像瀑布一样刷屏，开发组长阿杰当场崩溃：「312 个？我们团队就 8 个人，就算什么都不干，一个月也修不完啊！」

CTO 把安全负责人和老李叫到办公室：「我不关心有多少个漏洞，我只关心两个问题：第一，有没有漏洞能让黑客直接拿到用户银行卡信息？第二，如果要修，按什么顺序修？」

安全负责人拿着报告看了半天，发现这 312 个漏洞里有 180 个是 MEDIUM，90 个是 HIGH，42 个是 CRITICAL。但奇怪的是，很多 CRITICAL 漏洞对应的包只是一个内部工具引用的开发依赖，根本不会暴露在公网；而有一个 HIGH 级别的漏洞却位于直接处理支付请求的 OpenSSL 组件中，且公开网络上已经存在完整的利用代码（PoC）。

更棘手的是，团队内部对「严重性」的理解完全不一致。开发看 Trivy 的 `Severity` 列，认为 CRITICAL 必须全部立即修复；运维看 `Fixed Version`，认为只要能升级就能解决；而安全负责人知道，真正决定风险大小的不是 Severity  alone，而是「可利用性 × 资产暴露面 × 业务影响」。

与此同时，项目经理老王在准备下周的客户审计。客户要求提供「漏洞风险评估报告」，老王把 Trivy 的 JSON 导出直接发了过去，结果客户退回了三回：「你们这份报告只有 CVE 编号和 Severity，没有风险评分、没有修复优先级、没有业务影响分析，这不符合我们的安全治理要求。」

### 痛点放大

**第一，Severity 不等于风险。** Trivy 的 Severity 主要来自漏洞数据库的 CVSS 评分，它衡量的是漏洞的「技术危害性」，但没有考虑你的环境是否真的会被攻击。一个 CRITICAL 的缓冲区溢出，如果发生在完全离线的内网工具中，实际风险可能接近于零。

**第二，漏洞数量超出团队的修复能力。** 中型企业的镜像动辄几百个漏洞，而安全/开发团队的修复带宽有限。如果没有优先级排序，团队会陷入「修完 10 个又来 20 个」的无限循环，最终导致安全疲劳——所有人都对漏洞报告麻木了。

**第三，缺少可利用性情报。** 传统的漏洞报告只告诉你「这个漏洞存在」，但不告诉你「这个漏洞有没有被黑客实际利用」。如果某个 CVE 已经出现在 CISA KEV（Known Exploited Vulnerabilities）目录中，或者 EPSS 评分超过 0.9（表示 90% 概率近期会被利用），它的修复优先级应该远高于普通的 CRITICAL。

**第四，报告无法支撑决策。** 管理层需要知道「今天必须修什么」「这周计划修什么」「哪些可以接受」。原始的 CVE 列表无法满足这种分层决策需求。

**本章的核心目标是：建立一套基于 Severity + EPSS + CISA KEV + 资产暴露面的漏洞优先级排序体系，让团队把有限的修复资源投入到真正的「高价值目标」上。**

## 2. 项目设计

**场景**：云帆科技的漏洞优先级评审会，安全负责人、小胖（开发代表）和小白（运维代表）正在讨论如何处理支付网关的 312 个漏洞。

---

**小胖**：（把打印出来的漏洞报告摔在桌上）「312 个！其中 42 个 CRITICAL！安全团队这是要我们停掉所有业务专门修漏洞吗？」

**小白**：「你先别激动。Trivy 报的 CRITICAL 不等于都要今晚加班修。你看这个 `CVE-2023-XXXX`，它是 `lodash` 的一个原型污染漏洞，CRITICAL 级别，但我们的支付网关根本没用到 `lodash`，是构建工具链里的间接依赖。」

**小胖**：「那我怎么能一眼看出哪些是真正的威胁？」

**大师**：「这需要引入『风险评分』的概念。Trivy 的 Severity 只是风险的一个维度，就像考试分数只是能力的一个维度。一个高考 700 分的学生，如果志愿填错了，也可能落榜。同样，一个 CRITICAL 的漏洞，如果不在攻击路径上，风险也要打折。」

**小胖**：「那怎么判断它在不在攻击路径上？」

**大师**：「四个维度叠加：

1. **技术严重性（Severity / CVSS）**：漏洞本身有多危险？能否远程利用？是否需要认证？
2. **可利用性预测（EPSS）**：这个漏洞近期被实际利用的概率有多大？EPSS 是一个机器学习模型，综合了漏洞特征、暗网情报、Exploit 发布情况等因素，输出 0-1 的概率值。
3. **已知利用状态（CISA KEV）**：这个漏洞是否已经被黑客组织实际利用？CISA 维护了一个已知被利用漏洞的目录，进了这个名单的漏洞必须优先处理。
4. **资产暴露面**：存在漏洞的组件是否直接暴露在公网？是否处理敏感数据？是否是核心支付链路的一部分？」

**小白**：「也就是说，一个 HIGH 级别但 EPSS 0.95、在 CISA KEV 上、位于公网入口的漏洞，优先级应该高于一个 CRITICAL 但 EPSS 0.01、在内网工具里的漏洞？」

**大师**：「技术映射：完全正确。你可以把漏洞想象成『疾病』，Severity 是『疾病的致死率』，EPSS 是『这种疾病正在流行的概率』，CISA KEV 是『WHO 已宣布的传染病警报』，资产暴露面是『你是否有基础病』。一个致死率低的流行病（如新冠），对社会的威胁远大于一个致死率高但极罕见的疾病（如埃博拉在北极）。」

**小胖**：「那具体怎么排优先级？有没有公式？」

**大师**：「我给团队设计了一个优先级矩阵：

| 优先级 | 条件组合 | 处理时限 | 行动 |
|--------|----------|----------|------|
| P0 | CISA KEV 或 EPSS > 0.5 + CRITICAL/HIGH + 公网暴露 | 24 小时 | 立即修复，可中断发版 |
| P1 | CRITICAL + 生产环境 + 有 Fixed Version | 72 小时 | 排入当前 Sprint |
| P2 | HIGH + 生产环境 + 有 PoC/EPSS > 0.1 | 7 天 | 排入下一个 Sprint |
| P3 | MEDIUM + 生产环境 或 HIGH + 内网工具 | 30 天 | 常规迭代修复 |
| P4 | LOW 或 无 Fixed Version 或 开发测试环境 | 下次发版 | 视资源处理 |

关键原则是：**CISA KEV 一票否决**。只要进了 KEV 名单，无论 Severity 是什么，都按 P0 处理。」

**小白**：「Trivy 的报告里有 EPSS 和 CISA KEV 信息吗？」

**大师**：「Trivy 的 JSON 报告中有 `CweIDs`、`Cvss` 等字段，但 EPSS 和 CISA KEV 需要外部数据源补充。目前 Trivy 不直接内置 EPSS 查询，但你可以通过 `VulnerabilityID`（CVE 编号）去调用 EPSS API（`https://api.first.org/data/v1/epss`）和 CISA KEV JSON（`https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json`），然后和 Trivy 报告做关联。」

**小胖**：「听起来我们需要写个脚本把 Trivy 报告和这些外部数据打通？」

**小白**：「对。我可以写一个 Python 脚本：读取 Trivy 的 JSON → 提取所有 CVE → 批量查询 EPSS API → 检查是否在 CISA KEV 列表中 → 根据优先级矩阵打分 → 输出一份按 P0-P4 分层的修复清单。」

**大师**：「这个脚本应该成为团队的安全治理基础设施。每周一早上自动运行，把 P0 和 P1 推送到 Slack 的 #security-alerts 频道，把完整的优先级报告发到安全团队的邮箱。」

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+，已安装
- **Python 3.8+**：用于编写优先级排序脚本
- **网络**：可访问 `api.first.org` 和 `www.cisa.gov`（或通过代理）
- **测试镜像**：一个包含多个漏洞的镜像（如 `python:3.4-alpine`）

### 步骤二：编写漏洞优先级排序脚本

**目标**：整合 Trivy、EPSS 和 CISA KEV 数据，生成分层修复清单。

创建 `priority-scorer.py`：

```python
#!/usr/bin/env python3
"""
Trivy 漏洞优先级排序器
输入：Trivy JSON 报告
输出：按 P0-P4 分层的修复清单
"""

import json
import sys
import urllib.request
from datetime import datetime
from pathlib import Path

class VulnerabilityPrioritizer:
    def __init__(self):
        self.kev_set = self._load_cisa_kev()
        self.epss_cache = {}
    
    def _load_cisa_kev(self) -> set:
        """加载 CISA Known Exploited Vulnerabilities 列表"""
        try:
            url = "https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json"
            with urllib.request.urlopen(url, timeout=30) as resp:
                data = json.loads(resp.read())
            return {item["cveID"] for item in data.get("vulnerabilities", [])}
        except Exception as e:
            print(f"Warning: Failed to load CISA KEV: {e}", file=sys.stderr)
            return set()
    
    def _get_epss(self, cve_id: str) -> float:
        """查询 EPSS 评分，带缓存"""
        if cve_id in self.epss_cache:
            return self.epss_cache[cve_id]
        try:
            url = f"https://api.first.org/data/v1/epss?cve={cve_id}"
            with urllib.request.urlopen(url, timeout=10) as resp:
                data = json.loads(resp.read())
            if data.get("data"):
                score = float(data["data"][0]["epss"])
                self.epss_cache[cve_id] = score
                return score
        except Exception:
            pass
        self.epss_cache[cve_id] = 0.0
        return 0.0
    
    def _calculate_priority(self, vuln: dict, is_exposed: bool = True) -> str:
        """根据优先级矩阵计算漏洞等级"""
        cve = vuln.get("VulnerabilityID", "")
        severity = vuln.get("Severity", "UNKNOWN")
        has_fix = bool(vuln.get("FixedVersion"))
        
        # 查询外部情报
        in_kev = cve in self.kev_set
        epss = self._get_epss(cve)
        
        # P0: CISA KEV 或 高 EPSS + 高危 + 暴露
        if in_kev or (epss > 0.5 and severity in ("CRITICAL", "HIGH") and is_exposed):
            return "P0"
        
        # P1: CRITICAL + 生产 + 有修复
        if severity == "CRITICAL" and has_fix and is_exposed:
            return "P1"
        
        # P2: HIGH + 生产 + 有利用迹象
        if severity == "HIGH" and has_fix and (epss > 0.1 or in_kev):
            return "P2"
        
        # P3: MEDIUM + 生产 或 HIGH + 内网
        if severity in ("MEDIUM", "HIGH"):
            return "P3"
        
        # P4: 其他
        return "P4"
    
    def process(self, report_path: str, output_path: str):
        with open(report_path) as f:
            report = json.load(f)
        
        prioritized = {"P0": [], "P1": [], "P2": [], "P3": [], "P4": []}
        
        for result in report.get("Results", []):
            target = result.get("Target", "unknown")
            for vuln in result.get("Vulnerabilities", []):
                cve = vuln.get("VulnerabilityID", "N/A")
                priority = self._calculate_priority(vuln)
                epss = self._get_epss(cve)
                in_kev = cve in self.kev_set
                
                item = {
                    "cve": cve,
                    "pkg": vuln.get("PkgName", "N/A"),
                    "installed": vuln.get("InstalledVersion", "N/A"),
                    "fixed": vuln.get("FixedVersion", "N/A"),
                    "severity": vuln.get("Severity", "UNKNOWN"),
                    "epss": round(epss, 4),
                    "in_kev": in_kev,
                    "target": target,
                    "description": vuln.get("Description", "")[:100] + "..."
                }
                prioritized[priority].append(item)
        
        # 生成报告
        with open(output_path, 'w') as f:
            f.write(f"# 漏洞优先级报告\n\n")
            f.write(f"生成时间：{datetime.now().isoformat()}\n\n")
            
            for level in ["P0", "P1", "P2", "P3", "P4"]:
                items = prioritized[level]
                f.write(f"## {level} 级别（{len(items)} 个）\n\n")
                if not items:
                    f.write("无\n\n")
                    continue
                f.write("| CVE | 包名 | 当前版本 | 修复版本 | Severity | EPSS | CISA KEV |\n")
                f.write("|-----|------|----------|----------|----------|------|----------|\n")
                for item in items:
                    kev_badge = "🔴 YES" if item["in_kev"] else "No"
                    f.write(f"| {item['cve']} | {item['pkg']} | {item['installed']} | "
                           f"{item['fixed']} | {item['severity']} | {item['epss']} | {kev_badge} |\n")
                f.write("\n")
        
        print(f"Report generated: {output_path}")
        print(f"Summary: P0={len(prioritized['P0'])}, P1={len(prioritized['P1'])}, "
              f"P2={len(prioritized['P2'])}, P3={len(prioritized['P3'])}, P4={len(prioritized['P4'])}")

if __name__ == "__main__":
    if len(sys.argv) < 3:
        print(f"Usage: {sys.argv[0]} <trivy-report.json> <output.md>")
        sys.exit(1)
    
    prioritizer = VulnerabilityPrioritizer()
    prioritizer.process(sys.argv[1], sys.argv[2])
```

运行脚本：

```bash
python priority-scorer.py vuln-report.json priority-report.md
```

**预期输出**：

```
Summary: P0=2, P1=5, P2=8, P3=45, P4=252
```

**修复决策流程**：

1. **P0 级别**：立即通知值班工程师，24 小时内修复。如果修复版本可用，直接升级；如果修复版本不可用，评估临时缓解措施（如 WAF 规则、网络隔离）。
2. **P1 级别**：创建紧急工单，排入当前 Sprint，72 小时内完成修复验证。
3. **P2 级别**：创建标准工单，排入下一个 Sprint。
4. **P3 级别**：纳入技术债务 backlog，按资源情况安排。
5. **P4 级别**：记录到 `.trivyignore`（如果已评估接受），或留待下次发版处理。

### 步骤五：自动化周报生成

**目标**：每周一自动生成并推送漏洞优先级报告。

创建 `weekly-report.sh`：

```bash
#!/bin/bash
set -e

IMAGE="python:3.4-alpine"
DATE=$(date +%Y%m%d)
REPORT_DIR="/var/reports/security"
mkdir -p "$REPORT_DIR"

# 扫描
trivy image --format json --output "$REPORT_DIR/vuln-${DATE}.json" "$IMAGE"

# 排序
python priority-scorer.py "$REPORT_DIR/vuln-${DATE}.json" "$REPORT_DIR/priority-${DATE}.md"

# 推送到 Slack（需配置 webhook）
P0_COUNT=$(grep -c "## P0" "$REPORT_DIR/priority-${DATE}.md" || true)
if [ "$P0_COUNT" -gt 0 ]; then
    curl -X POST -H 'Content-type: application/json' \
        --data "{\"text\":\"🚨 周报：发现 P0 级别漏洞，请立即查看 $REPORT_DIR/priority-${DATE}.md\"}" \
        "$SLACK_WEBHOOK_URL"
fi

echo "Weekly report generated at $REPORT_DIR/priority-${DATE}.md"
```

加入 crontab：

```bash
# 每周一早上 9 点执行
0 9 * * 1 /opt/security/weekly-report.sh >> /var/log/weekly-report.log 2>&1
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 数据维度 | 整合 Severity + EPSS + KEV + 资产暴露面，比单维评分更科学 | EPSS 和 KEV 需要网络查询，离线环境需预缓存 |
| 优先级矩阵 | 分层清晰，便于管理层决策和资源分配 | 矩阵阈值（如 EPSS > 0.5）需要根据团队实际情况调整 |
| 自动化 | Python 脚本易于集成到 CI/定时任务 | 脚本需要持续维护，CISA KEV 格式变化时需适配 |
| 可复现性 | 基于 Trivy JSON 标准化输入，结果可复现 | 不同数据库版本的 Trivy 报告可能产生不同优先级结果 |
| 可操作性 | 输出直接对应修复时限和 Sprint 排期 | 缺少与 JIRA/工单系统的直接集成 |

### 适用场景

1. **漏洞数量爆炸的治理**：当单镜像漏洞超过 100 个时，用优先级排序避免团队陷入安全疲劳。
2. **红蓝对抗后的修复**：攻防演练后，用 EPSS + KEV 快速定位被利用风险最高的漏洞。
3. **客户安全审计**：提供分层修复计划和风险评分，满足企业级安全治理要求。
4. **Sprint 安全排期**：将 P0/P1 漏洞作为 Sprint 的必做项，P2/P3 作为常规技术债务。
5. **安全运营中心（SOC）**：自动化周报生成，持续监控漏洞态势。

**不适用场景**：
1. 漏洞数量极少（<10 个）的小型项目——直接全部修复即可，无需复杂排序。
2. 需要实时威胁情报（如 APT 组织正在利用的 0-day）——EPSS 和 KEV 有数天的滞后性。

### 注意事项

- **EPSS 的时效性**：EPSS 评分在漏洞公开后会快速变化，通常 CVE 公开后的前 7 天波动最大。建议对新生成的 P0 条目在 24 小时后重新评估 EPSS。
- **CISA KEV 的覆盖范围**：KEV 主要收录美国政府机构观测到的被利用漏洞，对某些地域或行业的特定威胁可能覆盖不足。
- **资产暴露面的动态变化**：今天在内网的组件，明天可能因配置变更暴露到公网。建议定期（如每月）重新评估资产暴露面。

### 常见踩坑经验

**踩坑案例 1：EPSS API 调用频繁触发速率限制**
- **现象**：脚本运行到一半报错 `HTTP 429 Too Many Requests`。
- **根因**：FIRST.org 的 EPSS API 对未认证用户有速率限制，批量查询几百个 CVE 容易触发。
- **解法**：在脚本中加入 `time.sleep(0.5)` 降低请求频率；或批量查询（EPSS API 支持一次查询多个 CVE，如 `?cve=CVE-2021-44228,CVE-2022-0778`）。

**踩坑案例 2：CISA KEV 列表格式变更导致解析失败**
- **现象**：脚本突然报错 `KeyError: 'vulnerabilities'`。
- **根因**：CISA 更新了 JSON 结构的字段名。
- **解法**：在脚本中增加异常处理，字段缺失时优雅降级；并设置监控，检测到解析失败时发送告警。

**踩坑案例 3：P0 漏洞过多导致团队恐慌**
- **现象**：首次运行脚本后输出 30 个 P0，团队以为被大规模攻击。
- **根因**：初始数据库版本较旧，一次性累积了大量历史 KEV 漏洞。
- **解法**：首次运行时应区分「新增 P0」和「历史遗留 P0」。建议脚本增加与上周报告的 diff 功能，只高亮新增的 P0。
