> 版本：Trivy v0.50+
> 面向人群：系统架构师、SRE、性能工程师

## 1. 项目背景

### 业务场景

云帆科技的业务增长超出了所有人的预期。从 50 个微服务扩展到 500 个，从 3 个 K8s 集群扩展到 20 个（跨 5 个 Region），镜像仓库中的镜像总数突破了 10 万个。安全团队的扫描基础设施面临前所未有的压力。

第 24 章的批量扫描优化在 50 个镜像规模下效果显著（从 4 小时降到 26 分钟），但在 10 万级镜像面前，一切都需要重新设计。当前的瓶颈数据令人窒息：

- **全量扫描耗时**：即使优化到每个镜像 30 秒（已经很优秀），10 万镜像 × 30 秒 = 833 小时（约 35 天）。串行模式下，一轮全量扫描需要超过一个月。
- **镜像仓库增长速度**：每天新增约 500 个镜像 tag，删除 200 个。10 万存量 + 日均 300 净增长。
- **重复扫描比例**：由于 80% 的镜像共享相同的基础层，如果不做去重，80% 的 CPU 时间浪费在重复计算上。
- **流量高峰**：每天凌晨 2 点 CI/CD 批量构建完成后，5 点之前必须完成所有新镜像的扫描——时间窗口只有 3 小时。

「我们需要一个能在 3 小时内完成 10 万镜像扫描的分布式系统。」CTO 给出了硬性指标，「并且能在 50 台机器上线性扩展。」

### 痛点放大

**第一，I/O、CPU、网络的三角瓶颈。** 镜像拉取是网络 I/O 密集、解压分析是 CPU 密集、DB 查询也是网络密集。如果三种瓶颈同时存在，单纯增加机器数量并不能获得线性增长。

**第二，镜像去重策略需要全局化。** 单机上的分层缓存已经实现（Applier 缓存），但跨机器的去重——两台不同机器上都扫描了 ubuntu:22.04 作为基础层——目前仍是重复计算。需要一个全局的 content-addressable 缓存。

**第三，DB 成为瓶颈。** trivy-db 是单文件 BoltDB，写入串行。10 万个镜像同时写入结果，BoltDB 直接被打垮。需要从同步写入改为异步批量写入。

**第四，故障恢复与幂等性。** 在 50 台机器的分布式系统中，部分节点必然出现临时故障（网络抖动、OOM、磁盘满）。扫描任务必须支持自动重试且不重复计算已完成的层。

**本章的核心目标是：设计一个能支撑 10 万+ 镜像规模的分布式扫描架构，包括全局去重缓存、数据库分片、任务队列优化，以及性能剖析（pprof/火焰图）定位热点。**

## 2. 项目设计

**场景**：云帆科技的性能峰会，小白（架构师）、大师（技术总监）、小胖（SRE）在讨论如何突破天花板。

---

**小胖**：「直接加机器啊——50 台机器同时扫，10 万镜像不就 2000 个/台，2000 × 30 秒 = 16 小时——还是太久。」

**小白**：「关键不是机器数量，是去重效率。10 万个镜像中，真正的『唯一基础层组合』可能只有 200 种。如果你能做到『每个唯一的基础层全集群只分析一次』，那 CPU 密集部分的计算量就从 10 万降到了 200——几乎 500 倍的减少。」

**大师**：「技术映射：分布式扫描就像流水线工厂。每个工件（镜像）到了流水线上，首先通过一个『X 光机』（全局缓存查询）扫描它的 digest。如果发现这个工件已经在数据库里有完整的分析记录了（另一台机器分析过），直接跳过——贴个标签（缓存命中）。只有全新的工件才真正进入流水线加工。」

**小胖**：「那具体的架构分几层？」

**小白**：「五层架构：

```
┌─────────────────────────────────────────────────┐
│  1. 资产发现层（Asset Discovery）                  │
│     自动同步 Harbor / ECR / ACR 镜像清单          │
│     每分钟增量更新；记录新增/删除 tag              │
├─────────────────────────────────────────────────┤
│  2. 智能调度层（Smart Scheduler）                  │
│     基于镜像变更频率、严重性、依赖关系的优先级       │
│     Critical 基础镜像 → 优先级最高                 │
│     3 个月未变动的老旧镜像 → 优先级最低              │
├─────────────────────────────────────────────────┤
│  3. 去重缓存层（Deduplication Cache）              │
│     全局 Content-Addressable 缓存（基于 layer digest）│
│     Redis Cluster（热层）+ S3（冷层）               │
│     命中率目标：> 95%                              │
├─────────────────────────────────────────────────┤
│  4. 分布式 Worker 层（Scan Workers）               │
│     K8s StatefulSet × 50 Pod（HPA 自动扩缩）       │
│     每个 Worker 独立拉取任务、执行扫描、上报结果      │
│     单 Worker 故障不丢失任务（Redis 队列 ACK 机制）   │
├─────────────────────────────────────────────────┤
│  5. 结果聚合层（Result Aggregation）               │
│     PostgreSQL TimescaleDB（时序存储）             │
│     增量报告：只展示相对于上次的新增/修复漏洞        │
│     Grafana 全局态势展示                           │
└─────────────────────────────────────────────────┘
```

**大师**：「其中最关键的设计是第三层——全局去重缓存。传统的单机缓存（Applier 的 FS/Redis 缓存）只能在一个节点内复用。全局去重缓存的原理是：用 layer digest 的 SHA256 作为 key，已分析的结果作为 value。任何节点在分析一个 layer 前，先查全局缓存——命中则零 CPU 开销。」

**小胖**：「那 DB 分片是什么？」

**小白**：「扫描结果不需要全写到一个数据库里。按镜像的 registry（harbor/ecr/acr）或按镜像名的 hash 分片。比如：

- Shard 0：Harbor 生产镜像
- Shard 1：ECR 镜像
- Shard 2：开发/测试镜像

每个 Shard 是独立的 PostgreSQL 实例。汇总查询时通过一个轻量的 query router 并发查询所有 Shard 再聚合。这样任何一个 Shard 挂掉都不影响其他 Shard。」

### 步骤一：构建全局去重缓存

**目标**：实现跨节点的 content-addressable 缓存，消除重复分析。

```go
// cache/global-dedup.go
package cache

import (
    "context"
    "crypto/sha256"
    "encoding/hex"
    "encoding/json"
    "time"

    "github.com/redis/go-redis/v9"
)

// GlobalDedupCache 全局去重缓存
type GlobalDedupCache struct {
    redis *redis.Client
    ttl   time.Duration
}

func NewGlobalDedupCache(redisAddr string, ttl time.Duration) *GlobalDedupCache {
    return &GlobalDedupCache{
        redis: redis.NewClient(&redis.Options{Addr: redisAddr}),
        ttl:   ttl,
    }
}

// LayerDigest 计算 layer 内容的 SHA256 摘要
func LayerDigest(content []byte) string {
    h := sha256.Sum256(content)
    return "sha256:" + hex.EncodeToString(h[:])
}

// GetLayerAnalysis 从全局缓存中获取 layer 的分析结果
func (c *GlobalDedupCache) GetLayerAnalysis(ctx context.Context, digest string) (*LayerAnalysis, error) {
    key := "dedup:layer:" + digest
    data, err := c.redis.Get(ctx, key).Bytes()
    if err != nil {
        if err == redis.Nil {
            return nil, nil // 缓存未命中
        }
        return nil, err
    }

    var analysis LayerAnalysis
    if err := json.Unmarshal(data, &analysis); err != nil {
        return nil, err
    }

    return &analysis, nil
}

// SetLayerAnalysis 存储 layer 的分析结果
func (c *GlobalDedupCache) SetLayerAnalysis(ctx context.Context, digest string, analysis *LayerAnalysis) error {
    data, err := json.Marshal(analysis)
    if err != nil {
        return err
    }

    key := "dedup:layer:" + digest
    return c.redis.Set(ctx, key, data, c.ttl).Err()
}

// LayerAnalysis 单个 layer 的分析结果（从缓存恢复，避免重复分析）
type LayerAnalysis struct {
    Digest       string              `json:"digest"`
    AnalyzedBy   string              `json:"analyzed_by"`    // Worker ID
    AnalyzedAt   time.Time           `json:"analyzed_at"`
    OS           *OSInfo             `json:"os,omitempty"`
    Packages     []PackageInfo       `json:"packages,omitempty"`
    Applications []AppInfo           `json:"applications,omitempty"`
    CacheHit     bool                `json:"cache_hit"`
}

type OSInfo struct {
    Family string `json:"family"`
    Name   string `json:"name"`
}

type PackageInfo struct {
    Name    string `json:"name"`
    Version string `json:"version"`
}

type AppInfo struct {
    Type     string `json:"type"`
    FilePath string `json:"file_path"`
    Libs     []PackageInfo `json:"libs"`
}
```

### 步骤二：实现画像去重扫描逻辑

**目标**：基于全局去重缓存优化扫描流程。

```go
// worker/scanner.go
package worker

import (
    "context"
    "fmt"
    "time"

    "github.com/cloud-sail/dist-scanner/cache"
)

type DistributedScanner struct {
    cache      *cache.GlobalDedupCache
    workerID   string
    scanStats  *ScanStats
}

type ScanStats struct {
    LayersTotal      int64
    LayersCacheHit   int64
    LayersAnalyzed   int64
    TotalDuration    time.Duration
}

func (s *DistributedScanner) ScanImage(ctx context.Context, imageRef string) (*ImageScanResult, error) {
    start := time.Now()
    defer func() { s.scanStats.TotalDuration += time.Since(start) }()

    // Step 1: 获取镜像的 layer 列表
    layers, err := s.getImageLayers(ctx, imageRef)
    if err != nil {
        return nil, fmt.Errorf("get layers: %w", err)
    }

    result := &ImageScanResult{
        Image:  imageRef,
        Layers: make([]cache.LayerAnalysis, 0, len(layers)),
    }

    // Step 2: 逐层检查全局去重缓存
    var layersToAnalyze []Layer
    for _, layer := range layers {
        s.scanStats.LayersTotal++

        cached, err := s.cache.GetLayerAnalysis(ctx, layer.Digest)
        if err == nil && cached != nil {
            // 缓存命中！跳过分析
            cached.CacheHit = true
            result.Layers = append(result.Layers, *cached)
            s.scanStats.LayersCacheHit++
            continue
        }

        layersToAnalyze = append(layersToAnalyze, layer)
    }

    if len(layersToAnalyze) == 0 {
        result.AllCached = true
        result.Duration = time.Since(start)
        fmt.Printf("[%s] %s: 100%% cache hit (%d layers)\n",
            s.workerID, imageRef, len(layers))
        return result, nil
    }

    // Step 3: 只对未缓存的层做分析
    for _, layer := range layersToAnalyze {
        analysis, err := s.analyzeLayer(ctx, layer)
        if err != nil {
            return nil, fmt.Errorf("analyze layer %s: %w", layer.Digest[:12], err)
        }

        analysis.AnalyzedBy = s.workerID
        analysis.AnalyzedAt = time.Now()
        analysis.Digest = layer.Digest

        // 写入全局缓存供其他节点复用
        s.cache.SetLayerAnalysis(ctx, layer.Digest, analysis)

        result.Layers = append(result.Layers, *analysis)
        s.scanStats.LayersAnalyzed++
    }

    result.Duration = time.Since(start)
    hitRate := float64(s.scanStats.LayersCacheHit) / float64(s.scanStats.LayersTotal) * 100
    fmt.Printf("[%s] %s: done in %v (cache hit rate: %.1f%%)\n",
        s.workerID, imageRef, result.Duration, hitRate)

    return result, nil
}

type Layer struct {
    Digest string
    Size   int64
}

type ImageScanResult struct {
    Image     string
    AllCached bool
    Duration  time.Duration
    Layers    []cache.LayerAnalysis
}
```

### 步骤三：实现智能任务调度器

**目标**：基于镜像优先级和变更频率的任务调度。

```go
// scheduler/scheduler.go
package scheduler

import (
    "context"
    "encoding/json"
    "sort"
    "time"

    "github.com/redis/go-redis/v9"
)

type Priority int

const (
    PriorityCritical Priority = iota  // 0: 基础镜像 + CRITICAL 漏洞
    PriorityHigh                       // 1: 生产镜像
    PriorityMedium                     // 2: 最近 24h 内变更
    PriorityLow                        // 3: 3 天内变更
    PriorityBackground                 // 4: 老旧镜像
)

type ScanTask struct {
    ID        string    `json:"id"`
    Image     string    `json:"image"`
    Priority  Priority  `json:"priority"`
    CreatedAt time.Time `json:"created_at"`
    Retries   int       `json:"retries"`
    MaxRetry  int       `json:"max_retry"`
}

type SmartScheduler struct {
    redis  *redis.Client
    queues map[Priority]string // priority -> Redis queue name
}

func NewSmartScheduler(redisAddr string) *SmartScheduler {
    return &SmartScheduler{
        redis: redis.NewClient(&redis.Options{Addr: redisAddr}),
        queues: map[Priority]string{
            PriorityCritical:    "scan:queue:p0",
            PriorityHigh:        "scan:queue:p1",
            PriorityMedium:      "scan:queue:p2",
            PriorityLow:         "scan:queue:p3",
            PriorityBackground:  "scan:queue:p4",
        },
    }
}

func (s *SmartScheduler) Enqueue(ctx context.Context, task ScanTask) error {
    data, _ := json.Marshal(task)
    queue := s.queues[task.Priority]
    return s.redis.LPush(ctx, queue, data).Err()
}

func (s *SmartScheduler) Dequeue(ctx context.Context, workerID string) (*ScanTask, error) {
    // 按优先级从高到低尝试拉取
    for p := PriorityCritical; p <= PriorityBackground; p++ {
        queue := s.queues[p]
        result, err := s.redis.BRPop(ctx, 1*time.Second, queue).Result()
        if err != nil {
            continue
        }

        var task ScanTask
        json.Unmarshal([]byte(result[1]), &task)

        // 将任务标记为 in-progress（防止重复分配）
        s.redis.Set(ctx, "scan:in_progress:"+task.ID, workerID, 30*time.Minute)

        return &task, nil
    }

    return nil, nil // 所有队列为空
}

// PrioritizeImage 根据镜像元数据计算优先级
func PrioritizeImage(image string, metadata *ImageMetadata) Priority {
    // Priority 0: 基础镜像 + 有已知 CRITICAL 漏洞
    if metadata.IsBaseImage && metadata.HasCriticalVulns {
        return PriorityCritical
    }

    // Priority 1: 生产标签
    if metadata.IsProduction {
        return PriorityHigh
    }

    // Priority 2: 24 小时内变更
    if time.Since(metadata.LastModified) < 24*time.Hour {
        return PriorityMedium
    }

    // Priority 3: 3 天内变更
    if time.Since(metadata.LastModified) < 72*time.Hour {
        return PriorityLow
    }

    // Priority 4: 老旧镜像（3 个月未变更 + 无已知漏洞）
    return PriorityBackground
}
```

### 步骤五：数据库分片与结果聚合

**目标**：将扫描结果分片写入，通过 Query Router 聚合查询。

```go
// storage/shard_router.go
package storage

import (
    "context"
    "crypto/md5"
    "encoding/hex"
    "fmt"
    "sync"

    "github.com/jackc/pgx/v5/pgxpool"
)

type ShardRouter struct {
    shards    []*pgxpool.Pool
    shardName []string
    mu        sync.RWMutex
}

func NewShardRouter(connStrings []string) (*ShardRouter, error) {
    router := &ShardRouter{
        shards:    make([]*pgxpool.Pool, len(connStrings)),
        shardName: make([]string, len(connStrings)),
    }

    for i, cs := range connStrings {
        pool, err := pgxpool.New(context.Background(), cs)
        if err != nil {
            return nil, fmt.Errorf("connect to shard %d: %w", i, err)
        }
        router.shards[i] = pool
        router.shardName[i] = fmt.Sprintf("shard-%d", i)
    }

    return router, nil
}

// GetShard 通过镜像名的 hash 确定归属 Shard
func (r *ShardRouter) GetShard(imageName string) *pgxpool.Pool {
    h := md5.Sum([]byte(imageName))
    index := int(h[0]) % len(r.shards)
    return r.shards[index]
}

// InsertScanResult 写入扫描结果到对应 Shard
func (r *ShardRouter) InsertScanResult(ctx context.Context, imageName string, result json.RawMessage) error {
    pool := r.GetShard(imageName)
    _, err := pool.Exec(ctx, `
        INSERT INTO scan_results (image_name, result_json, scanned_at)
        VALUES ($1, $2, NOW())
    `, imageName, result)
    return err
}

// QueryAll 跨所有 Shard 并发查询并聚合
func (r *ShardRouter) QueryAll(ctx context.Context, query string, args ...interface{}) ([]interface{}, error) {
    var wg sync.WaitGroup
    resultsCh := make(chan []interface{}, len(r.shards))
    errorsCh := make(chan error, len(r.shards))

    for _, pool := range r.shards {
        wg.Add(1)
        go func(p *pgxpool.Pool) {
            defer wg.Done()

            rows, err := p.Query(ctx, query, args...)
            if err != nil {
                errorsCh <- err
                return
            }
            defer rows.Close()

            // 收集结果...
            resultsCh <- nil
        }(pool)
    }

    wg.Wait()
    close(resultsCh)
    close(errorsCh)

    // 聚合所有 Shard 的结果
    var allResults []interface{}
    for r := range resultsCh {
        allResults = append(allResults, r...)
    }

    return allResults, nil
}
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 全局去重缓存 | 500x 计算量减少；线性扩展 | Redis 内存容量是瓶颈（需集群模式） |
| 智能调度 | 高危镜像秒级响应；老旧镜像不占紧急资源 | 元数据同步需实时准确 |
| DB 分片 | 写入吞吐量线性扩展 | 跨 Shard 查询增加复杂度 |
| pprof 剖析 | 精确量化性能瓶颈 | 需要 Go 源码理解能力 |
| Worker 弹性伸缩 | K8s HPA 按队列深度自动扩缩 | 冷启动延迟（约 30 秒） |

### 适用场景

1. **日均扫描 > 5000 次**：必须上分布式架构。
2. **多地域部署**：Worker 就近拉取镜像，减少跨 Region 网络延迟。
3. **基础镜像频繁变更**：全局去重缓存在基础镜像变更时收益最大。
4. **SLA 严格的场景**：P0 镜像的扫描延迟从「等待资源」降到「秒级」。
5. **成本优化**：通过去重 + 智能调度，用 1/5 的计算资源完成相同的扫描量。

**不适用场景**：
1. 扫描量 < 2000 次/天——单机+缓存即可。
2. 团队无 Redis/PostgreSQL 运维能力——分布式复杂度可能超出收益。

### 注意事项

- **Redis Cluster 必须开启持久化（AOF）**。如果 Redis 宕机重启后所有去重数据丢失，所有 Worker 将重新全量分析——系统短暂瘫痪。
- **Worker 的 blob 下载并发数要控制**。100 个 Worker 同时从同一个 Harbor 下载镜像层，可能打满 Registry 连接数。
- **PostgreSQL TimescaleDB 的 chunk 配置**。按日分 chunk，保留策略建议 90 天热数据 + 归档到 S3。
- **Worker 的资源 limit 要精确设置**。不要让单个 Worker 无限膨胀内存——大镜像（8GB+）可能需要独立的高内存 Worker 池。

### 常见踩坑经验

**踩坑案例 1：Redis 全局缓存 OOM 导致所有 Worker 卡死**
- **现象**：大规模扫描进行到 50% 时，所有 Worker 同时变慢。
- **根因**：Redis 内存达到 maxmemory，eviction 策略（LRU）开始淘汰缓存。Worker 发现缓存未命中，开始重复分析已扫过的层——恶性循环。
- **解法**：监控 Redis 内存使用率（告警阈值 70%）；配置 Redis Cluster 的横向扩展；设置去重缓存的 TTL 为 14 天自动过期。

**踩坑案例 2：Cross-Regional 镜像拉取导致 Worker 大量超时**
- **现象**：上海 Worker 扫描存放在新加坡 ECR 的镜像，耗时是预期的 10 倍。
- **根因**：Worker 没有感知 Registry 的地理位置，随机分配了跨 Region 的镜像。
- **解法**：Worker 分区域部署——每个 Region 的 Worker 只扫本 Region 的 Registry；在 Region 间同步结果即可。

**踩坑案例 3：PostgreSQL Shard 的 JOIN 查询性能差**
- **现象**：跨 Shard 查询「过去 7 天新增 CRITICAL 漏洞」耗时 5 秒+。
- **根因**：各 Shard 独立返回大数据集，聚合 node 在内存中做 JOIN——等价于把本属于数据库的工作搬到了应用层。
- **解法**：每个 Shard 预先计算好聚合结果返回（如 `SUM(critical_count)`），聚合 node 只做简单的加法 merge。
