## 1. 项目背景

### 业务场景

云帆科技已经成为一家拥有 500+ 微服务、20 个 K8s 集群（跨 5 个 Region）、日均 1 万+ 镜像推送量的互联网企业。安全团队从最初的「一人一把梭」发展到 15 人的安全工程团队。但安全基础设施却还停留在「散落的脚本 + 开源工具组合」的阶段：

- 镜像扫描：Trivy CLI + Harbor 内置扫描器，但两者没有统一调度。
- K8s 安全：Trivy Operator，但配置基线、漏洞趋势、合规报告分散在 Grafana、DefectDojo、Confluence 三个地方。
- CI/CD 门禁：GitHub Actions Reusable Workflow（第 31 章），但随项目数量增长，Workflow 的版本管理成为灾难。
- 漏洞闭环：DefectDojo + JIRA，但两套系统之间的同步偶尔断裂，出现孤儿工单。
- 策略管理：Polices Git 仓库（第 30 章），但新策略的生效周期长达 1 周——从 PR 合并到所有执行点同步到生效。

CTO 下达了终极任务：「在 3 个月内，将散落的安全工具替换为一个自研的统一安全平台（代号『Sentinel』）。这个平台必须满足：日均扫描 1 万+ 镜像、P99 扫描延迟 < 2 分钟、平台可用性 99.9%、所有安全事件在 5 分钟内可追溯。」

本章是高级篇的综合实战，也是整个专栏的收官之作。我们将整合前 39 章的所有知识点，从零设计并实现一个企业级安全扫描平台。

### 痛点放大

**第一，工具整合难度。** 不是简单的 Docker Compose up——而是要设计统一的架构，使得 Trivy Library、OPA Rego 引擎、自研 Dashboard、通知系统、任务调度器在同一个架构下协同工作。

**第二，数据一致性。** 同一个镜像在 CI、Registry、K8s 三个执行点扫描出的结果可能不同（DB 版本差异、缓存过期）。平台需要一个统一的数据模型来消弭这种差异。

**第三，平台自身的可靠性。** 如果安全平台挂了，所有依赖它的服务（CI 门禁、镜像准入、K8s 准入控制）全部受影响。安全平台本身就是最高优先级的服务——但它自己也需要安全扫描。

**第四，多租户隔离。** 500 个微服务分属 8 个业务团队。每个团队只能看到自己的安全数据，不能看到其他团队的。同时，平台管理员需要全公司视角。

**小白**：「六个层的职责非常清晰。我需要强调几个关键设计决策：

**决策 1：统一数据模型（Unified Finding Model）**

所有扫描结果——不管来自 CI、Registry、K8s Operator——都归一化为一个 Finding：

```go
type Finding struct {
    ID          string    // UUID v7
    Source      string    // ci / registry / k8s-operator
    Artifact    string    // 镜像名 / K8s 资源名 / 文件路径
    Type        string    // vulnerability / misconfiguration / secret / license
    Severity    string    // CRITICAL / HIGH / MEDIUM / LOW
    Status      string    // active / fixed / risk_accepted / false_positive
    FirstSeen   time.Time
    LastSeen    time.Time
    FixedAt     *time.Time
    CVE         string    // CVE ID (for vulnerabilities)
    Package     string
    Version     string
    FixedVer    string
    Metadata    json.RawMessage  // 灵活扩展
}
```

这个统一模型的好处是：同一个镜像的同一个漏洞如果同时在 CI 和 K8s 中出现，只创建一条 Finding，归并 `Sources` 数组。平台查询时不重复显示。

**决策 2：事件驱动架构 + 异步处理**

所有触发（镜像推送、PR 提交、K8s 事件）都先转化为『事件』，进入事件总线（NATS JetStream），再分发到对应处理器。好处：
- 触发方和处理方解耦——Harbor 不知道谁是扫描器，只知道『我产生了一个 image.pushed 事件』。
- 事件持久化可重放——如果扫描 Worker 故障重启，未处理的事件不会丢失。
- 分布式追踪——每个事件带 trace ID，跨服务的调用链可追踪。

**决策 3：分级故障隔离**

平台各层独立部署、独立扩缩、独立故障恢复：
- 扫描引擎挂了 → 新扫描请求排队等待 → 已有结果不受影响。
- 数据存储挂了 → 只读模式（已有结果可查，但不能入库新结果）→ 后续事件继续排队。
- 门户层挂了 → 不影响后台扫描和告警。
- 即使全平台挂了 → CI 门禁切回纯 Trivy CLI fallback 模式。

**决策 4：自举——平台扫描自身**

Sentinel 平台自己也是容器化的。它的部署 YAML、Dockerfile、依赖包也必须经过 Trivy 扫描——而且这个扫描不能用 Sentinel 自身（鸡蛋篮子问题）。解决方案：CI 中保留一份最小化的 Trivy CLI fallback，专门扫描 Sentinel 自己的代码和镜像。

### 步骤一：搭建事件驱动的核心架构

**目标**：所有安全事件通过 NATS JetStream 流转。

```go
// event/bus.go
package event

import (
    "encoding/json"
    "time"

    "github.com/nats-io/nats.go"
)

type EventBus struct {
    js nats.JetStreamContext
}

type SecurityEvent struct {
    ID          string          `json:"id"`
    Type        string          `json:"type"`      // image.pushed, scan.completed, finding.created
    Source      string          `json:"source"`    // harbor, github-actions, trivy-operator
    Subject     string          `json:"subject"`   // 影响对象
    Timestamp   time.Time       `json:"timestamp"`
    TraceID     string          `json:"trace_id"`
    Data        json.RawMessage `json:"data"`
}

// 事件类型常量
const (
    EventImagePushed      = "image.pushed"
    EventImageScanned     = "image.scanned"
    EventFindingCreated   = "finding.created"
    EventFindingResolved  = "finding.resolved"
    EventPolicyViolation  = "policy.violation"
    EventSLAWarning       = "sla.warning"
)

func (b *EventBus) Publish(ctx context.Context, event SecurityEvent) error {
    // 发布到 JetStream 主题，确保持久化
    subject := fmt.Sprintf("sentinel.events.%s", event.Type)
    data, _ := json.Marshal(event)
    _, err := b.js.Publish(subject, data)
    return err
}

func (b *EventBus) Subscribe(eventType string, handler func(SecurityEvent) error) (*nats.Subscription, error) {
    subject := fmt.Sprintf("sentinel.events.%s", eventType)
    return b.js.Subscribe(subject, func(msg *nats.Msg) {
        var event SecurityEvent
        json.Unmarshal(msg.Data, &event)
        handler(event)
    })
}
```

### 步骤二：实现资产发现服务

**目标**：自动从 Harbor/ECR/ACR 发现所有镜像，维护全公司镜像清单。

```go
// discovery/discoverer.go
package discovery

type AssetDiscoverer struct {
    registries []RegistryClient
    eventBus   *event.EventBus
    db         *sql.DB
}

type RegistryClient interface {
    ListImages(ctx context.Context, project string) ([]ImageAsset, error)
    WatchEvents(ctx context.Context, project string) (<-chan ImageEvent, error)
}

type ImageAsset struct {
    Registry   string    // harbor / ecr / acr
    Project    string
    Repository string
    Tags       []string
    Digest     string
    PushTime   time.Time
    Size       int64
}

func (d *AssetDiscoverer) Run(ctx context.Context) error {
    // 1. 初次启动：全量同步
    for _, reg := range d.registries {
        images, _ := reg.ListImages(ctx, "*")
        for _, img := range images {
            d.upsertAsset(ctx, img)
            // 发送新资产发现事件
            d.eventBus.Publish(ctx, SecurityEvent{
                Type:    "asset.discovered",
                Subject: img.Digest,
                Data:    marshal(img),
            })
        }
    }

    // 2. 持续监听：增量更新
    for _, reg := range d.registries {
        events, _ := reg.WatchEvents(ctx, "*")
        go func(ch <-chan ImageEvent) {
            for evt := range ch {
                switch evt.Type {
                case "push":
                    d.upsertAsset(ctx, evt.Image)
                    d.eventBus.Publish(ctx, SecurityEvent{
                        Type:    EventImagePushed,
                        Subject: evt.Image.Digest,
                    })
                case "delete":
                    d.deleteAsset(ctx, evt.Image.Digest)
                }
            }
        }(events)
    }

    return nil
}
```

### 步骤三：实现智能调度器（整合第 24、38 章）

**目标**：根据资产发现服务的事件，自动生成扫描任务并按优先级调度。

```go
// scheduler/scheduler.go
package scheduler

type SentinelScheduler struct {
    eventBus    *event.EventBus
    taskQueue   *TaskQueue
    workerPool  *WorkerPool
    db          *sql.DB
}

func (s *SentinelScheduler) Run(ctx context.Context) error {
    // 监听 image.pushed 事件 → 生成扫描任务
    s.eventBus.Subscribe(EventImagePushed, func(evt SecurityEvent) error {
        var img ImageAsset
        json.Unmarshal(evt.Data, &img)

        task := ScanTask{
            Image:    img.FullName(),
            Priority: calculatePriority(img),
            Source:   "asset-discoverer",
        }

        return s.taskQueue.Enqueue(ctx, task)
    })

    // 监听 policy.updated 事件 → 重新扫描受影响的镜像
    s.eventBus.Subscribe("policy.updated", func(evt SecurityEvent) error {
        // 找到所有涉及到的镜像 → 生成重扫任务
        return s.scheduleRescanForPolicy(ctx, evt)
    })

    // 定时：每日全量扫描低优先级镜像
    go s.dailyBackgroundScan(ctx)

    return nil
}

func calculatePriority(img ImageAsset) Priority {
    // Critical: 基础镜像 / 有已知 CRITICAL 漏洞
    if img.IsBaseImage || img.Tag == "latest" {
        return PriorityCritical
    }
    // High: 生产环境 tag
    if strings.HasPrefix(img.Tag, "prod-") || img.Tag == "stable" {
        return PriorityHigh
    }
    // Medium: 24 小时内推送
    if time.Since(img.PushTime) < 24*time.Hour {
        return PriorityMedium
    }
    // Low: 其余
    return PriorityLow
}
```

// api/dashboard.go
package api

type DashboardAPI struct {
    findingRepo *FindingRepository
    policyRepo  *PolicyRepository

type DashboardData struct {
    // 风险概览

}

type TrendPoint struct {
    Date        string `json:"date"`
    Critical    int    `json:"critical"`
    High        int    `json:"high"`
    Resolved    int    `json:"resolved"`
}

type ImageRisk struct {
    Image        string `json:"image"`
    Critical     int    `json:"critical"`
    High         int    `json:"high"`
    LastScanned  string `json:"last_scanned"`
}

func (api *DashboardAPI) GetOverview(ctx context.Context) (*DashboardData, error) {
    data := &DashboardData{}

    // 并发查询各维度数据
    var wg sync.WaitGroup

    wg.Add(1)
    go func() {
        defer wg.Done()
        data.FindingsBySeverity, _ = api.findingRepo.CountBySeverity(ctx)
        data.FindingsByStatus, _ = api.findingRepo.CountByStatus(ctx)
    }()

    wg.Add(1)
    go func() {
        defer wg.Done()
        data.TotalImages, _ = api.assetRepo.CountTotal(ctx)
        data.ScannedToday, _ = api.findingRepo.CountScannedToday(ctx)
    }()

    wg.Add(1)
    go func() {
        defer wg.Done()
        data.WeeklyTrend, _ = api.findingRepo.GetWeeklyTrend(ctx, 12)
    }()

    wg.Add(1)
    go func() {
        defer wg.Done()
        data.TopRiskyImages, _ = api.findingRepo.GetTopRiskyImages(ctx, 20)
    }()

    wg.Wait()
    return data, nil
}

```bash
#!/bin/bash
# sentinel-acceptance.sh
# Sentinel 平台验收测试

echo "=== Sentinel Platform Acceptance Test ==="

# === 指标 1：日均扫描能力 (> 1万) ===
echo "[Test 1] Throughput: 10,000 scans/day"
SCANS_24H=$(curl -s http://sentinel-api/metrics | grep 'sentinel_scans_total' | tail -1 | awk '{print $2}')
if [ "$SCANS_24H" -ge 10000 ]; then
    echo "  PASS: $SCANS_24H scans in last 24h (target: >= 10,000)"
else
    echo "  FAIL: Only $SCANS_24H scans (target: >= 10,000)"
fi

# === 指标 2：P99 扫描延迟 (< 2 分钟) ===
echo "[Test 2] Latency: P99 < 2 minutes"
P99_MS=$(curl -s 'http://prometheus:9090/api/v1/query?query=histogram_quantile(0.99,sentinel_scan_duration_seconds_bucket)' | jq -r '.data.result[0].value[1]')
P99_SEC=$(echo "scale=1; $P99_MS" / 1 | bc)
if [ "$(echo "$P99_SEC < 120" | bc)" -eq 1 ]; then
    echo "  PASS: P99 latency = ${P99_SEC}s (target: < 120s)"
else
    echo "  FAIL: P99 latency = ${P99_SEC}s (target: < 120s)"
fi

# === 指标 3：平台可用性 (> 99.9%) ===
echo "[Test 3] Availability: > 99.9%"
UPTIME=$(curl -s http://sentinel-api/health | jq -r '.uptime_seconds')
AVAILABILITY=$(echo "scale=3; 1 - ($(curl -s http://sentinel-api/metrics | grep 'sentinel_downtime_seconds_total') / $UPTIME)" | bc)
if [ "$(echo "$AVAILABILITY > 0.999" | bc)" -eq 1 ]; then
    echo "  PASS: Availability = $AVAILABILITY (target: > 0.999)"
else
    echo "  FAIL: Availability = $AVAILABILITY (target: > 0.999)"
fi

# === 指标 4：端到端延迟（镜像推送 → Dashboard 可查）(< 5 分钟) ===
echo "[Test 4] End-to-end: push → scan → dashboard < 5min"
TEST_IMAGE="sentinel-test-$(date +%s)"
docker tag alpine:latest harbor.internal/test/$TEST_IMAGE:latest
docker push harbor.internal/test/$TEST_IMAGE:latest

# 轮询 Dashboard 直到结果出现（最多 5 分钟）
START=$(date +%s)
while true; do
    FOUND=$(curl -s "http://sentinel-api/api/v1/findings?artifact=$TEST_IMAGE" | jq '.total')
    if [ "$FOUND" -gt 0 ]; then
        ELAPSED=$(($(date +%s) - START))
        echo "  PASS: Finding appeared in $ELAPSED seconds (target: < 300s)"
        break
    fi
    if [ $(($(date +%s) - START)) -gt 300 ]; then
        echo "  FAIL: Timeout after 300s"
        break
    fi
    sleep 5
done

echo "=== Acceptance Test Complete ==="
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 统一数据模型 | 跨源去重；统一查询接口 | 初期定义和维护成本高 |
| 事件驱动 | 松耦合；可追踪（TraceID） | NATS JetStream 运维较复杂 |
| 智能调度 | 高危镜像秒级响应 | 优先级算法需要持续调优 |
| 分级故障隔离 | 部分故障不影响整体 | 增加了部署和运维的层数 |
| 自举扫描 | 平台自身也在安全审计范围内 | fallback 机制的切换逻辑需要充分测试 |

### 平台指标对比

### 注意事项

- **渐进式替换，不是大爆炸迁移。** 不要一次性替换所有旧工具。先让 Sentinel 和新工具的并行运行 2 周，确认数据一致性后再停用旧工具。
- **数据库的备份策略是最重要的。** 安全数据是合规审计的核心证据。PostgreSQL 必须每日全量备份 + WAL 归档。
- **NATS JetStream 的持久化配置。** `RetentionPolicy=Limits, MaxAge=7d, Storage=File`——确保事件不丢失，也不无限堆积。
- **平台的 API 限流。** Sentinel API 可能被滥用（如恶意大批量扫描）。必须配置 per-tenant 的限流策略。

### 常见踩坑经验

**踩坑案例 1：统一 Finding 模型的数据膨胀**
- **现象**：运行 3 个月后，findings 表达到 500 万行，查询变慢。
- **根因**：每次扫描都创建新 Finding 记录（即使状态未变）。
- **解法**：实现 Upsert 而非 Insert——同一个 {artifact + finding_key} 只保留最新一条，历史变更存储到 changes 日志表。

**踩坑案例 2：事件总线延迟导致 Dashboard 数据不一致**
- **现象**：Dashboard 显示「扫描完成 5 分钟后，Finding 仍为 0」。
- **根因**：NATS 消费者处理速度跟不上生产速度（500 QPS），事件积压。
- **解法**：增加 NATS consumer 的并发处理数（MaxAckPending=1000）；监控 consumer lag 指标。

**踩坑案例 3：自举扫描的循环依赖**
- **现象**：Sentinel 自己触发了一个扫描，扫描结果又触发了一个 Sentinel 自身的告警——循环。
- **根因**：Sentinel 的 image.pushed 事件触发了自己的扫描 Worker，扫描 Worker 的 finding.created 事件又触发了自己的 alert 处理器。
- **解法**：在事件中标记 source=sentinel-self，所有处理器检查 source 字段，忽略来自自身的递归事件。

### 思考题

1. Sentinel 平台服务了 500 个微服务，日均扫描 1 万+。如果 3 年后微服务增长到 5000 个，日均扫描增长到 10 万+，当前的六层架构中哪个层最可能成为瓶颈？如何提前设计水平扩展方案？
2. Sentinel 平台的 Web Dashboard 需要展示全公司的安全数据。如果 CTO 和 Team Lead 看到的数据范围应该不同（CTO 全局，Team Lead 只看自己团队），如何设计一个高效的多租户数据隔离方案，不通过 SQL WHERE tenant_id 来过滤（避免查询性能下降）？
