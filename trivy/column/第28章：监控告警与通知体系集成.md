# 第28章：监控告警与通知体系集成

> 版本：Trivy v0.50+
> 面向人群：SRE、DevOps、安全工程师

---

## 1. 项目背景

### 业务场景

云帆科技的扫描体系已经相当完善：CI/CD 门禁、Harbor 镜像扫描、Trivy Operator 持续监控、供应链签名验证——一切自动化运行。安全负责人老李甚至觉得「终于可以睡个安稳觉了」。但这个感觉只持续了 72 小时。

周三凌晨 3 点，一名运维为调试问题，临时修改了某个生产 Deployment 的安全上下文，去掉了 `readOnlyRootFilesystem: true`——这个变更让 Trivy Operator 触发了 HIGH 级别的配置错误检查（AVD-KSV-0017）。但直到周五上午的周例会上，安全团队才在 Grafana Dashboard 上偶然发现这个配置漂移。

「3 天，整整 3 天！如果这段时间有攻击者利用了这个缺口，我们都不知道。」老李铁青着脸说。

更深层的问题是：Trivy Operator 虽然发现了问题并写入了 VulnerabilityReport / ConfigAuditReport CRD，但这些 CRD 静静地躺在 K8s 的 etcd 里，没有人去「翻阅」。团队不可能每天手动检查成千上万个 CRD——需要一套自动化的「侦察兵」，在发现问题时主动通知。

「我们的扫描能力已经很强了，但我们没有『眼睛』。扫描好比守夜人拿着手电筒巡逻，但如果没有对讲机，发现了问题也没法告诉指挥中心。」CTO 说得一针见血。

### 痛点放大

**第一，报告被生产却没有被消费。** Trivy 的扫描结果以文件或 CRD 形式存在，但如果没有人主动去看，它们等于不存在。安全团队需要「拉」信息，而不是「推」到眼前。

**第二，告警延迟可能导致安全窗口被利用。** 从漏洞/配置错误出现到被发现之间的时间窗口（Time-to-Detect），是衡量安全运营效率的核心指标。人工巡检的 TTD 以天为单位，自动化告警可以压缩到秒级。

**第三，告警泛滥。** 如果每次出现一个 MEDIUM 级别的漏洞就全频道广播，不出三天团队就会「告警疲劳」——耳朵选择性关闭，真正重要的告警也被淹没。

**第四，多通知渠道的对接成本。** 钉钉、企业微信、Slack、PagerDuty——每个团队有自己惯用的沟通工具，安全告警需要覆盖所有这些渠道才能确保通知到达。

**本章的核心目标是：基于 Prometheus Metrics + Alertmanager + Webhook 构建分层告警体系，实现「发现问题 → 秒级告警 → 团队响应」的自动化闭环。同时设计告警降噪策略，避免「狼来了」的疲劳效应。**

## 2. 项目设计

**场景**：云帆科技的监控告警体系设计会，小胖（SRE）、小白（安全工程师）、大师正在设计告警逻辑。

---

**小胖**：「告警还不简单？写个 CronJob，每 5 分钟扫一次扫描结果，发现有 Critical 就发消息。半小时搞定。」

**小白**：「『有 Critical 就发消息』——你知道昨晚凌晨 2 点到早上 6 点之间，K8s 集群里发生了多少次 Pod 调度导致的「瞬时」Critical 告警吗？某个支付服务扩容了一个 Pod，这个 Pod 基于旧版本镜像，带了 3 个已知的 Critical 漏洞，被 Trivy Operator 检测到。然后 Pod 在 3 分钟后被自动回收了——因为新版本镜像也被调过来了。如果你的告警系统每 5 分钟轮询一次，你会在这 3 分钟的窗口内收到一条告警，然后 5 分钟后发现告警消失了，以为误报。凌晨 3 点被叫起来处理一个『已经不存在的问题』——你说同事会不会想打你？」

**小胖**：「那…那设置『持续时间』过滤？只有存在超过 30 分钟才告警？」

**大师**：「技术映射：告警设计就像医院的『分诊台』。不是所有咳嗽都直接送 ICU——分诊台会根据体温、血压、血氧等指标判断优先级。安全告警也需要分流：

| 告警级别 | 严重程度 | 响应时间 | 通知方式 | 举例 |
|----------|----------|----------|----------|------|
| P0 紧急 | CRITICAL 漏洞 + CISA KEV | 15分钟内 | 电话 + 短信 + 钉钉 | Log4Shell (CVE-2021-44228) |
| P1 高 | CRITICAL 漏洞（新发现） | 1小时内 | 钉钉 + Slack | 新公开的 0day |
| P2 中 | HIGH 漏洞或配置错误 | 4小时内 | Slack + 邮件 | 安全组对外开放 22 端口 |
| P3 低 | MEDIUM 漏洞 | 24小时内 | 邮件/日报 | 过期的 TLS 证书 |

关键是 P0/P1 才会触发即时通知（特别是夜间），P2/P3 只在工作时间内聚合推送。」

**小胖**：「哪些指标需要监控？」

**小白**：「主要三套 Metrics：

**第一套：漏洞指标（来自 Trivy Operator）**

```
trivy_vulnerability_id{severity, namespace, image}  # 各 namespace 的漏洞数量
trivy_image_vulnerabilities{severity, image_registry, image_repository, image_tag}
```

Prometheus 每隔 30 秒从 Operator 的 `/metrics` 端点抓取这些指标。你可以配置告警规则：

- `trivy_vulnerability_id{severity="CRITICAL"} > 0` 持续 5 分钟 → P1 告警
- `trivy_vulnerability_id{severity="HIGH"} > 10` 持续 30 分钟 → P2 告警
- `increase(trivy_vulnerability_id{severity="CRITICAL"}[1h]) > 0` → 新增 CRITICAL → P0 告警

**第二套：配置错误指标（来自 Trivy Operator）**

```
trivy_configaudit{severity, namespace, resource_kind, resource_name}
```

告警规则：

- `trivy_configaudit{severity="CRITICAL"} > 0` → 高危配置（如特权容器）→ P1 告警
- `trivy_configaudit{severity="HIGH"} > 5` → 多资源高危配置 → P2 告警

**第三套：Secret 泄露指标**

```
trivy_exposedsecret{severity, namespace, resource_name}
```

告警规则：

- `trivy_exposedsecret` 任何值 > 0 → P0 告警（Secret 泄露必须零容忍）」

**大师**：「除了这三个核心指标，你还需要基础设施层面的健康指标：

- `up{job="trivy-operator"}`：Trivy Operator Pod 是否存活？
- `trivy_db_last_update_timestamp`：trivy-db 最后一次更新时间是否在 24 小时内？
- `trivy_scan_duration_seconds`：扫描耗时 P99 是否超过阈值？

如果 Trivy Operator 挂了，或者 DB 过期了，你的『没有告警』可能意味着『什么都没扫』——这是最危险的状态。」

**小胖**：「那把这些指标接到 Prometheus 之后呢？Alertmanager 怎么配置？」

**大师**：「Alertmanager 的核心是『路由 + 分组 + 抑制』：

1. **路由（Route）**：根据告警的 labels 决定发给谁。比如 `severity="CRITICAL"` 发给 PagerDuty，`severity="HIGH"` 发给 Slack，# 不同 team 负责不同的 namespace。
2. **分组（Group）**：把同类告警合并。比如同一个镜像的 5 个 HIGH 漏洞，合并成一条消息而不是 5 条。
3. **抑制（Inhibit）**：如果某个 Pod 是 Deployment Rolling Update 期间的旧 Pod（即将被新 Pod 替换），抑制它的告警。
4. **静默（Silence）**：已知的计划维护窗口内（如 K8s 升级），静默指定规则的告警。

四种机制环环相扣，最终让运维收到的告警『每一条都能行动』，而不是『每一条都要排查』。」

## 3. 项目实战

### 环境准备

- **K8s 集群**：已部署 Trivy Operator
- **Prometheus**：v2.45+（自带 Alertmanager）
- **kube-prometheus-stack**：Helm Chart 一键部署 Prometheus + Grafana + Alertmanager
- **Slack / 钉钉 Webhook URL**：用于告警推送到 IM

### 步骤一：暴露 Trivy Operator 的 Prometheus Metrics

**目标**：配置 ServiceMonitor 让 Prometheus 抓取 Trivy Operator 的指标。

首先确认 Trivy Operator 的安装参数中已启用 Metrics：

```bash
# 安装 Trivy Operator 时启用 Metrics
helm install trivy-operator aqua/trivy-operator \
  --namespace trivy-system \
  --create-namespace \
  --set="trivyOperator.metrics.enabled=true" \
  --set="trivyOperator.metrics.serviceMonitor.enabled=true" \
  --set="trivyOperator.metrics.serviceMonitor.labels.release=prometheus"
```

> **坑点**：`serviceMonitor.labels.release=prometheus` 必须与 `kube-prometheus-stack` 安装时设置的 `prometheus.prometheusSpec.serviceMonitorSelector.matchLabels` 匹配。否则 Prometheus 不会自动发现 Trivy Operator 的 ServiceMonitor。

**手动验证 Metrics 端点**：

```bash
# 端口转发访问 Trivy Operator 的 /metrics
kubectl port-forward -n trivy-system svc/trivy-operator 8080:80

# 查询指标
curl http://localhost:8080/metrics | grep trivy_
```

**典型输出**：

```
# HELP trivy_vulnerability_id Number of exposed vulnerabilities per severity
# TYPE trivy_vulnerability_id gauge
trivy_vulnerability_id{image_registry="harbor.internal",image_repository="cloud-sail/payment",image_tag="v2.3.1",severity="CRITICAL"} 3
trivy_vulnerability_id{image_registry="harbor.internal",image_repository="cloud-sail/payment",image_tag="v2.3.1",severity="HIGH"} 12
trivy_vulnerability_id{image_registry="harbor.internal",image_repository="cloud-sail/payment",image_tag="v2.3.1",severity="MEDIUM"} 45

# HELP trivy_exposedsecret Number of exposed secrets
```

### 步骤二：配置 PrometheusRule（告警规则）

**目标**：定义分层告警规则，从 P0 到 P3 按严重性和紧迫性分级。

创建 `prometheus-rules.yaml`：

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: trivy-alert-rules
  namespace: trivy-system
  labels:
    release: prometheus
spec:
  groups:
  # ============ P0: 紧急告警 ============
  - name: trivy.p0.critical
    interval: 30s
    rules:
    - alert: TrivyNewCriticalVulnerability
      expr: |
        (trivy_vulnerability_id{severity="CRITICAL"} 
         - trivy_vulnerability_id{severity="CRITICAL"} offset 1h) > 0
      for: 5m
      labels:
        severity: critical
        priority: P0
        team: security
      annotations:
        summary: "New CRITICAL vulnerability detected in production"
        description: |
          Image {{ $labels.image_repository }}:{{ $labels.image_tag }} 
          has NEW CRITICAL vulnerabilities in the last 1 hour.
          Current count: {{ $value }}
          Run: trivy image {{ $labels.image_registry }}/{{ $labels.image_repository }}:{{ $labels.image_tag }}

    - alert: TrivySecretExposed
      expr: trivy_exposedsecret > 0
      for: 1m
      labels:
        severity: critical
        priority: P0
        team: security
      annotations:
        summary: "SECRET EXPOSED in {{ $labels.namespace }}/{{ $labels.resource_name }}"
        description: |
          A secret has been detected in Kubernetes resource:
          Namespace: {{ $labels.namespace }}
          Resource: {{ $labels.resource_name }}
          Severity: {{ $labels.severity }}
          ACTION REQUIRED: Rotate the exposed credential immediately.

  # ============ P1: 高优先级告警 ============
  - name: trivy.p1.high
    interval: 1m
    rules:
    - alert: TrivyCriticalVulnerabilityExists
      expr: trivy_vulnerability_id{severity="CRITICAL"} > 0
      for: 15m
      labels:
        severity: warning
        priority: P1
        team: security
      annotations:
        summary: "CRITICAL vulnerabilities exist in production images"
        description: |
          {{ $labels.image_repository }}:{{ $labels.image_tag }} 
          has {{ $value }} CRITICAL vulnerabilities for > 15 minutes.

    - alert: TrivyCriticalMisconfiguration
      expr: trivy_configaudit{severity="CRITICAL"} > 0
      for: 10m
      labels:
        severity: warning
        priority: P1
        team: platform
      annotations:
        summary: "CRITICAL misconfiguration in {{ $labels.namespace }}/{{ $labels.resource_name }}"
        description: |
          Resource {{ $labels.resource_kind }}/{{ $labels.resource_name }}
          in namespace {{ $labels.namespace }} has a CRITICAL misconfiguration.

  # ============ P2: 中优先级告警 ============
  - name: trivy.p2.medium
    interval: 2m
    rules:
    - alert: TrivyHighVulnerabilityThreshold
      expr: trivy_vulnerability_id{severity="HIGH"} > 10
      for: 30m
      labels:
        severity: info
        priority: P2
        team: security
      annotations:
        summary: "High number of HIGH vulnerabilities ({{ $value }} > 10)"

    - alert: TrivyHighMisconfigurationThreshold
      expr: trivy_configaudit{severity="HIGH"} > 5
      for: 30m
      labels:
        severity: info
        priority: P2
        team: platform
      annotations:
        summary: "Multiple HIGH misconfigurations in {{ $labels.resource_name }}"

  # ============ 基础设施健康 ============
  - name: trivy.infra
    interval: 1m
    rules:
    - alert: TrivyOperatorDown
      expr: up{job="trivy-operator"} == 0
      for: 5m
      labels:
        severity: critical
        priority: P0
        team: platform
      annotations:
        summary: "Trivy Operator is DOWN"
        description: "The Trivy Operator Pod is not responding. No vulnerability scans are being performed."

    - alert: TrivyDBOutdated
      expr: (time() - trivy_db_last_update_timestamp) > 86400
      for: 30m
      labels:
        severity: warning
        priority: P1
        team: security
      annotations:
        summary: "Trivy vulnerability database is over 24 hours old"
        description: "The vulnerability database has not been updated in over 24 hours. Scans may miss newly disclosed vulnerabilities."
```

### 步骤三：配置 Alertmanager（路由 + 通知渠道）

**目标**：根据告警级别自动选择通知渠道——P0 走 PagerDuty/电话，P1/P2 走 Slack/钉钉。

创建 `alertmanager-config.yaml`：

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: alertmanager-main
  namespace: monitoring
stringData:
  alertmanager.yaml: |
    global:
      resolve_timeout: 5m
      slack_api_url: 'https://hooks.slack.example/services/<workspace>/<channel>/<token>'
      pagerduty_url: 'https://events.pagerduty.com/v2/enqueue'

    # 路由树：根据 labels 决定告警去向
    route:
      receiver: 'null'                          # 默认接收者（兜底）
      group_by: ['alertname', 'severity']       # 分组键
      group_wait: 30s                           # 首次等待（收集同类告警）
      group_interval: 5m                        # 分组内的发送间隔
      repeat_interval: 4h                       # 重复发送间隔
      routes:
        # P0 紧急 → PagerDuty + Slack
      - match:
          priority: P0
        receiver: 'pagerduty-critical'
        continue: true                          # 继续向下匹配（同时发 Slack）

        # P0 / P1 → Slack #security-alerts 频道
      - match_re:
          priority: 'P0|P1'
        receiver: 'slack-security-alerts'

        # P2 → Slack #platform-ops 频道
      - match:
          priority: P2
        receiver: 'slack-platform-ops'

        # platform team 的告警分流
      - match:
          team: platform
        receiver: 'slack-platform-ops'

    # 抑制规则：如果 Trivy Operator 挂了，抑制所有扫描结果告警
    inhibit_rules:
    - source_match:
        alertname: TrivyOperatorDown
      target_match_re:
        alertname: 'Trivy.*'
      equal: []

    # ===== 接收者配置 =====
    receivers:
    - name: 'null'    # 兜底，不发送

    - name: 'pagerduty-critical'
      pagerduty_configs:
      - routing_key: '<PAGERDUTY_INTEGRATION_KEY>'
        severity: critical
        description: '{{ range .Alerts }}{{ .Annotations.summary }}\n{{ end }}'

    - name: 'slack-security-alerts'
      slack_configs:
      - channel: '#security-alerts'
        title: '{{ if eq .Status "firing" }}🔴{{ else }}🟢{{ end }} [{{ .CommonLabels.priority }}] {{ .CommonLabels.alertname }}'
        text: |
          *Summary*: {{ .CommonAnnotations.summary }}
          *Description*: 
          ```
          {{ .CommonAnnotations.description }}
          ```
          *Links*:
          - <https://grafana.internal/d/trivy-overview|Trivy Dashboard>
          - <https://playbook.internal/security/{{ .CommonLabels.alertname }}|Runbook>

    - name: 'slack-platform-ops'
      slack_configs:
      - channel: '#platform-ops'
        title: '[{{ .CommonLabels.priority }}] {{ .CommonLabels.alertname }}'
        text: '{{ .CommonAnnotations.description }}'

    # ===== 钉钉 Webhook 接收者 =====
    - name: 'dingtalk-webhook'
      webhook_configs:
      - url: 'https://oapi.dingtalk.com/robot/send?access_token=xxx'
        send_resolved: false
        http_config:
          follow_redirects: true
```

### 步骤四：自定义钉钉/企业微信 Webhook 通知

**目标**：将 Alertmanager 告警转换为钉钉 Markdown 消息格式。

如果 Alertmanager 原生不支持钉钉格式，可以部署一个 `prometheus-webhook-dingtalk` 转换器：

```bash
# 安装 dingtalk webhook 转换器（Helm）
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm install dingtalk prometheus-community/prometheus-webhook-dingtalk \
  --namespace monitoring \
  --set config.webhooks[0].name=trivy \
  --set config.webhooks[0].url="https://oapi.dingtalk.com/robot/send?access_token=xxx" \
  --set config.webhooks[0].secret="your-dingtalk-secret"
```

Alertmanager 配置中增加一个 receiver 指向这个 dingtalk 服务：

```yaml
    - name: 'dingtalk-trivy'
      webhook_configs:
      - url: 'http://dingtalk-prometheus-webhook-dingtalk.monitoring:8060/dingtalk/trivy/send'
        send_resolved: true
```

### 步骤六：编写自动补救脚本（Semi-Automated）

**目标**：针对常见配置错误，自动发送修复建议或执行预授权的修复。

```python
#!/usr/bin/env python3
"""
监听 Prometheus Alertmanager 的 Webhook，
对特定类型的告警自动生成修复建议
"""

from flask import Flask, request, jsonify
import json
import requests

app = Flask(__name__)

# 修复建议映射
REMEDIATION_MAP = {
    "TrivyCriticalMisconfiguration": {
        "AVD-KSV-0001": "Set `securityContext.runAsNonRoot: true` in your Deployment YAML.",
        "AVD-KSV-0002": "Set `securityContext.readOnlyRootFilesystem: true` in your Deployment YAML.",
        "AVD-KSV-0012": "Set `securityContext.capabilities.drop: [\"ALL\"]` in your Deployment YAML.",
        "AVD-KSV-0017": "Set `securityContext.privileged: false` in your Deployment YAML.",
    },
}

DINGTALK_WEBHOOK = "https://oapi.dingtalk.com/robot/send?access_token=xxx"


@app.route("/alert", methods=["POST"])
def handle_alert():
    data = request.json

    for alert in data.get("alerts", []):
        alertname = alert.get("labels", {}).get("alertname", "")
        annotations = alert.get("annotations", {})
        status = alert.get("status", "")  # firing / resolved

        if alertname in REMEDIATION_MAP:
            suggestion = REMEDIATION_MAP[alertname]
            send_dingtalk(alertname, annotations, suggestion, status)

    return jsonify({"status": "ok"})


def send_dingtalk(alertname, annotations, suggestion, status):
    """发送带修复建议的钉钉消息"""
    icon = "🔴" if status == "firing" else "🟢"

    text = f"""### {icon} {alertname}
**摘要**: {annotations.get('summary', 'N/A')}

**描述**: 
```
{annotations.get('description', 'N/A')}
```

**修复建议**: 

{suggestion.get('default', '请参考 Runbook 手动处理。')}

---
*状态: {status} | 时间: {annotations.get('startsAt', 'N/A')}*
"""

    payload = {"msgtype": "markdown", "markdown": {"title": alertname, "text": text}}
    requests.post(DINGTALK_WEBHOOK, json=payload, timeout=5)


if __name__ == "__main__":
    app.run(host="0.0.0.0", port=9093)
```

### 测试验证

1. 部署告警规则后，向测试 Deployment 添加不安全的配置（如 `privileged: true`），验证 Alertmanager 触发 `TrivyCriticalMisconfiguration` 告警。
2. 故意停止 Trivy Operator Pod，验证 `TrivyOperatorDown` 告警被触发，且抑制了其他所有 Trivy 告警。
3. 模拟夜间 2 点出现 CRITICAL 漏洞——向测试 Namespace 部署含漏洞镜像，验证 PagerDuty / Slack / 钉钉均收到通知。
4. 测试告警分组：同时向 3 个 Namespace 部署含漏洞镜像，验证 Alertmanager 将同类告警合并为一条消息。
5. 验证修复建议：触发 `AVD-KSV-0017`（特权容器）告警，确认钉钉消息包含修复命令。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| PrometheusRule | 告警规则即代码；版本可控 | 规则语法（PromQL）有一定学习曲线 |
| Alertmanager 路由 | 灵活的分组/抑制/静默策略 | 配置文件较复杂，调试困难 |
| 多通知渠道 | Slack + 钉钉 + PagerDuty 全覆盖 | 每个渠道需要独立的 Webhook 维护 |
| 分层告警 | P0-P3 分流，减少噪点 | 需要团队共识每个级别的响应 SLA |
| 自动修复建议 | 降低 MTTR（平均修复时间） | 需要持续更新修复建议库 |

### 适用场景

1. **7×24 安全运维**：必须有人值班的场景，告警体系是第二道防线。
2. **多团队/多业务线**：不同 Namespace 的告警分流到不同的 Slack 频道/团队。
3. **合规审计**：告警历史记录（Prometheus TSDB + Alertmanager 日志）可证明「安全事件被发现的时间」和「响应时间」。
4. **平台工程团队**：通过 Grafana Dashboard 直观监控整体安全态势。
5. **自动化修复探索**：告警触发自动 JIRA 工单或修复脚本——成为安全自动化的起点。

**不适用场景**：
1. 团队 < 5 人且无专人运维——告警疲劳的风险极高，不如从日报/周报开始。
2. 不使用 K8s 的纯 VM 环境——Prometheus + Alertmanager 仍然可用，但 Trivy Operator 的 Metrics 不适用。

### 注意事项

- **告警静默要有限期**。不要创建「永久静默」规则。静默窗口到期后自动恢复告警。
- **夜间告警策略要严格区分**。凌晨 2 点只有 P0（即将造成业务损失）的告警才允许唤醒人。P1/P2 缓存到早上 8 点统一发送。
- **告警信息的可操作性**。每条告警的 annotation 中必须包含：影响范围、修复步骤、相关 Dashboard 链接。
- **Prometheus 不要被用作事件存储**。告警历史应导入 ELK/Loki 或长期日志系统，Prometheus TSDB 的保留期通常较短（15-30天）。

### 常见踩坑经验

**踩坑案例 1：Alertmanager 静默配置了但告警仍然发送**
- **现象**：在 Alertmanager UI 中设置了静默规则，但 Slack 仍然收到告警。
- **根因**：静默规则的 matcher 与告警 labels 不完全匹配（如大小写、多余空格）。
- **解法**：在 Alertmanager UI 的 `/alerts` 页面检查告警的完整 labels，确保 matcher 精确匹配。

**踩坑案例 2：Trivy Operator 升级后指标名称变更**
- **现象**：Dashed 的 Grafana 面板突然全部显示 "No data"。
- **根因**：Trivy Operator 新版本将 `trivy_vulnerability_id` 重命名为 `trivy_image_vulnerabilities`。
- **解法**：在 Trivy Operator 的 CHANGELOG 中追踪 Metrics 名称变更；在 Grafana 的 Dashboard 中使用 `__name__` 正则匹配兼容新旧名称。

**踩坑案例 3：钉钉 Webhook 限流导致告警丢失**
- **现象**：大规模事件（如全集群的 Trivy Operator 重扫）触发数百条告警，钉钉返回 HTTP 429。
- **根因**：钉钉机器人每分钟最多接收 20 条消息。
- **解法**：利用 Alertmanager 的 `group_by` 和 `group_interval` 合并告警；在告警消息中只展示 Top N（最严重的 5 个）而非全部；考虑用自定义 HTTP Gate 做钉钉侧的限流队列。

### 思考题

1. Trivy Operator 的扫描间隔默认是 1 小时。如果在此期间有一个容器被部署、被攻击、被自动重启，且扫描仅在重启后发生，告警系统是否会遗漏这个短暂的漏洞窗口？如何设计一种机制来捕获「生命周期极短的容器」的安全风险？

2. 你的公司有 200 个微服务，每个微服务每天可能触发 0-5 条安全告警。在高峰期，Alertmanager 每分钟可能收到 100+ 条告警。请设计一个「告警优先级动态调整」方案：根据告警的持续时间、影响范围、历史频率，自动提升或降低告警级别。

> **答案提示**：第 40 章「从零构建企业级安全扫描平台」将讨论大型平台中的告警治理和智能路由方案。
