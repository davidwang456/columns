# 第2章：多环境安装部署与首个漏洞扫描

> 版本：Trivy v0.50+
> 面向人群：开发、运维、测试

---

## 1. 项目背景

### 业务场景

云帆科技在上次会议后决定全面采用 Trivy 作为安全扫描工具。CTO 给运维负责人老李下了死命令：「一周之内，让公司所有人——不管是用 Mac 的开发、用 Windows 的测试、还是 Linux 服务器上的 Jenkins——都能跑通 Trivy，扫出第一份漏洞报告。」

老李一开始觉得这事简单：「不就是个二进制吗，扔上去不就行了？」结果现实给他上了生动的一课。

开发小王的 MacBook M2 用 Homebrew 装完后，扫描公司镜像时死活报 `architecture mismatch`；测试组的小张在 Windows 上按照官方文档操作，路径里的反斜杠把 Trivy 的配置文件解析搞崩了；而 Jenkins 的 Slave 节点是 CentOS 7，yum 源里根本没有 Trivy 的包，老李只能从 GitHub Release 手动下载，结果 CI 流水线又因为权限问题无法写入缓存目录而失败。

更麻烦的是，团队第一次扫描报告出来后，开发们看着满屏的红色 Critical 标签慌了神，纷纷在群里问：「这报告里的 CVE-2021-44228 是啥？要不要连夜发版修？」而测试同学则困惑：「为什么同一个镜像，小王扫出来 12 个 High，我扫出来 15 个？」

### 痛点放大

**第一，安装碎片化。** 不同操作系统、不同架构（x86 / ARM）、不同部署模式（裸机 / Docker / K8s）的安装方式各不相同。没有一份覆盖全场景的部署指南，每个环境都要踩一遍坑。

**第二，权限与缓存的隐形陷阱。** Trivy 默认把漏洞数据库缓存到用户主目录的 `~/.cache/trivy/` 下。Jenkins 以 `jenkins` 用户运行时，主目录可能在 `/var/lib/jenkins/`，也可能在容器里的 `/root/`，权限配不对就下载失败。

**第三，报告解读不统一。** 团队不清楚 `Severity` 的定义、不知道 `--severity` 过滤怎么用、不理解 `exit-code` 对 CI 门禁的意义。结果扫描成了「形式主义」——大家看一眼密密麻麻的表格，叹了口气，继续合并代码。

**本章的核心目标就是：让 Trivy 在任何环境都能一键跑起来，让第一份报告能被正确解读。**

## 2. 项目设计

**场景**：云帆科技的运维室，老李正带着小王和小张排查安装问题。白板上贴着三个环境的报错截图。

---

**小胖**：（端着奶茶走进来）「老李，我按你说的 brew install 了，怎么扫咱们公司的 `arm64` 镜像时报架构不匹配啊？」

**小白**：「你的 Mac 是 M2 芯片吧？Trivy 本身没问题，但你要扫的镜像如果是 `amd64` 构建的，Docker Desktop 默认会走 Rosetta 转译。Trivy 解析 layer 时如果直接读宿主机的二进制，就可能出现架构混淆。你应该加上 `--platform linux/amd64` 或者在本地构建一个 `arm64` 版本的测试镜像。」

**小胖**：「等等，什么叫 amd64、arm64？这不就是电脑型号吗，跟扫描有啥关系？」

**大师**：「小胖，你可以把 CPU 架构想象成『语言』。同样一句话，说中文的人和说法语的人互相听不懂。`amd64` 是 Intel/AMD 的『语言』，`arm64` 是苹果 M 系列芯片的『语言』。Trivy 扫描镜像时，需要读取镜像里的二进制文件（比如 `dpkg` 数据库），如果镜像的『语言』和运行环境的『语言』不一致，解析就可能出错。」

**小胖**：「哦……那 Windows 呢？小张说她那边路径炸了。」

**小白**：「Windows 的坑我踩过。Trivy 的配置文件路径、缓存路径在 Windows 上涉及反斜杠和空格的转义。比如 `C:\Program Files\Trivy\` 这种路径，如果不加引号或者不用 WSL2，命令行解析就会截断。最省事的办法是在 Windows 上直接用 WSL2 安装 Linux 版 Trivy，或者全程用 Docker 运行，把路径挂载到 Linux 容器里处理。」

**大师**：「技术映射：WSL2 和 Docker 本质上是给 Windows 包了一层 Linux 翻译官，让 Trivy 以为自己运行在 Linux 上，从而避开 Windows 路径和权限的历史包袱。」

**小胖**：「那 Jenkins 呢？老李说流水线挂了。」

**老李**：（叹气）「Jenkins Slave 以非 root 用户跑 Trivy，默认缓存目录写不进去，下载漏洞数据库时报 `permission denied`。我想改到 `/tmp/` 下，又发现每次容器重启 `/tmp/` 被清空，下次扫描还得重新下载几 GB 的数据库。」

**大师**：「这是个经典的缓存持久化问题。Trivy 的缓存分两层：一层是漏洞数据库（`trivy.db`、`java.db`），一层是扫描缓存（镜像 layer 的 BLOB）。对于 CI 环境，我建议用三个策略：

1. **固定缓存目录**：通过 `--cache-dir /mnt/trivy-cache` 把缓存挂到持久化存储（如 NFS、EBS、HostPath）。
2. **容器卷挂载**：在 Jenkins Pipeline 里用 `docker run -v /host/cache:/root/.cache/trivy` 把宿主机的缓存目录挂进容器。
3. **预置数据库**：在 Jenkins Agent 的镜像里预装好最新版 `trivy.db`，每天通过 CronJob 更新，避免每次构建都下载。」

**小白**：「还有一个问题——为什么我和小王扫同一个镜像，漏洞数量不一样？」

**大师**：「原因可能有三个：数据库版本不同、缓存状态不同、命令参数不同。Trivy 的数据库每天更新两次，如果小王是早上扫的、你是下午扫的，下午数据库里可能新入库了几个 CVE。另外，如果你们一个用了 `--severity HIGH,CRITICAL` 过滤，一个没用，结果当然不同。」

**小胖**：「那怎么保证团队输出一致？」

**大师**：「统一配置。把 `severity`、`exit-code`、`format`、`cache-dir` 这些参数写进 `trivy.yaml` 配置文件，提交到 Git 仓库，所有人用同一套配置扫描。再加上 `trivy version` 的输出检查，确保数据库更新时间一致。」

**小胖**：「原来一条 `trivy image` 后面藏了这么多门道……」

**大师**：「安装只是开始，统一才是目的。今天我们把三类环境的安装 SOP 定下来，明天所有人就能输出可对比的扫描报告了。」

## 3. 项目实战

### 环境准备

- **目标环境**：开发机（macOS / Windows WSL2）、CI 服务器（Linux）、本地实验（Docker）
- **Trivy 版本**：v0.50+
- **Docker**：已安装（用于容器化运行和测试镜像拉取）
- **网络**：可访问 GitHub Releases 或已配置镜像加速

### 步骤一：macOS 开发机安装与配置

**目标**：在 Mac（Intel 或 Apple Silicon）上安装 Trivy，完成首次扫描。

```bash
# 使用 Homebrew 安装
brew install trivy

# 验证版本和数据库状态
trivy version

# 扫描一个测试镜像
trivy image alpine:3.14
```

**Apple Silicon 注意事项**：

如果你需要扫描 `linux/amd64` 架构的镜像，确保 Docker Desktop 已开启 Rosetta 支持，或在扫描前显式指定平台：

```bash
# 拉取 amd64 版本的镜像
docker pull --platform linux/amd64 alpine:3.14

# 扫描时 Trivy 会自动识别镜像内的 OS 包
trivy image alpine:3.14
```

### 步骤二：Windows WSL2 安装（推荐方案）

**目标**：避开 Windows 原生路径和权限问题，在 WSL2 Ubuntu 中运行 Trivy。

```bash
# 在 WSL2 中执行
sudo apt-get update
sudo apt-get install -y wget apt-transport-https gnupg lsb-release

# 导入 Aqua 公钥
wget -qO - https://aquasecurity.github.io/trivy-repo/deb/public.key | sudo gpg --dearmor -o /usr/share/keyrings/trivy.gpg

# 添加 apt 源
echo "deb [signed-by=/usr/share/keyrings/trivy.gpg] https://aquasecurity.github.io/trivy-repo/deb $(lsb_release -sc) main" | sudo tee /etc/apt/sources.list.d/trivy.list

# 安装
sudo apt-get update
sudo apt-get install -y trivy

# 验证
trivy version
```

> **可能遇到的坑**：如果 WSL2 中 Docker 不可用，需要确保 Docker Desktop 的「WSL2 后端」已开启，并在 WSL2 中执行 `docker ps` 测试连通性。

### 步骤三：Linux 服务器与 CI 环境安装

**目标**：在 CentOS/Ubuntu 服务器上安装 Trivy，并配置持久化缓存。

```bash
# 通用方案：直接下载二进制（适合无公网 apt/yum 源的内网环境）
TRIVY_VERSION=0.50.0
wget https://github.com/aquasecurity/trivy/releases/download/v${TRIVY_VERSION}/trivy_${TRIVY_VERSION}_Linux-64bit.tar.gz
tar zxvf trivy_${TRIVY_VERSION}_Linux-64bit.tar.gz
sudo mv trivy /usr/local/bin/

# 验证
trivy version
```

**创建持久化缓存目录**（CI 环境关键步骤）：

```bash
# 创建全局缓存目录
sudo mkdir -p /var/cache/trivy
sudo chmod 777 /var/cache/trivy

# 设置环境变量，让所有用户使用同一缓存
export TRIVY_CACHE_DIR=/var/cache/trivy

# 写入 /etc/profile 持久化
echo 'export TRIVY_CACHE_DIR=/var/cache/trivy' | sudo tee /etc/profile.d/trivy.sh
```

### 步骤四：Docker 方式运行 Trivy

**目标**：在完全不安装 Trivy 二进制的情况下，用 Docker 容器完成扫描。

```bash
# 扫描远程镜像（Trivy 容器内运行，需挂载 docker.sock）
docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
  -v $HOME/.cache/trivy:/root/.cache/trivy \
  aquasec/trivy:0.50.0 image python:3.4-alpine

# 扫描本地文件系统（挂载目标目录）
docker run --rm -v /path/to/project:/project \
  -v $HOME/.cache/trivy:/root/.cache/trivy \
  aquasec/trivy:0.50.0 fs /project
```

**参数解读**：
- `-v /var/run/docker.sock:/var/run/docker.sock`：让容器内的 Trivy 能调用宿主机的 Docker Daemon 拉取镜像。
- `-v $HOME/.cache/trivy:/root/.cache/trivy`：将宿主机的缓存目录挂载到容器内，避免每次运行都重新下载数据库。

> **可能遇到的坑**：如果宿主机是 rootless Docker，`docker.sock` 路径可能在 `$XDG_RUNTIME_DIR/docker.sock`，需调整挂载路径。

### 步骤五：统一配置与团队规范

**目标**：创建 `trivy.yaml` 配置文件，统一团队的扫描行为。

创建文件 `trivy.yaml`：

```yaml
# 全局配置
severity:
  - HIGH
  - CRITICAL

# 报告格式
format: table

# 扫描完成后退出码策略
exit-code: 1

# 缓存目录（CI 环境中需确保此目录持久化）
cache-dir: /var/cache/trivy

# 忽略未修复的漏洞（减少噪音，根据团队策略决定是否开启）
# ignore-unfixed: true

# 扫描器选择（默认全部，可精简以加速）
scanners:
  - vuln
  - secret
```

**使用配置文件扫描**：

```bash
# Trivy 会自动查找当前目录的 trivy.yaml，或通过 --config 指定
trivy image --config trivy.yaml python:3.4-alpine

# 或直接放在默认路径
mkdir -p ~/.config/trivy
cp trivy.yaml ~/.config/trivy/
trivy image python:3.4-alpine
```

### 步骤六：解读首份报告并建立基线

**目标**：理解报告字段，设置Severity过滤，建立团队的漏洞处理基线。

```bash
# 仅查看 HIGH 和 CRITICAL，且已修复的漏洞
trivy image --severity HIGH,CRITICAL --ignore-unfixed python:3.4-alpine
```

**输出示例**：
```
python:3.4-alpine (alpine 3.9.2)
================================
Total: 8 (HIGH: 5, CRITICAL: 3)

┌─────────────┬────────────────┬──────────┬────────┬───────────────────┐
│   Library   │ Vulnerability  │ Severity │ Status │   Fixed Version   │
├─────────────┼────────────────┼──────────┼────────┼───────────────────┤
│ openssl     │ CVE-2022-0778  │ HIGH     │ fixed  │ 1.1.1n-r0         │
│ sqlite      │ CVE-2019-8457  │ CRITICAL │ fixed  │ 3.28.0-r0         │
└─────────────┴────────────────┴──────────┴────────┴───────────────────┘
```

**基线决策表**：

| Severity | 处理时限 | 责任人 | 行动 |
|----------|----------|--------|------|
| CRITICAL | 24 小时内 | 开发负责人 | 必须修复，阻断发布 |
| HIGH | 7 天内 | 模块 owner | 评估修复，可接受则记录 |
| MEDIUM | 30 天内 | 团队排期 | 常规迭代修复 |
| LOW | 下次发版 | 团队排期 | 视资源情况处理 |

将这张基线表写入团队 Wiki，并在 `trivy.yaml` 中配置对应的 `severity` 和 `exit-code`。

### 测试验证

1. **安装验证**：在三类环境（Mac / WSL2 / Linux）分别执行 `trivy version`，确认版本一致。
2. **扫描验证**：用同一镜像（如 `python:3.4-alpine`）在三类环境扫描，对比 `Total` 数量是否一致（允许因数据库版本差异有少量不同）。
3. **缓存验证**：在 Linux 环境执行两次相同扫描，第二次应在 10 秒内完成（证明缓存生效）。
4. **配置验证**：使用 `trivy.yaml` 扫描，确认只输出 HIGH 和 CRITICAL 漏洞。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 安装方式 | 支持 brew/apt/yum/Docker/二进制，覆盖全平台 | Windows 原生支持较弱，推荐 WSL2 |
| 配置管理 | 支持 YAML 配置文件，便于团队统一 | 配置优先级规则（Flag > Env > YAML）较复杂 |
| 缓存机制 | 分层缓存（DB + BLOB），二次扫描极快 | 默认缓存目录因用户而异，CI 环境需显式指定 |
| 报告可读性 | 表格输出直观，JSON 便于集成 | 默认输出信息量过大，初学者容易迷失 |
| CI 集成 | exit-code 机制天然支持门禁 | 未提供内置的增量扫描能力 |

### 适用场景

1. **开发机快速自检**：开发者在提交代码前，用 `trivy fs` 扫描项目依赖。
2. **CI/CD 安全门禁**：在构建阶段阻断包含高危漏洞的镜像进入仓库。
3. **多架构团队统一**：通过 Docker 方式运行，抹平 macOS/Windows/Linux 的差异。
4. **离线/内网环境**：预置数据库 + 持久化缓存，实现无外网扫描。
5. **临时审计**：安全团队对特定镜像或目录进行一次性深度扫描。

**不适用场景**：
1. 需要 GUI 图形界面的团队——Trivy 是纯 CLI 工具，没有官方桌面端。
2. 零命令行基础的纯业务人员——需要一定的终端操作能力。

### 注意事项

- **缓存目录的磁盘占用**：`trivy.db` 约 300MB，`java.db` 约 500MB，加上镜像 layer 缓存，建议为 CI 节点预留至少 5GB 磁盘空间。
- **权限最小化**：CI 环境中的 Trivy 不需要 root 权限运行，但需要对 `--cache-dir` 有读写权限。
- **数据库时间戳检查**：在 CI 日志中加入 `trivy version` 输出，便于排查因数据库版本差异导致的扫描结果不一致。

### 常见踩坑经验

**踩坑案例 1：Jenkins 容器内扫描报 `permission denied`**
- **现象**：Jenkins Pipeline 中执行 Trivy 时，数据库下载失败。
- **根因**：Jenkins 以非 root 用户运行，默认 `~/.cache/trivy` 目录不存在或无写权限。
- **解法**：在 Pipeline 中设置 `TRIVY_CACHE_DIR=/tmp/trivy-cache`，并确保目录已 `chmod 777`；或挂载持久化卷到 `/root/.cache/trivy`。

**踩坑案例 2：Mac M2 扫描 amd64 镜像时报错**
- **现象**：`unsupported image architecture` 或解析异常。
- **根因**：Trivy 尝试读取镜像中的二进制文件，跨架构时某些特征识别可能出错。
- **解法**：优先扫描与宿主机同架构的镜像；若必须扫描异架构镜像，确保 Docker Desktop 的「Use Rosetta for x86/amd64 emulation」已开启。

**踩坑案例 3：Docker 方式运行 Trivy 时数据库反复下载**
- **现象**：每次 `docker run` 都要重新下载 trivy-db。
- **根因**：未挂载缓存卷，容器销毁后缓存丢失。
- **解法**：始终加上 `-v $HOME/.cache/trivy:/root/.cache/trivy` 参数，或构建一个预置数据库的自定义 Trivy 镜像。

### 思考题

1. 为什么 Trivy 在 CI 环境中建议使用 `exit-code: 1` 策略？如果设置 `exit-code: 0`，对流水线的门禁机制会有什么影响？
2. 假设你的团队有 50 个开发者，每人每天构建 5 次镜像。请设计一套缓存策略，既能保证数据库新鲜度，又能将外部下载流量降低 90% 以上。
