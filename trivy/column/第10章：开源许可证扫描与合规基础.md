# 第10章：开源许可证扫描与合规基础

> 版本：Trivy v0.50+
> 面向人群：法务、技术负责人、架构师、开源治理专员

---

## 1. 项目背景

### 业务场景

云帆科技经过近两年的快速发展，产品已经从单一 SaaS 工具扩展成了包含 12 个微服务、3 个移动端 App、2 个桌面客户端的平台型产品。CTO 在一次全员大会上宣布：「我们准备启动 B 轮融资，审计机构需要对我们的技术栈做一次全面的尽职调查。」

三天后，审计机构的法务顾问发来了一份长长的问卷，其中有一条让技术团队措手不及：「请提供贵司所有产品中使用到的第三方开源组件的完整清单，以及每个组件对应的开源许可证类型。请特别说明是否存在 GPL 系列许可证的依赖。」

小胖（后端负责人）第一次意识到问题的严重性。他翻看了主服务的 `package.json`，里面列出了 40 多个直接依赖；但当他用 `npm ls` 展开后，发现间接依赖超过 1200 个。前端的情况更夸张——一个 React 项目的 `node_modules` 里有 2800 多个包。

法务部的张律师随后补了一刀：「我查了一下，你们后端服务里有一个依赖链引用了 `GPL-3.0` 的库。如果你们的代码和这个库有链接关系，按照 GPL 的传染性条款，你们整个后端服务的源代码都可能需要以 GPL 协议开源。」

CTO 当场脸色铁青。公司最核心的业务代码，怎么可能因为引入了一个日志工具库就「被迫开源」？

与此同时，另一个问题也浮出水面。公司的 iOS App 使用了一个标注为 `CC-BY-NC-4.0`（知识共享-非商业性使用）的图标库，而公司的产品恰恰是商业收费的。虽然技术团队认为「只是几个图标而已」，但法务顾问指出这构成了明确的许可证违约。

### 痛点放大

**第一，依赖网络的许可证盲区。** 现代软件的依赖关系呈指数级增长。一个直接依赖可能引入 10 个间接依赖，每个间接依赖又有自己的许可证。团队对直接依赖的许可证也许有感知，但对间接依赖几乎完全失控。

**第二，GPL 传染性的法律风险。** GPL（尤其是 GPL-2.0/3.0 和 AGPL）的传染性条款要求：任何与 GPL 代码「链接」或「组合」的软件，都必须以 GPL 协议开源。对于商业闭源产品，这几乎是不可接受的。而LGPL虽然传染性较弱，但在静态链接场景下仍然存在风险。

**第三，许可证冲突。** 某些开源许可证之间是互不兼容的。比如，你不能把 `GPL-2.0` 的代码和 `Apache-2.0` 的代码混在一起发布，因为它们的专利条款和再授权条款存在冲突。当间接依赖中同时出现这两种许可证时，团队可能陷入了无法察觉的「许可证死锁」。

**第四，合规流程缺失。** 很多团队在选择技术栈时只考虑功能和性能，从不审查许可证。等到融资审计、上市尽调、客户合规审查时才临时抱佛脚，往往发现「木已成舟」——核心系统已经深度依赖了有问题的库，替换成本极高。

**本章的核心目标是：用 Trivy 的 License Scanner 建立开源许可证的自动化发现、分类、风险评估和合规治理流程，避免「代码写完了，公司却要开源」的悲剧。**

## 2. 项目设计

**场景**：云帆科技的法务与技术联席会，法务张律师、小胖（技术代表）和小白（架构师）正在讨论开源许可证的排查方案。

---

**张律师**：（推过来一叠打印纸）「我列了 20 个『高风险许可证』，你们技术团队看看，产品里有没有用到？」

**小胖**：（翻了几页）`GPL-2.0`、`GPL-3.0`、`AGPL-3.0`、`SSPL`、`CC-BY-NC`……张律师，我们这几十个服务加起来可能有上万个依赖，你让我人工一个个查？查到明年也查不完啊。」

**小白**：「而且依赖关系每天都在变。昨天合并了一个 PR，可能就引入了一个新的间接依赖。人工查根本跟不上节奏。」

**大师**：「你们说的对，人工审查在 modern software 的依赖规模下已经不可行了。但好消息是，Trivy 内置了 License Scanner，可以自动识别依赖包的开源许可证。它的原理是：解析 `package.json`、`pom.xml`、`go.mod` 等清单文件 → 提取每个依赖包的版本 → 查询许可证数据库 → 输出每个包的许可证类型。」

**张律师**：「那它能识别所有许可证吗？那些『假开源』的协议，比如 MongoDB 的 SSPL，能认出来吗？」

**大师**：「Trivy 的许可证数据库覆盖了 SPDX（Software Package Data Exchange）标准定义的所有许可证，包括主流许可证（MIT、Apache-2.0、BSD、GPL 系列）和近年来出现的新许可证（SSPL、Elastic License、BSL 等）。而且它不仅识别直接依赖，还能穿透到间接依赖。你执行一条命令，就能拿到整个依赖树的许可证全景图。」

**小胖**：「那如果我一个项目里同时有 MIT 和 GPL 的依赖，Trivy 能告诉我它们之间冲不冲突吗？」

**大师**：「技术映射：这是个好问题。Trivy 目前主要做『发现』和『分类』，不直接做『兼容性判断』。但它识别出来的许可证清单，可以作为兼容性分析工具的输入。你可以把 Trivy 的输出导入 FOSSology 或 OpenChain 等合规平台，做深度的兼容性分析。或者，更简单的方式是制定团队的『许可证白名单』：只允许使用 MIT、Apache-2.0、BSD、ISC 等宽松许可证；一旦发现 GPL、AGPL、SSPL 等严格许可证，自动阻断构建。」

**张律师**：「白名单这个思路好。但万一我们确实需要用某个 GPL 的工具，怎么办？」

**小白**：「有几种合法的规避方式：

1. **进程隔离**：把 GPL 组件放在独立的进程中，通过 IPC（如 HTTP、gRPC）与本进程通信。GPL 的传染性只作用于『链接』或『同一进程地址空间』的代码，跨进程调用不触发传染。
2. **动态链接 + 用户自行替换**：对于 LGPL 的库，如果采用动态链接，且允许用户自行替换 so 文件，可以不触发传染性。
3. **替代方案**：寻找功能相近但许可证更宽松的替代品。比如用 `Fluentd` 替代 `Logstash`（避免 Elastic License），用 `PostgreSQL` 替代某些 GPL 的数据库工具。
4. **商业授权**：联系版权方购买商业授权，绕过 GPL 的开源要求。」

**大师**：「无论用哪种方式，前提是『先知道存在风险』。很多团队的悲剧不是『故意用了 GPL』，而是『根本不知道用了 GPL』。Trivy 的 License Scanner 就是解决这个『不知道』的问题。」

**张律师**：「那如果 Trivy 扫描发现项目里有 50 个 GPL 依赖，我是不是应该让开发全部替换掉？」

**小白**：「不要一刀切。首先要区分『开发依赖』和『生产依赖』。GPL 的测试框架（如 `JUnit` 早期版本）只在测试阶段使用，不打包到生产镜像，通常不构成传染风险。其次要区分『工具依赖』和『链接依赖』。构建工具（如 GCC、Maven）本身可能是 GPL 的，但它们只参与编译过程，不参与最终分发。」

**大师**：「技术映射：GPL 的传染性就像感冒病毒——只有密切接触（链接/同一进程）才会传染。如果只是和感冒患者擦肩而过（不同进程/不打包），不会被传染。Trivy 的报告会标注每个依赖的『分类』（library、application、tool 等），帮助你判断传染风险。」

**小胖**：「那我们今天定个规矩吧：以后所有新项目引入依赖前，必须先过一遍许可证扫描；所有老项目，这个月内全部扫一遍，出一份许可证合规报告给法务。」

**大师**：「我建议分三步走：

1. **扫描摸底**：对所有存量项目执行 `trivy fs --scanners license`，输出许可证全景报告。
2. **制定策略**：法务和技术联合制定《开源许可证治理规范》，定义白名单、灰名单、黑名单。
3. **嵌入流程**：在 CI 中配置许可证门禁，黑名单许可证直接阻断构建；灰名单许可证需要审批流程。」

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+，已安装
- **测试项目**：一个包含多种许可证依赖的 Node.js 和 Java 项目
- **网络**：可访问 Trivy 许可证数据库（或已预置）

### 步骤一：扫描项目许可证全景

**目标**：对一个真实项目执行许可证扫描，了解依赖的许可证分布。

```bash
# 创建测试项目
mkdir license-demo && cd license-demo
npm init -y

# 安装不同许可证的依赖
npm install lodash  # MIT
npm install express # MIT
npm install request # Apache-2.0
npm install left-pad # WTFPL
npm install json-schema  # AFL-2.1 OR BSD-3-Clause (dual license)

# 执行许可证扫描
trivy fs --scanners license .
```

**预期输出**（截取关键部分）：
```
Node.js (node-pkg)
==================
Total: 156 (UNKNOWN: 12, MIT: 98, Apache-2.0: 35, BSD-3-Clause: 8, ISC: 3)

┌─────────────────────────────┬─────────────────────────────┐
│         Package             │          License            │
├─────────────────────────────┼─────────────────────────────┤
│ lodash                      │ MIT                         │
│ express                     │ MIT                         │
│ request                     │ Apache-2.0                  │
│ left-pad                    │ WTFPL                       │
│ json-schema                 │ AFL-2.1, BSD-3-Clause       │
│ inherits                    │ ISC                         │
└─────────────────────────────┴─────────────────────────────┘
```

**关键观察**：
- `Total: 156`：这个项目有 156 个依赖（包括间接依赖），只有 5 个是直接安装的。
- `UNKNOWN: 12`：有 12 个包没有明确的许可证声明。这在法律上是高风险——没有声明不等于「可以随意使用」。
- `WTFPL`：`left-pad` 使用的是「Do What The F*ck You Want To Public License」，虽然宽松，但某些企业法务可能不接受这种非标准表述。

### 步骤二：配置许可证白名单与黑名单策略

**目标**：在 CI 中阻断高风险许可证的引入。

创建 `trivy-license.yaml`：

```yaml
license:
  full: true  # 显示所有依赖的许可证，而非仅显示冲突的
  forbidden:
    - AGPL-3.0
    - GPL-2.0
    - GPL-3.0
    - SSPL-1.0
    - CC-BY-NC-4.0
    - proprietary  # 禁止未授权的商业软件
  restricted:
    - LGPL-2.1
    - LGPL-3.0
    - MPL-2.0
    - EPL-2.0
  permissive:
    - MIT
    - Apache-2.0
    - BSD-2-Clause
    - BSD-3-Clause
    - ISC
```

**策略解读**：
- `forbidden`（黑名单）：一旦发现，立即阻断构建。这些许可证具有强传染性或商业限制。
- `restricted`（灰名单）：允许使用，但需要法务审批。这些许可证有一定传染性或兼容性问题。
- `permissive`（白名单）：自由使用，无需额外审批。

执行带策略的扫描：

```bash
trivy fs --scanners license --license-full --config trivy-license.yaml .
```

> **可能遇到的坑**：Trivy 的 License Scanner 默认只显示「有问题的」许可证。加上 `--license-full` 可以输出所有依赖的许可证，便于生成完整的 SBOM 合规报告。

### 步骤三：生成许可证合规报告

**目标**：为审计和法务提供标准化的许可证清单。

```bash
# 输出 JSON 格式
trivy fs --scanners license --license-full --format json --output license-report.json .

# 用 jq 提取关键字段
cat license-report.json | jq -r '.Results[]?.Licenses?[]? | [.PkgName, .Version, (.License[]? // "UNKNOWN")] | @tsv'
```

**进阶：生成 CSV 报告供法务审核**

创建 `license-to-csv.py`：

```python
#!/usr/bin/env python3
import json
import csv
import sys

def convert_to_csv(input_path, output_path):
    with open(input_path) as f:
        report = json.load(f)
    
    rows = []
    for result in report.get("Results", []):
        for lic in result.get("Licenses", []):
            rows.append({
                "Package": lic.get("PkgName", "N/A"),
                "Version": lic.get("Version", "N/A"),
                "License": ", ".join(lic.get("License", ["UNKNOWN"])),
                "Confidence": lic.get("Confidence", "N/A"),
                "Target": result.get("Target", "N/A")
            })
    
    with open(output_path, 'w', newline='') as f:
        writer = csv.DictWriter(f, fieldnames=["Package", "Version", "License", "Confidence", "Target"])
        writer.writeheader()
        writer.writerows(rows)
    
    print(f"CSV report generated: {output_path} ({len(rows)} packages)")

if __name__ == "__main__":
    convert_to_csv(sys.argv[1], sys.argv[2])
```

运行：

```bash
python license-to-csv.py license-report.csv license-report.csv
```

### 步骤四：Java 项目的许可证扫描

**目标**：验证 Trivy 对 Java Maven 项目的许可证识别能力。

创建 `pom.xml`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.example</groupId>
  <artifactId>license-java-demo</artifactId>
  <version>1.0</version>
  <dependencies>
    <dependency>
      <groupId>org.apache.commons</groupId>
      <artifactId>commons-lang3</artifactId>
      <version>3.12.0</version>
    </dependency>
    <dependency>
      <groupId>junit</groupId>
      <artifactId>junit</artifactId>
      <version>4.13.2</version>
      <scope>test</scope>
    </dependency>
  </dependencies>
</project>
```

扫描：

```bash
trivy fs --scanners license .
```

**预期输出**：
```
Java (jar)
==========
Total: 2

┌─────────────────────────────┬─────────────────────────────┐
│         Package             │          License            │
├─────────────────────────────┼─────────────────────────────┤
│ org.apache.commons:commons… │ Apache-2.0                  │
│ junit:junit                 │ EPL-1.0                     │
└─────────────────────────────┴─────────────────────────────┘
```

**关键观察**：
- `commons-lang3` 是 `Apache-2.0`，属于宽松许可证。
- `junit:4.13.2` 是 `EPL-1.0`（Eclipse Public License），具有文件级传染性。但由于它是 `test` scope，不进入生产镜像，风险可控。

> **可能遇到的坑**：某些 Java 包的 POM 文件中缺少 `<licenses>` 声明，Trivy 会标记为 `UNKNOWN`。对于这类包，需要人工核实其 GitHub 仓库或官网的许可证信息。

### 步骤五：镜像中的许可证扫描

**目标**：扫描生产镜像中的所有依赖许可证，确保没有「带病上线」。

```bash
# 扫描一个生产镜像
trivy image --scanners license myapp:latest

# 如果镜像基于多阶段构建，Trivy 仍能识别最终镜像中的所有包
```

**与 `trivy fs` 的差异**：
- `trivy fs` 扫描项目目录的锁定文件，可能包含开发依赖。
- `trivy image` 扫描实际进入镜像的包，更贴近生产环境的真实情况。

### 步骤六：CI 中配置许可证门禁

**目标**：在代码合并前阻断黑名单许可证的引入。

**GitHub Actions 示例**：

```yaml
name: License Compliance
on: [pull_request]
jobs:
  license-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Trivy License Scanner
        uses: aquasecurity/trivy-action@master
        with:
          scan-type: 'fs'
          scanners: 'license'
          scan-ref: '.'
          exit-code: '1'
          severity: 'HIGH,CRITICAL'
```

> **注意**：Trivy 的 License Scanner 目前不直接支持按「许可证类型」设置 exit-code。更精细的控制可以通过解析 JSON 输出来实现：

```bash
#!/bin/bash
# license-gate.sh
trivy fs --scanners license --format json --output license.json .

# 检查是否包含黑名单许可证
FORBIDDEN=$(cat license.json | jq -r '.Results[]?.Licenses?[]?.License[]?' | grep -iE "GPL|AGPL|SSPL" || true)

if [ -n "$FORBIDDEN" ]; then
    echo "ERROR: Forbidden licenses detected:"
    echo "$FORBIDDEN"
    exit 1
fi

echo "License check passed."
```

### 测试验证

1. 执行 `trivy fs --scanners license .`，确认能识别 `lodash`（MIT）、`request`（Apache-2.0）等包的许可证。
2. 检查报告中是否包含 `UNKNOWN` 许可证条目，并制定人工核实计划。
3. 运行 `license-to-csv.py`，确认 CSV 报告生成正确。
4. 扫描 Java 项目，确认 `EPL-1.0` 被正确识别。
5. 运行 `license-gate.sh`，验证包含 GPL 依赖时脚本返回非零退出码。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 覆盖范围 | 支持 npm/Maven/Go/Python 等主流生态 | 对 C/C++ 等无标准包管理器的生态支持有限 |
| 间接依赖 | 能穿透到深层依赖 | 某些间接依赖的许可证声明不完整 |
| 识别准确率 | 基于 SPDX 标准，覆盖主流许可证 | 双许可证场景下可能无法判断实际选用的许可证 |
| 集成能力 | JSON/CSV 输出便于对接合规平台 | 不直接提供许可证兼容性分析 |
| CI 集成 | 易于嵌入构建流水线 | 缺少按许可证类型的内置门禁机制 |

### 适用场景

1. **融资/上市尽调**：快速生成第三方依赖的许可证清单，满足投资人审计要求。
2. **企业开源治理**：建立许可证白名单/黑名单，防止高风险许可证进入代码库。
3. **产品出海合规**：满足不同国家/地区对开源许可证的披露要求。
4. **供应链安全**：识别具有商业限制条款（如 SSPL、Elastic License）的依赖，避免未来被「反授权」。
5. **客户交付**：向企业客户提供 SBOM 和许可证清单，作为安全合规的交付物。

**不适用场景**：
1. 需要精确的法律意见——Trivy 只能识别许可证类型，不能替代律师的专业判断。
2. 需要自动化的许可证兼容性分析——需配合 FOSSology、OpenChain 等专业平台。

### 注意事项

- **UNKNOWN 许可证的风险**：Trivy 标记为 `UNKNOWN` 的包不代表「无许可证」，而是「无法自动识别」。这类包必须人工核实，否则存在法律风险。
- **双许可证的解析**：某些包声明「Licensed under MIT OR Apache-2.0」，表示用户可以选择适用其中一个。Trivy 会输出两个许可证，团队需要根据策略选择更有利的一个。
- **开发依赖 vs 生产依赖**：测试框架、构建工具等开发依赖通常不进入生产分发，其许可证风险低于生产依赖。建议在扫描时区分 scope。

### 常见踩坑经验

**踩坑案例 1：间接依赖引入 GPL 代码**
- **现象**：团队确认直接依赖都是 MIT/Apache，但 Trivy 扫描发现某个深层依赖是 GPL。
- **根因**：依赖树的某个节点引入了 GPL 子依赖。
- **解法**：使用 `npm ls <package>` 或 `mvn dependency:tree` 定位引入路径；评估是否可以替换上层依赖，或采用进程隔离方案。

**踩坑案例 2：许可证声明与实际不符**
- **现象**：Trivy 识别为 MIT，但包的 GitHub 仓库显示是 GPL。
- **根因**：包的 `package.json` 或 POM 文件中的许可证声明错误或过时。
- **解法**：以源代码仓库的 LICENSE 文件为准，向包维护者提交 PR 修正元数据；在 `.trivyignore` 中备注实际许可证。

**踩坑案例 3：镜像扫描遗漏开发依赖**
- **现象**：`trivy fs` 发现了 200 个依赖，`trivy image` 只发现了 80 个。
- **根因**：多阶段构建中，开发依赖没有进入最终镜像。
- **解法**：如果目的是「生产合规」，以 `trivy image` 的结果为准；如果目的是「完整审计」，以 `trivy fs` 的结果为准。

### 思考题

1. 假设你的团队发现一个核心依赖从 `Apache-2.0` 迁移到了 `SSPL-1.0`（如 MongoDB 的历史事件）。请设计一个应急预案：如何在不影响业务连续性的前提下，评估替换成本、寻找替代方案、并通知所有受影响的团队？
2. Trivy 扫描出一个包的许可证是 `UNKNOWN`，但该包在你的系统中处于关键路径。请设计一个「UNKNOWN 许可证处理流程」，包括信息收集、风险评估和决策记录的标准化模板。

> **答案提示**：第 11 章「SBOM 生成、消费与供应链安全初探」将介绍如何把许可证信息嵌入 SBOM，实现全链路合规追踪。
