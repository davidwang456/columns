# 第24章：性能调优与大规模镜像扫描策略

> 版本：Trivy v0.50+
> 面向人群：DevOps、平台工程师、SRE

---

## 1. 项目背景

### 业务场景

云帆科技的微服务架构发展到 80+ 个服务，每个服务对应一个容器镜像。随着每周 2 次的全量发版节奏，安全团队需要在发版窗口内完成所有镜像的漏洞扫描。起初，这个任务由一台 8 核 16GB 的 Jenkins Slave 执行——串行扫描，一个一个来。

问题在镜像数量突破 50 个后集中爆发。某次发版前夜，全量扫描耗时 4 小时 23 分钟——从晚上 10 点开始，到凌晨 2 点多才完成。更致命的是，扫描过程中 Jenkins Slave OOM 了两次——支付网关镜像解压后达到 3.8GB，Trivy 的内存峰值冲到了 12GB，直接触发 Container Runtime 的 OOM Killer。

DevOps 负责人小李在复盘会上拿出了令人焦虑的数据：

- **扫描耗时增长率**：每增加 10 个镜像，全量扫描时间增加约 45 分钟（近似线性增长）。
- **资源占用峰值**：单次扫描的瞬时内存可达镜像大小的 3-4 倍（解压 + 分析 + 数据库查询）。
- **缓存命中率**：不足 15%——每次发版都重建镜像，即使基础镜像层完全相同，Trivy 也在重复分析。
- **失败重扫成本**：OOM 或网络超时导致的失败，需要全量重扫（而非断点续扫）。

CTO 下了死命令：「下个月我们要上线 30 个新服务，按现在的速度全量扫描要 6 个小时，这绝对不行。我要你在一个月内把全量扫描时间降到 30 分钟以内。」

### 痛点放大

**第一，串行扫描浪费 I/O 等待时间。** 镜像拉取和解压是 I/O 密集型操作，CPU 大部分时间在等网络和磁盘。串行模式下，一个镜像在等待 docker pull 时，CPU 几乎完全空闲。这种「人等车」的模式在大规模场景下极其低效。

**第二，大镜像的内存黑洞。** 某些 AI 推理镜像（含 PyTorch、CUDA）体积可达 5-8GB。Trivy 的解压分析过程需要将镜像层加载到内存，瞬时内存需求可能超过 20GB。如果不做资源限制，一台 64GB 的机器可能被 3 个并发扫描打满。

**第三，重复分析的浪费。** 80 个微服务镜像中，有 60 个共享同一个 `cloud-sail/base:2.0` 基础镜像层。但在没有层缓存的情况下，Trivy 每次都要重新分析这 60 个相同的层——重复计算占扫描时间的 40% 以上。

**第四，增量扫描缺失。** 大多数发版只修改了业务代码层（最后 1-2 层），基础镜像层和依赖层几乎不变。但全量扫描模式对每一层都重新分析，大量时间浪费在没有变化的层上。

**本章的核心目标是：通过并行调度、分层缓存、增量扫描和资源限制四大优化手段，将大规模镜像扫描时间从「小时级」压降到「分钟级」。**

## 2. 项目设计

**场景**：云帆科技的性能优化专题会，小李（DevOps 负责人）、小胖（开发代表）、大师正在分析扫描瓶颈。

---

**小胖**：「4 个小时扫 80 个镜像，平均每个镜像 3 分钟，好像也不慢啊？我们开发环境跑个全量测试还要 40 分钟呢。」

**小李**：「账不是这么算的。平均 3 分钟包括了大量等待时间——实际分析过程可能就 1 分钟，但拉镜像花了 2 分钟。而且 OOM 导致的重扫成本没算进去。那天晚上实际上扫了 110 个『镜像次』——30 个因为 OOM 失败了重新扫的。」

**大师**：「技术映射：扫描性能就像高速公路的通行效率。串行扫描相当于只有一条车道——一辆车停收费站交费时，后面所有车都等着。并行扫描相当于多车道——10 个收费站同时服务，每辆车占用一个窗口。但收费站数量（并发度）不是越多越好——如果收费站比你有的车道还多，后面的车照样排队。」

**小胖**：「那就多开几个并行度嘛，把 `--parallel` 调大不就行了？」

**小白**：「并行度不是灵丹妙药。要考虑三个限制因素：

1. **网络带宽**：同时拉 10 个 2GB 的镜像，网络可能直接打满，每个镜像的拉取速度反而更慢。
2. **磁盘 I/O**：解压是磁盘密集型操作，10 个镜像同时解压可能导致磁盘队列深度爆满。
3. **内存容量**：你调了 10 个并行，但机器只有 32GB 内存，每个大镜像可能吃 4GB——实际只能跑 6-7 个并发，多的就排队。」

**大师**：「所以我们需要的不是简单的『提高并行度』，而是一个系统化的优化策略。核心就四个方向：

**方向一：分层缓存。** Trivy 在分析镜像时，会逐层解压并生成该层的包清单（PkgInfo）。如果多个镜像共享同一个基础层（比如都用 `ubuntu:22.04`），那这一层的包清单只需要生成一次，后续扫描直接从缓存读取。这就是 `--cache-dir` 的核心价值。

**方向二：增量扫描。** 基于 SBOM 的差异分析。假设上周你已经扫描过 `payment:v2.3.0`，并保存了 SBOM。这周发版 `payment:v2.3.1`，只改了业务代码层。你可以只对新增层做分析，基础层直接用缓存的 SBOM。

**方向三：高效调度。** 用消息队列（Redis/RabbitMQ）做任务分发，配合 Worker 池做并发控制。每个 Worker 独立拉取任务、执行扫描、上报结果。这是一个经典的生产者-消费者模型。

**方向四：资源配置。** 不只用 `--parallel`。还要控制容器的 `--memory` 限制，设置 GOMAXPROCS、GOMEMLIMIT 等 Go runtime 参数，甚至用 Linux cgroup 做严格的资源隔离。」

**小胖**：「等一下，增量扫描听起来很厉害，但 Trivy 本身有这个功能吗？」

**小白**：「Trivy 没有内置的『仅扫描变更层』命令，但可以通过组合现有功能实现：

1. 先用 `trivy image --format cyclonedx` 导出上一版本的 SBOM。
2. 对于新版本镜像，找到变更的层（通过比较 layer digest）。
3. 用 `trivy sbom` 从上一版本的 SBOM 反推漏洞——这部分结果可复用。
4. 只对变更层的依赖做增量分析。
5. 合并结果。

实际上，更实用的做法是『基于缓存加速全量扫描』——有了分层缓存后，即使做全量扫描，重复层的分析时间也接近零。这与增量扫描的效果趋同，但实现更简单。」

**小李**：「那扫描队列怎么设计？我们是 Jenkins Pipeline，要给每个镜像建一个 Stage 吗？」

**大师**：「Jenkins 的并行 Stage 可以做，但更推荐的是『Worker 池』模式：

```
                         ┌─────────────┐
                         │  消息队列    │
                         │  (Redis)    │
                         └──────┬──────┘
              ┌─────────────────┼─────────────────┐
        ┌─────▼─────┐    ┌─────▼─────┐    ┌─────▼─────┐
        │  Worker 1  │    │  Worker 2  │    │  Worker N  │
        │  (Trivy)   │    │  (Trivy)   │    │  (Trivy)   │
        └─────┬─────┘    └─────┬─────┘    └─────┬─────┘
              └─────────────────┼─────────────────┘
                         ┌──────▼──────┐
                         │  结果收集器  │
                         │  (PostgreSQL)│
                         └─────────────┘
```

每个 Worker 监听 Redis 的队列（如 `scan:queue`），弹出任务、执行扫描、推送结果。Jenkins Pipeline 只需要发一个『开始扫描』信号，然后轮询结果状态即可。这种架构的扩展性极佳——加一台机器就能加 N 个 Worker。」

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+，已安装
- **Redis**：v7.0+（用于任务队列）
- **Go**：1.21+（Worker 程序编译）
- **测试镜像**：准备 50 个不同大小的测试镜像（100MB ~ 5GB）

### 步骤一：建立分层缓存基线

**目标**：配置 Trivy 缓存目录，预热基础镜像层的分析结果。

```bash
# 创建专用缓存目录
export TRIVY_CACHE_DIR=/data/trivy/cache
mkdir -p $TRIVY_CACHE_DIR

# 预热常用基础镜像
for base in ubuntu:22.04 alpine:3.19 debian:12-slim node:20-alpine python:3.12-slim; do
    echo "=== Pre-warming: $base ==="
    trivy image --cache-dir $TRIVY_CACHE_DIR \
        --offline-scan \
        --no-progress \
        $base
done

# 查看缓存大小和命中统计
du -sh $TRIVY_CACHE_DIR
```

**预期效果**：缓存目录填充完成后，后续扫描共享这些基础层的镜像时，Trivy 跳过已分析层的重复解压和分析，仅处理业务层。

```bash
# 对比扫描耗时：无缓存 vs 有缓存
# 假设 payment:v2.3.1 基于 node:20-alpine

# 无缓存（清除缓存后首次扫描）
rm -rf $TRIVY_CACHE_DIR
time trivy image --cache-dir $TRIVY_CACHE_DIR payment:v2.3.1
# 输出：scanned in 95s

# 有缓存（基础层已预热）
time trivy image --cache-dir $TRIVY_CACHE_DIR payment:v2.3.1
# 输出：scanned in 28s（业务层分析时间）
```

> **坑点**：缓存目录必须是持久化卷（如 K8s PVC 或 EBS），不能是容器临时存储。否则容器重启后缓存丢失，预热效果归零。

### 步骤二：编写并行扫描调度器

**目标**：用 Redis 队列 + Go Worker 实现可控并发的批量扫描。

创建 `scanner-worker/main.go`：

```go
package main

import (
    "context"
    "encoding/json"
    "fmt"
    "log"
    "os"
    "os/exec"
    "sync"
    "time"

    "github.com/redis/go-redis/v9"
)

// ScanTask 扫描任务定义
type ScanTask struct {
    Image    string `json:"image"`
    Severity string `json:"severity"`
    JobID    string `json:"job_id"`
}

// ScanResult 扫描结果定义
type ScanResult struct {
    JobID      string `json:"job_id"`
    Image      string `json:"image"`
    Status     string `json:"status"` // success / failed
    Duration   string `json:"duration"`
    Critical   int    `json:"critical"`
    High       int    `json:"high"`
    ErrorMsg   string `json:"error_msg,omitempty"`
}

func main() {
    redisAddr := os.Getenv("REDIS_ADDR")
    if redisAddr == "" {
        redisAddr = "localhost:6379"
    }
    cacheDir := os.Getenv("TRIVY_CACHE_DIR")
    if cacheDir == "" {
        cacheDir = "/data/trivy/cache"
    }

    rdb := redis.NewClient(&redis.Options{Addr: redisAddr})

    // 从环境变量读取并发数
    concurrency := 5
    taskQueue := "scan:queue"
    resultQueue := "scan:results"

    var wg sync.WaitGroup
    sem := make(chan struct{}, concurrency)

    for {
        // 从 Redis 队列拉取任务（阻塞式，超时 5 秒）
        result, err := rdb.BRPop(context.Background(), 5*time.Second, taskQueue).Result()
        if err != nil {
            continue // 队列为空，继续等待
        }

        var task ScanTask
        json.Unmarshal([]byte(result[1]), &task)

        sem <- struct{}{} // 获取信号量
        wg.Add(1)

        go func(t ScanTask) {
            defer wg.Done()
            defer func() { <-sem }() // 释放信号量

            scanResult := executeScan(t, cacheDir)

            data, _ := json.Marshal(scanResult)
            rdb.LPush(context.Background(), resultQueue, string(data))

            fmt.Printf("[%s] %s scanned: %d CRITICAL, %d HIGH\n",
                time.Now().Format("15:04:05"), t.Image, scanResult.Critical, scanResult.High)
        }(task)
    }
}

func executeScan(task ScanTask, cacheDir string) ScanResult {
    start := time.Now()

    cmd := exec.Command("trivy", "image",
        "--cache-dir", cacheDir,
        "--severity", task.Severity,
        "--format", "json",
        "--no-progress",
        "--quiet",
        task.Image,
    )

    output, err := cmd.CombinedOutput()
    duration := time.Since(start)

    result := ScanResult{
        JobID:    task.JobID,
        Image:    task.Image,
        Duration: duration.String(),
    }

    if err != nil {
        result.Status = "failed"
        result.ErrorMsg = string(output)
        return result
    }

    result.Status = "success"

    // 解析 JSON 输出提取漏洞数量
    var report struct {
        Results []struct {
            Vulnerabilities []struct {
                Severity string `json:"Severity"`
            } `json:"Vulnerabilities"`
        } `json:"Results"`
    }
    json.Unmarshal(output, &report)

    for _, r := range report.Results {
        for _, v := range r.Vulnerabilities {
            switch v.Severity {
            case "CRITICAL":
                result.Critical++
            case "HIGH":
                result.High++
            }
        }
    }

    return result
}
```

> **关键设计**：用 buffered channel (`sem`) 实现信号量模式，限制最大并发数。当并发 Worker 达到上限时，新任务在 `sem <- struct{}{}` 处阻塞，直到有 Worker 完成并释放槽位。

### 步骤三：配置 Go Runtime 参数与环境变量

**目标**：控制 Trivy 的内存和 CPU 使用，防止 OOM。

```bash
# 设置 Go Runtime 参数
export GOMAXPROCS=2           # 限制 CPU 核心数
export GOMEMLIMIT=4096MiB     # 软限制 4GB 内存
export TRIVY_CACHE_DIR=/data/trivy/cache

# 对于大镜像，使用 --parallel 限制内部并行
# Trivy 内部分析镜像层时也有并行操作
trivy image \
  --parallel 2 \
  --cache-dir $TRIVY_CACHE_DIR \
  --timeout 10m \
  --severity HIGH,CRITICAL \
  my-large-image:5gb
```

**Docker 容器模式下更严格的控制**：

```yaml
# docker-compose.yml
services:
  trivy-worker:
    image: aquasec/trivy:0.50.0
    environment:
      - GOMAXPROCS=3
      - GOMEMLIMIT=3072MiB
      - TRIVY_CACHE_DIR=/cache
    deploy:
      resources:
        limits:
          cpus: '4'
          memory: 4096M
        reservations:
          cpus: '2'
          memory: 2048M
    volumes:
      - trivy-cache:/cache
      - /var/run/docker.sock:/var/run/docker.sock

volumes:
  trivy-cache:
    driver: local
```

> **坑点**：`GOMEMLIMIT` 是 Go 1.19+ 引入的软限制——当堆内存接近限制时，GC 会更频繁地运行，但不会强杀进程。硬限制应该由容器运行时的 `--memory` 提供。两者配合使用：`GOMEMLIMIT` 让 Go 主动控制自己的行为，`--memory` 兜底。

### 步骤四：编写任务分发器（Jenkins Pipeline 调用）

**目标**：在 CI/CD 中一键触发所有待扫描镜像的批量扫描。

创建 `batch-scan.py`：

```python
#!/usr/bin/env python3
"""
批量扫描分发器
从 HARBOR_URL 获取镜像列表，推送到 Redis 队列
"""

import json
import os
import time
import uuid

import redis
import requests

HARBOR_URL = os.environ["HARBOR_URL"]
HARBOR_USER = os.environ["HARBOR_USER"]
HARBOR_PASS = os.environ["HARBOR_PASS"]
REDIS_HOST = os.environ.get("REDIS_HOST", "localhost")
SCAN_SEVERITY = os.environ.get("SCAN_SEVERITY", "HIGH,CRITICAL")

class BatchScanner:
    def __init__(self):
        self.rdb = redis.Redis(host=REDIS_HOST, port=6379, decode_responses=True)
        self.session = requests.Session()
        self.session.auth = (HARBOR_USER, HARBOR_PASS)

    def get_images(self, project_name):
        """从 Harbor API 获取指定项目下的所有镜像和 tag"""
        url = f"{HARBOR_URL}/api/v2.0/projects/{project_name}/repositories"
        resp = self.session.get(url, params={"page_size": 100})
        resp.raise_for_status()

        images = []
        for repo in resp.json():
            # 获取每个仓库的 artifacts（tags）
            name = repo["name"].replace("/", "%2F")
            artifacts_url = f"{HARBOR_URL}/api/v2.0/projects/{project_name}/repositories/{name}/artifacts"
            art_resp = self.session.get(artifacts_url, params={"page_size": 50})
            art_resp.raise_for_status()

            for art in art_resp.json():
                for tag in art.get("tags", []):
                    images.append(f"{HARBOR_URL.split('//')[1]}/{repo['name']}:{tag['name']}")

        return images

    def push_tasks(self, images):
        """将扫描任务推送到 Redis 队列"""
        batch_id = str(uuid.uuid4())[:8]
        for i, img in enumerate(images):
            task = {
                "image": img,
                "severity": SCAN_SEVERITY,
                "job_id": f"{batch_id}-{i:04d}",
            }
            self.rdb.lpush("scan:queue", json.dumps(task))
            print(f"[{batch_id}] Queued: {img}")

        print(f"\nTotal: {len(images)} images queued. Batch ID: {batch_id}")
        return batch_id

    def wait_and_collect(self, batch_id, timeout=3600):
        """等待扫描完成并收集结果"""
        start = time.time()
        total = len(self.rdb.lrange("scan:queue", 0, -1))  # 初始
        results = []

        while time.time() - start < timeout:
            # 收集已完成的结果
            while True:
                raw = self.rdb.lpop("scan:results")
                if not raw:
                    break
                result = json.loads(raw)
                results.append(result)
                print(f"  [{result['status']}] {result['image']}: "
                      f"CRITICAL={result.get('critical', 0)}, HIGH={result.get('high', 0)}")

            queue_len = self.rdb.llen("scan:queue")
            if queue_len == 0:
                print(f"\nAll jobs completed. {len(results)} results collected.")
                break

            print(f"  Waiting... {queue_len} jobs remaining. Elapsed: {int(time.time()-start)}s")
            time.sleep(10)

        # 输出汇总
        critical_found = sum(1 for r in results if r.get("critical", 0) > 0)
        failed = sum(1 for r in results if r["status"] == "failed")
        print(f"\n{'='*60}")
        print(f"Batch ID: {batch_id}")
        print(f"Total scanned: {len(results)}")
        print(f"Images with CRITICAL: {critical_found}")
        print(f"Scans failed: {failed}")
        print(f"Total time: {int(time.time()-start)}s")
        print(f"{'='*60}")

        return results


if __name__ == "__main__":
    scanner = BatchScanner()
    project = os.environ.get("HARBOR_PROJECT", "production")
    images = scanner.get_images(project)
    print(f"Found {len(images)} images in project '{project}'")
    batch_id = scanner.push_tasks(images)
    scanner.wait_and_collect(batch_id)
```

### 步骤五：性能基准测试

**目标**：对比串行、并行、优化三种模式的性能差异。

准备 50 个测试镜像（5 种基础镜像 × 10 个业务变体）：

```bash
#!/bin/bash
# perf-benchmark.sh
echo "=== Trivy 扫描性能基准测试 ==="

# 模式一：串行（逐个扫描）
echo "--- 串行模式 ---"
START=$(date +%s)
for img in $(cat images.txt); do
    trivy image --format json --no-progress -q "$img" > /dev/null
done
SERIAL_TIME=$(($(date +%s) - START))
echo "串行耗时: ${SERIAL_TIME}s"

# 模式二：简单并行（xargs -P）
echo "--- 多进程并行模式 ---"
START=$(date +%s)
cat images.txt | xargs -P 5 -I {} trivy image --format json --no-progress -q {} > /dev/null
PARALLEL_TIME=$(($(date +%s) - START))
echo "并行耗时: ${PARALLEL_TIME}s"

# 模式三：优化模式（缓存 + 并发控制 + GOMEMLIMIT）
echo "--- 优化模式 ---"
export GOMEMLIMIT=3072MiB
export TRIVY_CACHE_DIR=/data/trivy/cache
START=$(date +%s)
python3 batch-scan.py
OPTIMIZED_TIME=$(($(date +%s) - START))
echo "优化耗时: ${OPTIMIZED_TIME}s"

# 输出对比表
echo ""
echo "| 模式 | 耗时(s) | 相对串行提升 | OOM次数 |"
echo "|------|---------|-------------|---------|"
echo "| 串行 | ${SERIAL_TIME} | - | - |"
echo "| 并行 | ${PARALLEL_TIME} | $((100 - PARALLEL_TIME * 100 / SERIAL_TIME))% | - |"
echo "| 优化 | ${OPTIMIZED_TIME} | $((100 - OPTIMIZED_TIME * 100 / SERIAL_TIME))% | - |"
```

**典型测试结果**：

| 模式 | 耗时(s) | 相对串行提升 | OOM次数 | 说明 |
|------|---------|-------------|---------|------|
| 串行 | 11480 (3h11m) | - | 2次 | 单 Worker 逐个扫描 |
| 并行(xargs -P5) | 3120 (52m) | 73% | 1次 | 5进程并发，无缓存 |
| 优化模式 | 1580 (26m) | 86% | 0次 | 缓存预热+并发控制+资源限制 |

> **关键发现**：缓存预热是性价比最高的优化手段——在不改变代码逻辑的情况下，仅靠复用基础层分析结果就能缩短 30-40% 的扫描时间。

### 测试验证

1. 配置 Redis 并启动 Worker 池（3 个 Worker），观察并发扫描时系统资源占用。
2. 推送 10 个镜像任务到队列，监控每个任务的处理时间和完成顺序。
3. 验证缓存命中：清除缓存后扫描 `ubuntu:22.04`，记录耗时；再次扫描同一镜像，确认耗时大幅缩短。
4. 模拟大镜像 OOM：用 `docker run --memory=512m` 限制容器内存，扫描 5GB 镜像，验证 Worker 能否优雅处理失败。
5. 运行性能基准脚本，确认优化模式比串行模式提升 5 倍以上。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 分层缓存 | 大幅减少重复分析；零代码侵入 | 缓存占用磁盘空间（50镜像约2-5GB） |
| 并行调度 | 充分利用多核；天然支持水平扩展 | Redis 依赖增加运维复杂度 |
| 增量分析 | 理论上最优的加速策略 | 需要额外维护 SBOM 和版本映射 |
| 资源限制 | 防止 OOM；保障其他服务稳定 | 限制过紧导致扫描变慢 |
| Worker 池 | 弹性伸缩；故障隔离 | 需要编排层（如 K8s）管理 Worker 生命周期 |

### 适用场景

1. **大型微服务集群**（50+ 镜像）：必选优化方案，否则扫描成为发版瓶颈。
2. **多基础镜像的混合环境**（如同时用 Alpine、Debian、CentOS）：分层缓存收益最高。
3. **频繁发版场景**（日发版或每周多次发版）：缓存命中率高，二次扫描几乎秒级。
4. **CI/CD 流水线的扫描阶段**：用 Worker 池替代单机扫描，流水线无需等待。
5. **镜像镜像刷新**：对已有镜像仓库执行定期重扫时，缓存带来显著加速。

**不适用场景**：
1. 镜像总数 < 10 个的小团队——优化投入可能超过收益。
2. 每次发版都更换基础镜像的场景——缓存命中率低，预热成本高。

### 注意事项

- **缓存目录必须是持久化存储**。容器重建时必须保留 cache-dir。
- **并行度不是越高越好**。推荐 `并发数 = CPU 核心数 × 1.5`，上限受制于内存和网络。
- **Go Runtime 参数在生产环境需充分测试**。`GOMEMLIMIT` 设置过低会导致 GC 频繁，反而拖慢扫描。
- **Redis 队列需设置 TTL**。防止失败的或超时的任务无限堆积。

### 常见踩坑经验

**踩坑案例 1：缓存目录权限问题导致 Worker 启动失败**
- **现象**：Worker 日志报错 `mkdir /data/trivy/cache: permission denied`。
- **根因**：容器以非 root 用户运行（UID 1000），但挂载的 PVC 所有权是 root。
- **解法**：用 `initContainer` 做 `chown 1000:1000 /data/trivy/cache`，或在 K8s 中配置 `fsGroup: 1000`。

**踩坑案例 2：concurrency 设置过高导致 Redis 连接数耗尽**
- **现象**：Worker 日志大量 `dial tcp: too many open files`。
- **根因**：每个 Worker 主循环持有一个连接，但 Go 的 goroutine 泄漏可能导致连接数飙升。
- **解法**：限制单机 Worker 数（`ulimit -n 65536` 兜底）；使用 Redis 连接池；增加 Worker 健康检查和重启机制。

**踩坑案例 3：批量扫描中个别镜像超时导致整体卡死**
- **现象**：batch-scan.py 在 `wait_and_collect` 阶段卡死，反复打印 "Waiting..."
- **根因**：某个镜像的扫描因网络超时无限等待，Worker 未能将其标记为 "failed" 并推送结果。
- **解法**：在 `executeScan` 函数中设置 `cmd` 的超时（`context.WithTimeout`）；为每个任务加入 TTL，超时后自动标记失败。

### 思考题

1. 假设你的公司有 500 个镜像，每天凌晨 3 点执行一次全量扫描。如果扫描节点从 5 台扩展到 20 台，扫描总时间是否一定会线性缩短到原来的 1/4？不是的话，瓶颈可能在哪里？请从网络、存储、缓存一致性三个角度分析。

2. 在 Worker 池架构中，如果一个 Worker 在处理某个大镜像时 OOM 被 kill，该任务可能丢失（未返回结果）。请设计一个「任务重试与去重」机制：确保每个任务最终都被执行且只被成功执行一次。
