# 第27章：软件供应链安全端到端实践

> 版本：Trivy v0.50+
> 面向人群：安全架构师、DevSecOps、合规专员

---

## 1. 项目背景

### 业务场景

2023 年 12 月，某知名开源日志组件被发现植入了恶意后门，攻击者通过社会工程学获得了该项目的维护权限，在发版流程中悄悄注入了恶意代码。这一事件波及全球数万家企业——云帆科技也在受影响名单中。幸运的是，云帆科技当时还没有将这个组件升级到受影响版本。但安全负责人老李却彻夜未眠。

「如果那个后门不是植入在日志组件里，而是植入在我们的某个基础镜像里呢？」老李在次日紧急会议上抛出这个假设。瞬间，会议室安静了。

云帆科技的 CI/CD 流水线是这样的：开发提交代码 → GitHub Actions 构建镜像 → 推送至 Harbor → K8s 部署。整个链路中，唯一的安全检查是 Trivy 对最终镜像的漏洞扫描。但这个扫描只能发现「已知 CVE」，无法回答三个更根本的问题：

1. **谁构建的？** 这个镜像是从公司官方的 CI 流水线构建的，还是某个人在本地 `docker build` 然后手动 push 的？
2. **怎么构建的？** 构建过程是否使用了安全的构建环境？构建参数是否被篡改？
3. **里面有什么？** 镜像里包含的所有依赖（包括传递依赖、构建工具、基础镜像层），是否都可信？

如果没有这些信息，即使镜像没有已知漏洞，也无法排除它被投毒的可能性。这就是「供应链安全」的本质——不仅仅是扫漏洞，而是要回答「这个制品从头到尾的可信度」。

CTO 在会议总结时说了句狠话：「Log4j 告诉我们依赖可能有漏洞，XZ Utils 告诉我们维护者可能被攻破。下一次攻击，可能就藏在咱们自己的镜像里。供应链安全不是『要不要做』，是『什么时候被人利用之前做』。」

### 痛点放大

**第一，来源不可追溯。** 镜像是谁构建的、构建环境是什么、构建参数有哪些——这些关键信息在镜像元数据中几乎不存在。当安全事件发生时，无法快速从数百万镜像中区分「官方构建」和「潜在可疑镜像」。

**第二，SBOM 缺失或不可信。** 虽然有第 11 章的基础，但团队实践中 SBOM 的生成是「手动 + 偶尔」——出事了才想到生成 SBOM，平时没有自动化。更糟的是，SBOM 文件本身也可能被篡改——攻击者修改 SBOM 说「没有引入恶意依赖」，你无从验证。

**第三，依赖关系的「冰山效应」。** 一个 Java 应用引入了一个库，这个库又引入了 50 个间接依赖。如果其中一个深层依赖被投毒，你几乎不可能通过代码审查发现。只能通过「构建 provenance」来确认所有依赖在编译时是干净的。

**第四，监管压力。** 金融、政务客户越来越频繁地要求提供「软件供应链安全证明」。美国行政令 EO 14028、欧盟 Cyber Resilience Act 都在推动 SBOM 和供应链透明度。没有自动化体系，合规成本将指数级上升。

**本章的核心目标是：构建「SBOM 生成 → 签名 → 存储 → 验证 → VEX 发布」的完整供应链安全闭环，实现制品来源可追溯、内容可验证、风险可管理。**

## 2. 项目设计

**场景**：云帆科技的供应链安全策略会，老李（安全负责人）、小白（架构师）、大师正在设计端到端方案。

---

**小胖**：「供应链攻击听起来好遥远啊。我们一个小公司，谁会来攻击我们？」

**小白**：「攻击者的目标不是『云帆科技』，而是『用了某个镜像的公司们』。一次成功的供应链攻击可以同时影响数千家企业。比如攻击者入侵了 `node:20-alpine` 的构建流程，在所有基于这个镜像构建的应用里埋下后门。你不是被定向了，而是被『无差别收割』了。」

**大师**：「技术映射：供应链安全就像食品溯源。你去超市买牛奶，你可以扫描二维码看到『这瓶奶来自哪个牧场、哪个挤奶站、经过哪个加工厂、冷链车车牌号多少』。软件供应链安全要做的是同样的事——你能查到『这个镜像是哪个 CI 系统构建的、用了哪个基础镜像、构建过程中拉取了哪些依赖、每个依赖来自哪里』。」

**老李**：「那我们具体要怎么做？从哪开始？」

**大师**：「供应链安全的『四件套』：

1. **SBOM（Software Bill of Materials）**：软件物料清单。列出镜像里的所有组件、版本、许可证、依赖关系。相当于牛奶的『配料表』。

2. **Provenance（构建溯源）**：构建过程的元数据。谁、在什么环境、用什么命令、拉取了什么输入、产出了什么制品。相当于牛奶的『生产记录』。

3. **签名（Signature）**：对 SBOM 和 Provenance 的加密签名。证明这些文件确实是由可信的构建系统生成的，没有被中间人篡改。相当于牛奶瓶盖上的『防伪封签』。

4. **VEX（Vulnerability Exploitability eXchange）**：漏洞可利用性声明。说明哪些已知漏洞『不影响我们的产品』，以及为什么。相当于『过敏原提示：本品不含花生』。」

**小胖**：「所以我们现在 Scattered 地做了 1 和 4（SBOM 和 VEX），但 2 和 3（Provenance 和签名）完全没做？」

**小白**：「对。而且即使 SBOM 也没做全——我们只对部分镜像手动生成 SBOM，没有自动化。我们需要一条 CI/CD 流水线，在构建镜像时自动完成这四件事。」

**大师**：「技术映射：这就是 SLSA（Supply-chain Levels for Software Artifacts）框架。SLSA 定义了 4 个安全等级：

- **Level 1**：构建过程有文档记录（最低要求）。
- **Level 2**：构建过程使用版本控制和托管服务，生成 Provenance。
- **Level 3**：构建过程经过审计和控制（如两阶段代码审查、隔离构建）。
- **Level 4**：最高级别——构建过程的所有步骤都经过双人审查和加密验证。

大多数企业将 Level 2-3 作为目标。Trivy + cosign + SLSA GitHub Generator 这套组合可以帮助你达到 Level 2-3。」

**老李**：「那具体的技术栈是什么？」

**小白**：

| 环节 | 工具 | 作用 |
|------|------|------|
| 生成 SBOM | `trivy image --format cyclonedx` | 镜像内所有组件的物料清单 |
| 生成 Provenance | `slsa-framework/slsa-github-generator` | 构建过程的可验证溯源 |
| 签名 SBOM | `cosign sign` / `cosign attest` | 对 SBOM 和 Provenance 进行数字签名 |
| 存储与分发 | Harbor / OCI Registry | SBOM 和签名作为 OCI artifact 附加到镜像 |
| 验证签名 | `cosign verify` / `trivy image --sbom-sources oci` | 消费方验证签名和 SBOM |
| 生成 VEX | `vexctl create` | 声明漏洞的可利用性状态 |
| 验证全链路 | `trivy image --vex --sbom-sources` | 一键验证镜像、SBOM、签名、VEX |

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+
- **cosign**：v2.2+
- **oras**：v1.1+（用于 OCI artifact 操作）
- **Harbor**：v2.9+（需启用 cosign 支持）
- **GitHub Actions**：用于构建流水线

### 步骤一：在 CI/CD 中自动生成并签名 SBOM

**目标**：构建镜像后自动生成 SBOM，用 cosign 签名，并将 SBOM 作为 OCI artifact 附加到镜像。

创建 `.github/workflows/build-secure.yml`：

```yaml
name: Secure Build Pipeline
on:
  push:
    tags:
      - 'v*'
    branches:
      - main

env:
  REGISTRY: harbor.internal.example.com
  IMAGE_NAME: ${{ github.repository }}
  COSIGN_EXPERIMENTAL: "1"   # 启用 cosign 的 experimental 特性（keyless signing）

jobs:
  build-and-attest:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
      id-token: write        # 用于 cosign keyless signing（OIDC）
      actions: read

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      # Step 1: 登录容器仓库
      - name: Login to Harbor
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ secrets.HARBOR_USER }}
          password: ${{ secrets.HARBOR_PASSWORD }}

      # Step 2: 构建镜像
      - name: Build Docker Image
        run: |
          IMAGE_TAG="${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}"
          docker build -t "$IMAGE_TAG" .
          docker push "$IMAGE_TAG"
          echo "IMAGE_TAG=$IMAGE_TAG" >> $GITHUB_ENV

      # Step 3: 生成 SBOM（CycloneDX 格式）
      - name: Generate SBOM
        run: |
          trivy image --format cyclonedx \
            --output sbom.cdx.json \
            ${{ env.IMAGE_TAG }}

      # Step 4: 生成 SPDX 格式 SBOM（备用）
      - name: Generate SPDX SBOM
        run: |
          trivy image --format spdx-json \
            --output sbom.spdx.json \
            ${{ env.IMAGE_TAG }}

      # Step 5: 用 cosign sign 对镜像签名
      - name: Sign Image with cosign
        run: |
          cosign sign \
            --yes \
            --recursive \
            ${{ env.IMAGE_TAG }}

      # Step 6: 用 cosign attest 将 SBOM 附加到镜像
      - name: Attach SBOM as In-Toto Attestation
        run: |
          cosign attest \
            --yes \
            --type https://cyclonedx.org/schema \
            --predicate sbom.cdx.json \
            ${{ env.IMAGE_TAG }}

          cosign attest \
            --yes \
            --type https://spdx.org/Document \
            --predicate sbom.spdx.json \
            ${{ env.IMAGE_TAG }}

      # Step 7: 生成 SLSA Provenance（使用官方 Action）
      - name: Generate SLSA Provenance
        uses: slsa-framework/slsa-github-generator/.github/actions/generate-provenance@v1
        with:
          base64-subjects: "${{ needs.build.outputs.digest }}"

      # Step 8: Trivy 漏洞扫描
      - name: Vulnerability Scan
        run: |
          trivy image \
            --severity HIGH,CRITICAL \
            --format json \
            --output scan-result.json \
            ${{ env.IMAGE_TAG }}

          cat scan-result.json | jq '[.Results[]?.Vulnerabilities?[]?] | length'

      # Step 9: 上传扫描结果
      - name: Upload Scan Reports
        uses: actions/upload-artifact@v4
        with:
          name: security-reports
          path: |
            sbom.cdx.json
            sbom.spdx.json
            scan-result.json
```

> **坑点**：`cosign attest` 默认使用 OIDC keyless signing，依赖 GitHub OIDC Provider 提供短期签名密钥。如果企业使用自建 CI（如 Jenkins），需使用 `cosign generate-key-pair` 生成长期密钥对，并通过 KMS 管理私钥。

### 步骤二：消费方验证签名和 SBOM

**目标**：在部署前验证镜像签名和 SBOM 的真实性。

```bash
# 直接拉取镜像前验证
IMAGE="harbor.internal.example.com/cloud-sail/payment:v1.2.3"

# 1. 验证镜像的 cosign 签名
cosign verify \
  --certificate-identity "https://github.com/cloud-sail/payment/.github/workflows/build-secure.yml@refs/heads/main" \
  --certificate-oidc-issuer "https://token.actions.githubusercontent.com" \
  "$IMAGE"

# 输出（签名有效时）：
# Verification for harbor.internal.example.com/cloud-sail/payment@sha256:abc123... --
# The following checks were performed on each of these signatures:
#   - The cosign claims were validated
#   - Existence of the claims in the transparency log was verified offline
#   - The code-signing certificate was verified using trusted certificate authority certificates

# 2. 查看镜像附加的 SBOM
cosign download attestation "$IMAGE" | jq -r '.payload' | base64 -d | jq .

# 3. 用 Trivy 扫描同时验证 SBOM
trivy image --sbom-sources oci "$IMAGE"
# 输出中会显示 "SBOM source: oci", 表明 SBOM 是从 OCI Registry 拉取的
```

**在 K8s 中实现准入控制**（用 Kyverno 验证签名）：

```yaml
# kyverno-policy.yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signature
spec:
  validationFailureAction: Enforce
  rules:
  - name: check-cosign-signature
    match:
      any:
      - resources:
          kinds:
          - Pod
    verifyImages:
    - imageReferences:
      - "harbor.internal.example.com/*"
      attestors:
      - entries:
        - keyless:
            subject: "https://github.com/cloud-sail/*"
            issuer: "https://token.actions.githubusercontent.com"
            rekor:
              url: https://rekor.sigstore.dev
```

> **效果**：任何未签名的镜像或签名不匹配的镜像，都无法在 K8s 集群中部署。这是供应链安全的「最后一公里」。

### 步骤三：构建 provenance 的使用与验证

**目标**：生成并验证 SLSA Provenance，确保制品构建链路可追溯。

SLSA Provenance 是一个 JSON 文件，包含构建过程的完整信息。GitHub Actions 的 `slsa-github-generator` 会自动生成：

```json
{
  "_type": "https://in-toto.io/Statement/v1",
  "subject": [
    {
      "name": "harbor.internal.example.com/cloud-sail/payment",
      "digest": {
        "sha256": "abc123def456..."
      }
    }
  ],
  "predicateType": "https://slsa.dev/provenance/v1",
  "predicate": {
    "buildDefinition": {
      "buildType": "https://slsa.dev/container-based-build/v1",
      "resolvedDependencies": [
        {
          "uri": "git+https://github.com/cloud-sail/payment@refs/heads/main",
          "digest": {"sha1": "def789..."}
        }
      ]
    },
    "runDetails": {
      "builder": {
        "id": "https://github.com/slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@refs/tags/v1.10.0"
      },
      "buildConfig": {
        "command": ["docker", "build", "-t", "payment:latest", "."],
        "environment": {
          "platform": "ubuntu-latest",
          "os": "Ubuntu 22.04.3 LTS"
        }
      }
    }
  }
}
```


### 步骤四：SBOM 的长期存储与索引

**目标**：将所有 SBOM 集中存储，支持按组件反查「哪些镜像包含某个特定的依赖版本」。

```python
#!/usr/bin/env python3
"""
SBOM Indexer：将 CycloneDX SBOM 索引到 PostgreSQL，支持反向查询
"""

import json
import os
import sys

import psycopg2
from psycopg2.extras import execute_values

DATABASE_URL = os.environ["DATABASE_URL"]


def index_sbom(sbom_path, image_name):
    """将 SBOM 中的组件信息写入数据库"""
    with open(sbom_path) as f:
        sbom = json.load(f)

    conn = psycopg2.connect(DATABASE_URL)
    cur = conn.cursor()

    # 插入镜像记录
    cur.execute("""
        INSERT INTO sbom_images (image_name, sbom_format, sbom_json)
        VALUES (%s, %s, %s)
        ON CONFLICT (image_name) DO UPDATE SET sbom_json = EXCLUDED.sbom_json
        RETURNING id
    """, (image_name, sbom.get("bomFormat", "CycloneDX"), json.dumps(sbom)))

    image_id = cur.fetchone()[0]

    # 插入组件记录
    components = sbom.get("components", [])
    component_rows = [
        (image_id, c.get("name"), c.get("version"), c.get("purl", ""),
         c.get("type", ""), json.dumps(c.get("licenses", [])))
        for c in components
    ]

    execute_values(cur, """
        INSERT INTO sbom_components 
            (image_id, name, version, purl, type, licenses)
        VALUES %s
        ON CONFLICT (image_id, purl) DO NOTHING
    """, component_rows)

    conn.commit()
    cur.close()
    conn.close()

    print(f"Indexed {len(components)} components from {image_name}")


def query_affected_images(package_name, version_range=None):
    """查询哪些镜像包含指定包"""
    conn = psycopg2.connect(DATABASE_URL)
    cur = conn.cursor()

    cur.execute("""
        SELECT DISTINCT si.image_name, sc.version
        FROM sbom_components sc
        JOIN sbom_images si ON sc.image_id = si.id
        WHERE sc.name = %s
        ORDER BY si.image_name
    """, (package_name,))

    results = cur.fetchall()
    cur.close()
    conn.close()

    if results:
        print(f"Package '{package_name}' found in {len(results)} images:")
        for img, ver in results:
            print(f"  - {img} (version: {ver})")
    else:
        print(f"Package '{package_name}' not found in any image.")

    return results


if __name__ == "__main__":
    if sys.argv[1] == "index":
        index_sbom(sys.argv[2], sys.argv[3])
    elif sys.argv[1] == "query":
        query_affected_images(sys.argv[2])
```

**建表 SQL**：

```sql
CREATE TABLE IF NOT EXISTS sbom_images (
    id BIGSERIAL PRIMARY KEY,
    image_name VARCHAR(500) UNIQUE NOT NULL,
    sbom_format VARCHAR(50),
    sbom_json JSONB,
    indexed_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS sbom_components (
    id BIGSERIAL PRIMARY KEY,
    image_id BIGINT REFERENCES sbom_images(id),
    name VARCHAR(500) NOT NULL,
    version VARCHAR(200),
    purl VARCHAR(1000),
    type VARCHAR(50),
    licenses JSONB,
    UNIQUE(image_id, purl)
);

CREATE INDEX idx_sbom_comp_name ON sbom_components(name);
CREATE INDEX idx_sbom_comp_purl ON sbom_components(purl);
```

### 步骤五：结合 VEX 完成端到端验证

**目标**：部署前一次性验证：镜像签名 → SBOM 完整性 → VEX 过滤 → 漏洞扫描。

```bash
#!/bin/bash
# verify-and-deploy.sh
# 部署前的最后一道安全验证

IMAGE="$1"
if [ -z "$IMAGE" ]; then
    echo "Usage: $0 <image:tag>"
    exit 1
fi

VEX_FILE="/etc/cloud-sail/vex/${IMAGE//\//_}.vex.json"

echo "=== 供应链安全验证: $IMAGE ==="
echo ""

# 1. 验证镜像签名
echo "[1/5] Verifying image signature..."
if cosign verify "$IMAGE" 2>/dev/null; then
    echo "  ✓ Signature valid"
else
    echo "  ✗ Signature verification FAILED"
    exit 1
fi

# 2. 验证 SLSA Provenance
echo "[2/5] Verifying SLSA provenance..."
if cosign verify-attestation \
    --type https://slsa.dev/provenance/v1 \
    "$IMAGE" 2>/dev/null; then
    echo "  ✓ Provenance valid"
else
    echo "  ⚠ Provenance not found (non-blocking)"
fi

# 3. 验证 SBOM 签名
echo "[3/5] Verifying SBOM attestation..."
if cosign verify-attestation \
    --type https://cyclonedx.org/schema \
    "$IMAGE" 2>/dev/null; then
    echo "  ✓ SBOM verified"
else
    echo "  ✗ SBOM verification FAILED"
    exit 1
fi

# 4. VEX 过滤 + 漏洞扫描
echo "[4/5] Running Trivy scan (with VEX)..."
VEX_FLAG=""
if [ -f "$VEX_FILE" ]; then
    VEX_FLAG="--vex $VEX_FILE"
    echo "  Using VEX: $VEX_FILE"
fi

trivy image \
    --severity HIGH,CRITICAL \
    --sbom-sources oci \
    $VEX_FLAG \
    --format json \
    --output /tmp/scan-result.json \
    "$IMAGE"

CRITICAL_COUNT=$(jq '[.Results[]?.Vulnerabilities?[]? | select(.Severity=="CRITICAL")] | length' /tmp/scan-result.json)

if [ "$CRITICAL_COUNT" -gt 0 ]; then
    echo "  ✗ Found $CRITICAL_COUNT CRITICAL vulnerabilities"
    exit 1
else
    echo "  ✓ No CRITICAL vulnerabilities"
fi

# 5. 通过全部验证
echo "[5/5] All checks passed!"
echo "  ✓ Image is ready for deployment"
exit 0
```

### 测试验证

1. 推送一个带签名的测试镜像到 Harbor，运行 `cosign verify` 确认签名有效。
2. 用 `cosign download attestation` 下载 SBOM，验证其中的组件列表与镜像内容一致。
3. 修改 SBOM 文件后尝试用 `cosign attest` 覆盖原有签名——应该失败（签名不匹配）。
4. 运行 `verify-and-deploy.sh` 对测试镜像执行全链路验证，确认五道检查全部通过。
5. 查询「哪些镜像包含 `log4j-core`」，验证 SBOM 索引的反向查询能力。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| SBOM 自动化 | 零人工介入；每次构建自动生成 | SBOM 格式版本演进可能不兼容 |
| cosign 签名 | OIDC keyless 无需管理私钥 | keyless 依赖外部透明度日志（Rekor） |
| SLSA Provenance | 构建过程完全可追溯 | 需要 GitHub Actions 环境（自建 CI 适配成本高） |
| SBOM 索引 | 支持反向查询，0Day 响应快 | 存储和维护成本（随组件数量增长） |
| 准入控制 | 部署前强制验证，零信任落地 | 误拒绝可能阻塞业务发版 |

### 适用场景

1. **金融、政务、军工**：监管要求必须提供 SBOM 和签名验证的企业。
2. **开源项目维护者**：向用户证明「你下载的制品和我们的源码是一致的」。
3. **多云/多集群部署**：统一的签名验证策略防止跨云的供应链攻击。
4. **0Day 应急响应**：当新 CVE 爆发时，通过 SBOM 索引 1 分钟内定位受影响的所有镜像。
5. **并购/外包审计**：第三方代码的供应链透明度核查。

**不适用场景**：
1. 纯内部开发、无外部依赖的单体应用——SBOM 信息价值有限。
2. 构建流程极度简化的场景（如 `docker build` 一键部署）——引入全套供应链工具的复杂度可能高于收益。

### 注意事项

- **cosign keyless signing 依赖第三方**（Fulcio CA + Rekor 日志）。如果这些服务不可用，签名验证会失败。对可用性要求极高的场景，考虑用自建 PKI。
- **SBOM 不能替代漏洞扫描**。SBOM 只是告诉你「有什么」，不告诉你「有什么风险」。两者必须配合使用。
- **签名密钥的保护是最高优先级**。如果签名私钥泄漏，攻击者可以为恶意镜像伪造签名。考虑使用 HSM（硬件安全模块）或云 KMS 管理密钥。

### 常见踩坑经验

**踩坑案例 1：cosign 签名后镜像 tag 移动导致验证失败**
- **现象**：`cosign verify payment:latest` 报错 `no matching signatures`。
- **根因**：cosign 签名绑定的是镜像 digest（SHA256），不是 tag。当 `latest` 标签被重新指向另一个 digest 时，新 digest 没有签名。
- **解法**：在 CI 中使用 immutable tag（如 `v1.2.3` 或 git SHA），避免 `latest`；或每次 push 后自动签名。

**踩坑案例 2：SBOM 生成不完整**
- **现象**：90% 的依赖都正确，但缺少某些传递依赖。
- **根因**：`trivy image --format cyclonedx` 依赖于镜像内的包管理器元数据（如 `node_modules` 目录、maven 依赖树）。如果构建时清理了这些元数据，SBOM 就不完整。
- **解法**：不要在 Dockerfile 中清理包管理器缓存（或至少保留到 SBOM 生成后再清理）；考虑用 `syft` 直接从 filesystem 生成 SBOM。

**踩坑案例 3：Harbor 不支持 OCI Artifact 导致 attestation 丢失**
- **现象**：`cosign attest` 成功执行但 `cosign download attestation` 返回空。
- **根因**：Harbor v2.7 以下不完全支持 OCI Referrers API，导致 cosign 的 attestation 无法正确存储。
- **解法**：升级 Harbor 到 v2.8+；或使用 `cosign attach`（而非 attest）将 SBOM 作为普通 tag 推送到 Registry。

### 思考题

1. 假设你的公司使用 Jenkins（自建 CI/CD），无法使用 GitHub Actions 的 OIDC keyless signing。请设计一套不使用 keyless 的签名方案：如何生成、存储、轮换签名密钥，并确保 Jenkins Pipeline 中安全使用私钥。
2. SLSA Level 3 要求「构建过程经过隔离和审查」。在你的 CI/CD 环境中，如何验证「隔离构建」的真正生效？请设计一个验证脚本，检查构建环境是否满足 SLSA Level 3 的要求。
