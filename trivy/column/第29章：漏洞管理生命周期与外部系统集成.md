# 第29章：漏洞管理生命周期与外部系统集成

> 版本：Trivy v0.50+
> 面向人群：安全工程师、DevOps、PM

---

## 1. 项目背景

### 业务场景

云帆科技的安全扫描体系已经相当成熟——CI/CD 自动扫描、Harbor 镜像扫描、Trivy Operator 持续监控、Prometheus 告警自动化。但安全负责人老李发现了一个最根本的问题没有解决。

「我们扫描了、发现了、告警了，但是——谁来修？」

这个问题在一次安全事件中暴露得非常尖锐。某个金融客户要求在 7 个工作日内修复生产环境的所有 CRITICAL 漏洞。团队加班加点修了两天，但事后复盘时发现：有 3 个 CRITICAL 漏洞已经在扫描报告中存在了 40 多天，但一直没有被真正修复。原因令人尴尬：

- 漏洞 A 被扫描器发现后，安全团队给开发发了封邮件，但邮件进了开发经理的垃圾箱。
- 漏洞 B 已经在 Issue 里跟踪了，但负责的开发离职了，工单没有重新指派，变成「孤儿工单」。
- 漏洞 C 开发说「修了」，但没人重新扫描验证，实际根本没修好。

「发现问题只是 10%，修复才是 90%。但我们把 90% 的精力都花在了发现上。」老李意识到：**漏洞管理不是一次性扫描，而是一个生命周期闭环。**

CTO 要求：「在下一个季度结束前，建立从『扫描发现』到『修复验证』的完整闭环流程。每条 CRITICAL 漏洞都要有唯一的追踪 ID、明确的责任人、可量化的修复 SLA 和自动化的验证机制。」

### 痛点放大

**第一，发现与修复脱节。** 扫描结果停留在「报告」中——JSON 文件、Trivy Table 输出、Grafana 面板。这些信息进入不了团队的实际工作流（JIRA / Asana / Linear），开发不知道「我被分配了一个漏洞修复任务」。

**第二，漏洞生命周期无追踪。** 一条漏洞经历了「新发现 → 已确认 → 已指派 → 修复中 → 待验证 → 已验证 → 已关闭」的状态流转，但现实是「扫到了 → 开发说知道了 → 忘了」。没有标准化的状态机，就没有可被管理的流程。

**第三，重复造轮子。** 不同的项目、不同的团队、不同的扫描结果，手动汇总到 Excel 里跟踪，安全团队每月花 40+ 小时做「漏洞台账」的手工维护。这种重复劳动无意义且容易出错。

**第四，修复验证依赖人工。** 开发说「修好了」，安全团队需要手动重新执行扫描来验证——这又是一个瓶颈。如果每个漏洞都走人工验证，100 个漏洞就需要 100 次手动扫描。

**第五，SLA 缺乏约束力。** CRITICAL 漏洞应该 7 天修完还是 30 天？没人定义、没人追踪、没人考核。结果就是「重要但不够紧急」——永远排在业务需求后面。

**本章的核心目标是：将 Trivy 扫描结果无缝接入 DefectDojo、JIRA 等企业工具链，建立「发现 → 评估 → 指派 → 修复 → 验证 → 关闭」的漏洞管理闭环，实现安全风险的端到端治理。**

## 2. 项目设计

**场景**：云帆科技的漏洞管理流程设计会，老李（安全负责人）、小胖（开发代表）、大师正在讨论如何闭环。

---

**小胖**：「我们不是有 JIRA 吗？扫出漏洞就手动创建 JIRA Issue，指派给开发不就行了？」

**小白**：「『手动创建 JIRA Issue』——这就是瓶颈。一个镜像可能扫出 50 个漏洞，50 个 JIRA Issue 你手动创建？然后下一周你又扫一次，同样的镜像又有 50 个漏洞——你怎么判断哪些是新的、哪些是旧的、哪些已经修好了？」

**大师**：「技术映射：漏洞管理就像医院的『电子病历系统』。病人每次就诊（每次扫描），医生会记录新的症状（新漏洞），但也会参考历史记录（历史漏洞）。如果有医生手工在纸上写病历（手动创建 JIRA），效率极低还不容易查找历史。而电子病历系统可以自动关联——你挂一次号，系统就知道你以前有什么病史、上次吃了什么药、这次和上次有什么变化。」

**小胖**：「那我们缺的『电子病历系统』是什么？」

**小白**：「DefectDojo——专门做漏洞管理的开源平台。它的核心能力：

1. **自动导入**：支持导入 Trivy JSON / SARIF 报告，自动解析漏洞。
2. **智能去重**：同一个镜像的不同扫描，DefectDojo 能自动识别「这是同一个漏洞的新一次扫描」，不会重复创建。
3. **状态流转**：每个漏洞有完整的生命周期状态（Active → Verified → Mitigated → False Positive → Out of Scope → Risk Accepted）。
4. **SLA 管理**：每种严重级别设置修复截止日期，超时自动提醒。
5. **增量对比**：比较两次扫描，只显示新增/修复的漏洞——精确展示趋势。」

**老李**：「那我们怎么和 JIRA 对接？」

**大师**：**「DefectDojo → JIRA 的双向同步，这是闭环的关键。** 流程是这样的：

```
Trivy 扫描
    │
    ▼
DefectDojo（自动导入）    ←── 漏洞去重、状态管理
    │
    ▼
JIRA（自动创建工单）       ←── 按严重级别分派不同项目/负责人
    │
    ▼
开发修复代码
    │
    ▼
CI/CD 自动重新扫描
    │
    ▼
DefectDojo 发现漏洞消失   ←── 自动标记为 Mitigated
    │
    ▼
JIRA 工单自动关闭          ←── 通知开发「验证通过」
```

每条漏洞在 JIRA 中有唯一 ID（如 SEC-1234），在 DefectDojo 中有唯一 Finding ID。两者的状态通过 Webhook 同步——开发在 JIRA 中标记为 Done，DefectDojo 收到通知后等待下一次扫描验证。」

**小胖**：「等扫描验证？那不是又要人工去触发扫描？」

**小白**：「不需要。如果你的 CI/CD 流水线每次构建后都自动推送到 DefectDojo（我们第 17 章已经做了），那这个验证是完全自动的。开发修完代码、提交 PR、合并到主分支、CI 自动构建并扫描、DefectDojo 自动消费新报告、发现 CRITICAL 消失了 → JIRA 自动关闭。全链路无需人工触发。」

**大师**：「我们还需要定义 SLA。我建议：

| 严重级别 | 修复 SLA | JIRA 优先级 | 超时动作 |
|----------|----------|-------------|----------|
| CRITICAL | 7 天 | Highest | 通知 CTO |
| HIGH | 30 天 | High | 通知部门经理 |
| MEDIUM | 90 天 | Medium | 通知团队 Lead |
| LOW | 180 天 | Low | 纳入季度规划 |

这些 SLA 在 DefectDojo 的 Product 级别配置，超时后会自动标记并触发通知。」

### 步骤二：配置 DefectDojo Product 和 Engagement

**目标**：在 DefectDojo 中创建镜像对应的 Product 和 Engagement，建立扫描报告的接收容器。

```python
#!/usr/bin/env python3
"""
setup-dojo.py：通过 DefectDojo API 创建 Product 和 Engagement
"""

import os
import requests

DOJO_URL = os.environ["DOJO_URL"]     # e.g. http://defectdojo:8080
DOJO_TOKEN = os.environ["DOJO_TOKEN"]
HEADERS = {"Authorization": f"Token {DOJO_TOKEN}", "Content-Type": "application/json"}


def create_product(name, description, prod_type=1):
    """创建 Product（如一个微服务）"""
    resp = requests.post(f"{DOJO_URL}/api/v2/products/", headers=HEADERS, json={
        "name": name,
        "description": description,
        "prod_type": prod_type,  # 1 = Research and Development
        "enable_simple_risk_acceptance": True,
        "enable_full_risk_acceptance": True,
        "sla_configuration": 1,  # 使用默认 SLA 配置
    })
    if resp.status_code == 201:
        print(f"Created product: {name}")
        return resp.json()["id"]
    elif resp.status_code == 400 and "already exists" in resp.text:
        # 获取已存在的 ID
        existing = requests.get(
            f"{DOJO_URL}/api/v2/products/?name={name}", headers=HEADERS
        ).json()
        if existing.get("results"):
            return existing["results"][0]["id"]
    print(f"Failed to create product: {resp.text}")
    return None


def create_engagement(product_id, name, target_start, target_end):
    """创建 Engagement（如一次版本扫描周期）"""
    resp = requests.post(f"{DOJO_URL}/api/v2/engagements/", headers=HEADERS, json={
        "product": product_id,
        "name": name,
        "target_start": target_start,  # e.g. "2024-01-01"
        "target_end": target_end,      # e.g. "2024-12-31"
        "status": "In Progress",
        "engagement_type": "CI/CD",
        "deduplication_on_engagement": True,  # 开启去重
    })
    if resp.status_code == 201:
        print(f"Created engagement: {name}")
        return resp.json()["id"]
    print(f"Failed to create engagement: {resp.text}")
    return None


if __name__ == "__main__":
    # 为每个微服务创建 Product
    services = [
        ("cloud-sail/payment-gateway", "支付网关"),
        ("cloud-sail/user-service", "用户服务"),
        ("cloud-sail/order-service", "订单服务"),
    ]

    for name, desc in services:
        product_id = create_product(name, desc)
        if product_id:
            create_engagement(
                product_id,
                f"{name} - Q1 2024 Scan",
                "2024-01-01",
                "2024-03-31",
            )
```

### 步骤三：自动导入 Trivy 扫描结果到 DefectDojo

**目标**：在 CI/CD 扫描完成后，自动将 Trivy JSON 报告推送到 DefectDojo。

```bash
#!/bin/bash
# import-to-dojo.sh
# 在 CI/CD 扫描步骤后调用

DOJO_URL="${DOJO_URL:-http://defectdojo:8080}"
DOJO_TOKEN="${DOJO_TOKEN}"
SCAN_REPORT="${1:-trivy-scan.json}"
IMAGE_NAME="${2:-unknown-image}"
SCAN_DATE=$(date -Iseconds)

# 先找到对应 Product 的 Engagement ID
# 这里简化处理：假设 engagement_id 已知或通过 API 查询
ENGAGEMENT_ID=$(curl -s -H "Authorization: Token ${DOJO_TOKEN}" \
  "${DOJO_URL}/api/v2/engagements/?name=${IMAGE_NAME}" | \
  jq -r '.results[0].id // empty')

if [ -z "$ENGAGEMENT_ID" ]; then
    echo "Error: No engagement found for $IMAGE_NAME. Run setup-dojo.py first."
    exit 1
fi

# 导入扫描结果
# DefectDojo 支持 Trivy JSON 格式直接导入
curl -X POST "${DOJO_URL}/api/v2/import-scan/" \
  -H "Authorization: Token ${DOJO_TOKEN}" \
  -F "engagement=${ENGAGEMENT_ID}" \
  -F "scan_type=Trivy Scan" \
  -F "file=@${SCAN_REPORT}" \
  -F "scan_date=${SCAN_DATE}" \
  -F "active=true" \
  -F "verified=false" \
  -F "close_old_findings=true" \
  -F "skip_duplicates=true" \
  -F "minimum_severity=Medium" \
  | jq .

echo "Import completed. Check: ${DOJO_URL}/engagement/${ENGAGEMENT_ID}"
```

> **重要参数说明**：
> - `close_old_findings=true`：如果上次扫描存在的漏洞在本次扫描中消失了，自动标记为 "Mitigated"。这是自动闭环的关键。
> - `skip_duplicates=true`：同一个 endpoint + CVE 的扫描结果只保留一条，避免重复工单。
> - `minimum_severity=Medium`：只导入 MEDIUM 及以上的漏洞，减少噪音。

### 步骤四：配置 DefectDojo → JIRA 双向同步

**目标**：当 DefectDojo 创建新 Finding 时，自动在 JIRA 创建工单；当 Finding 被 Mitigated 时，自动关闭 JIRA 工单。

**DefectDojo 端配置**（管理员 → System Settings）：


**配置 JIRA Webhook（JIRA → DefectDojo）**：

```python
#!/usr/bin/env python3
"""
dojo-jira-webhook.py
监听 JIRA Issue 状态变更，同步到 DefectDojo
"""

from flask import Flask, request, jsonify
import requests
import os

app = Flask(__name__)

DOJO_URL = os.environ["DOJO_URL"]
DOJO_TOKEN = os.environ["DOJO_TOKEN"]
HEADERS = {"Authorization": f"Token {DOJO_TOKEN}"}


@app.route("/jira-webhook", methods=["POST"])
def jira_webhook():
    """接收 JIRA 的 Webhook 通知"""
    data = request.json

    issue_key = data.get("issue", {}).get("key", "")
    changelog = data.get("changelog", {}).get("items", [])

    # 查找 JIRA Issue ID → DefectDojo Finding ID 的映射
    finding_id = get_finding_by_jira(issue_key)
    if not finding_id:
        return jsonify({"status": "ok", "note": "No finding matched"})

    # 处理状态变更
    for change in changelog:
        field = change.get("field", "")
        if field == "status":
            new_status = change.get("toString", "")

            if new_status in ["Done", "Resolved"]:
                # 标记为待验证（等待下次扫描确认）
                update_finding(finding_id, {"active": True, "verified": False})
                note = f"JIRA {issue_key} marked as {new_status}. Awaiting re-scan verification."
                add_finding_note(finding_id, note)

            elif new_status in ["Won't Do", "Cancelled"]:
                # 标记为 False Positive 或 Risk Accepted
                update_finding(finding_id, {"active": False, "risk_accepted": True})
                note = f"JIRA {issue_key} marked as {new_status}. Marking as Risk Accepted."
                add_finding_note(finding_id, note)

    return jsonify({"status": "ok"})


def get_finding_by_jira(jira_key):
    """通过 JIRA key 查找 DefectDojo Finding"""
    # Finding 的 jira_issue 字段存储了 JIRA key
    resp = requests.get(
        f"{DOJO_URL}/api/v2/findings/?jira_issue={jira_key}",
        headers=HEADERS,
    )
    results = resp.json().get("results", [])
    return results[0]["id"] if results else None


def update_finding(finding_id, data):
    resp = requests.patch(
        f"{DOJO_URL}/api/v2/findings/{finding_id}/",
        headers=HEADERS,
        json=data,
    )
    return resp.status_code


def add_finding_note(finding_id, note_text):
    resp = requests.post(
        f"{DOJO_URL}/api/v2/findings/{finding_id}/notes/",
        headers=HEADERS,
        json={"entry": note_text},
    )
    return resp.status_code


if __name__ == "__main__":
    app.run(host="0.0.0.0", port=8000)
```

### 步骤五：漏洞 SLA 追踪与超时告警

**目标**：自动监控漏洞修复 SLA，超时自动升级通知。

```python
#!/usr/bin/env python3
"""
sla-monitor.py
每天检查 DefectDojo 中超过 SLA 的 Finding，自动升级
"""

import os
import json
from datetime import datetime, timedelta

import requests

DOJO_URL = os.environ["DOJO_URL"]
DOJO_TOKEN = os.environ["DOJO_TOKEN"]
HEADERS = {"Authorization": f"Token {DOJO_TOKEN}"}

SLA_HOURS = {
    "Critical": 168,    # 7 天
    "High": 720,        # 30 天
    "Medium": 2160,     # 90 天
    "Low": 4320,        # 180 天
}


def check_sla_violations():
    """检查所有超过 SLA 的 Active Finding"""
    resp = requests.get(
        f"{DOJO_URL}/api/v2/findings/?active=true&limit=500",
        headers=HEADERS,
    )
    findings = resp.json().get("results", [])

    violations = []
    now = datetime.now()

    for finding in findings:
        severity = finding.get("severity", "")
        created_str = finding.get("created", "")

        if not created_str or severity not in SLA_HOURS:
            continue

        created = datetime.fromisoformat(created_str.replace("Z", "+00:00"))
        elapsed_hours = (now - created.replace(tzinfo=None)).total_seconds() / 3600

        sla_limit = SLA_HOURS[severity]

        if elapsed_hours > sla_limit:
            violations.append({
                "finding_id": finding["id"],
                "title": finding["title"],
                "severity": severity,
                "created": created_str,
                "elapsed_hours": round(elapsed_hours, 1),
                "sla_hours": sla_limit,
                "sla_breached_hours": round(elapsed_hours - sla_limit, 1),
                "jira_issue": finding.get("jira_issue", ""),
            })

    return violations


def escalate_violations(violations):
    """对 SLA 违规进行升级处理"""
    print(f"SLA Violations found: {len(violations)}")

    for v in violations:
        severity = v["severity"]
        breached = v["sla_breached_hours"]

        # 升级逻辑
        if breached > 168:  # 超时超过 1 周
            level = "CTO"
            action = f"Notify CTO about finding {v['finding_id']}"
        elif breached > 72:  # 超时超过 3 天
            level = "Manager"
            action = f"Notify manager about finding {v['finding_id']}"
        else:
            level = "Reminder"
            action = f"Send reminder about finding {v['finding_id']}"

        print(f"  [{severity}] {v['title']} - "
              f"Elapsed: {v['elapsed_hours']}h | "
              f"Breached: {breached}h | "
              f"Escalation: {level}")

        # 在 DefectDojo 中添加 note
        requests.post(
            f"{DOJO_URL}/api/v2/findings/{v['finding_id']}/notes/",
            headers=HEADERS,
            json={
                "entry": f"[SLA ALERT] Severity={severity}, "
                         f"Elapsed={v['elapsed_hours']}h, "
                         f"SLABreach={breached}h. "
                         f"Escalation: {level}. {action}"
            },
        )


if __name__ == "__main__":
    violations = check_sla_violations()

    if violations:
        escalate_violations(violations)
        # 写入状态文件供 Prometheus / Alertmanager 抓取
        sla_count = len(violations)
        print(f"\nTotal SLA violations: {sla_count}")

        with open("/var/metrics/trivy_sla.prom", "w") as f:
            f.write(f"trivy_sla_violations {sla_count}\n")
    else:
        print("No SLA violations. All findings within compliance window.")
```

> **坑点**：DefectDojo 的 `created` 字段是 Finding 的创建时间，不是漏洞首次被发现的时间。如果扫描报告多次导入（不同版本），DefectDojo 去重后 Finding 的 `created` 可能不会更新。建议 SLA 以 `found_date` 或自定义字段为准。

### 步骤六：漏洞趋势报告（给管理层）

**目标**：自动生成漏洞趋势报表，展示「新增 vs 修复」的动态。

```python
#!/usr/bin/env python3
"""
vuln-trend-report.py
从 DefectDojo API 获取漏洞趋势数据，生成 Markdown 报告
"""

import requests
from datetime import datetime, timedelta
from collections import defaultdict

DOJO_URL = "http://defectdojo:8080"
TOKEN = "your-token"
HEADERS = {"Authorization": f"Token {TOKEN}"}


def get_trend(days=90):
    """获取最近 N 天的漏洞趋势"""
    findings = []

    # 分页获取所有 Finding
    url = f"{DOJO_URL}/api/v2/findings/?limit=500"
    while url:
        resp = requests.get(url, headers=HEADERS).json()
        findings.extend(resp.get("results", []))
        url = resp.get("next")

    # 按日期和严重级别分组
    trend = defaultdict(lambda: defaultdict(int))
    now = datetime.now()

    for f in findings:
        created = datetime.fromisoformat(
            f.get("created", "").replace("Z", "+00:00")
        )
        if (now - created.replace(tzinfo=None)).days > days:
            continue

        date_key = created.strftime("%Y-%m-%d")
        severity = f.get("severity", "Info")
        status = f.get("active", True)

        if status:
            trend[date_key][severity] += 1
        else:
            trend[date_key][f"mitigated_{severity}"] += 1

    return trend


def generate_report(trend):
    """生成 Markdown 报告"""
    report = f"""# 漏洞趋势报告

**生成时间**: {datetime.now().strftime('%Y-%m-%d %H:%M:%S')}

## 最近 90 天漏洞发现趋势

| 日期 | CRITICAL 新增 | HIGH 新增 | CRITICAL 修复 | HIGH 修复 | 净增长 |
|------|--------------|-----------|--------------|-----------|--------|
"""

    for date_key in sorted(trend.keys()):
        d = trend[date_key]
        critical_new = d.get("Critical", 0)
        high_new = d.get("High", 0)
        critical_fix = d.get("mitigated_Critical", 0)
        high_fix = d.get("mitigated_High", 0)
        net = (critical_new + high_new) - (critical_fix + high_fix)

        report += f"| {date_key} | {critical_new} | {high_new} | {critical_fix} | {high_fix} | {'+' + str(net) if net >= 0 else str(net)} |\n"

    return report


if __name__ == "__main__":
    trend = get_trend(90)
    report = generate_report(trend)

    with open("vulnerability-trend-report.md", "w") as f:
        f.write(report)

    print("Report saved to vulnerability-trend-report.md")
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| DefectDojo | 漏洞去重、状态管理、SLA 追踪 | 部署和配置较复杂；API 文档不完善 |
| JIRA 集成 | 将漏洞纳入开发标准工作流 | 双向同步的断路可能导致数据不一致 |
| 自动化导入 | 零人工介入的漏洞入库 | 误报/重复依赖正确的去重配置 |
| SLA 监控 | 量化安全团队的工作效率 | SLA 需要各团队共识支持 |
| 漏洞趋势 | 管理层可见的安全改进进度 | 需要至少 1-2 个月的数据积累 |

### 适用场景

1. **中大型企业（50+ 开发）**：漏洞数量多到无法手工管理，必须工具化。
2. **有 JIRA 等工单系统的团队**：DefectDojo 与 JIRA 的双向同步是核心价值。
3. **安全合规审计**：NIST SP 800-53、ISO 27001 都要求漏洞管理的书面记录和追踪。
4. **CISO 需要量化安全运营**：SLA 达标率、MTTR、漏洞趋势等 KPI 可以直接从 DefectDojo 提取。
5. **多项目多团队共用安全基础设施**：统一的漏洞管理平台避免信息孤岛。

**不适用场景**：
1. 团队 < 10 人且漏洞 < 20 个——DefectDojo 的管理成本高于 Excel。
2. 不需要 JIRA/工单系统跟踪的场景——直接用 `.trivyignore` + 定时 Cron 重扫更简单。

### 注意事项

- **DefectDojo 的去重配置**。正确配置 `deduplication_on_engagement=True` 和合适的去重算法（hash_code / unique_id_from_tool），否则同一漏洞会重复创建。
- **JIRA 工单自动关闭需谨慎**。某些团队可能希望在 JIRA 中手动确认关闭，而不是自动关闭。提供配置开关。
- **SLA 定义要务实**。CRITICAL 7 天看起来很美好，但如果平均修复速度是 14 天，7 天 SLA 只会产生大量的「虚假违规」。
- **DefectDojo 的生产部署**。Docker Compose 适合 POC，生产环境推荐 K8s + PostgreSQL + Redis + Celery Worker 的完整部署。

### 常见踩坑经验

**踩坑案例 1：DefectDojo 的 `close_old_findings` 误关闭 Finding**
- **现象**：部分 Finding 被错误标记为 Mitigated。
- **根因**：导入了不同工具的扫描结果（如 Trivy + Grype），DefectDojo 将不在同一份报告中的 Finding 当作「已修复」。
- **解法**：为每个扫描工具创建独立的 Engagement；或将去重算法调整为 `unique_id_from_tool`，确保不同工具生成不同的 ID。

**踩坑案例 2：JIRA 双向同步导致无限循环**
- **现象**：DefectDojo 通知 JIRA → JIRA Webhook 回通知 DefectDojo → DefectDojo 再通知 JIRA……死循环。
- **根因**：JIRA 的 Webhook 触发条件配置不当（如所有 issue 变更都触发）。
- **解法**：在 JIRA Webhook 中只监听特定事件（如 `jira:issue_updated` 的 status 变更）；在 Webhook 处理器中检查变更来源，避免 DefectDojo 触发的变更再次回传。

**踩坑案例 3：SLA 监控脚本的时区问题**
- **现象**：SLA 违规报警的时间与实际不符（如 UTC vs Asia/Shanghai）。
- **根因**：DefectDojo 的 `created` 字段是 UTC 时间，但本地时区是 UTC+8，导致 SLA 计算偏了 8 小时。
- **解法**：在 SLA 计算中统一使用 UTC；在报警信息中自动转换为本地时区显示。

### 思考题

1. 如果你的公司不使用 JIRA（使用 Linear / Asana / Notion 等其他工具），如何设计与 DefectDojo 的集成？抽象出一个「Issue Tracker 适配层」的接口设计是怎样的？
2. 假设你的 DefectDojo 中积累了一万条 Finding，其中 30% 是误报（False Positive）但尚未标记。请设计一个自动化脚本，通过分析 Finding 的描述、CVE 信息、影响范围，自动推荐哪些 Finding 可能是误报，供安全团队批量审批。
