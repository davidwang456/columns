# 第8章：报告输出过滤与基线管理

> 版本：Trivy v0.50+
> 面向人群：开发、运维、安全工程师、项目经理

---

## 1. 项目背景

### 业务场景

云帆科技的 Trivy 扫描全面铺开后，安全团队的邮箱被报告淹没了。每天早上，小李打开邮箱都能看到几十封来自不同 CI 流水线的扫描报告——有的以 JSON 附件形式发送，有的以 Markdown 贴在 GitLab Merge Request 评论里，还有的以纯文本形式打印在 Jenkins 控制台日志中。

最混乱的一次，项目经理老王拿着三份报告来找安全负责人：「同一个镜像，开发给的报告里有 8 个 Critical，测试给的报告里有 5 个 Critical，运维给的报告里有 12 个 Critical。到底信谁？这个版本能不能发？」

安全负责人仔细一看，三份报告的差异原因五花八门：

- 开发的报告用了 `--severity HIGH,CRITICAL`，测试的报告用了默认的 `UNKNOWN,LOW,MEDIUM,HIGH,CRITICAL`。
- 运维的报告加了 `--ignore-unfixed`，过滤掉了厂商尚未发布补丁的漏洞。
- 开发的报告是三天前跑的，数据库版本比测试的旧，少了两个新入库的 CVE。
- 最离谱的是，有一份报告把 `npm audit` 的结果和 Trivy 的结果混在一起，表格格式完全不对。

与此同时，开发组的阿杰也满腹怨言。他的项目扫描稳定报出 15 个 Medium 级别的漏洞，全是某个深层依赖的 issue， upstream 已经确认不会修复，团队也评估过不影响业务。但每次发版前，这 15 个漏洞都会出现在报告中，安全评审会议上都要重新解释一遍——「这是已知接受的风险」。

### 痛点放大

**第一，报告格式不统一，沟通成本爆炸。** 不同人用不同的参数、不同的格式、不同的数据库版本扫描同一个目标，输出的报告像「罗生门」一样互相矛盾。管理层看不懂 JSON，开发不想看表格，安全团队需要将多种格式转换成统一的视图。

**第二，噪音淹没真正重要的风险。** 一个中型项目的扫描报告可能包含 200 多个漏洞，其中 80% 是已修复的低危漏洞、开发依赖漏洞、或者已评估接受的已知风险。开发人员面对满屏的红色标记，直接选择「全部忽略」，反而可能漏掉真正需要立即处理的 Critical 漏洞。

**第三，基线管理缺乏工程化手段。** 团队的漏洞接受策略停留在「口头约定」或「Excel 表格」阶段。某个 CVE 被评估为「可接受」，但三个月后新人接手项目，重新提问「这个漏洞为什么不修？」知识没有沉淀，同样的讨论反复发生。

**第四，报告无法直接对接企业流程。** JSON 报告虽然结构化，但安全团队需要把它导入 JIRA、DefectDojo、或者企业微信通知。每次都要写临时的 Python 脚本做格式转换，脚本东一个西一个，维护困难。

**本章的核心目标是：建立统一的报告输出规范、掌握多级过滤策略、用 `.trivyignore` 和 Go Template 实现工程化的基线管理与报告定制。**

## 2. 项目设计

**场景**：云帆科技的周会上，项目经理老王、开发阿杰（小胖）和安全工程师小白正在争论扫描报告的规范问题。

---

**小胖**：（把三份报告拍在桌上）「老王，这三份报告都是扫同一个镜像，漏洞数量都不一样。你让我怎么给你准确的发布时间？」

**老王**：「所以我才要问你们啊！到底哪个是对的？」

**小白**：「三个都是对的，只是参数不同。开发的报告只看了 HIGH 和 CRITICAL，测试的报告看了所有级别，运维的报告过滤了没修复的。就像三个人用不同筛子过滤同一堆沙子，筛出来的数量当然不一样。」

**大师**：「技术映射：报告差异的本质是『筛选器』不同。Trivy 的默认行为是展示所有漏洞，但不同角色需要看不同视角。开发只需要关心『我现在能修什么』，运维关心『哪些会影响生产稳定性』，管理层关心『有没有 Critical 风险会阻碍发版』。我们需要为每个角色定义统一的『筛子规格』。」

**小胖**：「那能不能直接让 Trivy 只输出我想看的？」

**小白**：「可以。Trivy 提供了多层过滤机制：

1. **Severity 过滤**：`--severity HIGH,CRITICAL` 只展示高危和严重漏洞。
2. **状态过滤**：`--ignore-unfixed` 过滤掉厂商尚未发布补丁的漏洞，避免『无法修复』的噪音。
3. **类型过滤**：`--vuln-type os` 只看 OS 包漏洞，`--vuln-type library` 只看语言包漏洞。
4. **包名过滤**：`--ignored-licenses` 或 Rego 策略可以排除特定的包或 CVE。
5. **基线过滤**：`.trivyignore` 文件可以把已评估接受的漏洞永久排除。」

**老王**：「那基线管理怎么做？我们不能每次开会都重新讨论一遍『这 15 个 Medium 修不修』。」

**大师**：`.trivyignore` 就是答案。它不只是『忽略列表』，更是『团队共识的书面化』。每一条忽略规则都应该包含：

- **CVE ID**：明确标识哪个漏洞。
- **忽略原因**：为什么可以接受（如「开发依赖，不进入生产」）。
- **到期时间**：什么时候重新评估（防止永久忽略）。
- **审批人**：谁做了这个决定。

比如：
```
# CSAT-2024-003: lodash <4.17.21 的 prototype pollution
# 原因：仅用于测试框架，不进入生产镜像
# 到期：2024-12-31
# 审批：security@cloud-sail.com
CVE-2021-23337
```

**小胖**：「听起来像给漏洞开『请假条』。但谁保证这些请假条不会泛滥成灾？」

**小白**：「所以需要审计。我建议每月跑一次脚本，检查 `.trivyignore` 中快到期的条目，重新评估。对于没有到期时间或没有原因的条目，CI 直接拒绝通过。」

**老王**：「还有一个问题。你们给的报告格式五花八门的，我看不懂 JSON，能不能给我一份像 Excel 一样的表格，只显示关键字段？」

**大师**：「Trivy 支持 Go Template 自定义报告。你可以定义一个模板，只输出：镜像名、CVE ID、包名、当前版本、修复版本、严重级别。然后用这个模板生成 Markdown 或 HTML 表格，直接贴到邮件或 Wiki 里。」

**小胖**：「那如果我想把报告自动发到企业微信呢？」

**小白**：「JSON 格式天然适合程序处理。写一个简单的 Python 脚本读取 Trivy 的 JSON 输出，提取 Critical 漏洞，调用企业微信的 Webhook API 推送。或者更高级一点，用 Trivy 的 SARIF 输出，直接导入 GitHub/GitLab 的安全面板。」

**大师**：「总结今天的讨论：

1. **统一参数**：团队发布一份《Trivy 扫描规范》，规定不同场景的 severity、ignore-unfixed、format 标准。
2. **基线即代码**：`.trivyignore` 提交到 Git 仓库，接受代码审查，定期审计。
3. **分层报告**：开发用 table 看细节，管理层用 custom template 看摘要，CI 用 JSON 做自动化。」

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+，已安装并配置好数据库
- **测试镜像**：`python:3.4-alpine`（已知有多个漏洞，便于演示过滤效果）
- **工具**：`jq`（JSON 处理）、文本编辑器

### 步骤一：理解报告格式的差异与适用场景

**目标**：对比 Trivy 支持的各种输出格式，为不同场景选择最合适的格式。

```bash
# Table 格式（默认，适合人类阅读）
trivy image --format table python:3.4-alpine

# JSON 格式（适合程序处理）
trivy image --format json --output report.json python:3.4-alpine

# SARIF 格式（适合导入 GitHub/GitLab Security Dashboard）
trivy image --format sarif --output report.sarif python:3.4-alpine

# CycloneDX 格式（SBOM 标准）
trivy image --format cyclonedx --output sbom.xml python:3.4-alpine

# SPDX 格式（另一种 SBOM 标准）
trivy image --format spdx-json --output sbom.spdx.json python:3.4-alpine
```

**格式对比表**：

| 格式 | 适用场景 | 人类可读 | 机器可读 | 包含修复建议 |
|------|----------|----------|----------|-------------|
| table | 终端快速查看、邮件正文 | 高 | 低 | 是 |
| json | CI 自动化、脚本解析、数据库存储 | 中 | 高 | 是 |
| sarif | GitHub/GitLab Security 面板集成 | 低 | 高 | 部分 |
| cyclonedx | SBOM 生成、供应链审计 | 低 | 高 | 否 |
| spdx | 开源合规、法律审计 | 低 | 高 | 否 |
| template | 企业自定义报告、管理层摘要 | 高 | 中 | 自定义 |

### 步骤二：多层过滤策略实战

**目标**：掌握 severity、状态、类型三层过滤，输出精准报告。

```bash
# 基础扫描：显示所有漏洞
trivy image python:3.4-alpine

# 第一层过滤：只看 HIGH 和 CRITICAL
trivy image --severity HIGH,CRITICAL python:3.4-alpine

# 第二层过滤：只看已有修复方案的漏洞
trivy image --severity HIGH,CRITICAL --ignore-unfixed python:3.4-alpine

# 第三层过滤：只看 OS 包漏洞（排除语言包噪音）
trivy image --severity HIGH,CRITICAL --ignore-unfixed --vuln-type os python:3.4-alpine

# 组合效果统计（用 jq 提取数量）
trivy image --format json python:3.4-alpine | jq '[.Results[]?.Vulnerabilities?[]?] | length'
trivy image --severity HIGH,CRITICAL --ignore-unfixed --format json python:3.4-alpine | jq '[.Results[]?.Vulnerabilities?[]?] | length'
```

**预期效果**：
- 无过滤时：漏洞总数可能超过 50 个。
- 加 `--severity HIGH,CRITICAL`：降至约 10-15 个。
- 再加 `--ignore-unfixed`：降至约 5-8 个（只保留有补丁的）。
- 再加 `--vuln-type os`：可能只剩 3-5 个。

> **可能遇到的坑**：`--ignore-unfixed` 会过滤掉状态为 `unfixed` 的漏洞。在某些场景下（如需要向客户证明「所有已知漏洞都已评估」），你可能需要保留 unfixed 漏洞，但在报告中单独标注。

### 步骤三：自定义 Go Template 报告

**目标**：为管理层生成只包含关键字段的摘要报告。

创建模板文件 `report-template.tpl`：

```
{{- /* 管理层摘要模板 */ -}}
# 安全扫描摘要

**扫描目标**：{{ .ArtifactName }}  
**扫描时间**：{{ now }}  
**严重漏洞统计**：

| 级别 | 数量 |
|------|------|
| CRITICAL | {{ len (where .Results "Vulnerabilities" "severity" "CRITICAL") }} |
| HIGH | {{ len (where .Results "Vulnerabilities" "severity" "HIGH") }} |

**需立即处理的漏洞（CRITICAL & HIGH & fixed）**：

| 包名 | CVE | 当前版本 | 修复版本 | 级别 |
|------|-----|----------|----------|------|
{{- range .Results }}
{{- range .Vulnerabilities }}
{{- if or (eq .Severity "CRITICAL") (eq .Severity "HIGH") }}
{{- if .FixedVersion }}
| {{ .PkgName }} | {{ .VulnerabilityID }} | {{ .InstalledVersion }} | {{ .FixedVersion }} | {{ .Severity }} |
{{- end }}
{{- end }}
{{- end }}
{{- end }}

**结论**：{{ if gt (len (where .Results "Vulnerabilities" "severity" "CRITICAL")) 0 }}🔴 存在 CRITICAL 漏洞，建议阻断发布。{{ else }}🟡 无 CRITICAL 漏洞，HIGH 级别漏洞请在 7 天内修复。{{ end }}
```

执行扫描并应用模板：

```bash
trivy image --format template --template @report-template.tpl \
  --output summary.md python:3.4-alpine
```

查看生成的 `summary.md`：

```bash
cat summary.md
```

**预期输出**：
```markdown
# 安全扫描摘要

**扫描目标**：python:3.4-alpine  
**扫描时间**：2024-03-15 14:30:00  
**严重漏洞统计**：

| 级别 | 数量 |
|------|------|
| CRITICAL | 3 |
| HIGH | 8 |

**需立即处理的漏洞（CRITICAL & HIGH & fixed）**：

| 包名 | CVE | 当前版本 | 修复版本 | 级别 |
|------|-----|----------|----------|------|
| openssl | CVE-2022-0778 | 1.1.1g-r0 | 1.1.1l-r0 | HIGH |
| sqlite | CVE-2019-8457 | 3.32.1-r0 | 3.34.1-r0 | CRITICAL |

**结论**：🔴 存在 CRITICAL 漏洞，建议阻断发布。
```

### 步骤四：建立 `.trivyignore` 基线管理体系

**目标**：创建结构化的漏洞忽略文件，实现「已知风险的可追溯管理」。

创建 `.trivyignore`：

```
# ============================================
# 云帆科技 - Trivy 漏洞基线文件
# 规则：每条忽略必须包含原因、到期日、审批人
# 审计频率：每月 1 日由安全团队审查到期条目
# ============================================

# --- 开发依赖（不进入生产）---
# CVE-2021-23337: lodash prototype pollution
# 原因：仅用于单元测试框架，不打包到生产镜像
# 到期：2024-12-31
# 审批：security@cloud-sail.com
CVE-2021-23337

# --- OS Vendor Backport 已修复但版本号不匹配 ---
# CVE-2023-0464: OpenSSL 证书验证问题
# 原因：Debian 已 backport 补丁到 1.1.1n-0+deb11u5，Trivy 误报
# 到期：2024-06-30
# 审批：ops@cloud-sail.com
# pkg: openssl
CVE-2023-0464

# --- 无修复方案且影响可控 ---
# CVE-2023-29469: libxml2 已知问题
# 原因：上游未发布补丁，团队已评估不暴露外部 XML 解析接口
# 到期：2024-09-30
# 审批：security@cloud-sail.com
CVE-2023-29469
```

执行带基线的扫描：

```bash
trivy image --severity HIGH,CRITICAL \
  --ignorefile .trivyignore \
  python:3.4-alpine
```

**验证效果**：报告中不再出现 `.trivyignore` 中列出的三个 CVE。

> **可能遇到的坑**：`.trivyignore` 的匹配逻辑是「精确匹配 CVE ID」。如果同一个 CVE 出现在多个包中，所有实例都会被忽略。如果只想忽略特定包的实例，需要使用 VEX（见第 23 章）。

### 步骤五：自动化基线审计脚本

**目标**：防止 `.trivyignore` 中的条目无限期累积。

创建 `audit-trivyignore.py`：

```python
#!/usr/bin/env python3
"""
审计 .trivyignore 文件，检查：
1. 已过期的忽略条目
2. 缺少原因/到期日/审批人的条目
3. 输出审计报告
"""

import re
from datetime import datetime, timezone
from pathlib import Path

def audit_trivyignore(filepath: str) -> dict:
    content = Path(filepath).read_text()
    lines = content.splitlines()
    
    ignores = []
    current_meta = {}
    
    for line in lines:
        line = line.strip()
        if line.startswith('# 到期：'):
            current_meta['expiry'] = line.replace('# 到期：', '').strip()
        elif line.startswith('# 原因：'):
            current_meta['reason'] = line.replace('# 原因：', '').strip()
        elif line.startswith('# 审批：'):
            current_meta['approver'] = line.replace('# 审批：', '').strip()
        elif line and not line.startswith('#') and not line.startswith('==='):
            # 这是一个 CVE ID
            ignores.append({
                'cve': line,
                **current_meta
            })
            current_meta = {}
    
    now = datetime.now(timezone.utc)
    expired = []
    incomplete = []
    
    for item in ignores:
        if not item.get('reason') or not item.get('approver'):
            incomplete.append(item)
        elif item.get('expiry'):
            try:
                expiry_date = datetime.strptime(item['expiry'], '%Y-%m-%d').replace(tzinfo=timezone.utc)
                if expiry_date < now:
                    expired.append(item)
            except ValueError:
                incomplete.append(item)
    
    return {
        'total': len(ignores),
        'expired': expired,
        'incomplete': incomplete
    }

if __name__ == '__main__':
    result = audit_trivyignore('.trivyignore')
    print(f"审计结果：总条目 {result['total']}")
    print(f"已过期：{len(result['expired'])}")
    for item in result['expired']:
        print(f"  - {item['cve']} (到期: {item.get('expiry', 'N/A')})")
    print(f"信息不完整：{len(result['incomplete'])}")
    for item in result['incomplete']:
        print(f"  - {item['cve']}")
    
    if result['expired'] or result['incomplete']:
        exit(1)  # CI 失败，强制清理
```

在 CI 中集成审计：

```yaml
# .gitlab-ci.yml 片段
trivy_ignore_audit:
  stage: security
  image: python:3.11-slim
  script:
    - python audit-trivyignore.py
  allow_failure: false
```

### 步骤六：报告自动化分发

**目标**：将 Trivy 的 JSON 报告自动发送到企业微信/钉钉/Slack。

创建 `notify.py`：

```python
#!/usr/bin/env python3
"""
读取 Trivy JSON 报告，提取 Critical 漏洞，推送到企业微信
"""

import json
import sys
import urllib.request

def send_wechat(webhook_url: str, report_path: str):
    with open(report_path) as f:
        report = json.load(f)
    
    critical_count = 0
    messages = []
    
    for result in report.get('Results', []):
        for vuln in result.get('Vulnerabilities', []):
            if vuln.get('Severity') == 'CRITICAL':
                critical_count += 1
                messages.append(
                    f"- {vuln['VulnerabilityID']} ({vuln['PkgName']} "
                    f"{vuln.get('InstalledVersion', 'unknown')})"
                )
    
    if critical_count == 0:
        content = f"✅ 安全扫描通过：{report['ArtifactName']} 未发现 Critical 漏洞"
    else:
        content = (
            f"🚨 Critical 漏洞告警：{report['ArtifactName']}\n\n"
            f"共发现 {critical_count} 个 Critical 漏洞：\n"
            + "\n".join(messages[:10])  # 最多显示 10 条
        )
    
    data = json.dumps({"msgtype": "text", "text": {"content": content}}).encode()
    req = urllib.request.Request(
        webhook_url, data=data,
        headers={"Content-Type": "application/json"}
    )
    urllib.request.urlopen(req)

if __name__ == '__main__':
    if len(sys.argv) < 3:
        print(f"Usage: {sys.argv[0]} <webhook_url> <report.json>")
        sys.exit(1)
    send_wechat(sys.argv[1], sys.argv[2])
```

在 CI 中使用：

```bash
# 扫描并输出 JSON
trivy image --format json --output report.json python:3.4-alpine

# 推送到企业微信
python notify.py "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=YOUR_KEY" report.json
```

### 测试验证

1. 执行五种格式的扫描命令，确认每种格式都能正常生成文件。
2. 对比 `--severity HIGH,CRITICAL` 和默认扫描的输出数量，确认过滤生效。
3. 使用自定义 Go Template 生成 `summary.md`，确认只包含关键字段和结论。
4. 配置 `.trivyignore` 后重新扫描，确认被忽略的 CVE 不再出现。
5. 运行 `audit-trivyignore.py`，验证能正确检测过期和不完整的条目。
6. 执行 `notify.py` 推送报告到 Webhook，验证消息格式正确。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 格式丰富 | 支持 table/json/sarif/cyclonedx/spdx/template 六种格式 | 某些格式（如 SARIF）的字段映射不够完整 |
| 过滤灵活 | severity/状态/类型/包名多层过滤，精准降噪 | 过滤参数组合过多，新手容易混淆 |
| 基线管理 | `.trivyignore` 简单易用，适合团队共识沉淀 | 缺乏内置的到期提醒和审批工作流 |
| 模板定制 | Go Template 强大，可生成任意格式的报告 | 模板语法学习曲线较陡，调试困难 |
| 自动化 | JSON 输出便于脚本处理，易于集成通知系统 | 需要自行编写脚本，无内置通知渠道 |

### 适用场景

1. **管理层摘要报告**：用 Go Template 生成只含 Critical/HIGH + fixed 漏洞的一页纸摘要。
2. **CI 安全门禁**：用 `--severity CRITICAL --exit-code 1` 阻断包含严重漏洞的发布。
3. **安全工单自动化**：用 JSON + Python 脚本将漏洞自动导入 JIRA/DefectDojo。
4. **SBOM 合规交付**：用 `--format cyclonedx` 生成符合客户要求的物料清单。
5. **已知风险基线管理**：用 `.trivyignore` + 审计脚本实现漏洞接受的版本化和可追溯。

**不适用场景**：
1. 需要实时的交互式漏洞管理平台（如漏洞趋势图、历史对比）——Trivy 是扫描器，不是漏洞管理系统。
2. 需要自动修复漏洞（如自动提 PR 升级依赖）——Trivy 只负责发现，修复需配合 Renovate/Dependabot。

### 注意事项

- **`.trivyignore` 的继承关系**：Trivy 会查找当前目录、用户主目录、系统目录的 `.trivyignore` 文件，多层叠加。建议项目级 `.trivyignore` 纳入代码审查。
- **Template 中的 nil 安全**：Go Template 中访问嵌套字段时，如果上层字段为 nil，会报错。建议使用 `{{ if .Field }}...{{ end }}` 做非空判断。
- **SARIF 的 Severity 映射**：SARIF 格式的 severity 与 Trivy 内部的 severity 有一一映射关系，但某些平台（如 GitHub）对 SARIF 的展示有额外要求，需确保 Trivy 版本 >= 0.45。

### 常见踩坑经验

**踩坑案例 1：Go Template 报错 `nil pointer evaluating`**
- **现象**：自定义模板运行时 panic。
- **根因**：模板中直接访问了可能为空的嵌套字段，如 `{{ .Vulnerabilities[0].FixedVersion }}`。
- **解法**：使用 `{{ range }}` 遍历，或在外层加 `{{ if .Vulnerabilities }}` 判断。

**踩坑案例 2：`.trivyignore` 不生效**
- **现象**：扫描报告中仍然出现 `.trivyignore` 中列出的 CVE。
- **根因**：Trivy 默认查找当前工作目录的 `.trivyignore`，如果扫描目标在其他目录，可能找不到文件。
- **解法**：显式指定 `--ignorefile /path/to/.trivyignore`，或在目标目录下放置 `.trivyignore`。

**踩坑案例 3：JSON 报告中漏洞总数与 Table 不一致**
- **现象**：`jq` 统计的 JSON 漏洞数比 table 输出的少。
- **根因**：Trivy 的 JSON 输出中，`Results` 数组可能包含多个条目（如 OS 包和语言包分开），`jq` 查询时漏掉了某些层级。
- **解法**：使用递归查询 `.. | .Vulnerabilities? | arrays | .[]` 确保遍历所有层级。
