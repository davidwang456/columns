## 1. 项目背景

### 业务场景

云帆科技的架构师小白已经理解了 Fanal 引擎（第 33 章）如何从镜像中提取结构化信息，现在要揭开最后一个黑盒：「有了包清单（OpenSSL 1.1.1k），Trivy 如何判断它是否有漏洞？」

这个问题在小白的团队中引发了一个真实需求。云帆科技有一套内部的私有 Python 包索引（类似企业版 PyPI），管理了 200 多个内部包。这些内部包偶尔也会被发现安全漏洞——某个版本引入了已知的 CVE。但 Trivy 的 `trivy-db` 只覆盖公共包（PyPI、npm、Maven Central 等），不包含云帆科技的内部包。

安全团队问小白：「我们能不能把内部包的漏洞信息也录入到 Trivy 的检测体系里？比如我们自研的 `cloudsail-crypto` 包 1.2.0 版本有一个已知的 XML 注入漏洞，我们希望 Trivy 扫描镜像时能报出来。」

这就触及了 Trivy 漏洞检测引擎的核心——Detector 的接口抽象、版本比较算法、以及 trivy-db 的 BoltDB 查询机制。

### 痛点放大

**第一，Detector 接口看似简单实则内涵丰富。** `Detect()` 方法只有寥寥几行签名，但每个 OS 发行版（Alpine、Debian、RHEL）和每种语言生态（npm、pip、maven）的版本比较规则完全不同。例如，`3.0.0_p1-r5` 是 Alpine 的 apk 版本格式，`1.1.1n-0+deb11u5` 是 Debian 的 dpkg 版本格式——两者无法用同一个版本比较器处理。

**第二，trivy-db 的 BoltDB 存储结构对外部开发者不够透明。** 虽然知道它是一个 key-value 数据库，但 key 的命名规则、value 的编码格式、bucket 的层级结构——这些都需要从源码中反推。

**第三，多数据源的合并逻辑。** Trivy 的漏洞数据不仅来自 trivy-db（OS + 语言），还有 java-db（Maven）、CISA KEV、EPSS 等外部数据。Detector 如何合并这些来源？如何解决同一个 CVE 在多个数据源中的评分差异？

**本章的核心目标是：深入 Detector 源码，理解 OS 包与语言包的漏洞匹配算法（版本比较器），掌握 trivy-db 的 BoltDB 查询结构，并实战实现一个自定义 Library Detector 对接私有漏洞数据源。**

## 2. 项目设计

**场景**：云帆科技的安全数据内部化研讨会，小白、老李和大师在讨论如何把内部包的漏洞数据接入 Trivy。

---

**老李**：「我们内部的安全研究员每周都会发现一些内部包的安全问题。现在这些信息记录在 Confluence 里，靠人工比对——运维扫出了 `cloudsail-crypto==1.2.0`，然后去 Confluence 查『这个版本有没有已知漏洞』。太原始了。」

**大师**：「技术映射：Trivy 的 Detector 就是『漏洞匹配引擎』。它的核心逻辑很简单——你给它一个包名 + 版本号，它去漏洞数据库里找：『有没有一个 Advisories 说这个包的这个版本是受影响的？』如果找到了，就标记为漏洞。」

**小白**：「但版本比较不是简单的字符串比较。`1.2.0 < 1.2.10` 在语义化版本下是对的，但在某些生态中不同。更麻烦的是操作系统包——Alpine 的 `openssl-1.1.1k-r0` 和 `openssl-1.1.1n-r0`，你很难用通用算法比较。

Trivy 的解决策略是『每种生态一套版本比较器』：

```go
// pkg/detector/library/driver.go
func NewDriver(libType ftypes.LangType) (Driver, error) {
    switch libType {
    case ftypes.Npm, ftypes.Yarn, ftypes.Pnpm:
        return newDriver(ecosystem.Npm, npm.Comparer{})
    case ftypes.Pip, ftypes.Pipenv, ftypes.Poetry:
        return newDriver(ecosystem.Pip, pep440.Comparer{})
    case ftypes.GemSpec, ftypes.Bundler:
        return newDriver(ecosystem.RubyGems, rubygems.Comparer{})
    case ftypes.GoModule, ftypes.GoBinary:
        return newDriver(ecosystem.Go, genericComparer{})
    // ... 等等
    }
}
```

每个 Comparer 实现了 `compare.Comparer` 接口：

```go
// pkg/detector/library/compare/compare.go
type Comparer interface {
    IsVulnerable(installedVersion string, fixedVersion string) bool
}
```

对于 npm，`IsVulnerable("1.2.0", "1.2.10")` 返回 `true`（需要升级）。对于 Python PEP 440，`IsVulnerable("1.2.0", "1.2.0.post1")` 也要正确处理。」

**大师**：「再看 OS 包检测。`pkg/detector/ospkg/` 下为每个 OS 发行版实现了独立的 Driver：

```
pkg/detector/ospkg/
├── alpine/
│   └── alpine.go       # Alpine apk 的检测逻辑
├── debian/
│   └── debian.go       # Debian dpkg 的检测逻辑
├── redhat/
│   └── redhat.go       # RHEL/CentOS rpm 的检测逻辑
├── ubuntu/
│   └── ubuntu.go       # Ubuntu 的检测逻辑
├── suse/
│   └── suse.go         # SUSE 的检测逻辑
└── ...
```

每个 Driver 的核心逻辑都是：

1. 从 trivy-db 的 BoltDB 中查询该 OS 版本的 Advisories。
2. 遍历每个包，在 Advisories 中查找匹配的 CVE。
3. 用 OS 专属的版本比较逻辑判断 `installedVersion < fixedVersion`。

以 Alpine 为例（简化自源码）：
```go
func (s Scanner) Detect(osVer string, pkgs []ftypes.Package) ([]types.DetectedVulnerability, error) {
    // 1. 从 trivy-db 中读取 Alpine 的 Advisories
    advisories, err := s.advisoryDB.Get("alpine", osVer)
    // osVer 如 "3.19", "3.18", "edge"

    var vulns []types.DetectedVulnerability
    for _, pkg := range pkgs {
        // 2. 查找这个包对应的 Advisories（CVE 列表）
        for _, adv := range advisories[pkg.Name] {
            // 3. 版本比较：installed vs fixed
            if pkg.Version < adv.FixedVersion {
                vulns = append(vulns, types.DetectedVulnerability{
                    VulnerabilityID: adv.VulnerabilityID,
                    PkgName:         pkg.Name,
                    InstalledVersion: pkg.Version,
                    FixedVersion:    adv.FixedVersion,
                })
            }
        }
    }
    return vulns, nil
}
```

## 3. 项目实战

### 环境准备

- **Go**：1.21+
- **Trivy 源码**：已 clone 到本地
- **BoltDB 浏览器**：`go install github.com/br0xen/boltbrowser@latest`（可选，用于直接查看 trivy-db）

### 步骤一：理解 trivy-db 的 BoltDB 结构

**目标**：直接查看 trivy-db 文件，理解 key-value 的存储格式。

```bash
# 下载最新的 trivy-db
trivy image --download-db-only alpine:latest

# 找到 trivy-db 的位置
# macOS/Linux: ~/.cache/trivy/db/trivy.db
# 或通过环境变量查看
echo $TRIVY_CACHE_DIR

# 用 Go 读取 BoltDB（临时脚本）
cat > read-db.go <<'EOF'
package main

import (
    "fmt"
    "log"

    bolt "go.etcd.io/bbolt"
)

func main() {
    db, err := bolt.Open(os.ExpandEnv("$HOME/.cache/trivy/db/trivy.db"), 0600, nil)
    if err != nil {
        log.Fatal(err)
    }
    defer db.Close()

    db.View(func(tx *bolt.Tx) error {
        // 列出所有 bucket
        return tx.ForEach(func(name []byte, b *bolt.Bucket) error {
            fmt.Printf("Bucket: %s (items: %d)\n", name, b.Stats().KeyN)
            return nil
        })
    })
}
EOF
```

**BoltDB 的 Bucket 结构**：

```
trivy.db (BoltDB)
├── vulnerability/     # 核心漏洞数据
│   ├── alpine/
│   │   ├── 3.19/     # key: package_name, value: []Advisory
│   │   └── 3.18/
│   ├── debian/
│   │   ├── 11/       # Debian 11 (bullseye)
│   │   └── 12/       # Debian 12 (bookworm)
│   ├── redhat/
│   └── ...
├── vulnerability-info/ # 漏洞详情（描述、CVSS、参考链接）
│   └── CVE-2023-XXXX ↦ {title, description, cvss, references}
├── data-source/       # 数据源元信息
│   └── ...
└── metadata/          # 数据库版本信息
    ├── version: 2
    └── updated-at: 2024-03-15T00:00:00Z
```

> **关键发现**：trivy-db 中的 key 不是 CVE ID，而是 `OS/version/package_name`。查询逻辑是「给定 OS 版本和包名，找到所有 Advisory」，而不是「给定 CVE，找到所有受影响的包」。这意味着 Trivy 在扫描时是「包驱动」的，不是「CVE 驱动」的。

### 步骤二：理解版本比较算法

**目标**：深入 Go 语义化版本、Debian 版本号和 RPM 版本的比较逻辑。

Trivy 的版本比较逻辑在 `pkg/version/` 目录下：

```bash
pkg/version/
├── version.go          # 通用版本比较函数
├── semver/             # 语义化版本比较（用于 Go modules）
├── pep440/             # Python PEP 440 版本比较
├── rpm/                # RPM 版本比较（Red Hat / CentOS）
├── dpkg/               # Debian dpkg 版本比较
├── apk/                # Alpine apk 版本比较
└── rust/               # Rust Cargo 版本比较
```

**Debian 版本号格式示例**：

```
1.1.1n-0+deb11u5
│      │  │       │
│      │  │       └─ debian revision (deb11u5)
│      │  └───────── upstream version (1.1.1n)
│      └──────────── upstream version
└─────────────────── epoch (3.x 系统使用)
```

**Debian 版本比较规则**（`pkg/version/dpkg/`）：
- epoch 有最高优先级（但通常不存在）
- upstream version 按数字/字母分割比较
- debian revision 以 `~` 优先于字母优先于数字

这是一个在 Go 中极其精巧的实现——约 500 行代码处理了所有边界条件。

**实战测试版本比较**：

```go
package main

import (
    "fmt"
    "strings"

    "golang.org/x/xerrors"

    // Trivy 的版本比较包
    "github.com/aquasecurity/trivy/pkg/detector/library/compare"
    "github.com/aquasecurity/trivy/pkg/detector/library/compare/npm"
    "github.com/aquasecurity/trivy/pkg/detector/library/compare/semver"
)

func main() {
    // npm 版本比较
    npmComparer := npm.Comparer{}
    fmt.Println(npmComparer.IsVulnerable("1.2.0", "1.2.10"))  // true (需要升级)
    fmt.Println(npmComparer.IsVulnerable("1.2.10", "1.2.5"))  // false (已是最新)

    // 语义化版本比较
    semverComparer := semver.Comparer{}
    fmt.Println(semverComparer.IsVulnerable("1.0.0", "1.0.1")) // true
    fmt.Println(semverComparer.IsVulnerable("2.0.0", "1.9.9")) // false
}
```

### 步骤三：实现自定义 Library Detector

**目标**：为云帆科技的内部 Python 包索引创建自定义漏洞检测。

创建 `custom-detector/cloudsail/detector.go`：

```go
package cloudsail

import (
    "context"
    "encoding/json"
    "fmt"
    "net/http"
    "strings"

    "github.com/aquasecurity/trivy/pkg/detector/library"
    "github.com/aquasecurity/trivy/pkg/fanal/types"
    vulnTypes "github.com/aquasecurity/trivy/pkg/types"
)

// InternalVulnDB 内部漏洞数据库的接口
type InternalVulnDB interface {
    Query(ctx context.Context, pkgName string) ([]InternalAdvisory, error)
}

// InternalAdvisory 内部漏洞公告
type InternalAdvisory struct {
    VulnerabilityID string `json:"vulnerability_id"` // CS-2024-001
    PkgName         string `json:"pkg_name"`         // cloudsail-crypto
    FixedVersion    string `json:"fixed_version"`    // 1.3.0
    Severity        string `json:"severity"`         // CRITICAL
    Title           string `json:"title"`
    Description     string `json:"description"`
}

// Detector 云帆内部包检测器
type Detector struct {
    vulnDB InternalVulnDB
}

func NewDetector(vulnDB InternalVulnDB) *Detector {
    return &Detector{vulnDB: vulnDB}
}

// Detect 对镜像中的 Python 包执行内部漏洞检测
func (d *Detector) Detect(
    ctx context.Context,
    pkgs []types.Package,
) ([]vulnTypes.DetectedVulnerability, error) {

    var detected []vulnTypes.DetectedVulnerability

    // 只处理 Python 包（pip / poetry 安装的）
    for _, pkg := range pkgs {
        // 只检查公司内部包（以 cloudsail- 开头）
        if !strings.HasPrefix(pkg.Name, "cloudsail-") {
            continue
        }

        // 查询内部漏洞数据库
        advisories, err := d.vulnDB.Query(ctx, pkg.Name)
        if err != nil {
            return nil, fmt.Errorf("failed to query internal vuln DB for %s: %w", pkg.Name, err)
        }

        for _, adv := range advisories {
            // 版本比较：使用 Python PEP 440 比较器
            // 简化处理：语义化版本字符串比较
            if compareVersions(pkg.Version, adv.FixedVersion) < 0 {
                detected = append(detected, vulnTypes.DetectedVulnerability{
                    VulnerabilityID:  adv.VulnerabilityID,
                    PkgName:          adv.PkgName,
                    InstalledVersion: pkg.Version,
                    FixedVersion:     adv.FixedVersion,
                    Severity:         adv.Severity,
                    Title:            adv.Title,
                    Description:      adv.Description,
                    SeveritySource:   "cloudsail-internal",
                    DataSource: &vulnTypes.DataSource{
                        ID:   "cloudsail-vuln-db",
                        Name: "CloudSail Internal Vulnerability Database",
                        URL:  "https://vulndb.internal.cloud-sail.com",
                    },
                })
            }
        }
    }

    return detected, nil
}

// compareVersions 简易的语义化版本比较
func compareVersions(v1, v2 string) int {
    parts1 := strings.Split(strings.TrimPrefix(v1, "v"), ".")
    parts2 := strings.Split(strings.TrimPrefix(v2, "v"), ".")

    for i := 0; i < len(parts1) && i < len(parts2); i++ {
        if parts1[i] < parts2[i] {
            return -1
        }
        if parts1[i] > parts2[i] {
            return 1
        }
    }

    if len(parts1) < len(parts2) {
        return -1
    }
    if len(parts1) > len(parts2) {
        return 1
    }
    return 0
}
```

### 步骤四：实现内部漏洞数据库的 HTTP Client

**目标**：对接公司内部的漏洞数据库 API。

创建 `custom-detector/cloudsail/vulndb.go`：

```go
package cloudsail

import (
    "context"
    "encoding/json"
    "fmt"
    "net/http"
    "time"
)

// HTTPVulnDB 通过 HTTP API 访问内部漏洞数据库
type HTTPVulnDB struct {
    baseURL    string
    httpClient *http.Client
}

func NewHTTPVulnDB(baseURL string) *HTTPVulnDB {
    return &HTTPVulnDB{
        baseURL: baseURL,
        httpClient: &http.Client{
            Timeout: 10 * time.Second,
        },
    }
}

func (h *HTTPVulnDB) Query(ctx context.Context, pkgName string) ([]InternalAdvisory, error) {
    url := fmt.Sprintf("%s/api/v1/advisories?package=%s", h.baseURL, pkgName)

    req, err := http.NewRequestWithContext(ctx, "GET", url, nil)
    if err != nil {
        return nil, err
    }
    req.Header.Set("Accept", "application/json")
    req.Header.Set("X-API-Key", "cloudsail-internal-key") // 生产中使用 Secret Manager

    resp, err := h.httpClient.Do(req)
    if err != nil {
        return nil, err
    }
    defer resp.Body.Close()

    if resp.StatusCode != http.StatusOK {
        return nil, fmt.Errorf("vuln DB returned status %d", resp.StatusCode)
    }

    var advisories []InternalAdvisory
    if err := json.NewDecoder(resp.Body).Decode(&advisories); err != nil {
        return nil, err
    }

    return advisories, nil
}
```

### 步骤五：集成到扫描流程

**目标**：将内部 Detector 嵌入到 Trivy 扫描的 post-processing 阶段。

创建 `custom-detector/main.go`：

```go
package main

import (
    "context"
    "encoding/json"
    "fmt"
    "log"
    "os"
    "os/exec"

    "github.com/cloud-sail/trivy-extensions/cloudsail"
)

func main() {
    if len(os.Args) < 2 {
        log.Fatal("Usage: custom-scanner <image>")
    }
    image := os.Args[1]
    ctx := context.Background()

    // Step 1: 使用标准 Trivy 扫描（获取包清单）
    cmd := exec.Command("trivy", "image",
        "--format", "json",
        "--output", "/tmp/trivy-raw.json",
        "--scanners", "vuln",
        image,
    )
    cmd.Stderr = os.Stderr
    if err := cmd.Run(); err != nil {
        // exit code 1 表示发现了漏洞——这是正常的
        log.Printf("Trivy scan completed with exit code (may be non-zero): %v", err)
    }

    // Step 2: 读取 Trivy 报告的 JSON，提取 Python 包信息
    var report struct {
        Results []struct {
            Packages []struct {
                Name    string `json:"Name"`
                Version string `json:"Version"`
            } `json:"Packages"`
        } `json:"Results"`
    }

    raw, _ := os.ReadFile("/tmp/trivy-raw.json")
    json.Unmarshal(raw, &report)

    // Step 3: 对所有 Python 包运行内部检测
    vulnDB := cloudsail.NewHTTPVulnDB("https://vulndb.internal.cloud-sail.com")
    detector := cloudsail.NewDetector(vulnDB)

        }
    }

    // Step 4: 输出内部漏洞检测结果

    fmt.Printf("Total packages checked: %d\n", len(allPkgs))
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| OS Driver 抽象 | 每个发行版独立实现，不互相干扰 | 新增 OS 支持需完整实现 Driver 接口 |
| 版本比较器 | 精确匹配各生态的版本规则 | 维护成本高——每种生态都要维护一套规则 |
| trivy-db BoltDB | 单文件、零依赖、嵌入式的 KV 存储 | 并发写入性能差；不支持复杂查询 |
| Library Detector | 覆盖 15+ 种语言生态 | 语言生态新增包管理器时需适配 |
| 自定义数据源 | 通过实现接口对接私有漏洞库 | 数据一致性需自行保证 |

### 适用场景

1. **企业私有漏洞数据源接入**：将内部安全团队的漏洞发现自动集成到 Trivy 扫描流程。
2. **特定生态的深度定制**：如新增 Rust/Cargo 的支持。
3. **安全数据研究员**：理解 CVE 匹配算法、假阳性根因分析。
4. **构建独立的漏洞检测工具**：复用 Trivy 的 Detector + Version 比较器。
5. **trivy-db 的数据质量审计**：直接读取 BoltDB 验证数据的准确性和时效性。

**不适用场景**：
1. 对 Trivy 只需 CLI 使用的场景——不需要改动 Detector。
2. 内部漏洞数据极少（< 10 条）的场景——手动 `.trivyignore` 管理更简单。

### 注意事项

- **trivy-db 的 schema 在不同版本间可能变化**。读取 BoltDB 时需匹配当前 Trivy 版本的 schema。
- **版本比较器的边界条件极其复杂**。不要轻易尝试自己实现——优先使用 Trivy 内置的 `pkg/version/` 包。
- **内部漏洞数据库的 SLA**：如果自定义 Detector 依赖 HTTP API，API 的超时和降级策略至关重要——API 挂了不能 Block 住整个扫描流程。

### 常见踩坑经验

**踩坑案例 1：BoltDB 的 bucket 名称大小写敏感**
- **现象**：程序读取 `bolt.Open("trivy.db")` 后找不到 `vulnerability` bucket。
- **根因**：BoltDB 的 bucket name 是严格的 byte-for-byte 比较，`"Vulnerability"` ≠ `"vulnerability"`。
- **解法**：查看 Trivy 源码中 `pkg/db/` 下定义的 bucket name 常量。

**踩坑案例 2：版本比较显示 "1.2.0" 比 "1.2.0-1" 新，但实际上后者更新**
- **现象**：Debian 镜像中 `libssl1.1=1.1.1n-0+deb11u5` 被 Trivy 误报为受 CVE 影响。
- **根因**：使用了错误的版本比较器——用语义化版本比较 Debian 包的 `+deb11u5` 后缀。
- **解法**：Debian 包必须用 `pkg/version/dpkg/` 的比较器，不能用通用 Comparer。

**踩坑案例 3：java-db 的查询性能在首次使用时极慢**
- **现象**：首次扫描 Java 项目时，`library.Detect()` 耗时超过 30 秒。
- **根因**：`java-db` 是一个独立的 SQLite 数据库（约 1GB），首次查询需要加载索引。
- **解法**：在后台预热 java-db；或在 CI 中使用有缓存的 Runner。

### 思考题

1. Trivy 的检测是「包驱动」的——给定包名查 CVE。但在某些场景下（如 Log4Shell），需要在不知道包名的情况下，找到所有使用了受影响库的镜像。这种「CVE 驱动」的反向搜索能否在现有 trivy-db 结构上高效实现？如果不能，需要怎样的数据结构改造？
2. 如果公司使用 3 个不同的私有包管理器（Python internal PyPI、Node.js internal npm registry、Java internal Maven），如何设计一个统一的「自定义 Detector 注册表」，让每类包的检测逻辑独立维护但共享同一个扫描框架？
