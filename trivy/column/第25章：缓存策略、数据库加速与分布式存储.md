# 第25章：缓存策略、数据库加速与分布式存储

> 版本：Trivy v0.50+
> 面向人群：平台工程师、SRE、安全架构师

---

## 1. 项目背景

### 业务场景

云帆科技上一章落地了并行扫描调度，全量扫描时间从 4 小时缩短到 26 分钟，CTO 很满意。但新问题也随之而来——随着业务扩展到 3 个地域（北京、上海、新加坡），每个地域独立部署了 Trivy 扫描节点，挑战升级了。

首先是「缓存孤岛」。北京集群的扫描节点已经预热了 `ubuntu:22.04` 和 `node:20-alpine` 的缓存，但上海集群的节点一无所知。每个节点都要独立下载 `trivy-db`（约 300MB 的漏洞数据库）、`java-db`（约 1GB 的 Java 漏洞索引），以及预热常用基础镜像层。3 个地域 × 2 台节点 = 6 份独立的全量数据，总浪费存储约 8GB，且每次数据库更新都要 6 个节点分别下载。

其次是「数据库下载风暴」。每天 UTC 时间 trivy-db 更新时，6 个节点几乎同时触发 `--download-db-only`，对外网的 GitHub Release 发起了 6 次下载请求。某天 GitHub 限流（rate limit），3 个节点的 DB 下载失败，当天的所有扫描都依赖过期数据库——安全团队直到第二天才发现这个问题。

更让平台工程师小白揪心的是「报告散落」。每个扫描节点的报告存储在本地磁盘中。运维想归档一个月前的扫描报告做审计时，发现部分节点的磁盘被日志打满，旧的报告已被自动清理。管理层的安全周报因此缺失了一周的数据，审计留下缺口。

CTO 在季度规划会上说了句话引发了思考：「我们要像管理代码一样管理安全数据。代码有 Git，构建产物有 Artifactory，为什么安全数据（漏洞库、报告、SBOM）就没有统一的存储和加速层？」

### 痛点放大

**第一，分布式缓存的缺失。** 单个节点的本地缓存无法跨节点共享。相同基础镜像层的分析结果、相同版本的 trivy-db，每个节点都要独立构建——计算和存储的重复浪费随节点数线性增长。

**第二，漏洞数据库的同步延迟与单点依赖。** 所有节点直接依赖 GitHub（外网），不仅受限于带宽和限流，在内网隔离环境中更是完全不可用。缺乏本地託管的 DB 镜像，扫描基础设施的可用性严重依赖外网。

**第三，扫描结果的无序散落。** 每个节点的结果存储在本地，缺乏统一的持久化、索引和查询能力。这对于需要长期留存（合规要求通常为 1-3 年）和跨节点聚合的安全数据来说，是不可接受的。

**本章的核心目标是：构建 Redis 共享缓存、本地 DB 镜像仓库、对象存储（MinIO/S3）三层存储基础设施，将 Trivy 扫描从「无状态单机运行」升级为「有状态分布式服务」。**

## 2. 项目设计

**场景**：云帆科技的基础设施优化会，小白（平台工程师）、小胖（SRE）和大师正在讨论如何让多个 Trivy 节点高效协作。

---

**小胖**：「这问题简单啊——给 6 个节点各挂一个 SSD，各用各的，互不干扰。不就几 GB 数据吗？」

**小白**：「你这个方案的问题是『数据漂移』。假如节点A的 DB 是昨天的版本，节点B因为下载失败还在用前天的版本，同一个镜像在两个节点上扫出的漏洞数量可能不一致。到时候安全团队问你『到底有没有漏洞』，你怎么回答？」

**大师**：「技术映射：本地缓存就像每个家庭自建一个小仓库，存储自己的粮食。共享缓存就像社区统一建了一个大冷库，所有家庭按需取用。冷库的运营成本（维护、更新、防过期）比分散的小仓库低得多，而且保证所有人拿到的是同一批粮食。」

**小胖**：「那具体怎么实现共享缓存？」

**小白**：「Trivy 支持 `TRIVY_CACHE_BACKEND=redis` 的环境变量。设置后，Trivy 不再将分析结果写入本地磁盘，而是写入 Redis。所有配置了相同 Redis 地址的 Trivy 节点，都能从 Redis 读取其他节点已经分析过的层。

```bash
export TRIVY_CACHE_BACKEND=redis
export TRIVY_CACHE_REDIS_ENDPOINT=redis://redis-cluster:6379
```

Trivy 会把每个镜像层的 SHA256 作为 key、该层的包分析结果作为 value，存入 Redis。当另一个节点扫描共享同一基础层的镜像时，直接命中 Redis 缓存，跳过解压与分析。」

**小胖**：「那 Redis 内存够吗？公司几百个镜像的缓存数据得多大？」

**大师**：「技术映射：Redis 内存就像共享冰箱的容量。你不能把所有东西都塞进去，需要有策略地决定什么该放、什么可以丢。Trivy 的 Redis 后端支持 TTL（过期时间），你可以设置 7 天或 30 天的过期策略。而且 Redis Cluster 可以横向扩展——当你发现内存不够时，加节点就行。」

**小白**：**「还有一个更关键的优化点：trivy-db 的本地化。** 如果我们 6 个节点都从 GitHub 下载 DB，不仅慢（每个约 300MB），而且受限于 GitHub API 的 rate limit。更好的方案是：

1. **外网节点**：一台专门的机器（或 CronJob）负责从 GitHub 下载最新的 trivy-db 和 java-db。
2. **本地 OCI Registry**：将下载好的 DB 打包成 OCI 镜像，推送到公司内部的 Harbor。
3. **内网节点**：所有扫描节点通过 `--db-repository` 指定内部 Harbor 的 DB 地址，Trivy 直接从内网 OCI Registry 拉取 DB。

```bash
# 外网节点：下载并打包 DB
trivy db update --download-db-only
oras push harbor.internal/trivy/trivy-db:latest ./trivy-db/

# 内网节点：从内网拉取
trivy image --db-repository harbor.internal/trivy/trivy-db my-app:latest
```

**小胖**：「那扫描结果的持久化呢？总不能全靠 Redis 吧，Redis 挂了数据就没了。」

**大师**：「对，Redis 是『快存储』，不适合做『持久存储』。我们需要另一个『慢存储』层——对象存储（如 MinIO 或 AWS S3）。架构是这样的：

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Node 1    │     │   Node 2    │     │   Node 3    │
│   Trivy     │     │   Trivy     │     │   Trivy     │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                    │                    │
       │  读取/写入         │                    │
       └────────────────────┼────────────────────┘
                     ┌──────▼──────┐
                     │    Redis    │  ← 热缓存（层分析结果、DB缓存）
                     │ (Cluster)   │
                     └─────────────┘
                            │
                     ┌──────▼──────┐
                     │ Harbor / OCI│  ← trivy-db 镜像托管
                     │  Internal   │
                     └─────────────┘
                            │
       ┌────────────────────┼────────────────────┐
       │                    │                    │
┌──────▼──────┐     ┌──────▼──────┐     ┌──────▼──────┐
│   MinIO /   │     │ PostgreSQL │     │  Report UI  │
│     S3      │     │  (元数据)   │     │  (Grafana)  │
│ SBOM/报告   │     │  扫描历史   │     │             │
└─────────────┘     └─────────────┘     └─────────────┘
```

三层数据流：
- **Redis（热层）**：高频读写、TTL 过期、极致速度；存储层分析缓存和 DB 元数据。
- **Harbor/OCI（温层）**：DB 镜像分发；对内网所有节点可达，无限带宽。
- **MinIO/S3 + PostgreSQL（冷层）**：持久化报告、SBOM、扫描历史；支持查询、聚合、审计。」

**小胖**：「这么复杂的架构，会不会比原来单机更容易出问题？」

**小白**：「恰恰相反。单机模式下，节点的本地磁盘坏了，该节点上所有的历史报告都丢失。在分布式架构中，报告存在 MinIO（3 副本），元数据在 PostgreSQL（主从），Redis 是纯缓存（挂了只是慢，不会丢数据）。整体的数据可靠性反而提高了。」

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+，多节点
- **Redis**：v7.0+（推荐 Cluster 模式，至少 3 主 3 从）
- **Harbor**：v2.9+（用于托管 trivy-db OCI 镜像）
- **MinIO**：集群模式（用于持久化报告和 SBOM）
- **ORAS CLI**：用于推送 trivy-db OCI 镜像

### 步骤一：搭建 Redis 共享缓存

**目标**：配置 Trivy 使用 Redis 作为分析缓存的共享后端。

```bash
# 1. 每个 Trivy 节点配置环境变量
export TRIVY_CACHE_BACKEND=redis
export TRIVY_CACHE_REDIS_ENDPOINT=redis://redis-master.trivy.svc.cluster.local:6379
export TRIVY_CACHE_TTL=168h  # 缓存有效期 7 天

# 或通过 trivy.yaml 配置文件
cat <<EOF > /etc/trivy/trivy.yaml
cache:
  backend: redis
  redis:
    endpoint: redis://redis-cluster:6379
  ttl: 168h
EOF

# 2. 验证缓存功能
# 在 Node-1 上扫描一个基础镜像
trivy image --cache-dir /data/trivy/cache ubuntu:22.04
# 输出: ... cached layers: 0 (首次，无缓存)

# 在 Node-2 上扫描同一个镜像
trivy image --cache-dir /data/trivy/cache ubuntu:22.04
# 输出: ... cached layers: 5 (命中 Node-1 写入的缓存)

# 3. 检查 Redis 中的缓存 Key
redis-cli --scan --pattern "trivy:cache:*" | head
# 输出: trivy:cache:image:sha256:abc123...
```

> **坑点**：Redis 缓存 key 的前缀与 Trivy 版本相关。升级 Trivy 版本后，旧缓存可能因格式不兼容而无法命中。建议在升级 Trivy 后执行 `FLUSHDB` 清空旧缓存，或用版本号前缀隔离。

### 步骤二：搭建本地 trivy-db 镜像仓库

**目标**：将 trivy-db 和 java-db 托管到内网 Harbor，所有节点从内网拉取。

创建 `sync-trivy-db.sh`（部署在外网可访问的机器上作为定时任务）：

```bash
#!/bin/bash
set -e

HARBOR_URL="harbor.internal.example.com"
HARBOR_PROJECT="trivy"
DB_DIR="/tmp/trivy-db-sync"
TRIVY_DB_IMAGE="${HARBOR_URL}/${HARBOR_PROJECT}/trivy-db:2"
JAVA_DB_IMAGE="${HARBOR_URL}/${HARBOR_PROJECT}/java-db:1"

# 登录 Harbor
echo "$HARBOR_PASSWORD" | oras login "$HARBOR_URL" -u "$HARBOR_USER" --password-stdin

# === 同步 trivy-db ===
echo ">>> Syncing trivy-db..."

# 下载最新版本 trivy-db
mkdir -p "$DB_DIR/trivy-db"
docker run --rm \
  -v "$DB_DIR/trivy-db:/root/.cache/trivy/db" \
  aquasec/trivy:latest \
  image --download-db-only --reset alpine:latest

# 打包成 OCI 镜像并推送
cd "$DB_DIR/trivy-db"
oras push "$TRIVY_DB_IMAGE" \
  trivy.db:application/octet-stream \
  metadata.json:application/json

# === 同步 java-db ===
echo ">>> Syncing java-db..."
mkdir -p "$DB_DIR/java-db"
docker run --rm \
  -v "$DB_DIR/java-db:/root/.cache/trivy/java-db" \
  aquasec/trivy:latest \
  image --download-java-db-only alpine:latest

cd "$DB_DIR/java-db"
oras push "$JAVA_DB_IMAGE" \
  trivy-java.db:application/octet-stream \
  metadata.json:application/json

echo ">>> Sync completed. Images:"
echo "  - $TRIVY_DB_IMAGE"
echo "  - $JAVA_DB_IMAGE"

# 清理临时文件
rm -rf "$DB_DIR"
```

配置 CronJob（每天凌晨 2 点执行）：

```yaml
# k8s-cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: trivy-db-sync
spec:
  schedule: "0 2 * * *"
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: sync
            image: alpine/trivy-db-sync:latest
            env:
            - name: HARBOR_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: harbor-credentials
                  key: password
          restartPolicy: OnFailure
```

**扫描节点配置**（指定内网 DB 仓库）：

```bash
# 方法一：环境变量
export TRIVY_DB_REPOSITORY=harbor.internal.example.com/trivy/trivy-db:2
export TRIVY_JAVA_DB_REPOSITORY=harbor.internal.example.com/trivy/java-db:1

# 方法二：trivy.yaml
cat <<EOF > /etc/trivy/trivy.yaml
db:
  repository: harbor.internal.example.com/trivy/trivy-db:2
  java-repository: harbor.internal.example.com/trivy/java-db:1
EOF

# 验证内网 DB 可用
trivy image --db-repository harbor.internal.example.com/trivy/trivy-db:2 alpine:latest
```

> **坑点**：`oras push` 推送的 OCI 镜像，需要用 Harbor 的「Proxy Cache」项目类型或手动创建项目。如果 Harbor 不支持 ORAS 推送，可以用 `docker pull/push` 通过 `docker save` + `docker load` 的迂回方式。

### 步骤三：搭建 MinIO 报告持久化存储

**目标**：所有扫描报告和 SBOM 自动上传到 MinIO，按日期和镜像组织目录结构。

创建 `report-archiver.py`（扫描完成后自动归档）：

```python
#!/usr/bin/env python3
"""
报告归档器：将 Trivy 输出自动上传到 MinIO
"""

import json
import os
import sys
from datetime import datetime
from pathlib import Path

from minio import Minio

MINIO_ENDPOINT = os.environ["MINIO_ENDPOINT"]
MINIO_ACCESS_KEY = os.environ["MINIO_ACCESS_KEY"]
MINIO_SECRET_KEY = os.environ["MINIO_SECRET_KEY"]
MINIO_BUCKET = os.environ.get("MINIO_BUCKET", "trivy-reports")


class ReportArchiver:
    def __init__(self):
        self.client = Minio(
            MINIO_ENDPOINT,
            access_key=MINIO_ACCESS_KEY,
            secret_key=MINIO_SECRET_KEY,
            secure=os.environ.get("MINIO_SECURE", "false").lower() == "true",
        )

        # 确保 Bucket 存在
        if not self.client.bucket_exists(MINIO_BUCKET):
            self.client.make_bucket(MINIO_BUCKET)

    def archive_scan_result(self, image_name, report_path, report_format):
        """归档单次扫描结果"""
        now = datetime.now()
        date_prefix = now.strftime("%Y/%m/%d")
        image_safe = image_name.replace("/", "_").replace(":", "-")

        # 组织 MinIO 路径：bucket/YYYY/MM/DD/image/type.format
        for ext in ['json', 'md', 'csv', 'spdx.json', 'cdx.json', 'sarif.json']:
            file_path = Path(report_path) / f"report.{ext}"
            if not file_path.exists():
                continue

            object_name = f"{date_prefix}/{image_safe}/{ext}/report.{ext}"
            self.client.fput_object(
                MINIO_BUCKET,
                object_name,
                str(file_path),
                content_type="application/json" if "json" in ext else "text/plain",
            )
            print(f"  Archived: {object_name} ({file_path.stat().st_size} bytes)")

    def list_reports(self, date=None, image=None):
        """列出指定条件的报告"""
        prefix = date or ""
        if image:
            prefix = f"{prefix}/{image.replace('/', '_').replace(':', '-')}"
        objects = self.client.list_objects(MINIO_BUCKET, prefix=prefix, recursive=True)
        return [obj.object_name for obj in objects]

    def get_report_stats(self, days=30):
        """获取最近 N 天的报告统计"""
        from datetime import timedelta
        cutoff = datetime.now() - timedelta(days=days)
        stats = {"total": 0, "size_bytes": 0, "images": set()}
        
        objects = self.client.list_objects(MINIO_BUCKET, recursive=True)
        for obj in objects:
            if obj.last_modified.replace(tzinfo=None) >= cutoff:
                stats["total"] += 1
                stats["size_bytes"] += obj.size
                # 从路径中提取镜像名
                parts = obj.object_name.split("/")
                if len(parts) >= 3:
                    stats["images"].add(f"{parts[0]}/{parts[1]}/{parts[2]}")  # YYYY/MM/DD/image
        
        stats["images"] = len(stats["images"])
        return stats


if __name__ == "__main__":
    archiver = ReportArchiver()

    # 示例：归档一个扫描结果
    if sys.argv[1] == "archive":
        archiver.archive_scan_result(sys.argv[2], sys.argv[3], sys.argv[4])

    # 示例：统计最近 30 天的报告量
    if sys.argv[1] == "stats":
        stats = archiver.get_report_stats(30)
        print(json.dumps(stats, indent=2))
```

### 步骤四：搭建 PostgreSQL 元数据索引

**目标**：将扫描结果的关键字段（镜像名、时间、漏洞数量、严重级别）存入 PostgreSQL，支持跨时间查询分析。

```sql
-- schema.sql
CREATE TABLE IF NOT EXISTS scan_records (
    id            BIGSERIAL PRIMARY KEY,
    image_name    VARCHAR(500) NOT NULL,
    image_digest  VARCHAR(100),
    scanned_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    trivy_version VARCHAR(20),
    db_updated_at TIMESTAMPTZ,
    
    -- 漏洞汇总
    critical_count INT DEFAULT 0,
    high_count     INT DEFAULT 0,
    medium_count   INT DEFAULT 0,
    low_count      INT DEFAULT 0,
    
    -- MinIO 中的报告路径
    report_json_path   VARCHAR(1000),
    report_sbom_path   VARCHAR(1000),
    report_html_path   VARCHAR(1000),
    
    -- 扫描元数据
    scan_status    VARCHAR(20) DEFAULT 'success', -- success / failed
    scan_duration  INTERVAL,
    error_message  TEXT,
    node_hostname  VARCHAR(200),
    
    UNIQUE(image_digest, scanned_at)
);

-- 索引设计
CREATE INDEX idx_scan_records_image ON scan_records(image_name);
CREATE INDEX idx_scan_records_scanned ON scan_records(scanned_at DESC);
CREATE INDEX idx_scan_records_critical ON scan_records(critical_count) WHERE critical_count > 0;

-- 常用查询视图
CREATE VIEW v_weekly_trend AS
SELECT
    date_trunc('week', scanned_at) AS week,
    SUM(critical_count) AS total_critical,
    SUM(high_count) AS total_high,
    COUNT(DISTINCT image_name) AS images_scanned
FROM scan_records
WHERE scanned_at > NOW() - INTERVAL '12 weeks'
GROUP BY date_trunc('week', scanned_at)
ORDER BY week DESC;
```

插入扫描记录：

```python
# db-recorder.py
import json
import os
import subprocess
from datetime import datetime

import psycopg2

def record_scan(image_name, report_json_path):
    """从扫描结果 JSON 中提取字段并写入 PostgreSQL"""
    with open(report_json_path) as f:
        report = json.load(f)

    # 汇总各严重级别漏洞数量
    counts = {"CRITICAL": 0, "HIGH": 0, "MEDIUM": 0, "LOW": 0}
    for result in report.get("Results", []):
        for vuln in result.get("Vulnerabilities", []):
            sev = vuln.get("Severity", "")
            if sev in counts:
                counts[sev] += 1

    # 获取镜像 digest
    digest = ""
    try:
        output = subprocess.check_output(
            ["docker", "inspect", "--format", "{{.RepoDigests}}", image_name],
            text=True
        )
        if output:
            digest = output.strip().split("@")[-1].rstrip("]")
    except Exception:
        pass

    conn = psycopg2.connect(os.environ["DATABASE_URL"])
    cur = conn.cursor()
    cur.execute("""
        INSERT INTO scan_records 
            (image_name, image_digest, critical_count, high_count, 
             medium_count, low_count, report_json_path, node_hostname)
        VALUES (%s, %s, %s, %s, %s, %s, %s, %s)
        ON CONFLICT (image_digest, scanned_at) DO NOTHING
    """, (
        image_name, digest,
        counts["CRITICAL"], counts["HIGH"],
        counts["MEDIUM"], counts["LOW"],
        report_json_path,
        os.uname().nodename,
    ))
    conn.commit()
    cur.close()
    conn.close()

    print(f"Recorded: {image_name} -> C:{counts['CRITICAL']} H:{counts['HIGH']}")
```

### 步骤五：编写统一扫描流水线（整合三层存储）

**目标**：一次扫描自动完成「缓存读取 → 扫描执行 → DB 记录 → 报告归档」。

创建 `unified-scan.sh`：

```bash
#!/bin/bash
set -e

# 配置
IMAGE="$1"
REDIS_ENDPOINT="${REDIS_ENDPOINT:-redis://localhost:6379}"
DB_REPO="${DB_REPO:-harbor.internal/trivy/trivy-db:2}"
REPORT_DIR="/tmp/trivy-reports/$(date +%Y%m%d_%H%M%S)"
MINIO_BUCKET="${MINIO_BUCKET:-trivy-reports}"

if [ -z "$IMAGE" ]; then
    echo "Usage: $0 <image:tag>"
    exit 1
fi

mkdir -p "$REPORT_DIR"

echo "=== [$(date)] Scanning: $IMAGE ==="
echo "  Redis Cache: $REDIS_ENDPOINT"
echo "  DB  Source:  $DB_REPO"

# 1. 扫描（使用 Redis 共享缓存 + 内网 DB）
START_TIME=$(date +%s)

trivy image \
  --cache-backend redis \
  --redis-endpoint "$REDIS_ENDPOINT" \
  --db-repository "$DB_REPO" \
  --severity HIGH,CRITICAL \
  --format json \
  --output "$REPORT_DIR/report.json" \
  --timeout 15m \
  "$IMAGE"

SCAN_EXIT_CODE=$?
SCAN_DURATION=$(( $(date +%s) - START_TIME ))

# 2. 生成多格式报告
if [ $SCAN_EXIT_CODE -eq 0 ] || [ $SCAN_EXIT_CODE -eq 1 ]; then
    # 生成 Markdown 摘要
    trivy convert --format template \
      --template @/etc/trivy/templates/exec-summary.tpl \
      --output "$REPORT_DIR/report.md" \
      "$REPORT_DIR/report.json"

    # 生成 CycloneDX SBOM
    trivy image --format cyclonedx \
      --output "$REPORT_DIR/report.cdx.json" \
      "$IMAGE"
fi

echo "=== Scan completed in ${SCAN_DURATION}s (exit code: $SCAN_EXIT_CODE) ==="

# 3. 记录到 PostgreSQL
if command -v db-recorder.py &> /dev/null; then
    python3 /usr/local/bin/db-recorder.py "$IMAGE" "$REPORT_DIR/report.json"
fi

# 4. 归档到 MinIO
if command -v report-archiver.py &> /dev/null; then
    python3 /usr/local/bin/report-archiver.py archive "$IMAGE" "$REPORT_DIR" "all"
fi

# 5. 清理临时文件（7 天后自动清理）
find /tmp/trivy-reports -type d -mtime +7 -exec rm -rf {} \; 2>/dev/null || true

exit $SCAN_EXIT_CODE
```

### 测试验证

1. 在两台独立的 Trivy 节点上分别执行 `unified-scan.sh`，验证 Redis 缓存共享生效（第二台节点的层分析耗时明显减少）。
2. 将 trivy-db 推送到内网 Harbor，断开外网后执行扫描，验证能通过 `--db-repository` 完成离线扫描。
3. 检查 MinIO 中的报告文件组织（`YYYY/MM/DD/image/format/`），确认目录结构清晰。
4. 查询 PostgreSQL `v_weekly_trend` 视图，验证能展示 12 周的漏洞趋势。
5. 模拟 Redis 宕机：停止 Redis 后扫描，验证 Trivy 自动回退到本地缓存目录，且不影响扫描结果正确性。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| Redis 共享缓存 | 跨节点复用分析结果；减少磁盘 I/O | Redis 宕机影响扫描速度（但不影响结果） |
| 本地 DB 镜像 | 内网无限带宽；不受 GitHub Rate Limit | 需要维护同步 CronJob |
| MinIO 对象存储 | 高可用（3副本）；低成本；S3 兼容 | 比本地磁盘多一层网络开销 |
| PostgreSQL 元数据 | 强大的查询能力；支持趋势分析 | 需要维护数据库实例 |
| 统一流水线 | 一次配置，全局生效 | 各组件依赖增加了系统复杂度 |

### 适用场景

1. **多地域部署**：各区域扫描节点共享全局缓存和统一 DB 源。
2. **内网隔离环境**（政务、金融、军工）：本地 DB 镜像 + Redis 缓存 = 完全离线运行。
3. **大规模定期扫描**（每日 > 500 次扫描）：MinIO 持久化确保历史报告不丢失。
4. **安全审计与合规**：PostgreSQL 提供结构化的扫描历史，支持任意时间跨度的查询。
5. **平台化建设**：三层存储是上层安全平台（Dashboard、告警、工单系统）的数据底座。

**不适用场景**：
1. 单机 + 扫描量 < 20 次/天——分布式存储的运维成本超过收益。
2. 严格无第三方依赖的环境（连 Redis/MinIO 都不能部署）——只能用本地文件缓存。

### 注意事项

- **Redis TTL 设置要合理**。太短（< 1 天），缓存命中率低；太长（> 30 天），内存可能不够。推荐 7-14 天。
- **DB 同步需要校验完整性**。同步脚本中应加入 `oras manifest fetch` 验证远程镜像完整，避免推送损坏的 DB 镜像到内网。
- **MinIO 启用版本控制**。防止误覆盖报告；同时配置生命周期策略，自动清理超过 3 年的报告。
- **PostgreSQL 适时清理**。`scan_records` 表随时间增长，建议按季度分表或配置自动归档。

### 常见踩坑经验

**踩坑案例 1：Redis 内存打满导致写入失败**
- **现象**：Trivy 日志出现 `OOM command not allowed when used memory > 'maxmemory'`。
- **根因**：未设置 Redis 的 maxmemory-policy。默认 `noeviction` 策略在内存满时拒绝所有写入。
- **解法**：设置 `maxmemory-policy allkeys-lru` 自动淘汰最少使用的 key；同时增大 Redis 内存或增加节点。

**踩坑案例 2：trivy-db 版本不匹配导致扫描结果异常**
- **现象**：内网节点扫描结果与 GitHub 上显示的漏洞数量不一致。
- **根因**：DB 同步 CronJob 失败了 3 天但没人注意，内网节点一直在用过期的 DB。
- **解法**：在 DB 同步脚本中增加健康检查——推送后立即 `trivy image --db-repository <内网> test:latest` 验证；加告警通知。

**踩坑案例 3：MinIO 对象名冲突导致报告被覆盖**
- **现象**：同一天连续两次扫描同一镜像的最新版本，第二次的报告覆盖了第一次的。
- **根因**：对象名没有唯一标识（只有 `date/image/`），当 `latest` 标签移动后，新旧版本的同名报告冲突。
- **解法**：在对象名中嵌入镜像 digest（SHA256）的前 12 位：`YYYY/MM/DD/image_digestShort/report.json`。

### 思考题

1. 在你的三层存储架构中，如果 Redis 集群宕机超过 24 小时，Trivy 能继续正常工作吗？扫描结果会变差吗？请设计一个「Redis 不可用时的降级方案」——确保扫描 100% 可行，即使速度变慢。

2. 你的公司有 10 个 K8s 集群，每个集群 3 个 Trivy 节点，总共 30 个节点。请设计一个「DB 镜像的逐级分发」方案：外网 → 总部 Harbor → 区域 Harbor → 集群级 Harbor → 节点，确保每一级都有缓存、每次 DB 更新都能在 5 分钟内传播到所有节点。
