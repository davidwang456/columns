# 第4章：文件系统与代码仓库扫描实战

> 版本：Trivy v0.50+
> 面向人群：开发、运维、测试

---

## 1. 项目背景

### 业务场景

云帆科技的安全事件再次发生。这次不是线上镜像出问题，而是后端工程师阿杰在 GitHub 上不小心提交了一个包含数据库密码的 `config.properties` 文件。虽然 5 分钟后他就 force push 删除了，但密码已经被 GitHub 的公开 API 缓存，当天晚上公司的测试数据库就遭到了暴力破解。

CTO 紧急召集会议：「我们必须把安全扫描左移到开发阶段！等镜像构建完再扫描，密钥已经泄露了；等发版后再修漏洞，攻击者已经进来了。我要每个开发者在提交代码前，就能发现自己引入了漏洞或泄露了密钥。」

老李接下了这个任务，但遇到了现实的阻碍：

- 开发小王说：「我本地没有 Docker，怎么扫镜像？」
- 前端小张说：「我的项目是纯静态页面，没有容器化计划，是不是就扫不了？」
- 测试组更关心：「我们能不能直接扫描 Git 仓库，不用每个人都 clone 代码到本地？」

### 痛点放大

**第一，安全扫描与开发工作流脱节。** 很多团队把 Trivy 当成「运维工具」，只在 CI 构建镜像时跑一遍。但开发阶段引入的漏洞、泄露的密钥、不合规的依赖，如果在写代码时就被发现，修复成本几乎为零；如果拖到生产环境才发现，修复成本可能暴涨 100 倍。

**第二，非容器项目的安全盲区。** 不是每个项目都有 Dockerfile。公司的前端项目、Python 脚本、内部工具库，可能永远不需要打包成镜像。如果安全扫描只覆盖容器，这些项目就成了「法外之地」。

**第三，Git 历史中的秘密残留。** 开发以为 `git rm` 或 force push 就能删掉敏感信息，但 Git 的提交历史里永远留着记录。除非专门扫描 Git 历史，否则这些「已删除」的密钥就是定时炸弹。

**第四，扫描范围不可控。** 项目里的 `node_modules/`、`vendor/`、测试数据目录，往往包含大量第三方依赖的漏洞噪音。如果不加过滤，一次扫描输出几百条结果，开发人员直接选择无视。

**本章的核心目标是：让 Trivy 走出容器，深入开发工作流——在文件系统、Git 仓库、甚至 pre-commit 阶段建立安全防线。**

## 2. 项目设计

**场景**：云帆科技的开发室，老李正在向小王和小张演示如何把 Trivy 集成到日常开发中。

---

**小胖**：「老李，你说要把扫描左移到开发阶段。可我本地连 Docker 都没装，怎么扫？」

**小白**：「这就是 `trivy fs` 存在的意义。你不需要 Docker，不需要镜像，直接扫你电脑上的代码目录就行。Trivy 会分析你项目里的 `package-lock.json`、`pom.xml`、`go.mod`，匹配漏洞数据库，跟扫镜像里的依赖是一个原理。」

**小胖**：「哦！那跟 `npm audit` 有什么区别？我跑 `npm audit` 也能发现漏洞啊。」

**大师**：「`npm audit` 只能扫 npm 生态的漏洞。如果你的项目同时用了 npm、Python pip、还有一个内部的 Go 工具，`npm audit` 只能看到三分之一。Trivy 是『全能体检』，一份报告覆盖所有语言的依赖漏洞，再加上密钥泄露和配置错误检查。你可以把它理解为 `npm audit` + `gitleaks` + `checkov` 的合体版。」

**小胖**：「这么强？那 Git 仓库呢？我想扫一个 GitHub 上的开源项目，不想 clone 到本地。」

**小白**：「用 `trivy repo`。比如 `trivy repo https://github.com/yourcompany/backend-api`，Trivy 会远程拉取仓库内容，直接在内存里分析，不需要你本地有代码副本。」

**大师**：「技术映射：`trivy fs` 是『家庭医生上门体检』，医生带着设备到你家（本地目录）检查；`trivy repo` 是『远程诊所初筛』，你把病历传过去（Git URL），医生看完给你发报告。两者诊断能力一样，只是样本获取方式不同。」

**小胖**：「那密钥检测呢？我听说阿杰上次把密码提交到 GitHub 了，Trivy 能扫出来？」

**小白**：「不仅能扫出来，还能扫 Git 历史。你执行 `trivy fs --scanners secret /path/to/repo`，Trivy 会遍历目录下的所有文件，用正则表达式和熵值算法识别 AWS Key、GitHub Token、私钥、数据库连接串等敏感信息。更厉害的是加上 `--scan-git-history`，它会深入 `.git` 目录，检查每一次提交中是否曾经出现过密钥。」

**小胖**：「那岂不是所有开发的历史『黑料』都会被翻出来？」

**大师**：「正是如此。很多团队以为『删了文件就安全了』，但 Git 历史是只增不减的。Trivy 的 `--scan-git-history` 就是专门用来『翻旧账』的。我建议每个团队在新引入 Trivy 时，都先用这个模式扫一遍所有仓库，把历史遗留的密钥全部清理掉——该轮换的轮换，该 revoke 的 revoke。」

**小白**：「还有一个问题。我前端项目里的 `node_modules/` 有 3 万多个文件，每次 `trivy fs` 都要扫很久，而且报出来的漏洞大部分是我无法控制的深层依赖。怎么减少噪音？」

**大师**：「三个策略：

1. **排除目录**：用 `--skip-dirs node_modules/ --skip-dirs .git/` 跳过不需要分析的大目录。Trivy 对语言包的识别主要依赖锁定文件（`package-lock.json`、`yarn.lock`），不需要逐文件扫描 `node_modules/`。
2. **基线管理**：创建 `.trivyignore` 文件，把已评估接受的漏洞写进去。后续扫描会自动过滤，只显示新增风险。
3. **精准扫描**：如果只关心密钥和配置错误，可以用 `--scanners secret,misconfig` 关闭漏洞扫描，速度提升 5 倍以上。」

**小胖**：「听起来 `trivy fs` 比 `trivy image` 更适合日常开发。那我能不能在每次 `git commit` 之前自动触发扫描？」

**小白**：「当然可以。用 pre-commit hook 或者 Husky（前端项目常用）。开发人员每次提交代码前，Trivy 自动扫一遍，发现密钥或 Critical 漏洞就直接阻止提交。」

**大师**：「技术映射：pre-commit hook 就像机场的安检门。你不通过安检（扫描），就上不了飞机（提交代码）。这是把安全控制『嵌入』到开发肌肉记忆里的最佳方式。」

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+，已安装并配置缓存
- **Git**：已安装，用于仓库扫描和历史分析
- **测试项目**：一个模拟的 Java Spring Boot 项目，包含已知漏洞和故意植入的密钥

### 步骤一：准备一个含漏洞和密钥的测试项目

**目标**：创建一个真实的业务项目结构，用于验证文件系统扫描。

```bash
# 创建测试项目目录
mkdir -p trivy-fs-demo/src/main/java/com/example/demo
mkdir -p trivy-fs-demo/src/main/resources
cd trivy-fs-demo

# 初始化 Git 仓库
git init

# 创建一个包含旧版 Log4j 的 pom.xml（模拟 Log4Shell 漏洞）
cat > pom.xml << 'EOF'
<?xml version="1.0" encoding="UTF-8"?>
<project>
  <modelVersion>4.0.0</modelVersion>
  <groupId>com.example</groupId>
  <artifactId>demo</artifactId>
  <version>1.0</version>
  <dependencies>
    <dependency>
      <groupId>org.apache.logging.log4j</groupId>
      <artifactId>log4j-core</artifactId>
      <version>2.14.1</version>
    </dependency>
    <dependency>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-web</artifactId>
      <version>2.6.0</version>
    </dependency>
  </dependencies>
</project>
EOF

# 创建一个包含密钥的配置文件
cat > src/main/resources/config.properties << 'EOF'
db.url=jdbc:mysql://localhost:3306/prod
db.user=admin
db.password=SuperSecret123!
aws.access.key=AKIAIOSFODNN7EXAMPLE
aws.secret.key=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
EOF

# 创建应用入口
cat > src/main/java/com/example/demo/Application.java << 'EOF'
package com.example.demo;
public class Application {
    public static void main(String[] args) {
        System.out.println("Demo app running");
    }
}
EOF

# 提交初始版本
git add .
git commit -m "Initial commit with vulnerable deps and secrets"
```

### 步骤二：执行文件系统漏洞扫描

**目标**：扫描本地项目，发现依赖漏洞。

```bash
# 在项目目录的父级执行
cd ..
trivy fs --severity HIGH,CRITICAL trivy-fs-demo
```

**预期输出**（截取关键部分）：
```
Java (jar)
==========
Total: 2 (HIGH: 1, CRITICAL: 1)

┌─────────────────────────────────┬────────────────────┬──────────┬────────┬───────────────────┐
│            Library              │   Vulnerability    │ Severity │ Status │   Fixed Version   │
├─────────────────────────────────┼────────────────────┼──────────┼────────┼───────────────────┤
│ org.apache.logging.log4j:log4j… │ CVE-2021-44228     │ CRITICAL │ fixed  │ 2.15.0            │
│ org.springframework.boot:spri…  │ CVE-2022-27772     │ HIGH     │ fixed  │ 2.6.6             │
└─────────────────────────────────┴────────────────────┴──────────┴────────┴───────────────────┘
```

**关键观察**：
- Trivy 通过解析 `pom.xml` 发现了 `log4j-core:2.14.1`，正确匹配到 `CVE-2021-44228`（Log4Shell）。
- 状态为 `fixed`，修复版本为 `2.15.0`，意味着升级依赖即可消除漏洞。

### 步骤三：执行密钥检测扫描

**目标**：发现代码目录中泄露的敏感信息。

```bash
trivy fs --scanners secret trivy-fs-demo
```

**预期输出**（截取关键部分）：
```
trivy-fs-demo/src/main/resources/config.properties (secrets)
============================================================
Total: 2 (UNKNOWN: 0, LOW: 0, MEDIUM: 0, HIGH: 1, CRITICAL: 1)

┌─────────────┬──────────────────────┬──────────┬────────────┬─────────────────────────────────────┐
│   Rule      │        Category      │ Severity │ Confidence │                Match                │
├─────────────┼──────────────────────┼──────────┼────────────┼─────────────────────────────────────┤
│ AWS Access… │   Cloud Provider     │ CRITICAL │    High    │ aws.access.key=AKIAIOSFODNN7EXAMPLE │
│ AWS Secret… │   Cloud Provider     │ HIGH     │    High    │ aws.secret.key=wJalrXUtnFEMI/K7M…  │
└─────────────┴──────────────────────┴──────────┴────────────┴─────────────────────────────────────┘
```

> **可能遇到的坑**：如果项目中没有明显的密钥格式，Trivy 可能输出为空。确保测试文件中的密钥符合标准格式（如 AWS AKIA 开头、符合熵值阈值）。

### 步骤四：扫描 Git 提交历史

**目标**：验证 Trivy 能否发现已删除但留在 Git 历史中的密钥。

```bash
# 先删除配置文件中的密钥，模拟「发现问题后紧急删除」
cd trivy-fs-demo
cat > src/main/resources/config.properties << 'EOF'
db.url=jdbc:mysql://localhost:3306/prod
db.user=admin
db.password=${DB_PASSWORD}
aws.access.key=${AWS_KEY}
aws.secret.key=${AWS_SECRET}
EOF

git add .
git commit -m "Remove hardcoded secrets"

# 扫描当前文件系统（应该没有密钥了）
trivy fs --scanners secret .

# 扫描 Git 历史（应该能发现之前提交的密钥）
trivy fs --scanners secret --scan-git-history .
```

**预期结果**：
- 普通 `trivy fs`：Total 为 0（当前文件已无密钥）。
- 带 `--scan-git-history`：仍然能发现历史提交中的 AWS Key 和 Secret Key。

### 步骤五：远程仓库扫描

**目标**：无需 clone，直接扫描远程 Git 仓库。

```bash
# 扫描一个公开的已知有漏洞的仓库（示例）
trivy repo --severity HIGH,CRITICAL https://github.com/aquasecurity/trivy-ci-test
```

**参数解读**：
- `repo` 子命令会克隆仓库到临时目录，分析后自动清理。
- 支持私有仓库：通过 `GITHUB_TOKEN` 环境变量认证。

```bash
export GITHUB_TOKEN=ghp_xxxxxxxxxxxx
trivy repo --severity HIGH,CRITICAL https://github.com/yourorg/private-repo
```

### 步骤六：配置 pre-commit Hook

**目标**：在开发人员提交代码前自动触发扫描，阻断密钥和 Critical 漏洞。

在项目根目录创建 `.git/hooks/pre-commit`：

```bash
#!/bin/bash
# pre-commit hook for Trivy scanning

echo "Running Trivy secret scan..."
trivy fs --scanners secret --exit-code 1 .

if [ $? -ne 0 ]; then
    echo "ERROR: Trivy found secrets in your commit. Please remove them before committing."
    exit 1
fi

echo "Running Trivy vulnerability scan (CRITICAL only)..."
trivy fs --scanners vuln --severity CRITICAL --exit-code 1 .

if [ $? -ne 0 ]; then
    echo "ERROR: Critical vulnerabilities found. Please fix or update .trivyignore with justification."
    exit 1
fi

echo "Trivy scan passed."
```

赋予执行权限：

```bash
chmod +x .git/hooks/pre-commit
```

**测试 Hook**：

```bash
# 尝试提交一个包含新密钥的文件
echo "api_key=sk-1234567890abcdef" > secret.txt
git add secret.txt
git commit -m "Test pre-commit hook"
```

**预期结果**：提交被阻止，终端显示 `ERROR: Trivy found secrets in your commit.`。

> **可能遇到的坑**：pre-commit hook 只在当前仓库生效，每个开发者都需要手动配置。如果想统一管理，可使用 [pre-commit 框架](https://pre-commit.com/)，将 Trivy 配置写入 `.pre-commit-config.yaml`，团队成员只需执行 `pre-commit install` 即可。

### 步骤七：排除目录与基线管理

**目标**：减少扫描噪音，建立团队基线。

创建 `.trivyignore`：

```
# 忽略测试框架中的已知漏洞（开发依赖，不进入生产）
CVE-2021-12345

# 忽略特定路径的密钥误报（测试数据中的假密钥）
path:src/test/resources/mock-aws-credentials.properties
```

创建 `trivy.yaml`：

```yaml
severity:
  - HIGH
  - CRITICAL
scanners:
  - vuln
  - secret
skip-dirs:
  - ./target
  - ./node_modules
  - ./.git
exit-code: 1
```

执行扫描：

```bash
trivy fs --config trivy.yaml .
```

### 测试验证

1. 执行 `trivy fs --severity HIGH,CRITICAL trivy-fs-demo`，确认发现 Log4j 漏洞。
2. 执行 `trivy fs --scanners secret trivy-fs-demo`，确认发现 AWS Key 泄露。
3. 执行带 `--scan-git-history` 的扫描，确认能发现历史提交中的密钥。
4. 触发 pre-commit hook，确认包含密钥的提交被阻止。
5. 配置 `.trivyignore` 后重新扫描，确认被忽略的条目不再出现在报告中。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 扫描范围 | 无需 Docker，直接扫本地代码和远程仓库 | 远程仓库扫描需要网络权限，大仓库耗时较长 |
| 密钥检测 | 支持文件系统和 Git 历史，覆盖主流密钥格式 | 自定义密钥格式需要编写额外规则 |
| 集成能力 | 天然支持 pre-commit、CI 流水线 | pre-commit hook 需要每个开发者手动配置 |
| 多语言覆盖 | 一份命令覆盖 Java/Node/Go/Python 等 | 对无锁定文件的项目（如纯 pip）识别率下降 |
| 误报控制 | `.trivyignore` 和 skip-dirs 可有效降噪 | 初学者容易过度忽略，导致漏报 |

### 适用场景

1. **开发阶段安全自检**：开发者在本地用 `trivy fs` 扫描项目，在提交前修复漏洞和密钥泄露。
2. **密钥泄露应急响应**：发现密钥泄露后，用 `--scan-git-history` 检查所有历史提交，评估泄露范围。
3. **开源项目安全审计**：安全团队用 `trivy repo` 快速评估第三方依赖的风险，无需 clone 整个仓库。
4. **CI 前置门禁**：在代码合并前（Pre-Merge）执行 `trivy fs`，阻断包含密钥或 Critical 漏洞的 PR。
5. **非容器项目安全覆盖**：前端、脚本、工具库等没有 Dockerfile 的项目，通过文件系统扫描补齐安全短板。

**不适用场景**：
1. 需要扫描编译后的二进制文件内部漏洞（如静态链接的 C 库）——Trivy 主要依赖包管理器清单，对无元数据的二进制分析能力有限。
2. 需要深度代码逻辑分析（如 SQL 注入、XSS）——Trivy 不是 SAST 工具，不做代码流分析。

### 注意事项

- **锁定文件的重要性**：Trivy 对语言包的识别高度依赖 `package-lock.json`、`yarn.lock`、`go.sum`、`pom.xml` 等锁定文件。确保这些文件已提交到 Git，且与生产部署的依赖一致。
- **Git 历史扫描的性能**：`--scan-git-history` 会遍历所有提交，对大仓库可能耗时数分钟。建议在 CI 中定期执行（如每周一次），而不是每次提交都跑。
- **密钥规则的更新**：Trivy 内置的密钥规则库会随版本更新。如果发现新型密钥格式漏报，可通过 `trivy-secret.yaml` 自定义规则。

### 常见踩坑经验

**踩坑案例 1：扫描结果为空，但项目明显有漏洞**
- **现象**：`trivy fs` 输出 `Total: 0`。
- **根因**：项目中没有锁定文件（如只有 `package.json` 没有 `package-lock.json`），Trivy 无法确定依赖的确切版本。
- **解法**：确保提交锁定文件到 Git；对于 Python 项目，使用 `pip freeze > requirements.txt` 生成精确版本清单。

**踩坑案例 2：pre-commit hook 执行太慢，开发抱怨**
- **现象**：每次提交都要等 30 秒以上。
- **根因**：Trivy 默认全量扫描所有 Scanner，且没有排除 `node_modules/` 等大目录。
- **解法**：在 hook 中限制 Scanner（如只扫 secret 和 CRITICAL vuln），并添加 `--skip-dirs` 排除无需扫描的目录。

**踩坑案例 3：远程仓库扫描私有项目报错 404**
- **现象**：`trivy repo https://github.com/org/private-repo` 报 `authentication required`。
- **根因**：Trivy 默认使用匿名方式拉取仓库，私有项目需要认证。
- **解法**：设置 `GITHUB_TOKEN` 环境变量（需有 `repo` 权限），或在 URL 中嵌入 token（不推荐，会留痕）。

### 思考题

1. 假设你的团队有 100 个 Git 仓库，如何设计一个自动化方案，定期扫描所有仓库的 Git 历史中的密钥泄露，而不需要手动 clone 每一个仓库？
2. `trivy fs` 和 `trivy image` 对一个同时包含 `pom.xml` 和 Dockerfile 的 Java 项目，扫描结果会有什么差异？在什么场景下应该同时使用两者？

> **答案提示**：第 17 章「GitHub Actions 集成与 Pull Request 安全门禁」将介绍如何在 CI 中自动化远程仓库扫描和 pre-commit 的统一管理。

> **推广计划**：本章是开发团队的必读内容。建议每个项目仓库都配置 `trivy.yaml` 和 `.trivyignore`，并将 pre-commit hook（或 pre-commit 框架配置）写入项目模板。安全团队应每月用 `--scan-git-history` 对所有仓库进行一次「历史密钥大扫除」。运维同学可将 `trivy fs` 纳入开发环境初始化脚本，确保新入职员工的第一天就能用上安全扫描。
