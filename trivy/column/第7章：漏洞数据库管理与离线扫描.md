# 第7章：漏洞数据库管理与离线扫描

> 版本：Trivy v0.50+
> 面向人群：运维、安全工程师、架构师

---

## 1. 项目背景

### 业务场景

云帆科技最近接了一个金融行业的私有化部署项目。客户的核心交易系统运行在一个完全物理隔离的内网环境中，连一条能访问外网的专线都没有。合同里的等保条款明确要求：「部署前必须完成所有镜像和依赖组件的漏洞扫描，并提供完整的安全报告。」

老李带着 Trivy 进了客户机房，兴致勃勃地执行了第一条扫描命令，结果卡在 `Downloading vulnerability database...` 整整二十分钟，最后报了个 `connection timeout`。客户的信息安全负责人站在旁边，面无表情地问：「你们不是说这个工具能离线用吗？」

老李满头大汗。他确实记得 Trivy 支持离线扫描，但从来没真正操作过。之前在公司里都是「即开即用」，Trivy 自动从 GitHub 下载数据库，他根本没关心过数据库长什么样、有多大、怎么搬运。

与此同时，公司的 CI/CD 流水线也遇到了数据库相关的性能瓶颈。每天两百次镜像构建，每次构建都要触发 Trivy 扫描，而扫描的第一步就是检查数据库是否需要更新。GitHub Releases 的下载速度不稳定，高峰期经常只有 50KB/s，导致原本 30 秒的扫描变成了 10 分钟的等待。开发群里开始刷屏抱怨：「Trivy 又卡住了，能不能关掉自动更新？」

### 痛点放大

**第一，离线环境是 enterprise 的硬门槛。** 金融、政府、军工、能源等行业的生产环境几乎都处于物理隔离或逻辑强隔离状态。没有外网，Trivy 的默认工作流就直接瘫痪。很多团队因为这个原因被迫放弃 Trivy，转而选择自带离线数据库的商业产品——尽管那些产品的扫描引擎可能还不如 Trivy。

**第二，数据库更新的「惊群效应」。** 默认情况下，Trivy 的每次扫描都会先检查数据库是否过期（超过 12 小时）。如果 50 个 Jenkins Slave 同时启动扫描，它们会同时向 GitHub 发起数据库下载请求，造成带宽争抢和速率限制。GitHub 对未认证用户的 API 速率限制是每小时 60 次，大规模 CI 场景下很容易触发。

**第三，数据库版本不一致导致扫描结果不可比。** 开发小王早上扫了一个镜像，报告里有 10 个漏洞；运维老李下午扫同一个镜像，报告里变成了 12 个——因为下午的 `trivy-db` 更新了两个新入库的 CVE。团队在安全评审会议上拿着两份不同的报告争吵，根本原因在于没有统一的数据库版本管理。

**第四，Java 项目的扫描被「java-db」拖累。** 很多团队不知道 Trivy 对 Maven 依赖的漏洞检测需要额外的 `java-db`（约 500MB）。首次扫描 Java 镜像时，Trivy 会突然开始下载一个巨大的数据库，而官方文档对这个过程的描述非常简略，新手往往以为工具「卡死了」。

**本章的核心目标是：掌握 Trivy 漏洞数据库的完整生命周期管理——下载、缓存、同步、离线搬运、版本控制——让 Trivy 在任何网络环境下都能稳定运行。**

## 2. 项目设计

**场景**：云帆科技的运维室，老李、小胖和小白正在讨论如何在内网环境和 CI 流水线中优化 Trivy 的数据库策略。

---

**小胖**：「老李，客户在机房里瞪着你的时候，你当时咋想的？『离线扫描』不就是断网也能扫吗？」

**老李**：「我一开始确实以为 Trivy 自带漏洞库，像杀毒软件的本地病毒库一样。结果发现它默认要从 GitHub 下载，而且一更新就是几百兆。」

**小白**：「这其实是个设计取舍。Trivy 把漏洞数据库独立出来，本体只有几十兆，数据库可以每天更新。这样你不用每次升级 Trivy 二进制，就能拿到最新的漏洞数据。但代价就是首次使用必须有网。」

**大师**：「技术映射：你可以把 Trivy 二进制想象成『体检仪器』，把 `trivy-db` 想象成『疾病百科全书』。仪器本身不会变，但医学知识每天都在更新。所以仪器启动时要先问一句：『百科全书是不是最新版？』如果不是，就去图书馆（GitHub）借一本新的。」

**小胖**：「那内网没有图书馆怎么办？自己带书进去？」

**大师**：「对，这就是离线扫描的核心思路。Trivy 的数据库本质上就是几个放在 `~/.cache/trivy/db/` 目录下的文件。你可以在外网机器上执行 `trivy image --download-db-only`，把这些文件打包，通过 U 盘、堡垒机文件传输、或者内部 OCI Registry 搬运到内网。内网的 Trivy 加上 `--skip-db-update` 参数，就会直接用本地缓存，不再尝试连外网。」

**小白**：「但搬运文件太原始了。我们 CI 流水线有 20 个节点，难道每次更新数据库都要运维手动复制 20 份？」

**大师**：「当然不用。Aqua 提供了一个更优雅的方案：把数据库托管到自己的 OCI Registry（如 Harbor、Docker Hub、AWS ECR）上。Trivy 支持 `--db-repository` 参数指定自定义的数据库镜像地址。你只需要在外网机器上定期把最新数据库 push 到内网的 Registry，所有扫描节点从内部 Registry 拉取，既避开了 GitHub 的速率限制，又实现了集中化更新。」

**小胖**：「哦！就像公司内部建了个『图书馆分馆』，所有员工去分馆借书，不用都跑总馆？」

**大师**：「技术映射：完全正确。OCI Registry 作为数据库分发介质，解决了三个问题：

1. **网络隔离**：内网节点只访问内网 Registry，不需要外网权限。
2. **带宽优化**：Registry 有缓存和 CDN 能力，多台节点同时拉取也不会冲击单一出口。
3. **版本可控**：你可以决定什么时候 push 新版本数据库，甚至保留历史版本，方便回滚和审计。」

**小白**：「那 `java-db` 呢？我之前扫 Java 镜像时，它突然开始下载一个 500MB 的东西，差点把 Jenkins Slave 的磁盘撑爆。」

**大师**：「`java-db` 是 Trivy 专门给 Maven Central 准备的索引。Java 的生态太复杂了，同一个 `groupId:artifactId` 可能对应几百个版本，Trivy 需要这个数据库来建立『Java 包 → CVE』的映射。但它确实很大，而且默认也是走 GitHub 下载。你可以用和 `trivy-db` 同样的方式管理：预下载、Registry 托管、或者对非 Java 项目干脆不启用 Java 扫描。」

**老李**：「还有一个问题。我们团队经常因为数据库版本不一致吵起来。上午开发和下午运维扫同一个镜像，漏洞数量不一样。有没有办法锁定数据库版本？」

**小白**：「可以在 CI 中显式指定数据库的下载时间戳，或者干脆把数据库文件打包成 Docker 镜像，作为 CI 的基础镜像层。这样所有扫描节点使用的是同一个『快照』。」

**大师**：「更标准的做法是『数据库即代码』。把 `trivy.db` 的 SHA256 校验值记录在项目的 `trivy.lock` 文件中，CI 扫描前检查本地数据库的校验值是否匹配。如果不匹配，从公司内部 Registry 拉取对应版本。这保证了『同一代码、同一数据库、同一结果』的可复现性。」

**小胖**：「听起来数据库管理是个系统工程啊。那我们今天定个方案吧：公司内外网、CI 流水线、开发机，分别用什么策略？」

**大师**：「我画张表：

| 场景 | 网络环境 | 推荐策略 |
|------|----------|----------|
| 开发机 | 有外网 | 默认自动更新，本地缓存 |
| CI 流水线 | 有外网 | 自托管 Registry + 定时同步，避免 GitHub 限速 |
| 内网机房 | 无外网 | Registry 代理或预置数据库镜像 + `--skip-db-update` |
| 离线交付 | 完全隔离 | U 盘/光盘搬运数据库文件，版本锁定 |

关键是：不要让每个节点都直接访问 GitHub，要把数据库当成一种『基础设施』来管理。」

## 3. 项目实战

### 环境准备

- **Trivy**：v0.50+，两台机器（一台模拟外网、一台模拟内网）
- **OCI Registry**：内网 Harbor 或 Docker Registry（可选，用于进阶方案）
- **存储**：足够存放 `trivy.db`（~300MB）和 `java-db`（~500MB）的磁盘空间

### 步骤一：了解数据库的存储结构和更新机制

**目标**：理解 Trivy 默认缓存目录的内容和数据库元数据。

```bash
# 查看 Trivy 缓存目录结构
ls -lah ~/.cache/trivy/
```

**预期输出**：
```
drwxr-xr-x  5 user user 4.0K Mar 15 10:00 .
drwxr-xr-x  3 user user 4.0K Mar 15 09:00 db
drwxr-xr-x  2 user user 4.0K Mar 15 09:00 fanal
```

```bash
# 查看数据库文件
ls -lah ~/.cache/trivy/db/
```

**预期输出**：
```
-rw-r--r-- 1 user user 280M Mar 15 09:00 trivy.db
-rw-r--r-- 1 user user 520M Mar 15 09:00 trivy-java.db
-rw-r--r-- 1 user user  256 Mar 15 09:00 metadata.json
```

查看元数据：

```bash
cat ~/.cache/trivy/db/metadata.json
```

**预期输出**：
```json
{
  "Version": 2,
  "NextUpdate": "2024-03-16T00:00:00Z",
  "UpdatedAt": "2024-03-15T12:00:00Z",
  "DownloadedAt": "2024-03-15T09:00:00Z"
}
```

**字段解读**：
- `Version`：数据库 Schema 版本。
- `UpdatedAt`：数据库内容的实际更新时间（数据源的同步时间）。
- `NextUpdate`：Trivy 建议的下次更新时间（默认 12 小时后）。
- `DownloadedAt`：本地下载时间。

### 步骤二：外网预下载数据库并打包

**目标**：在外网机器上下载完整数据库，为搬运到内网做准备。

```bash
# 仅下载漏洞数据库，不执行扫描
trivy image --download-db-only

# 同时下载 Java 数据库（如果团队需要扫描 Java 镜像）
trivy image --download-java-db-only

# 确认文件已下载
ls -lh ~/.cache/trivy/db/

# 打包数据库文件
tar czvf trivy-db-backup-$(date +%Y%m%d).tar.gz -C ~/.cache/trivy/db .
```

**预期输出**：
```
trivy-db-backup-20240315.tar.gz
```

> **可能遇到的坑**：如果 GitHub 访问慢，可以设置代理或镜像。国内用户可以通过 `https://ghproxy.com/` 等代理加速，但 Trivy 默认的数据库下载 URL 是硬编码的。更好的方案是自托管 Registry（见步骤四）。

### 步骤三：内网离线环境导入数据库

**目标**：在完全无外网的机器上，导入预下载的数据库并执行扫描。

```bash
# 将打包文件通过 U 盘/堡垒机传输到内网机器
# 解压到 Trivy 缓存目录
mkdir -p ~/.cache/trivy/db/
tar xzvf trivy-db-backup-20240315.tar.gz -C ~/.cache/trivy/db/

# 验证数据库文件存在
ls -lh ~/.cache/trivy/db/

# 执行离线扫描（必须加 --skip-db-update，否则会尝试联网更新）
trivy image --skip-db-update --severity HIGH,CRITICAL alpine:3.14
```

**预期输出**：
```
alpine:3.14 (alpine 3.14.10)
=============================
Total: 5 (HIGH: 3, CRITICAL: 2)
...（漏洞列表）...
```

**关键参数**：
- `--skip-db-update`：禁止 Trivy 检查数据库更新，完全使用本地缓存。
- 如果缺少 `--skip-db-update`，Trivy 会尝试连接 GitHub，在离线环境中导致超时或报错。

### 步骤四：通过 OCI Registry 托管数据库（企业级方案）

**目标**：搭建内部数据库分发通道，让内网节点自动从内部 Registry 拉取更新。

**在外网机器上推送数据库到 Harbor**：

```bash
# 安装 oras 工具（OCI Registry 客户端）
curl -LO https://github.com/oras-project/oras/releases/download/v1.1.0/oras_1.1.0_linux_amd64.tar.gz
tar xzvf oras_1.1.0_linux_amd64.tar.gz -C /usr/local/bin/ oras

# 登录 Harbor
oras login harbor.cloud-sail.internal

# 将本地数据库作为 OCI Artifact 推送到 Harbor
oras push harbor.cloud-sail.internal/security/trivy-db:latest \
  ~/.cache/trivy/db/trivy.db:application/vnd.aquasec.trivy.db.layer.v1.tar+gzip \
  ~/.cache/trivy/db/metadata.json:application/vnd.aquasec.trivy.db.config.v1+json

# 同样推送 java-db
oras push harbor.cloud-sail.internal/security/trivy-java-db:latest \
  ~/.cache/trivy/db/trivy-java.db:application/vnd.aquasec.trivy.java-db.layer.v1.tar+gzip
```

**在内网机器上配置 Trivy 使用内部 Registry**：

```bash
# 方式一：命令行参数
trivy image --db-repository harbor.cloud-sail.internal/security/trivy-db \
  --java-db-repository harbor.cloud-sail.internal/security/trivy-java-db \
  alpine:3.14

# 方式二：环境变量
export TRIVY_DB_REPOSITORY=harbor.cloud-sail.internal/security/trivy-db
export TRIVY_JAVA_DB_REPOSITORY=harbor.cloud-sail.internal/security/trivy-java-db
trivy image alpine:3.14
```

**预期效果**：
- 内网 Trivy 不再访问 GitHub，而是从内部 Harbor 拉取数据库。
- 拉取速度由内部网络带宽决定，通常比外网快 10 倍以上。
- Harbor 可以配置缓存策略、访问控制和版本保留策略。

> **可能遇到的坑**：Harbor 默认不允许推送非镜像类型的 OCI Artifact。需要在 Harbor 的「配置管理」中开启「允许推送 Helm Chart、CNAB、OCI 索引等」选项。

### 步骤五：CI 流水线中的数据库缓存策略

**目标**：在 Jenkins/GitLab CI 中避免每次构建都重复下载数据库。

**GitLab CI 缓存配置示例**：

```yaml
variables:
  TRIVY_CACHE_DIR: "$CI_PROJECT_DIR/.trivy-cache"

stages:
  - security

trivy_scan:
  stage: security
  image: aquasec/trivy:0.50.0
  cache:
    key: trivy-db
    paths:
      - .trivy-cache/db/
    policy: pull-push
  script:
    - mkdir -p $TRIVY_CACHE_DIR
    - trivy image --cache-dir $TRIVY_CACHE_DIR --severity HIGH,CRITICAL $CI_REGISTRY_IMAGE:$CI_COMMIT_SHA
  only:
    - merge_requests
    - main
```

**策略解读**：
- `cache:key: trivy-db`：所有 Job 共享同一个缓存键。
- `cache:policy: pull-push`：首次运行 push 缓存，后续运行 pull 缓存。
- `TRIVY_CACHE_DIR`：将缓存目录指向项目内的 `.trivy-cache/`，以便 GitLab Runner 能正确归档。

**Jenkins Pipeline 示例**：

```groovy
pipeline {
    agent any
    environment {
        TRIVY_CACHE_DIR = '/var/cache/trivy'
    }
    stages {
        stage('Security Scan') {
            steps {
                sh '''
                    mkdir -p $TRIVY_CACHE_DIR
                    trivy image --cache-dir $TRIVY_CACHE_DIR \
                      --severity HIGH,CRITICAL \
                      --exit-code 1 \
                      myapp:${BUILD_NUMBER}
                '''
            }
        }
    }
}
```

### 步骤六：数据库版本锁定与可复现扫描

**目标**：确保同一镜像在不同时间、不同节点扫描的结果一致。

创建数据库版本锁定脚本 `lock-trivy-db.sh`：

```bash
#!/bin/bash
# 锁定 Trivy 数据库版本，确保团队使用同一快照

DB_DIR="${TRIVY_CACHE_DIR:-$HOME/.cache/trivy}/db"
LOCK_FILE="trivy-db.lock"

# 计算当前数据库的校验值
current_sha=$(sha256sum "$DB_DIR/trivy.db" | awk '{print $1}')

# 如果 lock 文件存在，检查是否匹配
if [ -f "$LOCK_FILE" ]; then
    expected_sha=$(cat "$LOCK_FILE")
    if [ "$current_sha" != "$expected_sha" ]; then
        echo "ERROR: trivy.db SHA256 mismatch!"
        echo "Expected: $expected_sha"
        echo "Current:  $current_sha"
        echo "Please run: trivy image --download-db-only"
        exit 1
    fi
else
    # 首次运行，写入 lock 文件
    echo "$current_sha" > "$LOCK_FILE"
    echo "Locked trivy.db version: $current_sha"
fi

# 输出数据库元数据
cat "$DB_DIR/metadata.json"
```

在 CI 中使用：

```bash
chmod +x lock-trivy-db.sh
./lock-trivy-db.sh
trivy image --skip-db-update myapp:latest
```

**收益**：
- 开发、测试、生产环境的扫描使用同一版本数据库。
- 安全评审会议上的报告不再因为数据库版本差异而「打架」。
- 问题追溯时，可以精确复现当时的扫描环境。

### 测试验证

1. 在外网机器执行 `trivy image --download-db-only`，确认 `~/.cache/trivy/db/trivy.db` 已生成。
2. 将数据库打包并导入内网机器，执行带 `--skip-db-update` 的扫描，确认漏洞报告正常输出。
3. 配置 `--db-repository` 指向内部 Harbor，确认 Trivy 从内部 Registry 拉取数据库。
4. 在 CI 中配置缓存，确认第二次扫描不再下载数据库，耗时缩短 80% 以上。
5. 运行 `lock-trivy-db.sh`，验证 SHA256 校验机制能检测数据库版本漂移。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 离线支持 | 数据库文件可直接搬运，支持完全隔离环境 | 需要运维额外维护数据库同步流程 |
| 更新机制 | 每日自动同步 NVD/GHSA/厂商通告，数据新鲜 | 首次下载体积大（~800MB 含 java-db），对弱网不友好 |
| Registry 托管 | 内网分发高效、可控、可审计 | 需要维护 Harbor 等 OCI Registry 基础设施 |
| CI 缓存 | 二次扫描极快，大幅降低构建耗时 | 缓存过期策略需要精心设计，避免脏缓存 |
| 版本锁定 | 保证扫描结果可复现、可对比 | 锁定后可能错过新发布的紧急漏洞通告 |

### 适用场景

1. **等保/密网环境**：金融、政府、军工等物理隔离内网的漏洞扫描需求。
2. **大规模 CI/CD 集群**：通过 Registry 托管或共享缓存，避免 GitHub 速率限制和带宽争抢。
3. **跨国/多地域团队**：在各地域部署 Registry 缓存节点，加速数据库分发。
4. **扫描结果审计**：锁定数据库版本，确保安全报告在法律或合规审计中可复现。
5. **Java 密集型团队**：预置 `java-db`，避免首次 Java 扫描时的长时间等待。

**不适用场景**：
1. 追求「零运维」的个人开发者——默认的自动更新模式已足够。
2. 需要实时漏洞情报（如 PoC 发布后分钟级检测）——Trivy 数据库有 12 小时的更新周期，无法满足极致的实时性。

### 注意事项

- **数据库的双重结构**：`trivy-db`（通用）和 `java-db`（Java 专用）是独立更新的，管理时不要遗漏 `java-db`。
- **Schema 版本升级**：Trivy 重大版本升级时，数据库 Schema 可能变化，需要同时更新 Trivy 二进制和数据库。
- **缓存清理策略**：`trivy clean --scan-cache` 不会删除数据库，只会删除镜像 layer 的 BLOB 缓存。如需清理数据库，需手动删除 `~/.cache/trivy/db/`。
- **OCI Artifact 的兼容性**：推送数据库到 Harbor 时，确保 Harbor 版本 >= 2.5，且已开启 OCI Artifact 支持。

### 常见踩坑经验

**踩坑案例 1：离线扫描时 `--skip-db-update` 漏加**
- **现象**：内网环境执行 `trivy image alpine:3.14`，卡住 10 分钟后报连接超时。
- **根因**：Trivy 默认行为是先检查数据库更新，离线环境无法连接 GitHub。
- **解法**：所有离线扫描命令都必须显式添加 `--skip-db-update`，或在 `trivy.yaml` 中配置 `skip-db-update: true`。

**踩坑案例 2：Java 镜像首次扫描极慢**
- **现象**：扫描一个 Spring Boot 镜像时，Trivy 在 `Downloading Java DB...` 阶段卡了 20 分钟。
- **根因**：`java-db` 约 500MB，首次扫描时必须下载。很多运维不知道这个数据库的存在。
- **解法**：在 CI 基础镜像中预置 `java-db`；或在非 Java 项目的扫描中禁用 Java 分析器（`--scanners vuln` 默认包含，但可通过过滤减少）。

**踩坑案例 3：CI 缓存导致数据库过期**
- **现象**：GitLab CI 的扫描结果比本地开发机少几个漏洞。
- **根因**：CI 缓存了 7 天前的 `trivy.db`，而本地开发机每天自动更新。
- **解法**：在 CI 缓存策略中设置 `policy: pull-push` 并定期（如每周）手动清除缓存；或在 CI Job 开始时执行 `trivy image --download-db-only` 强制更新（如果 CI 节点有外网）。

### 思考题

1. 假设你的公司在北京、上海、新加坡各有一个数据中心，每个中心有 30 台 CI 节点。请设计一套数据库分发架构，使得三个地域的节点都能在一分钟内获取到最新的 `trivy-db`，且总出口带宽消耗不超过每天 1GB。
2. Trivy 的数据库更新周期是 12 小时。如果在这 12 小时内出现了一个被广泛利用的 0-day 漏洞（如 Log4Shell 级别），你的团队如何在数据库正式更新前进行应急扫描？
