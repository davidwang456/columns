# 第3章：容器镜像分层扫描与漏洞原理

> 版本：Trivy v0.50+
> 面向人群：开发、运维、安全工程师

---

## 1. 项目背景

### 业务场景

云帆科技的开发组最近遇到了一件怪事。后端工程师阿杰构建了一个新版本的订单服务镜像，发版前用 Trivy 扫描，居然报出了 30 多个漏洞。阿杰很委屈：「我明明只改了三行业务代码，为什么漏洞比上个版本多了十几个？」

运维老李拿着两份扫描报告对比了半天，发现新增的漏洞全部来自基础镜像层。原来，阿杰上个版本用的是 `node:16-alpine3.14`，这次为了「跟上节奏」换成了 `node:18-alpine3.17`，以为版本号升高就更安全，殊不知新基础镜像里自带了一个有漏洞的 `libxml2` 版本。

更棘手的是，安全团队要求阿杰给出「每个漏洞出现在镜像的哪一层」。阿杰一脸茫然：「镜像不就是一层打包好的文件系统吗？还有分层？」

与此同时，测试组在扫描一个 Java 项目时发现了大量 CVE，但开发负责人坚称「我们的依赖版本已经是官方推荐的安全版本了」。双方争执不下，最后发现这些 Java 包是通过 `yum` 安装的，Trivy 优先匹配了 OS Vendor（RHEL）的安全通告，而开发看的是 Maven Central 的上游版本号——同一个 CVE，两个来源的 Fixed Version 完全不同。

### 痛点放大

**第一，开发者不理解镜像分层。** 很多人认为镜像就是一个「压缩包」，改一行代码重新构建，整个镜像就全变了。实际上 Docker 镜像是多层叠加的（Layer），每一层对应 Dockerfile 里的一条指令。基础镜像的漏洞会逐层继承到最终镜像，而业务代码层通常只影响应用依赖。

**第二，漏洞来源无法追溯。** 当扫描报出 50 个漏洞时，团队不知道哪些来自基础镜像、哪些来自应用依赖、哪些来自构建工具残留。没有分层视角，修复就变成「盲人摸象」。

**第三，OS Vendor Backport 导致认知冲突。** 开发习惯看 NVD（美国国家漏洞数据库）或 GitHub Security Advisory 的上游版本号，而 Trivy 对 OS 包优先采用厂商自己的安全通告。Debian 给 OpenSSL 打了个补丁，版本号从 `1.1.1n` 变成 `1.1.1n-0+deb11u5`，开发以为没修复，Trivy 却认为已修复——这种信息不对称浪费了无数沟通成本。

**本章的核心目标是：让团队理解镜像分层扫描的原理，掌握漏洞追溯方法，建立对 OS Vendor Backport 机制的正确认知。**

## 2. 项目设计

**场景**：云帆科技的技术复盘会，阿杰、老李和安全工程师小白正在分析订单服务镜像的漏洞报告。

---

**小胖**：（挠头）「我就加了三行代码，怎么漏洞多了十几个？这扫描器是不是抽风了？」

**小白**：「你把 Dockerfile 给我看看……（看了一眼）问题找到了。你上次用 `node:16-alpine3.14`，这次改成了 `node:18-alpine3.17`。基础镜像整个换了，里面的系统库自然也全变了。」

**小胖**：「那我升级基础镜像还有错了？不是越新越安全吗？」

**大师**：「升级本身没错，但『新』不等于『安全』。你可以把镜像分层想象成汉堡包：最底下是面包（基础镜像），中间是生菜番茄（系统工具），最上面是肉饼（你的业务代码）。你换了一家面包供应商，虽然肉饼没变，但新面包里可能有更多的防腐剂——哦，我是说漏洞。」

**小胖**：「技术映射：所以基础镜像就是『面包层』，它的材料变了，整个汉堡的安全性就变了？」

**大师**：「对。Docker 镜像的每一层都是只读的，构建时从下往上叠加。Trivy 扫描时会逐层解压，分析每一层里的文件。对于 OS 包来说，它会在特定路径找数据库——比如 Alpine 的 `/lib/apk/db/installed`，Debian 的 `/var/lib/dpkg/status`——然后把所有层里的包信息合并起来，得到一个完整的软件清单。」

**小白**：「那如果上一层把下一层的文件覆盖了呢？比如我在 Dockerfile 里 `RUN apt-get upgrade`？」

**大师**：「好问题。Trivy 不是只看单一层，而是把所有层叠加后的最终文件系统作为分析对象。如果你在某一层升级了 OpenSSL，新的文件会覆盖旧层中的同名文件，Trivy 看到的是升级后的版本。但如果 upgrade 不彻底，或者只是部分升级，旧版本的 so 库可能还残留在某一层里——这就是分层扫描的复杂性所在。」

**小胖**：「原来如此。那我怎么知道这 30 个漏洞里，哪些来自基础镜像、哪些是我自己的依赖？」

**小白**：「Trivy 本身没有直接输出『漏洞属于哪一层』的功能，但我们可以通过构建策略来隔离。比如采用多阶段构建（Multi-stage Build），把基础镜像的 OS 包和应用的 npm 包分开到不同的构建阶段，最后只把编译产物拷进最终镜像。这样最终镜像里几乎没有 OS 包，漏洞面就小多了。」

**大师**：「更实用的方法是：先单独扫描基础镜像，记录它的漏洞基线；再扫描最终镜像，对比差异。差异部分就是你引入的。」

**小胖**：「还有个问题。这个 `CVE-2021-44228`，我看 NVD 上说 Log4j 2.15.0 才修复，但咱们镜像里装的是 `log4j-2.17.1`，为什么 Trivy 还报？」

**小白**：「给我看看报告……哦，这个不是 Maven 依赖，是通过 `yum` 安装的 `log4j` RPM 包。RHEL 对 Log4j 做了 backport，把补丁打到了 2.14.1 版本上，版本号后面加了 `.el8_5`。Trivy 检测到 RPM 版本是 `2.14.1-5.el8_5`，查 RHEL 的安全通告，认为已修复。但开发看的是 Maven 的 `2.17.1`， confusion 就在这里。」

**大师**：「技术映射：OS Vendor Backport 就像汽车厂商的召回维修。上游厂家说『换新款发动机才算修好』，但 4S 店可以在老款发动机上打个补丁，让它不再漏油。Trivy 信任 4S 店（OS Vendor）的维修记录，因为 4S 店更了解这辆车的具体情况。如果你只看上游厂家的公告，就会误以为老款发动机的车都不安全。」

**小胖**：「那我作为开发，以后怎么判断一个漏洞到底修没修？」

**大师**：「三步法：

1. **看 Status 字段**：Trivy 报告里的 `Status` 如果是 `fixed`，且 `Fixed Version` 一栏有值，说明 OS Vendor 已经发布了补丁。只要你的镜像版本号大于等于 Fixed Version，就是安全的。
2. **看 SeveritySource**：Trivy 会标明 severity 来自哪个数据源。如果是 `debian`、`redhat`、`alpine`，说明采用了 Vendor 评级，通常比 NVD 更准确。
3. **交叉验证**：对于语言包（如 Maven/npm），Trivy 没有 Vendor Backport 的问题，直接看 GitHub Advisory 的上游版本号即可。」

**小白**：「还有一个边缘情况：如果我在 Dockerfile 里手动编译安装了某个软件，没有走包管理器，Trivy 能扫出来吗？」

**大师**：「这就涉及到 Trivy 对『非打包软件』的识别能力。对于没有走 `apk`/`dpkg`/`rpm` 的软件，Trivy 会尝试通过其他线索识别——比如二进制文件的路径、内置版本字符串、SBOM 文件等。但识别率不如包管理器高，这也是我们强烈建议用官方包管理器安装系统工具的原因。」

## 3. 项目实战

### 环境准备

- **Docker**：已安装，可正常构建镜像
- **Trivy**：v0.50+，已配置好缓存
- **测试镜像**：我们将构建一个故意包含已知漏洞的多层镜像

### 步骤一：构建一个故意含漏洞的多层镜像

**目标**：通过 Dockerfile 的多个 RUN 指令，创建多层镜像，验证 Trivy 的扫描原理。

创建 `Dockerfile.vulnerable`：

```dockerfile
# 第一层：基础镜像（Alpine 3.12 已知有漏洞）
FROM alpine:3.12 AS base

# 第二层：安装有漏洞的系统包
RUN apk add --no-cache openssl=1.1.1g-r0 \
    && apk add --no-cache sqlite=3.32.1-r0

# 第三层：业务代码层
WORKDIR /app
COPY package.json ./
RUN echo '{"name":"demo","dependencies":{"lodash":"4.17.20"}}' > package.json

# 第四层：最终层，添加入口脚本
COPY entrypoint.sh ./
RUN chmod +x entrypoint.sh
CMD ["./entrypoint.sh"]
```

创建 `entrypoint.sh`：

```bash
#!/bin/sh
echo "Vulnerable app running"
```

构建镜像：

```bash
docker build -f Dockerfile.vulnerable -t vuln-demo:latest .
```

查看镜像分层：

```bash
docker history vuln-demo:latest
```

**预期输出**（截取关键部分）：
```
IMAGE          CREATED         CREATED BY                                      SIZE
<missing>      2 minutes ago   COPY entrypoint.sh ./ # buildkit                45B
<missing>      2 minutes ago   COPY package.json ./ # buildkit                 67B
<missing>      2 minutes ago   RUN apk add --no-cache openssl=1.1.1g-r0 s…   5.12MB
<missing>      3 years ago     /bin/sh -c #(nop) ADD file:xxx in /             5.57MB
```

> **注意**：`<missing>` 是因为使用了 BuildKit，层信息存储在 build cache 中，不影响 Trivy 扫描。

### 步骤二：扫描并分析漏洞来源

**目标**：扫描镜像，识别漏洞来自 OS 包还是应用依赖。

```bash
trivy image --severity HIGH,CRITICAL vuln-demo:latest
```

**预期输出**（截取关键部分）：
```
vuln-demo:latest (alpine 3.12.12)
=================================
Total: 15 (HIGH: 10, CRITICAL: 5)

┌─────────────┬────────────────┬──────────┬────────┬───────────────────┐
│   Library   │ Vulnerability  │ Severity │ Status │   Fixed Version   │
├─────────────┼────────────────┼──────────┼────────┼───────────────────┤
│ openssl     │ CVE-2021-3711  │ HIGH     │ fixed  │ 1.1.1l-r0         │
│ openssl     │ CVE-2021-3449  │ HIGH     │ fixed  │ 1.1.1k-r0         │
│ sqlite      │ CVE-2021-36690 │ MEDIUM   │ fixed  │ 3.34.1-r0         │
└─────────────┴────────────────┴──────────┴────────┴───────────────────┘

Node.js (node-pkg)
==================
Total: 1 (HIGH: 1, CRITICAL: 0)

┌─────────┬────────────────┬──────────┬────────┬───────────────────┐
│ Library │ Vulnerability  │ Severity │ Status │   Fixed Version   │
├─────────┼────────────────┼──────────┼────────┼───────────────────┤
│ lodash  │ CVE-2021-23337 │ HIGH     │ fixed  │ 4.17.21           │
└─────────┴────────────────┴──────────┴────────┴───────────────────┘
```

**分析要点**：
- `alpine 3.12.12` 部分：漏洞来自 OS 包（`openssl`、`sqlite`），是我们通过 `apk add` 安装的。
- `Node.js (node-pkg)` 部分：漏洞来自应用依赖（`lodash`），是 `package.json` 中声明的。
- 如果基础镜像 `alpine:3.12` 本身也有漏洞，它们会出现在 OS 包列表中。

### 步骤三：多阶段构建优化

**目标**：使用多阶段构建，将构建依赖与运行依赖隔离，减少最终镜像的漏洞面。

创建 `Dockerfile.multistage`：

```dockerfile
# 构建阶段：安装依赖、编译（包含大量构建工具）
FROM node:16-alpine AS builder
WORKDIR /app
COPY package.json ./
RUN npm install

# 运行阶段：仅保留运行所需的最小环境
FROM alpine:3.18
WORKDIR /app
# 只复制构建产物，不复制 node_modules 和构建工具
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
COPY entrypoint.sh ./
RUN chmod +x entrypoint.sh
CMD ["./entrypoint.sh"]
```

构建并扫描：

```bash
docker build -f Dockerfile.multistage -t multi-stage-demo:latest .
trivy image --severity HIGH,CRITICAL multi-stage-demo:latest
```

**对比结果**：
- `vuln-demo`：包含构建工具、旧版 Alpine、大量 OS 包漏洞。
- `multi-stage-demo`：只包含 Alpine 3.18 运行层 + 应用文件，漏洞数量通常减少 70% 以上。

### 步骤四：OS Vendor Backport 验证

**目标**：验证 Debian/RHEL 的 backport 机制，消除对 Fixed Version 的误解。

创建一个基于 Debian 的测试镜像：

```dockerfile
FROM debian:bullseye-slim
RUN apt-get update && apt-get install -y openssl=1.1.1n-0+deb11u4
```

构建并扫描：

```bash
docker build -t debian-openssl-demo:latest .
trivy image --severity HIGH debian-openssl-demo:latest
```

查看报告中 OpenSSL 的 `Fixed Version` 和 `Status`：

```
┌─────────────┬────────────────┬──────────┬────────┬──────────────────────┐
│   Library   │ Vulnerability  │ Severity │ Status │   Fixed Version      │
├─────────────┼────────────────┼──────────┼────────┼──────────────────────┤
│ openssl     │ CVE-2022-0778  │ HIGH     │ fixed  │ 1.1.1n-0+deb11u5     │
└─────────────┴────────────────┴──────────┴────────┴──────────────────────┘
```

**关键观察**：
- 镜像中安装的版本是 `1.1.1n-0+deb11u4`。
- 修复版本是 `1.1.1n-0+deb11u5`。
- 这意味着 Debian 在 `1.1.1n` 的基础上打了 5 个安全补丁（`deb11u5` 表示 Debian 11 的第 5 次更新）。
- 如果开发只看上游 OpenSSL 的版本号 `1.1.1n`，会误以为没修复；但 Trivy 识别了 Debian 的补丁后缀，正确判断出需要升级到 `u5`。

### 步骤五：对比 `trivy image` 与 `trivy fs`

**目标**：理解镜像扫描和文件系统扫描的差异。

```bash
# 将镜像导出为文件系统
docker create --name temp-container vuln-demo:latest
docker cp temp-container:/ ./extracted-fs
docker rm temp-container

# 用 fs 模式扫描提取的文件系统
trivy fs --severity HIGH,CRITICAL ./extracted-fs
```

**差异总结**：

| 维度 | `trivy image` | `trivy fs` |
|------|---------------|------------|
| 分析对象 | OCI 镜像的分层结构 | 普通目录或文件系统 |
| OS 包识别 | 通过镜像内的包管理器数据库 | 通过宿主机的包管理器数据库（如有） |
| 语言包识别 | 扫描镜像内的锁定文件 | 扫描目录内的锁定文件 |
| Secret 检测 | 扫描所有层中的文件 | 扫描目录中的文件 |
| 典型场景 | CI 构建后扫描镜像 | 开发机扫描本地项目 |

> **可能遇到的坑**：`trivy fs` 扫描提取的镜像文件系统时，可能会误报大量宿主机的文件。更推荐直接用 `trivy image` 扫描镜像，让 Trivy 正确处理分层元数据。

### 测试验证

1. 执行 `docker history vuln-demo:latest`，确认镜像有 4 层以上。
2. 执行 `trivy image vuln-demo:latest`，确认报告中同时包含 OS 包漏洞和语言包漏洞。
3. 对比 `vuln-demo` 和 `multi-stage-demo` 的扫描结果，确认多阶段构建后漏洞数量显著下降。
4. 检查 Debian OpenSSL 报告的 `Fixed Version`，确认带有 `+deb11u5` 格式的 Vendor Backport 版本号。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 分层分析 | 能识别 OS 包和语言包的来源，便于分类修复 | 不直接输出漏洞属于 Dockerfile 的哪一行 |
| OS Vendor 数据 | 减少 Backport 误报，准确率更高 | 版本号格式复杂，初学者容易困惑 |
| 多阶段构建支持 | 与 Docker Multi-stage 天然配合，减少漏洞面 | 需要开发者理解镜像构建优化 |
| 非打包软件识别 | 支持二进制分析和 SBOM 识别 | 识别率不如包管理器，可能漏报 |
| 扫描速度 | 分层缓存机制，重复 layer 不重复分析 | 超大镜像（>10GB）分析耗时较长 |

### 适用场景

1. **镜像漏洞溯源**：需要区分漏洞来自基础镜像、系统包还是应用依赖，制定针对性修复策略。
2. **基础镜像选型评估**：在替换基础镜像前，先用 Trivy 扫描候选镜像，对比漏洞基线。
3. **多阶段构建优化**：通过 `builder` 阶段和最终运行阶段隔离，大幅缩减生产镜像的攻击面。
4. **OS 包合规审计**：对 Alpine、Debian、RHEL 等发行版的安全补丁状态进行自动化检查。
5. **语言包漏洞发现**：识别镜像内嵌的 `package-lock.json`、`pom.xml` 等依赖文件中的已知漏洞。

**不适用场景**：
1. 需要精确到 Dockerfile 指令行的漏洞定位——Trivy 不输出漏洞与 Dockerfile 行号的映射。
2. 纯 Windows 容器（Windows Server Core/Nano Server）的 OS 包扫描——Trivy 对 Windows 容器支持有限。

### 注意事项

- **基础镜像的漏洞继承**：无论你上层怎么优化，基础镜像的 OS 包漏洞都会带入最终镜像。选型基础镜像时，务必先单独扫描一次。
- **Distroless 镜像的特殊性**：Distroless 镜像没有 shell 和包管理器，Trivy 无法通过传统方式识别 OS 包。如果使用了 Distroless，确保应用依赖的锁定文件已包含在镜像中，并开启语言包扫描。
- **缓存层与构建参数**：Docker 的层缓存可能导致「旧漏洞被缓存隐藏」。修改 `ARG` 或 `ENV` 时，注意下游层的缓存失效情况。

### 常见踩坑经验

**踩坑案例 1：多阶段构建后仍然有大量漏洞**
- **现象**：使用了 `FROM alpine:latest AS builder` + `FROM alpine:latest`，最终镜像还是有很多漏洞。
- **根因**：`COPY --from=builder` 把 `node_modules` 全部复制过来了，其中包含大量有漏洞的依赖。
- **解法**：在 builder 阶段执行 `npm ci --only=production`，只安装生产依赖；或使用 `npm audit fix` 提前修复。

**踩坑案例 2：Debian 镜像漏洞报告里的 Fixed Version 无法直接 apt-get 安装**
- **现象**：Trivy 报告说 `openssl` 需要 `1.1.1n-0+deb11u5`，但 `apt-get install openssl=1.1.1n-0+deb11u5` 报错找不到包。
- **根因**：没有执行 `apt-get update`，本地 apt 缓存中的包列表是旧的。
- **解法**：在 Dockerfile 中确保 `apt-get update` 和 `apt-get upgrade` 或 `apt-get install` 在同一条 RUN 指令中执行。

**踩坑案例 3：Alpine 的 `apk add` 安装了旧版本包**
- **现象**：明明指定了 `apk add openssl`，扫描时还是报漏洞。
- **根因**：Alpine 的仓库中可能同时存在多个版本，`apk add` 默认安装的是该 Alpine 版本仓库中的最新版，但不一定是安全版。
- **解法**：在 Dockerfile 中显式指定安全版本号，如 `apk add --no-cache openssl=1.1.1w-r1`；或使用 `apk upgrade` 强制升级。

### 思考题

1. 如果你有一个 5 层的基础镜像，第 3 层安装了一个有漏洞的 `curl`，第 5 层通过 `RUN apk upgrade curl` 升级到了安全版本。Trivy 扫描最终镜像时会报这个漏洞吗？为什么？
2. 假设你的团队正在评估 `alpine:3.18`、`debian:bookworm-slim`、`distroless` 三种基础镜像。请设计一个评估脚本，自动扫描这三种镜像并输出漏洞数量、镜像大小、启动时间的对比报告。

> **答案提示**：第 14 章「私有镜像仓库与 Harbor 集成扫描」将介绍如何对基础镜像进行持续监控和准入控制。
