# 第10章 日志与连接池调优（正文初稿）

> 对应总纲：**驯兽师修炼 · 可观测性与数据层入口**。读完本章，你应能 **给 Tomcat 与业务日志做降噪与防刷盘**，并能为 **`tomcat-jdbc` 连接池** 设一套 **可监控、可回滚** 的容量与泄漏策略。  
> **说明**：若应用使用 **HikariCP / Druid** 等，原理与 **池化、超时、校验** 相通；本章 **源码锚点** 按总纲落在 **`org.apache.tomcat.jdbc.pool.ConnectionPool`**。

---

## 本章导读

- **你要带走的三件事**
  1. **日志是负载**：同步写盘 + 高 QPS = **IO wait、线程阻塞**（与第8章线程打满叠加）。
  2. **连接池不是越大越好**：过大 **拖垮数据库**、过小 **请求排队**；必须 **`maxWait`、校验、驱逐** 三件事一起设计。
  3. **泄漏是 P1**：`removeAbandoned` 只能 **止血**，根因是 **未 close** 或 **异常路径漏关**。

- **阅读建议**：先对你当前环境填 **10.7 手册模板** 的「现状」列，再改 **一个** 日志或池参数做压测（第6章纪律）。

## 10.2 Tomcat 与容器日志（JUL）

### 10.2.1 默认机制

- Tomcat 内置常用 **`java.util.logging`（JUL）**，配置在 **`conf/logging.properties`**。
- **`catalina.out`**：若用 **`catalina.sh run`** 或 systemd 未分离 stdout，**stdout/stderr** 可能持续追加 → **极易撑满磁盘**。

### 10.2.2 生产建议（原则）

1. **按包分级**：`org.apache.catalina`、`org.apache.coyote` 等 **WARN/INFO** 分治；开发 DEBUG 不上生产。
2. **滚动与上限**：`FileHandler` 的 **limit / count**（或交给 **logrotate** / 容器侧限制）。
3. **不要把「全量 DEBUG」当排障常态**：用 **请求级 traceId** + **短时上调**。

### 10.2.3 `logging.properties` 片段（示意）

```properties
handlers = 1catalina.org.apache.juli.AsyncFileHandler, java.util.logging.ConsoleHandler
.level = INFO
org.apache.catalina.core.ContainerBase.[Catalina].[localhost].level = INFO
org.apache.coyote.level = WARN
1catalina.org.apache.juli.AsyncFileHandler.level = INFO
1catalina.org.apache.juli.AsyncFileHandler.directory = ${catalina.base}/logs
1catalina.org.apache.juli.AsyncFileHandler.prefix = catalina.
1catalina.org.apache.juli.AsyncFileHandler.maxDays = 30
```

> **AsyncFileHandler**：异步写出，降低 **请求线程** 被同步 IO 拖慢的概率；仍以 **队列有界** 与 **落盘能力** 为上限。

## 10.3 业务日志（Log4j2 / Logback）

### 10.3.1 同步 vs 异步

| 模式 | 优点 | 风险 |
|------|------|------|
| **同步** | 简单、进程崩溃少丢日志 | 高并发下 **RT 与 IO** 压力 |
| **异步** | 吞吐好 | 队列满 **阻塞或丢日志**；kill -9 **丢缓冲** |

**Log4j2** 常见：`AsyncAppender` / `AsyncLogger`；**Logback** 常见：`AsyncAppender` + **discardingThreshold** 等。

### 10.3.2 过载保护思路

- **滚动**：按 **大小 + 时间** 双触发；**总容量 cap**（`totalSizeCap` 等）。
- **采样**：错误全量、高频 INFO **按比例** 或 **每 N 秒一条**。
- **独立磁盘或卷**：日志与 **数据库数据盘** 分离。
- **敏感字段脱敏**：密码、token、证件号（合规）。

## 10.4 Access Log（访问日志）

- **`AccessLogValve`**：按请求一条，量大时 **磁盘与 IO** 显著。
- 优化方向：
  - **按需字段**：不必记录 **完整 Cookie/Query**。
  - **缓冲**：`buffered="true"`（以文档为准）减少 **频繁 flush**。
  - **异步实现**（若版本/扩展支持）或 **前置 Nginx access log** 分流。
- **与第9章**：别把 Access log 当「缓存命中率」唯一来源，需 **解析脚本**。

## 10.5 JDBC 连接池：`tomcat-jdbc` 配置要点

在 **`META-INF/context.xml`** 或 **`conf/context.xml`** 中声明 `Resource`（属性名 **以官方文档为准**）：

```xml
<Resource name="jdbc/AppDS"
          auth="Container"
          type="javax.sql.DataSource"
          factory="org.apache.tomcat.jdbc.pool.DataSourceFactory"
          driverClassName="com.mysql.cj.jdbc.Driver"
          url="jdbc:mysql://db:3306/app?useSSL=false&amp;characterEncoding=utf8"
          username="app"
          password="secret"
          maxTotal="50"
          maxIdle="20"
          minIdle="5"
          maxWaitMillis="15000"
          testOnBorrow="false"
          testWhileIdle="true"
          validationQuery="SELECT 1"
          validationInterval="30000"
          timeBetweenEvictionRunsMillis="30000"
          minEvictableIdleTimeMillis="60000"
          removeAbandoned="true"
          removeAbandonedTimeout="120"
          logAbandoned="true"/>
```

### 10.5.1 参数直觉

| 参数 | 直觉 |
|------|------|
| **`maxTotal`** | 池内 **最大物理连接**；**全局**（多实例）× **每实例** 不得超过 DB 承载 |
| **`maxIdle` / `minIdle`** | 空闲保留与下限；过小 **频繁建连**，过大 **占 DB 连接** |
| **`maxWaitMillis`** | 取连接 **最大等待**；`-1` 无限等易导致 **线程堆积**（第8章） |
| **`testOnBorrow`** | 每次取连 **校验**；**最稳**但 **RT +1 次查询** |
| **`testWhileIdle` + `validationQuery`** | 定时/空闲时校验；常与 **`validationInterval`** 配合防 **过于频繁** |
| **`removeAbandoned`** | **泄漏止血**：超时未归还 **强制回收**；可能误伤 **合法长事务** |

### 10.5.2 容量粗算（起点，不是公式）

- **池大小** 与 **`maxThreads`**、**下游 RT**、**事务持有时间** 相关：线程可能在 **等连接** 或 **持连接等下游**。
- **经验起点**：`maxTotal` 从 **较小值**（如 10～30）压测 **排队与错误**，**小步增加**，观察 **DB 连接数与慢查询**。

### 10.5.3 泄漏排查

1. **日志**：`logAbandoned="true"` 打出 **疑似泄漏栈**（有开销，短时开启）。
2. **代码**：**try-with-resources**、`finally` 中 `close`；框架 **事务边界** 是否正确。
3. **DB**：长时间 **Sleep** 会话是否与应用实例数匹配。

## 10.6 源码锚点：`org.apache.tomcat.jdbc.pool.ConnectionPool`

**读什么**：

1. **`getConnection` 等待队列**：与 **`maxWaitMillis`** 对应。
2. **`borrow` / `return`**：连接如何 **包装**、**校验**、**销毁**。
3. **后台驱逐线程**：与 **`timeBetweenEvictionRunsMillis`** 的关系。

**读法提示**：对照 **`PooledConnection`** 与 **拦截器链**（若启用），理解 **一次 SQL 前后** 池做了什么。

## 10.8 与第7、8、11章的联动

- **堆与 GC 恶化**：有时是 **日志对象过大** 或 **异步队列** 堆积（第7章）。
- **线程打满**：可能是 **`getConnection` 长时间阻塞**（本章）而非 Connector（第8章）。
- **Session 复制**：长 Session + 大属性 → **堆与日志** 同时膨胀（第11章预告）。

## 本章小结

- **日志**：默认 **JUL + catalina.out 风险**；生产要 **分级、滚动、异步、限流**。
- **连接池**：**maxTotal** 对齐 **DB 与实例数**；**校验与驱逐** 是稳定性三角；**泄漏** 靠 **代码 + 日志 + DB** 三角验证。
- **`ConnectionPool` 源码** 把 **参数与现象** 钉死，避免「拍脑袋 200 连接」。
