# 第15章：【基础篇综合实战】构建电商数据中台调度体系

> **定位**：融会贯通基础篇全部知识，构建生产级电商数据中台调度体系。
> **核心内容**：项目需求拆解、架构设计决策、12步完整实施、全链路验证与运维收尾。
> **实战目标**：为大麦电商搭建日→周→月三级调度体系，覆盖Shell/Python/SQL/SubProcess/Dependent/Conditions/告警/权限/资源中心/补数全能力。

## 1. 项目背景

大麦电商是一家中型电商平台，日均订单量500万，覆盖3C数码、美妆、食品三大品类。公司设有五个核心部门——财务部、市场部、运营部、工程部、管理层——每个部门都依赖数据平台产出关键决策报表。

现状堪忧：当前的调度体系是一套"crontab + 手动Email + Excel"的手工作坊。凌晨两点，运维小王被闹钟叫醒，远程登录一台跳板机，按顺序手动执行15个脚本。脚本之间的依赖全凭一本皱巴巴的《操作手册》：第3步的订单清洗脚本跑完，才能运行第4步的销售聚合；如果第2步挂了，后面13步全部白跑——但小王可能已经睡过去了。上个月，小王请假三天，代班的同事漏跑了一个市场部的渠道归因脚本，导致"双十一活动复盘"数据缺失，市场总监在周会上当场发飙。

CTO拍板：必须在两个月内，用Apache DolphinScheduler重建整个数据调度体系。具体需求如下：

- **数据源整合**：MySQL（订单库、用户库）、Hive（数仓ODS/DWD/DWS层）、ClickHouse（实时分析），需统一管理数据源连接。
- **每日流水线**：凌晨2点ODS→DWD ETL → 3点DWD→DWS聚合 → 4点DWS→ADS应用层报表 → 5点全链路数据质量校验 → 6点邮件推送各部门报表。
- **每周汇总**：每周一早8点，自动汇总前7天的日报产出周报，发送管理层。
- **月度仪表盘**：每月1号早9点，汇总前4周的周报产出月度CEO看板。
- **补数能力**：支持任意历史日期范围的数据回填，用于修复因上游系统故障导致的数据缺失。
- **权限隔离**：财务、市场、运营、工程四个部门各自独立的项目和工作流，互不可见也不可篡改。

**成功标准**：支持1000并发任务实例、P99延迟<100ms、系统可用性99%、30天零人工干预。

---

**大师**（放下茶杯，拿起白板笔，在架构图上画了三个圈）：

> "小白这三个问题，个个问到点子上。我一个一个答。"

> "第一，SubProcess vs Dependent——记住一条铁律：**复用逻辑用SubProcess，等待外部结果用Dependent**。数据准备流程是'逻辑复用'——两个日报都要跑同一套ETL，那就把它封装成子流程，在父流程里调用，像函数调用一样传参。Depentent是'状态检查'——周报要等7个日报全跑完，这不是复用，是跨工作流的依赖判定。混淆了就会把子流程当成'被等的人'，把Dependent当成'调用的方式'，那架构就乱了。"

> "第二，告警疲劳——你说得对，要建**告警严重级别矩阵**。P0级：数据准备失败 → 双通道告警（钉钉+电话），5分钟内负责人必须响应。P1级：报表产出延迟 → 钉钉+邮件，30分钟内处理。P2级：非核心任务超时 → 仅邮件，次日处理即可。此外还要配置**告警静默窗口**——凌晨2-6点是正常的批量调度窗口，同类型告警60分钟内只发一次。这样既能保证关键告警不遗漏，又不会被刷屏。"

> "第三，权限粒度——DolphinScheduler的项目权限模型是项目级的读/写/执行，但生产环境还需要配合**数据层权限**。财务部的项目不可见不等于数据不可见——如果报表产出后以邮件附件发送或写入共享数据仓库的表，那就需要数据源的列级/表级授权。简单来说：调度权限看DS的项目授权，数据权限看Hive/MySQL的GRANT。跨部门临时共享，走数据层的视图授权，不走调度层的项目越权。"

> **技术映射**：SubProcess的本质是工作流级的函数调用——子流程独立运行、拥有自己的实例和日志，通过参数传递实现数据交换。Dependent的本质是跨工作流的状态查询——Master扫描依赖条件时，会去元数据库检查目标流程实例的状态是否为SUCCESS。告警静默通过AlertServer的Redis缓存最近告警时间戳实现去重。项目权限由`t_ds_relation_project_user`表的perm字段（0=无权限/1=读/2=写/4=执行）按位控制。

**`/DaMai/etl/python/quality_check.py`**：
```python
#!/usr/bin/env python3
import sys, json
from datetime import datetime

biz_date = "${biz_date}"
dwd_table = "${DWD_TABLE}"

# 模拟质量检查：检查空值率、行数校验
null_ratio = 0.02       # 模拟结果：空值率2%
row_count = 4823015     # 模拟结果
expected_min = 4500000
expected_max = 5500000

checks = []
pass_all = True

if null_ratio > 0.05:
    checks.append(f"空值率超标: {null_ratio:.2%} > 5%")
    pass_all = False

if row_count < expected_min:
    checks.append(f"行数偏低: {row_count} < {expected_min}")
    pass_all = False
elif row_count > expected_max:
    checks.append(f"行数偏高: {row_count} > {expected_max}")
    pass_all = False

quality_score = 0.95 if pass_all else max(0.0, 1.0 - null_ratio * 10)

result = {
    "biz_date": biz_date,
    "table": dwd_table,
    "null_ratio": null_ratio,
    "row_count": row_count,
    "quality_score": round(quality_score, 3),
    "checks": checks
}

print("::set-output name=QUALITY_SCORE::" + str(result["quality_score"]))
print(json.dumps(result, indent=2, ensure_ascii=False))

if quality_score < 0.95:
    sys.exit(1)
else:
    sys.exit(0)
```

**`/DaMai/reports/sql/daily_sales.sql`**：
```sql
-- 每日销售日报聚合（DWD → DWS）
SELECT
    '${biz_date}'                            AS report_date,
    product.category                         AS category,
    COUNT(DISTINCT o.order_id)               AS order_cnt,
    SUM(o.amount)                            AS total_revenue,
    AVG(o.amount)                            AS avg_order_value,
    COUNT(DISTINCT o.user_id)                AS active_users
FROM dwd.dwd_orders_day o
JOIN dim.dim_product product ON o.product_id = product.product_id
WHERE o.dt = '${biz_date}'
GROUP BY product.category
ORDER BY total_revenue DESC;
```

## 4. 项目总结

### 架构全景回顾

下表总结了调度体系中每个组件的作用、运行位置和监控方式：

| 组件 | 作用 | 运行位置 | 监控指标 |
|------|------|----------|----------|
| **prepare_data 子流程** | 统一ETL入口，拉取→清洗→质检→入仓 | Worker (etl_heavy组) | null_ratio, row_count, quality_score |
| **daily_sales_report** | 财务销售日报，DWS聚合→CSV→邮件 | Worker (report组) | 执行耗时、行数差异 |
| **daily_user_report** | 用户行为分析，留存计算→报表 | Worker (report组) | 用户数据完整性 |
| **weekly_executive_report** | 管理层周报，依赖7个日报→合并→PPT | Worker (default组) | 日报依赖满足率 |
| **monthly_dashboard** | CEO月报，依赖4个周报→KPI汇总 | Worker (default组) | 周报依赖满足率 |
| **告警体系** | 多通道分级告警，P0-P2三级 | AlertServer | 告警到达率、误报率 |
| **权限体系** | 4租户 + 4项目 + 3Worker组 | ApiServer + 元数据库 | 越权访问拦截次数 |
| **资源中心** | 脚本/JAR集中管理 + 版本控制 | 存储后端（本地/HDFS/S3） | 文件数、存储量 |

### 五个核心经验

1. **先设计依赖拓扑，再创建节点**。很多新手一上来就拖节点、写脚本，写到第8个节点才发现依赖关系画错了，全部推倒重来。应该在白纸上先画出完整的DAG拓扑，标注每个节点的输入输出参数和数据类型，评审通过后再在DS画布上落地。

2. **SubProcess的边界要清晰**。"数据准备"之所以适合做子流程，是因为它具备三个特征：① 多个父流程共用 ② 有清晰的输入（biz_date）和输出（DWD_TABLE, ROW_COUNT） ③ 内部逻辑独立，不应该被父流程打断。不符合这三个条件的，不要强行封装。

3. **告警不是越多越好，分级静默是关键**。第11章学习告警配置时觉得"每个工作流都配告警"很稳妥，但在综合项目中体会到：11个工作流如果同时失败，钉钉群会被刷爆。必须建立P0/P1/P2三级矩阵，并为P1/P2设置静默窗口。

4. **Dependent节点的依赖周期要和业务日历对齐**。周报依赖"上周一至周日"，如果配置成"过去7天"而今天恰好是周三，那就错了——会依赖上周三至本周二，数据窗口对不上。必须用DS的"上周"周期选择器，而不是手动算日期。

5. **运维文档是生产系统的一部分，不是附属品**。没有Runbook的调度系统等于没有使用说明书的药品——出了问题谁也不知道怎么处理。模板化Runbook（常见故障→处理步骤→责任人→升级流程）能把平均故障恢复时间从小时级降到分钟级。

### 方案对比

| 维度 | Crontab方案 | Apache Airflow方案 | **DolphinScheduler方案（本项目）** |
|------|-------------|-------------------|----------------------------------|
| 依赖管理 | 无，全靠`sleep` | DAG代码定义（Python） | **可视化DAG编辑 + Dependent节点** |
| 子流程复用 | 复制粘贴脚本 | TaskGroup/SubDag | **SubProcess节点，传参调用** |
| 条件分支 | 无，`if-else`写脚本 | BranchOperator（代码） | **Conditions节点，画布配置** |
| 告警 | 手动curl发消息 | EmailBackend + Callback | **AlertServer独立服务，12种渠道** |
| 多租户 | 无 | RBAC（需配置） | **Tenant + Worker Group + 项目权限** |
| 补数 | 写for循环脚本 | backfill CLI | **UI一键补数，串行/并行可选** |
| 学习门槛 | 低（懂cron就会） | 高（Python代码编程） | **中（拖拽+配置，门槛远低于Airflow）** |
| 运维成本 | 高（分散脚本难管） | 中（需维护DAG代码） | **低（可视化运维，日志集中查看）** |
