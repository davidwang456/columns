## 1. 项目背景

小张所在的数据团队上周刚刚完成了 DolphinScheduler 3.x 的集群部署，从单机测试到三节点高可用，折腾了整整一周。环境终于跑起来了，PM 在群里发了一条消息："明天全员开始用 DS 管理 ETL 任务，大家先熟悉一下。"

结果第二天，群里炸了——"这个首页的数字啥意思？""工作流定义和工作流实例有什么区别？""安全中心里有六个菜单，我该点哪一个？"新人们打开浏览器，面对二十多个侧边栏菜单，直接进入"我是谁、我在哪"的状态。

PM 找到架构师老李："我们花了一周搭建环境，结果大家打开界面就懵了——赶紧出一个新手导航指南。"老李苦笑："DS 的 UI 确实功能多，但日常用的就那么几个。我来带大家过一遍。"

这就是本章的定位：一份面向新手的 DolphinScheduler 控制台全功能导航指南。我们将按使用频率从高到低，逐一拆解首页、项目管理、工作流定义与实例、安全中心、数据源中心、监控中心等九个核心模块，帮助团队在半小时内建立对 DS 界面的整体认知。

## 2. 项目设计——剧本式交锋对话

> 场景：周一晨会后，老李带着小胖和小白坐在会议室大屏前，浏览器已打开 DolphinScheduler 控制台。

**小胖**（盯着满屏菜单，挠头）："李哥，这也太吓人了。首页、项目管理、工作流定义、工作流实例……加上下面还有安全中心、数据源中心、监控中心，二十多个子菜单。我一个做数据开发的，从哪开始啊？首页这几个数字——项目数、工作流数——是啥意思？"

**老李**（不急不慢，鼠标切到首页 Dashboard）："别慌，你每天做开发，真正高频使用的菜单不超过五个。首页就是你的'驾驶舱仪表盘'——项目总数告诉你团队有多少个项目在跑，工作流总数反映业务覆盖面，下面两个饼图分别展示当天工作流和任务的运行状态分布。注意右下的 24 小时执行统计图，绿色柱子代表成功，红色代表失败——一眼就能看出昨晚的 ETL 有没有炸。"

**小白**（指着侧边栏下半部分）："安全中心为啥有六个子菜单？租户和用户是什么关系？我登录用的账号算用户还是租户？"

**老李**："好问题，这是个经典的概念混淆。租户（Tenant）本质是 Linux 操作系统用户——DS 的任务最终以某个 OS 用户身份去执行脚本，这个 OS 用户就是租户。用户（User）是控制台登录账号。一个用户绑定一个租户，这样 DS 就知道'这个登录的人的任务应该用哪个系统账号来跑'。安全中心的其他菜单——告警组用来配邮件/钉钉通知，队列是 YARN 队列映射，令牌给 API 调用用，Worker 分组用于任务隔离——你现在只需要了解就行，真正配置时再细看。"

**小胖**（鼠标点进"项目管理"）："所以第一步是先建项目？"

**老李**："对。项目管理是万物的容器。你点右上角的'创建项目'按钮，填项目名和描述，点确定就完事了。一个项目对应一个业务线或一个团队，比如'数据仓库ETL''用户画像计算'。创建完项目，你就能在这个项目下创建工作流了。"

**小白**："那工作流定义和工作流实例有什么区别？名字太像了，我每次都会点错。"

**老李**（切换到工作流定义页面，展示列表）："记住一句话——**定义是模板，实例是执行记录**。工作流定义是你'画'出来的那套 DAG 图，定义里面只有任务的拓扑结构和参数配置。当你把定义上线、设置定时，到了调度时刻，系统会把定义'复印'一份出来跑，这份复印件就是工作流实例。所以你修改定义不会影响已经生成的实例。"

**小胖**："定义页面的工具栏好多按钮——创建、保存、导入、导出、上线、下线……"

**老李**："先说核心操作：点'创建工作流'会进入 DAG 画布，左侧是任务节点类型——Shell、SQL、Spark、Flink、Python、子流程等等，你把它拖到画布上，连线，就搭出了任务依赖关系。保存后回到列表，点'上线'才能被调度。导入导出用于跨环境迁移——在测试环境导出 JSON，到生产环境导入即可。"

**小白**（切到工作流实例页）："实例页面更热闹了，左边是甘特图，右边是任务列表……"

**老李**："实例页是你排障的主战场。列表里每条记录代表一次执行，你可以点'重跑'修复失败的实例，点'恢复'让暂停的流程继续跑，点'停止'中断异常流程。切到甘特图视图，你能直观看到每个任务的耗时和并行度——如果某个任务占了总时长的 80%，那它就是你该优化的瓶颈。"

**小胖**："数据源中心和监控中心呢？感觉是配置型的，平时不太会进来吧？"

**老李**："数据源中心是给 SQL 类任务用的——你在这里注册 MySQL、Hive、ClickHouse、PostgreSQL 的连接信息，之后在 DAG 里选数据源就行，不用每次写 JDBC 地址。监控中心显示 Master 和 Worker 节点的 CPU、内存、负载状态，如果任务大量堆积，先到这里看是不是 Worker 资源吃紧了。"

**小胖**（长舒一口气）："这么一梳理，清晰多了。我来总结一下：先看首页仪表盘→创建项目→画工作流定义→上线→等调度→查看实例。安全中心和数据源中心是初始化配置，监控中心用来排障，对吧？"

**老李**："总结到位。接下来我们挨个实操一遍。"

## 3. 项目实战

### Step 1：登录并熟悉首页 Dashboard

打开浏览器进入 DolphinScheduler 控制台，登录后映入眼帘的就是首页。

**界面描述**：页面顶部横向排列四个指标卡片——项目总数、工作流总数、工作流实例状态分布、任务实例状态分布。每个卡片都用不同颜色区分：项目和工作流用蓝色，工作流实例状态用环形图（绿色=成功、橙色=运行中、红色=失败），任务实例类似。右下方是 24 小时执行统计柱状图，X 轴是时间段，Y 轴是执行次数，绿色柱为成功、红色柱为失败。

**关键解读**：
- 项目总数 = 团队已创建的项目数量，一般 5-20 个为正常范围
- 工作流总数 = 所有项目下工作流定义的总和（含上线和下线状态）
- 工作流实例状态分布 = 当天所有运行过的工作流的最终状态比例
- 任务实例状态分布 = 比工作流更细的粒度，展示每个任务节点的执行结果

> **实操提示**：每天早上到公司，先扫一眼 24 小时执行统计图——如果红柱偏高，说明昨晚的调度出了问题，马上去工作流实例页排查。

### Step 2：创建第一个项目

点击左侧菜单"项目管理"，进入项目列表页。

**界面描述**：顶部有搜索框（可按项目名搜索），右上角是蓝色的"创建项目"按钮。列表中每行显示项目名称、描述、创建时间、更新时间，行末有编辑和删除操作按钮。

**操作步骤**：
1. 点击"创建项目"按钮，弹出对话框
2. 填入项目名称，例如 `test-learning`，描述写"新人学习测试项目"
3. 点击"确定"，列表中出现新建项目
4. 点击项目名称，进入项目内部详情页

> **注意**：项目名称在 DS 中必须全局唯一，建议使用英文+下划线命名，如 `dw_ods_etl`。

### Step 3：浏览工作流定义列表

进入某个项目后，默认显示该项目的"工作流定义"标签页。

**界面描述**：表格列包括工作流名称、状态（上线/下线）、创建人、创建时间、修改时间、操作按钮（编辑、上线、下线、运行、定时、复制、导出、删除等）。顶部工具栏有创建工作流、导入工作流按钮。

**各列含义**：
| 列名 | 说明 |
|------|------|
| 工作流名称 | 定义的唯一标识，上线后不可重名 |
| 状态 | 上线=会被调度触发；下线=仅可手动触发 |
| 创建人/时间 | 便于追溯谁创建、何时创建 |
| 操作按钮 | 编辑=进入画布，上线/下线=控制调度，运行=手动触发一次 |

导航小技巧：点击"编辑"进入 DAG 画布后，左侧节点库包含 Shell、SQL、Spark、Flink、Python、Dependent、SubProcess、Switch 等节点类型，拖拽到画布即可编排。节点之间连线代表执行依赖——上游成功后才执行下游。

### Step 4：探索安全中心

点击左侧菜单"安全中心"，展开六个子菜单。

#### 4.1 创建租户

点击"安全中心 > 租户管理"，进入租户列表页。

**操作步骤**：
1. 点击"创建租户"，弹出表单
2. 租户编码填 `bigdata`（必须与操作系统用户一致）
3. 队列填 `default`（对应 YARN 队列名）
4. 描述填"大数据平台租户"
5. 点击"确定"

**关键理解**：这里的租户编码必须对应 Linux 系统真实存在的用户。比如你的 Worker 节点上有一个系统用户叫 `hadoop`，那租户编码就填 `hadoop`。任务执行时，DS 会切换到这个系统用户来运行脚本。

#### 4.2 创建用户并绑定租户

点击"安全中心 > 用户管理"。

**操作步骤**：
1. 点击"创建用户"
2. 用户名填 `xiaowang`，密码自设
3. 租户下拉选择刚创建的 `bigdata`
4. 邮箱填真实邮箱（用于告警通知）
5. 点击"确定"

**关键理解**：一个用户只能绑定一个租户，但多个用户可共享同一个租户。如果你是为团队新人创建账号，先确认他的系统账号是哪个租户编码，再创建对应 DS 用户并绑定。

#### 4.3 告警组与 Worker 分组

- **告警组管理**：创建告警组，关联用户并配置告警渠道（邮件、钉钉、企业微信）。当任务失败时，DS 向该组内所有成员发送告警。
- **Worker 分组管理**：为 Worker 节点打标签。比如将几台高配节点划入 `high_memory` 组，在创建任务时可指定该分组，实现资源隔离。
- **令牌管理**：为 API 调用生成访问令牌，适用于 CI/CD 流程中通过 HTTP API 触发任务。
- **队列管理**：映射 YARN 集群的资源队列，在创建租户时需要选择一个队列。

### Step 5：探索数据源中心

点击"数据源中心"菜单，进入数据源列表。

**操作步骤——注册 MySQL 数据源**：
1. 点击"创建数据源"
2. 数据源类型选择 `MySQL`
3. 填写连接信息：主机 IP、端口 3306、数据库名、用户名、密码
4. 点击"测试连接"，确认返回"连接成功"
5. 数据源名称填 `prod_mysql_report`
6. 点击"确定"

**界面描述**：创建数据源页面是一个居中表单，顶部选择数据源类型后，下方动态展示该类型的专属字段。MySQL 类型包含主机、端口、数据库、用户、密码等标准字段；Hive 类型则额外需要填写 HDFS 配置；ClickHouse 需要填写 HTTP 端口等。

**关键理解**：数据源注册后，在工作流定义的 SQL 类任务节点中可以直接引用数据源名称，无需每次填写 JDBC 连接串。支持的数据源类型包括 MySQL、PostgreSQL、Hive/Impala、Spark、ClickHouse、Oracle、SQL Server、DB2 等主流数据库。

### Step 6：探索监控中心

点击"监控中心"菜单，分为 Master 监控和 Worker 监控两个子页面。

**界面描述**：页面以表格形式展示所有 Master/Worker 节点信息，包括主机地址、端口、状态（正常/异常）、CPU 使用率、内存使用率、系统负载。表格支持按节点搜索和刷新，异常节点会以红色高亮显示。

**关键解读**：
- Master 负责调度（分配任务到 Worker）和 DAG 执行编排，通常 1-3 台即可
- Worker 负责实际执行任务，数量决定了并发吞吐能力
- 如果 CPU 或内存持续高于 85%，说明该节点已过载，需考虑扩容或调整任务分配
- 如果某个节点状态为"异常"，检查对应服务器上 DS 的 Master/Worker 进程是否存活

## 4. 项目总结

### 菜单使用频率分类

DS 控制台菜单虽多，但使用频率差异很大。对数据开发工程师而言，建议按以下优先级掌握：

| 分类 | 菜单名称 | 使用场景 |
|------|----------|----------|
| 🔴 每日必用 | 首页 Dashboard | 到公司先看一眼执行统计，确认昨晚调度是否正常 |
| 🔴 每日必用 | 工作流实例 | 排查失败任务，重跑/恢复异常实例 |
| 🔴 每日必用 | 工作流定义 | 修改 DAG 逻辑，上线新版本 |
| 🔴 每日必用 | 任务实例 | 查看失败任务日志，追溯上下游依赖 |
| 🟡 每周使用 | 项目管理 | 新建业务项目或归档旧项目 |
| 🟡 每周使用 | 定时管理 | 调整调度频率或暂停/恢复调度 |
| 🟢 偶尔使用 | 安全中心 | 新增人员账号、配置告警通知 |
| 🟢 偶尔使用 | 数据源中心 | 新增数据源或修改连接信息 |
| 🟢 偶尔使用 | 监控中心 | 节点扩容或排查集群异常时查看 |

### 常见操作误区与解决

**误区一：修改工作流定义后，已生成的实例为什么没变？**

> **原因**：定义是模板，实例是副本。实例在生成时将定义的配置快照了一份，之后修改定义不会影响已生成的实例。
> **解决**：如果修改的定义需要影响后续执行，直接正常操作即可——新版本的实例会使用最新定义；如果希望当前失败的实例重跑时使用新定义，需要先终止原实例，等下一次调度触发即可。

**误区二：在安全中心创建了用户，但登录时提示"该用户未绑定租户"。**

> **原因**：创建用户时忘记选择租户，或者选择的租户编码在 Worker 节点上不存在对应的 OS 用户。
> **解决**：进入安全中心 > 用户管理 > 编辑该用户，绑定一个有效的租户。同时登录 Worker 节点确认该租户对应的系统用户确实存在（`id <tenant_code>` 命令验证）。

**误区三：数据源连接测试成功，但任务执行时报"Access denied for user"。**

> **原因**：连接测试是在 DS 服务器（Master 节点）上进行的，但任务执行是在 Worker 节点上。如果 Worker 节点的网络或权限与 Master 不一致，就会出现测试通过但执行失败的情况。
> **解决**：排查目标数据库的授权白名单是否包括所有 Worker 节点的 IP；如果有防火墙，确保 Worker 到数据库端口的网络是通的。在 Worker 节点上用相同连接信息手动测试数据库连接。

### 思考题

1. 如果你的团队有 3 条业务线（数据仓库、实时计算、算法平台），每条线有 2 位开发人员，你会如何规划 DS 中的项目、租户和用户体系？画出你的规划方案。

2. 某天你发现首页 Dashboard 的任务失败率突然飙升到 40%，你会按照什么顺序在 DS 控制台的哪些页面中排查问题？请列出你的排查路径。
