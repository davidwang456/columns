# 第34章：注册中心与分布式协调

> **关键词**：ZooKeeper、Etcd、JDBC Registry、临时节点、Leader选举、分布式锁、心跳检测、故障转移、Curator、Session超时、Split-Brain

---

## 1. 项目背景

大麦数据中台遭遇了一次教科书级的"脑裂"故障。凌晨3点17分，核心交换机固件升级导致Master-1、Master-2与ZooKeeper集群之间出现短暂的网络分区——TCP包能发出去，但ACK迟迟不到，ZK认为两个Master都还活着。Master-1原本是Leader，但它的ZK Session还没过期（超时设了30秒）；Master-2发现Leader心跳断了（实际上只是网络分区），它认为自己应该接管。于是——从3:17:05到3:17:20这15秒内，两台Master同时从`t_ds_command`表消费命令，将同一批工作流各分发了一遍。用户早上醒来发现：30个工作流产生了60个实例，数据重复写入，下游BI报表全乱了。

排查人员熬了两个通宵才发现根因：`zookeeper.session.timeout`配置为30000（30秒），而生产环境中交换机升级导致的分区恰好持续了15~20秒——正好落在"ZK还没判定Master下线，但备Master已经等不及了"的灰色地带。如果将session超时降到8秒，ZK能在第一波心跳丢失后迅速清理临时节点，备Master不会提前抢权。这次事故暴露了一个铁律：**分布式系统中ZooKeeper的配置不是"设大一点保险"，而是要精细计算故障转移窗口——session超时 × 2 + 选举时间，就是系统无法调度工作流的盲区时长**。注册中心在DS中承担服务发现、Leader选举、分布式锁、心跳存活检测四大核心职责，任何一环的配置失误都会引发连锁反应。理解ZK在DS中的节点结构、Watch机制和选举协议，是拿到生产环境"许可证"的前提。

## 2. 项目设计——剧本式交锋对话

三人刚从RPC通信栈的讨论中走出来，白板上还留着Protobuf的二进制编码草图。大师把话题一转。

**小胖伸了个懒腰：**"ZooKeeper不就是个'服务中心'嘛！Master和Worker启动的时候去ZK注册一下，互相发现对方在哪，没了。跟我们在公司打卡一样——来上班滴一下，前台就知道你今天来了。三句话能讲明白的东西，为什么要写一整章？"

**小白推了推眼镜，连珠炮似地发问：**"那我来追问四层——第一，临时节点消失后ZK能立即通知Master吗？通知延迟多久？Master收到通知前这段时间，已经派发给那个Worker的任务怎么办？第二，多个Master怎么通过ZK选出Leader？如果Master-A刚选上Leader，ZK集群突然挂了，Master-A继续调度吗？第三，分布式锁怎么保证同时只有一个Master消费`t_ds_command`？如果用JDBC注册中心代替ZK实现同样的锁，性能差多少？第四，你刚才说'打卡'——如果员工进了公司但忘了打卡呢？Worker进程还在但网络到ZK断了，Master认为它死了把任务移到别的Worker——原来的Worker其实还在跑同样的任务，这不就是重复执行吗？"

**大师挥手示意小白坐下，拿起白板笔：**"小胖的'前台打卡'比喻方向对，但太浅了。我来把它展开——ZK在DS里不是简单的前台，它是公司的全套行政系统。每个Master和Worker启动时去ZK创建**临时节点**——这个临时节点的本质是跟ZK维持一条心跳连接（Session），连接断了，ZK在Session超时后自动删除节点。这就像员工的工牌内置了生命体征传感器——心跳一停，行政系统立刻把工位标记为'空缺'。"

"**服务发现**就是查'谁今天在岗'——Master Watch `/dolphinscheduler/worker/`，这个目录下有任何子节点变化（新增/删除），ZK都会推送给Master。**Leader选举**是看谁工号最小——所有Master在`/dolphinscheduler/lock/master-leader/`下创建临时顺序节点，序号最小的当Leader，其他人Watch紧挨着自己前面的那个节点。Leader挂了，它的临时节点消失，排在后面的立刻被ZK通知→新的最小序号当选，整个过程在1秒内完成。**分布式锁**用的是同一种机制——`/lock/command-consumer`下谁序号小谁就获得消费权，只有Leader才能消费命令。**故障转移**则靠Master Watch Worker节点——Worker的临时节点消失→Master立刻知道它退出了→把该Worker上正在跑的任务重新分配到其他Worker。"

**小胖挠头：**"那网络分区呢？就是我刚才说的——Master还活着，ZK也活着，但中间的网络断了。这叫'假死'，怎么办？"

**大师在白板上画了一个两条竖线中间一道闪电的图：**"这就是分布式系统最经典的脑裂（Split-Brain）问题。网络分区的本质是——同一个集群被切成两半，每半都认为对方死了，每半都试图接管全局。DS的防护手段有三道：第一，Session超时不要设太大——建议8~10秒。Session超时 = ZK判定客户端死亡的时间窗口。8秒意味着网络分区持续8秒以上，ZK才会删除临时节点、通知其他Master重新选举。第二，分布式锁 + 命令消费互斥——即使真的出现极端情况的脑裂，只有拿到`command-consumer`锁的Master才能消费命令。不过大麦那场事故是因为命令消费锁和Leader选举走的不是同一把锁，才会出现双写。第三，任务幂等检查——Worker执行前检查`t_ds_task_instance`的版本号，如果版本号已经被其他Worker改过，说明任务已经被执行过了，拒绝重复执行。"

**小白追问：**"你提到JDBC注册中心可以替代ZK——它怎么实现同样的临时节点和分布式锁？性能差距有多大？"

**大师放下笔："JDBC注册中心是给没有ZK运维能力的小集群用的降级方案——它在数据库里建两张表：`t_ds_heartbeat`和`t_ds_lock`。服务注册就是每隔5秒UPDATE一条心跳记录。Master发现Worker 30秒没更新心跳就认为它死了。分布式锁则是`INSERT INTO t_ds_lock VALUES (lock_key, lock_holder, expire_time)`——利用主键唯一约束，INSERT失败说明锁被抢了。**核心差距在一件事上：延迟**。ZK的Watch是推送机制——Worker注册/下线，ZK主动推给Master，延迟在亚秒级。JDBC靠轮询——Master每隔5秒SELECT一次`t_ds_heartbeat`表，平均发现延迟2.5秒，最坏5秒。故障转移：ZK ~8秒，JDBC ~30秒。所以JDBC Registry只适合10台以内的小集群，大规模生产环境还是必须ZK或Etcd。"

**小胖恍然大悟：**"哦——ZK就是公司内部的对讲机系统，及时推送；JDBC就是公告栏，你得自己走过去看。"

**大师点头：**"就是这个比喻。ZK的Watch机制是**事件推送**，服务状态变化立刻通知订阅者；JDBC的心跳表是**轮询拉取**，信息更新和被发现之间有窗口时差。在调度系统中，这个时差就是故障转移的盲区——盲区越长，重复执行和任务卡死的风险越高。"

### Step 2: 理解Master Leader选举源码

```java
// dolphinscheduler-registry/dolphinscheduler-registry-zookeeper
public class ZookeeperLeaderElection {

    private volatile boolean isLeader = false;

    public void electLeader() {
        String lockPath = "/dolphinscheduler/lock/master-leader";
        String serverInfo = "192.168.1.101:5679"; // 本机地址

        ensurePathExists(lockPath);

        // 创建临时顺序节点——ZK自动在路径后追加10位递增序号
        String ourNode = zkClient.create()
            .creatingParentsIfNeeded()
            .withMode(CreateMode.EPHEMERAL_SEQUENTIAL)
            .forPath(lockPath + "/lock-", serverInfo.getBytes());
        // ourNode = /dolphinscheduler/lock/master-leader/lock-0000000003

        // 获取lock目录下所有子节点，按序号排序
        List<String> children = zkClient.getChildren().forPath(lockPath);
        Collections.sort(children); 
        // ["lock-0000000001", "lock-0000000002", "lock-0000000003"]

        // 提取我们自己的序号（去掉"lock-"前缀）
        String ourSequence = ourNode.substring(ourNode.lastIndexOf("/lock-") + 6);

        // 如果我们是第一个（序号最小）→ 成为Leader
        if (ourSequence.equals(children.get(0).substring(5))) {
            isLeader = true;
            logger.info("本Master当选为Leader, 节点: {}", ourNode);
            return;
        }

        // 否则Watch紧挨在我们前面的那个节点
        int ourIndex = children.indexOf("lock-" + ourSequence);
        String toWatchPath = lockPath + "/" + children.get(ourIndex - 1);
        // 例如Watch lock-0000000002

        // 注册Watcher：前一个节点消失→重新选举
        zkClient.getData().usingWatcher((WatchedEvent event) -> {
            if (event.getType() == Watcher.Event.EventType.NodeDeleted) {
                logger.info("前序Master节点 {} 已消失，重新选举", toWatchPath);
                electLeader(); // 递归触发新一轮选举
            }
        }).forPath(toWatchPath);
    }
}
```

选举机制的三个关键点：（1）只有序号最小的Master是Leader，所有备节点Watch的不是所有更小节点，而仅仅是**紧挨在自己前面的那一个**——这避免了"惊群效应"（所有备节点同时被唤醒竞争）。（2）Leader宕机后，其临时节点随Session过期被ZK删除，排在第二位的Master的Watcher立即触发，1秒内即可完成接管。（3）如果新Master加入时序号最小，它直接成为Leader，原Leader退位为Follower——这是正常的Leader切换，不会中断调度。

### Step 3: 生产环境ZK配置调优

```yaml
# registry.properties —— 注册中心配置文件
registry:
  type: zookeeper
  zookeeper:
    connect-string: "zk1:2181,zk2:2181,zk3:2181"  # 奇数节点，推荐3或5
    namespace: dolphinscheduler                     # 所有DS节点在此命名空间下
    session-timeout: 8000         # 8秒 — Session超时，建议8~10秒
    connection-timeout: 3000      # 3秒 — 首次连接ZK的超时
    retry:
      base-sleep-time: 1000      # 1秒 — 重试退避基数
      max-retries: 3             # 最多重试3次
      max-sleep-time: 5000       # 最多等5秒
    digest: "ds:ds_secret_2024"  # ACL认证 —— 生产环境必须开启
```

参数调优黄金法则：
- **session-timeout**：太小（<3s）→ 一次Full GC就可能触发"假死"，导致不必要的故障转移；太大（>15s）→ Leader宕机后长时间无人接管，调度盲区过长。**8秒**是经过大量生产验证的平衡值。
- **connect-string**：务必写**奇数个**ZK节点（3或5）。ZK的Zab协议要求超过半数的节点存活才能选举Leader，偶数节点白白浪费一个名额。
- **namespace**：所有DS节点共享同一个namespace，实现多套调度系统之间的隔离——测试环境和生产环境使用不同的namespace。

### Step 6: 分布式锁——命令消费互斥

```java
// 基于Curator的分布式锁实现
public class CommandConsumerLock {

    private final InterProcessMutex lock;

    public CommandConsumerLock(CuratorFramework client) {
        this.lock = new InterProcessMutex(client,
            "/dolphinscheduler/lock/command-consumer");
    }

    public void consumeCommands() {
        while (running) {
            try {
                // 尝试获取锁，最多等3秒——如果备Master拿不到锁就跳过本轮
                if (lock.acquire(3, TimeUnit.SECONDS)) {
                    try {
                        // === 临界区开始 ===
                        // 只有拿到锁的Master才能执行这些操作
                        List<Command> commands = commandDao.queryReadyCommands(100);
                        for (Command cmd : commands) {
                            WorkflowExecuteRunnable runnable =
                                new WorkflowExecuteRunnable(cmd);
                            workflowThreadPool.submit(runnable);
                            commandDao.delete(cmd.getId());
                        }
                        // === 临界区结束 ===
                    } finally {
                        lock.release(); // 无论是否异常都要释放锁
                    }
                }
            } catch (Exception e) {
                logger.error("命令消费异常", e);
            }
            // 等待1秒后再尝试下一轮
            Thread.sleep(1000);
        }
    }
}
```

InterProcessMutex底层原理：在`/dolphinscheduler/lock/command-consumer`下创建临时顺序节点，序号最小的获得锁。`lock.acquire()`阻塞等待直到本节点的前序节点被删除，底层也是Watch机制而非忙等。与Leader选举的区别在于：这是一个**短期持有**的锁（一次命令消费后就释放），而Leader选举是**长期持有**的锁（直到Leader宕机才释放）。

### Step 7: JDBC注册中心——ZK不可用时的降级方案

```sql
-- JDBC Registry使用的两张核心表

-- 心跳表：服务通过更新此表证明自己存活
CREATE TABLE t_ds_heartbeat (
    id INT AUTO_INCREMENT PRIMARY KEY,
    server_type VARCHAR(20) NOT NULL,    -- MASTER / WORKER / API / ALERT
    server_host VARCHAR(100) NOT NULL,
    server_port INT NOT NULL,
    heartbeat_time DATETIME NOT NULL,    -- 最后心跳时间
    cpu_usage DOUBLE DEFAULT 0,
    memory_usage DOUBLE DEFAULT 0,
    UNIQUE KEY uk_server (server_type, server_host, server_port)
);

-- 分布式锁表：利用主键唯一约束实现互斥
CREATE TABLE t_ds_lock (
    lock_key VARCHAR(100) PRIMARY KEY,  -- 锁的唯一标识
    lock_holder VARCHAR(200) NOT NULL,  -- 当前持有者（服务器地址）
    expire_time DATETIME NOT NULL       -- 锁过期时间（避免死锁）
);
```

```java
// JDBC Registry的核心运作逻辑

// 服务心跳：每5秒更新一次
public void heartbeat() {
    jdbcTemplate.update(
        "INSERT INTO t_ds_heartbeat (server_type, server_host, server_port, " +
        "heartbeat_time, cpu_usage, memory_usage) VALUES (?,?,?,NOW(),?,?) " +
        "ON DUPLICATE KEY UPDATE heartbeat_time = NOW(), " +
        "cpu_usage = VALUES(cpu_usage), memory_usage = VALUES(memory_usage)",
        serverType, host, port, cpuUsage, memoryUsage
    );
}

// Master检测死Worker：查询30秒内没有心跳的Worker
public List<String> detectDeadWorkers() {
    return jdbcTemplate.queryForList(
        "SELECT CONCAT(server_host, ':', server_port) FROM t_ds_heartbeat " +
        "WHERE server_type = 'WORKER' " +
        "AND heartbeat_time < DATE_SUB(NOW(), INTERVAL 30 SECOND)",
        String.class
    );
}

// 分布式锁：利用INSERT的主键唯一约束
public boolean tryLock(String lockKey, String lockHolder, int expireSeconds) {
    try {
        jdbcTemplate.update(
            "INSERT INTO t_ds_lock (lock_key, lock_holder, expire_time) " +
            "VALUES (?, ?, DATE_ADD(NOW(), INTERVAL ? SECOND))",
            lockKey, lockHolder, expireSeconds
        );
        return true; // INSERT成功 = 获得锁
    } catch (DuplicateKeyException e) {
        return false; // INSERT冲突 = 锁已被持有
    }
}

// 释放锁
public void releaseLock(String lockKey, String lockHolder) {
    jdbcTemplate.update(
        "DELETE FROM t_ds_lock WHERE lock_key = ? AND lock_holder = ?",
        lockKey, lockHolder
    );
}
```

### Step 8: 三种注册中心对比选型

| 特性 | ZooKeeper | Etcd | JDBC |
|------|-----------|------|------|
| 事件通知延迟 | ~1ms（Watch推送） | ~1ms（gRPC Watch） | ~2500ms（轮询平均） |
| 故障转移速度 | ~8s | ~3s | ~30s |
| 运维复杂度 | 中等（需独立ZK集群） | 中等（需独立Etcd集群） | 低（复用业务DB） |
| 适用集群规模 | 任意 | 任意 | < 20台服务器 |
| 原生K8s集成 | 需额外部署 | 原生支持 | 不适用 |
| 脑裂风险 | 中等（需调Session超时） | 低（Lease机制更精细） | 高（轮询窗口大） |
| 性能上限 | ~10万 QPS | ~10万 QPS | 受DB连接池限制 |

选型建议：
- **生产环境大型集群（>20台）**：ZooKeeper（成熟稳定，Java生态最佳）或 Etcd（K8s环境首选）
- **生产环境小型集群（<20台）**：ZooKeeper 仍推荐，运维成本可接受
- **开发/测试环境**：JDBC Registry（无需额外部署，快速搭建）
- **POC/Demo**：JDBC Registry

### Step 10: 生产环境ZK常见问题与排查

| 故障现象 | 根因 | 诊断命令 | 修复方案 |
|---------|------|---------|---------|
| Master频繁切换Leader | Session超时设得太小（<3s），Full GC触发假死 | `echo stat \| nc zk1 2181 \| grep Latency` | 调大session-timeout到8~10s，优化JVM GC参数 |
| Worker下线30s+ Master仍无感知 | session-timeout设得太大（>15s），ZK判定慢 | 检查`registry.properties`中`session-timeout` | 降低到8~10s |
| ZK磁盘满，DS所有服务注册失败 | ZK快照和事务日志持续增长未清理 | `df -h /data/zookeeper` | 配置ZK自动清理：`autopurge.snapRetainCount=3` |
| Watcher数量持续增长导致ZK OOM | DS频繁Watch/UnWatch但UnWatch未执行 | `echo wchs \| nc zk1 2181` | 检查代码中是否有Watcher泄漏 |
| 网络分区时出现双Leader | session-timeout高于网络中断时长 | 检查交换机日志中的中断时长 | 降低session-timeout，启用ACL限制 |
| ZK集群宕机后DS完全不可用 | 所有ZK节点同时故障（极罕见） | `echo ruok \| nc zk1:2181` 全部失败 | 紧急切换至JDBC Registry作为临时恢复方案 |

## 4. 项目总结

ZK在DolphinScheduler中扮演了四种不可替代的角色，每一种都对应着分布式协调的经典范式：

| 职责 | ZK机制 | DS中的应用 | 如果失效会怎样 |
|------|--------|-----------|-------------|
| 服务发现 | 临时节点 + Watch推送 | Master发现Worker上线/下线 | 任务无法分发或发到死Worker |
| Leader选举 | 临时顺序节点 + 最小序号优先 | 多个Master竞争Leader | 无Master消费命令，调度停止 |
| 分布式锁 | InterProcessMutex | 命令消费互斥、故障转移互斥 | 重复消费命令、重复故障转移 |
| 健康检查 | Session心跳 + 临时节点自动清理 | 检测Master/Worker存活 | 脑裂、脏数据、任务重复执行 |

**注册中心选型决策树**：集群规模>20台→ZK或Etcd；K8s环境→优先Etcd；运维能力不足→JDBC（接受30s故障转移延迟）；开发测试→JDBC。

**三个源自注册中心的生产故障**：
1. **session-timeout=30s导致脑裂**（本章开篇事故）：网络分区15~20s，ZK未判定Master下线，备Master抢权→双Master同时调度。**修复**：session-timeout降至8秒。
2. **ZK磁盘满致DS全面瘫痪**：某节点事务日志未清理，磁盘使用率达100%→ZK拒绝所有写入→Master/Worker无法注册、无法选举、无法获取锁。**修复**：配置`autopurge.snapRetainCount=3, autopurge.purgeInterval=1`。
3. **JDBC Registry下Worker假死漏检**：数据库连接池耗尽→Worker心跳UPDATE失败→Master误判全部Worker离线→所有任务卡在DISPATCH状态。**修复**：JDBC Registry的心跳需使用独立连接池，不与业务SQL共用。

**思考题**：
1. 如果DS集群有5台Master，当前Leader突然掉线，ZK的Session超时设为8秒。请推演从Leader掉线到新Leader开始消费命令的完整时间线，并计算在此期间可能积压的命令上限（假设平均每秒产生10条命令）。
2. 设计一个方案：在不修改DS源码的前提下，如何利用ZK的Watch机制实现一个"Worker负载告警"——当某个Worker的CPU使用率连续3次心跳超过80%时，自动发送钉钉告警通知运维。提示：考虑写一个独立的Python程序去Watch ZK节点数据。
