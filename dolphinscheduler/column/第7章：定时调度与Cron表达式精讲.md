# 第7章：定时调度与Cron表达式精讲

> **定位**：让调度自动化运转起来。
> **核心内容**：Cron表达式七字段详解、Quartz调度引擎、定时配置与通知策略、定时管理页面、定时与工作流定义解耦。
> **实战目标**：为日报/周报/月报/季报配置多套定时调度，掌握线上/下线/修改/删除的完整生命周期操作。

---

## 1. 项目背景

某金融公司的报表团队负责四份定期报告：日报每天凌晨2:00生成，汇总前一日交易流水；周报每周一凌晨3:00生成，合并上一周的日汇总数据；月报每月1号凌晨4:00生成，计算当月各业务线的KPI指标；季报在每年1月、4月、7月、10月的第一个工作日早晨8:00生成，需呈报给监管机构。

这四份报告都有固定的时间窗口——必须在开盘前产出入库，迟到就意味着合规风险。然而，团队目前的执行方式原始得令人难以置信：运维小王在手机闹钟里设了四个提醒——每天1:55、每周一2:55、每月1号3:55、每季度第一个月1号7:55——闹钟一响，她放下手头一切，SSH登录服务器，手动敲命令启动对应的工作流。如果她请病假，就得打电话让同事帮忙代跑；如果服务器临时维护需要推迟执行，后续所有时间节点全部乱套，全凭她Excel表格逐行修正。

去年第三季度，悲剧发生了。小王国庆休假回了老家，手机闹钟响了但网络不通，她远程连不上服务器，季报没有按时提交。周一上班后，监管机构的问询函已经躺在CTO的收件箱里。最终整个团队被通报批评，年度绩效打了折扣。部门总监在复盘会上拍了桌子："我们每天管着几十亿的交易数据，居然要靠闹钟来驱动调度——这简直是金融科技圈的耻辱！DolphinScheduler的定时调度功能上线后，所有报告必须自动化触发，小王你把闹钟全删了。"

## 2. 项目设计——剧本式交锋对话

会议室里，小胖、小白和大师三人围坐在白板前，开始讨论定时调度方案。

**小胖**（不以为然）："Cron不就是那五个星星嘛！`* * * * *`每分钟跑一次，搞定！咱们把每个报告写成一行cron，往配置文件一塞，比调闹钟强一百倍。"

**小白**（立刻皱起眉头）："五个星星？DolphinScheduler里的定时配置我瞄过一眼，明明是七个字段，比Linux多出来的是什么？而且我之前在一个Spring项目里用过Quartz，它的星期字段周日是1，跟Linux cron的0~6也不一样，这里会不会有坑？"

**小胖**（托着下巴）："七个字段？多出来的……是不是加了秒和年？那周的1到底代表周一还是周日？哎大师你权威，给讲讲。"

**大师**走到白板前画了一张表：

```
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│ 秒  │ 分  │ 时  │ 日  │ 月  │ 周  │ 年  │
│0-59 │0-59 │0-23 │1-31 │1-12 │1-7  │选项 │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┘
```

"DolphinScheduler底层的调度引擎是Quartz，不是Linux的crontab。Quartz的Cron表达式有七个字段：秒、分、时、日、月、周、年。年内可选，但DS默认会带上年份。小胖说的五个星是Linux标准Cron，只含分、时、日、月、周——在DS里这样写会直接报错，因为它要求至少六个字段。"

**技术映射**：DolphinScheduler使用Quartz Cron（7字段），与Linux Crontab（5字段）互不兼容，跨平台迁移时必须重写表达式。

**小白**追问："那周的1到7到底怎么对应星期几？还有特殊字符，我见过Python写的`croniter`项目里`L`表示最后一天、`?`表示不指定——Quartz是不是也有自己的一套？"

**大师**在白板上追加了说明：

"Quartz的星期编号：1=周日、2=周一、3=周二、4=周三、5=周四、6=周五、7=周六。这跟Linux Cron完全不同——Linux是0=周日、1=周一……7=周日。如果你把Linux的`0 3 * * 1`（每周一凌晨3点）原样搬到DS，Quartz会把'0'解释为第5个字段（月），直接报错。正确写法是`0 0 3 ? * 2 *`——注意星期用了2代表周一。"

**技术映射**：Quartz星期字段1=周日，Linux Cron星期0或7=周日，跨系统迁移时这是最高频的踩坑点。

"再说特殊字符。Quartz支持这些通配符：`*`所有值、`?`不指定（只能用在日和星期字段，两者互斥）、`-`范围、`,`列举、`/`步长、`L`最后一天/最后一个周几、`W`最近工作日、`#`第N个周几。比如'每月最后一个周五'可以写成`0 0 18 ? * 6L *`，读作'每月的最后一个周五下午6点'。这在Linux Cron里根本做不到。"

**小胖**（恍然大悟）："卧槽，`6L`就是最后一个周五？这语法还挺形象。那`W`和`#`呢？"

**大师**："`W`表示最近的工作日。比如`0 0 9 15W * ?`表示'每月15号最近的工作日上午9点'——如果15号是周六，就会在14号周五执行；如果15号是周日，就会在16号周一执行。`#`表示第N个周几，比如`0 0 10 ? * 2#3`就是'每月的第3个周一下午10点'。"

**小白**合上笔记本："那定时上线之后，如果我发现表达式写错了，修改了Cron后重新上线，之前已经排队但还没有执行的任务会怎样？"

**大师**："这是个好问题。DolphinScheduler的设计中，定时配置和工作流定义是解耦的——你创建了一个工作流定义，再独立为它创建一个定时配置。一个工作流有且只有一个活跃的定时配置。当你修改Cron表达式并重新上线时，旧的Cron调度会立即停止，Quartz按新的表达式重新计算下一次触发时间。但已经在`t_ds_command`表中排队的Command记录不会被自动清除——它们会被Master正常消费执行。所以如果你的需求是'旧调度产生的待执行任务全部作废'，需要在修改表达式之前，先下线定时、手动清理Command记录，再修改、重新上线。"

**技术映射**：定时上线/下线本质是控制Quartz Scheduler中对应Trigger的启停。修改表达式后重新上线会删除旧Trigger、创建新Trigger，但已入队的Command不受影响。

**小胖**追问："那定时配置的通知策略和工作流启动时候的通知策略，有什么关系？"**大师**："两个独立的维度。定时配置的通知策略控制的是调度层面——比如'Cron表达式非法导致调度失败'、'定时下线'这类事件发生时通知谁。启动工作流时，运行弹窗里的通知策略控制的是本次运行实例的通知——'任务失败了发给谁'。一个管定时维护者的告警，一个管业务负责人的告警，各司其职。"

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x Standalone或集群模式已部署（参见第2章）
- 浏览器打开DS控制台，默认地址`http://localhost:12345/dolphinscheduler`
- 确认左侧菜单中【定时管理】入口可见
- 准备好第4章创建的"每日数据处理"项目，或用本节步骤新建

### Step 1：构建Cron表达式速查表

在动手配置之前，先建立一份Cron表达式参考表，覆盖金融报表团队的四种典型场景：

| 场景 | Cron表达式 | 字段分解 | 说明 |
|------|-----------|----------|------|
| 每分钟 | `0 * * * * ? *` | 秒=0，分=*，时=*，日=*，月=*，周=?，年=* | 每分0秒触发，适用于调试阶段快速验证 |
| 每小时整点 | `0 0 * * * ? *` | 秒=0，分=0，时=* | 每整点触发，常用于低频数据同步 |
| 每天凌晨2:00 | `0 0 2 * * ? *` | 秒=0，分=0，时=2 | 日报标准时间，给上游系统留足数据就绪窗口 |
| 每周一凌晨3:00 | `0 0 3 ? * MON *` | 周=MON | 周报标准时间，收集上周七天数据 |
| 每月1号凌晨4:00 | `0 0 4 1 * ? *` | 日=1 | 月报标准时间，月初汇总上月数据 |
| 每季度首月1号8:00 | `0 0 8 1 1,4,7,10 ? *` | 月=1,4,7,10，日=1 | 季报，逗号列举特定月份 |
| 工作日早9:00 | `0 0 9 ? * MON-FRI *` | 周=MON-FRI | 工作日触发，避开周末 |
| 每月最后一天23:00 | `0 0 23 L * ? *` | 日=L | L表示Last，月末催款/结账常用 |
| 每月最后一个周五18:00 | `0 0 18 ? * 6L *` | 周=6L | 期权结算日等金融场景 |

> **关键提示**：日和星期字段不能同时指定具体值——至少一个必须用`?`。`?`的含义是"不关心此维度"。比如你指定了每月1号（日=1），那么星期就不能再写具体的`MON`，必须写`?`；反之如果指定了星期`MON-FRI`，日就必须写`?`。这是Quartz的硬性约束，违反后表达式校验会报错。

### Step 2：创建工作流"daily_report"

登录DS控制台，进入【项目管理】→ 选择"每日数据处理"项目（如未创建则先创建）→ 点击【工作流定义】Tab。

创建一个新工作流，命名为`daily_report`。工作流内部结构从简——拖入一个Shell节点，脚本内容为：

```bash
#!/bin/bash
echo "===== 日报开始执行 ====="
echo "业务日期: ${system.biz.date}"
echo "执行时间: $(date '+%Y-%m-%d %H:%M:%S')"

# 模拟日报生成逻辑——实际场景中此处为ETL + 汇总 + 导出
sleep 5
echo "日报生成完成，共处理10,234条交易记录。"

exit 0
```

点击【保存】，然后点击【上线】。注意：此处只是工作流定义上线，尚未配置定时——工作流本身随时可以手动运行，但不会自动触发。

### Step 3：配置定时调度

工作流上线后，点击工作流操作区的【定时管理】按钮（或直接在顶部菜单进入【定时管理】页面），点击【新建定时】。

在弹出的配置窗中填写：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| Cron表达式 | `0 0 2 * * ? *` | 每天凌晨2:00:00触发 |
| 开始时间 | `2024-01-01 00:00:00` | 从2024年1月1日起生效 |
| 结束时间 | `2025-12-31 23:59:59` | 到2025年底失效（也可留空表示永久有效） |
| 时区 | `Asia/Shanghai (UTC+8)` | 明确指定时区，避免服务器UTC时区偏差 |
| 通知策略 | 仅失败通知 | 只在调度失败/任务失败时发送告警 |

点击【确定】。DS会将此Cron注册到Quartz Scheduler中，并在元数据库`t_ds_schedules`表中持久化一条调度记录。

此时进入【定时管理】页面，可以看到新增的定时记录，状态为"在线"。此时Quartz已经开始监听时间——每到凌晨2:00:00，自动触发该工作流实例。

### Step 4：理解定时触发到底层链路

定时触发到工作流实例启动的完整链路如下：

```
Quartz Trigger触发
       ↓
MasterSchedulerBootstrap监听
       ↓
生成Command记录写入t_ds_command表
(command_type=START_PROCESS, process_definition_code=xxx)
       ↓
Master轮询消费Command（默认间隔1秒查库）
       ↓
Master创建ProcessInstance
       ↓
Master启动WorkflowExecuteRunnable编排DAG
       ↓
任务下发给Worker执行
```

整个链路中，Quartz只负责"到点唤醒"，它并不直接操作工作流，而是向Command表插入一条START_PROCESS类型的记录。Master Server才是真正的调度消费者，它轮询Command表，拿到记录后执行完整的工作流编排逻辑。这种设计将定时引擎和工作流引擎彻底解耦，即使Quartz异常暂停，Command积压在表中也不会丢失，Master恢复后仍可逐一消费。

### Step 5：快速验证——将启动时间改为3分钟后

生产环境定时动辄每天一次，验证等不起24小时。最快捷的验证方法：将Cron的开始时间改为"当前时间+3分钟"，观察3分钟后是否自动触发。

假设当前时间为`2024-03-15 14:00:00`，操作步骤：

1. 进入【定时管理】页面，找到`daily_report`的定时记录，先点击【下线】。
2. 点击【编辑】，修改Cron表达式为当前3分钟后的精确时刻：
   - 改为`0 3 14 15 3 ? 2024`（即2024年3月15日14:03:00触发一次）
   - 或者更灵活的方式——保持原来的`0 0 2 * * ? *`不变，但修改**开始时间**为`2024-03-15 14:03:00`。
3. 重新点击【上线】。
4. 进入【工作流实例】页面，静等3分钟。观察是否出现一条新的实例记录，执行状态从"提交"→"运行"→"成功"流转。

> **踩坑提示**：如果3分钟后实例没有出现，先检查Worker节点是否在线（监控中心→Worker管理）、再检查Master日志中是否有`schedule complete`日志输出、最后确认服务器时区——DS默认使用服务器本地时区，如果服务器是UTC而Cron期望北京时间，会有8小时偏差。

验证成功后，将定时恢复为正式配置`0 0 2 * * ? *`。

### Step 7：定时生命周期操作演练

现在进行一次完整的定时生命周期实操，覆盖日常运维中的高频操作：

**(1) 下线一个定时**

在【定时管理】页面，找到`daily_report`的定时记录，点击【下线】。确认后，该定时状态从"在线"变为"下线"。下线后，Quartz会将该Cron对应的Trigger暂停，不再触发新的工作流实例。已存在的实例不受影响，继续运行至结束。

验证方法：将Cron表达式改为每分钟一次（`0 * * * * ? *`）用于测试，观察上线时有每分钟的实例、下线后不再有新实例生成。

**(2) 修改Cron表达式**

点击`daily_report`定时记录的【编辑】，将Cron表达式从`0 0 2 * * ? *`（凌晨2:00）改为`0 0 5 * * ? *`（凌晨5:00），然后重新【上线】。

此时Quartz内部的操作序列：
- 删除旧的Trigger（凌晨2:00）
- 创建新的Trigger（凌晨5:00）
- 新Trigger立即开始按照新表达式计算下一次触发时间

进入【定时管理】页面，确认该记录的Cron表达式已更新为新的值。

**(3) 删除定时**

点击`daily_report`定时记录的【删除】，确认操作。该定时记录从元数据库`t_ds_schedules`中被移除，Quartz中的对应Trigger也被注销。删除定时不影响工作流定义本身——`daily_report`的工作流依然存在，只是不再被自动触发，需要时可手动运行。

### Step 8：常见踩坑与排查

| 现象 | 可能原因 | 排查方法 |
|------|----------|----------|
| 配置完定时后一直没有实例触发 | 定时虽创建但未点击"上线"按钮——定时记录处于"下线"状态 | 进入【定时管理】页面检查状态列是否为"在线" |
| 实例触发时间比预期晚了8小时 | 服务器时区是UTC，而Cron表达式按北京时间编写 | 查看服务器`date`命令输出；在定时配置中明确指定时区为`Asia/Shanghai` |
| 修改Cron表达式后仍按旧时间触发 | 没有先下线→修改→重新上线，而是直接修改了表达式 | 修改定时必须执行完整的"下线→编辑→上线"三步，单纯编辑不生效 |
| 工作流实例频繁失败 | 旧的实例还没跑完，新的定时触发又启动了新实例，产生并发冲突 | 检查实例是否并发执行——可在工作流编辑页设置"并行度"为1，或拉长Cron间隔 |
| 保存Timing时提示"表达式非法" | 日和星期同时指定了具体值，违反Quartz的`?`规则 | 修正表达式：`0 0 2 * * MON *` → `0 0 2 ? * MON *`（日改为`?`） |

## 4. 项目总结

### Cron最佳实践速查

| 维度 | 建议 | 反例 |
|------|------|------|
| 日和星期字段 | 永远只指定其中一个，另一个用`?` | `0 0 2 15 * MON *`——日和星期同时指定，结果不可预测 |
| 表达式可读性 | 优先使用星期缩写`MON/WED`而非数字`2/4` | `0 0 3 ? * 2 *`——阅读者不知道2是周一还是周二 |
| 触发密度 | 避免高频调度（<1分钟），加重Master轮询和Command表膨胀压力 | 高频数据同步场景应使用消息队列触发，而非Cron轮询 |
| 时区指定 | 总是在定时配置中显式指定时区，不依赖服务器默认值 | 跨国部署时因时区混乱导致报表时间偏差 |
| 验证方法 | 上线前先用"3分钟后触发"快速验证，确认无误再改回正式时间 | 直接写正式时间上线，等24小时后才发现表达式写错 |
| 通知区分 | 定时配置的通知策略配置运维人员，工作流启动通知配置业务负责人 | 所有告警发给同一个人，后续无法定位责任 |

### Quartz Cron vs Linux Cron vs Airflow schedule_interval

| 特性 | Quartz Cron (DolphinScheduler) | Linux Crontab | Airflow schedule_interval |
|------|-------------------------------|---------------|---------------------------|
| 字段数 | 7（秒/分/时/日/月/周/年） | 5（分/时/日/月/周） | 2（cron_preset / timedelta） |

### 三个常见深层陷阱

1. **夏令时（DST）跳变**：在实行夏令时的地区，春季调快一小时可能造成该时段内的Cron丢失一次触发，秋季调慢一小时可能造成同一Cron重复触发。Quartz默认不处理DST，建议为金融/监管类报表配置`CronScheduleBuilder.dailyAtHourAndMinute()`替代原生Cron表达式（需修改DS源码实现），或在DST切换日通过DS的补数功能手动覆盖缺失的实例。

2. **Cron表达式"静默失败"**：如果Cron表达式语法本身合法，但表达的业务含义与实际需求不符，DS不会给出任何警告。比如`0 0 2 ? * 2 *`你可能以为是每月2号触发，实际是每周一触发（星期2=周一）。建议在团队内部建立表达式评审制度——任意Cron表达式上线前，至少一人核对解析后的"下次触发时间"是否与预期一致。

3. **工作流运行时长超过调度间隔**：如果工作流平均运行时长是50分钟，而Cron设置为`0 */30 * * * ?`（每30分钟），那么每30分钟触发一次的新实例会与尚未完成的旧实例产生并发。虽然在技术上是合法的，但可能导致数据竞争、资源耗尽。建议通过工作流的"并行度"配置限制同一定义最多同时运行的实例数，或拉大Cron间隔至任务平均耗时的2倍以上。

### 思考题

1. 某工作流需要实现"每月最后一个工作日上午10:00执行"，Quartz的`L`和`W`字段能否直接组合写成`0 0 10 LW * ? *`？如果可以，它的行为是什么？如果不可以，应该如何变通实现？请查阅Quartz官方文档验证你的判断。
