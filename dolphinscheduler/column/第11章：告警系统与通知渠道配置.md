# 第11章：告警系统与通知渠道配置

> **定位**：让调度系统"会说话"——构建多通道、分级别的告警体系，确保关键任务失败时第一时间通知到人。
> **核心内容**：AlertServer架构、插件化告警通道（Email/企业微信/钉钉/飞书/Slack/HTTP等12种）、告警组管理、超时策略（告警vs失败）、通知策略层级（工作流vs调度）、收件人与抄送人机制。
> **实战目标**：为一套凌晨ETL关键链路配置"钉钉+邮件+企业微信"三通道告警，并设计告警严重级别矩阵，确保任何单通道故障都不会漏掉告警。

---

## 1. 项目背景

某电商公司的核心数据管道每天凌晨2点自动运行：从MySQL抽取订单数据 → ETL清洗 → 生成销售报表 → 推送到财务系统。财务团队要求每天早晨8点前拿到前一天完整的销售报告。

一个星期三的早晨，运营总监发现销售报表迟迟未出。他打电话给运维工程师老张，老张睡眼惺忪地打开电脑，才发现凌晨3点ETL就因为源数据库连接超时报错了——任务挂了整整5个小时，没有一个人知道。财务总监怒气冲冲地给CTO打电话："为什么ETL挂了五个小时才被发现？我拿什么给CEO做晨会汇报？"

事后盘查发现，团队确实在工作流上配置了"失败通知"，但没有人验证过这个通知是否真的有效——原来两周前SMTP邮件服务器升级、密码过期了，告警邮件从未被送达。

CTO在复盘会上拍着桌子说："第一，关键任务必须走多通道告警——钉钉、邮件、电话，至少两个渠道同时通知。第二，告警服务本身也要被监控。第三，除了失败告警，每天早晨还要有一个成功摘要，让我知道系统是否健康。"

这暴露了典型的"告警盲区"问题：依赖单一通知渠道、告警配置后从不验证、告警服务本身缺乏监控、缺少"正向健康报告"。DolphinScheduler的告警系统正是为解决这些问题设计的——它拥有独立的AlertServer服务、12种插件化告警通道、灵活的告警组与通知策略，让你可以构建一个"打不掉的告警网"。

## 2. 项目设计——剧本式交锋对话

周一技术分享会结束后，小胖用筷子夹着一块红烧肉，边嚼边说。

**小胖**（嘴里含糊不清）：

> "告警不就是发个消息嘛！钉钉机器人webhook一配，我的Shell脚本里两行curl就搞定了——`curl 'https://oapi.dingtalk.com/robot/send' -d '{"msgtype":"text","text":{"content":"任务挂了"}}'`，哪个任务失败就在任务脚本末尾调一下，简单到不行。为啥DS还要专门搞一个独立的AlertServer？多此一举吧？"

**小白**（放下手里的咖啡杯，皱眉）：

> "四个问题。第一，你这种'每个任务自己发告警'的模式，如果任务脚本本身就因为OOM被kill了，根本没机会执行到发告警那行代码——告警丢失。第二，钉钉服务本身也会宕机，如果webhook调用超时了，你的告警就丢了，有没有内置重试机制？第三，你配了超时告警，但如果任务还没超时、Worker进程先挂了，谁替你发这个告警？第四，告警策略层次怎么定——工作流级别的通知和调度级别的通知，到底是什么关系？如果我既在任务上配了告警组，又在调度上配了告警组，用户会收到两遍告警吗？"

**小胖**（筷子停在半空，愣住了）：

> "呃……OOM…webhook超时…这还真是盲区。"

**大师**（从会议室走出来，手里拿着白板笔，在旁边的白板上画了一个医院平面图）：

> "小胖的curl方案就像每个病房的病人给自己打120——如果病人已经昏迷，电话就打不出去。而DolphinScheduler的告警系统更像医院的**护士站**。"

大师在白板上圈出几个区域继续解释：

"每个病房——也就是你配置的每个工作流任务——发生异常时，信号会先传到护士站，由护士（AlertServer）集中判断：这个告警是哪个级别的？要通知哪个告警组的医生？走什么渠道——微信群、企业微信、邮件还是电话？AlertServer是独立的Spring Boot服务，端口50052走RPC协议、50053走HTTP协议，它不跟你任何Worker或者Master耦合在一起。哪怕你的Worker因为OOM崩溃了，Master作为调度大脑，依然能识别出任务失败事件，推送给AlertServer发告警。"

**技术映射**：AlertServer是集中式告警分发中心，解耦了"故障检测"和"通知发送"。Master/Worker负责检测任务状态变化，AlertServer负责告警策略匹配与多渠道投递。这种架构保证了"检测者挂了不等于告警者挂了"。

**小白**（眼睛一亮，飞快地在笔记本上记录）：

> "那告警渠道是怎么扩展的？如果公司用的飞书，目前系统支持吗？还有，超时告警和超时失败这两种策略区别在哪？告警组和收件人/抄送人之间又是什么关系？"

**大师**：

> "告警渠道用的是**SPI插件机制**。目前DolphinScheduler内置了12种插件：Email、企业微信、钉钉、飞书、Slack、Telegram、HTTP、PagerDuty、Script、Prometheus AlertManager、WebexTeams、阿里云语音。每种插件打成一个jar包放在`alert-server/lib`下，AlertServer启动时自动加载。如果12种还不够你用，你甚至可以自定义插件实现`AlertChannel`接口。"

"**超时策略**的选择很关键——这两种配置在任务的'超时告警'属性页里。选'超时告警'，任务超时后AlertServer发一条告警，但任务本身继续运行，不会置为失败状态。选'超时失败'，任务超时后不仅发告警，还会强制将任务标记为失败，然后触发工作流的失败处理逻辑（重试、跳过、结束等）。选哪个？看你的业务——如果时序要求严格，ETL超时必须阻断下游报表，选'超时失败'；如果是数据质量巡检类任务，超时只是慢了点但不影响数据正确性，选'超时告警'就够了。"

"**告警组的收件人vs抄送人**：收件人是直接责任人——任务失败了你要立刻处理；抄送人是知情者——比如团队leader、跨部门接口人，只关心大事不负责执行。创建告警组时，收件人和抄送人分开配置，不同渠道投递方式也不同——比如Email渠道收件人在To字段、抄送人在CC字段；钉钉、企业微信只支持@所有人或指定群成员。"

**技术映射**：告警组 = 通知接收者集合；告警实例 = 一个告警事件触发一个告警实例；告警插件 = 一种消息发送通道。一个告警组可以绑定多种插件同时发送。

**小胖**（终于接上话茬）：

> "那如果告警服务自己挂了怎么办？总不能发不了告警的时候也没人知道吧？"

**大师**：

> "这正是告警系统高可用的核心课题。DolphinScheduler支持AlertServer集群部署——多台AlertServer同时注册到Zookeeper，Master通过服务发现拿到所有可用的AlertServer地址。一台AlertServer宕机，Master自动切换到另一台继续发送告警。同时，你应该对AlertServer本身做健康监控——比如配置一个独立的HTTP健康检查，周期性调AlertServer的`/actuator/health`端点，失败则通过外部手段告警到运维。另外，AlertServer内部有失败重试队列——一次投递失败，会按退避策略重试，重试多次仍失败才会丢弃。"

"记住两个原则：**多渠道冗余**——关键链路至少覆盖两个不同供应商的渠道（比如钉钉+邮件），降低单渠道故障风险；**正向健康报告**——不仅失败了告警，每天早上发送一条汇总消息，告诉你哪些任务准点完成、哪些延迟了、哪些跳过，这就是'无告警不等于健康'。"

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x 完整部署（AlertServer已启动）
- QQ邮箱开通SMTP服务并获取授权码
- 企业微信管理员权限（或已创建群机器人）
- 钉钉群机器人webhook地址
- 确认AlertServer健康：`curl http://<alert-server-host>:50053/actuator/health`

### Step 1：配置Email告警通道

**目标**：让AlertServer通过SMTP协议发送邮件告警。

编辑AlertServer的配置文件`alert-server/conf/application.yaml`（或通过环境变量注入），添加Email插件配置：

```yaml
alert:
  plugin:
    email:
      mail-smtp-host: smtp.qq.com
      mail-smtp-port: 465
      mail-sender: your-email@qq.com
      mail-user: your-email@qq.com
      mail-password: your-auth-code     # ★ QQ邮箱的SMTP授权码，不是登录密码
      mail-smtp-ssl-enable: true
      mail-smtp-starttls-enable: false
```

保存后重启AlertServer：

```bash
# 先停掉AlertServer进程
bin/dolphinscheduler-daemon.sh stop alert-server
# 再启动
bin/dolphinscheduler-daemon.sh start alert-server
# 验证启动成功
tail -f logs/dolphinscheduler-alert-server.log | grep "started"
```

> **关键细节**：QQ邮箱`mail-password`必须填**SMTP授权码**，不是QQ密码。获取路径：QQ邮箱 → 设置 → 账户 → POP3/IMAP/SMTP服务 → 开启 → 生成授权码。163邮箱同理。如果使用企业邮箱（如`mail.company.com`），port通常为25或587，SSL设为false、STARTTLS设为true，具体咨询公司IT。

### Step 2：创建告警组"finance_alert_group"

**目标**：为财务数据管道创建专门的告警接收组，包含至少2名成员（冗余）。

在DolphinScheduler控制台中：

1. 导航到：**安全中心 → 告警组管理 → 创建告警组**
2. 填写信息：

| 字段         | 值                                      |
|--------------|----------------------------------------|
| 告警组名称    | finance_alert_group                    |
| 描述         | 财务核心ETL链路告警组，凌晨2点-8点值班   |
| 收件人        | zhangwei@company.com, lihong@company.com |
| 抄送人        | cto@company.com                        |

3. 保存。告警组创建后，在任何一个工作流定义的"通知策略"或任务的"告警配置"中都可以选择它。

> **设计原则**：收件人至少配2人——当一人休假时，另一人能收到告警。抄送人放领导或接口人，让管理层了解故障但不打扰其正常工作。告警组的命名建议用清晰的组织或系统前缀，比如`etl_core_group`、`ops_oncall_group`、`ad_hoc_monitor_group`。

### Step 3：配置企业微信告警通道

**目标**：通过企业微信应用或群机器人发送告警消息。

编辑`alert-server/conf/application.yaml`，在`alert.plugin`下添加：

```yaml
alert:
  plugin:
    # ... Email配置保持不变 ...
    wechat:
      # 方式一：企业微信应用（推荐，可控性强）
      corp-id: ww123456789abcdef
      agent-id: 1000002
      secret: your-app-secret
      # 方式二：企业微信群机器人（更简单，适合小团队）
      # webhook: https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx
```

重启AlertServer。

> **踩坑提示**：企业微信群机器人有频率限制——**每分钟最多20条消息**。如果同一时间段大量任务批量失败，可能导致告警被限流丢弃。此时建议优先使用应用方式，或对告警做聚合降噪。另外，群机器人的webhook key不要提交到Git仓库，建议用环境变量注入。

### Step 4：配置钉钉告警通道

**目标**：通过钉钉群机器人发送告警消息。

```yaml
alert:
  plugin:
    dingtalk:
      webhook: https://oapi.dingtalk.com/robot/send?access_token=xxx
      secret: your-robot-secret    # ★ 安全设置中的加签密钥，非必填但强烈建议
```

钉钉机器人的创建路径：钉钉群 → 群设置 → 智能群助手 → 添加机器人 → 选择"自定义" → 安全设置勾选"加签" → 复制webhook地址和签名密钥。

> **注意**：钉钉的消息格式支持markdown，DS会自动将告警内容格式化为钉钉Markdown消息。如果你要自定义格式，可以在AlertServer插件目录中找到`dingtalk`的插件jar，通过扩展`AlertChannel`接口实现。

### Step 5：创建测试工作流"alert_test"

**目标**：通过一个必定失败的Shell任务验证三通道告警能正常投递。

在DolphinScheduler工作流定义页创建新工作流`alert_test`，拖入一个Shell任务节点：

- **节点名称**：`critical_failure_sim`
- **脚本内容**：
```bash
#!/bin/bash
echo "===== 模拟核心ETL任务失败 ====="
echo "执行时间: $(date '+%Y-%m-%d %H:%M:%S')"
echo "正在连接源数据库 10.0.2.100:3306..."
sleep 3
echo "ERROR: 数据库连接超时 - Connection refused"
exit 1
```

### Step 6：配置通知策略与告警

在`alert_test`工作流定义页，打开**通知策略**配置：

| 配置项               | 值                        | 说明                                    |
|----------------------|---------------------------|-----------------------------------------|
| 通知策略             | 失败发送                   | 仅工作流执行失败时触发告警               |
| 告警组               | finance_alert_group       | 对应Step 2创建的告警组                  |
| 通知类型             | 钉钉、企业微信、邮件       | 三通道并行发送                          |

保存工作流并上线运行。运行后约10-30秒内，你应该在钉钉群、企业微信和邮箱中分别收到告警消息。消息内容包含：工作流名称、任务名称、失败时间、错误码（exit 1）、以及跳转到DS控制台查看详情的链接。

> **验证检查清单**：
> - [ ] 钉钉群收到markdown格式告警，包含任务名和失败原因
> - [ ] 企业微信收到告警消息
> - [ ] 邮箱收到告警邮件，收件人To字段包含zhangwei和lihong, CC字段包含cto
> - [ ] 三通道的告警到达时间差在15秒以内

### Step 7：测试超时告警场景

**目标**：验证"超时告警"和"超时失败"两种策略的差异。

新增一个Shell任务节点`timeout_sleep_task`：

```bash
#!/bin/bash
echo "这个任务将休眠2分钟，但超时策略设为1分钟..."
sleep 120
echo "正常情况下这行不会执行到"
exit 0
```

在任务的**超时告警**属性配置中：

- **超时间隔**：1分钟
- **超时策略**：先选**超时告警**

上线运行并观察：
1. 任务运行1分钟后，AlertServer发出"任务超时告警"
2. 任务状态仍为"运行中"，继续sleep直到120秒后正常结束

第二次测试，将超时策略改为**超时失败**：
1. 任务运行1分钟后，AlertServer发出告警
2. 同时任务状态变为"失败"
3. 工作流级别的失败通知策略被触发，再次发告警（注意和超时告警是两条独立消息）

> **建议**：生产环境中超时策略的选择是一个业务决策而非技术决策。ETL任务推荐选"超时失败"——任务超时了说明数据产出已延迟，继续运行可能产出过期数据。数据质量巡检类任务推荐选"超时告警"——跑慢一点也比不跑强。

### Step 8：高级——HTTP告警通道对接自定义告警平台

**目标**：将告警推送到公司自建的监控平台（或第三方平台如PagerDuty、Prometheus AlertManager）。

```yaml
alert:
  plugin:
    http:
      url: https://your-monitoring-api.company.com/api/v1/alert
      method: POST
      headers: Authorization=Bearer eyJhbGciOiJIUzI1NiIs...
      body: |-
        {
          "alertName": "${alertName}",
          "workflowName": "${workflowName}",
          "taskName": "${taskName}",
          "state": "${state}",
          "severity": "critical"
        }
```

HTTP插件支持变量替换，`${...}`中的变量名对应告警实例的属性值。你的自定义告警平台收到POST请求后可以做二次分发、升级通知（如电话外呼）、创建工单等。

> **注意**：如果HTTP接收端返回非200状态码，AlertServer会触发退避重试。建议接收端实现幂等处理（根据告警实例ID去重），避免同一条告警因重试被创建多个工单。

### Step 9：构建告警严重级别矩阵

**目标**：为不同级别的故障定义清晰的响应SOP。

| 级别        | 触发条件                  | 通道组合              | 责任人            | 响应时间 |
|-------------|--------------------------|----------------------|-------------------|----------|
| P0-Critical | 核心ETL失败、源库不可达   | 电话 + 钉钉 + 邮件   | 值班工程师（主+备） | 5分钟    |
| P1-High     | 报表生成失败、数据延迟>1h | 钉钉 + 邮件          | 数据团队负责人     | 15分钟   |
| P2-Medium   | 数据质量检查告警          | 邮件 only            | 数据分析师         | 1小时    |
| P3-Low      | 测试环境任务失败          | 不通知               | —                  | 次工作日 |

在DolphinScheduler中，你可以通过创建多个告警组来对应不同级别：
- `p0_critical_group`：收件人=值班工程师, 通知类型=全部通道
- `p1_high_group`：收件人=团队负责人, 通知类型=钉钉+邮件
- `p2_medium_group`：收件人=数据分析师, 通知类型=邮件

每个工作流定义根据自身重要性选择对应的告警组。

### Step 10：常见踩坑清单

| 陷阱                                    | 现象                                  | 根因与解决                                                                                           |
|-----------------------------------------|---------------------------------------|------------------------------------------------------------------------------------------------------|
| **邮件密码填了登录密码而非SMTP授权码**     | 邮件告警始终发送失败，日志报Auth failed | QQ/163等邮箱的SMTP使用独立授权码。解决：去邮箱设置中开启SMTP服务并生成授权码，填入`mail-password`字段   |
| **企业微信机器人被频率限流**              | 批量告警时部分消息丢失                 | 企业微信群机器人限制20条/分钟。解决：切换到企业微信应用方式（无此限制），或做告警聚合——将多条告警合并为一条 |
| **告警组为空**                           | 测试告警时控制台提示"告警组没有成员"    | 创建告警组时必须添加至少一个收件人。解决：检查告警组的收件人或抄送人列表，确保至少有1名有效成员        |
| **告警时间戳与本地时间不一致**            | 告警消息中的时间比实际时间晚/早8小时    | AlertServer的JVM默认时区与用户时区不一致。解决：在AlertServer启动参数中加`-Duser.timezone=Asia/Shanghai` |
| **多通道配置顺序影响性能**                | 某通道超时导致所有通道告警延迟          | AlertServer默认串行投递——先发钉钉再发邮件。解决：确保各通道的超时时间配置合理，或将耗时通道放最后    |
| **忘记验证告警能否正常送达**              | 配置完上线后告警从未触发或从未到达      | 变更后不测试是最大的人为风险。解决：每次修改告警配置后，必须用测试工作流触发一次真实告警并验证所有通道 |

## 4. 项目总结

### 告警渠道对比

| 维度         | Email                  | 企业微信                      | 钉钉                          | HTTP（自定义）            |
|--------------|------------------------|------------------------------|-------------------------------|---------------------------|
| **成本**     | 免费（SMTP账号）        | 免费（企业应用内）            | 免费（群机器人）               | 取决于对接平台             |
| **延迟**     | 3-15秒（SMTP传输）      | 1-3秒（即时通讯）             | 1-3秒（即时通讯）              | 取决于接收端响应速度        |
| **可靠性**   | 中（依赖SMTP服务器）    | 高（企业级基础设施）          | 中（有频率限制）               | 取决于接收端可用性          |
| **配置复杂度** | 低                    | 中（corp-id/secret/app配置） | 低（webhook一键生成）          | 中（需自定义HTTP接收端）    |
| **信息丰富度** | 高（HTML正文+附件）    | 中（Markdown文本）            | 中（Markdown文本）             | 高（任意自定义格式）        |
| **适用场景**   | 日报摘要、审计留档    | 即时故障通知、值班群通知       | 即时故障通知、机器人交互       | 对接内部工单系统、电话外呼 |

### 告警最佳实践

1. **关键任务双通道打底**：P0/P1级任务至少覆盖两个不同供应商的渠道（如钉钉+邮件），任一通道故障不影响告警送达。
2. **避免告警疲劳**：不要对所有任务开启"失败通知"。P3级测试任务的失败用定时健康检查报告覆盖即可，减少"狼来了"效应。
3. **变更后必须测试**：每次修改告警配置（增删渠道、更换密码、换webhook地址等），必须上线测试工作流触发一次真实告警并验证所有通道。
4. **告警服务本身要监控**：AlertServer的`/actuator/health`端点应被外部监控系统周期性探测，AlertServer宕机本身是最高优先级告警。
5. **正向健康报告**：除了失败告警，每天定时发一条汇总消息（如："今日20个任务全部按时完成，数据管道健康。"），这样"没收到告警"等于"一切正常"。

### 思考题

1. 如果某天凌晨核心ETL失败，但钉钉和企业微信同时因为网络波动不可达，邮件告警是唯一的通知渠道——但邮件服务器恰好在维护中。请问在这种极端情况下，还有什么其他机制可以确保故障被发现？DolphinScheduler的Script插件能否作为"最后一道防线"（比如调用电话外呼API或触发声光报警）？

2. 团队有20个工作流，每个工作流都配了告警。当凌晨数据库整机宕机时，20个工作流同时失败，AlertServer会产生20条独立的告警实例。如何设计告警降噪机制，将这20条告警合并为一条"数据库集群不可达，影响20个工作流"的聚合告警？
