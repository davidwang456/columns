# 第2章：单机快速部署与开发环境搭建

## 1. 项目背景

小娜是一名刚入职某电商公司的数据开发工程师。入职第一天，Leader 就给她布置了任务："咱们团队所有数据调度都跑在 DolphinScheduler 上，你先在本地搭一套环境，把 Shell、SQL、Python 这几个任务类型跑通，下周开始接手日报 ETL 流程。"

小娜打开 DolphinScheduler 官网，一看架构图就懵了：API Server、Master Server、Worker Server、Alert Server，中间还夹着 Zookeeper 和 MySQL——这哪是一个"调度工具"啊，分明是一套分布式系统！她之前只用过 crontab 和 Airflow 单机版，面对这么多组件，一时不知从何下手。

**痛点放大**：DolphinScheduler 是一个分布式调度系统，其核心优势在于高可用和水平扩展，但这也意味着部署门槛比单机调度工具高出一个量级。新手直接上来搭集群，至少要面对以下问题：

1. **组件多**：4 个独立服务 + Zookeeper + MySQL，任何一个配置不对都跑不起来。
2. **配置散**：环境变量、数据源、注册中心、服务地址分布在至少 4 个配置文件中。
3. **版本敏感**：JDK 版本、MySQL 字符集、ZK 客户端版本，任何一个不匹配都会出诡异错误。
4. **学习成本高**：即使跑起来了，日志分散在不同目录，出问题不知道看哪个日志。

但是，DolphinScheduler 官方提供了一种"Standalone 模式"——将所有服务运行在同一个 JVM 中，使用嵌入式 H2 数据库，零依赖即可启动。这就像一款模拟飞行器：仪表盘和真飞机一模一样，但不需要真飞机的地勤和塔台。本章的目标，就是帮小娜用四种方式把"模拟飞行器"开起来，并在过程中理解核心配置文件的作用、各部署方案的取舍，为后续章节的深入实战打好地基。

## 2. 项目设计——剧本式交锋对话

小娜把自己的困惑发到了团队群里。刚发出去，群里三个人就吵起来了。

---

**小胖**：（秒回）"直接装个 Docker 不就行了嘛，一分钟搞定！我上周试过了，一个 `docker-compose up -d` 就能跑，还能带 Zookeeper 和 MySQL，多省事！"

**小白**：（过了一分钟，推了推眼镜）"等一下，Docker 是方便，但我有几个问题。第一，Docker 里挂了怎么排查？日志在容器里，jstack、jmap 都用不了。第二，如果我要看源码、改配置、加插件，Docker 镜像里的东西我能改吗？第三，二进制包和 Docker 到底选哪个？总不能每个场景都 docker pull 吧？"

**小胖**：（挠头）"这个……我就跑个 Demo 看看嘛，谁想那么多。"

**大师**：（放下咖啡杯）"你们都说得有道理，但都没抓到本质。部署方式的选择，关键在于你的**目的**。我给你打个比方——"

"Standalone 模式就像是**模拟飞行器**，仪表盘、操纵杆和真飞机一样，但你不能真用它飞上天。它用 H2 嵌入式数据库，所有服务塞在一个 JVM 里，你的电脑就是它的全世界。这适合什么？适合你**学习工作流、调试任务逻辑、写 Demo 验证想法**。"

"集群模式才是**真飞机**。MySQL/PostgreSQL 做元数据库，Zookeeper 做注册中心，每个服务独立部署、独立扩缩容，Master 挂了另一个能顶上。这才是生产环境的标配。"

**小胖**：（恍然大悟）"哦，那我 Docker 装的其实也是 Standalone 对吧？"

**大师**："对，不管你用 Docker、二进制包还是源码编译，只要跑的是 Standalone 模式，本质上都一样。区别在于——"

"- **Docker Compose**：开箱即用，适合快速体验。但镜像封装好了一切，你想改配置、加自定义插件就比较麻烦。"
"- **二进制包**：官方编译好的 tgz 压缩包，解压就能用。适合正式开发环境，配置灵活，所有文件可见可改。"
"- **源码编译**：最重但也最灵活。适合需要深度定制、或者参与开源贡献的场景。"

**小白**：（若有所思）"那就是说，如果我只是想跑通一个 Shell 任务，Docker 最快；如果我要长期开发测试，二进制包最合适；如果我要改源码，才需要编译……但我想搞清楚那个 `dolphinscheduler_env.sh` 里到底配了什么，Docker 里也能看吗？"

**大师**："问得好！`dolphinscheduler_env.sh` 是 DolphinScheduler 的**环境变量枢纽**。你用 Docker 的话，这些变量通过 `docker-compose.yml` 的 `environment` 字段注入，不是直接修改 shell 文件。如果你想看懂每一个配置项，二进制包或源码方式会更直观。来，我们看看这个文件里到底有什么——"

```
# dolphinscheduler_env.sh 核心变量
export JAVA_HOME=${JAVA_HOME:-/usr/local/java}       # JDK 路径
export PYTHON_HOME=${PYTHON_HOME:-/usr/bin/python}   # Python 解释器
export HADOOP_HOME=${HADOOP_HOME:-/opt/hadoop}       # Hadoop 客户端
export SPARK_HOME=${SPARK_HOME:-/opt/spark}          # Spark 客户端
export FLINK_HOME=${FLINK_HOME:-/opt/flink}          # Flink 客户端
export DATAX_HOME=${DATAX_HOME:-/opt/datax}          # DataX 客户端
```

**技术映射**：`dolphinscheduler_env.sh` 就像飞机的地勤补给系统——告诉每个组件去哪里加油（JAVA_HOME）、去哪里装卸货物（PYTHON_HOME）、用哪个货柜（SPARK_HOME）。配错了路径，飞机就在跑道上趴窝。

**小胖**：（举起手机）"那我用 Docker 装完，是不是访问 `http://localhost:12345/dolphinscheduler` 就能看到界面了？"

**大师**："没错，默认用户名密码是 `admin / dolphinscheduler123`。但我建议你至少也走一遍二进制包部署，这样你才能理解 `install_config.conf` 里那些 `ips="localhost"`、`masters="localhost"`、`workers="localhost:default"` 各代表什么——那对应的是集群拓扑。比如说 `workers="localhost:default"`，这里 `default` 是 Worker 分组的名称，在生产环境中你可以配置 `workers="node1:ETL,node2:REPORT"` 来让不同类型的任务跑在不同的机器上。这些概念如果不亲手配置一遍，光看文档是记不住的。"

**小白**："明白了。我今晚先 Docker 跑起来看一眼，再拆二进制包看配置，周末再去下源码编译。大师，源码编译那个 `mvnw clean install -Prelease -DskipTests` 会跑多久？"

**大师**："首次编译大概 15-20 分钟，Maven 要把所有依赖拉下来。之后增量编译就快了。记得 JDK 一定要 1.8，这是当前大多数 DolphinScheduler 3.x 版本的硬性要求——你要是装了 JDK 17，编译能过，跑起来就报 module 相关错误，坑过不少人了。"

## 3. 项目实战

### 环境准备

| 软件 | 版本要求 | 说明 |
|------|---------|------|
| JDK | 1.8 | 硬性要求，OpenJDK 或 Oracle JDK 均可 |
| Maven | 3.6.3+ | 仅源码编译时需要 |
| MySQL | 5.7+ | 仅二进制包/集群模式需要 |
| Zookeeper | 3.8.0 | 仅二进制包/集群模式需要 |
| Node.js | 16+ | 仅前端开发需要 |
| Docker | 20.10+ | 仅 Docker 方案需要 |
| Docker Compose | 2.0+ | 仅 Docker 方案需要 |

### 方案 A：Docker Compose 一键部署

**步骤目标**：5 分钟内通过 Docker 启动一套完整的 Standalone 环境，包含 DS 所有服务。

**Step 1：编写 docker-compose.yml**

```yaml
version: '3.8'
services:
  zookeeper:
    image: bitnami/zookeeper:3.8.0
    container_name: ds-zookeeper
    environment:
      ALLOW_ANONYMOUS_LOGIN: "yes"
    ports:
      - "2181:2181"
    networks:
      - ds-network

  dolphinscheduler:
    image: apache/dolphinscheduler-standalone-server:3.2.1
    container_name: ds-standalone
    ports:
      - "12345:12345"
      - "25333:25333"
    environment:
      - DATABASE_TYPE=mysql
      - DATABASE_HOST=ds-mysql
      - DATABASE_PORT=3306
      - DATABASE_DATABASE=dolphinscheduler
      - DATABASE_USERNAME=root
      - DATABASE_PASSWORD=dolphinscheduler
      - REGISTRY_PLUGIN_DIR=/opt/dolphinscheduler/lib/plugin/registry/zookeeper
      - REGISTRY_SERVERS=ds-zookeeper:2181
    depends_on:
      - mysql
      - zookeeper
    networks:
      - ds-network

  mysql:
    image: mysql:5.7
    container_name: ds-mysql
    environment:
      MYSQL_ROOT_PASSWORD: dolphinscheduler
      MYSQL_DATABASE: dolphinscheduler
    ports:
      - "3306:3306"
    networks:
      - ds-network

networks:
  ds-network:
    driver: bridge
```

**Step 2：启动并验证**

```bash
# 启动所有服务
docker-compose up -d

# 查看容器状态
docker-compose ps

# 查看 DS 启动日志
docker logs -f ds-standalone
```

**验证结果**：浏览器访问 `http://localhost:12345/dolphinscheduler`，使用 `admin / dolphinscheduler123` 登录。看到仪表盘页面即部署成功。

**常见坑**：
- **端口冲突**：如果 12345 被占用，修改 compose 文件中的宿主机端口映射。
- **MySQL 启动慢**：取决于网络，首次拉取镜像可能较慢。DS 容器可能因 MySQL 未就绪而启动失败，手动 `docker-compose restart dolphinscheduler` 重试即可。
- **H2 vs MySQL**：官方 Standalone 镜像默认使用 H2 内嵌数据库，无需 MySQL 和 ZK 就能直接跑。上面配置为了展示完整架构特意挂载了 MySQL+ZK。

### 方案 B：二进制包部署

**步骤目标**：下载官方二进制包，手动配置环境，启动 Standalone 服务并验证。

**Step 1：下载并解压**

```bash
# 下载 DolphinScheduler 3.2.1 二进制包
wget https://dlcdn.apache.org/dolphinscheduler/3.2.1/apache-dolphinscheduler-3.2.1-bin.tar.gz

# 解压
tar -zxvf apache-dolphinscheduler-3.2.1-bin.tar.gz
cd apache-dolphinscheduler-3.2.1-bin
```

**Step 2：创建 MySQL 数据库**

```sql
-- 登录 MySQL
mysql -uroot -p

-- 创建数据库
CREATE DATABASE dolphinscheduler DEFAULT CHARACTER SET utf8mb4 DEFAULT COLLATE utf8mb4_general_ci;

-- 创建用户并授权
CREATE USER 'ds'@'%' IDENTIFIED BY 'DolphinScheduler@2024';
GRANT ALL PRIVILEGES ON dolphinscheduler.* TO 'ds'@'%';
FLUSH PRIVILEGES;
```

> **注意**：字符集必须为 `utf8mb4`，否则初始化脚本执行时会因 emoji 等特殊字符导致异常。

**Step 3：修改配置文件**

首先确认 `dolphinscheduler_env.sh` 中的环境变量：

```bash
# 编辑 bin/env/dolphinscheduler_env.sh
export JAVA_HOME=${JAVA_HOME:-/usr/local/jdk1.8.0_202}
export PYTHON_HOME=${PYTHON_HOME:-/usr/bin/python3}
export HADOOP_HOME=${HADOOP_HOME:-/opt/hadoop}
export SPARK_HOME=${SPARK_HOME:-/opt/spark}
```

然后配置数据源 `api-server/conf/datasource.properties`：

```properties
# api-server/conf/datasource.properties
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver
spring.datasource.url=jdbc:mysql://127.0.0.1:3306/dolphinscheduler?useUnicode=true&characterEncoding=utf8mb4&useSSL=false
spring.datasource.username=ds
spring.datasource.password=DolphinScheduler@2024
```

**Step 4：初始化数据库并启动**

```bash
# 执行数据库初始化脚本
bash tools/bin/upgrade-schema.sh

# 启动 Standalone 服务
bash bin/dolphinscheduler-daemon.sh start standalone-server

# 查看启动状态
bash bin/dolphinscheduler-daemon.sh status standalone-server
```

**验证结果**：访问 `http://localhost:12345/dolphinscheduler`，登录后进入首页。

**Step 5：停止服务**

```bash
bash bin/dolphinscheduler-daemon.sh stop standalone-server
```

**常见坑**：
- **`JAVA_HOME` 未设置或版本不对**：启动脚本会显式检查 `JAVA_HOME`，确保指向 JDK 1.8。
- **MySQL 连接失败**：检查 `datasource.properties` 中 URL、用户名、密码是否正确，检查 MySQL 是否允许远程连接。
- **`upgrade-schema.sh` 执行失败**：查看 `tools/bin/../logs/` 目录下的日志，常见原因为字符集不是 utf8mb4。

### 方案 C：源码编译

**步骤目标**：从 GitHub 克隆源码，本地编译构建，产出可部署的二进制包。

**Step 1：克隆仓库**

```bash
git clone https://github.com/apache/dolphinscheduler.git
cd dolphinscheduler
```

**Step 2：全量编译**

```bash
# Windows 使用 mvnw.cmd，Linux/Mac 使用 ./mvnw
# -Prelease：使用 release profile 打包
# -DskipTests：跳过单元测试以加快编译速度
./mvnw clean install -Prelease -DskipTests
```

编译成功后会输出类似信息：
```
[INFO] Reactor Summary for Apache DolphinScheduler 3.2.1:
[INFO] Apache DolphinScheduler ........................ SUCCESS [  0.542 s]
[INFO] dolphinscheduler-common ........................ SUCCESS [  8.123 s]
[INFO] dolphinscheduler-spi ........................... SUCCESS [  1.234 s]
...
[INFO] dolphinscheduler-dist .......................... SUCCESS [01:15 min]
[INFO] BUILD SUCCESS
```

**Step 3：查找构建产物**

编译产物位于：
```
dolphinscheduler-dist/target/apache-dolphinscheduler-3.2.1-SNAPSHOT-bin.tar.gz
```

解压后与方案 B 的二进制包结构完全一致，按同样方式部署即可。

**Step 4：直接运行 Standalone 模块（开发阶段推荐）**

```bash
cd dolphinscheduler-standalone-server
./mvnw spring-boot:run -Pstandalone
```

此方式无需打包，直接在 IDE 外也能快速启动 Standalone 服务。

**常见坑**：
- **首次编译慢**：Maven 需要下载全部依赖（1GB+），建议配置国内镜像源（阿里云 Maven 仓库）。
- **Node 编译失败**：前端模块需要 Node.js 16+，如果不需要前端修改，可加 `-pl '!dolphinscheduler-ui'` 跳过前端模块。
- **JDK 版本**：必须使用 JDK 1.8，如果用 JDK 11/17 编译，部分模块（如 `dolphinscheduler-extract`）会因模块化系统报错。

### 方案 D：IntelliJ IDEA 开发环境搭建

**步骤目标**：将 DolphinScheduler 源码导入 IDEA，配置断点调试 Standalone 模式。

**Step 1：导入项目**

1. 打开 IntelliJ IDEA → File → Open → 选择 DolphinScheduler 根目录（即包含根 `pom.xml` 的目录）。
2. IDEA 会自动识别为 Maven 多模块项目，点击"Trust Project"。
3. 等待 Maven 依赖下载和索引构建（首次约 10-15 分钟）。

**Step 2：配置 JDK**

```
File → Project Structure → Project
  Project SDK: 1.8 (选择本地安装的 JDK 1.8)
  Project language level: 8
```

**Step 3：配置 Run Configuration**

1. Run → Edit Configurations → 点击 "+" → 选择 "Application"。
2. 配置参数如下：

```
Name: StandaloneServer
Main class: org.apache.dolphinscheduler.standalone.StandaloneServer
VM options: -Dlogging.config=classpath:logback-spring.xml -Dspring.profiles.active=standalone
Working directory: $MODULE_DIR$
Use classpath of module: dolphinscheduler-standalone-server
Environment variables:
  DATABASE_TYPE=h2
  SPRING_DATASOURCE_DRIVER_CLASS_NAME=org.h2.Driver
  SPRING_DATASOURCE_URL=jdbc:h2:mem:dolphinscheduler;MODE=MySQL;DB_CLOSE_DELAY=-1
```

**Step 4：启动并调试**

1. 点击 Debug 按钮启动 StandaloneServer。
2. 在 `dolphinscheduler-service/src/main/java/org/apache/dolphinscheduler/service/process/ProcessServiceImpl.java` 的 `startProcess` 方法打上断点。
3. 通过 UI 创建一个简单工作流并运行，观察断点命中情况。

**Step 5：远程调试 Worker 任务**

对于集群模式下需要调试 Worker 执行逻辑的场景：

```
# Worker 节点启动时添加 JVM 参数
-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005

# IDEA 中配置 Remote JVM Debug
Host: Worker 节点 IP
Port: 5005
```

**常见坑**：
- **IDEA 内存不足**：DolphinScheduler 模块众多（26+），建议将 IDEA 的 VM 内存调整为 `-Xmx4096m`（Help → Edit Custom VM Options）。
- **Maven 依赖爆红**：右键根 `pom.xml` → Maven → Reload Project。
- **Lombok 告警**：确保已安装 Lombok 插件，并在 Settings → Compiler → Annotation Processors 中启用注解处理器。

## 4. 项目总结

### 四种部署方案对比

| 维度 | Docker Compose | 二进制包 | 源码编译 | IDE 开发 |
|------|---------------|---------|---------|---------|
| 上手难度 | ★☆☆☆☆ | ★★★☆☆ | ★★★★☆ | ★★★★☆ |
| 部署速度 | 1-5 分钟 | 10-20 分钟 | 20-40 分钟 | 20-30 分钟 |
| 配置灵活度 | 低（环境变量注入） | 高（直接编辑文件） | 最高（可修改任何代码） | 最高 |
| 可调试性 | 低（需进容器） | 中（远程调试） | 高（本地打断点） | 最高 |
| 适用场景 | 快速体验/Demo | 日常开发测试 | 二次开发/贡献 | 源码学习/调试 |
| 资源占用 | 高（需拉取多个镜像） | 中 | 中（仅编译时高） | 高（IDE + 编译） |
| 生产可用 | 不推荐 | 不推荐（需切集群模式） | 可选（需切集群模式） | 不可用 |

### 各方案适用场景

- **Docker Compose**：新人第一天入职快速体验、给产品经理演示调度能力、CI 流水线做集成测试。
- **二进制包**：团队统一的开发环境标准配置、日常任务逻辑开发与验证。
- **源码编译**：需要修改 SPI 插件、新增任务类型、参与开源社区提交 PR。
- **IDE 开发**：源码学习、架构理解、排查疑难 Bug、准备技术分享时的源码走读。

不适用场景：以上四种 Standalone 部署方案均**不可直接用于生产**。生产环境必须采用集群模式，独立部署各组件并配置 MySQL + ZK 高可用。

### 常见踩坑经验

1. **MySQL 字符集陷阱**：建库时若使用 `utf8` 而非 `utf8mb4`，初始化脚本中涉及 4 字节字符（如某些 emoji 测试数据）会报 `Incorrect string value` 错误。务必使用 `CREATE DATABASE ... DEFAULT CHARACTER SET utf8mb4`。

2. **JDK 版本引发的玄学问题**：有人在 JDK 11 下编译一切正常，但启动时抛 `java.lang.reflect.InaccessibleObjectException`，这是 Java 9+ 模块化系统对反射的限制。DolphinScheduler 部分模块使用了反射，JDK 1.8 是最稳妥的选择。

3. **Zookeeper 超时导致服务频繁重启**：集群模式下如果 ZK 网络不稳定，Worker 会因心跳超时被 Master 标记为离线，导致任务不断重分配。适当地将配置 `registry.zookeeper.session.timeout` 从默认的 30s 调大到 60s 可以缓解。同时建议将 ZK 与 DS 各组件部署在同一网段，避免跨机房延迟导致的误判。

4. **内存配置不足导致 OOM**：Standalone 模式的默认 JVM 堆内存为 1GB，如果同时运行多个复杂任务（如大数据量的 Spark/DataX 任务），很容易触发 OutOfMemoryError。可以通过修改 `standalone-server/conf/application.yaml` 中的 JVM 参数，将 `-Xmx` 调大到 2GB 或 4GB，但前提是宿主机有足够的物理内存。

### 思考题
