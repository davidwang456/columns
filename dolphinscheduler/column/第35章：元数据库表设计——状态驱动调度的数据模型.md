## 一、项目背景

大麦数据（Damai）的运维值班工程师周凯接到了凌晨两点的告警电话："daily_sales_report 工作流没有按时跑！财务部门已经在催销售日报了。"周凯睡眼惺忪地打开笔记本，开始逐层排查：第一步，检查 Cron 表达式——`0 0 2 * * ?`，每天凌晨两点触发，配置正确。第二步，确认定时调度状态——`t_ds_schedules.release_state = 1`（上线状态），没有问题。第三步，翻阅 Master 节点日志——`grep daily_sales_report /var/log/dolphinscheduler/master-server.log`——没有异常，也没有错误。周凯开始冒冷汗。

第四步，他连接到元数据库，敲下了一条决定性的 SQL：

```sql
SELECT COUNT(*) FROM t_ds_command WHERE state = 0;
-- 结果: 5238
```

五千多行命令在排队，而 `daily_sales_report` 那条 `SCHEDULER` 类型的命令排在第 4127 位。Master 以每秒约 100 条的速率消费命令队列——这意味着这条工作流还需要约 41 秒才能被调度引擎拾取。实际上它并没有"挂掉"，只是在"排队"。

问题解决了，但周凯心底浮起一个更深层的困惑：这条命令到底从哪儿来的？是谁、在什么时刻、以什么形式把它写进了 `t_ds_command` 表？从用户在 UI 上画一个 DAG、配置一个 Cron，到凌晨两点 Master 从命令表里拉出一条记录开始执行——中间的每一步数据都是如何流转、存在哪张表的什么字段里的？理解 DolphinScheduler 的元数据模型——定义 → 调度 → 命令 → 实例 → 任务实例——这条数据链路——不是"加分项"，而是严肃调试和运维的必修课。

## 二、项目设计——剧本式交锋对话

三人在会议室白板前讨论上周的凌晨排查事故，话题从"为什么工作流没跑"自然过渡到"命令到底是怎么来的"。

**小胖翘着二郎腿，嘴里咬着吸管：**"数据库不就是存点配置和运行记录嘛，几张表的事儿。`t_ds_process_definition` 存工作流定义，`t_ds_task_instance` 存任务运行结果——跟记流水账一样。哪需要一百多张表？我猜其他那些都是系统生成的冗余表。"

**小白推了推眼镜，一连串问题砸了过去：**"那命令队列 `t_ds_command` 是怎么变成工作流实例的？定义表 `t_ds_process_definition` 和实例表 `t_ds_process_instance` 是什么关系——是同一个表加个 `is_running` 字段，还是两张完全独立的表？DAG 的拓扑结构——节点和边的 JSON——存在哪个表、哪个字段里？依赖节点 depend 跨工作流依赖的数据存在哪个表？一百多张表里，核心的就那十几张，到底是哪几张？"

**大师从保温杯里倒出茶水，缓缓开口：**"DS 的数据模型就像工厂的 ERP 系统。BOM 表（Bill of Materials，物料清单）描述了产品的结构——一台发动机由哪些零件组成、零件之间怎么装配——这对应定义表 `t_ds_process_definition` 和 `t_ds_task_definition`，它是你画在 UI 上的 DAG 蓝图。工单表记录每次实际的生产过程——2024年1月15日第3批次生产了50台——这对应实例表 `t_ds_process_instance` 和 `t_ds_task_instance`，每次运行都是新的一行。排产表决定什么时候开工——每天凌晨两点生产一批——这对应 `t_ds_schedules`。而待办表就是流水线上的任务卡——'工序A已完成，请开始工序B'——这对应 `t_ds_command`。每个表各司其职，合在一起就是一个完整的数字工厂。"

**小胖挠头：**"那为什么不把定义和实例放在一张表里？比如加一个 `run_id` 字段区分不同批次？我看很多小系统就是这么干的。"

**大师在白板上画了两条并行的线：**"这就是 DS 设计中最核心的一个原则：**定义与实例分离**。定义表的数据是**用户资产**——工作流画成什么样、任务参数配置成什么样，这是业务逻辑的沉淀，一年不变都很正常。实例表的数据是**运行痕迹**——每天凌晨两点跑一次，一次就生成一行，一个月就是上百行。如果把两者混在同一张表，定义要改一个任务名——删掉重建？不行，历史实例的数据还引用了旧的任务名。更新？那历史数据也跟着变了，审计链路就断了。分开之后，定义的 version 字段自增——v1、v2、v3 各版本并存，运行的实例引用的是当时那个版本的定义码——谁改了工作流、什么时候改的、改成什么样——清清楚楚。"

**小白追问：**"那命令表又是什么角色？为什么需要一个 `t_ds_command` 中间表？定时调度直接触发执行不行吗？"

**大师点头：**"命令表是 DS 架构中最优雅的一层解耦。你想——触发工作流执行的来源太多了：用户手动点击'运行'、Cron 定时触发、补数命令、失败重试、从属节点触发、API 调用……如果每个触发源都直接调用 Master 的执行逻辑，Master 就要同时处理'接收触发'和'执行编排'两件事——耦合得一塌糊涂。命令表把一个异步的、生产-消费模型引入进来：所有的触发源都只做一件事——**往 `t_ds_command` 表里 INSERT 一行**——然后拍拍屁股走人。Master 作为独立的消费者，用一个轮询线程不停地 `SELECT ... WHERE state = 0 LIMIT 100`，把命令一条条拉出来消费。触发方不知道 Master 有几个实例、在哪个节点上、现在是忙是闲；Master 不知道这条命令是谁放的、为什么要放。两者通过一张表实现完美解耦——这是我们常说的**命令队列模式**。"

**小胖恍然大悟：**"哦！所以那天凌晨的问题——5000 条命令卡住——本质上是消费端处理不过来？不是生产端写错了？"

**大师：**"对。命令表里的 `state` 字段是关键的信号：`0=WAIT` 表示命令在排队，`1=PROCESSING` 表示 Master 正在处理，处理完后直接 `DELETE` 掉。所以排查'工作流为什么不跑'的第一站不是日志文件，而是这条 SQL：

```sql
SELECT state, COUNT(*) FROM t_ds_command GROUP BY state;
```

如果 `state=0` 数量持续增长，说明 Master 消费速度跟不上生产速度——可能是一个大工作流占用了 Master 线程池，也可能是 Master 节点资源不足。"

**技术映射**：`t_ds_process_definition` + `t_ds_task_definition` + `t_ds_process_task_relation` = 数字工厂的 BOM 表（蓝图）；`t_ds_process_instance` + `t_ds_task_instance` = 工单表（生产记录）；`t_ds_schedules` = 排产表（排期）；`t_ds_command` = 待办/任务卡（异步解耦）；定义与实例分离 = 代码的"类"与"对象"——定义是 class，实例是 new 出来的 object。

## 三、项目实战

### Step 1: 核心实体 ER 关系总览

DS 超过 100 张表，但核心表可以归纳为五大类，围绕一条清晰的数据链路：

```
t_ds_process_definition ──1:N── t_ds_task_definition
        │                              │
        │ 1:N                          │ 1:N
        ▼                              ▼
t_ds_process_instance  ◄── t_ds_task_instance
        ▲
        │
t_ds_schedules ──triggers──▶ t_ds_command ──consumed by──▶ Master

t_ds_task_definition ──N:M── t_ds_process_task_relation
  (边通过 pre_task_code / post_task_code 定义)

t_ds_process_instance ──N:M── t_ds_relation_process_instance
  (依赖节点的跨工作流实例关联数据)
```

**五类核心表**：

| 类别 | 代表表 | 职责 |
|------|--------|------|
| 定义表 | `t_ds_process_definition`、`t_ds_task_definition`、`t_ds_process_task_relation` | 用户创建的静态蓝图（模板） |
| 实例表 | `t_ds_process_instance`、`t_ds_task_instance`、`t_ds_relation_process_instance` | 每次运行生成的动态记录 |
| 调度表 | `t_ds_schedules`、`t_ds_command` | 定时触发 + 异步命令队列 |
| 配置表 | `t_ds_tenant`、`t_ds_queue`、`t_ds_worker_group`、`t_ds_alert_group`、`t_ds_datasource` | 环境与资源映射 |
| 安全表 | `t_ds_user`、`t_ds_project`、`t_ds_access_token`、`t_ds_relation_project_user` | 用户、项目与权限 |

### Step 2: 从定义到执行——追踪一条工作流的完整数据链路

打开数据库客户端，跟随 SQL 走一遍完整的数据流转：

```sql
-- ① 用户创建了一个工作流定义
SELECT code, name, version, release_state, global_params, create_time
FROM t_ds_process_definition
WHERE name = 'daily_sales_report';
-- code 是全局唯一标识（数字），version 自增（每次保存+1）
-- global_params 是 JSON 格式的全局参数定义

-- ② 这个工作流包含哪些任务定义
SELECT code, name, task_type, task_params, priority
FROM t_ds_task_definition
WHERE process_definition_code = 12345;
-- task_type: 'SHELL' / 'SQL' / 'SPARK' / 'DEPENDENT' ...
-- task_params: JSON，不同 task_type 的结构不同

-- ③ 任务之间的边（DAG 连线）
SELECT pre_task_code, post_task_code, pre_task_version, post_task_version,
       process_definition_code, condition_type
FROM t_ds_process_task_relation
WHERE process_definition_code = 12345;
-- pre_task_code = 0 表示该节点是起始节点（无上游）

-- ④ 定时调度配置
SELECT crontab, start_time, end_time, release_state
FROM t_ds_schedules
WHERE process_definition_code = 12345;
-- release_state: 0=下线, 1=上线
```

此时凌晨两点到了，Quartz 触发 Cron 回调：

```sql
-- ⑤ Cron 触发 → API 生成一条 Command
SELECT id, command_type, process_definition_code, command_params,
       state, priority, create_time
FROM t_ds_command
WHERE process_definition_code = 12345
ORDER BY create_time DESC
LIMIT 5;
-- command_type: 1=SCHEDULER(定时触发), 0=START_PROCESS(手动运行),
--   5=REPEAT_RUNNING(补数), 6=START_FAILURE_TASK_PROCESS(失败重跑)
-- state: 0=WAIT(等待消费), 1=PROCESSING(Master处理中), 执行完则DELETE
-- command_params: JSON, 携带启动参数和业务流程日期

-- ⑥ Master 轮询消费 Command → 创建工作流实例
SELECT id, name, process_definition_code, state, host, start_time,
       end_time, run_times, biz_date
FROM t_ds_process_instance
WHERE process_definition_code = 12345
ORDER BY id DESC
LIMIT 1;
-- state: 运行中态 → 最终 SUCCESS(7) / FAILURE(6)

-- ⑦ 工作流实例下的任务实例
SELECT id, name, task_type, state, start_time, end_time, log_path,
       retry_times, host
FROM t_ds_task_instance
WHERE process_instance_id = <上面查到的id>
ORDER BY id;
-- state 枚举: SUBMITTED_SUCCESS(0) → DISPATCH(2) →
--   RUNNING_EXECUTION(3) → SUCCESS(6) / FAILURE(7)
```

### Step 3: 命令表——调度系统的异步解耦核心

`t_ds_command` 是调度系统的"消息队列"，用一张数据库表实现了生产者-消费者模式：

```sql
CREATE TABLE t_ds_command (
    id INT PRIMARY KEY AUTO_INCREMENT,
    command_type TINYINT NOT NULL,          -- 0=START_PROCESS, 1=SCHEDULER,
                                            -- 4=RECOVER_TOLERANCE_FAULT_PROCESS,
                                            -- 5=REPEAT_RUNNING, 6=START_FAILURE_TASK_PROCESS
                                            -- 9=RECOVER_SERIAL_WAIT
    process_definition_code BIGINT NOT NULL,
    command_params TEXT,                    -- JSON: {"processDefinitionCode":12345,
                                            --   "startParams":{"biz_date":"20240115"}}
    task_depend_type TINYINT,               -- 任务依赖策略（执行时）
    failure_strategy TINYINT,               -- CONTINUE(0) / END(1)
    warning_type TINYINT,                   -- NONE(0) / SUCCESS(1) / FAILURE(2) / ALL(3)
    priority TINYINT DEFAULT 2,             -- HIGHEST(0) / HIGH(1) / MEDIUM(2) / LOW(3) / LOWEST(4)
    worker_group VARCHAR(64),               -- 目标 WorkerGroup 名称
    state TINYINT DEFAULT 0,                -- 0=WAIT, 1=PROCESSING, 2=SUCCESS(随即DELETE)
    create_time DATETIME,
    update_time DATETIME
);
```

Master 的消费逻辑伪代码：

```java
// MasterSchedulerBootstrap 主循环
while (!stopped) {
    // 按优先级 + 创建时间排序，每次拉取100条
    List<Command> commands = commandMapper
        .selectList(new LambdaQueryWrapper<Command>()
            .eq(Command::getState, 0)         // 只取待消费
            .orderByAsc(Command::getPriority)  // 优先级高者先
            .orderByAsc(Command::getCreateTime)
            .last("LIMIT 100"));

    for (Command cmd : commands) {
        // CAS 更新状态为 PROCESSING，避免多个 Master 实例重复消费
        int updated = commandMapper.update(null,
            new LambdaUpdateWrapper<Command>()
                .eq(Command::getId, cmd.getId())
                .eq(Command::getState, 0)
                .set(Command::getState, 1));
        if (updated == 0) continue; // 已被其他 Master 消费

        // 创建 ProcessInstance → 进入 DAG 状态机编排
        workflowExecuteRunnablePool.submit(() -> executeCommand(cmd));
    }
    Thread.sleep(1000); // 轮询间隔
}
```

`state` 字段是关键——`0→1→DELETE` 构成了一个轻量级的分布式任务队列。`priority` 字段决定了消费顺序，`command_type` 决定了处理逻辑的分支。

### Step 4: 任务实例的状态机——状态驱动的执行模型

`t_ds_task_instance.state` 是一个典型的状态机，所有任务生命周期都由状态流转驱动：

```
SUBMITTED_SUCCESS(0)     ← 任务已持久化到数据库，等待分发
  → SUBMITTED_SUCCESS_X(1) ← 任务已分发到 Worker
  → DISPATCH(2)          ← Worker 已接收
  → RUNNING_EXECUTION(3) ← Worker 正在执行物理任务
  → SUCCESS(6) / FAILURE(7) / KILL(8) / STOP(9) / PAUSE(4)

异常路径：
  → NEED_FAULT_TOLERANCE(5) ← Worker 宕机，需要故障转移
```

运维常用查询：

```sql
-- 当前工作流实例的任务状态分布
SELECT state, COUNT(*) as cnt
FROM t_ds_task_instance
WHERE process_instance_id = ?
GROUP BY state;

-- 某个任务的完整生命周期耗时
SELECT state, start_time, end_time,
       TIMESTAMPDIFF(SECOND, start_time, end_time) as duration_sec,
       log_path
FROM t_ds_task_instance
WHERE id = ?;

-- 过去24小时最慢的10个任务（性能诊断）
SELECT name, task_type,
       TIMESTAMPDIFF(SECOND, start_time, end_time) as duration_s
FROM t_ds_task_instance
WHERE start_time > DATE_SUB(NOW(), INTERVAL 24 HOUR)
  AND state IN (6, 7)
ORDER BY duration_s DESC
LIMIT 10;
```

**重要警示**：永远不要手动 `UPDATE t_ds_task_instance SET state = 6`。DS 的状态机逻辑不仅更新 `state` 字段，还会同时写日志、发回调、更新上游依赖、触发下游就绪判断——手动改一行数据会导致状态机错乱，轻则工作流卡住，重则数据一致性崩溃。

### Step 5: 依赖节点（Dependent）的数据模型

Dependent 节点的依赖配置存储为 JSON：

```sql
-- t_ds_task_definition.task_params 字段内容（JSON）:
{
  "dependence": {
    "relation": "AND",               -- 多依赖项的逻辑关系: AND / OR
    "dependTaskList": [{
      "dependItemList": [{
        "projectCode": 100,          -- 依赖的目标项目
        "definitionCode": 200,       -- 依赖的目标工作流定义
        "depTasks": "ALL",           -- 依赖全部任务 或 指定任务名
        "cycle": "day",              -- 时间周期: day / hour / week / month
        "dateValue": "last1Days",    -- 相对时间偏移
        "state": "SUCCESS"           -- 要求的完成状态
      }]
    }]
  }
}
```

运行时，Dependent 任务处理逻辑执行如下查询：

```sql
-- 检查目标工作流在上一个业务日期的实例是否已达 SUCCESS 状态
SELECT state FROM t_ds_process_instance
WHERE process_definition_code = 200
  AND biz_date = ?   -- 由 cycle + dateValue 计算得出
ORDER BY id DESC
LIMIT 1;
```

跨工作流的依赖关系记录在 `t_ds_relation_process_instance` 表中——它关联了父流程实例、子流程实例和依赖节点任务实例三者，是多对多的关系表。

### Step 6: 软删除模式——一切皆标记，永不真删

DS 极少物理删除行（`t_ds_command` 是个例外），绝大多数表使用 `flag` 字段实现逻辑删除：

```sql
-- "删除"一个工作流定义（实际上是软删除）
UPDATE t_ds_process_definition
SET flag = 1, update_time = NOW()
WHERE code = 12345;

-- "删除"一个项目
UPDATE t_ds_project SET flag = 1 WHERE code = 100;

-- 所有查询的标准模式——必须加 flag = 0
SELECT * FROM t_ds_process_definition WHERE flag = 0;
SELECT * FROM t_ds_task_definition WHERE flag = 0;
SELECT * FROM t_ds_datasource WHERE flag = 0;
```

软删除的三重动机：
1. **审计追溯**：谁在什么时候删了什么——`flag=1` 保留了全部记录；
2. **引用完整性**：正在运行的工作流实例引用了已被"删除"的定义——物理删除会导致外键级联异常；
3. **误删恢复**：一条 `UPDATE flag = 0` 就能撤销，比从备份里捞数据快一万倍。

DS 中 `flag` 的值约定：`0 = 有效（启用）`，`1 = 无效（已删除）`。另外，所有表统一包含 `create_time` 和 `update_time` 两个审计字段，贯穿全部 100+ 张表。

### Step 7: 日常运维必备 SQL 速查

```sql
-- 最近7天失败次数最多的10个工作流（稳定性排行）
SELECT d.name, COUNT(*) as failure_count
FROM t_ds_process_instance i
JOIN t_ds_process_definition d ON i.process_definition_code = d.code
WHERE i.state = 6    -- FAILURE
  AND i.start_time > DATE_SUB(NOW(), INTERVAL 7 DAY)
GROUP BY d.name
ORDER BY failure_count DESC
LIMIT 10;

-- Worker 负载分布（各 Worker 当前运行中任务数）
SELECT worker_group, host, COUNT(*) as running_tasks
FROM t_ds_task_instance
WHERE state = 3    -- RUNNING_EXECUTION
GROUP BY worker_group, host;

-- 待消费命令的分布（判断 Master 是否拥堵）
SELECT command_type, priority, COUNT(*) as cnt
FROM t_ds_command
WHERE state = 0    -- WAIT
GROUP BY command_type, priority;

-- 数据源使用统计
SELECT type, COUNT(*) as cnt
FROM t_ds_datasource
WHERE flag = 0
GROUP BY type;
```

### Step 8: 历史数据清理与分区优化

DS 不会自动清理历史实例数据——时间一长，`t_ds_task_instance` 和 `t_ds_process_instance` 可能膨胀到百万行甚至千万行，导致 UI 加载变慢、查询超时：

```sql
-- 清理90天前的任务实例（建议做成定时任务每周执行）
DELETE FROM t_ds_task_instance
WHERE end_time < DATE_SUB(NOW(), INTERVAL 90 DAY) AND state IN (6, 7, 8, 9);

DELETE FROM t_ds_process_instance
WHERE end_time < DATE_SUB(NOW(), INTERVAL 90 DAY) AND state IN (6, 7, 8, 9);

-- 清理7天前的已消费命令（留作问题排查窗口）
DELETE FROM t_ds_command WHERE create_time < DATE_SUB(NOW(), INTERVAL 7 DAY);
```

对于日均 5000+ 任务实例的大型部署，建议启用 MySQL 表分区：

```sql
-- 按 start_time 按月分区（示例）
ALTER TABLE t_ds_task_instance PARTITION BY RANGE (TO_DAYS(start_time)) (
    PARTITION p202401 VALUES LESS THAN (TO_DAYS('2024-02-01')),
    PARTITION p202402 VALUES LESS THAN (TO_DAYS('2024-03-01'))
);
```

### Step 9: 元数据库版本升级机制

DS 的 SQL 迁移脚本统一存放在 `dolphinscheduler-dao/src/main/resources/sql/` 目录下：

```
sql/
├── dolphinscheduler_h2.sql      ← H2 全新安装
├── dolphinscheduler_mysql.sql   ← MySQL 全新安装
├── dolphinscheduler_pg.sql      ← PostgreSQL 全新安装
├── upgrade/
│   ├── 3.0.0_schema/
│   └── 3.1.0_schema/
│       └── mysql/
│           └── dolphinscheduler_ddl.sql  ← ALTER TABLE ... 增量变更
```

升级命令通过 `dolphinscheduler-tools` 模块执行：

```bash
sh bin/upgrade-schema.sh
# 背后逻辑: 读取 upgrade/ 目录下的版本号 → 依次执行高于当前版本的 DDL
```

DS 的表结构变更遵循**追加式演进**原则——只 ADD COLUMN，不 DROP COLUMN，不 MODIFY COLUMN 类型。这保证了跨版本滚动升级时，新老服务可以同时读写同一张表而不会因为 schema 不一致崩溃。

### Step 10: 常见踩坑与避坑指南

| 踩坑行为 | 后果 | 正确做法 |
|---------|------|---------|
| 直接 `UPDATE task_instance SET state=6` | 状态机断裂，Master 卡在等待状态 | 让 DS 管理状态，通过 API 或 UI 操作 |
| 忘记定期清理历史数据 | 实例表膨胀到千万行，UI 翻页超时 | 部署定时清理脚本，生产环境启用分区表 |
| 手动 `DELETE FROM process_definition` | 正在运行的实例引用异常 | 使用 UI 的"删除"按钮（内部走软删除） |
| 直接读取 `datasource` 表的密码字段 | 看到的是 AES 加密后的密文 | 通过 API 获取，API 内部会解密 |
| Master 宕机后不清理积压命令 | 重启后瞬间涌入数千条命令，Master 被打爆 | 先评估积压量，必要时手动清理过期命令 |
| 改 `t_ds_process_definition.version` 字段 | 工作流实例的引用链路发生错乱 | version 由系统自动管理，保存即自增 |

## 四、项目总结

回看 DS 的元数据模型，它的设计哲学可以用五个原则概括：

**1. 定义与实例分离**：`t_ds_process_definition` 是蓝图，`t_ds_process_instance` 是产品。每次运行生成新的一行实例数据，互不影响。版本号自增，历史版本留痕——本质上是一个不可变日志（Event Sourcing）模式的变体。

**2. 状态驱动执行**：任务不靠回调也不是消息驱动——全靠 `t_ds_task_instance.state` 字段的状态机驱动。`SUBMITTED → DISPATCH → RUNNING → SUCCESS/FAILURE`，每个状态都有明确的转换条件和后续动作。状态机是 DS 可靠性的基石。

**3. 命令队列解耦**：`t_ds_command` 是一张充当消息队列的数据库表。触发方只管写，Master 只管消费。`state` 字段的 `0→1→DELETE` 充当了 ACK 信号和去重机制。这是典型的生产者-消费者模式在数据库层的最佳实践。

**4. 软删除为常态**：`flag=0` 是有效，`flag=1` 是已删——保留审计链、保证引用完整性、支持一键恢复。所有表统一携带 `create_time` 和 `update_time`。

**5. 元数据即文档**：`dolphinscheduler-dao/src/main/resources/sql/` 目录就是活的 schema 文档。搞不清楚某个字段的含义？直接打开 SQL 文件看注释——比任何博客都权威。

**数据全生命周期链路图**：

```
用户创建 → t_ds_process_definition (静态蓝图, version=vN)
  ↓
配置定时 → t_ds_schedules (Cron + 起止时间)
  ↓
Cron触发 → t_ds_command (生产端写入, state=WAIT)
  ↓
Master消费 → t_ds_process_instance (运行时实例, biz_date= 业务日期)
  ↓
任务分派 → t_ds_task_instance (每任务一行, state 按状态机流转)
  ↓
执行完成 → Master将实例状态置为 SUCCESS/FAILURE
  ↓
历史归档 → 90天后批量清理（或保留用于审计）
```

**核心表速查矩阵**：

| 想知道... | 去这里找 |
|-----------|---------|
| 工作流是怎么定义的（DAG结构、参数） | `t_ds_process_definition` + `t_ds_task_definition` + `t_ds_process_task_relation` |
| 某次执行的结果（成功/失败、耗时） | `t_ds_process_instance` + `t_ds_task_instance` |
| 为什么工作流没有按时触发 | `t_ds_schedules`（Cron+状态）+ `t_ds_command`（队列积压量） |
| 数据源的连接信息（加密） | `t_ds_datasource` |
| 用户有什么项目权限 | `t_ds_relation_project_user` |

**思考题**：

1. 如果 `t_ds_command` 表用 Redis List 或 Kafka 替代，会带来哪些收益和哪些新的问题？在什么场景下，用数据库表做命令队列比用专门的消息队列更合适？

2. DS 的 `t_ds_process_definition` 用 `version` 字段做版本管理——每次保存自增，老版本保留。如果要实现"回滚到 v2 版本的定义"，在现有表结构下应该执行什么 SQL？需要同时修改哪些关联表的数据？
