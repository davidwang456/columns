# 第25章：日志管理与故障排查实战

## 一、项目背景

大麦数据（Damai）的 DolphinScheduler 集群过去一个月连遭四起事故：周一，20 个工作流卡在"已提交"两小时，根因是 ZK 连接超时；周三，Spark 任务静默失败但 DS 显示"成功"——退出码为 0 但输出为空；周五，API 持续 503 半小时，终因 Master OOM 而 API 侧毫无感知；周日，夏令时切换导致凌晨 2 点的调度漂移到 3 点。

复盘发现，团队 80% 的排障时间花在"找日志在哪"——SSH 逐台登录、手工 grep、盲猜路径。亟需三件事：（1）统一日志入口；（2）每种故障的系统化排查 SOP；（3）主动告警代替被动救火。

## 二、项目设计——剧本式交锋对话

**小胖**："日志不就是 `tail -f` 嘛！出问题直接 SSH 看 `dolphinscheduler-master.log`，一眼找到错误。"

**小白**："3 个 Master、5 个 Worker 分散不同网段，你准备开几个终端？日志级别怎么动态调整？任务实例日志和 Worker 日志是什么关系？为什么不把日志集中到 ELK 统一检索？"

**大师**："日志排查就像刑侦破案——任务实例日志是案发现场的**第一线索**，告诉你 exit code 和报错内容；Master/Worker 服务日志是**证人证词**，告诉你系统在那一刻做了什么决策、为什么任务分配给了 Worker-3；GC 日志和 Metrics 是**监控录像**，提供完整时间线。"

大师接着说："ELK（Filebeat → ES → Kibana）就像把所有证据汇总到一个案情分析室。但前提是日志格式统一、多行异常正确合并，否则就像证据撕碎后扔进碎纸机——ELK 也无能为力。每种常见故障都需要标准 SOP——工作流卡住、任务失败、调度不触发、API 503、Worker 掉线——有了 SOP，凌晨三点也能按清单五分钟定位。"

**小胖**："原来日志不只是 `tail -f`……得先规划目录结构、统一格式、配置轮转，再谈集中化。"

## 三、项目实战

### Step 1：日志目录结构与分类

```
DS_INSTALL_DIR/
├── api-server/logs/        # API 主日志 + 错误日志
├── master-server/logs/     # Master 主日志 + GC 日志
├── worker-server/logs/     # Worker 主日志 + 错误日志 + 任务日志
├── alert-server/logs/      # 告警服务日志
```

四类日志：**服务日志**（模块自身运行状态）、**任务实例日志**（Worker 捕获的脚本 stdout/stderr，路径存于 `t_ds_task_instance.log_path`）、**访问日志**（API HTTP 请求）、**GC 日志**（JVM 内存诊断）。

### Step 2：Logback 生产配置要点

```xml
<appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
    <file>logs/dolphinscheduler-master.log</file>
    <rollingPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedRollingPolicy">
        <fileNamePattern>logs/dolphinscheduler-master.%d{yyyy-MM-dd}.%i.log</fileNamePattern>
        <maxFileSize>100MB</maxFileSize>
        <maxHistory>30</maxHistory>
        <totalSizeCap>10GB</totalSizeCap>
    </rollingPolicy>
</appender>
<root level="INFO"><appender-ref ref="FILE"/></root>
<!-- 排查时临时改为 DEBUG，上线前务必恢复 INFO -->
<logger name="org.apache.dolphinscheduler.server.master.runner" level="DEBUG"/>
```

铁律：生产环境用 INFO，曾有人 DEBUG 开了一周，120GB 日志撑爆 Worker 磁盘致全部掉线。

### Step 3-7：四大故障 SOP 速查

**SOP-1：工作流卡"已提交"**
```bash
# 查命令状态: 0=待处理 1=处理中 2=成功
mysql> SELECT id, state FROM t_ds_command WHERE process_definition_id=X ORDER BY create_time DESC LIMIT 5;
# 查 Master Leader
$ zkCli.sh get /dolphinscheduler/master/leader
# 查 Master 日志
$ grep "command.*<id>" dolphinscheduler-master.log | tail -20
```
常见根因：无 Master Leader → 重启 Master；Worker 全忙 → 增大 `worker.exec.threads`；command 卡在 processing → Master 崩溃后 ZK 锁未释放。

**SOP-2：任务执行失败**
```bash
# UI"查看日志"聚焦最后 50 行 → 看 exit code 和错误栈
# 交叉查 Worker 日志
$ grep "taskId=7890" dolphinscheduler-worker.log -A 10
```

| 日志关键词 | 原因 | 修复 |
|-----------|------|------|
| `command not found: python3` | PYTHON_HOME 错误 | 修正 `dolphinscheduler_env.sh` |
| `Permission denied` | 租户无执行权限 | `chmod +x` / `chown` |
| `No space left on device` | Worker 磁盘满 | 清理日志/扩容 |
| `Connection refused` | 数据源不可达 | 检查防火墙/白名单 |
| `OutOfMemoryError` | 任务堆内存不足 | 增大 `-Xmx` |
| exit=0 但输出为空 | Spark/Flink 静默失败 | 脚本末尾加产出校验 |

**SOP-3：调度不触发**
```bash
# 确认调度"在线"、cron 合法、服务器时区正确
$ timedatectl | grep "Time zone"
$ grep "quartz" dolphinscheduler-master.log | tail -20
```
常见原因：调度离线（编辑后忘上线）、cron 非法日期（如 25 号在 2 月）、UTC 时区差 8 小时、上次实例未结束（默认禁止并行）。

**SOP-4：API 返回 503**
```bash
$ curl http://api:12345/dolphinscheduler/health
$ jps -l | grep ApiApplicationServer
$ tail -200 dolphinscheduler-api.log | grep -E "ERROR|503"
```
常见原因：Master 宕机（API 代理请求透传失败）、DB 连接池耗尽（增大 `hikari.maximum-pool-size`）、Jetty 线程池满。

### Step 8-9：集中式日志 + Kibana 查询

**Filebeat 配置要点**：

```yaml
filebeat.inputs:
  - type: log
    paths: /opt/dolphinscheduler/*/logs/*.log
    multiline:               # 必须配置！否则异常栈被拆成多行
      pattern: '^\d{4}-\d{2}-\d{2}'
      negate: true
      match: after
output.elasticsearch:
  hosts: ["http://es:9200"]
  index: "ds-logs-%{+yyyy.MM.dd}"
```

**Kibana 常用查询**：`level: ERROR AND @timestamp: [now-1h TO now]`（近期错误）；`workflow: "日报" AND exit_code: >0`（特定工作流失败）；`message: "leader" OR "failover"`（Master 切换）；`log_type: gc AND pause: >1.0`（GC 暂停 > 1 秒）。

### Step 10：主动告警

用 ElastAlert 或 Grafana Alerting：ERROR 速率 > 10 条/分钟持续 5 分钟 → P2；任意 `OutOfMemoryError` → P0-Critical；DB 连接拒绝 > 5 次/10 分钟 → P1；Worker 无心跳 > 60 秒 → P1。

### Step 11：关键踩坑

1. **多行合并**：Filebeat 不配 `multiline`，异常栈拆成几十条，搜索 `NullPointerException` 只能看到异常名看不见调用链。
2. **日志轮转**：必须同时设 `maxHistory` + `totalSizeCap` 双保险。
3. **时区一致性**：DS、DB、ES、Kibana 统一 `Asia/Shanghai`，否则日志时间轴错乱。
4. **Filebeat 权限**：将 filebeat 用户加入 DS 用户组，确保可读 Worker 任务日志。

## 四、项目总结

**日志体系全景**：DS 日志分四大类——服务日志、任务实例日志、访问日志、GC 日志，各有明确位置和用途。任务日志通过 DB 的 `log_path` 字段与磁盘文件关联，前端"查看日志"本质是读取 Worker 本地的 `dolphinscheduler-worker-task-{id}.log`。

**故障排查速查表**：

| 故障现象 | 第一检查点 | 常见根因 | 快速修复 |
|---------|-----------|---------|---------|
| 工作流卡"已提交" | Master Leader | ZK 超时/Master 崩溃 | 重启 Master |
| 任务失败 | 任务日志最后 50 行 | 环境变量/权限/磁盘 | 修正配置/chmod |
| 调度不触发 | 定时管理页状态 | 离线/时区/并行策略 | 上线/修时区 |
| API 503 | API+Master 进程 | Master OOM/连接池满 | 重启/增大连接池 |
| Worker 掉线 | ZK Worker 节点 | 磁盘满/网络/OOM | 清理/检查网络 |

**集中化日志选型**：ELK 功能全但资源消耗大（日均 > 50GB 日志）；Loki+Grafana 轻量兼容 Prometheus；云上直接用 CloudWatch/Azure Monitor 零运维。

**排障成熟度演进**：阶段一"被动救火"（手动 grep，MTTR 2h）→ 阶段二"主动告警"（ERROR 速率、OOM 关键词自动告警，MTTR 30min）→ 阶段三"预测运维"（历史日志+指标训练的异常检测模型，提前预警堆使用率上涨），这才是日志管理的终极目标。
