# 第16章：集群架构部署与高可用设计

> **定位**：中级篇开篇之作，从单机Standalone跨入生产级集群部署。
> **核心内容**：集群拓扑设计、分布式组件协作、HA高可用机制、12步部署完整实战、故障演练与横向扩容。
> **实战目标**：在3台物理节点上部署DolphinScheduler集群，实现Master双活、Worker动态扩缩、99.9%可用性。

---

## 1. 项目背景

大麦电商自上线DolphinScheduler以来，Standalone模式平稳运行了一个月。基础篇中搭建的日报、周报、月报三级调度体系已经产出稳定数据，CTO在月度总结会上还表扬了调度平台的效率提升。

但好景不长。上周三凌晨3点17分，唯一的Master进程因JVM内存溢出悄然崩溃。没有备用的Master接管，整个调度系统瘫痪——11条工作流全部卡死在"等待执行"状态，订单日报、用户分析、周报预览全部中断。而此时运维工程师正在熟睡，直到早晨7点才发现告警邮件（更讽刺的是，告警邮件也因为Alert Server随Master一起挂了而没能发出去）。业务方等了两个小时没有收到报表，运营总监的电话直接打到了CTO的手机上。

祸不单行。上周五正值"618预热大促"，订单量暴涨5倍，唯一的Worker节点CPU占用率冲到100%，Shell任务排队长达30分钟以上。数据团队眼睁睁看着监控曲线飙红却束手无策——想加一台Worker，Standalone模式根本不支持。

业务方给技术团队下了最后通牒：99.9%可用性（月故障时间不超过43分钟），支持高峰期动态扩容。解决方案只有一个：**从单机Standalone迁移到分布式集群架构**。

## 2. 项目设计——剧本式交锋对话

**场景**：周二上午，运维作战室。一块白板上画满了方框和箭头，三台服务器的配置单贴在旁边。三人开始做集群架构方案评审。

---

**小胖**（自信满满地在白板上画了3个一模一样的方框）：

> "集群不就是多装几台机器嘛！在node01、node02、node03上各装一套Standalone，然后在前面架一个Nginx做负载均衡——搞定！简单粗暴，一小时完事！"

---

**小白**（眉头紧锁，拿起红笔在小胖的图上画了三个叉）：

> "胖哥你这个方案有三个致命问题。第一，**Master主备切换**——你说在每个节点上都装Master，那两台Master同时看到Command表里有一条待派发的指令，是不是两个Master都去派？同一个工作流被两个Master各跑一遍，数据不重复写入才怪！第二，**Worker故障转移**——假设一个Worker正在跑一个耗时2小时的Spark任务，跑到第50分钟的时候Worker所在的物理机宕机了。这个跑到一半的任务谁来接管？Master怎么知道它挂了？挂掉的任务是重跑还是标记失败？第三，**API无状态但Session有状态**——你确实可以在API前面架Nginx做负载均衡，但用户登录后的Session信息存哪？轮询到另一个API节点，Session丢了用户就得重新登录。"

---

**大师**（放下茶杯，拿起白板笔，逐一圈出小白的三个问题）：

> "小白好问题！这就是为什么DolphinScheduler的集群不是单纯的'复制粘贴'。我们一个一个拆。"
>
> "**第一，Master Leader选举**。所有Master都连同一个Zookeeper集群——Zookeeper的`/dolphinscheduler/master`路径下，每个Master注册一个临时顺序节点。序号最小的那个当选Leader，Leader负责扫描Command表、拆解DAG、派发任务。其他Master作为Follower处于待命状态，不干活。Leader每隔几秒向ZK发心跳，如果Leader挂了，它的临时节点自动消失，ZK立刻通知序号第二的Master——'轮到你了，你来当Leader'。整个过程叫**ZK选主**，切换时间通常在30秒以内。你根本不用担心'两个Master同时调度'的问题——只有Leader有资格干活，这是分布式系统的经典模式。"
>
> "**第二，Worker故障转移**。每个Worker启动时在ZK的`/dolphinscheduler/worker`路径下注册一个临时节点，并定期发送心跳。Master Leader维护了所有Worker的心跳状态。如果某个Worker超过配置的超时时间（默认60秒）没有心跳——ZK临时节点消失触发watch回调——Master立即可感知到'这个Worker失联了'。然后Master检查这个Worker上正在运行的任务实例，根据工作流的失败策略处理：默认情况下，任务会被标记为失败，Master将失败任务重新派发给其他可用的Worker进行重试。当然，对于已经跑了50分钟的Spark任务，支持**容错检查点**机制——如果你的Spark任务本身配置了checkpoint，Worker重跑时会从检查点恢复，而不是从零开始。"
>
> "**第三，API无状态**。DolphinScheduler的API Server确实是无状态的——它不存储任何会话数据，所有状态都在元数据库（MySQL）中。用户的登录认证通过JWT Token实现：用户登录成功后，API Server签发一个Token返给前端，前端每次请求都带上这个Token。API Server验证Token的签名和有效期，不依赖服务端Session。所以你随便在API前面挂Nginx或HAProxy做负载均衡，用`ip_hash`或`least_conn`策略都可以——API节点想加几个加几个。"
>
> "总结一下集群拓扑——**4类服务分布式部署**。API Server × 2（无状态，随意扩缩），Master Server × 2（ZK选主，一主一备），Worker Server × 3（动态注册，任务自动分发），Alert Server × 1（单点没问题，挂了不影响主链路）。外部依赖方面，Zookeeper 3节点集群（过半存活即可用），MySQL主从复制（Master负责读写，Slave纯备份）。"

**小胖**（恍然大悟，开始在白板上重新画）：

> "我明白了！集群的核心不是'多装几台'，而是**各司其职**——API负责接收请求、Master负责DAG调度和任务派发、Worker负责干活、ZK负责协调。如果我想加容量，就加Worker；想提高请求吞吐，就加API；想保证调度不中断，就加Master。每个组件独立扩缩！"

**大师**（赞许地点头）：

> "胖胖总结得很好。最后加一个细节——Worker是**自动注册**的。你不需要改任何Master配置去'添加'一台新Worker。新Worker启动后，自动向ZK注册临时节点，Master扫描ZK发现新节点，自动纳入可用Worker池。反之，Worker下线——无论是正常关机还是物理机宕机——临时节点自动消失，Master自动将其移出可用池。这就是**动态服务发现**，也是集群运维最方便的地方。"

## 3. 项目实战

### 环境规划

**节点角色分配：**

| 节点 | IP | API Server | Master Server | Worker Server | Alert Server |
|------|-----|:---:|:---:|:---:|:---:|
| node01 | 192.168.1.101 | ✓ | ✓ | ✓ | ✓ |
| node02 | 192.168.1.102 | ✓ | ✓ | ✓ | — |
| node03 | 192.168.1.103 | — | — | ✓ | — |

**外部依赖：**

| 组件 | 部署节点 | 端口 | 说明 |
|------|----------|------|------|
| Zookeeper | node01/02/03 | 2181 | 3节点集群，过半存活机制 |
| MySQL（主） | node01 | 3306 | 读写，元数据存储 |
| MySQL（从） | node02 | 3306 | 只读备份，主从复制 |

**前置条件：** 三台节点均已安装JDK 1.8+、配置SSH免密互通、创建`dolphinscheduler`用户、建立`/opt/dolphinscheduler`目录。

---

### Step 1：配置SSH免密登录（10分钟）

```bash
# 在 node01 上，以 dolphinscheduler 用户执行
ssh-keygen -t rsa -P "" -f ~/.ssh/id_rsa

# 将公钥分发到三台节点
for host in 192.168.1.101 192.168.1.102 192.168.1.103; do
    ssh-copy-id dolphinscheduler@$host
done

# 验证免密登录
ssh node02 "hostname"   # 应输出 node02
ssh node03 "hostname"   # 应输出 node03
```

---

### Step 2：部署Zookeeper集群（15分钟）

在三台节点上分别安装Zookeeper（以3.8.0为例）：

```bash
# 三台节点均执行
cd /opt
wget https://archive.apache.org/dist/zookeeper/zookeeper-3.8.0/apache-zookeeper-3.8.0-bin.tar.gz
tar -xzf apache-zookeeper-3.8.0-bin.tar.gz
mv apache-zookeeper-3.8.0-bin zookeeper

# 创建数据目录
mkdir -p /opt/zookeeper/data

# 每台节点创建唯一 myid 文件
# node01:
echo "1" > /opt/zookeeper/data/myid
# node02:
echo "2" > /opt/zookeeper/data/myid
# node03:
echo "3" > /opt/zookeeper/data/myid

# 编辑 conf/zoo.cfg（三台相同）
cat > /opt/zookeeper/conf/zoo.cfg << 'EOF'
tickTime=2000
initLimit=10
syncLimit=5
dataDir=/opt/zookeeper/data
clientPort=2181
server.1=192.168.1.101:2888:3888
server.2=192.168.1.102:2888:3888
server.3=192.168.1.103:2888:3888
EOF

# 启动ZK集群（三台均执行）
/opt/zookeeper/bin/zkServer.sh start

# 验证集群状态
/opt/zookeeper/bin/zkServer.sh status
# 应输出: Mode: leader（其中一个节点）/ Mode: follower（其余两个节点）
```

### Step 4：配置 install_config.conf（5分钟）

在node01上编辑配置文件`/opt/dolphinscheduler/bin/env/install_config.conf`：

```properties
# 部署节点IP列表
ips="192.168.1.101,192.168.1.102,192.168.1.103"

# SSH端口
sshPort="22"

# Master服务部署在哪些节点
masters="192.168.1.101,192.168.1.102"

# Worker服务部署在哪些节点
workers="192.168.1.101,192.168.1.102,192.168.1.103"

# Alert服务部署在哪个节点
alertServer="192.168.1.101"

# API服务部署在哪些节点
apiServers="192.168.1.101,192.168.1.102"

# 安装路径
installPath="/opt/dolphinscheduler"

# 部署用户
deployUser="dolphinscheduler"

# Zookeeper集群地址
zkRoot="/dolphinscheduler"
zkQuorum="192.168.1.101:2181,192.168.1.102:2181,192.168.1.103:2181"

# 元数据库配置
dbType="mysql"
dbHost="192.168.1.101"
dbPort="3306"
dbUsername="dolphinscheduler"
dbPassword="ds_pass_2024"
dbName="dolphinscheduler"

# 资源中心存储类型（可选择 HDFS/S3/NONE）
resourceStorageType="NONE"

# YARN配置（如使用Hadoop相关任务）
yarnHaIps=
singleYarnIp=
hdfsRootUser="hdfs"
```

核心要点回顾：**`masters`** 决定哪些节点运行Master服务（Leader/Follower自动选举），**`workers`** 决定哪些节点可以执行任务，**`apiServers`** 决定前端请求的入口。每个服务独立配置，互不干扰。

### Step 5：配置 dolphinscheduler_env.sh（5分钟）

在每个节点的 `/opt/dolphinscheduler/bin/env/dolphinscheduler_env.sh` 中配置环境变量：

```bash
export JAVA_HOME=/usr/lib/jvm/java-1.8.0-openjdk
export PYTHON_HOME=/usr/bin/python3
export HADOOP_HOME=/opt/hadoop
export HADOOP_CONF_DIR=/opt/hadoop/etc/hadoop
export SPARK_HOME=/opt/spark
export FLINK_HOME=/opt/flink
export DATAX_HOME=/opt/datax

# 以上路径根据各节点实际环境调整
# 不使用大数据组件的节点，对应变量可不配置
```

**常见坑：** 三台节点的JDK安装路径可能不同（例如一台是`/usr/lib/jvm/java-1.8.0`，另一台是`/usr/local/jdk1.8.0_202`），部署前务必在每台节点上执行`echo $JAVA_HOME`确认路径一致，否则Worker启动时会报"找不到JAVA_HOME"。

### Step 6：执行分布式安装（5分钟）

```bash
# 在 node01 上，以 dolphinscheduler 用户执行
cd /opt/dolphinscheduler

# 一键安装——脚本会：
# 1. 读取 install_config.conf
# 2. 通过SSH将二进制包分发到 node02、node03
# 3. 在每台节点上按角色配置对应的服务
# 4. 创建各服务的启动脚本和配置目录
sh bin/dolphinscheduler-daemon.sh install
```

安装过程的底层逻辑：`dolphinscheduler-daemon.sh install` 依次读取 `masters`、`workers`、`apiServers`、`alertServer` 四个列表，对每个列表中的IP地址执行 `scp` 将对应服务的JAR包和配置分发过去，然后在目标节点上解压并生成启动脚本。如果某台节点SSH不通，该步骤会失败并中断安装——这就是为什么Step 1的SSH免密配置如此关键。

### Step 7：启动集群（3分钟）

```bash
# 在 node01 上执行一键启动
sh bin/dolphinscheduler-daemon.sh start-all

# 等价于分别在各自节点启动：
#   node01: start-master.sh, start-worker.sh, start-api.sh, start-alert-server.sh
#   node02: start-master.sh, start-worker.sh, start-api.sh
#   node03: start-worker.sh
```

启动顺序遵循以下约定：先启动ZK确认集群可用 → 再启动各DS服务 → DS服务启动后各自向ZK注册临时节点。

### Step 9：配置Nginx负载均衡（5分钟）

在任意一台可访问的节点上安装Nginx，配置API双活：

```nginx
upstream ds_api_cluster {
    # 负载均衡策略：ip_hash 保证同一用户会话路由到同一后端
    # 也可使用 least_conn 将请求发给连接数最少的节点
    ip_hash;
    server 192.168.1.101:12345 weight=1 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:12345 weight=1 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;
    server_name ds.damai.com;

    # 请求体大小限制（大文件上传场景）
    client_max_body_size 100m;

    location /dolphinscheduler {
        proxy_pass http://ds_api_cluster;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_connect_timeout 60s;
        proxy_read_timeout 300s;

        # WebSocket支持（DS日志推流依赖）
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}
```

验证：分别关闭node01的API Server，访问`http://ds.damai.com/dolphinscheduler`，请求应自动路由到node02，用户体验无中断。

### Step 10：Master HA故障演练（10分钟）

这是验证集群高可用的关键步骤：

```bash
# 1. 确定当前Master Leader
# 方法一：查看ZK中的Leader信息
/opt/zookeeper/bin/zkCli.sh -server 192.168.1.101:2181 get /dolphinscheduler/master/leader
# 方法二：查看Master日志
tail -f /opt/dolphinscheduler/logs/dolphinscheduler-master.log | grep "leader"

# 2. 假设 node01 是当前Leader，模拟故障——杀掉Master进程
ssh node01 "kill -9 \$(jps | grep MasterServer | awk '{print \$1}')"

# 3. 观察故障转移过程
# 在 node02 上实时查看日志：
ssh node02 "tail -f /opt/dolphinscheduler/logs/dolphinscheduler-master.log"

# 关键日志输出示例：
# [INFO] 注册中心连接断开，当前Master失去Leader角色
# [INFO] 检测到Leader节点丢失，启动新一轮选举
# [INFO] 当前节点当选为新的Leader，开始接管调度任务
# [INFO] 扫描Command表，恢复未完成的工作流实例

# 4. 验证
# - ZK中 /dolphinscheduler/master/leader 已指向 node02
# - 之前卡住的Pending工作流正在继续执行
# - 整个切换时间控制在 30 秒以内
```

**故障转移时序解读：** Master Leader每5秒向ZK发送心跳。如果心跳连续3次丢失（约15秒），ZK将临时节点标记为删除。Follower节点上的watch回调被触发，进入选举流程。选举本身耗时约5-10秒。新Leader当选后，执行一次"故障恢复扫描"——遍历所有处于`RUNNING`状态的工作流实例，检查其任务实例是否已完成、是否需重新派发。整个从故障到完全恢复的耗时（RTO）通常在25-35秒之间。

### Step 11：Worker故障转移与任务容错验证（10分钟）

```bash
# 1. 创建一个长时间运行的Shell测试任务
#    脚本内容: sleep 120; echo "done"
#    在DS UI中新建工作流，配置失败重试次数=2，上线并运行

# 2. 在任务执行期间，找到运行该任务的Worker节点
#    在"工作流实例 → 任务实例"页面查看"执行主机"列

# 3. 模拟Worker节点宕机
ssh node03 "kill -9 \$(jps | grep WorkerServer | awk '{print \$1}')"

# 4. 观察故障转移过程
# - Master Leader检测到Worker心跳超时（约60秒）
# - 相关任务被标记为失败
# - Master自动触发重试，将任务重新派发给 node01 或 node02 上的Worker
# - 工作流实例状态从"失败"变为"正在运行"
# - 重试次数减1（配置了2次，还剩1次）

# 5. 重新启动被杀掉的Worker
ssh node03 "cd /opt/dolphinscheduler && sh bin/dolphinscheduler-daemon.sh start-worker.sh"
# 验证：ZK中 /dolphinscheduler/worker 路径重新出现 node03 的临时节点
# 验证：DS UI监控中心Worker列表恢复为3个在线
```

**关键理解：** Worker宕机后Master接管是**异步**的。从Worker心跳丢失到Master检测到（心跳超时窗口 + ZK watch回调延迟），再到任务重派发，整个过程有几十秒的延迟窗口。对于分钟级的ETL任务这个延迟可接受；对于秒级敏感的实时任务，需要另行考虑。

### Step 12：横向扩容——动态添加Worker节点（10分钟）


## 4. 项目总结

### 集群架构全景回顾

| 组件 | 部署数量 | 高可用机制 | 故障影响 | 恢复时间 |
|------|:---:|------|------|:---:|
| API Server | 2 | Nginx负载均衡 + JWT Token无状态 | 单节点故障：Nginx自动剔除 | <5s |
| Master Server | 2 | ZK临时节点选主，一主一备 | Leader故障：Follower 30s内接管 | <30s |
| Worker Server | 3 | ZK临时节点注册，任务自动重派发 | 单节点故障：任务转移到其他Worker | <60s |
| Alert Server | 1 | 单点部署（轻量级） | 故障期间告警暂停，不影响主调度链路 | 重启即恢复 |
| Zookeeper | 3 | 过半存活即可用（2/3） | 1节点故障：集群正常运行 | 无感知 |
| MySQL | 主+从 | 主从复制，从库纯备份 | 主库故障：需手动切从库提升为主库 | 5-10分钟 |

### HA方案对比

| 调度系统 | 高可用方案 | 优势 | 劣势 |
|------|------|------|------|
| **DolphinScheduler** | ZK选主 + 多Master热备 | 架构原生支持，切换<30s | 依赖ZK集群稳定性 |
| Apache Airflow | Celery + Redis/RabbitMQ | 灵活的Executor扩展 | Scheduler单点需额外HA配置，组件多运维重 |
| Azkaban | Solo模式单Executor | 部署简单、学习成本低 | 单点故障全停，无HA能力 |

### 生产踩坑三则

**坑1：安装脚本卡在SSH步骤。** 表面现象是`dolphinscheduler-daemon.sh install`执行到一半报错退出，日志里只显示"连接失败"。根因是目标节点的`~/.ssh/authorized_keys`权限不是600，SSH服务拒绝公钥认证。**解法：** 在三台节点的目标用户目录下执行`chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys`。

**坑2：Worker注册到ZK但Master不派发任务。** 检查Master日志发现重复的"Worker不在可用列表"警告。根因是`install_config.conf`中Master和Worker列表不一致——Master列表中包含了Worker未注册的节点，导致Master的分发策略认为匹配不到Worker。**解法：** 确保`masters`和`workers`两个配置项的IP集合没有语义冲突，所有Master节点要么不运行Worker，要么必须也在`workers`列表中明确声明。

**坑3：MySQL主从延时导致数据不一致。** 某次主库负载突增，从库复制延迟达到5分钟。此时如果DS的读操作打到从库，可能会读到过期数据。**解法：** 在`datasource.properties`中将读操作强制指向主库（牺牲一部分读写分离优势换取一致性），或配置MySQL半同步复制（`rpl_semi_sync_master_enabled=1`），确保至少一个从库确认后才提交事务。
