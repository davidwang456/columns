# 第8章：依赖（Dependent）节点与跨流程编排

> **定位**：打破单工作流边界，实现跨流程的条件依赖编排。
> **核心内容**：Dependent节点原理、依赖周期选择逻辑、AND/OR组合、轮询机制、补数交互、超时与失败处理。
> **实战目标**：搭建"日报→周报→月报"三级流水线，验证跨工作流依赖的完整行为。

---

## 1. 项目背景

某传媒公司的数据平台维护着一条五级数据流水线：(1) 每日内容采集——爬取全网的新闻、短视频元数据，入库约200万条记录；(2) 每日内容打标——NLP模型对采集内容进行分类、情感分析和实体识别；(3) 每日推荐模型训练——基于当日新内容增量更新推荐算法；(4) 每周内容质量报告——汇总七天采集量、打标覆盖率、异常内容占比，生成周报；(5) 月度高管驾驶舱——汇总四周质量报告，输出内容团队KPI看板。

这条链路的依赖关系一目了然：周报必须等七天采集全部完成，月报必须等四周周报全部完成。然而当前调度方案用的是"硬编码等待"——运维在(2)和(3)之间塞了一个`sleep 7200`（等2小时），在(4)前面塞了`sleep 14400`（等4小时）。这套方案在正常时段勉强可用，但一到业务高峰期就原形毕露：上游数据源偶尔延迟到凌晨5点才推送，采集脚本跑到了早上7点，而此时周报的固定等待早已到期，拿到的是半成品数据——星期二到星期六的采集记录全在，唯独星期一和星期天的数据缺席。高管看报告时质问："上周到底发了多少篇文章？"运营一脸懵："报告说0篇，但后台明明有数据。"

本质问题在于：**固定时间等待 ≠ 依赖关系完成**。你需要的是"检查另一个工作流是否真的跑完了"，而不是"猜它大概多久能跑完"。DolphinScheduler的Dependent（依赖）节点正是为此而生——它通过轮询元数据库，验证目标工作流在指定日期内的实例状态，只有状态满足条件，当前工作流才继续执行。

## 2. 项目设计——剧本式交锋对话

会议室里，小胖、小白和大师正在讨论如何把"sleep大法"替换成真正的依赖机制。

**小胖**（指着白板上的流程图）：
"这不就是等人嘛！A流程跑完了B才能跑，跟去食堂吃饭一样——大师傅菜炒好了，我们才能打饭。但是有个问题：如果A流程跑了3小时，B流程就这么干等着？万一A卡住了跑到第二天，B岂不是饿死了？"

**大师**笑了笑："你的'等人'比喻说对了一半，但Dependent节点的核心不是'等'，是'查'。它不是坐在那里傻等A跑完，而是每隔一段时间去查一下数据库，看A流程在某个特定日期的实例是不是跑成功了。如果成功，立刻放行；如果还没成功，继续轮询；如果一直不成功，就超时失败。这就像用手机刷新外卖订单状态——你不会站在门口干等，而是隔几分钟刷一下App，看到'已送达'再去开门。"

**技术映射**：Dependent节点通过定时轮询`t_ds_process_instance`表检查目标工作流实例的`state`字段，而非阻塞等待。轮询间隔和超时时间可在节点配置中设定。

**小白**（停下记笔记的笔，皱眉追问）：
"等一下，这个'特定日期'怎么定？今天的工作流C依赖昨天的A，但A可能在凌晨12点之前就跑了，也可能在凌晨2点才跑——依赖节点怎么知道查的是'昨天'还是'前天'？更麻烦的是，如果A因为数据延迟补跑了一次——比如数据半夜才到，运维手动触发了一次补数，那依赖节点到底依赖A的第一次还是第二次？这不会搞混吗？"

**小胖**（挠头）：
"对对对，还有——如果A直接挂了呢？依赖节点是无限等下去，还是跟着一起挂？总得有个说法吧？"

**大师**走到白板前，画了一张时间线图：

```
调度日期: 8月25日
┌─────────────────────────────────────────────────┐
│  昨天(8/24)    今天(8/25)    明天(8/26)         │
│  ← A实例已跑    ← B依赖A昨天   ← A实例未生成     │
│                  的状态                        │
└─────────────────────────────────────────────────┘
```

"Dependent节点的核心配置有三个维度：**目标项目**、**目标工作流**、**依赖周期**。依赖周期决定了去查哪个日期的实例——'昨天'对应`scheduler_time - 1 day`，'上周一'对应上周一那天的调度日期。这不是根据实例的实际运行时间匹配的，而是根据**调度日期**匹配。你Trigger工作流时指定的那个日期，就是我查数据库时用的匹配Key。就算A在凌晨2点才跑完，只要是'8月24日的调度实例'，状态对了就行。"

"补数的场景更特殊。假设8月24日的A正常跑了一次，后来运维又补了一次，那么数据库里有两条8月24日的实例。Dependent节点默认取的是**最新创建的那一条实例**的状态。所以如果你的补数实例失败了，依赖节点会认为8月24日的A是失败状态。"

**技术映射**：依赖周期枚举值包括"今天/昨天/本周/上周/本月/上月/自定义"，运行时由Master根据调度日期计算目标日期，再通过`project_name + process_definition_name + schedule_time`三元组精确匹配实例。

**小白**追问："那AND和OR逻辑呢？周报依赖七天的采集全部成功——这是AND。但如果我有个降级策略——A成功或者B成功任一满足就行——能用OR吗？"

**大师**点头："这正是Dependent节点最强大的地方。一个Dependent节点可以配置多个**依赖项**——每个依赖项就是一个'目标工作流+周期+期望状态'的组合。依赖项之间可以用AND连接（全部满足才放行），也可以用OR连接（任一满足即放行）。周报场景就是典型的AND：七个依赖项，分别是周一到周日的采集工作流，全部SUCCESS才算满足。OR逻辑则适用于'主备链路'——比如主ETL链路和备用ETL链路，任一跑通就算数据就绪。"

"And当然也会带来新挑战：七个依赖项意味着节点要轮询七条实例的状态，如果其中某一天采集延迟了，整个周报就要等。总轮询时间 = 依赖项数量 × 每条实例的轮询次数 × 轮询间隔，超过超时上限就失败。所以生产环境需要合理设置超时时间。"

**小胖**恍然大悟："我明白了！Dependent节点就是一个检查岗，拿着一张'依赖清单'，挨个检查每个工作流有没有完成，都完成了才放行。没完成就等，等太久就报错。"

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x 集群已部署（参见第2章）
- 确保浏览器能打开控制台（`http://<master>:12345/dolphinscheduler`）
- 管理员账号 `admin / dolphinscheduler123`
- 至少有一个Worker节点正常运行
- 准备好一个项目，如"依赖节点实战"

### 步骤1：创建工作流A ——"每日内容采集"

进入项目管理页面，点击【工作流定义】→【创建工作流】，拖入一个Shell任务节点。

**任务名**：`content_ingestion`
**脚本内容**：

```bash
#!/bin/bash
echo "===== 开始每日内容采集 ====="
SCHEDULE_DATE="${system.biz.date}"
echo "调度日期: ${SCHEDULE_DATE}"

# 模拟采集耗时（实际场景可能是调用爬虫平台API）
sleep $(( RANDOM % 30 + 10 ))
echo "采集完成，收录文章数: $(( RANDOM % 50000 + 150000 ))"

# 输出关键指标供下游引用
echo "INGESTION_STATUS=SUCCESS"
echo "INGESTION_DATE=${SCHEDULE_DATE}"
echo "===== 每日内容采集结束 ====="
exit 0
```

保存工作流，命名为`daily_content_ingestion`，点击【上线】。然后在"定时管理"中添加一个定时调度：cron表达式 `0 0 2 * * ?`（每天凌晨2点执行）。

### 步骤2：创建工作流B ——"每日内容打标"（带Dependent节点）

新建工作流，先拖入一个Dependent节点。

**节点配置**：
- 节点名称：`wait_for_ingestion`
- 项目：选择当前项目
- 工作流：`daily_content_ingestion`
- 依赖周期：`昨天`（依赖昨天的采集是否完成）
- 期望状态：`SUCCESS`

> **配置说明**：左侧"依赖项"列表中点击"添加"按钮，弹出依赖配置弹窗。先选择目标项目，再选择目标工作流，下拉选择依赖周期和期望状态。依赖周期的含义——"昨天"：检查昨日的调度实例是否达到期望状态；"今天"：检查今日的调度实例；"本周/上周"：检查本周/上周范围内是否存在符合条件的实例；"本月/上月"：同理。此处的"昨天"是相对于当前工作流的调度日期来计算的。

再拖入一个Shell任务节点，命名为`content_tagging`，将其连线在Dependent节点之后：

```bash
#!/bin/bash
echo "===== 开始每日内容打标 ====="
echo "上游采集已确认完成，开始打标处理..."

# 模拟NLP打标耗时
sleep $(( RANDOM % 20 + 5 ))
TAGGED_COUNT=$(( RANDOM % 100000 + 50000 ))
echo "打标完成，处理文章数: ${TAGGED_COUNT}"
echo "TAGGING_STATUS=SUCCESS"
echo "===== 每日内容打标结束 ====="
exit 0
```

保存工作流，命名为`daily_content_tagging`，上线并添加定时：cron `0 0 4 * * ?`（每天凌晨4点执行——采集一般在2点启动，留2小时缓冲）。

完整DAG：
```
[wait_for_ingestion (Dependent)] → [content_tagging (Shell)]
```

### 步骤3：创建工作流C ——"每周内容质量报告"（多依赖AND逻辑）

这需要Dependent节点同时检查七天的采集实例。进入新建工作流，拖入一个Dependent节点。

**节点配置**——在"依赖项"列表中依次添加七个条目：

| 序号 | 目标工作流               | 依赖周期 | 期望状态 |
|------|--------------------------|----------|----------|
| 1    | daily_content_ingestion  | 上周一   | SUCCESS  |
| 2    | daily_content_ingestion  | 上周二   | SUCCESS  |
| 3    | daily_content_ingestion  | 上周三   | SUCCESS  |
| 4    | daily_content_ingestion  | 上周四   | SUCCESS  |
| 5    | daily_content_ingestion  | 上周五   | SUCCESS  |
| 6    | daily_content_ingestion  | 上周六   | SUCCESS  |
| 7    | daily_content_ingestion  | 上周日   | SUCCESS  |

**逻辑运算符**：选择 `AND`。

> **UI操作提示**：在依赖节点配置面板中，"依赖项"是一个可动态增删的表格。每点击一次"+"，新增一个依赖行。每一行包含：目标项目（下拉选择）、目标工作流（搜索选择）、依赖周期（枚举下拉）、状态（多选：SUCCESS/FAILURE等）。最上方有一个"关系"单选框，值为`AND`或`OR`，决定所有依赖项之间的逻辑关系。

再拖入一个Shell任务节点，命名为`weekly_report`，连线在Dependent节点之后：

```bash
#!/bin/bash
echo "===== 开始生成周报 ====="
echo "确认上周七天采集全部完成，开始汇总..."

# 模拟周报计算
sleep 15
echo "=============================================="
echo "           每周内容质量报告"
echo "=============================================="
echo " 统计周期: 上周一 ～ 上周日"
echo " 采集文章总数: 1,250,000+"
echo " 打标覆盖率: 98.7%"
echo " 异常内容占比: 1.2%"
echo "----------------------------------------------"
echo " 本周Top3话题:"
echo "   1. 新品发布会报道 (32,000篇)"
echo "   2. 行业政策解读   (28,500篇)"
echo "   3. 用户评测分析   (21,000篇)"
echo "=============================================="
echo "WEEKLY_REPORT_STATUS=SUCCESS"
echo "===== 周报生成结束 ====="
exit 0
```

保存工作流，命名为`weekly_quality_report`，上线并添加定时：cron `0 0 9 * * MON`（每周一早9点执行）。

### 步骤4：模拟七天的每日采集

为验证跨工作流依赖，我们需要让`daily_content_ingestion`在连续的七天中都有成功的实例。实际生产环境这些实例会自动由定时调度产生，在实验环境中我们通过手动"补数"来模拟。

在`daily_content_ingestion`工作流的"工作流定义"页，点击【运行】按钮右侧的下拉箭头，选择【补数】。在弹出的补数对话框中：
- 起始日期：`2025-04-07`（周一）
- 结束日期：`2025-04-13`（周日）
- 串行/并行：选择"串行"

点击确认后，系统会为这七天各创建一个工作流实例并按序执行。等待所有七天全部变为绿色（成功）。

> **补数说明**：补数功能会为指定日期范围内的每一天都生成一个独立的工作流实例。这些实例的`schedule_time`字段各自不同，Dependent节点正是根据这个字段进行匹配。

### 步骤5：执行周报工作流并验证

回到`weekly_quality_report`工作流，手动点击【运行】，调度日期可以选择`2025-04-14`（下周一）。

观察执行过程：Dependent节点开始执行后，你可以在"工作流实例"页点击该节点，切换到日志标签页，会看到类似以下日志：

```
[INFO] Dependent task start
[INFO] Checking dependency item: project=依赖节点实战, workflow=daily_content_ingestion, cycle=上周一, status=SUCCESS
[INFO] Matched instance: schedule_time=2025-04-07, state=SUCCESS → PASS
[INFO] Checking dependency item: project=依赖节点实战, workflow=daily_content_ingestion, cycle=上周二, status=SUCCESS
[INFO] Matched instance: schedule_time=2025-04-08, state=SUCCESS → PASS
...
[INFO] Checking dependency item: project=依赖节点实战, workflow=daily_content_ingestion, cycle=上周日, status=SUCCESS
[INFO] Matched instance: schedule_time=2025-04-13, state=SUCCESS → PASS
[INFO] All 7 dependency items passed (AND logic), dependent task succeeded
```

七个依赖项全部检查通过后，Dependent节点变为绿色，`weekly_report`任务开始执行。

### 步骤6：测试失败场景

现在模拟一个生产事故：假设上周三（4月9日）的采集工作流失败了。

在`daily_content_ingestion`的"工作流实例"页，找到`schedule_time`为`2025-04-09`的实例，这通常已经是成功状态（因为步骤4中补数全部成功了）。我们可以模拟失败：将该实例的Shell脚本临时改为`exit 1`，然后单独重跑这一天。

或者更直接的方式：在`weekly_quality_report`的Dependent节点配置中，将"上周三"的期望状态从`SUCCESS`改为`FAILURE`，这样依赖逻辑就变成了"上周三的采集必须是失败状态才算满足"——这显然不会通过，因为实际实例是成功的。

保持正常配置，将4月9日的实例改为失败后，重新运行`weekly_quality_report`。观察结果：

Dependent节点的日志会显示：
```
[INFO] Checking dependency item: workflow=daily_content_ingestion, cycle=上周三, status=SUCCESS
[WARN] Matched instance: schedule_time=2025-04-09, state=FAILURE → NOT SATISFIED
[INFO] Dependency condition not met, will retry...
```

Dependent节点会持续轮询，直到超时。超时后节点标记为失败，下游`weekly_report`不会执行。

### 步骤7：测试OR逻辑

新建一个工作流`daily_redundant_etl`，里面放一个简单的Shell任务（sleep 10秒即可），模拟备用ETL链路。

再新建工作流，命名为`or_dependency_demo`，拖入一个Dependent节点，配置两个依赖项，选择OR逻辑：

| 依赖项 | 目标工作流             | 依赖周期 | 期望状态 |
|--------|------------------------|----------|----------|
| 1      | daily_content_ingestion | 昨天    | SUCCESS  |
| 2      | daily_redundant_etl     | 昨天    | SUCCESS  |

策略：任一成功即可。你可以手动让其中一个失败、另一个成功，观察Dependent节点依然判定依赖满足。OR逻辑的本质是"降级容错"：主链路挂了，备用链路顶上，数据就绪即可，不纠结是哪条链跑出来的。

### 步骤8：常见陷阱与规避

| 陷阱 | 现象 | 根因与解决 |
|------|------|------------|
| 依赖周期选错 | 依赖节点一直找不到目标实例 | "昨天"和"今天"容易混淆——如果你在今天凌晨1点跑B依赖A，A的定时是在今天凌晨2点，那么依赖周期应该选"昨天"而非"今天" |
| 目标工作流从未运行 | 依赖节点轮询超时失败 | 依赖的目标日期内，该工作流根本没有创建过实例。Dependent不会凭空"创造一个实例来检查"，找不到实例=状态不满足。确保目标工作流的定时调度在依赖周期之前触发 |
| AND依赖项过多导致超时 | 七个依赖项逐个轮询，总耗时超过超时上限 | 适当增加轮询间隔（每次轮询开销降低），或调大超时时间。极端场景考虑拆分为多个Dependent节点串行 |
| 补数与正常调度冲突 | 一个日期出现多个实例，依赖节点检查结果不符合预期 | 补数会产生第二个同日期实例。依赖节点取"最新创建的实例"状态。流程设计时需约定：补数操作在所有依赖节点执行完毕后进行，避免竞争窗口 |
| 跨项目依赖找不到工作流 | 保存时提示"目标工作流不存在" | 检查目标工作流是否已上线。仅"已上线"的工作流才能被Dependent节点搜索到，草稿状态的工作流在依赖项目列表中不出现 |
| 依赖周期"自定义"未填日期 | 依赖检查逻辑异常 | 选择"自定义"周期后，需要在下方日期输入框中填入具体的日期值，否则依赖条件为空导致检查逻辑失效 |

## 4. 项目总结

### Dependent节点的优点与局限

| 维度 | Dependent节点（轮询式） | 事件驱动（Kafka/Webhook） | API回调 |
|------|---------------------------|----------------------------|---------|
| 实现复杂度 | 低，纯配置，零代码 | 高，需消息队列基础设施 | 中，需编写回调接收端 |
| 实时性 | 轮询级（秒～分钟延迟） | 毫秒～秒级 | 近实时 |
| 可靠性 | 高（依赖DB状态，Master内置重试） | 依赖MQ的持久化与重试 | 依赖回调接收方的高可用 |
| 跨系统依赖 | 仅限同一DS集群内的项目 | 可跨任意系统 | 可跨任意系统 |
| 运维成本 | 低，无需额外组件 | 中（Kafka集群维护） | 中（回调服务维护） |
| 历史补数交互 | 原生支持，自动匹配日期 | 需自行实现日期逻辑 | 需自行实现日期逻辑 |

### 适用场景

- **ETL→报表父子链**：采集→清洗→分析→日报→周报→月报，天然的跨工作流级联依赖
- **数据质量门禁**：在模型训练前增加Dependent节点，检查数据质量稽核工作流是否通过
- **跨团队工作流协调**：A团队负责数据入湖，B团队负责特征工程，B的调度通过Dependent依赖A的输出
- **主备链路容灾**：主ETL和备用ETL用OR依赖，任一完成即触发下游

### 不适用场景

- **毫秒级实时依赖**：Dependent的轮询粒度最低为秒级，高频交易、实时风控等场景不适用
- **依赖非DS外部系统**：如果下游任务依赖于第三方平台（如Airflow、自定义脚本）的执行结果，Dependent无法直接检测，需通过Shell节点调用API间接实现

### 核心注意事项

1. **依赖周期与调度时间的关系**：Dependent计算"昨天/今天"的基准是当前工作流实例的`schedule_time`，而非系统当前时间。补数时注意基准日期的变化。
2. **超时配置务必合理**：多依赖AND场景下，轮询总耗时 = 超时时间。建议根据上游最长执行时间 + 缓冲余量来设置，一般周报设置2-4小时超时。
3. **上线后方可依赖**：目标工作流必须先上线，否则Dependent节点无法在搜索列表中找到它。修改目标工作流后，依赖方不会自动感知，需手动重新保存依赖节点。
4. **实例清理策略**：DS会定期清理历史实例。如果目标工作流的历史实例已被清理，依赖节点将无法找到匹配实例，表现等同于"依赖不满足"。

### 思考题

1. 假设你的周报依赖七天的采集工作流，但其中有一天（例如上周三）的采集工作流因为数据源故障超时失败了。Dependent节点在检测到这个失败后应该怎样处理？如果利用OR逻辑设置一个"降级方案"——上周三的采集失败时，自动切换到"手动补录数据"的备用工作流——请设计这个依赖的配置方式。

2. 当你在对`daily_content_ingestion`进行"补数"操作时，正在运行中的`weekly_quality_report`的Dependent节点也在检查上周的依赖。补数产生的额外实例会不会干扰依赖节点的判断？请描述竞争窗口的具体发生条件和规避方案。补数完成后，如果发现依赖节点已经超时失败，又该如何补救？
