# 第26章：补数策略与大规模数据回刷

> **定位**：从"批量生成历史实例"升级到"工业级大规模补数工程方案"。
> **核心内容**：超大规模补数（180天+）的分批策略、资源容量规划、失败恢复机制、正常调度共存策略、下游系统保护、SubProcess与Conditions在补数中的行为。
> **实战目标**：为AI推荐模型ETL工作流执行730天历史补数，设计分批方案、容量估算、失败恢复SOP，最终在48小时内完成全量回刷。

## 1. 项目背景

大麦网（Damai）算法团队近期研发了一套新一代AI推荐模型，需要2年（730天）的历史训练数据。好消息是：特征ETL工作流`daily_ai_feature_etl`已经稳定运行，每天凌晨2点从业务库抽取前一天的用户行为数据，经清洗、特征工程后写入特征宽表——这正是模型所需的训练集。坏消息是：这个工作流自上线以来从未做过历史回刷——宽表中只有近3个月的数据，其余21个月完全是空白。

第一轮尝试——"全量并行"。运维直接将730天一次性提交并行补数。3分钟后，MySQL元数据库连接池耗尽，Master进程OOM，500多个工作流实例全部卡死在"正在运行"状态。僵尸实例堆积、Worker线程池饱和，连正常调度的其他工作流也全部排队饿死。整整1小时才完成紧急止损（重启所有Master/Worker、手工清理积压Command记录）。

第二轮尝试——"全量串行"。改为180天串行补数。结果：180个实例依次执行，每个实例平均耗时12分钟（ETL涉及Spark批处理），总耗时36小时。数据是跑完了，但模型团队只剩12小时的训练窗口——远远不够。

团队面临一个经典的工程难题：**730天的数据必然要回刷，但系统资源不是无限的，时间窗口也不是无限的。如何在"快"和"稳"之间找到最优解？** 这就是本章要回答的问题——工业级大规模补数策略的设计与落地。

**技术映射**：分批并行策略的核心公式——`MaxParallel = min(Master线程池 × 0.5, DB连接池 × 0.3, Worker任务线程 × 0.7)`，安全并发数取其50%，再按此设定每批大小。

**大师**走到白板前，画了一张时间线和命令队列图：

```
命令队列（t_ds_command）：
┌──────────────────────────────────────────────────┐
│  正常调度Command (优先级HIGH)  ← 凌晨2点CRON触发   │
│  补数Command_1 (优先级LOW)     ← 批量补数触发      │
│  补数Command_2 (优先级LOW)     ← 批量补数触发      │
│  ...                                              │
│  Master轮询消费 → 优先级高者先出队                   │
└──────────────────────────────────────────────────┘
```

"第一个问题——命令优先级。DolphinScheduler的Command表中`process_instance_priority`字段决定了消费顺序：HIGH > MEDIUM > LOW。大规模补数时，**必须将补数实例的优先级设为LOW**，正常CRON触发的保持HIGH。这样凌晨2点的生产调度永远优先消费，不被补数挤压。如果补数窗口极度紧张，也可以直接**暂停定时调度**——补数完成并验证后，再恢复定时。两种策略选其一，视业务容忍度而定。"

"第二个问题——依赖匹配更精妙。Dependent节点检查依赖的依据是`schedule_time`——不是`create_time`，不是`run_time`。所以你补数A的`2022-03-15`实例，它的`schedule_time`就是`2022-03-15`；B工作流在某天运行时，它的Dependent节点计算'昨天'就是`schedule_time - 1 day`，然后精确匹配到A的`schedule_time = 2022-03-15`那条记录。**补数实例和定时调度实例的区分维度从不在Dependent作用域内**——只要`schedule_time`匹配上且状态正确，依赖就满足。唯一需要防范的是：同一天有两条实例（一条定时、一条补数），此时Dependent节点默认取最新`create_time`的那条。"

**技术映射**：Dependent节点的核心匹配SQL等价于`SELECT state FROM t_ds_process_instance WHERE process_definition_code=? AND schedule_time=? ORDER BY create_time DESC LIMIT 1`。`schedule_time`由Master在生成实例时按补数日期范围精确计算，保证依赖链路可追溯。

**小胖**（总算消化完，挠挠头）：
"那SubProcess节点和Conditions节点呢？如果我补数的是一个引用子工作流的父工作流，子工作流里的条件分支还会正常评估吗？"

**大师**点头：
"SubProcess在补数时会递归生成子工作流实例——每个父实例对应一个子实例，子实例的`schedule_time`继承自父实例。Conditions节点在补数期间完全正常评估——因为条件节点的上游参数值在补数生成实例时就已经注入，跟是否是补数实例无关。但有一个隐藏坑：如果条件节点的判断依赖的是`system.biz.date`这类系统参数，补数实例中的`system.biz.date`等于你填的`补数起始日期 + 序号`，而不是当前真实日期。这意味着条件节点可能会走向你预期之外的路径——务必用固定参数值做条件判断，而非依赖系统时间。"

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x 集群：Master × 2、Worker × 4（每Worker配置20个task-executor线程）
- 元数据库：MySQL 8.0，HikariCP连接池`maximum-pool-size=50`
- 特征宽表：`dwd.ai_training_features`，分区键`dt`（格式`yyyy-MM-dd`）
- 目标：为`daily_ai_feature_etl`工作流补数730天（2022-01-01至2023-12-31），48小时内完成

### Step 1：评估系统容量上限

**目标**：在发起大规模补数前，先摸清集群的资源天花板，避免第一轮的惨案重演。

```sql
-- 1. 检查当前运行中的实例数
SELECT COUNT(*) AS running_instances 
FROM t_ds_process_instance 
WHERE state IN (4, 5, 6);  -- 4=DISPATCH, 5=READY_PAUSE, 6=READY_STOP

-- 2. 检查当前运行中的任务实例数
SELECT COUNT(*) AS running_tasks 
FROM t_ds_task_instance 
WHERE state IN (3, 4, 5);  -- 3=DISPATCH, 4=READY_PAUSE, 5=READY_STOP

-- 3. 查看各Worker分组的负载分布
SELECT worker_group, COUNT(*) AS task_cnt
FROM t_ds_task_instance 
WHERE state IN (3, 4, 5)
GROUP BY worker_group;
```

**容量公式推导**：

| 资源维度 | 总量 | 系数 | 可用 |
|---------|------|------|------|
| Master线程池（消费Command+调度DAG） | 2 Master × 20线程 = 40 | × 0.5 | **20** |
| 元数据库连接池（HikariCP maximum=50） | 50 | × 0.3 | **15** |
| Worker任务执行线程（4 Worker × 20线程） | 80 | × 0.7 | **56** |

**最大并行实例数** = min(20, 15, 56) = **15**
**安全并行度** = 最大并行 × 0.5 = **7～8个实例同时运行**

> **关键洞察**：此场景中瓶颈在数据库连接池（仅15个可用连接）。补数期间每个实例的元数据读写（状态更新、日志写入）都要消耗连接，留足余量防止连接耗尽是首要任务。

### Step 2：设计730天的分批策略

**目标**：在48小时内完成730天数据回刷，同时不压垮集群资源。

```
容量规划：
  单实例平均耗时：8分钟（含Spark任务）
  安全并行度：7个（基于Step 1的容量评估）
  每批执行时间：8分钟（7个并行，取最长单实例耗时）
  批次间缓冲时间：2分钟（状态确认 + 失败清理）

方案A：每批30天，并行7个
  总批数：ceil(730 / 30) = 25批（第25批10天）
  每批耗时：ceil(30 / 7) × 8分钟 ≈ 40分钟
  总预估时间：25 × (40 + 2) = 1050分钟 ≈ 17.5小时 ✓ （48h内可完成）

方案B（激进）：每批60天，并行7个
  每批耗时：ceil(60 / 7) × 8 ≈ 72分钟
  总批数：13批
  总预估时间：13 × (72 + 2) = 962分钟 ≈ 16小时

方案C（保守）：每批15天，并行7个
  总批数：49批
  总预估时间：49 × (24 + 2) ≈ 1274分钟 ≈ 21小时
```

选择**方案A**——30天/批是速度和稳定性的最优平衡点。每批内7个实例并行跑，批次之间串行等前一批全部完成。

### Step 3：实现分批补数控制器脚本

**目标**：用Python脚本编排25个批次，自动提交、监控、验收。

```python
#!/usr/bin/env python3
# backfill_controller.py
"""730天大规模补数分批编排控制器"""
import subprocess
import time
import json
from datetime import datetime, timedelta

API_BASE = "http://api-server:12345/dolphinscheduler"
TOKEN = "your-api-token-here"
PROJECT = "damai"
WORKFLOW = "daily_ai_feature_etl"

def backfill_batch(start_date, end_date, parallelism=7):
    """提交一批补数，返回提交结果"""
    payload = {
        "processDefinitionCode": get_workflow_code(WORKFLOW),
        "startDate": start_date,
        "endDate": end_date,
        "scheduleTime": "00:00",
        "failureStrategy": "CONTINUE",      # 单日期失败不阻塞同批次其他日期
        "warningType": "FAILURE",
        "warningGroupId": 1,
        "processInstancePriority": "LOW",   # 补数低优先级，让路给正常调度
        "workerGroup": "backfill_workers",  # 专用Worker分组，避免影响生产
        "execType": "PARALLEL",             # 批量内并行
        "complementDependentMode": "OFF_MODE"
    }
    cmd = f"""
    curl -s -X POST "{API_BASE}/projects/{PROJECT}/executors/complement-data" \\
      -H "token: {TOKEN}" \\
      -H "Content-Type: application/json" \\
      -d '{json.dumps(payload)}'
    """
    result = subprocess.run(cmd, shell=True, capture_output=True, text=True)
    return json.loads(result.stdout)

def get_workflow_code(wf_name):
    """根据工作流名称获取definition_code"""
    cmd = f"""
    curl -s "{API_BASE}/projects/{PROJECT}/process-definition?searchVal={wf_name}" \\
      -H "token: {TOKEN}"
    """
    result = subprocess.run(cmd, shell=True, capture_output=True, text=True)
    data = json.loads(result.stdout)
    return data["data"]["totalList"][0]["code"]

def monitor_batch(start_date, end_date, expected_count):
    """轮询监控一批补数的完成情况，返回失败日期列表"""
    while True:
        # 查询此批次日期范围内的实例状态
        cmd = f"""
        curl -s "{API_BASE}/projects/{PROJECT}/process-instances?processDefineCode={get_workflow_code(WORKFLOW)}&startDate={start_date}&endDate={end_date}" \\
          -H "token: {TOKEN}"
        """
        result = subprocess.run(cmd, shell=True, capture_output=True, text=True)
        instances = json.loads(result.stdout).get("data", {}).get("totalList", [])
        
        completed = [i for i in instances if i["state"] in ("SUCCESS", "FAILURE", "STOP")]
        failed = [i for i in completed if i["state"] != "SUCCESS"]
        
        print(f"  [{start_date}~{end_date}] 进度: {len(completed)}/{expected_count} "
              f"成功: {len(completed) - len(failed)} 失败: {len(failed)}")
        
        if len(completed) >= expected_count:
            return failed
        
        time.sleep(30)  # 30秒轮询一次
    
    return []

def quality_check(start_date, end_date):
    """每批完成后数据质量校验"""
    check_sql = f"""
    SELECT dt, COUNT(*) as row_cnt
    FROM dwd.ai_training_features
    WHERE dt BETWEEN '{start_date}' AND '{end_date}'
    GROUP BY dt
    HAVING COUNT(*) < 1000
    """
    # 执行SQL检查，过滤出数据量异常的日期
    print(f"  [质检] 批次 {start_date}~{end_date} 数据完整性校验中...")
    # 实际场景中通过JDBC/API执行
    return []  # 返回数据异常的日期列表

# ── 主流程 ──
TOTAL_DAYS = 730
BATCH_SIZE = 30
BASE_DATE = datetime(2022, 1, 1)
all_failed_dates = []

print("=" * 60)
print(f"  大规模补数启动: {WORKFLOW}")
print(f"  范围: {BASE_DATE.date()} ~ {BASE_DATE + timedelta(days=730-1)}")
print(f"  策略: {BATCH_SIZE}天/批, 并行7, 共{-(TOTAL_DAYS // -BATCH_SIZE)}批")
print("=" * 60)

for offset in range(0, TOTAL_DAYS, BATCH_SIZE):
    batch_start = BASE_DATE + timedelta(days=offset)
    batch_end = BASE_DATE + timedelta(days=min(offset + BATCH_SIZE, TOTAL_DAYS) - 1)
    batch_num = offset // BATCH_SIZE + 1
    expected = (batch_end - batch_start).days + 1
    
    print(f"\n▶ 批次 {batch_num}: {batch_start.date()} → {batch_end.date()} ({expected}天)")
    
    # 提交补数
    backfill_batch(batch_start.strftime("%Y-%m-%d"), 
                   batch_end.strftime("%Y-%m-%d"))
    
    # 监控完成
    failed = monitor_batch(batch_start.strftime("%Y-%m-%d"), 
                           batch_end.strftime("%Y-%m-%d"), expected)
    
    if failed:
        print(f"  ⚠ 本批失败 {len(failed)} 个日期: {[f['scheduleTime'] for f in failed]}")
        all_failed_dates.extend(failed)
    
    # 质量校验
    bad_dates = quality_check(batch_start.strftime("%Y-%m-%d"), 
                               batch_end.strftime("%Y-%m-%d"))
    
    print(f"  ✓ 批次 {batch_num} 完成")

print("\n" + "=" * 60)
print(f"  全部批次执行完毕")
if all_failed_dates:
    print(f"  失败日期汇总 ({len(all_failed_dates)}条): 需要单独重跑")
else:
    print(f"  730天数据全部回刷成功! ✓")
print("=" * 60)
```

> **关键设计**：`failureStrategy: "CONTINUE"`确保单天失败不阻塞整批。失败日期收集到`all_failed_dates`列表，全部批次跑完后统一重跑——这比中途中断修复高效得多。

### Step 4：配置补数实例的保护参数

**目标**：通过补数参数配置，确保大规模回刷不影响正常生产调度。

```json
{
  "failureStrategy": "CONTINUE",
  "warningType": "FAILURE",
  "processInstancePriority": "LOW",
  "workerGroup": "backfill_workers",
  "warningGroupId": 1,
  "complementDependentMode": "OFF_MODE"
}
```

| 参数 | 值 | 作用 |
|------|-----|------|
| `failureStrategy` | CONTINUE | 单日期失败后继续，不中断同批次其他日期 |
| `warningType` | FAILURE | 批次中任一日期失败即发送告警 |
| `processInstancePriority` | LOW | 补数命令优先级低于正常调度（HIGH） |
| `workerGroup` | backfill_workers | 隔离到专用Worker分组，避免抢占生产Worker |
| `complementDependentMode` | OFF_MODE | 补数期间不触发依赖检查，避免级联触发下游 |

### Step 5：下游系统保护措施

**目标**：730天数据写入不能压垮目标数据仓库。

**策略一——读写分离**：
```
补数ETL读取：业务库只读副本（reader-slave.bigdata.com）
补数ETL写入：特征宽表（dwd.ai_training_features），限速写入
```

**策略二——DataX通道限速**（如果ETL使用DataX节点）：
```yaml
# 正常生产配置
"speed": {
  "channel": 8,
  "byte": 10485760
}

# 补数专用配置（降速保护目标库）
"speed": {
  "channel": 2,
  "byte": 5242880
}
```

**策略三——目标库写入限流**（Shell/Spark节点场景）：
在ETL脚本中控制写入批次间隔：
```bash
# 在大数据量写入的目标表每条INSERT后加间歇
INSERT INTO dwd.ai_training_features PARTITION(dt='${biz_date}')
SELECT ... FROM source_wide_table WHERE dt='${biz_date}';
# 每写一批sleep 2秒，给目标库IO喘息时间
sleep 2
```

### Step 6：处理补数期间的正常调度冲突

**目标**：避免凌晨2点CRON触发与正在进行的补数产生竞态。

**方案一——暂停法（推荐）**：
```bash
# 补数开始前：暂停定时调度
curl -X POST "${API_BASE}/projects/${PROJECT}/schedules/${schedule_id}/offline" \
  -H "token: ${TOKEN}"

# 补数全部完成后：恢复定时调度（跳过已回刷的日期）
curl -X POST "${API_BASE}/projects/${PROJECT}/schedules/${schedule_id}/online" \
  -H "token: ${TOKEN}"
```

**方案二——优先级法**：
保持定时调度运行，但设置正常调度优先级为`HIGH`，补数优先级为`LOW`。Master消费者优先处理高优先级Command——正常调度不会排队等待。

> **场景推演**：凌晨2点，队列中积压了400条LOW优先级的补数Command。此时CRON触发一条HIGH优先级的正常调度Command。Master消费者轮询时先检查`priority`字段——HIGH优先出队，正常调度丝毫不受补数影响。

### Step 7：失败恢复与数据回滚

### Step 9：Conditions与SubProcess在补数中的行为

**目标**：理解含条件分支和子工作流的DAG在补数时的行为差异。

| 节点类型 | 补数行为 | 关键注意事项 |
|---------|---------|-------------|
| **Conditions节点** | 正常评估条件表达式，按条件走成功/失败分支 | 若条件依赖`system.biz.date`，需确认补数日期范围内该参数值与预期一致。建议改用固定IN参数传值 |
| **SubProcess节点** | 递归为每个父实例生成对应的子工作流实例，子实例继承父实例的`schedule_time` | 如果子工作流内部还有Dependent节点，补数时需要确保被依赖方先于依赖方完成补数 |
| **Dependent节点**（`complementDependentMode=ON`） | 自动检查依赖方在相同`schedule_time`下的实例状态 | 此模式下补数一个工作流会自动级联补数其上游依赖——适合"全链路回刷"场景 |

### Step 10：常见大规模补数事故清单

| 事故 | 现象 | 根因 | 修复 |
|------|------|------|------|
| 连接池耗尽 | Worker批量报`HikariPool-1 - Connection is not available` | 补数实例数远超DB连接池容量 | 按容量公式限制并行度 |
| Master OOM | Master进程内存飙升至上限后崩溃退出 | 一次性生成730个实例，元数据全部加载到JVM堆内存 | 必须拆分批次，每批不超过30天 |
| 下游库写崩 | 目标数据库CPU 100%，锁等待超时 | 730天数据同时写入，触发器/索引维护打满IO | 限速写入、降低DataX channel、专用Worker分组 |
| 补数覆盖生产数据 | 生产报表出现2022年的重复数据 | 补数写入模式为"overwrite"而非"append" | 确认ETL脚本的写入语义为INSERT INTO（追加） |
| 依赖节点级联失败 | 下游Dependent超时，虽然上游实则已成功 | 补数期间上游新增实例导致依赖匹配到"补数实例"而非"原定实例" | 先补上游 → 确认全成功 → 再补下游（严格顺序） |
| 补数日期顺序错误 | 数据按`schedule_time`排序后发现日期错乱 | 并行补数不保证执行顺序，`schedule_time`虽是顺序的但实例创建时间可能交错 | 依赖日期顺序的场景必须使用串行模式 |

## 4. 项目总结

### 补数策略决策矩阵

| 场景 | 天数范围 | 推荐模式 | 分批策略 | 风险等级 |
|------|---------|---------|---------|---------|
| 单日修复 | 1～3天 | 并行 | 无需分批 | 低 |
| 小范围回刷 | 4～30天 | 并行（限制并发度） | 1批完成 | 低 |
| 中等回刷 | 30～180天 | 分批并行 | 15～30天/批 | 中 |
| 大规模回刷 | 180～365天 | 分批并行 | 20～30天/批 | 高 |
| 超大规模回刷 | 365天以上 | 分批并行 + 串行批间 | 15～30天/批 | 极高 |

### 分批大小计算公式

```
安全并发数 S = min(Master线程池×0.5, DB连接池×0.3, Worker线程池×0.7)
每批大小 B = min(S × 2, 30)  // 30天是经验上限
批次数 N = ceil(总天数 / B)
预估总时间 T = N × (ceil(B/S) × 单实例平均耗时 + 缓冲时间)
```

### 三种补数模式总成本对比（以730天为例）

| 模式 | 耗时 | 资源峰值 | 失败恢复复杂度 | 适用场景 |
|------|------|---------|-------------|---------|
| 全量并行 | ~2h（理想）→ 系统崩溃（现实） | 极高（730并发） | 无法局部恢复，全量重跑 | 理论参考，不可用于实际生产 |
| 全量串行 | ~146h（6天） | 极低（1并发） | 简单，失败从断点续跑 | 时间充裕且数据量小的场景 |
| **分批并行** | ~17.5h | 可控（7并发） | 中等，仅重跑失败批次 | ✅ 生产环境最佳实践 |

### 生产实战三则

**案例1——730天并行血案**（已在本章背景中详述）：全量并行导致数据库连接池枯竭、Master OOM。教训：**永远不要在不评估系统容量的情况下发起大规模补数**。

**案例2——Dependent节点级联雪崩**：某团队补数上游ETL 365天，同时保留下游Dependent工作流的定时调度。补数过程中上游新增365条实例，下游Dependent节点每次轮询都匹配到"补数实例"而非"正常调度实例"，导致7个下游工作流全部因依赖检查失败而超时。教训：**补数期间必须暂停下游Dependent工作流，或按"上游→下游"严格顺序分批补数**。

**案例3——DataX通道风暴**：某团队对90天数据补数，ETL中使用DataX节点，channel保持生产环境的8通道。90 × 8 = 720个并发写入连接涌入ClickHouse目标库，导致MergeTree后台merge操作排队堆积，整个集群写入延迟从50ms飙升到30秒。教训：**大规模补数必须降速——将DataX的channel降至正常值的1/4，或在脚本中人为插入sleep间隔**。

1. 假设你的`daily_ai_feature_etl`工作流中包含一个SubProcess节点（子工作流为`feature_aggregation`），该子工作流内部又有一个Dependent节点（依赖`source_data_ready`）。当你对顶层工作流补数730天时，设置`complementDependentMode=ON`和`OFF_MODE`分别会导致什么行为差异？如果因某个上游依赖始终未满足，导致中间200天的子工作流全部失败——你应该如何设计恢复方案？提示：思考"全量重跑"与"跳过失败日期仅补缺失"两种策略的利弊。
