# 第1章：DolphinScheduler术语全景与分布式调度架构原理

---

## 1. 项目背景

某中型电商公司，数据团队每天要跑200多个数据处理任务：订单清洗、库存同步、用户行为分析、日报推送、推荐模型训练……这些任务之间存在复杂的上下游依赖——库存同步必须等订单清洗完成，日报推送又依赖所有上游产出结果。

早期他们全靠crontab脚本硬撑。运维小张在服务器上写了上百条cron表达式，用`&&`和`sleep`来勉强控制任务顺序。问题很快暴露：凌晨3点上游ETL挂了，下游几十个任务照样到点执行，产出全是空数据和脏数据；一次服务器重启导致cron丢失，小张半夜打车回公司补配置；老板想看"今天到底哪些任务失败了"，小张翻了半天`/var/log`也讲不清楚。

这就是典型的工作流调度失控。为什么？因为crontab本质上是"时间驱动"的，它只管"几点几分启动某个命令"，完全不关心依赖关系、执行状态、资源分配。当任务从几十增长到几百时，crontab就像一座纸牌屋——一个节点塌了，整条链路全崩。

调度系统的核心价值就是解决这三个问题：**依赖管理**（谁先谁后）、**状态追踪**（成功/失败/重试/超时）、**资源协调**（分布式节点负载均衡）。Apache DolphinScheduler正是为此而生——它是一款分布式、易扩展的可视化DAG工作流调度平台，支持多租户、多集群、多云环境。

## 2. 项目设计——剧本式交锋对话

**场景**：周三下午，会议室。黑板上画着一些方框和箭头。三个人围坐讨论新调度系统的技术选型。

---

**小胖**（咬了一口面包，指着屏幕上的架构图）：

> "这不就跟食堂打饭排队一样吗？到点了大家刷卡进食堂，排队打菜——大数据调度不就是'到点了启动任务，一个接一个跑'嘛！"

**大师**（微笑，推了推眼镜）：

> "小胖你说的对了一半。食堂打饭是线性的——一根队列排到底。但数据任务不是线性的，是**网状的**。一个报表任务，可能同时依赖订单任务和库存任务都跑完才能启动。这叫**DAG——有向无环图**。DAG里每个节点叫一个**Task（任务）**，整张图叫一个**Workflow（工作流定义，Process Definition）**。每次真正跑起来的那份就叫**Process Instance（工作流实例）**，里面的每个任务就是**Task Instance（任务实例）**。"

**小胖**（又咬了一口面包）：

> "那上游下游是啥？就跟食堂里'主食窗口'必须排在'打菜窗口'前面一样？"

**大师**：

> "对。如果任务A没跑完，任务B就不能开始，那A就是B的**上游（upstream）**，B是A的**下游（downstream）**。DolphinScheduler里，上游失败了还能决定下游要不要继续跑——这叫失败策略，crontab根本做不到。"

---

**小白**（一直在安静记笔记，突然抬头）：

> "我有个问题。公司200多个任务，一台机器跑得过来吗？万一凌晨机器挂了，是不是全部废掉？"

**大师**：

> "好问题。这就是为什么不能用单机crontab，必须用**分布式调度**。DolphinScheduler有四个核心服务，我把它们拆成公司里的大白话讲：

> - **API Server**：公司前台+管理层入口。你通过界面拖拽画DAG、点执行按钮，都走API Server（Spring Boot + Jetty，端口12345）。它把请求写入数据库的**Command（命令）**表，相当于"你在OA系统里下了一个工单"。
> - **Master Server**：项目经理。它不停扫描数据库里有没有新的Command，就像项目经理早上起来翻一遍**待办事项清单**。拿到命令后，Master负责把整个DAG"拆活儿"——哪个Task先跑、哪个后跑、失败了要不要重试、怎么通知下游。它自己维护一个DAG状态机，通过gRPC/Netty（端口5679）把任务分发给工人。
> - **Worker Server**：一线执行工人。工人不关心DAG长什么样，只管拿到活儿就干——Shell脚本、Spark作业、Flink任务……全由**任务插件（Task Plugin）**来执行（端口1235）。干完了通过状态回调告诉项目经理"我搞定了"。
> - **Alert Server**：保安队长。谁出错了、超时了，它通过邮件、钉钉、微信、飞书等渠道通知相应的人（端口50052/50053）——"警告！凌晨ETL任务又挂了！""

**小胖**（面包吃完了，舔舔手指）：

> "那如果项目经理Master自己挂了怎么办？比如Master当机了，谁接手？"

**大师**：

> "这就靠**注册中心（Registry）**了——Zookeeper/Etcd都能用。它就像**公司前台**，时刻知道公司里哪些员工在岗、哪些请假了。Master挂了，Registry通过临时节点的watch机制立刻感知到，然后触发**Leader选举**——另一个Master节点立刻接替当项目经理。Worker也是这样，新节点上线自动注册，挂了自动下线。分布式锁也靠它，保证同一个任务不会被两个Worker重复执行。"

**小白**：

> "那任务分给哪个Worker执行由谁决定？"

**大师**：

> "这里有三个关键概念。**Worker Group（工作组）**——比如'ETL组'只处理Spark任务，'算法组'专门跑Python。Master派活时会根据Worker Group匹配。**Tenant（租户）**——不同部门有不同Linux用户，比如数据组用`datauser`，算法组用`algouser`，Worker执行任务时会`sudo`到对应租户。**Queue（队列）**——YARN队列，决定了任务提交到哪个资源池。另外资源文件（脚本、JAR包）统一存放在**Resource Center（资源中心）**，支持S3/HDFS/本地存储，免得每个Worker节点上都得复制一份。"

## 3. 项目实战

### 动手第1步：画出架构全景图

找一张白纸（或用draw.io），画出以下组件并标注关系和端口：

```
┌──────────────────────────────────────────────────────────┐
│                      用户 / UI                            │
└────────────┬─────────────────────────────────────────────┘
             │ HTTP (12345)
┌────────────▼────────────┐
│      API Server         │  ← 入口：接收请求，写Command到DB
└────────────┬────────────┘
             │
┌────────────▼─────────────────────────────────────────────┐
│               Metadata DB (MySQL/PostgreSQL)              │
│  工作流定义 | 实例 | 任务 | Command | 历史 | 租户/队列   │
└──────────────────────────────────────────────────────────┘
             ▲                    ▲
             │                    │
┌────────────┴────────┐  ┌───────┴──────────────┐
│   Master Server(s)  │  │  Registry            │
│  (DAG状态机+派发)    │  │  (ZK/Etcd/JDBC)      │
│  gRPC:5679          │  │  服务发现+选举+锁     │
└─────────┬───────────┘  └──────────────────────┘
          │ 任务派发(gRPC/Netty)
┌─────────▼───────────┐      ┌──────────────────┐
│  Worker Server(s)   │      │   Alert Server   │
│  Task Plugins:      │      │  邮件/钉钉/飞书   │
│  Shell/Spark/SQL... │      │  :50052/:50053   │
│  :1235              │      └──────────────────┘
└─────────┬───────────┘
          │
┌─────────▼───────────────────────────────────┐
│   Resource Center (S3 / HDFS / Local)       │
│  脚本、JAR、UDF等资源文件统一存储             │
└─────────────────────────────────────────────┘
```

核心流转路径：**Cron定时触发 → 生成Command行写入DB → Master扫描消费Command → DAG状态机拆解任务 → Master通过gRPC派发到Worker → Worker执行并回调状态 → Master更新实例状态 → 完成后通知下游 → 异常时Alert Server告警。**

### 动手第2步：制作"术语速查表"

| 术语 | 英文 | 一句话解释 | 类比 |
|------|------|-----------|------|
| 工作流定义 | Process Definition | 一张描述任务拓扑结构的DAG图 | 工程图纸 |
| 工作流实例 | Process Instance | 工作流每次实际运行的快照 | 一次施工过程 |
| 任务 | Task | DAG图中的一个节点，描述要做什么 | 图纸上一个零件 |
| 任务实例 | Task Instance | 任务每次实际运行的记录 | 一个零件被加工了一次 |
| 命令 | Command | 触发工作流执行的指令，存于数据库中 | 工地上"开工"的口令 |
| 上游/下游 | Upstream/Downstream | 任务的依赖方向，A先于B则A是B的上游 | 地基是墙的上游 |
| 租户 | Tenant | 执行任务时使用的操作系统用户 | 各个承包队 |
| 队列 | Queue | 提交到YARN/K8S时使用的资源队列 | 不同施工车间的队列 |
| 工作组 | Worker Group | Worker的逻辑分组，用于任务隔离 | 焊工组、电工组 |
| 告警组 | Alert Group | 接收告警通知的人员集合 | 安全值班名单 |
| 资源中心 | Resource Center | 统一存放脚本/JAR/文件的地方 | 工具仓库 |
| 注册中心 | Registry | 服务发现与分布式协调组件 | 公司前台/花名册 |

### 动手第3步：写一个模拟crontab的调度脚本（理解其局限）

创建一个文件`/opt/scripts/cron_scheduler.sh`：

```bash
#!/bin/bash
# 一个简陋的crontab级联调度脚本 — 大量问题，仅供对比

LOG_DIR="/var/log/datapipe"
mkdir -p "$LOG_DIR"

# 第1步：订单同步
echo "[$(date)] 开始订单同步..." | tee -a "$LOG_DIR/order.log"
sleep 30  # 模拟执行
if [ $? -ne 0 ]; then
    echo "订单同步失败！" | tee -a "$LOG_DIR/order.log"
    exit 1
fi

# 第2步：库存计算（依赖订单同步）
echo "[$(date)] 开始库存计算..." | tee -a "$LOG_DIR/inventory.log"
sleep 60
if [ $? -ne 0 ]; then
    echo "库存计算失败！" | tee -a "$LOG_DIR/inventory.log"
    exit 1
fi

# 第3步：日报生成（依赖前两步）
echo "[$(date)] 开始生成日报..." | tee -a "$LOG_DIR/report.log"
sleep 45

echo "[$(date)] 所有任务完成！" | tee -a "$LOG_DIR/pipeline.log"
```

**问题分析**：这个脚本耦合了执行顺序、重试、告警逻辑；日志分散且难以检索；脚本挂了不知道哪个环节失败；多台服务器间无法协同；缺少监控与可视化。

### 动手第4步：crontab方案 vs DolphinScheduler方案对比

| 维度 | 手动crontab方案 | DolphinScheduler方案 |
|------|----------------|---------------------|
| 依赖管理 | sleep/&&硬编码，极易出错 | 可视化DAG拖拽，自动分析上下游 |
| 失败处理 | 脚本exit，后面全断 | 可配置失败策略：继续/重试/终止/等待 |
| 多机分布式 | 需要手工rsync脚本到每台机器 | 天然分布式，Master-Worker架构 |
| 监控告警 | 翻日志文件，靠人发现 | Alert Server多通道推送，有历史记录 |
| 重试机制 | 手工写while循环，简陋 | 原生支持：重试次数、间隔、超时 |
| 多租户隔离 | 难以实现 | 内置Tenant、Worker Group、Queue机制 |
| 任务类型 | 仅shell | 插件化：SQL/Spark/Flink/Python/DataX等30+种 |
| 资源管理 | 脚本散落各节点 | 统一资源中心(S3/HDFS/本地) |
| 补数/重启 | 手工重新跑 | 界面一键补数、恢复失败、暂停继续 |
| 可观测性 | 无 | Gantt图、血缘图、任务耗时分析、审计日志 |

## 4. 项目总结

### 竞品对比：DolphinScheduler vs Azkaban vs Airflow

| 对比维度 | DolphinScheduler | Azkaban | Airflow |
|---------|------------------|---------|---------|
| 架构 | 原生分布式Master-Worker | 单点Web+Executor | 单点Scheduler（需Celery扩展） |
| DAG编辑 | 可视化拖拽 | 需写flow文件（.job） | 纯Python代码定义 |
| 高可用 | ZK选主，多Master热备 | Executor单点风险 | 需额外配置 |
| 任务插件 | 30+种（Shell/Spark/Flink/DataX...） | 较少 | 丰富（社区Operator） |
| 学习曲线 | 低（拖拽+配置） | 中 | 高（需Python编程） |
| 多租户 | 原生支持 | 弱 | 需Airflow RBAC配置 |
| 社区活跃度 | Apache顶级项目，中国企业积极贡献 | 维护变慢 | CNCF孵化，活跃但复杂 |

### 适用场景

1. **多任务依赖调度**：每日数百个ETL/报表任务，DAG拓扑复杂
2. **多租户数据平台**：不同部门共享调度资源，需要隔离
3. **混合云/多云环境**：部分任务跑在Hadoop集群、部分在K8s
4. **可视化需求强的团队**：业务人员也能看懂DAG依赖关系
5. **需要易运维的调度**：不想折腾Airflow的Python代码和Celery配置

### 不适用场景

1. **纯实时流式处理**：如果你想做毫秒级事件驱动处理，Flink/Kafka Streams更合适
2. **极简场景（<5个任务）**：杀鸡不必用牛刀，crontab足够了

### 生产踩坑三则

**坑1：Command表堆积** —— 某团队配置了每分钟触发一次工作流，但下游Worker处理不过来，Command表日积月累上百万行，导致Master扫表越来越慢。**解法**：合理控制触发频率，定期清理历史Command，设置并行度上限。

**坑2：Worker Group配置遗忘** —— 搭建时忘了创建Worker Group，所有Worker默认挤在`default`组。一个资源消耗大的Flink任务占满了组内所有Worker，导致其他任务抢不到资源。**解法**：提前规划Worker Group，按任务类型和优先级隔离。

**坑3：租户sudo权限缺失** —— Worker节点上配置了Tenant `hadoopuser`，但漏配了`/etc/sudoers`免密权限。任务提交时Worker执行`sudo -u hadoopuser`提示输入密码，全部任务卡死。**解法**：部署时严格检查所有Worker节点上租户的sudo免密配置。

### 思考题（留给下一章）

1. 如果Master扫描到Command后还没来得及派发就挂了，等新Master选出来，这个Command会不会被重复执行？DolphinScheduler是如何通过"ack"和"超时重调度"来保证既不丢失也不重复的？
