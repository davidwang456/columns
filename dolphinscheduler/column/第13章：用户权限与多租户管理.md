# 第13章：用户权限与多租户管理

> **定位**：从"全员admin"到角色隔离——用租户、用户、队列、Worker分组四层模型构建多部门数据安全体系。
> **核心内容**：租户与Linux用户的映射关系、用户-租户绑定机制、队列与YARN资源隔离、Worker分组与任务亲和性、API令牌管理、项目级读写执行权限矩阵。
> **实战目标**：为财务、市场、工程三个部门搭建完全隔离的DolphinScheduler多租户环境，实现"各自的数据源只能被各自的用户看到，各自的任务只跑在各自的Worker上"。

---

## 1. 项目背景

某互联网公司数据团队规模50人，分属三个部门：财务部（8人，负责利润报表、成本核算、税务合规——数据极其敏感，泄露意味着合规风险）、市场部（20人，负责用户画像、广告投放归因、活动效果分析）、工程部（22人，负责基础数据管道、ETL清洗、实时计算平台）。三个部门的日常调度任务全部跑在同一个DolphinScheduler集群上。

然而安全现状令人心惊：所有50人共用一个`admin/admin`账号登录。权限体系形同虚设，任何人登录后都能看到所有项目、所有工作流定义、所有数据源配置。上个月，一位刚入职三天的实习生想删除自己练习用的测试工作流，却不小心选中了"财务部周报生成流程"，点击删除后扬长而去。当晚定时调度静默失败，财务总监第二天打开邮箱——空的。团队花了整整两个小时才从Git备份中找回工作流JSON定义，手动重建后又逐条核对定时配置。

比误删更可怕的是数据泄露风险。市场部同事在测试数据源连通性时，赫然发现自己能浏览财务部的数据源——里面静静躺着公司尚未公布的季度损益明细表，P&L数据一览无余。CISO在安全审计会议上拍了桌子："这不是运维问题，这是数据治理事故。一个实习生能删掉核心报表，一个市场分析师能看到财务核心数据——你们觉得下一次审计还能过关吗？我们需要部门级别的访问隔离、基于角色的权限控制、以及完整的审计追溯。"

这恰好是DolphinScheduler安全模块的用武之地——租户隔离任务执行身份、用户控制UI访问范围、队列管理计算资源、Worker分组限定执行位置，四层防线构成完整的多租户安全体系。

## 2. 项目设计——剧本式交锋对话

下午茶时间，小胖端着一杯奶茶晃到大师工位前，手里拿着一张纸，上面歪歪扭扭画了几个框。

**小胖**："大师，CISO昨天那个会开完，我琢磨了一下——这不就是开三个账号嘛！财务一个、市场一个、工程一个，每个人用自己账号登录，各管各的项目就完事了。这有什么难的？"

**大师**接过奶茶喝了一口："你的思路方向对了，但粒度远远不够。我问你——财务部的小张登录DS之后，创建了一个Shell任务执行`cat /data/finance/salary.csv`，这个命令究竟是以哪个Linux用户的身份在Worker机器上运行的？"

**小胖**愣了一下："呃……Worker进程用什么用户启动，任务就用什么用户跑？"

**小白**从屏幕后探出头来："不对。如果Worker进程用`dolphinscheduler`用户启动，那所有任务都以`dolphinscheduler`身份执行。财务部的敏感数据目录只要对`dolphinscheduler`用户开放了读权限，市场部同事写个脚本一样能读到。你这三个账号的方案解决不了操作系统层面的权限隔离。"

**小胖**："那……每个任务在Worker上执行之前，先`su`到对应的Linux用户？"

**大师**赞许地点头："就是这个思路，DolphinScheduler里专门有一个概念叫**租户（Tenant）**。租户不是一个UI登录账号，而是一个Linux操作系统用户的映射。你在DS中创建租户`finance_tenant`，把它关联到Worker节点上的Linux用户`finance_user`。那么任何绑定了`finance_tenant`的DS用户提交的任务，在Worker上运行时都会以`finance_user`的身份执行——HDFS权限、本地文件访问权限、Hive表读写权限，全部由这个Linux用户决定。"

**技术映射**：租户 = Linux OS用户的DS代理。任务执行身份不由DS UI登录账号决定，而由登录账号所绑定的租户决定——这是一层关键的间接映射。

**小白**放下笔，一连串追问："那我有四个问题。第一，用户和租户之间是一对一还是多对一？比如财务部8个人，是8个用户绑1个租户，还是每人一个租户？第二，刚才提到HDFS权限——如果财务和市场两个部门都要往HDFS写数据，怎么防止市场部的任务不小心覆盖财务部的文件？第三，Worker节点那么多，能不能让财务部的任务只跑在特定几台机器上，不和市场的任务抢资源？第四，那个队列管理是干什么用的？"

**大师**在白板上画了一个四层架构图：

```
[用户层]  zhang_wei (财务)  →  li_na (市场)  →  wang_tech (工程)
              ↓                    ↓                  ↓
[租户层]  finance_tenant   marketing_tenant  engineering_tenant
              ↓                    ↓                  ↓
[系统层]  Linux:finance_user Linux:marketing_user Linux:eng_user
              ↓                    ↓                  ↓
[资源层]  finance_queue(50%) marketing_queue(30%) default(20%)
              ↓                    ↓                  ↓
[执行层]  Finance_WG        Marketing_WG        Default(共享)
```

"一个个来。**用户与租户**：一个用户可以绑定一个租户（DS 3.x支持多租户绑定，但生产环境建议一对一以简化审计）。财务部8个用户可以全部绑定`finance_tenant`——这样8个人提交的任务都以`finance_user`身份运行，拥有相同的Linux层权限。

**HDFS隔离**：创建HDFS目录时直接按租户划分配额和权限：

```bash
# 以hdfs超级用户执行
hdfs dfs -mkdir -p /data/finance  /data/marketing  /data/engineering
hdfs dfs -chown finance_user:hadoop /data/finance
hdfs dfs -chown marketing_user:hadoop /data/marketing
hdfs dfs -chown engineering_user:hadoop /data/engineering
hdfs dfs -chmod 750 /data/finance  # 同组可读，其他用户无权限
```

这样即使市场部的任务脚本里写了`hdfs dfs -cat /data/finance/salary.csv`，操作系统层面也会直接报`Permission denied`——不是DS管，是Linux/HDFS管，防线在更底层。

**Worker分组**：DS支持按标签将Worker节点分组。假设你有5台Worker，可以配置`Finance_WG`包含worker-01和worker-02，`Marketing_WG`包含worker-03和worker-04，worker-05作为公共池。创建任务时在'Worker分组'下拉框中选择对应分组，任务就会被调度到指定组的Worker上执行——既实现了计算资源物理隔离，也避免了某个部门的失控任务拖垮全集群。

**队列管理**：这里的'队列'不是消息队列，而是YARN或Kubernetes的资源队列。比如你配置YARN集群的容量调度器，给`root.finance`分配50%的集群资源，`root.marketing`分配30%。然后在DS中创建同名队列并绑定到租户——租户下的所有任务提交Spark/Flink作业时，自动使用对应的YARN队列名。没有DS这层映射，你需要在每个脚本里硬编码`--queue root.finance`，改起来很痛苦。"

**技术映射**：Worker分组解决"在哪里跑"，队列解决"用多少资源跑"，租户解决"以谁的身份跑"——三个维度正交，组合出灵活的资源隔离策略。

**小胖**恍然大悟："所以admin/admin的问题不光在于密码太弱，更在于只有一个租户、一个队列、一个Worker分组——所有人在同一个身份下操作，边界完全不存在！另外，大师你说admin只有一个是什么意思？"

**大师**："DolphinScheduler默认只有admin账号拥有管理员角色。管理员能看到所有项目、管理安全中心（创建/删除租户、用户、队列、令牌）、修改系统配置。普通用户登录后，安全中心是只读的，只能看到自己有权限的项目列表。而且，admin账号不能删除，也不建议改名——它是系统最后的安全防线。普通用户即使绑定了有权限的租户，也绝对不应该被授予管理员角色，除非你希望他也能随意删掉别人的租户定义。"

**小白**最后问道："那API调用怎么办？总不能让自动化脚本也用账号密码登录吧？"

**大师**："令牌（Token）。在安全中心为每个用户生成独立的API Token，有效期可设。外部脚本通过HTTP Header携带Token即可调用API——Token泄露的影响范围仅限于该用户的项目权限，而不是整个系统。如果Token泄露，在UI上一键失效即可，不用改所有人的密码。"

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x 集群已部署，至少3个Worker节点在线
- 所有Worker节点操作系统为CentOS 7/Ubuntu 20.04+，拥有root或sudo权限
- Worker节点已安装Hadoop/HDFS客户端（如无HDFS环境可跳过HDFS隔离步骤）
- YARN集群已配置容量调度器（如无YARN环境可跳过队列创建步骤）

### 第1步：在所有Worker节点上创建Linux用户

SSH登录到每一台Worker节点，执行以下命令创建三个部门的Linux用户：

```bash
# 在每台Worker机器上执行（worker-01 ~ worker-05）
sudo useradd -m finance_user
sudo useradd -m marketing_user
sudo useradd -m engineering_user

# 设置密码（可选，DS通过sudo切换用户，不依赖密码）
echo "finance_user:ChangeMe123" | sudo chpasswd
echo "marketing_user:ChangeMe123" | sudo chpasswd
echo "engineering_user:ChangeMe123" | sudo chpasswd

# 验证用户创建成功
id finance_user && id marketing_user && id engineering_user
```

> **踩坑提醒**：这是最容易漏掉的步骤！DolphinScheduler不会自动在Worker节点上创建Linux用户，它只是用`sudo -u <linux_user>`切换到指定用户执行任务。如果某台Worker上不存在对应的Linux用户，任务提交到该Worker后会直接报`user not found`错误。

### 第2步：在DS安全中心创建租户

使用admin账号登录DolphinScheduler Web UI，进入【安全中心】→【租户管理】→【创建租户】。

依次创建三个租户：

| 租户名称 | Linux用户 | 队列 | 描述 |
|---------|----------|------|------|
| `finance_tenant` | `finance_user` | `finance_queue` | 财务部租户 |
| `marketing_tenant` | `marketing_user` | `marketing_queue` | 市场部租户 |
| `engineering_tenant` | `engineering_user` | `default` | 工程部租户 |

> **注意**：队列名称（`finance_queue`、`marketing_queue`、`default`）必须在第3步的队列管理中先创建好，创建租户时才能在下拉框中选到。如果队列管理页面尚无任何队列，可先创建租户时选择`default`，后续再回来修改。

### 第3步：创建资源队列

进入【安全中心】→【队列管理】→【创建队列】：

| 队列名称 | 队列值（YARN队列路径） | 描述 |
|---------|----------------------|------|
| `finance_queue` | `root.finance` | 映射YARN `root.finance`队列 |
| `marketing_queue` | `root.marketing` | 映射YARN `root.marketing`队列 |

```xml
<!-- 对应YARN capacity-scheduler.xml 配置示例 -->
<property>
    <name>yarn.scheduler.capacity.root.queues</name>
    <value>default,finance,marketing</value>
</property>
<property>
    <name>yarn.scheduler.capacity.root.finance.capacity</name>
    <value>50</value>
</property>
<property>
    <name>yarn.scheduler.capacity.root.marketing.capacity</name>
    <value>30</value>
</property>
<property>
    <name>yarn.scheduler.capacity.root.default.capacity</name>
    <value>20</value>
</property>
```

DS中队列名称`finance_queue`并不要求和YARN队列名完全一致——通过"队列值"字段建立映射关系。但建议保持命名一致以降低排查成本。

### 第4步：配置Worker分组

进入【安全中心】→【Worker分组管理】→【创建Worker分组】：

| 分组名称 | 包含Worker | 用途 |
|---------|-----------|------|
| `Default` | worker-01 ~ worker-05 | 默认公共资源池 |
| `Finance_WG` | worker-01, worker-02 | 财务部专用 |
| `Marketing_WG` | worker-03, worker-04 | 市场部专用 |

在Worker节点配置文件中指定分组（`worker.properties`或`application.yaml`，视DS版本而定）：

```properties
# worker.properties 或 application.yaml 中的配置项
# 此处为逻辑示意，实际配置需参考对应DS版本的文档
# worker.groups=Finance_WG,Default
# worker.host=worker-01
```

> **踩坑提醒**：Worker分组变更后，对应的Worker节点必须重启才能生效。如果不重启，任务仍然会被调度到旧分组配置所对应的节点上。

### 第5步：创建用户并绑定租户

进入【安全中心】→【用户管理】→【创建用户】：

| 用户名 | 密码 | 租户 | 角色 |
|-------|------|------|------|
| `zhang_wei` | `Finance@2025!` | `finance_tenant` | 普通用户 |
| `li_na` | `Marketing@2025!` | `marketing_tenant` | 普通用户 |
| `wang_tech` | `Engineering@2025!` | `engineering_tenant` | 普通用户 |
| `admin` | （保持原有密码） | `default` | 管理员 |

- 用户名建议遵循公司AD/LDAP命名规范，便于后续对接统一认证
- 每个用户在创建时必须绑定至少一个租户，否则无法提交任务
- 管理员角色只保留给admin账号，普通用户一律选"普通用户"
- 邮箱字段必填——告警通知会发送到此邮箱

### 第6步：创建项目并分配权限

使用admin账号创建三个项目：

进入【项目管理】→【创建项目】：

| 项目名称 | 负责人 | 描述 |
|---------|-------|------|
| `finance_reports` | `zhang_wei` | 财务部报表调度项目 |
| `marketing_analytics` | `li_na` | 市场部分析调度项目 |
| `engineering_infra` | `wang_tech` | 工程部基础设施项目 |

创建项目后，进入每个项目的【权限】Tab页，为各用户分配细粒度权限：

| 项目 | 用户 | 读权限 | 写权限 | 执行权限 |
|------|------|--------|--------|---------|
| `finance_reports` | zhang_wei | ✓ | ✓ | ✓ |
| `finance_reports` | li_na | ✗ | ✗ | ✗ |
| `finance_reports` | wang_tech | ✗ | ✗ | ✗ |
| `marketing_analytics` | zhang_wei | ✗ | ✗ | ✗ |
| `marketing_analytics` | li_na | ✓ | ✓ | ✓ |
| `marketing_analytics` | wang_tech | ✗ | ✗ | ✗ |
| `engineering_infra` | zhang_wei | ✓ | ✗ | ✗ |
| `engineering_infra` | li_na | ✓ | ✗ | ✗ |
| `engineering_infra` | wang_tech | ✓ | ✓ | ✓ |

- **读权限**：查看工作流定义、任务实例、历史运行记录
- **写权限**：创建/修改/删除工作流定义、定时配置、资源文件
- **执行权限**：手动运行/停止/重跑工作流实例

注意：工程部的`engineering_infra`项目对其他两个部门开放了读权限——因为基础设施的ETL产出（比如公共维度表）需要被财务和市场的下游报表消费。但写权限和执行权限仅开放给工程部人员，防止误修改核心管道。

### 第7步：创建专属数据源

进入【安全中心】或项目内的【数据源管理】，为每个项目创建独立的数据源：

| 数据源名称 | 类型 | 连接信息 | 所属项目 | 可见用户 |
|-----------|------|---------|---------|---------|
| `finance_mysql` | MySQL | `jdbc:mysql://db-cluster/finance_db` | `finance_reports` | zhang_wei |
| `marketing_mysql` | MySQL | `jdbc:mysql://db-cluster/marketing_db` | `marketing_analytics` | li_na |
| `engineering_hive` | Hive | `jdbc:hive2://hive-server/ods` | `engineering_infra` | wang_tech |

数据源创建时指定所属项目后，只有拥有该项目权限的用户才能在SQL任务节点的"数据源"下拉框中看到它。这是防止跨部门数据源泄露的关键配置——财务数据源的连接信息、库名、表名对市场部完全不可见。

### 第8步：创建API令牌

进入【安全中心】→【令牌管理】→【创建令牌】：

| 用户 | 过期时间 | 用途 |
|------|---------|------|
| `zhang_wei` | 2026-12-31 | 财务日报自动触发 |
| `li_na` | 2026-12-31 | 市场分析Jenkins集成 |

点击生成后立即复制Token值——Token仅在创建时显示一次，关闭弹窗后无法再次查看（只能重新生成）。

验证API访问：

```bash
# 使用Token列出项目（zhang_wei只能看到finance_reports）
curl -H "token: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
     http://api-server:12345/dolphinscheduler/projects

# 使用Token触发工作流运行
curl -X POST \
     -H "token: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
     -H "Content-Type: application/json" \
     -d '{"failureStrategy":"CONTINUE","processDefinitionCode":12345678901}' \
     http://api-server:12345/dolphinscheduler/projects/finance_reports/executors/start-process-instance
```

> **安全警告**：绝对不要把Token硬编码在脚本中提交到Git仓库。推荐做法是将Token存储在环境变量或公司内部的密钥管理服务（如Vault）中：
> ```bash
> # 正确做法：从环境变量读取
> TOKEN="${DS_FINANCE_TOKEN}"  # 由CI/CD或密钥管理服务注入
> curl -H "token: ${TOKEN}" http://api-server:12345/dolphinscheduler/projects
> ```

### 第9步：验证隔离效果

分别以不同用户身份登录DS UI，验证权限隔离：

**验证1：登录zhang_wei（财务部普通用户）**
- 首页项目列表只显示`finance_reports`
- 点击"安全中心"——各项菜单均为只读（无法创建/删除租户/用户/队列）
- 进入`finance_reports`，创建Shell任务并运行，在Worker上执行`whoami`验证：输出应为`finance_user`
- 在SQL任务节点的数据源下拉框中，只能看到`finance_mysql`

**验证2：登录li_na（市场部普通用户）**
- 首页项目列表只显示`marketing_analytics`
- 尝试在浏览器地址栏直接输入URL访问`finance_reports`项目——返回"无权限访问"提示
- 在Worker上执行任务时，`whoami`输出为`marketing_user`

**验证3：登录admin（管理员）**
- 首页项目列表显示所有三个项目
- 安全中心全部可操作
- 能查看任意用户的租户绑定信息和权限配置

### 第10步：常见踩坑

**坑1：忘了在全部Worker节点上创建Linux用户**

`finance_user`只在worker-01上创建了，但worker-03上没有。任务被调度到worker-03后报错`sudo: unknown user: finance_user`，任务失败。修复后再提交，任务被调度到同一台Worker——因为重试默认不切换Worker节点，除非配置了`failover`策略。

**解决**：使用Ansible或SaltStack在所有Worker节点上批量创建用户，运维脚本中加入`id <username>`检查作为预检步骤。

**坑2：队列名称不匹配**

DS中队列的"队列值"填的是`root.finance`，但YARN容量调度器里实际的队列路径是`root.finance_queue`。任务提交Spark作业时报`Queue does not exist`错误，排查耗时——日志中只显示了队列名，没有显示完整路径。

**解决**：进入YARN ResourceManager UI（默认端口8088），确认`Scheduler`页面中队列的完整路径名，然后回到DS队列管理中核对"队列值"字段。

**坑3：Token泄露到版本控制**

开发人员在测试脚本中硬编码了Token并推送到Git仓库，Token在公共仓库中暴露了3天。虽然该Token只有`zhang_wei`的普通用户权限，但攻击者可以利用它提交大量垃圾任务，或者读取财务项目的所有工作流定义。

**解决**：在DS安全中心立即对该Token执行"失效"操作。在CI/CD中配置Token环境变量注入。在`.gitignore`中添加`*token*`规则。定期审查Git历史中的敏感信息。

**坑4：Worker分组变更不生效**

在UI中修改了`Finance_WG`的成员节点，保存后新提交的任务仍然跑在旧节点上。原因是Worker服务没有重启——Worker分组信息在Worker启动时加载到内存，运行期间不会动态刷新。

**解决**：修改Worker分组后，对涉及的Worker节点逐台执行滚动重启：先停一台Worker → 确认任务已调度到其他节点 → 启动该Worker → 验证分组生效 → 处理下一台。

**坑5：数据源跨项目可见**

在项目A中创建数据源时忘了指定"所属项目"，导致该数据源变为全局可见——项目B的用户在SQL节点配置中也能选到它。

**解决**：创建数据源时务必勾选所属项目。如果已经创建了全局数据源，可以编辑修改其归属。管理员定期检查数据源列表，筛选出"所属项目为空"的条目并逐一修正。

## 4. 项目总结

### 多租户架构对比：DolphinScheduler vs Airflow vs Azkaban

| 维度 | DolphinScheduler | Apache Airflow | Azkaban |
|------|-----------------|----------------|---------|
| 租户机制 | 原生支持——租户映射Linux用户，与用户体系解耦 | 无原生租户概念，依赖OS用户或Kerberos | 无租户概念，权限管理极其简单 |
| 用户与权限 | 项目级读/写/执行三级权限，管理员/普通用户角色 | RBAC粒度更细（DAG级、Variable级），但配置复杂 | 仅Reader/Writer/Admin三种角色 |
| 队列管理 | 内置队列管理，映射YARN/K8s队列 | 无内置队列管理，需在DAG代码中自行指定 | 无队列概念 |
| Worker分组 | 原生支持标签分组，任务可绑定到指定Worker组 | 通过Celery Queue或K8s Pod Affinity间接实现 | 不支持 |
| Token管理 | 内置Token生成、过期、失效管理 | 通过Airflow API的`@auth`装饰器手动管理 | 不直接支持 |
| 学习成本 | 中——概念较多但图形化配置 | 高——需用Python编写DAG | 低——功能少，配置简单 |
| 适用规模 | 中大型团队（20-200人） | 中大型团队，Python技术栈 | 小型团队（<20人） |

### 权限模型最佳实践

1. **最小权限原则**：用户只授予完成工作所需的最小权限。普通用户默认连安全中心都看不到编辑按钮——这比事后审计有效得多。
2. **租户即边界**：一个租户对应一个业务域。不要多个部门共用一个租户——一旦共用了，HDFS权限无法拆分，数据隔离形同虚设。
3. **定期权限审计**：每季度导出用户-租户-项目-权限矩阵到Excel，由各部门Leader签字确认。离职员工权限在最后工作日当天回收——Token失效、用户禁用。
4. **Token轮换**：API Token设置明确的过期时间（建议不超过6个月），过期前一周通过告警通知负责人续签。轮换时采用"新建Token → 更新下游 → 删除旧Token"的流程，避免因忘记更新导致生产任务静默失败。
5. **Worker分组与队列协同**：Worker分组解决的是计算节点层面的物理隔离，队列解决的是YARN层面的资源配额隔离。两者互补而非替代——在共用Worker的场景下，仍可以通过不同的YARN队列限制Spark任务的资源使用上限。

### 适用场景

- **多部门共享DolphinScheduler集群**：不同业务部门需要数据隔离但不想维护多套DS实例
- **金融/医疗等强合规行业**：数据分类分级管理，必须实现HDFS级别和数据库级别的权限隔离
- **开发/测试/生产环境共用集群**：通过不同租户和队列区分环境，开发任务的失控不会影响生产任务
- **SaaS多租户数据平台**：为每个客户创建独立租户和项目，通过API Token对接自动化系统
- **外部合作伙伴接入**：给合作伙伴分配租户和项目，限制其可见的数据源和可执行的操作

### 不适用场景

- **单用户研究/实验环境**：一个人用DS学习或做实验时，多租户配置的维护成本远大于安全收益。一个admin账号足够。
- **团队规模极小且完全互信**：3人以下团队、数据全部公开共享的情况下，全套多租户配置反而是过度设计。

### 思考题

1. 如果财务部需要新增一个只读用户（实习生小王，只能查看工作流运行状态，不能修改任何定义也不能手动触发执行），请设计小王的用户配置方案——绑哪个租户、配什么权限、项目权限如何设置。

2. 市场部每天凌晨2点定时运行一组Spark任务，平均占用集群60%的计算资源。某天市场部的数据量暴涨导致任务跑了3小时还没结束，恰好财务部在凌晨4点也有定时任务需要执行，却因为YARN队列资源被市场部占满而一直排队等待。请分析：如果在DS层面调整队列的容量配置（将`root.marketing`从50%降到30%），是否需要在Worker分组层面也做调整？两者之间是什么关系？
