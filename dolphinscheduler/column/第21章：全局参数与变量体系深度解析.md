# 第21章：全局参数与变量体系深度解析

> **定位**：把DolphinScheduler的参数体系从头捋一遍——从系统内置变量到自定义参数的IN/OUT传递，再到启动参数的动态覆盖，让参数不再"神出鬼没"。
> **核心内容**：系统内置参数、自定义参数IN/OUT机制、${setVar=}语法解析原理、全局参数与局部参数的作用域层级、启动参数覆盖规则、Worker侧参数替换全流程。
> **实战目标**：搭建一条"日期参数→数据加工→产出文件路径→下游消费"的完整参数链路，掌握参数调试三板斧。

---

## 1. 项目背景

大麦电商的数据调度体系已演进到15+条工作流、上百个任务节点的规模，但随着规模增长，参数管理逐渐失控。数据团队面临四个典型痛点：

(1) **硬编码日期泛滥**：部分老工作流里日期直接写死为`2024-01-15`，每天早上分析师第一件事就是手动改日期、重新上线。上周二小明请假，替班同事不知道要改日期，导致报表跑的是旧数据，管理层晨会上当场发飙。

(2) **参数在任务间丢失**：上游Shell任务计算出了`ROW_COUNT=15000`，下游Python任务却拿不到这个值——分析师只好在上游跑完后手动抄下结果，再粘贴到下游脚本里重新跑。有一次抄错了数字，15000抄成了150000，下游直接触发了一级告警，运维半夜爬起来发现是虚惊。

(3) **补数场景参数混乱**：运营要求重跑去年"双十一"当天的报表做对比分析，但工作流一启动，`system.biz.date`自动取的是昨天，而非2023-11-11。分析师不得不在工作流定义里临时改全局参数，跑完再改回去——来回切换参数配置，一不小心就污染了生产环境的正常运行。

(4) **参数无法跨节点传递**：任务A生成了一个文件路径`/data/output/20240315/summary.parquet`，任务B需要将这个文件作为输入。但在目前的脚本写法里，路径要么写死在代码中，要么靠约定俗称的命名规律来"猜"——一旦文件落盘位置变动，下游全部断裂。

本质问题在于：**缺乏一套系统性的参数传递框架**——包括日期参数的自动推算、衍生参数的跨任务传递（行数、文件路径、状态码）、以及人工手动覆盖的能力。DolphinScheduler的参数体系正是为解决这类问题而设计，本章带你逐一拆解。

## 2. 项目设计——剧本式交锋对话

周一晨会，老张把上周年度"双十一"数据回补时参数混乱的糗事讲了一遍。大师说："今天我们不聊新功能，先把参数体系掰扯清楚——这是调度器的'血脉'。"小胖和小白立刻凑了过来。

**小胖**（手里转着笔，一脸不屑）：

> "参数不就是环境变量嘛！上游`export ROW_COUNT=100`，下游直接`echo $ROW_COUNT`不就行了？我们Linux用了这么多年，环境变量传参不是基本操作吗？"

**小白**（放下手里的保温杯，眉头一挑）：

> "你这思路放在单机脚本里没问题，但DS是分布式调度——上游任务跑在Worker-1上，下游可能被调度到Worker-3上。你Worker-1上export的环境变量，Worker-3读得到吗？再说了，`system.biz.date`是怎么算出来的？为什么取的是昨天而不是今天？全局参数和启动参数同时存在时，到底用哪个？OUT参数在Worker侧怎么解析——它是去日志里抠字符串吗？传的值能是JSON对象吗？参数替换发生在什么时候——是Master调度时还是Worker执行前？"

**小胖**（笔啪嗒掉桌上，愣了两秒）：

> "呃……你这一口气问了七八个问题。环境变量跨机器确实不行，那DS到底怎么传的？"

**大师**（从工位起身，走到白板前画了一个接力赛的场景）：

> "DS的参数体系就像一个**接力赛**。每个参赛选手（任务节点）手里都有一根接力棒（参数）。棒上有两个方向：**IN就是接棒**——从上游手里拿过来，**OUT就是交棒**——跑完自己的赛段后交给下一个人。"

> "全局参数呢？就像**比赛规则**——'每棒必须跑100米'，适用所有选手。启动参数呢？就像**裁判临时吹哨改规则**——'本场比赛改为200米'，裁判说了算，覆盖原规则。"

> "那`system.biz.date`呢？这是DS自带的'官方计时器'——把'昨天'作为默认的业务日期，因为多数数据仓库的ETL都在凌晨跑前一天的数据，今天0点产生T-1分区，咱早上起跑正好对得上。"

**技术映射**：DS的参数分为四层——系统内置参数（如`system.biz.date`、`system.biz.curdate`、`system.datetime`）、全局参数（工作流级别定义，所有节点可见）、局部参数（单个任务节点自定义，优先级高于全局）、启动参数（工作流实例启动时手动传入，优先级最高）。优先级链条：启动参数 > 局部参数 > 全局参数 > 系统默认值。Worker侧在fork子进程之前完成参数替换，将`${参数名}`替换为实际值后写入临时执行脚本。

**小白**（在笔记本上快速画了一个嵌套的环形图）：

> "那`${setVar=ROW_COUNT=15000}`这条语法是DS自己定义的吧？Worker是去stdout里用正则把这行抠出来吗？抠出来之后存到哪了？下游拿的是字符串还是数字？如果我同时输出三行setVar，Worker会全部解析吗？"

**大师**：

> "问得好。`${setVar=VAR_NAME=VALUE}`是DS定义的特殊输出协议——Worker通过管道采集子进程stdout，用正则匹配`\$\{setVar=([^=]+)=(.+)\}`模式。一旦匹配成功，Worker将参数名和参数值写入任务实例的参数上下文，存储到DS元数据库（MySQL）的`t_ds_task_instance`表里。下游节点启动时，Master从数据库读取上游OUT参数，作为下游IN参数的默认值注入。三行setVar当然全都能解析，但注意这是字符串解析——所有值在元数据库里都是VARCHAR存着，下游拿到的是字符串。你要在Python里做数值比较，记得`int(row_count)`转一手。"

**技术映射**：setVar解析位于`dolphinscheduler-task-executor`模块中。Worker采集到输出行后调用参数解析器，提取键值对并持久化到任务实例上下文。下游任务在参数替换阶段从上下文查询同名IN参数即可获取上游传递的值。特别需要注意：参数名严格大小写敏感，`ROW_COUNT`和`row_count`是两个不同参数；OUT参数只能传播到当前工作流内的直接下游节点，不能穿透SubProcess边界。

**小胖**（终于找到话头）：

> "那启动参数是怎么玩的？是不是我手动点'运行'的时候，弹窗里填的那个？填了就覆盖全局参数，不填就取默认值？"

**大师**点点头：

> "正是。启动参数是'一次性临时覆盖'——工作流定义本身不动，但本次运行用你指定的值。比如运营要看去年双十一的数据，你启动时填`biz_date=20231111`，整个工作流实例的所有`${biz_date}`全变成20231111。跑完自动恢复，不会污染定义。这就是'一份定义，多场景运行'。"

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x 集群已部署运行
- 至少一个Worker节点在线
- 工作流项目名称：`param_system_demo`
- 一个可执行Shell和Python的Worker节点

### Step 1：定义工作流级全局参数

创建新工作流`daily_sales_pipeline`，在工作流定义页面的"全局参数"中配置：

| 参数名 | 参数值 | 说明 |
|--------|--------|------|
| biz_date | `${system.biz.date}` | 业务日期，取昨天 |
| ds_env | `production` | 环境标识，静态字符串 |
| alert_threshold | `100000` | 告警阈值，数值型 |

> **要点**：`${system.biz.date}`是DS系统的内置参数，格式为yyyyMMdd（如20210825）。常用的内置参数还有`${system.biz.curdate}`（当天日期）和`${system.datetime}`（当前日期时间，yyyyMMddHHmmss格式）。全局参数在整个工作流的所有任务节点中均可通过`${参数名}`引用。

### Step 2：在Shell任务中使用全局参数

创建Shell任务节点，命名为`fetch_data`，脚本如下：

```bash
#!/bin/bash
echo "========================================="
echo "Processing data for date: ${biz_date}"
echo "Environment: ${ds_env}"
echo "Alert threshold: ${alert_threshold}"
echo "========================================="

# 使用biz_date查询数据库
mysql -h db.damai.com -u analyst -e "
  SELECT COUNT(*) FROM dw.orders WHERE dt='${biz_date}'
"
```

> **原理**：Worker在执行前会将脚本中的`${biz_date}`替换为实际值（如`20240315`），然后才创建子进程执行。这意味着在UI日志里看到的已经是替换后的结果，方便排查。

### Step 3：输出自定义OUT参数

在上一步的脚本末尾，加上参数输出逻辑：

```bash
#!/bin/bash
# ... 业务逻辑 ...
ROW_CNT=$(mysql -h db.damai.com -u analyst -sN \
  -e "SELECT COUNT(*) FROM dw.orders WHERE dt='${biz_date}'")

echo "Processed ${ROW_CNT} rows"

# ★核心语法★：setVar输出三连
echo '${setVar=ROW_COUNT='${ROW_CNT}'}'
echo '${setVar=OUTPUT_PATH=/data/output/'${biz_date}'}'
echo '${setVar=PROCESS_STATUS=SUCCESS}'
```

在任务的"自定义参数"标签页中声明这三个OUT参数：

| 参数名 | 参数值 | 方向 |
|--------|--------|------|
| ROW_COUNT | | OUT |
| OUTPUT_PATH | | OUT |
| PROCESS_STATUS | | OUT |

> **要点**：`${setVar=参数名=参数值}`必须输出到stdout，Worker会自动捕获并解析。参数值可以是数字、路径字符串、状态码等任意文本——但本质都是字符串。注意：如果OUTPUT_PATH的值包含空格或特殊字符，建议用引号包裹，避免解析截断。另外，参数名要大写+下划线命名，团队约定统一。

### Step 4：下游任务接收IN参数（Python）

创建Python任务节点，命名为`generate_report`，自定义参数配置：

| 参数名 | 参数值 | 方向 |
|--------|--------|------|
| ROW_COUNT | | IN |
| OUTPUT_PATH | | IN |
| PROCESS_STATUS | | IN |

Python脚本：

```python
#!/usr/bin/env python3
import os, sys

# DS将上游OUT参数注入为环境变量
row_count = os.environ.get('ROW_COUNT', '0')
output_path = os.environ.get('OUTPUT_PATH', '/tmp/default')
status = os.environ.get('PROCESS_STATUS', 'UNKNOWN')

print(f"Received: ROW_COUNT={row_count}, PATH={output_path}, STATUS={status}")

# 根据数据量选择分析策略
if int(row_count) > 100000:
    print("Large dataset → running Spark aggregation")
else:
    print("Small dataset → running local pandas summary")
```

> **原理**：Worker在执行Python任务前，将上游OUT参数的值作为环境变量注入子进程，Python通过`os.environ.get()`读取。注意：所有参数值都是字符串，做数值比较时务必`int()`转换。

### Step 5：配置DS UI中的参数传递

工作流DAG连线为：`[fetch_data] → [generate_report]`。两个节点之间的参数传递靠**参数名匹配**——上游OUT参数名与下游IN参数名必须完全一致（大小写敏感）。不需要额外配置"连线传参"，DS会自动根据参数名来传递。

### Step 6：参数优先级测试

在同一工作流中添加三个参数来源冲突的场景：

```
全局参数：    MY_VAR = "global_value"
启动参数：    MY_VAR = "startup_value"
局部参数：    MY_VAR = "local_value"
```

测试结果：
- **正常上线运行** → 各任务取全局值`"global_value"`
- **启动时填写MY_VAR=startup_value** → 所有任务取`"startup_value"`（启动参数覆盖全局）
- **某任务配置了局部参数MY_VAR=local_value** → 该任务取`"local_value"`（局部参数覆盖启动和全局）

优先级链条：**局部参数 > 启动参数 > 全局参数 > 系统默认值**

### Step 7：三节点参数链式传递

扩展工作流为三节点链式传递：

```
[Task1: 数据采集] → [Task2: 数据清洗] → [Task3: 报告生成]
```

- **Task1**（Shell）输出：`echo '${setVar=RAW_FILE=/data/raw/20240115.csv}'`
- **Task2**（Shell）接收`RAW_FILE`（IN），处理后输出：`echo '${setVar=CLEAN_FILE=/data/clean/20240115.parquet}'`
- **Task3**（Python）接收`CLEAN_FILE`（IN），生成最终报告

> **验证方法**：运行后依次查看每个任务实例的"自定义参数"标签页——可以看到该节点实际接收到的IN参数值，以及它产出的OUT参数值。这是参数调试的第一板斧。

### Step 8：日期参数的高级转换

在Shell任务中对`biz_date`进行格式变换：

```bash
#!/bin/bash
BIZ_DATE_STR="${biz_date}"                          # 20240115
BIZ_DATE_DASH=$(echo $BIZ_DATE_STR | sed 's/\(....\)\(..\)\(..\)/\1-\2-\3/')  # 2024-01-15
NEXT_DAY=$(date -d "${BIZ_DATE_DASH} + 1 day" +%Y%m%d)  # 20240116
PREV_WEEK=$(date -d "${BIZ_DATE_DASH} - 7 day" +%Y%m%d) # 20240108

echo "biz_date:      ${BIZ_DATE_STR}"
echo "Formatted:     ${BIZ_DATE_DASH}"
echo "Next day:      ${NEXT_DAY}"
echo "Prev week:     ${PREV_WEEK}"

echo '${setVar=NEXT_BIZ_DATE='${NEXT_DAY}'}'
echo '${setVar=PREV_WEEK_DATE='${PREV_WEEK}'}'
```

### Step 9：参数调试三板斧

**第一板斧——看日志**：进入任务实例→"日志"标签页，查看实际执行的命令。Worker已经将`${变量}`全部替换为实际值，你可以看到最终执行了什么。如果参数没替换，说明参数名写错了。

**第二板斧——看自定义参数**：进入任务实例→"自定义参数"标签页，可以看到该节点接收到的所有IN参数及其实际值。如果某个参数显示为空，说明上游OUT没有正确产出，或参数名不匹配。

**第三板斧——env全量导出**：在调试阶段，可以临时插入一个Shell任务节点，脚本只写一行`env | sort`。执行后查看日志，就能看到Worker注入的所有环境变量，一目了然。

### Step 10：常见踩坑清单

| 陷阱 | 现象 | 根因与解决 |
|------|------|-----------|
| **参数名大小写不匹配** | 下游拿到的参数值为空 | DS参数名严格大小写敏感。`ROW_COUNT`和`row_count`是两个不同的参数。解决：统一使用`UPPER_CASE`命名规范 |
| **setVar语法未换行** | Worker没有解析到OUT参数 | setVar行夹杂在其他日志中可能被忽略。建议将每个`${setVar=}`放在独立一行 |
| **特殊字符导致截断** | 参数值不完整 | 参数值中含空格、`$`、引号等特殊字符时，Worker的正则解析可能截断。解决：对特殊值做base64编码后传递 |
| **SubProcess参数隔离** | 父流程的参数在子流程中不可用 | SubProcess本质是一个独立的工作流实例，OUT参数不穿透子流程边界。解决：在SubProcess节点上显式声明IN/OUT参数桥接 |
| **数值比较陷阱** | Conditions节点中`"1000"`不等于1000 | 所有参数值都是字符串。在Conditions节点做数值比较时使用"数值型比较"而非"字符串型比较" |
| **启动参数忘记填写** | 手动启动后跑的还是默认值 | 启动弹窗中参数名必须与全局参数名完全一致才会覆盖。拼写错误会导致静默回退到全局默认值 |

## 4. 项目总结

### 参数作用域对比

| 维度 | 系统内置参数 | 全局参数 | 局部参数 | 启动参数 |
|------|------------|---------|---------|---------|
| 定义位置 | DS系统硬编码 | 工作流定义页 | 任务节点自定义参数 | 启动对话框 |
| 作用范围 | 所有工作流 | 当前工作流所有节点 | 仅当前任务节点 | 当前工作流实例 |
| 优先级 | 最低（可被覆盖） | 中等 | 较高 | 最高 |
| 典型用途 | 日期、时间等通用值 | 环境标识、配置常量 | 任务特有的临时变量 | 补数、回刷时的临时覆盖 |
| 可修改性 | 不可修改，只能引用 | 可修改（影响后续所有实例） | 可修改（仅影响当前节点） | 一次性（当次实例有效） |

### 最佳实践

1. **命名规范统一**：自定义参数全部采用`UPPER_CASE`下划线命名法（如`ROW_COUNT`、`OUTPUT_PATH`），团队内部形成书面约定，避免大小写引发的幽灵Bug。
2. **启动参数做补数**：需要重跑历史数据时，不要修改工作流定义中的全局参数，而是用启动参数临时覆盖。跑完即生效、随即恢复，不污染定义。
3. **端到端测试参数链**：任何新增的参数传递链路，务必完整跑一遍并把每个节点的"自定义参数"标签页截图留档——上游OUT和下游IN的值要一一对应上。
4. **避免深层嵌套传参**：参数链控制在3层以内（A→B→C）。超过3层的依赖关系建议用文件系统或数据库表做中间存储，减少参数丢失风险。

### 常见反模式

1. **将敏感信息作为参数传递**：数据库密码、API Key等写在setVar里会明文出现在日志中。敏感信息应走DS资源中心或密钥管理服务。
2. **在一个节点输出20+个OUT参数**：参数越多，调试越困难。建议将批量数据打包为JSON文件，只传递文件路径。
3. **用启动参数替代全局参数定义**：几乎所有参数都空着，全靠启动时填——这种做法让工作流失去了"一键上线"的能力。默认值该写在定义里，启动参数只做异常场景的临时覆盖。

### DS参数体系 vs Airflow XCom

| 对比维度 | DS自定义参数 | Airflow XCom |
|----------|------------|-------------|
| 传递机制 | setVar输出→Worker解析→MySQL存储 | task_instance.xcom_push/pull |
| 数据类型 | 全部为字符串 | 支持JSON序列化的任意类型 |
| 跨DAG传递 | 不支持（SubProcess需显式桥接） | 支持（指定dag_id即可拉取） |
| 调试难度 | UI可直接查看 | 需进Airflow元数据库或用CLI |
| 存储上限 | 受MySQL VARCHAR限制 | 受元数据库列大小限制 |

### 思考题

1. 假设一个工作流有三个阶段共10个节点，其中节点5输出了一个参数`FILE_PATH`，节点9需要读取这个参数。但节点9的实际执行日志显示`FILE_PATH`为空，而节点5的日志中明确输出了`${setVar=FILE_PATH=/data/v3/result.parquet}`。请列出至少三种可能导致参数传递失败的原因，并给出排查顺序。

2. 如果业务需要跨工作流传递参数（工作流A的产出作为工作流B的输入），目前DS的自定义参数体系不支持直接跨DAG传递。你会设计怎样的替代方案？请综合考虑时效性、可靠性和实现复杂度，对比"文件落地+轮询"、"数据库状态表"、"DS的Dependent节点+全局参数"三种方案的优劣。
