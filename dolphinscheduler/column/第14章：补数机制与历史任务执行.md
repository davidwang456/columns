# 第14章：补数机制与历史任务执行

> **定位**：批量生成历史时间范围内的调度实例，解决数据修复、新表初始化、模型回训等批量历史执行需求。
> **核心内容**：补数原理、串行补数与并行补数模式的选择与权衡、补数配置参数、补数与依赖节点的交互机制、大规模补数分批策略。
> **实战目标**：为一套ETL工作流执行180天历史补数，掌握串行/并行两种模式的操作与监控，处理补数中的部分失败、依赖联动及大规模场景的拆分策略。

---

## 1. 项目背景

某金融科技公司计划上线一套全新的风险评分模型。模型团队提出需求：该模型需要180天（6个月）的历史订单数据作为训练集，以覆盖淡季、旺季、促销季的完整业务周期，确保模型对不同市场环境都有足够的泛化能力。好消息是，数据ETL工作流早已在DolphinScheduler中定义好——`order_etl_daily`每天凌晨从业务库抽取前一天的订单数据，清洗、聚合后写入评分模型所需的特征宽表，已经稳定运行了半年。坏消息是，这个工作流的每个实例只处理**一个调度日**的数据——`biz_date`参数决定了它抽取的是哪一天的订单。要凑齐180天的训练数据，就必须让这个工作流跑180次，每次传入不同的历史日期。

运维工程师小周起初手动逐天触发：打开控制台，选中`order_etl_daily`，点击"运行"，填入`2024-01-01`；跑完后（约8分钟），再填入`2024-01-02`……第一天点了30次，第二天又点了30次。中途有3天因数据源问题失败——他得逐一排查、修数据、找失败日期重新触发。整个流程断断续续搞了一周，比模型训练本身还耗时。

DolphinScheduler的**补数（Backfill）**机制正是为此而生——你只需指定一个历史日期范围，DS会自动为每一天生成独立的工作流实例，并按选定的执行模式（串行或并行）逐个或批量调度。这些实例与正常定时产生的实例完全一致——有独立的`schedule_time`、状态流转和日志——唯一的区别是它们源于一次性批量生成而非cron触发。

## 2. 项目设计——剧本式交锋对话

周五下午，小胖从楼下便利店买了一袋泡芙回来，看到小白正对着屏幕上密密麻麻的工作流实例列表发愁。

**小胖**（嘴里塞着泡芙，含糊不清地）：
"这不就是跑180遍嘛！写个for循环，Shell里`for i in $(seq 1 180); do curl -X POST 'http://ds-api/api/run' -d 'date=xxx'; done`，一行搞定！"

**小白**（放下咖啡杯，皱着眉）：
"180个实例一下子全发出去，Worker线程池扛得住吗？假如中间有5个失败了怎么办——全部重跑还是只挑那5个？还有，下游有个模型训练工作流用Dependent节点依赖ETL，依赖节点怎么知道你要查的是'1月15日的实例'而不是'今天最新那次'？补数跟定时调度会不会打架？"

**小胖**（泡芙差点噎住，灌了口水）：
"呃……依赖、并发、失败重试……确实不是一行for能兜住的。"

**大师**（从身后走过来，端着红茶）：
"'批量触发'确实是补数的核心，但补数不是简单的批量API调用，而是DS内**实例生成 + 模式调度 + 状态感知**的工程机制。打个比方——补数就像快递处理积压包裹。串行补数好比一条分拣线，一个接一个过，前一个签收了后一个才投递，稳但慢。并行补数好比多个快递员同时出发，每人一个包裹，快但容易爆单——一次发180个，配送站直接瘫痪。"

**技术映射**：串行补数模式中，Master为每个日期依次生成实例，前一个完成后才创建下一个调度命令；并行补数则一次性生成所有实例并提交到任务队列，Worker线程池并发执行。

**小白**（快速在笔记本上画了一张图）：
"那依赖节点的问题呢？假设B工作流依赖A的'昨天'，现在我在补数——B的实例调度日期是1月8日，它依赖A'昨天'也就是1月7日。但如果1月7日既有正常的定时调度实例，又有后来补数的实例，数据库里就有两条`schedule_time=2024-01-07`的记录，依赖节点取哪一条？"

**大师**：
"好问题。Dependent节点根据`schedule_time`匹配目标实例，而补数生成的实例同样带有正确且独立的`schedule_time`——所以依赖节点能精准地找到对应历史日期的实例。当同一日期存在多条实例时（比如原定实例失败后补数覆盖），依赖节点默认取**最新创建的那一条**的状态。这意味着你可以先大规模补数上游工作流，再补数下游依赖工作流，整条链路的历史依赖关系就能完整恢复。但要注意——补数期间如果正常定时调度也在跑同一个工作流，同一天可能会意外生成第三条实例，这就会产生竞争窗口。"

**技术映射**：补数实例与定时调度实例在底层共用`t_ds_process_instance`表，区分方式是`command_type`字段（START_PROCESS表示补数启动，REPEAT_RUNNING表示定时调度）以及`schedule_time`字段。依赖匹配的核心SQL是`SELECT state FROM t_ds_process_instance WHERE schedule_time=? ORDER BY create_time DESC LIMIT 1`。

**小胖**（挠挠头）：
"那我什么时候用串行，什么时候用并行？有没有个简单规则？"

**大师**：
"三条判断标准。第一，下游扛不扛得住——如果你的ETL每跑一天要往里插200万行数据，并行30天意味着6000万行同时灌入，数据库写锁直接打满，这种情况必须用串行或小批量并行。第二，实例之间有没有隐含的顺序依赖——如果2月的数据处理逻辑依赖1月的聚合结果，串行是唯一选择。第三，资源池够不够——并行模式下Worker线程瞬间被占满，其他租户的工作流可能排队等到天荒地老。一句话口诀：小规模无依赖用并行抢时间，大规模或有依赖用串行保稳定。"

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x 集群已部署，至少一台Master和一台Worker正常运行
- 浏览器打开控制台（`http://<master-ip>:12345/dolphinscheduler`）
- 管理员账号 `admin / dolphinscheduler123`
- 创建一个项目，命名为"补数机制实战"

### 步骤1：创建基础ETL工作流"daily_order_etl"

**目标**：搭建一个接收`biz_date`参数、按日期处理数据的基础ETL工作流。

进入项目"补数机制实战"，点击【工作流定义】→【创建工作流】，拖入一个Shell任务节点，命名为`etl_job`。脚本内容如下：

```bash
#!/bin/bash
echo "=========================================="
echo "  订单ETL任务开始执行"
echo "  业务日期: ${biz_date}"
echo "=========================================="

# 模拟ETL处理：拉取数据、清洗、聚合
sleep 5
echo "[INFO] 数据抽取完成 - 源表: ods_orders"
echo "[INFO] 数据清洗完成 - 脏数据率: 0.3%"
echo "[INFO] 数据聚合完成 - 写入目标表: dwd_orders_daily"

echo "=========================================="
echo "  ETL任务执行成功: ${biz_date}"
echo "=========================================="
exit 0
```

保存工作流，命名为`daily_order_etl`。在「工作流定义」页面，点击【定时管理】旁的下拉箭头，选择【运行】。在弹出对话框中：
- 调度日期：选择今天（例如`2024-07-15`）
- 自定义参数：点击添加，参数名`biz_date`，值填入`2024-07-14`（模拟处理昨天数据）

点击确认，观察实例运行完成，验证基础工作流可正常执行。

### 步骤2：执行串行补数

**目标**：使用串行模式，为过去7天生成历史实例。

在`daily_order_etl`的「工作流定义」页面，点击【运行】按钮旁边的下拉箭头，选择【补数】。在弹出的补数对话框中配置：

| 配置项       | 值                  |
|-------------|---------------------|
| 起始日期     | `2024-01-01`        |
| 结束日期     | `2024-01-07`        |
| 补数方式     | **串行补数**         |
| 并行度       | （串行模式忽略）      |

点击确认后，系统会依次为`2024-01-01`到`2024-01-07`生成7个工作流实例。切换到「工作流实例」页面，你会看到实例列表逐条出现并逐个执行。串行模式下，第N个实例开始执行时，前N-1个实例已经全部结束。总耗时约为`7 × 5秒 = 35秒`。

> **观察要点**：串行模式下，同一时刻只有一个实例处于"运行中"状态。如果某个实例失败，后续实例仍会继续执行——串行只保证"依次执行"，不保证"前一个成功才执行下一个"。

### 步骤3：执行并行补数

**目标**：对比串行模式，体验并行补数的速度差异。

再次点击【补数】，配置相同日期范围，但将"补数方式"改为**并行补数**：

| 配置项       | 值                  |
|-------------|---------------------|
| 起始日期     | `2024-01-01`        |
| 结束日期     | `2024-01-07`        |
| 补数方式     | **并行补数**         |
| 并行度       | 7（或留空使用默认）   |

点击确认后，7个实例同时进入"运行中"状态。由于每个Shell任务仅sleep 5秒，且Worker有足够线程，这7个实例几乎同时完成——总耗时约**5秒**，相比串行模式提速7倍。

> **注意**：并行度是指同时运行的实例数量上限。如果设为3，则7个实例中最多同时有3个处于运行状态，其余排队等待。这个参数是控制资源消耗的关键开关。

### 步骤4：处理补数中的部分失败

**目标**：模拟部分日期ETL失败，学习如何在补数后单独重跑失败实例。

修改`etl_job`的Shell脚本，注入随机失败逻辑：

```bash
#!/bin/bash
echo "业务日期: ${biz_date}"

# 模拟特定日期数据缺失
if [ "${biz_date}" = "2024-01-05" ]; then
    echo "[FATAL] 2024-01-05源数据缺失，分区不存在！"
    exit 1
fi

sleep 3
echo "ETL完成: ${biz_date}"
exit 0
```

再次执行并行补数（日期范围`2024-01-01`到`2024-01-07`）。结果：六个实例成功（绿色），`2024-01-05`对应实例失败（红色）。

处理方式：进入「工作流实例」页面，找到`schedule_time`为`2024-01-05`的失败实例，点击其操作列的【重跑】按钮，在弹出的对话框中选择"**恢复失败**"模式——这会仅重新执行该实例内失败状态的节点，而非整个工作流。若失败原因是临时性的（如网络抖动），也可直接选择"重跑"模式完全重新执行该实例。

### 步骤5：补数与Dependent节点联动

**目标**：验证补数上下游工作流的依赖关系是否正确联动。

创建第二个工作流，命名为`daily_order_report`。拖入一个Dependent节点（命名为`wait_etl`）和一个Shell任务节点（命名为`gen_report`）。

Dependent节点配置：
- 项目："补数机制实战"
- 工作流：`daily_order_etl`
- 依赖周期：`昨天`
- 期望状态：SUCCESS

Shell任务节点脚本：

```bash
#!/bin/bash
echo "===== 日报生成 ====="
echo "依赖日期(昨天ETL): ${biz_date}"
echo "报告已生成，数据来源为 daily_order_etl 的昨日实例"
exit 0
```

连线：`wait_etl` → `gen_report`，上线工作流。

现在对`daily_order_report`执行补数：起始日期`2024-01-02`，结束日期`2024-01-08`，串行补数。观察：对于调度日期为`2024-01-02`的report实例，其Dependent节点会检查`daily_order_etl`在`2024-01-01`（昨天）的实例状态——由于步骤2中已经补数跑过，该实例为SUCCESS，Dependent节点判定通过。依此类推，7个report实例的依赖全部满足，依次生成报告。

> **关键认知**：补数操作是对调度日期的批量扩展，而非改变工作流本身的依赖逻辑。Dependent节点的"昨天"始终相对于当前实例的`schedule_time`来计算，无论该实例是定时产生还是补数产生。

### 步骤6：大规模补数分批策略

**目标**：学习180天历史补数的分批提交与管理方法。

直接对180天执行并行补数是危险的——180个实例同时涌入Worker线程池，可能导致：
- Worker线程耗尽，正常调度的其他工作流饥饿
- 数据库连接池打满，元数据库响应变慢甚至超时
- 如果每天ETL操作大量数据库写入，180天并行写入可能直接压垮目标库

推荐的分批策略：

| 策略             | 每批天数 | 模式 | 总批次数 | 适用场景                           |
|-----------------|---------|------|---------|-----------------------------------|
| 保守分批         | 7天     | 串行 | 26批    | 生产环境、下游数据库负载高           |
| 适中分批         | 15天    | 串行 | 12批    | 一般生产环境                        |
| 激进分批         | 30天    | 串行 | 6批     | 测试环境、目标库性能充裕             |
| 渐进式验证       | 3天→7天→30天 | 串行 | 验证后放大 | 不确定数据质量，需逐批验证          |

具体操作：
1. 首批补数3天（验证数据完整性和实例成功率）
2. 确认无异常后，第二批补数7天
3. 继续观察，第三批补数30天
4. 此后每批30天，直至覆盖全部180天
5. 每批结束后，检查失败实例列表，修复数据后单独重跑

> **运维建议**：大规模补数不要在业务高峰期执行。选择凌晨或周末窗口，提前通知DBA和下游团队。每次提交补数后在「工作流实例」页面打开该工作流的实例列表，设置筛选条件为该批次的日期范围，集中监控执行进度。

### 步骤7：补数与数据质量门禁

**目标**：在ETL任务前加入数据存在性校验，避免补数时因源数据缺失产生大量失败。

在`daily_order_etl`工作流中，在`etl_job`节点前面新增一个Python任务节点，命名为`data_quality_check`：

```python
#!/usr/bin/env python3
import sys
import os

biz_date = os.environ.get('biz_date', 'unknown')
print(f"[质检] 检查业务日期 {biz_date} 的源数据可用性...")

# 实际场景中这里连接数据库查询源表分区是否已生成
# 此处用模拟逻辑：假设 2024-03-15 源数据缺失
MISSING_DATES = ["2024-03-15"]

if biz_date in MISSING_DATES:
    print(f"[质检] 业务日期 {biz_date} 源数据缺失，跳过此日期")
    # 策略1：退出码0，下游依赖正常执行但结果为空（适合"尽力补数"场景）
    # 策略2：退出码1，标记失败，后续人工介入（适合"数据必须完整"场景）
    sys.exit(0)
else:
    print(f"[质检] 业务日期 {biz_date} 数据就绪，放行")
    sys.exit(0)
```

连线：`data_quality_check` → `etl_job`。对质检节点设置下游条件为"成功"才执行。这样当源数据缺失时，质检通过（exit 0）但`etl_job`不跑——相当于**优雅跳过**无数据的日期。

### 步骤8：补数常见陷阱速查

| 陷阱 | 现象 | 根因与解决 |
|------|------|-----------|
| 起始/结束日期反填 | 补数只生成了一天或根本没生成 | 起始日期必须早于结束日期。DS不会自动纠正顺序，需人工核对 |
| 并行补数打爆连接池 | Worker大量报`Connection pool exhausted` | 减少并行度参数。对于数据库操作密集的ETL，建议并行度不超过Worker数量的2倍 |
| 补数与定时调度竞态 | 同一日期出现两条实例，Dependent节点检查异常 | 补数前暂停目标工作流的定时调度；补数完成后恢复定时，避免同一日期两条实例共存 |
| 依赖节点在补数期间超时 | 上游ETL实例已成功，但依赖节点超时失败 | 大规模补数时依赖节点的轮询间隔可能需要增大。每批补数结束后，再对依赖方补数，而非同时进行 |
| 365+天补数导致Master OOM | Master进程内存飙升后崩溃 | DS在生成超大量实例时会将实例元数据一次性加载到内存。较大范围（如365天以上）必须拆分批次执行 |
| 补数日期里混入了未来日期 | 部分实例报"参数越界"或直接失败 | 补数范围的上限不应超过当前日期。如果业务需要"预跑未来几天的空分区"，应使用正常定时调度而非补数 |

## 4. 项目总结

### 串行补数 vs 并行补数

| 维度 | 串行补数 | 并行补数 |
|------|---------|---------|
| 执行速度 | 慢（实例数 × 单实例耗时） | 快（接近单实例耗时 × 1/并行度） |
| 资源消耗 | 低（同一时刻仅1个实例运行） | 高（N个实例同时占Worker线程） |
| 失败影响面 | 单实例失败不阻塞后续 | 多个实例可能同时失败 |
| 适用规模 | 任意规模，天然适合大批量 | 小范围（<30天），需配合并行度限制 |
| 典型场景 | 180天模型回训、年度数据审计 | 单周数据修复、单月报表重刷 |

### 补数最佳实践

1. **分批优先**：超过30天的补数务必拆分批次，每批完成后验证再继续，避免一次性提交后大面积失败难以定位。
2. **监控到位**：每批补数提交后，在实例列表页按日期范围筛选，盯住失败数。结合DS的整体监控大盘关注Worker线程使用率。
3. **失败闭环**：补数中的失败实例不要直接忽略——记录失败日期清单，排查根因后统一重跑，确保最终每个日期的目标实例都达到期望状态。
4. **依赖对齐**：如果工作流存在上下游依赖关系，补数顺序必须从上向下逐级推进——先补数所有上游，确认全部成功后，再补数下游。反向操作会导致下游依赖大面积超时失败。
5. **定时暂停**：在对已上线的工作流执行补数前，建议暂停其定时调度，避免补数与正常调度产生竞争。

### 常见踩坑经验

**案例1——并行补数导致数据库死锁**：某团队对`user_behavior_etl`执行30天并行补数，30个实例同时向MySQL的`dwd_behavior`表执行`INSERT INTO ... SELECT`。第3分钟数据库出现死锁风暴，锁等待超时导致25个实例失败。根因是该表缺少针对日期分区键的索引，并行写入时大面积行锁冲突。解决：先对该目标表添加日期分区索引，再将补数改为串行。

**案例2——依赖节点在补数窗口内超时**：下游工作流`weekly_report`的Dependent节点依赖上游7天ETL实例，超时时间设置为2小时。但在补数场景下，7天ETL串行补数总耗时超过2小时，导致Dependent节点在轮询中途超时退出。解决：将补数拆分为"先补ETL"和"后补report"两阶段，避免依赖轮询与上游补数重叠。

**案例3——补数范围误解**：运维以为补数结束日期是"不包含"的（左闭右开），填写了`2024-01-01`到`2024-01-07`，预期生成6个实例。实际DS的日期范围是**左右均包含**，实际生成了7个。解决：补数对话框中明确标注"起止日期均包含"，提交前核对实例数量预期。

### 反思题

1. 假设你的`daily_order_etl`工作流每天凌晨2点运行，处理昨天的数据（`biz_date = schedule_time - 1`）。现在你需要补数6个月（180天）的历史数据。补数时`biz_date`参数应该如何传递？如果在补数过程中，恰好跨过了凌晨2点，正常定时调度触发了一次新的实例——这会导致什么问题？如何防范？

2. 你的工作流中有三个SubProcess节点（子工作流），每个子工作流内部又有Dependent节点依赖其他工作流。当你对顶层工作流执行补数时，子工作流和依赖检查都会被递归触发。这种级联补数场景下，如果中间某个子工作流实例失败，整个补数链路会怎样？你应如何设计补数后的失败恢复流程？提示：思考补数的"全或无"语义与"尽力补"语义的权衡。
