# 第9章：条件分支（Conditions）节点与动态路由

> **定位**：告别手工分流，让工作流根据运行时数据自动选择执行路径。
> **核心内容**：Conditions节点的运算符体系、成功/失败分支路由、多条件AND/OR组合、参数类型处理、与SWITCH任务节点的对比。
> **实战目标**：搭建一条根据日订单量自动分流的销售分析工作流，掌握条件分支在生产环境中的正确用法。

---

## 1. 项目背景

某电商公司的数据团队每天要跑一套销售分析工作流：凌晨ETL产出前一天的订单数据后，需要根据订单量级决定后续分析路径——如果昨日订单量超过10万单，说明有促销活动或大促，必须跑完整的分析套件（Spark ML销量预测模型 + 多维可视化报表 + 异常品类预警）；如果订单量不足10万单，只需跑一个简版摘要（计算总GMV和Top 10单品），节省计算资源。

这个分流逻辑目前靠人工操作。数据分析师小王每天早晨6点爬起来，先登录数据库查订单量，然后根据数字手动启动两套工作流中的一个。问题接踵而至：小王偶尔闹钟没响，全部门等到9点才发现报表还没跑；查数时SQL写错了条件，拿到了前天的数据，导致跑错了分支；更危险的是"双十一"当天——订单量是平时的15倍，小王习惯性地启动了"简版"流程，Spark集群资源被大量简化任务挤占，真正的预测模型反而没跑成，导致运营错过了黄金调货窗口。

这本质上是"工作流缺少条件判断能力"的问题。当DAG中出现"如果X则跑A，否则跑B"的逻辑时，用手工判断代替自动路由，不仅低效，而且引入了人为错误的风险。Apache DolphinScheduler的**Conditions（条件分支）节点**正是为此而生——它像一个内置的if-else路由器，在运行时根据上游产出的参数值，自动将工作流引导到不同的下游分支。

## 2. 项目设计——剧本式交锋对话

周三午休后，小胖叼着棒棒糖晃到大师工位前，看见小白正对着屏幕皱眉。

**小胖**（棒棒糖从嘴里抽出来，指着屏幕上的DAG图）：

> "这不就是if-else嘛！上游输出个订单量数字，下游根据数字走不同分支——Python里两行代码的事：`if count > 100000: spark.run() else: simple.run()`。DS里加个节点搞这么复杂干嘛？"

**小白**（抬起头，笔在手里转了两圈）：

> "你这种代码if-else是写在Python脚本里的，跟调度器的条件分支是两回事。我好奇的是——如果上游SQL查询失败，输出的ORDER_COUNT是null或者根本不存在，条件节点怎么处理？还有，ORDER_COUNT在数据库里是INT类型，但上游Shell脚本echo出来的都是字符串——DS拿字符串'"150000"'和数字100000做大于号比较，能算对吗？另外，能不能同时判断好几个条件，比如'订单量>10万 AND 错误率<0.05'？AND和OR能混在一起用吗？会不会有优先级歧义？"

**小胖**（棒棒糖差点掉地上，手忙脚乱接住）：

> "呃……null和字符串比较……这题超纲了。"

**大师**（从旁边会议室走出来，手里端着咖啡）：

> "小白问的每一个问题都打在要害上。我打个比方——条件节点就像一个**分岔路口**，路口上有块路标牌。上游任务把路标值写上牌子，条件节点就是那个看牌指路的人。路标说'前方200公里'，条件节点判断'大于100公里就走高速，小于等于走国道'——**成功分支**就是走高速那条路，**失败分支**是走国道那条。如果路标牌被风吹走了（null），看牌人会怎么做？DolphinScheduler里，参数值缺失时条件直接判false，走失败分支。如果路标写的是'两百公里'而不是'200'——字符串和数字直接比较在DS里会做自动类型转换，但生产环境你最好别赌这个，上游严格保证输出的类型正确。多条件AND/OR的混合，DS里不支持嵌套括号，但你可以在同一节点里配多条条件规则，逐条判断，规则之间选择AND或OR关系——全部AND为真走成功，OR中任一为真走成功。"

**技术映射**：Conditions节点评估在Worker端执行。上游任务的OUT参数先由Master收集，派发条件节点任务时注入到Worker环境。Worker解析条件表达式、执行运算符比较，返回成功/失败信号给Master，Master再决定激活哪个下游分支。这与Airflow的BranchPythonOperator不同——Airflow是在Python代码中返回task_id来决定路由，而DS是在UI界面上配置条件表达式，零代码。

**小白**（在笔记本上快速记了几行）：

> "那这个条件判断是Master做的还是Worker做的？每个条件节点都要占用一个Worker线程吗？还有——如果我的分支特别多，比如按订单量分10个区间各自走不同的任务链，条件节点是不是就不够用了？"

**大师**：

> "判断逻辑在Worker上执行——Master负责DAG调度和状态机维护，Worker负责具体任务执行，条件表达式评估也是一种'任务'。不过条件节点的开销极小，几乎不占计算资源，它是一个轻量级路由决策，运行耗时通常毫秒级。至于你说的10个分支场景，Condition节点只有'成功'和'失败'两条线，十路分岔确实力不从心。这时你应该用DolphinScheduler 3.2+新增的**SWITCH任务节点**，它支持多路分支，类似于编程语言中的switch-case语句。但90%的生产场景——比如数据质量门禁、A/B测试分流、错误类型差异化退避——二路分支已经够用，且配置更简单直观。"

**技术映射**：条件节点是二路分支路由器（if-else），SWITCH节点是多路分支路由器（switch-case）。资源消耗：条件判断本身毫秒级，主要延迟来自Worker的任务调度开销（通常<1秒）。

**小胖**（终于找到机会插嘴）：

> "那我能把一个条件节点的失败分支，再连到另一个条件节点上吗？这样能不能实现多级if-else？"

**大师**：

> "完全正确。你可以串联多个条件节点——第一个判断订单量，成功走大促分析；失败分支再连第二个条件节点，判断'是否周五'，是周五走周报汇总，不是周五走常规简报。这就是**级联条件路由**，灵活度很高。但要避免一个坑——不要设计出让条件分支产生环形依赖的拓扑，比如条件A的成功分支最终又绕回到了条件A的上游。DS保存时会做环检测，但逻辑上的'死循环'有时候不一定表现为图论环，要自己把关。"

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x 已部署并正常运行
- 至少一台Worker节点已注册（Standalone模式下master和worker共用同一进程）
- Worker节点上具备Bash执行环境，本例仅需Shell任务
- 获取MySQL连接信息，用于模拟订单量查询（也可用`echo`写死模拟数据）

### Step 1：创建上游Shell任务"check_order_volume"

**目标**：查询昨日订单总量，并通过setVar语法输出为OUT参数供条件节点读取。

在工作流画布中拖入一个Shell任务节点，命名为"check_order_volume"。「节点名称」填`check_order_volume`，任务类型选`SHELL`，脚本内容如下：

```bash
#!/bin/bash
# 查询数据库中昨日订单总量
YESTERDAY=$(date -d "yesterday" +%Y-%m-%d)

# ★ 生产环境使用实际数据库查询
ORDER_COUNT=$(mysql -h 10.0.0.12 -u datauser -pData@2024 bigdata_db \
    -e "SELECT COUNT(*) FROM dwd_orders WHERE order_date='$YESTERDAY'" -sN 2>/dev/null)

# ★ 如果数据库查询失败，给一个安全默认值，避免下游条件节点拿到空值
if [ -z "$ORDER_COUNT" ] || [ "$ORDER_COUNT" -lt 0 ] 2>/dev/null; then
    echo "WARN: MySQL查询失败，使用默认值0"
    ORDER_COUNT=0
fi

echo "昨日日期: $YESTERDAY"
echo "昨日订单量: $ORDER_COUNT"

# 通过setVar语法将订单量和日期输出为OUT参数
echo "${setVar=ORDER_COUNT=$ORDER_COUNT}"
echo "${setVar=ORDER_DATE=$YESTERDAY}"
```

保存任务后，在「自定义参数」标签页添加两个OUT参数：

| 参数名        | 方向 | 值   |
|---------------|------|------|
| ORDER_COUNT   | OUT  | 留空 |
| ORDER_DATE    | OUT  | 留空 |

> **关键细节**：`${setVar=KEY=VALUE}`必须原样输出到标准输出，中间等号不空格。参数值允许为空，但上游务必做空值兜底——第8行的空值检查保证了ORDER_COUNT至少为0。

---

### Step 2：创建Conditions节点"decide_pipeline"

**目标**：根据上游ORDER_COUNT值，将工作流路由到"大促报告"分支或"常规报告"分支。

在画布中拖入一个Conditions节点，命名为"decide_pipeline"。双击打开配置面板：

- **上游任务**：在下拉框中将`check_order_volume`设置为上游
- **条件列表**：点击「添加条件」，配置如下：

| 上游参数        | 运算符 | 比较值   |
|-----------------|--------|----------|
| ${ORDER_COUNT} | >      | 100000   |

条件关系选择默认的"AND"（单条件时AND/OR效果相同）。

- **成功分支**：在"成功分支"标签旁填入`大促报告`——当ORDER_COUNT > 100000时走此分支。下游任务需要把节点名命名为完全一致的`大促报告`才能正确连线。
- **失败分支**：在"失败分支"标签旁填入`常规报告`——当ORDER_COUNT ≤ 100000或参数缺失时走此分支。

保存节点。

> **关键设计决策**：分支标签（"大促报告"/"常规报告"）是条件节点与下游任务建立的**连接标识**。下游任务的节点名称必须与标签**完全一致**（包括中文字符，严格匹配），否则条件节点找不到对应下游，工作流会报"下游节点不存在"的异常。

---

### Step 3：创建成功分支下游任务"full_analysis"

**目标**：订单量超过10万时运行完整分析套件。

在工作流画布中依次创建以下节点并连线（从`decide_pipeline`成功分支拖线到第一个节点）：

**3a. Spark任务"sales_prediction"**（类型SPARK）：
```
程序类型：JAVA
主程序包：resource://spark_jobs/sales_ml_predict_v3.jar
主类名：com.bigdata.SalesPrediction
主参数：${ORDER_DATE}
```
> 如果环境中没有Spark集群，此节点可替换为Shell任务做模拟：
> ```bash
> echo "[大促模式] 启动Spark ML销量预测..."
> echo "分析日期: ${ORDER_DATE}，订单量: ${ORDER_COUNT}"
> sleep 10
> echo "预测完成：明日预计销量 +35%"
> echo "${setVar=PREDICT_GROWTH=35}"
> ```

**3b. Shell任务"anomaly_alert"**：
```bash
#!/bin/bash
echo "===== [大促模式] 异常品类预警 ====="
echo "扫描各品类库存周转率..."
echo "手机品类安全库存不足！已发送钉钉告警"
```

将两个节点串行连接：`decide_pipeline`(成功) → `sales_prediction` → `anomaly_alert`

### Step 4：创建失败分支下游任务"simple_summary"

**目标**：订单量不足10万时仅生成简要汇总。

在工作流画布中创建一个Python任务节点，命名为"simple_summary"。从`decide_pipeline`的失败分支拖线到该节点。

```python
import os
import sys

order_count = os.environ.get('ORDER_COUNT', '0')
order_date = os.environ.get('ORDER_DATE', 'unknown')

print(f"[常规报告模式] 分析日期: {order_date}")
print(f"昨日订单量: {order_count}")
print("由于订单量低于10万，仅生成简要汇总")

# 模拟汇总逻辑
try:
    count_int = int(order_count)
    if count_int > 0:
        print(f"日均客单价估算: {count_int * 150} 元")
except ValueError:
    print("WARN: ORDER_COUNT为非数值，跳过汇总")

print("常规报告生成完毕")
sys.exit(0)
```

> **注意**：Python任务通过`os.environ`获取上游参数。上游`check_order_volume`是Shell任务，ORDER_COUNT通过setVar输出后，Master会将值注入到下游Python任务的环境变量中——前提是`simple_summary`的「自定义参数」里配置了ORDER_COUNT的IN参数。

在"simple_summary"的「自定义参数」中配置：

| 参数名      | 方向 | 值   |
|-------------|------|------|
| ORDER_COUNT | IN   | 留空 |
| ORDER_DATE  | IN   | 留空 |

### Step 5：创建末端汇聚任务"send_report"

**目标**：无论是"大促报告"还是"常规报告"分支，最终都需要将结果发送给管理层。这是一个典型的"分支后汇聚"模式。

创建Shell任务节点"send_report"，从`sales_prediction`（成功分支的最后一个节点）和`simple_summary`（失败分支的最后一个节点）都拖线连接到它。

```bash
#!/bin/bash
echo "===== 发送日报邮件 ====="
echo "日期: ${ORDER_DATE}"
echo "订单量: ${ORDER_COUNT}"

# 模拟邮件发送
echo "收件人: management@bigdata.com"
echo "日报已发送，工作流执行完毕"
```

右键打开`sales_prediction`到`send_report`的连线，设置依赖条件为"成功"；同样设置`simple_summary`到`send_report`的连线。这样**只要连接线的上游成功，send_report就会被触发**，不要求两条线同时满足。

> **避免的坑**：如果你把send_report设置为同时依赖两条上游线（AND依赖），那么一次运行只会走一条分支，另一条分支不会触发，send_report将永远不会执行。正确做法是用**OR依赖**——任意一个上游成功后触发。在连线配置中将依赖策略设为"任一上游成功即执行"（DS中连线默认就是成功即触发）。

### Step 6：完整DAG拓扑与连线

最终工作流拓扑如下：

```
                    [check_order_volume]
                            │
                            ▼
                    [decide_pipeline]
                     成功↙    失败↘
              [大促报告]         [常规报告]
                  │                   │
                  ▼                   ▼
         [sales_prediction]    [simple_summary]
                  │                   │
                  ▼                   │
         [anomaly_alert]              │
                  │                   │
                  └───────┬───────────┘
                          ▼
                   [send_report]
```

### Step 8：进阶——多条件AND组合

**场景**：只有同时满足"订单量 > 10万"**且**"数据错误率 < 5%"时，才走大促分析分支，否则走常规分支（哪怕订单量高但数据质量差也不能冒险跑预测模型）。

修改`check_order_volume`脚本，额外输出一个ERROR_RATE参数：
```bash
# 在脚本末尾追加
ERROR_RATE=$(mysql -h 10.0.0.12 -u datauser -pData@2024 bigdata_db \
    -e "SELECT ROUND(SUM(CASE WHEN amount < 0 THEN 1 ELSE 0 END)*100.0/COUNT(*),2) FROM dwd_orders WHERE order_date='$YESTERDAY'" -sN)
echo "数据错误率: ${ERROR_RATE}%"
echo "${setVar=ERROR_RATE=$ERROR_RATE}"
```

在`decide_pipeline`的条件列表中增加第二条：
| 上游参数        | 运算符 | 比较值 |
|-----------------|--------|--------|
| ${ORDER_COUNT} | >      | 100000 |
| ${ERROR_RATE}  | <      | 0.05   |

条件关系选**AND**。只有当两条规则都满足时走成功分支，任一条不满足走失败分支。

### Step 9：常见踩坑清单

| 陷阱                               | 现象                                                  | 根因与解决                                                                 |
|------------------------------------|-------------------------------------------------------|----------------------------------------------------------------------------|
| **参数类型不匹配**                  | 条件表达式`${ORDER_COUNT} > 100000`永远为false         | Shell脚本的echo输出默认是字符串。DS的Worker端会尝试将字符串转为数字与数值比较值比较，但隐式转换依赖Java的`Double.parseDouble()`，非纯数字字符串会抛异常并降级为false。**解决**：上游确保输出的数字不含空格/逗号/单位符 |
| **下游节点名与分支标签不一致**      | 条件节点执行后下游全部灰色，工作流提前终止            | "成功分支"字段填的是`大促报告`，但你给下游节点命名为`full_analysis`——这是匹配不上的。字段内容必须与下游节点的「节点名称」严格一致（含中英文、大小写、空格） |
| **条件节点的下游里又有条件节点**    | 级联条件时内层条件节点的参数取不到值                  | 级联的每个条件节点也需要配置参数的IN方向声明。如果第二个条件节点引用了`${ORDER_ERROR}`这个参数，你必须在第二个条件节点的上游任务中也声明OUT，或者在工作流级别定义全局参数 |
| **分支后汇聚任务永远不触发**        | 两条分支都成功但汇聚节点不跑                          | 汇聚节点默认需要**所有上游都成功**。而一次运行只走一条分支。解决：连线配置改为"任一上游成功即触发" |
| **条件节点形成隐式环**              | 保存时保存成功但运行时报"工作流进入死循环"            | 条件节点的失败分支重新连回了条件节点上游链路上的某个节点（比如连回check_order_volume），形成图中不可见的逻辑环 |
| **in运算符的边界行为**              | `${STATUS} IN (1,2,3)` 当STATUS为字符串"01"时判断为false | IN/NOT IN运算符做的是精确字符串匹配，不会做数值零填充。`"01" IN ("1","2","3")`返回false。**解决**：统一数据格式，上游用`printf "%d"`等强转类型 |

## 4. 项目总结

### Conditions节点 vs 其他分流方案

| 对比维度                | DS Conditions节点                               | Airflow BranchPythonOperator        | 手工判断启动两套工作流     |
|-------------------------|-------------------------------------------------|-------------------------------------|---------------------------|
| 配置方式                | UI界面选择上游参数+运算符+阈值，零代码           | 编写Python函数返回task_id           | SSH登录→查数→手动点击    |
| 分支数量                | 二路（成功/失败）                                | 理论无限（函数返回任意task_id）      | 手动选一套                |
| 条件组合                | AND/OR多条件，不支持嵌套括号                     | 任意Python逻辑                      | 人工脑判                  |
| 自动重跑友好度           | 高，重跑时条件重新评估                           | 高                                  | 低，需人工重新判断         |
| 适用角色                | 数据分析师、运维、业务人员                       | 开发工程师                          | —                         |
| 类型安全                | 宽松（隐式字符串→数值转换）                     | Python原生类型安全                  | 人脑判断                  |

### 适用场景

1. **数据质量门禁**——上游ETL输出的脏数据率 > 阈值时，走失败分支发送告警，阻断下游报表生成
2. **大促/常规自动分流**——根据订单量/流量指标自动切换轻量或全量分析链路，避免资源浪费
3. **A/B测试流水线**——根据实验分组参数，将不同实验组的用户数据路由到不同分析路径
4. **错误恢复差异化**——上游失败时根据错误码（通过自定义参数传递）走不同的退避或补偿分支
5. **跨环境同步控制**——开发环境产出数据量小走简化链路，生产环境走全链路校验

### 不适用场景

1. **分支数超过2个且无层级关系**——如按10个城市分别走不同子工作流，用SWITCH节点更合适
2. **需要持续监控并动态切换**——条件判断只在节点激活那一刻执行一次，不是持续监听。如需"执行到一半发现数据量超了自动切分支"，应在子任务内部实现

### 思考题

1. 如果`check_order_volume`脚本执行成功但ORDER_COUNT参数被意外输出为`"10,0000"`（中间有逗号），条件节点`${ORDER_COUNT} > 100000`会判断为true还是false？条件节点内部的字符串到数值转换逻辑是怎样处理的？如何在Shell脚本中防御此类参数格式问题？
