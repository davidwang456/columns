# 第36章：容错与故障恢复机制深度解析

> **关键词**：Master故障转移、Worker故障转移、任务重试、NEED_FAULT_TOLERANCE、恢复失败、暂停恢复、幂等性、ZooKeeper临时节点、Command队列、工作流恢复操作

---

## 一、项目背景

大麦（Damai）的数据平台在月末结算当天遭遇了一次灾难级故障。当天凌晨2点，数据中心发生了持续约2分钟的电压波动——Master-01所在机架瞬时断电后恢复，ZooKeeper会话超时（8秒未发心跳），Master-02通过ZK选举机制接过Leader角色。然而恢复结果却令人心惊：15个正在运行的工作流实例中，5个未能正常恢复，卡在了`NEED_FAULT_TOLERANCE`状态（需要故障容忍）；与此同时，Worker-03因网络闪断丢失了与ZK的连接，其上8个任务处于"失联"状态；最致命的是——财务月报工作流处于半完成状态，上游ETL已完成但下游汇聚任务未触发，导致月报产出推迟了整整3小时。CFO一早看到空荡荡的报表面板，电话直接打到了CTO。

事后复盘时，团队发现一个可怕的认知漏洞：所有人下意识以为"容错=重试"——Master挂了换新的、Worker挂了换别的、任务失败重跑——却从未深入追问：新Master如何知道旧Master上哪些任务已完成、哪些待重跑？Worker宕机后，正在执行的任务能否无缝迁移到另一台Worker？暂停、停止、重跑、恢复失败四种操作分别对应什么底层语义？"从失败节点开始"和"重跑整条工作流"的区别在哪里？

这些问题的答案，不在控制台的按钮上，而在DolphinScheduler容错架构的每一层设计里。容错不是"出问题了修一修"，而是一套从故障检测、状态持久化、任务重派、幂等提交到恢复策略的完整机制链。理解它，是确保生产环境工作流"永不停机"的前提。

## 二、项目设计——剧本式交锋对话

凌晨故障复盘会后，三人在会议室白板前坐下，白板上画着一个大大的"故障恢复"四字。

**小胖**（瘫在椅背上，打了个哈欠）："容错不就是重试嘛！失败了重跑一次，三次不行就报错。Master挂了换个新的，Worker挂了换个跑，多简单。我写的Shell脚本都是`while true; do retry; done`，三行搞定。DS搞那么多状态机、ZK监听、命令队列，是不是过度设计了？"

**小白**（翻开笔记本，上面密密麻麻记满了凌晨的故障细节）："那我问你五个问题——第一，Master故障恢复时，新Leader怎么知道哪些任务已经跑完了、哪些要重跑？第二，Worker宕机后，正在跑的任务是丢了还是能迁移到别的Worker？迁移过程中数据会不会重复执行？第三，暂停和停止有什么区别？暂停状态下Master也宕机了，恢复暂停还能继续吗？第四，重跑和恢复失败——一个是新建实例全部重来，一个是只重跑失败任务——这两个操作的底层数据库SQL到底差在哪？第五，你今天凌晨亲眼看到的NEED_FAULT_TOLERANCE状态，到底是容错的开端还是卡死的死循环？"

**大师**（从保温杯里倒出一杯茶，缓缓开口）："DS的容错就像航空公司的航班保障体系。机长（Master）突然生病了，备用机长（新Leader）不能坐进驾驶舱就说'起飞'——他必须先查飞行计划（DB中的ProcessInstance状态），搞清楚航班现在飞到哪个航路了（哪些Task已完成）、乘客名单对不对（参数上下文）、燃油还剩多少（任务执行进度），然后才能无缝接手操纵杆。如果某个空乘（Worker）请假了，签派中心要立刻找到替补空乘、重新分配职责，而不是让旅客干等。更重要的是，每一步操作——无论是'继续飞'、'返航'还是'换飞机'——都有应急预案。容错的本质不是'等故障发生后修一修'，而是'即使故障发生了，系统也能在15秒内自动恢复到正常状态'。"

**小胖**（拿起一根粉笔）："飞机比喻我懂了——但飞行员换班要30分钟，DS凭什么十几秒就搞定？"

**大师**："因为飞行计划不是写在小本子上，而是持久化在数据库里——每一行数据的UPDATE都是一个确定的状态转移。新Master不是'猜测'上一位干了什么，而是直接查询`t_ds_process_instance`和`t_ds_task_instance`：哪些任务的state是SUCCESS？哪些是RUNNING但host指向已经死掉的Worker？哪些从未提交过？有了这些数据，新Master在10秒内就可以重建整棵工作流DAG的状态树。这就像备用机长拿到了一份实时更新的电子飞行包，而不是一张手写的纸质清单。"

**小白**（若有所思）："那如果Master和Worker同时挂了呢？比如整个机架断电？"

**大师**："这就是DS容错架构最精妙的地方——它依赖的不是内存里的临时状态，而是数据库里的持久化状态。只要数据库还活着，即使Master和Worker全部重启，系统也能从DB中恢复出完整的工作流状态。具体来说有四层保障：第一，**状态持久化**——所有任务和工作流的状态变更都实时写入DB，不做内存缓存；第二，**ZK临时节点检测**——Master和Worker挂了，ZK的临时节点在8秒内消失，其他节点立刻感知；第三，**任务重派幂等性**——同一个TaskInstance ID不会在同一个时间段被派遣两次；第四，**Command命令队列**——调度指令不是内存消息队列，而是消费型数据库行，Master重启后可以继续消费未处理的Command。"

**小白**："那我们今天凌晨那5个卡在NEED_FAULT_TOLERANCE的工作流是怎么回事？为什么没有自动恢复？"

**大师**："这正是DS容错机制的一个边界条件——NEED_FAULT_TOLERANCE是一个中间态，表示'系统已经发现这个任务所在的Worker死了，需要重新分派'。正常情况下，FailoverCoordinator每10秒扫描一次，发现NEED_FAULT_TOLERANCE任务后，检查重试预算、重新提交。但大麦的情况特殊——那些任务的maxRetryTimes设为了0，即'不允许重试'。于是系统判断'重试次数已耗尽'，跳过了这些任务，转而去执行下游——但问题刚好出在：这个任务恰好是一条关键路径上的唯一节点，它一停，整个下游都被级联阻塞了。"

## 三、项目实战

### Step 1：为核心工作流配置重试策略

不同等级的工作流应配置差异化的容错策略：

```yaml
# 金融结算类工作流（高标准容错）
task:
  失败重试次数: 3          # mastRetryTimes = 3
  失败重试间隔: 60          # retryInterval = 60秒
workflow:
  失败策略: END             # 任何一个任务失败 → 停止所有运行中任务 → 工作流置为FAILURE

# 数据入仓类工作流（允许部分失败）
task:
  失败重试次数: 2
  失败重试间隔: 30
workflow:
  失败策略: CONTINUE         # 失败任务的并行兄弟继续运行，仅阻塞下游
```

### Step 3：追踪Master故障恢复核心代码路径

故障恢复的入口在`MasterFailoverCoordinator`中，它作为独立线程每10秒执行一次扫描：

```java
// MasterFailoverCoordinator.java —— 故障恢复核心逻辑
public void checkAndRecover() {
    // 第1步：查找"孤儿"流程实例 —— host字段不等于当前Master的RUNNING实例
    List<ProcessInstance> orphans = processInstanceDao
        .findRunningButNoCurrentMaster();

    logger.info("Found {} orphaned process instances to recover", orphans.size());

    for (ProcessInstance pi : orphans) {
        // 第2步：找出所有 NEED_FAULT_TOLERANCE 状态的任务
        List<TaskInstance> faultTasks = taskInstanceDao
            .findByStateAndInstanceId(pi.getId(), TaskState.NEED_FAULT_TOLERANCE);

        for (TaskInstance task : faultTasks) {
            // 第3步：检查重试预算
            if (task.getRetryTimes() < task.getMaxRetryTimes()) {
                // 还有重试次数 —— 重新提交
                task.setState(TaskState.SUBMITTED);
                task.setRetryTimes(task.getRetryTimes() + 1);
                taskInstanceDao.updateById(task);
                dispatchTask(task);  // 根据Worker分组分配到可用的Worker
                logger.info("Task {} re-dispatched (retry {}/{})",
                    task.getId(), task.getRetryTimes(), task.getMaxRetryTimes());
            } else {
                // 重试次数耗尽 —— 标记为失败
                task.setState(TaskState.FAILURE);
                taskInstanceDao.updateById(task);
                logger.warn("Task {} retries exhausted, marked FAILURE", task.getId());
            }
        }

        // 第4步：重建工作流执行状态，提交到线程池恢复调度
        WorkflowExecuteRunnable runnable = new WorkflowExecuteRunnable(pi);
        workflowExecuteThreadPool.submit(runnable);
    }
}
```

### Step 5：理解任务重试的完整逻辑

重试由两个关键参数驱动：`maxRetryTimes`（最大重试次数）和`retryInterval`（重试间隔秒数），数据存于`t_ds_task_instance`表。

```java
// TaskInstance 实体中的重试能力判断
public boolean canRetry() {
    return retryTimes < maxRetryTimes
        // 必须等待重试间隔到期，避免"闪电重试"打垮下游系统
        && (System.currentTimeMillis() - endTime.getTime()) > retryInterval * 1000L;
}

// WorkflowExecuteRunnable 中的失败处理逻辑
private void handleTaskFailure(TaskInstance task) {
    if (task.canRetry()) {
        // 延迟 retryInterval 秒后重新提交
        scheduler.schedule(() -> {
            task.setRetryTimes(task.getRetryTimes() + 1);
            submitTask(task);  // 重新通过RPC分派到Worker
        }, task.getRetryInterval(), TimeUnit.SECONDS);

    } else {
        // 重试次数耗尽 —— 标记失败
        task.setState(TaskState.FAILURE);
        taskInstanceDao.updateById(task);

        // 根据工作流级别的失败策略决定后续行为
        if (processInstance.getFailureStrategy() == FailureStrategy.END) {
            // 立即终止所有正在运行的任务，工作流整体标记为 FAILURE
            killAllRunningTasks();
            processInstance.setState(WorkflowState.FAILURE);
        } else {
            // CONTINUE: 失败任务的并行兄弟节点继续运行，仅阻塞下游
            checkAndBlockDownstreamTasks(task);
        }
    }
}
```

### Step 6：图解五种工作流恢复操作

DS控制台提供五种恢复操作，它们的底层语义截然不同。理解差异是故障应急的前提：

```bash
# ┌──────────────┬────────────────────────────────────────────────────┐
# │  操  作      │  底层语义与数据库影响                                │
# ├──────────────┼────────────────────────────────────────────────────┤
# │  重跑(Rerun) │  创建全新的ProcessInstance，新ID，新start_time      │
# │              │  INSERT INTO t_ds_process_instance (新记录)          │
# │              │  所有任务重新执行，不计历史状态                      │
# │              │  适用场景：参数修正后全量重跑                        │
# ├──────────────┼────────────────────────────────────────────────────┤
# │  恢复失败     │  在同一个ProcessInstance内，仅重跑FAILURE/NEED_     │
# │  (Recover    │  FAULT_TOLERANCE状态的任务                           │
# │   from       │  UPDATE t_ds_task_instance SET state=SUBMITTED      │
# │   Failure)   │  WHERE state=FAILURE AND process_instance_id=?       │
# │              │  SUCCESS的任务不会重跑                               │
# │              │  适用场景：修复上游数据问题后，从断点继续            │
# ├──────────────┼────────────────────────────────────────────────────┤
# │  从失败节点   │  创建新ProcessInstance，但跳过失败节点之前的所有任务│
# │  开始(From   │  新实例中：前置任务状态设为SUCCESS（虚拟完成），     │
# │  failed node)│  从失败节点开始真正执行                              │
# │              │  适用场景：长流程中途失败，不想重做前期成功步骤      │
# ├──────────────┼────────────────────────────────────────────────────┤
# │  暂停(Pause) │  状态：RUNNING_EXECUTION → READY_PAUSE → PAUSE      │
# │              │  正在执行的任务等待完成，待执行任务不启动            │
# │              │  UPDATE 状态为PAUSE，保留所有进度                    │
# │              │  适用场景：需要临时修改配置/数据，稍后继续           │
# ├──────────────┼────────────────────────────────────────────────────┤
# │  停止(Stop)  │  立即终止所有运行中任务（kill命令），工作流结束      │
# │              │  UPDATE state=KILL 所有RUNNING的任务                 │
# │              │  工作流状态变为STOP                                  │
# │              │  适用场景：发现严重逻辑错误，必须终止运行            │
# └──────────────┴────────────────────────────────────────────────────┘
```

### Step 8：设计容错友好的工作流架构

工作流本身的架构设计决定了容错恢复的便捷程度：

```
# 反模式 —— 单体巨型工作流
┌─────────────────────────────────────────────────────────┐
│  一个工作流：100个任务、预估执行8小时、最后一步易失败     │
│  问题：第95步失败 → 前7小时白跑                           │
│  恢复方式：仅能从失败节点开始 或 全部重跑                  │
│  风险：从失败节点开始时，上游数据可能已被修改              │
└─────────────────────────────────────────────────────────┘

# 推荐模式 —— 管道化小型工作流
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│ 工作流1:采集  │ → │ 工作流2:清洗  │ → │ 工作流3:汇聚  │
│ 30min         │   │ 30min         │   │ 1h            │
│ retry=3       │   │ retry=2       │   │ retry=2       │
│ 策略:CONTINUE │   │ 策略:CONTINUE │   │ 策略:END      │
└──────────────┘   └──────────────┘   └──────────────┘
         ↑ 失败仅重做此段       ↑ 失败仅重做此段
```

每个阶段独立可恢复：汇聚阶段失败，只需恢复汇聚即可，采集和清洗的结果已经安全落地。将一个8小时的巨型流程拆分为三个1小时左右的小流程，容错粒度从"全部重来"退化为"按阶段重来"。

### Step 9：配置容错相关核心参数

```yaml
# Master 配置文件 —— 故障转移参数
master:
  failover:
    interval: 10          # 扫描孤儿流程的周期（秒）
    max-threads: 5        # 并发恢复线程数（过大可能撑满DB连接池）

# Worker 配置文件 —— 心跳与存活检测
worker:
  heartbeat:
    interval: 10          # 向ZK发送心跳的周期（秒）
    max-miss: 3           # 连续3次未发心跳（30秒）→ Master判定Worker死亡
  # 注意：ZK自身的session_timeout通常设为8秒
  # 实际Worker死亡检测：max(zk_session_timeout, heartbeat.interval * max-miss) ≈ 30秒

# 任务级重试配置（在DS UI的任务定义页设置）
retry:
  max-times: 3            # 建议核心任务 ≥3，非核心任务 1~2
  interval: 60            # 建议 ≥30秒，防止"闪电重试"造成资源耗尽
  strategy: ALL           # 对所有失败原因都重试（备选：按退出码重试）
```

### Step 10：生产环境中六大常见容错坑与解决方案

| 问题 | 现象 | 根因 | 解决 |
|------|------|------|------|
| **双重执行** | 同一个任务产生两条SUCCESS记录，下游数据翻倍 | Worker宕机后任务重派，旧Worker恢复后也把任务跑完了 | Master在接收Worker执行结果时，先检查该TaskInstance的state是否为RUNNING，非RUNNING则丢弃结果 |
| **无限重试** | CPU打满，日志瞬间爆GB | retryInterval设为0秒，任务失败立即重试、又立即失败 | 最小重试间隔不得低于30秒 |
| **故障转移风暴** | Leader频繁切换（flapping），系统不停重建状态 | ZK session超时设得过短（<3秒），网络轻微抖动引发反复选举 | ZK session_timeout ≥ 8秒，适当增大至15秒 |
| **数据不一致** | Worker执行成功但崩溃前未上报，Master以为失败了 | Worker处理完任务后，RPC回执在网络或客户端侧丢失 | 任务设计遵循幂等原则：使用`INSERT OVERWRITE`而非`INSERT INTO`，用`REPLACE`而非`INSERT` |
| **暂停/停止卡死** | 工作流长时间停在READY_PAUSE状态 | 某任务在Worker上一直运行不结束（Spark任务卡在YARN队列中等待资源） | 设置任务级超时（timeout参数），超时后系统自动kill该任务 |
| **Command队列丢失** | Master重启后，"应该被调度"的任务没有被调度 | 虽然Command存于DB，但消费标记（消费时间戳）在异常关闭时未正确回滚 | 升级至DS 3.2+，该版本修复了Command消费的事务原子性问题 |

## 四、项目总结

DolphinScheduler的容错体系围绕一个核心思想构建：**状态不下内存，一切看数据库**。

**容错能力矩阵**：

| 故障类型 | 自动恢复 | 恢复时间 | 需要人工介入 |
|---------|---------|---------|-------------|
| Master进程崩溃 | ✅ 自动 | ~10-15秒 | 否 |
| Worker进程崩溃 | ✅ 自动 | ~30秒 | 否（除非重试耗尽） |
| 任务执行失败 | ✅ 自动重试 | 按retryInterval | 否（除非重试耗尽） |
| ZK集群短暂中断 | ✅ 自动重连 | ~8秒 | 否 |
| 数据库短暂不可用 | ❌ | — | 是（重启服务或等待DB恢复） |
| 网络分区（脑裂） | ⚠️ 部分 | 1-30秒 | 视严重程度 |
| 历史数据错误需回刷 | ❌ | — | 是（使用补数或重跑） |

**生产容错设计三条铁律**：
1. **幂等性优先**：所有任务必须是幂等的——`INSERT OVERWRITE`、`REPLACE`、`MERGE`而非`INSERT INTO`。这样即使任务被重派两次，数据也不会翻倍。
2. **化整为零**：将8小时的巨型工作流拆分为3-5个小流程，失败时只需重做失败的阶段，提升恢复效率。
3. **设置重试预算**：核心任务retry=3、interval=60s；非核心任务retry=1。不给无限重试留余地。

**大麦凌晨故障的复盘结论**：(1) 关键路径任务maxRetryTimes=0是配置错误——应设为≥2；(2) 缺失故障告警——应该在工作流进入NEED_FAULT_TOLERANCE超过3分钟时自动发出钉钉通知；(3) 未做过Master故障转移的演练——在非关键时段定期进行"混沌工程"演练，每季度一次。

**思考题**：
1. DS中暂停操作会等待当前运行任务完成后才进入PAUSE状态。如果某个Spark任务在YARN上排队等了2小时还没获得资源，暂停就一直卡在READY_PAUSE。你会如何设计一个"强制暂停"能力——既能快速暂停，又不丢失数据？
2. 假设你要为DS增加一个"故障自愈"Dashboard——在一个面板上展示所有当前处于NEED_FAULT_TOLERANCE、FAILURE、READY_PAUSE等异常状态的工作流，并支持一键批量恢复。请描述前后端各自需要新增哪些接口和数据结构。
