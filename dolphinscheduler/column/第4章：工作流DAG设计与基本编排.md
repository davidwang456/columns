# 第4章：工作流DAG设计与基本编排

> **定位**：从画布到运行——上手第一个工作流。
> **核心内容**：DAG基本概念、画布操作、工作流核心配置、全局变量、运行状态机。
> **实战目标**：创建一个包含Shell + Python两个节点的串行DAG，跑通并查看日志。

---

## 1. 项目背景

某互联网公司的运营团队每天都要处理一套"老三样"数据流程：凌晨跑一个数据清洗的Shell脚本把原始日志规整入库，接着跑一个Python分析脚本计算前一天的日活和留存，最后把分析结果汇总成一份报表邮件发给各部门老大。

听起来不复杂，但他们现在的做法堪称"人肉调度"——运维小张定了凌晨1点的闹钟，爬起来手动执行第一步脚本，盯着终端等它跑完，确认没报错，再敲命令执行第二步，最后手动发邮件。如果第一步在凌晨3点因为上游日志迟到挂了，第二步和第三步就只能干等到白天有人来处理——等小张早上8点半到公司一看，报表还是空的，部门会已经开了，老板的脸比炭还黑。

这种手工串联的方式有三个致命痛点：一是**人工依赖**，凌晨没人值守就停摆；二是**串联脆弱**，一步出错全线瘫痪，没有自动重试也没有告警；三是**无迹可寻**，跑了多少次、每次耗时多久、哪天失败过，全凭小张的Excel手工记录，数据可靠性堪忧。

如果把这三个任务变成一个有向无环图（DAG），让调度器自动依次执行、失败自动告警、完成自动通知，小张就能睡个安稳觉了。这一章，我们就来动手搭这个"自动驾驶"的工作流。

---

## 2. 项目设计——剧本式交锋对话

办公室里，小胖端着刚泡好的奶茶晃过来，看见大师正在DolphinScheduler的控制台上画东西。

**小胖**："大师你在搭积木呢？这个画布上拖方块、拉线条的，看着跟幼儿园画画似的。"

**大师**："这叫有向无环图——DAG。你看这三个方块是任务节点，这条线是执行依赖，意思就是A做完才能做B，B做完才能做C。"

**小胖**："嗨，这不就是把三个任务用线连起来，一个接一个跑嘛，跟地铁线路图一样！每站到了才能去下一站，简单得很。"

**小白**听见"地铁"两个字，从工位探过头来："那如果第二个任务失败了怎么办？第一个要不要重跑？还有——万一有人手抖把最后一个节点连回了第一个节点，那不就成了环线地铁，无限循环了？"

**大师**笑了笑："小白问到了两个关键点。先说环形——DAG的'有向无环'四个字，'无环'是硬约束。DolphinScheduler在保存工作流时会做拓扑检测，一旦发现环，直接拒绝保存，弹红框告诉你'检测到循环依赖'。这就像多米诺骨牌——每张牌推倒前必须等前一张倒了，但你不可能把最后一张牌绕回去推倒第一张，那物理上就推不起来。"

**技术映射**：DAG的"无环"约束由调度器的拓扑排序算法保障，画布保存时实时校验入度/出度，有环则拦截。

**小白**点头，继续追问："那失败处理呢？如果我第一个数据清洗脚本因为源文件没到失败了，后面两个任务怎么办？是直接停掉，还是跳过报错的任务继续跑后面的？"

**大师**："这就涉及工作流级别的'失败策略'配置了。有两种选择：选'结束'——任何节点失败，整个工作流立刻停止，后续节点不再执行；选'继续'——当前节点失败后，标记它失败，但下游节点照常运行。你们这个场景，清洗失败了后面的分析肯定没数据可用，应该选'结束'并配合失败告警，而不是盲目继续。"

**小胖**眼睛一亮："那能不能给它加个超时？比如我那个清洗脚本正常跑10分钟就够了，万一卡住了等半小时还没完——"

**大师**："对，这就是'超时告警'配置。你可以设超时时长30分钟，超时策略有两种：'超时告警'——仅发通知但不中断执行；'超时失败'——直接标记任务失败并触发后续失败策略。生产环境建议用后者，配合告警渠道让运维第一时间收到通知。"

**小白**若有所思："那如果同时有好几个工作流在跑，服务器资源紧张，能不能让重要的先跑？"

**大师**："优先级系统就是干这个的。五个等级——HIGHEST、HIGH、MEDIUM、LOW、LOWEST。核心报表优先级拉满，非关键的日志归档降到LOW。另外你还可以通过Worker分组把执行节点隔离开，ETL任务指定ETL组的Worker，报表任务指定报表组的Worker，互不抢资源。"

**技术映射**：优先级影响任务在Master执行队列中的排序权重；Worker分组通过标签机制将任务绑定到指定节点组。

**小胖**挠挠头："那参数怎么传？我第一步生成了一个日期，第二步需要用这个日期——"

**大师**："全局变量+`${}`语法。比如你定义一个全局变量`${biz_date}`，赋值为系统参数`system.biz.date`，它自动取调度日期的前一天。三个节点都能引用它。如果上游节点输出了某个值要传给下游，用自定义参数的OUT/IN方向配置，这就是工作流的血脉。"

**技术映射**：`${varName}`是DS的变量占位符语法，运行时由Worker端执行参数替换；`system.biz.date`等内置参数由Master在创建实例时注入。

## 3. 项目实战

### 环境准备

- DolphinScheduler 3.x 已部署（Standalone 或集群模式均可，参见第2章）
- 确保浏览器能打开 DS 控制台（默认 `http://localhost:12345/dolphinscheduler`）
- 默认账号：`admin` / `dolphinscheduler123`
- 确保 Worker 节点已正常注册（在"监控中心"可查看节点状态）

### Step 1：创建项目

登录 DS 控制台，点击左侧导航栏【项目管理】→ 右上角【创建项目】按钮，弹出创建表单。

- **项目名称**：`每日数据处理`
- **描述**：运营数据清洗、分析与报表编排

点击【确认】，项目创建成功。此时你进入了该项目的画布管理页。每个项目是工作流定义的命名空间，不同项目间的参数、数据源、资源文件默认隔离。

### Step 2：进入工作流定义画布

在【每日数据处理】项目页，点击顶部 Tab【工作流定义】→ 点击【创建工作流】按钮。系统会打开一个全屏的画布编辑器。

画布中央是绘图区，左侧是可拖拽的节点面板（Shell、Python、SQL、Spark、Flink、SubProcess、Dependent、Conditions 等），顶部是工具栏（保存、运行、导入导出），画布底部是状态栏。

将左上角"工作流名称"栏的默认值改为：`daily_data_pipeline`。

### Step 3：设置全局变量

在画布右侧（如果未展开，点击右上角齿轮图标打开配置面板），找到"全局参数"区域。点击【添加】按钮，新增一行：

| 参数名 | 值 | 方向 |
|--------|-----|------|
| `biz_date` | `${system.biz.date}` | IN |

这个全局变量将自动取 DolphinScheduler 的"业务日期"——即调度时间的前一天（T-1），正是运营团队需要的昨天数据日期。工作流中所有节点都可以通过 `${biz_date}` 引用它。

### Step 4：拖拽三个 Shell 节点

从左侧节点面板依次拖拽三个 Shell 节点到画布上：

**节点 A"数据清洗"**：

右键节点 → 编辑，或直接双击节点进入编辑弹窗。
- 节点名称：`数据清洗`
- 脚本内容：
```bash
#!/bin/bash
echo "========== 数据清洗开始 =========="
echo "业务日期: ${biz_date}"
# 模拟清洗逻辑：将原始日志格式化后写入清洗目录
INPUT_DIR="/data/logs/raw/${biz_date}"
OUTPUT_DIR="/data/logs/cleaned/${biz_date}"
mkdir -p ${OUTPUT_DIR}
if [ -d ${INPUT_DIR} ]; then
    echo "清洗完成，输出目录: ${OUTPUT_DIR}"
    exit 0
else
    echo "错误: 原始日志目录不存在 ${INPUT_DIR}" >&2
    exit 1
fi
```
- 点击【确认】关闭弹窗。

**节点 B"数据分析"**：

拖拽第二个 Shell 节点（分析用 Python，但 Shell 节点中调用 Python 更直观，方便查看退出码），设置：
- 节点名称：`数据分析`
- 脚本内容：
```bash
#!/bin/bash
echo "========== 数据分析开始 =========="
echo "业务日期: ${biz_date}"
python3 -c "
import json, random
# 模拟日活/留存计算
dau = random.randint(80000, 120000)
retention = round(random.uniform(0.3, 0.5), 3)
result = {'biz_date': '${biz_date}', 'DAU': dau, 'retention': retention}
print(json.dumps(result, indent=2, ensure_ascii=False))
" || { echo "分析失败"; exit 1; }
```
- 点击【确认】。

**节点 C"数据归档"**：

- 节点名称：`数据归档`
- 脚本内容：
```bash
#!/bin/bash
echo "========== 数据归档开始 =========="
echo "业务日期: ${biz_date}"
# 模拟归档：将结果文件压缩并移动到历史目录
ARCHIVE_DIR="/data/archive/${biz_date}"
mkdir -p ${ARCHIVE_DIR}
tar -czf ${ARCHIVE_DIR}/result_${biz_date}.tar.gz /data/output/${biz_date}/ 2>/dev/null
echo "归档完成: ${ARCHIVE_DIR}/result_${biz_date}.tar.gz"
```
- 点击【确认】。

此时画布上有三个独立的节点块，彼此没有连线，像三个孤岛。

### Step 5：连接上下游依赖

连线操作非常简单：鼠标悬停在节点 A 的下边缘，会出现一个蓝色锚点。按住拖拽到节点 B 的上边缘，松开即完成连线。同样的操作，从节点 B 拖锚点连到节点 C。

完成后，画布呈现 `数据清洗 → 数据分析 → 数据归档` 的线性 DAG。你可以在画布空白处拖拽整组节点调整布局，也可以选中节点后用方向键微调。

**常见操作备忘**：
- `Ctrl + 滚轮`：缩放画布
- 选中节点 → `Delete`：删除节点（也会删除关联连线）
- `Ctrl + C/V`：复制/粘贴节点（含内部配置）
- 按住 `Ctrl` 框选多个节点可批量操作

### Step 6：配置工作流级参数

点击画布顶部的设置齿轮（或右侧配置面板），配置以下工作流级参数：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 失败策略 | 结束 | 任何一个节点失败后停止后续节点 |
| 通知策略 | 失败发 | 仅在失败时发送告警通知（需预先配置告警组） |
| 流程优先级 | MEDIUM | 中等优先级，一般业务流推荐 |
| Worker分组 | default | 使用默认Worker分组 |
| 超时告警 | 开启，30分钟，超时失败 | 任一个任务卡住超过30分钟则标记失败 |
| 租户 | default | 以默认Linux用户身份执行 |

配置完成后，点击顶部【保存】按钮。系统会校验 DAG 环路、节点名称唯一性、脚本非空等规则。校验通过后弹出"保存成功"提示。

### Step 7：上线并运行

保存只是把工作流定义存到了数据库，要让定时调度生效还需要"上线"。但对于手工跑一次测试，直接在【工作流定义】列表页找到 `daily_data_pipeline`，点击操作栏的【运行】按钮。

弹出启动参数确认框，可在此补充或覆盖全局参数（本次不修改，直接确认）。系统会创建一个工作流实例，跳转到【工作流实例】页面。

### Step 8：观察执行过程与结果

在【工作流实例】页面，你会看到一条新记录，状态栏依次经历：

`SUBMITTED（已提交）` → `RUNNING_EXECUTION（运行中）` → `SUCCESS（成功）`

点击该实例名称，进入详情页，可以看到三个任务节点的实时执行状态。每个节点旁边有颜色标记：
- 绿色 ✓ = 成功
- 红色 ✗ = 失败
- 蓝色 ⟳ = 运行中
- 灰色 — = 等待中

右键任意节点 →【查看日志】，可在弹出窗口中逐行查看 Shell 脚本的 stdout/stderr 输出，这是日常排查问题最常用的入口。

点击顶部【甘特图】选项卡，能看到三个节点在时间轴上的排布——数据清洗从 0 秒开始，数据分析在清洗结束后才开始，数据归档则更靠后。这就是 DAG 严格按照依赖顺序执行的直观体现。

### Step 9：新手常见问题排查

**问题一：节点一直卡在"提交中"（蓝色旋转图标不消失）**

大概率是 Worker 不可用。去【监控中心】→【Worker】页面查看对应 Worker 是否显示"在线"。如果离线，检查 Worker 进程是否启动、注册中心是否可达。

**问题二：报错"exit code: 1"但脚本本地能跑**

DS Worker 的执行环境可能和本地不同：
- 检查脚本中引用的路径在 Worker 机器上是否存在
- Python 节点需确认 Worker 节点的 `PYTHON_HOME` 配置正确（参见 `dolphinscheduler_env.sh`）
- 如果脚本中使用了 `${biz_date}` 等变量但未定义，会被替换为空字符串

**问题三：保存时报"检测到循环依赖"**

检查是否误将下游节点连回了上游，或者在更复杂的 DAG 中形成了间接环路（A→B→C→…→A）。删掉非预期的连线后重新保存。

**问题四：运行结束后看不到告警通知**

检查两步前提：① 在【安全中心】→【告警组管理】中已创建告警组并绑定用户；② 告警渠道已在 `alert.properties` 中正确配置（SMTP/企业微信等参数）。告警仅在工作流级"通知策略"为非"无"且"告警组"已选择时生效。

## 4. 项目总结

### DAG 调度的优势与局限

| 维度 | 优势 | 局限 |
|------|------|------|
| 可靠性 | 依赖关系自动保障执行顺序，不会跳步或乱序 | 单个节点失败可能阻塞整条链路，需合理配置失败策略 |
| 可观测性 | 甘特图、实例日志、状态流转一览无余 | 日志默认分散在各 Worker 节点，集中采集需额外部署 |
| 可维护性 | 画布拖拽修改拓扑，比写 Cron 脚本直观得多 | 节点数超过 50 个时画布拥挤，排查连线困难 |
| 灵活性 | 全局参数+变量系统实现运行时传参，解耦静态配置 | OUT/IN 参数传递链路长时易写错变量名 |

### 常见 DAG 设计模式

| 模式 | 图例 | 适用场景 |
|------|------|----------|
| 串行 | A → B → C | 数据清洗→分析→归档，前后严格依赖 |
| 并行 | A → (B1, B2) → C | 多数据源同时拉取，汇总后统一处理 |
| 菱形 | A → (B, C) → D | B和C独立完成两部分计算，D合并结果 |

### 三个常见踩坑

1. **循环依赖**：在复杂 DAG 中，如果不小心将 C 连回 A，看似只有3个节点也可能形成环。保存前务必检查每条边的方向是否与意图一致。
2. **参数拼写不一致**：全局参数定义时用了 `${biz_date}`，某个节点的脚本里却写了 `${bizdate}`（少了下面线），Worker 运行时不会报错，只会原样输出 `${bizdate}`，导致逻辑静默出错。
3. **失败策略选错**：选了"继续"策略，清洗节点失败后分析节点照样跑，结果分析出了一个错误的日活数发给了老板——而且你还收不到失败通知，因为工作流最终可能是"成功"状态。

### 思考题

1. 如果节点 B 执行失败，工作流失败策略设为"结束"，那么节点 C 会处于什么状态？此时若修改失败策略为"继续"并重跑该实例，节点 C 是否会被执行？
2. 假设节点 A 成功执行后输出了一个参数 `record_count=5000`，节点 B 需要读取这个值做分支判断。请设计参数传递方案，并指出 OUT/IN 方向分别配置在哪个节点上。
