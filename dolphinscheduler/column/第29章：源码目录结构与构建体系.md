# 第29章：源码目录结构与构建体系

## 一、项目背景

大麦数据（Damai）的调度平台已经连续运行一年半，期间 DolphinScheduler 承载了 200+ 业务工作流，覆盖从 ETL 到模型训练的全部场景。然而，随着公司自研的机器学习平台"DM-Prophet"即将上线，技术团队面临一个全新的需求：需要为 DolphinScheduler 添加一个专用于 DM-Prophet 的自定义任务类型，支持在调度系统中直接提交分布式训练作业。

负责这项任务的是高级开发工程师 Lucy——她在 UI 上编排了上百个工作流，对各任务类型了如指掌，但从未打开过 DolphinScheduler 的源码。当她第一次克隆仓库后发现：26 个 Maven 模块、1500+ 个 Java 源文件、33 个任务插件、28 个数据源插件、7 个存储插件……"这不像一个项目，这像一座城市。"Lucy 盯着 IDEA 里展开的目录树感叹道。

她尝试找到任务执行的入口——搜索 `ShellTask`，结果在 `dolphinscheduler-task-plugin` 下看到了几十个类似的子模块；搜索 `MasterServer`，一路从 `dolphinscheduler-service` 追到 `dolphinscheduler-dao`，最终迷失在 MyBatis-Plus 的 Mapper 层。最要命的是——她不确定该从哪里开始写代码。Task Plugin 需要实现什么接口？SPI 注册文件放在哪里？新模块如何挂接到 Maven 构建体系中？每个问题都像一座迷宫。

更大的困惑来自构建流程：`pom.xml` 定义了 26 个 `<module>`，依赖关系层层嵌套——`dolphinscheduler-common` 被所有模块依赖，`dolphinscheduler-spi` 是所有插件模块的"宪法"，`dolphinscheduler-dao` 和 `dolphinscheduler-service` 被 API、Master、Worker 三个服务同时依赖。Lucy 试图只构建自己关心的 `dolphinscheduler-task-plugin`，结果 Maven 报了一屏的 `Could not resolve dependencies`。她终于明白：想在 DolphinScheduler 的源码城市里"建一栋新楼"，必须先读懂它的城市规划和施工规范。

本专栏由此进入**高级篇**——从"使用 DS"到"理解 DS"再到"改造 DS"。本章作为高级篇的起点，将系统拆解 DolphinScheduler 的源码目录结构、Maven 模块依赖图、构建体系与调试环境搭建，为你建立一张完整的"源码城市地图"。

## 二、项目设计——剧本式交锋对话

**小胖**嘴里嚼着辣条，眼睛盯着屏幕上的 26 个 Maven 模块目录，一头雾水："二十六个模块？我就想改一个 Shell 任务的超时逻辑，需要搞清楚这么多东西吗？看什么源码啊——UI 上能用就行了呗！API 文档不是写得很清楚吗？再说了，这代码量，没半个月连读完都困难。"

**大师**笑了笑，端起保温杯："那假如 Lucy 现在要加一种新的任务类型——DM-Prophet 训练任务——不看源码，你怎么知道 `TaskChannel` 接口应该怎么实现？SPI 注册文件应该放在哪个 Maven 模块、哪个 package 下？编译的时候怎么让 DolphinScheduler 的 Master 和 Worker 加载你的插件？这些可都不是 UI 和 API 文档能告诉你的。"

**小白**推了推眼镜，声音很低但思路清晰："我更关心的是逆向理解。DS 的很多边界行为——比如超时重试到底是在 Master 层处理还是在 Worker 层处理、任务间的变量传递如何在 RPC 序列化中保真——不读源码就只能靠猜或者看二三手博客。Lucy 要加自定义任务，光知道接口怎么实现是不够的，还必须理解 Master 怎么派发任务、Worker 怎么回传状态、异常时如何优雅降级。"

**小胖**挠了挠头："那这堆目录……有没有一张地图？common 是干嘛的，extract 又是啥？听着像跑物流的。"

**大师**起身走到白板前，画了一张分层的模块依赖图："DS 的源码是一座规划严密的城市——`dolphinscheduler-common` 是城市的基础设施，水电管网，所有楼都要通；`dolphinscheduler-spi` 是建筑规范，规定了每种插件的接口标准——你想建一座'DM-Prophet 任务'大楼，先得按 `TaskChannel` 这张设计图施工；`dolphinscheduler-dao` 是档案馆，数据怎么存、表怎么改都有规矩；`dolphinscheduler-service` 是行政中心，业务逻辑在这里流转；`dolphinscheduler-extract` 是交通规则，Master、Worker、Alert 三个服务之间怎么通信，全由它定义。"

他顿了顿继续道："`dolphinscheduler-registry` 是地址簿——集群里各个服务的地址谁来管？ZooKeeper、Etcd 还是数据库？它把注册中心的具体实现抽象掉，让上层只用 SPI 接口。`dolphinscheduler-meter` 是体检中心——Prometheus 指标采集、负载保护、限流熔断，都在这里。跑起来的服务——API、Master、Worker、Alert、Standalone——就是城市里的各个功能区：API 是政务服务大厅，Master 是总调度室，Worker 是工厂车间，Alert 是消防站。"

**小白**追问："可是模块这么多，依赖关系怎么梳理？比如我想确认 '改 common 里一个工具方法会不会影响所有模块'——怎么快速判断影响面？构建单模块的时候怎么只编需要的部分？"

**大师**点头："三个实战视角。第一，`common` 位于依赖链的最底层——它的 `pom.xml` 里几乎不依赖任何 DolphinScheduler 内部模块，但所有其他模块的 `pom.xml` 都依赖它。所以改 `common` 要极其谨慎——影响全局。第二，构建命令 `./mvnw -pl dolphinscheduler-master -am clean install` 中 `-pl` 指定模块、`-am` 代表 also-make（同时构建依赖），Maven 会自动推算最少构建范围。第三，用 `mvn dependency:tree -pl <module>` 打印依赖树，比手翻 pom.xml 快一百倍。"

**小胖**忽然一拍脑袋："等等！我自己用 Maven 这么多年，但是这里都是 `./mvnw` 而不是 `mvn`——这个 wrapper 是干嘛用的？还有 `spotless:apply` 是什么鬼？"

**大师**欣然道："`./mvnw` 是 Maven Wrapper——它在 `.mvn/wrapper/` 下写死了 Maven 版本号，无论开发者的机器上装了什么版本的 Maven，`./mvnw` 都会自动下载并使用项目所需的版本，保证所有人构建环境一致。`spotless:apply` 是代码格式化插件——DS 对代码风格极其严格，导入顺序、缩进风格都有精确规范。不先跑 `spotless:apply` 就提交 PR，CI 会直接挂掉。这些小工具恰恰体现了优秀开源项目的工程素养。"

**大师**最后总结："源码城市里建楼，三步走——先看懂地图（模块依赖），再读懂规范（SPI 接口），最后开工（写插件）。建筑图纸是 `TaskChannel`，施工规范是 `META-INF/services/` 下的 SPI 注册文件，验收标准是 `./mvnw clean install` 能通过。理解这一层，26 个模块就不吓人了——它们只是城市规划的不同功能分区。"

**技术映射**：`dolphinscheduler-common` 映射 Java 开发中的"工具包"，所有模块的通用基础；`dolphinscheduler-spi` 映射"接口定义层"，就是 Java SPI 机制——`META-INF/services/` 文件的实现即声明；`dolphinscheduler-dao` 映射 DAO 层，对应数据库表 `t_ds_*`；`dolphinscheduler-service` 映射 Spring `@Service`；`dolphinscheduler-extract` 映射 gRPC/Netty 的 `.proto` 协议文件；`./mvnw` 映射 `gradlew`，统一构建工具版本。

### Step 2：理解 Maven 模块依赖层次

完整的模块依赖关系树如下，**箭头指向表示"依赖"（A → B 表示 A 依赖 B）**：

```
Parent POM（根项目）
├── dolphinscheduler-bom（Bill of Materials —— 集中版本管控）
│    作用：统一所有第三方依赖的版本号，类似 Spring Boot 的 dependencyManagement
│
├── dolphinscheduler-common（基础设施层 ★ 不依赖任何 DS 内部模块）
│    ↑─── 一切从这里开始，所有模块都声明 <dependency>common</dependency>
│
├── dolphinscheduler-spi（插件契约层 ★ 仅依赖 common）
│   ├── dolphinscheduler-task-spi       ← TaskChannel 接口定义在此
│   ├── dolphinscheduler-datasource-spi ← DatasourceChannel 接口定义在此
│   └── dolphinscheduler-storage-spi   ← StorageOperate 接口定义在此
│
├── dolphinscheduler-dao（MyBatis 持久层 ★ 依赖 common）
│    核心位置：resources/sql/ —— 所有数据库 DDL/DML 迁移脚本
│
├── dolphinscheduler-service（业务层 ★ 依赖 common + dao）
│
├── dolphinscheduler-extract（RPC 契约 ★ 依赖 common）
│   ├── dolphinscheduler-extract-master   ← Master 暴露的 RPC 接口
│   ├── dolphinscheduler-extract-worker   ← Worker 暴露的 RPC 接口
│   └── dolphinscheduler-extract-alert    ← Alert 暴露的 RPC 接口
│
├── dolphinscheduler-registry（注册中心 ★ 依赖 common + spi）
│
├── dolphinscheduler-meter（指标与保护 ★ 依赖 common）
│
├── 可运行服务（各依赖 service + extract + registry）
│   ├── dolphinscheduler-api         → REST Server（port 12345）
│   ├── dolphinscheduler-master      → 编排引擎（port 5679 RPC）
│   ├── dolphinscheduler-worker      → 任务执行器（port 1235 RPC）
│   ├── dolphinscheduler-alert-server → 告警服务（port 50052 RPC / 50053 HTTP）
│   └── dolphinscheduler-standalone-server → 一体化开发服务器
│
├── 插件家族（各依赖 spi + common）
│   ├── dolphinscheduler-task-plugin/      33个子模块（shell/sql/python/spark/...）
│   ├── dolphinscheduler-datasource-plugin/ 28个子模块（mysql/hive/postgresql/...）
│   ├── dolphinscheduler-storage-plugin/    7个子模块（s3/hdfs/oss/gcs/...）
│   ├── dolphinscheduler-scheduler-plugin   （Cron 调度器 / Quartz 实现）
│   └── dolphinscheduler-dao-plugin         （数据库方言 MySQL/PostgreSQL/H2）
│
└── dolphinscheduler-dist（发布模块 ★ 依赖所有子模块）
```

这个依赖层次回答了 Lucy 最初的问题：**新任务插件该放在哪里？答案是 `dolphinscheduler-task-plugin/` 下新建一个子模块，该子模块只需依赖 `dolphinscheduler-task-spi`，编译时通过 `./mvnw -pl` 指定即可。**

### Step 5：追踪一次 Shell 任务的全链路调用

这是理解 DS 架构的核心 Debug 路径。依次在以下位置打断点：

```java
// 断点① API 接收请求
// 文件：dolphinscheduler-api/.../controller/ExecutorController.java
// 方法：startProcessInstance()
// 作用：用户点击"运行"→ REST API 接收 → 写入 t_ds_command 表

// 断点② Master 消费命令
// 文件：dolphinscheduler-master/.../runner/MasterSchedulerBootstrap.java
// 方法：run()
// 作用：Master 的主循环 → 不断从 t_ds_command 表拉取待执行命令

// 断点③ Master 派发任务
// 文件：dolphinscheduler-master/.../engine/WorkflowExecuteRunnable.java
// 方法：submitTask()
// 作用：Master 检查 DAG → 找到第一个就绪任务 → 通过 RPC 发给 Worker

// 断点④ Worker 接收任务
// 文件：dolphinscheduler-worker/.../processor/TaskExecuteProcessor.java
// 方法：process()
// 作用：Worker 收到 RPC 请求 → 根据任务类型查找对应 TaskChannel 实现

// 断点⑤ Shell 任务执行
// 文件：dolphinscheduler-task-plugin/.../shell/ShellTask.java
// 方法：run() 或 handle()
// 作用：构建 ProcessBuilder → 执行 Shell 脚本 → 捕获 stdout/stderr → 上报状态
```

请求调用链（文本描述，便于讲解）：

```
用户点击"运行"（UI Web）
  → API: ExecutorController.startProcessInstance()
  → API 将命令写入数据库 t_ds_command 表
  → Master: MasterSchedulerBootstrap 轮询 t_ds_command
  → Master: WorkflowExecuteRunnable 启动工作流实例
  → Master: 检查 DAG → 定位第一个就绪任务 → RPC 派发给 Worker
  → Worker: TaskExecuteProcessor.process() 接收
  → Worker: 查找 TaskChannel SPI 实现 → 调用 ShellTask.run()
  → Worker: Shell 进程执行 → 捕获输出 → 持久化日志
  → Worker: RPC 回传执行结果给 Master
  → Master: 更新 t_ds_task_instance → 触发下一 DAG 节点
  → UI: 轮询 API 获取状态更新 → 渲染到前端
```

### Step 6：读懂一个最简单的插件——Shell 任务

打开四个关键文件，理解插件开发的标准模式：

```
dolphinscheduler-task-plugin/dolphinscheduler-task-shell/src/main/
├── java/org/apache/dolphinscheduler/plugin/task/shell/
│   ├── ShellTask.java          ← 核心执行逻辑（ProcessBuilder 构建 shell 进程）
│   ├── ShellTaskChannel.java   ← TaskChannel 接口实现（工厂 + 参数解析）
│   └── ShellParameters.java    ← 任务参数 POJO
└── resources/META-INF/services/
    └── org.apache.dolphinscheduler.plugin.task.api.TaskChannel   ← SPI 注册文件
```

`META-INF/services/` 下的 SPI 注册文件内容只有一行（样例）：

```
org.apache.dolphinscheduler.plugin.task.shell.ShellTaskChannel
```

这是 Java SPI 机制的核心——**DS 会在 classpath 中扫描所有 `META-INF/services/TaskChannel` 文件，自动发现并加载所有 TaskChannel 实现类。** 任何新任务插件的注册都遵循这个模式，没有其他额外的 XML 配置。

### Step 8：创建自定义任务插件模块（框架）

Lucy 要做的实际开发的第一步——新建一个空模块框架：

```bash
# 创建目录结构
mkdir -p dolphinscheduler-task-plugin/dolphinscheduler-task-dmprophet/src/main/java/org/apache/dolphinscheduler/plugin/task/dmprophet
mkdir -p dolphinscheduler-task-plugin/dolphinscheduler-task-dmprophet/src/main/resources/META-INF/services

# 复制已有插件 pom.xml 作为模板（熟悉标准依赖）
cp dolphinscheduler-task-plugin/dolphinscheduler-task-shell/pom.xml \
   dolphinscheduler-task-plugin/dolphinscheduler-task-dmprophet/pom.xml
```

然后编辑 `dolphinscheduler-task-plugin/dolphinscheduler-task-dmprophet/pom.xml`：

```xml
<parent>
    <groupId>org.apache.dolphinscheduler</groupId>
    <artifactId>dolphinscheduler-task-plugin</artifactId>
    <version>${revision}</version>
</parent>
<artifactId>dolphinscheduler-task-dmprophet</artifactId>
<name>dolphinscheduler-task-dmprophet</name>
```

**关键一步**——在父模块 `dolphinscheduler-task-plugin/pom.xml` 的 `<modules>` 中加入新模块，否则编译时不会识别：

```xml
<modules>
    <!-- ...已有 33 个模块... -->
    <module>dolphinscheduler-task-dmprophet</module>
</modules>
```

验证模块是否正确挂接：

```bash
./mvnw -pl dolphinscheduler-task-plugin/dolphinscheduler-task-dmprophet -am clean install
# 输出 BUILD SUCCESS 则模块已进入构建体系
```

### Step 9：源码开发常见规范与陷阱

| 陷阱                                   | 后果                                       | 正确做法                                                     |
|----------------------------------------|--------------------------------------------|--------------------------------------------------------------|
| 使用 Java 11 API（如 `var`、`String.isBlank()`） | 在 JDK 1.8 环境中编译失败                   | 确认 IDE Project SDK 为 1.8；提交前 `./mvnw clean compile -pl <模块>` |
| 未运行 `spotless:apply` 即提交        | CI 格式检查失败，PR 被拒绝                  | 养成习惯：`./mvnw spotless:apply` → `git add` → `git commit` |
| 提交信息不规范                        | 不通过 commit lint 检查                     | 格式必须为 `[Fix-issue号] [模块] 简短描述`，如 `[Fix-18168] [Worker] Fix task retry counter` |
| 新建插件模块未注册到父 pom.xml         | 编译时模块被忽略，插件无法加载              | 在父模块 `pom.xml` 的 `<modules>` 列表中加入新模块            |
| 使用系统 Maven 而非内置 wrapper       | Maven 版本不一致导致构建行为差异             | 始终使用 `./mvnw`（Windows 下为 `mvnw.cmd`）                  |
| 在 `common` 中引入重量级依赖          | 所有模块被迫加载该依赖，JAR 体积膨胀        | `common` 只放纯工具类和无外部依赖的模型类；第三方依赖放在使用它的模块中 |

### 构建 vs 调试 vs 生产三环境差异

| 维度         | 源码构建（开发）                    | 本地调试（Standalone）            | 生产部署（集群）                    |
|--------------|-------------------------------------|-----------------------------------|-------------------------------------|
| 数据库       | H2 内存库（默认）或本地 MySQL       | H2 或本地 MySQL                   | MySQL 8.0+ / PostgreSQL 13+         |

### 给新贡献者的三条建议

1. **读 SPI 接口，不读具体实现**——理解系统从 `META-INF/services/` 入手。所有插件的"入口"就是 SPI 注册文件，由此找到接口定义，再由接口回溯实现类。比「打开一个文件一行行啃」高效十倍。
2. **先跑通全链路调试，再写代码**——在 ShellTask 打断点、从 UI 触发一次完整执行，亲眼看到数据如何在 API → Master → Worker → Plugin 之间流转。一次完整追踪胜过读十篇源码分析博客。
3. **构建只编需要的，依赖树必不可少**——`./mvnw -pl <目标> -am clean install` 构建自己关心的模块；`mvn dependency:tree` 排查依赖问题。掌握这两个命令就能摆脱"全量编译半小时坐等"的低效循环。
