# 第23章：一致性级别与可见性控制

> **定位**：解释为什么刚写入的数据不一定立刻被搜到。
> **版本**：Milvus 2.5.x
> **源码关联**：internal/proxy/task.go、internal/rootcoord/timestamp.go、internal/querynodev2/search.go

---

## 1. 项目背景

某实时推荐系统每天有 50 万次用户行为事件写入 Milvus，要求写入后 100ms 内对搜索可见。开发小刘发现，即使 Insert 返回 success、也调了 Flush，搜索仍然偶尔漏掉最近写入的数据——平均每 100 次搜索有 3-5 次看不到刚写入的数据。

小刘在代码里加了重试逻辑：搜不到就重搜，再搜不到就告警。结果告警邮件每天收到 200 封——大部分是假阳性（数据其实在，只是可见性延迟），但其中有 2-3 次是真的数据丢失。

他向王工请教，发现核心问题在**一致性级别（Consistency Level）**——Milvus 默认的 `Bounded` 一致性允许一定程度的"刚写入不可见"，用于换取更高的吞吐。而小刘需要的"写入立即可见"应该用 `Strong` 一致性。

更深入的问题是：不同组件内部有自己的时钟——RootCoord 的 Timestamp Oracle、DataNode 的消费位点、QueryNode 的加载状态。这些时钟不同步时，就会出现"Proxy 说写入成功，但 QueryNode 还没加载到这段数据"的时间窗口。

## 2. 项目设计（剧本式交锋对话）

**第一幕：四种一致性级别——Strong/Bounded/Eventually/Session**

*（小刘在 Grafana 上指着一条"写入成功但搜索为 0"的线）*

**小胖**（盯着监控）："写入成功了，数据量也涨了，但就是搜不到——Milvus 在骗我？"

**大师**："Milvus 没有骗你，是你在用默认的 Bounded 一致性——这是一个'性能换可见性'的取舍。"

**大师**（在白板上画出四条线）：

```
Strong一致性          Bounded一致性         Eventually一致性      Session一致性
─────────────────    ─────────────────    ─────────────────    ────────────────
写入 ──→ 立即可见   │  写入 ──→ ts秒后可  │  写入 ──→ 不知何   │  写入 ──→ 同一
       (等所有副本   │         见(默认5s) │         时可见     │         Session
        确认)       │                    │                    │         内可见
                    │                    │                    │
    强一致 慢         │    弱一致 快        │    最终一致 最快    │    读写绑定
```

| 一致性级别 | 保证 | 延迟 | 适用场景 |
|-----------|------|------|---------|
| **Strong** | 写入立即可见，所有副本一致 | 高（等所有 QueryNode 确认） | 金融交易、库存扣减 |
| **Bounded**（默认） | 写入后 GuaranteeTimestamp 秒内可见 | 中（默认 5s 内） | 大多数在线搜索 |
| **Eventually** | 写入最终可见（无时间保证） | 低 | 离线分析、日志搜索 |
| **Session** | 同一 Session 内写入立即可见 | 类似 Strong | 用户自己写的数据自己立刻查 |

**大师**："代码中的设置方式——"

```python
# 方式 1: 全局设置（连接级别）
connections.connect(host="localhost", port="19530",
                    consistency_level="Strong")

# 方式 2: 单次搜索设置
collection.search(..., consistency_level="Strong")

# 方式 3: 通过 GuaranteeTimestamp 精确控制（Bounded 模式）
from pymilvus import utility
ts = utility.get_query_segment_info(collection.name)[0].timestamp
collection.search(..., guarantee_timestamp=ts)
```

> **技术映射**：Strong = 银行柜台（每笔都确认到账才给你回执）；Bounded = 快递签收（承诺 5 秒内录入系统）；Eventually = 平信邮寄（不知道什么时候到，但一定会到）；Session = 同一个人操作的连续性。

**第二幕：Timestamp Oracle 与 Guarantee Timestamp**

**小白**："那 Bounded 一致性里的 'Guarantee Timestamp' 是什么？和普通时间戳有什么区别？"

**大师**："这是 Milvus 一致性模型的核心——**逻辑时钟**。"

```
Milvus 中的时间戳体系:

    物理时间: 2024-05-17 14:30:00.123
    逻辑时间: Timestamp Oracle (TSO) 分配的单调递增 uint64
    
RootCoord ──→ 分配 TSO ──→ Proxy 用 TSO 标记每条写入数据
    
    Guarantee Timestamp (GTS):
    搜索时告诉系统 "我需要看到 GTS 之前所有写入的数据"
    系统保证所有 ts ≤ GTS 的写入都已完成并可见
    
    ┌──────────────────────────────────────┐
    │ 写入时间线:                           │
    │ T1: 写入数据A (ts=1000)              │
    │ T2: 写入数据B (ts=1001)              │
    │ T3: 搜索 (GTS=1001)                  │
    │   → 保证能看到 A 和 B                │
    │ T4: 搜索 (GTS=1000)                  │
    │   → 只能看到 A（看不到 B）           │
    └──────────────────────────────────────┘
```

**大师**："不同一致性级别的本质区别就是 GTS 的取值策略——"

| 一致性 | GTS 取值 | 含义 |
|--------|---------|------|
| Strong | `latest_tso`（最新 TSO） | 等所有正在进行的写入完成 |
| Bounded | `latest_tso - grace_period`（默认 5 秒前） | 容忍 5 秒内的写入不可见 |
| Eventually | `0`（最早） | 不保证任何时效 |
| Session | 该 Session 最后写入的 TSO | 保证本 Session 内的写入可见 |

> **技术映射**：TSO = 银行柜台的取号机（每个人都拿到一个唯一的递增号）；GTS = "我要看 50 号以前的所有交易记录"；Strong = 必须等到最新叫号；Bounded = 可以接受 5 个号之前的记录。

**第三幕：不同业务场景下的一致性选择**

**小胖**："那我这个实时推荐应该用 Strong 还是 Bounded？"

**大师**："给你一份选择决策树——"

```
业务场景 → 一致性选择:

需要写入立即可见？ ──Yes──→ 延迟能否接受 +50%？ ──Yes──→ Strong
    │                              │
    │                              └──No──→ 降低搜索目标或接受 Bounded(1s)
    │
    └──No──→ 最大容忍不可见时长？
                │
                ├── 1s ──→ Bounded + guarantee_timestamp=1
                ├── 5s ──→ Bounded（默认）
                ├── 60s ──→ Bounded + guarantee_timestamp=60
                └── 无要求 ──→ Eventually

具体场景:
  股票交易/支付:     Strong
  电商搜索:          Bounded(默认 5s)  // 用户搜不到刚上架的商品 5 秒后就有了
  实时推荐Feed:      Bounded(1s)       // 用户行为 1 秒内影响推荐
  离线数据挖掘:      Eventually
  用户个人中心:      Session           // 用户自己修改的资料自己立即可见
```

**大师**："如果你的实时推荐对延迟敏感但又能接受 1 秒的可见性延迟，推荐 Bounded + guarantee_timestamp=1。这样既能获得接近 Strong 的体验，又不用付出 Strong 的全同步成本。"

**小白**："那 Strong 到底有多慢？"

**大师**："比 Bounded 慢 30-80%，因为 Strong 需要 Proxy 等待所有相关 QueryNode 确认 TSO 已推进到最新值。在 QueryNode 数量多（>10 个）的集群中，Strong 的额外开销更明显。"

> **技术映射**：实时推荐 = 越快越好但能接受 1 秒延迟（Bounded 1s）；支付 = 必须精确即时（Strong）；用户修改个人资料 = 自己看得到就行（Session）。

```python
from pymilvus import CollectionSchema, FieldSchema, DataType

COLL_NAME = "consistency_test"
```

#### 步骤 2：一致性级别对比实验

```python
# step2_consistency_compare.py
"""对比不同一致性级别下写入后的搜索可见性"""
import time
import numpy as np
from pymilvus import Collection

collection = Collection("consistency_test")
DIM = 128

results = {"Strong": [], "Bounded": [], "Eventually": []}

for level in results.keys():
    visible_count = 0
    latencies = []
    
    for i in range(100):
        # 写入一条唯一数据
        unique_id = int(time.time() * 1000000) + i
        collection.insert([
            [unique_id],
            [np.random.rand(DIM).astype(np.float32).tolist()],
            [f"test_{level}_{i}"],
        ])
        
        # 立即搜索（不 Flush）
        t0 = time.time()
        r = collection.search(
            data=[np.random.rand(DIM).astype(np.float32).tolist()],
            anns_field="vec",
            param={"metric_type": "COSINE", "params": {"ef": 16}},
            expr=f"id == {unique_id}",
            limit=1,
            consistency_level=level,  # ★ 关键参数
            output_fields=["id"]
        )
        latencies.append((time.time() - t0) * 1000)
        
        if len(r[0]) > 0:
            visible_count += 1
    
    results[level] = {
        "visible_rate": f"{visible_count}%",
        "avg_latency_ms": round(np.mean(latencies), 2),
        "p95_latency_ms": round(np.percentile(latencies, 95), 2),
    }

print("=" * 60)
print("一致性级别对比实验 (写入后立即搜索)")
print("=" * 60)
print(f"{'级别':<15} {'可见率':<10} {'平均延迟':<12} {'P95延迟'}")
print("-" * 50)
for level, r in results.items():
    print(f"{level:<15} {r['visible_rate']:<10} {r['avg_latency_ms']:<10.1f}ms {r['p95_latency_ms']:.1f}ms")

print(f"\n> Strong: 最高可见率，最高延迟")
print(f"> Bounded: 中等可见率（默认5s窗口），中等延迟")
print(f"> Eventually: 最低可见率（刚写入大概率搜不到），最低延迟")
```

#### 步骤 4：写入-搜索可见性端到端验证

```python
# step4_e2e_visibility.py
"""端到端验证：写入后多久能搜到？"""
import time
import numpy as np
from pymilvus import Collection

collection = Collection("consistency_test")

DIM = 128
test_runs = 50

print("=" * 60)
print("写入→搜索可见性 端到端验证")
print("=" * 60)
print(f"测试 {test_runs} 次, 每次写入后立即搜索")

for level, desc in [("Strong", "立即可见"), ("Bounded", "5秒内可见"),
                     ("Eventually", "最终可见(可能延迟)")]:
    visible = 0
    latencies = []
    
    for i in range(test_runs):
        uid = int(time.time() * 1000000) + i
        # 写入
        collection.insert([
            [uid],
            [np.random.rand(DIM).astype(np.float32).tolist()],
            [f"visibility_{i}"],
        ])
        
        # 立即搜索
        t0 = time.time()
        r = collection.search(
            data=[np.random.rand(DIM).astype(np.float32).tolist()],
            anns_field="vec",
            param={"metric_type": "COSINE", "params": {"ef": 8}},
            expr=f"id == {uid}",
            limit=1,
            consistency_level=level,
            output_fields=["id"]
        )
        latencies.append((time.time() - t0) * 1000)
        if len(r[0]) > 0:
            visible += 1
    
    print(f"  {level:<12} ({desc:<15}): "
          f"可见率={visible}/{test_runs} ({visible*100//test_runs}%), "
          f"平均延迟={np.mean(latencies):.1f}ms")

print(f"\n> Strong: 写入后立即可见（可见率 ~100%）")
print(f"> Bounded: 依赖 GTS 窗口，5 秒以内写入的数据可能不可见")
print(f"> Eventually: 不保证可见性时延，刚写入大概率搜不到")
```

```python
# Session 一致性需要建立连接时指定

    [uid],
    [np.random.rand(128).astype(np.float32).tolist()],
    ["session_test"],

print(f"  Session 内搜索: {'可见' if len(r[0]) > 0 else '不可见'}")
```

## 4. 项目总结

### 4.1 一致性级别选择速查

| 业务需求 | 推荐级别 | 配置示例 |
|---------|---------|---------|
| 支付/交易/库存 | Strong | `consistency_level="Strong"` |
| 在线搜索（容忍数秒延迟） | Bounded（默认） | 不设（默认 = Bounded 5s） |
| 实时推荐/个性化 | Bounded + GTS=1 | `guarantee_timestamp=now()-1` |
| 个人中心/用户设置 | Session | `consistency_level="Session"` |
| 后台分析/报表 | Eventually | `consistency_level="Eventually"` |

### 4.2 注意事项

- **Strong 不是银弹**：集群越大（QueryNode 越多），Strong 的延迟惩罚越大。
- **Bounded 的默认 5 秒不是绝对的**：如果 DataNode 消费慢，5 秒内数据可能仍然不可见。
- **Session 是 PyMilvus 特有的**：Java/Go SDK 的 Session 实现可能不同，使用前确认。

### 4.3 思考题

1. 如果 DataNode 消费 MQ 的速度跟不上写入速度，即使 Consistency=Strong，数据也不可见——为什么？Strong 不是等所有副本同步吗？
2. 在微服务架构中，如果写入服务（Service A）和搜索服务（Service B）是两个独立进程，如何保证 A 写入的数据被 B 搜索到？Session 一致性在此场景下有意义吗？
