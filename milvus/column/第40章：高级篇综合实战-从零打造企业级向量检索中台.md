# 第40章：【高级篇综合实战】从零打造企业级向量检索中台

> **定位**：融会贯通高级篇全部知识，产出可交付的生产级平台方案。
> **版本**：Milvus 2.5.x
> **源码关联**：全模块（cmd/、internal/、pkg/、deployments/）

---

## 1. 项目背景

某大型互联网公司有 15 个业务线共同使用 Milvus 做向量检索。现状是各业务线自己搭自己的 Milvus 实例——导致"15 个 Milvus 孤岛"：资源利用率低（有的 QPS 爆满有的空转）、运维成本高（15 套需要分别维护）、安全管控差（无统一权限管理）、成本不可控（各管各的花钱）。

CTO 要求架构团队建设"企业级向量检索中台"——用一套基础设施服务所有业务线，支持 50+ 业务方、十亿级向量规模、核心检索 P99 < 500ms、具备灾备恢复能力、可定制调度策略。

这是整个专栏的终章——融汇贯通基础篇（部署/建模/搜索）、中级篇（分布式/调优/监控/RAG）和高级篇（源码/引擎/SRE/极端规模）的全部知识。

| 设计决策 | 选择 | 理由 |
|---------|------|------|
| **多集群而非单集群** | 按业务等级分 4 个集群 | 避免核心业务被非核心拖垮 |
| **租户隔离** | Partition Key + RBAC | 50 租户 < 4096 上限，自动路由 |
| **索引选型** | 按业务分级 | 核心=HNSW，一般=IVF_SQ8，离线=DISKANN |
| **高可用** | Replica=2（核心） | 换 2 倍内存成本，得 0 秒故障恢复 |
| **一致性** | Bounded 5s | 多数业务容忍 5s 延迟 |

**第二幕：定制调度策略与源码扩展**

**小白**："中台需要定制调度策略——比如让大客户的搜索优先执行。Milvus 能做吗？"

**大师**："可以，但需要改源码。Milvus 的 QueryCoord 调度器支持扩展——"

```go
// 定制调度策略示例: 优先级调度

// 在 internal/querycoordv2/load_balancer.go 中添加

type PriorityLoadBalancer struct {
    defaultBalancer LoadBalancer
}

// GetLeastLoadedNode 覆盖默认的负载均衡
// 高优先级租户优先分配低负载 QN
func (p *PriorityLoadBalancer) GetLeastLoadedNode(
    ctx context.Context, 
    collectionID int64,
    ) (*QueryNode, error) {
    
    // 查询租户优先级
    priority := p.getTenantPriority(collectionID)
    
    if priority == HIGH {
        // 高优先级: 选负载最低的 QN, 且不允许与低优先级共用
        return p.getDedicatedLowLoadNode()
    } else {
        // 低优先级: 可以共用, 选负载不超过 50% 的 QN
        return p.getSharedNode(maxLoad=0.5)
    }
}
```

**大师**："可定制的扩展点——"

| 扩展点 | 源码位置 | 定制内容 |

| 索引选择 | `querynodev2/search.go` | 按数据量自动选择索引类型 |

# step2_platform_sdk.py
"""中台统一接入 SDK"""
from pymilvus import Collection, connections, utility

class PlatformVectorSearch:
    """企业级向量检索中台 SDK"""
    
    def __init__(self, platform_endpoint: str, tenant_id: str,
                 api_key: str):
        self.tenant = tenant_id
        self.api_key = api_key
        connections.connect(host=platform_endpoint, port=19530,
                          user=tenant_id, password=api_key)
    
    def create_collection(self, name: str, dim: int, 
                          index_type: str = None):
        """创建 Collection（中台自动分配资源组）"""
        # 中台根据 index_type 自动分配到对应集群
        pass
    
    def search(self, collection: str, query_vector: list,
               top_k: int = 10, filters: dict = None) -> dict:
        """统一搜索 API（自动路由到正确的 Milvus 集群）"""
        # 中台根据 Collection 自动路由到对应集群和 Resource Group
        coll = Collection(collection)
        
        expr = self._build_expr(filters)
        
        results = coll.search(
            data=[query_vector],
            anns_field="vec",
            param={"metric_type": "COSINE", "params": {"ef": 64}},
            expr=f'tenant_id == "{self.tenant}"' if expr is None 
                 else f'tenant_id == "{self.tenant}" and {expr}',
            limit=top_k,
            output_fields=["*"]
        )
        
        return self._format_results(results)
    
    def _build_expr(self, filters: dict) -> str:
        """构造过滤表达式"""
        parts = []
        for k, v in (filters or {}).items():
            if isinstance(v, (int, float)):
                parts.append(f"{k} == {v}")
            elif isinstance(v, str):
                parts.append(f'{k} == "{v}"')
        return " and ".join(parts)
    
    def _format_results(self, results) -> dict:
        """格式化搜索结果"""
        items = []
        for batch in results:
            for hit in batch:
                item = {"id": hit.id, "score": round(hit.distance, 4)}
                if hit.entity:
                    for field in hit.entity.fields:
                        item[field] = hit.entity.get(field)
                items.append(item)
        return {"results": items, "count": len(items)}

# 使用
platform = PlatformVectorSearch(
    platform_endpoint="platform.internal.com",
    tenant_id="ecommerce-team",
    api_key="sk-xxx"
)

results = platform.search(
    collection="product_search",
    query_vector=[0.1] * 768,
    top_k=10,
    filters={"category": "electronics", "in_stock": True}
)

for item in results["results"]:
    print(f"  {item['title']} (score={item['score']:.3f})")

### 4.2 最终思考题

1. 如果向量检索中台需要支持 500 个业务方（远超 Partition 4096 上限），架构应该如何演进？
2. 在 AI 日新月异的今天（新 Embedding 模型、新索引算法、新硬件），向量检索中台如何保持架构的"可进化性"——既不过度设计，又不至于每次变革都推倒重来？

本专栏基于 Milvus 2.5.x 编写。Milvus 是一个快速迭代的开源项目，本专栏提供的是方法论的"渔"，具体 API 和配置请以最新官方文档为准。
