# 第21章：稠密向量、稀疏向量与混合检索

> **定位**：让 Milvus 支持更贴近搜索引擎的召回能力。
> **版本**：Milvus 2.5.x
> **源码关联**：internal/proxy/hybrid_search.go、internal/querynodev2/hybrid_search.go、internal/core/src/index/sparse/

---

## 1. 项目背景

某电商搜索团队发现一个尴尬现象：用户搜"iPhone 15 手机壳"时，纯语义搜索返回了一堆"三星手机壳""华为手机壳"——语义上确实相似（都是"手机+壳"），但用户明确要的是 iPhone 15 型号。与此同时，搜"轻便露营折叠椅 便携"返回的结果中，Top3 没有一个包含"折叠椅"这个关键词，用户觉得搜索结果"不相关"。

团队分析发现这是**纯稠密向量（Dense Vector）搜索的天然缺陷**：Embedding 模型擅长理解语义（"车"和"汽车"能匹配），但弱化了精确关键词匹配（"iPhone 15"和"iPhone 14"在向量空间里距离很近，区分度不足）。

运维同学提议"加个 ES 做关键词过滤"——但这意味着一套搜索要调两个系统（ES 做关键词过滤 + Milvus 做语义召回），复杂度翻倍。算法同学调研后发现：Milvus 2.4+ 原生支持了**稀疏向量（Sparse Vector）和混合检索（Hybrid Search）**，可以在一次请求中同时做语义召回和关键词匹配，并在 Milvus 内部完成融合排序。

本章将实现一个"关键词 + 语义"的混合检索系统，对比纯向量召回与混合召回的效果差异。

## 2. 项目设计（剧本式交锋对话）

**第一幕：Dense vs Sparse——为什么两个比一个好**

*（搜索产品经理在会上拍桌子："iPhone 15 手机壳"为什么搜出三星的？）*

**小胖**（困惑地）："我就想搜'iPhone 15 手机壳'，Milvus 给我一堆'三星'和'华为'——这不合理吧！"

**大师**："这不是 Milvus 的错，是纯语义搜索的边界。Dense Vector（稠密向量）把'iPhone 15 手机壳'编码成 1024 维的语义向量——它能理解这是'一个电子产品的保护套'，但区分不出是 iPhone 还是三星。就像你描述一个人的'高个子、戴眼镜、穿西装'——听起来像很多人。"

**大师**（在白板上画）：

```
Dense Vector（稠密向量）           Sparse Vector（稀疏向量）
───────────────────────           ─────────────────────────
"iPhone 15 手机壳"                "iPhone 15 手机壳"
    ↓ Embedding Model                ↓ BM25 / SPLADE 分词
[0.12, 0.83, -0.45, ..., 0.67]   {"iPhone": 2.3, "15": 1.8,
 (1024维浮点数，稠密)                "手机壳": 2.1, ...}
                                    (大部分为0，只有关键词有值)
                                    
擅长: 语义理解                     擅长: 精确关键词匹配
弱点: 无法精确区分专业术语            弱点: 不懂同义词/语义变体
```

| 对比维度 | Dense Vector | Sparse Vector |
|---------|-------------|---------------|
| 表示方式 | 固定维度浮点数组（如1024维） | 高维稀疏字典（大部分为0） |
| 生成方式 | Embedding 模型（BERT/CLIP） | BM25 / TF-IDF / SPLADE |
| 语义理解 | 强（"汽车"="轿车"） | 弱（只做字面匹配） |
| 关键词区分 | 弱（"iPhone15"和"iPhone14"很近） | 强（严格按词匹配） |
| 跨语言能力 | 强（多语言模型） | 弱（需要同语言分词器） |

**大师**："混合检索的精髓就是——**用 Dense 保证查全（语义覆盖），用 Sparse 保证查准（关键词对齐）。**"

> **技术映射**：Dense Vector = 画像描述（能说出一个人的气质和特征）；Sparse Vector = 身份证号（精确到人）；混合检索 = 拿着画像和身份证号一起找人，又快又准。

**第二幕：Hybrid Search 的执行流程与 Rerank 策略**

**小白**："那混合检索在 Milvus 内部是怎么执行的？是两个检索并行吗？"

**大师**："四步流程——"

```
Hybrid Search 完整流程:

Step 1: 生成双向量
  查询文本 "iPhone 15 手机壳"
      ├→ Dense Model (BGE-M3) → dense_vec (1024维)
      └→ Sparse Model (BM25/SPLADE) → sparse_vec (词汇表大小维)

Step 2: 双路并行检索
  ┌─ Dense Search ─┐     ┌─ Sparse Search ─┐
  │ HNSW索引       │     │ 倒排索引 (WAND)  │
  │ 召回 TopK=20   │     │ 召回 TopK=20    │
  └───────┬────────┘     └────────┬────────┘
          │                       │
Step 3: 归一化分值
          ├── Dense scores → 归一化到 [0, 1]
          └── Sparse scores → 归一化到 [0, 1]

Step 4: 融合排序（Rerank）
          ┌─────────────────────────────┐
          │ 融合策略:                     │
          │  WeightedRanker: 加权求和      │
          │     score = w1*dense + w2*sparse │
          │  RRFRanker: 倒数排名融合        │
          │     score = Σ 1/(k + rank_i)  │
          └─────────────────────────────┘
                    ↓
            最终 TopK 结果返回
```

**大师**："两种融合策略的选择——"

| 策略 | 公式 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **WeightedRanker** | `score = w₁×dense + w₂×sparse` | 可调节权重，灵活 | 需要调 w₁/w₂ | 已知语义和关键词的重要性比例 |
| **RRFRanker** | `score = Σ 1/(k + rank)` | 无需调权，鲁棒 | 无法微调 | 不确定两者权重的通用场景 |

**小胖**："那 w₁ 和 w₂ 怎么设？"

**大师**："这取决于你的搜索场景——商品搜索偏语义（w₁=0.7, w₂=0.3），文档搜索偏关键词（w₁=0.4, w₂=0.6），通用场景 w₁=0.5, w₂=0.5 起步。"

> **技术映射**：双路检索 = 两个侦探分别从"长相"和"身份证号"两个线索查案；Rerank = 两个侦探碰头后综合判断谁最可能是目标；WeightedRanker = 一个侦探的意见更重要（加权）；RRFRanker = 两个侦探平权投票。

**第三幕：BM25/SPLADE/BGE-M3——稀疏模型选型**

**小胖**："那稀疏向量用什么模型生成？"

**大师**："三种主流方案——"

| 模型 | 生成方式 | 优点 | 缺点 | 推荐 |
|------|---------|------|------|------|
| **BM25** | 传统统计算法（TF-IDF 升级版） | 零推理成本、可解释 | 不懂语义、需要分词 | 快速起步 |
| **SPLADE** | 神经网络（BERT-based） | 端到端、效果好 | 推理需要 GPU、维度高 | 精度优先 |
| **BGE-M3** | 单一模型同时输出 Dense+Sparse | 一个模型出双向量！ | 配套生态尚在完善 | **推荐首选** |

**大师**："BGE-M3 是目前最省事的方案——一个模型同时输出 1024 维 Dense Vector 和词汇表大小的 Sparse Vector，不需要维护两个模型的版本一致性。"

```python
# BGE-M3 同时生成 Dense + Sparse
from FlagEmbedding import BGEM3FlagModel
model = BGEM3FlagModel("BAAI/bge-m3", use_fp16=True)

output = model.encode(["iPhone 15 手机壳"], return_dense=True, return_sparse=True)
dense_vec = output["dense_vecs"][0]    # 1024维稠密向量
sparse_vec = output["lexical_weights"][0]  # 稀疏词权: {"iPhone": 0.3, "15": 0.2, ...}
```

### 3.3 分步实现

#### 步骤 1：创建支持双向量字段的 Collection

```python
# step1_hybrid_collection.py
"""创建支持 Dense + Sparse 双向量字段的 Collection"""
from pymilvus import connections, Collection, utility
from pymilvus import CollectionSchema, FieldSchema, DataType

connections.connect(host="localhost", port="19530")

COLL_NAME = "hybrid_search_demo"
if utility.has_collection(COLL_NAME):
    utility.drop_collection(COLL_NAME)

fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="text", dtype=DataType.VARCHAR, max_length=1024),
    # Dense Vector: 稠密语义向量
    FieldSchema(name="dense_vec", dtype=DataType.FLOAT_VECTOR, dim=1024),
    # Sparse Vector: 稀疏关键词向量
    FieldSchema(name="sparse_vec", dtype=DataType.SPARSE_FLOAT_VECTOR),
    FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64),
]

schema = CollectionSchema(fields, description="混合检索实战")
collection = Collection(COLL_NAME, schema)

# 为稠密向量建索引
collection.create_index("dense_vec", {
    "index_type": "HNSW", "metric_type": "COSINE",
    "params": {"M": 16, "efConstruction": 200}
})

# 为稀疏向量建索引
collection.create_index("sparse_vec", {
    "index_type": "SPARSE_WAND", "metric_type": "IP"
})

utility.wait_for_index_building_complete(COLL_NAME, timeout=120)
collection.load()
print(f"Collection {COLL_NAME} 创建完成，支持混合检索")
```

#### 步骤 2：生成 Dense+Sparse 双向量并写入

```python
# step2_dual_embedding.py
"""生成双向量并写入 Milvus"""
from FlagEmbedding import BGEM3FlagModel
from pymilvus import Collection
import numpy as np

model = BGEM3FlagModel("BAAI/bge-m3", use_fp16=True)
collection = Collection("hybrid_search_demo")

# 模拟商品数据
products = [
    "iPhone 15 Pro Max 手机壳 防摔硅胶保护套",
    "三星 Galaxy S24 手机壳 透明防摔",
    "华为 Mate 60 手机壳 超薄磁吸",
    "iPhone 15 钢化膜 全屏覆盖高清",
    "户外露营折叠椅 便携超轻铝合金",
    "便携式野餐垫 防水防潮大号",
]

embeddings = model.encode(products, return_dense=True, return_sparse=True)

# 写入数据
entities = []
for i, (text, dense, sparse) in enumerate(
    zip(products, embeddings["dense_vecs"], embeddings["lexical_weights"])
):
    entities.append({
        "id": i + 1,
        "text": text,
        "dense_vec": dense.tolist(),
        "sparse_vec": sparse,  # {词: 权重}
        "category": "电子产品" if "手机" in text else "户外用品",
    })

collection.insert([
    [e["id"] for e in entities],
    [e["text"] for e in entities],
    [e["dense_vec"] for e in entities],
    [e["sparse_vec"] for e in entities],
    [e["category"] for e in entities],
])

collection.flush()
print(f"已写入 {len(entities)} 条数据")
```

#### 步骤 3：混合检索 vs 纯检索对比

```python
# step3_hybrid_vs_pure.py
"""对比纯 Dense、纯 Sparse、Hybrid 三种检索效果"""
from pymilvus import Collection, AnnSearchRequest, WeightedRanker, RRFRanker
from FlagEmbedding import BGEM3FlagModel

model = BGEM3FlagModel("BAAI/bge-m3", use_fp16=True)
collection = Collection("hybrid_search_demo")

# 测试查询
queries = [
    "iPhone 15 手机壳防摔",          # 关键词明确
    "户外轻便折叠椅子",              # 语义为主
]

for query_text in queries:
    q_emb = model.encode([query_text], return_dense=True, return_sparse=True)
    q_dense = q_emb["dense_vecs"][0].tolist()
    q_sparse = q_emb["lexical_weights"][0]

    print(f"\n{'='*60}")
    print(f"查询: {query_text}")
    print(f"{'='*60}")
    print(f"{'方法':<20} {'结果':<50} {'分数':<10}")

    # —— 方法 1: 纯 Dense ——
    r_dense = collection.search(
        data=[q_dense], anns_field="dense_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        limit=3, output_fields=["text"]
    )
    print(f"{'纯Dense(语义)':<20}", end="")
    for hit in r_dense[0]:
        print(f" {hit.entity.get('text')[:30]}({hit.distance:.3f})  ", end="")
    print()

    # —— 方法 2: 纯 Sparse ——
    r_sparse = collection.search(
        data=[q_sparse], anns_field="sparse_vec",
        param={"metric_type": "IP"},
        limit=3, output_fields=["text"]
    )
    print(f"{'纯Sparse(关键词)':<20}", end="")
    for hit in r_sparse[0]:
        print(f" {hit.entity.get('text')[:30]}({hit.distance:.3f})  ", end="")
    print()

    # —— 方法 3: Hybrid (Weighted) ——
    dense_req = AnnSearchRequest(data=[q_dense], anns_field="dense_vec",
                                  param={"metric_type": "COSINE", "params": {"ef": 64}}, limit=10)
    sparse_req = AnnSearchRequest(data=[q_sparse], anns_field="sparse_vec",
                                   param={"metric_type": "IP"}, limit=10)
    r_hybrid_w = collection.hybrid_search(
        reqs=[dense_req, sparse_req],
        rerank=WeightedRanker(0.5, 0.5),  # 语义和关键词各50%
        limit=3, output_fields=["text"]
    )
    print(f"{'Hybrid(语义50+词50)':<20}", end="")
    for hit in r_hybrid_w[0]:
        print(f" {hit.entity.get('text')[:30]}({hit.distance:.3f})  ", end="")
    print()

    # —— 方法 4: Hybrid (RRF) ——
    r_hybrid_r = collection.hybrid_search(
        reqs=[dense_req, sparse_req],
        rerank=RRFRanker(k=60),
        limit=3, output_fields=["text"]
    )
    print(f"{'Hybrid(RRF)':<20}", end="")
    for hit in r_hybrid_r[0]:
        print(f" {hit.entity.get('text')[:30]}({hit.distance:.3f})  ", end="")
    print()
```

**预期输出**：
```
查询: "iPhone 15 手机壳防摔"
纯Dense(语义)         三星 Galaxy S24 手机壳...  iPhone 15 Pro Max 手机壳...  华为 Mate 60 手机壳...
纯Sparse(关键词)      iPhone 15 Pro Max 手机壳...  iPhone 15 钢化膜...  三星 Galaxy S24 手机壳...
Hybrid(语义50+词50)   iPhone 15 Pro Max 手机壳...  iPhone 15 钢化膜...  三星 Galaxy S24 手机壳...
```

## 4. 项目总结

### 4.1 三种检索方式对比

| 维度 | 纯 Dense | 纯 Sparse | Hybrid |
|------|---------|----------|--------|
| 语义理解 | 强 | 弱 | 强 |
| 关键词区分 | 弱 | 强 | 强 |
| 跨语言 | 强 | 弱 | 取决于 Dense 模型 |
| 推理成本 | 中 | 低（BM25）/ 中（SPLADE） | 双份 |
| 存储成本 | 中 | 低（稀疏存非零项） | 高（双向量字段） |
| 召回率 | 中 | 中 | **高** |

### 4.2 适用场景

- **电商搜索**：Hybrid 解决"iPhone 15"型号精确匹配 + "轻便"语义理解
- **知识库 RAG**：Hybrid 解决专业术语匹配 + 同义词覆盖
- **法律/医疗检索**：Sparse 保证法条/药名精确匹配
- **多语言搜索**：Dense 负责跨语言语义对齐

**不适用场景**：纯数值计算搜索（不需要语义）、极实时场景（双路检索延迟翻倍）。

### 4.3 注意事项

- **Sparse Vector 的 Collection 需要单独建 SPARSE_WAND 索引**，不能用 HNSW。
- **BGE-M3 内存需求较高**（~2GB GPU 内存），小批量推理可用 CPU。
- **Hybrid Search 的 RRFRanker 不需要调权**，适合快速上线。
