## 1. 项目背景

运维工程师老周所在团队计划将 Milvus 从测试环境推上生产。业务方预估初期 500 万条商品向量，日均搜索 QPS 300，P95 延迟要求 < 50ms。老周被问到三个问题：

1. 需要几台服务器？什么配置？
2. 500 万数据占多少内存和磁盘？
3. 300 QPS 能撑住吗？峰值能到多少？

老周一开始直接查了 Milvus 官方文档里的"推荐配置"——但那是针对通用场景的，没法回答"我们的数据和搜索模式需要什么配置"。团队内部开始争论：有人说 2 台 32GB 就够了，有人说至少 4 台 64GB，有人说还得配 GPU 做索引……

更关键的是，老周在测试环境跑了压测后发现——1000 QPS 并发下，P95 延迟不是 50ms 而是 350ms，远超业务要求。但究竟是哪个环节慢？是 Proxy 瓶颈、QueryNode 瓶颈、还是 Embedding 模型推理慢？没有数据支撑，根本无法定位。

本章将建立一套从"算容量"到"跑压测"到"出报告"的方法论，让你能回答领导/客户问的"需要多少台机器"。

## 2. 项目设计（剧本式交锋对话）

**第一幕：数据量、维度、索引类型如何影响资源**

*（老周在 Excel 里列出了 5 种不同的服务器配置方案，但不知道选哪个）*

**小胖**（看傻眼了）："一个 Milvus 而已，需要这么复杂的资源规划？MySQL 我都是 4C8G 起步随便跑——"

**大师**："Milvus 和 MySQL 的资源模型完全不同。MySQL 的资源=磁盘+Buffer Pool。Milvus 的资源=QueryNode 内存 + 对象存储 + IndexNode CPU + etcd 磁盘。给你一个速算公式——"

**大师**（在白板上写下公式）：

```
┌─────────────────────────────────────────────────────────────┐
│              Milvus 资源估算速算公式                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 原始向量数据大小                                          │
│     = N × D × 4 bytes                                       │
│     N = 向量条数, D = 维度, 4 = float32 每个分量 4 字节       │
│     例: 500万 × 1024维 × 4B = 20.48 GB                      │
│                                                             │
│  2. HNSW 索引内存                                            │
│     ≈ 原始向量 × 1.8 (图结构开销约 80%)                      │
│     例: 20.48 GB × 1.8 = 36.9 GB                            │
│                                                             │
│  3. 磁盘存储（对象存储）                                       │
│     ≈ 原始向量 × 1.5 (Binlog + Index + Delta Log)            │
│     例: 20.48 GB × 1.5 = 30.7 GB                            │
│                                                             │
│  4. QueryNode 内存（加载到内存的集合）                          │
│     ≈ 索引内存 + Segment 元数据开销 (~10%)                    │
│     例: 36.9 GB × 1.1 = 40.6 GB                             │
│                                                             │
│  5. etcd 存储                                                │
│     ≈ 元数据条数 × 1 KB (粗略)                                │
│     例: 5000 Segment × 1KB = 5 MB (通常不是瓶颈)              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

| 索引类型 | 内存倍率（相对原始数据） | 示例：500万×1024维 |
|---------|---------------------|-------------------|
| FLAT | ×1.0（无图结构） | 20.4 GB |
| IVF_FLAT | ×1.1（聚类中心开销小） | 22.4 GB |
| IVF_SQ8 | ×0.3（量化压缩） | 6.1 GB |
| HNSW | ×1.5~2.0（图结构） | 30.6~40.8 GB |
| DISKANN | ×0.3（大部分在磁盘） | 6.1 GB（内存）+ 30 GB（磁盘） |

**小白**："那我用 IVF_SQ8 内存只有 HNSW 的 1/6——为什么不全用 IVF_SQ8？"

**大师**："因为召回率。IVF_SQ8 的量化压缩会损失精度，典型召回率 90-95%。HNSW 的召回率可达 99%+。如果你的业务对'搜不到正确结果'容忍度低，就得多掏内存。这是典型的**空间换精度**。"

> **技术映射**：原始向量 = 书的正文内容（最占地方）；HNSW 图结构 = 书本的目录和索引页（额外开销）；IVF_SQ8 = 书的摘要版（省地方但丢细节）；估算是按公式速算，精确值要实测。

**第二幕：压测维度设计与脚本编写**

**小胖**："那压测怎么跑？我只会用 `ab -n 1000` 打 HTTP——Milvus 是 gRPC，怎么压？"

**大师**："Milvus 压测有四个核心维度，每个都不能少——"

| 维度 | 变量 | 为什么重要 | 如何测试 |
|------|------|-----------|---------|
| **数据规模** | 1万/10万/100万/1000万 | 数据规模直接影响索引效率和搜索延迟 | 准备不同规模的数据集，分别测试 |
| **并发数** | 1/10/50/100/500 | 并发越高，排队延迟越明显 | 逐步提高并发，记录延迟变化 |
| **TopK** | 10/50/100/500 | TopK 越大，排序开销越大 | 固定其他变量，只变 TopK |
| **过滤条件** | 无过滤/简单过滤(price>100)/复杂过滤(3个AND) | 过滤条件增加 CPU 开销 | 对比有无过滤的延迟差异 |

**大师**："压测脚本的核心结构——"

```python
import time
import numpy as np
from concurrent.futures import ThreadPoolExecutor, as_completed
from pymilvus import Collection, connections

def search_latency_test(collection, query_vectors, concurrency=10,
                        total_requests=1000):
    """搜索延迟压测（多线程并发）"""
    results = []
    
    def single_search(qv):
        t0 = time.time()
        collection.search(
            data=[qv], anns_field="title_vec",
            param={"metric_type": "COSINE", "params": {"ef": 64}},
            limit=10
        )
        return (time.time() - t0) * 1000  # 毫秒
    
    with ThreadPoolExecutor(max_workers=concurrency) as executor:
        futures = [executor.submit(single_search, 
                   query_vectors[i % len(query_vectors)])
                   for i in range(total_requests)]
        
        for f in as_completed(futures):
            results.append(f.result())
    
    results = np.array(results)
    return {
        "avg": np.mean(results),
        "p50": np.percentile(results, 50),
        "p95": np.percentile(results, 95),
        "p99": np.percentile(results, 99),
        "max": np.max(results),
        "qps": total_requests / (sum(results) / 1000),
    }
```

**小白**："为什么不是只看平均值？P95 和 P99 有什么用？"

**大师**："平均值是最大的谎言。如果 100 次请求中 99 次是 1ms、1 次是 10 秒，平均值是 100ms——看起来还不错。但实际上那 1 次 10 秒的用户体验极差。**P95/P99 关注的是长尾延迟**，这才是真正影响用户体验的指标。"

> **技术映射**：并发 = 多少人同时来办事；P50 = 一半的人等了多久；P95 = 最慢的 5% 的人等了多久（这个数字才是用户真实感受）；QPS = 一小时能办多少人的事。

#### 步骤 1：容量估算脚本

```python
# step1_capacity_calc.py
"""Milvus 容量估算器"""
import math

class MilvusCapacityEstimator:
    """根据数据量和维度估算所需资源"""
    
    def __init__(self, vector_count: int, dim: int):
        self.N = vector_count
        self.D = dim
    
    def raw_vector_size_gb(self) -> float:
        """原始向量数据大小 (GB)"""
        return self.N * self.D * 4 / (1024 ** 3)
    
    def index_memory_gb(self, index_type: str = "HNSW") -> float:
        """索引内存占用 (GB)"""
        multipliers = {
            "FLAT": 1.0,
            "IVF_FLAT": 1.1,
            "IVF_SQ8": 0.3,
            "HNSW": 1.8,
            "DISKANN": 0.3,  # 内存部分
        }
        return self.raw_vector_size_gb() * multipliers.get(index_type, 1.8)
    
    def storage_size_gb(self) -> float:
        """对象存储占用 (GB)，约原始数据的 1.5 倍"""
        return self.raw_vector_size_gb() * 1.5
    
    def recommend_query_node_memory_gb(self, index_type: str = "HNSW",
                                       loaded_ratio: float = 1.0) -> float:
        """推荐 QueryNode 内存 (GB)，含 20% buffer"""
        return self.index_memory_gb(index_type) * loaded_ratio * 1.2
    
    def report(self) -> str:
        """生成容量估算报告"""
        raw = self.raw_vector_size_gb()
        return f"""
═══════════════════════════════════════════
  Milvus 容量估算报告
═══════════════════════════════════════════
  数据规模: {self.N:,} 条 × {self.D} 维
  
  原始向量数据:    {raw:.1f} GB
  HNSW 索引内存:   {self.index_memory_gb('HNSW'):.1f} GB (×1.8)
  IVF_SQ8 索引内存:{self.index_memory_gb('IVF_SQ8'):.1f} GB (×0.3)
  对象存储占用:     {self.storage_size_gb():.1f} GB (×1.5)
  
  推荐 QueryNode 内存 (HNSW, 全量加载):
    {self.recommend_query_node_memory_gb('HNSW'):.1f} GB (含 20% buffer)
═══════════════════════════════════════════
"""

# 使用
est = MilvusCapacityEstimator(vector_count=1_000_000, dim=768)
print(est.report())

# 不同索引类型对比
print("索引类型内存对比:")
for idx_type in ["FLAT", "IVF_FLAT", "IVF_SQ8", "HNSW", "DISKANN"]:
    mem = est.index_memory_gb(idx_type)
    print(f"  {idx_type:<12}: {mem:.2f} GB")
```

**预期输出**：

```python
# step2_benchmark.py
"""Milvus 搜索压测"""
import time
import random
import numpy as np
from concurrent.futures import ThreadPoolExecutor, as_completed
from pymilvus import Collection, connections
from sentence_transformers import SentenceTransformer

connections.connect(host="localhost", port="19530")
model = SentenceTransformer("all-MiniLM-L6-v2")
collection = Collection("product_search")
collection.load()

class SearchBenchmark:
    """搜索压测工具"""
    
    def __init__(self, collection, query_vectors):
        self.collection = collection
        self.query_vectors = query_vectors
    
    def run(self, concurrency: int, total_requests: int = 1000) -> dict:
        """一次压测运行"""
        latencies = []
        errors = 0
        
        def do_search(qv):
            try:
                t0 = time.time()
                self.collection.search(
                    data=[qv],
                    anns_field="title_vec",
                    param={"metric_type": "COSINE", "params": {"ef": 64}},
                    limit=10,
                    timeout=10,
                )
                return (time.time() - t0) * 1000, None
            except Exception as e:
                return 0, str(e)
        
        with ThreadPoolExecutor(max_workers=concurrency) as pool:
            futures = []
            for i in range(total_requests):
                qv = self.query_vectors[i % len(self.query_vectors)]
                futures.append(pool.submit(do_search, qv))
            
            for f in as_completed(futures):
                lat, err = f.result()
                if err:
                    errors += 1
                else:
                    latencies.append(lat)
        
        latencies = np.array(latencies)
        total_time = sum(latencies) / 1000
        return {
            "concurrency": concurrency,
            "total_requests": total_requests,
            "errors": errors,
            "error_rate": f"{errors/total_requests*100:.1f}%",
            "qps": total_requests / total_time if total_time > 0 else 0,
            "latency_ms": {
                "avg": np.mean(latencies),
                "min": np.min(latencies),
                "p50": np.percentile(latencies, 50),
                "p90": np.percentile(latencies, 90),
                "p95": np.percentile(latencies, 95),
                "p99": np.percentile(latencies, 99),
                "max": np.max(latencies),
            }
        }

# 准备查询向量
query_texts = [
    "户外露营折叠椅", "轻便蓝牙耳机", "家用收纳盒",
    "防水登山鞋", "大容量背包", "静音键盘",
]
query_vectors = model.encode(query_texts).tolist()

bench = SearchBenchmark(collection, query_vectors)

# 分并发级别测试
print("=" * 70)
print("Milvus 搜索压测报告")
print("=" * 70)
print(f"{'并发':<6} {'QPS':<10} {'P50':<10} {'P95':<10} {'P99':<10} {'Max':<10} {'错误率'}")
print("-" * 70)

for c in [1, 5, 10, 20, 50]:
    result = bench.run(concurrency=c, total_requests=500)
    lat = result["latency_ms"]
```

#### 步骤 3：Milvus Sizing Tool 思路

```python
# step3_sizing_tool.py
"""基于压测数据推算扩容方案"""
def sizing_recommendation(current_qps: float, current_p95: float,
                          target_qps: float, target_p95: float,
                          current_nodes: int) -> dict:
    """根据当前压测数据推算扩容方案
    
    假设条件：
    - 搜索延迟与 QPS 近似线性关系（在资源未饱和时）
    - 每个 QueryNode 承担 1/N 的查询负载
    """
    # QPS 缺口
    qps_gap = target_qps / current_qps
    
    # P95 是否超标
    p95_ok = current_p95 <= target_p95
    
    recommendations = []
    
    if not p95_ok:
        # P95 超标：需要增加 QueryNode 或用更快索引
        recommendations.append(
            f"当前 P95={current_p95:.1f}ms > 目标 {target_p95}ms"
        )
        # 估算：增加 QueryNode 能线性降低延迟（理想情况）
        needed_nodes = max(
            int(current_nodes * (current_p95 / target_p95)),
            current_nodes + 1
        )
        recommendations.append(
            f"建议 QueryNode: {current_nodes} → {needed_nodes} 台"
        )
    
    if qps_gap > 1.5:
        recommendations.append(
            f"当前 QPS={current_qps:.0f} < 目标 {target_qps:.0f}（差距 {qps_gap:.1f}x）"
        )
        qps_nodes = int(current_nodes * qps_gap)
        recommendations.append(
            f"为满足目标 QPS，建议至少 {qps_nodes} 台 QueryNode"
        )
    
    return {
        "current": {"qps": current_qps, "p95": current_p95, "nodes": current_nodes},
        "target": {"qps": target_qps, "p95": target_p95},
        "recommendations": recommendations,
    }

# 示例
result = sizing_recommendation(
    current_qps=100, current_p95=80,
    target_qps=300, target_p95=50,
    current_nodes=1
)
for r in result["recommendations"]:
    print(f"  - {r}")
```

## 4. 项目总结

### 4.1 优缺点对比

| 维度 | 公式估算 | 压测实测 | Sizing Tool |
|------|---------|---------|-------------|
| 速度 | 秒级 | 小时级 | 分钟级（基于压测数据） |
| 精度 | 中（±30%） | 高（±10%） | 中（取决于压测质量） |
| 适用阶段 | 项目初期选型 | 上线前验证 | 扩容规划 |

### 4.2 适用场景

- **项目启动前的容量规划**：需要几台机器、多少内存
- **上线前的性能验证**：300 QPS 真的能扛住吗
- **扩容决策**：业务增长到 1000 QPS 需要加几台
- **索引选型决策**：HNSW vs IVF_SQ8 的内存-召回率 tradeoff

**不适用场景**：极端精准预算（误差约 20-30%，最终以压测为准）。

### 4.3 注意事项

- **容量估算公式只作参考**：实际内存占用受索引参数、Segment 数量、碎片化等因素影响。
- **压测要用真实数据集**：随机的正态分布向量和真实 Embedding 模型的输出在搜索性能上有显著差异。
- **预热**：压测前先跑 100 次请求预热（建立 gRPC 连接、QueryNode 缓存）。

### 4.4 常见踩坑经验

1. **忘了算 HNSW 图结构内存**：只算了 20G 原始数据，结果 QueryNode 32G 内存不够（实际需要 36G+）。
2. **用平均值代替 P95 汇报**：P95 是 200ms 但平均值只有 30ms，领导以为性能很好，上线后用户投诉。
3. **压测时没模拟标量过滤**：纯向量搜索测得 P95=10ms，加上复杂 Expr 后 P95=80ms。

### 4.5 思考题

1. 如果 1000 万向量用 HNSW 索引需要 80GB 内存，但 QueryNode 只有 32GB，有哪些减少内存占用的方案？
2. 压测时发现 P99 延迟比 P95 高出一个数量级（比如 P95=10ms, P99=200ms），这通常意味着什么？如何排查长尾问题？
