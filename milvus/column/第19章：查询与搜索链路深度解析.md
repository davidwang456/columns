# 第19章：查询与搜索链路深度解析

> **定位**：理解搜索请求如何被调度到 QueryNode 并执行。
> **版本**：Milvus 2.5.x
> **源码关联**：internal/proxy/search_task.go、internal/querycoordv2/server.go、internal/querynodev2/search.go、internal/core/src/index/

---

## 1. 项目背景

搜索业务方投诉：同样的搜索词，有时 3ms 返回、有时 800ms——长尾延迟严重。运维小李看了 Proxy 的 P95 延迟，30ms 看起来"还行"，但把 QueryNode 的延迟分布拉出来一看——QueryNode 2 号节点的 P95 是 350ms，QueryNode 1 号和 3 号只有 15ms。

排查发现：2 号节点上加载了一个 8GB 的大 Segment（因为 Compaction 没及时触发），搜索时要遍历大量数据。而 QueryCoord 的 Load Balancer 没有按 Segment 大小做均衡——它只看"每个 QueryNode 加载了几个 Segment"，不看"每个 Segment 多大"。

小李意识到，这个问题的根因在**搜索链路的两层调度**——QueryCoord 如何分配 Segment 给 QueryNode（加载调度）& QueryNode 如何执行搜索（执行调度）。不理解这两层，就无法排查长尾延迟。

本章将追踪一次 Search 请求从 Proxy 到 QueryNode 的完整路径，包括 Query Plan 生成、Segment 路由、检索执行和结果归并。

---

## 2. 项目设计（剧本式交锋对话）

**第一幕：一次 Search 请求的 5 段旅程**

*（小李在 Grafana 上盯着 QueryNode 延迟分位图，P95 曲线像过山车）*

**小胖**（指着 Grafana）："不都是搜同一个 Collection 嘛——凭什么有的快有的慢？Milvus 内部到底怎么分配工作的？"

**大师**："搜索请求在 Milvus 内部走了 5 段路——每一段都可能成为瓶颈。"

```
一次 Search 请求的完整链路:

① Client → Proxy          ② Proxy → QueryCoord      ③ QueryCoord 计划
─────────────────         ────────────────────       ────────────────
gRPC SearchRequest        Get segment routing        "这个 Collection 的 Segment
到达 Proxy 19530          info for this collection   分布在哪些 QueryNode 上？
                          获取加载了目标 Segment     每个 QueryNode 的健康状态?"
                          的 QueryNode 列表

④ QueryNode 执行检索      ⑤ Proxy 归并 → Client
───────────────────       ───────────────────
"在加载的 Segment 上       收集各 QueryNode 的 TopK
执行向量检索 + 标量过滤"    做全局排序，取最终 TopK
                         返回给客户端
```

**大师**（展开每一步的细节）：

```
① Proxy 入口 (proxy/search_task.go)
───────────────────────────────────
SearchTask.execute():
  1. 校验参数: 向量维度、TopK、Expr 语法
  2. 向 QueryCoord 发送 GetSegmentInfo 请求
  3. 等待 QueryCoord 返回 Segment → QueryNode 映射
  4. 为每个 QueryNode 构造子请求 (sub-search)
  
② QueryCoord 调度 (querycoordv2/server.go)
────────────────────────────────────────
getSegmentInfo():
  1. 查本地缓存: Collection 的 Segment 分布在哪些 QueryNode
  2. 返回 [QueryNode1: [segment_A, segment_B], 
           QueryNode2: [segment_C, segment_D]]
  3. 策略: 最小负载优先 (Least Loaded First)

③ QueryNode 执行 (querynodev2/search.go)
───────────────────────────────────────
VectorSearch():
  1. 遍历本地加载的 Segment 列表
  2. 对每个 Sealed Segment: 走 HNSW/IVF 索引检索
  3. 对 Growing Segment: 走暴力检索 (无索引)
  4. 应用 Expr 标量过滤
  5. 返回本地 TopK 给 Proxy

④ Proxy 归并 (proxy/search_task.go)
──────────────────────────────────
Reduce():
  1. 收集所有 QueryNode 返回的结果
  2. 全局排序 (按 distance/score)
  3. 取最终 TopK
  4. 格式化为 SearchResponse → 返回客户端
```

> **技术映射**：Proxy = 项目经理（接需求、分任务、汇总结果）；QueryCoord = 资源调度员（知道谁有空、谁手上有资料）；QueryNode = 研究员（在分配的文档里找答案）。

**第二幕：Growing Segment vs Sealed Segment 搜索差异**

**小白**："你刚才说 Growing Segment 走暴力检索——那如果 Growing 里有很多数据，岂不是很慢？"

**大师**："对，这就是搜索延迟波动的核心原因之一。来看对比——"

| 搜索维度 | Sealed Segment | Growing Segment |
|---------|---------------|-----------------|
| 索引 | 有（HNSW/IVF） | 无（暴力检索） |
| 复杂度 | O(log N) | O(N×D) |
| 10万条搜索耗时 | ~2ms | ~50ms |
| 100万条搜索耗时 | ~5ms | ~500ms |
| 是否常见 | 占总数据 95%+ | 占总数据 <5%（正常情况） |

**大师**："正常情况下 Growing Segment 占比很小，开销可以忽略。但有两种场景会让 Growing Segment 膨胀——"

1. **Flush 间隔太长**：默认 60 分钟才触发一次 Flush，中间写入的 100 万条数据全在 Growing Segment 里。
2. **写入高峰**：大促期间短时间内写入 500 万条，Growing Segment 爆满。

**大师**："解决方案：调小 `dataNode.flush.interval` 或 `segment.maxSize`，让它更频繁 Flush。但不要调太小——过于频繁的 Flush 会产生大量小 Sealed Segment，反而影响搜索性能（需要遍历更多 Segment）。"

> **技术映射**：Sealed Segment = 图书馆已编目的书（有目录卡，查得快）；Growing Segment = 新到还没编目的书（只能逐页翻，但数量少）；Flush = 给新书编目上架。

---

**第三幕：Expr 过滤与向量检索的执行协同**

**小胖**："那搜索时既要向量匹配又要 Expr 过滤——是先过滤再搜，还是边搜边过滤？"

**大师**："这就引出了 Milvus 查询引擎的核心——执行计划（Query Plan）。"

```
Query Plan 的执行决策树:

Search Request: vector=[...], expr="price > 100 and category == '户外'", limit=10
                              │
                     ┌────────▼────────┐
                     │ 分析 Expr 选择性  │
                     └────────┬────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
    选择性高 (category=='户外' 命中 5% 数据)   选择性低 (price>100 命中 80% 数据)
              │                               │
    ┌─────────▼─────────┐           ┌─────────▼─────────┐
    │ Pre-filtering     │           │ Iterative Filtering│
    │ 先用 Expr 过滤     │           │ 先向量搜索一批      │
    │ 再在结果上做向量搜索│           │ 再过滤，不够继续搜  │
    └───────────────────┘           └───────────────────┘
    
    默认策略: Iterative Filtering (边搜边过滤)
    源码位置: internal/querynodev2/search.go → Search() → doSearch()
```

**大师**："两种策略的选择由 QueryNode 内部根据 Expr 的'选择性'自动判断——如果过滤后数据量很小，走 Pre-filtering 更高效；如果过滤后数据量仍然很大，走 Iterative 更省 GPU。"

> **技术映射**：Pre-filtering = 先筛简历再面试（人少效率高）；Iterative Filtering = 群面中边问边筛（人多时更快出结果）。

### 3.3 分步实现

#### 步骤 1：搜索链路追踪脚本

```python
# step1_search_trace.py
"""追踪一次 Search 请求的完整链路"""
import time
import logging
from pymilvus import connections, Collection, utility
from sentence_transformers import SentenceTransformer

logging.basicConfig(level=logging.INFO)

connections.connect(host="localhost", port="19530")
model = SentenceTransformer("all-MiniLM-L6-v2")
collection = Collection("product_search_prod")

# 准备查询
query_text = "户外露营折叠椅"
query_vec = model.encode([query_text]).tolist()

print("=" * 70)
print("Search 链路追踪")
print("=" * 70)

# ─── 阶段 1: 客户端 → Proxy (gRPC) ───
t0 = time.time()

# ─── 阶段 2: Proxy 查询计划 + QueryCoord 调度 ───
# (内部透明，通过延迟推算)
plan_start = time.time()

# ─── 阶段 3: QueryNode 执行检索 ───
search_start = time.time()
results = collection.search(
    data=query_vec,
    anns_field="title_vec",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    expr="in_stock == true and price >= 50",
    limit=10,
    output_fields=["title", "price", "in_stock"],
    timeout=30,
)
search_end = time.time()

total_time = (search_end - t0) * 1000
search_time = (search_end - search_start) * 1000
plan_time = (search_start - plan_start) * 1000

print(f"\n  总耗时: {total_time:.1f}ms")
print(f"  Proxy 计划阶段: ~{plan_time:.1f}ms (估)")
print(f"  QueryNode 搜索阶段: ~{search_time:.1f}ms (估)")
print(f"  客户端 gRPC 往返: ~{total_time - search_time - plan_time:.1f}ms (估)")

print(f"\n  TopK 结果:")
for i, hit in enumerate(results[0]):
    print(f"    #{i+1}: {hit.entity.get('title')[:30]}, "
          f"${hit.entity.get('price'):.1f}, "
          f"distance={hit.distance:.4f}")
```

#### 步骤 2：对比不同 Expr 复杂度对延迟的影响

```python
# step2_expr_latency.py
"""对比不同 Expr 复杂度对搜索延迟的影响"""
import time
import numpy as np
from pymilvus import Collection
from sentence_transformers import SentenceTransformer

collection = Collection("product_search_prod")
model = SentenceTransformer("all-MiniLM-L6-v2")
query_vec = model.encode(["户外折叠椅"]).tolist()

expr_tests = [
    ("无过滤", None),
    ("简单: price>100", "price > 100"),
    ("中等: price>100 AND in_stock", "price > 100 and in_stock == true"),
    ("复杂: 2类目+价格+库存", "category in ['户外运动','家居日用'] and price >= 50 and price <= 500 and in_stock == true"),
]

print("=" * 70)
print("Expr 复杂度对搜索延迟的影响 (10轮取平均)")
print("=" * 70)

for name, expr in expr_tests:
    times = []
    for _ in range(10):
        t0 = time.time()
        collection.search(
            data=query_vec, anns_field="title_vec",
            param={"metric_type": "COSINE", "params": {"ef": 64}},
            expr=expr, limit=10
        )
        times.append((time.time() - t0) * 1000)
    
    avg = np.mean(times)
    cnt = len(collection.search(
        data=query_vec, anns_field="title_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        expr=expr, limit=10, output_fields=["id"]
    )[0])
    print(f"  {name:<35} avg={avg:.2f}ms, results={cnt}")
```

## 4. 项目总结

### 4.1 搜索链路各阶段耗时参考

| 阶段 | 耗时占比 | 优化方向 |
|------|---------|---------|
| Proxy 校验+计划 | 10-15% | 增大 Proxy CPU |
| QueryCoord 调度 | 5-10% | 缓存 Segment 映射（已默认） |
| QueryNode 检索 | 60-80% | 索引类型、ef 参数、Segment 大小 |
| Proxy 归并 | 5-10% | TopK 不太大时不是瓶颈 |
| gRPC 往返 | 5% | 就近部署 |

### 4.2 搜索延迟优化速查

| 问题 | 现象 | 解决方案 |
|------|------|---------|
| 个别 QueryNode 慢 | P50 正常 P95 高 | 检查 Segment 大小均衡，触发 Compaction |
| 全 QueryNode 慢 | P50 也高 | 增大 QueryNode 内存/CPU，或用更快的索引 |
| Expr 复杂导致慢 | 加过滤条件后延迟翻倍 | 为标量字段建索引，简化 Expr |
| Growing Segment 拖累 | 写入高峰时搜索变慢 | 缩短 Flush 间隔 |

### 4.3 注意事项

- **QueryCoord 的 Segment 缓存不是实时的**：新增 Segment 可能要几秒后才能被搜索到（取决于心跳间隔）。
- **Proxy 的并发搜索超时是累加的**：如果 10 个 QueryNode 中 1 个超时，Proxy 可能等待整个 `timeout` 周期才返回错误。
- **长尾延迟排查应从 QueryNode 开始**：90% 的搜索延迟问题出在 QueryNode 段而非 Proxy/Coordinator。
