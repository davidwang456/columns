> **版本**：Milvus 2.5.x
> **源码关联**：internal/proxy/search.go、internal/querynodev2/expr/、internal/querynodev2/search.go

## 1. 项目背景

某电商平台的搜索产品经理提了一个需求：用户搜"适合露营的轻便折叠椅"时，不仅要语义匹配，还要能按价格区间（100-300 元）、商品类目（仅"户外运动"）、库存状态（仅"有货"）三个维度同时过滤。开发张工拍了拍胸脯说"简单，向量搜索完再过滤一下就行了"——然后他在代码里这么写：

```python
results = collection.search(data=[query_vec], limit=100, ...)
filtered = [h for h in results[0] 
            if h.entity.get("price") >= 100 
            and h.entity.get("price") <= 300
            and h.entity.get("category") == "户外运动"
            and h.entity.get("in_stock") == True][:10]
```

很快用户投诉了：搜"户外折叠椅"结果有 8 条缺货商品、3 条价格上万的奢侈品牌、2 条"家居日用"类的仓库椅。明明过滤了，为什么结果跟没过滤一样？张工一查日志——向量搜索 `limit=100`，但这 100 条里满足"户外+有货+100-300 元"三条件的只有 4 条，最终返回 4 条，远少于用户期待的 10 条。

张工这才意识到：**先搜后过滤**的逻辑有致命缺陷——如果向量召回的前 100 条里符合条件的不到 10 条，最终结果就是不足 10 条甚至为空。这就是"召回不足"问题。正确的做法是用 Milvus 原生的 **Scalar Filtering（标量过滤）**，在向量检索的过程中同步执行标量过滤，确保在符合条件的候选集上做向量搜索。

本章将深入 Milvus 的 Search 与 Query 接口差异、Expr 表达式语法、向量召回+标量过滤的执行方式，并实现一个支持类目、价格、库存组合过滤的商品搜索接口。

## 2. 项目设计（剧本式交锋对话）

**第一幕：Search vs Query——从概念上分清**

*（张工正在给团队演示"先向量搜索再客户端过滤"的翻车案例）*

**小胖**（困惑地）："大师，Milvus 有 Search 和 Query 两个接口——一个搜向量，一个查数据。我到现在都分不清什么时候用哪个。不就是搜吗？有啥区别？"

**大师**："这就像你去图书馆找书，有两种方式——"

**大师**（在白板上画）：

```
Search (向量检索)                        Query (精确查询)
─────────────────────                    ────────────────────
"帮我找和这本书内容最相似的 10 本书"        "帮我找 ISBN=978-7-xxx 的那本书"
              │                                        │
              ▼                                        ▼
      输入：查询向量                           输入：精确条件 (Expr)
      返回：TopK 相似结果                       返回：所有匹配的完整记录
      核心：相似度排序                         核心：精确匹配过滤
```

| 特性 | Search | Query |
|------|--------|-------|
| 输入 | 向量 + TopK + 可选标量过滤 | 标量过滤表达式（无向量） |
| 排序 | 按向量相似度排序 | 无排序（或按标量字段排序） |
| 返回 | 最多 TopK 条 | 所有匹配记录（可配合 limit/offset） |
| 核心用途 | 语义搜索、相似推荐 | 按 ID 查详情、按条件导出数据 |
| 性能 | 依赖索引效率 | 依赖标量索引效率 |

**小白**："那什么时候两个都要用？"

**大师**："典型的'两步走'模式——先用 Search 找到语义相似的商品 ID 列表，再用 Query 根据 ID 查完整信息。当然，Search 本身也能返回标量字段（通过 `output_fields`），如果返回的字段够用，就不需要再调 Query。"

**小胖**："哦！所以我之前那个需求——搜相似商品 + 过滤价格/类目/库存——应该用 Search 加 Expr 过滤，而不是 Search 完了再客户端过滤！"

> **技术映射**：Search = 图书管理员根据"内容相似度"找书 + 可选分类筛选；Query = 管理员根据精确索书号找书；两步走 = Search 找 ID → Query 查详情。

---

**第二幕：Expr 表达式语法完全指南**

**小白**："大师，Expr 表达式能写多复杂？不会只能写 `price > 100` 这种简单的吧？"

**大师**："Expr 的能力比你想的强得多。我把常用的表达式语法列成一张表——"

**大师**（在白板上列语法表）：

| 表达式类型 | 语法示例 | 业务场景 |
|-----------|---------|---------|
| **比较运算** | `price >= 100`、`price <= 300` | 价格区间 |
| **逻辑组合** | `category == "户外" and in_stock == true` | 多条件筛选 |
| **范围运算** | `price in [99, 199, 299]` | 特定定价档位 |
| **字符串匹配** | `title like "露营%"` | 商品标题前缀匹配 |
| **JSON 字段** | `attributes["color"] == "red"` | JSON 嵌套字段过滤 |
| **数组运算** | `tags contains "热卖"` 或 `JSON_CONTAINS(tags, "热卖")` | 标签过滤 |
| **IS NULL** | `price is null` | 查找缺失数据 |
| **算术运算** | `price * 0.9 < 100` | 折后价过滤 |

**小白**："等等，Milvus 有 LIKE 和 JSON 字段支持？这不是 MySQL 的功能吗？"

**大师**："Milvus 从 2.3 开始引入了表达式引擎，支持了大多数 SQL 风格的过滤操作。但要注意边界——"

1. **LIKE 只支持前缀匹配**：`title like "露营%"` ✓；`title like "%露营%"` ✗（不支持全模糊）。
2. **JSON 字段需要先建索引**：默认情况下 JSON 字段过滤走全扫描，需要为 JSON 的特定 key 建索引。
3. **表达式长度有限制**：单条表达式的解析深度和复杂度有上限，超复杂条件建议在业务层拆分。
4. **不同字段类型的运算符不同**：VARCHAR 用 `==` 和 `like`，INT64/FLOAT 用 `>`/`<`/`in`，BOOL 用 `==`，ARRAY 用 `contains`。

**小胖**："那如果我要写 `(category == '户外' or category == '运动') and price between 100 and 300 and in_stock == true`——这个能行吗？"

**大师**："完全正确！但要留意两点：第一，`between` 目前不支持，得写成 `price >= 100 and price <= 300`。第二，`or` 条件可能导致索引失效——如果 `or` 的两边走的是不同的索引路径，Milvus 可能会退化为全扫描后再过滤。"

> **技术映射**：Expr = 类似 SQL 的 WHERE 子句；LIKE = 前缀模糊匹配（非全模糊）；JSON 字段 = 半结构化数据的过滤能力；Expr 复杂度直接影响搜索性能，越复杂越慢。

---

**第三幕：向量召回 + 标量过滤的执行方式——为什么"先搜后过滤"是错的**

**小白**："那 Search + Expr 在 Milvus 内部到底是怎么执行的？是搜完向量再过滤，还是过滤完再搜向量？"

**大师**："这才是问题的关键！Milvus 内部有两套执行策略——"

**大师**（画执行流程图）：

```
策略 1: "先过滤再搜索" (Pre-filtering)
────────────────────────────────────────
Query ──→ [标量过滤] ──→ 符合条件的候选集 ──→ [向量检索] ──→ TopK
         
         先缩小范围，再在范围内                          ✓ 保证 TopK 都在过滤范围内
         做向量相似度排序                                 ✓ 召回率稳定
                                                        ✗ 如果过滤条件很窄，候选集太小
                                                           可能搜不到足够结果

策略 2: "边搜索边过滤" (Iterative Filtering)
────────────────────────────────────────
Query ──→ [向量检索 + 标量过滤交替执行] ──→ TopK
         
         每次向量搜索返回一批结果，                     ✓ 不会因过滤太窄而返回不足
         过滤后再决定是否需要继续搜索                    ✓ 实际生产中更常用
                                                        ✗ 执行复杂度更高

策略 3: "先搜后过滤" (Post-filtering) —— 你自己写的！
────────────────────────────────────────
Query ──→ [向量检索 TopK=100] ──→ [客户端过滤] ──→ 最终 ≤10 条
         
         ✗ 这是最差的做法！                              ✗ 过滤后结果可能不足 TopK
         ✗ 没有利用 Milvus 的过滤能力                     ✗ 浪费了网络传输和排序开销
```

**大师**："Milvus 默认使用'边搜索边过滤'策略，在 QueryNode 执行层面叫 Iterative Filtering。具体来说——"

1. QueryNode 收到搜索请求后，首先分析 Expr 表达式，确定哪些 Segment 包含相关数据。
2. 在目标 Segment 上执行向量检索，每检索出一批候选结果，立刻用 Expr 过滤。
3. 如果过滤后仍然不到 TopK 条，继续检索下一批，直到凑够 TopK 或遍历完所有候选。

**小胖**："那如果我设了 `limit=10`，但符合条件的商品总共只有 7 个，会返回几？"

**大师**："只返回 7 个。Milvus 不会用不符合条件的结果来凑数。这就是我们之前在'先搜后过滤'模式里碰到的问题——只不过 Milvus 原生过滤至少能保证这 7 条是正确的，而客户端过滤可能连这 7 条都找不到（如果在向量检索时这 7 条没出现在 Top100 里的话）。"

**小白**："过滤条件太窄导致召回不足——这个有办法解决吗？"

**大师**："三个抓手：第一，增大向量检索的搜索范围（比如增大 HNSW 的 ef 参数），让更多候选参与过滤；第二，如果过滤条件本身是业务强需求且确实数据很少，在业务层做降级处理——比如放宽价格区间或提示用户'暂无更多商品'；第三，在数据集层面检查为什么符合条件的商品这么少，是不是数据没入库。"

> **技术映射**：Pre-filtering = 先缩小范围再搜（快但可能漏）；Iterative Filtering = 边搜边筛（Milvus 默认，兼顾准确性和数量）；Post-filtering = 搜完再筛（最差，已弃用）。

"""创建商品搜索 Collection 并写入带标量字段的测试数据"""
import random
from pymilvus import connections, Collection, utility
from pymilvus import CollectionSchema, FieldSchema, DataType

connections.connect(host="localhost", port="19530")

COLLECTION_NAME = "product_filter_search"

if utility.has_collection(COLLECTION_NAME):
    utility.drop_collection(COLLECTION_NAME)

fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=512),
    FieldSchema(name="title_vec", dtype=DataType.FLOAT_VECTOR, dim=384),
    FieldSchema(name="price", dtype=DataType.FLOAT),
    FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="brand", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="in_stock", dtype=DataType.BOOL),
    FieldSchema(name="rating", dtype=DataType.FLOAT),
    FieldSchema(name="tags", dtype=DataType.ARRAY, element_type=DataType.VARCHAR, max_capacity=10, max_length=32),
]

schema = CollectionSchema(fields, description="标量过滤实战 - 商品搜索")
collection = Collection(COLLECTION_NAME, schema)
print(f"✓ Collection '{COLLECTION_NAME}' 创建完成")

# 生成并写入测试数据
from sentence_transformers import SentenceTransformer
model = SentenceTransformer("all-MiniLM-L6-v2")

categories = ["户外运动", "数码电子", "家居日用", "服饰鞋包", "食品饮料"]
brands = ["极地户外", "科技先锋", "宜家优选", "潮流前线", "有机农场"]
statuses = [True, True, True, False]

products = []
for i in range(1, 5001):
    cat = random.choice(categories)
    products.append({
        "id": i,
        "title": f"{random.choice(brands)} {cat} {random.choice(['便携','防水','轻便','大容量','静音'])} "
                 f"{random.choice(['帐篷','耳机','收纳盒','T恤','坚果'])}",
        "price": round(random.uniform(9.9, 1999.9), 2),
        "category": cat,
        "brand": random.choice(brands),
        "in_stock": random.choice(statuses),
        "rating": round(random.uniform(1.0, 5.0), 1),
        "tags": random.sample(["热卖", "新品", "限时", "包邮", "正品"], k=random.randint(1, 3)),
    })

# 批量写入
for i in range(0, len(products), 500):
    batch = products[i:i+500]
    titles = [p["title"] for p in batch]
    embeddings = model.encode(titles).tolist()
    collection.insert([
        [p["id"] for p in batch],
        titles,
        embeddings,
        [p["price"] for p in batch],
        [p["category"] for p in batch],
        [p["brand"] for p in batch],
        [p["in_stock"] for p in batch],
        [p["rating"] for p in batch],
        [p["tags"] for p in batch],
    ])

utility.flush([COLLECTION_NAME])
print(f"✓ 写入完成: {collection.num_entities} 条")

# 构建索引
collection.create_index("title_vec", {
    "index_type": "HNSW", "metric_type": "COSINE",
    "params": {"M": 16, "efConstruction": 200}
})
utility.wait_for_index_building_complete(COLLECTION_NAME, timeout=120)
collection.load()
print(f"✓ 索引构建完成并已 Load")

#### 步骤 3：Search + Expr 的完整业务接口封装

```python
# step3_product_search_service.py
"""商品搜索服务：支持多条件标量过滤"""
from pymilvus import Collection, connections
from sentence_transformers import SentenceTransformer
from typing import List, Dict, Optional

class ProductSearchService:
    """商品语义搜索服务"""
    
    def __init__(self, collection_name: str, model_name: str = "all-MiniLM-L6-v2"):
        connections.connect(host="localhost", port="19530")
        self.collection = Collection(collection_name)
        self.model = SentenceTransformer(model_name)
    
    def _build_expr(self, 
                    category: Optional[str] = None,
                    categories: Optional[List[str]] = None,
                    min_price: Optional[float] = None,
                    max_price: Optional[float] = None,
                    in_stock_only: bool = False,
                    brand_prefix: Optional[str] = None,
                    min_rating: Optional[float] = None) -> str:
        """构造过滤表达式"""
        conditions = []
        
        if category:
            conditions.append(f"category == '{category}'")
        if categories and len(categories) > 0:
            quoted = ", ".join(f"'{c}'" for c in categories)
            conditions.append(f"category in [{quoted}]")
        
        if min_price is not None:
            conditions.append(f"price >= {min_price}")
        if max_price is not None:
            conditions.append(f"price <= {max_price}")
        
        if in_stock_only:
            conditions.append("in_stock == true")
        
        if brand_prefix:
            conditions.append(f"brand like '{brand_prefix}%'")
        
        if min_rating is not None:
            conditions.append(f"rating >= {min_rating}")
        
        return " and ".join(conditions) if conditions else None
    
    def search(self, 
               query_text: str,
               top_k: int = 10,
               **filter_kwargs) -> List[Dict]:
        """搜索相似商品
        
        Args:
            query_text: 用户搜索文本
            top_k: 返回结果数
            **filter_kwargs: 标量过滤条件
                - category: 限定单个类目
                - categories: 限定多个类目
                - min_price: 最低价格
                - max_price: 最高价格
                - in_stock_only: 仅搜有货
                - brand_prefix: 品牌前缀匹配
                - min_rating: 最低评分
        
        Returns:
            [{"id": ..., "title": ..., "price": ..., "distance": ...}, ...]
        """
        expr = self._build_expr(**filter_kwargs)
        query_vec = self.model.encode([query_text]).tolist()
        
        results = self.collection.search(
            data=query_vec,
            anns_field="title_vec",
            param={"metric_type": "COSINE", "params": {"ef": 64}},
            expr=expr,
            limit=top_k,
            output_fields=["title", "price", "category", "brand", "in_stock", "rating"]
        )
        
        return [
            {
                "id": hit.id,
                "title": hit.entity.get("title"),
                "price": hit.entity.get("price"),
                "category": hit.entity.get("category"),
                "brand": hit.entity.get("brand"),
                "in_stock": hit.entity.get("in_stock"),
                "rating": hit.entity.get("rating"),
                "score": round(hit.distance, 4),
            }
            for hit in results[0]
        ]

# 使用示例
service = ProductSearchService("product_filter_search")

# 场景 1: 纯语义搜索（无过滤）
print("=== 场景 1: 纯语义搜索 ===")
for item in service.search("露营帐篷", top_k=5):
    print(f"  {item['title'][:30]:<35} ¥{item['price']:>7.1f}  {item['score']:.3f}")

# 场景 2: 类目 + 价格 + 库存组合过滤
print("\n=== 场景 2: 户外+有货+100-500元 ===")
for item in service.search("露营帐篷", top_k=5, 
                            category="户外运动", min_price=100, 
                            max_price=500, in_stock_only=True):
    print(f"  {item['title'][:30]:<35} ¥{item['price']:>7.1f}  [{item['category']}]  有货={item['in_stock']}")

# 场景 3: 品牌前缀 + 高评分
print("\n=== 场景 3: 极地品牌+4星以上 ===")
for item in service.search("露营帐篷", top_k=5, 
                            brand_prefix="极地", min_rating=4.0):
    print(f"  {item['title'][:30]:<35} ¥{item['price']:>7.1f}  {item['brand']:<8}  评分={item['rating']}")
```

## 4. 项目总结

### 4.1 优缺点对比

| 维度 | Milvus Search + Expr | 先搜后客户端过滤 | Elasticsearch Filter |
|------|---------------------|----------------|---------------------|
| 召回完整性 | 高（边搜边过滤，保证 TopK） | 低（搜索窗口之外的符合条件的记录被丢失） | 高（原生过滤+打分排序） |
| 性能 | 好（过滤在 QueryNode 内存中执行） | 差（浪费网络传输和 CPU） | 好（过滤集成在搜索流程中） |
| 表达式复杂度 | 中（支持 =/>/</in/like/json/contains） | 任意（客户端代码灵活） | 高（DSL 非常灵活） |
| 大范围 OR | 可能性能下降 | 不受影响 | 优化较好 |

### 4.2 适用场景

- **电商商品搜索**：语义匹配 + 价格/类目/库存/品牌多维度过滤
- **RAG 知识库**：向量召回 + 文档来源/时间范围/权限过滤
- **推荐系统**：用户向量召回 + 年龄/地域/偏好标签过滤
- **风控检索**：相似样本召回 + 时间窗口/金额阈值过滤

**不适用场景**：需要跨 Collection JOIN 的过滤（Milvus 不支持）、需要正则匹配的过滤。

### 4.3 注意事项

- **AND 条件比 OR 条件快**：OR 可能导致不同索引路径的合并，性能下降。
- **空 Expr 不等于没有 Expr**：传 `expr=""` 和 `expr=None` 行为不同，建议显式传 `None`。
- **过滤字段需建标量索引**：大量数据的标量过滤如果没有索引会全扫描。
- **每次 Search 调用建议设置 expr timeout**：复杂表达式可能导致单个搜索耗时过长。

### 4.4 常见踩坑经验

1. **用了 `price between A and B` 语法报错**：Milvus Expr 不支持 SQL 的 BETWEEN，需写 `price >= A and price <= B`。
2. **LIKE 全模糊不生效**：`title like '%露营%'` 不抛错但也不匹配，Milvus 只支持前缀 `like '露营%'`。
3. **布尔值过滤写法错误**：正确写法 `in_stock == true`，不能写 `in_stock`（裸字段不合法），也不能写 `in_stock == True`（Python 关键字）。

### 4.5 思考题

1. 如果一个 Search 请求的 Expr 过于复杂（10 个 AND/OR 条件），Milvus 内部如何优化执行计划？提示：研究 `internal/querynodev2/expr/` 中的表达式解析和短路径优化。
2. 对于"类目过滤"这种高基数字段过滤，应该为标量字段建什么类型的索引来加速？如何在建索引时权衡写入性能和查询性能？
