> **定位**：形成可复用的压测方法论。
> **版本**：Milvus 2.5.x
> **源码关联**：internal/proxy/search_task.go、internal/querynodev2/search.go、internal/datanode/data_sync_service.go

## 1. 项目背景

某电商平台准备启动"双十一"大促，预估搜索 QPS 将从日常 500 飙到 8000。技术负责人需要回答三个问题：当前配置能扛多少 QPS？瓶颈在哪里？需要扩容多少？

运维团队跑了一轮压测——用 50 并发打了 100 万次搜索请求。P95 延迟 45ms，看起来不错。但大促当天凌晨 0 点刚过，QPS 到 3000 时 P95 就飙升到 800ms。事后复盘发现：压测脚本只测了纯向量搜索（无 Expr 过滤），而生产 90% 的搜索都带标量过滤条件。加上过滤后，P95 应该在压测阶段就暴露出来。

更关键的是，压测团队没有做"瓶颈逐层定位"——不知道延迟的 45ms 里多少花在 Proxy、多少在 QueryCoord、多少在 QueryNode。结果优化方向全凭猜测：有人说加 Proxy，有人说换索引，最后发现瓶颈其实是 Embedding 模型推理占了 15ms。

本章将建立一套从"数据准备 → 压测执行 → 瓶颈定位 → 调优验证"的系统方法论。

**第二幕：瓶颈逐层定位——延迟拆解**

**小胖**："延迟 45ms 里到底哪些环节在花时间？"

**大师**："用'延迟拆解法'——把一次搜索请求的时间拆成 5 段。"

```
搜索延迟拆解 (以 500 万条/768 维/HNSW 为例):

┌──────────────┬─────────┬──────────┬─────────────────────┐
│   阶段        │ 典型耗时 │ 占比     │ 如何测量             │
├──────────────┼─────────┼──────────┼─────────────────────┤
│ ① 客户端 gRPC │ ~1ms    │ 3%       │ SDK 侧打点 t1-t0   │
│ ② Proxy 处理  │ ~1ms    │ 3%       │ milvus_proxy_req_latency │
│ ③ QueryCoord │ ~1ms    │ 3%       │ Proxy 日志中的调度延迟│
│ ④ QueryNode  │ ~25ms   │ 76%      │ milvus_querynode_search_latency │
│ ⑤ Proxy 归并  │ ~1ms    │ 3%       │ Proxy 日志中的 Reduce 耗时│
├──────────────┼─────────┼──────────┼─────────────────────┤
│  总计         │ ~30ms   │ 100%     │                     │
│  加上 Embedding│ +15ms  │ 额外      │ Embedding 服务独立   │
└──────────────┴─────────┴──────────┴─────────────────────┘
```

**大师**："瓶颈定位的关键指标——"

| 瓶颈在哪里 | 如何判断 | 优化方向 |
|-----------|---------|---------|
| Proxy | Proxy CPU > 80%, 但 QueryNode CPU < 50% | 增加 Proxy 副本 |
| QueryCoord | QueryCoord 调度延迟 > 5ms | 检查心跳间隔、Segment 数量 |
| QueryNode | QueryNode CPU/Memory > 80% | 加 QueryNode、换索引、调整参数 |
| Embedding 服务 | Embedding API 延迟 > 10ms | GPU 加速、模型量化、批量推理 |

**小白**："那如果我优化 QueryNode（换了更快的索引），但 Embedding 模型推理还是要 15ms，总延迟还是下不去——"

**大师**："这就是**端到端思维**的重要性。优化不能只看 Milvus 内部，还要看上游——Embedding 生成、网络传输、结果后处理——都在端到端延迟里。"

> **技术映射**：延迟拆解 = 看病时量体温/血压/心电图分别排查；瓶颈定位 = 找到最慢的那一步（木桶最短的板）；端到端 = 从挂号到拿药的总耗时（不是只看医生的问诊时间）。

**第三幕：写入和查询的调优矩阵**

**大师**："调优要分两条线——写入和查询。各有一个调优矩阵。"

**写入调优矩阵：**

| 调优方向 | 措施 | 效果 | 副作用 |
|---------|------|------|--------|
| 批量大小 | 100→500 条/批 | 吞吐 ×3-5 | Proxy 内存增加 |
| Shard 数 | 1→4 | 写入并行度 ×4 | Segment 碎片增加 |
| Flush 间隔 | 60min→10min | 减少 Growing Segment 膨胀 | 小 Segment 变多 |
| DataNode 数 | 2→4 | 消费吞吐 ×2 | 成本增加 |
| Segment 大小 | 1GB→512MB | Flush 更快，减少黑洞时间 | Segment 数量翻倍 |

**查询调优矩阵：**

| 调优方向 | 措施 | 效果 | 副作用 |
|---------|------|------|--------|
| 索引类型 | IVF_FLAT→HNSW | 延迟 ×0.3, 召回 ↑5% | 内存 ×1.8 |
| ef 参数 | ef=64→128 | 召回 ↑1%, 延迟 ×1.5 | 搜索变慢 |
| Replica | 1→2 | 读吞吐 ×1.8, 高可用 | 内存 ×2 |
| QueryNode | 3→6 | 读吞吐 ~×2 | 成本增加 |
| 标量索引 | 为 category 建索引 | Expr 过滤 ×5 快 | 内存和写入开销 |

**大师**："调优的黄金顺序——**先硬件再软件再参数**。先看是不是资源不够（加 QueryNode），再看是不是索引选错了（换索引），最后才微调参数（调 ef/nprobe）。"

> **技术映射**：写入调优 = 优化快递入库流（分拣员数、传送带速度、装箱大小）；查询调优 = 优化快递出库流（目录索引、取件窗口数、快递员数）；先硬件后软件 = 先看工人够不够，再看流程对不对。

"""生成指定规模的测试数据集"""
import numpy as np
from pymilvus import connections, Collection, utility
from pymilvus import CollectionSchema, FieldSchema, DataType

connections.connect(host="localhost", port="19530")

def create_bench_collection(name: str, dim: int = 768):
    if utility.has_collection(name):
        utility.drop_collection(name)
    fields = [
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vec", dtype=DataType.FLOAT_VECTOR, dim=dim),
        FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64),
        FieldSchema(name="price", dtype=DataType.FLOAT),
    ]
    return Collection(name, CollectionSchema(fields, description="压测用"))

# 生成 100 万条测试数据（实际测试可扩展到 1000 万）
N = 1_000_000
DIM = 768
coll = create_bench_collection("bench_1m", DIM)

BATCH = 10000
for i in range(0, N, BATCH):
    end = min(i+BATCH, N)
    size = end - i
    coll.insert([
        list(range(i+1, end+1)),
        np.random.rand(size, DIM).astype(np.float32).tolist(),
        [f"cat_{np.random.randint(1,11)}" for _ in range(size)],
        np.round(np.random.uniform(1, 1000, size), 2).tolist(),
    ])
    if (i // BATCH + 1) % 10 == 0:
        print(f"  生成 {end}/{N} 条")

utility.flush(["bench_1m"])
coll.create_index("vec", {"index_type": "HNSW", "metric_type": "COSINE",
                          "params": {"M": 16, "efConstruction": 200}})
utility.wait_for_index_building_complete("bench_1m", timeout=600)
coll.load()
print(f"> 压测数据集就绪: {N} 条 {DIM} 维")

```python
# step2_benchmark_framework.py
"""可配置的压测框架"""
import time
import numpy as np
from concurrent.futures import ThreadPoolExecutor, as_completed
from pymilvus import Collection

class SearchBenchmark:
    """Milvus 搜索压测框架"""
    
    def __init__(self, collection_name: str, query_vectors: list):
        self.collection = Collection(collection_name)
        self.queries = query_vectors
    
    def run(self, concurrency: int, total: int = 1000,
            top_k: int = 10, expr: str = None,
            ef: int = 64) -> dict:
        """执行一轮压测"""
        latencies = []
        errors = 0
        sp = {"metric_type": "COSINE", "params": {"ef": ef}}
        
        def task(qv):
            try:
                t0 = time.time()
                self.collection.search(
                    data=[qv], anns_field="vec",
                    param=sp, expr=expr, limit=top_k, timeout=30
                )
                return (time.time() - t0) * 1000, None
            except Exception as e:
                return 0, str(e)
        
        with ThreadPoolExecutor(max_workers=concurrency) as pool:
            futures = [pool.submit(task, self.queries[i % len(self.queries)])
                       for i in range(total)]
            for f in as_completed(futures):
                lat, err = f.result()
                if err: errors += 1
                else: latencies.append(lat)
        
        lat = np.array(latencies)
        return {
            "qps": total / (sum(latencies) / 1000) if latencies else 0,
            "errors": errors,
            "p50": np.percentile(lat, 50),
            "p95": np.percentile(lat, 95),
            "p99": np.percentile(lat, 99),
            "avg": np.mean(lat),
        }

# 准备 20 条查询向量
query_vecs = [np.random.rand(768).astype(np.float32).tolist() for _ in range(20)]
bench = SearchBenchmark("bench_1m", query_vecs)

# 实验 1: 并发压力测试
print("=" * 60)
print("并发压力测试 (TopK=10, 无Expr)")
print("=" * 60)
for c in [1, 10, 50, 100, 200]:
    r = bench.run(concurrency=c, total=500, top_k=10)
    print(f"  并发={c:<4}  QPS={r['qps']:<8.0f}  P95={r['p95']:.1f}ms  "
          f"P99={r['p99']:.1f}ms  err={r['errors']}")

# 实验 2: Expr 复杂性测试
print(f"\n{'='*60}")
print("Expr 复杂性测试 (并发=50, TopK=10)")
print("=" * 60)
for expr_name, expr in [("无Expr", None),
                          ("简单", "price >= 100"),
                          ("复杂", "price >= 100 and category == 'cat_1'")]:
    r = bench.run(concurrency=50, total=200, top_k=10, expr=expr)
    print(f"  {expr_name:<10}  P95={r['p95']:.1f}ms  QPS={r['qps']:.0f}")
```

"""基于压测数据生成扩容建议"""
def scaling_plan(current: dict, target_qps: int, target_p95: float):
    """扩容规划"""
    
    # 当前 P95 是否达标
    p95_gap = target_p95 / current["p95"] if current["p95"] > 0 else 1
    
    # 扩缩容建议
    advice = []
    
    if current["p95"] > target_p95:
        advice.append({
            "问题": f"P95={current['p95']:.0f}ms > 目标{target_p95:.0f}ms",
            "建议": [
                "• 换更快的索引 (IVF_FLAT→HNSW, 延迟可降60%)",
                "• 增加 QueryNode 数量 (线性加速)",
                "• 降低 ef 参数 (ef=64→32, 延迟降40%, 召回降2%)",
            ]
        })
    
    qps_gap = target_qps / current["qps"] if current["qps"] > 0 else 1
    if qps_gap > 1.5:
        needed_qn = max(1, qps_gap * 1.5)
        advice.append({
            "问题": f"QPS={current['qps']:.0f} < 目标{target_qps}(缺口{qps_gap:.1f}x)",
            "建议": [
                f"• QueryNode: 建议 ≥ {needed_qn:.0f} 台",
                "• Proxy: 建议 ≥ 2 台 (分散请求入口)",
                "• 考虑开启 Replica=2 (提升读吞吐)",
            ]
        })
    
    return {
        "current": {"qps": current["qps"], "p95": current["p95"]},
        "target": {"qps": target_qps, "p95": target_p95},
        "advice": advice,
    }

# 使用示例

# step4_latency_breakdown.py
"""端到端搜索延迟拆解定位瓶颈"""
import time
import requests
import numpy as np
from pymilvus import Collection, connections
from contextlib import contextmanager
from collections import defaultdict

connections.connect(host="localhost", port="19530")
collection = Collection("bench_1m")

class LatencyProfiler:
    """搜索链路延迟分析器"""
    
    def __init__(self):
        self.measurements = defaultdict(list)
    
    @contextmanager
    def measure(self, name: str):
        t0 = time.time()
        yield
        self.measurements[name].append((time.time() - t0) * 1000)
    
    def report(self):
        print("=" * 60)
        print("搜索延迟链路拆解")
        print("=" * 60)
        total_ms = 0
        for name, values in self.measurements.items():
            avg = np.mean(values)
            p95 = np.percentile(values, 95)
            total_ms += avg
            print(f"  {name:<20}: avg={avg:.2f}ms, P95={p95:.2f}ms")
        print(f"  {'─'*40}")
        print(f"  {'总计':<20}: avg={total_ms:.2f}ms")
        print(f"\n  瓶颈分析:")
        sorted_items = sorted(self.measurements.items(),
                             key=lambda x: np.mean(x[1]), reverse=True)
        for name, values in sorted_items[:3]:
            pct = np.mean(values) / total_ms * 100
            print(f"  {'▶' if pct > 50 else '▷'} {name}: {pct:.0f}% 占比")

# 使用
profiler = LatencyProfiler()

for _ in range(20):
    with profiler.measure("Embedding生成"):
        # 实际场景中这里调用 Embedding 模型
        query_vec = [np.random.rand(384).astype(np.float32).tolist()]
    
    with profiler.measure("Milvus检索"):
        collection.search(
            data=query_vec, anns_field="vec",
            param={"metric_type": "COSINE", "params": {"ef": 64}},
            limit=10, timeout=10
        )

profiler.report()

  │ P95(ms)  │ {before.get('p95', 'N/A'):<8} │ {after.get('p95', 'N/A'):<8} │ {improvement(before.get('p95',999), after.get('p95',0))} │
  │ P99(ms)  │ {before.get('p99', 'N/A'):<8} │ {after.get('p99', 'N/A'):<8} │ {improvement(before.get('p99',999), after.get('p99',0))} │

  └──────────┴──────────┴──────────┴──────────┘
╚══════════════════════════════════════════════════════════════╝
"""
    return report

def improvement(before_val, after_val):
    if before_val == 0 or before_val == 999: return "N/A"
    change = (before_val - after_val) / before_val * 100
    arrow = "↓" if change > 0 else "↑"
    return f"{arrow}{abs(change):.0f}%"

# 使用
before = {"qps": 500, "p50": 8, "p95": 45, "p99": 80, "err_rate": "0.2%"}
after = {"qps": 2100, "p50": 3, "p95": 12, "p99": 25, "err_rate": "0.02%"}
print(generate_tuning_report(before, after, ["HNSW M=16→32", "QueryNode 3→6台", "ef=64→128"]))

## 4. 项目总结

### 4.1 压测方法论速查

| 步骤 | 内容 | 工具/方式 |
|------|------|----------|
| 数据准备 | 生成与生产相似的测试数据集 | Python numpy 随机或真实数据集 |
| 单变量实验 | 每次只变一个参数 (数据量/并发/TopK/Expr) | 压测脚本 + 参数化 |
| 延迟拆解 | 分 Proxy/Coordinator/Node 定位瓶颈 | Prometheus + 日志 |
| 调优验证 | 每次优化后重新压测对比 | 前后对比表 |
| 容量规划 | 基于线性外推估算扩容需求 | 扩容建议脚本 |

### 4.2 注意事项

- **压测数据集要尽量接近生产数据分布**：随机正态向量 ≠ 真实 Embedding（语义上有聚集效应）。
- **预热很重要**：正式压测前先跑 100 次请求做预热（建立 gRPC 连接 + QueryNode 缓存）。
- **不要在生产环境压测**：使用独立隔离的压测集群。

### 4.3 思考题

1. 压测时发现 QPS 到 500 后就线性增长不动了（QPS=490→500→505，并发从 50→100→200 提升微弱）——瓶颈最可能在哪里？
