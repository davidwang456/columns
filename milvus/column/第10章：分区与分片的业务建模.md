# 第10章：分区与分片的业务建模

> **定位**：用正确的数据组织方式降低查询成本。
> **版本**：Milvus 2.5.x
> **源码关联**：internal/rootcoord/partition.go、internal/datacoord/channel_manager.go、internal/querycoordv2/segment.go

---

## 1. 项目背景

某 SaaS 知识库平台接入了 500 个企业租户，每个租户有独立的知识库（平均 5 万条文档 Chunk）。团队最初的设计很简单：一个 Collection 存所有租户的数据，搜索时用 `tenant_id == 'xxx'` 做过滤。

初期 10 个租户时一切正常，但当租户增长到 500 个时问题集中爆发：

1. **搜索延迟飙升**：每次搜索都要在全部 2500 万条数据中做向量检索 + 标量过滤，即使只搜一个租户。P95 延迟从 30ms 涨到了 800ms。
2. **写入不均衡**：大租户（50 万 Chunk）和小租户（1000 Chunk）共用一个 Shard，写入流量被大租户占满，小租户的写入经常超时。
3. **Load 时间太长**：Collection 越来越大，每次 Load 需要 20 分钟——而只搜一个租户的用户也得等 20 分钟。
4. **资源浪费**：即使一个租户的数据很少被访问，QueryNode 也要为它保留内存。
5. **扩容困难**：Cluster 模式下 Segment 按 Shard 分布，但无法按租户粒度迁移。

技术负责人张工意识到，根本问题是**数据组织方式错了**——把 500 个逻辑上完全独立的数据集塞进了同一个物理容器。他需要重新设计 Partition 和 Shard 策略。

## 2. 项目设计（剧本式交锋对话）

**第一幕：Partition 与 Shard——各司其职的两个概念**

*（张工在白板上画了一个大圆，里面密密麻麻写着 500 个租户 ID）*

**小胖**（困惑地）："大师，Partition 和 Shard 都是'分'，到底有什么区别？我看了半天文档还是晕——"

**大师**："这俩确实特别容易混淆。我用一个图书馆的比喻区分它们——"

**大师**（在白板上画）：

```
Partition（逻辑分区）                      Shard（物理分片）
─────────────────────                     ───────────────────
"按主题把书分到不同书架"                    "每个书架有多宽，能同时几个人放书"

  Collection: 图书馆                        Collection: 图书馆
  ├── Partition: 文学区（A-G书架）           ├── Shard 1: 入口通道 1
  ├── Partition: 科技区（H-M书架）           └── Shard 2: 入口通道 2
  ├── Partition: 艺术区（N-Z书架）
  └── Partition: _default（未分类）         每个 Shard = 一个消息队列 Channel
                                             写入请求按哈希分散到不同 Shard
                                     
  搜索时可以指定 Partition                    写入时自动按哈希分 Shard
  只搜"科技区"，跳过文学区和艺术区              Shard 越多 = 写入并行度越高
  缩小搜索范围 = 更快更省资源                    Shard 数量在创建时固定，不可动态修改
```

**大师**："核心区别就一句——**Partition 是给查询用的（缩小搜索范围），Shard 是给写入用的（提升写入并行度）。**"

| 维度 | Partition | Shard |
|------|-----------|-------|
| 类型 | 逻辑分区 | 物理分片 |
| 目的 | 查询时缩小数据扫描范围 | 写入时分散数据流量 |
| 创建时机 | 随时可创建/删除 | Collection 创建时设定，不可更改 |
| 数量限制 | 最多 4096 个 | 建议 1-16 个（Standalone 通常 1 个） |
| 搜索指定 | 可指定特定 Partition 搜索 | 对用户透明，系统自动路由 |
| 主键唯一性 | 跨 Partition 全局唯一 | N/A（同属 Collection） |

**小白**："那 Shard 数量设多少合适？是不是越多越好？"

**大师**："不是。Shard 每增加一个，意味着多一个消息队列 Channel，对应多一组 DataNode 消费。设太大有两个坏处：第一，管理开销增加；第二，小数据量下 Shard 太多，每个 Shard 上的 Segment 太小太碎，搜索时 QueryNode 要加载大量小 Segment，反而影响性能。"

**大师**："经验公式：**Shard 数 = 写入 QPS / 5 万（取 2 的次幂）**。Standalone 下 1 个 Shard 完全够用。"

> **技术映射**：Partition = 图书馆的主题分区（帮你缩小寻找范围）；Shard = 图书馆的入口通道数（决定同时能进多少人）；Segment = 书架上的书（实际存储单元）。

**第二幕：租户隔离的三种方案**

**小胖**："那 500 个租户应该怎么隔离数据？建 500 个 Collection？500 个 Partition？还是全塞一个 Collection 里用 tenant_id 过滤？"

**大师**："这是多租户架构的经典三选一，各有代价——"

| 方案 | 隔离方式 | 优点 | 缺点 | 适用租户数 |
|------|---------|------|------|-----------|
| **Collection 隔离** | 每个租户一个 Collection | 完全物理隔离；独立索引和 Load | 管理成本高；QueryNode 内存压力大 | < 50 |
| **Partition 隔离** | 每个租户一个 Partition | 逻辑隔离；按需 Load 指定 Partition | 索引共用；Partition 上限 4096 | 50-2000 |
| **字段过滤隔离** | 同一 Collection 用 tenant_id 过滤 | 实现最简单；无额外管理开销 | 搜索要扫全量数据再过滤；大租户影响小租户 | 不限但性能差 |

**大师**："对于你这个 500 租户的知识库平台，推荐 **Partition 隔离**方案。理由——"

1. 搜索时 `Partition("tenant_123")` 可以把搜索范围从 2500 万缩小到 5 万，延迟降低 99%。
2. 不需要 Load 全部 500 个 Collection（那样 QueryNode 内存撑不住）。
3. 可以按租户的热度灵活管理——大租户的 Partition 常驻内存，小租户按需 Load。
4. 备份和迁移可以按 Partition 粒度进行。

**小白**："那 Partition 隔离有什么坑需要注意吗？"

**大师**："有两个致命陷阱——"

**陷阱 1：Partition 过多导致管理开销爆炸**
每个 Partition 都有一组元数据（记录在 etcd 中）。500 个 Partition × 每个 50 个 Segment = 25000 个元数据条目。etcd 的存储和处理能力有限，Partition 数量超过 1000 时，etcd 可能成为瓶颈。

**陷阱 2：忘了指定 Partition 等于没隔离**
如果你写 `collection.search(expr="...")` 而不指定 Partition，它会搜索所有 Partition。如果你知道只搜一个租户但忘了指定 Partition，等于没利用 Partition 隔离的优势。

**小胖**："那我能不能把冷门租户的 Partition Release 掉？搜的时候再 Load？"

**大师**："可以！这就是'按需加载'策略——"

```python
# 热租户：服务启动时 Load
collection.load(partition_names=["tenant_big_1", "tenant_big_2"])

# 冷租户：搜索前按需 Load，搜完 Release
if not is_loaded("tenant_cold_99"):
    collection.load(partition_names=["tenant_cold_99"])
results = collection.search(..., partition_names=["tenant_cold_99"])
collection.release(partition_names=["tenant_cold_99"])
```

> **技术映射**：Collection 隔离 = 每人一间独立书房（奢华但空间不够）；Partition 隔离 = 大阅览室里的独立书桌（经济实用）；字段过滤 = 所有人共用一张大桌子（最省事但最乱）。

**第三幕：Growing Segment 与 Sealed Segment——为什么写入一会儿快一会儿慢**

**小胖**："大师，我做写入压测时发现一个诡异现象——前 10 秒写入吞吐 1 万条/秒，但 10 秒后突然降到 500 条/秒！这是为什么？"

**大师**："你碰到的是 Segment 的生命周期切换——从 Growing 到 Sealed。"

**大师**（画 Segment 状态机）：

```
┌──────────┐    数据写入     ┌──────────┐   Flush    ┌──────────┐
│ Growing  │ ───────────────→│ Growing  │ ────────→ │  Sealed  │
│ (内存)   │                 │ (内存满) │            │ (磁盘)   │
└──────────┘                 └──────────┘            └──────────┘
  写入快（内存操作）              触发 Flush               写入停！
  ~1万条/s                    DataNode 序列化            ~500条/s
                              Segment → Binlog         (等新 Growing Segment)
```

**大师**："Growing Segment 写内存，快；一旦触发 Flush 变成 Sealed Segment，这个写通道就'封口'了，DataCoord 需要分配新的 Growing Segment。如果分配不及时，就会出现写入暂停——也就是你说的吞吐骤降。"

**小白**："那怎么优化？"

**大师**："三个调优方向——"

1. **增大 Segment 大小阈值**：默认 Growing Segment 达到 1GB 或 100 万行触发 Flush。但注意，Segment 太大 Load 慢。
2. **增加 Shard 数**：Shard 多意味着并行写入通道多，一个 Shard 的 Segment 封口时其他 Shard 还在写。
3. **预分配 Growing Segment**：让 DataCoord 提前准备多个空 Segment 备用。

**小胖**："那搜索的时候 Growing 和 Sealed 表现有区别吗？"

**大师**："有，这是关键——"

| 搜索阶段 | Growing Segment | Sealed Segment |
|---------|----------------|----------------|
| 索引 | 无索引（暴力检索） | 有索引（HNSW/IVF） |
| 速度 | 慢（O(N)） | 快（O(log N)） |
| 数据量 | 小（通常 < 100 万行） | 大（已积累） |
| 总体影响 | 微弱（占比通常 < 10%） | 占搜索延迟的大头 |

"所以在生产环境，Growing Segment 对搜索延迟的影响通常不大——除非你的 Flush 策略太激进导致 Growing Segment 占比过高。"

> **技术映射**：Growing Segment = 餐厅开放厨房（随点随做）；Sealed Segment = 中央厨房预制菜（标准化包装）；Flush = 把开放厨房的菜标准化后送入中央厨房。

#### 步骤 1：按类目创建 Partition 并写入数据

```python
# step1_partition_setup.py
"""按类目创建 Partition 并写入模拟数据"""
import random
from pymilvus import connections, Collection, utility, Partition
from pymilvus import CollectionSchema, FieldSchema, DataType
from sentence_transformers import SentenceTransformer

connections.connect(host="localhost", port="19530")

COLL_NAME = "product_partition_demo"

if utility.has_collection(COLL_NAME):
    utility.drop_collection(COLL_NAME)

fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=256),
    FieldSchema(name="title_vec", dtype=DataType.FLOAT_VECTOR, dim=384),
    FieldSchema(name="price", dtype=DataType.FLOAT),
    FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="in_stock", dtype=DataType.BOOL),
]

schema = CollectionSchema(fields, description="分区实战")
collection = Collection(COLL_NAME, schema)

# 创建类目 Partition
categories = ["户外运动", "数码电子", "家居日用", "服饰鞋包", "食品饮料"]
partitions = {}
for cat in categories:
    pname = f"cat_{cat}"
    partitions[cat] = Partition(collection, pname)
    partitions[cat].create()
    print(f"> Partition '{pname}' 已创建")

# 按 Partition 写入数据
model = SentenceTransformer("all-MiniLM-L6-v2")
counter = 1

for cat in categories:
    batch_data = []
    for _ in range(1000):
        batch_data.append({
            "id": counter,
            "title": f"{cat} {random.choice(['便携','防水','轻便','大容量'])} {random.choice(['商品A','商品B','商品C'])}",
            "price": round(random.uniform(9.9, 999.9), 2),
            "category": cat,
            "in_stock": random.choice([True, False]),
        })
        counter += 1

    titles = [p["title"] for p in batch_data]
    embeddings = model.encode(titles).tolist()

    collection.insert(
        [
            [p["id"] for p in batch_data],
            titles,
            embeddings,
            [p["price"] for p in batch_data],
            [p["category"] for p in batch_data],
            [p["in_stock"] for p in batch_data],
        ],
        partition_name=f"cat_{cat}"
    )
    utility.flush([COLL_NAME])
    print(f"> Partition 'cat_{cat}': 写入 {len(batch_data)} 条")

print(f"\n> 共写入 {collection.num_entities} 条, {len(partitions)} 个 Partition")

# 构建索引
collection.create_index("title_vec", {
    "index_type": "HNSW", "metric_type": "COSINE",
    "params": {"M": 16, "efConstruction": 200}
})
utility.wait_for_index_building_complete(COLL_NAME, timeout=120)
```

#### 步骤 2：对比全库搜索 vs 指定 Partition vs Expr 过滤

```python
# step2_partition_benchmark.py
"""对比三种搜索策略的延迟差异"""
import time
from pymilvus import Collection, connections
from sentence_transformers import SentenceTransformer

connections.connect(host="localhost", port="19530")
model = SentenceTransformer("all-MiniLM-L6-v2")
collection = Collection("product_partition_demo")
collection.load()

query_text = "户外露营帐篷"
query_vec = model.encode([query_text]).tolist()
sp = {"metric_type": "COSINE", "params": {"ef": 64}}

print("=" * 60)
print(f"查询: {query_text}  (10轮取平均)")
print("=" * 60)
print(f"{'策略':<35} {'延迟(ms)':<10} {'结果数':<8}")
print("-" * 55)

# 策略 1: 全库搜索
times1 = []
for _ in range(10):
    t0 = time.time()
    r1 = collection.search(data=query_vec, anns_field="title_vec",
        param=sp, limit=10, output_fields=["id", "category"])
    times1.append((time.time() - t0) * 1000)
print(f"{'全库搜索（无过滤）':<35} {sum(times1)/len(times1):<10.2f} {len(r1[0]):<8}")

# 策略 2: 指定 Partition
times2 = []
for _ in range(10):
    t0 = time.time()
    r2 = collection.search(data=query_vec, anns_field="title_vec",
        param=sp, limit=10, partition_names=["cat_户外运动"],
        output_fields=["id", "category"])
    times2.append((time.time() - t0) * 1000)
print(f"{'指定 Partition（户外运动）':<35} {sum(times2)/len(times2):<10.2f} {len(r2[0]):<8}")

# 策略 3: Expr 过滤（不指定 Partition）
times3 = []
for _ in range(10):
    t0 = time.time()
    r3 = collection.search(data=query_vec, anns_field="title_vec",
        param=sp, limit=10,
        expr="category == 户外运动",
        output_fields=["id", "category"])
    times3.append((time.time() - t0) * 1000)
print(f"{'Expr 过滤（category==户外运动）':<35} {sum(times3)/len(times3):<10.2f} {len(r3[0]):<8}")

print("\n> 结论: 指定 Partition 跳过无关数据分区，搜索范围最小，延迟最低。")
```

```python
# step3_on_demand_load.py
"""演示冷门 Partition 的按需加载"""
import time
from pymilvus import Collection, Partition, connections
from sentence_transformers import SentenceTransformer

connections.connect(host="localhost", port="19530")
model = SentenceTransformer("all-MiniLM-L6-v2")

# 模拟冷数据 Partition：Release 掉食品饮料分区
cold_partition = "cat_食品饮料"
print(f"> Release 冷 Partition: {cold_partition}")
collection.release(partition_names=[cold_partition])

# 搜索时按需 Load
query_text = "休闲零食推荐"
query_vec = model.encode([query_text]).tolist()
sp = {"metric_type": "COSINE", "params": {"ef": 64}}

print(f"\n> 搜索: {query_text}")

# 步骤 1: 按需 Load 指定 Partition
t0 = time.time()
collection.load(partition_names=[cold_partition])
load_time = time.time() - t0
print(f"  Load {cold_partition} 耗时: {load_time:.2f}s")

# 步骤 2: 搜索
t0 = time.time()
results = collection.search(
    data=query_vec, anns_field="title_vec",
    param=sp, limit=5,
    partition_names=[cold_partition],
    output_fields=["id", "title", "price"]
)
search_time = (time.time() - t0) * 1000
print(f"  搜索耗时: {search_time:.1f}ms, 结果数: {len(results[0])}")
for i, hit in enumerate(results[0]):
    print(f"    #{i+1}: {hit.entity.get('title')[:30]}, ${hit.entity.get('price'):.1f}")

# 步骤 3: 搜完释放
collection.release(partition_names=[cold_partition])
print(f"\n> 已 Release {cold_partition}，内存释放完成")
```

#### 步骤 4：展示多租户分区管理最佳实践

```python
# step4_tenant_manager.py
"""多租户 Partition 管理最佳实践"""
from pymilvus import Collection, Partition, utility

class TenantPartitionManager:
    """租户分区管理器"""
    
    def __init__(self, collection_name: str):
        self.collection = Collection(collection_name)
    
    def create_tenant_partition(self, tenant_id: str):
        """为新租户创建独立 Partition"""
        pname = f"tenant_{tenant_id}"
        if utility.has_partition(self.collection.name, pname):
            return
        partition = Partition(self.collection, pname)
        partition.create()
        return pname
    
    def get_tenant_partition(self, tenant_id: str) -> str:
        """获取租户 Partition 名称"""
        return f"tenant_{tenant_id}"
    
    def search_tenant(self, query_vec, tenant_id: str, limit: int = 10):
        """在指定租户 Partition 中搜索（缩小搜索范围）"""
        pname = self.get_tenant_partition(tenant_id)
        return self.collection.search(
            data=query_vec,
            anns_field="title_vec",
            param={"metric_type": "COSINE", "params": {"ef": 64}},
            limit=limit,
            partition_names=[pname],  # 只搜该租户
            output_fields=["title", "price"],
        )
    
    def load_hot_tenants(self, tenant_ids: list):
        """启动时只加载热租户"""
        pnames = [self.get_tenant_partition(tid) for tid in tenant_ids]
        self.collection.load(partition_names=pnames)
    
    def release_cold_tenant(self, tenant_id: str):
        """释放冷租户内存"""
        self.collection.release(
            partition_names=[self.get_tenant_partition(tenant_id)]
        )

# 使用示例
mgr = TenantPartitionManager("product_partition_demo")
mgr.create_tenant_partition("company_a")
mgr.create_tenant_partition("company_b")
mgr.load_hot_tenants(["company_a"])  # 只加载热租户
print("> 热租户 company_a 已加载，冷租户 company_b 按需加载")
```

## 4. 项目总结

### 4.1 优缺点对比

| 维度 | 无分区（单 Collection） | Partition 隔离 | Collection 隔离 |
|------|----------------------|---------------|----------------|
| 搜索延迟 | 高（全量扫描） | 低（缩小范围） | 低 |
| 管理复杂度 | 低 | 中 | 高（N 倍 Collection 管理） |
| 内存效率 | 低（全量常驻） | 高（按需 Load） | 中（按需 Load Collection） |
| 资源隔离 | 无 | 逻辑隔离 | 完全物理隔离 |
| 扩容灵活性 | 低 | 中 | 高 |
| 租户上限 | 不限 | < 4096 | < 100 |

### 4.2 适用场景

- **多租户 SaaS 平台**：每个租户一个 Partition（本章核心场景）
- **按时间分区**：按月份/季度分区，搜索时可以只搜最近几个月
- **按地域分区**：多数据中心部署时按地域分区
- **A/B 测试**：实验组和对照组各一个 Partition

**不适用场景**：租户数超过 1000（etcd 元数据压力太大）、需要频繁跨分区搜索的场景。

### 4.3 注意事项

- **Shard 数量创建即固定**：不可后期修改，需在创建 Collection 前规划好。
- **Partition 不是越多越好**：超过 500 个 Partition 会产生显著的 etcd 开销。
- **主键全局唯一**：跨 Partition 的主键必须唯一——不能两个 Partition 里有相同主键。
- **索引参数共享**：同一个 Collection 下所有 Partition 共用同一套索引参数。

### 4.4 常见踩坑经验

1. **忘记在搜索时指定 partition_names**：代码里写了 Partition 隔离但搜索时没传参数，等于白隔离了。
2. **在热租户上频繁 Load/Release**：每次都重新 Load 的延迟远高于常驻内存，热租户应一直保持 Load 状态。
3. **Shard 设为 16 但数据只有 10 万条**：16 个 Shard 产生了 16 个 Channel，大量小 Segment 严重影响搜索性能。

### 4.6 推广计划提示

- **开发团队**：在服务启动时配置热租户列表，自动 Load 对应 Partition。
- **运维团队**：监控每个 Partition 的 Segment 数量和 Load 状态，发现大量小 Segment 及时触发 Compaction。
