# 第22章：分区键、动态字段与多租户隔离

> **定位**：面向真实 SaaS 业务设计数据隔离方案。
> **版本**：Milvus 2.5.x
> **源码关联**：internal/rootcoord/partition.go、internal/proxy/partition_key.go、pkg/v2/proto/schema.pb.go

---

## 1. 项目背景

某 SaaS 知识库平台已接入 800 个企业租户，每个租户平均 3 万条 Chunk。初期团队用了最简单的方案：一个 Collection 存所有租户，搜索时用 `tenant_id == 'xxx'` 做 Expr 过滤。随着租户增长，问题像多米诺骨牌一样倒下来：

1. **搜索延迟失控**：每次搜索都要扫描全部 2400 万条数据再过滤，即使只搜单个租户的 3000 条。P95 延迟从 30ms 涨到 600ms。
2. **大租户拖垮小租户**：某大租户（50 万 Chunk）的 Segment 占了大半内存，小租户的 Segment 被挤出 QueryNode 内存，搜索时需要重新从磁盘加载。
3. **权限隔离不够**：虽然有 Expr 过滤，但开发在日志里打印了搜索结果，泄露了其他租户的商品数据——Collection 级别的隔离不彻底。
4. **备份不灵活**：一个租户要求"恢复 3 天前的数据"，但备份是 Collection 级的，恢复意味着把所有租户的数据回滚 3 天。

架构师决定引入 **Partition Key**（分区键）——Milvus 2.4+ 的新特性，让数据按 `tenant_id` 自动分区，搜索时自动路由到对应分区，无需手动管理 Partition。

---

## 2. 项目设计（剧本式交锋对话）

**第一幕：Partition Key —— 自动分区的革命**

*（架构师张工在白板上对比"手动 Partition"和"Partition Key"两种方案）*

**小胖**（困惑地）："Partition Key 和普通 Partition 有啥区别？不就是换个名字吗？"

**大师**："区别巨大。手动 Partition 是你自己管理——创建、命名、维护分区。Partition Key 是 Milvus 帮你管理——你只需指定'按哪个字段分区'，Milvus 自动帮你做一切。"

**大师**（画对比图）：

```
手动 Partition（旧方式）            Partition Key（新方式）
───────────────────────            ──────────────────────
# 自己创建 800 个 Partition        # 只需在 Schema 中声明一次
for tenant in 800_tenants:         schema = CollectionSchema(
    Partition(col, f"t_{tenant}").create()     fields, partition_key_field="tenant_id")

# 写入时手动指定 Partition        # 写入时 Milvus 自动路由
collection.insert(data,             collection.insert(data)
    partition_name=f"t_{tenant}"   # ← 不需要指定 partition_name!
)
                                    # Milvus 根据 tenant_id 的值
# 搜索时手动指定 Partition          # 自动分配到对应分区
collection.search(...,             # 搜索时同理，Expr 中有 tenant_id
    partition_names=[f"t_{tenant}"] # 自动路由到对应分区
)
```

| 对比维度 | 手动 Partition | Partition Key |
|---------|---------------|---------------|
| 分区创建 | 需手动为每个租户创建 | Schema 声明一次即可 |
| 分区数量 | 受 etcd 元数据限制（< 4096） | 同样受限，但管理自动化 |
| 写入路由 | 需显式指定 `partition_name` | 自动按 Partition Key 值哈希路由 |
| 搜索路由 | 需显式指定 `partition_names` | Expr 中包含 Partition Key 字段时自动路由 |
| 数据均衡 | 依赖人工规划 | 基于哈希，天然均衡 |

**小白**："那 Partition Key 底层也是 Partition？还是新的数据结构？"

**大师**："底层还是 Partition，但带了自动哈希路由能力。Partition Key 的值经过哈希后，映射到固定的几个物理 Partition 中。好处是你不需要关心多少个租户、租户叫什么名——Milvus 全自动处理。"

> **技术映射**：手动 Partition = 自己整理书架（每本书手动放）；Partition Key = 智能书架（你只需要把书放上去，书架自动按标签归类）。

---

**第二幕：多租户隔离——三种方案的深度对比**

**小胖**："那 800 个租户到底用哪种隔离方案？Collection 隔离、Partition 隔离、还是字段过滤？"

**大师**："给你一份升级版决策表——"

| 方案 | 隔离强度 | 管理成本 | 最大租户数 | 推荐场景 |
|------|---------|---------|-----------|---------|
| **Collection 隔离** | 完全物理隔离 | 高（N个Collection×N个操作） | < 50 | 金融/医疗等强合规场景 |
| **Partition 隔离**（手动） | 逻辑隔离 | 中（需手动管理Partition） | 50-500 | 中等租户数的 SaaS |
| **Partition Key 隔离** | 逻辑隔离 + 自动路由 | 低（只需声明一次） | 50-4096 | **推荐：大多数 SaaS 场景** |
| **字段过滤** | 弱（仅查询层隔离） | 最低 | 不限 | POC 验证（不建议生产） |

**大师**："Partition Key 是最优解，但有一个重要限制——"

```python
# Partition Key 限定一个 Collection 最多 4096 个实际 Partition
# 如果 800 个租户 × 每个租户不同 partition_key 值
# → Milvus 内部哈希到 N 个物理 Partition（N 创建时设定）
# → 默认 N=16，意味着 800 个租户的数据分布在 16 个 Partition 中
# → 搜索时即使只搜一个租户，Expr "tenant_id=='xxx'" 自动路由到该租户所在的 Partition

# ❌ 常见错误：把 Partition Key 设成高基数字段（如 user_id）
# Partition Key 适用于: tenant_id / region / department (基数 < 4096)
# Partition Key 不适用于: user_id / order_id (基数可达百万级)
```

**小白**："那我怎么决定 Partition Key 映射到几个物理 Partition？"

**大师**："用 `num_partitions` 参数——创建 Collection 时指定。对于 800 个租户，建议 `num_partitions=64` 或 `128`，确保数据均匀分布。"

> **技术映射**：Collection 隔离 = 每人一个独立保险柜（最安全但成本最高）；Partition Key = 大保险柜里的分类抽屉（安全+低成本+自动分类）；字段过滤 = 所有东西扔一个大箱子里（最便宜但最乱）。

---

**第三幕：动态字段——便利与代价**

**小胖**："那 Dynamic Field 要不要开？不开的话 Schema 定死了很不灵活——"

**大师**："Dynamic Field 就像'随意堆放区'——可以临时放东西，但长期管理成本高。"

```python
# 开启 Dynamic Field
schema = CollectionSchema(fields, enable_dynamic_field=True)

# 写入时可以携带 Schema 中没有定义的字段
collection.insert([
    ...,
    [{"color": "red", "material": "cotton"}],  # 动态字段，JSON 格式
    ...
])

# 搜索时也可以用动态字段过滤
collection.search(..., expr='color == "red"')
```

**大师**："Dynamic Field 的利弊——"

| 优点 | 缺点 |
|------|------|
| Schema 灵活，新增字段不重建 Collection | 不能为动态字段建标量索引（搜索慢） |
| 适合快速迭代的 POC 阶段 | 类型推断可能出错（"123" 被存成字符串而非数字） |
| 减少数据迁移次数 | 动态字段占用额外 JSON 解析开销 |

**大师**："最佳实践：**POC 阶段开 Dynamic Field 快速迭代，上线生产前关闭，把常用过滤字段升级为显式 FieldSchema。**"

#### 步骤 1：创建带 Partition Key 的 Collection

```python
# step1_partition_key_schema.py
"""创建带 Partition Key 的多租户 Collection"""
from pymilvus import connections, Collection, utility
from pymilvus import CollectionSchema, FieldSchema, DataType

connections.connect(host="localhost", port="19530")

COLL_NAME = "multi_tenant_kb"
if utility.has_collection(COLL_NAME):
    utility.drop_collection(COLL_NAME)

fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                is_partition_key=True),  # ★ 声明为 Partition Key
    FieldSchema(name="chunk_text", dtype=DataType.VARCHAR, max_length=2048),
    FieldSchema(name="chunk_vec", dtype=DataType.FLOAT_VECTOR, dim=384),
    FieldSchema(name="doc_title", dtype=DataType.VARCHAR, max_length=256),
    FieldSchema(name="created_at", dtype=DataType.INT64),
]

schema = CollectionSchema(
    fields,
    description="多租户知识库 - Partition Key 隔离",
    enable_dynamic_field=False,  # 生产环境关闭
    num_partitions=64,           # ★ 物理分区数：64 个
)

collection = Collection(COLL_NAME, schema)
print(f"Collection {COLL_NAME} 创建完成, num_partitions=64")

# 建索引
collection.create_index("chunk_vec", {
    "index_type": "HNSW", "metric_type": "COSINE",
    "params": {"M": 16, "efConstruction": 200}
})
utility.wait_for_index_building_complete(COLL_NAME, timeout=120)
collection.load()
```

#### 步骤 2：多租户写入 + 自动路由验证

```python
# step2_tenant_write.py
"""验证 Partition Key 的自动路由"""
import random
import numpy as np
from pymilvus import Collection
from sentence_transformers import SentenceTransformer

collection = Collection("multi_tenant_kb")
model = SentenceTransformer("all-MiniLM-L6-v2")

# 模拟 10 个租户写入数据
tenants = [f"company_{i:03d}" for i in range(1, 11)]
total = 0

for tenant in tenants:
    batch = []
    for j in range(100):  # 每租户 100 条
        batch.append({
            "id": total + j + 1,
            "tenant_id": tenant,
            "chunk_text": f"{tenant} 的文档片段 {j} - {random.choice(['合同条款','技术规范','HR政策'])}",
            "doc_title": f"{tenant}_doc_{j//5}",
            "created_at": 1700000000 + random.randint(0, 86400*365),
        })
    total += len(batch)
    
    texts = [b["chunk_text"] for b in batch]
    embeddings = model.encode(texts, batch_size=64).tolist()
    
    collection.insert([
        [b["id"] for b in batch],
        [b["tenant_id"] for b in batch],  # Partition Key 字段
        texts,
        embeddings,
        [b["doc_title"] for b in batch],
        [b["created_at"] for b in batch],
    ])
    # 注意: 没有指定 partition_name！Milvus 自动路由
    print(f"  {tenant}: 写入 100 条（自动路由）")

collection.flush()
print(f"\n总数据: {collection.num_entities} 条, 分布 {len(tenants)} 个租户")

# 检查分区自动创建
for p in collection.partitions:
    print(f"  Partition: {p.name} ({p.num_entities} 条)")
```

#### 步骤 3：租户级搜索 + 性能对比

```python
# step3_tenant_search.py
"""验证 Partition Key 的搜索自动路由 + 性能"""
import time
import numpy as np
from pymilvus import Collection
from sentence_transformers import SentenceTransformer

collection = Collection("multi_tenant_kb")
model = SentenceTransformer("all-MiniLM-L6-v2")

query_text = "员工年假申请流程"
query_vec = model.encode([query_text]).tolist()

# 测试 1: 不指定租户（全库搜索）——对比基线
times_all = []
for _ in range(10):
    t0 = time.time()
    r = collection.search(
        data=query_vec, anns_field="chunk_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        limit=5, output_fields=["tenant_id", "chunk_text"]
    )
    times_all.append((time.time() - t0) * 1000)

# 测试 2: 指定租户（Partition Key 自动路由）
times_tenant = []
target_tenant = "company_005"
for _ in range(10):
    t0 = time.time()
    r = collection.search(
        data=query_vec, anns_field="chunk_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        expr=f'tenant_id == "{target_tenant}"',  # ★ Partition Key 自动路由
        limit=5, output_fields=["tenant_id", "chunk_text"]
    )
    times_tenant.append((time.time() - t0) * 1000)

print(f"全库搜索平均延迟: {np.mean(times_all):.2f}ms")
print(f"指定租户平均延迟: {np.mean(times_tenant):.2f}ms")
print(f"加速比: {np.mean(times_all)/np.mean(times_tenant):.1f}x")
print(f"\n> Partition Key 自动路由让指定租户的搜索跳过了无关分区")
```

"""演示动态字段的正确使用方式"""
from pymilvus import Collection, CollectionSchema, FieldSchema, DataType

# 场景：POC 阶段用动态字段存储灵活的租户自定义属性
COLL_POC = "tenant_poc"

fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="tenant_id", dtype=DataType.VARCHAR, max_length=64,
                is_partition_key=True),
    FieldSchema(name="chunk_vec", dtype=DataType.FLOAT_VECTOR, dim=384),
    FieldSchema(name="chunk_text", dtype=DataType.VARCHAR, max_length=2048),
]

# POC 开启 dynamic field
schema = CollectionSchema(fields, enable_dynamic_field=True,
                          description="POC阶段-允许动态字段")
collection = Collection(COLL_POC, schema)

# 写入时携带 Schema 外的动态字段
collection.insert([
    [1],
    ["company_001"],
    [[0.1] * 384],
    ["测试文档"],
    # 动态字段（JSON 格式）:
    [{"custom_tag": "重要", "priority": 3, "department": "法务部"}],
])

# 可以用动态字段做过滤
r = collection.search(
    data=[[0.1]*384], anns_field="chunk_vec",
    param={"metric_type": "COSINE", "params": {"ef": 16}},
    expr='custom_tag == "重要" and priority >= 2',
    limit=5, output_fields=["*"]
)
print(f"动态字段过滤结果: {len(r[0])} 条")
print(f"> ⚠ POC 结束后建议关闭 enable_dynamic_field，将常用字段升级为显式 FieldSchema")

#### 步骤 5：多租户权限过滤实战

```python
# step5_tenant_acl.py
"""多租户场景下的权限过滤最佳实践"""
from pymilvus import Collection, connections

connections.connect(host="localhost", port="19530")
collection = Collection("multi_tenant_kb")

# ── 场景 1: 单租户搜索（最简单） ──
def search_tenant_1(tenant_id: str, query_vec, top_k=10):
    """直接搜索指定租户的数据"""
    return collection.search(
        data=query_vec,
        anns_field="chunk_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        expr=f'tenant_id == "{tenant_id}"',
        limit=top_k,
        output_fields=["chunk_text", "tenant_id", "doc_title"]
    )

# ── 场景 2: 多租户联合搜索（合作关系） ──
def search_multi_tenants(tenant_ids: list, query_vec, top_k=10):
    """多租户联合搜索"""
    tenants_expr = ", ".join(f'"{t}"' for t in tenant_ids)
    return collection.search(
        data=query_vec,
        anns_field="chunk_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        expr=f'tenant_id in [{tenants_expr}]',
        limit=top_k,
        output_fields=["chunk_text", "tenant_id", "doc_title"]
    )

# ── 场景 3: 租户 + 文档级别双重过滤 ──
def search_tenant_doc(tenant_id: str, doc_title: str, query_vec, top_k=10):
    """指定租户 + 指定文档"""
    return collection.search(
        data=query_vec,
        anns_field="chunk_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        expr=f'tenant_id == "{tenant_id}" and doc_title == "{doc_title}"',
        limit=top_k,
        output_fields=["chunk_text", "tenant_id", "doc_title"]
    )

# ── 场景 4: 带时间范围的租户搜索 ──
def search_tenant_with_time(tenant_id: str, query_vec,
                             start_ts: int, end_ts: int, top_k=10):
    """租户 + 时间范围（如查某租户近30天的文档）"""
    return collection.search(
        data=query_vec,
        anns_field="chunk_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        expr=f'tenant_id == "{tenant_id}" and created_at >= {start_ts} and created_at <= {end_ts}',
        limit=top_k,
        output_fields=["chunk_text", "tenant_id", "doc_title", "created_at"]
    )

print("> 多租户权限过滤模式:")
print("  ① 单租户 → Partition Key 自动路由，性能最优")
print("  ② 多租户 → 使用 IN 子句，自动路由到相关分区")
print("  ③ 租户+文档 → 双重过滤，精确定位")
print("  ④ 租户+时间 → 配合时间戳做数据生命周期管理")
```

## 4. 项目总结

### 4.1 优缺点对比

| 维度 | Partition Key | 手动 Partition | Collection 隔离 |
|------|--------------|---------------|----------------|
| 管理成本 | 极低（自动路由） | 中 | 高 |
| 搜索自动路由 | 是（Expr 中检测到 PK 字段） | 否（需手动指定） | N/A |
| 写入自动路由 | 是 | 否 | N/A |
| 数据均衡 | 哈希均衡 | 取决于设计 | 手动 |
| 最大分区数 | 4096 | 4096 | 不限（但管理受不了） |

### 4.2 适用场景

- **多租户 SaaS**：Partition Key = tenant_id（本章核心）
- **按地域分区**：Partition Key = region（如 cn-us-eu）
- **按时间分区 + Partition Key**：手动按月份分区 + Partition Key 按租户再分

**不适用场景**：超多租户（> 10000，etcd 压力）、租户间需要频繁跨分区搜索。

### 4.3 注意事项

- **Partition Key 只支持 VARCHAR 和 INT64 类型**。
- **num_partitions 创建后不可更改**。
- **Partition Key 的值不能为 NULL**。
- **dynamic_field 在生产中尽量关闭**：升级为显式 FieldSchema 并建标量索引。
