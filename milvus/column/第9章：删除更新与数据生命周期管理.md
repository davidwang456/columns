> **版本**：Milvus 2.5.x
> **源码关联**：internal/datacoord/compaction.go、internal/datanode/delete.go、internal/querynodev2/delete.go

## 1. 项目背景

商品运营团队找到张工，提出三个需求：1）每天有 200 个商品下架，需要从 Milvus 中删除对应向量；2）商品价格每天变动，需要更新价格字段；3）已下架超过 90 天的商品数据需要彻底清理以节省存储成本。

张工心想"不就是 DELETE FROM 和 UPDATE 吗"，调了一把 Milvus 的 delete 和 upsert 接口，就告诉运营搞定了。三天后，运营投诉：下架的商品还能搜到、更新的价格在搜索结果里是旧值、明明删了 200 条数据但磁盘占用一点没减少。

张工排查了一整天，才发现 Milvus 的数据管理模型和关系型数据库完全不同：

1. **删除不是即时生效**：Milvus 的 Delete 是标记删除（Soft Delete），数据在 Compaction 之前物理上仍然存在且能被搜索到（取决于一致性级别设置）。
2. **Upsert 不是原地更新**：Milvus 的 Upsert 实际上是"标记旧数据删除 + 插入新数据"，旧数据同样要等 Compaction 才能清理。
3. **磁盘不释放**：即使 Compaction 完成了，对象存储（MinIO/S3）中的旧文件也不会立刻删除，需要等 GC（Garbage Collection）周期到达。
4. **查询可见性**：删除后的数据能否被查到，取决于 Guarantee Timestamp 的设置——如果设置了 Strong 一致性，删完立刻搜不到；如果设置了 Bounded，可能要等一段时间。

更令人头疼的是，张工调了 3 天 `delete` 接口发现删了 10 万条数据后，Collection 的搜索性能反而下降了——原来是大量删除留下的 Delta Log 增加了每次搜索的过滤开销。

## 2. 项目设计（剧本式交锋对话）

**第一幕：Delete vs Upsert vs Compaction——数据变更的三重奏**

*（张工正在监控大屏前，磁盘使用率曲线一直在涨）*

**小胖**（挠头）："大师，我昨天删了 10 万条数据，磁盘一点没少！Milvus 是不是有 bug？"

**大师**："不是 bug，是设计。你理解的关系型数据库删除是'立刻从磁盘上擦除'，但 Milvus 的删除是'三步走'——"

**大师**（在白板上画时间线）：

```
时间线：Milvus 数据删除的完整生命周期

T0: Delete 请求到达
    │  Proxy 校验 → 写入 Delete Log (Delta Log)
    │  数据标记为"已删除"，但物理文件未变
    │
T1: 搜索时（取决于一致性级别）
    │  Strong: 立刻不可见 ✓
    │  Bounded: 可能需要等 ts = Guarantee Timestamp 之后才不可见
    │  Eventually: 可能仍可见一段时间
    │
T2: Compaction 触发
    │  DataCoord 检测到 Delete Log 积累到阈值
    │  触发 Compaction：合并 Segment + 物理删除已标记的数据
    │  生成新的 Sealed Segment（不含已删除数据）
    │
T3: GC 周期到达
    │  旧 Segment 文件和 Delta Log 进入 GC 队列
    │  等待 retention 时间（默认 10 分钟）后被对象存储删除
    │  磁盘空间终于释放 ✓
    
    全程耗时: T0 → T3 = 分钟级到小时级（取决于数据量和配置）
```

**小白**："所以软删除和硬删除的区别就是——软删除只打标记，硬删除才真正清理文件？"

**大师**："没错。更准确地说——"

| 概念 | Milvus 中的实现 | 何时发生 |
|------|----------------|---------|
| **软删除（Soft Delete）** | 写入 Delta Log，标记数据不可见 | `delete()` 调用后立即生效 |
| **Compaction** | 物理合并 Segment，丢弃已标记删除的行 | DataCoord 自动触发（根据 compaction 策略） |
| **GC** | 删除对象存储中的过期文件 | 周期性后台任务（默认 30 分钟检查一次） |

**小胖**："那 Upsert 呢？我更新商品价格的时候，它是直接覆盖还是删了再写？"

**大师**："Upsert 的底层实现比你想象的复杂。它不是'原地修改'，而是——"

```
Upsert(id=123, price=299) 的实际执行:

Step 1: 查找主键 123 是否存在
        如果存在 → 标记旧记录为"已删除"（写入 Delta Log）
        如果不存在 → 跳过此步

Step 2: 插入新记录 (id=123, price=299)
        写入 Growing Segment（此时有两份 id=123 的数据：一份旧的已标记删除，一份新的活跃）

Step 3: Compaction 之后
        旧记录被物理清除，只剩一条 (id=123, price=299)
```

**小白**："那如果 Upsert 之后还没 Compaction，搜索能看到两条吗？"

**大师**："看不到。因为搜索时 QueryNode 会先加载 Delta Log，过滤掉已标记删除的记录。所以你只会看到最新的那条。但从存储角度看，旧数据确实还占着磁盘。"

> **技术映射**：软删除 = 图书馆里把书标记为"报废"但还没搬走；Compaction = 员工定期清理报废书并整理书架；GC = 清洁工把报废书拉走回收；Upsert = 先标记旧书报废再放本新的在架子上。

**第二幕：主键删除 vs 表达式删除——怎么删又快又准**

**小胖**："那删除到底有几种方式？我总是分不清——"

**大师**："两种删除方式，适用场景完全不同——"

```python
# 方式 1: 按主键删除（精确删除）
collection.delete(expr="id in [1, 2, 3, 4, 5]")

# 方式 2: 按表达式删除（条件删除）
collection.delete(expr="category == '已下架' and in_stock == false")
```

| 删除方式 | 语法 | 性能 | 适用场景 | 注意事项 |
|---------|------|------|---------|---------|
| **主键删除** | `id in [1, 2, 3]` 或 `id == 123` | 快（直接定位） | 业务侧已知要删除的 ID 列表 | 主键列表不宜过大（单次 < 10000 个） |
| **表达式删除** | `price > 1000 or category == '过期'` | 慢（需扫描） | 按条件批量清理过期数据 | 复杂表达式可能触发全扫描 |

**大师**："生产环境有三条删除铁律——"

1. **优先用主键删除**：如果你知道要删哪些 ID，直接传 ID 列表。主键删除走索引，O(1) 定位；表达式删除走扫描，O(N)。
2. **单次删除量控制在 10000 条以内**：超过这个量，建议分批删除。否则 Delta Log 文件过大，Compaction 吃紧。
3. **删除后不要立刻搜**：如果业务允许，建议删除操作和搜索操作之间间隔至少一个 Compaction 周期（通常是数分钟），给系统时间把数据从"软删除"过渡到"硬删除"。

**小白**："那如果我有 100 万条需要清理，怎么分批？"

**大师**："这种方式——"

```python
def batch_delete(collection, ids, batch_size=10000):
    """分批删除大量数据"""
    for i in range(0, len(ids), batch_size):
        batch = ids[i:i+batch_size]
        expr = f"id in {batch}"
        collection.delete(expr)
        if (i // batch_size + 1) % 10 == 0:
            print(f"  已删除 {i + len(batch)}/{len(ids)} 条")
```

> **技术映射**：主键删除 = 点名删除（精准快）；表达式删除 = 按条件筛查（范围广但慢）；分批删除 = 大扫除分批做（避免一次搬太多）。

**第三幕：TTL 数据过期、冷热分层与归档策略**

**小白**："大师，运营说要把 90 天以前的下架商品永久删除。除了手动删，还有别的办法吗？"

**大师**："当然有，Milvus 支持 **TTL（Time-To-Live）** 自动过期机制——"

```python
# 创建 Collection 时设置 TTL（单位：秒）
# 90 天 = 90 * 24 * 3600 = 7776000 秒
schema = CollectionSchema(fields, 
    enable_dynamic_field=False,
    ttl=7776000  # 数据写入 90 天后自动过期
)
```

**大师**："TTL 的工作原理：每条数据在写入时都会记录一个时间戳。当 `当前时间 - 写入时间 > TTL` 时，该数据会被自动标记为过期。Compaction 时会扫描并清理过期数据。"

**小白**："TTL 和手动删除有什么区别？"

| 维度 | 手动 Delete | TTL 自动过期 |
|------|-----------|------------|
| 触发方式 | 业务代码显式调用 | Milvus 后台自动扫描 |
| 精度 | 精确（想删哪条删哪条） | 粗粒度（按写入时间） |
| 适用场景 | 业务驱动的精确删除（如商品下架） | 数据保留策略（如日志类数据 30 天归档） |
| 系统开销 | 即时产生 Delta Log | 周期性产生清理开销 |

**大师**："但 TTL 不是万能的——如果你的数据不是按写入时间判断过期，而是按业务字段（如 `last_access_time`），那就得用手动删除。"

**小胖**："那冷热分层呢？我们有些老数据很少搜到，但又不舍得删——"

**大师**："冷热分层的核心思路：把数据按访问频率分为——"

```
热数据（近 7 天）        温数据（7-90 天）        冷数据（90 天+）
─────────────────      ─────────────────      ──────────────────
常驻 QueryNode 内存      按需 Load/Release       存储在对象存储中
HNSW 高性能索引           IVF_SQ8 压缩索引        不建索引或 DISKANN
Collection A              Collection B           Collection C
(按时间 Partition)        (按时间 Partition)      (按时间 Partition)
```

**大师**："具体做法是按时间建三个 Collection 或 Partitions。搜索时优先搜热数据，不够再搜温数据，冷数据只在极少量场景下按需加载。"

> **技术映射**：TTL = 食品保质期（到期自动清理）；手动删除 = 主动退货（你想退哪件退哪件）；冷热分层 = 冰箱管理（常用的放冷藏、不常用的放冷冻）。

```python
# step1_setup.py
"""创建测试 Collection 并写入 5000 条商品数据"""
import random, json
from pymilvus import connections, Collection, utility
from pymilvus import CollectionSchema, FieldSchema, DataType
from sentence_transformers import SentenceTransformer

connections.connect(host="localhost", port="19530")

COLL_NAME = "product_lifecycle_test"
if utility.has_collection(COLL_NAME):
    utility.drop_collection(COLL_NAME)

fields = [
    FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
    FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=256),
    FieldSchema(name="title_vec", dtype=DataType.FLOAT_VECTOR, dim=384),
    FieldSchema(name="price", dtype=DataType.FLOAT),
    FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64),
    FieldSchema(name="status", dtype=DataType.VARCHAR, max_length=32),  # online/offline
    FieldSchema(name="created_at", dtype=DataType.INT64),  # Unix timestamp
]

schema = CollectionSchema(fields, description="数据生命周期测试")
collection = Collection(COLL_NAME, schema)

model = SentenceTransformer("all-MiniLM-L6-v2")

# 生成并写入数据
products = []
for i in range(1, 5001):
    products.append({
        "id": i,
        "title": f"商品_{i}_" + random.choice(["便携折叠椅", "防水帐篷", "静音耳机", "大容量背包"]),
        "price": round(random.uniform(9.9, 999.9), 2),
        "category": random.choice(["户外运动", "数码电子", "家居日用"]),
        "status": "online",
        "created_at": 1700000000 + random.randint(0, 90*86400),
    })

for i in range(0, len(products), 500):
    batch = products[i:i+500]
    titles = [p["title"] for p in batch]
    embeddings = model.encode(titles).tolist()
    collection.insert([
        [p["id"] for p in batch], titles, embeddings,
        [p["price"] for p in batch], [p["category"] for p in batch],
        [p["status"] for p in batch], [p["created_at"] for p in batch],
    ])

utility.flush([COLL_NAME])

collection.create_index("title_vec", {
    "index_type": "HNSW", "metric_type": "COSINE",
    "params": {"M": 16, "efConstruction": 200}
})
utility.wait_for_index_building_complete(COLL_NAME, timeout=120)
collection.load()

print(f"✓ 写入 {collection.num_entities} 条数据并 Load 完成")
```

```python
# step2_offline_and_verify.py
"""模拟商品下架操作并验证搜索可见性"""
import time
from pymilvus import Collection, connections
from sentence_transformers import SentenceTransformer

connections.connect(host="localhost", port="19530")
model = SentenceTransformer("all-MiniLM-L6-v2")
collection = Collection("product_lifecycle_test")

# ---------- 1. 下架前搜索 ----------
query_text = "便携折叠椅"
query_vec = model.encode([query_text]).tolist()

print("=== 下架前搜索 ===")
results_before = collection.search(
    data=query_vec, anns_field="title_vec",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    limit=5, output_fields=["id", "title", "status"]
)
for i, hit in enumerate(results_before[0]):
    print(f"  #{i+1}: id={hit.id}, title={hit.entity.get('title')[:30]}, "
          f"status={hit.entity.get('status')}")

# ---------- 2. 批量下架（删除前 3 条搜索结果） ----------
ids_to_delete = [hit.id for hit in results_before[0][:3]]
print(f"\n正在下架 {len(ids_to_delete)} 个商品: {ids_to_delete}")

start = time.time()
collection.delete(expr=f"id in {ids_to_delete}")
print(f"  Delete 调用耗时: {(time.time()-start)*1000:.1f}ms")

# ---------- 3. 下架后立即搜索 ----------
print(f"\n=== 下架后立即搜索（可能仍可见！） ===")
results_after = collection.search(
    data=query_vec, anns_field="title_vec",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    limit=5, output_fields=["id", "title", "status"]
)
for i, hit in enumerate(results_after[0]):
    deleted = " [已删除但可能仍可见]" if hit.id in ids_to_delete else ""
    print(f"  #{i+1}: id={hit.id}, title={hit.entity.get('title')[:30]}{deleted}")

# ---------- 4. 使用 Query 验证删除状态 ----------
print(f"\n=== Query 验证被删除的数据 ===")
try:
    deleted_data = collection.query(
        expr=f"id in {ids_to_delete}",
        output_fields=["id", "title", "status"],
        consistency_level="Strong"
    )
    print(f"  Strong 一致性: 查询到 {len(deleted_data)} 条（应为 0 条）")
except Exception as e:
    print(f"  查询异常: {e}")

# ---------- 5. 搜索时排除已下架商品 ----------
print(f"\n=== 搜索 + 过滤（status='online'） ===")
results_filtered = collection.search(
    data=query_vec, anns_field="title_vec",
    param={"metric_type": "COSINE", "params": {"ef": 64}},
    expr="status == 'online'",
    limit=5, output_fields=["id", "title", "status"],
    consistency_level="Strong"
)
for i, hit in enumerate(results_filtered[0]):
    print(f"  #{i+1}: id={hit.id}, title={hit.entity.get('title')[:30]}, "
          f"status={hit.entity.get('status')}")
```

```python
import time

print("=== Compaction 触发前 ===")
segments = utility.get_query_segment_info(collection.name)
growing = len([s for s in segments if s.state_name == "Growing"])
sealed = len([s for s in segments if s.state_name == "Sealed"])
print(f"Growing Segments: {growing}")
print(f"Sealed Segments:  {sealed}")
print(f"总实体数: {collection.num_entities}")

# 模拟大量删除（删除 50% 的数据）
all_ids = [q["id"] for q in collection.query(
    expr="id >= 0", output_fields=["id"], limit=10000
)]
delete_count = len(all_ids) // 2
ids_to_delete = all_ids[:delete_count]

print(f"\n正在删除 {delete_count} 条数据...")
collection.delete(expr=f"id in {ids_to_delete}")
print(f"✓ Delete 完成")

# 手动触发 Compaction
print("\n正在触发 Compaction...")
try:
    collection.compact()
    time.sleep(3)
    print("  Compaction 已触发（后台异步执行）")
except Exception as e:
    print(f"  Compaction 触发: {e}（Milvus Standalone 可能不支持手动触发）")

print("\n=== Compaction 观察结论 ===")
print("1. Delete 后数据立即不可见（Strong 一致性下）")
print("2. 但物理存储不会立即释放")
print("3. 需要等 Compaction + GC 两个周期后磁盘空间才释放")

print("5. 大量 Delete 后搜索性能可能下降（Delta Log 过滤开销）")
```

## 4. 项目总结

### 4.1 优缺点对比

| 维度 | Milvus Delete/Upsert | MySQL DELETE/UPDATE | Elasticsearch Delete/Update |
|------|---------------------|--------------------|----------------------------|
| 即时生效 | 取决于一致性级别 | 事务提交后立刻 | Refresh 间隔后 |
| 物理清理 | Compaction + GC（分钟级） | 立刻（覆盖写） | Segment Merge（分钟级） |
| 磁盘释放 | 延迟（GC 周期） | 立刻（可能留下碎片） | 延迟（Merge） |
| Upsert 语义 | 标记删除 + 新增 | 原地更新（或 DELETE+INSERT） | 标记旧版 + 新增 |
| 大删除性能影响 | Delta Log 过滤开销 | 表锁 / 行锁 | Tombstone 过滤开销 |
| TTL 支持 | 原生 Collection 级 TTL | 需应用层实现 | ILM（Index Lifecycle Management） |

### 4.2 适用场景

- **商品上下架**：状态频繁变更的数据（本章示例）
- **数据合规**：GDPR "被遗忘权"——用户要求删除其所有数据
- **数据归档**：将历史数据迁移到低成本存储（配合 TTL）
- **增量同步**：从 MySQL 通过 Upsert 同步变更数据到 Milvus

**不适用场景**：需要事务保证的更新（Upsert 不是原子事务）、需要实时物理删除的场景（如果必须落盘即删，Milvus 不适合）。

### 4.3 注意事项

- **Upsert 需要提供完整字段**：Upsert 会覆盖整条记录，所有字段都要传入——不仅是变更的字段。如果需要部分字段更新，建议在业务层合并全量数据后再 Upsert。
- **一致性级别对删除可见性的影响**：Strong 一致性下删除立刻不可见；Bounded/Eventually 下有延迟。
- **Compaction 可手动触发但有代价**：可以调用 `compact()` 手动触发，但频繁 Compaction 会影响写入和搜索性能。
- **Delta Log 膨胀**：大量删除未 Compaction 时，每次搜索都要加载大量 Delta Log，导致搜索延迟增加。

### 4.4 常见踩坑经验

1. **Upsert 后向量变了，搜索召回率下降**：价格更新时，很多开发者只改价格字段而保留旧向量，看起来"省钱"。但如果商品标题也变了（比如加了个关键词），旧向量就不再准确。最佳实践：Upsert 时重新生成向量。
2. **删除 100 万条后搜索慢了 3 倍**：排查发现 Delta Log 文件超过 200MB，QueryNode 加载和过滤 Delta Log 成为瓶颈。解决方案：主动触发 Compaction 或在业务低峰期做删除操作。
3. **删除后又插入了相同主键的数据，但搜不到**：如果旧数据还没 Compaction，新数据的主键可能和"已删除但未清理"的旧数据冲突。用 Upsert 替代 Delete + Insert 可以避免这种问题。

### 4.5 思考题

1. 如果业务要求"删除操作必须在 5 秒内生效，且搜索结果中不能有任何已被删除的数据"，应该如何配置一致性级别和搜索参数？
2. Milvus 的 Compaction 和 LSM-Tree 的 Compaction（如 RocksDB）有何异同？为什么 Milvus 的 Compaction 不能像 RocksDB 那样频繁触发？
