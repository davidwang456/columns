# 第18章：写入链路深度解析

> **定位**：理解一条向量从 SDK 到存储落盘的全过程。
> **版本**：Milvus 2.5.x
> **源码关联**：internal/proxy/insert.go、internal/datacoord/channel_manager.go、internal/datanode/data_sync_service.go、internal/datanode/flush_manager.go

---

## 1. 项目背景

电商平台的数据管道团队负责将 MySQL 商品数据实时同步到 Milvus。日均 5 万条新增/变更商品，峰值 QPS 2000 写入。系统运行一段时间后出现诡异现象：写入延迟偶尔从 2ms 飙到 10 秒，但搜不到任何报错。

数据工程师小李开启全链路日志追踪后发现：

1. **Proxy 层耗时正常（2ms）**——请求校验和转发很快。
2. **消息队列层耗时正常（< 1ms）**——数据写入 Pulsar 很快返回 ACK。
3. **但是 DataNode 到 Flush 完成之间，时不时有 8-12 秒的"黑洞时间"**——没有任何日志输出，也没有报错。

深入排查发现：DataNode 在生成 Binlog 时，触发了对象存储的慢 IO（MinIO 磁盘负载高），但 DataNode 的代码中没有对这块做超时上报。更讽刺的是——写入在 Proxy 层就已经返回客户端"成功"了（因为 Proxy 只负责写到 MQ），用户完全不知道自己的数据其实卡在了存储层。

本章将追踪一条 Insert 请求从客户端到 Binlog 落盘的完整链路，包括 Proxy 分片、DataCoord Segment 分配、DataNode 缓冲写入、Flush 机制和异常处理。

## 2. 项目设计（剧本式交锋对话）

**第一幕：一条 Insert 请求的旅程——从 Proxy 到 MQ**

*（小李在白板上画了一条横跨 7 个组件的写入链路图）*

**小胖**（震惊地）："我就写了个 `collection.insert()`，数据居然要经过 7 个组件？！这延迟还不得上天？"

**大师**："延迟比你想象的小，因为大部分步骤是异步的。客户端收到'写入成功'的时候，数据其实只走完了前两步——"

**大师**（画时间线）：

```
客户端视角:  collection.insert(data) ──→ "写入成功" (2ms)
                    │
实际链路:  ┌────────────────────────────────────────────────────┐
           │ Step 1: Proxy 校验 + 分片         (0.5ms)          │
           │ Step 2: 写入 Message Queue          (1.0ms) ← 在这返回成功 │
           │ Step 3: DataNode 消费 MQ            (异步, 不确定)   │
           │ Step 4: DataNode 写入 Growing Seg   (异步)          │
           │ Step 5: Flush → Binlog → S3         (异步)          │
           │ Step 6: IndexNode 建索引            (异步)          │
           └────────────────────────────────────────────────────┘
           
  客户端只等 Step 1+2（同步），Step 3-6 全部异步！
```

**大师**："详细拆解每个 Step——"

```
Step 1: Proxy 校验 + 分片（同步，~0.5ms）
─────────────────────────────────────────
Proxy 收到 InsertRequest:
  ① 参数校验: 字段数量匹配？维度一致？主键合法？
  ② 向 RootCoord 申请 ID（如果 auto_id=True）
  ③ 向 RootCoord 申请时间戳（Timestamp）
  ④ 按 Shard 哈希分片: 决定数据发往哪个 Channel
  ⑤ 构造 InsertMsg 序列化为 Protobuf

源码位置: internal/proxy/insert_task.go
关键函数: InsertTask.Execute() → assignChannels() → produceMsg()
```

```
Step 2: 写入 Message Queue（同步，~1.0ms）
─────────────────────────────────────────
Proxy 将 InsertMsg 发送到对应的 Pulsar/Kafka Topic：
  ✓ 消息写入后立即返回 ACK
  ✓ 如果 MQ 写入失败 → 返回客户端错误
  ✓ 如果 MQ 写入成功 → 返回客户端成功（Step 3-6 异步）

关键点: 客户端收到"成功"只代表数据在 MQ 中安全了！
       数据还未进入 Milvus 存储层（Growing Segment）。
```

**小白**："所以如果 MQ 写入成功但 DataNode 消费之前 DataNode 崩溃了——数据会丢吗？"

**大师**："不会。因为数据在 MQ 中还没被消费（ACK），DataCoord 检测到 DataNode 宕机后，会把对应的 Channel 重新分配给其他 DataNode，新 DataNode 从断点继续消费。"

> **技术映射**：Proxy = 前台收银（验货+开小票）；MQ = 传送带（小票放上传送带就算'已受理'）；DataNode = 后厨（从传送带上取单做菜，这一步顾客看不到）。

---

**第二幕：DataNode 的内部世界——缓冲区、Growing Segment 与 Flush**

**小白**："那 Step 3-5 在 DataNode 内部是怎么执行的？为什么偶尔会有'黑洞时间'？"

**大师**："DataNode 的内部是一个典型的生产者-消费者模型——"

```
DataNode 内部架构:

  ┌─────────────────────────────────────────────────┐
  │                 DataNode                         │
  │                                                  │
  │  ┌──────────────┐    ┌───────────────────────┐  │
  │  │ MQ Consumer  │───→│  Write Buffer (内存)   │  │
  │  │ 从 Channel    │    │  - 按 Segment 分组     │  │
  │  │ 拉取消息      │    │  - 积累到阈值触发 Flush │  │
  │  └──────────────┘    └───────────┬───────────┘  │
  │                                  │               │
  │                   ┌──────────────▼───────────┐  │
  │                   │  Flush Manager            │  │
  │                   │  - 序列化为 Binlog        │  │
  │                   │  - 上传到对象存储          │  │
  │                   │  - 通知 DataCoord         │  │
  │                   └──────────────────────────┘  │
  └─────────────────────────────────────────────────┘
```

**大师**："Flush 触发的三个条件（任一满足即触发）——"

| 触发条件 | 默认值 | 调节参数 |
|---------|-------|---------|
| Segment 大小达到阈值 | 1 GB | `dataNode.segment.maxSize` |
| Segment 行数达到阈值 | 100 万行 | `dataNode.segment.maxRows` |
| 距离上次 Flush 超时 | 60 分钟 | `dataNode.flush.interval` |

**小胖**："那'黑洞时间'是什么原因？"

**大师**："最常见的三种——"

1. **对象存储慢 IO**：MinIO/S3 的写延迟偶尔飙到 10 秒，Binlog 上传卡住。排查：`milvus_datanode_flush_latency` 指标。
2. **Segment 太大**：如果 `segment.maxSize=4GB`，Flush 时需要序列化 4GB 数据，内存拷贝耗时数秒。解决：降低 maxSize 到 512MB。
3. **无新 Growing Segment 可用**：DataCoord 没有及时分配新 Segment，当前 Segment Flush 后写入通道"断档"。解决：增加 DataCoord 的 Segment 预分配数。

> **技术映射**：Write Buffer = 后厨备菜台（食材先放这儿积累）；Flush = 备菜台满了开始装盘上菜（序列化+写磁盘）；慢 IO = 传菜电梯卡住了（影响的是装盘速度而非备菜速度）。

**第三幕：写入乱序、重复、幂等——工程上的灵魂三问**

**小胖**："那如果我同时写两条数据，它们到达的顺序和写入的顺序不一样怎么办？如果网络断了一次，同一条数据被发了两次怎么办？"

**大师**："这是写入系统中经典的三问——"

**1. 写入乱序怎么办？**
Milvus 不保证写入顺序的严格一致。因为数据按 Shard 哈希分流到不同 Channel，不同 Channel 的消费速度可能不同。如果需要严格顺序，同一业务实体的数据应保证 hash 到同一 Shard（通过 Partition Key）。

**2. 重复写入怎么办？**
Insert 模式下主键重复会报错（`duplicate primary key`）。Upsert 模式下主键重复会覆盖——Upsert 天然幂等。推荐增量同步场景使用 Upsert。

**3. 如何做到 Exactly-Once（精确一次）？**
Milvus 目前不提供事务级别的 Exactly-Once 保证。推荐在业务层维护一个"已处理消息 ID"表，消费者消费 MQ 消息前先查重。

```python
# 业务层幂等写入设计
def idempotent_upsert(collection, products, processed_ids_store):
    """幂等 Upsert：基于业务去重表"""
    new_products = []
    for p in products:
        if p["id"] not in processed_ids_store:
            new_products.append(p)
            processed_ids_store.add(p["id"])
    
    if new_products:
        collection.upsert([...])  # Upsert 本身也幂等
```

#### 步骤 2：批量大小压测——找最佳吞吐

```python
# step2_batch_size_bench.py
"""对比不同批量大小下的写入吞吐"""
import time
import numpy as np
from pymilvus import connections, Collection

connections.connect(host="localhost", port="19530")
collection = Collection("insert_trace_test")

DIM = 128
TOTAL = 50000

# 预生成数据
all_ids = list(range(TOTAL))
all_vecs = np.random.rand(TOTAL, DIM).astype(np.float32).tolist()
all_texts = [f"text_{i}" for i in range(TOTAL)]

print("=" * 70)
print(f"批量大小对写入吞吐的影响 (总数据: {TOTAL} 条, dim={DIM})")
print("=" * 70)
print(f"{'批量大小':<10} {'总耗时(s)':<12} {'吞吐(条/s)':<12} {'Proxy压力'}")
print("-" * 60)

results = []
for bs in [10, 50, 100, 500, 1000, 5000]:
    # 清空旧数据
    try:
        collection.delete("id >= 0")
    except:
        pass
    
    t0 = time.time()
    for i in range(0, TOTAL, bs):
        end = min(i + bs, TOTAL)
        collection.insert([
            all_ids[i:end],
            all_vecs[i:end],
            all_texts[i:end],
        ])
    elapsed = time.time() - t0
    throughput = TOTAL / elapsed
    
    pressure = "低" if bs <= 100 else ("中" if bs <= 1000 else "高")
    results.append((bs, elapsed, throughput, pressure))
    print(f"{bs:<10} {elapsed:<12.2f} {throughput:<12.0f} {pressure}")

best_bs, _, best_tp, _ = max(results, key=lambda x: x[2])
print(f"\n> 最佳批量大小: {best_bs} 条/批, 吞吐: {best_tp:.0f} 条/s")
print(f"> 建议: 批量 100-500 是写入吞吐的'甜点区间'")
```

"""验证 MQ 的数据安全性和 Upsert 幂等性"""
from pymilvus import Collection, connections
import numpy as np

connections.connect(host="localhost", port="19530")
collection = Collection("insert_trace_test")

print("=" * 60)
print("幂等性验证实验")

test_data = [
    [1, 2, 3],

]
r1 = collection.upsert(test_data)
print(f"第1次 Upsert: insert={r1.insert_count}, upsert={r1.upsert_count}")

# 第二次 Upsert（完全相同的数据）
r2 = collection.upsert(test_data)
print(f"第2次 Upsert（相同数据）: insert={r2.insert_count}, upsert={r2.upsert_count}")
print(f"  → upsert_count={r2.upsert_count} 说明 3 条全部覆盖（幂等）")

# 第三次 Upsert（部分相同，部分更新）
test_data_changed = [
    [1, 4],
    np.random.rand(2, 128).astype(np.float32).tolist(),

]

print(f"第3次 Upsert（部分更新）: insert={r3.insert_count}, upsert={r3.upsert_count}")
print(f"  → 1 条覆盖(id=1), 1 条新增(id=4)")

print("\n> 结论: Upsert 天然幂等，适合做增量同步的写入方式")

## 4. 项目总结

### 4.1 写入链路关键指标

| 阶段 | 耗时（参考） | 是否阻塞客户端 | 监控指标 |
|------|-----------|-------------|---------|
| Proxy 校验+分片 | 0.1-0.5ms | 是 | `proxy_insert_latency` |
| MQ 写入 | 0.5-2ms | 是 | `mq_produce_latency` |
| DataNode 消费 | 10-100ms（异步） | 否 | `datanode_consume_lag` |
| Growing Segment 写入 | 0.1-1ms/批 | 否 | `datanode_buffer_size` |
| Flush（Binlog→S3） | 100ms-数秒 | 否 | `datanode_flush_latency` |

### 4.2 写入优化速查

| 优化方向 | 具体措施 | 效果 |
|---------|---------|------|
| 批量大小 | 100-500 条/批 | 吞吐提升 3-5 倍 vs 单条写入 |
| Shard 数 | 写入 QPS/5 万 | 并行写入通道增加 |
| Segment 大小 | 512MB-1GB（不要太大） | 避免 Flush 黑洞时间 |
| 消息队列 | 使用高性能 MQ（Pulsar 优于 Kafka） | 降低 MQ 写入延迟 |

### 4.3 注意事项

- **客户端 Insert 成功 ≠ 数据已 Flush**：客户端只需等 Proxy→MQ，数据真正落盘可能还要数秒。
- **Upsert 是增量同步的首选**：幂等、可重复执行、不丢数据。
- **不要每个 Insert 后都 Flush**：频繁 Flush 产生大量小 Segment 影响搜索性能。
