> **版本**：Milvus 2.5.x

## 1. 项目背景

某视频推荐平台有 10 亿条视频 Embedding（768 维），每天新增 500 万条。团队最初用 HNSW 索引 + 3 台 64GB QueryNode 扛了 1 亿条，P95 延迟 15ms。当数据量突破 3 亿后，QueryNode 内存告警频发，搜索延迟涨到 200ms。

架构师算了笔账：10 亿 × 768 × 4B（float32）= 约 3TB 原始数据，HNSW 索引内存约 5.4TB。如果全放内存，需要 16 台 512GB 的 QueryNode，年成本 200 万——远超预算。

CTO 要求找一条"成本减半、性能尚可"的路线。团队开始探索：DISKANN 磁盘索引、冷热分层、IVF_SQ8 量化压缩、多副本成本优化。最终发现，组合拳能让成本降到原来的 40%。

本章将探讨十亿级向量规模下的索引选择、冷热分层、弹性扩缩容和成本核算。

**第一幕：亿级向量的索引选择——HNSW 还是 DISKANN**

**小胖**（瞪大眼睛）："200 万一年？！就为了存向量？还不如用 ES——"

**大师**："别急。200 万是'全内存 HNSW'的方案。我们有更便宜的选择——DISKANN。"

**大师**（画出索引选择图）：

```
不同索引的"成本-性能"曲线:

  内存成本 ↑
    │
    │  HNSW (M=32)    速度最快, 成本最高
    │  HNSW (M=16)    速度很快, 成本高
    │  IVF_FLAT       速度中等, 成本中等
    │  IVF_SQ8        速度尚可, 成本低
    │  IVF_PQ         速度慢, 成本很低
    │  DISKANN        速度中等, 成本最低 (索引存磁盘)
    │
    └──────────────────────→ 查询延迟

十亿级向量推荐方案 (768维, 10亿条):

┌──────────────┬─────────┬──────────┬───────────┐
│ 索引类型       │ 内存占用  │ 磁盘占用   │ P95延迟   │
├──────────────┼─────────┼──────────┼───────────┤
│ HNSW (全内存)  │ 5.4 TB  │ 3 TB     │ 5ms       │
│ DISKANN       │ 0.3 TB  │ 6 TB     │ 15ms      │
│ IVF_SQ8       │ 0.9 TB  │ 3 TB     │ 20ms      │
│ 冷热分层       │ 0.5 TB  │ 6 TB     │ 10ms(P50) │
│ (热HNSW+冷DISK)│         │          │ 50ms(P99) │
└──────────────┴─────────┴──────────┴───────────┘
```

**大师**："DISKANN 是 Microsoft 开源的一种磁盘辅助索引——大部分图结构存磁盘，只在内存中缓存最热门的导航节点。内存只要 HNSW 的 5-10%。代价是搜索时需要读磁盘，延迟从 5ms 涨到 15ms。"

**大师**："DISKANN 的工作原理——"

```
DISKANN 内存/磁盘分工:

  内存中 (常驻):                      磁盘上 (按需加载):
  ┌────────────────┐               ┌─────────────────────┐
  │ 入口点 (Entry Points)│           │  完整图结构         │
  │ 缓存的热门节点      │    ←──→   │  (NN 亿条边)        │
  │ (~1% 的节点)       │  按需加载  │                     │
  └────────────────┘               └─────────────────────┘
  
  搜索时:
  ① 从内存的 Entry Point 开始导航
  ② 当需要访问不在内存中的节点时 → 从磁盘加载
  ③ 对热门节点做 LRU 缓存 → 下次访问命中内存
  ④ 最终 TopK 从内存 + 磁盘候选结果中选出
```

> **技术映射**：HNSW = 把整本电话簿背下来（快但费脑）；DISKANN = 只背常用号码，不常用的翻电话簿（省脑但慢点）；冷热分层 = 冰箱+冰柜组合（常用的冷藏、不常用的冷冻）。

```
冷热分层存储架构:

┌─────────────────────────────────────────────────────────┐
│  热数据 (近 7 天)                                        │
│  ├─ Collection: product_hot                             │
│  ├─ 索引: HNSW (全内存)                                  │
│  ├─ 数据量: ~5000 万条                                   │

├─────────────────────────────────────────────────────────┤
│  温数据 (7-90 天)                                        │
│  ├─ Collection: product_warm                            │
│  ├─ 索引: IVF_SQ8 (压缩内存)                             │

├─────────────────────────────────────────────────────────┤
│  冷数据 (90 天+)                                         │
│  ├─ Collection: product_cold                            │

│  └─ 内存: ~30GB (只缓存入口点)                            │
└─────────────────────────────────────────────────────────┘

搜索策略:
  ① 先搜热数据 (Priority=1)
  ② 如果结果不够 TopK → 搜温数据 (Priority=2)
  ③ 如果还不够 → 搜冷数据 (Priority=3)
  ④ 合并去重 → 返回 TopK

数据流转:
  热→温: 每晚定时 Job (近7天→7-90天)
  温→冷: 每周定时 Job (90天前的数据)
```

**大师**："批量导入策略——"

| 数据量 | 导入方式 | 吞吐 | 注意 |
|--------|---------|------|------|
| < 100 万 | `collection.insert()` 逐批 | ~1000 条/s | 简单 |
| 100 万 - 1000 万 | `collection.insert()` 大批 | ~5000 条/s | 控制批量大小 |
| 1000 万 - 1 亿 | Bulk Insert (`import()`) | ~5 万条/s | 跳过 MQ，直写对象存储 |
| > 1 亿 | 离线建库 + Load | ~10 万条/s | 生成 Binlog → 直接上传 S3 |

> **技术映射**：冷热分层 = 图书馆的新书区（热）+ 常规书架（温）+ 地下书库（冷）；Bulk Insert = 包机货运（跳过分拣中心直飞目的地）；离线建库 = 预制建筑模块（在工厂造好→现场组装）。

**第三幕：成本核算与弹性扩缩容**

**小胖**："200 万一年太贵了。有没有更详细的成本估算模型？"

**大师**："成本 = 计算 + 内存 + 存储 + 网络。给你一个成本计算器——"

```python
def cost_estimator(vectors: int, dim: int, index_type: str,
                   query_nodes: int, storage_gb: int) -> dict:
    """十亿级成本估算"""
    
    raw_gb = vectors * dim * 4 / (1024**3)
    
    memory_multiplier = {
        "HNSW": 1.8, "IVF_FLAT": 1.1,
        "IVF_SQ8": 0.3, "DISKANN": 0.1
    }
    
    mem_gb = raw_gb * memory_multiplier[index_type]
    
    # 云成本参考 ($/月)
    compute_cost = query_nodes * 200       # 每台 $200/月 (8C 64G)
    memory_cost = mem_gb * 3               # 每 GB $3/月
    storage_cost = storage_gb * 0.02       # S3 $0.02/GB/月
    
    total_monthly = compute_cost + memory_cost + storage_cost
    
    return {
        "raw_data_gb": raw_gb,
        "index_memory_gb": mem_gb,
        "compute_monthly": compute_cost,
        "memory_monthly": memory_cost,
        "storage_monthly": storage_cost,
        "total_monthly": total_monthly,
        "total_yearly": total_monthly * 12,
    }

# 示例: 10亿条 × 768维 × DISKANN
result = cost_estimator(
    vectors=1_000_000_000, dim=768,
    index_type="DISKANN", query_nodes=6,
    storage_gb=6000
)
print(f"年成本: ${result['total_yearly']:,.0f} (DISKANN方案)")
# 输出: 年成本: ~$45,000 (vs HNSW方案的 ~$200,000)
```

## 3. 项目实战

### 3.1 实战目标

为 10 亿 768 维向量设计容量方案，估算机器、内存、存储、QPS 和成本。

### 3.2 分步实现

#### 步骤 1：容量计算器

```python
# step1_capacity_planner.py
"""十亿级 Milvus 容量方案计算器"""

class CapacityPlanner:
    
    def __init__(self, total_vectors: int, dim: int, 
                 daily_growth: int, target_p95_ms: float):
        self.N = total_vectors
        self.D = dim
        self.growth = daily_growth
        self.target_p95 = target_p95_ms
    
    def plan(self) -> dict:
        raw_gb = self.N * self.D * 4 / (1024**3)
        
        plans = {}
        
        # 方案 A: 全 HNSW
        plans["HNSW全内存"] = {
            "内存(GB)": round(raw_gb * 1.8),
            "磁盘(GB)": round(raw_gb * 1.5),
            "QueryNode(台)": max(3, round(raw_gb * 1.8 / 30)),  # 每台 30GB 可用
            "P95延迟(ms)": "< 10",
            "年成本(万美元)": round(raw_gb * 1.8 * 3 * 12 / 10000 + 
                                 max(3, round(raw_gb*1.8/30)) * 200 * 12 / 10000, 1),
        }
        
        # 方案 B: DISKANN
        plans["DISKANN"] = {
            "内存(GB)": round(raw_gb * 0.1),
            "磁盘(GB)": round(raw_gb * 2.0),
            "QueryNode(台)": max(3, round(raw_gb / 200_000_000)),  # 每台 2 亿条
            "P95延迟(ms)": "< 20",
            "年成本(万美元)": round(raw_gb * 0.1 * 3 * 12 / 10000 + 
                                 max(3, round(raw_gb/200_000_000)) * 200 * 12 / 10000, 1),
        }
        
        # 方案 C: 冷热分层
        hot_ratio = 0.05  # 5% 热数据
        plans["冷热分层"] = {
            "内存(GB)": round(raw_gb * hot_ratio * 1.8 + raw_gb * 0.95 * 0.1),
            "磁盘(GB)": round(raw_gb * 2.0),
            "QueryNode(台)": max(3, round(raw_gb * hot_ratio * 1.8 / 30)),
            "P95延迟(ms)": "< 10 (热) / < 50 (冷)",
            "年成本(万美元)": round(raw_gb * 0.18 * 3 * 12 / 10000 + 
                                 max(3, round(raw_gb*hot_ratio*1.8/30)) * 200 * 12 / 10000, 1),
        }
        
        return {
            "数据规模": f"{self.N/1e8:.1f}亿 × {self.D}维",
            "原始数据量": f"{raw_gb:.0f} GB",
            "日均增长": f"{self.growth/1e4:.0f}万条",
            "方案对比": plans,
        }

# 使用
planner = CapacityPlanner(
    total_vectors=1_000_000_000,  # 10亿
    dim=768,
    daily_growth=5_000_000,       # 日增500万
    target_p95_ms=20,             # 目标P95<20ms
)

plan = planner.plan()
print(f"数据规模: {plan['数据规模']}")
print(f"原始数据: {plan['原始数据量']}")
print(f"\n方案对比:")
for name, detail in plan["方案对比"].items():
    print(f"\n  [{name}]")
    for k, v in detail.items():
        print(f"    {k}: {v}")
```

## 4. 项目总结

### 4.1 十亿级方案决策树

```
数据量 < 1 亿 → HNSW (全内存), 3-6 台 QN
数据量 1-5 亿 → IVF_SQ8 (压缩), 6-10 台 QN
数据量 5-10 亿 → DISKANN (磁盘索引), 4-8 台 QN
数据量 > 10 亿 → 冷热分层 + DISKANN, 分 Collection 管理
```

### 4.2 注意事项

- **DISKANN 需要高性能磁盘**：NVMe SSD 是标配，HDD 延迟不可接受。
- **冷热分层的搜索逻辑**：需要业务侧实现多 Collection 搜索+去重，比单 Collection 复杂。
- **成本估算要留 Buffer**：实际成本通常比公式估算高 20-30%（碎片化、元数据开销等）。

### 4.3 大规模数据迁移策略

从旧索引迁移到新索引的生产方案:

```
迁移步骤 (HNSW → DISKANN):
  ① 新建目标 Collection (使用 DISKANN)
  ② 从旧 Collection 流式导出数据
     → collection.query(offset=N, limit=10000)
  ③ 写入新 Collection (Bulk Insert 模式)
     → 跳过 MQ, 直写对象存储, 吞吐 10万+/s
  ④ 新 Collection 建索引 (DISKANN)
  ⑤ 灰度切换搜索流量 (10% → 100%)
  ⑥ 下线旧 Collection

  迁移窗口估算:
    10亿条 × 768维:
      导出: ~2 小时 (流式读取)
      导入: ~3 小时 (Bulk Insert)
      建索引: ~6 小时 (DISKANN)
      总计: ~11 小时 (可在夜间低峰期完成)
```

### 4.4 十亿级生产运维清单

```
□ QueryNode 内存分配: 每个 QN 不超过物理内存的 70%
□ 索引选择: 热数据 HNSW, 冷数据 DISKANN
□ Segment 治理: 定期 Compaction, Segment 数量控制在 5000 以内
□ 监控重点: QN 内存 + Segment 数量 + 对象存储 IO
□ 备份策略: 只备份热数据 (每日), 冷数据 (每周)
□ 成本审计: 每月核算各业务线成本
```
