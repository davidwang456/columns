## 1. 项目背景

算法团队对商品搜索的召回率不满意——用 HNSW 索引（默认参数 M=16, efConstruction=200, ef=64），Recall@10 只有 91%。业务方可以接受最大 5% 的漏召回，但 9% 已经影响用户体验（每页 10 个结果里就有一个不理想）。

算法工程师小周开始调参：把 ef 从 64 提高到 256，Recall@10 涨到 97%——但搜索延迟从 5ms 涨到 18ms。再把 M 从 16 提高到 32，Recall@10 涨到 99%——但内存占用从 40GB 涨到 65GB，QueryNode 内存告警。

小周发现自己陷入了"召回-延迟-内存"三体问题——提升任何一个都会影响另外两个。他需要系统性地理解每种索引类型的参数含义、相互制约关系，以及如何找到召回/延迟/内存的帕累托最优解。

更让他头疼的是：同一个数据集上，IVF_FLAT 的 Recall@10=95%、HNSW=99%、IVF_SQ8=91%——差距这么大，但网上资料都只说"推荐 HNSW"，没人解释为什么在某些场景下 IVF 反而更合适。

本章将对同一数据集构建三类索引，产出召回率、延迟、内存和构建耗时的四维对比报告。

## 2. 项目设计（剧本式交锋对话）

**第一幕：HNSW 参数全景——M、efConstruction、ef 三兄弟**

*（小周在 Excel 里列出了 20 组参数组合的实验结果，但没有清晰的规律）*

**小胖**（看着满屏数字头晕）："M、efConstruction、ef——这三个参数到底是干啥的？我随便调调行不行？"

**大师**："这三个参数控制着 HNSW 的三个阶段——"

**大师**（在白板上画）：

```
HNSW 三参数控制三阶段:

┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│   构建阶段    │      │   搜索阶段    │      │   存储阶段    │
│              │      │              │      │              │
│ efConstruction│     │     ef        │      │     M        │
│ (构建时搜索宽度)│      │ (搜索时搜索宽度)│      │ (图连接数)    │
│              │      │              │      │              │
│ 越大→索引越好  │      │ 越大→召回越高  │      │ 越大→图越密  │
│ 越大→构建越慢  │      │ 越大→搜索越慢  │      │ 越大→内存越大│
│              │      │              │      │              │
│ 建议: 200-500 │      │ 建议: 16-128  │      │ 建议: 8-32   │
└──────────────┘      └──────────────┘      └──────────────┘

                   ┌───────────────────────────┐
                   │    召回率 vs 延迟 vs 内存    │
                   │                              │
                   │  M ↑ → 内存 ↑, 召回 ↑        │
                   │  efConstruction ↑ → 构建慢   │
                   │  ef ↑ → 搜索慢, 召回 ↑        │
                   │                              │
                   │  M 影响最大（改了要重建索引）    │
                   │  ef 可在线调整（不影响内存）     │
                   └───────────────────────────┘
```

**大师**："详细解释每个参数——"

**1. M（图连接数）**
HNSW 是一个分层图结构，每个节点连接 M 个邻居。M 越大，图的"导航能力"越强，搜索时可以更快地跳转到目标区域。代价是内存——每个连接的邻居 ID 都要存，M=32 比 M=8 多 4 倍图结构内存。

**2. efConstruction（构建搜索宽度）**
构建索引时，每插入一个节点，系统先搜索 efConstruction 个候选邻居，再从中选 M 个最好的连接。efConstruction 越大，连接质量越高，但构建时间指数增长。efConstruction=500 的索引质量比 efConstruction=100 好约 5%，但构建时间多 4-6 倍。

**3. ef（搜索搜索宽度）**
搜索时，系统从入口点出发，维护一个大小为 ef 的候选队列。ef 越大，搜索越彻底，召回越高——但搜索时间线性增长。**ef 是唯一可以在搜索时动态调整的参数**（不改索引，不重建）。

**小胖**："所以调参的顺序是——先定 M（考虑内存预算），再定 efConstruction（考虑构建时间容忍度），最后在线调 ef（按需调节召回率）？"

**大师**："完全正确！"

> **技术映射**：M = 每个人认识几个朋友（朋友越多消息传得越快，但通讯录也越厚）；efConstruction = 交朋友时面试多少人（面试越多人越能找到好闺蜜，但要花时间）；ef = 找人问路时问几个人（多问几个更可能找对，但也要多跑腿）。

**第二幕：IVF 参数——nlist 与 nprobe 的黄金比例**

**小胖**："那 IVF 的参数 nlist 和 nprobe 呢？"

**大师**：

```
IVF 两参数:

nlist (聚类数)                           nprobe (搜索聚类数)
─────────────                           ────────────────
数据被聚成 nlist 个簇                    搜索时只扫描最近的 nprobe 个簇
nlist 越大 → 簇越细 → 搜索越准            nprobe 越大 → 搜索越彻底 → 召回越高
nlist 越大 → 构建越慢                     nprobe 越大 → 搜索越慢

黄金比例: nprobe ≈ nlist / 10 ~ nlist / 4

常见配置:
  10万条: nlist=128,  nprobe=16-32
  100万条: nlist=1024, nprobe=64-128
  1000万条: nlist=4096, nprobe=128-256
```

**大师**："nlist 和 HNSW 的 M 不一样的地方是——nlist 太小，聚类太粗，搜索精度下降严重；nlist 太大，每个簇里数据太少，聚类中心开销反而变大。有个经验公式：**nlist = 4 × sqrt(N)**。"

**小白**："10 万条数据 nlist = 4 × sqrt(100000) = 1265——为什么文档推荐 128？"

**大师**："因为 nlist 太大导致聚类中心太多，每个 DataNode 要维护的元数据量增加。而且 IVF 中搜索要扫描 nprobe 个簇，簇太多意味着每个簇里数据量很小，搜索覆盖的绝对数据量可能不够。128 或 256 适合大多数 1 万到 100 万的场景。"

> **技术映射**：nlist = 把全城分成几个大区（区划太细管理成本高）；nprobe = 搜房时去几个区看（多看几个区找到好房的概率高，但跑腿多）。

**第三幕：召回率评估——怎么用 Ground Truth 验证**

**小胖**："那怎么知道我的索引参数调好了没有？总不能每次调完都人工看吧——"

**大师**："这就是**召回率评估（Recall@K）**的价值。流程是——"

```python
# 1. 用 FLAT（暴力检索）跑一遍 → 得到 Ground Truth TopK
gt_results = collection_flat.search(query_vector, limit=100, ...)
gt_ids = [hit.id for hit in gt_results[0][:10]]  # 精确 Top10

# 2. 用目标索引（如 HNSW ef=64）跑一遍 → 得到近似 TopK
approx_results = collection_hnsw.search(query_vector, limit=10, ...)
approx_ids = [hit.id for hit in approx_results[0]]

# 3. 计算 Recall@10
hits = len(set(gt_ids) & set(approx_ids))
recall_at_10 = hits / 10
```

**大师**："完整的召回率评估需要——"

| 要素 | 说明 |
|------|------|
| **Ground Truth** | 用 FLAT 暴力检索跑出来的精确 TopK（数据集不能太大，否则 FLAT 跑不动） |
| **测试 Query 集** | 至少准备 100+ 条真实查询（不是随机向量） |
| **Recall@K** | 近似结果中正确命中 Ground Truth TopK 的比例 |
| **多次平均** | 单次 Recall 有偶然性，至少跑 50 次取平均 |

**大师**："生产环境 Recall@K 基线建议：**Recall@10 ≥ 99%**。如果你的场景是 RAG 知识库，Recall@10 = 98% 也能接受（剩下 2% 由 LLM 兜底）。但如果是电商推荐，Call@10 < 95% 就必须调参了。"

> **技术映射**：Ground Truth = 标准答案（用最慢但最准确的方式得到）；Recall@K = 近似答案和标准答案的重合度；参数调优 = 不断缩小近似答案和标准答案的差距。

#### 步骤 1：准备多版数据集

```python
# step1_prepare.py
"""为三类索引分别准备 Collection"""
import numpy as np
from pymilvus import connections, Collection, utility
from pymilvus import CollectionSchema, FieldSchema, DataType

connections.connect(host="localhost", port="19530")

INDEX_TYPES = ["FLAT", "IVF_FLAT", "IVF_SQ8", "HNSW"]
DIM = 384
N = 10000

def create_collection_for_index(name, dim):
    if utility.has_collection(name):
        utility.drop_collection(name)
    
    fields = [
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="vec", dtype=DataType.FLOAT_VECTOR, dim=dim),
        FieldSchema(name="label", dtype=DataType.VARCHAR, max_length=64),
    ]
    collection = Collection(name, CollectionSchema(fields))
    return collection

# 写入相同的数据到各 Collection
vecs = np.random.rand(N, DIM).astype(np.float32)
vecs = vecs / np.linalg.norm(vecs, axis=1, keepdims=True)

for idx_type in INDEX_TYPES:
    coll_name = f"idx_bench_{idx_type.lower()}"
    coll = create_collection_for_index(coll_name, DIM)
    coll.insert([
        list(range(N)),
        vecs.tolist(),
        [f"item_{i}" for i in range(N)],
    ])
    utility.flush([coll_name])
    print(f"  {idx_type}: 写入 {N} 条")

print(f"> 4 个 Collection 准备完成，每个 {N} 条 {DIM}维向量")
```

#### 步骤 2：建索引 + 四维对比

```python
# step2_index_benchmark.py
"""对四类索引做召回率、延迟、内存、构建耗时对比"""
import time
import numpy as np
from pymilvus import Collection, connections, utility
import json

connections.connect(host="localhost", port="19530")

DIM = 384
N = 10000
TOP_K = 10

# 测试查询向量 (10 条)
np.random.seed(42)
test_vecs = np.random.rand(10, DIM).astype(np.float32)
test_vecs = test_vecs / np.linalg.norm(test_vecs, axis=1, keepdims=True)

# ====== Step 1: 获取 Ground Truth（用 FLAT 暴力检索） ======
print("获取 Ground Truth (FLAT 暴力检索)...")
coll_flat = Collection("idx_bench_flat")
coll_flat.create_index("vec", {"index_type": "FLAT", "metric_type": "COSINE"})
coll_flat.load()

gt_results = coll_flat.search(
    data=test_vecs.tolist(), anns_field="vec",
    param={"metric_type": "COSINE"},
    limit=TOP_K, output_fields=["id"]
)
gt_ids = [[h.id for h in batch] for batch in gt_results]

# ====== Step 2: 对每种索引做建索引 + 搜索 + 评估 ======
index_configs = {
    "IVF_FLAT": {
        "index": {"index_type": "IVF_FLAT", "metric_type": "COSINE",
                   "params": {"nlist": 128}},
        "search": {"metric_type": "COSINE", "params": {"nprobe": 16}},
    },
    "IVF_SQ8": {
        "index": {"index_type": "IVF_SQ8", "metric_type": "COSINE",
                   "params": {"nlist": 128}},
        "search": {"metric_type": "COSINE", "params": {"nprobe": 16}},
    },
    "HNSW_default": {
        "index": {"index_type": "HNSW", "metric_type": "COSINE",
                   "params": {"M": 16, "efConstruction": 200}},
        "search": {"metric_type": "COSINE", "params": {"ef": 64}},
    },
    "HNSW_high": {
        "index": {"index_type": "HNSW", "metric_type": "COSINE",
                   "params": {"M": 32, "efConstruction": 400}},
        "search": {"metric_type": "COSINE", "params": {"ef": 256}},
    },
}

results_table = []

for name, config in index_configs.items():
    coll_name = f"idx_bench_{name.split('_')[0].lower()}"
    coll = Collection(coll_name)
    
    # 建索引
    t0 = time.time()
    coll.create_index("vec", config["index"])
    utility.wait_for_index_building_complete(coll_name, timeout=120)
    build_time = time.time() - t0
    
    coll.load()
    
    # 搜索 + 延迟
    latencies = []
    recalls = []
    for qv, gt in zip(test_vecs, gt_ids):
        t0 = time.time()
        r = coll.search(
            data=[qv.tolist()], anns_field="vec",
            param=config["search"], limit=TOP_K, output_fields=["id"]
        )
        latencies.append((time.time() - t0) * 1000)
        
        result_ids = [h.id for h in r[0]]
        recall = len(set(result_ids) & set(gt)) / TOP_K
        recalls.append(recall)
    
    avg_recall = np.mean(recalls)
    avg_latency = np.mean(latencies)
    
    results_table.append({
        "索引": name,
        "构建耗时(s)": round(build_time, 1),
        "平均延迟(ms)": round(avg_latency, 2),
        "Recall@10": f"{avg_recall:.1%}",
    })
    
    coll.release()
    print(f"  {name}: build={build_time:.1f}s, latency={avg_latency:.2f}ms, "
          f"recall={avg_recall:.1%}")

# ====== Step 3: 输出对比表 ======
print("\n" + "=" * 70)
print("索引四维对比报告")
print("=" * 70)
print(f"{'索引':<16} {'构建耗时':<10} {'延迟(ms)':<10} {'Recall@10':<10}")
print("-" * 50)
for r in results_table:
    print(f"{r['索引']:<16} {r['构建耗时(s)']:<10} {r['平均延迟(ms)']:<10} {r['Recall@10']:<10}")
```

"""分析 HNSW ef 参数对召回率和延迟的影响"""

print("-" * 35)

for ef in [4, 8, 16, 32, 64, 128, 256, 512]:
    times = []
    recalls = []
    for _ in range(10):
        t0 = time.time()
        r = coll_hnsw.search(
            data=[test_vec.tolist()], anns_field="vec",
            param={"metric_type": "COSINE", "params": {"ef": ef}},
            limit=10, output_fields=["id"]

        times.append((time.time() - t0) * 1000)
        result_set = set(h.id for h in r[0])
        recalls.append(len(result_set & gt_set) / 10)
    
    bar = "█" * int(np.mean(recalls) * 20)
    print(f"{ef:<8} {np.mean(times):<10.2f} {np.mean(recalls)*100:<10.1f}% {bar}")

print(f"\n> 结论: ef=64 是大多数场景下的'甜点'(99%+召回, <5ms)")

## 4. 项目总结

### 4.1 索引选择决策矩阵

| 条件 | 推荐索引 | 参数建议 |
|------|---------|---------|
| 数据量 < 1万, 极致召回 | FLAT | 无需调参 |
| 1万-500万, 内存充裕 | HNSW | M=16, ef=64 |
| 1万-500万, 内存紧张 | IVF_SQ8 | nlist=128, nprobe=16 |
| 500万-1亿, 召回率 < 95% 可接受 | IVF_FLAT + 大 nlist | nlist=4096, nprobe=128 |
| 数据频繁更新 | IVF_FLAT | HNSW 不支持增量 |
| > 1亿, 内存放不下 | DISKANN | （高级篇详述） |

### 4.2 调参优先级

1. **先定索引类型**（根据数据量和内存预算）
2. **再定 M/nlist**（影响内存和构建时间，改了要重建索引，先定）
3. **再定 efConstruction**（影响构建时间，非在线影响）
4. **最后在线调 ef/nprobe**（不影响索引，可动态调节）

### 4.3 注意事项

- **HNSW 不支持增量更新**：新增数据需要重建索引（或接受 Growing Segment 的暴力检索）。
- **ef 不要设太大**：ef=512 比 ef=64 慢 5-8 倍，但召回率只提升 0.3%。
- **Metric Type 必须与建索引时一致**：用 COSINE 建索引但用 L2 搜索，距离分值无意义。
- **nlist 推荐取 2 的次幂**：64/128/256/512/1024，方便数据对齐。
