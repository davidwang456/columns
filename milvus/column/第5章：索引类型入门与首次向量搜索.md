## 1. 项目背景

接第4章，李明已经成功将 10 万条商品数据的 Embedding 向量写入了 Milvus。他兴奋地打开 Python 终端准备做第一次搜索——

```python
results = collection.search(
    data=[query_vector],
    anns_field="title_embedding",
    param={"metric_type": "COSINE", "params": {"nprobe": 16}},
    limit=10
)
```

然后他等来的不是 Top10 结果，而是一个冷冰冰的错误：

```
MilvusException: <MilvusException: (code=1, message=collection product_search_v4 
was not loaded)> 
```

李明一拍脑门："忘了 Load！" 赶紧补上：

```python
collection.load()
```

这次终于有结果了，但是——等了足足 8.7 秒才返回。10 万条数据搜 Top10 要 8.7 秒？这个延迟足以让任何一个搜索产品被用户抛弃。

李明查阅文档才发现：他只插入了数据，没有构建索引！Milvus 在无索引的情况下会对每个 Segment 执行暴力检索（Flat Search），时间复杂度 O(N×D)——10 万 × 1024 维 = 1 亿次浮点乘法。而构建 ANN（近似最近邻）索引后，搜索复杂度可以降到 O(log N)，延迟从秒级降到毫秒级。

但当他开始选择索引类型时，又被一堆选项搞晕了：FLAT、IVF_FLAT、IVF_SQ8、IVF_PQ、HNSW、DISKANN、SCANN……每个都有一组参数（nlist、nprobe、M、efConstruction、efSearch）。该选哪个？参数怎么调？

更头疼的是，索引构建需要时间、占用 CPU 和内存，而参数一旦选错，要么召回率太低用户搜不到想找的商品，要么索引太大内存撑爆。李明花了一整周做对比实验，才有了清晰的选择逻辑。

本章的目标：让你理解 ANN 索引的核心理念，掌握 FLAT、IVF_FLAT、HNSW 三种入门索引的选择与参数调优，并完成一次从建索引到搜索、评估的完整闭环。

## 2. 项目设计（剧本式交锋对话）

**第一幕：为什么需要 ANN 索引——暴力检索与近似检索的取舍**

*（李明正在给团队做技术分享，演示了无索引搜索的 8.7 秒延迟）*

**小胖**（瞪大眼睛）："10 万条数据搜 Top10 要 8.7 秒？！如果我有 1000 万商品，岂不是要等 15 分钟？这不就跟在图书馆里从头到尾翻每一本书找人名吗？"

**大师**："你这个比喻非常精准——暴力和精确检索就是在图书馆里一本一本翻。但如果图书管理员聪明一点，他可以做索引：先按书名首字母建一张卡片目录，你查'三体'的时候直接定位到 S 区，然后再在那个小范围里翻。这就是 ANN 索引的核心思想——**用索引结构把搜索空间从 100% 缩小到 1% 甚至更小**。"

**大师**（在白板上画）：

```
暴力检索 (Flat / Brute Force):          近似检索 (ANN with Index):
                                         
Query                                  Query
  │                                      │
  ▼                                      ▼
┌─────────────────────────┐            ┌─────────────────────────┐
│ 对比每一条向量 (10万次)    │            │ 索引定位 → 候选子集 (~1000条) │
│ ┌───┬───┬───┬───┬───┐  │            │ ┌───┬───┐                │
│ │ ✓ │ ✓ │ ✓ │ ✓ │ ✓ │  │            │ │ ✓ │ ✓ │                │
│ └───┴───┴───┴───┴───┘  │            │ └───┴───┘                │
│ 精确 TopK                │            │ 近似 TopK (可能漏掉一些)    │
└─────────────────────────┘            └─────────────────────────┘
  100% 召回率                           99.x% 召回率
  O(N×D) 复杂度                          O(log N + small_const) 复杂度
  10万条 = 8.7秒                        10万条 = 2ms
```

**小白**："近似检索会漏结果——那为什么还要用？精确的结果不是更好吗？"

**大师**："因为'够用'比'完美'更实用。我给你算笔账："

| 数据量 | 暴力检索延迟 | ANN 检索延迟（HNSW） | 召回率 |
|--------|------------|-------------------|--------|
| 1 万 | 50ms | 1ms | 99.9% |
| 10 万 | 500ms | 2ms | 99.5% |
| 100 万 | 5s | 5ms | 99.2% |
| 1000 万 | 50s | 10ms | 99.0% |

**大师**："对于一个电商搜索系统来说，Top10 结果里有 9.9 条是正确的，已经足够好了——用户不会注意到第 10 条和第 11 条的细微差异。但 50 秒和 10 毫秒的体验差距，是用户绝对无法接受的。"

**小胖**："那 1% 的漏召回会丢到什么样的商品？"

**大师**："通常是那些处于'语义边界'的商品——比如搜'户外折叠椅'，第 11 名可能是一把'户外钓鱼凳'，虽然也相关但略逊一筹。丢失它并不影响用户体验。而真正的不相关商品（比如'办公转椅'），在暴力检索中本来就不会排进 Top10。"

> **技术映射**：暴力检索 = 翻遍所有书（100% 准确 + O(N) 慢）；ANN 索引 = 目录卡片定位（99% 准确 + O(log N) 快）；召回率 = 近似结果中含有多少真正的 TopK；工业界常用 99% 的召回率换取 1000 倍的性能提升。

**第二幕：FLAT、IVF_FLAT、IVF_SQ8、HNSW 的入门对比**

**小白**："那 FLAT、IVF_FLAT、HNSW 这三种索引到底是什么关系？我该选哪个？"

**大师**："这三种索引代表了三种不同的'空间划分'思路。我用一个城市找房子的比喻——"

**大师**（在白板上画）：

```
┌──────────────────────────────────────────────────────────────────┐
│ FLAT（暴力检索）                                                  │
│ "把全城每个小区都走一遍，挨个看房"                                 │
│ FLAT 不建任何索引，搜索时直接计算所有向量距离                       │
│ 优点: 100% 召回率、不占额外内存                                    │
│ 缺点: 慢！n 条向量就要算 n 次                                     │
│ 适用: n < 1 万 的极小数据集，或作为 Ground Truth 基准              │
└──────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│ IVF_FLAT（倒排索引）                                              │
│ "把全城分成 128 个区，搜房时只去最相关的 16 个区看"                │
│ 先将所有向量聚类为 nlist=128 个簇                                  │
│ 搜索时只扫描与 query 最近的 nprobe=16 个簇                         │
│ 扫描量 = (nprobe / nlist) × N ≈ 12.5% 的原始数据                  │
│ 优点: 内存占用小、构建快、适合百万级数据                            │
│ 缺点: 召回率受 nprobe 影响、聚类中心需要内存                       │
└──────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│ IVF_SQ8（量化压缩版 IVF）                                         │
│ "每个房子的资料从 10 页缩减到 1 页摘要"                            │
│ 向量从 float32（4字节/分量）压缩到 uint8（1字节/分量）              │
│ 内存降为 IVF_FLAT 的 1/4，搜索稍慢但幅度不大                       │
│ 适用: 内存紧张但向量维度高（768+）的场景                          │
└──────────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────────┐
│ HNSW（分层导航小世界图）                                           │
│ "每个房子记住附近最熟的几个邻居，通过熟人网络快速跳转"               │
│ 构建一个分层图结构：上层跳得远（长距离导航），下层找得准（局部搜索）  │
│ 参数 M: 每个节点连接多少邻居（越大召回越高，内存越大）               │
│ 参数 efConstruction: 构建时搜索宽度（越大索引越好，构建越慢）        │
│ 参数 efSearch: 搜索时搜索宽度（越大召回越高，搜索越慢）             │
│ 优点: 搜索极快、召回率高（通常 > 99%）                              │
│ 缺点: 内存占用大（约为 FLAT 的 1.5-2 倍）、构建慢                  │
└──────────────────────────────────────────────────────────────────┘
```

**小白**："所以我应该根据什么维度来选择？"

**大师**："给你一张决策表——"

| 你的情况 | 推荐索引 | 原因 |
|---------|---------|------|
| 数据量 < 1万，需要 100% 精确 | FLAT | 小数据集暴力也很快 |
| 数据量 1万-500万，内存充足 | HNSW | 搜索最快、召回最高 |
| 数据量 1万-500万，内存紧张 | IVF_SQ8 | 压缩存储，内存只有 FLAT 的 1/4 |
| 数据量 500万-1亿，内存足够 | HNSW 或 IVF_FLAT | HNSW 仍是首选，但需关注内存 |
| 数据量 > 1亿，内存放不下 | DISKANN | 磁盘辅助索引（高级篇再讲） |

**小胖**："我要找'适合露营的轻便折叠椅子'——10万条商品，内存也够，那我选 HNSW 就对了吧？"

**大师**："没错！HNSW 是当前向量检索领域的'标配'，绝大多数生产系统都首选 HNSW。不过你要注意它的内存开销——10 万条 1024 维向量，HNSW 索引大约占用 1.5GB 内存（向量数据 400MB + 图结构 1.1GB）。如果你的 QueryNode 只有 2GB 内存，可能就不够用了。"

> **技术映射**：FLAT = 精确但慢（基准线）；IVF = 聚类分桶（空间换时间，效果线性）；HNSW = 图导航（当前最优方案，内存换速度）；nlist/nprobe/M/efSearch = 控制精度与速度的旋钮。

**第三幕：Load、Release、TopK、Distance 的含义和正确理解**

**小胖**："大师，还有个问题——Load Collection 和 Release Collection 到底发生了什么？为什么每次搜之前都要 Load？"

**大师**："Load 就是把数据从对象存储（MinIO/S3）加载到 QueryNode 的内存中。你可以想象成——"

```
未 Load 状态:                        已 Load 状态:
┌─────────────────┐                 ┌─────────────────┐
│  Object Storage │                 │  QueryNode 内存  │
│  ┌───────────┐  │                 │  ┌───────────┐  │
│  │ Segment 1 │  │   Load  ──→     │  │ Segment 1 │  │  ← 可直接搜索
│  │ Segment 2 │  │                 │  │ Segment 2 │  │
│  │ Segment 3 │  │                 │  │ Segment 3 │  │
│  └───────────┘  │                 │  └───────────┘  │
│  数据在硬盘上     │                 │  数据在内存中     │
│  不可搜索 ✗      │                 │  可搜索 ✓        │
└─────────────────┘                 └─────────────────┘
```

**大师**："Release 则相反——把 QueryNode 中的数据卸载，释放内存。这在多个 Collection 共享同一组 QueryNode 时很有用：你可以 Release 不常用的冷 Collection，把内存让给热 Collection。"

**小白**："那 Load 一次要多久？如果数据量很大呢？"

**大师**："取决于数据量、维度和索引大小。10 万条 1024 维 HNSW 索引 Load 大约 10-30 秒；100 万条可能需要 2-5 分钟。所以生产环境一般是启动时 Load，用完之后不轻易 Release。"

**小胖**："那 TopK、Distance、Score 这三个又是什么？"

**大师**："搜索结果里最重要的三个概念——"

| 概念 | 含义 | 示例 | 使用建议 |
|------|------|------|---------|
| TopK | 返回的最相似的 K 条结果 | `limit=10` 返回 10 条 | K 太小漏语义相关结果；K 太大浪费带宽和计算 |
| Distance | 查询向量与结果之间的"距离" | 0.8923 | 用来判断结果质量、设置阈值过滤低质量结果 |
| Score | 距离的归一化分数（PyMilvus 特有） | 0.95 | 更方便跨查询比较（但不同 Metric Type 不可比） |

**大师**："特别注意——Distance 的解读取决于 Metric Type。COSINE 越大越好（越接近 1 越相似），L2 越小越好（越接近 0 越相似）。如果你混合使用了不同 Metric Type，Distance 的数值是不能直接比较的。"

**小白**："那如果我想过滤掉不相关的结果，比如只保留相似度 > 0.7 的，应该怎么做？"

**大师**："两种方式——"

```python
# 方式 1: 搜索后客户端过滤
results = collection.search(data=query, limit=100, ...)
filtered = [h for h in results[0] if h.distance > 0.7]

# 方式 2: 搜索时指定 radius（Milvus 2.5 新特性）
search_params = {
    "metric_type": "COSINE",
    "params": {"radius": 0.7, "range_filter": 1.0}  # 只返回 distance ∈ [0.7, 1.0]
}
results = collection.search(data=query, limit=100, param=search_params, ...)
```

"方式 2 更好——它是在 QueryNode 内部过滤，不会浪费网络传输带宽。但需要注意，当 radius 设置过严时，可能返回少于 K 条结果，甚至 0 条。"

> **技术映射**：Load = 把书从仓库搬到书架（磁盘 → 内存）；Release = 把书撤回仓库（释放内存）；TopK = 只翻最靠前的 K 本；Distance = 两本书内容的相似程度；Radius = 相似度的最低门槛（低于此门槛直接丢弃）。

```python
# step1_prepare.py
"""准备一个有 Ground Truth 的测试数据集"""
import json
import numpy as np
from sentence_transformers import SentenceTransformer
from pymilvus import connections, Collection, utility
from pymilvus import CollectionSchema, FieldSchema, DataType

connections.connect(host="localhost", port="19530")

# 加载 Embedding 模型
model = SentenceTransformer("all-MiniLM-L6-v2")

# 用第4章的数据（如果不存在则生成模拟数据）
try:
    with open("products_clean.json", "r", encoding="utf-8") as f:
        products = json.load(f)
except FileNotFoundError:
    # 生成简单测试数据
    products = [
        {"id": i, "title": f"测试商品_{i}", "price": i*10.0,
         "category": "测试", "in_stock": True}
        for i in range(1, 2001)
    ]

titles = [p["title"] for p in products]
embeddings = model.encode(titles)

# 创建 Collection（两个版本用于对比）
def create_collection(name):
    if utility.has_collection(name):
        utility.drop_collection(name)
    fields = [
        FieldSchema(name="id", dtype=DataType.INT64, is_primary=True),
        FieldSchema(name="title", dtype=DataType.VARCHAR, max_length=512),
        FieldSchema(name="title_vec", dtype=DataType.FLOAT_VECTOR, dim=384),
        FieldSchema(name="price", dtype=DataType.FLOAT),
        FieldSchema(name="category", dtype=DataType.VARCHAR, max_length=64),
        FieldSchema(name="in_stock", dtype=DataType.BOOL),
    ]
    collection = Collection(name, CollectionSchema(fields, description="索引对比测试"))
    return collection

# 写入数据
col_flat = create_collection("idx_flat_test")
col_hnsw = create_collection("idx_hnsw_test")

for col in [col_flat, col_hnsw]:
    for i in range(0, len(products), 500):
        batch = products[i:i+500]
        emb = embeddings[i:i+500].tolist()
        col.insert([
            [p["id"] for p in batch],
            [p["title"] for p in batch],
            emb,
            [p["price"] for p in batch],
            [p["category"] for p in batch],
            [p["in_stock"] for p in batch],
    ])
    utility.flush([col.name])

print(f"✓ FLAT 测试集: {col_flat.num_entities} 条")
print(f"✓ HNSW 测试集: {col_hnsw.num_entities} 条")
```

#### 步骤 2：构建索引并对比

```python
# step2_index_compare.py
"""对比 FLAT 与 HNSW 索引的构建、搜索性能"""
import time
import numpy as np
from pymilvus import Collection, connections
from sentence_transformers import SentenceTransformer
import json

connections.connect(host="localhost", port="19530")
model = SentenceTransformer("all-MiniLM-L6-v2")

# ---------- FLAT 索引 ----------
print("=" * 60)
print("FLAT 索引（无索引，暴力检索）")
print("=" * 60)

col_flat = Collection("idx_flat_test")
# FLAT 不需要真正构建索引，但如果调用 create_index 可作为一种占位
t0 = time.time()
col_flat.create_index(
    field_name="title_vec",
    index_params={
        "index_type": "FLAT",
        "metric_type": "COSINE",
    }
)
print(f"  FLAT '建索引'耗时: {time.time() - t0:.4f}s（几乎为0，FLAT 不建索引）")

col_flat.load()
print(f"  FLAT Load 完成")

# ---------- HNSW 索引 ----------
print("\n" + "=" * 60)
print("HNSW 索引")
print("=" * 60)

col_hnsw = Collection("idx_hnsw_test")
t0 = time.time()
col_hnsw.create_index(
    field_name="title_vec",
    index_params={
        "index_type": "HNSW",
        "metric_type": "COSINE",
        "params": {
            "M": 16,                      # 每个节点的边数
            "efConstruction": 200,        # 构建时搜索宽度
        }
    }
)
print(f"  HNSW 建索引耗时: {time.time() - t0:.2f}s")

# 等待索引构建完成
print("  等待索引构建完成...")
utility.wait_for_index_building_complete("idx_hnsw_test", timeout=300)
print(f"  HNSW 索引构建完成")

col_hnsw.load()
print(f"  HNSW Load 完成")

# ---------- 搜索性能对比 ----------
print("\n" + "=" * 60)
print("搜索性能对比 (10 轮取平均)")
print("=" * 60)

# 准备查询向量
query_texts = ["轻便折叠户外露营椅", "便携式无线蓝牙耳机", "舒适记忆棉枕头"]
query_vectors = model.encode(query_texts).tolist()

# FLAT 搜索
print(f"\n{'指标':<20} {'FLAT':<15} {'HNSW':<15}")
print("-" * 50)

flat_times = []
hnsw_times = []

for qv in query_vectors:
    # FLAT
    t0 = time.time()
    result_flat = col_flat.search(
        data=[qv], anns_field="title_vec",
        param={"metric_type": "COSINE", "params": {}},
        limit=10, output_fields=["title"]
    )
    flat_times.append(time.time() - t0)

    # HNSW
    t0 = time.time()
    result_hnsw = col_hnsw.search(
        data=[qv], anns_field="title_vec",
        param={"metric_type": "COSINE", "params": {"ef": 64}},
        limit=10, output_fields=["title"]
    )
    hnsw_times.append(time.time() - t0)

avg_flat = np.mean(flat_times) * 1000
avg_hnsw = np.mean(hnsw_times) * 1000

print(f"{'平均搜索延迟':<20} {avg_flat:<14.2f}ms {avg_hnsw:<14.2f}ms")
print(f"{'加速比':<20} {'1x (基准)':<15} {avg_flat/avg_hnsw:<13.1f}x")
print(f"\n{'结果数':<20} {len(flat_times):<15} {len(hnsw_times):<15}")

# ---------- 内存占用对比 ----------
print("\n" + "=" * 60)
print("内存占用对比")
print("=" * 60)

flat_index_size = col_flat.index().params.get("index_size", "N/A")
hnsw_index_size = col_hnsw.index().params.get("index_size", "N/A")
```

## 4. 项目总结

### 4.1 优缺点对比

| 维度 | FLAT | IVF_FLAT | HNSW |
|------|------|----------|------|
| 搜索速度 | 最慢（O(N×D)） | 快（遍历 ~12.5% 数据） | 最快（图导航 O(log N)） |
| 召回率 | 100% | 95-99%（取决于 nprobe） | 98-99.9%（取决于 ef） |
| 内存占用 | 低（只存原始向量） | 中（向量 + 聚类中心） | 高（向量 + 图结构，约 1.5-2x） |
| 构建速度 | 无额外构建 | 快（只需聚类） | 慢（需构建多层图） |
| 增量更新 | 天然支持 | 需重建或增量聚类 | 需重建（HNSW 不支持增量） |
| 最佳数据量 | < 1万 | 1万-100万 | 1万-1000万 |

### 4.2 适用场景

- **HNSW**：在线搜索服务（电商、RAG、推荐），需要极低延迟和高召回率。
- **IVF_FLAT**：数据频繁更新的场景（HNSW 不支持增量更新），或内存紧张的部署环境。
- **IVF_SQ8**：向量维度很高（768+），内存预算有限，且对召回率要求不是极致（95% 可接受）。
- **FLAT**：极小数据集（< 1万），或需要 100% 精确结果的基准测试。
- **DISKANN**：十亿级数据，内存放不下索引，需要磁盘辅助（高级篇第 20 章详述）。

**不适用场景**：需要频繁实时更新的场景（HNSW 每次更新需要重建索引，不适合）。

### 4.3 注意事项

- **efConstruction 影响构建速度**：efConstruction 越大，HNSW 索引质量越好，但构建时间显著增长。200-500 是常见的生产值。
- **efSearch 不影响内存**：efSearch 只在搜索时生效，增加 ef 只增加搜索延迟，不会增加内存。
- **nlist 的经验公式**：nlist = 4 × sqrt(N)，10 万条数据 nlist 约 1265。常见值 128/256/512/1024。
- **建索引后要重新 Load**：部分索引类型（HNSW）在 `create_index` 之后需要重新 `load()` 才能生效。

### 4.4 常见踩坑经验

1. **"建了索引但搜索还是很慢"** —— 忘记 Load Collection。`create_index` 只是定义了索引结构并触发构建，但不会自动加载。需要在 `create_index` + `wait_for_index_building_complete` 之后再 `load()`。
2. **"HNSW 内存占用远超预算"** —— 忘记了图结构的内存开销。HNSW 的内存 ≈ 向量数据 × 1.5-2 倍。计算资源前务必留出余量。
3. **"索引构建一直在 running"** —— IndexNode 资源不足或数据量太大。检查 IndexNode 的 CPU/内存使用，必要时增加 IndexNode 实例。

### 4.5 思考题

1. 如果需要在一个已经上线服务的 Collection 上重建索引（比如从 IVF 切换到 HNSW），应该如何设计灰度切换方案以保证搜索不停服？
2. HNSW 的 M 参数设置为 8 vs 32，对搜索延迟和内存占用有什么定量影响？请设计一个实验来量化这个关系。
