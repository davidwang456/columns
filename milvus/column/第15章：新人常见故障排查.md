# 第15章：新人常见故障排查

> **定位**：从"能跑"到"知道哪里坏了"。
> **版本**：Milvus 2.5.x
> **源码关联**：internal/proxy/task.go、internal/rootcoord/service.go、internal/datacoord/service.go

## 1. 项目背景

新人小吴入职第一周就被分配维护公司的 Milvus 服务。周二凌晨 2 点，告警电话把他吵醒——商品搜索接口返回 500 错误。他迷迷糊糊打开笔记本，看到日志里一堆 `connection refused` 和 `collection not loaded`。

他对着 6 个终端窗口（Milvus 日志、etcd 日志、MinIO 日志、SDK 日志、Grafana 大盘、业务服务日志），像个无头苍蝇一样逐个翻，2 个小时过去了还是没定位到根因。最后是王工远程帮忙——3 分钟就发现是 QueryNode 内存 OOM 导致 Load 状态异常。

小吴事后统计了一下，Milvus 常见故障其实就 6 大类，每类都有固定的排查路径。但他没有建立排查 SOP（Standard Operating Procedure），每次出问题都从零开始摸索。

更让团队头疼的是，不同角色的同事排查问题的方式不同——运维用 `docker logs` 和 Prometheus，开发用 PyMilvus 跑测试脚本，算法用 Attu 可视化检查。三个角色各看各的，信息不对齐，排故效率极低。

本章将汇总新人最常见的 6 类故障，提供每类的"三分法排查路径"（运维命令 + SDK 验证 + Attu 可视化），形成一份可打印、可张贴的排故 SOP。

故障 1: 连接失败 / 端口不通

┌──────────────────────────────────────────────────────────┐
│ 排查三步走                                                │
├──────────────────────────────────────────────────────────┤
│                                                          │
│ Step 1: 端口检测（运维）                                   │
│   curl http://localhost:19530/healthz                    │
│   # 如果返回 OK，服务正常                                 │
│   # 如果 connection refused，检查:                        │

│ Step 2: SDK 验证（开发）                                   │
│   from pymilvus import connections, utility              │
│   connections.connect(host="localhost", port="19530")    │

│ ✗ 容器未启动: docker compose ps 显示 Exited             │
│ ✗ 防火墙: 安全组/iptables 阻断了端口                      │
│ ✗ Docker 网络: 容器网络与宿主机网络未打通                  │

**小白**："端口冲突怎么排查？"

**大师**："`netstat -ano | findstr 19530`（Windows）或 `lsof -i :19530`（Linux/macOS），看看是谁占的。如果是其他程序，要么停掉它，要么改 Milvus 的端口映射。"

> **技术映射**：端口不通 = 店门关了或地址写错了；healthz = 敲门看有没有人应；Attu = 推门进去看店里什么情况。

**第二幕：Collection 未 Load / 索引未构建 / 维度不匹配**

**小胖**："我上周还碰见一个怪事——数据明明写进去了，但搜不到，啥报错也没有！"

**大师**："这是'Collection 没 Load'的经典症状。而且经常伴随'建了索引但忘记重新 Load'——"

```
故障 2: 搜索无结果 / collection not loaded

┌──────────────────────────────────────────────────────────┐
│ 检查清单（按顺序执行）                                     │
├──────────────────────────────────────────────────────────┤
│                                                          │
│ □ 1. Collection 是否已 Load?                              │
│      collection.load()                                   │
│      utility.load_state(collection_name)                 │
│      # 返回 LoadState.Loaded 才是真的可搜索               │
│                                                          │
│ □ 2. 索引是否已构建?                                      │
│      collection.has_index()                              │
│      # 如果 False:                                       │
│      collection.create_index(...)                        │
│      utility.wait_for_index_building_complete(...)       │
│      collection.load()  # 建完索引必须重新 Load!          │
│                                                          │
│ □ 3. 向量维度是否一致?                                    │
│      schema = collection.schema                          │
│      for f in schema.fields:                             │
│          if f.dtype == DataType.FLOAT_VECTOR:            │
│              print(f"Schema dim: {f.params['dim']}")      │
│      # 对比 Embedding 模型的输出维度                       │
│      # 不一致 = 写入会报错或搜索无效                       │
│                                                          │
│ □ 4. Metric Type 是否一致?                                │
│      # 建索引用 COSINE，搜索用 L2 → 结果无意义             │
│      index = collection.index()                          │
│      print(index.params)  # 查看 metric_type             │
└──────────────────────────────────────────────────────────┘
```

**大师**："90% 的'搜不到'问题都在上面这四项里。按顺序排查，5 分钟搞定。"

> **技术映射**：Load = 把书放到书架上（不放到架子上没法翻阅）；索引 = 目录页（没有目录只能逐页翻）；维度匹配 = 书的厚度要和书架格子大小匹配。

---

**第三幕：Insert 成功但数据怎么不见了**

**小胖**："还有更诡异的——我 Insert 返回 success，`num_entities` 也涨了，但 Query 查不到刚插入的数据——"

**大师**："这是'可见性'问题。新人经常忘记两点——"

```
故障 3: Insert 成功但 Search 找不到新数据

原因 1: 数据还在 Growing Segment 中
  解决: collection.flush() 然后 collection.load()

原因 2: 一致性级别为 Bounded/Eventually
  解决: 搜索时设置 consistency_level="Strong"
        collection.search(..., consistency_level="Strong")

原因 3: 搜的是其他 Collection
  解决: print(collection.name) 确认没错

原因 4: 插入了但主键与已有数据冲突（Insert 不是 Upsert）
  解决: 检查 insert_result.insert_count vs upsert_count
```

**大师**："最典型的是这个场景——"

```python
collection.insert([...])   # 返回 success
utility.flush(["coll"])     # 持久化
# 但忘记 collection.load()  # ← 新数据不会自动可见
results = collection.search(...)  # 搜不到刚插入的数据!
```

> **技术映射**：Insert 成功 ≠ 搜索可见；Growing Segment = 待上架的新书（还没放到书架上）；Flush = 把新书入库登记；Load = 把新书放上书架（这步才能被搜到）。

#### 步骤 1：故障模拟与排查脚本

```python
# milvus_troubleshooter.py
"""Milvus 6 类常见故障的自动诊断工具"""
from pymilvus import connections, utility, Collection
import sys

class MilvusTroubleshooter:
    """Milvus 故障自动排查器"""
    
    def __init__(self, host="localhost", port=19530):
        self.host = host
        self.port = port
    
    def check_all(self):
        """执行全部 6 项检查"""
        checks = [
            self.check_connection,
            self.check_etcd_minio,
            self.check_collection_load,
            self.check_index,
            self.check_dimension_match,
            self.check_visibility,
        ]
        
        results = []
        for check in checks:
            try:
                name, status, detail = check()
                results.append((name, status, detail))
            except Exception as e:
                results.append((check.__name__, "ERROR", str(e)))
        
        return results
    
    # ---- 检查 1: 连接 ----
    def check_connection(self):
        """检查 Milvus 连接是否正常"""
        try:
            connections.connect(
                alias="troubleshoot",
                host=self.host,
                port=self.port,
                timeout=5,
            )
            version = utility.get_server_version(using="troubleshoot")
            return ("连接检查", "PASS", f"Milvus v{version} @ {self.host}:{self.port}")
        except Exception as e:
            return ("连接检查", "FAIL", f"无法连接: {e}\n"
                    f"  排查: ① docker compose ps | grep milvus\n"
                    f"       ② curl http://{self.host}:{self.port}/healthz\n"
                    f"       ③ netstat -ano | findstr {self.port}")
    
    # ---- 检查 2: etcd/MinIO ----
    def check_etcd_minio(self):
        """检查外部依赖是否正常"""
        import requests
        issues = []
        
        try:
            r = requests.get(f"http://{self.host}:2379/health", timeout=3)
            if r.json().get("health") != "true":
                issues.append("etcd 健康检查失败")
        except Exception:
            issues.append("etcd 不可达 (2379)")
        
        try:
            r = requests.get(f"http://{self.host}:9001/minio/health/live", timeout=3)
            if r.status_code != 200:
                issues.append("MinIO 健康检查失败")
        except Exception:
            issues.append("MinIO 不可达 (9001)")
        
        if issues:
            return ("外部依赖", "FAIL", "\n  ".join(issues))
        return ("外部依赖", "PASS", "etcd & MinIO 健康")
    
    # ---- 检查 3: Collection Load ----
    def check_collection_load(self):
        """检查 Collection 是否已 Load"""
        collections = utility.list_collections(using="troubleshoot")
        if not collections:
            return ("Collection Load", "SKIP", "无 Collection")
        
        results = []
        for name in collections:
            state = utility.load_state(name, using="troubleshoot")
            results.append(f"  {name}: {state.name}")
        
        all_loaded = all("Loaded" in r for r in results)
        status = "PASS" if all_loaded else "WARN"
        detail = "\n".join(results)
        if not all_loaded:
            detail += "\n  解决: collection.load()"
        
        return ("Collection Load", status, detail)
    
    # ---- 检查 4: 索引 ----
    def check_index(self):
        """检查索引是否已构建"""
        collections = utility.list_collections(using="troubleshoot")
        if not collections:
            return ("索引检查", "SKIP", "无 Collection")
        
        from pymilvus import Collection
        results = []
        for name in collections:
            col = Collection(name, using="troubleshoot")
            has_idx = col.has_index()
            status_icon = "✓" if has_idx else "✗"
            results.append(f"  {status_icon} {name}: {'已构建' if has_idx else '未构建'}")
        
        all_indexed = all("✓" in r for r in results)
        status = "PASS" if all_indexed else "WARN"
        detail = "\n".join(results)
        if not all_indexed:
            detail += "\n  解决: ① collection.create_index(...) ② collection.load()"
        
        return ("索引检查", status, detail)
    
    # ---- 检查 5: 维度匹配 ----
    def check_dimension_match(self):
        """检查 Collection 中向量字段的维度"""
        from pymilvus import Collection, DataType
        collections = utility.list_collections(using="troubleshoot")
        if not collections:
            return ("维度检查", "SKIP", "无 Collection")
        
        results = []
        for name in collections:
            col = Collection(name, using="troubleshoot")
            for f in col.schema.fields:
                if f.dtype == DataType.FLOAT_VECTOR:
                    results.append(f"  {name}.{f.name}: dim={f.params['dim']}")
        
        return ("维度检查", "INFO", "\n".join(results) + 
                "\n  注意: Embedding 模型输出维度需与 Schema 一致")
    
    # ---- 检查 6: 可见性 ----
    def check_visibility(self):
        """检查数据可见性"""
        collections = utility.list_collections(using="troubleshoot")
        if not collections:
            return ("可见性检查", "SKIP", "无 Collection")
        
        from pymilvus import Collection
        results = []
        for name in collections:
            col = Collection(name, using="troubleshoot")
            count = col.num_entities
            results.append(f"  {name}: {count:,} 条")
        
        return ("可见性检查", "INFO", "\n".join(results) + 
                "\n  注意: 新写入数据需 Flush + Load 后才可搜索")


# 运行排障
troubleshooter = MilvusTroubleshooter()
print("=" * 60)
print("Milvus 故障排查报告")
print("=" * 60)

for name, status, detail in troubleshooter.check_all():
    icon = {"PASS": "✅", "FAIL": "❌", "WARN": "⚠️", "SKIP": "⏭️", "INFO": "ℹ️", "ERROR": "💥"}[status]
    print(f"\n{icon} {name}: [{status}]")
    print(f"   {detail}")

connections.disconnect("troubleshoot")
```

## 4. 项目总结

### 4.1 优缺点对比

| 维度 | 手动排查 | 排查 SOP | 自动化诊断工具 |
|------|---------|---------|--------------|
| 排查速度 | 慢（30min+） | 中（10min） | 快（1min） |
| 覆盖全面性 | 取决于个人经验 | 覆盖常见 6 类 | 覆盖代码能检查的范围 |
| 维护成本 | 低 | 中 | 高（需随版本更新） |
| 新人友好 | 差 | 好 | 最好 |

### 4.2 排故黄金法则

1. **先外部后内部**：先查 etcd → MinIO → 然后才是 Milvus 自身。
2. **先状态后日志**：先看 `ps`、`healthz`、`load_state`，定位问题类别后再看详细日志。
3. **先缩小范围**：确认是整个集群挂了还是单个 Collection 的问题。
4. **保留现场**：不要急着重启容器，先 `docker compose logs` 导出日志留作复盘。
5. **一次只改一个变量**：排查时不要同时改多个配置，否则不知道是哪步修复的。

### 4.3 思考题

1. 如果 Milvus 搜索延迟突然从 5ms 涨到 500ms 但没有任何报错，你应该按什么顺序排查哪些组件？
2. 设计一个自动健康检查脚本，每 5 分钟运行一次，覆盖连接、Load、索引三大类检查，异常时发送告警。

### 4.4 推广计划提示

- 将故障速查表打印贴在团队工位上（物理版）。
- 将 `MilvusTroubleshooter` 脚本集成到 CI 健康检查中。
- 每出现一种新故障，更新一次 SOP（保持文档活力）。
