## 1. 项目背景

运维工程师老周负责的公司 Milvus 集群已经稳定运行了 3 个月，存了 500 万条商品向量数据。一个周五下午，实习生误操作在 Attu 上执行了 `drop_collection("product_search")`，整个商品搜索库瞬间蒸发。更让人崩溃的是——没有任何备份。

老周花了整整一个周末重跑数据流水线：从 MySQL 导出商品数据 → 重新生成 Embedding → 重新写入 Milvus → 重建索引。整个过程耗时 14 小时，期间商品搜索功能完全不可用，业务部门投诉电话打爆。

事故复盘会上，CTO 拍桌子："为什么没有备份？" 老周委屈地说："我上个月就提了备份方案，但当时大家觉得 Milvus 数据都是从 MySQL 同步来的，'丢了重新导就行了'——没想到重新导需要 14 个小时。"

这个血的教训让团队开始认真对待备份，但在实操中又发现：Milvus 的备份不是简单的"cp 一个文件"：

1. **对象存储**：数据不是存为可直接读取的文件格式，而是 Binlog + Delta Log + Index 的组合体。
2. **元数据**：etcd 存储了 Collection Schema、Partition 定义、Segment 分布等关键元数据，只备份对象存储不够。
3. **索引文件**：是否备份索引是一个权衡——备份可以免去重建索引的时间，但大大增加备份体积。
4. **一致性**：边写入边备份时，如何保证备份的快照一致性。
5. **跨环境迁移**：从开发环境备份的数据，在测试环境 restore 时可能因为 Milvus 版本不同而失败。

本章将学会使用 Milvus Backup 工具，并手动理解备份背后的文件关系。

## 2. 项目设计（剧本式交锋对话）

**第一幕：为什么不能直接 cp 文件——理解 Milvus 的存储结构**

*（老周正在给团队介绍"14小时恢复事故"的复盘）*

**小胖**（不解地）："不就几个文件吗？Milvus 的数据不是都存 MinIO 里了？直接把 MinIO bucket cp 一份不就行了？"

**大师**："我也想这么简单。但 Milvus 的数据不是普通文件，它分成三块——"

**大师**（在白板上画）：

```
┌──────────────────────────────────────────────────────────────┐
│                    Milvus 数据存储全景                         │
│                                                              │
│  ┌─────────────────┐  ┌──────────────────┐  ┌──────────────┐ │
│  │    etcd         │  │   Object Storage  │  │  Message Queue│ │
│  │  (元数据存储)     │  │   (MinIO / S3)    │  │  (WAL)       │ │
│  └────────┬────────┘  └────────┬─────────┘  └──────────────┘ │
│           │                    │                              │
│  ┌────────▼────────┐  ┌────────▼─────────────────────────┐   │
│  │ Collection 定义  │  │  Binlog (数据日志)                │   │
│  │ Schema 结构     │  │  Delta Log (删除标记日志)          │   │
│  │ Partition 定义  │  │  Index Files (索引文件)            │   │
│  │ Segment 映射    │  │  DDL Log (DDL 操作日志)            │   │
│  └─────────────────┘  └──────────────────────────────────┘   │
│                                                              │
│  ⚠ 备份必须同时保存 etcd 元数据 + Object Storage 文件！         │
│  只备份 Object Storage = 有数据但不知道是哪个 Collection 的      │
│  只备份 etcd = 有结构但没有数据                                 │
└──────────────────────────────────────────────────────────────┘
```

| 存储位置 | 存了什么 | 丢了会怎样 | 备份方式 |
|---------|---------|-----------|---------|
| **etcd** | Schema、Partition、Segment 映射、Collection 别名 | 整个 Milvus 集群失忆——不知道有哪些 Collection | etcd snapshot |
| **Object Storage** | Binlog（实际数据）、Delta Log（删除记录）、Index（索引文件） | 数据丢失，恢复后 Collection 是空的 | 对象存储同步 |
| **Message Queue** | 写入中的飞行数据（未 Flush 的） | 最近几秒的数据丢失 | 通常不备份（恢复性价比低） |

**小白**："那 Milvus Backup 工具做了什么？"

**大师**："它帮你同时做了 etcd 元数据快照 + 对象存储文件同步，保证两者的一致性。并提供了 restore 命令一键恢复。"

> **技术映射**：etcd 备份 = 图书馆的索引目录备份；Object Storage 备份 = 书的内容备份；MQ 备份 = 传送带上还没上架的书（一般不备份，丢了重新投递）。

**第二幕：备份策略——全量 vs 增量 vs 索引要不要备份**

**小胖**："那备份应该多久做一次？全量还是增量？索引要备份吗？"

**大师**："三个问题，一份决策表——"

| 备份类型 | 优点 | 缺点 | 频率建议 |
|---------|------|------|---------|
| **全量备份**（Collection 级） | 恢复最简单、最快 | 数据量大时耗时长、占空间 | 每天 1 次（凌晨低峰） |
| **增量备份** | 备份快、占空间少 | 恢复复杂（需先恢复全量 + 所有增量） | 每小时 1 次 |
| **混合策略** | 平衡 | 管理复杂 | 每周全量 + 每天增量 |

| 索引备份 | 优点 | 缺点 | 建议 |
|---------|------|------|------|
| **备份索引** | 恢复后立即可搜索 | 备份体积大（HNSW 索引可达原始数据 1.5-2 倍） | 数据量 < 100 万条，建议备份 |
| **不备份索引** | 备份体积小 | 恢复后需重建索引（耗时长） | 数据量 > 1000 万条，不建议备份 |

**大师**："核心原则：**评估你的 RTO（Recovery Time Objective，恢复时间目标）和 RPO（Recovery Point Objective，恢复点目标）**——"

- RTO = 能容忍的最长恢复时间。如果 RTO = 1 小时，那就必须备份索引（否则重建索引可能超过 1 小时）。
- RPO = 能容忍的最大数据丢失量。如果 RPO = 5 分钟，那就需要高频增量备份。

**小白**："那备份验证呢？怎么知道备份是真的可用的？"

**大师**："备份不验证等于没备份。建议每个月做一次恢复演练——把最近一次备份 restore 到一个隔离环境中，随机抽样搜索 100 条 query 对比结果。这个应该编成自动化脚本，而不是等人手工做。"

> **技术映射**：全量备份 = 给整个房子拍照存档；增量备份 = 只拍今天变化了的东西；RTO = 房子着火了多久能住回去；RPO = 能接受丢失多少最近买的东西。

**第三幕：跨环境迁移——版本兼容与配置适配**

**小胖**："那我从开发环境备份的，能直接恢复到测试环境吗？"

**大师**："不一定。跨环境恢复有三个卡点——"

| 卡点 | 问题 | 解决方案 |
|------|------|---------|
| **Milvus 版本** | Milvus 2.4 备份无法恢复到 2.5 | 备份和恢复两端版本必须一致（或官方文档支持的升级路径） |
| **对象存储差异** | 开发用 MinIO，生产用 S3 | 备份文件格式兼容（都是 S3 协议），但 restore 时要指定新的存储配置 |
| **etcd 集群差异** | 开发 1 节点 etcd，生产 3 节点 | etcd snapshot 格式兼容，但 restore 时需适配 etcd endpoint |

**大师**："标准跨环境迁移流程——"

```bash
# 1. 在源环境备份
milvus-backup create -n product_backup_20240501 -c product_search

# 2. 拷贝备份文件到目标环境
# 备份文件存在 backup 工具配置的 backup_storage 里（MinIO/S3）
aws s3 sync s3://source-bucket/backups/ s3://target-bucket/backups/

# 3. 在目标环境恢复
milvus-backup restore -n product_backup_20240501 \
    -c product_search \
    --suffix _restored  # 恢复后 Collection 名加后缀，避免冲突

# 4. 验证
# 对比源环境和目标环境的 Collection 数据量、索引状态、随机搜索一致性
```

#### 步骤 2：Python 封装备份操作

```python
# step1_backup.py
"""Milvus 备份与恢复操作封装"""
import subprocess
import json
import time
from pymilvus import connections, utility, Collection

class MilvusBackupManager:
    """Milvus 备份管理器"""
    
    def __init__(self, milvus_host: str = "localhost", milvus_port: int = 19530,
                 config_file: str = "backup.yaml"):
        self.host = milvus_host
        self.port = milvus_port
        self.config = config_file
        connections.connect(host=milvus_host, port=milvus_port)
    
    def create_backup(self, collection_name: str, backup_name: str = None) -> dict:
        """创建 Collection 级备份"""
        if backup_name is None:
            backup_name = f"{collection_name}_backup_{int(time.time())}"
        
        result = subprocess.run([
            "milvus-backup", "create",
            "-n", backup_name,
            "-c", collection_name,
        ], capture_output=True, text=True)
        
        print(f"备份请求: {result.stdout}")
        if result.returncode != 0:
            print(f"错误: {result.stderr}")
        
        return {"name": backup_name, "collection": collection_name}
    
    def list_backups(self) -> list:
        """列出所有备份"""
        result = subprocess.run(
            ["milvus-backup", "list"],
            capture_output=True, text=True
        )
        print(result.stdout)
        return result.stdout
    
    def restore_backup(self, backup_name: str,
                       target_collection: str = None, suffix: str = "_restored"):
        """恢复备份"""
        cmd = ["milvus-backup", "restore", "-n", backup_name]
        if target_collection:
            cmd.extend(["-c", target_collection])
        if suffix:
            cmd.extend(["--suffix", suffix])
        
        result = subprocess.run(cmd, capture_output=True, text=True)
        print(f"恢复结果: {result.stdout}")
        if result.returncode != 0:
            print(f"错误: {result.stderr}")
        return result.returncode == 0
    
    def verify_restore(self, original_collection: str, restored_collection: str,
                       test_queries: list, sample_size: int = 100) -> dict:
        """验证恢复后的数据一致性"""
        orig_coll = Collection(original_collection)
        rest_coll = Collection(restored_collection)
        
        # 检查 1: 数据量一致
        orig_count = orig_coll.num_entities
        rest_count = rest_coll.num_entities
        count_match = orig_count == rest_count
        print(f"数据量: 原始={orig_count}, 恢复={rest_count} {'✓' if count_match else '✗'}")
        
        # 检查 2: Schema 一致
        orig_schema = str(orig_coll.schema)
        rest_schema = str(rest_coll.schema)
        schema_match = orig_schema == rest_schema
        print(f"Schema: {'✓ 一致' if schema_match else '✗ 不一致'}")
        
        # 检查 3: 索引状态一致
        orig_index = orig_coll.index()
        # 恢复后可能还没建索引，需要验证
        rest_has_index = rest_coll.has_index()
        print(f"索引: 原始=已构建, 恢复={'已构建' if rest_has_index else '未构建'}")
        
        return {
            "count_match": count_match,
            "schema_match": schema_match,
            "orig_count": orig_count,
            "rest_count": rest_count,
        }

# 使用示例
mgr = MilvusBackupManager()
mgr.create_backup("product_search")
mgr.list_backups()
mgr.restore_backup("product_search_backup_xxx")

verify = mgr.verify_restore(
    "product_search", "product_search_restored",
    test_queries=["户外折叠椅", "蓝牙耳机"]
)
print(f"\n验证结果: {json.dumps(verify, indent=2)}")
```

#### 步骤 3：使用 PyMilvus 原生 API 做轻量备份

```python
# step2_lightweight_backup.py
"""使用 PyMilvus API 做轻量级备份导出/导入（适合小数据量）"""
import json
import numpy as np
from pymilvus import Collection, connections, utility

connections.connect(host="localhost", port="19530")

def export_collection_data(collection_name: str, output_file: str,
                           batch_size: int = 10000):
    """导出 Collection 全部数据到 JSON"""
    collection = Collection(collection_name)
    collection.load()
    
    total = collection.num_entities
    print(f"导出 {collection_name}: {total} 条")
    
    all_data = []
    expr = "id >= 0"
    
    # 分页查询（Milvus 的 query 不能一次返回全量，需分页）
    offset = 0
    while offset < total:
        results = collection.query(
            expr=expr,
            output_fields=["*"],  # 返回所有字段
            limit=batch_size,
            offset=offset,
        )
        if not results:
            break
        all_data.extend(results)
        offset += len(results)
        print(f"  进度: {offset}/{total} ({offset*100//total}%)")
    
    # 向量字段 numpy 转 list（JSON 兼容）
    for item in all_data:
        for key, val in item.items():
            if isinstance(val, np.ndarray):
                item[key] = val.tolist()
    
    with open(output_file, "w", encoding="utf-8") as f:
        json.dump(all_data, f, ensure_ascii=False, indent=2)
    
    print(f"导出完成: {output_file} ({len(all_data)} 条)")
    collection.release()

def import_collection_data(collection_name: str, input_file: str,
                           batch_size: int = 500):
    """从 JSON 导入数据到 Collection"""
    with open(input_file, "r", encoding="utf-8") as f:
        all_data = json.load(f)
    
    collection = Collection(collection_name)
    total = len(all_data)
    print(f"导入 {collection_name}: {total} 条")
    
    for i in range(0, total, batch_size):
        batch = all_data[i:i+batch_size]
        entities = [[] for _ in range(len(batch[0]))]
        keys = list(batch[0].keys())
        
        for item in batch:
            for idx, key in enumerate(keys):
                entities[idx].append(item[key])
        
        collection.insert(entities)
        
    utility.flush([collection_name])
    print(f"导入完成: {total} 条")
```

## 4. 项目总结

### 4.1 优缺点对比

| 维度 | milvus-backup 工具 | PyMilvus 手动导出 | 对象存储快照 |
|------|-------------------|------------------|------------|
| 一致性保证 | 高（同时备份 etcd + S3） | 中（只导出数据） | 依赖对象存储的快照一致性 |
| 恢复便利性 | 一条命令 | 需写脚本 | 需手动组装 etcd |
| 大型 Collection | 支持（流式） | 慢（全量 query 内存压力大） | 快（对象存储原生） |
| 适用场景 | 生产环境标准方案 | 小数据量开发调试 | 有 S3 备份基础的团队 |

### 4.2 适用场景

- **灾难恢复**：误删 Collection 后快速恢复
- **跨环境迁移**：开发 → 测试 → 生产
- **版本升级前保险**：升级 Milvus 前全量备份
- **合规审计**：定期保留数据快照

**不适用场景**：需要实时数据同步的场景（应使用 Kafka 双写或 Change Data Capture）。

### 4.3 注意事项

- **备份前确保数据已 Flush**：否则 Growing Segment 中的数据不会被备份。
- **备份期间避免大量写入**：可能导致备份快照不一致。
- **恢复后需要重新 Load**：restore 只恢复了数据，不会自动 Load。
- **验证恢复结果**：每次备份后至少做一次抽样验证。

### 4.4 常见踩坑经验

1. **备份文件被自动清理**：Milvus Backup 有 retention 机制，默认保留最近 7 个备份。需要长期保留的备份要手动拷贝到冷存储。
2. **跨版本恢复失败**：Milvus 2.3 的备份不能恢复到 2.5。升级版本前务必确认备份兼容性。
3. **备份时等太久**：大型 Collection 备份需数小时，建议在业务低峰期（凌晨 2-6 点）执行，并设置合理超时。

### 4.5 思考题

1. 如果有 1TB 的 Milvus 数据需要每日备份，直接全量备份需要 4 小时。如何设计增量备份方案，把每日备份窗口缩短到 30 分钟以内？
2. 备份恢复后，搜索性能下降了 30%——可能是什么原因？如何排查？
