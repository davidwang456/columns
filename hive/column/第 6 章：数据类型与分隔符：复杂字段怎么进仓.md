## 1 项目背景（约 500 字）

埋点 JSON 里 `ext` 字段是嵌套对象：设备信息、实验参数、AB 桶号。数据工程把 JSON 打成 CSV 给数仓，结果 **逗号与引号转义** 一团糟，`SPLIT` 拆列对不齐，夜间 ETL **大量 NULL**。团队决定：在 Hive 层用 **struct/map/array** 显式建模，并统一 **SerDe（如 OpenCSVSerde / JSON SerDe）** 与 **NULL 语义**。本章用最小表结构演示 **类型选择、分隔符、脏行处理策略**。

再补一个**真实脏数据样例**：埋点里 `device.os` 偶发为 **空串** 而非 `NULL`，下游 `WHERE os IS NULL` 与 `WHERE os=''` 统计结果分裂；若再混入 **全角逗号** 与 **制表符**，CSV 手工拆分几乎必败。治理上要在 ODS 层约定：**空串是否规范化为 NULL**、**非法 UTF-8 字节是否丢弃或替换**、**超长 string 是否截断并打标**——这些都不是 Hive 类型系统替你决定的，但要写进 **数据契约**。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：为啥不全部 STRING，查的时候再 `CAST`？

**小白**：可以，但会失去 **早期校验** 与 **统计信息** 机会，且宽表全是 STRING 可读性差。折中是 **ODS 保留 raw string**，DWD 强类型化。

**大师**：技术映射：**Schema-on-read** 不等于 **不要 schema**；分层越深，类型应越严。

**小胖**：struct 和 map 怎么选？

**小白**：字段名稳定、结构演进可控 → `struct`；键集合开放（如实验参数）→ `map<string,string>`。array 适合序列，如曝光链路 ID 列表。

**大师**：还要考虑 **下游 Spark/Presto** 是否对复杂类型支持一致，避免「Hive 能写、Spark 读崩」。

**技术映射**：类型设计 ≈ **跨引擎契约**。

**小胖**：脏数据咋办？

**小白**：源头清洗优先；Hive 侧可用 `NULLIF`、正则 UDF、或 **bad record 旁路表**；极端用 `LATERAL VIEW` 容错。

**小胖**：JSON 一行一条不是挺好吗，为啥还要 CSV？

**小白**：历史包袱与 **采集端性能**：有些链路先落 **gzip 文本**，再批量转 ORC。JSON 行存对 **半结构化演进** 友好，但对 **极宽 JSON** 解析 CPU 更贵。

**大师**：技术映射：**格式选择 = 演进速度 × 解析成本 × 下游引擎一致性**；Hive 只是其中一站。

**技术映射**：把「用不用 JSON」落实为 **ODS 是否保留 raw_json 列 + DWD 强类型展开列** 的两段式。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：struct 示例

```sql
CREATE DATABASE IF NOT EXISTS demo_types;
USE demo_types;

CREATE TABLE IF NOT EXISTS user_event (
  uid BIGINT,
  device STRUCT<brand:STRING, os:STRING, ver:STRING>,
  tags MAP<STRING,STRING>
)
ROW FORMAT SERDE 'org.apache.hadoop.hive.serde2.lazy.LazySimpleSerDe'
WITH SERDEPROPERTIES ('collection.delim'='|', 'mapkey.delim'=':')
STORED AS TEXTFILE;
```

> 注：生产更常用 JSON/ORC + 对应 SerDe；此处演示 **集合分隔符** 概念，键名以环境可用 SerDe 为准。

### 步骤 2：插入与查询

```sql
INSERT INTO user_event VALUES (
  10001,
  named_struct('brand','ACME','os','Android','ver','14'),
  map('exp','B','bucket','3')
);

SELECT uid, device.brand, tags['exp'] FROM user_event;
```

**坑**：`collection.delim` 与数据内容冲突 → 行炸裂；需换分隔符或换格式。

### 步骤 3：array 与 `explode`

```sql
CREATE TABLE paths (uid BIGINT, path ARRAY<STRING>) STORED AS ORC;
-- INSERT 视环境 ORC 写入方式（可用 CTAS 或临时表）
SELECT uid, p FROM paths LATERAL VIEW explode(path) e AS p;
```

### 步骤 4：脏数据策略文档化

写清三选一：**拒绝整行 / 字段置 NULL / 写入 quarantine 分区**。

**验证**：用 10 条含异常字符的样本跑通解析，不使作业失败。

### 步骤 5（推荐）：JSON 行 + `get_json_object` 探查（无需改表也可写 SQL）

对 ODS 仍为 `STRING` 的 `payload` 列：

```sql
SELECT
  get_json_object(payload, '$.device.os') AS os,
  COUNT(*) AS cnt
FROM ods_event_raw
WHERE dt = '2026-04-20'
GROUP BY get_json_object(payload, '$.device.os')
ORDER BY cnt DESC
LIMIT 20;
```

**目的**：在 **未强类型化** 前快速发现 **空串/异常枚举/字段缺失率**；再决定是否 `ALTER TABLE ADD COLUMNS` 或建新 DWD 表做 `LATERAL VIEW json_tuple`。

### 步骤 6：`explode` 后的行数膨胀自检

```sql
-- 对单 uid 观察 path 长度与 explode 行数是否一致
SELECT uid, SIZE(path) AS n, COUNT(*) AS exploded_rows
FROM paths
LATERAL VIEW explode(path) e AS p
GROUP BY uid, SIZE(path);
```

**预期**：`exploded_rows = n`（在 `path` 无 NULL 元素时）；若不等，检查 **NULL 元素** 或 **重复 lateral**。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 复杂嵌套一次建模可读性高 | SerDe 配置学习曲线陡 |
| map 灵活承载扩展字段 | map 键爆炸影响统计与优化 |
| array+explode 表达力强 | explode 易放大行数导致倾斜 |
| TEXTFILE 调试直观 | 生产大表更推荐列存 |

### 适用与不适用

- **适用**：半结构化埋点、实验参数。  
- **不适用**：极高基数 map 当 join key（性能灾难）。  

### 注意事项

- **NULL** 与空字符串区别要写进口径。  
- **版本升级** 时 SerDe 行为可能变化，需回归。  

### 常见生产踩坑

1. CSV 内嵌换行未转义。  
2. UTF-8 与 GBK 混用。  
3. struct 字段改名导致下游 `device.brand` 全 NULL。

### 思考题

1. 将 JSON 直接落 ORC 与落 TEXT+JSON SerDe，运维与查询成本如何权衡？  
2. `MAP` 的 key 是否应限制长度与字符集？如何治理？  
3. 若 `tags` 的 key 从 20 个膨胀到 2 万个，对 **Metastore、统计信息、查询优化** 各可能产生什么副作用？
