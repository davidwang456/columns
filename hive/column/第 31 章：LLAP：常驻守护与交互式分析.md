## 1 项目背景（约 500 字）

BI 希望 **Hive 也能亚秒级** 探索，传统 Tez 冷启动与 **容器申请延迟** 不满足。引入 **LLAP（Live Long and Process）**：**常驻 Daemon + 缓存 + 向量化** 组合。成本：**YARN 长期占位**、**内存与 GC 调优**、**与批队列资源争用**。本章给 **架构组件（LLAP Daemon、IO elevator、cache）** 与 **何时值得上 LLAP** 的决策树。

补充：LLAP 的 ROI 不要只看 **查询 P95**，还要计入 **常驻容器费用** 与 **额外运维人力**。若 BI 实际只有 **少量高频查询** 值得加速，往往 **物化视图 / 预聚合表 / OLAP 引擎** 更便宜——本章给的是 **判断树**，不是推销 LLAP。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：LLAP 像便利店夜班员工一直待着？

**小白**：Daemon **常驻**，减少 **AM 反复申请**；并缓存 **热数据页**。

**大师**：技术映射：**LLAP = 用常驻资源换交互延迟**。

**小胖**：和 Presto 比呢？

**小白**：Presto **MPP 全内存倾向**；LLAP **更贴近 Hive 元数据与 ORC 生态**，但 **不是万能亚秒**。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：LLAP 像常驻便利店——夜里没人买也要付房租吗？

**小白**：**常驻守护进程 + cache** 有 **固定成本**；适合 **高并发交互查询** 或 **重复扫描热数据集**，不适合纯离线大吞吐独占。

**大师**：做 **TCO 表**：CPU、内存、SSD、与 **HS2-only** 对比同一套黄金 SQL；没有数据别先上规模。

**技术映射**：**LLAP 收益 ≈（热数据复用率 × 向量化收益）− 常驻开销**；冷启动与预热策略决定首查体验。

列出 **节点角色**：`HiveServer2 Interactive`、`LLAP daemons`、`Slider/YARN` 等（依发行版）。

### 步骤 2：PoC 指标

- 冷/热查询 **P50/P95**  
- **缓存命中率**  
- **与批作业** CPU 抢占

### 步骤 3：`SET hive.execution.mode=llap`（示意）跑 **TPC-H 子集**

**坑**：**ORC stripe** 与 cache 对齐。  
**坑**：**UDF 非线程安全** 在 LLAP 放大。

**验证**：ROI 表：**节省 BI 导出次数 × 人时** vs **额外 YARN 资源费**。

### 步骤 4：LLAP 容量估算（极简）

- 统计 BI **并发活跃会话** `C`  
- 估计单查询平均 **daemon vcores** `v`  
- 需要约 **`C * v`** 的常驻容量 + **20% headroom**（经验值，需压测校正）

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 显著降低交互延迟 | 常驻成本高 |
| 与 Hive 语义一致性好 | 部署与升级复杂 |
| 缓存可复用 ORC 页 | 缓存一致性与失效需设计 |

### 适用与不适用

- **适用**：高并发探索、数据科学家自助。  
- **不适用**：无稳定运维的小团队。  

### 注意事项

- **安全**：LLAP 与 Kerberos 集成。  
- **隔离**：与重批 **节点标签** 分离。  

### 常见生产踩坑

1. **缓存污染** 导致 **错误结果**（bug 或配置）。  
2. **小查询风暴** 打满 daemon。  
3. **升级** 不兼容 **cache 格式**。
