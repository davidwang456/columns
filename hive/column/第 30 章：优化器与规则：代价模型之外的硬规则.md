## 1 项目背景（约 500 字）

开启 CBO 后，某 **维度表实际膨胀** 但统计未更新，优化器仍 **广播** 导致 **Driver OOM**。另一案例：**规则优先** 于代价，强制 **分区裁剪** 反而 **过滤掉应保留分区**（谓词推导 bug 版本）。需要理解：**规则优化（RBO）与代价优化（CBO）** 的协作、**hint 优先级**、以及如何 **禁用单条规则** 做验证。

补充：生产排障时可用 **二分法**：先关闭 CBO 看是否回到「可接受计划」，再逐项打开规则（若版本支持细粒度开关）；每一步都保存 **EXPLAIN 文本 diff**，避免「感觉快了」但无人知道改了什么。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：代价模型像「导航选最短路径」？

**小白**：类似，但 **边权（基数、选择性）** 来自统计；错了就选错路。

**大师**：技术映射：**硬规则 = 无论代价都成立的安全/语义变换**。

**小胖**：能关掉某个规则吗？

**小白**：部分版本提供 **hint/参数** 关闭或 **fallback**；需要读源码/文档而非猜。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：硬规则像交规——会不会有时候「合法但不合理」？

**小白**：会：**过度裁剪谓词** 或 **版本 bug** 导致计划变差；所以要 **计划回归测试** 与 **灰度升级**。

**大师**：把 **「规则开关」** 纳入变更评审：谁有权在生产关、关多久、如何回补统计与重写。

**技术映射**：**规则层 = 语义安全网**；**CBO 层 = 代价选择**；两者冲突时优先 **语义正确**，再谈性能。

### 步骤 1：收集 **前后计划 diff**

升级前后对同一 SQL `EXPLAIN CBO`（若可用）文本 diff。

### 步骤 2：在源码搜索 **Optimizer 规则类名**

例如 `PartitionPruner` 相关（示意），记录 **触发条件**。

### 步骤 3：实验 **hint 固定 join 顺序** 与 **不 hint** 对比代价

**坑**：**过度 hint** 使 CBO **失效** 长期技术债。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 自动发现更优计划 | 黑盒感强 |
| 与统计联动可自我改进 | 误估时放大错误 |
| 规则保证语义安全（理想） | 版本 bug 影响面大 |

### 适用与不适用

- **适用**：复杂 join、多表星型模型。  
- **不适用**：统计长期缺失且无人治理环境。  

### 注意事项

- **hint 文档化** 原因与移除条件。  
- **升级必跑计划 diff**。  

### 常见生产踩坑

1. **空表** 统计导致 **join reorder 极端**。  
2. **动态分区裁剪** 与 **视图** 交互 bug。  
3. **常量折叠** 与 **时区** 组合问题。

### 思考题

1. 如何设计 **内部规则开关** 做金丝雀？  
2. **多物理计划枚举** 的边界在哪里？  
3. 当 **同一条 SQL** 在升级后计划变好但 **结果集变化**，你优先怀疑统计、规则还是语义变更？如何证明？
