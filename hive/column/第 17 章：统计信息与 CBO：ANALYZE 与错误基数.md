## 1 项目背景（约 500 字）

升级 Hive 小版本后，同一 `JOIN` SQL **从 20 分钟变 3 小时**。`EXPLAIN` 显示 **Join 顺序翻转**，广播表与大表角色互换。根因：**新优化器更依赖统计信息**，而团队从未跑 **`ANALYZE TABLE`**，基数全是默认值。本章覆盖 **表级/分区级统计、直方图（视版本）、自动收集参数**，以及 **错误统计导致更差计划** 时的回滚策略。

补充：**统计不是越新越好，而是越准越好**。大删大改后若只 `ANALYZE` 表级不刷新分区，CBO 仍可能拿着 **过期的分区行数**；反过来，**过于频繁的 ANALYZE** 会在高峰期拖垮元数据库。常见做法是 **分区就绪后异步统计** + **变更事件触发**（如每日分区写入完成信号）。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：CBO 不就是「更聪明」吗？

**小白**：聪明需要 **输入**：行数、NDV、min/max；缺省猜错会 **更蠢**。

**大师**：技术映射：**CBO = 代价函数 + 统计；Garbage in, garbage out**。

**小胖**：`ANALYZE` 会不会很慢？

**小白**：全表统计贵；常用 **分区级增量**、或 **抽样**（视功能）、或与 **写入作业绑定** 触发。

**大师**：要平衡 **统计新鲜度** 与 **Metastore/DB 压力**。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：统计信息像体检报告——多久做一次才不算「过期」？

**小白**：看 **表增长斜率** 与 **查询对基数敏感度**。大表日增、join 键高基数的，**日更或分区级增量**；小维表可以周更。关键是 **变更可观测**：谁触发了 ANALYZE、耗时、行数。

**大师**：把 **「统计新鲜度」** 写进数据质量看板：超过阈值未分析就 **告警 + 自动任务**，而不是等 CBO 选错路再人肉救火。

**技术映射**：**CBO 可信度 ∝ stats 新鲜度 × 直方图/NDV 粒度**；缺失时用默认估计等价于「蒙眼开车」。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：收集基本统计

```sql
ANALYZE TABLE dwd_impression PARTITION(dt='2026-04-20') COMPUTE STATISTICS;
-- 或表级，视 Hive 版本语法
```

### 步骤 2：对比 EXPLAIN 代价

对典型 Join SQL 在 **收集前/后** 各保存一份 `EXPLAIN CBO`（若可用）。

### 步骤 3：配置自动统计（谨慎）

查阅 `hive.stats` 相关配置，评估 **写入后自动 analyze** 对峰值的影响。

**坑**：**空分区** 或 **单行分区** 导致估算极端。  
**坑**：**多版本统计格式** 混存。

**验证**：建立 **周度分区统计作业** 与 SLA。

### 步骤 4：对比「无统计 / 有统计」计划文本 diff

1. 保存 `EXPLAIN` 文本到 `plan_before.txt`。  
2. `ANALYZE TABLE ... PARTITION(...)` 后保存 `plan_after.txt`。  
3. `diff -u plan_before.txt plan_after.txt | head -n 80` 观察 join 顺序是否变化。

### 步骤 5：当怀疑「错误统计」时的应急

- 会话级关闭部分 CBO（参数名依版本，如 `hive.cbo.enable` 类）**仅用于验证**；  
- 或对单条 SQL 使用 **hint 固定 join**（第 30 章）；  
- **切勿长期关闭 CBO** 作为默认手段。
### 环境准备（模板对齐）

- **依赖**：HiveServer2 + Beeline + HDFS（或 Docker），参见 [第 2 章](<第 2 章：HDFS 与 Hive 的最小可运行环境.md>)。
- **版本**：以 [source/hive/pom.xml](../source/hive/pom.xml) 为准；仅在非生产库验证。
- **权限**：目标库 DDL/DML 与 HDFS 路径写权限齐备。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 计划更稳、更少shuffle | 收集有成本 |
| Join reorder 更合理 | 错误统计放大错误 |
| 与动态分区 pruning 协同 | 元数据库压力 |

### 适用与不适用

- **适用**：大表 join、多阶段复杂 DAG。  
- **不适用**：极小表或测试库可关闭部分 CBO 以提速解析（权衡）。  

### 注意事项

- **升级后** 强制安排统计回填窗口。  
- **权限**：ANALYZE 触发的元数据写入权限。  

### 常见生产踩坑

1. 只收集表级未收集 **新分区**。  
2. **统计过期**（删大量数据后未更新）。  
3. 与 **Spark 统计** 两套口径冲突。

### 思考题

1. 如何用 **部分列统计** 降低开销？  
2. 当 CBO 选错广播侧时，**hint** 与 **session 参数** 如何选？  
3. 若 `ANALYZE` 本身成为 **Metastore DB 慢查询 Top1**，你如何拆分窗口与并行度？

### 跨部门推广提示

- **调度**：在分区就绪 DAG 尾部挂 **轻量 ANALYZE**。  
- **测试**：对关键 SQL 做 **计划快照 diff 告警**。
