## 1 项目背景（约 500 字）

业务希望 **一条 Hive SQL** 同时扫 **HDFS 历史** 与 **Kafka 近线** 做归因。现实：**延迟、语义、谓词下推、权限** 与原生表差异巨大；Kafka handler 更适合 **受控的联邦探查** 而非 **重 ETL**。本章读官方 handler README，列出 **能力矩阵：可下推谓词、可写回、吞吐上限、故障模式**。

补充：联邦查询最大的组织风险是 **「谁为结果负责」**——Kafka 延迟、schema drift、权限差异都会导致 **同一 SQL 不同时间不同答案**。若不上治理，最终会变成 **数据团队与业务团队互相甩锅**；需要事前签 **SLA：延迟上界、一致性级别、失败可观测性**。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 探索便捷 | 性能与稳定性风险 |
| 减少数据拷贝 | 语义与 Hive 原生不一致 |
| 与 BI 工具串联 | 运维边界模糊 |

### 适用与不适用

- **适用**：小范围探查、PoC、与 Druid 报表联邦。  
- **不适用**：核心 SLA 批处理链路。  

### 注意事项

- **配额与 ACL**。  
- **监控 lag**。  

### 常见生产踩坑

1. **SQL 误写 cross join** 放大拉取。  
2. **Kerberos** 与 broker 认证不一致。  
3. **handler jar** 版本与集群不匹配。

### 思考题

1. **Trino** 联邦 vs Hive handler 选型？  
2. 如何把 **热数据** 渐进 **落 ORC**？  
3. 联邦查询结果若与 **离线 Hive 真相层** 冲突，以谁为准？如何写进 SLA？

### 跨部门推广提示

- **架构**：联邦能力纳入 **技术雷达评审**。  
- **SRE**：broker **容量隔离**。
