## 1 项目背景（约 500 字）

公司 Hive 作业长期在 **MapReduce** 上运行，升级 CDP/自建栈后平台默认切到 **Tez**，试点团队尝试 **Hive on Spark**。现象：部分作业 **快了 3 倍**，少数作业 **OOM 或卡死**；调度器里 **失败重试风暴** 影响 SLA。需要一份 **引擎对比与迁移清单**：谁设置 `hive.execution.engine`、与 YARN 队列、shuffle 服务、小文件策略如何联动。

补充：**引擎切换不是「SET 一下」**。同一 SQL 在 Tez 上可能更吃 **container 内存与 shuffle 服务**，在 MR 上更吃 **阶段数与中间落盘**；Spark 则要关心 **dynamic allocation、shuffle service、与 Hive warehouse connector** 是否启用。迁移清单里务必写清 **回滚开关**（会话级、队列级、集群默认三级）。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：Tez 不就是「更快的 MR」？

**小白**：Tez 有 **DAG 级** 优化、容器复用，减少 MR 多轮落盘；但对 **内存与 shuffle 配置** 更敏感。

**大师**：技术映射：**MR = 粗粒度阶段模型**；**Tez = 细粒度 DAG + 容器常驻倾向**。

**小胖**：Spark 呢？

**小白**：Hive on Spark 把 Spark 当执行后端，适合 **与 Spark 生态统一**；但要处理 **版本对齐、shuffle 服务、与原生 Spark SQL 职责重叠**。

**大师**：迁移三问：**是否依赖 MR 独有特性**？**UDF 是否线程安全**？**统计与 CBO 是否已收集**？

**小胖**：那 ACID 表、LLAP 这种「高级货」换引擎会怎样？

**小白**：要走 **支持矩阵**：有的路径 **仅 Tez 验证**，Spark 上 **语义或锁** 可能不同。迁移前把 **高风险 SQL** 单列一类做 **金丝雀**。

**大师**：技术映射：**引擎 = 运行时契约**；换引擎等于换一半契约。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：换引擎像换交通工具——地铁换打车，会不会卡在上车点（兼容）？

**小白**：**向量算子、hint 语义、ACID、权限钩子** 都可能随引擎分叉；要维护 **引擎矩阵表**（特性 × 版本）。

**大师**：切换用 **金丝雀队列 + 黄金 SQL 集**；回滚开关写清 **「会话级 SET」** 与 **「集群默认」** 谁优先。

**技术映射**：**引擎切换 = 执行后端替换**；风险集中在 **语义差分 + 资源模型 + 客户端超时**。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：查看当前引擎

```sql
SET hive.execution.engine;
-- 或 hive.execution.engine=tez / spark / mr（以版本为准）
```

### 步骤 2：同一条 SQL 切换引擎对比

```sql
SET hive.execution.engine=mr;
EXPLAIN SELECT ...;

SET hive.execution.engine=tez;
EXPLAIN SELECT ...;
```

记录 **Vertex/Stage 数量、Map Join 提示** 差异。

### 步骤 3：迁移清单（写入团队 Wiki）

- [ ] YARN 内存 headroom 是否满足 Tez/Spark container  
- [ ] `tez.lib.uris` / Spark jars 分发是否正常  
- [ ] 历史 UDF 是否在 **多线程** 环境安全  
- [ ] 回滚开关：**会话级** 与 **队列级** 默认引擎

**坑**：混用引擎导致 **ORC split** 行为差异。  
**坑**：Spark shuffle 服务未起 → **shuffle fetch failed**。

**验证**：选 3 条 **黄金 SQL**（小中大）在预发全引擎跑通并归档耗时。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 引擎 | 优点 | 缺点 |
|------|------|------|
| MR | 成熟、内存门槛低 | 慢、阶段 IO 多 |
| Tez | 延迟与吞吐平衡较好 | 调优项多 |
| Spark | 与 Spark 栈统一、部分工作负载快 | 运维与版本耦合复杂 |

### 适用与不适用

- **默认 Tez**：多数离线 Hive 集群主流选择（视发行版）。  
- **不适用**：未评估就全量切 Spark 导致 **生产救火**。  

### 注意事项

- **队列默认** 与会话覆盖策略。  
- **回归测试** 含 ACID/LLAP 等特殊路径。  

### 常见生产踩坑

1. 客户端与集群 **Tez 版本不一致**。  
2. `spark.yarn.jars` 过大导致 **应用提交慢**。  
3. MR 硬编码 **Counter 名** 在监控里失效。

### 思考题

1. 若一条 SQL 在 Tez 上 **广播 join OOM**，如何在不回 MR 的前提下处理？  
2. Hive on Spark 与 **Spark SQL 读 Hive 表** 的职责如何划分？  
3. 若公司决定 **默认引擎切 Tez**，你如何设计 **「仅对失败作业自动 fallback」** 而不掩盖根因？

### 跨部门推广提示

- **运维**：维护 **引擎版本矩阵** 与镜像。  
- **测试**：对引擎切换做 **金丝雀批次**。
