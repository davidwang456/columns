## 1 项目背景（约 500 字）

大促前压测，BI 并发把 **单台 HS2 打满**，JDBC **排队超时**；夜间 ETL 与 **交互查询** 同队列，**相互饿死**。目标：**多 HS2 实例 + 负载均衡**、**YARN 队列隔离（探索 vs 生产）**、**连接与查询超时**、（可选）**ZooKeeper 服务发现**（扩展章）。

补充：HA 不只 HS2，还要考虑 **JDBC 客户端超时三连**：`connectTimeout`、`socketTimeout`、以及 BI 工具自己的 **查询超时**。否则服务端已断，客户端线程仍占满 **应用服务器连接池**，引发 **级联雪崩**。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：加机器不就行？

**小白**：要 **无状态水平扩展** 前提：会话不绑死单机 sticky（除非有状态）；**元数据与执行** 仍在共享服务。

**大师**：技术映射：**HS2 HA ≈ 多实例 + 统一接入层 + 后端共享**。

**小胖**：队列怎么切？

**小白**：`mapreduce.job.queuename` / Tez 队列映射，把 **大作业** 与 **小查询** 分队列，配 **容量与最大应用数**。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**大师**：HA 评审别只看 **VIP 漂移**，要看 **故障注入** 下 JDBC 是否 **指数退避**；队列隔离要写清 **批不得挤占交互** 的硬上限与超限处置。

**技术映射**：**接入层高可用 ≈ 多 HS2 + 无不当粘滞 + 客户端重试**；**隔离 ≈ YARN 队列 + HS2 并发上限 + 超时** 的交集。

**小胖**：连接池到底配多大？

**小白**：看 **单会话峰值内存**（fetchSize、大结果集）和 **HS2 线程/队列**；池过大易把 **连接风暴** 传到服务端，过小则业务空等。

**大师**：把 **max pool、checkoutTimeout、stmt timeout** 与 YARN **max apps** 放在一张评审表上，用 **固定混合压测脚本** 验收。

**技术映射**：**连接池 sizing ≈ min(HS2 吞吐, RM 容量, Metastore 承载)** 的漏斗最窄处。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：绘制当前拓扑

标出 **VIP/ZK、HS2×N、Metastore、YARN RM**。

### 步骤 2：配置片段（键名示意）

- `hive.server2.active.passive.ha.enable`（视发行版）  
- `hive.server2.zookeeper.namespace`  
- YARN **Capacity Scheduler** 队列：`explore`、`batch`

### 步骤 3：压测脚本

`beeline` 并行 `SELECT 1` 与 **真实大 SQL** 混合，记录 **P99 连接时间**。

**坑**：**Zookeeper session** 过短导致 **假死切换**。  
**坑**：**JDBC 重试风暴** 放大故障。

**验证**：单台 HS2 维护窗口，**VIP 摘除** 后客户端自动连另一台。

### 步骤 4：容量粗算（写在白板即可）

设峰值 **并发 JDBC 会话 = C**，单 HS2 可稳定承载 **S**（压测得），则至少需要 **`ceil(C/S)`** 台 HS2（再 +N 冗余）。若 **C 主要来自 BI**，还要考虑 **每个仪表盘的并发查询数** 而不是只看用户数。

### 步骤 5：发布检查（rolling restart）

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 提升可用与吞吐 | 拓扑复杂 |
| 队列隔离保护 SLA | 配置错误难排查 |
| 滚动升级友好 | ZK 依赖运维成本 |

### 适用与不适用

- **适用**：多团队共享集群。  
- **不适用**：极小集群硬上 HA（过度设计）。  

### 注意事项

- **Kerberos** 下 principal 与 DNS。  
- **审计日志** 汇聚。  

### 常见生产踩坑

1. **负载均衡长连接** 与健康检查冲突。  
2. **各 HS2 默认配置不一致**。  
3. **队列 max-am-percent** 过低饿死大作业。

### 思考题

1. **HS2 无状态** 有哪些例外（如某些 session hook）？  
2. 如何为 **交互式 BI** 单独限流？  
3. 若 LB 健康检查用 **短查询**，但长查询占满 worker，健康检查仍绿——如何避免「假健康」？

### 跨部门推广提示

- **SRE**：把 HS2 纳入 **标准故障演练**。  
- **开发**：客户端 **超时 + 指数退避**。
