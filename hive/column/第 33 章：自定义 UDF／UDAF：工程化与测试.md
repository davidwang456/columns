## 1 项目背景（约 500 字）

风控要把 **复杂规则**（多段条件、版本化系数）下沉到 SQL，纯内置函数 **200 行难维护**。团队开发 **GenericUDF**，但线上出现 **NPE、非确定性结果、LLAP 线程安全问题**。需要 **工程规范**：`@Description`、**类型推断**、`DeferredObject` 空值、**单元测试**、**Fat jar 分发**、**版本灰度**。

补充：**UDF 发布 = 供应链事件**。Fat jar 里若打入 **冲突版本的 protobuf/guava**，可能只在 **特定查询计划** 下才爆炸，极难排查。建议用 **shade relocations** 或 **provided scope** 严格管理依赖，并在 CI 跑 **`mvn dependency:tree` diff**。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：脚手架

在 `udf` 模块参考官方示例创建 `MyMaskUDF extends GenericUDF`。

### 步骤 2：本地 `mvn test` 用 `MiniHS2` 或轻量单测

对 **NULL、中文、超长输入** 断言。

### 步骤 3：部署

```sql
ADD JAR hdfs:///lib/udf-mask-1.0.jar;
CREATE TEMPORARY FUNCTION mask_phone AS 'com.example.hive.MyMaskUDF';
SELECT mask_phone(phone) FROM t LIMIT 10;
```

**坑**：**临时函数** 会话丢失 → 持久注册需 **创建永久函数** 流程。  
**坑**：**Guava/Protobuf** 与集群冲突。

**验证**：Code Review 检查 **无静态可变状态**。

### 步骤 4：UDF 单测最小集（建议必须覆盖）

- `NULL` 输入  
- 空串 / 超长 / 非 BMP 字符  
- **并发调用**（多线程压 1 分钟）  
- **与不同 Writable 类型** 的边界（`Text` vs `String`）

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 表达复杂逻辑 | 破坏向量化/增加维护面 |
| 可复用 | 依赖冲突风险 |
| 单测可保障质量 | 滥用导致 SQL 黑盒化 |

### 适用与不适用

- **适用**：经评估无法用 SQL 简洁表达。  
- **不适用**：可用 **内置+CTE** 解决的逻辑。  

### 注意事项

- **semver** 与 **CHANGELOG**。  
- **回滚 jar** 预案。  

### 常见生产踩坑

1. **时区/Locale** 导致结果漂移。  
2. **日志打爆** 在 map 中。  
3. **SecurityManager** 限制反射。

### 思考题

1. 如何将 UDF **迁移为 Native vector**（若团队有能力）？  
2. UDAF **部分聚合** 在 combiner 阶段的约束？  
3. 若 UDF 内部缓存 **静态 Map** 做字典，在 LLAP/多线程下会出现什么故障模式？

### 跨部门推广提示

- **平台**：UDF **注册表 + 所有者 + SLA**。  
- **测试**：强制 **覆盖率门槛**。
