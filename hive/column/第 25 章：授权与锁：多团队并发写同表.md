## 1 项目背景（约 500 字）

两团队同时对 **同一张 DWS 表** 做 `INSERT OVERWRITE` 与 `ALTER TABLE ADD COLUMNS`，夜间 DAG **互锁超时**。另有 **Ranger 拒绝** 与 **Hive 内部锁** 报错混淆。需要分清：**SQLStdAuth / Ranger 策略** vs **Hive LockManager（ZooKeeper/Db）** 的行为与 **排障顺序**。

补充：线上工单里先把报错分成 **「Permission denied」类** 与 **「Lock wait timeout / Could not acquire locks」类**，再决定找 **安全团队** 还是 **平台 on-call**。混谈会导致两边同时改配置，**越修越乱**。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：`SHOW LOCKS`（若启用）

对问题表执行，识别 **TABLE/ PARTITION** 级锁。

### 步骤 2：复现最小互斥

两个会话：一个 **长时间 SELECT**（是否加锁视配置），一个 **ALTER**。

### 步骤 3：阅读 `hive高级话题.md` 中锁与权限章节

整理 **你们发行版** 的 **LockManager 实现类名**。

**坑**：**Spark 与 Hive** 同时写 ACID 表锁冲突。  
**坑**：**ZK 会话丢失** 导致 **幽灵锁**。

**验证**：文档化 **解锁 SOP**（谁有权、如何审计）。

### 步骤 4：锁等待工单模板（复制即用）

1. 表名 + 库名 + **query id**（HS2）  
2. `SHOW LOCKS` 输出（如可用）  
3. 阻塞方 SQL 文本（脱敏）  
4. 决策：**等待 / kill / 调整 DDL 窗口** + 审批人

### 步骤 5：与 Ranger 的「责任边界」一句话

**Ranger 回答能不能做；LockManager 回答现在能不能做。** 两者都放行仍可能因 **HDFS 权限** 失败——排障顺序：**Ranger → HMS/HDFS ACL → 锁**。
### 环境准备（模板对齐）

- **依赖**：HiveServer2 + Beeline + HDFS（或 Docker），参见 [第 2 章](<第 2 章：HDFS 与 Hive 的最小可运行环境.md>)。
- **版本**：以 [source/hive/pom.xml](../source/hive/pom.xml) 为准；仅在非生产库验证。
- **权限**：目标库 DDL/DML 与 HDFS 路径写权限齐备。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 保护数据一致性 | 锁等待放大尾延迟 |
| 与 Ranger 组合细粒度授权 | 配置复杂 |
| 可观测（部分实现） | 误配导致死锁感知差 |

### 适用与不适用

- **适用**：多写入者、共享核心表。  
- **不适用**：纯读集群可弱化锁复杂度。  

### 注意事项

- **超时** 与 **重试退避**。  
- **变更窗口** 统一 DDL。  

### 常见生产踩坑

1. **长时间未提交事务** 占锁。  
2. **Impala/Hive** 混写同表无规范。  
3. **测试环境锁机制关闭** 与生产不一致。

### 思考题

1. **乐观锁 + 对象存储** 场景下 Hive 锁的局限？  
2. 如何用 **队列时间窗** 减少锁争用？  
3. 若 `kill query` 后锁未释放，你的 SOP 下一步是什么（ZK session / HMS / 重启边界）？

### 跨部门推广提示

- **数据平台**：发布 **DDL 变更日历**。  
- **SRE**：锁等待 **Prometheus 指标**（若可导出）。
