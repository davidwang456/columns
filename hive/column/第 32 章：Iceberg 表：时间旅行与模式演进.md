## 1 项目背景（约 500 字）

合规要求 **6 个月内任意时点重放** 广告曝光口径；传统 Hive 分区 **覆盖写** 难审计。评估 **Apache Iceberg** 表：**快照、时间旅行、schema evolution、隐藏分区**。Hive 作为 **catalog 与 SQL 引擎** 之一，需要理解 **Catalog 实现、与 Metastore 映射、Compaction 与 expire_snapshots** 运维。

补充：Iceberg 引入后，**「删数据」** 变成 **「切断快照指针 + 异步清理文件」** 的组合拳；合规团队会问：**删除可证明吗？** 答案通常在 **审计日志 + 快照元数据 + 对象存储版本控制** 的组合里，而不是一句 `DROP TABLE`。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：快照像 git commit？

**小白**：每次写入产生 **新快照指针**，旧数据文件可 **异步清理**（策略可控）。

**大师**：技术映射：**Iceberg = 表格式 + 事务日志 + 列统计**。

**小胖**：Hive SQL 都能用吗？

**小白**：受 **版本与语法覆盖** 限制；MERGE/DDL 与原生 Hive 表有差异，要以 **发行版矩阵** 为准。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：时间旅行听着酷——业务问「上周二的数」我能不能随便指一个快照？

**小白**：要约定 **快照与调度水位对齐**：否则「上周二」可能是 **未 compaction 的中间态** 或 **迟到数据未并入**。

**大师**：把 **查询默认分支（main）** 与 **审计读历史** 的流程分开；合规读必须 **显式 snapshot id + 审批**。

**技术映射**：**Iceberg 时间旅行 = snapshot 链 + 文件级元数据**；业务语义要对齐 **watermark 与提交边界**。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 强审计与演进 | 学习曲线陡 |
| 隐藏分区降元数据压力 | 运维 compaction 必须跟上 |
| 多引擎可读同表 | 版本组合敏感 |

### 适用与不适用

- **适用**：强合规、频繁 schema 变更、湖仓一体。  
- **不适用**：极简静态分区且团队无运维能力。  

### 注意事项

- **expire_snapshots** 与 **法务保留** 对齐。  
- **权限** 与 HMS 同步。  

### 常见生产踩坑

1. **Hive 与 Spark** 写同一 Iceberg 表 **catalog 配置不一致**。  
2. **忘记清理 orphan files**。  
3. **时间旅行** 与 **分区裁剪** 误解。

### 思考题

1. Iceberg **v2 删除文件** 对读放大影响？  
2. 如何用 **标签快照** 管理发版？  
3. 若 **法务要求删除单用户数据**，在 Iceberg 上「物理删除」与「逻辑删除」各如何落地？成本差异？

### 跨部门推广提示

- **合规**：把 **快照 ID** 写入审计日志。  
- **架构**：制定 **唯一 catalog 入口**。
