## 1 项目背景（约 500 字）

新同学写了一条「看似简单」的 `JOIN + GROUP BY`，跑了一小时。Leader 要求先贴 **`EXPLAIN`**，同学贴了一屏 **Stage/Vertex** 日志却说不清瓶颈在 **扫描、shuffle 还是倾斜**。本章建立 **阅读顺序**：先看 **分区裁剪与表扫描量** → **Join 类型（Common Join / Map Join）** → **Reduce 个数与数据分布提示**，并区分 **Tez/Spark/MR** 计划外形差异（细节见调优笔记）。

补充：**EXPLAIN 不是免死金牌**。在 Tez 上有时会出现 **「计划看起来很美，但运行时被 skew/数据倾斜击垮」**——因此第 2 步读完计划后，必须去 YARN 看 **task 直方图**（第 19 章）。本章训练的是 **把问题分桶** 的能力：先判断大类，再决定是调 SQL、调统计、还是调资源。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：EXPLAIN 能像编译器 `-O0` 那样告诉我优化了吗？

**小白**：它展示 **逻辑/物理计划草图**，不是精确耗时；但能看出 **是否走了广播 join、是否全表扫**。

**大师**：技术映射：**EXPLAIN ≈ 静态计划透视**；真实瓶颈还要 **日志 + YARN UI + metrics**（第 36 章）。

**小胖**：`EXTENDED` 和 `AUTHORIZATION` 类后缀要看吗？

**小白**：`EXTENDED` 常带更多 **序列化/统计** 信息；初学先 `EXPLAIN` 再逐步加。

**大师**：建立个人 **检查清单**：分区、Join、文件格式、倾斜 hint。

**小胖**：`EXPLAIN ANALYZE`（若版本支持）和 `EXPLAIN` 差在哪？

**小白**：带 **运行时统计** 的 explain 才能告诉你 **哪一步真慢**；纯 `EXPLAIN` 只是编译期视图。生产排障应 **两者结合**。

**大师**：技术映射：**编译期计划 + 运行时 counters = 完整故事**。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：`EXPLAIN` 像看地图——会不会地图画对了，路上照样堵车？

**小白**：会：**统计过期、数据倾斜、外部系统慢** 都不会完整出现在静态计划里；要 **EXPLAIN + 运行时指标** 一起看。

**大师**：给团队一张 **「计划红线词」** 表：如 `Cartesian`、`Full Table Scan`、`wrong column stats`；出现即触发 **二级排查模板**。

**技术映射**：**静态计划验证必要不充分条件**；**充分条件 ⊆ 计划 ∧ 统计 ∧ 资源 ∧ 数据分布**。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：基线 SQL

```sql
EXPLAIN
SELECT a.channel, COUNT(*)
FROM   dwd_impression a
JOIN   dim_channel b ON a.channel = b.id
WHERE  a.dt = '2026-04-20'
GROUP BY a.channel;
```

### 步骤 2：EXTENDED

```sql
EXPLAIN EXTENDED
SELECT ... -- 同上
```

在输出中搜索关键字：`Filter`、`Map Join`、`Partition`、`Stats`。

### 步骤 3：对比去掉分区条件

```sql
EXPLAIN SELECT ... -- 去掉 WHERE dt
```

观察 **扫描分区数** 或 **TableScan 行估算** 变化。

**坑**：统计信息缺失导致 **估算行数=未知** → 计划乱；`ANALYZE TABLE`（第 17 章）。  
**坑**：只看 **最外层** 忽略 **子查询物化**。

**验证**：能口述本 SQL 的三条 **潜在瓶颈假设** 并各给验证手段。

### 步骤 4：建立「计划阅读笔记」模板（每次慢 SQL 必填）

| 项 | 记录 |
|----|------|
| 引擎版本 | Tez ? / MR ? / Spark ? |
| 分区裁剪 | 是否只扫 1 个 dt |
| Join 类型 | Map / Shuffle / Hybrid |
| 广播侧 | 哪张表、估算行数 |
| Reducer 数 | 估算 vs 实际 |
| 可疑算子 | Union/Distinct/Window |

### 步骤 5：把 `EXPLAIN` 输出 **脱敏** 后贴工单

替换真实表名为 `t1/t2`，保留 **结构特征**；避免泄露 **敏感列名/业务活动代号**。
### 环境准备（模板对齐）

- **依赖**：HiveServer2 + Beeline + HDFS（或 Docker），参见 [第 2 章](<第 2 章：HDFS 与 Hive 的最小可运行环境.md>)。
- **版本**：以 [source/hive/pom.xml](../source/hive/pom.xml) 为准；仅在非生产库验证。
- **权限**：目标库 DDL/DML 与 HDFS 路径写权限齐备。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 零成本初步诊断 | 与真实执行有偏差 |
| 可版本化附在工单 | 输出冗长难读 |
| 帮助学习优化器行为 | 不同引擎字段名不同 |
| 与 CBO 信息联动 | 无统计时信息量骤降 |

### 适用与不适用

- **适用**：慢 SQL 初筛、Code Review。  
- **不适用**：精确定位 GC 或磁盘热点（需 profiling）。  

### 注意事项

- **保存计划文本** 附 issue。  
- **脱敏**：EXPLAIN 可能含表名/用户。  

### 常见生产踩坑

1. 只看 **第一个 Stage** 误判。  
2. **动态分区** 计划与实际分区数不符（统计过旧）。  
3. **视图展开** 后计划复杂难读 → 分层验证。

### 思考题

1. 如何判断 **Map Join** 条件是否满足且数据未撑爆内存？  
2. `VECTORIZED` 执行在计划中如何体现？（第 16 章预习）  
3. 若 `EXPLAIN` 显示 map join，但运行时报 **OOM**，可能是哪些 **估算错误** 或 **重复膨胀** 导致？

### 跨部门推广提示

- **测试**：对关键 SQL **基线化 EXPLAIN 摘要**，发版对比 diff。  
- **开发**：工单模板强制附 **EXPLAIN EXTENDED**。
