## 1 项目背景（约 500 字）

合规要求输出 **PII 字段血缘**：`user_phone` 从哪些 ODS 表、经哪些 JOIN/UDF 流入报表。手工维护 Excel **一周即过期**。目标：**从 Hive SQL 文本或执行计划提取列级血缘**，对接 **DataHub/Collibra/OpenLineage**。本章参考仓库 [hive数据血缘](../hive数据血缘/) 思路，讲 **Hook 点、AST 遍历、视图展开** 的挑战。

补充：血缘平台最怕 **「动态 SQL」**：调度器模板渲染后的 SQL 每次不同，AST 集合爆炸。治理策略通常是 **绑定到模板 ID + 参数域**，而不是强行解析每一次渲染结果全量入库。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：血缘是不是画箭头？

**小白**：是 **图数据库里的有向边**：表/列级，带 **变换类型**（投影、聚合、join）。

**大师**：技术映射：**AST lineage = 语法结构上的依赖传播**。

**小胖**：为啥难？

**小白**：**视图嵌套、子查询别名、星号展开、动态分区** 都增加不确定性。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**大师**：血缘工程要约定 **解析失败时的降级**：精确 / 近似 / 未知三档；合规材料引用的是 **置信度**，不是一张「全绿」的图。

**技术映射**：**血缘 SLA = f(解析成功率, 列级覆盖率, 与调度 DAG 一致性)**。

**小胖**：审计问「删用户数据」影响面，我甩哪张表？

**小白**：输出 **列血缘影响集** 与 **调度表级依赖** 两张视图，**差集** 必须人工签字说明（例如 UDF 黑盒、动态 SQL）。

**大师**：Hive Hook 产 **不可变事实事件**；合并、归因、报表在 **异步管道** 做，避免拖慢提交路径。

**技术映射**：**删除权分析 ≈ 列血缘反向传播 ∩ PII 标记 ∩ 保留期策略**。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 支撑 GDPR/删除权 | 自动化难完美 |
| 影响分析辅助重构 | 性能开销需异步 |
| 与数据目录协同 | 多引擎需统一模型 |

### 适用与不适用

- **适用**：中大型企业数据治理。  
- **不适用**：无治理编制仅技术自嗨。  

### 注意事项

- **版本漂移**：SQL 方言升级。  
- **隐私**：血缘图含敏感名需权限。  

### 常见生产踩坑

1. **只采表级** 无法满足合规。  
2. **忽略临时表**。  
3. **调度 DAG 边** 与 **列血缘** 混为一谈。

### 思考题

1. **OpenLineage** 与 Hive Hook 如何组合？  
2. **删除权** 请求如何用血缘 **定位影响面**？  
3. 当 UDF 为黑盒时，血缘应标记为 **精确/近似/未知** 哪一级？对合规声明有何影响？

### 跨部门推广提示

- **法务**：定义 **PII 分类法**。  
- **平台**：血缘 **与调度解耦存储**。
