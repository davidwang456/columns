## 1 项目背景（约 500 字）

工单描述：**「查询很慢但没报错」**。一线打开 YARN 看到 **单 reduce 99%**，HS2 日志只有 **INFO**。需要 **标准 SOP**：客户端 **session id** → HS2 **query id** → YARN **app id** → **DAG 视图** → **GC/Spill** → **HDFS slow node**。本章沉淀 **日志字段、指标项、审计表** 与 **Runbook 模板**。

补充：可观测性的敌人是 **「指标太多但无人 on-call 看」**。首屏建议只保留 **5 个黄金信号**：提交成功率、P95 排队时间、YARN 拒绝率、Metastore 慢查询数、NN RPC 延迟。其余指标进 **二级 dashboard**。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。

**小胖**：可观测性是不是打更多 log？

**小白**：**结构化日志 + 指标 + 链路关联** 比刷屏 `println` 有用。

**大师**：技术映射：**三件套 = logs + metrics + traces（可用 query id 串联）**。

**小胖**：审计和排障啥关系？

**小白**：审计偏 **合规谁查了啥**；排障要 **延迟分解**；字段可部分复用。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：指标多了像体检单太长——到底先看哪三项？

**小白**：**HS2 活跃会话、查询 P95、YARN pending** 往往是最短路径；再下钻 **GC、Metastore 延迟、HDFS slow node**。

**大师**：把 **SOP 第一屏** 固定成「三板斧」链接到仪表盘与日志检索模板；新人按图索骥，少在群里问「现在该看谁」。

**技术映射**：**可观测性闭环 = 指标（golden signals）→ 日志（trace id）→ 计划（EXPLAIN）** 三角互证。

### Runbook：慢但不挂（15 分钟版）

1. 取 **query id** → HS2 Web UI / 日志定位 **YARN application id**  
2. YARN UI 看 **最慢 stage** 是 map / shuffle / reduce  
3. 若 reduce 长尾 → 查 **倾斜/数据量**（第 19 章）  
4. 若 map 长尾 → 查 **小文件/split**（第 20 章）  
5. 全局慢 → 查 **NN/HMS/网络** 基线

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 缩短 MTTR | 存储与隐私成本 |
| 支撑容量规划 | 指标过多噪声大 |
| 合规审计兼备 | 需要平台治理 |

### 适用与不适用

- **适用**：所有生产 HS2。  
- **不适用**：无集中日志栈且无人看。  

### 注意事项

- **保留周期** 与 GDPR。  
- **敏感列** 哈希化。  

### 常见生产踩坑

1. **时钟漂移** 导致链路对不齐。  
2. **query id** 未回传给客户端。  
3. **Tez AM 日志** 未集中采集。

### 思考题

1. 如何把 **Hive metrics** 接入 Prometheus？  
2. **OpenTelemetry** 在 HS2 侧可行性调研？  
3. 审计日志里若不能存 SQL 全文，你如何用 **hash + 参数化模板** 兼顾排障与隐私？

### 跨部门推广提示

- **SRE**：把 Hive 纳入 **统一 on-call Runbook**。  
- **安全**：审计日志 **不可篡改存储**。
