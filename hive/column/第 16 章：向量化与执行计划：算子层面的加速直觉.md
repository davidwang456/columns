## 1 项目背景（约 500 字）

宽表 200 列，查询只取 10 列，Profiler 显示 CPU 大量耗在 **逐行解析与非向量化算子**。平台开启 **vectorized execution** 后，部分查询 **CPU 降半**。也有反例：某 UDF 非 **VectorUDF** 导致 **向量化整条链路降级**。本章讲 **开关、支持类型、与 EXPLAIN 中的 vectorized hint**，建立「**批处理 1024 行一批**」心智。

补充：向量化收益在 **「过滤 + 投影 + 简单聚合」** 链路上最明显；若热点在 **复杂 join 或大量 shuffle**，向量化只能算「锦上添花」。压测时务必同时打开 **OS 级 CPU perf**（如 `perf top`）确认热点确实在 **解释执行** 而非 **磁盘或网络**。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：向量化是不是 GPU？

**小白**：这里指 **CPU SIMD/批量迭代**，一次处理一批行，减少解释开销。

**大师**：技术映射：**向量化 = 算子从行迭代 → 列批迭代**。

**小胖**：为啥一用自定义 UDF 就慢？

**小白**：若 UDF 不能向量化，优化器可能 **整段降级** 或 **部分算子回退**。

**大师**：热点路径尽量用 **内置函数** 或官方 **VectorUDF** 模式（进阶开发）。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：那我开 A/B 对比，怎么确认「慢」不是磁盘而是向量化没开？

**小白**：同一 SQL、同一数据量，看 **CPU user%** 与 **每条算子耗时**；向量化开起来通常 **user 时间更饱满**、扫描阶段 **行吞吐** 上去。再配合 `EXPLAIN` 里是否出现 **vectorized** 描述。

**大师**：把对比结果写成 **三条固定样例**（可向量 / 必降级 / 边界类型），新人照抄就能自证；别只贴「快了 30%」这种无法复现的结论。

**技术映射**：**向量化有效性 ≈（计划走 VectorizedRowBatch）×（热点算子占 wall time 比例）**；否则优化的是「不痛的点」。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：开关（键名以版本为准）

```sql
SET hive.vectorized.execution.enabled=true;
SET hive.vectorized.execution.reduce.enabled=true;
```

### 步骤 2：对比查询

构造 **仅内置函数** 的过滤聚合 vs **含 Java UDF** 版本，比较 **Wall time** 与 `EXPLAIN` 中 vectorized 描述。

### 步骤 3：阅读执行日志

在 Tez/LLAP 日志中搜索 `vectorized` / `Vectorized` 关键字。

**坑**：**复杂数据类型** 或 **非 UTF-8** 路径限制向量化。  
**坑**：**Decimal 精度** 版本差异。

**验证**：记录 **三条可向量查询** 与 **一条必降级查询** 作为培训样例。

### 步骤 4：用 `SET` 快速 A/B（同一会话）

```sql
SET hive.vectorized.execution.enabled=true;
-- 跑同一条 SELECT，记录耗时

SET hive.vectorized.execution.enabled=false;
-- 再跑同一条 SELECT
```

> 生产变更应走配置中心；此处用于 **自证瓶颈是否在向量化路径**。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| CPU 友好 | 与部分 UDF/类型不兼容 |
| 与 ORC 列存协同 | 排障需理解降级原因 |
| 吞吐提升明显 | 内存峰值可能变化 |

### 适用与不适用

- **适用**：大扫描+简单表达式过滤聚合。  
- **不适用**：重度依赖非向量化 UDF 且无替代。  

### 注意事项

- **升级回归**：向量化相关 bug 常在 x.x.1 修复。  
- **JOIN** 部分路径向量化策略因版本而异。  

### 常见生产踩坑

1. 全局开启后 **个别作业错误结果** → 快速会话级关闭定位。  
2. **统计信息缺失 + 向量化** 组合误判（少见但需警惕）。  

### 思考题

1. 如何将高频自定义逻辑改写为 **内置函数组合**？  
2. LLAP 与 Tez 在向量化路径上的差异？  
3. 若 `EXPLAIN` 显示向量化开启但 CPU 仍高，你如何区分 **解压瓶颈** 与 **算子瓶颈**？

### 跨部门推广提示

- **开发**：UDF 需求评审增加 **向量化兼容性** 勾选。  
- **运维**：监控 **CPU profile** 前后对比。
