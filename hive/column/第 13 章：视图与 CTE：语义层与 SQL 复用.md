## 1 项目背景（约 500 字）

三家报表团队各自复制了一份「有效曝光」过滤逻辑：`click>0 AND fraud_flag=0 AND ...`，版本漂移后 **同一周三个 GMV**。数据组决定在 Hive 建 **视图 `v_valid_impression`** 作为 **语义层入口**，并鼓励复杂脚本用 **CTE** 分段。需要澄清：**视图默认非物化**，每次查询展开；**权限** 与 **底层表变更** 的影响链。

补充：**视图不是文档**。若不在仓库里对 `CREATE VIEW` 文本做版本管理，只靠线上 `SHOW CREATE VIEW` 导出，极易出现 **「谁改了视图却不发公告」**。推荐把视图 DDL 与 **指标字典** 绑定同一 MR：视图变更 = 指标版本号 bump。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：视图是不是把结果存下来加速？

**小白**：Hive 传统视图多为 **逻辑视图**，查询时 **内联子查询**；要加速用 **物化视图**（中级篇）或预计算表。

**大师**：技术映射：**视图 ≈ 命名查询 + 口径封装**。

**小胖**：那 CTE 和子查询有啥差别？

**小白**：可读性更好，可 **链式** 表达多步清洗；是否 **多次物化** 取决于优化器（不要想当然只算一次）。

**大师**：复杂 CTE 仍可能 **重复扫描**，要用 `EXPLAIN` 验证。

**小胖**：视图能隐藏敏感列吗？

**小白**：可以 **列级投影**，但若用户有底层表权限，仍可能绕过视图；需 **授权模型** 配合（第 24～25 章）。

**小胖**：视图能带参数吗？比如只看自己的渠道？

**小白**：Hive 没有「真正的参数化视图」像存储过程那样通用；常见替代是 **下游用 CTE + 变量替换**（调度器渲染 SQL）或 **用 `LATERAL VIEW` + 维表过滤**。

**大师**：技术映射：**参数化需求 ≈ 调度模板或 BI 语义层**，不要硬塞进视图定义里导致字符串拼接地狱。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：视图套视图像俄罗斯套娃——拆开会不会只剩空气？

**小白**：**视图合并** 可能改变 **谓词下推边界**；CTE 也可能 **物化与否** 因版本/配置不同，导致 **性能与语义坑**。

**大师**：把 **「可合并 / 必须物化」** 写成团队规范；关键报表用 **命名 CTE + 注释版本号**，避免暗改。

**技术映射**：**语义层 = 逻辑封装**；**性能层 = 展开后与优化器交互**；两者要分开评审。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：创建视图

```sql
CREATE VIEW IF NOT EXISTS v_valid_impression AS
SELECT id, channel, creative, dt
FROM   ods_impression
WHERE  fraud_flag = 0 AND click > 0;
```

### 步骤 2：CTE 分段

```sql
WITH base AS (
  SELECT * FROM v_valid_impression WHERE dt='2026-04-20'
),
agg AS (
  SELECT channel, COUNT(*) cnt FROM base GROUP BY channel
)
SELECT * FROM agg WHERE cnt > 1000;
```

### 步骤 3：`SHOW CREATE VIEW`

确认依赖表列表，纳入 **血缘** 文档。

**坑**：视图基于 **可变更 UDF** → 结果漂移。  
**坑**：在视图定义里写 **当前日期函数** 导致 **非确定性**（慎用 `current_date()` 于口径视图）。

**验证**：修改底层表新增一列，确认视图 **兼容** 或需 `ALTER VIEW`（视版本能力）。

### 步骤 4：`EXPLAIN` 看视图展开后的扫描

```sql
EXPLAIN EXTENDED
SELECT channel, COUNT(*) FROM v_valid_impression WHERE dt='2026-04-20' GROUP BY channel;
```

在输出中确认 **`ods_impression` 仍被读取** 且 **谓词下推** 是否如预期；若出现 **意外 cross product**，多半是视图内 join 被误改。

### 步骤 5：CTE **物化**与否的快速判别思路

1. 把 CTE 改成 **临时表** `CREATE TEMPORARY TABLE tmp AS ...` 再查，对比耗时。  
2. 若耗时显著下降，说明原 CTE 路径存在 **重复计算** 或 **优化器未折叠** 的迹象（非绝对）。
### 环境准备（模板对齐）

- **依赖**：HiveServer2 + Beeline + HDFS（或 Docker），参见 [第 2 章](<第 2 章：HDFS 与 Hive 的最小可运行环境.md>)。
- **版本**：以 [source/hive/pom.xml](../source/hive/pom.xml) 为准；仅在非生产库验证。
- **权限**：目标库 DDL/DML 与 HDFS 路径写权限齐备。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 统一口径、减少复制粘贴 | 非物化不提速 |
| 简化下游 SQL | 深层嵌套影响优化 |
| 可做列裁剪辅助安全 | 权限绕过需系统治理 |
| CTE 提升可读性 | 误用可能导致重复扫描 |

### 适用与不适用

- **适用**：口径稳定的过滤/投影。  
- **不适用**：强实时低延迟查询（需物化或 OLAP）。  

### 注意事项

- **版本管理**：视图定义进 Git。  
- **Breaking change**：改视图前先通知下游。  

### 常见生产踩坑

1. 视图 **循环依赖**。  
2. 视图引用 **临时表** 被删。  
3. `SELECT *` 视图在 schema 演进中 **悄悄变列**。

### 思考题

1. 如何检测 CTE 是否被 **物理化多次**？  
2. 视图与 **RLS（行级权限）** 组合时的边界？  
3. 若 `ods_impression` 增加敏感列 `device_id`，旧视图 `SELECT *` 是否会「自动暴露」？如何防范？
