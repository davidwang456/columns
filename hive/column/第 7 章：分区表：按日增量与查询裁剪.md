## 1 项目背景（约 500 字）

「用户曝光明细」单日 200 亿行，分析师查询却常写 `WHERE dt = '2026-04-20'` 仍扫描全表——原因是 **未按日期分区** 或 **分区键与过滤列不一致**。另一起事故：回溯三个月历史做活动复盘，作业扫描 **900 个分区**，YARN 队列被占满。团队要求：**事实表默认按业务日期分区**；查询模板强制带分区条件；分区数 **上限与生命周期** 写入规范。

再记 **「伪分区」反模式**：把 `dt` 拼进 **普通列**（如 `pdate`）里却不 `PARTITIONED BY`，或把分区键放在 **文件路径里却不注册到 Metastore**——这两种都会让 `WHERE dt=...` **无法裁剪**，扫描成本与「没分区」几乎一样。治理上可启用 **严格模式**（如 `hive.strict.checks.largequery` 一类配置，名称以版本为准）在预发 **拒绝无分区过滤的全表扫**，把问题左移到开发自测阶段。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：分区是不是把大表切成很多小表？

**小白**：逻辑上是一张表，物理上 **每个分区子目录** 存一段数据。Metastore 记录分区键 → 路径映射，优化器可做 **分区裁剪**。

**大师**：技术映射：**分区 ≈ 粗粒度索引 + 生命周期管理单元**。

**小胖**：分区键选 `dt` 还是 `hour`？

**小白**：看查询模式与单分区大小。过细分区 → 元数据膨胀；过粗 → 单分区扫描仍大。常见：**天分区 + 桶** 或 **小时分区用于大促专项**。

**大师**：还要考虑 **数据到达延迟**：业务日与自然日不一致时要有 **对齐规则**（如按事件时间还是入库时间）。

**技术映射**：分区键 = **时间与到达语义的选择**。

**小胖**：为啥会扫全分区？

**小白**：函数包分区列 `WHERE substr(dt,1,7)='2026-04'` 可能阻止裁剪；或动态分区写入未 `MSCK`；或统计信息过旧。

**小胖**：双分区 `dt + hour` 会不会很难写 SQL？

**小白**：查询要写全两个键才裁剪干净；收益是 **大促小时级回滚/重算** 更细粒度。常见折中：**平时天分区**，大促窗口临时切换 **小时子目录**（需作业与元数据协同）。

**大师**：技术映射：**分区粒度 = 查询模式 × 元数据成本 × 单分区大小** 的三角平衡。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：创建分区表

```sql
CREATE DATABASE IF NOT EXISTS demo_part;
USE demo_part;

CREATE TABLE IF NOT EXISTS impression (
  id BIGINT,
  channel STRING,
  creative STRING
)
PARTITIONED BY (dt STRING)
STORED AS PARQUET;
```

### 步骤 2：静态分区写入

```sql
INSERT OVERWRITE TABLE impression PARTITION (dt='2026-04-20')
SELECT 1, 'c1', 'cr1' FROM dual
UNION ALL
SELECT 2, 'c2', 'cr2' FROM dual;
```

> `dual` 若不存在可用 `VALUES` 或已有小表替代。

### 步骤 3：验证裁剪（阅读 EXPLAIN）

```sql
EXPLAIN EXTENDED
SELECT COUNT(*) FROM impression WHERE dt='2026-04-20';
```

观察计划中 **分区个数** 是否为 1。

### 步骤 4：展示分区元数据

```sql
SHOW PARTITIONS impression;
```

**坑**：`WHERE dt BETWEEN ...` 却分区为 `string` 非 `yyyy-MM-dd` 规范 → 字典序比较错误。  
**坑**：外部数据已落 HDFS 但未 `MSCK REPAIR TABLE` → 查不到新分区。

**验证**：同一条 SQL 带与不带 `dt` 条件，对比 **读取数据量**（日志或 YARN counters）。

### 步骤 5：`VALUES` 写法（Hive 3+ 常见，可替代 `dual`）

```sql
INSERT OVERWRITE TABLE impression PARTITION (dt='2026-04-20')
VALUES
  (1, 'c1', 'cr1'),
  (2, 'c2', 'cr2');
```

若集群版本不支持，退回 `UNION ALL` 子查询或 `SELECT ... FROM staging LIMIT n`。

### 步骤 6：分区注册与「能查但慢」的区分

| 症状 | 更可能原因 | 下一步 |
|------|------------|--------|
| `SELECT` 直接报 **partition not found** | 元数据缺分区 | `MSCK` / `ADD PARTITION` |
| 能出结果但 **扫描量巨大** | 裁剪失效 / 谓词包函数 | 改 SQL 或改分区策略 |
| 结果 **偶发旧数据** | 缓存/视图/外表路径竞态 | 查写入与刷新顺序 |

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 显著降低扫描成本 | 分区数爆炸拖累 Metastore |
| 便于归档与 TTL | 错误分区键导致静默错误结果 |
| 与调度按日对齐自然 | 跨分区大 Join 仍需优化 |

### 适用与不适用

- **适用**：按时间增长的事实表。  
- **不适用**：维度很小且更新频繁、不适合大量子目录的场景（考虑其他组织方式）。  

### 注意事项

- **字符串分区** 规范 `yyyy-MM-dd`。  
- **时区** 全链路统一 UTC 或业务时区。  

### 常见生产踩坑

1. 动态分区打开导致 **一夜创建上万个分区**。  
2. `INSERT` 未指定分区列写入 **默认分区**。  
3. 外部表新增 HDFS 目录未修复分区。

### 思考题

1. 将 `dt` 设计为 `STRING` 与 `DATE` 类型在裁剪与工具链上各有什么影响？  
2. 如何为「迟到数据」设计 **回补分区** 流程？  
3. 若分析师习惯写 `WHERE dt between '2026-04-01' and '2026-04-30'`，你如何验证 **不会扫到其它年月的同名字典序**？（提示：`yyyy-MM-dd` 与 `yyyyMMdd` 混用的灾难。）
