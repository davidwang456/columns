## 1 项目背景（约 500 字）

资深开发在升级 Hive 后，将 `NOT IN` 子查询改写为 `LEFT ANTI JOIN`，性能 **突增**；另一处仅调整 **谓词位置**，CBO **选择不同广播侧** 导致 **OOM**。团队需要 **源码级鸟瞰**：`Driver` compile 流程、**AST**、**QB（QueryBlock）**、**逻辑计划算子**、**物理 Task（MapWork/ReduceWork）** 与 Tez/Spark **DAG 映射**。本章不逐行读代码，给 **阅读地图与断点建议**。

补充：读源码时建议 **自底向上**：先锁定一条 SQL 的 **AST 节点类型**（如 `TOK_QUERY` 相关），再跟踪到 **SemanticAnalyzer** 产出的 **QB/QBMetaData**，最后到 **Task 工厂**（`MapRedTask`/`TezTask` 等，视版本）。不要从 `main()` 随机漂流，否则一周只能读半个类。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：本地构建或 IDE 打开 `ql` 与 `parser` 模块

在 `org.apache.hadoop.hive.ql` 包搜索 `Driver`、`SemanticAnalyzer`。

### 步骤 2：对简单 SQL 打开 debug

```sql
SELECT COUNT(*) FROM t WHERE dt='2026-04-20';
```

断点观察 **parse → analyze → genPlan** 调用栈（示意）。

### 步骤 3：`EXPLAIN EXTENDED` 输出与 **算子名** 对照

把 `TS`（TableScan）、`FIL`、`JOIN` 等与 **源码 Operator** 概念对齐。

**坑**：**不同引擎后端** 同一逻辑计划 **物理化不同**。  
**坑**：**视图展开** 在 AST 哪阶段发生需查证版本。

**验证**：能向同事 **白板画出三阶段方块图**。

### 步骤 4：推荐断点清单（IntelliJ 友好）

1. `Driver.compile`（或等价入口）  
2. `ParseDriver.parse`  
3. `SemanticAnalyzer.analyzeInternal`  
4. `GenMapRedUtils` / `TezCompiler`（依引擎）

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 读懂编译链利于深度排障 | 源码体量大 |
| 支撑自定义规则扩展 | 版本差异显著 |
| 与优化器/权限天然衔接 | 学习曲线陡 |

### 适用与不适用

- **适用**：平台开发、疑难 SQL、二次开发。  
- **不适用**：仅需写 SQL 的业务分析师（用 EXPLAIN 即可）。  

### 注意事项

- **分支对齐**：阅读与线上一致 tag。  
- **贡献回社区** 前先最小复现。  

### 常见生产踩坑

1. **自定义 hook** 插入点错误导致 **计划被改坏**。  
2. **升级** 未读 **Release note** 的语义变更。  
3. **视图+子查询** 展开顺序差异。

### 思考题

1. **Correlation subquery** 在 Hive 中如何脱相关？  
2. **Calcite**（若使用）与原生优化器边界？  
3. 若你在 hook 里修改 AST，如何防止 **与后续优化规则冲突** 导致不可复现计划？

### 跨部门推广提示

- **架构**：把 **编译阶段图** 放入内部培训。  
- **SRE**：重大升级前安排 **SQL 回归集**。
