## 1 项目背景（约 500 字）

大促当天 **`channel='LIVE_TOP1'`** 曝光量占全站 40%，Reduce 阶段 **单 task 运行 5 小时**，其余 task 早已结束——典型 **数据倾斜**。团队总结三板斧：**识别（skew key 分布）**、**参数（skew join、split）**、**改写（加盐、两阶段聚合）**。本章用 **GROUP BY 与 JOIN** 两类场景演示。

补充：**倾斜有时是「假倾斜」**。例如某分区 **总量正常** 但 **文件极不均匀**（少数超大 ORC 文件），map 阶段就会出现 **长尾 map**；这与 reduce 侧 key 倾斜不同，治理手段是 **合并小文件/调 split**（第 20 章）。读 YARN 时先看 **map 长尾还是 reduce 长尾**，再决定三板斧用哪板。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：倾斜是不是「有人排队特别长」？

**小白**：shuffle 时同一 key 去同一 reducer，热点 key → **单 reducer 撑爆**。

**大师**：技术映射：**倾斜 = 分区函数与数据分布失配**。

**小胖**：先改 SQL 还是先调参？

**小白**：**先定位 key**：`GROUP BY channel ORDER BY cnt DESC LIMIT 20`；或 **stage 任务耗时直方图**。

**大师**：确认是 **真倾斜** 而非 **单分区过大**。

**技术映射**：加盐 = **人为打散 key 再二次聚合**。

**小胖**：JOIN 倾斜和 GROUP BY 倾斜一样治吗？

**小白**：思路类似（打散/拆分热点），但 join 还要处理 **空键、重复键、广播与 shuffle 组合**；有时要先 **过滤 NULL join key** 再谈倾斜。

**大师**：技术映射：**倾斜治理 = 先消灭无意义键，再打散有意义热点**。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：参数调了还是慢，是不是说明「三板斧」没用？

**小白**：可能是 **倾斜键集合变了**（活动、大促），或 **上游空分区** 导致估算又飘了。三板斧是 **定位框架**，不是一次配置管一年。

**大师**：把每次倾斜治理的 **SQL_ID、倾斜键 TopN、采用策略（随机盐/两阶段聚合/改写）** 记进知识库，下次同类活动直接复用。

**技术映射**：**倾斜治理 = 观测（skew metric）→ 分类（键/空值/cartesian）→ 代价可控的改写**；缺少观测只有参数是赌命。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：诊断 SQL

```sql
SELECT channel, COUNT(*) cnt
FROM dwd_impression WHERE dt='2026-04-21'
GROUP BY channel
ORDER BY cnt DESC
LIMIT 20;
```

### 步骤 2：尝试 skew join 参数（名称以版本为准）

```sql
SET hive.optimize.skewjoin=true;
SET hive.skewjoin.key=100000; -- 示例阈值
```

### 步骤 3：改写——加盐聚合

```sql
SELECT channel, SUM(subcnt) cnt
FROM (
  SELECT channel, CONCAT(channel, '_', CAST(FLOOR(RAND()*10) AS STRING)) sk, COUNT(*) subcnt
  FROM dwd_impression WHERE dt='2026-04-21'
  GROUP BY channel, CONCAT(channel, '_', CAST(FLOOR(RAND()*10) AS STRING))
) t
GROUP BY channel;
```

> 生产常用 **确定性 salt**（如 hash(uid)%N）而非 `RAND()`，此处仅示意打散。

**坑**：`RAND()` 每次不同导致 **结果不稳定**——禁止用于生产指标。  
**坑**：skew 参数过大导致 **过度拆分**。

**验证**：YARN UI 中 **reduce 最大耗时/均值** 比值显著下降。

### 步骤 4：确定性加盐示例（替代 `RAND()`，仅示意）

```sql
-- 假设 impression_id 可作为稳定盐粒（生产请按业务选择）
SELECT channel, SUM(cnt) AS cnt
FROM (
  SELECT
    channel,
    CONCAT(channel, '#', CAST(pmod(hash(impression_id), 8) AS STRING)) AS salt_key,
    COUNT(*) AS cnt
  FROM dwd_impression
  WHERE dt = '2026-04-21'
  GROUP BY channel, CONCAT(channel, '#', CAST(pmod(hash(impression_id), 8) AS STRING))
) x
GROUP BY channel;
```

**说明**：`hash` 函数名/可用性依版本；核心是 **同一行每次跑批盐值不变**。

### 步骤 5：JOIN 倾斜的「拆分热点 key」模板（文字）

1. 将 topK 热点 `channel` 写入 **小表 hot_channels**。  
2. `UNION ALL` 两条路径：`channel IN hot` 走 **加盐 join 或 mapjoin+复制维表扩展行**；`channel NOT IN hot` 走普通 join。  
3. 结果 `UNION ALL` 后去重/聚合（注意 **口径**）。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 方法 | 优点 | 缺点 |
|------|------|------|
| 参数 skew join | 侵入小 | 不万能 |
| 加盐改写 | 可控 | SQL 复杂、易出错 |
| 提高并行+更大 container | 简单 | 成本上升 |

### 适用与不适用

- **适用**：长尾幂律分布、促销热点。  
- **不适用**：全表均匀但仍慢（查 IO/格式/统计）。  

### 注意事项

- **口径**：两阶段聚合要与 **财务/数据科学** 对齐。  
- **随机盐** 与 **可重复跑批** 冲突。  

### 常见生产踩坑

1. 把 **NULL join key** 当倾斜主因未处理。  
2. **Broadcast** 与 **倾斜** 同时误判。  
3. **Map-only** 任务看似快但 **下游放大**。
