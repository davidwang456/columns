## 1 项目背景（约 500 字）

测试集群 **匿名 JDBC** 暴露在办公网，被爬虫扫到 **10000 端口**，有人误连 **生产同名库**（DNS 配置错误）。安全组要求：**Kerberos 或等价强认证**、**TLS 加密**、**仅内网/VPN**、**审计日志落 SIEM**。本章给 **最小安全基线清单** 与 **常见配置坑**，不深究 Ranger 策略细节（可结合高级话题笔记）。

补充：**认证解决「你是谁」之后，还要解决「从哪来」**——内网扫描与 **SSRF 跳板** 仍是常见攻击面。生产 HS2 应默认 **拒绝公网路由**，并通过 **堡垒机 / Zero Trust** 收敛入口；TLS 证书建议使用 **企业 CA** 并开启 **双向 TLS（mTLS）** 若合规要求更高（实现复杂度上升）。

## 2 项目设计（约 1200 字）


> **角色（对齐 [template.md](../template.md)）**：**小胖**（生活化比喻、抛问题）· **小白**（原理、边界、风险与备选）· **大师**（选型与「**技术映射**」承接）。  
> **对话结构**：小胖开球 1～2 轮 → 小白追问 2～3 轮 → 大师解答并引出下一子话题；全文循环 **2～3 次**，覆盖本章核心概念。

**小胖**：Kerberos 像「每次进门先对暗号」？

**小白**：Ticket **限时**、**双向验证**，防止 **明文密码** 在网络上飘。

**大师**：技术映射：**认证解决你是谁；授权解决你能做什么**（下一章）。

**小胖**：TLS 还要吗？

**小白**：Kerberos 管身份，**TLS 管传输层窃听与中间人**；生产建议 **叠加**。


> **§2·第三轮**（对齐 [template.md](../template.md) 的第三循环）

**小胖**：Kerberos 像门禁卡——我本地开发能不能关掉图省事？

**小白**：短期可以走 **公司沙箱账号**，但长期会养出 **「只在非 Kerberos 环境能跑」** 的 SQL 与脚本，上生产全炸。

**大师**：开发机也走 **keytab + kinit 自动化**（或公司 SSO 代理），把 **认证失败** 纳入 CI 冒烟，别当「环境问题」私了。

**技术映射**：**认证边界 = 身份（principal）× 票据生命周期 × 网络可达域**；任何一角松动都是横向移动风险面。

## 3 项目实战（约 1500～2000 字）

### 步骤 1：基线检查表

- [ ] `hive.server2.authentication` 非 `NONE`（生产）  
- [ ] `hive.server2.use.SSL` 或网关 SSL  
- [ ] 安全组 **仅白名单 CIDR**  
- [ ] **keytab** 分发与 **定期轮换** 流程

### 步骤 2：Beeline Kerberos 示例骨架

```bash
kinit -kt app.keytab app/principal@REALM
beeline -u "jdbc:hive2://hs2:10000/default;principal=hive/_HOST@REALM"
```

### 步骤 3：证书与 hostname 校验

确认 **CN/SAN** 与 JDBC URL 主机名一致。

**坑**：**时钟漂移** 导致 `Clock skew too great`。  
**坑**：**反向 DNS** 缺失导致 principal 校验失败。

**验证**：用 **非授权 principal** 尝试 `SELECT` 应被拒绝。

### 步骤 4：安全基线「最小勾选集」

- [ ] 禁止 `hive.server2.authentication=NOSASL`（或等价弱模式）暴露公网  
- [ ] `hive.server2.thrift.sasl.qop=auth-conf`（若需加密完整性+保密性，依版本）  
- [ ] HS2 **bind address** 非 `0.0.0.0` 或配合安全组白名单  
- [ ] 审计日志落 **集中 SIEM** 并设保留周期  
- [ ] **keytab rotation** Runbook（每季度演练）

### 步骤 5：TLS 常见错配对照

| 症状 | 可能原因 |
|------|----------|
| ` PKIX path building failed` | 客户端 truststore 缺链 |
| hostname verify failed | URL 用 IP 但证书只签域名 |
| handshake OK 但 Hive auth fail | principal 与加密层混淆 |
### 环境准备（模板对齐）

- **依赖**：HiveServer2 + Beeline + HDFS（或 Docker），参见 [第 2 章](<第 2 章：HDFS 与 Hive 的最小可运行环境.md>)。

## 4 项目总结（约 500～800 字）

### 优点与缺点

| 优点 | 缺点 |
|------|------|
| 显著降低误用与窃听风险 | 配置与排障成本高 |
| 与企业 IAM 集成 | 客户端分发 keytab 需流程 |
| 满足合规审计 | 性能微小开销 |

### 适用与不适用

- **适用**：所有非本机 toy 环境。  
- **不适用**：无运维承接的「半套 Kerberos」。  

### 注意事项

- **keytab 权限** `0400`。  
- **审计** 与 **PII 脱敏** 联动。  

### 常见生产踩坑

1. **混用 auth 模式** 导致部分客户端仍匿名。  
2. **负载均衡 SSL 终结** 与后端 plain 混配错误。  
3. **YARN 容器** 未取 delegation token。

### 思考题

1. **SPNEGO** 与 JDBC Kerberos 在 BI 工具链上的差异？  
2. 如何用 **网关（Knox）** 收敛入口？  
3. 若要求 **mTLS**，JDBC URL 与 truststore 分发流程应如何设计以避免「每人一份私钥」？

### 跨部门推广提示

- **安全**：把 Hive 端口纳入 **漏洞扫描范围**。  
- **开发**：禁止在仓库提交 **keytab**。
