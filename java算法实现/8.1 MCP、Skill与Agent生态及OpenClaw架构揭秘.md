# 8.1 MCP、Skill 与 Agent 生态及 OpenClaw 架构揭秘

> **第8章 智能体与工具链** | Model Context Protocol · Agent 循环 · Cursor Skill · OpenClaw

## 背景

2024–2026 年间，以大语言模型（LLM）为核心的应用形态从「纯对话」快速演进到 **工具调用（Tool Use）**、**长程任务（Agent）** 与 **可插拔上下文（MCP）**。同一时期，编辑器与 IDE（如 Cursor）通过 **Skill（技能包）** 把领域知识以文件形式注入模型行为。开源社区也出现 **OpenClaw** 等 **本地优先、多通道、可扩展技能** 的 Agent 运行时，常与 MCP 组合使用。

本章从 **协议与数据流** 出发，说明 MCP、Skill、典型 Agent 控制流的关系，并概括 **OpenClaw** 类系统的常见分层实现（不绑定某一版本的内部源码行号——生态迭代极快，实现以各项目官方仓库为准）。

**参考与延伸阅读**（建议优先阅读官方文档）：

- Model Context Protocol 架构与规范：<https://modelcontextprotocol.io/docs/concepts/architecture>、<https://modelcontextprotocol.io/specification/latest>
- MCP 参考实现与示例 Server：<https://github.com/modelcontextprotocol/servers>
- OpenClaw 项目（开源 Agent 框架，持续迭代）：<https://github.com/openclaw/openclaw>

## 8.1.1 MCP（Model Context Protocol）：在「揭秘」什么？

MCP 要解决的问题可以一句话概括：**把「模型能用的上下文与动作」从宿主应用里拆出来，变成标准、可复用的 Server**，由 **Host** 内的 **Client** 按统一协议连接。

### 角色划分（官方架构）

| 角色 | 职责 |
|------|------|
| **MCP Host** | 承载 AI 能力的应用（如 Claude Desktop、VS Code、各类 Agent 壳），负责协调多个 Client |
| **MCP Client** | 与 **某一个** MCP Server 维持一条连接，替 Host 拉取上下文、转发调用 |
| **MCP Server** | 对外提供工具、资源、提示模板等能力；可在本机或远程运行 |

Host 与 Server 之间 **一对多**：每个 Server 对应一个 Client 实例、一条独立连接（概念上如此；实现上可能是进程、子进程或 HTTP 会话）。

### 两层结构：数据层 + 传输层

**数据层（内层）** —— 基于 **JSON-RPC 2.0** 的消息语义：

- **生命周期**：初始化、能力协商（capabilities）、关闭连接等。
- **Server 侧原语（Primitives）**（最常被开发者扩展的部分）：
  - **Tools**：可执行动作（读文件、调 API、跑查询等），通常带 JSON Schema 描述参数，由模型或编排层决定何时调用。
  - **Resources**：只读或偏「上下文供给」的数据源（文件片段、URI 内容、业务对象快照等）。
  - **Prompts**：可复用的提示模板或交互范式，由 Server 声明、Client 侧选用。
- **Client 侧能力**：例如让 Server 请求 Host 对 **LLM 采样**、向用户 **elicitation**、写日志等（具体以规范版本为准）。

**传输层（外层）** —— 只负责 **怎么把 JSON-RPC 消息送过去**，常见两类：

1. **Stdio**：Host 拉起本地子进程，**标准输入/输出** 作为双向通道；延迟低、无网络暴露，适合本机工具链（文件、Git、本地 DB）。
2. **Streamable HTTP**：远程 Server；客户端多用 **HTTP POST** 发请求，**SSE（Server-Sent Events）** 等支持流式响应；配合 **Bearer / API Key / OAuth** 等做鉴权。

**底层实现揭秘（实现者视角）**：

- 你在调试器里看到的往往不是「魔法」，而是：**JSON 序列化后的 RPC 帧** + **传输适配器**（pipe 或 HTTP+SSE）。
- Server 进程内：**注册表**（tools/resources 列表）+ **分派器**（根据 method 名调用对应 handler）+ **可选鉴权/限流**。
- Host 侧：**把模型输出的 tool call 映射为 MCP 的 `tools/call`**，再把结果 **回填到下一轮对话上下文**。

MCP **不规定** Host 内部如何用 LLM、如何做记忆；它只规范 **上下文与动作的交换格式**——这是它和「Agent 框架」的分界线。

## 8.1.2 Skill（以 Cursor Skill 为例）：和 MCP 是什么关系？

**Skill** 在 Cursor 等环境中通常指：**仓库或全局目录下的 `SKILL.md`（及附属资源）**，在对话或任务匹配时被加载，作为 **持久、可版本化的系统级说明**，约束模型 **如何做事、遵循何种步骤、何时用何种工具**。

### 典型结构（概念）

- **元数据**：名称、适用场景、是否只读等（具体字段随产品演进）。
- **正文**：分步骤工作流、约束、输出格式、安全边界（例如禁止擅自改生产配置）。
- **可选附件**：脚本、模板、检查清单。

### 与 MCP 的互补关系（实务理解）

| 维度 | Skill | MCP |
|------|--------|-----|
| 主要载体 | Markdown / 项目内文件 | 独立进程或远程服务 |
| 强项 | **行为规范、流程、团队约定** | **可执行能力、数据源、统一工具接口** |
| 更新方式 | Git 与 Code Review | 部署/升级 Server、改配置 |
| 典型失败 | 模型未遵守说明（需规则+评测） | 连接失败、权限过大、工具注入攻击 |

**揭秘**：Skill 的「底层」在宿主侧往往是——**启动任务前做文件发现与注入**（把 SKILL 内容并入 system/developer 消息或等价上下文槽位）；而 MCP 的「底层」是 **RPC + 进程/HTTP**。二者常 **同时使用**：Skill 定义「何时、以何纪律调用工具」，MCP 提供「工具真身」。

## 8.1.3 Agent：控制流与「工具调用」如何实现？

**Agent** 在此指：**以 LLM 为决策核心、在循环中反复调用外部工具直到满足停止条件** 的系统（单智能体视角；多 Agent 则是多个此类循环 + 编排）。

### 最小控制循环（ReAct 式抽象）

1. **输入**：用户目标 + 当前上下文（含 Skill、RAG、MCP 拉取的 Resource）。
2. **推理**：模型生成自然语言 + **结构化 tool call**（常见为 JSON：工具名 + 参数）。
3. **执行**：Host 解析 tool call → 调 MCP / 本地 API / 浏览器自动化等。
4. **观察**：将工具输出 **追加为对话历史**（或写入状态机）。
5. **终止条件**：得到最终答案、达到步数上限、预算耗尽、或显式 `stop`。

**底层实现揭秘**：

- **与 MCP 的衔接点**：Tool 定义（name、description、parameters schema）常被 **原样或映射** 进模型 API 的 `tools` 字段；模型返回的 `tool_calls` 被 Host 转成 **MCP Client** 的 RPC。
- **状态**：除对话历史外，常见 **任务队列、子目标栈、检查点**；生产系统会加 **幂等、超时、重试、审计日志**。
- **安全**：工具即代码执行边界——**最小权限 MCP Server**、**人工确认敏感操作**、**沙箱** 是标配讨论题，而非可选项。

## 8.1.4 OpenClaw：开源 Agent 框架的常见架构（揭秘「通常怎么做」）

**OpenClaw** 指社区中一类 **本地优先、可连接多种聊天通道与模型后端** 的开源 Agent 运行时（实现语言多为 **Node.js/TypeScript** 生态，具体以 [openclaw/openclaw](https://github.com/openclaw/openclaw) 仓库为准）。公开资料与架构文章中反复出现的 **模式** 如下（用于理解「底层在干什么」，而非替代阅读其源码）。

### 常见分层（逻辑模块）

1. **通道适配层（Channel Adapters）**  
   将 WhatsApp、Telegram、Slack、Discord 等平台消息 **归一化** 为内部事件（文本、附件、会话 ID、用户 ID）。

2. **会话与路由层（Session / Router）**  
   维护 **多用户、多会话** 状态；决定某条消息交给哪个 **Agent 会话**、是否新建线程、如何做 **速率限制**。

3. **模型后端抽象（LLM Provider）**  
   统一调用 Claude、OpenAI、Gemini、本地 Ollama 等；处理 **流式输出、重试、计费与降级**。

4. **工具与上下文层**  
   - 内置能力：浏览器自动化、定时任务、文件工作区等（以项目文档为准）。  
   - **MCP Client**：把外部 MCP Server 的 Tools/Resources **挂到同一条 Agent 循环** 上，与上一节描述一致。

5. **持久化与「文件即记忆」**  
   不少设计采用 **工作区目录下的 Markdown**（如会话摘要、用户偏好、长期记忆索引）实现 **可审计、可 Git 管理** 的记忆，而不是完全黑盒数据库——本质是 **约定目录结构 + 读写工具**。

6. **技能/插件生态**  
   社区贡献的自动化工作流、第三方集成常以 **插件包** 或 **预置 Skill** 形式分发；**供应链安全**（恶意插件、过度权限）已被安全研究与监管机构关注——部署时需 **白名单、签名、最小权限**。

### 与 MCP + Skill 的合流

- **OpenClaw 类框架** ≈ **通道 + 会话路由 + LLM +（可选）MCP 工具 + 文件记忆**。  
- **Cursor** ≈ **编辑器 Host + Skill 文件 +（可选）MCP**。  
二者差异主要在 **产品形态与默认通道**，**底层积木**（RPC 工具、提示注入、循环决策）高度同构。

## 8.1.5 Java 生态中的对应能力（简要）

本章主题不依赖 Java，但若需在 **JVM** 侧对齐同一套思想，可关注：

- **MCP Java SDK**（若官方或社区维护）：实现 **MCP Client/Server** 与 stdio/HTTP 传输。
- **Spring AI** 等框架中的 **Tool Calling、MCP 集成**（版本与包名随发行说明变更）。
- **自研**：用 **Jackson** 处理 JSON-RPC，**ProcessBuilder** 托管 stdio 子进程，或用 **WebClient** 对接 HTTP+SSE。

原则不变：**协议层与业务层解耦**；Java 侧重点是 **可靠的连接生命周期、线程模型与超时**。

## 总结

1. **MCP** 的「底层」是 **JSON-RPC 数据层** + **Stdio/HTTP 传输层**；核心价值是 **Tools / Resources / Prompts** 的标准化交换，而非替代 LLM。  
2. **Skill** 的「底层」多是 **宿主在运行时把 SKILL 文档注入上下文**；适合 **流程与规范**，与 MCP **能力供给** 互补。  
3. **Agent** 的「底层」是 **推理—工具执行—观察** 的循环，Host 负责 **解析 tool call、调用 MCP、写回历史**。  
4. **OpenClaw** 类框架的「底层」通常是 **通道归一化 + 会话路由 + 多模型后端 +（可选）MCP + 文件化记忆与插件**；具体类名与模块以官方仓库为准。  
5. **安全与治理**（权限、审计、插件供应链）与算法正确性同等重要，尤其在把 MCP 接到生产环境时。

生态变化快，**以各项目官方文档与 spec 版本为最终依据**；本章侧重 **概念骨架与实现分界**，便于你阅读源码时快速对号入座。
