# 第21章：权限、组织与项目模板治理

## 1. 项目背景

**业务场景**：某集团级研发组织在 SonarQube 中管理着 200+ 个项目，涉及 500 名开发者、30 个团队。最初 SonarQube 只有一个 admin 账户，所有人都用这个账户登录和操作。不久后，出现了一系列安全事故：

- 某开发者在"清理 Issue"时误操作了核心交易系统的项目，将状态为 "Confirmed" 的 80 个安全 Vulnerability 标记为 "False Positive"——导致安全团队丢失了 3 个月的跟踪记录。
- 外包团队的开发者在新项目创建时默认获得了对所有项目的 Execute Analysis 权限，导致外包团队可以扫描任何项目的代码并查看 Issue 列表——这违反了公司的数据隔离政策。
- 当一个开发者离职时，管理员忘记吊销其 Token，该 Token 在离职后仍被用来访问内部项目 2 周。

这些事故的根本原因是**权限治理缺失**——没有角色分离、没有最小权限原则、没有自动化权限管理。

**痛点放大**：

- **开发/测试/运维的职责不分**：测试人员可以修改 Quality Gate，开发者可以管理用户权限
- **新项目权限黑洞**：创建新项目后权限设置没有任何默认模板，需要人工配置
- **离职权杖管理缺失**：开发者离职后 Token 和账户仍然存活
- **审计不可追溯**：谁在什么时候改了哪个项目的权限？没有任何记录

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（发现自己可以进入任何项目的设置页面修改 Quality Gate）："大师！我发现我居然能把支付系统的 Quality Gate 从 'Core System Gate' 改成 'Internal Tool Gate'——门禁瞬间就从 5 个条件变成了 1 个。这也太危险了吧？"

**大师**："这就是权限治理的意义。在 SonarQube 中，权限分为三个层级：全局权限、项目权限、组件权限。你之所以能修改 Quality Gate，是因为你的账户被授予了项目的 'Administer' 权限——这通常不应该给普通开发者。"

**小白**："权限模型是怎样的？有哪些角色和权限？"

**大师**："SonarQube 的权限模型包括：

**全局权限**（影响整个 SonarQube 实例）：
- Administer System：安装插件、管理系统配置（最高权限）
- Administer Quality Profiles：管理 Quality Profile
- Administer Quality Gates：管理 Quality Gate
- Execute Analysis：执行分析扫描
- Create Projects：创建新项目

**项目权限**（影响单个项目）：
- Administer：管理项目配置（用户、权限、Quality Gate 绑定等）
- Administer Issues：管理 Issue（分配、标记误报、批量修改）
- Execute Analysis：执行分析扫描
- Issue Administer：管理 Issue 安全变更
- Security Hotspot Administer：审查 Security Hotspot
- User：查看项目页面（最低权限）
- Code Viewer：查看源码（不包含 Issue 管理）

在实际团队中：

- 开发者 → User + Execute Analysis + Issue Administer（自己创建的 Issue）
- 测试/QA → User + Issue Administer
- 架构师/Tech Lead → Administer + Administer Issues
- 运维/管理员 → Administer System（全局）"

**小胖**："我们有 200 个项目，每个项目都要手动配置一遍权限？"

**大师**："用 Permission Template。你可以创建权限模板，比如 'Developer Template'、'QA Template'、'Security Auditor Template'，每个模板定义了不同角色在项目上的默认权限。当新项目创建时，自动应用某个默认模板。"

## 3. 项目实战

### 3.1 环境准备

- SonarQube 管理员权限
- 至少已有 3 个项目和 5 个用户

### 3.2 分步实现

**步骤 1：创建用户和组**

进入 **Administration → Security → Users**，手动创建用户，或配置 LDAP/SAML 集成。

通过 API 批量创建：

```bash
# 创建用户
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/users/create" \
  -d "login=zhangsan" \
  -d "name=张三" \
  -d "password=Changeme@123" \
  -d "email=zhangsan@company.com"

# 创建组
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/user_groups/create" \
  -d "name=backend-developers" \
  -d "description=后端开发组"
```

**步骤 2：设计角色权限矩阵**

| 权限 | 开发组 | 测试组 | 架构组 | 安全组 | 运维组 |
|------|--------|--------|--------|--------|--------|
| Administer System | ❌ | ❌ | ❌ | ❌ | ✅ |
| Administer Quality Profiles | ❌ | ❌ | ✅ | ❌ | ❌ |
| Administer Quality Gates | ❌ | ❌ | ✅ | ❌ | ❌ |
| Execute Analysis (全局) | ✅ | ❌ | ✅ | ❌ | ❌ |
| Create Projects | ❌ | ❌ | ✅ | ❌ | ❌ |
| Administer (项目) | ❌ | ❌ | ✅ | ❌ | ❌ |
| Administer Issues (项目) | ✅ | ✅ | ✅ | ✅ | ❌ |
| Security Hotspot Admin | ❌ | ❌ | ✅ | ✅ | ❌ |
| User (项目) | ✅ | ✅ | ✅ | ✅ | ✅ |
| Code Viewer (项目) | ✅ | ✅ | ✅ | ✅ | ❌ |

**步骤 3：创建 Permission Template**

进入 **Administration → Security → Permission Templates**。

创建 "Backend Developer Template"：

1. 点击 "Create"
2. Name: `Backend Developer Template`
3. Project Key Pattern: `com\.company\.backend\..*`（自动匹配后端项目）
4. 添加组权限：
   - Group `backend-developers`: User, Execute Analysis, Issue Administer
   - Group `qa-team`: User, Issue Administer
   - Group `architects`: Administer
5. 设为默认模板（勾选 "Set as default"）

通过 API 批量配置：

```bash
# 创建模板
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/permissions/create_template" \
  -d "name=Frontend Developer Template" \
  -d "description=前端开发团队的默认权限模板"

# 获取模板 ID
TEMPLATE_ID=$(curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/permissions/search_templates" \
  | python3 -c "import sys,json;d=json.load(sys.stdin);print([t['id'] for t in d['permissionTemplates'] if t['name']=='Frontend Developer Template'][0])")

# 添加组权限到模板
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/permissions/add_group_to_template" \
  -d "templateId=$TEMPLATE_ID" \
  -d "groupName=frontend-developers" \
  -d "permission=user"

curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/permissions/add_group_to_template" \
  -d "templateId=$TEMPLATE_ID" \
  -d "groupName=frontend-developers" \
  -d "permission=codeviewer"
```

**步骤 4：为新项目自动应用模板**

设置 Project Key Pattern 实现自动匹配：

```bash
# 更新模板的匹配规则
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/permissions/update_template" \
  -d "id=$TEMPLATE_ID" \
  -d "projectKeyPattern=com\.company\.frontend\..*"

# 将此模板设为默认（所有新项目自动应用）
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/permissions/set_default_template" \
  -d "templateId=$TEMPLATE_ID"
```

**步骤 5：管理 Token 安全**

进入 **Administration → Security → Users**，管理中定期审计 Token：

```bash
# 列出所有活跃 Token
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/user_tokens/search" \
  | python3 -c "
import sys, json
data = json.load(sys.stdin)
for token in data.get('userTokens', []):
    print(f\"{token['login']:20s} {token['name']:25s} last used: {token.get('lastConnectionDate','never')}\")
"

# 吊销闲置超过 30 天的 Token（需要编写脚本批量处理）
```

**步骤 6：配置 LDAP/SAML 集成**

集成企业目录服务，实现用户自动同步：

**LDAP 示例（`sonar.properties`）**：

```properties
# LDAP 配置
sonar.security.realm=LDAP
ldap.url=ldap://ldap.company.com:389
ldap.bindDn=cn=sonarqube,cn=Users,dc=company,dc=com
ldap.bindPassword=secret
ldap.user.baseDn=cn=Users,dc=company,dc=com
ldap.user.request=(&(objectClass=user)(sAMAccountName={login}))
ldap.user.realNameAttribute=displayName
ldap.user.emailAttribute=mail

# 组映射
ldap.group.baseDn=cn=Groups,dc=company,dc=com
ldap.group.request=(&(objectClass=group)(member={dn}))
ldap.group.idAttribute=cn
```

### 3.3 权限审计检查

```bash
#!/bin/bash
# 审计所有项目的权限
echo "=== SonarQube 权限审计 $(date) ==="

# 1. 检查具有全局管理权限的用户
echo -e "\n[1] 全局管理员："
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/permissions/global_permissions" \
  | python3 -c "
import sys,json
data = json.load(sys.stdin)
for perm in data['permissions']:
    if 'Administer System' in perm['key']:
        print(f'  User: {perm.get(\"login\",\"Group\")}')"

# 2. 检查有 Administer 权限的项目（高风险）
echo -e "\n[2] 项目管理员："
for proj in proj1 proj2 proj3; do
  users=$(curl -s -u admin:Sonar@2024Admin \
    "http://localhost:9000/api/permissions/users?projectKey=$proj&permission=admin")
  echo "  $proj: $(echo $users | python3 -c 'import sys,json;print(len(json.load(sys.stdin)["users"]))') 个管理员"
done
```

## 4. 项目总结

### 4.1 权限治理检查清单

| 检查项 | 标准 |
|--------|------|
| 最少管理员数 | 全局管理员 ≤ 3 人 |
| Token 轮换周期 | 每 90 天轮换 |
| 离职权杖吊销 | 离职当日吊销所有 Token |
| 权限模板覆盖 | 所有新项目自动应用默认模板 |
| 审计日志 | 保留至少 6 个月的权限变更记录 |

### 4.2 适用场景

- **多团队协作的企业环境**（≥ 3 个团队、≥ 10 个项目）
- **有安全合规要求的行业**（金融、医疗、政务）
- **外包/外部贡献者管理**

### 4.3 注意事项

1. **最小权限原则**：任何角色只拥有完成其工作所需的最小权限——不要为了方便给所有人 Admin 权限。
2. **定期审计**：每季度检查一次权限分配，吊销不再需要的权限。
3. **离职流程**：将 SonarQube 账户吊销纳入 HR 离职 checklist。

### 4.4 思考题

1. 如果一个外包团队的成员需要访问一个指定的项目进行 Issue 修复，但不应看到其他项目的代码——你如何配置权限和项目可见性？
2. SonarQube 支持 LDAP 组映射。如果公司的 AD 中有一个 "All Employees" 组，你会将它映射到 SonarQube 的哪个权限级别？

> **答案提示**：第1题将项目可见性设为 Private，仅为外包用户所在组授予该项目的 User + Issue Administer 权限。第2题建议只给 User（或 Nothing），不要给 Execute Analysis——不是所有人都需要执行扫描。

---

> **推广计划提示**：权限治理应在推广 SonarQube 的第 2 阶段（约第 2-3 周）引入——太早会让团队觉得繁琐，太晚会暴露安全风险。建议安全团队和 DevOps 一起设计权限矩阵，然后由质量负责人在周会上花 15 分钟讲解"你的角色能做什么"。
