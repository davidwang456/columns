# 第23章：覆盖率、测试报告与质量数据可信度

## 1. 项目背景

**业务场景**：某公司的质量大盘显示整体代码覆盖率为 87%，CI 中的 Quality Gate 自三个月前就一直显示绿色。CTO 在董事会上汇报时自豪地展示了这个数字——直到一次安全渗透测试发现，核心支付模块的实际覆盖率只有 23%，因为大量无效测试被计入统计。

进一步审查发现五个问题：
- 13 个 JaCoCo 报告路径中，有 4 个指向了空目录（配置错误但没报错）
- 集成测试的覆盖率被错误地合并到了单元测试报告中（导致覆盖范围虚高）
- SonarQube 中的 `lines_to_cover` 和 JaCoCo 报告中的 `LINE` counter 对不上——因为 exclude 配置不一致
- 覆盖率报告生成在 Docker 容器内，路径是 `/app/src/`，但 SonarScanner 在宿主机执行，路径是 `/home/jenkins/workspace/src/`
- 1 个核心 Service 类的测试覆盖率是 100%，但所有断言都是 `assertNotNull(result)`——没有任何业务逻辑验证

**痛点放大**：覆盖率数据的可信度取决于整个数据链路的完整性——从测试框架生成报告，到构建工具传输报告，到 SonarScanner 上传，到 CE 计算入库。任何一个环节出错，最终数字就是错的。

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（看着 SonarQube 上的覆盖率数字 87%，困惑地打开一个模块的代码）："大师，这个 `RiskCalculator.java` 显示覆盖率 100%，但我翻遍测试代码都没找到对应的测试类。这个 100% 是哪来的？"

**大师**："小胖，你发现了覆盖率可信度的核心问题。这个 100% 有两种可能性：一是测试在某个你没有检查到的测试类中，二是——覆盖率报告配置错误，它根本不是这个类的真实覆盖率。我们先查一下数据链路。"

**小白**（在 Jenkins 构建日志中搜索）："'JaCoCo XML report generated at /workspace/backend/target/site/jacoco/jacoco.xml'——路径看起来没问题。"

**大师**："路径看起来没问题，但内容可能有问题。让我们用脚本检查 JaCoCo 报告中的源文件路径："

```bash
python3 -c "
import xml.etree.ElementTree as ET
tree = ET.parse('target/site/jacoco/jacoco.xml')
for sf in tree.findall('.//sourcefile'):
    pkg = next(tree.iter('package'))
    print(f'{pkg.get(\"name\")}/{sf.get(\"name\")}')
" | head -5
```

"如果输出是 `/app/src/main/java/com/company/RiskCalculator.java`（绝对路径），而 SonarScanner 执行时的源码路径是 `/home/jenkins/workspace/src/main/java/com/company/RiskCalculator.java`，两个路径对不上——SonarQube 找不到这个文件，覆盖率数据根本没有被匹配。"

**小胖**："那为什么最终显示的数字还是 87%？如果数据没匹配上，不应该是 0% 吗？"

**大师**："这取决于其他文件的覆盖率数据是否正确匹配了。假设：
- Java 后端有 100 个文件，其中 80 个文件（包括被正确测试的）覆盖率路径匹配正确
- 20 个文件路径不匹配，SonarQube 不统计它们的覆盖率
- 最终覆盖率 = (80 个文件的覆盖行) / (80 个文件的总行数) × 100%

是的——路径不匹配的文件被'静默排除'了。SonarQube 不会告诉你'有 20 个文件的覆盖率被忽略了'，它只是把不匹配的文件从统计中移除了。这就是覆盖率数据失真的最常见原因。"

**小白**："那怎么保证路径一致？"

**大师**："三种方法（按推荐度排序）：

1. **在容器内完成构建+扫描**（不分离）：Maven 构建 → JaCoCo 生成报告 → SonarScanner 上传，全都发生在同一个 Docker 容器内。路径天然一致。
2. **统一 CI 工作目录**：Jenkins 和 Docker 挂载使用相同的绝对路径，如 `/home/jenkins/workspace`。
3. **使用 `sonar.coverage.jacoco.xmlReportPaths` 的路径修正功能**：某些版本的 SonarScanner 支持 `sonar.coverage.jacoco.reportPaths` + `sonar.coverage.jacoco.sourcePath` 进行路径映射。"

## 3. 项目实战

### 3.1 环境准备

- 一个配置了 JaCoCo 的 Java 项目
- SonarQube 实例

### 3.2 分步实现

**步骤 1：检查覆盖率数据完整度**

编写数据校验脚本 `validate-coverage.py`：

```python
#!/usr/bin/env python3
"""覆盖率数据链路完整性检查"""

import xml.etree.ElementTree as ET
import os
import sys

def check_jacoco_report(xml_path, source_root):
    """检查 JaCoCo 报告中的源文件是否在源码目录中存在"""
    tree = ET.parse(xml_path)
    root = tree.getroot()

    issues = []
    for pkg in root.findall('.//package'):
        pkg_name = pkg.get('name', '').replace('.', '/')
        for sf in pkg.findall('sourcefile'):
            sf_name = sf.get('name')
            expected_path = os.path.join(source_root, pkg_name, sf_name)

            if os.path.exists(expected_path):
                issues.append(f"✅ {sf_name}")
            else:
                # 检查报告中的路径格式
                counters = sf.findall('counter')
                for c in counters:
                    covered = int(c.get('covered', 0))
                    if covered > 0:
                        issues.append(
                            f"❌ MISSING: {sf_name} (有 {covered} 行覆盖数据但文件不存在: {expected_path})")

    print(f"检查 {xml_path}...")
    for issue in issues:
        print(f"  {issue}")
    print(f"总计: {len(issues)} 个文件\n")

if __name__ == "__main__":
    check_jacoco_report("target/site/jacoco/jacoco.xml", "src/main/java")
```

**步骤 2：修复路径不一致**

**(a) 在 Dockerfile 中统一路径**：

```dockerfile
FROM maven:3.9-eclipse-temurin-17

# 使用固定的工作目录
WORKDIR /workspace
COPY . .

# 在容器内完成全流程
RUN mvn clean verify sonar:sonar \
    -Dsonar.host.url=$SONAR_HOST_URL \
    -Dsonar.token=$SONAR_TOKEN
```

**(b) 在 JaCoCo 中强制使用相对路径**：

在 `pom.xml` 中配置：

```xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <configuration>
        <outputDirectory>${project.basedir}/target/jacoco-report</outputDirectory>
    </configuration>
</plugin>
```

然后配置 SonarQube 扫描：

```properties
sonar.coverage.jacoco.xmlReportPaths=target/jacoco-report/jacoco.xml
```

**步骤 3：多模块覆盖率聚合**

对于 Maven 多模块项目，确保所有模块的报告都被收集：

```xml
<!-- 父 pom.xml -->
<build>
    <plugins>
        <plugin>
            <groupId>org.jacoco</groupId>
            <artifactId>jacoco-maven-plugin</artifactId>
            <executions>
                <execution>
                    <id>report-aggregate</id>
                    <phase>verify</phase>
                    <goals>
                        <goal>report-aggregate</goal>
                    </goals>
                    <configuration>
                        <outputDirectory>${project.basedir}/target/aggregate-jacoco</outputDirectory>
                    </configuration>
                </execution>
            </executions>
        </plugin>
    </plugins>
</build>
```

```properties
# sonar-project.properties
sonar.coverage.jacoco.xmlReportPaths=target/aggregate-jacoco/jacoco.xml
```

**步骤 4：覆盖率数据可信度评级**

建立覆盖率数据质量检查清单：

```bash
#!/bin/bash
# coverage-trust-check.sh
echo "=== 覆盖率数据可信度检查 ==="

# 1. 检查 JaCoCo XML 是否存在且非空
JACOCO_XML="target/site/jacoco/jacoco.xml"
if [ ! -s "$JACOCO_XML" ]; then
    echo "❌ 严重：Jacoco XML 文件不存在或为空"
    exit 1
fi

# 2. 检查有覆盖率>0的类
HAS_DATA=$(python3 -c "
import xml.etree.ElementTree as ET
tree = ET.parse('$JACOCO_XML')
total_covered = sum(int(c.get('covered',0))
    for c in tree.findall('.//counter[@type=\"LINE\"]'))
print(total_covered)
")
if [ "$HAS_DATA" -eq "0" ]; then
    echo "❌ 严重：所有类的覆盖率都是 0"
fi

# 3. 检查源文件路径
echo "源文件路径（前 5 个）："
python3 -c "
import xml.etree.ElementTree as ET
tree = ET.parse('$JACOCO_XML')
for i, pkg in enumerate(tree.findall('.//package')):
    for sf in pkg.findall('sourcefile')[:1]:
        print(f'  {pkg.get(\"name\")}/{sf.get(\"name\")}')
    if i >= 4: break
"

# 4. 检查 SonarQube 导入后与 JaCoCo 的一致性
echo ""
echo "SonarQube API 覆盖率 vs JaCoCo 报告覆盖率："
COVERAGE_API=$(curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/measures/component?component=com.company:my-service&metricKeys=coverage,lines_to_cover")
echo "$COVERAGE_API" | python3 -c "
import sys, json
for m in json.load(sys.stdin)['component']['measures']:
    print(f'  {m[\"metric\"]}: {m.get(\"value\",\"N/A\")}')"
```

## 4. 项目总结

### 4.1 覆盖率数据质量检查清单

| 检查项 | 操作 | 频率 |
|--------|------|------|
| JaCoCo 报告非空 | `test -s jacoco.xml` | 每次构建 |
| 源文件路径一致性 | 对比 JaCoCo 报告路径与源码路径 | 每次构建 |
| 覆盖率 > 0% 的文件占比 | 统计覆盖率 > 0% 的文件数 | 每周 |
| 多模块覆盖率聚合完整性 | 确认所有模块都有数据 | 每次扫描 |
| SonarQube 导入 vs JaCoCo 差异 | 对比 API 和本地报告 | 每次扫描 |

### 4.2 注意事项

1. **Docker 内构建时路径是覆盖率失真的头号杀手**：优先选择"容器内扫描"而非"宿主机扫描容器内生成的报告"。
2. **exclusion 配置在 JaCoCo 和 SonarQube 中必须一致**：如果 JaCoCo 排除了某些类但 SonarQube 没有排除，会导致分母变大覆盖率偏低。
3. **多行 lambda 的行覆盖率计数可能不准确**：JaCoCo 对 lambda 表达式的行覆盖有时不精确，这是已知限制。

### 4.3 思考题

1. 如果 SonarQube 显示覆盖率 80%，但检查 JaCoCo 报告发现实际上覆盖了 95% 的代码行——是什么导致了这 15% 的差异？
2. 集成测试的覆盖率是否应该纳入 SonarQube 统计？为什么？

> **答案提示**：第1题差异可能来自 exclude 配置不一致、路径不匹配导致部分文件被排除、或 SonarQube 的 CE 计算逻辑与 JaCoCo 的 counter 算法不同。第2题集成测试覆盖率建议分开统计——SonarQube 统一展示（用标签区分），不混入单元测试覆盖率。
