# 第15章：日常运维与常见扫描失败排查

## 1. 项目背景

**业务场景**：某公司的 CI 流水线每天早上 9 点集中触发 30 个项目的 SonarQube 扫描。三天前的早晨，运维收到开发团队的投诉："SonarQube 网页打不开了！" 登录服务器后发现，SonarQube 进程虽然还在运行，但 Elasticsearch 已经因为磁盘满而进入只读模式——所有新扫描的报告都无法写入。

排查后发现：SonarQube 的日志文件在 `/opt/sonarqube/logs/` 目录下没有配置日志轮转，`sonar.log` 已经膨胀到 12GB，撑爆了磁盘。而运维团队从未关注过 SonarQube 的磁盘使用情况，因为在他们的监控面板里，SonarQube 不属于"基础设施"范畴。

更糟糕的是，扫描失败的开发者只能看到 CI 上的 "Sonar Scanner returned non-zero exit code" 提示——没有详细的错误原因，不知道是网络问题、内存问题、还是代码问题。开发者的第一反应是"SonarQube 又挂了"，而不是"我要修代码"。

**痛点放大**：

- **日志混乱**：SonarQube 有 4 个日志文件（web.log、ce.log、es.log、sonar.log），但大多数运维人员不知道每个日志文件记录什么内容。
- **CE 队列黑洞**：Compute Engine 任务积压时，新任务的等待时间可能超过 15 分钟，但没有人监控队列深度。
- **扫描失败原因不透明**：Scanner 的退出码和信息不够直观，开发者缺乏自助排障能力。
- **ES 运维知识缺失**：Elasticsearch 内嵌在 SonarQube 中，但运维团队的 ES 知识为零。

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（早上 9:05，发现 SonarQube 页面白屏，慌张地拉大师看）："大师！SonarQube 挂了！我一早上提交的 PR 全卡在 Quality Gate 上了！"

**大师**（淡定地打开终端）："别慌，先做三件事：看进程、看磁盘、看日志。"

```bash
docker compose ps
df -h
docker compose logs sonarqube --tail 50
```

**小胖**："……磁盘使用率 100%？！什么东西占满了？"

**大师**："看日志目录。SonarQube 的 `logs/` 目录默认不轮转，长时间运行后 `sonar.log` 和 `access.log` 可能大到惊人。"

```bash
du -sh logs/*
```

**小胖**："`sonar.log` 12GB？！为什么会这么大？"

**大师**："因为 SonarQube 的日志级别被设成了 DEBUG——通常是有人在排查问题时改了配置，调试完忘了改回来。对于日常运维，日志级别应该是 INFO。"

**小白**："我注意到 SonarQube 有四个日志文件，每个记录什么内容？"

**大师**：

| 日志文件 | 内容 | 常见排查用途 |
|---------|------|-------------|
| `sonar.log` | 应用启动日志、整体状态 | 服务无法启动、插件加载失败 |
| `web.log` | Web 服务器日志、API 请求 | 认证失败、401/403 错误 |
| `ce.log` | Compute Engine 任务日志 | 扫描成功但数据未入库 |
| `es.log` | Elasticsearch 运行日志 | 搜索不可用、索引损坏 |

"日常运维中，`ce.log` 是最重要的——90% 的'扫描成功但页面没数据'问题都是 CE 任务失败了。`es.log` 是最致命的——ES 一旦进入只读模式，整个平台就不可用了。"

**小胖**："还有什么常见故障？我好提前写个 SOP。"

**大师**："最常见的 5 类故障：

1. **内存不足（OOM）**：ES 的 JVM 堆设置过大，超过机器可用内存。症状：容器反复重启，日志中有 `OutOfMemoryError`。
2. **数据库连接失败**：PostgreSQL 故障或网络不通。症状：登录时提示'无法连接到数据库'。
3. **ES 启动失败**：`vm.max_map_count` 不足。症状：日志中报 `max virtual memory areas vm.max_map_count [65530] is too low`。
4. **磁盘满**：日志未轮转或历史数据过多。症状：页面操作报 500 错误，ES 进入只读。
5. **CE 队列积压**：大量项目同时扫描，CE Worker 处理不过来。症状：扫描成功但 Web UI 长时间不更新。"

## 3. 项目实战

### 3.1 环境准备

- 正在运行的 SonarQube 实例
- 具有服务器管理权限（或 Docker 控制权限）

### 3.2 分步实现

**步骤 1：建立健康检查 SOP**

编写 `check-sonarqube.sh`：

```bash
#!/bin/bash
echo "=== SonarQube 健康检查 $(date) ==="

# 1. 容器状态
echo -e "\n[1/6] 容器状态："
docker compose ps --format "table {{.Name}}\t{{.Status}}\t{{.Health}}"

# 2. 系统健康 API
echo -e "\n[2/6] 系统健康："
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/system/health" \
  | python3 -m json.tool

# 3. 磁盘使用
echo -e "\n[3/6] 磁盘使用："
df -h /var/lib/docker | tail -1

# 4. CE 队列
echo -e "\n[4/6] CE 任务队列："
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/ce/activity?statuses=PENDING,IN_PROGRESS&ps=1" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(f'待处理: {d[\"paging\"][\"total\"]}')" 2>/dev/null || echo "API 不可达"

# 5. 日志最近 Error
echo -e "\n[5/6] 最近错误（web.log 最后 10 行 ERROR）："
docker compose exec -T sonarqube \
  tail -100 /opt/sonarqube/logs/web.log 2>/dev/null \
  | grep ERROR | tail -5 || echo "N/A"

# 6. ES 健康
echo -e "\n[6/6] ES 集群健康："
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/system/info" \
  | python3 -c "import sys,json; print(json.load(sys.stdin).get('System',{}).get('Elasticsearch',{}).get('State','N/A'))" 2>/dev/null

echo -e "\n=== 检查完成 ==="
```

**步骤 2：排查"扫描成功但 Web UI 无数据"**

```bash
# 1. 查看 CE 任务状态
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/ce/activity?component=com.example:order-service&ps=5" \
  | python3 -c "
import sys, json
data = json.load(sys.stdin)
for task in data['tasks']:
    print(f\"ID: {task['id']}\")
    print(f\"Status: {task['status']}\")  # SUCCESS / FAILED
    print(f\"Type: {task['type']}\")       # REPORT
    print(f\"Submitted: {task['submittedAt']}\")
    if task['status'] == 'FAILED':
        print(f\"Error: {task.get('errorMessage', 'N/A')}\")
    print('---')
"

# 2. 查看具体失败原因
# 记录下 Task ID（如 AXeqW7sdf123）
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/ce/task?id=AXeqW7sdf123" \
  | python3 -m json.tool | grep -A5 errorMessage

# 3. 查看 ce.log
docker compose exec sonarqube \
  tail -500 /opt/sonarqube/logs/ce.log | grep -i error
```

**步骤 3：常见故障排查速查表**

**(a) 内存不足（OOM）**

```bash
# 查看 SonarQube 容器内存
docker stats sonarqube --no-stream

# 查看 OOM 日志
docker compose logs sonarqube | grep -i "outofmemory\|java.lang.OutOfMemoryError"
```

解决：增加 Docker 容器的内存限制，或在 `sonar.properties` / 环境变量中降低 JVM 堆大小：

```yaml
environment:
  SONAR_SEARCH_JAVAOPTS: "-Xms512m -Xmx1g"
  SONAR_WEB_JAVAOPTS: "-Xms256m -Xmx512m"
  SONAR_CE_JAVAOPTS: "-Xms256m -Xmx512m"
```

**(b) ES 磁盘满（Read-Only）**

```bash
# 检查 ES 的只读状态
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/system/info" \
  | python3 -c "import sys,json; print(json.load(sys.stdin).get('System',{}).get('Elasticsearch',{}))"

# 如果是只读模式，需要手动将索引设为可写
# ⚠ 但这只是一个临时修复，核心是要释放磁盘空间
```

解决核心步骤：
1. 清理旧日志：`find /opt/sonarqube/logs -type f -name "*.log" -mtime +7 -delete`
2. 检查数据库大小：`docker compose exec postgres psql -U sonar -c "SELECT pg_size_pretty(pg_database_size('sonar'))"`
3. 清理历史数据：SonarQube 的 Housekeeping（自动清理历史数据）默认每日执行，确认其未暂停

**(c) CE 队列积压**

```bash
# 查看积压情况
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/ce/activity?statuses=PENDING" \
  | python3 -c "import sys,json; print(f'积压任务: {json.load(sys.stdin)[\"paging\"][\"total\"]}')"

# 查看失败的 CE 任务
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/ce/activity?statuses=FAILED&ps=5" \
  | python3 -m json.tool
```

解决：
- 增加 CE Worker 线程：`SONAR_CE_JAVAOPTS=-XX:ActiveProcessorCount=4`
- 检查是否有大量失败后又重试的任务，手动取消它们
- 确保数据库连接池足够大

**步骤 4：建立日志轮转**

在 `docker-compose.yml` 中添加日志轮转：

```yaml
services:
  sonarqube:
    logging:
      driver: "json-file"
      options:
        max-size: "10m"
        max-file: "3"
```

对于 SonarQube 内部日志，可以通过设置配置管理：

```bash
# 在 sonar.properties 中或通过环境变量设置
SONAR_LOG_ROLLING_POLICY=size:10MB
SONAR_LOG_MAXFILES=7
```

### 3.3 Scanner 端排障

**Scanner 常见错误解读**：

| 错误信息 | 根因 | 解决 |
|---------|------|------|
| `Connection refused` | SonarQube URL 不可达或端口错误 | 检查 `sonar.host.url` 和网络配置 |
| `Not authorized (401)` | Token 无效或权限不足 | 重新生成 Token，确认权限 |
| `Project not found` | projectKey 不存在或项目被删除 | 确认 projectKey 拼写 |
| `Java heap space` | Scanner 内存不足 | 设置 `SONAR_SCANNER_OPTS=-Xmx2g` |
| `No files nor directories` | `sonar.sources` 路径错误 | 确认相对路径或绝对路径 |
| `Unsupported Java version` | JDK 版本不兼容 | 参考 SonarQube 版本要求 |

**Scanner 调试模式**：

```bash
sonar-scanner -X -Dsonar.verbose=true
```

关键输出行：
- `Load quality profiles` → 规则加载情况
- `Sensor JavaSensor` → 实际执行的分析器
- `Analysis report uploaded in XXXms` → 上传成功
- `EXECUTION SUCCESS` or `EXECUTION FAILURE` → 最终结果

### 3.4 验证

```bash
# 验证 Scanner 连接
sonar-scanner -Dsonar.host.url=http://localhost:9000 \
  -Dsonar.token=squ_xxx \
  -Dsonar.projectKey=test-connection \
  -Dsonar.sources=. \
  -Dsonar.scm.disabled=true \
  -X 2>&1 | grep -E "SUCCESS|FAILURE"
```

## 4. 项目总结

### 4.1 日常运维检查清单

| 频率 | 检查项 | 命令/工具 |
|------|--------|----------|
| 每日 | 系统健康 API | `api/system/health` |
| 每日 | CE 任务状态 | `api/ce/activity` |
| 每日 | 磁盘使用率 | `df -h` |
| 每周 | 日志大小检查 | `du -sh logs/` |
| 每周 | 数据库连接池 | `api/system/info` → Database |
| 每月 | ES 索引健康 | `api/system/info` → Elasticsearch |

### 4.2 适用场景

- **生产环境**：所有生产级 SonarQube 部署都需要本章的运维知识
- **CI/CD 支持**：当流水线中扫描失败时，需要快速定位是代码问题还是平台问题
- **容量规划**：通过监控 CE 队列深度和磁盘趋势，预判扩容时机

**不适用场景**：
- 个人学习环境（对稳定性和故障恢复要求较低）
- 每天扫描 < 5 次的小团队（运维压力小）

### 4.3 注意事项

1. **不要在生产环境随意修改 SonarQube 自身配置**：任何 `sonar.properties` 的修改都需要先测试、后上线。
2. **备份数据库是最重要的**：只要 PostgeSQL 数据在，SonarQube 的所有质量数据都可以恢复（ES 数据可以从数据库重建）。
3. **CE Worker 数量不是越多越好**：过多的 Worker 会增加数据库和 ES 的并发压力。

### 4.4 常见踩坑经验

**故障 1：SonarQube 启动成功，但一直显示 "Embedded Elasticsearch is not operational"**

根因：ES 没有分配足够的堆内存（默认要求至少 512MB），或者 `vm.max_map_count` 未设置。在 `sonar.properties` 中增加 `sonar.search.javaOpts=-Xmx1g`。

**故障 2：升级 SonarQube 后，所有旧插件报错**

根因：插件版本与 SonarQube 主版本不兼容。升级前必须检查所有已安装插件的兼容性，并在升级后先禁用不兼容的插件、再安装兼容版。

**故障 3：大量 CE 任务堆积，但不是因为并发过高**

根因：某些项目的 CE 任务处理超时（可能是代码量太大或数据库查询慢）。查看 CE 任务日志，对大型项目考虑拆分为多个子项目或优化数据库索引。

### 4.5 思考题

1. 如果一个 500+ 项目的 SonarQube 实例每天 CE 队列积压 200+ 任务，你如何从 CE Worker、数据库连接池、ES 索引三个维度做容量扩容？
2. SonarQube 的数据库备份只需要备份 PostgreSQL 吗？ES 数据需要单独备份吗？为什么？

> **答案提示**：第1题容量规划详见第26章和第38章。第2题 ES 数据是数据库的派生数据，备份数据库即可恢复（但重建 ES 索引需要时间，生产环境可能需要直接备份 ES 以减少恢复时长）。
