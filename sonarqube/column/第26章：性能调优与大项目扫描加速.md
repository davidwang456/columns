# 第26章：性能调优与大项目扫描加速

## 1. 项目背景

**业务场景**：某银行的核心交易系统是一个百万行级别的 Java 项目，首次 SonarQube 扫描耗时 52 分钟。团队在 CI Pipeline 中配置了每次 PR 都触发扫描——结果每天有 30+ 个 PR，累计扫描时间超过 26 小时/天。CI 队列堵塞，开发者抱怨"提交代码后等扫描结果比写代码的时间还长"。

同时，SonarQube 服务器端也出现了性能问题：CE 队列高峰期积压 80+ 任务，单个任务处理有时超过 30 分钟。数据库连接池频繁耗尽（当前配置 10 个连接），导致 CE 任务因等待数据库连接而超时。

**痛点放大**：

- **扫描时间 = 开发等待时间**：52 分钟的扫描让开发者转去做其他事，上下文切换成本高
- **全量扫描浪费资源**：即使只改了 1 行代码，也需要分析所有 100 万行代码
- **服务端资源瓶颈**：数据库连接池、CE Worker 数量、JVM 堆配置都可能成为瓶颈
- **插件和规则膨胀**：780 条激活规则中，有 200 条是"可能有用但没人看"的规则
- **覆盖率报告解析是隐藏的性能杀手**：JaCoCo 生成的 exec 文件如果包含大量类，解析时间可能比 Java 分析还长
- **Scanners 缓存未生效**：SonarScanner 支持分析缓存（analysis cache），但很多人不会配置——导致每次扫描都全量分析，浪费大量 CPU
- **CI 中的重复开销**：每个 PR 扫描都拉取完整规则集、都执行完整的文件索引——这些步骤可以缓存
- **内存配置的"Goldilocks 问题"**：Scanner 内存太小 → GC 频繁；内存太大 → GC 停顿时间长。找到"刚刚好"的配置是艺术

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（提交 PR 后等了 52 分钟才看到扫描结果）："大师！扫了 52 分钟！如果结果是通过还好，不通过的话修了还得再等 52 分钟——这谁受得了？"

**大师**："52 分钟确实太久了。我们来拆解一下这段时间花在哪了——Scanner 端和 Server 端各占多少。"

**小白**（查看扫描日志）："从日志看：文件索引 2 分钟、规则下载 5 分钟、Java 分析 35 分钟、报告生成 3 分钟、上传和等待 CE 处理 7 分钟。瓶颈在 Java 分析这个环节。"

**大师**："35 分钟的分析时间，我们可以从四个维度优化：

1. **排除不需要分析的文件**：生成的代码、第三方 jar 的拆包源码、测试辅助类——这些不应该浪费分析时间。
2. **精简规则**：700+ 条规则中，有多少是你真的需要的？关闭不需要的规则可以减少 20-30% 的分析时间。
3. **增加 Scanner 内存**：默认 Scanner 的 JVM 堆可能不够大，GC 频繁导致性能下降。
4. **服务端资源扩容**：CE Worker 线程数、数据库连接池、ES 内存。"

**小胖**："那具体到配置上，我应该怎么调？"

**大师**："看实战部分。但先记一个原则：**优化 Scanner 端比优化 Server 端效果更明显**。Scanner 端的优化能让每个扫描都受益（因为每次扫描都要在 Scanner 端分析），而 Server 端的优化只在 CE 处理时生效。"

**小胖**："我们试了一下排除文件——排除了 target/ 目录后，分析时间从 52 分钟降到了 45 分钟——但为什么只少了 7 分钟？target/ 目录里有几千个生成的类。我预期应该减少 20 分钟才对。"

**大师**："因为 SonarQube 的 Java 分析器本来就会跳过没有源码的文件——target/ 目录中的 .class 文件不会被分析，只有 target/generated-sources/ 中的 .java 文件才会。而且 SonarQube 默认排除 `**/target/classes/**` 等路径。

真正需要排除的'意外开销大户'：

1. **Protobuf 生成的代码**：一个大的 proto 文件可能生成 5000+ 行 Java 代码，其中包含大量 getter/setter——这些会被风格类规则重复检查。
2. **JAXB / XJC 生成的代码**：类似 Protobuf，生成的代码结构重复性高。
3. **Lombok 生成的代码**：SonarQube 在字节码层面分析时会重复检查 Lombok 生成的方法。
4. **第三方库的源码被打包进项目**：有些人把依赖的源码直接复制到 `src/main/java/lib/`。

优化建议——不要乱排，用数据驱动：

```bash
# 找出代码行数最多的文件（它们通常是分析耗时的大头）
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/measures/component_tree?component=com.company:core-system&metricKeys=ncloc&ps=20&s=metric&asc=false" \
  | python3 -c "
import sys, json
for c in json.load(sys.stdin).get('components', []):
    ncloc = next((m['value'] for m in c.get('measures',[]) if m['metric']=='ncloc'), '?')
    print(f'{ncloc:>6s} lines  {c[\"path\"]}')" | head -20
```

超过 2000 行但非核心业务代码的文件 → 优先排除。"

**小白**："SonarScanner 的缓存到底怎么配置？文档里提到了 `sonar.cpd.cross_project` 和增量分析，但具体怎么用？"

**大师**："确切地说，SonarQube 社区版不支持真正的增量分析（只分析变更的文件），那是 DataCenter 版的功能。但你可以通过以下方式减少重复工作：

**1. 利用 Scanner 的本地缓存（sonar.scanner.cache）**：

```properties
# sonar-project.properties
sonar.scanner.skipJreProvisioning=true
sonar.scanner.cache.enable=true
sonar.scanner.cache.home=.sonar-cache
```

这会缓存规则和插件的下载——避免每次扫描都从 Server 下载几十 MB 的规则 JAR。

**2. Maven/Gradle 缓存机制**：

```xml
<!-- Maven: 使用增量编译 + Sonar 缓存 -->
<properties>
    <sonar.java.enablePreview>false</sonar.java.enablePreview>
</properties>
```

```bash
# 在 CI 中配置缓存路径
# GitLab CI 示例
cache:
  paths:
    - .sonar/cache
    - .m2/repository
```

**3. 针对 PR 扫描的最有效优化**：

```yaml
# GitLab CI: 只有在 MR 时才触发完整扫描
sonarqube-scan:
  script:
    - |
      if [ -n "$CI_MERGE_REQUEST_IID" ]; then
        # MR 扫描：使用 sonar.branch 参数，分析变更范围
        mvn sonar:sonar \
          -Dsonar.branch.name=$CI_MERGE_REQUEST_SOURCE_BRANCH_NAME \
          -Dsonar.branch.target=$CI_MERGE_REQUEST_TARGET_BRANCH_NAME
      else
        # 主干扫描：完整分析
        mvn sonar:sonar
      fi
```

这些优化的关键是：**减少 Scanner 端的不必要工作，而非减少分析本身的时间。**"

**小胖**："另一个问题——我们的 SonarQube Server 端，CE 队列经常积压 80+ 个任务。我看监控显示 CPU 使用率只有 30%——说明 CE 并不是被 CPU 瓶颈了。那瓶颈在哪？"

**大师**："CPU 30% 但队列积压，说明瓶颈在 I/O 等待——最可能是数据库连接池或者磁盘 I/O。排查步骤：

```bash
# 1. 检查数据库连接池使用情况
# 在 PostgreSQL 中查询活跃连接数
docker compose exec postgres psql -U sonar -c \
  "SELECT count(*) as active_connections FROM pg_stat_activity WHERE state = 'active';"

# 如果 active_connections >= maxActive * 0.8 → 连接池瓶颈

# 2. 检查 PostgreSQL 的慢查询
docker compose exec postgres psql -U sonar -c \
  "SELECT query, calls, mean_exec_time, total_exec_time 
   FROM pg_stat_statements 
   WHERE query LIKE '%ce_activity%' OR query LIKE '%project_measures%'
   ORDER BY mean_exec_time DESC LIMIT 5;"

# 3. 检查磁盘 I/O 等待
docker compose exec sonarqube iostat -x 1 3

# 4. 检查 ES 写入延迟
curl -s "http://localhost:9001/_nodes/stats/indices" \
  | python3 -c "import sys,json;d=json.load(sys.stdin);print('ES写入延迟:', ...)"
```

常见优化方向：
- 数据库连接池 → `sonar.jdbc.maxActive=50`（默认 60 可能太大，反而让 PG 压力大）
- CE Worker → `sonar.ce.workerCount=2`（不是越多越好，2-3 是甜点）
- PG 配置 → `shared_buffers=1GB`（默认很小），`work_mem=64MB`
- 磁盘 → 如果用的是机械硬盘，升级到 SSD 会有数量级提升"

## 3. 项目实战

### 3.1 环境准备

- 大型 Java 项目（代码量 ≥ 50 万行）
- SonarQube 管理员权限
- 可修改项目配置和 Scanner 参数

### 3.2 分步实现

**步骤 1：精准排除不需要扫描的文件**

```properties
# sonar-project.properties
# 排除生成的代码
sonar.exclusions=\
  **/target/generated-sources/**,\
  **/build/generated/**,\
  **/target/generated-test-sources/**,\
  **/*.pb.java,\
  **/*.proto.java,\
  **/Q*.java

# 排除第三方代码（如 Ant 任务的依赖）
sonar.exclusions=\
  **/lib/**,\
  **/vendor/**,\
  **/thirdparty/**

# 排除测试资源
sonar.exclusions=\
  **/src/test/resources/**,\
  **/*TestData*.java
```

通过 API 批量获取耗时最长的文件：

```bash
# 获取分析耗时长的文件列表
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/measures/component_tree?component=com.company:core-system&metricKeys=ncloc,complexity&s=metricPeriod&metricSortFilter=complexity&asc=false&ps=10" \
  | python3 -c "
import sys, json
data = json.load(sys.stdin)
for comp in data.get('components', [])[:5]:
    measures = {m['metric']: m.get('value','?') for m in comp.get('measures', [])}
    print(f'{comp[\"path\"]:60s} ncloc={measures.get(\"ncloc\",\"?\"):6s} complexity={measures.get(\"complexity\",\"?\"):6s}')"
```

**步骤 2：精简规则集**

进入 Quality Profile → 查看已激活规则数：

```bash
# 统计当前激活的规则数
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/rules/search?languages=java&activation=true&qprofile=AXeqW7sdf123&ps=1" \
  | python3 -c "import sys,json;print(f'激活规则: {json.load(sys.stdin)[\"total\"]}')"
```

精简策略：
- 停用所有 Info 级别的规则（这些通常是最低优先级的建议）
- 停用所有由 `common-*` 仓库提供的通用规则（如果有其他检查工具覆盖）
- 停用所有仅在特定框架场景下有效的规则（如果项目不使用该框架）

```bash
# 批量获取 Info 级别规则并停用
INFO_RULES=$(curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/rules/search?languages=java&activation=true&qprofile=AXeqW7sdf123&severities=INFO&ps=200" \
  | python3 -c "import sys,json;[print(r['key']) for r in json.load(sys.stdin).get('rules',[])]")

for rule in $INFO_RULES; do
  curl -s -X POST -u admin:Sonar@2024Admin \
    "http://localhost:9000/api/qualityprofiles/deactivate_rule" \
    -d "qualityProfile=Team Java Profile v1" \
    -d "rule=$rule" > /dev/null
done
```

**预期效果**：关闭 50-80 条低价值规则，扫描时间可降低 15-25%。

**步骤 3：调整 Scanner JVM 参数**

**步骤 5：建立扫描性能基线**

```bash
#!/bin/bash
# scan-benchmark.sh - 扫描性能基线测试
echo "=== SonarQube 扫描性能基线 ==="

START=$(date +%s)

sonar-scanner -Dsonar.token=$SONAR_TOKEN 2>&1 | tee scan.log

END=$(date +%s)
DURATION=$((END - START))

echo ""
echo "扫描总耗时: ${DURATION}s ($((DURATION / 60))m $((DURATION % 60))s)"

# 提取各阶段耗时
echo "阶段耗时分析："
grep -oP '.*done.*\| time=\K[0-9]+' scan.log | while read ms; do
    echo "  $((ms / 1000))s"
done
```

**步骤 7：SonarQube Server 端数据库优化**

```sql
-- PostgreSQL 性能优化
-- 查看慢查询（执行前需启用 pg_stat_statements 扩展）
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;

-- 分析表以确保执行计划最优
ANALYZE issues;
ANALYZE project_measures;
ANALYZE components;

-- 查看 Buffer Cache 命中率（应 > 99%）
SELECT 
    sum(heap_blks_read) as heap_read,
    sum(heap_blks_hit) as heap_hit,
    sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read)) * 100 as ratio
FROM pg_statio_user_tables;

-- 如果命中率 < 99%，增加 shared_buffers
-- postgresql.conf: shared_buffers = 1GB (推荐物理内存的 25%)
```

**步骤 9：建立性能监控面板**

```bash
#!/bin/bash
# perf-dashboard.sh - 每分钟采集关键性能指标

SONAR_URL="http://localhost:9000"
TOKEN="squ_xxx"

while true; do
    TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')
    
    # CE Pending 任务数
    PENDING=$(curl -s -u "$TOKEN:" \
        "$SONAR_URL/api/ce/activity?statuses=PENDING&ps=1" \
        | python3 -c "import sys,json;print(json.load(sys.stdin)['paging']['total'])")
    
    # CE In-Progress 任务数
    IN_PROGRESS=$(curl -s -u "$TOKEN:" \
        "$SONAR_URL/api/ce/activity?statuses=IN_PROGRESS&ps=1" \
        | python3 -c "import sys,json;print(json.load(sys.stdin)['paging']['total'])")
    
    # Web Server 健康状态
    HEALTH=$(curl -s -o /dev/null -w "%{http_code}" "$SONAR_URL/api/system/health")
    
    echo "[$TIMESTAMP] Pending=$PENDING | InProgress=$IN_PROGRESS | Health=$HEALTH"
    
    # 告警逻辑
    if [ "$PENDING" -gt 20 ]; then
        echo "⚠️ CE 队列积压！Pending=$PENDING"
    fi
    
    sleep 60
done
```

## 4. 项目总结

### 4.1 性能调优排查路径图

```
扫描很慢 → 看日志哪个阶段耗时最长
│
├── 文件索引慢 → sonar.exclusions 排除无关目录
├── 规则执行慢 → 精简 Quality Profile 规则数
├── 报告上传慢 → 检查网络带宽和 Server 负载
└── CE 处理慢 → 扩容 CE Worker / 数据库连接池 / 检查 ES 状态
```

### 4.2 注意事项

1. **排除文件要先测试**：过度排除可能遗漏真正的质量问题。先在一个测试项目上验证，确认无误再推广。
2. **不要无限增加 CE Worker**：Worker 数超过 CPU 核心数后会造成线程争抢，反而降速。
3. **数据库连接池不宜过大**：连接池过大会让 PostgreSQL 产生大量 idle 连接，占用内存。
4. **规则精简是最被低估的优化**：关闭 50 条无用规则的收益往往大于加 4GB Scanner 内存——因为规则执行是 O(n) × 规则数的关系。
5. **性能优化是循环迭代的**：排除文件 → 重新扫描测基线 → 精简规则 → 再测 → 调整内存 → 再测——每次只做一个变更，记录效果，才能定位真正的瓶颈。
6. **CI 中的扫描时间包含网络开销**：如果 SonarQube Server 在另一台机器上，检查网络延迟（报告上传可能占扫描总时间的 15-25%）。

### 4.3 扫描耗时成分分析模型

一次完整扫描的时间组成：

```
T_total = T_file_index + T_analysis + T_report_gen + T_upload + T_ce_process

T_file_index     → sonar.exclusions 优化
T_analysis       → 规则精简 + Scanner JVM 优化（最大优化空间）
T_report_gen     → 基本不可优化（报告序列化）
T_upload         → 网络带宽 + report 大小（排除生成代码可减小 report）
T_ce_process     → CE Worker + 数据库 + ES（Server 端优化）
```

对于大型 Java 项目（50 万+ 行），T_analysis 通常占总时间的 60-70%——这是优化的优先级最高点。

### 4.4 不同规模项目的推荐配置

| 项目规模 | 代码行数 | Scanner 内存 | CE Worker | DB 连接池 | ES 内存 | 预期扫描时间 |
|---------|---------|-------------|-----------|----------|---------|------------|
| 小型 | < 5 万行 | 512MB | 1 | 10 | 512MB | < 2 min |
| 中型 | 5-20 万行 | 1GB | 1-2 | 20 | 1GB | 2-8 min |
| 大型 | 20-50 万行 | 2GB | 2 | 30 | 2GB | 8-20 min |
| 超大 | 50-200 万行 | 4GB | 2-3 | 50 | 4GB | 20-45 min |
| 巨型 | > 200 万行 | 8GB+ | 3-4 | 60 | 8GB | > 45 min |

### 4.5 Scanner JVM GC 调优参考

```bash
# 不同规模项目的 SONAR_SCANNER_OPTS 推荐
# 小型项目
export SONAR_SCANNER_OPTS="-Xmx512m -Xms256m"

# 中型项目
export SONAR_SCANNER_OPTS="-Xmx2g -Xms1g -XX:+UseG1GC -XX:MaxGCPauseMillis=200"

# 大型项目
export SONAR_SCANNER_OPTS="-Xmx4g -Xms2g -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -XX:+ParallelRefProcEnabled"

# 巨型项目
export SONAR_SCANNER_OPTS="-Xmx8g -Xms4g -XX:+UseG1GC -XX:MaxGCPauseMillis=200 -XX:G1HeapRegionSize=16m -XX:+ParallelRefProcEnabled -XX:+UseStringDeduplication"
```

1. 如果扫描时间是 28 分钟，但其中 25 分钟是 Java Sensor 的执行时间——你还能从哪里优化？
2. 使用 `sonar.inclusions`（仅包含指定文件）而不是 `sonar.exclusions`（排除指定文件），哪种方式更安全？为什么？
3. CE 队列经常积压，你在监控面板上看到 Pending 任务数在高峰时段飙升至 120。但增加 CE Worker 数从 2 → 4 后，瓶颈并未改善——你下一步排查什么？
4. 你有一个微服务架构的项目群（20 个独立服务），应该拆分成 20 个 SonarQube 项目分别扫描，还是作为一个聚合项目扫描？从性能角度分析。

> **答案提示**：第1题 Java Sensor 的耗时瓶颈可能在 AST 解析或字节码分析，检查是否使用了 `sonar.java.binaries` 指向的编译产物是否包含完整符号信息。第2题 `sonar.inclusions` 更安全——新增文件不会自动被包含，避免"忘记排除新目录"。第3题排查数据库连接池耗尽或磁盘 I/O——增加 Worker 数只是在数据库前面加了更多排队者，没解决后端瓶颈。第4题拆分为 20 个项目更好——每个项目独立分析、可并行执行、CE 任务粒度更小、失败隔离；聚合扫描一次失败全部丢失且 CE 处理超时风险更高。

---

> **推广计划提示**：性能优化不是一次性的活动——建议在每次 SonarQube 版本升级或规则变更后重新执行一次扫描性能基线测试，记录在团队的 SonarQube Wiki 上。
