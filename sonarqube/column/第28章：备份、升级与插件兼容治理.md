# 第28章：备份、升级与插件兼容治理

## 1. 项目背景

**业务场景**：某公司的 SonarQube 从 9.9 LTS 版本运行了 18 个月，积累了 200+ 项目的历史数据。安全团队要求升级到 10.7（修复了 3 个已知漏洞），但运维团队非常犹豫——上次从 8.9 升级到 9.9 时，数据库迁移失败导致服务中断了 6 小时。更糟糕的是，升级后 12 个已安装的插件中有 5 个不兼容，导致 SonarQube 启动即崩溃。

这次事故的根因分析揭示了三个致命问题：没有升级前的备份、没有插件兼容性检查、没有灰度验证流程。运维 Leader 决定建立一套完整的 SonarQube 运维 SOP——覆盖备份策略、升级路径、插件管理和回滚预案。

**痛点放大**：

- **升级路径复杂**：不能跨版本跳跃（9.9 → 10.7 必须经过 10.0），社区版无自动升级
- **插件兼容性黑盒**：升级前不知道哪些插件会"爆炸"，升级后才发现
- **备份无验证**：有备份文件，但从未演练过恢复——备份可能根本不可用
- **数据库迁移失败回滚难**：升级过程中数据库 Schema 被修改后，回滚需要数据库级回退
- **多版本并存的灰度验证困难**：无法在同一套基础设施上同时运行新旧版本进行对比测试，这使得验证升级效果变得非常被动
- **ES 索引重建的时间黑洞**：升级后如果 ES 索引不兼容需要重建，对于 200+ 项目的实例，重建可能需要数小时，这段时间内搜索功能不可用
- **插件版本矩阵的复杂性**：SonarQube 核心升级后，每个插件可能有自己的兼容版本范围，部分插件可能已经被 abandon，需要寻找替代品
- **安全补丁升级的紧迫性与流程冗长的矛盾**：安全团队要求的 CVE 修复需要在 72 小时内完成，但完整的升级+验证流程需要 4-6 小时，如果发现不兼容需要回滚则时间更长

**升级风险评估矩阵**：

| 风险类型 | 发生概率 | 影响范围 | 缓解措施 |
|---------|---------|---------|---------|
| 插件不兼容导致启动失败 | 高（40%） | 全部用户不可用 | 预升级兼容性检查 + 灰度验证 |
| 数据库迁移超时 | 中（15%） | 全部用户不可用 | 增加迁移超时配置 + 预热数据库 |
| ES 索引损坏 | 中（10%） | 搜索功能不可用 | 重建索引脚本预准备 |
| 性能回退 | 高（35%） | 用户体验下降 | 升级前后性能基线对比 |
| 数据丢失 | 低（2%） | 极严重 | 全量备份 + 恢复演练 |
| API 行为变更导致集成工具异常 | 中（20%） | CI/CD 流水线失败 | API 兼容性测试套件 |

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（看到 SonarQube 10.7 的 Release Note 里列了一长串 Bug Fixes，心痒痒）："大师，我们该升级 SonarQube 了。10.7 修复了好多问题——尤其是那个 CE 偶尔卡死的 Bug，太影响体验了！"

**大师**："升级欲望是好的，但先问自己三个问题：备份了吗？检查过兼容性吗？有回滚预案吗？"

**小白**："官方说不能跳版本升级——9.9 → 10.7 必须先经过 10.0。为什么？"

**大师**："SonarQube 的数据库迁移脚本是串行的——每个版本有自己的数据库 Schema 变更。10.0 的迁移脚本（可能新增表、修改字段）必须在 10.7 的脚本之前执行。如果跳过 10.0，10.7 的迁移脚本可能找不到 10.0 创建的字段/表——直接失败。

标准升级路径：9.9 LTS → 10.0 → 10.7。

建议：先在测试环境练一遍，记录每个步骤的耗时和遇到的问题。这份记录就是你的生产升级 Runbook。"

**小胖**："上次我们升级到 10.0 后，发现社区版的一个插件（sonar-dependency-check）不兼容——启动直接报错，SonarQube 无法正常运行。这种情况还能抢救吗？"

**大师**："这种情况说明你升级前缺少了一个关键步骤——**兼容性矩阵检查**。每个 SonarQube 版本都有一个'插件兼容性矩阵'（Plugin Version Matrix），你可以在 SonarSource 的官方文档或 Marketplace 中找到。

预防方案是在升级前做一次'模拟升级'：

```bash
# 在独立测试环境中验证插件兼容性
# 1. 复制生产环境配置到测试环境
# 2. 执行升级
# 3. 启动后检查启动日志中的插件加载错误
docker compose logs sonarqube | grep -i "plugin.*fail\|plugin.*error\|Plugin.*not"

# 4. 如果有插件失败，检查是否需要更新插件版本
# 5. 记录不兼容的插件清单，制定替代方案
```

如果已经启动失败，紧急救援方案：

```bash
# 紧急救援：禁用所有第三方插件启动
docker compose exec sonarqube bash

# 在容器内
cd /opt/sonarqube/extensions/plugins/
mkdir -p /tmp/disabled-plugins-backup
# 只保留核心插件（sonar-java-plugin, sonar-javascript-plugin 等）
# 移动第三方插件到备份目录
for jar in sonar-dependency-*.jar sonar-custom-*.jar; do
    mv "$jar" /tmp/disabled-plugins-backup/ 2>/dev/null
done
exit

docker compose restart sonarqube
# 启动成功后，逐个恢复第三方插件，每次恢复一个后重启
```

关键教训：**升级前必须对每个第三方插件做兼容性验证**。如果一个插件已经 2 年没更新且不在 Marketplace 中——做好它可能不兼容并需要移除的准备。"

**小白**："我们用的是 PostgreSQL 15 + SonarQube 9.9。升级 SonarQube 时，PostgreSQL 版本也需要跟着升级吗？升级顺序是怎样的？"

**大师**："好问题。SonarQube 和 PostgreSQL 的版本兼容性也是升级路径的一部分。以 SonarQube 10.x 为例：

| SonarQube 版本 | 最低 PostgreSQL 版本 | 推荐 PostgreSQL 版本 |
|---------------|---------------------|---------------------|
| 9.9 LTS | PG 11 | PG 13/14 |
| 10.0 | PG 13 | PG 14/15 |
| 10.7 | PG 13 | PG 15/16 |

升级顺序遵循'基础设施先行'原则：

```
正确顺序：PostgreSQL 先升级 → SonarQube 再升级
错误顺序：SonarQube 先升级 → 发现 PG 版本不兼容 → 回滚
```

安全的双升级流程：

```bash
# 步骤 1：备份数据库（最重要！）
docker compose exec postgres pg_dump -U sonar -Fc sonar > pre_all_upgrade.dump

# 步骤 2：升级 PostgreSQL
docker compose stop sonarqube  # 只停 SonarQube，保留 PG
docker compose down postgres    # 停止旧版 PG
# 修改 docker-compose.yml 中 PG 版本
sed -i 's/postgres:14/postgres:16/' docker-compose.yml
docker compose up -d postgres
# 等待 PG 启动完成
sleep 15

# 步骤 3：验证 PostgreSQL 数据完整
docker compose exec postgres psql -U sonar -c "SELECT count(*) FROM projects;"

# 步骤 4：升级 SonarQube
sed -i 's/sonarqube:9.9.6-community/sonarqube:10.7.0-community/' docker-compose.yml
docker compose up -d sonarqube

# 步骤 5：观察迁移日志
docker compose logs -f sonarqube
```

如果中间 PG 升级失败——用备份恢复到原始状态，重新来过。"

**小胖**："还有一个实际场景——我们每周日凌晨有 2 小时维护窗口。如果升级在 2 小时内完不成怎么办？"

**大师**："2 小时窗口内完成升级，需要做'时间预算'：

| 步骤 | 操作 | 预计耗时 | 弹性时间 |
|------|------|---------|---------|
| 备份 | pg_dump + tar | 30 min | +10 min |
| 停止服务 | docker compose stop | 2 min | +2 min |
| 镜像切换 | docker compose up -d | 5 min | +5 min |
| DB 迁移 | 自动执行 | 10-30 min | +15 min |
| 启动验证 | 健康检查 + 数据核对 | 10 min | +10 min |
| 插件处理 | 逐个启用验证 | 20 min | +20 min |
| **总计** | | **77-97 min** | **+62 min** |

如果总计接近 2 小时——你应该：
1. **提前执行备份**：维护窗口前 30 分钟开始备份，这样备份不计入窗口时间
2. **预下载镜像**：提前 `docker pull` 新版本镜像
3. **演练过 2 次以上**：演练时记录每步的实际耗时（而非估计）
4. **准备回滚脚本**：如果 1 小时 45 分还未完成 → 立即触发回滚

```bash
# 维护窗口定时回滚脚本
#!/bin/bash
# rollback-if-timeout.sh
DEADLINE=$(date -d "+105 minutes" +%s)  # 1h45m 硬截止
CURRENT=$(date +%s)

if [ $CURRENT -gt $DEADLINE ]; then
    echo "超过时间限制，触发回滚！"
    # 回滚到之前的镜像版本
    sed -i 's/sonarqube:10.7.0-community/sonarqube:9.9.6-community/' docker-compose.yml
    docker compose down
    docker compose up -d postgres
    docker compose exec -T postgres pg_restore -U sonar -d sonar < pre_upgrade.dump
    docker compose up -d sonarqube
    # 发送回滚通知
    echo "SonarQube 升级超时，已回滚至 9.9.6" | mail -s "升级回滚通知" ops@company.com
fi
```

核心原则：**宁可推迟升级，不要冒险上线。有 2 小时窗口就给升级预留 1 小时 30 分，剩下 30 分钟是回滚时间。**"

### 3.2 分步实现

**步骤 1：数据库备份**

```bash

docker compose exec postgres pg_dump -U sonar \
  --schema=public --exclude-table=ce_activity --exclude-table=ce_queue \
```

**步骤 3：预升级检查清单**

```bash
#!/bin/bash
# pre-upgrade-check.sh

echo "=== SonarQube 升级前检查 ==="

# 1. 检查当前版本
CURRENT_VERSION=$(curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/system/info" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['System']['Version'])")

# 2. 检查数据库版本

# 3. 列出所有已安装插件

curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/plugins/installed" \

    print(f\"  {p['name']:30s} v{p.get('version','?')}\")"

# 5. 验证备份
echo "最近备份:"
ls -lt sonar_backup_*.dump | head -1
```

**步骤 4：执行升级（9.9 → 10.0 → 10.7）**

```bash
# 1. 停止 SonarQube（保留 PostgreSQL）
docker compose stop sonarqube

# 2. 备份数据库
docker compose exec postgres pg_dump -U sonar -Fc sonar > pre_upgrade_$(date +%Y%m%d).dump

# 3. 切换到 10.0 版本
sed -i 's/sonarqube:9.9.6-community/sonarqube:10.0.0-community/' docker-compose.yml
docker compose up -d sonarqube

# 4. 等待数据库迁移完成（观察日志）
docker compose logs -f sonarqube | grep -E "Database migration|SonarQube is operational"

# 5. 验证升级成功
curl -u admin:admin http://localhost:9000/api/system/info | python3 -c "import sys,json;print(json.load(sys.stdin)['System']['Version'])"
# 预期输出: 10.0.0.x

# 6. 重复步骤 1-5，升级到 10.7
sed -i 's/sonarqube:10.0.0-community/sonarqube:10.7.0-community/' docker-compose.yml
docker compose up -d sonarqube
```

**步骤 5：插件兼容性治理**

升级前：检查每个插件的兼容目标版本。

```bash
# 导出插件清单
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/plugins/installed" \

for p in json.load(sys.stdin)['plugins']:
```

升级后：如果插件不兼容导致启动失败，禁用所有非核心插件：

```bash

# 手动移除不兼容插件的 JAR 文件

mv sonar-xxxx-plugin-*.jar /tmp/disabled-plugins/  # 移动不兼容的插件

docker compose restart sonarqube
```

**步骤 7：自动化备份脚本（含保留策略）**

```bash
#!/bin/bash
# auto-backup.sh - 定期备份 + 自动清理旧备份

BACKUP_DIR="/opt/backups/sonarqube"
RETENTION_DAYS=30  # 保留 30 天

TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/sonar_db_${TIMESTAMP}.dump"
DATA_BACKUP="${BACKUP_DIR}/sonar_data_${TIMESTAMP}.tar.gz"

# 数据库备份
docker compose exec -T postgres pg_dump -U sonar -Fc sonar > "$BACKUP_FILE"

# 文件备份
tar -czf "$DATA_BACKUP" data/ extensions/ conf/

# 验证备份完整性
if [ -s "$BACKUP_FILE" ] && [ -s "$DATA_BACKUP" ]; then
    echo "[$(date)] 备份成功: $BACKUP_FILE ($(du -h $BACKUP_FILE | cut -f1))"
else
    echo "[$(date)] 备份失败！"
    exit 1
fi

# 清理超过保留期的旧备份
find "$BACKUP_DIR" -name "sonar_db_*.dump" -mtime +$RETENTION_DAYS -delete
find "$BACKUP_DIR" -name "sonar_data_*.tar.gz" -mtime +$RETENTION_DAYS -delete

# 记录备份日志
echo "[$(date)] $(ls -lh $BACKUP_FILE | awk '{print $5}')" >> "${BACKUP_DIR}/backup.log"
```

**步骤 9：插件兼容性自动化检查工具**

```bash
#!/bin/bash
# plugin-compat-check.sh - 检查插件与目标版本的兼容性

TARGET_VERSION="10.7"
SONAR_URL="http://localhost:9000"

echo "=== 检查插件兼容性 (目标版本: $TARGET_VERSION) ==="

# 获取当前已安装插件
curl -s -u admin:Sonar@2024Admin "$SONAR_URL/api/plugins/installed" \
  | python3 -c "
import sys, json

plugins = json.load(sys.stdin)['plugins']
print(f'{\"插件名称\":<35s} {\"当前版本\":<12s} {\"风险等级\":<10s}')
print('-' * 60)

# 已知不兼容插件清单（需要维护）
INCOMPATIBLE = {
    'sonar-dependency-check': '4.0.0 (需更新至 4.2.0+)',
    'sonar-apex-plugin': '已停止维护，无 10.x 兼容版本',
}

for p in plugins:
    key = p['key']
    name = p['name'][:32]
    version = p.get('version', '?')
    
    if key in INCOMPATIBLE:
        risk = '🔴 高风险'
        note = INCOMPATIBLE[key]
    elif 'sonar-' in key and key not in ('sonar-java-plugin', 'sonar-javascript-plugin'):
        risk = '🟡 需验证'
        note = ''
    else:
        risk = '🟢 兼容'
        note = ''
    
    print(f'{name:<35s} {version:<12s} {risk:<10s} {note}')
"
```

## 4. 项目总结

### 4.1 升级 SOP 模板

| 步骤 | 操作 | 预计耗时 |
|------|------|---------|
| 1. 通知 | 发送升级通知和维护窗口 | 提前 1 周 |
| 2. 测试环境 | 在测试环境完整演练 | 2 小时 |
| 3. 生产备份 | 数据库 + 文件系统全量备份 | 30 分钟 |
| 4. 停止服务 | 停止 SonarQube（保留 DB） | 5 分钟 |
| 5. 执行升级 | 切换镜像版本 + 启动 | 15 分钟 |
| 6. 验证 | 健康检查 + 数据完整性 | 15 分钟 |
| 7. 监控 | 持续观察 24 小时 | 1 天 |

### 4.2 注意事项

1. **备份必须经过恢复验证**：一个从未恢复过的备份不是真正的备份。
2. **数据库备份 + 文件备份缺一不可**：只有数据库备份可以恢复质量数据，但插件和 ES 索引也需要文件备份（或接受重建 ES 的时间成本）。
3. **升级前先禁用所有第三方插件**：启动成功后逐个启用，隔离问题。
4. **LTS 到 LTS 是最安全的升级策略**：9.9 LTS → 10.7（非 LTS）→ 等待下一个 LTS；如果不需要新功能，直接 9.9 LTS → 下一个 LTS。
5. **维护窗口预留回滚时间**：2 小时窗口 → 90 分钟升级 + 30 分钟回滚兜底。
6. **升级后的"静默观察期"**：升级完成后不要立即宣布"成功"，持续观察 24 小时，监控 CE 队列、ES 状态、API 响应时间。
7. **版本控制备份文件**：docker-compose.yml 和 sonar.properties 纳入版本管理，每次变更都有 commit 记录。

### 4.3 插件生命周期管理策略

| 插件类型 | 管理策略 | 升级频率 | 示例 |
|---------|---------|---------|------|
| SonarSource 官方语言插件 | 随 SQ 版本升级自动更新 | 每版本 | sonar-java, sonar-js |
| 社区维护的热门插件 | 升级前 1 周检查新版本 | 按需 | sonar-dependency-check |
| 内部自定义插件 | 升级前重新编译测试 | 每次 SQ 主版本升级 | company-java-rules |
| 废弃/无人维护插件 | 寻找替代品或移除 | 不升级 | 部分老插件 |

### 4.4 备份策略分级

| 级别 | 备份频率 | 保留策略 | 恢复演练 | 适用场景 |
|------|---------|---------|---------|---------|
| 基础 | 每周全量备份 | 保留 4 周 | 每季度 1 次 | 个人/小团队 |
| 标准 | 每日增量 + 每周全量 | 增量 7 天 + 全量 90 天 | 每月 1 次 | 中型组织（50-200 项目）|
| 高级 | 每日全量 + WAL 归档 | 全量 1 年 + WAL 7 天 | 每月 1 次 | 大型企业（200+ 项目）|

### 4.5 升级失败的应急处理决策树

```
升级启动后 → 观察日志
    │
    ├── 启动成功 → 健康检查通过 → 🟢 升级成功
    │
    ├── 启动失败（插件不兼容）
    │   └── 禁用第三方插件 → 重启 → 逐个恢复
    │       ├── 成功 → 🟡 部分插件需更新
    │       └── 失败 → 🔴 回滚
    │
    ├── 启动成功但 DB 迁移失败
    │   └── 检查迁移日志 → 
    │       ├── 超时 → 增加 sonar.ce.ceTimeoutInMs → 重试
    │       └── 脚本错误 → 🔴 回滚（恢复数据库备份）
    │
    └── 启动成功但 ES 状态异常
        └── 尝试重建索引 → 
            ├── 成功 → 🟡 索引重建完成
            └── 失败 → 🔴 回滚
```

### 4.6 升级后验证清单

```bash
#!/bin/bash
# post-upgrade-verify.sh - 升级后验证脚本

SONAR_URL="http://localhost:9000"
TOKEN="squ_xxx"

echo "=== SonarQube 升级后验证 ==="

# 1. 系统健康检查
HEALTH=$(curl -s -o /dev/null -w "%{http_code}" "$SONAR_URL/api/system/health")
echo "1. 健康检查: $([ "$HEALTH" == "200" ] && echo '✅' || echo '❌') (HTTP $HEALTH)"

# 2. 版本确认
VERSION=$(curl -s -u "$TOKEN:" "$SONAR_URL/api/system/info" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['System']['Version'])")
echo "2. 当前版本: $VERSION"

# 3. 项目数据完整性
PROJ_COUNT=$(curl -s -u "$TOKEN:" "$SONAR_URL/api/projects/search?ps=1" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['paging']['total'])")
echo "3. 项目总数: $PROJ_COUNT"

# 4. CE 状态
CE_STATUS=$(curl -s -u "$TOKEN:" "$SONAR_URL/api/ce/activity?ps=1" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['tasks'][0]['status'] if json.load(sys.stdin)['tasks'] else 'EMPTY')")
echo "4. 最新 CE 任务: $CE_STATUS"

# 5. Quality Gate
GATE_COUNT=$(curl -s -u "$TOKEN:" "$SONAR_URL/api/qualitygates/list" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['qualitygates'].__len__())")
echo "5. Quality Gate 数量: $GATE_COUNT"

# 6. ES 状态
ES_STATUS=$(curl -s -u "$TOKEN:" "$SONAR_URL/api/system/info" \
  | python3 -c "import sys,json;print(json.load(sys.stdin)['System'].get('Elasticsearch',{}).get('State','UNKNOWN'))")
echo "6. Elasticsearch: $ES_STATUS"

echo ""
echo "=== 验证完成 ==="
```

### 4.7 思考题

1. SonarQube 升级后数据库迁移失败，你发现数据库中已经新增了部分表，回滚应该怎么做——直接恢复数据库备份，还是回退 Schema 变更？
2. 如果你只能在维护窗口（周日凌晨 2-4 点）内完成升级，如何确保 2 小时内能完成并验证？
3. 你的 SonarQube 实例使用内嵌 ES，但 ES 索引占用了 80GB 磁盘空间，磁盘剩余不足 10%。你需要在升级前解决这个问题——有哪些策略？
4. 升级后发现一个关键的自定义插件完全不兼容（源码已丢失，只有 JAR 包），你有哪些方案？

> **答案提示**：第1题直接恢复数据库备份（pg_restore 覆盖现有数据）是最安全的方式。手动回退 Schema 极易出错。第2题：提前在测试环境完整演练并记录每步耗时，把升级脚本预先准备好，执行时不需要临时决策。第3题策略：① 清理历史分析数据（Administration → Housekeeping）；② 删除不活跃的已归档项目；③ 对 ES 做 force merge。第4题方案：① 反编译 JAR 查看源码并基于新 API 重写；② 联系原始作者；③ 评估是否可以改用内建规则替代该自定义功能。

---

> **推广计划提示**：升级不是运维一个人的事——需要在升级前 1 周通过邮件/群聊通知所有使用 SonarQube 的团队（告知维护窗口）。升级完成后发送"升级完成通知"，包含新版本的功能变化和已知问题。备份脚本应纳入 Cron/Task Scheduler 自动执行，而非手动触发——"自动化是运维的第一原则"。本章涉及的脚本集合建议整理为一个 `sonarqube-ops-toolkit` 仓库，供全团队使用。
