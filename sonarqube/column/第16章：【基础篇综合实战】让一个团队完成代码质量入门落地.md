# 第16章：【基础篇综合实战】让一个团队完成代码质量入门落地

## 1. 项目背景

**业务场景**：某 20 人规模的创业公司，核心产品是 SaaS 报销管理系统（Spring Boot + Vue.js）。团队在过去半年快速迭代中积压了大量技术债务：线上平均每月 3-4 次事故，测试覆盖率不可见（实际上仅 12%），代码审查完全依赖人工（审查时间平均 4 小时/次），安全漏洞无人关注。CTO 决定引入 SonarQube 作为统一质量平台，要求在一个 Sprint（2 周）内完成从部署到门禁的完整落地。

这是一个典型的企业质量治理起点场景：团队有动力但没经验，有需求但没标准。基础篇 15 章的所有知识需要在一个综合实战中融会贯通——从环境部署、项目扫描、规则定制、门禁设计、IDE 集成到日常排障，形成完整的质量治理闭环。

**痛点放大**：

- 两周时间紧任务重：部署、扫描、规则评审、门禁配置、团队培训——每一项都充满未知
- 多技术栈挑战：后端 Java + 前端 TypeScript，需要两套扫描方案
- 团队抵触情绪：部分资深开发者认为"工具会降低开发效率"
- 历史代码包袱：已有 5 万行代码、500+ 历史 Issue，如何让门禁可落地而不引发团队反感？

### 1.3 成功标准 vs 失败标准

在 Kickoff 会议上，CTO 与团队明确约定了什么算"成功"、什么算"失败"，避免项目结束时出现"你说成功了、我说没效果"的分歧。

| 维度 | ✅ 成功标准（2周后必须达到） | ❌ 失败标准（出现任一即宣告失败） |
|------|----------------------------|--------------------------------|
| 门禁覆盖率 | New Code Coverage ≥ 70%，CI 自动检查 | 门禁从未成功触发过阻断 |
| Issue 处理 | Blocker Bug = 0，Critical Vulnerability = 0 | 新增 Blocker 未被发现，线上出现可预防事故 |
| 团队覆盖 | 100% 开发者安装 SonarLint Connected Mode | 超过 3 人拒绝安装且无正当理由 |
| 扫描频率 | 每次提交触发自动扫描 | 扫描仅手动执行，CI 中未集成 |
| 流程闭环 | 开发者能在 1 小时内处理 CI 门禁失败 | 门禁失败后无人知道如何排查和修复 |
| 认知提升 | 团队问卷得分从 0 分提升到 ≥ 60 分（满分 100） | 两周后仍有人问"SonarQube 是干什么的" |
| 业务影响 | 两周内零新增线上事故 | 推广期间引发超过 1 次因工具配置导致的线上事故 |
| 文档沉淀 | Wiki 上有排障 SOP + Issue 修复指南 | 知识仅存于个人脑中，人员离职即丢失 |

> **关键理念**：成功标准不是"修完 500 个历史 Issue"——那需要 3-6 个月。两周的目标是建立**质量防护网**：新代码不变坏，团队养成质量意识，工具链融入日常工作流。把"止血"和"还债"分阶段看待，是落地成功的第一原则。

**小胖**（Day 2 下午，盯着 Dashboard 上 503 个 Issue，额头冒汗）："大师，首次扫描结果出来了——503 个 Issue！我刚在群里发截图，老张直接回了一句'这么多问题，两周怎么可能修得完，这工具是不是太严格了？'。团队士气明显受挫，好几个人说'这玩意儿就是来找茬的'。怎么办？"

**大师**："小胖，你踩了落地过程中最常见的一个坑——**把未经筛选的全部 Issue 暴露给整个团队**。这就像医生做完体检，把 200 页报告直接甩给病人，病人不恐慌才怪。

正确的做法是 **Issue 分级展示策略**，分三步走：

**第一步：Dashboard 只展示 New Code 指标**

在 SonarQube 项目首页，默认展示的是全部代码（Overall Code）指标。你要做的第一件事是——把所有人的视角强制切换到 'New Code' 视图。历史代码的 503 个 Issue 不是不修，而是不能作为当前 Sprint 的主要目标。操作：

```bash
# 在 SonarQube 项目设置中将默认视图改为 New Code
# Quality Profile 也设置为仅对 New Code 强制
```

**第二步：给 Issue 贴标签，分三批处理**

把 503 个 Issue 按优先级分为三批：

- **🔥 红色批次（必须本周修，约 15 个）**：Blocker Bug、Critical Vulnerability、安全相关。这是底线，不修就是不负责。
- **🟡 黄色批次（下个 Sprint 修，约 100 个）**：Major Bug、高重复率代码块。分配给具体开发者，纳入 Sprint Backlog。
- **🟢 绿色批次（长期还债，约 388 个）**：Minor Code Smell、命名规范类。标记为 'Won't Fix' 或 'Confirmed'（改天再修），不在本次 Sprint 追踪。

用 API 批量打标签：

```bash
#!/bin/bash
# 批量将 Minor Code Smell 标记为 Won't Fix
PROJECT="com.company:expense-backend"
TOKEN="squ_xxx"
SONAR_URL="http://localhost:9000"

# 查询所有 Minor Code Smell（按严重性排序后取前 200 个作为示例）
curl -s -u $TOKEN: \
  "$SONAR_URL/api/issues/search?projectKeys=$PROJECT&severities=MINOR&types=CODE_SMELL&ps=200" \
  | python3 -c "
import sys, json
data = json.load(sys.stdin)
for issue in data.get('issues', []):
    print(issue['key'])
" | while read issue_key; do
  curl -s -X POST -u $TOKEN: \
    "$SONAR_URL/api/issues/do_transition" \
    -d "issue=$issue_key" \
    -d "transition=wontfix"
  echo "  → $issue_key marked as Won't Fix"
done
```

**第三步：在团队沟通中主动引导叙事**

不要在群里甩链接说'大家看，我们代码有 503 个问题'。换一种说法：'好消息！SonarQube 帮我们找到了 **8 个可能导致线上事故的高危问题**，这周我们集中精力把这 8 个修掉，就能大幅降低下个月的事故概率。其余 495 个是历史Code Smell，不紧急，会逐步分配。'

把威胁转化为收益——工具不是来找茬的，是来帮我们**找到最危险的问题优先处理**的。"

**小白**（Day 2 晚上，Jenkins 日志刷了一屏红色）："大师救命！前端覆盖率一直显示 0%，但我本地 `npx jest --coverage` 是正常的，`coverage/lcov.info` 文件也确实生成了。我在 `sonar-project.properties` 里配了 `sonar.javascript.lcov.reportPaths=coverage/lcov.info`，但 SonarQube 就是读不到。搞了 3 个小时了！"

**大师**："小白，你遇到的问题 80% 是 **路径问题**，尤其是在 Docker 或 Jenkins 构建环境里。覆盖率 0% 排查有五个关键检查点，按顺序逐一排查，5 分钟就能定位：

**检查点 1：lcov.info 是否真的生成在正确位置**

```bash
# 在 sonar-scanner 执行前，确认文件存在且非空
ls -la coverage/lcov.info
wc -l coverage/lcov.info         # 应该 > 0 行
head -5 coverage/lcov.info       # 应看到 SF:src/... 开头的记录
```

常见陷阱：Jest 默认输出到 `coverage/lcov-report/` 而非 `coverage/`，或者 Jest 配置了 `coverageDirectory` 改变了输出路径。

**检查点 2：sonar-project.properties 中的路径是相对于项目根目录的**

```properties
# ✅ 正确：相对于 sonar-project.properties 所在目录
sonar.javascript.lcov.reportPaths=coverage/lcov.info

# ❌ 错误：用了绝对路径（在 Docker 中会找不到）
sonar.javascript.lcov.reportPaths=/home/jenkins/workspace/coverage/lcov.info
```

**检查点 3：sonar-scanner 的执行目录必须正确**

```bash
# ❌ 错误：从上级目录执行，相对路径失效
cd /tmp && sonar-scanner -Dproject.settings=/path/to/sonar-project.properties

# ✅ 正确：在项目根目录执行
cd /path/to/project && npx jest --coverage && sonar-scanner
```

**检查点 4：验证 SonarQube 是否真的读取了覆盖率文件**

```bash
# 查看 SonarQube CE 任务日志（在 SonarQube Web UI → Administration → Background Tasks）
# 搜索日志中的 'Sensor JavaScript/TypeScript Coverage'
# 应该看到: '1 file(s) found' 或类似的覆盖率文件导入信息
# 如果看到 'No coverage report files found' → 路径一定有问题
```

**检查点 5：Docker/Jenkins 环境中的特殊问题**

如果你在 Docker 容器中运行 sonar-scanner，而 Jest 在宿主机上运行，路径会完全不同：

```bash
# Jenkins Pipeline 中的完整正确写法
stage('Frontend Analysis') {
    steps {
        sh '''
            cd frontend
            npm ci
            npx jest --coverage --coverageReporters=lcov
            # 确认生成
            ls -la coverage/lcov.info
            # 在正确的目录下执行 sonar-scanner
            sonar-scanner \
              -Dsonar.projectKey=com.company:expense-frontend \
              -Dsonar.host.url=$SONAR_HOST_URL \
              -Dsonar.token=$SONAR_TOKEN \
              -Dsonar.javascript.lcov.reportPaths=coverage/lcov.info
        '''
    }
}
```

最后一招——如果以上都确认正确但覆盖率还是 0%，检查 `lcov.info` 中的文件路径是否与 SonarQube 认知的源码路径一致：

```bash
# lcov.info 中的路径示例：
# SF:src/components/ExpenseForm.vue
# 但 SonarQube 的项目根目录下确实有 src/components/ExpenseForm.vue 吗？
grep 'SF:' coverage/lcov.info | head -5
```

如果 SF 路径包含 `/home/jenkins/workspace/` 这样的绝对路径前缀，需要用 `genhtml` 或 `sed` 做路径替换。"

**小胖**（Day 11，门禁正式上线第三天）："大师，出事了！今天早上小李提交了一个 PR，里面有明显的 SQL 注入漏洞——但他把 Issue 标记成了 False Positive，门禁就通过了。我发现的时候代码已经合入 dev 分支了。这样下去，门禁不就形同虚设了吗？"

**大师**："你发现了一个关键治理问题——**False Positive 审计机制**。工具再好，如果开发者可以随意把安全问题标记为 False Positive，那门禁就是纸老虎。但也不能完全不让人标记——有些确实是误报（如 Lombok 生成的代码、框架自动代码）。

解决这个问题的正确方式不是禁止标记 False Positive，而是建立**三级审计机制**：

**审计机制设计：**

| 级别 | 条件 | 处理方式 |
|------|------|---------|
| 🟢 自动批准 | Code Smell (Minor) + 标记者备注长度 ≥ 20 字 | 自动通过 |
| 🟡 同行审批 | Bug (Major) 或 Vulnerability + 任何标记 | 需要同一 Team 的另一位开发者 Approve |
| 🔴 强制审批 | Blocker / Critical + 任何标记 | 需要质量负责人（小胖）或架构师 Approve |

**技术实现——用 SonarQube Webhook + 自建审计看板：**

```bash
#!/bin/bash
# False Positive 审计脚本
# 用途：每日运行，检测最近 24 小时内被标记为 False Positive 的可疑 Issue
# 放到 cron 中：0 9 * * * /path/to/fp_audit.sh

SONAR_URL="http://localhost:9000"
TOKEN="squ_xxx"
AUDIT_LOG="/var/log/sonarqube/fp_audit_$(date +%Y%m%d).json"

echo "=== False Positive Audit Report - $(date) ==="

# 1. 查询所有标记为 False Positive 的 Issue（最近 7 天）
curl -s -u $TOKEN: \
  "$SONAR_URL/api/issues/search?resolutions=FALSE-POSITIVE&createdAfter=$(date -d '7 days ago' +%Y-%m-%d)&ps=500" \
  | python3 -c "
import sys, json
from collections import Counter

data = json.load(sys.stdin)
issues = data.get('issues', [])

if not issues:
    print('✅ 最近 7 天无 False Positive 标记')
    sys.exit(0)
```

**Spring Boot 项目规则调整重点**：

| 规则 | 操作 | 原因 |
|------|------|------|
| java:S1186（空方法体） | 停用 | Spring Data JPA Repository 的空方法由框架实现 |
| java:S1118（工具类构造器） | 调参 | 添加 `@UtilityClass` 注解（Lombok）为异常 |
| java:S1181（捕获 Throwable） | 保留 | 全局异常处理中也应捕获 Exception |
| java:S1192（字符串字面量重复） | 调参 | 阈值从 3 调为 5，减少命名中的误报 |

**前端项目规则调整重点**：

| 规则 | 操作 | 原因 |
|------|------|------|
| javascript:S1523（动态代码执行） | 保留 | 禁止 `eval()` 使用 |
| javascript:S5693（文件上传大小限制） | 保留 | 防止 DoS |
| typescript:S4325（any 类型） | 调参 | 从 Error 降为 Warning |

```bash
#!/bin/bash
# frontend-coverage-debug.sh — 前端覆盖率故障诊断脚本
# 使用: ./frontend-coverage-debug.sh [项目根目录]

PROJECT_DIR="${1:-.}"
cd "$PROJECT_DIR" || exit 1

echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"

echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo ""

PASS=0
FAIL=0

check() {
    local desc="$1"
    shift
    if "$@"; then
        echo "  ✅ $desc"
        PASS=$((PASS + 1))
    else
        echo "  ❌ $desc"
        FAIL=$((FAIL + 1))
    fi
}

echo "📁 1. 项目结构检查"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
check "package.json 存在" test -f package.json
check "node_modules 存在" test -d node_modules
check "Jest 已安装" test -f node_modules/.bin/jest
check "sonar-project.properties 存在" test -f sonar-project.properties
echo ""

echo "📝 2. sonar-project.properties 配置验证"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
if [ -f sonar-project.properties ]; then
    check "包含 sonar.javascript.lcov.reportPaths" \
        grep -q "sonar.javascript.lcov.reportPaths" sonar-project.properties
    
    LCOV_PATH=$(grep "sonar.javascript.lcov.reportPaths" sonar-project.properties | cut -d= -f2- | tr -d ' ')
    echo "    配置的路径: $LCOV_PATH"
    check "lcov 路径非空" test -n "$LCOV_PATH"
fi
echo ""

echo "🧪 3. 测试执行与覆盖率生成"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
check "Jest 可执行" npx jest --version > /dev/null 2>&1

# 实际执行测试
echo "    正在运行测试..."
npx jest --coverage --coverageReporters=lcov --silent 2>&1 | tail -5
JEST_EXIT=$?
check "Jest 测试通过 (exit=$JEST_EXIT)" test $JEST_EXIT -eq 0
echo ""

echo "📄 4. lcov.info 文件完整性验证"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
if [ -n "$LCOV_PATH" ]; then
    check "lcov.info 文件存在" test -f "$LCOV_PATH"
    if [ -f "$LCOV_PATH" ]; then
        SIZE=$(wc -c < "$LCOV_PATH")
        LINES=$(wc -l < "$LCOV_PATH")
        SF_COUNT=$(grep -c '^SF:' "$LCOV_PATH" 2>/dev/null || echo 0)
        check "lcov.info 非空 (${SIZE} bytes, ${LINES} 行)" test "$SIZE" -gt 0
        check "包含源文件记录 (SF: ${SF_COUNT} 个文件)" test "$SF_COUNT" -gt 0
        
        echo ""
        echo "    📋 lcov.info 中的源文件路径 (前 10 个):"
        grep '^SF:' "$LCOV_PATH" | head -10 | while read line; do
            echo "      $line"
        done
        
        # 检查路径是否与项目匹配
        echo ""
        FIRST_SF=$(grep '^SF:' "$LCOV_PATH" | head -1 | cut -d: -f2-)
        if echo "$FIRST_SF" | grep -q "^/"; then
            echo "    ⚠️ 发现绝对路径！在 Docker/Jenkins 中可能无法匹配"
        fi
    fi
else
    echo "  ❌ 无法检查（lcov 路径未配置）"
fi
echo ""

echo "📊 5. 诊断总结"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "  通过: $PASS 项"
echo "  失败: $FAIL 项"
echo ""

if [ $FAIL -eq 0 ]; then
    echo "✅ 所有检查通过！如果覆盖率仍为 0%，请检查："
    echo "   1. SonarQube CE 任务日志中是否有 'No coverage report files found'"
    echo "   2. lcov.info 中的路径是否匹配 SonarQube 的源码路径"
    echo "   3. sonar-scanner 是否在正确的目录下执行"
else
    echo "❌ 发现 $FAIL 个问题，请根据以上诊断结果逐一修复。"
```

**步骤 15：自动化环境搭建脚本（Day 0 准备）**

```bash
#!/bin/bash
# setup-sonarqube-env.sh — 一键搭建 SonarQube 完整环境
# 使用: chmod +x setup-sonarqube-env.sh && ./setup-sonarqube-env.sh
# 前置: Docker + Docker Compose 已安装，≥ 8GB 可用内存

set -euo pipefail

echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "  🚀 SonarQube 团队环境一键部署脚本 v1.0"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo ""

# ====== 配置区 ======
SONAR_VERSION="${SONAR_VERSION:-10.4.1-community}"
SONAR_PORT="${SONAR_PORT:-9000}"
ADMIN_PASSWORD="${ADMIN_PASSWORD:-Sonar@2024Admin}"
COMPOSE_DIR="./sonarqube-docker"

# ====== 1. 系统检查 ======
echo "[1/7] 系统环境检查..."
if ! command -v docker &> /dev/null; then
    echo "❌ Docker 未安装，请先安装 Docker"
    exit 1
fi

if ! command -v docker-compose &> /dev/null && ! docker compose version &> /dev/null 2>&1; then
    echo "❌ Docker Compose 未安装"
    exit 1
fi

AVAILABLE_MEM=$(free -m 2>/dev/null | awk '/Mem:/{print $7}' || echo 8192)
if [ "$AVAILABLE_MEM" -lt 4096 ]; then
    echo "⚠️  可用内存不足 4GB (当前 ${AVAILABLE_MEM}MB)，SonarQube 可能无法正常启动"
fi

# 设置内核参数（Linux 需要）
if [ "$(uname)" = "Linux" ]; then
    echo "  设置 vm.max_map_count..."
    sudo sysctl -w vm.max_map_count=262144 > /dev/null 2>&1 || \
        echo "  ⚠️  无法设置 vm.max_map_count，请手动执行: sudo sysctl -w vm.max_map_count=262144"
fi

echo "  ✅ 系统检查通过"
echo ""

# ====== 2. 生成 docker-compose.yml ======
echo "[2/7] 生成 docker-compose.yml..."
mkdir -p "$COMPOSE_DIR"

cat > "$COMPOSE_DIR/docker-compose.yml" << 'DOCKEREOF'
version: "3.8"
services:
  sonarqube:
    image: sonarqube:10.4.1-community
    container_name: sonarqube
    depends_on:
      postgresql:
        condition: service_healthy
    environment:
      SONAR_JDBC_URL: jdbc:postgresql://postgresql:5432/sonarqube
      SONAR_JDBC_USERNAME: sonar
      SONAR_JDBC_PASSWORD: sonar@db2024
      SONAR_ES_BOOTSTRAP_CHECKS_DISABLE: "true"
    volumes:
      - sonarqube_data:/opt/sonarqube/data
      - sonarqube_extensions:/opt/sonarqube/extensions
      - sonarqube_logs:/opt/sonarqube/logs
    ports:
      - "${SONAR_PORT}:9000"
    restart: unless-stopped
    mem_limit: 4096m

  postgresql:
    image: postgres:15-alpine
    container_name: sonarqube-db
    environment:
      POSTGRES_USER: sonar
      POSTGRES_PASSWORD: sonar@db2024
      POSTGRES_DB: sonarqube
    volumes:
      - postgresql_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U sonar -d sonarqube"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

volumes:
  sonarqube_data:
  sonarqube_extensions:
  sonarqube_logs:
  postgresql_data:
DOCKEREOF

echo "  ✅ docker-compose.yml 已生成"
echo ""

# ====== 3. 启动服务 ======
echo "[3/7] 启动 SonarQube 服务（首次启动可能需要 3-5 分钟）..."
cd "$COMPOSE_DIR"
docker compose up -d
echo "  ✅ 容器已启动"
echo ""

# ====== 4. 等待就绪 ======
echo "[4/7] 等待 SonarQube 就绪..."
MAX_WAIT=300
WAITED=0
while [ $WAITED -lt $MAX_WAIT ]; do
    HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" "http://localhost:${SONAR_PORT}/api/system/health" 2>/dev/null || echo "000")
    if [ "$HTTP_CODE" = "200" ]; then
        HEALTH=$(curl -s "http://localhost:${SONAR_PORT}/api/system/health")
        if echo "$HEALTH" | grep -q '"health":"GREEN"'; then
            echo "  ✅ SonarQube 已就绪 (耗时 ${WAITED}s)"
            break
        fi
    fi
    sleep 10
    WAITED=$((WAITED + 10))
    echo "  等待中... (${WAITED}s/${MAX_WAIT}s, HTTP $HTTP_CODE)"
done

if [ $WAITED -ge $MAX_WAIT ]; then
    echo "❌ SonarQube 启动超时，请检查日志: docker compose logs sonarqube"
    exit 1
fi
echo ""

# ====== 5. 修改默认密码 ======
echo "[5/7] 修改默认管理员密码..."
curl -s -X POST -u "admin:admin" \
    "http://localhost:${SONAR_PORT}/api/users/change_password" \
    -d "login=admin" \
    -d "password=${ADMIN_PASSWORD}" \
    -d "previousPassword=admin" > /dev/null
echo "  ✅ 密码已更新"
echo ""

# ====== 6. 生成 API Token ======
echo "[6/7] 生成 API Token..."
TOKEN_RESPONSE=$(curl -s -X POST -u "admin:${ADMIN_PASSWORD}" \
    "http://localhost:${SONAR_PORT}/api/user_tokens/generate" \
    -d "name=ci-scanner-token")

TOKEN=$(echo "$TOKEN_RESPONSE" | python3 -c "import sys,json; print(json.load(sys.stdin).get('token',''))" 2>/dev/null || echo "")
if [ -n "$TOKEN" ]; then
    echo "  ✅ Token 已生成: $TOKEN"
    echo "$TOKEN" > "$COMPOSE_DIR/../sonar-token.txt"
    echo "  📄 Token 已保存到 sonar-token.txt"
else
    echo "  ⚠️  Token 生成失败，请手动通过 Web UI 创建"
fi
echo ""

# ====== 7. 创建默认 Quality Gate ======
echo "[7/7] 创建默认 Quality Gate..."
curl -s -X POST -u "admin:${ADMIN_PASSWORD}" \
    "http://localhost:${SONAR_PORT}/api/qualitygates/create" \
    -d "name=Team Quality Gate" > /dev/null 2>&1

# 添加条件
GATE_ID=$(curl -s -u "admin:${ADMIN_PASSWORD}" \
    "http://localhost:${SONAR_PORT}/api/qualitygates/show?name=Team%20Quality%20Gate" \
    | python3 -c "import sys,json; print(json.load(sys.stdin).get('id',''))" 2>/dev/null)

if [ -n "$GATE_ID" ]; then
    # New Code Coverage < 70% → Fail
    curl -s -X POST -u "admin:${ADMIN_PASSWORD}" \
        "http://localhost:${SONAR_PORT}/api/qualitygates/create_condition" \
        -d "gateId=$GATE_ID" \
        -d "metric=new_coverage" \
        -d "op=LT" \
        -d "error=70" > /dev/null

    # New Blocker Issues > 0 → Fail
    curl -s -X POST -u "admin:${ADMIN_PASSWORD}" \
        "http://localhost:${SONAR_PORT}/api/qualitygates/create_condition" \
        -d "gateId=$GATE_ID" \
        -d "metric=new_blocker_violations" \
        -d "op=GT" \
        -d "error=0" > /dev/null

    # New Critical Issues > 0 → Fail
    curl -s -X POST -u "admin:${ADMIN_PASSWORD}" \
        "http://localhost:${SONAR_PORT}/api/qualitygates/create_condition" \
        -d "gateId=$GATE_ID" \
        -d "metric=new_critical_violations" \
        -d "op=GT" \
        -d "error=0" > /dev/null

    echo "  ✅ Quality Gate 'Team Quality Gate' 已创建"
else
    echo "  ⚠️  Quality Gate 可能已存在，跳过创建"
fi
echo ""

# ====== 完成 ======
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "  🎉 SonarQube 环境搭建完成！"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo ""
echo "  访问地址:  http://localhost:${SONAR_PORT}"
echo "  管理员:    admin / ${ADMIN_PASSWORD}"
if [ -n "$TOKEN" ]; then
    echo "  API Token: $TOKEN"
fi
echo ""
echo "  下一步:"
echo "  1. 登录 Web UI 创建项目"
```

## 4. 项目总结

### 4.1 两周落地计划回顾

| 阶段 | 关键决策 | 避坑经验 |
|------|---------|---------|
| 部署 | Docker Compose 单机部署 | 不要在配置上过度工程化——20 人团队单机够用 |
| 首扫 | 同时扫后端和前端 | 首次扫描等待时间可能较长（15+ 分钟），提前通知团队 |
| 规则评审 | 5 人小组，1 小时达成 80% 共识 | 不要追求所有规则一次完美——剩余规则在后续 Sprint 中迭代 |
| 门禁设计 | New Code Only，Coverage ≥ 70% | 70% 是合理起点，最低不应低于 50% |
| 推广 | Lunch & Learn + 一对一协助 | 强制安装工具会引发抵触——先展示价值 |

### 4.2 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 部署速度 | Docker 一键启动 | 需要 Docker 基础知识 |
| 覆盖范围 | 前后端全覆盖 | 前端规则不如后端完善 |
| 团队接受度 | 渐进式推广，阻力小 | 需要 Leader 持续推动 |
| 维护成本 | 初期几乎零维护 | 随着项目增多需要运维投入 |

### 4.3 注意事项

1. **首次扫描不要直接开 Quality Gate**：先让团队看到扫描结果，进行一轮修复，第二周再开户禁。
2. **覆盖率门槛不是一蹴而就的**：如果当前覆盖率只有 12%，先设 50%，2 个 Sprint 后提到 70%，再提到 80%。
3. **尊重"Won't Fix"**：允许开发者标记"Won't Fix"，但要求填写 Comment 说明原因。这是团队信任的基础。

### 4.4 思考题

1. 如果团队在两周推广后，Quality Gate 通过率只有 30%，你应该采取什么措施？
2. 前端项目的覆盖率一直徘徊在 40%，门禁设 70% 导致大部分 PR 被阻断——如何调整策略？

> **答案提示**：第1题应先分析 Fail 的主要原因（覆盖率不足？新增 Bug？），针对性改进而非降低门槛。第2题可以分段提升覆盖率目标（先 50%，后 70%），同时优先补充核心组件的测试。

### 4.5 落地成功关键要素（Checklist）

两周落地能否成功，取决于以下 10 条关键要素。每一条都是真实项目中踩过的坑总结而来——对照检查，缺一条都可能翻车：

| # | 关键要素 | 为什么重要 | 验证方法 |
|---|---------|-----------|---------|
| 1 | **CTO/技术负责人亲自站台 Kickoff** | 没有自上而下的支持，质量工具会被视为"额外负担" | Kickoff 会议 CTO 是否讲了 5 分钟以上？ |
| 2 | **明确"先止血，后还债"的核心原则** | 避免团队因 500 个历史 Issue 集体崩溃 | 团队能否用一句话复述这个原则？ |
| 3 | **选定 1 名质量负责人全职投入 2 周** | 工具落地不是"顺手做"能完成的，必须有人盯全程 | 质量负责人的日历是否每天有 SQ 相关任务？ |
| 4 | **首次扫描后先展示价值，再开通门禁** | 先让团队发现"这个工具确实找到了我写的 Bug"，再强制 | Day 1-5 是否有至少 3 个开发者主动反馈"有用"？ |
| 5 | **规则评审不超过 5 人、不超过 90 分钟** | 全员评审会变成辩论赛，永远谈不完 | 规则评审会是否按时结束且有明确输出？ |
| 6 | **覆盖率目标分阶段设定，不做跳跃式要求** | 12% → 70% 直接跳变是反人性的，必然引发抵触 | 下个 Sprint 的目标是否不超过当前值的 2 倍？ |
| 7 | **CI Pipeline 先用 Warning 模式灰度 2 天** | 直接 Fail 会让开发者把门禁视为"敌人" | 灰度期间是否有误报？是否及时调整了规则？ |
| 8 | **全员安装 SonarLint 但容许 1-2 周学习期** | IDE 实时提示是最好的培训——但需要适应期 | 第 1 周末安装率是否 ≥ 80%？第 2 周末是否 ≥ 100%？ |
| 9 | **建立 Wiki 文档，知识不能只存在一个人脑中** | 质量负责人休假/离职时，整个体系不能崩塌 | Wiki 是否至少有排障 SOP + Issue 修复指南两个页面？ |
| 10 | **两周结束时做数据复盘，用数字说话** | 没有数据的落地是"感觉良好"，有数据的落地是"持续改进" | 是否产出了包含覆盖率/Issue/门禁 3 类数据的前后对比报告？ |

### 4.6 两周内常见紧急情况及应对 SOP

在实际落地过程中，以下 5 种紧急情况出现概率极高。提前准备好应对方案，不要临时抱佛脚：

| 紧急情况 | 触发信号 | 严重程度 | 应对 SOP | 责任人 |
|---------|---------|---------|---------|--------|
| **扫描导致 CI 队列雪崩** | Jenkins 同时触发 5+ 个扫描，CE 队列排到 50+ | 🔴 P0—阻塞所有开发 | ①暂停非必要分支的自动扫描 ②联系 SonarQube 管理员检查 CE Worker 数量 ③临时改为仅 master 分支触发扫描 ④必要时重启 SonarQube CE 服务 | 质量负责人 + DevOps |
| **门禁误杀——覆盖率计算方式引发争议** | 开发者抱怨"我加了 3 行测试，门禁还说覆盖率不足" | 🟡 P1—影响士气 | ①确认是 New Code Coverage（不是 Overall） ②确认 JaCoCo/lcov 报告是否正确生成 ③如果是框架代码（Lombok/Config 类）被计入分母，通过 sonar.exclusions 排除 ④临时调低阈值至 50%，2 天后恢复 | 质量负责人 |

### 4.7 追加思考题

3. **跨团队推广**：质量门禁在团队 A（20 人）已成功落地，CTO 要求在下个 Sprint 推广到团队 B（30 人，技术栈是 Go + Python）。团队 B 的 Tech Lead 私下找到你说"我们团队不需要这种工具，我们有自己的 Code Review 规范"。你会如何说服他？推广策略与第一次落地有哪些不同？

4. **门禁误杀危机**：某周五下午 5 点，一个紧急 Hotfix 需要上线，但 Jenkins 中 Quality Gate 报 Fail——新增代码的覆盖率只有 55%（阈值 70%）。值班开发者在群里说"这工具在阻止我们修线上 Bug"。作为质量负责人，你会怎么处理？如何在"质量红线"和"业务需求"之间平衡？

> **答案提示**：
> **第 3 题**：先不带门禁去推广，先让团队 B 用 SonarLint Connected Mode 体验 1 周。用团队 A 的数据（如"发现并修复了 3 个高危安全漏洞"）作为案例。跨团队推广的关键是找到团队 B 自己的痛点（比如"你们上次那个 P0 事故是因为空指针——SonarQube 能自动检测这类问题"），而不是简单复制团队 A 的配置。Go/Python 需要额外的扫描器配置（如 golangci-lint、pylint 与 SonarQube 的集成）。
> **第 4 题**：核心原则——**质量门禁是保护线，不是阻碍线**。紧急 Hotfix 场景的处理流程：(1) 立即判断门禁失败的原因——如果是覆盖率不足（而非新增 Blocker Bug），可以临时豁免；(2) 记录技术债：创建一张 Jira Ticket，写明"HOTFIX-123 豁免了覆盖率门禁，Sprint 结束后补充测试"，由 Tech Lead 审批；(3) 在 SonarQube 中将该 Issue 标记为"Confirmed"（确认后修）而非 Won't Fix；(4) 事后在团队复盘会上讨论：为什么 Hotfix 没有测试？能否在 Hotfix 分支上至少写 1 个关键路径的测试？这条记录会推动团队养成"修 Bug 必加测试"的习惯。

### 4.8 从本综合实战中学到的 5 条核心教训

这 5 条教训来自真实项目中被反复验证的血泪经验——每一条都价值一个 Sprint 的试错成本：

> **教训 1：技术落地本质是"人的工程"，不是"工具的工程"**
>
> 部署 SonarQube 只需 1 天。但让 20 个人接受它、信任它、日常使用它——这才是真正的硬仗。规则评审、Issue 分级展示、Lunch & Learn、一对一协助——这些"软性工作"占了两周落地时间的 60% 以上。不要低估人的因素：一个资深开发者的公开反对，足以让整个推广行动功亏一篑。

> **教训 2：没有基线的改进都是自我安慰**
>
> "覆盖率提升了""代码质量变好了"——这些话在没有数据基线之前毫无意义。项目启动前，花 30 分钟用精确数字记录现状（覆盖率 12%、503 个 Issue、每月 3-4 次事故）。两周后，用同样的测量方法对比数据。有了数字，团队的成就感是真实的；没有数字，所有的"进步"都只是感觉。

> **教训 3：门禁是最后一道防线，不是第一道**
>
> 如果把 Quality Gate 当作质量保障的唯一手段，你就是把 20 个开发者变成了"门禁绕过黑客"。正确的顺序是：SonarLint（IDE 中实时发现）→ Code Review（人工二次确认）→ Quality Gate（CI 自动兜底）。三道防线层层递进，才能在质量和效率之间取得平衡。

> **教训 4：允许"合理的例外"，但要求"不合理的代价"**
>
> 绝对化的门禁（"任何情况都不能绕过"）在真实业务场景中行不通——Hotfix 就是最好的反例。正确的治理策略是：允许例外（如 Hotfix 豁免覆盖率），但要求记录和后续偿还（如创建技术债 Ticket）。这让开发者感到"被信任"，同时保持了质量的长期可持续性。

> **教训 5：两周落地只是开始，持续运营才是真正的挑战**
>
> 两周能建立质量防护网，但不能建立质量文化。第 3 周起，新的挑战会接踵而至：新项目的接入、规则的持续迭代、覆盖率的缓慢爬升、False Positive 的持续审计、新人 Onboarding……质量治理是一场没有终点的马拉松。两周落地成功的标志不是"所有指标都绿了"，而是——**团队的每个人在下班前会习惯性地看一眼 SonarQube Dashboard**。
