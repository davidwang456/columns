# 第7章：规则、问题与修复闭环

## 1. 项目背景

**业务场景**：某物流公司的运单调度系统在上线第一个月产生了 1,276 个 SonarQube Issue。开发 Leader 打开项目页面后当场崩溃——"这么多问题，我们怎么修得完？"运维负责人也质疑："这么多告警，真正重要的有几个？全部修完要多久？"

团队陷入了典型的"Issue 焦虑"：数量多到让人不想看，但每条又标着 Blocker 或 Critical，看起来都很严重。在第一次质量周会上，团队讨论了 45 分钟也没能决定"先修哪 10 个 Issue"。

这个场景暴露了 SonarQube 推广中最核心的挑战：Issue 管理不是技术问题，而是工程管理问题。规则引擎可以自动发现问题，但判断哪些问题真正重要、如何分配修复工作、如何跟踪验证——这些都需要人工决策和团队协作。

**痛点放大**：

- **Issue 泛滥**：一次全量扫描可能产生数千个 Issue，团队无法消化
- **误报干扰**：某些规则对框架代码或设计模式产生误报，团队需要人工标记 False Positive
- **修复闭环断裂**：Issue 被分配给某人后石沉大海，无人跟踪验证
- **优先级混乱**：Blocker 级别的 Issue 有 50 个，但真正会导致线上事故的可能只有 3 个
- **上下文丢失**：Issue 的修复理由、设计方案、风险接受记录没有地方存档

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（打开 SonarQube 项目页，用颤抖的手指数着 Issues）："大师……1267 个 Issue！光 Blocker 就有 18 个！我们项目总共才 3 万行代码，平均每 24 行就有一个问题。这是不是说明我们代码写得像屎一样？"

**大师**："冷静，小胖。这 1267 个 Issue 不全是'真正的问题'。我用体检来类比：你去做全身体检，报告上可能有 30 项指标异常——但其中 10 项是'体检项目本身不适合你的年龄'（误报），8 项是'指标偏高但不构成疾病'（Minor），5 项是'去年就有的老毛病'（历史债务），真正需要立即干预的可能只有 2-3 项。"

**小白**（盯着 Issue 详情页）："我注意到每条 Issue 都有几个关键属性：Type、Severity、Status、Assignee。这几个属性怎么用才能让团队不乱？"

**大师**："这几个属性构成了 Issue 管理的四个维度：

- **Type**（类型）：Bug 是错误，Vulnerability 是安全漏洞，Code Smell 是可维护性问题，Security Hotspot 需要人工审核。不同类型的修复优先级不同——Bug 和 Vulnerability 应该优先于 Code Smell。

- **Severity**（严重级别）：Blocker > Critical > Major > Minor > Info。但注意——Severity 是规则作者的主观判断，不一定匹配你的业务影响。一个 Blocker Code Smell（如复杂度过高）可能不如一个 Major Bug（如偶发的 NPE）紧急。

- **Status**（状态）：Open → Confirmed → Resolved → Closed。团队需要约定每个状态谁来操作、什么条件触发状态变更。

- **Assignee**（经办人）：Issue 必须有明确的负责人，否则就像医院里没人签字的检查报告。"

**小胖**："那怎么区分误报？ESLint 里我可以在行末加 `// eslint-disable-next-line`，SonarQube 里怎么办？"

**大师**："SonarQube 有三种方式处理误报或不需要修复的问题：

1. **标记为 False Positive（误报）**：这行代码实际上没问题，规则判断错了。需要添加 Comment 说明原因。
2. **标记为 Won't Fix（不再修复）**：这行代码确实有问题，但团队评估后决定接受这个风险（比如修复成本太高、下个版本会重写）。
3. **Accept（接受）**：用于 Security Hotspot，表示安全人员审核后认为没问题。

三种操作都会关闭 Issue，但含义完全不同。我强烈建议团队给 Won't Fix 的 Issue 加一个 Comment 说明理由——否则三个月后新人看到了会以为老代码故意留坑。"

**小白**："我看到还有一个 Bulk Change 功能——可以一次操作修改很多 Issue。这个在什么场景下用？"

**大师**："典型场景：你在一个模块中发现 50 个 Issue 都是因为某个第三方库的 API 设计导致的——比如所有 `PreparedStatement` 都被标记为'资源可能未关闭'，但实际上这个库帮你管理了生命周期。这 50 个就可以一次标记为 False Positive，并统一添加 Comment：'使用 XX 库的连接池管理，资源自动回收'。"

**小胖**："那怎么确认一个 Issue 真的被修复了？我改了代码，重新扫描，Issue 就自动消失了吗？"

**大师**："是的，但有前提条件。SonarQube 的 Issue 跟踪机制是：

1. 你修复代码 → 提交 → CI 重新扫描 → 上传报告到 SonarQube
2. SonarQube 比对两次扫描结果：如果某行之前有 Issue，新报告中该行没有 Issue → 自动将 Status 改为 Resolved
3. 经过一定时间（默认 30 天）或手动确认 → Status 改为 Closed

但有一个常见误区：如果你移动了代码（重构），原行的 Issue 会变为 'Closed（No longer exists）'，新行会生成一个新的 Issue。所以大规模重构后建议先处理一遍 Issue，再开始重构。"

**小白**："那团队怎么定修复节奏？一天修 10 个？一周 50 个？"

**大师**："最有效的方法是分级治理：

- **第 0 级：阻断性修复**：New Code 上的 Blocker Bug 和 Vulnerability，必须是 0。这是合并代码的硬门槛（Quality Gate）。
- **第 1 级：每日清零**：每天下班前确保当日没有新增 Issue（New Code 上的所有 Issue）。
- **第 2 级：每周治理**：每周挑选 3-5 个历史 Issue 修复——优先选高严重级别 + 频繁被访问的模块。
- **第 3 级：每月复盘**：每月团队一起看质量趋势图，决定下个月的重点治理方向。

这个节奏下，历史 Issue 是会持续减少的，而不会越积越多——关键是守好第 0 级和第 1 级。"

## 3. 项目实战

### 3.1 环境准备

- SonarQube 实例已部署
- 已有至少一个分析过的项目（可从第3章继续）
- 具有项目管理权限的账号

### 3.2 分步实现

**步骤 1：理解 Issue 的完整信息结构**

打开任意一个 Issue 详情页，观察以下字段：

```
Type:          Bug / Vulnerability / Code Smell / Security Hotspot
Severity:      BLOCKER / CRITICAL / MAJOR / MINOR / INFO
Status:        OPEN / CONFIRMED / REOPENED / RESOLVED / CLOSED
Resolution:    FIXED / FALSE-POSITIVE / WONTFIX / REMOVED
Rule:          规则 Key（如 java:S1141）
Message:       问题描述
Tags:          标签（如 cwe, owasp-a1, cert）
Assignee:      经办人
Creation Date: 创建时间
Comments:      讨论记录
```

**步骤 2：Issue 状态转换实战**

以第3章的 `BrokenCalculator` 为例：

**(a) 确认 Issue（OPEN → CONFIRMED）**

找到 "Divide by zero" Issue（Bug/Blocker），右上角点击 "Confirm" → Status 变为 CONFIRMED。这个操作表示："我已经看到这个问题了，我承认它需要修复"。

**(b) 标记误报（OPEN → RESOLVED as FALSE-POSITIVE）**

找到 "equals() without hashCode()" Issue。如果你认为这个类不需要放到 HashMap 中，不应要求覆写 hashCode：

1. 点击 Issue 详情
2. 点击 "Change" → Resolution → "False Positive"
3. 在 Comment 中输入："This class is never used in hash-based collections. Intentional."
4. Status 变为 RESOLVED，Resolution 为 FALSE-POSITIVE

**(c) 分配 Issue（Assignee）**

将 "SQL Injection" Issue 分配给具体的开发者：

1. 在 Issue 列表页，勾选目标 Issue
2. 点击 "Bulk Change" → "Assign" → 选择团队成员
3. 或在 Issue 详情页直接点击 Assign

**(d) 添加标签（Tags）**

为 Issue 添加自定义标签便于筛选，如 `backend-priority`、`sprint-21`、`security-review-needed`：

```bash
# 通过 API 批量添加标签
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/issues/bulk_change" \
  -d "issues=com.example:broken-calculator:java:S1141" \
  -d "add_tags=tech-debt,sprint-22"
```

**步骤 3：批量操作演练**

假设你有 20 个由"未使用 import"引起的 Code Smell（规则 `java:S1128`）：

1. 在 Issues 列表页，筛选规则为 `java:S1128`
2. 全部选中（共 20 个）
3. 点击 "Bulk Change"
4. 选择 Action: "Assign"，Assignee: "小胖"
5. 添加 Comment: "Please clean up unused imports. Low effort, high impact."

**步骤 4：验证修复闭环**

修改 `BrokenCalculator.java` 中的除零问题：

```java
public int divide(int a, int b) {
    if (b == 0) {
        throw new IllegalArgumentException("Divisor cannot be zero");
    }
    return a / b;
}
```

重新编译并扫描：

```bash
javac src/main/java/com/example/BrokenCalculator.java -d target/classes
sonar-scanner
```

刷新 Web UI，原来的 "Divide by zero" Issue 应变为 **RESOLVED**（Resolution: FIXED）。

### 3.3 通过 API 进行 Issue 管理

```bash
# 列出所有 OPEN 状态的 Blocker Issue
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/issues/search?projectKeys=com.example:broken-calculator&severities=BLOCKER&statuses=OPEN" \
  | python3 -m json.tool

# 按类型统计 Issue 数量
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/issues/search?projectKeys=com.example:broken-calculator&statuses=OPEN&facets=types" \
  | python3 -m json.tool | grep -E '"val"|"count"'

# 分配 Issue 给特定用户
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/issues/assign" \
  -d "issue=com.example:broken-calculator:java:S2111" \
  -d "assignee=developer1"

# 添加 Comment
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/issues/add_comment" \
  -d "issue=com.example:broken-calculator:java:S2111" \
  -d "text=Fixing this in sprint 23. Adding null guard to constructor."
```

### 3.4 建立 Issue 治理看板

编写一个简单的 Python 脚本，汇总团队 Issue 数据：

```python
#!/usr/bin/env python3
"""SonarQube Issue 治理看板导出脚本"""
import requests
import json

SONAR_URL = "http://localhost:9000"
TOKEN = "squ_xxxxxxxx"
PROJECT = "com.example:order-service"

headers = {"Authorization": f"Bearer {TOKEN}"}

# 获取 Issue 统计
def get_issue_stats():
    resp = requests.get(
        f"{SONAR_URL}/api/issues/search",
        params={"projectKeys": PROJECT, "statuses": "OPEN",
                "facets": "severities,types,assignees"},
        headers=headers, auth=("admin", "Sonar@2024Admin")
    )
    data = resp.json()
    print(f"=== {PROJECT} Issue 看板 ===")
    print(f"Open Issues Total: {data['total']}")
    for facet in data.get("facets", []):
        for val in facet["values"]:
            print(f"  {val['val']}: {val['count']}")

# 获取 Top 5 最急需修复的 Issue
def get_top_issues():
    resp = requests.get(
        f"{SONAR_URL}/api/issues/search",
        params={"projectKeys": PROJECT, "statuses": "OPEN",
                "severities": "BLOCKER,CRITICAL", "ps": 5,
                "s": "CREATION_DATE", "asc": "false"},
        headers=headers, auth=("admin", "Sonar@2024Admin")
    )
    print("\n=== Top 5 需紧急修复 ===")
    for issue in resp.json().get("issues", []):
        print(f"  [{issue['severity']}] {issue['message'][:80]}")

if __name__ == "__main__":
    get_issue_stats()
    get_top_issues()
```

## 4. 项目总结

### 4.1 优点与缺点

| 维度 | SonarQube Issue 管理 | GitHub Issues/Jira |
|------|---------------------|-------------------|
| 自动关联代码行 | ✅ 精确定位到行号 | ❌ 手动粘贴链接 |
| 自动状态同步 | ✅ 修复后自动关闭 | ❌ 手动更新状态 |
| 批量操作 | ✅ 支持 Bulk Change | ✅ 支持 Label/Assignee 批量 |
| 自定义工作流 | ❌ 固定状态机（5 个状态） | ✅ 灵活的工作流引擎 |
| 与外部系统集成 | 🟡 通过 Webhook 间接通知 | ✅ 原生 API 集成 |
| 研发流程嵌入 | ✅ 和 CI/PR 深度绑定 | 🟡 需要手动关联 |

### 4.2 适用场景

- **日常缺陷治理**：所有扫描出的问题统一管理
- **质量周会**：以 Issue 看板为基础，讨论 Block/Critical Issue 的修复计划
- **安全审计**：Security Hotspot Review 流程全程在 SonarQube 中完成
- **新成员培训**：让新人从修复 Minor Issue 开始熟悉代码

**不适用场景**：
- 需要复杂审批流（三级审批、跨部门 Sign-off）的场景 → 应使用 Jira/ServiceNow
- 需要和需求/Bug 跟踪系统双向同步的场景 → 需额外开发 API 脚本

### 4.3 注意事项

1. **Status 状态机不可自定义**：Open → Confirmed → Resolved → Closed 是固定流程，不能添加自定义状态。
2. **False Positive vs Won't Fix**：两者都是关闭 Issue，但含义不同。False Positive 是"规则错了"，Won't Fix 是"我选择不修"。对审计来说，Won't Fix 的 Issue 仍可能被统计为"未修复的安全问题"。
3. **Comment 权限**：只有 Issue 的 Assignee 和有项目管理权限的人可以添加 Comment。

### 4.4 常见踩坑经验

**故障 1：标记为 False Positive 后，下次扫描又被重新打开**

根因：规则基于代码模式匹配，而非 Issue ID。如果你标记 False Positive 后没有在代码中添加抑制注释（如 Java 的 `@SuppressWarnings`），重扫时如果规则再次匹配到同一行，会生成新 Issue。解决方案：代码中加 `// NOSONAR` 或在 Quality Profile 中永久关闭该规则。

**故障 2：Bulk Change 操作后 Issue 没有变化**

根因：Bulk Change 操作异步处理，并且有权限限制（批量操作需要 Administer Issues 权限）。等 30 秒后刷新，或检查 Background Tasks 中的 Bulk Change 任务状态。

**故障 3：分配 Issue 给某人后，对方登录前看不到**

根因：SonarQube 的通知功能默认关闭，需要配置邮件服务器或在 CI 中调用 Web API 发送通知。

### 4.5 思考题

1. 团队有 500 个历史 Issue，其中 200 个是 Code Smell（Minor），50 个是 Bug（Major），10 个是 Bug（Critical）。你如何制定一份 3 个月的修复计划？
2. "False Positive" 和 "Won't Fix" 的区别是什么？为什么在安全审计报告中，两者要分开统计？

> **答案提示**：第1题建议先用 New Code 门禁止血，再按"Critical Bug → Major Bug → 核心模块的 Code Smell"顺序修复。第2题见本章正文。

---

> **推广计划提示**：Issue 管理是团队推广中最容易"落地失败"的环节——如果 Issue 太多没人修，工具就成了装饰品。建议质量负责人每周发布一份"Issue 治理周报"：本周新增 Issue、本周修复 Issue、Top 3 责任人。数据可视化能有效驱动修复意愿。
