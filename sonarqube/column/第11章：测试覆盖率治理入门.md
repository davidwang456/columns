# 第11章：测试覆盖率治理入门

## 1. 项目背景

**业务场景**：某金融支付团队接入 SonarQube 后，Quality Gate 中设置了"New Code Coverage >= 80%"。一个月后，覆盖率数字确实达标了——90%！但产品的 Bug 数量没有明显下降。质量负责人疑惑地抽查了一些测试代码，发现了令人啼笑皆非的现象：

- `assertTrue(true)` 测试占 15%
- 只有 getter/setter 的测试占 30%
- 测试方法名称为 `test1()`, `test2()`, `test3()`，不知道测了什么
- 有一个 `if/else` 分支的测试覆盖率是 100%，但 `else` 分支永远不会被执行（逻辑错误）
- 一个异常路径的测试写了 `try/catch` 后什么都不断言

数字好看，质量没变。这是"覆盖率陷阱"的典型案例——团队把覆盖率从"质量问题指示器"变成了"KPI 数字游戏"。

**痛点放大**：

- **覆盖率≠测试质量**：行覆盖率告诉你"这行代码被执行过"，但不告诉你"执行结果被正确断言"。
- **无效测试泛滥**：为凑数字而写的测试，反而增加了维护成本。
- **路径映射错误**：Docker 容器中的覆盖率报告路径和 SonarQube 源码路径不一致，覆盖率显示 0%。
- **覆盖率门禁滥用**：对所有代码（包括 DTO/配置/常量类）要求覆盖率，导致团队反感。

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（看着 SonarQube 上的覆盖率大盘，骄傲地指着 92%）："大师你看！我们项目的覆盖率 92%！超过了 80% 的门禁线！我们团队牛不牛？"

**大师**（点开测试代码，随机翻了几页）："小胖，你来看这个测试——"

```java
@Test
void testSomething() {
    assertTrue(true);
}
```

"这个覆盖率是真实的吗？你对它有什么意义？"

**小胖**（脸红）："这个……这个是建模生成代码的测试。反正这些 DTO 也没什么逻辑，写个测试覆盖一下就行……"

**大师**："这就是覆盖率骗局。行覆盖率说'这行被执行了'，但它不说'执行结果被验证了'。`assertTrue(true)` 永远不会失败——它让 JaCoCo 觉得代码被执行了，但实际上没有测试任何逻辑。这就好比你去健身房打卡但不运动——卡是刷了，肉是一斤没少。"

**小白**："那覆盖率到底有没有用？如果容易作弊，为什么还要作为 Quality Gate 条件？"

**大师**："覆盖率有用，但要正确理解它。覆盖率是一个**必要但非充分**的代码质量信号。'必要'的意思是：如果覆盖率低，说明很多代码没有测试保护——这是危险信号。'非充分'的意思是：覆盖率高不代表测试质量好。

正确的使用方式是：用覆盖率发现'该测但没测的代码'，而不是用覆盖率考核'谁写得多'。"

**小白**："我注意到 SonarQube 显示的是 Line Coverage，但 JaCoCo 实际上还支持 Branch Coverage 和 Instruction Coverage。这些有什么区别？"

**大师**："三种覆盖率维度：

- **Line Coverage（行覆盖）**：某个源码行是否被执行。最直观，但也是最粗糙的——一行可能包含多个条件分支。
- **Branch Coverage（分支覆盖）**：`if/else`, `switch`, `try/catch` 的分支是否都被覆盖过。比行覆盖率更有意义——因为未覆盖的分支往往藏着 Bug。
- **Instruction Coverage（指令覆盖）**：JVM 字节码指令级别的覆盖。最精确，但对开发者不够直观。

SonarQube 默认显示 Line Coverage，但也支持 Branch Coverage。我建议团队把门禁条件改为 Branch Coverage，因为它更体现真实的测试充分性。"

**小胖**："那路径问题呢？我们 Docker 里跑 JaCoCo 生成的报告，源文件路径是 `/app/src/main/java/...`，但 SonarScanner 在宿主机上的路径是 `/home/user/project/src/main/java/...`，这两边对不上，覆盖率一直是 0%。"

**大师**："这是容器化构建中最常见的问题。解决方案有三种：

1. **使用相对路径**：在 `build.gradle.kts` 或 `pom.xml` 中配置 JaCoCo 使用相对路径生成报告。
2. **统一构建和扫描的根目录**：在 Docker 容器内完成编译、测试、扫描三个步骤，保证路径一致。
3. **使用 `sonar.coverage.jacoco.xmlReportPaths` 指定绝对路径**：如果在容器外扫描，需要手动 mount 报告并映射路径。

方案 1 最简单、最可靠，强烈推荐。"

## 3. 项目实战

### 3.1 环境准备

- Java 项目（Maven/Gradle），已有单元测试
- SonarQube 实例

### 3.2 分步实现

**步骤 1：Maven 项目配置 JaCoCo**

在父 `pom.xml` 中配置：

```xml
<properties>
    <jacoco.version>0.8.12</jacoco.version>
</properties>

<build>
    <plugins>
        <plugin>
            <groupId>org.jacoco</groupId>
            <artifactId>jacoco-maven-plugin</artifactId>
            <version>${jacoco.version}</version>
            <executions>
                <execution>
                    <id>prepare-agent</id>
                    <goals><goal>prepare-agent</goal></goals>
                </execution>
                <execution>
                    <id>report</id>
                    <phase>verify</phase>
                    <goals><goal>report</goal></goals>
                </execution>
            </executions>
        </plugin>

        <!-- SonarQube 覆盖率配置 -->
        <plugin>
            <groupId>org.sonarsource.scanner.maven</groupId>
            <artifactId>sonar-maven-plugin</artifactId>
            <version>4.0.0.4121</version>
        </plugin>
    </plugins>
</build>
```

在 `sonar-project.properties` 中无需任何覆盖率配置——Maven 插件会自动发现 `jacoco.xml`。

**步骤 2：Gradle 项目配置 JaCoCo**

在 `build.gradle.kts` 中：

```kotlin
plugins {
    jacoco
}

tasks.test {
    useJUnitPlatform()
    finalizedBy(tasks.jacocoTestReport)
}

tasks.jacocoTestReport {
    dependsOn(tasks.test)
    reports {
        xml.required.set(true)   // SonarQube 需要 XML 格式
        html.required.set(true)  // 开发者在浏览器中查看
        csv.required.set(false)
    }
}

sonarqube {
    properties {
        property("sonar.coverage.jacoco.xmlReportPaths",
            "${buildDir}/reports/jacoco/test/jacocoTestReport.xml")
    }
}
```

**步骤 3：排除不需要覆盖的代码**

在 SonarQube 配置中添加排除：

```properties
# 排除 DTO、POJO、配置类、启动类
sonar.coverage.exclusions=\
  **/dto/**,\
  **/model/**,\
  **/config/**,\
  **/*Application.java,\
  **/*Properties.java
```

**Gradle 方式**：

```kotlin
sonarqube {
    properties {
        property("sonar.coverage.exclusions",
            "**/dto/**,**/model/**,**/config/**")
    }
}
```

**步骤 4：解决覆盖率路径不一致问题**

**Docker 场景**——在 Dockerfile 中使用相对路径构建：

```dockerfile
FROM maven:3.9-eclipse-temurin-17

# 关键：WORKDIR 必须和 sonar-scanner 执行时的目录一致
WORKDIR /app
COPY . .

# 在容器内一并完成构建、测试、扫描
RUN mvn clean verify sonar:sonar \
    -Dsonar.token=$SONAR_TOKEN \
    -Dsonar.host.url=$SONAR_HOST_URL
```

这样 JaCoCo 生成的报告中源文件路径是 `/app/src/main/java/...`，SonarScanner 也在 `/app` 目录执行——路径一致。

**步骤 5：验证覆盖率数据**

```bash
# 查看覆盖率
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/measures/component?component=com.example:payment&metricKeys=coverage,branch_coverage,lines_to_cover,uncovered_lines" \
  | python3 -m json.tool
```

输出示例：

```json
      {"metric": "uncovered_lines", "value": "266"}

```

**步骤 6：识别覆盖率漏洞**

```bash
# 找出覆盖率最低的 5 个文件
curl -s -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/measures/component_tree?component=com.example:payment&metricKeys=coverage&s=metric&asc=true&ps=5" \
  | python3 -m json.tool | grep -E '"name"|"value"'
```

### 3.3 JaCoCo 报告验证脚本

编写脚本检查 JaCoCo 报告是否正确生成：

```bash
#!/bin/bash
# check-jacoco.sh - 验证 JaCoCo 覆盖率和源文件路径

JACOCO_XML="target/site/jacoco/jacoco.xml"
if [ ! -f "$JACOCO_XML" ]; then
    echo "❌ jacoco.xml 未生成！请先执行 mvn verify"
    exit 1
fi

echo "📊 覆盖率汇总："
python3 -c "
import xml.etree.ElementTree as ET
tree = ET.parse('$JACOCO_XML')
root = tree.getroot()
for counter in root.findall('.//counter'):
    print(f\"  {counter.get('type'):20s} missed={counter.get('missed'):6s} covered={counter.get('covered'):6s}\")
"

echo ""
echo "📁 源文件路径（前 5 个）："
python3 -c "
import xml.etree.ElementTree as ET
tree = ET.parse('$JACOCO_XML')
root = tree.getroot()
for i, pkg in enumerate(root.findall('.//package')):
    for sf in pkg.findall('sourcefile'):
        print(f\"  {sf.get('name')}\")
    if i >= 5: break
"

echo ""
echo "✅ 如果源文件路径是相对路径（不以 / 开头），SonarQube 可正确匹配"
```

### 3.4 覆盖率治理策略

**反模式识别清单**：

| 反模式 | 表现 | 检测方式 |
|--------|------|----------|
| 无效断言 | `assertTrue(true)` | Code Review |
| 只测 getter/setter | 大量对 DTO 的简单断言 | 检查测试文件中 `@Test` 方法的内容 |
| 覆盖但不验证 | `doSomething(); // 没有 assert` | 检查测试方法是否有断言 |
| 复杂的 setUp 无人看 | @BeforeEach 中做了大量逻辑但没测 | 检查 setUp 方法中的代码 |

## 4. 项目总结

### 4.1 覆盖率的最佳实践对照

| 实践 | ✅ 推荐 | ❌ 不推荐 |
|------|---------|----------|
| 门禁阈值 | New Code Coverage ≥ 80%（Branch Coverage） | Overall Coverage ≥ 95%（数字游戏） |
| 排除策略 | 排除 DTO/配置/生成的代码 | 什么都不排除（引发反感） |
| 测试质量审查 | Code Review 检查测试有效性 | 只看覆盖率数字 |
| 指标维度 | Branch Coverage + Line Coverage | 只看 Line Coverage |
| 目标设定 | "找出未覆盖的关键路径" | "全项目达到 90%" |

### 4.2 适用场景

- **核心业务逻辑**：支付、结算、风控等模块要求高覆盖率
- **公共库/组件**：被多处引用的基础库
- **复杂算法**：排序、加密、状态机等容易出错的计算逻辑

**不适用场景**：
- 已有充分集成测试的遗留系统（盲目要求单元测试覆盖率可能破坏现有架构）
- ORM 映射类、序列化类（框架生成的样板代码）

### 4.3 注意事项

1. **Branch Coverage vs Line Coverage**：优先用 Branch Coverage 作为门禁条件，因为它更准确反映测试充分性。
2. **排除清单公开透明**：覆盖率排除规则要写入团队规范文档，不能让开发者"私下"排除文件。
3. **不用覆盖率考核个人绩效**：覆盖率是代码质量信号，不是 KPI。一旦和绩效挂钩，"刷数据"就不可避免。

### 4.4 常见踩坑经验

**故障 1：多模块 Maven 聚合覆盖率重复计算**

根因：JaCoCo 的 `report-aggregate` goal 和子模块独立报告同时存在。解决：只在父模块使用 `report-aggregate`，子模块不独立生成报告。

**故障 2：JaCoCo 和 Lombok 冲突**

根因：Lombok 生成的 `@Data` 注解方法（如 getter/setter）被 JaCoCo 视为需要覆盖。解决：添加 `lombok.config` 文件，配置 `lombok.addLombokGeneratedAnnotation = true`。

### 4.5 思考题

1. 如果 Branch Coverage 达到 85%，但团队的 Bug 率并没有下降，可能的原因是什么？如何改进？
2. 在 microservice 架构中，单元测试、集成测试、端到端测试的覆盖率如何分配？哪种覆盖率在 SonarQube 中更有价值？

> **答案提示**：第1题说明测试质量低（缺少断言或场景不全），需要在 Code Review 中加入测试质量审查。第2题见第23章。

---

> **推广计划提示**：覆盖率治理应从 2-3 个核心模块开始，避免一开始就在全项目推行 80% 门禁。建议测试团队和开发团队一起制定覆盖率基线：先跑一次全量覆盖率报告，识别出覆盖率低于 20% 的关键模块，优先补充测试。
