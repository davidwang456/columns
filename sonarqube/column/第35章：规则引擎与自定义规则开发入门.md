# 第35章：规则引擎与自定义规则开发入门

## 1. 项目背景

**业务场景**：某金融公司的架构团队制定了一套内部编码规范——"Controller 层不允许直接操作 DAO，必须经过 Service 层"。这条规则被写入开发手册，但在 Code Review 中只能靠人工检查——审查者经常忘记，且不同审查者的判断标准不一致。

团队决定将这条规范实现为 SonarQube 自定义规则，让机器自动检查——违反时在 PR 中自动标记为 Block Issue。但开发团队对 SonarQube 插件开发完全陌生，不知道从何入手。

**痛点放大**：

- **内部规范依赖人工审查**：容易遗漏，标准不一
- **SonarQube 内建规则无法覆盖所有团队规范**：框架约定、架构分层、命名模式等需要定制
- **插件开发门槛高**：不理解 AST、Visitor 模式、SSLR 语法树等底层机制
- **测试规则困难**：如何验证自定义规则不会产生大量误报？
- **性能影响未知**：自定义规则是否是"扫描时间的隐形杀手"——开发者担心引入的自定义规则会显著拖慢扫描速度
- **规则维护成本被低估**：写一条规则只需 2 小时，但维护它（适配新版本、修复误报、优化性能）可能需要 20 小时
- **规则优先级和严重程度的困惑**：Blocker / Critical / Major / Minor / Info 五级体系在自定义规则中如何合理分配？错误分级会导致"狼来了"效应
- **多语言规则的一致性挑战**：同一条规范（如"禁止使用 System.out.println"）在 Java 和 Kotlin 中需要分别实现——规则越多，维护负担越重

**自定义规则开发的技术栈全景图**：

```
规则需求分析 → 选择实现方式 → 编写规则代码 → 编写单元测试 
    → 在真实项目上验证 → 注册到 Quality Profile → 按需推广
```

| 实现方式 | 适用场景 | 技术门槛 | 性能 |
|---------|---------|---------|------|
| SSLR AST Visitor | 模式匹配、命名约定、导入检查 | 低 | 优秀 |
| sonar-java Checks API | 语义分析、数据流分析、类型检查 | 中 | 良好 |
| XPath 规则（社区版不支持）| 简单的 AST 模式匹配 | 低 | 一般 |
| 自定义 Sensor | 非标准文件格式的分析 | 高 | 可控 |

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（对着架构文档里的 20 条编码规范）："大师，这些规范写在文档里根本没人看。能不能把它们做成 SonarQube 规则——谁违反了直接报 Issue？"

**大师**："可以。SonarQube 支持自定义规则——你写一个插件，注册到 SonarQube，激活在 Quality Profile 中，就像内建规则一样工作。对于 Java 项目，你有两种方式：

1. **使用 SSLR（SonarSource Language Recognizer）**：编写基于语法树（AST）的 Visitor，适合模式匹配型规则（如'禁止在 Controller 中 import DAO'）。
2. **使用 sonar-java 提供的 Checks API**：编写基于语义分析的规则，适合需要类型信息、数据流分析的规则（如'检查某个方法返回类型是否正确'）。

入门推荐方式 1——SSLR Visitor 模式简单直接，不需要理解完整的类型系统。"

**小白**："自定义规则的执行效率和内建规则一样吗？会不会拖慢扫描速度？"

**大师**："自定义规则和内建规则使用相同的 AST 和 Visitor 框架——性能上几乎没有差异，前提是你的 Visitor 实现不包含高复杂度的循环或 IO 操作。每条自定义规则的额外耗时通常是毫秒级的。"

**小胖**："我们团队还有一个规范：'禁止在 Service 层的方法中捕获异常后只打印日志不重新抛出'，因为吞掉异常会导致调用方感知不到错误。这个能用自定义规则检测吗？"

**大师**："这需要做**数据流分析**——不仅检查语法，还要跟踪异常对象的流向。用 SSLR Visitor 实现会比较复杂，建议使用 sonar-java 的 Checks API：

```java
@Rule(key = "NoSwallowedException",
      name = "Exception should not be swallowed in Service layer",
      priority = Priority.CRITICAL)
public class NoSwallowedExceptionRule 
        extends IssuableSubscriptionVisitor {

    @Override
    public List<Kind> nodesToVisit() {
        // 只关注 Catch 块
        return List.of(Kind.CATCH);
    }

    @Override
    public void visitNode(Tree tree) {
        CatchTree catchTree = (CatchTree) tree;
        BlockTree block = catchTree.block();
        
        // 检查 Catch 块内是否只有日志而没有 throw
        boolean hasThrow = hasThrowStatement(block);
        boolean hasLog = hasLogStatement(block);
        
        // 如果当前文件在 service 包下
        if (isInServicePackage() && hasLog && !hasThrow) {
            reportIssue(catchTree.catchParameter(), 
                "Service 层捕获异常后必须重新抛出或包装后抛出，" +
                "禁止只记录日志不抛出。");
        }
    }
    
    private boolean hasThrowStatement(BlockTree block) { /* 遍历 block */ }
    private boolean hasLogStatement(BlockTree block) { /* 检查 logger.error */ }
    private boolean isInServicePackage() { /* 检查包名包含 .service */ }
}
```

这类规则的难点不在于写代码，而在于**精确找到你需要的 AST 节点类型**。我的建议是：先用 IntelliJ 的 PSI Viewer 工具打开一个示例 Java 文件，看 AST 结构——然后对照写 Visitor。"

**小白**："规则写好后，怎么在正式上线前做充分的回归测试？我们担心规则会在某些边缘情况下产生误报。"

**大师**："自定义规则的测试分三层：

**L1 单元测试**（CheckVerifier）：
测试文件要覆盖 4 种情况：
- ✅ 应该被检测到的违规代码（用 `// Noncompliant` 标记）
- ✅ 应该被检测到的违规代码（用 `// Noncompliant {{expected message}}` 精确验证消息）
- ✅ 不应该被检测到的合规代码（正常写法）
- ✅ 边界情况：空文件、只有注释、嵌套在内部类中

```java
// src/test/files/NoDaoInControllerRule.java
package com.company.user.controller; // 在 controller 包

import com.company.user.dao.UserDao; // Noncompliant {{Controller 不应直接导入 DAO}}

import com.company.user.service.UserService; // 合规

public class UserController {
    private UserDao userDao; // Noncompliant
    private UserService userService; // 合规
}

// 边界情况测试
class InnerHelper {  // 内部类
    private UserDao dao; // Noncompliant - 仍在 controller 文件内
}
```

**L2 真实项目验证**（试跑）：

```bash
# 将新规则部署到测试环境 SonarQube
# 对 2-3 个真实的公司项目执行扫描
mvn sonar:sonar -Dsonar.host.url=http://test-sonar:9000 -Dsonar.token=xxx

# 然后手动检查所有 Issues：
# 1. 统计总 Issue 数
# 2. 人工抽查 20% 的 Issues 判断是否误报
# 3. 如果误报率 > 10% → 回炉修改规则
# 4. 如果漏报（有违规但规则没检测到）→ 补充规则逻辑
```

**L3 灰度推广**（分阶段激活）：

不是直接在全员 Quality Profile 中激活——而是：
1. Week 1：只在测试环境的 Profile 中激活，观察误报率
2. Week 2：在 1-2 个试点团队（早期接入者）的 Profile 中激活
3. Week 3：收集反馈，调整规则（调整 message、降低 Severity 等）
4. Week 4：全员推广

只有经过这三层测试的规则才能真正上线。跳过任何一层的规则上线都是在制造信任危机。"

**步骤 2：编写自定义规则**

`src/main/java/com/company/rules/NoDaoInControllerRule.java`：

```java
package com.company.rules;

import org.sonar.check.Rule;
import org.sonar.plugins.java.api.JavaFileScanner;
import org.sonar.plugins.java.api.JavaFileScannerContext;
import org.sonar.plugins.java.api.tree.*;

@Rule(key = "NoDaoInController",
      name = "Controller should not directly use DAO",
      description = "Controller 层不应直接操作 DAO，必须通过 Service 层",
      priority = Priority.MAJOR,
      tags = {"architecture", "company-standard"})
public class NoDaoInControllerRule
        extends BaseTreeVisitor
        implements JavaFileScanner {

    private JavaFileScannerContext context;

    @Override
    public void scanFile(JavaFileScannerContext context) {
        this.context = context;
        scan(context.getTree());
    }

    @Override
    public void visitImport(ImportTree tree) {
        String importName = tree.qualifiedIdentifier().toString();
        if (importName.endsWith("Dao") || importName.endsWith("Repository")) {
            // 检查当前文件是否在 controller 包下
            String fileName = context.getInputFile().filename();
            String packageName = getPackageName(context.getTree());
            if (packageName != null &&
                packageName.contains(".controller")) {
                context.reportIssue(this, tree,
                    "Controller 不应直接导入 DAO: " + importName +
                    "。请通过 Service 层访问。");
            }
        }
        super.visitImport(tree);
    }

    private String getPackageName(Tree tree) {
        if (tree.is(Tree.Kind.COMPILATION_UNIT)) {
            CompilationUnitTree cu = (CompilationUnitTree) tree;
            if (cu.packageDeclaration() != null) {
                return cu.packageDeclaration()
                         .packageName().toString();
            }
        }
        return null;
    }
}
```

import org.sonar.plugins.java.api.JavaRulesDefinition;
import org.sonar.plugins.java.api.JavaCheck;

public class MyJavaRulesDefinition
        implements RulesDefinition, JavaRulesDefinition {

    @Override
    public void define(Context context) {
        NewRepository repo = context
            .createRepository("company-java-rules", "java")
            .setName("Company Java Rules");

        // 这里注册每条规则（SonarQube API 会自动从 @Rule 注解读取）
        repo.done();
    }

    @Override
    public Iterable<Class<? extends JavaCheck>> checkClasses() {
        return List.of(NoDaoInControllerRule.class);
    }
}

**步骤 7：编写更复杂的规则——检查方法参数非空注解**

```java
@Rule(key = "MissingNonNullAnnotation",
      name = "Public method parameters should have @NonNull annotation",
      description = "所有 public 方法的引用类型参数必须标注 @NonNull（如适用）",
      priority = Priority.MAJOR,
      tags = {"null-safety"})
public class MissingNonNullAnnotationRule 
        extends IssuableSubscriptionVisitor {

    @Override
    public List<Kind> nodesToVisit() {
        return List.of(Kind.METHOD);
    }

    @Override
    public void visitNode(Tree tree) {
        MethodTree method = (MethodTree) tree;
        
        // 只检查 public 方法
        if (!method.symbol().isPublic()) return;
        
        // 检查每个参数
        for (VariableTree param : method.parameters()) {
            TypeTree type = param.type();
            // 跳过原始类型（int, long 等）
            if (type.is(Kind.PRIMITIVE_TYPE)) continue;
            
            // 检查是否有 @NonNull 或 @Nullable 注解
            boolean hasNullAnnotation = param.modifiers().annotations().stream()
                .anyMatch(a -> a.annotationType().toString()
                    .matches(".*(NonNull|Nullable|NotNull).*"));
            
            if (!hasNullAnnotation) {
                reportIssue(param.simpleName(),
                    "缺少空安全注解: 参数 '" + param.simpleName().name() + 
                    "' 应标注 @NonNull 或 @Nullable");
            }
        }
    }
}
```

## 4. 项目总结

### 4.1 自定义规则开发要点

| 概念 | 说明 |
|------|------|
| `@Rule` | 标记一个类为规则，指定 key、name、priority |
| `JavaFileScanner` | 规则的入口接口——`scanFile()` 被 Scanner 调用 |
| `BaseTreeVisitor` | AST 访问器基类——覆写 `visit*` 方法检查特定语法节点 |
| `context.reportIssue()` | 在指定 AST 节点上报告一个 Issue |
| `CheckVerifier` | 测试框架——验证规则在测试文件上的行为 |

### 4.2 注意事项

1. **规则 key 必须全局唯一**：使用带命名空间前缀的 key（如 `company-java:NoDaoInController`）。
2. **自定义规则必须经过充分测试**：在 2-3 个真实项目上试跑，收集误报反馈，调整规则逻辑后再推广。
3. **规则版本与 SonarQube 兼容**：每次 SonarQube 主版本升级需重新编译插件。
4. **规则描述要包含示例**：开发者看到 Issue 时需要理解"为什么这么写不行"以及"怎么写才对"——好的描述有违规示例和合规示例。
5. **Severity 分级要慎重**：不要把所有自定义规则都设为 Blocker——"狼来了"会损害规则的权威性。建议：架构违规=Major，安全漏洞=Blocker，代码风格=Minor。
6. **性能优化原则**：`visitNode` 中避免调用外部服务或复杂 I/O 操作；优先使用 SubscriptionVisitor 而非完整 TreeVisitor（只订阅感兴趣的节点类型）。
7. **插件的 SonarLint 支持**：如果希望开发者在 IDE 中实时看到规则提示，需要在打包时设置 `<sonarLintSupported>true</sonarLintSupported>`。

### 4.3 规则开发排错指南

| 症状 | 可能原因 | 排查方法 |
|------|---------|---------|
| 规则在测试中通过但实际扫描无 Issue | 规则未在 Quality Profile 中激活 | 检查 Profile 中是否有该规则 |
| 规则产生了大量误报 | 规则逻辑过于宽泛 | 增加更多上下文判断（包名、类注解、方法修饰符等）|
| 上传插件后 SonarQube 启动失败 | 插件版本与 SQ 版本不兼容 | 检查 sonar-plugin-api 版本是否匹配 |
| 测试中 verifyIssues 报 no issue found | 测试文件路径/内容不对 | 确认 `Noncompliant` 注释格式 |
| 扫描时间显著增加 | 规则中做了重计算操作 | 检查 visitNode 中是否有不必要的循环 |

### 4.4 自定义规则开发清单（Checklist）

```
□ 需求明确：规范是否适合自动化？（有些规范属于设计层面，不适合静态分析）
□ 选择技术路线：SSLR Visitor / Checks API / 自定义 Sensor
□ 编写规则类（@Rule 注解 + 核心逻辑）
□ 编写单元测试（覆盖：违规、合规、边界情况）
□ 编写规则元数据（description含示例、remediation时间估算）
□ 注册规则（MyJavaRulesPlugin + MyJavaRulesDefinition）
□ 打包部署验证（mvn package → 部署 → 扫描验证）
□ 误报评估（在2-3个真实项目上试跑，误报率 < 10%）
□ 灰度推广（试点团队先激活，收集反馈后全员推广）
□ 文档归档（规则背景、决策理由、维护记录）
```

### 4.5 思考题

1. 如果一条自定义规则在 2 个项目上产生了 500 个 Issue，但团队评估后发现其中 60% 是误报——你应该怎样调整规则？
2. AST Visitor 模式和正则表达式模式在代码检查中各有什么优劣？
3. 你需要检查"所有 REST API 的返回类型必须是 `ResponseEntity<T>` 而非裸类型"。这是属于语法检查还是语义检查？你会选择哪种实现方式？
4. 公司的编码规范有 50 条，全部做成自定义规则可行吗？请分析"规则性价比"——哪些适合自动化，哪些保留在 Code Review 文档中？

> **答案提示**：第3题这是语义检查——需要解析返回类型的泛型参数，涉及类型系统和符号解析，建议用 Checks API。第4题评估标准：技术可行性（是否能用 AST 检测到）、误报率（规则越精确误报越低）、修复频率（团队是否真的经常违反这条规范）——通常 50 条规范中只有 10-15 条适合做成自动化规则，其余保留在文档中。
