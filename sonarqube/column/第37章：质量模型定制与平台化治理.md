# 第37章：质量模型定制与平台化治理

## 1. 项目背景

**业务场景**：某集团经过 1 年的 SonarQube 推广，已将 500+ 项目全部接入。但现在面临"大锅饭"困境——同一个 Quality Gate 适用于所有项目，无论该项目是核心交易系统还是内部管理工具。架构师认为需要精细化的质量模型——不同系统等级有不同的质量要求，质量豁免有规范的审批流程，新项目创建自动继承团队的规则和门禁。

这需要将 SonarQube 从"点状工具"升级为"企业质量平台"——通过 Web API 二次开发、项目分级管理、治理流程标准化，实现对 500+ 项目的精细化质量管控。

**痛点放大**：

- 500+ 项目的人工管理不可持续，必须自动化
- 质量豁免（Won't Fix / False Positive）需要审批流，否则滥用
- 新项目接入时需要自动创建项目、应用权限模板、绑定 Quality Gate、设置 New Code Period
- 质量平台的状态数据需要开放给其他系统（如项目管理工具、BI 报表）
- **治理流程的审计追溯缺失**：豁免审批、规则变更、权限修改——这些操作都需要可追溯。谁在什么时间做了什么、为什么——如果没有审计日志，合规审计时会很被动
- **不同团队对"质量"的定义不一致**：支付团队认为"安全第一"，数据团队认为"性能第一"——单一的质量模型无法满足所有场景
- **平台 API 的稳定性与版本兼容性**：每次 SonarQube 升级都可能导致自定义平台服务因 API 变更而中断——需要 API 兼容性测试套件
- **数据所有权和安全边界模糊**：500+ 项目的质量数据集中在一个平台上——谁有权看哪些项目的数据？如何防止跨团队数据泄露？

**大师**："基于 SonarQube Web API 构建一个自动化服务（Quality Platform Service）。它可以是一个 Python/Go/Java 应用，提供以下能力：

1. **项目生命周期管理**：创建项目 → 应用模板 → 更新设置 → 归档清理
2. **质量豁免审批**：开发者申请 Won't Fix / False Positive → Tech Lead 审批 → 自动标记
3. **质量报表聚合**：跨项目的指标采集和汇总"

**小白**："审批流怎么和 SonarQube 打通？SonarQube 自身没有审批功能。"

**大师**："审批在质量平台层实现。流程是：
1. 开发者在 SonarQube 中标记 Issue 为 Won't Fix，并在 Comment 中填写 '#waiver-request' 标签
2. 质量平台定时扫描带有 '#waiver-request' 标签的 Issue
3. 质量平台向 Tech Lead 发送审批通知（邮件/IM）
4. Tech Lead 在平台后台点'批准'或'拒绝'
5. 平台通过 API 正式标记 Issue，并追加 Comment 'Approved by Tech Lead on 2024-xx-xx'"

**小胖**："我们的质量平台上线后，架构师团队想根据不同业务域定制质量模型——支付域关注安全+可靠性、数据域关注性能+可维护性。SonarQube 的 Quality Profile 能做到这个粒度吗？"

**大师**："可以，但需要通过'Profile + Gate 组合'来实现领域化质量模型：

```
支付域质量模型:
  Quality Profile: "Payment Java Profile"（包含额外的安全规则）
  Quality Gate: "P0 - Core System Gate"（严格门禁）
  自定义指标: security_rating, new_vulnerabilities, owasp_top10

数据域质量模型:
  Quality Profile: "Data Java Profile"（包含性能相关规则）
  Quality Gate: "P1 - Business System Gate"（标准门禁）
  自定义指标: sqale_debt_ratio, duplicated_blocks, complexity_distribution
```

具体的 Profile 差异化：在 SonarQube 中创建两个 Java Profile——一个激活了所有安全相关规则（用于支付域），另一个额外激活了性能相关规则（用于数据域）。然后通过 API 按项目 Key 前缀自动分配：

```python
class ProfileManager:
    """按业务域自动分配 Quality Profile"""
    
    DOMAIN_PROFILE_MAP = {
        "com.company.payment": "Payment Java Profile",
        "com.company.trading": "Payment Java Profile",  # 同支付域
        "com.company.data": "Data Java Profile",
        "com.company.infra": "Infra Java Profile",       # 基础设施域
    }
    
    def assign_profile(self, project_key):
        # 根据项目 Key 前缀匹配域
        for prefix, profile_name in self.DOMAIN_PROFILE_MAP.items():
            if project_key.startswith(prefix):
                # 调用 API 为项目关联 Profile
                requests.post(
                    f"{self.sonar_url}/api/qualityprofiles/add_project",
                    params={
                        "language": "java",
                        "qualityProfile": profile_name,
                        "project": project_key
                    },
                    headers=HEADERS
                )
                return profile_name
        # 默认 Profile
        return "Default Java Profile"
```

这样做的好处：每个域的质量模型独立演进——支付域新增了安全规则不会影响数据域的项目。"

**小白**："质量豁免审批流里面有一个很实际的问题：如果一个 Issue 被标记为 Won't Fix，会不会在下次扫描时被重新打开？SonarQube 的行为是怎样的？"

**大师**："这是最容易踩的坑。SonarQube 的 Won't Fix 规则必须理解清楚：

**Won't Fix 的持久性**：一旦标记为 Won't Fix（或 False Positive），只要满足以下条件就不会被重新打开：
- Issue 所在文件的**代码行没有变化**
- Issue 的**规则 key 没有变化**

但如果：
1. **你修改了这一行代码**（即使只是加了一个空格）→ Issue 被重新打开
2. **你重命名了文件** → Issue 被重新打开
3. **规则 key 发生了变化**（如升级插件后规则重命名）→ Issue 被重新打开

所以，在质量平台层面要做豁免的"持久化保护"：

```python
class WaiverPersistenceManager:
    """豁免持久化管理——防止 Won't Fix 被意外重新打开"""
    
    def scan_reopened_waivers(self):
        """扫描被重新打开的豁免 Issue"""
        resp = requests.get(
            f"{self.sonar_url}/api/issues/search",
            params={
                "statuses": "OPEN,CONFIRMED,REOPENED",
                "tags": "waiver-approved"
            },
            headers=HEADERS
        )
        reopened = resp.json().get("issues", [])
        
        for issue in reopened:
            # 检查是否在审批白名单中
            waiver = self.db.find_waiver(issue['key'])
            if waiver and waiver['status'] == 'approved':
                # 自动重新标记
                self.re_apply_waiver(issue['key'], waiver['approver'])
                print(f"🔄 重新应用豁免: {issue['key']} (审批人: {waiver['approver']})")
        
        return len(reopened)
    
    def re_apply_waiver(self, issue_key, approver):
        """重新应用已过期的豁免标记"""
        requests.post(
            f"{self.sonar_url}/api/issues/bulk_change",
            params={
                "issues": issue_key,
                "set_type": "FALSE_POSITIVE",
                "do_transition": "wontfix",
                "add_tags": "waiver-approved,auto-reapplied"
            },
            headers=HEADERS
        )
```

这个脚本应该在质量平台的定时任务中执行——每 24 小时扫描一次，自动重新应用被意外重新打开的豁免。"

```python
#!/usr/bin/env python3
"""质量平台 - 项目自动开户服务"""

import requests
import os

SONAR_URL = os.getenv("SONAR_URL")
TOKEN = os.getenv("SONAR_TOKEN")
HEADERS = {"Authorization": f"Bearer {TOKEN}"}

class ProjectManager:
    def __init__(self):
        self.sonar_url = SONAR_URL

    def create_project(self, project_key, project_name, level):
        """创建项目并应用对应等级的配置"""
        # 1. 创建项目
        resp = requests.post(
            f"{self.sonar_url}/api/projects/create",
            params={"project": project_key, "name": project_name,
                    "visibility": "private"},
            headers=HEADERS
        )
        if resp.status_code != 200:
            return {"error": f"创建项目失败: {resp.text}"}

        # 2. 设置 New Code Period
        requests.post(
            f"{self.sonar_url}/api/new_code_periods/set",
            params={"project": project_key,
                    "type": "NUMBER_OF_DAYS", "value": "30"},
            headers=HEADERS
        )

        # 3. 根据等级分配 Quality Gate
        gate_map = {
            "P0": "P0 - Core System Gate",
            "P1": "P1 - Business System Gate",
            "P2": "P2 - Internal Tool Gate",
            "P3": "P3 - Advisory Only"
        }
        gate_name = gate_map.get(level, "P2 - Internal Tool Gate")
        requests.post(
            f"{self.sonar_url}/api/qualitygates/select",
            params={"gateName": gate_name, "projectKey": project_key},
            headers=HEADERS
        )

        return {
            "status": "success",
            "project_key": project_key,
            "gate": gate_name
        }


if __name__ == "__main__":
    pm = ProjectManager()
    result = pm.create_project(
        "com.company:new-payment-service",
        "New Payment Service",
        "P0"
    )
    print(result)
```

**步骤 2：质量豁免审批流程**

```python
class WaiverManager:
    """质量豁免管理"""

    def find_pending_waivers(self):
        """查找待审批的豁免申请"""
        resp = requests.get(
            f"{self.sonar_url}/api/issues/search",
            params={
                "statuses": "OPEN,CONFIRMED",
                "tags": "waiver-request"
            },
            headers=HEADERS
        )
        return resp.json().get("issues", [])

    def approve_waiver(self, issue_key, approver):
        """批准豁免"""
        # 移除 waiver-request 标签
        requests.post(
            f"{self.sonar_url}/api/issues/bulk_change",
            params={
                "issues": issue_key,
                "remove_tags": "waiver-request",
                "add_tags": "waiver-approved",
                "set_type": "FALSE_POSITIVE",
                "do_transition": "wontfix"
            },
            headers=HEADERS
        )
        # 添加审批评论
        requests.post(
            f"{self.sonar_url}/api/issues/add_comment",
            params={
                "issue": issue_key,
                "text": f"豁免已批准，审批人: {approver}"
            },
            headers=HEADERS
        )
        return {"status": "approved"}

    def reject_waiver(self, issue_key, reviewer):
        """拒绝豁免"""
        requests.post(
            f"{self.sonar_url}/api/issues/bulk_change",
            params={
                "issues": issue_key,
                "remove_tags": "waiver-request",
                "add_tags": "waiver-rejected"
            },
            headers=HEADERS
        )
        requests.post(
            f"{self.sonar_url}/api/issues/add_comment",
            params={
                "issue": issue_key,
                "text": f"豁免已拒绝，需修复。审核人: {reviewer}"
            },
            headers=HEADERS
        )
        return {"status": "rejected"}
```

**步骤 3：质量聚合报表**

```python
class ReportGenerator:
    """质量数据聚合"""

    def generate_org_report(self):
        """生成组织级质量周报"""
        # 获取所有项目
        projects = requests.get(
            f"{self.sonar_url}/api/projects/search?ps=500",
            headers=HEADERS
        ).json().get("components", [])

        report = {
            "total_projects": len(projects),
            "gate_pass_rate": 0,
            "avg_coverage": 0,
            "top_risks": []
        }

        for proj in projects:
            key = proj["key"]
            # 获取 Quality Gate 状态
            gate = requests.get(
                f"{self.sonar_url}/api/qualitygates/project_status",
                params={"projectKey": key},
                headers=HEADERS
            ).json().get("projectStatus", {})
            if gate.get("status") == "OK":
                report["gate_pass_rate"] += 1

            # 获取关键指标
            measures = requests.get(
                f"{self.sonar_url}/api/measures/component",
                params={
                    "component": key,
                    "metricKeys": "coverage,new_bugs,security_rating"
                },
                headers=HEADERS
            ).json().get("component", {}).get("measures", [])
            for m in measures:
                if m["metric"] == "new_bugs" and int(m.get("value", 0)) > 0:
                    report["top_risks"].append({
                        "project": key,
                        "new_bugs": m["value"]
                    })

        report["gate_pass_rate"] = \
            report["gate_pass_rate"] / len(projects) * 100
        return report
```

class PermissionManager:

    ROLE_PERMISSIONS = {
        "developer": {
            "can_view_own_projects": True,
            "can_request_waiver": True,

            "can_view_org_reports": False,

            "can_approve_waiver": True,
            "can_manage_rules": False,
            "can_view_org_reports": True,
        },
        "architect": {
            "can_view_all_projects": True,
            "can_approve_waiver": True,
            "can_manage_rules": True,
            "can_manage_quality_gates": True,
        },
        "admin": {
            "can_all": True
        }
    }
    
    def check_permission(self, user_role, action):
        if user_role not in self.ROLE_PERMISSIONS:
            return False
        perms = self.ROLE_PERMISSIONS[user_role]
        return perms.get("can_all", False) or perms.get(action, False)

class PlatformHealthChecker:
    """平台运行健康检查"""
    
    def check_all(self):
        checks = {
            "sonarqube_reachable": self._check_sq_connectivity(),
            "database_ok": self._check_db_connectivity(),
            "api_rate_limit": self._check_api_rate(),
            "waiver_queue_size": self._check_pending_waivers(),
            "project_count_match": self._verify_project_count(),
        }
        
        # 如果有任何检查失败，发送告警
        failed = [k for k, v in checks.items() if not v]
        if failed:
            self._send_alert(f"质量平台健康检查失败: {failed}")
        
        return checks
    
    def _check_sq_connectivity(self):
        try:
            resp = requests.get(f"{SONAR_URL}/api/system/health", timeout=5)
            return resp.status_code == 200
        except:
            return False
    
    def _check_pending_waivers(self, max_pending=50):
        """检查待审批豁免是否堆积"""
        count = self.db.count_pending_waivers()
        return count < max_pending  # 超过 50 个待审批说明审批瓶颈

## 4. 项目总结

### 4.1 质量平台核心能力矩阵

| 能力 | 实现方式 | 依赖 |
|------|---------|------|
| 项目自动开户 | Web API 创建 + 模板应用 | SonarQube API |
| 质量豁免审批 | 标签扫描 + 审批工作流 | 数据库 + 通知系统 |
| 质量报表 | 指标采集 + 聚合 | Web API |
| 权限治理 | Permission Template API | SonarQube API |
| 规则推广通知 | Webhook + IM | SonarQube Webhook |

### 4.2 注意事项

1. **审批记录要持久化**：不要依赖 SonarQube 的 Issue Comment 作为审批记录的唯一存储——建立独立的审批数据库。
2. **API 频率控制**：大规模轮询（如每分钟扫描 500 个项目）会压垮 SonarQube。建议频率 ≤ 30 分钟/次。
3. **令牌安全管理**：质量平台使用专用的高权限 Token，该 Token 必须存储在 Vault/Secrets Manager 中。
4. **豁免审批的"四眼原则"**：审批人和申请人不能是同一个人——这个校验必须在质量平台层实现，SonarQube 不会帮你做。
5. **项目生命周期管理**：建立项目归档/删除策略——超过 90 天无扫描的项目自动标记为"休眠"，180 天无扫描的项目自动归档（备份后从 SonarQube 中删除以释放资源）。
6. **API 变更监控**：SonarQube 升级后，第一时间运行 API 兼容性测试套件——验证平台所有功能是否正常。
7. **质量平台也要有 Quality Gate**：平台自身的代码质量、测试覆盖率、部署流程——也应该受到门禁约束。"管质量的平台如果自己没有质量门禁，凭什么说服别人？"

### 4.3 平台化治理成熟度路线图

|------|------|---------|------|

| L2 脚本化 | Shell/Python 脚本 | API 批量操作、自动采集 | 3-6 月 |

### 4.5 平台 API 设计最佳实践

```python
# 质量平台 API 设计示例 (FastAPI)
from fastapi import FastAPI, Depends, HTTPException
from pydantic import BaseModel

app = FastAPI(title="Quality Platform API")

class ProjectCreateRequest(BaseModel):
    project_key: str
    project_name: str
    quality_level: str  # P0, P1, P2, P3
    domain: str         # payment, data, infra
    team: str

@app.post("/api/v1/projects")
async def create_project(req: ProjectCreateRequest, user=Depends(get_current_user)):
    """创建项目并自动配置质量门禁"""
    if not permission_manager.check_permission(user.role, "create_project"):
        raise HTTPException(status_code=403)
    
    result = project_manager.create_project(
        req.project_key, req.project_name, req.quality_level
    )
    
    # 记录审计日志
    audit_log.record("project_created", req.project_key, user.username)
    
    # 发送通知
    webhook_manager.send("new_project_created", req.project_key, result)
    
    return result

@app.get("/api/v1/waivers/pending")
async def get_pending_waivers(user=Depends(get_current_user)):
    """获取待审批的豁免列表"""
    if not permission_manager.check_permission(user.role, "approve_waiver"):
        raise HTTPException(status_code=403)
    return waiver_manager.find_pending_waivers()

@app.post("/api/v1/waivers/{issue_key}/approve")
async def approve_waiver(issue_key: str, user=Depends(get_current_user)):
    """批准豁免"""
    if waiver_manager.is_requester(issue_key, user.username):
        raise HTTPException(status_code=400, detail="不能批准自己的豁免申请")
    return waiver_manager.approve_waiver(issue_key, user.username)
```

### 4.6 思考题

1. 如果质量豁免审批人和申请人是同一个人（如 Tech Lead 自己申请 Won't Fix），审批流应该怎么设计，防止"自己批准自己"？
2. 质量平台需要知道项目是"活跃"还是"已废弃"——如何通过 SonarQube API 判断项目最近是否有扫描活动？
3. 质量平台收到 500+ 项目的质量数据，但 CEO 要求每天出一份"一句话质量摘要"——你需要用哪 3 个指标来代表整个组织的代码质量？为什么？
4. 如果 SonarQube 的 Web API 在某个版本中改变了返回 JSON 的字段名（比如 `componentKey` 变成了 `projectKey`），你的质量平台如何避免因此崩溃？

> **答案提示**：第3题三个指标：门禁通过率（P0项目）、新增 Blocker 安全问题数、覆盖率趋势——这是安全+可靠性+测试投入的最简表达。第4题方案：① 在平台中封装一层 API Client，所有 API 调用经过这个 Client；② 对于每个 API 响应做 schema 校验（pydantic/marshmallow）；③ 升级 SonarQube 前在测试环境运行 API 兼容性测试套件。
