# 第36章：外部分析结果导入与安全工具联动

## 1. 项目背景

**业务场景**：某安全团队使用 Semgrep 做深度 SAST 扫描，发现了一批 SonarQube 未能检测到的安全问题。但目前工作流是碎片化的——开发者需要查看 SonarQube（日常质量）+ Semgrep 报告（安全）+ OWASP Dependency-Check（依赖漏洞）三份独立报告。

质量负责人希望将所有的质量数据统一展示在 SonarQube 中——包括外部分析工具的结果。这样开发者在 PR 页面就能看到所有问题，不需要切换多个工具。

**痛点放大**：

- **多工具多报告**：开发者需要在 3-4 个工具间切换，信息孤岛
- **外部工具结果无法参与 Quality Gate**：Semgrep 发现了问题但不能在 CI 门禁中自动阻断
- **缺少统一视图**：管理层无法在一处看到全局质量状况

**更多现实场景**：

- **场景一：Semgrep 发现 SQL 注入但 SonarQube 没检测到**：安全团队使用 Semgrep 的 `p/sql-injection` 规则集扫描一个 Spring Boot 项目，发现了 3 个高危 SQL 注入漏洞（动态拼接 SQL 语句）。但 SonarQube 的 Java 规则集中没有覆盖这类模式——结果就是：CI 质量门禁"通过"了，但安全团队说"不能上线"。两套标准冲突，开发者夹在中间无所适从。

- **场景二：依赖漏洞检查结果无处安放**：OWASP Dependency-Check 扫描出了 15 个高危 CVE（包括 Log4Shell）。安全团队发了一封邮件列出这些漏洞，但开发者说过了一周"没看到邮件"。如果有统一的质量视图，这些 CVE 直接显示在 SonarQube 项目主页上，开发者每天看代码质量时就能看到。

- **场景三：自定义团队规范无法融入标准流程**：架构团队制定了 20 条编码规范（如"禁止使用 System.out.println"、"Service 层不能直接调用 DAO 层"）。他们写了一个简单的 grep 脚本检查这些规则，但结果只有控制台输出，无法和 SonarQube 的 Issue 一起在 PR 页面展示。

**外部工具导入的核心价值**：
1. **统一质量视图**：所有类型的 Issue（代码缺陷、安全漏洞、依赖风险、团队规范）在一个页面展示
2. **统一门禁流程**：外部工具的发现可以影响 CI 门禁（需要额外脚本支持）
3. **降低工具切换成本**：开发者只需要关注一个平台
4. **可追溯**：每次扫描的外部 Issue 都有关联的代码版本和时间戳

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（打开三个浏览器窗口——SonarQube、Semgrep Dashboard、Dependency-Check Report）："大师，每次代码审查我要看三个工具的扫描结果——SonarQube 的 Issue、Semgrep 的安全报告、还有依赖检查的漏洞列表。三个工具三个页面，累死了。"

**大师**："你可以把外部工具的结果导入 SonarQube——这样开发者在 SonarQube 一个地方就能看到所有问题。SonarQube 支持两种导入方式：

1. **Generic Issue Import**：通用格式——将任何外部工具的输出转换为 SonarQube 的标准 Issue JSON 格式，通过 API 导入。
2. **Web API 批量导入**：通过 `api/issues/import` 上传符合格式的 Issue 列表。"

**小白**："那这些外部 Issue 的严重级别怎么映射？Semgrep 的 'ERROR' 等同于 SonarQube 的 'Blocker' 吗？"

**大师**："需要自己建立映射关系。导入时指定每个外部 Issue 对应的规则 Key、严重级别、类型。推荐做法是：为每个外部工具创建独立的前缀规则（如 `semgrep:sql-injection`），这样在 Quality Profile 中可以独立管理。"

**小白**："那这些外部 Issue 导入后，我怎么区分哪些是 SonarQube 自己发现的、哪些是外部工具导入的？会不会混在一起分不清？"

**大师**："有几个区分维度：

**1. 规则 Key 前缀**：这是最直观的方式。为每个外部工具创建独立的规则仓库和规则 Key：
- SonarQube 内建规则：`java:S1234`（标准命名）
- ESLint 导入：`eslint:no-unused-vars`
- Semgrep 导入：`semgrep:sql-injection`
- SpotBugs 导入：`spotbugs:DM_DEFAULT_ENCODING`

**2. Issue 标签（Tags）**：导入时给 Issue 添加标签，方便在 Web UI 中筛选：
```python
issue = {
    "engineId": "semgrep",
    "ruleId": "semgrep:sql-injection",
    "severity": "BLOCKER",
    "type": "VULNERABILITY",
    "tags": ["external", "semgrep", "security"],  # 添加标签
    ...
}
```

**3. 创建独立的 Quality Profile**：如果你希望外部工具的规则和 SonarQube 内建规则完全独立管理，可以创建一个新的 Quality Profile 专门用于外部规则——但它不能和内建 Profile 合并到同一个项目中。更好的方式是在同一个 Profile 中同时激活内建规则和外部规则，通过规则 Key 前缀区分来源。"

**小胖**："我之前试过导入 ESLint 结果，但发现有些 Issue 导入后和 SonarQube 自己报的重复了——比如 ESLint 报了 `no-console`，SonarQube 也报了同一条规则。怎么去重？"

**大师**："这是外部导入中最难处理的问题。去重策略有三个层次：

**层次 1：规则层面去重（推荐）**：在 SonarQube 中禁用与外部工具重叠的内建规则。

例如，如果你用 ESLint 做前端代码检查，可以在 SonarQube 的 Quality Profile 中把等价的 JavaScript 规则设置为"Deactivated"。这样 Scanner 不会报，只有 ESLint 导入的 Issue 会显示。

**层次 2：导入脚本中去重**：在转换脚本中，维护一个"排除的规则列表"——这些规则 SonarQube 已经覆盖了，不需要再导入：

```python
# 排除规则：SonarQube 已覆盖的规则
EXCLUDED_ESLINT_RULES = {
    'no-unused-vars',       # SQ 内建规则 S1481
    'no-undef',             # SQ 内建规则 S3827
    'eqeqeq',               # SQ 内建规则 S124
    'no-eval',              # SQ 内建规则 S1523
}

# 转换时过滤
if msg.get("ruleId") in EXCLUDED_ESLINT_RULES:
    continue  # 跳过，避免重复
```

**层次 3：Issue 层面去重**：基于文件路径 + 行号 + 规则类型的 Hash 去重。如果外部 Issue 和数据库中的 Issue 在同一位置、同一类型，跳过导入。但这需要先查询数据库，实现较复杂，不推荐作为首选方案。"

**小白**："还有个问题——外部 Issue 导入后不会自动清理。每次 CI 跑一次就导入一次，那旧 Issue 怎么办？"

**大师**："这是外部 Issue 导入最大的工程挑战。SonarQube 内建 Issue 会自动追踪状态——修复后自动标记为 CLOSED。但通过 API 导入的外部 Issue 没有这个能力——它们只是被插入到数据库中，不会自动更新或清理。

**推荐的自动清理方案**——每次导入前先清理同一个 engineId 的旧 Issue：

```python
#!/usr/bin/env python3
"""导入前先清理，避免僵尸 Issue 累积"""
import requests
import json
import sys

SONAR_URL = "http://localhost:9000"
SONAR_TOKEN = "your-token"
PROJECT_KEY = "com.company:my-service"
ENGINE_ID = "semgrep"

# 1. 查询当前 engineId 下的所有已存在 Issue
resp = requests.get(
    f"{SONAR_URL}/api/issues/search",
    auth=(SONAR_TOKEN, ""),
    params={
        "projectKeys": PROJECT_KEY,
        "tags": "external," + ENGINE_ID,  # 通过标签筛选
        "statuses": "OPEN,CONFIRMED,REOPENED",
        "ps": 500
    }
)

existing_issues = resp.json().get("issues", [])
print(f"发现 {len(existing_issues)} 个旧的 {ENGINE_ID} Issue")

# 2. 批量标记为已修复（而不是删除，以保留历史记录）
# 注意：外部 Issue 不支持 Transition API，变通方案是用规则标记
for issue in existing_issues:
    # 可选：如果 Issue 确实已修复，通过 API Transition 关闭
    # 但由于外部 Issue 的特性，最简单的方式是依赖标签和状态管理
    pass

# 3. 导入新的 Issue 列表
with open("semgrep-sonar.json") as f:
    new_issues = json.load(f)

resp = requests.post(
    f"{SONAR_URL}/api/issues/import",
    auth=(SONAR_TOKEN, ""),
    headers={"Content-Type": "application/json"},
    json=new_issues
)

print(f"导入完成: {resp.status_code}")
```

更好的做法是——在导入脚本中，对有状态的 Issue 使用 `api/issues/do_transition` 关闭旧的，然后导入新的。但这要求外部工具也能提供 Issue 修复状态（如 Semgrep 的 baseline 机制）。"

## 3. 项目实战

### 3.1 分步实现

**步骤 1：ESLint 结果导入**

生成 ESLint 报告：

```bash
npx eslint src --format json > eslint-report.json
```

转换为 SonarQube 的 Generic Issue Import 格式：

```python
#!/usr/bin/env python3
"""将 ESLint JSON 报告转换为 SonarQube Generic Issue Import 格式"""

import json
import sys

def convert_eslint_to_sonarqube(eslint_report_path):
    with open(eslint_report_path) as f:
        eslint_data = json.load(f)

    issues = []
    for file_report in eslint_data:
        file_path = file_report.get("filePath", "")
        for msg in file_report.get("messages", []):
            severity_map = {2: "MAJOR", 1: "MINOR", 0: "INFO"}
            issue = {
                "engineId": "eslint",
                "ruleId": msg.get("ruleId", "eslint-custom"),
                "severity": severity_map.get(msg.get("severity", 1), "MINOR"),
                "type": "CODE_SMELL",
                "primaryLocation": {
                    "message": msg.get("message", ""),
                    "filePath": file_path,
                    "textRange": {
                        "startLine": msg.get("line", 1),
                        "endLine": msg.get("endLine", msg.get("line", 1)),
                        "startColumn": msg.get("column", 0),
                        "endColumn": msg.get("endColumn", msg.get("column", 0))
                    }
                }
            }
            issues.append(issue)

    return {"issues": issues}

if __name__ == "__main__":
    result = convert_eslint_to_sonarqube("eslint-report.json")
    with open("sonarqube-generic-issues.json", "w") as f:
        json.dump(result, f, indent=2)
    print(f"✅ 转换了 {len(result['issues'])} 个 Issue")
```

**步骤 2：通过 API 导入到 SonarQube**

```bash
# 导入通用 Issue
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/issues/import" \
  -H "Content-Type: application/json" \
  -d @sonarqube-generic-issues.json
```

**步骤 3：Semgrep 结果导入**

```bash
# 运行 Semgrep 并生成 JSON 报告
semgrep --config=auto --json -o semgrep-report.json .

# 转换并导入
python3 convert_semgrep_to_sonarqube.py semgrep-report.json > semgrep-sonar.json
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/issues/import" \
  -H "Content-Type: application/json" \
  -d @semgrep-sonar.json
```

Semgrep 映射关系：

| Semgrep Severity | SonarQube Severity | 类型 |
|-----------------|-------------------|------|
| ERROR | BLOCKER | BUG or VULNERABILITY |
| WARNING | MAJOR | BUG or CODE_SMELL |
| INFO | MINOR | CODE_SMELL |

**步骤 4：SpotBugs 结果导入**

```bash
# Maven 项目运行 SpotBugs
mvn spotbugs:spotbugs

# 转换 SpotBugs XML 报告
python3 convert_spotbugs_to_sonarqube.py target/spotbugsXml.xml > spotbugs-sonar.json
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/issues/import" \
  -H "Content-Type: application/json" \
  -d @spotbugs-sonar.json
```

**步骤 5：将外部结果集成到 CI**

```yaml

external-scan-import:
```

**步骤 6：OWASP Dependency-Check 结果导入**

Dependency-Check 的报告较大（通常包含数百个 CVE），不适合直接作为 Issue 导入。推荐做法是将摘要信息导入为项目级度量数据：

```bash
#!/bin/bash
# odc-to-metrics.sh - 将 OWASP Dependency-Check 结果转为 SonarQube 度量

# 运行 OWASP Dependency-Check
dependency-check --project my-service --scan ./ --format XML -o odc-report.xml

# 提取摘要数据
HIGH_CVSS=$(grep -c '<severity>HIGH</severity>' odc-report.xml)
CRITICAL_CVSS=$(grep -c '<severity>CRITICAL</severity>' odc-report.xml)
MEDIUM_CVSS=$(grep -c '<severity>MEDIUM</severity>' odc-report.xml)

# 通过 API 写入自定义度量
curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/custom_measures/create" \
  -d "metricKey=odc_critical_cve&value=$CRITICAL_CVSS" \
  -d "projectKey=com.company:my-service" \
  -d "description=OWASP Dependency-Check Critical CVEs"

curl -X POST -u admin:Sonar@2024Admin \
  "http://localhost:9000/api/custom_measures/create" \
  -d "metricKey=odc_high_cve&value=$HIGH_CVSS" \
  -d "projectKey=com.company:my-service"
```

**步骤 7：统一导入 Pipeline 设计**

将所有外部工具的导入集成到一个统一的脚本中：

```bash
#!/bin/bash
# external-import-pipeline.sh - 统一外部分析结果导入

SONAR_URL="${SONAR_HOST_URL:-http://localhost:9000}"
SONAR_TOKEN="${SONAR_TOKEN:-admin:Sonar@2024Admin}"
PROJECT_KEY="$1"

if [ -z "$PROJECT_KEY" ]; then
    echo "用法: $0 <projectKey>"
    exit 1
fi

echo "=== 外部分析结果导入 Pipeline ==="
echo "项目: $PROJECT_KEY"
echo ""

# 1. ESLint（如果项目包含 JavaScript/TypeScript）
if [ -f "eslint-report.json" ]; then
    echo "[1/4] 导入 ESLint 结果..."
    python3 /scripts/convert_eslint.py eslint-report.json > /tmp/eslint-sonar.json
    
    RESP=$(curl -s -w "%{http_code}" -X POST -u "$SONAR_TOKEN" \
        "$SONAR_URL/api/issues/import" \
        -H "Content-Type: application/json" \
        -d @/tmp/eslint-sonar.json)
    echo "  ESLint 导入: HTTP $RESP"
fi

# 2. Semgrep（如果项目有 Semgrep 报告）
if [ -f "semgrep-report.json" ]; then
    echo "[2/4] 导入 Semgrep 结果..."
    python3 /scripts/convert_semgrep.py semgrep-report.json > /tmp/semgrep-sonar.json
    
    RESP=$(curl -s -w "%{http_code}" -X POST -u "$SONAR_TOKEN" \
        "$SONAR_URL/api/issues/import" \
        -H "Content-Type: application/json" \
        -d @/tmp/semgrep-sonar.json)
    echo "  Semgrep 导入: HTTP $RESP"
fi

# 3. SpotBugs（如果项目有 SpotBugs 报告）
if [ -f "target/spotbugsXml.xml" ]; then
    echo "[3/4] 导入 SpotBugs 结果..."
    python3 /scripts/convert_spotbugs.py target/spotbugsXml.xml > /tmp/spotbugs-sonar.json
    
    RESP=$(curl -s -w "%{http_code}" -X POST -u "$SONAR_TOKEN" \
        "$SONAR_URL/api/issues/import" \
        -H "Content-Type: application/json" \
        -d @/tmp/spotbugs-sonar.json)
    echo "  SpotBugs 导入: HTTP $RESP"
fi

# 4. OWASP Dependency-Check（如果有）
if [ -f "odc-report.xml" ]; then
    echo "[4/4] 导入 OWASP Dependency-Check 摘要..."
    bash /scripts/import-odc-summary.sh "$PROJECT_KEY"
fi

echo ""
echo "=== 导入完成 ==="

# 验证导入结果
echo "验证导入..."
TOTAL_EXTERNAL=$(curl -s -u "$SONAR_TOKEN" \
    "$SONAR_URL/api/issues/search?projectKeys=$PROJECT_KEY&tags=external&statuses=OPEN" \
    | python3 -c "import sys,json;print(json.load(sys.stdin)['total'])")
echo "当前未关闭的外部 Issue 数: $TOTAL_EXTERNAL"
```

## 4. 项目总结

### 4.1 外部工具导入清单

| 工具 | 类型 | 导入方式 | 推荐导入 |
|------|------|---------|---------|
| ESLint | JS Lint | Generic Issue Import | 是 |
| Semgrep | SAST | Generic Issue Import | 是 |
| SpotBugs | Java 缺陷 | Generic Issue Import | 是（如不用内建规则） |
| OWASP Dependency-Check | SCA | 自定义度量导入（摘要） | 摘要级别 |
| PMD | Java Lint | Generic Issue Import | 可选（SQ 已有内建） |
| Checkstyle | Java 风格 | Generic Issue Import | 可选 |
| KubeLinter | K8s 配置 | Generic Issue Import | 是 |

### 4.2 Issue 严重级别映射速查

| 来源工具 | 来源级别 | SonarQube 级别 | 建议类型 |
|---------|---------|---------------|---------|
| ESLint | error (2) | MAJOR | CODE_SMELL |
| ESLint | warn (1) | MINOR | CODE_SMELL |
| Semgrep | ERROR | BLOCKER | VULNERABILITY |
| Semgrep | WARNING | MAJOR | BUG |
| Semgrep | INFO | MINOR | CODE_SMELL |
| SpotBugs | Scariest (1) | BLOCKER | BUG |
| SpotBugs | Scary (2) | MAJOR | BUG |
| SpotBugs | Troubling (3) | MINOR | CODE_SMELL |
| Checkstyle | error | MAJOR | CODE_SMELL |
| Checkstyle | warning | MINOR | CODE_SMELL |

### 4.3 外部 Issue 生命周期管理

外部 Issue 与内建 Issue 的核心区别：

| 特性 | 内建 Issue | 外部导入 Issue |
|------|-----------|--------------|
| 自动清理 | ✅ 重新扫描时自动更新状态 | ❌ 需手动/脚本清理 |
| 状态追踪 | ✅ OPEN→CONFIRMED→CLOSED | 🟡 只支持基本状态 |
| Quality Gate 参与 | ✅ 直接参与门禁评分 | ❌ 不能直接参与 |
| 增量分析 | ✅ 自动计算 New Code 指标 | ❌ 不支持 |
| 规则详情 | ✅ 有规则描述和修复建议 | 🟡 取决于导入数据 |

**推荐的自动清理策略**：
1. 每次 CI 运行前，先通过 `api/issues/bulk_change` 清理旧的同一 engineId 的 Issue
2. 或在 Issue 导入时携带 `hash` 字段——SonarQube 通过 hash 去重

### 4.4 注意事项

1. **外部 Issue 的去重**：如果 SonarQube 内建规则已经覆盖了某个问题，外部导入会产生重复 Issue。在导入前做好规则映射，避免重叠规则的双重报告。
2. **外部 Issue 的清理**：外部导入的 Issue 不会像 SonarQube 内建问题那样在重新扫描时自动清理——需要每次导入前先清理旧的导入 Issue。
3. **Quality Gate 联动**：外部 Issue 不能直接参与 Quality Gate。如果需要门禁阻断，需通过 CI 脚本检查导入工具的输出。

### 4.5 思考题

1. 外部工具（如 Semgrep）发现了 50 个新 Issue，但你已经在 SonarQube 中配置了 600 条规则——这些 Issue 中哪些可能是重复的？如何避免重复？
2. 如果每次 CI 运行都导入外部 Issue，但外部 Issue 不会自动清理——久而久之 SonarQube 中会积累大量"僵尸 Issue"。如何设计自动清理机制？

---

> **推广计划提示**：外部分析结果导入是统一质量视图的关键。建议先从一个工具开始试点（如 ESLint），等流程跑通后再逐步接入 Semgrep、OWASP Dependency-Check 等。每种工具的结果导入脚本应保存在团队的 CI 仓库中。
