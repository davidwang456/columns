# 第12章：重复代码与复杂度治理

## 1. 项目背景

**业务场景**：某 SaaS 公司的订单管理模块在连续迭代 15 个版本后，代码行数从 3000 行膨胀到 12000 行。新来的开发者发现，同一个枚举值转换逻辑在 4 个不同的 Service 类中各写了一遍——每次需求变更时，开发者只改了自己发现的那份，导致系统中存在 4 种不同的"订单状态→中文描述"映射表。

一次客户投诉暴露了问题的严重性：用户看到的订单状态和后台数据库中的实际状态不一致——原因是有两份映射表没被同步更新。开发 Leader 打开 SonarQube 的 Duplications 面板，发现项目的重复率高达 18%（SonarQube 默认阈值是 3%）。

更糟糕的是，两个核心方法 `calculateTotalPrice()` 和 `calculateDiscountPrice()` 的逻辑几乎一模一样，但被复制到了两个微服务中。当产品经理调整了折扣算法后，只有一个服务的开发者修改了代码，另一个服务的逻辑仍按旧算法执行——导致不同入口的用户看到了不同的价格。

**痛点放大**：

- **重复代码=维护噩梦**：N 份重复代码意味着 N 个需要同步修改的地方，每次漏改都可能产生 Bug。
- **高复杂度=理解障碍**：一个认知复杂度 35 的方法，新开发者需要 30 分钟才能理解，而复杂度 5 的方法只需要 3 分钟。
- **主观性**：什么是"合理的重复"？什么是"该消除的重复"？标准不清晰。

## 2. 项目设计

### 剧本式交锋对话

---

**小胖**（打开 SonarQube 的 Duplications 面板，看到鲜红的 18%）："大师！我们的重复率 18%！这意味着我写的代码平均 5 行里就有 1 行是抄的？我有那么懒吗？"

**大师**："别急，先搞清楚 SonarQube 怎么算重复率。它不是简单的字符串匹配——它会在 token 化后进行比较。Token 化会把代码里的变量名、字面量、注释等都抽象成 token，然后找连续 10 个以上 token 的相同序列（默认至少 100 个 token）。

比如下面两段代码在 SonarQube 看来是重复的："

```java
// 文件 A
public double calculateOrderTotal(List<Order> orders) {
    double total = 0;
    for (Order order : orders) {
        total += order.getAmount() * order.getQuantity();
    }
    return total;
}

// 文件 B
public double calculateInvoiceTotal(List<Invoice> invoices) {
    double total = 0;
    for (Invoice inv : invoices) {
        total += inv.getPrice() * inv.getCount();
    }
    return total;
}
```

"尽管变量名不同，但 token 序列完全一致——都是 `double = 0; for ( X : X ) { += X * X } return`。这就是重复代码。"

**小白**："那什么是 Cognitive Complexity（认知复杂度）？它和 Cyclomatic Complexity（圈复杂度）有什么不同？"

**大师**："这是两个容易混淆的概念：

- **圈复杂度（Cyclomatic Complexity）**：数一数有多少个分支路径。`if(x) { a; } else { b; }` 的复杂度是 2（两条路径）。它度量的是'测试需要覆盖多少条路径'。
- **认知复杂度（Cognitive Complexity）**：度量的是'人阅读这段代码有多难'。它不仅考虑分支数量，还考虑嵌套深度、逻辑运算符、递归等。比如：

```java
// 圈复杂度 = 4，但认知复杂度更高（嵌套 + 逻辑运算符）
if (a > 0) {
    if (b > 0 && c > 0) {
        if (d > 0 || e > 0) {
            doSomething();
        }
    }
}
```

SonarQube 从 2017 年开始逐步用认知复杂度替代圈复杂度作为可维护性的主要度量标准。认知复杂度更贴合'代码好不好读'的直觉。"

**小胖**："那我的 `complexLogic` 方法复杂度 35，怎么降下来？把它拆成 7 个方法就行了吗？"

**大师**："拆方法可以降低每个方法的复杂度，但有前提——拆分要有业务意义。如果把一个方法无脑拆成 7 个，只是把复杂度从'一个方法的维度'移到了'一个类的维度'——整体可理解性没变。

正确的重构策略：
1. **提取 Guard Clause**：把前置校验提出来，减少主流程的 if 嵌套
2. **使用策略模式**：`if-else` 链通常是策略选择的信号
3. **提取纯函数**：把无副作用的计算逻辑提取成独立方法
4. **早期返回**：用 `return` 提前结束，减少 else 嵌套

关键是：**拆分是为了表达业务意图，不是为了降低数字**。"

**步骤 3：实战重构——消除重复**

**重构前**（两个 Service 中相似的方法）：

```java
// PaymentService.java
public double calculatePrice(List<OrderItem> items, double taxRate) {
    double subtotal = 0;
    for (OrderItem item : items) {
        subtotal += item.getPrice() * item.getQuantity();
    }
    double tax = subtotal * taxRate;
    double discount = subtotal > 100 ? subtotal * 0.1 : 0;
    return subtotal + tax - discount;
}

// InvoiceService.java（几乎相同的逻辑，只有折扣率不同）
public double calculateInvoiceTotal(List<InvoiceItem> items, double vatRate) {
    double subtotal = 0;
    for (InvoiceItem item : items) {
        subtotal += item.getPrice() * item.getQuantity();
    }
    double vat = subtotal * vatRate;
    double discount = subtotal > 200 ? subtotal * 0.15 : 0;
    return subtotal + vat - discount;
}
```

**重构后**（提取公共逻辑到工具类）：

```java
// PricingCalculator.java —— 公共类
public class PricingCalculator {
    public double calculate(List<PricedItem> items,
                            double taxRate,
                            double discountThreshold,
                            double discountRate) {
        double subtotal = items.stream()
            .mapToDouble(item -> item.getPrice() * item.getQuantity())
            .sum();
        double tax = subtotal * taxRate;
        double discount = subtotal > discountThreshold
            ? subtotal * discountRate : 0;
        return subtotal + tax - discount;
    }
}

// PricedItem.java —— 公共接口
public interface PricedItem {
    double getPrice();
    int getQuantity();
}

// OrderItem 和 InvoiceItem 实现 PricedItem 接口
public class OrderItem implements PricedItem { ... }
public class InvoiceItem implements PricedItem { ... }
```

重构后重新扫描，重复率应显著下降。

**步骤 4：降低认知复杂度**

**重构前**（复杂度 35 的方法）：

```java
public String getShipMethod(int weight, String dest, boolean isExpress,
                             boolean isHoliday, int distance) {
    String method;
    if (isExpress) {
        if (weight > 50) {
            method = "Freight";
        } else {
            if (dest.equals("Local")) {
                method = "Sameday";
            } else if (distance < 100) {
                method = "Overnight";
            } else {
                method = "2-Day";
            }
        }
    } else {
        if (isHoliday) {
            method = "Holiday Delivery";
        } else {
            if (weight < 10) {
                method = "Standard";
            } else if (weight < 30) {
                method = "Ground";
            } else {
                method = "Freight";
            }
        }
    }
    if (method == null) {
        method = "Standard";
    }
    return method;
}
```

**重构后**（复杂度降到 8）：

```java
public String getShipMethod(int weight, String dest, boolean isExpress,
                             boolean isHoliday, int distance) {
    if (isExpress) return getExpressMethod(weight, dest, distance);
    if (isHoliday) return "Holiday Delivery";
    return getStandardMethod(weight);
}

private String getExpressMethod(int weight, String dest, int distance) {
    if (weight > 50) return "Freight";
    if (dest.equals("Local")) return "Sameday";
    return distance < 100 ? "Overnight" : "2-Day";
}

private String getStandardMethod(int weight) {
    if (weight < 10) return "Standard";
    return weight < 30 ? "Ground" : "Freight";
}
```

关键手法：
- 用 Guard Clause（`if (isExpress) return ...`）消除嵌套
- 提取有意义的方法名（`getExpressMethod`、`getStandardMethod`）
- 用三元运算符替代简单 `if/else`

## 4. 项目总结

### 4.1 重复代码消除策略对比

| 策略 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 提取公共方法 | 同一类中的重复 | 改动最小 | 只在本地有效 |
| 提取工具类 | 跨类的同质逻辑 | 全局复用 | 需要设计接口或泛型 |
| 继承/模板方法 | 同一族类的骨架重复 | 语义清晰 | 增加了继承关系 |
| 策略模式 | 变化部分的 if/else 重复 | 可扩展 | 学习成本高 |

### 4.2 适用场景

- **多模块项目**：跨模块重复是最大的技术债务来源
- **快速迭代的早期产品**：MVP 阶段常出现"先复制一份改改"的临时方案
- **同一套逻辑以不同语言实现**：如 Java 和 JavaScript 同时实现了相同的校验函数

**不适用场景**：
- 看似重复但业务语义不同的代码（"两个计算器长得很像但计算公式不同"）
- 框架强制要求的重复模式（如 Controller 中的 `@RequestMapping`）

### 4.3 注意事项

1. **合理的重复不需要消除**：如果两个方法"暂时相同"但未来演进方向不同，合并不如保持重复。
2. **认知复杂度 < 15 是好的**：SonarQube 默认阈值 15，超过 15 建议重构。
3. **重复率 < 3% 是健康的**：如果项目重复率超过 5%，建议将重复治理纳入 Sprint Backlog。

### 4.4 常见踩坑经验

**故障 1：重复消除后引入了 Bug**

根因：消除重复时没有充分考虑两个原始方法之间的细微差异（如异常处理方式不同、空值处理不同）。提取公共方法前必须用单元测试覆盖所有差异场景。

**故障 2：复杂度从 1 个方法转移到了 1 个类，SonarQube 没有变好**

根因：拆方法时没有减少逻辑——只是把复杂度从"方法的维度"转移到了"文件/类的维度"。重构必须减少整体逻辑复杂度，而不仅仅是重新分配。

### 4.5 思考题

1. SonarQube 检测到的重复代码中，哪些是"必须消除的"，哪些是"合理保留的"？请给出判断标准。
2. 圈复杂度和认知复杂度的核心区别是什么？为什么 SonarQube 逐渐用认知复杂度替代圈复杂度？

> **答案提示**：第1题的关键判断标准是"未来是否需要独立演进"。如果两段重复代码被相同的需求驱动，应合并；如果被不同的需求驱动，应保持独立。

---

> **推广计划提示**：重复代码治理建议分两阶段。第一阶段：设定 New Code Duplication < 3% 的门禁，防止新增重复。第二阶段：每 Sprint 选 1-2 个重复代码块进行重构，持续降低历史重复率。架构师应在 Sprint Planning 中预留"技术债务修复时间"（建议占 10-15% 的 Sprint 产能）。
