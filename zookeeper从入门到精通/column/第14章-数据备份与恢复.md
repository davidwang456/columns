# 第14章：数据备份与恢复

## 1. 项目背景

### 业务场景

ZooKeeper 集群在生产环境运行了两年，存储了服务注册信息、配置数据、分布式锁状态等核心数据。某天运维人员发现：

- 事务日志文件所在的磁盘满了，只有 100MB 剩余空间
- ZooKeeper 无法写入新的事务日志
- 强制清理日志时，误删了最新的几个事务日志文件
- 重启 ZooKeeper 后，数据回退到了 2 小时前的状态

更糟的是，由于没有定时备份，丢失的 2 小时数据无法恢复。分布式锁状态丢失导致部分业务死锁了 30 分钟，直到人工介入才恢复。

这个案例说明：**ZooKeeper 的数据备份和恢复策略不是可选项，而是必选项**。

### 痛点放大

ZooKeeper 的数据存储有两部分：

1. **快照（Snapshot）**：内存数据在某个时间点的完整副本
2. **事务日志（Transaction Log）**：每次写操作的增量记录

启动时，ZooKeeper 加载最新的快照，然后回放快照之后的事务日志，重建内存数据。如果事务日志损坏或丢失，数据可能部分丢失。

## 2. 项目设计

### 剧本式交锋对话

**场景**：小白的 ZooKeeper 集群遇到了磁盘故障，需要从备份中恢复数据。

**小白**：ZooKeeper 的数据到底存在哪里？我看了下数据目录，文件很多。

**大师**：ZooKeeper 的数据目录在 `dataDir` 中配置，包含两种文件：

```
dataDir=./data/
```

`data/` 目录的内容：

```
data/
├── version-2/
│   ├── snapshot.1a00000001    ← 快照文件
│   ├── snapshot.1a0000000f    ← 快照文件
│   ├── log.1a00000001         ← 事务日志
│   ├── log.1a00000009         ← 事务日志
│   └── log.1a00000011         ← 事务日志
└── myid                       ← 节点 ID
```

- **快照文件**命名格式：`snapshot.{zxid}`，zxid 即创建快照时的最后提交事务 ID
- **事务日志**命名格式：`log.{zxid}`，zxid 为该日志文件包含的第一个事务 ID

**小胖**：我看了下我的 ZooKeeper 数据目录有几十个快照和日志文件，太多了，能删吗？

**大师**：可以，但要小心。ZooKeeper 启动时会：
1. 找到所有快照文件，选择 zxid 最大的一个
2. 加载该快照
3. 找到该快照之后的所有事务日志
4. 逐个回放日志，重建内存数据库

所以只要保留**最新的快照** + **该快照之后的所有事务日志**就够了。`snapCount` 参数控制每写入多少事务触发一次快照（默认 100000）。

**小白**：那备份怎么做？直接拷文件就行吗？

**大师**：备份有两层含义：

**1. 冷备份**：停掉 ZooKeeper，拷贝 `dataDir` 目录
**2. 热备份**：ZooKeeper 运行时，拷贝快照和日志

热备份要注意：事务日志和快照文件是**增量追加**的，拷贝时可能存在"未写完"的风险。安全做法：

```bash
#!/bin/bash
# 热备份脚本 —— 优先拷贝已经完成的事务日志
BACKUP_DIR="/backup/zookeeper/$(date +%Y%m%d_%H%M%S)"
DATA_DIR="/var/lib/zookeeper/data"

mkdir -p "$BACKUP_DIR"

# 使用 zkServer.sh 的快照命令，获取一致性快照
# --snapshot 参数强制创建快照，确保数据一致性
zkServer.sh snapshot "$DATA_DIR" > "$BACKUP_DIR/snapshot.txt"

# 拷贝事务日志
cp "$DATA_DIR/version-2/log.*" "$BACKUP_DIR/"

# 保留 30 天备份
find /backup/zookeeper/ -type d -mtime +30 -exec rm -rf {} \;
```

**小胖**：那数据恢复呢？怎么恢复备份？

**大师**：恢复分为"完全恢复"和"部分恢复"：

**完全恢复**：

```bash
# 1. 停止 ZooKeeper
zkServer.sh stop

# 2. 备份当前数据（以防恢复失败）
mv /var/lib/zookeeper/data /var/lib/zookeeper/data.bak

# 3. 创建新的数据目录
mkdir -p /var/lib/zookeeper/data/version-2

# 4. 拷贝备份的快照
cp /backup/20250314/snapshot.* /var/lib/zookeeper/data/version-2/

# 5. 拷贝备份的事务日志
cp /backup/20250314/log.* /var/lib/zookeeper/data/version-2/

# 6. 检查 myid 文件
echo "1" > /var/lib/zookeeper/data/myid

# 7. 启动 ZooKeeper
zkServer.sh start

# 8. 验证数据完整性
echo mntr | nc 127.0.0.1 2181 | grep zk_znode_count
```

> **技术映射**：快照 = 数据库的 full dump，事务日志 = binlog，备份 = mysqldump

**小白**：如果事务日志损坏了怎么办？不是按顺序回放的吗？

**大师**：这个问题问得好。ZooKeeper 加载事务日志时，会逐条读取并校验。如果某条记录校验失败（Checksum 错误），ZooKeeper 会**截断**——丢弃该条记录及之后的所有记录。这意味着你可能丢失部分数据。

有一个工具 `zkSnapshotComparer` 可以对比两个快照的差异：

```bash
# 对比正常节点的快照和恢复节点的快照
java -cp zookeeper-3.9.2.jar:lib/* org.apache.zookeeper.server.persistence
    .SnapshotComparer snapshot.1a0000000f snapshot.1a00000011
```

## 3. 项目实战

### 环境准备

- ZooKeeper 3.9.x 单机模式运行中
- 有一些已创建的测试节点

### 分步实现

#### 步骤 1：创建测试数据并手动触发快照

```bash
# 1. 连接 ZooKeeper 并创建一些测试节点
./bin/zkCli.sh -server 127.0.0.1:2181

create /backup-test "backup-test-root"
create /backup-test/app1 "app1-config"
create /backup-test/app2 "app2-config"
create /backup-test/app1/db-url "jdbc:mysql://prod-db:3306/app1"
create /backup-test/app1/redis-host "prod-redis:6379"
set /backup-test/app1/db-url "jdbc:mysql://new-prod-db:3306/app1"

quit

# 2. 查看当前快照文件
ls /var/lib/zookeeper/data/version-2/
# 看到类似：
# log.100000001  ← 事务日志
# snapshot.100000001  ← 快照

# 3. 查看 ZooKeeper 当前状态
echo mntr | nc 127.0.0.1 2181 | grep zk_znode_count
```

#### 步骤 2：使用四字命令查看数据目录配置

```bash
echo conf | nc 127.0.0.1 2181 | grep dataDir
# 输出类似：
# dataDir=/var/lib/zookeeper/data

# 查看快照触发配置
echo conf | nc 127.0.0.1 2181 | grep snapCount
# snapCount=100000
```

#### 步骤 3：手动创建事务日志损坏并恢复

```bash
#!/bin/bash
# 模拟事务日志损坏并恢复

# Step 1: 记录当前数据状态
echo "=== Step 1: 查看当前数据 ==="
./bin/zkCli.sh -server 127.0.0.1:2181 ls /backup-test
./bin/zkCli.sh -server 127.0.0.1:2181 get /backup-test/app1/db-url

# Step 2: 创建新数据（模拟故障前新增的配置）
echo "=== Step 2: 新增配置，模拟故障前写入 ==="
./bin/zkCli.sh -server 127.0.0.1:2181 create /backup-test/app3 "app3-config"
./bin/zkCli.sh -server 127.0.0.1:2181 create /backup-test/app3/timeout "5000ms"

# Step 3: 查找并损坏最新的事务日志
echo "=== Step 3: 模拟事务日志损坏 ==="
DATA_DIR=$(echo conf | nc 127.0.0.1 2181 | grep dataDir | awk '{print $2}')
LATEST_LOG=$(ls -t "$DATA_DIR/version-2/log."* | head -1)
echo "最新事务日志: $LATEST_LOG"

# 损坏日志：覆盖最后 100 字节
dd if=/dev/urandom of="$LATEST_LOG" bs=1 seek=$(( $(stat -c%s "$LATEST_LOG") - 100 )) count=100 conv=notrunc 2>/dev/null
echo "事务日志已损坏（最后100字节被随机数据覆盖）"

# Step 4: 停止 ZooKeeper 并重启
echo "=== Step 4: 重启 ZooKeeper ==="
./bin/zkServer.sh stop
./bin/zkServer.sh start
sleep 3

# Step 5: 检查数据恢复情况
echo "=== Step 5: 检查恢复后数据 ==="
echo mntr | nc 127.0.0.1 2181 | grep zk_znode_count
./bin/zkCli.sh -server 127.0.0.1:2181 ls /backup-test
# 注意：事务日志损坏后，损坏位置之后的数据可能丢失
```

**说明**：由于本章是保护性操作，这里不实际执行此脚本。请在有备份的环境中测试。

#### 步骤 4：使用 zkSnapshotComparer 工具

```bash
# 1. 创建两个时间点的快照
# 通过写入大量数据触发快照，或者设置 snapCount 触发

# 2. 使用 SnapshotComparer 对比
java -cp zookeeper-3.9.2.jar:lib/* \
```

#### 步骤 5：编写完整的备份恢复脚本

创建 `zk-backup.sh`：

```bash
#!/bin/bash
# ZooKeeper 数据备份脚本
# 推荐配置 crontab: 0 */6 * * * /path/to/zk-backup.sh

set -e

ZK_HOME="/opt/zookeeper"
ZK_DATA_DIR="/var/lib/zookeeper/data"
BACKUP_ROOT="/backup/zookeeper"
RETENTION_DAYS=30
DATE_TAG=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="${BACKUP_ROOT}/${DATE_TAG}"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1"
}

# 1. 检查 ZooKeeper 是否运行
log "检查 ZooKeeper 运行状态..."
if ! echo ruok | nc -w 3 127.0.0.1 2181 | grep -q imok; then
    log "ERROR: ZooKeeper 未运行，跳过备份"
    exit 1
fi

# 2. 创建备份目录
mkdir -p "$BACKUP_DIR"

# 3. 触发快照（确保有最新的快照）
log "触发快照..."
echo isro | nc -w 3 127.0.0.1 2181
# 注意：没有直接触发快照的四字命令，snapCount 触发是自动的
# 这里通过 sync 命令确保数据落盘
echo srvr | nc -w 3 127.0.0.1 2181 > /dev/null

# 4. 备份快照文件
log "备份快照文件..."
cp "$ZK_DATA_DIR/version-2/snapshot."* "$BACKUP_DIR/" 2>/dev/null
SNAPSHOT_COUNT=$(ls "$BACKUP_DIR/snapshot."* 2>/dev/null | wc -l)
log "已备份 $SNAPSHOT_COUNT 个快照文件"

# 5. 备份事务日志
log "备份事务日志..."
cp "$ZK_DATA_DIR/version-2/log."* "$BACKUP_DIR/" 2>/dev/null
LOG_COUNT=$(ls "$BACKUP_DIR/log."* 2>/dev/null | wc -l)
log "已备份 $LOG_COUNT 个事务日志文件"

# 6. 备份配置文件和 myid
cp "$ZK_HOME/conf/zoo.cfg" "$BACKUP_DIR/"
cp "$ZK_DATA_DIR/myid" "$BACKUP_DIR/"

# 7. 创建恢复说明
cat > "$BACKUP_DIR/RESTORE.md" << 'RESTORE'
# ZooKeeper 数据恢复步骤

## 完全恢复
1. 停止 ZooKeeper: zkServer.sh stop
2. 备份当前数据: mv data data.bak
3. 创建新目录: mkdir -p data/version-2
4. 拷贝快照和日志: cp backup/* data/version-2/
5. 恢复 myid: cp backup/myid data/
6. 启动 ZooKeeper: zkServer.sh start
7. 验证: echo ruok | nc localhost 2181

## 事务日志损坏恢复
1. 停止 ZooKeeper
2. 删除损坏的日志: rm data/version-2/log.*
3. 只保留最新快照
4. 启动 ZooKeeper（会从快照恢复完整数据）
RESTORE

# 8. 打包压缩（可选）
# cd "$BACKUP_ROOT" && tar -czf "${DATE_TAG}.tar.gz" "$DATE_TAG" && rm -rf "$DATE_TAG"

# 9. 清理过期备份
log "清理 ${RETENTION_DAYS} 天前的过期备份..."
find "$BACKUP_ROOT" -type d -mtime +${RETENTION_DAYS} -exec rm -rf {} \; 2>/dev/null
# 或保留 tar 文件:
# find "$BACKUP_ROOT" -name "*.tar.gz" -mtime +${RETENTION_DAYS} -delete

log "备份完成: $BACKUP_DIR"
log "备份文件列表:"
ls -lh "$BACKUP_DIR/"
```

### 可能遇到的坑

| 问题 | 原因 | 解决 |
|------|------|------|
| 热备份的日志文件损坏 | 拷贝时事务日志正在写入 | 优先选择低峰期备份 |
| Snapshot 和 Log 不对应 | 备份了旧的 snapshot 和新的 log | 使用 ZooKeeper 的 `sync` 命令后备份 |
| 恢复后 ZooKeeper 无法启动 | myid 文件丢失或错误 | 手动创建 myid，值和 server.x 对应 |
| 恢复后数据量比预期少 | 事务日志有坏块，被截断了 | 找更早的备份 |

## 4. 项目总结

### 备份策略推荐

| 策略 | 方法 | 频率 | 适用 |
|------|------|------|------|
| 冷备份 | 停止 ZooKeeper，拷贝 dataDir | 每周 | 可停服场景 |
| 热备份 | 运行时拷贝快照和日志 | 每天（低峰期） | 生产环境 |
| 日志归档 | 保留 N 天的完整事务日志 | 持续 | 可接受增量备份 |
| 异地备份 | 备份文件传输到另一数据中心 | 每天 | 容灾需求 |

### 适用场景

- **日常运维**：热备份脚本定时执行
- **容灾演练**：定期检查恢复流程是否可执行
- **数据迁移**：将快照和日志迁移到新集群

**不适用场景**：
- 数据量极大（快照 > 10GB）的场景（恢复时间长）

### 注意事项

- 快照和事务日志的版本兼容性——不要用 3.4 的快照恢复到 3.9 集群
- 事务日志可以和快照放在不同的磁盘（`dataLogDir`），性能更好
- ZooKeeper 3.5+ 支持自动清理：`autopurge.snapRetainCount=3` 和 `autopurge.purgeInterval=1`

### 常见踩坑经验

**故障 1：FastLeaderElection 恢复后数据不一致**

现象：从备份恢复了一个节点后，该节点始终无法加入集群，一直处于 LOOKING 状态。

根因：备份的 epoch 和当前集群的 epoch 不一致。新 Leader 启动时发现旧节点的 epoch 较大，拒绝接受它加入集群。

解决方案：删除该节点 `dataDir` 下的所有文件（包括 myid），让 ZooKeeper 重新生成数据并从集群同步。

**故障 2：快照加载速度极慢**

现象：启动 ZooKeeper 后 30 分钟仍不可用，日志显示在加载快照。

根因：快照文件在内存中反序列化时，大量 ZNode 导致 OOM，操作系统在使用 Swap。快照加载是 CPU 和内存密集型的，堆内存不足会导致极慢。

解决方案：在启动时增加 JVM 堆内存，或使用 SSD 加速快照读取。

### 思考题

1. ZooKeeper 的自动清理机制 `autopurge.snapRetainCount=3` 只保留 3 个快照。如果这三个快照都损坏了怎么办？如何确保至少有一个可恢复的备份？
2. 在数据恢复场景中，如果 ZooKeeper 的 "最新快照 + 部分事务日志" 损坏了，但你的备份有一个 1 小时前的完整快照。这 1 小时的数据差异能恢复吗？如何实现"增量恢复"？

### 推广计划提示

- **开发**：了解 ZooKeeper 的数据存储结构，有助于理解为什么 ZooKeeper 的数据量受内存限制
- **运维**：务必将备份脚本集成到定时任务中（crontab），并定期演练恢复流程
- **测试**：测试环境定期演练从备份恢复整个 ZooKeeper 集群的流程，记录恢复时间
