# 第15章：常见应用场景概览

## 1. 项目背景

### 业务场景

通过前 14 章的学习，你已经掌握了 ZooKeeper 的配置管理、分布式锁、服务注册发现、命名服务等具体场景的实现。但 ZooKeeper 的应用远不止这些——**Kafka、Dubbo、Hadoop、HBase 等知名项目都依赖 ZooKeeper 做协调**。

本章作为基础篇的收官理论章节，目的是建立 ZooKeeper 应用场景的全景地图，让你知道 ZooKeeper 在大数据生态中的位置。

### 痛点放大

不了解 ZooKeeper 在各技术栈中的实际用法，可能在架构选型和问题排查时走弯路：

- 以为 ZooKeeper 只能做配置中心，不知道它还能做 Leader 选举
- 看到 Kafka 的日志报 `ConnectionLoss`，不知道是 ZooKeeper 的问题还是 Kafka 本身的问题
- Dubbo 的服务发现用 ZooKeeper，但后来发现大规模集群下性能不如 Nacos 的 AP 方案

## 2. 项目设计

### 剧本式交锋对话

**场景**：三人讨论 ZooKeeper 在主流中间件中的使用方式。

**小胖**：Kafka 好像用了 ZooKeeper，但最近的新版本又在去 ZooKeeper 依赖。ZooKeeper 在 Kafka 里到底管什么的？

**大师**：Kafka 对 ZooKeeper 的依赖是逐步减少的。在 Kafka 2.8+ 引入了基于 Raft 的 KRaft 模式来替代 ZooKeeper，但在此之前（以及很多生产部署），ZooKeeper 是 Kafka 不可或缺的一部分。

Kafka 在 ZooKeeper 中存储的内容：

```
/kafka                 ← Kafka 根节点
├── /brokers           ← Broker 注册
│   ├── /ids/0 → {"host":"kafka-1","port":9092}     ← Broker 1 信息
│   ├── /ids/1 → {"host":"kafka-2","port":9092}     ← Broker 2 信息
│   └── /ids/2 → {"host":"kafka-3","port":9092}     ← Broker 3 信息
├── /controller        ← Controller（类似 Leader）
│   → 0x0000000001     ← 当前 Controller 的 Broker ID
├── /topics            ← Topic 元数据
│   ├── /orders → {"partitions":{"0":[1,2],"1":[2,3],"2":[3,1]}}
│   ├── /users
│   └── /products
├── /config            ← 配置
│   ├── /topics/orders → {"retention.ms":86400000}
│   └── /clients/producer-1 → {"compression.type":"gzip"}
└── /admin             ← 管理操作
    └── /reassign_partitions
```

**小白**：那 Kafka 的 Controller 选举就是 ZooKeeper 的 Leader 选举机制吗？

**大师**：对。Kafka 使用 ZooKeeper 的临时节点 + Watcher 做 Controller 选举：

```java
// Kafka 的 Controller 选举简化版
// 每个 Broker 启动时尝试在 /controller 创建临时节点
// 创建成功的成为 Controller
// 创建失败的监听 /controller 的变化

// 当前 Controller 宕机时：
// 临时节点 /controller 被自动删除
// 所有候选 Broker 的 Watcher 被触发
// 触发重新选举，第一个重建 /controller 的成为新 Controller
```

**小胖**：那 Dubbo 服务发现是用 ZooKeeper 做的？怎么实现的？

**大师**：Dubbo 用 ZooKeeper 做服务注册和发现：

```
/dubbo                     ← Dubbo 根节点
├── /com.example.UserService   ← 接口全限定名
│   ├── /providers
│   │   ├── dubbo://192.168.1.1:20880/com.example.UserService
│   │   ├── dubbo://192.168.1.2:20880/com.example.UserService
│   │   └── dubbo://192.168.1.3:20880/com.example.UserService
│   ├── /consumers
│   │   ├── consumer://192.168.1.100/com.example.UserService
│   │   └── consumer://192.168.1.101/com.example.UserService
│   ├── /configurators
│   │   └── override://0.0.0.0/com.example.UserService?timeout=5000
│   └── /routers
│       └── condition://0.0.0.0/com.example.UserService?...
└── /com.example.OrderService
    └── ...
```

- **providers**：服务提供者注册的临时节点（断连自动删除）
- **consumers**：服务消费者注册的临时节点（用于统计和监控）
- **configurators**：动态配置覆盖
- **routers**：路由规则

**小白**：我注意到 providers 的 URL 中有 `dubbo://` 协议、IP、端口，还有其他参数。但不像之前我们做的服务发现只存 IP:Port，Dubbo 存储的信息丰富得多。

**大师**：对的，Dubbo 把**完整的 URL** 作为节点数据存进去了，包括协议版本、序列化方式、超时时间等。这也是 ZooKeeper 作为配置中心的体现——服务发现的节点数据里就包含了配置信息。

> **技术映射**：Kafka 的 ZooKeeper = 集群管理面板，存储 Broker 成员、Topic 配置、Controller 选举；Dubbo 的 ZooKeeper = 服务注册表 + 配置中心

**小白**：HBase 也依赖 ZooKeeper？它用来做什么？

**大师**：HBase 对 ZooKeeper 的依赖更深：

```
/hbase                 ← HBase 根节点
├── /master            ← 当前 Master 节点（临时节点）
├── /backup-masters    ← 备用 Masters（临时节点）
├── /rs                ← RegionServer 注册
│   ├── hbase-1,16020,...
│   ├── hbase-2,16020,...
│   └── hbase-3,16020,...
├── /meta-region-server  ← Meta 表所在的 RegionServer
├── /table-lock        ← 表级别分布式锁
├── /draining          ← 正在退役的 RegionServer
└── /splitlog          ← Split 操作的 WAL 日志跟踪
```

- **Master 选举**：谁是 HBase Master（只有一个活跃 Master）
- **RegionServer 注册**：哪些 RegionServer 在运行
- **Meta 表定位**：存有所有 Region 元数据的表在哪
- **分布式锁**：表级别操作加锁

## 3. 项目实战

### 环境准备

- Kafka 和 Dubbo 环境不是必须的，本章是纯概念讲解 + 实证分析
- 运行中的 ZooKeeper 集群（用于验证目录结构）

### 分步实现

#### 步骤 1：查看 Kafka 在 ZooKeeper 中的注册信息

如果你有 Kafka 集群，可以查看它在 ZooKeeper 中的数据：

```bash
# 连接 ZooKeeper
./bin/zkCli.sh -server 127.0.0.1:2181

# 查看 Kafka 注册的 Broker
ls /brokers/ids
# 输出：如果有 Kafka 集群运行，这里会列出 Broker ID

# 查看特定 Broker 信息
get /brokers/ids/0
# 输出 JSON：{"listener_security_protocol_map":..., "endpoints":..., "host":"...","port":..., ...}

# 查看 Controller
get /controller
# 输出：{"version":1,"brokerid":0,"timestamp":"..."}

# 查看 Topic
ls /brokers/topics
```

#### 步骤 2：查看 Dubbo 在 ZooKeeper 中的注册信息

```bash
# 连接 ZooKeeper
./bin/zkCli.sh -server 127.0.0.1:2181

# 如果有 Dubbo 服务运行
ls /dubbo
# 输出：接口列表

# 查看某个接口的服务提供者
ls /dubbo/com.example.UserService/providers
# 输出：提供者 URL
```

#### 步骤 3：各场景对应 Curator Recipe 速查表

| 应用场景 | 底层机制 | Curator Recipe | 内部原理 |
|---------|---------|---------------|---------|
| 配置管理 | 持久节点 + NodeCache | `NodeCache` | 监听单节点数据变化 |
| 服务发现 | 临时节点 + PathChildrenCache | `ServiceDiscovery` | 监听子节点列表变化 |
| 分布式锁 | 临时顺序节点 | `InterProcessMutex` | 创建顺序节点，监听前驱节点 |
| 读写锁 | 临时顺序节点 + 读写区分 | `InterProcessReadWriteLock` | 读锁监听写锁，写锁监听前驱 |
| Leader 选举 | 临时节点 / 临时顺序节点 | `LeaderSelector` / `LeaderLatch` | 创建节点，抢到即为主 |
| 队列 | 顺序节点 | `DistributedQueue` | FIFO，按序号最小的出队 |
| 屏障 | 持久节点 + Watcher | `DistributedBarrier` | 等待子节点数满足条件 |
| 计数器 | 持久节点 + 版本号 CAS | `DistributedAtomicLong` | 读取-修改-写回（CAS 乐观锁） |

#### 步骤 4：场景适用性对比表

| 特性 | ZooKeeper | Redis | Etcd |
|------|-----------|-------|------|
| CAP | CP（一致性+分区容错） | AP（可用性+分区容错） | CP |
| 数据模型 | 树形（ZNode） | KV | KV（版本化） |
| 通知机制 | Watcher（一次性） | Pub/Sub / Keyspace Notification | Watch（持续） |
| 典型应用 | 配置中心、分布式锁 | 缓存、Session | K8s、服务发现 |
| 一致性 | 强一致（ZAB） | 最终一致 | 强一致（Raft） |
| 写性能 | 中等（~10W QPS） | 极高（~10W+ QPS） | 中等 |
| 大数据量 | 受限（内存） | 可持久化（RDB/AOF） | 中等（内存+磁盘） |
| 客户端生态 | Curator（Java） | Jedis/Redisson | etcdctl / gRPC |

**Kafka、Dubbo、Hadoop、HBase 等技术为什么选择 ZooKeeper？**

| 技术 | 为什么选 ZooKeeper | 核心用途 |
|------|-------------------|---------|
| Kafka | 强一致性保证 Controller 选举的可靠性 | Controller 选举、Broker 注册、配置存储 |
| Dubbo | 树形模型天然适合 URL 层级存储 | 服务注册发现、路由规则、动态配置 |
| HBase | 临时节点自动检测 RegionServer 崩溃 | Master 选举、RegionServer 注册、Meta 表定位 |
| Solr | ZooKeeper 一致性保证集群状态正确 | 集群状态管理、配置下发、Leader 选举 |

### 可能遇到的坑

| 问题 | 原因 | 解决 |
|------|------|------|
| Kafka 连接 ZooKeeper 超时 | Kafka 版本与 ZooKeeper 版本不匹配 | 检查 Kafka + ZooKeeper 版本兼容矩阵 |
| Dubbo 服务注册后找不到 | Dubbo 的 namespace 和 ZooKeeper 路径不一致 | 检查 Dubbo 配置中的 `dubbo.registry.group` |
| ZooKeeper 节点数过多 | 应用未清理临时节点 | 排查连接泄漏，配置 `autopurge` |

## 4. 项目总结

### ZooKeeper 场景全景图

```
                    ┌─────────────────┐
                    │   ZooKeeper 集群 │
                    └────────┬────────┘
          ┌──────────────────┼──────────────────┐
          │                  │                  │
    ┌─────▼──────┐    ┌─────▼──────┐    ┌─────▼──────┐
    │ 配置管理    │    │ 服务发现    │    │ 分布式锁    │
    │            │    │            │    │            │
    │ /config/   │    │ /services/ │    │ /locks/    │
    └────────────┘    └────────────┘    └────────────┘
          │                  │                  │
    ┌─────▼──────┐    ┌─────▼──────┐    ┌─────▼──────┐
    │ 选主/选举   │    │ 分布式队列  │    │ 计数器     │
    │            │    │            │    │            │
    │ /leader/   │    │ /queue/    │    │ /counter/  │
    └────────────┘    └────────────┘    └────────────┘

    生态集成:
    ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
    │ Kafka│  │ Dubbo│  │ HBase│  │ Solr │
    └──────┘  └──────┘  └──────┘  └──────┘
```

### ZooKeeper 的局限与新趋势

- **Kafka**：2.8+ 引入 KRaft 模式，逐步去 ZooKeeper 依赖
- **Dubbo**：3.0+ 支持 Nacos、Consul 作为注册中心
- **服务发现**：Nacos（AP+CP 双模式）在微服务场景更适合
- **配置中心**：Apollo（携程）、Nacos（阿里）在功能丰富度上优于 ZooKeeper

### 思考题

1. 为什么 HBase 选择 ZooKeeper 做 Master 选举而不是自己实现一个 Raft 算法？如果今天重新选型，HBase 会不会选择 Etcd？
2. Dubbo 的 ZooKeeper 目录结构中，`/dubbo/com.example.UserService/providers` 下的节点是临时节点。如果 Dubbo 服务实例突然宕机，从 ZooKeeper 上注销到 Consumer 感知到变化隔离，大致需要多长时间？这个时间受哪些因素影响？

### 推广计划提示

- **开发**：本章帮助你建立 ZooKeeper 的开阔视野——不只是作为独立学习的技术，更是 Kafka、Dubbo、HBase 等热门技术的底层依赖
- **运维**：排查 Kafka/Dubbo/HBase 问题时，先检查 ZooKeeper 集群的健康状态，往往是根因
- **架构师**：选型时注意 ZooKeeper 的 CP 特性，如果业务场景更需要 AP，考虑 Nacos（服务发现）或 Etcd（K8s 原生）
