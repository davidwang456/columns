# 第29章：跨机房部署与多数据中心方案

## 1. 项目背景

### 业务场景

你的电商系统有两个数据中心——一个是主数据中心（北京），一个是灾备数据中心（上海）。需要实现：

- 北京机房发生灾难时，上海机房可以接管服务
- 客户端读取的请求尽量在本地机房处理（降低延迟）
- 写入请求必须保证数据不丢失

但 ZooKeeper 的 ZAB 协议假设所有节点在同一个网络中，延迟很低。跨机房部署面临：**高延迟（几十毫秒 vs 零点几毫秒）、分区风险、一致性与可用性的权衡**。

### 痛点放大

跨机房部署的困难：

- **写入延迟大**：写请求需要跨机房广播，延迟增加 10-100 倍
- **脑裂风险**：机房之间的网络中断，两个机房各自选出 Leader
- **Quorum 分裂**：3 个节点分布在 2 个机房，一个机房宕机后 Quorum 不足

## 2. 项目设计

### 剧本式交锋对话

**场景**：小胖需要在两个机房部署 ZooKeeper，但不知道怎么做。

**小胖**：我有 5 台机器，北京 3 台，上海 2 台。这样部署可以吗？

**大师**：这是典型的跨机房部署陷阱。5 个节点中，3 个在北京，2 个在上海。Quorum = 3。如果北京机房网络故障，上海的 2 个节点无法达到 Quorum，整个集群不可用。

更好的方案：

**方案一：Observer 模式（推荐）**

```
北京机房（主）：3 个投票节点（1 Leader + 2 Follower）
上海机房（灾备）：2 个 Observer（只读，不投票）

优点：写请求只需北京机房内的 Quorum（2/3），不受上海机房延迟影响
优点：上海机房可以服务本地读请求
缺点：上海机房的数据是异步同步的，轻微落后
```

**小白**：那不投票的 Observer 能紧急升级为投票节点吗？如果北京机房全挂了。

**大师**：不能。Observer 不会参与选举。如果要灾备，可以在上海机房部署 2 个 Follower，但要接受"写请求要跨机房确认"的延迟。

**方案二：3 机房 5 节点（2+2+1）**

```
北京机房：2 个节点
上海机房：2 个节点
广州机房：1 个节点
Quorum = 3

任意一个机房故障，剩余节点仍能组成 Quorum
广州机房作为"仲裁者"，不一定会被选为 Leader，但能帮助形成 Quorum
```

> **技术映射**：Observer = 只读副本，就像图书馆的"参考阅览室"（可以看书但不能借走）；仲裁者 = 投票观察员，没有决策权但参与了表决

**小白**：Proxy 方案呢？Nginx 做 ZooKeeper 代理？

**大师**：ZooKeeper **不支持 HTTP 代理**。客户端必须直接连接 ZooKeeper 的 NIO 端口。不过可以**用 DNS 做客户端就近路由**：

```bash
# 客户端连接字符串
# DNS 返回不同机房的 ZooKeeper 地址
zk-client.example.com → 根据客户端 IP 返回最近的 ZooKeeper 节点

# 北京客户端 → 北京 ZooKeeper
# 上海客户端 → 上海 ZooKeeper（Observer）
```

**小胖**：那跨机房写入怎么保证？北京客户端写请求到北京 Leader，上海客户端写请求也要到北京 Leader。

**大师**：对，**写请求必须由 Leader 处理**。客户端可以通过连接字符串同时指定多个节点：

```java
// 客户端连接所有节点，由 ZooKeeper 客户端自动发现 Leader
CuratorFramework client = CuratorFrameworkFactory.builder()
    .connectString("zk-beijing:2181,zk-shanghai:2181,zk-guangzhou:2181")
    // 客户端连接任意节点，写请求自动转发到 Leader
```

客户端连接任意节点后，如果该节点不是 Leader，会自动将写请求**透明转发**到 Leader。

#### 步骤 5：ObserverMaster 协议（3.6+）

ZooKeeper 3.6+ 引入了 **ObserverMaster** 协议，Observer 不再直接从 Leader 同步数据，而是从某个指定的 Follower（ObserverMaster）同步：

```properties
# 在 Observer 的 zoo.cfg 中配置
observerMasterPort=2889   # Observer 从此端口从 Follower 同步
```

优势：
- **减少 Leader 负担**：Leader 不需要直接为所有 Observer 提供数据同步
- **更好的扩展性**：一个 Follower 可以服务多个 Observer

#### 步骤 6：客户端就近路由

创建 `ZoneAwareClient.java`：

```java
package com.zkdemo.dc;

import org.apache.curator.framework.CuratorFramework;
import org.apache.curator.framework.CuratorFrameworkFactory;
import org.apache.curator.retry.ExponentialBackoffRetry;

import java.net.InetAddress;

/**
 * 就近路由客户端——根据部署区域选择 ZooKeeper 节点
 */
public class ZoneAwareClient {
    // 不同机房的连接串
    private static final String ZK_BEIJING = "zk-bj-1:2181,zk-bj-2:2181,zk-bj-3:2181";
    private static final String ZK_SHANGHAI = "zk-sh-1:2181,zk-sh-2:2181";

    public static CuratorFramework createClient() {
        String zkConnectString = selectNearestZone();

        System.out.println("连接 ZooKeeper: " + zkConnectString);

        return CuratorFrameworkFactory.builder()
                .connectString(zkConnectString)
                .retryPolicy(new ExponentialBackoffRetry(1000, 3))
                .build();
    }

    private static String selectNearestZone() {
        String hostname;
        try {
            hostname = InetAddress.getLocalHost().getHostName().toLowerCase();
        } catch (Exception e) {
            return ZK_BEIJING; // 默认连接北京
        }

        // 根据主机名判断区域
        if (hostname.contains("sh") || hostname.contains("shanghai")) {
            System.out.println("[路由] 上海机房 → 连接上海 Observer");
            return ZK_SHANGHAI;
        } else {
            System.out.println("[路由] 北京机房 → 连接北京集群");
            return ZK_BEIJING;
        }
    }

    public static void main(String[] args) throws Exception {
        CuratorFramework client = createClient();
        client.start();

        // 读请求（上海机房就近读取 Observer）
        // 写请求（ZooKeeper 会自动转发到北京 Leader）
        System.out.println("当前集群状态:");
        client.getChildren().forPath("/").forEach(path ->
                System.out.println("  /" + path));

        client.close();
    }
}
```

### 可能遇到的坑

| 问题 | 原因 | 解决 |
|------|------|------|
| 跨机房写入超时 | 机房之间延迟大，syncLimit 太小 | 增大 syncLimit（按延迟估算） |
| Observer 数据落后 | Observer 同步延迟 | 使用 ObserverMaster 协议优化 |
| 脑裂 | 机房之间网络断开，各自选举 Leader | 3 机房 2+2+1 方案 |

## 4. 项目总结

### 三种方案对比

| 方案 | 可用性 | 写入延迟 | 读一致性 | 复杂度 |
|------|--------|---------|---------|--------|
| Observer 模式 | 高（3/3 Quorum 在北京） | 低（北京局域网内） | 最终一致（Observer 落后） | 低 |
| 2+2+1 三机房 | 极高（任意机房故障可用） | 高（跨机房确认） | 强一致 | 中 |
| 双集群同步 | 极高 | 中 | 最终一致 | 高 |

### 推荐方案

- **读多写少场景**：Observer 模式（简单可靠）
- **容灾要求高**：3 机房 2+2+1（能容忍单机房故障）
- **完全灾备**：双集群 + 消息同步（但太复杂，不推荐）

### 注意事项

- Observer 不参与投票，Quorum 只在投票节点中计算
- 跨机房部署时，`syncLimit` 和 `initLimit` 要适当增大（跨机房延迟更高）
- Observer 的 `peerType=observer` 要在 `zoo.cfg` 中配置

### 常见踩坑经验

**故障 1：跨机房部署的 syncLimit 配置太小**

现象：机房之间的网络偶尔抖动（延迟从 2ms 到 200ms），Follower 被判定为"失联"，频繁触发选举。

根因：`syncLimit=5`（默认 5 tickTime = 10 秒），但跨机房网络波动时 Follower 超过 10 秒无法同步数据，被 Leader 剔除。

解决方案：`syncLimit=20`（40 秒，根据机房之间的延迟估算）。

**故障 2：Observer 数据明显落后**

现象：上海机房的 Observer 读到的是 30 秒前的数据。

根因：Observer 从 Leader 直接同步，Leader 繁忙时 Observer 同步队列被排到后面。ZooKeeper 3.6+ 的 ObserverMaster 协议可以缓解。

### 思考题

1. 3 机房 2+2+1 方案中，广州机房的 1 个节点被称为"仲裁者"。如果北京和上海之间的网络断了，但广州和双方都能通信，会发生什么？ZooKeeper 如何决定数据应该从哪边导入？
2. Observer 模式下，上海机房的客户端也能读取配置中心的配置。如果上海 Observer 同步延迟了 5 秒，客户端刚刚读取到"旧配置"后就处理了一个业务请求，这会导致什么后果？如何避免？

### 推广计划提示

- **开发**：客户端连接字符串中包含多个机房的地址，ZooKeeper 客户端自动发现 Leader
- **运维**：跨机房部署时需要重点关注各节点之间的网络延迟和带宽
- **架构师**：跨机房部署的核心是"权衡"——一致性、可用性、延迟三者在跨机房场景中不可兼得
