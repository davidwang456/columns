# 第11章：配置管理与命名服务实战

## 1. 项目背景

### 业务场景

你的微服务架构中，每个服务都需要读取配置：数据库连接串、Redis 地址、限流阈值、功能开关……传统做法是把配置写在 `application.properties` 里。但问题来了：

- 配置改了需要重新打包、部署，周期太长
- 多个环境的配置（dev/staging/prod）散落在不同仓库中
- 紧急需要关闭某个功能开关，不能快速生效

ZooKeeper 做配置中心，天然适合：存储配置（持久节点）、感知变更（Watcher）、多环境隔离（路径隔离）。

另一个需求：分布式系统中需要生成**全局唯一、单调递增**的 ID，用于订单号、流水号等。ZooKeeper 的顺序节点正好满足这个需求。

### 痛点放大

没有 ZooKeeper 配置中心时：
- **改配置 = 重启服务**：数据库切换需要修改配置文件 → 重启所有实例 → 耗时数十分钟
- **配置散落**：不同环境（dev/staging/prod）的配置缺乏统一管理，经常出现"测试环境改好了，生产忘了改"

缺乏分布式 ID 时：
- **数据库自增**：依赖数据库，高并发下成为瓶颈
- **UUID**：生成快但无序，不利于数据库索引
- **雪花算法**：依赖机器时钟，时钟回拨会导致 ID 冲突

## 2. 项目设计

### 剧本式交锋对话

**场景**：运维在 ZooKeeper 上改了数据库地址，希望所有服务实例都能自动感知并切换。

**小胖**：我已经在第 5 章用 Watcher 实现了配置刷新，难道不够吗？

**大师**：手写 Watcher 有很多细节要考虑。我们对比一下"玩具"和"生产级"配置中心的区别：

| 特性 | 玩具方案 | 生产方案 |
|------|---------|---------|
| 路径结构 | `/config/db-url` | `/config/{app}/{env}/{key}` |
| Watcher 注册 | 一次性，每次手动 | NodeCache 自动持续监听 |
| 多环境 | 无 | 按路径隔离（/dev /staging /prod） |
| 容错 | 连不上就崩溃 | 本地缓存 + ZooKeeper 不可用时降级 |
| 版本管理 | 无 | 保留历史版本，支持回滚 |

**小白**：路径结构为什么要设计得这么复杂？

**大师**：我们可以设计成这样：

```
/config                        ← 配置根目录
  /config/order-service        ← 订单服务
    /config/order-service/dev  ← 开发环境
      /config/order-service/dev/db-url = "jdbc:mysql://dev-db:3306/order"
      /config/order-service/dev/redis-host = "dev-redis:6379"
    /config/order-service/staging
    /config/order-service/prod
  /config/user-service
    /config/user-service/dev
    /config/user-service/prod
```

每个应用 + 环境独立路径，天然隔离。再配合 ACL，不同团队只能操作自己的路径。

**小胖**：那灰度发布怎么搞？我想先让 10% 的实例用新配置，90% 的老配置。

**大师**：按实例 ID 灰度：

```
/config/order-service/prod/db-url = "jdbc:mysql://prod-db:3306/order"
/config/order-service/prod/db-url/gray-instances
    → 内容: ["instance-001", "instance-002", ...]
```

灰度实例读取 `/gray-instances` 判断自己是否在名单中，如果在，使用灰度配置节点；否则使用默认节点。

**小白**：那命名服务呢？用顺序节点生成唯一 ID 的原理是什么？

**大师**：非常简单：

```java
// 每次创建顺序节点，ZooKeeper 自动生成递增序号
String idPath = zk.create("/id-generator/order-", new byte[0],
        CreateMode.PERSISTENT_SEQUENTIAL);
// 返回：/id-generator/order-0000000042
// 从路径中提取：42
String orderId = idPath.substring(idPath.lastIndexOf("-") + 1);
```

这就是一个全局唯一、单调递增的 ID。和 UUID 的对比：

| 特性 | 顺序节点 ID | UUID |
|------|-----------|------|
| 唯一性 | 全局唯一（ZK 保证） | 全局唯一（概率） |
| 单调递增 | ✓ | ✗ |
| 可排序 | ✓（数字序） | ✗（字符串序） |
| 长度 | 可变 | 36 字符 |
| 性能 | 依赖 ZK 写入吞吐 | 本地生成，无网络开销 |

### 分步实现

#### 步骤 1：配置中心核心实现

创建 `ConfigCenter.java`：

```java
package com.zkdemo.config;

import org.apache.curator.RetryPolicy;
import org.apache.curator.framework.CuratorFramework;
import org.apache.curator.framework.CuratorFrameworkFactory;
import org.apache.curator.framework.recipes.cache.NodeCache;
import org.apache.curator.retry.ExponentialBackoffRetry;

import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

public class ConfigCenter {
    private final CuratorFramework client;
    private final String appName;
    private final String env;
    private final Map<String, String> localCache = new ConcurrentHashMap<>();
    private final Map<String, NodeCache> watchers = new ConcurrentHashMap<>();

    public ConfigCenter(String connectString, String appName, String env) {
        this.appName = appName;
        this.env = env;
        RetryPolicy retryPolicy = new ExponentialBackoffRetry(1000, 3);
        this.client = CuratorFrameworkFactory.builder()
                .connectString(connectString)
                .namespace("config")  // 所有配置在 /config 下
                .retryPolicy(retryPolicy)
                .build();
        this.client.start();
    }

    // 初始化配置：从 ZooKeeper 拉取并注册 Watcher
    public void initConfig(Map<String, String> defaultConfig) throws Exception {
        for (Map.Entry<String, String> entry : defaultConfig.entrySet()) {
            String configPath = buildPath(entry.getKey());

            // 如果节点不存在，用默认值创建
            if (client.checkExists().forPath(configPath) == null) {
                client.create().creatingParentsIfNeeded()
                        .forPath(configPath, entry.getValue().getBytes());
            }

            // 读取配置到本地缓存
            byte[] data = client.getData().forPath(configPath);
            localCache.put(entry.getKey(), new String(data));

            // 注册 NodeCache 持续监听
            startWatcher(entry.getKey(), configPath);
        }
    }

    private void startWatcher(String key, String path) throws Exception {
        NodeCache cache = new NodeCache(client, path);
        cache.getListenable().addListener(() -> {
            if (cache.getCurrentData() != null) {
                String newValue = new String(cache.getCurrentData().getData());
                String oldValue = localCache.put(key, newValue);
                System.out.println("[配置变更] " + key + ": " + oldValue + " → " + newValue);
                onConfigChanged(key, oldValue, newValue);
            }
        });
        cache.start();
        watchers.put(key, cache);
    }

    // 配置变更回调（子类可覆盖）
    protected void onConfigChanged(String key, String oldValue, String newValue) {
    }

    public String getConfig(String key) {
        return localCache.get(key);
    }

    public void setConfig(String key, String value) throws Exception {
        String path = buildPath(key);
        client.setData().forPath(path, value.getBytes());
    }

    private String buildPath(String key) {
        return "/" + appName + "/" + env + "/" + key;
    }

    public void close() {
        watchers.values().forEach(NodeCache::close);
        client.close();
    }
}
```

#### 步骤 2：使用配置中心

创建 `ConfigCenterDemo.java`：

```java
package com.zkdemo.config;

import java.util.HashMap;
import java.util.Map;

public class ConfigCenterDemo {
    public static void main(String[] args) throws Exception {
        // 创建订单服务的配置中心（开发环境）
        ConfigCenter configCenter = new ConfigCenter(
                "127.0.0.1:2181", "order-service", "dev") {

            @Override
            protected void onConfigChanged(String key, String oldValue, String newValue) {
                // 配置变更时刷新连接池
                if (key.equals("db.url")) {
                    System.out.println(">>> 刷新数据库连接池: " + newValue);
                } else if (key.equals("redis.host")) {
                    System.out.println(">>> 刷新 Redis 连接: " + newValue);
                } else if (key.equals("feature.switch")) {
                    System.out.println(">>> 功能开关变更: " + newValue);
                }
            }
        };

        // 初始化配置（默认值）
        Map<String, String> defaults = new HashMap<>();
        defaults.put("db.url", "jdbc:mysql://dev-db:3306/order");
        defaults.put("db.max-active", "20");
        defaults.put("redis.host", "dev-redis:6379");
        defaults.put("redis.timeout", "3000");
        defaults.put("feature.switch", "true");
        defaults.put("rate.limit", "1000");

        configCenter.initConfig(defaults);

        // 读取配置
        System.out.println("\n=== 当前配置 ===");
        System.out.println("db.url: " + configCenter.getConfig("db.url"));
        System.out.println("redis.host: " + configCenter.getConfig("redis.host"));
        System.out.println("rate.limit: " + configCenter.getConfig("rate.limit"));

        // 手动修改配置模拟变更
        System.out.println("\n模拟管理员修改配置...");
        System.out.println("请在另一个终端执行：");
        System.out.println("  set /config/order-service/dev/db.url \"jdbc:mysql://new-dev-db:3306/order\"");

        // 等待修改
        Thread.sleep(60000);
        configCenter.close();
    }
}
```

#### 步骤 3：分布式 ID 生成器

创建 `IdGenerator.java`：

```java
package com.zkdemo;

import org.apache.curator.RetryPolicy;
import org.apache.curator.framework.CuratorFramework;
import org.apache.curator.framework.CuratorFrameworkFactory;
import org.apache.curator.retry.ExponentialBackoffRetry;
import org.apache.zookeeper.CreateMode;

public class IdGenerator {
    private static final String ID_ROOT = "/id-generator";
    private final CuratorFramework client;

    public IdGenerator(String connectString) {
        RetryPolicy retryPolicy = new ExponentialBackoffRetry(1000, 3);
        this.client = CuratorFrameworkFactory.newClient(connectString, retryPolicy);
        this.client.start();
    }

    // 生成全局唯一 ID
    public long generateId(String type) throws Exception {
        // 确保父节点存在
        String parentPath = ID_ROOT + "/" + type;
        if (client.checkExists().forPath(parentPath) == null) {
            client.create().creatingParentsIfNeeded()
                    .forPath(parentPath, new byte[0]);
        }

        // 创建顺序节点
        String path = client.create()
                .withMode(CreateMode.PERSISTENT_SEQUENTIAL)
                .forPath(parentPath + "/", new byte[0]);

        // 从路径提取序号：/id-generator/order/00000000042 → 42
        String seqStr = path.substring(path.lastIndexOf("/") + 1);
        long id = Long.parseLong(seqStr);

        // 清理这个占位节点（ID 已生成，不需要保留了）
        client.delete().forPath(path);

        return id;
    }

    public void close() {
        client.close();
    }

    public static void main(String[] args) throws Exception {
        IdGenerator idGen = new IdGenerator("127.0.0.1:2181");

        // 生成 10 个订单 ID
        System.out.println("=== 生成订单 ID ===");
        for (int i = 0; i < 10; i++) {
            long id = idGen.generateId("order");
            System.out.println("订单 ID: " + id);
        }

        // 生成 10 个支付流水 ID
        System.out.println("\n=== 生成支付流水 ID ===");
        for (int i = 0; i < 5; i++) {
            long id = idGen.generateId("payment");
            System.out.println("支付流水 ID: " + id);
        }

        idGen.close();
    }
}
```

#### 步骤 4：灰度推送

创建 `GrayReleaseConfigCenter.java`：

```java
package com.zkdemo.config;

import org.apache.curator.framework.CuratorFramework;
import org.apache.curator.framework.recipes.cache.ChildData;
import org.apache.curator.framework.recipes.cache.PathChildrenCache;
import org.apache.zookeeper.CreateMode;

import java.util.List;
import java.util.concurrent.CopyOnWriteArrayList;

public class GrayReleaseConfigCenter {
    private final CuratorFramework client;
    private final String grayListPath;
    private final List<String> grayInstances = new CopyOnWriteArrayList<>();

    public GrayReleaseConfigCenter(CuratorFramework client, String grayListPath) throws Exception {
        this.client = client;
        this.grayListPath = grayListPath;

        if (client.checkExists().forPath(grayListPath) == null) {
            client.create().creatingParentsIfNeeded()
                    .withMode(CreateMode.PERSISTENT)
                    .forPath(grayListPath, "[]".getBytes());
        }

        // 监听灰度列表变化
        loadGrayList();
        watchGrayList();
    }

    private void loadGrayList() throws Exception {
        byte[] data = client.getData().forPath(grayListPath);
        String grayJson = new String(data);
        grayInstances.clear();
        // 简化的 JSON 解析
        String cleaned = grayJson.replace("[", "").replace("]", "")
                .replace("\"", "").trim();
        if (!cleaned.isEmpty()) {
            String[] parts = cleaned.split(",");
            for (String part : parts) {
                grayInstances.add(part.trim());
            }
        }
    }

    private void watchGrayList() throws Exception {
        NodeCache cache = new NodeCache(client, grayListPath);
        cache.getListenable().addListener(() -> {
            if (cache.getCurrentData() != null) {
                loadGrayList();
                System.out.println("灰度列表已更新，当前灰度实例: " + grayInstances);
            }
        });
        cache.start();
    }

    public boolean isGrayInstance(String instanceId) {
        return grayInstances.contains(instanceId);
    }

    public String getConfig(String defaultConfig, String grayConfig, String instanceId) {
        if (isGrayInstance(instanceId)) {
            return grayConfig;
        }
        return defaultConfig;
    }
}
```

### 可能遇到的坑

| 问题 | 原因 | 解决 |
|------|------|------|
| NodeCache 不触发 | 节点路径错误（有 namespace） | 检查 namespace 配置 |
| 配置变更丢失 | 变更发生在 NodeCache 初始化之前 | 启动时先 `getData` 再 `start` |
| ID 生成重复 | ZooKeeper 连接断开后重连 | 改用 `DistributedAtomicLong` |
| 父节点不存在 | 没有创建前置路径 | 使用 `creatingParentsIfNeeded()` |

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 配置中心 | 实时推送、版本管理、环境隔离 | 1MB 单节点限制，配置量大时不适用 |
| ID 生成器 | 全局唯一、单调递增、强一致 | QPS 受 ZooKeeper 写性能限制 |
| 灰度推送 | 按实例灰度，可控 | 灰度列表需要手动维护 |

### 适用场景

- **配置中心**：微服务配置管理（< 100 个配置项/服务）
- **命名服务**：小流量 ID 生成（< 10 万/天）
- **灰度发布**：配置层面的灰度切换

**不适用场景**：
- 大规模配置存储（建议 Nacos / Apollo）
- 高频 ID 生成（建议雪花算法 / Redis）

### 注意事项

- ID 生成器的顺序节点用完记得 `delete`，否则会留下大量空节点
- 配置中心需要本地缓存兜底，ZooKeeper 不可用时使用缓存配置
- 灰度列表建议用独立路径，不影响主配置结构

### 常见踩坑经验

**故障 1：配置变更后客户端没有收到通知**

现象：运维修改了 ZooKeeper 上的配置，但客户端 10 分钟后仍未感知。

根因：NodeCache 的 `start()` 需要在连接建立后调用。如果 ZooKeeper 连接断开后重连，NodeCache 自动重新注册，但如果在断连期间配置发生了变化，这个变更不会触发回调。需要启动时主动拉取一次最新值。

**故障 2：顺序节点 ID 回跳**

现象：ID 生成器从 `42` 变成了 `35`。

根因：ZooKeeper 使用了旧的快照恢复数据，顺序节点的计数器回退到旧值。这是 ZooKeeper 3.4.x 及以下的已知问题，3.5+ 已经修复。

### 思考题

1. 配置中心的灰度推送是按"实例 ID"手动维护列表的。如何设计一个"按百分比"灰度推送的方案？比如：新配置先推送给 10% 的实例，观察 5 分钟无问题再推给全部。
2. 分布式 ID 生成器在高并发下（每秒 10 万+），ZooKeeper 单机写入 QPS 可能成为瓶颈。如何设计一个"预分配"策略来提升 ID 生成性能？

### 推广计划提示

- **开发**：配置中心 + 命名服务是 ZooKeeper 最基础的两个应用场景，建议在生产环境优先落地配置中心
- **运维**：修改配置后需要确认所有实例都已收到变更通知，可以观察 zkCli 的 stat 查看 dataVersion 递增
- **测试**：测试内容覆盖：配置变更是否触发回调、ZooKeeper 断连后本地缓存是否生效、断连恢复后配置是否重新同步
