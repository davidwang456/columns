# 第1章：ZooKeeper 术语全景与架构原理

## 1. 项目背景

### 业务场景

假设你是一家电商公司的后端开发，团队正在搭建一套微服务架构。订单服务、库存服务、用户服务各自独立部署，但遇到了一个头疼的问题：**分布式协调**。

- **配置管理**：数据库连接串、限流阈值等配置项散落在每个服务的配置文件中，改一个参数得重启所有实例。
- **服务发现**：订单服务要调用库存服务，但库存服务的 IP 是动态的（容器化部署），如何让订单服务总能找到可用的库存服务实例？
- **分布式锁**：双十一秒杀，100 个订单服务实例同时扣减库存，怎么保证只有一个实例能扣减成功？

这些问题的共同本质是：**在分布式环境下，多个节点需要就某一状态达成一致**。ZooKeeper 正是为解决这类问题而生的分布式协调服务。

### 痛点放大

没有 ZooKeeper 这类协调服务之前，团队通常用数据库来实现分布式协调——建一张 `config` 表存配置，用 `SELECT ... FOR UPDATE` 做分布式锁。这种方式的问题很明显：

- **性能瓶颈**：数据库行锁在高并发下迅速成为热点，秒杀场景下数据库会被打垮
- **单点故障**：数据库本身也要做主从，主库切换期间协调能力完全丧失
- **缺乏通知机制**：配置改了，服务实例不会自动感知，需要轮询数据库
- **CAP 权衡**：数据库通常优先保证一致性（CP），在分区时拒绝写入，导致整个系统不可用

一个专门为分布式协调设计的服务，应该具备：高可用、顺序一致性、实时通知、高性能。ZooKeeper 就是这样的服务。

## 2. 项目设计

### 剧本式交锋对话

**场景**：会议室，小胖、小白、大师在讨论新系统的架构方案。

**小胖**：（啃着面包）我听说要引入一个叫 ZooKeeper 的东西？这不就跟食堂打饭排队一样吗？搞个号就行了，为啥要搞那么复杂？

**大师**：（笑）你这个比喻其实挺准的。食堂打饭确实是"协调"——每个人都得排队，谁先到谁先打。但分布式系统里，排队的人可能在不同楼层、不同的食堂窗口，怎么保证所有人看到的排队顺序是一样的？

**小白**：我理解 ZooKeeper 是一个"分布式协调服务"，但它具体协调什么？怎么协调？

**大师**：你可以把 ZooKeeper 想象成一个**超级公告板**。这个公告板有几个特点：

1. 它有多个副本（集群），任何一个副本坏了，其他副本继续服务
2. 你在公告板上写的东西，所有副本保证看到的内容是一样的
3. 你可以盯着公告板的某个区域看，一旦内容变了，公告板主动通知你

**小胖**：哦！就是 MySQL 主从同步呗，写到主库同步到从库？

**大师**：不是简单的 MySQL 主从。MySQL 主从是异步的——你写进主库，从库可能还没同步完。ZooKeeper 用的是**过半写入**：写请求必须得到超过半数的节点确认才算成功。这保证了你读任何一个节点，数据都是一致的。

**小白**：过半写入？那写性能是不是很慢？

**大师**：好问题。ZooKeeper 的写入性能并不慢——因为它**不需要等待所有节点**，只需要等待半数以上的节点确认。对于 3 节点集群，只需等 2 个节点；5 节点集群，等 3 个节点。而且 ZAB 协议是流式处理的，不像两阶段提交（2PC）那样阻塞。

> **技术映射**：食堂排队 = 顺序一致性，多窗口复制 = 集群 + Quorum 机制，盯着公告板 = Watcher 机制

**小白**：那 ZooKeeper 的架构是怎样的？Leader 和 Follower 分别做什么？

**大师**：ZooKeeper 集群采用 **Master-Slave 模式**，但称呼不同：

- **Leader**：集群的大脑，处理所有写请求，负责广播和提交数据
- **Follower**：处理读请求，参与投票（选举和写确认）
- **Observer**：只处理读请求，不参与投票（专门扩展读能力）

**小胖**：那 Leader 挂了咋办？没人写数据了？

**大师**：ZooKeeper 集群会自动选出一个新 Leader，这个过程叫 **Leader 选举**。ZooKeeper 用的是 FastLeaderElection 算法，3 节点集群通常 200ms 内就能选出新 Leader。选举期间集群仍然可以处理读请求，只是写入暂停。

> **技术映射**：Leader 挂掉 = 食堂大师傅走了，新大师傅通过投票选出来

**小白**：那 ZooKeeper 的数据模型是什么样的？和文件系统比呢？

**大师**：ZooKeeper 的数据模型是一个**树形命名空间**，和 Linux 文件系统非常像。每个节点叫 **ZNode**，每个 ZNode 可以存储一段数据（最大 1MB）。但和文件系统有几个关键区别：

- ZNode 的路径就是它的名字（不能重命名）
- ZNode 可以创建子节点，形成树结构
- ZNode 有类型：持久、临时、顺序等
- 每个 ZNode 都有版本号，用于乐观锁并发控制

> **技术映射**：文件系统 = ZNode 树形命名空间，文件类型 = ZNode 类型（持久/临时等），文件版本 = ZNode 版本号

## 3. 项目实战

### 环境准备

本章是概念入门，不需要安装 ZooKeeper。我们将绘制一张 ZooKeeper 整体架构图。

**工具**：Draw.io（在线）、Excalidraw、ProcessOn 任选其一

### 分步实现：绘制 ZooKeeper 架构图

#### 步骤 1：绘制集群角色图

画三个圆角矩形框，分别标注：Leader、Follower、Observer（Observer 用虚线边框表示"非投票成员"）。

#### 步骤 2：客户端连接

在集群下方画多个客户端图标，用箭头连接到 Leader/Follower/Observer。读请求（Read）用蓝色箭头直接打到 Follower/Observer，写请求（Write）用红色箭头，标出"写请求转发到 Leader"的流程。

```
[Client A] ──Read──> [Follower 1]
[Client B] ──Write──> [Follower 2] ──forward──> [Leader]
```

#### 步骤 3：ZAB 协议流程

在 Leader 右侧画一个流程框，展示一次写入：

```
Client Write → Leader → Proposal(广播到所有Follower)
                         → Follower返回ACK
                         → 收到多数ACK → Commit(广播COMMIT消息)
                         → 数据生效 → 返回Client
```

#### 步骤 4：数据模型树

画一棵倒置的树：

```
/ (根节点)
├── /services
│   ├── /services/user-service
│   │   ├── /services/user-service/192.168.1.1:8080 (临时节点)
│   │   └── /services/user-service/192.168.1.2:8080 (临时节点)
│   └── /services/order-service
├── /config
│   ├── /config/db_url
│   └── /config/redis_url
└── /locks
    └── /locks/seckill-lock-0000000001 (临时顺序节点)
```

#### 步骤 5：标注术语

在图四周加上术语解释框：

- **ZNode**：树上的每个节点，存储数据和元数据
- **Session**：客户端与服务器的 TCP 长连接
- **Watcher**：客户端在 ZNode 上注册的监听，节点变化时触发通知
- **ACL**：访问控制列表，控制谁可以对 ZNode 做什么操作
- **zxid**：事务 ID，全局唯一单调递增，用于保证写入顺序
- **epoch**：Leader 任期编号，每选举一次 +1

### 测试验证

用文字描述架构图的正确性：

1. **架构完整性检查**：图中是否包含了 Leader、Follower、Observer 三种角色？
2. **流程正确性检查**：写请求的转发、Proposal、ACK、Commit 链路是否完整标记？
3. **数据模型检查**：ZNode 树是否包含了不少于 3 层深度，是否同时包含持久节点和临时节点？

### 可能遇到的坑

- **混淆 CAP 理论**：ZooKeeper 是 CP 系统（保证一致性和分区容错性），不是 AP 系统。在极端网络分区下，ZooKeeper 可能会选择不可用来保证一致性。
- **把 ZooKeeper 当数据库用**：ZNode 数据限制 1MB，不适合存大数据。ZooKeeper 是协调服务，不是数据库。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 一致性 | 强一致性，ZAB 保证写入顺序 | 写性能受限于 Leader 单点 |
| 可用性 | 自动 Leader 选举，故障恢复快 | 选举期间写入不可用（一般 <200ms） |
| 可观测性 | 四字命令、JMX 监控丰富 | 缺少内置可视化控制台 |
| 使用成本 | 概念少、学习曲线平缓 | 1MB 数据限制，不适合大容量场景 |
| 生态 | Kafka、Dubbo、HBase 等广泛集成 | 不适合服务发现场景（已被 Nacos 等替代趋势） |

### 适用场景

- **配置中心**：小规模配置存储和动态通知
- **分布式锁**：CP 强一致性，适合对一致性要求高的场景
- **Leader 选举**：自动选主，保障单 Master 架构
- **命名服务**：全局唯一 ID 生成（顺序节点）
- **服务发现**：中小规模服务注册（大规模建议 Nacos/Consul）

**不适用场景**：
- 大规模数据存储（单节点 1MB 限制，总数据量受内存限制）
- 高并发写场景（单 Leader 瓶颈，写 QPS 通常 ~10万/s）

### 注意事项

- ZooKeeper 3.5+ 才支持容器节点、TTL 节点、动态重配置
- 客户端版本需和服务端版本兼容（推荐使用相同大版本）
- 不要将 ZooKeeper 暴露在公网，需配合防火墙或 TLS

### 常见踩坑经验

**故障 1：ZooKeeper 频繁选主**

现象：集群每隔几分钟就选举一次，客户端频繁断开重连。

根因：磁盘 IO 性能不足，事务日志写盘延迟过高，导致 Follower 与 Leader 之间的心跳超时。

**故障 2：ZNode 数据量过大导致 OOM**

现象：ZooKeeper 进程内存持续增长，最终 OOM Killer 杀掉进程。

根因：ZNode 数量和存储数据量过大（单集群几十万个节点），内存消耗殆尽。ZooKeeper 将所有数据存储在内存中。

**故障 3：Watcher 泄漏导致性能下降**

现象：客户端断开后，服务端 Watcher 列表不断增长。

根因：客户端异常断开时，服务端清理 Watcher 的逻辑存在竞态条件，部分 Watcher 未正确清理。

### 思考题

1. ZooKeeper 为什么选择 CP 而不是 AP？如果换成 AP 系统（比如 Eureka），在场景上有哪些不同？
2. 假设你有一个 3 节点 ZooKeeper 集群，网络分区导致 1 个节点被隔离，剩余 2 个节点还能正常服务吗？为什么？
