# 第10章：会话与连接管理——心跳与超时

## 1. 项目背景

### 业务场景

你的服务每天通过 Curator 连接到 ZooKeeper 集群，运行很稳定。但有一天，网络发生了一次短暂的抖动（1-2 秒），然后你的服务开始疯狂报错：

```
Session expired, closing connection...
```

更糟的是，注册在 ZooKeeper 上的临时节点全部被删除，其他服务发现你的服务"下线"了，流量不再分发过来。虽然你的服务其实还活着，但 ZooKeeper 不知道。

这个场景揭示了一个核心问题：**ZooKeeper 是如何判断客户端"活着"的？** 答案是**Session（会话）**。

### 痛点放大

Session 管理是 ZooKeeper 最容易被低估的复杂机制：

- **Session Timeout 设置太小**：正常 GC 停顿（几百毫秒）都可能导致会话过期
- **Session Timeout 设置太大**：客户端死了但 ZooKeeper 几分钟后才感知，服务发现不及时
- **连接断开 ≠ 会话过期**：连接断开后客户端自动重连，只要在 timeout 内重连成功，会话仍然有效
- **会话过期 ≠ 数据丢失**：过期后临时节点被删除，但持久节点还在

## 2. 项目设计

### 剧本式交锋对话

**场景**：小胖服务最近频繁出现 Session 过期，每次过期后都需要重新初始化连接池，业务受到严重影响。

**小胖**：我设的 Session Timeout 是 15 秒，但服务每隔几分钟就 Session 过期一次。日志显示 `Disconnected` 后一两秒就连上了，但会话过期了！

**大师**：Session 的过期时间是从 ZooKeeper 服务端**最后一次收到心跳**开始计算的。客户端断线期间，服务端收不到心跳，`tickTime` 的计数器继续走。如果断线时间超过 `Session Timeout`，服务端认定客户端死亡，清除会话和所有临时节点。

来看 Session 的生命周期：

```
客户端连接服务端
    → 服务端分配 sessionId，设置 timeout
    → 客户端定时发送心跳（PING）
    → 服务端刷新会话到期时间
    → 如果服务端超过 timeout 没收到心跳 → 会话过期
    → 如果客户端断线 → 客户端尝试重连
    → 如果重连成功且在 timeout 内 → 会话恢复
    → 如果重连成功但 timeout 已过 → SessionExpired
```

**小白**：那 Session Timeout 到底设多少合适？我看有推荐 2*tickTime 的，也有推荐 20*tickTime 的。

**大师**：Session Timeout 的范围由 ZooKeeper 服务端控制：

```
// 最小 = 2 * tickTime（默认 4s）
// 最大 = 20 * tickTime（默认 40s）
minSessionTimeout <= userValue <= maxSessionTimeout
```

如果你设了 60 秒，但服务端 `maxSessionTimeout=40000`（默认），实际生效的是 40 秒。

推荐值参考：

| 场景 | 推荐 Session Timeout | 理由 |
|------|---------------------|------|
| 开发环境 | 5-10 秒 | 快速发现断线，调试方便 |
| 生产（常规） | 15-30 秒 | 容忍正常 GC 抖动 |
| 生产（GC 频繁） | 30-60 秒 | 大堆内存 GC 可能达数秒 |
| 生产（非常重要） | 60-120 秒 | 宁可延迟感知，不让会话随意过期 |

**小胖**：那我控制台输出 `Session expired`，是不是会话过期了必须重新创建 `ZooKeeper` 对象？

**大师**：是的！Session 过期后，旧的 `ZooKeeper` 对象已经无效了。你必须创建一个新的 `ZooKeeper` 对象重新连接。

Curator 自动帮你处理了——当 Curator 检测到 `SessionExpired` 时，它会创建一个新的内部 `ZooKeeper` 对象，然后重新注册所有 Watcher 和锁。

**小白**：连接断开和会话过期到底是什么关系？我经常把这两个搞混。

**大师**：这是最核心的区别：

```
连接断开（Disconnected）    会话过期（Expired）
     ↓                              ↓
TCP 连接断了                  服务端认为客户端死了
     ↓                              ↓
客户端还在，正在重连          客户端必须重建会话
     ↓                              ↓
只要在 timeout 内重连成功，    临时节点已全部被删除
会话仍然有效
     ↓                              ↓
临时节点不会删除              需要重新注册所有临时节点
```

用一个比喻：**连接是电话线，会话是你的身份**。电话线断了（连接断开），只要在约定时间内重新拨通，你还是你（会话有效）。但如果你超时没有回电话，对方挂断后你就算再打过去，也需要重新介绍自己（会话过期）。

> **技术映射**：Session = 你在网吧的登录状态，连接 = 你和网吧电脑之间的网络。网线断了，只要 30 分钟内插回去，你还能继续上机（Session 有效）。超过 30 分钟，系统自动注销（Session 过期），需要重新登录。

**小胖**：那心跳（PING）的频率是多少？我能不能控制？

**大师**：心跳频率**不是**由 ZooKeeper 客户端配置控制的，它有一个固定算法：

```
心跳间隔 = Session Timeout / 3
```

如果你的 Session Timeout = 30 秒，客户端每 10 秒发送一次 PING。如果 ZooKeeper 服务端在 30 秒内没有收到任何消息（包括 PING、读写请求），则认为会话过期。

**小白**：那 "tickTime" 又是什么？它和 Session Timeout 的关系？

**大师**：`tickTime` 是 ZooKeeper 服务端的基本时间单位：

- 服务端**每隔 tickTime**（默认 2 秒）会检查一次会话的到期时间
- `minSessionTimeout = 2 * tickTime`（4 秒）
- `maxSessionTimeout = 20 * tickTime`（40 秒）
- 心跳超时检测：`syncLimit * tickTime`（默认 10 秒）

所以检查会话是否过期的精度是 tickTime 级别（2 秒），不是更细的毫秒级。

## 3. 项目实战

### 环境准备

- ZooKeeper 3.9.x 单机模式（便于模拟断连）
- JDK 11+
- Maven

### 分步实现

#### 步骤 1：观察 Session 生命周期

创建 `SessionLifecycleDemo.java`：

```java
package com.zkdemo;

import org.apache.zookeeper.*;
import org.apache.zookeeper.data.Stat;

import java.util.concurrent.CountDownLatch;
import java.util.concurrent.TimeUnit;

public class SessionLifecycleDemo {
    private static final String ZK_URL = "127.0.0.1:2181";

    public static void main(String[] args) throws Exception {
        System.out.println("=== ZooKeeper Session 生命周期演示 ===");
        System.out.println("Session Timeout 设置为 10 秒\n");

        CountDownLatch connectedLatch = new CountDownLatch(1);
        CountDownLatch expiredLatch = new CountDownLatch(1);

        // 1. 创建客户端连接（Session Timeout = 10 秒）
        ZooKeeper zk = new ZooKeeper(ZK_URL, 10000, event -> {
            System.out.println("[Watcher] 状态变化: " + event.getState()
                    + " | 类型: " + event.getType()
                    + " | 路径: " + event.getPath());

            switch (event.getState()) {
                case SyncConnected:
                    System.out.println("[事件] 连接成功！");
                    connectedLatch.countDown();
                    break;
                case Disconnected:
                    System.out.println("[事件] 连接断开！");
                    break;
                case Expired:
                    System.out.println("[事件] 会话过期！！！");
                    expiredLatch.countDown();
                    break;
            }
        });

        // 2. 等待连接建立
        connectedLatch.await();
        long sessionId = zk.getSessionId();
        System.out.println("\n会话已建立，sessionId: " + Long.toHexString(sessionId));

        // 3. 创建临时节点
        zk.create("/session-test", "temp".getBytes(),
                ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.EPHEMERAL);
        System.out.println("临时节点 /session-test 已创建");

        // 4. 查看当前会话状态
        System.out.println("\n当前客户端状态: " + zk.getState());

        // 5. 模拟网络断开——关闭 ZooKeeper 服务端或在防火墙 block 端口
        System.out.println("\n========= 模拟场景 1: 连接断开后快速恢复 =========");
        System.out.println("请在 8 秒内重启 ZooKeeper 或恢复网络连接...");
        System.out.println("（如果 8 秒内恢复，临时节点不会删除）");

        // 等待 8 秒（模拟短时间断连）
        Thread.sleep(8000);

        if (zk.getState() == ZooKeeper.States.CONNECTED) {
            System.out.println("\n客户端已重连成功！");
            Stat stat = zk.exists("/session-test", false);
            System.out.println("临时节点 /session-test 存在: " + (stat != null));
        } else {
            System.out.println("\n客户端仍在断连状态或已过期...");
        }

        // 6. 模拟 Session 过期
        System.out.println("\n========= 模拟场景 2: 会话过期 =========");
        System.out.println("正在等待会话过期（可能需要 10-20 秒）...");

        // 等待会话过期
        boolean expired = expiredLatch.await(30, TimeUnit.SECONDS);
        if (expired) {
            System.out.println("\n会话已过期，验证临时节点是否被删除...");
        } else {
            System.out.println("\n未等到会话过期事件，手动关闭...");
        }

        zk.close();
        System.out.println("\n演示完成");
    }
}
```

**运行说明**：这个程序需要在运行过程中手动停止 ZooKeeper 服务来模拟断连。

#### 步骤 2：Curator 连接状态监听

创建 `ConnectionStateDemo.java`：

```java
package com.zkdemo;

import org.apache.curator.RetryPolicy;
import org.apache.curator.framework.CuratorFramework;
import org.apache.curator.framework.CuratorFrameworkFactory;
import org.apache.curator.framework.state.ConnectionState;
import org.apache.curator.framework.state.ConnectionStateListener;
import org.apache.curator.retry.ExponentialBackoffRetry;

import java.util.concurrent.CountDownLatch;

public class ConnectionStateDemo {
    public static void main(String[] args) throws Exception {
        System.out.println("=== Curator 连接状态监听演示 ===\n");

        RetryPolicy retryPolicy = new ExponentialBackoffRetry(1000, 3);
        CuratorFramework client = CuratorFrameworkFactory.builder()
                .connectString("127.0.0.1:2181")
                .sessionTimeoutMs(10000)   // 10 秒
                .connectionTimeoutMs(5000)  // 5 秒
                .retryPolicy(retryPolicy)
                .build();

        // 注册连接状态监听器
        client.getConnectionStateListenable().addListener(new ConnectionStateListener() {
            @Override
            public void stateChanged(CuratorFramework client, ConnectionState newState) {
                System.out.println("[ConnectionStateListener] 状态变更: " + newState);

                switch (newState) {
                    case CONNECTED:
                        System.out.println("  → 首次连接成功");
                        break;
                    case RECONNECTED:
                        System.out.println("  → 断线后重连成功（会话未过期！）");
                        break;
                    case LOST:
                        System.out.println("  → 连接丢失（正尝试重连）");
                        break;
                    case SUSPENDED:
                        System.out.println("  → 连接挂起（可能网络抖动）");
                        break;
                    case READ_ONLY:
                        System.out.println("  → 只读模式（集群发生分区）");
                        break;
                }
            }
        });

        client.start();
        System.out.println("Curator 客户端已启动");

        // 模拟网络故障
        System.out.println("\n请手动停止 ZooKeeper 服务模拟断连...");
        System.out.println("然后在 10 秒内重新启动 ZooKeeper 服务...");
        System.out.println("观察 CONNECTED → LOST → RECONNECTED 状态变化\n");

        // 保持运行
        CountDownLatch latch = new CountDownLatch(1);
        Runtime.getRuntime().addShutdownHook(new Thread(() -> {
            System.out.println("\n关闭客户端...");
            client.close();
            latch.countDown();
        }));
        latch.await();
    }
}
```

**预期输出（模拟断连再恢复）**：

```
=== Curator 连接状态监听演示 ===

Curator 客户端已启动
[ConnectionStateListener] 状态变更: CONNECTED
  → 首次连接成功

请手动停止 ZooKeeper 服务模拟断连...
[ConnectionStateListener] 状态变更: SUSPENDED
  → 连接挂起（可能网络抖动）
[ConnectionStateListener] 状态变更: LOST
  → 连接丢失（正尝试重连）

重启 ZooKeeper 后...
[ConnectionStateListener] 状态变更: RECONNECTED
  → 断线后重连成功（会话未过期！）
```

#### 步骤 3：Session 过期后的正确处理

创建 `SessionExpireHandlerDemo.java`：

```java
package com.zkdemo;

import org.apache.curator.RetryPolicy;
import org.apache.curator.framework.CuratorFramework;
import org.apache.curator.framework.CuratorFrameworkFactory;
import org.apache.curator.framework.recipes.locks.InterProcessMutex;
import org.apache.curator.framework.state.ConnectionState;
import org.apache.curator.framework.state.ConnectionStateListener;
import org.apache.curator.retry.ExponentialBackoffRetry;

import java.util.concurrent.atomic.AtomicBoolean;

public class SessionExpireHandlerDemo {
    private static final AtomicBoolean isHealthy = new AtomicBoolean(true);

    public static void main(String[] args) throws Exception {
        RetryPolicy retryPolicy = new ExponentialBackoffRetry(1000, 5);
        CuratorFramework client = CuratorFrameworkFactory.builder()
                .connectString("127.0.0.1:2181")
                .sessionTimeoutMs(10000)
                .retryPolicy(retryPolicy)
                .build();

        // 监听会话过期
        client.getConnectionStateListenable().addListener((c, newState) -> {
            if (newState == ConnectionState.LOST) {
                System.out.println("连接丢失，标记服务不可用");
                isHealthy.set(false);
            } else if (newState == ConnectionState.RECONNECTED) {
                System.out.println("重连成功，标记服务可用");
                isHealthy.set(true);
                // 重新注册服务节点
                reRegisterService(c);
            }
        });

        client.start();
        System.out.println("服务已启动，健康状态: " + isHealthy.get());

        // 获取分布式锁
        InterProcessMutex lock = new InterProcessMutex(client, "/locks/my-lock");

        // 模拟健康检查和自愈
        new Thread(() -> {
            while (true) {
                try {
                    Thread.sleep(5000);
                    if (isHealthy.get()) {
                        if (lock.acquire(1, java.util.concurrent.TimeUnit.SECONDS)) {
                            try {
                                System.out.println("获取锁成功，执行任务...");
                            } finally {
                                lock.release();
                            }
                        }
                    } else {
                        System.out.println("服务不健康，跳过任务");
                    }
                } catch (Exception e) {
                    System.out.println("操作失败: " + e.getMessage());
                }
            }
        }).start();

        Thread.sleep(60000);
        client.close();
    }

    private static void reRegisterService(CuratorFramework client) {
        try {
            // 会话过期后临时节点被删除，重连后需要重新注册
            client.create().orSetData()
                    .creatingParentsIfNeeded()
                    .forPath("/services/my-service/instance-1", "127.0.0.1:8080".getBytes());
            System.out.println("服务节点已重新注册");
        } catch (Exception e) {
            System.out.println("服务注册失败: " + e.getMessage());
        }
    }
}
```

### 可能遇到的坑

| 问题 | 原因 | 解决 |
|------|------|------|
| `Session expired` 频繁出现 | Session Timeout 太小 | 增大 Session Timeout |
| 临时节点未及时删除 | Session Timeout 太长 | 减小 Session Timeout |
| 断连后 Watcher 不生效 | 连接恢复后 Watcher 未重新注册 | Curator Cache 自动处理 |
| 重连后锁状态丢失 | 会话过期后锁节点已删除 | 使用 `ConnectionStateListener` 重新初始化 |

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 心跳机制 | 自动维护，无需客户端干预 | 心跳间隔无法自定义 |
| 会话恢复 | 断连重连后会话可恢复 | 恢复期间的操作全部失败 |
| 临时节点 | 会话过期自动清理，避免僵尸数据 | 需要重新注册 |
| Curator 封装 | 自动处理重连和会话恢复 | 内部逻辑复杂，排查困难 |

### 适用场景

- **常规开发**：Session Timeout 15-30s
- **GC 停顿大的应用**：Session Timeout 30-60s（如大堆内存 Java 应用）
- **网络不稳定的环境**：Session Timeout 60-120s

**不适用场景**：
- 需要秒级服务发现（Session Timeout 本身至少 4 秒）

### 注意事项

- Session Timeout 是服务端强制执行的范围：[2*tickTime, 20*tickTime]
- 客户端心跳频率 = Session Timeout / 3，无法修改
- `zookeeper.session.timeout.ms` 的默认值是 40 秒（Curator/原生 API 各不相同）
- 容器环境中 GC 停顿更频繁，建议适当增大 Session Timeout

### 常见踩坑经验

**故障 1：Full GC 导致 Session 过期**

现象：应用每隔几小时出现一次 Session 过期，和 Full GC 周期吻合。

根因：应用堆内存 8G，Fully GC 耗时 3-5 秒，超过了 Session Timeout（10 秒内可能多次 GC 停顿）。解决方案：增大 Session Timeout 到 30 秒，或使用 ZGC 减少 GC 停顿。

**故障 2：Session 过期后临时节点错误恢复**

现象：会话过期重连后，应用尝试删除旧的临时节点（以为是自己创建的），结果报了 NoAuth 错误。

根因：会话过期后，旧会话创建的临时节点已被服务端清理。新的会话需要用新的路径重新注册，不能复用旧的路径或试图操作旧节点。

### 思考题

1. 假设一个客户端设置了 Session Timeout = 30 秒。客户端断线后 20 秒重连成功，此时会话是否应该恢复？如果断线 35 秒后重连成功呢？为什么？
2. 在网络分区场景下，ZooKeeper 集群中的少数派节点无法联系 Leader。如果客户端正好连接在少数派节点上，会发生什么？客户端应该如何处理？

### 推广计划提示

- **开发**：生产环境 Session Timeout 至少设置为 15-30 秒，开发环境可适当减小
- **运维**：通过四字命令 `stat` 或 JMX 查看当前活跃会话数，异常增长说明可能有连接泄漏
- **测试**：在故障注入测试中模拟网络断开和恢复，验证服务的重连和自愈能力
