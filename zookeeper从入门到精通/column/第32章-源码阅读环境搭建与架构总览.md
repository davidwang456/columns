# 第32章：源码阅读环境搭建与架构总览

## 1. 项目背景

### 业务场景

经过基础篇和中级篇的学习，你已经掌握了 ZooKeeper 的使用、集群部署、分布式原语和性能调优。现在你从"使用者"向"贡献者"迈进——阅读 ZooKeeper 源码，理解它内部是如何工作的。

本章的目标是搭建 ZooKeeper 源码阅读环境，并浏览整体源码架构，为后续的源码分析章节做准备。

### 痛点放大

直接阅读 ZooKeeper 源码的困难：

- **代码量大**：zookeeper-server 模块 1000+ Java 源文件
- **包结构复杂**：20+ 核心包，分散在不同模块中
- **协议细节隐蔽**：ZAB 协议、选举机制等的核心逻辑分布在多个文件中
- **缺少入口引导**：不知道从哪里开始读

## 2. 项目设计

### 剧本式交锋对话

**场景**：小胖想读 ZooKeeper 源码，但不知道从哪里开始。

**小胖**：我下载了 ZooKeeper 源码，解压后看到一大堆目录，完全不知道从哪里开始看。

**大师**：ZooKeeper 源码主要分为这几个模块：

```
zookeeper/
├── zookeeper-server/        ← 核心实现（重点阅读）
│   ├── src/main/java/org/apache/zookeeper/
│   │   ├── server/           ← 服务端核心
│   │   │   ├── quorum/       ← 集群协议（ZAB、选举）
│   │   │   └── persistence/  ← 数据持久化
│   │   └── client/           ← 客户端实现
├── zookeeper-recipes/       ← Curator recipes
├── zookeeper-docs/          ← 官方文档
├── zookeeper-it/            ← 集成测试
└── zookeeper-jute/          ← 序列化框架（Jute）
```

**小白**：那启动流程的入口在哪里？ZooKeeper 是怎么启动起来的？

**大师**：启动入口是 `QuorumPeerMain.main()`：

```
QuorumPeerMain.main()
  └── QuorumPeerConfig.parse()    ← 解析 zoo.cfg
  └── ServerCnxnFactory.create()  ← 创建网络层
  └── ZooKeeperServer.init()      ← 初始化服务端
  └── QuorumPeer.start()          ← 启动集群节点
       ├── startLeaderElection()  ← 开始选举
       ├── super.start()          ← 启动线程
       └── run()                  ← 主循环
            ├── LOOKING  → lookForLeader()  ← 选举
            ├── LEADING  → lead()          ← Leader 角色
            └── FOLLOWING → followLeader() ← Follower 角色
```

## 3. 项目实战

### 环境准备

- JDK 17+
- IntelliJ IDEA（推荐 Ultimate 版）
- Git
- Maven 3.8+

### 分步实现

#### 步骤 1：获取和编译源码

```bash
# 1. 克隆官方仓库
git clone https://github.com/apache/zookeeper.git
cd zookeeper

# 2. 切换到 3.9 分支
git checkout branch-3.9

# 3. 编译（跳过测试，加快速度）
mvn clean install -DskipTests -Dspotbugs.skip=true \
  -Dcheckstyle.skip=true -Drat.skip=true

# 4. 验证编译成功
# 查看编译后的 JAR
ls zookeeper-server/target/zookeeper-*.jar

# 5. 导入 IntelliJ IDEA
# File → Open → 选择 zookeeper 目录
# IntelliJ 会自动识别 Maven 项目
```

#### 步骤 2：关键包结构导览

```bash
# 核心包结构
cd zookeeper-server/src/main/java/org/apache/zookeeper/

# 1. 服务端核心
server/
├── ZooKeeperServer.java        ← 服务端主类
├── FinalRequestProcessor.java   ← 请求处理管道终点
├── PrepRequestProcessor.java    ← 预处理请求
├── SyncRequestProcessor.java    ← 同步事务日志
├── DataTree.java                ← 内存数据库
├── DataNode.java                ← ZNode 节点
├── ServerCnxn.java              ← 客户端连接抽象
├── NIOServerCnxn.java           ← NIO 连接实现
└── SessionTrackerImpl.java      ← 会话管理

# 2. 集群协议
server/quorum/
├── QuorumPeer.java              ← 集群节点主类
├── Leader.java                  ← Leader 角色
├── Follower.java                ← Follower 角色
├── Observer.java                ← Observer 角色
├── FastLeaderElection.java      ← FLE 选举算法
├── QuorumCnxManager.java        ← 选举网络通信
└── Learner.java                 ← Learner（Follower/Observer）基类

# 3. 持久化
server/persistence/
├── FileTxnLog.java              ← 事务日志
├── FileSnap.java                ← 快照
└── Util.java                    ← 工具方法

# 4. 客户端
client/
├── ZooKeeper.java               ← 客户端主类
├── ClientCnxn.java              ← 客户端连接
├── SendThread.java              ← 发送线程
└── EventThread.java             ← 事件线程
```

#### 步骤 3：在 IDE 中启动源码版 ZooKeeper

创建 IDEA Run Configuration：

**Main Class**：`org.apache.zookeeper.server.quorum.QuorumPeerMain`

**VM Options**：
```bash
-Dzookeeper.log.dir=/tmp/zk-logs
-Dzookeeper.root.logger=DEBUG,CONSOLE
-Dzookeeper.serverCnxnFactory=org.apache.zookeeper.server.NIOServerCnxnFactory
```

**Program Arguments**：`conf/zoo.cfg`

**Working Directory**：`zookeeper-server/`

**或者直接运行单机 ZooKeeper**：


#### 步骤 5：代码追踪——一条写入请求的完整源码链路

```
客户端发送 setData("/config", "new-value")

ZooKeeper 服务端入口：
  NIOServerCnxn.doIO()
  → readRequest()
  → ZooKeeperServer.processRequest()
  → PrepRequestProcessor.processRequest()
  → SyncRequestProcessor.processRequest()
  → ProposalRequestProcessor.processRequest()  [Leader only]
  → CommitProcessor.processRequest()
  → FinalRequestProcessor.processRequest()
  → DataTree.setData()           ← 实际修改数据
  → WatchManager.triggerWatch()  ← 触发 Watcher
  → NIOServerCnxn.sendResponse() ← 返回响应
```

#### 步骤 7：阅读路线建议

```bash
# 推荐阅读顺序

# 第 1 步：启动流程
QuorumPeerMain.main() → QuorumPeer.start() → 角色初始化

# 第 2 步：请求处理管道（第 33 章详述）
requestPath: Prep → Sync → Proposal(Leader) → Commit → Final

# 第 3 步：Leader 选举（第 34 章详述）
FastLeaderElection.lookForLeader() → QuorumCnxManager

# 第 4 步：ZAB 协议（第 35 章详述）
Leader.propose() → Follower.processPacket() → Leader.processAck()

# 第 5 步：会话管理（第 36 章详述）
SessionTrackerImpl → touchSession() → expire()

# 第 6 步：数据管理（第 37 章详述）
DataTree → FileTxnLog → FileSnap

# 第 7 步：Watcher 机制（第 38 章详述）
WatchManager → triggerWatch() → ServerCnxn.process()
```

### 可能遇到的坑

| 问题 | 原因 | 解决 |
|------|------|------|
| IDEA 编译报错 | JDK 版本不匹配 | 使用 JDK 17+ |

## 4. 项目总结

### ZooKeeper 源码架构总览

```
┌──────────────────────────────────────────────────────────┐
│                      ZooKeeper 集群                       │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │                  QuorumPeer                        │  │
│  │  (集群节点主类，管理角色状态机)                       │  │
│  │  状态: LOOKING → LEADING / FOLLOWING / OBSERVING   │  │
│  └──┬───────────────┬───────────────┬────────────────┘  │
│     │               │               │                    │
│  ┌──▼──┐       ┌────▼────┐    ┌────▼────┐              │
│  │Leader│       │Follower │    │Observer │              │
│  │      │       │         │    │         │              │
│  │·ZAB  │       │·只读+投票│    │·只读    │              │
│  │·广播  │       │·转发写入  │    │·转发写入 │              │
│  └──┬──┘       └────┬────┘    └────┬────┘              │
│     │               │               │                    │
│  ┌──▼───────────────▼───────────────▼────────────────┐  │
│  │            RequestProcessor 管道                   │  │
│  │  PrepRequest → SyncRequest → Commit → Final       │  │
│  └──────────────────────┬───────────────────────────┘  │
│                         │                              │
│  ┌──────────────────────▼───────────────────────────┐  │
│  │                    DataTree                       │  │
│  │  (内存数据库: ConcurrentHashMap<路径, DataNode>)   │  │
│  └──────────────────────┬───────────────────────────┘  │
│                         │                              │
│  ┌──────────────────────▼───────────────────────────┐  │
│  │            FileTxnLog + FileSnap                  │  │
│  │  (持久化: 事务日志 + 快照)                        │  │
│  └──────────────────────────────────────────────────┘  │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │            FastLeaderElection                       │  │
│  │  (选举算法: epoch > zxid > sid)                    │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

### 注意事项

- ZooKeeper 源码约 50 万行 Java 代码，不需要全部阅读

- 建议在 IDE 中 Enable "Mute Breakpoints" 来过滤不感兴趣的断点

### 思考题

1. ZooKeeper 的 `QuorumPeer` 类中有一个 `peerState` 变量，它的值在 LOOKING、LEADING、FOLLOWING、OBSERVING 之间流转。除了这几种状态，还有哪些状态？分别在什么情况下进入？
2. `RequestProcessor` 链在 Leader 和 Follower 上是否有不同？Leader 上增加了 `ProposalRequestProcessor`，Follower 上没有。这个区别说明了什么？
