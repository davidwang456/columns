# 第7章：数据类型转换与内存优化入门

## 1. 项目背景

某互联网公司的数据平台团队维护着一张用户画像宽表（user_profile_wide.csv），涵盖全平台 500 万用户，包含用户 ID、注册时间、年龄、性别、城市、会员等级、累计消费、近 30 日活跃天数、设备品牌、注册渠道等 25 个字段。这张表每天被 30 多个下游任务引用（推荐算法、运营分析、风控模型等）。

数据工程师小陈最近遇到了一个棘手的性能问题：

**痛点一：读取一张 2GB 的 CSV 需要 4 分钟**。小陈用 `pd.read_csv` 读取时观察任务管理器，内存峰值接近 8GB——CSV 文件的大小和内存中的 DataFrame 大小不是 1:1 关系。更糟的是他在 16GB 内存的笔记本电脑上跑这个任务时，系统开始疯狂交换内存（swap），鼠标都动不了。

**痛点二：所有字符串列都是 `object` dtype**。`object` 类型本质上存的是 Python 对象指针，每条记录都是指向堆中字符串对象的指针——500 万行 × 15 个字符串列，内存开销巨大。小陈疑惑："为什么不能像数据库那样用一个紧凑的字符串类型？"

**痛点三：年龄列是 float 类型**。因为数据源中有少量用户年龄缺失，导入时自动变成了 `float64`（因为 `int64` 不支持 NaN）。结果年龄变成了 `28.0`、`35.0` 这种形式，在前端展示很别扭。更糟的是，`float64` 占 8 字节，而年龄只需要 0-120 的整数——用 `int8` 只占 1 字节。

痛点流程：

```
用户画像 CSV (2GB)
  ↓ read_csv 直接读
内存占用 8GB+，读取 4 分钟
  ↓
所有字符串 = object (Python 指针，昂贵)
年龄 = float64 (8 字节，浪费)
会员等级 = int64 (8 字节，0-4 足够了)
  ↓
笔记本爆内存 → swap → 任务失败
```

本章将系统讲解 pandas 的数据类型体系、类型转换方法和内存优化策略，将 2GB 的用户画像数据压缩到可在笔记本上分析。

## 2. 项目设计：剧本式交锋对话

> 场景：小陈在技术分享会上演示了一张幻灯片"我们的用户画像表快把机器搞死了"，引起了讨论。

**小胖**："2GB 数据很大吗？我电脑游戏都装了好几百 GB，2GB 不是小意思？"

**大师**："存储大小和运行时内存是两回事。CSV 文件 2GB 存的是文本——比如一个整数 '1000000' 占 8 个字符（8 字节），但读进 pandas 变成 `int64` 只占 8 字节。看起来一样？别急——字符串 '北京市朝阳区望京街道' 在 CSV 里占约 13 字节（UTF-8），但读进 `object` 列里，除了字符串本身，还有一个 8 字节的指针、Python 对象的开销（约 49 字节）——总共约 70 字节。"

**小胖**："什么？多了 5 倍？"

**大师**："对。这就是为什么 CSV 2GB 读进来可能是 8GB——数字类字段没膨胀甚至压缩了，但字符串列严重膨胀。如果改用 `string[pyarrow]` dtype，内存占用可以降回到接近原始文本的水平。"

**【技术映射：object dtype = 每个人都随身带了一个仓库钥匙（指针）——钥匙本身也占空间】**

**小白**："你说的 `string[pyarrow]` 是什么？和普通 `object` 有啥区别？"

**大师**："pandas 有几种主要的字符串存储方式：

| dtype | 底层实现 | 缺失值支持 | 内存效率 |
|-------|---------|-----------|---------|
| `object` | Python 对象数组 | None | 低（每个值一个 Python 对象） |
| `string` | pandas StringArray | pd.NA | 中 |
| `string[pyarrow]` | PyArrow 字符串数组 | pd.NA | 高（列式连续存储） |

`string[pyarrow]` 利用了 Apache Arrow 的列式存储格式——所有字符串连续存放在一块内存里，一次读取整块而不是逐个对象跳转。对于 500 万行数据，`object` → `string[pyarrow]` 通常能节省 50-70% 的内存。"

**【技术映射：string[pyarrow] = 书店的连续书架（分类排列），object = 散落在读者手中的书（分散在各处）】**

**小白**："那年龄为什么是 float64 而不是 int？我看到数据里年龄没有小数点啊。"

**大师**："这是 pandas 的一个经典设计决策——`int64`（注意小写的 i）是 NumPy 原生类型，**不支持 NaN**。当数据中有缺失值时，pandas 只能退而求其次用 `float64`（支持 NaN）。解决方案是 pandas 1.0+ 引入的 **nullable integer**——`Int64`（大写 I），它用 `pd.NA` 而非 `NaN` 表示缺失，完美解决了整数 + 缺失值的矛盾。"

**小胖**："那我到底应该用 int64 还是 Int64？"

**大师**："规则很简单：
- **数据永远不会有缺失值** → 用 `int64`（小写），与 NumPy 生态兼容更好，计算略快；
- **数据可能有缺失值** → 用 `Int64`（大写），保留整数语义和缺失值支持。
- **如果在用 PyArrow 引擎** → 统一用 `int64[pyarrow]` 或相关 Arrow 类型。"

**【技术映射：Int64 = 考场座位号——有人缺考座位空着（pd.NA），但座位号本身是整数】**

**小胖**："那 category 类型呢？我听说明星电商都在用它压内存？"

**大师**："`category` 是 pandas 的'杀手锏'类型。它适合**低基数（唯一值少）的字符串列**。比如你的：
- `gender` 只有 '男'/'女'/'未知' 三个值
- `member_level` 只有 0-4 五个等级
- `city` 可能有几百个城市

`category` 的内部原理是一个整数编码数组（存 0、1、2...）加一个映射表（categories：'北京'→0，'上海'→1...）。500 万行 '男'/'女' 数据：
- `object` 存法：500 万个 Python 字符串对象 → 约 250 MB
- `category` 存法：500 万个 int8 编码 + 一个 2 元素映射表 → 约 5 MB

**内存压缩 50 倍。**"

**【技术映射：category = 餐厅菜单——顾客点菜只需说编号（整数编码），不需要重复说完整的菜名】**

**小白**："但我听说 category 有坑？"

**大师**："对，三个坑要记住：
1. `category` 的加法/乘法等算术运算不支持——它是分类数据，不是数值；
2. `category` 在 `groupby` 时 `observed=True` 很重要，否则会把未出现过的类别也输出一行 0；
3. 高基数列（如用户 ID，500 万个唯一值）千万别用 category——500 万整数编码 + 500 万个字符串的映射表，比直接存字符串还大。"

**大师总结**："今天讲的 dtype 优化，总结起来就是四项'军规'：
1. 有缺失值的整数 → `Int64` 而非 `float64`
2. 字符串列 → `string[pyarrow]` 而非 `object`
3. 低基数字符串 → `category`
4. 所有操作前 → 先用 `memory_usage(deep=True)` 看内存分布

下面我们就把一份 2GB 的用户画像压缩到笔记本能用的大小。"

import numpy as np

#### 步骤 2：数值类型优化

**目标**：将 int64→合适的小 int 类型，float64→Int64/nullable int。

```python
# step2_numeric_optimize.py
import pandas as pd
import numpy as np

df = pd.read_csv('user_profile_wide.csv', parse_dates=['reg_date'])

# === 优化 1：年龄 float64 → Int8（支持缺失值） ===
# 先检查取值范围
print(f"age: min={df['age'].min():.0f}, max={df['age'].max():.0f}, nulls={df['age'].isna().sum()}")

# float → nullable integer (Int8 = 1 字节，范围 -128~127 够用)
df['age'] = df['age'].astype('Int8')  # 大写 I 的 Int8 支持 NA
print(f"  age dtype: {df['age'].dtype}")

# === 优化 2：member_level int64 → int8 ===
# 会员等级 0-4，不需要 8 字节
print(f"member_level: unique={df['member_level'].nunique()}")
df['member_level'] = df['member_level'].astype('int8')
print(f"  member_level dtype: {df['member_level'].dtype}")

# === 优化 3：active_days_30d int64 → uint8 ===
# 0-30 天，用无符号 1 字节
df['active_days_30d'] = df['active_days_30d'].astype('uint8')

# === 优化 4：score float64 → float32 ===
# 评分精度不需要 float64
df['score'] = df['score'].astype('float32')

# === 优化 5：has_order bool → bool (已经是 bool 通常不需要改) ===

# === 查看优化后内存 ===
mem_after_numeric = df.memory_usage(deep=True).sum()
print(f"\n数值列优化后内存: {mem_after_numeric / 1024**2:.2f} MB")
```

**常用数值 dtype 选择参考表**：

| 数据类型 | 推荐 dtype | 范围 | 字节 |
|---------|-----------|------|------|
| 年龄、评分（0-100）| `int8` / `Int8` | -128~127 | 1 |
| 天数、次数（0-365）| `int16` / `uint16` | 0~65535 | 2 |
| 金额、用户ID | `int32` / `float32` | ±21 亿 | 4 |
| 大数据量 ID | `int64` / `float64` | ±9e18 | 8 |

#### 步骤 3：字符串和分类类型优化

**目标**：object → string[pyarrow]，低基数列 → category。

```python
# step3_string_category.py
import pandas as pd

df = pd.read_csv('user_profile_wide.csv', parse_dates=['reg_date'])

# 检查每列唯一值比例
print("===== 列唯一值基数分析 =====")
for col in df.select_dtypes(include='object').columns:
    unique_ratio = df[col].nunique() / len(df) * 100
    print(f"  {col}: {df[col].nunique()} unique ({unique_ratio:.2f}%), "
          f"top3: {df[col].value_counts().head(3).index.tolist()}")

# === 策略：低基数 → category，高基数字符串 → string[pyarrow] ===
CAT_THRESHOLD = 0.05  # 唯一值比例 < 5% 的用 category

for col in df.select_dtypes(include='object').columns:
    unique_ratio = df[col].nunique() / len(df)
    if unique_ratio < CAT_THRESHOLD:
        df[col] = df[col].astype('category')
        print(f"  → {col} → category")
    else:
        # 改用 string[pyarrow] (需 pyarrow 已安装)
        try:
            df[col] = df[col].astype('string[pyarrow]')
            print(f"  → {col} → string[pyarrow]")
        except Exception:
            df[col] = df[col].astype('string')
            print(f"  → {col} → string (fallback)")

# 查看优化后的 dtypes
print("\n===== 优化后 dtypes =====")
memory_before = 0  # 需要与步骤1对比
memory_after = df.memory_usage(deep=True).sum()
print(f"优化后内存: {memory_after / 1024**2:.2f} MB")
```

#### 步骤 4：convert_dtypes 自动优化 + 基准对比

**目标**：使用 `convert_dtypes()` 方法自动选择合适的 dtype。

```python
# step4_auto_convert.py
import pandas as pd
import numpy as np
import time

df_raw = pd.read_csv('user_profile_wide.csv', parse_dates=['reg_date'])

# 测量原始内存
mem_raw = df_raw.memory_usage(deep=True).sum()

# === 方法一：convert_dtypes 自动推断 ===
df_auto = df_raw.convert_dtypes()
mem_auto = df_auto.memory_usage(deep=True).sum()

# === 方法二：手动精细化优化 ===
df_manual = df_raw.copy()

# 数值优化
df_manual['age'] = df_manual['age'].astype('Int8')
df_manual['member_level'] = df_manual['member_level'].astype('int8')
df_manual['active_days_30d'] = df_manual['active_days_30d'].astype('uint8')
df_manual['last_login_days'] = df_manual['last_login_days'].astype('int16')
df_manual['score'] = df_manual['score'].astype('float32')

# 字符串优化
for col in df_manual.select_dtypes(include='object').columns:
    n_unique = df_manual[col].nunique()
    if n_unique / len(df_manual) < 0.05:
        df_manual[col] = df_manual[col].astype('category')
    else:
        df_manual[col] = df_manual[col].astype('string[pyarrow]')

mem_manual = df_manual.memory_usage(deep=True).sum()

# 对比报告
print("=" * 55)
print(f"{'方案':<15} {'内存(MB)':>10} {'压缩比':>10}")
print("-" * 55)
print(f"{'原始 read_csv':<15} {mem_raw/1024**2:>10.2f} {'--':>10}")
print(f"{'convert_dtypes':<15} {mem_auto/1024**2:>10.2f} {mem_raw/mem_auto:>9.1f}x")
print(f"{'手动精细化':<15} {mem_manual/1024**2:>10.2f} {mem_raw/mem_manual:>9.1f}x")
print("=" * 55)
```

### 3.5 常见坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| `string[pyarrow]` 报错 `ImportError` | 未安装 pyarrow | `pip install pyarrow` 或降级使用 `string` |
| `category` 列无法做算术运算 | category 是分类类型 | 先 `astype(int)` 转数值再运算 |
| `Int8`（大写）遇到除 0 返回 NaN | nullable int 支持缺失 | 用 `.fillna(0)` 处理后再运算 |
| `convert_dtypes()` 把 int 转成 Int64 而非更小的 | 保守策略，不猜测范围 | 手动指定 `astype('int8')` 等 |
| `to_numeric` 报错 `ValueError` | 字符串中含特殊字符 | 加 `errors='coerce'` 将无效值转 NaN |
| `memory_usage(deep=True)` 很慢 | 需要遍历 object 列计算每个字符串大小 | 只在开发阶段用，生产脚本去掉 |

### 3.6 测试验证

```python
# test_ch07.py
import pandas as pd
import numpy as np

def test_memory_reduction():
    """验证 dtype 优化后内存明显下降"""
    df = pd.DataFrame({
        'id': range(10000),
        'city': np.random.choice(['北京','上海','广州'], 10000),
        'val': np.random.random(10000)
    })
    mem_before = df.memory_usage(deep=True).sum()
    # 优化
    df['id'] = df['id'].astype('int16')
    df['city'] = df['city'].astype('category')
    df['val'] = df['val'].astype('float32')
    mem_after = df.memory_usage(deep=True).sum()
    assert mem_after < mem_before, "优化后内存应减少"
    print(f"内存: {mem_before} → {mem_after} ({mem_before/mem_after:.1f}x)")

def test_nullable_integer():
    """验证 Int64 支持缺失值"""
    s = pd.Series([1, 2, None, 4], dtype='Int64')
    assert s.isna().sum() == 1
    assert s.dtype == 'Int64'
    assert s.sum() == 7

def test_category_memory():
    """验证 category 显著节省低基数字段内存"""
    s_obj = pd.Series(np.random.choice(['A','B'], 10000))
    s_cat = s_obj.astype('category')
    mem_obj = s_obj.memory_usage(deep=True)
    mem_cat = s_cat.memory_usage(deep=True)
    assert mem_cat < mem_obj * 0.3, f"category 应明显更省内存，实际: {mem_obj}→{mem_cat}"

def test_astype_bounds():
    """验证 int8 溢出会报错"""
    s = pd.Series([0, 127, 128])
    try:
        s.astype('int8')
        assert False, "应有溢出错误"
    except (OverflowError, ValueError):
        pass

if __name__ == '__main__':
    test_memory_reduction()
    test_nullable_integer()
    test_category_memory()
    test_astype_bounds()
    print("✓ 所有测试通过")
```

## 4. 项目总结

### 4.1 优点 & 缺点

| 维度 | pandas dtype 优化 | 不优化直接读 | 数据库存储 |
|------|------------------|-------------|-----------|
| 内存效率 | 可压至原始的 1/5~1/10 | 2-5 倍 CSV 大小 | 由数据库引擎管理 |
| 操作速度 | 小类型运算更快（CPU 缓存友好） | 慢 | 依赖索引 |
| 类型安全 | 显式 dtype 约束 | 依赖自动推断，易出错 | Schema 约束强 |
| 缺失值 | Int64/string 原生支持 NA | float64 无奈之举 | NULL 原生支持 |
| 部署门槛 | 需了解 dtype 体系 | 无需额外知识 | 需数据库环境 |
| 可移植性 | 需 pyarrow/libraries | 开箱即用 | 特定数据库 |

### 4.2 适用场景

- **适用场景**：
  1. 大数据量（百万级+）在有限内存机器上的分析任务
  2. 需要频繁读取的数据集——一次优化，长期受益（可导出 Parquet 固化）
  3. 需要处理缺失值的整数/布尔列场景
  4. 低基数维度列（性别、等级、省份等）的存储和计算加速
  5. 机器学习特征工程——减小特征矩阵的内存占用
- **不适用场景**：
  1. 小型数据集（<10 万行）——优化收益不值花费的时间
  2. 临时探索分析——先用默认 dtype 快速出结果，确定后统一优化

### 4.3 注意事项

- **Int8/Int16 溢出**：赋值前先用 `describe()` 确认数据范围，`int8` 只能存 -128~127；
- **category 的唯一值限制**：唯一值超过 50% 行数时不建议用 category，反而浪费内存；
- **PyArrow 引擎**：需要额外安装 `pyarrow`，但好处明显——读写速度 + 内存效率双提升；
- **to_numeric 的 errors**：`errors='raise'` 遇到无效值报错，`'coerce'` 转成 NaN，`'ignore'` 保持原样。

### 4.4 常见踩坑经验

1. **category 列 merge 导致类型丢失**：两个 category 列做 `merge` 后，结果列会被"退化"为 `object`。如果后续还要 merge，会重复触发这个退化。解决方案：merge 后立即 `astype('category')` 保持类型。
2. **`astype('int8')` 静默溢出**：`pd.Series([128, 0]).astype('int8')` 会报错，但 `pd.Series([200, 0]).astype('int8', errors='ignore')` 静默失败（列还是 int64）。务必在转换前检查值域。
3. **`convert_dtypes()` 的推断不是最优的**：它会把整数列转成 `Int64`（支持 NA）但不会缩小到 `Int8`——因为 pandas 不知道你的业务范围。手动精细化优化始终是内存最优解。
