# 第29章：任务调度、日志与参数化运行

## 1. 项目背景

某零售公司的数据分析师小吴维护着一套"门店经营日报"脚本——每天早上 7 点自动运行，从数据库拉取前一天的各门店销售数据，做清洗聚合，生成一份带图表的 Excel 报告，发送到运营群。

这套脚本的核心逻辑（pandas 数据处理）已经稳定了，但运维方面有三个痛点：

**痛点一：脚本无法识别运行失败**。有一次数据库凌晨维护，脚本连接不上 MySQL，报了 `OperationalError` 直接退出——小吴直到上午 10 点运营来问"今天的日报呢？"才发现脚本根本没跑成功。没有任何告警机制。

**痛点二：重复运行导致数据翻倍**。运维同事在排查失败时又手动执行了一次脚本——但脚本没有"幂等"设计，数据被重复写入了两遍。日报里的销售额直接翻倍，运营震惊。

**痛点三：切换日期靠改代码**。运营偶尔需要补跑历史某天的日报——小吴需要在代码里把 `date = '2025-03-31'` 改成 `date = '2025-03-15'`，改完再改回来。来来回回中间还改错过一次，生成了错误日期的报告。

痛点流程：

```
门店日报脚本
  ├── DB 挂了 → 脚本静默退出 → 无告警 → 3小时后才发现
  ├── 失败后重跑 → 重复写入 → 销售额翻倍
  └── 切换日期 → 手工改代码 → 容易改错
```

本章将讲解如何用命令行参数、结构化日志、幂等设计和 cron 调度，将 pandas 脚本从"能跑"推向"生产可信赖"。

## 2. 项目设计：剧本式交锋对话

> 场景：小吴的日报又出问题了——销售翻了 2 倍。小胖看着报表哈哈大笑。

**小胖**：（咬着一个苹果）"销售额翻倍还不好？你们公司的 KPI 直接超额完成！老板肯定高兴！"

**小吴**："高兴个鬼——这是数据重复写入导致的计算错误。运维同事重跑脚本的时候没有清空上次的结果。"

**大师**："这就是'幂等性'的缺失。**幂等**意味着同一个操作执行一次和执行多次，结果相同。对于日报脚本，幂等的关键是：写入数据前先清空要写入的目标日期范围。

```python
# 幂等写入：先删除再插入
def sync_report(engine, report_date, df):
    with engine.connect() as conn:
        conn.execute(text(f'DELETE FROM daily_report WHERE date = :date'),
                     {'date': report_date})
        conn.commit()
    df.to_sql('daily_report', engine, if_exists='append', index=False)
```

先 DELETE 再 INSERT——无论跑多少次，结果都是一份数据。这就是幂等。"

**【技术映射：幂等 = 用橡皮擦先擦掉再写字——写 10 遍和写 1 遍留下的都是同一行字】**

**小白**："那参数化运行呢？每次改日期好麻烦。"

**大师**："用命令行参数或环境变量，而不是硬编码：

```python
# ❌ 硬编码
REPORT_DATE = '2025-03-31'

# ✅ 命令行参数
import argparse
parser = argparse.ArgumentParser()
parser.add_argument('--date', required=True, help='日报日期 YYYY-MM-DD')
parser.add_argument('--output', default='./output', help='输出目录')
args = parser.parse_args()
REPORT_DATE = args.date
```

然后 `python daily_report.py --date 2025-03-31` ——改日期不需要改代码。这就是**配置外置**原则。"

**【技术映射：命令行参数 = 遥控器——同样的电视（代码），不同的人用不同的频道（参数）】**

**小胖**："日志是干啥的？我以前都靠 print 调试，不行吗？"

**大师**："print 会输出到 stdout，脚本在 cron 里运行时 stdout 没人看。你需要的是 **logging**——结构化、带时间戳、可以输出到文件：

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s',
    handlers=[
        logging.FileHandler('daily_report.log'),
        logging.StreamHandler()  # 同时输出到控制台
    ]
)
logger = logging.getLogger(__name__)

logger.info(f'开始生成日报，日期: {date}')
logger.info(f'读取数据: {len(df):,} 条')
logger.info(f'处理耗时: {elapsed:.2f}s')
logger.info('日报生成完成')
```

日志的输出示例：

```
2025-03-31 07:00:01 [INFO] 开始生成日报，日期: 2025-03-30
2025-03-31 07:00:03 [INFO] 读取数据: 52,310 条
2025-03-31 07:00:05 [INFO] 处理耗时: 2.34s
2025-03-31 07:00:05 [INFO] 日报生成完成
```

出问题时，日志比 print 有价值 10 倍——因为你能看到**什么时间、什么步骤、处理了多少数据**。"

**【技术映射：logging = 黑匣子——飞机飞完看黑匣子才知道全程发生了什么，而不是靠飞行员回忆】**

**小白**："那监控和告警呢？怎么让脚本失败时通知我？"

**大师**："三层防护：

1. **脚本级**：`try/except` 捕获异常 → 记录 ERROR 日志
2. **调度级**：cron/Airflow 检测进程退出码——非 0 就发通知
3. **数据级**：脚本末尾做质量检查（第 27 章）——数据不对也告警

```python
import sys

def main():
    try:
        run_report(args.date)
    except Exception as e:
        logger.error(f'日报失败: {e}', exc_info=True)
        sys.exit(1)  # 非零退出码 → cron 会检测到

if __name__ == '__main__':
    main()
```

配合 cron 的 `MAILTO` 或企业微信 Webhook——脚本失败时自动通知。"

**大师总结**："pandas 脚本生产化的四个关键步骤：
1. 参数化：argparse/配置文件替代硬编码
2. 结构化日志：logging 替代 print
3. 幂等设计：DELETE + INSERT 替代纯 APPEND
4. 异常退出：非零退出码 + 日志告警

记住：**代码写得再漂亮，跑不起来就是零。**"

#### 步骤 1：命令行参数——argparse

**目标**：用 argparse 接收日期和输出目录参数。

```python
# step1_argparse.py
import argparse
import pandas as pd
from datetime import datetime

def parse_args():
    parser = argparse.ArgumentParser(description='门店经营日报生成器')
    parser.add_argument('--date', type=str, required=True,
                        help='日报日期 YYYY-MM-DD')
    parser.add_argument('--output', type=str, default='./output',
                        help='输出目录')
    parser.add_argument('--debug', action='store_true',
                        help='调试模式：只处理少量数据')
    return parser.parse_args()

if __name__ == '__main__':
    args = parse_args()
    # 验证日期格式
    try:
        report_date = datetime.strptime(args.date, '%Y-%m-%d')
    except ValueError:
        print(f'错误：日期格式应为 YYYY-MM-DD，收到 "{args.date}"')
        exit(1)

    print(f'日报日期: {report_date.date()}')
    print(f'输出目录: {args.output}')
    print(f'调试模式: {args.debug}')
```

#### 步骤 2：结构化日志——logging

**目标**：用 logging 替代 print，记录关键步骤。

```python
# step2_logging.py
import logging
import time
import pandas as pd
import numpy as np

logger = logging.getLogger(__name__)
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s',
    handlers=[logging.FileHandler('report.log', encoding='utf-8'), logging.StreamHandler()]
)

def generate_report(date):
    logger.info(f'===== 开始生成日报 {date} =====')

    # 读取数据
    t0 = time.perf_counter()
    df = pd.DataFrame({
        'store': ['A','B','A'], 'sales': [1000, 2000, 1500],
        'orders': [10, 15, 12]
    })
    logger.info(f'读取数据: {len(df)} 条 (模拟)')

    # 计算
    summary = df.groupby('store').agg(
        total_sales=('sales','sum'),
        total_orders=('orders','sum')
    ).reset_index()
    logger.info(f'计算完成: {len(summary)} 家门店')

    # 导出
    output = f'report_{date}.csv'
    summary.to_csv(output, index=False)
    logger.info(f'已导出: {output}')

    elapsed = time.perf_counter() - t0
    logger.info(f'===== 完成 耗时 {elapsed:.2f}s =====')
    return summary

if __name__ == '__main__':
    generate_report('2025-03-31')
```

#### 步骤 3：幂等设计——DELETE + INSERT

**目标**：先删后写，确保重复运行不产生重复数据。

```python
# step3_idempotent.py
import pandas as pd
import numpy as np

# 模拟数据库表
class FakeDB:
    def __init__(self):
        self.data = pd.DataFrame(columns=['date','store','sales'])

    def delete_date(self, date):
        before = len(self.data)
        self.data = self.data[self.data['date'] != date]
        return before - len(self.data)

    def insert(self, df):
        self.data = pd.concat([self.data, df], ignore_index=True)

db = FakeDB()

# 第一次插入
report = pd.DataFrame({'date':['2025-03-31','2025-03-31'], 'store':['A','B'], 'sales':[100,200]})
db.delete_date('2025-03-31')
db.insert(report)
print(f"第1次: {len(db.data)} 行")  # 2 行

# 第二次插入（幂等）
db.delete_date('2025-03-31')
db.insert(report)
print(f"第2次: {len(db.data)} 行")  # 还是 2 行——幂等
print(db.data.to_string(index=False))
```

#### 步骤 4：异常退出 + cron 接入

**目标**：异常时记录 ERROR 并返回非零退出码。

```python
# step4_exception.py
import sys
import logging
import traceback

logger = logging.getLogger(__name__)
logging.basicConfig(level=logging.INFO, format='%(asctime)s [%(levelname)s] %(message)s')

def risky_operation(date):
    # 模拟可能失败的操作
    if date == '2025-02-30':  # 非法日期
        raise ValueError(f'日期不存在: {date}')
    return f'日报 {date} 生成成功'

def main(date):
    try:
        result = risky_operation(date)
        logger.info(result)
        return 0
    except Exception as e:
        logger.error(f'日报生成失败: {e}')
        logger.error(traceback.format_exc())  # 完整堆栈
        return 1  # 非零退出码

if __name__ == '__main__':
    exit_code = main('2025-03-31')  # 正常
    # exit_code = main('2025-02-30')  # 异常 → 返回 1
    sys.exit(exit_code)
```

#### 步骤 5：完整主脚本

```python
# daily_report.py —— 完整可交付的日报脚本
import argparse
import logging
import sys
import time
import pandas as pd
from pathlib import Path

def setup():
    p = argparse.ArgumentParser(description='门店经营日报')
    p.add_argument('--date', required=True, help='YYYY-MM-DD')
    p.add_argument('--output', default='./output')
    p.add_argument('--debug', action='store_true')
    return p.parse_args()

def main():
    args = setup()
    # 日志
    logging.basicConfig(
        level=logging.DEBUG if args.debug else logging.INFO,
        format='%(asctime)s [%(levelname)s] %(message)s',
        handlers=[logging.FileHandler('daily_report.log'), logging.StreamHandler()]
    )
    logger = logging.getLogger(__name__)
    logger.info(f'日报开始: {args.date}')

    try:
        t0 = time.perf_counter()
        # 模拟：读取→处理→导出
        df = pd.DataFrame({'store':['A','B'], 'sales':[1000,2000]})
        logger.info(f'处理 {len(df)} 条数据')
        Path(args.output).mkdir(parents=True, exist_ok=True)
        df.to_csv(f'{args.output}/report_{args.date}.csv', index=False)
        logger.info(f'完成, 耗时 {time.perf_counter()-t0:.2f}s')
    except Exception as e:
        logger.error(f'失败: {e}', exc_info=True)
        sys.exit(1)

if __name__ == '__main__':
    main()
```

### 3.5 常见坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| cron 里 Python 找不到 | cron 的环境变量和交互 shell 不同 | 在脚本中使用 Python 的绝对路径 |
| logging 多进程冲突 | 多个进程写同一个日志文件 | 用 RotatingFileHandler 或独立日志文件 |
| argparse 的 date 没有校验 | 传了非法日期不会报语法错 | 加 `type=lambda s: datetime.strptime(s, '%Y-%m-%d')` |
| cron 的 % 被转义 | cron 中 % 是特殊字符 | 用 `\%` 转义或使用脚本封装 |

## 4. 项目总结

### 4.1 优点 & 缺点

| 维度 | argparse+logging | 硬编码+print | Airflow/Prefect |
|------|-----------------|-------------|----------------|
| 参数管理 | 命令行灵活 | 改代码 | Web UI 配置 |
| 错误感知 | 日志+退出码 | 无 | 内置告警 |
| 幂等性 | DELETE+INSERT | 依赖脚本外 | 内置回填机制 |
| 部署门槛 | 极低 (cron) | 无 | 需要部署 Airflow |
| 适用场景 | 轻量定时任务 | 临时分析 | 复杂 DAG |

### 4.2 适用场景

- **适用场景**：
  1. 每日/每周定时运行的 pandas 分析脚本
  2. 需要灵活切换参数（日期/渠道/区域）的分析任务
  3. 需要审计日志和数据回溯的生产任务
  4. 单机或少量依赖的轻量 ETL
- **不适用场景**：
  1. 复杂的多步骤依赖（A→B→C→D）——用 Airflow/Prefect
  2. 需要分布式调度——用 Airflow K8s Executor

### 4.3 注意事项

- **argparse 的 type 参数可以做验证**：`type=lambda s: datetime.strptime(s, '%Y-%m-%d')` 在参数解析阶段就校验格式；
- **logging 的 level**：生产用 INFO（关键步骤），调试用 DEBUG（详细变量）；
- **幂等性不是 DELETE 一定要**：对于大表 DELETE 很慢，可以在表上建日期分区，用 `TRUNCATE PARTITION`；
- **cron 的 PATH 问题**：cron 不加载 `.bashrc`，所有命令用绝对路径或脚本开头 `source ~/.bashrc`。

### 4.4 常见踩坑经验

1. **cron 跑脚本时报 `ModuleNotFoundError: No module named 'pandas'`**：因为 cron 的环境没有激活虚拟环境。解决方案：在脚本中用虚拟环境的 Python 绝对路径，如 `/home/user/.venv/bin/python daily_report.py`。
2. **log 文件越来越大直到撑满磁盘**：`FileHandler` 默认无限追加。改用 `RotatingFileHandler('app.log', maxBytes=10*1024*1024, backupCount=5)` 自动切割。
3. **重复运行幂等脚本时出现竞态条件**：两个实例同时 DELETE 同一天数据再 INSERT——最后一条 INSERT 可能不是最新的。用文件锁或数据库 Advisory Lock。
