# 第26章：窗口、排名与漏斗分析

## 1. 项目背景

某短视频 App 的增长团队需要做一次"用户转化漏斗分析"，评估从 App 安装到完成首条视频发布的各环节转化率。埋点系统记录了 5 个关键事件的触发日志：`app_install`（安装）、`account_register`（注册）、`first_browse`（首次浏览）、`first_like`（首次点赞）、`first_post`（首次发布）。数据来源是 `user_events.csv`，约 80 万行，包含：user_id、event_type、event_time。

增长分析师小钱需要回答三个核心问题：
- **漏斗转化率**：从安装到各环节的留存比例——安装了 App 的用户中有多少完成了注册？注册了的有多少发布了首条视频？
- **用户路径序列**：每个用户的事件触发顺序——是否按预期路径（安装→注册→浏览→点赞→发布），还是有跳跃行为？
- **同期群分析（Cohort）**：按周分组的新用户，在后续各周的留存率变化趋势。

小钱目前的做法是 SQL 查询 + Excel 图表组合。但遇到了显著困难：

**痛点一：漏斗转化需要多次 GROUP BY 和 JOIN**。先按 event_type 做 COUNT DISTINCT user_id，再手工算各环节的比例——SQL 需要写 5 个 CTE 嵌套。一旦需求变成"分渠道看漏斗"，嵌套层级翻倍。

**痛点二：用户路径排序需要窗口函数**。找出每个用户的第二个事件是什么——在 SQL 里需要 `ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY event_time)`，然后取 `rn=2`——但这个查询在 80 万行上跑一次需要 2 分钟。

**痛点三：同期群（Cohort）分析在 Excel 里完全是噩梦**。"每个周队列在后续 8 周的留存率"是一个 52×8 的矩阵——Excel 需要手动做 52 次透视表。

痛点流程：

```
埋点日志 (80万行)
  ├── 漏斗转化 → SQL 5层 CTE → 修改需求就重构
  ├── 用户路径 → ROW_NUMBER窗口 → 80万行2分钟
  └── 同期群 → 52次透视表 → Excel 噩梦
```

本章将用 pandas 的 rank、cumcount、cumsum、shift 以及分组窗口函数，构建转化漏斗、用户路径和同期群三大增长分析模型。

## 2. 项目设计：剧本式交锋对话

> 场景：小钱在会议上展示了用 5 个 SQL CTE 拼出来的漏斗分析，小胖看得眼睛都直了。

**小胖**：（拿着一包薯片）"漏斗不就是五个数相除吗？100 人安装、80 人注册、50 人浏览......每一步除以上一步——小学生都会啊！"

**大师**："数据层面没错，但当你要分渠道、分版本、分时间段看漏斗时，就不是五个数了。是 5 事件 × 10 渠道 × 4 版本 = 200 个组合。pandas 的 **cumcount + pivot_table** 组合可以在一个管道里完成所有维度的漏斗分析。"

**小白**："cumcount 是什么？和 count 有什么区别？"

**大师**："`cumcount` 是 pandas 的'分组内累计计数'。它在 groupby 后给每条记录分配一个递增的序号：

```python
# 按 user_id 分组，按 event_time 排序，给每个用户的事件编号
df['event_seq'] = df.sort_values('event_time').groupby('user_id').cumcount() + 1
# user_id=1:  event1→1, event2→2, event3→3
# user_id=2:  event1→1, event2→2
```

`cumcount` 和 SQL 的 `ROW_NUMBER()` 完全对应。配合 `rank`（排名）和 `cumsum`（累计和），构成了 pandas 的'分组窗口三剑客'：

| 方法 | 作用 | SQL 对应 |
|------|------|---------|
| `cumcount` | 组内序号 (0,1,2...) | ROW_NUMBER() |
| `rank` | 组内排名 (按值) | RANK() / DENSE_RANK() |
| `cumsum` | 组内累计和 | SUM() OVER (ORDER BY) |
| `cummax/cummin` | 组内累计最大/最小 | MAX/MIN() OVER (ORDER BY) |
| `shift` | 组内位移 | LAG() / LEAD() |"

**【技术映射：cumcount = 排队发号牌——同一窗口（用户）内的每个人按到达顺序领号】**

**小白**："那漏斗分析具体怎么做？"

**大师**："漏斗分析的核心是**每个事件类型的去重用户数**。关键是要处理'用户可能跳过某些环节'的情况：

```python
# 1. 每个用户最早触发每个事件的时间
funnel = df.groupby(['user_id','event_type'])['event_time'].min().unstack()

# 2. 漏斗逻辑：要进入"浏览"环节，必须先完成"注册"
#    即浏览时间 > 注册时间（且注册时间存在）
funnel['first_browse'] = funnel['first_browse'].where(
    funnel['first_browse'] > funnel['account_register'],
    other=pd.NaT
)
```

这样确保漏斗的每个环节遵循先后顺序——用户必须先注册才能浏览。"

**【技术映射：漏斗 = 筛子——每层筛掉一部分人，剩下的人进入下一层，且必须按顺序通过】**

**小胖**："那同期群（Cohort）分析呢？就是'1 月注册的用户在 2 月还有多少活跃'？"

**大师**："对。同期群 = 按首次激活时间分组 + 跟踪后续行为。核心步骤：

1. 找出每个用户的'首周'（注册周）
2. 计算每个用户每周的活跃天数
3. 按 (首周, 第N周) 做透视表

```python
# 1. 每个用户的首周
df['cohort_week'] = df.groupby('user_id')['event_time'].transform('min').dt.isocalendar().week

# 2. 每条事件的第几周（相对于首周）
df['week_number'] = (df['event_time'].dt.isocalendar().week - df['cohort_week'])

# 3. 同期群矩阵
cohort = df.pivot_table(index='cohort_week', columns='week_number',
                         values='user_id', aggfunc='nunique')
```

一行 pivot_table 搞定整个同期群矩阵。"

**大师总结**："增长分析的三大模型：
1. **漏斗**：cumcount + groupby + pivot 按顺序筛选
2. **用户路径**：cumcount + shift 计算事件间隔和顺序
3. **同期群**：transform min 找首周 + pivot_table 做矩阵

核心工具就四个：cumcount、rank、cumsum、shift——掌握了它们，80% 的增长分析需求三行代码搞定。"

```python
# generate_event_data.py
import pandas as pd
import numpy as np

np.random.seed(42)
N_USERS = 5000
events = ['app_install','account_register','first_browse','first_like','first_post']
channels = ['抖音','微信','应用商店','地推']

# 模拟漏斗转化率（从安装到各环节）
funnel_rates = [1.0, 0.75, 0.55, 0.35, 0.15]

data = []
for uid in range(1, N_USERS+1):
    channel = np.random.choice(channels)
    install_time = pd.Timestamp('2025-01-01') + pd.Timedelta(hours=np.random.randint(0, 8760))
    data.append({'user_id': uid, 'event_type': 'app_install', 'event_time': install_time, 'channel': channel})

    current_time = install_time
    for i, event in enumerate(events[1:], 1):
        if np.random.random() < funnel_rates[i] / funnel_rates[i-1]:
            current_time = current_time + pd.Timedelta(minutes=np.random.randint(1, 1440))
            data.append({'user_id': uid, 'event_type': event, 'event_time': current_time, 'channel': channel})

df = pd.DataFrame(data)
df['event_time'] = df['event_time'].dt.strftime('%Y-%m-%d %H:%M:%S')
df.to_csv('user_events.csv', index=False)
print(f"已生成 {len(df)} 条事件, {df['user_id'].nunique()} 用户")
```

### 3.3 分步实现

#### 步骤 1：漏斗转化分析

**目标**：计算从安装到发布的各环节转化率。

```python
# step1_funnel.py
import pandas as pd

df = pd.read_csv('user_events.csv', parse_dates=['event_time'])

# 1. 每个用户每个事件的最早触发时间
funnel = df.pivot_table(index='user_id', columns='event_type',
                        values='event_time', aggfunc='min')

# 2. 按顺序约束：后面的环节必须晚于前面的环节
event_order = ['app_install','account_register','first_browse','first_like','first_post']
for i in range(1, len(event_order)):
    prev, curr = event_order[i-1], event_order[i]
    if prev in funnel.columns and curr in funnel.columns:
        funnel[curr] = funnel[curr].where(funnel[curr] > funnel[prev], other=pd.NaT)

# 3. 各环节人数和转化率
print("===== 用户转化漏斗 =====")
funnel_counts = {}
for event in event_order:
    if event in funnel.columns:
        count = funnel[event].notna().sum()
        funnel_counts[event] = count

prev_count = funnel_counts[event_order[0]]
print(f"{'环节':<18} {'人数':>8} {'总体转化率':>10} {'步骤转化率':>10}")
for event in event_order:
    if event in funnel_counts:
        count = funnel_counts[event]
        overall_rate = count / funnel_counts[event_order[0]] * 100
        step_rate = count / prev_count * 100
        print(f"{event:<18} {count:>8} {overall_rate:>9.1f}% {step_rate:>9.1f}%")
        prev_count = count

# 4. 分渠道漏斗
print("\n===== 分渠道漏斗 (注册转化率) =====")
channel_funnel = df[df['event_type'].isin(['app_install','account_register'])].pivot_table(
    index='channel', columns='event_type', values='user_id', aggfunc='nunique')
channel_funnel['注册率%'] = (channel_funnel['account_register'] / channel_funnel['app_install'] * 100).round(1)
print(channel_funnel[['app_install','account_register','注册率%']].to_string())
```

#### 步骤 2：用户路径排序——cumcount 和 shift

**目标**：分析用户的事件触发顺序和间隔。

```python
# step2_user_path.py
import pandas as pd

df = pd.read_csv('user_events.csv', parse_dates=['event_time'])
df = df.sort_values(['user_id','event_time'])

# === cumcount：给每个用户的事件编号 ===
df['event_seq'] = df.groupby('user_id').cumcount() + 1

# === shift：计算相邻事件的时间间隔 ===
df['prev_event_time'] = df.groupby('user_id')['event_time'].shift(1)
df['time_gap_minutes'] = (df['event_time'] - df['prev_event_time']).dt.total_seconds() / 60
df['time_gap_minutes'] = df['time_gap_minutes'].round(1)

# === 每个用户第 2 个事件是什么？（安装后下一步） ===
second_event = df[df['event_seq'] == 2]
print("===== 安装后的第二步行为分布 =====")
print(second_event['event_type'].value_counts().to_string())

print(f"\n===== 各步骤平均时间间隔 (分钟) =====")
print(df.groupby('event_seq')['time_gap_minutes'].mean().round(1).to_string())

# === 异常路径检测：跳过注册直接浏览 ===
df_pivot = df.pivot_table(index='user_id', columns='event_seq', values='event_type', aggfunc='first')
# 找出 seq=2 不是 'account_register' 的用户
skip_users = df_pivot[df_pivot[2] != 'account_register']
print(f"\n跳过注册的用户: {len(skip_users)} 人")
```

#### 步骤 3：rank 排名分析——各渠道用户活跃度排名

**目标**：用 rank 给各渠道按用户活跃事件数排名。

```python
# step3_rank.py
import pandas as pd

df = pd.read_csv('user_events.csv', parse_dates=['event_time'])

# 每个用户的事件总数
user_events_count = df.groupby('user_id').size().reset_index(name='event_count')

# 关联渠道信息
user_channel = df.groupby('user_id')['channel'].first().reset_index()
user_stats = user_events_count.merge(user_channel, on='user_id')

# === rank：在各渠道内按事件数排名 ===
user_stats['rank_in_channel'] = user_stats.groupby('channel')['event_count'].rank(ascending=False, method='dense').astype(int)
user_stats['pct_in_channel'] = user_stats.groupby('channel')['event_count'].rank(pct=True).round(2)

# === 各渠道 Top 10 活跃用户 ===
print("===== 各渠道 Top 3 活跃用户 =====")
for ch in user_stats['channel'].unique():
    top = user_stats[user_stats['channel']==ch].nsmallest(3, 'rank_in_channel')
    print(f"\n{ch}:")
    print(top[['user_id','event_count','rank_in_channel']].to_string(index=False))

# === 渠道活跃度整体对比 ===
print(f"\n===== 渠道活跃度中位数 =====")
print(user_stats.groupby('channel')['event_count'].median().round(0).sort_values(ascending=False).to_string())
```

#### 步骤 4：同期群（Cohort）分析

**目标**：按周分组新用户，追踪留存率。

```python
# step4_cohort.py
import pandas as pd
import numpy as np

df = pd.read_csv('user_events.csv', parse_dates=['event_time'])

# 1. 每个用户的首周（注册周 = 安装时间所在的 ISO 周）
df['cohort_week'] = df.groupby('user_id')['event_time'].transform('min').dt.isocalendar().week.astype(int)

# 2. 每条事件的相对周数
df['event_week'] = df['event_time'].dt.isocalendar().week.astype(int)
df['week_number'] = df['event_week'] - df['cohort_week']

# 3. 同期群矩阵：每个首周队列在各 relative_week 的活跃用户数
cohort_matrix = df.pivot_table(
    index='cohort_week', columns='week_number',
    values='user_id', aggfunc='nunique', fill_value=0
)

# 4. 留存率：以 week 0 为基准
cohort_retention = cohort_matrix.div(cohort_matrix[0], axis=0).round(3)

# 只看前 8 周
print("===== 同期群留存率矩阵 (前5个队列 × 前8周) =====")
print(cohort_retention.iloc[:5, :8].to_string())

print(f"\n===== 平均周留存率 =====")
avg_retention = cohort_retention.mean()
for wk in range(8):
    if wk in avg_retention.index:
        print(f"  Week {wk}: {avg_retention[wk]*100:.1f}%")
```

### 3.4 常见坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| cumcount 编号从 0 开始 | pandas 设计 | 加 1：`cumcount() + 1` |
| shift 在分组边界处取到了上一组的值 | 跨组 shift | shift 必须在 groupby 内执行 |
| rank 的 method 选错导致排名不连续 | `method='min'` 跳过序号 | 用 `method='dense'` 得到连续排名 |
| pivot_table 生成超大矩阵 | cohort_week 太多 | 限制到 Top 10 队列或按月而非按周 |

```python
def test_cumcount():
    df = pd.DataFrame({'g':['a','a','b','b','b'], 'v':[1,2,3,4,5]})
```

## 4. 项目总结

### 4.1 优点 & 缺点

| 维度 | pandas 增长分析 | SQL 窗口函数 | Excel 手工 |
|------|---------------|-------------|-----------|
| 漏斗计算 | pivot_table + where 灵活 | 多层 CTE | 手工除法 |
| 用户路径 | cumcount + shift 一行搞定 | ROW_NUMBER + LAG | 手工排序 |
| 同期群 | pivot_table 秒出矩阵 | 需要复杂 PIVOT | 52 次透视表 |
| 多维度下钻 | groupby 自然支持 | 需动态 SQL | 多次手工筛选 |
| 可复用性 | 函数封装 | 存储过程 | 无 |

### 4.2 适用场景

- **适用场景**：
  1. App/网站的用户转化漏斗分析
  2. 用户行为路径的顺序分析和异常路径检测
  3. 同期群留存率追踪（周/月粒度）
  4. 分组内排名和累计指标（如用户累计消费）
  5. 事件间隔分析（如从注册到首购的平均天数）
- **不适用场景**：
  1. 实时漏斗（需要实时流计算框架）
  2. 十亿级事件日志——考虑用 Spark/Dask

### 4.3 注意事项

- **cumcount 需要预排序**：`sort_values(['user_id','event_time'])` 是必须的，否则序号无意义；
- **漏斗的 where 约束**：确保后面的事件时间晚于前面——否则漏斗会出现"先发布后注册"的荒谬结果；
- **同期群的基准**：通常以 `cohort_week` 的首个值 week=0 为基准 100%，需确认 week=0 有足够用户。

### 4.4 常见踩坑经验

1. **cumcount 在分组外使用返回全局序号**：`df['seq'] = df.cumcount()` 没加 groupby，返回的是全局行号。必须 `df.groupby('user_id').cumcount()`。
2. **漏斗转化率超过 100%**：因为漏斗约束（先注册后浏览）没生效，导致浏览用户数超过了注册用户数。检查 `where` 条件是否正确。
3. **同期群矩阵行数爆炸**：按天分组导致 365 个 cohort——矩阵 365×365 不可读。先按月或按周聚合，控制 cohort 数量在 20 以内。
