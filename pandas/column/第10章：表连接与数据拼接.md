# 第10章：表连接与数据拼接

## 1. 项目背景

某电商平台的运营团队需要搭建一张"订单全景宽表"，将四张独立表整合为一：

- **订单主表**（orders.csv）：订单号、用户 ID、下单时间、订单金额、订单状态
- **支付表**（payments.csv）：订单号、支付方式、支付金额、支付时间
- **退款表**（refunds.csv）：订单号、退款金额、退款原因、退款时间
- **用户画像表**（user_profiles.csv）：用户 ID、城市、年龄段、注册渠道、VIP 等级

运营分析师小马需要基于这张宽表回答以下问题：
- "哪个城市的支付后退款率最高？"
- "哪种支付方式的用户平均客单价最高？"
- "VIP 用户的退款原因分布和普通用户有什么不同？"

目前小马的做法是：在 Excel 里用 VLOOKUP 把四张表手工关联——先 VLOOKUP 支付信息到订单表（30 万条），再 VLOOKUP 退款信息（一次只能返回第一个匹配值但一个订单可能有多次退款），最后 VLOOKUP 用户画像。全程下来：

**痛点一：VLOOKUP 只能返回第一个匹配值**。一个订单如果有两次退款（部分退款 + 全额退款），VLOOKUP 只会返回第一次，第二次退款数据就丢失了。小马只能先在退款表里手工合并退款记录再 VLOOKUP。

**痛点二：连接结果无声膨胀**。由于用户画像表中部分 `user_id` 有重复记录（历史导入时未去重），VLOOKUP 返回了错误匹配，导致订单数凭空多了 2000 行。小马直到总监质疑"为什么订单数比昨天多了？"才发现。

**痛点三：无法验证关联质量**。小马只能在心里默念"关联对了"，但没有任何机制确认多少订单没关联到支付、多少用户没关联到画像。每次数据异常排查都是一场噩梦。

痛点流程：

```
订单表 --+
支付表 --+-- VLOOKUP x4次 --> 宽表
退款表 --+     |
画像表 --+   只能匹配第一个 + 重复画像导致膨胀 + 无法验证关联质量
```

本章将系统讲解 pandas 的表连接机制（merge/concat/join），以及如何用 indicator 和 validate 参数保障连接质量。

## 2. 项目设计：剧本式交锋对话

> 场景：小马在周报里被质疑"订单数怎么跟昨天不一样"，小胖好奇地问是怎么回事。

**小胖**：（拆开一包坚果）"关联两个表，不就是在一张表里找另一张表的对应行嘛？SQL 里 LEFT JOIN 不就一行代码？pandas 也有吧？"

**大师**："pandas 确实有——核心是 `merge` 函数。它和 SQL JOIN 几乎一一对应。参数含义：
- `on`：连接键（可传单个列名或列表）
- `how`：连接方式：'left'、'right'、'inner'、'outer'、'cross'
- `left_on` / `right_on`：当左右表连接键列名不同时使用
- `suffixes`：重名列的后缀（默认 '_x', '_y'）"

**【技术映射：merge = 相亲配对——on 是匹配条件，how 决定匹配不到的人怎么处理（都保留/只保留配对的）】**

**小白**："那 `merge`、`concat`、`join` 三个有什么区别？我以前经常用错。"

**大师**："这是最经典的 pandas 三兄弟区分：

- **merge**：按列值横向关联（像 SQL JOIN）。适合两表按业务键左右拼列。
- **concat**：纵向/横向堆叠。axis=0 纵向堆行（Union All），axis=1 横向拼列（按 Index 对齐）。适合把多个结构相同的表上下拼起来，如各门店月度数据。
- **join**：按 Index 关联（merge 的特化版）。先 set_index 再 join，语法更简洁。

简单记忆：表合并列 → `merge`，表堆叠行 → `concat(axis=0)`，按 Index 关联 → `join`"

**【技术映射：merge = 拼图（左右拼），concat = 叠纸（上下叠），join = 按编号配对】**

**小胖**："那一个订单如果有两条退款记录，merge 之后会怎么样？"

**大师**："这是关键！如果右边表（退款表）中同一个 order_id 出现多次，merge 会产生笛卡尔积：订单表该 order 有 1 行，退款表有 2 行，merge 结果该 order 膨胀为 2 行。这就是订单数'凭空多了 2000 行'的根因。**连接前检查连接键的唯一性是必须步骤**。"

**小白**："怎么检查？"

**大师**："两个利器：

1. **indicator 参数**——在结果中加一列标记每行来源。`result['_merge'].value_counts()` 能让你看到 'both'（两表都有）、'left_only'（仅左表）、'right_only'（仅右表）三种情况。
2. **validate 参数**——在连接时做断言。可选 '1:1'（左右键都唯一）、'1:m'（左唯一右可多）、'm:1'（左可多右唯一）、'm:m'（两边都可多）。不符合预期直接抛 MergeError，比连接后才发现行数不对要好得多。"

**【技术映射：indicator = 相亲后的记录卡，标注谁跟谁配上了；validate = 相亲前的户口核验，防止有人重婚】**

**小白**："那 DataFrame.join 和 merge 有什么区别？什么时候用哪个？"

**大师**："`join` 本质上是 `merge` 的语法糖，按 Index 关联时更简洁。核心原则：如果按 Index 做关联 → 用 `join`，如果按普通列做关联 → 用 `merge`。`join` 也支持 `on` 参数指定左表的列作为连接键。"

**大师总结**："今天覆盖了 pandas 表连接的完整工具箱：
1. `merge` 四个核心参数：`on`、`how`、`indicator`、`validate`
2. `merge` vs `concat` vs `join` 的场景区分
3. 连接前检查唯一性 → 连接中用 validate 做断言 → 连接后用 indicator 验证结果

最后一条铁律：**永远不要相信连接前的原始数据——用户 ID 可能有重复、订单号可能跨系统不统一、支付记录可能缺少某些订单。**"

### 3.2 模拟四张数据表

```python
# generate_order_tables.py
import pandas as pd
import numpy as np
from datetime import datetime, timedelta

np.random.seed(42)
N_ORDERS = 30000

# 1. 订单主表
orders = pd.DataFrame({
    'order_id': [f'ORD{100000+i:06d}' for i in range(N_ORDERS)],
    'user_id': np.random.randint(10000, 30000, N_ORDERS),
    'order_time': [(datetime(2025,3,15) + timedelta(minutes=np.random.randint(0,10080))).strftime('%Y-%m-%d %H:%M:%S') for _ in range(N_ORDERS)],
    'amount': np.round(np.abs(np.random.normal(200, 80, N_ORDERS)), 2),
    'status': np.random.choice(['已完成','待支付','已取消'], N_ORDERS, p=[0.7, 0.2, 0.1])
})

# 2. 支付表（仅已完成订单有支付记录，造5条重复）
paid_orders = orders[orders['status']=='已完成']['order_id'].sample(frac=0.95).tolist()
payments = pd.DataFrame({
    'order_id': paid_orders,
    'pay_method': np.random.choice(['支付宝','微信','银行卡'], len(paid_orders), p=[0.5,0.4,0.1]),
    'pay_amount': np.round(np.abs(np.random.normal(200, 80, len(paid_orders))), 2),
    'pay_time': [(datetime(2025,3,15) + timedelta(minutes=np.random.randint(5,10080))).strftime('%Y-%m-%d %H:%M:%S') for _ in range(len(paid_orders))]
})
duplicates = payments.sample(5).copy()
payments = pd.concat([payments, duplicates], ignore_index=True)

# 3. 退款表（部分订单有退款，含重复order_id）
refund_orders = np.random.choice(paid_orders, size=int(len(paid_orders)*0.08), replace=True)
refunds = pd.DataFrame({
    'order_id': refund_orders,
    'refund_amount': np.round(np.abs(np.random.normal(50, 30, len(refund_orders))), 2),
    'refund_reason': np.random.choice(['商品问题','物流问题','不想要了','其他'], len(refund_orders)),
    'refund_time': [(datetime(2025,3,16) + timedelta(hours=np.random.randint(1,72))).strftime('%Y-%m-%d %H:%M:%S') for _ in range(len(refund_orders))]
})

# 4. 用户画像表（造20条重复user_id模拟脏数据）
unique_users = orders['user_id'].unique()
profiles = pd.DataFrame({
    'user_id': unique_users,
    'city': np.random.choice(['北京','上海','广州','深圳','杭州'], len(unique_users)),
    'age_group': np.random.choice(['18-25','26-35','36-45','46+'], len(unique_users), p=[0.3,0.4,0.2,0.1]),
    'reg_channel': np.random.choice(['微信','抖音','搜索','地推'], len(unique_users)),
    'vip_level': np.random.randint(0, 4, len(unique_users))
})
dup_users = profiles.sample(20).copy()
profiles = pd.concat([profiles, dup_users], ignore_index=True)

orders.to_csv('orders.csv', index=False, encoding='utf-8-sig')
payments.to_csv('payments.csv', index=False, encoding='utf-8-sig')
refunds.to_csv('refunds.csv', index=False, encoding='utf-8-sig')
profiles.to_csv('user_profiles.csv', index=False, encoding='utf-8-sig')
print(f"orders:{len(orders)} payments:{len(payments)} refunds:{len(refunds)} profiles:{len(profiles)}")
print(f"profiles重复user_id:{profiles['user_id'].duplicated().sum()}条")
```

### 3.3 分步实现

#### 步骤 1：连接前数据质量检查

**目标**：在连接前检查每个表的连接键唯一性。

```python
# step1_precheck.py
import pandas as pd

orders = pd.read_csv('orders.csv')
payments = pd.read_csv('payments.csv')
profiles = pd.read_csv('user_profiles.csv')

print("===== 连接键唯一性检查 =====")
for name, df in [('orders', orders), ('payments', payments), ('profiles', profiles)]:
    key = 'order_id' if 'order_id' in df.columns else 'user_id'
    dups = df[key].duplicated().sum()
    if dups > 0:
        print(f"  !! {name}.{key}: {dups} 条重复!")
        dup_vals = df[df[key].duplicated(keep=False)][key].value_counts().head(5)
        print(f"     重复的 {key}: {dup_vals.index.tolist()}")
    else:
        print(f"  OK {name}.{key}: 唯一")

# 处理重复：profiles 按 user_id 去重（保留第一条）
profiles_clean = profiles.drop_duplicates(subset='user_id', keep='first')
print(f"\nprofiles 去重后: {len(profiles_clean)} 条 (原 {len(profiles)} 条)")
profiles_clean.to_csv('user_profiles_clean.csv', index=False)
```

#### 步骤 2：merge 基本连接实验

**目标**：用四种连接方式做实验，理解每种方式的结果差异。

```python
# step2_merge_experiment.py
import pandas as pd

orders = pd.read_csv('orders.csv')
payments = pd.read_csv('payments.csv')

# LEFT JOIN：保留所有订单
left_result = orders.merge(payments, on='order_id', how='left', indicator=True)
print(f"\nLEFT JOIN: {len(left_result)} 行")
print(left_result['_merge'].value_counts().to_string())

# INNER JOIN：只保留有支付的订单
inner_result = orders.merge(payments, on='order_id', how='inner')
print(f"\nINNER JOIN: {len(inner_result)} 行")

# OUTER JOIN：保留所有订单和所有支付
outer_result = orders.merge(payments, on='order_id', how='outer', indicator=True)
print(f"\nOUTER JOIN: {len(outer_result)} 行")
print(outer_result['_merge'].value_counts().to_string())

# 检查重复 key 导致的膨胀
pay_dup_check = payments['order_id'].value_counts()
multi_pay = pay_dup_check[pay_dup_check > 1]
if len(multi_pay) > 0:
    print(f"\n!! 有 {len(multi_pay)} 个订单有多条支付记录")
    print(f"   膨胀: {len(orders)} -> {len(left_result)} (+{len(left_result)-len(orders)})")
```

#### 步骤 3：多表连接构建完整宽表

**目标**：四表联合，构建订单全景宽表，并验证每一步的连接质量。

```python
# step3_wide_table.py
import pandas as pd

orders = pd.read_csv('orders.csv')
payments = pd.read_csv('payments.csv').drop_duplicates(subset='order_id', keep='first')
refunds = pd.read_csv('refunds.csv')
profiles = pd.read_csv('user_profiles_clean.csv')

# 第1层：订单 + 支付 (LEFT JOIN)
print("第1层: 订单 + 支付")
wide = orders.merge(
    payments[['order_id','pay_method','pay_amount']],
    on='order_id', how='left', validate='1:m'
)
print(f"  结果: {len(wide)} 行 (订单:{len(orders)})")

# 第2层：+ 退款 (LEFT JOIN)
print("第2层: + 退款")
wide = wide.merge(
    refunds[['order_id','refund_amount','refund_reason']],
    on='order_id', how='left', indicator='退款标记'
)
print(f"  结果: {len(wide)} 行 (有退款:{(wide['退款标记']=='both').sum()})")

# 第3层：+ 用户画像 (LEFT JOIN)
print("第3层: + 用户画像")
wide = wide.merge(
    profiles,
    on='user_id', how='left', validate='m:1',
    indicator='画像标记'
)
print(f"  有画像:{(wide['画像标记']=='both').sum()}, 无画像:{(wide['画像标记']=='left_only').sum()}")

# 最终摘要
print(f"\n===== 宽表摘要 =====")
print(f"总行数:{len(wide):,}  订单数:{wide['order_id'].nunique():,}")
print(f"有支付:{(wide['pay_method'].notna()).sum():,}")
print(f"有退款:{(wide['refund_amount'].notna()).sum():,}")
print(f"有画像:{(wide['city'].notna()).sum():,}")

# 保存
output_cols = ['order_id','user_id','order_time','amount','status',
               'pay_method','pay_amount','refund_amount','refund_reason',
               'city','age_group','reg_channel','vip_level']
wide[output_cols].to_csv('order_wide_table.csv', index=False, encoding='utf-8-sig')
print("\nOK 订单全景宽表已导出")
```

# step4_concat.py
import pandas as pd
import numpy as np

# 模拟两个月的订单数据（结构相同）
np.random.seed(0)
m1 = pd.DataFrame({
    'order_id': [f'ORD{i:06d}' for i in range(100)],
    'month': '2025-03',
    'amount': np.random.uniform(50, 500, 100).round(2)
})
m2 = pd.DataFrame({
    'order_id': [f'ORD{i:06d}' for i in range(100, 200)],
    'month': '2025-04',
    'amount': np.random.uniform(50, 500, 100).round(2)
})

# 纵向拼接（axis=0，增加行）
combined = pd.concat([m1, m2], axis=0, ignore_index=True)
print(f"concat 纵向: {len(m1)}+{len(m2)} = {len(combined)} 行")

# 横向拼接（axis=1，按 Index 对齐）
left_df = pd.DataFrame({'A': [1,2,3]}, index=['x','y','z'])
right_df = pd.DataFrame({'B': [4,5,6]}, index=['x','y','z'])
side_by_side = pd.concat([left_df, right_df], axis=1)
print(f"\nconcat 横向:\n{side_by_side}")

# join='inner' 只保留共有 Index
left_df2 = pd.DataFrame({'A': [1,2,3]}, index=['x','y','z'])
right_df2 = pd.DataFrame({'B': [4,5,6]}, index=['x','y','w'])
inner_concat = pd.concat([left_df2, right_df2], axis=1, join='inner')
print(f"\nconcat inner join:\n{inner_concat}")
```

**目标**：基于宽表回答业务问题。

# step5_post_analysis.py
import pandas as pd

wide = pd.read_csv('order_wide_table.csv')

# Q1: 哪个城市的支付后退款率最高？
city_refund = wide.groupby('city').agg(
    有支付订单=('pay_method', 'count'),
    有退款订单=('refund_amount', 'count'),
).fillna(0)
city_refund['退款率'] = (city_refund['有退款订单'] / city_refund['有支付订单'] * 100).round(2)
print("===== 各城市退款率 =====")
print(city_refund.sort_values('退款率', ascending=False).to_string())

# Q2: 哪种支付方式的平均客单价最高？
pay_kpi = wide.groupby('pay_method').agg(
    订单数=('order_id', 'nunique'),
    平均客单价=('amount', 'mean'),
    总金额=('amount', 'sum'),
).round(2)
print(f"\n===== 支付方式 KPI =====")
print(pay_kpi.to_string())

# Q3: VIP 用户退款原因分布
vip_refund = wide[(wide['vip_level'] >= 2) & wide['refund_reason'].notna()]
print(f"\n===== VIP用户退款原因分布 =====")
print(vip_refund['refund_reason'].value_counts().to_string())
```

### 3.4 常见坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| merge 后行数暴增 | 连接键在右表中有重复 | 连接前检查 `df[key].duplicated().sum()` |
| merge 后列名冲突 | 两表有同名非连接列 | 指定 `suffixes=('_ord', '_pay')` |
| `validate='1:1'` 报 MergeError | 连接键不唯一 | 先去重：`df.drop_duplicates(subset=key)` |
| concat 后 Index 重复 | `ignore_index` 默认为 False | `pd.concat(..., ignore_index=True)` |
| join 用 `on` 参数报错 | `on` 是指定左表的列 | 右表必须先 `set_index` |
| merge 大量 NaN | how='left' 但右表缺数据 | 检查 indicator 列的分布 |

### 3.5 测试验证

```python
# test_ch10.py
import pandas as pd

def test_merge_left():
    left = pd.DataFrame({'id': [1,2,3], 'a': ['a1','a2','a3']})
    right = pd.DataFrame({'id': [1,2], 'b': ['b1','b2']})
    result = left.merge(right, on='id', how='left')
    assert len(result) == 3
    assert result.loc[2, 'b'] == 'b2'
    assert pd.isna(result.loc[result['id']==3, 'b'].iloc[0])

def test_merge_validate():
    left = pd.DataFrame({'id': [1,2,2], 'a': [10,20,30]})
    right = pd.DataFrame({'id': [1,2,3], 'b': [1,2,3]})
    try:
        left.merge(right, on='id', validate='1:1')
        assert False, "应抛出 MergeError"
    except pd.errors.MergeError:
        pass  # 预期行为

def test_concat_axis():
    a = pd.DataFrame({'x': [1,2]})
    b = pd.DataFrame({'x': [3,4]})
    result = pd.concat([a, b], axis=0, ignore_index=True)
    assert len(result) == 4
    assert result['x'].tolist() == [1,2,3,4]

def test_indicator():
    left = pd.DataFrame({'id': [1,2], 'a': [10,20]})
    right = pd.DataFrame({'id': [1,3], 'b': [30,40]})
    result = left.merge(right, on='id', how='outer', indicator=True)
    vc = result['_merge'].value_counts()
    assert vc['left_only'] == 1
    assert vc['right_only'] == 1
    assert vc['both'] == 1

def test_drop_duplicates_before_merge():
    """验证先去重可防止 merge 膨胀"""
    left = pd.DataFrame({'id': [1,2,3], 'x': [10,20,30]})
    right = pd.DataFrame({'id': [1,1,2], 'y': [1,2,3]})  # id=1 重复
    result_bad = left.merge(right, on='id', how='left')
    assert len(result_bad) == 4  # 膨胀了
    right_clean = right.drop_duplicates(subset='id', keep='first')
    result_good = left.merge(right_clean, on='id', how='left')
    assert len(result_good) == 3  # 正确

if __name__ == '__main__':
    test_merge_left()
    test_merge_validate()
    test_concat_axis()
    test_indicator()
    test_drop_duplicates_before_merge()
    print("OK 所有测试通过")
```

## 4. 项目总结

### 4.1 优点 & 缺点

| 维度 | pandas merge/concat | Excel VLOOKUP | SQL JOIN |
|------|-------------------|---------------|----------|
| 多字段关联 | 传入列表 `on=['a','b']` 即可 | 需辅助列拼接 | ON a AND b |
| 连接质量验证 | indicator + validate 内置 | 无 | 用 COUNT 手工验证 |
| 一对多处理 | 自动笛卡尔积（可预期） | 只返回第一个匹配 | 自动笛卡尔积 |
| 多种连接方式 | left/right/inner/outer/cross | 仅 left（近似） | 完整 JOIN 语法 |
| 纵向拼接 | concat(axis=0) 一行搞定 | 手工复制粘贴 | UNION ALL |
| 可复现性 | 脚本化 | 手工（不可追踪） | SQL 脚本化 |

### 4.2 适用场景

- **适用场景**：
  1. 多源数据整合为宽表——订单+支付+退款+用户画像等
  2. 多期数据纵向合并——各月/各门店的结构化数据堆叠
  3. 需要验证关联质量的场景——indicator 逐行标记来源
  4. 按业务规则做多表 join 的 ETL 管道
  5. DataFrame 按 Index 的快速关联（join 比 merge 更简洁）
- **不适用场景**：
  1. 需要保留连接前原始表结构的场景——merge 会改变列数和行数
  2. 超大数据量（TB 级）的多表 join——建议在数据库/Spark 中完成

### 4.3 注意事项

- **validate 参数生产环境必加**：它能提前暴露数据质量问题，比你分析完才发现行数不对更有效；
- **suffixes 默认值是 ('_x','_y')**：几乎无法理解——建议始终显式指定有业务含义的后缀；
- **merge 的 copy 参数**：pandas 2.0+ copy-on-write 下 merge 默认不复制数据，安全但可能导致与原表共享内存；
- **concat 的 axis 默认是 0**：纵向拼接是默认行为，横向拼接需显式传 `axis=1`。

### 4.4 常见踩坑经验

1. **merge 后 silently 膨胀导致下游分析全错**：一位同事做"订单 + 支付"merge 后行数从 10 万变成 12 万——因为支付表中部分订单号重复。他在 GroupBy 统计销售额时，这些重复支付的订单被多次计算，虚增了 20% 的 GMV。教训：每次 merge 后打印 `len(before)` vs `len(after)` 并解释差异原因。
2. **concat 时 ignore_index 忘加导致 loc 混乱**：两个 DataFrame 纵向 concat 后，Index 分别是 0-99 和 0-99——concat 后 Index 有重复值。后续用 `loc[0]` 会返回两行。教训：除非 Index 有业务含义，否则 `concat` 始终加 `ignore_index=True`。
3. **对 category 类型列做 merge 导致 dtype 退化**：两个 DataFrame 的 category 列做 merge 后，结果列退化为 object——因为 merge 内部重建了数据类型。如果后续还要做大量此列的 groupby 操作，性能会暴跌。解决：merge 后立即 `astype('category')` 恢复类型。
