# 第37章：IO 子系统源码与解析器扩展

## 1. 项目背景

某物联网公司的数据团队每天需要处理来自 5000 个设备的传感器日志——格式是自定义的二进制格式（`.slog`），每个文件约 200MB。日志格式是公司 5 年前设计的——不是 CSV、不是 JSON、不是 Parquet——而是一种用 `|` 分隔的定长字段格式，加上自定义的时间戳编码。

数据工程师小齐目前的做法是：用 Python 的 `struct` 模块逐行解析 → 构建字典 → pandas DataFrame。处理一个 200MB 的文件需要 90 秒。但需求是 30 秒以内——因为每天有 24 个文件滚动到达，需要在 10 分钟内全部处理完。

**痛点一：逐行 Python 解析太慢**。`struct.unpack` 逐行调用 200 万次，Python 函数调用开销累积巨大。如果日志格式是 CSV，pandas 的 C 引擎 2 秒就能读完。

**痛点二：不知道如何让 pandas 原生支持自定义格式**。小齐知道 pandas 的 `read_csv` 底层有 C 引擎——超快。但他的日志不是 CSV——他能复用 C 引擎吗？还是需要自己写一个解析器？

**痛点三：不同引擎的 dtype 推断行为不一致**。pandas 有至少三个 CSV 解析引擎（C 引擎、Python 引擎、PyArrow 引擎），它们在 dtype 推断、NA 值识别、日期解析上的行为各不相同。

痛点流程：

```
自定义 .slog 文件 (200MB)
  └── struct.unpack 逐行 → 90s
        ├── 逐行 Python 调用 → 慢
        ├── 能用 pandas C 引擎吗？
        └── 不同引擎行为不一致
```

本章将深入 pandas IO 子系统源码——read_csv 的统一入口、C/Python/PyArrow 三个引擎的差异、以及如何实现一个自定义文件格式的 pandas 读取器。

## 2. 项目设计：剧本式交锋对话

> 场景：小齐在技术分享会上展示了.log 解析的性能问题，小胖第一反应是"换 Parquet 啊"。

**小胖**：（吃着饼干）"那还不简单！先把 .slog 转成 CSV，再用 pandas 的 `read_csv`——2 秒读完！"

**大师**："小胖，你多了一次中间格式转换的开销——.slog → CSV → pandas。更好的方式是**直接让 pandas 读取你的自定义格式**——实现一个自定义解析器，输出 DataFrame，享受 pandas C 引擎的性能。"

**小白**："pandas 的 read_csv 底层到底有多少个引擎？它们有什么区别？"

**大师**："pandas 有三个 CSV 解析引擎：

| 引擎 | 语言 | 速度 | 特性 |
|------|------|------|------|
| **C 引擎** (默认) | C | 极快 | 基本功能完整，不支持 `skipfooter` |
| **Python 引擎** | Python | 慢 | 功能最全（支持所有参数），纯 Python 实现 |
| **PyArrow 引擎** | C++ | 极快 | 多线程读取，直接返回 Arrow 类型 |

关键是：这三个引擎通过一个统一的 `read_csv` 入口调用——你传入的参数决定了使用哪个引擎。pandas 默认用 C 引擎，当你的参数中包含 C 引擎不支持的特性时（如 `skipfooter`），自动回退到 Python 引擎。"

**【技术映射：三个引擎 = 三条高速公路——C 引擎（国道，大多数情况够用）、PyArrow 引擎（高铁，最新最快）、Python 引擎（村道，什么路都能走但慢）】**

**小白**："那我能写一个自己的解析器吗？不用这三个引擎。"

**大师**："当然可以——这就是 IO 子系统的扩展点。pandas 提供了 `pandas.io.common.file_exists`、`pandas.io.parsers.TextFileReader` 等工具类，你可以基于它们构建自己的读取器。

核心思路：写一个函数，接收文件路径和参数，返回 DataFrame：

```python
def read_slog(filepath, **kwargs):
    # 1. 打开文件
    # 2. 解析二进制格式 → 构建列数据
    # 3. 返回 pd.DataFrame(data, columns=...)
    pass
```

如果你想要 chunksize 支持和更高效的内存使用，可以参考 `TextFileReader` 的实现——它内部维护了一个文件句柄和一个 `_engine` 对象，支持迭代读取。"

**小胖**："那 dtype 推断呢？为什么有时 `read_csv` 把数字推断成字符串？"

**大师**："C 引擎的 dtype 推断策略是**分块采样**——读取前 100 行，尝试 `float()` 转换，如果大多数成功就推断为 float64。如果某列混合了数字和字符串——推断为 object。

关键问题：**dtype 推断是不确定的**——取决于采样块的内容。如果前 100 行某列恰好全是数字，推断为 int64；但第 101 行出现一个 'N/A'——读取时报错或转为 NaN（取决于 `na_values` 设置）。

最佳实践：**生产环境始终显式指定 `dtype`**，不要依赖自动推断。"

**大师总结**："IO 子系统的三个层次：
1. **使用者**：用 `read_csv` 的 30+ 参数精确控制读取行为
2. **调优者**：选择合适的引擎（C/Python/PyArrow）和参数组合
3. **扩展者**：实现自定义文件格式的 pandas 读取器

记住：**一个自定义解析器 + DataFrame 构造函数 = pandas 原生读取任何格式。**"

# generate_slog.py
import numpy as np

np.random.seed(42)
N = 200_000  # 模拟 20 万条

# .slog 格式: "timestamp|device_id|sensor_type|value|unit"
# 字段定长，用 | 分隔

sensors = ['TEMP','HUMID','PRESS','FLOW','VOLT']
units = ['C','%','Pa','L/min','V']

with open('data.slog', 'w') as f:
    for i in range(N):
        ts = 1711929600 + i * 60  # epoch 秒
        line = f"{ts}|{np.random.choice(devices)}|{np.random.choice(sensors)}|{np.random.uniform(0,100):.3f}|{np.random.choice(units)}\n"

print(f"已生成 {N:,} 条的 .slog 文件")

#### 步骤 2：自定义解析器 —— read_slog

```python
# step2_custom_parser.py
import pandas as pd
import numpy as np

def read_slog(filepath, **kwargs):
    """自定义 .slog 文件解析器，输出 DataFrame"""
    timestamps = []
    devices = []
    sensors = []
    values = []
    units = []

    with open(filepath, 'r') as f:
        for line in f:
            parts = line.strip().split('|')
            if len(parts) == 5:
                timestamps.append(int(parts[0]))
                devices.append(parts[1])
                sensors.append(parts[2])
                values.append(float(parts[3]))
                units.append(parts[4])

    df = pd.DataFrame({
        'timestamp': pd.to_datetime(timestamps, unit='s'),
        'device': pd.Categorical(devices),
        'sensor': pd.Categorical(sensors),
        'value': values,
        'unit': pd.Categorical(units),
    })
    return df

import time
start = time.perf_counter()
df = read_slog('data.slog')
t_custom = time.perf_counter() - start
print(f"自定义解析器: {t_custom:.3f}s, {len(df):,} 行")
print(f"dtypes:\n{df.dtypes}")
```

#### 步骤 3：预编译正则加速版

```python
# step3_compiled_regex.py
import pandas as pd
import re
import time

# 预编译正则比 split 更快（对于定长格式）
PATTERN = re.compile(r'(\d+)\|([^|]+)\|([^|]+)\|([\d.]+)\|([^|]+)')

def read_slog_regex(filepath):
    data = []
    with open(filepath, 'r') as f:
        for line in f:
            m = PATTERN.match(line)
            if m:
                data.append((int(m.group(1)), m.group(2), m.group(3),
                            float(m.group(4)), m.group(5)))
    # DataFrame 从元组列表构建（最快）
    df = pd.DataFrame(data, columns=['ts','device','sensor','value','unit'])
    df['ts'] = pd.to_datetime(df['ts'], unit='s')
    for col in ['device','sensor','unit']:
        df[col] = df[col].astype('category')
    return df

start = time.perf_counter()
df_re = read_slog_regex('data.slog')
t_regex = time.perf_counter() - start
print(f"预编译正则: {t_regex:.3f}s, {len(df_re):,} 行")
```

### 3.4 常见坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| PyArrow 引擎不支持某些参数 | PyArrow 解析器功能仍在完善 | 检查 pandas 版本对 `dtype_backend='pyarrow'` 的支持 |
| 自动 dtype 推断不稳定 | 推断依赖前 N 行采样 | 生产环境始终显式 `dtype={}` |
| C 引擎遇到 `skipfooter` 静默降级 | C 引擎不支持此参数 | 显式 `engine='python'` 或不用 skipfooter |

## 4. 项目总结

### 4.1 优点 & 缺点

| 维度 | C 引擎 | Python 引擎 | PyArrow 引擎 | 自定义解析器 |
|------|--------|------------|-------------|------------|
| 速度 | 极快 | 慢 | 极快 | 取决于实现 |
| 功能完整度 | 高 | 最高 | 中 | 完全自定义 |
| 自定义格式 | CSV/TSV/FWF | CSV/TSV/FWF | CSV/TSV | 任意格式 |
| 学习成本 | 低 | 低 | 低 | 高 |

### 4.2 深入：CSV 三个引擎的内部实现

**C 引擎（`pandas/io/parsers/c_parser_wrapper.py`）**：核心是一个 C 扩展模块 `pandas/_libs/parsers.pyx`。它使用**状态机**来解析 CSV——逐个字符扫描，记录字段开始/结束位置，在遇到分隔符或换行符时产出字段值。C 引擎的 dtype 推断策略是分块采样——读取前 100 行（默认），对这些行的每列尝试 `strtod`（字符串转 float）和 `strtol`（字符串转 int）。如果某一列的采样数据中 80% 以上能成功转为数字，就推断为数值类型；否则保持 object。

**Python 引擎（`pandas/io/parsers/python_parser.py`）**：纯 Python 实现的解析器。它用 `csv` 标准库（CPython 内置）做基础解析，然后逐列做类型推断。Python 引擎的特点是功能完整——支持 C 引擎不支持的所有参数（如 `skipfooter`、`quoting=QUOTE_NONE`、复杂的 `comment` 正则等）。当用户使用了这些参数时，pandas 自动切换到 Python 引擎（可能有性能损失）。

**PyArrow 引擎（`pandas/io/parsers/arrow_parser_wrapper.py`）**：调用 PyArrow 的 `csv.read_csv()`。PyArrow 解析器是多线程的——可以利用多核并行解析大 CSV 文件的多个 chunk。而且它直接返回 Arrow Table（列式内存格式），避免了 C 引擎"先解析为 Python 对象再转 NumPy"的中间步骤。当配合 `dtype_backend='pyarrow'` 使用时，pandas 的 DataFrame 底层直接持有 Arrow 数组——实现了从磁盘到内存的零拷贝链路。

**引擎选择决策表**：
- 标准 CSV/TSV 文件 + 追求极致速度 → 默认 C 引擎
- CSV 文件超大（> 10GB）+ 多核机器 → `engine='pyarrow'`
- CSV 有特殊格式（尾部注释行、特殊引用规则）→ `engine='python'`
- 非标准格式 → 自定义解析器 + DataFrame 构造函数

### 4.2 适用场景

- **适用场景**：自定义日志/二进制格式需要 pandas 原生速度、需要精确控制 dtype/NA 推断
- **不适用场景**：标准格式（CSV/Parquet）直接用内置读取器

### 4.3 注意事项

- **自定义解析器返回的 dtype 要显式设置**：否则自动推断可能退化为 object；
- **C 引擎的 `na_values` 参数**：指定哪些字符串应被识别为 NA——默认包括 `'', '#N/A', 'N/A', 'NULL', 'NaN', 'n/a', 'nan', 'null'`；
- **PyArrow 引擎的日期解析**：目前支持有限——复杂的日期格式建议用 C 引擎 + `parse_dates`。

### 4.4 常见踩坑经验

1. **C 引擎静默降级到 Python 引擎**：当你使用 `skipfooter`、`quoting=QUOTE_NONE` 等参数时——性能骤降但无提示。可以用 `engine='c'` 显式指定，如果用了不支持的参数会直接报错而非静默降级。
2. **`low_memory=False` 和 `low_memory=True` 的 dtype 推断不同**：前者全量扫描推断，后者分块推断——可能导致同一文件两次读取的 dtype 不同。`low_memory=True` 时，第一个 chunk 的 dtype 推断会影响整个文件的解析——如果后面的数据类型和第一个 chunk 不同，可能报错或转为 object。生产环境用 `low_memory=False` + 显式 `dtype={}` 最安全。
3. **`date_format='epoch'` 只适用于整型时间戳**：如果时间戳是浮点（含毫秒），epoch 模式会失败。使用 `parse_dates=['ts']` 让 pandas 自动推断。
4. **`read_csv` 的 `na_values` 和 `keep_default_na` 的交互**：`keep_default_na=True` 时，pandas 自动将 `'NA'`, `'N/A'`, `'null'` 等识别为缺失值。如果你有一列实际名为 "NA" 的数据（如化学元素钠的缩写——虽然现实中不太可能），需要设置 `keep_default_na=False` + 自定义 `na_values`。
5. **编码检测的陷阱**：`encoding='utf-8'` 对带 BOM 的 UTF-8 文件会保留 BOM 头（`\ufeff`）出现在第一列的列名中。使用 `encoding='utf-8-sig'` 自动去除 BOM。
