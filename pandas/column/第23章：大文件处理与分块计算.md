# 第23章：大文件处理与分块计算

## 1. 项目背景

某视频平台的数据团队每天凌晨需要处理前一天的 CDN 访问日志（cdn_access.log），单日日志约 50GB，包含约 1.5 亿行。日志格式为 JSON-like 字段，包含：时间戳、IP、视频 ID、流量字节数、状态码、省份、运营商。分析任务包括：各视频的播放量 Top 100、各运营商的流量分布、各小时的请求峰值、4xx/5xx 错误率统计。

数据工程师小方当前的脚本是一行 `pd.read_csv` 全量读入——他在 32GB 内存的开发机上跑的时候还能勉强撑住，但公司要求迁移到 8GB 内存的 Docker 容器中运行。全量读入在 8GB 内存下直接 OOM Killed。

**痛点一：全量读取内存爆炸**。50GB 日志如果包含大量字符串字段，读进 pandas 后内存可能膨胀到 80-100GB——远远超出任何常规机器的内存。小方尝试过用 `nrows=1000000` 只读前 100 万行"抽样分析"，但这个结果不能代表全天数据。

**痛点二：分析任务各自为战**。小方为了统计"各视频播放量 Top 100"写了 `df.groupby('video_id').size().nlargest(100)`，全量跑一次 30+ 分钟。当产品经理临时追加一个"各运营商流量分布"需求时，又得重新全量扫描一次。

**痛点三：失败后需要从头再来**。一次分析跑了 25 分钟后因为连接超时失败，所有中间结果丢失。小方需要"断点续传"能力——失败后可以从上次失败的分块继续。

痛点流程：

```
CDN 日志 (50GB, 1.5亿行)
  └── pd.read_csv 全量读入
        ├── 内存 8GB → OOM Killed
        ├── 多个分析任务 → 各自全量扫描
        └── 失败 → 从头再来 → 时间浪费
```

本章将系统讲解 pandas 的分块处理策略——chunksize 分块读取、分块聚合、分块过滤、MapReduce 思路简化实现、以及内存峰值监控。

## 2. 项目设计：剧本式交锋对话

> 场景：小方打开监控看到 OOM 告警，小胖凑过来说"换个大内存机器啊"。

**小胖**：（喝着奶茶）"50GB 日志就爆了？换一台 128GB 内存的机器不就好了？公司又不是没钱！"

**大师**："小胖，横向扩容（加内存）不是免费的——128GB 的云服务器一个月几千块。而且数据量半年后可能涨到 200GB，难道一直加内存？更好的思路是**纵向降载**——在有限资源内把事情做完。pandas 的 `chunksize` 就是干这件事的。"

**小白**："chunksize 是什么意思？读进来的是多个 DataFrame？"

**大师**："`chunksize` 把一次全量读取变成**多次小块读取**。当你传入 `chunksize=100000` 时，`read_csv` 不再返回一个 DataFrame，而是返回一个**迭代器**——每次都吐出一个 10 万行的 DataFrame：

```python
# 传统（全量）
df = pd.read_csv('big.csv')  # 一口气全吞 → 爆内存

# 分块（迭代）
for chunk in pd.read_csv('big.csv', chunksize=100000):
    process(chunk)  # 一次只处理 10 万行
```

每次 for 循环拿到的是 10 万行的 chunk，处理完就可以释放。**任何时候内存中只存在一个 chunk**——假设 10 万行占 30MB，那总内存占用始终在 30MB 左右，不管原始文件是 1GB 还是 100GB。"

**【技术映射：chunksize = 吃自助餐——不是把整个餐台搬过来，而是一次端一盘，吃完再拿】**

**小白**："那统计'各视频播放量 Top 100'这种全局排序怎么办？分块后每个 chunk 只有局部信息。"

**大师**："这就是分块计算的核心挑战——**把全局聚合拆成多个局部聚合，再合并**。本质上是 MapReduce 的简化版：

1. **Map 阶段（分块）**：每个 chunk 内做局部聚合
2. **Reduce 阶段（合并）**：把所有局部聚合结果合并

对于"各视频播放量 Top 100"：

```python
local_counts = []
for chunk in pd.read_csv('log.csv', chunksize=100000):
    # Map: chunk 内的局部计数
    local = chunk['video_id'].value_counts()
    local_counts.append(local)

# Reduce: 合并所有局部计数并取 Top 100
total_counts = pd.concat(local_counts).groupby(level=0).sum()
top100 = total_counts.nlargest(100)
```

核心技巧：chunk 内只做聚合（不保留明细），输出远小于输入。"

**【技术映射：MapReduce = 各班级算自己的平均分（Map），然后年级汇总所有班级的平均（Reduce）】**

**小胖**："那只做过滤不聚合呢？比如只保留 4xx 错误的日志？"

**大师**："过滤比分块聚合更简单——逐块过滤，逐块写出：

```python
for chunk in pd.read_csv('log.csv', chunksize=100000, iterator=True):
    errors = chunk[chunk['status_code'].between(400, 499)]
    errors.to_csv('errors.csv', mode='a', index=False, header=not os.path.exists('errors.csv'))
```

这里 `mode='a'` 是追加模式，`header=not os.path.exists(...)` 保证只在第一次写入时写表头。"

**小白**："如果分块处理到一半失败了怎么办？"

**大师**："加**断点续传**机制——记录已经处理到文件的第几行：

```python
# 从上次失败的断点继续
skip_rows = load_checkpoint()  # 上次已处理的行数
reader = pd.read_csv('log.csv', chunksize=100000, skiprows=range(1, skip_rows+1))
for i, chunk in enumerate(reader):
    process(chunk)
    save_checkpoint((i+1) * 100000)  # 更新断点
```

这样即使 Docker 容器重启，也能从断点继续，不需要从头扫描。"

**大师总结**："分块处理的四种模式：
1. **分块过滤**：逐块筛选，逐块写出
2. **分块聚合**：chunk 内做 local agg，chunk 间做 reduce
3. **分块统计**：chunk 内做 local count/sum，chunk 间做 global sum
4. **断点续传**：记录行数进度，失败后恢复

记住：**chunksize 是应对"数据量 > 内存"的第一道防线，但真正的解决方案是"数据量 > 内存"时把聚合逻辑往数据端推动（如用 SQL 或 Spark）。**"

### 3.3 分步实现

#### 步骤 1：分块过滤——逐块读，逐块筛选写出

**目标**：从大文件中筛选出所有 4xx/5xx 错误日志。

```python
# step1_chunk_filter.py
import pandas as pd
import os

output_file = 'cdn_errors.log'
if os.path.exists(output_file):
    os.remove(output_file)

error_count = 0
total_count = 0
for i, chunk in enumerate(pd.read_csv('cdn_access.log', chunksize=50000)):
    total_count += len(chunk)
    errors = chunk[chunk['status_code'] >= 400]
    error_count += len(errors)

    # 追加写入（第一次写表头，后面不写）
    errors.to_csv(output_file, mode='a', index=False,
                  header=(i == 0))

    if i % 5 == 0:
        print(f"  chunk {i}: {total_count:,} 行已处理, {error_count} 条错误")

print(f"\n总计: {total_count:,} 行, {error_count} 条错误 ({error_count/total_count*100:.2f}%)")
```

#### 步骤 2：分块聚合——MapReduce 全局聚合

**目标**：用 MapReduce 模式统计各视频播放量和各 ISP 流量。

```python
# step2_chunk_aggregate.py
import pandas as pd

# Map 阶段：每个 chunk 内局部聚合
video_counts = []
isp_bytes = []

for chunk in pd.read_csv('cdn_access.log', chunksize=50000):
    # 局部：各视频播放量
    vc = chunk['video_id'].value_counts()
    video_counts.append(vc)
    # 局部：各 ISP 总流量
    ib = chunk.groupby('isp')['bytes'].sum()
    isp_bytes.append(ib)

# Reduce 阶段：合并
video_total = pd.concat(video_counts).groupby(level=0).sum()
isp_total = pd.concat(isp_bytes).groupby(level=0).sum()

print("===== Top 10 视频播放量 =====")
print(video_total.nlargest(10).to_string())

print(f"\n===== ISP 流量分布 =====")
isp_total_gb = (isp_total / 1024**3).round(2)
for isp, gb in isp_total_gb.items():
    print(f"  {isp}: {gb:.2f} GB")
```

#### 步骤 3：分块多指标统计——HLL 近似 UV

**目标**：在不爆内存的前提下统计 UV 和 PV。

```python
# step3_chunk_uv_pv.py
import pandas as pd
import numpy as np

# 精确 UV 需要存储所有唯一 IP → 内存可能不足
# 替代方案：分块统计各维度
hourly_pv = []
hourly_uv_ip = []

for chunk in pd.read_csv('cdn_access.log', chunksize=50000, parse_dates=['timestamp']):
    chunk['hour'] = chunk['timestamp'].dt.hour

    # 每小时 PV
    hp = chunk['hour'].value_counts()
    hourly_pv.append(hp)

    # 每小时 UV（每小时内去重 IP）
    hu = chunk.groupby('hour')['ip'].nunique()
    hourly_uv_ip.append(hu)

# 合并
pv_per_hour = pd.concat(hourly_pv).groupby(level=0).sum().sort_index()
# UV 不能直接 sum（不同 chunk 的 IP 可能重复），用近似
# 简单近似：chunk 内的 UV 取 max（上界估计）
uv_per_hour = pd.concat(hourly_uv_ip).groupby(level=0).max().sort_index()

print("===== 每小时 PV/UV =====")
summary = pd.DataFrame({'PV': pv_per_hour, 'UV(近似)': uv_per_hour})
print(summary.to_string())
```

#### 步骤 4：断点续传——失败后继续

**目标**：记录已处理的行数，重启后从断点继续。

```python
# step4_checkpoint.py
import pandas as pd
import json
import os

CHECKPOINT_FILE = 'cdn_checkpoint.json'

def load_checkpoint():
    if os.path.exists(CHECKPOINT_FILE):
        with open(CHECKPOINT_FILE) as f:
            return json.load(f)['rows_processed']
    return 0

def save_checkpoint(rows):
    with open(CHECKPOINT_FILE, 'w') as f:
        json.dump({'rows_processed': rows}, f)

# === 带断点续传的处理 ===
skip = load_checkpoint()
print(f"从第 {skip:,} 行开始...")

total_processed = skip
for i, chunk in enumerate(pd.read_csv('cdn_access.log', chunksize=50000, skiprows=range(1, skip+1) if skip > 0 else None)):
    total_processed += len(chunk)

    # 模拟处理
    result = chunk['status_code'].value_counts()

    # 每 10 个 chunk 保存一次断点
    if i % 10 == 0:
        save_checkpoint(total_processed)
        print(f"  断点: {total_processed:,} 行已处理")

save_checkpoint(total_processed)
print(f"完成: {total_processed:,} 行")
os.remove(CHECKPOINT_FILE)  # 完成清理断点
```

### 3.4 常见坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| chunksize 设为 1 导致内存不足 | 每 1 行为一个 chunk → 函数调用开销巨大 | chunksize 至少 10000 行 |
| 分块聚合中 groupby.agg 用错了 reduce | chunk 间 sum 可行，但 mean 不能直接 sum | mean = sum/chunk 数（加权），或存 sum+count |
| `to_csv(mode='a')` 行头重复 | 每个 chunk 都写了 header | 只在第一个 chunk 写 header |
| 分块读取时 dtypes 不一致 | 不同 chunk 自动推断不同 dtype | 在全量读之前显式指定 `dtype={}` |

## 4. 项目总结

### 4.1 优点 & 缺点

| 维度 | 分块计算 (chunksize) | 全量读入 | Spark/Dask |
|------|--------------------|---------|-----------|
| 内存占用 | 恒定（chunk 大小） | 线性增长 | 分布式管理 |
| 实现复杂度 | 中 | 低 | 高 |
| 全局排序 | 困难（需 extra logic） | 简单 | 内置 |
| 断点续传 | 手写简单 | 不支持 | 内置容错 |
| 部署门槛 | 单机 pandas | 单机 pandas | 集群 |
| 性能 | 单机 CPU 全利用 | 单机 | 分布式 |

### 4.2 适用场景

- **适用场景**：
  1. 数据量大于内存但仍在单机可处理范围（2-50GB）
  2. 只需要过滤/聚合/采样，不需要全局排序
  3. 日志清洗和初步聚合（像 CDN 日志分析）
  4. 在 Docker/K8s 等限制内存的容器中运行
- **不适用场景**：
  1. 需要全局 Top-N 且 N 很大——MapReduce 合并开销大
  2. 需要多次全表扫描的复杂分析——用数据库更合适
  3. TB 级数据——超出单机能力，用 Spark

### 4.3 注意事项

- **chunksize 选择**：太小（<1000）→函数调用开销大，太大→内存接近上限；建议 50K-200K；
- **显式指定 dtype**：不同 chunk 可能推断出不同 dtype（尤其日期列），必须显式指定 `dtype` 和 `parse_dates`；
- **groupby 的 reduce 策略**：`sum`/`count` 可以直接 sum，`mean` 需要存 sum+count 再除，`max`/`min` 可以直接 max/min，`nunique` 不可分块精确计算；
- **分块不适用于所有操作**：`sort_values`、`shift`、`rolling` 等依赖全局顺序的操作无法直接分块。

### 4.4 常见踩坑经验

1. **用 `df.append(chunk)` 在循环中累积**：每次 append 都会复制整个 DataFrame——O(n^2) 复杂度。替代：用 `list` 收集各 chunk 结果，循环结束后 `pd.concat`。
2. **分块时忘记传 dtype 导致不同 chunk 的 category 编码不一致**：第一个 chunk 的 category codes 是 0/1/2，第二个 chunk 可能重新编码——concat 后 category 列退化。解决：显式定义 `CategoricalDtype`。
3. **断点续传的 `skiprows` 性能陷阱**：`skiprows` 在 C 解析器中需要逐行扫描跳过的行——跳过的行数越多越慢。如果文件巨大且经常失败，考虑按日期分区存储（Parquet），失败只需重跑当天的分区。
