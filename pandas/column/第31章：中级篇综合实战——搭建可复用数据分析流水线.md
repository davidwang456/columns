# 第31章：【中级篇综合实战】搭建可复用数据分析流水线

## 1. 项目背景

某 SaaS 公司为电商客户提供"用户行为分析"服务。每天凌晨需要处理客户上传的前一天用户行为日志（约 200 万条/天，Parquet 格式），加工成用户级特征指标（留存率、活跃度、转化漏斗），生成一份 Excel 报告并写回数据库供 BI 查询。

数据工程师老马负责维护这套数据处理流水线。目前系统有四个严重问题：

**痛点一：代码耦合度高，增加一个指标要改 5 处**。从数据读取 → 清洗 → 特征加工 → 指标聚合 → 报告导出的每个环节的代码都在同一个 `main.py` 文件中（约 800 行）。加一个"30 日留存率"指标需要修改特征加工函数、聚合函数、导出列名——三个地方各改一处。上次运营要求加一个"分设备类型的页面停留时长"指标，老马改了 6 个地方才勉强跑通。

**痛点二：增量处理缺失，全量扫描 OOM**。线上客户数据每天增量上传，但脚本每次都全量扫描所有历史数据。老马在 16GB 内存的机器上跑 2 个月数据就 OOM 了，客户却需要看半年的趋势。他试图用 `chunksize` 分块处理，但因为缺乏增量设计，每个 chunk 都需要和全量历史数据做 join——越跑越慢。

**痛点三：质量保障靠"肉眼看"**。脚本末尾的 `print(df.describe())` 是老马唯一的"质量检查"。上个月因为上游日志格式变更（新增字段导致列名偏移），pandas 列名匹配失败，某客户的"活跃用户数"被计算为 0——直到客户投诉报表数据不对才知道。

**痛点四：无重试和断点续传机制**。凌晨 3 点数据库连接超时导致脚本失败——整个流水线需要从第 1 步重新执行，又因为全量扫描——再次 OOM——进入死循环。老马手动修复了 5 次数据后，决定彻底重构。

痛点流程：

```
用户行为日志 (200万条/天, Parquet 分区)
  └── main.py (800行, 全耦合)
        ├── 加指标 → 改 5-6 处代码
        ├── 无增量设计 → 全量扫描 → OOM
        ├── 无质检 → print(df.describe()) → 客户投诉才发现
        └── 无重试 → 失败后从头来 → 死循环
```

本章将中级篇 14 章（17-30 章）的核心能力融会贯通——pipe 管道（第 17 章）、高级 GroupBy（第 18 章）、MultiIndex（第 19 章）、特征工程（第 20 章）、SQL 集成（第 21 章）、Parquet 分区（第 22 章）、分块计算（第 23 章）、向量化（第 24 章）、质量监控（第 27 章）、日志与参数化（第 29 章）——搭建一条可配置、可测试、可重跑、可观测的生产级数据分析流水线。

## 2. 项目设计：剧本式交锋对话

> 场景：老马在代码评审会上展示了 800 行的 main.py，大师的表情逐渐凝重。小胖还在旁边吃辣条。

**小胖**：（吃着辣条）"800 行一个文件怎么了？我见过 3000 行的！代码多说明牛！"

**大师**："代码行数从来不等同于能力。800 行拆成 8 个文件——前者是技术债的复利，后者是可维护的资产。**单一职责**是老马的流水线最需要的设计原则：读、洗、算、存、查——每一步是一个独立模块。"

**小白**："那具体的模块结构怎么设计？每个模块的输入输出是什么？"

**大师**："参考 pipe 管道模式（第 17 章）+ 参数化运行（第 29 章）：

```
pipeline/
├── config.yaml            # 配置：日期/客户/质量阈值
├── main.py                # 主入口：串联各模块
├── src/
│   ├── loader.py          # 数据加载（Parquet 分区 + 增量过滤）
│   ├── cleaner.py         # 数据清洗（pipe 管道）
│   ├── features.py        # 特征加工（transform/rank/cumcount）
│   ├── aggregator.py      # 指标聚合（pivot_table + groupby 分层）
│   ├── state.py           # 状态管理（增量书签 + 断点续传）
│   ├── exporter.py        # 报告导出（ExcelWriter + openpyxl 格式化）
│   └── quality.py         # 质量检查（五维质量 + 统计基线 + 告警）
├── tests/                 # 每个 src 模块的单元测试
└── logs/                  # 运行日志（按日期滚动）
```

每个模块的契约：输入是 DataFrame(s)，输出是 DataFrame(s)——纯函数无副作用。加一个新指标？只改 `features.py` 和 `aggregator.py`。"

**【技术映射：模块化流水线 = 装配线——每个工位做一个工序，加一个新工序只需插一个工位】**

**小白**："增量处理怎么做？既要增量读（省内存），又要能看半年趋势（全量分析）。"

**大师**："这是增量流水线的经典设计——**明细即抛，聚合持久化**：

1. **增量读取**：用 Parquet 分区过滤（第 22 章），只读 `date > last_synced_date` 的数据
2. **实时聚合**：把增量数据按用户聚合成"今日指标快照"
3. **合并历史**：把今日快照追加到历史指标 Parquet 表中
4. **全量报告**：从历史指标表中读取需要的日期范围做趋势计算

核心思想：**明细数据只处理一次并立即丢弃，存储为聚合指标后增量追加。生成报告时只读聚合表——数据量从 200 万条/天 × 180 天 = 3.6 亿行，缩小为 5000 用户 × 180 天 = 90 万行。**"

**【技术映射：增量流水线 = 流水账本——每天记当天的收支（增量），月底翻账本算总账（全量报告），不需要把每天的账本全部摊开】**

**小胖**："那重试和断点续传呢？我凌晨 3 点跑挂了总不能 5 点起来手动重跑吧？"

**大师**："用 `state.py` 管理两个关键状态：
1. **同步书签**：记录上次成功处理的日期。失败后重跑自动从断点继续。
2. **重试计数**：如果连续失败 3 次，发送告警并停止自动重试（防止死循环）。

```python
# state.py —— 断点续传
def get_last_synced_date(): ...
def set_last_synced_date(date): ...
def increment_retry_count(): ...
def reset_retry_count(): ...
```

结合第 29 章的 `sys.exit(1)` 和 `logging.error`——失败时退出码非零，cron/Airflow 检测到后触发重试或告警。"

**【技术映射：断点续传 = 游戏存档——失败了从存档点继续，不用从头打】**

**小白**："那质量检查的三层架构呢？和基础篇的五维检查有什么区别？"

**大师**："基础篇的五维检查是'进入和离开时各检查一次'。中级篇升级为**三层质量架构**：

1. **规则层**（进入时）：完整性/唯一性/合法性——硬约束，任何违反直接中止流水线
2. **统计层**（离开时）：均值和标准差基线对比——软约束，偏离 3σ 告警但不中止
3. **一致性层**（聚合前 vs 聚合后）：汇总值 = 明细总和——逻辑约束，不通过则告警

这才是'生产可信赖'的流水线——而不是'跑起来靠运气'。"

**大师总结**："中级篇收官——我们串联了 14 章的核心能力到一条流水线中：
- 数据加载：Parquet 分区 + 增量过滤（第 22/23 章）
- 清洗：pipe 管道 + where/mask（第 17 章）
- 特征：transform/rank/cumcount（第 18/26 章）
- 聚合：pivot_table + groupby 分层（第 11/19/20 章）
- 导出：ExcelWriter + openpyxl 格式（第 14 章）
- 运维：质量监控 + logging + argparse（第 27/29 章）
- 状态管理：增量书签 + 断点续传（本章新增）

下面我们搭建一条生产级流水线。"

### 3.4 分步实现

#### 步骤 1：loader.py —— 增量 Parquet 读取

**目标**：读取指定日期范围的 Parquet 分区数据，支持增量模式。

```python
# src/loader.py
import pandas as pd
from pathlib import Path
import logging

logger = logging.getLogger(__name__)

def load_events(data_dir='data', start_date=None, end_date=None, incremental=False):
    """读取 Parquet 分区数据，支持增量过滤"""
    all_dates = sorted([
        d.name.split('=')[1]
        for d in Path(data_dir).glob('date=*')
        if (d / 'events.parquet').exists()
    ])

    if start_date:
        all_dates = [d for d in all_dates if d >= str(start_date)]
    if end_date:
        all_dates = [d for d in all_dates if d <= str(end_date)]

    if not all_dates:
        logger.warning(f"无匹配数据: {start_date} ~ {end_date}")
        return pd.DataFrame()

    # 使用 PyArrow 的 filters 做分区裁剪
    dfs = []
    for d in all_dates:
        partition_path = Path(data_dir) / f'date={d}' / 'events.parquet'
        chunk = pd.read_parquet(partition_path)
        dfs.append(chunk)

    df = pd.concat(dfs, ignore_index=True)
    logger.info(f"加载 {len(all_dates)} 天, {len(df):,} 条事件, {df['user_id'].nunique():,} 用户")
    return df
```

#### 步骤 2：state.py —— 断点续传与状态管理

**目标**：管理同步书签，支持失败重试和断点续传。

```python
# src/state.py
import json
import os
import logging

logger = logging.getLogger(__name__)

STATE_FILE = 'pipeline_state.json'

def load_state():
    if os.path.exists(STATE_FILE):
        with open(STATE_FILE) as f:
            return json.load(f)
    return {'last_synced_date': None, 'retry_count': 0, 'total_runs': 0}

def save_state(state):
    with open(STATE_FILE, 'w') as f:
        json.dump(state, f, indent=2, default=str)

def get_last_synced_date():
    return load_state().get('last_synced_date')

def mark_synced(date):
    state = load_state()
    state['last_synced_date'] = str(date)
    state['retry_count'] = 0
    state['total_runs'] += 1
    save_state(state)
    logger.info(f"状态已更新: last_synced={date}")

def should_retry(max_retries=3):
    state = load_state()
    if state['retry_count'] >= max_retries:
        logger.error(f"已达最大重试次数 {max_retries}，停止重试")
        return False
    state['retry_count'] += 1
    save_state(state)
    logger.info(f"重试 {state['retry_count']}/{max_retries}")
    return True
```

#### 步骤 3：features.py —— 用户特征加工

```python
# src/features.py
import pandas as pd
import numpy as np
import logging

logger = logging.getLogger(__name__)

def compute_user_features(events):
    logger.info("特征加工...")

    # 事件计数（pivot_table 宽表化）
    event_counts = events.pivot_table(
        index='user_id', columns='event_type', values='event_time',
        aggfunc='count', fill_value=0
    ).add_prefix('cnt_')

    # 活跃天数、设备使用
    active_days = events.groupby('user_id')['date'].nunique().rename('active_days')
    device_counts = events.pivot_table(
        index='user_id', columns='device', values='event_time',
        aggfunc='count', fill_value=0
    ).add_prefix('device_')

    # 最后活跃时间
    last_active = events.groupby('user_id')['event_time'].max().rename('last_active')

    # 合并特征表
    features = pd.concat([
        event_counts, active_days, device_counts, last_active
    ], axis=1).reset_index()
    features = features.fillna(0)

    # 派生指标：转化率、参与度评分
    features['purchase_rate'] = (
        features['cnt_purchase'] / features['cnt_page_view'].replace(0, 1) * 100
    ).round(2)
    features['engagement_score'] = (
        features['cnt_click'] * 2 +
        features['cnt_add_cart'] * 5 +
        features['cnt_purchase'] * 10
    ).clip(0, 100)

    logger.info(f"特征完成: {len(features):,} 用户, {len(features.columns)} 个特征")
    return features
```

#### 步骤 4：aggregator.py + quality.py + exporter.py

```python
# src/aggregator.py
import pandas as pd
import logging

logger = logging.getLogger(__name__)

def compute_kpi(events, features):
    logger.info("指标聚合...")
    kpi = {}

    kpi['总用户数'] = events['user_id'].nunique()
    kpi['总事件数'] = len(events)
    kpi['人均事件数'] = round(kpi['总事件数'] / kpi['总用户数'], 1)

    funnel = events.groupby('event_type')['user_id'].nunique()
    for event in ['page_view','click','add_cart','purchase']:
        if event in funnel.index:
            kpi[f'{event}_用户'] = funnel.get(event, 0)
    if 'page_view' in funnel.index and funnel['page_view'] > 0:
        kpi['浏览→点击率'] = round(funnel.get('click',0) / funnel['page_view'] * 100, 2)
        kpi['浏览→购买率'] = round(funnel.get('purchase',0) / funnel['page_view'] * 100, 2)

    # 用户分层
    hi = features[features['engagement_score'] >= 60]
    mid = features[(features['engagement_score'] >= 20) & (features['engagement_score'] < 60)]
    low = features[features['engagement_score'] < 20]
    kpi['高活跃用户'] = len(hi)
    kpi['中活跃用户'] = len(mid)
    kpi['低活跃用户'] = len(low)

    daily_active = events.groupby('date')['user_id'].nunique().sort_index().reset_index()
    daily_active.columns = ['日期','DAU']

    kpi_df = pd.DataFrame([kpi]).T.reset_index()
    kpi_df.columns = ['指标','数值']

    logger.info(f"KPI 完成: {len(kpi_df)} 个指标")
    return kpi_df, daily_active
```

```python
# src/quality.py
import logging
logger = logging.getLogger(__name__)

def check_quality(df, config):
    checks = []
    qc = config.get('quality', {})
    if len(df) < qc.get('min_rows', 0):
        checks.append(f"[FAIL] 行数 {len(df):,} < 阈值 {qc['min_rows']:,}")
    null_pct = df.isna().mean().max() * 100
    if null_pct > qc.get('max_null_pct', 10):
        checks.append(f"[FAIL] 最大空值率 {null_pct:.1f}% > {qc['max_null_pct']}%")
    if 'user_id' in df.columns:
        dup_pct = df['user_id'].duplicated().sum() / len(df) * 100
        if dup_pct > qc.get('max_dup_pct', 2):
            checks.append(f"[FAIL] 重复率 {dup_pct:.1f}% > {qc['max_dup_pct']}%")
    for c in checks:
        logger.warning(c)
    return len([c for c in checks if 'FAIL' in c]) == 0
```

import pandas as pd

def export_report(kpi_df, daily_active, features, config):

#### 步骤 5：main.py —— 主入口（带重试和断点续传）

```python
# main.py
import yaml, logging, sys, time
from pathlib import Path
from src.loader import load_events
from src.features import compute_user_features
from src.aggregator import compute_kpi
from src.quality import check_quality
from src.exporter import export_report
from src.state import get_last_synced_date, mark_synced, should_retry

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s',
    handlers=[
        logging.FileHandler('logs/pipeline.log', encoding='utf-8'),
        logging.StreamHandler()
    ]
)
logger = logging.getLogger(__name__)

def main():
    with open('config.yaml', encoding='utf-8') as f:
        config = yaml.safe_load(f)

    t0 = time.perf_counter()
    try:
        # 增量模式：从上次断点继续
        start_date = config['dates']['start']
        if config.get('incremental'):
            last_date = get_last_synced_date()
            if last_date:
                start_date = last_date
                logger.info(f"增量模式: 从 {start_date} 继续")

        events = load_events(start_date=start_date, end_date=config['dates']['end'])
        if len(events) == 0:
            logger.info("无新数据，跳过")
            return

        if not check_quality(events, config):
            logger.error("质量检查失败，流水线中止")
            if not should_retry(config.get('retry', {}).get('max_retries', 3)):
                sys.exit(1)
            return

        features = compute_user_features(events)
        kpi_df, daily_active = compute_kpi(events, features)
        export_report(kpi_df, daily_active, features, config)

        # 更新状态
        mark_synced(config['dates']['end'])
        logger.info(f"流水线完成, 耗时 {time.perf_counter()-t0:.2f}s")

    except Exception as e:
        logger.error(f"失败: {e}", exc_info=True)
        should_retry(config.get('retry', {}).get('max_retries', 3))
        sys.exit(1)

if __name__ == '__main__':
    main()
```

### 3.5 常见坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|----------|
| loader 拉取过多分区 OOM | 没限制日期范围 | 配置显式 start/end |
| 特征表 join 后行数膨胀 | 左连接产生 NaN 行 | fillna(0) + 检查行数 |
| 增量模式第一天无数据 | state 表为空 | 初始化为 start_date |
| 报告覆盖前次结果 | 文件名固定 | 文件名加时间戳后缀 |

## 4. 项目总结

### 4.1 优点 & 缺点

| 维度 | 本章模块化流水线 | 单体 main.py | Airflow DAG |
|------|---------------|-------------|-------------|
| 可维护性 | 独立模块，单改 | 800 行耦合 | 配置驱动 |
| 可测试性 | 每模块独立测试 | 不可测试 | 难测试 |
| 可观测性 | logging + quality + 状态文件 | print | Airflow 内置 |
| 增量处理 | 内置书签机制 | 无 | 内置 backfill |
| 断点续传 | state.py 管理 | 无 | 内置 retry |
| 部署复杂度 | cron 一行 | cron 一行 | Airflow 集群 |

### 4.2 适用场景

- **适用场景**：中小规模数据管道（< 100GB/天）、需要快速迭代的数据产品、需要增量+全量双模的场景
- **不适用场景**：超大规模（> 1TB/天）→ Spark/Dask；复杂多步骤依赖 → Airflow

### 4.3 注意事项

- **状态文件并发安全**：如果多个 cron 实例同时跑，`state.json` 可能被竞态写入——加文件锁或使用数据库；
- **增量 + 全量双模**：同一套代码通过配置切换——`incremental: true/false`；
- **指标历史表的 schema 演进**：新增指标时，Parquet 的 schema 需要兼容——用 `schema_evolution` 或新建列。

### 4.4 常见踩坑经验

1. **模块间传递 DataFrame 的 dtype 退化**：pipe 传递中 category → object。在每个模块出口统一 `astype`。
2. **配置文件修改不生效**：YAML 被 `importlib` 缓存——流水线重启才能生效。长期方案：用 `watchdog` 监控配置变更。
3. **增量模式下的"数据空洞"**：某天数据完全缺失但 state 表已推进——下次增量从空洞之后开始，空洞日的数据永远丢失。需要回溯机制（手动设置 start_date）。
