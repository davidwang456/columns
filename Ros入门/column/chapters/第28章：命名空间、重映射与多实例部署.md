# M03 · 命名空间、重映射与多实例部署

> 本章目标字数：3000–5000。统一环境见 [ENV.md](../ENV.md)。

## 1 项目背景

### 业务场景

仿真要在同一 Gazebo 世界起 **两台 TurtleBot3**，若都占用 **`/cmd_vel`、`/scan`**，控制器与 RViz 无法区分数据源。**命名空间（namespace）** 把资源名前缀化：**`/bot1/scan`** vs **`/bot2/scan`**。**Remapping** 则解决「**算法写死了 `scan` 话题名**」但**硬件驱动发布 `lidar/front/scan`**」的对接问题——**不改 C++**，在 **Launch** 层完成布线。

### 痛点放大

1. **多层 Launch** `include` 时 **namespace 被 push 两次** → 出现 `/bot1/bot1/...`。
2. **TF frame** `base_link` 未随 namespace 自动改名，和 topic 不一致。
3. **参数文件**里节点名与真实 **FQName** 不一致 → 参数未加载。

```mermaid
flowchart TB
  subgraph ns1 [namespace bot1]
    scan1[/bot1/scan]
    cmd1[/bot1/cmd_vel]
  end
  subgraph ns2 [namespace bot2]
    scan2[/bot2/scan]
  end
```

**本章目标**：熟练使用 **Python Launch** 的 **`PushRosNamespace`**、**`GroupAction`**、**`remappings`**；用 CLI 验证 **topic/param** 全名；说明 **TF** 与 **namespace** 的边界。

## 2 项目设计

### 剧本对话

**小胖**：namespace 不就是字符串拼接吗？我自己在话题名前写 `/bot1` 行不行？

**小白**：那 Nav2 里几百个参数也要手改吗？

**大师**：手动拼接**易漏**且难维护。**Launch** 用 **`PushRosNamespace`** 一次性给子树内所有节点加前缀；**参数文件**用 **相对节点名**（如 `nav2_params.yaml` 内层级）配合 namespace **解析成绝对名**。

**技术映射**：**namespace** = **graph 资源前缀**（topics/services/actions/params 的规则集合）。

---

**小胖**：remap 和 param 重载分不清。

**大师**：**Remap** 改**连接关系**（逻辑名→实际名）；**param** 改**数值**。把 remap 想成「**重接线**」，param 想成「**调旋钮**」。

**技术映射**：**RemappingRule** 在 **Node 构造**时生效。

---

**小白**：TF 树里还是 `base_link` 冲突怎么办？

**大师**：TF **frame_id 不按 namespace 自动改名**——需 **多机器人 URDF / prefix** 或 **static TF 前缀策略**（`bot1/base_link` 等），与 topic namespace **分别设计**。

**技术映射**：**TF 命名空间** = **frame_id 字符串**，与 ROS graph namespace **正交**。

## 3 项目实战

### 环境准备

与 [ENV.md](../ENV.md) 一致：**Ubuntu 22.04 + ROS 2 Humble**，`source /opt/ros/humble/setup.bash`。

本章额外依赖：`sudo apt install ros-humble-turtlesim`；多机/Nav2 演示再加 **`ros-humble-nav2-bringup`**（按需，`apt search` 核对包名）。

### 分步实现

#### 步骤 1：PushRosNamespace 起双 turtlesim

- **目标**：同一 ROS 图中 **两套路由** 前缀隔离（`/r1/...`、`/r2/...`）。
- **命令**：新建 `launch/two_turtles.launch.py`：

```python
from launch import LaunchDescription
from launch.actions import PushRosNamespace
from launch_ros.actions import Node

def generate_launch_description():
    return LaunchDescription([
        PushRosNamespace('r1'),
        Node(package='turtlesim', executable='turtlesim_node', name='sim'),
    ])
```

第二实例用 `PushRosNamespace('r2')` 包另一个 `Node`（或 **`GroupAction`** 嵌套）。

- **预期输出**：`ros2 topic list` 出现 **`/r1/turtle1/pose`** 等（以实际为准）；**两个 turtlesim** 窗口。
- **坑与解法**：**重复 PushRosNamespace** → `/r1/r1/...` —— 用 **一层 Group** 管清边界（本章正文）。

#### 步骤 2：remapping `cmd_vel`

- **目标**：**不改源码**，在 Launch 层把 **逻辑名** 接到 **硬件实际名**。
- **代码**：

```python
Node(
    package='teleop_twist_keyboard',
    executable='teleop_twist_keyboard',
    namespace='r1',
    remappings=[('cmd_vel', 'diff_drive_controller/cmd_vel_unstamped')],
)
```

（具体 **remap 对** 按底盘驱动调整。）

- **预期输出**：`ros2 topic info /r1/cmd_vel`（或 remap 后路径）与 **teleop** 联通。
- **坑与解法**：**remap 不改 TF** —— **frame_id** 仍须与 **URDF** 一致（[B09](第21章：TF2-坐标系与静态变换.md)、[M12](第37章：多机器人协同与通信隔离.md)）。

#### 步骤 3：CLI 验证

- **目标**：确认 **topic / node / param** 全名无串台。
- **命令**：

```bash
ros2 topic list
ros2 node list
ros2 param list /r1/sim
```

- **预期输出**：**r1 与 r2** 列表可区分；`param list` **层级**与 launch 一致。
- **坑与解法**：**参数写在 YAML 里用相对名**时，须理解 **FQName** 解析 —— 与 **namespace** 乘在一起。

## 4 项目总结

### 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| namespace | 一图多实例 | TF/参数容易漏改 |
| remap | 不动源码接硬件 | Launch 复杂度上升 |

### 适用场景

- 仿真多机、真车 + 笔记本调试并存。

### 不适用场景

- 单机器人全绝对名极简 demo。

### 注意事项

- **`global topic` 以 `/` 开头**不受当前 namespace 影响——慎用绝对名。

### 常见踩坑经验

1. **重复 push** 导致路径爆炸。
2. **RViz fixed frame** 仍用 `map` 但 TF 实际带前缀。
3. **ros2 param get** 名字写错层级。

### 思考题

1. **绝对话题名** `/scan` 与相对名 `scan` 在带 namespace 的节点内如何解析？
2. Nav2 **bringup** 中 **`use_namespace`** 类参数常用模式？

**答案**：见 [APPENDIX-answers.md](../APPENDIX-answers.md#m03)；Nav2 概览 [M04](第29章：Nav2 栈概览与行为树入门.md)。
