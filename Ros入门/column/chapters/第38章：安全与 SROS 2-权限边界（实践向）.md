# 第38章：安全与 SROS 2-权限边界（实践向）

> 本章目标字数：3000–5000。统一环境见 [ENV.md](../ENV.md)。

## 1 项目背景

### 业务场景

产线 Wi‑Fi 与办公网混接时，**未鉴权 ROS graph** 意味着任意笔记本可发 **`/cmd_vel`**。**SROS 2** 提供基于 **DDS Security** 的认证与加密路径（成熟度随发行版变化）。

### 痛点放大

1. **以为域隔离等于安全**。
2. **加密 CPU 开销**未评估。
3. **密钥轮换**无流程。

**本章目标**：理解 **governance / permissions.xml** 概念；**最小** demo（参考官方）；**纵深防御**：**防火墙 + VPN + 单向网关**。

## 2 项目设计

### 剧本对话

**小胖**：我们 **`ROS_DOMAIN_ID` 都不一样**了，隔壁项目还能控制咱车吗？

**小白**：听说 **SROS 2** 配证书特别劝退，能不能就 **防火墙关 DDS 端口**？

**大师**：**域隔离**只是「不在同一 DDS 发现里偶遇」，**不是身份认证**；能进二层网的人仍可能**伪造参与者**（取决于网络位置）。**防火墙**是必要但不充分：**纵向访问**要 **VPN**、**管理口与业务口分离**。**SROS 2 / DDS Security** 解决的是「**参与者彼此证明身份 + 可选加密**」——工程上常与**运维成本**博弈：证书轮换、时钟漂移、性能开销，都需要**Runbook**。

**技术映射**：**DDS Security** = **Authentication + Access control + Encryption**（子集启用）。

---

**小胖**：那我只加密 **`/cmd_vel`** 行不行？

**大师**：DDS Security 多在 **DomainParticipant** 粒度起作用，细到单个 topic 的策略涉及 **governance / permissions.xml** 编写与**厂商支持度**。现实折中常是：**物理网络隔离 + VPN + 最小镜像**，再评估 **SROS2**；别把「没配证书」当成**已安全**。

**技术映射**：**最小权限** 先在网络层落地，再在 **DDS 层**细化。

---

**小白**：**加密**会让本来就紧张的 CPU 更炸吧？

**大师**：要实测：**典型负载下 CPU headroom**、**尾延迟**。有些场景只对**跨公网链路**加密，**车间内**走**物理隔离**——这是**威胁模型**问题，不是纯技术胜负。

**技术映射**：**性能–安全** trade-off 需数据支撑。

---

**大师**：牢记 **急停硬线** 与 **软件急停** 的分工：后者失败不能阻止机械伤人；**SROS2** 再强也替不了**安全标准**流程。

**技术映射**：**纵深防御** = **机械/电气/网络/软件** 分层叠加。

## 3 项目实战

### 环境准备

与 [ENV.md](../ENV.md) 一致：**Ubuntu 22.04 + ROS 2 Humble**，`source /opt/ros/humble/setup.bash`。

本章额外依赖：**SROS2** 工具链随 **`ros-humble-sros2`**（或文档指定包，`apt search sros2`）；**OpenSSL** 用于 **本地 CA**（`apt install openssl` 通常已有）。

### 分步实现

#### 步骤 1：阅读并准备证书目录（离线）

- **目标**：理解 **DDS-Security** 三件套：**Identity / Permissions / Governance**（概念见本章正文）。
- **命令**：跟随 **[ros2/sros2](https://github.com/ros2/sros2)** 官方教程：**生成本地 CA**、**节点证书**、**权限 XML**。
- **预期输出**：**`keystore/`** 或文档约定目录下 **文件齐全**；**权限位** 合理（**私钥不可组读**）。
- **坑与解法**：**路径写死** → **launch 迁移** 失败 —— 用 **环境变量** 统一 **KEYSTORE**。

#### 步骤 2：配置环境变量启动（分析向）

- **目标**：在无 **全套策略** 时，练习 **「从日志反推缺啥」**。
- **命令**：按教程设置 **`ROS_SECURITY_KEYSTORE`**、**`ROS_SECURITY_ENABLE`** 等；故意 **漏一项**，启动 **`demo_nodes`** 或自有节点。
- **预期输出**：**RMW** 日志出现 **security / authentication** 相关错误码；能 **对照文档** 指出 **缺文件** 或 **权限 XML 不匹配**。
- **坑与解法**：**不同 RMW** 报错格式不同 —— **记录你方所用 Fast-DDS/Cyclone** 样例一条。

#### 步骤 3：`RMW_IMPLEMENTATION` 与 **失败日志**

- **目标**：建立 **「换 RMW = 换安全栈行为」** 的意识（与 [A02](第02章：rmw 与 DDS 实现切换（Fast-DDS-Cyclone 等）.md) 联动）。
- **命令**：**同一 keystore** 下切换 **`RMW_IMPLEMENTATION=rmw_fastrtps_cpp`** / **`rmw_cyclonedds_cpp`**（若均启用 security），对比 **能否互通**。
- **预期输出**：笔记中 **表格**：RMW、**是否连上**、**日志摘录**。
- **坑与解法**：**混 RMW** 未在 **治理文件** 中允许 —— **预期失败**。

#### 步骤 4：运维向最小 checklist（非纯软件）

- **目标**：把 **纵深防御** 落到 **可勾选项**（本章正文「机械/电气/网络/软件」）。
- **检查项**：

  - [ ] **急停**硬线独立于计算机。
  - [ ] **远程调试**仅 VPN。
  - [ ] **生产镜像**关闭 dev 工具。

- **预期输出**：**现场评审** 可 walkthrough。
- **坑与解法**：**只谈 SROS2 不谈急停** → **不安全**。

### 完整代码清单

- **`security/README.md`**（占位）：**证书生成命令**、**目录结构**、**禁止提交私钥** 说明。
- **Launch 片段**：**env-hook** 注入 **`ROS_SECURITY_*`**。
- Git 占位：**`.gitignore` 包含 `*.key`、`keystore/`**。

### 测试验证

- **手工验收**：**故意错误配置** 时，你能 **从日志定位到「缺哪类文件/策略」**（截图或摘录）。
- **联动**：**性能** 对比 **开/关 security** 的 **CPU%**（[M09](第34章：性能与带宽-topic hz-bw、系统剖析入门.md) 一条数据即可）。

## 4 项目总结

### 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 工程价值 | **安全与 SROS 2 / 权限边界（实践向）** 能把隐性经验显式化，便于新人复现与团队协作。 | 需要配套环境、日志与记录表，否则容易停留在概念层。 |
| 可维护性 | 通过标准命名、参数、Launch 或测试约束降低沟通成本。 | 规则一多会增加初期学习曲线。 |
| 可观测性 | 便于用 CLI、日志、bag 或监控指标定位问题。 | 指标若没有业务阈值，仍可能变成“看起来很多但不能决策”。 |
| 扩展性 | 可与前后章节串联，逐步走向真实系统。 | 跨 RMW、跨发行版或跨硬件时需要重新验证边界。 |

### 适用场景

- 团队需要把 **安全与 SROS 2 / 权限边界（实践向）** 从个人经验沉淀为可复用流程。
- 新人、测试与运维需要用同一套命令与术语对齐问题现象。
- 项目进入联调阶段，需要记录参数、话题、日志与验收结果。
- 需要为后续源码阅读、性能优化或生产复盘提供上下文。

### 不适用场景

- 只做一次性演示且不需要交接、回归或复盘的临时脚本。
- 现场约束尚未明确时，不宜把 **安全与 SROS 2 / 权限边界（实践向）** 的示例参数直接当作生产标准。

### 注意事项

- **版本兼容**：所有命令以 Humble 与 [ENV.md](../ENV.md) 为基线，其他发行版需查 `--help` 与官方文档。
- **配置边界**：不要把实验参数直接带入生产；先记录硬件、RMW、QoS、网络与时钟条件。
- **安全边界**：涉及远程调试、容器权限、证书或硬件接口时，先按最小权限原则收敛。

### 常见踩坑经验

1. **只看现象不记录环境**：同一命令在不同 RMW、Domain、QoS 或硬件上结果不同。根因通常是缺少版本与环境快照。
2. **一次改多个变量**：参数、Launch、网络与代码同时变化，导致无法归因。解决方法是每次只改一项并保存日志或 bag。
3. **忽略跨角色交接**：开发能跑通但测试/运维无法复现。根因是缺少最小验收命令、预期输出与失败处理路径。
