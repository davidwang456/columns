## 2 项目设计

### 剧本对话

**小胖**：学完能去考啥证？我妈问起来我总不能说「会 spin」吧？

**小白**：招聘 JD 写「熟悉 ROS2」、面试却问 **DDS 域** 和 **executor**，我们这本读完算啥水平？

**大师**：ROS **没有全球统一考级**。企业更认三件事：**可复现仓库**（固定发行版、README 一键起）、**公开 Issue/PR**（证明协作）、**作品集**能讲清**约束→架构→权衡**（Nav2、硬件、网络各占多少坑）。**特种机器人或功能安全**认证属于另一条线，和「会调 Nav2」不是包含关系——需要**流程与证据链**，本书给你的是**技术底座**。

**技术映射**：**能力证明** = **工件（artifact）** + **可追溯决策**。

---

**小胖**：那我下一步刷 **MoveIt2** 还是 **Gazebo** 还是直接找工作实习？

**大师**：按岗位切：**感知**就深挖 **sensor + TF + 标定 + 数据**；**移动机器人**就 **Nav2 插件 + 系统集成**；**嵌入式**就 **micro-ROS / 刷写 / 实时**（**A07/A09**）。别同时开五条线——用**一条垂直切片**做到「能给别人 clone 跑」比收藏 fifty 个 launch 强。

**技术映射**：**T 型能力**：一竖（可演示项目）+ 一横（本书通识）。

---

**小白**：**社区**里吵架最多的话题是啥？我们要站队吗？

**大师**：常有 **Fast-DDS vs Cyclone**、**Python vs C++**、**Launch XML vs Python**——多数是**场景变量不同**。成熟团队输出的是：**我们在何种网络/硬件/团队约束下选了 A，代价是 X**。站队不如**写清决策记录（ADR）**。

**技术映射**：**Discourse + REP** 是「标准战场」，读结论前的**假设**。

---

**大师**：若真想「背书」，可考虑 **厂商培训 / 高校慕课证书** 作为辅助，但面试官仍会用白板追问 **QoS 与 TF**——那一关本书已经反复操练了（**B05/B09/M02**）。

## 3 项目实战（结业项目：按 B04 结构落地）

> 本章无「实验室代码」，而是把 **环境准备 → 分步 → 清单 → 验收** 固化为 **你个人的 30 天路线**；与 [B04](第16章：话题与消息-发布订阅第一印象.md) 同级结构，便于写入 **简历/述职**。

### 环境准备

- **机器**：**Ubuntu 22.04 + ROS 2 Humble**（[ENV.md](../ENV.md)）。
- **账号**：**GitHub/GitLab** 可建公开或**可见范围**仓库；**CI 分钟数**自行评估。
- **时间**：建议 **≥20h** 有效编码（**含**联调与文档）。

### 分步实现

#### 步骤 1：选题 — 一条垂直切片

- **目标**：从下列 **选一**（勿多选）：
  1. **移动底盘**：**Nav2** 最小仿真 + **一张自定义地图** + **记录 bag**；或
  2. **机械臂/抓手**：**MoveIt 2** hello（若硬件无，仅仿真）；或
  3. **工具向**：为本仓库 **38 章** 做 **`markdown` 校验脚本** + **GitHub Action**（非 ROS 但可证明工程化）。
- **预期输出**：**`README.md` 一句话电梯陈述** + **约束列表**（传感器、算力、网络）。
- **坑与解法**：范围蔓延 —— **先跑通再美化**。

#### 步骤 2：仓库骨架

- **目标**：**README** 具备 **「一键 reproduce」** 段落（与 [A08](第08章：构建系统深入-ament、依赖与私有仓库.md) 社会工程对齐）。
- **命令/结构**：

```text
your-project/
  README.md          # 必须：环境、安装、运行、测试
  .github/workflows/ci.yaml
  ros2_ws/src/...    # 若选 ROS
```

- **预期输出**：**clone → install → run** 三步可被**陌生人**执行（你自己 **48h 后 cold run** 测一次）。
- **坑与解法**：**漏写 `source install/setup.bash`** —— README 显式写出。

#### 步骤 3：最小 CI

- **目标**：**`colcon build && colcon test`**（ROS）或 **`pytest`/`shellcheck`**（工具向）。
- **命令（GitHub Actions 示意）**：

```yaml
# .github/workflows/ci.yml — 骨架，按需补 ROS 容器
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-22.04
    steps:
      - uses: actions/checkout@v4
      - run: |
          sudo apt update
          sudo apt install -y ros-humble-desktop python3-colcon-common-extensions
          echo "在此 source 并 colcon build"
```

- **预期输出**：**CI 绿**；失败时能 **log 定位**。
- **坑与解法**：**无测试** → 至少 **`launch_test` smoke** 或 **`ros2 doctor`**（[A10](第10章：SRE-故障演练、SLA、告警与回滚.md)）。

#### 步骤 4：复盘文章 / ADR

- **目标**：**一篇公开 blog** 或 **一条 ADR**：**约束 → 取舍 → 未解决问题**（剧本）。
- **预期输出**：**URL** 或 **`docs/adr/0001-....md`**。
- **坑与解法**：只贴代码无叙事 —— **招聘方无法评估思维**。

### 完整代码清单

- **你的仓库 URL**（完成后填入简历）。
- **依赖**：固定 **`ros-humble-*` 包列表** 或 **Dockerfile digest**。

### 测试验证

- **结业标准（自检表）**：
  - [ ] **他人可读 README 复现**；
  - [ ] **CI 绿**；
  - [ ] **至少 1 条 issue/PR**（协作痕迹）或 **导师 review 记录**；
  - [ ] 能 **白板讲清**：**DDS 域**（[M01](第26章：DDS 发现、域（Domain）与跨机通信.md)）+ **QoS**（[M02](第27章：QoS 深度-history、deadline、durability.md)）+ **Executor**（[A01](第01章：rcl-rclcpp 执行模型与源码导读.md)）三者之一 **10 分钟**。
