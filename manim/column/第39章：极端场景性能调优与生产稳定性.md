# 第39章：极端场景性能调优与生产稳定性

---

## 1. 项目背景

某在线教育平台已经用 Manim 生产了 200+ 集课程动画，累计渲染时长超过 300 小时。随着生产规模的扩大，之前被忽略的"极端场景"问题集中爆发：

1. **长视频渲染失败**：一段 60 分钟的"高等数学总复习"动画在第 47 分钟时 FFmpeg 内存溢出，前 46 分钟的渲染成果全部丢失，需要从头重跑。运维需要"断点续渲"能力。
2. **字体和 LaTeX 缓存爆炸**：`build/tex_cache/` 目录已膨胀到 8GB，包含 3000+ 个 LaTeX 编译产物。其中 60% 是历史迭代版本留下的垃圾缓存。
3. **渲染机差异**：开发者 macOS 上能正常显示的字体，在 Linux CI 容器中变成方块；开发机 16GB 内存下能渲染的场景，在 8GB 的 CI runner 上 OOM（Out of Memory）。
4. **缺乏监控**：运维不知道哪一集渲染最慢、哪个公式编译最耗时、哪台机器的资源利用率最高——一切全凭"感觉"。

这些痛点的本质是将 Manim 从"开发者工具"升级为"**生产系统**"所需的工程化能力。本章要系统讲解长视频断点续渲、缓存治理、容器化渲染和日志监控四大主题。

## 2. 剧本式交锋对话

> **场景**：运维老黄正在检查昨晚的 CI 渲染日志——60 分钟的长视频在第 47 分钟崩溃了，日志最后一行是 `Killed`。

**小胖**（撕开一根牛肉干）：

"黄哥，你这 `Killed` 是啥意思？被暗杀了？"

**老黄**（头疼）：

"OOM Killer——Linux 内核发现 Manim 占用了太多内存（超过 CI runner 的 8GB），直接把进程杀了。47 分钟的渲染成果，一张帧都没留下来。现在又要从头再跑 47 分钟。"

**小白**（打开 `SceneFileWriter` 源码）：

"问题出在 Manim 默认的渲染模式——它把所有帧的 PNG 缓存在内存里，等到 `finish()` 才一次性喂给 FFmpeg。47 分钟 × 30fps = 84600 帧 × 每帧 8MB（1080p）≈ 677GB！8GB 内存当然是奢望。

解决方案是 `disable_caching = true`——让每帧渲染完直接写入磁盘，不堆在内存里。但这样磁盘 IO 会成为瓶颈。折中方案是**分段渲染**——把 60 分钟视频切成 12 个 5 分钟的段，分别渲染为 `part01.mp4` 到 `part12.mp4`，最后用 FFmpeg concat 拼接。"

**大师**（补充）：

"分段渲染还有一个好处——**断点续渲**。如果第 7 段渲染失败，只需要重跑第 7 段，不需要从头再来。实现策略是：每个段渲染前检查对应的 `partXX.mp4` 是否已存在且完整，存在就跳过。"

> **技术映射**：`ffmpeg -f concat -safe 0 -i filelist.txt -c copy output.mp4` 将多个 MP4 段无损拼接为完整视频。`filelist.txt` 格式：`file 'part01.mp4'\nfile 'part02.mp4'...`。

**小胖**（牛肉干嚼得嘎嘣响）：

"那 LaTeX 缓存 8GB 是怎么回事？清理一下不就完了？"

**小白**：

"问题是"清哪些"。全删了下次渲染 3000 个公式全部重编译——耗时 40 分钟起。正确的缓存治理策略是：

1. **保留最新版本**：每个公式的最新编译结果。
2. **清理孤立缓存**：已经被改动过的旧公式版本（哈希值变了，旧缓存永远不再被引用）。
3. **监控缓存增长**：在 build 脚本中加一句 `du -sh build/tex_cache/`，超过 2GB 时告警。

Manim 的 LaTeX 缓存基于公式内容的 MD5 哈希命名目录。你每次修改公式字符串——哪怕加一个空格——都会创建新目录。"

#### 步骤一：分段渲染 + 断点续渲

**步骤目标**：将长视频切分为多段渲染，支持失败重试和跳过已完成段。

```python
# scripts/chunked_render.py
"""分段渲染脚本——支持断点续渲"""
import subprocess, sys, time
from pathlib import Path

SCENE_FILE = "scenes/long_video.py"
SCENE_NAME = "LongScene"
TOTAL_DURATION = 3600  # 60 分钟 = 3600 秒
CHUNK_SIZE = 300       # 每段 5 分钟 = 300 秒
QUALITY = "h"
OUTPUT_DIR = Path("build/chunks")
OUTPUT_DIR.mkdir(parents=True, exist_ok=True)

chunks = list(range(0, TOTAL_DURATION, CHUNK_SIZE))

def render_chunk(chunk_start, chunk_index):
    """渲染一个时间段"""
    chunk_end = min(chunk_start + CHUNK_SIZE, TOTAL_DURATION)
    output_file = OUTPUT_DIR / f"part_{chunk_index:03d}.mp4"

    # 断点续渲：如果文件已存在且完整，跳过
    if output_file.exists() and output_file.stat().st_size > 100 * 1024:
        print(f"  ⏭ 跳过已完成的段 {chunk_index} ({output_file.stat().st_size // 1024}KB)")
        return True

    # 策略：修改场景代码中的 start_time/end_time 参数
    # 更稳健的做法：渲染完整视频后用 ffmpeg 裁剪出各段
    # 这里展示 ffmpeg 裁剪方案

    # 假设完整视频已渲染为 full.mp4
    full_video = OUTPUT_DIR / "full.mp4"
    cmd = [
        "ffmpeg", "-y",
        "-ss", str(chunk_start),
        "-i", str(full_video),
        "-t", str(CHUNK_SIZE),
        "-c", "copy",  # 无损裁剪（关键帧对齐）
        str(output_file),
    ]
    result = subprocess.run(cmd, capture_output=True, text=True)
    if result.returncode == 0:
        print(f"  ✅ 段 {chunk_index:03d}: {chunk_start}s-{chunk_end}s, {output_file.stat().st_size // 1024}KB")
        return True
    else:
        print(f"  ❌ 段 {chunk_index:03d} 失败: {result.stderr[-200:]}")
        return False

# 执行分段
success_count = 0
for i, start_time in enumerate(chunks):
    print(f"[{i+1}/{len(chunks)}] 时间段 {start_time}s-{min(start_time+CHUNK_SIZE, TOTAL_DURATION)}s")
    if render_chunk(start_time, i):
        success_count += 1

print(f"\n完成: {success_count}/{len(chunks)} 段成功")

# 拼接所有段
if success_count == len(chunks):
    filelist = OUTPUT_DIR / "filelist.txt"
    with open(filelist, "w") as f:
        for i in range(len(chunks)):
            f.write(f"file 'part_{i:03d}.mp4'\n")

    output_final = Path("build/media/long_video/1080p60/LongScene.mp4")
    output_final.parent.mkdir(parents=True, exist_ok=True)
    subprocess.run([
        "ffmpeg", "-y", "-f", "concat", "-safe", "0",
        "-i", str(filelist), "-c", "copy", str(output_final),
    ])
    print(f"✅ 拼接完成: {output_final}")
```

#### 步骤二：缓存治理

**步骤目标**：编写缓存清理脚本，保留最新版本，清理孤立缓存。

```python
# scripts/cache_cleanup.py
"""LaTeX 缓存清理——只保留每个公式的最新版本"""
import shutil
from pathlib import Path
from collections import defaultdict

TEX_CACHE = Path("build/tex_cache")

def analyze_cache():
    """分析缓存中每个公式的版本"""
    formula_versions = defaultdict(list)
    for item in TEX_CACHE.iterdir():
        if item.is_dir():
            # 目录名格式: <hash>_<timestamp>
            parts = item.name.split("_")
            if len(parts) >= 2:
                formula_hash = parts[0]
                formula_versions[formula_hash].append(item)

    # 保留每个公式的最新版本
    kept_size = 0
    removed_size = 0
    for h, dirs in formula_versions.items():
        dirs.sort(key=lambda d: d.stat().st_mtime, reverse=True)
        for old_dir in dirs[1:]:  # 保留第一个（最新），删除其余
            old_size = sum(f.stat().st_size for f in old_dir.rglob("*"))
            shutil.rmtree(old_dir)
            removed_size += old_size
        kept_size += sum(f.stat().st_size for f in dirs[0].rglob("*"))

    print(f"保留: {kept_size // 1024 // 1024}MB")
    print(f"删除: {removed_size // 1024 // 1024}MB")

if __name__ == "__main__":
    analyze_cache()
```

```python
# scripts/render_monitor.py
"""渲染资源监控——记录 CPU/内存峰值"""
import subprocess, time, psutil, json
from datetime import datetime

def monitor_render(cmd, output_report="build/render_stats.json"):
    process = subprocess.Popen(cmd)
    cpu_peaks = []
    mem_peaks = []

    try:
        while process.poll() is None:
            if psutil.pid_exists(process.pid):
                proc = psutil.Process(process.pid)
                cpu_peaks.append(proc.cpu_percent(interval=0.1))
                mem_peaks.append(proc.memory_info().rss / 1024 / 1024)  # MB
            time.sleep(0.5)
    except psutil.NoSuchProcess:
        pass

    stats = {
        "timestamp": datetime.now().isoformat(),
        "exit_code": process.returncode,
        "cpu_max_pct": max(cpu_peaks) if cpu_peaks else 0,
        "mem_max_mb": max(mem_peaks) if mem_peaks else 0,
        "wall_time_s": time.time() - (process._start_time if hasattr(process, '_start_time') else 0),
    }

    Path(output_report).parent.mkdir(exist_ok=True)
    Path(output_report).write_text(json.dumps(stats, indent=2))
    print(f"📊 渲染统计: {json.dumps(stats, indent=2)}")

if __name__ == "__main__":
    cmd = ["python", "-m", "manim", "-qh", "scenes/video.py", "MainScene"]
    monitor_render(cmd)
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 断点续渲 | 失败只重跑失败段，节省 90% 时间 | ffmpeg 无损裁剪需要关键帧对齐 |
| 缓存治理 | 清理后缓存体积可降 60-80% | 清理策略过于激进可能误删仍引用的旧缓存 |
| 容器化 | Docker 消除平台差异，CI 和本地输出一致 | Docker 镜像体积大（含 TeX Live 完整版 ~3GB） |
| 资源监控 | psutil 实时记录 CPU/内存峰值 | 监控本身消耗约 1-3% CPU |
| 生产稳定性 | 组合四招后渲染成功率 >99% | 基础搭建耗时 1-2 天 |

### 适用场景

| 场景 | 说明 |
|------|------|
| 长视频制作 | 10 分钟以上的课程动画 |
| CI 自动化 | GitHub Actions / GitLab CI 中的渲染 |
| 多平台验证 | 确保 Windows/macOS/Linux 输出一致 |
| 大规模生产 | 200+ 集的批量渲染 |
| 云计算渲染 | AWS/GCP 上的弹性渲染节点 |

**不适用场景**：单次渲染的短动画（<2 分钟，直接 `manim -pqh scene.py` 更简单）、开发阶段的快速迭代（`.py` 文件频繁改动，缓存变动快）。

### 注意事项

1. **ffmpeg 无损裁剪对齐**：`-c copy` 只能在关键帧处精确切割。如果起始时间点不是关键帧，画面会有一个短暂的花屏。解决：`-ss` 放在 `-i` 之前进行精确 seek（但会慢一些）。
2. **容器镜像的字体**：Dockerfile 必须安装中文字体（`fonts-noto-cjk`），否则中文全方块。
3. **psutil 的跨平台差异**：Windows 上的 `Process.memory_info().rss` 与 Linux 上的含义不同，跨平台比较时注意。

### 常见踩坑经验

**故障一：分段渲染后拼接的视频在段衔接处有黑帧**

根因：不同段的关键帧不对齐。段 A 的最后一帧和段 B 的第一帧之间缺少过渡。

解决：使用 `ffmpeg -ss START -to END -i input.mp4 -c:v libx264 -crf 18 output.mp4` 强制重编码（非 copy 模式）确保每帧精确。

**故障二：Docker 容器内 LaTeX 编译极慢**

根因：Docker 的 AUFS/overlay2 文件系统对大量小文件 IO 不够优化。

解决：在 Dockerfile 中将 `TEXMFVAR` 环境变量指向 `/tmp`（tmpfs 内存文件系统）。

**故障三：`psutil` 监控导致渲染变慢 10-20%**

根因：每 0.5 秒采样一次 `cpu_percent` 会执行上下文切换。

解决：增大采样间隔到 2-5 秒，或只在渲染开始和结束时各采样一次。

**故障四：容器间 LaTeX 缓存不共享导致重复编译**

根因：每次 `docker run` 创建新容器，`build/tex_cache/` 是空的。

解决：用 Docker volume 挂载宿主机目录：`-v /host/tex_cache:/workspace/build/tex_cache`。

**故障五：缓存清理误删了上游依赖的中间产物**

根因：`part_03.mp4` 被删后，拼接脚本仍然引用它。

解决：在拼接前做 `filelist.txt` 的有效性检查：`for f in parts: assert f.exists()`。

### 思考题

1. 扩展 `chunked_render.py`，增加一个"自适应分段"功能——对每段渲染进行耗时统计，如果某段耗时超过前几段的 2 倍，自动将该段拆分为 2 段重试。提示：记录每段的渲染耗时到 JSON，用 `np.std` 检测异常值。

2. 设计一套"金丝雀渲染"机制：每天凌晨 3 点 CI 自动渲染 3 个"金丝雀场景"（覆盖文字、公式、3D 曲面），如果任一失败则凌晨 3:05 发送告警到运维群。提示：用 `schedule` 或 `cron` 定时触发 `build_v2.py --scene Canary1 --scene Canary2 --scene Canary3`。

3. 将 `Dockerfile.manim` 改造为多阶段构建：第一阶段安装 TeX Live 完整版（约 3GB），第二阶段只复制必要的运行时文件（约 800MB）。用 `COPY --from=builder` 优化最终镜像体积。
