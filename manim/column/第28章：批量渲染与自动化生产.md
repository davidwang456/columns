# 第28章：批量渲染与自动化生产

---

## 1. 项目背景

某教育平台的 Manim 课程项目已经积累了 30+ 个场景，分布在 8 个章节目录中。每当教研团队修改一个公式或一个颜色，就需要重新渲染受影响的所有场景。

第 12 章已经介绍了基础的 `build.py` 单文件渲染脚本。但随着项目规模膨胀（30 个场景、8 个目录、3 种输出质量），单文件脚本的局限性暴露了：

1. **场景依赖关系复杂**：第 5 章的"排序动画"依赖第 4 章的 `make_node` 工厂函数。如果不按顺序渲染，依赖关系可能断裂。
2. **渲染失败无声**：CI 中批量渲染 30 个场景，第 17 个场景的 LaTeX 编译失败了，但 CI 继续渲染了后面 13 个。没人注意到有一个失败了，直到教研老师查看成片才发现"第 17 集是空的"。
3. **增量渲染缺失**：只改了一个文件中的一行注释，却重新渲染了全部 30 个场景——浪费了 45 分钟的 CI 时间。
4. **多平台差异**：开发者在 Mac 上能渲染的场景，Windows CI 上却报字体找不到——因为 CI 容器的字体配置和桌面环境不同。

这些痛点的本质是 Manim 的**生产化工程**。本章要在第 12 章的基础上，建立场景依赖管理、增量渲染、失败监控和多环境适配的完整生产流水线。

## 2. 剧本式交锋对话

> **场景**：运维小马盯着 CI 面板上的 30 个渲染任务。有 3 个红的，但他不知道是哪 3 个。

**小胖**（捧着一碗关东煮）：

"马哥你这 CI 像俄罗斯轮盘赌——30 个任务有 3 个挂了，但挂了的是哪三个？不影响最终产物的要不要重新跑？全部重跑的话又要等 45 分钟，CI 账单在燃烧啊。"

**小马**（指着 CI 日志）：

"你看这个日志——30 个场景混在一起。Manim 报错在第 300 行的中间，我需要从 2000 行日志里用肉眼找 `[FAIL]` 字样。每次 CI 跑完，我要花 10 分钟人工排查失败。"

**小白**（打开了一个 Well-structured 的 `build_v2.py`）：

"你需要的是**结构化渲染报告**。改造 `build.py`，让它输出 JSON 格式的渲染结果——每个场景一个条目，含成功/失败、渲染耗时、产物大小、错误信息。

```json
{
  "total": 30, "passed": 27, "failed": 3,
  "scenes": [
    {"name": "HelloManim", "status": "ok", "time_s": 12.3, "output_mb": 4.2},
    {"name": "Dijkstra",   "status": "fail", "error": "LaTeX Error: ..."}
  ]
}
```

"解析这份 JSON，一目了然。CI 可以基于它做决策——'失败超过 10% 则阻止发布'。"

**大师**（补充）：

"除了监控，还有三个生产化要点：

**第一，增量渲染**——用 Git diff 找出哪些 `.py` 文件改动过，只渲染受影响的场景。可以用 `git diff --name-only HEAD~1` 找出变动的文件，然后只渲染对应的场景类。

简易实现：每个场景在文件头注释中声明它所依赖的模块，构建脚本对比变动文件和依赖声明，只渲染受影响者。

**第二，依赖排序**——场景之间如果共享组件（如第 25 章的 `components/` 包），需要确保依赖链完整。可以在 `manim.cfg` 或独立 `deps.yaml` 中声明依赖关系。

**第三，平台适配层**——在 `build.py` 中检测当前平台（`platform.system()`）和环境变量（`CI=true`），自动切换渲染参数。CI 环境不用 `-p` 预览，Windows 需要设置字体 PATH，macOS 需要确保 Homebrew 的 FFmpeg 在 PATH 中。"

> **技术映射**：`platform.system()` 返回 `'Windows'`/`'Darwin'`/`'Linux'`。`os.environ.get('CI', 'false')` 检测 CI 环境。

**小胖**（关东煮里的鱼丸掉桌上了）：

"那 100 个场景的大项目呢？一个一个串行渲染是不是要跑到明年？"

**大师**：

"并行渲染是中大型项目的必选项。Manim 本身不支持多场景并行——但 `build.py` 可以用 `ProcessPoolExecutor` 或 `concurrent.futures` 同时启动多个 manim 进程。

关键限制：每个 manim 进程会创建独立的 `media/` 子目录，不会互相覆盖。但 LaTeX 缓存是共享的——两个进程同时编译同一个公式可能产生竞态。解决方案是每个进程指定独立的 `--media_dir`。"

**小胖**（蹭蹭鼻子）：

"最后问个实际的——我渲染完 30 集，拿到了 30 个 MP4。但怎么知道每一个 MP4 都是完整可播放的？万一第 17 集渲染到一半 FFmpeg 崩了，文件还在但是坏的。"

**大师**：

"产物验证是生产流水线的最后一道防线。简单的验证策略：

1. **文件存在性**：`path.exists() and path.stat().st_size > 100KB`（非空文件）
2. **时长检查**：用 `ffprobe -v error -show_entries format=duration` 提取时长，确认大于 1 秒。
3. **元数据校验**：`ffprobe` 确认分辨率、帧率、编码格式与期望一致。

把这些检查写成 `validate_artifacts()` 函数，放在 `build.py` 的渲染循环之后——生成报告前先验产物。"

> **技术映射**：`ffprobe -v error -show_entries format=duration -of default=noprint_wrappers=1:nokey=1 video.mp4` 返回 MP4 时长（秒）。

#### 步骤一：结构化渲染报告

**步骤目标**：改造 `build.py`，输出 JSON 格式的渲染报告。

```python
# build_v2.py —— 结构化渲染报告版
import subprocess, sys, time, json, re
from pathlib import Path
from datetime import datetime

SCENES_DIR = Path("scenes")
OUTPUT_DIR = Path("build/media")

def discover_scenes(directory):
    scenes = []
    for py_file in sorted(directory.glob("*.py")):
        content = py_file.read_text(encoding="utf-8")
        found = re.findall(r"class\s+(\w+)\(\w*Scene\w*\)", content)
        for name in found:
            scenes.append({"file": str(py_file), "name": name})
    return scenes

def render_scene(file_path, scene_name, quality="m"):
    """渲染单个场景，返回结果字典"""
    cmd = [
        sys.executable, "-m", "manim",
        "-q", quality,
        "--media_dir", str(OUTPUT_DIR),
        file_path, scene_name,
    ]
    start = time.time()
    result = subprocess.run(cmd, capture_output=True, text=True)
    elapsed = time.time() - start

    # 查找产物 MP4 文件
    mp4_files = list(OUTPUT_DIR.rglob(f"**/{scene_name}.mp4"))
    output_mb = round(mp4_files[0].stat().st_size / 1024 / 1024, 2) if mp4_files else 0

    return {
        "scene": scene_name,
        "file": file_path,
        "status": "ok" if result.returncode == 0 else "fail",
        "time_s": round(elapsed, 1),
        "output_mb": output_mb,
        "error": result.stderr[-300:] if result.returncode != 0 else None,
    }

def main():
    quality = sys.argv[1] if len(sys.argv) > 1 else "m"
    scenes = discover_scenes(SCENES_DIR)
    print(f"[{datetime.now().strftime('%H:%M:%S')}] 发现 {len(scenes)} 个场景，质量: -q{quality}")

    results = []
    total_start = time.time()
    for i, scene in enumerate(scenes):
        print(f"  [{i+1}/{len(scenes)}] {scene['name']} ...", end=" ", flush=True)
        result = render_scene(scene["file"], scene["name"], quality)
        results.append(result)
        print(f"{result['status'].upper()} ({result['time_s']}s)")

    total_elapsed = time.time() - total_start
    passed = sum(1 for r in results if r["status"] == "ok")
    failed = len(results) - passed

    report = {
        "timestamp": datetime.now().isoformat(),
        "quality": quality,
        "total": len(results), "passed": passed, "failed": failed,
        "total_time_s": round(total_elapsed, 1),
        "scenes": results,
    }

    # 写入 JSON 报告
    report_path = Path("build/report.json")
    report_path.parent.mkdir(exist_ok=True)
    report_path.write_text(json.dumps(report, indent=2, ensure_ascii=False), encoding="utf-8")

    print(f"\n报告: {report_path}")
    print(f"结果: {passed}/{len(results)} 成功, {failed} 失败, 总耗时 {total_elapsed:.1f}s")

    sys.exit(0 if failed == 0 else 1)

if __name__ == "__main__":
    main()
```

#### 步骤二：增量渲染

**步骤目标**：只渲染 Git 变更影响的场景。

```python
# build_incremental.py —— 增量渲染版
import subprocess, sys
from pathlib import Path

def get_changed_files():
    """获取 Git 变动的 Python 文件"""
    result = subprocess.run(
        ["git", "diff", "--name-only", "HEAD~1"],
        capture_output=True, text=True,
    )
    return [f for f in result.stdout.splitlines() if f.endswith(".py")]

def find_affected_scenes(changed_files, scenes_dir):
    """找出受影响的场景——简单策略：变了哪个场景文件就渲染哪个"""
    affected = set()
    for f in changed_files:
        f_path = Path(f)
        if f_path.parent.name == "scenes" or "scenes" in str(f_path):
            affected.add(f_path.stem)
    return list(affected)

if __name__ == "__main__":
    changed = get_changed_files()
    print(f"变更文件: {changed}")
    affected = find_affected_scenes(changed, Path("scenes"))
    print(f"受影响场景: {affected}")

    if not affected:
        print("无受影响场景，跳过渲染")
        sys.exit(0)

    # 调用 build_v2.py 渲染受影响的场景
    for scene_file in affected:
        subprocess.run([sys.executable, "build_v2.py", scene_file])
```

#### 步骤三：并行渲染

**步骤目标**：利用多进程并行渲染加速生产。

```python
# build_parallel.py —— 并行渲染版
import subprocess, sys, time
from concurrent.futures import ProcessPoolExecutor, as_completed
from pathlib import Path

def render_one(scene_info):
    """单个场景渲染（独立进程）"""
    file_path, scene_name, quality = scene_info
    cmd = [
        sys.executable, "-m", "manim",
        "-q", quality,
        "--media_dir", f"build/media/worker_{scene_name}",
        file_path, scene_name,
    ]
    start = time.time()
    result = subprocess.run(cmd, capture_output=True, text=True)
    elapsed = time.time() - start
    return {
        "scene": scene_name,
        "status": "ok" if result.returncode == 0 else "fail",
        "time_s": elapsed,
    }

def main():
    quality = sys.argv[1] if len(sys.argv) > 1 else "m"
    workers = int(sys.argv[2]) if len(sys.argv) > 2 else 4

    # 发现场景
    scenes = []
    for py_file in sorted(Path("scenes").glob("*.py")):
        import re
        content = py_file.read_text(encoding="utf-8")
        for name in re.findall(r"class\s+(\w+)\(\w*Scene\w*\)", content):
            scenes.append((str(py_file), name, quality))

    print(f"并行渲染 {len(scenes)} 个场景, workers={workers}")

    start = time.time()
    with ProcessPoolExecutor(max_workers=workers) as executor:
        futures = {executor.submit(render_one, s): s for s in scenes}
        for i, future in enumerate(as_completed(futures)):
            result = future.result()
            print(f"  [{i+1}/{len(scenes)}] {result['scene']}: {result['status'].upper()} ({result['time_s']:.1f}s)")

    elapsed = time.time() - start
    print(f"并行渲染完成, 耗时 {elapsed:.1f}s")

if __name__ == "__main__":
    main()
```

**运行命令**：

```bash
python build_v2.py m                  # 批量渲染 720p + JSON 报告
python build_incremental.py           # 增量渲染
python build_parallel.py m 8          # 8 进程并行渲染
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 结构化报告 | JSON 格式可被 CI 系统解析做决策 | 需要手动编写解析逻辑 |
| 增量渲染 | 只渲染变更场景，节省 CI 时间 | Git diff 判断较粗糙（改注释也会触发） |
| 并行渲染 | 多进程利用多核 CPU | 进程间 LaTeX 缓存可能冲突 |
| 平台适配 | `platform.system()` 自动切换参数 | 容器环境的平台检测可能不准确 |
| 可扩展性 | build_v2 可扩展为生产平台核心 | 脚本越来越多，维护负担增加 |

### 适用场景

| 场景 | 说明 |
|------|------|
| 课程批量生产 | 30+ 场景的系列课程 |
| CI/CD 流水线 | GitHub Actions 自动渲染 + 发布 |
| 多平台验证 | 确保 Windows/macOS/Linux 渲染一致 |
| 自动化测试 | 每 PR 渲染受影响的场景做视觉回归 |
| 云端渲染 | 将渲染任务分发到多台机器 |

**不适用场景**：只维护 2-3 个场景的小项目（纯 `manim -pql scene.py` 足够）、实时预览和调试阶段（需要 `-p` 参数和人工观察）。

### 注意事项

1. **并行渲染的 `--media_dir` 必须隔离**：每个 worker 使用独立的 `media_dir` 子目录，避免多个 manim 进程同时写同一个文件。
2. **增量渲染的依赖分析不够精确**：改 `components/` 中的组件会应影响很多场景，但 Git diff 只看到 `components/cards.py` 一个文件。需要维护依赖映射表。
3. **CI 中的字体和 LaTeX**：CI 容器的字体和 LaTeX 宏包通常不完整。Dockerfile 中需要安装中文字体和 TeX Live 完整版。

### 常见踩坑经验

**故障一：并行渲染后 `media/` 中出现空目录**

根因：多个 manim 进程的 `media_dir` 子目录在进程启动时创建但某些进程未产出文件。

解决：在 `build.py` 末尾清理空目录：`for d in sorted(OUTPUT_DIR.rglob("*"), reverse=True): d.rmdir() if d.is_dir() and not list(d.iterdir()) else None`。

**故障二：增量渲染的 Git diff 在 CI 中返回空**

根因：CI 的 shallow clone 只有最近 1 次提交，`HEAD~1` 不存在。

解决：在 CI 配置中设置 `fetch-depth: 0` 获取完整历史，或使用 `git diff --name-only ${{ github.event.before }} ${{ github.sha }}`。

**故障三：JSON 报告中有中文乱码**

根因：`json.dumps` 默认 ASCII 编码，`ensure_ascii=True` 会将中文转义为 `\uxxxx`。

解决：`json.dumps(report, indent=2, ensure_ascii=False)` 保留原始中文字符，文件编码设为 UTF-8。

### 思考题

1. 扩展 `build_v2.py`，增加 **"重试失败场景"** 功能：对 `status=fail` 的场景最多重试 2 次（每次间隔 5 秒）。如果 3 次都失败才标记为最终失败。提示：用 `while try_count < 3` 循环包裹 `render_scene` 调用。

2. 设计一个 **依赖图**（`deps.yaml`）机制：每个场景声明它所依赖的模块（如 `depends_on: [components/cards.py, constants/theme.py]`），增量渲染脚本通过对比 Git diff 和依赖图精确确定受影响场景。
