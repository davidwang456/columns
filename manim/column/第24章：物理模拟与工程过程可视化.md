# 第24章：物理模拟与工程过程可视化

---

## 1. 项目背景

某机械工程系的教授正在用 Manim 制作"振动力学"课程动画。主题是"双摆运动"——一个经典的混沌系统。双摆的运动轨迹非常复杂，不能像简谐运动那样用解析式直接描述，需要用数值方法（龙格-库塔法或欧拉法）求解微分方程，然后把解出来的轨迹用 Manim 渲染。

教授遇到的技术挑战有三层：

1. **数值计算与动画分离**：不能把 `scipy.integrate.solve_ivp` 放在 Manim 的渲染循环里——那会导致每帧都重新求解微分方程，性能灾难。正确做法是**预计算全部轨迹，再给 Manim 播放**。
2. **Manim 的帧率与仿真步长不一致**：微分方程可能需要 dt=0.001 的精度，但动画只需要 30fps（dt=0.033）。需要做数据重采样或插值。
3. **工程结构复杂**：双摆有 2 个连杆、2 个关节、2 条轨迹线、2 个角度标注。如果用第 4 章的"手动管理每个 Arrow 的位置"，代码量爆炸。

本章要系统讲解：用 Python 做数值计算 → 将结果序列传递给 Manim → 用 Updater/always_redraw 渲染物理运动，以及如何处理仿真步长与渲染帧率的不匹配。

## 2. 剧本式交锋对话

> **场景**：教授的屏幕上左边是 scipy 的计算结果图，右边是 Manim 的空白画布。

**小胖**（啃着一根玉米棒）：

"教授，您这左边 matplotlib 画的轨迹挺好看，右边 Manim 怎么还是空白？您是不是以为 Manim 能直接运行 scipy 的微分方程？"

**教授**（无奈）：

"我确实试了——把 `solve_ivp` 放到 `always_redraw` 里。结果渲染一帧要 30 秒。方程求一次解 0.1 秒，但 30fps 意味着每秒求 30 次，机器直接冒烟。"

**小白**（打开一个预计算的脚本）：

"科学计算的黄金法则是——**计算和渲染分离**。你先把 0 到 10 秒的轨迹完整计算出来，存入 Python list 或者 numpy array，然后把这份'关键帧序列'喂给 Manim。Manim 只负责在时间轴上行走到对应的数据点并渲染画面。

```python
# 计算阶段（离线）
t_span, dt = (0, 10), 0.01
sol = solve_ivp(equations, t_span, y0, t_eval=np.arange(0, 10, dt))
angle1 = sol.y[0]  # 摆 1 的角度序列
angle2 = sol.y[1]  # 摆 2 的角度序列

# 渲染阶段（Manim）
time_tracker = ValueTracker(0)
def get_positions():
    idx = int(time_tracker.get_value() / dt)
    return angle1[idx], angle2[idx]
```

"这样，Manim 的每一帧只需要一次数组索引——比重新求微分方程快 10000 倍。"

**大师**（补充）：

"更进一步——如果步长 dt=0.001（10000 个数据点），而 Manim 只需要 30fps（300 个关键帧），你可以做**降采样**：每 33 个数据点取一个。或者保留原始精度，让轨迹绘制用所有数据点（更平滑的曲线），而摆杆只更新到当前帧对应的位置。"

> **技术映射**：`numpy.arange(0, T, dt)` 生成均匀步长的仿真时间点。`sol.y[idx]` 按时间索引取出对应状态，复杂度 O(1)。

**小胖**（玉米啃完了）：

"那双摆的结构怎么画？两根杆子、两个关节——这比画一个箭头复杂多了。"

**大师**（在白板上画了双摆的几何关系）：

"双摆的 Manim 表达本质是几何关系翻译：

- 第一根杆子：从固定点到关节 1，长度 L1，角度 θ1
- 第一关节：在杆 1 的末端
- 第二根杆子：从关节 1 到关节 2，长度 L2，角度 θ2
- 轨迹：关节 2 的路径集合

用 `always_redraw` 绑定四样东西——两条杆（Line）、两个关节（Dot）、两条轨迹（VMobject）。都不需要复杂的 Animation 逻辑。"

#### 补充对话：处理仿真步长与渲染帧率的不匹配

**小胖**（玉米啃得只剩棒子了）：

"教授，您刚才说计算和渲染要分离。但我有个实际困扰——仿真用的 dt=0.01（1000fps），动画是 30fps。这中间差 33 倍！如果把 1000 个数据点直接映射到 30 帧，跳跃感会不会很强？"

**小白**：

"当然会。处理方案有三种：

1. **取整索引法**：`data[int(timer / dt)]`。最简单，但在 30fps 下会有可见的"阶梯感"。
2. **线性插值法**：`np.interp(timer, t_array, data)`。在 30fps 下平滑但无法还原高频细节（已被采样定理限制）。
3. **多采样平均法**：在每帧取该时间段内的平均值。适合展示平滑趋势（如能量曲线），不适合精确位置。

物理轨迹建议用**插值法**——它至少能保证位置的一阶连续性，视觉上不会'跳帧'。"

**大师**（补充）：

"还有一个重要技巧——**不要每帧重建整条轨迹曲线**。轨迹应该是一个递增的 `VMobject`，用 `add_points_as_corners` 追加新点，而不是每帧 `set_points_as_corners` 重建全部历史点。前者的复杂度是 O(新增点数)，后者是 O(总点数)。"

> **技术映射**：`np.interp(x, xp, fp)` 做一维线性插值。`VMobject.add_points_as_corners(new_pts)` 追加新点到路径末尾，避免每次重建整个路径。

#### 步骤一：数值求解双摆方程

**步骤目标**：用 scipy 求解双摆的拉格朗日方程，导出角度序列。

```python
# scenes/chapter24_double_pendulum_solver.py
import numpy as np
from scipy.integrate import solve_ivp

def double_pendulum_deriv(t, y, L1=2.0, L2=1.5, m1=1.0, m2=1.0, g=9.8):
    """双摆的拉格朗日方程导数"""
    theta1, omega1, theta2, omega2 = y
    delta = theta2 - theta1

    denom1 = (m1 + m2) * L1 - m2 * L1 * np.cos(delta) ** 2
    denom2 = (L2 / L1) * denom1

    dtheta1 = omega1
    domega1 = (m2 * L1 * omega1 ** 2 * np.sin(delta) * np.cos(delta)
               + m2 * g * np.sin(theta2) * np.cos(delta)
               + m2 * L2 * omega2 ** 2 * np.sin(delta)
               - (m1 + m2) * g * np.sin(theta1)) / denom1

    dtheta2 = omega2
    domega2 = (-m2 * L2 * omega2 ** 2 * np.sin(delta) * np.cos(delta)
               + (m1 + m2) * (g * np.sin(theta1) * np.cos(delta)
               - L1 * omega1 ** 2 * np.sin(delta)
               - g * np.sin(theta2))) / denom2

    return [dtheta1, domega1, dtheta2, domega2]

def solve_pendulum(T=10, dt=0.02):
    """求解双摆轨迹"""
    L1, L2 = 2.0, 1.5
    y0 = [np.pi / 2, 0.0, np.pi / 2, 0.0]  # 初始角度和角速度
    t_eval = np.arange(0, T, dt)
    sol = solve_ivp(double_pendulum_deriv, (0, T), y0, t_eval=t_eval, method='RK45')
    theta1, theta2 = sol.y[0], sol.y[2]
    # 转换为笛卡尔坐标
    x1 = L1 * np.sin(theta1)
    y1 = -L1 * np.cos(theta1)
    x2 = x1 + L2 * np.sin(theta2)
    y2 = y1 - L2 * np.cos(theta2)
    return t_eval, (x1, y1, x2, y2)
```

#### 步骤二：双摆动画

**步骤目标**：将数值解用 Manim 渲染为物理动画。

```python
# scenes/chapter24_pendulum.py
from manim import *
from chapter24_double_pendulum_solver import solve_pendulum

class DoublePendulum(Scene):
    def construct(self):
        # 预计算轨迹
        T, dt = 8.0, 0.02
        t_eval, (x1, y1, x2, y2) = solve_pendulum(T, dt)

        origin = UP * 2.5  # 固定点
        time_tracker = ValueTracker(0)

        # 摆杆 1（从 origin 到关节 1）
        rod1 = always_redraw(lambda: Line(
            origin,
            origin + np.array([x1[int(time_tracker.get_value() / dt)],
                               y1[int(time_tracker.get_value() / dt)], 0]) * 1,
            color=BLUE, stroke_width=3,
        ))

        # 摆杆 2（从关节 1 到关节 2）
        rod2 = always_redraw(lambda: Line(
            origin + np.array([x1[int(time_tracker.get_value() / dt)],
                               y1[int(time_tracker.get_value() / dt)], 0]) * 1,
            origin + np.array([x2[int(time_tracker.get_value() / dt)],
                               y2[int(time_tracker.get_value() / dt)], 0]) * 1,
            color=ORANGE, stroke_width=3,
        ))

        # 两个关节
        joint1 = always_redraw(lambda: Dot(
            origin + np.array([x1[int(time_tracker.get_value() / dt)],
                               y1[int(time_tracker.get_value() / dt)], 0]) * 1,
            color=BLUE, radius=0.08,
        ))
        joint2 = always_redraw(lambda: Dot(
            origin + np.array([x2[int(time_tracker.get_value() / dt)],
                               y2[int(time_tracker.get_value() / dt)], 0]) * 1,
            color=RED, radius=0.12,
        ))

        # 轨迹（关节 2 的历史路径）
        trail_points = []
        def get_trail():
            if len(trail_points) < 2:
                return VGroup()
            pts = [origin + np.array([x, y, 0]) * 1 for (x, y) in trail_points]
            return VMobject().set_points_as_corners(pts).set_color(YELLOW).set_stroke(width=1)

        trail = always_redraw(get_trail)

        # 固定点
        pivot = Dot(origin, color=WHITE, radius=0.08)
        self.add(pivot)

        title = Text("双摆运动——混沌系统演示", font_size=30, color=WHITE).to_edge(UP, buff=0.3)
        self.play(Write(title))

        self.add(rod1, rod2, joint1, joint2, trail)
        self.play(time_tracker.animate.set_value(T - dt), run_time=10, rate_func=linear)

        # 动画期间收集轨迹
        for i in range(0, len(t_eval), 5):  # 每 5 个采样点取一个轨迹点
            trail_points.append((x2[i], y2[i]))

        self.wait(1.5)
        self.play(FadeOut(VGroup(title, rod1, rod2, joint1, joint2, trail, pivot)), run_time=1.5)
```

**请注意**：上面的代码有个已知问题——轨迹点是在整个 play 开始前一次性采集的（`for i in range(...)` 在 `play` 之前执行），所以轨迹不会在动画期间"实时生长"。正确的实时轨迹收集需要更复杂的 `ValueTracker` 回调模式，这里简化为"完整轨迹一次性出现"来展示核心结构。

#### 步骤三：修正——实时轨迹生长

**步骤目标**：轨迹随动画实时生长（模拟真实摆动过程）。

修正方案：用一个全局列表 `trail_points` 在 `always_redraw` 中累积轨迹点：

```python
trail_points = []
last_idx = [0]

def get_trail():
    idx = int(time_tracker.get_value() / dt)
    if idx > last_idx[0]:
        for i in range(last_idx[0] + 1, idx + 1, 4):
            trail_points.append((x2[i], y2[i]))
        last_idx[0] = idx
    if len(trail_points) < 2:
        return VGroup()
    pts = [origin + np.array([x, y, 0]) * 1 for (x, y) in trail_points]
    return VMobject().set_points_as_corners(pts).set_color(YELLOW).set_stroke(width=1)

trail = always_redraw(get_trail)
```

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 计算分离 | 预计算轨迹后渲染飞流畅 | 需要管理两套时间参数（仿真 dt vs 动画 fps） |
| 精度可控 | 仿真可用 dt=0.001 级精度 | 高精度意味着大数据量（内存占用） |
| 复用性 | 求解器可复用于任何常微分方程 | 需要微分方程知识（拉格朗日/牛顿力学） |
| 视觉效果 | 双摆的混沌轨迹极富观赏性 | 轨迹点累积后内存持续增长 |
| 扩展性 | 可更换方程模拟任何机械系统 | 增加约束（碰撞/摩擦）后求解器变复杂 |

### 适用场景

| 场景 | 说明 |
|------|------|
| 物理教学 | 摆锤、弹簧、轨道力学 |
| 机械工程 | 四杆机构、曲柄滑块 |
| 流体力学 | 涡旋、流线的简化模拟 |
| 电气工程 | RLC 电路的瞬态响应 |
| 生物医学 | 种群动力学、传染病模型 |

**不适用场景**：需要实时物理引擎的交互应用（使用 Box2D/PhysX）、需要碰撞检测和刚体模拟的场景（使用 Mujoco/Bullet）。

#### 扩展阅读：用 sympy 推导运动方程

如果你不想手动写拉格朗日方程（双摆的推导有十几行导数），可以用 `sympy` 做符号推导。sympy 的 `Lagrangian` 类能自动生成运动方程，然后 `lambdify` 转成数值函数喂给 `solve_ivp`。这虽然不是 Manim 的范畴，但在制作复杂力学动画时能省下大量推导时间。

```python
import sympy as sp
# 符号定义
t = sp.symbols('t')
theta1, theta2 = sp.Function('theta1')(t), sp.Function('theta2')(t)
# ... 计算拉格朗日量 L = T - V
# ... sp.Lagrangian(L) 生成运动方程
# ... sp.lambdify 转数值函数
```

### 注意事项

1. **`solve_ivp` 的 `method='RK45'` 是自适应步长**：返回的 t 不是等间距的。使用 `t_eval` 参数强制指定输出时间点。
2. **索引越界保护**：`int(time_tracker.get_value() / dt)` 可能超过数组长度。用 `min(idx, len(data)-1)` 限制边界。
3. **numpy 数组复制**：在 `always_redraw` 中直接引用 numpy 数组是安全的（只读），但如果修改需先 `.copy()`。

### 常见踩坑经验

**故障一：双摆动画中杆长视觉上变化**

根因：原始数据 x1, y1 是实际物理坐标，单位是米。Manim 的画面坐标单位与米不匹配。

解决：乘以缩放因子，或直接在求解器中将 L1, L2 设为与实际画面尺寸匹配的值（如 L1=1.5 对应画面 1.5 单位）。

**故障二：轨迹累积导致渲染越来越慢**

根因：`trail_points` 持续增长的 list 在 `set_points_as_corners` 中全部重新绘制。

解决：限制轨迹长度（如只保留最近 500 个点），或只用最后 N 个点。

**故障三：`dt` 选择不当导致索引跳跃**

根因：仿真 `dt=0.02` 产生 500 个数据点/10秒，但 `time_tracker.animate(run_time=10)` 包含更多帧。两帧可能对应同一个数组索引。

解决：使用 `time_tracker` 的连续值做插值，而非取整索引：`np.interp(time, t_eval, x1)`。
