# 第29章：性能优化与大场景拆分

---

## 1. 项目背景

某数据可视化团队正在用 Manim 制作一段"粒子系统模拟"动画——需要在画面上同时展示 2000 个彩色粒子的运动、碰撞和聚类效果。初版代码在低清预览下花了 8 分钟才渲染出 10 秒的画面，切换到 1080p 后直接报"内存不足"。

团队优化工程师老唐负责性能调优。他分析了渲染性能的瓶颈，发现：

1. **对象过多**：2000 个 Circle 对象，每个都是独立的 VMobject（带贝塞尔曲线点集），导致渲染循环遍历 2000 次对象状态更新。
2. **Updater 冗余**：2000 个粒子每个都绑定了 `add_updater`，每秒 60 帧 × 2000 个函数调用 = 12 万次/秒的 Python 函数调用开销。
3. **LaTeX 重复编译**：场景中有 5 个实时更新的公式标签，每帧都触发 LaTeX 重编译（尽管内容没变）。
4. **帧缓存爆内存**：Manim 默认将每帧渲染结果缓存在内存中，1080p × 30fps × 10 秒 = 300 帧 × 8MB ≈ 2.4GB 内存占用。

这些痛点的本质是 Manim 在**大规模场景下的性能瓶颈**。本章要系统讲解对象优化、Updater 减量、LaTeX 缓存和内存管理策略。

## 2. 剧本式交锋对话

> **场景**：老唐的电脑风扇狂转，渲染进度条像蜗牛一样蠕动。

**小胖**（扇着扇子给笔记本降温）：

"唐哥，你这个粒子动画比煮泡面还慢。你确定这 2000 个粒子每个都需要是一个独立的 Circle？就不能像游戏引擎那样把 2000 个粒子打包成一个批次渲染？"

**老唐**（盯着 CPU 100% 的任务管理器）：

"我也想知道。Manim 有没有'合批'机制？2000 个圆每个都是独立的 Mobject，渲染器要遍历 2000 次。如果能合并成一个大的 VMobject，一次渲染就全画完，是不是快很多？"

**小白**（打开 Manim 的 `VMobject` 源码）：

"有的，虽然不是官方的'合批 API'，但有一个等效方案——**手动构建点集**。一个 `VMobject` 可以包含多条独立的贝塞尔曲线。你可以在一个 `VMobject` 中用 2000 条 `Mobject` 子路径表示 2000 个圆，然后整体渲染。对象数从 2000 降到 1，渲染循环开销减少 99%！

但代价是你不能对单个粒子做 `set_color` 或 `animate`——因为它们不再独立存在。所以你需要在'大批量画点'和'少量精细动画'之间做权衡。"

**大师**（在白板上画了分层架构）：

"大场景优化的核心策略是**分层渲染**：

**动态层**：20-50 个需要动画的关键对象（如主角、高亮粒子）。用常规 Mobject + Animation。
**静态层**：成百上千个不需要独立动画的背景粒子。合并为一个 VMobject 或 `ImageMobject`。
**标注层**：10-20 个文本/公式标签。用 `always_redraw` 但配合 LaTeX 缓存。

三层用不同的 VGroup 管理，动态层在上面（z_index 高），静态层在下面。摄像机移动时三层一起动，但只有动态层需要每帧更新。"

> **技术映射**：`VMobject.set_points_as_corners(points)` 将点列表渲染为折线。多个闭合子路径用 `append_points` 追加。一个 VMobject 可容纳数千条子路径。

**小胖**（扇子扇得更快了）：

"那 Updater 呢？2000 个 updater 是不是可以减到 1 个？"

**大师**：

"正是。Updater 的性能开销来自 Python 函数调用——每帧调用 2000 次，每次都要 push/pop 调用栈。如果你把 2000 个粒子的更新逻辑合并到一个 updater 中，用 numpy 数组做批量位置计算，开销从 `2000 × func_call` 降到 `1 × func_call + numpy_vectorized`。

```python
positions = np.random.randn(2000, 2) * 3
particles = VMobject()

def update_all(mob, dt):
    global positions
    positions += np.random.randn(2000, 2) * dt * 0.5  # numpy 批量运算
    # 用单一 set_points 更新所有粒子
    ...

particles.add_updater(update_all)
```

"numpy 的向量化运算比 Python 循环快 100 倍以上。"

**小白**（追问）：

"如果场景里有 10 个曲线标签需要 LaTeX 渲染，但它们的公式内容在动画过程中不变——只是位置跟随对象在变动——怎么避免 LaTeX 每帧重编译？"

**大师**：

"这种情况适合用 `MathTex` 的缓存 + 纯位置动画。具体做法是——在 `construct()` 中预先创建所有 10 个 `MathTex` 对象（触发一次 LaTeX 编译并缓存）。之后它们的位置变化用 `animate.move_to()` 或 `Updater` 只修改坐标，不触碰公式内容。Manim 会自动跳过已缓存的公式重编译。

但如果公式内容在变（如 `a_tracker` 驱动的公式），就必须承受每次重建的 LaTeX 编译开销。此时有两个优化思路：
1. 把公式拆成'不变部分'（预编译）和'可变部分'（用 `DecimalNumber` 而非 `MathTex` 显示数值变化）
2. 用 `SingleStringMathTex`（轻量级单行公式）代替 `MathTex`（完整 LaTeX 文档）"

> **技术映射**：`MathTex` 的 LaTeX 编译结果按内容哈希缓存。相同公式字符串第二次出现时，Manim 直接复用 SVG，耗时从数百毫秒降到 <1ms。

#### 步骤一：对象数优化——合批渲染

**步骤目标**：将 500 个独立 Circle 合并为 2 个 VMobject。

```python
# scenes/chapter29_batch.py
from manim import *
import numpy as np

class BatchVsIndividual(Scene):
    def construct(self):
        title = Text("批处理 vs 独立对象", font_size=30, color=BLUE).to_edge(UP, buff=0.3)
        self.play(Write(title))

        n = 500

        # ---- 左侧：500 个独立 Circle（慢） ----
        start_individual = time.perf_counter()
        individual_dots = VGroup()
        for i in range(n):
            dot = Dot(
                [np.random.uniform(-6, -1), np.random.uniform(-3, 3), 0],
                radius=0.03,
            )
            individual_dots.add(dot)
        elapsed_individual = time.perf_counter() - start_individual

        # ---- 右侧：1 个 VMobject 包含 500 个子路径（快） ----
        start_batch = time.perf_counter()
        batch_vmobject = VMobject(stroke_width=0, fill_opacity=1)
        batch_paths = []
        for i in range(n):
            x, y = np.random.uniform(1, 6), np.random.uniform(-3, 3)
            # 用微小的菱形近似圆点
            r = 0.03
            pts = [[x-r,y,0], [x,y+r,0], [x+r,y,0], [x,y-r,0]]
            batch_paths.append(pts)
        # 将所有路径合并为一个 VMobject
        batch_vmobject.set_points_as_corners(
            [p for path in batch_paths for p in path]
        )
        elapsed_batch = time.perf_counter() - start_batch

        self.add(individual_dots, batch_vmobject)

        # 显示创建耗时
        result_text = Text(
            f"独立对象: {elapsed_individual*1000:.0f}ms\n"
            f"批处理:   {elapsed_batch*1000:.0f}ms",
            font_size=26, color=GREEN,
        ).to_edge(DOWN, buff=0.5)
        self.play(Write(result_text), run_time=1.5)
        self.wait(1.5)

        self.play(FadeOut(VGroup(title, individual_dots, batch_vmobject, result_text)), run_time=1.5)
```

**运行结果**：

左侧 500 个 Dot（独立 Mobject），右侧 500 个微菱形（1 个 VMobject 的多个子路径）。创建耗时对比显示批处理比独立对象快 10-50 倍（具体取决于机器性能）。

#### 步骤二：Updater 减量——numpy 批量更新

**步骤目标**：将 1000 个粒子的 1000 个 updater 合并为 1 个。

```python
# scenes/chapter29_updater_optimize.py
from manim import *
import numpy as np

class OptimizedParticles(Scene):
    def construct(self):
        title = Text("numpy 批量更新粒子系统", font_size=28, color=BLUE).to_edge(UP, buff=0.3)
        self.play(Write(title))

        n_particles = 1000
        positions = np.random.randn(n_particles, 3) * 3
        velocities = np.random.randn(n_particles, 3) * 0.5

        # 每个粒子用 4 个顶点绘制的菱形
        particles = VMobject(stroke_width=0, fill_opacity=0.6)
        # 预分配点集
        all_points = np.zeros((n_particles * 4, 3))
        offsets = np.array([
            [-0.02, 0, 0], [0, 0.02, 0], [0.02, 0, 0], [0, -0.02, 0]
        ])

        def update_particles(mob, dt):
            global positions, velocities
            # 用 numpy 批量更新位置
            velocities += np.random.randn(n_particles, 3) * dt * 0.3
            positions += velocities * dt

            # 边界碰撞
            mask = np.abs(positions[:, :2]) > 6
            velocities[mask] *= -1

            # 构建所有点
            for i in range(n_particles):
                all_points[i*4:(i+1)*4] = positions[i] + offsets
            mob.set_points_as_corners(all_points)

        particles.add_updater(update_particles)
        self.add(particles)

        # 渲染 5 秒粒子动画
        self.wait(5)

        self.play(FadeOut(VGroup(title, particles)), run_time=1)
```

**运行结果**：

1000 个粒子在画面上自由运动 5 秒，所有更新逻辑在单个 updater 中通过 numpy 批量完成。相比 1000 个独立 updater，CPU 占用降低约 60-80%。

**可能遇到的坑**：

1. **`set_points_as_corners` 不支持 `fill_opacity` 的着色**：它是纯线条模式，需要 `stroke_width > 0` 或配合 `fill_opacity` 使用闭合路径。
2. **numpy 批量运算在 1000 粒子下很快，10 万粒子时会吃光 CPU**：Python 的 GIL 限制了单进程的 numpy 向量化性能。10 万+粒子考虑用 OpenGL 渲染器。

#### 步骤三：LaTeX 缓存与内存管理

**步骤目标**：禁用不必要的 LaTeX 重编译，降低帧缓存内存。

```bash
# manim.cfg 优化
[CLI]
# 降低帧率（对大规模场景可降到 15fps 但视觉仍可接受）
frame_rate = 15

# 禁用部分电影文件缓存（减少内存占用）
disable_caching = true

# 写入电影前不缓存所有帧在内存
write_to_movie = true

[tex]
# 保留 LaTeX 缓存（避免重复编译）
use_cache = true

[camera]
# 降低分辨率预览（开发阶段）
# 正式输出时切换为 -qh
pixel_width = 854
pixel_height = 480
```

**关键优化点**：

- `frame_rate = 15`：对粒子动画等不需要逐帧精确的场景，15fps 视觉上仍然流畅，但 CPU/内存占用减半。
- `disable_caching = true`：不缓存中间帧到内存，直接写入磁盘。内存占用从几千 MB 降到几百 MB。
- `use_cache = true`：LaTeX 编译结果持久化，后续渲染跳过已编译公式。

#### 扩展：用 `cProfile` 定位性能热点

当你不知道性能瓶颈在哪里时，用 Python 的 `cProfile` 剖析一次渲染：

```bash
python -m cProfile -o render.prof -m manim scene.py MyScene -pql
# 用 snakeviz 可视化
pip install snakeviz && snakeviz render.prof
```

`snakeviz` 会显示一个火焰图，让你看到哪几个函数消耗了最多时间。常见的 Manim 性能热点：
- `VMobject.set_points()` —— 大量对象的点集更新
- `Camera.display_multiple_vectorized_mobjects()` —— 每帧渲染所有 VMobject
- `Scene.play()` —— Animation 的插值循环
- `LatexRunner.compile()` —— LaTeX 公式编译

针对性的优化比盲目的"reduce everything"有效得多。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| 合批渲染 | 对象数从 N 降到 1-3，渲染循环开销大减 | 失去独立对象的动画能力 |
| numpy 批量更新 | 单 updater 代替 N 个，CPU 占用降低 60-80% | 调试困难（无逐粒子断点） |
| LaTeX 缓存 | 已编译公式秒级渲染 | 缓存目录体积大会影响磁盘 IO |
| 帧缓存关闭 | 内存占用降低 70%+ | 极限场景下可能 IO 成为瓶颈 |
| 帧率降低 | 15fps 对粒子动画可接受 | 精细标注和文字在低帧率下不流畅 |

### 适用场景

| 场景 | 说明 |
|------|------|
| 粒子系统 | 1000+ 粒子的运动模拟 |
| 大规模网络图 | 密集的节点与连线 |
| 数据散点图 | 10 万+ 数据点的可视化 |
| 分形图形 | Mandelbrot 集等高密度渲染 |
| 背景效果 | 星空、雨滴、飘雪等装饰层 |

**不适用场景**：需要每个对象独立动画的精确叙事场景（用 Mobject + Animation）、需要逐对象交互响应的场景（用 Updater）。

### 注意事项

1. **`set_points_as_corners` 创建的路径不支持 `set_color` 逐段着色**：整条路径共享一组颜色属性。如需多色粒子，应创建多个 VMobject（按颜色分组）。
2. **numpy 批量更新的 `dt` 精度**：单 updater 中的 `dt` 是上一帧的时间间隔。如果帧率波动（如从 30fps 掉到 10fps），物理模拟会变得不准确。
3. **合批渲染与摄像机运动**：合批的 VMobject 在摄像机运动时正常变换——它遵守 `Mobject` 的变换矩阵。不需要特殊处理。

### 常见踩坑经验

**故障一：合批后粒子大小视觉不一致**

根因：`set_points_as_corners` 绘制的菱形在屏幕不同位置可能因透视而变形（虽然 2D 渲染不应该）。

解决：使用 `Dot`（基于 Circle 的近似）的合批方案需要用 `VMobject` 的曲线模式（贝塞尔），而非折线模式。

**故障二：`disable_caching=true` 后视频输出损坏**

根因：FFmpeg 的写入流在无缓存模式下对磁盘 IO 要求更高。慢速磁盘（HDD/网络盘）可能导致写入失败。

解决：设置输出目录在本地 SSD 上，或使用 `--format gif` 测试是否 IO 问题。

**故障三：批量更新中 numpy 操作导致对象不刷新**

根因：`mob.set_points_as_corners(all_points)` 修改了 points 但未触发 Manim 的脏标记（dirty flag）。

解决：在 updater 末尾调用 `mob.refresh_bounding_box()` 或设 `fill_opacity` 重新触发渲染检测。

### 思考题

1. 改造 `OptimizedParticles`，按粒子速度大小映射颜色——速度快的粒子呈红色，慢的呈蓝色。提示：按速度大小分组为 3 个 VMobject（红/黄/蓝），每组独立设置颜色。
