# 第13章：错误排查与调试工作流

---

## 1. 项目背景

新人小何加入团队的第三天，被分配了一个"简单"任务：用 Manim 制作一段"因式分解 x² - 5x + 6"的教学动画。小何花两小时写好了代码，然后花了整整一天调试——来来回回遇到：

1. **`NameError: name 'Scene' is not defined`** —— 忘了 `from manim import *`，但报错信息没指明是哪个类缺失。
2. **`LaTeX Error: Missing $ inserted`** —— LaTeX 字符串中混入了普通字符，但报错堆栈长达 300 行，小何不会看。
3. **动画渲染完成但画面是一片黑** —— 所有对象都被创建但初始位置在画面外。
4. **`AttributeError: 'VGroup' object has no attribute 'set_fill'`** —— 对 VGroup 调用了不存在的方法，但 VGroup 的方法列表没有文档。

这一天结束后，小何对 Manim 的信心从 80% 掉到了 30%——不是因为功能不够，而是因为"不知道报错了怎么办"。

Manim 的调试痛点不同于普通 Python 程序——它有**三层错误源**：Python 层（语法/类型/引用错误）、LaTeX 层（公式编译错误）、渲染层（对象位置/颜色/可见性异常）。每层有自己的日志格式和排查思路。本章的目标就是建立这三层的排查 SOP，让"报错"不再可怕。

## 2. 剧本式交锋对话

> **场景**：小何的屏幕上，从 `manim` 命令敲下到满屏红色报错不到 3 秒。小胖端着一碗酸辣粉路过。

**小胖**（吹了吹热气）：

"小何，你这屏幕像是俄罗斯方块玩到 Game Over 了——红彤彤一片。Manim 报错这么多行，你得从哪行开始看？"

**小何**（快哭了）：

"我不知道！这 200 行堆栈我全看完了，还是不知道哪错了。它说 LaTeX 编译失败，但我明明写的是 `MathTex(r"x^2")`，这不是标准的 LaTeX 吗？"

**小白**（冷静地滚动报错日志）：

"教你一个读报错的口诀——**从下往上看，找第一个你自己的文件名**。Manim 的堆栈从底层库（`manim/animation/...`、`manim/renderer/...`）往上抛，最顶上可能是 `subprocess` 或 `os` 的底层异常。真正有用的信息在"堆栈的上半部分"——找你自己的 `.py` 文件，那行附近才是问题的根源。"

**大师**（接过键盘演示）：

"来，我们现场拆一个典型报错。"

（屏幕上显示着 LaTeX 错误）

```
RuntimeError: latex error converting to svg

Error logs:
! Missing $ inserted.
<inserted text> ... <*> ... x^2 + 2x + 1
```

"这个报错的根因在倒数第 3 行——`Missing $ inserted`。意思是 LaTeX 发现你的公式里混入了数学符号但没有用 `$...$` 包裹。虽然你写了 `MathTex(r"x^2 + 2x + 1")`，但可能是 `r` 前缀丢了导致 `\^` 被转义。排查方向：检查字符串是否真的以 `r"` 开头。"

> **技术映射**：`MathTex` 在后台调用 `tex_to_svg_file()`，该函数用 `subprocess.run(["latex", ...)` 调用 LaTeX 编译器。如果 LaTeX 返回非零退出码，Manim 抛出 `ValueError` 并附带 stderr 输出。

**小胖**（吸了一口粉）：

"那画面一片黑呢？没报错啊，但什么东西都没有。"

**小白**：

"没报错不代表没问题。画面黑的常见原因就四个——我列个排查表（PULL 法则）：

- **P**osition（位置）—— 对象在画面外？在 `construct` 最后加 `self.wait(3)` 并用鼠标滚轮缩放看看画面角落。
- **U**nadded（未添加）—— 忘了 `self.add()` 或 `self.play(Create())`？对象存在于内存但不在渲染列表。
- **L**ighting（颜色/透明度）—— fill_opacity=0 或 color 和背景色相同？黑底黑字当然看不见。
- **L**aTeX（公式为空）—— MathTex 编译成功了但内容为空？检查是否有 `\phantom` 或 `\vspace` 导致内容为空白。"

**大师**：

"我再补充一个调试神器——**辅助线**。当你怀疑对象位置不对时，在 `construct` 开头加一行代码就能厘清坐标："

```python
# 调试技巧：添加坐标网格和辅助点
self.add(NumberPlane(
    x_range=[-7, 7, 1], y_range=[-4, 4, 1],
    background_line_style={"stroke_opacity": 0.2},
))
```

"有了坐标网格，你就能直观看到每个对象在画面上的坐标。调试完记得删掉这行。"

> **技术映射**：`NumberPlane` 是 Manim 内置的坐标网格 Mobject，每个交点代表 (x, y) 坐标。它帮你把"抽象的坐标值"映射为"可视的像素位置"。

**小胖**（酸辣粉已经吃完了）：

"我还是有个疑问——最可怕的不是报错，而是**不报错但效果不对**。比如我上次写的动画，在 play 里明明写了 `(circle.animate.shift(RIGHT * 5))`，但画出来圆圈只动了一丁点。没报错，我盯着代码看了半小时才发现——是前面某行误把它 scale(0.1) 了，它真的在 shift(RIGHT * 5)，但因为它太小了，move 的效果几乎看不见。"

**小白**：

"这种情况就是**逻辑错误/状态污染**——比语法错误更难排查。核心问题是 Manim 对象是**有状态的**（mutable），前面某步改了它的 scale 或 color 或 opacity，后面的所有 animation 都在这个"被污染的状态"上执行。小胖你 shift(RIGHT*5) 的代码没错，但被之前的 scale(0.1) 污染了。"

**大师**（总结性地列出排查清单）：

"针对不报错的诡异结果，我给出**六条排查法则**（SCOPE-C）：

- **S**cale check：对象是否被意外缩放？（插入 `print(obj.get_width())`）
- **C**olor check：颜色是否被覆盖？（插入 `print(obj.get_color())`）
- **O**pacity check：是否透明到看不见？（`print(obj.get_fill_opacity())`）
- **P**osition check：坐标是否超出画面？（用 NumberPlane）
- **E**xisting objects：旧对象是否残留？（`print(len(self.mobjects))`）
- **C**opy/Reference：是否误用了同一对象的引用？（`print(obj1 is obj2)`）

每一条排查法，一行 `print` 或 `self.add(NumberPlane())` 就够了。养成在关键节点插入诊断输出的习惯，能省下 80% 的排查时间。"

> **技术映射**：`Mobject` 是可变对象，`animate` 和 `set_*` 方法直接修改原始对象。在每次 transform 前用 `obj.save_state()` 保存快照，必要时 `obj.restore()` 回退。

"那我还有一个问题——Manim 的 `print` 能打到哪？我加了 `print(obj.get_center())` 但终端没显示。"

**大师**：

"这就是 Manim 的 **logger 系统**。默认的 `print()` 在 Manim 的渲染进程中会被重定向到 logger。你应该用 `import logging` + `logging.getLogger("manim").info("...")` 来输出。或者更简单——在 `construct` 中直接 print，观察 `manim` 命令终端的输出（不在 IDE 里跑，在终端里跑）。"

#### 步骤一：常见错误的识别与修复

**步骤目标**：模拟 5 类常见错误并修复。

```python
# scenes/chapter13_bugs.py
# 注意：本文件包含刻意制造的错误，
# 用于演示调试流程。正确代码见注释部分。
from manim import *
import logging
logger = logging.getLogger("manim")


class Scene1_MissingImport(Scene):
    """错误演示：忘记导入"""
    def construct(self):
        # 下面这行会报 NameError 因为没 import Text
        # t = Text("Hello")  # 取消注释看效果
        # 正确写法：
        from manim import Text   # 在文件顶部统一 import
        t = Text("已修复", font_size=36, color=WHITE)
        self.play(Write(t))
        self.wait(1)


class Scene2_LatexError(Scene):
    """错误演示：LaTeX 语法错误"""
    def construct(self):
        # 错误写法：\frav 是错误的命令
        # wrong = MathTex(r"\frav{a}{b}")  # 会引发 LaTeX Error
        # 修复写法：
        correct = MathTex(r"\frac{a}{b}", font_size=48, color=WHITE)
        self.play(Write(correct))
        self.wait(1)


class Scene3_BlackScreen(Scene):
    """错误演示：画面全黑（对象在画面外）"""
    def construct(self):
        # 错误写法：对象远在画面之外
        # circle = Circle(radius=0.5, color=RED).shift(RIGHT * 20)

        # 修复前先加网格调试
        grid = NumberPlane(
            x_range=[-10, 10, 1], y_range=[-6, 6, 1],
            background_line_style={"stroke_opacity": 0.15},
        )
        self.add(grid)

        circle = Circle(radius=0.5, color=RED, fill_opacity=0.5)
        # 调试：打印位置
        logger.info(f"Circle position: {circle.get_center()}")
        self.play(Create(circle))
        self.wait(1)


class Scene4_OpacityZero(Scene):
    """错误演示：透明度为 0"""
    def construct(self):
        # 错误写法：
        # ghost = Square(fill_opacity=0, stroke_opacity=0)  # 完全看不见

        # 修复时先创建可见版本，再动画过渡
        square = Square(side_length=2, color=BLUE, fill_opacity=0.3)
        self.play(Create(square))
        self.wait(0.5)
        # 如果想要淡出，用 FadeOut 而不是直接 set_opacity(0)
        self.play(FadeOut(square))
        self.wait(0.5)
```

#### 步骤二：建立最小复现脚本

```python
# scenes/chapter13_minimal.py
"""
最小复现原则：当你遇到一个 bug，在完整场景中无法定位时，
立即创建一个新脚本，只保留最少的代码来触发这个 bug。
"""

from manim import *

class MinimalRepro(Scene):
    """骨架：只创建一个对象 + 一个动画。
    如果这个跑不通，说明环境有问题。
    如果跑通了但原场景有问题，对比差异定位。
    """
    def construct(self):
        # 骨架元素（删掉不干事的，只保留最简）
        dot = Dot(ORIGIN, color=YELLOW)
        self.play(Create(dot), run_time=1)
        self.wait(1)

        # 打印诊断信息
        print(f"frame_width: {config.frame_width}")
        print(f"dot position: {dot.get_center()}")
        print(f"dot color: {dot.get_color()}")

        # 测试完毕后逐步加回原始复杂代码
        # ...
```

```bash
# 如果成功，逐步合并原始场景的代码直到问题浮现
```

#### 步骤四：逐帧调试——慢速观察动画中间态

**步骤目标**：掌握 `--write_to_movie=false` + `--format=png` 逐帧观察技巧。

```python
# scenes/chapter13_frame_debug.py
from manim import *

class FrameByFrame(Scene):
    """逐帧调试演示"""
    def construct(self):
        circle = Circle(radius=1, color=BLUE, fill_opacity=0.3)
        circle.shift(LEFT * 3)

        square = Square(side_length=2, color=YELLOW, fill_opacity=0.3)
        square.shift(RIGHT * 3)

        self.play(Create(circle), run_time=1.5)

        # 调试方法一：在 play 中间插入 wait + print
        print(f"[Frame Debug] After Create, circle at: {circle.get_center()}")
        self.wait(0.5)

        # Transform 动画期间的逐帧检查
        self.play(Transform(circle, square), run_time=2)

        # 调试方法二：检查对象合并后的状态
        print(f"[Frame Debug] After Transform, circle.color={circle.get_color()}")
        print(f"[Frame Debug] circle.submobjects count: {len(circle.submobjects)}")

        self.wait(1)
```

**逐帧渲染命令**：

```bash
# 方式一：只渲染关键帧为 PNG 序列
manim -pql --write_to_movie=false --format=png scenes/chapter13_frame_debug.py FrameByFrame

# 方式二：降低帧率观察
manim -pql --frame_rate=5 scenes/chapter13_frame_debug.py FrameByFrame

# 方式三：渲染特定片段（修改代码中 run_time=0.1 加速后再检查）
manim -pql scenes/chapter13_frame_debug.py FrameByFrame
```

**可能遇到的坑**：

1. **`--write_to_movie=false` 仍渲染所有帧**：这个参数只跳过 FFmpeg 合成步骤，每帧的 PNG 仍然生成。如果场景有 30 秒 × 60fps = 1800 帧，会生成 1800 张 PNG。
2. **PNG 调试后磁盘占用**：记得在 `media/images/` 目录清理不再需要的帧。

## 4. 项目总结

### 优点 & 缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| Python 层调试 | 标准 try/except + print + logger 可用 | Manim 内部错误被封装后堆栈较长 |
| LaTeX 层调试 | 完整 stderr 输出到终端，可逐行分析 | LaTeX 报错术语晦涩（如 Misplaced \noalign） |
| 渲染层调试 | NumberPlane + SurroundingRectangle 直观可视化 | 需手动添加/删除调试代码 |
| 最小复现 | 剥离无关逻辑后快速定位 | 复杂场景（Updater + 多对象联动）不易剥离 |
| 团队协作 | debug_utils.py 统一调试入口 | 调试代码误提交到主线仓库的风险 |

### 适用场景

| 场景 | 说明 |
|------|------|
| 新人入门 | 遇到报错不知如何看日志 |
| 复杂动画 | 多对象联动时排查单个对象的问题 |
| 团队审查 | 代码 PR 时预检常见错误 |
| CI 守护 | 构建前扫描脚本的常见语法问题 |
| 教学演示 | 展示错误修复过程 |

**不适用场景**：性能瓶颈定位（需要使用 profiler 或计时器，而非本文的日志调试）、OpenGL 硬件兼容问题（需要 GPU 驱动诊断工具）。

### 注意事项

1. **调试代码勿提交**：`add_coord_grid()` 等调试代码应在验证完成后的最终版本中删除。建议在文件末尾标记 `# DEBUG-ONLY` 注释，方便搜索清理。
2. **`logger.info` vs `print`**：在 `construct` 内部 `print` 会输出但可能被 Manim 缓冲。建议用 `sys.stdout.flush()` 或 `logger.info` 确保实时输出。
3. **LaTeX 缓存更新**：修改 MathTex 字符串后，必须确保 LaTeX 缓存已刷新。如果遇到"修改了公式但渲染没变"——运行 `manim --clear_cache`。

### 常见踩坑经验

**故障一：`self.play()` 后 `print` 不立即执行**

根因：`play()` 内部设置了渲染锁，`print` 在渲染锁释放后才执行。

解决：在 `self.wait(0.1)` 之后 `print`，确保渲染已完成一轮帧。

**故障二：修改代码后 Manim 用旧缓存渲染**

根因：LaTeX 缓存和 `.partial_movie_files` 目录未清理。

解决：`rm -rf media/Tex` 或 `manim --clear_cache`。

**故障三：`Indicate` 不生效以为是代码 bug**

根因：`Indicate` 在对象颜色和背景色相近时视觉上几乎看不到——它通过短暂地改变对象颜色来"闪烁"。

解决：用 `Flash` 替代（添加一个包围框的闪光）。
