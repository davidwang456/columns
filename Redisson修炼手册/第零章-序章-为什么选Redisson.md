# 第零章：序章——为什么选 Redisson？

[← 目录](README.md)

---

## 茶馆开场（趣味锚点）

城西「缓存茶馆」里，小白端着 `Jedis` 泡的浓茶抱怨：「我不就存个 key 吗，整这么多花样？」  
掌柜大师把茶盏一推：「你若只与 Redis 说三句话，自写命令无妨；若要让**十个服务、百个线程、一套锁与队列**都和同一片内存打交道，你就需要一位**会说 Java 的跑堂**——Redisson 就是那位跑堂。」  
门外又进来一单：**订单服务要防超卖、网关要限流、活动页要排行榜**——大师笑：「看见没？**业务一来，就不止三句话了。**」

---

## 需求落地（由浅入深）

1. **浅（项目里先喊疼）**：PRD 写「库存不能卖穿」「接口别被刷爆」「多实例要一致」——你手里若只有裸 `SET`，很快会在各服务里复制粘贴 Lua、重试、锁 key 命名，**评审会上谁也说不清边界**。  
2. **中（架构上要定调）**：哪些能力算**协调层**（锁、队列、信号量），哪些仍归 **DB 约束与幂等**；多客户端是否**共用 key 空间**；升级与 Codec 谁负责——**先画责任，再选库**。  
3. **深（落到 Redisson）**：用 `RedissonClient` 工厂拿 `RMap`/`RLock`/`RTopic` 等抽象，把拓扑与线程模型收敛到 **一份 `Config`**，让业务代码读得像**领域服务**，而不是**Redis 命令草稿纸**。

---

## 对话钩子

**小白**：师父，Redis 客户端那么多，我直接 `SET` / `GET` 拼命令不行吗？  
**大师**：行。但你会很快把「协议版本、连接拓扑、重试策略、线程安全、分布式语义」腌进业务类里——**技术债闻起来像隔夜 Redis**。Redisson 做的是另一件事：**把 Valkey/Redis 变成一组线程安全的 Java 抽象**（`RMap`、`RLock`、`RTopic`……），让你留在**领域模型**里思考。

**小白**：那我不就「绑死」在 Redisson 上了？  
**大师**：绑的是 **Redis 语义**；Redisson 是**表达层**。真怕换库，就把 **key 规范、TTL、失效模式** 写成团队契约——换客户端是换跑堂，**契约仍是账本**。

---

## Redisson 是什么（一句话 + 三点）

1. **定位**：面向 Valkey 与 Redis 的 Java 客户端 + 实时数据平台能力，见仓库 [README.md](../../README.md)。  
2. **分布式对象与服务**：不止封装命令；提供**分布式集合、锁、队列、执行器**等「拼命令很疼」的能力。  
3. **多部署形态**：单机、主从、哨兵、集群、复制、多集群、代理等，详见 [configuration.md](../configuration.md)。

## 和 Jedis / Lettuce / Spring Data Redis 怎么分工？

| 维度 | Jedis | Lettuce | Spring Data Redis | Redisson |
|------|--------|---------|-------------------|----------|
| 核心风格 | 命令式 | 异步/响应式友好 | Spring 抽象 + Template | **分布式 Java 对象与服务** |
| 典型用途 | 简单直连、脚本 | 高并发 Netty | Repository、Template | 锁、本地缓存 Map、队列、调度 |
| 学习曲线 | 低 | 中 | 中（Spring 栈内） | 中偏高（概念多，样板少） |

**选型口诀**：

- 主要是 **CRUD 缓存 + Repository** → Spring Data Redis 往往更贴。  
- 要 **可重入锁、信号量、延迟队列、分布式执行器** → Redisson 常更省命。  
- **可以共存**：同一项目里 Data 管 CRUD，Redisson 管协调——但务必 **约定 key 空间**，别两套客户端互踩。

---

## 心智模型（最重要）

1. **`RedissonClient` 是工厂**：`getMap("myMap")` 绑定逻辑名，背后是一组 **Redis key 模式** + **Lua/多命令编排**。  
2. **`R*` 不是 `HashMap`**：每次操作可能是一次或多次网络往返；**延迟与失败是常态**。  
3. **语义对齐 Redis**：Redis 事务 ≠ 分布式事务；Pub/Sub 默认不持久——后面章节会反复校准直觉。

### 反例（加深记忆）

- 以为 `RMap.size()` 和本地 Map 一样「免费」→ 大 Map 上可能是 O(N) 网络扫描，**生产慎用**。  
- 以为「我用了 Redisson 所以自动强一致」→ **除非**你在架构上显式设计（锁、版本号、DB 约束），否则没有魔法。

## 本章实验室（20 分钟）

1. 本地起 Redis 或 Valkey，用 Redisson `RBucket<String>` 写入 `"hello"`。  
2. `redis-cli` 里 `SCAN 0 MATCH *`（避免生产 `KEYS *`），观察 **key 前缀与结构**，拍照贴进团队 Wiki。  
3. **追问**：同一个 `getBucket("user:1")`，换一台应用实例连同一 Redis，读到的值是否一致？这说明了什么？

## 大师私房话（深度）

把 Redis 想成**外包的内存部**：你下的是业务订单（`put`/`lock`），对方机房负责搬箱子。订单**丢了、慢了、重复了**，你要有 **超时、重试、幂等、监控**——**可观测性不是上线后补丁，是契约的一部分**。
