# 第六章：分布式集合选型——List、Set、ZSet 与兵器谱

[← 目录](README.md)

---

## 趣味：青龙偃月刀削苹果

**小白**：排行榜我用 `RList` 行不行？  
**大师**：排行榜要 **按分数排序**，该请 **Sorted Set**；List 是 **队列语义**。兵器谱读错页，上阵先输一半。  
**小白**：那我 `List.sort()` 再塞回 Redis？大师：**「把排序外包给 Redis」和「把 Redis 当临时 ArrayList」是两码事。**

---

## 需求落地（由浅入深）

1. **浅**：运营要 **日榜周榜、Top 10、点赞数**——先问清是 **精确排序** 还是 **可接受近似**；是否 **高并发写同一 member**（热 key 预警）。  
2. **中**：选型 `RScoredSortedSet` 等承载排行；**并发更新** 用原子增量而非「读改写」；**时间窗口榜** 要想清楚 **过期与分桶**，别用一个大 ZSet 扛所有活动。  
3. **深**：大 key、全表 `ZRANGE`、复杂统计下推——该 **分片、异步聚合、或换专用分析链路**；Redisson 只是 **Redis 上的 Java 脸**，**救不了错误建模**。

---

## 选型地图

| 需求 | Redisson 方向（概念） |
|------|------------------------|
| Top N / 排行榜 | `RScoredSortedSet` 等 |
| 去重 | `RSet` |
| 有序去重 | `RSortedSet` / 按业务权衡 |
| 时间序列 / 窗口指标 | `RTimeSeries` 等（查文档版本） |
| 向量场景 | `VectorSet` 等（模块与版本敏感） |

总览：[collections.md](../data-and-services/collections.md)。

---

## 对话钩子

**小白**：我们榜单就几千人，`RList` 也能排序吧？  
**大师**：几千人时**怎么都行**；问题是 PRD 明天变 **百万级、实时刷新**——**现在偷懒的选型，会在流量曲线拐点找你讨债**。

---

## 大 key / 热 key（生死线）

- **大 key**：一次操作阻塞 Redis 单线程、拖慢主从复制 → **拆分、分片、聚合下推**。  
- **热 key**：单机 QPS 打穿 → **本地缓存、随机后缀、或架构层副本**（配合一致性要求）。  
**与 Redisson 无关，但 Redisson 也救不了错误建模。**

---

## 深度：并发更新排行榜

- `ZINCRBY` 风格更新：**原子**、简单。  
- 「先读排名再写」：**竞态**——要么 Lua，要么接受短暂不一致 + 对账。

---

## 本章实验室

迷你 leaderboard：多线程 `ZINCRBY`，定时拉 Top 10；观察 **P99** 与 **CPU**。  
**加分**：模拟 **热 key**（全员打同一个 member），讨论拆分策略。

---

## 大师私房话

**读多写少**：缓存、副本读（能接受延迟时）。**写多读少**：优先 **key 设计** 与 **批量写**，别先堆机器。
