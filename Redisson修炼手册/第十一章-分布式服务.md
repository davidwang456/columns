# 第十一章：分布式服务——Remote、Executor、Scheduler 与 Live Object

[← 目录](README.md)

---

## 趣味：用信鸽当 RPC

**小白**：我想用 Redis 当 RPC，是不是很分布式？  
**大师**：信鸽也能传书，但你不会用它传 **4K 视频**。Redisson 的 Remote/Executor **能**做轻量协调，**别**和 gRPC + 专业 MQ 比生态与可观测性。  
小白眼睛一亮：「那我们不用部署 MQ 了？」大师：**「省部署」和「省事故」往往是反义词。**

---

## 需求落地（由浅入深）

1. **浅**：团队想 **跨服务扔个后台任务、定时扫表、轻量回调**——Executor/Scheduler 看起来「开箱即用」。  
2. **中**：必须设计 **任务幂等、超时、misfire、慢任务隔离**；和 **消息队列、调度平台** 比 **可观测、死信、回放** 是否接受降级。  
3. **深**：Live Object **魔法与调试成本** 并存；默认优先 **显式 DTO + RMap**；MapReduce 教学向——**大数据请找专业引擎**。

---

## 对话钩子

**小白**：Executor 和我们自己起线程池有啥区别？  
**大师**：区别在 **多实例协调与任务投递语义**；若你连 **任务去重、监控、版本兼容** 都没想好，区别只是 **坑分布得更广**。

---

## Remote Service / Executor / Scheduler

总览：[services.md](../data-and-services/services.md)。

| 能力 | 边界思考 |
|------|-----------|
| Remote | 像远程调用；**超时、版本、序列化演进** |
| Executor | 分布式线程池味；**慢任务隔离、限流、背压** |
| Scheduler | 定时/延迟；**时钟漂移、重复执行、misfire** |

---

## Live Object

- **魔法**：对象状态在 Redis，Java 侧像本地引用。  
- **坑**：调试难、引用图复杂、版本与序列化 **团队认知成本高**。  
- **建议**：中小团队优先 **显式 DTO + RMap**；Live Object 留给 **真需要** 且 **有规范** 的团队。

---

## MapReduce

教学向了解即可；大数据计算常见 **专用引擎 + 对象存储**，Redis 不是主战场。

---

## 生产清单

- [ ] **任务幂等**：重试必然发生。  
- [ ] **超时与线程池**：Executor 队列深度监控。  
- [ ] **可观测**：任务成功率、延迟直方图、失败原因分类。

---

## 本章实验室

提交 **故意超时** 的任务，观察客户端行为；记录 **重试是否双执行**，给出 **幂等键** 设计。
