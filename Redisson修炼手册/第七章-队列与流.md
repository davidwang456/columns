# 第七章：队列与流——从「丢消息」到「至少一次」

[← 目录](README.md)

---

## 趣味：店小二的托盘

**小白**：`BRPOP` 多爽，弹出即消费！  
**大师**：托盘端到客人桌前，**小二摔了一跤**——菜没了，客人也没吃上。你要的是 **「菜从厨房到客人」的 SLA**，不是 **「离开厨房」** 那一下。  
掌柜补刀：「财务问 **订单消息谁赔**，你可别答 **Redis 很快**。」

---

## 需求落地（由浅入深）

1. **浅**：业务要 **异步下单、发通知、对账任务**——先写清：**能丢吗？能重复吗？** 再选 List 队列还是 Stream；**没写清就写代码，等于蒙眼接飞刀**。  
2. **中**：至少一次语义下，消费端必须有 **幂等键、状态机、死信与重试上限**；监控 **队列深度、pending、消费滞后**，和 **业务 SLA** 绑定。  
3. **深**：`RStream` + 消费者组：**ACK、pending reclaim、分区**；与 **第九章 Pub/Sub** 边界对照——**要堆积就别用纯 Pub/Sub**。

---

## 对话钩子

**小白**：我们就想简单点，List 队列不行吗？  
**大师**：行，只要你书面接受 **「进程死在 pop 之后」的丢失窗口**。不能接受 → **ReliableQueue / Stream** 或 **专业 MQ**，别在中间态骗自己。

List 当队列：**进程死在 `BRPOP` 之后、业务处理之中** → 消息 **蒸发**。必须分清：

- **至多一次**：可能丢，不重复。  
- **至少一次**：可能重复，尽量不丢。  
- **恰好一次**：分布式里极贵，常靠 **幂等 + 去重** 模拟。

**小白**：「恰好一次」我加分布式锁行不行？  
**大师**：锁保的是 **互斥**，不保 **消息不重复投递**。**幂等** 才是「重复来了也安全」的底裤。

## 能力谱系（语义优先）

详见 [queues.md](../data-and-services/queues.md)。

| 形态 | 直觉 |
|------|------|
| `RQueue` / `RDeque` | 简单 FIFO；理解 **丢失窗口** |
| `RBlockingQueue` | 阻塞；仍有 **处理中崩溃** 问题 |
| `ReliableQueue` | 向可靠靠拢；读清 **限制与成本** |
| `RStream` | 消费者组、**ACK**、**pending**；标配 **至少一次 + 幂等** |
| Ring Buffer / Transfer Queue | 特定场景 |

## RStream 是什么（本手册里的「介绍」）

**`RStream<K, V>`** 是 Redisson 对 **Redis Stream**（Redis 5.0+）的封装：底层仍是 `XADD`、`XREADGROUP`、`XACK` 等，在 Java 里表现为**可过期**的流对象，并支持 **同步 / 异步 / Reactive / RxJava** 四种调用方式。

- **入口**：`redisson.getStream("业务流名")`，需要自定义序列化时重载 `getStream(name, codec)`。  
- **权威说明与完整示例**（含各 API 风格）：仓库 [queues.md 的 Stream 小节](../data-and-services/queues.md#stream)。  
- **接口细节**：以当前版本 [RStream JavaDoc](https://www.javadoc.io/doc/org.redisson/redisson/latest/org/redisson/api/RStream.html) 为准；部分方法要求 **更高 Redis 版本**（如个别与 consumer / ack 相关的扩展），升级前要对表核对。

**和 List 队列一句话差别**：Stream 里消息**持久在流上**，消费进度由**消费者组**维护；List 的 `BRPOP` 弹出后，消息就从列表里**没了**，进程死在「弹出之后、ACK 等价物之前」时语义更难兜。

## 典型工作场景（何时优先考虑 RStream）

| 场景 | 为何合适 | 仍须自备 |
|------|-----------|-----------|
| **异步任务 / 事件溯源雏形** | 多条生产者 `add`，多消费者组横向扩展，`readGroup` 按组瓜分进度 | **幂等**、业务唯一键；重复投递按常态设计 |
| **可堆积的通知 / 订单流水** | 消息可积压在流里，不像普通 Pub/Sub 那样「不在线就丢」 | **长度与内存**：`MAXLEN`、修剪策略；监控流长度 |
| **失败重试与认领** | pending 列表 + `XAUTOCLAIM` / claim 类 API 做**超时消息转移** | **重试上限、死信**；避免无限 pending |
| **多租户 / 多业务线隔离** | 不同 key（不同 `getStream` 名）即不同流 | key 规范、ACL、避免单流热 key |

**不太适合硬上的情况**：海量消息、强顺序跨分片、复杂路由——更常见做法是 **专业 MQ（Kafka、RocketMQ 等）** 或 **DB + Outbox**；Redis Stream 更适合 **中等体量、延迟敏感、已与 Redis 共栈** 的队列型需求。

## 极简示意（读懂即可，细节以文档为准）

```java
RStream<String, String> stream = redisson.getStream("orders:events");
// 生产：向流追加一条（字段名-值 由 StreamAddArgs 描述）
StreamMessageId id = stream.add(StreamAddArgs.entry("type", "PAID").entry("orderId", "1001"));

// 消费组：先创建组，再 readGroup；处理成功后 ack，失败则进入 pending，需 reclaim / 重试策略
stream.createGroup(StreamCreateGroupArgs.name("bill-workers").makeStream());
// ... readGroup、ack、pendingRange、autoClaim 等见官方 queues.md 示例
```

---

## Stream：pending 与 reclaim

- **已投递未 ACK** → pending；要设计 **重试上限、死信、超时 reclaim**。  
- **反例**：只 `XREADGROUP` 不处理 pending → **幽灵积压**，磁盘与内存慢慢炸。

---

## 与 Pub/Sub 的边界（预告第九章）

Pub/Sub：**不保证离线投递**、默认无积压语义 → **别当 MQ**。要堆积与重试 → **本章队列/Stream**。

## 生产清单

- [ ] 消费逻辑 **幂等**（业务唯一键、状态机）。  
- [ ] **队列深度** 监控 + 告警 + 背压（限流生产或扩容消费）。  
- [ ] ** poison message** 策略：跳过重试、进 DLQ、人工介入。
