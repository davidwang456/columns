## 需求落地（由浅入深）

1. **浅**：跨 JVM **调个方法、扔个任务、定个时**——先看 **是否已与 Redis 强绑定**，否则专业 RPC / MQ / 调度平台可能更省长期成本。  
2. **中**：任务路径必须 **幂等、超时、监控**；Scheduler 要处理 **多实例、misfire、时钟**。  
3. **深**：Live Object **魔法与维护成本** 并存；默认可优先 **显式 DTO + RMap**（第五章），再评估 RLO。

## 生产清单（全章）

- [ ] **任务幂等**：重试必然发生。  
- [ ] **超时与线程池**：Executor / Worker **队列深度** 可观测。  
- [ ] **可观测**：成功率、延迟、失败原因；忌 **静默失败**。

## 综合实验室（约 90～120 分钟）

**环境**：同一 Redis；至少 **两个 JVM 进程**（或一台机器上两个 `main`）效果最佳——一个扮演 worker/服务端，一个扮演客户端/提交端。参考 [services.md](../data-and-services/services.md)。

### 步骤

1. **Remote Service 超时**  
   - 服务端 `register` 一个接口方法，内部 `Thread.sleep` **大于** `RemoteInvocationOptions` 的 ack/执行超时。  
   - 客户端调用，捕获异常类型与堆栈，记录 **是否可区分「超时 / 无 worker / 序列化失败」**。  
   - 对照文档调整超时后，确认 **调用成功**。

2. **Executor 故意失败与重试**  
   - 提交 `Callable`：第一次执行抛异常或返回错误码，第二次成功（用静态计数器，**仅实验**）。  
   - 若配置了 **taskRetryInterval** 等，观察 **是否执行两次**；在 Redis 或日志中统计 **副作用次数**。  
   - 写出：**若业务不可重试，幂等键应放在何处**（参数、Redis key、DB 唯一约束）。

3. **Scheduled（可选）**  
   - `schedule` 延迟 30s 的任务，**kill worker 进程** 再拉起，观察 **是否 misfire / 是否重复**（与文档语义对照）。

4. **Live Object（可选）**  
   - 两进程 `attach` 同一 `@RId`，交替改字段，`redis-cli HGETALL` 对照 **字段是否即时反映**。

### 验证标准

- 实验 1：**有一份** 超时异常样本 + 修正后的成功调用日志。  
- 实验 2：**明确结论**「重试下是否会双执行」+ **幂等设计** 一段话（可贴 Wiki）。  
- 实验 3～4：至少完成其一，并有 **可复述的语义结论**。

### 记录建议

- 表格：| 组件 | 超时配置项 | 客户端表现 | 是否幂等 |  
- **告警建议**：Remote/Executor **队列深度、失败率** 初稿指标名。
