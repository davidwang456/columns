## 需求落地（由浅入深）

1. **浅**：先分清要的是 **顺序列表**、**唯一集合**，还是 **带分排序**——三者对应 List / Set / ZSet，**选错结构比调参更痛**。  
2. **中**：排行榜用 **`addScore` 原子加分**；避免大集合上的 **全量遍历**；活动榜按 **时间分 key**，减轻单 key 压力。  
3. **深**：大 key、热 key 需 **拆分、缓存、或异步聚合**；Redisson 只是封装，**错误建模救不了**。

---

## 大 key / 热 key（全章必读）

- **大 key**：单次操作阻塞 Redis 单线程、拖慢复制 → **拆分、分片、聚合下推**。  
- **热 key**：单机 QPS 打穿 → **本地缓存、随机后缀、或架构副本**（在一致性允许时）。  

---

## 综合实验室（约 60 分钟）

**环境**：单 Redis；JUnit 或 `main` 可多线程；`redis-cli INFO cpu` / `LATENCY DOCTOR`（可选）。

### 步骤

1. **ZSet 并发加分**  
   - `getScoredSortedSet("lab:ch6:board")`，10 线程各循环 1000 次对 **不同 member** `addScore(member, 1)`。  
   - 结束后 `entryRangeReversed(0, 9)` 打印 Top10，核对总分与次数一致。

2. **P99 粗测**  
   - 在线程内记录每次 `addScore` 耗时（`nanoTime`），算 **99 分位**（可导出 CSV 用表格算）。  
   - 同时 `redis-cli INFO` 看 **used_cpu_sys** 前后差（粗略）。

3. **热 member 加压**  
   - 10 线程 **只对同一 member** `addScore("hot", 1)` 共 1 万次，重复步骤 2。  
   - 对比 P99 与 CPU 与步骤 1 的差异，**截图或记数字**。

4. **（加分）分榜讨论**  
   - 书面回答：若步骤 3 延迟不可接受，**拆 key / 随机后缀 / 本地聚合** 你选哪种，为什么？

### 验证标准

- TopN 结果与 **数学期望**一致（无丢更新）。  
- 能展示：**热 member 时延迟或 CPU 变差** 的对比数据。

### 记录建议

- 一页纸结论：**「本业务排行榜是否会出现热 member」** 及监测指标（QPS、P99）。
