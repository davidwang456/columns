## 综合实验室（贯穿四篇，约 2～3 小时，可拆天）

**环境**：Redis 单节点即可；**两个 JVM 进程**（或两台机器）共连同一 Redis；`redis-cli`；可选 **JMeter 线程组** 双实例压限流。

### 步骤

1. **RBucket + TTL + 多实例**  
   - 进程 1：`getBucket("lab:ch5:flag").set("ON", 5, TimeUnit.MINUTES)`。  
   - 进程 2：仅 `get()`，应得到 `ON`；等待 TTL 过期后再 `get()`，应为 `null`（以实际为准）。  
   - `redis-cli` 对匹配 key 执行 `TTL`，与 Java 侧设定对照。

2. **RMap 与 Hash 结构**  
   - `RMap<String,String> m = getMap("lab:ch5:order:1")`，`put` 至少 3 个 field。  
   - `redis-cli` 使用 `TYPE` + `HGETALL`（key 名用 `SCAN` 查找，**禁止生产 `KEYS *`**），截图字段与 Java 一致。

3. **RBloomFilter 假阳性体感**  
   - `tryInit(1000, 0.03)`，循环 `add` 5000 个随机字符串；再随机生成 **未 add** 的串若干次 `contains`。  
   - 记录：**「不存在」为 false 的比例** 与 **「存在」为 true 但未 add 的次数**（假阳性计数）。

4. **RRateLimiter OVERALL**  
   - 两进程共享 `getRateLimiter("lab:ch5:rl")`，`trySetRate(OVERALL, 100, Duration.ofMinutes(1))`（仅首次成功）。  
   - 各进程循环 `tryAcquire` 打满 1 分钟，合计成功次数应 **接近 100**（允许误差，记录环境说明）。

### 验证标准

- 能演示：**多 JVM 看到同一 Bucket / Map / Limiter 状态**。  
- 能口头解释：**Bloom 的 false positive 与 tryInit 参数关系**；**OVERALL 与多实例总配额**。

### 记录建议

- 一页纸：**四篇 API 在本实验中的 key 前缀规范 `lab:ch5:*`**，便于以后清理测试数据。
