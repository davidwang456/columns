## 需求落地（由浅入深）

1. **浅**：上线前检查清单：**连得上、shutdown 有、密码/TLS 对**——这是 **科目一**。  
2. **中**：把 **客户端超时、池水位、重试、反序列化失败** 与 **Redis slowlog、内存、复制积压** 放进同一套大盘；故障演练 **拔节点、拔从、拉长延迟**。  
3. **深**：SRE 问「**最坏多糟**」——你要能指着面板答：**P99、错误预算、锁失败时的降级**。可观测性 **写进架构评审**，不是上线后补丁。

---

## 对话钩子

**小白**：指标要采多少才够？  
**大师**：先保证 **能回答三个问题**：谁慢了？是 Redis 还是客户端？最近发布动了吗？**采一万个没人看的指标，不如五个能定责的。**

## 客户端可观测

- **日志**：超时、重连、反序列化失败 → **结构化 + traceId**。  
- **指标**：QPS、P50/P99、连接池 active/idle、重试次数。  
- **Tracing**：把 Redis 段纳入全链路（代理或客户端 span，视栈而定）。

参考：[observability.md](../observability.md)、[fault-tolerance-and-recovery.md](../fault-tolerance-and-recovery.md)。

## 本章实验室（约 45～90 分钟）

**环境**：**测试** Redis（勿在生产执行破坏性命令）；应用已接 Redisson，日志 **带时间戳与 traceId**（若尚无 traceId，至少统一 `yyyy-MM-dd HH:mm:ss.SSS`）。

### 步骤

1. **制造慢查询（仅测试）**  
   - 任选其一（安全范围内）：Lua 里 **大循环**、`DEBUG SLEEP`（若版本允许且环境隔离）、或 **超大 payload 连续 SET**。  
   - 同时从应用发起 **一批超时敏感调用**（短 timeout 的 `getBucket` 等）。

2. **对齐时间线**  
   - `redis-cli SLOWLOG GET 128`，记下 **慢命令 id、unix 时间戳、耗时**。  
   - 从应用日志截取 **同一时间段** 的 **TimeoutException / 重试** 行。  
   - 换算时区，画 **一条时间轴**：T0 慢命令开始 → T1 应用报错 → T2 重连（若有）。

3. **客户端池与命令数**  
   - 记录压测前后 **`INFO clients`** 或应用侧 **连接池 active**（若有指标）。  
   - 回答：**变慢主要来自 Redis CPU、网络，还是客户端等待池？**

4. **故障注入（桌面演练亦可）**  
   - 对照上文「故障注入清单」选 **1 项**（如停从节点、iptables 延迟），写出 **预期面板变化** 与 **第一步操作**（不必真做若环境不允许）。

### 验证标准

- **一张图或表**：slowlog 条目 ↔ 应用日志行（至少 **1 条** 可对应关系）。  
- 能回答 **三个问题** 之一用数据支撑：谁慢了？Redis 还是客户端？与发布相关吗？

### 记录建议

- 将本实验产出 **附在团队 Runbook**：「Redis 慢」章节的第一页。  
- 列出 **待补指标**（持锁时间、池水位、重试次数）对照第十四章 Checklist 打勾。
