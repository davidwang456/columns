# 第二章：配置篇——拓扑决定 Redisson 的「脾气」

[← 目录](README.md)

---

## 类比（趣味）

大师指着地图：「单机 Redis 是**独栋**；哨兵是**物业 + 备用钥匙**；集群是**分片楼盘**——你导航 App 选错模式，就会天天 **MOVED/ASK 绕路**。」  
小白指着云控制台：「可页面上都叫 Redis 啊？」大师叹气：**名字一样，户口本三页。**

---

## 需求落地（由浅入深）

1. **浅**：上线前运维丢来一串地址——你要回答的是：**这是单机入口、哨兵列表，还是集群 seed？** 抄错 `useSingleServer` / `useClusterServers`，症状往往是 **偶发 MOVED** 或 **永远连错节点**。  
2. **中**：跨 AZ 延迟、故障 **RTO**、读从是否接受 **复制延迟**；这些决定 **超时、重试、是否读写分离**，不是「调大一点超时」能糊弄过去的。  
3. **深**：在 `Config` 里选对 `use*`，让客户端与 **槽路由、故障转移** 对齐；禁止 **Jedis 直连某分片 + Redisson 集群** 混写同一批 key——否则会出现「写 A 读 B」的灵异现场。

---

## 对话钩子

**小白**：本地单节点好好的，上集群谜之超时、偶发读不到刚写的值……  
**大师**：先问运维：**读写落在什么拓扑？** 再问自己：**是不是把集群当单机用、或混了多个客户端直连？**

**小白**：我把 `connectionPoolSize` 拉满，是不是就更稳？  
**大师**：对 Redis 端是 **压力**；对本机是 **FD 与线程**。池子要和 **QPS、命令耗时、实例数** 一起算——**灌满池子不等于灌满吞吐量**。

## 模式速查表

| 模式 | API（概念） | 典型场景 | 直觉 |
|------|-------------|----------|------|
| Single | `useSingleServer()` | 开发、可接受单点 | 一个入口 |
| Master-Slave | `useMasterSlaveServers()` | 读多写少 | 副本延迟要承认 |
| Sentinel | `useSentinelServers()` | 故障自动转移 | master 名 + 哨兵列表 |
| Cluster | `useClusterServers()` | 水平扩展 | 槽、迁移、多 master |
| Replicated | `useReplicatedServers()` | 云厂商复制拓扑 | 与 Sentinel 勿混淆 |
| Multi-* | 对应 `use...` | 跨集群/多哨兵 | 先读懂运维模型 |
| Proxy | `useProxy()` | 统一代理入口 | 把代理当服务端 |

**权威字段与示例**：[configuration.md](../configuration.md)。

---

## 调参维度（不讲死数字）

| 旋钮 | 思考 |
|------|------|
| 命令超时 | 与业务 P99、Redis `slowlog` 联动 |
| 连接超时 | 网络抖动、跨 AZ 延迟 |
| 连接池 | 过小排队、过大 FD 与 Redis 连接数压力 |
| 重试 | 与 **幂等** 一起设计，避免重试放大写压力 |
| SSL | `rediss://` / `valkeys://`、证书轮换 |
| DNS | K8s Service 变更时，是否需 **重建客户端** |

---

## 深度：MOVED / ASK

集群扩缩容、槽迁移时，服务端会返回 **MOVED**（永久迁移）或 **ASK**（临时导入）。成熟客户端应自动跟路由。  
**反模式**：业务代码里混用 **Jedis 直连某节点 + Redisson 集群模式** 操作同一批 key → 易出现「写 A 读 B」的幽灵 bug。

---

## 实战片段：Cluster 地址形态（示意）

```java
Config config = new Config();
config.useClusterServers()
      .addNodeAddress("redis://10.0.0.1:6379", "redis://10.0.0.2:6379");
RedissonClient redisson = Redisson.create(config);
```

真实环境以运维交付的 **discovery 方式** 为准（DNS、K8s Headless Service 等）。

## 本章实验室（约 60～90 分钟，可分两次做）

**环境**：Docker Compose 或运维提供的 **单机**、**Cluster 或 Sentinel** 三套环境中至少两套；本机可跑同一 `main` 仅改 `Config`。

### 步骤

1. **同一业务代码，两套 Config**  
   - 分支 A：`useSingleServer()`，向 `getBucket("lab:topo:a").set("1")`。  
   - 分支 B：`useClusterServers()`（或 Sentinel），**相同逻辑 key** 再 `set("2")` 或 `get`。  
   - 记录：启动日志中是否出现 **MOVED/ASK**（仅 Cluster）、或 Sentinel 的 **master 切换日志**。

2. **Cluster 槽位（若适用）**  
   - 在 `redis-cli -c` 下对 `lab:topo:slotkey` 执行 `CLUSTER KEYSLOT lab:topo:slotkey`（命令以集群版为准）。  
   - 将 key 名 **改一两个字符** 再查槽位，观察是否变化，理解 **分片与 key 有关**。

3. **故障或延迟**  
   - 在测试 Cluster：**停掉一个从节点** 或 `tc`/`iptables` 模拟 Redis 端 **200ms RTT**，对同一操作打 **P99 延迟**（简单循环计次即可）。  
   - 记录：客户端是否 **自动重试/超时**，业务侧是否需 **调大 commandTimeout**（仅实验结论，不写死数字）。

### 验证标准

- 能口述：**单机 / Sentinel / Cluster** 各解决什么问题。  
- Cluster 下能解释：**为何错误拓扑配置会出现 MOVED 或读不到刚写的值**（与槽与路由有关）。

### 记录建议

- 输出一页：**本服务 Redis 拓扑 + 对应 Redisson `use*API` 名称**，评审签字。
