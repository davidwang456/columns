# 第八章（上）：分布式锁——`RLock`、租约与看门狗

[← 目录](README.md)

---

## 趣味：客栈包间与钥匙

**大师**：分布式锁不是「万能门神」，是 **租了一间包间**：到点不交租，别人就能进——**租约（lease）** 就是时间。  
**小白**：那我一直续租？  
**大师**：可以，那叫 **看门狗（watch dog）**——但续租的前提是 **你还活着且业务仍在合理执行**；不是 **死锁了还续命**。

---

## 需求落地（由浅入深）

1. **浅**：产品一句「同一订单不能并发改」——落到工程是：**哪段代码互斥**、**锁 key 怎么拼**、**失败是重试还是报错**。  
2. **中**：临界区要 **短、可预测、无长 RPC**；锁外完成 **幂等与校验**；主从切换、时钟漂移下 **锁不是司法公证**，**DB 唯一约束** 仍要就位。  
3. **深**：`lock()` 与 `tryLock(wait, lease, unit)` 差在 **是否带固定租约、是否依赖看门狗续期**；持锁时长要 **打指标**——见第十四章。

---

## 对话钩子

**小白**：我 `lock` 了还超卖！  
**大师**：三问：**锁粒度对吗？** **临界区里有没有远程调用拖长尾？** **解锁是否只删自己的锁（token/fencing）？**

**小白**：我 `unlock` 写在 `if` 里，少执行了一次……  
**大师**：**`finally` + `isHeldByCurrentThread`** 不是洁癖，是 **防你未来某次改需求时手抖**。锁题一半挂在 **模板代码** 上。

---

## `RLock` 基础

- **可重入**：同线程多次 `lock` 须配对 `unlock`。  
- **`lock()`**：未指定 `leaseTime` 时，Redisson **可**通过内部机制在默认租约内**续期**（以当前版本文档为准）。  
- **`tryLock(wait, lease, unit)`**：指定 `leaseTime` 后通常 **不续期**——到期自动释放，适合 **「N 秒内必完成」** 的任务。

权威：[locks-and-synchronizers.md](../data-and-services/locks-and-synchronizers.md)。

---

## 典型反模式（深度）

1. **锁内 HTTP/RPC**：超时不可控 → **长持锁** → 吞吐雪崩。  
2. **锁粒度太粗**：「全局一把锁」→ 分布式系统 **退化成串行机**。  
3. **只加锁不做幂等**：锁边界与业务不一致、或主从切换场景下，仍需 **DB 唯一约束 / 版本号** 兜底。  
4. **误解「有了锁就强一致」**：锁解决 **互斥**，不解决 **可见性以外的所有业务 bug**。

---

## 实战片段（示意）

```java
RLock lock = redisson.getLock("order:lock:" + orderId);
if (lock.tryLock(3, 10, TimeUnit.SECONDS)) {
    try {
        // 临界区：短、无远程、可预测
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

**注意**：`isHeldByCurrentThread` 等 API 以版本为准；**unlock 必须在 finally**，且避免 **非持有者释放**。

## 本章实验室（上）（约 60～90 分钟）

**环境**：单 Redis；同一 `RedissonClient`；准备两个 JVM 进程或同一进程内两个线程均可（进程间更贴近生产）。

### 步骤

1. **固定租约穿透**  
   - 线程 A：`tryLock(1, 5, TimeUnit.SECONDS)`，进入临界区后 `Thread.sleep(8000)`。  
   - 线程 B：在 A sleep 期间 `tryLock(1, 5, TimeUnit.SECONDS)`，记录 **是否在约 5s 后拿到锁**。  
   - 结论写在实验记录：**带 lease 时到期释放，长临界区会导致互斥失效**。

2. **无 lease 的 `lock()`（看门狗语义）**  
   - 线程 A：`lock()`，临界区 `sleep` **短于** 默认续期周期（例如 3s，以你环境为准）。  
   - 观察日志：**锁是否一直被 A 持有**、B 是否阻塞到 A `unlock`。  
   - 再试：A 临界区故意 **极长**（若安全可试）或查文档说明续期上限——记录 **「续期不是无限保命」** 的理解要点。

3. **`finally` + 持有者校验**  
   - 写标准模板：`try { ... } finally { if (lock.isHeldByCurrentThread()) lock.unlock(); }`。  
   - 故意制造 **`tryLock` 失败仍走 finally** 的路径，确认 **不会 NPE / 非法解锁**（若 API 行为与版本有关，注明版本）。

4. **持锁时长指标（可选）**  
   - 在临界区前后打时间戳，输出 **持锁毫秒数** 分布（至少 20 次加锁），记下 **P50/P95** 量级。

### 验证标准

- 能 **复现**「lease 过短 + 长临界区 → 另一线程进入临界区」。  
- 能口述：**`lock()` 与 `tryLock(wait, lease, unit)`** 在续期上的差异。  
- 代码路径上 **unlock 仅在 finally 且带 isHeldByCurrentThread**（或等价安全写法）。

### 记录建议

- 一页纸：**订单类业务的 key 命名规范** + **推荐 lease 量级**（与临界区实测挂钩）。  
- **追问**（书面）：主从切换、节点宕机时锁语义——与运维 **拓扑** 对齐结论（勿拍脑袋）。
