# 第五章（分篇四）：`RRateLimiter`——多实例共享配额

[← 第五章导览](05-分布式对象基础.md)｜[目录](README.md)

---

## 1. 项目背景

开放平台要对「**每个调用方 + 每个 API**」限制 **每秒请求数**，避免某一大客户把线程池打满。单机 `Guava RateLimiter` **无法跨 JVM 共享**；网关层限流又拆不到 **细粒度业务规则**。  
此时在 Redis 侧用 Redisson 的 **`RRateLimiter`**，可在 **多应用实例** 间共享 **同一组 permit**，实现 **分布式限流**（仍建议与 **网关/WAF** 分层配合，而非单点依赖应用层）。

---

## 2. 项目设计（大师 × 小白）

**小白**：每个实例 new 一个限流器，为啥说不准？  
**大师**：每台 JVM 各算各的配额，**四台机器就变成 4 倍流量**。`RRateLimiter` 把状态放在 **Redis**，全集群看见 **同一把尺子**。

**小白**：我限流设 100 QPS，业务高峰全 429？  
**大师**：先分清 **保护系统** 还是 **合同 SLA**；再配 **突发、预热、分层**（网关粗、应用细）。限流是 **产品体验与稳定性** 的折中，不是调一个 magic number。

---

## 3. 项目实战（主代码片段）

```java
import org.redisson.api.RRateLimiter;
import org.redisson.api.RateType;
import org.redisson.api.RedissonClient;

import java.time.Duration;

RedissonClient redisson = /* ... */;

// key 建议包含租户/应用/API，避免全局互撞
RRateLimiter limiter = redisson.getRateLimiter("ratelimit:app:pay-api:partner:" + partnerId);

// 全集群共享总配额：例如每 1 分钟 1000 次（仅首次设置成功，具体语义见 JavaDoc）
limiter.trySetRate(RateType.OVERALL, 1000, Duration.ofMinutes(1));

if (!limiter.tryAcquire()) {
    response.setStatus(429);
    response.setHeader("Retry-After", "60");
    return;
}
// 执行业务逻辑
```

**`RateType` 取舍（简述）**：`OVERALL` 表示 **所有使用该限流器 key 的实例共享** 总额；`PER_CLIENT` 与 **同一 Redisson 客户端实例** 相关（见枚举注释）。上线前用压测验证 **是否符合你对「每应用 / 每集群 / 每用户」的预期**。

## 4. 项目总结

| 维度 | 说明 |
|------|------|
| **优点** | **多实例一致**；API 接近 `Semaphore`/`RateLimiter` 直觉；可与 **指标、429、Retry-After** 打通。 |
| **缺点** | 依赖 **Redis 可用性**；极端故障时要定义 **fail-open 还是 fail-close**；非网关层，**恶意流量**仍应先在外层挡。 |
| **适用场景** | API 配额、内部服务互相调用节流、活动抢券前的 **软限流**（硬防刷还需业务与风控）。 |
| **注意事项** | **`trySetRate` vs `setRate`** 语义不同（是否覆盖已有配置）；**key 爆炸**（每用户一个 limiter）要评估 Redis key 数量与内存。 |
| **常见踩坑** | 误用 `PER_CLIENT` 却以为 **全集群限一次**；限流 key 无 **租户/环境前缀** 导致测试冲生产；无 **监控**（被拒次数、剩余 permit 估算）导致排障盲飞。 |

## 本章实验室（约 40 分钟）

**环境**：单 Redis；两个 JVM 或两个线程同时抢配额。

### 步骤

1. 共享 `getRateLimiter("lab:rl:overall")`，`trySetRate(RateType.OVERALL, 50, Duration.ofMinutes(1))`。  
2. 进程 A、B 各循环 **60 秒**：仅统计 `tryAcquire()==true` 的次数并打印。  
3. 两进程 **合计** 成功次数应 **接近 50**（记录机器时钟误差说明）。  
4. 换用 **`RateType.PER_CLIENT`**（若适用你的理解），两进程各跑 60 秒，观察 **合计是否约为 100**，并与文档中 **PER_CLIENT 语义**对照。

### 验证标准

- 能用数据说明：**OVERALL 是集群总桶还是按客户端分桶**（以你实验与文档为准）。  
- 日志中保留：**每进程成功/失败次数**。

### 记录建议

- 团队结论写清：**本服务限流 key 命名规则 `ratelimit:{tenant}:{api}`**。
