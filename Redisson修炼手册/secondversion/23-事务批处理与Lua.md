# 第十章：事务、批处理与 Lua——局部原子与全局幻觉

[← 目录](README.md)

---

## 趣味：算盘 vs 银行核心

**小白**：我开了 Redisson 事务，Redis + MySQL 一起 ACID 了吧？  
**大师**：醒醒。Redis 事务是 **命令打包 + 乐观式冲突检测** 这一路，不是 **2PC**。跨资源一致性要靠 **Outbox、幂等、补偿、对账**——**江湖规矩**和 **银监局规矩**别混。  
账房先生探头：「你们 Redis 里减了库存，MySQL 没写上——**谁对账？**」

---

## 需求落地（由浅入深）

1. **浅**：接口要 **一次改多个 key**（余额 + 流水）——先想是 **减少 RTT**（Pipeline）还是 **原子一致**（Lua/事务），**两者不是同一张药方**。  
2. **中**：Redis 事务失败要 **重试策略**；与 DB 双写要有 **顺序与补偿**（Outbox/Saga），禁止 **「先缓存后库」拍脑袋**。  
3. **深**：Lua 在 **单实例** 内原子执行；**跨集群、跨系统** 没有免费分布式事务——**把「最终一致」写进设计文档**，比把 `MULTI` 写进代码更难。

---

## 对话钩子

**小白**：Lua 里能调 HTTP 吗？  
**大师**：能写不代表该写——**阻塞单线程**等于给全站 Redis **拔网线**。脚本要 **短、纯内存、可预测**。

---

## Redisson Transactions API

见 [transactions.md](../transactions.md)。

- **WATCH / MULTI / EXEC 思维**：竞争 key 多时要接受 **失败重试**。  
- **与 DB 同事务**：常见反模式是 **先写 Redis 再写 DB** 或反过来，中间崩溃 → **双写不一致**——用 **事务消息 / Outbox** 或 **明确最终一致**。

---

## Pipelining

[../pipelining.md](../pipelining.md)：减少 RTT；**不等于原子**。部分失败时，**哪些成功哪些失败** 要可读日志。

---

## Lua / 服务端脚本

[../server-side-scripting.md](../server-side-scripting.md)：

- **优点**：多步 **在服务端原子执行**（单实例语义内）。  
- **禁区**：脚本里 **慢逻辑、大循环、外部 IO** → 堵死 Redis **单线程**。  
- **深度**：Lua 原子 ≠ 跨 **多个 Redis 集群** 的分布式事务。

---

## 实战对照表

| 目标 | 手段 |
|------|------|
| 减 RTT | Pipeline |
| 多命令原子（相关 key） | Lua 或事务（注意限制） |
| 跨 Redis + DB | Outbox / Saga / 对账 |

## 本章实验室（约 60～90 分钟）

**环境**：单 Redis；两个 key：`lab:tx:a`、`lab:tx:b`，初值各 **1000**；`RBucket<Long>` 或 `getBucket` + 长整型均可。

### 步骤

1. **非原子转账（故意错）**  
   - 循环 5000 次：线程随机执行「A 减 1、B 加 1」两步 **无事务无锁**，两步之间 `Thread.yield()` 或短 sleep。  
   - 多线程（如 8 线程）并发跑至总转账次数达标。  
   - 验证：`A + B` **不等于** 2000，或出现 **负余额**；统计 **违规样本数** 或最终 `A,B` 与 2000 的差值。

2. **Redis 事务（Redisson Transaction API）**  
   - 用文档 [transactions.md](../transactions.md) 中方式，将「读 A、写 A、读 B、写 B」包进事务（注意 **WATCH 冲突重试**）。  
   - 同样并发压测，记录 **EXEC 失败重试次数** 与 **最终 A+B**。

3. **Lua 脚本原子转账**  
   - 编写服务端脚本：`A>=amount` 则 `A-=amount; B+=amount` 否则返回错误；`redisson.getScript().eval` 调用。  
   - 同并发模型下跑，验证：**A+B 恒为 2000**，且 **无负余额**。

4. **Pipeline 对照（非原子但快）**  
   - 用 [pipelining.md](../pipelining.md) 批量发送多条独立命令，观察 **RTT 下降**；同时说明 **为何仍可能出现中间状态被其他客户端读到**。

### 验证标准

- 实验 1：**能复现** 不一致；实验 3：**数值守恒** 与业务约束满足。  
- 能口述：**Pipeline / 事务 / Lua** 三者分别解决的是 **延迟** 还是 **原子性**。

### 记录建议

- 表格：| 方案 | A+B 最终 | 负余额出现 | 重试/失败次数 | QPS 量级 |  
- 一句话设计原则：**跨 Redis 与 DB 的转账** 在本系统中用 **Outbox / 对账** 的落点。
