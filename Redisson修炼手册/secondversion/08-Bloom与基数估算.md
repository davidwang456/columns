# 第五章（分篇三）：`RBloomFilter` 与基数工具——「可能在」与「大约有多少」

[← 第五章导览](05-分布式对象基础.md)｜[目录](README.md)

---

## 1. 项目背景

风控要在 **登录接口** 前快速判断：「这个设备 ID **是否从未见过**？」若从未见过，可走一道额外校验；若**见过**，则走快速通道。  
精确集合（数据库或 Set）在 **亿级** 体量下存储与查询成本极高；**布隆过滤器**用少量位图换 **O(1) 量级的判断**：  
- **`contains == false`**：**一定不存在**（可放心走「新设备」分支）。  
- **`contains == true`**：**可能存在**（需二次确认，不能当作铁证）。  

Redisson 提供 **`RBloomFilter<T>`**（基于 Highway 128-bit hash，见接口注释）。另有 **`RHyperLogLog`** 用于 **UV 估算**（非本章代码重点，总结中对比）。

---

## 2. 项目设计（大师 × 小白）

**小白**：布隆说「存在」我就当存在，行不行？  
**大师**：**不行。** 那会误伤：假阳性（false positive）是设计内代价。正确姿势是：**「不存在」当强结论，「存在」当弱提示**，后面接 **DB / 精确 Set** 或业务规则。

**小白**：我布隆当「精确去重」做报表？  
**大师**：报表对不上账时，别怪 Redis。**布隆不能删元素**（经典实现下），也不能给你「集合里到底有几个」的精确答案——要数 UV，看 **HyperLogLog** 那条路。

## 3. 项目实战（主代码片段）

```java
import org.redisson.api.RBloomFilter;
import org.redisson.api.RedissonClient;

RedissonClient redisson = /* ... */;

RBloomFilter<String> seenDevice = redisson.getBloomFilter("risk:device:seen");
// 首次按预期插入量与可接受假阳性率初始化；已初始化则返回 false
boolean fresh = seenDevice.tryInit(1_000_000L, 0.01);

String deviceId = request.getHeader("X-Device-Id");
if (!seenDevice.contains(deviceId)) {
    // 高置信：从未记录过（在假阳性率约束下）
    runExtraChallenge();
}
seenDevice.add(deviceId); // 记录本次见过；若已存在则 add 返回 false
```

**说明**：`tryInit` 的参数需按业务 **峰值插入量** 与可接受的 **误判率** 评估；初始化后若要改参数，通常需 **新 key 迁移**，不可想当然原地改。

## 4. 项目总结

| 维度 | 说明 |
|------|------|
| **优点** | **省空间**、判断极快；适合 **前置过滤**、降低下游压力。 |
| **缺点** | **假阳性**；经典场景 **不支持删除**（若需删除需换方案如布谷鸟过滤器等，以业务与版本为准）；参数选错则误判率或空间失控。 |
| **适用场景** | 爬虫/风控粗筛、缓存穿透防护的「空值洞」前置、去重第一道闸。 |
| **注意事项** | 与 **第四章 Codec** 一致：`T` 的相等与 hash 要稳定；多实例共享同一 filter name 即 **同一份** 位图。 |
| **常见踩坑** | 把 **`contains(true)` 当司法证据**；**低估插入量**导致误判飙升；需要 **精确去重/对账** 仍只用布隆。 |

**关联**：精确基数 **估算** 用 **`RHyperLogLog`**（「大约多少 UV」）；与布隆解决的是不同问题。详见 [objects.md](../data-and-services/objects.md) 等官方文档。

## 本章实验室（约 30 分钟）

**环境**：单 Redis；准备随机字符串生成（`UUID` 即可）。

### 步骤

1. `getBloomFilter("lab:bloom:uv")`，`tryInit(5000, 0.01)`。  
2. 循环 `add` 恰好 **5000** 个不同字符串；再取 **1000 个从未 add 的** 串执行 `contains`，统计 **true 的次数**（假阳性样本）。  
3. 对 **已知已 add** 的串抽 100 个 `contains`，应 **全为 true**（在实现语义下）。  
4. （可选）将 `tryInit` 的第一参数改为 **500** 但同样 add 5000 个，重复步骤 2，对比假阳性 **是否明显增加**。

### 验证标准

- 能填表：**预期插入量 / 实际插入量 / 观测假阳性次数**。  
- 能结论：**不能把 `contains==true` 当作司法级存在证明**。
