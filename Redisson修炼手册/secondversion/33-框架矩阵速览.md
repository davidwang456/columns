## 需求落地（由浅入深）

1. **浅**：非 Spring 栈先找 **`redisson-*` 扩展**，目标仍是 **一个（或多个命名）`RedissonClient` + 正确 shutdown**。  
2. **中**：**框架大版本 ↔ Redisson artifact 后缀** 对齐 [dependency-list.md](../dependency-list.md) 或 BOM；Hibernate/MyBatis **勿混用两套缓存实现写同一逻辑命名空间**。  
3. **深**：GraalVM Native：**反射、资源、Codec** 按各扩展文档配置；CI 加 **Testcontainers 冒烟**。

---

## 原则（全章）

1. **先跑通最小 `RedissonClient`**，再叠 Cache / Session。  
2. **版本矩阵**手写进项目 Wiki，随升级 **回归**。  
3. **Native / AOT**：按框架扩展文档 **逐项勾选**。

---

## 综合实验室（约 60～120 分钟，按栈裁剪）

**环境**：团队 **真实目标栈**（Quarkus / Micronaut / Helidon / Hibernate / MyBatis / Tomcat 之一）；**Testcontainers** 或 docker-compose Redis；CI 可跑 **一条** 集成测试。

### 步骤

1. **脚手架最小 Client**  
   - 按对应分篇与 [dependency-list.md](../dependency-list.md) 引入 **正确 artifact 后缀**。  
   - 启动后打印：`Redisson` **版本字符串**、`Config` 的 **连接模式**（single/sentinel/cluster 之一）。

2. **冒烟用例**  
   - 单测或 `@QuarkusTest` / Micronaut test：`getBucket("lab:stack:smoke").set("ok")` → `get` 断言。  
   - Redis 容器 **重启一次**，重复断言（观察 **重连** 是否自动恢复，记录日志关键词）。

3. **CI 挂钩**  
   - 将上述测试纳入 **默认 pipeline**（PR 即跑）；失败时 artifact 附上 **完整依赖树** 片段（`mvn dependency:tree` 或 Gradle 等价）。

4. **Native / AOT（若适用）**  
   - 对照 [microservices-integration.md](../microservices-integration.md) 与框架扩展文档，勾选 **反射、资源、Codec** 配置项；至少 **一次** `native-image` 或框架等价构建试跑（可仅文档记录卡点）。

### 验证标准

- 日志或测试输出中可看到：**版本 + 模式** 两行信息。  
- CI **绿**；本地与 CI **同一 Redis 镜像版本**（写在 README）。  
- 文档：**artifact 后缀 ↔ 框架大版本** 手抄表更新到团队 Wiki。

### 记录建议

- **数据 Ownership**：本栈下 **缓存 / Session / 锁** 各用哪套 API，**禁止混写** 的 key 前缀一条。
