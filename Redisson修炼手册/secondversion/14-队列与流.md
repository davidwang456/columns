## 需求落地（由浅入深）

1. **浅**：先写清 **能丢吗、能重复吗**（至多一次 / 至少一次 / 用幂等模拟恰好一次），再选 API。  
2. **中**：至少一次路径上必须有 **幂等键、状态机、死信与重试上限**；监控 **队列深度、pending、消费滞后**。  
3. **深**：`RStream` / `RReliableQueue` 的 **配置与 Redis 版本** 对齐；大流量仍可能需 **专业 MQ** 或 **Outbox**。

---

## 语义速记

- **至多一次**：可能丢，不重复。  
- **至少一次**：可能重复，尽量不丢。  
- **恰好一次**：分布式里极贵，常靠 **幂等 + 去重**。

---

## 生产清单（全章）

- [ ] 消费逻辑 **幂等**（业务唯一键、状态机）。  
- [ ] **队列深度** 监控 + 告警 + 背压。  
- [ ] **毒消息** 策略：跳过重试、进 DLQ、人工介入。

## 综合实验室（约 90～120 分钟，可分两次）

**环境**：单 Redis；两个终端或两台机器跑消费者；准备 `kill -9` **仅测试进程**。

### 步骤（任选一条链路做深）

**A. `RBlockingQueue`**

1. 生产者 `offer` 消息 ID 1～20。  
2. 消费者 A `poll(10s)` 循环消费，**每取一条 sleep 2s** 模拟慢处理。  
3. 消费到第 5 条时对消费者 **`kill -9`**。  
4. 重启消费者 B，继续 `poll`，记录：**是否丢消息、是否重复、从哪条开始**（对照 Redis List 长度与业务日志）。

**B. `RStream` + Group**

1. `XADD` 等价：`stream.add(...)` 写入 10 条。  
2. `createGroup` + `readGroup`，消费若干条后 **不 ACK** 即杀进程。  
3. 重启后 `pendingRange` 查看 pending，再 `ack` 或按文档 reclaim。  
4. 填表：**pending 条数 / 是否重复投递**。

**C. `RReliableQueue`**

1. 按分篇三配置 `QueueConfig`，`add` 后 `poll`，**故意不 acknowledge** 或 **negativeAcknowledge**（按文档）。  
2. 观察消息是否 **再次可 poll**。  
3. 正常路径 `acknowledge` 后确认 **不可再取到**。

### 验证标准

- 书面结论 200 字内：**本业务接受「至多一次」还是「至少一次」**；若为至少一次，**幂等键**是什么。  
- 能指出：**A/B/C 哪条链路在杀进程后仍能从 Redis 找回未完成工作**。

## 大师私房话

**消息系统里，没有「默认恰好一次」这种免费午餐。** 把「重复投递」当成常态设计，你会少很多凌晨电话。
