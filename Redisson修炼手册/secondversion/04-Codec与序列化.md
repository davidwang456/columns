# 第四章：Codec 与序列化——对象怎么变成 Redis 里的字节？

[← 目录](README.md)

---

## 趣味：契约比剑法重要

小白升级 DTO 多了一个 `Optional` 字段，线上反序列化炸成烟花。大师叹气：「Redis 只认**字节**；你改的是 Java 类，动的是**跨版本契约**。剑招可以花哨，**剑谱版本号**不能乱。」  
后厨传来一声惨叫：有人把 **生产缓存** 当成了 **临时 Session 垃圾箱**，Codec 混用——大师摇头：**「能读出来」不等于「该读出来」。**

---

## 需求落地（由浅入深）

1. **浅**：发版后告警「反序列化失败」——根因往往是 **DTO 改字段**、**换 Codec**、或与 **Node/Python 服务** 共用 key 却各写各的格式。  
2. **中**：定 **允许使用的 Codec 列表**、**key 命名空间与版本**（v1/v2 双写迁移）；灰度时盯 **错误率与延迟**，而不是只看「功能有没有亮绿灯」。  
3. **深**：在 `Config` 层统一 Codec；大 value 考虑 **压缩**；对不可信来源的字节坚持 **白名单与升级 CVE**——**反序列化面 = 安全面**，与业务字段同等对待。

---

## 对话钩子

**小白**：反序列化失败、升级读不出老数据、不同语言服务读同 key 乱码……  
**大师**：**Codec 是数据契约**，与「接口 JSON 字段」同级，要上 **评审与迁移清单**。

**小白**：Kryo 快，我全项目 Kryo 行不行？  
**大师**：快的前提是你控得住 **类演进与跨语言**。团队若有多语言或频繁改模型，**JSON + 显式 schema** 往往更省钱——**快几天开发，慢半年排障**，不划算。

---

## Codec 三角：兼容 × 性能 × 安全

| 方案 | 兼容 | 性能 | 安全 |
|------|------|------|------|
| JSON（Jackson） | 较好 | 中 | 注意多态、默认 typing |
| Kryo 等二进制 | **需谨慎设计** | 高 | 白名单类 |
| JDK Serialization | 差 | 中 | **高风险**，遗留系统除外 |

权威说明：[data-serialization.md](../data-and-services/data-serialization.md)。

---

## 演进策略（实战）

1. **加字段**：默认向后兼容；禁止「改字段语义不换版本」。  
2. **破坏性变更**：新 **key 前缀** 或 **namespace v2**，双写迁移后再切读。  
3. **灰度**：小流量实例上新 codec，看 **反序列化错误率** 与 **延迟**。  
4. **多语言**：优先 **JSON + 显式 schema**（如 OpenAPI/Protobuf 定义），别指望 Kryo 跨语言。

---

## 压缩（LZ4 / Snappy 等）

- **大 value** 可考虑压缩；小 value 可能 **CPU 换不来空间收益**。  
- **敏感数据**：先谈 **加密、脱敏、ACL**，压缩不是安全。

---

## 反例（深度）

- **信任 Redis 里的字节**：若 key 可被其他组件写入，反序列化即 **RCE 攻击面**——最小权限 + 校验。  
- **同一 key 混用两种 Codec**：「有时能读有时不能」——**比全挂更难查**。

---

## 本章实验室（约 45～60 分钟）

**环境**：同一 `User` POJO（含 2～3 个字段）；`redisson.yaml` 或代码里可切换 `Codec`。

### 步骤

1. **体积与可读性**  
   - 实例 A：`config.setCodec(new JsonJacksonCodec())`，`getBucket("lab:codec:user:1").set(user)`。  
   - 实例 B：换 **Kryo** 或默认 Codec（与 A 不同即可），写入 **同一逻辑 key 的另一副本** 如 `lab:codec:user:2`（避免覆盖混淆）。  
   - `redis-cli --raw GET`（或 `STRLEN`）对比 **字节长度**；判断是否 **人类可读**。

2. **不兼容演练**  
   - 用 A 写入 `lab:codec:ver:v1`，再 **给 `User` 增字段或改类型**，用 B 读取 **旧 key**（或反过来）。  
   - 捕获异常，保存 **完整栈顶 5 行**，在笔记中标注：**「若线上出现类似栈，优先查哪次发版改了 DTO」**。

3. **规范落地**  
   - 起草 **《允许 Codec 列表》**：列名、适用场景、禁止场景（如 JDK 序列化）、**升级流程**（双写/新前缀）。

### 验证标准

- `redis-cli` 中能看出 **JSON vs 二进制** 差异。  
- 团队仓库或 Wiki 上存在 **Codec 白名单** 文档链接。

### 记录建议

- Code Review checklist 增加一条：**改 DTO 是否影响 Redis 中已有 key**。

---

## 大师私房话

**反序列化 = 远程执行代码的潜在入口**（视实现而定）。生产环境：**禁用不安全的默认反序列化**、限制类白名单、升级依赖跟进 CVE。
