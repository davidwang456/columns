# 第33章：调度器调度算法与性能调优

## 1 项目背景

某电商平台在生产环境部署了 Airflow 来调度 200+ 个 Dag，总计超过 5000 个 Task，高峰期每 5 分钟需要调度近 800 个 TaskInstance。上线三个月后，系统开始出现令人头疼的调度延迟：Scheduler 的日志显示心跳间隔从 5 秒逐渐拉长到 60 秒，大量 Task 从 SCHEDULED 状态排队超过 2 分钟才被提交给 Executor。

运维团队进行了多方排查：数据库 CPU 使用率 35%，Executor 空闲槽位充足，网络延迟正常。直到用 `pg_stat_statements` 揪出了元凶：Scheduler 在一次心跳中执行了超过 2000 条 SQL 查询，其中有一个 N+1 问题尤为突出——遍历 150 个 DagRun 时，每个 DagRun 又各自触发了一次 eager-load 查询去获取 TaskInstance。

更隐蔽的问题是"Parsimonious Parsing"（吝啬解析策略）的配置错误：团队为了"安全"开启了全量 Dag 解析，导致每次心跳都要重新解析所有 Dag 文件。200 个 Dag 的 import 操作累计耗时超过 30 秒，严重拖慢了心跳周期。

> 本章将深入 `airflow-core/src/airflow/jobs/scheduler_job_runner.py` 的核心方法 `_do_scheduling()`，完整走读 DagRun 创建 → TaskInstance 可调度性检查 → Executor 提交的全链路，同时系统讲解数据库查询优化、Scheduler HA 和调度延迟根因分析。

## 2 项目设计

**小胖**（看着 Grafana 监控面板上一条锯齿状的调度延迟曲线）："为什么 Scheduler 每次心跳要这么久？不就是判断哪些任务该跑吗，在我印象里就是一个 for 循环的事情。"

**大师**："你说的是 5 个 Dag 的情况，for 循环确实可行。但对于 200 个 Dag、5000 个 Task，for 循环里每次都做一次数据库查询，2000 条 SQL 打下去，PostgreSQL 也扛不住。而且还有一个你没看到的问题：调度器不是一个简单的'检查依赖 → 提交任务'的线性流程，它还有并发控制、Pool 槽位分配、DagRun 状态转换、僵尸任务回收——所有这些在一个心跳周期内完成。"

**小白**："那 `_do_scheduling()` 的具体执行顺序是什么？"

**大师**："我们来看源码中的执行阶段。`_do_scheduling()` 的逻辑分四个阶段，被 `prohibit_commit` 上下文管理器严格管控：

第一阶段——创建 DagRun。`_create_dagruns_for_dags()` 查询 `DagModel` 表中 `next_dagrun_create_after <= now()` 的记录，通过 timetable 计算下一个 DagRun 的逻辑日期，创建 `DagRun` 记录。这里有个重要的优化：每次最多创建 10 个 DagRun（`max_dagruns_to_create_per_loop`），防止创建操作占用太多时间。

第二阶段——调度 DagRun。`_schedule_all_dag_runs()` 选取'最久未被调度'的 N 个活跃 DagRun（默认 N=20，`max_dagruns_per_loop_to_schedule`），逐个检查每个 Task 的可调度性。这里有个'行级锁'的设计——用 `SELECT ... FOR UPDATE SKIP LOCKED` 确保多个 Scheduler 实例不会同时处理同一个 DagRun。

第三阶段——Dag 回调处理。调度完成后，在 `prohibit_commit` 之外发送 Dag 级别的成功/失败回调，使用 LRU 缓存避免重复获取同一个 Dag 的序列化数据。

第四阶段——临界区（Critical Section）。这是最关键的性能热点。通过在 Pool 表上加 Advisory Lock，确保同一时刻只有一个 Scheduler 执行此处。在此阶段，Scheduler 查询所有 `SCHEDULED` 状态的 TaskInstance，按照优先级排序，分配给各个 Executor。"

**小胖**："临界区为什么是性能热点？加个锁能有多慢？"

**大师**："因为这把锁是全局的。举例来说，如果你有 2 个 Scheduler 实例，第一个拿到了锁，在处理 500 个 TaskInstance。第二个 Scheduler 就只能在外面等着——哪怕它有充足的 CPU 和内存。更糟的是，拿到锁的 Scheduler 在临界区里要执行大量操作：查询 TaskInstance、检查 Executor 槽位、批量更新状态、提交任务——这全部是串行的。"

**小白**："那么 N+1 查询问题具体是怎么发生的？"

**大师**："典型场景在 `_schedule_all_dag_runs()` 中。假设你拿到了 20 个活跃 DagRun，需要为每个 DagRun 获取对应的 Dag 对象。如果不做优化，代码会写：`for dr in dag_runs: dag = SerializedDagModel.get(dr.dag_id)`——这会产生 20 次独立的数据库查询。Airflow 源码使用 `lru_cache` 装饰器缓存 `get_dag_for_run` 的结果，避免了对同一个 Dag 的重复查询。但更重要的优化在 `selectinload`——用 JOIN 一次性加载关联数据，而不是用 N+1 次查询逐个获取。"

**小胖**："那 Parsimonious Parsing 是什么？公司为什么要用？"

**大师**："Parsimonious Parsing 是一种'按需解析'策略。默认情况下，Scheduler 不需要解析任何 Dag 文件——它只读数据库中的 `serialized_dag` 表。Dag Processor 已经完成了所有解析工作。但有些场景——比如 `airflow dags test` 命令、或者 Scheduler 需要执行 Dag 回调时——才需要'按需'解析。如果错误地把 `parsing_processes` 配置到 Scheduler 进程上，每个心跳周期都触发 Dag 解析，就是调度延迟的主要元凶之一。"

> **技术映射**：`_do_scheduling()` = 机场调度塔台的工作流程（分配廊桥 → 检查飞机状态 → 指挥起飞）。Critical Section = 只能一个人使用的跑道调度权。N+1 查询 = 每检查一架飞机就打电话问一次燃料厂"有油吗"，而不是一次问清所有飞机。

### 3.2 阶段一：_do_scheduling() 完整源码走读

**步骤目标**：逐段理解 `_do_scheduling()` 的四个阶段。

```python
# 关键源码: airflow-core/src/airflow/jobs/scheduler_job_runner.py:1737

def _do_scheduling(self, session: Session) -> int:
    """
    在一次心跳中执行完整的调度决策。
    :return: 本次迭代中进入队列的 TaskInstance 数量
    """
    with prohibit_commit(session) as guard:
        # 阶段1: 创建 DagRun
        if self._scheduler_use_job_schedule:
            self._create_dagruns_for_dags(guard, session)
        
        # 启动 queued 状态的 DagRun
        self._start_queued_dagruns(session)
        guard.commit()
        
        # 阶段2: 获取活跃 DagRun 并调度
        # get_running_dag_runs_to_examine() 使用行级锁
        dag_runs = DagRun.get_running_dag_runs_to_examine(session=session)
        
        # _schedule_all_dag_runs 是耗时最长的部分
        callback_tuples = self._schedule_all_dag_runs(guard, dag_runs, session)
    
    # 阶段3: 在 prohibit_commit 之外发送回调
    cached_get_dag = lru_cache()(partial(
        self.scheduler_dag_bag.get_dag_for_run, session=session
    ))
    for dag_run, callback_to_run in callback_tuples:
        dag = cached_get_dag(dag_run)
        if dag:
            self._send_dag_callbacks_to_processor(dag, callback_to_run)
    
    with prohibit_commit(session) as guard:
        session.expunge_all()
        
        # 阶段4: 临界区 - 将 SCHEDULED 任务提交给 Executor
        total_free_executor_slots = sum(
            executor.slots_available for executor in self.executors
        )
        if total_free_executor_slots <= 0:
            self.log.debug("All executors are full, skipping critical section")
            return 0
        
        try:
            timer = stats.timer("scheduler.critical_section_duration")
            timer.start()
            num_queued_tis = self._critical_section_enqueue_task_instances(
                session=session
            )
            timer.stop(send=True)
        except OperationalError as e:
            timer.stop(send=False)
            if is_lock_not_available_error(error=e):
                # 另一个 Scheduler 持有锁，本实例跳过本次临界区
                self.log.debug("Critical section lock held by another Scheduler")
                stats.incr("scheduler.critical_section_busy")
                session.rollback()
                return 0
            raise
        
        guard.commit()
    
    return num_queued_tis
```

### 3.3 阶段二：数据库查询优化实战

**步骤目标**：理解并实践 `selectinload`、`joinedload` 和批量更新。

```python
"""
数据库查询优化模式对比
"""
from sqlalchemy import select, update
from sqlalchemy.orm import selectinload, joinedload, lazyload

# 反模式1: N+1 查询
# for dr in dag_runs:
#     tis = session.scalars(select(TI).where(TI.dag_id == dr.dag_id)).all()

# 优化1: 使用 selectinload 批量预加载关联
# 一次查询 DagRun, 一次查询所有关联的 TaskInstance
stmt = (
    select(DagRun)
    .where(DagRun.state == DagRunState.RUNNING)
    .options(
        selectinload(DagRun.task_instances),
        joinedload(DagRun.consumed_asset_events),
    )
)

# 优化2: 批量更新替代逐行更新
# 反模式: 循环中逐条 update
# 优化: 使用 WHERE IN 批量更新
stmt = (
    update(TaskInstance)
    .where(
        TaskInstance.task_id.in_(task_ids),
        TaskInstance.dag_id == dag_id,
        TaskInstance.run_id == run_id,
    )
    .values(state=TaskInstanceState.SCHEDULED)
    .execution_options(synchronize_session=False)
)
session.execute(stmt)
```

## 4 项目总结

### 优点 & 缺点对比

| 维度 | Airflow Scheduler 设计 | 传统 Cron + 脚本调度 |
|------|----------------------|---------------------|
| 依赖感知 | 原生 DAG 依赖和 Trigger Rule | 需手动编排 |
| 并发控制 | Pool + parallelism + max_active_tasks | 需自行实现信号量 |
| 高可用 | Advisory Lock + 多实例 | 需外部锁（如 ZooKeeper） |
| 性能诊断 | 内置 statsd 指标和 SQL 追踪 | 需自行埋点 |
| 复杂度 | 源码 3000+ 行，理解成本高 | 脚本简单但功能有限 |

### 适用场景

1. **大规模 Dag 集群**（>100 Dag）：Scheduler HA 和按需解析策略必不可少。
2. **高吞吐调度**（>500 Task/min）：需要精细调整并发参数和数据库连接池。
3. **多租户环境**：通过 Team 机制隔离不同团队的调度策略。
4. **资产驱动的管道**：Asset-triggered Dag 需要调度器精确支持。
5. **混合调度模式**：同时支持定时和事件驱动的 Dag。

### 不适用场景

1. **3 个以内 Dag 的环境**：简单配置即可，过度优化成本高。
2. **实时性要求极高的场景**（<1s 延迟）：Airflow 的调度粒度不适合毫秒级。

### 注意事项

- **数据库连接池**：Scheduler 的 `sql_alchemy_pool_size` 建议设为 `(max_dagruns_per_loop + parsing_processes) × 1.5`。
- **临界区优化**：`max_dagruns_per_loop_to_schedule` 不宜超过 50，过大会加长单一 Scheduler 的临界区持有时间。
- **版本兼容**：Airflow 3.x 的 `prohibit_commit` 机制替代了 2.x 的 `using` 上下文，升级时注意 session 管理变化。

### 常见踩坑经验

1. **Scheduler 卡在 DagRun 创建**：某个 Dag 的 timetable 计算耗时过长。解决：`max_dagruns_to_create_per_loop` 调小，检查 timetable 的实现效率。
2. **临界区持续被占用**：只有一个 Scheduler 拿到锁，其他都在空转。解决：`max_dagruns_per_loop_to_schedule` 调大到 30-50，让拿到锁的实例一次处理更多。
3. **Task 在 SCHEDULED 状态滞留**：Executor 槽位已满或 Pool 限制。解决：增大 parallelism、检查 Pool 配置、增加 Worker 数量。

### 思考题

1. **进阶题**：如果要实现一个"无锁临界区"——允许多个 Scheduler 实例同时在临界区中操作，但通过分区策略（如按 dag_id hash 分片）避免冲突——需要修改 `_critical_section_enqueue_task_instances()` 的哪些部分？请给出伪代码实现。

2. **设计题**：假设 Scheduler 的 `_schedule_all_dag_runs()` 中检测 Task 依赖的时间占总耗时的 40%。请设计一个依赖缓存方案——将 Dag 的依赖关系图缓存到 Redis，Scheduler 从 Redis 读取依赖而非每次查询数据库。考虑缓存失效策略（Dag 版本变化时失效）和一致性保证。

> **推广计划提示**：开发团队重点阅读 `_do_scheduling()` 和 `_critical_section_enqueue_task_instances()` 的源码，运维团队重点掌握数据库查询优化和 Scheduler HA 配置。测试团队建议使用 `pg_stat_statements` 设计调度器性能基准测试。
