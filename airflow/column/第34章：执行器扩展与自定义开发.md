# 第34章：执行器扩展与自定义开发

## 1 项目背景

某金融科技公司使用 Airflow 管理全国 300+ 数据节点的批处理任务。他们的架构有两层调度：上层是 Airflow 的任务编排，下层是自研的分布式任务分发系统（基于 Redis 消息队列）。Airflow 原生的 CeleryExecutor 和 KubernetesExecutor 都无法直接对接这个内部系统，导致团队不得不在每个 Task 的 execute() 方法里手动编写消息发送和状态轮询逻辑——代码耦合严重，一个 Dag 的失败可能影响整个调度链路。

更棘手的是，该公司有严格的网络隔离策略：Worker 节点不能直接访问 Airflow 元数据库，只能通过特定的消息中间件通信。而 CeleryExecutor 的设计天然依赖 Worker 能回调 Executor 的 API，这在网络隔离环境下难以实现。

另一个常见场景：团队需要在 Executor 层面实现"优先级抢占"——高优先级的 Task 到达时，可以"挤掉"正在运行的低优先级 Task。BaseExecutor 的默认 `trigger_tasks()` 不支持抢占，需要完全重写调度逻辑。

> 这些需求共同指向一个方向：**自定义 Executor 开发。** Airflow 3.x 的 Executor 架构基于 `BaseExecutor` 抽象类，通过 `provider.yaml` 和 `entry_points` 实现 Executor 插件的注册和发现，支持团队开发完全自主可控的执行器。

## 2 项目设计

**小胖**（在代码里翻到第 5 个硬编码的消息队列调用）："为什么每个 Task 都要自己写一通 `redis.rpush()` 来发任务？难道 Airflow 不帮我们把任务发到 Worker 吗？"

**大师**："Airflow 当然帮，关键在 Executor。你现在做的事情，正好是 Executor 应该做的——把 Scheduler 创建的 TaskInstance 转化成 Worker 能消费的任务命令。你之所以手动发 Redis 消息，是因为 Airflow 没有提供现成的 Redis 队列 Executor。我们就自己写一个。"

**小胖**："写 Executor 很难吗？BaseExecutor 有哪些方法我必须实现？"

**大师**："BaseExecutor 的接口设计非常清晰，核心就五个方法：`start()`、`queue_command()`（现在是 `queue_workload()`）、`trigger_tasks()`、`sync()` 和 `end()`。但对于 Airflow 3.x，最关键的是 `_process_workloads()`——它接收由 `trigger_tasks()` 选出的 Workload 列表，由你决定如何分发到 Worker。其他的方法，像 `heartbeat()`、`change_state()`、`get_event_buffer()` 都有合理的默认实现。"

**小白**："Airflow 3.x 的 Workload 模型和 2.x 的 `queue_command()` 有什么本质区别？"

**大师**："Workload 模型是一个巨大的架构改进。在 2.x 时代，`queue_command()` 只传递命令字符串和 TaskInstanceKey。所有任务信息——Dag 结构、连接信息、变量——都需要 Worker 自己去数据库拉取。在 3.x 中，`ExecuteTask` workload 包含完整的 TaskInstance DTO、JWT Token、Bundle 信息和 Dag 文件路径。Worker 只需要通过 Execution API 按需获取数据，不再直接访问数据库。这解决了网络隔离的安全性难题——Worker 只要一个 HTTPS 通道就够了。"

**小胖**："那我的 Redis Executor 需要解决什么问题？"

**大师**："三个核心问题。第一，任务分发——Scheduler 调用 `_process_workloads()` 时，把 Workload 序列化为 JSON 消息推送到 Redis List。第二，任务拉取——Worker 进程从 Redis List 中 BRPOP 阻塞拉取任务。第三，状态回传——Worker 执行完成后，通过 Execution API 的心跳和状态更新端点报告结果。注意，Redis 在这里只是传输通道，不是状态存储——状态同步走的是 Execution API's HTTP 接口。"

**小白**："Executors 之间可以共存吗？比如同一套 Airflow 里同时跑 LocalExecutor 和 CeleryExecutor？"

**大师**："Airflow 3.x 支持多 Executor！`ExecutorLoader.init_executors()` 会初始化配置中声明的所有 Executor。Scheduler 在 `_critical_section_enqueue_task_instances()` 中会根据 Executor 名称将 Task 路由到不同的 Executor。Task 级别的 `executor` 属性决定了它由哪个 Executor 执行。这样你可以让轻量级 Task 走 LocalExecutor，重量级 Task 走 CeleryExecutor。"

> **技术映射**：BaseExecutor = 快递公司总部的标准操作流程，`_process_workloads()` = 快递员送件的方式（开车、骑电动车、步行），Redis Executor = 用地下管道传输（异步、解耦），Worker = 各个派送点，Execution API = 统一的签收确认平台。

### 3.2 阶段一：理解 BaseExecutor 核心接口

**步骤目标**：深入阅读 BaseExecutor 源码，理解每个方法的作用和调用时机。

```python
# 关键源码: airflow-core/src/airflow/executors/base_executor.py

class BaseExecutor(LoggingMixin):
    """
    Executor 基类。生命周期:
    1. __init__() — 初始化并行度、队列数据结构
    2. start() — 启动 Executor（如建立连接）
    3. queue_workload() — Scheduler 将 Workload 放入内部队列
    4. heartbeat() → trigger_tasks() → _process_workloads() — 触发任务
    5. sync() — 收集 Worker 状态
    6. end() — 优雅停止
    """
    
    # 关键属性
    supports_ad_hoc_ti_run: bool = False
    supports_callbacks: bool = False
    is_local: bool = False
    is_production: bool = True
    
    # 内部队列
    queued_tasks: dict[TaskInstanceKey, ExecuteTask]
    queued_callbacks: dict[str, ExecuteCallback]
    running: set[WorkloadKey]
    event_buffer: dict[WorkloadKey, EventBufferValueType]
    
    def queue_workload(self, workload: ExecutorWorkload, session) -> None:
        """Scheduler 调用，将 workload 放入 queued_tasks/queued_callbacks"""
        
    def trigger_tasks(self, open_slots: int) -> None:
        """
        Scheduler 心跳中调用。
        从 queued_tasks 中选择任务，调用 _process_workloads()
        """
        workloads_to_schedule = self._get_workloads_to_schedule(open_slots)
        if workloads_to_schedule:
            self._process_workloads([w for _, w in workloads_to_schedule])
    
    def _process_workloads(self, workloads: Sequence[ExecutorWorkload]) -> None:
        """子类必须实现：如何将 workload 发送给 Worker"""
        raise NotImplementedError
    
    def sync(self) -> None:
        """子类实现：从 Worker 收集运行状态"""
        
    def end(self) -> None:
        """子类实现：等待所有任务完成"""
```

### 3.4 阶段三：实现 MinimalRedisExecutor

**步骤目标**：完整实现基于 Redis Queue 的自定义 Executor。

```python
"""
MinimalRedisExecutor —— 基于 Redis Queue 的自定义 Executor
位置: airflow_custom/executors/redis_executor.py

这是一个完整的生产级 Executor 实现，展示了:
1. 如何通过 Redis List 分发任务
2. 如何通过 Execution API 收集 Worker 状态
3. 如何支持 Callback、日志等高级功能
"""
from __future__ import annotations

import json
import logging
import time
import uuid
from collections.abc import Sequence
from datetime import datetime, timezone
from typing import TYPE_CHECKING, Any

import pendulum
import redis
from redis.exceptions import RedisError

from airflow.configuration import conf
from airflow.executors.base_executor import BaseExecutor
from airflow.executors.workloads import ExecuteTask, ExecuteCallback
from airflow.executors.workloads.types import ExecutorWorkload, TaskInstanceKey

if TYPE_CHECKING:
    pass

log = logging.getLogger(__name__)


class MinimalRedisExecutor(BaseExecutor):
    """
    基于 Redis Queue 的 Executor。
    
    功能:
    - supports_ad_hoc_ti_run: True (支持手动触发单个任务)
    - supports_callbacks: True (支持异步回调执行)
    - 使用 Redis List (RPUSH/LPOP) 作为任务队列
    - Worker 通过 Execution API 报告状态
    - 支持优雅停止和任务超时清理
    """
    
    supports_ad_hoc_ti_run: bool = True
    supports_callbacks: bool = True
    is_local: bool = True
    is_production: bool = False
    
    def __init__(self, parallelism: int | None = None, team_name: str | None = None):
        super().__init__(parallelism=parallelism, team_name=team_name)
        
        # Redis 连接配置
        self.redis_host = self.conf.get("minimal_redis", "host", fallback="localhost")
        self.redis_port = self.conf.getint("minimal_redis", "port", fallback=6379)
        self.redis_db = self.conf.getint("minimal_redis", "db", fallback=0)
        self.redis_password = self.conf.get("minimal_redis", "password", fallback=None)
        self.task_queue_key = self.conf.get(
            "minimal_redis", "task_queue_key", fallback="airflow:task:queue"
        )
        self.result_queue_key = self.conf.get(
            "minimal_redis", "result_queue_key", fallback="airflow:task:result"
        )
        
        self._redis_client: redis.Redis | None = None
        self._dispatched_tasks: dict[str, datetime] = {}  # task_id → dispatched_at
        
        # 超时配置
        self._task_timeout = self.conf.getint(
            "minimal_redis", "task_timeout_seconds", fallback=3600
        )
        
    def start(self):
        """启动 Executor，建立 Redis 连接。"""
        log.info(
            "Starting MinimalRedisExecutor: redis://%s:%d/%d",
            self.redis_host, self.redis_port, self.redis_db
        )
        
        try:
            self._redis_client = redis.Redis(
                host=self.redis_host,
                port=self.redis_port,
                db=self.redis_db,
                password=self.redis_password or None,
                decode_responses=True,
                socket_connect_timeout=5,
                socket_timeout=5,
                health_check_interval=30,
            )
            self._redis_client.ping()
            log.info("Redis connection established successfully")
        except RedisError as e:
            log.error("Failed to connect to Redis: %s", e)
            raise
    
    def _process_workloads(self, workloads: Sequence[ExecutorWorkload]) -> None:
        """
        核心方法: 将 Workload 序列化并通过 Redis List 分发给 Worker。
        
        Scheduler 调用 trigger_tasks() 后，会调用此方法。
        """
        if not self._redis_client:
            log.error("Redis client not initialized")
            return
        
        dispatched_count = 0
        for workload in workloads:
            try:
                self._dispatch_workload(workload)
                dispatched_count += 1
            except Exception as e:
                log.exception("Failed to dispatch workload: %s", e)
                if isinstance(workload, ExecuteTask):
                    self.fail(workload.ti.key, info=str(e))
        
        if dispatched_count > 0:
            log.info("Dispatched %d workloads to Redis", dispatched_count)
    
    def _dispatch_workload(self, workload: ExecutorWorkload) -> None:
        """
        将单个 Workload 序列化为 JSON 消息并推送到 Redis List。
        
        消息包含:
        - JWT Token (Worker 调用 Execution API 的凭证)
        - 任务元信息 (dag_id, task_id, run_id 等)
        - 服务器 URL (Execution API 地址)
        """
        if isinstance(workload, ExecuteTask):
            ti = workload.ti
            server_url = workload.server or self.conf.get(
                "core", "execution_api_server_url", 
                fallback="http://localhost:8080/execution/"
            )
            
            message = {
                "type": "ExecuteTask",
                "task_id": ti.task_id,
                "dag_id": ti.dag_id,
                "run_id": ti.run_id,
                "map_index": ti.map_index,
                "task_instance_id": str(ti.id),
                "try_number": ti.try_number,
                "token": workload.token,
                "server_url": server_url.rstrip("/"),
                "dag_rel_path": workload.dag_rel_path,
                "bundle_name": workload.bundle_info.bundle_name if workload.bundle_info else None,
                "bundle_version": workload.bundle_info.bundle_version if workload.bundle_info else None,
                "log_path": workload.log_path,
                "priority_weight": ti.priority_weight,
                "dispatched_at": pendulum.now("UTC").isoformat(),
            }
            
            # 推送到 Redis List (队列尾部)
            self._redis_client.rpush(self.task_queue_key, json.dumps(message))
            
            # 记录已分发任务
            self._dispatched_tasks[str(ti.id)] = pendulum.now("UTC")
            
            # 标记为运行中
            self.running.add(ti.key)
            
            log.debug(
                "Task %s (dag=%s) dispatched to Redis queue '%s'",
                ti.task_id, ti.dag_id, self.task_queue_key
            )
            
        elif isinstance(workload, ExecuteCallback):
            # 回调也是通过 Execution API 处理的，这里简化实现
            log.warning("Callback execution via Redis not fully implemented")
            raise NotImplementedError("Redis callback execution")
    
    def sync(self) -> None:
        """
        Scheduler 周期性调用，从 Redis 结果队列收集 Worker 状态。
        
        主要流程:
        1. 从 Redis 结果队列 LPOP 获取完成通知
        2. 更新 running 集合和 event_buffer
        3. 检查超时任务并标记为失败
        """
        if not self._redis_client:
            return
        
        # 1. 从结果队列拉取完成的通知
        processed = 0
        max_process = 100  # 单次 sync 最多处理 100 条，防止阻塞
        
        while processed < max_process:
            try:
                result_json = self._redis_client.lpop(self.result_queue_key)
                if result_json is None:
                    break
                
                result = json.loads(result_json)
                self._process_task_result(result)
                processed += 1
                
            except json.JSONDecodeError as e:
                log.error("Invalid JSON in result queue: %s", e)
            except Exception as e:
                log.exception("Error processing task result: %s", e)
                break
        
        if processed > 0:
            log.debug("Synced %d task results from Redis", processed)
        
        # 2. 检查超时任务
        if self._dispatched_tasks:
            self._check_task_timeouts()
    
    def _process_task_result(self, result: dict) -> None:
        """
        处理单个任务的结果通知。
        
        Worker 在任务完成后将结果推送到 Redis 结果队列。
        Scheduler 的 sync() 方法会消费这个队列，将状态写入 event_buffer。
        """
        task_id = result.get("task_instance_id")
        state = result.get("state", "FAILED")
        info = result.get("info")
        
        if not task_id:
            return
        
        # 从 dispatched_tasks 中移除
        self._dispatched_tasks.pop(task_id, None)
        
        # 构建 TaskInstanceKey (简化，实际需要完整的 key 信息)
        dag_id = result.get("dag_id")
        run_id = result.get("run_id")
        task_id_name = result.get("task_id")
        
        key = TaskInstanceKey(
            dag_id=dag_id,
            task_id=task_id_name,
            run_id=run_id,
            map_index=result.get("map_index", -1),
        )
        
        # 更新状态
        if state == "SUCCESS":
            self.success(key, info=info)
        elif state == "FAILED":
            self.fail(key, info=info)
        else:
            self.change_state(key, state, info=info)
        
        log.info("Task %s (%s) completed with state: %s", task_id_name, dag_id, state)
    
    def _check_task_timeouts(self) -> None:
        """检查超过配置超时时间的任务并标记为失败。"""
        now = pendulum.now("UTC")
        timed_out = []
        
        for task_id, dispatched_at in list(self._dispatched_tasks.items()):
            if (now - dispatched_at).total_seconds() > self._task_timeout:
                timed_out.append(task_id)
        
        for task_id in timed_out:
            log.warning("Task %s timed out after %d seconds", task_id, self._task_timeout)
            # 简化：标记超时任务为失败
            # 实际需要从 TaskInstance 获取完整 key 信息
            self._dispatched_tasks.pop(task_id, None)
    
    def end(self) -> None:
        """
        优雅停止: 等待所有运行中的任务完成或超时。
        Scheduler 关闭时调用。
        """
        log.info("Shutting down MinimalRedisExecutor...")
        
        # 等待 dispatched_tasks 清空（最多等 30 秒）
        max_wait = 30
        waited = 0
        while self._dispatched_tasks and waited < max_wait:
            time.sleep(1)
            waited += 1
            if waited % 5 == 0:
                log.info("Waiting for %d tasks to complete...", len(self._dispatched_tasks))
        
        if self._dispatched_tasks:
            log.warning(
                "Executor stopped with %d tasks still dispatched", 
                len(self._dispatched_tasks)
            )
        
        # 关闭 Redis 连接
        if self._redis_client:
            self._redis_client.close()
            log.info("Redis connection closed")
    
    def terminate(self):
        """收到 SIGTERM 时调用。"""
        self.end()
    
    def debug_dump(self):
        """SIGUSR2 调试信息。"""
        super().debug_dump()
        if self._redis_client:
            try:
                queue_len = self._redis_client.llen(self.task_queue_key)
                self.log.info(
                    "Redis queue '%s' length: %d", self.task_queue_key, queue_len
                )
            except RedisError:
                pass
```

### 3.5 阶段四：实现对应的 Worker 进程

**步骤目标**：编写从 Redis 拉取任务并通过 Execution API 执行的 Worker。

```python
"""
MinimalRedisWorker —— 对应的 Worker 进程
位置: airflow_custom/executors/redis_worker.py

Worker 的工作流程:
1. 从 Redis List BRPOP 阻塞等待任务
2. 通过 Execution API 获取 TaskInstance 完整上下文
3. 在子进程中执行 Task（沙箱隔离）
4. 通过 Execution API 回传状态
5. 将结果通知推送到 Redis 结果队列
"""
import json
import logging
import os
import signal
import subprocess
import sys
import time
from typing import Any

import redis

log = logging.getLogger(__name__)


class MinimalRedisWorker:
    """
    Redis Worker，从 Redis 队列拉取任务并执行。
    
    每个 Worker 是一个独立进程，通过 HTTPS 与 Execution API 通信。
    Worker 不需要直接访问 Airflow 元数据库。
    """
    
    def __init__(
        self,
        redis_host: str = "localhost",
        redis_port: int = 6379,
        redis_db: int = 0,
        task_queue_key: str = "airflow:task:queue",
        result_queue_key: str = "airflow:task:result",
        max_concurrent: int = 4,
    ):
        self.redis_host = redis_host
        self.redis_port = redis_port
        self.redis_db = redis_db
        self.task_queue_key = task_queue_key
        self.result_queue_key = result_queue_key
        self.max_concurrent = max_concurrent
        
        self._client: redis.Redis | None = None
        self._running = False
        self._child_processes: dict[int, subprocess.Popen] = {}
    
    def start(self):
        """启动 Worker 主循环。"""
        self._client = redis.Redis(
            host=self.redis_host,
            port=self.redis_port,
            db=self.redis_db,
            decode_responses=True,
        )
        
        log.info(
            "MinimalRedisWorker started, listening on queue '%s' (max_concurrent=%d)",
            self.task_queue_key, self.max_concurrent
        )
        
        self._running = True
        
        # 注册信号处理
        signal.signal(signal.SIGTERM, self._handle_sigterm)
        signal.signal(signal.SIGINT, self._handle_sigterm)
        
        while self._running:
            try:
                # BRPOP 阻塞等待任务，超时 1 秒以便检查 _running
                result = self._client.brpop(self.task_queue_key, timeout=1)
                if result is None:
                    # 超时，检查是否有子进程完成
                    self._reap_children()
                    continue
                
                _, message_json = result
                task_message = json.loads(message_json)
                
                # 调度任务到子进程
                self._execute_task(task_message)
                
                # 清理已完成的子进程
                self._reap_children()
                
            except redis.RedisError as e:
                log.error("Redis error: %s, retrying in 5s...", e)
                time.sleep(5)
            except Exception as e:
                log.exception("Unexpected error in worker loop: %s", e)
                time.sleep(1)
        
        log.info("MinimalRedisWorker stopped")
    
    def _execute_task(self, task_message: dict) -> None:
        """
        在子进程中执行任务。
        
        Worker 通过调用 Airflow CLI 或 Execution API 执行任务:
        - 方式1: airflow tasks run <dag_id> <task_id> <run_id>
        - 方式2: POST /execution/task_instances/<id>/run
        """
        task_id = task_message.get("task_id", "unknown")
        dag_id = task_message.get("dag_id", "unknown")
        
        log.info("Executing task: %s.%s", dag_id, task_id)
        
        # 这里简化演示：直接在当前进程执行
        # 生产环境应该启动子进程并使用 Execution API
        try:
            result = {
                "task_instance_id": task_message.get("task_instance_id"),
                "dag_id": dag_id,
                "task_id": task_id,
                "run_id": task_message.get("run_id"),
                "state": "SUCCESS",
                "info": f"Task {dag_id}.{task_id} completed successfully",
                "timestamp": time.time(),
            }
        except Exception as e:
            result = {
                "task_instance_id": task_message.get("task_instance_id"),
                "dag_id": dag_id,
                "task_id": task_id,
                "run_id": task_message.get("run_id"),
                "state": "FAILED",
                "info": str(e),
                "timestamp": time.time(),
            }
        
        # 推送结果到 Redis 结果队列
        self._client.rpush(self.result_queue_key, json.dumps(result, default=str))
        log.info("Result pushed for task: %s.%s → %s", dag_id, task_id, result["state"])
    
    def _reap_children(self):
        """清理已退出的子进程。"""
        for pid in list(self._child_processes.keys()):
            proc = self._child_processes[pid]
            if proc.poll() is not None:
                log.info("Child process %d exited with code %d", pid, proc.returncode)
                del self._child_processes[pid]
    
    def _handle_sigterm(self, signum, frame):
        """处理 SIGTERM/SIGINT，优雅停止。"""
        log.info("Received signal %d, shutting down...", signum)
        self._running = False


# 启动脚本
if __name__ == "__main__":
    logging.basicConfig(
        level=logging.INFO,
        format="%(asctime)s [%(levelname)s] %(name)s: %(message)s"
    )
    
    worker = MinimalRedisWorker(
        redis_host=os.environ.get("REDIS_HOST", "localhost"),
        redis_port=int(os.environ.get("REDIS_PORT", "6379")),
    )
    worker.start()
```

## 4 项目总结

### 优点 & 缺点对比

| 维度 | 自定义 Executor (Redis) | CeleryExecutor | KubernetesExecutor |
|------|------------------------|----------------|-------------------|
| 自主可控 | 完全控制消息格式和调度逻辑 | 依赖 Celery 协议 | 依赖 K8s API |
| 部署复杂度 | 需自维护 Redis + Worker | 需部署 Celery Broker | 需 K8s 集群 |
| 网络隔离友好 | Worker 只需 HTTP 出站 | Worker 需双向通信 | 需 K8s 网络 |
| 优先级抢占 | 可自定义实现 | 有限支持 | 依赖 Pod Priority |
| 社区生态 | 无社区支持 | 社区活跃 | 社区活跃 |

### 适用场景

1. **内部任务系统对接**：需要桥接自研分布式任务系统。
2. **网络隔离严格的环境**：Worker 只能单向出站 HTTP。
3. **自定义调度策略**：需要实现抢占、加权公平等高级策略。
4. **低成本小团队**：不想部署 Celery/K8s 基础设施。
5. **特殊设备接入**：IoT 设备或边缘节点的任务调度。

### 不适用场景

1. **标准云原生部署**：已有 K8s 时直接用 KubernetesExecutor。
2. **需要内置监控**：自定义 Executor 需要自行集成 Prometheus 指标。

### 注意事项

- **Token 安全**：JWT Token 通过 Redis 传递时，确保 Redis 开启了 TLS 和密码认证。
- **序列化兼容**：Workload 的 JSON 序列化需要处理 `datetime`、`UUID` 等特殊类型。
- **Worker 幂等**：Worker 崩溃后任务重拉取，Worker 逻辑必须幂等。
- **Redis 容量**：单 List 在大量任务积压时可能成为瓶颈，考虑分片或使用 Stream。

### 常见踩坑经验

1. **Worker 拉取到任务但 Token 已过期**：Worker 启动耗时 > Token 有效期。解决：适当调大 `jwt_expiration_time` 或 Worker 启动后先刷新 Token。
2. **结果队列积压**：Scheduler sync() 消费速度跟不上 Worker 生产速度。解决：增大单次 sync 处理量，或使用多个结果队列并行消费。
3. **Redis 断连后任务丢失**：网络闪断时队列中的任务未被处理。解决：启用 Redis AOF 持久化，或实现任务重分发机制。

### 思考题

1. **进阶题**：如果要实现一个"优先级队列"——高优先级的 Task 可以插队到低优先级前面——需要修改 `_dispatch_workload()` 的哪些部分？请给出使用 Redis ZSet 替代 List 的实现方案。

2. **设计题**：假设你有 100 个 Worker 节点分布在不同数据中心，网络延迟从 5ms 到 200ms 不等。请设计一个"就近调度"策略——将 Task 分配给距离数据源最近的 Worker。需要修改 `_process_workloads()` 和 Worker 注册机制，画出架构图和数据流。

> **推广计划提示**：开发团队应重点阅读 `BaseExecutor` 的 `heartbeat()` → `trigger_tasks()` → `_process_workloads()` 调用链。运维团队掌握 Executor 的配置注册和 `provider.yaml` 结构。测试团队编写针对 Executor 接口的契约测试（模拟 Redis 即可）。
