# 第23章：任务重试、超时与容错——打造健壮的流水线

## 1. 项目背景

深夜两点，监控告警声划破寂静。生产环境的数据管道中，一个关键 ETL 任务因上游 API 短暂不可用而失败，下游五个小时的依赖任务全部阻塞。值班同事揉着惺忪睡眼，手动重跑了那个任务——一切恢复正常。这正是**缺乏健壮容错机制**的典型代价。

在真实的大数据调度场景中，以下痛点反复出现：

- **瞬态故障频发**：第三方 API 偶尔返回 503，数据库连接池瞬时耗尽，网络抖动导致 SSH 超时——这些问题根本不需要人工介入，一次自动重试就能解决。
- **耗时失控**：一个解析任务本应 5 分钟完成，却因为上游数据量激增跑了 2 小时，占满了 Worker 资源，拖垮整个集群。
- **僵尸任务**：Worker 节点意外宕机，但调度器以为任务仍在运行，导致该任务永远卡在 "Running" 状态，对应的 Dag Run 永远无法完成。
- **SLA 违约无感知**：核心报表任务要求在每天 8:00 前产出，但是否按时完成，全靠人工盯着仪表板。

Airflow 作为 Apache 顶级项目，其 `retries`、`retry_delay`、`execution_timeout`、`dagrun_timeout`、`SLA` 以及心跳机制构成的容错体系，正是为解决这些问题而设计的。本章将带你深入 Airflow 的任务重试、超时控制与容错机制，通过源码剖析与实战案例，构建一条真正"自愈"的数据流水线。

## 2. 项目设计

> **场景设定**：某数据团队负责每日金融数据的接入与清洗。上游行情 API 偶发不可用，下游数据校验规则严格。小胖刚入职两周，被线上 Task 频繁失败折磨得不轻。

**小胖**：大师，我快疯了！那个 `fetch_market_data` 任务每天早上都要失败两三次，API 偶尔抽风个十几秒就好了，但我每次都得手动点重跑。Airflow 能不能自己重试啊？

**大师**（抿了一口茶）：你连 `retries` 参数都没配置吧？Airflow 在 `BaseOperator` 里内置了完整的重试机制。你看看 `task-sdk/src/airflow/sdk/bases/operator.py`，每个 Operator 都继承自 `BaseOperator`，而重试相关的核心参数都在这里定义。

```python
# operator.py:871-874
retries: int | None = DEFAULT_RETRIES          # 默认0，即不重试
retry_delay: timedelta = DEFAULT_RETRY_DELAY    # 默认300秒
retry_exponential_backoff: float = 0            # 指数退避乘数
max_retry_delay: timedelta | float | None = None  # 最大重试间隔上限
```

**小白**（翻看代码）：所以只要给任务设 `retries=3`，Airflow 就会自动在失败后重试 3 次？

**大师**：没错，但仅仅是固定间隔重试还不够。你看 `retry_exponential_backoff` 这个参数——它是一个乘数因子。假设 `retry_delay=4分钟`，`retry_exponential_backoff=5`：

- 第 1 次重试等待：4 分钟
- 第 2 次重试等待：4 × 5 = 20 分钟
- 第 3 次重试等待：20 × 5 = 100 分钟

这是**指数退避（Exponential Backoff）**策略。

**小胖**：啊，这个我懂！API 如果挂了，短时间内恢复的概率不大，盲目重试只会加重服务端负担。指数退避给系统留了更多"喘息"时间。

**大师**：对，而且你还可以用 `max_retry_delay` 来给指数增长"戴个帽子"：

```python
# 无论指数退避怎么算，单次等待不超过30分钟
fetch_task = PythonOperator(
    task_id="fetch_market_data",
    python_callable=fetch_from_api,
    retries=5,
    retry_delay=timedelta(minutes=4),
    retry_exponential_backoff=2,  # 每次翻倍
    max_retry_delay=timedelta(minutes=30),
)
```

**小白**：那万一任务干脆不失败了，直接卡住了怎么办？我们上周有个 Spark 任务死循环，跑了 6 个小时都没停。

**大师**：好问题！这就是 `execution_timeout` 的用武之地。Airflow 支持**任务级别**和 **DagRun 级别**两种超时控制。

任务级超时——`execution_timeout`，定义在 `operator.py:891`：

```python
# 任务若执行超过 1 小时，直接标记为失败
execution_timeout: timedelta | None = DEFAULT_TASK_EXECUTION_TIMEOUT
```

Scheduler 在将任务下发到 Worker 时，会将 `execution_timeout` 注入到 `trigger_timeout` 字段中（见 `airflow-core/src/airflow/models/taskinstance.py:1697-1703`）：

```python
# 若设了 execution_timeout，则取 execution_timeout 和 trigger_timeout 中较早的那个
if execution_timeout := self.task.execution_timeout:
    if self.trigger_timeout:
        self.trigger_timeout = min(
            self.start_date + execution_timeout, self.trigger_timeout
        )
    else:
        self.trigger_timeout = self.start_date + execution_timeout
```

DagRun 级别超时——`dagrun_timeout`，定义在 `dag.py:488-491`。Scheduler 在每次调度循环中检查（`scheduler_job_runner.py:2367-2370`）：

```python
if (
    dag_run.start_date
    and dag.dagrun_timeout
    and dag_run.start_date < timezone.utcnow() - dag.dagrun_timeout
):
    dag_run.set_state(DagRunState.FAILED)
    # 将所有未完成的任务标记为 SKIPPED
```

**小胖**：那 `on_failure_callback` 和 `on_retry_callback` 有什么区别？

**大师**：问得精准。看 `operator.py:893-895`：

```python
on_failure_callback: Sequence[TaskStateChangeCallback] = ()   # 任务最终失败时触发
on_retry_callback: Sequence[TaskStateChangeCallback] = ()     # 每次重试前触发
on_success_callback: Sequence[TaskStateChangeCallback] = ()   # 任务成功时触发
```

关键区别：`on_retry_callback` 在**每次重试之前**触发——你可以在里面发送通知"任务正在第 N 次重试"；而 `on_failure_callback` 只在**所有重试耗尽、任务最终标记为 FAILED** 时触发一次。这是两段不同时间线上的回调。

**小白**：那如果 Worker 进程直接挂了，调度器能发现吗？

**大师**：这就是 Airflow 的**僵尸任务检测**机制（Zombie / Heartbeat Timeout）。Scheduler 在 `scheduler_job_runner.py:2898-3016` 中实现了一套完整的心跳检测逻辑。每个运行中的 Task Instance 都有一个 `last_heartbeat_at` 字段。Scheduler 定期扫描处于 `RUNNING` 或 `RESTARTING` 状态、且心跳超过 `task_instance_heartbeat_timeout` 秒没更新的任务：

```python
# scheduler_job_runner.py:2918-2937
def _find_task_instances_without_heartbeats(self, *, session):
    limit_dttm = timezone.utcnow() - timedelta(
        seconds=self._task_instance_heartbeat_timeout_secs
    )
    return session.scalars(
        select(TI)
        .where(
            TI.state.in_((TaskInstanceState.RUNNING, TaskInstanceState.RESTARTING)),
            TI.last_heartbeat_at < limit_dttm,
        )
    )
```

一旦被发现，这些"僵尸"任务会被标记为 FAILED，并生成 `TaskCallbackRequest` 交由 DAG Processor 处理。默认心跳超时可在 `airflow.cfg` 中通过 `[scheduler] task_instance_heartbeat_timeout` 配置。

## 3. 项目实战

### 3.1 环境准备

首先创建一个完整的 DAG 文件 `robust_pipeline.py`：

```python
from __future__ import annotations

import random
import time
import textwrap
from datetime import datetime, timedelta

from airflow.sdk import DAG
from airflow.providers.standard.operators.bash import BashOperator
from airflow.providers.standard.operators.python import PythonOperator


# ============================================================
# 工具函数：模拟不稳定的 API 调用
# ============================================================
def fetch_market_data(**context):
    """模拟从行情 API 拉取数据，70% 概率失败"""
    task_instance = context["task_instance"]
    try_number = task_instance.try_number
    max_tries = task_instance.max_tries

    print(f"[尝试 {try_number}/{max_tries}] 正在请求行情 API...")

    # 模拟 70% 概率失败
    if random.random() > 0.3:
        msg = f"行情 API 返回 503 Service Unavailable (模拟)"
        print(f"[失败] {msg}")
        raise ConnectionError(msg)

    # 模拟网络延迟
    time.sleep(random.uniform(2, 5))
    print("[成功] 行情数据拉取完成，共 12,500 条记录")
    return {"records": 12500, "timestamp": str(datetime.now())}


def validate_data(**context):
    """模拟数据校验，校验不过抛异常"""
    task_instance = context["task_instance"]
    try_number = task_instance.try_number

    print(f"[尝试 {try_number}] 正在进行数据质量校验...")

    # 从 XCom 取上游数据
    ti = context["ti"]
    upstream_data = ti.xcom_pull(task_ids="fetch_market_data")

    if upstream_data and upstream_data.get("records", 0) > 10000:
        print("[成功] 数据校验通过")
        return True
    else:
        raise ValueError(f"数据量不足：{upstream_data}")


def on_retry_callback(context):
    """重试时触发的回调 —— 发送企业微信/钉钉通知"""
    ti = context["task_instance"]
    print(
        f"[通知] 任务 '{ti.task_id}' 第 {ti.try_number} 次重试于 {datetime.now()}，"
        f"原因：{context.get('exception')}"
    )


def on_failure_callback(context):
    """最终失败时触发的回调 —— 升级告警"""
    ti = context["task_instance"]
    print(
        f"[告警] 任务 '{ti.task_id}' 在 {ti.try_number} 次尝试后彻底失败！"
        f"请立即排查。DAG: {ti.dag_id}, Run: {context['dag_run'].run_id}"
    )


def sla_miss_callback(dag, task_list, blocking_task_list, slas, blocking_tis):
    """SLA 违约回调 —— 发送高优先级告警"""
    print(f"[SLA告警] DAG '{dag.dag_id}' 以下任务未在 SLA 时间内完成：")
    for task in task_list:
        print(f"  - {task.task_id} (SLA: {task.sla})")


# ============================================================
# DAG 定义
# ============================================================
with DAG(
    dag_id="robust_finance_pipeline",
    description="健壮的金融数据流水线 —— 含重试、超时、回调、SLA",
    schedule=timedelta(days=1),
    start_date=datetime(2026, 1, 1),
    catchup=False,
    tags=["production", "finance"],
    # DagRun 级别的超时 —— 整个 Run 不能超过 2 小时
    dagrun_timeout=timedelta(hours=2),
    # DAG 级别的失败回调
    on_failure_callback=lambda ctx: print(f"[DAG失败] {ctx['dag_run'].dag_id} 执行失败"),
) as dag:

    # ============================================================
    # Task 1: 数据拉取 —— 配置指数退避重试
    # ============================================================
    fetch_task = PythonOperator(
        task_id="fetch_market_data",
        python_callable=fetch_market_data,
        # ---- 重试配置 ----
        retries=5,                                  # 最多重试 5 次
        retry_delay=timedelta(seconds=30),          # 初始等待 30 秒
        retry_exponential_backoff=2,                # 每次翻倍：30s → 60s → 120s → 240s
        max_retry_delay=timedelta(minutes=5),       # 单次等待不超过 5 分钟
        # ---- 超时配置 ----
        execution_timeout=timedelta(minutes=15),    # 单次执行不超过 15 分钟
        # ---- 回调配置 ----
        on_retry_callback=on_retry_callback,
        on_failure_callback=on_failure_callback,
        # ---- 重试邮件（已弃用，仅作示例） ----
        email_on_failure=False,
        email_on_retry=False,
    )

    # ============================================================
    # Task 2: 数据校验 —— 固定间隔重试
    # ============================================================
    validate_task = PythonOperator(
        task_id="validate_data",
        python_callable=validate_data,
        retries=3,
        retry_delay=timedelta(minutes=2),
        execution_timeout=timedelta(minutes=10),
        on_failure_callback=on_failure_callback,
    )

    # ============================================================
    # Task 3: 数据入库 —— BashOperator 示例
    # ============================================================
    load_task = BashOperator(
        task_id="load_to_warehouse",
        bash_command=textwrap.dedent("""\
            echo "正在写入数仓..."
            sleep 30
            echo "写入完成: $(date)"
        """),
        retries=1,
        retry_delay=timedelta(minutes=5),
        execution_timeout=timedelta(hours=1),
    )

    # 定义依赖链
    fetch_task >> validate_task >> load_task
```

### 3.2 指数退避 vs 固定间隔：实际效果对比

假设 `fetch_market_data` 任务连续失败，我们来对比两种重试策略下任务真正失败的时间线：

**固定间隔（retries=5, retry_delay=30s）：**

| 尝试次数 | 执行耗时 | 等待间隔 | 累计耗时 |
|----------|---------|---------|---------|
| Try 1    | 10s     | —       | 10s     |
| Retry 1  | 8s      | 30s     | 48s     |
| Retry 2  | 7s      | 30s     | 1m 25s  |
| Retry 3  | 9s      | 30s     | 2m 4s   |
| Retry 4  | 8s      | 30s     | 2m 42s  |
| Retry 5  | 10s     | —       | 2m 52s  |

→ 总共不到 **3 分钟**，5 次重试全部打出去。如果 API 正在经历 5 分钟的故障恢复，这些请求全白费。

**指数退避（retries=5, retry_delay=30s, retry_exponential_backoff=2, max_retry_delay=5min）：**

| 尝试次数 | 执行耗时 | 等待间隔 | 累计耗时 |
|----------|---------|---------|---------|
| Try 1    | 10s     | —       | 10s     |
| Retry 1  | 8s      | 30s     | 48s     |
| Retry 2  | 7s      | 60s     | 1m 55s  |
| Retry 3  | 9s      | 120s    | 4m 4s   |
| Retry 4  | 8s      | 240s    | 8m 12s  |
| Retry 5  | 10s     | —       | 8m 22s  |

→ 总耗时约 **8 分钟**，给 API 留了充足恢复时间，避免"雪上加霜"。

### 3.3 回调机制实战

在 `robust_pipeline.py` 中我们定义了两个关键回调：

```python
def on_retry_callback(context):
    """每次重试前触发 —— 轻量级通知"""
    ti = context["task_instance"]
    # 实际可接入企业微信/钉钉/Slack
    send_wechat_notification(
        f"任务 {ti.task_id} 正在重试 ({ti.try_number}/{ti.max_tries})"
    )

def on_failure_callback(context):
    """所有重试耗尽后触发 —— 升级为 P1 告警"""
    ti = context["task_instance"]
    send_pagerduty_alert(
        f"任务 {ti.task_id} 彻底失败！Run: {context['dag_run'].run_id}"
    )
```

两者的时间线关系：

```
Task 执行
  │
  ├─ 失败 (Try 1) ──→ on_retry_callback() ──→ 等待 retry_delay
  ├─ 失败 (Try 2) ──→ on_retry_callback() ──→ 等待 retry_delay
  ├─ 失败 (Try 3) ──→ on_retry_callback() ──→ 等待 retry_delay
  │                        ⬆ 重试耗尽，以下不触发
  └─ 失败 (最终) ──→ on_failure_callback()  ← 只有这一处最终告警
```

### 3.4 SLA 与 Deadline Alert

> **注意**：SLA 功能在 Airflow 3.0 中已被移除，替换为 >=3.1 的 `DeadlineAlert` 机制。如果你的 Airflow 版本 < 3.0，仍可使用 `sla` 参数。

在 DAG 定义中配置 Task 级别的 SLA：

```python
fetch_task = PythonOperator(
    task_id="fetch_market_data",
    python_callable=fetch_market_data,
    sla=timedelta(hours=6),  # 要求在 schedule 后 6 小时内完成
    ...
)
```

DAG 级别配置：

```python
with DAG(
    dag_id="robust_finance_pipeline",
    sla_miss_callback=sla_miss_callback,  # DAG 级别的 SLA 违约回调
    ...
) as dag:
    ...
```

一旦任务的实际执行时间超过了 `sla` 阈值，Airflow 就会触发 `sla_miss_callback`，输出违约任务列表，方便运维团队追责和优化。

### 3.5 僵尸任务心跳检测配置

心跳检测相关的配置项在 `airflow.cfg` 中：

```ini
[scheduler]
# Task Instance 心跳超时时间（秒），默认 300 秒
# 超过此时间没有心跳更新的 RUNNING 任务将被标记为 FAILED
task_instance_heartbeat_timeout = 300

# 心跳检测的执行间隔（秒），默认 10 秒
task_instance_heartbeat_timeout_detection_interval = 10.0

# Scheduler 自身的心跳间隔
SCHEDULER_HEARTBEAT_SEC = 5
```

工作流程（源码 `scheduler_job_runner.py:1565-1664`）：

1. Scheduler 主循环中，每个 `_schedule_dag_run` 方法处理 DagRun 的调度。
2. Scheduler 同时周期性地调用 `_find_and_purge_task_instances_without_heartbeats()`。
3. 该方法查询所有 `state IN (RUNNING, RESTARTING)` 且 `last_heartbeat_at < now - timeout` 的 Task Instance。
4. 对每个"僵尸"任务：
   - 写入 `Log` 记录心跳超时事件。
   - 生成 `TaskCallbackRequest` 通知 DAG Processor 执行 `on_failure_callback`。
   - 通过 Executor 将任务状态改为 `FAILED`。

### 3.6 高级：RetryPolicy 异常驱动的重试决策

Airflow Task SDK 还提供了 `RetryPolicy` 接口，允许你**根据异常类型**做出精细化重试决策。源码位置 `task-sdk/src/airflow/sdk/definitions/retry_policy.py`。

```python
from airflow.sdk.definitions.retry_policy import (
    ExceptionRetryPolicy, RetryRule, RetryAction
)

# 自定义重试策略
retry_policy = ExceptionRetryPolicy(
    rules=[
        RetryRule(
            exception=ConnectionError,
            action=RetryAction.RETRY,
            retry_delay=timedelta(seconds=60),
            reason="网络连接错误，自动重试",
        ),
        RetryRule(
            exception=ValueError,
            action=RetryAction.FAIL,
            reason="数据校验失败，重试无意义",
        ),
    ],
    default=RetryAction.RETRY,  # 未匹配的异常默认重试
)

fetch_task = PythonOperator(
    task_id="smart_retry_task",
    python_callable=fetch_market_data,
    retry_policy=retry_policy,
    retries=5,
    retry_delay=timedelta(seconds=30),
)
```

这样当 `ConnectionError` 发生时系统自动重试（带 60 秒延迟），但 `ValueError` 导致的数据校验失败则立即标记为 FAILED，不再浪费重试次数。

## 4. 项目总结

### 4.1 容错机制对比总表

| 机制 | 作用层级 | 核心参数 | 解决的问题 | 源码位置 |
|------|---------|---------|-----------|---------|
| `retries` + `retry_delay` | Task | `retries`, `retry_delay`, `retry_exponential_backoff`, `max_retry_delay` | 瞬态故障自动恢复 | `operator.py:871-874` |
| `execution_timeout` | Task Instance | `execution_timeout: timedelta` | 单次执行耗时失控 | `operator.py:891`, `taskinstance.py:1697` |
| `dagrun_timeout` | DagRun | `dagrun_timeout: timedelta` | 整条流水线超时 | `dag.py:488`, `scheduler_job_runner.py:2367` |
| `sla` / `DeadlineAlert` | Task / DAG | `sla: timedelta`, `sla_miss_callback` | 产出时效性监控 | `dag.py:503` (已弃用于 3.0) |
| `on_retry_callback` | Task | 回调函数或列表 | 重试时的轻量通知 | `operator.py:895` |
| `on_failure_callback` | Task | 回调函数或列表 | 最终失败的升级告警 | `operator.py:893` |
| Heartbeat / Zombie 检测 | Task Instance | `task_instance_heartbeat_timeout` | Worker 宕机导致的任务僵死 | `scheduler_job_runner.py:2898` |
| `RetryPolicy` | Task | `retry_policy: RetryPolicy` | 按异常类型精细化重试 | `retry_policy.py:92` |

### 4.2 最佳实践建议

1. **重试次数不宜过多**：`retries=3~5` 是合理区间。超过 5 次的瞬态故障大概率是"根因问题"，重试只是浪费资源。
2. **生产环境必须用指数退避**：`retry_exponential_backoff=2` 配合 `max_retry_delay` 是标配，保护下游服务不被打垮。
3. **execution_timeout 设得比正常耗时大 2~3 倍**：既要防止死循环，也要给正常波动留足余量。
4. **DagRun timeout 大于所有 Task 超时之和**：否则 DagRun 级别的超时会先触发，导致正常任务被强制 SKIPPED。
5. **回调中不要做耗时操作**：`on_retry_callback` 和 `on_failure_callback` 在 Task Runner 进程中同步执行，阻塞回调会延长任务生命周期。
6. **SLA 阈值应参考历史 p99 耗时**：不要设得过于激进，否则每天都会被 SLA 告警淹没，产生告警疲劳。
7. **心跳超时不应太小**：默认 300 秒是合理值。如果设得太小（如 60 秒），网络轻微抖动就会导致大量任务被误杀。

### 4.3 思考题

1. **场景题**：你有一个任务，正常执行只需 2 分钟，但偶发地会因为上游数据未就绪而失败。数据通常会在 5~15 分钟内就绪。你会如何配置 `retries`、`retry_delay` 和 `retry_exponential_backoff`？请给出具体参数并说明理由。
