# 第31章：【中级篇综合实战】构建分布式数据平台调度中心

## 1 项目背景

某互联网公司旗下拥有广告投放、推荐系统、风险控制三条核心业务线，每条业务线日均产出 150+ 条数据处理 Dag。过去三年，各团队各自维护独立的调度系统：广告团队用 Azkaban 管理竞价日志回灌，推荐团队用自研 Python cron 脚本驱动特征工程，风控团队则依赖 Oozie 编排实时规则引擎。三套系统像三座孤岛，彼此无感知，运维成本居高不下——运维团队需要同时维护 Azkaban Executor 集群、cron 服务器集群和 Oozie Namenode HA，每周光巡检就要花掉 2 个人天。

更致命的痛点在于：**缺少全局资源管控**。2025 年双十一大促期间，推荐团队的 80 条特征计算 Dag 集中触发，瞬间占满了所有 Hadoop YARN 队列容量，导致广告团队的反作弊检测 Dag 因无可用资源而延迟 45 分钟——结果就是价值数百万的黑产刷量订单未被及时拦截。事后复盘发现，几条大 SQL 查询（SELECT * FROM ad_events WHERE dt >= ... 无分区裁剪）是资源泄漏的根因，但因为没有跨系统的资源隔离机制，推荐团队的"放飞"直接殃及了风控团队。

CTO 拍板：**用 Airflow 构建统一的分布式数据平台调度中心**，核心需求如下：

1. **多租户资源隔离**：三条业务线共用 Airflow 集群，但各自拥有独立的资源池（Pool），任何一个业务线的任务暴涨都不能挤占其他业务线的调度资源。
2. **混合执行器分层**：日常离线 ETL 走 CeleryExecutor（稳定可控），弹性计算任务（如训练数据采样、模型评估）走 KubernetesExecutor（按需扩缩容）。
3. **统一监控与分级告警**：Dag 失败发送钉钉通知（P0）、任务 SLA 延迟发送邮件（P1）、集群资源水位异常发送 Slack（P2），构建"无死角"的可观测体系。
4. **跨 Dag 依赖编排**：广告 Dag 的产出表是风控 Dag 的输入源，两者通过 ExternalTaskSensor + Asset 实现声明式级联触发，杜绝"靠预估时间等上游"的定时炸弹。
5. **日志聚合与检索**：所有 Worker 节点的任务日志统一推送到 ELK（Elasticsearch + Logstash + Kibana），支持按 Dag ID / Task ID / 时间范围秒级检索，告别 SSH 到 Worker 机器上 grep 日志的原始时代。

> **技术映射**：本章相当于中级篇的"毕业设计"——将第 17 章到第 30 章学到的元数据库、调度器架构、执行器选型、动态任务映射、连接池管理、容错与重试、Web UI 定制、REST API、监控告警、ELK、K8s 等知识点，在真实的生产级场景中串联起来。

## 2 项目设计

**小胖**（啃着巧克力棒，盯着白板上的架构图）："师傅，咱们这次要搞的事情好大啊——三条业务线、混合执行器、还得跨 Dag 编排，这比之前基础篇那个 ETL 流水线复杂了不止一个数量级。"

**大师**："对。基础篇是'在驾校操场里练倒库'，中级篇是'在城市快速路上换道并线'。这次综合实战，你得上高速了。"

**小白**："我先把架构捋一遍。"小白拿起白板笔，画了一个简图。"三条业务线的 Dag 通过一套 Airflow 集群调度，但执行层拆成两部分：Celery Worker 集群跑常规 ETL，K8s Pod 跑弹性任务。中间用 Redis 做 Celery 的消息队列，元数据存 PostgreSQL。监控侧 Prometheus 采指标，ELK 收日志，告警走钉钉/邮件/Slack 三通道。这样理解对吗？"

**大师**："方向对，但少了两个关键设计点。第一，资源隔离怎么做？刚才小胖说的'Azkaban 的慢查询拖垮 Oozie'问题，在 Airflow 里你会怎么防？"

**小胖**："Pool！中级篇第 22 章讲过的——给每个业务线建独立的 Pool，设不同的 slots 上限。"

**大师**："没错，但具体怎么分配？三个 Pool 各设多少 slots？"

**小白**："这个要算一下。假设 Celery 集群总共 64 个 Worker 并发槽位。广告线日均 2000 个 TaskInstance，峰值 QPS 最高；推荐线日均 3000 个，但单个任务时长波动大（5 分钟 ~ 2 小时）；风控线任务量最小但时效性要求最高——延迟 30 秒就会漏过欺诈交易。所以我倾向于：广告 Pool 32 slots（够用且有弹性），推荐 Pool 20 slots（防止长任务占坑），风控 Pool 12 slots（量小但要保证即时响应）。"

**大师**："基本合理。但你再想想——风控只有 12 个 slots，万一集中来了 50 个紧急任务怎么办？"

**小白**眼神一紧："那就要靠 Priority Weight 了。给风控任务设高优先级（比如 priority_weight=10），广告默认 3，推荐默认 1。这样当 Pool 满时，调度器优先把风控任务推进 Executor 的队列头部。"

**大师**："好。第二个设计点：跨 Dag 依赖。广告线的 `ad_bidding_report` Dag 产出表 `warehouse.ad_daily_report`，风控线的 `risk_detection` Dag 必须等这个表到位才能跑。你怎么确保不靠硬编码 `wait 30 minutes`？"

**小胖**："ExternalTaskSensor！第 9 章讲过的——可以在 Dag B 的起点放一个 Sensor，一直探测 Dag A 的 Task 是否成功。"

**小白**补充："而且我们还可以结合 Asset 调度（第 12 章）做第二道保险：把 `warehouse.ad_daily_report` 注册为 Asset，下游 Dag 声明对它的依赖。这样即使 ExternalTaskSensor 因为时区或执行日期映射问题漏检，Asset 调度也能感知到数据就绪。"

**大师**："ElasticSearch 日志聚合呢？日志量巨大——每天 50K TaskInstance，每个 Task 产生几百到几千行日志，你怎么设计日志采集管道？"

**小白**："用 Filebeat 替代 Celery Worker 上的本地日志文件 tail，统一推送到 Logstash 做字段解析（提取 dag_id、task_id、try_number、log_level），然后索引到 ES。Kibana 建几张核心大盘：按业务线的任务失败率趋势图、Top 10 慢任务排行、单 Task 日志全文检索页。"

**大师**（在白板上敲定最终架构）："很好，现在把执行器路由策略也敲定。Celery Worker 和 K8s Pod 如何分流？"

**小胖**："用 Airflow 的 `executor_config` 字段！在 Task 级别通过 `queue` 参数指定：`queue='celery'` 走 Celery，`queue='kubernetes'` 走 K8s。CeleryExecutor 会按 queue 路由到对应 Worker 节点，KubernetesExecutor 则按 queue 创建对应命名空间的 Pod。"

**大师**："> 技术映射：多租户资源隔离 = 商场里给每家店铺独立的配电箱（Pool slots），一个店铺用电过载只会跳自己家的闸；Cross-Dag 编排 = 汽车工厂的 JIT 流水线——引擎车间不凭'感觉'猜测车身车间的进度，而是通过看板信号（Sensor / Asset）精确获知上游就绪时间。"

### 3.3 核心代码：多租户 Pool 隔离 + 混合执行器路由

创建文件 `dags/comprehensive_platform_dag.py`：

```python
"""
中级篇综合实战：分布式数据平台调度中心
演示：多租户 Pool 隔离、混合执行器路由、跨 Dag 依赖、分级告警
"""
from datetime import datetime, timedelta
from airflow.sdk import DAG
from airflow.sdk.operators.python import task
from airflow.sdk.operators.bash import BashOperator
from airflow.sdk.operators.empty import EmptyOperator
from airflow.models.variable import Variable
from airflow.models.pool import Pool
import requests
import json
import smtplib
from email.mime.text import MimeText

# ============================================================
# 通用工具：分级告警模块
# ============================================================
def send_dingtalk_alert(message: str):
    """P0 级告警：任务失败，钉钉即时通知"""
    webhook = Variable.get("dingtalk_webhook")
    payload = {
        "msgtype": "markdown",
        "markdown": {
            "title": "Airflow P0 告警",
            "text": f"## 🚨 P0 任务失败\n{message}\n\n请立即处理！",
        },
    }
    requests.post(webhook, json=payload, timeout=10)

def send_email_alert(subject: str, body: str):
    """P1 级告警：SLA 延迟，邮件通知"""
    smtp_host = Variable.get("alert_email_smtp_host")
    to_addr = Variable.get("alert_email_to")
    msg = MimeText(body, "plain", "utf-8")
    msg["Subject"] = f"[Airflow P1] {subject}"
    msg["From"] = "airflow-alert@company.com"
    msg["To"] = to_addr
    with smtplib.SMTP(smtp_host, 25, timeout=10) as smtp:
        smtp.sendmail(msg["From"], [to_addr], msg.as_string())

def send_slack_alert(message: str):
    """P2 级告警：水位异常，Slack 频道通知"""
    webhook = Variable.get("slack_webhook")
    payload = {"text": f"⚙️ *Airflow P2 通知*\n{message}"}
    requests.post(webhook, json=payload, timeout=10)

def graded_alert(context, level: str = "P1"):
    """统一分级告警入口"""
    dag_id = context["task_instance"].dag_id
    task_id = context["task_instance"].task_id
    message = f"Dag: {dag_id}\nTask: {task_id}"

    if level == "P0":
        send_dingtalk_alert(message)
    elif level == "P1":
        send_email_alert(f"任务 SLA 延迟: {dag_id}.{task_id}", message)
    elif level == "P2":
        send_slack_alert(f"资源水位异常 - {message}")


# ============================================================
# 广告业务线 Dag：竞价日志 ETL（使用 ad_pool）
# ============================================================
with DAG(
    dag_id="ad_bidding_etl",
    description="广告竞价日志 ETL（CeleryExecutor + ad_pool）",
    schedule="0 3 * * *",
    start_date=datetime(2025, 1, 1),
    catchup=False,
    max_active_tasks=8,
    default_args={
        "owner": "ad-team",
        "pool": "ad_pool",                    # 绑定广告资源池
        "priority_weight": 3,                  # 中等优先级
        "retries": 2,
        "retry_delay": timedelta(minutes=5),
        "on_failure_callback": lambda ctx: graded_alert(ctx, "P0"),
        "on_sla_miss_callback": lambda *a, **k: graded_alert({}, "P1"),
    },
    tags=["ad", "production", "celery"],
) as ad_dag:

    start = EmptyOperator(task_id="start")
    end = EmptyOperator(task_id="end")

    # CeleryExecutor 任务：queue="celery"
    extract_logs = BashOperator(
        task_id="extract_logs",
        queue="celery",                        # 路由到 Celery Worker
        bash_command="""
            echo "抽取广告竞价日志: {{ ds }}"
            # 模拟从 Kafka 消费近 24h 的竞价事件
            COUNT=$((500000 + RANDOM % 200000))
            mkdir -p /tmp/airflow/ad/{{ ds }}/
            echo $COUNT > /tmp/airflow/ad/{{ ds }}/bidding_count.txt
            echo "抽取完成: $COUNT 条竞价记录"
        """,
    )

    @task(pool="ad_pool", priority_weight=5)
    def analyze_bidding(context):
        """分析竞价效果（高优先级任务）"""
        ds = context["ds"]
        with open(f"/tmp/airflow/ad/{ds}/bidding_count.txt") as f:
            count = int(f.read().strip())
        ctr = 0.035 + (context["dag_run"].id % 10) / 1000.0
        result = {
            "date": ds,
            "impressions": count * 2,
            "clicks": int(count * ctr),
            "ctr": f"{ctr:.4f}",
            "cost": count * 0.15,
        }
        print(f"广告效果分析完成: {json.dumps(result, ensure_ascii=False)}")
        return result

    @task
    def write_to_warehouse(results: dict):
        """写入数仓（产出 Asset）"""
        print(f"写入 warehouse.ad_daily_report，日期={results['date']}")
        return results

    start >> extract_logs >> analyze_bidding() >> write_to_warehouse() >> end


# ============================================================
# 推荐业务线 Dag：特征工程（KubernetesExecutor + rec_pool）
# ============================================================
with DAG(
    dag_id="rec_feature_engineering",
    description="推荐特征工程（KubernetesExecutor 弹性计算）",
    schedule="@daily",
    start_date=datetime(2025, 1, 1),
    catchup=False,
    default_args={
        "owner": "rec-team",
        "pool": "rec_pool",
        "priority_weight": 1,
        "retries": 3,
        "retry_delay": timedelta(minutes=10),
    },
    tags=["recommendation", "production", "kubernetes"],
) as rec_dag:

    @task(
        queue="kubernetes",                    # 路由到 K8s Pod
        executor_config={
            "KubernetesExecutor": {
                "image": "python:3.11-slim",
                "request_memory": "2Gi",
                "request_cpu": "1",
                "limit_memory": "4Gi",
                "limit_cpu": "2",
            }
        },
    )
    def feature_extraction(**context):
        """在 K8s Pod 中执行大规模特征提取"""
        ds = context["ds"]
        print(f"[K8s Pod] 开始特征提取，日期={ds}")
        # 模拟 Spark 特征工程
        features = {
            "user_count": 5000000,
            "item_count": 200000,
            "feature_dim": 512,
            "date": ds,
        }
        print(f"[K8s Pod] 特征提取完成: dim={features['feature_dim']}")
        return features

    @task(queue="kubernetes")
    def model_evaluation(features: dict):
        """在 K8s Pod 中执行模型评估"""
        print(f"[K8s Pod] 模型评估: 使用 {features['feature_dim']} 维特征")
        eval_result = {"auc": 0.872, "log_loss": 0.341}
        print(f"[K8s Pod] 评估完成: AUC={eval_result['auc']}")
        return eval_result

    feature_extraction() >> model_evaluation()


# ============================================================
# 风控业务线 Dag：风险检测（高优先级 + risk_pool）
# ============================================================
with DAG(
    dag_id="risk_detection",
    description="风控规则引擎（依赖 ad_bidding_report 产出表）",
    schedule="0 5 * * *",
    start_date=datetime(2025, 1, 1),
    catchup=False,
    default_args={
        "owner": "risk-team",
        "pool": "risk_pool",
        "priority_weight": 10,                 # 最高优先级
        "retries": 1,
        "retry_delay": timedelta(minutes=1),
        "on_failure_callback": lambda ctx: graded_alert(ctx, "P0"),
    },
    tags=["risk", "production", "celery"],
) as risk_dag:

    from airflow.sdk.operators.sensors.external_task import ExternalTaskSensor
    from airflow.sdk.operators.empty import EmptyOperator

    start = EmptyOperator(task_id="start")

    # 跨 Dag 依赖：等待广告 Dag 的 write_to_warehouse 完成
    wait_for_ad_report = ExternalTaskSensor(
        task_id="wait_for_ad_report",
        external_dag_id="ad_bidding_etl",
        external_task_id="write_to_warehouse",
        allowed_states=["success"],
        failed_states=["failed", "skipped"],
        timeout=3600,                          # 最多等 1 小时
        poke_interval=60,                      # 每 60 秒探测一次
        mode="reschedule",                     # 释放 Worker 槽位，不占坑
    )

    @task(pool="risk_pool", priority_weight=15)
    def fraud_detection(**context):
        """欺诈检测：读取 warehouse.ad_daily_report"""
```

### 方案优缺点对比

| 维度 | 本方案 | 替代方案 |
|------|--------|---------|
| 资源隔离 | Pool + Priority（应用层） | cgroup / K8s ResourceQuota（OS 层） |
| 弹性计算 | KubernetesExecutor 按需扩缩 | 固定 Celery Worker 数 |
| 告警 | 多渠道分级（钉钉/邮件/Slack） | 单通道（仅邮件） |
| 日志检索 | ELK 集中索引 | SSH + grep 本地日志 |
| 跨 Dag 依赖 | ExternalTaskSensor + Asset | 硬编码 wait 时间 |

**优点**：
1. 多租户 Pool 隔离彻底杜绝"某个业务线任务拖垮全局"的问题。
2. 混合执行器兼顾稳定性和弹性——Celery 处理常规 ETL，K8s 吸收突发计算。
3. 分级告警确保运维人员第一时间收到关键通知，同时避免告警疲劳。
4. ELK 日志聚合让排查效率从"SSH 登录多台机器 grep"提升到"Kibana 搜索框 3 秒出结果"。
5. Cross-Dag 依赖声明式管理，不再依赖预估时间，做到精准触发。

**缺点**：
1. 混合执行器运维复杂度较高——需要同时维护 Celery Worker 集群和 K8s 集群。
2. ExternalTaskSensor 的 `execution_date` 映射在时区差异或 Dag 更改 schedule 时可能失效。
3. Pool 槽位数需要持续调优——设得太少导致排队，设得太多失去隔离意义。
4. ELK 日志管道产生额外存储和计算成本——日增 100GB+ 日志需要合理设置索引生命周期。

### 适用场景

- **多业务线共用调度集群**：每条业务线有独立 SLA 和资源预算。
- **离线 + 在线任务混合调度**：部分任务需要 K8s 弹性扩容。
- **跨团队 Dag 依赖复杂**：需要 Sensor/Asset 做声明式级联。
- **对可观测性有高要求**：需要集中日志检索和分级告警。

### 不适用场景

- **单业务线、Dag 数量 < 20**：引入 Pool 隔离和混合执行器的复杂度远超收益。
- **所有任务都是秒级延迟**：Airflow 的最小调度间隔（scheduler loop delay）通常在数十秒级别，不适合实时系统。

### 验收标准

| 指标 | 目标 | 验证方法 |
|------|------|---------|
| Dag 总量 | ≥ 500 条 | `airflow dags list | wc -l` |
| 日 TaskInstance 数 | ≥ 50K | Postgres 查询 `COUNT(*) FROM task_instance WHERE start_date >= CURRENT_DATE` |
| P99 调度延迟 | < 30s | Prometheus `histogram_quantile(0.99, airflow_scheduler_task_instance_delay)` |
| Pool 隔离性 | 任一 Pool 满载不影响其他 Pool | 压测填满 ad_pool，验证 rec_pool 和 risk_pool 任务正常调度 |
| 告警送达 | P0 < 1 min, P1 < 5 min, P2 < 15 min | 人工注入故障后计时 |
| 日志检索延迟 | < 10s 可检索最新日志 | Kibana 搜索最新 1 分钟的日志条目 |
| 零数据丢失 | 无因调度系统自身故障导致的数据丢失 | 月报对比上游数据量和数仓入库量 |

### 常见踩坑经验

1. **Pool 槽位计入时机陷阱**：TaskInstance 在 `queued` 状态即占用 Pool 槽位，而非 `running`。如果 Executor 延迟拉取任务，Task 会白白占坑几十秒甚至几分钟。解决方案：警惕 Deferrable Operator 的 `include_deferred` 配置，避免 deferred 任务长期占用槽位。
2. **ExternalTaskSensor 死等**：如果上游 Dag 被删除或改名，Sensor 会永久 `poke` 直到 `timeout`。解决方案：始终设置 `timeout` 和 `failed_states`，配合 `soft_fail=True` 让 Dag 优雅降级而非卡死。
3. **KubernetesExecutor 镜像拉取超时**：大规模任务并发时，K8s 节点可能因镜像拉取带宽打满而超时。解决方案：使用镜像预热（DaemonSet 预拉）、本地 Registry 缓存、设置合理的 `startup_timeout_seconds`。

### 思考题

1. 广告 Pool 设了 32 slots，推荐 Pool 设了 20 slots。如果某天广告团队临时需要紧急跑一批竞价回溯任务（共 50 个 TaskInstance），而广告 Pool 已全部占满（32/32），有哪些不修改代码的应急方案？每种方案的优缺点是什么？

2. ExternalTaskSensor 依赖上游 Dag 的 `execution_date` 默认按 `same_date` 模式匹配（即 Sensor 的 `execution_date` 等于上游的 `execution_date`）。如果上游 Dag 的 schedule 从 `@daily` 改为 `@hourly`，下游的 Sensor 行为会如何变化？如果上游 Dag 的 `start_date` 晚于下游 Sensor 所探测的日期，会发生什么？
