# 第35章：ExecutionAPI内部机制与安全模型

## 1 项目背景

某安全审计团队在对 Airflow 部署环境进行渗透测试时，发现了一个危险的 Token 泄漏场景：一个 Worker 节点因为 OOM 被 K8s 重启后，旧的 JWT Token 没有被立即撤销，攻击者利用从内存 dump 中恢复的 Token 在 15 分钟内持续调用 Execution API 的 `/task_instances/{id}/heartbeat` 端点伪造心跳，导致本应被标记为 zombie 的 Task 一直保持在 running 状态，最终拖死了整个 DagRun。

另一个案例更隐蔽：团队在 Dag 文件中使用了 `get_connection()` 从 Airflow 元数据库读取加密的连接信息，Worker 进程通过 Execution API 的 `/connections/{conn_id}` 端点获取数据库密码。但开发者在调试日志中不小心打印了完整的 API 响应，密码以明文形式出现在了 Worker 的 stdout 日志中，被日志采集系统（ELK）索引后，任何有日志查询权限的人都能看到数据库密码。

这些案例揭示了 Airflow 3.x Execution API 安全模型的核心挑战：**如何在 Worker 与 API Server 之间的每一次交互中，确保最小权限、Token 生命周期可控、敏感数据不泄漏？**

> 本章将深入 Execution API 的端点设计、JWT Token 的完整生命周期、Worker 隔离模型的边界条件，以及 Supervisor 进程管理的底层机制。

## 2 项目设计

**小胖**（惊恐地盯着渗透测试报告）："Token 从内存 dump 中被提取出来还能用？这不就跟门禁卡丢了一样吗——谁捡到都能刷？"

**大师**："你的比喻很准确。JWT Token 本身是无状态的——一旦签发，在过期之前就是有效的。这就是为什么 Execution API 设计了两层防护：第一层是 Token 类型限制，第二层是 `ti:self` 范围约束。简单来说，Worker 拿到的 Token 只能在特定端点使用，且只能操作属于它自己的 TaskInstance。"

**小白**："Token 类型和 `ti:self` 具体是怎么工作的？"

**大师**："来看源码中的实现。Execution API 定义了两类 Token 类型：`execution` 和 `workload`。`execution` Token 是默认类型，Worker 用它来访问标准端点——如心跳上报、状态更新、连接获取。`workload` Token 是限制性更强的类型，只能访问明确标了 `Security(scopes=["token:workload"])` 的端点。

`ti:self` 更精妙——它通过 FastAPI 的路径参数和 Token 的 `sub` 字段做鉴权：`require_auth` 依赖检查 `request.path_params["task_instance_id"]` 是否等于 `token.id`。如果不等，直接返回 403。这意味着即使 Token 泄露了，攻击者也只能操作这个特定的 TaskInstance，不能横向移动。"

**小胖**："那 Worker 是怎么获取 Token 的？我不记得在代码里写过获取 Token 的逻辑。"

**大师**："Token 是由 Scheduler 在创建 Workload 时通过 `BaseExecutor.jwt_generator` 自动生成的。Worker 不需要主动获取——它从 Executor 分发的 Workload 中直接拿到 Token。源码中 `BaseExecutor.run_workload()` 调用 `supervise_task()` 时，Token 作为参数传入。Worker 通过 Execution API Client 发起请求时，自动在 HTTP Authorization 头中携带 `Bearer <token>`。"

**小白**："Supervisor 进程管理具体做了什么？它跟 Worker 是什么关系？"

**大师**："Supervisor 是 Task SDK 中的一个关键组件，它扮演'监工'角色。每个 Task 在 Worker 上执行时，不是直接运行在 Worker 主进程中——Supervisor 先 fork 出一个子进程，子进程运行实际的 Task 代码，Supervisor 通过 Unix socket pair 与子进程通信。

Supervisor 的职责包括三项：第一，**超时控制**——如果 Task 执行时间超过 `execution_timeout`，Supervisor 发送 SIGTERM 给子进程，超时后 SIGKILL。第二，**内存限制**——监控子进程的内存使用，超过限制时触发 kill。第三，**API 代理**——子进程不能直接访问网络，所有 Execution API 请求都由 Supervisor 代理转发。这形成了一个安全包围圈：子进程（Task 代码）→ Unix Socket → Supervisor → HTTPS → Execution API Server。"

**小胖**："如果 Supervisor 崩了怎么办？"

**大师**："好问题。`WatchedSubprocess` 设计了多层次的心跳机制。子进程定期通过 socket 发送心跳帧，Supervisor 检测心跳超时（默认 60 秒无心跳视为异常）。此外，Scheduler 也有独立的心跳超时检测——如果 TaskInstance 在 `task_instance_heartbeat_timeout` 秒内没有更新状态，Scheduler 会将其标记为 zombie 并 fail。这意味着即使整个 Worker 节点宕机，Scheduler 也能在超时后回收 TaskInstance。"

**小白**："Connection 泄漏的场景中，Token 过期后会怎样？"

**大师**："Token 过期后的处理分两个层面。第一，Execution API 的 `JWTBearer` 中间件在每次请求时校验 Token 的 `exp` 字段——过期的 Token 直接返回 401，Worker 必须获取新 Token。第二，`JWTReissueMiddleware` 可以在 Token 即将过期前自动签发新 Token，Worker 无需中断执行。关键是 Token 的有效期配置——默认 3600 秒（1 小时），需要在'安全性'（越短越安全）和'可用性'（避免频繁刷新）之间权衡。"

> **技术映射**：JWT Token = 酒店房卡（限定房间、过期失效），`ti:self` = 房卡只能开自己的房间，不能开别人的。Supervisor = 房间里的管家（代你打电话叫前台、送餐），隔离了住客与酒店后台。Token 过期 = 房卡到退房时间失效，需要前台续卡。

```python
# 来源: airflow-core/src/airflow/api_fastapi/execution_api/security.py

# JWTBearer 中间件 —— 每次请求执行一次加密验证
class JWTBearer(HTTPBearer):
    async def __call__(self, request: Request, services=DepContainer) -> TIToken | None:
        # 1. 检查缓存（同一个 request 内只验证一次）
        if cached := request.scope.get(_REQUEST_SCOPE_TOKEN_KEY):
            return cached
        
        # 2. 提取 Bearer Token
        creds = await super().__call__(request)
        
        # 3. 加密验证 + 签名校验
        validator: JWTValidator = await services.aget(JWTValidator)
        claims = await validator.avalidated_claims(creds.credentials, self.required_claims)
        
        # 4. 构造 TIToken，设置默认 scope
        claims.setdefault("scope", "execution")
        claim_model = TIClaims(**claims)
        token = TIToken(id=claims["sub"], claims=claim_model)
        
        # 5. 缓存到 request.scope
        request.scope[_REQUEST_SCOPE_TOKEN_KEY] = token
        return token


# require_auth 依赖 —— 路由级别的权限控制
async def require_auth(
    security_scopes: SecurityScopes,
    request: Request,
    token: TIToken = Depends(_jwt_bearer),
) -> TIToken:
    # 1. 验证 Token 类型白名单
    route = request.scope.get("route")
    allowed_token_types = getattr(route, "allowed_token_types", frozenset({"execution"}))
    
    if token.claims.scope not in allowed_token_types:
        raise HTTPException(status_code=403, detail="Token type not allowed")
    
    # 2. ti:self 检查（如果端点声明了此 scope）
    if "ti:self" in security_scopes.scopes:
        ti_self_id = str(request.path_params["task_instance_id"])
        if str(token.id) != ti_self_id:
            raise HTTPException(status_code=403, detail="Token subject mismatch")
    
    return token
```

### 3.3 阶段二：JWT Token 生命周期追踪

**步骤目标**：从创建到过期，完整追踪 JWT Token 的生命周期。

```python
"""
JWT Token 完整生命周期演示

阶段1: Token 创建（Scheduler 端）
    ↓
阶段2: Token 分发（通过 Workload 传递给 Executor）
    ↓
阶段3: Token 使用（Worker 调用 Execution API）
    ↓
阶段4: Token 刷新（JWTReissueMiddleware 自动续期）
    ↓
阶段5: Token 过期（JWTBearer 返回 401）
"""

import jwt
import time
from datetime import datetime, timezone, timedelta
from uuid import uuid4

# === 阶段1: Token 创建 ===
# 来源: BaseExecutor.jwt_generator (base_executor.py)
def create_task_token(task_instance_id: str, secret: str) -> str:
    """
    Scheduler 为每个 TaskInstance 创建一个短期 JWT Token。
    
    Token claims:
    - sub: task_instance_id (UUID 字符串)
    - scope: "execution" (默认)
    - exp: 签发时间 + jwt_expiration_time (默认3600秒)
    - aud: "airflow-execution"
    - iat: 签发时间 (UNIX 时间戳)
    """
    now = int(time.time())
    claims = {
        "sub": task_instance_id,
        "scope": "execution",
        "exp": now + 3600,      # 1 小时后过期
        "aud": "airflow-execution",
        "iat": now,
        "jti": str(uuid4()),    # 唯一 Token ID (方便追踪)
    }
    token = jwt.encode(claims, secret, algorithm="HS256")
    print(f"[创建] Token 已签发: sub={task_instance_id}, "
          f"exp={datetime.fromtimestamp(claims['exp'], tz=timezone.utc)}")
    return token

# === 阶段2: Token 分发 ===
# Scheduler → Workload → Worker
def dispatch_workload(token: str, task_info: dict):
    """
    模拟 Scheduler 将 Token 嵌入 Workload 分发给 Worker。
    """
    print(f"[分发] Workload 包含 Token: {token[:20]}...")
    return {"token": token, **task_info}

# === 阶段3: Token 使用 ===
# Worker 每次 API 请求都携带此 Token
def worker_make_api_call(token: str, endpoint: str) -> dict:
    """
    模拟 Worker 通过 Execution API Client 发起请求。
    Client 自动设置 Authorization 头。
    """
    headers = {"Authorization": f"Bearer {token}"}
    print(f"[使用] {endpoint} | Auth: Bearer {token[:20]}...")
    return {"status": "ok"}

# === 阶段4: 模拟 Token 刷新 ===
def should_refresh_token(claims: dict) -> bool:
    """
    JWTReissueMiddleware 的刷新判断逻辑:
    如果 Token 剩余有效期 < jwt_reissue_threshold (默认600秒)，自动签发新 Token
    """
    remaining = claims["exp"] - int(time.time())
    threshold = 600  # 10 分钟
    should_refresh = remaining < threshold
    print(f"[刷新检查] 剩余 {remaining}s, 阈值 {threshold}s, "
          f"需要刷新: {should_refresh}")
    return should_refresh

# === 阶段5: Token 过期处理 ===
def verify_expired_token(token: str, secret: str):
    """
    模拟 Token 过期后的验证失败。
    """
    try:
        claims = jwt.decode(token, secret, algorithms=["HS256"], audience="airflow-execution")
        print(f"[验证] Token 有效: sub={claims['sub']}")
        return claims
    except jwt.ExpiredSignatureError:
        print("[验证] Token 已过期!")
        return None

# 完整流程演示
if __name__ == "__main__":
    SECRET = "my-secret-key-for-demo-only"
    ti_id = str(uuid4())
    
    # 创建
    token = create_task_token(ti_id, SECRET)
    
    # 分发
    workload = dispatch_workload(token, {"dag_id": "etl_pipeline", "task_id": "extract"})
    
    # 使用
    worker_make_api_call(workload["token"], "POST /task_instances/{id}/heartbeat")
    
    # 刷新检查
    claims = jwt.decode(token, SECRET, algorithms=["HS256"], audience="airflow-execution")
    should_refresh_token(claims)
    
    # 过期验证（模拟未来时间）
    print("\n--- 模拟 Token 过期 ---")
    old_token = create_task_token(ti_id, SECRET)
    print(f"原本应该: 验证通过，但为了演示，这里展示过期时的 401 响应")
```

### 3.5 阶段四：Supervisor 进程管理实战

**步骤目标**：理解 Supervisor 的超时控制和子进程监控机制。

```python
"""
Supervisor 进程管理核心代码走读
来源: task-sdk/src/airflow/sdk/execution_time/supervisor.py

关键类: WatchedSubprocess → ActivitySubprocess
"""

# Supervisor 的工作流程（简化伪代码）
def supervise_task(ti, token, server, log_path, ...):
    """
    Supervisor 的主入口。
    
    1. fork 子进程执行实际 Task 代码
    2. 通过 socket pair 与子进程通信
    3. 代理子进程的 API 请求
    4. 处理超时和资源限制
    """
    # 创建通信 socket pair
    parent_sock, child_sock = socketpair()
    
    # fork 子进程
    pid = os.fork()
    if pid == 0:
        # === 子进程: 执行 Task 代码 ===
        child_sock.close()
        os.environ["_AIRFLOW_PROCESS_CONTEXT"] = "client"
        # 执行 task.execute()
        # 所有 API 调用通过 parent_sock 发送给 Supervisor
        task.execute(context)
        sys.exit(0)
    else:
        # === 父进程 (Supervisor): 监控和代理 ===
        parent_sock.close()
        
        # 事件循环: 同时处理
        # a) 子进程 → Supervisor 的 API 请求（转发给 Execution API）
        # b) 子进程的心跳超时检测
        # c) 子进程的内存监控
        # d) 子进程的退出信号处理
        
        supervisor_loop(parent_sock, ti, timeout_seconds=3600)


class WatchedSubprocess:
    """
    子进程观察者基类。
    
    核心属性:
    - _check_subprocess_exit(): 检查子进程是否仍然存活
    - _open_sockets: 与子进程通信的 socket 列表
    - is_ready: 进程退出 + socket 关闭 → True
    - heartbeat_timeout: 心跳超时阈值
    """
    
    def _monitor_heartbeat(self):
        """监控子进程心跳。超时后发送 SIGTERM，再超时发送 SIGKILL。"""
        now = time.monotonic()
        if now - self.last_heartbeat > self.heartbeat_timeout:
            self.log.warning("Subprocess heartbeat timeout, sending SIGTERM")
            os.kill(self.pid, signal.SIGTERM)
            self.sigterm_sent_at = now
            
            # 等待 grace_period 后强制 kill
            if now - self.sigterm_sent_at > self.grace_period:
                self.log.error("Subprocess did not exit after SIGTERM, sending SIGKILL")
                os.kill(self.pid, signal.SIGKILL)
```

```python
# Execution API 返回连接信息时，对密码字段做掩码处理
def get_connection(conn_id: str, session) -> ConnectionResponse:
```

## 4 项目总结

### 优点 & 缺点对比

| 维度 | Airflow 3.x Execution API | Airflow 2.x 直连数据库 |
|------|--------------------------|----------------------|
| 安全性 | Token + ti:self 双重鉴权，Worker 零 DB 权限 | Worker 直接持有 DB 连接 |
| 审计性 | 每次 API 调用可独立审计 | DB 日志混杂，难以区分来源 |
| 网络隔离 | Worker 只需 HTTPS 出站 | Worker 需 DB 端口可达 |
| Token 管理复杂度 | 需处理签发、刷新、过期 | 无 Token 概念 |
| 性能开销 | 每次 API 调用增加 HTTP 往返 | 本地数据库调用零延迟 |

### 适用场景

1. **安全合规环境**：需要审计每个 API 调用的金融、医疗行业。
2. **多租户部署**：不同团队的 Worker 通过不同 Token 隔离。
3. **网络隔离严格**：Worker 节点只能出站 HTTP/HTTPS。
4. **云原生部署**：Worker 运行在动态伸缩的 K8s Pod 中。
5. **敏感数据处理**：需要精细控制连接、变量访问权限。

### 不适用场景

1. **单机开发环境**：API 调用增加不必要的网络开销。
2. **极低延迟要求**：每个额外 HTTP 往返增加毫秒级延迟。

### 注意事项

- **Token 签名密钥**：生产环境必须使用非对称密钥（RS256），而非 HS256 的共享密钥。
- **Token 过期窗口**：`jwt_expiration_time` 应大于最长 Task 执行时间的 1.5 倍。
- **HTTPS 强制**：Execution API 应始终通过 HTTPS 提供服务，Token 明文传输在 HTTP 下可被中间人截获。
- **日志安全**：配置 Worker 日志级别时，Production 环境应至少为 WARNING，避免 API 响应被打印。

### 常见踩坑经验

1. **Token 过期导致 Worker 日志刷屏**：长时间运行的 Task 在 Token 过期后每个 API 调用都失败。解决：启用 `JWTReissueMiddleware` 自动续期，或将 `jwt_expiration_time` 调大。
2. **Supervisor OOM**：Supervisor 本身也消耗内存（用于 API 代理缓存），在大量并行 Task 时可能先于子进程 OOM。解决：设置 Supervisor 进程的内存上限，使用 `psutil` 监控。
3. **`ti:self` 限制过严导致回调失败**：Dag 级别回调没有对应的 TaskInstance ID。解决：Dag 回调使用独立的 Token，不绑定特定 TI，通过回调专用端点处理。

### 思考题

1. **进阶题**：如果要在 Execution API 中实现"Token 撤销"机制——一旦发现 Token 泄漏，可以立即使其失效——需要如何修改 `JWTBearer` 和 `JWTValidator`？请设计一个基于 Redis 的 Token 黑名单方案，考虑高可用和性能影响。

2. **设计题**：假设你的 Airflow 集群分布在 3 个数据中心，每个数据中心有自己的 Execution API Server 和 Worker 节点。请设计一个"跨数据中心 Token"方案——Worker 可以携带 Token 跨数据中心调用 Execution API，而不需要为每个数据中心签发独立 Token。考虑 `aud` (audience) 字段的使用和 JWT 的非对称签名方案。

> **推广计划提示**：安全团队重点阅读 `security.py` 中的 Token 验证流程和 `ti:self` 机制。开发团队理解 `Supervisor` 的进程管理模式。运维团队配置 Execution API 的 HTTPS、Token 过期时间和审计日志。测试团队设计 Token 泄漏和过期场景的集成测试。
