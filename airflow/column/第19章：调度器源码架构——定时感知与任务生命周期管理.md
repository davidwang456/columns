# 第19章：调度器源码架构——定时感知与任务生命周期管理

## 1 项目背景

某团队在生产环境运行 Airflow 一年后，遇到了一个难以复现的问题：偶尔有 Task 从 queued 状态直接跳到 skipped，中间没有 running 也没有 failed。排查整个日志链路后发现：在 Task 从 queued 到 Worker 取走的极短窗口内（< 1 秒），Worker 进程因为 OOM 被 K8s kill 了，Task 失联。Scheduler 的心跳检测发现超时后直接标记为 zombie 然后跳过。

更奇怪的是：同一天，有个 DagRun 的状态一直停留在 running，尽管所有 Task 都已经成功完成。原因是 DagRun 的状态更新逻辑在一个异常分支中被跳过了。

> 这些边界 case 暴露了对调度器源码理解的重要性。表面上看 Airflow 的调度"只是"创建 DagRun、分配 Task，但实际上涉及大量的边界条件判断、异常处理和并发控制。理解调度器的内部逻辑，是定位生产级疑难杂症的必备技能。

## 2 项目设计

**小胖**（对着一堆日志试图找出 Skipped Task 的原因）："调度器到底是怎么决定一个 Task 能不能跑的？"

**大师**："调度器的心跳循环大约每 5 秒运行一次。每次循环做三件事：1. 检查是否有新 DagRun 需要创建（基于 timetable 和 catchup）；2. 遍历所有 scheduled/queued 的 TaskInstance，检查能否提交执行；3. 处理超时、zombie 等异常情况。第二件事是核心——它包含了你刚才问的'可调度性检查'。"

**小白**："可调度性检查具体检查什么？"

**大师**："按顺序检查：第一，依赖是否就绪——上游 Task 是否都满足了 trigger_rule；第二，Pool 是否有可用槽位——这是计数器级别的检查；第三，并发限制——parallelism、max_active_tasks_per_dag、max_active_runs_per_dag；第四，Executor 容量——Executor 是否还能接收新任务。任何一个不满足，Task 就留在队列里等待下一次心跳。"

**小胖**："那 Adisory Lock 又是干啥的？"

**大师**："这是保证多 Scheduler 实例安全的机制。当你有 2 个 Scheduler 实例时，它们可能同时尝试调度同一个 Dag。Advisory Lock（数据库行级锁）确保同一时刻只有一个 Scheduler 实例在处理同一个 Dag——避免 DagRun 被重复创建、Task 被重复提交。"

> **技术映射**：Scheduler 心跳 = 交通信号灯的循环检测，可调度性检查 = 红绿灯转换前的路口情况评估（有没有行人、有没有紧急车辆），Advisory Lock = 交通管制的排他权（只有一名交警在指挥）。

### 3.2 可调度性检查实战

创建一个专门用于观察调度行为的 Dag：

```python
"""
调度行为观察 Dag
用于理解 Task 为什么排队、为什么被调度、什么时候被执行
"""
from datetime import datetime, timedelta
from airflow.sdk import DAG
from airflow.sdk.operators.python import PythonOperator
from airflow.sdk.operators.empty import EmptyOperator

with DAG(
    dag_id="scheduler_behavior_demo",
    schedule=None,
    start_date=datetime(2025, 1, 1),
    max_active_tasks=2,
    tags=["demo", "scheduler"],
) as dag:

    start = EmptyOperator(task_id="start")

    # 创建 8 个 Task，但限制 max_active_tasks=2
    # 观察哪些先跑、哪些排队
    def _worker(idx, **ctx):
        import time
        print(f"Task {idx} 开始执行...")
        time.sleep(15)  # 每个跑 15 秒，模拟耗时
        print(f"Task {idx} 完成")

    tasks = []
    for i in range(8):
        t = PythonOperator(
            task_id=f"worker_{i:02d}",
            python_callable=_worker,
            op_kwargs={"idx": i},
            pool="default_pool",
        )
        tasks.append(t)

    end = EmptyOperator(task_id="end")

    start >> tasks >> end
```

触发后观察：
1. Grid 视图中，同时只有 2 个 Task 在 running
2. 剩下的 6 个排队（queued 状态）
3. 当一个完成后，Scheduler 的下一个心跳周期从队列中取出下一个

### 3.4 调度延迟追踪

```python
def _track_scheduling_latency(**context):
    """追踪 Task 从创建到执行的每个时间节点"""
    ti = context["task_instance"]
    dag_run = context["dag_run"]

    print("=== 任务延迟分析 ===")
    print(f"DagRun 创建时间: {dag_run.queued_at}")      # DagRun 入队时间
    print(f"DagRun 开始时间: {dag_run.start_date}")      # DagRun 开始执行
    print(f"Task 入队时间: {ti.queued_dttm}")            # Task 被提交给 Executor
    print(f"Task 开始时间: {ti.start_date}")             # Worker 开始执行
    print(f"Task 结束时间: {ti.end_date}")               # Worker 执行完成

    # 计算各阶段延迟
    if dag_run.queued_at and dag_run.start_date:
        dr_delay = (dag_run.start_date - dag_run.queued_at).total_seconds()
        print(f"DagRun 排队延迟: {dr_delay:.1f}s")

    if ti.queued_dttm and ti.start_date:
        executor_delay = (ti.start_date - ti.queued_dttm).total_seconds()
        print(f"Executor 调度延迟: {executor_delay:.1f}s")

    if ti.start_date and ti.end_date:
        exec_time = (ti.end_date - ti.start_date).total_seconds()
        print(f"Task 执行耗时: {exec_time:.1f}s")
```

## 4 项目总结

### 调度器关键决策点

```
TaskInstance 在 queued 状态是否进入 running？
  ├── 1. 依赖检查：上游任务是否满足 trigger_rule？
  │   └── 否 → 留在 queued，等上游完成
  ├── 2. Pool 检查：目标 Pool 是否有空闲槽位？
  │   └── 否 → 留在 queued，等其他任务释放槽位
  ├── 3. 并发检查：是否超过 parallelism/max_active_tasks 限制？
  │   └── 是 → 留在 queued，等 running 数下降
  └── 4. Executor 检查：Executor 是否接受新任务？
      └── 否 → 留在 queued，等待 Executor 空闲
      └── 是 → 进入 running！
```

### 常见调度异常排查

| 症状 | 可能原因 | 排查方法 |
|------|---------|---------|
| Task 长时间 queued | Pool 槽位不足 | Admin → Pools 查看 |
| DagRun 一直 running | 最后一次 Task 状态未更新 | 检查 zombie Task |
| DagRun 不创建 | Dag Pause 或 catchup=False | 检查 dag 表状态 |
| 重复 DagRun | 多 Scheduler 抢锁 | 检查 advisory lock |
