# 第12章：数据资产与数据集调度——让数据驱动工作流

## 1 项目背景

某数据中台团队管理着 200+ 个 Dag，其中 70% 的 Dag 之间存在着隐式的数据依赖关系：Dag A 产出 `dwd_orders` 表，Dag B 消费 `dwd_orders` 生成 `dws_daily_sales`，Dag C 再消费 `dws_daily_sales` 生成报表。

在 Airflow 2.x 时代，这种依赖通过两种方式管理：
1. **时间假设**：Dag B 的 schedule 设得比 Dag A 晚 1 小时，期望 A 在 B 启动前完成
2. **ExternalTaskSensor**：Dag B 中用 Sensor 去等 Dag A 的某个 Task 完成

这两种方式都很脆弱。时间假设一旦上游延迟就全盘崩溃；ExternalTaskSensor 需要硬编码对方 Dag 的 dag_id 和 task_id，跨团队协作时沟通成本极高——A 团队的 Dag 改了 task_id，B 团队完全不知道，直到 B 的 Sensor 持续超时才被发现。

> Airflow 3.x 引入了 **Asset（数据资产）**机制来解决这个问题。它的核心理念是：**定义数据，而非定义依赖**。你不是说"Dag B 依赖 Dag A"，而是说"Dag B 消费 `dwd_orders` 表，当该表有新数据时自动触发"。具体的上游是谁、在哪个 Dag 中、什么时间产出——这些对下游完全透明。

## 2 项目设计

**小胖**（被跨团队依赖问题折磨了一周）："ExternalTaskSensor 太脆弱了——上游团队随便改个 task_id 我的 Dag 就挂了。Asset 真的能解决这个问题？"

**大师**："Asset 的本质是'数据的 URI 表示'，它把依赖从'谁生产的'变成了'生产了什么'。类比供应链：传统方式是你指定'必须从张三的工厂进货'（耦合到具体生产者），Asset 方式是你说'我需要零件 X'（只看产品规格，谁都可以生产）。张三分包给李四了？不影响你，只要零件 X 还在产出。"

**小白**："具体怎么用？"

**大师**："生产者在 Dag 定义中声明 `outlets=[Asset("my_table")]`——意思是'我这个 Dag 会产出 my_table 这个数据资产'。消费者用 `schedule=[Asset("my_table")]` 替代传统的 cron 表达式——意思是'当 my_table 有新数据时触发我这个 Dag'。Airflow 的 Scheduler 会自动追踪每个 Asset 的最新更新时间，在消费者 Dag 的调度条件满足时触发。"

**小胖**："那如果一个 Dag 既需要按时间跑，又需要等数据呢？"

**大师**："可以混合使用：`schedule=(某timetable) | Asset("table")`——满足任意条件就触发。或者用 `&` 表示必须同时满足。这是 Airflow 3.x 提供的 AND/OR 组合语义。"

**小白**："跨团队怎么避免 Asset URI 冲突？命名有什么规范？"

**大师**："Asset URI 是一个字符串，你需要团队的命名规范。比如 `team://data-warehouse/dwd/orders`、`s3://analytics-bucket/reports/daily/`。Airflow 支持通配符 `AssetAll`——例如消费者等的不是一个具体 Asset，而是某个前缀下的所有 Asset 更新。这在数据平台级依赖时非常有用。"

> **技术映射**：Asset 调度 = 快递到付通知——你不用盯着物流看快递到没到，到了自动通知你去取。传统 cron = 定时去快递柜看看——不管快递到没到，你都得跑一趟。

## 3 项目实战

### 3.1 基础 Asset 调度：生产-消费模型

**场景**：Dag A 每小时产出订单表，Dag B 在订单表更新后自动触发。

```python
"""
生产者 Dag：每小时产出 order_data asset
"""
from airflow.sdk import DAG, Asset
from airflow.sdk.operators.python import PythonOperator
from datetime import datetime, timedelta

order_asset = Asset(name="order_data", uri="table://warehouse/dwd/orders")

with DAG(
    dag_id="produce_orders",
    schedule=timedelta(hours=1),
    start_date=datetime(2025, 1, 1),
    catchup=False,
    outlets=[order_asset],  # 声明产出
    tags=["asset", "producer"],
) as producer:

    def _generate_orders(**context):
        print(f"生成 {context['ds']} 的订单数据...")
        return "order_data_updated"

    generate = PythonOperator(task_id="generate", python_callable=_generate_orders)
```

```python
"""
消费者 Dag：order_data 更新后自动触发
"""
from airflow.sdk import DAG, Asset
from airflow.sdk.operators.python import PythonOperator
from datetime import datetime

order_asset = Asset(name="order_data", uri="table://warehouse/dwd/orders")

with DAG(
    dag_id="consume_orders",
    schedule=[order_asset],  # 数据驱动：当 order_asset 更新时触发
    start_date=datetime(2025, 1, 1),
    catchup=False,
    tags=["asset", "consumer"],
) as consumer:

    def _process_orders(**context):
        # 获取触发本次运行的 Asset 列表
        triggering_assets = context.get("triggering_assets", [])
        print(f"触发资产: {[a.uri for a in triggering_assets]}")
        print(f"处理订单数据...")

    process = PythonOperator(task_id="process", python_callable=_process_orders)
```

**关键理解**：消费者 Dag 不再需要 cron 表达式或 timetale——它的调度完全由数据驱动。Scheduler 检测到 `order_asset` 被生产者更新后，自动创建消费者的 DagRun。

### 3.2 多 Asset 依赖：AND/OR 组合

```python
from airflow.sdk import DAG, Asset
from airflow.sdk.timetables.simple import CronDataIntervalTimetable

users_asset = Asset("table://warehouse/dwd/users")
orders_asset = Asset("table://warehouse/dwd/orders")

with DAG(
    dag_id="multi_asset_consumer",
    # 两个 Asset 都更新后触发（AND 逻辑）
    schedule=(users_asset & orders_asset),
    start_date=datetime(2025, 1, 1),
) as dag:
    ...
```

```python
with DAG(
    dag_id="any_asset_consumer",
    # 任意一个 Asset 更新就触发（OR 逻辑）
    schedule=(users_asset | orders_asset),
    start_date=datetime(2025, 1, 1),
) as dag:
    ...
```

### 3.3 Asset + Timetable 混合调度

```python
from airflow.sdk import DAG, Asset
from airflow.sdk.timetables.simple import CronDataIntervalTimetable

report_asset = Asset("table://analytics/daily_report")

with DAG(
    dag_id="hybrid_schedule",
    # 每天早上 8 点 + asset 更新时都触发
    schedule=(CronDataIntervalTimetable("0 8 * * *") | report_asset),
    start_date=datetime(2025, 1, 1),
) as dag:
    ...
```

### 3.4 Asset 别名与跨团队解耦

```python
from airflow.sdk import DAG, Asset

# 团队 A 产出一个内部 Asset
team_a_asset = Asset("s3://team-a-bucket/output/")

# 团队 B 定义一个别名，指向团队 A 的 Asset
# 如果团队 A 换了 bucket 或路径，只需改别名，不用改 B 的代码
AssetAlias(
    name="customer_data_ready",
    targets=[team_a_asset],
)

# 团队 B 消费别名
with DAG(
    dag_id="team_b_consumer",
    schedule=[Asset("customer_data_ready")],
) as dag_b:
    ...
```

### 3.5 通配符 Asset（AssetAll）

```python
from airflow.sdk import DAG, AssetAll

# 等待 warehouse 前缀下的所有 Asset 更新
with DAG(
    dag_id="aggregator",
    schedule=[AssetAll("table://warehouse/")],
) as dag:
    ...
```

## 4 项目总结

### Asset 调度 vs ExternalTaskSensor

| 维度 | ExternalTaskSensor | Asset 调度 |
|------|-------------------|-----------|
| 依赖方式 | 指定 dag_id + task_id | 指定数据 URI |
| 耦合度 | 强耦合（耦合到具体实现） | 弱耦合（只耦合到数据接口） |
| 跨团队 | 需沟通 task_id 变更 | 需约定 Asset URI 命名 |
| 一对多生产 | 不支持（一个 Sensor 等一个 Dag） | 天然支持（多个产生同一个 Asset） |
| 可观测性 | 无专门视图 | 独立的 Asset 视图 |

### 适用场景

1. **数据仓库分层调度**：ODS → DWD → DWS → ADS 各层自动级联
2. **ML 流水线**：特征工程完成后自动触发模型训练
3. **报表自动更新**：源表更新后报表自动刷新
4. **跨部门数据协作**：A 部门产出数据，B/C/D 部门各自消费

### 注意事项

1. **Asset URI 命名规范**：建议采用 `scheme://domain/entity` 格式，如 `table://warehouse/dwd/orders`
2. **Asset 的更新检测**：Airflow 通过 Task 的 `outlets` 声明来追踪更新，确保生产者的每个 Task 正确声明了产出
3. **不要滥用通配符**：`AssetAll("s3://")` 会匹配所有 S3 asset，几乎任何 Dag 完成都会触发——这相当于回到了 cron 的"到点就跑"
