# 第13章：回填与历史任务重跑——时间的倒带键

## 1 项目背景

周一早上 9 点，数据团队收到了产品经理的一条紧急消息："上周五的销售报表数据有问题，周五的订单数据在周日凌晨才补录完成，但我们的报表在周六就生成了，用的是不完整的数据。"

这意味着一件事：需要重跑上周五（以及受影响的周六、周日）的所有数据任务。整个链路是：MySQL 订单表（修复后）→ ODS 层 → DWD 层 → DWS 层 → 报表。工程师小刘需要从上周五开始，重跑整条链路 3 天的数据。

如果是在 cron 时代，小刘需要手动写一个 Shell 脚本，循环 3 天，逐个调用各脚本。更麻烦的是：有些依赖关系比较隐蔽——比如 DWD 层的一个表实际被 5 个下游报表使用，小刘得一个一个去找，确保没有遗漏。

在 Airflow 中，这个问题有一个专业的名字叫 **Backfill（回填）**。但"回填"不仅仅是"重跑"那么简单——它涉及 execution_date 的语义、幂等性设计、数据库压力控制、DagRun 的状态管理等诸多考量。

## 2 项目设计

**小胖**（刚被要求重跑 7 天数据）："还好有 Airflow！我直接把 Dag 的 start_date 往前改 7 天，然后设置 catchup=True 就行了吧？"

**大师**："千万别！这是一个非常危险的操作。修改 start_date + catchup=True 在技术上是可行的，但你要考虑：(1) 你的 start_date 原来可能是 2020 年，catchup=True 会瞬间创建上千个 DagRun；(2) 它会跑所有 7 天的数据，但你可能只需要跑部分 Task；(3) 如果已有部分成功的数据，会被覆盖或产生冲突。"

**小白**："那正确的姿势是什么？"

**大师**："两种方式：一是 **Backfill 命令**——`airflow dags backfill`，它能精准控制重跑的时间范围和是否重置已有的 DagRun。二是 **Clear 操作**——在 Web UI 中选择特定的 TaskInstance，清除状态让它重新执行。Backfill 适合整段时间的重跑，Clear 适合指定 Task 的修复。"

**小胖**："那幂等性呢？重跑不会导致数据重复吗？"

**大师**："这就是 Airflow 管不了的地方——**幂等性是你作为 Dag 作者的责任**。你的 Task 要设计成重复执行不会产生副作用。比如写入数据库时，使用 INSERT OVERWRITE 分区而不是 INSERT INTO，或者先检查数据是否已存在。Airflow 保证的是'按顺序执行'，但不保证'执行的结果不重复'。"

**小白**："Clear 操作会重跑下游吗？"

**大师**："这取决于你在 Clear 时选择的选项。默认 Clear 行为是：只清除当前选中的 TaskInstance，下游保持不变。但你可以选择 'Clear Downstream'（清除当前 + 所有下游），这在修复上游任务后需要级联重跑时非常有用。还可以选择 'Clear Upstream'（清除当前 + 所有上游），这很少用到。"

> **技术映射**：Backfill = 时间机器的"倒带键"——回到过去的某个时间点，重新执行那段历史。Clear = 手术刀——精准切除有问题的 TaskInstance，让它愈合（重新执行）。

## 3 项目实战

### 3.1 Backfill 命令详解

**步骤目标**：掌握 Backfill 命令的完整参数和用法。

```bash
# 基本语法
airflow dags backfill \
  -s START_DATE \      # 开始日期（含）
  -e END_DATE \        # 结束日期（含）
  --reset-dagruns \    # 是否重置已存在的 DagRun
  --rerun-failed-tasks \  # 仅重跑失败的任务
  dag_id
```

**常用模式**：

```bash
# 1. 重跑过去 3 天的所有 DagRun（即使之前成功过）
docker exec airflow-scheduler-1 airflow dags backfill \
  -s 2025-01-12 \
  -e 2025-01-14 \
  --reset-dagruns \
  order_etl_v1

# 2. 仅重跑失败的 Task，不碰成功的
docker exec airflow-scheduler-1 airflow dags backfill \
  -s 2025-01-12 \
  -e 2025-01-14 \
  --rerun-failed-tasks \
  order_etl_v1

# 3. 试运行（dry-run，不真正执行）
docker exec airflow-scheduler-1 airflow dags backfill \
  -s 2025-01-12 \
  -e 2025-01-14 \
  --dry-run \
  order_etl_v1

# 4. 限制并发 DagRun 数（避免冲垮数据库）
docker exec airflow-scheduler-1 airflow dags backfill \
  -s 2025-01-12 \
  -e 2025-01-14 \
  --max-active-runs 2 \
  order_etl_v1
```

### 3.2 Web UI Clear 操作

在 `dags/` 下创建 `backfill_demo.py` 来练习 Clear 操作：

```python
from datetime import datetime, timedelta
from airflow.sdk import DAG
from airflow.sdk.operators.python import PythonOperator
from airflow.sdk.operators.empty import EmptyOperator

with DAG(
    dag_id="backfill_demo",
    schedule="@daily",
    start_date=datetime(2025, 1, 12),
    catchup=True,
    tags=["demo", "backfill"],
) as dag:

    start = EmptyOperator(task_id="start")

    def _step1(**ctx):
        print(f"Step1: processing {ctx['ds']}")
        return f"data_{ctx['ds']}"

    step1 = PythonOperator(task_id="step1", python_callable=_step1)

    def _step2(**ctx):
        prev = ctx["task_instance"].xcom_pull(task_ids="step1")
        print(f"Step2: got {prev}")

    step2 = PythonOperator(task_id="step2", python_callable=_step2)

    def _step3(**ctx):
        import random
        if random.random() < 0.3:  # 30% 概率失败（模拟不稳定的任务）
            raise ValueError("模拟随机失败！")
        print("Step3: completed")

    step3 = PythonOperator(
        task_id="step3",
        python_callable=_step3,
        retries=2,
        retry_delay=timedelta(seconds=10),
    )

    start >> step1 >> step2 >> step3
```

**Clear 操作练习**：

1. 让 Dag 跑几个 DagRun，观察部分 `step3` 失败
2. 在 Grid 视图中右键失败的 `step3` 方块 → Clear → 选择 "Clear"（仅当前 Task）
3. 观察 `step3` 重试，但 `step2` 不变
4. 再试一次：右键失败的 `step3` → Clear → 勾选 "Downstream" → 观察下游也被清除

### 3.3 幂等性设计实战

```python
def _write_partition(**context):
    """
    幂等的分区写入
    无论执行多少次，结果都一样
    """
    ds = context["ds"]
    target_path = f"s3://warehouse/dwd_orders/dt={ds}/"

    # 幂等检查：如果分区已存在且有 _SUCCESS 标记，跳过
    import boto3
    s3 = boto3.client("s3")

    try:
        s3.head_object(Bucket="warehouse", Key=f"dwd_orders/dt={ds}/_SUCCESS")
        print(f"分区 {ds} 已存在且完整，跳过")
        return {"status": "skipped", "partition": ds}
    except Exception:
        pass  # 分区不存在，需要写入

    # 使用临时目录 + 原子重命名保证一致性
    tmp_path = f"/tmp/dwd_orders/{ds}/"
    os.makedirs(tmp_path, exist_ok=True)

    # 写入数据到临时目录
    with open(f"{tmp_path}/data.parquet", "w") as f:
        f.write("...")

    # 原子提交：上传完成后写入 _SUCCESS 标记
    s3.put_object(Bucket="warehouse", Key=f"dwd_orders/dt={ds}/_SUCCESS", Body=b"")

    print(f"分区 {ds} 写入完成")
    return {"status": "created", "partition": ds}
```

### 3.4 生产事故恢复 SOP

```
1. 确认影响范围
   - 哪个 Dag 的哪些 DagRun 受到影响？
   - 影响了哪些下游表/报表？

2. 修复根因
   - 修复数据源或代码 bug
   - 验证修复后单个 Task 的正确性（airflow tasks test）

3. Dry-run 验证
   - airflow dags backfill -s ... -e ... --dry-run
   - 确认将创建的 DagRun 数量和 Task 列表

4. 执行回填
   - 选择非业务高峰时段
   - 限制 --max-active-runs 避免数据库压力
   - 监控 Scheduler 和 DB 的负载

5. 验证结果
   - 检查所有 DagRun 状态为 success
   - 抽样验证数据正确性
   - 通知下游消费者
```

## 4 项目总结

### Backfill vs Clear vs 手动 Trigger

| 操作 | 适用场景 | 粒度 | 下游影响 |
|------|---------|------|---------|
| Backfill | 批量重跑一段时间 | 时间范围 | 按依赖自动级联 |
| Clear | 修复特定 Task | 单个/多个 TaskInstance | 可选：仅当前/含下游 |
| 手动 Trigger | 临时测试/补跑单次 | 单个 DagRun | 按依赖自动级联 |

### 常见踩坑经验

1. **"Backfill 不创建 DagRun"**：检查 Dag 是否被 Pause。Pause 状态下的 Dag 不会创建新的 DagRun，即使 Backfill 命令也不例外。需要先 Unpause。

2. **"Clear 后 Task 不自动重试"**：这是新手的常见误解。Clear 只是清除状态，Task 的重新执行是由 Scheduler 在下一个调度周期中触发的。如果 Scheduler 被 Pause，就不会重试。

3. **"Backfill 跑了一半，数据库满了"**：大量历史 DagRun 会创建大量 TaskInstance 和 Log 记录。解决方案：(1) 使用 `--max-active-runs` 限制并发；(2) 分批次执行，不要一次性回填几个月。

### 思考题

1. 一个 Dag 包含 50 个 Task，其中第 25 个 Task（`data_quality`）的数据质量检查逻辑需要修改，导致过去 7 天的 DagRun 都需要重新执行 `data_quality` 及其下游任务。你会选择 Backfill 还是 Clear？为什么？
2. Backfill 命令的 `--reset-dagruns` 参数会将已存在的 DagRun 的状态重置——但已经写入 S3 的数据文件是否也会被重置？如果不会，你怎么保证幂等性？
