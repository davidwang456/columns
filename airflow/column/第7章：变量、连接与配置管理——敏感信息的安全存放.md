# 第7章：变量、连接与配置管理——敏感信息的安全存放

## 1 项目背景

某电商团队的数据平台中，Airflow Dag 需要连接 12 个数据源：MySQL、PostgreSQL、Hive、Redis、MongoDB、S3、Kafka……每个数据源的连接信息（Host、端口、用户名、密码）散落在各个 Dag 文件的硬编码中。

一场安全事故彻底改变了这个现状：代码被意外推送到公共 Git 仓库，MySQL 生产库的密码随之泄露。安全团队紧急轮换所有密码，但由于密码硬编码在 30 多个 Dag 文件中，修改过程耗时 2 天。

与此同时，运维团队也在叫苦——测试环境和生产环境用的是同一套 Dag 代码，但数据库地址不同。每次部署前都需要手工修改十几个文件中的连接字符串。

> 这两个痛点指向同一个解决方案：Airflow 的配置管理体系——**Variable（变量）、Connection（连接）、环境变量注入**。它们让敏感信息和环境差异配置从代码中解耦出来，实现"代码不变，配置可变"。

## 2 项目设计

**小胖**（看着一个 Dag 文件里的明文密码直皱眉）："这也太不安全了吧……密码就这样写在代码里？"

**大师**："这正是 Airflow 设计 Connection 和 Variable 的原因。Connection 专门用来存储外部系统的连接信息——数据库、API、消息队列等。它在数据库中加密存储（如果配置了 fernet key），不在 Dag 代码中显式出现。Variable 则是通用的 key-value 存储，用来存放非连接类的配置——比如 API 超时时间、版本号、功能开关。"

**小白**："那它们的底层实现是什么？也是存数据库里？"

**大师**："是的，Connection 和 Variable 都有对应的数据库表。Connection 表包含 conn_id、conn_type、host、schema、login、password、port、extra 等字段。其中 password 如果配置了加密密钥（fernet key），会以密文形式存储。extra 字段是用 JSON 格式存储额外参数——比如 MySQL 连接的 charset、S3 连接的 endpoint_url。"

**小胖**："环境变量也能传配置，那什么时候用 Variable，什么时候用环境变量？"

**大师**："环境变量适合'部署时确定、运行时不变'的配置——比如数据库地址、执行器类型。Variable 适合'运行时可能修改'的配置——比如 Dag 的参数阈值、某个功能的开关。最大的区别：修改环境变量需要重启 Scheduler，修改 Variable 实时生效。"

**小白**："那 Secrets Backend 又是什么？"

**大师**："Connection 和 Variable 默认存在 Airflow 的元数据库中。但在企业级部署中，你通常已经有了一套密钥管理系统——比如 HashiCorp Vault、AWS Secrets Manager。Secrets Backend 让 Airflow 从这些外部系统读取密码，而不是自己存。这样即使 Airflow 的数据库被拖库，攻击者也拿不到密码。"

> **技术映射**：Connection = 银行卡（统一管理各支付渠道），Variable = 手机设置（随时可改，无需重启），Secrets Backend = 保险箱（加密存储，专人保管）。

## 3 项目实战

### 3.1 Connection 管理实战

**步骤目标**：创建和管理各类 Connection，在 Dag 中安全使用。

**步骤 1：创建 MySQL Connection**

方式一：Web UI

Admin → Connections → 点击 "+" 添加新连接：

```
Connection Id: mysql_warehouse
Connection Type: MySQL
Host: 10.0.1.50
Schema: warehouse_db
Login: etl_user
Password: ********
Port: 3306
Extra: {"charset": "utf8mb4", "connect_timeout": 30}
```

方式二：CLI

```bash
docker exec airflow-scheduler-1 airflow connections add mysql_warehouse \
  --conn-type mysql \
  --conn-host 10.0.1.50 \
  --conn-schema warehouse_db \
  --conn-login etl_user \
  --conn-password 'SecurePass123!' \
  --conn-port 3306 \
  --conn-extra '{"charset": "utf8mb4", "connect_timeout": 30}'
```

方式三：环境变量

```bash
export AIRFLOW_CONN_MYSQL_WAREHOUSE='mysql://etl_user:SecurePass123!@10.0.1.50:3306/warehouse_db?charset=utf8mb4'
```

方式四：Python 代码（批量创建）

```python
from airflow.models.connection import Connection
from airflow.utils import db

conn = Connection(
    conn_id="mysql_warehouse",
    conn_type="mysql",
    host="10.0.1.50",
    schema="warehouse_db",
    login="etl_user",
    password="SecurePass123!",
    port=3306,
    extra='{"charset": "utf8mb4"}',
)

# 保存到数据库
session = db.create_session()
session.add(conn)
session.commit()
```

**步骤 2：在 Dag 中使用 Connection**

```python
from airflow.providers.common.sql.hooks.sql import DbApiHook

def _use_connection(**context):
    """通过 Hook 使用 Connection"""
    # Airflow 自动根据 conn_id 从数据库加载连接信息
    hook = DbApiHook(mysql_conn_id="mysql_warehouse")
    with hook.get_conn() as conn:
        with conn.cursor() as cursor:
            cursor.execute("SELECT COUNT(*) FROM orders")
            count = cursor.fetchone()[0]
            print(f"订单总数: {count}")
    return count
```

**步骤 3：从 Connection 中获取特定字段**

```python
from airflow.hooks.base import BaseHook

def _get_connection_details(**context):
    """直接获取 Connection 对象的各个字段"""
    conn = BaseHook.get_connection("mysql_warehouse")
    print(f"Host: {conn.host}")
    print(f"Port: {conn.port}")
    print(f"Schema: {conn.schema}")
    print(f"Login: {conn.login}")
    # password 以密文存储，但通过 get_connection() 可以获取明文
    print(f"Password: {'*' * len(conn.password)}")

    # 解析 extra 中的 JSON 配置
    extra = conn.extra_dejson
    charset = extra.get("charset", "utf8")
    print(f"Charset: {charset}")
```

### 3.2 Variable 管理实战

**步骤目标**：创建并使用 Variable，理解其生命周期。

**步骤 1：创建 Variable**

```bash
# CLI
docker exec airflow-scheduler-1 airflow variables set api_timeout 30
docker exec airflow-scheduler-1 airflow variables set etl_config '{"threshold":0.15,"batch_size":1000}'

# 查看
docker exec airflow-scheduler-1 airflow variables get api_timeout
docker exec airflow-scheduler-1 airflow variables list
```

**步骤 2：在 Dag 中使用 Variable**

```python
from airflow.models.variable import Variable

def _use_variable(**context):
    """使用 Variable 读取配置"""
    # 方式一：直接获取（每次调用都查询数据库）
    timeout = int(Variable.get("api_timeout", default_var=30))
    print(f"API Timeout: {timeout}s")

    # 方式二：获取 JSON 并反序列化
    config = Variable.get("etl_config", deserialize_json=True)
    threshold = config["threshold"]
    batch_size = config["batch_size"]
    print(f"Threshold: {threshold}, Batch Size: {batch_size}")

    # 方式三：通过 Jinja 模板（适用于 BashOperator）
    # bash_command: 'echo {{ var.value.api_timeout }}'
    # bash_command: 'echo {{ var.json.etl_config.threshold }}'
```

**步骤 3：Variable 的缓存特性**

```python
# 注意：Variable.get() 在每次调用时默认查询数据库
# 对于高频调用的场景，使用环境变量替代或实现本地缓存

# 全局缓存示例（在 Dag 文件顶层）
from functools import lru_cache

@lru_cache()
def _get_config():
    return Variable.get("etl_config", deserialize_json=True)  # noqa
```

### 3.3 环境变量注入

**步骤目标**：通过环境变量覆盖配置，实现多环境部署。

```bash
# Airflow 环境变量命名规则
# 格式: AIRFLOW__{SECTION}__{KEY}

# 覆盖数据库连接
export AIRFLOW__CORE__SQL_ALCHEMY_CONN=postgresql+psycopg2://prod_user:pass@prod-host/airflow

# 覆盖 Variable
export AIRFLOW_VAR_API_TIMEOUT=60

# 覆盖 Connection
export AIRFLOW_CONN_MYSQL_WAREHOUSE=mysql://prod_user:prod_pass@prod-host:3306/prod_db
```

最佳实践：在 Docker Compose 中使用 `.env` 文件：

```bash
# .env
AIRFLOW_VAR_ENVIRONMENT=production
AIRFLOW_VAR_LOG_LEVEL=INFO
AIRFLOW_CONN_MYSQL_WAREHOUSE=mysql://prod_user:pass@10.0.1.50:3306/warehouse_db
```


### 3.5 完整 Dag：配置管理综合示例

```python
"""
配置管理综合示例
演示 Connection + Variable + 环境变量的最佳实践
"""
from datetime import datetime
from airflow.sdk import DAG
from airflow.sdk.operators.python import PythonOperator
from airflow.hooks.base import BaseHook
from airflow.models.variable import Variable
import os

default_args = {"owner": "config-demo", "retries": 1}

with DAG(
    dag_id="config_management_demo",
    default_args=default_args,
    schedule=None,
    start_date=datetime(2025, 1, 1),
    catchup=False,
    tags=["demo", "config"],
) as dag:

    def _read_config(**context):
        # 1. 读取环境变量（部署级配置，运行时不变）
        env = os.getenv("AIRFLOW_VAR_ENVIRONMENT", "development")
        print(f"Environment: {env}")

        # 2. 读取 Variable（运行时配置，可动态修改）
        timeout = Variable.get("api_timeout", default_var=30)
        print(f"API Timeout: {timeout}s")

        # 3. 读取 Connection（敏感信息，加密存储）
        conn = BaseHook.get_connection("mysql_warehouse")
        print(f"DB Host: {conn.host}:{conn.port}")
        print(f"DB Schema: {conn.schema}")
        print(f"Password stored: {'YES' if conn.password else 'NO'}")

        return {"env": env, "timeout": timeout}

    _read_config()
```

## 4 项目总结

### 配置管理方式对比

| 方式 | 存储位置 | 加密 | 热更新 | 适用场景 |
|------|---------|------|--------|---------|
| 环境变量 | OS 进程 | 否 | 需重启 | 部署级配置 |
| Variable | 元数据库 | Fernet | 实时 | 运行时参数 |
| Connection | 元数据库 | Fernet | 实时 | 外部系统登录 |
| Secrets Backend | Vault/AWS | 原生 | 实时 | 企业级密钥 |

### 注意事项

1. **Fernet Key 是加密的基础**：如果没有配置 fernet_key，Connection 的密码以明文存储。生成方式：`python3 -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`
2. **Variable 不适合存大数据**：每次读取都会查询数据库并序列化/反序列化。如果存一个 10MB 的 JSON，会严重影响性能。
3. **不要在 Dag 文件顶层调用 Variable.get()**：Dag Processor 每次解析都会执行，如果 Variable 很大会拖慢解析速度。
4. **Connection extra 字段是 JSON**：格式必须是合法的 JSON 字符串，否则 `extra_dejson` 会解析失败。

### 思考题

1. 你的团队有 dev / staging / prod 三套环境，如何设计 Variable 和 Connection 的命名和管理策略，确保以下两点：(1) 相同的 Dag 代码可以在三套环境运行；(2) 不会因为误操作把 prod 的 Dag 跑在 dev 的数据库上？
2. Fernet Key 也是密钥——它本身存在哪里？如果 Fernet Key 泄露了，你的 Connection 密码还安全吗？
