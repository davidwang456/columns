# 第9章：传感器与等待机制——灵敏的"守门员"

## 1 项目背景

某广告平台的实时报表系统依赖上游多个数据源：竞价日志由 Flink 实时写入 S3（每 5 分钟一个分区），用户画像数据由 Spark 批量更新（每天凌晨 4 点完成），第三方转化数据通过 SFTP 上传（时间不固定，通常上午 10 点前）。

过去使用 cron 调度，工程师只能"猜"时间——比如假设 S3 数据 5 分钟后到位，SFTP 文件 10 点前到。但实际上 Flink 偶尔延迟 20 分钟，SFTP 偶尔 11 点才到。一旦提前执行，就拿不到完整数据。

后来用了一个简单粗暴的解决方案：Shell 脚本里写 `sleep 1800`（等 30 分钟）。这在一段时间内有效——直到某天 Flink 延迟了 45 分钟，脚本还是 30 分钟后就开始跑，结果报表全错了。

> 这正是 Airflow **Sensor（传感器）**要解决的问题。Sensor 是一个"等待型"算子——它每隔 N 秒检查一次条件是否满足，条件满足即执行下游，超时则失败。它把"猜时间"变成了"等条件"。

## 2 项目设计

**小胖**（看着 Sensor 文档）："这不就是轮询吗？写个 while 循环 + sleep 也能实现啊。"

**大师**："技术上是的，但你写的 while 循环会一直占着一个 Worker 槽位。假设你有 10 个 Worker，10 个 Sensor 同时等，你的整个集群就被占满了——其他任务根本排不上队。Airflow 的 Sensor 有两种模式：`poke` 和 `reschedule`。`poke` 模式就是你说的死循环占用 Worker。`reschedule` 模式会把 Sensor 任务标记为 `up_for_reschedule`，释放 Worker 槽位，等下一个 poke_interval 再重新分配。"

**小白**："那 Deferrable Sensor 又是什么？跟 reschedule 有什么区别？"

**大师**："Deferrable Sensor 是 Airflow 3.x 的一大进步。传统的 reschedule 模式虽然释放了 Worker，但每次重调度都需要 Scheduler 参与——Scheduler 要把任务重新放入队列，Executor 再分配给 Worker。对于几百个 Sensor 在等的场景，Scheduler 会成为瓶颈。Deferrable Sensor 把'等待'的职责转移给了 Triggerer——一个专门处理异步等待的进程。Sensor 提交等待请求后，Worker 完全释放，Triggerer 在条件满足时通过回调触发下游。"

**小胖**："那常用的 Sensor 有哪些？"

**大师**："最常用的五种：
1. **FileSensor**：等待文件出现在指定路径
2. **ExternalTaskSensor**：等待另一个 Dag 的 Task 完成
3. **S3KeySensor**：等待 S3 对象的出现
4. **HttpSensor**：等待 HTTP 端点返回 200
5. **SqlSensor**：等待 SQL 查询返回特定结果"

**小白**："Sensor 超时了会怎么样？"

**大师**："会被标记为 failed。如果设置了 retries，还会重试。但注意 timeout 和 retry_delay 的计算——如果 timeout=3600，retry_delay=600，retries=3，实际最长等待时间 = 3600 × (1 + 3) = 4 小时。这个经常被低估。"

> **技术映射**：poke 模式 = 人在银行排队（一直占着柜台等），reschedule 模式 = 拿号等叫号（释放柜台但需要反复去确认），Deferrable Sensor = 手机APP排队提醒（完全不用管，到号就通知）。

## 3 项目实战

### 3.1 FileSensor：等待文件到达

**场景**：上游 Flink 任务写完文件后，下游 Airflow 才开始处理。

```python
from airflow.sdk import DAG
from airflow.sdk.operators.python import PythonOperator
from airflow.sdk.sensors.filesystem import FileSensor
from datetime import datetime, timedelta

with DAG(
    dag_id="file_sensor_demo",
    schedule="@daily",
    start_date=datetime(2025, 1, 1),
    catchup=False,
) as dag:

    # 等待上游数据文件
    wait_for_data = FileSensor(
        task_id="wait_for_data",
        filepath="/data/incoming/{{ ds }}/_SUCCESS",  # Hadoop 风格的成功标记
        fs_conn_id="fs_default",                       # 文件系统连接 ID
        poke_interval=60,                              # 每 60 秒检查一次
        timeout=3600,                                  # 最多等 1 小时
        mode="reschedule",                             # 释放 Worker 槽位
        soft_fail=True,                                # 超时后标记为 skipped 而非 failed
    )

    process_data = PythonOperator(
        task_id="process_data",
        python_callable=lambda **ctx: print(f"处理 {ctx['ds']} 的数据"),
    )

    wait_for_data >> process_data
```

**关键参数**：

| 参数 | 作用 | 典型值 |
|------|------|--------|
| filepath | 要等待的文件路径 | `/data/{{ ds }}/done.txt` |
| poke_interval | 检查间隔（秒） | `60` |
| timeout | 最大等待时间（秒） | `3600` |
| mode | poke / reschedule | `reschedule` |
| soft_fail | 超时后 skip 而非 fail | `True` |

### 3.2 ExternalTaskSensor：跨 Dag 依赖

**场景**：Dag B 必须在 Dag A 的某个 Task 完成后才能执行。

```python
from airflow.sdk.sensors.external_task import ExternalTaskSensor

with DAG(
    dag_id="dependent_dag_b",
    schedule="@daily",
    start_date=datetime(2025, 1, 1),
    catchup=False,
) as dag_b:

    # 等待上游 DagA 的 final_task 完成
    wait_for_upstream = ExternalTaskSensor(
        task_id="wait_for_dag_a",
        external_dag_id="upstream_dag_a",
        external_task_id="final_task",          # 具体等待哪个 Task
        execution_date_fn=lambda dt: dt,        # 匹配相同 execution_date
        timeout=7200,
        poke_interval=120,
        mode="reschedule",
    )

    my_task = PythonOperator(
        task_id="my_task",
        python_callable=lambda **ctx: print("DagB 开始执行"),
    )

    wait_for_upstream >> my_task
```

**关键注意事项**：
- `execution_date_fn` 决定了如何匹配两个 Dag 的 execution_date。默认是相同时间，但有些场景需要偏移（比如 DagB 的 execution_date = DagA 的 execution_date - 1 天）。
- ExternalTaskSensor 依赖 Dag A 的 Task 成功——如果是失败或 skipped，需要结合 `allowed_states` 和 `failed_states` 参数处理。

### 3.3 Deferrable Sensor：异步等待

在 Airflow 3.x 中，Sensor 默认支持 deferrable 模式：

```python
from airflow.sdk.sensors.filesystem import FileSensor

wait_for_data = FileSensor(
    task_id="wait_for_data",
    filepath="/data/{{ ds }}/_SUCCESS",
    mode="reschedule",     # 在 Airflow 3.x 中也支持 deferrable
    deferrable=True,       # 启用异步模式（由 Triggerer 处理）
    poke_interval=60,
    timeout=3600,
)
```

当 `deferrable=True` 时：
1. Sensor 提交等待请求给 Triggerer
2. Worker 释放槽位去执行其他任务
3. Triggerer 在后台异步等待条件满足
4. 条件满足后，Triggerer 通知 Scheduler 重新调度该 Task

**必须启动 Triggerer**：

```bash
# 在 Docker Compose 中确保 triggerer 服务已启动
docker compose up -d airflow-triggerer
```

### 3.4 自定义 Sensor

如果内置 Sensor 不满足需求，可以自定义：

```python
from airflow.sdk.sensors.base import BaseSensorOperator

class RedisKeySensor(BaseSensorOperator):
    """等待 Redis Key 存在"""

    def __init__(self, redis_conn_id: str, key: str, **kwargs):
        super().__init__(**kwargs)
        self.redis_conn_id = redis_conn_id
        self.key = key

    def poke(self, context):
        """每次检查时调用，返回 True 表示条件满足"""
        import redis
        from airflow.hooks.base import BaseHook

        conn = BaseHook.get_connection(self.redis_conn_id)
        r = redis.Redis(
            host=conn.host,
            port=conn.port,
            password=conn.password,
        )
        return r.exists(self.key)
```

**步骤目标**：避免 Sensor 耗尽所有 Worker 槽位。

```python
    ...

# 解决方案 3：使用 Deferrable 模式
# 将等待转移给 Triggerer，彻底释放 Worker
```

## 4 项目总结

### Sensor 模式对比

| 模式 | Worker 占用 | Scheduler 负载 | 适用场景 |
|------|------------|---------------|---------|
| poke | 全程占用 | 无 | 短暂等待（< 5 分钟） |
| reschedule | 释放，重调度 | 高（频繁重调度） | 中等等待（5 分钟 ~ 2 小时） |
| deferrable | 完全释放 | 低（Triggerer 接管） | 长时间等待（> 2 小时） |

### 适用场景

1. **文件到达等待**：Hadoop/S3 分区就绪、_SUCCESS 标记文件
2. **跨系统同步**：A 系统的 ETL 完成 → B 系统开始
3. **API 就绪检查**：新部署的服务健康检查
4. **数据质量前置**：等待某条件满足才继续
5. **时间窗口控制**：等待特定时间点（TimeSensor）

### 常见踩坑

1. **Sensor 死锁**：Worker 全被 Sensor 占满，其他任务无法执行。解决：独立 pool + deferrable 模式。
2. **ExternalTaskSensor 匹配不上**：execution_date 不匹配或 Dag 被 Pause。解决：检查两个 Dag 的 schedule 和 start_date 是否一致。
3. **timeout 设置过短**：上游偶尔延迟导致 Sensor 频繁超时，但不影响最终执行。解决：区分"等待超时"和"执行失败"——前者用 soft_fail=True。
