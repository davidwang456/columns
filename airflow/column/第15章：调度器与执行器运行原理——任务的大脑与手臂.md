# 第15章：调度器与执行器运行原理——任务的"大脑"与"手臂"

## 1 项目背景

某数据平台团队遇到了一个诡异的问题：50 个 Dag 中，有 3 个 Dag 每天准时创建 DagRun，但 Task 总是在 queued 状态卡住 10-30 分钟才开始 running。其他 47 个 Dag 一切正常。

初步排查发现：这 3 个 Dag 都有一个共同的 Sensor 在等待同一个外部文件——但 Sensor 的 poke_interval 是 60 秒，不应该阻塞其他任务。更奇怪的是，`airflow.cfg` 中 `parallelism` 设为了 32，实际运行的 Task 只有 10 个左右，远未触及上限。

问题出在哪里？答案在调度器的任务排队逻辑中：Pool 槽位不足。Sensor 虽然设置了 `mode=reschedule`，但每次 re-poke 时会重新申请 Pool 槽位。当 Sensor 数量和频率上升时，Pool 槽位被 Sensor 反复占用，其他任务只能排队等待。

> 这个案例揭示了理解调度器内部机制的重要性。Airflow 的调度器和执行器不是"魔法黑盒"——它们的每一步决策（创建 DagRun、判断可调度性、提交 Executor、分配 Worker）都有明确的逻辑和限制条件。

## 2 项目设计

**小胖**（看着 Pool 的 slots 配置）："Pool 不就是个计数器吗？为什么会导致任务排队？"

**大师**："Pool 确实是计数器，但它卡在调度链路的关键路径上。Scheduler 在决定'是否把一个 TaskInstance 提交给 Executor'前，会依次检查 5 个条件：1. 上游依赖是否就绪；2. 是否符合 trigger_rule；3. Pool 是否有可用槽位；4. 并发上限是否达到；5. Executor 是否有容量。任何一个条件不满足，Task 就继续排队。Sensor 的问题在于：它虽然释放了 Worker，但没有释放 Pool 槽位——或者说，每次重新调度都要再次申请槽位，如果槽位被别的 Sensor 占着，它就得等。"

**小白**："能画一下调度器和执行器之间的具体交互流程吗？"

**大师**："调度器循环大约每 5 秒（scheduler_heartbeat_sec）运行一次。每次循环中：首先检查哪些 Dag 需要创建新 DagRun（基于 timetable 和 catchup），然后遍历所有 newly-scheduled 的 TaskInstance，逐个做可调度性检查，最后将符合条件的 TaskInstance 通过 `executor.queue_command()` 提交给 Executor。Executor 拿到命令后，在自有的事件循环中把命令分配给 Worker 执行。LocalExecutor 直接在本地 fork 子进程；CeleryExecutor 通过 Redis 把消息发给远程 Worker。"

**小胖**："那 parallelism 和 dag_concurrency 的区别呢？"

**大师**："parallelism 是全局天花板——所有 Dag 加起来同时 running 的 Task 最多这么多。dag_concurrency（现在叫 max_active_tasks_per_dag）是单个 Dag 的天花板。二者是全域和局部的关系。还有一个容易被忽略的 max_active_runs_per_dag——它限制的是同一个 Dag 最多同时有几个 DagRun 在跑。比如每天一次的任务，如果上次跑了 23 小时还没结束，catchup 可能创建新的 DagRun——这时候 max_active_runs_per_dag 就起作用了。"

> **技术映射**：Scheduler = 机场塔台（决定哪架飞机可以起飞），Executor = 跑道和登机口（提供实际的起飞能力），Pool = 航空公司柜台（每个柜台只能服务一个乘客）。

### 3.2 Pool 机制实战

**步骤目标**：理解 Pool 如何影响任务调度。

**创建 Pool**：

```bash
# 创建一个只有 2 个槽位的 Pool
docker exec airflow-scheduler-1 airflow pools set slow_api_pool 2 "Slow API Pool"
```

**Dag 中使用 Pool**：

```python
from airflow.sdk import DAG
from airflow.sdk.operators.python import PythonOperator
from datetime import datetime, timedelta

with DAG(
    dag_id="pool_demo",
    schedule=None,
    start_date=datetime(2025, 1, 1),
    tags=["demo", "pool"],
) as dag:

    for i in range(8):
        PythonOperator(
            task_id=f"api_task_{i}",
            python_callable=lambda i=i, **ctx: print(f"Task {i} running"),
            pool="slow_api_pool",     # 使用自定义 Pool
            pool_slots=1,             # 占用 1 个槽位
        )
```

触发这个 Dag，观察 Grid 视图——8 个 Task 中只有 2 个会同时 running，其余 6 个排队等待。

### 3.3 三层并发控制测试

```python
"""
并发控制实战：parallelism vs max_active_tasks vs max_active_runs
"""
from airflow.sdk import DAG
from airflow.sdk.operators.bash import BashOperator
from datetime import datetime

# 全局配置（airflow.cfg）：
# parallelism = 32           ← 全局并发上限
# 单个 Dag 配置：

with DAG(
    dag_id="concurrency_demo",
    schedule=None,
    start_date=datetime(2025, 1, 1),
    max_active_tasks=4,       # 单 Dag 最多同时跑 4 个 Task
    max_active_runs=1,        # 单 Dag 最多同时 1 个 DagRun
    concurrency=4,            # 可同时执行的 TaskInstance 数（deprecated, 同 max_active_tasks）
    tags=["demo", "concurrency"],
) as dag:

    # 创建 20 个并发 Task，观察受限情况
    for i in range(20):
        BashOperator(
            task_id=f"task_{i:02d}",
            bash_command=f'echo "Task {i} running" && sleep 10',
        )
```

### 3.5 调度延迟观测

**步骤目标**：量化调度延迟。

```python
def _measure_latency(**context):
    """记录从 scheduled 到 running 的延迟"""
    ti = context["task_instance"]

    # start_date 是 Worker 开始执行的时间
    # queued_dttm 是 Scheduler 将任务入队的时间
    if ti.queued_dttm:
        delay = (ti.start_date - ti.queued_dttm).total_seconds()
        print(f"调度延迟: {delay:.2f} 秒")
    else:
        print("无法计算调度延迟（缺少 queued_dttm）")

    return delay
```

## 4 项目总结

### 调度链路瓶颈排查

```
TaskInstance 长时间 queued？
├── 检查 Pool 槽位是否充足
│   └── Admin → Pools → 查看 used_slots vs slots
├── 检查 parallelism 是否达到上限
│   └── 对比 running tasks 数 vs parallelism 配置
├── 检查 max_active_tasks_per_dag 限制
│   └── 对比该 Dag 的 running tasks 数
├── 检查 Executor 容量
│   └── Celery: 查看 Worker 数量和并发配置
│   └── K8s: 查看 Pod 创建是否成功
└── 检查上游依赖是否完成
    └── 检查 trigger_rule 和 upstream 状态
```

### 关键配置参数速查

| 参数 | 级别 | 作用 |
|------|------|------|
| `parallelism` | 全局 | 所有 Dag 同时 running 的最大 Task 数 |
| `max_active_tasks_per_dag` | Dag 级 | 单个 Dag 同时 running 的最大 Task 数 |
| `max_active_runs_per_dag` | Dag 级 | 单个 Dag 同时 running 的最大 DagRun 数 |
| `pool.slots` | Pool 级 | 资源池的最大并发数 |
| `pool_slots` | Task 级 | Task 占用的槽位数 |
| `scheduler_heartbeat_sec` | 全局 | 调度器循环间隔 |

### 常见踩坑经验

1. **"Task 一直 queued 但 Worker 空闲"**：通常是 Pool 槽位不够。检查 Admin → Pools，增加 slots。
2. **"Sensor 导致其他任务饿死"**：Sensor 的 reschedule 模式虽然释放 Worker，但占用 Pool 槽位。解决：给 Sensor 单独分配 Pool。
3. **"parallelism=32 但只有 10 个在跑"**：max_active_tasks_per_dag 限制了单 Dag 并发。如果只有 1 个 Dag 在跑，它最多跑 max_active_tasks 个。
