# 第 5 章：主键策略与字段填充注解

示例模块：`mybatis-plus-sample-crud` + 全局/实体策略（深入验证建议配合 `mybatis-plus-sample-id-generator`）。

## 1）项目背景

单表 CRUD 的第一步是：**主键怎么来**。自增、雪花、UUID、业务号段各有约束；MP 用 `@TableId(type = …)` 与 `global-config.db-config.id-type` 把策略声明在实体或配置里，避免每个 `insert` 前手写赋值。与主键相邻的 **`FieldFill`**（插入/更新填充）常与第 20 章 `MetaObjectHandler` 配合，本章先建立**注解层概念**。

**痛点放大**：策略与数据库实际类型不一致时，表现为**插入失败、回填为 null、分布式 ID 冲突**；团队若未文档化「新表默认主键策略」，每个服务会各自为政。

**本章目标**：能区分 `IdType` 常见取值；知道全局配置与实体注解的覆盖关系；能指出回填依赖 JDBC 与驱动的点。

## 2）项目设计：小胖、小白与大师的对话

**小胖**：主键不就让数据库自增吗？

**大师**：单库单体可以；分库分表、合并数据、对外暴露顺序敏感时，自增不一定是最佳。

**技术映射**：**@TableId** ≈ 身份证号规则；**IdType.ASSIGN_ID** ≈ 派出所按规则发号。

---

**小白**：`ASSIGN_ID` 和雪花啥关系？

**大师**：实现上常用雪花算法族；要关注时钟、长度与前端精度（大整数转 JSON）。

**小白**：插入后 `user.getId()` 何时有值？

**大师**：依赖 `useGeneratedKeys`、回填与 MP 的 `TableInfo` 处理；批量插入更要单独测。

**本章金句**：主键是**架构决策**，MP 帮你少写样板配置。

## 4）项目总结

| 优点 | 缺点 / 边界 |
|------|-------------|
| 声明式策略、减少重复赋值 | 选错策略线上难纠 |
| 与全局配置可分层治理 | 多数据源需每套核对 |

**适用场景**：新表设计；统一团队主键规范。

**不适用场景**：由数据库触发器/外部系统独占发号且禁止应用层干预。

**注意事项**：Oracle/PostgreSQL 序列与 `IdType.INPUT` 等组合见第 17 章。

**常见踩坑（案例化）**

1. **现象**：批量插入 id 全 null。**根因**：驱动回填与 `saveBatch` 实现差异。**处理**：对照版本文档与集成测试。
2. **现象**：前端 id 精度丢失。**根因**：JS Number。**处理**：字符串传输或 BigInt。
3. **现象**：与 DBA 自增策略冲突。**根因**：应用又发号。**处理**：单一真相来源。

**思考题**

1. `IdType.NONE` 与未写 `@TableId` 的实体在 MP 解析时有何差异？
2. 全局 `id-type` 与实体 `@TableId` 同时存在时谁优先？
