# 第 27 章：关联查询与 N+1 意识

示例模块：`mybatis-plus-sample-association`。

## 1）项目背景

多表关系在 MyBatis 中通常用 **嵌套查询 / 嵌套结果** 表达；MP 不试图用单表 Wrapper 替代所有关联。业务常见「用户带公司」「订单带明细」——需在 **一次 JOIN** 与 **N+1 查询** 之间做性能权衡。

**痛点放大**：`list()` 后懒加载或隐式触发多次查询，会在生产流量下放大为数据库压力；自定义分页 + 关联 SQL 时，count 与 list 语义需一起审。

**本章目标**：阅读 `AssociationTest` 与 Service 层自定义分页；理解 `userService.list()` 打印关联对象的含义；建立 N+1 排查意识。

## 2）项目设计：小胖、小白与大师的对话

**小胖**：关联查出来对象树一次搞定，多爽。

**大师**：爽的是**对象图**；痛的是**发了多少条 SQL**——注意 N+1。

**技术映射**：**嵌套查询** ≈ 分窗口取餐；**JOIN** ≈ 一次大托盘。

---

**小白**：MP 能像 JPA 一样 `@OneToMany` 全自动吗？

**大师**：MyBatis 系强调 **SQL 可控**；策略要你在 XML/注解里显式写。

**本章金句**：对象图有多丰满，SQL 就要有多清醒。

## 4）项目总结

| 优点 | 缺点 / 边界 |
|------|-------------|
| SQL 可控 | 映射维护成本高 |
| 与 Service 分页结合 | N+1 风险 |

**适用场景**：明确需要对象图、关联筛选。

**不适用场景**：可单表 + 缓存字典替代时。

**注意事项**：分页 count；与第 13 章 join count 优化对照。

**常见踩坑（案例化）**

1. **现象**：列表接口慢。**根因**：N+1。**处理**：改 JOIN 或批量预取。
2. **现象**：公司信息为空。**根因**：外键或映射列错。**处理**：核对 XML。
3. **现象**：事务内懒加载失败。**根因**：会话关闭。**处理**：Open Session In View 或 eager fetch。
