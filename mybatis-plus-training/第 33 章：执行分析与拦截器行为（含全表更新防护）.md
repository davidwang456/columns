# 第 33 章：执行分析与拦截器行为（含全表更新防护）

示例模块：`mybatis-plus-sample-execution-analysis`。在单元测试中演示**更新/删除**在「空条件 Wrapper」下的行为及异常捕获，用于强化**安全更新意识**。

## 1）项目背景

`update(entity, wrapper)`、`delete(wrapper)` 一类 API 若传入**空条件**的 `QueryWrapper`，在错误配置或误调用下可能触发**全表更新/删除**——这是生产事故高频区。MyBatis-Plus 通过**拦截器与策略**（随版本演进）对危险操作加以限制或抛错，行为需在**测试用例**中显式验证，而不是依赖「我记得我不会传 null」。

**痛点放大**：ORM 越方便，**一行代码毁全表**的风险越大；代码评审很难肉眼覆盖所有 Wrapper 组合。必须在团队规范里约定：**带条件的更新/删除默认走显式条件构造**，并对全表操作单独审批。

**本章目标**：跑通 `ExecutionTest`；理解示例中对 `MyBatisSystemException` 的捕获语义；将「执行分析」与日常 CRUD 习惯结合。

## 2）项目设计：小胖、小白与大师的对话

**小胖**：我就图省事写个 `update(new Student(), new QueryWrapper<>())`，为啥不行？

**小白**：空条件在语义上等于「没有 WHERE」，等价于全表更新；框架若放行，责任在谁？

**大师**：生产上**显式全表**与**误操作全表**要区分；框架侧往往默认**拒绝模糊语义**或要求插件开关，具体以当前 MP 版本文档为准。

**技术映射**：**空 Wrapper** ≈ 没写 WHERE 的 UPDATE/DELETE。

---

**小白**：那合法的全表更新怎么写？

**大师**：用**明确语义**的 API 或配置允许的策略，并在变更单里留痕；绝不能用「看起来像有条件其实没有」的写法糊弄过去。

**本章金句**：**危险不在 SQL 而在语义不清**——Wrapper 要写成人话能读懂的条件。

## 3）项目实战

**环境准备**：模块 `mybatis-plus-sample-execution-analysis`；`mvn -pl mybatis-plus-sample-execution-analysis -am test`。

**步骤 1：阅读集成测试**

目标：观察 `selectList`、`deleteById`、`insert`、`update` 组合，以及对空条件 `update`/`delete` 的异常处理。

```26:44:d:\software\workspace\mybatis-plus\mybatis-plus-samples\mybatis-plus-sample-execution-analysis\src\test\java\com\baomidou\samples\execution\ExecutionTest.java
    @Test
    void test() {
        studentMapper.selectList(new QueryWrapper<>());
        studentMapper.deleteById(1L);
        Student student = new Student();
        student.setName("test_update");
        studentMapper.insert(new Student(1L, "test", 12));
        studentMapper.update(student, new QueryWrapper<Student>().eq("id", 1L));
        try {
            studentMapper.update(new Student(), new QueryWrapper<>());
        } catch (MyBatisSystemException e) {
        }
        try {
            studentMapper.delete(new QueryWrapper<>());
        } catch (MyBatisSystemException e) {
            System.err.println("执行了全表删除拦截，删除无效！异常：" + e.getMessage());
        }
        Assertions.assertTrue(CollectionUtils.isNotEmpty(studentMapper.selectList(new QueryWrapper<>())), "数据都被删掉了.(┬＿┬)");
    }
```

**可能遇到的坑**：不同 MP 版本对「空条件」拦截策略不同，升级需回归本测试；勿在生产关闭安全策略换取「方便」。

## 4）项目总结

| 优点 | 缺点 / 边界 |
|------|-------------|
| 用短测试固化「危险操作」心智 | 需随版本更新用例断言与文案 |
| 与 Code Review 清单天然契合 | 不能替代权限与审计 |
| 可与全局拦截、租户插件组合理解 | 过度依赖异常路径影响性能时需评估 |

**适用场景**：团队规范培训；升级 MP 大版本后的回归；与运维「变更窗口」流程配套。

**不适用场景**：已使用存储过程全表维护且绕过 MyBatis——需在 DB 层单独管控。

**注意事项**：异常类型与信息**不要**作为稳定契约依赖在业务分支里；仅用于测试与日志。

**常见踩坑（案例化）**

1. **现象**：预发拦截、生产未拦截。**根因**：Profile 或 Bean 未加载同一套拦截器。**处理**：配置与打包一致性检查。
2. **现象**：误以为 `eq("id", null)` 会忽略条件。**根因**：对 Wrapper 生成规则理解错误。**处理**：显式判空与单元测试。
3. **现象**：批量任务误用 `delete(wrapper)`。**根因**：复用 Wrapper 未清空。**处理**：每次 new 或使用链式不可变风格。

**思考题**

1. 若业务确实需要「按租户 ID 全表软删除」，Wrapper 应如何写才不算「空条件」？
2. 与第 21 章租户插件叠加时，全表防护与租户条件谁先谁后？
