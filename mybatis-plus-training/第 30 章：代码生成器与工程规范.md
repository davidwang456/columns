# 第 30 章：代码生成器与工程规范

示例模块：源码仓 `mybatis-plus` 内 **`mybatis-plus-generator`**（如 `FastAutoGeneratorTest`）。

## 1）项目背景

从数据库元数据批量生成 Entity、Mapper、Service、XML，可加速脚手架。生成物是**起点**：表结构变更后的增量合并、手写代码与生成代码冲突，要靠流程解决。

**痛点放大**：生成器连接串、账号提交到 Git；生成目录覆盖手写业务——都是常见事故。

**本章目标**：阅读 `FastAutoGeneratorTest` 交互式入口；理解包名、作者、表名参数；建立「生成后必评审」的规范。

## 2）项目设计：小胖、小白与大师的对话

**小胖**：生成器跑完就完事。

**大师**：跑完是**第一公里**；**合并与维护**才是长途。

**技术映射**：**生成器** ≈ 毛坯房交付；**工程规范** ≈ 装修与物业。

---

**小白**：能进 CI 自动刷代码吗？

**大师**：可以，但要**确定性输入**与**diff 审查**，避免无人值守覆盖。

**本章金句**：生成器提升**起步速度**，不消灭**设计责任**。

## 3）项目实战

**环境准备**：克隆 `mybatis-plus` 源码；配置可连数据库；运行 `FastAutoGeneratorTest`（以当前包路径为准）。

**步骤 1：入口示例**

```28:46:d:\software\workspace\mybatis-plus\mybatis-plus\mybatis-plus-generator\src\test\java\com\baomidou\mybatisplus\generator\samples\FastAutoGeneratorTest.java
    public static void main(String[] args) throws SQLException {
        // 初始化数据库脚本
        initDataSource(DATA_SOURCE_CONFIG.build());
        FastAutoGenerator.create(DATA_SOURCE_CONFIG)
            // 数据库配置
            .dataSourceConfig((scanner, builder) -> builder.schema(scanner.apply("请输入表名称")))
            // 全局配置
            .globalConfig((scanner, builder) -> builder.author(scanner.apply("请输入作者名称")))
            // 包配置
            .packageConfig((scanner, builder) -> builder.parent(scanner.apply("请输入包名")))
            // 策略配置
            .strategyConfig((scanner, builder) -> builder.addInclude(scanner.apply("请输入表名，多个表名用,隔开")))
            .execute();
    }
```

## 4）项目总结

| 优点 | 缺点 / 边界 |
|------|-------------|
| 极大提速 | 模板与策略需统一 |
| 可团队封装 | 版本升级需回归 |

**适用场景**：新服务、表多且结构规整。

**不适用场景**：极度动态表、无固定 schema。

**注意事项**：勿提交数据源密钥；模板与 Lombok/注解策略对齐。

**常见踩坑（案例化）**

1. **现象**：生成包名错乱。**根因**：`parent` 配置错误。**处理**：模板校验。
2. **现象**：覆盖手写 Service。**根因**：输出目录与业务混放。**处理**：分目录或生成到临时区再合并。
3. **现象**：字段类型映射错。**根因**：JDBC 类型与驱动。**处理**：自定义 TypeConvert。
