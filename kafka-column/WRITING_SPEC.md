# Kafka 专栏写作规范（冻结版）

本文档冻结**单章结构、技术栈、示例仓库约定**，供所有作者与评审共用。修改本规范需经专栏编辑与至少一名技术负责人书面确认。

## 1. 单章结构与字数

每章必须包含四段，且按此顺序出现（与 [`templates/技术专栏优化模板：趣味性、实战性、深度兼顾.md`](../templates/技术专栏优化模板：趣味性、实战性、深度兼顾.md) 对齐；Kafka 专栏在「字数与冻结技术栈」上以本文件为准）：

1. **项目背景**：业务/技术动机、痛点、本章产出物（读者学完能做什么）；章首可有 **读法提示**（三角色如何分工）。
2. **项目设计**：**小胖、小白与大师** 三角色对话引出概念——**小胖**提直觉与「土问题」，**小白**追问原理与边界，**大师**给类比、选型与架构级结论；推进方式为「误解 → 纠正 → 类比 → 边界」。
3. **项目实战**：**主代码片段**驱动；至少包含「最小可运行示例」+「贴近生产的一种变体」（参数显式化 / 错误处理 / 指标埋点 三选一）；并含 **运维视角（提要）**、**测试视角（提要）**（可简写，与附录呼应）。
4. **项目总结**：**优点/缺点**、**适用场景**、**注意事项**、**常见踩坑**（固定清单，便于检索）；并含 **跨部门协作提要**（研发/运维/测试各 1～2 条，可简写）。

**字数**：**3000～5000 汉字**（含标点）。若含大量英文代码，可略上浮，但正文仍以汉字为主。

**比例建议**：对话与代码各占约 **25%～35%**；背景与总结占 **30%～50%**。

## 2. 冻结技术栈

| 组件 | 版本/约定 | 说明 |
|---|---|---|
| JDK | **17**（LTS） | 示例代码默认目标版本；若需 21 需脚注说明 |
| Kafka Broker | **3.7.x**（与 `examples/docker-compose.yml` 一致） | 升级小版本时同步更新本表与 compose |
| 客户端 | `org.apache.kafka:kafka-clients` **与 Broker 同主版本** | 在 `examples/pom.xml` 中声明 |
| 本地集群 | **KRaft 单节点**（`examples/docker-compose.yml`） | 避免引入 ZK 依赖，降低入门成本 |
| 操作系统 | 文档中说明 **Windows / Linux / macOS** 均可；路径用正斜杠或说明 PowerShell |

## 4. 代码与可复现性

- 每章实战须给出：**如何启动**（指向 `examples/docker-compose.yml` 或本章补充命令）、**如何运行**、**期望输出**（至少 2 行关键日志或指标）。
- 若使用 CLI，注明命令适用于 **Kafka 3.x** 脚本路径（容器内 `kafka-topics.sh` 等）。
- 禁止粘贴「依赖团队私服」的坐标；若需 Schema Registry / Connect，在章首说明 **可选** 并单独提供 compose 片段。

## 5. 对话与风格（小胖 / 小白 / 大师）

- **小胖**：生活化、口语化，提出直觉问题或常见误区（可与业务场景挂钩）；**不负责**长篇技术推导。
- **小白**：允许「队列直觉」类错误，但**每章最多 3 个典型误解**，且与后文「踩坑」一一对应；追问原理、边界与风险。
- **大师**：用类比与边界条件，不堆术语表；把结论落到 **业务约束、成本、团队协作**。
- 禁止：纯段子、与 Kafka 无关的冗长故事。

三角色 **开场白** 与 **读法提示** 句式见 `scripts/chapter_three_role_openers.py`；批量套用见 `scripts/apply_three_role_template.py`（幂等）。

## 6. 跨部门附录（每章文末可选）

- **研发**：设计评审问题清单（3～5 条）。
- **测试**：用例模板（正常/重复/乱序/断网/Broker 重启 中至少 3 类）。
- **运维**：指标或日志与告警映射（2～4 条）。

若正文已超字数上限，可将附录移至 `rollout/` 中对应模板，章内仅保留链接。
