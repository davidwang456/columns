# 第 3 章：Topic 与 Partition——并行与有序边界

> 级别：基础  
> 预计阅读：约 3000～5000 字  
> 配套示例：`examples/src/main/java/org/example/column/ch03/PartitionOrderingDemo.java`

---

## 1. 项目背景

在引入 Kafka 的团队里，最常见的争论之一是：**「我们的消息到底有没有顺序？」** 业务方希望「先下单、后支付、再发货」这样的因果顺序在系统里始终成立；架构师则反复强调「Kafka 只能保证分区内的顺序」。如果双方对 **Topic、Partition、Key** 的关系没有共同语言，设计评审很容易变成各说各话：开发按「全局队列」去想象，测试按「单线程」去造数据，运维看到分区不均衡时也不知道该找产品改键还是改分区数。

本章的项目背景可以概括成一句话：**Kafka 的并行能力主要来自 Partition，而「顺序」被严格限制在「同一分区内的追加顺序」**。读者在真实项目里经常遇到的痛点包括：把「多分区」当成「多副本」、把「提高并行」误解成「增加副本」、把「全局有序」硬塞进 Kafka 而不改键设计。本章不追求讲完所有分区策略细节（那会在生产调优章节展开），而是建立**可验证的心智模型**：你能用一段最小代码证明「相同 Key 进同一分区」「不同 Key 分散到不同分区」，并能在总结里向业务方解释**代价与边界**。

本章的产出物包括三项：**（1）** 能口述 Topic 与 Partition 的职责分工；**（2）** 能说明「有序」在 Kafka 中的精确定义；**（3）** 能在本机跑通示例程序，观察发送结果中的 `partition` 字段，并解释它与 Key 的关系。

从数据建模角度，**Topic** 更像「逻辑表名」或「一类事件的名字空间」：生产者与消费者通过同名 Topic 对接契约（字段、版本、兼容性在序列化章节展开）。**Partition** 则是 Topic 在物理上的分片：Broker 为每个分区维护一段**只追加日志**，每条消息在分区内获得单调递增的 **offset**。因此，「顺序」若指 offset 的增长方向，则只在**同一分区内**成立；跨分区比较 offset 或时间戳都没有统一语义。很多团队第一次压测时发现 TPS 上不去，第一反应是加机器，但若分区数不足，消费者并行度仍被卡住——这进一步说明：**并行与顺序的杠杆首先是分区设计，而不是单纯扩容硬件**。

## 2. 项目设计：大师与小白的对话

**小白**：我们 Topic 叫 `orders`，我打算开一个消费者线程不断 `poll`，这样所有订单就按时间顺序处理，对吗？

**大师**：可以把它想成一本**可并行阅读的账本**，而不是超市门口唯一的排队通道。`orders` 这个名字下面，通常会切成多个 **Partition（分区）**。每个分区内部是**只追加的日志**，所以分区内顺序是明确的；但**跨分区之间没有全局先后顺序**。你用一个消费者线程去读，只是「读法」上可能只订阅了一个分区，或者组内分配让你看起来串行——这不等于 Kafka 承诺了全局有序。

**小白**：那我把分区数设成 1，不就全局有序了吗？

**大师**：对，**单分区**时，Topic 层面确实可以谈「全 Topic 有序」，但代价是**并行度为 1**：写入与消费扩展都受这一根「独木桥」限制。生产里常见的是：业务上只需要「同一用户」或「同一订单」有序，而不是全站有序——这时应该用 **Key** 把相关消息路由到同一分区，让并行与有序在「业务键维度」上同时成立。

**小白**：Key 我懂，就是消息的主键嘛，我随机生成 UUID 当 Key，每条消息都独一无二，性能最好？

**大师**：这恰恰是「看起来对但不对」的典型。**UUID 当 Key** 往往会让分区器把消息均匀打散——并行度上去了，但**同一业务实体**的消息可能落到不同分区，**顺序与因果就丢了**。反过来，如果所有消息都用**同一个 Key**，又会把并行度打回单分区热点。边界在于：**Key 的粒度要与你需要的「有序单元」一致**，而不是越随机越好。

**小白**：那分区数和副本数是不是一回事？我多建几个副本，是不是就等于多分几个并行通道？

**大师**：不是。**副本（Replication）**解决的是**可用性与持久化**，同一条数据在不同 Broker 上的拷贝；**分区**解决的是**伸缩与并行**。你可以有「3 个分区、每个分区 3 副本」这种组合；但副本不会增加「可并行的消费者分片」——消费者组里的并行度，基本以**分区数**为上限（在常规语义下）。

**收束**：本章实战的验收标准是：在 **3 分区** Topic 上，**固定 Key** 连续发送时，`RecordMetadata` 中的分区号保持不变；**随机 Key** 发送时，分区号在多轮运行中通常分散。你能用日志向团队演示这一点，并把「全局有序 vs 分区内有序」说清楚。

补充三个「看起来对但不对」的推论，便于与后文踩坑对照：**推论 A**，「分区越多，顺序越差」——顺序与分区数量无直接因果，只与**是否跨分区**有关；**推论 B**，「只要消费者单线程就全局有序」——若 Topic 多分区，单线程轮询仍可能按分区交错读到不同顺序的「业务时间」；**推论 C**，「没有 Key 的消息会进固定分区」——在默认分区器下，null Key 往往走**轮询或粘性分区**策略，与「固定分区」不是一回事。

## 3. 项目实战

### 3.1 环境说明

按 [examples/docker-compose.yml](../examples/docker-compose.yml) 启动本地 KRaft 单节点 Broker，监听 `localhost:9092`。客户端版本与 [examples/pom.xml](../examples/pom.xml) 中的 `kafka-clients` 对齐（当前冻结为 **3.7.0**）。

### 3.2 主代码片段：分区与 Key 的关系

下面示例做两件事：**（1）** 用 `AdminClient` 在开发环境创建 `column.ch03.orders`（3 分区、副本因子 1，单节点仅能 1）；**（2）** 用 `KafkaProducer` 分别发送「固定 Key」与「随机 Key」消息，并打印每条记录落到的 **partition**；最后用**临时消费者组**从 `earliest` 读取，观察消费侧看到的分区与位移。

核心类：`org.example.column.ch03.PartitionOrderingDemo`。

```java
// 节选：同一固定 key 连续发送；随机 key 发送；打印 RecordMetadata.partition()
ProducerRecord<String, String> fixed =
    new ProducerRecord<>(TOPIC, fixedKey, "order-" + i);
var md = producer.send(fixed).get();
// md.partition() 即为落盘分区

ProducerRecord<String, String> rnd =
    new ProducerRecord<>(TOPIC, "sess-" + UUID.randomUUID(), "evt-" + i);
var md2 = producer.send(rnd).get();
```

完整实现见仓库文件，此处强调阅读点：**`ProducerRecord` 的 Key 参与分区选择**（默认分区器下），Value 不参与；**null Key** 时的行为（粘性分区等）与「显式 Key」不同，属于后续进阶章节。

读者可以在 IDE 中单步调试 `producer.send` 返回的 `RecordMetadata`：对比 `partition()`、`offset()` 与 `serializedKeySize()`。若团队使用**自定义分区器**（实现 `Partitioner` 接口），则「Key → 分区」的映射可能与默认哈希不同，这是多租户或地域路由场景的常见扩展点——本章仍建议先掌握默认行为，再进入定制。

消费侧示例使用**随机 `group.id`**，是为避免本地反复调试时因已提交位移而「看不到消息」；生产环境则必须显式治理消费者组与位移策略。读取时打印 `ConsumerRecord` 的 `partition()` 与 `offset()`，可与发送侧 `RecordMetadata` 对照，建立端到区映射直觉。

**验证步骤**：

1. `cd kafka-column/examples && docker compose up -d`，等待 Broker 就绪。
2. `mvn -q compile exec:java`（默认主类已指向 `PartitionOrderingDemo`）。
3. 观察控制台：前五行「固定 Key」应对应**同一 partition 号**；后五行「随机 Key」的 partition 号**不必相同**。

**期望输出（节选，partition 数字以本机为准）**：

```text
已创建 topic: column.ch03.orders partitions=3
--- 发送结果：分区号（同一固定 key 应落在同一分区） ---
fixed:0 -> partition 2
fixed:1 -> partition 2
...
random:0 -> partition 1
random:1 -> partition 0
...
消费: partition=2 offset=... key=user-42 value=order-0
```

若固定 Key 却出现不同分区，优先排查：**是否误用了不同 Key**、**是否中途更换了分区器**、**是否连到了另一集群**。

**与 CLI 的对照（运维同屏）**：在容器内使用 `kafka-topics.sh --describe` 查看 Topic 的分区与 Leader 分布；使用 `kafka-console-producer.sh` 指定 `--property parse.key=true` 与键值分隔符时，可手工验证「键 → 分区」是否与客户端一致。Java 示例侧重**可编程重复**，CLI 侧重**现场排障**；两者结合能减少「代码里是一个分区、命令行里是另一个」的沟通成本。

### 3.3 贴近生产的变体：显式参数

生产环境很少使用「每次 `send().get()`」这种同步写法；通常会配置 **`linger.ms`、`batch.size`、`compression.type`** 等以权衡延迟与吞吐。本章变体建议：在 `ProducerConfig` 中显式设置 `linger.ms=5`、`batch.size=32768`、`compression.type=lz4`，再对比端到端延迟与批次行为（可用日志或指标观察）。注意：**分区选择发生在发送管道内**，与批处理并行不矛盾，但异步发送时要用回调或 `flush()` 保证演示时的输出顺序可读。

## 4. 项目总结

### 4.1 优点

- **并行扩展清晰**：增加分区（在合理范围内）可提高消费者并行度，与业务吞吐对齐。
- **顺序语义可组合**：通过 Key 将「需要顺序的业务单元」绑定到同一分区，其余分区仍可并行处理其他实体。
- **模型与运维指标一致**：分区是 Lag、重平衡、热点排查的基本单位，便于开发运维使用同一套语言。

### 4.2 缺点

- **无免费的全局有序**：多分区下若坚持全局顺序，只能牺牲并行或在外层做排序（成本高）。
- **Key 设计错误代价大**：不当 Key 会导致热点分区或顺序丢失，且往往在流量放大后才暴露。
- **分区数不是越多越好**：过多分区会带来元数据、文件句柄、选举与复制开销（高级篇详述）。

### 4.3 适用场景

- **需要「实体级顺序」**：如用户行为、订单生命周期、设备事件，用用户 ID、订单 ID 等做 Key。
- **可接受「分区内顺序即可」的管道**：日志、点击流、异步任务，通常用随机 Key 或不指定 Key 换更高并行。

### 4.4 注意事项

- 修改分区数、分区器或 Key 策略前，评估**消费者依赖顺序的代码路径**。
- 单分区 Topic 适合极低并行或强全局顺序场景，但要预先评估**峰值写入与单线程消费**瓶颈。
- 测试环境 Topic 自动创建策略与生产可能不一致，示例中显式 `createTopics` 可避免「本地能跑、上线分区数不对」。

### 4.5 常见踩坑

- **误以为「Topic 名」自带顺序**：顺序来自**分区内日志**，不是来自名字。
- **用时间戳当 Key 追求顺序**：若时间戳在业务上每毫秒唯一，分区会极度分散；若粗到「按分钟」，又可能热点。
- **忽略消费者组与分区的关系**：再均衡时分区在实例间迁移，处理逻辑必须**幂等**，不能假设「永远只处理某分区」。

**跨团队沟通提示**：当产品说「订单必须按时间处理」时，应追问时间戳是**事件发生时间**还是**写入 Kafka 时间**；前者若跨分区，Kafka 不保证全局序，后者还受时钟与重试影响。把「有序」定义成**分区内 offset 递增**或**业务键维度因果**，能显著减少返工。

**与后续章节的衔接**：当你已经理解「分区是并行与有序的交界」，第 7、15、31 章会进一步解释消费者组如何把分区分配给实例，以及再均衡为何会让「看起来像顺序」的业务在短暂窗口内出现重复或缺口；第 14、18、30 章则讨论重试与事务如何与分区语义叠加。把本章的直觉带过去，阅读那些章节时就不容易迷失在参数名里。

## 附录（可选）

### 研发：设计评审问题清单

- 业务要求的「有序」是全局还是「用户/订单」维度？
- Key 的基数与分布是否会导致热点分区？
- 分区数与消费者实例数、下游并行度是否匹配？

### 测试：用例模板

- 正常：同一 Key 多条消息，断言处理顺序与发送顺序一致（单分区内）。
- 重复：重放同一分区消息，业务是否幂等？
- 断网：Producer 重试后是否可能重复写入（与第 14 章衔接）。

### 运维：变更与告警映射

- **分区不均衡**：部分 Broker 磁盘与请求热点 → 检查 Key 分布与分区分配。
- **消费 Lag 按分区倾斜**：常与热点 Key 或慢消费者有关，而非单纯「加机器」能解决。
