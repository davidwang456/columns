# 第 6 章：Consumer 基础——poll、提交位移

> 级别：基础  
> 预计阅读：约 3000～5000 字（正文以汉字为主，含标点）  
> 配套示例：`examples/src/main/java/org/example/column/ch06/ConsumerPollCommitDemo.java`

---

## 1. 项目背景

生产者把消息写入分区日志后，**消费者**通过 **`poll`** 批量拉取 **`ConsumerRecord`**，并在处理进度上维护 **位移（offset）**。新手上常见的痛点是：**以为 `poll` 像 HTTP 一样「请求-响应」**、**搞不清自动提交与手动提交**、**处理失败却提交了位移** 导致 **丢消息**，或 **重复处理** 却未做幂等。

本章产出物：**（1）** 能解释 **`poll(Duration)`** 的语义与「可能返回空集合」；**（2）** 能区分 **`enable.auto.commit`** 与 **`commitSync`/`commitAsync`**；**（3）** 能跑通示例：关闭自动提交、处理一条、**同步提交**，并理解位移写入 **内部 Topic**（第 8 章展开）。

从 **API 角度**，`KafkaConsumer` 不是「拉一条算一条」的朴素接口：`poll` 返回的是 **ConsumerRecords**，内部可含 **多个分区** 的多条记录。处理循环里常见的两种风格是：**逐条处理 + 逐条提交**（简单但 RPC 多），与 **批量处理 + 按分区提交**（吞吐更好但代码复杂）。入门阶段先掌握 **同步提交**，再演进到 **异步与批量**，避免一上来就叠加 **多线程消费** 与 **EOS** 两个复杂度。

**参数预告**：`max.poll.records` 控制单次 `poll` 可能返回的最大条数；`fetch.min.bytes` / `fetch.max.wait.ms` 影响 **拉取侧批聚**；`session.timeout.ms` 与 **`max.poll.interval.ms`** 与 **存活与重平衡** 强相关（第 7、31 章）。本章只要求建立「**poll 是批、提交是进度**」的直觉。

## 2. 项目设计：大师与小白的对话

**小白**：我 `while(true)` 里 `poll`，没有消息就 sleep，对吗？

**大师**：`poll` 本身会 **阻塞至多 Duration**；返回空表示 **当前没有可拉取数据** 或 **仍在重平衡** 等。盲目再套一层 `sleep` 可能增加延迟；更重要的是设置合理的 **Duration**，并在循环里处理 **再均衡监听器**（第 7、15 章）。

**小白**：消费成功就 `commit`，失败就不 `commit`，这样最简单？

**大师**：思路对，但要注意：**处理失败** 时若不 commit，可能 **反复消费同一条**——业务要 **幂等**；若先 commit 再处理，可能 **丢处理**。**提交时机**与 **业务原子性** 要对齐（第 14 章）。

**小白**：位移存在消费者本地吗？

**大师**：消费者本地有缓存，但 **已提交位移** 最终记录在 **组协调器** 管理的内部 Topic（`__consumer_offsets`）；重启后按组 **恢复**（第 8 章）。

**收束**：验收标准：示例跑通，日志中可见 **partition/offset**，并打印 **手动 commit** 提示。

三个推论：**推论 A**，「`poll` 一次就一条」——常是 **一批**；**推论 B**，「不 commit 就永远从头读」——取决于 **auto.offset.reset** 与 **是否曾提交**；**推论 C**，「消费慢就加线程」——可能要先看 **分区数** 与 **并行模型**。

## 3. 项目实战

### 3.1 环境说明

[examples/docker-compose.yml](../examples/docker-compose.yml)，`localhost:9092`；客户端 **3.7.0**。

### 3.2 主代码片段

类 `org.example.column.ch06.ConsumerPollCommitDemo`：**创建 Topic** → **Producer 写入** → **Consumer `subscribe`** → **`enable.auto.commit=false`** → 循环 **`poll`** → 每条 **`commitSync`**。

```java
// 节选：关闭自动提交 + 手动 commitSync
props.put(ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG, "false");
ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(500));
for (ConsumerRecord<String, String> r : records) {
    // 业务处理
    consumer.commitSync(); // 演示：逐条提交；生产常按批次/分区
}
```

**验证**：`mvn -q compile exec:java -Dexec.mainClass=org.example.column.ch06.ConsumerPollCommitDemo`

**期望输出**：打印 `partition`、`offset`，并出现提示 **已写入 __consumer_offsets**（语义提示，非强制字面）。

**与 CLI 对照**：`kafka-consumer-groups.sh --describe --group ...` 查看 **CURRENT-OFFSET** 与 **LAG**（第 8、10 章）。

### 3.3 贴近生产的变体

- 改为 **`commitAsync`** + 回调记录失败；或 **批量提交**（按分区批量）。  
- 加入 **反序列化异常** 与 **死信** 处理（第 14、19 章）。

## 4. 项目总结

### 4.1 优点

- **pull 模型**利于吞吐与批处理；**位移**提供可恢复进度。

### 4.2 缺点

- **语义复杂**：提交与处理顺序组合不当易丢或重复。

### 4.3 适用场景

- 事件处理、流式 ETL、微服务异步集成。

### 4.4 注意事项

- 处理时长超过 **`max.poll.interval.ms`** 会触发 rebalance（第 7 章）。

### 4.5 常见踩坑

- 先 commit 后 **业务失败**；  
- **长耗时处理** 放在 `poll` 循环里不触发心跳；  
- 与 **多线程消费** 混用不当（第 16 章）。

**与后续章节的衔接**：第 7 章消费者组；第 8 章位移；第 14 章交付语义。

## 6. 与线程模型的关系（入门）

**一个分区在同一消费者组内通常只分配给组内一个 consumer 实例**，因此「多线程并发消费同一分区」并不是 Kafka 的标准用法。常见模式是：**每分区单线程处理**、或 **异步处理 + 队列** 但要非常小心 **提交位移** 与 **顺序**（第 16 章）。入门阶段不要急于上多线程，先把 **单线程 poll 循环** 写稳。

---

## 7. 与「至少一次」语义的默认路径

在 **`enable.auto.commit=true`**（默认）时，位移可能 **周期性自动提交**，这往往意味着 **至少一次**：消息可能被处理多次，如果 **提交发生在处理之后** 且 **处理失败后进程崩溃**，也可能出现 **丢失处理** 的感知（取决于提交时机与失败点）。因此生产环境常见做法是：**关闭自动提交** 或 **精确控制提交点**，并把 **幂等** 做在业务层（第 14 章）。

---

## 8. 与第 5 章的对应关系

生产者写出的 **Key/Headers** 在 `ConsumerRecord` 中原样可读：`key()`、`value()`、`headers()`。调试时建议打印 **分区、offset、timestamp**，与生产者日志中的 **trace-id** 对齐，形成端到端追踪（第 5、22 章）。

## 10. 排障速查（消费侧）

1. **无消息**：是否 **订阅成功**、**Topic 是否有数据**、**group 位移是否已在末尾**、**ACL** 是否拒绝读取。  
2. **重复消费**：是否 **rebalance**、是否 **处理失败未提交**、是否 **业务非幂等**。  
3. **越消费越慢**：是否 **单分区热点**、是否 **下游阻塞**、是否 **一次 poll 拉太多**。

## 13. 与「消费语义」的一句话提醒

Kafka 默认不替你保证「**处理成功**」与「**位移提交**」是同一事务；除非你引入 **事务型消费/生产链路** 或 **外部事务日志**（第 14、18、30 章）。因此：**业务幂等** 往往是第一防线。

## 14. 术语对照（巩固）

- **ConsumerRecord**：单条消费记录。  
- **ConsumerRecords**：一次 poll 的批次容器。  
- **Offset**：分区内序号。  
- **Committed offset**：已提交进度。

## 23. 与「最小可运行」标准

本章合格线：**同一台机器**上，你能 **重复运行** 示例并得到 **稳定输出**；并能口述 **commit 前后崩溃** 的差异。达不到这一点，不建议进入 EOS 章节。

## 24. 与「监控」的最低配置

至少能看到：**消费速率**、**处理耗时**、**commit 失败次数**（若有统计）、**Lag**。没有 Lag 就很难判断是 **生产慢** 还是 **消费慢**（第 10、22 章）。

---

## 25. 与「数据质量」的衔接

若消费后写入 **数仓/OLTP**，要明确 **迟到数据**、**重复数据** 的容忍度与 **去重策略**；Kafka 只解决管道层的一部分（第 9、21 章）。

## 27. 本章自测（判断题）

1. `poll` 返回空一定表示「没有消息」（错：可能重平衡等）。  
2. 关闭自动提交后，不 commit 就永远不会前进（在处理成功路径下如此，但要区分 **异常退出**）。  
3. 同一组内两个 consumer 能同时读同一分区（错）。

## 29. 与「延迟敏感业务」的提醒

若你的业务对 **端到端延迟** 极敏感，需要同时观察：**poll 等待**、**批大小**、**下游 RPC**、**rebalance** 暂停窗口。Kafka 往往只是链路中的一段；没有全链路追踪会把问题误判为「Kafka 慢」（第 22 章）。

---

## 30. 与「消费者组命名」规范

建议 **环境前缀 + 服务名 + 版本**（如 `stg.order-svc.v2`），避免与别人的组冲突；**不要随意改 group.id**，否则可能 **从头读** 或 **重复处理**。

## 33. 与「失败处理」模式

常见模式：**重试**（带上限与退避）、**死信队列**（写入侧或消费侧）、**告警人工介入**。不要把「无限重试」当成默认策略，否则可能拖垮下游或放大热点（第 14、19 章）。

## 35. 与「On-Call」常见问法对照

- 问「消费挂了吗？」先回答 **Lag 是否上升** 与 **是否 rebalance**。  
- 问「是不是 Kafka 丢了消息？」先区分 **未生产**、**未消费**、**消费了但业务失败**。  
- 问「要不要 reset？」先确认 **影响面** 与 **是否双活**。
