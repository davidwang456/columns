现在他有机会实现他当初的梦想，但难度比以前的接龙游戏大多了，不仅有更加复杂的规则，更有更多的“纸牌”，这些对他都是挑战，我们来说说他最近遇到的挑战吧。

**“纸牌”的挑战**

如果“纸牌”数量少还好，但如果假设“纸牌”数量有 1k，1w，10w，100w，亦或更多时“纸牌”接龙该如何连接呢？

传统接龙方式：

```java
private static final int loops=100000;	
static String concat(String org) {
 	for(int i=0;i<loops;i++) {
 		org=org+""+i;
 	}
 	return org;
}
```

程序运行耗时 14849 毫秒即 15 秒，此时的性能堪忧，如何解决性能问题呢？

**Java中纸牌接龙新方法**

考虑性能的情况下，可以使用Java中的 StringBuffer 和 StringBuilder 两种方式：

```java
static String append(String org) {		
	StringBuilder sb=new StringBuilder(org);
	for(int i=0;i<loops;i++) {
		sb.append(""+i);
	}
	return sb.toString();
}
	
static String buffer(String org) {
	StringBuffer sb=new StringBuffer(org);
	for(int i=0;i<loops;i++) {
		sb.append(""+i);
	}
	return sb.toString();
}
```

性能的事情，当然要从测试的结果来看，我们来测一下：

```java
	private static final int loops=100000;
	public static void main(String[] args) {
        //字符串连接方式
		long start=System.currentTimeMillis();
		concat("hello world ! ");
		System.out.println(System.currentTimeMillis()-start);
		//StringBuilder连接方式
		start=System.currentTimeMillis();
		append("hello world!");
		System.out.println(System.currentTimeMillis()-start);
		//StringBuffer连接方式
		start=System.currentTimeMillis();
		buffer("hello world!");
		System.out.println(System.currentTimeMillis()-start);
	}
```

测试结果如下：

​		1000 					5 				1 				0

​		1w 					   225 			    2 				3

​		10w 					 15569 			11 				10

​		100w 					很久。。 		 124 				96

通过测试发现，传统的 concate（“+”）方式的性能太差，使用   或者 Strinbuffer 才能满足项目需求。但是可以看出来，他们的性能差别不大，又应该如何选择呢？

StringBuilder 是线程安全的，StringBuffer 不是，在多线程情况下一定要使用 StringBuilder。

**建议不管在单线程还是多线程都直接使用 StringBuilder 来连接字符串**。
