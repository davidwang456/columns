# 第27章 性能调优：反压诊断与资源配置

## 1 项目背景

### 业务场景：Flink CDC作业延迟从1秒涨到5分钟

一个运行稳定的Flink CDC作业，某天突然处理延迟从1秒飙升到5分钟。运维打开Web UI，看到Source算子的`currentFetchDelay`显示为300000ms（5分钟），Sink算子的`backPressuredTimeMsPerSecond`为1000ms/秒——Sink被反压打满了。

增加Sink并行度后延迟没降下来，反而看到TaskManager的GC时间大幅增加。问题进一步复杂化——到底瓶颈在哪个环节？是Source读取太慢？还是Transform计算太重？还是Sink写入太慢？

**系统化的反压诊断和性能调优是解决这类问题的唯一途径。**

### Flink CDC性能模型

```
预期吞吐 = min(Source吞吐, Transform吞吐, Sink吞吐)

Source吞吐 ≈ MySQL Binlog读取速率 × 反序列化效率
  → 受Debezium配置、Binlog文件大小、server-id数量影响

Transform吞吐 ≈ 列数 × 数据量 × 表达式复杂度
  → 受projection列数、filter条件、UDF性能影响

Sink吞吐 ≈ 目标系统写入速率 × 批处理大小
  → 受Kafka分区数 / Doris导入频次 / MySQL写入速度影响
```

## 2 项目设计 · 三人交锋对话

**小胖**（疑惑）：Flink的反压（Backpressure）机制我理解——下游处理不过来时，上游会减速。但为什么反压会导致Checkpoint也变慢？

**大师**：这是反压最危险的连锁反应。Checkpoint需要**Barrier对齐**——Barrier从Source发出，经过所有Operator到达Sink。如果某个Operator反压严重（Buffer中积压了大量数据），Barrier得等这些数据处理完才能通过。

```
正常情况:
  Source → [Barrier] → Transform → [Barrier] → Sink
    耗时: 10ms

反压情况:
  Source → [Barrier 卡住] → Transform(积压10000条) → [Barrier 卡住] → Sink(写入慢)
    耗时: 30秒  → Checkpoint超时 → 作业失败！
```

**技术映射**：反压就像"高速公路堵车"——Checkpoint的Barrier就像救护车，需要从入口（Source）开到出口（Sink）。堵车时（反压），救护车也开不动（Checkpoint慢），最后病人死在路上（Checkpoint超时 → 作业失败）。

**小白**：那反压的根因怎么定位？Web UI上的反压状态有OK/LOW/HIGH三种，我该从哪个算子开始查？

**大师**：反压定位遵循**"从后往前"**原则：

```
Step 1：找到第一个出现 BACKPRESSURED=HIGH 的算子
  Source (OK) → Transform (OK) → Sink (HIGH)
                                   ↑ 从这里开始查

Step 2：检查这个算子的上游
  如果Sink=HIGH → 检查Sink的写入目标是否有问题
  → Kafka分区Leader选举？Doris BE负载高？MySQL慢查询？

Step 3：如果没有算子显示HIGH但延迟依然高
  → 可能Source本身有瓶颈（如Binlog读取慢）
  → 检查Source的currentFetchDelay

Step 4：如果所有算子都OK
  → 可能是外部系统延迟（如Kafka Consumer消费慢）
  → 检查下游消费者的消费速率
```

## 3 项目实战

### 分步实现

#### 步骤1：反压诊断五步法

```bash
# 第一步：查看作业概览
# 在Flink Web UI (http://localhost:8081) 选择你的CDC作业

# 第二步：找到反压最高的算子
# 点击 "Backpressure" 标签
# 观察每个算子的 "Backpressure Status"
# Source → Map → Sink
# 如果Sink = HIGH，从Sink开始查

# 第三步：检查Sink算子的详细信息
# 点击Sink算子 → "Metrics" 标签
# 查看:
#   numRecordsOutPerSecond: 500   ← 当前写入速率
#   numRecordsInPerSecond: 5000   ← 当前接收速率
#   backPressuredTimeMsPerSecond: 900  ← 90%的时间在反压
#   busyTimeMsPerSecond: 100      ← 只有10%的时间在忙

# 第四步：检查Source算子的延迟
# 点击Source算子 → "Metrics" 标签
# 查看:
#   currentFetchDelay: 5000      ← Binlog延迟5秒（正常）
#   numRecordsInPerSecond: 5000  ← Source读取速率
#                              ↑ Source读得快，Sink写得慢 → 反压

# 第五步：确认根因——Sink写入慢
# 检查目标系统（Kafka）的写入性能
echo "检查Kafka性能..."
docker exec kafka-cdc kafka-producer-perf-test \
  --topic test_perf \
  --num-records 100000 \
  --record-size 1000 \
  --throughput -1 \
  --producer-props bootstrap.servers=localhost:9092
```

#### 步骤2：Sink写入速度优化

```yaml
# 方案A：增大批次大小（减少网络请求次数）
sink:
  type: kafka
  sink:
    batch-size: 16384               # 每批字节数（默认16384）
    linger-ms: 100                   # 等待100ms或满16384字节发送
    buffer-memory: 33554432         # 缓冲区大小（32MB）

# 方案B：增加并行度（需要目标系统支持多分区写入）
pipeline:
  parallelism: 4                     # 从2提升到4

# 方案C：使用Buffer缓解反压
# 在Source和Sink之间加Kafka作为Buffer
```

#### 步骤3：内存配置优化

```yaml
# flink-conf.yaml — Flink CDC作业内存配置

# TaskManager总内存
taskmanager.memory.process.size: 4g

# 托管内存（用于RocksDB State Backend）
taskmanager.memory.managed.size: 1g

# 网络Buffer（影响反压传播速度）
taskmanager.memory.network.max: 1g
taskmanager.memory.network.min: 512m

# 堆外内存（用于Debezium引擎）
taskmanager.memory.off-heap.size: 512m

# JVM参数
env.java.opts.taskmanager: >
  -XX:+UseG1GC
  -XX:MaxGCPauseMillis=100
  -XX:+PrintGCDetails
  -Xloggc:/tmp/flink-gc.log
```

#### 步骤4：并行度计算——公式与实践

```java
/**
 * Flink CDC并行度计算公式
 * 
 * 参数:
 *   totalRecordsPerSec   - MySQL峰值TPS（如50000条/秒）
 *   avgRecordSize        - 平均记录大小（如500字节）
 *   sinkThroughputPerTask - 单个Sink并行度的峰值吞吐（如10000条/秒）
 *   networkBufferLimit   - 单TaskManager的网络Buffer限制（如100MB）
 * 
 * 计算结果:
 *   sourceParallelism = MIN(chunkCount, totalRecordsPerSec / chunkSize)
 *                      = 通常4~8
 *   transformParallelism = sourceParallelism（建议相同）
 *   sinkParallelism    = CEIL(totalRecordsPerSec / sinkThroughputPerTask)
 *                      = CEIL(50000 / 10000) = 5
 */
public class ParallelismCalculator {
    
    public static int calcSinkParallelism(
            long totalRps, long sinkRpsPerTask) {
        return (int) Math.ceil((double) totalRps / sinkRpsPerTask);
    }
    
    public static void main(String[] args) {
        // 示例：MySQL峰值5万条/秒写入Kafka
        long totalRps = 50000;
        long sinkRpsPerTask = 12000;  // 实测Kafka Sink单并行度吞吐
        int sinkParallelism = calcSinkParallelism(totalRps, sinkRpsPerTask);
        System.out.println("推荐Sink并行度: " + sinkParallelism);
        // 输出: 推荐Sink并行度: 5
    }
}
```

#### 步骤5：Debezium引擎调优

Debezium是Flink CDC的底层引擎，它的配置直接影响Source读取性能：

```yaml
source:
  type: mysql

  # Debezium引擎调优参数（以debezium.开头）
  debezium:
    # 批量读取大小
    max.batch.size: 20480              # 每次最多读取20480条（默认2048）
    max.queue.size: 81920             # 事件队列大小（默认8192）
    
    # 增量快照调优
    snapshot.fetch.size: 4096          # 全量快照每批读取行数
    
    # 连接调优
    poll.interval.ms: 300              # 轮询间隔（毫秒）
    connect.timeout.ms: 30000          # 连接超时
    heartbeat.interval.ms: 30000       # 心跳间隔（Binlog保活）
    
    # 事件过滤（减少不必要的事件）
    table.include.list: shop.orders   # 只监听需要的表
    column.exclude.list: shop.orders.internal_remark  # 排除不需要的列
    
    # 异常处理
    event.deserialization.failure.handling.mode: warn  # 反序列化失败时告警而非报错
    inconsistent.schema.handling.mode: warn            # Schema不一致时告警
```

#### 6个关键性能指标与告警阈值

| 指标 | Flink Web UI路径 | 正常值 | 告警阈值 |
|------|-----------------|-------|---------|
| `currentFetchDelay` | Source算子Metrics | < 1000ms | > 10000ms |
| `backPressuredTimeMsPerSecond` | 各算子Metrics | < 100ms | > 500ms |
| `numRecordsInPerSecond` | Source→Sink对比 | 上下游接近 | 差异 > 30% |
| Checkpoint Duration | Checkpoints页 | < 5000ms | > 30000ms |
| Checkpoint Size | Checkpoints页 | < 100MB | > 500MB |
| TaskManager GC | TaskManager → Metrics | < 1000ms/分 | > 10000ms/分 |

#### 常见陷坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| 增加并行度后性能更差 | 并行度 > MySQL连接池上限或Kafka分区数 | Source并行度 ≤ MySQL连接池大小；Sink并行度 ≤ Kafka分区数 |
| 全量快照慢但增量正常 | Chunk Size设置不当或无主键 | 调小Chunk Size，确保表有主键 |
| 反压症状在算子之间跳变 | 频繁的Auto-Rescaling | 设置固定的并行度，关闭Operator Chaining观察 |
| Debezium max.queue.size导致OOM | 生产速度 > 消费速度，队列撑爆内存 | 调小`max.queue.size`的同时增大`max.batch.size` |

## 4 项目总结

### Flink CDC性能调优清单

| 类别 | 优化项 | 效果 | 难度 |
|------|-------|------|------|
| **Source** | 增加server-id范围（匹配并行度） | ✅ 提升快照速度 | 低 |
| | 调小Chunk Size（2000~5000） | ✅ 降低单Chunk内存 | 中 |
| | 配置`max.batch.size` | ✅ 提升增量吞吐 | 低 |
| **Transform** | 减少projection列数 | ✅ 降低序列化开销 | 低 |
| | Filter靠近Source | ✅ 尽早减少数据量 | 低 |
| | UDF避免创建大对象 | ✅ 降低GC压力 | 中 |
| **Sink** | 增大batch-size | ✅ 减少网络请求 | 低 |
| | Sink并行度 = 目标分区数 | ✅ 匹配写入能力 | 低 |
| | 使用幂等写入而非事务 | ✅ 降低写入开销 | 中 |
| **Flink** | 合理配置TaskManager内存 | ✅ 降低GC频率 | 中 |
| | 启用Unaligned Checkpoint | ✅ 反压时Checkpoint不超时 | 中 |
| | 选择RocksDB状态后端 | ✅ 大状态场景性能稳定 | 低 |

### 常见踩坑经验

**故障案例1：增加并行度反而导致延迟增加**
- **现象**：Sink并行度从2增到4后，延迟从5秒涨到15秒
- **根因**：Kafka目标Topic只有3个分区。Flink Sink并行度=4，其中有3个Sink Subtask正常写入，1个Subtask永远空闲（没有分区可写），反而增加了调度开销
- **解决方案**：Sink并行度 ≤ 目标Topic分区数。在增加并行度前，先增加Kafka分区数

**故障案例2：RocksDB状态后端导致反压**
- **现象**：每次Checkpoint触发时，Source的延迟都飙升一次
- **根因**：RocksDB的Compaction在Checkpoint时被触发，Compaction消耗大量磁盘IO，导致状态读写变慢，进而引发Operator反压
- **解决方案**：设置`state.backend.rocksdb.thread.num=2`限制Compaction线程数，或错开Compaction和Checkpoint的时间

**故障案例3：网络带宽成为瓶颈**
- **现象**：Flink CDC作业的延迟持续增长，但所有Operator的反压都是OK
- **根因**：每个CDC事件平均2KB，每秒5万条事件 = 100MB/s网络吞吐。而TaskManager的网络Bandwidth是1Gbps（125MB/s），接近饱和
- **解决方案**：在Source端使用projection减少列数（从20列降到5列，数据量降为1/4），或在Source和Sink之间压缩数据

### 思考题

1. **进阶题①**：在Flink CDC作业中，如果要处理的表有200列（宽表），但下游只需要其中10列。在不修改MySQL表结构的情况下，你能在哪些环节减少数据传输量？从Binlog事件、Debezium解析、Flink网络传输、Sink写入四个维度分别说明。

2. **进阶题②**：假设你的Flink CDC作业的Checkpoint耗时突然从2秒涨到了60秒，经排查是RocksDB Compaction导致。此时你同时收到"反压"和"Checkpoint超时"告警。你应该先解决反压还是先解决Checkpoint？为什么？提示：考虑Checkpoint和反压的因果关系。
