# 第38章 极端场景优化：万级表与大事务

## 1 项目背景

### 业务场景：50000张表的实时同步

某大型SaaS公司，一个Flink CDC Pipeline需要同时同步来自50000张业务表的CDC数据。遇到了几个极端问题：

1. **Source端**：50000张表的Metadata（Schema信息）超过2GB，TaskManager堆内存溢出
2. **Checkpoint端**：每个Checkpoint都需要遍历50000张表的Schema状态，Checkpoint耗时从5秒涨到15分钟
3. **网络端**：50000张表的Binlog事件每秒产生超过10万条，网络传输成为瓶颈

另一个场景：某金融公司一笔"批量利息计算"事务，修改了1.2亿行数据。Flink CDC在接收这个事件时，Debezium引擎缓存了整个事务的变更记录，直接导致了OOM。

### 极端场景优化维度

```
万级表挑战:
  ├── Schema管理 → SchemaRegistry内存优化
  ├── Chunk切分 → 动态表发现 + 批量切分
  ├── Checkpoint → 增量状态存储
  └── 路由规则 → 正则表达式预编译

大事务挑战:
  ├── Debezium内存 → 流式事务处理
  ├── Binlog解析 → 大事务拆分
  ├── 事件积压 → 反压传导
  └── 恢复耗时 → 事务位点优化
```

## 2 项目设计 · 三人交锋对话

**小胖**（目瞪口呆）：5万张表！我第15章做个双写5张表就觉得很复杂了。5万张表怎么管得过来？

**大师**：5万张表的管理核心不是"人工配置"，而是**模式匹配 + 动态发现**。

**万级表配置方案：**
```yaml
source:
  type: mysql
  # 不列出50000张表名，而是用正则匹配
  database-list: tenant_.*        # 匹配所有租户库
  table-list: .*\.orders, .*\.users  # 匹配所有库的orders和users表
  capture-new-tables: true        # 动态发现新增表
```

**原理：**
1. 用正则模式匹配所有表名，而不是逐个列举
2. `capture-new-tables=true`：Flink CDC定期扫描MySQL的`INFORMATION_SCHEMA.TABLES`，发现新模式匹配的新表后自动添加
3. 新表通过`SchemaChangeEvent`（CreateTableEvent）通知下游

**小白**：那5万张表的Schema信息都存内存里，不爆吗？Flink State能存得下吗？

**大师**：问到了关键！SchemaRegistry默认将所有表的Schema存在**堆内存**中。5万张表的Schema（每张表平均20列，每列约100字节） ≈ 5万 × 20 × 100 = **100MB**。看起来不大，但加上各种索引、缓存、序列化对象，实际可能膨胀到500MB~1GB。

**优化方案：**

**方案1：按需加载Schema**
只有被Binlog事件命中的表才加载Schema，而不是启动时全部加载。

**方案2：Schema缓存淘汰**
使用LRU（最近最少使用）缓存，只保留最近N张表的Schema。如果一张表超过30分钟没有变更事件，Schema从缓存中淘汰。

**方案3：Schema序列化优化**
使用二进制序列化（而非JSON）存储Schema，减少内存和Checkpoint大小。

```yaml
pipeline:
  schema.cache:
    max-size: 10000          # 最多缓存10000张表的Schema
    evict-strategy: lru       # 淘汰策略：最近最少使用
    evict-idle-time: 30m     # 闲置30分钟后淘汰
```

**技术映射**：万级表的Schema管理像"图书馆的图书索引"——5万本书的索引卡片不能全摊在桌上（堆内存），而是用电子目录（LRU缓存），最近被借阅的书（活跃表）的索引卡片才放在前台。

## 3 项目实战

### 分步实现

#### 步骤1：万级表配置——正则匹配 + 动态发现

```yaml
source:
  type: mysql
  hostname: ${MYSQL_HOST}
  username: ${MYSQL_USER}
  password: ${MYSQL_PASSWORD}
  
  # 万级表配置：使用正则而非枚举
  database-list: "tenant_\\d{4}"        # 匹配tenant_0000~tenant_9999
  table-list: ".*\\.(orders|users|products|payments)"
  
  # 动态表发现
  capture-new-tables: true              # 自动发现新增表
  scan.newly-added-table.enabled: true  # 扫描新增表
  
  # 内存优化
  debezium:
    schema.cache:
      enabled: true
      max.size: 10000
  
  server-id: 5400-5408                  # 覆盖并行度

pipeline:
  parallelism: 8
  # 非活跃表Schema不参与Checkpoint
  state.backend.incremental: true
```

#### 步骤2：SchemaRegistry内存优化配置

```java
// 源码中的优化点: SchemaRegistry的缓存策略
import org.apache.flink.cdc.common.schema.Schema;
import java.util.concurrent.ConcurrentHashMap;
import java.util.LinkedHashMap;
import java.util.Map;

/**
 * LRU模式的Schema缓存——替代默认的ConcurrentHashMap
 */
public class LruSchemaCache {

    private final int maxSize;
    private final Map<TableId, Schema> cache;

    public LruSchemaCache(int maxSize) {
        this.maxSize = maxSize;
        this.cache = new LinkedHashMap<TableId, Schema>() {
            @Override
            protected boolean removeEldestEntry(
                    Map.Entry<TableId, Schema> eldest) {
                return size() > maxSize;
            }
        };
    }

    public Schema getSchema(TableId tableId) {
        return cache.get(tableId);
    }

    public void putSchema(TableId tableId, Schema schema) {
        cache.put(tableId, schema);
    }
}
```

#### 步骤3：大事务处理的Debezium配置

```yaml
source:
  type: mysql
  debezium:
    # ========== 大事务优化配置 ==========
    
    # 大事务拆分：超过10000条的事务，分批次读取
    # 避免一次性缓存整个事务到内存导致OOM
    max.batch.size: 10240
    max.queue.size: 81920
    
    # 大事务的Binlog事件分割
    # SCN (System Change Number) 间隔
    # 让Debezium在事务中间也能输出事件
    # 而不是等整个事务提交后才输出
    event.deserialization.failure.handling.mode: warn
    inconsistent.schema.handling.mode: warn
    
    # Streaming配置
    stream:
      # 每批最大行数（流式读取避免OOM）
      max.rows.per.batch: 50000
      # 每批最大时间（毫秒）
      max.time.per.batch.ms: 1000
    
    # 大事务超时保护
    snapshot.locking.mode: none
    database.history:
      skip.unparseable.ddl: true
```

#### 步骤4：万级表环境下的Checkpoint优化

```yaml
# flink-conf.yaml — 万级表环境Checkpoint配置

# 使用RocksDB状态后端（支持大状态）
state.backend: rocksdb
state.backend.incremental: true

# RocksDB调优
state.backend.rocksdb.memory.managed: true
state.backend.rocksdb.writebuffer.count: 4
state.backend.rocksdb.writebuffer.size: 128m

# Checkpoint配置
state.checkpoints.num-retained: 3     # 只保留最近3个
state.checkpoints.cleanup-strategy: DELETE_ON_CANCELLATION

# 异步Checkpoint（减少对主处理的影响）
execution.checkpointing.unaligned: true
execution.checkpointing.alignment-timeout: 30s
```

#### 步骤5：万级表路由优化

```yaml
route:
  # 万级表路由：使用正则预编译
  - source-table: "tenant_\\d{4}\\.orders"
    sink-table: "ods.orders"
    description: "所有租户的orders表汇总到ods.orders"
    
  - source-table: "tenant_\\d{4}\\.users"
    sink-table: "ods.users"
    
  - source-table: "tenant_\\d{4}\\.products"
    sink-table: "ods.products"

# 路由模式：ALL_MATCH（可能匹配多个）vs FIRST_MATCH（仅第一个）
pipeline:
  route-mode: first_match
```

#### 常见陷坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| 万级表下Checkpoint超时 | 每个Checkpoint遍历所有表Schema | 启用增量Checkpoint + LRU Schema缓存 |
| 动态发现表后数据丢失 | 新表的第一批变更在Schema加载前就到了 | 增加`scan.newly-added-table.interval`扫描频率 |
| 大事务导致OOM | Debezium缓存整个事务的所有变更 | 设置`max.batch.size`限制每批最大行数 |
| 路由规则正则匹配太慢 | 5万条规则逐一匹配 | 减少规则数量、使用更具体的前缀匹配 |

## 4 项目总结

### 万级表优化总结

| 优化维度 | 问题 | 方案 | 效果 |
|---------|------|------|------|
| Schema管理 | 5万张表Schema占1GB内存 | LRU缓存 + 按需加载 | 内存降至100MB |
| Checkpoint | 遍历5万张表状态耗时15分钟 | 增量Checkpoint + 非对齐 | 降至30秒 |
| 正则路由 | 5万条规则匹配耗时100ms/条 | 前缀匹配 + 预编译 | 降至1ms/条 |
| 动态表发现 | 新表数据丢失 | 定期扫描INFORMATION_SCHEMA | 自动发现新增表 |
| 大事务 | 1亿行事务OOM | 流式批处理 | 稳定运行 |

### 大事务优化总结

| 优化项 | 配置 | 说明 |
|-------|------|------|
| 分批处理 | `max.batch.size: 10240` | 每批最多处理10240条 |
| 流式输出 | `max.rows.per.batch: 50000` | 到达上限不管事务是否完成都输出 |
| 反压机制 | Handover背压传导 | Debezium慢下来让MySQL Slave延迟增加 |
| RocksDB状态 | 增量Checkpoint | 避免全量Checkpoint的大事务状态快照 |
