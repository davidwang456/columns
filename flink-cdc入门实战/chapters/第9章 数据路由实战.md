# 第9章 数据路由实战

## 1 项目背景

### 业务场景：多环境数据同步与表名规范化

某中型电商公司维护了三套环境：开发环境、测试环境、生产环境。Flink CDC需要将生产环境的订单数据实时同步到测试环境，供QA团队进行回归测试验证。

但有一个棘手的问题——生产环境的表名是`orders`、`order_items`、`users`，而测试环境由于历史原因表名加了前缀：`test_orders`、`test_order_items`、`test_users`。如果直接使用CDC的默认行为（表名不变），数据会写入错误的表。

更复杂的是，公司还有数据合规要求：**订单金额超过10000元的订单需要路由到"高价值订单"审计表**，其余订单走普通订单表。

数据路由（Routing）就是解决这类"源表→目标表灵活映射"问题的核心功能。

### 路由场景分类

```
场景1：同名映射（默认）
  shop.orders → shop.orders   （源和目标表名相同）

场景2：前缀/后缀映射
  shop.orders → shop_ods.orders   （增加库名后缀）
  orders → test_orders           （增加表名前缀）

场景3：正则映射
  shop.order_* → ods.order_*     （匹配多张表）

场景4：条件路由
  orders (amount>10000) → vip_orders   （高价值订单分流）
  orders (amount<=10000) → normal_orders

场景5：表合并
  orders_202401, orders_202402 → orders_all   （分区表合并）
```

## 2 项目设计 · 三人交锋对话

**小胖**（一脸困惑）：路由不就是改个表名吗？感觉像URL重写一样。为啥要专门一章来讲？

**大师**：表面上看是改表名，但路由背后涉及几个核心问题：
1. **正则匹配的效率**——几百张表同时匹配，正则引擎扛不扛得住？
2. **匹配优先级**——如果多条规则都匹配，用哪条？
3. **Schema一致性**——路由到不同目标表后，Schema变了怎么办？

**小胖**：那ALL_MATCH和FIRST_MATCH是啥？还分两种模式？

**大师**：这是Flink CDC路由的核心配置——`route-mode`。

**ALL_MATCH**：只要规则匹配，就**复制一份数据**到每个匹配的目标表。适用于"一份数据分发到多个目标"的场景。

**FIRST_MATCH**：按规则顺序匹配，**只有第一个匹配的规则生效**。适用于"条件路由"场景。

**技术映射**：ALL_MATCH像广播——一条消息发到所有群。FIRST_MATCH像客服分配——谁先接单谁处理，其他人不处理。

**小白**：那路由的表名映射支持变量吗？比如`${database}_${table}`这种格式？

**大师**：Flink CDC Pipeline API的路由配置支持以下变量：

| 变量 | 含义 | 示例（源: shop.orders） | 结果 |
|------|------|-----------------------|------|
| `${db}` | 源数据库名 | `ods_${db}` | `ods_shop` |
| `${table}` | 源表名 | `${table}_copy` | `orders_copy` |
| `*` | 通配符 | `shop.*` → `ods.*` | 匹配所有shop库的表 |

但需要注意：Flink CDC的路由是基于正则表达式的，不是基于变量的。上面提到的变量只是通过`RegexRouter`的替换规则实现的。核心是`source-table`（正则匹配）+ `sink-table`（替换表达式）的组合。

**小白**：还有一个问题让我担心——如果路由导致多个源表路由到同一个目标表（比如分表合并），Schema不一致怎么办？一张表有10列，另一张有11列？

**大师**：这就是路由和Schema Evolution的配合之处了。Flink CDC 3.0提供了两个关键机制：

1. **Schema统一**：多源表路由到同一目标表时，Schema Evolution机制会自动合并所有列的并集
2. **缺失列填充**：如果源表缺少目标表的某个列，用NULL或默认值填充

但这需要设置`schema.change.behavior=EVOLVE`。如果设置为`IGNORE`，Schema不一致时直接报错。

#### 步骤1：准备多表数据

```sql
USE shop;

-- 创建订单主表
CREATE TABLE orders (

-- 创建用户表

    id INT AUTO_INCREMENT PRIMARY KEY,

);

-- 创建高价值订单表（模拟VIP订单路由）
```

#### 步骤2：使用DataStream API实现路由

```java
package com.example;

import org.apache.flink.api.common.functions.MapFunction;
import org.apache.flink.cdc.connectors.mysql.source.MySqlSource;
import org.apache.flink.cdc.debezium.JsonDebeziumDeserializationSchema;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;

/**
 * 数据路由演示——基于DataStream API自定义路由
 */
public class DataRoutingDemo {

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        env.enableCheckpointing(5000);

        MySqlSource<String> source = MySqlSource.<String>builder()
            .hostname("localhost").port(3306)
            .databaseList("shop").tableList("shop.orders", "shop.order_items", "shop.users")
            .username("cdc_user").password("cdc_pass")
            .deserializer(new JsonDebeziumDeserializationSchema())
            .serverId("5400-5401")
            .startupOptions(
                org.apache.flink.cdc.connectors.mysql.table.StartupOptions.latest())
            .build();

        DataStream<String> cdcStream = env.fromSource(source,
            org.apache.flink.api.common.eventtime.WatermarkStrategy.noWatermarks(),
            "MySQL CDC");

        // 自定义路由实现：按表名和金额分流
        DataStream<String> routed = cdcStream.map(
            new MapFunction<String, String>() {
                @Override
                public String map(String json) throws Exception {
                    // 解析JSON提取表名和金额
                    // 简化处理：用字符串匹配演示
                    String targetTable;
                    if (json.contains("\"table\":\"orders\"")) {
                        if (json.contains("\"amount\":")
                            && json.matches(".*\"amount\"\\s*:\\s*[0-9]{5,}.*")) {
                            // 金额>=10000，路由到vip_orders
                            targetTable = "vip_orders";
                        } else {
                            targetTable = "orders";
                        }
                    } else if (json.contains("\"table\":\"order_items\"")) {
                        targetTable = "order_items";
                    } else if (json.contains("\"table\":\"users\"")) {
                        targetTable = "users";
                    } else {
                        targetTable = "unknown";
                    }

                    // 在JSON中添加目标表信息
                    return json.replace(
                        "\"op\"",
                        "\"target_table\":\"" + targetTable + "\",\"op\"");
                }
            });

        routed.print();

        env.execute("Data Routing Demo");
    }
}
```

#### 步骤3：使用Pipeline YAML实现声明式路由

创建 `pipeline-routing.yaml`：

```yaml
source:
  type: mysql
  hostname: localhost
  port: 3306
  username: cdc_user
  password: cdc_pass
  tables: shop.orders, shop.order_items, shop.users

sink:
  type: kafka
  properties:
    bootstrap.servers: localhost:9092
    auto.create.topics.enabled: true

# 核心：路由规则
route:
  # 规则1：前缀映射——给所有表加ods_前缀
  - source-table: shop.orders
    sink-table: shop.ods_orders
    description: "订单表加ods前缀"

  - source-table: shop.order_items
    sink-table: shop.ods_order_items
    description: "订单明细表加ods前缀"

  - source-table: shop.users
    sink-table: shop.ods_users
    description: "用户表加ods前缀"

  # 规则2：正则表达式匹配——所有shop库的表
  # 匹配shop.orders, shop.users等
  - source-table: shop\.(.*)
    sink-table: ods_shop.$1
    description: "正则匹配shop库所有表"

  # 规则3：ALL_MATCH模式——数据同时发送到多个目标
  # 适用于数据分发场景
  # - source-table: shop.vip_.*
  #   sink-table: audit.vip_orders
  #   route-mode: all_match

  # 规则4：FIRST_MATCH模式（默认）——只匹配第一条规则
pipeline:
  name: Pipeline Routing Demo
  parallelism: 1
  route-mode: first_match    # first_match (默认) | all_match
```

#### 步骤4：路由规则详解——正则替换

Flink CDC的路由使用**正则捕获组 + 替换表达式**：

```yaml
# 正则规则格式
- source-table: <正则表达式>
  sink-table: <替换表达式（支持 $1, $2 等捕获组引用）>
```

**常见路由模式示例：**

| 源表 | source-table正则 | sink-table替换 | 匹配结果 |
|------|----------------|---------------|---------|
| `shop.orders` | `shop\.(.*)` | `ods.$1` | `ods.orders` |
| `shop.orders` | `(shop)\.(.*)` | `$1_ods.$2` | `shop_ods.orders` |
| `prod.orders_202401` | `prod\.(.*)_\d{6}` | `ods.$1` | `ods.orders` |
| `stg.users` | `stg\.(.*)` | `dev.$1` | `dev.users` |
| `shop.orders_shard_01` | `shop\.(.*)_shard_\d{2}` | `ods.$1` | `ods.orders` |

#### 常见陷坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| 路由规则不生效 | source-table正则写错了 | 使用`shop\\.`（注意转义）代替`shop.` |
| 数据没有写入预期表 | 多条规则命中但模式是FIRST_MATCH | 调整规则顺序，或改用ALL_MATCH |
| 路由后Schema丢失某些列 | 源表和目标表的列不完全一致 | 开启`schema.change.behavior=EVOLVE` |
| 正则匹配性能差 | 几千条正则规则逐个尝试 | 减少规则数，使用更精确的匹配前缀 |

## 4 项目总结

### 路由策略对比

| 策略 | 匹配方式 | 路由结果 | 适用场景 |
|------|---------|---------|---------|
| 等同映射 | `shop.orders`→`shop.orders` | 一对一 | 默认行为 |
| 前缀/后缀 | `shop.*`→`ods.*` | 一对多 | 多环境隔离 |
| 正则分组 | `shop\.(.+)`→`ods.$1` | 多对多 | 复杂映射 |
| 条件路由 | 基于数据内容 | 一对多 | 数据分流 |
| ALL_MATCH | 广播所有匹配目标 | 一对多 | 数据分发 |
| FIRST_MATCH | 优先匹配 | 一对一 | 条件路由 |

### 注意事项

1. **正则转义**：`.`在正则中匹配任意字符，如果只想匹配点号，用`\\.`（YAML中需要双斜杠）。
2. **路由顺序敏感**：FIRST_MATCH模式下，最具体的规则放前面，通用的放后面。
3. **ALL_MATCH与幂等性**：ALL_MATCH模式下，数据被复制多份。下游消费处理必须做到幂等，否则产生重复数据。
4. **目标表预先创建**：路由到的新目标表需要预先存在，或开启`auto.create.table`。

### 常见踩坑经验

**故障案例1：正则规则写错导致数据丢失**
- **现象**：增加路由规则后，某些表的数据不再同步
- **根因**：所有规则都不匹配该表，Flink CDC丢弃了不匹配的数据
- **解决方案**：添加一条兜底规则如`- source-table: .*`（匹配所有），或开启`pipeline.route.unmatched: IGNORE`配置

**故障案例2：ALL_MATCH模式下数据重复**
- **现象**：下游收到重复的CDC事件（同一条数据收到2次）
- **根因**：配置了ALL_MATCH且两条规则都匹配了，数据被复制了2份
- **解决方案**：确认ALL_MATCH是否是预期行为。如果不是，改为FIRST_MATCH

**故障案例3：路由替换后表名超长**
- **现象**：路由后目标表名超过数据库命名长度限制
- **根因**：替换规则添加了过多前缀后缀
- **解决方案**：控制替换后表名长度，或在路由前对表名做截断处理

1. **进阶题①**：在ALL_MATCH模式下，Flink CDC如何实现"一份数据同时写入多个目标表"？是复制数据流还是复制写入操作？对性能有什么影响？
