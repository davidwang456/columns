# 第18章 增量快照原理与调优（FLIP-27）

## 1 项目背景

### 业务场景：无锁全量同步亿级大表

某金融公司需要将MySQL核心交易表（1.2亿行，约80GB）全量同步到数据湖。团队调研了三种方案：

1. **mysqldump + 导入**：需要锁表，线上无法接受（锁表期间业务停摆）
2. **Canal全量同步**：Canal的`zk-offset`全量模式会触发MySQL短时锁
3. **Flink CDC增量快照**：无锁、并行读取

最终选择Flink CDC，但1.2亿行数据的全量同步仍然是个挑战——如果Chunk大小设置不当，可能出现OOM、长事务、或Chunk空转等问题。

**理解增量快照（Incremental Snapshot）的原理**，是在生产环境中用好Flink CDC的必修课。

### 增量快照 vs 传统快照

```
传统快照（有锁）:
  FLUSH TABLES WITH READ LOCK;   ← 全局锁表
  读全部数据;
  UNLOCK TABLES;
  读取增量Binlog（从锁表时的位点开始）
  └── 问题：锁表期间业务停止

Flink CDC增量快照（无锁）:
  将表按主键分成N个Chunk;
  并行读取每个Chunk（无锁SELECT）;
  同时记录每个Chunk读取时的Binlog位点;
  读取完成后做水位对齐（合并全量和增量）;
  └── 优点：全程无锁，数据库零侵入
```

## 2 项目设计 · 三人交锋对话

**小胖**（疑惑）：增量快照的核心思想我懂——分块读取。但MySQL的SELECT不是MVCC的吗？每个SELECT看到的是不同时间点的快照，那分块读出来的数据在时间上能一致吗？

**大师**：问到了核心！MySQL的InnoDB MVCC确实让每个SELECT看到的是自己启动时的快照。如果在快照过程中有数据变更，每个Chunk看到的可能是不同版本的数据。

Flink CDC通过**水位对齐算法（Watermark Alignment）**解决这个问题：

```
时间线：─────────────────────────────────────────────►
                     Chunk 1 (id: 1-1000)  ──► 完成@T1
                     Chunk 2 (id: 1001-2000) ──► 完成@T2
                     Chunk 3 (id: 2001+ ) ──► 完成@T3

水位线 = min(T1, T2, T3)  ← 取所有Chunk完成时间的最小值

增量流从 水位线 开始读取Binlog，补上：
  - T1~水位线之间的变更（Chunk1读取期间产生的变更）
  - T2~水位线之间的变更（Chunk2读取期间产生的变更）
  - T3~水位线之间的变更（Chunk3读取期间产生的变更）

最终结果 = 所有Chunk数据 + 水位线之后的增量数据
          = 在时间上完全一致的完整数据集
```

**小白**：那Chunk是怎么切分的？如果表有主键，按主键范围切分我可以理解。但如果表没有主键怎么办？

**大师**：这是增量快照最棘手的case。Flink CDC的`MySqlChunkSplitter`切分策略分三种情况：

**情况1：有主键（默认使用）**
```sql
-- 查询主键的MIN和MAX
SELECT MIN(id), MAX(id) FROM orders;  -- (1, 5000000)
-- 按步长切成N个Chunk
-- Chunk 1: id BETWEEN 1 AND 100000
-- Chunk 2: id BETWEEN 100001 AND 200000
-- ...
```

**情况2：有唯一索引（无主键）**
```sql
-- 使用唯一索引列作为切分键
-- 如果唯一索引是UUID（字符串），按字母序切分
```

**情况3：无主键、无唯一索引（最差case）**
- Flink CDC退化为**单线程读取整张表**
- 性能骤降，且无法并行
- 强烈建议：**生产环境的大表必须有主键！**

**技术映射**：有主键的表像"图书馆按编号排好的书架"——你可以说"我要第1~1000号的书"，多个人同时搬。没有主键的表像"全堆在地上的书"——只能一个人默默地整理。

**小胖**：那Chunk大小怎么配置？太大和太小各有什么风险？

**大师**：通过`scan.incremental.snapshot.chunk.size`配置（默认8096行/Chunk）：

| Chunk Size | 优点 | 缺点 |
|-----------|------|------|
| 小（1000行） | 内存占用低，单个Chunk处理快 | Chunk数量多，快照阶段总时间可能变长 |
| 中（8096行） | 平衡内存和速度（默认值） | 需要根据主键数据类型调整 |
| 大（10万行） | Chunk数量少，快照总时间短 | 内存占用高，单个Chunk失败恢复代价大 |

**最佳实践**：对于普通表（百万级），保持默认8096即可。对于大表（亿级），建议调小到2000-5000。

#### 步骤5：配置调优——大表增量快照的最佳实践

```yaml
source:
  type: mysql
  tables: shop.user_action_log

  # 增量快照配置
  scan.incremental.snapshot:
    enabled: true
    chunk.size: 4000              # 每个Chunk行数（提高并行度，降低单Chunk内存）
    chunk.key.columns:            # 自定义切分列（非主键列作为切分键）
      user_action_log: id
    parallelism: 8                # 快照读取并行度

  # 连接配置
  connect.timeout: 30s
  connect.max-retries: 3
  server-id: 5400-5407            # 必须覆盖并行度
  heartbeat.interval: 30s         # Binlog心跳间隔

pipeline:
  parallelism: 1                  # Sink并行度保持1（保证写入顺序）
```

#### 常见陷坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| 增量快照全量阶段非常慢 | Chunk Size太小，导致Chunk数量过多 | 适当增大Chunk Size（5000~10000） |
| Chunk读取时报错"BIGINT UNSIGNED value is out of range" | 无符号BIGINT超出Java的Long.MAX_VALUE | 使用`DecimalType`替代`LongType`处理主键 |
| 增量快照切换后数据大量重复 | 水位对齐算法有Bug或Chunk切分重叠 | 升级Flink CDC到最新版本，确保使用FLIP-27稳定版 |
| 无限循环"Assigning splits..." | Enumerator和Reader之间的RPC通信卡死 | 增加RPC超时时间，检查网络连通性 |
| 全量快照阶段MySQL CPU飙升 | 并行度太高导致MySQL并发查询过载 | 降低并行度（从8降到4），同时调小Chunk Size |

## 4 项目总结

### 增量快照关键配置参数

| 参数 | 默认值 | 说明 | 推荐值 |
|------|-------|------|-------|
| `scan.incremental.snapshot.enabled` | true | 是否启用增量快照算法 | true |
| `scan.incremental.snapshot.chunk.size` | 8096 | 每个Chunk的行数 | 2000~5000（大表） |
| `chunk-key.even-distribution.factor.lower-bound` | 0.05 | Chunk均匀分布下限 | 默认 |
| `chunk-key.even-distribution.factor.upper-bound` | 1000 | Chunk均匀分布上限 | 默认 |
| `scan.snapshot.fetch.size` | 1024 | 每批读取行数 | 1024~4096 |
| `connect.max-retries` | 3 | 连接失败最大重试次数 | 3~5 |

### 适用场景

**增量快照适用的场景：**
1. 大表（>100万行）的全量初始化
2. 需要无锁读取的线上生产库
3. 需要并行读取加速全量同步
4. 有自增数字主键的表（最佳效果）

**不适用的场景：**
1. 无主键表（必需有唯一标识列）
2. 主键为UUID等无序字符串（Chunk范围切分效果差）
3. 源库MySQL版本低于5.6（不支持GTID）

### 常见踩坑经验

**故障案例1：Chunk切分遇到NULL值主键**
- **现象**：增量快照阶段卡在"Chunk 0"永不完成
- **根因**：主键列包含NULL值，MySqlChunkSplitter在计算MIN/MAX范围时出现NULL，导致Chunk范围无限循环
- **解决方案**：升级到Flink CDC 3.0+（已修复NULL key问题），或确保主键列有NOT NULL约束

**故障案例2：水位对齐导致大量重复数据**
- **现象**：全量快照完成后，增量流又输出了大量在快照中已经在的数据
- **根因**：Chunk Size设置过大（100万行/Chunk），读取一个Chunk太耗时，导致Chunk读取期间产生的Binlog增量数据在后续的"水位对齐"中被重复读取
- **解决方案**：调小Chunk Size（从100万降到5000），缩短单个Chunk的读取时间

**故障案例3：高并发写入场景下水位对齐失败**
- **现象**：Flink CDC作业报错"Watermark alignment failed"，进入无限重试
- **根因**：MySQL的Binlog写入频率太高（每秒数万行），水位对齐时增量数据追不过来
- **解决方案**：增加Source并行度（提升Binlog读取速度），或使用Kafka作为中间缓冲
