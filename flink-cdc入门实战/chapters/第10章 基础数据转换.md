# 第10章 基础数据转换

## 1 项目背景

### 业务场景：CDC数据不是"拿来就用"的

数据同步到Kafka后，下游的数据分析团队发现了一个问题：**CDC原始数据包含了太多不需要的信息**，比如：
- MySQL的`TINYINT`类型（0/1）希望转为"是/否"
- 价格字段单位是"分"，需要转为"元"
- 手机号、身份证等敏感字段需要脱敏
- 不需要的字段（如`internal_remark`内部备注）需要去掉
- 需要排除某些操作的记录（比如LOGIN事件不需要同步）

简单来说：源数据库的设计是为"业务交易"服务的，不是为"数据分析"服务的。CDC原始数据需要进行**转换（Transform）**才能被下游使用。

### 数据转换层次

```
源数据（MySQL）
    │
    ▼
┌──────────────────────┐
│ 1. 列裁剪 (Projection)│  去掉不需要的列
│ 2. 条件过滤 (Filter)  │  排除特定条件的行
│ 3. 计算列 (Computed)  │  衍生新列
│ 4. 类型转换           │  调整数据类型
│ 5. 敏感脱敏           │  掩盖敏感信息
└──────────────────────┘
    │
    ▼
目标数据（Kafka / 数据湖 / 下游应用）
```

## 2 项目设计 · 三人交锋对话

**小胖**（喝着奶茶）：转换这词我懂，不就是SQL里的`SELECT col1, col2 FROM table WHERE col > 10`吗？搞Flink转换是不是一样的？

**大师**：本质确实是"读原始数据 → 应用变换规则 → 输出新数据"。但Flink CDC的数据转换和传统的SQL查询有两点核心区别：

1. **流式 vs 批式**：SQL是"对已有数据的一次性查询"，Flink是"对新来的每条数据持续应用转换"。转换函数需要是**纯函数**——输出只依赖于输入。
2. **行变更语义**：CDC的INSERT/UPDATE/DELETE事件在经过转换后，必须保持正确的行变更语义。比如`UPDATE`转换后，`before`和`after`都需要对应转换。

**小白**：那我们说的projection和filter分别对应什么？我看了Flink CDC YAML里transform块有`projection`和`filter`两个字段。

**大师**：这两个是CDC数据转换的两个最重要的操作：

**Projection（投影）**——决定输出哪些列：
```sql
-- 原始数据列: id, order_id, user_id, product, amount, status, create_time, internal_remark
-- 投影后的列（只保留需要的5列）:
projection: "id, order_id, product, amount * 0.01 AS amount_yuan, status"
```

**Filter（过滤）**——决定保留哪些行：
```sql
-- 过滤条件：只保留PAID和SHIPPED状态的订单
filter: "status = 'PAID' OR status = 'SHIPPED'"
```

这两个操作的组合可以实现绝大多数CDC数据清洗需求。

**小白**：那projection支持表达式操作吗？我知道在SQL中可以做`amount * 0.01`、`CONCAT(first_name, ' ', last_name)`这种操作，Flink CDC支持吗？

**大师**：Flink CDC 3.x的Transform API基于**Calcite SQL解析引擎**，支持以下表达式：

| 类别 | 支持的表达式 | 示例 |
|------|------------|------|
| 算术 | `+`, `-`, `*`, `/`, `%` | `amount * 0.01` |
| 字符串 | `||`, `CONCAT`, `UPPER`, `LOWER`, `SUBSTRING` | `UPPER(product)` |
| 条件 | `CASE WHEN ... THEN ... ELSE ... END` | `CASE WHEN status='PAID' THEN '已支付' ELSE '其他' END` |
| 类型 | `CAST` | `CAST(amount AS INT)` |
| 时间 | `EXTRACT`, `YEAR`, `MONTH`, `DAY` | `EXTRACT(YEAR FROM create_time)` |
| NULL | `COALESCE`, `IFNULL`, `NULLIF` | `COALESCE(discount, 0)` |

**技术映射**：Projection就像"餐厅菜单"——你想吃什么菜就点什么菜（选列），还可以要求"加辣"（表达式转换）。Filter就像"门禁"——满足条件的人才能进去（保留行），不满足的挡在外面（丢弃行）。

**小胖**：那如果我想给某个列改个名字怎么办？MySQL叫`user_id`，但我下游系统叫`uid`。

**大师**：Projection支持**别名（Alias）**：

```yaml
projection: "user_id AS uid, order_id AS order_no, product, amount"
```

只需要在列名后面加`AS 新名称`就可以了。

## 3 项目实战

### 环境准备

使用第3章的Docker环境。Flink CDC Transform功能是内置的，不需要额外依赖。

### 分步实现

#### 步骤1：准备测试表和数据

```sql
USE shop;

CREATE TABLE orders_full (
    id                INT AUTO_INCREMENT PRIMARY KEY,
    order_id          VARCHAR(64),
    user_id           INT,
    product           VARCHAR(128),
    amount_cents      INT           COMMENT '金额（单位：分）',
    status            VARCHAR(32),
    payment_method    VARCHAR(32),
    internal_remark   VARCHAR(256)  COMMENT '内部备注（敏感）',
    phone             VARCHAR(32)   COMMENT '手机号（敏感）',
    create_time       TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO orders_full VALUES
(1, 'ORD001', 101, 'iPhone 15', 699900, 'PAID', 'wechat', '内部优惠券抵扣50', '13800138001', NOW()),
(2, 'ORD002', 102, 'MacBook Air', 899900, 'SHIPPED', 'alipay', '企业客户', '13900139002', NOW()),
(3, 'ORD003', 103, 'AirPods', 199900, 'PENDING', 'credit_card', NULL, '13700137003', NOW());
```

#### 步骤2：使用DataStream API实现自定义转换

```java
package com.example;

import org.apache.flink.api.common.functions.MapFunction;
import org.apache.flink.cdc.connectors.mysql.source.MySqlSource;
import org.apache.flink.cdc.debezium.JsonDebeziumDeserializationSchema;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;

/**
 * CDC数据转换演示——在DataStream层做投影、过滤、计算
 */
public class BasicTransformDemo {

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        env.enableCheckpointing(5000);

        MySqlSource<String> source = MySqlSource.<String>builder()
            .hostname("localhost").port(3306)
            .databaseList("shop").tableList("shop.orders_full")
            .username("cdc_user").password("cdc_pass")
            .deserializer(new JsonDebeziumDeserializationSchema())
            .serverId("5400-5401")
            .startupOptions(
                org.apache.flink.cdc.connectors.mysql.table.StartupOptions.latest())
            .build();

        DataStream<String> cdcStream = env.fromSource(source,
            org.apache.flink.api.common.eventtime.WatermarkStrategy.noWatermarks(),
            "MySQL CDC");

        // 数据转换：投影 + 过滤 + 计算 + 脱敏
        DataStream<String> transformed = cdcStream.map(
            new MapFunction<String, String>() {
                @Override
                public String map(String json) throws Exception {
                    // 1. 判断操作类型——只保留INSERT和UPDATE
                    if (!json.contains("\"op\":\"c\"")
                        && !json.contains("\"op\":\"u\"")
                        && !json.contains("\"op\":\"r\"")) {
                        return null; // 过滤掉DELETE
                    }

                    // 2. 投影：去掉internal_remark，新增金额（元）字段
                    // 简化处理：用字符串替换模拟投影
                    String result = json;

                    // 3. 过滤：只保留PAID和SHIPPED状态的记录
                    if (!json.contains("\"status\":\"PAID\"")
                        && !json.contains("\"status\":\"SHIPPED\"")) {
                        return null; // PENDING状态的订单不过滤掉
                    }

                    // 4. 金额单位转换：分→元，并去掉internal_remark和phone
                    result = result.replaceAll("\"internal_remark\":\"[^\"]*\",?", "");
                    result = result.replaceAll("\"phone\":\"[^\"]*\",?", "");

                    result = result.replaceAll(
                        "\"amount_cents\":(\\d+)",
                        "\"amount_yuan\":$1");

                    // 5. 金额除以100（分转元）
                    result = result.replaceAll(
                        "\"amount_yuan\":(\\d+)",
                        "\"amount_yuan\":" + (Long.parseLong(
                            result.replaceAll(".*\"amount_yuan\":(\\d+).*", "$1")) / 100.0));

                    return result;
                }
            }).filter(json -> json != null); // 去掉被过滤的null记录

        transformed.print();

        env.execute("Basic Transform Demo");
    }
}
```

**预期输出（原始JSON vs 转换后）：**

原始：
```json
{"before":null,"after":{"id":1,"order_id":"ORD001","amount_cents":699900,"status":"PAID","internal_remark":"内部优惠券抵扣50","phone":"13800138001",...},"op":"c",...}
```

转换后：
```json
{"before":null,"after":{"id":1,"order_id":"ORD001","amount_yuan":6999.0,"status":"PAID",...},"op":"c",...}
```

#### 步骤3：使用Pipeline YAML实现声明式转换

创建 `pipeline-transform.yaml`：

```yaml
source:
  type: mysql
  hostname: localhost
  port: 3306
  username: cdc_user
  password: cdc_pass
  tables: shop.orders_full

sink:
  type: kafka
  properties:
    bootstrap.servers: localhost:9092
    auto.create.topics.enabled: true

transform:
  - source-table: shop.orders_full
    # 投影：只保留指定列，并做类型转换和别名
    projection: |
      id,
      order_id,
      user_id,
      product,
      CAST(amount_cents / 100 AS DECIMAL(10,2)) AS amount_yuan,
      status,
      payment_method,
      create_time
    # 过滤：只保留支付和已发货的订单
    filter: "status = 'PAID' OR status = 'SHIPPED'"
    # 可选：重定义主键（默认使用源表主键）
    primary-keys: id
    # 可选：定义分区键
    partition-keys: status

pipeline:
  name: Transform Demo
  parallelism: 1
```

#### 步骤4：实现敏感数据脱敏（自定义UDF）

创建脱敏UDF（User-Defined Function）：

```java
package com.example;

import org.apache.flink.cdc.common.udf.UserDefinedFunction;

/**
 * CDC脱敏UDF：手机号中间4位脱敏
 * 输入: 13800138001 → 输出: 138****8001
 */
public class PhoneMaskUDF extends UserDefinedFunction {

    public String eval(String phone) {
        if (phone == null || phone.length() < 7) {
            return phone;
        }
        return phone.substring(0, 3)
            + "****"
            + phone.substring(phone.length() - 4);
    }

    // 重载：支持整数类型输入
    public String eval(Long phone) {
        return eval(String.valueOf(phone));
    }
}
```

配置到Pipeline中：

```yaml
transform:
  - source-table: shop.orders_full
    projection: |
      id,
      order_id,
      phone_mask(phone) AS phone_masked,   # 使用UDF脱敏
      amount_yuan,
      status
    filter: "status = 'PAID'"

# UDF定义
udfs:
  - function-name: phone_mask
    class-name: com.example.PhoneMaskUDF
```

#### 常见陷坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| Projection后DELETE事件的before为null | 投影只处理了after字段 | 确认投影同时覆盖before和after |
| Filter条件中的字符串比较失败 | SQL方言差异 | 使用单引号`'PAID'`而非双引号 |
| CAST类型转换失败 | 源数据包含非法值 | 使用`TRY_CAST`安全转换 |
| UDF编译失败 | Janino语法限制 | 确认UDF中不含泛型、lambda等Janino不支持的语法 |
| 过滤后DELETE事件丢失 | filter条件排除了DELETE的before数据 | DELETE事件需要特殊处理，不应被过滤掉 |

## 4 项目总结

### 转换操作对比

| 操作 | 作用范围 | 性能影响 | DataStream | SQL API | Pipeline YAML |
|------|---------|---------|-----------|---------|--------------|
| Projection | 列级别 | 低 | ✅ map() | ✅ SELECT | ✅ |
| Filter | 行级别 | 低 | ✅ filter() | ✅ WHERE | ✅ |
| 计算列 | 列级别 | 中 | ✅ map() | ✅ SELECT expr | ✅ |
| 类型转换 | 列级别 | 低 | ✅ CAST | ✅ CAST | ✅ |
| 脱敏 | 列级别 | 中 | ✅ 自定义 | ❌ 需UDF | ✅ UDF |
| 行转列 | 行级别 | 高 | ✅ flatMap | ❌ | ❌ |
| 多流Join | 跨流 | 高 | ✅ coGroup/connect | ✅ JOIN | ❌ |

### 最佳实践

1. **Projection优先于Filter**：先裁剪列减少数据量，再过滤行减少数据量，最后做计算。这个顺序能最大化性能。
2. **Filter近Source**：在转换链路中，filter尽量靠近Source执行，避免不必要的序列化和网络传输。
3. **UDF设计原则**：UDF必须是**确定性的**（同样的输入总是得到同样的输出），否则Checkpoint恢复后结果不一致。
4. **避免复杂计算**：CDC数据转换中避免递归、循环、外部IO等操作，这些会阻塞数据流。
5. **保留原始数据**：建议在转换后的数据中保留`op`、`ts_ms`等元数据字段，方便下游审计和排错。

### 注意事项

1. **Decimal精度**：`CAST(amount / 100 AS DECIMAL(10,2))`可能丢失精度，优先使用`CAST(amount AS DECIMAL(10,2)) / 100`。
2. **NULL传播**：表达式中任何操作数的值为NULL，整个表达式结果也为NULL，需要`COALESCE`处理。
3. **DELETE事件过滤风险**：如果filter条件过滤掉了DELETE事件，下游写入了该行的DELETE后，将无法接收到"该行已被删除"的信息，导致下游和目标库数据不一致。

### 常见踩坑经验

**故障案例1：Filter过滤了DELETE导致下游数据删不掉**
- **现象**：下游数据仓库中某行数据已被MySQL删除，但数仓中仍然存在
- **根因**：Filter条件写的是`status = 'PAID'`，DELETE事件的after为null，filter条件计算为false，DELETE事件被丢弃
- **解决方案**：Filter条件中增加对DELETE事件的保留：`filter: "op = 'd' OR (status = 'PAID')"`

**故障案例2：Projection字段太多导致性能下降**
- **现象**：每秒处理量从10000条下降到2000条
- **根因**：一个表有200列，原样全部投影到下游，增加了序列化和网络开销
- **解决方案**：只投影下游需要的列（一般不超过20列），不需要的列尽早裁剪

**故障案例3：UDF中的静态变量在多并行度下数据错乱**
- **现象**：脱敏手机号在不同并行度下结果不同
- **根因**：UDF中使用了`static SimpleDateFormat`（非线程安全），多个并行子任务共享导致数据竞争
- **解决方案**：使用ThreadLocal或实例变量（每个并行实例独立创建）

### 思考题

1. **进阶题①**：在Flink CDC中，`UPDATE`事件经过投影和过滤转换后，`before`和`after`的对应关系是否会改变？比如投影只保留了3列，但源表有10列，UPDATE的`before`和`after`是否仍然对应同一条记录？

2. **进阶题②**：假设源表中有一个`JSON`类型的字段`ext_info`，其中包含了`{"coupon":"DISCOUNT50", "source":"APP"}`。如何在Flink CDC的Projection中从JSON字段提取`coupon`和`source`作为独立列？提示：考虑UDF配合JSON解析，或Flink SQL的JSON_VALUE函数。
