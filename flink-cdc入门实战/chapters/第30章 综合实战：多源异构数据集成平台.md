# 第30章 综合实战：多源异构数据集成平台

> **本文属于中级篇综合实战**，整合第16~29章的核心知识：Pipeline YAML、增量快照、Schema Evolution、数据湖（Iceberg）、OLAP（Doris）、性能调优、可观测性、生产部署。构建一个完整的企业级多源异构数据集成平台。

## 1 项目背景

### 业务场景：企业级数据中台的CDC底座

某互联网公司的数据中台团队需要构建一个**统一的CDC数据集成平台**，将公司所有业务数据库的变更数据实时汇聚到数据中台。需求如下：

| 数据源 | 数据库 | 表数量 | 数据量 | 用途 |
|-------|--------|--------|-------|------|
| 订单系统 | MySQL | 12张表 | ~3TB | 实时OLAP（Doris） |
| 用户系统 | PostgreSQL | 8张表 | ~500GB | 数据湖（Iceberg） |
| 日志系统 | MongoDB | 5个集合 | ~2TB/天 | 流式下游（Kafka） |

**架构要求：**
1. 所有数据源统一接入、统一管理
2. CDC数据同时写入：
   - **Kafka**：供下游微服务/实时计算消费
   - **Iceberg**：数据湖存储，用于历史回溯和分析
   - **Doris**：实时OLAP，用于运营看板
3. 全流程监控告警
4. 支持DDL变更自动同步（Schema Evolution）
5. 端到端数据一致性（Exactly-Once）

## 2 项目设计 · 三人交锋对话

**小胖**（搓手）：终于到综合实战了！这次要搞定三个数据源、三个目标的集成。相当于把第8章的多数据源、第23章的Iceberg、第25章的Doris串起来。

**大师**：是的。但这里有个架构设计题——**为什么要用3个独立的Flink Pipeline，而不是1个整合的Pipeline？**

**小胖**（想）：1个Pipeline看起来更酷？统一管理嘛。

**大师**：实际生产环境恰恰相反——**每个数据源独立一个Pipeline**。原因有三：

1. **故障隔离**：MongoDB Pipeline挂了不影响MySQL Pipeline
2. **独立版本升级**：PostgreSQL连接器升级不需要影响其他Pipeline
3. **资源独立配置**：MySQL数据量大（3TB）需要更多资源，MongoDB（2TB/天）需要更高的写入吞吐，独立配置更灵活

**小白**：那这三个Pipeline之间有没有依赖关系？比如订单数据（MySQL）关联用户数据（PostgreSQL），需要Join才能得到宽表？

**大师**：好问题！宽表Join在Flink CDC Pipeline YAML中不直接支持。解决方案分层：

```
架构分层:
  Layer 1: CDC数据摄取层（当前实战的内容）
    把原始数据（MySQL/PG/MongoDB）直接同步到Kafka和Iceberg
    → 保持原始Schema，不做转换

  Layer 2: 宽表合并层（需要额外的Flink SQL作业）
    从Kafka消费3个Topic，做Streaming Join生成宽表
    → 宽表写入Doris

  Layer 3: 数据服务层
    Doris/Iceberg对外提供查询服务
```

所以本章实现的是Layer 1（数据摄取层），Layer 2的宽表合并在后续章节可以作为拓展题。

**技术映射**：多源集成平台像"一个国际化机场的行李处理系统"——不同航班（MySQL/PG/MongoDB）的行李（CDC数据）通过不同的传送带（独立Pipeline）进入同一个分拣中心（Kafka/Iceberg）。行李不会因为一个航班的传送带坏了就影响其他航班。

## 3 项目实战

### 环境准备

**统一配置目录：**

```bash
mkdir -p /data/cdc-platform/{pipeline,lib,checkpoints,conf}
```

### 分步实现

#### 步骤1：Pipeline 1 — MySQL → Kafka + Iceberg

```yaml
# pipeline-mysql-iceberg.yaml
source:
  type: mysql
  hostname: ${MYSQL_HOST}
  port: 3306
  username: ${MYSQL_USER}
  password: ${MYSQL_PASSWORD}
  tables: shop.orders, shop.order_items, shop.products, shop.payments
  server-id: 5400-5404
  scan.startup.mode: initial
  capture-new-tables: true
  debezium:
    snapshot.locking.mode: none
    max.batch.size: 20480
    max.queue.size: 81920

sink:
  - type: kafka
    properties:
      bootstrap.servers: ${KAFKA_BOOTSTRAP}
    topic-prefix: cdc_mysql_
    sink:
      semantic: exactly-once
      batch-size: 16384

  - type: iceberg
    catalog-type: hadoop
    warehouse: s3a://cdc-lake/iceberg
    catalog-database: cdc_db
    table-prefix: mysql_
    auto-create-table: true
    upsert-mode: merge-on-read
    commit-interval: 60s

route:
  - source-table: shop.orders
    sink-table: orders

pipeline:
  name: CDC MySQL Platform
  parallelism: 4
  schema.change.behavior: EVOLVE
  execution.runtime-mode: STREAMING
```

**提交命令：**
```bash
flink-cdc.sh pipeline-mysql-iceberg.yaml \
  --target kubernetes-application \
  --kubernetes.container.image flink:1.20.3 \
  --name "cdc-mysql-platform" \
  --jobmanager.memory 2048m \
  --taskmanager.memory 4096m \
  --taskmanager.slots 4
```

#### 步骤2：Pipeline 2 — PostgreSQL → Kafka + Iceberg

```yaml
# pipeline-postgres-iceberg.yaml
source:
  type: postgres
  hostname: ${POSTGRES_HOST}
  port: 5432
  username: ${POSTGRES_USER}
  password: ${POSTGRES_PASSWORD}
  database: user_db
  schema-list: public
  tables: public.users, public.user_addresses, public.user_levels
  slot.name: cdc_platform_slot
  publication.name: cdc_publication
  decoding.plugin.name: pgoutput

sink:
  - type: kafka
    properties:
      bootstrap.servers: ${KAFKA_BOOTSTRAP}
    topic-prefix: cdc_pg_

  - type: iceberg
    catalog-type: hadoop
    warehouse: s3a://cdc-lake/iceberg
    catalog-database: cdc_db
    table-prefix: pg_
    auto-create-table: true

route:
  - source-table: public.users
    sink-table: users

pipeline:
  name: CDC PostgreSQL Platform
  parallelism: 2
  schema.change.behavior: EVOLVE
```

#### 步骤3：Pipeline 3 — MongoDB → Kafka

```yaml
# pipeline-mongo-kafka.yaml
source:
  type: mongodb
  hosts: ${MONGO_HOST}:27017
  username: ${MONGO_USER}
  password: ${MONGO_PASSWORD}
  database-list: logs_db
  collection-list: logs_db.app_logs, logs_db.access_logs
  connection.options: replicaSet=rs0&authSource=admin
  copy.existing: true
  heartbeat.interval.ms: 10000

sink:
  type: kafka
  properties:
    bootstrap.servers: ${KAFKA_BOOTSTRAP}
  topic-prefix: cdc_mongo_
  sink:
    semantic: exactly-once

route:
  - source-table: logs_db.app_logs
    sink-table: app_logs
  - source-table: logs_db.access_logs
    sink-table: access_logs

pipeline:
  name: CDC MongoDB Platform
  parallelism: 2
  schema.change.behavior: IGNORE     # MongoDB没有DDL，使用IGNORE
```

#### 步骤4：配置统一监控

```yaml
# prometheus-cdc-rules.yml — 统一告警规则
groups:
  - name: cdc-platform-alerts
    rules:
      - alert: MySQLPipelineLag
        expr: flink_taskmanager_job_task_operator_currentFetchDelay{job_name=~".*mysql.*"} > 30000
        for: 2m
        labels: { severity: critical, pipeline: mysql }

      - alert: PGPipelineLag
        expr: flink_taskmanager_job_task_operator_currentFetchDelay{job_name=~".*postgres.*"} > 60000
        for: 2m
        labels: { severity: critical, pipeline: postgres }

      - alert: MongoPipelineLag
        expr: flink_taskmanager_job_task_operator_currentFetchDelay{job_name=~".*mongo.*"} > 60000
        for: 2m
        labels: { severity: critical, pipeline: mongo }
```

#### 步骤5：验证端到端数据流

```bash
#!/bin/bash
echo "=== CDC平台端到端验证 ==="

# 验证1：MySQL → Kafka
echo "1. MySQL写入 → Kafka消费..."

  USE shop; INSERT INTO orders VALUES (99, 'ORD_PLATFORM', 1001, 'Platform Test', 99.99, 'PAID', NOW(), NOW());

# 验证2：PostgreSQL → Iceberg
echo "2. PostgreSQL写入 → Iceberg验证..."

  INSERT INTO users VALUES (999, 'platform_test', 'test@cdc.com', 'VIP', 1000, NOW());
"
sleep 10
# 使用Spark SQL或Trino查询Iceberg表
echo "验证Iceberg中是否有新数据..."

# 验证3：MongoDB → Kafka
echo "3. MongoDB写入 → Kafka验证..."
docker exec mongodb-cdc mongosh -u cdc_user -p cdc_pass --eval "
  use logs_db;
  db.app_logs.insertOne({level: 'INFO', message: 'Platform test log', timestamp: new Date()});
"
sleep 5
docker exec kafka-cdc kafka-console-consumer \
  --topic cdc_mongo_logs_db.app_logs \

  --from-beginning --max-messages 1
```

#### 常见陷坑及解决方法

| 问题 | 原因 | 解决方法 |
|------|------|---------|
| Kafka Topic命名冲突 | 多个Pipeline使用相同的topic-prefix | 不同Pipeline使用不同的prefix（cdc_mysql_, cdc_pg_, cdc_mongo_） |
| Iceberg表被多个Pipeline覆盖写入 | 两个Pipeline写到同一个Iceberg表 | 每个Pipeline使用不同的`table-prefix`隔离 |
| MongoDB全量快照慢 | 大量文档无索引 | 为MongoDB集合的`_id`字段创建索引 |
| Checkpoint超大（>1GB） | 三路Pipeline共享Flink Session的State | 使用Application模式隔离State |
| S3写入权限错误 | Iceberg的S3凭证未正确配置 | 配置`fs.s3a.access.key`和`fs.s3a.secret.key` |

### 平台拓展思路

1. **拓展1**：增加Doris Sink，将Kafka中的CDC数据通过Flink SQL写入Doris做实时分析
2. **拓展2**：增加Schema Registry（如Confluent Schema Registry），统一管理Kafka Topic的Schema版本
3. **拓展3**：增加CDC数据质量监控——设置数据完整性校验规则（如表A的行数 = 表B的行数）
4. **拓展4**：增加动态表发现（`capture-new-tables=true`），自动同步新增表

### 思考题

1. **进阶题①**：在本章的三Pipeline架构中，如果MySQL的订单表和PostgreSQL的用户表需要在Doris中Join为宽表，你需要在什么环节实现这个Join？在Flink CDC Pipeline中？还是在Doris中？还是在两者之间的Flink SQL作业中？各自有什么优缺点？

2. **进阶题②**：本章使用3个独立的Flink Pipeline（Application模式）来分别处理MySQL/PG/MongoDB。如果未来需要增加到50个数据源（50个Pipeline），K8s上会有50个JM + 50组TM Pod。这种架构的资源开销和运维成本如何？是否有更高效的方案？提示：考虑Flink Session模式和Pipeline Partitioning策略。
