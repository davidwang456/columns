## 上一章思考题回顾

1. **超时先于熔断**：无超时则线程/连接**堆积**，熔断**来不及**触发；**超时**是基础，**熔断**是**系统性止损**。  
2. **重试 + 幂等**：仅对**读操作**或**幂等写**重试；配合 **`Idempotency-Key`** 与 **去重表**。

---

## 1 项目背景

下游 WMS **延迟飙升**，线程池打满，**级联故障**。需 **Resilience4j** / **Sentinel** 实现 **限流、熔断、舱壁**。

**痛点**：  
- **重试风暴** 放大故障。  
- **全链路超时** 未传递。

---

## 2 项目设计（剧本式对话）

**角色**：小胖 / 小白 / 大师。  
**结构**：超时 → 重试 → 熔断 → 舱壁；与「错误预算」对齐。

**小胖**：下游慢我就加超时时间，用户多等一会儿嘛。

**大师**：超时是**失败快速**；没有超时，线程/连接会**堆积**，熔断**来不及**触发——雪崩从这儿开始。

**技术映射**：**TimeLimiter + CircuitBreaker + Bulkhead**。

**小白**：**重试** 不是能提升成功率吗？

**大师**：对**读**或**幂等写**；对**非幂等**重试是**重复下单**；还要 **指数退避 + 抖动**，避免重试风暴。

**大师**：**SRE 黄金信号**：**延迟、流量、错误、饱和度**；配合 **错误预算**（SLO）决定何时熔断、何时降级。

## 3 项目实战

### 3.1 Resilience4j 示例（概念）

```java
@CircuitBreaker(name = "wms")
@TimeLimiter(name = "wms")
public Mono<String> ship(String orderId) { /* WebClient call */ return Mono.just("ok"); }
```

### 3.2 分步实现

- **WebClient** 配置 **`responseTimeout`**。  
- **Reactor** 上 **`timeout(Duration)`**。  
- **网关** 层 **全局限流**。

**步骤 4 — 目标（故障注入）**：用 **Toxiproxy** 给下游加 **200ms→2000ms** 延迟，观察 **线程池队列** 与 **熔断状态机**。

**可能遇到的坑**

| 现象 | 原因 | 处理 |
|------|------|------|
| 重试风暴 | 全站重试 | 限制重试次数与退避 |

## 4 项目总结

### 常见踩坑经验

1. **fallback** 返回错误业务结果未监控。  
2. **线程池隔离** 配置过小。  
3. **重试** 与 **事务** 叠加。
