# 第 26 章：多数据源与读写分离——路由与坑

> **业务线**：电商 / 订单履约微服务（拟真场景）。本章可独立阅读；与全书案例弱关联。

> **篇章**：中级篇（全书第 19–35 章；架构与分布式、性能、可观测性）

## 上一章思考题回顾

1. **路由键**：常用 **`ThreadLocal` 持有当前租户或读写标记**，在 **`AbstractRoutingDataSource#determineCurrentLookupKey`** 中返回 **DataSource key**；由 **AOP** 在 `@Transactional(readOnly=true)` 前后设置。  
2. **只读路由**：**主从延迟** 导致读不到刚写入数据 → 关键读走 **主库** 或 **强制同步**；**事务传播** 跨数据源需 **JTA** 或业务拆分。

---

## 1 项目背景

报表 **慢查询** 影响 **OLTP**；团队将 **只读** 流量切到 **从库**。若代码里手写两个 `DataSource`，事务与 **连接泄漏** 风险高。

**痛点**：  
- **同一事务** 跨库 **XA** 复杂度高。  
- **路由错误** 写落到从库失败。  
- **连接池** 配置重复。

---

## 2 项目设计（剧本式对话）

**角色**：小胖 / 小白 / 大师。  
**结构**：先否定「两个 DataSource 随便 new」→ 路由键与 ThreadLocal → 主从延迟。

**小胖**：我写两个 `@Bean DataSource`，一个查一个写，不就行了？

**大师**：行，但事务与连接绑定需要 **TransactionManager** 协同；更复杂的是 **读写路由**、**事务只读标记**、**连接池**——否则你会得到 **随机连库** 与 **连接泄漏**。

**技术映射**：**AbstractRoutingDataSource** + **TransactionRouting**。

**小白**：路由键放哪？

**大师**：常用 **`ThreadLocal`** 保存当前 **READ/WRITE**；在 `determineCurrentLookupKey()` 返回；由 **AOP** 在 `@Transactional(readOnly=true)` 前后设置/清理。

**小白**：主从延迟导致「刚下单查不到」咋办？

**大师**：关键读走 **主** 或 **同步复制**；业务上给 **读己写** 的短暂窗口（体验层）。

## 3 项目实战

### 3.1 环境准备

| 项 | 说明 |
|----|------|
| 数据库 | 双 H2（不同 URL）或 Testcontainers MySQL 主从 |
| 目标 | 验证 **同线程** 读写路由切换 |

### 3.2 分步实现

**骨架**

```java
public class RoutingDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        return DataSourceContextHolder.get();
    }
}
```

**AOP**：在事务开始前根据 `readOnly` 设置 `READ/WRITE`（**务必 finally 清理 ThreadLocal**）。

**步骤 3 — 目标（对照实验）**：同一事务内禁止切换（应报错或强制走主），验证「**一事务一连库**」约束。

### 3.3 完整代码清单与仓库

`chapter22-routing-ds`。

### 3.4 测试验证

双 H2 或 Testcontainers MySQL 主从模拟。

**命令**：`mvn -q test`。

**可能遇到的坑**

| 现象 | 原因 | 处理 |
|------|------|------|
| ThreadLocal 泄漏 | 未清理 | 用 try/finally |
| 懒加载跨库 | 会话与数据源不一致 | 事务边界与 fetch 策略 |

## 4 项目总结

### 常见踩坑经验

1. **@Transactional** 自调用路由失效。  
2. **懒加载** 跨库。  
3. **批处理** 误用从库。
