# 第 27 章：Bean 创建与循环依赖——三级缓存与边界

> **业务线**：电商 / 订单履约微服务（拟真场景）。本章可独立阅读；与全书案例弱关联。

## 上一章思考题回顾

1. **构造器注入** 循环依赖：无法在创建前放入 **早期引用**（对象未构造），**Spring 默认抛异常**；解决方向是 **重构** 或 **`ObjectFactory`/`@Lazy`**。  
2. **`ObjectFactory` 注入**：延迟获取依赖，**打破循环** 或 **prototype 按需创建**。

---

## 1 项目背景

`OrderService` ↔ `PaymentService` **setter 注入** 互相依赖，启动时 **奇迹般成功**；若改成 **构造器**，则失败。团队需理解 **边界** 避免 **线上埋雷**。

---

## 2 项目设计（剧本式对话）

**角色**：小胖 / 小白 / 大师。  
**结构**：先讲「神奇能启动」→ 三级缓存直觉 → 构造器为何不行。

**小胖**：我就喜欢两个 Service 互相注入，Spring 能启动就说明没问题吧？

**大师**：能启动，只说明 **setter/字段** 的循环在 **单例** 场景下被 **早期引用** 接住；换成 **构造器**，对象还没构造出来，**早期引用**放不进去——于是爆炸。

**技术映射**：三级缓存 **只解决单例 setter/字段循环**；**AOP 代理** 会加 **getEarlyBeanReference** 复杂度；三张表对应 **singletonFactories → earlySingletonObjects → singletonObjects**。

**小白**：那我把循环依赖当「设计模式」行不行？

**大师**：行，但像**互相欠账**：短期能周转，长期维护成本会指数上升；**解耦**（事件、Facade、拆分模块）才是正路。

---

## 3 项目实战

### 3.1 实验

**Setter 循环**：A→B→A，启动成功。  
**构造器循环**：失败，提示 `BeanCurrentlyInCreationException`。

### 3.2 分步实现

**推荐**：`@Lazy` 注入一方，或 **事件驱动** 解耦。

**步骤 3 — 目标（观察日志）**：在 `application.properties` 打开 `logging.level.org.springframework.beans.factory=DEBUG`（仅本地），观察 **Creating shared instance** 与 **circular** 相关日志（不同版本输出略有差异）。

### 3.4 测试验证

|------|------|------|

---

## 4 项目总结

### 常见踩坑经验

1. **prototype Bean** 不参与三级缓存。  
2. **@Async 代理** 与循环依赖交织。  
3. **设计层** 应消除循环，而非依赖框架技巧。
