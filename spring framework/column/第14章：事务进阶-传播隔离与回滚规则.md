# 第 14 章：事务进阶——传播、隔离与回滚规则

> **业务线**：电商 / 订单履约微服务（拟真场景）。本章可独立阅读；与全书案例弱关联。

## 上一章思考题回顾

1. **`REQUIRES_NEW`**：总是**新开**事务，外层挂起；内层提交/回滚**独立**。**`NESTED`**：使用**保存点**，内层回滚可**部分回滚**外层，外层仍可提交（底层依赖 DB savepoint）。  
2. **多 `@Transactional` 切面顺序**：由 **`@Order`** 与 **Advisor 顺序**决定；一般**事务切面应在外层**，否则可能出现「只读事务包裹写操作」等异常组合。

---

## 1 项目背景

下单需要 **写订单** 与 **记积分**。**积分失败**不应影响订单主单（业务妥协），或 **审计日志** 必须独立提交——这涉及 **传播行为**。大促时 **读多写少**，要评估 **读已提交** vs **可重复读** 与 **幻读** 风险。

**痛点**：  
- 误用 `REQUIRES_NEW` 导致 **死锁** 窗口变大。  
- **隔离级别**与 **MySQL 默认 RR** 行为不一致导致「同 SQL 不同结果」。  
- **checked exception** 未回滚，资金类 Bug。

---

## 2 项目设计（剧本式对话）

**角色**：小胖 / 小白 / 大师。  
**结构**：从「一把梭大事务」→ 传播/隔离语义 → 回滚与异常类型。

**小胖**：为啥不一个事务包全部？我 `@Transactional` 一贴，全世界都回滚，多安全。

**大师**：**一致性**不止数据库：**积分失败要不要挡下单**？**审计日志要不要跟订单同生共死**？这些属于**产品/业务妥协**，传播行为是在代码里把妥协**写清楚**。

**技术映射**：**Propagation** + **Isolation** + **rollbackFor** + **timeout**。

**小白**：`REQUIRES_NEW` 和 `NESTED` 听起来都像「里面一套、外面一套」，差别在哪？

**大师**：`REQUIRES_NEW` 是**另一条独立事务**（提交/回滚与外层无关）；`NESTED` 是**保存点**语义——内层回滚像「撤销一段」，外层还能继续（取决于异常与配置）。

**技术映射**：**NESTED** 依赖 **JDBC Savepoint**；某些数据源/驱动组合要谨慎验证。

**小胖**：隔离级别我选 `SERIALIZABLE`，是不是最稳？

**小白**：最稳也最慢，还容易 **死锁**；多数 OLTP 在 **RR/RC** 间权衡，配合 **乐观锁版本号** 更常见。

**大师**：像**地铁高峰期**：你把车厢锁成「一次只进一个人」（串行化），安全，但**吞吐归零**。

**小白**：`readOnly=true` 真有用吗？

**大师**：对 Hibernate/JPA 可优化；对 JDBC 主要是**声明意图**，在读写分离路由里常作为**读走从库**的开关（视基础设施而定）。

**小胖**：那我 `catch(Exception e){ log... }` 不抛，事务是不是就稳了？

**大师**：这叫**稳在日志里，烂在数据里**：默认只对 **RuntimeException/Error** 回滚，checked 异常不回滚；吞异常更是**提交脏数据**捷径。

## 3 项目实战

### 3.1 环境准备

| 项 | 说明 |
|----|------|
| 栈 | Spring Boot + JDBC + H2（或 Testcontainers MySQL 做隔离级别实验） |
| 目标 | 复现 **REQUIRES_NEW**「外层回滚、内层已提交」与 **rollback-only** 传播 |

### 3.2 分步实现

**步骤 1 — 目标**：准备两张表 `orders`、`points`（最小字段即可）。

**步骤 2 — 目标**：实现 `OrderWriteService#createOrder`（外层事务）与 `PointsService#addPoints`（`REQUIRES_NEW`）。

```java
@Service
public class OrderWriteService {
    private final PointsService points;

    public OrderWriteService(PointsService points) {
        this.points = points;
    }

    @Transactional
    public void createOrder(String id, String userId) {
        // insert order...
        points.addPoints(userId, 10);
        throw new IllegalStateException("simulate rollback outer");
    }
}

@Service
public class PointsService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void addPoints(String userId, int p) {
        // insert points...
    }
}
```

**运行结果（文字描述）**：外层异常后 **orders 回滚**；若 `addPoints` 已提交，**points 可能保留**——用 SQL 查询验证「**部分提交**」现象，并在需求上明确是否接受。

**步骤 3 — 目标（对照）**：把 `REQUIRES_NEW` 改为默认 `REQUIRED`，观察 **points 是否一并回滚**。

**可能遇到的坑**

| 现象 | 原因 | 处理 |
|------|------|------|
| 传播不生效 | 同类自调用 | 拆 Bean（第 28 章） |
| NESTED 不工作 | 数据源不支持 savepoint | 换引擎或改方案 |
| 死锁增多 | 事务粒度过大/索引缺失 | SQL 与索引联合治理 |

## 4 项目总结

### 优点与缺点

| 维度 | 显式传播 | 单一大事务 |
|------|----------|------------|
| 灵活性 | 高 | 低 |
| 复杂度 | 高 | 低 |

### 常见踩坑经验

1. **自调用** 传播不生效。  
2. **NESTED** 在不支持 savepoint 的引擎上行为差异。  
3. **超时** `timeout` 与慢 SQL。
