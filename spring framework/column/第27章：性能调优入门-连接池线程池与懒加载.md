# 第 27 章：性能调优入门——连接池、线程池与懒加载

> **业务线**：电商 / 订单履约微服务（拟真场景）。本章可独立阅读；与全书案例弱关联。

> **篇章**：中级篇（全书第 19–35 章；架构与分布式、性能、可观测性）

## 上一章思考题回顾

1. **HikariCP `maximum-pool-size`**：应 **小于** DB 最大连接且与 **并发模型** 匹配；一般 **略大于** 典型并行查询线程，避免过大导致 **DB 侧排队** 与 **上下文切换**。  
2. **`@Lazy`**：延迟初始化 Bean，**缩短启动**，但**首次请求变慢**且**隐藏循环依赖**；不适合关键路径。

---

## 1 项目背景

大促时 **RT P99** 飙高。profiler 显示 **连接等待**、**线程池满**、**懒加载 N+1**。

**痛点**：  
- **池化资源** 默认值在生产不适用。  
- **`@Async` 无界队列** OOM。

---

## 2 项目设计（剧本式对话）

**角色**：小胖 / 小白 / 大师。  
**结构**：先反对「拍脑袋调参」→ 指标驱动 → 池与队列的直觉。

**小胖**：我把连接池调到 200，线程池调到 200，是不是就快了？

**大师**：先问 **瓶颈在 DB 还是应用**：池太大可能把 **数据库打满**；线程太多只会增加 **上下文切换** 与 **锁竞争**。调优顺序：**指标 → 定位瓶颈 → 改一处验证**。

**技术映射**：**Micrometer** + **Hikari metrics** + **线程池指标**。

**小白**：`connection-timeout` 设很大，是不是更「稳」？

**大师**：更「慢」：等待连接的时间变长，**尾延迟**更差。超时是**失败快速**的一部分。

**小胖**：`@Lazy` 能让启动飞快，为啥生产不爱用？

**大师**：**首次请求**替启动还债；还可能**掩盖循环依赖**。像**把行李托运到落地再取**——省登机时间，但落地要等转盘。

## 3 项目实战

### 3.1 环境准备

| 项 | 说明 |
|----|------|
| 指标 | Actuator + Prometheus（第 16 章） |
| 压测 | k6 / JMeter / Gatling |

### 3.2 分步实现

**`application.yml`**

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      connection-timeout: 3000
```

**异步线程池**

```yaml
spring:
  task:
    execution:
      pool:
        core-size: 8
        max-size: 32
        queue-capacity: 200
```

**步骤 1 — 目标**：打开 **Hikari 注册指标**（`register-mbeans` 或通过 Micrometer）。

**步骤 2 — 目标**：压测时观察 **`hikaricp.connections.pending`** 与 **`threads.active`**。

**运行结果（文字描述）**：pending 长期大于 0 → **池过小或 SQL 慢**；队列持续增长 → **线程池或下游过慢**。

### 3.3 完整代码清单与仓库

`chapter24-perf`。

### 3.4 测试验证

压测工具（JMeter/k6）+ Grafana 看 **pool waiting**；压测脚本应包含 **思考时间** 与 **混合读写**。

**命令**：`mvn -q package` 后执行压测 CLI。

**可能遇到的坑**

| 现象 | 原因 | 处理 |
|------|------|------|
| 池等待高 | 慢 SQL / 长事务 | SQL 优化与缩短事务 |
| OOM | 无界队列 | 设队列上限与拒绝策略 |

## 4 项目总结

### 常见踩坑经验

1. **池大小** 与 **K8s replica** 相乘压垮 DB。  
2. **事务过长** 占连接。  
3. **日志同步写** 阻塞。
