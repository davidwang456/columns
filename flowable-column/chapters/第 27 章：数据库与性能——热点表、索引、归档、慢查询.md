# 第 27 章：数据库与性能——热点表、索引、归档、慢查询

> 版本说明：`ACT_RU_*` 运行时；`ACT_HI_*` 历史；不同数据库优化手段各异。[VERSION_MATRIX.md](../VERSION_MATRIX.md)

## 1）项目背景

星云生产半年 **`ACT_HI_ACTINST`** 达 **亿级**；报表「按部门统计平均审批时长」拖垮只读副本。DBA 要求：**索引治理**、**分区/归档**、**批量删除窗口**。**本章聚焦**：读路径热点、写放大来源、以及**归档 ≠ delete 全部**的风险。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：删历史行吗？

**大师**：行，但要 **法务保留年限** 与 **技术分批**；通常 **搬迁到冷存储** 而不是 `DROP`。

**小白**：读写分离？

**大师**：报表走 **只读副本**；接受 **秒级延迟**。

**小胖**：引擎表能乱加索引吗？

**大师**：**写入路径**会受损；必须 **DBA 评审**。

## 2.1）场景深化：「宽表」与数仓分层

引擎历史适合 **事实表**；给企业管理者看的「部门口径」常在数仓 **维度表** 补组织信息。**星云** T+1 批处理将 **`HistoricProcessInstance`** 与 **HR 维度** join 后入仓；线上 **OLTP** 避免大聚合。

## 3）项目实战（主代码片段）

### 3.1 慢查询定位（示意）

```sql
-- EXPLAIN ANALYZE 以 DBA 指导为准
SELECT * FROM ACT_HI_ACTINST_ WHERE PROC_INST_ID_ = ?;
```

### 3.2 归档思路（业务脚本，示意）

1. 选定 **endTime** 早于 cutoff 的实例集合；  
2. **搬运**到 **历史归档库**；  
3. 生产库 **分批删除**（或标记删除）。

### 3.3 应用侧

对超级大户 `businessKey` **限制历史查询范围**；提供 **下载 CSV** 而非在线超大 result。

## 4）项目总结

### 优点

- 控制 **成本**与 **延迟**。  
- **可持续运营**。

### 缺点

- **归档后**单一 SQL 追溯不再「一条出全」。

### 典型使用场景

- 年结清库、监管要求的 **冷存**。

### 注意事项

- **批量删除**锁表；  
- **外键/级联**顺序。

### 反例

高峰期跑 **大 DELETE**。**纠正**：**夜批 + 限流**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 慢 | 缺索引 | EXPLAIN |
| 膨胀 | full 历史 | 降级 |
| 主从延迟 | 大事务 | 拆分 |

### 给测试一句话

归档后 **抽样对账** 业务事实。

### 给运维一句话

**表空间告警** + **备份窗口** + **归档作业 SLA**。

## 附录：星云案例深潜——一次「归档脚本误伤生产」演习教训

夜批误把仍在运行的实例变量一并归档——虽及时中止，却暴露了 **WHERE 条件**缺少 `END_TIME_ IS NOT NULL` 的防呆。附录建议：**所有批量脚本**在沙箱先做 **count 校验**，再生产 **limit 小批次试运行**。
