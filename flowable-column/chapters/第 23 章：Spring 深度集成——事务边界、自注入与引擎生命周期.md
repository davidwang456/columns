# 第 23 章：Spring 深度集成——事务边界、自注入与引擎生命周期

> 版本说明：Spring Boot + Flowable Starter；`PlatformTransactionManager` 与引擎命令共事。[VERSION_MATRIX.md](../VERSION_MATRIX.md)

---

## 1）项目背景

星云 `OrderService` 在 `@Transactional` 方法里 **`save(采购单)`** 与 **`startProcessInstance`** 同行；一旦 DB 写入失败，期望流程也别落库。反之，异步作业线程里调用业务服务时，**事务边界**已变——很多「看似应该在同一事务」的操作其实跨线程。**本章聚焦**：Spring 事务与 Flowable 命令、**同类自调用**陷阱、以及 **Listener/Delegate** 中的传播行为。

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：引擎 API 算同一个事务吗？

**大师**：在同一线程、同一 Spring 管理的事务上下文中，一般 **参与**；具体以 **DataSource** 与 **TransactionManager** 绑定为准。

**小白**：`REQUIRES_NEW` 呢？

**大师**：会把流程写库与业务写库 **拆分提交**；一旦一边失败，另一边可能已成功——必须写清 **补偿**。

**小胖**：为什么 Listener 里注入的 Bean 有时是 null？

**大师**：Listener 若 **非 Spring 管理** 而用 `class=` 全限定名，需要 `delegateExpression`。

**小白**：async 任务呢？

**大师**：异步线程 **默认不在**原请求事务里——不要把「用户已提交成功」与「后台 Job 已成功」划等号。

**小胖**：ApplicationContext 手动拿 Bean？

**大师**：最后手段；优先 **依赖注入**。

## 2.1）场景深化：「先业务后流程」还是「先流程后业务」

两种顺序都有拥趸。**星云**在资金相关场景采用：**先写业务影子单 + pending 状态 → 再起流程**；流程失败可标记业务单失败。**若先起流程后写业务**，则要有 **补偿关闭流程** 策略。顺序不是教条，关键是**可逆操作**与 **幂等** 设计。

## 3）项目实战（主代码片段）

### 3.1 同事务启动

```java
@Service
public class ProcurementService {
  @Autowired RuntimeService runtimeService;

  @Transactional
  public void createDraftAndStart(String bk, Map<String,Object> vars) {
    // dao.insertDraft(...);
    runtimeService.startProcessInstanceByKey("procurement", bk, vars);
  }
}
```

### 3.2 避免自调用绕过 AOP

```java
@Service
public class A {
  @Autowired ApplicationContext ctx;
  public void outer() { ctx.getBean(A.class).inner(); }
  @Transactional public void inner() { ... }
}
```

### 3.3 JUnit

`@Transactional` 测试 + `@Rollback` 验证 **无流程实例** 残留（注意引擎测试配置）。

## 4）项目总结

### 优点

- Spring 生态无缝；  
- 声明式事务降低样板代码。

### 缺点

- **异步**与 **多数据源** 让心智负担指数上升。

### 典型使用场景

- 单体/模块化单体中的强一致写入。

### 注意事项

- **同类自调用**与 **代理**；  
- **只读副本延迟**若参与决策要想清楚。

### 反例

在异步 Job 里调用 **RequiresNew** 业务方法却假设失败能一起 rollback 流程前半段。**纠正**：**Saga/补偿**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 部分提交 | 传播级别错 | 明确设计 |
| 监听无事务 | 非 public / 自调用 | 修 AOP |
| 死锁 | 锁序不一致 | 统一资源序 |
| 脏读 | 隔离级别 | 业务容忍度 |

### 给测试一句话

**失败注入**：DB 抛错 vs Delegate 抛错 各测一轮。

### 给运维一句话

监控 **长事务** 与 **lock wait**；高峰与 Job 窗口错峰。

## 附：延展阅读——多数据源下的事务管理器绑定

若流程库与业务库分离，需两套 `DataSourceTransactionManager`；跨库一致性靠 **最终一致** 而非 XA（多数互联网架构取舍）。

## 附录：星云案例深潜——跨服务 Saga 与本章事务的错位

有团队把「流程 + 本地库」搞顺后，又接了外部支付。仍想用本地事务「一把梭」，结果出现**支付成功本地回滚**的不可恢复状态。附录再次强调：**跨服务请用消息/Saga**，Flowable 解决的是**人机协同与状态外化**，不是**分布式数据库**。
