# 第 20 章：DMN 决策表——规则外置与流程联动

> 版本说明：`dmn` 资源与 **Decision Service**；命中策略 **HIT POLICY** 以 DMN 1.x 为准。

---

## 1）项目背景

星云的折扣与**风控等级**规则每季度市场部都要调整；若写在 Java `if-else` 或 BPMN 网关条件， 발版节奏跟不上。**DMN** 把规则表 **独立资源** 部署，业务方可参与评审。**本章聚焦** `.dmn` 资源、**决策表**、以及 BPMN **Decision Task** 或 Java API 调用 **DecisionService** 将结果写回变量。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：DMN 和数据库规则表？

**大师**：DMN 是 **标准**、可视化；规则表适合**运营配置**但缺少引擎级版本治理时可自建。**混合**：DMN 管核心口径，运营参数走配置中心。

**小白**：命中多行？

**大师**：**Hit Policy** 决定取首行、聚合、收集等——要选对。

**小胖**：和流程网关？

**大师**：`decisionResult` 变量给排他网关分流。

**小白**：测试？

**大师**：对每个 **规则行** 做 **参数化单测**。

## 2.1）场景深化：FEEL 类型与边界测试

DMN 对 **日期**、**string vs number** 有严格 FEEL 规则；采购场景常在月末改价，若测试只覆盖「整数金额」，上线后第一个带小数的合同就可能 **未命中**。建议在评审会附带 **FEEL 单元测试导出**（若工具链支持），并把**默认输出**策略写清：未命中时是拒绝、人工还是兜底价。

## 4）项目总结

### 优点

- **规则与流程解耦**；变更可灰度。  
- 非开发可读性较好。

### 缺点

- **类型**与 **null** 处理要严。  
- 复杂决策可能不如代码清晰。

### 典型使用场景

- 折扣阶梯、信用评分、路由等级。

### 注意事项

- **版本**与 **回滚**。  
- **输入事实**校验。

### 反例

决策表复制粘贴多份「几乎相同」——改一处漏一处。**纠正**：**统一子决策** + **引用**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 未命中 | 无默认行 | FIRST+默认 |
| 类型错 | FEEL 类型 | 显式转换 |
| 部署找不到 | key 不一致 | 命名规范 |
| 性能 | 大行宽表 | 预编译/缓存 |

### 给测试一句话

**决策表行覆盖** + **边界值**；与 **BPMN 网关**联调集成测。

### 给运维一句话

DMN 与 BPMN **同期发布**；监控 **evaluate 延迟**。

## 附：延展阅读——业务可读的决策表治理

星云产品侧可在 Confluence 维护「决策表镜像」，每次 DMN 变更必须 **同步更新镜像**与 **变更原因**，满足业务方 sign-off；技术再通过 deployment 推向各环境，避免「口头改规则」。

---

## 附录：星云案例深潜——业务部门「手改规则」幻想

有的团队希望运营在页面直接改 DMN，却不改流程版本。现实是：**决策表也是版本化制品**，应走与 BPMN 同级评审。若确实需要自助，另建**参数中心**而非直接写生产 DMN。附录再次强调：**无评审的高频改规则 = 金融风险**。
