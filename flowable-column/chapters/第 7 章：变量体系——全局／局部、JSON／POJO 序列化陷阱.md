# 第 7 章：变量体系——全局／局部、JSON／POJO 序列化陷阱

> 版本说明：持久化变量依赖数据库与 `VariableType`；大 JSON 慎用。

---

## 1）项目背景

星云采购审批链上需要传递 **金额、币种、供应商 ID、合同草案附件引用** 等。业务方希望「流程里什么都能塞」，而平台组关心：**变量放哪一级**、**能否版本升级**、**能否序列化**、**会不会把 DB 撑爆**。**本章聚焦** `RuntimeService`/`TaskService` 的变量 API：**全局（流程实例）**与**局部（execution/task）**作用域，以及 JSON、Serializable POJO、JPA 实体等类型的注意点。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：全局变量不就是 `setVariable` 吗？

**大师**：默认多指 **流程实例级**，所有子执行路径可读（除非你用 **local** 覆盖）。并行分支里若多人改同一全局变量——**竞态**就来了，要设计锁或局部变量。

**小白**：局部变量给谁用？

**大师**：**`setVariableLocal`** 常见在：多实例每条分支、子执行、任务级临时标记。理解「**哪个 executionId**」在非主线并行里是关键（第 9、18 章）。

**小胖**：能放 BigDecimal 吗？

**大师**：可以，但要全链路类型一致；从 JSON 反序列化时小心 **Double**。

**小白**：放整个 Order 实体行不行？

**大师**：JPA 实体当变量要小心 **懒加载与会话**；更稳的是只存 **ID** 或 **DTO 快照**。否则流程实例与 ORM 生命周期绑死。

**小胖**：JSON 字符串大行吗？

**大师**：小对象可以；巨型 JSON 影响 **历史表膨胀** 与查询。考虑**外部存对象存储**，变量里放指针。

**小白**：变量名谁定？

**大师**：团队约定 **前缀+业务域**，与网关条件、EL 一致；别用魔法字符串散落。

**小胖**：Transient 变量有吗？

**大师**：有「非持久」变量类型配置；用于**中间计算不落库**——重启丢失，慎用。

**小白**：版本迁移时变量结构变了？

**大师**：第 26 章；本章先养成 **向后兼容** 习惯：增字段不删语义、用 Optional 读。

**小胖**：能存 null 吗？

**大师**：能，但 UEL 条件 `amount > 50000` 遇 null 易 NPE——网关要防御。

## 2.1）场景深化：并行尚未出现前的变量纪律

在并行网关与多实例尚未加入前，团队可以用一次「沙盘推演」建立共同语言：想象采购流程在经理审批后要拆成两条并行线，如果仍然在两条分支里频繁写入同一个全局变量 `riskScore`，那么后写者会覆盖先写者，审计上也无法区分「谁在什么时刻改成了 3 分」。因此，在进入第 9、18 章之前，建议把**变量命名规范**写成一页团队共识：哪些字段**永远全局**（如 `businessKey`、`supplierId`），哪些字段**只允许局部**（如某个并行分支里的临时评分），以及网关条件**仅读取**哪些字段。**星云案例回放**：曾有项目组把「附件列表」以 `List` 变量持续 `add`，但未注意在一次驳回重提时未清空，导致审批人看到过期的旧附件引用——本质是**变量生命周期**与**业务重置策略**没有在需求里交代清楚。

## 3）项目实战（主代码片段）

### 3.1 全局与局部

```java
runtimeService.setVariable(executionId, "amount", 120000L);

runtimeService.setVariableLocal(executionId, "branchScore", 3);
```

### 3.2 任务完成时写入

```java
taskService.complete(taskId, Map.of(
    "approved", true,
    "comment", "同意，预算充足"
));
```

### 3.3 查询变量

```java
Object amount = runtimeService.getVariable(processInstanceId, "amount");
```

### 3.4 JUnit

```java
assertThat(runtimeService.getVariable(pi.getId(), "amount")).isEqualTo(120000L);
```

### 3.5 curl（REST 修改变量）

```bash
curl -s -u demo:demo -X PUT \
  "http://localhost:8080/flowable-rest/service/runtime/process-instances/{id}/variables/amount" \
  -H "Content-Type: application/json" \
  -d '{"name":"amount","type":"long","value":130000}'
```

## 4）项目总结

### 优点

- 变量是**业务与流程的契约**；良定义后报表与网关稳定。  
- 局部变量能隔离并行分支状态。

### 缺点

- 类型与版本治理成本高。  
- 大体序列化对象影响性能与存储。

### 典型使用场景

- 审批意见、金额、合同编号、风控评分。

### 注意事项

- **类型一致**与 **null 防御**写进评审清单。  
- 历史级别为 `full` 时变量历史膨胀更快。

### 反例

把整个 **Hibernate 代理对象**塞进变量，序列化到字节数组再进库——下一分支反序列化炸懒加载。**纠正**：**只存 ID** 或 **显式 DTO**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 网关不走期望分支 | 变量类型变化 / EL NPE | 打印运行时变量快照 |
| 并行下变量被覆盖 | 写全局而非 local | 用 local 或业务锁 |
| DB 急剧变大 | 大 JSON 当变量 | 外置存储 |
| 类找不到反序列化失败 | 改类名删字段 | migration 或兼容策略 |

### 给测试的一句话

**固定变量类型表**与 **边界值表**（null、0、负数、超大）；断言 **`HistoricVariableInstance`**（若历史打开）。

### 给运维一句话

监控 **变量表** 与 **历史变量** 增长速度，异常飙升优先查业务流程是否塞大对象。

## 附：延展阅读——与 JSON / Map 的纠缠

若你们使用 REST 传入 JSON，经 Jackson 反序列化为 `Map` 再塞入流程变量，务必在评审里固定：**数字是 Integer 还是 Long**、日期是字符串还是 `Instant`。星云团队在灰度曾出现「测试环境走 Integer、生产走 Long」导致网关心照不宣地与真实分支「漂移」。建议把变量契约沉淀为 **OpenAPI/JSON Schema**，并对照 BPMN 条件做一轮静态检查；这样第 8 章的排他网关才不会成为「随机分支生成器」。
