# 第 21 章：Form 引擎与表单场景——内置与外链

> 版本说明：Form 引擎为 **可选组件**；生产系统多数采用 **自研前端 + 变量 API**。[VERSION_MATRIX.md](../VERSION_MATRIX.md)

---

## 1）项目背景

星云采购审批页需展示「金额、供应商、附件」并收集「审批意见、是否加急」。产品倾向沿用 Flowable 生态的 **Form Definition**（`.form` JSON），而体验团队更想全栈掌控 UI。**本章聚焦**：`formKey` 与流程节点的关系、**表单字段如何映射为流程变量**、以及在企业里常见的折中——**外链 SPA + BFF 组装**。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：一定要用 Form 引擎吗？

**大师**：不必须。若你们已有「统一审批壳」与设计体系，直接在应用层渲染表单，调用 `complete(taskId, variables)` 即可。Form 引擎更适合 **PoC、内部工具** 或 **Flowable UI** 开箱。

**小白**：那 Modeler 里配的 form 有啥用？

**大师**：它能驱动 **低代码页面** 快速生成；但字段一复杂，你会撞上可定制性天花板。

**小胖**：移动端 H5 怎么做？

**大师**：关键是 **契约稳定**：无论表单由谁渲染，**变量名、类型、必填规则**要在 OpenAPI 中落成文档；移动端只是另一客户端。

**小白**：表单版本和流程版本谁跟谁？

**大师**：**一起发版**。`formKey` 若指向 `approval-v3.form`，而流程定义还停在 v2，会出现**字段缺失**或**多传变量**。

**小胖**：审批 outcome「同意 / 驳回」怎么建模？

**大师**：可映射布尔、枚举字符串，或专用 `outcome` 变量；要与网关条件对齐（第 8、12 章）。

**小白**：能嵌套子表吗？

**大师**：视 Form 能力与前端实现；复杂 aggregate 往往落 **业务表**，流程只存 **引用 ID**。

**小胖**：附件呢？

**大师**：表单存 **fileId**；二进制走对象存储；流程变量里别塞整文件 bytes。

**小白**：谁做 XSS 防御？

**大师**：**前后端一起做**；引擎侧二次校验不可省。

**小胖**：国际化？

**大师**：表单 label 走 i18n 资源；**变量名**保持 ASCII 稳定。

## 2.1）场景深化：BFF 聚合「业务主数据 + 表单默认值」

审批页不只是「填变量」，还要显示 **ERP 供应商评级**、**历史逾期记录**。这些不适合全部塞进 Flowable 变量。**星云**实践：BFF 调业务服务拿主数据，动态拼 **只读区块**；可编辑字段按 taskDefinitionKey 决定**白名单**。提交时 BFF **校验**后写变量——避免把 Flowable 当 CRUD。

## 3）项目实战（主代码片段）

### 3.1 UserTask 声明 formKey

```xml
<userTask id="managerApprove" name="经理审批"
          flowable:formKey="forms/procurement-approve.form"/>
```

### 3.2 部署

表单 JSON 与 BPMN **同一 deployment**（或通过 API 追加），避免跨部署找不到 resource。

### 3.3 通过 REST 提交变量（外链等价）

```bash
curl -s -u demo:demo -X POST \
  "http://localhost:8080/flowable-rest/service/runtime/tasks/{taskId}/complete" \
  -H "Content-Type: application/json" \
  -d '{"action":"complete","variables":[
        {"name":"approved","value":true,"type":"boolean"},
        {"name":"comment","value":"建议通过","type":"string"}
      ]}'
```

### 3.4 JUnit（核心断言）

```java
taskService.complete(task.getId(), Map.of(
    "approved", true,
    "comment", "OK"
));
var ok = runtimeService.getVariable(pi.getId(), "approved");
assertThat(ok).isEqualTo(true);
```

## 4）项目总结

### 优点

- **表单即配置**：变更可走评审与 Git。  
- 与 Flowable UI **零胶水**试用。

### 缺点

- 交互深度受限；  
- **版本治理**成本高。

### 典型使用场景

- PoC、内部运维表单、轻量审批。

### 注意事项

- **必填**在服务端二次验证；  
- **文件上传**与病毒扫描链路。

### 反例

前端把「是否副总审批」开关存在 localStorage——未进引擎，合规**全灭**。**纠正**：**所有决策入变量**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 表单不显示 | form 未同包部署 | 核对 deployment |
| 字段丢失 | key 改名 | 版本矩阵 |
| 类型错 | boolean 当 string | Schema |
| XSS | 未转义 | 输出编码 |

### 给测试一句话

**恶意字段名**、**超长意见**、**并发双提**。

### 给运维一句话

**静态资源缓存** bust；监控 **表单接口 413/400**。

## 附：延展阅读——从 Form 引擎迁离的迁移策略

当团队决定全面自建 UI，应保留 **变量名稳定**，前端改造可渐进；切勿一次性改所有 BPMN 字段名——配合第 26 章迁移策略评估影响面。

## 附录：星云案例深潜——移动端「字段顺序」引发的合规投诉

法务发现移动表单把「风险确认」折叠到第二屏导致用户未读即提交。之后团队在表单 schema 增加**强阻断**与**滚动到底检测**。此事提醒我们：**表单体验即合规模型**的一部分，不能丢给前端「随心所欲」。
