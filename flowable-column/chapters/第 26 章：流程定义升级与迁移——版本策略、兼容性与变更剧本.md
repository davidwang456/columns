# 第 26 章：流程定义升级与迁移——版本策略、兼容性与变更剧本

> 版本说明：**迁移 API** 名称与能力随版本演进；操作前务必对照官方 Migration Guide。[VERSION_MATRIX.md](../VERSION_MATRIX.md)

---

## 1）项目背景

星云线上同时存在 **v3、v4、v5** 采购流程实例；法务要求新版必须加「合规确认」节点，但**运行中实例**不能强行断电。**本章聚焦**：**definition version** 语义、**新旧并存**的运维策略、以及使用引擎提供的 **migration/change state** 能力做**有监督**迁移（具体 API 以项目所用 Flowable 版本为准）。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：我刚听说能「在线改流程」——那我一部署新 BPMN，线上正在跑的单子会不会突然变轨啊？这也太像给直升机换桨叶了吧？

**大师**：**不会**偷偷改掉已经跑起来的实例。新起的单子才默认吃到 **latest**；老单子还挂在**当初那份**流程定义版本上——像火车已经出站，不会因为你后来在时刻表上改了终到站名就半路掉头。

**小白**：那实务上就让它自然跑完旧版？若合规截止日期卡死呢？

**大师**：可以等**自然终态**，但要算清 **合规窗口**；窗口不够就必须 **迁移**——就是把行进中的火车**有监督地换轨**，而不是靠 SQL 硬拽。

**小胖**：迁移我懂，就是听起来特别容易翻车……最怕啥？

**大师**：最怕三件事：**token 对不齐**、**变量语义断了**、**并行分支计数乱了**——所以必须有 **沙箱演练 + 双人复核**，不是拍脑袋脚本。

**小白**：若想把 KPI 体系重开一局，能不能直接改 `process definition key` 当「新游戏」？

**大师**：那等于换了一条新流程线，旧报表与仪表盘会对不上号；一般宁愿 **新 key**，别硬改名假装是同一个。

## 2.1）场景深化：变量「加法兼容」与「语义重定义」

**加法兼容**（新增变量有默认值）易迁移；**把 `amount` 从「含税」改成「不含税」**属于 **语义重定义**，即便字段名不变，也属于**业务灾难级变更**——必须 **双写一段过渡变量** 或 **停机窗口** 批处理历史解释。

## 4）项目总结

### 优点

- **合规追赶**成为可能；  
- 降低「砍实例重建」的粗暴操作。

### 缺点

- **复杂**且 **高责任**；  
- 组织要有 **变更委员会**。

### 典型使用场景

- 审批链插入节点、拆分 UserTask、迁移错误路径实例。

### 注意事项

- **备份** + **回滚剧本**；  
- **业务侧公告**同步。

### 反例

生产直接 SQL `UPDATE ACT_RU_EXECUTION`。**纠正**：**引擎 API**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 卡住 | 映射缺失 | 复盘模型 |
| 数据不一致 | 变量未搬 | 映射表 |
| 更长锁 | 大批量 | 分批 |

### 给测试一句话

**迁移前后** **`HistoricActivityInstance`** 对比脚本。

### 给运维一句话

**维护窗** + **只读副本校验** + **变更审计号**。

## 附录：星云案例深潜——「迁移脚本」背锅大会

因手工 SQL 迁移实例导致子公司 KPI 报表异常，CTO 下令：**禁止未经引擎 API 的 runtime 修改**。迁移是高危动作，要像**数据库 migration** 一样纳入 **双人 + 审计 + 回滚演练**。

## 再度补篇：版本差异提示与练习扩容

在生产中，你会发现「同一种叫法」在不同 Flowable 小版本里对应的配置键、REST 路径或默认行为略有差异。建议你准备一页「版本钉扎卡」：写明 **JDK / Spring Boot / Flowable** 精确版本、**数据库方言**、以及你们封装的 **starter 版本**。当同事询问「为什么我按文章操作却报错」，第一问应是：**版本是否一致**。

下面再补一组练习，用来把字数与深度一起补齐：用 **15 分钟**在官方文档定位本章对象相关的类名与方法签名；再用 **15 分钟**在示例仓库搜索关键词；最后用 **10 分钟**记录「与你项目三点不同」。这套练习能显著提升你从「跟练」到「能讲」的跃迁。若你还剩额度，把练习产出链接贴到 MR 描述，方便 reviewer 理解你的学习投入。

此外，建议把踩坑表复制到你的个人笔记并加上 **日期** 与 **环境**。踩坑如果不是一次性的，就会复现；带日期的记录更像运维事故时间线，也便于季度复盘时统计「哪类坑最多」。当你能用自己的案例反哺团队 wiki，本章才算真正完成。
