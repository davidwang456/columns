# 第 18 章：多实例——会签、或签、动态集合

> 版本说明：`multiInstanceLoopCharacteristics`、`completionCondition`。

---

## 1）项目背景

星云采购会签：**三位部门负责人全同意**即通过；或 **任意一人否决**即驳回。审批人列表来自 HR 接口动态返回。**本章聚焦** **Multi-Instance**：**顺序 vs 并行**、`collection` 表达式、`elementVariable`、`completionCondition`（如 `${nrOfCompletedInstances == nrOfInstances}`）。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：多实例和并行网关？

**大师**：多实例是**同一 UserTask 复制多份**；并行网关是**不同节点**并行。

**小白**：集合哪里来？

**大师**：变量放 **用户 id 列表**；`${approverList}`。

**小胖**：一票否决？

**大师**：在 **completionCondition** 或监听里**短路**——实现要小心引擎语义。

**小白**：空集合？

**大师**：**边界**：会签无人时跳过或报错？业务决定。

## 2.1）场景深化：「一票否决」的语义落地

会签规则有三种常见落地：全通过、多数决、一票否决。**completionCondition** 只能表达部分逻辑；「一票否决」往往还要在 **TaskListener** 检测某个变量后 **触发子路径** 或 **取消未完成任务**（依语义谨慎使用）。务必在需求里写：**否决后其他人的待办是否自动消失**——这关系到用户体验与合规。

## 3）项目实战（主代码片段）

### 3.1 UserTask 多实例（示意）

```xml
<userTask id="miApprove" name="会签">
  <multiInstanceLoopCharacteristics isSequential="false"
    flowable:collection="${approvers}" flowable:elementVariable="approver">
    <completionCondition>${nrOfCompletedInstances == nrOfInstances}</completionCondition>
  </multiInstanceLoopCharacteristics>
  <extensionElements>
    <flowable:assignee>${approver}</flowable:assignee>
  </extensionElements>
</userTask>
```

### 3.2 Java

```java

```

## 4）项目总结

### 优点

- **动态人数**不需改 BPMN。  
- 与业务 **组织数据**联动。

### 缺点

- **completionCondition** 出错导致**卡死**。  
- **大量实例**性能压力。

### 典型使用场景

- 部门会签、投票、轮流签字。

### 注意事项

- **变量类型**（List vs JSON 字符串）。  
- **弃权**与**否决**业务口径。

### 反例

`approvers` 为 null 导致运行时异常。**纠正**：默认 **空列表策略** + 网关分支。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 只生成了一个任务 | collection 不是 List | 修正变量 |
| 永不完成 | 条件写错 | 打印计数变量 |
| assignee 错 | elementVariable 名错 | 对齐字段 |
| 重复批 | 重试 | 幂等 complete |

### 给测试一句话

覆盖 **1人、2人、0人、否决短路**。

### 给运维一句话

峰值 **任务数 = 人数 × 实例**；关注 **ACT_RU_TASK**。

## 附：延展阅读——与组织旋转门

当审批人列表来自 HR 的「当前负责人」而非固定名单，需定义 **中途换人**策略：已生成任务是否迁移？星云一般选择**下一回合**生效，避免「审批中改组织」引发权限争议。

## 附录：星云案例深潜——动态名单与「离职瞬移」

审批进行中途，名单里的经理离职，待办应指向继任者还是原人？业务若未定义，研发只能猜。星云落地规范：**已生成任务不自动迁移**；除非在监听里显式 `delegate`。多实例+组织变更是一组**长期演进问题**，别怕把它写进需求首屏。
