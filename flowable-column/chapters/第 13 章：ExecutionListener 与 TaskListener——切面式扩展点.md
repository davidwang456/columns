# 第 13 章：ExecutionListener 与 TaskListener——切面式扩展点

> 版本说明：`class` 或 `delegateExpression`；字段注入同服务任务。

---

## 1）项目背景

星云希望在 **流程创建**、**活动开始/结束**、**任务创建/完成** 等点统一写**审计日志**、**发 MQ 通知**、**同步 HR 兼职身份**。用 Java 在每个业务方法硬编码耦合太高；BPMN **Listener** 提供类似 AOP 的扩展点。**本章聚焦** `ExecutionListener`（take/start/end）与 `TaskListener`（create/assignment/complete/delete）的配置方式及 **Spring 事务中的顺序**。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：监听和 Delegate 区别？

**大师**：Delegate 多在 **ServiceTask** 主体；Listener 挂在 **生命周期钩子**，粒度更细。

**小白**：能改变量吗？

**大师**：可以，但要清楚**事务边界**——监听抛错可**回滚**当前步骤。

**小胖**：顺序？

**大师**：监听器**顺序**可用 `order` 字段控制（以具体元素支持为准）；团队要文档化。

**小白**：take 是？

**大师**：**顺序流被经过**时触发，常用于跟踪路由。

**小胖**：TaskListener 记录审批意见？

**大师**：`complete` 前可取表单变量写入日志；与业务表**双写**需幂等。

**小白**：Spring `@Transactional`？

**大师**：监听若在 Spring Bean 内，同一线程默认参与事务；远程调用应短小可靠。

## 2.1）场景深化：监听器的「可观测顺序」

当一个 UserTask 同时挂了 `create` 与 `assignment` 监听，又叠加全局 `ExecutionListener`，现场排障常变成「谁先谁后」的罗生门。星云要求：关键路径监听必须在设计文档列出 **order 序号**与 **失败策略**（阻断 vs 吞掉写补偿队列）。另外，监听里发 MQ 失败时，不要让整个提交流程失败——除非业务明确「没通知成功即不得审批」。

## 3）项目实战（主代码片段）

### 3.1 BPMN 片段

```xml
<userTask id="approve1" name="审批">
  <extensionElements>
    <flowable:taskListener event="create"
      delegateExpression="${taskCreateListener}"/>
    <flowable:taskListener event="complete"
      delegateExpression="${taskCompleteListener}"/>
  </extensionElements>
</userTask>
```

### 3.2 Spring Bean

```java
@Component("taskCreateListener")
public class TaskCreateListener implements TaskListener {
    @Override
    public void notify(DelegateTask delegateTask) {
        // 推送待办、写审计
    }
}
```

 Mockito **verify** 监听被调用；或查自定义审计表。

### 3.4 curl

间接：完成任务触发监听，查侧车日志或 MQ。

## 4）项目总结

### 优点

- **非侵入**主业务代码。  
- 与模型一体可视化。

### 缺点

- 过量监听让排障困难（**监听地狱**）。

### 典型使用场景

- 待办通知、指标打点、权限裁剪。

### 注意事项

- **幂等**：监听可能重试。  
- **时延**：监听里避免同步重 RPC。

### 反例

在 `create` 监听里调用外部 HTTP 支付——失败卡住任务生成。**纠正**：异步消息或后置补偿。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 监听未触发 | event 名写错/未部署 | 检查 XML |
| 事务不提交 | 监听抛错 | 明确异常策略 |
| Spring 注入空 | 非 Spring 管理 | 用 delegateExpression |
| 重复通知 | 重试 | 幂等键 |

### 给测试一句话

对每个 event 写 **负例**（监听抛错是否阻断）。

### 给运维一句话

监听失败常在 **业务日志**；给监听 **统一错误码** 与 **重试队列**。

## 附：延展阅读——与 AOP 的边界

Spring AOP 与 Flowable Listener **不是二选一**：AOP 适合纯业务服务方法；Listener 适合「与 BPMN 生命周期强绑定」的动作。混用时注意：**同类 Bean 的自调用**不吃事务代理——与第 23 章联动。

---

## 附录：星云案例深潜——监听里发 MQ 的「半成功」困局

监听向 MQ 发送「待办已创建」消息时，若 MQ 短暂不可用，工程师一度选择「吞异常」避免阻断流程——结果 BI 与待办数字长期对不上。最终方案改为：**监听失败进入本地 Outbox**，由补偿 job 重试；业务上允许「待办晚到几秒」。这说明：副作用失败要有**显式策略**，不要用「吃异常」骗自己。
