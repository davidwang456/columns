# 第 11 章：REST 集成——引擎 REST 与前后端职责切分

> 版本说明：`flowable-spring-boot-starter-rest` 或与业务网关组合。

---

## 1）项目背景

星云前端团队希望用 Vue/React 自建待办中心，不直接使用 Flowable UI；后端需要决定：**是否暴露原生引擎 REST**、哪些资源对公网开放、**BFF（Backend For Frontend）** 是否包一层聚合。**本章聚焦** Flowable REST 的典型资源：**runtime process-instances、tasks、variables** 与 **鉴权、分页、幂等** 边界。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：直接开 `/flowable-rest` 给前端不行吗？

**大师**：内网 PoC 可以；对外要**统一 OAuth2**、**租户隔离**、**数据范围**。往往业务网关做「只返回本人待办」比让前端直连引擎更安全。

**小白**：REST 和 Java API 差啥？

**大师**：语义接近，REST 多一层 **HTTP/JSON**，要少做粗大查询暴露。

**小胖**：幂等怎么保证？

**大师**：`startProcessInstance` 可用 **businessKey** + 业务幂等键去重；`complete` 可传 **client token**（若版本支持）防重放。

**小白**：大列表性能？

**大师**：**分页**、**排序字段索引**、必要时投影字段到业务库宽表。

**小胖**：文件上传呢？

**大师**：REST 层一般不管大文件；走对象存储，流程里只记 **fileRef** 变量。

**小白**：错误码？

**大师**：映射为统一业务错误体；不把 **堆栈** 给前端。

| 输入 | 输出 | 适用场景 | 不适用/慎用 | 与上下游章节关系 |
|------|------|----------|-------------|----------------|
| HTTP 客户端 | JSON 资源 | 跨语言、移动 | 极高频内网 Java 直接 API 更快 | 第 33 章安全 |

## 2.1）场景深化：BFF 为什么是常态

直接把 Flowable 原生 REST 暴露给互联网移动端，常见的坑不只是鉴权，还包括**字段爆炸**——引擎返回大量 UUID 与技术字段，前端却要拼业务主数据。星云 BFF 层会聚合 **采购单详情 + 摘要变量 + 待办 metadata**，并把「下一可办动作」翻译成产品话术（「去补充资料」而不是 `complete task xyz`）。这样 Flowable 仍是系统记录真相的后台，而用户体验不被 JSON 噪声绑架。

## 3）项目实战（主代码片段）

### 3.1 curl：分页任务

```bash
curl -s -u user:pass \
  "http://localhost:8080/flowable-rest/service/runtime/tasks?assignee=zhangsan&start=0&size=20"
```

### 3.2 curl：完成

```bash
curl -s -u user:pass -X POST \
  "http://localhost:8080/flowable-rest/service/runtime/tasks/{taskId}" \
  -H "Content-Type: application/json" \
  -d '{"action":"complete","variables":[{"name":"approved","value":true,"type":"boolean"}]}'
```

### 3.3 Java 客户端（示意）

使用 `RestTemplate`/`WebClient` 调以上路径，与嵌入式引擎二选一。

### 3.4 JUnit

使用 `@SpringBootTest(webEnvironment=RANDOM_PORT)` + `TestRestTemplate` 调本地 REST，断言 200 与 body。

## 4）项目总结

### 优点

- **解耦**前端与引擎 JVM。  
- 天然适合 **跨团队**接口契约。

### 缺点

- HTTP 序列化开销；  
- 安全面扩大。

### 典型使用场景

- SPA 待办、移动审批、异构语言服务编排。

### 注意事项

- **HTTPS**、**鉴权**、**限流**。  
- **敏感字段**脱敏。

### 反例

把管理员账号写在公开仓库的 curl 示例里传播。**纠正**：示例用占位符+环境变量。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 401/403 | Basic/OAuth 配置错 | 对齐网关 |
| 数据越权 | 未按用户过滤 | BFF 强校验 |
| 重复起实例 | 未幂等 | businessKey+唯一索引 |
| 超时 | 大查询 | 分页与索引 |

### 给测试一句话

契约测试覆盖 **分页边界**、**非法 taskId**、**并发 complete**。

### 给运维一句话

**接入 API 网关日志**与 **429 限流**；监控 **4xx/5xx** 比例。

## 附：延展阅读——OpenAPI 契约与向后兼容

一旦前后端并发迭代，建议在 BFF 暴露稳定 DTO，把引擎字段变动隔离在内部适配层；对客户端发布 **SemVer**。星云事故库里有一条典型条目：引擎升级后 `assignee` null 语义微调，导致旧版 App 「待办为空」——本质是把**技术 PATCH**当成了**应用 PATCH**。

## 附录：星云案例深潜——网关鉴权与「公共 read」事故

曾有外包团队在 BFF 层「图省事」暴露了按 `processInstanceId` 查询的调试接口但未鉴权，导致内部安全演练被刷出他人流程摘要。修复方案是：所有引擎相关 REST **必经 OAuth2**，并在服务层强制 **`tenantId` + 用户成员关系**双校验。该附录强调：**技术可用性**不等于**产品安全性**；列一份对外 API 清单并做季度渗透。
