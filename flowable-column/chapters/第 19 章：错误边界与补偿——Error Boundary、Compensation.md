# 第 19 章：错误边界与补偿——Error Boundary、Compensation

> 版本说明：`errorEvent` 需 `errorCode` 对齐；compensation handler 触发顺序与 BPMN 语义相关。

---

## 1）项目背景

星云在调用第三方支付冻结预算失败时，不应让用户任务卡在中间；需要走 **人工处理** 分支。已扣库存的步骤在订单取消时要 **冲正**。**本章聚焦** **边界错误事件**（捕获 Java 异常映射）与 **补偿边界**（Saga 式回滚意图）。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：Delegate 抛异常就够吗？

**大师**：未映射时可能**整条失败**；`errorCode` 可将**业务异常**路由到**处理问题**路径。

**小白**：补偿和事务回滚？

**大师**：传统 **DB ACID** 只覆盖单库；跨服务要用 **补偿**。BPMN compensation 是一种**显式补救**建模。

**小胖**：与消息 Saga？

**大师**：思想相近；选型看团队习惯与监控。

**小白**：补偿处理器写啥？

**大师**：**反向业务**：退款、恢复库存、发冲正请求。

## 2.1）场景深化：BpmnError 与 Java Exception 的分工

不是所有 RuntimeException 都应冒泡给客户；业务上可预期的 **支付渠道拒绝** 应映射为 **BpmnError** 进入错误边界，而 **空指针** 更应在 Delegate 内转译。星云规定：**第三方 HTTP 5xx** 默认进入 **可重试队列**，**4xx** 进入 **人工处理**——两者 errorCode 不同，便于客服分流。

## 4）项目总结

### 优点

- **失败显式**、利于运维 playbook。  
- 补偿**可视化**。

### 缺点

- 模型复杂；  
- 补偿自身可能失败——要 **重试+人工**。

### 典型使用场景

- 支付、物流、第三方审批接口。

### 注意事项

- **errorCode** 管理与枚举。  
- **补偿幂等**。

### 反例

所有异常 catch 后「只做日志」——用户任务假完成。**纠正**：映射到 **可见业务节点**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 未捕获 | code 不一致 | 对齐字符串 |
| 补偿未执行 | 未标记可补偿活动 | 查 BPMN 规范 |
| 双重扣款 | 补偿重试 | 幂等 token |
| 事务混用 | 远程已提交 DB回滚 | 分离一致性策略 |

### 给测试一句话

矩阵：**错误码 × 恢复路径**；测 **补偿失败** 二次补救。

### 给运维一句话

告警 **error boundary** 进入次数；补偿 job **与死信联动**。

## 附：延展阅读——与 Saga 编排器的比较

若组织已有成熟 Saga 编排（如自研状态机 + MQ），Flowable 承担的是**人机协同段**；纯机器段可放在外部。**避免**两套事务语义打架：谁在主、谁在从，要在架构图写明。

---

## 附录：星云案例深潜——补偿失败后的人工「止滑器」

即便补偿 Handler 也可能调用失败 API。星云在 Runbook 要求：补偿失败进入**专用人工队列**，与常规待办分区，并限制**单笔金额**触发阈值。别把补偿失败静默记录成 debug 日志——那是下一单 P1 的伏笔。
