# 第 30 章：运维排障——卡住、死信、时钟、悬挂与人工介入

> 版本说明：结合 `ACT_RU_EXECUTION`、`ACT_RU_JOB`、应用日志；**ManagementService** API 依版本。[VERSION_MATRIX.md](../VERSION_MATRIX.md)

---

## 1）项目背景

星云 on-call 高频问题：**待办不出现**、**流程卡住**、**死信暴涨**、**时间相关误触发**。需 **问题树** 与 **Runbook**。**本章给出**通用排查顺序（非穷尽）：从 **实例状态** → **等待类型** → **作业失败** → **外部依赖**。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：第一步干啥？

**大师**：拿到 **`processInstanceId` / businessKey**，确认实例 **是否运行中**、当前 **activityId**。

**小白**：卡住怎么定义？

**大师**：先定义清什么叫「卡」：**超过 SLA** 且无进展；再区分是「等人」还是「等系统」。

**小胖**：死信如何处理？

**大师**：**先看异常栈**，区分下游 500 与数据错误；再决定 **重试 / 修数据 / 人工办结**。

## 2.1）场景深化：Runbook 模板（摘录）

1. **确认影响面**：单实例 / 批量 / 某 tenant。  
2. **runtime 表**：是否停在 **receive task / message**？  
3. **job 表**：是否有 **FAILED**、**retries=0**。  
4. **应用日志**：同一时间窗 **stacktrace**。  
5. **决策**：**resume** 消息 / **move job** / **补丁变量** / **升级定义迁移**。  
6. **复盘**：是否需 **监控项** 补洞。

## 4）项目总结

### 优点

- **标准化**缩短恢复时间；  
- **知识库**可传承。

### 缺点

- 误操作风险；  
- 需要 **权限分级**。

### 典型使用场景

- 7x24、关键业务、政府级客户。

### 注意事项

- **只读优先**；  
- **双人**重大操作。

### 反例

为「赶快恢复」删 `ACT_RU_*` 行。**纠正**：**API + 备份**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 悬挂 | 等消息 | correlate |
| 双花 | 重试 | 幂等 |
| timer 乱 | 时钟 | NTP |

## 附录：星云案例深潜——「一键修复脚本」与权限失控

某运维脚本误对全租户 `move execution`，酿成重大故障。此后：**任何写操作必须显式 tenant 过滤 + dry-run**，并与 **双人授权**绑定。Runbook 也要 **版本化**，旧的「民间脚本」一律废止。

## 再度补篇：版本差异提示与练习扩容

在生产中，你会发现「同一种叫法」在不同 Flowable 小版本里对应的配置键、REST 路径或默认行为略有差异。建议你准备一页「版本钉扎卡」：写明 **JDK / Spring Boot / Flowable** 精确版本、**数据库方言**、以及你们封装的 **starter 版本**。当同事询问「为什么我按文章操作却报错」，第一问应是：**版本是否一致**。
