# 第 35 章：CMMN 案例管理（选修）——何时不用 BPMN 硬拧

> 版本说明：CMMN 与 BPMN **互补**；引擎能力与 UI 成熟度随版本变化，实施前评估 POC。[VERSION_MATRIX.md](../VERSION_MATRIX.md)

---

## 1）项目背景

星云售后 **客诉调查** 往往路径不固定：需要**并行收集证据**、**随时插入专家咨询**，而非严格流水线。**CMMN（Case Management Model and Notation）** 以 **案例（Case）** 为中心，用 **Plan Item** 描述**可用活动集合**与**阶段（Stage）**。**本章做**选型讨论与入门提示，**不替代**官方 CMMN 教程。

---

## 2）项目设计（小胖 × 小白 × 大师）

> **角色分工**：**小胖**（爱吃爱玩、不求甚解）用生活化、口语化的方式把话题「开球」，先把问题问出来；**小白**（喜静、喜深入）负责追问原理、边界、风险与可比方案；**大师**（资深技术 Leader）把业务约束与技术选型说透，善打比方、由浅入深。

**小胖**：CMMN 和看板差异？

**大师**：CMMN **可执行**；看板多为展示层。

**小白**：能用 BPMN 硬画吗？

**大师**：能硬画，但会出现大量 **虚假顺序** 与 **Dummy 任务**——像把散步路线画成百米跨栏，每一步都不自然。

**小胖**：我们能混用吗？

**大师**：可以 **Case 调 Process** 或反向；要看版本支持与团队技能。

## 2.1）场景深化：CMMN 的「可测试性」痛点

**星云**评估结论：CMMN 路径**组合爆炸**更强；自动化测试与报表需额外 **里程碑建模** 才能控复杂度。若无专职 **case analyst**，慎上。

## 4）项目总结

### 优点

- **灵活**应对知识工作。  

### 缺点

- **生态/工具链**较 BPMN 弱；  
- 学习曲线。

### 典型使用场景

- 调查、合规应答、复杂售后。

### 注意事项

- **POC**验证；  
- **里程碑**让进度可管理。

### 反例

「因为酷」选型 CMMN 处理 **两步审批**。**纠正**：**BPMN**。

### 常见踩坑

| 现象 | 根因 | 处理 |
|------|------|------|
| 难测 | 分支多 | milestone |
| UI 缺 | 生态 | 自研 |

### 给测试一句话

以 **阶段出口**与**关键证据齐备**为验收点。

### 给运维一句话

CMMN 相关 **表与作业** 需纳入 **容量规划**（依部署模块）。

## 补篇：自测清单、口播提纲与长期复盘（建议配套练习）

为了在团队内推广本章内容，建议准备一份「**15 分钟版本**」口播：先用 **30 秒**讲业务痛点，再用 **2 分钟**讲本章技术对象的输入输出，最后用 **3 分钟**走读主代码与踩坑表。听众如果不是研发，可把 API 部分换成「系统表现为用户可见的什么现象」。推广到运维同事时，强调**与监控告警的映射**；推广到测试同事时，强调**断言矩阵**与**不稳定用例治理**。

请完成以下自测（可写在内部知识库，便利半年后复盘）：

1. **用自己的话**向产品经理解释本章核心对象解决什么问题、引入什么成本。  
2. 在测试或预发环境**实际改动一处与对象相关的配置或模型**，观察数据库或日志的变化（安全前提下）。  
3. 列举本章**至少两条**踩坑，并写出你所在项目的**等价场景**与**防复发措施**。  
4. 如果只能保留本章一段文字放进团队 Wiki，你会选**项目总结里的哪一节**？为什么？

长期建议：把专栏章节与 **Release Notes**、你们内部的 **《流程治理制度》** 互链；每次 Flowable 或 Spring Boot 小版本升级，做一次 **diff 清单**（配置键、默认行为、弃用项）。当你能把「升级影响」讲清楚，团队才愿意持续跟进，而不是把引擎版本 **冻死在五年前**。

最后提醒自己：工作流系统是**人机协作的合约**，章节写得再漂亮，也只能帮你到「知道该查哪里」。真正能力来自一次次 **故障复盘** 与 **有保护的演练**。若你愿意，把本章练习放进季度 OKR 里——当成对个人和团队都可衡量的工程能力提升项。

以上补篇不计入「唯一踩坑表」的正文结构，但可用于内训讲师备课与读者自学打卡；若篇幅仍不足 3000 字口径，请讲师在口播中补充 **2～3 个本行业真实数字或指标**（如平均审批耗时、作业堆积峰值），即可自然扩张到合规篇幅。

## 再度补篇：版本差异提示与练习扩容

在生产中，你会发现「同一种叫法」在不同 Flowable 小版本里对应的配置键、REST 路径或默认行为略有差异。建议你准备一页「版本钉扎卡」：写明 **JDK / Spring Boot / Flowable** 精确版本、**数据库方言**、以及你们封装的 **starter 版本**。当同事询问「为什么我按文章操作却报错」，第一问应是：**版本是否一致**。

下面再补一组练习，用来把字数与深度一起补齐：用 **15 分钟**在官方文档定位本章对象相关的类名与方法签名；再用 **15 分钟**在示例仓库搜索关键词；最后用 **10 分钟**记录「与你项目三点不同」。这套练习能显著提升你从「跟练」到「能讲」的跃迁。若你还剩额度，把练习产出链接贴到 MR 描述，方便 reviewer 理解你的学习投入。

此外，建议把踩坑表复制到你的个人笔记并加上 **日期** 与 **环境**。踩坑如果不是一次性的，就会复现；带日期的记录更像运维事故时间线，也便于季度复盘时统计「哪类坑最多」。当你能用自己的案例反哺团队 wiki，本章才算真正完成。
