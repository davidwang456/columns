# 第40章：【高级篇综合实战】自研企业级可观测性平台

## 1. 项目背景

"某金融科技公司需要自研一套统一的可观测性平台。需求包括：统一Dashboard门户（200+研发团队）、自定义品牌（White-labeling）、告警聚合路由（对接企业级通知平台）、多云数据联邦（AWS+Azure+自建IDC）、合规审计报告自动生成。目前各团队各自维护监控工具，数据不互通，合规部门无法审计。"

这就是第40章的使命——运用全书所学知识，完成一个从零到一的"企业级可观测性平台"方案设计。这不是一个工具的使用教程，而是一个架构级的产品交付。

## 2. 项目设计

**小胖**（翻完全书40章）：大师，40章学完了。如果现在让我独立设计一个企业级可观测性平台，我能做到吗？

**大师**：平台设计的核心不是"会多少工具"，而是"怎么组合工具解决问题"。我带你梳理一下这个金融科技公司案例的平台架构。

**平台核心需求**：
1. 200+研发团队，每个团队独立的数据空间和Dashboard
2. 多云环境：AWS(美东)、Azure(新加坡)、自建IDC(北京)
3. 统一的告警体系——对接企业统一通知平台
4. SOC2合规——所有操作可审计
5. 自定义品牌——对外展示时白标
6. 自研告警聚合网关——防止告警风暴

**平台架构设计**：

```
┌─────────────────────────────────────────────────┐
│                  统一门户 (Grafana)               │
│  ┌─────────┐ ┌─────────┐ ┌───────────────────┐  │
│  │研发Org1 │ │研发Org2 │ │平台管理Org (Admin) │  │
│  └─────────┘ └─────────┘ └───────────────────┘  │
├─────────────────────────────────────────────────┤

├─────────────────────────────────────────────────┤
│  数据服务层                                      │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────────┐       │
│  │Mimir │ │ Loki │ │Tempo │ │Pyroscope │       │
│  └──────┘ └──────┘ └──────┘ └──────────┘       │
├─────────────────────────────────────────────────┤
│  采集层 (Grafana Alloy × 多云部署)               │
│  ┌────────┐ ┌────────┐ ┌────────┐              │
│  │AWS节点 │ │Azure节点│ │IDC节点 │              │
│  └────────┘ └────────┘ └────────┘              │
├─────────────────────────────────────────────────┤
│  告警聚合网关 (自研)                             │
│  Grafana Alerting → Alertmanager → 企业通知平台  │
│                  → OnCall (值班升级)              │
├─────────────────────────────────────────────────┤
│  合规审计层                                      │
│  API调用日志 → Loki → 合规Dashboard → 自动报告    │
└─────────────────────────────────────────────────┘
```

**关键设计决策**：

1. **为什么用多Organization而不是多Grafana实例？**
   - 200个团队如果每个一个Grafana实例→200个运维对象→管理成本爆炸
   - 多Organization共享同一套Grafana集群→统一升级、统一备份、统一监控
   - 但需要每个Org独立的数据空间（Dashboard/DataSource隔离）

2. **多云怎么统一？**
   - 每个云部署一套Grafana Alloy采集→数据统一写入Mimir/Loki/Tempo（部署在中心）
   - Dashboard通过`$cluster`变量切换查看不同云的数据
   - 跨云网络延迟不可避免，但通过查询缓存和Recording Rules补偿

3. **告警聚合网关怎么设计？**
   ```
   Grafana Alerting → Alertmanager (分组降噪) → 自研网关
   自研网关做的事情：
   - 去重：同一个告警2分钟内只发一次
   - 聚合：同一服务10条告警打包为1条
   - 路由：根据severity+team路由到企业通知平台（钉钉/飞书/短信/电话）
   - 审计：记录所有告警发送历史
   ```

4. **SOC2合规怎么实现？**
   - API访问日志全部发送到Loki（审计专用索引）
   - 所有配置变更记录在PostgreSQL + Git（Provisioning）
   - 自动生成每周合规报告（Python脚本 + Grafana API）
   - 保留180天操作日志

**技术映射**：平台架构 = 智慧城市（统一的水电网路+独立的小区管理），多云联邦 = 跨国物流网络（各地仓库独立运作但数据统一汇总），告警聚合网关 = 110报警中心（过滤重复报警、分发到正确片区）。

**步骤二：Organization批量创建与权限配置**

Python脚本创建200个团队Org：
```python
orgs = [
    {"name": "支付平台", "team": "payment"},
    {"name": "用户中心", "team": "user"},
    # ... 200个团队
]

for org in orgs:
    # 创建Org
    resp = requests.post(f"{GRAFANA_URL}/api/orgs", json={"name": org["name"]})
    org_id = resp.json()["orgId"]
    
    # 在Org中创建默认Folder
    requests.post(f"{GRAFANA_URL}/api/folders", 
                  json={"title": f"{org['name']}监控", "uid": f"team-{org['team']}"},
                  headers={"X-Grafana-Org-Id": str(org_id)})
    
    # 创建Team并分配权限
    team_resp = requests.post(f"{GRAFANA_URL}/api/teams",
                              json={"name": org["team"]},
                              headers={"X-Grafana-Org-Id": str(org_id)})
    
    # 配置数据源（各Org共享Mimir但标签隔离）
    requests.post(f"{GRAFANA_URL}/api/datasources",
                  json={
                      "name": "Mimir",
                      "type": "prometheus",
                      "url": "http://mimir:9009/prometheus",
                      "jsonData": {"httpHeaderName1": "X-Scope-OrgID"},
                      "secureJsonData": {"httpHeaderValue1": org["team"]}
                  },
                  headers={"X-Grafana-Org-Id": str(org_id)})
```

**步骤三：告警聚合网关开发**

```go
// alert-gateway/main.go
package main

import (
    "net/http"
    "sync"
    "time"
)

type AlertDeduplicator struct {
    mu     sync.Mutex
    cache  map[string]time.Time
    window time.Duration
}

func NewDeduplicator(window time.Duration) *AlertDeduplicator {
    return &AlertDeduplicator{
        cache:  make(map[string]time.Time),
        window: window,
    }
}

func (d *AlertDeduplicator) ShouldSend(key string) bool {
    d.mu.Lock()
    defer d.mu.Unlock()
    
    lastSent, exists := d.cache[key]
    if exists && time.Since(lastSent) < d.window {
        return false // 窗口内已发送过，去重
    }
    
    d.cache[key] = time.Now()
    return true
}

// 接收Alertmanager告警 → 去重 → 转发到企业通知平台
func handleAlert(w http.ResponseWriter, r *http.Request) {
    var alerts []Alert
    json.NewDecoder(r.Body).Decode(&alerts)
    
    dedup := NewDeduplicator(2 * time.Minute) // 2分钟窗口
    
    for _, alert := range alerts {
        key := fmt.Sprintf("%s:%s", alert.Labels["alertname"], alert.Labels["service"])
        
        if !dedup.ShouldSend(key) {
            continue // 去重
        }
        
        // 路由到不同的通知渠道
        switch alert.Labels["severity"] {
        case "critical":
            sendToPagerDuty(alert)
            sendToDingTalk(alert)
        case "warning":
            sendToDingTalk(alert)
        default:
            sendToEmail(alert)
        }
        
        // 审计日志
        logAlertToLoki(alert)
    }
}
```

## 4. 项目总结

**全书知识体系在平台中的应用**

| 章节 | 知识 | 平台应用 |
|------|------|---------|
| 1-5章 基础概念 | Dashboard/Panel/术语 | 门户设计、面板选型 |
| 6-10章 数据源 | DataSource/Prometheus/Transform | Mimir数据源配置 |
| 11-15章 管理 | 用户/告警/API/Provisioning | 200团队隔离、Provisioning |
| 17章 认证 | LDAP/OAuth | 企业AD+SSO |
| 20章 Loki | 日志聚合 | 审计日志存储 |
| 21章 Tempo | 链路追踪 | 三柱联动 |
| 22-23章 告警 | 高级告警/统一告警 | 告警聚合网关 |
| 24章 Mimir | 长期存储 | 指标中心化存储 |
| 25章 HA | 高可用 | Grafana集群 |
| 28章 SLO | 服务等级目标 | 核心服务SLO |
| 29章 OnCall | 值班管理 | 告警升级 |
| 30章 K8s Operator | 声明式管理 | Dashboard as Code |
| 32-34章 源码 | Go/React架构 | 自研网关、品牌定制 |
| 35-38章 插件开发 | Panel/App/DataSource | 审计插件、聚合网关 |
| 39章 大规模优化 | 性能调优 | 平台容量规划 |

**运维指标目标**

| 指标 | 目标值 | 度量方式 |
|------|--------|---------|
| 平台可用性 | 99.9% | Grafana自身监控 |
| Dashboard加载时间 | P99 < 3s | Grafana自身指标 |
| 告警延迟 | < 30s | 告警触发→通知时间差 |
| MTTR (平均修复时间) | < 15min | 从告警到恢复 |
| 团队覆盖率 | 100% | 接入平台团队数/总团队数 |

**思考题**
1. 该平台上线6个月后，审计团队要求新增一个功能：管理员能查看所有Org中的所有Dashboard（打破Org隔离）。这与"Org隔离"的设计原则冲突，如何平衡？
2. 如果公司收购了另一家公司，需要把对方已有的Grafana（100+ Dashboard）平滑迁移到本平台，最小化对双方业务的影响。请设计迁移方案。
