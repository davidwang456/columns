# 第11章：用户、团队与组织管理

## 1. 项目背景

"开发团队说看不到运维团队的Dashboard，运维团队又担心开发会把告警规则给改了。安全部门要求所有操作要有审计追溯……一个Grafana怎么满足这么多不同角色的需求？"

平台架构师老郑最近接到任务：为全公司200多号研发人员搭建统一的Grafana监控平台。需求听起来合理——一个平台、统一入口、各自看各自的数据。但实际落地时问题层出不穷：开发想看应用指标但不能改告警规则，运维需要管理所有Dashboard但不能干预开发团队的私有面板，测试团队只需要看看结果不需要编辑权限，安全审计还想知道"谁在什么时间看了什么Dashboard"。

Grafana的三级权限体系——Organization（组织）→ Team（团队）→ User（用户）——正是为此设计的。Organization提供完全隔离的数据空间（数据源、Dashboard、告警规则全部隔离），Team提供细粒度的Dashboard和文件夹级权限控制，User角色（Admin/Editor/Viewer）定义了用户在组织内的基本能力边界。

本章将通过一个"为电商平台三个部门搭建Grafana"的实战，让你彻底掌握这套权限体系。

## 2. 项目设计

**小胖**（困惑地盯着Grafana左下角的一排小字）：大师，我注意到Grafana界面左侧菜单最下面有个"当前组织：Main Org."，这是什么意思？难道Grafana还能像游戏一样切换"服务器"？

**大师**（点头）：这个类比挺好。Organization就像游戏的服务器——不同服务器上的角色、装备、金币完全隔离。同理，不同Organization之间的Dashboard、DataSource、User都是独立不可见的。一个用户可以在多个Organization中存在，但每个Org里的权限是独立的。

**小白**（推了推眼镜）：那什么情况下该多创建Organization，什么情况下用Team就够了？

**大师**：这是权限设计中最核心的决策。简单说——程度不同。

Organization隔离程度最高：两个Org之间不仅Dashboard看不到，连DataSource都可以完全隔离。财务部门和生产部门的Grafana互不相干——各自连各自的Prometheus，各自建各自的Dashboard。

Team隔离程度适中：同一个Organization内的不同Team，可以共享DataSource（比如大家都用同一个Prometheus），但各自有私有Dashboard。开发Team的Dashboard运维Team默认看不到——除非开发者明确授权。

**小胖**：那User角色呢？Admin、Editor、Viewer各有什么区别？

**大师**：三个默认角色的权限矩阵如下：

| 能力 | Admin | Editor | Viewer |
|------|-------|--------|--------|
| 查看Dashboard | ✅ | ✅ | ✅ |
| 编辑Dashboard | ✅ | ✅ | ❌ |
| 管理DataSource | ✅ | ❌ | ❌ |
| 管理用户/团队 | ✅ | ❌ | ❌ |
| 管理插件 | ✅ | ❌ | ❌ |
| 创建告警 | ✅ | ✅ | ❌ |
| 导出Dashboard JSON | ✅ | ✅ | ✅ |

**小白**（追问）：那如果我想让某个团队的Leader既能编辑自己团队的Dashboard，又能查看（但不能编辑）其他团队的Dashboard，这怎么实现？

**大师**：这正是Dashboard/Folder级别权限的用武之地。在Dashboard或Folder的Settings → Permissions中，你可以为特定用户或Team添加细粒度权限：
- **View**：只能看
- **Edit**：可以修改
- **Admin**：可以修改权限

比如开发Leader在开发Team的Folder里是Admin，在运维Team的Folder里只是View。这个权限覆盖（Override）了该用户的默认Org角色。

**小胖**：最近公司要求所有自动化操作不能绑定到具体个人账号（因为员工离职后账号就失效），这怎么办？

**大师**：用Service Account（服务账号）。Service Account是Grafana 8.0引入的——它是一个不绑定具体用户的、专门的API访问身份。创建Service Account时生成一个API Key，这个Key用作自动化脚本的认证。它的好处是：
1. 不依赖个人账号，开发离职不收影响
2. 可以独立设置权限，范围比Admin小
3. 有独立的Token管理，方便轮换

**小白**：说到安全，有没有办法禁止用户自己注册账号？

**大师**：在grafana.ini中设置：
```ini
[users]
allow_sign_up = false
allow_org_create = false
auto_assign_org = true
auto_assign_org_id = 1
```

这样用户只能通过管理员创建或SSO登录自动创建。再加上禁止用户自己创建Org，整个平台的管理就完全集中了。

**技术映射**：Organization = 独立服务器（完全隔离的世界），Team = 工作小组（共享资源但有私人空间），Role = 工作牌级别（决定你能进哪些门），Service Account = 机器人账号（不依赖人类员工）。

## 3. 项目实战

**环境准备**

使用之前的Docker Compose环境。

**步骤一：创建组织隔离环境**

管理员登录Grafana → Administration → Organizations → New org。

创建三个组织：

| 组织名 | 用途 |
|--------|------|
| 生产运维 | 生产环境监控，仅运维团队 |
| 应用研发 | 开发环境监控，开发团队 |
| 安全审计 | 只读审计，安全团队 |

注意：创建Org后你的当前上下文自动切换到新Org，你可以通过左下角的Org切换回到原Org。

**步骤二：创建用户和组**

Administration → Users → New user。

创建以下用户：

| 用户名 | 角色 | 所属Org |
|--------|------|---------|
| ops_admin | Admin | 生产运维 |
| ops_viewer | Viewer | 生产运维 |
| dev_leader | Editor | 应用研发 |
| dev1 | Editor | 应用研发 |
| dev2 | Viewer | 应用研发 |
| auditor | Viewer | 安全审计 |

**步骤三：在应用研发Org中创建Team**

切换到"应用研发"Org → Administration → Teams → New team。

创建两个Team：

**Team 1：订单服务组**
- 成员：dev_leader、dev1、dev2

**Team 2：用户服务组**
- 成员：dev_leader、dev1

接下来，创建两个Folder来组织Dashboard：

Dashboards → New folder → 创建：
- Folder：`订单服务监控`
- Folder：`用户服务监控`

**步骤四：设置Folder级权限**

进入"订单服务监控"Folder → Settings → Permissions → Add a permission：

| 角色/团队 | 权限 |
|-----------|------|
| 订单服务组 (Team) | Edit |
| 用户服务组 (Team) | View |
| dev2 (User) | View |

此时：
- dev_leader和dev1属于订单服务组，可以编辑该Folder下的Dashboard
- dev_leader也属于用户服务组，但在这个Folder里只有View权限
- dev2属于订单服务组但被单独覆盖为View（禁止编辑）

**步骤五：创建Service Account及API Token**

Administration → Service accounts → Add service account。

| 属性 | 值 |
|------|-----|
| Display name | `ci-bot` |
| Role | Editor |

创建后自动生成Token，保存这个Token（只显示一次）。

验证Token：
```bash
TOKEN="glsa_xxxxxxxxxxxx"
curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:3000/api/org | jq
```

如果返回组织信息，Token有效。

通过API创建Dashboard：
```bash
curl -X POST -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "dashboard": {
      "title": "CI自动创建的Dashboard",
      "panels": [],
      "schemaVersion": 38
    },
    "overwrite": false
  }' \
  http://localhost:3000/api/dashboards/db | jq
```

**步骤六：安全加固配置**

在grafana.ini中添加：
```ini
[security]
# 禁止Gravatar头像（防止邮箱泄露）
disable_gravatar = true
# Cookie安全
cookie_secure = true
cookie_samesite = strict

[users]
# 禁止用户自行注册
allow_sign_up = false
# 禁止用户自行创建组织
allow_org_create = false
# 用户登录失败5次后锁定
login_maximum_inactive_lifetime_days = 30
login_maximum_lifetime_days = 30

[auth]
# 启用登录失败锁定
login_maximum_incorrect_lock_attempts = 5
```

重启Grafana使配置生效。

**步骤七：权限验证测试**

用不同用户登录验证：
1. dev1登录 → 可以看到订单服务监控和用户服务监控两个Folder → 可以编辑订单服务的Dashboard
2. auditor登录 → 只能看到被授权的Dashboard → 所有Dashboard上都没有编辑按钮
3. 匿名用户访问 → 如果未开启匿名访问，直接跳转登录页

**常见坑点**
1. **Org切换后Dashboard消失**：每个Org有独立的Dashboard空间，切换Org后之前看到的所有Dashboard不见了是正常的。
2. **Service Account Token 权限大于所属User**：Service Account的权限继承创建时的Role设置，可以设为Editor但实际不遵守Dashboard级别的权限覆盖，需注意。
3. **删除用户后其创建的Dashboard还在**：Dashboard不属于个人，属于Org。用户被删除后Dashboard由Org Admin接管。
4. **Team多成员设置相同权限**：如果用户同时在两个Team中且两个Team对同一Dashboard有不同权限，最终取二者中权限高的（编辑>查看）。

## 4. 项目总结

**权限体系总览**

| 层级 | 隔离范围 | 适用场景 |
|------|---------|---------|
| Organization | 完全隔离（数据源、Dashboard、用户） | 不同部门、不同产品线 |
| Team | 部分隔离（Dashboard共享可控） | 同一部门的不同小组 |
| Role (Admin/Editor/Viewer) | 功能权限 | 确定用户的默认能力范围 |
| Folder/Dashboard 权限 | 细粒度覆盖 | 特殊权限需求 |

**优点**
| 特性 | 说明 |
|------|------|
| 多租户隔离 | Organization提供完全的数据空间隔离 |
| 细粒度权限 | Team+Foler+Dashboard三级细粒度权限 |
| Service Account | 不依赖个人账号的API认证 |
| 无额外依赖 | 权限全部内置在Grafana中，不需要外部系统 |

**缺点**
| 特性 | 说明 |
|------|------|
| 无跨Org共享 | Organization之间完全隔离，无法共享DataSource或Dashboard |
| 无审批流程 | 权限变更即时生效，无审批机制 |
| RBAC有限 | OSS版不支持自定义角色 |
| 审计不足 | 默认无详细的操作审计日志 |

**适用场景**
1. 多部门统一平台：不同部门各自Org，互不干扰
2. 多环境管理：开发/测试/生产各自Org或Team
3. 对外服务：为客户创建独立Org展示其SaaS监控数据
4. 内部工具化：CI/CD系统使用Service Account自动管理Dashboard

**注意事项**
1. 用户可属于多个Org，但同一时刻只在一个Org中操作
2. Service Account在每个Org中可以是不同角色
3. Org一旦删除，其中的Dashboard/DataSource/告警全部永久删除，无法恢复
4. Grafana OSS版不支持自定义角色（需Enterprise版）

**常见踩坑经验**
1. **权限不够但不报明确错误**：Editor角色看不到Administration菜单，会误以为功能不存在；实际是权限限制。
2. **Service Account Token泄露**：Token在创建后只显示一次，忘记保存只能删了重建。
3. **邀请用户流程中断**：如果邮件服务未配置，邀请用户功能会失败。解决：用API直接创建用户。
