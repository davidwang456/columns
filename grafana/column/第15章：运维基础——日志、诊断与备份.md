# 第15章：运维基础——日志、诊断与备份

## 1. 项目背景

"Grafana突然打不开了，白屏。查了半小时也没找到原因——因为根本不知道Grafana自己的日志在哪里。"

这是运维工程师小杨的真实经历。他在凌晨接到告警：所有依赖Grafana的监控大屏全部白屏，值班人员无法看到系统状态。小杨SSH到Grafana服务器，尝试了`systemctl status grafana-server`（服务显示running），` docker logs grafana`（日志全是INFO，没有ERROR），`curl localhost:3000`（返回200 OK）。但打开浏览器就是白屏——前端JS加载失败。

最终定位：运维昨天升级了Nginx的CSP（内容安全策略）配置，阻止了Grafana前端的某些脚本来源，导致React渲染失败。但Grafana自身的错误日志里没有任何记录，因为这是一个前端问题。

Grafana作为基础设施的核心组件，其自身的可运维性往往被忽视。数据库选型（SQLite vs PostgreSQL）、备份策略、日志分析、健康检查、故障诊断——这些"元运维"能力是保障监控系统本身不成为盲点的前提。本章将深入Grafana运维的四个核心战场：日志系统配置、数据库管理、备份恢复策略、以及常见故障的SOP（标准处理流程）。

## 2. 项目设计

**小胖**（焦急地拍键盘）：大师大师！Grafana崩了！所有Dashboard都显示"Loading..."，但是Grafana服务还在运行，数据库也没挂，Prometheus也正常。这种情况怎么办？

**大师**（不急不慢地放下茶杯）：你这个属于经典的Grafana运维故障。按照我的四步排查法：日志→数据库→网络→前端。

**小白**（立刻打开笔记本）：具体每一步要看什么？

**大师**：第一步——看Grafana自己的日志。很多人不知道Grafana的日志分两类：Server日志和HTTP访问日志。

Server日志记录了Grafana后端的运行状态——启动过程、插件加载、数据库连接、查询执行情况。默认输出到`stdout`，如果用的是二进制部署可能输出到`/var/log/grafana/grafana.log`。

HTTP访问日志记录了每一次API请求——谁在什么时候访问了哪个端点，响应状态码是多少。默认不开启，需要在grafana.ini中设置`router_logging = true`。

**小胖**：我看了`docker logs grafana`，就一行`INFO[05-15] HTTP Server Listen`，其他什么都没有。

**大师**：其他什么都没有——这意味着后端启动是正常的，问题在前端。你可以用浏览器开发者工具的Network和Console标签查看。具体看：JS/CSS资源是否加载成功？API请求是否都返回200？如果没有报错，打开Console查看有没有JavaScript异常。

我之前遇到过一个和你类似的白屏问题，根因是CDN挂了——Grafana前端依赖的几个静态资源是从CDN加载的，CDN不可达导致React渲染失败。后来我们把所有静态资源都走本地加载解决了。

**小白**（若有所思）：那数据库的运维呢？我知道Grafana默认用SQLite，但生产环境是不是该换成PostgreSQL？

**大师**：SQLite到PostgreSQL是Grafana运维中最重要的升级决策之一。

SQLite的优点：零配置、单文件、备份简单（cp一个文件）。缺点：不支持并发写入，多Grafana实例不共用，WAL日志可能膨胀。

PostgreSQL的优点：支持多Grafana实例共享、高并发、连接池、读写分离。缺点：需要独立部署和维护、配置比SQLite复杂。

如果你团队少于10人，Grafana只部署一个实例，SQLite足够了。如果团队超过10人或者需要HA（高可用），必须用外部数据库。

**小胖**：那怎么从SQLite迁移到PostgreSQL？

**大师**：Grafana提供了内置的数据迁移工具。流程是：

1. 先部署好PostgreSQL
2. 在PostgreSQL中创建空数据库`grafana`
3. 修改grafana.ini指向PostgreSQL
4. 重启Grafana，它会自动建表
5. 用grafana-cli工具迁移数据（或者用社区工具）

```bash
# 使用Grafana内置迁移
grafana-cli db migrate \
  --source sqlite3:///var/lib/grafana/grafana.db \
  --target postgres://grafana:password@postgres:5432/grafana
```

**小白**：备份策略呢？我发现很多人根本不做Grafana备份，出了问题就傻眼。

**大师**：这是最容易被忽视的运维基础。Grafana备份分三块：

**数据库备份**：Dashboard定义、DataSource配置、用户信息、告警规则全部在数据库里。最简单的备份是数据库dump：
```bash
# SQLite
cp /var/lib/grafana/grafana.db /backup/grafana-$(date +%Y%m%d).db

# PostgreSQL
pg_dump -h postgres -U grafana grafana > /backup/grafana-$(date +%Y%m%d).sql
```

**Dashboard JSON备份**：即使数据库备份了，再单独备份一份JSON也是好习惯。用API批量导出（见第13章）。

**配置文件备份**：grafana.ini、provisioning目录、插件目录。这些在恢复时需要。

理想的备份方案是：数据库每日自动备份（保留最近7天）+ Dashboard通过Provisioning存在Git仓库（天然备份）+ grafana.ini和provisioning配置在Git仓库。

**小胖**（举手）：那经常遇到的故障有哪些？

**大师**：我总结Grafana最常见的5个故障及排查方案：

**故障1：登录失败/密码不记得了**
```bash
# 重置管理员密码
grafana-cli admin reset-admin-password newpassword

# 或通过API（如果有Admin Token）
curl -X PUT -H "Authorization: Bearer $TOKEN" \
  -d '{"password":"newpassword"}' \
  http://localhost:3000/api/admin/users/2/password
```

**故障2：数据源连接超时**
```bash
# 检查Grafana到数据源的网络连通性
docker exec grafana curl -v http://prometheus:9090/api/v1/status/buildinfo

# 检查grafana.ini中的超时设置
# [dataproxy] timeout = 30
```

**故障3：Dashboard加载缓慢**
查看Grafana自身监控指标：
```bash
curl http://localhost:3000/metrics | grep grafana_db_request_duration_seconds
```
如果数据库查询耗时>1秒，检查数据库连接数和索引。

**故障4：内存泄漏**
Grafana是Go程序，正常内存占用100-500MB。如果持续增长不回收，可能是插件或查询结果缓存导致的。

**故障5：前端白屏**
浏览器F12 → Network → 寻找红字404/500 → Console → 查看JS错误堆栈。90%的白屏问题是静态资源加载失败或JS异常。

**技术映射**：Grafana日志 = 行车记录仪（事后回溯的凭据），数据库备份 = 游戏存档（没了就得重头开始），健康检查 = 体检报告（表面看着没事，指标可能已异常），SOP = 应急预案（出了事知道第一步该做什么）。

## 3. 项目实战

**环境准备**

继续使用之前的Docker Compose环境。

**步骤一：Grafana日志配置与诊断**

在docker-compose.yml中配置Grafana日志级别：

```yaml
grafana:
  environment:
    - GF_LOG_LEVEL=debug        # 可选: debug, info, warn, error, critical
    - GF_LOG_MODE=console file  # 同时输出到控制台和文件
    - GF_LOG_CONSOLE_FORMAT=json  # JSON格式方便日志分析
    # 启用访问日志
    - GF_ROUTER_LOGGING=true
```

重启后查看日志：
```bash
# 查看实时日志
docker logs -f grafana

# 按级别过滤
docker logs grafana 2>&1 | grep -E "lvl=(eror|warn)"
```

GitHub上的日志分析命令：
```bash
# 统计错误类型分布
docker logs grafana 2>&1 | grep "lvl=eror" | awk -F'msg=' '{print $2}' | sort | uniq -c | sort -rn

# 查找慢查询
docker logs grafana 2>&1 | grep -E "duration=[0-9]{3,}ms"
```

默认情况下，Grafana日志包含：
- 启动信息（版本、数据库类型、插件加载）
- 数据库查询（级别为debug时）
- 数据源代理请求
- 告警评估结果
- 认证/授权事件

**步骤二：Grafana自身监控**

Grafana暴露了`/metrics`端点，可以用Prometheus抓取Grafana自身的指标：

```yaml
# 在prometheus.yml中添加
scrape_configs:
  - job_name: 'grafana'
    static_configs:
      - targets: ['grafana:3000']
```

关键指标及其含义：

| 指标 | 含义 | 告警建议 |
|------|------|---------|
| `grafana_http_request_duration_seconds` | API请求延迟 | P99 > 2s 需关注 |
| `grafana_db_request_duration_seconds` | 数据库查询延迟 | P99 > 500ms 需关注 |
| `grafana_stat_active_users` | 活跃用户数 | 异常突增可能被攻击 |
| `grafana_stat_total_dashboards` | 总Dashboard数 | 监控增长趋势 |
| `grafana_alerting_active_alerts` | 活跃告警数 | 平时约0-3条为正常 |
| `grafana_api_user_sign_up_total` | 用户注册总数 | 异常增长需调查 |
| `process_resident_memory_bytes` | 内存占用 | >1GB需关注 |
| `go_goroutines` | Go协程数量 | >1000需关注 |

创建Grafana自己的健康Dashboard：
```promql
# Grafana自身API请求QPS
sum(rate(grafana_http_request_duration_seconds_count[5m])) by (handler)

# Grafana数据库查询P99
histogram_quantile(0.99, 
  sum(rate(grafana_db_request_duration_seconds_bucket[5m])) by (le))

# Grafana内存使用
process_resident_memory_bytes / 1024 / 1024
```

**步骤三：从SQLite迁移到PostgreSQL**

```bash
# 1. 确认PostgreSQL已运行
docker exec grafana-db pg_isready -U grafana

# 2. 备份SQLite数据库
docker cp grafana:/var/lib/grafana/grafana.db ./backup/grafana-before-migration.db

# 3. 修改docker-compose.yml中的Grafana配置
# environment:
#   - GF_DATABASE_TYPE=postgres
#   - GF_DATABASE_HOST=postgres:5432
#   - GF_DATABASE_NAME=grafana
#   - GF_DATABASE_USER=grafana
#   - GF_DATABASE_PASSWORD=grafana_secret

# 4. 重启Grafana（会自动在PostgreSQL中建表）
docker compose restart grafana

# 5. 验证迁移（检查Dashboard和DataSource是否还在）
docker logs grafana | grep "migration"
```

注意：这样切换数据库后，Grafana会自动建新表但没有迁移旧数据。如需迁移数据，用额外的迁移工具。

**步骤四：备份脚本**

创建 `backup-grafana.sh`：

```bash
#!/bin/bash
BACKUP_DIR="/backup/grafana"
DATE=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=7

mkdir -p "$BACKUP_DIR"

# 1. PostgreSQL数据库备份
echo "[1/4] 备份数据库..."
docker exec grafana-db pg_dump -U grafana grafana > "$BACKUP_DIR/grafana-db-$DATE.sql"
gzip "$BACKUP_DIR/grafana-db-$DATE.sql"

# 2. Dashboard JSON备份
echo "[2/4] 备份Dashboard JSON..."
curl -s -H "Authorization: Bearer $GRAFANA_TOKEN" \
  "http://localhost:3000/api/search?type=dash-db&limit=5000" | \
  jq -r '.[].uid' | while read uid; do
    curl -s -H "Authorization: Bearer $GRAFANA_TOKEN" \
      "http://localhost:3000/api/dashboards/uid/$uid" | \
      jq '.dashboard' > "$BACKUP_DIR/dashboards/${uid}.json"
done

# 3. 配置文件备份
echo "[3/4] 备份配置文件..."
tar czf "$BACKUP_DIR/grafana-config-$DATE.tar.gz" \
  /etc/grafana/grafana.ini \
  /etc/grafana/provisioning/

# 4. 清理过期备份
echo "[4/4] 清理过期备份..."
find "$BACKUP_DIR" -type f -mtime +$RETENTION_DAYS -delete

echo "备份完成: $BACKUP_DIR"
```

添加到crontab每日执行：
```bash
# crontab -e
0 2 * * * /opt/scripts/backup-grafana.sh >> /var/log/grafana-backup.log 2>&1
```

**步骤五：恢复演练**

定期演练恢复流程（至少每季度一次）：

```bash
# 1. 停止Grafana
docker compose stop grafana

# 2. 恢复PostgreSQL数据库
docker exec -i grafana-db psql -U grafana grafana < /backup/grafana-db-20250115.sql

# 3. 恢复配置文件
tar xzf /backup/grafana-config-20250115.tar.gz -C /

# 4. 启动Grafana
docker compose start grafana

# 5. 验证
curl -s http://localhost:3000/api/health | jq
curl -s -H "Authorization: Bearer $TOKEN" \
  http://localhost:3000/api/search?type=dash-db | jq 'length'
```

**步骤六：常见故障SOP**

创建排查流程图（运维手册）：

```markdown
## Grafana故障排查SOP

### 1. Grafana无法访问（HTTP 502/503）
→ 检查Grafana进程是否运行: `docker ps | grep grafana`
→ 检查端口是否监听: `netstat -tlnp | grep 3000`
→ 检查数据库连接: `docker logs grafana | grep database`
→ 检查内存是否耗尽: `docker stats grafana`

### 2. 所有Dashboard加载缓慢（>5s）
→ 查看Grafana日志中的慢查询
→ 检查数据库是否有死锁: `SELECT * FROM pg_stat_activity WHERE state = 'active'`
→ 检查Prometheus是否正常: `curl prometheus:9090/api/v1/status/buildinfo`
→ 临时方案: 重启Grafana释放连接池

### 3. 用户无法登录
→ 检查LDAP/OAuth服务是否正常
→ 检查grafana.ini中的[auth]配置
→ 查看Grafana日志中的认证失败记录
→ 临时方案: 使用admin账号本地登录，关闭SSO

### 4. 磁盘空间不足
→ 检查Grafana日志文件大小: `du -sh /var/log/grafana/`
→ 清理旧日志: `find /var/log/grafana/ -name '*.log.*' -mtime +7 -delete`
→ 检查数据库大小: `SELECT pg_size_pretty(pg_database_size('grafana'))`
→ 清理历史Dashboard版本: 调整`versions_to_keep`参数
```

**常见坑点**
1. **日志级别debug导致磁盘爆满**：debug级别日志量巨大（每条数据库查询都记录），生产环境务必用info或warn。
2. **SQLite文件在NFS上**：SQLite不支持网络文件系统（NFS），会导致数据库损坏。
3. **PostgreSQL版本不兼容**：Grafana 11.x要求PostgreSQL 12+，10.x版本无法启动。
4. **备份脚本缺少错误处理**：备份失败但cron没有报错，导致几个月后发现备份都是空的。

## 4. 项目总结

**运维成熟度模型**

| 级别 | 描述 | 关键实践 |
|------|------|---------|
| L1: 初始级 | 无备份、无监控、凭记忆恢复 | 尽快升级到L2 |
| L2: 可管理级 | 有备份、有基本监控、但人工操作多 | 数据库每日备份、Grafana自身指标监控 |
| L3: 规范化级 | 自动化备份、有SOP、定期演练 | Provisioning + GitOps、恢复演练 |
| L4: 量化级 | SLO驱动、自动化恢复、告警完备 | 恢复时间<30min、自动修复常见故障 |
| L5: 优化级 | 混沌工程、自愈系统 | 故障自动转移、零人工干预 |

**数据库选型决策**

| 场景 | 推荐数据库 | 理由 |
|------|-----------|------|
| 个人学习/开发 | SQLite | 零配置 |
| 小团队(<10人) | SQLite + 定期备份 | 够用 |
| 中团队(10-50人) | PostgreSQL | 并发支持 |
| 大团队(>50人) | PostgreSQL + 读写分离 | 性能要求 |
| 多实例HA | PostgreSQL + Redis | Session共享 |

**适用场景**
1. 生产环境上线前：完成日志配置、数据库选型、备份策略三件套
2. 故障排查：按照SOP快速定位问题
3. 容量规划：通过Grafana自身指标预测资源需求
4. 审计合规：日志保留和审计追踪

**注意事项**
1. 切勿在生产环境使用SQLite + NFS的组合
2. PostgreSQL连接池默认100，高并发时可能需要调大
3. Grafana自身不产生业务数据，备份主要价值在于恢复配置
4. 恢复演练一定要在独立的恢复环境中做，不要在生产环境演练

**常见踩坑经验**
1. **日志中充满TLS handshake error**：客户端使用HTTP访问但Grafana配置了HTTPS，或者TLS证书过期/不匹配。
2. **Dashboard版本过多导致数据库膨胀**：每个Dashboard的每次保存都会新生成一个版本快照。解决：在grafana.ini中设置`versions_to_keep=20`限制保留的版本数。
3. **插件不兼容导致Grafana启动失败**：升级Grafana版本后，某些旧插件可能不兼容。解决：在grafana.ini中设置`plugin_admin_enabled=true`禁用有问题的插件。

**思考题**
1. 如果需要在5分钟内部署一套全新的Grafana实例并恢复所有Dashboard（假设数据全部丢失只剩昨天的备份），请设计这个自动化脚本。
2. Grafana自身监控显示数据库查询P99延迟突然从50ms飙升至2000ms，但Grafana服务没有报错。可能的原因有哪些？排查步骤是什么？
