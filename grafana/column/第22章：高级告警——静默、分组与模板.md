# 第22章：高级告警——静默、分组与模板

## 1. 项目背景

"上周末数据库做例行维护，2小时里收到了247条告警通知，钉钉群直接被刷成'消息瀑布'。运维同事把群消息设成了免打扰——这等于整个告警体系废了。"

这并非个例。大多数团队在搭建告警体系时，重点放在"告警规则怎么写"，却忽视了"告警怎么管理、怎么降噪、怎么表达"。结果是：告警规则越来越多（50+条属于正常），通知越来越频繁，但真正有效（被处理）的告警比例越来越低。这就是"告警疲劳"——当告警像狼来了一样频繁，真出问题时反而没人注意了。

Grafana Alerting提供的静默、分组、通知策略树和模板功能，正是对抗告警疲劳的武器。本章将深入告警管理的高级话题——不仅学怎么配置，更要学告警设计的工程方法论。

## 2. 项目设计

**小胖**（揉着被消息震麻的手腕）：大师，昨天晚上整整收到183条告警。我一条条看过去，发现实际上只有2个是真问题，其他181条要么是维护窗口的正常波动，要么是可以合并的同类告警。

**大师**：你这个情况我总结为"告警管理的三个失效"：

1. **Silence失效**：维护窗口的告警没被静默
2. **Grouping失效**：同类告警没有被合并
3. **Notification Policy失效**：不重要的告警走了高优先级的通知渠道

我们一个一个来解决。

**小白**（翻开笔记）：先说Silence。静默除了手动在UI上点"创建静默"，还有别的方式吗？

**大师**：Grafana支持三种静默方式：

**定时静默**（Scheduled Silence）：在UI中创建。比如"每周六凌晨2:00-4:00全量备份，静默`service=database`的告警"。可以设置重复周期。

**API静默**：在CI/CD流程中自动创建。发布前自动创建静默，发布完成后自动删除。
```bash
curl -X POST -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "matchers": [{"name": "service", "value": "order-svc"}],
    "startsAt": "2025-01-15T22:00:00Z",
    "endsAt": "2025-01-15T23:00:00Z",
    "comment": "版本发布v3.4.0"
  }' \
  http://localhost:3000/api/alertmanager/grafana/api/v2/silences
```

**日历静默**（配合OnCall）：在Grafana OnCall中通过值班日历自动静默非值班时间的低优先级告警。

**小胖**：那Group by呢？我设置了`group by alertname`，但好像没效果？

**大师**：Grouping的效果取决于两个参数：Group by的Label选择，以及Group wait/interval。

Group by的Label决定了哪些告警会被"打包"在一起。比如你按`alertname`分组，所有"CPU高负载"告警都会被合并成一条通知（里面包含所有触发的实例）。

但你按`instance`分组，每个实例的告警就是独立的一条通知——等于没分组。

最佳实践是多层分组：
```yaml
Group by:
  - alertname     # 第一层：按告警名
  - severity      # 第二层：按严重级别
  - service       # 第三层：按服务
Group wait: 30s   # 收到第一条告警后等30秒再发通知（等同类告警到齐）
Group interval: 5m # 同组告警有新触发时，每5分钟发一次更新
Repeat interval: 4h # 告警持续未恢复的重复提醒间隔
```

**小白**：那Notification Policy Tree怎么设计？我看默认就一个"Default policy"。

**大师**：通知策略树是告警路由的核心。设计原则是"从上到下匹配，命中停止"。我给你看一个生产级的策略树设计：

```
根策略 (Default)
├── 不匹配任何子策略的告警走这里 → email-default
│
├── severity = critical
│   └── → PagerDuty + 钉钉群 (立即通知)
│
├── severity = warning
│   ├── service = infrastructure  
│   │   └── → 运维钉钉群 (Group by alertname, 30s等待)
│   │
│   ├── service = application
│   │   └── → 开发钉钉群 + 邮件 (Group by alertname+service, 1m等待)
│   │
│   └── (其他warning) → 邮件
│
└── severity = info
    └── → 只发邮件 (不打扰IM渠道)
```

关键设计原则：
1. 根据severity分层——不同严重级别走不同渠道
2. 根据service分流——基础设施告警通知运维，应用告警通知开发
3. 低优先级走异步渠道（邮件），高优先级走同步渠道（PagerDuty/电话）

**小胖**：通知模板呢？我收到的告警消息信息太少，连个Dashboard链接都没有。

**大师**：好的告警通知模板应该包含：
1. **告警摘要**：什么告警、当前值、阈值
2. **上下文**：哪个服务、哪个环境、什么时间开始
3. **可操作链接**：直达Dashboard的URL（带时间范围和变量值）
4. **静默链接**：如果确认是误报，一键创建静默

Grafana使用Go模板语法。一个完整的Slack通知模板：

```
{{ define "slack.title" }}
[{{ .Status | toUpper }}{{ if eq .Status "firing" }}:{{ .Alerts.Firing | len }}{{ end }}] 
{{ .CommonLabels.alertname }}
{{ end }}

{{ define "slack.text" }}
{{ range .Alerts.Firing }}
*告警名称*: {{ .Labels.alertname }}
*严重级别*: {{ .Labels.severity }}
*当前值*: {{ .Annotations.value }}
*阈值*: {{ .Annotations.threshold }}
*服务*: {{ .Labels.service }}
*环境*: {{ .Labels.env }}
*触发时间*: {{ .StartsAt.Format "2006-01-02 15:04:05" }}
📊 [查看Dashboard]({{ .GeneratorURL }})
🔕 [静默2小时]({{ .SilenceURL }})
---
{{ end }}
{{ end }}
```

**小胖**：等等，`.GeneratorURL`、`.SilenceURL`这些变量是Grafana内置的吗？

**大师**：是的，Grafana Alerting模板支持的变量包括：

| 变量 | 含义 |
|------|------|
| `.Status` | firing 或 resolved |
| `.Alerts.Firing` | 当前触发的告警列表 |
| `.CommonLabels` | 所有告警共享的Label |
| `.Labels.xxx` | 单个告警的特定Label |
| `.Annotations.xxx` | 告警规则的Annotation |
| `.StartsAt` | 告警触发时间 |
| `.EndsAt` | 告警恢复时间 |
| `.GeneratorURL` | 带参数的Dashboard链接 |
| `.DashboardURL` | Dashboard链接 |
| `.SilenceURL` | 创建静默的链接 |

**技术映射**：Silence = 请假条（这个时间段别找我），Group by = 打包（同类快递合并一个箱子），Notification Policy Tree = 分拣流水线（不同目的地走不同传送带），模板 = 通知格式化（把机器数据转成人话）。

## 3. 项目实战

**环境准备**

使用之前的Docker Compose环境，Grafana Alerting已配置。

**步骤一：设计Notify Policy Tree**

Alerting → Notification policies → 编辑Default policy → Contact point: email-ops。

添加子策略：

```
Policy 1: severity = critical
  Contact point: webhook-pagerduty
  Group by: alertname, severity
  Group wait: 0s (立即发送)
  Group interval: 5m
  Repeat interval: 1h

Policy 2: severity = warning, service = infrastructure
  Contact point: email-ops
  Group by: alertname
  Group wait: 30s
  Group interval: 10m
  Repeat interval: 4h
  Continue matching siblings: false

Policy 3: severity = warning, service = application
  Contact point: webhook-dingtalk
  Group by: alertname, service
  Group wait: 1m
  Group interval: 5m
  Repeat interval: 2h
  Continue matching siblings: false

Policy 4: severity = info
  Contact point: email-ops
  Group by: alertname
  Group wait: 5m
  Group interval: 30m
  Repeat interval: 24h
  Continue matching siblings: false
```

验证策略树：
```bash
# 查看当前策略树
curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:3000/api/v1/provisioning/policies | jq
```

**步骤二：创建通知模板**

Alerting → Contact points → 选择一个Contact Point → Notification settings → 点击"Edit"图标。

Slack/Microsoft Teams模板：
```
{{ define "msteams.message" }}
### {{ if eq .Status "firing" }}🚨 Firing{{ else }}✅ Resolved{{ end }}: {{ .CommonLabels.alertname }}

**严重级别**: {{ .CommonLabels.severity }}
**影响服务**: {{ .CommonLabels.service }}
**环境**: {{ .CommonLabels.env }}
**当前时间**: {{ now.Format "2006-01-02 15:04:05" }}

{{ if eq .Status "firing" }}
| 实例 | 当前值 | 阈值 | 触发时间 |
|------|--------|------|---------|
{{ range .Alerts.Firing }}| {{ .Labels.instance }} | {{ .Annotations.value }} | {{ .Annotations.threshold }} | {{ .StartsAt.Format "15:04:05" }} |
{{ end }}
{{ end }}

[📊 查看Dashboard]({{ .CommonAnnotations.dashboard_url }})
{{ end }}
```

Email模板（HTML）：
```html
{{ define "email.subject" }}
[{{ .Status | toUpper }}] {{ .CommonLabels.alertname }} - {{ .CommonLabels.service }}
{{ end }}

{{ define "email.html" }}
<h2>{{ if eq .Status "firing" }}🔴 Firing{{ else }}🟢 Resolved{{ end }}</h2>
<table border="1" cellpadding="5">
  <tr><th>Alert</th><td>{{ .CommonLabels.alertname }}</td></tr>
  <tr><th>Severity</th><td>{{ .CommonLabels.severity }}</td></tr>
  <tr><th>Service</th><td>{{ .CommonLabels.service }}</td></tr>
</table>
<p><a href="{{ .CommonAnnotations.dashboard_url }}">View Dashboard</a></p>
{{ end }}
```

**步骤三：创建定时静默**

```bash
# 创建每周六凌晨2:00-4:00的定时静默
curl -X POST -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "matchers": [
      {"name": "service", "value": "infrastructure", "isRegex": false}
    ],
    "startsAt": "2025-01-18T18:00:00Z",
    "endsAt": "2025-01-18T20:00:00Z",
    "comment": "每周维护窗口 - 数据库备份",
    "id": "weekly-maintenance-silence"
  }' \
  http://localhost:3000/api/alertmanager/grafana/api/v2/silences
```

如果需要重复周期，需要通过API或使用`cron`定期创建新的Silence。

**步骤四：CI/CD集成——发布自动静默**

在部署脚本中集成：

```bash
#!/bin/bash
# deploy.sh
SERVICE=$1
GRAFANA_URL="http://localhost:3000"
TOKEN="glsa_xxx"

# 1. 创建静默
SILENCE_ID=$(curl -s -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{
    \"matchers\": [{\"name\": \"service\", \"value\": \"$SERVICE\"}],
    \"startsAt\": \"$(date -u +%Y-%m-%dT%H:%M:%SZ)\",
    \"endsAt\": \"$(date -u -d '+30 minutes' +%Y-%m-%dT%H:%M:%SZ)\",
    \"comment\": \"部署${SERVICE}\"
  }" \
  $GRAFANA_URL/api/alertmanager/grafana/api/v2/silences | jq -r '.silenceID')

echo "创建静默: $SILENCE_ID"

# 2. 执行部署
kubectl rollout restart deployment/$SERVICE
kubectl rollout status deployment/$SERVICE

# 3. 部署成功后在静默备注中标记
# 注意：Grafana API不直接支持修改静默，需要删除后重建
```

**步骤五：告警规则规范设计**

为团队建立告警规则模板规范：

```yaml
# 告警规则标准模板
groups:
  - name: application_health
    rules:
      - alert: ServiceErrorRateHigh
        expr: |
          sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)
          / sum(rate(http_requests_total[5m])) by (service) * 100 > 1
        for: 3m
        labels:
          severity: warning
          team: dev
          service: "{{ $labels.service }}"
          component: application
        annotations:
          summary: "{{ $labels.service }} 错误率 {{ $value | humanize }}%"
          description: "服务 {{ $labels.service }} 错误率超过1%阈值"
          runbook_url: "https://wiki.example.com/runbooks/service-error-rate"
          dashboard_url: "https://grafana.example.com/d/service-dashboard?var-service={{ $labels.service }}"
```

规范要点：
- `for` 至少2倍评估间隔
- Label必须有`severity`、`team`、`service`
- Annotations必须有`summary`、`description`、`runbook_url`

**常见坑点**
1. **Group wait设为0导致不及时**：虽然第一条告警立即发送，但会错过同一组的其他告警。建议至少30s。
2. **Repeat interval过短**：设为5m意味着每5分钟重复提醒一次未恢复的告警——这是"通知炸弹"。
3. **Silence的Matchers配置太宽**：`service=~".*"`会静默所有告警，生产事故。
4. **模板中的字段不存在导致渲染空白**：如果告警规则的Annotation缺少某字段，模板引用时会渲染为空字符串。

## 4. 项目总结

**告警管理成熟度**

| 级别 | 告警管理实践 |
|------|------------|
| L1: 裸告警 | 有告警规则，无分组无降噪 |
| L2: 基础管理 | 有静默和通知策略，但配置随意 |
| L3: 规范化 | 统一模板、标准化Label体系、CI/CD静默 |
| L4: 智能化 | 基于历史数据的动态阈值、自适应告警窗口 |
| L5: 可观测 | 告警有效性追踪、MTTR持续优化 |

**优势**
| 特性 | 说明 |
|------|------|
| 策略树路由 | 告警自动分流到正确的团队和渠道 |
| 分组降噪 | 同类告警合并，避免通知风暴 |
| 定时静默 | 维护窗口自动静默，减少误报 |
| 通知模板 | 自定义消息格式，带Dashboard链接 |
| CI/CD集成 | 发布时自动静默和恢复 |

**注意事项**
1. Group by的Label越多 → 分组越细 → 通知越多
2. 静默不是万能的——长期静默的告警可能掩盖真问题
3. Repeat interval不要设太短（建议至少1小时）
4. 模板中引用的Annotation必须在每条告警规则中都定义

**常见踩坑经验**
1. **Notification Policy匹配不到告警**：确认Label key拼写完全一致（大小写敏感），且策略的优先级正确（顺序自上而下）。
2. **静默过期后告警依然不触发**：静默过期后告警规则需要重新经过Pending→Firing。如果Pending期还没过，不会立即触发。
3. **模板调试困难**：在Contact Point的Test按钮中可以测试模板效果，利用这个功能调试。

**思考题**
1. 如果告警规则中`for=5m`，但静默只有3分钟，静默结束后告警会立即触发还是会重新计算5分钟的等待期？
2. 如何设计一套"告警有效性"度量体系——追踪哪些告警规则产生的告警真正被处理了，哪些是"狼来了"？
