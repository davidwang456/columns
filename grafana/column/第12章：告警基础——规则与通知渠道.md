# 第12章：告警基础——规则与通知渠道

## 1. 项目背景

"凌晨3点数据库连接池耗尽，所有接口502，PagerDuty没响，原因是Grafana的告警规则配置有问题——'评估'状态挂了整整5小时没人知道。"

这是某电商公司一次真实的生产事故复盘。他们用了Grafana的告警功能，但配置上犯了一个致命错误：告警规则的"evaluation interval"设置了10分钟，而数据库连接池从枯竭到完全不可用只用了3分钟——告警还没评估，服务已经挂了。更糟糕的是，Notification Policy的匹配规则写错了，本应发给运维的告警发给了开发Leader，而leader凌晨手机静音。

Grafana新告警系统（Grafana Alerting，区别于传统Dashboard告警）在Grafana 9.0之后成为了标配，它提供了一套完整的多维告警引擎。但"能告警"和"告警能有效运转"之间有巨大的鸿沟——规则配置、评估行为、通知模板、静默策略、分组降噪，每一个环节不恰当都可以导致告警失效或告警风暴。

本章将从0开始搭建一套生产可用的告警体系，让你真正理解"什么是好的告警"。

## 2. 项目设计

**小胖**（看着手机上的99+未读消息）：大师，我昨天睡觉前Grafana给我发了83条告警，全是"CPU超过80%"。我爬起来一看，CPU确实超了，但3分钟后就恢复了，这期间发了83条是几个意思？

**大师**（叹气）：你这是经典的"告警风暴"问题。Grafana的告警评估机制是这样的：你设置规则"CPU > 80%，evaluation interval = 1m"。CPU从80%涨到90%再降到75%，5分钟的时间内每1分钟评估一次，每次都满足条件，于是发了5条告警。

但真正的问题不是告警发多了，而是你缺少分组和降噪。

**小白**（认真记录）：分组具体是怎么做的？

**大师**：Grafana Alerting体系有四个核心概念，理解它们的关系才能设计好的告警。

**Alert Rule**（告警规则）：定义"什么条件触发告警"。比如"CPU使用率 > 80% 持续 5分钟"。这里容易忽略的是"持续"——Grafana默认不是"瞬时超标就告警"，而是需要"连续N个评估周期都超标"才触发。

**Contact Point**（通知触点）：定义"告警发给谁、用什么渠道"。比如Email、Slack、钉钉、Webhook、PagerDuty、Telegram等。Grafana内置了20+种通知渠道。

**Notification Policy**（通知策略）：定义"哪类告警发给哪个Contact Point"。这是分组降噪的核心——你可以在策略树中按Label匹配，比如所有`severity=critical`的告警发给PagerDuty，`severity=warning`的只发Slack。

**Silence**（静默）：临时屏蔽某些告警。比如"今晚2点到4点数据库维护，预计会有告警，先静默掉"。

**小胖**（挠头）：那"grouping"又是怎么回事？我看配置里面有个Group by？

**大师**：Group by是通知策略中最重要的参数之一。举个例子，你有10台服务器，每台都有一个独立的CPU告警规则。不加Group by的话，10台服务器的CPU都是独立的告警事件，意味着10条通知。加了`Group by alertname`后，同一个alertname的所有告警会被合并成一条消息。

更进一步，设置Group wait = 30s，Group interval = 5m。意思是第一条告警触发后等30秒，把这段时间内触发的同类告警收集齐，打包成一条消息发出。后续每5分钟发送一次汇总更新。

**小白**（沉思）：这和Prometheus Alertmanager的分组机制很像。

**大师**：对，Grafana Alerting的架构就是借鉴Alertmanager。但Grafana Alerting的优势在于——你可以直接在Dashboard面板上看到告警状态。一个Time series图上如果有一段时间变红了，那就是告警期间。告警规则和可视化的天然联动，这是纯Alertmanager做不到的。

**小胖**：那具体怎么判断一个告警规则应该怎么设？比如"数据库连接池使用率"这个指标。

**大师**：好问题。设计告警规则有三个核心决策：

**阈值怎么设**：不是拍脑袋。要基于历史数据。用Grafana的Explore功能查看过去7天的数据分布——P95值是多少？P99是多少？阈值通常设在P95和P99之间。设太低会告警疲劳，设太高会有盲区。

**持续多久**："大于阈值5分钟"是经验默认值。对于快速自愈的波动（比如瞬时CPU spike），加持续时间可以过滤噪音。对于关键指标（如错误率），持续时间要设短。

**评估间隔**：默认1分钟。对于变化缓慢的指标（如磁盘使用率），评估间隔可以放长到5-10分钟。对于关键指标（如QPS突然降为0），评估间隔需要15-30秒。

**小胖**：最后一个问题——告警模板。我收到的告警消息就一行字"CPU > 80% on server01"，连个链接都没有，我还要自己去Grafana里找是哪个Dashboard。

**大师**：Grafana支持Go模板语法自定义通知内容。一个好的告警模板应该包含：
- Title：告警名称和严重级别
- Description：当前值、阈值、触发时间
- 一个直达Dashboard的链接（带时间范围和变量值）
- 如果是Slack/钉钉等渠道，还可以加颜色标记

模板示例（用于Slack）：
```
*{{ .Alerts.Firing | len }} firing, {{ .Alerts.Resolved | len }} resolved*
{{ range .Alerts.Firing }}
• *{{ .Labels.alertname }}*: {{ .Annotations.description }}
  Value: {{ .Annotations.value }}
  Dashboard: {{ .DashboardURL }}
{{ end }}
```

**技术映射**：Alert Rule = 火警感应器（感知异常），Contact Point = 消防队电话（通知谁），Notification Policy = 调度中心（什么火警通知谁），Silence = 维护公告（修设备期间别报警）。

## 3. 项目实战

**环境准备**

使用之前的Docker Compose环境。确保Prometheus和Node Exporter运行。

由于Grafana Alerting可以直接查询Prometheus，不需要额外部署Alertmanager即可完成告警闭环。

**步骤一：创建Contact Point（通知触点）**

Alerting → Contact points → Add contact point。

**方式一：Email通知**

| 参数 | 值 | 说明 |
|------|-----|------|
| Name | `email-ops` | |
| Integration | Email | |
| Addresses | `ops@example.com` | 多个地址用分号分隔 |

Email需要在grafana.ini中配置SMTP：
```ini
[smtp]
enabled = true
host = smtp.example.com:587
user = grafana@example.com
password = your-smtp-password
from_address = grafana@example.com
from_name = Grafana Alerting
```

测试按钮 → Send test notification → 确认收到邮件。

**方式二：Webhook通知（通用方案）**

创建一个Webhook Contact Point：
| 参数 | 值 |
|------|-----|
| Name | `webhook-dingtalk` |
| Integration | Webhook |
| URL | `https://oapi.dingtalk.com/robot/send?access_token=xxx` |
| HTTP Method | POST |

对于钉钉/飞书等IM工具，消息格式需要自定义：
```json
{
  "msgtype": "markdown",
  "markdown": {
    "title": "{{ .CommonLabels.alertname }}",
    "text": "## {{ .Status | title }}\n**Alert:** {{ .CommonLabels.alertname }}\n**Value:** {{ .CommonAnnotations.value }}\n[查看Dashboard]({{ .DashboardURL }})"
  }
}
```

**步骤二：创建第一条告警规则**

Alerting → Alert rules → New alert rule。

**规则1：CPU高负载告警**

| 属性 | 值 |
|------|-----|
| Rule name | `CPU使用率过高` |
| Type | Grafana managed alert |
| Folder | (选择存储位置) |
| Evaluation group | `infra-checks`（新建，interval = 1m）|
| Pending period | `2m`（持续2分钟才告警）|

Query部分：
```promql
100 - (avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)
```

Expression部分（从Query结果中创建告警条件）：
- Input: `A` (Query的结果)
- Expression type: `Threshold`
- Condition: `IS ABOVE 80`

设置Labels：
- `severity: warning`
- `team: ops`
- `service: infrastructure`

设置Annotations：
- `summary: CPU使用率 {{ $values.A.Value }}%`
- `description: 主机CPU使用率超过80%阈值，当前值：{{ $values.A.Value }}%`
- `value: {{ $values.A.Value }}%`

点击Save and exit。

**规则2：内存高负载告警**

Query：
```promql
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100
```

Expression: `IS ABOVE 90`，Pending period: `5m`。

**规则3：磁盘空间告警**

Query：
```promql
(1 - (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"})) * 100
```

Expression: `IS ABOVE 85`，Pending period: `10m`（磁盘变化慢）。

**规则4：服务不可用告警（No Data告警）**

对于一个必须存在的指标（如`up`指标），如果连续3分钟没有数据，说明服务下线。

Query：
```promql
up{job="node_exporter"}
```

Expression:
- Type: `Classic condition`
- Condition: `WHEN last() OF A HAS NO VALUE`

Pending period: `3m`。

**步骤三：配置Notification Policy通知策略树**

Alerting → Notification policies → 默认策略的Contact point设为`email-ops`。

添加子策略：

**Policy 1：严重告警走Webhook**
```
Matching labels: severity = critical
Contact point: webhook-dingtalk
Group by: alertname, service
Group wait: 10s
Group interval: 5m
Repeat interval: 4h
```

**Policy 2：警告告警走邮件**
```
Matching labels: severity = warning
Contact point: email-ops
Group by: alertname
Group wait: 30s
Group interval: 10m
Repeat interval: 12h
Continue matching siblings: Disabled
```

**Policy 3：基础设施告警**
```
Matching labels: service = infrastructure
Contact point: email-ops
Continue matching siblings: Disabled
```

策略树优先级：从上到下匹配，匹配到第一个`Continue matching siblings = Disabled`的策略后停止。

**步骤四：创建Silence静默**

Alerting → Silences → New Silence。

| 属性 | 值 |
|------|-----|
| Start | 立即开始或指定时间 |
| End | 2小时后或指定时间 |
| Matchers | `service = infrastructure` |
| Comment | "计划维护窗口，预期告警" |

**步骤五：告警状态验证**

主动触发告警（如运行CPU压测工具）：
```bash
# 在宿主机运行CPU压测，使CPU飙升
docker exec node_exporter stress --cpu 2 --timeout 120s
```

在Grafana Alerting → Alert rules中观察：
- Normal (绿) → Pending (黄) → Firing (红)
- 恢复后：Firing → Normal

在Contact points中查看通知发送历史。

**常见坑点**
1. **Pending period设太短**：设成0秒会导致每次评估只要超标就发告警。建议至少2倍evaluation interval。
2. **Group by设置的Label不存在**：如果Label在告警规则中没定义，Group by不会生效，所有告警会合并为一条。
3. **Email SMTP不工作**：很多企业邮箱需要App Password而非登录密码，且端口25常被云服务商封禁，需用587或465。
4. **告警规则中的Variables不适用**：带变量的Dashboard查询在创建告警规则时可能无法正确解析变量值，需要写死或使用相对时间范围。

## 4. 项目总结

**告警设计原则**

| 原则 | 说明 |
|------|------|
| 告警必须可操作 | 每条告警必须对应一个已知的处理动作 |
| 避免告警疲劳 | 持续时间过滤瞬时抖动，分组合并同类告警 |
| 告警有优先级 | 用severity label区分critical/warning/info |
| 携带上下文 | 通知中带上Dashboard链接、当前值、阈值 |
| 定期回顾 | 每月梳理告警有效性，清理无效规则 |

**优点**
| 特性 | 说明 |
|------|------|
| 可视化联动 | Dashboard面板直接显示告警状态和注释线 |
| 多维度告警 | 支持Metric/Label维度告警 |
| No Data/Error告警 | 支持检测数据缺失和查询错误 |
| 多通知渠道 | Email/Slack/钉钉/Webhook/PagerDuty等20+种 |
| 分组降噪 | Group by + Group wait抑制告警风暴 |

**缺点**
| 特性 | 说明 |
|------|------|
| 无告警确认 | 没有原生的告警认领/处理流程（需OnCall补足） |
| 无值班调度 | 没有排班功能，所有通知发给固定的Contact Point |
| 评估引擎单一 | 所有规则运行同一评估引擎，海量规则可能瓶颈 |
| 无告警升级 | 超时未处理的告警无法自动升级通知对象 |

**适用场景**
1. 基础设施告警：CPU/内存/磁盘/网络异常
2. 应用性能告警：QPS骤降、错误率飙升、P99延迟劣化
3. 业务指标告警：订单量异常、支付成功率下降
4. 数据完整性告警：关键指标No Data检测

**注意事项**
1. Alert Rule的Query必须返回有效的数值型数据，字符串类型无法做阈值比较
2. Notification Policy的Matching Labels支持正则（如`service =~ .*-prod$`）
3. Repeat interval默认4小时，如果希望持续提醒设为更短的值
4. Evaluation group的interval影响所有该组内的规则评估频率

**常见踩坑经验**
1. **Pending时期告警已发送**：误解了Pending period的含义——Pending期间告警不会发通知，只有进入Firing状态才发。
2. **告警规则修改后没有重新触发**：已触发的告警在规则修改后不会自动重评估，需要等待下一个evaluation周期。
3. **Notification Policy匹配不到规则**：检查Label key和value是否完全匹配，注意大小写敏感。

**思考题**
1. 如何设计一套"三级告警体系"——Info级发群消息、Warning级发邮件和群、Critical级邮件+电话+值班升级？
2. 如果一个告警规则覆盖了100个实例，其中1个实例触发告警，如何做到只发一条通知？如果50个实例同时触发，又如何做到消息不炸群？
