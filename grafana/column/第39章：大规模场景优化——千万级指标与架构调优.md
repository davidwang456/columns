# 第39章：大规模场景优化——千万级指标与架构调优

## 1. 项目背景

"当前Grafana管理着10000+个Dashboard、500+个DataSource、1500+个活跃用户。每天SQLite数据库写入量超过500MB，Dashboard列表加载需要10秒。更头疼的是，企业版渲染服务一小时能渲染2000张截图用于邮件报告，但Chromium进程频繁OOM。"

这是一家大型金融科技公司的Grafana平台现状。当Grafana从小团队工具变为企业级平台后，早期不作特殊优化的架构暴露出各种问题：数据库读写得慢、前端加载Dashboard列表卡顿、告警评估引擎在高基数时过载、渲染服务不稳定。

本章聚焦大规模场景下的架构调优——不是"怎么用Grafana"，而是"怎么让Grafana撑住"。涵盖数据库优化、渲染服务、缓存架构、查询治理和前端性能调优。

## 2. 项目设计

**小胖**：大师，我们公司Grafana有500个数据源，每次打开数据源列表页面转圈30秒。这正常吗？

**大师**：500个数据源意味着每次打开数据源管理页面，Grafana要查数据库→返回500条记录→其中50条还要做健康检查→每条健康检查是一个HTTP请求→总共50个并发HTTP请求。慢是正常的，但可以优化。

**小白**：从哪开始优化？

**大师**：大规模Grafana优化分五个维度：

**数据库优化**：
- `max_open_conn`从默认100调整到实际需求（如200-500）
- 对高频查询字段建立索引（`org_id`、`uid`、`folder_uid`）
- 分离读写：主库PostgreSQL、从库只读查询
- 历史数据归档：定期清理老旧的Dashboard版本（versions_to_keep设小）

**渲染服务优化**：
- 使用Remote Rendering Service（独立渲染节点），与主Grafana进程隔离
- 渲染节点池化：多节点负载均衡，Chromium进程独立管理
- 渲染超时控制：设置最大渲染时间（如60s），超时自动kill
- 截图策略：不是每次Dashboard刷新都渲染，批量定时渲染

**缓存架构优化**：
```
Grafana → Redis (查询结果缓存 + Session)
     → 内存 (热点Dashboard模板)
     → CDN (静态资源)
```

- Redis缓存查询结果（TTL 60-300s），大幅减少重复查询
- CDN缓存前端静态资源（JS/CSS），减少Grafana自身流量
- Dashboard数据缓存：多人同时打开同一Dashboard时，后端只查询一次

**查询治理优化**：
- 限制单个Dashboard的面板数量（如不超过30个）
- 限制查询的超时时间（dataproxy timeout）
- 配置查询速率限制（grafana.ini中的`concurrent_query_limit`）
- 禁止全表扫描式查询（对SQL数据源设置查询白名单）

**前端性能优化**：
- Dashboard列表分页：不一次性加载10000个Dashboard
- 懒加载：Dashboard面板按需加载（只加载可见区域的面板）
- Code splitting：按路由分割JS bundle

**小胖**：这么多优化点，哪些是"花小钱办大事"的？

**大师**：按优先级排序：
1. **数据库迁PostgreSQL**（收益最大，一劳永逸）
2. **Dashboard列表分页**（前端响应从30s→1s）
3. **查询结果缓存**（减少50%+数据源查询）
4. **限制面板数量**（治理层面，避免单个Dashboard过大）
5. **渲染服务独立部署**（对于有截图需求的企业）

**技术映射**：数据库优化 = 给仓库装更多货架和更快的传送带，查询缓存 = 常用物品放前台不每次进仓库拿，CDN = 各地设分仓就近取货，渲染服务 = 独立的打印间和主操作间隔开。

**步骤二：Dashboard治理**

批量分析Dashboard的"健康度"：
```python
import requests

def analyze_dashboards(grafana_url, token):
    headers = {"Authorization": f"Bearer {token}"}
    
    # 获取所有Dashboard
    resp = requests.get(f"{grafana_url}/api/search", 
                       params={"type": "dash-db", "limit": 5000},
                       headers=headers)
    dashboards = resp.json()
    
    issues = []
    for db in dashboards:
        detail = requests.get(
            f"{grafana_url}/api/dashboards/uid/{db['uid']}", headers=headers
        ).json()
        
        panels = detail["dashboard"].get("panels", [])
        panel_count = len(panels)
        
        # 检查面板数量
        if panel_count > 30:
            issues.append({
                "dashboard": db["title"],
                "issue": f"面板过多({panel_count}个)，建议拆分",
                "severity": "warning"
            })
        
        # 检查无用的Dashboard（超过90天未更新）
        # (需要从dashboard_version表查询)
        
        # 检查面板查询数量
        total_targets = sum(
            len(panel.get("targets", [])) for panel in panels
        )
        if total_targets > 100:
            issues.append({
                "dashboard": db["title"],
                "issue": f"查询过多({total_targets}个)，可能影响性能",
                "severity": "warning"
            })
    
    return issues
```

Dashboard治理SOP：
1. 标记"僵尸Dashboard"（90天无人访问）
2. 将面板过多的Dashboard拆分为多个小Dashboard
3. 为Dashboard添加Owner标签，落实责任人
4. 设置`versions_to_keep=10`减少数据库膨胀

**常见坑点**
1. **PostgreSQL连接数不够**：Grafana + OnCall + 其他服务共用同一个PostgreSQL，连接数超限。
2. **Redis单点故障**：缓存全走单节点Redis→Redis挂了→Grafana每次查询都请求后端→数据源被打爆。
3. **渲染服务OOM**：同时渲染10个Dashboard的截图，每个Chromium进程500MB→10个=5GB→OOM。

**步骤六：大规模场景性能压测**

```bash
#!/bin/bash
# load-test-grafana.sh
USERS=500
DURATION=300  # 5分钟

echo "=== Grafana负载测试 ($USERS用户/$DURATION秒) ==="

# 1. 模拟用户登录并打开Dashboard
for i in $(seq 1 $USERS); do
    (
        TOKEN=$(curl -s -X POST http://grafana:3000/login \
            -d "{\"user\":\"user$i\",\"password\":\"password\"}" \
            -c /tmp/cookie_$i | jq -r '.message')
        
        while true; do
            # 随机选择一个Dashboard并加载
            DASH_UID=$(curl -s -H "Authorization: Bearer $TOKEN" \
                "http://grafana:3000/api/search?type=dash-db&limit=1&start=$((RANDOM % 1000))" | \
                jq -r '.[0].uid')
            
            if [ "$DASH_UID" != "null" ]; then
                curl -s -o /dev/null -w "$i: %{time_total}s\n" \
                    -H "Authorization: Bearer $TOKEN" \
                    "http://grafana:3000/api/dashboards/uid/$DASH_UID" &
            fi
            
            sleep $((RANDOM % 10 + 5))  # 每5-15秒刷新一次
        done
    ) &
done

# 2. 监控Grafana自身指标
while true; do
    echo "--- $(date) ---"
    curl -s http://grafana:3000/metrics | grep -E \
        "grafana_http_request_duration_seconds_count|grafana_db_request_duration_seconds|grafana_stat_active_users"
    sleep 10
done

wait
```

**容量规划模型**：
```python
# capacity-plan.py
def estimate_resources(users, dashboards, sample_rate):
    """
    估算Grafana集群需要的资源
    
    users: 并发用户数
    dashboards: Dashboard数量
    sample_rate: 每秒写入的samples数
    """
    grafana_cpu_cores = max(4, users / 50)
    grafana_memory_gb = max(8, users / 30)
    db_connections = max(50, users * 2)
    redis_memory_gb = max(2, users / 100)
    
    mimir_ingester_memory_gb = sample_rate * 86400 / 1e6 * 3
    
    return {
        "grafana_cpu_cores": grafana_cpu_cores,
        "grafana_memory_gb": grafana_memory_gb,
        "db_connections": db_connections,
        "redis_memory_gb": redis_memory_gb,
        "mimir_ingester_memory_gb": mimir_ingester_memory_gb,
    }

# 示例：2000用户，10000 Dashboard，每秒50万samples
print(estimate_resources(2000, 10000, 500000))
# 输出：grafana需40核/60GB，数据库需4000连接
```

**大规模优化清单**

| 优化维度 | 措施 | 收益 | 难度 |
|---------|------|------|------|
| 数据库 | PostgreSQL + 索引 + 读写分离 | 🔥🔥🔥🔥🔥 | 中 |
| 缓存 | Redis查询缓存 + CDN静态资源 | 🔥🔥🔥🔥 | 低 |
| 治理 | Dashboard拆分 + 面板限制 | 🔥🔥🔥🔥 | 低 |
| 渲染 | 独立渲染节点 + 集群模式 | 🔥🔥🔥 | 中 |
| 前端 | 分页 + 懒加载 + Gzip | 🔥🔥🔥 | 低 |
| 查询 | 超时控制 + 速率限制 | 🔥🔥 | 低 |

**容量规划建议**
- <100 用户 + <1000 Dashboard：SQLite单实例
- 100-500 用户：PostgreSQL + Redis
- 500-2000 用户：PostgreSQL读写分离 + Redis集群 + CDN + 独立渲染
- >2000 用户：以上 + 多Grafana实例 + 数据库分库分表
