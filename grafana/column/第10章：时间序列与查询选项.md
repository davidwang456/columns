# 第10章：时间序列与查询选项

## 1. 项目背景

"为什么我选了'Last 7 days'后Dashboard要等20秒才出来？""为什么同样的PromQL，在'Last 1 hour'下曲线很平滑，到'Last 30 days'就变成了锯齿状？""Max data points 这个参数到底设多少合适？"

Grafana作为时序数据可视化的核心工具，时间维度的处理贯穿了从查询到渲染的整个链路。很多Grafana使用者在享受Dashboard便利的同时，却忽略了背后一系列关键的时间参数配置：时间范围（from/to）、时间偏移（Time Shift）、采样间隔（Min interval）、数据点限制（Max data points）、查询缓存等。这些参数如果配置不当，轻则Dashboard加载慢，重则展示的数据完全失真。

本章将深入Grafana的时间模型，从Dashboard级别的时间范围控制，到面板级别的查询精度参数，再到Grafana内置的查询缓存机制，帮你建立起对时间序列可视化的完整理解，让你的Dashboard在任何时间范围下都能既快又准。

## 2. 项目设计

**小胖**（焦急地敲着桌子）：大师救命！老板要看过去30天的CPU趋势，我把Dashboard时间范围从"Last 6 hours"改成了"Last 30 days"，结果页面加载了25秒，最后还超时报错了。这不科学啊，明明数据就在Prometheus里！

**大师**（镇定地端起咖啡）：你触发了Grafana时间序列中最常见也是最致命的性能问题——无限制数据量查询。我打个比方：你想看一部电影的发展脉络，是把120分钟从头看到尾快，还是看2小时的逐帧截图快？

**小胖**：当然是正常播放快啊，逐帧截图好几万张呢！

**大师**：对。你现在的查询就像在要"逐帧截图"——过去30天的CPU数据，如果15秒一个采样点，那就是172800个数据点。你让Grafana把这些点全拿回来再渲染，当然慢。

**小白**（若有所思）：所以需要降精度？

**大师**：正是。Grafana有一个叫Max data points的参数，默认是图形的像素宽度。比如你的Time series面板在屏幕上宽800像素，Max data points就自动设为800。Grafana的工作逻辑是：不管你查询的时间范围多大，最终只需要大约800个点来画成图。所以它会把原始数据按时间聚合，把172800个点压缩到800个。

**小胖**（追问）：那具体怎么聚合的？取平均值吗？

**大师**：取决于你在Query options里选择的`Interval`和`Min interval`。

`Min interval`是你允许的最小采样间隔。比如你设Min interval=1m，意思是"不管原始数据多密，每1分钟最多取一个点"。对于30天的查询，Grafana自动计算需要的interval = 30天 / 800点 ≈ 54分钟。如果你的Min interval是1m，最终interval会取max(自动计算值, Min interval)。

`Interval`是硬编码的采样间隔。如果你手动设Interval=5m，Grafana就按固定5分钟间隔聚合（忽略Min interval和自动计算）。

**小胖**（挠头）：那这个"聚合"具体是avg、max还是min？我能选吗？

**大师**：这就是容易忽视的细节。在`Query options`的最下方有一个下拉框"Relative time"，里面可以设Group by的聚合函数。默认是`avg`。但实际上这个参数不总是生效——它只在Grafana执行降精度时起作用（即原始数据点多于Max data points）。

如果你的数据本来就只有100个点（小于800），Grafana不做任何降精度，你设什么聚合函数都没影响。

**小白**：有一个相关概念我一直混淆——$__interval和$__rate_interval在Prometheus查询中有什么区别？

**大师**：这是Prometheus用户的必修课。

`$__interval`是Grafana计算出的应该使用的采样间隔。在Dashboard时间范围为1h、Max data points=800的情况下，$__interval = 1h/800 ≈ 4.5s。

`$__rate_interval`是为rate函数优化的间隔，等于max($__interval + 容错偏移, 4×scrape_interval)。为什么加这个？因为rate()函数的工作窗口如果恰好小于一个采集周期，可能一个数据点都捕获不到。

所以正确的写法是：
```promql
rate(http_requests_total[$__rate_interval])
```
而不是：
```promql
rate(http_requests_total[$__interval])
```

**小胖**（眼睛一亮）：那Query caching查询缓存又是怎么回事？我开了缓存但感觉没效果。

**大师**：Grafana的查询缓存有两个层面的含义。

第一层是**浏览器端缓存**。如果你在Dashboard设置里开启了"Caching"，同一个查询在同一个时间范围内不会重复发送。比如你有两个面板复制了同一个PromQL，打开Dashboard时只发一次查询。

第二层是**服务端缓存**（需要Enterprise版或配置外部缓存）。Grafana把查询结果存在内存或Redis中，TTL内相同的查询直接返回缓存。这对于热门Dashboard（比如公司大屏很多人看）效果显著——100个人同时打开同一个Dashboard，后端只查一次。

但缓存有局限性——它按查询字符串+时间范围做Key，稍微不一样就命中不了。而且默认TTL只有60秒，长周期的历史数据查询其实很适合长TTL。

**小白**：还有Time Shift时间偏移。我经常需要比较今天和昨天的数据，或者这周和上周。

**大师**：Time Shift有两种实现方式。

方法一：在查询中直接用Prometheus的`offset`修饰符。`rate(http_requests_total[5m] offset 1d)`就是昨天同一时段的QPS。优点：在一个面板里同时画今天和昨天两根线。缺点：每个时间偏移需要单独写一个查询。

方法二：在面板的Query options中设置Relative time。比如设置`1d`，面板就会显示24小时前的数据。优点：不需要改查询。缺点：面板只看偏移后的数据，看不到同时段对比。

**小胖**（总结）：所以最佳实践是——短时间（<6h）用原始精度，长时间（>24h）开Max data points限制，所有rate函数用$__rate_interval，热点Dashboard开启查询缓存。对吗？

**大师**：90分！再补充一个：Min interval不要设太小。很多人出于"精度焦虑"把Min interval设成1s，结果就是任何Dashboard都得拉取高密度数据，白白拖慢速度。建议至少设为scrape_interval的值（如15s、30s）。

**技术映射**：Max data points = 像素宽度（屏幕只能显示这么多点，多了浪费），Min interval = 粒度控制（至少这个间隔才取一个点，防精度失控），Query caching = 图书馆预约（别人借过的书直接给，不用重新入库）。

## 3. 项目实战

**环境准备**

基于之前的Docker Compose环境。

**步骤一：理解时间范围的UI控制**

在Dashboard右上角的时间选择器，可以进行以下操作：

1. **Quick ranges**：Last 5 minutes / 15 minutes / 1 hour / 6 hours / 12 hours / 24 hours / 2 days / 7 days / 30 days / 90 days
2. **Custom range**：精确到秒的自定义起止时间
3. **Time zone**：默认Browser Time（用户浏览器时区），可改为UTC
4. **Refresh interval**：关闭(Off) / 5s / 10s / 30s / 1m / 5m / 15m / 30m / 1h

在Dashboard Settings → Time options中：
- **Now delay**：默认0s。设置30s意味着"现在"是30秒前，避免查询到"未来"的数据点
- **Hide time picker**：Kiosk/TV Mode时隐藏时间选择器

**步骤二：面板级Query Options配置**

创建Time series面板，重点配置Query options参数。

**测试1：观察Max data points对图表的影响**

查询：
```promql
rate(node_cpu_seconds_total{mode!="idle"}[5m])
```

时间范围选"Last 30 days"。

在Query options中：
- 默认不设Max data points → 图可能加载慢且线密（Grafana自动取图形宽度）
- 设置Max data points = `200` → 30天数据压缩到200个点，加载快但细节丢失
- 设置Max data points = `2000` → 更多数据点，更平滑但更慢

最佳实践：一般设为图形宽度的1-2倍，如800-1600。

**步骤二：Min interval实战**

同样查询，时间范围"Last 7 days"，面板宽度约800px。

| Min interval配置 | 自动计算Interval | 实际数据点 | 效果 |
|-----------------|-----------------|-----------|------|
| 不设（默认）| 7d/800 ≈ 12.6min | ~800 | 正常 |
| 1m | max(12.6min, 1m)=12.6min | ~800 | 正常（Min interval小于自动值，不生效）|
| 30m | max(12.6min, 30m)=30min | ~336 | 曲线粗，细节少 |
| 1h | max(12.6min, 1h)=1h | ~168 | 曲线很粗，大量细节丢失 |

推荐：Min interval设为与Prometheus scrape_interval匹配的值（15s或30s），让Grafana自动决定长周期查询的interval。

**步骤三：$__interval vs $__rate_interval 对比**

创建两个面板，A用$__interval，B用$__rate_interval。

**面板A**：
```promql
rate(node_network_receive_bytes_total[${__interval}])
```

**面板B**：
```promql
rate(node_network_receive_bytes_total[${__rate_interval}])
```

时间范围分别设为"Last 15 minutes"和"Last 6 hours"对比观察。

在短时间范围下，`$__interval`可能仅为1-2秒，而Prometheus的scrape_interval是15秒——此时rate的计算窗口没有数据点，结果为空。而`$__rate_interval`会自动扩大到至少4×scrape_interval。

**步骤四：时间偏移（Time Shift）对比**

**方案A：Prometheus offset**

Query A（今天）：
```promql
sum(rate(http_requests_total[5m]))
```

Query B（昨天同一时段）：
```promql
sum(rate(http_requests_total[5m] offset 1d))
```

使用Transform → Merge和Rename，同时展示两根线。

**方案B：Query Options Relative time**

在面板Query options中设置Relative time = `1d`。

查询保持不变：
```promql
sum(rate(http_requests_total[5m]))
```

此时面板显示的是昨天同一时段的QPS，但左下角会显示时间偏移提示。

**步骤五：查询缓存配置**

在grafana.ini中配置缓存：
```ini
# 服务端查询缓存（需要Redis或内存缓存）
[query_history]
enabled = true

# 数据源代理超时
[dataproxy]
timeout = 60
dialTimeout = 30
keep_alive_seconds = 30
```

Dashboard级别的缓存设置：
在Dashboard Settings → General → Cache timeout：设为60（秒）。表示60秒内重复打开该Dashboard，数据从缓存读取不重查。

使用API检查缓存状态：
```bash
# 查看Grafana自身指标中的缓存统计
curl -s http://localhost:3000/metrics | grep grafana_cache
```

**步骤六：Now delay——时间的艺术**

Now delay是一个容易忽略但重要的参数。它的作用是让"现在"等于"N秒前"。

Dashboard Settings → Time options → Now delay → 设为`30s`。

为什么需要？假设现在是14:00:00。Prometheus最近一次采集是13:59:45（15s采集间隔）。如果你查询`now`到`now-5m`，Grafana请求的是`14:00:00`到`13:55:00`。但`13:59:45`到`14:00:00`这个15秒区间内没有数据，会导致Rate计算出现不自然的下降趋势。

Now delay=30s解决了这个问题——实际查询端点退回到最后一个完整采集周期之前。

**常见坑点**
1. **User time zone与数据源时间不一致**：Prometheus内部使用UTC时间，如果Dashboard时区选了Asia/Shanghai，`$__from`和`$__to`变量会自动转换为用户时区——但在PromQL中，最好保持UTC避免混淆。
2. **Long query timeout**：默认查询超时30s。如果你查30天的原始精度数据，很容易超时。解决：要么加大超时（grafana.ini的dataproxy timeout），要么降低精度（增大Min interval）。
3. **Max data points 设太大**：有人为了"最高精度"把Max data points设成0（不限制）或100000，结果浏览器渲染直接卡死。
4. **Relative time与offset同时使用效果叠加**：Query options里设置Relative time = 1d，查询里又用了offset 1d，实际看到的是2天前的数据。

## 4. 项目总结

**关键参数速查表**

| 参数 | 位置 | 含义 | 推荐值 |
|------|------|------|--------|
| Time range | Dashboard顶部 | 查询的起止时间范围 | 根据场景选择 |
| Now delay | Dashboard Settings | "现在"的回退秒数 | 30s |
| Refresh interval | Dashboard顶部 | 自动刷新间隔 | 关闭或5m（非TV模式）|
| Max data points | Panel Query options | 渲染的最大数据点数 | 800-2000（图形宽度的1-2倍） |
| Min interval | Panel Query options | 最小采样间隔 | 15s-30s（=采集间隔）|
| Interval | Panel Query options | 固定采样间隔 | 留空（自动计算） |
| Relative time | Panel Query options | 查询的时间偏移量 | 按需 |
| Cache timeout | Dashboard Settings | 查询结果缓存时间 | 60-300s |
| $__interval | PromQL变量 | 自动计算的采样间隔 | 用于非rate函数 |
| $__rate_interval | PromQL变量 | rate优化的采样间隔 | 用于rate/increase函数 |

**适用场景**
1. 运维实时监控：短时间范围(15m-1h)，密集刷新，Now delay=30s
2. 日周月报告：长时间范围(7d-30d)，Max data points=2000，Min interval=5m
3. 同比环比分析：Time Shift offset + Merge Transform
4. TV大屏展示：自动刷新(30s-1m)，隐藏时间选择器

**注意事项**
1. 查询时间范围和Max data points共同决定实际采样间隔：interval = (to - from) / max_data_points
2. `$__interval`的最小值受Min interval约束，避免单个查询请求过多数据
3. 缓存对实时监控场景（5分钟以内）意义不大，对日报/周报等历史查询意义重大
4. TV Mode（Kiosk Mode）下，时间选择器自动隐藏，通过URL参数控制时间范围

**常见踩坑经验**
1. **Refresh interval设太短导致查询风暴**：TV大屏设5s自动刷新，50个面板每5秒100+次查询。解决：延长刷新间隔到30s或1m。
2. **Relative time与Absolute time混淆**：Relative time = `1d`是"当前时间往前推1天"；如果Dashboard右上角选了Last 7 days，Relative time再设1d，实际看的是"8天前到1天前"。容易产生时序错乱。
3. **Max data points对Heatmap面板无效**：Heatmap面板的精度由Y轴bucket控制，不是Max data points。Heatmap大数据量场景需要单独优化查询。

**思考题**
1. 用户报告说Dashboard在'Last 24h'下查询正常(3s)，但在'Last 7d'下查询超时(30s)。你能设计一个优化方案吗？（提示：考虑Recording rules）
2. 某团队使用Grafana的"Compare time ranges"功能对比"生产环境vs预发布环境"的QPS，发现两条曲线的峰值位置差了15秒。这15秒的偏移是什么原因造成的？如何消除？
