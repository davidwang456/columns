# 第16章：【基础篇综合实战】企业级运维监控大盘搭建

## 1. 项目背景

"三个月了，我们Grafana上的Dashboard数量从5个涨到了53个，但没有一个人能说清楚'系统现在到底健康不健康'。老板要看一个综合报告，我只能把十几个Dashboard的截图拼在一起发给他。"

运维主管老周终于意识到问题的严重性。三个月前，他的团队引入了Grafana+Prometheus做监控。团队热情高涨，每个人建了各自的Dashboard——有人监控网络，有人看数据库，有人关注应用。但53个Dashboard之间毫无关联，信息碎片化严重，没有一个"总览视图"能让任何人一眼看出系统健康状态。

这就是本章要解决的问题：基于前15章所学的全部知识，从零搭建一套企业级监控体系，包括主机监控、应用监控、业务监控三层，并用变量实现多环境复用，用告警实现异常主动发现。本章将基础篇的所有知识点串联起来，完成一次端到端的实战交付。

## 2. 项目设计

**小胖**（盯着自己的53个Dashboard列表）：大师，我们学完了Dashboard、Panel、DataSource、变量、告警、API、Provisioning……学了这么多，现在老板说"给我一个能看的监控系统"，我能把这些全用上吗？

**大师**：（翻看小胖的Dashboard列表）：让我看看你的作品。这个叫"CPU测试1"、这个"test-勿删"、这个"内存监控-新-最终版-不要再改了"……你这53个Dashboard里，有40个是草稿和测试页。

**小白**（扶额）：这是典型的"Dashboard膨胀"问题。没有规划就开建，最后长得像城中村一样。

**大师**：对。做一个好的监控体系，首先要做好Dashboard的分层设计，而不是见什么建什么。我给你的方案是"三层监控金字塔"：

第一层：基础设施层（Infrastructure）——CPU、内存、磁盘、网络。受众：运维团队。1-2个Dashboard。

第二层：应用服务层（Application）——QPS、延迟、错误率、依赖服务健康。受众：开发团队。每个核心服务1个Dashboard。

第三层：业务指标层（Business）——订单量、支付成功率、活跃用户、转化率。受众：管理层/产品。1个Dashboard。

三层从上到下递进：管理层看到支付成功率低了→点链接跳到支付服务的应用层Dashboard→发现P99延迟高了→点链接跳到主机监控Dashboard→发现某台服务器CPU飙高。

**小胖**：那这三个层级的Dashboard怎么设计？

**大师**：我给你每层一个标准化模板。

**基础设施层模板**：顶部一行Stat（CPU均值、内存均值、磁盘均值告警数），中间Time series趋势图（24小时变化），底部Table（排名靠前的异常实例，点击可跳转详情）。

**应用服务层模板**：左上Stat（QPS/错误率/P99延迟——RED三大指标），右上Service Graph（依赖拓扑），中间Time series（各指标24小时趋势），底部Table（各实例健康度排行）。

**业务指标层模板**：顶部Stat（今日核心KPI），中间Time series（7天/30天趋势），补充Pie chart（渠道/地区占比），底部Table（详细数据）。

这三个模板设计好之后，通过变量$service、$env、$instance切换——1个模板Dashboard可以覆盖几十个服务的场景。

**小白**：告警规则怎么统一管理？

**大师**：告警也分层。基础设施告警（CPU>80%持续5分钟，磁盘>85%持续30分钟）由运维统一管理。应用告警（错误率>1%，P99>500ms）由各服务的开发Owner配置。业务告警（订单量同比跌>20%，支付成功率<95%）由业务负责人配置。

所有告警统一走Notification Policy路由——Critical级发PagerDuty+钉钉，Warning级发钉钉，Info级发邮件。

**小胖**：那一口气建这么多，我是不是又得手工点来点去？

**大师**：这就是Provisioning+API派上用场的时候。把基础层的Dashboard JSON模板写好，然后通过API脚本批量创建服务的Dashboard。告警规则同样用API批量创建。

**技术映射**：三层监控金字塔 = 医院三级诊疗（发热门诊→专科检查→专家会诊），Dashboard模板 = 标准化户型（同一套图纸建不同的楼），告警分层 = 消防等级（小火自灭、中火报警、大火全员出动）。

**步骤一：基础设施层Dashboard**

创建 `provisioning/dashboards/infrastructure-overview.json`

核心面板设计：

```
Row 1: 基础设施总览
┌─────────────┬─────────────┬─────────────┬─────────────┐
│ CPU 均值    │ 内存 均值   │ 磁盘 均值   │ 网络流量    │
│ Stat 58%    │ Stat 72%   │ Stat 45%   │ Stat 1.2GB │
│ (Green)     │ (Yellow)   │ (Green)    │ (Green)    │
└─────────────┴─────────────┴─────────────┴─────────────┘

Row 2: 趋势图
┌──────────────────┬──────────────────┐
│ CPU趋势(24h)      │ 内存趋势(24h)    │
│ Time series      │ Time series      │
└──────────────────┴──────────────────┘

Row 3: 实例详情表
┌────────────────────────────────────┐
│ 实例健康度排行榜                    │
│ Table: 实例名 | CPU | 内存 | 磁盘 |  │
│       排序: CPU降序, 着色: 阈值    │
└────────────────────────────────────┘
```

Key PromQL查询：

```promql
# CPU全局均值（Stat面板）
avg(100 - (rate(node_cpu_seconds_total{mode="idle"}[5m]) * 100))

# CPU趋势（Time series，按实例分组）
100 - (avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100)

# 内存使用率
(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100

# 磁盘使用率
(1 - (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"})) * 100

# 网络流量（每秒字节）
rate(node_network_receive_bytes_total{device!="lo"}[5m])
rate(node_network_transmit_bytes_total{device!="lo"}[5m])
```

变量配置：
- `$env`：Custom（dev, staging, prod）
- `$instance`：Query（多选，Include All）
- `$interval`：Interval（1m, 5m, 15m）

**步骤二：应用服务层Dashboard**

每个服务一个Dashboard，但用变量复用。

Dashboard模板设计：

```
Row 1: RED指标（Rate/Errors/Duration）
┌─────────┬──────────┬──────────┐
│ QPS     │ Error%   │ P99 Lat  │
│ 1200/s  │ 0.05%    │ 45ms     │
│ Stat+   │ Stat+    │ Stat+    │
│ Spark   │ Color BG │ Spark    │
└─────────┴──────────┴──────────┘

Row 2: 趋势图
┌──────────────────────────────────┐
│ QPS + Error Rate (双Y轴)         │
│ Time series                      │
└──────────────────────────────────┘

Row 3: 延迟分析
┌────────────────┬─────────────────┐
│ P50/P90/P99    │ Latency Heatmap │
│ Time series    │ (分桶分布)      │
└────────────────┴─────────────────┘

Row 4: 实例健康度
┌──────────────────────────────────┐
│ 实例 | QPS | Error% | P99 | CPU │
│      Table (可排序,可下钻)       │
└──────────────────────────────────┘
```

变量配置：
- `$service`：Query (label_values(http_requests_total, service))
- `$instance`：Query (级联依赖$service)
- `$percentile`：Custom (50, 90, 95, 99, 999)

核心PromQL：
```promql
# QPS
sum(rate(http_requests_total{service=~"$service"}[5m]))

# 错误率
sum(rate(http_requests_total{service=~"$service",status=~"5.."}[5m]))
/ sum(rate(http_requests_total{service=~"$service"}[5m])) * 100

# P99延迟
histogram_quantile(0.99,
  sum(rate(http_request_duration_seconds_bucket{service=~"$service"}[5m])) by (le)
)

# 延迟多分位数（动态变量$percentile）
histogram_quantile($percentile/100,
  sum(rate(http_request_duration_seconds_bucket{service=~"$service"}[5m])) by (le)
)
```

**步骤三：业务指标层Dashboard**

业务Dashboard展示核心业务数据（假设从MySQL获取）：

```sql
-- 今日订单量（Stat面板）
SELECT COUNT(*) as value FROM orders 
WHERE $__timeFilter(created_at)

-- 订单趋势（Time series）
SELECT 
  $__timeGroup(created_at, '1h') as time,
  COUNT(*) as value
FROM orders
WHERE $__timeFilter(created_at)
GROUP BY time ORDER BY time

-- 各渠道订单占比（Pie chart）
SELECT channel, COUNT(*) as value
FROM orders
WHERE $__timeFilter(created_at)
GROUP BY channel
```

**步骤四：统一告警规则配置**

基础设施告警：
```yaml
- alert: HostCPUHigh
  expr: 100 - (avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
  for: 5m
  labels:
    severity: warning
    team: ops
  annotations:
    summary: "主机CPU使用率 {{ $value }}%"

- alert: HostDiskFull
  expr: (1 - (node_filesystem_avail_bytes / node_filesystem_size_bytes)) * 100 > 85
  for: 10m
  labels:
    severity: critical
    team: ops
```

应用服务告警：
```yaml
- alert: ServiceHighErrorRate
  expr: |
    sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)
    / sum(rate(http_requests_total[5m])) by (service) * 100 > 1
  for: 3m
  labels:
    severity: warning
    team: dev
```

**步骤五：Dashboard关联与下钻**

在基础设施Dashboard的Table面板中设置Data Link：
```
Data Link → Type: Dashboard → Target: "应用服务监控"
→ 传递变量: ${__data.fields.instance}
```

在应用服务Dashboard的Panel上设置向下跳转到基础设施：
```
Data Link → Type: Dashboard → Target: "基础设施监控"
→ 传递变量: ${__data.fields.instance}
```

**步骤六：自动化批量创建**

使用Python脚本批量创建服务Dashboard：

```python
import requests, json

SERVICES = ["order", "user", "payment", "inventory", "notification"]
TEMPLATE_FILE = "service-dashboard-template.json"

with open(TEMPLATE_FILE) as f:
    template = json.load(f)

for svc in SERVICES:
    dash = json.loads(json.dumps(template))  # deep copy
    dash["dashboard"]["title"] = f"{svc} 服务监控"
    dash["dashboard"]["uid"] = f"service-{svc}"
    dash["dashboard"]["templating"]["list"][0]["query"] = svc
    
    resp = requests.post(
        "http://localhost:3000/api/dashboards/db",
        headers={"Authorization": "Bearer xxx", "Content-Type": "application/json"},
        json={"dashboard": dash["dashboard"], "overwrite": True}
    )
    print(f"[{resp.status_code}] {svc} Dashboard: {resp.json().get('uid')}")
```

**常见坑点**
1. **Dashboard模板的变量覆盖**：批量创建时注意每个Dashboard的变量值要独立，不要共享。
2. **Data Link的变量传递**：`${__data.fields.xxx}`中的字段名必须和实际数据中的字段名完全匹配。
3. **三层Dashboard的权限隔离**：不同团队看不同的Folder，避免"基础设施Dashboard被开发误改"。

## 4. 项目总结

**三层监控体系总览**

| 层级 | Dashboard数 | 主要受众 | 关键指标 | 告警Owner |
|------|------------|---------|---------|----------|
| 基础设施 | 1-2个 | 运维 | CPU/内存/磁盘/网络 | 运维团队 |
| 应用服务 | 1个/服务 | 开发 | QPS/Error/P99/Cache命中 | 开发Owner |
| 业务指标 | 1个 | 管理层 | 订单量/转化率/用户活跃 | 业务负责人 |

**本次实战涉及的基础篇知识点**

| 知识点 | 应用位置 |
|--------|---------|
| Dashboard JSON模型 | 设计Dashboard模板 |
| 面板类型 | Stat用于KPI、Time series用于趋势、Table用于排行 |
| 数据源 | Prometheus用于指标、MySQL用于业务数据 |
| 变量模板 | $env/$service/$instance多级联动 |
| Transform | 合并多个查询结果到Table面板 |
| 告警 | 三层告警体系统一管理 |
| API | 批量创建Dashboard，Provisioning管理配置 |

**思考题**
1. 如果公司规模扩大10倍，从50个服务增加到500个服务，服务Dashboard从1个模板创建500个Dashboard会带来什么问题？如何解决？
2. 基础设施Dashboard中的"CPU均值"Metric覆盖了所有主机，但如果一台主机的CPU达到95%而其他99台都在30%，均值可能只有31%——这种"平均值陷阱"如何规避？
