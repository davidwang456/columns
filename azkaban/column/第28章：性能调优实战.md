# 第28章：性能调优实战

## 1. 项目背景

### 业务场景

某电商平台大促期间，Azkaban需要在一小时内触发200个ETL Flow。但运维发现：凌晨0点到1点之间，Web Server的CPU飙升到90%，API响应从平时的50ms涨到5秒；Executor的Job提交延迟从100ms涨到30秒——Flow排着队等待执行。最严重的是，30%的Flow因为超时被标记为失败，但实际任务本身没有问题。

大促结束后，团队对Azkaban进行了全面性能分析，发现瓶颈集中在三个方面：MySQL连接池不足、JVM GC频繁、调度时Job文件解析重复。

### 痛点放大

性能瓶颈不解决时：

1. **高峰雪崩**：大促凌晨流量暴增，系统响应变慢→重试增多→系统更慢→最终崩溃
2. **资源利用率低**：16核32G的机器只跑了30%的负载，但响应已经很慢
3. **定位困难**：不知道是CPU瓶颈、内存瓶颈还是IO瓶颈

## 2. 项目设计——剧本式交锋对话

**小胖**（望着Grafana上触目惊心的CPU曲线）：大师，大促那天Azkaban差点崩了。200个Flow挤压在凌晨0点同时触发，Web Server CPU直接飙到90%！

**大师**：性能调优需要系统地排查瓶颈。我给你一个"五步调优法"：

1. **负载画像**：先搞清楚是什么导致了高峰——是200个Flow同时调度？还是某个大Job占用资源？
2. **瓶颈定位**：用`top`、`jstat`、MySQL慢查询日志确定瓶颈是CPU、内存还是IO
3. **参数调优**：调整JVM参数、MySQL连接池、线程池等
4. **代码优化**：如果是频繁的热点代码，考虑缓存或异步化
5. **验证效果**：压测验证优化效果

**小白**：那Azkaban常见的性能瓶颈有哪些？

**大师**（列出清单）：

| 瓶颈区域 | 典型症状 | 根因 | 优化方向 |
|---------|---------|------|---------|
| MySQL连接池 | 请求排队、超时 | 连接数不够 | 增大pool size |
| JVM GC | CPU高、间歇卡顿 | 堆太小或GC算法不对 | 调整-Xmx和GC算法 |
| 线程池 | Job提交延迟 | 线程数不够 | 增大executor线程池 |
| 日志IO | 磁盘util 100% | 日志写太频繁 | 异步日志、降低级别 |
| Flow文件解析 | 上传慢 | 重复解析 | 缓存解析结果 |

**小胖**：那JVM参数怎么调？我现在是默认的。

**大师**（写下推荐配置）：

```bash
# Executor JVM优化
-Xms4g -Xmx8g                          # 堆大小：与物理内存匹配
-XX:+UseG1GC                           # 使用G1 GC（适合大堆）
-XX:MaxGCPauseMillis=200               # G1暂停目标200ms
-XX:ParallelGCThreads=8                # GC并行线程数
-XX:ConcGCThreads=2                    # 并发GC线程数
-XX:InitiatingHeapOccupancyPercent=45  # G1开始并发标记的堆占用阈值
-XX:+PrintGCDetails                    # 输出GC日志
-XX:+PrintGCDateStamps
-Xloggc:/opt/azkaban/logs/gc.log

# Web Server JVM优化（侧重点不同）
-Xms2g -Xmx4g
-XX:+UseG1GC
-XX:MaxGCPauseMillis=100               # Web对延迟更敏感
```

**小白**：MySQL怎么调？

**大师**：Azkaban中最繁忙的查询是`execution_flows`表的状态更新和查询。关键优化：


```bash
#!/bin/bash
# benchmark.sh —— Azkaban性能基准测试

AZKABAN_URL="http://localhost:8081"

TOTAL_REQUESTS=100

echo "=== Azkaban性能基准测试 ==="

      -X POST "$AZKABAN_URL" \

    if [ $((i % CONCURRENT_USERS)) -eq 0 ]; then
        wait
    fi

# 统计
AVG=$(awk '{sum+=$1} END {printf "%.3f", sum/NR}' /tmp/login_times.txt)
P99=$(sort -n /tmp/login_times.txt | tail -n +$(( TOTAL_REQUESTS * 99 / 100 )) | head -1)
echo "  平均延迟: ${AVG}s"
echo "  P99延迟: ${P99}s"

for i in $(seq 1 20); do

    curl -s -b /tmp/cookie \
      -X POST "$AZKABAN_URL/executor?ajax=executeFlow" \
      --data "project=benchmark&flow=test_flow_${i}" > /dev/null
    END=$(date +%s%N)
    echo "scale=3; ($END - $START) / 1000000000" | bc
```

```python
#!/usr/bin/env python3
# monitor_connection_pool.py

import requests
import time

JMX_URL = "http://localhost:8081/jmx"

def get_pool_metrics():
    """获取连接池指标"""
    resp = requests.get(JMX_URL)
    data = resp.json()
    
    for bean in data.get("beans", []):
        name = bean.get("name", "")
        if "HikariPool" in name:
            active = bean.get("ActiveConnections", 0)
            idle = bean.get("IdleConnections", 0)
            total = bean.get("TotalConnections", 0)
            waiting = bean.get("ThreadsAwaitingConnection", 0)
            
            print(f"连接池状态:")
            print(f"  活跃连接: {active}")
            print(f"  空闲连接: {idle}")
            print(f"  总连接数: {total}")
            print(f"  等待线程: {waiting}")
            
            usage = (active / total * 100) if total > 0 else 0
            print(f"  使用率: {usage:.1f}%")
            
            if waiting > 0:
                print(f"  ⚠️  有 {waiting} 个线程在等待连接！")
            if usage > 80:
                print(f"  ⚠️  连接池使用率超过80%，建议扩容")

if __name__ == '__main__':
    get_pool_metrics()
```

**分析GC日志的Python脚本**：

```python
#!/usr/bin/env python3
# gc_analyzer.py —— GC日志分析

import re
import sys
from collections import defaultdict

def analyze_gc_log(log_file):
    """分析GC日志"""
    stats = {
        "young_gc": {"count": 0, "total_time_ms": 0},
        "full_gc": {"count": 0, "total_time_ms": 0},
        "max_pause_ms": 0,
    }
    
    with open(log_file, 'r') as f:
        for line in f:
            # [GC pause (G1 Evacuation Pause) (young), 0.0234567 secs]
            match = re.search(r'\[GC.*?(\d+\.\d+)\s*secs\]', line)
            if match:
                pause_ms = float(match.group(1)) * 1000
                
                if "young" in line:
                    stats["young_gc"]["count"] += 1
                    stats["young_gc"]["total_time_ms"] += pause_ms
                elif "Full" in line or "mixed" in line.lower():
                    stats["full_gc"]["count"] += 1
                    stats["full_gc"]["total_time_ms"] += pause_ms
                
                if pause_ms > stats["max_pause_ms"]:
                    stats["max_pause_ms"] = pause_ms
    
    print("=== GC 分析报告 ===")
    print(f"Young GC: {stats['young_gc']['count']}次, "
          f"平均{stats['young_gc']['total_time_ms']/max(stats['young_gc']['count'],1):.1f}ms, "
          f"总耗时{stats['young_gc']['total_time_ms']:.0f}ms")
    print(f"Full GC: {stats['full_gc']['count']}次, "
          f"总耗时{stats['full_gc']['total_time_ms']:.0f}ms")
    print(f"最大暂停: {stats['max_pause_ms']:.1f}ms")
    
    if stats["full_gc"]["count"] > 0:
        print("⚠️  存在Full GC，建议增大堆内存或检查内存泄漏")
    
    return stats
```

## 4. 项目总结

### 调优清单速查

| 优化项 | 配置 | 效果 |
|--------|------|------|
| JVM堆大小 | -Xmx8g | 减少Full GC |
| GC算法 | G1GC | 降低暂停时间 |
| MySQL池大小 | 200 | 减少等待 |
| 慢查询索引 | 复合索引 | 查询提速10x |
| 线程池 | 80 | 提高并发 |
| 日志级别 | WARN | 减少IO |

### 适用场景

- **适用**：Flow数>200的大规模集群、有严格SLA的生产环境、大促/活动期间的高并发场景
- **不适用**：小规模部署（默认配置足够）、开发环境

### 注意事项

- 增大JVM堆可能带来更长的GC暂停，需权衡
- MySQL索引优化在生产环境需要评估影响
- 调优不是一次性工作——随着负载增长需要持续优化
- 每次变更后都要压测验证，避免"负优化"

### 常见踩坑经验

1. **Xmx设得太大**：32G堆用了G1GC，但region太多导致元数据开销大。建议：单台机器的Xmx不超过物理内存的60%。
2. **MySQL连接池开得太大**：`mysql.numconnections=500`导致MySQL `max_connections`被撑爆。建议：单台Azkaban的连接池≤MySQL max_connections/4。
3. **启用query_cache反而变慢**：Azkaban的表更新频繁，query cache的失效开销大于命中收益。建议：关闭query cache。
