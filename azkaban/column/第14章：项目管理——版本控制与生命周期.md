# 第14章：项目管理——版本控制与生命周期

## 1. 项目背景

### 业务场景

数据团队维护着一个包含30个Job的复杂ETL项目。上个月，一个新人对Flow做了优化，上传了新版本。结果第二天调度系统异常——一个看似无关的Job被误删除了，但因为项目没有版本历史，也没人记得原来的Flow长什么样。

运维翻了半天Git日志才找到旧版的`.job`文件，手动恢复了删掉的Job，折腾了整整一个上午。更让人后怕的是，这次故障只影响了非核心报表，如果是核心数据管道，后果不堪设想。

### 痛点放大

没有有效的项目生命周期管理时：

1. **不可回滚**：新版本Flow有Bug，但无法一键回滚到上次正常运行的版本。
2. **僵尸项目**：一年前废弃的测试项目还留在系统里，占用资源且新人误点执行。
3. **依赖断裂**：删除了一个项目，却不知道其他项目的Flow依赖了它的输出。
4. **配置漂移**：同一个项目的Flow在三个环境（dev/test/prod）的配置不一致，导致"测试环境正常，生产环境报错"。

## 2. 项目设计——剧本式交锋对话

**小胖**（心急如焚）：大师，江湖救急！我上传了新版的Flow，结果那个新人把"数据清洗"Job的名字改了，下游10个Job的dependsOn全断了。怎么回到旧版本啊？

**大师**（冷静地）：Azkaban的项目管理有三个关键能力你要知道：

1. **版本历史**：每次上传新的zip包，Azkaban会保留旧版本。在Project页面的"History"标签中能找到。
2. **项目克隆**：可以对一个项目做全量克隆，适合做"测试版本 → 生产版本"的迁移。
3. **项目删除/归档**：废弃的项目可以删除，但建议先导出备份。

**小白**：那Azkaban的"版本"具体指什么？是每次上传的zip包吗？

**大师**：对。Azkaban的版本管理是以"上传事件"为单位的。每次上传zip包，会生成一个新的版本号。旧版本的文件不会被立即删除——但这里有个容易踩的坑：

**Azkaban不会像Git那样做增量存储，每次上传都是全量覆盖。** 旧版本的`.job`文件虽然保存在数据库中，但如果你传了新的zip包，旧版本中独有的文件会被"逻辑删除"。

**小胖**：那怎么恢复旧版本呢？

**大师**：

```
版本恢复流程：
1. 进入Project → History
2. 选择一个历史上传记录
3. 下载当时的zip包
4. 重新上传这个zip包（会生成新版本号）
```

或者你可以从Git仓库中恢复——Azkaban本身不提供"一键回滚到版本N"的功能。

**小白**：那有什么最佳实践吗？怎么防止这类问题再发生？

**大师**：给你五条"金科玉律"：

1. **Git是真相源**：所有的`.job`文件和`.flow`文件必须放在Git仓库中管理，Azkaban只是执行引擎。
2. **上传前做diff**：修改Flow后，先`git diff`确认变更。
3. **Tag标记生产版本**：每次上传到Azkaban生产环境后，在Git中打一个tag（如`v2.3.0-2025-01-15`）。
4. **上传前验证**：在上传到生产Azkaban前，先在测试环境验证Flow能跑通。
5. **项目生命周期规划**：定义"开发→测试→灰度→生产→归档→删除"的标准流程。

**小胖**：那这些环节在不同环境之间怎么保持一致？比如测试环境的参数和生产环境肯定不一样……

**大师**：这就是"配置外提"的价值。核心思想是：**Flow的Job逻辑在所有环境中都一样，只有参数随环境变化**。

```
项目结构示例（Git仓库）：
etl_pipeline/
├── flows/                    # Flow定义文件（所有环境共用）
│   ├── daily_report.flow
│   ├── data_sync.flow
│   └── user_profile.flow
├── jobs/                     # Job定义文件（所有环境共用）
│   ├── mysql_export.job
│   ├── spark_etl.job
│   └── hive_load.job
├── config/                   # 环境相关配置（每个环境不同）
│   ├── dev.properties
│   ├── test.properties
│   └── prod.properties
└── scripts/                  # Shell/Python脚本
    ├── export_mysql.sh
    └── data_processor.py
```

上传到Azkaban时，根据目标环境选择对应的properties文件。这样Flow逻辑保持一致，配置按环境隔离。

### 技术映射总结

- **版本历史** = 文档的修改记录（可以看到"谁在什么时候改了哪里"）
- **Git + Azkaban** = 图书馆书库（Git是真迹，Azkaban是展出的副本）
- **配置外提** = 手机的主题模式（核心功能不变，换个皮肤和环境）
- **项目生命周期** = 产品的上架流程（设计→测试→发布→运营→下架）

#### 步骤2：上传新版本并标记

**目标**：在Azkaban上传Flow并建立版本关联。

```bash
#!/bin/bash
# deploy_to_azkaban.sh —— 自动化部署到Azkaban

ENV=${1:-dev}  # dev/test/prod
AZKABAN_URL=${AZKABAN_URL:-http://localhost:8081}
PROJECT_NAME="etl_pipeline"
VERSION=$(date +%Y%m%d-%H%M%S)

echo "=== 部署 ETL Pipeline 到 Azkaban ($ENV) ==="
echo "版本: $VERSION"

# 1. Git打标签
git tag "deploy-${ENV}-${VERSION}"
echo "[1/5] Git tag: deploy-${ENV}-${VERSION}"

# 2. 创建部署包
TEMP_DIR=$(mktemp -d)
cp flows/*.flow $TEMP_DIR/
cp jobs/*.job $TEMP_DIR/
cp scripts/*.sh $TEMP_DIR/

# 复制环境配置作为project.properties
cp config/${ENV}.properties $TEMP_DIR/project.properties

# 打包（确保zip根目录就是文件）
DEPLOY_ZIP="deploy-${ENV}-${VERSION}.zip"
cd $TEMP_DIR && zip -r /tmp/${DEPLOY_ZIP} ./*
echo "[2/5] 打包完成: ${DEPLOY_ZIP}"

# 3. 登录Azkaban
curl -c /tmp/azkaban.cookie \
  -X POST "${AZKABAN_URL}" \
  --data "action=login&username=${AZK_USER}&password=${AZK_PASS}"
echo "[3/5] 已登录Azkaban"

# 4. 确保项目存在（如不存在则创建）
curl -b /tmp/azkaban.cookie \
  -X POST "${AZKABAN_URL}/manager?action=create" \
  --data "name=${PROJECT_NAME}&description=ETL Pipeline (${ENV})" \
  2>/dev/null

# 5. 上传Flow
UPLOAD_RESULT=$(curl -s -b /tmp/azkaban.cookie \
  -X POST "${AZKABAN_URL}/manager?project=${PROJECT_NAME}&ajax=upload" \
  -F "file=@/tmp/${DEPLOY_ZIP}")

echo "[4/5] 上传结果: ${UPLOAD_RESULT}"

# 6. 验证
if echo "$UPLOAD_RESULT" | grep -q "error"; then
    echo "[5/5] ❌ 部署失败！"
    echo "错误信息: $UPLOAD_RESULT"
    # 移除失败标签
    git tag -d "deploy-${ENV}-${VERSION}"
else
    echo "[5/5] ✓ 部署成功！版本: ${VERSION}"
    echo "部署时间: $(date)"
    echo "Azkaban URL: ${AZKABAN_URL}/manager?project=${PROJECT_NAME}"
fi

# 清理临时文件
rm -rf $TEMP_DIR /tmp/${DEPLOY_ZIP}
```

**使用**：

```bash
# 部署到测试环境
./deploy_to_azkaban.sh test

# 部署到生产环境
./deploy_to_azkaban.sh prod
```

```python
#!/usr/bin/env python3
# rollback_project.py

import requests
import json
import sys

def rollback_to_version(base_url, project, version, session):
    """回滚到指定版本"""
    
    # 1. 获取历史版本列表
    resp = requests.get(
        f"{base_url}/manager",
        params={"project": project, "ajax": "fetchProjectLogs"},
        cookies={"azkaban.browser.session.id": session}
    )
    history = resp.json()
    
    # 2. 找到目标版本
    target = None
    for entry in history:
        if entry.get("version") == version:
            target = entry
            break
    
    if not target:
        print(f"版本 {version} 不存在")
        return False
    
    print(f"准备回滚到版本 {version}，上传人: {target['uploader']}")
    print(f"包含文件: {', '.join(target['files'])}")
    
    # 3. 确认操作
    confirm = input("确认回滚? (yes/no): ")
    if confirm != "yes":
        print("已取消")
        return False
    
    # 4. 下载对应版本的zip（需要从Git或其他备份获取）
    # Azkaban不直接提供下载，所以应从Git恢复
    print("请从Git恢复到对应tag:")
    print(f"  git checkout deploy-{version}")
    print(f"  然后重新打包上传")
    
    return True

if __name__ == '__main__':
    rollback_to_version("http://localhost:8081", "etl_pipeline", 3, "your_session")
```

#### 步骤4：项目生命周期管理脚本

**目标**：实现项目的创建→活跃→封存→删除的完整生命周期管理。

```bash
#!/bin/bash
# project_lifecycle_manager.sh

AZKABAN_URL="http://localhost:8081"

lifecycle_stage() {
    local PROJECT=$1
    local STAGE=$2  # active/archive/delete

    case $STAGE in
        active)
            echo "激活项目: $PROJECT"
            # 恢复项目的所有调度
            curl -b cookies.txt -X POST "$AZKABAN_URL/schedule" \
              --data "ajax=resumeAllSchedules" \
              --data "project=$PROJECT"
            ;;
        
        archive)
            echo "归档项目: $PROJECT"
            # 1. 暂停所有调度
            curl -b cookies.txt -X POST "$AZKABAN_URL/schedule" \
              --data "ajax=pauseAllSchedules" \
              --data "project=$PROJECT"
            
            # 2. 导出项目配置备份
            curl -b cookies.txt \
              "$AZKABAN_URL/manager?project=$PROJECT&ajax=fetchProjectProperties" \
              > "backup_${PROJECT}_$(date +%Y%m%d).json"
            
            echo "归档备份已保存: backup_${PROJECT}_$(date +%Y%m%d).json"
            ;;
        
        delete)
            echo "⚠️  即将永久删除项目: $PROJECT"
            read -p "确认删除? 输入项目名确认: " confirm
            if [ "$confirm" == "$PROJECT" ]; then
                # 1. 先归档（备份）
                lifecycle_stage "$PROJECT" "archive"
                
                # 2. 删除项目
                curl -b cookies.txt -X POST "$AZKABAN_URL/manager" \
                  --data "delete=true&project=$PROJECT"
                
                echo "项目 $PROJECT 已删除"
            else
                echo "删除已取消"
            fi
            ;;
    esac
}

# 使用示例
# lifecycle_stage "old_etl_pipeline" "archive"
# lifecycle_stage "test_project_2023" "delete"
```

```python
#!/usr/bin/env python3
# build_deploy_package.py —— 构建带环境配置的部署包

import os
import sys
import shutil
import zipfile

def build_package(env, project_dir="."):
    """根据环境构建Azkaban部署包"""
    
    print(f"=== 构建 {env} 环境部署包 ===")
    
    deploy_dir = f"/tmp/azkaban_deploy_{env}"
    zip_path = f"/tmp/etl_pipeline_{env}_{__import__('datetime').datetime.now().strftime('%Y%m%d')}.zip"
    
    # 清理并创建临时目录
    if os.path.exists(deploy_dir):
        shutil.rmtree(deploy_dir)
    os.makedirs(deploy_dir)
    
    # 1. 复制Flow和Job文件（所有环境共用）
    for subdir in ["flows", "jobs", "scripts"]:
        src = os.path.join(project_dir, subdir)
        dst = os.path.join(deploy_dir, os.path.basename(subdir))
        if os.path.exists(src):
            shutil.copytree(src, dst)
    
    # 2. 复制环境对应的配置文件（作为project.properties）
    env_config = os.path.join(project_dir, "config", f"{env}.properties")
    if os.path.exists(env_config):
        shutil.copy(env_config, os.path.join(deploy_dir, "project.properties"))
    
    # 3. 打包为zip
    with zipfile.ZipFile(zip_path, 'w', zipfile.ZIP_DEFLATED) as zf:
        for root, dirs, files in os.walk(deploy_dir):
            for f in files:
                file_path = os.path.join(root, f)
                arcname = os.path.relpath(file_path, deploy_dir)
                zf.write(file_path, arcname)
    
    # 4. 验证zip包结构
    print(f"\n部署包内容:")
    with zipfile.ZipFile(zip_path, 'r') as zf:
        for name in zf.namelist():
            print(f"  {name}")
    
    print(f"\n✓ 部署包已生成: {zip_path}")
    return zip_path

if __name__ == '__main__':
    env = sys.argv[1] if len(sys.argv) > 1 else "dev"
    build_package(env)
```

## 4. 项目总结

### 版本管理方案对比

| 维度 | Azkaban内置 | Git + Azkaban | Git + CI/CD |
|------|-----------|--------------|------------|
| 版本回滚 | ★☆☆ 手动下载重传 | ★★☆ Git回退后上传 | ★★★ 自动化 |
| 变更追溯 | ★★☆ 上传记录 | ★★★ 完整Git历史 | ★★★ 完整历史 |
| 多人协作 | ★☆☆ 无分支概念 | ★★★ Git分支/PR | ★★★ CI门禁 |
| 环境一致性 | ★☆☆ 手动维护 | ★★☆ properties分环境 | ★★★ 自动打包 |

### 适用场景

- **适用**：多环境部署的核心数据流水线、需要变更追溯的合规项目、多人协作的大项目
- **不适用**：一次性的临时调试Flow、单环境的简单ETL任务

### 注意事项

- Azkaban的项目"删除"是物理删除，不可恢复，操作前务必导出备份
- 项目克隆不会复制调度配置（Schedule需在新项目中重建）
- 每次上传zip包都会清空该项目的所有文件并替换为新包内容
- Git中的tag务必与Azkaban的版本号对应，减少追踪成本

### 常见踩坑经验

1. **删项目忘关调度**：直接在Azkaban中删除了项目，但对应的调度计划没有自动取消，导致下次Cron触发时报错。解决：删除项目前先用API暂停所有调度。
2. **Git仓库与Azkaban不一致**：有人绕过Git直接在Azkaban Web界面修改并上传Flow，导致Git仓库中找不到最新的变更。解决：强制要求变更流程，关闭Web上传（仅API上传）。
3. **多环境properties混用**：开发环境的`prod.properties`文件和测试环境的搞混了，导致测试环境连上了生产数据库。解决：在properties文件中加环境标识注释，自动化部署脚本做校验。

### 思考题

1. 如何实现Azkaban项目的"灰度发布"——新版本Flow先只让10%的调度周期使用，如果连续3次成功则全量切换，否则自动回滚？
2. 如果你的Azkaban中有100个项目，如何快速扫描出所有"3个月内无任何执行记录"的僵尸项目，并生成待清理列表？
