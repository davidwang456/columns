# 第13章：权限管理与多租户隔离

## 1. 项目背景

### 业务场景

某公司数据平台组将Azkaban开放给公司内三个业务团队使用：电商团队、风控团队和财务团队。每个团队有自己的ETL流水线和调度计划。

开放使用一周后，噩梦开始了：

- 电商团队的开发同学不小心删除了风控团队的一个核心Flow，导致风控今天的数据中断4小时。
- 财务团队的Job调度占用了大量Executor资源，导致电商团队凌晨的批处理超时。
- 新来的实习生用admin账号登录Azkaban，不小心把整个"核心日报"Flow的调度给暂停了——所有人都以为任务在跑，实际上已经停了3天。

### 痛点放大

权限管理不到位时：

1. **越权操作**：任何人都可以修改、删除、暂停他人的Flow和调度。
2. **账号共享**：10个人共用一个admin账号，无法追溯是谁做出了某个操作。
3. **资源争抢**：所有团队的Job在同一个Executor池中争抢，没有资源隔离。
4. **数据泄露风险**：不同团队的Job能看到彼此的配置参数，包括数据库密码等敏感信息。

## 2. 项目设计——剧本式交锋对话

**小胖**（气喘吁吁地跑进来）：大师，出大事了！风控组的老王刚才打电话骂我，说我把他的Flow删了。但我发誓我没动过他的项目，是不是系统出Bug了？

**大师**（皱眉）：不是Bug。我查了一下操作日志，确实是用你的账号删的——但你那个账号有ADMIN权限，你可能是误点了。

**小胖**：啊？我怎么会有ADMIN权限？我就是一个写SQL的数据开发啊……

**大师**：这就是问题所在。现在你们全组都在用同一个admin账号。Azkaban默认只有一个`azkaban/azkaban`管理员账号，权限系统需要额外配置。

**小白**：Azkaban的权限模型是什么样的？跟数据库的RBAC（基于角色的访问控制）一样吗？

**大师**（在白板上画图）：

```
Azkaban权限体系：
┌─────────────────────────────────────────────┐
│  用户 (User)                                  │
│    ↓ 赋予                                    │
│  角色 (Role)                                  │
│    ├── ADMIN     可以管理所有项目、用户、系统配置  │
│    ├── READER     只能查看项目内容               │
│    ├── WRITER     可以编辑、上传Flow              │
│    └── EXECUTOR   可以手动执行Flow               │
│                                                  │
│  Group (用户组)                                  │
│    ├── 电商组 → 只读电商项目、可写电商项目         │
│    ├── 风控组 → 只读风控项目、可写风控项目         │
│    └── 财务组 → 只读财务项目、可写财务项目         │
│                                                  │
│  每个Project可以独立配置哪些用户/组有什么权限      │
└─────────────────────────────────────────────┘
```

**小胖**：那怎么限制每个团队的资源使用呢？我发现财务组的一个Spark任务，开了50个executor，把我们的Spark任务全挤掉了。

**大师**：这涉及到"多租户资源隔离"，Azkaban原生支持有限。你需要从两个层面做隔离：

1. **Azkaban层面**：不同团队分配不同的Executor Server，物理隔离。
2. **Yarn层面**：不同团队使用不同的Yarn队列（`spark.yarn.queue`），资源配额由Yarn控制。

**小白**：那用户认证怎么办？难道每个人都要在Azkaban里手动建账号？

**大师**：你可以对接公司的身份认证系统。Azkaban支持两种外部认证：

1. **LDAP**：对接公司AD/LDAP，用户用域账号登录。
2. **OAuth2**：对接SSO系统，走统一认证。

```xml
<!-- conf/azkaban-users.xml（XML认证方式） -->
<azkaban-users>
    <user username="admin" password="admin" roles="admin" groups="admin"/>
    <user username="zhangsan" password="pass123" roles="writer"/>
    <user username="lisi" password="pass456" roles="reader"/>
    
    <user username="zhangsan" groups="ecommerce_team"/>
    <user username="lisi" groups="risk_control_team"/>
</azkaban-users>
```

但生产环境不推荐XML方式——密码明文存储，且无法和公司统一认证打通。推荐LDAP。

**小胖**：那操作审计呢？怎么知道谁在什么时候做了什么？

**大师**：Azkaban的审计能力比较基础，主要通过两个渠道：

1. **Project Logs**：每个项目有操作日志，记录了谁上传了Flow、谁修改了调度等。
2. **azkaban.log**：系统日志记录了所有的API调用、登录、权限变更。

对于详细审计需求（比如"谁在什么时候看了哪个Job的日志"），通常需要把Azkaban的accesslog接入ELK或Splunk来分析。

### 技术映射总结

- **RBAC** = 公司门禁系统（不同员工刷工卡能进入不同区域）
- **Group隔离** = 部门隔离（市场部的人不能随便进出研发部的实验室）
- **LDAP/OAuth** = 统一门禁卡（一张卡刷全公司，不用每扇门配一把钥匙）
- **Yarn队列隔离** = 食堂窗口分流（财务窗口、开发窗口、测试窗口，各有各的配额）

## 3. 项目实战

### 3.1 环境准备

Azkaban运行中，准备3个测试项目分别模拟不同团队的场景。

### 3.2 分步实现

#### 步骤1：创建多用户与角色

**目标**：为不同团队创建独立用户账号。

**方式A：XML文件认证**（适合测试环境）

```xml
<!-- conf/azkaban-users.xml -->
<azkaban-users>
    <!-- 管理员 -->
    <user username="admin" password="admin" roles="admin" groups="admin"/>
    
    <!-- 电商团队成员 -->
    <user username="wangxiaoming" password="xm123" roles="writer"/>
    <user username="lixiaohong" password="xh456" roles="executor"/>
    
    <!-- 风控团队成员 -->
    <user username="zhaoda" password="zd789" roles="writer"/>
    
    <!-- 财务团队成员 -->
    <user username="sunxiao" password="sx111" roles="reader"/>
    
    <!-- 运维 -->
    <user username="ops_admin" password="ops789" roles="admin"/>
</azkaban-users>
```

在`azkaban.properties`中启用XML认证：

```properties
user.manager.class=azkaban.user.XmlUserManager
user.manager.xml.file=conf/azkaban-users.xml
```

**方式B：通过API创建项目级别权限**（推荐，无需重启）

```bash
#!/bin/bash
# setup_project_permissions.sh

BASE_URL="http://localhost:8081"
ADMIN_SESSION="xxx"

# 为电商项目添加权限
echo "=== 配置电商项目权限 ==="

# 添加wangxiaoming为WRITER
curl -X POST "$BASE_URL/manager" \
  --data "ajax=addProjectUserPermission" \
  --data "project=ecommerce_etl" \
  --data "user=wangxiaoming" \
  --data "permission=WRITER" \
  -b azkaban.browser.session.id=$ADMIN_SESSION

# 添加电商组
curl -X POST "$BASE_URL/manager" \
  --data "ajax=addProjectGroupPermission" \
  --data "project=ecommerce_etl" \
  --data "group=ecommerce_team" \
  --data "permission=EXECUTOR" \
  -b azkaban.browser.session.id=$ADMIN_SESSION

# 为风控项目添加权限
echo "=== 配置风控项目权限 ==="
curl -X POST "$BASE_URL/manager" \
  --data "ajax=addProjectUserPermission" \
  --data "project=risk_control" \
  --data "user=zhaoda" \
  --data "permission=WRITER" \
  -b azkaban.browser.session.id=$ADMIN_SESSION

echo "权限配置完成"
```

#### 步骤2：权限验证实验

**目标**：通过对比实验验证各角色权限边界。

| 操作 | ADMIN | WRITER | EXECUTOR | READER |
|------|-------|--------|----------|--------|
| 查看项目 | ✓ | ✓ | ✓ | ✓ |
| 上传Flow | ✓ | ✓ | ✗ | ✗ |
| 执行Flow | ✓ | ✓ | ✓ | ✗ |
| 创建调度 | ✓ | ✓ | ✗ | ✗ |
| 删除项目 | ✓ | ✓ | ✗ | ✗ |
| 修改权限 | ✓ | ✗ | ✗ | ✗ |
| 管理用户 | ✓ | ✗ | ✗ | ✗ |
| 查看全局调度 | ✓ | ✗ | ✗ | ✗ |

**验证脚本**：

```python
#!/usr/bin/env python3
# perm_test.py —— 权限边界测试

import requests

USERS = {
    "admin": {"session": "admin_session_id", "role": "ADMIN"},
    "wangxiaoming": {"session": "xm_session_id", "role": "WRITER"},
    "lixiaohong": {"session": "xh_session_id", "role": "EXECUTOR"},
    "sunxiao": {"session": "sx_session_id", "role": "READER"},
}

BASE_URL = "http://localhost:8081"

def test_permission(user_key, action):
    """测试指定用户能否执行某个操作"""
    session = USERS[user_key]["session"]
    
    tests = {
        "view_project": f"{BASE_URL}/manager?project=ecommerce_etl",
        "execute_flow": f"{BASE_URL}/executor?ajax=executeFlow&project=ecommerce_etl&flow=daily_report",
        "upload_flow": f"{BASE_URL}/manager?project=ecommerce_etl&ajax=upload",
    }
    
    url = tests.get(action)
    if not url:
        return "UNKNOWN"
    
    try:
        resp = requests.get(url, cookies={"azkaban.browser.session.id": session})
        if resp.status_code == 200 and "error" not in resp.text:
            return "ALLOWED"
        else:
            return "DENIED"
    except:
        return "ERROR"

# 执行权限矩阵验证
print("=== Azkaban 权限矩阵验证 ===\n")
print(f"{'操作':<20}", end="")
for user in USERS:
    print(f"{user:<15}", end="")
print()

for action in ["view_project", "execute_flow", "upload_flow"]:
    print(f"{action:<20}", end="")
    for user in USERS:
        result = test_permission(user, action)
        print(f"{result:<15}", end="")
    print()
```

#### 步骤3：配置LDAP认证

**目标**：对接公司AD/LDAP实现统一认证。

```properties
# conf/azkaban.properties —— LDAP配置
user.manager.class=azkaban.user.LdapUserManager

# LDAP服务器
user.manager.ldap.host=ldap.company.com
user.manager.ldap.port=389
user.manager.ldap.use.ssl=false

# 用户搜索
user.manager.ldap.user.base.dn=ou=people,dc=company,dc=com
user.manager.ldap.user.id.property=uid

# 组映射（LDAP组 → Azkaban角色）
user.manager.ldap.group.base.dn=ou=groups,dc=company,dc=com
user.manager.ldap.group.mapping=data_team=admin,data_dev=writer

# 绑定用户（用于搜索）
user.manager.ldap.bind.account=cn=admin,dc=company,dc=com
user.manager.ldap.bind.password=bind_password
```

#### 步骤4：多租户资源隔离（Yarn队列）

**目标**：通过Yarn队列让不同团队的Spark任务使用隔离资源。

```bash
# ecommerce_spark.job —— 电商团队Spark任务
type=spark
spark.master=yarn
spark.yarn.queue=ecommerce  # 电商专属队列
spark.num.executors=10
spark.executor.memory=4g
# ... 其他配置 ...

# risk_spark.job —— 风控团队Spark任务
type=spark
spark.master=yarn
spark.yarn.queue=risk_control  # 风控专属队列
spark.num.executors=8
spark.executor.memory=6g
# ... 其他配置 ...
```

**Yarn队列配置**（capacity-scheduler.xml）：

```xml
<property>
    <name>yarn.scheduler.capacity.root.queues</name>
    <value>default,ecommerce,risk_control,finance</value>
</property>

<property>
    <name>yarn.scheduler.capacity.root.ecommerce.capacity</name>
    <value>40</value>  <!-- 电商团队占40% -->
</property>

<property>
    <name>yarn.scheduler.capacity.root.risk_control.capacity</name>
    <value>30</value>  <!-- 风控团队占30% -->
</property>

<property>
    <name>yarn.scheduler.capacity.root.finance.capacity</name>
    <value>20</value>  <!-- 财务团队占20% -->
</property>
```

#### 步骤5：操作审计日志收集

**目标**：将Azkaban操作日志集中收集分析。

```bash
# 在azkaban.properties中启用详细访问日志
jetty.request.log=true

# 配置access log格式
jetty.accesslog.format=%{client}a - %u %t "%r" %s %O %Dms
```

**审计脚本**：

```bash
#!/bin/bash
# audit.sh —— 从系统日志中提取关键操作

LOG_FILE="/opt/azkaban/logs/azkaban.log"

echo "=== Azkaban 操作审计报告 ==="
echo "报告时间: $(date)"
echo ""

echo "--- 登录记录（最近3天）---"
grep "login" $LOG_FILE | tail -20

echo ""
echo "--- Flow操作记录 ---"
grep -E "upload|delete.*flow|schedule.*flow|executeFlow" $LOG_FILE | tail -20

echo ""
echo "--- 权限变更记录 ---"
grep -E "add.*Permission|remove.*Permission|change.*Role" $LOG_FILE | tail -10

echo ""
echo "--- 异常操作（ERROR） ---"
grep "ERROR" $LOG_FILE | tail -20
```

## 4. 项目总结

### 认证方式对比

| 维度 | XML认证 | LDAP认证 | OAuth2/SSO |
|------|---------|---------|-----------|
| 配置复杂度 | ★☆☆ 极低 | ★★☆ 中等 | ★★★ 较高 |
| 密码安全 | ★☆☆ 明文 | ★★★ 加密 | ★★★ Token |
| 用户同步 | ★☆☆ 手动 | ★★☆ 自动 | ★★★ 自动 |
| 与公司SSO打通 | ★☆☆ 不支持 | ★★☆ 部分 | ★★★ 完全 |
| 适用环境 | 开发/测试 | 生产环境 | 生产(推荐) |

### 适用场景

- **适用**：多团队共用Azkaban的生产环境、有合规审计需求的公司、需要细粒度权限控制的团队
- **不适用**：单团队的小型数据平台、纯开发/测试环境

### 注意事项

- XML认证方式中密码明文存储，绝不能用于生产
- 修改权限配置后，已登录用户需重新登录才能生效
- `azkaban.properties`中的`default.timezone.id`需与LDAP服务器时区一致
- 移除用户权限时，该用户正在执行的Job不会被中断

### 常见踩坑经验

1. **LDAP连接失败但Azkaban仍启动**：LDAP配置错误不会阻止Azkaban启动，但所有用户都无法登录。建议部署LDAP前先确认网络可达性。
2. **权限缓存**：修改了XML用户文件但未重启Azkaban，新用户仍无法登录。Azkaban启动时一次性加载XML。
3. **跨项目可见性**：默认情况下，所有用户可以看到所有项目的存在（但内部内容不可见）。如需完全隐藏，需修改Azkaban源码。
