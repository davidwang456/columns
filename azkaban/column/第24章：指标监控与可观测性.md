# 第24章：指标监控与可观测性

## 1. 项目背景

### 业务场景

运维团队接到一个任务：老板要求"数据平台的可观测性达到99%"。解释一下就是：
- 平台出了任何问题，必须在1分钟内被发现
- 每个Flow的执行成功率、耗时要实时可查
- 各个Executor的CPU/内存/线程池使用率要可视化

当前团队的做法是：每天上班手工查看Azkaban Web界面，看有没有红色失败标记。这显然不满足"1分钟发现"的目标。他们需要一套完整的监控体系——指标采集、可视化大盘、阈值告警。

### 痛点放大

可观测性不足时：

1. **盲飞**：不知道集群真实健康状态，只有当用户投诉了才知道有问题
2. **无法定位瓶颈**：Executor慢了，是CPU高还是线程池满还是网络延迟？
3. **容量规划靠猜**：不知道目前用了多少比例的资源，扩容的时候该加几台机器
4. **历史趋势丢失**：一个月前的性能数据无从查起

## 2. 项目设计——剧本式交锋对话

**小胖**（被老板叫去办公室）：大师，老板要求"1分钟发现平台故障"。我们现在靠人工看网页，响应时间至少15分钟起。怎么办？

**大师**：你需要构建三层监控体系——**指标采集、可视化、告警**。

**小白**：Azkaban自带什么监控指标吗？

**大师**：Azkaban内置了几个关键指标：

1. **JMX指标**：通过JMX暴露JVM和业务指标（内存、线程、Flow数）
2. **stub_status**（Solo模式）：`/status`端点返回基础状态
3. **数据库查询**：从execution_flows表统计成功率、耗时

但这些指标都很基础。你需要把它们接入一个专业的监控系统来做聚合、可视化和告警。

**小胖**：推荐什么方案？

**小白**：那我需要采集哪些具体指标？

**大师**：四类"黄金指标"：

| 类别 | 指标 | 含义 | 目标 |
|------|------|------|------|
| 吞吐量 | 每日执行Flow数 | 系统负载 | 趋势观察 |
| 延迟 | P50/P99 Flow耗时 | 执行效率 | P99 < 目标值 |
| 错误率 | Flow失败率 | 稳定性 | < 1% |
| 饱和度 | Executor CPU/线程池使用率 | 资源利用率 | < 80% |

### 技术映射总结

- **Prometheus** = 体检中心（定时给你抽血化验各项指标）
- **Grafana** = 体检报告可视化（把化验结果画成曲线图）
- **AlertManager** = 体检异常报警（某项指标超标、立刻通知你）
- **黄金指标** = 生命体征（脉搏、血压、体温——判断健康的核心参数）

### 3.2 分步实现

#### 步骤1：自定义Azkaban Exporter

**目标**：编写Prometheus Exporter暴露Azkaban核心指标。

```python
#!/usr/bin/env python3
# azkaban_exporter.py —— Azkaban Prometheus Exporter

import time
import json
import requests
from prometheus_client import start_http_server, Gauge, Counter, Histogram, REGISTRY

class AzkabanExporter:
    """Azkaban Prometheus指标采集器"""
    
    def __init__(self, azkaban_url, session_id):
        self.base_url = azkaban_url.rstrip('/')
        self.session = session_id
        
        # ===== 定义指标 =====
        # 执行状态指标
        self.flow_status_gauge = Gauge(
            'azkaban_flow_status',
            'Flow execution status (1=SUCCESS, 0=FAILED/RUNNING)',
            ['project', 'flow', 'status']
        )
        
        # Flow耗时指标
        self.flow_duration_seconds = Histogram(
            'azkaban_flow_duration_seconds',
            'Flow execution duration in seconds',
            ['project', 'flow'],
            buckets=[60, 300, 600, 1800, 3600, 7200]
        )
        
        # 执行计数
        self.flow_execution_total = Counter(
            'azkaban_flow_execution_total',
            'Total number of flow executions',
            ['project', 'flow', 'status']
        )
        
        # Executor指标
        self.executor_count = Gauge(
            'azkaban_executor_count',
            'Number of active executors'
        )
        
        self.executor_flow_count = Gauge(
            'azkaban_executor_flow_count',
            'Number of running flows per executor',
            ['executor_id', 'host']
        )
        
        # 调度指标
        self.schedule_count = Gauge(
            'azkaban_schedule_count',
            'Number of active schedules'
        )
        
        # JVM指标（从JMX获取）
        self.jvm_heap_used = Gauge(
            'azkaban_jvm_heap_used_bytes',
            'JVM heap memory used'
        )
        self.jvm_threads = Gauge(
            'azkaban_jvm_threads_current',
            'Current number of JVM threads'
        )
    
    def collect_metrics(self):
        """采集所有指标"""
        try:
            self._collect_executor_metrics()
            self._collect_schedule_metrics()
            self._collect_execution_metrics()
            self._collect_jvm_metrics()
        except Exception as e:
            print(f"Metric collection error: {e}")
    
    def _collect_executor_metrics(self):
        """采集Executor指标"""
        resp = requests.get(
            f"{self.base_url}/executor",
            params={"ajax": "fetchallexecutors"},
            cookies={"azkaban.browser.session.id": self.session}
        )
        data = resp.json()
        executors = data.get("executors", [])
        
        active_count = 0
        for executor in executors:
            if executor.get("status") == "ACTIVE":
                active_count += 1
                self.executor_flow_count.labels(
                    executor_id=str(executor.get("id")),
                    host=executor.get("host", "unknown")
                ).set(executor.get("runningFlows", 0))
        
        self.executor_count.set(active_count)
    
    def _collect_schedule_metrics(self):
        """采集调度指标"""
        resp = requests.get(
            f"{self.base_url}/schedule",
            params={"ajax": "fetchAllScheduledFlows"},
            cookies={"azkaban.browser.session.id": self.session}
        )
        data = resp.json()
        self.schedule_count.set(len(data.get("items", [])))
    
    def _collect_execution_metrics(self):
        """采集最近的执行指标"""
        # 获取最近20条执行记录
        resp = requests.get(
            f"{self.base_url}/executor",
            params={"ajax": "fetchexecflowhistory", "start": 0, "length": 20},
            cookies={"azkaban.browser.session.id": self.session}
        )
        data = resp.json()
        
        for exec_info in data.get("executions", []):
            status = exec_info.get("status", "UNKNOWN")
            project = exec_info.get("projectName", "unknown")
            flow = exec_info.get("flowId", "unknown")
            
            self.flow_execution_total.labels(
                project=project, flow=flow, status=status
            ).inc()
            
            duration = (exec_info.get("endTime", 0) - exec_info.get("startTime", 0)) / 1000
            if duration > 0:
                self.flow_duration_seconds.labels(
                    project=project, flow=flow
                ).observe(duration)
    
    def _collect_jvm_metrics(self):
        """采集JVM指标"""
        try:
            resp = requests.get(
                f"{self.base_url}/jmx",
                cookies={"azkaban.browser.session.id": self.session}
            )
            data = resp.json()
            
            for bean in data.get("beans", []):
                if "java.lang:type=Memory" in bean.get("name", ""):
                    self.jvm_heap_used.set(bean.get("HeapMemoryUsage", {}).get("used", 0))
                
                if "java.lang:type=Threading" in bean.get("name", ""):
                    self.jvm_threads.set(bean.get("ThreadCount", 0))
        except:
            pass  # JMX可能不可用

if __name__ == '__main__':
    exporter = AzkabanExporter(
        "http://localhost:8081",
        "your_session_id_here"
    )
    
    # 启动HTTP server暴露/metrics端点
    start_http_server(9100)
    print("Azkaban Exporter started on port 9100")
    
    # 定期采集
    while True:
        exporter.collect_metrics()
        time.sleep(30)
```

#### 步骤4：Prometheus告警规则

**目标**：配置关键告警规则。

```yaml
# azkaban_alerts.yml
groups:
  - name: azkaban_critical
    rules:
      # 规则1：Flow成功率过低
      - alert: AzkabanFlowSuccessRateLow
        expr: |
          sum(rate(azkaban_flow_execution_total{status="SUCCEEDED"}[15m])) /
          sum(rate(azkaban_flow_execution_total[15m])) < 0.9
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "Azkaban Flow 成功率低于90%"
          description: "最近15分钟Flow成功率仅为{{ $value | humanizePercentage }}"
      
      # 规则2：Executor全部离线
      - alert: AzkabanNoActiveExecutor
        expr: azkaban_executor_count == 0
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "Azkaban 没有活跃的Executor！"
          description: "所有Executor已离线，无法执行任何Flow"
      
      # 规则3：活跃Executor数减少
      - alert: AzkabanExecutorCountDecreased
        expr: azkaban_executor_count < 2
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Azkaban Executor 数量不足"
          description: "当前只有 {{ $value }} 个活跃Executor"
      
      # 规则4：JVM内存使用过高
      - alert: AzkabanHighHeapUsage
        expr: azkaban_jvm_heap_used_bytes > 7 * 1024 * 1024 * 1024
        for: 10m
        labels:
          severity: warning
        annotations:
          summary: "Azkaban JVM 堆内存使用超过7GB"
          description: "当前堆内存使用 {{ $value | humanize }}B，接近上限"
      
      # 规则5：单个Executor负载过高
      - alert: AzkabanExecutorHighLoad
        expr: azkaban_executor_flow_count > 20
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Executor({{ $labels.executor_id }}) 运行中的Flow超过20个"
          description: "可能接近线程池上限"
```

## 4. 项目总结

### 监控方案对比

| 维度 | 人工巡检 | Prometheus + Grafana | 商业APM |
|------|---------|---------------------|---------|
| 发现延迟 | 15分钟+ | < 1分钟 | < 1分钟 |
| 可视化 | Web界面 | 自定义大盘 | 丰富模板 |
| 历史数据 | 无 | 15天-数年 | 数年 |
| 成本 | 免费 | 免费 | 昂贵 |
| 学习曲线 | ★☆☆ | ★★★ | ★★☆ |

### 适用场景

- **适用**：生产环境Azkaban集群、需要SLA保障的企业级数据平台、多团队共享的Azkaban服务
- **不适用**：个人开发环境、单机Solo Server的简单使用

### 注意事项

- Prometheus对时序数据的磁盘消耗较大（约1-2字节/样本），注意配置retention
- 自定义Exporter需要处理Azkaban session过期问题
- 告警规则的`for`子句非常重要——避免因瞬时抖动产生告警风暴
- JMX端点在生产环境中建议增加认证

### 常见踩坑经验

1. **Prometheus拉取失败**：Azkaban Exporter的端口被防火墙拦截。解决：开放Exporter端口到Prometheus服务器。
2. **histogram的分位数不准确**：`histogram_quantile(0.99, ...)`需要足够的bucket配置，否则P99会偏差很大。
3. **AlertManager重复告警**：`repeat_interval`设得太短，同一个故障每30分钟发一次告警。合理配置静默和重复间隔。
