# 第11章：日志系统与问题定位

## 1. 项目背景

### 业务场景

凌晨3点，运维小王被报警电话吵醒——"用户画像构建"Flow失败了，下游所有依赖的报表都没生成。他睡眼惺忪地登录Azkaban Web界面，看到红彤彤的失败标记。

点开失败Job的日志，只有一行：`ERROR: Job failed with exit code 1`。没有堆栈信息，没有错误原因，没有上下文。他需要知道：是SQL查询超时？还是HDFS磁盘满了？还是上游数据没到齐？

他不得不ssh到Azkaban服务器，从`/opt/azkaban/logs/`目录中翻找更详细的执行日志。等他终于定位到问题（磁盘空间不足），已经过去了40分钟——而如果日志系统设计得好，这本该是5分钟就能解决的问题。

### 痛点放大

日志系统不完善时：

1. **日志太简略**：只记录了"失败了"，没有失败原因。
2. **日志太多**：一个Job跑3小时产生了几万行日志，失败信息埋在海量的INFO日志中。
3. **日志分散**：Azkaban日志、Job执行日志、Hadoop/Spark集群日志散落在不同系统中，无法串联。
4. **日志丢失**：H2数据库重启后历史日志丢失，Azkaban的日志保留机制不清楚。

## 2. 项目设计——剧本式交锋对话

**小胖**（对着界面上的红色失败标记发呆）：大师，这个Job又失败了，但日志就一行"exit code 1"，我完全不知道什么地方出了错……

**大师**：这就是经典的"日志不足"问题。Azkaban的日志分为两个层面：

1. **Azkaban自身日志**（azkaban.log）：记录了调度、执行、资源管理的系统级信息。
2. **Job执行日志**：每条`command`的stdout和stderr，这是你在Web界面看到的。

"exit code 1"只是告诉你"进程退出码不是0"，具体原因需要看Job输出的stderr。

**小白**：那为什么有的Job失败后有详细的堆栈，有的就一行"exit 1"呢？

**大师**（解释）：这取决于你的脚本怎么写。一个粗心的脚本可能是这样的：

```bash
python important_task.py
```

如果`important_task.py`抛异常后`sys.exit(1)`，stdout和stderr中可能没有任何错误信息就直接退出了。

正确的写法应该捕获异常并输出到stderr：

```bash
python important_task.py 2>&1 || {
    echo "FATAL: important_task.py failed with code $?" >&2
    exit 1
}
```

**小胖**：那日志太多怎么办？我看过一些Job跑3个小时，日志有5万行，浏览器都加载不出来了……

**大师**：这种时候你需要"日志分层"策略。我的建议是：

- **INFO级别**：记录关键步骤（如"开始数据导出""已导出100万行"）
- **WARNING级别**：记录非关键异常（如"某个分区无数据，跳过"）
- **ERROR级别**：记录致命异常（如"数据库连接失败"）

在Azkaban中，虽然Job日志不分级别，但你可以把脚本的输出设计为三层结构。

**小白**：我怎么区分Azkaban的系统日志和Job执行日志？

**大师**（在白板上画）：

```
日志体系：
┌─────────────────────────────────────────────┐
│ Azkaban系统日志 (azkaban.log)                │
│   位置: conf/log4j.properties 配置的路径      │
│   内容: Web服务请求、调度触发、执行器状态      │
│   容量: 通常较大，需要logrotate               │
├─────────────────────────────────────────────┤
│ Job执行日志                                  │
│   位置: MySQL execution_logs表 / H2文件       │
│   内容: Job的stdout/stderr                   │
│   保留: 受azkaban.log.retention.days控制     │
├─────────────────────────────────────────────┤
│ 外部系统日志                                  │
│   位置: HDFS / Yarn / Spark History Server    │
│   内容: MR/Spark作业详细日志                  │
│   保留: 各系统独立配置                        │
└─────────────────────────────────────────────┘
```

**小胖**：那故障排查的标准流程应该是怎样的？

**大师**：我给你总结一个"故障排查五步法"：

1. **看状态**：Flow状态 → 哪个Job失败了？（绿色=成功，红色=失败）
2. **看日志**：失败Job → 点击查看日志 → 搜索"ERROR""FATAL""Exception"
3. **看时间线**：Job开始时间、结束时间，附近时间段的系统变更（是否有部署、扩容？）
4. **看资源**：Executions页面 → 查看同时间段其他Flow是否也失败了？（可以判断是"点故障"还是"面故障"）
5. **看外部**：检查上游系统（数据源是否可访问？Hadoop集群是否健康？）

### 技术映射总结

- **exit code = 1** = 收银台的错误指示灯亮了，但没说明是一毛没找到还是收银机坏了
- **stderr vs stdout** = 警报广播（stderr）vs 日常播报（stdout）——关键时刻听警报
- **log retention** = 监控录像保留天数（超时自动覆盖）
- **故障排查五步法** = 医生问诊流程（望闻问切 + 化验报告）

#### 步骤1：配置Azkaban系统日志

**目标**：定制Azkaban自身的日志输出。

`conf/log4j.properties`：

```properties
# Azkaban系统日志配置
log4j.rootLogger=INFO, Console, File

# 控制台输出
log4j.appender.Console=org.apache.log4j.ConsoleAppender
log4j.appender.Console.layout=org.apache.log4j.PatternLayout
log4j.appender.Console.layout.ConversionPattern=%d{yyyy-MM-dd HH:mm:ss} %-5p [%t] %c{1}:%L - %m%n

# 文件输出（主日志）
log4j.appender.File=org.apache.log4j.RollingFileAppender
log4j.appender.File.File=logs/azkaban.log
log4j.appender.File.MaxFileSize=100MB
log4j.appender.File.MaxBackupIndex=10
log4j.appender.File.layout=org.apache.log4j.PatternLayout
log4j.appender.File.layout.ConversionPattern=%d{ISO8601} %-5p [%t] %c{2}:%L - %m%n

# 关键类的日志级别
log4j.logger.azkaban.executor=INFO
log4j.logger.azkaban.scheduler=DEBUG
log4j.logger.azkaban.webapp=WARN
log4j.logger.azkaban.jobtype=INFO
```

#### 步骤4：编写日志分析脚本

**目标**：自动从日志中提取关键信息。

```python
#!/usr/bin/env python3
# log_analyzer.py —— 分析Azkaban Job日志

import re
import sys
from collections import Counter

def analyze_job_log(log_text):
    """分析Job日志，提取关键信息"""
    
    analysis = {
        'total_lines': 0,
        'errors': [],
        'warnings': [],
        'duration': None,
        'status': 'UNKNOWN',
        'key_metrics': {}
    }
    
    lines = log_text.strip().split('\n')
    analysis['total_lines'] = len(lines)
    
    for line in lines:
        # 提取错误
        if re.search(r'(ERROR|FATAL|Exception|failed)', line, re.IGNORECASE):
            analysis['errors'].append(line.strip())
        
        # 提取警告
        if re.search(r'WARN(ING)?', line, re.IGNORECASE):
            analysis['warnings'].append(line.strip())
        
        # 提取状态
        if 'SUCCESS' in line or 'completed successfully' in line:
            analysis['status'] = 'SUCCESS'
        elif 'FAILED' in line or 'exit code' in line:
            analysis['status'] = 'FAILED'
        
        # 提取关键指标
        match = re.search(r'Total rows:\s*(\d+)', line)
        if match:
            analysis['key_metrics']['rows'] = int(match.group(1))
        
        match = re.search(r'Duration:\s*(\d+)', line)
        if match:
            analysis['key_metrics']['duration_seconds'] = int(match.group(1))
    
    # 错误分类
    error_types = Counter()
    for err in analysis['errors']:
        if 'Connection' in err or 'connect' in err:
            error_types['Connection Error'] += 1
        elif 'timeout' in err.lower():
            error_types['Timeout'] += 1
        elif 'Permission' in err or 'denied' in err:
            error_types['Permission Error'] += 1
        elif 'memory' in err.lower() or 'heap' in err.lower():
            error_types['Memory Error'] += 1
        else:
            error_types['Unknown Error'] += 1
    
    analysis['error_types'] = dict(error_types)
    
    return analysis

def print_report(analysis):
    """打印分析报告"""
    print("=" * 60)
    print("           Azkaban Job 日志分析报告")
    print("=" * 60)
    print(f"  总行数:    {analysis['total_lines']}")
    print(f"  执行状态:  {analysis['status']}")
    print(f"  错误数:    {len(analysis['errors'])}")
    print(f"  警告数:    {len(analysis['warnings'])}")
    
    if analysis['key_metrics']:
        print(f"\n  关键指标:")
        for k, v in analysis['key_metrics'].items():
            print(f"    - {k}: {v}")
    
    if analysis['errors']:
        print(f"\n  错误详情（前5条）:")
        for err in analysis['errors'][:5]:
            print(f"    ❌ {err[:100]}")
    
    if analysis['error_types']:
        print(f"\n  错误类型分布:")
        for etype, count in analysis['error_types'].items():
            bar = '█' * min(count, 20)
            print(f"    {etype:<20}: {bar} {count}")
    
    print("=" * 60)

if __name__ == '__main__':
    if len(sys.argv) > 1:
        with open(sys.argv[1], 'r', encoding='utf-8') as f:
            log_text = f.read()
    else:
        log_text = sys.stdin.read()
    
    analysis = analyze_job_log(log_text)
    print_report(analysis)
```

## 4. 项目总结

### 日志能力对比

| 维度 | Azkaban原生 | Azkaban + ELK | 自研日志系统 |
|------|-----------|--------------|------------|
| 日志搜索 | ★☆☆ 仅Web界面 | ★★★ 全文检索 | ★★☆ 定制搜索 |
| 日志聚合 | ★☆☆ 分散存储 | ★★★ 集中收集 | ★★★ 集中存储 |
| 保留策略 | ★★☆ 基础清理 | ★★★ 索引生命周期 | ★★★ 灵活策略 |
| 可视化 | ★☆☆ 纯文本 | ★★★ Kibana | ★★☆ 自建大盘 |
| 运维成本 | ★☆☆ 零成本 | ★★★ 需维护ELK | ★★★ 开发+维护 |

### 适用场景

- **适用**：中小规模团队（10-50人）、日志量每天<10GB、故障排查SOP已建立的团队
- **不适用**：大规模集群（100+ Executor，日日志量100GB+）、需要全文检索或多维度关联分析的场景

### 注意事项

- H2数据库下的Job日志在服务重启后可能丢失，生产环境务必使用MySQL
- `azkaban.log.retention.days`设得太小会导致故障回溯困难，建议≥30天
- 定期清理`execution_logs`表，该表增长极快（每个Job每次执行都有一条记录）
- 日志中不要输出密码、Token等敏感信息

### 常见踩坑经验

1. **Web界面日志加载超时**：Job日志超过10MB时，浏览器可能卡死。解决：用API分页获取日志，`offset`和`length`参数控制。
2. **日志编码乱码**：Windows环境下编辑的.sh脚本包含GBK编码的中文注释，上传到Linux服务器后日志乱码。解决：统一使用UTF-8编码。
3. **H2数据库锁**：Solo Server模式下，查看日志时H2数据库被锁定，无法同时执行其他Flow。解决：升级到多Executor模式+MySQL。

2. 如果需要对最近7天内所有失败Job的日志进行汇总分析，提取Top 10失败原因。如何实现这个需求？
