# 第10章：告警通知——邮件与失败回调

## 1. 项目背景

### 业务场景

某电商公司的数据团队有一个"核心日报生成"Flow，每天凌晨2点自动运行。上周二凌晨，因为上游日志采集延迟，Flow中"数据汇聚"Job跑了3小时后超时失败。直到早上9点运营部同事在周会上提问"昨天的GMV数据呢？"，数据团队才意识到日报根本没生成。

开发负责人被严厉批评——"凌晨失败的任务，为什么到早上9点都没人知道？"更糟糕的是，这不是第一次——之前还发生过"数据导出"Job因为磁盘空间不足静默失败，整整三天数据缺失没人发现。

### 痛点放大

没有有效的告警机制时：

1. **盲区运行**：凌晨任务失败后，直到有人上班主动查看Azkaban界面，才可能发现——最严重的情况是任务失败一整个周末都没人知道。
2. **告警风暴**：配置了邮件告警后，一个任务失败重试3次，每次都发一封邮件；加上它阻塞了下游10个Job也失败，邮件箱瞬间被50封邮件淹没。
3. **告警不可操作**：只通知"任务失败了"，但为什么失败？错误日志在哪？应该联系谁？邮件中都没有信息。
4. **告警疲劳**：每天收到几十封"Presto连接超时——自动恢复"的不重要告警，真正重要的告警被淹没在邮件堆里。

## 2. 项目设计——剧本式交锋对话

**小胖**（双手合十）：大师，救救我！这周又漏了一个失败任务，运营总监发了封抄送CEO的邮件。我该怎么搞告警？

**大师**（严肃地）：告警是数据平台的生命线。没有告警的数据平台就像没有警报器的核电站——看着安静，实际上随时可能出大事。

Azkaban提供了两个层面的告警能力：

1. **Job级别的告警**：某个Job成功或失败时发邮件
2. **Flow级别的SLA告警**：根据时间约束发告警（比如"Flow必须在凌晨5点前跑完"）

**小白**：这两个有什么区别？什么时候用哪个？

**大师**（拿出一张纸画图）：

```
Job告警（细粒度）：                   SLA告警（时间维度）：
┌──────────────────────┐         ┌──────────────────────┐
│ job_data_export       │         │ Flow: daily_report   │
│  failure.emails=...   │         │   SLA: finish by 5am │
│  success.emails=...   │         │   如果5点还没跑完     │
│                       │         │   触发告警            │
│  "只要这个Job挂了"     │         │   "不管哪个Job卡住了"  │
│  就告警               │         │   只要整体超时就告警    │
└──────────────────────┘         └──────────────────────┘
```

Job告警适合关键Job的精确监控；SLA告警适合"流程整体健康度"监控。

**小胖**：那邮件通知怎么配置？我在Jetty服务上是不是还要装个SMTP？

**大师**：不需要额外装SMTP。Azkaban本身集成了Java Mail，你只需要在`azkaban.properties`中配置一个已存在的邮件服务器地址即可。可以对接公司内部的SMTP服务器，也可以用QQ邮箱、网易邮箱等第三方服务。

**小白**：那告警邮件的内容能自定义吗？默认的模板挺丑的……

**大师**：Azkaban的邮件模板是可以自定义的。在`conf/`目录下可以找到邮件模板文件。你可以自定义HTML模板，添加CSS样式、公司logo、甚至直接附上失败Job的日志链接。

**小白**：还有一个关键问题——假设一个场景：上游数据延迟了，导致"数据汇聚"Job重试了3次，每次失败都发一封告警。再加上因为"数据汇聚"失败，下游20个Job也失败了……我的邮件箱会爆炸吗？

**大师**：这就是著名的"告警雪崩"问题。Azkaban有几个机制来缓解：

1. **failure.emails vs failure.email**：前者每次都发，后者只在首次失败时发（注意单复数！）
2. **SLA基于Flow维度**：即使50个Job失败，SLA只触发一次告警（因为SLA绑定在Flow上）
3. **自定义阈值**：比如"只有当重试2次后还失败"才告警

但说实话，Azkaban原生的告警能力比较基础。很多团队会在Azkaban外面套一层Prometheus + AlertManager来做更精细的告警管理：聚合、静默、分级、值班表。

**小胖**：那如果邮件服务器挂了怎么办？告警不就也丢了？

**大师**：这就需要对冲方案。推荐的做法是——**邮件告警 + Webhook回调**双通道。Azkaban支持在Flow执行结束后回调一个HTTP接口，你可以对接企业微信、钉钉、Slack等IM工具：

```bash
# execution.options 中配置
failure.action=webhook.call
webhook.url=https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx
```

这样即使邮件服务挂了，IM通知也能送到。当然，最保险的是邮件 + IM + 监控大盘三层防御。

### 技术映射总结

- **failure.emails** = 每失败一次发一条（像唠叨的闹钟，反复提醒）
- **failure.email** = 只发一次（像一次性闹钟，响完就安静了）
- **SLA告警** = 总裁令（不管内部什么原因，外部截止时间到了没交差就问责）
- **Webhook回调** = 传呼机（主动推送关键事件到指定设备）

### 3.2 分步实现

#### 步骤1：配置SMTP邮件服务

**目标**：让Azkaban能发送邮件。

在`conf/azkaban.properties`中添加：

```properties
# ===== 邮件配置 =====
mail.sender=azkaban-alert@company.com
mail.sender.display=Azkaban Alert System
mail.host=smtp.company.com
mail.port=25
mail.user=azkaban-alert@company.com
mail.password=your_mail_password_here

# 是否启用TLS/SSL
mail.tls=true
mail.ssl=false

# 邮件主题前缀
azkaban.mail.subject.prefix=[Azkaban]
```

**使用QQ邮箱示例**（开发/测试环境）：

```properties
mail.sender=your_qq@qq.com
mail.host=smtp.qq.com
mail.port=587
mail.user=your_qq@qq.com
mail.password=授权码  # 不是QQ密码！在QQ邮箱设置中生成授权码
mail.tls=true
```

**验证邮件服务**：

```bash
# 在Azkaban服务器上测试SMTP连通性
echo "EHLO localhost" | nc -w 5 smtp.company.com 25
# 预期输出：250-smtp.company.com Hello ...

# 或使用Python测试发送邮件
python3 -c "
import smtplib
smtp = smtplib.SMTP('smtp.company.com', 25)
smtp.ehlo()
smtp.starttls()
smtp.login('user@company.com', 'password')

```

#### 步骤2：配置Job级别的邮件告警

**目标**：在指定Job失败或成功时发送邮件。

```bash
# critical_job.job —— 带告警配置的关键Job
type=command
command=echo "Starting critical data pipeline..."
command.1=sleep 5
command.2=echo "Processing done."

# ===== 告警配置 =====

# 失败时发送的邮箱（复数形式=每次失败都发）
failure.emails=oncall@company.com,data-lead@company.com

# 成功时发送的邮箱
success.emails=data-team@company.com

# 添加自定义邮件标题
failure.email.subject=[紧急] 核心数据流水线失败

# 重试也告警
notify.failure.first=false   # false=每次失败都发; true=仅首次失败
notify.failure.last=true     # 最后一次失败也发

retries=2
retry.backoff=60000
```

**关键参数说明**：

| 参数 | 说明 | 示例值 |
|------|------|--------|
| `failure.emails` | 失败时告警（每次重试都发） | `admin@co.com` |
| `failure.email` | 失败时告警（仅首次） | 注意单复数！|
| `success.emails` | 成功时告警 | `team@co.com` |
| `failure.email.subject` | 自定义邮件标题 | `[紧急] xxx失败` |
| `notify.failure.first` | 首次失败是否通知 | `true/false` |
| `notify.failure.last` | 最后一次重试失败是否通知 | `true/false` |

#### 步骤4：企业微信Webhook告警集成

**目标**：实现Flow失败时自动发送企业微信消息。

```python
#!/usr/bin/env python3
# wechat_notifier.py —— 企业微信机器人通知脚本

import requests
import json
import sys
import os

def send_wechat_notification(webhook_url, title, content, level='warning'):
    """发送企业微信通知"""
    
    # 根据级别选择颜色
    colors = {
        'info': '#3498db',      # 蓝色
        'warning': '#f39c12',   # 橙色
        'error': '#e74c3c',     # 红色
    }
    
    markdown_content = f"""## {title}
> 项目: <font color="info">{os.environ.get('azkaban.project.name', 'N/A')}</font>
> Flow: <font color="info">{os.environ.get('azkaban.flow.name', 'N/A')}</font>
> 执行ID: <font color="info">{os.environ.get('azkaban.flow.execid', 'N/A')}</font>
> 提交用户: <font color="info">{os.environ.get('azkaban.submit.user', 'N/A')}</font>
> 开始时间: {os.environ.get('azkaban.flow.starttimestamp', 'N/A')}

### 详细内容
{content}

### 快速操作
- [查看执行详情](http://azkaban.company.com/executor?execid={os.environ.get('azkaban.flow.execid', '')})
- [查看Flow日志](http://azkaban.company.com/manager?project={os.environ.get('azkaban.project.name', '')})
"""

    payload = {
        "msgtype": "markdown",
        "markdown": {
            "content": markdown_content
        }
    }
    
    try:
        resp = requests.post(webhook_url, json=payload, timeout=10)
        resp.raise_for_status()
        result = resp.json()
        if result.get('errcode') == 0:
            print(f"[INFO] 企业微信通知发送成功")
        else:
            print(f"[ERROR] 企业微信通知失败: {result}")
        return result.get('errcode') == 0
    except Exception as e:
        print(f"[ERROR] 发送企业微信通知异常: {e}")
        return False

if __name__ == '__main__':
    webhook_url = sys.argv[1] if len(sys.argv) > 1 else os.environ.get('WECHAT_WEBHOOK')
    title = sys.argv[2] if len(sys.argv) > 2 else "Azkaban Flow通知"
    content = sys.argv[3] if len(sys.argv) > 3 else "Flow执行完成"
    send_wechat_notification(webhook_url, title, content)
```

## 4. 项目总结

### 告警通道对比

| 维度 | 邮件 | 企业微信/钉钉 | 短信 | Prometheus+AlertManager |
|------|------|------------|------|------------------------|
| 送达速度 | ★☆☆ 延迟分钟级 | ★★★ 秒级 | ★★☆ 秒级 | ★★★ 秒级 |
| 信息丰富度 | ★★★ 可带图附件 | ★★☆ Markdown | ★☆☆ 纯文本 | ★★★ 可定制 |
| 配置复杂度 | ★★☆ | ★☆☆ | ★★★ | ★★★ |
| 告警聚合 | ★☆☆ 无 | ★☆☆ 无 | ★☆☆ 无 | ★★★ 强聚合 |
| 值班管理 | ★☆☆ 需手工 | ★☆☆ 手工 | ★★☆ | ★★★ 内置 |

### 适用场景

- **适用**：核心数据流水线的失败监控、生产环境的任务健康检查、SLA合规监控、多角色通知（开发+运维+业务）
- **不适用**：非关键性任务（告警太多会疲劳）、测试环境的调试任务

### 注意事项

- `failure.emails`（复数）和`failure.email`（单数）行为完全不同，注意区分
- 邮件密码不要写在`azkaban.properties`中，建议用环境变量注入
- Webhook URL中包含key等敏感信息，不要硬编码在Job文件中，用`env.*`从外部注入
- 避免在Job中直接发送告警（增加Job复杂度），优先使用Azkaban内置告警机制

### 常见踩坑经验

1. **SMTP端口被防火墙拦截**：公司内网可能只开放了25端口的出站流量。尝试切到587端口（STARTTLS）或465端口（SSL）。
2. **QQ邮箱授权码问题**：使用QQ邮箱做告警时，`mail.password`不是QQ登录密码，而是在QQ邮箱设置→账户→SMTP中生成的"授权码"。
3. **告警风暴**：一个Flow有50个Job，全部设置了`failure.emails`，一次数据源不可用导致全部失败——邮件箱收到50封告警。解决：只在根因Job设置告警，或者在Flow级别设置SLA。

### 思考题

1. 如何设计一个"告警升级"机制：Job失败后先通知值班开发，15分钟未处理通知技术Leader，30分钟未处理通知运维总监？
2. 假设你需要为100个Flow配置告警，如果逐个Job配置`failure.emails`，维护成本极高。请设计一种"批量配置"方案，让所有Flow共享统一的告警策略。
