# 第22章：高级调度——多时区与日历调度

## 1. 项目背景

### 业务场景

某跨国电商公司有三大数据中心：上海、新加坡、法兰克福。每个地区的数据团队需要按当地时间执行ETL任务——上海团队凌晨2点（CST）、新加坡凌晨2点（SGT）、法兰克福凌晨2点（CET）。更复杂的是，各地还有不同的节假日：中国春节、新加坡国庆、欧洲圣诞假期，这些天部分报表不需要生成。

Azkaban默认只配一个时区（`default.timezone.id`），当三个团队的Cron调度都在同一个Azkaban中时，时区混乱导致上海团队的任务跑到了新加坡时间、欧洲团队周末被凌晨告警吵醒。

### 痛点放大

多时区/多日历调度不当时：

1. **夏令时灾难**：欧洲夏令时切换那天，任务提前/推迟1小时，有些数据出现了双倍或缺失。
2. **节假日报表空跑**：春节连续7天没人看报表，但批处理每天照常运行，浪费计算资源。
3. **时区计算错误**：一个Cron任务配置了UTC时间但大家以为是本地时间，关键报表晚了8小时。

## 2. 项目设计——剧本式交锋对话

**小胖**（看着满屏的时区错误告警）：大师，欧洲团队投诉说他们的任务在夏令时切换那天全乱了！有的任务跑了两次，有的压根没跑。

**大师**：这是多时区调度的经典难题。让我解释三个核心概念：

1. **Azkaban时区配置**：`default.timezone.id` 是全局默认时区，每个Schedule还可以设置独立的时区。
2. **Cron时区语义**：`0 0 2 * * ?` 在不同时区下含义不同——在CST是北京时间凌晨2点，在CET是欧洲中部时间凌晨2点。
3. **夏令时**：对于用了夏令时的地区（如欧洲CET→CEST），在切换日那天会出现"一天只有23小时"或"一天有25小时"的情况。

**小白**：那为什么我的Schedule设置里明明选了`Europe/Berlin`时区，到了夏令时切换还是出问题了？

**大师**（解释）：问题出在Azkaban使用的Quartz框架对夏令时的处理。Quartz在遇到"不存在的时刻"（夏令时跳跃产生的时间空洞）时，默认行为是跳过。比如：

- 柏林夏令时开始日：凌晨1:59之后直接变成3:00。2:00~2:59这几个时刻不存在。
- 如果你的Cron配了`0 0 2 * * ?`，Quartz会直接跳过这次触发，等价于这天不跑了。

反之，在夏令时结束日（多出一小时），`0 0 2 * * ?`可能会触发两次。

**小胖**：那怎么解决？总不能手工改Cron吧？

**大师**：有几个实用策略：

1. **避开切换时刻**：把Cron触发时间设在凌晨4点以后（避开1:00-3:00的危险区间）。
2. **用UTC时区配本地逻辑**：Cron配UTC时间，在Job内部根据业务时区判断是否执行。
3. **日历调度**：Azkaban支持排除特定日期的调度。

**小白**：日历调度是指什么？

**大师**：Azkaban的Schedule有一个"Excluded Dates"功能——你可以指定某些日期不触发。比如把春节7天、国庆7天加进去，这些天的调度自动跳过。

```
调度日历示例：
Schedule: 每天凌晨2点触发
排除日期:
  - 2025-01-29 ~ 2025-02-04 (春节)
  - 2025-05-01 ~ 2025-05-03 (劳动节)
  - 2025-10-01 ~ 2025-10-07 (国庆)
```

### 技术映射总结

- **时区** = 手表的设置（同一个UTC时间，在不同地方显示不同本地时间）
- **夏令时** = 手动拨钟（春天少睡1小时，秋天多睡1小时）
- **排除日期** = 公司放假通知（公告了今天不上班，闹钟别响）
- **时间空洞** = 不存在的时间（就像没有2月30日）

#### 步骤1：多时区Flow配置

**目标**：三个团队使用不同时区的Schedule。

```bash
#!/bin/bash
# create_multizone_schedules.sh

AZKABAN_URL="http://localhost:8081"

# 上海团队 —— CST (UTC+8)
curl -b cookies.txt -X POST "${AZKABAN_URL}/schedule" \
  --data "ajax=scheduleCronFlow" \
  --data "projectName=shanghai_etl" \
  --data "flow=daily_report" \
  --data "cronExpression=0 0 2 * * ?" \
  --data "scheduleTimezone=Asia/Shanghai"

# 新加坡团队 —— SGT (UTC+8，与上海相同)
curl -b cookies.txt -X POST "${AZKABAN_URL}/schedule" \
  --data "ajax=scheduleCronFlow" \
  --data "projectName=singapore_etl" \
  --data "flow=daily_report" \
  --data "cronExpression=0 0 2 * * ?" \
  --data "scheduleTimezone=Asia/Singapore"

# 法兰克福团队 —— CET/CEST
curl -b cookies.txt -X POST "${AZKABAN_URL}/schedule" \
  --data "ajax=scheduleCronFlow" \
  --data "projectName=frankfurt_etl" \
  --data "flow=daily_report" \
  --data "cronExpression=0 0 4 * * ?" \   # 凌晨4点(避开夏令时空洞)
  --data "scheduleTimezone=Europe/Berlin"
```

#### 步骤2：时区感知的Job脚本

**目标**：在Job内部根据时区做条件判断。

```bash
# timezone_aware.job
type=command
command=bash -c '
# 获取当前时区
SYSTEM_TZ=$(timedatectl 2>/dev/null | grep "Time zone" | awk "{print \$3}" || echo "UTC")
echo "System Timezone: $SYSTEM_TZ"
echo "Local Time: $(date)"

# 检查是否为夏令时
IS_DST=$(python3 -c "
import datetime, pytz
tz = pytz.timezone(\"Europe/Berlin\")
now = datetime.datetime.now(tz)
print(now.dst() != datetime.timedelta(0))
")
echo "Is DST: $IS_DST"

# 根据不同时区采用不同的数据处理参数
case "$SYSTEM_TZ" in
    Asia/Shanghai)
        MARKET="CN"
        CURRENCY="CNY"
        ;;
    Asia/Singapore)
        MARKET="SG"
        CURRENCY="SGD"
        ;;
    Europe/Berlin)
        MARKET="EU"
        CURRENCY="EUR"
        ;;
esac

echo "Processing market: ${MARKET}, currency: ${CURRENCY}"
'
```

#### 步骤3：节假日日历排除

**目标**：配置排除日期列表，让调度自动跳过节假日。

```python
#!/usr/bin/env python3
# holiday_calendar.py —— 管理Azkaban的节假日排除日历

import json
from datetime import date, timedelta

# 中国法定节假日 2025
CHINA_HOLIDAYS_2025 = {
    "元旦":       [(date(2025,1,1), date(2025,1,1))],
    "春节":       [(date(2025,1,28), date(2025,2,4))],
    "清明节":     [(date(2025,4,4), date(2025,4,6))],
    "劳动节":     [(date(2025,5,1), date(2025,5,5))],
    "端午节":     [(date(2025,5,31), date(2025,6,2))],
    "中秋节":     [(date(2025,9,15), date(2025,9,17))],
    "国庆节":     [(date(2025,10,1), date(2025,10,7))],
}

# 欧洲主要节假日
EUROPE_HOLIDAYS_2025 = {
    "Christmas":  [(date(2025,12,24), date(2025,12,26))],
    "New Year":   [(date(2025,12,31), date(2026,1,1))],
    "Easter":     [(date(2025,4,18), date(2025,4,21))],
}

def get_holidays(region="CN", year=2025):
    """获取指定地区和年份的节假日列表"""
    holidays_map = {
        "CN": CHINA_HOLIDAYS_2025,
        "EU": EUROPE_HOLIDAYS_2025,
    }
    
    all_dates = set()
    for holiday_name, ranges in holidays_map.get(region, {}).items():
        for start, end in ranges:
            current = start
            while current <= end:
                all_dates.add(current.strftime("%Y-%m-%d"))
                current += timedelta(days=1)
    
    return sorted(all_dates)

def generate_exclusion_string(region="CN", year=2025):
    """生成Azkaban排除日期字符串"""
    dates = get_holidays(region, year)
    return ",".join(dates)

# 设置到Azkaban Schedule的排除日期
def set_schedule_exclusions(project, flow, region, session):
    """通过API设置Schedule的排除日期"""
    import requests
    
    exclusions = generate_exclusion_string(region)
    url = "http://localhost:8081/schedule"
    
    resp = requests.post(url, cookies={"azkaban.browser.session.id": session}, data={
        "ajax": "setExcludedDates",
        "projectName": project,
        "flowName": flow,
        "excludedDates": exclusions
    })
    
    print(f"设置排除日期: {len(exclusions.split(','))} 天")
    return resp.json()

if __name__ == '__main__':
    print("=== 节假日日历 ===")
    print(f"中国节假日: {generate_exclusion_string('CN')}")
    print(f"欧洲节假日: {generate_exclusion_string('EU')}")
```

#### 步骤4：夏令时安全Job

**目标**：在Job中检测夏令时切换并做对应处理。

```bash
# dst_safe.job —— 夏令时安全的Job
type=command
command=bash -c '
# 检测当前日期是否在夏令时切换窗口
TODAY=$(date +%Y-%m-%d)
YEAR=$(date +%Y)

echo "=== 夏令时安全检查 ==="
echo "日期: $TODAY"
echo "当前系统时区: $(timedatectl | grep "Time zone" | awk "{print \$3}")"

# 检测是否为欧洲夏令时切换日
# 欧洲夏令时：3月最后一个周日开始，10月最后一个周日结束
LAST_SUN_MAR=$(date -d "$YEAR-03-31" +%Y-%m-%d)  # 从3月31日往回到周日
LAST_SUN_OCT=$(date -d "$YEAR-10-31" +%Y-%m-%d)  # 从10月31日往回到周日

# 获取真正的最后一个周日
LAST_SUN_MAR=$(date -d "last sunday $YEAR-04-01" +%Y-%m-%d)
LAST_SUN_OCT=$(date -d "last sunday $YEAR-11-01" +%Y-%m-%d)

echo "3月最后一个周日: $LAST_SUN_MAR"
echo "10月最后一个周日: $LAST_SUN_OCT"

if [ "$TODAY" = "$LAST_SUN_MAR" ]; then
    echo "⚠️  今天是夏令时开始日！凌晨2:00→3:00（少1小时）"
    echo "  [措施] 跳过可能导致重复的数据窗口"
    # UTC时间2:00-2:59的Cron会被跳过
fi

if [ "$TODAY" = "$LAST_SUN_OCT" ]; then
    echo "⚠️  今天是夏令时结束日！凌晨3:00→2:00（多1小时）"
    echo "  [措施] 检查是否有重复数据，如有则去重"
    # UTC时间1:00-1:59的Cron可能触发两次
    # 去重：如果今天的数据已经存在，则跳过
    if hdfs dfs -test -e "/data/daily/${TODAY}/_PROCESSED"; then
        echo "  数据已处理，跳过重复执行"
        exit 0
    fi
fi

echo "✓ 夏令时检查通过，正常执行"
hdfs dfs -touchz "/data/daily/${TODAY}/_PROCESSED"
'

# 业务Job随后执行
command.1=echo "Starting business logic..."
command.2=python3 /opt/etl/process_daily_data.py
```

#### 步骤5：全时区对比脚本

**目标**：对比同一Cron在不同时区下的执行时间。

```python
#!/usr/bin/env python3
# timezone_comparator.py —— 多时区触发时间对比

from croniter import croniter
from datetime import datetime
import pytz

TIMEZONES = [
    "Asia/Shanghai",
    "Asia/Singapore", 
    "Asia/Tokyo",
    "Europe/London",
    "Europe/Berlin",
    "America/New_York",
    "America/Los_Angeles",
    "UTC"
]

def compare_cron_across_timezones(cron_expr, count=5):
    """对比Cron在不同时区下的触发时间"""
    
    print(f"Cron: {cron_expr}")
    print(f"未来 {count} 次触发的本地时间:")
    print("=" * 80)
    
    # 表头
    print(f"{'触发#':<8}", end="")
    for tz_name in TIMEZONES:
        print(f"{tz_name:<20}", end="")
    print()
    print("-" * 80)
    
    # 为每个时区计算触发时间
    trigger_times = {tz: [] for tz in TIMEZONES}
    
    for tz_name in TIMEZONES:
        tz = pytz.timezone(tz_name)
        base = datetime.now(tz)
        cron = croniter(cron_expr, base)
        
        for _ in range(count):
            next_time = cron.get_next(datetime)
            trigger_times[tz_name].append(next_time.strftime("%m-%d %H:%M"))
    
    # 打印
    for i in range(count):
        print(f"{i+1:<8}", end="")
        for tz_name in TIMEZONES:
            print(f"{trigger_times[tz_name][i]:<20}", end="")
        print()

if __name__ == '__main__':
    # 对比每天凌晨2点的Cron
    compare_cron_across_timezones("0 0 2 * * ?")
    
    # 对比工作日上午9点的Cron
    compare_cron_across_timezones("0 0 9 ? * 2-6")
```

## 4. 项目总结

### 多时区方案对比

| 方案 | 复杂度 | 可靠性 | 维护成本 |
|------|--------|--------|---------|
| 多Azkaban实例（每时区一个） | ★★☆ | ★★★ | ★★☆ |
| 单Azkaban + 多时区Schedule | ★☆☆ | ★★☆ | ★☆☆ |
| UTC统一+Job内时区转换 | ★★☆ | ★★★ | ★★☆ |
| 基于日历的调度系统（如Airflow timetable） | ★★★ | ★★★ | ★★★ |

### 适用场景

- **适用**：跨国企业的多数据中心调度、需要避开节假日的批处理、夏令时地区的定时任务
- **不适用**：纯单时区部署、非周期性任务

### 注意事项

- Azkaban的时区配置在Schedule级别生效，但Job执行时的时区取决于Executor所在服务器的系统时区
- 排除日期的字符串格式必须为`yyyy-MM-dd`，多个用逗号分隔
- DST切换当天建议将关键任务触发时间设在UTC时间4:00之后

### 常见踩坑经验

1. **global.properties中的default.timezone.id不生效**：只对没有明确设置时区的Schedule生效，如果Schedule创建时选了时区，以Schedule为准。
2. **中国时区夏令时历史遗留**：`Asia/Shanghai`历史上记录了中国曾使用过夏令时的数据（1986-1991），如果你处理1991年以前的历史数据，时间会有1小时的偏移。
3. **排除日期误配**：在`setExcludedDates` API中传入了`2025-01-01`，但实际上想排除的是2025-01-01到2025-01-03，结果只排除了1月1日这一天的触发。
