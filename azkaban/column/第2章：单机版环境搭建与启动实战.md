# 第2章：单机版环境搭建与启动实战

## 1. 项目背景

### 业务场景

小李是公司新入职的数据开发工程师，第一天报到就被安排了一个任务：在公司开发服务器上部署一套Azkaban调度系统，用于管理团队现有的50个ETL任务。开发环境是一台CentOS 7服务器，8核16G内存。

"这还不简单，看官网文档装一下就行。"小李信心满满。但他很快发现，Azkaban的部署文档对新手并不友好——Gradle编译、配置文件修改、数据库初始化、启动脚本调试……每一步都可能踩坑。

### 痛点放大

没有正确的部署指引时，新手可能会遇到：

1. **构建失败**：Gradle版本不对、依赖下载超时、Java版本不匹配，编译阶段就会报错几十行。
2. **启动闪退**：配置文件少了一个参数、MySQL连接不通、端口被占用，服务启动后立即退出，日志中找不到有效错误信息。
3. **环境混乱**：Java_HOME未正确设置、文件权限不足、防火墙拦截8041端口，运维同学排查半天才发现是基础环境问题。

## 2. 项目设计——剧本式交锋对话

**小胖**（抓耳挠腮，盯着满屏的报错日志）：大师，救命！我搞了三个小时了，这Azkaban死活装不上。Gradle编译到一半就报错，这是什么鬼东西？

**大师**（喝着保温杯里的枸杞茶）：别急，说说你报的什么错？

**小胖**：就是这个——`Failed to download kotlin-stdlib.jar`，还有一堆timeout。

**大师**：这就是典型的"医院没开门，你就要看病"的问题。Azkaban用Gradle构建，Gradle需要从Maven中央仓库下载大量依赖包。在国内直接下载，经常超时。你需要配置一个国内Maven镜像源。

**小白**（在旁边安静地听，突然抬头）：说到Maven镜像，我有个疑问——网上很多人都用Docker部署Azkaban，为什么还要手动编译？直接用镜像不更方便吗？

**大师**：问得好。这就像"买现成的房子"和"自己盖房子"的区别。Docker镜像确实方便，但有三点局限：
1. 官方镜像版本滞后，Azkaban 3.90.0之后官方没有维护Docker镜像；
2. 你需要自定义配置时（比如加插件、改端口），还是要理解容器内部结构；
3. 生产环境中很多公司不允许直接用外网镜像，需要你从源码构建并推送到私有仓库。

**小胖**：那我理解了，但编译真的好慢啊，能不能跳过不必要的东西？

**大师**：当然可以。你有两个选择——`./gradlew build -x test` 跳过测试加速，或者 `./gradlew build installDist -x test -x javadoc` 连文档也不生成。我给你举个例子：

如果把Azkaban的编译比作做饭，"build"是做一桌满汉全席——所有模块、所有测试、所有文档全编译。但你只需要一个"Solo Server"来开发用，加 `-x test` 就等于只做一碗面——核心功能有了，速度还快。

**小白**：那production环境该怎么部署？

**大师**：生产环境部署要分三步走。第一步，基础环境就绪（JDK、MySQL、Gradle）；第二步，编译并分发（Web Server和Executor Server是分离的）；第三步，配置和启动。我们接下来一步步走一遍，你就心里有数了。

### 技术映射总结

- **Gradle** = 建筑工程队（负责把源码"盖"成可运行的程序）
- **镜像源** = 建材市场（国内镜像源=本地建材市场，下载更快）
- **Solo Server** = 样板间（单机版，用于开发测试）
- **Web+Executor分离** = 大公司职能分离（Web管调度，Executor管执行，各司其职）

## 3. 项目实战

### 3.1 环境准备

| 组件 | 版本要求 | 安装方式 |
|------|---------|---------|
| JDK | 1.8.0_202 | yum安装或手工解压 |
| Gradle | 4.10+ | SDKMAN安装或手工解压 |
| MySQL | 5.7.x | yum安装（生产用） |
| 操作系统 | CentOS 7+ / Ubuntu 18+ | — |

### 3.2 分步实现

#### 步骤1：基础环境安装

**目标**：安装JDK和Gradle，配置环境变量。

```bash
# === JDK安装 ===
# 检查是否已安装Java
java -version

# 如未安装，使用yum安装OpenJDK 8
sudo yum install -y java-1.8.0-openjdk java-1.8.0-openjdk-devel

# 验证安装
java -version
# 输出：openjdk version "1.8.0_302"

# 设置JAVA_HOME
echo 'export JAVA_HOME=/usr/lib/jvm/java-1.8.0-openjdk' >> ~/.bashrc
echo 'export PATH=$JAVA_HOME/bin:$PATH' >> ~/.bashrc
source ~/.bashrc

# === Gradle安装 ===
# 下载Gradle 4.10
cd /opt
sudo wget https://services.gradle.org/distributions/gradle-4.10-bin.zip
sudo unzip gradle-4.10-bin.zip
sudo ln -s /opt/gradle-4.10/bin/gradle /usr/local/bin/gradle

# 验证
gradle --version
```

**可能遇到的坑**：如果服务器无法访问外网，需要先在内网仓库或跳板机下载好，再scp传输到服务器。

#### 步骤2：配置Gradle国内镜像

**目标**：加速依赖下载，避免超时。

```bash
# 在用户目录下创建init.gradle
mkdir -p ~/.gradle
cat >> ~/.gradle/init.gradle << 'EOF'
allprojects {
    repositories {
        maven { url 'https://maven.aliyun.com/repository/public' }
        maven { url 'https://maven.aliyun.com/repository/central' }
        maven { url 'https://maven.aliyun.com/repository/gradle-plugin' }
        mavenCentral()
    }
}
EOF
```

**运行结果**：再次执行Gradle命令时，依赖包会从阿里云镜像下载，速度从几个小时降到几分钟。

#### 步骤3：编译Azkaban

**目标**：从源码编译出可运行的安装包。

```bash
# 克隆源码
cd /opt
git clone https://github.com/azkaban/azkaban.git
cd azkaban
git checkout 3.90.0

# 编译全部模块，跳过测试
./gradlew build installDist -x test

# 编译成功后的输出
# BUILD SUCCESSFUL in 8m 42s
# 36 actionable tasks: 36 executed
```

**目录结构**（编译后）：
```
azkaban-solo-server/build/install/azkaban-solo-server/
├── bin/
│   ├── azkaban-solo-start.sh      # 启动脚本
│   ├── azkaban-solo-shutdown.sh   # 停止脚本
│   └── internal/                  # 内部脚本
├── conf/
│   ├── azkaban.properties         # 核心配置文件
│   ├── global.properties          # 全局属性
│   └── log4j.properties           # 日志配置
├── lib/                           # 依赖jar包
└── temp/                          # 运行时临时文件
```

#### 步骤4：配置azkaban.properties

**目标**：根据实际环境修改核心配置。

```properties
# conf/azkaban.properties 关键配置项

# Web服务端口
jetty.port=8081

# 数据库配置（Solo模式默认H2）
database.type=h2
h2.path=./temp/h2

# 时区
default.timezone.id=Asia/Shanghai

# 执行器配置
executor.port=12321
azkaban.use.multiple.executors=true

# 邮件告警（可选）
mail.sender=
mail.host=
mail.port=25
```

**配置说明**：
- `jetty.port`：Web界面访问端口，确保不被防火墙拦截
- `database.type`：开发测试用H2即可，生产务必用MySQL
- `default.timezone.id`：务必设为`Asia/Shanghai`，否则Cron调度会按UTC时间触发，比预期晚8小时

#### 步骤5：启动服务（Solo模式）

**目标**：启动Azkaban Solo Server并验证。

```bash
# 启动
cd /opt/azkaban/azkaban-solo-server/build/install/azkaban-solo-server
bin/azkaban-solo-start.sh

# 输出示例
# Azkaban Solo Server started.
# PID: 28341

# 查看启动日志
tail -f azkaban-solo-server/temp/azkaban.log
```

**常见启动报错及解决**：

| 错误信息 | 原因 | 解决方法 |
|---------|------|---------|
| `Address already in use` | 端口被占用 | `lsof -i:8081` 找到占用进程并关闭 |
| `java.lang.OutOfMemoryError` | 内存不足 | 修改`JAVA_OPTS`增大`-Xmx`参数 |
| `Access denied for user` | MySQL连接被拒 | 检查MySQL用户权限和密码 |
| `Could not find or load main class` | 编译不完整 | 重新执行`./gradlew build installDist` |

#### 步骤6：通过API验证服务状态

**目标**：使用REST API验证Azkaban服务正常运行。

```bash
# 登录并获取session
curl -X POST "http://localhost:8081" \
  --data "action=login&username=azkaban&password=azkaban" \
  -c cookies.txt 2>/dev/null

# 获取所有项目列表（首次应为空）
curl -s "http://localhost:8081/index?ajax=fetchuserprojects" \
  -b cookies.txt | python -m json.tool

# 输出
# {"projects": [], "error": null}
```

#### 步骤7：停止服务

**目标**：正确关闭Azkaban服务。

```bash
bin/azkaban-solo-shutdown.sh
# 输出：Azkaban Solo Server shutdown.

# 如需强制杀死
ps aux | grep azkaban | awk '{print $2}' | xargs kill -9
```

### 3.3 测试验证

```bash
# 编写测试脚本 verify_install.sh
#!/bin/bash
BASE_URL="http://localhost:8081"

echo "=== Azkaban安装验证 ==="

# 1. 检查进程
if pgrep -f azkaban > /dev/null; then
    echo "[PASS] Azkaban进程运行中"
else
    echo "[FAIL] Azkaban进程未运行"
    exit 1
fi

# 2. 检查端口
if ss -tlnp | grep -q 8081; then
    echo "[PASS] 端口8081监听中"
else
    echo "[FAIL] 端口8081未监听"
fi

# 3. 检查Web界面
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" $BASE_URL)
if [ "$HTTP_CODE" = "200" ]; then
    echo "[PASS] Web界面正常返回200"
else
    echo "[FAIL] Web界面返回 $HTTP_CODE"
fi

# 4. 检查登录
RESP=$(curl -s -X POST "$BASE_URL" \
    --data "action=login&username=azkaban&password=azkaban")
if echo "$RESP" | grep -q "session.id"; then
    echo "[PASS] 登录功能正常"
else
    echo "[FAIL] 登录失败"
fi

echo "=== 验证完成 ==="
```

## 4. 项目总结

### 部署方式对比

| 维度 | Solo Server | Multi-Executor模式 | Docker部署 |
|------|------------|-------------------|-----------|
| 适用场景 | 开发/测试 | 生产环境 | 测试/临时环境 |
| 部署难度 | ★☆☆ | ★★★ | ★☆☆ |
| 性能上限 | 低 | 高 | 中等 |
| 高可用 | 不支持 | 支持 | 取决于编排 |
| 数据持久化 | H2文件/Memory | MySQL | 需挂载卷 |

### 适用场景

- **适用**：本地开发调试、功能验证、CI/CD流水线中的集成测试、10人以下团队的轻型调度
- **不适用**：生产级高可用环境、需要MySQL数据库的生产集群、需要执行器隔离的多租户场景

### 注意事项

- 务必修改默认密码`azkaban/azkaban`，生产环境密码长度>16位
- `default.timezone.id`不设好，Cron调度会以UTC时间触发
- Gradle编译缓存路径`~/.gradle/caches/`会持续增长，定期清理

### 常见踩坑经验

1. **时区陷阱**：Cron表达式`0 0 8 * * ?`本该每天早上8点触发，结果下午4点才跑——因为时区是UTC，比北京时间晚8小时。修改`azkaban.properties`中的`default.timezone.id=Asia/Shanghai`。
2. **H2并发限制**：H2在文件模式下只支持单连接，多人同时操作Web界面会导致`Database may be already in use`错误。
3. **内存溢出**：默认JVM内存1G，处理大Flow时容易OOM。在启动脚本中增加`-Xmx4g -Xms2g`。
