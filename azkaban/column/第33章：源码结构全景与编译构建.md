# 第33章：源码结构全景与编译构建

## 1. 项目背景

### 业务场景

数据平台团队收到一个需求：为Azkaban开发一个"大数据组件版本自动检测"的插件——当一个Spark Job执行前，自动检查目标集群的Spark版本是否与Job声明的一致。要想实现这个功能，必须深入理解Azkaban的源码结构、模块划分、插件机制和编译流程。

团队中有两位Java开发者，但对Azkaban的源码完全不熟悉。代码仓150+个文件，不知道该从哪里看起。

### 痛点放大

不熟悉源码结构时：

1. **调试困难**：出了问题不知道哪个模块负责
2. **二次开发缓慢**：一个简单的功能改动可能需要几天摸索
3. **版本升级风险大**：不清楚改了A模块会不会影响B模块

## 2. 项目设计——剧本式交锋对话

**小胖**（盯着IDE里150+个Java文件一脸茫然）：大师，Azkaban源码这么多文件，我该从哪里开始看？

**大师**：跟我之前画的那个路线一样——先宏观后微观。Azkaban的模块结构其实非常清晰：

```
azkaban/
├── azkaban-common/       ← 公共基础库
├── azkaban-web-server/    ← Web服务（调度管理）
├── azkaban-exec-server/   ← 执行器服务（Job执行）
├── azkaban-solo-server/   ← 单机版（Web+Exec合一）
├── azkaban-hadoop-security-plugin/ ← Hadoop安全
├── azkaban-db/            ← 数据库脚本
├── azkaban-spi/           ← 服务接口定义
└── azkaban-plugins/       ← 插件目录
```

**小白**：每个模块的核心类是什么？

**大师**：我给你一张"源码路标图"：

| 模块 | 入口类 | 核心职责 |
|------|--------|---------|
| common | — | Props、Flow、Job数据结构 |
| web-server | AzkabanWebServer | Jetty启动、Schedule管理、API接口 |
| exec-server | AzkabanExecutorServer | Executor启动、FlowRunner、JobRunner |
| solo-server | AzkabanSoloServer | 组合Web+Exec |

### 技术映射总结

- **模块化** = 乐高积木（每块有独立功能，最后拼装起来）
- **Gradle多模块** = 总装车间调度系统（先造零件，再组装）
- **SPI机制** = USB接口标准（只要符合接口规范，插上就能用）

## 3. 项目实战

### 3.1 分步实现

#### 步骤1：编译环境搭建

```bash
# 1. 克隆源码
git clone https://github.com/azkaban/azkaban.git
cd azkaban
git checkout 3.90.0

# 2. 编译全部模块（IntelliJ IDEA导入）
./gradlew clean build installDist -x test

# 3. 导入IDE
# File → Open → 选择 azkaban/ 目录 → Import Gradle project
```

#### 步骤2：关键类依赖关系

```
调用关系图：
AzkabanWebServer.main()
  → AzkabanWebServer.launch()
    → WebServer.setupAndStartServer()
      → TriggerManager(quartzScheduler)
      → ScheduleManager
      → ExecutorManager(executorLoader)

AzkabanExecutorServer.main()
  → AzkabanExecutorServer.launch()
    → FlowRunnerManager.submitFlow()
      → FlowRunner.run()
        → JobRunner.run()
          → ProcessJob.run() // command类型入口
```
