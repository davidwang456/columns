# 第25章：动态参数注入与运行时属性

## 1. 项目背景

### 业务场景

数据团队维护的ETL流水线需要每天处理"昨天"的数据。默认情况下，每个Job内部通过`date -d "yesterday"`获取日期。但遇到以下场景时，静态的参数获取方式就不够用了：

1. **补跑历史数据**：上周末的数据因上游延迟没生成，周一需要补跑周六、周日的数据。此时`date -d "yesterday"`返回的仍是"昨天"而不是"周六"。
2. **A/B测试**：同一个Flow需要在两组不同参数下同时运行（如两组不同的Spark Executor配置），对比性能。
3. **依赖上游输出**：Job B需要用到Job A输出的文件路径——这个路径只有Job A跑完才知道。

### 痛点放大

参数管理不灵活时：

1. **补数据痛苦**：每次补跑都要手动修改.job文件中的日期，跑完再改回来
2. **参数传递困难**：上游Job的输出无法自然地传给下游Job
3. **环境参数泄露**：生产环境的敏感参数通过环境变量注入后，可能在日志中意外打印出来

## 2. 项目设计——剧本式交锋对话

**小胖**（正在第5次手动改.job文件里的日期）：大师，我要补跑上周六到周日的数据，每次都要改`process_date=2025-01-11`然后上传打包，跑完再改回`$(date -d yesterday)`...有没有办法不改文件直接传参数？

**大师**：当然有。在Azkaban中，你可以在执行Flow时通过"Flow Parameters"动态传入参数，覆盖.job文件中的默认值。

**小白**：Flow Parameters能覆盖哪些内容？

**大师**：Flow Parameters可以覆盖.job文件中的任何参数，包括`command`、`dependsOn`、`retries`等——但需注意它不能新增参数，只能覆盖已存在的参数。

**小胖**：那具体怎么操作呢？

**大师**：

```
方式1: Web界面 → 执行Flow → Flow Parameters输入框
方式2: REST API → executeFlow + flowOverride参数
```

```bash
# API方式传入参数
curl -X POST "http://localhost:8081/executor?ajax=executeFlow" \
  --data "project=etl&flow=daily_report" \
  --data "flowOverride[process_date]=2025-01-11" \
  --data "flowOverride[batch_size]=20000"
```

**小白**：那上游Job的输出怎么传给下游？目前Azkaban的Job是独立进程，跑完就没了。

**大师**：这是Azkaban设计上的一个局限——Job之间没有内置的"通道"来传递数据。但有几种变通方案：

1. **文件通道**：上游Job把输出写入一个已知路径的文件，下游Job去读。
2. **数据库通道**：上游Job把输出写入MySQL一个临时表，下游Job去查。
3. **环境变量通道**：利用Azkaban的`env.*`机制，但不跨Job。

我最推荐"文件通道"——它最简单且解耦：

```bash
# 上游Job
command=echo "OUTPUT_PATH=/data/etl/output_2025-01-15" > /tmp/azkaban_ctx.txt

# 下游Job  
command=source /tmp/azkaban_ctx.txt && echo "上游输出: $OUTPUT_PATH"
```

**小胖**：那敏感参数呢？比如数据库密码，我不想在日志里看到。

**大师**：Azkaban 3.80+支持"加密参数"。在Web界面的Flow Parameters中，输入框旁边有一个"锁"图标——点击后该参数的值会被加密存储，且不会在日志中明文显示。

同时在Job脚本中访问时，`${db.password}`会被替换为实际值（执行时解密），但日志中会显示为`********`。这是通过Azkaban的`azkaban.encryption`模块实现的。

### 技术映射总结

- **Flow Parameters** = 快递单上的备注（每次可以写不同的备注，覆盖默认的寄件人信息）
- **文件通道** = 便签条（上游做完事留张便签，下游按便签干活）
- **加密参数** = 保险箱（密码放保险箱里，使用时临时打开，日志只显示"***"）
- **动态注入** = 万能遥控器（不修改电视内部设置，通过遥控器改变频道）

## 3. 项目实战

### 3.1 环境准备

Azkaban运行中，准备支持动态参数的Flow。

### 3.2 分步实现

#### 步骤1：支持动态日期的Job

**目标**：Job可以在默认日期和指定日期之间切换。

```bash
# dynamic_date.job —— 支持动态日期
type=command

# 默认使用昨天，但可以通过Flow Parameters覆盖
env.process_date=${process_date:-$(date -d "yesterday" +%Y-%m-%d)}
env.batch_size=${batch_size:-10000}

command=echo "=== Dynamic Date Job ==="
command.1=echo "Processing date: ${process_date}"
command.2=echo "Batch size: ${batch_size}"
command.3=bash -c '
  echo "Running data pipeline for date: ${process_date}"
  
  # 验证日期格式
  if ! date -d "${process_date}" "+%Y-%m-%d" > /dev/null 2>&1; then
    echo "ERROR: Invalid date format: ${process_date}"
    exit 1
  fi
  
  # 执行实际的数据操作
  echo "Query: SELECT * FROM orders WHERE dt='\'${process_date}\''"
  
  # 将处理日期写入上下文文件（供下游Job使用）
  echo "PROCESS_DATE=${process_date}" > /tmp/azkaban_context.txt
  echo "BATCH_SIZE=${batch_size}" >> /tmp/azkaban_context.txt
'
```

#### 步骤2：补数据脚本

**目标**：批量补跑历史数据。

```bash
#!/bin/bash
# backfill.sh —— 批量补跑历史数据

AZKABAN_URL="http://localhost:8081"
PROJECT="etl_pipeline"
FLOW="daily_report"
START_DATE="2025-01-10"
END_DATE="2025-01-12"

echo "=== 批量补数据 ==="
echo "项目: $PROJECT"
echo "Flow: $FLOW"
echo "范围: $START_DATE ~ $END_DATE"
echo ""

CURRENT_DATE=$START_DATE
while [[ "$CURRENT_DATE" < "$END_DATE" ]] || [[ "$CURRENT_DATE" == "$END_DATE" ]]; do
    echo "[$CURRENT_DATE] 开始补数据..."
    
    # 执行Flow，指定process_date
    RESULT=$(curl -s -b /tmp/azkaban.cookie \
      -X POST "${AZKABAN_URL}/executor?ajax=executeFlow" \
      --data "project=${PROJECT}" \
      --data "flow=${FLOW}" \
      --data "flowOverride[process_date]=${CURRENT_DATE}" \
      --data "concurrentOption=skip")
    
    EXEC_ID=$(echo "$RESULT" | grep -o '"execid":[0-9]*' | grep -o '[0-9]*')
    
    if [ -z "$EXEC_ID" ]; then
        echo "  ✗ 执行失败: $RESULT"
        CURRENT_DATE=$(date -d "$CURRENT_DATE + 1 day" +%Y-%m-%d)
        continue
    fi
    
    echo "  执行ID: $EXEC_ID"
    
    # 等待该日期完成再执行下一天
    echo "  等待完成..."
    while true; do
        STATUS=$(curl -s -b /tmp/azkaban.cookie \
          "${AZKABAN_URL}/executor?execid=${EXEC_ID}&ajax=fetchexecflow" \
          | python3 -c "import json,sys; print(json.load(sys.stdin).get('status','UNKNOWN'))")
        
        if [ "$STATUS" = "SUCCEEDED" ]; then
            echo "  ✓ 完成"
            break
        elif [ "$STATUS" = "FAILED" ] || [ "$STATUS" = "KILLED" ]; then
            echo "  ✗ 失败 ($STATUS)"
            break
        fi
        
        sleep 30
    done
    
    # 移动到下一天
    CURRENT_DATE=$(date -d "$CURRENT_DATE + 1 day" +%Y-%m-%d)
done

echo "=== 补数据完成 ==="
```

#### 步骤3：跨Job参数传递——文件通道

**目标**：实现Job之间的参数传递。

```bash
# upstream_job.job —— 上游Job输出参数
type=command
command=bash -c '
echo "=== Upstream Job ==="

# 模拟数据处理并产出输出路径
OUTPUT_PATH="/data/etl/output_${process_date}"
mkdir -p "$OUTPUT_PATH"

# 执行数据处理...
echo "Data processed to: $OUTPUT_PATH"
echo "Processing 50000 records"
echo "Start time: $(date)"

# 将关键信息写入上下文文件
CTX_FILE="/tmp/azkaban_ctx_${azkaban.flow.execid}.txt"
cat > "$CTX_FILE" << EOF
OUTPUT_PATH=$OUTPUT_PATH
RECORD_COUNT=50000
PROCESS_DATE=${process_date}
START_TIME=$(date -d "today 02:00:00" +%s)000
EOF

echo "Context written to: $CTX_FILE"
'

# downstream_job.job —— 下游Job读取参数
type=command
command=bash -c '
echo "=== Downstream Job ==="

# 读取上游上下文
CTX_FILE="/tmp/azkaban_ctx_${azkaban.flow.execid}.txt"

if [ ! -f "$CTX_FILE" ]; then
    echo "ERROR: Context file not found: $CTX_FILE"
    exit 1
fi

source "$CTX_FILE"
echo "Upstream output path: $OUTPUT_PATH"
echo "Upstream record count: $RECORD_COUNT"
echo "Process date: $PROCESS_DATE"

# 验证上游输出
if [ ! -d "$OUTPUT_PATH" ]; then
    echo "ERROR: Upstream output path does not exist: $OUTPUT_PATH"
    exit 1
fi

echo "Proceeding with downstream processing..."
'
dependsOn=upstream_job
```

#### 步骤4：敏感参数加密

**目标**：保护数据库密码等敏感参数。

```bash
# secure_job.job —— 敏感参数保护
type=command

# 数据库密码不会在日志中明文显示
env.db.password=${db.password}
# 注意：在env.*中引用加密参数

command=bash -c '
echo "=== Secure Job ==="

# 使用加密参数（在日志中会显示为****）
echo "DB Host: ${db.host}"
echo "DB User: ${db.user}"
# 下面的输出会被Azkaban自动屏蔽
echo "DB Password length: ${#db.password}"

# 实际使用（传递到子进程时解密）
mysql -h "${db.host}" -u "${db.user}" -p"${db.password}" \
  -e "SELECT COUNT(*) FROM orders WHERE dt='"${process_date}"'"
'
```

**在azkaban.properties中启用加密**：

```properties
# 加密密钥配置
azkaban.encryption.enabled=true
azkaban.encryption.key.path=/opt/azkaban/conf/encryption.key

# 标记需要加密的参数前缀
azkaban.encryption.param.prefix=password,secret,token,key
```

#### 步骤5：运行时参数验证

**目标**：在执行前验证参数的合法性。

```python
#!/usr/bin/env python3
# param_validator.py —— 运行时参数验证

import re
import sys
from datetime import datetime

class ParamValidator:
    """参数合法性验证器"""
    
    RULES = {
        "process_date": {
            "pattern": r"^\d{4}-\d{2}-\d{2}$",
            "required": True,
            "description": "处理日期，格式: YYYY-MM-DD"
        },
        "batch_size": {
            "type": int,
            "min": 100,
            "max": 1000000,
            "default": 10000,
            "description": "批处理大小"
        },
        "env": {
            "enum": ["dev", "test", "prod"],
            "default": "prod",
            "description": "执行环境"
        },
        "db.host": {
            "pattern": r"^[\w\.-]+$",
            "required": True,
            "description": "数据库主机地址"
        }
    }
    
    @classmethod
    def validate(cls, params):
        """验证参数"""
        errors = []
        warnings = []
        
        for param_name, rule in cls.RULES.items():
            value = params.get(param_name)
            
            # 必填检查
            if rule.get("required") and not value:
                if "default" in rule:
                    params[param_name] = rule["default"]
                    warnings.append(f"{param_name}: 未提供，使用默认值 {rule['default']}")
                else:
                    errors.append(f"{param_name}: 必填参数缺失")
                continue
            
            if value is None:
                continue
            
            # 正则检查
            if "pattern" in rule:
                if not re.match(rule["pattern"], str(value)):
                    errors.append(f"{param_name}: 格式不符合 {rule['pattern']}")
            
            # 类型检查
            if "type" in rule:
                try:
                    value = rule["type"](value)
                except (ValueError, TypeError):
                    errors.append(f"{param_name}: 类型错误，期望 {rule['type'].__name__}")
                    continue
            
            # 范围检查
            if "min" in rule and value < rule["min"]:
                errors.append(f"{param_name}: {value} < 最小值 {rule['min']}")
            if "max" in rule and value > rule["max"]:
                errors.append(f"{param_name}: {value} > 最大值 {rule['max']}")
            
            # 枚举检查
            if "enum" in rule and value not in rule["enum"]:
                errors.append(f"{param_name}: {value} 不在允许值 {rule['enum']} 中")
        
        return errors, warnings

# 在Job的command中调用
# python3 param_validator.py --process_date=2025-01-15 --batch_size=10000 --env=prod
```

## 4. 项目总结

### 参数传递方案对比

| 方案 | 实时性 | 复杂度 | 安全性 | 适用场景 |
|------|--------|--------|--------|---------|
| Flow Parameters API | 即时 | ★☆☆ | ★★☆ | 手动/外部触发 |
| 文件通道 | 准实时 | ★★☆ | ★★☆ | Job间数据传递 |
| 数据库通道 | 准实时 | ★★★ | ★★★ | 大量结构化数据 |
| 加密参数 | 即时 | ★★☆ | ★★★ | 密码等敏感信息 |

### 适用场景

- **适用**：需要补跑历史数据的场景、上线前AB测试、需要外部系统传参的集成
- **不适用**：参数极度简单的固定流程（默认值就够用）

### 注意事项

- Flow Parameters不能新增参数，只能覆盖已存在于.job文件中的参数
- 加密参数的加密密钥务必妥善保管并定期轮换
- 文件通道的临时文件需注意并发冲突（使用exec_id区分）
- Web界面Flow Parameters的加密标记在3.80+版本才支持

### 常见踩坑经验

1. **Flow Parameters未生效**：在.job中写了`env.my_param=default`，API传了`flowOverride[my_param]=new_value`，但日志中仍显示default。原因：Flow Parameters用`${my_param}`引用时才生效，直接`env.my_param=值`不会被覆盖。
2. **文件通道并发冲突**：两个Flow同时运行，都往`/tmp/azkaban_ctx.txt`写文件。解决：使用`${azkaban.flow.execid}`作为文件名后缀。
3. **加密参数解密失败**：加密密钥文件权限为600（只有Azkaban用户可读），但部署时设为644，导致解密时报权限异常。
