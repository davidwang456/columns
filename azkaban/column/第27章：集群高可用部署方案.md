# 第27章：集群高可用部署方案

## 1. 项目背景

### 业务场景

某金融科技公司的数据平台承载着每日的核心风控计算。凌晨2点，Azkaban Web Server所在的服务器因硬件故障宕机——所有调度停止、所有API不可用、所有监控大盘空白。数据团队从2点忙到6点，手动在Executor服务器上执行了所有遗留的Shell脚本，才勉强赶在开盘前完成风控计算。

事后复盘：Azkaban的单点故障导致了4小时的业务中断。CEO要求"数据平台必须具备99.99%的可用性"，也就是全年停机时间不超过53分钟。

### 痛点放大

没有高可用架构时：

1. **Web Server单点**：Web一挂，调度全停，API全断
2. **Executor单点**：唯一的Executor宕机，正在运行的Flow全部中断
3. **MySQL单点**：数据库不可用，所有读写操作全部失败
4. **无自动切换**：故障后全部依赖人工发现和恢复

## 2. 项目设计——剧本式交锋对话

**小胖**（刚从机房出来，满身灰）：大师，昨晚Web Server的硬盘烧了，整个集群瘫痪4小时。CEO现在要求99.99%可用性！Azkaban能做到吗？

**大师**：能，但需要改造。高可用=消除所有单点故障。Azkaban的HA可以从三个层面构建：

**小白**：三个层面？

**大师**（画图）：

```
Azkaban 高可用架构：

┌─────────────────────────────────────────────────────┐
│                    客户端层                           │
│        Nginx (Keepalived VIP) ← 负载均衡 + HA        │
│            ├──────────┼──────────┐                   │
│            ▼          ▼          ▼                   │
│      ┌─────────┐┌─────────┐                       │
│      │Web-1    ││Web-2    │  ← Web Server HA        │
│      │(Active) ││(Standby)│    (双活或主备)          │
│      └────┬────┘└────┬────┘                         │
│           └────┬─────┘                               │
│                │                                     │
│      ┌─────────┼─────────┐                           │
│      ▼         ▼         ▼                           │
│  ┌────────┐┌────────┐┌────────┐                     │
│  │Exec-1  ││Exec-2  ││Exec-3  │ ← Executor HA       │
│  │(Active)││(Active)││(Active)│    (多活，Web自动切换) │
│  └───┬────┘└───┬────┘└───┬────┘                     │
│      └──────────┼─────────┘                           │
│                 │                                     │
│      ┌──────────┴──────────┐                          │
│      ▼                     ▼                          │
│  ┌────────┐           ┌────────┐                     │
│  │MySQL   │←─复制──→ │MySQL   │ ← MySQL HA           │
│  │(Master)│           │(Slave) │    (主从 + 自动切换) │
│  └────────┘           └────────┘                     │
└─────────────────────────────────────────────────────┘
```

**小胖**：Web Server的两个实例怎么协调？会不会两个同时调度同一个Flow？

**大师**：关键——Azkaban的Web Server默认会**互斥**。通过MySQL中的`active_executing_flows`表加锁，保证同时只有一个Web Server在处理调度。Standby节点只接收请求但不处理调度。

但如果你的Web Server也处理API请求，可以配置为"双活"模式，用Nginx做负载均衡：

```nginx
upstream azkaban_web {
    ip_hash;  # session sticky, 同一个用户始终路由到同一台Web
    server web-01:8081 weight=1;
    server web-02:8081 weight=1;
}
```

**小白**：Executor的HA呢？

**大师**：Executor本身就是多活的。Web Server会自动检测Executor的健康状态，如果一个Executor离线，Web会把它上面的Flow重新分配给其他Executor。但这有一个前提——**Flow必须是幂等的**，可以在新的Executor上重跑。

**小胖**：那MySQL的HA呢？

**大师**：MySQL可以用Keepalived+MHA或ProxySQL实现主从自动切换。要点：
1. 一主一从或一主两从
2. 半同步复制，确保数据不丢
3. 自动故障检测和切换

### 技术映射总结

- **Nginx + Keepalived** = 酒店双前台（一个前台歇了，客人自动被引导到另一个）
- **Web Server HA** = 值班经理AB角（一个在岗，一个备勤，自动交接）
- **Executor HA** = 装修队多个工人（一个累了，其他工人接替他未完成的活）
- **MySQL HA** = 账本双备份（主账本每笔都同步给备账本，主账本丢了立马切备）

### 3.1 环境准备

| 组件 | 数量 | 用途 |

### 3.2 分步实现

#### 步骤1：Web Server双实例部署

#### 步骤2：Nginx负载均衡配置

**目标**：配置Nginx实现Web Server的负载均衡和健康检查。

```nginx
# /etc/nginx/conf.d/azkaban.conf

upstream azkaban_web_backend {
    # ip_hash确保同一客户端的请求始终发到同一台Web Server
    ip_hash;
    
    server web-01.company.com:8081 
        weight=1 
        max_fails=3 
        fail_timeout=30s;
    
    server web-02.company.com:8081 
        weight=1 
        max_fails=3 
        fail_timeout=30s;
    
    # 长连接支持（Azkaban有些API是长轮询）
    keepalive 32;
}

server {
    listen 80;
    server_name azkaban.company.com;
    
    client_max_body_size 100M;  # Flow zip包上传限制
    
    # 通用代理设置
    location / {
        proxy_pass http://azkaban_web_backend;
        
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        # 关键超时设置
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 300s;  # 长轮询需要较长的read_timeout
        
        # WebSocket支持
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
    
    # 健康检查端点
    location /health {
        access_log off;
        return 200 "healthy\n";
        add_header Content-Type text/plain;
    }
}

# Keepalived VIP配置（可选）
# vrrp_instance azkaban_vip {
#     state MASTER
#     interface eth0
#     virtual_router_id 51
#     priority 100
#     virtual_ipaddress {
#         192.168.1.100/24
#     }
# }
```

#### 步骤3：Executor健康检查与自动切换

**目标**：配置Web Server自动检测Executor状态。

```properties
# azkaban.properties —— Web Server侧配置

# 心跳间隔（毫秒）
azkaban.executor.ping.interval=30000

# 健康检查间隔
azkaban.executor.health.check.interval=60000

# 连续失败多少次后标记为DEAD
azkaban.executor.max.failurecount=3

# Executor选择器（负载均衡策略）
azkaban.executorselector.filters=StaticRemainingFlowSize,CpuStatus
azkaban.executorselector.comparator.NumberOfAssignedFlowComparator=1
azkaban.executorselector.comparator.Memory=1
azkaban.executorselector.comparator.LastDispatched=1
azkaban.executorselector.comparator.CpuUsage=1
```

**Executor故障转移验证**：

```bash
#!/bin/bash
# test_executor_failover.sh

echo "=== Executor故障转移测试 ==="

# 1. 查看当前活跃Executor
echo "[1] 当前活跃Executor:"
curl -s "http://localhost:8081/executor?ajax=fetchallexecutors" \
  | python3 -c "import json,sys; [print(f\"  ID={e['id']} Status={e['status']}\") for e in json.load(sys.stdin)['executors']]"

# 2. 提交一个测试Flow
echo "[2] 提交测试Flow..."
EXEC_ID=$(curl -s -b /tmp/cookie \
  -X POST "http://localhost:8081/executor?ajax=executeFlow" \
  --data "project=ha_test&flow=long_running" \
  | grep -o '"execid":[0-9]*' | grep -o '[0-9]*')
echo "  Execution ID: $EXEC_ID"

# 3. 获取Flow被分配到的Executor
ASSIGNED_TO=$(curl -s "http://localhost:8081/executor?execid=$EXEC_ID&ajax=fetchexecflow" \
  | python3 -c "import json,sys; print(json.load(sys.stdin).get('executorId','N/A'))")
echo "  分配到: Executor-$ASSIGNED_TO"

# 4. 停止该Executor（模拟故障）
echo "[3] 停止 Executor-$ASSIGNED_TO..."
ssh "exec-0${ASSIGNED_TO}" "cd /opt/azkaban-exec && bin/shutdown-exec.sh"
sleep 30

# 5. 检查Executor状态
echo "[4] 检查Executor状态变化..."
curl -s "http://localhost:8081/executor?ajax=fetchallexecutors" \
  | python3 -c "import json,sys; [print(f\"  ID={e['id']} Status={e['status']}\") for e in json.load(sys.stdin)['executors']]"

# 6. 恢复Executor
echo "[5] 恢复Executor..."
ssh "exec-0${ASSIGNED_TO}" "cd /opt/azkaban-exec && bin/start-exec.sh"
sleep 15
# 激活Executor
curl -s -X POST "http://localhost:8081/executor?ajax=activate" \
  --data "executorId=${ASSIGNED_TO}"

echo "=== 故障转移测试完成 ==="
```

#### 步骤4：MySQL主从复制配置

**目标**：配置MySQL主从复制。

```sql
-- Master配置 (my.cnf)
-- server-id=1
-- log-bin=mysql-bin
-- binlog-format=ROW
-- sync_binlog=1
-- innodb_flush_log_at_trx_commit=1

-- 创建复制用户
CREATE USER 'repl'@'%' IDENTIFIED BY 'repl_password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';
FLUSH PRIVILEGES;

-- Slave配置
-- server-id=2
-- relay-log=mysql-relay-bin
-- read_only=ON

-- 在Slave上执行
CHANGE MASTER TO
    MASTER_HOST='mysql-master.company.com',
    MASTER_USER='repl',
    MASTER_PASSWORD='repl_password',
    MASTER_LOG_FILE='mysql-bin.000001',
    MASTER_LOG_POS=107;

START SLAVE;
SHOW SLAVE STATUS\G
```

#### 步骤5：MySQL自动故障切换

**目标**：使用ProxySQL实现透明的数据库故障切换。

```bash
# proxysql配置
# /etc/proxysql.cnf

# 后端MySQL节点
mysql_servers = (
    {
        address = "mysql-master.company.com"
        port = 3306
        hostgroup = 0  # 写组
        max_connections = 200
    },
    {
        address = "mysql-slave.company.com" 
        port = 3306
        hostgroup = 1  # 读组
        max_connections = 200
    }
)

# 读写分离规则
mysql_query_rules = (
    {
        rule_id = 1
        active = 1
        match_pattern = "^SELECT.*"
        destination_hostgroup = 1  # 读操作到Slave
        apply = 1
    },
    {
        rule_id = 2
        active = 1
        match_pattern = ".*"
        destination_hostgroup = 0  # 写操作到Master
        apply = 1
    }
)

# 用户配置
mysql_users = (
    {
        username = "azkaban"
        password = "prod_pass"
        default_hostgroup = 0
    }
)
```

**Azkaban连接ProxySQL**：

```properties
# azkaban.properties
mysql.host=proxysql.company.com
mysql.port=6033  # ProxySQL默认端口
mysql.user=azkaban
mysql.password=prod_pass
```

## 4. 项目总结

### HA方案对比

| 方案 | RTO | RPO | 复杂度 | 成本 |
|------|-----|-----|--------|------|
| 单机 | ∞ | ∞ | ★☆☆ | 1台 |
| 主备切换 | 1-5分钟 | <1秒 | ★★☆ | 3-5台 |
| 多活 | <30秒 | 0 | ★★★ | 5-8台 |
| K8s + Operator | <30秒 | 0 | ★★★ | 5+台 |

### 适用场景

- **适用**：核心数据平台（金融、电商）、对可用性有SLA要求的场景、多团队共享的调度中心
- **不适用**：开发测试环境、Solo Server模式

### 注意事项

- HA架构测试必须定期进行（建议每月一次故障演练）
- Nginx的`ip_hash`是session sticky的必要条件
- MySQL主从延迟在繁忙时可能达到数秒，影响读写一致性
- Executor故障时，正在运行的Job会丢失，需保证Flow幂等性

### 常见踩坑经验

1. **双Web同时调度**：两个Web Server都认为自己Master，导致Flow被重复提交。解决：设置`azkaban.should.verify.active.server=true`。
2. **MySQL主从切换后Azkaban连接失败**：ProxySQL未正确检测到Master故障，仍将写请求路由到旧Master。解决：配置ProxySQL的`mysql-monitor`机制。
3. **Session丢失导致登录失败**：用户登录Web-01，但因ip_hash路由到Web-02，session无效。解决：确保ip_hash一致性或使用Redis共享session。
