# 第15章：常见生产故障排查手册

## 1. 项目背景

### 业务场景

凌晨4点，运维值班工程师收到告警——"核心数据同步"Flow已经失败，10分钟后自动重试再次失败。他打开Azkaban Web界面，看到一连串红色的失败标记。糟糕的是，同时有5个Flow处于失败或卡死状态。

他需要快速判断：这些失败是同一个根因导致的"连锁反应"，还是多个独立故障？是临时性的（等着就好），还是需要立即手动干预的严重故障？他只有30分钟的时间窗口——如果5点前数据还没同步完成，早晨的业务报表就会全部延迟。

### 痛点放大

没有故障排查SOP时：

1. **盲目重启**：看到失败就点"重试"，连续点5次全部失败，浪费了宝贵的时间窗口。
2. **根因迷失**：下游3个Job失败了，挨个排查，最后发现是上游数据采集延迟——但已经过了40分钟。
3. **症状误判**："502 Bad Gateway"可能来自Azkaban自身、Nginx代理、或网络层，每个的修复方式完全不同。
4. **告警疲劳后麻木**：收到太多无关告警，真正严重的问题反而错过了处理窗口。

## 2. 项目设计——剧本式交锋对话

**小胖**（满头大汗地盯着监控大盘）：大师，现在凌晨4点半，5个Flow全挂了，我该先看哪个？完全没头绪啊！

**大师**（淡定地）：记住——**故障排查的核心不是修好每一个问题，而是找到根因**。就像医生看病，不是头疼医头脚疼医脚，而是找出感染的源头。

给你的SOP口诀——"ABCDE五步法"：

```
A - Appearance（现象）：明确看到了什么异常？
B - Boundary（边界）：影响范围有多大？是单点还是全局？
C - Cause（根因）：直接的触发原因是什么？
D - Decision（决策）：立即修？等自动恢复？人工介入？
E - Execution（执行）：执行修复 → 验证恢复
```

**小白**：那最常见的Azkaban故障有哪些类型？每种类型的标准处理流程是什么？

**大师**（拿出提前准备好的卡片）：
我把Azkaban故障分为五大类：

| 类型 | 典型症状 | 常见根因 | 优先处理层 |
|------|---------|---------|-----------|
| 服务不可达 | 502/504/连接拒绝 | JVM OOM、端口冲突、进程挂掉 | Web/Executor Server |
| Job执行异常 | Job超时、OOM、无日志 | 脚本Bug、资源不足、数据格式错 | Job脚本/资源 |
| 调度失效 | 到了时间不触发 | Cron配置错误、时区问题、Schedule被暂停 | Schedule配置 |
| 数据库问题 | 列表加载慢、无法创建项目 | MySQL连接池满、表锁、磁盘满 | MySQL |
| 外部依赖 | 所有Job都FAILED | HDFS/Yarn/Hive故障 | 上游系统 |

**小胖**：那我现在看到的是——5个Flow失败，错误信息都是"Connection refused to HDFS"。这属于哪一类？

**大师**：明显是"外部依赖"类——问题不在Azkaban，而在HDFS。你现在要做的是：
1. 检查HDFS NameNode是否在failover
2. 检查网络是否可达（`hdfs dfs -ls /`）
3. 等HDFS恢复后重试失败的Flow

不要急着修Azkaban——它只是"信使"，不是"病因"。

**小白**：那502错误呢？这个最常见但我一直搞不懂。

**大师**：502 Bad Gateway是Azkaban Web Server给你返回的。根源可能是：
1. Web Server的线程池满了（大量请求排队）
2. Web Server → Executor的RPC连接断了
3. Executor进程挂了但Web Server还在

排查路径是这样的：

```
502故障排查树：
┌─ 检查Azkaban进程是否活着？
│   ├─ 活着 → curl http://localhost:8081
│   │   ├─ 200 → 可能是前端LB的问题
│   │   └─ 502 → Web Server线程耗尽或Executor不可达
│   └─ 没活着 → 重启Azkaban服务
│
├─ 检查线程池状态
│   └─ curl http://localhost:8081/jmx → 看ThreadPool指标
│
└─ 检查日志
    └─ grep "502\|OutOfMemory\|too many open files" azkaban.log
```

### 技术映射总结

- **ABCDE五步法** = 急诊分诊（先判断病人严重程度，再对症治疗）
- **根因 vs 症状** = 发烧（症状）vs 细菌感染（根因）——退烧药治标不治本
- **502 Bad Gateway** = 传达室大爷说"你找的人不在办公室"（不是没人上班，是中间人找不到了）
- **外部依赖故障** = 快递公司的分拣中心停电了（不是你的包裹寄丢，是整个中心停了）

#### 步骤1：502/504故障排查

**目标**：掌握Web服务不可达的标准排查流程。

```bash
#!/bin/bash
# diagnose_502.sh —— 502故障排查SOP

echo "=== Azkaban 502 故障排查 SOP ==="
echo "时间: $(date)"
echo ""

# Step 1: 检查Azkaban进程
echo "[Step 1] 检查Azkaban进程状态..."
if pgrep -f azkaban > /dev/null; then
    PID=$(pgrep -f azkaban | head -1)
    echo "  ✓ Azkaban进程存在, PID: $PID"
    UPTIME=$(ps -o etime= -p $PID)
    echo "  运行时间: $UPTIME"
else
    echo "  ✗ Azkaban进程不存在！需要重启服务"
    exit 1
fi

# Step 2: 检查JVM内存
echo "[Step 2] 检查JVM内存使用..."
PID=$(pgrep -f azkaban | head -1)
jcmd $PID VM.uptime 2>/dev/null
jcmd $PID GC.heap_info 2>/dev/null | grep -E "total|used"

HEAP_USED=$(jcmd $PID GC.heap_info 2>/dev/null | grep "used" | head -1 | awk '{print $NF}')
echo "  堆内存使用: $HEAP_USED"

# Step 3: 检查端口监听
echo "[Step 3] 检查端口..."
for PORT in 8081 12321; do
    if ss -tlnp | grep -q ":$PORT"; then
        echo "  ✓ 端口 $PORT 正在监听"
    else
        echo "  ✗ 端口 $PORT 未监听！"
    fi
done

# Step 4: 测试本地连接
echo "[Step 4] 测试本地连接..."
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" --connect-timeout 5 http://localhost:8081)
echo "  HTTP状态码: $HTTP_CODE"

# Step 5: 检查线程池
echo "[Step 5] 检查线程状态..."
THREAD_COUNT=$(jcmd $PID Thread.print 2>/dev/null | grep "^\"http" | wc -l)
echo "  HTTP线程数: $THREAD_COUNT"

# Step 6: 查看最近的错误日志
echo "[Step 6] 最近的错误日志..."
LOG_FILE="/opt/azkaban/logs/azkaban.log"
if [ -f "$LOG_FILE" ]; then
    RECENT_ERRORS=$(grep "ERROR\|FATAL\|OutOfMemory" "$LOG_FILE" | tail -5)
    if [ -n "$RECENT_ERRORS" ]; then
        echo "$RECENT_ERRORS"
    else
        echo "  无近期错误日志"
    fi
fi

echo ""
echo "=== 排查完成，请根据以上信息判断故障原因 ==="
```

#### 步骤2：Job执行卡死排查

**目标**：定位Job长时间"RUNNING"但无进展的问题。

```bash
#!/bin/bash
# diagnose_stuck_job.sh

EXEC_ID=$1
if [ -z "$EXEC_ID" ]; then
    echo "用法: $0 <execution_id>"
    exit 1
fi

echo "=== Job卡死诊断 (Execution ID: $EXEC_ID) ==="

# 1. 获取执行状态
echo "[1] Flow执行状态..."
FLOW_STATUS=$(curl -s -b cookies.txt \
  "http://localhost:8081/executor?execid=${EXEC_ID}&ajax=fetchexecflow")

# 提取RUNNING状态的Job
RUNNING_JOBS=$(echo "$FLOW_STATUS" | python3 -c "
import json, sys
data = json.load(sys.stdin)
for node in data.get('nodes', []):
    if node.get('status') == 'RUNNING':
        print(f\"{node['id']} (开始: {node.get('startTime')})\")
")

if [ -z "$RUNNING_JOBS" ]; then
    echo "  无RUNNING状态的Job，检测是否有等待状态的Job..."
else
    echo "  等待中的Job: $RUNNING_JOBS"
fi

# 2. 检查Executor资源
echo "[2] Executor资源状态..."
PID=$(pgrep -f azkaban.executor | head -1)
if [ -n "$PID" ]; then
    # CPU使用率
    CPU=$(ps -p $PID -o %cpu --no-headers)
    echo "  Executor CPU: ${CPU}%"
    
    # 内存使用
    MEM=$(ps -p $PID -o %mem --no-headers)
    echo "  Executor 内存: ${MEM}%"
    
    # 打开的线程
    THREADS=$(jcmd $PID Thread.print 2>/dev/null | grep "^\"JobRunner" | wc -l)
    echo "  JobRunner线程数: $THREADS"
fi

# 3. 检查外部资源连接
echo "[3] 检查可能的外部阻塞..."
# 检查HDFS
timeout 5 hdfs dfs -ls / > /dev/null 2>&1
if [ $? -eq 0 ]; then
    echo "  ✓ HDFS可访问"
else
    echo "  ✗ HDFS不可访问（可能NameNode故障或网络问题）"
fi

# 检查Yarn
timeout 5 yarn application -list 2>/dev/null | head -3
if [ $? -eq 0 ]; then
    echo "  ✓ Yarn可访问"
else
    echo "  ✗ Yarn不可访问"
fi

echo "=== 卡死诊断完成 ==="
```

#### 步骤3：调度不触发排查

**目标**：定位Cron调度没有按时执行的原因。

```bash
#!/bin/bash
# diagnose_schedule.sh

PROJECT=$1
FLOW=$2

echo "=== 调度不触发诊断 ($PROJECT/$FLOW) ==="

# 1. 检查调度是否存在
echo "[1] 检查调度状态..."
SCHEDULE_INFO=$(curl -s -b cookies.txt \
  "http://localhost:8081/schedule?ajax=fetchAllScheduledFlows")

SCHEDULE_EXISTS=$(echo "$SCHEDULE_INFO" | python3 -c "
import json, sys
data = json.load(sys.stdin)
for item in data.get('items', []):
    if item.get('projectName') == '${PROJECT}' and item.get('flowName') == '${FLOW}':
        print('YES')
        print('nextExecTime:', item.get('nextExecTime'))
        print('cronExpression:', item.get('cronExpression'))
        print('timezone:', item.get('scheduleTimezone'))
        break
else:
    print('NO')
")

if [ "$(echo "$SCHEDULE_EXISTS" | head -1)" = "NO" ]; then
    echo "  ✗ 该调度不存在！请重新创建"
    exit 1
fi

echo "$SCHEDULE_EXISTS"

# 2. 检查Cron表达式
echo "[2] 验证Cron表达式..."
CRON_EXPR=$(echo "$SCHEDULE_EXISTS" | grep "cronExpression:" | cut -d' ' -f2-)
echo "  原始: $CRON_EXPR"

# 使用Python验证接下来的触发时间
python3 -c "
from croniter import croniter
from datetime import datetime
import pytz

cron = '$CRON_EXPR'
tz = pytz.timezone('Asia/Shanghai')
now = datetime.now(tz)
iter = croniter(cron, now)
print(f'  基准时间: {now}')
print('  接下来3次触发:')
for _ in range(3):
    t = iter.get_next(datetime)
    print(f'    {t.strftime(\"%Y-%m-%d %H:%M:%S\")}')
"

# 3. 检查是否有并发执行的限制
echo "[3] 检查是否有上一个实例还在运行..."
RECENT_EXECS=$(curl -s -b cookies.txt \
  "http://localhost:8081/manager?project=${PROJECT}&ajax=fetchFlowExecutions&flow=${FLOW}&start=0&length=3")

RUNNING_COUNT=$(echo "$RECENT_EXECS" | python3 -c "
import json, sys
data = json.load(sys.stdin)
running = [e for e in data.get('executions', []) if e.get('status') == 'RUNNING']
print(len(running))
")

if [ "$RUNNING_COUNT" -gt 0 ]; then
    echo "  ✗ 有 $RUNNING_COUNT 个实例正在运行，调度可能因为skip策略被跳过"
fi

# 4. 检查系统时间
echo "[4] 系统时间检查..."
echo "  系统时间: $(date)"
echo "  时区: $(timedatectl 2>/dev/null | grep "Time zone" || echo 'UTC')"

echo "=== 调度诊断完成 ==="
```

#### 步骤5：故障总结与预防脚本

**目标**：建立故障知识库，持续改进。

```python
#!/usr/bin/env python3
# incident_report.py —— 故障总结脚本

import json
from datetime import datetime

def generate_incident_report(exec_id, description, root_cause, fix_action, prevention):
    """生成故障总结报告"""
    report = {
        "incident_id": f"INC-{datetime.now().strftime('%Y%m%d')}-{exec_id}",
        "timestamp": datetime.now().isoformat(),
        "execution_id": exec_id,
        "severity": "P2",  # P1=严重 P2=一般 P3=轻微
        "description": description,
        "root_cause": root_cause,
        "fix_action": fix_action,
        "prevention": prevention,
        "resolution_time_minutes": 0,
        "affected_flows": [],
        "lessons_learned": []
    }
    
    print("=== 故障总结报告 ===")
    print(f"事件ID: {report['incident_id']}")
    print(f"时间: {report['timestamp']}")
    print(f"描述: {report['description']}")
    print(f"根因: {report['root_cause']}")
    print(f"修复措施: {report['fix_action']}")
    print(f"预防措施: {report['prevention']}")
    print("=" * 40)
    
    # 保存到JSON文件
    filename = f"incident_{report['incident_id']}.json"
    with open(filename, 'w', encoding='utf-8') as f:
        json.dump(report, f, indent=2, ensure_ascii=False)
    print(f"报告已保存: {filename}")
    
    return report

# 故障模式库
KNOWN_FAILURE_PATTERNS = {
    "hdfs_failover": {
        "symptom": "多个Job同时失败，错误信息包含'Connection refused'或'NameNode'",
        "cause": "HDFS NameNode failover 或网络抖动",
        "fix": "等待HA自动切换完成后重试失败的Job",
        "retry_delay_seconds": 180
    },
    "executor_oom": {
        "symptom": "Azkaban进程频繁重启，日志中出现OutOfMemoryError",
        "cause": "Executor堆内存不足或内存泄漏",
        "fix": "增大JVM -Xmx参数，检查是否有大对象未释放",
        "retry_delay_seconds": 0
    },
    "db_connection_pool_exhausted": {
        "symptom": "Web界面加载缓慢或超时，创建项目失败",
        "cause": "MySQL连接池耗尽，可能因慢查询或连接泄漏",
        "fix": "检查MySQL连接数，重启Azkaban清空连接池",
        "retry_delay_seconds": 0
    },
}

def match_failure_pattern(error_logs):
    """匹配已知故障模式"""
    for pattern_name, pattern in KNOWN_FAILURE_PATTERNS.items():
        # 简单的关键词匹配
        if any(kw in str(error_logs).lower() for kw in pattern["cause"].lower().split()):
            print(f"匹配到已知故障模式: {pattern_name}")
            print(f"  > 症状: {pattern['symptom']}")
            print(f"  > 根因: {pattern['cause']}")
            print(f"  > 建议修复: {pattern['fix']}")
            return pattern
    return None

if __name__ == '__main__':
    # 示例：匹配故障模式
    sample_error = "Connection refused to NameNode at hdfs-prod:8020"
    matched = match_failure_pattern(sample_error)
    
    if matched:
        # 自动生成处理建议
        generate_incident_report(
            exec_id="42",
            description=matched["symptom"],
            root_cause=matched["cause"],
            fix_action=matched["fix"],
            prevention="添加HDFS监控告警，在failover期间自动暂停相关Flow"
        )
```

### 3.3 常见故障速查表

| 错误码/现象 | 可能原因 | 快速排查命令 | 修复方式 |
|-----------|---------|------------|---------|
| 502 Bad Gateway | Executor RPC不可达 | `netstat -tlnp \| grep 12321` | 重启Executor |
| 504 Gateway Timeout | Web Server处理超时 | `tail -f azkaban.log \| grep Timeout` | 增大超时配置 |
| Connection refused | 目标服务未启动 | `telnet host port` | 启动目标服务 |
| OutOfMemoryError | JVM堆内存不足 | `jcmd <pid> GC.heap_info` | 增大-Xmx |
| Flow stuck RUNNING | Job内部死循环或等待 | `jstack <pid> \| grep WAITING` | 取消Flow，修复脚本 |
| Schedule not triggering | Cron错误或时区不对 | 见步骤3的诊断脚本 | 修正Cron或时区 |
| OOM Killer kills Azkaban | 系统内存不足 | `dmesg \| grep -i kill` | 增加系统内存或减少并发 |
| MySQL Gone Away | DB连接超时断开 | `mysql -e "SELECT 1"` | 调整wait_timeout |

## 4. 项目总结

### 故障排查方法对比

| 方法 | 效率 | 准确性 | 学习成本 | 适用团队 |
|------|------|--------|---------|---------|
| 经验驱动（老手记忆） | ★★★ 快 | ★★☆ 可能误判 | ★★★ 需长期积累 | 小团队 |
| SOP文档 | ★★☆ 中 | ★★★ 标准 | ★☆☆ 低 | 所有团队 |
| 自动化诊断脚本 | ★★★ 极快 | ★★★ 精准 | ★★☆ 中 | 大型团队 |
| AI辅助分析 | ★★★ 极快 | ★★☆ 待优化 | ★☆☆ 低 | 前沿团队 |

### 适用场景

- **适用**：生产环境运维值班、故障演练、新员工培训、故障复盘
- **不适用**：一次性调试验证（不值得写SOP）、极其罕见的边界Case

### 注意事项

- 故障排查时先做"隔离"——确认影响范围后再行动，避免小问题被扩大
- 修改Azkaban配置后必须重启才生效，但重启期间所有运行中的Job都会被中断
- 不要在生产Azkaban上直接调试——复现问题在测试环境进行
- 每次故障处理后须记录到知识库，积累团队经验

### 常见踩坑经验

1. **反复重启不解决问题**：Executor OOM后重启，跑几小时后再次OOM。根因是某个Job内存泄漏。解决方案：排查具体是哪个Job导致的，而不是不断增大JVM内存。
2. **忽略时间同步问题**：Azkaban服务器和NTP时间不同步，导致Cron调度时间偏差。解决：配置NTP服务，定期检查时间偏移。
3. **日志级别的坑**：排查问题时调整日志级别为DEBUG，事后忘记恢复，导致日志文件暴涨。解决：故障排查后记录检查清单——恢复日志级别。
