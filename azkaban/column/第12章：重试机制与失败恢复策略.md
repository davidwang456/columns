# 第12章：重试机制与失败恢复策略

## 1. 项目背景

### 业务场景

凌晨3点，"用户画像构建"Flow中的"数据同步"Job因为HDFS临时的NameNode切换（failover，通常持续30-60秒）而失败了。这本就是一个短暂的可恢复错误，但因为没有配置重试机制，整个Flow因此中断。下游10个Job全部处于"等待"状态，直到第二天早上人工介入。

更糟糕的是，值班同事早上8点手动重跑这个Flow时，选择的是"从头开始"——上游的"日志采集"Job也重新跑了一遍，导致Hive中出现了重复数据，需要花2小时清理。

### 痛点放大

没有完善的重试与恢复策略时：

1. **过度重试**：配置了`retries=10`，一个因为"SQL语法错误"必然失败的Job，被重试了10次，每次失败发一封邮件，团队收到10封告警。
2. **重试不足**：网络抖动导致的临时连接超时，一次失败就放弃，实际上等1分钟再试就能成功。
3. **恢复位置错误**：Flow中途失败后，不知道从哪里重新执行——从头开始浪费资源且可能产生重复数据，从失败点开始又担心遗漏了前置步骤。
4. **重试依赖破坏**：一个Job失败重试3次后最终成功了，但下游Job已经因为等待超时被跳过了。

## 2. 项目设计——剧本式交锋对话

**小胖**（懊恼地看着执行日志）：大师，昨晚就一个NameNode切换，30秒的事情，结果整个Flow都挂了。能不能让Azkaban碰到这种错误时自动等一下再试？

**大师**：当然能。Azkaban提供了两组重试配置：

```bash
retries=3              # 失败后最多重试3次
retry.backoff=60000    # 每次重试间隔60秒（毫秒为单位）
```

这两个参数就是为"瞬态故障"设计的。NameNode failover一般30-60秒恢复，你配个`retries=2, retry.backoff=30000`就够了。

**小白**：但是重试会把所有错误都重试吗？如果一个Job是"配置错误"（比如写错了SQL表名），那重试100次也没用啊……

**大师**：这正是我们要解决的问题。在理想的重试机制中，应该区分"可恢复错误"和"不可恢复错误"。

- **可恢复错误**：网络超时、连接中断、资源暂时不足 → 值得重试
- **不可恢复错误**：配置错误、权限错误、数据格式错误 → 重试无意义

但Azkaban本身不做这个区分。你需要在Job脚本中明确退出码的含义。我的建议是：

```bash
# 在脚本中区分错误类型
exit 0   # 成功
exit 1   # 一般错误（可能是瞬时的）
exit 2   # 配置错误（不可恢复，重试无意义）
exit 3   # 资源不足（可恢复，等一会再试）
```

然后在`.job`文件中通过`command`的返回值判断，而不是依赖Azkaban的retry。

**小胖**：那如果Flow中间某个Job失败了，怎么不从头开始，而是从失败点恢复？

**大师**：这就是"断点续跑"的需求。Azkaban支持几种恢复策略：

```
Flow恢复方式：
┌──────────────────────────────────────────────────────┐
│  原始Flow:  A → B → C → D → E                        │
│  如果C失败了：                                        │
│                                                       │
│  ① 从头执行(restart):                                 │
│     A → B → C → D → E  （全部重跑）                   │
│                                                       │
│  ② 仅重试失败Job(retry failed):                       │
│     C → D → E  （C重试，然后继续D,E）                  │
│                                                       │
│  ③ 从失败点执行(execute from failed):                 │
│     C → D → E  （从C开始往下跑）                       │
│                                                       │
│  ④ 执行指定Job(run specific job):                     │
│     D → E  （跳过C，直接从D开始）                      │
└──────────────────────────────────────────────────────┘
```

在Web界面的"Executions"页面，每个失败的Flow旁边都有"重试"按钮，点进去可以选择重试模式。

**小白**：那重试状态怎么传给下游？比如B重试了3次才成功，下游C知道B曾经失败过吗？

**大师**：在Azkaban中，只要B最终状态是SUCCEEDED，C就认为一切正常——它不关心B是第一次就成功还是重试了3次。这其实是一个设计上的考量：下游只关心"上游的数据是否准备好了"，不关心过程。

但如果你想在日志中保留重试历史，你可以在Job脚本中写checkpoint：

```bash
RETRY_COUNT=${azkaban.retry.count:-0}
echo "Current attempt: $((RETRY_COUNT + 1))"
```

**小胖**：那如果我想要更复杂的重试策略，比如"失败后等5分钟，再失败等10分钟，再失败等30分钟"，能做到吗？

**大师**：Azkaban原生不支持指数退避（exponential backoff）。`retry.backoff`是固定的间隔。但有变通方案：

1. 在Job脚本中自己实现退避逻辑
2. 把你的Job拆成多个——每个代表一次尝试，失败后呼叫下一个"重试Job"

我更推荐第二种，因为每个"尝试"都是独立的Job，日志清晰，可以被独立监控。

### 技术映射总结

- **retries** = 外卖重送次数（下雨天配送员可能晚到，再送一次就好）
- **retry.backoff** = 等待间隔（别连续打电话催，给配送员一点时间）
- **可恢复 vs 不可恢复** = 感冒 vs 骨折（感冒可以自愈，骨折必须就医）
- **断点续跑** = 视频的"继续播放"（不必从头开始看）

#### 步骤2：区分可恢复与不可恢复错误

**目标**：通过退出码区分错误类型，实现智能重试。

```bash
# smart_retry.job —— 智能重试Job
type=command
retries=3
retry.backoff=30000

command=bash -c '
echo "[$(date)] Starting smart retry job..."

# 检查网络连通性（可恢复错误）
echo "[$(date)] Checking network connectivity..."
ping -c 1 -W 2 192.168.1.100 > /dev/null 2>&1
if [ $? -ne 0 ]; then
    echo "[$(date)] Network unreachable - retrying (exit code 10)"
    exit 10  # 自定义退出码10 = 瞬态网络错误
fi

# 检查配置文件（不可恢复错误）
CONFIG_FILE="/opt/etl/config.yaml"
if [ ! -f "$CONFIG_FILE" ]; then
    echo "[$(date)] FATAL: Config file not found: $CONFIG_FILE"
    echo "[$(date)] This error cannot be resolved by retry"
    exit 20  # 自定义退出码20 = 配置错误
fi

# 检查磁盘空间（可恢复错误——等一会可能被清理）
AVAIL_GB=$(df -BG /tmp | tail -1 | awk "{print \$4}" | sed "s/G//")
if [ "$AVAIL_GB" -lt 1 ]; then
    echo "[$(date)] Low disk space: ${AVAIL_GB}GB available"
    exit 30  # 自定义退出码30 = 资源不足
fi

echo "[$(date)] All checks passed. Proceeding with ETL..."
exit 0
'
```

**外部重试控制脚本**（替代Azkaban原生的retry）：

```python
#!/usr/bin/env python3
# smart_retry_controller.py

import subprocess
import time
import sys

# 错误码定义
RETRYABLE_ERRORS = {10, 30}  # 网络错误、资源不足 → 可重试
FATAL_ERRORS = {20}           # 配置错误 → 不重试

def execute_with_smart_retry(command, max_retries=3, backoff_seconds=30):
    """智能重试执行"""
    for attempt in range(1, max_retries + 1):
        print(f"\n[{time.strftime('%Y-%m-%d %H:%M:%S')}] Attempt {attempt}/{max_retries}")
        
        result = subprocess.run(command, shell=True, capture_output=True, text=True)
        print(result.stdout)
        
        if result.returncode == 0:
            print(f"[SUCCESS] Command completed on attempt {attempt}")
            return True
        
        if result.returncode in FATAL_ERRORS:
            print(f"[FATAL] Non-recoverable error (exit code {result.returncode}), "
                  f"stopping retry. Error: {result.stderr[:200]}")
            return False
        
        if result.returncode in RETRYABLE_ERRORS:
            if attempt < max_retries:
                print(f"[RETRY] Recoverable error, waiting {backoff_seconds}s "
                      f"before next attempt...")
                time.sleep(backoff_seconds)
            else:
                print(f"[FAILED] Max retries ({max_retries}) exhausted")
                return False
        
    return False

if __name__ == '__main__':
    exit(0 if execute_with_smart_retry('bash smart_retry.sh') else 1)
```

#### 步骤4：指数退避重试策略

**目标**：实现失败后等待时间递增的退避策略。

```bash
# exponential_backoff.job —— 指数退避重试
type=command
retries=4

command=bash -c '
MAX_RETRIES=4

for attempt in $(seq 1 $MAX_RETRIES); do
    echo "[Attempt $attempt/$MAX_RETRIES] Starting..."
    
    # 执行实际任务
    mysql -h prod-db -u etl_user -e "SELECT COUNT(*) FROM orders WHERE dt=CURDATE()"
    
    if [ $? -eq 0 ]; then
        echo "[Attempt $attempt] SUCCESS"
        exit 0
    fi
    
    if [ $attempt -lt $MAX_RETRIES ]; then
        # 指数退避: 2^attempt 分钟
        BACKOFF=$((2 ** attempt * 60))
        echo "[Attempt $attempt] FAILED. Waiting ${BACKOFF}s before retry..."
        sleep $BACKOFF
    else
        echo "[Attempt $attempt] FAILED permanently"
        exit 1
    fi
done
'
```

**退避时间表**：
| 尝试次数 | 等待时间 | 累计等待 |
|---------|---------|---------|
| 第1次失败 | 2分钟 | 2分钟 |
| 第2次失败 | 4分钟 | 6分钟 |
| 第3次失败 | 8分钟 | 14分钟 |
| 第4次失败 | — | 放弃 |

#### 步骤5：Flow级别的失败恢复

**目标**：当Flow中某个Job失败后，选择最优恢复策略。

```bash
# flow_recovery.flow
nodes=init_check,data_sync,data_process,report_gen

# Flow配置
failure.emails=oncall@company.com
failure.action=finishPossible   # 继续执行不依赖失败Job的其他Job
```

```python
#!/usr/bin/env python3
# flow_recovery_tool.py

import requests
import json

class AzkabanRecoveryTool:
    def __init__(self, base_url, session_id):
        self.base_url = base_url
        self.session = session_id
    
    def get_failed_jobs(self, exec_id):
        """获取失败的Job列表"""
        resp = requests.get(
            f"{self.base_url}/executor",
            params={"execid": exec_id, "ajax": "fetchexecflow"},
            cookies={"azkaban.browser.session.id": self.session}
        )
        data = resp.json()
        failed = [n for n in data.get("nodes", []) if n.get("status") == "FAILED"]
        return failed
    
    def retry_failed_only(self, exec_id):
        """仅重试失败的Job"""
        resp = requests.post(
            f"{self.base_url}/executor",
            data={"execid": exec_id, "ajax": "retryFailedJobs"},
            cookies={"azkaban.browser.session.id": self.session}
        )
        return resp.json()
    
    def skip_and_continue(self, exec_id, failed_jobs):
        """跳过失败的Job，继续执行下游"""
        disabled = ",".join([j["id"] for j in failed_jobs])
        resp = requests.post(
            f"{self.base_url}/executor",
            data={
                "ajax": "executeFlow",
                "project": "etl_pipeline",
                "flow": "daily_report",
                "disabled": disabled
            },
            cookies={"azkaban.browser.session.id": self.session}
        )
        return resp.json()

# 使用示例
tool = AzkabanRecoveryTool("http://localhost:8081", "session_xxx")
failed_jobs = tool.get_failed_jobs(42)
print(f"发现 {len(failed_jobs)} 个失败Job: {[j['id'] for j in failed_jobs]}")

# 根据失败Job的类型决定恢复策略
if any(j["id"] == "data_sync" for j in failed_jobs):
    print("数据同步任务失败，建议: 等待数据源恢复后重试")
    result = tool.retry_failed_only(42)
else:
    print("非关键任务失败，跳过并继续")
    result = tool.skip_and_continue(42, failed_jobs)
```

## 4. 项目总结

### 重试策略对比

| 策略 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 固定间隔重试 | 已知恢复时间的故障 | 简单 | 可能过早或过晚 |
| 指数退避 | 不清楚恢复时间的故障 | 给系统更多自我修复时间 | 等待时间可能很长 |
| 带抖动的退避 | 多客户端同时重试 | 避免雪崩效应 | 实现稍复杂 |
| 不重试 | 配置错误等不可恢复错误 | 快速失败，减少告警 | 可能遗漏可恢复错误 |

### 适用场景

- **适用**：网络不稳定的环境、依赖外部系统（HDFS/Yarn）的ETL任务、凌晨非人工值守的批处理任务
- **不适用**：秒级延迟敏感的任务（重试会增加延迟）、幂等性无法保证的任务（重复执行会产生副作用）

### 注意事项

- `retry.backoff`单位是毫秒，写`retry.backoff=60`等于等待0.06秒，基本没意义
- 重试不会改变Job的`start time`，只会更新`end time`
- 幂等性是重试的前提——确保你的Job重复执行不会产生副作用
- 对于Spark/Hadoop任务，不建议在Job级别重试，因为Yarn本身有重试机制

### 常见踩坑经验

1. **重试导致数据重复**：一个SQL INSERT Job失败后重试成功，但第一批数据已经写入了一半。解决：在每个Job开始前检查是否已有部分输出了，有则先清理。
2. **retry.backoff设得太短**：设了`retry.backoff=1000`（1秒），HDFS failover需要60秒，Job在1秒后立即重试再次失败。解决：根据已知恢复时间设置合理的backoff。
3. **重试耗尽Executor线程**：10个Job同时失败并进入重试等待，占用了大量Executor线程。解决：配置`executor.max.threads`限制并发重试数。

### 思考题

1. 如何设计一个"熔断器"机制：如果某个Job连续失败超过5次（包括重试次数），自动跳过该Job和它下游的所有Job，并向管理员发送特殊告警？
2. 在一个跨系统的ETL Flow中（MySQL → HDFS → Hive → ClickHouse），中间某个环节失败后，如何保证"回滚到上一个成功checkpoint"的状态，避免数据不一致？
