# 第 24 章：高级 RAG —— 元数据与溯源（Metadata & Sources）

## 1. 项目背景

### 业务场景（拟真）

政策、合同、医疗、财务口径类回答需要 **可审计**：用户与合规要知道 **来自哪份文档、哪一版、哪一段原文**。ingest 时为每个 `TextSegment` 写入 **metadata**；回答时以 **来源列表** 返回给调用方/前端。

### 痛点放大

无溯源时 **无法回放条款**；**metadata** 过大或含 **PII** 会 **放大合规与性能风险**。示例两路：

- `_04_Advanced_RAG_with_Metadata_Example.java` —— **写入侧**  
- `_09_Advanced_RAG_Return_Sources_Example.java` —— **响应侧 Return sources**

与第 25 章 **过滤**、第 33 章 **索引漂移** 强相关。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：metadata 像图书标签？贴多了书包沉不沉？

**小白**：**metadata 会不会太大**？用户不看出处，我们还要做吗？

**大师**：存 **短键 + 可联结外部 ID**；大文本放 **CMS/对象存储**，向量库 **指针**。B2B **审计** 仍需要；UI 可 **默认折叠**，导出 **PDF** 带齐。**技术映射**：**metadata = 可追溯索引，非全文副本**。

**小胖**：引用和片段不一致咋办？

**小白**：**版本号** 放哪？**PII** 能进 metadata 吗？**溯源链接失效**？

**大师**：**同一批** `EmbeddingMatch` 进 **prompt 与 API 响应**，避免 **二次检索漂移**。版本由 **文档发布流水线** 写入。**PII 尽量避免**；必须则 **访问控制 + 日志脱敏**。CMS 链接要 **健康检查 + 备用纯文本引用**。**技术映射**：**溯源一致性 = 单次召回事实源**。

**小胖**：模型复述了片段外内容？

**大师**：提示策略：**无检索依据则拒答或声明推测**；评测集约束。**技术映射**：**生成层不能覆盖溯源层**。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | Metadata + Return sources | 无溯源 RAG | 仅模型口述 |
|------|---------------------------|------------|------------|
| 审计 | 强 | 弱 | 无 |
| 工程成本 | 高 | 低 | 低 |
| 典型缺点 | schema 治理 | 合规风险 | 不可信 |

### 适用场景

- 金融、政务、医疗、内部政策助理。

### 不适用场景

- **无合规要求** 的闲聊——可简化。

### 注意事项

- **缓存键版本化**；**对话记忆** 勿与 **来源** 混为一谈。

### 常见踩坑经验（生产向根因）

1. **只展示自然语言** 不展示来源。  
2. **版本号缺失** → 引用过时条款。  
3. **metadata 泄漏** 内部代号到公网。

### 进阶思考题

1. **多语言同一文档** 时 metadata **语言维度** 如何设计？  
2. **CDN 缓存旧答案** 与 **文档版本** 的失效策略？
