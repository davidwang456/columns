# 第 24 章：高级 RAG —— 元数据与溯源（Metadata & Sources）

## 1. 项目背景

当回答涉及 **政策、合同、医疗建议、财务口径** 时，「**模型这么说不代表公司立场**」已不够；用户与审计需要知道：**来自哪一份文档、哪一个版本、哪一段原文**。技术上，这要求在 **ingest** 时为每个 `TextSegment` 写入 **`metadata`**（文档 ID、版本号、章节、页码、部门标签等），在 **回答** 时以 **「来源列表」** 形式返回并展示。

示例分两路互补阅读：

- `.../_04_Advanced_RAG_with_Metadata_Example.java` —— 强调 **写入侧** 如何组织 metadata。  
- `.../_09_Advanced_RAG_Return_Sources_Example.java` —— 强调 **响应侧** 如何把 **检索来源** 带回给调用方/前端。

本章是 **合规与可信设计** 的枢纽，与第 25 章「过滤」、第 33 章「索引漂移」强相关。

## 2. 项目设计：大师与小白的对话

**小白**：metadata 会不会太大？

**大师**：应存储 **短键** 与 **可联结外部系统的 ID**；**大文本**留在对象存储或 CMS，**向量库只存指针**。

**小白**：用户不关心出处怎么办？

**大师**：B2B **审计**仍需要；可在 UI **默认折叠**，但 **导出/PDF** 要带齐。

**小白**：如何保证「引用」与「片段」一致？

**大师**：**同一批** `EmbeddingMatch` 同时进入 **prompt 组装**与 **API 响应**；避免 **二次检索**导致漂移。

**小白**：版本号存在哪？

**大师**：**文档发布流水线**写入；**检索时**带 `docVersion <= 用户可见版本` 过滤（与业务规则对齐）。

**小白**：PII 能塞进 metadata 吗？

**大师**：**尽量避免**；若必须存在，**访问控制**与 **日志脱敏**同步升级。

**小白**：多语言同一文档？

**大师**：**语言**作为 metadata 维度；或 **分索引**，路由时选中语言。

**小白**：如果模型复述了片段外的内容？

**大师**：产品策略：若 **无检索依据**，要求 **模型拒答或显性声明推测**（提示工程 + 评估集）。

**小白**：溯源链接失效？

**大师**：**强依赖 CMS**时要做 **健康检查**与 **备用展示**（仅文本引用）。

## 4. 项目总结

### 优点

- **可解释、可审计**，增强 **信任**。  
- 为 **法务回放**提供抓手。

### 缺点

- **schema 治理**成本高；**跨团队**对字段含义要对齐。  
- **调试**更_verbose_。

### 适用场景

- 金融、政务、医疗、**内部政策**助理。

### 注意事项

- **缓存**：CDN **可能缓存旧答案**，需 **版本化缓存键**。  
- **一致性**：**对话记忆**中的用户陈述 **不应**写进「来源」混为一谈。

### 常见踩坑

1. **只展示模型自然语言**不展示 **检索来源**。  
2. **版本号缺失**导致引用 **过时条款**。  
3. **metadata 泄漏**内部项目代号到公网。
