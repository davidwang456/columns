# 第 17 章：结构化输出与业务类型映射

## 1. 项目背景

### 业务场景（拟真）

下游系统要 **`OrderDTO`、`RiskLabel`、JSON 数组**——不是散文。LangChain4j 在 `service.output` 提供 **`StringListOutputParser`、`PojoOutputParser`、`EnumSetOutputParser`** 等，把 **模型文本** 拉回 **Java 类型**。失败包括：**漏字段**、**markdown 围栏**、**locale 数字格式**。

### 痛点放大

若 **手写 Jackson**：**噪声与重试** 散落各处；若 **把解析当安全边界**：**业务校验** 仍不可省。与第 6 章模板、第 14 章工具形成三角：**模板/工具约束行为，解析器拉回类型系统**。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：为啥不直接 `ObjectMapper` 一把梭？

**小白**：和厂商 **response_format / JSON mode** 啥关系？**解析失败要重试吗？** **enum 映射失败**呢？

**大师**：可行，但要 **统一噪声、retry、修复提示**；`PojoOutputParser` **集中** 减少复制。若 provider **原生 JSON mode**，**优先组合**；解析器仍处理 **部分响应、越界字段**。失败 **有限次重试**，超阈值 **降级人工** 并 **告警解析率**。**enum** 提供 **UNKNOWN** 或 **Optional**，勿崩溃；日志保留 **脱敏** 原文。**技术映射**：**解析层 = 契约恢复，非授权层**。

**小胖**：大 JSON 一次吐完卡死了？

**大师**：**复杂嵌套** 可 **分步**：先关键字段再第二跳 LLM，或 **工具链取数**。**技术映射**：**输出复杂度与成本线性相关**。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | PojoOutputParser 等 | 裸 Jackson | 仅 prompt 要 JSON |
|------|---------------------|------------|---------------------|
| 集中治理 | 高 | 低 | 低 |
| 脆弱性 | 中（可监控） | 中 | 高 |
| 与 JSON mode | 可组合 | 可组合 | 视厂商 |
| 典型缺点 | 格式变动 | 重复代码 | 无统一 retry |

### 适用场景

- 表单预填、风控标签、抽取式 NLU；对接 **工单/CRM**。

### 不适用场景

- **超大嵌套一次生成**——应分步或工具。

### 注意事项

- **schema 演进** 与历史消息兼容；**多语言** 日期金额。

### 常见踩坑经验（生产向根因）

1. **未处理 markdown 代码块**。  
2. **盲目重试** 放大成本。  
3. **解析器当安全边界**——仍需业务校验。
