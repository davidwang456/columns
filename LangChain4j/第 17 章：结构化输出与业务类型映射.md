# 第 17 章：结构化输出与业务类型映射

## 1. 项目背景

企业接口往往需要的是 **`OrderDTO`、`RiskLabel`、`JSON` 数组**——不是散文。LangChain4j 在主模块的 `service.output` 包提供 **`StringListOutputParser`、`PojoOutputParser`、`EnumSetOutputParser`** 等，将 **模型文本**转换为 **Java 类型**。失败场景包括：模型 **漏字段**、**夹杂 markdown 围栏**、**用了非预期 locale 的数字格式**。

本章与第 6 章模板、第 14 章工具形成三角：**模板/工具约束行为**，**解析器把输出拉回类型系统**。读源码时从 `OutputParserFactory` 与若干 `*OutputParser` 入手。

## 2. 项目设计：大师与小白的对话

**小白**：为什么不让模型只输出 JSON 再由 Jackson 反序列化？

**大师**：可行，但要 **统一处理噪声**、**retry**、**修复提示**；`PojoOutputParser` 把这套逻辑 **集中**，减少复制粘贴。

**小白**：和厂商 `response_format` 什么关系？

**大师**：若 provider **原生支持 JSON mode**，应**优先组合使用**；解析器仍要处理 **部分响应** 与 **越界字段**。

**小白**：解析失败要重试吗？

**大师**：**有限次**；超过阈值应 **降级**为人工或规则引擎，并 **告警解析失败率**。

**小白**： enum 映射失败呢？

**大师**：提供 **UNKNOWN** 或 **Optional.empty()**，不要让系统崩溃；**日志**要保留原始文本 **脱敏后**片段。

**小白**：复杂嵌套 POJO？

**大师**：考虑 **分步**：先抽取关键字段，再第二跳 LLM——避免一次性大 JSON。

**小白**：测试如何做？

**大师**：**表驱动**：`input text → expected POJO`；对 **坏样本**单独建集。

#### 闯关任务

| 难度 | 破坏实验 | 期待反应 |
|------|----------|----------|
| ★ | 让模型输出 **多一行废话** 再解析 | 体验 **fragile** |
| ★★ | 输出 **合法 JSON 但缺字段** | `Optional` / **默认值策略** |
| ★★★ | **中英文数字混排** | 与 **locale** 相关的格式化坑 |

#### 挖深一层

- **schema 与提示同步**：改字段名要 **双改**——否则 **静默解析错误**。  
- **大对象**：一次 JSON 过大 → **拆分多轮** 或 **工具链取数**。

## 4. 项目总结

### 优点

- **类型安全**贯穿服务层。  
- 解析失败可 **集中监控**。

### 缺点

- **脆弱**于模型格式变动。  
- 大对象解析 **成本高**。

### 适用场景

- 表单预填、风控标签、抽取式 NLU。  
- 与 **工单系统/CRM** 对接。

### 注意事项

- **schema 演进**与 **历史消息** 兼容。  
- **多语言**日期与金额。

### 常见踩坑

1. **未处理 markdown 代码块** 包裹。  
2. 盲目重试 **放大成本**。  
3. 把 **解析器当安全边界**——仍需业务校验。
