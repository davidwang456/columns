# 第 8 章：多模态一瞥 —— 图像模型

## 1. 项目背景

### 业务场景（拟真）

某电商平台的市场部有个固定需求：每次大促活动都需要一批配图——主视觉、社交媒体的分享图、App 内 push 的配图。以前的方法是设计师用 Photoshop 做，一套图要 3-5 个工作日。市场部总监问技术团队：「能不能在运营后台加个功能——文案我写，图你生成，一句话的事情？」后端团队发现，这需要集成 **图像生成能力**。

与文本 Chat 不同，图像模型有几个重要的差异点：

- **计费单元不同**：Chat 按 token 计费（一个回答几分钱），图像按张计费（一张图几毛到几块）。
- **输出形态不同**：Chat 返回文本字符串，图像返回的是一段 **有时效的临时 URL**——你需要在过期前下载到自己的存储。
- **延迟更大**：一张图通常需要 5-30 秒，不像 Chat 那样 1-3 秒就有回音。
- **合规要求更多**：生成的图有没有版权问题？肖像权？NSFW 内容怎么过滤？

### 痛点放大

如果每个团队各自直连 OpenAI 的 Image API 生成图片，会出现：

- **成本无法拆分**：市场部做了 1000 张图，客服部做了 50 张——账单混在同一个 OpenAI 账户里，分不清谁用了多少。
- **临时 URL 当永久链接**：前端直接用了模型返回的 URL 展示图片，三天后链接过期——用户看到 403 或空白。
- **无内容审核**：模型生成了不合规的图片，未经审核直接展示给了所有用户——这是法务事故。

LangChain4j 把图像能力抽象为 **`ImageModel`** 接口，与 ChatModel 的 BOM 治理和测试策略完全一致。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：文生图是不是跟点外卖配图一样——下单等半天，送来还不一定是你要的那盘菜？而且别人碗里的图（URL）你也不能夹来吃吧？

**大师**：点外卖的比喻很形象。首先「等半天」确实是真的——图像生成通常需要 5-30 秒，比 Chat 长很多，所以 **不能用同步等待的方式在用户请求线程里阻塞**，必须用异步任务队列。其次「别人碗里的图」——模型返回的 URL 是 **临时链接**，通常几小时后过期。你不能把这个 URL 直接扔给前端展示，**必须由你的系统下载到对象存储（OSS/S3）**，再返回你自己的永久链接。下载时还要考虑：内网环境能不能拉到外网地址？要不要做病毒扫描？CDN 预热要不要做？

**小白**：那 `ImageModel` 的版本也能走 BOM 统一管理吗？还有——CI 里如果每次都真调 DALL·E 生成图片来验证，这预算是不是太离谱了？

**大师**：BOM 当然统一管版本——`langchain4j-open-ai` 模块包含 `OpenAiImageModel`，版本跟在 BOM 里，和 ChatModel 完全一致。CI 的问题你说到点子上了：**绝对不要在 CI 里每次跑都真调图像 API**。图像比 Chat 贵得多——一张 DALL·E 3 的图大约 $0.04，如果有 200 个测试用例一次 CI 就跑掉 $8。策略是：单元测试用 **Mock `ImageModel`**（返回固定的假 URL）；端到端验证跑 **nightly 定时任务**，用少量样本验证连通性；或者用提供免费额度的沙箱模型。**技术映射**：**ImageModel 的图像生成能力只是「契约层」——模型负责生成，你的系统负责下载、存储、审核、分发——资产入库与生命周期管理是你自己的管道**。

**小胖**：图像生成失败了重试——会不会扣两次钱？

**大师**：图像 API 比 Chat 更容易被限流（429），因为单张图计算成本高。重试策略要谨慎：用 **指数退避 + 随机抖动**，并且 **不自动重试非幂等的操作**（如果调用成功了但下载超时，图片其实已经生成并付费了，重试会再生成一张新图扣两次钱）。更稳妥的方案是把图像生成放到 **任务队列**（如 RabbitMQ/Redis Queue）后端异步执行，不阻塞用户请求线程。用户看到的反馈是「任务已提交、预计 X 秒后完成、完成后通知你」。**技术映射**：**图像生成 = 异步长任务 + 配额治理 + 人工审核，不能当 Chat 那样的同步调用来设计——把「生成」和「交付」拆成两个独立的步骤**。

## 3. 项目实战

### 环境准备

```bash
cd langchain4j-examples/tutorials
export OPENAI_API_KEY="sk-your-key-here"
```

### 步骤 1：跑通图像生成

```java
import dev.langchain4j.model.image.ImageModel;
import dev.langchain4j.model.openai.OpenAiImageModel;
import dev.langchain4j.model.output.Response;
import dev.langchain4j.data.image.Image;
import static dev.langchain4j.model.openai.OpenAiImageModelName.DALL_E_3;

public class ImageDemo {

    public static void main(String[] args) {

        ImageModel model = OpenAiImageModel.builder()
                .apiKey(System.getenv("OPENAI_API_KEY"))
                .modelName(DALL_E_3)
                .build();

        Response<Image> response = model.generate(
                "Swiss software developers with cheese fondue, a parrot and a cup of coffee");

        String imageUrl = response.content().url().toString();
        System.out.println("Generated image URL:");
        System.out.println(imageUrl);
    }
}
```

**预期输出**：
```
Generated image URL:
https://oaidalleapiprodscus.blob.core.windows.net/private/org-xxx/...
```

### 步骤 2：下载图片并计算哈希

```java
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.security.MessageDigest;

// 在 main 中追加
HttpClient client = HttpClient.newHttpClient();
HttpRequest request = HttpRequest.newBuilder()
        .uri(URI.create(imageUrl))
        .build();

byte[] imageBytes = client.send(request, HttpResponse.BodyHandlers.ofByteArray()).body();

MessageDigest digest = MessageDigest.getInstance("SHA-256");
byte[] hash = digest.digest(imageBytes);
System.out.println("SHA-256: " + bytesToHex(hash));
System.out.println("Image size: " + imageBytes.length + " bytes");
```

**预期输出**：
```
SHA-256: a3f2b8c9d1e4f5a6...
Image size: 523456 bytes
```

### 步骤 3：观察随机性

```bash
# 同一个 prompt 运行两次，比较 URL 是否变化
# 第一次
mvn exec:java -Dexec.mainClass="ImageDemo"
# 第二次（再跑一次）
mvn exec:java -Dexec.mainClass="ImageDemo"
```

**预期**：URL 每次不同（每次生成新图），这意味这 **不能把 URL 当永久链接**。

### 闯关任务

| 难度 | 动手 | 过关标准 |
|------|------|----------|
| ★ | 同 prompt 跑两次 | 确认 URL 变化，理解不能当永久链 |
| ★★ | 下载并 SHA-256 | 入库前需要做去重 |
| ★★★ | 检查 Response 中的 `revised_prompt` | 了解模型可能改写你的 prompt |

```java
// 检查 revised_prompt
String revisedPrompt = response.content().revisedPrompt();
if (revisedPrompt != null) {
    System.out.println("Model revised prompt: " + revisedPrompt);
}
```

### 可能遇到的坑

| 坑 | 表现 | 解法 |
|----|------|------|
| 临时 URL 当永久链 | 七天后 403 | 下载到对象存储 |
| 未设预算 | 一夜打穿配额 | 设置预算告警 |
| 无人工审核对公众发图 | 合规事故 | 审核流程前置 |

## 4. 项目总结

### 优点与缺点

| 维度 | LangChain4j `ImageModel` | 直连 HTTP | 专用 Python 服务 |
|------|-------------------------|-----------|-----------------|
| 依赖治理 | 与 BOM 一致 | 自管 | 双栈 |
| 单测 | 易 mock | 中 | 中 |
| 合规责任 | 仍在业务 | 仍在业务 | 仍在业务 |

### 适用 / 不适用场景

**适用**：营销草稿、配图探索、内部设计协作、与文本资源隔离部署。

**不适用**：高 QPS 同步阻塞热路径、强实时视频。

### 常见踩坑

1. 临时 URL 当前端永久资源
2. 未设预算告警，图像打穿配额
3. 无审核对公众自动发图

### 进阶思考题
