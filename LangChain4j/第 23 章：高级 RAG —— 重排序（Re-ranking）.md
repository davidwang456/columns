# 第 23 章：高级 RAG —— 重排序（Re-ranking）

## 1. 项目背景

### 业务场景（拟真）

**Bi-encoder 向量召回** 要快、要能 **海选** 大批候选，但对 **细粒度相关性**（多义词、跨句推理）不足。**Re-ranking** 用 **更强更慢** 的模型（cross-encoder 或小 LLM）对 **候选列表** 重打分，优化 **topN** 再交给生成模型。

### 痛点放大

代价：**额外延迟与费用**。工程上常见 **两阶段**：**粗召回 topK 大**、**精排后 topN 小**。与第 21 章对比：**压缩** 处理 **输入**；**重排** 处理 **召回侧**。示例：[`_03_Advanced_RAG_with_ReRanking_Example.java`](../../langchain4j-examples/rag-examples/src/main/java/_3_advanced/_03_Advanced_RAG_with_ReRanking_Example.java)。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：为啥不一开始就用最强调检索？

**小白**：**K 与 N** 咋定？**能用大模型 rerank** 吗？**英文 rerank 中文** 会怎样？

**大师**：**成本与延迟**——超强模型扫全库 **不可扩展**。K/N 靠 **压测** 与 **Recall@N vs P95 vs 账单**。大模型 rerank **仅小候选集** + **缓存/超时降级**。**语言域不匹配** → 排序可信度降——用 **多语言 reranker** 或 **分路**。**技术映射**：**两阶段 = 可扩展 vs 可精**。

**小胖**：rerank 会把合规段落排后面吗？

**小白**：**失败怎么办**？和 **MMR** 啥关系？

**大师**：**训练偏见** 或 **候选缺文档** 会误排——保留 **audit log** 与 **人工复核**。失败：**超时/异常 → 使用向量序子集** 并提示 **检索质量可能受限**。**MMR** 偏 **多样性**；**cross-encoder** 偏 **相关性**——可 **串联**。**技术映射**：**rerank 降级 = SLA 必备**。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | Re-ranking | 仅向量 topN | 超大 cross-encoder 全库 |
|------|------------|-------------|---------------------------|
| 相关性 | 高 | 中 | 理论高 |
| 延迟/成本 | 中 | 低 | 不可接受 |
| 典型缺点 | 运维复杂 | 胡说风险 | 不可用 |

### 适用场景

- **高精度** 知识助理、**法条/规格**、长文档库。

### 不适用场景

- **低延迟聊天**、**成本敏感** 且 **召回已足够好**。

### 注意事项

- **监控 rerank 降级率**；**数据主权**（部分 rerank API 境外）。

### 常见踩坑经验（生产向根因）

1. **粗召回过小** → rerank **无力回天**。  
2. **未设超时** 拖垮 SLA。  
3. **语言不一致**。

### 进阶思考题

1. **离线黄金集提升微弱** 时，上线决策树？  
2. **reranker 与 generator 同厂商** vs **异构** 的故障隔离？
