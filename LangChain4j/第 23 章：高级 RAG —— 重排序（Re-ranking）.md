# 第 23 章：高级 RAG —— 重排序（Re-ranking）

## 1. 项目背景

**Bi-encoder 向量召回**追求 **快**，在巨大语料上先 **海选** 一批候选；但它对 **细粒度相关性**（例如「同一词多义」「需要跨句推理」）往往不足。**Re-ranking（重排序）** 在第二阶段用 **更强但更慢** 的模型（常见为 cross-encoder，或小型 LLM）对 **候选列表** 重新打分，**优化 topN 的排序**，再送给最终生成模型。

代价非常现实：**额外延迟**与 **额外费用**。因此工程上几乎总是 **两阶段**：**粗召回 topK 大**、**精排后 topN 小**。示例：

- `langchain4j-examples/rag-examples/src/main/java/_3_advanced/_03_Advanced_RAG_with_ReRanking_Example.java`

本章与第 21 章「压缩」对比：**压缩**处理 **输入侧**；**重排**处理 **召回结果侧**。

## 2. 项目设计：大师与小白的对话

**小白**：为什么不一开始就用最强的检索？

**大师**：**成本与延迟**。超强模型扫描全库 **不可扩展**；必须先 **向量或倒排索引海选**。

**小白**：K 和 N 怎么定？

**大师**：靠 **压测**：观察 **Recall@N** 相对 **仅向量 topN** 的提升，对比 **P95 延迟**与 **账单**。

**小白**：能用大模型做 reranker 吗？

**大师**：可以，但通常 **只对小候选集**；并要 **缓存**与 **超时降级**。

**小白**：英文模型 rerank 中文段落会怎样？

**大师**：**语言域不匹配**时，排序可信度 **显著下降**；要选用 **多语言 reranker** 或 **按语言分路**。

**小白**：rerank 会不会把 **合规段落**排到后面？

**大师**：会，如果训练 **偏见**或 **候选本身缺文档**。要保留 **audit log** 与 **人工复核入口**。

**小白**：失败时怎么办？

**大师**：**超时/异常 → 使用向量排序的子集** 并 **降低模型「过度自信」**（在提示中声明「检索质量可能受限」）。

**小白**：和 MMR（最大边际相关）的关系？

**大师**：**MMR** 更像 **多样性重排**；**cross-encoder** 更像 **相关性重排**——可以 **串联**。

**小白**：如何验证 rerank 真有用？

**大师**：**离线黄金集** + **线上 A/B**；若离线提升微弱，谨慎上线。

## 4. 项目总结

### 优点

- **显著提升** topN 相关性，减少 **胡说** 与 **抄错片段**。  
- 对 **法规/医疗** 等 **高后果**场景价值大。

### 缺点

- **延迟叠加**；**系统复杂度**上升。  
- **额外模型**运维与配额。

### 适用场景

- **高精度**知识助理、**长文档库**、**法条/规格**检索。

### 注意事项

- **监控 rerank 降级率**。  
- **数据主权**：部分 rerank API **境外**。

### 常见踩坑

1. **粗召回过小** → rerank **无力回天**。  
2. **未设超时** 拖垮整体 SLA。  
3. **忽略语言一致性**。
