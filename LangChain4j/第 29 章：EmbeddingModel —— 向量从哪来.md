# 第 29 章：EmbeddingModel —— 向量从哪来

## 1. 项目背景

### 业务场景（拟真）

RAG **召回上限** 很大程度上由 **嵌入模型** 决定：语言域（中英）、领域（法律/代码）、**维度**、**量化** 影响 **精度与吞吐**。平台组要在 **本地 CPU ONNX** 与 **云端 embed API** 间选型，平衡 **成本、延迟、数据出境**。

### 痛点放大

**换嵌入模型** 通常需 **全量重建索引**（迁移映射一般不划算）。若 **存与搜模型不一致**：**语义空间** 漂移 → **指鹿为马**。Aggregator 下 `embeddings/langchain4j-embeddings-*` 提供 **可打包的本地模型**；云端用各 provider 的 `EmbeddingModel`。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：嵌入像给每句话发「身份证照片」？

**小白**：**聊天模型能当嵌入用吗**？**换模型要重建索引吗**？**量化 vs 全精度**？

**大师**：嵌入一般是 **单独模型族**。换模型 **要重建**（破坏性变更）——**蓝绿索引**。**量化** 省内存加速，**略损召回**——用 **离线 hit@k** 决策。**技术映射**：**embedding 版本须写入 metadata（第 24 章）**。

**小胖**：多语言文档咋办？

**小白**：**合规** 上文本出不出境？

**大师**：**多语言向量模型** 或 **分语言分索引再融合**。**本地模型** 数据主权友好；云端 API 要 **法务评估**。**技术映射**：**批量吞吐 = 批大小 × 线程 × ONNX 参数**。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | 本地 ONNX embed | 云端 embed API | 与 chat 同厂商 |
|------|-----------------|----------------|----------------|
| 成本/延迟 | 可控 | 按量 | 视套餐 |
| 合规 | 易 | 难 | 视区域 |
| 典型缺点 | 运维模型文件 | 出境 | 绑定 |

### 适用场景

- 大批量历史索引；**数据不出域** 政企。

### 不适用场景

- **极小文档量**——过度工程。

### 注意事项

- **维度一致**；**预处理** 与训练一致。

### 常见踩坑经验（生产向根因）

1. **中英混用错选纯英文模型**。  
2. **embed 与 chat 语言不一致**。  
3. **无版本号** 无法复现线上问题。

### 进阶思考题

1. **embedding 与 chat 不同厂商** 时幻觉风险在哪？  
2. **CPU pinning** 在 K8s 内对批嵌入的影响？
