# 第 15 章：动态工具注册

## 1. 项目背景

静态 `new Calculator()` 适合 Demo；真实系统里，不同租户、不同用户角色、甚至「插件市场」下的 **可见工具集合**会变化。动态工具能力允许在运行时 **增删或筛选** 工具元数据，使模型 **只看到当前上下文允许** 的那一批。典型风险是：工具描述 **过多过长** → 提示膨胀；或 **权限泄露** → 模型学会调用不该调的东西。

教程 `_11_ServiceWithDynamicToolsExample.java`（`langchain4j-examples/tutorials/src/main/java/_11_ServiceWithDynamicToolsExample.java`）展示动态注册思路。本章与第 14 章共享安全原则，但强调 **策略化治理**。

## 2. 项目设计：大师与小白的对话

**小白**：动态是不是等于「模型自己下载插件」？

**大师**：不是。**代码仍在你的 JVM**；动态的仅是 **暴露哪些描述与入口**。

**小白**：如何避免描述爆炸？

**大师**：**分层**：先路由意图，再展开子工具集；或使用 **向量检索工具名**（主模块有 tool search 策略相关类，按需读源码）。

**小白**：如何做 AB？

**大师**：**按 user/tenant hash** 切工具包版本，并把实验指标 **埋到同一套观测**。

**小白**：CI 怎么测？

**大师**：为 **每类角色**准备一组 **期望可见工具名列表**，对生成 schema 做快照（脱敏）。

**小白**：和热部署冲突吗？

**大师**：要避免 **旧会话句柄** 指向已卸载工具；需要 **会话世代号**。

**小白**：和 MCP 什么关系？

**大师**：MCP 更像 **进程级协议**；动态工具是在 **单次 JVM 调用链** 内筛选 provider。二者可组合（第 37 章）。

## 3. 项目实战：主代码片段

> **场景入戏**：静态工具像 **固定技能栏**；动态工具像 **MMO 天赋树**——点错一行配置，团灭（泄露高危 API）的可能**比 boss 技能还高**。

请打开 [`_11_ServiceWithDynamicToolsExample.java`](../../langchain4j-examples/tutorials/src/main/java/_11_ServiceWithDynamicToolsExample.java)，完成：

1. **荧光笔**标出 **运行时决定工具列表** 的入口（方法或 Builder）。  
2. 自问：**新增租户策略** 要改 **几处**？理想答案：**一处策略类** + **配置**，别散落十个 `if`。  
3. 在笔记本写伪代码：`if (role == ADMIN) allowDangerousTools()` ——然后 **立刻划掉**，改成 **PolicyEngine.evaluate(context)`**。

**调试彩蛋**：在日志里打印 **每次请求的工具 schema 摘要（哈希）**，你会震惊于：**多十个工具**，延迟与 **prompt 长度**怎么一起飞——这就是 **工具爆炸**。

本文不粘贴全文以防版本差异。

### 延伸案例（情景演练）：多租户「工具超市」与提示膨胀

**背景**：某 HR SaaS 为 **不同企业客户** 开放可选插件：`打卡查询`、`薪资估算（沙箱）`、`假期政策（只读）`。若 **一次性** 把 **二十个工具描述** 塞进系统提示，**gpt-4o-mini** 在长会话里开始 **叫错工具名**。团队的动态策略是：**首轮只暴露「路由工具」+「2～3 个高频工具」**；当用户明确说出「假期」时再 **惰性展开** 该域工具列表（记录一次 **工具集变更事件** 便于追踪）。

**事故变体**：测试环境 **误把 ADMIN 工具包**装配给 **试用租户**，导致模型 **尝试调用** `exportAllEmployees()`——幸好在 **工具实现层**二次校验 **租户 id** 拒绝。**教训**：**动态≠自动安全**；**每个工具方法仍要是「零信任」**。可将本案例改编为 **红蓝对抗演练** 脚本：蓝方只拿到 **普通租户 token**，红方观察日志里 **工具候选集**是否泄漏高危名。

## 4. 项目总结

### 优点

- **多租户**与 **插件化** 的自然实现。  
- 可随业务开关 **灰度 AI 能力**。

### 缺点

- **可观测复杂度**上升。  
- **调试**需记录每次 **工具候选集** 快照。

### 适用场景

- SaaS、企业内 **按部门开放不同能力**。  
- 「技能包」式产品运营。

### 注意事项

- **工具 cardinality 上限**。  
- **描述中不要含密钥或内网 URL**。

### 常见踩坑

1. **提示超长**导致模型忽略工具规格。  
2. **缓存**了错误版本的工具列表。  
3. **并发修改**工具注册表。
