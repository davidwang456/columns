# 第 15 章：动态工具注册

## 1. 项目背景

### 业务场景（拟真）

某 SaaS 平台有不同付费等级：免费版用户只能使用基础工具（查天气、算计算器）；专业版用户能看到数据导出、报表生成工具；企业版用户还能看到管理员工具（用户管理、审计日志导出）。如果一次性把所有工具的描述都塞给模型——即使免费用户用不了管理员工具——模型会在生成的回答里提到「你可以使用用户管理功能」，而免费用户根本没有这个功能。这造成了产品承诺与实际权限之间的不一致。

解决方案是 **动态工具注册**：每次请求时，根据当前租户的角色和权限，**只暴露该租户有权看到的那些工具的描述和调用接口**。

### 痛点放大

沿用静态全量工具列表时，三个典型问题：

- **提示膨胀**：20 个工具，每个平均 200 tokens 描述——光工具列表就占了 4000 tokens，模型处理长列表时更容易选错工具名。
- **权限幻觉**：模型看到了「删除用户」工具的描述（尽管调用时会被拒绝），但它可能对用户说「我可以帮你删除用户」——承诺了不存在的功能。
- **安全假象**：「模型看不到所以安全」是一个错误假设。正确的安全模型是：即使模型看到了，服务端也会拒绝。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：动态工具是不是模型自己上网下插件——像手机应用商店，需要什么功能就去装一个？

**大师**：你正好说反了——不是模型去下载代码，而是在调用模型之前，你的代码已经决定好了「模型可以看到哪些工具」。代码始终在你的 JVM 里，动态的只是 **元数据描述**。这就像餐厅给不同顾客不同菜单——VIP 菜单上有「私人订制」，普通菜单上没有——但两种菜单上的菜都是同一个厨房做的。

**小白**：如何避免工具描述爆炸？企业版 50 个工具每次全量发太贵了。AB 实验怎么做？

**大师**：三个策略：① **分层路由**——第一轮只暴露 2-3 个路由工具（`navigateTo(department)`、`searchTools(keyword)`），模型先确定意图域，再惰性展开该域下的工具。② **工具名检索**——LangChain4j 的 `service.tool.search.*` 提供了类似搜索引擎的机制，模型先搜再调。③ AB 实验——按 tenant hash 分工具包，指标统一观测。**技术映射**：**动态工具 = 元数据筛选层，不是代码热加载；给模型看什么和代码里有什么是两个完全不同的关注点——前者影响决策质量，后者影响系统安全**。

**小胖**：那 ADMIN 工具的描述误给试用租户咋办？免费用户调「删除所有订单」呢？

**大师**：**最危险的安全假设就是认为「模型看不到所以安全」。** 正确的安全模型是 **零信任**：即使模型看到了 ADMIN 工具的描述甚至发起了调用，`@Tool` 方法内部第一行就拒绝。动态工具只是「效率优化层」——减少无关描述噪声、降低 token 成本。安全层永远在工具方法内部：取当前认证用户的租户 ID，校验权限，不通过就直接返回错误。**技术映射**：**动态工具解决的是性能和质量问题（模型选错工具、token 浪费），安全靠每个工具方法内部独立的零信任校验——两件事要用两层，不能把安全寄托在哪一层上**。

### 步骤 1：根据角色动态提供工具

```java
import dev.langchain4j.agent.tool.Tool;
import dev.langchain4j.service.AiServices;
import java.util.*;

// 两个不同的工具集
class AdminTools {
    @Tool("Deletes a user account (admin only)")
    String deleteUser(String userId) {
        // 即使被调用，内部也会校验权限
        return "User " + userId + " deleted.";
    }
    
    @Tool("Views system logs")
    String viewLogs() {
        return "System logs: [2024-01-01] all systems normal.";
    }
}

class UserTools {
    @Tool("Views your own profile")
    String viewProfile() {
        return "Your profile: John Doe, joined 2024";
    }
    
    @Tool("Submits a support ticket")
    String submitTicket(String description) {
        return "Ticket created: " + description;
    }
}

// 动态工具提供者
class DynamicToolProvider {
    
    private final AdminTools adminTools = new AdminTools();
    private final UserTools userTools = new UserTools();
    
    public Object[] getToolsForRole(String role) {
        if ("admin".equals(role)) {
            return new Object[]{adminTools};
        } else {
            return new Object[]{userTools};
        }
    }
}

interface Assistant {
    String chat(String userMessage);
}

public class DynamicToolsDemo {

    public static void main(String[] args) {
        ChatModel model = OpenAiChatModel.builder()
                .apiKey(System.getenv("OPENAI_API_KEY"))
                .modelName(GPT_4_O_MINI)
                .build();

        DynamicToolProvider provider = new DynamicToolProvider();
        
        // 模拟请求线程变量
        String currentRole = "user";  // 改 "admin" 试试
        
        Assistant assistant = AiServices.builder(Assistant.class)
                .chatModel(model)
                .tools(provider.getToolsForRole(currentRole))
                .build();

        // 普通用户看不到 admin 工具
        String result = assistant.chat("Can you delete user 123?");
        System.out.println("[" + currentRole + "] " + result);
    }
}
```

class RouterTool {
    
    private final Map<String, Object[]> toolMap = new HashMap<>();
    
    RouterTool() {
        toolMap.put("sales", new Object[]{new SalesTool()});
        toolMap.put("support", new Object[]{new SupportTool()});

### 延伸案例：工具超市

多租户「工具超市」——首轮只暴露路由工具 + 高频工具；用户说了「假期」再惰性展开。事故案例：试用租户误装了 ADMIN 工具包，但工具实现层二次校验租户拒绝执行。

### 可能遇到的坑

| 坑 | 表现 | 解法 |
|----|------|------|
| 一次性塞 20+ 个工具 | 提示膨胀、模型叫错工具名 | 分层路由，先高频后惰性 |
| 缓存错误版本的工具列表 | 工具已改名模型还在调旧的 | 验证缓存失效机制 |
| 并发修改注册表 | 运行时不一致 | 用不可变集合 |

## 4. 项目总结

### 优点与缺点

| 维度 | 动态 ToolProvider | 静态全量工具 | MCP 外置工具 |
|------|-----------------|-------------|-------------|
| 多租户 | 强 | 弱 | 强 |
| 提示长度 | 可控 | 易爆炸 | 视协议 |
| 调试 | 需候选集快照 | 简单 | 中 |

### 适用 / 不适用场景

**适用**：SaaS、按部门开放能力、技能包运营。

**不适用**：仅单租户工具少于 5 个（静态更简单）、无 PolicyEngine 与审计。

### 常见踩坑

1. 提示超长 → 模型忽略工具规格
2. 缓存错误版本的工具列表
3. 并发修改注册表
