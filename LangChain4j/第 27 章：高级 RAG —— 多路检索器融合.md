# 第 27 章：高级 RAG —— 多路检索器融合

## 1. 项目背景

### 业务场景（拟真）

企业数据不止 **一个向量库**：可有 **ES 关键词**、**Neo4j**、**工单 SQL**、**CRM API**。多个 `ContentRetriever` **并行拉取** 再 **融合、重排或 LLM 二次筛选**。`_07_Advanced_RAG_Multiple_Retrievers_Example.java` 演示 **异构检索** 同环问答。

### 痛点放大

若 **简单拼接片段**：**上下文重复**、**超长**；若 **一路超时无熔断**：**拖死整请求**；若 **未做租户过滤**（第 25 章）：**跨租户污染**。融合算法：**RRF**、**加权归一化**、**拼集后再 rerank**。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：多路像「豆瓣+IMDb+硬盘」一起搜？

**小白**：和 **路由** 啥不同？**一路超时**？**重复片段**？

**大师**：**路由** 选主路径；**多路** **并行再合并**——可先路由域再域内多路。**超时** → **部分成功** + **提示缺失源**，勿 **假装综合** 未检索数据。**去重**（hash/shingle）或 **MMR**。**技术映射**：**融合前必须租户过滤**。

**小胖**：成本咋控？

**小白**：**观测** 看啥？

**大师**：**便宜路宽召回、贵路窄召**；或 **按意图选路**。每路 **latency、hit、失败率**；融合后 **token 量**。**技术映射**：**可解释排序 = 审计基础**。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | 多路融合 | 单路向量 | 手工拼 SQL |
|------|----------|----------|------------|
| 召回覆盖 | 高 | 中 | 高 |
| 复杂度 | 高 | 低 | 中 |
| 典型缺点 | 故障面大 | 异构弱 | 难维护 |

### 适用场景

- 企业搜索中台、复杂客服（政策+订单+知识库）；**迁移期** 新旧索引并存。

### 不适用场景

- **单一数据源已足够**——勿过度工程。

### 注意事项

- **安全上下文** 传到每路；**最终排序** 可解释。

### 常见踩坑经验（生产向根因）

1. **简单拼接** → 重复与超长。  
2. **一路失败** 未捕获。  
3. **未记录** 哪路贡献断言。

### 进阶思考题

1. **RRF** 与 **加权分数** 的 A/B 设计？  
2. **CRM JSON 一行** 错误当长文本 → **摘要化表格** 的边界？
