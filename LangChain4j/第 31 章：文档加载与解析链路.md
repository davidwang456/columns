# 第 31 章：文档加载与解析链路

## 1. 项目背景

### 业务场景（拟真）

团队的 RAG 管线接入了各种格式的原始文档：PDF 扫描件（发票、合同）、Word 文档（产品说明书）、HTML 页面（公司官网 FAQ）。如果只是用最原始的 `Files.readString()` 来读取，这些文件不是乱码就是空字符串——向量库里存的全是噪声。**Document Loader** 负责拉取字节流（从本地文件、S3、HTTP），**Document Parser** 负责把字节流转为可读的 `Document` 对象。

### 痛点放大

解析质量直接决定了 RAG 管线质量的上限——如果你连 PDF 里的表格都解析不出来，那向量库里的内容就是残缺的。`langchain4j` 在 `document-loaders/` 和 `document-parsers/` 目录下提供了多种适配器。安全方面也同样重要——上传的文档可能携带宏病毒。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：Loader 和 Parser——是不是就像做饭前先洗菜切菜？不管后面炒得多好，菜没洗干净就白搭？

**大师**：这句话可以刻在 RAG 工程团队的墙上——**解析质量 = RAG 天花板**。你后面用再好的 embedding 模型、再贵的 LLM，如果原始文档的内容在解析阶段就被静默丢失了，那就已是无用功。比如一个 PDF 扫描件实际上是图片——文本提取结果可能是空字符串，但你的管线没有检查解析后的文本长度，直接把这个空串喂给了 embedding 模型——向量库里就多了一条空数据的向量，检索时可能命中但它毫无意义。

**小白**：那个入库前的最小字符数门禁——是强制要求的吗？

**大师**：**在我看应该是「必须的」。** 一条解析后字符数少于 50（或具体阈值根据业务定）的 segment，就不应该进入索引。这能拦截掉大多数因为解析失败造成的空内容或噪声内容。这是一个在代码里加上 if 语句就能解决的事，但它能省掉后面大量的排障时间。**技术映射**：**入库前的最小字符数门禁是最简单有效的防沉默失败手段——与其让空文档占用向量库空间、产生无意义召回，不如一开始就拦住并告警**。

## 3. 项目实战

### 环境准备

```bash
cd langchain4j-examples/rag-examples
# PDF 解析需要额外依赖（pom.xml 中加入）
```

### 分步实现

#### 步骤 1：用不同 Parser 加载不同格式

```java
import dev.langchain4j.data.document.Document;
import dev.langchain4j.data.document.loader.FileSystemDocumentLoader;
import dev.langchain4j.data.document.parser.TextDocumentParser;

// TXT 文件
Document txtDoc = FileSystemDocumentLoader.loadDocument(
    java.nio.file.Paths.get("documents/readme.txt"), new TextDocumentParser());
System.out.println("TXT: " + txtDoc.text().length() + " chars");

// PDF 文件（需引入 pdfbox 依赖，使用 PdfDocumentParser）
// Document pdfDoc = FileSystemDocumentLoader.loadDocument(
//     Paths.get("documents/manual.pdf"), new PdfDocumentParser());
// System.out.println("PDF: " + pdfDoc.text().length() + " chars");
```

#### 步骤 2：空文档门禁

```java
int MIN_CHARS = 50;
Document doc = parser.parse(inputStream);

if (doc.text().trim().length() < MIN_CHARS) {
    System.err.println("SKIP: Document too short (" + doc.text().length() 
        + " chars < " + MIN_CHARS + "), possible parse failure.");
    metrics.counter("document.rejected", "reason", "too_short").increment();
    return;  // 跳过，不进入向量库
}

// 通过门禁后，继续后续步骤
```

#### 步骤 3：对比解析前后依赖

```bash
# 只引 TextDocumentParser 的依赖树
mvn dependency:tree -Dincludes=langchain4j-document-parser

# 加了 PdfDocumentParser 后的依赖树
mvn dependency:tree -Dincludes=pdfbox
# 注意观察传递依赖膨胀
```

### 可能遇到的坑

| 坑 | 表现 | 解法 |
|----|------|------|
| 解析空未发现 | 空 segment 进索引 | 字符数门禁 |
| HTML 导航栏噪声 | 向量库索引了页头页脚无用内容 | 用 HTML Parser 过滤 main 区域 |
| 编码错误 | 中文乱码 | 指定 UTF-8 |

## 4. 项目总结

### 优点与缺点

| 维度 | 模块化 Loader/Parser | 手写读文件 | 外包解析服务 |
|------|--------------------|----------|------------|
| 格式覆盖率 | 高 | 窄 | 视供应商 |
| 依赖膨胀 | 可大（引入 Tika） | 小 | 联调成本 |
| 典型缺点 | 需治理依赖树 | 静默失败 | 出境 |

### 适用场景

- 知识库批量入湖、合规归档检索

### 不适用场景

- 仅纯 txt 小规模（可极简）

### 常见踩坑

1. 解析空未发现直接索引
2. HTML 导航栏噪声
3. 中文编码乱码
