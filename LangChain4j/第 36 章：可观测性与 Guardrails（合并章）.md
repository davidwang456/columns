# 第 36 章：可观测性与 Guardrails（合并章）

## 1. 项目背景

**可观测性**回答「**多慢、多贵、错在哪**」：`langchain4j-micrometer-metrics` 与 `langchain4j-observation` 帮助把 **模型调用、工具执行、向量检索**纳入统一指标与追踪。**Guardrails（护栏）**回答「**该不该回**」：输入/输出策略、正则拦截、二次审核模型——模块 `langchain4j-guardrails`。

上线 LLM 应用，**两者缺一不可**：没有指标你在 **财务事故** 面前失明；没有护栏你在 **合规事故** 面前失守。本章合并原大纲 **第 38～39 章** 的主题，便于运维与风控同一页面落地。

## 2. 项目设计：大师与小白的对话

**小白**：Micrometer 和 OpenTelemetry 冲突吗？

**大师**：概念层都 **可对接**；关键是选 **单一真理源** 做 **trace id** 串联（HTTP → LLM → DB）。

**小白**：要采哪些指标？

**大师**：**延迟 P95/P99**、**token 用量**、**工具调用率**、**检索空结果率**、**429/5xx**、**解析失败率**（与第 17 章联动）。

**小白**：Guardrails 放调用前还是后？

**大师**：通常 **两枪**：**入站清洗** + **出站审查**；高风险再加 **人工**。

**小白**：护栏会误杀吗？

**大师**：会。要有 **申诉通道**与 **动态调级**（灰度发布）。

**小白**：日志脱敏如何做？

**大师**：**结构化日志字段级** Redaction；禁止 **stdout 全文 prompt**。

**小白**：与 API 网关风控区别？

**大师**：网关防 **传统攻击**；护栏专注 **提示注入/敏感生成**。

**小白**：成本？

**大师**：**二次模型审核**最贵；可用 **轻量分类器** + **抽样**。

## 3. 项目实战：主代码片段

> **场景入戏**：没有 Listener 指标，只有 **HTTP 200**——像 **只看体温正常** 却不管 **体内癌细胞在开会**。**Guardrails** 则是 **安检仪**：太松漏 **PII**，太严把 **合法教材**当违禁品夹走（延伸案例 **账单惊雷** vs **误杀**）。

结合 Spring 示例 [`MyChatModelListener.java`](../../langchain4j-examples/spring-boot-example/src/main/java/dev/langchain4j/example/aiservice/MyChatModelListener.java) 思路，列出你要打到 **Micrometer**（或 OTel）的 **tags**：`model`, `route`, `tenant`, **`approxTokenTotal`（若可）**。

草拟 **Guardrails 策略表**：

| 阶段 | 策略 | 失败动作 |
|------|------|----------|
| 入站 | 邮箱/身份证正则屏蔽 | 422 + 提示 |
| 出站 | 违禁词 | 替换为安全文案 |

具体 API 以 **`langchain4j-guardrails`** 当前文档为准。

### 延伸案例（情景演练）：一次「账单惊雷」与护栏误杀

**案例 A（观测）**：某教育 SaaS 在 **晚高峰** 发现 **云账单中 embedding 调用突增 400%**。Grafana 上若仅有 **`http_server_requests`**，几乎无法定位——团队在 **`ChatModelListener`** 中增加 **`modelId`、`route`、`tenant`、近似 `tokenTotal`** 标签后，才看到 **某租户**在 **批量重放**历史作业。处置：**限流** + **租户级预算** + **夜间队列**。没有 **Micrometer/OTel** 这一层，你只能 **靠猜**。

**案例 B（护栏）**：上线首周，**身份证号**被 **站内合法教程**误触发 **出站替换**，用户怒斥「你们在掩盖什么」。根因是 **正则过宽** + **未结合上下文**。改进：**(1)** 入站与出站 **分级**：`\d{17}[\dXx]` 仅在 **`highRiskChannel`** 才拦截；**(2)** 规则 **版本化**与 **灰度**；**(3)** **申诉工单**自动附带 **规则 id**。这组故事提醒：**护栏与指标同样是「产品」**，需要 **产品经理**参与评审。

## 4. 项目总结

### 优点

- **平台化**：Listener/Filter 模式 **复用**。**缺点**：**误杀/漏杀**需持续调参。

### 适用场景

- 所有 **对公** Bot；**金融/未成年人**相关。

### 注意事项

- **审计**需求：**谁在何时改了哪条规则**。  
- **跨国**法域条款差异。

### 常见踩坑

1. **指标基数爆炸**（每个用户一个 tag）。  
2. **护栏日志**泄露更多隐私。  
3. **只看黄金指标**忽略 **解析失败**。
