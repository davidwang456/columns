# 第 5 章：模型参数 —— 温度、超时与日志

## 1. 项目背景

### 业务场景（拟真）

某 SaaS 团队维护着一套 Java 客服系统，同时服务两个产品线：**「智能客服摘要」**要求输出简洁、客观、一字不差地反映客户原意；**「营销文案生成」**则要求语气活泼、有创意、能吸引点击。两个产品线共用同一套 Java 服务、同一个 `ChatModel` 配置。

运营团队开始抱怨：「摘要太飘了——客户说『我要退款』，模型给我概括成『客户表达了对服务的不满』，情绪色彩加太多了！」而另一边营销团队也在抱怨：「文案太死板了——每次生成的广告语都一个调调，跟模板填的一样！」同一个模型，同一个 prompt 结构，只是 **参数不同**，结果风格差距巨大。

### 痛点放大

语言模型 API 暴露了多个参数——temperature（采样随机性）、top_p（核采样）、maxTokens（输出长度上限）、stop（停止词）、timeout（超时）——它们直接影响：

- **账单**：maxTokens 设得太大会让模型产生比你预期长得多的回答，白白浪费 token；timeout 太短导致频繁重试，翻倍消耗。
- **延迟**：maxTokens 直接控制输出长度的上限，值设大了用户等得久。
- **可复现性**：temperature > 0 让模型每次回答不一样——客服摘要需要稳定一致的输出，但营销文案反而需要变化。

如果全环境用同一套参数，或者只凭感觉调温度，就会出现「摘要飘、文案死」的对立困境。更严重的是：

- **成本失控**：maxTokens 设得过大导致延迟拖垮用户体验，账单还翻倍。
- **无法复现**：温度高导致线上用户报了一个 bug，你用自己的账号复现不出一样的截图——因为模型每次回答都不同。
- **合规事故**：`logRequests(true)` 在生产环境打开后，所有用户问题和模型回答全文打了日志——其中包含用户的姓名、电话号码、订单号。PII 泄露。

`OpenAiChatModel.builder()` 把这些参数变成 **强类型的链式配置**，让你不需要手写 JSON 或记厂商字段名。更重要的是：参数变了，`ChatModel` 的调用形态不变——业务代码始终是 `model.chat(prompt)`。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：调 temperature 是不是跟调奶茶甜度一样——全糖一定更好喝？0.7 就是七分甜，1.0 就是全糖？

**大师**：奶茶的比喻大致对，但在语言模型里，「全糖」不一定更好。temperature 控制的是 **采样的随机性**：温度越高，模型越倾向于选概率较低的词——所以回答看起来更有「创意」，但也更可能跑题。温度越低，模型越保守，每次都选概率最高的词——所以回答稳定、可预测，但也可能显得单调。但是注意：**低温度不等于「更正确」**——如果你问「1+1=？」，温度 0 和温度 1 可能都回答 2，这个问题不受温度影响。但如果你问「给我推荐一本书」，温度 0 每次都推荐同一本，温度 0.7 可能每次推荐不同——正确性取决于你的评判标准。

**小白**：那 temperature 设为 0，是不是就 100% 可复现了？我可以在测试里断言两次输出完全一样吗？

**大师**：**不能依赖这个**。绝大多数实现即使 temperature=0，仍有极小概率的随机性——因为 GPU 计算的浮点数精度、批处理顺序等都可能引入微小差异。所以 **不要拿 temperature=0 当作精确复现的唯一手段**。严格的回归测试需要 **同时固定四样东西**：模型版本（`gpt-4o-mini-2024-07-18` 而不是 `gpt-4o-mini`）、提示词（精确到标点符号）、检索结果（如果用了 RAG）、随机种子（如果 provider 支持 `seed` 参数）。**技术映射**：**temperature = 采样随机性的控制旋钮，不是事实性的保证——低温度让输出更稳定，但不等于 100% 可复现**。

**小胖**：那 timeout 我设 600 秒，用户是不是就永远等得到？反正请求不会超时了。

**大师**：**千万不能这么想。** timeout 不是「设大就保险」的——它的两个极端都会出问题：太短（比如 1 秒）会让正常的长生成请求被误杀，导致频繁重试、用户体验差、token 被双重消耗；太长（比如 600 秒）会让线程/连接长期被一个慢请求占用，一旦并发上来，**连接池耗尽**，其他所有正常请求都被阻塞。正确的做法是 **根据你的 P95 延迟来标定 timeout**——如果 P95 是 15 秒，设 30 秒作为上限。并且 timeout 必须与 **重试策略（第 9 章）和断路器（第 36 章）** 一起配置，不能独立存在。**技术映射**：**timeout = 单个请求的最长等待时间，不是 SLA 保障——它保护的是你不被一个慢请求拖垮整个线程池，而不是保证请求一定成功**。

**小白**：`logRequests(true)` 这个方法看着很方便——能直接在生产上打开，把请求和响应全文都打到日志里方便排查吗？

**大师**：**极不建议在生产环境长期开启。** 请求体里可能包含完整的用户问题、系统提示、甚至 API Key 的片段；响应体里是模型生成的全文，可能包含 PII。如果你开了全文日志——这些数据未经脱敏就进了你的 ELK 或 Splunk——这就是一桩合规事件。正确做法：**开启请求日志只能在预发环境短时间 Debug 用**，并且开启前必须经安全审批。生产环境的观测应该用 **结构化指标和脱敏后的采样日志**——第 36 章的 `ChatModelListener` 是实现这个的标准方式。**技术映射**：**日志策略 = 可观测性与数据合规之间的折中——你需要在「能定位问题」和「不泄露用户数据」之间找到平衡点，而不是非黑即白地全开或全关**。

**小胖**：想省钱，我把 maxTokens 压到最小行不行？反正模型说多了我也用不上。

**大师**：压缩 maxTokens 确实省钱，但 **太短了会让输出被硬截断**——尤其是当你的下游期望 JSON 或工具调用时。想象一下：模型输出到一半，maxTokens 达到了上限，输出被截断——结果是 JSON 语法错误、工具调用参数不完整、或者句子只说了半句。解法：**先用一批离线样本验证业务可接受的最短长度**。你取 50 条真实用户请求，让模型用不同的 maxTokens（100、200、500）生成回答，然后人工判断「截断率」——找到那个保证 95% 的回答完整的最小值。**技术映射**：**maxTokens = 输出预算的上限，与账单呈线性关系，但与回答完整性呈阶梯关系——你需要在曲线上找到最经济的拐点**。

### 代码清单（全文）

```java
import dev.langchain4j.model.chat.ChatModel;
import dev.langchain4j.model.openai.OpenAiChatModel;
import static dev.langchain4j.model.openai.OpenAiChatModelName.GPT_4_O_MINI;
import static java.time.Duration.ofSeconds;

public class _01_ModelParameters {

    public static void main(String[] args) {

        ChatModel model = OpenAiChatModel.builder()
                .apiKey(System.getenv("OPENAI_API_KEY"))
                .modelName(GPT_4_O_MINI)
                .temperature(0.3)
                .timeout(ofSeconds(60))
                .logRequests(true)    // 只在调试时开，生产关
                .logResponses(true)   // 只在调试时开，生产关
                .build();

        String prompt = "Explain in three lines how to make a beautiful painting";

        String response = model.chat(prompt);

        System.out.println("Prompt: " + prompt);
        System.out.println("Response: " + response);
    }
}
```

### 可能遇到的坑

| 坑 | 表现 | 解法 |
|----|------|------|
| 双层超时 | 客户端 60s 但网关只有 5s | 确保客户端 timeout < 网关 timeout |
| 两行 `.temperature()` | 后者覆盖前者 | Builder 每个参数只设一次 |
| 日志全开生产 | PII 进集中日志平台 | 用配置审查或 Listener 脱敏 |
| maxTokens 过小 | JSON/工具结果被截断 | 离线验证业务可接受的最短长度 |

## 4. 项目总结

### 优点与缺点

| 维度 | Builder 强类型参数 | 手写 HTTP JSON | 仅控制台调参 |
|------|-------------------|---------------|-------------|
| 可审阅性 | 高（defaults 可见） | 低 | 无代码沉淀 |
| 迁移厂商 | 中（对照矩阵） | 高 | 不适用 |
| 日志风险 | 需治理 | 同样需治理 | 易忽略 |

### 适用 / 不适用场景

**适用**：需要文体稳定的摘要/分类（偏低温）、需要头脑风暴文案（偏高温+人工审核）。

**不适用**：完全不需要控制生成的一次性 curl 试验。

### 常见踩坑

1. maxTokens 过小 → 工具/JSON 断裂
2. 全局高温 → 线上风格漂移，无法复现工单截图
3. 日志全开 → PII 进集中日志平台
