# 第 5 章：模型参数 —— 温度、超时与日志

## 1. 项目背景

同一个问题用不同「旋钮」去问，模型会得到不同风格的答案。语言模型 API 通常暴露 **temperature**、**top_p**、**max_tokens**、**stop**、**timeout** 等参数。它们不仅影响「好不好看」，也直接影响：**账单（token）**、**延迟（生成长度）**、**是否可复现（采样随机性）**。在 Java 侧，`OpenAiChatModel.builder()` 把这些厂商字段变成强类型链式配置，并与 `ChatModel` 接口的使用方式保持不变。

运维同学往往更关心：**配额（RPM/TPM）**、**超时导致的 5xx**、**日志里能否区分不同模型别名**；开发同学则关心：**评测集上基线参数**与**提示词版本如何一起打 tag**。官方教程 `_01_ModelParameters.java`（`langchain4j-examples/tutorials/src/main/java/_01_ModelParameters.java`）用最小代码把这些旋钮一次性展示出来，是学习「可观测的第一步」的合适入口。

## 2. 项目设计：大师与小白的对话

**小白**：温度调低是不是就「更严肃、更正确」？

**大师**：**更低温通常更确定、更保守**，但不等价于「更正确」。幻觉与事实性主要取决于模型、提示与检索，而不是单靠把温度拉到 0。

**小白**：那温度为 0 就完全一样吗？

**大师**：多数实现仍会留有**极小随机性**或受批处理影响；不要把它当 **100% 可复现**的唯一手段——如需严格回归，要把 **模型版本、提示、检索片段、随机种子策略** 一并固定，并配合评测脚本。

**小白**：`maxTokens` 和「截断」是什么关系？

**大师**：它限制**模型输出**的长度上限。设太小，JSON、工具调用或长答案会被**硬截断**，下游解析直接失败；设太大，**成本与延迟**上升。

**小白**：timeout 设 60 秒会不会太大？

**大师**：要看 **P95 延迟与你的 SLA**。timeout 过短会误杀长尾请求；过长会**占满线程/连接**。应与 **重试策略**、**断路器** 一起看（第 9 章）。

**小白**：`logRequests(true)` 能开在生产吗？

**大师**：**极不建议**长期全文打开。请求里可能有 **PII、密钥片段、内部系统字段**。生产应使用 **脱敏**、**采样**、或专用 **观测链路**（第 36 章），而不是把整包 JSON 打到 stdout。

**小白**：运维怎么看「模型维度」的成本？

**大师**：把 **模型别名**、**输入输出 token**（若可得）、**错误码**打到指标与日志字段。LangChain4j 可在 `ChatModelListener` 侧统一收口（第 34 章 Spring 示例有_listener 示范）。

**小白**：我想省点钱，把 maxTokens 压得很低行吗？

**大师**：可以，但要先用 **离线样本** 验证「业务可接受的最短答案长度」。否则你会得到大量「半成品 JSON」与客服投诉。

**小白**：GPT-4o-mini 这类枚举名换了怎么办？

**大师**：用库内的 **`OpenAiChatModelName` 枚举**能减少拼写错误；若厂商新增模型而枚举滞后，可暂时用字符串（注意**版本发布说明**）。

## 3. 项目实战：主代码片段

> **场景入戏**：把 `ChatModel` 想成一台可调**奶泡厚度**的咖啡机——`temperature` 像奶泡旋钮（越大越「飘」），`maxTokens` 是杯子高度（越高账单和等待越久），`timeout` 是顾客耐心，`log*` 则是「是否允许路人拍你的配方小票」。

仓库中与官方一致的片段如下（节选）：

```java
ChatModel model = OpenAiChatModel.builder()
        .apiKey(ApiKeys.OPENAI_API_KEY)
        .modelName(GPT_4_O_MINI)
        .temperature(0.3)
        .timeout(ofSeconds(60))
        .logRequests(true)
        .logResponses(true)
        .build();

String prompt = "Explain in three lines how to make a beautiful painting";

String response = model.chat(prompt);
```

**仓库锚点**：[`langchain4j-examples/tutorials/src/main/java/_01_ModelParameters.java`](../../langchain4j-examples/tutorials/src/main/java/_01_ModelParameters.java)（源文件内注释指向 OpenAI Chat Create 官方文档，适合和 Builder 字段 **逐列对照**）。

#### 闯关任务（由浅入深）

| 难度 | 动手 | 你要观察什么 |
|------|------|----------------|
| ★ | 固定 `prompt`，`temperature` 取 `0.0` / `0.7`（各跑 3 次） | **随机性**：低温是否更「千篇一律」 |
| ★★ | `timeout` 改为 `ofMillis(1)` 再恢复 `ofSeconds(60)` | **异常类型名** + **是否可重试**——抄进你们「故障手册」词条 |
| ★★★ | 写 5 行脚本：循环改 `temperature`，把 **每次 completion 长度**记 CSV | 体会「旋钮 → 实际字数 → **账单**」这条**隐性链条** |

#### 挖深一层（原理与边界）

- **双层超时**：`builder.timeout` 管 **客户端**；网关 / LB 可能还有 **更短的 idle 超时**——P95 要在**整条链路上**量。  
- **`logResponses(true)`**：响应里常含 **用户问题复述**；在个保审计下属于**敏感日志**，生产请换 **脱敏 Listener**（第 36 章）。  
- **复制粘贴事故**：真实 PR 里偶见 **两行 `.temperature(...)`** 后者覆盖前者——Code Review 时盯 **Builder 终端差异**。  
- **与成本**：token 计费；`temperature` 不直接出现在发票行项目，但会通过**输出长度**偷走预算——参数变更走 **评审 + 评测集**。

## 4. 项目总结

### 优点

- Builder 把常用参数字段 **显式化**，代码审阅一眼能看到「defaults 被改过」。  
- `timeout` 与客户端层配置可组合，便于对齐网关超时。

### 缺点

- 不同厂商 **同名参数语义略有差异**，迁移时要对照矩阵。  
- **日志选项**容易被误用在生产，造成合规事件。

### 适用场景

- 需要 **文体稳定** 的摘要、分类、抽取（偏低温度）。  
- 需要 **头脑风暴/文案**（偏高温度，配合人工审核）。

### 注意事项

- **评测先行**：参数与提示词必须绑定版本号一起回归。  
- **租户隔离**：多租户应用要把 **模型名与参数** 配成可策略化管理，不硬编码在分支里。

### 常见踩坑

1. **maxTokens 过小** 导致工具/JSON **断裂**。  
2. **全局高温** 让线上输出风格漂移，无法复现用户截图。  
3. **日志全开** 把用户隐私打进集中日志平台。

### 本期给测试 / 运维的检查清单

**测试**：为「截断类失败」设计断言（答案明显未完、JSON 不完整）；用同一 fixture 跑 **低/中/高温度** 三组快照，标记为 **非确定性预期**，只比对结构不比对逐字。  
**运维**：将 `modelName`、`timeout`、429/5xx 与 **retry 次数** 纳入仪表盘；对 **logRequests/logResponses** 做配置审查，禁止预发/生产默认开启全文日志。
