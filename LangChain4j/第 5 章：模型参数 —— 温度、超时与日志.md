# 第 5 章：模型参数 —— 温度、超时与日志

## 1. 项目背景

### 业务场景（拟真）

某 SaaS 团队在 **客服摘要**与 **营销文案** 两条产品线上共用同一套 Java 服务，运营抱怨「摘要有时太飘、文案有时太死板」。同一个问题用不同「旋钮」去问，模型会得到不同风格的答案。语言模型 API 通常暴露 **temperature**、**top_p**、**max_tokens**、**stop**、**timeout** 等参数——它们影响 **账单（token）**、**延迟（生成长度）**、**是否可复现（采样随机性）**。在 Java 侧，`OpenAiChatModel.builder()` 把厂商字段变成强类型链式配置，而 **`ChatModel` 调用形态**可保持不变。

### 痛点放大

若 **全环境同一套参数** 或 **只凭感觉调温度**，会出现：**成本**上 `maxTokens` 过大拖垮延迟；**稳定性**上温度高导致 **无法复现用户截图**；**合规**上 `logRequests(true)` 把 **PII** 打进集中日志。运维关心 **RPM/TPM、超时 5xx**；开发关心 **评测集基线参数与提示版本打 tag**。官方教程 `_01_ModelParameters.java` 用最小代码展示旋钮，是 **可观测的第一步**。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：调温度是不是跟调奶茶甜度一样？全糖一定更好喝？

**小白**：温度低是不是就更「正确」？**温度为 0 就完全一样吗？** `maxTokens` 和「截断」啥关系？

**大师**：**更低温通常更确定、更保守**，但不等于「更正确」；幻觉取决于模型、提示与检索。多数实现仍有 **极小随机性** 或批处理影响——**不要当 100% 可复现**唯一手段；要严格回归需固定 **模型版本、提示、检索、随机策略**。**技术映射**：**temperature = 采样随机性旋钮，非事实性保证**。

**小胖**：那我 timeout 设 600 秒，用户是不是永远等得到？

**小白**：`logRequests(true)` 能开在生产吗？运维咋按 **模型维度** 看成本？

**大师**：timeout 要看 **P95 与 SLA**；过短误杀长尾，过长 **占满线程/连接**，须与 **重试、断路器** 一起看（第 9 章）。**极不建议**生产长期全文打日志——请求含 **PII、密钥片段**；应 **脱敏、采样或观测链路**（第 36 章）。成本要把 **模型别名、token、错误码** 打进指标；可用 **`ChatModelListener`**（第 34 章）。**技术映射**：**超时 = 单请求上限；日志策略 = 合规与排障的折中**。

**小胖**：想省钱把 maxTokens 压到最小行不行？

**大师**：先用 **离线样本** 验证「业务可接受的最短长度」，否则 **JSON/工具调用被硬截断**，下游解析全挂。**技术映射**：**maxTokens = 输出预算上限，与账单、截断强相关**。

## 3. 项目实战

### 环境准备

- `langchain4j-examples/tutorials` 可构建；有效 **API Key** 与网络（或换兼容端点）。  
- 仓库文件：[`_01_ModelParameters.java`](../../langchain4j-examples/tutorials/src/main/java/_01_ModelParameters.java)。

### 分步实现

**核心片段（节选，与仓库一致）：**

```java
ChatModel model = OpenAiChatModel.builder()
        .apiKey(ApiKeys.OPENAI_API_KEY)
        .modelName(GPT_4_O_MINI)
        .temperature(0.3)
        .timeout(ofSeconds(60))
        .logRequests(true)
        .logResponses(true)
        .build();

String prompt = "Explain in three lines how to make a beautiful painting";

String response = model.chat(prompt);
```

| 步骤 | 目标 | 操作与观察 |
|------|------|------------|
| 1 | 感受随机性 | 固定 `prompt`，`temperature` 取 `0.0` / `0.7` 各跑 3 次 |
| 2 | 认识超时异常 | `timeout` 改为 `ofMillis(1)` 再恢复，记录 **异常类型** |
| 3 | 链接成本 | 循环改 `temperature`，记录 **completion 长度** 与体感 |

**可能遇到的坑**：**双层超时**（客户端 vs 网关更短）；**两行 `.temperature`** 后者覆盖前者；**日志含用户问题复述**——生产须脱敏。

### 测试验证

- 预发/生产 **禁止** 默认 `logRequests(true)` 全文；用配置审查或 Listener。  
- 将 **模型名、timeout、429/5xx、retry** 纳入仪表盘（与运维对齐）。

**仓库锚点**：[`langchain4j-examples/tutorials/src/main/java/_01_ModelParameters.java`](../../langchain4j-examples/tutorials/src/main/java/_01_ModelParameters.java)（内附 OpenAI Chat Create 文档链接，可与 Builder **逐列对照**）。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | Builder 强类型参数 | 手写 HTTP JSON | 仅控制台调参 |
|------|-------------------|----------------|--------------|
| 可审阅性 | 高（defaults 可见） | 低 | 无代码沉淀 |
| 迁移厂商 | 中（对照矩阵） | 高 | 不适用 |
| 日志风险 | 需治理 | 同样需治理 | 易忽略 |
| 典型缺点 | 同名参数语义略有差异 | 易漏字段 | 不可回归 |

### 适用场景

- 需要 **文体稳定** 的摘要、分类、抽取（偏低温度）。  
- 需要 **头脑风暴/文案**（偏高温度 + 人工审核）。  
- 需要把 **timeout** 与网关 **SLA** 对齐时。

### 不适用场景

- **完全不需要控制生成**的一次性 curl 试验——不必引入 Builder 讨论。  
- **强依赖厂商独有采样参数**且 LangChain4j 未暴露——需查 **底层客户端扩展**或 issue。

### 注意事项

- **评测先行**：参数与提示词 **绑定版本** 回归。  
- **租户隔离**：模型名与参数 **策略化配置**，勿硬编码在分支里。

### 常见踩坑经验（生产向根因）

1. **maxTokens 过小** → 工具/JSON **断裂**。  
2. **全局高温** → 线上风格漂移，**无法复现**工单截图。  
3. **日志全开** → **PII** 进集中日志平台，合规事件。

### 进阶思考题

1. 如何用 **`ChatModelListener`** 在 **不打全文请求体** 的前提下记录 **延迟与 token**？（答案线索：第 34、36 章。）  
2. **GPT-4o-mini 枚举滞后于厂商新模型**时，如何 **安全** 使用字符串 `modelName`？（提示：版本说明 + 回归集。）

### 本期给测试 / 运维的检查清单

**测试**：为「截断类失败」设计断言（答案明显未完、JSON 不完整）；用同一 fixture 跑 **低/中/高温度** 三组快照，标记为 **非确定性预期**，只比对结构不比对逐字。
