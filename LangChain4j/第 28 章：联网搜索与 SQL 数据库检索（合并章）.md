# 第 28 章：联网搜索与 SQL 数据库检索（合并章）

## 1. 项目背景

### 业务场景（拟真）

向量库擅 **非结构化相似度**；**联网搜索** 补 **最新公共信息**；**SQL 数据库检索** 提供 **强结构化事实**（订单、库存、合同状态）。二者与 RAG 混用即 **混合检索**。合并章对应示例：

- `_08_Advanced_RAG_Web_Search_Example.java` —— `WebSearchEngine`  
- `_10_Advanced_RAG_SQL_Database_Retreiver_Example.java` —— SQL 检索器

### 痛点放大

**联网** 可能 **假新闻**（域名白名单、时间过滤、UI 标明来源）；**SQL** 怕 **注入与越权**（参数化、只读账号、RLS/视图）。**仲裁**：结构化与网络矛盾时 **通常以内部 DB 为准**。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：联网像「看窗外下不下雪」？

**小白**：联网会不会 **假新闻**？**SQL 最怕啥**？**两路并行** 谁说了算？

**大师**：联网要 **白名单、二次校验、来源标注**。**SQL：注入 + 越权**——**参数化、只读、行级安全**。并行时要 **仲裁规则**：**通常 DB > 向量 > 网络摘要**。**技术映射**：**事实源优先级写死**。

**小胖**：合规呢？

**小白**：**运维** 差异？

**大师**：**爬虫 ToS、跨境、隐私**——联网单独审计。**联网** 依赖 **API 配额**；**SQL** 依赖 **连接池与只读副本延迟**。**技术映射**：**混合检索 = 最大风险面，须分域治理**。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | Web + SQL + 向量 | 仅向量 RAG | 仅联网 |
|------|------------------|------------|--------|
| 时效/事实 | 强 | 弱 | 中 |
| 风险 | 最大 | 中 | 高（假新闻） |
| 典型缺点 | 治理复杂 | 无实时 | 无结构化 |

### 适用场景

- 研究助理、经营分析、**合规允许** 的实时行情。

### 不适用场景

- **禁止出网** 或 **禁止动态 SQL**——须换设计。

### 注意事项

- **网络结果缓存** 防重复计费；**SQL 与向量口径** 同一业务真相表。

### 常见踩坑经验（生产向根因）

1. **联网摘要** 当事实不核对。  
2. **动态拼 SQL**。  
3. **未熔断** 外部搜索拖垮核心路径。
