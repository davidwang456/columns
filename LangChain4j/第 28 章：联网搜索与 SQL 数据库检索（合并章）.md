# 第 28 章：联网搜索与 SQL 数据库检索（合并章）

## 1. 项目背景

向量库擅长 **非结构化相似度**；**联网搜索**解决「**最新公共信息**」；**SQL 数据库检索**把 **强结构化事实**（订单额、库存、合同状态）以可控方式带给模型。二者若与 RAG 混用，统称 **混合检索**。本教材将原大纲两章合并，对应示例：

- `_08_Advanced_RAG_Web_Search_Example.java` —— `WebSearchEngine`  
- `_10_Advanced_RAG_SQL_Database_Retreiver_Example.java` —— SQL 作为检索器

## 2. 项目设计：大师与小白的对话

**小白**：联网会不会把假新闻检进来？

**大师**：**会**。要 **域名白名单、发布时间过滤、摘要二次校验**，并在 UI **标明网络来源**。

**小白**：SQL 路径最怕什么？

**大师**：**注入**与 **越权**。必须 **参数化**、**只读账号**、**行级安全** 或 **预定义视图**。

**小白**：两路能并行吗？

**大师**：可以，但要定义 **仲裁**：若 **结构化**与 **网络**矛盾，以谁为准？通常 **以内部 DB 为准**。

**小白**：合规？

**大师**：**爬虫 ToS**、**数据跨境**、**个人隐私**——联网段要单独立项审计。

**小白**：运维差异？

**大师**：联网依赖 **外部 API 配额**；SQL 依赖 **连接池**与 **只读副本**延迟。

### 延伸案例（情景演练）：投研助手的「内数外网」

**场景**：证券分析团队的内网助理需要回答「**某公司昨天收盘市值与财报电话会要点是否一致**」。**结构化事实**来自 **只读数据仓库**（SQL 检索器）：取 **收盘价、总股本**；**非结构化要点**来自 **内部研报向量库**；**公开舆情**来自 **Web Search**（受限 **白名单财经媒体**）。**仲裁规则**写死：**行情与股本**以 **数仓时间戳最新** 为准；**网络摘要**只能作为 **参考**，须在 UI **标注「外部来源」**。

**事故预演**：若 **搜索 API** 返回 **过时转载**，模型可能 **错误归因**。运营策略：**(1)** 网络结果要求 **`publishedAfter=当日 00:00 UTC`**；**(2)** 对 **冲突**触发 **二次提示**：「内部数与公开摘要不一致，请人工复核」。**SQL 侧**强制 **参数绑定**，禁止拼接 **股票代码字符串**；账户仅有 **`SELECT` on 视图 `v_market_daily`**。把本案例画成 **泳道图**，可作为 **合规与风控** 联席评审附件。

## 4. 项目总结

### 优点

- **时效 + 结构化**能力显著扩展。**缺点**：**风险面积**最大。

### 适用场景

- 研究助理、经营分析、需 **实时行情** 的场景（在合规允许范围内）。

### 注意事项

- **网络结果缓存**策略防 **重复计费**。  
- **SQL** 与 **向量索引**的 **口径一致性**（同一张「业务真相表」）。

### 常见踩坑

1. 把 **联网摘要**当 **事实**不做核对。  
2. **动态拼 SQL 字符串**。  
3. **未设熔断**导致外部搜索拖垮核心路径。
