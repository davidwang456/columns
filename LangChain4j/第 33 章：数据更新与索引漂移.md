# 第 33 章：数据更新与索引漂移

## 1. 项目背景

### 业务场景（拟真）

公司的「退货政策」文档在 2 月 1 日更新了——退货期限从 30 天改成了 45 天。但向量库里的旧版本片段还是 30 天。用户问「退货期限是多久」，模型检索到了旧版本的片段，回答「30 天」——用户按 30 天的理解寄回了商品，实际政策是 45 天，双方都造成了损失。这就是 **索引漂移**——文档已更新，但向量索引没有同步。

### 痛点放大

漂移的治理需要：① 检测漂移——定时跑评测集看 hit@k 是否下降；② 增量更新——文档变了，只更新变化的那段；③ 全量重建的蓝绿切换——建好新索引再切别名，建一半出问题可以回滚。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：索引漂移是不是就像超市价签没更新——货架上摆的是新价格，价签还贴着昨天的？

**大师**：一针见血。更麻烦的是超市可能有三个价签位（DB、向量库、缓存），你只更新了 DB 的那个，向量库和缓存还是旧的——用户问「多少钱」模型去向量库搜到了旧价格。所以索引新鲜度的治理必须作为整体来看，不是一个组件的事。

**小白**：增量更新怎么定义「变了」？所有文档重新嵌入一遍太贵了。

**大师**：增量更新靠 **业务主键的 hash 变化** 来判断。每个 segment 有一个 `docId + chunkIndex` 的复合主键。当文档来源变化时（比如 PDF 重新上传），计算新文本的 hash，与已有的 hash 对比——变了才重新嵌入。这是一种低成本的变化检测机制。同时保留旧版本的别名，万一新索引有问题可以秒级切回。**技术映射**：**索引新鲜度 SLO 必须写入 SLA 文档——「文档发布后最多 X 分钟内更新到向量库」这条指标必须被监控，否则没人知道索引落后了多久；出了问题，就是价签没更新的连锁反应**。

**小白**：全量重建一次要多久？如果建到一半发现新索引有问题怎么办？

**大师**：全量重建的时间取决于数据量和嵌入模型的速度——先用 **生产数据量的 10% 做一次压测**，推算出全量窗口。例如 100 万条 segments 用本地 ONNX 模型做全量嵌入耗时 2 小时，那就把重建窗口安排在业务低峰期的 4 小时内。重建一半出问题的解决方案是 **索引别名机制**：新索引用一个新名字（`embeddings_v3`），建好并验证数据完整性后，把别名（`embeddings_prod`）从旧索引切到新索引。旧索引保留至少一个发布周期用于回滚。**技术映射**：**索引漂移的发现不能仅靠投诉——必须建立自动化检测机制；定时跑黄金评测集对比 hit@k，或者监控「文档发布时间戳 vs 索引更新时间戳」的差值，差值超过阈值就告警**。

### 分步实现

#### 步骤 1：增量更新——hash 对比

```java
import java.security.MessageDigest;

public class IncrementalIndexer {

    private final Map<String, String> hashIndex = new ConcurrentHashMap<>();

    // 计算文档 hash
    public String computeHash(String text) {
        return bytesToHex(MessageDigest.getInstance("SHA-256")
            .digest(text.getBytes(StandardCharsets.UTF_8)));
    }

    // 判断是否需要重新索引
    public boolean needsReindex(String docId, String newText) {
        String newHash = computeHash(newText);
        String oldHash = hashIndex.get(docId);
        
        if (newHash.equals(oldHash)) {
            return false;  // 内容没变，跳过
        }
        // 内容变了，更新 hash 索引
        hashIndex.put(docId, newHash);
        return true;
    }
}
```

## 4. 项目总结

### 优点与缺点

| 维度 | 版本化+增量+别名 | 仅全量重建 | 无评测 |
|------|----------------|----------|-------|
| 新鲜度 | 可控 | 差 | 未知 |
| 运维复杂度 | 重 | 中 | 轻 |

### 适用场景

- 政策、条款、价格等频繁变更的业务（如电商、金融）
- 多团队协作、文档版本管理需求强的组织
- 需要满足合规审计「答案可追溯至特定版本」的场景

### 不适用场景

- 静态文库极少更新（如图书馆档案）
- 文档和向量库之间允许长时间不一致

### 注意事项

- **CDN/客户端缓存**——更新了向量库但前端 CDN 还缓存着旧答案，需设置合理的缓存失效策略
- **多语言文档**——中英文版本更新不同步时，每种语言需独立管理索引版本
- **向量删除原子性**——删除源文件时同步删除对应的 embedding，避免孤立脏数据

### 常见踩坑

1. **只更关系库未更向量库**——业务数据库改了，但向量索引还是旧数据，用户问到的仍是过时信息
2. **删文档忘删向量**——源文件已物理删除，但向量库里还残留着对应片段，检索时仍会命中
3. **无监控评测掉线**——索引漂移发生后无人发现，直到用户投诉才被动响应

### 进阶思考题

1. Kafka 积压与索引新鲜度 SLO 的联合告警——当 ingest 队列积压超过阈值时是否触发告警？
2. 蓝绿 warm 完成再切别名的探针设计——全量重建完成后如何自动验证数据完整性？
