# 第 33 章：数据更新与索引漂移

## 1. 项目背景

文档会 **修改、下架、合并**；向量索引若不同步，用户会得到 **过期答案**——所谓 **索引漂移**。治理手段包括：**版本号**、**增量任务**、**黄金问答回归**、以及 **双写/蓝绿索引切换**。测试团队关心：**评测集 fixture** 与 **生产文档版本**对齐。

## 2. 项目设计：大师与小白的对话

**小白**：全量重建要多长时间？

**大师**：要用 **真实数据量压测**；大库往往 **夜间窗口** + **分批**。

**小白**：增量如何定义？

**大师**：以 **业务主键**（文档 URI + hash）判断是否 **重新嵌入**。

**小白**：如何发现漂移？

**大师**：**定时评测** hit@k；用户 **点踩**回流；**法务变更**主动触发。

**小白**：向量与 DB 事务？

**大师**：通常 **最终一致**；用 **outbox 模式** 把变更排队到索引管道。

**小白**：回滚？

**大师**：保留 **上一版本索引别名**；切换别名即回滚。

## 3. 项目实战：主代码片段

> **场景入戏**：索引漂移是 **超市价签滞后**：收银机扫出来 **还是上周特价**，顾客会 **拍照发微博**——你的 RAG 也一样：**CMS 已更新，向量还在昨晚**，客服机器人就成了 **虚假促销直播**。

写一份 **SOP 草稿**（无需代码）：

1. 文档源更新事件到达 → **队列** → **解析→切分→嵌入** → **写入新 segment** → **旧 segment 失效**。  
2. 失败重试 **DLQ**。  
3. 完成后 **触发评测任务**（**黄金集**回放，对比 **policyVersion**）。

与团队讨论：你们 **主数据系统** 是 Kafka / DB trigger / 定时拉取？

#### 深度一行

在 SOP 里**必须**写清：**「用户可见答案」如何展示 `docId + version`**，否则无法与延伸案例里的 **公关危机**对齐。

### 延伸案例（情景演练）：「双 11」价格条款乌龙

某零售企业在 **大促前 6 小时**更新了 **退换货政策 PDF**（版本 **v2026.11**），CMS 已发布，但 **向量索引作业**因 **Kafka 积压**延迟 **90 分钟**才完成。期间客服机器人仍引用 **旧片段**，多名用户在社交平台 **晒截图** 投诉。事故复盘四要点：**(1)** **索引新鲜度 SLO**（例如「文档生效后 ≤15 分钟可检索」）写入 **服务级协议**；**(2)** **用户可见回答**携带 **`policyVersion`**，与 CMS **比对**不一致时 **降级为人工**；**(3)** **黄金问答集**在每次大促前 **全量回放**；**(4)** **蓝绿索引**：新索引 **warm** 完成后再 **切别名**，失败 **一键回滚**。

第二个迷你案例：**多语言并行文档** 仅更新了英文，中文向量滞留旧版——评测虽未报警（英文集通过），但 **华语用户**体感极差。团队在 **metadata** 增加 **`locale`**，并在 **管线**里禁止 **跨语言复用 segment**。这两则故事适合写成 **一页「索引漂移」海报** 贴在看板旁。

## 4. 项目总结

### 优点

- **可控的**知识新鲜度。**缺点**：流程 **重**。

### 适用场景

- 条款/价格 **频繁变更**业务。

### 注意事项

- **缓存层**（CDN/客户端）与索引一致性。  
- **国际化**多语言文档版本。

### 常见踩坑

1. **只更新关系库**未更新向量。  
2. **删除文档**忘了删向量残留。  
3. **无监控**评测掉线。
