# 第 22 章：高级 RAG —— 查询路由（Query Routing）

## 1. 项目背景

### 业务场景（拟真）

企业内部存在 **多知识域**（产品、法务、人力、运维 Runbook），**一个全局向量库混存一切** 往往 **不是最佳解**：权限、更新节奏、嵌入模型可能不同。**查询路由** 在检索前决定：**本条问题应检索哪个/哪些子索引**，降低 **噪声命中** 与 **无关 token 注入**。

### 痛点放大

`QueryRouter` / `LanguageModelQueryRouter` 等描述 **决策节点**。若无路由：**安全**上可能 **跨域误召回**；**性能**上单次检索数据量过大；**治理**上无法按部门 **独立迭代索引**。与第 21 章 **压缩**、第 26 章 **跳过检索** 构成 **进入向量库前的阀门**：**变形 / 选域 / 是否检索**。

示例：[`_02_Advanced_RAG_with_Query_Routing_Example.java`](../../langchain4j-examples/rag-examples/src/main/java/_3_advanced/_02_Advanced_RAG_with_Query_Routing_Example.java)。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：路由像商场楼层导览？走错楼层买错东西？

**小白**：路由和 **第 27 章多路融合** 差别？**能用规则吗？** 何时上 **小模型分类**？

**大师**：路由常 **互斥或主次**——先决定 **往哪走**；融合是 **并行多源再合并**。第一版可用 **关键词/正则**；**误路由代价高** 且规则 **打地鼠** 时再上 **LLM/小模型路由**。**技术映射**：**路由 = 检索 scope 决策**。

**小胖**：LLM 路由会被用户忽悠吗？

**小白**：**多租户** 咋约束？**路由与权限不一致** 会怎样？

**大师**：**提示勿暴露内部分类名**；**输入清洗**（第 6、36 章）。**租户 id** 须 **硬约束**，不能仅靠软提示。**路由与权限不一致** = **严重安全事故**——与 **元数据过滤**（第 25 章）**双重校验**。**技术映射**：**routeId 可观测、可审计**。

**小胖**：路由错了咋办？

**大师**：**安全默认**：**缩小召回**或 **澄清**，勿 **扩大可见集**；记录 **routeId** 与 **点踩** 抽样。**技术映射**：**评测 = 路由准确率 + 端到端答案**。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | Query Routing | 单库全量 | 仅关键词过滤 |
|------|---------------|----------|----------------|
| 噪声 | 低 | 高 | 中 |
| 治理复杂度 | 高 | 低 | 中 |
| 误路由代价 | 可高 | 中 | 中 |
| 典型缺点 | 与业务变更耦合 | 难扩展 | 泛化差 |

### 适用场景

- 多部门共享同一助理；**强权限隔离** 企业搜索。

### 不适用场景

- **单一域、小文档**——路由层过重。

### 注意事项

- 与 **IAM/ABAC** 对齐；**审计**路由决策；**灰度**新策略。

### 常见踩坑经验（生产向根因）

1. **提示泄露**内部索引代号。  
2. **只测准确率**不测 **权限侧信道**。  
3. **无 fallback** → **冷启动用户** 无法回答。

### 进阶思考题

1. **先路由再压缩** 与 **先压缩再路由** 的 A/B 设计要点？
