# 第 22 章：高级 RAG —— 查询路由（Query Routing）

## 1. 项目背景

当组织内存在 **多个知识域**（产品说明、法务合规、人力资源、运维 Runbook）时，「**一个全局向量库混存一切**」往往 **不是最佳答案**：权限模型不同、更新节奏不同、嵌入模型可能也不同。**查询路由（Query Routing）** 在检索之前决定：**本条问题应该检索哪一个或哪几个子索引**。它降低 **噪声命中** 与 **无关节 token 注入**，也可能减少 **单次检索数据量**。

在 `langchain4j-core` 中，`QueryRouter`（以及 **`LanguageModelQueryRouter`** 等实现）描述这种「**决策节点**」。示例：

- `langchain4j-examples/rag-examples/src/main/java/_3_advanced/_02_Advanced_RAG_with_Query_Routing_Example.java`

本章与第 21 章 **压缩**、第 26 章 **跳过检索** 构成「**进入向量库之前的三类阀门**」：**变形 / 选域 / 是否检索**。

## 2. 项目设计：大师与小白的对话

**小白**：路由和「多路检索器融合」有什么差别？

**大师**：**路由**通常是 **互斥或主次选择**——先决定 **往哪儿走**；**融合**是 **并行检索多个来源再合并**（第 27 章）。实践中二者可组合：先 **路由到事业部**，再在部内 **多路融合**。

**小白**：路由能不能用规则？

**大师**：**能**，而且很多企业第一版就是 **关键词 + 正则 + 简单打分**；优点是 **可解释、成本低**；缺点是 **泛化差**。

**小白**：那何时引入小模型分类？

**大师**：当 **误路由代价** 高（例如把 **医疗**问题送进 **营销话术库**），而规则维护已进入 **打地鼠** 状态。

**小白**：路由错误如何观测？

**大师**：记录 **`routeId`**、**候选集摘要**与用户反馈 **点踩**；定期 **人工抽样**校准。

**小白**：LLM 路由会不会被用户话术攻击？

**大师**：会。**提示里**不应暴露 **内部分类名**或 **敏感索引标签**；需要 **输入清洗**（第 6、36 章）。

**小白**：多租户呢？

**大师**：**租户 id** 必须是 **路由约束的硬条件**，不能只做「软提示」；否则 **跨租户数据**理论上可能被检索（若索引层未隔离）。

**小白**：路由和权限不一致会怎样？

**大师**：**最严重的安全事故之一**：用户看到 **不该看的知识**。要与 **Embedding 元数据过滤**（第 25 章） **双重校验**。

**小白**：失败降级？

**大师**：**安全默认**：**缩小召回**或 **要求澄清**，而不是 **扩大可见集**。

**小白**：评测指标？

**大师**：**路由准确率**（分类任务） + **端到端答案准确率**；后者更贴近业务。

## 4. 项目总结

### 优点

- **降低噪声**、缩小 **单次检索范围**。  
- 使 **组织知识治理** 与 **技术架构**对齐。

### 缺点

- **治理复杂**：路由规则与 **业务变更** 强耦合。  
- **错误会级联** 到最终答案。

### 适用场景

- 多部门共享同一助理入口。  
- **强权限隔离** 的企业搜索。

### 注意事项

- **与 IAM/ABAC** 对齐；**审计**路由决策。  
- **灰度发布**新路由策略。

### 常见踩坑

1. **提示中泄露**内部索引代号。  
2. **只测准确率**不测 **权限侧信道**。  
3. **无 fallback** 导致 **冷启动用户** 无法回答。
