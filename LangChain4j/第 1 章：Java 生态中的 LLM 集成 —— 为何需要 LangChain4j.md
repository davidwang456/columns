# 第 1 章：Java 生态中的 LLM 集成 —— 为何需要 LangChain4j

## 1. 项目背景

### 业务场景（拟真）

某连锁零售企业的 **Java 订单中心**有 12 人团队，维护着订单查询、退款处理、物流跟踪三个微服务。2024 年初，产品总监在季度规划会上提出需求：**在订单详情页右上角加一个「问状态」的输入框**，用户可以直接输入「我的退货到哪了」「上周的订单发货没」，而不是依次点击「我的订单 → 查看详情 → 物流信息 → 滚动到最下面」。产品要求 **Java 团队独立交付、两周出可演示版本**——不等数据团队排期，也不引入新的编程语言。

团队负责人老张评估后发现：这不仅仅是调一个 API 的事。订单数据在 MySQL、物流状态在第三方接口、退款的审批流在 BPM 引擎——要把这些信息「理解」并组织成自然语言回答，需要一个 **编排层** 来统一调度。

### 痛点放大

最快的方式是接 OpenAI 的 Java SDK，直接在订单服务里写 HTTP 调用。但老张想到一个问题：**三个月后采购要求评估国内模型怎么办？** 如果今天接 OpenAI，明天接百度文心，后天还要试本地部署的 Ollama，每个服务里散落的 if-else 和 JSON 字段映射会呈现 **指数级别的膨胀**。没有统一抽象的时候：

- **性能**：每个服务的连接池、超时、重试策略各写各的，无法统一治理。线上出现超时，说不清是订单服务连 OpenAI 超了，还是退款服务连 Azure 超了。
- **一致性**：同一个「订单状态」的概念——订单服务传 `"status": "shipped"`，退款服务传 `"status": 3`（用数字枚举），后续做数据分析和报表时需要对映射。
- **可维护性**：团队里来了新人，打开订单服务的代码看到的是直连 OpenAI SDK，打开退款服务看到的是另一套 HTTP 客户端封装，而物流服务干脆用 ProcessBuilder 调 Python 脚本——「半套框架、半套直连 SDK」的现状让交接成本翻倍。

LangChain4j 要解决的核心问题就是：**把 Chat / Embedding / Vector Store 统一成一套接口**，让业务代码面向 `ChatModel`、`ContentRetriever`、`Tool` 编程，而不是面向每家厂商不同的 SDK。

```text
无统一抽象：服务A -> OpenAI SDK（JSON 字段名不同）
           服务B -> Azure SDK（认证方式不同）  
           服务C -> 自写 HTTP（超时策略不同）
有 LangChain4j：全部面向 ChatModel / Retriever / Tool 编程
           换模型 = 改一行配置 + 换一个依赖坐标
```

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：这不就跟食堂打饭一样吗？窗口（模型）能出菜就行了，我爱吃哪个窗口就去哪个窗口排队——为啥还要搞个「LangChain」这么长的名字来多一层？

**大师**：食堂的比喻很好——但你想过没有，如果今天食堂换了一批厨师（从 OpenAI 换成国内模型），原来每个窗口的取餐方式（SDK 调用、认证方式、字段名）全变了，你每个窗口都得重新学怎么点菜。LangChain4j 统一的不是菜本身，而是 **点菜的方式**——不管你换哪个供应商，你的代码里都是 `model.chat("Hello")` 这一句话。

**小白**：我理解统一的好处。但我们 API 网关已经封了一层 OpenAI——外面有密钥管理、有流控、有审计日志。**应用里再加一层库，边界在哪？** 网关做了的事为什么要在应用里再做一遍？

**大师**：这是个非常关键的问题。网关管的是 **南北向流量**——谁可以调 API、每秒多少次、有没有过线。而 LangChain4j 管的是 **应用内部的组合逻辑**——多轮对话的历史怎么截断、工具调用的结果怎么拼回提示、检索出来的片段怎么跟用户问题组装在一起。你可以把网关想象成 **小区门禁**，LangChain4j 是 **家里怎么收纳家具**。门禁管得了谁进小区，但管不了你家客厅里堆五套不同牌子的螺丝刀。**技术映射**：**网关 ≠ 应用内 LLM 编排层——网关管准入与限流，编排层管组合与状态，两者是互补关系不是替代关系**。

**小胖**：哦……所以网关是物业，LangChain 是装修公司。那问题来了：统一成一套 API，会不会变成「最低公分母」——各家模型的独门绝活都暴露不出来？比如 OpenAI 的 strict tools 别的模型没有，抽象层会不会为了兼容就把这功能砍了？

**小白**：对，我也有这个担心——**如果某个模型不支持工具调用**，抽象层会不会把错误悄悄吞掉，到运行时才炸？而且我们怎么知道哪个模型支持什么、不支持什么？

**大师**：这两个问题问到点子上了。好的抽象设计原则是：**给的是默认路径和可替换实现**，而不是最低公分母。高级特性——比如 strict tools、json mode——仍然可以通过 **厂商专属的 Builder 参数** 直接暴露。比如 `OpenAiChatModel.builder().strictTools(true)` 是这个模型的专属能力，你不会因为用了抽象层就丢掉它。至于能力不一致的问题：LangChain4j 维护了一份 **能力矩阵表**（Capability Matrix），明确标注每个 provider 支持什么、不支持什么。不支持的工具调用会在 **编译期或配置期** 就暴露出来，而不是运行时静默失败。**技术映射**：**ChatModel 等接口是契约，不是魔法——它约定了你「怎么调」，但「能调什么」由具体 provider 的能力矩阵决定，要显式体现在配置与分支策略里**。

**小白**：那跟 Haystack、LlamaIndex 这类 Python 生态的框架比呢？我们合规还卡了一条——数据不能出域，是不是就没法用 LangChain4j 了？

**大师**：Haystack 和 LlamaIndex 同属「管线思维」，在概念层是相通的。但 LangChain4j 站在 **JVM 生态**，意味着它跟你的 Maven/Gradle 构建、JUnit 测试、Testcontainers 集成测试、Micrometer 监控体系是 **原生一致** 的——不需要在 Java 和 Python 之间搭桥。数据不出域的问题，用 **本地部署的模型**（Ollama、llama.cpp）或 **国产专有云端点** 就能解决。接口层的好处是：换端点主要改 **配置文件里的 baseURL 和 pom.xml 里的模块坐标**，业务代码一行都不动。**技术映射**：**选型看运行时与团队技能栈，不是看 GitHub star 数——对 Java 团队来说，能跟 Maven/JUnit 原生集成的价值，往往大于某个 Python 独有功能**。

### 可能遇到的坑

| 坑 | 表现 | 解法 |
|----|------|------|
| README 营销数字当能力 | 以为「30+ 向量库支持」= 公司已采购 | 区分「框架能接什么」与「你司已买什么」 |
| 新旧接口名混淆 | `ChatLanguageModel` vs `ChatModel` 混用 | 以当前仓库 Javadoc 为准 |
| 密钥写死在代码里 | API Key 被提交到 Git | 永远用环境变量或密钥管理服务 |

### 延伸案例：从「胶水代码」到统一抽象

某保险团队用 200 行胶水代码接入云厂商 SDK，完成了理赔问答的 PoC。采购要求评估第二家国产模型时，团队发现消息结构、流式事件名、工具调用的 JSON schema 完全不同，估计需要 **3 人周** 才能再抽象一层。引入 LangChain4j 后，业务包在 `ChatModel` + `AiServices` 之外，仅在配置中切换 provider bean；评测用 JUnit 跑同一批用例。另一收益：Micrometer 钩子比散落各处的 SDK 回调更容易统一埋点。

**反面例子**：只做一次性营销文案生成、且永不换模型的团队，继续直连也可接受——选型应匹配变更频率。

## 4. 项目总结

### 优点与缺点

| 维度 | LangChain4j（Java） | 直连厂商 SDK | 自建 Python 中台 |
|------|---------------------|-------------|-----------------|
| 与现有 Java 工程集成 | 高 | 中 | 低 |
| 多模型切换成本 | 低 | 高 | 中 |
| 学习曲线 | 中 | 低 | 高 |
| 社区与版本节奏 | 活跃，需跟 BOM | 跟厂商 | 依赖自研 |
| 典型风险 | 抽象与模型能力不匹配 | 供应商锁定 | 延迟与数据过网 |

### 适用 / 不适用场景

**适用**：Java 后端需要聊天/RAG/Agent 能力、多模型 PoC 与对比评测、需纳入统一观测体系。

**不适用**：一次性永不换模型的单轮文案生成、强依赖 Python 生态独有库且无意维护 Java 侧。

### 常见踩坑

1. **复制网上旧代码**中的过期类名 → 编译通过、文档对不上
2. **把框架当魔法**：不上评测与回归 → 线上「感觉变差」无法量化
3. **忽视观测**：首周流量小，第三周账单暴涨却无指标定位到具体接口

### 进阶思考题

1. 若公司要求推理必须在境内专区，你会如何在不改业务接口的前提下拆分配置与依赖模块？（提示：BOM + ChatModel 构造）
2. 网关已做脱敏，应用内是否还需要对 prompt 与检索片段再做一层策略？（提示：Guardrails）
