# 第 1 章：Java 生态中的 LLM 集成 —— 为何需要 LangChain4j

## 1. 项目背景

2023 年以来，大语言模型从实验室走入业务系统：智能客服、知识助理、运维排障、代码评审辅助等场景在 Java 企业中最常见的落地形态，是「**已有订单/工单/微服务之上，多一个能理解自然语言的接口**」。与数据团队偏好的 Python 笔记本不同，Java 团队需要：**与 Spring/Jakarta CDI 共存**、**与统一可观测（Micrometer/OpenTelemetry）对齐**、**与公司 SSO、审计、内网出口策略一致**。

若仅用厂商 REST SDK，短期可行；当模型从单一供应商扩展到「国内外多模型、多向量库、本地 Ollama、离线与在线混合」时，**散落在各处的 if-else 与重复的 JSON 映射**会指数增长。LangChain4j 在愿景上与官方 README 一致：**统一 Chat/Embedding/Vector Store 等抽象**，并把 RAG、工具调用、记忆等沉淀为可复用模式。

## 2. 项目设计：大师与小白的对话

**小白**：我们已经有 API 网关把 OpenAI 封了一层，还需要 LangChain4j 吗？

**大师**：网关解决的是 **南北向流量与密钥**；LangChain4j 解决的是 **应用内部的组合逻辑**—— 记忆如何截断、工具如何执行、检索结果如何拼进提示。两者可并存：网关管外，库管内。

**小白**：统一抽象会不会「最低公分母」，用不出各家特色能力？

**大师**：抽象提供 **默认路径**；高级特性仍可通过 **厂商专属 Builder 参数**、`Listener`、或直接使用 provider 客户端兜底。关键是领域代码不要被 **每家 SDK 的 DTO 名** 绑死。

**小白**：团队只会写 CRUD，学大模型的成本会不会拖垮交付？

**大师**：课程按章节递进：**第 4 章先跑一次 Hello World**，建立信心；**RAG 先 Easy 再 Low-level**，避免一上来读论文。组织上可用「平台组维护底座 + 业务组写提示与工具」。

**小白**：和 Haystack/LlamaIndex 这类比呢？

**大师**：同属「管线思维」；LangChain4j 站在 **JVM 生态**，与 Gradle/Maven、JUnit、Testcontainers 习惯一致。概念可迁移，代码不复制。

**小白**：合规说数据不能出域，是不是就没法用？

**大师**：可用 **本地模型**、**专有云**、**国产兼容端点**，LangChain4j 的接口层让「换端点」主要改配置与模块坐标。

**小白**：我们为什么要信一个社区项目？

**大师**：用 **许可证、发布节奏、issue 响应、你司 fork 策略** 四维评估；生产应 **版本钉死 + BOM**，而不是「每次构建拉最新 SNAPSHOT」。

### 延伸案例（情景演练）：从「直连 SDK」到「想换模型」的真实一周

**背景**：某保险 Java 团队为 **理赔问答** 第一次接入了云厂商官方 Java SDK，两个开发两周内用 **200 行胶水**完成了 **单轮问答**。Demo 广受好评，直到 **采购要求同时评估第二家国产模型** 做比价——团队发现：消息结构、**流式事件名**、**工具调用的 JSON schema** 完全不同，估计 **三人周**才能 **抽象一层**。

**转折**：他们引入 LangChain4j，把业务包在 **`ChatModel` + `AiServices`** 之外，仅在 `application-prod.yml` 切换 **provider bean**；评测集使用 **JUnit** 跑同一批用例，对两家输出做 **BLEU/人工双轨**（内部标准）。**另一收益**：**Micrometer** 钩子比散落在各处的 SDK 回调更易 **统一埋点**。这个案例不是说「必须上 LangChain4j」，而是说明：**当「换供应商」写进路线图** 时，尽早引入 **统一抽象**通常比 **临期重构**便宜。

**反面短故事**：另一家只做 **单次活动营销文案** 的团队，从未打算换模型，他们 **继续用直连** 也能接受——选型应匹配 **变更频率** 与 **组织技能**。读者可将两则对比写入自己的 **立项一页纸**。

## 4. 项目总结

### 优点

- **Java 原生**：无缝接入现有工程、依赖治理与 CI。  
- **模块多**：可按需引入，减少 fat jar。  
- **示例仓库** `langchain4j-examples` 覆盖从控制台到 Spring、向量库对接。

### 缺点

- **主版本迭代快**：需 BOM 与回归策略。  
- **「LangChain」一名易误解**：与 Python 版不是代码级兼容。  
- **模型能力差异**：抽象无法抹平「某模型不支持工具」的事实。

### 适用场景

- 企业 Java 后端需要 **聊天/RAG/Agent** 能力。  
- 需要 **多模型/多云** 切换与 PoC 对比。  
- 想把 LLM 调用纳入 **统一日志、指标、集成测试**。

### 注意事项

- **法务与数据分级**先于技术选型。  
- **成本模型**：token、向量库主机、批处理算力分开评估。  
- **团队技能**：安排提示工程与评测集维护职责，而非仅开发。

### 常见踩坑

1. **复制网上旧代码**中的过期类名（如接口Rename）。  
2. **把 LangChain4j 当魔法**：不设计评测与回归，上线后「感觉变差」无法量化。  
3. **忽视观测**：首周流量小，第三周账单暴涨却无指标定位。
