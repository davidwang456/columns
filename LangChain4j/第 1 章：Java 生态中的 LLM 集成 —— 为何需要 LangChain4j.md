# 第 1 章：Java 生态中的 LLM 集成 —— 为何需要 LangChain4j

## 1. 项目背景

### 业务场景（拟真）

2023 年以来，大语言模型从实验室进入业务系统。某连锁零售企业的 **Java 订单中心**接到需求：在 **订单详情页**增加「自然语言问状态」入口——用户可问「我的退货到哪了」而不是点五个菜单。产品希望 **首期由 Java 团队独立交付**，暂不依赖数据团队维护的 Python 中台，以便两周内出可演示版本。

这类诉求在 Java 企业里极为典型：**在已有订单、工单、微服务之上，多一个能理解自然语言的接口**。与数据团队偏好的 Python 笔记本不同，Java 团队需要：**与 Spring / Jakarta CDI 共存**、**与统一可观测（Micrometer / OpenTelemetry）对齐**、**与公司 SSO、审计、内网出口策略一致**。

### 痛点放大

若 **仅接某一家厂商的 REST SDK**，短期可行；当模型从单一供应商扩展到「国内外多模型、多向量库、本地 Ollama、离线与在线混合」时，**散落在各处的 if-else 与重复的 JSON 映射**会指数增长：记忆如何截断、工具如何执行、检索结果如何拼进提示，都变成 **复制粘贴的胶水**。结果是：**性能**上难以统一连接池与超时；**一致性**上同一业务概念在不同服务里被映射成不同 DTO；**可维护性**上新人面对「半套 LangChain4j、半套直连 SDK」无从下手。

LangChain4j 在愿景上与官方 README 一致：**统一 Chat / Embedding / Vector Store 等抽象**，并把 RAG、工具调用、记忆等沉淀为可复用模式——本章回答的是：**在 Java 生态里，为何值得为此多引一层库**，而不是继续手写胶水。

```text
[用户/工单系统] ──► [网关/SSO] ──► [Java 服务]
                                      │
                    无统一抽象时：各服务直连不同 SDK、各自拼 Prompt
                    有 LangChain4j：领域代码面向 ChatModel / Retriever / Tool
```

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：这不就跟食堂打饭一样吗？窗口（模型）能出饭不就行了，为啥还要「LangChain」这么长的名字？

**小白**：而且我们 API 网关已经封了一层 OpenAI，外面还有密钥和限流——**应用里再加一层库**，边界在哪？

**大师**：网关管的是 **南北向流量与密钥**；LangChain4j 管的是 **应用内部的组合逻辑**——记忆截断、工具执行、检索片段怎么拼进提示。你可以把网关想成 **小区门禁**，LangChain4j 是 **家里怎么收纳家具**：门禁挡不住你屋里乱堆五套螺丝刀。**技术映射**：**网关 ≠ 应用内 LLM 编排层**。

**小胖**：哦……那统一成一套 API，会不会变成「最低公分母」，各家模型的绝活都用不上？

**小白**：对，我还担心 **边界条件**：如果某模型不支持工具调用，抽象层会不会把错误藏到运行时？

**大师**：抽象给的是 **默认路径**和 **可替换实现**；高级特性仍可通过 **厂商专属 Builder 参数**、`Listener`、或 **直接使用 provider 客户端**兜底。不支持工具是 **能力矩阵**问题，好的抽象会让你在 **编译期或配置期**就发现，而不是散落在字符串里。**技术映射**：**ChatModel 等接口是契约，不是魔法**；能力差异要显式体现在配置与分支策略里。

**小胖**：我们组只会写 CRUD，学大模型会不会拖垮迭代？

**大师**：课程按章节递进：**第 4 章先跑通 Hello World**，**RAG 先 Easy 再 Low-level**。组织上可以 **平台组维护底座 + 业务组写提示与工具**，别把「会调 API」和「会设计评测集」绑在同一个人身上。**技术映射**：**降低学习曲线靠课程顺序与职责拆分**，不靠少写代码。

**小白**：和 Haystack、LlamaIndex 这类比呢？合规还说数据不能出域，是不是就没法用？

**大师**：同属「管线思维」；LangChain4j 站在 **JVM 生态**，与 Gradle / Maven、JUnit、Testcontainers 习惯一致。出域问题用 **本地模型、专有云、国产兼容端点**——接口层让「换端点」主要改 **配置与模块坐标**。**技术映射**：**选型看运行时与团队栈**，不是看 GitHub star。

**小白**：我们为什么要信一个社区项目？

**大师**：用 **许可证、发布节奏、issue 响应、你司 fork 策略**四维评估；生产应 **版本钉死 + BOM**，而不是每次构建拉最新 SNAPSHOT。**技术映射**：**治理与供应链**和选 Spring 一样严肃。

## 3. 项目实战

> **本章无编译型主代码**，侧重「**入职第一天词汇表**」与路径熟悉；完成下列步骤后，你在评审会上能 **少问十个基础问题**。

### 环境准备

- **仓库**：本地已克隆 [`langchain4j`](../../langchain4j) 与 [`langchain4j-examples`](../../langchain4j-examples)（与后续章节一致）。
- **工具**：JDK 17+、Maven 或 IDE、可访问文档的网络（内网则需代理说明）。

### 分步任务

| 步骤 | 目标（一句话） | 操作与产出 |
|------|----------------|------------|
| 1 | 用一句话向非技术同事解释「统一 API」 | 打开 [`langchain4j/README.md`](../../langchain4j/README.md)，用 **≤15 字**中文解释 **Unified APIs**，再用 **≤15 字**解释 **Comprehensive Toolbox** |
| 2 | 在磁盘上「定位」Hello World | 在 `langchain4j-examples/tutorials` 中 **物理找到** `_00_HelloWorld.java`，确认路径与包结构 |
| 3 | 建立安全与场景意识 | 手写表格：某场景 **要不要 LLM**、**数据敏感度**、**机密时找谁签字** |

**运行结果（文字描述）**：步骤 1 应能产出两句短定义；步骤 2 应能不看文档说出「从仓库根到该文件的相对路径」；步骤 3 应能区分「公开 FAQ」与「含订单号的对话」。

**可能遇到的坑**：把 README 里的营销数字当成 **自家已具备的能力**；**解法**：区分「框架能接什么」与「你司已买什么模型、批不批得出网」。

**深度彩蛋**：用荧光笔圈出 README 里 **「20+ LLM / 30+ embedding stores」**——含义是 **选型会议**要带 **竞品矩阵表**，而不是「听说 OpenAI 不错」。

### 延伸案例（情景演练）

**背景**：某保险 Java 团队为 **理赔问答**第一次接入云厂商官方 Java SDK，两周内用 **200 行胶水**完成 **单轮问答**。Demo 受好评，直到 **采购要求并行评估第二家国产模型**——团队发现：消息结构、**流式事件名**、**工具调用的 JSON schema** 各不相同，估计 **三人周**才能 **抽象一层**。

**转折**：引入 LangChain4j 后，业务包在 **`ChatModel` + `AiServices`** 之外，仅在配置中切换 **provider bean**；评测用 **JUnit** 跑同一批用例。**另一收益**：**Micrometer** 钩子比散落各处的 SDK 回调更易 **统一埋点**。

**反面短故事**：只做 **单次活动营销文案**、且 **从不打算换模型** 的团队，**继续直连**也可接受——**选型应匹配变更频率与组织技能**。

## 4. 项目总结

### 优点与缺点（与同类路径对比）

| 维度 | LangChain4j（Java） | 直连厂商 Java SDK | 自建 Python 中台 + HTTP |
|------|---------------------|---------------------|---------------------------|
| 与现有 Java 工程集成 | 高（BOM、JUnit、同仓库 CI） | 中（仍要自拼记忆/RAG） | 低（跨语言运维与联调） |
| 多模型 / 多向量库切换成本 | 相对较低（统一抽象 + 配置） | 高（每换一家一套胶水） | 中（取决于中台抽象质量） |
| 学习曲线 | 中（需理解 ChatModel / RAG 管线） | 低（仅调用） | 高（双栈团队） |
| 社区与主版本节奏 | 活跃，需跟 BOM | 跟厂商 | 依赖自研 |
| 典型风险 | 抽象与某模型能力不匹配 | 供应商锁定 + 胶水膨胀 | 延迟、SLA 边界、数据过网 |

### 适用场景

- 企业 Java 后端需要 **聊天 / RAG / Agent** 能力且希望 **与现有工程同构**。
- 需要 **多模型 / 多云** PoC 与对比评测。
- 想把 LLM 调用纳入 **统一日志、指标、集成测试**。

### 不适用场景

- **只做一次性的、永不换模型的单轮文案生成**，且团队 **零 JVM 意愿**——直连或外包脚本可能更便宜。
- **强依赖 Python 生态独有库**且 **无意维护 Java 侧**——不必硬上 JVM 抽象。

### 注意事项

- **法务与数据分级**先于技术选型。
- **成本模型**：token、向量库主机、批处理算力分开评估。
- **团队技能**：提示工程与 **评测集维护** 要有责任人，而非仅「会调接口的开发」。

### 常见踩坑经验（生产向根因）

1. **复制网上旧代码**中的过期类名（接口更名）导致 **编译通过、文档对不上**。
2. **把 LangChain4j 当魔法**：不设计评测与回归，上线后「感觉变差」**无法量化**。
3. **忽视观测**：首周流量小，第三周账单暴涨却 **无指标定位**到具体接口与模型。

### 进阶思考题

1. **若公司规定「推理必须在境内专区」**，你会如何在 **不改业务接口** 的前提下，拆分 **配置** 与 **依赖模块**？（提示：第 2 章 BOM、第 4 章 `ChatModel` 构造。答案见后续章节与附录。）
2. **网关已做脱敏**，应用内是否还需要对 **Prompt 与检索片段** 再做一层策略？由谁负责审计？（提示：第 36 章 Guardrails。）
