# 第 21 章：高级 RAG —— 查询压缩（Query Compression）

## 1. 项目背景

### 业务场景（拟真）

用户粘贴 **整封邮件**、长背景、情绪语句——直接 embed 会让检索向量被 **噪声支配**，召回 **不相关片段**；长查询还放大 **延迟与费用**。

### 痛点放大

**查询压缩** 在 **保持意图** 前提下，把输入变成 **更短、更像检索的查询**，再进入嵌入与召回。`QueryTransformer` 族（如 **`CompressingQueryTransformer`**，以版本为准）把逻辑插入 **Advanced RAG 管道**。风险：**压掉否定词**、**额外 LLM 调用**、**压缩胡说**——需 **回退与评测**。

示例：[`_01_Advanced_RAG_with_Query_Compression_Example.java`](../../langchain4j-examples/rag-examples/src/main/java/_3_advanced/_01_Advanced_RAG_with_Query_Compression_Example.java)。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：压缩是不是把用户废话删掉？删错了咋办？

**小白**：压缩和 **Query Rewriting** 一样吗？**会压掉「不要退款而要换货」吗？** **一定要多调一次 LLM？**

**大师**：重写可能 **同义扩展**，压缩偏 **去噪缩句**；Transformer = **检索前变形**。风险真实：**保留极性、实体、时间**；可 **规则校验** 丢关键 token 则 **回退原文**。常见用 **LLM 压缩**；启发式便宜但 **泛化差**。**技术映射**：**压缩 = 额外推理换检索质量**。

**小胖**：成本咋算账？

**小白**：压缩提示会 **泄露内部结构** 吗？和 **路由** 谁先谁后？**失败降级？**

**大师**：对比表：**无压缩 vs 有压缩** 的 Recall@k、延迟、额外 token；收益盖不过 **二次调用** 则不值。提示 **勿含内部路径**；日志脱敏。与路由 **无唯一顺序**——常 **先粗路由再域内压缩** 或 **先压缩再轻量路由**，需 **A/B**。失败：**超时用原文**；**输出空** 让用户重述。**技术映射**：**回退策略 = 可用性必备**。

## 4. 项目总结

### 优点与缺点（与同类做法对比）

| 维度 | Query Compression | 仅重写扩展 | 不处理长查询 |
|------|---------------------|------------|--------------|
| 降噪 | 强 | 视策略 | 无 |
| 成本 | +LLM 调用 | 可变 | 低 |
| 风险 | 压错、胡说 | 扩大召回噪声 | 检索差 |
| 典型缺点 | 延迟升 | 难控 | 不适用长邮件 |

### 适用场景

- 邮件/工单/IM **长文本** 内部助理；用户爱 **叙述背景** 时。

### 不适用场景

- **短关键词检索**已足够——不必加延迟。  
- **无预算做二次推理**——用启发式或规则。

### 注意事项

- 与 **权限** 正交；日志标 **`compressorModelId`**。

### 常见踩坑经验（生产向根因）

1. **过度压缩** 丢否定与条件。  
2. **无回退** 压缩失败全路瘫痪。  
3. **压缩提示** 无人评审。

### 进阶思考题

1. **多语言长邮件**：检测语言 → 压缩模型 **同语言** 的必要性？  
2. 如何将 **压缩步骤耗时占比** 纳入 **SLO**？
