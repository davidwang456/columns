# 第 21 章：高级 RAG —— 查询压缩（Query Compression）

## 1. 项目背景

在真实对话里，用户往往不会把问题写成「适合向量检索」的短句：他们可能粘贴 **整封邮件**、附上 **背景故事**、夹杂 **与主题无关的情绪性语句**。如果你直接把这一整段拿去 embed，**检索向量**可能被 **次要信息支配**，召回 **不相关的长文档片段**；同时 **更长的查询编码**也可能放大 **延迟与费用**。

**查询压缩（Query Compression）** 的目标是：在 **不改变用户真实意图** 的前提下，把输入压缩成 **更短、更像关键词组合的查询** 或 **显式问句**，再进入嵌入与召回链路。LangChain4j 在 `langchain4j-core` 中提供 **`QueryTransformer`** 抽象族（如 **`CompressingQueryTransformer`** 等具体实现，名称以当前版本为准），让你把这一逻辑 **插入 Advanced RAG 管道**。

示例代码：

- `langchain4j-examples/rag-examples/src/main/java/_3_advanced/_01_Advanced_RAG_with_Query_Compression_Example.java`

本章在架构上位于 **Naive RAG** 之后：你已经能「**把文档塞进索引**」，现在要优化「**问对问题**」。

## 2. 项目设计：大师与小白的对话

**小白**：压缩和「Query Rewriting」是同一个东西吗？

**大师**：范畴接近，但团队命名会不同。**重写**可能包括 **同义扩展**（扩大召回），**压缩**偏 **去噪与缩句**。LangChain4j 用 **Transformer** 这个词强调「**在进入检索前变形**」。

**小白**：会不会压掉否定词，比如「不要退款而要换货」？

**大师**：这是**真实风险**。实践中要 **保留极性词**、**实体名** 与 **时间约束**；或在压缩后用 **规则校验** 若输出丢失关键 token 则 **回退原始 query**。

**小白**：压缩一定要再调一次 LLM 吗？

**大师**：**常见是**，因为「理解长文本并缩写」是生成任务；也可用 **启发式**（抽取主题句）降低成本，但 **泛化差**。

**小白**：那成本怎么和收益算账？

**大师**：建立 **对比表**：同一评测集上，**无压缩 vs 有压缩** 的 **Recall@k**、**平均延迟**、**平均额外 token**；若召回提升不足以覆盖 **二次调用**，就不值得。

**小白**：压缩提示会不会泄露内部系统结构？

**大师**：会。提示里 **不要出现** 内部路径、员工姓名、未公开产品代号；**输出日志**要脱敏。

**小白**：和「路由」谁先谁后？

**大师**：无唯一答案。常见 **先粗路由**（选大致域）再 **域内压缩**；或 **先压缩** 再做 **轻量路由**。需要 **A/B** 与数据支撑。

**小白**：多语言长邮件怎么处理？

**大师**：**先检测语言**；压缩模型 **与该语言一致**；避免 **中英混杂提示** 压坏语义。

**小白**：失败时要降级吗？

**大师**：**必须**。典型：**压缩超时 → 用原文检索**；**压缩输出空 → 抛业务异常让用户重述**。

**小白**：测试怎么写才稳定？

**大师**：对 **压缩函数**输出做 **结构断言**（长度上限、必含关键词），不要对 **自然语言细节**逐字断言。

## 4. 项目总结

### 优点

- **降噪**，提升向量召回的相关性。  
- 可能 **缩短**嵌入输入，**略微**降低成本。

### 缺点

- **额外一次（或多次）推理调用**，整体延迟上升。  
- 引入 **新失败面**（压缩胡说）。

### 适用场景

- **邮件/工单/IM 长文本**驱动的内部助理。  
- 用户习惯 **叙述背景** 而非关键词检索。

### 注意事项

- **与权限模型**正交：压缩 **不改变** 谁可以看什么数据（第 25 章）。  
- **留存审计**：若压缩模型与主模型不同，**日志**要标 `compressorModelId`。

### 常见踩坑

1. **过度压缩** 丢失 **否定与条件**。  
2. **未设回退** 导致压缩失败 **整路瘫痪**。  
3. **压缩提示**写在代码深处 **无人评审**。
