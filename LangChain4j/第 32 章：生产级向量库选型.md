# 第 32 章：生产级向量库选型

## 1. 项目背景

### 业务场景（拟真）

平台组需要在 pgvector、Milvus 和 Elasticsearch 之间选择一个作为公司的标准向量库。这不是一个纯技术问题——它涉及团队现有的运维能力（DBA 熟悉 pg，但没运维过 Milvus）、数据量级（当前 500 万条 segments，预计一年后 2000 万）、延迟要求（P99 < 100ms）。

### 痛点放大

选型常见错误有三：**没有做压测就凭直觉选**（「听说 Milvus 快」）、**只看 feature list 不看运维负担**（Milvus 需要独立集群、pgvector 复用已有 PG）、**用不同评测集比较**（A 用自己的 100 条 query 测召回、B 用自己的 200 条测——结果不可比）。

## 2. 项目设计：小胖、小白与大师的对话

**小胖**：选向量库是不是就像选交通工具——高铁（pgvector）、轿车（Milvus）、自行车（内存库）——得看你去哪、带多少行李、有多少人一起走？

**大师**：这个比喻非常实用。选型不应该在会议室里对着 PPT 比功能清单，而应该用同一套评测集在候选方案上跑一遍，比的是 **Recall@K、P99 延迟、故障切换时间、on-call 响应负担** 四个维度。如果你公司已经重度使用 PostgreSQL，且数据量在千万级以下——选 pgvector 是自然路径，运维成本最低。如果你预期十亿级向量、高吞吐——Milvus 更适合。如果你全站已经是 ES——用 ES 的向量功能作为一路召回，不需要另起炉灶。**技术映射**：**选型的核心不是比 feature list，而是比「同一套评测集在不同后端的 Recall@K 和 P99」——用数据说话，不靠 PPT 选型**。

**小白**：那 TCO（总拥有成本）应该怎么算？云托管的向量库和自己搭的怎么比？

**大师**：TCO 要算三部分：① **基础设施**——pgvector 复用的是你已经有的 PG 集群，边际成本最低；Milvus 需要独立的 K8s 集群（至少 3 节点），资源开销大；② **运维人力**——DBA 已经会管 PG 了，但要学 Milvus 需要额外培训；③ **备份与容灾**——PG 有成熟的流复制和 WAL 归档，Milvus 的备份工具还在快速演进中。云托管省运维，但数据出境和长期成本需要法务和财务联合评估。**技术映射**：**选型不仅是技术决策，更是组织决策——团队熟悉的技术的运维负担往往比 feature list 上的差异更具决定性，人的因素不可忽视**。

### 分步实现

#### 步骤 1：同一评测集对比

```java
// 前提：EmbeddingModel 固定、查询黄金集固定
List<String> goldenQueries = loadGoldenQueries();  // 从文件加载

// 在 pgvector 上跑
```

| 维度 | pgvector | Milvus | Elasticsearch |

## 4. 项目总结

### 优点与缺点

| 维度 | LangChain4j 适配多后端 | 锁单一云向量 | 自研 |
|------|----------------------|-----------|------|
| 迁移成本 | 低（接口统一） | 高 | 最高 |
| 学习曲线 | 分散在各后端 | 低 | 高 |
| 典型缺点 | 特性矩阵 | 厂商锁定 | 维护 |

### 适用场景

- 企业 AI 中台基建决策，需要统一向量存储标准
- 数据量从百万到亿级、延迟要求从秒级到毫秒级的混合场景
- 需要对比多个向量库方案做技术选型评审

### 不适用场景

- 数据量极小（<10万条）——内存库已足够，不需要引入外部向量库
- 团队无任何数据库运维能力——应优先考虑全托管云服务

### 注意事项

- **换库 ≈ 全量重建**——切换向量库的时间窗口需要纳入排期
- **维度/距离度量必须一致**——HNSW vs IVF、cosine vs L2 的选择影响召回率
- **多活/灾备**——生产向量库必须做备份恢复演练

### 常见踩坑

1. **开发用内存库、上线直接换生产库**——未做压测，P95 从 5ms 飙到 500ms
2. **embedding 维度与向量库配置不一致**——启动时报错，临时重建索引影响上线时间
3. **监控缺失**——磁盘爆满或 compaction 堆积导致查询超时
