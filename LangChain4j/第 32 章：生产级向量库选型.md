# 第 32 章：生产级向量库选型

## 1. 项目背景

向量库选型是 **平台决策**：牵涉 **数据量、QPS、延迟、容灾、运维技能栈、许可证成本**。LangChain4j 通过 **`langchain4j-pgvector`、`langchain4j-milvus`、`langchain4j-elasticsearch`…** 提供适配。示例仓库中 **`pgvector-example`**（若存在）、**`milvus-example`** 等目录给出 **最小可运行**闭环，是 PoC 的起点。

## 2. 项目设计：大师与小白的对话

**小白**：PG+pgvector 什么时候选？

**大师**：团队 **已重度 PG**、数据量 **中等到大**、需要 **ACID+行级安全** 与 **SQL 报表**。

**小白**：Milvus 呢？

**大师**：**十亿级**、**高吞吐**、**分布式**诉求更强；运维门槛更高。

**小白**：Elasticsearch/OpenSearch？

**大师**：若全站已在 **搜索团队** 运维 ES，向量可作为 **另一路 recall**。

**小白**：Cloud 托管 vs 自建？

**大师**：看 **TCO + 合规**；云上 **运维省**、可能 **贵**。

**小白**：如何 PoC？

**大师**：**统一 RAG 管线**，只换 `EmbeddingStore` 实现，跑 **同一评测集**。

## 3. 项目实战：主代码片段

> **场景入戏**：选向量库像 **选交通工具**：**Milvus** 像 **高铁**——吞吐大、专网；**PG+pgvector** 像 **自家轿车**——能装 **业务数据同事务**，后备箱就是 **`metadata` SQL**。没有 **银弹**，只有 **约束条件下的最优**。

### 延伸案例（情景演练）：Milvus vs PG 的一场「联席 PoC」

某视频平台的内容安全团队计划 **1.2 亿条** 审核知识片段入库，峰值 **800 QPS** 语义检索。平台组倾向 **Milvus**（分布式、水平扩展），业务 DBA 倾向 **PostgreSQL + pgvector**（已有 **强一致备份** 与 **行级权限**）。联席 PoC 的 **统一评测脚本**使用 **同一 `EmbeddingModel`**、同一 **查询集（5k 条）**，比较：**(1) P99 查询延迟**；**(2) 故障注入下 **副本切换时间**；**(3) 运维团队 **on-call 负担**（以「一次凌晨演练人均小时数」粗估）。

结果并不非黑即白：Milvus 在 **吞吐**上胜出，但 **权限与 SQL 报表**要额外同步到 **OLTP**；PG 在 **合规叙事**上顺畅，但需要 **分库分表 + 向量分区** 规划。最终折衷：**冷数据 + 超高并发检索** 走 Milvus；**强关联订单/账号 metadata** 仍落 PG，应用层 **双写索引摘要** 或通过 **联邦查询**聚合。LangChain4j 的价值在于 **接口不变**，团队能以 **feature flag** 在 **EmbeddingStore** 实现间切换做灰度。你可把本案例改成表格里的 **权重打分** 练习。

## 4. 项目总结

### 优点

- **选择自由**，LangChain4j 保持接口统一。**缺点**：**学习曲线分散**在各后端。

### 适用场景

- 企业 **AI 中台** 基建决策材料。

### 注意事项

- **迁移成本**：换库 ≈ **全量重建索引**。  
- **多活/灾备**演练。

### 常见踩坑

1. **开发用内存**无压测就上线。  
2. **维度/距离度量**与训练不一致。  
3. **无监控**磁盘与 Compaction  backlog。
