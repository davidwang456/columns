# 滚动评审与跨部门校准

本文档用于落实计划中的 **rollout-review**：由**开发、测试、运维**各派代表，在课程推广前完成至少 **6 章（每角色 2 章）** 的交叉评审，并回填改进项。不要编辑计划文件本身；在本文件记录结论即可。

## 评审目标

1. **技术准确性**：示例路径、类名、模块名与当前仓库版本一致。  
2. **可执行性**：「测试/运维检查清单」能直接变成工单或 Runbook 条目。  
3. **风险措辞**：对实验模块、合规与数据出境描述无过度承诺。  
4. **篇幅**：样章（第 4、18、34 章）保持 **3000～5000 字**；其余章节若不足，按评审意见在后续迭代扩写。

## 推荐评审组合（示例）

| 角色 | 建议评审章节 | 关注面 |
|------|----------------|--------|
| 开发 | 第 4、12、20、34、38 章 | API 演进、SPI、扩展点 |
| 测试 | 第 14、17、24、28、35 章 | 工具调用、结构化输出、溯源、SQL 安全、端到端 |
| 运维 | 第 2、9、32、36、37 章 | BOM/密钥、HTTP 与超时、向量库运维、观测与护栏、MCP 安全 |

## 检查清单校准原则

- **测试**：每条清单应能映射到「自动化测试类型」（单元 / 契约 / e2e / 混沌 / 安全扫描）。  
- **运维**：每条清单应能映射到「监控或 Runbook 条目」（指标名、阈值、回滚触发条件之一）。  
- **开发**：每条清单应能映射到「设计决策或 ADR 要点」。

## 与 TODO 的对应

完成首轮评审并更新至少 **6 章**后，可将 `rollout-review` 待办标记为完成；后续季度复审仍使用本文件追加记录。
