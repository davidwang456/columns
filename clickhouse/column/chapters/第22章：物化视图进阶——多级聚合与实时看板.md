# 第22章：物化视图进阶——多级聚合与实时看板

## 1. 项目背景

某出行平台的数据团队遇到了一个典型的实时看板难题。公司业务需要三种不同粒度的实时数据：第一是城市维度的每秒订单数，运营人员要盯着大屏随时调度车辆；第二是司机维度的每分钟营收，用于实时绩效排名；第三是按小时汇总的账单数据，财务需要拿来做对账。

团队一开始用一张物化视图就上线了。`trips_minute_mv` 按分钟聚合，查询最近半小时的数据只需要几十毫秒，效果立竿见影。然而好景不长——当业务方开始对比本月和上月的分钟级数据时，查询耗时飙升到2秒以上。跨月查询甚至逼近5秒，大屏刷新频率根本扛不住。

问题的根源在于单张物化视图的粒度陷阱。分钟级MV每天产生1440行数据（每城市×每司机），一个月就是4万行，三个月累积12万行。虽然MergeTree的索引在分区裁剪后仍然有效，但聚合计算量随历史数据线性增长。更关键的是，业务并非真的需要以分钟粒度去扫描三个月的数据——看月趋势只需要日粒度就够了。

于是团队想到了"多级物化视图"的思路：源表入仓后，依次触发分钟级、小时级、日级三张MV，形成级联聚合链。但这带来了一连串新问题——级联MV会不会重复计算？为什么查询结果翻倍了？State函数和Merge函数是干什么的？Refreshable MV又是什么新东西？

本章将围绕"物化视图进阶"展开，从问题定位到方案设计，从状态函数到运维管理，带你掌握ClickHouse中多级聚合与实时看板的完整实践。

## 2. 项目设计：剧本式交锋对话

公司的小会议室里，白板上画着一张张歪歪扭扭的架构图。

**小胖**把马克笔往桌上一拍："这事儿简单得很！多建几张MV不就行了？一分钟建一张，一小时建一张，一天建一张，三层搞定。"

**小白**皱着眉头："等等，我有三个问题。第一个，假如源表插入一笔新订单记录，它会触发分钟级MV写入一行，然后这个INSERT又触发小时级MV写入一行……那小时级MV的数据来源是分钟级MV，它会不会把分钟级MV里已有的存量数据也重新扫一遍、导致重复计数？"

**小胖**被问住了，挠了挠头："应该……不会吧？不然这就不是级联，是套娃了。"

**大师**推门进来，正好听到这段对话，笑了笑说："小白的直觉很准，这是个好问题。答案是不会重复计算——原因在于物化视图的触发机制。当源表`trips_raw`插入一批数据块时，ClickHouse只会把**这个新数据块**传给分钟级MV做增量计算。分钟级MV写入后，它产生的新数据块再次作为'源'，**仅这个新增的数据块**被传给小时级MV。整个过程是一条增量管道，每一级只接收上一级的增量部分。存量数据不会被重复处理。"

小白追问："那假如我先建分钟级MV，过了两天才补建小时级MV，这两天里分钟级MV已有的数据怎么办？小时级MV不会自动吞掉历史数据吧？"

大师点头："对，这就是级联MV设计中最容易踩的坑之一。ClickHouse的物化视图不会回溯历史数据。你新建小时级MV的时候，它只会接收**之后**分钟级MV新产生的INSERT。已有的100万行分钟级数据，你得自己 `INSERT INTO trips_hour_mv SELECT ... FROM trips_minute_mv` 手动补一回。"

小白若有所思，继续抛出第二个问题："我刚才在看文档时还发现另一个概念——Refreshable MV。它跟传统MV有什么区别？咱们什么情况下该用哪个？"

大师在白板上画了两条线："传统MV是**增量触发**模式，每次INSERT到源表时自动执行查询，然后写入目标表。适合实时性要求高的场景，但前提是你的聚合逻辑必须是可累加的——比如SUM、COUNT这种，后来的数据直接跟现有的合并就行。而Refreshable MV是24.x版本引入的**定时全量刷新**模式，每到设定的时间间隔，比如每隔1小时，它会把QUERY定义的查询完整重跑一遍，然后通过`REPLACE`语义把目标表的对应分区替换掉。它适合不能用增量解决的问题，比如带TOP N的排名、需要JOIN多表的复杂报表、或者用到`UNIQ`精确去重的场景。"

**小胖**插嘴："但你刚才说传统MV也用到了countState/uniqState啊，那不是也能做精确去重？"

大师笑道："胖儿问到点子上了！这就是State/Merge函数组合的妙处。"

他在白板上写下：

```
写入时：countState() / sumState() / uniqState() → 存储中间状态
查询时：countMerge() / sumMerge() / uniqMerge() → 合并状态并求最终值
```

"传统MV + AggregatingMergeTree就是靠这套机制实现的。数据写入时不去算最终值，而是把'半成品状态'存起来；查询时再用Merge后缀函数把所有中间状态合并成最终结果。这相当于把计算延迟到了查询阶段，写路径极快，但读路径要吃一点状态合并的开销。"

"Refreshable MV则反过来——写路径重（定时全量计算），但读路径轻（查询普通表就行，不用Merge函数）。一重一轻，场景不同。"

小白又冒出了第三个问题："那State/Merge模式跟SummingMergeTree+简单聚合的写法相比，各有什么优劣？"

大师在白板上画了一张对比表：

| 维度 | SummingMergeTree + 简单聚合 | AggregatingMergeTree + State/Merge |
|------|---------------------------|-----------------------------------|
| 写路径 | 直接算SUM/COUNT，写即完成 | 存储中间状态对象，FIFO快速写入 |
| 读路径 | 需要用FINAL或GROUP BY再合并 | 必须用Merge函数解包 |
| 支持的函数 | SUM, COUNT, MIN, MAX, any | 几乎全量聚合函数都有State形式 |
| 精确去重 | 不支持（需COUNT DISTINCT） | 支持（uniqState/uniqMerge） |
| 中位数/分位 | 不支持 | quantileState/quantileMerge |
| 存储体积 | 较小 | 较大（状态对象比标量值大） |
| 查询复杂度 | 低（GROUP BY即可） | 较高（每个列都要配对Merge） |

"简单来说：如果你只需要SUM和COUNT，用SummingMergeTree就够了，省心省力。如果你需要精确去重、分位数等高级聚合，或者你的聚合逻辑涉及多表JOIN且必须延迟计算，那就得上State/Merge。"

小白合上笔记本：**"最后一个问题——MV上线后如果要改字段，必须删掉重建吗？那历史数据不就丢了？"**

大师摇头："24.x版本支持`ALTER MATERIALIZED VIEW ... MODIFY QUERY`，可以**在线修改MV的查询逻辑**。前提是目标表的ENGINE和ORDER BY不变，只是调整SELECT里的字段或表达式。这是24.x的一大改进，之前的版本确实只能DETACH → DROP → 重建。"

**小胖**站起来活动了一下筋骨："我彻底懂了——级联MV是管道，增量往下流；Refreshable MV是定时器，全量覆盖；State/Merge是半成品车间，先用后算。开工吧！"

### 分步实现

#### Step 1：创建源表和级联MV

首先建立原始事件表。这里使用MergeTree表引擎，按城市和时间排序，按月分区：

```sql
CREATE TABLE trips_raw (
    trip_id UInt64,
    driver_id UInt32,
    city LowCardinality(String),
    fare Decimal(10,2),
    distance Float32,
    start_time DateTime,
    status LowCardinality(String)
) ENGINE = MergeTree()
ORDER BY (city, start_time, trip_id)
PARTITION BY toYYYYMM(start_time);
```

源表建好后，开始构建三级级联MV。第一级按分钟聚合，粒度最细：

```sql
-- Level 1: 分钟级聚合，直接消费源表数据
CREATE MATERIALIZED VIEW trips_minute_mv
ENGINE = SummingMergeTree()
ORDER BY (minute, city, driver_id)
PARTITION BY toYYYYMM(minute)
AS SELECT
    toStartOfMinute(start_time) AS minute,
    city,
    driver_id,
    count() AS trip_count,
    sum(fare) AS total_fare,
    avg(distance) AS avg_distance,
    sumIf(fare, status = 'completed') AS completed_fare,
    countIf(status = 'cancelled') AS cancelled_count
FROM trips_raw
GROUP BY minute, city, driver_id;
```

注意这里使用`SummingMergeTree`而非`AggregatingMergeTree`——因为当前需求只需要SUM和COUNT类聚合，简单高效。

第二级按小时聚合，数据源不再是源表，而是分钟级MV：

```sql
-- Level 2: 小时级聚合，消费分钟级MV的输出
CREATE MATERIALIZED VIEW trips_hour_mv
ENGINE = SummingMergeTree()
ORDER BY (hour, city)
PARTITION BY toYYYYMM(hour)
AS SELECT
    toStartOfHour(minute) AS hour,
    city,
    sum(trip_count) AS trip_count,
    sum(total_fare) AS total_fare,
    sum(completed_fare) AS completed_fare,
    sum(cancelled_count) AS cancelled_count
FROM trips_minute_mv
GROUP BY hour, city;
```

第三级按天聚合，消费小时级MV的输出：

```sql
-- Level 3: 日级聚合，消费小时级MV的输出
CREATE MATERIALIZED VIEW trips_day_mv
ENGINE = SummingMergeTree()
ORDER BY (day, city)
PARTITION BY toYYYYMM(day)
AS SELECT
    toStartOfDay(hour) AS day,
    city,
    sum(trip_count) AS trip_count,
    sum(total_fare) AS total_fare
FROM trips_hour_mv
GROUP BY day, city;
```

这样一条**源表 → 分钟MV → 小时MV → 日MV**的级联链路就搭建完成了。

#### Step 2：插入数据验证级联

用存储过程生成10万条模拟数据，覆盖3天的时间跨度：

```sql
INSERT INTO trips_raw
SELECT
    number + 1000000 AS trip_id,
    rand(1) % 500 AS driver_id,
    ['北京','上海','广州','深圳','杭州'][rand(2) % 5 + 1] AS city,
    round(randUniform(3, 100), 2) AS fare,
    round(randUniform(0.5, 50.0), 1) AS distance,
    now() - INTERVAL (rand(4) % 259200) SECOND AS start_time,
    ['completed','cancelled','ongoing'][rand(5) % 3 + 1] AS status
FROM numbers(100000);
```

验证级联链路的数据一致性。理想情况下，三级MV的`trip_count`总和都应等于10万：

```sql
-- 分钟级验证
SELECT sum(trip_count) FROM trips_minute_mv;
-- 预期: 100000

-- 小时级验证
SELECT sum(trip_count) FROM trips_hour_mv;
-- 预期: 100000

-- 日级验证
SELECT sum(trip_count) FROM trips_day_mv;
-- 预期: 100000
```

如果小时级MV或日级MV的结果偏小，那很可能是建MV晚于源表写入导致的。此时需要手动回填：

```sql
INSERT INTO trips_hour_mv
SELECT
    toStartOfHour(minute) AS hour,
    city,
    sum(trip_count) AS trip_count,
    sum(total_fare) AS total_fare,
    sum(completed_fare) AS completed_fare,
    sum(cancelled_count) AS cancelled_count
FROM trips_minute_mv
GROUP BY hour, city;
```

> **注意：SummingMergeTree在同一分区内会自动合并重复主键的行**，但跨分区的合并是异步的，不做FINAL查询可能看到部分合并的中间结果。对于严格的一致性校验，查询时使用 `SELECT ... FROM table FINAL WHERE ...` 强制触发合并。

#### Step 3：State/Merge函数模式实战

当我们需要精确去重或分位数时，SimpleAggregateFunction系列就不够用了。这时候需要State/Merge模式。

建一张使用AggregatingMergeTree的MV，存储中间聚合状态：

```sql
CREATE MATERIALIZED VIEW trips_agg_state_mv
ENGINE = AggregatingMergeTree()
ORDER BY (minute, city, driver_id)
PARTITION BY toYYYYMM(minute)
AS SELECT
    toStartOfMinute(start_time) AS minute,
    city,
    driver_id,
    countState() AS trip_count_state,
    sumState(fare) AS fare_state,
    uniqState(trip_id) AS unique_trips_state,
    quantileState(0.95)(distance) AS p95_distance_state
FROM trips_raw
GROUP BY minute, city, driver_id;
```

写入时每一列存储的都是**中间状态对象**（二进制blob），不直接可读。查询时必须用对应的Merge函数解包：

```sql
SELECT
    minute,
    city,
    countMerge(trip_count_state) AS trips,
    sumMerge(fare_state) AS revenue,
    uniqMerge(unique_trips_state) AS unique_trips,
    quantileMerge(0.95)(p95_distance_state) AS p95_distance
FROM trips_agg_state_mv
WHERE minute >= today()
GROUP BY minute, city
ORDER BY minute DESC
LIMIT 20;
```

State/Merge的核心价值在于**聚合可叠加**。你可以把高峰期的数据先按分钟预聚合存State，低谷期再合并到小时级State，跨月查询时合并日级State——每次合并都能得到原始精度的结果。uniqState/uniqMerge支持HyperLogLog近似去重，quantileState/quantileMerge支持T-Digest分位数估计，都是可叠加的。

#### Step 4：Refreshable MV实践（24.x+）

传统MV解决不了的场景交给Refreshable MV。先开启功能开关：

```sql
SET allow_experimental_refreshable_materialized_view = 1;
```

创建一张每小时自动刷新的报表视图，只关注近30天数据：

```sql
CREATE MATERIALIZED VIEW trips_daily_report
REFRESH EVERY 1 HOUR
ENGINE = MergeTree()
ORDER BY (report_date, city)
AS SELECT
    toDate(start_time) AS report_date,
    city,
    count() AS trips,
    sum(fare) AS revenue,
    uniqExact(driver_id) AS active_drivers
FROM trips_raw
WHERE start_time >= today() - 30
GROUP BY report_date, city;
```

关键差异在这里：传统MV在GROUP BY中不能用`uniqExact`（它不属于聚合状态函数家族），而Refreshable MV没有这个限制——它就是一条普通的SELECT语句，每1小时全量重跑一次。

查看刷新状态：

```sql
SELECT
    database,
    view,
    status,
    last_refresh_time,
    last_exception,
    refresh_count
FROM system.view_refreshes
WHERE view = 'trips_daily_report'
FORMAT Vertical;
```

修改刷新频率（比如改为每30分钟）：

```sql
ALTER MATERIALIZED VIEW trips_daily_report REFRESH EVERY 30 MINUTE;
```

Refreshable MV的适用边界：适合以天/小时为粒度的报表、全量计算开销可控的场景。不适合秒级实时场景（延迟等于刷新周期），也不适合数据量超过内存承载范围的全量重算。

#### Step 5：MV运维管理

**停止与恢复MV**：当需要进行源表维护或MV重建时，暂停单张或多张MV的增量写入，防止产生错误数据：

```sql
-- 暂停分钟级MV
SYSTEM STOP VIEW trips_minute_mv;

-- 此时源表仍可正常INSERT，但分钟级MV不再写入新数据
INSERT INTO trips_raw VALUES (9999999, 100, '北京', 50.00, 10.0, now(), 'completed');
-- 这条数据不会出现在分钟级MV中！

-- 恢复
SYSTEM START VIEW trips_minute_mv;
-- 恢复后，之前积压的INSERT会立即触发增量计算
```

**在线修改MV查询**：不用删表重建，直接ALTER即可更新计算逻辑：

```sql
ALTER TABLE trips_hour_mv MODIFY QUERY
SELECT
    toStartOfHour(minute) AS hour,
    city,
    sum(trip_count) AS trip_count,
    sum(total_fare) AS total_fare,
    sum(completed_fare) AS completed_fare,
    sum(cancelled_count) AS cancelled_count,
    avg(avg_distance) AS avg_distance    -- 新增加的平均距离字段
FROM trips_minute_mv
GROUP BY hour, city;
```

修改完成后，后续新写入的小时级数据将包含`avg_distance`字段，存量数据需要手动回填。

**分离与挂载**：跨节点迁移或MV备份时使用：

```sql
-- 分离小时级MV（元数据保留，数据文件不移除）
DETACH TABLE trips_hour_mv;

-- 移动数据文件到目标节点...（略）

-- 在目标节点挂载
ATTACH MATERIALIZED VIEW trips_hour_mv
ENGINE = SummingMergeTree()
ORDER BY (hour, city)
PARTITION BY toYYYYMM(hour)
AS SELECT
    toStartOfHour(minute) AS hour,
    city,
    sum(trip_count) AS trip_count,
    sum(total_fare) AS total_fare,
    sum(completed_fare) AS completed_fare,
    sum(cancelled_count) AS cancelled_count
FROM trips_minute_mv
GROUP BY hour, city;
```

> DETACH不会删除分区文件夹下的数据，只是从元数据中移除表定义。这是安全操作。

### 测试验证

完成以上步骤后，跑一轮完整的功能验证：

```sql
-- 1. 源表插入1000条实时数据
INSERT INTO trips_raw
SELECT
    number + 2000000,
    rand(10) % 500,
    ['北京','上海','广州','深圳','杭州'][rand(11) % 5 + 1],
    round(randUniform(10, 200), 2),
    round(randUniform(1.0, 80.0), 1),
    now() - INTERVAL rand(12) % 600 SECOND,
    ['completed','cancelled'][rand(13) % 2 + 1]
FROM numbers(1000);

-- 2. 等几秒让级联链路触发完成

-- 3. 对比三级MV与源表的行数一致性
SELECT 'raw' AS level, count() FROM trips_raw WHERE start_time >= now() - INTERVAL 1 HOUR
UNION ALL
SELECT 'minute_mv', sum(trip_count) FROM trips_minute_mv WHERE minute >= toStartOfMinute(now() - INTERVAL 1 HOUR)
UNION ALL
SELECT 'hour_mv', sum(trip_count) FROM trips_hour_mv WHERE hour >= toStartOfHour(now() - INTERVAL 1 HOUR)
UNION ALL
SELECT 'day_mv', sum(trip_count) FROM trips_day_mv WHERE day >= today();

-- 4. 性能对比：State/Merge vs 直接聚合
-- State模式（预先存储中间状态）
SELECT minute, countMerge(trip_count_state) AS trips
FROM trips_agg_state_mv
WHERE minute >= now() - INTERVAL 3 HOUR
GROUP BY minute
ORDER BY minute DESC;

-- 直接聚合（扫源表）
SELECT toStartOfMinute(start_time) AS minute, count() AS trips
FROM trips_raw
WHERE start_time >= now() - INTERVAL 3 HOUR
GROUP BY minute
ORDER BY minute DESC;
```

预期结果：级联链路数据完整一致，State/Merge模式在数据量大时查询性能优于直接扫描源表。

## 4. 项目总结

### MV级联设计原则

经过本项目的完整实践，我们可以总结出多级物化视图的设计原则：

| Level | 粒度 | 保留周期 | 查询速度 | 数据膨胀 |
|-------|------|---------|---------|---------|
| Raw（源表） | 逐条事件 | 7天 | 慢（全表扫描） | 1x（基准） |
| Minute MV | 每分钟 | 30天 | 快（近期数据） | ~0.1x |
| Hour MV | 每小时 | 90天 | 快（历史数据） | ~0.02x |
| Day MV | 每天 | 2年 | 极快（趋势分析） | ~0.005x |

核心思想是**渐进式降采样**：随着时间推移，数据从高粒度流向低粒度，查询时根据时间跨度自动选择最近的足够粗的粒度。查最近1小时用分钟级MV，查最近1个月用小时级MV，查年度趋势用日级MV。

### State/Merge vs SummingMergeTree 选型指南

```
需要 SUM/COUNT/MIN/MAX → SummingMergeTree（简单高效）
需要 精确去重/分位数 → AggregatingMergeTree + State/Merge
需要 跨级联聚合链路 → AggregatingMergeTree（State可叠加）
需要 极简查询体验 → SummingMergeTree（不用写Merge函数）
```

### 适用场景

- **实时运维大屏**：秒级/分钟级MV，城市维度，数据保留一周以内，查询延迟低于100ms
- **业务监控报表**：分钟→小时级联，支持同比环比，数据保留3个月
- **离线对账系统**：小时→日级联，数据保留2年，支持按月汇总
- **多粒度时序分析**：存储传感器数据、日志指标，按不同粒度下钻分析

### 常见踩坑经验

1. **级联断裂**：级联MV中某一级因异常中断（如OOM、磁盘满），会导致下游全部阻塞。运维上需要对每张MV做心跳监控，使用`SYSTEM START/STOP VIEW`时更要按上下游顺序操作——先停下游再停上游，先启上游再启下游。

2. **SummingMergeTree忘记FINAL**：SummingMergeTree的合并是异步的，未合并的行在GROUP BY中会被重复加和，导致聚合值翻倍。查询时加上`FINAL`修饰符强制触发合并：`SELECT * FROM trips_minute_mv FINAL WHERE ...`。

3. **Refreshable MV全量刷新开销**：Refreshable MV在每次刷新时执行完整的SELECT，如果源表有几亿行，全量扫描可以轻易打满CPU和IO。合理做法是在WHERE子句中限制数据范围（如30天），并选择合适的刷新周期，避免与业务高峰期重叠。

4. **级联深度过大导致写入延迟**：每增加一级MV，源表的一次INSERT都要额外触发多级写入。三级以上建议评估写入吞吐量，必要时可以将非关键级别的MV改为Refreshable模式。

### 思考题

1. 级联MV中，如果第二级（小时）的聚合逻辑写错了（比如少加了一个field），现在发现分钟后和小时级的总和不一致。如何修复小时级的数据而不影响第三级（日级）？提示：考虑`SYSTEM STOP VIEW`的上下游顺序以及`INSERT INTO ... SELECT ...`的定向回填策略。

2. State/Merge模式和SummingMergeTree模式各有什么优劣？在以下场景中你分别会选择哪一种？
   - (a) 需要精确统计每天有多少独立用户（unique visitors）
   - (b) 需要统计每分钟的订单总金额和订单数
   - (c) 需要P50/P95/P99延迟分位数，同时支持从1分钟到1天不同粒度的下钻
