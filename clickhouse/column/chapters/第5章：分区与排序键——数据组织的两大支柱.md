# 第5章：分区与排序键——数据组织的两大支柱

> **版本**：ClickHouse 25.x LTS
> **定位**：基础篇核心章节。分区决定物理布局，排序键决定查询效率——理解这两者，才算真正入门 MergeTree。
> **前置阅读**：第4章（MergeTree 家族概览）
> **预计阅读**：35 分钟 | **实战耗时**：45 分钟

## 1. 项目背景

字节跳动旗下某日志分析平台，负责全公司所有业务线的 Nginx 访问日志采集与分析。日均入库量高达 **50 亿条**——每秒钟涌入近 6 万行。团队最初的设计非常简单粗暴：一张 MergeTree 表，直接按默认配置建表，连 `PARTITION BY` 都没指定。

```sql
-- 最初的建表语句（存在严重设计缺陷）
CREATE TABLE nginx_access_log (
    timestamp  DateTime,
    host       String,
    uri        String,
    status     UInt16,
    body_bytes UInt64,
    user_agent String,
    remote_ip  String
) ENGINE = MergeTree()
ORDER BY (host, timestamp);
```

三个月后噩梦降临。`system.parts` 显示该表的活跃 Part 数量突破了 **10 万个**，`SELECT * FROM system.parts WHERE table='nginx_access_log' AND active=1` 一跑就是十几秒。更致命的是——每次查询最近一小时的日志数据，ClickHouse 虽然只扫描少量 Granule，但仍需要遍历 10 万个 Part 的元数据，光是打开 `primary.idx` 和 `*.mrk` 文件就耗费上百毫秒。DBA 在告警群里哀嚎："查一条日志比拉面还慢，ZooKeeper 内存都快撑爆了。"

**核心痛点**：

- **无分区 = 全分区扫描**：ClickHouse 没有"全表"概念，默认将所有数据放进 `all` 分区。查询时必须遍历该分区下所有 Part 的元数据，Part 越多越慢。
- **排序键设计不当 = 索引形同虚设**：`ORDER BY` 就是稀疏索引。如果查询条件不匹配排序键的最左前缀，索引完全不起作用，退化为全表扫描，扫描的 Granule 数可能暴增 100 倍。
- **合并压力巨大**：10 万级 Part 意味着后台 Merge 任务堆积如山，CPU 和 IO 被持续消耗。

本项目的目标非常明确：**重新设计分区键和排序键**，将活跃 Part 数控制在可管理的范围内（建议 1000 以内），同时保证高频查询能够最大限度地利用索引裁剪，将 P99 查询延迟从秒级压到毫秒级。

## 2. 项目设计：剧本式交锋对话

会议室白板上画满了密密麻麻的表结构草图。小胖端着一杯奶茶推门进来，看到大师和小白正对着投影仪上的 `system.parts` 截图发愁。

**小胖**（吸了一口奶茶）：“分区不就是按日期切一下嘛，`PARTITION BY toYYYYMMDD(timestamp)` 一把梭，有什么好设计的？ORDER BY 不就等于 MySQL 的索引吗，把经常查询的列往里一扔就完事儿。”

**大师**（放下白板笔）：“小胖，如果你用 `toYYYYMMDD` 按天分区，这张表每天产生 50 亿数据、几百个 Part。运行一年就是 365 个分区——这还凑合。但你知道吗，上周有个实习生跑来问我，说为了做实时大屏，能不能按小时分区……”

**小胖**（抢答）：“按小时更精细啊，查得不是更快？”

**大师**：“按小时分区，一年就是 8760 个分区。如果每个分区再因为频繁写入积累几十个 Part，总 Part 数轻松破 50 万。到时候 `SELECT * FROM system.parts` 比你的业务查询还慢。记住：**ClickHouse 的 Part 元数据是加在内存里的，几十万个 Part → 内存炸 → ZooKeeper 炸 → 整个集群炸**。”

**技术映射 #1**：分区裁剪本质是"数据隔离"——查询时先按分区表达式确定需要扫描哪些分区，直接跳过不相关的数据目录。但分区不是越细越好，每个分区下有多个 Part，而 Part 的元数据需要常驻内存，过多分区会耗尽内存和 ZooKeeper 资源。

---

**小白**（一直在笔记本上记，抬起头）：“那大师，分区粒度到底怎么选？按天、按月、还是按业务 ID？我上家公司用按天分区，DBA 说每天凌晨的 Merge 任务会把 CPU 打满。”

**大师**：“好问题。来看三个维度的取舍：

**按天分区**（`toYYYYMMDD`）：最灵活，可以按天删除过期数据、按天备份。但分区数多，适合数据量极大且必须按天清理的场景——比如日志数据，GDPR 合规要求 90 天自动删除。缺点是多天查询需要跨分区读取，适合 `WHERE timestamp >= today() - 7` 这种短区间查询。

**按月分区**（`toYYYYMM`）：分区数适中，一年的数据只有 12 个分区。大多数分析查询按月粒度运行足够——比如"上月 PV 趋势图""本月环比增长"。缺点是删除只能按月，无法精细控制。适合数据量适中、不需要按天清理的场景。

**按业务 ID 分区**：比如按 `tenant_id % 10` 分区。查询单个租户的数据时，只命中一个分区。但注意——如果某些租户数据量远超其他，会造成严重的数据倾斜，大租户的查询拖慢整个集群。适合多租户 SaaS 平台，前提是租户数据量相对均匀。”

**小胖**（掰手指）：“那我们的日志平台，日增量 50 亿，还得按天清理……按天分区跑不掉啊。”

**大师**：“对。但额外需要控制写入频率——把 INSERT 批量合并，每个分区每天控制在 10-50 个 Part 以内，让后台 Merge 有足够时间消化。”

**技术映射 #2**：分区字段选择的关键词是"均匀"和"可控"。均匀指的是各分区数据量相当，避免数据倾斜；可控指的是分区总数不会爆炸，不会因 Merge 积压拖垮系统。

---

**小白**（翻到下一页笔记本）：“下一个问题。`ORDER BY (a, b, c)` 和 MySQL 的联合索引 `(a, b, c)` 一样遵循最左前缀原则吗？如果我的查询是 `WHERE b=2 AND c=3`，`ORDER BY (a, b, c)` 能命中索引吗？”

**大师**（在白板上画了两个对比图）：“你的直觉对了一半。相同的是——它们都遵循**最左前缀匹配规则**。不同的是——MySQL 的 B+Tree 索引是**稠密索引**（每行一个索引条目），不做覆盖索引的话还要回表；ClickHouse 的 ORDER BY 是**稀疏索引**——每 8192 行（一个 Granule）才记录一个索引条目。”

“所以在 ClickHouse 中：

- `WHERE a=1 AND b=2 AND c=3` → 最左前缀完全命中，索引精确缩小到少数几个 Granule。
- `WHERE a=1 AND c=3` → 用到了 `a`，跳过了 `b`，`c` 无法使用索引。
- `WHERE b=2 AND c=3` → 完全不命中索引，因为第一列 `a` 没有出现在 WHERE 中——**全表扫描**。”

**小胖**：“所以如果你经常按 `b` 查，就应该把 `b` 放第一列？”

**大师**：“正是。`ORDER BY (timestamp, user_id)` 和 `ORDER BY (user_id, timestamp)` 虽然只有顺序差别，查询场景完全不同：

- `ORDER BY (timestamp, user_id)`：适合 `WHERE timestamp >= '2025-04-01'`，按时间范围查所有用户的数据。
- `ORDER BY (user_id, timestamp)`：适合 `WHERE user_id = 10086`，查某个用户的历史行为数据。

排序键的顺序决定了你的查询能跑多快。”

**技术映射 #3**：ORDER BY 列的设计应当遵循——**过滤频率从高到低、基数从低到高排列**。把 WHERE 中几乎每次都会出现的列放最前面，基数低的（如 `event_type`）放前面，基数高的（如 `user_id`）放后面。列数不要超过 4-5 列，否则索引文件本身也变大。

---

**小白**：“还有一个问题。系统一直在后台做 Merge 把 Part 合并，那我怎么看到底有哪些 Part？有没有类似 'SQL Server 的 DMV' 这样的视图？”

**大师**：“`system.parts` 表就是你的眼睛。通过它你能看到每个表的每个分区下有多少 Part、每个 Part 有多少行、占用多少磁盘。配合 `WHERE active=1` 只看活跃 Part，配合 `WHERE table='your_table'` 看特定表的状态。”

“一个健康的 MergeTree 表，每个分区通常只有 1 个活跃 Part。如果某个分区超过 50 个 Part，说明 Merge 速度跟不上写入速度——要么提高 `max_bytes_per_sec` 让合并更快，要么批量插入减少 Part 产生。”

**小胖**（若有所思）：“所以分区决定数据放哪个文件夹，排序键决定查询时读多少数据。Part 是这两者的物理载体，被 Merge 不断地重组？我懂了——分区是'抽屉'，排序键是'抽屉里文件的排列顺序'，Part 就是那一沓一沓被整理的文件！”

**大师**（笑）：“……差不多是这个意思。”

## 3. 项目实战

### 环境准备

### 分步实现

#### Step 1：创建不同分区策略的表并对比

首先创建三张对比表——分别采用按天分区、按小时分区（反面教材）和无分区。

```sql
-- ===== 表1：按天分区（推荐方案） =====
CREATE TABLE logs_by_day (
    timestamp  DateTime,
    level      LowCardinality(String),
    message    String,
    user_id    UInt64
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDD(timestamp)
ORDER BY (level, timestamp);

-- ===== 表2：按小时分区（反面教材——太细了！） =====
CREATE TABLE logs_by_hour (
    timestamp  DateTime,
    level      LowCardinality(String),
    message    String,
    user_id    UInt64
) ENGINE = MergeTree()
PARTITION BY toYYYYMMDDhhmmss(timestamp)  -- 每秒一个分区！灾难！
ORDER BY (level, timestamp);

-- ===== 表3：无分区（原始错误设计） =====
CREATE TABLE logs_no_partition (
    timestamp  DateTime,
    level      LowCardinality(String),
    message    String,
    user_id    UInt64
) ENGINE = MergeTree()
ORDER BY (level, timestamp);

-- ===== 表4：按月分区（折中方案） =====
CREATE TABLE logs_by_month (
    timestamp  DateTime,
    level      LowCardinality(String),
    message    String,
    user_id    UInt64
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(timestamp)
ORDER BY (level, timestamp);
```

现在向四张表插入 30 天的模拟日志数据（每天 50 万行，共 1500 万行）。为了模拟生产中的小批量写入（这会导致大量小 Part），我们每次只插入约 1 万行：

```sql
-- 插入 30 天模拟数据，每天分 50 批写入（每批 1 万行，每张表共 1500 万行）
INSERT INTO logs_by_day
SELECT
    toDateTime('2025-04-01 00:00:00') + number * 60 AS timestamp,
    ['INFO','WARN','ERROR','DEBUG'][rand() % 4 + 1] AS level,
    concat('Log message #', toString(number % 1000), ': request from ip=',
           toString(rand() % 256), '.', toString(rand() % 256), '.',
           toString(rand() % 256), '.', toString(rand() % 256),
           ' processed in ', toString(randUniform(1, 5000)), 'ms') AS message,
    rand() % 1000000 AS user_id
FROM numbers(15000000);

INSERT INTO logs_by_hour   SELECT * FROM logs_by_day;
INSERT INTO logs_no_partition SELECT * FROM logs_by_day;
INSERT INTO logs_by_month  SELECT * FROM logs_by_day;
```

> **踩坑提醒**：如果数据量较大且内存有限，可以分批次 INSERT，每次 `FROM numbers(1000000)` 并更换时间范围。本文示例中的 1500 万行在 8GB 内存下可一次完成。

现在对比各表的 Part 数量——这就是"分区粒度"带来的直接后果：

```sql
SELECT
    table,
    count()     AS active_parts,
    sum(rows)   AS total_rows,
    uniq(partition) AS partition_count
FROM system.parts
WHERE database = currentDatabase()
  AND table IN ('logs_by_day', 'logs_by_hour', 'logs_no_partition', 'logs_by_month')
  AND active = 1
GROUP BY table
ORDER BY active_parts DESC;
```

运行结果大致如下（具体数值取决于写入并发度和 Merge 进度）：

```
┌─table──────────────┬─active_parts─┬─total_rows─┬─partition_count─┐
│ logs_by_hour       │         1273 │   15000000 │             538 │
│ logs_by_day        │          124 │   15000000 │              30 │
│ logs_by_month      │           53 │   15000000 │               1 │
│ logs_no_partition  │          107 │   15000000 │               1 │
└────────────────────┴──────────────┴────────────┴─────────────────┘
```

`logs_by_hour` 的 Part 数是 `logs_by_day` 的 **10 倍**，是 `logs_by_month` 的 **24 倍**。但这还不是最可怕的——这只是 30 天数据，生产环境运行一年后，`logs_by_hour` 的 Part 数将突破百万，ZooKeeper 内存和 ClickHouse 的内存会直接被 Part 元数据撑爆。

#### Step 2：分区裁剪效果对比

分区裁剪是分区键最重要的价值——查询只扫描相关分区，跳过无关数据。我们用 `EXPLAIN` 来验证：

```sql
-- 查看按天分区表的查询计划——应该只扫描1个分区
EXPLAIN
SELECT count(), avg(length(message))
FROM logs_by_day
WHERE timestamp >= '2025-04-15 00:00:00'
  AND timestamp <  '2025-04-16 00:00:00'
  AND level = 'ERROR';
```

输出关键行：

```
Expression ((Projection + Before ORDER BY))
  Aggregating
    Filter ((WHERE + ...))
      ReadFromMergeTree
      ┌─database─┬─table───────┬─ … ─┬─num_parts─┬─partitions─┐
      │ default  │ logs_by_day │      │         8 │ ['20250415']│
      └──────────┴─────────────┴──────┴───────────┴─────────────┘
```

`partitions: ['20250415']` —— 只扫描了一个分区。30 天的数据，只有 1/30 被纳入扫描范围。

再对比无分区表的查询计划：

```sql
EXPLAIN
SELECT count(), avg(length(message))
FROM logs_no_partition
WHERE timestamp >= '2025-04-15 00:00:00'
  AND timestamp <  '2025-04-16 00:00:00'
  AND level = 'ERROR';
```

输出显示 `partitions: ['all']` —— 虽然用到了 ORDER BY 索引来缩小 Granule 范围，但所有 107 个 Part 的元数据都需要被打开和读取，额外的元数据 IO 开销不可忽视。

```sql
-- 用 EXPLAIN indexes=1 对比实际扫描的 Granule 数
EXPLAIN indexes=1
SELECT count(), avg(length(message))
FROM logs_by_day
WHERE timestamp >= '2025-04-15 00:00:00'
  AND timestamp <  '2025-04-16 00:00:00'
  AND level = 'ERROR';
```

`indexes=1` 输出包含 `KeyCondition` 信息，确认时间范围和 level 都命中了索引，扫描的 Granule 数仅数百个，而非全表的上万个。

#### Step 3：ORDER BY 列顺序对查询性能的影响

这是本章最重要的动手实验。我们创建三张表，唯一区别是 ORDER BY 列的顺序：

```sql
-- 表A：最优顺序，过滤频率从高到低
CREATE TABLE test_order_abc (
    a UInt32, b UInt32, c UInt32, payload String
) ENGINE = MergeTree()
PARTITION BY a % 10
ORDER BY (a, b, c);

-- 表B：次优顺序
CREATE TABLE test_order_bac (
    a UInt32, b UInt32, c UInt32, payload String
) ENGINE = MergeTree()
PARTITION BY a % 10
ORDER BY (b, a, c);

-- 表C：最差顺序
CREATE TABLE test_order_cba (
    a UInt32, b UInt32, c UInt32, payload String
) ENGINE = MergeTree()
PARTITION BY a % 10
ORDER BY (c, b, a);

-- 向三张表写入相同的 1000 万行数据
INSERT INTO test_order_abc
SELECT
    rand() % 100     AS a,
    rand() % 1000    AS b,
    rand() % 1000000 AS c,
    repeat('x', 100) AS payload
FROM numbers(10000000);

INSERT INTO test_order_bac SELECT * FROM test_order_abc;
INSERT INTO test_order_cba SELECT * FROM test_order_abc;

-- 强制合并以确保索引结构完整（释放 Part 碎片对测试的干扰）
OPTIMIZE TABLE test_order_abc  FINAL;
OPTIMIZE TABLE test_order_bac  FINAL;
OPTIMIZE TABLE test_order_cba  FINAL;
```

现在用同样的查询（WHERE 条件完美匹配 `(a, b, c)` 顺序）测试三张表：

```sql
EXPLAIN indexes=1
SELECT count() FROM test_order_abc
WHERE a = 42 AND b = 500 AND c = 123456;
```

解读输出中的 `KeyCondition`：

- **test_order_abc**（`ORDER BY (a, b, c)`）：`KeyCondition: (a == 42) and (b == 500)` ——三列全部命中，扫描极少数 Granule。
- **test_order_bac**（`ORDER BY (b, a, c)`）：`KeyCondition: (b == 500) and (a == 42)` ——虽然也命中了两列，但由于 `b` 是第一列而查询中没有对 `b` 做精确前缀约束，实际使用了 `b` 的区间裁剪而非精确命中，Granule 扫描范围比表 A 大。
- **test_order_cba**（`ORDER BY (c, b, a)`）：查询中 `a` 在最前，但 ORDER BY 把 `c` 放在了第一列。`KeyCondition` 只能使用 `c` ——而 `c = 123456` 只会命中一个 Granule，这看起来反而更少。等等，这是看似反常的——事实上因为 `c` 基数极高（100 万），精确匹配 `c = 123456` 确实只扫描极少数 Granule。但换一个查询：

```sql
-- 这次只查 a 和 b，不查 c
EXPLAIN indexes=1
SELECT count() FROM test_order_abc
WHERE a = 42 AND b = 500;

EXPLAIN indexes=1
SELECT count() FROM test_order_cba
WHERE a = 42 AND b = 500;
```

对于 `test_order_abc`，`(a, b)` 是最左前缀，索引全部命中。对于 `test_order_cba`，第一列是 `c`，而 WHERE 中没有 `c` —— **索引完全失效，全表扫描**。

```
-- test_order_abc 输出:
-- ReadFromMergeTree (default.test_order_abc)
-- Parts: 1, Granules: 12/1235 (约 1% 的数据被扫描)

-- test_order_cba 输出:
-- ReadFromMergeTree (default.test_order_cba)
-- Parts: 1, Granules: 1235/1235 (全表扫描！)
```

**结论**：把 WHERE 中最频繁出现的列放在 ORDER BY 最前面，否则索引形同虚设。

#### Step 4：分区合并观察

现在观察 MergeTree 后台如何将小 Part 逐步合并为大 Part：

```sql
-- 分批插入数据，故意产生大量小 Part
INSERT INTO logs_by_day
SELECT
    toDateTime('2025-05-01 00:00:00') + number * 10 AS timestamp,
    ['INFO','WARN','ERROR','DEBUG'][rand() % 4 + 1] AS level,
    concat('batch insert message #', toString(number)) AS message,
    rand() % 1000000 AS user_id
FROM numbers(100000);

-- 重复执行上面这句 20 次，模拟 20 批写入
-- (在 clickhouse-client 中按上箭头 + 回车快速重复)
```

每插入一批，立即查看该分区的 Part 变化：

```sql
SELECT
    partition,
    count()   AS part_count,
    sum(rows) AS total_rows,
    min(min_time), max(max_time)
FROM system.parts
WHERE table = 'logs_by_day' AND database = currentDatabase() AND active = 1
GROUP BY partition
ORDER BY partition DESC
LIMIT 5;
```

你会看到分区 `20250501` 下的 Part 数在持续增长——从 5 到 10 到 20……然后突然减少到 1-2 个。这就是后台 Merge 在发挥作用。

手动触发合并：

```sql
OPTIMIZE TABLE logs_by_day PARTITION '20250501' FINAL;

-- 合并完成后再次查看——Part 数应收敛到 1
SELECT
    partition,
    count() AS part_count,
    sum(rows) AS total_rows
FROM system.parts
WHERE table = 'logs_by_day' AND database = currentDatabase() AND active = 1
  AND partition = '20250501'
GROUP BY partition;
```

> **踩坑**：`OPTIMIZE ... FINAL` 是阻塞操作，大表不要在生产高峰期执行。生产环境依赖后台自动 Merge 即可，手动 OPTIMIZE 仅用于紧急运维场景。

#### Step 5：监控 Part 健康度

生产中应对 Part 数量做持续监控，以下 SQL 可以放入告警脚本：

```sql
SELECT
    table,
    partition,
    count()                                          AS part_count,
    sum(rows)                                        AS total_rows,
    formatReadableSize(sum(bytes_on_disk))          AS disk_size,
    dateDiff('second', min(min_time), max(max_time)) AS time_span_seconds
FROM system.parts
WHERE database = currentDatabase()
  AND active = 1
GROUP BY table, partition
ORDER BY part_count DESC
LIMIT 10;
```

告警规则建议：
- 单分区 Part 数 > 50 → 警告级告警
- 单分区 Part 数 > 200 → 严重级告警，需检查 Merge 是否阻塞
- 单表总 Part 数 > 10000 → 立即扩容或合并

## 4. 项目总结

### 分区设计原则表

| 场景 | 推荐分区大小 | 示例表达式 | 原因 |
|------|------------|-----------|------|
| 大规模日志数据（日均 > 10 亿） | 按天 | `toYYYYMMDD(timestamp)` | 需要按天清理过期数据，数据量大值得精细化管理 |
| 中等规模分析数据（日均 100 万 ~ 10 亿） | 按月 | `toYYYYMM(timestamp)` | 平衡灵活性和 Part 数，大部分分析查询按月汇总 |
| 多租户 SaaS 平台 | 按租户 ID | `tenant_id` | 查询几乎都带 `WHERE tenant_id=xxx`，天然分区裁剪 |
| 维表 / 小表（总行数 < 100 万） | 不分区 | `tuple()` | 数据量小，分区开销大于收益 |
| 实时大屏（秒级刷新） | 不分区 + 物化视图 | — | 分区不能解决实时性问题，物化视图才是正解 |

### 排序键设计原则

1. **高频过滤列放最前面**：把 WHERE 中 80% 的查询都会带的列放在 ORDER BY 第一列。
2. **基数从低到高排列**：`event_type`（基数 10）放前面，`user_id`（基数 1 亿）放后面。这样索引能更高效地过滤。
3. **列数控制在 4-5 列以内**：超过 5 列的 ORDER BY 会导致索引文件膨胀，写入性能下降，而查询增益递减。
4. **与分区键形成互补**：分区键做"粗筛"（如按天），排序键做"精筛"（如按 user_id + timestamp）。两者配合形成两级过滤。
5. **避免将时间戳放在第一列**：除非你 90% 的查询都是 `WHERE timestamp BETWEEN ... AND ...`。大多数业务查询会先按某个业务维度（如用户、商品、事件类型）过滤，再限制时间范围。

### 常见踩坑经验

**坑1：按小时/分钟分区导致百万 Part 爆炸**

某团队为了做分钟级实时大屏，使用了 `PARTITION BY toYYYYMMDDhhmmss(timestamp)`。半年后分区数超过 15 万，Part 数突破 60 万。ClickHouse 启动加载元数据需要 3 分钟，ZooKeeper 内存耗尽导致整个集群只读。**解决方案**：改为按天分区 + 物化视图实现分钟级聚合，Part 数降至 300 以内。

**坑2：ORDER BY 列顺序反了导致全表扫描**

某用户表 `ORDER BY (create_time, user_id)`，但 90% 的查询是 `WHERE user_id = xxx`。由于 `user_id` 在第二列，查询无法利用索引，每次全表扫描 800 万行。**解决方案**：改为 `ORDER BY (user_id, create_time)`，查询延迟从 2.3 秒降至 12 毫秒。

**坑3：分区表达式使用函数导致裁剪失效**

建表 `PARTITION BY toDate(timestamp)`，但查询写的是 `WHERE timestamp >= '2025-04-01 00:00:00'`。ClickHouse 无法自动推导 `toDate(timestamp) == '2025-04-01'`，分区裁剪失效。**解决方案**：保持分区表达式与查询条件中的表达式一致——要么都用 `toDate()`，要么都用原始 `DateTime` 比较。

### 思考题

1. **为什么 ClickHouse 不推荐按小时分区？如果确实需要按小时查询最近 24 小时的数据，有什么更好的方案？**
   > 提示：考虑物化视图 + 按月/按天分区的组合，以及 `PREWHERE` 的作用。

2. **`ORDER BY (user_id, timestamp)` 和 `ORDER BY (timestamp, user_id)` 分别适用于什么查询场景？如果有张表两种查询都很频繁，应该如何设计？**
   > 提示：考虑 Projection（投影）作为第二索引的替代方案，参考第 21 章。
