## 1. 项目背景

某IoT监控公司，管理着10万台边缘设备。每台设备每10秒上报一次数据——温度、湿度、气压、电池电量四个指标。日增数据约3.3亿行，累积半年的原始数据已突破200TB。

公司的存储架构全跑在SSD上，月账单2万美元。CTO在月会上把AWS账单投影到幕布上时，会议室安静了整整十秒——存储成本已是人力成本的1.7倍，而且曲线还在以每月8%的斜率攀升。

团队调研了一圈压缩方案。LZ4是ClickHouse的默认Codec——写入快、查询快，但压缩率只有2-3倍。隔壁时序数据库团队用ZSTD把同样的传感器数据压到原体积的30%以下，声称换Codec就能把存储成本砍掉60%。更让人心动的是那些"专用Codec"——Delta编码针对递增序列能做到10倍压缩，DoubleDelta对时间戳能压到50倍，Gorilla对浮点数指标能实现10倍以上的压缩率。

但这中间有一个巨大的陷阱：压缩率和查询性能是跷跷板的两端。ZSTD(9)的解压开销足以让全表扫描的查询延迟翻3倍以上。Delta编码对乱序数据几乎无效，甚至可能膨胀。盲目给所有列套上"最强压缩"，结果可能是存储省了钱，查询慢了用户体验，CPU账单反手又把省下的钱吞了回去。

更关键的是，ClickHouse支持**列级Codec**——每一列都可以独立选择压缩算法和编码方式。时间戳用DoubleDelta，流水ID用Delta，随机UUID用LZ4打底，浮点指标用Gorilla。一张表里六列各用六种不同的Codec，这才是ClickHouse压缩策略的精髓所在。

本章将围绕"数据压缩与编码优化"展开，从LZ4与ZSTD的取舍，到Delta/DoubleDelta/Gorilla/T64四大专用编码的实战应用，再到在线修改Codec的运维操作，带你建立一套完整的列级压缩选型方法论。

## 2. 项目设计：剧本式交锋对话

周一晨会刚结束，小白拿着一份存储成本报表走进工位区。小胖正对着屏幕上的默认建表语句发呆。

**小胖**（头也不抬）："压缩不就是LZ4吗？ClickHouse默认的还能错？省那点空间不如直接加硬盘。200TB而已，加四块SSD不就搞定了？"

**小白**把报表往小胖桌上一拍："四块SSD？你看看这个——咱们200TB全是NVMe SSD，每个月光存储账单就是这个数。隔壁时序库团队用ZSTD把120TB压到了35TB，咱们用的什么？全是默认LZ4！"

**小胖**从椅子上坐直了："等下，压缩不就是让数据变小吗？LZ4不也是压缩？ZSTD能好到哪去？而且压缩率越高，查询的时候解压不就越慢？这不是脱了裤子放屁吗——省了存储费，CPU费又涨上去了？"

**大师**端着咖啡走过来，正好接住这个问题："小胖问到了压缩优化的核心矛盾——**空间与时间的权衡**。但你的假设有个盲区：不是所有数据都按同一种规律变化，所以也不是所有列都应该用同一种Codec。"

大师在白板上画了一张表：

```
LZ4：默认压缩，压缩率2-3x，解压速度最快（3GB/s/core），CPU开销低
ZSTD(1)：轻度压缩，压缩率3-5x，解压速度中（1.5GB/s/core）
ZSTD(3)：均衡压缩，压缩率4-7x，解压速度中
ZSTD(9)：重度压缩，压缩率6-15x，解压速度慢（0.8GB/s/core），CPU开销最高
```

"ZSTD的威力取决于数据本身的可压缩性。对于高基数的随机字符串——比如UUID——LZ4压到2x，ZSTD(9)也只能压到2.5x，省下的空间微乎其微，但解压却慢了。对于低基数、高重复的数据——比如状态码、设备类型——ZSTD(9)能把10GB压到500MB，这时候用ZSTD就非常划算。"

**技术映射 #1**：LZ4追求速度，ZSTD追求比率。核心差异在于LZ4使用固定大小的滑动窗口（64KB），算法简单，解压几乎等于"拷贝+还原指针"。ZSTD使用更大的搜索窗口、更复杂的熵编码（FSE/Huffman），压缩时对数据做深度分析，解压开销自然更高。因此选Codec的本质是——这列数据的查询频率和扫描量，是否值得用解压开销去换存储空间。

---

**小白**翻了翻文档，举手提问："大师，我看到ClickHouse除了LZ4和ZSTD，还有Delta、DoubleDelta、Gorilla、T64这些'编码'。它们跟压缩算法是什么关系？是替代还是叠加？"

**大师**："问得非常精准。编码（Encoding）和压缩（Compression）是两层独立的处理——它们可以叠加。编码是'数据变换'，不减少体积，而是改变数据的表示形式；压缩是'数据紧缩'，在编码的基础上真正减少体积。"

大师敲开终端，写了一段伪逻辑：

```
写入流程：
原始数据 → [编码预处理: Delta/DoubleDelta/Gorilla/T64] → [压缩: LZ4/ZSTD] → 落盘

读取流程：
磁盘数据 → [解压: LZ4/ZSTD] → [编码还原] → 原始数据
```

"以Delta编码为例。假设你有一列设备ID：`1001, 1002, 1003, 1004, 1005`。原始存储需要5个32位整数=20字节。Delta编码先计算相邻值的差：`1001, +1, +1, +1, +1`。此时差值变成了小整数，ZSTD再压这些小整数，压缩率可以从2x直接跳到15x——因为差值序列的熵比原始值低得多。"

"DoubleDelta更巧妙——它对差值再求一次差值，专门对付时间戳这种**匀速增长**的数据。比如 `2025-04-30 10:00:00, 10:00:10, 10:00:20, 10:00:30`。第一次差分得到 `10, 10, 10, 10` 秒，第二次差分得到 `10, 0, 0, 0`。一堆零和一两个非零值——压缩器看到这种数据高兴坏了，压缩率轻松突破50倍。"

"Gorilla则完全不同。它是Facebook专门为浮点时间序列设计的编码算法，利用了IEEE 754浮点数的比特级规律：相邻时间戳的毫秒差通常只变低32位，相邻测量值的浮点表示通常只有尾数的最后几位不同。Gorilla用XOR操作提取差异比特位，再对前导零和尾随零做游程编码，对平稳的时间序列指标能实现10-15x的压缩。"

"T64则是最简单的——它不分析值之间的关系，只压缩单个值的**存储宽度**。比如 `battery_level UInt8`，值的范围永远是0-100。如果ClickHouse内部用了64位存储，T64会把它压缩到实际有效位宽对应的最小空间。"

**技术映射 #2**：编码 = 数据变换，降低信息熵；压缩 = 数据紧缩，消除冗余。编码让压缩器的输入"更容易压"，二者叠加的压缩率远高于单独使用。列级Codec的精髓在于：为每列数据选择最适合其数学规律的编码+压缩组合。

**小胖**挠了挠头："听着是挺好，但我有几个实操问题。第一，我建表的时候已经用了LZ4默认，现在想改成ZSTD——能在线改吗？要停服吗？历史数据怎么处理？"

**大师**："这就是`ALTER TABLE ... MODIFY COLUMN ... CODEC(...)`的用途——可以在线修改列的压缩策略，不需要停服。"

大师写下SQL：

```sql
-- 在线修改列的Codec（只影响此后新写入的Part）
ALTER TABLE sensor_data_lz4
    MODIFY COLUMN temperature CODEC(Delta, ZSTD(3));

-- 存量数据需要Mutation来重压缩（会产生新的数据Part）
ALTER TABLE sensor_data_lz4
    UPDATE temperature = temperature WHERE 1;

-- 监控Mutation进度
SELECT table, command, is_done, latest_fail_reason
FROM system.mutations
WHERE table = 'sensor_data_lz4';
```

"注意两点。第一，`MODIFY COLUMN ... CODEC`只影响命令执行后新写入的数据块。已有的数据Part不会自动重压缩——它们的Codec是在写入时就已经锁定的。第二，`UPDATE ... WHERE 1`会触发一次Mutation，把全表数据读出来再写回去，这个过程会使用新的Codec。在100TB的线上表上跑全量Mutation是非常重的操作，要选在业务低峰期。更好的做法是分区分批迁移——`ALTER TABLE ... UPDATE ... WHERE partition_col = '2025-01'`，一次做一个月的量。"

**小白**追问："那`system.columns`里能看到每列用的什么Codec吗？我怎么知道现状？"

```sql
SELECT
    name,
    type,
    compression_codec
FROM system.columns
WHERE table = 'sensor_data_lz4'
  AND database = 'default';
```

**小胖**："最后再问一个——如果我搞不清楚一列该用什么Codec，有没有万能的安全做法？"

**大师**笑了笑："有。拿不准就用LZ4，拿得准才上专用Codec。这是我的'三句话选型口诀'：

1. **有顺序的列用Delta/DoubleDelta**——时间戳、自增ID、递增序号。它们相邻值有强规律，Delta编码收益最大，解压开销最小。
2. **有范围的列用T64**——状态码、枚举值、小整数。T64几乎不增加CPU开销，纯粹靠减少存储位宽来节省空间。
3. **有波动的浮点列用Gorilla**——温度、CPU使用率、股价。浮点数据的相邻值高度相关，Gorilla的XOR差分能精准捕获这种规律。
4. **什么都对不上的用ZSTD(3)打底**——随机字符串、URL、JSON日志。这些数据没有数学规律，纯靠压缩算法硬压，ZSTD(3)是性价比最优的平衡点。"

### 分步实现

#### Step 1：创建不同压缩策略的对比表

首先，创建三张结构相同但压缩策略不同的表，用于对比实验。

**表一：LZ4默认（基线）**

```sql
CREATE TABLE sensor_data_lz4 (
    device_id UInt32,
    timestamp DateTime,
    temperature Float32,
    humidity Float32,
    pressure Float32,
    battery_level UInt8
) ENGINE = MergeTree()
ORDER BY (device_id, timestamp)
PARTITION BY toYYYYMM(timestamp)
SETTINGS min_compress_block_size = 65536;
```

**表二：ZSTD(3)均衡方案 + 列级编码**

先创建基础表，再用ALTER添加列级Codec：

```sql
CREATE TABLE sensor_data_zstd3 (
    device_id UInt32,
    timestamp DateTime,
    temperature Float32,
    humidity Float32,
    pressure Float32,
    battery_level UInt8
) ENGINE = MergeTree()
ORDER BY (device_id, timestamp)
PARTITION BY toYYYYMM(timestamp)
SETTINGS min_compress_block_size = 65536;

-- 为每列选择合适的编码+压缩组合
ALTER TABLE sensor_data_zstd3
    MODIFY COLUMN timestamp CODEC(DoubleDelta, ZSTD(3)),
    MODIFY COLUMN temperature CODEC(Delta, ZSTD(3)),
    MODIFY COLUMN humidity CODEC(Delta, ZSTD(3)),
    MODIFY COLUMN pressure CODEC(Delta, ZSTD(3)),
    MODIFY COLUMN battery_level CODEC(T64, ZSTD(3));
```

> `min_compress_block_size` 设置了压缩块的最小大小，65536（64KB）意味着每攒够64KB的列数据才做一次压缩。块太小压缩率差，块太大则随机访问时解压开销大。64KB是一个合理的折中值。

**表三：ZSTD(9)极限方案 + 专用编码**

建表时直接在DDL中声明每列的CODEC：

```sql
CREATE TABLE sensor_data_zstd9 (
    device_id UInt32 CODEC(T64, ZSTD(9)),
    timestamp DateTime CODEC(DoubleDelta, ZSTD(9)),
    temperature Float32 CODEC(Gorilla, ZSTD(9)),
    humidity Float32 CODEC(Gorilla, ZSTD(9)),
    pressure Float32 CODEC(Gorilla, ZSTD(9)),
    battery_level UInt8 CODEC(T64, ZSTD(9))
) ENGINE = MergeTree()
ORDER BY (device_id, timestamp)
PARTITION BY toYYYYMM(timestamp);
```

> 注意：`device_id` 使用了 `T64`。虽然设备ID是可枚举的，但相邻值并不一定是连续的（取决于插入顺序）。如果数据按设备ID排序后再写入，Delta效果会更好。这里的假设是部分递增但仍有一定跳跃，因此T64压缩位宽比Delta更稳妥。

完成后验证三张表的Codec配置：

```sql
SELECT
    table,
    name,
    type,
    compression_codec
FROM system.columns
WHERE table IN ('sensor_data_lz4', 'sensor_data_zstd3', 'sensor_data_zstd9')
ORDER BY table, column_position;
```

#### Step 2：写入相同数据

向三张表分别导入1000万行模拟传感器数据。为了模拟真实的IoT场景，数据需要有一定的时序规律——温度在小范围内波动，气压缓慢变化，电池电量阶梯式递减。

```sql
-- 向LZ4表写入数据
INSERT INTO sensor_data_lz4
SELECT
    number % 10000 AS device_id,
    now() - INTERVAL (number % 86400 * 30) SECOND AS timestamp,
    20 + (number % 150) / 10.0 + (rand() % 5) / 10.0 AS temperature,
    40 + (number % 200) / 5.0 + (rand() % 3) / 10.0 AS humidity,
    1013.25 + (number % 1000 - 500) / 50.0 + (rand() % 10) / 100.0 AS pressure,
    (number * 7) % 101 AS battery_level
FROM numbers(10000000);

-- 从LZ4表拷贝到另外两张表（确保数据完全一致）
INSERT INTO sensor_data_zstd3 SELECT * FROM sensor_data_lz4;
INSERT INTO sensor_data_zstd9 SELECT * FROM sensor_data_lz4;
```

稍等片刻让后台Merge完成后，确认三张表行数一致：

```sql
SELECT
    'LZ4(default)' AS table_name,
    count() AS rows
FROM sensor_data_lz4
UNION ALL
SELECT 'ZSTD(3)+Delta', count()
FROM sensor_data_zstd3
UNION ALL
SELECT 'ZSTD(9)+Gorilla', count()
FROM sensor_data_zstd9;
```

#### Step 3：对比存储效果

这是本章最核心的对比——三种压缩策略的实际存储差异：

```sql
SELECT
    'LZ4(default)' AS codec,
    formatReadableSize(sum(bytes_on_disk)) AS disk_size,
    formatReadableSize(sum(data_uncompressed_bytes)) AS raw_size,
    round(sum(data_uncompressed_bytes) / sum(bytes_on_disk), 1) AS compression_ratio
FROM system.parts
WHERE table = 'sensor_data_lz4' AND active

UNION ALL

SELECT
    'ZSTD(3)+Delta' AS codec,
    formatReadableSize(sum(bytes_on_disk)) AS disk_size,
    formatReadableSize(sum(data_uncompressed_bytes)) AS raw_size,
    round(sum(data_uncompressed_bytes) / sum(bytes_on_disk), 1) AS compression_ratio
FROM system.parts
WHERE table = 'sensor_data_zstd3' AND active

UNION ALL

SELECT
    'ZSTD(9)+Gorilla' AS codec,
    formatReadableSize(sum(bytes_on_disk)) AS disk_size,
    formatReadableSize(sum(data_uncompressed_bytes)) AS raw_size,
    round(sum(data_uncompressed_bytes) / sum(bytes_on_disk), 1) AS compression_ratio
FROM system.parts
WHERE table = 'sensor_data_zstd9' AND active;
```

预期输出（基于1000万行传感器数据）：

| Codec | Disk Size | Raw Size | Ratio |
|-------|-----------|----------|-------|
| LZ4(default) | ~380 MiB | ~1.80 GiB | 4.8x |
| ZSTD(3)+Delta | ~140 MiB | ~1.80 GiB | 13.2x |
| ZSTD(9)+Gorilla | ~85 MiB | ~1.80 GiB | 21.7x |

如果想看每列的压缩详情，可以用这个查询：

```sql
SELECT
    table,
    column,
    formatReadableSize(sum(column_bytes_on_disk)) AS column_disk,
    formatReadableSize(sum(column_data_uncompressed_bytes)) AS column_raw,
    round(sum(column_data_uncompressed_bytes) / sum(column_bytes_on_disk), 1) AS column_ratio
FROM system.parts_columns
WHERE table IN ('sensor_data_lz4', 'sensor_data_zstd9')
  AND active
GROUP BY table, column
ORDER BY table, column;
```

这能让你看到每列的压缩效果差异——`timestamp` 列在 `DoubleDelta + ZSTD(9)` 下可能达到50倍压缩，而 `device_id` 在 `T64 + ZSTD(9)` 下可能只有3-4倍。

#### Step 4：对比查询性能

压缩省了空间，但查询会不会变慢？我们设计三种典型查询模式来全面评估。

**查询A：范围扫描（利用主键）**

```sql
-- 每个表执行3次取平均，\timing 开启计时
\timing

SELECT device_id, avg(temperature), max(pressure)
FROM sensor_data_lz4
WHERE timestamp >= now() - INTERVAL 7 DAY
GROUP BY device_id
LIMIT 100;
-- 记下耗时

SELECT device_id, avg(temperature), max(pressure)
FROM sensor_data_zstd3
WHERE timestamp >= now() - INTERVAL 7 DAY
GROUP BY device_id
LIMIT 100;
-- 记下耗时

SELECT device_id, avg(temperature), max(pressure)
FROM sensor_data_zstd9
WHERE timestamp >= now() - INTERVAL 7 DAY
GROUP BY device_id
LIMIT 100;
-- 记下耗时
```

**查询B：全表聚合（不能利用主键部分裁剪）**

```sql
SELECT
    count() AS total_rows,
    avg(temperature) AS avg_temp,
    max(pressure) AS max_pressure,
    min(battery_level) AS min_battery
FROM sensor_data_lz4;
-- 同样在三张表上各跑一次
SELECT
    count() AS total_rows,
    avg(temperature) AS avg_temp,
    max(pressure) AS max_pressure,
    min(battery_level) AS min_battery
FROM sensor_data_zstd3;

SELECT
    count() AS total_rows,
    avg(temperature) AS avg_temp,
    max(pressure) AS max_pressure,
    min(battery_level) AS min_battery
FROM sensor_data_zstd9;
```

**查询C：单点查询（按设备ID精确查找）**

```sql
SELECT * FROM sensor_data_lz4
WHERE device_id = 42
ORDER BY timestamp DESC
LIMIT 10;

SELECT * FROM sensor_data_zstd3
WHERE device_id = 42
ORDER BY timestamp DESC
LIMIT 10;

SELECT * FROM sensor_data_zstd9
WHERE device_id = 42
ORDER BY timestamp DESC
LIMIT 10;
```

> 如果你在测试中发现ZSTD(9)的查询比LZ4慢3倍以上——这不是Bug，而是正常的CPU-I/O权衡。ZSTD(9)的压缩块更小、更密集，但解压时CPU要做更多的工作。当查询只需要扫描少量Granule时，LZ4的优势明显（解压快，I/O本来就不多）；当查询需要全表扫描时，ZSTD(9)的I/O节省可能抵消解压开销，甚至反超LZ4。这就是为什么你需要"根据查询模式选Codec"而不是"无脑用最强压缩"。

#### Step 5：列级CODEC选择最佳实践表

基于前面的实验结论，建一张"教科书级"的Codec示范表：

```sql
CREATE TABLE best_practice_codec (
    -- 递增时间戳：DoubleDelta 完美匹配
    event_time DateTime CODEC(DoubleDelta, ZSTD(3)),

    -- 自增流水号：相邻值差为1，Delta压缩率极高
    order_id UInt64 CODEC(Delta, ZSTD(3)),

    -- 随机用户ID：值分布均匀，无序列规律，纯靠ZSTD
    user_id UInt64 CODEC(ZSTD(3)),

    -- 浮点CPU使用率：相邻采样点高度相关，Gorilla最擅长
    cpu_usage Float32 CODEC(Gorilla, ZSTD(3)),

    -- 小范围整数状态码：T64压缩存储位宽
    status_code UInt16 CODEC(T64, ZSTD(3)),

    -- 低基数事件类型：用LowCardinality字典编码替代通用Codec
    event_type LowCardinality(String),

    -- 大文本日志：高压缩比优先，查询频率低
    log_message String CODEC(ZSTD(9))
) ENGINE = MergeTree()
ORDER BY (event_time, order_id);
```

每一行注释都在告诉你——Codec的选择依据是什么。这比任何速查表都更有参考价值。

#### Step 6：在线修改压缩策略（生产环境实操）

假设你有一张用LZ4跑了大半年的线上表，现在想迁移到新的列级Codec方案。正确的操作流程是：

```sql
-- 1. 先改元数据（只影响新写入的Part）
ALTER TABLE sensor_data_lz4
    MODIFY COLUMN timestamp CODEC(DoubleDelta, ZSTD(3)),
    MODIFY COLUMN temperature CODEC(Delta, ZSTD(3)),
    MODIFY COLUMN humidity CODEC(Delta, ZSTD(3)),
    MODIFY COLUMN pressure CODEC(Delta, ZSTD(3)),
    MODIFY COLUMN battery_level CODEC(T64, ZSTD(3));

-- 2. 验证Codec已更新到元数据
SELECT name, compression_codec
FROM system.columns
WHERE table = 'sensor_data_lz4';

-- 3. 对存量数据执行Mutation（生产环境请逐分区执行！）
-- 错误做法：ALTER TABLE ... UPDATE temperature = temperature WHERE 1;  -- 全表一把梭！
-- 正确做法：逐分区迁移
ALTER TABLE sensor_data_lz4
    UPDATE temperature = temperature WHERE toYYYYMM(timestamp) = '202501',
    UPDATE humidity = humidity WHERE toYYYYMM(timestamp) = '202501',
    UPDATE pressure = pressure WHERE toYYYYMM(timestamp) = '202501',
    UPDATE battery_level = battery_level WHERE toYYYYMM(timestamp) = '202501',
    UPDATE timestamp = timestamp WHERE toYYYYMM(timestamp) = '202501';

-- 4. 监控Mutation进度
SELECT
    table,
    command,
    is_done,
    parts_to_do,
    latest_fail_reason
FROM system.mutations
WHERE table = 'sensor_data_lz4'
  AND NOT is_done
ORDER BY create_time;
```

> **关键要点**：`MODIFY COLUMN ... CODEC` 是元数据操作，秒级完成。但数据的实际重压缩需要通过Mutation（或OPTIMIZE）触发。Mutation会复制整个Part并在新Part中应用新Codec——这是一个后台异步过程。在单分区数据超过50GB的表上，全表Mutation可能跑数小时甚至数天。

### 测试验证

完成所有步骤后，汇总对比结果：

```sql
-- 最终压缩率汇总
SELECT
    codec,
    formatReadableSize(disk_size) AS disk,
    compression_ratio
FROM (
    SELECT
        'LZ4(default)' AS codec,
        sum(bytes_on_disk) AS disk_size,
        round(sum(data_uncompressed_bytes) / sum(bytes_on_disk), 1) AS compression_ratio
    FROM system.parts
    WHERE table = 'sensor_data_lz4' AND active

    UNION ALL

    SELECT
        'ZSTD(3)+Delta/T64',
        sum(bytes_on_disk),
        round(sum(data_uncompressed_bytes) / sum(bytes_on_disk), 1)
    FROM system.parts
    WHERE table = 'sensor_data_zstd3' AND active

    UNION ALL

    SELECT
        'ZSTD(9)+Gorilla/DoubleDelta',
        sum(bytes_on_disk),
        round(sum(data_uncompressed_bytes) / sum(bytes_on_disk), 1)
    FROM system.parts
    WHERE table = 'sensor_data_zstd9' AND active
);
```

预期结论：
- LZ4默认方案存储最大，但全表扫描查询最快
- ZSTD(3)+Delta均衡方案存储节省约65%，查询性能下降可控制在15%以内
- ZSTD(9)+Gorilla极限方案存储节省约78%，但高并发查询时CPU开销显著上升

## 4. 项目总结

### Codec选择速查表

经过本章的实验和对比，下面是面向不同列类型的推荐Codec方案：

| Column Type | Best Codec | Compression Ratio | Query Speed | 原理 |
|------------|-----------|-------------------|-------------|------|
| 递增时间戳 | DoubleDelta + ZSTD(3) | 10-50x | 极快 | 二阶差分消除线性规律 |
| 自增流水ID | Delta + ZSTD(3) | 5-20x | 极快 | 一阶差分后值域极小 |
| 随机ID/UUID | ZSTD(3) | 2-5x | 快 | 无序列规律，纯压缩 |
| 浮点监控指标 | Gorilla + ZSTD(3) | 5-15x | 中 | XOR位级差分专长 |
| 小整数(0-255) | T64 + ZSTD(3) | 3-10x | 极快 | 压缩存储位宽 |
| 大文本(>1KB) | ZSTD(9) | 5-15x | 慢 | 查询少，空间优先 |
| 低基数字符串 | LowCardinality(String) | — | 快 | 字典编码，不用Codec |
| 随机短字符串 | LZ4 | 2-3x | 最快 | 无规律可循，快最重要 |

### 决策树：选Codec的三步法

```
第一步：这列有没有数学规律？
  有 → 用专用编码（Delta/DoubleDelta/Gorilla/T64）
  无 → 跳到第二步

第二步：这列数据量大不大（>总容量的10%）？
  大 → 用 ZSTD(3)，空间收益可观
  小 → 用 LZ4，省CPU

第三步：这列的查询频率高不高？
  高 → 降低ZSTD级别（1-3），保查询速度
  低 → 提高ZSTD级别（5-9），保压缩率
```

### 适用场景

- **IoT传感器时序库**：timestamp用DoubleDelta + ZSTD(3)，指标列用Gorilla + ZSTD(3)。100TB原始数据可压至15TB以下。
- **金融逐笔成交数据**：成交时间用DoubleDelta，价格用Delta + ZSTD(3)，成交量用T64。TB级数据月存储成本可降70%。
- **应用日志与APM指标**：日志文本用ZSTD(9)极限压缩，结构化列按类型分别选Codec。
- **任何存储成本占总成本超过30%的数据平台项目**：简单地从LZ4迁移到ZSTD(3)就可能在查询性能基本不变的情况下，省掉一半的存储账单。

### 常见踩坑经验

1. **给所有列无脑套ZSTD(9)**：一张表里可能有20列，但只有2-3列占总数据量的80%。其余小列强行用ZSTD(9)只会拖慢解压速度，压缩率提升微乎其微。正确做法是——大列用心选，小列保持LZ4。

2. **MODIFY COLUMN CODEC后以为历史数据会自动转换**：无数次了——运维执行了`ALTER TABLE ... MODIFY COLUMN ... CODEC`，第二天一看磁盘使用量纹丝不动。因为元数据改了，Part里的数据还是老Format。必须执行Mutation或分区级的`OPTIMIZE FINAL`才会触发实际重压缩。请务必在`system.mutations`中确认进度。

3. **给乱序写入的数据用Delta编码**：Delta编码假设插入顺序就是数据本身的自然顺序。但如果你的数据是多个Kafka分区并发消费的（多线程乱序写入），`device_id`列的Delta编码效果会大打折扣——相邻值不再连续，差值变大，压缩率不升反降。解决方案是确保ORDER BY与数据的实际排列顺序一致，或者干脆不用Delta。

4. **Gorilla编码对整数列无效**：Gorilla是专门为Float32/Float64设计的——它操作的是IEEE 754比特位。对整数列使用Gorilla不仅没有效果，还会造成数据膨胀。ClickHouse 24.x版本开始会对此报Warning。

5. **混淆Codec中的编码和压缩的叠加顺序**：`CODEC(Delta, ZSTD(3))` 不是"Delta或者ZSTD(3)二选一"，而是"先Delta编码，再ZSTD(3)压缩"——管道叠加，顺序固定。如果你写`CODEC(Delta, LZ4)`，它的效果也比单独LZ4好很多，因为Delta降低了数据熵。
