# 第35章：分布式查询——计划拆分与结果合并

## 1. 项目背景

小明所在的数据平台团队最近遇到了一个令人困惑的性能问题。他们在3分片的ClickHouse集群上部署了一套用户行为分析系统，`events_dist` 表按 `user_id` 哈希分片，每个分片存储约2亿行数据。某天，产品经理临时要一个去重用户数统计：

```sql
SELECT count(DISTINCT user_id) FROM events_dist WHERE date = '2025-01-15';
```

查询跑了整整8秒才返回结果。团队起初怀疑是某个分片负载过高，于是逐个直连分片执行相同的查询：每个分片本地响应时间仅1秒左右。三个分片的本地计算加起来不过3秒，那多出来的5秒去哪儿了？

他们在 `system.query_log` 中发现，发起节点（initiator）在这8秒内持续高CPU消耗，内存占用飙升至12GB。原来，三个分片各返回了约500万行 `user_id`（共计1500万行中间结果），发起节点需要将这些数据全部接收、去重、再合并。这个"收尾"动作——即分布式查询的结果合并（merge）——成了整个查询的瓶颈。

这不是孤例。许多刚接触ClickHouse分布式表的工程师都有一个朴素认知："分布式查询不就是把SQL发到各个节点，然后汇总结果吗？跟MapReduce差不多吧？" 但现实远比这个模型复杂。哪些算子下推到分片执行？哪些必须拉回发起节点？两阶段聚合（two-phase aggregation）如何协同工作？`GLOBAL IN` 和 `GLOBAL JOIN` 究竟在背后做了什么？如果分片有多个副本，如何避免重复计算？

理解分布式查询计划的拆分逻辑，掌握子查询下推（pushdown）的边界条件，清楚两阶段聚合的中间状态传递机制，以及 `GLOBAL` 子句以临时表广播方式实现跨分片数据一致的原理——这些是确保ClickHouse集群查询性能可线性扩展的关键。本章将深入ClickHouse源码层面，剖析 `StorageDistributed` 如何将一条SQL拆分成"分片执行计划"与"发起节点合并计划"，并以3分片集群上的实际场景为线索，逐步构建一个完整的高性能分布式查询方案。

## 2. 项目设计：剧本式交锋对话

**场景**：周五下午的架构评审会上，团队正在讨论新上线的实时看板接口频繁超时的问题。白板上写满了一行行分布式SQL和EXPLAIN输出。

**小胖**（后端开发，2年经验）："我不太理解，咱们的 `events_dist` 表不就是把查询自动分发到各个分片吗？本质上就是MapReduce的Map阶段在各个节点跑，然后把结果Reduce回发起节点。这有什么需要深入优化的？"

**大师**（平台架构师，8年ClickHouse经验）："小胖，你这个理解对了一半。确实，`StorageDistributed` 会把查询扇出（fan-out）到各个分片，但它不是无脑地把原始SQL原样发送。关键问题是：**哪些算子在分片上执行，哪些拉回发起节点执行**？这个决策直接影响网络传输量和发起节点的CPU负载。"

**小白**（数据工程师，刚转岗）："大师，能举个例子吗？我们之前那条 `SELECT count(DISTINCT user_id) FROM events_dist WHERE date='2025-01-15'` 到底发到分片上的是什么SQL？"

**大师**拿起马克笔在白板上画起来：

"ClickHouse根据查询的SQL特征，决定一个**查询处理阶段（QueryProcessingStage）**。源码在 `src/Storages/StorageDistributed.cpp` 中有一个核心方法 `getQueryProcessingStage()`，它会检查SQL中是否存在 `GROUP BY`、`LIMIT`、`DISTINCT` 等子句，然后选择三种策略之一：

- **FetchColumns**：最轻量的策略，分片只负责读原始列数据，不执行任何聚合或过滤（除了下推的WHERE条件）。所有计算——包括`count(DISTINCT ...)`——全部在发起节点完成。你那个慢查询就属于这种。

- **WithMergeableState**：当你写 `GROUP BY` 时触发。分片执行部分聚合（partial aggregation），返回的是聚合中间状态（如 `countState()`、`sumState()`），而不是最终结果。发起节点收到这些中间状态后，执行合并聚合（merging aggregation），得到最终结果。

- **Complete**：当查询包含 `LIMIT` 但没有 `GROUP BY` 时，分片完整执行查询并返回最终行。发起节点只需要归并排序并再次应用 `LIMIT`。

小胖你的那条 `count(DISTINCT user_id)`，因为既没有 `GROUP BY` 也没有 `LIMIT`，所以走了 `FetchColumns` 路径。三个分片各把500万行 `user_id` 原样发给发起节点，由发起节点单线程做去重——不慢才怪。"

**小胖**："原来如此！那我加上 `GROUP BY user_id`，让分片做部分去重不就行了？"

**大师**摇摇头："`GROUP BY user_id` 会让每个分片返回去重后的 `user_id` 列表，但不同分片之间可能有相同的 `user_id`，发起节点还得再做一次去重。你只是把1500万行变成了1000万行，本质问题没解决。

真正的解法是用**两阶段聚合**配合组合聚合函数（combinators）。比如 `uniqExact` 本身就是一个去重计数函数，它的中间状态 `uniqExactState` 可以用 `uniqExactMerge` 合并。所以你的查询可以改写为显式的两阶段写法，或者直接使用 `count(DISTINCT user_id)` 配合 `distributed_aggregation_memory_efficient=1` 设置，让ClickHouse自动选择更优的分布式计划。"

**小白**："那 `GLOBAL IN` 又是怎么回事？我看文档说分布式表上做子查询必须加 `GLOBAL`，不然结果会错。"

**大师**："好问题。"他在白板上画了一个图：

```
-- 不加 GLOBAL（错误！）
SELECT * FROM orders_dist WHERE user_id IN (SELECT user_id FROM vip_users)

-- 每个分片独立执行：
--   Shard1: SELECT * FROM orders_local WHERE user_id IN (SELECT user_id FROM vip_users_local)
--   Shard2: SELECT * FROM orders_local WHERE user_id IN (SELECT user_id FROM vip_users_local)
--   Shard3: SELECT * FROM orders_local WHERE user_id IN (SELECT user_id FROM vip_users_local)
-- 问题：vip_users_local 只包含该分片上的VIP用户，不是全量VIP！
```

"而加了 `GLOBAL` 之后，ClickHouse的实现逻辑是：

1. 发起节点先在本机执行子查询 `SELECT user_id FROM vip_users`，得到全量VIP用户列表（比如1000个ID）。
2. 发起节点在**每个分片上**创建一张临时表（`CREATE TEMPORARY TABLE _data_xxx ... ENGINE=Memory`），将这1000个ID写入该临时表。
3. 发起节点将查询改写为 `SELECT * FROM orders_local WHERE user_id IN _data_xxx` 发往各分片。
4. 各分片执行完毕后，发起节点清理所有临时表。

核心思想是：**子查询在发起节点执行一次，结果广播到所有分片**，而不是每个分片各自执行一遍。"

**小胖**："那如果子查询结果很大——比如上百万行——临时表岂不是要炸？"

**大师**赞许地点头："这个意识很好。`GLOBAL IN` 的子查询结果集被完全加载到内存中，并通过网络复制到每个分片。如果你要筛选100万个ID，可能不如用 `GLOBAL JOIN` 或者重新设计分片键，让关联数据落在同一个分片上（co-location）。源码里 `StorageDistributed::createGlobalInSubQuery()` 和 `ExternalDistributedTable` 就负责这套广播逻辑。"

**小白**："最后一个问题，分布式表多副本的情况下，发起节点会不会重复查询所有副本导致结果翻倍？"

**大师**："ClickHouse对这种情况有参数控制。关键参数是 `load_balancing`，默认值是 `'random'`，即每个分片随机选一个副本执行。但更推荐在生产环境打开 `prefer_localhost_replica=1`，优先查询和发起节点在同一台机器上的副本，避免跨网络IO。

如果你的副本之间是内部复制（`internal_replication=true`），ClickHouse知道所有副本数据一致，所以每个分片只会查询一个副本。但如果关闭了内部复制且没有正确配置负载均衡策略，确实可能重复计算。这点在配置 `Distributed` 表引擎时必须留意。"

**小胖**若有所思："所以分布式查询本质上是一个**查询计划的拆分与重写**过程，而不是简单的SQL转发……"

**大师**："没错。你把它理解为一个分布式的查询优化器。它要做分片选择、算子下推、中间状态合并、临时表广播，还有网络IO调度。接下来我们动手实战，在3分片集群上把这些知识点逐个验证。"

在每个节点上创建本地表：

```sql
CREATE TABLE events_local (
```

### Step 1：分布式查询计划生成

这是整个分布式查询的决策核心。ClickHouse在 `StorageDistributed::read()` 被调用时，会分析查询的AST（抽象语法树），决定采用哪种查询处理阶段。简化后的核心逻辑如下：

```cpp
// src/Storages/StorageDistributed.cpp (simplified)

QueryProcessingStage::Enum StorageDistributed::getQueryProcessingStage(
    const ContextPtr & context,
    const ASTPtr & query,
    const ClusterPtr & cluster) const
{
    const auto & select = query->as<ASTSelectQuery &>();

    // 情况1: 带有GROUP BY但没有LIMIT BY
    // → 使用WithMergeableState，分片做部分聚合
    if (select.groupBy() && !select.limitBy()) {
        return QueryProcessingStage::WithMergeableState;
    }
    // 情况2: 带有LIMIT但明确需要完整结果
    // → 使用Complete，分片执行完整查询
    else if (select.limitLength()) {
        return QueryProcessingStage::Complete;
    }
    // 情况3: 简单SELECT或仅带WHERE
    // → 使用FetchColumns，分片仅返回原始列
    else {
        return QueryProcessingStage::FetchColumns;
    }
}
```

验证计划拆分效果：

```sql
-- 简单查询 → FetchColumns
EXPLAIN PLAN
SELECT count(DISTINCT user_id) FROM events_dist WHERE event_date = '2025-01-15';
-- 输出: ReadFromRemote → Expression → Aggregating

-- 带GROUP BY → WithMergeableState
EXPLAIN PLAN
SELECT event_type, count(), sum(amount)
FROM events_dist
WHERE event_date >= '2025-01-01'
GROUP BY event_type;
-- 输出: ReadFromRemote(MergingAggregated) → MergingAggregated
```

第二个查询的 `EXPLAIN PLAN json=1` 会展示更详细的计划树：

```json
{
  "Plan": {
    "Node Type": "MergingAggregated",
    "Header": ["event_type String", "count() UInt64", "sum(amount) Float64"],
    "Plans": [{
      "Node Type": "Union",
      "Plans": [
        {"Node Type": "Remote", "Description": "shard 1: WithMergeableState"},
        {"Node Type": "Remote", "Description": "shard 2: WithMergeableState"},
        {"Node Type": "Remote", "Description": "shard 3: WithMergeableState"}
      ]
    }]
  }
}
```

可以看到三层结构：`MergingAggregated`（发起节点合并聚合）→ `Union`（并行汇合）→ `Remote`（各分片远程查询）。这正是两阶段聚合在查询计划中的体现。

### Step 2：扇出查询到各分片

当查询计划确定后，`ReadFromRemote` 这个QueryPlan节点负责将查询实际发送到各分片。核心实现位于 `src/Processors/QueryPlan/ReadFromRemote.cpp`：

```cpp
// src/Processors/QueryPlan/ReadFromRemote.cpp (simplified)

class ReadFromRemoteStep : public QueryStep {
public:
    void initializePipeline(QueryPipelineBuilder & pipeline,
                            const BuildQueryPipelineSettings &) override
    {
        // 1. 获取集群拓扑信息
        auto clusters = context->getClusters().getCluster(
            modify_query->cluster_name);
        const auto & shards_info = clusters->getShardsInfo();
        const auto & shards = clusters->getShards();

        // 2. 为每个分片创建远程查询执行器
        Pipes remote_pipes;
        for (size_t i = 0; i < shards_info.size(); ++i) {
            // 选择副本：根据load_balancing策略
            auto pool = shards[i].getPool();
            auto connection = pool->get(
                context->getSettingsRef().load_balancing);

            // 3. 创建RemoteQueryExecutor
            // 它会将改写后的查询发往对应分片
            auto executor = std::make_shared<RemoteQueryExecutor>(
                connection,
                modified_query,     // 改写后的SQL
                block_io,           // 输入/输出格式
                context,
                /*throttler*/ nullptr,
                scalars,            // 全局标量值
                external_tables,    // GLOBAL子句产生的临时表
                query_processing_stage  // ← 关键：指定分片执行阶段
            );

            // 4. 将每个executor包装为Pipe
            auto source = std::make_shared<RemoteSource>(
                executor, /*add_aggregation_info*/ true,
                /*async_read*/ true, /*async_query_sending*/ false);
            remote_pipes.emplace_back(std::move(source));
        }

        // 5. 所有分片的输出通过Union并行合并
        Pipe union_pipe = Pipe::unitePipes(std::move(remote_pipes));

        // 6. 如果query_processing_stage为WithMergeableState，
        //    后续pipeline会追加MergingAggregatedTransform
        pipeline.init(std::move(union_pipe));
    }
};
```

关键点解读：

- **`RemoteQueryExecutor`** 是一个异步执行的查询代理。它通过网络连接将改写后的查询发送到目标分片，然后以流式方式（streaming）逐步拉取结果Block。不会等所有结果就绪才返回，而是边读边传。

- **`Pipe::unitePipes()`** 实现了从多个分片并发读取。默认是round-robin轮询，也可以配置为基于分片优先级的策略。

- **异步发送（`async_query_sending`）** 是ClickHouse 22.x+引入的优化，允许查询在多个分片上同时开始执行，避免了串行发送带来的前几个分片先跑完、最后一个分片还没收到查询的问题。

### Step 3：两阶段聚合深度解析

来看一个具体的两阶段聚合查询在分片和发起节点上分别执行的SQL：

```sql
-- 用户写的原始查询（对分布式表）
SELECT event_type, count(), sum(amount), avg(amount)
FROM events_dist
WHERE event_date >= '2025-01-01'
GROUP BY event_type;
```

ClickHouse会在内部分解为两个阶段：

**阶段一（每个分片上并行执行）：**

```sql
-- 发往各分片的改写SQL（伪代码表示）
SELECT
    event_type,
    countState()              AS _cnt_state,    -- 聚合中间状态
    sumState(amount)          AS _sum_state,
    -- avg = sum / count，所以传sumState和countState
    countState(amount)        AS _cnt_amt_state
FROM events_local
WHERE event_date >= '2025-01-01'
  AND (_shard_num IS NULL OR _shard_num = N)  -- 分片过滤（如有）
GROUP BY event_type;
```

每个分片完成部分聚合后，不返回最终数值，而是返回**聚合函数的序列化中间状态**（serialized aggregation state）。以 `countState()` 为例，它返回的是一个 `AggregateFunction(count, UInt64)` 类型的二进制blob，约8字节。

**阶段二（发起节点串行执行）：**

```sql
-- 发起节点上的合并查询
SELECT
    event_type,
    countMerge(_cnt_state)    AS `count()`,     -- 合并各分片的count状态
    sumMerge(_sum_state)      AS `sum(amount)`,
    -- avg = sumMerge / countMerge
    sumMerge(_sum_state) / countMerge(_cnt_amt_state)  AS `avg(amount)`
FROM (
    -- 这里是从各分片汇聚来的Block流
    -- 每个Block结构与阶段一的输出一致
)
GROUP BY event_type;
```

这个过程在查询pipeline中体现为一个 `MergingAggregatedTransform`，源码位置 `src/Processors/Transforms/MergingAggregatedTransform.cpp`：

```cpp
// src/Processors/Transforms/MergingAggregatedTransform.cpp (simplified)

void MergingAggregatedTransform::consume(Chunk chunk) {
    // chunk来自某个分片的partial aggregation结果
    auto num_rows = chunk.getNumRows();
    auto & columns = chunk.getColumns();

    for (size_t i = 0; i < num_rows; ++i) {
        // 以GROUP BY列为key，合并各分片的状态
        auto key = getGroupKey(columns, i);

        auto & aggregator = aggregators[key];
        // 如果不是第一次遇到这个key，创建新的聚合状态
        if (!aggregator) {
            aggregator = createAggregateState();
            // 从当前行反序列化聚合中间状态
            aggregator->mergeDeserialize(columns, i);
        } else {
            // 将当前行的聚合状态合并到已有的聚合状态中
            aggregator->merge(columns, i);
        }
    }
}

Block MergingAggregatedTransform::generate() {
    // 对每个最终group，将聚合状态序列化为最终结果
    // 例如: countMerge(countState_merged) → UInt64
    Block result;
    for (auto & [key, aggregator] : aggregators) {
        result.insert(aggregator->finalize());
    }
    return result;
}
```

另外，ClickHouse在 `max_threads > 1` 的情况下，发起节点的合并阶段也可以并行——它会使用 `ParallelAggregatingTransform` 将来自不同分片的数据流哈希分发到多个合并线程，每个线程负责一部分key的合并。

### Step 4：GLOBAL子句深度实现

`GLOBAL IN` 和 `GLOBAL JOIN` 是分布式表查询中最容易被误用的特性。下面来看其完整实现链路：

**场景：查询VIP用户的订单**

```sql
SELECT * FROM orders_dist
WHERE user_id GLOBAL IN (SELECT user_id FROM vip_users);
```

内部执行流程（源码参考 `src/Storages/StorageDistributed.cpp` 的 `rewriteQueryForExternalDatabase()` 和相关方法）：

```cpp
// 简化描述GLOBAL IN的执行流程

// 步骤1: 发起节点本地执行子查询
auto vip_ids = executeSubQuery("SELECT user_id FROM vip_users");
// 得到: [101, 203, 405, 607, ...] 共500个VIP用户ID

// 步骤2: 在发起节点创建ExternalDistributedTable
auto temp_table = std::make_shared<ExternalDistributedTable>(
    *context,
    "_data_8f3a2b1c",  // 随机生成的临时表名
    block_structure    // Block结构: {user_id: UInt64}
);

// 步骤3: 将子查询结果写入内存临时表
temp_table->writeData(vip_ids);  // 数据保留在RAM中

// 步骤4: 对于每个远端分片，执行以下操作：
for (auto & shard : shards) {
    auto conn = shard->getConnection();

    // 4a: 在远端分片上创建临时表
    conn->sendQuery(
        "CREATE TEMPORARY TABLE IF NOT EXISTS _data_8f3a2b1c "
        "(user_id UInt64) ENGINE = Memory"
    );

    // 4b: 将数据分批发送到远端临时表
    connectAndSendExternalData(
        conn,
        temp_table,
        shard_address,
        vip_ids                    // 500个VIP用户ID
    );

    // 4c: 发送改写后的查询
    conn->sendQuery(
        "SELECT * FROM orders_local "
        "WHERE user_id IN _data_8f3a2b1c"
    );
}

// 步骤5: 收集各分片结果并合并（标准流程）

// 步骤6: 清理所有分片上的临时表
for (auto & shard : shards) {
    shard->sendQuery("DROP TEMPORARY TABLE IF EXISTS _data_8f3a2b1c");
}
```

这个流程在源码中由 `ReadFromRemote` 的 `getExternalTables()` 方法负责构建。关键类 `ExternalDistributedTable` 实现了分布式临时表的生命周期管理：


**GLOBAL JOIN的实现**

```sql
SELECT o.*, u.name
FROM orders_dist AS o
GLOBAL LEFT JOIN users AS u ON o.user_id = u.user_id;
```

GLOBAL JOIN的流程稍有不同：发起节点不会将整张 `users` 表广播到所有分片。而是采用以下策略：

1. 发起节点从每个分片获取 `orders_dist` 中实际参与JOIN的 `user_id` 集合（去重后）。
2. 发起节点用这些ID去 `users` 表中查对应的 `name`，得到一个映射表。
3. 发起节点将这个**精简后的映射表**以临时表的方式广播到所有分片。
4. 各分片使用映射表完成本地JOIN。

这种方式避免了传输整张 `users` 表，但如果 `user_id` 集合极其分散（几乎覆盖全表），效果就打折扣了。

### Step 5：避免副本重复计数

在多副本环境中，如果不加控制，Distributed引擎可能查询同一分片的多个副本，导致结果重复。ClickHouse提供了多层防护：

```sql
-- 方式1: 优先查询本地副本（避免跨网络）
SET prefer_localhost_replica = 1;

-- 方式2: 指定每个分片只选一个副本
SET load_balancing = 'first_or_random';

-- 方式3: 配置Distributed表时声明内部复制
-- internal_replication = true 表示副本间数据自动同步，
-- 引擎知道不需要查询所有副本。
CREATE TABLE events_dist AS events_local
ENGINE = Distributed(
    'demo_cluster',
    'default',
    'events_local',
    shard_key
);
-- 默认识别集群配置中的 <internal_replication>true</internal_replication>
```

`load_balancing` 参数的源码逻辑在 `src/Common/ProxyConfigurationResolver.cpp` 中：

```cpp
// 负载均衡策略选择

ConnectionPool::Entry getConnection(
    const std::string & load_balancing,
    const Pools & pools)
{
    if (load_balancing == "random") {
        // 随机从所有可用副本中选一个（默认）
        return pools[rand() % pools.size()]->get();
    }
    else if (load_balancing == "nearest_hostname") {
        // 根据主机名相似度选择最近副本
        return selectNearest(pools);
    }
    else if (load_balancing == "first_or_random") {
        // 初始分片优先选第一个可用副本，之后维持粘性
        return selectFirstOrRandom(pools);
    }
    else if (load_balancing == "in_order") {
        // 严格按配置顺序，第一个不健康才选第二个
        return selectInOrder(pools);
    }
    // ...
}
```

## 4. 项目总结

### 分布式查询处理阶段总览

ClickHouse的分布式查询将一条SQL分解为两个层级的执行：

| 处理阶段 (Processing Stage) | 执行位置 | 是否并行 | 典型用途 |
|---|---|---|---|
| **FetchColumns** | 各分片 | 是 | 简单SELECT，仅读取原始列数据 |
| **WithMergeableState** | 各分片 → 发起节点 | 阶段一并行，阶段二可并行 | GROUP BY聚合查询 |
| **Complete** | 各分片 | 是 | 带LIMIT的排序查询 |
| **Merge** | 发起节点 | 否（可多线程） | 合并各分片的中间结果 |

### 各方案的适用场景

- **FetchColumns**：适用于返回数据量小、无需聚合的查询。如果返回数据量大（如 `SELECT *`），发起节点内存和网络压力巨大，应改用 `GROUP BY` 或配合 `SETTINGS max_bytes_before_external_group_by` 落盘。

- **WithMergeableState**：GROUP BY聚合查询的首选策略。聚合中间状态体积远小于原始数据，大幅减少网络传输。但需注意：同一key在不同分片上的中间状态合并需要发起节点的CPU和内存。

- **GLOBAL IN / GLOBAL JOIN**：当子查询或JOIN的驱动表**不能按分片键co-locate**时使用。代价是子查询结果需广播到所有分片，子查询结果集越大，代价越高。

### 注意事项

1. **发起节点是瓶颈**：大量分片返回数据时，发起节点的合并操作（尤其是排序、去重、JOIN）可能成为单点瓶颈。监控发起节点的CPU和内存是关键。

2. **GLOBAL临时表有开销**：每个 `GLOBAL IN` 子查询都会在每个分片上创建Memory引擎临时表，子查询结果集大会导致分片内存压力。如果子查询结果超过几百万行，考虑重新设计分片策略。

3. **长尾分片拖慢全局**：分布式查询的延迟 = max(所有分片延迟) + 合并延迟。如果某个分片因数据倾斜或负载过高而延迟较长，整个查询的响应时间将被拉长。

4. **max_parallel_replicas**：此参数控制同时从多少个副本获取数据，但不影响分片级别的并行度（分片始终全部并发查询）。

### 常见踩坑经验

1. **在Distributed表上使用 `ORDER BY ... LIMIT N`**：每个分片各自返回前N条，发起节点再做归并取前N条。但如果你要全局前N条，每个分片只给N条是不够的——可能某个分片里有更好的数据排在N条之外但理应进入全局TOP N。解决方案：要么加大每个分片的LIMIT（比如 `LIMIT N * shard_count`），要么在分片键上做全局排序。

2. **GLOBAL IN数据量过大导致临时表OOM**：500万的IN子查询结果集不适合做GLOBAL IN。应改用Hash分片让数据co-locate，或者在应用层做两阶段查询。

3. **忘记设置 `prefer_localhost_replica=1`**：发起节点本该读本地副本，结果却通过网络去读其他机器的副本，徒增网络开销。在 `users.xml` 中写入profile默认开启此设置。

4. **混用Distributed表和本地表**：`INSERT INTO events_dist SELECT * FROM events_dist` 这种操作会导致数据在分片间反复跳转，产生指数级的数据膨胀。

### 思考题

1. **如果3个分片中有一个分片特别慢（数据倾斜），分布式查询的整体性能会如何？**

   *提示*：分布式查询的结束时间 = max(各分片完成时间) + 合并时间。最慢的分片决定了查询的延迟下限。此外，如果使用 `WithMergeableState` 策略，发起节点必须等待所有分片返回部分聚合结果后才能开始合并。属于典型的"短板效应"。

2. **GLOBAL JOIN和普通JOIN在语义上有什么区别？为什么Distributed表必须用GLOBAL JOIN？**

   *提示*：普通JOIN在分布式表上执行时，每个分片独立执行JOIN，但JOIN的右表（`users`）在每个分片上只有部分数据（取决于该表是否也分片）。这意味着分片A上的 `orders` 行可能找不到它在分片B上的对应 `users` 行。GLOBAL JOIN通过将右表（或右表的子集）广播到所有分片，保证每个分片都能看到完整的右表数据，从而语义正确。本质是分布式场景下的数据co-location问题。
