## 1. 项目背景

某短视频App坐拥**5000万日活用户**，每天产生超**200亿条**用户行为事件——视频播放、点赞、分享、评论、广告曝光。数据团队肩负四大核心需求：

1. **实时大盘**：运营大屏需要展示全站QPS、在线用户数、热门视频排行，端到端延迟必须在**5秒以内**，大促峰值期间尤其不能断流。
2. **用户旅程分析**：广告投放团队想知道"用户看了哪些视频后最终下单"，需要跨事件类型的多步路径关联查询——从播放到点赞，从点赞到分享，从分享到购买。这类即席查询（Ad-hoc Query）无法预定义，必须支持灵活SQL。
3. **A/B实验实时指标**：推荐算法团队每天跑200+个A/B实验，每个实验需要分钟级的曝光数、转化率、置信区间。传统T+1的离线报表已经无法满足快速迭代的诉求。
4. **广告效果监控**：商业化团队需要按广告位、素材、人群包实时统计CPM、点击率、转化率。数据延迟每多1分钟，无效投放就多烧一笔预算。

现状堪忧——当前架构是 **Flink + HBase**。Flink负责实时流处理，将聚合结果写入HBase；HBase提供KV查询。这个方案表面上看起来很"实时"，但实际上存在三个致命缺陷：

- **即席查询不可行**：每新增一个分析维度，数据团队需要花**2小时以上**编写新的Flink Job——定义Source、Window、Aggregation、Sink，然后打包、上传、重启。运营同学提一个"按城市+内容分类看过去30分钟的分享率"，数据分析师要排期到下周二。
- **HBase的OLAP能力几乎为零**：HBase是KV存储，`GROUP BY`、`JOIN`、多维度下钻全都不支持。数据团队被迫维护一套复杂的Lambda架构——实时数据走Flink+HBase，离线数据走Spark+Hive，两套口径对不齐的鬼故事每周都在上演。
- **存储成本失控**：HBase的RowKey设计为服务写入吞吐做了大量冗余，加上HDFS三副本，每TB有效数据的存储成本高达$800/月。200亿事件/天 × 7天保留周期 = 1400亿行，存储账单让CTO皱眉头。

CTO拍板：**用ClickHouse替换HBase做实时分析引擎，Flink只管数据清洗和入湖，所有聚合和查询由ClickHouse承担。目标：即席查询从2小时降到2分钟，存储成本砍半，P99聚合查询< 500ms。**

这就是本章要交付的系统——从埋点上报到实时大屏，全链路贯通。

## 2. 项目设计：剧本式交锋对话

周三下午，大师把小白和小胖叫到作战室。白板上已经画好了一条流水线：**Kafka → ClickHouse → Grafana → S3**。

**小胖**（手里拿着一杯奶茶，扫了一眼白板）："大师，Flink不是挺好？实时流处理不就应该用Flink吗？我之前给运营做那个实时UV看板，Flink一个滑动窗口15分钟就写完了。ClickHouse不是做离线分析的吗，你拿来搞实时靠谱吗？"

**大师**（把奶茶从小胖手里拿过来放到一边）："小胖，我问你——上次运营提的那个'按城市+内容分类看分享率'需求，你用Flink写了多久？"

**小胖**（挠头）："那需求来来回回改了三次……第一次要按城市，第二次要加视频时长区间，第三次又要把城市改成省份。每次改完重跑Job都得从头消费Kafka，三天加起来至少花了8个小时。"

**大师**："这就是问题所在。Flink是**预计算模型**——你得提前定义好'要算什么、按什么维度算、用什么样的窗口'。定义错了就得改代码、重启Job。但运营和分析师的需求是**探索性的**——他们自己都说不清楚下一秒要查什么。ClickHouse是**即席计算模型**——数据先进来，查询时再聚合。你新增一个维度就是写一行SQL，30秒上线。Flink好比工厂流水线，开动前得把模具装好；ClickHouse好比乐高积木，数据就是砖块，怎么拼是查询时决定的。"

**技术映射 #1**：预计算（Flink）vs 即席计算（ClickHouse）的本质区别在于**计算发生的时机**。Flink在写入时做聚合（Write-time Aggregation），优势是查询快但灵活性低；ClickHouse在查询时做聚合（Query-time Aggregation），优势是灵活但需要更强悍的查询引擎。两者的结合点——**Flink负责ETL清洗+数据标准化入湖，ClickHouse负责灵活查询和预聚合加速**——才是现代实时数据栈的最佳实践。

---

**小白**（眼镜反着白板的光，笔在本子上飞快记着）："大师，这个架构我有三个问题。第一，Kafka是at-least-once语义，如果Flink或ClickHouse消费时挂了重启，数据会不会重复？第二，我们的埋点SDK经常加新字段——比如上个月加了`sdk_version`，这个月又要加`network_type`——ClickHouse怎么应对这种Schema变更？第三，有些用户的设备在离线环境下刷视频，数据迟到一两天才上报，这种延迟数据怎么处理？"

**大师**（赞许地点头）："三个问题都是生产环境的硬骨头。一个个来。"

"**第一：数据重复**。Kafka的at-least-once加上消费者重启，确实会写入重复数据。解决思路分两层：在MergeTree层，我们用`ReplacingMergeTree`做幂等去重——以`event_id`做主键去重依据，`ENGINE = ReplacingMergeTree(ver)`配合一个递增的版本号，后台Merge时自动清理旧版本。在物化视图层，聚合物化视图本身对重复写入有一定的容忍度——多写入一条数据，`count()`多1，`sum()`多加一个值——但总量级上误差可控。真正需要精确去重的场景（比如广告计费），在高版本ClickHouse中可以使用`Refreshable Materialized View`做定时修正。"

"**第二：Schema变更**。ClickHouse支持`ALTER TABLE ... ADD COLUMN`，而且是**即时生效的元数据操作**——不重写旧数据，新写入的行自动填上新字段。旧数据查询时新字段返回默认值。这就是列存的天然优势之一：加列几乎零成本。不过删除列就麻烦一些，得`ALTER TABLE ... DROP COLUMN`，底层是Mutation——异步重写Part。所以建议：**宁可加列不删列，用NULL或空字符串标记废弃字段**。"

"**第三：延迟到达**。这是个经典问题。对于迟到1-2天的数据，我们有两种策略：如果延迟数据比例小（< 1%），直接在原表写入，`event_time`用原始时间戳而非入库时间。物化视图是按`event_time`聚合的，所以延迟数据会自动归到正确的分钟/小时桶里——只是这个分钟的指标会在数据到达时有一次微小跳变。如果延迟比例大（比如离线批量回补），建议用`ALTER TABLE ... MODIFY QUERY`先暂停物化视图对该分区的写入，补完数据再手动触发一次OPTIMIZE聚合。"

**技术映射 #2**：实时数据管道的三座大山——重复（幂等性）、变更（Schema Evolution）、延迟（Late Arrival）——在ClickHouse中的解决方案分别是：ReplacingMergeTree（去重）+ ADD COLUMN元数据操作（加列）+ event_time保留原始时间戳（晚到数据归位）。三者都不完美，但在工程实践中够用且可控。

**大师**放下马克笔："这套架构核心有四层。**第一层：Kafka数据总线**——所有事件先入Kafka，解耦生产者和消费者。Kafka分区数要和ClickHouse的Consumer数对齐——12个分区 × 4个Consumer = 每Consumer负责3个分区。**第二层：ClickHouse实时计算**——Kafka Engine消费后写入MergeTree原始表，物化视图自动触发级联聚合。**第三层：Grafana+Prometheus**——从ClickHouse查聚合数据出图，同时通过内置的9363端口/metrics暴露集群健康指标。**第四层：S3冷归档**——超过90天的数据通过TTL自动迁移到S3，节省SSD成本。整个链路遵循KISS原则——Keep It Simple, Stupid——不强依赖Flink做预聚合，把所有灵活性留给ClickHouse。"

### 分步实现

#### Step 1：数据建模——三张核心表

启动环境后，进入ClickHouse Node1创建核心表结构：

```shell
docker exec -it ch-behavior-1 clickhouse-client
```

```sql
-- ═══════════════════════════════════════════
-- 数据库：行为分析系统
-- ═══════════════════════════════════════════
CREATE DATABASE IF NOT EXISTS behavior;

-- ── 表1：Kafka消费表（临时表，不存数据）──
CREATE TABLE behavior.events_kafka (
    event_id       String,
    event_type     LowCardinality(String),
    user_id        UInt32,
    video_id       UInt32,
    author_id      UInt32,
    session_id     String,
    properties     Map(String, String),
    event_time     DateTime,
    receive_time   DateTime DEFAULT now()
) ENGINE = Kafka()
SETTINGS
    kafka_broker_list = 'kafka:9092',
    kafka_topic_list = 'user_events',
    kafka_group_name = 'ch_analytics_consumer',
    kafka_format = 'JSONEachRow',
    kafka_num_consumers = 4;       -- 4个Consumer消费12个分区

-- ── 表2：MergeTree原始事件表（真相来源）──
-- 核心设计：按天分区 + (event_date, event_type, user_id)排序 + 90天TTL
CREATE TABLE behavior.events (
    event_id       String,
    event_type     LowCardinality(String),
    user_id        UInt32,
    video_id       UInt32,
    author_id      UInt32,
    session_id     String,
    properties     Map(String, String),
    event_time     DateTime,
    receive_time   DateTime,
    event_date     Date DEFAULT toDate(event_time)
) ENGINE = ReplicatedMergeTree('/clickhouse/tables/{shard}/behavior/events', '{replica}')
ORDER BY (event_date, event_type, user_id)
PARTITION BY toYYYYMM(event_date)
TTL event_date + INTERVAL 90 DAY DELETE
SETTINGS index_granularity = 8192;

-- 为常用查询字段添加跳数索引
ALTER TABLE behavior.events 
    ADD INDEX idx_video_id video_id TYPE bloom_filter() GRANULARITY 1;
ALTER TABLE behavior.events 
    ADD INDEX idx_session_id session_id TYPE bloom_filter() GRANULARITY 4;
ALTER TABLE behavior.events 
    ADD INDEX idx_user_id user_id TYPE minmax GRANULARITY 4;

-- ── 物化视图：Kafka → MergeTree（核心消费链路）──
CREATE MATERIALIZED VIEW behavior.events_mv TO behavior.events
AS SELECT * FROM behavior.events_kafka;
```

**设计要点解读**：

| 设计决策 | 原因 | 知识来源 |
|---------|------|---------|
| `ORDER BY (event_date, event_type, user_id)` | event_date首位确保时间范围裁剪，event_type次位支撑事件聚合，user_id末位支撑用户旅程查询 | 第5章 分区与排序键 |
| `PARTITION BY toYYYYMM(event_date)` | 按月分区，匹配90天TTL——每个月一个分区，删除时整个分区DROP，零碎片IO | 第5章 分区裁剪、第10章 TTL |
| `LowCardinality(String)` | event_type取值<20个（play/like/share/comment/ad_view/purchase等），字典编码压缩到十几KB | 第3章 数据类型 |
| `Map(String, String)` | 动态属性（如watch_duration、experiment_id、variant）无需固定Schema，灵活应对字段变更 | 第3章 复合类型 |
| `ReplicatedMergeTree` | 跨节点数据冗余，单节点故障时查询自动路由到副本 | 第19章 复制表 |
| `bloom_filter`跳数索引 | video_id和session_id在高基数场景下加速等值查找（如查某个视频的播放量、某个session的完整链路） | 第21章 高级索引 |

> **踩坑提醒**：`Map`类型的性能不如固定列。如果某个属性在99%的事件中都存在（比如`watch_duration`总是伴随`play`事件），应提出来作为独立列。`Map`适合长尾属性——只在部分事件中出现且未来会动态增减的字段。

#### Step 2：级联物化视图——秒→分钟→小时→A/B实验

这是整个系统的性能核心。我们设计四级级联聚合：

```
events (原始表, 200亿/天)
  ├── Level 0: events_mv (Kafka消费视图，直接写入events)
  ├── Level 1: events_minute_agg (分钟粒度聚合)
  │     ├── 按 minute + event_type + video_id 聚合
  │     └── 支撑实时大盘（最近5分钟QPS、热门视频）
  ├── Level 2: events_hour_agg (小时粒度聚合)
  │     └── 支撑日报/周报（趋势分析、流量对比）
  └── Level 3: events_ab_test (A/B实验专属聚合)
        └── 按 minute + experiment_id + variant 聚合
```

```sql
-- ═══════════════════════════════════════════
-- Level 1：分钟级聚合物化视图
-- 用途：实时大盘（最近几分钟的QPS和指标）
-- ═══════════════════════════════════════════
CREATE MATERIALIZED VIEW behavior.events_minute_agg
ENGINE = SummingMergeTree()
ORDER BY (minute, event_type, video_id)
PARTITION BY toYYYYMM(minute)
AS SELECT
    toStartOfMinute(event_time)     AS minute,
    event_type,
    video_id,
    count()                         AS event_count,
    uniqState(user_id)              AS unique_users_state,    -- 中间态去重
    countIf(properties['action'] = 'share')     AS share_count,
    countIf(properties['action'] = 'like')      AS like_count,
    countIf(properties['action'] = 'comment')   AS comment_count,
    sumOrNull(toFloat32OrNull(properties['watch_duration'])) 
                                     AS total_watch_seconds
FROM behavior.events
GROUP BY minute, event_type, video_id;

-- ═══════════════════════════════════════════
-- Level 2：小时级聚合物化视图（级联自分钟表）
-- 用途：运营日报、周报、趋势分析
-- ═══════════════════════════════════════════
CREATE MATERIALIZED VIEW behavior.events_hour_agg
ENGINE = SummingMergeTree()
ORDER BY (hour, event_type, video_id)
PARTITION BY toYYYYMM(hour)
AS SELECT
    toStartOfHour(minute)           AS hour,
    event_type,
    video_id,
    sum(event_count)                AS event_count,
    uniqMergeState(unique_users_state) AS unique_users_state,
    sum(share_count)                AS share_count,
    sum(like_count)                 AS like_count,
    sum(comment_count)              AS comment_count,
    sum(total_watch_seconds)        AS total_watch_seconds
FROM behavior.events_minute_agg
GROUP BY hour, event_type, video_id;

-- ═══════════════════════════════════════════
-- Level 3：A/B实验专属物化视图
-- 用途：分钟级实验指标（曝光、转化、置信区间）
-- ═══════════════════════════════════════════
CREATE MATERIALIZED VIEW behavior.events_ab_test
ENGINE = SummingMergeTree()
ORDER BY (minute, experiment_id, variant)
PARTITION BY toYYYYMM(minute)
AS SELECT
    toStartOfMinute(event_time)     AS minute,
    properties['experiment_id']     AS experiment_id,
    properties['variant']           AS variant,
    count()                         AS event_count,
    uniqState(user_id)              AS unique_users,
    countIf(event_type = 'purchase') AS conversions,
    sumOrNull(toFloat32OrNull(properties['order_amount'])) AS total_revenue
FROM behavior.events
WHERE properties['experiment_id'] != ''
GROUP BY minute, experiment_id, variant;
```

**关键技术细节**：

- **`uniqState` / `uniqMerge` 组合**：分钟级使用`uniqState`生成去重中间状态（HyperLogLog sketch），小时级用`uniqMergeState`合并这些中间状态。这比小时级直接用`uniq(user_id)`重新去重更准确——因为同一个user_id可能跨越多分钟，直接`uniqMerge`能正确去重。
- **`sumOrNull` 处理 Map 中的异常值**：`properties['watch_duration']`可能不存在（不是所有事件都有观看时长），直接`toFloat32OrNull`转成NULL，`sumOrNull`返回NULL而非0——避免把缺失值误当成0参与统计。
- **WHERE过滤在物化视图中是免费的**：A/B实验视图只处理有`experiment_id`的事件——这个过滤发生在源表扫描阶段，不消耗额外资源。

#### Step 3：核心分析查询SQL集合

这些SQL就是Grafana大屏、运营报表和算法实验的底层查询：

```sql
-- ═══════════════════════════════════════════
-- 【查询1】实时大盘——过去5分钟各事件类型QPS
-- 用途：Grafana时序折线图，运营大屏左上角
-- ═══════════════════════════════════════════
SELECT
    event_type,
    sum(event_count) / 300               AS qps,
    uniqMerge(unique_users_state)        AS unique_users
FROM behavior.events_minute_agg
WHERE minute >= now() - INTERVAL 5 MINUTE
GROUP BY event_type
ORDER BY qps DESC;

-- ═══════════════════════════════════════════
-- 【查询2】漏斗分析——视频播放→点赞→分享转化
-- 用途：内容运营评估视频质量
-- ═══════════════════════════════════════════
WITH
    plays AS (
        SELECT video_id, uniq(user_id) AS users
        FROM behavior.events
        WHERE event_type = 'play' AND event_time >= today()
        GROUP BY video_id
    ),
    likes AS (
        SELECT video_id, uniq(user_id) AS users
        FROM behavior.events
        WHERE event_type = 'like' AND event_time >= today()
        GROUP BY video_id
    ),
    shares AS (
        SELECT video_id, uniq(user_id) AS users
        FROM behavior.events
        WHERE event_type = 'share' AND event_time >= today()
        GROUP BY video_id
    )
SELECT
    p.video_id,
    dictGetOrDefault('behavior.videos_dict', 'category', p.video_id, 'Unknown') AS category,
    p.users                       AS plays,
    COALESCE(l.users, 0)          AS likes,
    COALESCE(s.users, 0)          AS shares,
    round(likes * 100.0 / plays, 2)  AS like_rate,
    round(shares * 100.0 / plays, 2) AS share_rate
FROM plays p
LEFT JOIN likes l ON p.video_id = l.video_id
LEFT JOIN shares s ON p.video_id = s.video_id
ORDER BY plays DESC
LIMIT 20;

-- ═══════════════════════════════════════════
-- 【查询3】Top 10热门视频（过去1小时）
-- 用途：实时内容推荐质量反馈
-- ═══════════════════════════════════════════
SELECT
    e.video_id,
    dictGetOrDefault('behavior.videos_dict', 'category', e.video_id, 'Unknown') AS category,
    dictGetOrDefault('behavior.videos_dict', 'duration', e.video_id, 0)    AS duration,
    count()                            AS views,
    sum(toFloat32OrNull(e.properties['watch_duration']))::Float64 AS total_watch_sec,
    round(total_watch_sec / greatest(views, 1), 1)               AS avg_watch_sec,
    uniq(e.user_id)                   AS unique_viewers
FROM behavior.events e
WHERE e.event_type = 'play'
  AND e.event_time >= now() - INTERVAL 1 HOUR
GROUP BY e.video_id
ORDER BY views DESC
LIMIT 10;

-- ═══════════════════════════════════════════
-- 【查询4】用户旅程——指定用户过去7天的行为序列
-- 用途：分析"看了什么→做了什么→买了什么"
-- ═══════════════════════════════════════════
SELECT
    event_time,
    event_type,
    video_id,
    dictGetOrDefault('behavior.videos_dict', 'category', video_id, 'Unknown') AS category,
    properties
FROM behavior.events
WHERE user_id = 12345
  AND event_date >= today() - 7
ORDER BY event_time ASC
LIMIT 100;

-- ═══════════════════════════════════════════
-- 【查询5】A/B实验实时指标
-- 用途：算法团队评估实验效果
-- ═══════════════════════════════════════════
SELECT
    experiment_id,
    variant,
    uniqMerge(unique_users)             AS users,
    sum(event_count)                    AS events,
    sum(conversions)                    AS conversions,
    round(sum(conversions) * 100.0 / greatest(sum(event_count), 1), 4) 
                                         AS conversion_rate,
    round(sum(total_revenue), 2)        AS total_revenue
FROM behavior.events_ab_test
WHERE minute >= now() - INTERVAL 1 DAY
GROUP BY experiment_id, variant
ORDER BY experiment_id, variant;

-- ═══════════════════════════════════════════
-- 【查询6】广告效果监控（最近1小时）
-- 用途：商业化团队评估广告ROI
-- ═══════════════════════════════════════════
SELECT
    properties['ad_id']                 AS ad_id,
    properties['ad_position']           AS position,
    countIf(event_type = 'ad_impression') AS impressions,
    countIf(event_type = 'ad_click')    AS clicks,
    round(clicks * 1000.0 / greatest(impressions, 1), 2) AS cpm,
    round(clicks * 100.0 / greatest(impressions, 1), 2)  AS ctr
FROM behavior.events
WHERE event_type IN ('ad_impression', 'ad_click')
  AND event_time >= now() - INTERVAL 1 HOUR
GROUP BY ad_id, position
ORDER BY impressions DESC
LIMIT 20;
```

#### Step 5：数据生成器（Python）

这个生成器模拟短视频App的用户行为事件，写入Kafka供ClickHouse消费：

```python
# data-generator/generate.py
"""
用户行为事件模拟生成器
生成 play/like/share/comment/ad_impression/ad_click/purchase 事件
写入Kafka → ClickHouse Kafka Engine消费 → MergeTree + 物化视图
"""
import json
import random
import time
import uuid
from datetime import datetime

from kafka import KafkaProducer

# ── 配置 ──
KAFKA_BOOTSTRAP = "kafka:9092"
TOPIC = "user_events"
EVENT_TYPES = ['play', 'like', 'share', 'comment', 'ad_impression', 'ad_click', 'purchase']
EVENT_WEIGHTS = [0.55, 0.15, 0.05, 0.05, 0.10, 0.03, 0.07]

VIDEO_POOL = list(range(1001, 1011))   # 10个模拟视频
USER_POOL = list(range(1, 100001))      # 10万模拟用户
ACTION_LIST = ['share', 'like', 'comment', 'skip', 'collect']

# ── A/B实验配置 ──
EXPERIMENTS = ['recommend_v5', 'recommend_v6', 'ui_redesign', 'ad_layout']
VARIANTS = ['control', 'treatment_a', 'treatment_b']

producer = KafkaProducer(
    bootstrap_servers=KAFKA_BOOTSTRAP,
    value_serializer=lambda v: json.dumps(v, ensure_ascii=False).encode('utf-8'),
    batch_size=32768,
    linger_ms=10,
    compression_type='lz4'
)

def generate_event():
    """生成一条模拟用户行为事件"""
    event_type = random.choices(EVENT_TYPES, weights=EVENT_WEIGHTS, k=1)[0]
    user_id = random.choice(USER_POOL)
    video_id = random.choice(VIDEO_POOL)
    author_id = video_id - 1000  # 简化映射

    # 按事件类型差异化填充属性
    props = {}
    if event_type == 'play':
        props['watch_duration'] = str(round(random.expovariate(1 / 30), 1))
        props['action'] = random.choice(ACTION_LIST)
    elif event_type == 'like':
        props['action'] = 'like'
        props['like_source'] = random.choice(['feed', 'search', 'recommend'])
    elif event_type == 'share':
        props['action'] = 'share'
        props['share_platform'] = random.choice(['wechat', 'weibo', 'qq', 'copy_link'])
    elif event_type == 'comment':
        props['action'] = 'comment'
        props['comment_length'] = str(random.randint(1, 500))
    elif event_type == 'ad_impression':
        props['ad_id'] = str(random.randint(5001, 5020))
        props['ad_position'] = random.choice(['feed_1', 'feed_3', 'feed_5', 'video_end'])
    elif event_type == 'ad_click':
        props['ad_id'] = str(random.randint(5001, 5020))
        props['ad_position'] = random.choice(['feed_1', 'feed_3', 'video_end'])
    elif event_type == 'purchase':
        props['order_amount'] = str(round(random.uniform(9.9, 299.9), 2))
        props['product_type'] = random.choice(['course', 'membership', 'physical', 'virtual'])

    # 10%的事件属于A/B实验
    if random.random() < 0.10:
        experiment = random.choice(EXPERIMENTS)
        props['experiment_id'] = experiment
        props['variant'] = random.choice(VARIANTS)

    return {
        "event_id": str(uuid.uuid4()),
        "event_type": event_type,
        "user_id": user_id,
        "video_id": video_id,
        "author_id": author_id,
        "session_id": f"{user_id}-{random.randint(1000, 9999)}-{int(time.time())}",
        "properties": props,
        "event_time": datetime.now().strftime('%Y-%m-%d %H:%M:%S'),
        "receive_time": datetime.now().strftime('%Y-%m-%d %H:%M:%S')
    }

def main():
    print("Waiting for Kafka to be ready...")
    time.sleep(15)

    total = 0
    batch = []
    BATCH_SIZE = 1000

    print("Starting event generation (target: ~10K events/sec)...")
    while True:
        batch.append(generate_event())

        if len(batch) >= BATCH_SIZE:
            for event in batch:
                producer.send(TOPIC, value=event)
            producer.flush()
            total += BATCH_SIZE
            if total % 50000 == 0:
                print(f"[{datetime.now().strftime('%H:%M:%S')}] Generated {total:,} events")
            batch = []

        # 控制生成速率 ~10K events/sec
        time.sleep(0.0001)

if __name__ == "__main__":
    main()
```

```yaml
# 生产环境架构（非Docker，物理机/虚拟机部署）：
# 
# Kafka Cluster (3 brokers, 12 partitions/topic)
#   ├── Broker 1: 10.0.1.1:9092
#   ├── Broker 2: 10.0.1.2:9092
#   └── Broker 3: 10.0.1.3:9092

#   ├── Shard 1: ch01 (10.0.2.1, primary) + ch02 (10.0.2.2, replica)
#   ├── Shard 2: ch03 (10.0.2.3, primary) + ch04 (10.0.2.4, replica)
#   └── Shard 3: ch05 (10.0.2.5, primary) + ch06 (10.0.2.6, replica)
#         ↓
#  每个节点: 64 cores / 256GB RAM / 4 × 4TB NVMe SSD (RAID 0)
#        ↓
# Grafana + Prometheus (10.0.3.x)
#   ├── Grafana (dashboard rendering)
#   ├── Prometheus (metrics collection from CH 9363 + node_exporter)
#   └── Alertmanager (告警路由)
#        ↓
# S3/MinIO (10.0.4.x)
#   └── 90天以上冷数据归档
#
# 关键网络要求：
#   - CH节点间: 25Gbps (跨分片数据传输)
#   - Kafka→CH: 10Gbps (实时消费)
#   - CH→Grafana: 1Gbps (查询结果传输)
```

**Kafka与ClickHouse分区对齐策略**：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| Kafka Topic分区数 | 12 | 3倍于CH Shard数，保证数据均匀分布 |
| `kafka_num_consumers` | 4 | 每个节点4个Consumer线程 |
| 每Consumer负责分区数 | 3 | 12分区 ÷ 4 Consumer = 3分区/Consumer |
| 分布表分片键 | `rand()` | 随机分片，避免热点 |

**验收指标验证SQL**：

```sql
-- 1. 写入吞吐检查（每秒事件数）
SELECT
    toStartOfHour(receive_time) AS hour,
    count() / 3600              AS events_per_second,
    formatReadableSize(sum(length(properties))) AS data_size_per_hour
FROM behavior.events
WHERE event_date = today()
GROUP BY hour
ORDER BY hour;

-- 2. P50/P95/P99查询延迟（从query_log）
SELECT
    quantile(0.50)(query_duration_ms) AS p50_ms,
    quantile(0.95)(query_duration_ms) AS p95_ms,
    quantile(0.99)(query_duration_ms) AS p99_ms,
    count()                           AS total_queries
FROM system.query_log
WHERE type = 'QueryFinish'
  AND query_start_time >= now() - INTERVAL 1 HOUR
  AND query NOT LIKE '%system.query_log%';

-- 3. 数据新鲜度（Kafka消费延迟）
SELECT
    dateDiff('second', max(event_time), max(receive_time)) AS max_lag_seconds,
    count() / dateDiff('second', min(receive_time), max(receive_time)) AS avg_throughput_rps
FROM behavior.events
WHERE event_date = today();

-- 4. 各表压缩率
SELECT
    table,
    formatReadableSize(sum(bytes_on_disk))    AS compressed,
    formatReadableSize(sum(data_uncompressed_bytes)) AS uncompressed,
    round(sum(data_uncompressed_bytes) / greatest(sum(bytes_on_disk), 1), 1) AS ratio,
    sum(rows)                                 AS rows,
    count()                                   AS parts
FROM system.parts
WHERE database = 'behavior' AND active
GROUP BY table
ORDER BY sum(bytes_on_disk) DESC;

-- 5. 副本健康检查
SELECT
    table,
    is_leader,
    absolute_delay,                           -- 副本复制延迟（秒）
    queue_size                                -- 待复制队列积压
FROM system.replicas
WHERE database = 'behavior'
ORDER BY absolute_delay DESC;
```

## 4. 项目总结

### 架构对比：Flink+HBase vs ClickHouse 实时分析

| 维度 | Flink + HBase | ClickHouse | 差距 |
|------|--------------|------------|------|
| 数据入库延迟 | < 100ms（流处理） | < 1s（Kafka消费+写入） | Flink更实时 |
| 查询灵活性 | 有限（预定义Job） | 完整SQL即席查询 | ClickHouse胜 |
| 新增指标耗时 | 2小时+（编写Flink Job，打包上线） | 2分钟（编写SQL） | **60x** |
| 存储成本 | 高（HBase RowKey膨胀 + HDFS三副本） | 低（列存 + LZ4/ZSTD压缩5-10x） | **5-10x** |
| 运维复杂度 | 高（Flink Checkpoint + HBase Compaction + HDFS均衡） | 中（单进程，无JVM，Merge自动管理） | ClickHouse更简单 |
| 即席聚合性能（10亿行） | 不支持（HBase无GROUP BY） | < 500ms（物化视图+向量化+列存） | — |
| 高可用 | 依赖HBase RegionServer故障转移 | 副本自动故障转移，ZooKeeper/Keeper选主 | 相当 |
| 学习曲线 | 高（Java + Flink算子 + HBase设计） | 低（SQL，DBA友好） | ClickHouse更友好 |

### 适用场景

✅ **最适合的四大场景**：

1. **用户行为分析**——社交、电商、游戏、短视频的埋点数据（play/like/share/purchase），需要多维度下钻和漏斗分析。
2. **实时推荐反馈环**——推荐系统发出推荐列表 → 用户产生行为 → ClickHouse分钟级聚合反馈CTR和转化率 → 模型在线更新。
3. **A/B实验平台**——分钟级实验指标（曝光数、转化率、置信区间），算法团队自助查询，无需等数仓T+1报表。
4. **广告效果监控**——按广告位/素材/人群包实时统计CPM、CTR、ROI，秒级反馈帮助广告主优化投放策略。

❌ **不适合的场景**：

1. **需要亚秒级点查的在线服务**（如用户画像API返回某个用户的所有标签）——ClickHouse设计面向OLAP聚合，单行点查延迟在10-50ms级别，不如Redis/HBase。
2. **需要事务和严格幂等的场景**（如支付记账）——MergeTree的Update/Delete是异步Mutation，不保证即时可见性。

### 注意事项

1. **Kafka分区数与Consumer数的对齐**：`kafka_num_consumers`必须≤Kafka分区总数。如果Consumer数 > 分区数，多余的Consumer会空闲。最佳实践：Consumer数 = 分区数 / 3 到 分区数 / 2，保证每个Consumer有2-3个分区可以处理。
2. **级联物化视图深度**：三层（分钟→小时→天）是实践经验的上限。四层开始，链路延迟累积、故障排查复杂度指数上升。如果需要更多维度，用并行的多张物化视图而非更深的级联。
3. **Map字段的性能陷阱**：`properties Map(String, String)`的读取和过滤性能远不如固定列。高频属性（如实验ID、广告ID）必须提出来做独立列。Map只用于长尾、偶发的动态属性。
4. **数据倾斜风险**：如果某个video_id或user_id产生的事件量远超平均（比如大V的一条视频10分钟百万播放），按`video_id`聚合时该分片会倾斜。建议在分片键中加入随机因子`rand()`均匀打散，查询时通过Distributed表自动合并。
5. **`uniqMerge` vs `uniqExact`**：在高精度去重场景（如广告计费UV），`uniqExact`精确但内存消耗大；`uniqMerge`基于HyperLogLog近似（误差约2%），内存恒定。根据业务要求选择——计费用精确，趋势分析用近似。

### 常见踩坑经验

**坑1：Kafka消息格式变更导致物化视图消费中断**

某天埋点SDK新增了`device_model`字段，消息JSON多了一个key，ClickHouse的`JSONEachRow`格式默认会忽略多余字段——但如果ClickHouse版本较低（< 22.x），严格模式下会直接报错并停止消费。**解法**：在Kafka Engine表的SETTINGS中显式设置`kafka_skip_broken_messages = 10`，容忍少量格式错误。更彻底的做法是用Flink做一层标准化清洗再入Kafka。

**坑2：级联物化视图某一级数据错误，需要从头重建**

某天发现小时级聚合表的`share_count`少了30%，排查发现是分钟级视图的`WHERE`条件写错了。**解法**：①先`DETACH VIEW`停掉下游视图；②用`INSERT INTO events_minute_agg SELECT ... FROM events WHERE ...`全量回填历史数据；③`ATTACH VIEW`恢复下游；④`OPTIMIZE TABLE events_minute_agg FINAL`触发Merge去重。切记操作前先备份原始表——物化视图的重建是不可逆的。

**坑3：大促峰值写入激增导致Merge积压，查询延迟飙升**

某电商大促期间，事件量从日均200亿飙升至500亿，ClickHouse的写入T+0完全没问题，但后台Merge速度跟不上写入速度，Part数从正常的200个暴增至8000个——每次查询都要`open()`更多文件，延迟从300ms涨到3秒。**解法**：①临时调大`background_pool_size`增加Merge并发线程；②临时降低`parts_to_delay_insert`阈值，触发写入限速保护；③短期方案——大促前手动`OPTIMIZE TABLE events FINAL`把历史Part合并好，为冲刺期腾出Merge带宽。

### 思考题

1. **如果用户行为数据延迟到达（如离线设备的数据晚到1-3天），如何更新已经聚合完成的物化视图？会不会导致已经出给运营的报表数值前后不一致？请设计一套补偿方案，并分析其对查询性能的影响。**

> 提示：考虑`event_time`与`receive_time`的双时间戳方案，让延迟数据归位到正确的历史分钟桶；同时设计一套版本号机制（如`version`字段+`ReplacingMergeTree`），让下游报表系统可以识别"某分钟的指标是否有更新"。

2. **该架构如何演进以支持实时特征工程（Feature Store）？例如，推荐模型需要"用户过去30分钟内看了多少个视频、平均观看时长、点赞率"作为实时特征。请设计一套基于ClickHouse的实时特征计算方案，要求特征更新延迟 < 1分钟，并讨论与现有物化视图层级的复用关系。**

> 提示：考虑在现有分钟聚合视图基础上，新增一张以`user_id`为维度的特征宽表；利用`Refreshable Materialized View`定时刷新特征；探讨特征如何通过HTTP接口被模型在线服务调用，以及如何避免特征查询对大盘查询造成资源竞争（第30章Workload调度）。
