# 第30章：多租户与资源隔离

> **版本**：ClickHouse 25.x LTS
> **定位**：中级篇核心章节。从"一个集群大家用"的混乱状态出发，通过用户/角色/配额/行策略/资源限制五层隔离体系，将共享集群升级为逻辑独立的多租户平台。
> **前置阅读**：第21章（用户与权限管理基础）、第25章（数据压缩与编码优化）、第26章（性能调优）
> **预计阅读**：35 分钟 | **实战耗时**：60 分钟

---

## 1. 项目背景

某数据平台团队管理着一套32核256GB内存的ClickHouse集群，为公司五个部门提供OLAP查询服务。市场营销部需要跑广告投放归因分析，财务部依赖实时风控查询——每一笔交易都必须在上线前完成黑名单校验，延迟超过500毫秒就算事故。运营部负责日常报表，需求不高但必须稳定跑完。数据科学部的分析师们偶尔会丢出一条"数据探索"类型的查询，动辄扫上百亿行数据。数据工程部则负责ETL写入，凌晨有批处理任务。

项目初期，架构很简单——一个集群、一个`default`数据库、一个`default`用户，所有人都用同一个账号连进来干活。前三个月相安无事。转折发生在第四个月：营销团队为了准备双十一大促，连续跑了一周的`SELECT DISTINCT`全表去重分析，每次都把32核CPU吃到95%以上。财务部的风控查询在这段时间里P99延迟从80毫秒飙升到了4秒——这意味着每100笔交易里有1笔可能要等4秒才能完成风控校验。风控接口超时直接触发熔断策略，交易系统被迫降级为"先放行、后审计"模式，合规部门当天就发了警告邮件。

"要不给每个部门单独部署一套ClickHouse？"运维组提出了最朴素的方案。但算了一笔账立刻否决了——5套32核256GB的集群，加上备份节点，光是硬件成本每月就超过15万，还不算运维人力。而且大部分时间各团队的用量都远低于物理上限，集群大量空转，ROI打骨折。

"那做时间切割呢？"产品经理提议，"财务上午跑，营销下午跑，科学晚上跑？"听起来合理，但实际行不通——风控查询必须7×24实时响应，不可能等到"轮到财务的时候"才跑。广告投放也不分时段，凌晨照样有流量。

CTO拍板：**必须在一个物理集群上实现逻辑多租户——每个部门看到的是"自己的ClickHouse"，不能访问彼此的数据，资源配额各自独立，高优先级查询不能被低优先级查询抢资源。** 这就是本章要解决的核心问题：在ClickHouse中构建生产级多租户架构。

## 2. 项目设计：剧本式交锋对话

周一架构评审会上，老张把五个部门的诉求列在白板上：隔离、保底、配额、可观测、可计费。小胖第一个跳了起来。

**小胖**："一个团队一个实例不就完了？5个团队5台ClickHouse，简单粗暴！出问题各找各的，谁也不影响谁。你们搞什么多租户、资源隔离，不就是把一台机器切碎分给五个人用吗？那还不如直接买五台！"

**小白**翻着ClickHouse的RBAC文档："但一台32核的机器，营销只用了白天的40%，财务只用了白天的20%，运营和科学加起来不到15%。五台机器总利用率不到30%。多租户的核心理念不就是共享削峰填谷吗？关键是能不能做到——营销跑重查询的时候，财务的轻查询延迟能不能稳在100毫秒以内？"

**大师**从公文包里掏出笔记本，打开ClickHouse终端："小胖的直觉没错——物理隔离永远是最彻底的。但公司不只有技术需求，还有成本约束。多租户解决的不是'能不能隔离'，而是'在不增加成本的前提下，能不能做到足够好的隔离'。ClickHouse从21.x版本开始，RBAC体系已经相当完善了——用户、角色、行策略、配额、设置档、工作负载调度，六层机制层层递进。"

大师在白板上画了一个洋葱图：

```
┌─ 网络层 ── <networks> IP白名单
│
├─ 数据库层 ── 每个租户独立Database（逻辑隔离）
│
├─ 表/列层 ── GRANT精确到SELECT(table.column)
│
├─ 行层 ── Row Policy按tenant_id过滤
│
├─ 资源层 ── Quota（时间窗口限流） + Settings Profile（硬资源上限）
│
└─ 调度层 ── Workload（查询优先级队列）
```

"第一层IP白名单最简单——限制连接来源，杜绝来自非授权网段的访问。第二层独立数据库，每个部门在自己的`tenant_xxx`库里操作，互不可见——这是最低成本的数据隔离。第三层GRANT到表和列级别，比如运营部只能读不能写，营销部可以在自己库内建表删表。第四层行策略才是最难也最精妙的部分——当多租户共享同一张事实表时（比如`shared_analytics.orders`），用`CREATE ROW POLICY`确保每个人只能看到自己租户的行。"

**技术映射 #1**：Row Policy本质上是一个隐式的WHERE子句。当你执行`SELECT * FROM shared_analytics.orders`时，ClickHouse会在查询编译阶段自动追加`AND tenant_id = 'marketing'`。这个WHERE条件是按行计算的——每读取一行就校验一次条件，因此存在一定的运行时开销（大约是每行一次字符串比较）。对于大数据量查询，建议将`tenant_id`放在ORDER BY的第一列，让ClickHouse能利用主键索引跳过不属于当前租户的Granule，大幅降低Row Policy的实际开销。

---

**小胖**挠了挠头："行策略我懂了。但配额这东西我有点糊涂——它到底是限制'同时能跑几个查询'，还是限制'一个小时总共能跑几个查询'？如果我设了一小时最多1000条查询，那我同时开100个连接、每个跑10条，是什么行为？"

**大师**："好问题。配额不是并发控制，而是**时间窗口内的累计量控制**。它像一个水表——不是控制水管同时能流出多少水，而是控制这个小时总共多少吨水流经你的水表。你一个小时发1000条查询，不管是用一个连接慢慢发、还是100个连接同时发，只要总数到了1000，第1001条就会被拒绝。而且配额不仅统计查询数量，还统计结果行数、结果字节数、总执行时间、错误次数——五个维度同时约束，先触碰哪个就以哪个为准。"

大师敲了一行SQL：

```sql
CREATE QUOTA quota_marketing
    FOR INTERVAL 1 HOUR
        MAX queries = 1000,
        MAX result_rows = 500000,
        MAX result_bytes = 500000000,
        MAX execution_time = 3600,
        MAX errors = 10
    FOR INTERVAL 1 DAY
        MAX queries = 5000,
        MAX result_rows = 5000000
    TO role_marketing;
```

"`FOR INTERVAL 1 HOUR`定义了一个滑动窗口——不是整点开始的固定窗口，而是从当前时刻往前推一小时。也就是说，任何时刻你前一小时内发了1000条查询，当前这条就会被拒绝。`FOR INTERVAL 1 DAY`是更长周期的限制——双重窗口确保某个租户不会在通宵跑批时把全天的配额一口气用完。"

**技术映射 #2**：Quota的计数器是纯内存维护的，ClickHouse重启后所有计数器清零。这意味着如果服务器在窗口期内重启，租户可以"白嫖"一次配额。在生产环境中建议配合监控告警——当配额使用率超过80%时提前通知，而不是等到100%再让查询失败。`system.quotas_usage`表可以实时查看每个用户当前窗口内的配额消耗量。

---

**小白**追问："资源限制呢？配额管的是'小时级'的累积量，但如果有人在配额允许的范围内，单条查询就吃掉40GB内存，怎么办？"

**大师**点头："这就需要Settings Profile了。配额管的是**宏观限流**，Settings Profile管的是**微观硬上限**。两者是正交的——配额说'你一小时最多花3600秒执行时间'，Profile说'你单条查询最多吃10GB内存、最多开5个并发'。Profile里可以设置十几个资源参数，最核心的是这四个：

- `max_memory_usage_for_user`：单个用户所有并发查询加起来最多用多少内存。这个参数是ClickHouse 23.x之后引入的，比老的`max_memory_usage`更精准——老的只限制单条查询，这个限制用户级别总内存。
- `max_concurrent_queries_for_user`：单个用户最多同时跑多少个查询。第6个连接到同一个用户发起查询时，会排队等待或直接拒绝。
- `max_execution_time`：单条查询最多跑多少秒，超时自动终止。这是防止"失速查询"的最后防线。
- `readonly`：设成1之后，这个角色的用户只能读不能写——非常适合运营和审计团队。

**小胖**："等等，你刚才说工作负载调度？Workload是什么？是不是能指定'营销的查询优先级低'，'财务的查询优先级高'，然后ClickHouse自动让财务的查询插队？"

**大师**："完全正确。这是24.x版本引入的新特性。你可以创建Workload对象，给每个Workload分配优先级。然后把Workload绑定到角色或者用户上。当多个租户的查询同时在排队时，优先级高的Workload的查询会被优先调度执行。"

```sql
CREATE WORKLOAD workload_critical SETTINGS priority = 100;
CREATE WORKLOAD workload_batch SETTINGS priority = 10;
```

"财务的风控查询绑定`workload_critical`，营销的归因分析绑定`workload_batch`。当两者同时抢计算资源时，critical优先拿到线程池和I/O带宽。注意——Workload调度只在查询排队时生效，已经拿到资源正在跑的查询不会被抢占（那属于preemption，ClickHouse目前不支持）。所以Workload更像'排队时的VIP通道'，而不是'跑着跑着把你踢出CPU'。"

**小白**："那综合来看，给一个租户做隔离，大概需要做哪些配置？"

**大师**合上笔记本总结："一套标准的多租户配置模板，包括五个对象的创建和五条GRANT语句。我给大家一个清单：每个租户=一个Database + 一个Role + 一个Quota + 一个Settings Profile + 一个Workload。五个对象各司其职，打出一套组合拳。开工吧。"

## 3. 项目实战

### 环境准备

启动ClickHouse容器时，必须配置`access_management`和`named_collection_control`，确保RBAC功能开启：

```bash
# 启动启用了用户管理功能的ClickHouse
docker run -d --name ch-tenant-lab \
  --cpus=4 --memory=8g \
  -p 8123:8123 -p 9000:9000 \
  -e CLICKHOUSE_USER=admin \
  -e CLICKHOUSE_PASSWORD=Admin@123 \
  clickhouse/clickhouse-server:25
```

进入容器并以admin身份连接：

```bash
docker exec -it ch-tenant-lab clickhouse-client -u admin --password Admin@123
```

检查access_management是否已启用：

```sql
SELECT name, value FROM system.settings WHERE name = 'access_management';
-- 应为 1
```

> **重要**：默认的`default`用户拥有全部权限且无密码。生产环境第一步永远是——给default用户设置密码并收紧权限，或者直接禁用。

### 分步实现

#### Step 1：创建租户架构

首先，为每个部门创建独立的数据库，实现最基础的逻辑隔离：

```sql
-- 各部门专属数据库（Schema级隔离）
CREATE DATABASE tenant_marketing;
CREATE DATABASE tenant_finance;
CREATE DATABASE tenant_operations;
CREATE DATABASE tenant_datascience;

-- 共享分析库（多租户共用，靠Row Policy隔离）
CREATE DATABASE shared_analytics;
```

每个租户在自己的数据库内拥有最高权限，彼此不可见。`shared_analytics`用于存放需要跨部门分析的公共数据——比如订单流水表，各部门都能查但只能看到属于自己的行。

#### Step 2：创建角色和用户

为每个部门创建角色和用户。角色是权限的载体，用户通过角色继承权限：

```sql
-- 营销团队
CREATE ROLE role_marketing;
CREATE USER marketing_alice IDENTIFIED BY 'Alice@Pass!2025' DEFAULT ROLE role_marketing;
CREATE USER marketing_bob IDENTIFIED BY 'Bob@Pass!2025' DEFAULT ROLE role_marketing;

-- 财务团队
CREATE ROLE role_finance;
CREATE USER finance_charlie IDENTIFIED BY 'Charlie@Pass!2025' DEFAULT ROLE role_finance;

-- 运营团队（只读角色，只能看不能改）
CREATE ROLE role_operations;
CREATE USER ops_diana IDENTIFIED BY 'Diana@Pass!2025' DEFAULT ROLE role_operations;

-- 数据科学团队（分析角色，可读可建临时表）
CREATE ROLE role_datascience;
CREATE USER ds_edward IDENTIFIED BY 'Edward@Pass!2025' DEFAULT ROLE role_datascience;

-- 管理员角色（保留给DBA使用）
CREATE ROLE role_admin;
CREATE USER admin_root IDENTIFIED BY 'Root@Secret#2025' DEFAULT ROLE role_admin;
GRANT ALL ON *.* TO role_admin WITH GRANT OPTION;

-- 第一时间收紧default用户
ALTER USER default IDENTIFIED BY 'Default@Complex#2025';
REVOKE ALL ON *.* FROM default;
```

> `DEFAULT ROLE`意味着用户登录后自动激活该角色。如果在创建用户时没有指定`DEFAULT ROLE`，用户登录后需要手动`SET ROLE xxx`才能拥有权限——这是常见的新手坑。

#### Step 3：配置权限隔离（GRANT/REVOKE）

权限粒度可以精确到数据库、表、甚至列级别。按最小权限原则分配：

```sql
-- 营销部门：在自己库内有完全控制权，共享库只读
GRANT SELECT, INSERT, ALTER, CREATE, DROP, TRUNCATE
    ON tenant_marketing.* TO role_marketing;
GRANT SELECT ON shared_analytics.* TO role_marketing;

-- 财务部门：同样拥有自己的库，共享库只读
GRANT SELECT, INSERT, ALTER, CREATE, DROP, TRUNCATE
    ON tenant_finance.* TO role_finance;
GRANT SELECT ON shared_analytics.* TO role_finance;

-- 运营部门：所有数据库只读（含共享库）
GRANT SELECT ON tenant_marketing.* TO role_operations;
GRANT SELECT ON tenant_finance.* TO role_operations;
GRANT SELECT ON tenant_operations.* TO role_operations;
GRANT SELECT ON shared_analytics.* TO role_operations;

-- 数据科学部门：自己库内可读写，共享库只读，不能删表
GRANT SELECT, INSERT, ALTER, CREATE
    ON tenant_datascience.* TO role_datascience;
GRANT SELECT ON shared_analytics.* TO role_datascience;

-- 验证权限配置
SHOW GRANTS FOR role_marketing;
SHOW GRANTS FOR role_operations;
```

关键设计：运营团队虽然可以`SELECT`所有数据库，但后续通过Row Policy限制其在共享库中能看到的行——权限和策略双层控制。

#### Step 4：行级安全策略（Row Policy）

在共享订单表中插入租户标识列，然后用Row Policy实现行级隔离：

```sql
-- 创建共享订单表（包含tenant_id用于行级分区）
CREATE TABLE shared_analytics.orders (
    order_id UInt64,
    tenant_id LowCardinality(String),
    customer_name String,
    amount Decimal(10, 2),
    created_at DateTime
) ENGINE = MergeTree()
ORDER BY (tenant_id, created_at, order_id)
PARTITION BY toYYYYMM(created_at);

-- 插入多租户模拟数据
INSERT INTO shared_analytics.orders VALUES
    (1, 'marketing', '客户A', 100.00, '2025-04-01 10:00:00'),
    (2, 'marketing', '客户B', 250.00, '2025-04-02 11:00:00'),
    (3, 'finance', '客户C', 500.00, '2025-04-03 09:00:00'),
    (4, 'operations', '客户D', 75.50, '2025-04-04 14:00:00'),
    (5, 'finance', '客户E', 1200.00, '2025-04-05 16:00:00');

-- 行策略1：营销部只能看到tenant_id='marketing'的行
CREATE ROW POLICY rp_marketing_isolation ON shared_analytics.orders
    FOR SELECT
    USING tenant_id = 'marketing'
    TO role_marketing;

-- 行策略2：财务部只能看到tenant_id='finance'的行
CREATE ROW POLICY rp_finance_isolation ON shared_analytics.orders
    FOR SELECT
    USING tenant_id = 'finance'
    TO role_finance;

-- 行策略3：运营部可以看到全部行（但不可写，权限已控制）
CREATE ROW POLICY rp_ops_full ON shared_analytics.orders
    FOR SELECT
    USING 1 = 1  -- 无过滤，全量可见
    TO role_operations;

-- 验证：以marketing_alice身份登录测试
-- docker exec -it ch-tenant-lab clickhouse-client -u marketing_alice --password 'Alice@Pass!2025'
-- SELECT DISTINCT tenant_id FROM shared_analytics.orders;
-- 预期：只返回 'marketing' 一行
```

> **行策略叠加规则**：当一个用户属于多个角色、且多个角色绑定了同一张表的不同Row Policy时，ClickHouse使用**OR语义**合并——即满足任意一条策略的行都对用户可见。这就是为什么设计角色时应避免一个用户同时拥有marketing和finance角色，否则他将看到两个租户的数据。

验证Row Policy实际生效的SQL——从`system.row_policies`查看：

```sql
SELECT
    database,
    table,
    name,
    storage_type,  -- 'table' = 本地表 / 'distributed' = 分布式表
    is_restrictive,
    conditions_count
FROM system.row_policies
WHERE database = 'shared_analytics';
```

#### Step 5：配额管理（Quota）

配额按角色分配，建立时间窗口内的累计消费上限：

```sql
-- 营销配额：宽松，支持大量ad-hoc分析
CREATE QUOTA quota_marketing
    FOR INTERVAL 1 HOUR
        MAX queries = 2000,
        MAX result_rows = 2000000,
        MAX result_bytes = 2000000000,  -- 2GB
        MAX execution_time = 7200,      -- 2小时总执行时间
        MAX errors = 20
    FOR INTERVAL 1 DAY
        MAX queries = 10000,
        MAX result_rows = 10000000,
        MAX execution_time = 36000
    TO role_marketing;

-- 财务配额：严格，保证稳定性优先
CREATE QUOTA quota_finance
    FOR INTERVAL 1 HOUR
        MAX queries = 500,
        MAX result_rows = 200000,
        MAX result_bytes = 500000000,
        MAX errors = 5
    FOR INTERVAL 1 DAY
        MAX queries = 3000,
        MAX result_rows = 2000000
    TO role_finance;

-- 运营配额：最小，只跑监控报表
CREATE QUOTA quota_operations
    FOR INTERVAL 1 HOUR
        MAX queries = 200,
        MAX result_rows = 100000
    FOR INTERVAL 1 DAY
        MAX queries = 1000
    TO role_operations;

-- 数据科学配额：中等，允许探索性查询但防止失控
CREATE QUOTA quota_datascience
    FOR INTERVAL 1 HOUR
        MAX queries = 300,
        MAX result_rows = 5000000,
        MAX execution_time = 3600
    FOR INTERVAL 1 DAY
        MAX queries = 1500,
        MAX result_rows = 20000000
    TO role_datascience;
```

重要提醒——配额中的`0`表示**无限制**而非**禁用**：

```sql
-- 错的：想禁用queries维度，设成0 → 反而变成了无限制！
CREATE QUOTA bad_quota FOR INTERVAL 1 HOUR MAX queries = 0 TO some_role;

-- 对的：不想限制某个维度，直接省略该MAX子句
CREATE QUOTA good_quota FOR INTERVAL 1 HOUR MAX result_rows = 50000 TO some_role;
```

监控配额使用情况的查询可以直接在admin连接中执行：

```sql
SELECT
    quota_name,
    user_name,
    start_time,
    queries,
    max_queries,
    result_rows,
    max_result_rows,
    formatReadableSize(result_bytes) AS result_bytes_fmt,
    formatReadableSize(max_result_bytes) AS max_result_bytes_fmt,
    execution_time,
    max_execution_time,
    -- 使用百分比标识最接近上限的维度（预警用）
    round(queries * 100.0 / nullIf(max_queries, 0), 1) AS queries_pct,
    round(result_rows * 100.0 / nullIf(max_result_rows, 0), 1) AS rows_pct
FROM system.quotas_usage
WHERE quota_name != 'default'
ORDER BY queries_pct DESC;
```

## 4. 项目总结

### 多租户隔离层次全景

| 层级 | 机制 | 隔离强度 | 性能开销 | 典型场景 |
|------|------|---------|---------|---------|
| 网络 | `<networks>` IP白名单 | 连接级 | 无 | 限制办公网/VPN接入 |
| 数据库 | 独立Database | Schema级 | 无 | 不同部门数据完全隔离 |
| 表/列 | GRANT/REVOKE | 操作级 | 无 | 只读角色、敏感列屏蔽 |
| 行 | Row Policy | 数据级 | 每条查询追加WHERE子句 | 多租户共享同一张事实表 |
| 资源 | Quota + Settings Profile | 计算/内存级 | 无（仅计数器） | 防止单租户耗尽集群资源 |
| 调度 | Workload | 排队优先级 | 无 | 核心业务查询优先于批处理 |

### 适用场景

- **企业数据平台**：多部门共享一个ClickHouse集群，每个部门需要独立的数据视图和资源配额。
- **SaaS多租户**：每个客户对应一个租户——通过Row Policy实现租户间的行级数据隔离，通过Quota限制单个客户的数据拉取量。
- **开发/测试/生产共用**：同一集群上划定三个环境，生产库只有只读权限，开发库可以自由建表删表，测试库有独立配额。
- **组织内分级的数据服务**：分析师可以查询但不能修改（readonly=1），数据工程师可以写入但不能删除（GRANT精确控制），DBA拥有全部权限。

### 注意事项

1. **Row Policy在分布式表上的行为差异**：当对`Distributed`表执行查询时，`currentUser()`在远端分片上返回的是ClickHouse的代理用户（通常为`default`），而非实际发起查询的用户。这意味着基于`currentUser()`的Row Policy在分布式场景下会失效。需要使用`EXECUTABLE`类型的Row Policy传入`currentUser()`作为参数，或者改用`tenant_id`和角色名之间的静态映射策略。

2. **配额计数器依赖内存**：`system.quotas_usage`中的计数器全部在内存中维护。ClickHouse重启后全部归零——这意味着如果你重启后不重新设置配额限制，用户实际上拥有了"无限制"的配额弹性。生产环境中应在配置文件中持久化Quota定义（`/etc/clickhouse-server/users.d/`目录下的XML/YAML文件），确保重启后自动加载。

3. **太多Profile会增加运维复杂度**：当租户数量从5个增长到50个时，维护50套Profile、50套Quota、50套Role会变得非常繁琐。建议将租户按业务特点分组——"核心业务型"、"分析探索型"、"只读监控型"——每组一套Profile模板，租户按组分配。新租户接入时只需创建Database+Role+User三个对象，其余配置继承组模板。

4. **忘记给default用户收紧权限**：默认情况下`default`用户拥有`ALL ON *.*`的完整权限且无密码。新集群上线后的第一步操作必须是给`default`设置密码并`REVOKE ALL ON *.* FROM default`。否则所有RBAC配置形同虚设——任何人用HTTP客户端访问8123端口，不带认证信息就能以`default`身份操作所有数据。

### 常见踩坑经验

1. **Row Policy中currentUser()的变量未正确配置**。在`users.xml`或`CREATE USER`语句中，需要显式为用户设置`access_management = 1`并确保用户有权限执行`currentUser()`函数。某些ClickHouse版本中，非admin用户调用`currentUser()`会返回空字符串，导致Row Policy匹配不到任何行——用户看到的数据为空而不是报错，排查起来非常隐蔽。

2. **Quota的`MAX queries = 0`是无限而非禁止**。这个坑几乎每个新手都会踩一次。0在ClickHouse配置语义中普遍代表"不限制"——包括`max_memory_usage`、`max_execution_time`等参数也是如此。若想禁止某个维度（比如完全禁止写操作），应该通过权限系统（`REVOKE INSERT`）实现，而非依赖配额。

3. **Settings Profile的`readonly`粒度不足**。`readonly`只有三级——0（无限制）、1（禁止写入和DDL）、2（禁止写入但允许DDL）。如果想实现"可以INSERT但不能ALTER"，readonly无能为力，必须在GRANT层面精确控制——`GRANT INSERT ON db.table TO role`但`REVOKE ALTER ON db.table FROM role`。

4. **分布式表上Row Policy的"幽灵行"问题**。在集群环境下，如果Row Policy使用`currentUser()`做过滤条件，而分布式查询在远端节点上以代理用户身份执行，Row Policy可能完全失效——用户看到的是全量数据。解决方法是使用条件`tenant_id IN (SELECT ... FROM local_tenant_mapping WHERE user = currentUser())`，或者在应用层做SQL改写——查询拼接前把`tenant_id = 'xxx'`条件硬编码进SQL。

### 思考题

1. **分布式表上的正确行级安全**：当`SELECT`通过Distributed表扇出到多个分片时，`currentUser()`在每个远端分片上返回的是什么？如何设计一套不依赖`currentUser()`的行级安全方案——比如通过在`EXECUTABLE`类型的Row Policy中调用外部服务做用户到租户ID的映射？

2. **租户费用分摊模型**：设计一套统计SQL，汇总各个租户在过去一个月内实际消耗的CPU时间（`ProfileEvents['RealTimeMicroseconds']`）、内存峰值（`MemoryUsage`）、读取字节数（`read_bytes`）和扫描行数（`read_rows`）。如何利用`system.query_log`和`system.quotas_usage`构建按租户维度的资源消费账单？——提示：利用`query_log`中的`user`字段关联到租户角色，利用`ProfileEvents`提取CPU和I/O指标。
