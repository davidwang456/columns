# 第3章：数据类型与建表入门

## 1. 项目背景

某中型电商公司技术团队正在做一件大事——把积累了3年的订单数据从MySQL迁移到ClickHouse。CTO给出的目标很明确："报表查询从30秒压到1秒以内，存储成本控制在原来的三分之一。"

DBA老张翻出当年设计的MySQL表结构，不禁感叹："当初建`orders`表的时候，`order_no`用的`VARCHAR(255)`，`amount`用的`FLOAT`，`status`用的`VARCHAR(20)`，`remark`用的`TEXT`……这在MySQL里跑得好好的，怎么到了ClickHouse就不灵了呢？"

问题很快暴露出来。按照MySQL的表结构原样迁移后，一张2亿行的订单表占了接近500GB的磁盘空间，而同样的数据用ClickHouse的最优类型建表，理论上只需要150GB左右。更糟糕的是，全表扫描`SELECT sum(amount) FROM orders WHERE status = 'paid'`跑了12秒，而CTO要求的是1秒以内。

团队里的小胖最先发出抱怨："不就是存个数据嘛，我全用String类型得了，省心省力。MySQL里我也是这么干的——能VARCHAR绝不用别的。"结果一张实验表跑下来，500GB变成了800GB，查询反而更慢了。大师看了一眼，叹了口气说："你这不是省心，是把卡车发动机装到了自行车上。"

这就是本章要解决的核心问题——ClickHouse的数据类型系统比MySQL丰富得多，也"讲究"得多。从无符号整数到高精度定点数，从低基数编码到复合嵌套类型，每一种类型的背后都是存储效率和查询性能的博弈。选错类型，轻则浪费几十GB的磁盘空间，重则让查询性能打折50%以上。

本章将以这家电商公司的订单表重构为线索，带你从零开始掌握ClickHouse的数据类型体系与建表最佳实践。目标很明确：**同样的2亿行订单数据，用最优的类型建表，存储从500GB降到150GB，查询从12秒降到0.8秒。**

## 2. 项目设计：剧本式交锋对话

*场景：周五下午，会议室。小白面前摊着一份ClickHouse官方文档，小胖抱着一袋薯片，大师端着保温杯推门进来。*

---

**小胖**（嚼着薯片）："大师，我实在想不通。MySQL里我用`INT`能存用户ID，用`VARCHAR(255)`能存订单号，用`FLOAT`能存金额，怎么到ClickHouse就这么多讲究？光整数就分了`UInt8`、`UInt16`、`UInt32`、`UInt64`，还有带符号和不带符号的区别，这是要考我二进制吗？我全用`Int64`不就好了——内存便宜嘛！"

**大师**（放下保温杯）："小胖，我问你一个问题。你每周从食堂打包，饭盒有大中小三个号。如果每次不管打多少菜都拿最大号饭盒，你觉得怎么样？"

**小胖**："那当然浪费啊！我就打两个菜，拿个小号饭盒就够了。大号饭盒占地方，洗碗阿姨还不高兴。"

**大师**："你这个比喻很精准。`UInt8`就是小号饭盒——只占1个字节，能存0到255。`UInt64`是大号饭盒——占8个字节。同样是存用户的`age`字段，用`UInt8`每行占1字节，用`Int64`每行占8字节。2亿行数据，光是这一列就差了1.4GB。而且，更大的数据类型意味着每一个Granule（默认为8192行）中的有效行数没变，但Granule的物理尺寸翻倍了——MergeTree在扫描时会读取更多的磁盘数据页。"

**技术映射**：整数类型的选择 = 饭盒选号。`UInt8=1字节`，`UInt16=2字节`，`UInt32=4字节`，`UInt64=8字节`。类型越大，存储越大，扫描越慢。始终选择能容纳数据范围的最小类型。

---

**小白**（放下文档）："大师，这个我能理解——数值类型选小的。但我有两个问题。第一个：`LowCardinality(String)`到底是怎么做到又存字符串又省空间的？第二个：`Nullable`在MySQL里很常用，为什么ClickHouse文档说它'代价高昂'？"

**大师**："好问题。先讲LowCardinality。"

**大师**（在白板上画起来）："`LowCardinality(String)`的核心思想是**字典编码**。假设你有一列`status`，只有`'pending'`、`'paid'`、`'shipped'`、`'cancelled'`四种值。不用LowCardinality，每行都存完整的字符串——比如`'pending'`是7个字节，2亿行就是1.4GB。用LowCardinality后，ClickHouse会先建一个字典：0→'pending'，1→'paid'，2→'shipped'，3→'cancelled'。字典里存了字符串本身，而数据文件里只存小小的整数索引——每个值只需要1到2个字节。"

**小胖**："这不就跟我们公司的快递柜一样吗？小号柜子放小的包裹，大号柜放大的。取件码就是那个整数索引，实际包裹在柜子里。"

**大师**："对，就是这个理。而且字典带来的好处不止存储压缩。在做`WHERE status = 'paid'`这种过滤时，ClickHouse只需要对字典编码后的整数列做比较，速度快得多。但要注意一点——当你的列基数（distinct values）超过100万条时，字典本身会变得很大，查询反而变慢。这就是为什么它叫'Low'Cardinality。"

**技术映射**：LowCardinality = 快递柜取件码。字典存实际值（包裹），数据列存整数引用（取件码）。适合基数小于100万的列，如状态、国家代码、设备类型。

---

**大师**："再说Nullable。它的代价来自底层存储格式——每一个Nullable列都会额外附带一个独立的UInt8标记列，负责记录每一行是NULL还是NOT NULL。也就是说，一个`Nullable(Int32)`在磁盘上实际占用的是5个字节——4字节存Int32，再加1字节的NULL标记。"

**小白**："那如果我只是偶尔有几个NULL值呢？5个字节也还好吧？"

**大师**："存储不是最伤的。真正的问题是**查询性能**。因为这个NULL标记列和实际数据列是分开存储的，执行任何操作时——过滤、聚合、排序——ClickHouse都要同时操作两个物理列。`WHERE amount > 100`这个简单条件，如果amount是Nullable，ClickHouse得先检查NULL标记，再检查数值。这会使查询执行计划的处理器变复杂，部分场景下性能下降可达50%。"

**小胖**："那我不要NULL了，全用默认值行不行？"

**大师**："正是最佳实践。很多场景下，NULL可以用哨兵值替代。比如`status`的NULL可以改为`'unknown'`，`amount`的NULL可以默认为0，`created_at`的NULL可以默认为`'1970-01-01'`。如果你的业务确实需要区分'值为0'和'值不存在'这两种语义，可以额外加一个布尔列来标记。这样数据更紧凑，查询也更快速。"

**技术映射**：Nullable列 = 每个数据带一个"小跟班"标记它是不是NULL。回避方案：用哨兵默认值 + 额外布尔列替代。

---

**小白**："还有一个问题。`FixedString(N)`听起来像是优化过的String——固定长度，应该更快吧？"

**大师**（连忙摆手）："小心！这是ClickHouse里最大的坑之一。FixedString(N)会**用NULL字节`\0`填充不足的部分**。比如你定义`FixedString(10)`，存'CHN'进去，实际存储的是`CHN\0\0\0\0\0\0\0`——7个NULL字符。这在`WHERE code = 'CHN'`时不会自动匹配，因为`'CHN'`和`'CHN\0\0\0\0\0\0\0'`是不同的字符串。"

**小胖**："啊？！那它存在的意义是什么？"

**大师**："它的应用场景非常狭窄——仅在二进制数据（如哈希值、UUID的二进制表示）且数据本身就是定长时有用，比如MD5的16字节原始二进制、IPV6地址的16字节。对于普通的字符串业务字段，永远用`String`或`LowCardinality(String)`，忘掉FixedString。"

---

**小结**

通过三人的交锋对话，我们达成了以下共识：

| 场景 | 推荐类型 | 避免类型 |
|------|---------|---------|
| 用户ID、计数 | UInt32/UInt64 | Int64（无符号场景浪费符号位） |
| 金额计算 | Decimal(18,2) | Float32/Float64 |
| 状态、类别（低基数） | LowCardinality(String) | String / Enum |
| 文本、JSON、日志 | String | FixedString(N) |
| 可选字段 | 带默认值的普通类型 | Nullable |
| 时间戳 | DateTime/DateTime64 | String存时间 |

### 分步实现

#### Step 1：设计最优订单表结构

**目标**：为2亿行电商订单数据设计存储高效、查询快速的最优表结构。

```sql
-- 创建数据库
CREATE DATABASE IF NOT EXISTS ecommerce;

-- 设计最优类型的订单表
CREATE TABLE ecommerce.orders (
    -- 订单ID：范围0~100亿，用UInt64（8字节），不用Int64浪费符号位
    order_id        UInt64,

    -- 用户ID：预计不超过43亿，UInt32足够（4字节 vs 8字节）
    user_id         UInt32,

    -- 商品ID列表：一个订单多个商品，用Array类型天然支持
    product_ids     Array(UInt32),

    -- 订单金额：Decimal(10,2)精度精确到分，不用Float避免浮点误差
    amount          Decimal(10,2),

    -- 订单状态：只有4种值，LowCardinality字典编码，比String省80%
    status          LowCardinality(String),

    -- 支付方式：5种左右，同样用LowCardinality
    pay_method      LowCardinality(String),

    -- 国家代码：ISO 3166-1，约249种，LowCardinality完美适配
    country_code    LowCardinality(String),

    -- 创建时间：DateTime精确到秒，存储为4字节UInt32
    created_at      DateTime,

    -- 更新时间：DateTime64(3)精确到毫秒，存储为8字节
    updated_at      DateTime64(3),

    -- 扩展信息：Map类型存储动态键值对（JSON风格）
    extra_info      Map(String, String)

) ENGINE = MergeTree()
ORDER BY (created_at, order_id)
PARTITION BY toYYYYMM(created_at)
SETTINGS index_granularity = 8192;
```

**设计要点说明**：

- `order_id`用`UInt64`而非`Int64`：订单ID从1开始自增，永不为负，无符号类型省掉符号位的1字节开销（实际上UInt64和Int64都是8字节，但语义更清晰，且在与外部系统对接时避免溢出）。
- `amount`用`Decimal(10,2)`而非`Float32`：`Decimal`是定点数，保证`0.1 + 0.2 = 0.3`精确成立。`Float32`用IEEE 754浮点数，`0.1 + 0.2`可能等于`0.30000000000000004`——在财务场景中是致命错误。
- `country_code`用`LowCardinality(String)`而非`String`：249种国家代码，每行存储从2字节的String变为1字节的字典索引，压缩率约50%。
- `ORDER BY (created_at, order_id)`：时间在前、ID在后，按时间范围查询时可以充分利用主键索引。
- `PARTITION BY toYYYYMM(created_at)`：按月分区，方便按时间范围裁剪Part和TTL管理。

#### Step 2：对比不同字符串类型的存储差异

**目标**：用10万行真实数据，对比`String`、`FixedString(20)`、`LowCardinality(String)`三种字符串类型在磁盘上的实际存储占用。

```sql
-- 创建三张结构相同、仅字符串类型不同的表
CREATE TABLE ecommerce.country_string (
    country_code String
) ENGINE = MergeTree() ORDER BY country_code;

CREATE TABLE ecommerce.country_fixedstring (
    country_code FixedString(20)
) ENGINE = MergeTree() ORDER BY country_code;

CREATE TABLE ecommerce.country_lowcard (
    country_code LowCardinality(String)
) ENGINE = MergeTree() ORDER BY country_code;

-- 生成10万行模拟数据：随机分配50种国家代码
INSERT INTO ecommerce.country_string
SELECT ['CN','US','JP','DE','GB','FR','IN','BR','CA','AU',
        'KR','IT','ES','MX','RU','NL','SE','CH','NO','DK',
        'FI','PL','AT','BE','IE','PT','GR','NZ','SG','ZA',
        'AE','TH','MY','ID','PH','VN','AR','CL','CO','PE',
        'EG','NG','KE','MA','SA','TR','PK','BD','RO','UA'][toInt32(randUniform(1, 51))]
FROM numbers(100000);

-- 复制数据到另外两张表
INSERT INTO ecommerce.country_fixedstring
SELECT country_code FROM ecommerce.country_string;

INSERT INTO ecommerce.country_lowcard
SELECT country_code FROM ecommerce.country_string;

-- 对比存储占用（执行前先执行 OPTIMIZE 触发合并）
OPTIMIZE TABLE ecommerce.country_string FINAL;
OPTIMIZE TABLE ecommerce.country_fixedstring FINAL;
OPTIMIZE TABLE ecommerce.country_lowcard FINAL;

SELECT
    table,
    formatReadableSize(sum(data_uncompressed_bytes)) AS uncompressed,
    formatReadableSize(sum(data_compressed_bytes)) AS compressed,
    round(sum(data_compressed_bytes) / sum(data_uncompressed_bytes) * 100, 1) AS compression_ratio
FROM system.parts
WHERE database = 'ecommerce'
  AND table IN ('country_string', 'country_fixedstring', 'country_lowcard')
  AND active = 1
GROUP BY table
ORDER BY table;
```

**预期输出**（实际数值因数据分布略有差异）：

| table                  | uncompressed | compressed | compression_ratio |
|------------------------|-------------|------------|-------------------|
| country_fixedstring    | 2.00 MiB    | 48.75 KiB  | 2.4%              |
| country_lowcard        | 97.66 KiB   | 1.13 KiB   | 1.2%              |
| country_string         | 203.13 KiB  | 29.77 KiB  | 14.7%             |

**结果解读**：

- `FixedString(20)`未压缩时最大（2000000字节 = 100000行 × 20字节/行），因为每行强制占满20字节，不足部分用NULL填充。压缩后反而最小——大量重复的`\0`字节被LZ4高效压缩了。
- `LowCardinality(String)`无论是压缩前还是压缩后都表现最优。压缩前只有约98KB（100000行 × 1字节字典索引 + 50个字符串的字典），压缩后仅1KB出头。
- **关键认知**：不要因为FixedString压缩后很小就用它——NULL填充会导致字符串比较、JOIN等操作行为异常，带来难以排查的Bug。LowCardinality才是正解。

#### Step 3：Nullable的代价实验

**目标**：量化Nullable列在存储和查询性能上的实际代价。

```sql
-- 实验表A：使用Nullable列
CREATE TABLE ecommerce.test_nullable (
    id      UInt32,
    value   Nullable(Int32),
    label   Nullable(String)
) ENGINE = MergeTree() ORDER BY id;

-- 实验表B：使用默认值替代Null
CREATE TABLE ecommerce.test_non_nullable (
    id      UInt32,
    value   Int32 DEFAULT 0,
    label   String DEFAULT 'N/A'
) ENGINE = MergeTree() ORDER BY id;

-- 插入100万行：其中30%的value为空，10%的label为空
INSERT INTO ecommerce.test_nullable
SELECT
    number AS id,
    if(rand() < 0.3, NULL, toInt32(rand() * 10000)) AS value,
    if(rand() < 0.1, NULL, ['paid','shipped','pending'][toInt32(randUniform(1, 4))]) AS label
FROM numbers(1000000);

-- 非Nullable表：NULL用默认值替代
INSERT INTO ecommerce.test_non_nullable
SELECT
    id,
    ifNull(value, 0),
    ifNull(label, 'N/A')
FROM ecommerce.test_nullable;

OPTIMIZE TABLE ecommerce.test_nullable FINAL;
OPTIMIZE TABLE ecommerce.test_non_nullable FINAL;

-- 存储对比
SELECT
    table,
    formatReadableSize(sum(data_compressed_bytes)) AS compressed_size
FROM system.parts
WHERE database = 'ecommerce'
  AND table IN ('test_nullable', 'test_non_nullable')
  AND active = 1
GROUP BY table;
```

**查询性能对比**：

```sql
-- 开启计时
SET send_logs_level = 'warning';

-- 测试1：聚合查询——Nullable表
SELECT count(), sum(value)
FROM ecommerce.test_nullable
WHERE value > 5000;
-- 预期耗时：约0.08秒

-- 测试2：同样的聚合——非Nullable表
SELECT count(), sum(value)
FROM ecommerce.test_non_nullable
WHERE value > 5000;
-- 预期耗时：约0.04秒（约为Nullable表的一半）

-- 测试3：GROUP BY查询——Nullable表
SELECT label, count()
FROM ecommerce.test_nullable
GROUP BY label
ORDER BY count() DESC;
-- 预期耗时：约0.12秒

-- 测试4：同样的GROUP BY——非Nullable表
SELECT label, count()
FROM ecommerce.test_non_nullable
GROUP BY label
ORDER BY count() DESC;
-- 预期耗时：约0.07秒（约为Nullable表的一半到60%）
```

**最佳实践总结**：

- 如果NULL值的语义是"未知"，用默认值`''`、`0`、`'unknown'`替代 — 查询快50%~100%。
- 如果NULL值的语义是"不存在"且必须与"为0"区分，额外加一个`has_value UInt8`布尔列 — 既清晰又高效。
- `Nullable(Int32)`在磁盘上占用5字节（4 + 1标记字节），而非Nullable的`Int32`仅占4字节——100万行多占约1MB。

#### Step 4：数组和嵌套类型实战

**目标**：掌握`Array`类型的写入、查询和展开操作。

```sql
-- 向orders表插入示例数据
INSERT INTO ecommerce.orders VALUES
(100001, 42, [101, 203, 305], 299.99, 'paid', 'wechat', 'CN', now(), now64(3), {'color':'red','size':'XL'}),
(100002, 57, [102, 204], 159.50, 'paid', 'alipay', 'US', now(), now64(3), {'color':'blue','note':'gift'}),
(100003, 99, [103], 89.00, 'pending', 'card', 'JP', now(), now64(3), {'color':'black'}),
(100004, 42, [101, 104, 205, 306], 599.00, 'shipped', 'wechat', 'CN', now(), now64(3), {'color':'red','size':'M','note':'urgent'}),
(100005, 128, [107, 108], 199.00, 'cancelled', 'alipay', 'GB', now(), now64(3), {'reason':'out_of_stock'});

-- 查询1：基础Array查询
SELECT
    order_id,
    user_id,
    product_ids,
    length(product_ids) AS product_count
FROM ecommerce.orders
WHERE status = 'paid';

-- 查询2：ARRAY JOIN 展开数组——每行变成一个商品一行
SELECT
    order_id,
    user_id,
    product_id
FROM ecommerce.orders
ARRAY JOIN product_ids AS product_id
WHERE status = 'paid';

-- 查询3：Map类型查询
SELECT
    order_id,
    extra_info['color'] AS color,
    extra_info['size'] AS size
FROM ecommerce.orders
WHERE has(mapKeys(extra_info), 'color');

-- 查询4：检查Map中是否存在某个键
SELECT
    order_id,
    mapContains(extra_info, 'note') AS has_note
FROM ecommerce.orders;

-- 查询5：聚合每个商品的订单数（展开后统计）
SELECT
    product_id,
    count() AS order_count,
    sum(amount) AS total_amount
FROM ecommerce.orders
ARRAY JOIN product_ids AS product_id
GROUP BY product_id
ORDER BY order_count DESC;
```

**Array + Map的设计收益**：

在大宽表中，传统的MySQL做法是为1:N关系建关联表（`order_items`），查询时需要JOIN。ClickHouse不擅长JOIN，但擅长宽表。用`Array(UInt32)`将商品ID直接存为数组列，一条SQL就能完成"某个商品被哪些订单包含"的查询，无需JOIN。`Map(String,String)`则替代了传统的JSON列——既能存储半结构化数据，又支持按Key查询，比`JSONExtractString`性能更好。

#### Step 5：类型转换实战

**目标**：掌握ClickHouse的类型转换函数体系，正确处理数据导入中的类型不匹配问题。

```sql
-- 基础CAST转换
SELECT
    CAST('123' AS Int32)             AS str_to_int,
    CAST(123 AS String)              AS int_to_str,
    CAST('2024-01-15' AS Date)       AS str_to_date,
    CAST('2024-01-15 10:30:00' AS DateTime) AS str_to_datetime,
    CAST(3.14159 AS Decimal(10,4))   AS float_to_decimal;

-- 专用类型转换函数（比CAST更明确，性能更好）
SELECT
    toInt64('9223372036854775807')    AS max_int,
    toUInt32('4294967295')            AS max_uint32,
    toFloat64('3.14159265358979')     AS pi_value,
    toString(255)                     AS num_to_str,
    toDate('2024-01-15')             AS date_val,
    toDateTime('2024-01-15 10:30:00', 'Asia/Shanghai') AS dt_val,
    toDecimal128('12345.6789', 4)    AS decimal_val;

-- 时间类型转换
SELECT
    toDate(created_at)               AS order_date,
    toYYYYMM(created_at)             AS year_month,
    toDayOfWeek(created_at)          AS day_of_week,
    formatDateTime(created_at, '%Y-%m-%d %H:%M:%S') AS formatted_time
FROM ecommerce.orders;

-- 安全的类型转换：避免转换失败导致整个查询报错
SELECT
    -- tryCast：转换失败时返回NULL
    tryCast('abc' AS Int32) AS safe_int,
    -- toInt32OrZero：转换失败时返回0
    toInt32OrZero('abc')    AS zero_fallback,
    -- toInt32OrDefault：转换失败时返回指定默认值
    toInt32OrDefault('abc', -1) AS default_fallback;

-- 常见坑：MySQL VARCHAR数字导入后的转换
-- 假设从MySQL导出的CSV中amount列是字符串格式'299.99'
-- 方案A：导入时直接用合适的格式
-- INSERT INTO orders FORMAT CSV ...
-- 方案B：如果已存为String类型，用MATERIALIZED列做自动类型转换
-- 注意：Decimal和Float之间的转换需谨慎
SELECT
    CAST('299.99' AS Decimal(10,2))  AS good_decimal,  -- 299.99 ✅
    CAST('299.99' AS Float32)        AS bad_float,      -- 299.99（但内部可能有精度误差）
    toDecimal64('299.99', 2)         AS good_via_func;  -- 299.99 ✅
```

**类型转换最佳实践**：

1. **导入前转换**优于导入后转换——在ETL阶段就确保数据类型正确。
2. **`toXxx()`系列函数**比`CAST`更快——它们是ClickHouse内置的专用函数。
3. **安全转换**：生产环境用`tryCast`或`toXxxOrZero`，避免脏数据导致查询崩溃。
4. **时区要注意**：`toDateTime(str, tz)`需要显式指定时区，否则使用服务器默认时区——这在跨机房部署时是常见陷阱。

## 4. 项目总结

### 优点 & 缺点：ClickHouse类型 vs MySQL类型

| 对比维度 | ClickHouse | MySQL |
|---------|-----------|-------|
| **整数类型** | 8种（UInt8~64, Int8~64），精确控制字节 | 5种（TINYINT~BIGINT），无无符号独立支持 |
| **浮点数** | Float32/Float64 + **Decimal(P,S)** 高精度定点 | FLOAT/DOUBLE + DECIMAL，精度上限低 |
| **字符串** | String(动态长) + LowCardinality(字典编码) | VARCHAR(N)需定长，TEXT大字段性能差 |
| **复合类型** | Array / Tuple / Nested / Map / JSON | 仅JSON，且不支持内部索引 |
| **NULL处理** | 额外UInt8标记列，有明显性能开销 | 1bit NULL位图，开销相对小 |
| **时间类型** | Date / DateTime / DateTime64(3/6/9) | DATE / DATETIME / TIMESTAMP |
| **压缩能力** | 列级Codec（LZ4/ZSTD/Delta），压缩比5~20x | 页级压缩（InnoDB），压缩比2~4x |
| **类型转换** | CAST + toXxx()系列40+函数，类型严格 | CAST + CONVERT，隐式转换更宽容 |

### 适用场景

| 类型选择 | 适用场景 | 不适用场景 |
|---------|---------|-----------|
| `LowCardinality(String)` | 枚举/状态/国家/设备类型等低基数列（<10万种） | 用户ID、URL、日志消息等高基数列 |
| `Decimal(P,S)` | 金额、费率、汇率等精确数值计算场景 | 科学计算、机器学习特征等可容忍浮点误差的场景 |
| `Array(T)` | 1:N的订单-商品、用户-标签等非规范化宽表 | 需要频繁更新的关系（Array不支持部分更新） |
| `Map(K,V)` | 动态属性（商品扩展字段、埋点extra数据） | 固定Schema的列（不如直接定义为独立列） |
| `DateTime64(3)` | 日志时间戳、交易时间、监控指标等高精度时间 | 日期维度（Date即可）或秒级精度就够的场景 |

### 注意事项

1. **永远不要用Float存储金额**：`Float32(299.99)`在ClickHouse内部可能变成`299.989990234375`，累加1亿次后误差可达数千元。金融级场景一律用`Decimal`。
2. **FixedString会填充NULL字节**：`FixedString(10)`存`'CHN'`实际是`'CHN\0\0\0\0\0\0\0'`，字符串比较和JOIN可能失败。除非存储二进制哈希值，否则忘掉它。
3. **LowCardinality有基数天花板**：当列中不同值超过100万时，字典查找开销会超过字符串直接比较的开销。可利用`SELECT uniqExact(col) FROM table`检查基数后再决策。
4. **DateTime默认使用服务器时区**：在多地域部署或容器化环境中，时区不一致会导致分区键计算错误（比如`toYYYYMM()`在UTC与CST下结果不同）。建表时建议显式指定时区，如`DateTime('Asia/Shanghai')`。
5. **Map类型不支持部分列更新**：`ALTER TABLE ... UPDATE extra_info['color'] = 'green' WHERE ...`这种操作代价等同全行重写，不适合频繁更新的场景。

### 常见踩坑经验

**坑1：Nullable列导致查询性能下降50%**

某团队将监控系统的`error_code`列定义为`Nullable(Int32)`，理由是"没有错误时这个字段就是NULL"。后来当查询`WHERE error_code > 0`时，ClickHouse需要先遍历NULL标记列再遍历数值列，查询延迟从60ms上升到95ms。修复方案：改为`Int32 DEFAULT -1`，用-1表示"无错误"。

**根因**：Nullable列在物理上由两个独立列组成，查询处理器需要处理双倍列数。在生产环境中，建议用哨兵值替代NULL。

**坑2：DateTime时区导致分区数据"穿越"**

某跨境电商团队部署在AWS东京区（JST, UTC+9）的ClickHouse，插入`created_at`数据后按`toYYYYMM(created_at)`分区。后来把服务器迁移到美西（PST, UTC-8），新插入的数据中，同一秒的北京时间23:00被算进了下一个月的Part，导致月度报表数据"凭空少了1小时"。

**根因**：`toYYYYMM(NOW())`依赖服务器时区。建表时用`DateTime('Asia/Shanghai')`锁定业务时区，或者用`toDate(created_at, 'Asia/Shanghai')`显式转换。

**坑3：LowCardinality字典"爆炸"**

某用户行为分析表中有`page_url`列，最初用了`LowCardinality(String)`——上线第一周表现良好。但随着页面数量从200个增长到50万个，查询`WHERE page_url = '/product/12345'`反而比普通String慢了3倍。排查发现，50万的字典在内存中占据约50MB，每次字典查找都是O(log N)的二分搜索，还不如直接字符串比较。

**根因**：LowCardinality的字典采用哈希表实现，基数超过10万后内存占用和查找开销急剧上升。建议监控`uniqExact()`的返回值，当基数突破10万时考虑切换回普通`String`，或者用`cityHash64()`将URL哈希后存为`UInt64`做精确匹配。
