# 第7章：主键与稀疏索引——查询加速的秘密

> **定位**：深入剖析ClickHouse主键与索引机制的本质差异。
> **核心内容**：稀疏索引原理与B+Tree对比、PRIMARY KEY与ORDER BY的关系、index_granularity调优、primary.idx与*mrk文件结构、左缀匹配原则、基数对索引效率的影响。
> **实战目标**：通过建表实验和EXPLAIN分析，直观感受稀疏索引的工作机制和调优效果。

---

## 1. 项目背景

双十一大促复盘会上，BI团队向全公司展示了一张实时大屏——用户行为分析看板。数十个指标卡片轮播，从PV/UV到转化漏斗到热力路径，数据源是那张已经攒了20亿行的`user_events`行为日志表。演示本来一切顺利，直到产品总监临时提了个需求："能不能帮我定位一下用户12345在1月15号那天的完整行为路径？"

BI同学在ClickHouse里敲下：

```sql
SELECT * FROM user_events 
WHERE user_id = 12345 AND event_date = '2025-01-15';
```

回车之后，看板空白了整整8秒。会议室里十几双眼睛盯着加载中的转圈，气氛尴尬得像在给投资人做DEMO时服务器宕机。

DBA紧急排查，扫了一眼建表语句就发现症结所在——

```sql
CREATE TABLE user_events (
    user_id UInt64,
    event_date Date,
    event_time DateTime,
    event_type String,
    page_url String,
    duration UInt32
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(event_date)
ORDER BY (event_date, event_time);
```

"你的PRIMARY KEY呢？" BI同学问。

"MergeTree里没写PRIMARY KEY，它默认就等于ORDER BY。"DBA摇头，"但这不是重点。重点是你们把`event_date`和`event_time`放在排序键首位，却用`user_id`做WHERE条件——**稀疏索引只认排序列的前缀**，这个查询根本用不上索引，每个分区都是全扫描，20亿行里硬搜。"

BI同学懵了："什么鬼？MySQL里PRIMARY KEY不就是建个B+Tree，按主键查就嗖的一下定位到行吗？ClickHouse的主键难道不是干这个的？"

这就是痛苦根源所在。**MySQL开发者对PRIMARY KEY的理解在ClickHouse里完全失效。** 在MySQL中，PRIMARY KEY承载三层语义：唯一约束、物理排序键、B+Tree索引。但在ClickHouse的MergeTree引擎里，PRIMARY KEY不保证唯一性，不提供行级索引，它只是一个**稀疏索引的骨架**——用来告诉查询引擎"这片数据区域大概包含哪些值"，以此决定"这片数据要不要读"。

更直白地说：MySQL的B+Tree主键索引是**精确定位到行**（Navigational），ClickHouse的稀疏索引是**粗略跳过数据块**（Skipping）。一个追求准，一个追求快。搞混这两者，轻则查询慢三五倍，重则慢一百倍——正如这场DEMO翻车事故。

小胖在群里感慨："早知道就不会拿着MySQL的思维写ClickHouse的DDL了。"大师回复："不怪你，99%的新人都会踩这个坑——他们把CREATE TABLE里那个`PRIMARY KEY`想当然地当成MySQL的索引来看。今天就专门把这层窗户纸捅破。"

## 2. 项目设计：剧本式交锋对话

**Scene**：数据团队会议室。白板上写着"PRIMARY KEY ≠ UNIQUE"几个大字。

**小胖（抱着一桶爆米花）**："大师，我先捋清楚。MySQL里我说`PRIMARY KEY (id)`，第一保证id不重复，第二数据按id排序存硬盘上，第三查`WHERE id=100`直接走B+Tree定位到那一页。ClickHouse倒好——`PRIMARY KEY (user_id)`建了之后，插入两条`user_id=12345`的行，它一声不吭全收进去了！这主键不去重，不是坑人吗？"

**大师**："你先回答我一个问题：你去图书馆查一本书，图书管理员是怎么找的？"

**小胖**："查索书号，索引卡片上是按分类+索书号排的，找到编号范围，去对应的书架找。"

**大师**："对。图书管理员不会给每一本书单独做一张索引卡——那得多少张？他给**每个书架**做一张索引卡，标上'这个书架上放的是索书号A100-A500的书'。你要找A230，他找到对应的书架，然后到书架上一本一本看。这就是稀疏索引。"

**小白（放下马克杯）**："所以稀疏索引的'稀疏'，意思是它不是每行一个索引条目，而是每个数据块一个索引条目？"

**大师**："完全正确。ClickHouse把数据按8192行切成一个颗粒（Granule）。排序列的数据落盘后，每个Granule里的第一行——或者说这个Granule的排序列的最小值——被拎出来组成一个索引文件`primary.idx`。查询的时候，引擎在这个小索引上做二分查找，确定哪些Granule可能包含目标数据，然后**只读那些Granule**。剩下的直接跳过。这就是**分区裁剪之外的第二个裁剪武器：颗粒裁剪（Granule Pruning）**。"

**技术映射**：B+Tree的索引条目数量 = 表行数（每个叶节点对应一行或多行，但粒度是行级），而ClickHouse稀疏索引的条目数量 = 表行数 / index_granularity（默认8192）。假设表有8亿行，B+Tree索引需要大约1.6亿个索引条目（含非叶节点）；而ClickHouse的primary.idx只需要约10万个条目。**索引文件大小差距可达三个数量级。**

---

**小胖**："等等，这跟PRIMARY KEY和ORDER BY的关系还没说清楚。我建表写`PRIMARY KEY (a, b)`跟`ORDER BY (a, b)`到底有什么区别？"

**大师**："这又是MySQL惯性带来的困惑。在MergeTree里，有三组密切相关的参数：`ORDER BY`是**数据物理排序的依据**；`PRIMARY KEY`是**稀疏索引包含哪些列**。如果只写了ORDER BY没写PRIMARY KEY，那么PRIMARY KEY就等于ORDER BY。如果PRIMARY KEY指定了，它必须是ORDER BY的**前缀**。举个例子——"

大师画了一个示意图：

```
ORDER BY (user_id, event_time, event_type)
PRIMARY KEY (user_id, event_time)

  primary.idx 只包含 (user_id, event_time) 列
  *.mrk 桥接到 *.bin 中的数据位置
  数据文件 *.bin 按 (user_id, event_time, event_type) 物理排序存储

  查询 WHERE user_id = 12345 AND event_time > '2025-01-01'
    → primary.idx 二分查找定位到包含 user_id=12345 的 Granule 范围
    → 通过 *.mrk 找到对应数据在 *.bin 中的偏移量
    → 只读这些 Granule，其他全部跳过
```

**大师**："大部分场景下，你不必把PRIMARY KEY写出来——就让PRIMARY KEY等于ORDER BY。除非ORDER BY列太多，主索引文件会变大，这时候可以只把ORDER BY的前几列设为主键，缩小索引文件。"

**小白**："那`index_granularity`又是啥？8192这个默认值有什么讲究？"

**大师**："这是控制稀疏索引'密度'的拧钮。8192的意思是：每隔8192行取一行作为索引标记。拧到256，索引密度就提高32倍，primary.idx文件大了32倍，但查询时颗粒裁剪更精细——点查询可能只需扫256行而不是8192行。拧到65536，索引极小，但一次扫描要多读8倍的无效数据。**这是一个空间换时间的连续调节旋钮。**"

**技术映射**：index_granularity = 8192意味着primary.idx条目数 ≈ 总行数/8192。如果表有10亿行，primary.idx大约有12.2万个标记值，文件大小仅约2-5MB（取决于键列类型），完全可以常驻内存。

---

**小胖**："那我还有个疑问——B+Tree支持二级索引、普通索引，ClickHouse呢？为什么没听说过二级索引？"

**大师**："好问题。ClickHouse没有传统意义上的二级B+Tree索引，因为列存引擎的架构决定了它在按主键排序之外建立另一个B+Tree索引的成本太高——每次INSERT都要维护多棵B+Tree，写入吞吐会崩塌。但ClickHouse提供了另一套方案：**Skip Index（跳数索引）**。它不索引到行，而是给每个Granule附加一个统计摘要。比如`bloom_filter`索引告诉引擎'这个Granule里有没有某个值'，`minmax`索引告诉引擎'这个Granule里值的范围'。查询时先通过Skip Index判断哪些Granule**不可能**包含目标数据，直接跳过。这跟B+Tree索引的思路完全不同——B+Tree是主动导航到目标，Skip Index是排除不可能的块。一种叫'Search Index'，一种叫'Pruning Index'。"

**小白**："那PRIMARY KEY列的基数对索引效率影响大吗？"

**大师**："非常大。假设ORDER BY (user_id, event_time)，如果user_id是高基数列，每个user_id只分布在少数几个Granule中——查询主键就能跳过99%的Granule。但如果第一列是低基数，比如event_type只有10个值，查询WHERE event_type='click'时会命中海量Granule，索引几乎等于没建。**稀疏索引的裁剪效果取决于排序列首列的基数和查询条件的过滤率。**首列基数越高，索引越有效。"

**小胖**（恍然大悟地放下爆米花）**："我全明白了！MySQL的PRIMARY KEY是一张高精度GPS导航地图，ClickHouse的PRIMARY KEY是一张告诉你在哪个书架的图书馆索引卡——粗粒度，但文件极小，适合批量扫描。"**

**大师**："比喻很到位。记住核心三句话：**稀疏索引不保证唯一性；稀疏索引只对ORDER BY前缀生效；稀疏索引靠跳过Granule来提速，不是靠精确定位来提速。**"

### 分步实现

#### Step 1：创建表并观察索引文件

**目标**：创建一个MergeTree表，插入数据后查看磁盘上生成了哪些索引相关的物理文件。

```sql
-- 创建用户行为表
DROP TABLE IF EXISTS user_events;

CREATE TABLE user_events (
    user_id UInt64,
    event_time DateTime,
    event_type String,
    page_url String,
    duration UInt32
) ENGINE = MergeTree()
ORDER BY (user_id, event_time)
PARTITION BY toYYYYMM(event_time);
```

> **注意**：这里没有显式写`PRIMARY KEY`，根据MergeTree规则，PRIMARY KEY默认等于ORDER BY，即`(user_id, event_time)`。

插入1000万行模拟数据：

```sql
INSERT INTO user_events 
SELECT 
    rand64() % 100000 AS user_id,
    toDateTime('2025-01-01 00:00:00') + rand() % (86400 * 365) AS event_time,
    ['click', 'view', 'scroll', 'submit', 'share'][rand() % 5 + 1] AS event_type,
    concat('/page/', toString(rand() % 5000)) AS page_url,
    rand() % 300 AS duration
FROM numbers(10000000);
```

> **插入耗时参考**：1000万行通常需要15-40秒，取决于机器性能。如果机器配置较低，可以调小`numbers()`的参数到200万。

现在退出ClickHouse客户端，去数据目录看看物理文件：

```bash
docker exec -it ch01-clickhouse bash -c "ls -lh /var/lib/clickhouse/data/analytics/user_events/"
```

预期看到类似下面的输出：

```
202501_1_1_0/
202502_1_1_0/
...
```

进入其中一个分区目录：

```bash
docker exec -it ch01-clickhouse bash -c "find /var/lib/clickhouse/data/analytics/user_events/*/ -type f | sort"
```

关键文件：

```
column.bin        ← 列数据文件（如 user_id.bin, event_time.bin）
column.mrk2       ← Mark标记文件（v2格式），桥接索引和数据
primary.idx       ← 主键稀疏索引文件
checksums.txt     ← 文件校验和
count.txt         ← 行数统计
```

**文件职责速查**：

| 文件 | 角色 | 内容 |
|------|------|------|
| `primary.idx` | 稀疏索引 | 每8192行（默认）取一行，记录该Granule中ORDER BY列的值 |
| `*.mrk2` | Mark标记文件 | 记录每个Granule在对应`*.bin`文件中的起始偏移量和压缩块位置 |
| `*.bin` | 列数据文件 | 压缩存储的列数据，按ORDER BY顺序排列 |
| `skp_idx_*.idx` | Skip Index索引 | 按Granule存储的附加统计摘要（本章暂不展开） |

> **踩坑提醒**：如果你的目录里看不到`primary.idx`文件，检查一下是否真的有数据写入且Merge完成。没有数据写入的分区不会产生`primary.idx`。

#### Step 2：用EXPLAIN分析索引命中

**目标**：通过`EXPLAIN indexes=1`直观看到哪些查询能命中稀疏索引，哪些不能。

重新进入ClickHouse客户端，执行：

```sql
-- 查询1：命中索引（使用ORDER BY首列 user_id）
EXPLAIN indexes=1 
SELECT * FROM user_events WHERE user_id = 12345;
```

预期输出关键信息：

```
┌─explain─────────────────────────────────────────────┐
│ Expression ((Projection + Before ORDER BY))          │
│   Filter: (WHERE ...)                                │
│     ReadFromMergeTree                                │
│     Parts: 12                                       │
│     Granules: 1200 ← 存在的总颗粒数                   │
│     Granules selected: 4  ← 实际需要读取的颗粒数!!     │
│       └─ Primary key: (user_id: [12345, 12345])      │
│          Condition: (user_id in [12345, 12345])       │
│          Parts: 4/12, Granules: 4/1200               │
└──────────────────────────────────────────────────────┘
```

> 输出会因实际数据分布略有差异。关键看`Granules selected`——如果远小于`Granules`总数，说明**索引生效，大部分Granule被跳过了**。

```sql
-- 查询2：无法命中索引（跳过ORDER BY首列，直接用第二列）
EXPLAIN indexes=1 
SELECT * FROM user_events WHERE event_time = '2025-01-15 10:00:00';
```

预期输出关键信息：

```
┌─explain─────────────────────────────────────────────┐
│   Filter: (WHERE ...)                                │
│     ReadFromMergeTree                                │
│     Parts: 12                                       │
│     Granules: 1200                                  │
│     Granules selected: 1200  ← 全部Granule都要读!!    │
│       └─ Primary key: unknown                        │
│          Condition: (event_time ...)                 │
│          Parts: 12/12, Granules: 1200/1200           │
└──────────────────────────────────────────────────────┘
```

> `Granules selected: 1200` 意味着**每个Granule都要扫描**，稀疏索引完全失效。因为稀疏索引按`user_id`排序，而`event_time`在同一个`user_id`范围内是无序的——引擎无法推断哪些Granule不含目标`event_time`值。

**这就是左缀匹配原则在ClickHouse中的严格体现**：稀疏索引只能利用ORDER BY的前缀列。ORDER BY (a, b, c) 时，以下查询能用到索引——

| 查询条件 | 是否命中索引 | 原因 |
|---------|------------|------|
| `WHERE a = 1` | ✅ | 命中最左列 |
| `WHERE a = 1 AND b = 2` | ✅ | 命中前两列 |
| `WHERE a = 1 AND c = 3` | ✅ (部分) | a命中，c不命中但a已经裁剪了 |
| `WHERE b = 2` | ❌ | 跳过了首列，索引无法定位 |
| `WHERE b = 2 AND c = 3` | ❌ | 同样跳过首列 |
| `WHERE a > 1 AND a < 100` | ✅ | 范围查询也能利用首列索引 |

#### Step 3：index_granularity 调优实验

**目标**：对比不同`index_granularity`对索引文件大小和查询性能的影响。

```sql
DROP TABLE IF EXISTS events_granule_8192;
DROP TABLE IF EXISTS events_granule_256;

-- 表A：默认颗粒度 8192
CREATE TABLE events_granule_8192 (
    user_id UInt64,
    event_time DateTime,
    value Float64
) ENGINE = MergeTree()
ORDER BY (user_id, event_time)
PARTITION BY toYYYYMM(event_time)
SETTINGS index_granularity = 8192;

-- 表B：细粒度索引 256
CREATE TABLE events_granule_256 (
    user_id UInt64,
    event_time DateTime,
    value Float64
) ENGINE = MergeTree()
ORDER BY (user_id, event_time)
PARTITION BY toYYYYMM(event_time)
SETTINGS index_granularity = 256;
```

插入同样的500万行数据：

```sql
INSERT INTO events_granule_8192 
SELECT 
    rand64() % 50000 AS user_id,
    toDateTime('2025-06-01 00:00:00') + rand() % (86400 * 180) AS event_time,
    rand() / 1000000.0 AS value
FROM numbers(5000000);

INSERT INTO events_granule_256 
SELECT * FROM events_granule_8192;
```

对比索引文件大小：

```bash
docker exec -it ch01-clickhouse bash -c "
  echo '=== index_granularity=8192 ==='
  ls -lh /var/lib/clickhouse/data/analytics/events_granule_8192/*/primary.idx
  echo '=== index_granularity=256 ==='
  ls -lh /var/lib/clickhouse/data/analytics/events_granule_256/*/primary.idx
"
```

预期结果（示意）：

```
=== index_granularity=8192 ===
-rw-r----- 1 clickhouse clickhouse  28K ... primary.idx
-rw-r----- 1 clickhouse clickhouse  32K ... primary.idx
=== index_granularity=256 ===
-rw-r----- 1 clickhouse clickhouse 880K ... primary.idx
-rw-r----- 1 clickhouse clickhouse 920K ... primary.idx
```

> **结论**：`index_granularity=256`的索引文件约为`8192`的32倍（8192/256 = 32），因为索引条目增多32倍。

现在对比点查性能：

```sql
-- 对比1：点查询（查特定user_id）
EXPLAIN indexes=1 
SELECT count() FROM events_granule_8192 WHERE user_id = 12345;

EXPLAIN indexes=1 
SELECT count() FROM events_granule_256 WHERE user_id = 12345;
```

预期观察：
- `events_granule_8192`：可能会读1-2个Granule（扫描8192-16384行）
- `events_granule_256`：可能会读1-2个Granule（但只扫描256-512行）

```sql
-- 对比2：范围查询（查一段时间）
EXPLAIN indexes=1 
SELECT count() FROM events_granule_8192 
WHERE user_id BETWEEN 10000 AND 20000 
  AND event_time BETWEEN '2025-07-01' AND '2025-07-31';

EXPLAIN indexes=1 
SELECT count() FROM events_granule_256 
WHERE user_id BETWEEN 10000 AND 20000 
  AND event_time BETWEEN '2025-07-01' AND '2025-07-31';
```

**观察与结论**：
- **点查询**：细粒度索引（256）扫描的行数更少，但差异在1万行级别内，对小数据量查询影响不明显
- **全表范围扫描**：两种`index_granularity`的扫描总量接近，`Granules selected`差异不大——因为范围查询本身就需要扫描大量Granule，细粒度索引的优势被稀释
- **trade-off**：`index_granularity`越小→索引更精确→点查更快，但`primary.idx`更大→占用更多内存/文件缓存；反之亦然。默认8192是广泛的工程平衡点

#### Step 4：主键基数对索引效率的影响

**目标**：通过实验验证排序列首列基数如何影响稀疏索引的裁剪效率。

```sql
DROP TABLE IF EXISTS events_low_card;
DROP TABLE IF EXISTS events_high_card;

-- 表A：低基数列为首列（event_type只有5个值）
CREATE TABLE events_low_card (
    event_type String,      -- 'click','view','scroll','submit','share'
    user_id UInt64,
    event_time DateTime
) ENGINE = MergeTree()
ORDER BY (event_type, user_id, event_time);

-- 表B：高基数列为首列（user_id有50000个不同值）
CREATE TABLE events_high_card (
    user_id UInt64,
    event_type String,
    event_time DateTime
) ENGINE = MergeTree()
ORDER BY (user_id, event_type, event_time);

-- 插入相同数据
INSERT INTO events_low_card SELECT 
    ['click','view','scroll','submit','share'][rand() % 5 + 1] AS event_type,
    rand64() % 50000 AS user_id,
    now() - rand() % 86400 * 30 AS event_time
FROM numbers(5000000);

INSERT INTO events_high_card SELECT user_id, event_type, event_time FROM events_low_card;
```

对比两个查询的Granule扫描量：

```sql
-- 在低基数字段表上查询（event_type='click'）
EXPLAIN indexes=1 
SELECT count() FROM events_low_card WHERE event_type = 'click';

-- 在高基数字段表上查询（user_id=12345）
EXPLAIN indexes=1 
SELECT count() FROM events_high_card WHERE user_id = 12345;
```

预期结果：

```
--- events_low_card (WHERE event_type='click') ---
Granules: 610    ← 总颗粒数
Granules selected: 122  ← 'click'约占20%的数据，索引也只能跳过80%

--- events_high_card (WHERE user_id=12345) ---
Granules: 610
Granules selected: 1    ← user_id=12345高度集中在1个颗粒中，跳过99.8%
```

> **结论**：稀疏索引的裁剪效果 = 排序列首列的区分度。**首列基数越高 → 等值查询命中的Granule越少 → 裁剪效果越好。** 反过来，用低基数列做首列（如event_type、status），稀疏索引几乎形同虚设。

#### Step 5：理解primary.idx的物理结构

**目标**：理解稀疏索引在磁盘上的二进制组织方式及其查找算法。

虽然不能直接用文本打开二进制的`primary.idx`文件，但我们可以用ClickHouse内置的工具来理解其结构。先看一下索引的元信息：

```sql
-- 通过系统表查看索引粒度
SELECT 
    name,
    sorting_key,
    primary_key,
    index_granularity
FROM system.tables 
WHERE database = 'analytics' AND name = 'user_events'
FORMAT Vertical;
```

```sql
-- 查看具体的Part及其标记数量
SELECT 
    partition,
    name AS part_name,
    rows,
    marks_count,
    round(rows / marks_count, 1) AS rows_per_mark
FROM system.parts
WHERE database = 'analytics' 
  AND table = 'user_events'
  AND active = 1
ORDER BY partition;
```

预期输出：

```
partition | part_name      | rows      | marks_count | rows_per_mark
202501    | 202501_1_1_0   | 849152    | 104         | 8164.9
202502    | 202502_2_2_0   | 767861    | 94          | 8168.7
...
```

> `marks_count`就是每个Part的索引条目数量，它乘以`index_granularity`（8192）大致等于该Part的行数（最后一个Granule可能不满8192行）。

**primary.idx的查找伪代码**：

```
FUNCTION find_matching_granules(condition: WHERE user_id = 12345):

    matching_ranges = []
    
    FOR EACH part IN table.active_parts:
        // 1. 加载该Part的primary.idx到内存
        //    primary.idx 结构: [granule_0_min, granule_1_min, ..., granule_N_min]
        //    每个条目是 (user_id, event_time) 二元组
        
        idx_entries = load_primary_idx(part) // 例如: [(10, Jan1), (50, Jan3), (100, Jan5), ...]
        
        // 2. 二分查找：找到第一个 user_id >= 12345 的 Granule
        start_granule = binary_search_lower_bound(idx_entries.user_id, 12345)
        
        // 3. 找到最后一个 user_id <= 12345 的 Granule
        end_granule = binary_search_upper_bound(idx_entries.user_id, 12345)
        
        // 4. 记录需要扫描的Granule范围
        IF start_granule <= end_granule:
            // 通过 *.mrk2 文件解析每个Granule在 *.bin 中的物理偏移
            matching_ranges.push( (part, start_granule, end_granule) )
    
    // 5. 只读 matching_ranges 中指定的Granule数据，其余全部跳过
    RETURN matching_ranges
```

**关键理解**：`primary.idx`中存的是每个Granule的**起始值**（即该Granule中ORDER BY列的最小值/首行值），而不是这个Granule中所有值的完整列表。查询引擎利用这些起始值判断"如果这个Granule的第一个user_id已经大于12345，说明12345不在这个Granule及之后的Granule中"——这就是二分查找能工作的基础。

**Mark文件（`*.mrk2`）的作用**：

```
primary.idx → 逻辑定位："目标数据在第3到第5个Granule"
    ↓
*.mrk2     → 物理定位：第3个Granule在user_id.bin偏移量4096字节处，event_time.bin偏移量4096字节处
    ↓
*.bin      → 读取数据
```

Mark文件里每个条目存储两个数字：`(offset_in_compressed_file, offset_in_decompressed_block)`，告诉引擎从哪个字节开始读、读多少。**没有*.mrk文件，即使primary.idx知道是哪个Granule，也找不到这个Granule在*.bin文件中的物理位置。**

```sql

-- 不指定user_id → 全表扫描
EXPLAIN indexes=1 
SELECT count() FROM user_events;

-- 指定user_id → 只扫描少数Granule
EXPLAIN indexes=1
```

2. 不含`user_id`前缀的查询`Granules selected`等于总数（全扫描）
3. 重复`user_id`插入不报错，PRIMARY KEY不强制唯一

## 4. 项目总结

### 稀疏索引 vs B+Tree 对比表

| 维度 | 稀疏索引（ClickHouse） | B+Tree（MySQL/PostgreSQL） |
|------|----------------------|---------------------------|
| 索引粒度 | Per Granule（默认8192行/条目） | Per Row（每行一个叶节点条目） |
| 索引文件大小 | 极小（10亿行 ≈ 2-5MB） | 较大（10亿行 ≈ 数十GB） |
| 点查询（等值） | 扫描1-2个Granule（最多16384行） | O(log n) 精确定位到行 |
| 范围查询 | 优秀（一次定位后顺序扫描连续Granule，磁盘顺序读） | 良好（叶节点链表遍历，可能有随机IO） |
| 聚合查询 | 极优（可直接跳到Granule边界开始聚合） | 一般（需要遍历叶节点） |
| 唯一性约束 | 不保证 | 保证 |
| 写入维护成本 | 极低（追加写入，索引只记首行值） | 较高（每次INSERT/UPDATE需维护树结构） |
| 内存占用 | 极小（常驻内存无压力） | 需要Buffer Pool缓存索引页 |

### 适用场景

**稀疏索引的三大主场**：

1. **时间序列范围扫描**：`WHERE ts BETWEEN '2025-01-01' AND '2025-01-31'`——稀疏索引直接定位起始Granule，然后顺序扫后续Granule，磁盘顺序读性能拉满

2. **有序维度的聚合分析**：`GROUP BY user_id`——数据已按user_id物理排序，相同user_id的行集中在相邻的Granule中，聚合计算缓存友好

3. **日志/事件流的过滤查询**：`WHERE app_id = 'xxx' AND event_time > '2025-06-01'`——稀疏索引先按app_id裁剪，再按event_time裁剪，两个维度叠加裁剪效果

**B+Tree的更优场景**：

1. **高频单行查询**：`SELECT * FROM orders WHERE order_id = 12345678`——OLTP的典型场景，B+Tree O(log n) 定位到一行远优于稀疏索引扫描8192行

2. **需要唯一约束的场景**：用户表、订单表的主键必须具备唯一性

### 注意事项

1. **PRIMARY KEY不强制唯一性**——这是对MySQL开发者最大的认知颠覆。如果需要去重，请使用`ReplacingMergeTree`引擎或应用层去重，不要指望主键帮你拒绝重复行

2. **ORDER BY列的顺序就是一切**——决定哪些查询能用到稀疏索引、哪些不能。设计ORDER BY时，必须把最常用的等值过滤列放在最前面，范围过滤列放在中间

3. **不要用UUID作为ORDER BY的首列**——UUID是随机无序的，完全不具备数据局部性。同一个用户的两次行为可能在完全不搭界的Granule里，稀疏索引对UUID列的裁剪效果为零

4. **index_granularity不是越低越好**——设为1会让索引变得和B+Tree一样大，失去了稀疏索引的核心优势。调优应基于实际负载在点查精度和索引尺寸之间找平衡

### 常见踩坑经验

**坑1：把PRIMARY KEY当作唯一约束导致重复数据**

某用户画像团队建表`PRIMARY KEY (user_id)`，以为跟MySQL一样能防止重复。结果同一天内多次ETL导入产生大量重复user_id，下游报表展示"用户数"比真实值高出30%。根因：MergeTree的PRIMARY KEY不保证唯一。解决方案：导入前DISTINCT去重，或者使用`ReplacingMergeTree + FINAL`在查询时去重。

**坑2：用随机ID做ORDER BY首列导致索引完全失效**

某团队创建`ORDER BY (request_id, timestamp)`，其中`request_id`是UUID生成的随机字符串。所有查询都带`timestamp`范围条件，但因为排序列首列是随机ID，稀疏索引完全无法利用timestamp做裁剪——timestamp在同一个request_id下是无序的。根因：不了解左缀匹配原则。修复方案改为`ORDER BY (toDate(timestamp), request_id)`。

**坑3：index_granularity设置过大导致点查扫描过多无用行**

某监控团队为了节省存储空间，把`index_granularity`设为65536。点查`WHERE host = 'web-server-01'`时需要扫描65K行才能找到目标行，导致API接口P99延迟升至2秒。根因：`index_granularity`过大致使每个Granule太大，点查效率雪崩。调回8192后P99降至50ms。

**坑4：低基数列排第一，稀疏索引名存实亡**

日志表`ORDER BY (log_level, timestamp)`，业务查询永远是`WHERE timestamp BETWEEN ...`。因为`log_level`只有5个值（INFO/WARN/ERROR等），作为排序列首列没有任何裁剪效果——它只能把几十亿行分成5个大块。根因：不了解基数对稀疏索引效率的影响。正确设计是`ORDER BY (timestamp, log_level)`。
