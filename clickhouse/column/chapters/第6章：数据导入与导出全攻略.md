> **版本**：ClickHouse 25.x LTS
> **定位**：基础篇核心章节。数据进得来、出得去，ClickHouse 才算真正跑通。
> **前置阅读**：第4章（MergeTree 家族概览）、第5章（分区与排序键）

## 1. 项目背景

某视频网站运维团队接手了一个令人头疼的任务：需要将过去两年累积的 Nginx 访问日志全部导入 ClickHouse，用于日常分析和故障排查。这些日志分散在 200 多个 CSV 文件中，总数据量约 **5TB**，压缩后仍有 2TB。每条日志包含时间戳、请求方法、URI、状态码、响应时间、User-Agent、源 IP 等 20 多个字段——典型的大批量、宽表导入场景。

小胖接到任务后的第一反应是："不就是把 CSV 导进去嘛？Python 写个循环，读一行 `INSERT ... VALUES` 一行，多简单。"他用 pymysql 写了个脚本，单行逐条插入，跑了一晚上——第二天早晨一看，才导完了 3%，按这个速度需要 **超过三天**才能全部完成。更糟的是，凌晨的服务器的内存告警响了：pymysql 的逐行 INSERT 把 ClickHouse 内存撑到了 85%，服务险些 OOM。

他赶紧换了个思路：用 Python 的 `executemany` 批量插入，结果发现有些行包含特殊字符——User-Agent 字段里嵌了换行符和引号，CSV 解析直接裂开，整批数据因一行出错而全部回滚。他又试着用 `LOAD DATA LOCAL INFILE`，结果字段顺序和表结构对不上，状态码被写进了时间戳列，DateTime 格式不匹配导致 2000 万行写入失败。

另一边，数据科学团队在催："导入后的聚合结果能导出成 Parquet 吗？我们下游用 Spark 做模型训练，不支持 ClickHouse 直连。"小胖用 `clickhouse-client --format CSV` 导了一份，数据科学团队反馈："CSV 没有 Schema 信息，状态码是整数还是字符串？时间戳是秒还是毫秒？每次都要人工对齐，太容易出错了。"

**核心痛点**：

- **导入慢**：逐行 `INSERT ... VALUES` 每次一条网络往返 + 一行解析，吞吐量惨不忍睹。
- **格式坑**：CSV 的换行符、引号、逗号与字段内容冲突，一行错导致整批失败；DateTime 格式不匹配直接拒绝写入。
- **大文件 OOM**：将整个大文件加载到内存再做 INSERT，单次插入数据量过大导致内存溢出。
- **导出格式不兼容**：CSV 无 Schema 元信息，下游 Spark/Flink 无法自动推断列类型；用错格式导致下游管道断裂。

本项目的目标非常明确：**建立一套高效、可靠的数据导入与导出方法论**——选对传输方式、选对数据格式、控制好批次大小、处理好异常行，让 5TB 日志在可控时间内完整入库，同时保证导出结果能被下游工具无缝消费。

## 2. 项目设计：剧本式交锋对话

会议室桌上摊着一张 A4 纸，上面画满了小胖尝试过的各种导入方案——每个方案旁边都打了一个大大的叉号。小白端着一杯美式咖啡走进来，大师紧跟其后，手里拿着一个 U 盘。

**小胖**（指着满纸的红叉）："我不理解。不就是把 CSV 导进 ClickHouse 吗？我一个 `INSERT ... VALUES` 循环不就搞定了？MySQL 我从来都是这么干的，怎么到 ClickHouse 这儿就三天三夜跑不完？"

**大师**（拉过白板）"小胖，你算一笔账。`INSERT ... VALUES (row1); INSERT ... VALUES (row2); ...`——每一条 SQL 都是一次完整的网络往返。客户端打包请求 → 网络传输 → 服务端解析 SQL → 写 Block → 落盘 → 返回响应。你的服务器延迟按 1ms 算，5 亿行就是 50 万秒，将近 6 天。这还不算服务端每次解析 SQL 的 CPU 开销。"

**小胖**（掰手指）"那我把多条 VALUES 拼一起呢？`INSERT INTO t VALUES (row1), (row2), (row3)...`"

**大师**："方向对了，但有上限。ClickHouse 的 `INSERT ... VALUES` 单次请求有一个 `max_insert_block_size` 限制，默认 1048576 行（约 100 万行）。但是——如果你一次拼 100 万行 VALUES 进去，客户端先把这 100 万行全部序列化到内存里，然后一股脑发给服务端。数据量一大，**客户端先 OOM**。而且如果这 100 万行中有一行格式不对，整批全部丢弃。"

**技术映射 #1**：`INSERT ... VALUES` 适合少量测试数据或维表维护（几百到几千行）。生产级大批量导入有两条更高效的路径：**流式管道导入**（clickhouse-client pipe）和 **表函数直接读取**（file/s3/url 等）。

---

**小白**（放下咖啡杯）"那正确的大批量导入方式是什么？我听说有什么'管道导入'，还有用 `FORMAT` 的，这两个有什么区别？"

**大师**："ClickHouse 的数据导入有'三步走'的路线图：

**第一层：本地批量写入**。`INSERT INTO ... SELECT ... FROM numbers()` ——直接在 ClickHouse 内部生成数据再写入，零网络开销，但需要编写 SELECT 逻辑构造数据。适合测试数据和 ETL 内部数据转换。

**第二层：客户端管道导入**。这是生产最常用的方式：

```sql
cat /data/logs.csv | clickhouse-client --query="INSERT INTO nginx_logs FORMAT CSV"
```

关键在于 `FORMAT CSV`——不是 `VALUES`。ClickHouse 的 Native Protocol 支持流式解析：客户端把 CSV 数据流式推送过去，服务端边收边解析边写 Block，满 8192 行（一个 Granule）就落盘一个 Part。整个过程是**流水线化的**——解析和写入并行，客户端和服务端的内存都不会膨胀。5TB 数据不会一次性加载，而是在流里持续消耗。"

**技术映射 #2**：`FORMAT` 是 ClickHouse 的流式数据协议——服务端不需要等到接收完所有数据才开始处理，而是一边读一边写。这从根本上避免了大文件导致的内存爆炸。

**小胖**（眼睛亮了）"哦！所以管道导入天然就是流式的，不会 OOM？"

**大师**："对。但它也有缺点——客户端中断了，已写入的数据不会回滚（已经落成 Part 了）。如果文件中间有一行格式错误，后面所有行都会丢失。所以正式导入前，建议先用 `head -100` 采样，用 `INSERT INTO ... SELECT ... FROM file()` 表函数试读几行，确认格式正确再全量导入。"

---

**小白**（翻开笔记本，开始记录）"那第三个层级呢？你说什么表函数直接读写文件？"

**大师**（在白板上画了一个文件图标，旁边画了一个表图标，中间画了一个等号）：

```
┌──────────┐       ┌─────────────────┐
│  磁盘文件  │  <=>  │  file() 表函数   │  <=>  SQL 查询
│ /data/*.csv│       │ (零拷贝/极低拷贝) │
└──────────┘       └─────────────────┘
```

"这是 ClickHouse 最强大的特性之一——**文件表函数**。你不需要把数据导入到 ClickHouse 表里才能查询，而是**直接把文件当成表来查**：

```sql
SELECT count(), avg(status)
FROM file('/data/logs.csv', 'CSV',
  'timestamp DateTime, level String, message String, status UInt16')
WHERE status >= 500;
```

`file()` 函数返回一个虚拟表，ClickHouse 直接扫描磁盘上的 CSV 文件，边读边解析边聚合。**不需要任何导入步骤**。同理，`s3()` 表函数可以直接查 S3 上的 CSV/Parquet 文件，`url()` 可以查 HTTP 上的数据，`hdfs()` 可以查 HDFS——省去了'先下载再导入'的中间环节。"

**小胖**（兴奋）"这也太酷了！那我直接把 5TB 的 CSV 文件挂到 file() 上查不就行了？还导入个啥？"

**大师**："file() 是即席查询利器，但不适合反复查询。它每次都要重新解析 CSV——CSV 是文本格式，没有索引，没有压缩的列存文件，5TB 的数据每次全扫一遍，谁也受不了。正确姿势是：**先用 file() 确认数据格式正确，再通过 INSERT INTO ... SELECT FROM file() 一次性导入到 MergeTree 表**。导入后 CSV 就可以删了，后续所有查询都享受列存压缩和稀疏索引的加速。"

**技术映射 #3**：文件表函数是"只读视图"——零存储成本，即刻可用，适合一次性分析和数据探索。MergeTree 是"物化存储"——有导入成本，但后续查询极快。两者是互补关系，不是替代关系。

---

**小白**："那数据导出呢？数据科学团队要 Parquet，运营团队要 Excel CSV，我总不能每个格式写一套导出脚本吧？"

**大师**："ClickHouse 支持丰富的导出格式——而且是**双向对称**的。数据怎么吃进来的，就可以怎么吐出去：

- **CSV / TSV**：人类可读，Excel 能打开。缺点——无 Schema、无类型信息、不能嵌套。
- **JSONEachRow**：每行一个 JSON 对象，下游工具（Python、Node.js）解析方便。缺点——体积大，压缩比差。
- **Parquet**：列存二进制格式，带完整 Schema，Spark/Flink/Pandas 原生支持。**这是数据科学导出首选**。
- **Arrow (ArrowStream / ArrowIPC)**：内存列存格式，适合进程间零拷贝数据交换。Python pandas 2.0+ 原生支持。
- **Avro**：行存二进制格式，带 Schema 演化能力，Kafka 生态常用。
- **ORC**：Hive 生态常用列存格式，Hadoop 系工具优先选择。
- **Native**：ClickHouse 内部二进制协议，跨 ClickHouse 实例传输效率最高。

导出的方式有两种。第一种，用 `INTO OUTFILE`：

```sql
SELECT level, count()
FROM nginx_logs
GROUP BY level
INTO OUTFILE '/var/lib/clickhouse/user_files/report.parquet'
FORMAT Parquet;
```

第二种，用 `INSERT INTO FUNCTION file()` ——把 SELECT 的结果'插入'到文件里：

```sql
INSERT INTO FUNCTION file('/tmp/report.parquet')
SELECT level, count() FROM nginx_logs GROUP BY level;
```

这两种效果一样——都是把查询结果按指定格式写入文件。哪种顺手用哪种。"

**技术映射 #4**：ClickHouse 的 FORMAT 体系是'即插即用'的——同一个 SQL，换一个 FORMAT 参数，输出完全不同。这保证了 ClickHouse 可以无缝嵌入任何数据管道。

---

**小胖**（挠头）"那 Kafka 呢？我们日志是实时推给 Kafka 的，能不能直接让 ClickHouse 从 Kafka 里读？省得每天手动导入 CSV。"

**大师**："好问题。这是第四种导入方式——**集成表引擎**。ClickHouse 内置了 Kafka 引擎、RabbitMQ 引擎、NATS 引擎，它们把消息队列当成一张'虚拟表'来看待：

```sql
CREATE TABLE kafka_logs_queue (
    timestamp DateTime, level String, message String, status UInt16
) ENGINE = Kafka()
SETTINGS
    kafka_broker_list = 'kafka:9092',
    kafka_topic_list = 'nginx-logs',
    kafka_group_name = 'ch_consumer',
    kafka_format = 'JSONEachRow';
```

然后建一张物化视图，自动把 Kafka 消息拉取 → 解析 → 写入 MergeTree 表。整个过程是**事件驱动的，实时持续运行**，无需任何外部调度脚本。"

**小白**（记完最后一笔）"所以总结一下——导入有四条路：本地 SELECT 写、管道 FORMAT 导入、文件表函数读、消息引擎消费。导出有两条路：INTO OUTFILE 和 INSERT INTO FUNCTION file()。选对路，事半功倍。"

**大师**："精准。记住了——**别一行一行 INSERT ... VALUES，那是'用勺子舀干一个湖'**。"

## 3. 项目实战

### 环境准备

启动 ClickHouse 并挂载本地数据卷，以便 ClickHouse 能直接读取宿主机上的文件：

```bash
# 创建数据共享目录
mkdir -p /tmp/clickhouse-data

# 启动 ClickHouse，将 /tmp/clickhouse-data 挂载到容器内
docker run -d --name ch06 \
  -p 8123:8123 -p 9000:9000 \
  -v /tmp/clickhouse-data:/var/lib/clickhouse/user_files \
  clickhouse/clickhouse-server:25.4-alpine

# 进入客户端
docker exec -it ch06 clickhouse-client
```

> **踩坑提醒**：`user_files` 目录是 ClickHouse 执行 `file()` 表函数和 `INTO OUTFILE` 的默认根目录。所有文件路径都必须是该目录下的相对路径（安全沙箱机制）。Windows 挂载时注意路径分隔符和权限问题。

```sql
CREATE TABLE nginx_logs (
```

#### Step 2：管道导入 CSV——生产级标准姿势

这是批量导入 CSV 最常用的方式。注意 `FORMAT CSV` 而非 `VALUES`：

```bash
# 退出 clickhouse-client，回到 bash
cat /var/lib/clickhouse/user_files/nginx_logs.csv \
  | clickhouse-client --query="INSERT INTO nginx_logs FORMAT CSV"

# 验证导入结果
clickhouse-client --query="SELECT count() FROM nginx_logs"
```

管道的原理是：`cat` 将文件内容逐块（通常 64KB buffer）发送到标准输出 → 管道将数据传递给 `clickhouse-client` 的标准输入 → clickhouse-client 将数据流式推送到 ClickHouse Server → Server 边收边解析，攒够 Granule 就落盘 Part。整个过程**内存占用恒定**，不随文件大小增长。

```sql
-- 查看导入后的 Part 分布
SELECT
    partition,
    count() AS parts,
    sum(rows) AS total_rows
FROM system.parts
WHERE table = 'nginx_logs' AND active = 1
GROUP BY partition
ORDER BY partition;
```

#### Step 3：使用表函数直接查询文件——免导入分析

`file()` 表函数让你无需导入就能查询磁盘上的 CSV/Parquet/JSON 文件：

```sql
-- 直接查询 CSV 文件（不导入）
SELECT count(), avg(status)
FROM file('nginx_logs.csv', 'CSV',
    'timestamp DateTime, level String, message String, status UInt16');
```

注意：列类型定义字符串 `'timestamp DateTime, level String, message String, status UInt16'` 必须与文件中的列顺序和类型严格一致。如果文件中包含表头行，需要额外指定 `SETTINGS format_csv_delimiter=','` 等参数。

```sql
-- 查看 5xx 错误的前 10 条
SELECT *
FROM file('nginx_logs.csv', 'CSV',
    'timestamp DateTime, level String, message String, status UInt16')
WHERE status >= 500
ORDER BY timestamp DESC
LIMIT 10;
```

也可以使用通配符一次查询多个文件：

#### Step 4：Parquet 格式导出——数据科学标准输出

将分析结果导出为 Parquet 格式，供下游 Spark/Pandas 消费：

```sql
-- 方式一：INSERT INTO FUNCTION file()
INSERT INTO FUNCTION file('nginx_report.parquet')
SELECT
    level,
    count()   AS cnt,
    avg(status) AS avg_status
FROM nginx_logs
GROUP BY level
ORDER BY cnt DESC;
```

```sql
-- 方式二：SELECT ... INTO OUTFILE
SELECT
    level,
    count()   AS cnt,
    avg(status) AS avg_status
FROM nginx_logs
GROUP BY level
ORDER BY cnt DESC
INTO OUTFILE 'nginx_report_v2.parquet'
FORMAT Parquet;
```

验证导出的 Parquet 文件：

```bash
ls -lh /var/lib/clickhouse/user_files/nginx_report.parquet
```

**格式对比实验**——同样的数据，不同格式的体积差异：

```sql
-- 导出 CSV 对比
INSERT INTO FUNCTION file('nginx_report.csv')
SELECT level, count() AS cnt
FROM nginx_logs GROUP BY level
FORMAT CSV;

-- 导出 JSONEachRow 对比
INSERT INTO FUNCTION file('nginx_report.json')
SELECT level, count() AS cnt
FROM nginx_logs GROUP BY level
FORMAT JSONEachRow;

-- 导出 Arrow IPC 对比
INSERT INTO FUNCTION file('nginx_report.arrow')
SELECT level, count() AS cnt
FROM nginx_logs GROUP BY level
FORMAT Arrow;
```

```bash
# 查看各格式文件大小
ls -lh /var/lib/clickhouse/user_files/nginx_report.*
```

典型结果（3 行数据）：

```
nginx_report.csv      ~80B    # 纯文本，无压缩
nginx_report.json     ~150B   # JSON 元数据开销大
nginx_report.parquet  ~450B   # 列存 + 内置压缩，小文件有 Schema 头部开销
nginx_report.arrow    ~1.1KB  # Arrow IPC 有完整 Schema 定义
```

> **说明**：小数据量时 Parquet/Arrow 的体积反而比 CSV 大，这是因为它们的 Schema 元数据有固定开销（几百字节）。**大数据量（百万行+）时，Parquet 的列存压缩优势会非常显著——压缩比通常可达 5-10x**（相对于 CSV）。CSV 是行式文本，Parquet 是列存二进制 + 压缩，不在一个量级。

```bash
# 在容器内验证——Pandas 读取 Parquet
```

#### Step 6：大文件分批导入最佳实践

对于超大文件（数百 GB 或 TB 级），一次性导入不仅风险高（断连则前功尽弃），也无法利用并行加速。最佳实践是**先按分区键拆分大文件，再并行管道导入**：

```bash
# 在容器内操作

# Step 6.1：用 awk 按日期字段拆分 CSV（假设日期在第一列）
# 原理：按 toYYYYMM(timestamp) 拆分成独立小文件
apt-get update -qq && apt-get install -y -qq moreutils 2>/dev/null || true

# 模拟拆分：按月份的日期前缀拆分
python3 << 'EOF'
import csv
import os
from collections import defaultdict

output_dir = "/var/lib/clickhouse/user_files/split_logs"
os.makedirs(output_dir, exist_ok=True)

files = defaultdict(list)
with open("/var/lib/clickhouse/user_files/nginx_logs.csv") as f:
    reader = csv.reader(f)
    for row in reader:
        # 按年月分组：如 2025-01, 2025-02 ...
        month_key = row[0][:7]  # "2025-01-15T..." → "2025-01"
        files[month_key].append(row)

for month, rows in files.items():
    filepath = os.path.join(output_dir, f"logs_{month}.csv")
    with open(filepath, "w") as f:
        writer = csv.writer(f)
        writer.writerows(rows)
    print(f"{filepath}: {len(rows)} rows")
EOF
```

```bash
# Step 6.2：并行导入拆分后的文件（多个 clickhouse-client 并发）
# 注意：这是一个演示脚本，生产中使用 xargs -P 或 GNU parallel

for f in /var/lib/clickhouse/user_files/split_logs/*.csv; do
    echo "Importing $f ..."
    cat "$f" | clickhouse-client --query="INSERT INTO nginx_logs FORMAT CSV" &
done

# 等待所有后台任务完成
wait
echo "All imports finished!"

# 验证总数
clickhouse-client --query="SELECT count() FROM nginx_logs"
```

```sql
-- Step 6.3：监控导入过程中的 Part 变化
-- 在另一个 clickhouse-client 窗口中持续观察
SELECT
    partition,
    count()   AS parts,
    sum(rows) AS rows
```

> **生产建议**：
> 1. 拆分文件时，**按分区键（如 `toYYYYMM`）拆分**，确保每个文件对应一个分区，这样导入后同分区数据天然聚合，减少跨分区 Merge 开销。
> 2. 并行度控制在 CPU 核心数的 50% 以内——导入是 CPU 密集型（解析 + 压缩），过度并行反而因上下文切换拖慢速度。
> 3. 导入完成后执行 `OPTIMIZE TABLE nginx_logs FINAL` 将每个分区的多个小 Part 合并为一个（仅非高峰期执行）。

#### Step 7：FORMAT 兼容性对比实验

故意制造一些格式不兼容的场景，观察报错行为：

```sql
-- 创建两张小表做对比测试
CREATE TABLE format_test (
    id   UInt32,
    name String,
    ts   DateTime
) ENGINE = MergeTree()
ORDER BY id;
```

```bash
# 场景A：正常 CSV 导入 —— 成功
echo '1,"Alice","2025-01-01 00:00:00"
2,"Bob","2025-01-02 00:00:00"' \
  | clickhouse-client --query="INSERT INTO format_test FORMAT CSV"

# 场景B：DateTime 格式不匹配 —— 失败
echo '1,"Alice","2025/01/01 00:00:00"' \
  | clickhouse-client --query="INSERT INTO format_test FORMAT CSV"
# 报错：Cannot parse DateTime

# 场景C：字段数不匹配 —— 失败
echo '1,"Alice"' \
  | clickhouse-client --query="INSERT INTO format_test FORMAT CSV"
# 报错：Number of columns does not match

# 场景D：JSONEachRow 导入 —— 列名自动匹配，顺序无关
echo '{"name":"Alice","id":3,"ts":"2025-01-03 00:00:00"}
{"id":4,"ts":"2025-01-04 00:00:00","name":"Bob"}' \
  | clickhouse-client --query="INSERT INTO format_test FORMAT JSONEachRow"
```

```sql
SELECT * FROM format_test ORDER BY id;
```

对比要点：
- **CSV** 依赖列顺序，列数不对或类型不匹配则整行失败（无容错）。
- **JSONEachRow** 按列名匹配，顺序无关，缺少的列自动填默认值（有容错）。
- **如果数据来自 Kafka/API、结构可能频繁变化，优先用 JSONEachRow**。如果数据来自 ETL 管道、列结构固定，CSV 性能更好。

## 4. 项目总结

### 导入方式性能对比表

| 方式 | 吞吐量 | 适用场景 | 优点 | 缺点 |
|------|--------|----------|------|------|
| `INSERT ... VALUES` 逐行 | 极低（百行/秒） | 手工维护几条数据 | 最简单直观 | 网络往返多，不适合批量 |
| `INSERT ... VALUES` 批量 | 中等（万行/秒） | 千~万行级维表初始化 | 无需外部工具 | 单行格式错全批回滚，客户端内存风险 |
| 管道 `FORMAT CSV` | 高（十万~百万行/秒） | **批量文件导入（首选）** | 流式零 OOM，客户端轻量 | 文件需在本地；中断后无断点续传 |
| `INSERT INTO ... SELECT FROM file()` | 高 | ETL 内部转换、单次批量导入 | 纯 SQL，不需要 shell 管道 | 文件解析阶段内存峰值较高 |
| `file()` 表函数直接查询 | 取决于文件大小 | **即席分析、临时探索** | 零导入成本，即刻可用 | 每次全扫，无索引无压缩优化 |
| Kafka 引擎 + 物化视图 | 持续流式 | **实时流数据接入** | 自动化，事件驱动 | 需维护 Kafka 集群和消费组 |
| `s3()`/`url()`/`hdfs()` 表函数 | 取决于网络带宽 | 云端/远端文件即席查询 | 无需下载即可分析 | 网络延迟，带宽受限 |

### 导出格式选型指南

| 下游消费者 | 推荐格式 | 原因 |
|------------|----------|------|
| Python Pandas | **Parquet** / Arrow | Pandas 2.0+ 原生支持，Schema 自动感知 |
| Apache Spark | **Parquet** | Spark 默认列存格式，谓词下推优化 |
| Apache Flink | **Avro** / JSONEachRow | Flink 流处理对 Avro Schema 演化支持好 |
| Excel / 人工查看 | **CSV** / TSV | 通用性最好，任何工具都能打开 |
| 跨 ClickHouse 实例 | **Native** | 内部二进制协议，零序列化开销 |
| BI 工具（Superset / Tableau） | TabSeparated / CSVWithNames | BI 工具普遍支持 TSV/CSV |
| Hive / Presto / Trino | **ORC** / Parquet | Hadoop 生态默认列存格式 |

### 注意事项

1. **CSV 表头处理**：如果 CSV 第一行是列名，导入时需要跳过。用 `clickhouse-client` 的 `--input_format_csv_allow_variable_number_of_columns=1` 或先用 `tail -n +2` 去掉表头再管道导入。表函数需要 `SETTINGS format_csv_allow_variable_number_of_columns=1`。

2. **DateTime 格式必须匹配**：ClickHouse 默认解析 `YYYY-MM-DD HH:MM:SS` 格式。如果源数据是 Unix 时间戳（秒/毫秒）或 ISO 8601（`2025-01-01T00:00:00Z`），需要指定 `--date_time_input_format='best_effort'` 或先用 `toDateTime()` 函数转换。

3. **大文件 INSERT ... VALUES 导致 OOM**：单次 `INSERT INTO t VALUES (...), (...), ...` 中如果 VALUES 数量过多（如数百万行），客户端会把所有数据序列化到内存再发送。解决：用管道 `FORMAT` 代替 `VALUES`，或分批 `LIMIT 100000`。

4. **s3() 表函数需要配置凭据**：使用 S3 表函数前需要在 ClickHouse 配置文件中设置 `access_key_id` 和 `secret_access_key`，或在 SQL 中通过 `SETTINGS s3_access_key_id='...'` 传入。

5. **Kafka 引擎不保证 Exactly-Once**：Kafka 表引擎默认是 At-Least-Once 语义。如果 ClickHouse 在消费 offset 后、写入 MergeTree 前崩溃，恢复后会重复消费。对数据一致性要求极高的场景，需配合 ReplacingMergeTree 或去重逻辑。

6. **Parquet 导出时的 Schema 兼容性**：ClickHouse 导出的 Parquet 文件使用的是 Arrow Schema。如果下游 Spark 版本过旧（< 2.4），可能无法正确读取。建议导出后先用下游工具试读一小批，验证兼容性。

### 常见踩坑经验

**坑1：CSV 中时间格式不匹配导致全部导入失败**

某团队从 MySQL 导出的 CSV，时间字段是 `2025/01/01 00:00:00`（斜杠分隔），直接用 ClickHouse 的 CSV 格式导入，5000 万行全部失败。根因是 ClickHouse 默认的 DateTime 解析器不认斜杠。解决方案：导入前用 `sed 's|/|-|g'` 替换，或导入后用 `String` 类型暂存 ETL 转换。更好的做法是导出时就指定标准格式。

**坑2：大文件 INSERT ... VALUES 一次性加载到内存 OOM**

某数据工程师用 Python 写了个脚本，`INSERT INTO t FORMAT CSV` 之前先把整个 30GB 的 CSV 读到内存里做预处理，结果 Python 进程直接 OOM Kill。根因是没有流式处理。解决方案：用 `cat file.csv | clickhouse-client --query="..."` 做管道流式导入，或用 Python `iter_lines()` 逐行/逐批处理。

**坑3：Parquet 导出时 Schema 不兼容下游 Spark**

某团队将 ClickHouse 的聚合结果导出为 Parquet，给下游 Spark 3.0 做模型训练。Spark 读取时报 `java.lang.UnsupportedOperationException: Unsupported type: UInt32`。根因是 Parquet 规范中没有 Unsigned 整数类型，ClickHouse 把 `UInt32` 映射成了 Parquet 的 `INT64` + 逻辑类型标注，而旧版 Spark 不识别这个逻辑类型。解决方案：导出前用 `toInt32()` / `toInt64()` 将有符号无符号转换一致，或升级 Spark 到 3.2+。

**坑4：管道导入中断后无断点续传**

某运维同学用管道导入一个 2TB 的 CSV 文件，跑了 8 小时后客户端因网络波动断开。已写入的 1.2TB 数据已经落盘成 Part 但尚未确定插入范围——重复执行导入会导致数据翻倍。解决方案：大文件导入前先按分区键拆分成小文件（参考 Step 6），逐个文件导入；每个文件导入成功后移动到 `archive/` 目录，这样即使中断也可以从上次断点继续。
