# 第26章：性能调优——从配置到内核

> **版本**：ClickHouse 24.x+ LTS
> **定位**：中级篇核心章节。从默认配置出发，通过系统性调优方法论，将单节点查询吞吐量从"能用"提升到"好用"。
> **前置阅读**：第4章（MergeTree家族）、第7章（主键与稀疏索引）、第15章（查询分析器与query_log诊断）
> **预计阅读**：40分钟 | **实战耗时**：90分钟

## 1. 项目背景

某数据平台团队最近焦头烂额。他们用一台64核、256GB内存、NVMe SSD的物理机部署了ClickHouse，为公司的运营仪表盘提供实时查询服务。仪表盘页面每30秒自动刷新一次，每次刷新触发6条SQL查询，白天整个集群承载着大约1000 QPS的并发量。架构看起来毫无问题——单节点ClickHouse处理这点查询量，绰绰有余。

然而诡异的事情发生了：仪表盘的P99延迟居然飙到了3秒以上，高峰期甚至出现超时重试。运维团队盯着Grafana监控，CPU使用率稳稳趴在30%，内存使用量也没超过60%。"机器明明没跑满，怎么就慢了？"这是他们每天晨会上必然出现的灵魂拷问。

更要命的是竞品对比的压力。隔壁部门用同样配置的机器跑Druid，轻轻松松撑到5000 QPS，而且平均响应时间在50毫秒以内。"ClickHouse不是号称世界上最快的OLAP引擎吗？是不是咱们没用对？"

技术负责人老张决定系统性地排查一下。他翻了翻ClickHouse的配置文件——`config.xml`里几乎全是出厂默认值。`max_threads`虽然设成了64，但`max_memory_usage`是0（不限制），`mark_cache_size`还是默认的5GB。老张又检查了操作系统层面：Transparent Huge Pages开着，Swap分区容量32GB且`swappiness`是默认的60，I/O调度器是`mq-deadline`，文件句柄限制是默认的1024。

这么多配置项，每个都可能是瓶颈，但不知道从哪个下手。盲目调参更危险——曾经就有同事把`max_threads`从64改到了256，结果上下文切换开销暴涨，查询反而更慢了。还有人把`max_memory_usage`调得太大，ClickHouse直接OOM被kill掉，整条数据写入链路中断了半小时。

"我们需要一套方法论。"老张在白板上写下：**基准测试 → 单变量调优 → 验证效果 → 记录变化**。"没有基准线的调优就是瞎猜，没有单变量的调优就是赌博。"

本章将带着你跟随老张团队，从系统内核到ClickHouse配置，一步步搭建起一套完整的性能调优体系。

## 2. 项目设计：剧本式交锋对话

会议室里，老张把监控大屏投影到墙上。CPU曲线像一条死鱼，趴在30%的位置上一动不动。

**小胖**把椅子转了个圈："这还不简单？性能不够加机器呗！咱们公司又不差那几台服务器。堆硬件不就行了？调参数能有多大提升？"

**老张**苦笑："胖儿，你算算账。64核256G的机器，一台月租八千。加一台是八千，加十台是八万。而且你没搞清楚——现在不是机器不够，是机器明明空着但是用不上。CPU才30%，这说明什么？说明瓶颈不在算力上，在别的地方。"

**小白**推了推眼镜，翻着ClickHouse文档："我在config.xml里看到几百个配置项。`max_threads`、`max_memory_usage`、`max_bytes_before_external_group_by`、`distributed_aggregation_memory_efficient`……这么多参数，什么时候该调哪个？参数之间会不会互相影响？比如我把内存调大了，线程数是不是也得跟着调？"

**大师**（公司请来的ClickHouse顾问）合上笔记本，走到白板前："小白的直觉非常对——调优最忌讳的就是大杂烩。一下子改十个参数，快了不知道是哪个的功劳，慢了更不知道是哪个的锅。我们遵循一个铁律：**一次只改一个参数，改完就测，测完就记，只留正向收益的改动。**"

大师在白板上画了一条横轴："任何性能调优都可以拆成三层。"

```
┌─ 应用层 ─ session级settings (max_threads, join_algorithm, ...)
├─ 引擎层 ─ config.xml 全局配置 (mark_cache_size, max_server_memory_usage, ...)
└─ 系统层 ─ OS内核参数 (THP, swap, ulimit, I/O scheduler, ...)
```

"第一层是Session级的settings，在SQL语句里用`SETTINGS`子句就能覆盖，适合按查询场景做精细化控制。比如白天跑Dashboard查询——低延迟优先，线程可以多开几个；晚上跑ETL——高吞吐优先，线程少开点但单线程跑满。第二层是config.xml里的全局配置，影响整个ClickHouse实例的行为，比如mark缓存大小、最大并发查询数。第三层是操作系统内核参数，直接影响ClickHouse能否高效地使用磁盘和内存。"

**小胖**举手："等等，为什么禁用THP和Swap这么重要？这不都是Linux内核的功能吗，关掉它们不会影响其他进程吗？"

**大师**："好问题。先说THP——Transparent Huge Pages。它的本意是减少TLB miss，提高大内存应用的效率。但ClickHouse自己管理内存——它用`mmap`把MergeTree的数据文件映射到虚拟地址空间。THP会在后台偷偷把4KB的页合并成2MB的大页，这个合并过程会阻塞`mmap`调用的进程。ClickHouse查询时频繁访问`mmap`映射的数据，THP的合并操作就变成了一颗随机炸弹——你永远不知道什么时候查询会卡上几十毫秒。"

"再说Swap。ClickHouse大量使用`mmap`，被映射的文件页面在内存不足时可以被操作系统swap到磁盘。如果数据被swap出去，下次访问的时候就要从磁盘读回来——这就相当于你的内存里明明有数据，但操作系统非要把一部分挪到磁盘上，需要的时候再挪回来。一来一回几十毫秒，query_latency直接起飞。所以ClickHouse的最佳实践是：**要么把swap彻底关掉（swapoff -a），要么把swappiness设成1，让内核尽可能不用swap。**"

**小白**追问："关掉swap之后，如果内存真的不够了呢？"

**大师**："那就是OOM Killer的直接介入——进程被杀掉。听起来很可怕，但至少你**知道**内存不够，可以去加内存或者优化查询。而swap让你陷入慢得莫名其妙、又找不到原因的'灰色地带'。所以关键是：关swap之前，要确保物理内存够用。一般建议`max_server_memory_usage_to_ram_ratio`设为0.9——让ClickHouse最多用90%的物理内存，留10%给操作系统和其他进程。这样基本不可能触发OOM。"

**小胖**来了兴趣："那multi-disk配置是什么鬼？一台机器上插多块NVMe SSD，ClickHouse能自动负载均衡？"

**大师**点头："对。在`storage_configuration`里可以声明多个磁盘，然后创建存储策略——JBOD就是最简单的轮询写入，每个part随机落到不同磁盘上。查询的时候多个磁盘可以并行读取，OPS接近线性扩展。但要记住——**这只是提升写入和扫描的I/O带宽，对查询延迟的改善有限**。如果一个查询只读一个part，它的I/O速度还是单盘的速度。"

**小白**："刚才说的async_insert是做什么的？是不是类似Buffer引擎的那种批量写入？"

**大师**："思路类似，但更现代。`async_insert=1`之后，多个并发的INSERT会被ClickHouse自动排队，攒满一个batch（由`async_insert_max_data_size`控制大小）或者到达超时时间（`async_insert_busy_timeout_ms`），再一次性刷写入表。这跟Buffer引擎的区别在于——async_insert是服务端Native协议的内置能力，不需要额外建Buffer表，不丢数据（只要`wait_for_async_insert=1`），也支持物化视图的级联触发。对于每秒几百上千条INSERT的高频写入场景，async_insert可以减少80%以上的合并开销。"

**小胖**恍然大悟："我懂了！调优跟健身差不多——不是吃一堆补剂就行的，得系统性地来。先测基础代谢，再调整单项，最后看整体效果。那我还有一个问题——怎么知道调对了还是调错了呢？"

**大师**在白板上写下最后一行字：

```
clickhouse-benchmark ← 你的成绩单
system.query_log   ← 你的体检报告
system.metrics     ← 你的心跳监测
```

"`clickhouse-benchmark`是自带的压测工具，一条命令就能输出QPS和P50/P95/P99延迟。每改一个参数就跑一遍，结果对比一目了然。`system.query_log`记录每一条SQL的耗时、内存、读行数——这是你调优的'显微镜'。三者结合，形成闭环：**压测发现问题 → query_log定位瓶颈 → 调整参数 → 重新压测验证。** 开工！"

### 分步实现

#### Step 1：基线性能测试

首先建立测试表并灌入1亿行模拟数据，作为性能对比的基准线：

```sql
-- 创建订单表
CREATE TABLE benchmark_orders (
    order_id UInt64,
    user_id UInt32,
    product_id UInt32,
    amount Decimal(10,2),
    status LowCardinality(String),
    created_at DateTime
) ENGINE = MergeTree()
ORDER BY (created_at, order_id)
PARTITION BY toYYYYMM(created_at);

-- 插入1亿行模拟数据（约3-5分钟）
INSERT INTO benchmark_orders SELECT
    number AS order_id,
    rand() % 100000 AS user_id,
    rand() % 10000 AS product_id,
    (rand() % 100000) / 100.0 AS amount,
    ['pending','completed','cancelled'][rand() % 3 + 1] AS status,
    now() - INTERVAL (rand() % 365) DAY AS created_at
FROM numbers(100000000);
```

在容器外部，使用`clickhouse-benchmark`建立性能基线。默认情况下，ClickHouse二进制包自带此工具：

```bash
# 安装clickhouse-client（如果未安装）
# macOS: brew install clickhouse
# Linux: apt/yum install clickhouse-client

# 运行基线压测：并发1，迭代10次
clickhouse-benchmark --host=localhost --port=9000 \
  --concurrency=1 --iterations=10 \
  --query="SELECT status, count(), sum(amount), avg(amount)
FROM benchmark_orders
WHERE created_at >= '2025-01-01'
GROUP BY status"
```

> **预期输出解读**：关注三个核心指标——QPS（每秒查询数）、P95延迟（95%查询在此时间内完成）、min/max/avg延迟。基线压测的QPS值将成为后续每一步调优的"基准线"。

```bash
# 并发16线程压测，模拟多用户同时查询
clickhouse-benchmark --host=localhost --port=9000 \
  --concurrency=16 --iterations=100 \
  --query="SELECT user_id, count(), sum(amount)
FROM benchmark_orders
WHERE created_at >= '2025-01-01'
GROUP BY user_id
ORDER BY sum(amount) DESC
LIMIT 100"
```

记录下基线数据。假设本次测试得到：并发16下QPS为12，P95延迟为1500ms。

#### Step 2：调优max_threads——线程数不是越大越好

`max_threads`控制单个查询可以使用的最大CPU线程数。默认值等于CPU核心数。很多人有误区：线程越多越能榨干CPU。实际上，线程数超过CPU核数后，上下文切换开销会反噬性能。

```sql
-- 查看当前max_threads值
SELECT name, value FROM system.settings WHERE name = 'max_threads';
-- 典型输出: max_threads | 4 (等于容器的CPU核数)
```

设计对比实验：固定SQL不变，只改`max_threads`：

```sql
-- 实验A: max_threads = 2（保守模式，适用于I/O密集型负载）
SELECT status, count(), sum(amount)
FROM benchmark_orders
WHERE created_at >= '2025-01-01'
GROUP BY status
SETTINGS max_threads = 2;

-- 实验B: max_threads = 4（等于CPU核数，默认模式）
SELECT status, count(), sum(amount)
FROM benchmark_orders
WHERE created_at >= '2025-01-01'
GROUP BY status
SETTINGS max_threads = 4;

-- 实验C: max_threads = 8（激进模式，2倍核数）
SELECT status, count(), sum(amount)
FROM benchmark_orders
WHERE created_at >= '2025-01-01'
GROUP BY status
SETTINGS max_threads = 8;
```

不要凭肉眼看耗时——从`system.query_log`里提取精确的`query_duration_ms`：

```sql
SELECT
    Settings['max_threads'] AS thread_setting,
    query_duration_ms,
    read_rows,
    memory_usage
FROM system.query_log
WHERE query LIKE '%benchmark_orders%'
  AND type = 'QueryFinish'
  AND query_start_time >= now() - INTERVAL 10 MINUTE
ORDER BY query_start_time DESC
LIMIT 10;
```

> **调优原则**：对于计算密集型（大量聚合、排序），`max_threads`设为CPU核数；对于I/O密集或混合负载（高并发Dashboard查询），设为核数的50%-75%，降低上下文切换开销。

#### Step 3：聚合优化——两级聚合与外部磁盘溢出

当`GROUP BY`的基数（不同key的数量）非常大时——比如按`user_id`分组聚合1亿行数据，有10万个不同的user——ClickHouse默认在内存中建HashMap做聚合。如果内存不够，直接OOM。

**两级聚合**是第一道防线。它将聚合拆成两个阶段：第一阶段在每个线程内用哈希表做局部聚合，第二阶段将各线程的中间结果汇总做全局聚合。这样可以显著降低内存峰值：

```sql
-- 对比：不开两级聚合 vs 开两级聚合
-- 不开（默认）
SELECT user_id, count(), sum(amount)
FROM benchmark_orders
WHERE created_at >= '2025-01-01'
GROUP BY user_id
SETTINGS group_by_two_level_threshold = 0;  -- 0表示禁用

-- 开启两级聚合，阈值为10万
SELECT user_id, count(), sum(amount)
FROM benchmark_orders
WHERE created_at >= '2025-01-01'
GROUP BY user_id
SETTINGS
    group_by_two_level_threshold = 100000,
    group_by_two_level_threshold_bytes = 50000000;
```

> `group_by_two_level_threshold`表示当GROUP BY的键数量超过这个值时启用两级聚合，`group_by_two_level_threshold_bytes`是内存阈值版本。两者任一命中即触发。

**外部聚合（External Aggregation）** 是第二道防线。当内存确实不够时，让ClickHouse把中间数据溢写到磁盘上：

```sql
-- 实验：限制内存为500MB，强制触发外部聚合
SELECT user_id, count(), sum(amount)
FROM benchmark_orders
GROUP BY user_id
SETTINGS
    max_bytes_before_external_group_by = 500000000,   -- 500MB触发写盘
    max_memory_usage = 1000000000;                      -- 总内存不超1GB
```

观察是否触发了外部聚合：

```sql
SELECT
    event_time,
    value
FROM system.events
WHERE event = 'ExternalAggregationWritePart'
  AND event_time >= now() - INTERVAL 10 MINUTE;
```

> 如果`ExternalAggregationWritePart`计数 > 0，说明聚合确实溢出到了磁盘。虽然比纯内存慢，但至少保证了查询不会OOM被kill。

#### Step 4：JOIN优化——五种算法的场景选择

建一张维度表用于JOIN实验：

```sql
CREATE TABLE users_dim (
    user_id UInt32,
    level LowCardinality(String),
    city String
) ENGINE = MergeTree()
ORDER BY user_id;

INSERT INTO users_dim SELECT
    number % 100000 AS user_id,
    ['basic','premium','vip'][rand() % 3 + 1] AS level,
    ['北京','上海','深圳','杭州','广州'][rand() % 5 + 1] AS city
FROM numbers(50000);
```

现在对四种JOIN算法进行横向对比，统一使用维表JOIN订单表的分组查询：

```sql
-- 查询模板（统一SQL，仅改join_algorithm）
-- 算法1: hash join（右表小，构建内存哈希表，默认行为）
SELECT u.level, count(), sum(o.amount)
FROM benchmark_orders o
JOIN users_dim u ON o.user_id = u.user_id
WHERE o.created_at >= '2025-01-01'
GROUP BY u.level
SETTINGS join_algorithm = 'hash';

-- 算法2: partial_merge join（右表大，按JOIN键排序后合并，内存友好）
SELECT u.level, count(), sum(o.amount)
FROM benchmark_orders o
JOIN users_dim u ON o.user_id = u.user_id
WHERE o.created_at >= '2025-01-01'
GROUP BY u.level
SETTINGS join_algorithm = 'partial_merge';

-- 算法3: grace_hash join（hash的升级版，内存不足时溢写磁盘）
SELECT u.level, count(), sum(o.amount)
FROM benchmark_orders o
JOIN users_dim u ON o.user_id = u.user_id
WHERE o.created_at >= '2025-01-01'
GROUP BY u.level
SETTINGS
    join_algorithm = 'grace_hash',
    grace_hash_join_initial_buckets = 8,        -- 初始桶数
    max_bytes_in_join = 100000000;               -- 100MB阈值

-- 算法4: parallel_hash join（多线程并行构建哈希表，适合大右表）
SELECT u.level, count(), sum(o.amount)
FROM benchmark_orders o
JOIN users_dim u ON o.user_id = u.user_id
WHERE o.created_at >= '2025-01-01'
GROUP BY u.level
SETTINGS join_algorithm = 'parallel_hash';
```

查看各算法的内存和耗时对比：

```sql
SELECT
    Settings['join_algorithm'] AS algo,
    avg(query_duration_ms) AS avg_ms,
    avg(memory_usage) AS avg_mem,
    count() AS queries
FROM system.query_log
WHERE query LIKE '%JOIN users_dim%'
  AND type = 'QueryFinish'
  AND query_start_time >= now() - INTERVAL 30 MINUTE
GROUP BY algo
ORDER BY avg_ms;
```

> **选型指南**：
> - `hash`：右表小于百万行且内存充足时的首选，构建快、查询快
> - `partial_merge`：右表大（千万行以上），需要低内存占用的稳定方案
> - `grace_hash`：右表中等但内存受限时的保底选择，自动溢写到磁盘
> - `parallel_hash`：右表大且CPU核数多时，利用并行加速哈希表构建

#### Step 5：异步写入优化——让INSERT飞起来

当写入频率很高时（每秒几十到几百条INSERT），每次INSERT都触发一次part写入和潜在的后台merge，磁盘I/O会成为瓶颈：

```sql
-- 开启异步写入
SET async_insert = 1;
SET wait_for_async_insert = 0;           -- 0=不等待刷盘即返回（性能优先）
SET async_insert_max_data_size = 10000000;  -- 10MB缓冲阈值
SET async_insert_busy_timeout_ms = 500;     -- 500ms超时刷新

-- 模拟高频写入场景
INSERT INTO benchmark_orders VALUES (100000001, 42, 999, 88.50, 'completed', now());
INSERT INTO benchmark_orders VALUES (100000002, 43, 998, 12.30, 'pending', now());
INSERT INTO benchmark_orders VALUES (100000003, 44, 997, 56.00, 'completed', now());
-- 以上三条INSERT会被自动合并为一个写入块
```

产线环境中，建议在`config.xml`的`profiles`中全局启用：

```xml
<profiles>
    <default>
        <async_insert>1</async_insert>
        <wait_for_async_insert>1</wait_for_async_insert>
        <async_insert_max_data_size>10000000</async_insert_max_data_size>
        <async_insert_busy_timeout_ms>1000</async_insert_busy_timeout_ms>
    </default>
</profiles>
```

> **关键取舍**：`wait_for_async_insert=0`时延迟最低，但ClickHouse进程崩溃可能导致缓冲区内的数据丢失。对于关键业务，应设置为`1`，确保客户端收到确认后数据已被刷写到磁盘。

检查异步写入队列状态：

```sql
SELECT
    table,
    status,
    bytes_uncompressed,
    first_insert_time,
    flush_time
FROM system.asynchronous_inserts;
```

#### Step 6：系统级调优——给ClickHouse一个干净的内核环境

以下命令在生产主机上执行（Docker环境在宿主机上执行），每一项都直接影响ClickHouse的性能：

```bash
# 1. 禁用Transparent Huge Pages（必须立即禁掉！）
echo never > /sys/kernel/mm/transparent_hugepage/enabled
echo never > /sys/kernel/mm/transparent_hugepage/defrag

# 持久化到 /etc/rc.local 确保重启后依然生效

# 2. 关闭Swap（或强制最少使用）
swapoff -a
# 或者保留swap文件但设为最少使用
sysctl -w vm.swappiness=1

# 3. 增大文件句柄限制（默认1024根本不够）
ulimit -n 262144
# 持久化: /etc/security/limits.conf 添加
# clickhouse soft nofile 262144
# clickhouse hard nofile 262144

# 4. 设置I/O调度器为none（NVMe/SSD最佳）
echo none > /sys/block/nvme0n1/queue/scheduler
# 传统SATA盘用deadline或mq-deadline
# echo mq-deadline > /sys/block/sda/queue/scheduler

# 5. CPU频率策略设为performance（避免节电降频）
echo performance > /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor

# 6. 验证以上全部生效
cat /sys/kernel/mm/transparent_hugepage/enabled
cat /proc/sys/vm/swappiness
ulimit -n
cat /sys/block/nvme0n1/queue/scheduler
cat /sys/devices/system/cpu/cpu0/cpufreq/scaling_governor
```

ClickHouse全局配置优化（`config.xml`或`config.d`目录中）：

```xml
<clickhouse>
    <!-- 连接与并发控制 -->
    <max_connections>4096</max_connections>
    <max_concurrent_queries>100</max_concurrent_queries>
    <max_concurrent_insert_queries>50</max_concurrent_insert_queries>

    <!-- 内存使用上限：使用90%的物理内存 -->
    <max_server_memory_usage_to_ram_ratio>0.9</max_server_memory_usage_to_ram_ratio>

    <!-- 线程池大小：建议设为 max_concurrent_queries * max_threads -->
    <max_thread_pool_size>10000</max_thread_pool_size>

    <!-- Mark缓存：稀疏索引的热数据缓存，对读性能影响极大 -->
    <!-- 建议设为物理内存的10%-20%，此处256GB * 20% ≈ 50GB -->
    <mark_cache_size>53687091200</mark_cache_size>

    <!-- 未压缩数据缓存：扫描时省掉解压缩开销 -->
    <uncompressed_cache_size>21474836480</uncompressed_cache_size>

    <!-- JBOD多磁盘存储策略 -->
    <storage_configuration>
        <disks>
            <disk_nvme0>
                <path>/data/clickhouse/disk0/</path>
            </disk_nvme0>
            <disk_nvme1>
                <path>/data/clickhouse/disk1/</path>
            </disk_nvme1>
            <disk_nvme2>
                <path>/data/clickhouse/disk2/</path>
            </disk_nvme2>
            <disk_nvme3>
                <path>/data/clickhouse/disk3/</path>
            </disk_nvme3>
        </disks>
        <policies>
            <jbod_4disks>
                <volumes>
                    <main>
                        <disk>disk_nvme0</disk>
                        <disk>disk_nvme1</disk>
                        <disk>disk_nvme2</disk>
                        <disk>disk_nvme3</disk>
                    </main>
                </volumes>
            </jbod_4disks>
        </policies>
    </storage_configuration>
</clickhouse>
```

> **JBOD vs RAID说明**：JBOD模式下各磁盘独立，写part时按轮询逐个写到不同磁盘，读时可以并行扫描多盘。适合宽表大批量写入和全表扫描。RAID0模式下操作系统层面条带化，对随机小块I/O更友好，但ClickHouse本身极少随机读——所以JBOD通常是更好的选择。

#### Step 7：调优效果记录——用数据说话

每完成一步调优，立即跑一遍压测并记录结果，避免遗忘因果。建议在ClickHouse中建一张调优日志表：

```sql
CREATE TABLE tuning_journal (
    step UInt8,
    change_desc String,
    benchmark_qps Float32,
    p50_ms Float32,
    p95_ms Float32,
    p99_ms Float32,
    avg_memory_bytes UInt64,
    change_time DateTime DEFAULT now()
) ENGINE = MergeTree()
ORDER BY change_time;

-- 示例：记录第一步max_threads调优效果
INSERT INTO tuning_journal VALUES
(1, 'max_threads: 4 -> 8', 12.5, 220, 890, 1500, 2100000000, now());
```

批量分析全天查询性能变化趋势：

```sql
SELECT
    toStartOfHour(query_start_time) AS hour,
    count() AS query_cnt,
    quantile(0.50)(query_duration_ms) AS p50_ms,
    quantile(0.95)(query_duration_ms) AS p95_ms,
    quantile(0.99)(query_duration_ms) AS p99_ms,
    quantile(0.999)(query_duration_ms) AS p999_ms,
    round(avg(query_duration_ms), 2) AS avg_ms,
    round(avg(memory_usage) / 1048576, 2) AS avg_mem_mb
FROM system.query_log
WHERE type = 'QueryFinish'
  AND query_start_time >= now() - INTERVAL 1 DAY
  AND query_kind = 'Select'
GROUP BY hour
ORDER BY hour;
```

如果发现某个小时的P99突然飙升，可以进一步按查询模式拆分：

```sql
SELECT
    normalized_query_hash,
    substr(query, 1, 120) AS query_sample,
    count() AS cnt,
    quantile(0.95)(query_duration_ms) AS p95_ms
FROM system.query_log
WHERE type = 'QueryFinish'
  AND query_start_time >= now() - INTERVAL 1 HOUR
  AND query_kind = 'Select'
GROUP BY normalized_query_hash, query_sample
ORDER BY p95_ms DESC
LIMIT 10;
```

## 4. 项目总结

### 调优检查清单

| 类别 | 参数 | 默认值 | 建议值 | 影响范围 | 注意事项 |
|------|------|--------|--------|---------|---------|
| CPU | `max_threads` | 核数 | 核数（计算密集）或核数/2（IO密集） | 高 | 超过核数×2上下文切换反噬 |
| CPU | `max_thread_pool_size` | 10000 | `max_concurrent_queries × max_threads` | 中 | 设太小会导致线程饥饿 |
| 内存 | `max_memory_usage` | 0(不限) | 80%物理内存 | 关键 | 设0可能导致OOM被kill |
| 内存 | `max_server_memory_usage_to_ram_ratio` | 0.9 | 0.8-0.9 | 关键 | 留出一部分给OS和其他进程 |
| 聚合 | `group_by_two_level_threshold` | 100000 | 根据基数调整 | 中 | 低基数GROUP BY无必要 |
| 聚合 | `max_bytes_before_external_group_by` | 0(禁用) | 物理内存的50% | 高 | 设为0 = 宁愿OOM也不写盘 |
| JOIN | `join_algorithm` | auto | `partial_merge`（大右表） | 高 | auto不一定选中最优算法 |
| JOIN | `max_bytes_in_join` | 0(不限) | 内存的30% | 中 | 限制hash join的内存占用 |
| 写入 | `async_insert` | 0(关) | 1(开) | 写入高 | 仅对高频INSERT有效 |
| 写入 | `wait_for_async_insert` | 1 | 0(高吞吐) 或 1(高可靠) | 中 | 0有数据丢失风险 |
| 缓存 | `mark_cache_size` | 5GB | 物理内存的10%-20% | 读高 | 缓存命中率直接影响磁盘读次数 |
| 缓存 | `uncompressed_cache_size` | 8GB | 物理内存的10% | 读中 | 仅对重复查询有效 |
| OS | THP | 开启 | **禁用** | 高 | 每个ClickHouse节点必备操作 |
| OS | `vm.swappiness` | 60 | 0或1 | 高 | 必须配合充足物理内存 |
| OS | `ulimit -n` | 1024 | 至少65536 | 中 | 限制的是同时打开的文件数 |
| OS | I/O调度器 | mq-deadline | none(NVMe) / deadline(SSD) | 读中 | 对全表扫描型查询影响最大 |

### 调优方法论总结

```
┌─ Step 1: 建立基线 ──── clickhouse-benchmark跑基准QPS + P95/P99
│
├─ Step 2: 单变量调优 ── 一次只改一个参数，改完即测
│
├─ Step 3: 记录效果 ──── tuning_journal表记录每次改动的量化收益
│
├─ Step 4: 系统层调优 ── THP/Swap/IOScheduler/fileHandle一步到位
│
├─ Step 5: 回归验证 ──── 跑完所有优化后重新跑完整基准对比
│
└─ Step 6: 持续监控 ──── system.query_log + Grafana 每日巡检
```

### 适用场景

- **生产集群上线前**：新集群默认配置保守，必须按照本清单逐项调优
- **大促前容量评估**：双11/618前用`clickhouse-benchmark`压出当前集群的天花板
- **故障排查**：CPU低但延迟高？检查THP、Swap、I/O调度器这三个"隐形杀手"
- **新表设计**：建表时为未来性能调优留好后路（分区键、排序键设计）

### 注意事项

1. **先在预发环境测试，再上生产**。别在生产机上跑`echo never > /sys/kernel/mm/transparent_hugepage/enabled`之前没在预发环境验证——虽然99%安全，但剩下1%的风险你扛不起。

2. **一次只改一个参数**。如果你同时改了`max_threads`、`mark_cache_size`、禁用Swap，然后发现性能提升了40%——你能说清楚是哪个改动贡献了这40%吗？下一次遇到新的瓶颈，你又该调哪个？

3. **内存限制是保命绳**。`max_memory_usage=0`意味着"查询可以用无限内存"——一个失控的`SELECT ... GROUP BY high_cardinality_col`就能把整台机器拖死。必须设置上限。

4. **async_insert有数据丢失风险**。当`wait_for_async_insert=0`时，客户端收到"写入成功"的响应后，数据可能还在内存缓冲区里。如果此时ClickHouse崩溃，这部分数据就丢了。对账单、交易等强一致性场景，必须设为`wait_for_async_insert=1`。

### 常见踩坑经验

1. **max_threads设太大反而变慢**。某团队把max_threads从32调到256，满以为能榨干64核CPU，结果p99延迟反而翻倍。原因是256个线程在32个物理核上疯狂争抢，上下文切换的开销超过了并行收益。经验规则：max_threads不超过物理核数的1.5倍。

2. **关了Swap但没加内存，OOM Kill整个实例**。Swap是慢，但至少能撑着。如果内存真的紧张——比如高峰期突然有用户跑了一条大查询——没有Swap兜底，OOM Killer就会直接杀掉ClickHouse进程。所以在关Swap之前，务必确认物理内存充足，并且`max_server_memory_usage_to_ram_ratio`设置合理。

3. **JBOD磁盘配置了但并行度没跟上**。配置了4块NVMe的JBOD策略，但`max_threads`还留在默认的8——8个线程只能同时读2-3块盘，剩下1-2块盘空转。多磁盘部署时，`max_threads`至少要等于磁盘数量×2。

4. **THP禁用不全导致间歇性延迟抖动**。`echo never > /sys/kernel/mm/transparent_hugepage/enabled`只禁用了当前分配，但已经存在的大页不会被拆分。机器重启后必须重新执行。务必加入开机启动脚本或使用`tuned`等系统工具持久化。

### 思考题

1. `max_bytes_before_external_group_by`设置为多少比较合适？如果一台256GB内存的机器上跑了10个并发查询，每个查询的`GROUP BY`基数各不相同，如何决定这个值？提示：考虑单个查询的内存占用峰值与并发数的乘积关系。

2. 同一台ClickHouse节点，白天跑Dashboard查询——高并发、低延迟要求、单个查询较小；晚上跑ETL任务——低并发、高吞吐、单个查询可能扫全表。如何通过`profiles`（配置Profile）为白天和晚上分别制定不同的参数策略？你怎么让ClickHouse自动在两种Profile之间切换？
