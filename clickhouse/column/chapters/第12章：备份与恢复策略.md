# 第12章：备份与恢复策略

> **版本**：ClickHouse 25.x LTS
> **定位**：运维篇核心章节。没有备份的数据库就是定时炸弹——FREEZE、clickhouse-backup、原生 BACKUP TABLE，三套方案撑起数据安全的最后一道防线。
> **前置阅读**：第4章（MergeTree 家族概览）、第5章（分区与排序键）
> **预计阅读**：35 分钟 | **实战耗时**：50 分钟

---

## 1. 项目背景

某大型物流公司的 ClickHouse 集群承载着全公司最核心的数据资产——物流轨迹追踪表 `tracking_data`。从包裹揽收、中转、运输、派送到签收，全链路每一个节点的经纬度、时间戳、操作员信息全部入仓。这套数据既是实时看板的唯一数据源（全国各分拨中心的揽收量、在途量、签收量实时大屏），也是客服系统的查询引擎（用户一个电话打过来："我的快递到哪了？"客服必须在 3 秒内拉出完整轨迹）。表规模：**5TB、50 亿行**，每天新增约 **2000 万行**，使用 12 节点 × ReplicatedMergeTree 架构。

上个月的一个周四，灾难降临了。

那天运维 DBA 老张接到一个工单：业务方说 2024 年 1 月 1 日的测试数据混进了生产表，需要清理。老张习惯性地打开 DBeaver，写了一条 `ALTER TABLE tracking_data DELETE WHERE date = '2024-01-01'`，选中，Ctrl+Enter。然而他万万没想到——DBeaver 的"执行选中语句"功能不知为何没有生效，它执行了**上一条还在编辑器里的残留 SQL**：

```sql
ALTER TABLE tracking_data DELETE WHERE 1=1
```

`WHERE 1=1`——无条件，全表删除。老张在按下回车的那一刻，50 亿行物流轨迹的 Mutation 任务进入了 ClickHouse 的异步队列。Mutation 一旦入队就**无法取消**——它不会立即删数据，但会在后台默默地为每一个 Part 生成一个被标记为"已删除"的新 Part。等到 Merge 任务把这些新 Part 合并完，原始数据就永久地变成了 `.tmp` 然后被回收。

老张发现时，已经有大约 40% 的 Part 被清理完毕，大量物流单号的轨迹查询返回空结果。全国 36 个分拨中心的实时大屏全部变成 0——揽收量 0、在途量 0、签收量 0。客服电话瞬间被打爆，业务方紧急通知所有分拨中心暂停卸车和装车，因为"系统不知道包裹在哪了"。

更致命的是——**没有备份**。没有 FREEZE，没有 clickhouse-backup，没有 S3，连个 `rsync` 脚本都没写过。运维团队的认知停留在"ClickHouse 有副本啊，三副本够了，要什么备份？"——但他们没想过，副本复制的是写入操作，`DELETE` Mutation 同样会被复制到所有副本上去。三副本对于误删数据的保护力是**零**。

恢复过程极其痛苦：从上游 Kafka 重新灌数，由于 Kafka 的 Retention 只有 7 天，历史数据需要回溯到 HDFS 上的原始日志，写 Spark 脚本重新 ETL 一遍。整个恢复流程持续了整整 **3 天**——其中 2 天业务完全停摆，物流轨迹实时追踪功能瘫痪，大量包裹在中转环节"失踪"，最终产生了几十起用户投诉和一笔不小的赔付。

事后复盘会上，CTO 拍着桌子定了三条铁律：

1. **RPO（恢复点目标）< 1 小时**：关键表的数据在任何时间点丢失，最多丢失 1 小时的数据。
2. **RTO（恢复时间目标）< 30 分钟**：从决定恢复到恢复完成，整个流程不能超过 30 分钟。
3. **备份必须异地存储**：备份文件不能和源数据在同一个机房甚至同一个云账号下，防止"一锅端"。

**核心痛点**：没有备份策略、缺乏恢复演练、团队不知道 FREEZE 机制和 clickhouse-backup 工具、误操作没有 SQL 审计拦截。备份不是"做了就行"，而是要能**在压力下、规定时间内，把数据完整无误地拉回来**。

## 2. 项目设计：剧本式交锋对话

事故复盘会结束后，大师把小白和小胖叫到了工位旁的白板前，写下四个字："**先备后战**。" 阳光透过百叶窗照在白板上，三人的影子斜斜地拉在墙上。

**小胖**（把椅子转过来，翘起腿）：“备份？直接把数据目录 `rsync` 一份不就行了？大不了一天传一次，凌晨三点跑个 cron 脚本嘛。MySQL 时代不就是 `mysqldump` 一把梭？ClickHouse 的数据不就是 `/var/lib/clickhouse` 下面的文件吗，我 `cp -r` 过去不就完了？”

**大师**（忍不住笑出声）：“小胖，你这种备份方式我给它起了个名——**薛定谔的备份**。在你 `cp` 的那一刻，ClickHouse 正在后台做 Merge——Part A 和 Part B 正在被合并成 Part C，旧 Part 正在被删除，新 Part 正在被写入。你的 `cp` 可能拷贝到一个半成品的 Part C、或者一个已经标记删除但还没回收的 Part A。等你哪天想用这个备份恢复——`File not found` 或者 `Checksum mismatch`。备份文件是坏的，但你不知道它是坏的，直到真需要的那一天——打开盒子，猫死了。”

**技术映射 #1**：直接拷贝文件系统是最危险的备份方式。MergeTree 的后台 Merge、Mutation、TTL 删除操作会持续修改数据目录结构。外部拷贝工具看不到 ClickHouse 内部的事务边界，必然产生**崩溃不一致**（Crash-Consistent）的备份——可能丢数据、可能文件残缺、可能元数据错乱。备份必须通过 ClickHouse 自身的机制来保证**应用层一致性**。

---

**小白**（皱着眉头在笔记本上飞快地记）：“那 ClickHouse 自己怎么保证备份的一致性？它总得有个机制把某个时间点的数据'冻结'住，不让后台合并干扰吧？就像 MySQL 的 `FLUSH TABLES WITH READ LOCK` 加全局读锁那样？”

**大师**：“思路对了，但 ClickHouse 不用锁——它用**硬链接（Hard Link）**。这就是 `ALTER TABLE ... FREEZE` 的核心原理。”

（大师转身在白板上画了一个示意图）

```
/var/lib/clickhouse/data/tracking_data/
├── 20240101_0_100_0/    ← Part 目录（原始数据）
├── 20240102_0_200_0/
└── 20240103_0_300_0/

执行 FREEZE 后，同一文件系统上：

/var/lib/clickhouse/shadow/1/data/tracking_data/
├── 20240101_0_100_0/    ← 硬链接（指向同一个 inode）
├── 20240102_0_200_0/    ← 硬链接
└── 20240103_0_300_0/    ← 硬链接
```

**大师**：“硬链接是什么？它和 `cp` 完全不一样。`cp` 是创建一个新文件、把数据拷过去——500GB 的表就要拷 500GB，时间超长，磁盘空间翻倍。而硬链接是**在文件系统层面给同一个 inode 起了一个别名**。创建硬链接几乎不占时间，也不占额外磁盘空间。更重要的是——创建硬链接之后，即使 ClickHouse 后台删除了原来的 Part 目录，只要 `shadow/` 下的硬链接还在，inode 就不会被真正回收，数据就还在磁盘上。”

**小胖**（若有所思）：“所以 FREEZE 其实就是给所有 Part 创建硬链接的快照？但这有个问题啊——硬链接必须在**同一个文件系统**上吧？不能跨盘？”

**大师**：“对，这正是 FREEZE 最大的限制。如果 `/var/lib/clickhouse/data` 在 XFS 文件系统上，你的备份目录在另一个 NFS mount 上——硬链接就创建不了。另外，FREEZE 后这些硬链接还躺在**同一块盘**上——万一磁盘坏了，备份跟着完蛋。所以 FREEZE 只是第一步，你还需要把 `shadow/` 下的文件搬到别处去。”

**技术映射 #2**：`ALTER TABLE ... FREEZE` 利用文件系统的硬链接机制，创建一个**表级别的一致性快照**。它的优势在于速度极快（O(Part数量)，不随数据量增长）、不占额外磁盘空间、不影响写入（非阻塞）。局限性：必须同文件系统、仍在同一物理盘上、需要手动搬到异地存储。

---

**小白**（合上笔记本）：“那 FREEZE 之后的数据怎么还原呢？我直接把 `shadow/` 里的目录拷回 `data/` 目录就行了吗？另外我们有很多 ReplicatedMergeTree 表，ZooKeeper 里存的元数据怎么办？光恢复数据文件没用吧？”

**大师**（点头赞许）：“问到第二个关键问题了。对于非复制表（普通 MergeTree），确实直接把 Part 拷回 `detached/` 目录再 `ATTACH` 即可。但对于 **ReplicatedMergeTree**，你还必须恢复 ZooKeeper 中的元数据路径 `/clickhouse/tables/<cluster>/<db>/<table>` ——里面有 Replica 信息、Block Numbers、Log Pointer 等。如果只恢复数据不恢复 ZK，ClickHouse 会认为自己是新副本，可能触发全量同步覆盖掉你恢复的数据。”

**小胖**：“天哪，那 ZK 的数据怎么备份？一台台手动 `zkCli.sh get`？这不是要疯？”

**大师**：“这就是 `clickhouse-backup` 这个开源工具的价值所在。它是 Altinity 公司维护的 Go 语言项目，专门解决 ClickHouse 的全量备份、增量备份、远程上传和完整恢复。它内部调用了 FREEZE 来保证一致性，然后自动把数据搬到指定存储（本地/S3/FTP/SFTP），同时它还支持 `--rbac` 备份用户权限、`--configs` 备份配置文件，以及——最重要的一点——**备份和恢复 ZooKeeper 元数据**。”

**技术映射 #3**：备份层次分为三层——**数据文件层**（Part + Metadata）、**元数据层**（ZooKeeper 路径）、**配置层**（RBAC + Server Config）。缺一层就可能导致恢复失败，尤其是 ReplicatedMergeTree 必须数据和 ZK 元数据一起恢复。

---

**小白**：“那 clickhouse-backup 是怎么做增量备份的？每天全量备份 5TB 数据，就算带宽受得了，S3 的存储费用也受不了啊。”

**大师**：“增量备份的原理巧妙得很。还记得 MergeTree 的 Part 是**不可变**的——一旦写入完成，Part 就不会再被修改。`20240101_0_100_0` 这个 Part 从生成到被 Merge 吞掉，它里面的数据一个字都不会变。那 clickhouse-backup 怎么做增量呢？它记录了每一个 Part 的元数据指纹（Part 名 + 大小 + Checksum），下次备份时对比：如果某个 Part 的指纹和上次完全一样，就跳过；如果不一样或者是新 Part，就拷贝。这样增量备份只备份**新增和变更的 Part**，通常只有全量的 5%~20%。”

**小胖**（恍然大悟）：“哦！所以不是按行做增量，是按 Part 做增量——因为 Part 本身就是不可变的！”

**大师**：“对。这也是为什么 ClickHouse 天生适合增量备份。另外 clickhouse-backup 的 `--diff-from` 参数允许你指定一个已有的全量备份作为基准，自动计算增量。恢复的时候，先恢复全量，再恢复增量——它内部是按 Part 粒度覆盖的，增量的 Part 会覆盖全量里的同名 Part。”

### 分步实现

#### Step 1：使用 FREEZE 进行冷备份

FREEZE 是 ClickHouse 内置的最轻量级备份机制，在 `data_path` 同文件系统下创建硬链接快照。

```sql
-- 对 tracking_data 表执行 FREEZE
-- 该操作瞬间完成，不阻塞读写
ALTER TABLE tracking_data FREEZE;

-- 查看冻结版本号
SELECT * FROM system.freezes WHERE table = 'tracking_data';
```

```bash
# FREEZE 后的文件存放在 ClickHouse 数据目录的 shadow/ 子目录中
# 每个 FREEZE 操作会生成一个递增编号的目录
ls /var/lib/clickhouse/shadow/

# 输出示例：
# 1/  2/  3/

# 查看最新的 FREEZE 目录结构
ls -la /var/lib/clickhouse/shadow/3/data/default/tracking_data/

# 输出示例——每个 Part 目录都是原始 Part 的硬链接：
# drwxr-x--- 2 clickhouse clickhouse  4096 20250101_0_100_0
# drwxr-x--- 2 clickhouse clickhouse  4096 20250102_0_200_0
# ...

# 验证硬链接：查看任意 Part 文件的 inode 和硬链接计数
stat /var/lib/clickhouse/shadow/3/data/default/tracking_data/20250101_0_100_0/checksums.txt
# Links: 2  ← 硬链接计数为 2，说明原始文件和 shadow 中的文件指向同一个 inode

# 将 FREEZE 数据搬到异地存储
cp -rl /var/lib/clickhouse/shadow/3/ /backups/freeze_$(date +%Y%m%d)/
# 或 rsync 到远程
rsync -avz /var/lib/clickhouse/shadow/3/ backup-server:/backups/clickhouse/
```

> **关键说明**：`cp -rl` 中的 `-l` 参数表示"尽可能创建硬链接而非拷贝"。如果 `/backups` 和 `/var/lib/clickhouse` 在同一个文件系统上，这能保持硬链接特性、节省空间和时间。如果不在同一文件系统，`cp` 会自动回退到真实拷贝。

清除过期 FREEZE 快照：

```sql
-- 清除所有 FREEZE（释放磁盘空间）
ALTER TABLE tracking_data UNFREEZE WITH NAME '3';
```

```bash
# 也可以直接删除 shadow 目录
rm -rf /var/lib/clickhouse/shadow/*
```

> **踩坑提醒**：FREEZE 后的硬链接虽然不占额外磁盘空间（共享同一个 inode），但如果原始数据之后被 Merge 删除了，shadow 中的硬链接就成了数据的唯一引用，此时磁盘空间**不会被释放**。很多运维同学忘记清理 shadow 目录，直到磁盘满了才发现——FREEZE 3 次 5TB 的表，虽然看起来没"复制"数据，但实际上老旧 Part 的数据全靠 shadow 里的硬链接"吊着命"，Merge 回收不了空间，磁盘水位悄悄上涨。

#### Step 2：安装和使用 clickhouse-backup

```bash
# 下载 clickhouse-backup（此处以 2.6.0 版本为例）
# 注意：请根据实际环境和版本选择合适的 release
cd /tmp
wget https://github.com/Altinity/clickhouse-backup/releases/download/v2.6.0/clickhouse-backup-linux-amd64.tar.gz
tar xzf clickhouse-backup-linux-amd64.tar.gz
sudo mv clickhouse-backup /usr/local/bin/

# 验证安装
clickhouse-backup --version
# clickhouse-backup version 2.6.0
```

创建配置文件 `/etc/clickhouse-backup/config.yml`：

```yaml
general:
  remote_storage: s3           # 远程存储类型：s3 / ftp / sftp / gcs / azblob / nfs
  disable_progress_bar: false
  backups_to_keep_local: 7     # 本地保留最近 7 个备份
  backups_to_keep_remote: 30   # 远端保留最近 30 个备份
  log_level: info

clickhouse:
  host: localhost
  port: 9000
  username: default
  password: ""
  data_path: /var/lib/clickhouse
  skip_tables:
    - system.*
    - INFORMATION_SCHEMA.*
  freeze_by_part: true         # 是否按 Part 粒度 FREEZE（建议开启，失败时可以重试单个 Part）

s3:
  access_key: "YOUR_ACCESS_KEY"
  secret_key: "YOUR_SECRET_KEY"
  bucket: "clickhouse-backups"
  path: "production/"
  region: us-east-1
  endpoint: ""
  force_path_style: false
  storage_class: STANDARD_IA   # 低频存储，降低 S3 费用
  compression_format: tar      # 上传前打包格式：tar / tar.zstd / tar.lz4
  part_size: 52428800          # 分片上传大小（50MB），大表上传建议开启
```

```bash
# 测试连接
clickhouse-backup -c /etc/clickhouse-backup/config.yml tables
# 应列出所有表（排除 skip_tables 规则中的系统表）
```

#### Step 4：增量备份

增量备份的核心原理前面已经讲过——MergeTree Part 不可变，增量只需备份新增和变更的 Part。

```bash
# 模拟增量数据写入——新的一天插入一批数据
clickhouse-client -q "
INSERT INTO tracking_data
SELECT
    concat('TRK', toString(rand64() % 10000000000)) AS tracking_id,
    concat('PKG', toString(number % 2000000)) AS package_id,
    now() - INTERVAL 1 DAY - INTERVAL rand64() % 86400 SECOND AS scan_time,
    concat('NODE_', toString(rand64() % 1000)) AS node_code,
    concat('分拨中心_', toString(rand64() % 50 + 1)) AS node_name,
    CAST((rand64() % 4 + 1), 'Enum8(\'揽收\'=1, \'中转\'=2, \'派送\'=3, \'签收\'=4)') AS event_type,
    30.0 + (rand64() % 1000) / 100 AS lat,
    120.0 + (rand64() % 1000) / 100 AS lng,
    toDate(now() - INTERVAL 1 DAY) AS date
FROM numbers(1000000);
"

# 基于全量备份创建增量备份
```

#### Step 6：使用原生 BACKUP TABLE 命令（23.x+）

ClickHouse 23.x 版本引入了原生的 `BACKUP TABLE` / `BACKUP DATABASE` 命令，可以无需第三方工具直接将表或数据库备份到磁盘或 S3。

```sql
-- 备份单表到本地磁盘目录（备份为 .zip 文件）
BACKUP TABLE tracking_data
TO Disk('backups', 'tracking_data_backup_20250115.zip');

-- 备份整库到 S3
BACKUP DATABASE default
TO S3(
    'https://s3.amazonaws.com/clickhouse-backups/production/backup-',
    'YOUR_ACCESS_KEY',
    'YOUR_SECRET_KEY'
);

-- 查看备份列表
SELECT * FROM system.backups;
-- ┌─name──────────────────────────────┬─status──┬─size──────┐
-- │ tracking_data_backup_20250115.zip │ Success │ 482 MiB   │
-- │ backup-2025-01-15...              │ Success │ 504 MiB   │
-- └───────────────────────────────────┴─────────┴───────────┘
```

```sql
-- 从备份恢复（注意：需要先 DROP 原表或恢复到新表名）
RESTORE TABLE default.tracking_data
FROM Disk('backups', 'tracking_data_backup_20250115.zip');

-- 恢复到新表
RESTORE TABLE default.tracking_data
FROM Disk('backups', 'tracking_data_backup_20250115.zip')
AS tracking_data_restored;
```

> **原生 BACKUP vs clickhouse-backup**：原生 BACKUP 命令的优势是零外部依赖、SQL 原生支持、语法简洁。但 clickhouse-backup 的功能更丰富——增量备份、ZooKeeper 元数据备份、RBAC 备份、定时调度、多种远程存储支持。生产环境建议 clickhouse-backup 为主，原生 BACKUP 为辅（快速 ad-hoc 备份场景）。

```bash
#!/bin/bash
# backup_drill.sh —— 备份恢复月度演练脚本
set -euo pipefail

BACKUP_NAME="drill_$(date +%Y%m%d_%H%M)"

START_TIME=$(date +%s)

# 3. 上传至 S3

clickhouse-client -q "TRUNCATE TABLE default.tracking_data_if_exists" || true

echo "Step 5: 执行恢复..." | tee -a "$LOG"
clickhouse-backup download -c "$CONFIG" "$BACKUP_NAME"
clickhouse-backup restore -c "$CONFIG" "$BACKUP_NAME"

echo "Step 6: 验证恢复结果..." | tee -a "$LOG"
END_TIME=$(date +%s)

else
    echo "✗ 验证失败：数据量不一致！备前=$BEFORE_COUNT 备后=$AFTER_COUNT" | tee -a "$LOG"
```

## 4. 项目总结

### 备份方式全方位对比

| 方式 | 原理 | 速度 | 额外存储 | 一致性级别 | 恢复难度 | 适用场景 |
|------|------|------|---------|-----------|---------|---------|
| **FREEZE** | 硬链接快照 | 极快（秒级） | 零（同 inode） | 表级一致 | 手动，需理解 Part 结构 | 紧急停服前快照、升级前保护 |
| **clickhouse-backup** | FREEZE + 自动化搬运 | 中等（受带宽限制） | 独立存储 | 表级一致 + ZK 元数据 | 半自动，一条命令恢复 | 日常备份主力、增量备份 |
| **原生 BACKUP TABLE** | SQL 原生导出 | 中等 | 磁盘/S3 | 表级一致 | 简单，SQL 直接恢复 | 快速 ad-hoc 备份、迁移 |
| **文件系统快照（LVM/ZFS）** | 存储层 COW 快照 | 极快（秒级） | 取决于变更量 | 崩溃一致 | 需文件系统知识 | 全集群级别的灾难恢复 |

### 适用场景与方案推荐

**场景一：核心业务表（RPO < 1h / RTO < 30min）**
```
全量备份：每天凌晨 2:00（clickhouse-backup → S3）
增量备份：每小时（clickhouse-backup --diff-from → S3）
立即可用快照：每次发版前（FREEZE + 本地临时保留）
```

**场景二：历史归档表（查询频率低，可容忍较长 RTO）**
```
全量备份：每周一次（原生 BACKUP TABLE → S3 Glacier）
增量备份：不需要（该表数据极少变更）
保留周期：S3 生命周期自动归档为 Glacier Deep Archive
```

**场景三：元数据与配置（高优先级，小数据量）**
```
clickhouse-backup --rbac --configs → 本地 + S3
每次修改用户/权限后手动触发一次
```

### 注意事项

1. **FREEZE 的硬链接限制**：不能跨文件系统，`/data` 在 XFS 上但备份目录在 NFS mount 上时，硬链接自动退化为真实拷贝（`cp` 不带 `-l` 时），备份时间从秒级变小时级。解决方案：在 `data_path` 所在磁盘上预留一个本地 `backup_staging` 目录做中转。

2. **Merge 期间的备份一致性**：FREEZE 操作本身是安全的——它列出当前活跃 Part 列表然后对其创建硬链接。但如果在 FREEZE 之后、拷贝完成之前发生了 Merge，旧的 Part 变成非活跃状态——不过这没关系，因为 shadow 里的硬链接依然有效。真正的风险是**带着正在进行的 Mutation 做备份**——建议备份前确认 `system.mutations` 中无未完成的 Mutation。

3. **ZooKeeper 元数据必须单独备份**：对于 ReplicatedMergeTree 表，clickhouse-backup 的 `restore` 命令默认会重建 ZK 元数据。但如果手动恢复，你必须从备份的 ZK 元数据中恢复 `/clickhouse/tables/` 路径下对应的节点，否则副本间可能出现数据冲突。

4. **恢复前必须停止写入**：恢复期间如果业务仍在写入同一张表，新写入的 Block Numbers 可能与恢复的 Block Numbers 冲突，导致 `KEEPER_EXCEPTION`。恢复时应先 `SYSTEM STOP FETCHES` 或临时切流，恢复完成后再打开。

### 常见踩坑经验

**坑 1：FREEZE 后忘记清理 shadow 目录，磁盘悄悄被占满**

FREEZE 创建硬链接时几乎不占额外空间，但如果原始 Part 被 Merge 删除，shadow 中的硬链接就成了数据的唯一持有者——此时磁盘空间不会释放。如果你一个月 FREEZE 了 30 次且从未清理，磁盘上的"幽灵 Part"可能占满整个盘。**经验**：定期执行 `ALTER TABLE ... UNFREEZE` 或 cron 清理 `shadow/` 目录，保留最近 3 次 FREEZE 即可。

**坑 2：clickhouse-backup 还原时表已存在导致失败**

`clickhouse-backup restore` 要求目标表**不存在**。如果原表还在（即使已清空），恢复会报错退出。**经验**：恢复前先 DROP 原表，或者在 `config.yml` 中保持 `restore_as_attach: true`——这会以 ATTACH 方式挂载 Part 而非重建表结构。

**坑 3：S3 备份路径配置错误导致静默失败**

原生 `BACKUP TABLE ... TO S3(...)` 如果目标 bucket 或路径不存在，**不会立即报错**，而是在 `system.backups` 中记录一条 `status = 'Failed'` 的条目。很多同学输完命令就关了终端，以为备份成功了，实际是失败的。**经验**：备份后务必执行 `SELECT * FROM system.backups ORDER BY start_time DESC LIMIT 1` 检查状态。

**坑 4：ReplicatedMergeTree 表恢复后副本数据被覆盖**

如果只恢复了数据文件，没有恢复 ZooKeeper 元数据就启动 ClickHouse，集群中其他副本会认为这个节点"落后"了，然后通过 ReplicatedMergeTree 的同步机制用**其他副本的当前数据**覆盖掉你刚恢复的 Part——等于白恢复了。**经验**：恢复 ReplicatedMergeTree 表时使用 clickhouse-backup 的 `restore --rm` 参数重建 Replica，它会自动清理旧的 ZK 节点并重新注册。
