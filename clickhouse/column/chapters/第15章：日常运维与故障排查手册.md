# 第15章：日常运维与故障排查手册

> **版本**：ClickHouse 25.x LTS
> **定位**：运维实战核心章节。建立标准化故障响应流程，将 MTTR（平均修复时间）从小时级压缩到分钟级。
> **前置阅读**：第4章（MergeTree 家族概览）、第12章（副本与复制机制）、第14章（监控与告警体系）
> **预计阅读**：40 分钟 | **实战耗时**：60 分钟

---

## 1. 项目背景

周一早晨九点零三分，某社交 App 数据团队的 Slack 频道炸了。客户端运营群里十几条@全部人的消息滚动得飞快——"数据后台白屏了！""昨天的新增用户报表呢？""老板的实时看板怎么全是 N/A？"

运维工程师阿杰冲进机房，眼前的 Grafana 监控大屏一片猩红：ClickHouse 四个节点全部 CPU 100%，所有查询卡死在队列中，连最简单的 `SELECT 1` 也要二十秒才返回。阿杰心一横，直接 `systemctl restart clickhouse-server`——五分钟过去，服务确实恢复了。可好景不长，十点半故障重现，一模一样的症状。这一回阿杰不敢再盲目重启，硬着头皮翻了三小时日志才找到真凶：一个实习生周末写了个 ETL 脚本，按一条一 INSERT 的方式往 `impressions_log` 表灌了 600 万行——每条一行，生成了 600 万个微型 Part。MergeTree 的后台 Merge 调度器当场崩溃，报错 `DB::Exception: Too many parts (3000). Merges are processing significantly slower than inserts.`

这不是这家公司第一次遭遇线上故障。过去半年，同一个集群上演过至少六种不同的"死法"：

- **CPU 风暴**：某个业务方写的 `SELECT DISTINCT` 子查询扫描了 200 亿行，把 48 核 CPU 全部打满。
- **OOM 连环杀**：一个 `GROUP BY` 语句的高基数列（数千万个唯一键）撑爆了 64GB 内存，OOM Killer 顺手把相邻的 ZooKeeper 进程也杀了，导致整个集群副本全部进入只读模式。
- **磁盘写满**：运维做过一次备份 `FREEZE` 操作后忘记清理 `/var/lib/clickhouse/shadow/` 下的冻结副本，两周累积了 800GB 残留文件，磁盘使用率达到 99%，Merge 线程无法申请临时空间，所有表的 Part 合并全部搁浅。
- **ZK 会话雪崩**：网络抖动导致 ClickHouse 与 ZooKeeper 之间心跳超时，11 张 ReplicatedMergeTree 表同时掉线，业务报表全部显示"只读模式"。

每次故障，团队的处理路径都惊人地相似：监控先报警 → 群聊里互相问"谁改了啥" → 找不到人背锅 → 重启服务碰运气 → 运气不好就拉大师远程支援 → 修好以后写个微信文档记录一下（然后就没人再看过了）。整个团队的 MTTR 长期在 2 小时以上，而且同类故障反复上演，因为根因从未被系统性地沉淀下来。

CTO 终于拍桌子了："给你们两周时间，把 ClickHouse 的运维 SOP 搞出来。以后值班的同学拿着手册就能处理 80% 的常见故障，不用每次都摇人。"

大师临危受命，带着小胖和小白开始了为期两周的故障排查体系搭建。

**核心痛点**：缺少标准化的故障分级、诊断、修复流程；每次事故都要高级工程师从零开始排查；根因不文档化导致同类故障反复出现；运维经验散落在个人大脑中，没有形成团队资产。

## 2. 项目设计：剧本式交锋对话

周三下午，大师在会议室白板上画了一张图——一棵倒立的树，根部写着"症状"，树干是"诊断树"，树冠分叉成各种"修复动作"。

**小胖**（把腿翘在旁边的椅子上）："大师，搞这么复杂干嘛？出问题了重启不就行了？大不了重装！我以前管 MySQL 就这么干的——先 `kill -9`，再 `service mysqld start`，三步走，一分钟搞定。哪要什么排查手册？"

**大师**（放下马克笔，转过头来）："小胖，你上次重启 ClickHouse 那次，故障好了吗？"

**小胖**（缩了缩脖子）："呃……好了五分钟。"

**大师**："五分钟——因为你没解决问题本身。那 600 万个 Part 还在磁盘上，重启只是中断了当前失败的 Merge 任务，但 ClickHouse 一启动，Merge 调度器又看到了那 600 万个小 Part，又冲上去合并——然后又卡死。你只是一遍又一遍地按下'重试'按钮，没清除真正的故障源。"

**小胖**（坐直了身体）："那怎么办？我总不能盯着日志看三小时吧？"

**大师**："关键不是'看多久'，而是'看哪里'和'看什么'。ClickHouse 的故障排查是有套路的——**90% 的故障都落在五个类别里**：CPU、内存、磁盘、ZooKeeper、连接数。你要做的不是逐个排查，而是按决策树走：先看 `system.processes` 定位当前请求，再看 `system.metrics` 和 `system.errors` 抓住异常指标，最后才去看 `clickhouse-server.err.log` 里的详细堆栈。从粗到细，从系统表到日志——这叫**漏斗式诊断法**。"

**技术映射 #1**：诊断优先级 = `system.processes`（当前运行状态）→ `system.metrics`（计数器快照）→ `system.query_log`（历史查询画像）→ `system.errors`（错误统计）→ 错误日志文件。从系统表入手比直接翻日志快 10 倍。

---

**小白**（翻开笔记本，推了推眼镜）："大师，我注意到 ClickHouse 的错误信息格式很统一——`DB::Exception: <message>, Stack trace`。有没有一个常用错误码的速查表？比如我看到 `Too many parts`、`Memory limit exceeded`、`Table is in readonly mode` 这些，能立刻知道下一步该查什么？"

**大师**（赞许地看了一眼小白）："好问题，这就是故障排查手册的核心价值——**错误码→自动关联→修复动作**。ClickHouse 的异常体系分三层：

**第一层：配置类错误**。比如 `DB::Exception: Limit for parts exceeded: 3000`——这是 `max_parts_in_total` 参数限制。它不告诉你'为什么 Parts 这么多'，只告诉你'触发了阈值'。排查方向：`system.parts` 查 Part 数量分布，找到攒了大量 Part 的源头表，然后 `SYSTEM STOP MERGES` 先止血，再考虑降插入频率或调参。

**第二层：资源类错误**。比如 `Memory limit (total) exceeded`——这说明查询的 `max_memory_usage` 设置不足以容纳中间计算结果。排查方向：看 `system.query_log` 里同一个 `query_id` 的 `memory_usage` 曲线，找出哪个算子（通常是 `GROUP BY` 或 `ARRAY JOIN`）吃掉了大部分内存。

**第三层：状态类错误**。比如 `Table default.some_table is in readonly mode (zookeeper session expired)`——这是 ZK 会话断开。排查方向：`system.replication_queue` 看副本积压队列，`system.zookeeper` 测连通性，优先 `SYSTEM RESTART REPLICA` 而非重启 ClickHouse。"

**技术映射 #2**：ClickHouse 的错误消息通常带有明确的错误码（如 `Error 241`、`Error 159`），在官方文档中可查详细说明。日志中的 `Stack trace` 部分应关注最顶层的调用栈——它通常是错误的直接触发点。

---

**小白**（又翻了一页）："那如果故障发生在凌晨两点，值班的同学可能不熟悉 ClickHouse 架构——手册上需要写清楚什么时候该止血，什么时候该深挖根因，什么时候该升级求助，对吗？"

**大师**："这就是**分级响应**的概念。我给个框架：

**Level 1（服务不可用）**：整个集群宕机或所有查询超时。不用深挖根因，优先止血——重启、切流、降级。目标是恢复服务，不是理解故障原理。耗时超过 15 分钟直接升级。

**Level 2（性能退化）**：查询能返回结果但比平时慢 10 倍以上。需要做快速诊断：`system.processes` 看有没有出格查询，`system.metrics` 看 `Query`、`Merge`、`ReadonlyReplica` 三个指标。如果是一个大查询造成的，直接 Kill；如果是 Merge 风暴，`SYSTEM STOP MERGES` 先止损。

**Level 3（异常预警）**：告警项目接近阈值但未突破，比如 Parts 数量达到 1500（上限 3000）。这时不需要紧急操作，但要做一份趋势报告，分析写入模式和 Merge 速率，在周五变更窗口统一处理。"

**小胖**（挠了挠头）："那是不是我把这些套路全记住了，以后就不用麻烦大师你了？"

**大师**（笑了笑）："你能处理 Level 1 和 Level 2 我就谢天谢地了。Level 3 的深水区——比如 `KEEPER_EXCEPTION` 背后的 ZK 事务日志膨胀、`Exchange` 算子的跨节点数据传输瓶颈、`S3 disk` 的读写延迟毛刺——这些确实需要经验值。但手册要有免责声明：**什么情况下该停止自救、立即升级**。最常见的升级信号：你执行了手册上的三个修复动作，故障依然存在——那就别试第四个，直接升级。"

**技术映射 #3**：SOP 手册必须包含**升级条件**——运维不是逞英雄，设置明确的升级阈值可以防止错误的"自救"操作引发二次故障。典型升级条件：同一故障 30 分钟内未恢复；单次操作影响超过 3 张核心业务表；需要修改 ZK 元数据或执行 `ALTER TABLE ... FREEZE` 等不可逆操作。

---

**小胖**（突然想到什么）："等等，那 ClickHouse 版本升级怎么办？上次我从 23.3 升到 24.8，搞崩了一个下午——好多 SQL 语法报错。"

**大师**："版本升级是另一个必须写进手册的专题。ClickHouse 的升级策略分两种：

**停机升级（Stop Upgrade）**：停掉所有节点，替换二进制，启动。最简单但中断时间长——适合非生产环境或小数据量集群。注意，停机前必须 `SYSTEM FLUSH LOGS`，确保所有日志落盘。

**滚动升级（Rolling Upgrade）**：一台一台来，每台逐次停→升→启，中间等待副本同步完成。滚动升级的核心风险是**版本不兼容**——新版本的 Server 可能修改了 ZK 里的元数据格式，导致旧版本节点无法读取副本状态。所以 ClickHouse 官方要求：滚动升级跨越的版本不能超过一个 LTS 版本；跨大版本（如 23.x → 25.x）强烈建议停机升级，并且在 `config.xml` 中关闭 `compatibility` 检查前先做一次灰度验证。

小白，过来记一笔——升级前必做的三件事：**备份 `config.xml` 和 `users.xml`**（新版本可能覆盖）；**跑一遍 `clickhouse-client --dry-run` 语法兼容检查**（用全量业务 SQL 集）；**首先从一台非核心节点开始**，升级后观察 30 分钟再推进。这些写不死你，但不写会害死你。"

## 3. 项目实战

### 环境准备

我们需要一个配置了真实运维参数的 ClickHouse 环境。使用 Docker 挂载自定义日志和系统配置：

```bash
# 创建配置目录
mkdir -p ~/clickhouse-ch15/config.d
mkdir -p ~/clickhouse-ch15/logs
mkdir -p ~/clickhouse-ch15/data
```

创建日志配置文件 `~/clickhouse-ch15/config.d/logger.xml`：

```xml
<clickhouse>
    <logger>
        <level>information</level>
        <log>/var/log/clickhouse-server/clickhouse-server.log</log>
        <errorlog>/var/log/clickhouse-server/clickhouse-server.err.log</errorlog>
        <size>100M</size>
        <count>10</count>
    </logger>
</clickhouse>
```

创建系统参数配置文件 `~/clickhouse-ch15/config.d/ops_settings.xml`：

```xml
<clickhouse>
    <max_concurrent_queries>100</max_concurrent_queries>
    <max_connections>1024</max_connections>
    <max_memory_usage>10000000000</max_memory_usage>
    <max_bytes_before_external_group_by>5000000000</max_bytes_before_external_group_by>
    <max_parts_in_total>3000</max_parts_in_total>
    <max_table_size_to_drop>0</max_table_size_to_drop>
    <log_queries>1</log_queries>
    <query_log_retention_size>1000</query_log_retention_size>
    <query_log_retention_time>30</query_log_retention_time>
</clickhouse>
```

启动 ClickHouse 容器：

```bash
docker run -d \
  --name clickhouse-ops \
  --ulimit nofile=262144:262144 \
  -p 8123:8123 -p 9000:9000 \
  -v ~/clickhouse-ch15/config.d:/etc/clickhouse-server/config.d \
  -v ~/clickhouse-ch15/logs:/var/log/clickhouse-server \
  -v ~/clickhouse-ch15/data:/var/lib/clickhouse \
  clickhouse/clickhouse-server:25.3
```

验证服务正常：

```bash
docker exec -it clickhouse-ops clickhouse-client -q "SELECT version()"
```

### 分步实现

#### Step 1：故障——CPU 100% 排查

**场景模拟**：某个分析同事提交了一个全表扫描的大查询，将所有 CPU 核心打满。

```bash
# 在 ClickHouse 客户端中执行以下查询（后台运行，持续消耗 CPU）
docker exec -it clickhouse-ops clickhouse-client -q "
SELECT groupArray(100000000)(number) FROM numbers(100000000)
" &
```

此时集群 CPU 使用率会迅速飙升。我们按照漏斗式诊断法逐步排查：

**第一层：检查当前运行查询（system.processes）**

```sql
SELECT
    query_id,
    user,
    toUInt32(elapsed) AS elapsed_sec,
    formatReadableSize(memory_usage) AS mem,
    read_rows,
    formatReadableSize(read_bytes) AS read_bytes,
    substring(query, 1, 120) AS query_preview
FROM system.processes
WHERE elapsed > 1
ORDER BY elapsed DESC;
```

查询结果中，`elapsed_sec` 最高的那条记录就是罪魁祸首。记下它的 `query_id`。

**第二层：快速止血——Kill 异常查询**

```sql
-- 替换为实际的 query_id
KILL QUERY WHERE query_id = 'a1b2c3d4-e5f6-7890-abcd-ef1234567890';
```

如果 `KILL QUERY` 无法终止（某些算子不支持中断），可以使用更暴力的方式：

```sql
-- Kill 特定用户的全部连接（谨慎使用）
SELECT
    format('KILL QUERY WHERE query_id = ''{0}'';', query_id)
FROM system.processes
WHERE user = 'default' AND elapsed > 60;
```

**第三层：事后溯源——从 query_log 定位历史 CPU 大户**

```sql
SELECT
    query_start_time,
    query_duration_ms,
    formatReadableSize(memory_usage) AS mem_usage,
    read_rows,
    read_bytes,
    formatReadableSize(result_size) AS result_size,
    substring(query, 1, 100) AS query_preview
FROM system.query_log
WHERE type = 'QueryFinish'
  AND query_duration_ms > 30000
ORDER BY memory_usage DESC
LIMIT 10;
```

**关键诊断指标对照**：

| 指标 | 正常范围 | 告警阈值 | 排查方向 |
|------|---------|---------|---------|
| `query_duration_ms` | <5000 | >60000 | 全表扫描、缺少分区过滤 |
| `read_rows` / `result_rows` | <10 | >10000 | 大量数据扫描后丢弃，索引失效 |
| `memory_usage` / `read_rows` | 合理 | 异常高 | GROUP BY 高基数列导致哈希表膨胀 |

#### Step 2：故障——内存溢出（OOM）

**场景模拟**：某业务方执行了一个超高基数的 GROUP BY 聚合查询。

```sql
-- 故意设置极低的内存限制来触发 OOM
SET max_memory_usage = 100000000;  -- 100MB

-- 执行高基数 GROUP BY，必然会触发内存溢出
SELECT
    number AS high_cardinality_key,
    count() AS cnt,
    sum(number) AS total
FROM numbers(50000000)
GROUP BY high_cardinality_key
FORMAT Null;
-- 错误：DB::Exception: Memory limit (total) exceeded: would use 512.00 MiB
```

**诊断第一步：确认当前内存配置**

```sql
SELECT name, value, changed
FROM system.settings
WHERE name LIKE '%memory%'
  AND name IN (
    'max_memory_usage',
    'max_memory_usage_for_user',
    'max_bytes_before_external_group_by',
    'memory_profiler_step',
    'max_server_memory_usage'
  )
ORDER BY name;
```

**诊断第二步：从 query_log 回溯 OOM 事故现场**

```sql
SELECT
    query_start_time,
    substring(query, 1, 120) AS query_preview,
    formatReadableSize(memory_usage) AS peak_memory,
    read_rows,
    exception
FROM system.query_log
WHERE type = 'ExceptionWhileProcessing'
  AND exception LIKE '%Memory limit%'
ORDER BY query_start_time DESC
LIMIT 10;
```

**诊断第三步：查看服务器级内存使用情况**

```sql
SELECT
    metric,
    formatReadableSize(value) AS size
FROM system.metrics
WHERE metric IN ('MemoryTracking', 'MemoryResident')
UNION ALL
SELECT
    event AS metric,
    toString(value) AS size
FROM system.events
WHERE event IN ('MemoryAlloc', 'MemoryFree')
ORDER BY metric;
```

**修复方案（由轻到重）**：

1. **SQL 级优化**：对于高基数 GROUP BY，可以考虑添加 `LIMIT` 限制返回值、使用 `SAMPLE` 子句降采样、或者用 `quantile` 等近似函数替代精确聚合。
2. **配置级优化**：启用 `max_bytes_before_external_group_by`——当 GROUP BY 哈希表超过此值，自动溢出到磁盘，以时间换空间。
   ```sql
   SET max_bytes_before_external_group_by = 2000000000;  -- 2GB 后磁盘溢出
   ```
3. **表结构优化**：如果业务确实需要超高基数 GROUP BY，考虑用 `LowCardinality` 修饰符压缩字典大小。
4. **集群级优化**：部署分布式查询，让 GROUP BY 阶段在分片节点上先做局部聚合，再在查询节点做全局聚合，减少单节点内存压力。

---

#### Step 3：故障——磁盘空间满

**场景模拟**：数据持续写入但未配置 TTL 清理策略，或 FREEZE 操作后残留文件未清理。

```bash
# 检查 ClickHouse 数据目录的磁盘使用情况
docker exec -it clickhouse-ops du -sh /var/lib/clickhouse/data/*
docker exec -it clickhouse-ops df -h /var/lib/clickhouse/
```

**诊断第一步：查询各表的磁盘占用排行**

```sql
SELECT
    database,
    table,
    formatReadableSize(sum(bytes_on_disk)) AS total_size,
    formatReadableSize(sum(data_compressed_bytes)) AS compressed,
    count() AS part_count,
    max(modification_time) AS latest_modification
FROM system.parts
WHERE active
GROUP BY database, table
ORDER BY sum(bytes_on_disk) DESC
LIMIT 15;
```

**诊断第二步：检查非活跃Part（待删除但尚未清理的旧Part）**

```sql
SELECT
    database,
    table,
    count() AS inactive_parts,
    formatReadableSize(sum(bytes_on_disk)) AS wasted_space
FROM system.parts
WHERE NOT active
GROUP BY database, table
ORDER BY sum(bytes_on_disk) DESC;
```

**诊断第三步：检查 Freeze 残留目录（冷备份未清理）**

```bash
# 进入 ClickHouse 容器检查 shadow 目录
docker exec -it clickhouse-ops ls -la /var/lib/clickhouse/shadow/
# 如果有残留的冻结备份文件，会显示类似：
# 20250101_01_01_0/  20250201_01_01_0/  ...
```

```sql
-- 同时确认是否有表做过 FREEZE 操作
SELECT
    database, table,
    formatReadableSize(freezed_bytes) AS freezed_size
FROM system.tables
WHERE freezed_bytes > 0
ORDER BY freezed_bytes DESC;
```

**修复方案**：

```sql
-- 方案A：删除指定历史分区（最直接）
ALTER TABLE ops_test_log DROP PARTITION '202401';

-- 方案B：对有 TTL 的表立即触发清理
ALTER TABLE ops_test_log MATERIALIZE TTL;

-- 方案C：清理非活跃Part（合并已完成，旧Part可以安全删除）
-- 手动触发：等待后台自动清理（默认 8 分钟后清理），或重启以加速清理
```

```bash
# 方案D：清理 Freeze 残留
# 警告：确认 shadow 目录下的数据确实已经备份完成！
docker exec -it clickhouse-ops rm -rf /var/lib/clickhouse/shadow/*
# 或使用 ClickHouse 系统命令
docker exec -it clickhouse-ops clickhouse-client -q "
SELECT 'ALTER TABLE ' || database || '.' || table || ' UNFREEZE NAME ''' || name || ''' WITH CLEANUP;'
FROM system.tables
WHERE freezed_bytes > 0
FORMAT TSV
"
```

**磁盘空间的"隐藏杀手"清单**：

1. **Freeze 残留**（`shadow/` 目录）——每次 `ALTER TABLE ... FREEZE` 都会在此创建一个硬链接副本，必须手动清理。
2. **Mutation 失败的临时目录**——`/var/lib/clickhouse/tmp/` 下的中间文件在异常中断后可能不会被清理。
3. **日志文件膨胀**——如果 `logger.xml` 中 `level` 设置为 `trace` 或 `debug`，短时间内可产生 GB 级日志。
4. **Detached Parts**——`ALTER TABLE ... DETACH PARTITION` 后 Part 不会被删除，只是不再参与查询，仍在占用磁盘。

#### Step 4：故障——ZooKeeper 会话超时

**场景模拟**：网络抖动导致 ClickHouse 节点与 ZooKeeper 之间的会话超时，ReplicatedMergeTree 表进入只读模式。

```sql
-- 症状观察：SELECT 或 INSERT 操作返回以下错误
-- DB::Exception: Table is in readonly mode (zookeeper session expired)
-- DB::Exception: Keeper error: connection loss
```

**诊断第一步：测试 ZooKeeper 连通性**

```sql
-- 检查 ZK 是否可达
SELECT * FROM system.zookeeper WHERE path = '/' LIMIT 1;

-- 如果返回结果正常，说明 ZK 连接已恢复
-- 如果持续返回 error，则 ZK 本身可能存在问题，需检查 ZK 集群
```

**诊断第二步：检查副本复制队列积压情况**

```sql
SELECT
    database,
    table,
    type,
    new_part_name,
    num_tries,
    num_postponed,
    last_exception_time,
    substring(last_exception, 1, 150) AS last_exception_preview,
    last_attempt_time
FROM system.replication_queue
ORDER BY last_attempt_time DESC
LIMIT 20;
```

**诊断第三步：检查副本状态与延迟**

```sql
SELECT
    database,
    table,
    is_leader,
    is_readonly,
    is_session_expired,
    zookeeper_path,
    replica_path,
    queue_size,
    absolute_delay,
    formatReadableSize(total_replicas_size) AS replica_size,
    active_replicas
FROM system.replicas
WHERE is_session_expired = 1
   OR absolute_delay > 300
ORDER BY absolute_delay DESC;
```

**修复方案（从轻到重）**：

```sql
-- 方案A：单独重启特定表的 ZK 连接（最小影响）
SYSTEM RESTART REPLICA <database>.<table_name>;
-- 例如：
SYSTEM RESTART REPLICA default.ops_test_log;

-- 方案B：如果大量表同时掉线，重启所有副本连接
SYSTEM RESTART REPLICAS;

-- 方案C：如果以上无效，考虑软重启 ClickHouse
-- 注意：重启前先 SYSTEM FLUSH LOGS 确保日志落盘
SYSTEM FLUSH LOGS;
-- 然后在宿主机上 systemctl restart clickhouse-server
```

**防止 ZK 会话超时的配置调优**：

```xml
<!-- config.xml 中添加以下配置 -->
<zookeeper>
    <node>
        <host>zk1.example.com</host>
        <port>2181</port>
    </node>
    <session_timeout_ms>30000</session_timeout_ms>
    <operation_timeout_ms>10000</operation_timeout_ms>
    <root>/clickhouse</root>
</zookeeper>
```

**ZK 故障排查清单**：

| 检查项 | 系统表/命令 | 正常标准 | 异常处理 |
|--------|------------|---------|---------|
| ZK 连通性 | `system.zookeeper WHERE path = '/'` | 返回结果 | 检查 ZK 集群状态和网络 |
| 会话是否过期 | `system.replicas.is_session_expired` | `0` | `SYSTEM RESTART REPLICA` |
| 副本只读比例 | `system.replicas.is_readonly` | 无只读副本 | 排查 ZK 连接，逐表恢复 |
| 复制队列积压 | `system.replication_queue` | `num_tries < 5` | 诊断积压原因（Merge慢/ZK慢） |
| 副本延迟大小 | `system.replicas.absolute_delay` | `< 300` 秒 | 增加 `max_replicated_fetches` 并发 |

#### Step 5：故障——连接数耗尽

**场景模拟**：某个微服务未正确关闭数据库连接，连接池泄露导致连接数达到上限。

```sql
-- 症状：客户端报错
-- DB::Exception: Too many simultaneous queries. Maximum: 100
-- DB::Exception: Cannot allocate new connection
```

**诊断第一步：查看连接池使用情况**

```sql
SELECT
    metric,
    value,
    formatReadableSize(value) AS display
FROM system.metrics
WHERE metric IN ('Connection', 'Query', 'TCPConnection', 'HTTPConnection')
ORDER BY metric;
```

**诊断第二步：找到长时间占用连接的会话**

```sql
SELECT
    query_id,
    user,
    toUInt32(elapsed) AS elapsed_sec,
    client_hostname,
    interface,
    substring(query, 1, 80) AS query_preview
FROM system.processes
ORDER BY elapsed DESC
LIMIT 20;
```

**诊断第三步：检查各用户的并发连接设置**

```sql
SELECT
    name,
    value
FROM system.settings
WHERE name LIKE '%max_concurrent%'
   OR name LIKE '%connection%'
   OR name LIKE '%thread%'
ORDER BY name;
```

**修复方案**：

```sql
-- 方案A：批量清理空闲连接（找出超过 10 分钟的查询并 Kill）
SELECT
    format('KILL QUERY WHERE query_id = ''{0}'';', query_id)
FROM system.processes
WHERE elapsed > 600;

-- 方案B：调整用户级最大并发限制
ALTER USER default SETTINGS max_concurrent_queries = 5;

-- 方案C：调整服务器级连接限制
-- 修改 config.xml 中的 max_concurrent_queries 和 max_connections
-- 然后热加载配置
SYSTEM RELOAD CONFIG;
```

**连接数管理的三条黄金法则**：

1. **每个客户端连接必须设置超时**：`SET idle_connection_timeout = 300`（5分钟无活动自动断开）。
2. **分离业务用户配额**：核心报表用户分配较高的 `max_concurrent_queries`，ETL 用户和临时查询用户各设较低的配额，互不干扰。
3. **应用层实现连接池**：如果使用 Java/Python 客户端，务必实现连接池（如 HikariCP），避免每次查询都建立新连接。

#### Step 6：运维命令速查表

以下命令构成 ClickHouse 日常运维的"三分钟急救面板"——所有命令均为非阻塞操作，可在不影响生产查询的前提下执行：

```sql
-- ==================== Merge 调度管控 ====================
-- 停止所有后台 Merge（止血：Part 过多导致的 Merge 风暴）
SYSTEM STOP MERGES;

-- 恢复 Merge 调度
SYSTEM START MERGES;

-- 停止特定表的 Merge
SYSTEM STOP MERGES default.ops_test_log;

-- 停止数据变更（Mutation）操作
SYSTEM STOP TTL MERGES;

-- 恢复 TTL Merge
SYSTEM START TTL MERGES;

-- ==================== 副本与分发管理 ====================
-- 停止向副本分发数据 Part
SYSTEM STOP FETCHES default.ops_test_log;

-- 恢复副本分发
SYSTEM START FETCHES default.ops_test_log;

-- 停止将本地 Part 发送给其他副本
SYSTEM STOP REPLICATED SENDS default.ops_test_log;

-- 恢复副本发送
SYSTEM START REPLICATED SENDS default.ops_test_log;

-- 手动触发副本同步
SYSTEM SYNC REPLICA default.ops_test_log;

-- ==================== 配置与缓存 ====================
-- 热加载配置文件（不重启服务）
SYSTEM RELOAD CONFIG;

-- 热加载用户和权限配置
SYSTEM RELOAD USERS;

-- 清除 DNS 缓存（上游 DNS 变更后必须执行）
SYSTEM DROP DNS CACHE;

-- 清除 Mark 缓存（发现查询结果异常时尝试）
SYSTEM DROP MARK CACHE;

-- 清除未压缩缓存
SYSTEM DROP UNCOMPRESSED CACHE;

-- ==================== 日志管理 ====================
-- 强制刷新所有日志缓冲到磁盘（重启前必执行）
SYSTEM FLUSH LOGS;

-- 手动触发 system.query_log 和 system.part_log 等日志落盘
SYSTEM FLUSH LOGS;

-- ==================== 信息查询类 ====================
-- 版本信息
SELECT version();

-- 运行时间
SELECT formatReadableTimeDelta(uptime()) AS uptime;

-- 各表磁盘占用排行
SELECT
    database, table,
    formatReadableSize(sum(bytes_on_disk)) AS size,
    count() AS parts
FROM system.parts
WHERE active
GROUP BY database, table
ORDER BY sum(bytes_on_disk) DESC
LIMIT 10;

-- 后台 Merge 任务进度
SELECT
    database, table,
    elapsed,
    progress,
    num_parts,
    total_size_bytes_compressed
FROM system.merges
ORDER BY elapsed DESC;

-- Mutation 任务执行状态
SELECT
    database, table,
    mutation_id,
    command,
    parts_to_do,
    is_done
FROM system.mutations
WHERE NOT is_done
ORDER BY create_time DESC;

-- 活跃 Part 总数（全局）
SELECT count() AS total_parts,
       countIf(active) AS active_parts,
       countIf(NOT active) AS inactive_parts
FROM system.parts;

-- 查看所有用户的配额和权限
SELECT name, storage_type FROM system.users;
SELECT * FROM system.quotas;
```

## 4. 项目总结

### 故障排查 SOP 速查表

| 症状 | 可能原因 | 首要检查 | 快速修复 | 根因解决 |
|------|---------|---------|---------|---------|
| **CPU 100%** | 大查询 / Merge风暴 | `system.processes` 找到长查询 | `KILL QUERY` 或 `SYSTEM STOP MERGES` | 优化 SQL；降低插入频率；增加 `max_bytes_to_merge_at_max_space_in_pool` |
| **内存溢出（OOM）** | 高基数 GROUP BY / ARRAY JOIN 膨胀 | `system.query_log` 中 Exception 记录 | 增加 `max_memory_usage`；启用 `max_bytes_before_external_group_by` | 改写 SQL 降基数；加 LIMIT；使用 `LowCardinality` 类型 |
| **磁盘满** | 数据无 TTL 清理 / Freeze残留 / 日志膨胀 | `system.parts` 按表汇总磁盘占用 | `DROP PARTITION`；清理 `shadow/`；调低 `logger.level` | 配置 TTL 自动清理；建立 Freeze → Backup → Cleanup SOP |
| **ZK 会话超时** | 网络抖动 / ZK 过载 / 会话超时配置过小 | `system.replicas.is_session_expired` | `SYSTEM RESTART REPLICA` / `SYSTEM RESTART REPLICAS` | 增大 `session_timeout_ms`；优化 ZK 集群；上 Keeper |
| **连接数耗尽** | 连接泄露 / 未设超时 / 瞬时高并发 | `system.metrics` 查看 Connection 指标 | `KILL QUERY` 清理长连接；`ALTER USER ... SETTINGS max_concurrent_queries` | 客户端连接池配置；设置 `idle_connection_timeout`；按用户分级配额 |
| **查询极慢** | 缺少分区裁剪 / 索引失效 / Merge 阻塞 | `system.query_log` 看 `read_rows` vs `result_rows` | `KILL QUERY`；`SYSTEM STOP MERGES` 释放 IO | 优化分区键和 ORDER BY；使用 `SAMPLE` 子句；物化视图预聚合 |
| **表只读** | ZK 磁盘满 / ZK 事务日志溢出 / 副本配置丢失 | `system.replicas.is_readonly` | `SYSTEM RESTART REPLICA` | 清理 ZK 事务日志；检查 `replicated_can_become_leader` 配置 |

### 生产运维巡检检查清单

**日度巡检（每天早上 9:00，耗时 5 分钟）**：

```sql
-- 1. 全局 Part 数量是否有异常增长
SELECT count() FROM system.parts WHERE active;

-- 2. 是否有长时间阻塞的查询
SELECT count() FROM system.processes WHERE elapsed > 300;

-- 3. 副本是否有明显的同步延迟
SELECT count() FROM system.replicas WHERE absolute_delay > 300;

-- 4. 是否有未完成的 Mutation
SELECT count() FROM system.mutations WHERE NOT is_done;

-- 5. 磁盘使用率
-- df -h /var/lib/clickhouse
```

**周度巡检（每周五下午，耗时 15 分钟）**：

```sql
-- 1. 本周 Top 10 慢查询
SELECT query_start_time, query_duration_ms, substring(query,1,120)
FROM system.query_log
WHERE type='QueryFinish' AND query_start_time >= now() - INTERVAL 7 DAY
ORDER BY query_duration_ms DESC LIMIT 10;

-- 2. 各节点内存峰值统计
SELECT hostName(), formatReadableSize(max(memory_usage))
FROM system.query_log
WHERE query_start_time >= now() - INTERVAL 7 DAY
GROUP BY hostName();

-- 3. Merge 成功率与耗时分布
SELECT database, table, count(), avg(elapsed), quantile(0.99)(elapsed)
FROM system.part_log WHERE event_type='MergeParts'
AND event_time >= now() - INTERVAL 7 DAY
GROUP BY database, table;

-- 4. ZK 事务统计（检查是否有异常高峰）
-- 5. 各表的冷热数据比例（验证 TTL 策略是否生效）
```

**月度巡检（每月第一周周一，耗时 30 分钟）**：

- 全量 SQL 查询模板分析（识别高频查询，评估是否需要物化视图加速）
- 各表压缩率评估（`data_compressed_bytes` / `data_uncompressed_bytes`）
- 存储容量趋势预测（根据最近 90 天 disk 增长曲线，推算扩容时间点）
- 用户权限和角色审查（清理已离职员工的用户账户）
- 配置文件变更审计（对比当月与上月的 `config.xml` / `users.xml` diff）

### 适用场景

- **应急响应（Incident Response）**：值班同学打开本手册，按症状→诊断→修复→验证的路径逐项排查，平均处理时间从 2 小时压缩到 15 分钟以内。
- **主动健康巡检（Proactive Health Check）**：通过日/周/月三级巡检脚本，在故障发生前发现潜在风险，实现从"被动救火"到"主动防火"的转变。
- **容量规划（Capacity Planning）**：基于磁盘增长趋势和查询量趋势，提前规划扩容和缩容策略，避免资源浪费或突然耗尽。
- **新人培训（Onboarding）**：新入职的运维工程师可通过本手册在两周内掌握 ClickHouse 基础运维能力，降低团队 bus factor。

### 注意事项

1. **`KILL QUERY` 不适用于 Mutation**：Mutation（`ALTER TABLE ... DELETE/UPDATE`）是异步后台操作，`KILL QUERY` 无法终止它们。要停止 Mutation，需使用 `KILL MUTATION WHERE mutation_id = '...'`，但已写入的临时数据不会回滚。

2. **`SYSTEM RESTART REPLICA` 可能导致短暂只读**：在 ZK 会话重建期间（通常 1-5 秒），该表的复制元数据处于不可用状态，这段时间内的写入请求会被拒绝。建议在业务低峰期执行，或先通过 `SYSTEM STOP REPLICATED SENDS` 将写入流量引导至其他副本。

3. **磁盘满不仅影响写入——也会阻止 Merge**：Merge 操作需要临时磁盘空间来重写 Part。当磁盘使用率超过 90% 时，ClickHouse 会拒绝启动新的 Merge 任务，导致 Part 持续堆积，形成恶性循环——Part 越多，Merge 越难完成，磁盘空间越紧张。这是 ClickHouse 运维中最危险的"死亡螺旋"。

4. **日志级别切换有风险**：临时将日志级别设为 `trace` 进行问题排查后，必须在 30 分钟内切回 `information`，否则日志文件会在数小时内撑满磁盘。

5. **`SYSTEM RELOAD CONFIG` 的生效范围**：并非所有配置都支持热加载。通常 `users.xml` 的权限和配额变更、宏变量、网络配置的变更可以热加载生效，但 `max_server_memory_usage`、`background_pool_size` 等底层线程池参数需要重启才能生效。官方文档中标注 "Reloadable: Yes/No" 的配置项支持热加载。

### 常见踩坑经验

1. **生产环境直接 `KILL MUTATION` 可能导致 Part 损坏**。Mutation 操作修改的是 Part 的物理文件，中断 Mutation 意味着可能存在半修改状态的 Part。正确的做法是：先 `KILL MUTATION`，再执行 `ALTER TABLE ... MODIFY SETTING ...` 重建元数据一致性。如果 Part 已经损坏，使用 `ALTER TABLE ... DETACH PARTITION` 隔离问题分区，再从副本恢复。

2. **`FREEZE` 后忘记清理 `shadow/` 目录持续占用磁盘**。`ALTER TABLE ... FREEZE` 创建的是硬链接（Hard Link），不占用额外的 inode 空间，但 ClickHouse 的磁盘监控并不区分"用户数据"和"冻结副本"——Freeze 残留会让你误判磁盘容量。务必建立 `FREEZE → 备份到 S3/HDFS → UNFREEZE WITH CLEANUP` 的三步 SOP，不要只做 `FREEZE` 而忘了后续清理。

3. **盲目重启 ClickHouse 而非先排查日志**。重启能解决一切问题——除了根因没解决的问题。重启后 ClickHouse 需要执行的恢复流程包括：加载元数据、与 ZK 同步副本状态、重新统计 Parts 信息。在大规模集群中，全量重启后恢复可能需要 20-30 分钟，且期间所有查询都不可用。**重启是最后的手段，不是第一反应**。

4. **忘记 `SYSTEM FLUSH LOGS` 直接关机**——日志缓冲区中未落盘的数据将永久丢失，导致 `query_log` 中出现时间断层，故障溯源时关键证据缺失。

5. **在一台机器上同时跑了 ClickHouse 和 ZooKeeper，但没给 ZK 设置内存上限**——ZK 是 Java 进程，默认 Heap 可以吃光系统内存。一旦 ZK 内存溢出导致 Full GC，ClickHouse 的 ZK 会话全部超时，整集群进入只读模式。Docker 部署时，务必用 `-Xmx` 参数限制 ZK 堆内存。
