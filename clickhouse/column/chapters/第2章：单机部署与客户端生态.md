# 第2章：单机部署与客户端生态

## 1. 项目背景

数据团队最近入职了一位新成员——小胖。入职第一天，小胖就被拉进了团队的ClickHouse群聊，看到前辈们讨论着各种MergeTree、物化视图、分布式表，他跃跃欲试，恨不得立刻上手写几条SQL练练手。

然而现实很快给他泼了一盆冷水。

"那个……测试环境在哪？我怎么连上去？"小胖在群里问。

"我们没有专门的测试环境，"一旁的同事小白苦笑着回答，"就那一套生产集群，大家公用。"

"那我能上去试试吗？我想跑几个聚合查询，看看阵列函数怎么用。"

"小心点！上次老王跑了个不带WHERE的ALTER，差点把整张表锁死，凌晨三点还在回滚数据……"

这就是许多数据团队的典型困境：只有一套生产ClickHouse集群，开发者没有独立的实验环境。每次有新功能想验证、新查询想优化，都要提心吊胆地在生产环境操作。久而久之，团队陷入了"不敢试→技术成长慢→更容易出错→更不敢试"的恶性循环。

小胖叹了口气："那怎么办？我在自己电脑上装个ClickHouse行不行？"

"当然行！而且这正是最佳实践。"数据组的大师从工位上探出头来，"每个开发者都应该有一套本机的ClickHouse沙箱环境。今天我们就来把这件事搞定——让你10分钟内拥有一套完整的单机ClickHouse开发环境，随便折腾，炸了也无所谓。"

本章我们将从零开始，完成ClickHouse的单节点部署、客户端生态配置，以及配置文件的分层管理。这不仅是开发者的第一块敲门砖，更是理解ClickHouse架构的起点。

## 2. 项目设计：剧本式交锋对话

大师把三人叫到白板前，准备先理清思路再动手。

**小胖**（迫不及待地打开终端）："这有什么好设计的？Docker一行命令不就跑起来了？我前天在网上看到的——"

```
docker run -d --name clickhouse clickhouse/clickhouse-server
```

"你看，这就跑起来了！"

**大师**（微微一笑）："跑起来是跑起来了，但我问你几个问题——你的数据存在哪了？容器删了数据还在吗？配置文件怎么改？暴露了哪几个端口？其他工具怎么连上来？"

**小胖**（愣住了）："呃……我还没想那么多。"

**小白**（举手提问）："等等，我也有问题。我之前看到别人装ClickHouse，有的是用yum install，有的是docker pull，我甚至见过有人直接下载二进制包./clickhouse server跑的。到底哪种方式是对的？"

**大师**："好问题！这正好是我们首先要厘清的——ClickHouse有四种主流部署方式，各有适用场景。"

大师在白板上列了一张表：

| 部署方式 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| **RPM/DEB包安装** | 原生性能最好，systemd管理，生产验证充分 | 需要root权限，依赖管理繁琐，卸载清理麻烦 | 生产环境裸金属部署 |
| **Docker** | 环境隔离，快速启停，无依赖冲突，一行命令跑起来 | 有少量性能损耗，需要理解volume和网络配置 | 开发测试、快速验证 |
| **二进制包** | 灵活性极高，可自定义编译参数 | 需要手动管理服务，升级复杂，门槛最高 | 源码级调优、定制化需求 |
| **源码编译** | 完全可控，可添加自定义补丁 | 编译时间长（30min+），C++依赖地狱 | 内核开发、性能极致优化 |

"对我们来说，开发环境选Docker是最合适的：快速、干净、可复现。"大师敲了敲白板，"但光一个docker run不够，我们需要用Docker Compose把ClickHouse和配套的Web UI一起管起来。"

**小白**："Web UI？ClickHouse不是命令行工具吗？"

**大师**："clickhouse-client确实是主力交互方式，但很多时候我们也需要一个可视化界面来快速浏览表结构、画图表。Tabix就是ClickHouse社区最流行的轻量级Web UI。另外，ClickHouse本身就内置了HTTP接口（8123端口），任何能发HTTP请求的工具都能连它。"

**小胖**："等等，端口这块我有点迷糊——8123和9000到底有什么区别？"

**大师**："这是理解ClickHouse客户端生态的关键。记住一句话：**9000端口走ClickHouse原生TCP协议，8123端口走HTTP协议。**"

大师在白板上继续写：

```
┌──────────────────────────────────────────┐
│         ClickHouse Client 生态           │
├─────────────┬────────────────────────────┤
│  Native TCP │    HTTP (REST + JDBC)      │
│  Port 9000  │    Port 8123               │
├─────────────┼────────────────────────────┤
│ clickhouse- │ curl / wget                │
│ client      │ Tabix / Play UI            │
│ C++ Native  │ JDBC Driver                │
│ Go Driver   │ Python clickhouse-driver   │
│ Node.js     │ DBeaver / DataGrip         │
└─────────────┴────────────────────────────┘
```

**小白**："所以如果我用Python连ClickHouse，是走8123还是9000？"

**大师**："这取决于你用什么库。Python社区最主流的`clickhouse-driver`走的是原生TCP协议（9000），而`clickhouse-connect`和JDBC驱动走的是HTTP协议（8123）。两种方式性能差别不大，选哪个主要看你的生态依赖。"

**小胖**（若有所思）："那配置文件呢？我看Docker里/etc/clickhouse-server/下面有一堆xml，到底该改哪个？"

**大师**："ClickHouse的配置是分层叠加的，记住这个原则：**主配置定义框架，conf.d目录做扩展，修改放conf.d而不是改主文件。**"

大师打开终端展示：

```
/etc/clickhouse-server/
├── config.xml          # 主配置（服务器参数）
├── users.xml           # 用户权限配置
├── config.d/           # 额外的服务器配置片段
└── users.d/            # 额外的用户配置片段
```

"config.xml和users.xml是系统自带的，升级时可能被覆盖。我们自己的定制全部放到config.d和users.d下面。ClickHouse启动时会自动合并所有配置——后加载的会覆盖先加载的同名配置项。"

**小胖**："懂了！那开始动手吧，我已经等不及了。"

## 3. 项目实战

### 环境准备

首先，确保你的机器上已安装Docker和Docker Compose。在项目目录下创建如下结构：

```
clickhouse-sandbox/
├── docker-compose.yml
├── config/
│   └── custom.xml
├── users/
│   └── test-user.xml
└── data/               # 数据卷，自动创建
```

我们先来看`docker-compose.yml`的核心内容：

```yaml
version: '3.8'
services:
  clickhouse:
    image: clickhouse/clickhouse-server:latest
    container_name: clickhouse-dev
    hostname: clickhouse-dev
    ports:
      - "8123:8123"     # HTTP 接口
      - "9000:9000"     # Native TCP 协议
    ulimits:
      nofile:
        soft: 262144
        hard: 262144
    volumes:
      - ./config:/etc/clickhouse-server/config.d
      - ./users:/etc/clickhouse-server/users.d
      - ./data:/var/lib/clickhouse
    environment:
      CLICKHOUSE_DB: sandbox
      CLICKHOUSE_USER: dev
      CLICKHOUSE_PASSWORD: dev123
    restart: unless-stopped

  tabix:
    image: spoonest/clickhouse-tabix-web-client
    container_name: clickhouse-tabix
    ports:
      - "8080:80"
    environment:
      CH_HOST: http://clickhouse-dev:8123
      CH_LOGIN: dev
      CH_PASSWORD: dev123
    depends_on:
      - clickhouse
```

这里有四个值得关注的点：

1. **ulimits配置**：ClickHouse运行时需要大量文件描述符，`nofile: 262144`是官方推荐值。如果不配置，容器默认的1024限制会导致高并发场景下报"Too many open files"错误。
2. **端口映射**：8123和9000都映射到了宿主机。这样宿主机上的DBeaver、Python脚本都可以通过`localhost`连接。
3. **数据卷挂载**：`./data`目录映射到容器内的`/var/lib/clickhouse`，确保容器重启后数据不丢失。
4. **Tabix依赖**：通过Docker Compose的内部DNS（服务名 `clickhouse-dev`），Tabix可以直接访问ClickHouse容器，无需暴露额外端口。

### 分步实现

#### Step 1: Docker Compose 一键启动 + 验证服务

在`docker-compose.yml`所在目录执行：

```bash
# 启动所有服务
docker compose up -d

# 检查容器运行状态
docker compose ps
```

正常情况下，你会看到两个容器都是`Up`状态。接下来验证ClickHouse是否正常响应：

```bash
# 通过 HTTP 接口发送一个简单查询
curl "http://localhost:8123/?query=SELECT+1"

# 预期输出：
# 1
```

如果看到那个孤零零的`1`，说明ClickHouse已经成功运行。接下来打开浏览器访问 `http://localhost:8080`，你应该能看到Tabix的界面。在连接配置中填入：

- Host: `clickhouse-dev`
- Port: `8123`
- Login: `dev`
- Password: `dev123`

登录成功后，Tabix会列出`sandbox`数据库。点击SQL编辑器，输入`SELECT version()`，如果返回版本号，说明Tabix也已经打通。

#### Step 2: clickhouse-client 交互式操作

Tabix很方便，但真正的ClickHouse开发中，`clickhouse-client`才是主角。我们通过Docker进入容器来使用它：

```bash
# 进入ClickHouse容器，打开交互式客户端
docker exec -it clickhouse-dev clickhouse-client -u dev --password dev123 -d sandbox
```

你会看到熟悉的提示符：

```
ClickHouse client version 24.x.x.
Connecting to localhost:9000 as user dev.
Connected to ClickHouse server version 24.x.x.

clickhouse-dev :)
```

现在来探索`clickhouse-client`的几个关键参数。先退出来（输入`exit`或按`Ctrl+D`），然后在宿主机终端尝试：

```bash
# 方式一：交互模式（默认）
docker exec -it clickhouse-dev clickhouse-client -u dev --password dev123

# 方式二：--query 参数，非交互式执行单条SQL
docker exec clickhouse-dev clickhouse-client \
  -u dev --password dev123 \
  --query "SELECT now(), version()"

# 方式三：--multiline 支持多行SQL（按分号执行）
docker exec -it clickhouse-dev clickhouse-client \
  -u dev --password dev123 --multiline
```

`--multiline`模式非常适合粘贴复杂SQL——不会每敲一个回车就执行一次，而是等到分号才提交。

接下来是`--format`参数，ClickHouse支持极其丰富的输出格式。我们来逐一体验：

```bash
# 经典表格格式
docker exec clickhouse-dev clickhouse-client \
  -u dev --password dev123 \
  --format PrettyCompact \
  --query "SELECT number, number*number FROM numbers(5)"

# JSON 每行一个对象，适合日志采集
docker exec clickhouse-dev clickhouse-client \
  -u dev --password dev123 \
  --format JSONEachRow \
  --query "SELECT number, number*number AS square FROM numbers(3)"

# TabSeparated，适合导入到 Excel / Shell 管道
docker exec clickhouse-dev clickhouse-client \
  -u dev --password dev123 \
  --format TabSeparated \
  --query "SELECT 1 AS a, 2 AS b"

# CSV 格式
docker exec clickhouse-dev clickhouse-client \
  -u dev --password dev123 \
  --format CSVWithNames \
  --query "SELECT number FROM numbers(3)"

# 垂直格式，适合字段很多的宽表
docker exec clickhouse-dev clickhouse-client \
  -u dev --password dev123 \
  --format Vertical \
  --query "SELECT * FROM system.build_options LIMIT 1"
```

输出格式的选型指南：
- **PrettyCompact**：日常交互查看数据，工整美观。
- **JSONEachRow**：对接日志系统（ELK、Fluentd），每行独立JSON，便于流式处理。
- **TabSeparated/CSV**：配合Shell管道（awk、sort、uniq）做轻量ETL，或导出给业务方。
- **Vertical**：宽表调优时，竖排显示每个字段的值，一目了然。

#### Step 3: 配置文件分层实践

现在让我们进入ClickHouse的配置系统。先看看容器内默认的主配置文件：

```bash
# 查看主配置文件的结构
docker exec clickhouse-dev cat /etc/clickhouse-server/config.xml | head -30
```

你会看到大量的XML配置项。但请记住原则：**不要直接修改config.xml**。我们通过`config.d`目录来扩展配置。

创建一个简单的自定义配置：

`config/custom.xml`：

```xml
<clickhouse>
    <!-- 自定义最大并发查询数 -->
    <max_concurrent_queries>100</max_concurrent_queries>

    <!-- 自定义HTTP超时时间（秒） -->
    <http_connection_timeout>30</http_connection_timeout>

    <!-- 开启SQL日志记录，便于调试 -->
    <logger>
        <level>information</level>
        <log>/var/log/clickhouse-server/clickhouse-server.log</log>
    </logger>
</clickhouse>
```

同时，我们为团队添加一个只读的测试用户：

`users/test-user.xml`：

```xml
<clickhouse>
    <users>
        <analyst>
            <password>readonly123</password>
            <networks>
                <ip>::/0</ip>
            </networks>
            <profile>readonly</profile>
            <quota>default</quota>
            <databases>
                <sandbox>
                    <readonly>1</readonly>
                </sandbox>
            </databases>
        </analyst>
    </users>
</clickhouse>
```

由于我们在`docker-compose.yml`中已经把`./config`映射到了`/etc/clickhouse-server/config.d`，`./users`映射到了`/etc/clickhouse-server/users.d`，所有修改只需要重启容器即可生效：

```bash
docker compose restart clickhouse
```

验证配置是否加载成功：

```bash
# 检查自定义配置项是否生效
docker exec clickhouse-dev clickhouse-client \
  -u dev --password dev123 \
  --query "SELECT value FROM system.server_settings WHERE name = 'max_concurrent_queries'"

# 用新用户连接，验证只读权限
docker exec clickhouse-dev clickhouse-client \
  -u analyst --password readonly123 --database sandbox
```

在`analyst`用户的会话中尝试创建表，你会看到权限被拒绝的错误——这说明用户配置生效了。

#### Step 4: 多客户端连接

现在我们的ClickHouse已经跑起来了，还有Tabix作为Web界面。但实际开发中，更多时候是通过编程语言或者IDE来操作的。我们来配置四种典型的客户端。

**客户端 A：DBeaver（JDBC连接）**

DBeaver是Java生态中最流行的数据库管理工具。连接ClickHouse需要以下步骤：

1. 打开DBeaver → 新建连接 → 选择"ClickHouse"
2. 连接配置：
   - Host: `localhost`
   - Port: `8123`（注意：JDBC走HTTP协议！）
   - Database: `sandbox`
   - Username: `dev`
   - Password: `dev123`
3. 点击"Test Connection"——DBeaver会自动下载ClickHouse JDBC驱动

如果遇到连接超时，检查几个点：
- Docker映射的8123端口是否被宿主机防火墙拦截
- JDBC URL是否写成了9000端口（JDBC不支持原生协议）
- 驱动的`socket_timeout`是否设置得过短

**客户端 B：Python clickhouse-driver**

这是Python社区使用最广的ClickHouse驱动，走原生TCP协议（9000端口）：

```python
from clickhouse_driver import Client

# 走 Native TCP 协议（9000端口）
client = Client(
    host='localhost',
    port=9000,
    user='dev',
    password='dev123',
    database='sandbox'
)

# 创建测试表
client.execute('''
    CREATE TABLE IF NOT EXISTS test_table (
        event_date Date,
        event_name String,
        event_value Float64
    ) ENGINE = MergeTree()
    ORDER BY event_date
''')

# 批量插入数据
data = [
    ('2026-04-01', 'page_view', 1.0),
    ('2026-04-01', 'click', 2.5),
    ('2026-04-02', 'page_view', 3.0),
]
client.execute(
    'INSERT INTO test_table (event_date, event_name, event_value) VALUES',
    data
)

# 聚合查询
result = client.execute('''
    SELECT event_name, sum(event_value) AS total
    FROM test_table
    GROUP BY event_name
    ORDER BY total DESC
''')

for row in result:
    print(f"{row[0]}: {row[1]}")
```

对比另一种流行的Python驱动`clickhouse-connect`，它走的是HTTP协议：

```python
# clickhouse-connect 走 HTTP 协议（8123端口）
import clickhouse_connect

client = clickhouse_connect.get_client(
    host='localhost',
    port=8123,        # 注意这里是 8123！
    username='dev',
    password='dev123',
    database='sandbox'
)
result = client.query('SELECT version()')
print(result.result_set)
```

两者的关键区别在于端口和协议。如果你的网络环境限制了非HTTP流量（比如某些企业代理），选择`clickhouse-connect`走8123端口会更顺畅。

**客户端 C：命令行 curl（HTTP接口）**

这是最"轻量"的客户端，不需要安装任何软件：

```bash
# 查询（GET请求）
curl "http://localhost:8123/?query=SELECT+now()&user=dev&password=dev123"

# 查询（POST请求，推荐用于长SQL）
curl -X POST "http://localhost:8123/" \
  -u "dev:dev123" \
  --data "SELECT database, name, engine FROM system.tables LIMIT 5"

# 指定输出格式
curl -X POST "http://localhost:8123/" \
  -u "dev:dev123" \
  --data "SELECT number FROM numbers(3) FORMAT JSONEachRow"
```

curl方式特别适合在Shell脚本中使用——比如定时任务中拉取ClickHouse数据，或者CI/CD流水线中做冒烟测试。

**客户端 D：DataGrip**

JetBrains系的DataGrip连接ClickHouse和DBeaver类似，也是走JDBC的8123端口。在DataGrip中：

1. 新建数据源 → 选择"ClickHouse"
2. Host: `localhost`, Port: `8123`, 填写用户名密码
3. 在Advanced标签页中，可以添加JDBC参数如`connect_timeout=30000`

DataGrip的优势是与IntelliJ IDEA深度集成，如果你在IDE中写Java/Scala代码同时需要操作ClickHouse，它会非常顺手。

## 4. 项目总结

### 部署方式对比

| 维度 | Docker | RPM/DEB | K8s |
|------|--------|---------|-----|
| 部署速度 | 秒级 | 分钟级 | 分钟级 |
| 环境隔离 | 容器级 | 无隔离 | Pod级 |
| 资源开销 | 较小 | 最小 | 较大（需K8s集群） |
| 运维复杂度 | 低 | 中 | 高 |
| 生产就绪 | 需额外配置 | 开箱即用 | 需熟悉Operator |

### 适用场景

- **Docker（本章方案）**：开发测试、技术验证、本地沙箱。快速、干净、可复现，是开发者入门的最佳选择。
- **RPM/DEB**：生产环境裸金属部署。性能最好，可充分利用硬件资源，与systemd深度集成，适合传统运维体系。
- **Kubernetes**：云原生场景。如果你的团队已经在K8s上运行微服务，使用ClickHouse Operator可以实现声明式管理、自动扩缩容和滚动升级。

### 核心知识点回顾

1. **两个端口，两种协议**：8123是HTTP接口，面向Web工具、JDBC驱动和RESTful调用；9000是原生TCP协议，面向clickhouse-client和Native驱动。选择哪个端口取决于你的客户端生态。

2. **配置文件分层管理**：主配置文件（config.xml / users.xml）不要直接修改，自定义配置放到`config.d/`和`users.d/`目录。ClickHouse启动时按字母序合并所有XML片段，后加载的覆盖先加载的。

3. **Docker ulimits不可忽略**：`nofile: 262144`不是可选配置，它是ClickHouse高性能运行的刚需。忘记配置会导致文件描述符耗尽，查询报错。

4. **默认用户无密码**：生产环境的`default`用户务必设置密码，或者通过users.xml禁用远程访问。但在本地沙箱中，这是便利性的体现——切记不要照搬到生产。

### 常见踩坑经验

**坑1：Docker容器内存不足导致ClickHouse OOM**

ClickHouse默认会使用宿主机的大部分内存用于查询缓存和聚合。如果你给Docker Desktop分配的内存太小（比如2GB），稍微大一点的GROUP BY查询就会触发OOM。解决方案：在Docker Desktop设置中给Docker分配至少4-8GB内存。

**坑2：配置文件加载顺序错误**

有同学把修改写到了`config.xml`里，升级ClickHouse版本时配置被覆盖，导致服务启动失败。正确的做法永远是：自定义配置写到`config.d/`和`users.d/`目录。

**坑3：JDBC连接超时**

使用DBeaver或DataGrip连接时，如果出现`socketTimeout`错误，别急着怀疑网络。先检查端口是否写成了9000——JDBC驱动只支持HTTP协议（8123端口）。另外，ClickHouse默认的`http_connection_timeout`较短，处理复杂分析查询时可能需要调大。

### 思考题

1. **为什么ClickHouse需要两个端口（8123和9000）？各有什么适用场景？**
   
   8123是HTTP协议端口，具有最好的跨语言和跨平台兼容性——任何语言都能通过HTTP请求操作ClickHouse，JDBC驱动、ODBC驱动、Web UI都基于此端口。9000是ClickHouse专有的原生TCP协议端口，性能更优、延迟更低，clickhouse-client和原生的语言驱动（C++、Go、Python clickhouse-driver）都走这个端口。两种协议并存的设计，让ClickHouse既能满足高性能场景，又不牺牲生态兼容性。

2. **如何在不重启ClickHouse的情况下热加载配置文件？**

   ClickHouse支持部分配置的在线热加载。对于用户权限配置（users.xml / users.d），执行`SYSTEM RELOAD CONFIG`即可使修改生效，无需重启。但对于大多数服务器级配置（如端口、内存限制、线程数等），仍然需要重启服务。详情可以参考官方文档中的"Reloadable Settings"部分。
