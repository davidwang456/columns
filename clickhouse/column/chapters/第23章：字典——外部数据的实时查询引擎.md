# 第23章：字典——外部数据的实时查询引擎

> **版本**：ClickHouse 25.x LTS
> **定位**：中级篇核心章节。用字典（Dictionary）替代昂贵的分布式 JOIN，将维表查询从秒级降至微秒级——一张 500 亿行事实表与百万级维表的博弈，字典是唯一的正确解法。
> **前置阅读**：第4章（MergeTree 家族）、第7章（主键与稀疏索引）、第14章（SQL 优化入门）
> **预计阅读**：40 分钟 | **实战耗时**：60 分钟

## 1. 项目背景

某内容平台的数据团队每天早上 9 点要对前一天的页面访问数据做聚合分析。核心事实表 `page_views` 存储了 500 亿行访问日志，每条日志只记录了 `page_id`、`user_id` 和 `event_time` 三个字段。但运营分析师要的不是这些枯燥的数字 ID——他们要的是"科技频道昨天有多少独立访客""Alice 写的文章总阅读量排名第二"这类带维度的指标。

维度信息在哪？在一百多公里外的 MySQL 服务器上。`page_metadata` 表存了 100 万篇文章的元数据——文章分类（category）、作者（author）、发布日期（publish_date）。每条分析 SQL 都要做一次 LEFT JOIN，把 `page_views` 的每行日志和 `page_metadata` 关联起来。

问题就出在这个 JOIN 上。凌晨 2 点的离线报表还好说，反正要跑半小时。但早上 9 点的实时交互查询就不行了——分析师敲一条 SQL，等 3 秒出结果；换个分类过滤条件再查，又等 3 秒。这 3 秒钟里 ClickHouse 在干什么？在内存中构建一张 100 万行的哈希表，然后用这张哈希表去匹配流经查询引擎的每一行数据。如果查询只扫描了 1 亿行，这个 Hash Join 的开销还能忍。但分析师经常做全表聚合，500 亿行流水全部参与 Join——构建哈希表 0.5 秒、600 亿次哈希探测 2 秒，加起来 3 秒已经算快的了。

更让人头疼的是 IP 地理定位场景。安全团队每查一条访问日志，都要解析出访问者的国家、城市和经纬度。数据来源是一个 50 万行的 MaxMind GeoIP 文件。如果用 JOIN，每次查询都要加载一个 50 万行的 CSV 文件到内存、构建哈希表、逐行探测——纯粹是大量的重复计算。

团队 DBA 尝试把 MySQL 的维表同步到 ClickHouse 本地的 MergeTree 表，然后用普通 JOIN。效果有一点提升——毕竟数据本地了，网络延迟没了——但哈希表构建的开销仍然存在。每一条查询都要重新构建一次哈希表，而这个维表的数据其实每天才变一次。

"难道就没有一种方法，让我建一次哈希表，后面所有查询共用吗？"DBA 在 ClickHouse 文档里翻到了 Dictionary。

字典（Dictionary）是 ClickHouse 为"外部维表常驻内存"这一需求量身定制的引擎。它在服务启动时或首次访问时，把外部数据源（MySQL、PostgreSQL、Redis、文件、HTTP 接口，甚至一个 shell 脚本的输出）全量加载到内存中，构建为键值对哈希结构。后续任何查询通过 `dictGet()` 函数访问字典，本质上就是一次 O(1) 的哈希表查找——不需要网络 IO，不需要构建哈希表，一次内存寻址就出结果。

同样一张 100 万行的 page_metadata 维表：用 MySQL 表函数 + LEFT JOIN，每次查询耗时 3 秒；用字典 + dictGet，每次查询耗时 0.003 秒。**1000 倍性能差距**。

这还只是第一个好处。字典自带缓存失效与自动刷新机制——你可以配置 `LIFETIME`，告诉 ClickHouse 每隔 5 分钟自动去 MySQL 拉取一次最新数据，保证维表数据不落后于业务。字典还支持多层次存储布局（flat、hashed、cache、ip_trie 等），根据主键特征选择最省内存、最快查询的布局策略。更强大的是层级字典——你可以把一棵组织架构树、一个商品类目树加载为字典，用 `dictGetHierarchy` 函数一键展开某个节点的所有祖先或子孙。

简单说：**字典就是把外部维表变成一个全局共享的内存哈希表，所有查询零成本复用**。本章将用 Docker 搭建 ClickHouse + MySQL 环境，手把手带你创建四种不同类型的字典，并用真实的 500 亿行场景测试字典与 JOIN 的性能差异。

## 2. 项目设计：剧本式交锋对话

**Scene**：数据团队工区，小白正盯着监控大盘上的慢查询列表出神。小胖端着一杯奶茶溜达过来。

**小胖**（瞥了一眼屏幕）："嗨，我当什么事呢——不就是 JOIN 一下嘛，MySQL 的 `page_metadata` 表通过 JDBC 引擎连过来查不就完了？ClickHouse 不是有个 `mysql()` 表函数吗，一行 SQL 搞定！"

**小白**："我试过了，`LEFT JOIN mysql('mysql:3306', 'meta', 'page_metadata', 'root', 'root123')`——查询跑了 4 秒才出结果。每换一个分类过滤条件就要重新执行，分析师一天查 50 多次，光 JOIN 就浪费了 3 分多钟。更何况有时候还要连阿里云那边的 PostgreSQL 维表——跨机房网络抖动，查询直接超时。"

**小胖**（愣了一下）："那……把 MySQL 的表定时全量同步到 ClickHouse 本地？搞个 MergeTree 表，用 ReplicatedMergeTree 同步，然后本地 JOIN，总快了吧？"

**大师**（从会议室出来，正好听到这段对话）："小胖，我问你——`page_metadata` 这张表，一天变几次？"

**小胖**："基本不变啊，偶尔编辑更新下文章分类，一天不超过 10 次变更。"

**大师**："那为什么要每次查询都重新构建哈希表呢？你每次去食堂打饭，难道每次都要重新看一眼菜单？把菜单贴墙上，看一眼就够了。Dictionary 就是这面墙。"

大师走到白板前，画了一张图：

```
┌──────────────┐     ┌──────────────────┐     ┌─────────────┐
│  page_views  │────▶│  dictGet() 调用   │────▶│  内存哈希表   │
│  (500亿行)    │     │  O(1) 查找        │     │  100万行维表  │
└──────────────┘     └──────────────────┘     └──────┬──────┘
                                                     │
                                           每5分钟自动刷新
                                                     │
                                            ┌────────┴──────┐
                                            │   MySQL 源表   │
                                            │ page_metadata  │
                                            └───────────────┘
```

**大师**："字典的核心逻辑一句话：**外部数据加载到内存，变成哈希表，查询时 O(1) 直接命中**。不需要网络 IO，不需要构建哈希表，不需要逐行探测。ClickHouse 在启动时或首次查询时把 MySQL 的 100 万行数据全部拉过来，按 `page_id` 做主键构建哈希表。之后的所有查询，只要调用 `dictGet('page_meta_dict', 'category', page_id)`，ClickHouse 就直接用 `page_id` 的值去哈希表里找——就像你用一个 Map<Integer, String>，put 一次，get 无数次。"

**小胖**："等等——那如果 MySQL 里的数据变了呢？比如运营不小心把 Alice 的文章改成了 Bob 的？字典岂不是永远返回旧数据？"

**大师**："问得好。这就是 `LIFETIME` 的作用。"大师在终端里敲出一段 DDL：

```sql
CREATE DICTIONARY page_meta_dict (
    page_id UInt32,
    category String,
    author String,
    publish_date Date
)
PRIMARY KEY page_id
SOURCE(MYSQL(
    PORT 3306
    USER 'root'
    PASSWORD 'root123'
    DB 'meta'
    TABLE 'page_metadata'
))
LAYOUT(HASHED())
LIFETIME(MIN 300 MAX 600);
```

**大师**："注意最后一行。`LIFETIME(MIN 300 MAX 600)` 的意思是：字典数据在 300 秒（5 分钟）后进入'可更新'状态，在 600 秒（10 分钟）后强制更新。ClickHouse 会随机选择一个在 300 到 600 秒之间的时间点重新从 MySQL 加载最新数据。这个随机化很重要——如果你有 20 个字典都用 300 秒，它们同时去 MySQL 拉数据，MySQL 瞬间就挂了。随机窗口可以把刷新风暴打散。"

"更新过程是非阻塞的——旧数据继续给查询用，新数据在后台加载完成后再原子切换，查询不会因为字典刷新而中断或变慢。"

**小白**（举手）："那如果我改了 MySQL 数据，但还没到 300 秒，分析师着急用最新数据呢？"

**大师**："`SYSTEM RELOAD DICTIONARY page_meta_dict`，手动触发立即刷新。或者 `SYSTEM RELOAD DICTIONARIES`，刷新全部字典。"

---

**技术映射 #1**：Dictionary 的内存布局选择。大师继续展开：

**大师**："字典的 `LAYOUT` 决定了数据在内存中的存储结构和查询方式。ClickHouse 提供了六种布局，各有各的适用场景。"

| 布局 | 原理 | 适用场景 | 内存效率 | 查询速度 |
|------|------|---------|---------|---------|
| **flat()** | 数组存储，key 就是数组下标 | 主键从 0 开始连续递增（如枚举状态码 0~255） | 最高 | 最快 |
| **hashed()** | 标准哈希表 | 任意主键，行数 < 1000 万 | 中等 | 快 |
| **complex_key_hashed()** | 多列组合键的哈希表 | 复合主键查询（如 IP 段） | 中等 | 快 |
| **cache()** | 只缓存热点数据，不加载全部 | 维表超大（ > 1000 万行），但热点访问集中 | 可配置 | 中等 |
| **ssd_cache()** | 热点数据在内存，冷数据在 SSD | 超大维表（ > 1 亿行），内存放不下 | 低 | SSD 读写速度 |
| **ip_trie()** | IP 前缀树（Trie） | IP 地址段 → 地理位置映射 | 优 | 极快 |

**小胖**："等等——既然 `flat()` 最快最省内存，那为什么不全用 `flat()`？"

**大师**："因为 `flat()` 有致命前提——主键必须从 0 开始连续递增。比如你的 `page_id` 是 `(1, 5, 18, 100, 500)` 这样的稀疏值，用 `flat()` 的话 ClickHouse 会分配一个大小为 501 的数组——0 到 4、6 到 17、19 到 99 这些位置全是空的，内存直接爆炸。`flat()` 只适合主键是稠密的连续整数。比如 HTTP 状态码 100~599，或者是自增的业务编号从 0 到 N。稀疏主键一律用 `hashed()`。"

**小白**："那 `cache()` 又是什么意思？既然 `hashed()` 是把全部维表加载到内存，那如果我的维表有 5000 万行——哈希表本身就要吃掉好几 GB 内存，再加上列数据，OOM 了怎么办？"

**大师**："这就是 `cache()` 的用武之地。`CACHE(SIZE_IN_CELLS 100000)` 只维护一个固定大小的 LRU 缓存——内存里最多放 10 万个单元格。查询 `dictGet(page_id=1)` 时，如果 page_id=1 在缓存里，直接返回；如果不在，去源数据（MySQL/Redis/文件）实时拉取一行，插入缓存并踢掉最久没用过的那行。这样你用几 GB 内存就能支撑 5000 万行维表的访问，代价是冷数据查询会有延迟（需要回源加载）。"

"`ssd_cache()` 是 `cache()` 的升级版——冷数据不丢，而是存在本地 SSD 上的 RocksDB 里。这样你不需要回源 MySQL，本地 SSD 读取比网络快 100 倍。"

---

**小胖**："还有一个问题——IP 地址怎么查？MaxMind 那个 GeoIP 文件里，IP 范围是 `start_ip, end_ip, country, city`，查询的时候是用某个具体 IP 去匹配 `start_ip <= ip <= end_ip` 的范围。哈希表没法做范围查询吧？"

**大师**："你说到点子上了。IP 范围查询不是等值匹配，而是'找包含这个 IP 的最小网段'。普通哈希表根本做不了。"大师敲出了 `IP_TRIE` 字典：

```sql
CREATE DICTIONARY ip_geo_dict (
    ip_range_start UInt32,
    ip_range_end UInt32,
    country String,
    city String
)
PRIMARY KEY ip_range_start, ip_range_end
SOURCE(FILE(PATH '/var/lib/clickhouse/user_files/ip_geo.csv' FORMAT 'CSVWithNames'))
LAYOUT(IP_TRIE());
```

**大师**："`IP_TRIE()` 布局在内部构建了一棵二进制 Trie 树（前缀树），每个节点代表一个 IP 前缀。50 万个 IP 网段压进一棵 Trie，查询时沿着 IP 地址的二进制位走一遍——最多走 32 层就能定位到最精确的网段。这比二分查找还快，因为 Trie 的深度是固定的 32 层，时间复杂度 O(32) = O(1)。"

---

**小白**（若有所思）："大师，还有个场景——我们的组织架构是棵树，员工表里每个人都有个 `manager_id` 字段指向他的上级。我要查'张三的各级领导链'或者'技术部下面所有员工'——字典能做树形查询吗？"

**大师**："那叫层级字典（Hierarchical Dictionary）。你需要在字典里标注父子关系，然后用 `dictGetHierarchy` 一键拉出整条祖孙链。"

```sql
CREATE DICTIONARY org_dict (
    employee_id UInt32,
    name String,
    parent_id UInt32
)
PRIMARY KEY employee_id
SOURCE(CLICKHOUSE(HOST 'localhost' PORT 9000 USER 'default' TABLE 'employees'))
LAYOUT(HASHED_ARRAY())
LIFETIME(3600);

-- 查"李四"的所有上级（从李四一直到 CEO）
SELECT dictGetHierarchy('org_dict', toUInt32(789));
-- 返回: [789, 456, 123, 1]  （李四 → 王五 → 张三 → CEO）
```

**技术映射 #2**：层级字典的存储原理——`HASHED_ARRAY()` 布局在哈希表的值里存了父节点指针。`dictGetHierarchy()` 沿指针链一直向上追溯，直到遇到 `parent_id=0`（根节点），返回完整路径数组。

## 3. 项目实战

在当前工作目录下新建 `docker-compose.yaml`：


```sql
CREATE PROCEDURE seed_page_metadata()
```

**字典三：flat() 布局 —— HTTP 状态码映射**

先从 ClickHouse 本地创建一个状态码表（模拟来源），再基于它建字典：

```sql
CREATE TABLE status_codes (
    status_id UInt8,
    status_name String
) ENGINE = TinyLog;

INSERT INTO status_codes VALUES
(200, 'OK'), (201, 'Created'), (301, 'Moved Permanently'),
(302, 'Found'), (400, 'Bad Request'), (401, 'Unauthorized'),
(403, 'Forbidden'), (404, 'Not Found'), (500, 'Internal Server Error'),
(502, 'Bad Gateway'), (503, 'Service Unavailable');

-- flat() 布局需要主键从 0 开始连续。这里 200~503 之间有空洞，
-- flat() 会自动填充空槽（内存按 504 个元素分配，每个元素一个指针 ≈ 4KB）
-- 对于范围小（<10000）的连续整数主键，flat() 是最优选择
CREATE DICTIONARY status_dict (
    status_id UInt8,
    status_name String
)
PRIMARY KEY status_id
SOURCE(CLICKHOUSE(
    HOST 'localhost'
    PORT 9000
    USER 'default'
    DB 'default'
    TABLE 'status_codes'
))
LAYOUT(FLAT())
LIFETIME(3600);
```

**字典四：cache() 布局 —— 超大商品维表（演示用）**

```sql
CREATE DICTIONARY product_dict (
    product_id UInt64,
    product_name String
)
PRIMARY KEY product_id
SOURCE(MYSQL(
    HOST 'ch23-mysql'
    PORT 3306
    USER 'root'
    PASSWORD 'root123'
    DB 'meta'
    TABLE 'page_metadata'
))
LAYOUT(CACHE(SIZE_IN_CELLS 50000))
LIFETIME(MIN 600 MAX 3600);
```

⚠️ **注意**：`CACHE(SIZE_IN_CELLS 50000)` 的意思是内存缓冲区最多放 50000 个单元格（每个单元格 = 哈希表中的一个桶）。字典源数据有 10 万行，缓存只能装一半，访问冷数据时会回源 MySQL 查询。生产环境中 SIZE_IN_CELLS 应设置为略大于热数据量的值。

### Step 4：使用 dictGet 系列函数查询

**（1）基础维表查询——替代 LEFT JOIN**

```sql
-- 传统写法（JOIN）
SELECT
    pv.page_id,
    count() AS views,
    pm.category,
    pm.author
FROM page_views pv
LEFT JOIN mysql('ch23-mysql:3306', 'meta', 'page_metadata', 'root', 'root123') AS pm
    ON pv.page_id = pm.page_id
GROUP BY pv.page_id, pm.category, pm.author
ORDER BY views DESC
LIMIT 10;

-- 字典写法（dictGet）
SELECT
    page_id,
    count() AS views,
    dictGet('page_meta_dict', 'category', page_id) AS category,
    dictGet('page_meta_dict', 'author', page_id) AS author
FROM page_views
GROUP BY page_id, category, author
ORDER BY views DESC
LIMIT 10;
```

⚠️ **注意**：上面的传统写法只是示意，实际测试中建议直接用 `clickhouse-benchmark` 做精确对比（见 Step 5）。

**（2）带默认值的安全查询**

```sql
-- 假设 page_id=999999 不存在于字典中
SELECT
    dictGetOrDefault('page_meta_dict', 'category', toUInt32(999999), 'Unknown') AS category,
    dictGetOrDefault('page_meta_dict', 'author', toUInt32(999999), 'N/A') AS author;
```

`dictGetOrDefault` 的第四个参数是"查不到时的默认值"。与 `dictGet` 不同，`dictGetOrDefault` 遇到缺失键不会抛异常，而是优雅地返回默认值。生产环境中推荐始终使用 `dictGetOrDefault`，避免因一两条脏数据撑爆你的报表任务。

**（3）字典过滤——用维表条件筛选事实表**

```sql
-- 筛选"Technology"分类下的所有访问记录
SELECT
    page_id,
    count() AS views
FROM page_views
WHERE dictGet('page_meta_dict', 'category', page_id) = 'Technology'
GROUP BY page_id
ORDER BY views DESC
LIMIT 10;
```

这条 SQL 的执行逻辑：逐行读取 `page_views` → 用 `page_id` 去字典查 `category` → 只保留 `category='Technology'` 的行 → 聚合。因为字典查询是 O(1) 内存操作，所以这比在 WHERE 里做子查询或 JOIN 快得多。

**（4）IP 地址地理定位查询**

```sql
-- 注意：IP_TRIE 字典要求把 IP 转为 UInt32（调用 IPv4StringToNum）再包装成 tuple
SELECT
    ip,
    dictGetOrDefault('ip_geo_dict', 'country',
        tuple(IPv4StringToNum(ip)), 'Unknown') AS country,
    dictGetOrDefault('ip_geo_dict', 'city',
        tuple(IPv4StringToNum(ip)), 'Unknown') AS city,
    dictGetOrDefault('ip_geo_dict', 'latitude',
        tuple(IPv4StringToNum(ip)), 0.0) AS latitude,
    dictGetOrDefault('ip_geo_dict', 'longitude',
        tuple(IPv4StringToNum(ip)), 0.0) AS longitude
FROM page_views
LIMIT 10;
```

`tuple()` 包装是 `complex_key_hashed` 和 `IP_TRIE` 布局的硬性要求——复合键必须以元组形式传递给 `dictGet`。

**（5）层级字典示例——组织架构查询**

```sql
-- 假设我们有一张员工表
CREATE TABLE employees (
    employee_id UInt32,
    name String,
    parent_id UInt32
) ENGINE = MergeTree() ORDER BY employee_id;

INSERT INTO employees VALUES
(1, 'CEO', 0),
(10, 'CTO', 1),
(100, '技术总监', 10),
(200, '产品总监', 10),
(1000, '后端组长', 100),
(1001, '前端组长', 100),
(2000, '张三（后端开发）', 1000),
(2001, '李四（前端开发）', 1001);

-- 创建层级字典（HASHED_ARRAY 支持父子指针）
CREATE DICTIONARY org_hierarchy_dict (
    employee_id UInt32,
    name String,
    parent_id UInt32
)
PRIMARY KEY employee_id
SOURCE(CLICKHOUSE(HOST 'localhost' PORT 9000 USER 'default' TABLE 'employees'))
LAYOUT(HASHED_ARRAY())
LIFETIME(3600);

-- 查"张三"的完整上级链路
SELECT dictGetHierarchy('org_hierarchy_dict', toUInt32(2000)) AS hierarchy;
-- 结果：[2000, 1000, 100, 10, 1]
--      张三 → 后端组长 → 技术总监 → CTO → CEO

-- 判断"李四"是否在"技术总监"的子树下
SELECT dictIsIn('org_hierarchy_dict', toUInt32(2001), toUInt32(100)) AS is_subordinate;
-- 结果：1 (true)
```

## 4. 项目总结

### Layout 选择速查表

| Layout | 主键特征 | 适用行数 | 查找复杂度 | 内存占用 | 典型场景 |
|--------|---------|---------|-----------|---------|---------|
| `flat()` | 0~N 连续整数 | < 50 万 | O(1) | 最低 | 状态码、枚举值、小维度表 |
| `hashed()` | 任意离散值 | < 1000 万 | O(1) | 中等 | 用户维表、文章维表、标签映射 |
| `complex_key_hashed()` | 多列组合键 | < 1000 万 | O(1) | 中等 | 多字段联合映射、双维度查询 |
| `cache()` | 任意，热点集中 | 100 万 ~ 1 亿 | O(1) 热点 / O(n) 冷 | 可配置 | 大维表、长尾数据、冷热分明 |
| `ssd_cache()` | 任意 | 1 亿 ~ 10 亿 | O(1) 热点 / SSD IO 冷 | 低 | 超大规模维表、内存预算紧张 |
| `ip_trie()` | IP 地址段 | < 100 万网段 | O(32) | 优 | IP 地理定位、ASN 查询、安全风控 |
| `hashed_array()` | 含父子关系的 ID | < 1000 万 | O(1) + 层级遍历 | 中等 | 组织架构、类目树、区域层级 |

### 适用场景

1. **维表 JOIN 替代**：任何需要频繁 JOIN 外部维表（MySQL/PostgreSQL/Redis）的查询，都应优先考虑字典。字典把"每个查询都构建一次哈希表"变成"全局共享一张哈希表"。
2. **IP 地理定位**：MaxMind GeoIP、IP 归属地、ASN 信息。`IP_TRIE` 布局是此场景的最佳解，比任何 JOIN 方案都高效。
3. **实时特征查询**：机器学习推理时，需要根据用户 ID 查用户画像特征（年龄、性别、会员等级）。用字典 + dictGet，一次内存查找得到所有特征。
4. **多语言 / 标签映射**：`id → 中文名`、`code → 描述` 这类静态映射表，`flat()` 布局几乎零内存开销。
5. **层级树遍历**：组织架构查询、商品类目路径展开，`HASHED_ARRAY()` + `dictGetHierarchy` 是最优雅的实现。

### 注意事项

- **字典首次加载会阻塞查询**：字典在首次被访问时才触发加载（lazy loading），加载期间引用该字典的查询会等待。如果字典数据量很大（比如 1000 万行），首次加载可能耗时几十秒。建议在应用启动后先执行一次 `SELECT dictGet(...)` 预热所有字典，或者使用 `SYSTEM RELOAD DICTIONARY` 手动触发。
- **LIFETIME 的选择**：LIFETIME 太短 → MySQL 频繁被轮询，数据库压力大。LIFETIME 太长 → 字典数据过时，业务看到脏数据。建议：业务日变更量 < 100 条的维表，LIFETIME 设 1800（30 分钟）；日变更量 < 1000 条，设 600（10 分钟）；变更非常频繁（上万次/天），考虑用 Redis 源 + 短 LIFETIME 或者改用 `cache()` 布局实时回源。
- **字典刷新是非阻塞的**：刷新期间旧数据继续服务查询，新数据加载完成后原子切换。但切换本身会有一个短暂的互斥锁——字典越大，切换耗时越长（1000 万行约 100ms）。
- **不支持跨字典 JOIN**：`dictGet` 每次只能查一个键的值。如果你想"查出所有 category='Technology' 的 page_id"，字典做不到——你只能用字典做"给定 page_id 查 category"的正向查找。反向查找需要额外建立反向字典，或者直接用物化视图。

### 常见踩坑经验

1. **flat() 布局主键从 0 开始**：如果你把 `user_id`（范围 10000200~10005000）映射到 `flat()` 布局，ClickHouse 会创建一个 10005001 个元素的数组——但只有 4800 个有值，其余全是空槽。内存直接炸掉。**检查方法**：`SELECT bytes_allocated FROM system.dictionaries WHERE name = 'your_dict'`——如果内存占用远超预期数据量，说明选错了布局。

2. **字典加载期间查询返回旧数据**：`SYSTEM RELOAD DICTIONARY` 之后立刻查询，理论上应该返回新数据——但实际上旧数据的哈希表还在服务，新数据后台加载。原子切换的时机不是你调 `RELOAD` 的时间，而是后台加载完成的时间。如果你需要在加载完成后立即确认，查询 `system.dictionaries` 的 `last_successful_update_time` 字段。

3. **MySQL 源字段类型不匹配**：字典的列定义必须和 MySQL 源表的列类型精确对应。比如 MySQL 的 `INT UNSIGNED` 在 ClickHouse 字典里必须用 `UInt32`，如果用 `Int32` 会导致字典加载失败（报 `DB::Exception: Type mismatch`）。**排查手段**：`SELECT * FROM system.dictionaries WHERE last_exception != ''` 可以快速定位加载失败的原因。

4. **字典的 `dictGet` 不能用于分布式查询的本地节点**：在 Distributed 表上使用 `dictGet` 时，字典必须在**每个分片节点上都存在**。如果字典只在某个节点上定义了，分布查询执行时会报 `Dictionary not found`。解决方案：在集群所有节点上统一建立同名字典，或将字典定义放在 `config.xml` 中通过 ClickHouse Keeper 同步。

5. **IP_TRIE 字典需要用 tuple 包装键**：这是最容易写错的 API。`dictGet('ip_geo_dict', 'country', IPv4StringToNum(ip))` 会直接报类型错误。正确写法是 `dictGet('ip_geo_dict', 'country', tuple(IPv4StringToNum(ip)))`。原因是 `IP_TRIE` 的 PRIMARY KEY 是两个字段（start 和 end），即使你建表时用了一个字段，ClickHouse 内部也将其视为复合键。

### 思考题

1. **CACHE(SIZE_IN_CELLS) 布局和 HASHED 布局的本质区别是什么？什么时候该用 CACHE 而不是 HASHED？**

   > 提示：HASHED 把源表全部加载到内存，查询永远是 O(1)；CACHE 只缓存热点数据，冷数据回源。如果维表有 5000 万行但 80% 查询只命中 5000 个热门 page_id，用 CACHE(50000) 就够，内存节省 99%。但如果你的查询模式是均匀访问所有行——那 CACHE 反而因为频繁的缓存未命中而变慢，不如直接 HASHED。

2. **如果 MySQL 维表有 1000 万行，全部加载到 ClickHouse 字典会不会 OOM？如何优化？**

   > 提示：1000 万行 × 每行 200 字节 = 约 2GB 原始数据。加上哈希表开销（约 30%~50%），总内存占用约 3~4GB。如果你的 ClickHouse 节点有 16GB 内存，完全能装下，用 `HASHED()` 即可。但如果节点只有 4GB 内存，就要用 `CACHE(SIZE_IN_CELLS 1000000)` 只缓存 100 万个单元格（约 500MB），或者用 `ssd_cache()` 把冷数据存本地 SSD。另外，检查是否有不必要的 String 列——把长字符串改成 `LowCardinality(String)` 或枚举类型可以减少 40% 内存占用。

3. **如果维表数据每天全量更换一次（比如凌晨跑 ETL 全量覆盖），字典应该怎么配置？**

   > 提示：设 `LIFETIME(MIN 0 MAX 3600)`。MIN=0 表示"随时可以刷新"，ClickHouse 会在 LIFETIME 窗口内随机选择一个时间点去 MySQL 重新拉数据。配合定时任务在 ETL 跑完后执行 `SYSTEM RELOAD DICTIONARY` 做到"ETL 完成 → 字典立即同步"。如果对时效要求极高，可以不用 LIFETIME 自动刷新，完全靠外部调度系统主动推送 `RELOAD`。

---

> **本章小结**：字典是 ClickHouse 在"高性能 JOIN"上最优雅的解法——把维表从慢速的查询时构建，变成一次性的内存常驻。选择合适的 LAYOUT（flat / hashed / cache / ip_trie）决定了你的字典是"利器"还是"炸弹"。下一章我们将进入 MergeTree 的表结构与 TTL 管理，探索 ClickHouse 数据生命周期的自动化运维能力。
