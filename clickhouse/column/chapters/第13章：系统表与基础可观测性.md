# 第13章：系统表与基础可观测性

> **版本**：ClickHouse 25.x LTS
> **定位**：基础篇核心章节。系统表是 ClickHouse 内置的"监控中心"——不需要 Prometheus、不需要 Grafana，一个 SQL 就能看清集群的健康状态。
> **前置阅读**：第4章（MergeTree 家族概览）、第5章（分区与排序键）、第10章（TTL 与数据生命周期管理）
> **预计阅读**：30 分钟 | **实战耗时**：45 分钟

---

## 1. 项目背景

凌晨 3 点 14 分，一阵刺耳的手机铃声把运维工程师老张从睡梦中拽了出来。

"老张！大盘全红了！所有看板都在转圈，业务那边电话已经打爆了！"值班同事的声音在电话那头急促得发抖。老张一个翻身爬起来，SSH 进 ClickHouse 节点，对着黑底白字的终端，大脑一片空白。

他下意识地敲下 `docker stats`——CPU 飙到 98%。又敲下 `df -h`——磁盘还剩 3%。"到底是哪条查询在吃 CPU？是有人在跑全表扫描，还是后台 Merge 炸了？磁盘是被数据撑满了，还是残留文件没清理？"

他不知道。他只知道集群快死了，于是他开始随机杀查询——`KILL QUERY WHERE query_id LIKE '%abc%'`，杀掉几个看起来"可疑"的查询，过两分钟 CPU 没降，再杀几个。像在黑暗中盲目挥舞拳头，打到谁算谁。

二十分钟后，CPU 突然自己降下来了——原来是一条离线 ETL 任务跑完了，和他杀的查询没有任何关系。老张瘫在椅子上，后背全是冷汗。

**这不是虚构的故事**——这是每一个没有建立可观测性的 ClickHouse 集群运维者的日常。

ClickHouse 的运维困境和 MySQL、PostgreSQL 有一个关键区别：ClickHouse 不只是一个数据库，它内部同时运行着几十个后台任务——Merge 合并 Part、Mutation 变更数据、TTL 清理过期行、Partition 自动删除——这些任务共享同一台机器的 CPU、内存和磁盘 IO。当集群变慢时，你必须在「前台查询」「后台合并」「变更任务」「磁盘碎片」之间快速定位真正的瓶颈。没有可观测性，你就是在蒙眼开车。

大多数团队的解法是加装外部监控——Prometheus + Grafana，配上一堆 exporter。这个思路本身没错，但有一个致命的时间差：从决定搭建监控到真正上线可用，往往需要 1-2 周。而故障不会等你两周。

**实际上，ClickHouse 自带了一套完整的"内置监控面板"——系统表（System Tables）**。它们就在 `system` 数据库下，不需要安装任何额外的组件，不需要配置文件修改（除了 `query_log`），一个 SQL 就能告诉你：当前谁在跑查询、谁在跑 Merge、哪个表的 Part 数量爆炸了、过去 24 小时最慢的 10 条 SQL 是什么、有没有 Mutation 卡住了。

本章的目标就是让你掌握这套"原生可观测性"——在接入 Prometheus 之前，先用系统表建起第一道防线。当你凌晨被叫醒的时候，不用再盲目杀查询，而是打开 `clickhouse-client`，跑一条健康检查脚本，30 秒内定位问题。

## 2. 项目设计：剧本式交锋对话

周一的技术分享会上，大师在屏幕上投出一张只有两行 SQL 的"健康检查脚本"。小胖看了一眼，皱起了眉头。

**小胖**（手里转着笔，不以为然地摇摇头）：“监控不就是装个 Prometheus 吗？看这些 `system.*` 表有啥用？Prometheus 还能配告警、画大盘、发钉钉通知，这些系统表能干啥？不就是一个数据库的元数据目录嘛。”

**大师**（笑了笑，没有直接反驳）：“好，那我问你一个问题。你的 Prometheus 从安装到第一条告警上线花了多久？”

**小胖**（愣了一下）：“呃……上次搞了差不多一星期吧。主要是配 Exporter、调 Grafana 面板、写告警规则，中间还被 AlertManager 的分组规则绕晕过。”

**大师**：“一星期。那你有没有想过——如果这一周里，凌晨三点集群挂了，你用什么诊断？难道告诉老板'等我一星期，装好 Prometheus 就知道了'？”

**小胖**（笔停了下来）：“啊……那就只能上机器瞎看了。”

**大师**：“对，'瞎看'。而系统表就是让你从'瞎看'变成'精准定位'。你甚至不需要记住所有表名——只要记住一个 `system.` 前缀，然后按 Tab 补全，就能看到 ClickHouse 为你准备的所有诊断窗口。”

**技术映射 #1**：`system` 数据库下的每一张表都是一扇"观测窗口"——`system.query_log` 看历史查询、`system.processes` 看当前负载、`system.parts` 看存储健康、`system.merges` 看后台合并。它们不需要额外部署，随 ClickHouse 启动就可用。

---

**小白**（翻开笔记本，扶了扶眼镜）：“那系统表到底有哪些？我不可能把所有表都记下来，有没有一个'核心清单'——日常巡检必看的几张表？”

**大师**（在白板上快速画了一张表）：

| 系统表 | 看什么 | 一句话用途 |
|--------|--------|-----------|
| `system.tables` | 所有表的元数据 | 哪些表最大？用的什么引擎？ |
| `system.parts` | 每个 Part 的状态 | 有没有 Part 爆炸？有没有脏数据没清？ |
| `system.query_log` | 历史查询记录 | 谁在跑慢查询？过去一天哪个 SQL 跑了最多资源？ |
| `system.processes` | 当前正在执行的查询 | 现在谁在吃 CPU？有没有查询卡住了？ |
| `system.merges` | 正在执行的 Merge | Merge 积压了吗？是不是 Merge 风暴？ |
| `system.mutations` | ALTER DELETE/UPDATE 任务 | Mutation 有没有卡住？还要多久完成？ |
| `system.metrics` | 实时数值指标 | 当前内存用了多少？连接数多少？ |
| `system.errors` | 系统错误记录 | 有没有反复报错的组件？ |

"把这八张表记住，你就覆盖了 90% 的日常诊断场景。"

**小白**（快速记下，又抬起头）：“但我有个疑问——`system.query_log` 和 `system.processes` 有什么区别？一个看历史，一个看当前？”

**大师**：“问得好。`system.processes` 就像你推开机房门看到的场景——谁正坐在终端前跑查询、跑了多久、吃了多少内存。但它只拍一张快照，查询一结束就消失。`system.query_log` 就像机房门口的签到本——谁、什么时候、进来做了什么、用了多少资源、什么时候走的，全记在上面。”

**小胖**（突然插话）：“等等，那 `system.query_log` 是默认就记的吗？我好像从来没在 `system` 库里见过这张表？”

**大师**：“这就是第一个坑——**`query_log` 默认不开启**。你需要在 `config.xml` 里显式配置，否则它只是一张空表。很多新手以为自己的 ClickHouse 没有查询日志功能，其实只是没打开开关。”

**技术映射 #2**：`system.processes` = 内存级快照（实时），`system.query_log` = 持久化日志（历史）。前者不需要配置，后者必须在配置文件中显式启用 `<query_log>` 块。二者配合使用：先用 `processes` 看当前谁在"搞事情"，再用 `query_log` 回溯它过去还干过什么。

---

**小白**（笔记本翻到下一页）：“还有一个场景——如果我发现磁盘 IO 很高，但 `system.processes` 里没有任何大查询，怎么判断是不是后台 Merge 或 Mutation 在搞鬼？”

**大师**（赞许地看了小白一眼）：“这就是系统表'关联诊断'的价值。一条 SQL 看不出全貌，但三张表交叉查询就能定位问题。比如这个场景，你应该同时查三张表：

- `system.merges`——看看是不是 Merge 正在大量读写 Part。
- `system.mutations`——看看是不是有未完成的 Mutation 在重写数据。
- `system.metrics`——查 `Merge` 和 `PartMutation` 两个指标的历史趋势。

如果 `system.merges` 里排了几十条任务，而 `system.metrics` 显示 `Merge` 数一直在飙升——这就是典型的 Merge 风暴。很可能是某个表写入太频繁，产生了大量小 Part，Merge 调度器忙不过来。”

**小胖**（恍然大悟）：“哦！所以 `system.parts` 里的 Part 数量也能提前预警——如果某个表的活跃 Part 数超过 100 甚至 1000，说明写入频率太高，合并跟不上，这时候就该调大 `parts_to_throw_insert` 或者降低写入并发。”

**大师**：“正是。`system.parts` 里的 `active` 字段尤其重要——`active = 1` 是活跃 Part，会被查询扫描。`active = 0` 的 Part 是已经被合并替代但还没物理删除的'僵尸 Part'。如果僵尸 Part 堆积太多，说明后台清理线程可能有问题，或者磁盘空间本身就不够用了。”

**技术映射 #3**：系统表诊断的核心方法论是"交叉关联"：`system.merges` + `system.mutations` 看后台负载、`system.parts` + `system.processes` 看前台压力、`system.metrics` 提供全局趋势。单表能发现问题，多表能定位根因。

---

**小白**（合上笔记本）：“最后一个问题——这些系统表的数据从哪来的？它们本身会占磁盘空间吗？会不会因为系统表太大反而拖慢集群？”

**大师**：“好问题。大多数系统表是**内存虚拟表**——它们的数据在查询时实时从 ClickHouse 内部的内存结构中读取，不占用磁盘。比如 `system.processes`、`system.merges`、`system.metrics`，你查它就是直接读内存。但有几个例外——`system.query_log` 和 `system.asynchronous_metric_log`，它们是真实的 MergeTree 表，数据会落盘，而且会持续增长。如果忘了给它们配 TTL，它们能悄悄把你的系统盘撑爆。”

**小胖**（倒吸一口气）：“所以之前我们有个环境的 `/var/lib/clickhouse` 满了，查出来全是 `system` 数据库在吃空间——原来是 `query_log` 没配 TTL！”

**大师**：“这个坑，待会实战环节会专门讲。”

## 3. 项目实战

### 环境准备

我们使用 Docker 启动一个 ClickHouse 实例，核心要求是**启用 `query_log`**——这是唯一需要手动配置的系统表。

首先创建配置目录和自定义配置文件：

```bash
mkdir -p ~/clickhouse-ch13/config.d
```

创建 `~/clickhouse-ch13/config.d/query_log.xml`：

```xml
<clickhouse>
    <query_log>
        <database>system</database>
        <table>query_log</table>
        <flush_interval_milliseconds>7500</flush_interval_milliseconds>
    </query_log>
</clickhouse>
```

> **参数说明**：`flush_interval_milliseconds` 控制 `query_log` 从内存缓冲区刷写到磁盘的间隔。默认 7500ms（7.5 秒）是一个合理的平衡——太短会增加 IO 压力，太长会丢失最近的查询记录。生产环境中可以根据查询频率调整到 1000-15000ms。

启动容器：

```bash
docker run -d --name ch13 \
  -p 8123:8123 -p 9000:9000 \
  -v ~/clickhouse-ch13/config.d:/etc/clickhouse-server/config.d \
  clickhouse/clickhouse-server:25.4-alpine

# 等待容器就绪后进入客户端
docker exec -it ch13 clickhouse-client
```

验证 `query_log` 是否已启用：

```sql
SELECT count() FROM system.query_log;
-- 如果返回 > 0，说明 query_log 已生效
-- 如果返回 0 或报错 "Table doesn't exist"，检查配置文件挂载路径
```

### 分步实现

#### Step 1：元数据总览——看清你的数据资产

**目标**：快速了解集群中有哪些表、各表的数据量和存储引擎。

```sql
-- 所有非系统库的表一览（按数据量降序）
SELECT 
    database,
    name,
    engine,
    total_rows,
    formatReadableSize(total_bytes) AS size,
    formatReadableSize(total_bytes / greatest(total_rows, 1)) AS bytes_per_row
FROM system.tables
WHERE database NOT IN ('system', 'INFORMATION_SCHEMA', 'information_schema')
ORDER BY total_bytes DESC
LIMIT 20;
```

`formatReadableSize()` 能将字节数自动转换为 KB/MB/GB/TB 的可读格式。`bytes_per_row` 帮你快速发现「某张表每行数据量异常大」——可能是压缩策略有问题，或者某列存了巨量文本。

再看列的详细信息——哪些列最占空间：

```sql
SELECT 
    table,
    name,
    type,
    formatReadableSize(sum(data_compressed_bytes)) AS compressed,
    formatReadableSize(sum(data_uncompressed_bytes)) AS uncompressed,
    round(sum(data_compressed_bytes) / greatest(sum(data_uncompressed_bytes), 1) * 100, 1) AS compression_pct
FROM system.columns
WHERE database = 'analytics'   -- 替换为你的业务库名
GROUP BY table, name, type
ORDER BY sum(data_compressed_bytes) DESC
LIMIT 20;
```

> **解读**：`compression_pct` 越低说明压缩效率越高。如果某列的压缩率异常低（接近 100%），可能是该列数据类型和压缩算法不匹配（比如对高基数字符串使用 Delta 编码）。

---

#### Step 2：Part 健康度监控——防止 Part 爆炸

**目标**：监控每个表的 Part 数量，Part 数超过 100 就是黄色预警，超过 1000 就是红色警报。

```sql
-- 找出 Part 数最多的表（> 100 需要关注）
SELECT 
    database,
    table,
    count() AS part_count,
    sum(rows) AS total_rows,
    formatReadableSize(sum(bytes_on_disk)) AS disk_usage,
    round(sum(rows) / greatest(count(), 1)) AS avg_rows_per_part
FROM system.parts
WHERE active
GROUP BY database, table
ORDER BY part_count DESC;
```

`avg_rows_per_part` 是一个关键指标——如果 Part 平均只有几千行，说明写入过于碎片化，Merge 跟不上写入速度。理想情况下，每个 Part 应该包含几十万到几百万行。

**僵尸 Part 检查**——`active = 0` 的 Part 是已合并但尚未物理删除的残留：

```sql
SELECT 
    database,
    table,
    partition,
    name,
    rows,
    formatReadableSize(bytes_on_disk) AS size,
    modification_time
FROM system.parts
WHERE NOT active
ORDER BY modification_time DESC
LIMIT 50;
```

> **踩坑提醒**：如果 `active = 0` 的 Part 数量很大且 `modification_time` 很久远，说明 ClickHouse 的后台清理线程可能因磁盘满或权限问题无法删除旧文件。这会导致磁盘空间被"僵尸 Part"悄悄占满——`df -h` 显示磁盘快满了，但 `system.parts` 中活跃 Part 的总大小远小于磁盘占用。

---

#### Step 3：慢查询分析——从 `query_log` 中挖出性能杀手

**目标**：找出过去 24 小时最慢的查询和最频繁的查询模式。

```sql
-- Top 10 最慢查询（过去 24 小时）
SELECT 
    type,
    query_start_time,
    query_duration_ms,
    formatReadableSize(memory_usage) AS memory_used,
    read_rows,
    read_bytes,
    result_rows,
    substring(query, 1, 150) AS query_preview
FROM system.query_log
WHERE type = 'QueryFinish'
  AND query_start_time >= now() - INTERVAL 1 DAY
ORDER BY query_duration_ms DESC
LIMIT 10;
```

> **注意**：`type` 字段必须过滤为 `'QueryFinish'`——`query_log` 中每条查询有两条记录：`QueryStart`（开始）和 `QueryFinish`（结束）。只有 `QueryFinish` 才包含 `query_duration_ms` 等执行统计指标。

**查询模式分析**——哪些 SQL 跑得最多、平均耗时多少、P99 耗时多少：

```sql
SELECT 
    normalizedQueryHash(query) AS query_hash,
    substring(query, 1, 80) AS query_preview,
    count() AS executions,
    round(avg(query_duration_ms), 1) AS avg_ms,
    round(quantile(0.50)(query_duration_ms), 1) AS p50_ms,
    round(quantile(0.99)(query_duration_ms), 1) AS p99_ms,
    round(max(query_duration_ms), 1) AS max_ms,
    sum(read_rows) AS total_read_rows,
    formatReadableSize(sum(memory_usage)) AS total_memory
FROM system.query_log
WHERE type = 'QueryFinish'
  AND query_start_time >= now() - INTERVAL 1 DAY
  AND query_kind = 'Select'
GROUP BY query_hash, query_preview
ORDER BY executions DESC
LIMIT 20;
```

> **关键解读**：
> - `avg_ms` 低但 `p99_ms` 高 → 说明大部分查询很快，但存在偶尔的"长尾延迟"——可能是某次查询碰上了大 Partition，或正好有 Merge 在竞争 IO。
> - `executions` 很高但查询本身很简单 → 可能是应用层没有做缓存，反复查询相同数据。这是优化的"低垂果实"——加一层 Redis 缓存或物化视图就能大幅降低集群负载。

---

#### Step 4：实时指标监控——此刻集群在做什么

**目标**：查看当前正在执行的查询、后台 Merge 和 Mutation 的积压情况。

**查询一：当前连接和执行中的查询**

```sql
SELECT 
    user,
    query_id,
    elapsed,
    formatReadableSize(memory_usage) AS memory,
    read_rows,
    substring(query, 1, 100) AS query_preview
FROM system.processes
ORDER BY elapsed DESC;
```

如果 `elapsed` 很大的查询（比如超过 300 秒），且 `read_rows` 也在不断增长——说明它大概率是一个大查询，而不是卡死。如果 `elapsed` 很大但 `read_rows` 一直不变——可能是被某个锁阻塞了。

**查询二：后台 Merge 积压**

```sql
SELECT 
    database,
    table,
    count() AS merge_count,
    sum(rows_read) AS total_rows_merging,
    formatReadableSize(sum(memory_usage)) AS total_memory
FROM system.merges
GROUP BY database, table
ORDER BY merge_count DESC;
```

`merge_count` 本身就是关键信号——如果某个表同时有 10+ 个 Merge 任务在执行，说明它的 Part 碎片化非常严重。配合 `system.parts` 的 `part_count` 一起看，就能判断是临时性的合并高峰还是长期性的写入-合并不平衡。

**查询三：Mutation 任务进度**

```sql
SELECT 
    database,
    table,
    mutation_id,
    command,
    is_done,
    parts_to_do,
    create_time,
    formatReadableTimeDelta(elapsed) AS running_for
FROM system.mutations
WHERE NOT is_done
ORDER BY create_time;
```

`parts_to_do` 是在这个 Mutation 完成前还需要处理多少个 Part。如果这个数字一直在减少，哪怕速度慢，也说明 Mutation 在正常推进。如果 `parts_to_do` 长时间不变，可能是被其他 Mutation 阻塞了（Mutation 对同一个 Part 是串行执行的）。

**查询四：系统资源指标快照**

```sql
SELECT 
    metric,
    value,
    description
FROM system.metrics
WHERE metric IN (
    'Query',               -- 当前正在执行的查询数
    'Merge',               -- 当前正在执行的 Merge 数
    'PartMutation',        -- 当前正在执行的 Mutation 数
    'MemoryTracking',      -- 当前追踪到的内存使用量（字节）
    'TCPConnection',       -- 当前 TCP 连接数
    'HTTPConnection',      -- 当前 HTTP 连接数
    'Read',                -- 当前正在读取的表数量
    'Write'                -- 当前正在写入的表数量
)
ORDER BY metric;
```

> **日常巡检阈值参考**：
> - `Query` > 50 → 并发查询较高，关注 `system.processes`。
> - `Merge` > 20 → Merge 积压严重，检查写入频率和 Part 碎片化。
> - `MemoryTracking` > 物理内存的 80% → 内存紧张，考虑限制 `max_memory_usage` 或扩容。
> - `TCPConnection` > 500 → 连接数较高，检查是否有连接泄漏。

---

#### Step 5：错误统计与诊断脚本

**目标**：快速了解系统级别的错误概况，并封装一套一键诊断脚本。

```sql
-- 最近系统错误总览
SELECT 
    name,
    last_error_time,
    last_error_message,
    last_error_trace
FROM system.errors
ORDER BY last_error_time DESC
LIMIT 20;
```

`system.errors` 记录的是 ClickHouse 内部组件的错误统计（如 `MEMORY_LIMIT_EXCEEDED`、`TOO_MANY_PARTS`、`REPLICA_IS_NOT_ACTIVE` 等）。它不是查询级别的错误（查询报错请在 `system.query_log` 的 `exception` 字段中查看），而是系统组件级别的"健康信号"。

---

#### Step 6：封装运维诊断脚本——一条 SQL 看清全局

将上述分散的诊断查询封装成一条"一键健康检查"SQL：

```sql
SELECT '========== 01. 当前活跃查询 ==========' AS check_item
UNION ALL
SELECT concat('活跃查询数：', toString(count())) FROM system.processes

UNION ALL
SELECT '========== 02. Merge 积压 ==========' AS check_item
UNION ALL
SELECT concat('正在执行的 Merge：', toString(count())) FROM system.merges

UNION ALL
SELECT '========== 03. 未完成 Mutation ==========' AS check_item
UNION ALL
SELECT concat('待完成 Mutation：', toString(count())) FROM system.mutations WHERE NOT is_done

UNION ALL
SELECT '========== 04. Part 爆炸预警（Part > 100）==========' AS check_item
UNION ALL
SELECT concat('Part > 100 的表数：', toString(count())) FROM (
    SELECT table FROM system.parts WHERE active GROUP BY table HAVING count() > 100
)

UNION ALL
SELECT '========== 05. 慢查询 TOP 3（24h）==========' AS check_item
UNION ALL
SELECT concat(
    formatDateTime(query_start_time, '%m-%d %H:%M'), ' | ',
    toString(query_duration_ms), 'ms | ',
    substring(query, 1, 80)
)
FROM system.query_log
WHERE type = 'QueryFinish'
  AND query_start_time >= now() - INTERVAL 1 DAY
ORDER BY query_duration_ms DESC
LIMIT 3

UNION ALL
SELECT '========== 06. 僵尸 Part 数量 ==========' AS check_item
UNION ALL
SELECT concat('非活跃 Part 数：', toString(count())) FROM system.parts WHERE NOT active

UNION ALL
SELECT '========== 07. 内存追踪 ==========' AS check_item
UNION ALL
SELECT concat(
    metric, '：',
    if(metric = 'MemoryTracking', formatReadableSize(value), toString(value))
)
FROM system.metrics
WHERE metric IN ('MemoryTracking', 'Query', 'Merge')

UNION ALL
SELECT '========== 诊断完成 ==========' AS check_item;
```

> **使用方式**：凌晨三点被叫起来时，SSH 进节点，打开 `clickhouse-client`，粘贴这段 SQL，回车——30 秒内你就能判断问题出在前台查询、后台 Merge、Mutation 积压还是磁盘空间。

## 4. 项目总结

### 关键系统表速查

| 系统表 | 数据来源 | 核心字段 | 诊断用途 | 是否默认开启 |
|--------|---------|---------|---------|------------|
| `system.tables` | 元数据内存 | `database`, `total_rows`, `total_bytes`, `engine` | 数据资产盘点、容量规划 | ✅ 是 |
| `system.columns` | 元数据内存 | `data_compressed_bytes`, `type`, `compression_codec` | 列级存储分析、压缩效率评估 | ✅ 是 |
| `system.parts` | MergeTree 元数据内存 | `active`, `partition`, `rows`, `bytes_on_disk`, `ttl_info` | Part 爆炸预警、磁盘泄漏排查、TTL 状态 | ✅ 是 |
| `system.query_log` | 磁盘（MergeTree 表） | `query`, `query_duration_ms`, `read_rows`, `memory_usage` | 慢查询审计、查询模式分析 | ❌ 需手动启用 |
| `system.processes` | 内存快照 | `query_id`, `elapsed`, `memory_usage`, `query` | 实时负载诊断、卡住查询定位 | ✅ 是 |
| `system.merges` | 内存快照 | `table`, `rows_read`, `memory_usage` | Merge 风暴检测、合并健康度 | ✅ 是 |
| `system.mutations` | 内存 + ZooKeeper | `command`, `is_done`, `parts_to_do`, `elapsed` | ALTER 任务进度、Mutation 阻塞排查 | ✅ 是 |
| `system.metrics` | 内存实时计数器 | `metric`, `value` | CPU / 内存 / 连接数实时监控 | ✅ 是 |
| `system.errors` | 内存计数器 | `name`, `last_error_time`, `last_error_message` | 系统级故障预警 | ✅ 是 |

### 适用场景

- **日常巡检**：每天上班跑一遍健康检查脚本，替代手动登机巡检——Part 数异常的、Mutation 卡住的、慢查询突增的，一目了然。
- **故障应急响应**：凌晨 3 点被叫醒的"三板斧"——先看 `system.processes` 找当前负载源、再看 `system.merges` + `system.mutations` 判断是否是后台任务、最后查 `system.query_log` 回溯历史异常。
- **容量规划**：通过 `system.tables` 的 `total_bytes` 增长趋势预测磁盘需求，通过 `system.parts` 的 `part_count` 评估写入策略是否需要调整。
- **查询性能审计**：定期导出 `system.query_log` 中的慢查询报告，推动业务方优化 SQL——从"被动救火"转向"主动治理"。
- **上线前检查**：新业务上线前，用 `system.tables` 确认表结构、`system.columns` 确认压缩策略、`system.parts` 确认分区策略是否正确写入。

**不适用场景**：
- 需要历史趋势图表和自动告警的场景——系统表只能查当前值和日志历史，无法自动生成趋势图或发送告警。这正是后续配套 Prometheus + Grafana 的原因（第 28 章会详细介绍）。
- 需要跨节点聚合的分布式集群诊断——`system.parts`、`system.merges` 等表是节点级别的，你需要逐节点查询或通过 Distributed 表汇总。

### 注意事项

1. **`query_log` 必须显式开启**：它是系统表中唯一需要手动配置才能工作的。如果没有在 `config.xml` 中配置 `<query_log>` 块，`system.query_log` 表中不会有任何数据——这是新手最常见的困惑。
2. **`query_log` 会持续增长，必须配 TTL**：`query_log` 底层是真实的 MergeTree 表（存储在 `/var/lib/clickhouse/system/query_log/` 下），如果不设置 TTL，它会无限增长。建议配置 30 天 TTL：
   ```sql
   ALTER TABLE system.query_log MODIFY TTL event_date + INTERVAL 30 DAY DELETE;
   ```
3. **`system.processes` 只显示当前快照**：查询一结束就从 `processes` 中消失。它不是历史记录——想回溯查询历史，唯一的选择是 `system.query_log`。
4. **`system.metrics` 是累计计数器**：部分指标（如 `Query`、`Merge`）是"当前值"类型，含义是"此刻正在发生的事"；而 `system.events` 中的大部分指标是"累计值"类型，含义是"自进程启动以来的总次数"。二者容易混淆——判断当前负载用 `metrics`，判断历史总量用 `events`。
5. **系统表查询本身也会影响 `system.query_log`**：你在执行诊断 SQL 时，这些查询也会被记入 `query_log`。在分析慢查询时，注意过滤掉来自 `system` 数据库的查询，避免"观察者污染数据"。

### 常见踩坑经验

**坑 1：`query_log` 未配 TTL，`/var/lib/clickhouse` 被悄悄填满**

某团队的 ClickHouse 磁盘使用率每隔几周就飙到 95%，最初怀疑是业务数据增长太快。排查后发现 `system.query_log` 表的数据目录占用了 200GB+——因为 `query_log` 从集群搭建第一天就启用了，至今从未设过 TTL。每天数千条查询的日志日积月累，最终吃光了系统盘。**教训**：启用 `query_log` 的第一天就给它设 TTL——`ALTER TABLE system.query_log MODIFY TTL event_date + INTERVAL 30 DAY DELETE`。

**坑 2：`system.processes` 看不到慢查询，认为"集群没问题"**

某 DBA 凌晨巡检时发现 CPU 95%，但 `system.processes` 里一条查询都没有。他以为 CPU 高是"正常的后台开销"，就没管。第二天发现是某张表的 Partition 有一万多个小 Part，Merge 线程全量在跑，前台查询因 IO Wait 全部超时。**教训**：`system.processes` 只能看到**前台查询**。后台 Merge、Mutation 的负载要通过 `system.merges` 和 `system.mutations` 来观察。CPU 高且 `processes` 空，大概率是后台任务在"搞鬼"。

**坑 3：重启后 `system.metrics` 和 `system.events` 数据归零**

某团队在 `system.events` 中看到了一个奇怪的现象——某个指标的值每到凌晨就断崖式下跌。排查后发现是定时重启脚本导致的。**教训**：`system.metrics` 和 `system.events` 的数据存储在进程内存中，重启即清空。如果需要长期保留趋势数据，需要启用 `system.asynchronous_metric_log`（这是一张落盘的 MergeTree 表，类似 `query_log`），或者通过 Prometheus Exporter 将指标导出到外部时序库。
