# 第11章：用户管理与权限控制

> **版本**：ClickHouse 25.x LTS
> **定位**：基础篇核心章节。让你的 ClickHouse 安全地开放给团队——从"全员 default"到"最小权限原则"，构建企业级的用户、角色、行策略和配额管理体系。
> **前置阅读**：第2章（单机部署与客户端生态）
> **预计阅读**：35 分钟 | **实战耗时**：60 分钟

---

## 1. 项目背景

某数据平台团队维护着一套 ClickHouse 集群，支撑公司三个核心业务组：

- **分析团队**（8 人）：每天跑报表、制做运营看板，只需读取已聚合的 `analytics_db` 数据，不应接触包含手机号、身份证号的 `orders_raw` 原始表。
- **数据工程团队**（5 人）：负责 ETL 数据的导入和清洗，需要对 `staging_db` 有 INSERT 和 ALTER 权限，但不能 DROP 生产表或触碰 `analytics_db`。
- **DBA 团队**（2 人）：负责集群运维，需要全部管理权限。

现实却让人捏一把汗——所有人都在用 `default` 用户，拥有所有库表的全部权限。

上周三凌晨，分析师小王加班跑一份月度报表。他在 SQL 客户端里开了三个 tab，本想在 `test` 库里验证一条 `DROP TABLE` 语句，却不小心在 `production` 库执行了 `DROP TABLE orders`——一张存着 2 亿行交易记录的 MergeTree 表。虽然从备份恢复了，但从事故发生到完成恢复的 4 个小时内，所有依赖该表的看板和 ETL 任务全部中断，CEO 在群里连发三个问号。事后复盘，团队发现：如果小王只有 `SELECT` 权限，这条命令根本不会执行成功。

更隐秘的风险还在数据层面。市场营销团队在做用户画像分析时，能查到其他业务线的订单明细，包括竞品部门的客单价和供应商信息——这违反了公司内部的 **数据隔离政策**。CTO 在安全审计会上拍了桌子："一个分析师能删生产表，一个市场专员能看到财务数据，这套系统跟裸奔有什么区别？"

**核心痛点**：共享集群缺乏用户隔离，权限模型缺失，误操作和越权访问随时可能发生。必须引入 **RBAC（基于角色的访问控制）**——把"谁能做什么"从人肉约定变成系统强制。

## 2. 项目设计：剧本式交锋对话

周一晨会，大师把事故复盘报告投到屏幕上。小胖缩在角落假装看手机，小白已经打开了 ClickHouse 官方文档的"Access Control"页面。

**小胖**（偷偷瞄了一眼屏幕，嘟囔道）："我们团队就三个人，用 `default` 用户不就行了？搞那么多用户角色，多麻烦！而且每次登录还得记密码……我这脑子，连自己银行卡密码都记不住。"

**大师**（指了指屏幕上的事故报告）："小胖，你觉得小王想删生产表吗？他不是故意的，但他有能力这么做——因为他用的 `default` 用户有这个权限。这不是信任问题，是**能力问题**。如果他的账户根本没有 DROP 权限，就算他删代码删到手抽筋，ClickHouse 也不会理他。再退一步，如果那张表上了 ROW POLICY，就算有 SELECT 权限，他也只能看到自己部门的订单——隔壁营销团队想看财务数据？门都没有。"

**小胖**（收起手机）："那我给每个人新建一个用户，给个密码，权限直接 `GRANT ALL ON *.* TO 小胖` 不就行了？反正大家都自己人，不会乱搞。"

**小白**（推了推眼镜）："等一下，你这等于把一把万能钥匙复制了十份分给所有人，只是锁没换而已。假设有 100 个用户，每个人都要单独 GRANT 一遍，离职的时候还要逐个 DROP USER——这不是人肉运维是什么？"

**大师**："小白问到点子上了。这就是 **RBAC 模型**的价值。Role（角色）是用户和权限之间的中间层——你把权限授予角色，再把角色赋予用户。想象一下：银行不会给每个员工单独配金库的钥匙。柜员有柜员的权限，大堂经理有大堂经理的权限，金库主管有金库主管的权限。新人入职给他分配一个'柜员角色'就行，他离职了就收回这个角色，不需要重新造一把锁。"

**技术映射 #1**：ClickHouse 支持完整的 RBAC 模型：`CREATE USER` 创建身份 → `CREATE ROLE` 创建角色 → `GRANT 权限 TO 角色` → `GRANT 角色 TO 用户`。角色是权限的"打包盒子"，用户是权限的"持有者"。

---

**小白**（翻开笔记本，眉头紧锁）："权限粒度呢？我能精确到某个表的某几列吗？比如让分析师 SELECT `orders_summary` 表的所有列，但不能 SELECT `orders_raw` 表的 `phone` 和 `id_card` 列？"

**大师**："ClickHouse 的权限粒度分四级——**全局**（`*.*`）、**数据库级**（`analytics_db.*`）、**表级**（`analytics_db.orders`）和**列级**（`analytics_db.orders(amount, status)`）。你没提的列，用户根本看不到，就像那些列不存在一样。这是数据库层级的访问控制，比应用层 if-else 靠谱一万倍——代码可能有 bug，但 GRANT 是引擎强制执行的。"

**小胖**（挠头）："那行级控制呢？比如我想让分析师大刘只能看华南区的订单，小马只能看华北区的——总不能给每个区建一张表吧？"

**大师**："这就是 **ROW POLICY（行策略）** 的用武之地。它是一种 SQL 级的过滤器，自动附加到满足条件的用户的每条查询上。比如：

```sql
CREATE ROW POLICY region_isolation ON orders
    FOR SELECT
    USING region = currentUser()
    TO analyst_role;
```

从此以后，`analyst_role` 的所有用户执行 `SELECT * FROM orders` 时，ClickHouse 会自动在 WHERE 条件中追加 `AND region = '大刘'`（假设大刘是当前用户）。这个过滤对用户完全透明——他以为自己查了全表，实际上只能看到自己的数据。这就是**多租户数据隔离**的核心。"

**小白**："那性能呢？每条查询都自动附加一个 WHERE 条件，会不会拖慢查询？如果 ROW POLICY 的 USING 表达式很复杂，比如要 JOIN 一张权限映射表？"

**大师**："问到要害了。ROW POLICY 确实会带来额外开销——每行数据在返回前都要通过 USING 表达式的检查。但 ClickHouse 做了两层优化：第一，USING 表达式会被合并到主查询的 WHERE 条件中，利用主键索引和分区裁剪加速过滤；第二，对于简单的 `column = currentUser()` 表达式，开销通常在 5% 以内，几乎可以忽略。但如果你的 USING 里嵌了子查询、JOIN 了外部表，那每条查询都会触发额外的 I/O——这种情况建议用**字典（Dictionary）**把权限映射关系缓存在内存里。"

**技术映射 #2**：ROW POLICY 本质是"自动注入的 WHERE 子句"，在查询分析阶段附加到 AST 上。开销取决于 USING 表达式的复杂度——常量表达式几乎零开销，子查询则需要额外 I/O。字典映射是性能最优的复杂行策略实现方案。

---

**小胖**（突然想起什么）："等等！既然权限这么细，那万一有个人发了疯，在合法的 SELECT 权限下跑了 1000 条查询，把 CPU 占满怎么办？权限管不了'滥用'吧？"

**大师**（笑了起来）："小胖，你今天状态在线啊。Resource Control 也是 ClickHouse 权限体系的一部分——**QUOTA（配额）**。你可以给每个角色设定时间窗口内的使用上限："

```sql
CREATE QUOTA analyst_quota
    FOR INTERVAL 1 HOUR
        MAX queries = 1000,
        MAX result_rows = 100000,
        MAX errors = 10,
    FOR INTERVAL 1 DAY
        MAX queries = 5000,
        MAX read_rows = 100000000
    TO analyst_role;
```

"超过配额，ClickHouse 直接抛异常拒绝查询——不是警告，是硬拒绝。这是防止'合法滥用'的最后一道防线。"

**小白**："那配额计数是存在哪里的？服务重启后会清零吗？"

**大师**："配额计数器存在内存中——**是的，重启会清零**。这不是 bug，是设计权衡。如果把配额持久化到 ZooKeeper 或磁盘，每次查询都要写一次——对一张天天跑几十万查询的表来说，这个代价太高了。生产环境中建议：日常配额设定宽松一些（比如每小时 5000 条），靠 DBA 人工巡检发现异常；关键时刻收紧到 200 条来应对恶意查询。"

**小胖**："还有一个问题——如果有人从公司外面连我们的 ClickHouse 怎么办？VPN 破了不就全暴露了？"

**大师**："三层防御。第一层，网络级——在 `users.xml` 或 SQL-driven 配置中通过 `<networks>` 标签限制来源 IP 白名单；第二层，用户级——强密码策略（ClickHouse 25.x 支持 SHA256 哈希存储密码）；第三层，传输级——开启 TLS 加密。这三层叠加，外部攻击者即使拿到了密码，从非白名单 IP 也无法登录。"

**技术映射 #3**：ClickHouse 的权限体系 = **身份认证**（User/Password/Network） + **权限授权**（Role/Grant） + **数据过滤**（Row Policy/Column Grant） + **资源限制**（Quota/Settings Profile）。四层环环相扣，构成完整的访问安全矩阵。

---

**小白**（合上笔记本，问出最后一个问题）："配置方式呢？我在文档里看到两种——SQL 语句管理（`CREATE USER`、`GRANT` 这些）和 XML 配置文件。哪种更好？能混用吗？"

**大师**："这是本场最重要的选择题。ClickHouse 默认使用 **XML 文件**（`users.xml`）管理用户和权限——修改后需要 `SYSTEM RELOAD CONFIG` 或重启才能生效，而且不支持 ROW POLICY 和 QUOTA 的精细控制。但如果你在 `config.xml` 中开启 `access_management=1`，就切换到 **SQL-driven 模式**——所有权限操作都用 SQL 语句，即时生效，不需要重启，支持全套 RBAC 功能。

**绝对不能混用！** `access_management=1` 开启后，`users.xml` 里配置的用户和权限会被**覆盖忽略**，ClickHouse 将权限元数据存储在 `access/` 目录下的 `.sql` 文件中。如果没开启 `access_management` 就去执行 `CREATE USER`，ClickHouse 会直接报错。生产环境的标准姿势：先在 `config.xml` 中开启 `access_management=1`，然后所有权限管理只走 SQL。"

## 3. 项目实战

### 环境准备

你需要一个开启了 SQL-driven 权限管理的 ClickHouse 实例。如果使用 Docker：

```bash
mkdir -p ~/clickhouse-ch11/config.d
```

创建配置文件 `~/clickhouse-ch11/config.d/access_management.xml`：

```xml
<clickhouse>
    <access_management>1</access_management>
</clickhouse>
```

启动 ClickHouse：

```bash
docker run -d --name clickhouse-ch11 \
  -p 8123:8123 -p 9000:9000 \
  -v ~/clickhouse-ch11/config.d:/etc/clickhouse-server/config.d \
  clickhouse/clickhouse-server:25-lts
```

> **注意**：Docker 镜像的 `default` 用户初始密码为空。生产环境务必在启动时通过环境变量 `CLICKHOUSE_PASSWORD` 设置密码。

验证 `access_management` 已启用：

```sql
SHOW ACCESS;
```

如果返回用户和角色列表而非报错，说明 SQL 权限管理模式已生效。

### 分步实现

#### Step 1：创建角色体系

**目标**：建立三层角色——只读分析员（`analyst_role`）、数据工程师（`engineer_role`）、超级管理员（`admin_role`）。

```sql
-- 创建三个角色
CREATE ROLE analyst_role;
CREATE ROLE engineer_role;
CREATE ROLE admin_role;

-- 验证角色创建成功
SELECT name FROM system.roles WHERE storage = 'local directory';
```

预期输出：`analyst_role`、`engineer_role`、`admin_role` 三行。

---

#### Step 2：创建用户并分配密码

**目标**：为每个角色创建对应的用户账号，启用 SHA256 密码加密。

```sql
-- 创建用户（ClickHouse 25.x 默认使用 SHA256 哈希存储密码）
CREATE USER analyst_user IDENTIFIED BY 'Analyst@2026#Secure';
CREATE USER engineer_user IDENTIFIED BY 'Engineer@2026#Secure';
CREATE USER admin_user IDENTIFIED BY 'Admin@2026#UltraSecure';

-- 可选：为 default 用户修改默认密码
ALTER USER default IDENTIFIED BY 'Default@Changed#2026';
```

> **坑点提醒**：ClickHouse 不强制密码复杂度校验，你需要通过组织规范要求用户使用强密码。特权用户（`default`）的密码必须在部署后第一时间修改。

---

#### Step 3：配置数据库和表级权限

**目标**：实现最小权限原则——每个角色只获得完成工作所必需的权限。

```sql
-- ======== 建库（先以 default 用户执行）========
CREATE DATABASE analytics_db;
CREATE DATABASE staging_db;

-- 在 analytics_db 创建聚合表
CREATE TABLE analytics_db.orders_summary (
    order_date Date,
    department String,
    total_amount Decimal(18,2),
    order_count UInt64
) ENGINE = SummingMergeTree()
ORDER BY (order_date, department);

-- 在 analytics_db 创建包含 PII 的原始数据表（分析师不应访问）
CREATE TABLE analytics_db.orders_raw (
    order_id UInt64,
    customer_name String,
    phone String,
    id_card String,
    amount Decimal(18,2),
    department String,
    created_at DateTime
) ENGINE = MergeTree()
ORDER BY (created_at, department);

-- 在 staging_db 创建 ETL 工作表
CREATE TABLE staging_db.raw_import (
    raw_data String,
    import_time DateTime
) ENGINE = MergeTree()
ORDER BY import_time;

-- ======== 分析师权限：只读聚合表，不可见原始表 ========
GRANT SELECT ON analytics_db.orders_summary TO analyst_role;
GRANT SHOW TABLES ON analytics_db.* TO analyst_role;
-- 故意不给 orders_raw 的 SELECT 权限
-- 列级权限示例：如果能 SELECT orders_summary，也只能看指定列
-- GRANT SELECT(order_date, department, total_amount) ON analytics_db.orders_summary TO analyst_role;

-- ======== 工程师权限：staging_db 可读写，但不可 DROP ========
GRANT SELECT, INSERT, ALTER, CREATE ON staging_db.* TO engineer_role;
GRANT SHOW TABLES ON staging_db.* TO engineer_role;
-- 给工程师 analytics_db 的只读权限（方便他们做数据校验）
GRANT SELECT ON analytics_db.orders_summary TO engineer_role;
GRANT SELECT ON analytics_db.orders_raw TO engineer_role;

-- ======== 管理员权限：全部 ========
GRANT ALL ON *.* TO admin_role WITH GRANT OPTION;
-- WITH GRANT OPTION 允许管理员将自己的权限转授给其他用户

-- ======== 将角色分配给用户 ========
GRANT analyst_role TO analyst_user;
GRANT engineer_role TO engineer_user;
GRANT admin_role TO admin_user;

-- ======== 设置用户默认角色（登录时自动激活） ========
SET DEFAULT ROLE analyst_role TO analyst_user;
SET DEFAULT ROLE engineer_role TO engineer_user;
SET DEFAULT ROLE admin_role TO admin_user;
```

---

#### Step 4：配置行级安全策略（Row Policy）

**目标**：实现部门间数据隔离——`analyst_role` 的每个用户只能看到自己所属部门的订单数据。

先向两张表插入测试数据（以 `default` 用户执行）：

```sql
INSERT INTO analytics_db.orders_summary VALUES
    ('2026-04-01', 'market', 150000.00, 300),
    ('2026-04-01', 'finance', 200000.00, 150),
    ('2026-04-01', 'tech', 80000.00, 100);

INSERT INTO analytics_db.orders_raw VALUES
    (1, '张三', '13800001111', '110101199001011234', 500.00, 'market', '2026-04-01 10:00:00'),
    (2, '李四', '13900002222', '110101199002022345', 300.00, 'finance', '2026-04-01 11:00:00'),
    (3, '王五', '13700003333', '110101199003033456', 200.00, 'tech', '2026-04-01 12:00:00');
```

创建行策略——当用户名与部门名对应时，按 `department = currentUser()` 过滤：

```sql
-- 为 analyst_role 创建行级过滤策略
-- USING 表达式会在查询时自动注入 WHERE 条件
CREATE ROW POLICY dept_isolation ON analytics_db.orders_raw
    FOR SELECT
    USING department = currentUser()
    TO analyst_role;
```

> **说明**：`currentUser()` 返回当前连接的用户名。如果你的用户命名规范是"部门名_人名"（如 `market_daliu`），则 USING 表达式应该提取部门前缀：
> ```sql
> CREATE ROW POLICY dept_by_user ON analytics_db.orders_raw
>     FOR SELECT
>     USING department = extract(currentUser(), '^([^_]+)')
>     TO analyst_role;
> ```

查看已创建的行策略：

```sql
SELECT * FROM system.row_policies
WHERE database = 'analytics_db' AND table = 'orders_raw';
```

#### Step 5：配置配额管理

**目标**：限制 `analyst_role` 的资源消耗，防止单个用户过度使用集群资源。

```sql
-- 创建配额：限制每小时的查询数、返回行数和错误数
CREATE QUOTA analyst_quota
    FOR INTERVAL 1 HOUR
        MAX queries = 1000,
        MAX result_rows = 100000,
        MAX errors = 10,
    FOR INTERVAL 1 DAY
        MAX queries = 5000,
        MAX read_rows = 100000000,
        MAX execution_time = INTERVAL 2 HOUR
    TO analyst_role;

-- 创建配额：限制工程师的 INSERT 速率（防止批量灌入打崩集群）
CREATE QUOTA engineer_quota
    FOR INTERVAL 1 HOUR
        MAX queries = 2000,
        MAX result_rows = 500000,
        MAX errors = 50
    TO engineer_role;

-- 查看配额使用情况
SELECT
    quota_name,
    quota_key,
    queries,
    max_queries,
    errors,
    max_errors
FROM system.quotas_usage;
```

```sql
-- 查看配额限制定义
SELECT * FROM system.quota_limits;
```

---

#### Step 6：网络访问控制与 Settings Profile

**目标**：通过来源 IP 白名单和查询级参数限制增强安全边界。

SQL-driven 模式下，来源 IP 限制需要在 `config.xml` 或 `users.xml` 的 `<networks>` 标签中配置（SQL 语句暂不支持直接管理 IP 白名单）。以下为示例配置：

```xml
<!-- 在 config.xml 或 users.xml 中添加 -->
<users>
    <analyst_user>
        <networks>
            <ip>10.0.1.0/24</ip>
            <ip>10.0.2.100/32</ip>
        </networks>
    </analyst_user>
</users>
```

SQL 可控的 Settings Profile（查询参数限制）：

```sql
-- 创建 Settings Profile 限制分析师的查询并发数
CREATE SETTINGS PROFILE analyst_profile
    SETTINGS
        max_concurrent_queries_for_user = 5,
        max_memory_usage_for_user = 10000000000,   -- 10GB
        max_execution_time = 300,                   -- 单查询最长 300 秒
        readonly = 1                                -- 只读模式
    TO analyst_role;
```

#### Step 7：验证权限边界

**目标**：模拟各角色用户登录，验证权限边界是否正确生效。

**测试 1：分析师用户——只读聚合表，无法访问原始数据**

```sql
-- 以 analyst_user 登录后执行
-- ✅ 应该成功：SELECT 聚合表
SELECT count() FROM analytics_db.orders_summary;

-- ✅ 应该成功：SHOW TABLES
SHOW TABLES FROM analytics_db;

-- ❌ 应该失败：INSERT 无权限
INSERT INTO analytics_db.orders_summary VALUES ('2026-04-02', 'market', 1000, 1);
-- 预期错误：DB::Exception: analyst_user: Not enough privileges

-- ❌ 应该失败：SELECT 原始表无权限
SELECT * FROM analytics_db.orders_raw;
-- 预期错误：DB::Exception: analyst_user: Not enough privileges

-- ❌ 应该失败：DROP 无权限
DROP TABLE analytics_db.orders_summary;
-- 预期错误：DB::Exception: analyst_user: Not enough privileges
```

**测试 2：工程师用户——可操作 staging_db，但不可 DROP 任何表**

```sql
-- 以 engineer_user 登录后执行
-- ✅ 应该成功：在 staging_db 中 CREATE TABLE
CREATE TABLE staging_db.test_table (x UInt8) ENGINE = MergeTree() ORDER BY x;

-- ✅ 应该成功：INSERT 数据
INSERT INTO staging_db.test_table VALUES (1), (2), (3);

-- ❌ 应该失败：DROP TABLE 无权限（即使在自己的 staging_db 中）
DROP TABLE staging_db.test_table;
-- 预期错误：DB::Exception: engineer_user: Not enough privileges

-- ❌ 应该失败：访问 analytics_db 的无权限操作
INSERT INTO analytics_db.orders_summary VALUES ('2026-04-02', 'market', 1000, 1);
-- 预期错误：DB::Exception: engineer_user: Not enough privileges
```

**测试 3：管理用户——拥有全部权限**

**测试 4：行策略隔离——不同用户查询 orders_raw 看到不同数据**

由于我们在示例中用户名为 `analyst_user`，而行策略过滤 `department = currentUser()`，这意味着 `analyst_user` 查询 `orders_raw` 时会被过滤为 `department = 'analyst_user'`——这个部门不存在，返回空结果。

为了演示更直观的效果，额外创建两个角色用户来模拟多部门场景：

```sql
-- 创建两个有明确部门标签的用户
CREATE USER analyst_market IDENTIFIED BY 'Mkt@2026#Test';
CREATE USER analyst_finance IDENTIFIED BY 'Fin@2026#Test';

-- 分配角色 + 修改用户名对应部门
GRANT analyst_role TO analyst_market;
GRANT analyst_role TO analyst_finance;

-- 修改行策略，按用户名精确匹配部门（模拟真实场景）
DROP ROW POLICY dept_isolation ON analytics_db.orders_raw;

CREATE ROW POLICY dept_isolation_v2 ON analytics_db.orders_raw
    FOR SELECT
    USING department = currentUser()
    TO analyst_role;
```

然后插入对应部门的数据，并创建匹配部门名的用户来验证（例如创建名为 `market` 的用户查询会匹配 `department = 'market'`）。

## 4. 项目总结

### RBAC 组件对照表

| 组件 | 用途 | 示例 |
|------|------|------|
| **User** | 身份认证——你是谁 | `CREATE USER analyst_user IDENTIFIED BY '***'` |
| **Role** | 权限打包——你能以什么身份操作 | `CREATE ROLE analyst_role` |
| **Grant** | 权限授予——你对哪些对象能做哪些操作 | `GRANT SELECT ON analytics_db.* TO analyst_role` |
| **Row Policy** | 行级数据过滤——你能看到哪些行 | `CREATE ROW POLICY ... USING department = currentUser()` |
| **Quota** | 资源限制——你能用多少资源 | `CREATE QUOTA ... MAX queries = 1000` |
| **Settings Profile** | 查询参数限制——你的查询能跑多狠 | `CREATE SETTINGS PROFILE ... SETTINGS max_memory_usage_for_user = 10G` |
| **Network** | 来源 IP 白名单——你能从哪里连接 | `<networks><ip>10.0.1.0/24</ip></networks>` |

### ClickHouse RBAC vs 传统数据库

| 特性 | ClickHouse (SQL-driven) | MySQL 8.0 | PostgreSQL |
|------|------------------------|-----------|------------|
| RBAC 支持 | ✅ 完整（User + Role + Grant） | ✅ 完整 | ✅ 完整 |
| 行级安全 | ✅ ROW POLICY | ❌ 需配合 VIEW | ✅ RLS (Row Level Security) |
| 配额管理 | ✅ 内置 QUOTA | ❌ 需外部工具 | ❌ 需外部工具 |
| 配置方式 | SQL + XML 双模式 | SQL only | SQL only |
| 即时生效 | ✅ SQL 模式下即时生效 | ✅ FLUSH 后生效 | ✅ 即时生效 |

### 适用场景

- **共享多团队集群**：一个 ClickHouse 集群服务分析、工程、运维三个团队，各团队数据逻辑隔离。
- **数据安全合规（GDPR / 个保法）**：行策略确保用户只能看到自己部门的数据，列级权限隐藏 PII 字段——降低数据泄露风险。
- **生产/预发布环境分离**：同一集群通过不同数据库 + 不同角色实现物理共享、逻辑隔离，节省硬件成本。
- **SaaS 多租户**：ROW POLICY 按 `tenant_id` 过滤，一行 SQL 实现数百个租户的数据隔离，无需为每个租户建表。
- **实习生/外包人员管控**：给临时人员分配只读 + 行策略 + 配额的组合，合同到期收回角色即可。

**不适用场景**：
- 需要精确到秒级的计费型配额（ClickHouse 配额计数器重启后会清零）。
- 纯 XML 配置模式下无法使用 ROW POLICY（必须开启 `access_management=1`）。
- 列级权限在 SELECT * 场景下会静默隐藏列，可能导致应用层代码报错（建议应用层显式指定列名）。

### 注意事项

1. **`access_management` 必须显式开启**：默认值 `0` 走 XML 配置。如果没开就去 `CREATE USER`，ClickHouse 会报 `DB::Exception: Access management is not enabled`。
2. **`default` 用户密码必须改**：Docker 镜像默认空密码，裸连即最高权限——相当于你家大门没锁。生产环境部署脚本第一条必须是 `ALTER USER default IDENTIFIED BY '...'`。
3. **角色扁平化与继承**：`GRANT analyst_role TO analyst_user` 后，`SHOW GRANTS FOR analyst_user` 显示的是角色引用而非展开的权限列表。要查看具体权限，需要 `SHOW GRANTS FOR analyst_role`。
4. **ROW POLICY 会附加到每一条查询**：如果你的 USING 表达式包含子查询或 JOIN，高频查询场景下会产生显著的性能开销。建议字典缓存复杂的权限映射。
5. **配额计数器重启后归零**：故障恢复后所有用户重新开始计数。不要用配额做精确的计费用途，只做"防滥用限流"。

### 常见踩坑经验

**坑 1：开启 `access_management` 后自己登录不进去了**

某运维在 `config.xml` 中启用了 `access_management=1` 并重启了服务，然后习惯性用 `default` 无密码登录——失败。因为 `access_management=1` 切换后，ClickHouse 会读取 `access/` 目录下的 SQL 文件重建权限。如果此前没有创建任何用户，`default` 用户可能没有预设密码。**教训**：开启 `access_management` 之前，先用 `CREATE USER` 创建至少一个有管理权限的用户，或者在 `users.xml` 中预配置 fallback 管理员。

**坑 2：ROW POLICY 使用 `currentUser()` 但用户名含特殊字符**

某团队的用户名采用邮箱格式（如 `daliu@company.com`）。创建行策略 `USING email = currentUser()` 后查询始终返回空数组——因为 `@` 和 `.` 被 ClickHouse 解析了。**教训**：用户名避免使用 `@`、`.`、`-` 等保留字符。如果无法避免，在 USING 表达式中用字符串匹配函数（如 `replaceOne()`）处理。

**坑 3：`SHOW GRANTS` 误导了权限排查**

某 DBA 执行 `SHOW GRANTS FOR analyst_user`，看到只有 `GRANT analyst_role TO analyst_user`，以为权限给少了，又手动 GRANT 了一堆表权限——导致权限膨胀。**教训**：`SHOW GRANTS` 不扁平展开角色的内含权限。需要用 `SHOW GRANTS FOR analyst_role` 查看角色内具体权限，或者用 `SHOW CURRENT GRANTS` 查看当前连接用户的全部有效权限（含角色继承）。

### 思考题

1. **如果 100 个用户各有不同的行级数据访问权限，ROW POLICY 在 Distributed 表上的执行效率如何？有什么优化方案？**
   > 提示：Distributed 表接收查询后会扇出到各个 Shard。ROW POLICY 是在每个 Shard 本地执行的，还是由发起节点统一过滤？如果 100 种行策略对应 100 个 USING 条件，Distributed 查询会做多少次过滤？考虑将权限表加载为字典，用 `dictGet()` 替换复杂的 CASE WHEN 链。

2. **`access_management=1` 模式和 XML 配置方式能否混用？如果可以，优先级是怎样的？如果用户既在 `users.xml` 中定义了又在 SQL 中创建了同名用户，会发生什么？**
   > 提示：阅读 ClickHouse 源码 `src/Access/` 目录下 `AccessControl` 类的实现，关注 `loadStores()` 和 `addUsersConfig()` 的加载顺序。
