# 第 27 章 · 集群与高可用

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：Cluster / Sentinel 为独立大话题；本文只钉**与 Stream、向量、JSON 共存的键设计**。

## 本话目标

- 会用 **hash tag** 让相关 key 落同槽。  
- 复述：**跨槽没有服务端 join**。

## 项目背景

**贯穿设定**：**某电商平台**单机 Redis 容量或单点风险不够用时，会上 **Redis Cluster**；此前章节学的 List、Hash、ZSet **语法不变**，但多了**槽位与多 key 原子性**约束。

**与本章关系**：购物车、库存、用户锁等需 **`{userId}` 等 hash tag** 让相关 key **同槽**，才能 **Lua/事务** 一次搞定；全国榜单一枚大 ZSet 则面临**单槽热点**，要单独做容量与降级设计。**集群没有跨槽 join**，键名设计必须前置考虑槽位。

## 步步引导：多 key 操作为啥突然报错？

**大师**：同一段 Lua，单机顺、上集群红——你心里先猜什么原因？

**小白**：……是不是集群脾气怪？

**大师**：怪在**规矩更严**。你且看看那三个 key：**槽位**是不是**同一路**？若**各奔东西**，集群常常**不肯在一口锅里炒**。

**小白**：那怎么让它们同路？

**大师**：**`{tag}`** 像给一串 key **系同一根绳**，绳里那段字参与算槽——**相关的一家人，坐一桌**。

**小白**：全国排行榜就一个 ZSet，简单明了。

**大师**：简单在**建模**。再把**访问量**叠上去想想：**一只碗装天下**，烫不烫手？大促时，**分榜、预聚合**常常是下一课。

**小白**：哨兵一切换，会不会少几笔？

**大师**：**切换**像**换班**，班与班之间常有**极短的空隙**。业务侧若心里有**重试、幂等、读从可能旧**这三件事，空隙就不容易变成**事故**。详见 [23-持久化与副本.md](23-持久化与副本.md)。

## 小剧场：分号联号

总号没有全国一本账，只有**各号分账**——你要**跨号汇总**，自己打算盘（客户端聚合）。

---

## 分片之后，再无「全局有序」

**大师**：**Cluster** 按 **slot** 分家；**相关 key** 要用 **hash tag** `{user:1}:cart` / `{user:1}:orders` 落同槽，**多键事务/Lua** 才安全。**Stream、Vector、JSON** 都要服从这条铁律——**槽位不对，神仙难救**。

**小白**：跨槽能聚合吗？

**大师**：**客户端聚合**或**重构数据模型**；别幻想服务端 magically 帮你 join。面试能画出 **slot → 节点** 的箭头，就赢一半。

**趣味比方**：Cluster 像**分号联号**——总号没有「全国一本总账」，只有「各号分账 + 总部规则」。

---

## hash tag 速记

```text
SET {user:42}:profile ...
SET {user:42}:sessions ...
```

`{...}` 内字符串参与 CRC16 计算 slot，**选对业务主键**。

---

## Sentinel 一句

**主从 + 自动故障转移**；切换瞬间仍有**窗口**——业务要**重试、幂等、读从延迟容忍**。与持久化、复制细节联动 [23-持久化与副本.md](23-持久化与副本.md)。

---

## 与消息/向量章节联动

- **Stream 消费组**：消费者与 key 同集群；跨集群复制是**另一套架构**。  
- **Vector set**：大索引 key 的**容量与迁移**要提前规划。  
- **JSON + 索引**：Query Engine 在集群上的限制读官方说明。

---

## 动手试一试（有集群环境时）

```text
CLUSTER SLOTS
CLUSTER KEYSLOT {user:1}:cart
```

故意创建 **不同槽** 的两个 key，在 `redis-cli` 里试 `MGET` / 简单 Lua，记录**错误信息**；再给它们加上**同一 hash tag** 重试，对比结果。

## 实战锦囊

- 设计 key 时先问：**会不会有多 key 原子需求？** 有则 **tag 先行**。  
- Stream / 大向量 key：**容量与迁移**提前评估。  
- 画一张 **slot → 节点** 草图，面试与排障都能用。
