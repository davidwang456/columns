# 第 21 章 · 语义缓存与路由

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：依赖 **Vector set** 或等价相似检索；应用层策略为主。

## 本话目标

- 画出 **规则 → 语义缓存 → 全量 RAG** 三层路由。  
- 列出语义缓存 **翻车** 时的三道保险。

## 项目背景

**贯穿设定**：承接第 20 章：**某电商平台**在已具备向量检索的前提下，希望**相似问法**直接走缓存答案，降低大模型成本与延迟。

**与本章关系**：**语义缓存**是「向量相似 + 业务规则」的路由层：**像 ≠ 对**，必须配 **阈值、知识版本、抽检、黑名单** 等兜底，避免错答被长期缓存。Redis 侧存条目与索引；**正确性底线在业务策略**，不在相似度分数本身。

## 步步引导：省钱不能省底线

**大师**：语义缓存最想省下的，是什么？

**小白**：少调一次大模型，省钱省延迟。

**大师**：方向对。你再品两个字：**「像」**与**「对」**——相似度高时，**一定**对吗？

**小白**：……不一定。那还得留后手。

**大师**：后手可以层层叠：**阈值**先挡一刀，**版本**记得菜谱换没换，**抽检**让掌柜偶尔尝一口，**黑名单**拦住爱出事的那类问法。

**小白**：我把阈值拉到 0.99，是不是就稳？

**大师**：门槛一高，**命中率**往往跟着瘦。阈值适合在**离线集**上试、在**线上小流量**里磨——**两边数字**对齐了，再谈省钱。

**小白**：文档更新了，旧答案还在缓存里……

**大师**：这时**版本号**要像**菜单上的日期**——旧日期不是不能吃，但要**降权或拒卖**；必要时，**按文档 ID** 清一圈邻域，别让陈货当新鲜卖。

## 小剧场：记错熟客

伙计记错熟客口味，上错菜——要有**掌柜试吃**（抽检）和**换菜单**（版本失效）。

---

## 同一道难题，别打两次大模型

**大师**：用户问句经嵌入后与缓存里**最近邻**相似度超阈值，则**直接返回答案**或**返回答案片段**，少一次 LLM 调用——省钱、省延迟。**路由**则是：简单 FAQ 走规则，复杂问句走 RAG，再往上才是「工具调用 / Agent」。

**小白**：缓存会不会「胡言乱语」？

**大师**：会。**阈值、版本、人工抽检、TTL、黑名单问句**缺一不可；把语义缓存当**精确 key 缓存**会翻车。要记录 **cache hit / near-miss / poison hit**。

**趣味比方**：语义缓存像**茶楼伙计记熟客**：「这位爷上次爱听这段」——记错了就得挨板子，所以要有**掌柜抽查**。

---

## 键与元数据设计

- 缓存 value：可 JSON（[18-JSON类型.md](18-JSON类型.md)）存「答案 + 元数据 + 模型版本 + 嵌入版本」。  
- 与向量索引：Vector set 存 **query embedding → cache_key** 或 chunk 级映射；**按业务选**，没有银弹。

---

## 路由三层（示例）

| 层级 | 触发 | 成本 |
|------|------|------|
| 规则 / 模板 | 关键词、正则、意图分类 | 极低 |
| 语义缓存 | 相似度 > τ | 中（嵌入推理） |
| 全量 RAG + LLM | 低置信、强时效 | 高 |

---

## 踩坑

- **阈值过低**：胡说八道；**阈值过高**：缓存形同虚设。  
- **文档更新**后旧答案滞留：要有 **版本号与失效策略**。  
- **多语言**：嵌入模型是否跨语言；混用时分别建桶。

---

## 动手试一试（可无向量，先做纸面）

1. 选 10 个真实用户问句，人工标「该走规则 / 语义缓存 / 全 RAG」。  
2. 对语义缓存路径：用 [20 章](20-VectorSet与RAG迷你管道.md) 的 `VSIM` 找近邻，**人工看** Top3 是否安全可返。  
3. 记录指标：`hit`、`near-miss`、`poison-hit`（返了但错）各多少。

## 实战锦囊

- value 里存 **`answer + sources + model_version + embedding_version + ttl`**。  
- **强时效**问题（股价、库存）慎用语义缓存。  
- 与成本核算联动：**每次缓存命中省多少 token**，才能说服老板。
