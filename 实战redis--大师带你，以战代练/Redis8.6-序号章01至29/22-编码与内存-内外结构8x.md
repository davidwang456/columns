# 第 22 章 · 编码与内存：内外结构 8.x

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：编码随版本演进；以 `OBJECT ENCODING` 与源码 `tryEncoding` 分支为准。

## 本话目标

- 用 **`OBJECT ENCODING` / `MEMORY USAGE`** 解释「同样逻辑，为何内存差很多」。  
- 说出为何生产慎用 **`DEBUG OBJECT`**。

## 项目背景

**贯穿设定**：**某电商平台**常见结构如「用户最近浏览 id 列表」：业务逻辑不变，数据量一变，**内存却非线性上涨**。

**与本章关系**：同一逻辑在不同规模下可能对应 **listpack/ziplist → quicklist** 等编码迁移（以版本与实测为准）。用 **`OBJECT ENCODING`、`MEMORY USAGE`** 对照**当前进程**，比背过期的「触发条件表」可靠；大促复盘时用来解释「key 数没翻倍，内存却爆」。

## 步步引导：别背编码表

**小白**：把 ziplist、listpack 触发条件背下来……

**大师**：背表不是坏事。你再想：**小版本一变，表还准吗**？若不准，**哪把尺子**永远跟你的进程在一起？

**小白**：……`OBJECT ENCODING`？

**大师**：正是。**亲手敲**出来的结果，比**背来的表**更贴近你这台机子。

**小白**：内存曲线抬头了，是不是 Redis 病了？

**大师**：先别下诊断。温和排查三件事：**数据是不是真多了**、**有没有从紧凑走向臃肿的编码**、**新特性（如字段 TTL）是否登场**——`MEMORY USAGE` 与抽样 key，会帮你讲故事。

**小白**：我想上生产 `DEBUG OBJECT` 瞄一眼……

**大师**：这一眼有时**很重**。不妨先问自己：**有没有更轻的窗口**——监控、`OBJECT ENCODING`、采样导出——**先看轻的，再看重的**。

两把剑外形一样，一把夹钢薄、一把厚背——**掂重量**才知哪把适合你（内存画像）。

## listpack 与紧凑编码（入门向）

**大师**：早年教程常讲 ziplist、linkedlist；如今主线是 **listpack** 等更紧凑、更少边界的编码。**Hash** 在字段少时可走紧凑路径，字段多或带 **字段 TTL** 时行为更复杂——**别死记一张表**，要会 **动手验**。

**小白**：我 `DEBUG OBJECT` 行不行？

**大师**：**生产慎用**；开发机配合 `OBJECT ENCODING`、`MEMORY USAGE`。`DEBUG` 类命令可能阻塞或泄露细节，**面试官常考的是你知道危险**。

**趣味比方**：编码像**兵刃内里的夹钢**——外表都是 Redis，里面有的是薄刃，有的是重剑；比武前**掂量重量**（`MEMORY USAGE`）。

## 动手试一试 · 加练

```text
SET demo:s1 a
SET demo:s2 "长字符串超过阈值时观察编码是否变化"
OBJECT ENCODING demo:s1
OBJECT ENCODING demo:s2
MEMORY USAGE demo:s1
MEMORY USAGE demo:s2
```

再建一个 **大 Hash**（循环 `HSET` 几千 field），观察 `OBJECT ENCODING` 是否变化（因版本而异，**记录现象**即可）。
