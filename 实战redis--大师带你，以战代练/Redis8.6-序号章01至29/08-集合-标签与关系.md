# 第 08 章 · 集合：标签与关系

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：Set 经典命令；大数据集注意 **SUNION** 等复杂度。

## 本话目标

- 用 Set 表达 **标签、好友、黑白名单**，并会 **交集/并集**。  
- 牢记：**`SSCAN` 分页，`SMEMBERS` 慎用在热 key**。

## 项目背景

**贯穿设定**：商品类目、用户兴趣、设备黑名单等，在平台上常抽象成**「一堆不重复的成员」**及**集合运算**。

**与本章关系**：**Set** 存标签 **`SADD`**，推荐里算重叠用 **`SINTER`**，风控用 **`SISMEMBER`** O(1) 判断。体量到亿级再考虑 Bloom/库表等——那是**量级换档**，不是 Set 设计错误。

## 步步引导：去重不是只有 Set

**大师**：要做「这个人注册过没有」，你最先想到的 Redis 结构？

**小白**：`SADD` / `SISMEMBER`，Set 很顺手。

**大师**：在**规模尚小**时，这几乎总是最清晰的一招。你再往远处望一眼：**若 member 数以亿计**，内存曲线会怎么走？

**小白**：……会陡。那是不是要换别的？

**大师**：路分几叉：**数据库唯一约束**、**概率结构**（如 [Bloom](12-概率模块-Bloom与伙伴.md)）都是常见下一站。关键不是「Set 错」，而是**量级变了，工具跟着变**。

**小白**：共同好友，我写 SQL join 也很熟。

**大师**：熟是好事。你再估一下：**两份好友名单若都很大**，往返与计算主要花在哪？有时把边落在 Redis，用 **SINTER** 换**更少的 RTT**——是否划算，要**量**而不是**信仰**。

**小白**：我要拉全量粉丝列表，`SMEMBERS` 一行最爽。

**大师**：爽在写，疼在读——尤其在**热 key**上。你不妨试试 **`SSCAN` 分页**，像翻书一页一页读，**CPU**与**停顿**都更温柔。

## 小剧场：名册与算盘

Set 是**花名册**：只问「在不在」；若要问「第几名」——你拿错本子了，换 ZSet。

---

## 去重、关系、六度人脉的玩具箱

**大师**：**`SADD`/`SISMEMBER`/`SINTER`/`SUNION`**——标签、好友关系、共同关注、黑白名单，一套够用。经典思考题「六度分隔」可以这么玩：**广度优先（BFS）写在应用里，Redis 只存每个用户的「好友 Set」**；Redis 不是图数据库，但是**极好的边缓存**。

**小白**：Set 能分页吗？

**大师**：用 **`SSCAN`** 分页；`SMEMBERS` 一把梭在热 key 上，常让 **CPU 喘不过气**。

**趣味比方**：Set 像**江湖好友录**——只看「在不在册」，不讲顺序；要讲排名请移步 ZSet（[09-有序集合-排行榜与延时.md](09-有序集合-排行榜与延时.md)）。

## 实战片段

### 共同关注（交集）

```text
SADD user:1:follows u2 u3 u5
SADD user:2:follows u3 u7
SINTER user:1:follows user:2:follows
```

### 标签检索

```text
SADD tag:redis article:101 article:205
SISMEMBER tag:redis article:101
```

### 随机抽奖

- `SRANDMEMBER`：可指定数量，**默认不放回需注意选项**。  
- `SPOP`：**弹出**，适合「中奖移出奖池」。

---

## 实战要点

- **布隆过滤器替代方案**：巨量去重可上 **Set**，内存顶不住则换 [12-概率模块-Bloom与伙伴.md](12-概率模块-Bloom与伙伴.md) 的 Bloom（模块）——**精确换空间**。  
- **SUNIONSTORE / SINTERSTORE**：结果落盘新 key，适合预计算；注意 **时间复杂度与峰值内存**。  
- **大 Set 拆分**：按用户尾号、按时间桶拆 key，避免单 key 过亿 member。

---

## 与 List、ZSet 的边界

| 结构 | 何时用 Set |
|------|------------|
| 需要分数/排序 | 用 ZSet |
| 需要重复、有序队列 | 用 List / Stream |
| 只要唯一成员、测交集并集 | Set |

---

## 踩坑

- `SMEMBERS` 在生产热 key 上慎用。  
- 集群多 key 集合运算：**同槽**才能一次算完，否则客户端分批或改模型。

## 动手试一试

```text
SADD demo:tags:redis post:1 post:2 post:3
SADD demo:tags:mysql post:2 post:4
SINTER demo:tags:redis demo:tags:mysql
SUNION demo:tags:redis demo:tags:mysql
SSCAN demo:tags:redis 0 COUNT 2
```

自测：`SISMEMBER` 与 `SMISMEMBER`（若版本支持）在接口设计上的差别。

## 实战锦囊

- **SUNION/SINTER** 前先估 **集合大小**与 **时间复杂度**，大促前压测。  
- 抽奖用 `SPOP` 时确认 **是否放回** 的业务规则。  
- 集群上做多 key 集合运算 → **同槽**或**客户端分批**。
