# 第 12 章 · 概率模块：Bloom 与伙伴

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：**必须** `make BUILD_WITH_MODULES=yes` 且二进制含对应模块；详见 [04-编译秘笈-BUILD与模块对照表.md](04-编译秘笈-BUILD与模块对照表.md)。

## 本话目标

- 先 **`MODULE LIST` 自检**，再谈 Bloom。  
- 说清 **假阳性** 时业务怎么办（二次校验）。

## 项目背景

**贯穿设定**：注册、领券等场景若每次都查库判重，成本高；可用**概率结构**先做**粗筛**。

**与本章关系**：**Bloom** `BF.ADD`/`BF.EXISTS`：返回「可能存在」时**必须回源库二次确认**；**假阳性是设计内行为**，不能当最终裁决。依赖第 04 章所述模块已加载；券「是否领过」等可类比，具体以模块能力与容量规划为准。

## 选型口诀

- **Bloom**：「**可能存在假阳性、不删**」——注册前判重、缓存穿透预防。  
- **Cuckoo**：需要**有限删除**时多看一眼。  
- **t-digest**：**P99 延迟**、分位数曲线、SLA 仪表盘。  
- **Top-k**：**最热搜索词**、直播礼物榜近似。  
- **CMS**：**频次估计**，销售漏斗粗算。

---

## Bloom 最小走通（示意）

```text
BF.RESERVE users:bf 0.01 1000000
BF.ADD users:bf user:10001
BF.EXISTS users:bf user:10001
```

参数 **错误率与容量** 上线前要估算；扩容策略读官方文档。

## 与 HLL、Set 同台

| 需求 | 选择 |
|------|------|
| 判断是否「可能见过」 | Bloom / Cuckoo |
| 只要计数、不要成员 | HLL（[11-HyperLogLog-近似去重.md](11-HyperLogLog-近似去重.md)） |
| 必须精确成员集合 | Set |

## 踩坑

- **假阳性**：业务要能接受「误判存在」后的二次校验（如回源 DB）。  
- **无法列出元素**：概率结构不是 Set。  
- **镜像与云裁剪**：生产环境 `MODULE LIST` 为准。

---

模块源码在仓库 `modules/` 下各子目录（以你 checkout 为准）；命令定义在 `src/commands` 生成链路中。

## 动手试一试（无模块则跳过）

```text
MODULE LIST
BF.RESERVE demo:bf 0.01 10000
BF.ADD demo:bf user:1
BF.EXISTS demo:bf user:1
BF.EXISTS demo:bf user:999999
```

观察：`user:999999` 是否可能 **EXISTS 为 1**（假阳性）？记录现象写进笔记。

## 实战锦囊

- 把 **误判率、容量、元素量级**写进设计文档，方便扩容评审。  
- 与缓存穿透方案联动：**Bloom + 短 TTL 空值** 常组合拳。  
- t-digest / Top-k / CMS 按上文 **选型口诀** 各练一个最小 demo，开会时能举例。
