# 第 24 章 · 淘汰策略：从 LRU 到 LRM

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：**`volatile-lrm` / `allkeys-lrm` 为 8.6 新增策略**；精确语义以 [配置文档](https://redis.io/docs/latest/develop/reference/eviction/) 为准。

## 本话目标

- 说清 **LRU / LFU / lrm** 各解决什么「踢谁」哲学。  
- 改 `maxmemory-policy` 前知道要 **灰度与看哪些指标**。

## 项目背景

**贯穿设定**：**某电商平台**大促时，Redis 中 **String、Hash、ZSet** 等缓存键总量膨胀，触及 **`maxmemory`** 后必须**淘汰**部分键以继续服务。

**与本章关系**：**LRU / LFU / lrm（8.6）** 回答「先踢谁」，哲学不同，**命中率与延迟毛刺**随之变化。改策略要**灰度 + 看监控**，避免活动页瞬间集体回源打穿源站。

## 步步引导：满了不是事故结束

**大师**：内存告警来了，你第一反应是什么？

**小白**：是不是要崩了？

**大师**：先看清**此刻的配置**：有没有 `maxmemory`、有没有策略？有策略时，常常**先踢人**；没策略时，故事可能走向**写失败**或**OOM**——**同名症状，病因未必相同**。

**小白**：我统一 LRU，听起来公平。

**大师**：公平是个好词。你再想一种菜：**写一次、很久才读**——它算不算「冷」？有的策略专盯这种**少改的陈货**（如 lrm），与 LRU **各有各的偏心**。

**小白**：加内存条，一了百了？

**大师**：常常是正道。也别忘了问：**热点是不是打在一个 key 上**、**能不能在边上挡一层**——有时**拆与挡**比**加条子**更划算。

## 小剧场：撤冷碟

LRU：久未碰面的离席；lrm：很久没改过菜的冷碟先撤——**宴席不同，撤法不同**。

---

## 内存满了要踢人

**大师**：**LRU** 看近似最近使用，**LFU** 看频次；**8.6** 的 **lrm** 强调**最近修改**——**冷数据先走**，适合「写一次读很久」的缓存：资料型 key 读完躺平，应优先被腾位置。

**小白**：和 LRU 啥区别？

**大师**：**读多写少**与**写一次躺很久**的业务，淘汰曲线不同。**没有万能策略，只有万能压测**——上线前用真实访问模式灌一灌。

**趣味比方**：LRU 像「久未碰面的客人请离席」；lrm 像「很久没改过菜的冷碟先撤」——宴席不同，撤法不同。

---

## 配置心智

```text
CONFIG GET maxmemory
CONFIG GET maxmemory-policy
# 示例：仅演示键名，生产用配置文件管理
# maxmemory-policy allkeys-lru
```

切到 **`*-lrm`** 前：**备份、灰度、监控驱逐次数**（`INFO stats` 中与 keyspace 相关指标）、业务错误率。

---

## 选型速查

| 业务形态 | 可考虑 |
|----------|--------|
| 纯缓存、键均可丢 | `allkeys-lru` / `allkeys-lfu` / `allkeys-lrm` |
| 仅带 TTL 的可当缓存 | `volatile-*` |
| 有持久要求的关键 key | **不要轻易依赖淘汰**；设合理 TTL 与容量规划 |

## 与第 22、25 章联动

- 大 key + 频繁驱逐 → **延迟尖刺**。  
- 热 key 被淘汰 → **缓存雪崩**；配合随机 TTL、互斥重建。

## 动手试一试（练习实例，勿生产乱改）

```text
CONFIG GET maxmemory
CONFIG GET maxmemory-policy
INFO stats
```

在**可丢数据**的库：设小 `maxmemory`，灌数据直到触发驱逐，观察 **`evicted_keys`** 与业务错误率；再切策略（若允许）对比曲线。

## 实战锦囊

- 改策略 **灰度**：先影子、再从库、最后主库。  
- **热 key 被误踢** → 缓存雪崩；配合 **随机 TTL** 与 **互斥重建**。  
- 与 [22-编码与内存-内外结构8x.md](22-编码与内存-内外结构8x.md)：**大 key 驱逐**也会抖延迟。
