# 第 11 章 · HyperLogLog：近似去重

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：HyperLogLog 为内置概率结构；**无需** `BUILD_WITH_MODULES`。与 Bloom/CMS 不同，勿混用场景。

## 本话目标

- 用 HLL 估 **UV/DAU**，并说出**与 Set 的取舍**。  
- 知道 **不能**用 HLL 回答「某个用户今天来过吗」。

## 项目背景

**贯穿设定**：**某电商平台**运营要看 **UV/DAU 类大盘**，关心「大约多少人」，不要求枚举每一个 id。

**与本章关系**：**HyperLogLog** 用固定空间估基数；**不能**用来回答「某个用户今天是否来过」——后者要 Set 或 Bloom。**分工**：HLL 答「规模量级」；成员判定另选结构。

## 步步引导：要数清，还是数「大约」？

**大师**：统计日活，你心里**最踏实**的一招是什么？

**小白**：`SADD` 用户 id，再 `SCARD`，看得见摸得着。

**大师**：在规模适中时，这几乎是最**可解释**的方案。你把用户数再**乘十倍、百倍**想象一下：**内存曲线**还让你同样踏实吗？

**小白**：……会陡。那有没有「不那么胖」的法子？

**大师**：HLL 走另一条路：**固定空间**，换**一点点误差**。它回答的是「大约多少人」，不是「张三在不在」。

**小白**：那我偏用 HLL 查张三……

**大师**：它名叫 HyperLogLog，心里却要当它**估堆人头**用——善答「约莫多少」，不善答「张三在不在」。要点名，仍要 **Set** 或 **Bloom** 这类角色。

**小白**：想合并七天呢？

**大师**：`PFMERGE` 是常见桥梁。合并后的结果，也记得像普通 key 一样**给期限**，免得**雪球**悄悄滚大。

## 小剧场：望气不点名

将军问：「张三在阵中吗？」望气师只能答：「约三万人。」——**别逼望气师点名**。

---

## 数不清的人头，估个大概

**大师**：**`PFADD`/`PFCOUNT`/`PFMERGE`**——UV、DAU、去重近似计数。这里打个比方：**用一口「小气」换一大块内存**——固定空间估海量基数，换的是**可接受误差**。

**小白**：和 Set 比呢？

**大师**：Set **精确**但内存随基数涨；HLL **固定开销**，适合**海量 UV**。你若要「张三来没来」这种**成员判定**，HLL 不伺候——请换 Set 或 Bloom。

**趣味比方**：Set 是**点名册**；HLL 是**望气**——远远一看「约莫三万人」，不用认清每张脸。

---

## 最小示例

```text
PFADD uv:20260407 user:1 user:2 user:1
PFCOUNT uv:20260407
PFMERGE uv:week uv:20260401 uv:20260402
PFCOUNT uv:week
```

---

## 实战注意

- 同一 key 多次 `PFADD` 合并为一次估计。  
- `PFMERGE` 合并多日数据做总 UV；注意 **目标 key 会变大**，规划 TTL。  
- 需要**精确去重**时别硬上 HLL——**误差是特性不是 bug**。  
- 极端小基数时误差比例可能难看；**低流量活动**用 Set 也行。

## 动手试一试

```text
PFADD demo:uv:1 ua:chrome ua:firefox ua:chrome
PFCOUNT demo:uv:1
PFADD demo:uv:2 ua:safari
PFMERGE demo:uv:merge demo:uv:1 demo:uv:2
PFCOUNT demo:uv:merge
```

对比：同样数据若用 `SADD` + `SCARD`，**内存**差多少（`MEMORY USAGE`）？

## 实战锦囊

- **低流量活动** HLL 误差比例可能难看，必要时换 Set。  
- 报表需求要**可解释**时，准备「估算值 ± 说明」。  
- 与 [12-概率模块-Bloom与伙伴.md](12-概率模块-Bloom与伙伴.md) 对照：**「在不在」用 Bloom，「多少 distinct」用 HLL**。
