# 第 18 章 · JSON 类型

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：**原生 JSON** 类型与命令以官方文档为准；与「字符串塞 JSON」兼容迁移需双写或离线转换。

## 本话目标

- 比较 **`SET` 字符串 JSON** vs **原生 JSON 类型** 的改字段成本。  
- 列出 **迁移双写** 时要约定的三件事。

## 步步引导：先别整页重抄

**小白**：文档改一个字段，我 `GET` 改 JSON 再 `SET` 回去……

**大师**：能跑，但**并发下易丢更新**、**网络胖**。JSON 类型可用路径更新（以文档为准）。

**小白**：我双写 String 与 JSON，以谁为准？

**大师**：必须定 **版本号/时间戳** + **灰度切读** + **回滚策略**，否则两本账。

**小白**：没有索引的 JSON 和字符串有啥本质区别？

**大师**：**类型与命令**不同；若都不建索引，**都是 blob**——别神化类型。

## 小剧场：活页夹

整本抄在一张纸上，改一个字重抄一页；活页夹只换那一页——**但夹子里没目录**，照样找不到（缺索引）。

---

## 别再把文档塞进字符串里硬啃

**大师**：旧招是 `SET doc '{"a":1}'`，查询全靠应用反序列化，更新要么全量覆盖要么 Lua。**新招**是 **JSON 类型 + JSONPath**——更新子字段、类型校验、与索引联动（若启用 Query Engine）。

**小白**：我还要字符串版吗？

**大师**：迁移期可以**双写**；稳定后**删字符串 key**，留 JSON key。双写时要定**以谁为准**的时钟与版本号，别两本账对不上。

**趣味比方**：字符串 JSON 像**整本秘籍抄在一张纸上**——改一个字重抄一页；JSON 类型像**活页夹**——换一页即可。

---

## 实战要点

- **路径表达式**：按文档版本使用 **JSONPath** 语法；升级客户端库时盯 **breaking changes**。  
- **与 [19-RedisQueryEngine导论.md](19-RedisQueryEngine导论.md)**：**有索引才有「数据库感」**；无索引仍是「结构化 blob」。  
- **与 Hash**：字段扁平、无嵌套、要字段 TTL → Hash（[06-哈希-字段级过期.md](06-哈希-字段级过期.md)）可能更直接；深层嵌套、数组、文档查询 → JSON。

---

## 迁移检查清单

- [ ] 序列化格式统一（日期、数字精度）  
- [ ] 大文档分片策略（单 key 大小上限）  
- [ ] 与缓存一致性（Cache Aside / 订阅失效）  
- [ ] 监控：`MEMORY USAGE`、慢查询

---

## 踩坑

- **超大文档**：网络、持久化、复制全变慢。  
- **频繁小更新**：评估是否更适合 Hash 或拆子文档。  
- **客户端模拟路径错误**：先在 `redis-cli` 用 `JSON.GET` 验证。

---

## 动手试一试（无 JSON 命令则跳过）

```text
COMMAND INFO JSON.SET
```

若存在：按文档建 `demo:doc`，`JSON.SET` 嵌套字段，再 `JSON.GET` 子路径；对比同等操作在 **String** 上要几次往返。

## 实战锦囊

- 单文档大小设 **上限**，超大拆 **多 key** 或对象存储 + Redis 索引。  
- 与 [19-RedisQueryEngine导论.md](19-RedisQueryEngine导论.md) 联动前，先确认 **本机是否有所需查询命令**。
