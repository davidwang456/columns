# 第 28 章 · 综合试剑：推荐、缓存、向量与流

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：架构示意；命令以 **8.6** 与模块实际可用性为准。

## 本话目标

- 用 **一张草图**串起：缓存、兴趣、点击流、向量、订单。  
- 每条链路写出 **一个失败模式 + 兜底**。

## 项目背景

**贯穿设定**：回到**某电商平台**全貌：推荐、交易、履约、客服等多条链路**同时**依赖 Redis，但**故障模式彼此独立**。

**与本章关系**：用一张草图串起典型分工：**String** 缓存详情、**Hash** 会话、**ZSet** 榜单、**Stream** 订单/对账事件、**Geo** 自提、**Vector** 客服召回、**Pub/Sub** 运营侧刷新等——每条链路要写清 **一种失败 + 兜底**（穿透、积压、模型漂移、热 key……），避免「Redis 一包治百病」的幻觉。

## 步步引导：先画块，再画箭

**大师**：白纸一张，你第一笔想画什么？

**小白**：Kafka、K8s、微服务，显得完整……

**大师**：完整是后来的事。为师建议你**先留四块空地**：接入、应用、Redis、下游——再问自己：**哪一类数据一刻不能停、哪一类丢了还能补**？

**小白**：点击流和订单，能不能共用一个 Stream，省事？

**大师**：可以共，也可以分。你且轻声数：**谁读、谁写、谁负责删旧账、谁担责**——若答案**两拨人**，**两把 key** 往往更清静。

**小白**：向量召回要快，详情也想快，都放 Redis？

**大师**：都快是好事。再算一笔**内存**：向量已经**不瘦**，若再把**整页详情**堆在旁边，**账**要先算清。常见是 **向量 + 指针**，详情**回源**或**旁路存**。

## 小剧场：一桌菜

冷盘、热炒、甜品各用合适的盘——**一个 Stream 装天下**往往烫手。

## 一页纸架构题

**大师**：电商「猜你喜欢」+ 实时点击埋点 + 订单事件。**不要求**上 K8s，要求**说清数据落点、读写路径、失败模式**——架构评审里要听你**像负责人一样说话**，不像背书童子。

**小白**：弟子试试……

**大师**：先画四方块：**接入层 / 应用 / Redis / 下游（DB、模型、MQ）**，再画箭头：**谁是主、谁是辅、谁可丢、谁不可丢**。

## 口述模板（可直接练）

1. **Happy path**：用户请求 → 缓存命中 / 向量召回 → 返回。  
2. **降级**：模型超时 → 规则推荐 → 静态榜单。  
3. **观测**：QPS、P99、Stream lag、向量命中率。  
4. **安全**：TLS、ACL（[26-安全-ACL与TLS.md](26-安全-ACL与TLS.md)）。

## 动手试一试（白板即可）

1. A4 纸横过来：**四格**写 接入 / 应用 / Redis / 下游。  
2. 用三种颜色画箭：**读路径、写路径、异步路径**。  
3. 每条路径旁写一个字：**可丢** 或 **不可丢**。  
4. 找一位同事 **90 秒复述**，听不懂的地方就是你的漏洞。

## 实战锦囊

- 评审时带 **数字**：QPS、P99、lag、内存上限，至少各一个。  
- **降级路径**写进 Runbook，别只写在 PPT 备注里。
