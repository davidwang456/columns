# 第 15 章 · List 与 Stream 混合战

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：无新增硬依赖；架构选型属**工程权衡**。

## 本话目标

- 能画一张**简图**：List / Stream / Pub/Sub / 外部 MQ 各管哪类数据。  
- 用 **一个具体数字**（延迟、QPS、留存天数）支撑你的选型，而不是「我们用了 Kafka 所以高级」。

## 步步引导：先量体型，再选镖局

**小白**：我们直接上 Kafka！

**大师**：**日消息量多少？留存多久？团队会运维吗？** 三口答不上来，Kafka 可能是**负资产**。

**小白**：五千 QPS，Redis 扛得住吗？

**大师**：看 **value 大小、持久化、副本、热点 key**——同一数字，两种命运。**压测**。

**小白**：List 缓冲 + Stream 入账，会不会太绕？

**大师**：**削峰**时常这么干；关键是 **漏斗满** 时的策略：拒绝、降级还是扩容——写进 Runbook。

## 小剧场：村口与跨国

村口寄信（List），省内公文（Stream），跨国贸易（Kafka）——**别为寄信雇海关**。

---

## 何时不必请 Kafka

**大师**：**小中台**常见套路：**Stream** 做业务事件与审计；**List** 做**简单工作队列**或**削峰缓冲**；**Pub/Sub** 推配置。Kafka 不是勋章，**过重也是债**——能讲清边界的人，面试加分。

**小白**：我们 QPS 五千，能上吗？

**大师**：看**持久化配置、副本延迟、热点 key、内存上限、value 大小**——五千可以是「轻松」，也可以是「作死」。**压测比口水准**；监控看 [25-可观测与HOTKEYS.md](25-可观测与HOTKEYS.md)。

**趣味比方**：消息中间件像**镖局规模**：村口送信 List 足够；跨省大宗 Stream；跨国 Kafka——别为送封信包下一整支镖队。

## 一种务实切法

| 数据 | 载体 | 备注 |
|------|------|------|
| 订单状态机事件 | Stream + 消费组 | 可回溯、可 ACK |
| 瞬时通知 | Pub/Sub | 见 [14-PubSub家族.md](14-PubSub家族.md) |
| 夜间批任务队列 | List + BLPOP | 简单、易理解 |
| 重试必须幂等的写入 | Stream + **IDMP**（8.6） | 见 [13-Stream-消费组与XADD幂等.md](13-Stream-消费组与XADD幂等.md) |
| 超大吞吐 + 长期留存 | 外部 Kafka / Pulsar | Redis 不是万能 MQ |

## 混合战故事：峰值削峰

**大师**：大促瞬时写爆：网关先 **`LPUSH` 到缓冲列表**，worker 按速率 **`BRPOP`** 消费，再 **`XADD` 入账流**——List 当**漏斗颈口**，Stream 当**总账**。关键是**漏斗满了怎么办**：拒绝、降级、还是扩容。

---

## 失败模式（提前写在架构图边上）

- 消费者全挂：List 长度暴涨、Stream 滞后飙升  
- 重复消费：业务幂等键 + Stream IDMP（写侧）  
- 单 key 热点：分片、多 Stream、按用户哈希
