# 第 06 章 · 哈希：字段级过期

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：**字段级过期**：`HEXPIRE` 等 **7.4+**；**`HSETEX` 8.0+**（见 `src/commands/hsetex.json`）。无需 `BUILD_WITH_MODULES`。

## 本话目标

- 分清：**整 key 过期** vs **单个 field 过期**。  
- 能写出一组 **session / token** 场景命令，并解释业务语义。

## 步步引导：别为一只闹钟拆十间房

**小白**：session 里 access token 要短过期，user_id 要长期，我是不是拆十个 String key？

**大师**：能拆，但 key 爆炸、事务难写。更干净的是 **一个 Hash，多 field，各自 TTL**——前提是版本 ≥7.4/8.0 支持你的命令族。

**小白**：field 过期了，`HGET` 空，我以为用户没登录……

**大师**：所以要设计 **哨兵**或**版本号**：区分「从未写入」与「写过但过期」。别让用户背锅。

**小白**：`HSETEX` 报错，我复制粘贴的啊？

**大师**：停。**参数顺序**是 Redis 的脾气——先 `HELP HSETEX`，再敲。复制粘贴不带脑，等于**蒙眼耍刀**。

## 小剧场：整柜扔鞋

房东说「租约到期全屋清空」，你只想扔门口一双鞋——字段级过期就是**只打开那一个抽屉**。

---

## 对象表 + 每列自己的「更漏」

**大师**：以前 Hash 像一张表，整张表共用一个 **key TTL**。如今每列（field）可自带闹钟——**token 一刻钟，user_id 永久**，不必拆成十个 key，也不必把 session 砍成「字符串套娃」。

**小白**：闹钟响了列没了，`HGET` 空？

**大师**：正是语义。业务要区分「从未设置」与「过期删除」，有时需 **版本号或哨兵字段**；审计日志要能说清**谁设的 TTL**。

**趣味比方**：以前是「整屋租约到期一起搬」；现在是**每个抽屉单独租**——省得为了扔一双鞋把整柜家具抬出去。

---

## 命令族（名称以 `HELP @hash` 为准）

- 设过期：`HEXPIRE` / `HPEXPIRE` / `HEXPIREAT` / `HPEXPIREAT`  
- 读写顺带过期：`HGETEX`、`HSETEX`（注意 **`HSETEX`：过期选项在 `FIELDS` 块之前**）  
- 查询 TTL：`HTTL` / `HPTTL`；绝对时间：`HEXPIRETIME` / `HPEXPIRETIME`  
- 清过期：`HPERSIST`（以文档为准）

**大师**：参数顺序错了，Redis 不跟你客气——**先看 HELP，再敲键盘**，比看十篇二手博客快。

---

## 最小示例

```text
HSET session:u1 user_id 42
HSETEX session:u1 PX 900000 FIELDS 1 token "abc"
HTTL session:u1 FIELDS 1 token
HGET session:u1 token
```

等 token 过期后再 `HGET`，应与业务预期一致（空或 nil，依客户端而定）。

---

## 实战故事：登录态三段式

| 字段 | TTL 策略 | 原因 |
|------|----------|------|
| `user_id` | 无或极长 | 账号标识不随 access token 变 |
| `token` | 短（15m） | 泄漏窗口小 |
| `refresh` | 中（7d） | 可轮换，配合服务端黑名单 |

**小白**：为啥不三个 key？

**大师**：可以，但 **一次 HGETALL** 与 **三次 GET** 的网络与键空间成本不同；更重要的是**事务边界**——同槽同 key 的 Lua/事务更简单（集群见 [27-集群与高可用.md](27-集群与高可用.md)）。

---

## 与 String JSON 的边界

- 字段多、要单字段过期、要部分更新：**Hash / JSON 类型**（[18-JSON类型.md](18-JSON类型.md)）更合适。  
- 整 blob 缓存、很少拆字段：**String** 仍可能更简单。

---

## 踩坑

- **复制与持久化**：字段 TTL 在故障恢复后的表现，以发行说明与测试为准；**上线前在从库演练**。  
- **大 Hash**：单 key 字段极多会影响编码与延迟；该分片就分片。  
- **误用 `HGETALL`**：生产热 key 上慎用，必要时 `HSCAN`。


## 实战锦囊

- 上线前在**从库**演练「主从切换 + 字段过期」是否符合预期。  
- 热 Hash 慎用 `HGETALL`，用 `HSCAN` 或只取必要 field。  
- 与 [18-JSON类型.md](18-JSON类型.md) 比：要**深层嵌套文档**时 JSON 更顺手；要**扁平字段 + 字段 TTL** 时 Hash 更直接。
