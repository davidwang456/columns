# 第 19 章 · Redis Query Engine 导论

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：Query Engine 能力集与**发行形态**强相关；**OSS 文档**与**本机命令**冲突时，以本机为准。

## 本话目标

- 分清 **Query Engine** 与 **Vector set / 简单二级索引** 的边界。  
- 评审时能问出：**索引延迟、删除一致性、分页边界**。

## 项目背景

**贯穿设定**：**某电商平台**在 Redis 里存了大量 **JSON 文档**（商品、活动）后，会出现需求：按**类目、标签、上架状态**等**组合条件**筛一批 key，而不想把全量拉回应用内存过滤。

**与本章关系**：**Redis Query Engine**（发行形态支持时）承担「Redis 内的条件查询/索引」角色。评审须问：**索引延迟、删除一致性、分页边界**——它与「单个 JSON 键 CRUD」不是同一层问题。

## 步步引导：别拿官网截图当环境

**大师**：文档里的查询跑得通，你本机红字——第一反应是什么？

**小白**：……是不是我装错了？

**大师**：这反应已经很好。下一步，不急着重装，先**轻声核对**：`INFO` 里的版本、`COMMAND` 里有没有那条词——**把「我这里的真相」说清楚**，往往就赢了一半面试分。

**小白**：有了 Query Engine，是不是可以少用一个 MySQL？

**大师**：先问**还要不要复杂事务、还要不要跨表 join**。若答案是「常常要」，关系库多半仍在台中央；Query Engine 更像**贴身的快查桌**。

**小白**：索引一加，写入好像钝了。

**大师**：索引是**多出来的活**，钝一点并不意外。要紧的是：**钝了多少、能不能接受**——用 **写入 TPS 与 P99** 说话，别只晒查询曲线。

## 小剧场：卷宗架

卷宗架（索引）再好用，也得有人**按规矩归档**；乱塞只会「找不到」与「找太慢」并存。

---

## 当 Redis 像「能查的文档库」

**大师**：在 **Hash/JSON** 上建**二级索引**，再做**全文、向量（在文档叙事中）、地理、聚合**——这是 **Query Engine** 的路线。**不是**替代 `GET`/`HGET` 的简单缓存，而是**查询引擎**：要能讲清**索引模型、查询语言、一致性**。

**小白**：和 **Vector set** 是一回事吗？

**大师**：**不是同一条兵器**。Vector set 是**原生类型 + `VADD`/`VSIM`**（[20-VectorSet与RAG迷你管道.md](20-VectorSet与RAG迷你管道.md)）；Query Engine 是**另一套索引与查询语言**，常与 JSON 文档搭配。**两篇文分开读，分开试**——面试混为一谈会扣分。

**趣味比方**：Vector set 像**独门暗器**；Query Engine 像**刑名师爷的卷宗架**——都能找人，一个靠「形似」，一个靠「条目」。

## 写作与读者纪律

1. 文内示例前运行 **`COMMAND INFO`** 或等价检查。  
2. 涉及 **RedisVL / Cloud**，单列一节标注环境。  
3. 别把官网截图当**你笔记本上的真相**。  
4. 设计评审要能回答：**更新索引延迟、删除是否同步、与副本一致性**。

---

## 实战落地顺序（建议）

1. 数据模型：JSON 文档结构、主键、标签字段。  
2. 索引：哪些字段可筛、哪些要全文、哪些要向量（若支持）。  
3. 查询：分页、排序、组合条件；**禁止**无界扫描。  
4. 观测：慢查询、索引大小、`INFO` 相关指标。

---

## 与关系库、搜索引擎边界

| 问题 | 倾向 |
|------|------|
| 复杂 join、强事务 | 关系库 |
| 海量日志检索、分词生态 | ES/OpenSearch 等 |
| 低延迟、与缓存同生命周期 | Query Engine + Redis |

## 动手试一试

1. 结合官方文档与 `HELP`、`COMMAND` 子命令（因版本而异），列出**你环境**里与查询/索引相关的命令清单。  
2. 对 **1k / 100k** 文档各跑一页查询，记录 **P99**。  
3. 做一次 **删除文档 + 再查**，看是否存在**旧索引残留窗口**。

- 云上/混合部署时，在文档页眉写 **环境标签**，避免团队误用命令。
