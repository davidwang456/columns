# 第 25 章 · 可观测与 HOTKEYS

本故事纯属虚构，如有雷同，纯属巧合。

> **版本说明**：**`HOTKEYS` 为 8.6**；可能带开销，生产启用前读官方警告。可与 `redis-cli --pipe` 灌库对照：灌完用热点分析**验伤**。

## 本话目标

- 建立排障顺序：**延迟 → 热 key → 大 key → 持久化抖动**。  
- 能安全试用 **`HOTKEYS` / 内存工具**，知道**不在主库瞎折腾**。

## 项目背景

**贯穿设定**：**某电商平台**大促时，**秒杀 SKU、全站开关、爆款缓存** 往往成为 **热 key**，与具体是 String 还是 Hash **无必然对应**——任何类型都可能被流量打烫。

**与本章关系**：延迟升高时按序：**延迟与慢日志 → commandstats →（版本允许时）`HOTKEYS`（注意开销）或代理统计**，用证据定位热点。忌在高峰对生产实例 `KEYS *` 做全表扫描式排查。

## 步步引导：慢，先别 KEYS

**大师**：latency 高了，你最先想抓什么？

**小白**：想把 key 都列出来看看……`KEYS *`？

**大师**：心急时都会这么想。你且缓半步：**全表扫描**对正在病着的实例，像**病人跑步测心跳**——我们先试试 **`LATENCY DOCTOR`、慢日志、commandstats** 这些**更轻的听诊器**。

**小白**：我感觉就是热 key……

**大师**：感觉常常是线索。能不能再**多一页数据**——客户端统计、代理、或版本允许时的热点命令——让**感觉**长成**证据**？

**小白**：压测我灌得很满，上线应该差不多吧？

**大师**：灌满只回答「**装不装得下**」。若访问**忽冷忽热、忽大忽小**，还要尽量**仿真实走路**——否则**抖**在上线后才露面。

## 热点、直方图、延迟

**大师**：线上慢，先问三句话：**是不是单个 key 被打穿？是不是大 key？是不是持久化/fsync 抖了？** **8.6** 给 **`HOTKEYS`** 与**内存直方图**利器，别只会 `KEYS *`——那是**自毁式排障**。

**小白**：我开 `HOTKEYS` 会不会把服务打挂？

**大师**：**读文档里的采样与代价**；在**从库或影子环境**先试。面试官想听你说出「**知道有风险所以怎么隔离试**」。

**趣味比方**：可观测像**把脉**：先分浮沉迟数，再开方；上来就「全库 KEYS」等于**剖肚验伤**。

---

## 组合拳

- **延迟**：`LATENCY DOCTOR`、慢日志、`INFO commandstats`。  
- **内存**：`MEMORY STATS`、大 key 扫描（`redis-cli --bigkeys` 等工具链）、`MEMORY DOCTOR`。  
- **灌库压测**：`redis-cli --pipe` + 限速 + 错误日志；对比 **QPS 与 P99**。  
- **热点**：`HOTKEYS`（8.6）、客户端采样、代理层统计。

---

## 热点治理思路（简表）

| 手段 | 说明 |
|------|------|
| 本地缓存 | 边缘扛读 |
| 分片 key | `foo:{slot}:1` 多副本分散（注意一致性） |
| 读写分离 | 从库分担读，容忍延迟 |
| 限流 | 保护源与 Redis |

集群槽位与 hash tag 见 [27-集群与高可用.md](27-集群与高可用.md)。

## 动手试一试

```text
LATENCY DOCTOR
INFO commandstats
MEMORY STATS
```

在**从库或空库**：若支持 `HOTKEYS`，小流量试跑并记录 **CPU 与延迟**；再用 `redis-cli --bigkeys`（或同类工具）扫一遍，对比两种手段的**结论是否一致**。
