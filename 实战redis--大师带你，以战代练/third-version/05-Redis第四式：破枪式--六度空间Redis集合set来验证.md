**大师**：这个六度空间系统要用到我们的 Redis 第四式：破枪式 Set。Set 是 string 类型的无序集合。集合成员是唯一的，这就意味着集合中不能出现重复的数据。Redis 中集合在较大规模时通常用哈希表实现，所以添加、删除、查找的平均复杂度为 O(1)。集合中最大的成员数为 2^32 - 1（4294967295，每个集合可存储 40 多亿个成员）。它的招式主要有：

**番外：Set 真是“一个哈希表”那么简单吗？**

**大师**：文档常说 Set 底层是哈希表，对你这份源码而言要更细一点：`t_set.c` 里会根据元素形态与规模在 **`intset`（全整型且够小）**、**`listpack`** 与 **`OBJ_ENCODING_HT`（dict）** 之间转换。`redis.conf` 里有 `set-max-intset-entries`、`set-max-listpack-entries`、`set-max-listpack-value` 等与 `server.h` 中 `redisServer` 字段对应。

**小白**：那我做六度人脉 `SINTER`，复杂度要注意啥？

**大师**：交集、并集在元素很多时会吃 CPU 与内存；你的 BFS 若每层 `SUNIONSTORE` 膨胀极快，要配合 **剪枝、分层落库、或换图引擎**。源码能告诉你：为什么同一命令在小集合上飞快、集合一大就陡增——正是编码切换与参与集合基数在起作用。
