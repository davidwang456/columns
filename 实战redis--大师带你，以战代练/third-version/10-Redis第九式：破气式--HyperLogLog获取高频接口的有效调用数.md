**Redis 第九式：破气式**

**大师**：监控是整个软件乃至整个产品生命周期中最重要的一环，事前及时预警发现故障，事后提供详实的数据用于追查定位问题。现在如果想要监控系统调用比较频繁的接口如根据用户 id 查询用户信息接口的有效访问数的话，有效访问数不能包含失败重试的次数，你如何设计呢？

**小白**：大师，我想到两种方法：1.在数据库里存放一个接口的调用情况，每次调用生成一个 token，统计 token 的数目即可以得到有效的调用次数。2.在 Redis 使用一个集合 set 结构存储，将每次调用的 token 存放的这个 set 集合中，使用 SCARD key 命令获取有效的调用次数。第一种方法适合调用量比较少的接口，因为数据库的读写性能决定的；第二种方法适合调用量比较大的接口。

**大师**：很不错，上面的方法可以适用于大部分场合。假设每天一亿次调用 ，1.如果是放到数据库中，io 性能问题，可能会影响到别的使用。2.假设一个 token 的大小为 4 个 Byte，那么如果一亿次有效访问要放到 set 中的话，就是 4 亿 Byte，大约 300~400G，放到 Redis 里显然是不合适的。有没有别的更节省内存的方式呢？

**小白**：那是否可以使用 Redis 破箭式中提到的 BloomFilter，将 token 做 hash 处理后，放入 BloomFilter 中进行统计呢？

**大师**：这是个好思路。1 亿 bit 大约占用 11MB，如果有 100 个这样的对象，就需要约 1.2GB 内存，可见占用仍然很大，在实际业务中单独用 Bitmap 做海量去重并不常见。其实还有一个更省内存的方式，这就是本次独孤九剑的最后一式：破气式。这里先提一个基本概念：基数计算（cardinality counting）指的是统计一批数据里不重复元素的个数，常见于独立用户数（UV）、维度独立取值数等。HyperLogLog 是一种基数估算算法，每个 HyperLogLog 键大约只需 12KB 内存，就可以估算接近 2^64 个不同元素的基数；成本相对更低，适合大数据量统计，例如页面访问量或用户访问量。

它的招式非常少：

> 1.PFADD key element [element ...]
> 
> 时间复杂度：O(1)
> 
> 将除了第一个参数以外的参数存储到以第一个参数为变量名的 HyperLogLog 结构中.
> 
> 如果一个 HyperLogLog 的估计的近似基数在执行命令过程中发了变化， PFADD 返回 1，否则返回 0，如果指定的 key 不存在，这个命令会自动创建一个空的 HyperLogLog 结构（指定长度和编码的字符串）.
> 
> 如果在调用该命令时仅提供变量名而不指定元素也是可以的，如果这个变量名存在，则不会有任何操作，如果不存在，则会创建一个数据结构（返回 1）
> 
> 2.PFCOUNT key [key ...]
> 
> 时间复杂度：O(1)
> 
> 当参数为一个 key 时,返回存储在 HyperLogLog 结构体的该变量的近似基数，如果该变量不存在,则返回 0.
> 
> 当参数为多个 key 时，返回这些 HyperLogLog 并集的近似基数，这个值是将所给定的所有 key 的 HyperLoglog 结构合并到一个临时的 HyperLogLog 结构中计算而得到的.
> 
> HyperLogLog 可以使用固定且很少的内存（每个 HyperLogLog 结构需要 12K 字节再加上 key 本身的几个字节）来存储集合的唯一元素.
> 
> 返回的可见集合基数并不是精确值， 而是一个带有 0.81% 标准错误（standard error）的近似值.
> 
> 例如为了记录一天会执行多少次各不相同的搜索查询， 一个程序可以在每次执行搜索查询时调用一次 PFADD， 并通过调用 PFCOUNT 命令来获取这个记录的近似结果.
> 
> 3.PFMERGE destkey sourcekey [sourcekey ...]
> 
> 时间复杂度：O(N)
> 
> 将多个 HyperLogLog 合并（merge）为一个 HyperLogLog ， 合并后的 HyperLogLog 的基数接近于所有输入 HyperLogLog 的可见集合（observed set）的并集.
> 
> 合并得出的 HyperLogLog 会被储存在目标变量（第一个参数）里面， 如果该键并不存在， 那么命令在执行之前， 会先为该键创建一个空的.

**小白**：大师，HyperLogLog 占用的内存那么少，还能做大数据量的统计，那么它的缺点是什么呢？

**番外：PFADD 在 `hyperloglog.c` 里做了什么？**

**大师**：HyperLogLog 的实现集中在 **`hyperloglog.c`**：`PFADD`/`PFCOUNT`/`PFMERGE` 的注册可在命令表里查到。稀疏与稠密两种编码在内存中切换，**标准误差约 0.81%** 来自 HLL 算法本身，不是 Redis 偷懒。`PFCOUNT` 多 key 时会做 **内部合并再估算**（源码注释写明是为加速多 key 场景）。

**小白**：和 Set 去重怎么选？

**大师**：要**精确成员集合**就用 Set；要**海量 UV、只关心量级**就用 HLL。读 `PFADD` 实现你能看到：每个 element 被 hash 到多个寄存器做 **stochastic averaging**——所以**不能删元素、不能当布隆过滤器用**，和 Bitmap/Bloom 的误判模型也不同。

**大师**：HyperLogLog 提供的是不精确的去重计数方案，但并非胡来，标准误差约在 0.81% 左右。HyperLogLog 是 Redis 的高级数据结构，非常有用，但会用的人不多。
