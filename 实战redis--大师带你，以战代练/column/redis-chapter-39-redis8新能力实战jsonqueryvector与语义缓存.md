# 第39章：Redis 8.x 新能力实战：JSON、Query、Vector 与语义缓存

## 1. 项目背景

很多团队最初使用 Redis 只是为了缓存商品详情、登录态和排行榜。但到了智能客服、知识库问答、推荐解释、风控画像等场景，业务不再只需要 `GET key`，而是希望在 Redis 里存结构化对象、按字段过滤、按向量相似度召回，再把高频问题的答案缓存起来，减少大模型调用成本。

传统做法通常拆成三套系统：对象放数据库，搜索放 Elasticsearch，向量放专门向量库，热点答案放 Redis。拆分带来灵活性，也带来链路长、数据同步复杂、延迟抖动和排障困难。Redis 8.x 及 Redis Stack 生态把 JSON、Query、Vector 等能力组合起来后，可以在中小规模、低延迟、实时更新的场景里承担“实时数据与 AI 应用底座”的角色。

本章实战是构建一个迷你 RAG 语义缓存系统：用户提问后，先把问题转成向量，在 Redis 中做相似检索；如果找到语义相近且仍在有效期内的答案，就直接返回缓存；否则回源到大模型或后端服务，写入 JSON 文档、索引字段和向量，供下次命中。重点不是追求模型效果，而是把 Redis 新能力串成可验证的工程闭环。

## 2. 项目设计

小胖先说：“语义缓存是不是高级版 `GET`？以前 key 是问题原文，现在 key 变成问题意思。用户问‘怎么退货’和‘我要退货怎么办’，都命中同一个答案。”

小白追问：“听起来不错，但问题原文不同，怎么判断意思一样？答案会不会过期？不同租户的数据会不会串？向量检索如果命中错了，用户会看到错误答案。”

大师把方案拆成四块：“第一，JSON 存文档，让问题、答案、租户、业务线、更新时间、来源版本都结构化。第二，Query 建索引，支持按租户、状态、分类过滤。第三，Vector 存嵌入向量，用相似度召回候选。第四，语义缓存要有阈值、TTL、版本和人工兜底，不是相似就一定返回。”

技术映射：JSON 负责对象局部更新，Query Engine 负责字段过滤和排序，Vector 能力负责近邻检索，普通 String 或 Hash 仍可用于短 TTL 的命中计数和熔断状态。

小胖问：“那 Redis 会不会变成万能数据库？对象、搜索、向量都放进去。”

小白说：“我更关心边界。大规模离线向量、复杂全文搜索、强一致主数据，Redis 未必是最合适的。”

大师回应：“本章定位是低延迟实时层，不是替代所有数据库。Redis 适合承接热文档、实时更新、小到中等规模向量、严格租户过滤后的候选召回。冷数据、训练数据、审计历史仍应放在数据库、对象存储或专门搜索系统里。”

技术映射：设计时先明确数据生命周期。热数据进入 Redis，冷数据回源；索引字段尽量少而稳定；向量维度、距离算法、内存容量要提前估算。

小胖继续问：“命中错了怎么办？用户问退款，缓存给了退货答案，客服会炸。”

小白补充：“需要验收标准，比如相似度阈值、租户隔离、答案版本、人工审核状态。”

大师总结：“语义缓存不是只看向量距离。它要先过滤租户、业务线、语言、状态，再做向量检索；检索后看分数阈值和答案版本；命中后记录命中日志；低置信度必须回源。工程上宁可少命中，也不能乱命中。”

## 3. 项目实战

### 3.1 环境准备

使用带 Redis Stack 能力的镜像或已集成模块的 Redis 8.x 实验环境：

```bash
docker run --name redis-ai-lab -p 6379:6379 -d redis/redis-stack-server:latest
redis-cli module list
```

确认 JSON、Search/Query、向量能力可用。不同版本命令细节可能略有差异，生产前必须锁定镜像版本和模块版本。

准备一个文档结构：

```json
{
  "tenant": "shop-a",
  "biz": "after_sale",
  "question": "如何申请退货",
  "answer": "进入订单详情，点击申请售后，选择退货原因并提交。",
  "status": "online",
  "version": "2026-04",
  "updated_at": 1777550000,
  "embedding": [0.12, -0.03, 0.44]
}
```

### 3.2 写入 JSON 文档

```bash
JSON.SET qa:39:1 $ '{"tenant":"shop-a","biz":"after_sale","question":"如何申请退货","answer":"进入订单详情，点击申请售后，选择退货原因并提交。","status":"online","version":"2026-04","updated_at":1777550000,"embedding":[0.12,-0.03,0.44]}'
JSON.GET qa:39:1 $.question $.answer
```

JSON 的价值在于局部更新。例如只更新答案版本：

```bash
JSON.SET qa:39:1 $.version '"2026-05"'
JSON.SET qa:39:1 $.updated_at 1780000000
```

如果只是把整个对象序列化成 String，每次局部变更都要读改写整块内容，也无法直接建立字段索引。

### 3.3 建立查询与向量索引

示例索引思路如下，实际语法以所用版本为准：

```bash
FT.CREATE qa_idx ON JSON PREFIX 1 qa:39: SCHEMA $.tenant AS tenant TAG $.biz AS biz TAG $.status AS status TAG $.updated_at AS updated_at NUMERIC $.embedding AS embedding VECTOR FLAT 6 TYPE FLOAT32 DIM 3 DISTANCE_METRIC COSINE
```

写入时要保证 embedding 维度一致，类型一致。真实项目里向量一般来自嵌入模型，应用层负责把用户问题和文档问题转成同一维度的 float 数组，再写入 Redis。

先做字段过滤验证：

```bash
FT.SEARCH qa_idx '@tenant:{shop-a} @biz:{after_sale} @status:{online}'
```

再做向量检索。伪命令如下：

```bash
FT.SEARCH qa_idx '(@tenant:{shop-a} @biz:{after_sale} @status:{online})=>[KNN 3 @embedding $vec AS score]' PARAMS 2 vec <binary-vector> SORTBY score RETURN 4 question answer version score DIALECT 2
```

命中后应用层判断 score 是否低于阈值，例如 cosine distance 小于 0.18 才认为可复用。阈值需要用历史问答集评测，不能凭感觉上线。

### 3.4 实现语义缓存流程

推荐流程：
1. 接收用户问题，读取 tenant、biz、language、用户权限。
2. 调用 embedding 服务生成向量。
3. 在 Redis Query 中先过滤租户、业务线、状态，再做 KNN。
4. 若分数达标且版本未过期，返回缓存答案并记录命中。
5. 若未命中，调用大模型或知识库回源。
6. 回源结果通过审核或规则校验后写入 JSON，并更新索引。
7. 设置逻辑 TTL 或版本字段，避免旧答案长期命中。

命中计数可以继续使用普通 Redis 命令：

```bash
INCR qa:39:hit:shop-a:after_sale
EXPIRE qa:39:hit:shop-a:after_sale 86400
```

### 3.5 验证与常见坑

验证指标：
- 同一租户的相似问题应命中缓存。
- 不同租户的相似问题不能串数据。
- 下线状态文档不能被召回。
- 低相似度问题必须回源。
- 更新答案版本后，新请求不能命中过期答案。
- 大模型调用次数、平均延迟、P95 延迟应明显下降。

常见坑包括：向量维度不一致导致写入失败；把所有字段都建索引导致内存暴涨；没有租户过滤导致数据泄露；阈值过松导致错误命中；只设置物理 TTL 不记录答案版本，无法解释为什么返回旧答案。

源码与实现观察点：模块能力可从 `src/module.c` 理解扩展接入方式；核心 Redis 仍负责命令执行、内存管理、复制和持久化；Query 与 Vector 的具体实现属于模块生态，排障时要同时看 Redis 日志、模块版本和客户端请求参数。

## 4. 项目总结

Redis 8.x 新能力让 Redis 不再只是简单缓存，而可以承担实时对象存储、索引过滤、向量召回和语义缓存的组合角色。对 AI 应用来说，它的价值在于低延迟、热数据实时更新和工程链路短。

优点：
- JSON 支持结构化对象和局部更新。
- Query 支持按字段过滤、排序和聚合。
- Vector 支持语义相似召回。
- 语义缓存能降低大模型成本和尾延迟。

缺点：
- 内存成本高，索引和向量都需要容量规划。
- 检索质量依赖 embedding、阈值和数据治理。
- 模块版本、命令语法和持久化兼容要严谨验证。
- 不适合无限规模冷数据和复杂离线分析。

适用场景是智能客服热点问答、推荐解释缓存、实时知识库、商品语义检索和低延迟 RAG。思考题：语义缓存命中率提高时，如何证明没有牺牲答案正确率？当 Redis 内存不足时，应该优先淘汰低频向量、旧版本答案，还是回退到传统 key-value 缓存？
