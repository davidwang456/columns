# 第1章：Redis 术语全景与单线程工作原理

## 1. 项目背景

公司准备把一个传统电商系统改造成“缓存优先”的高并发系统。商品详情、登录验证码、购物车、排行榜、附近门店、订单事件流都计划接入 Redis。问题是团队里每个人对 Redis 的理解并不一致：有人把它当成“更快的 HashMap”，有人把它当成“不会丢数据的数据库”，还有人以为 Redis 只是一个缓存中间件。

这种认知差异会直接带来生产风险。开发同学可能把所有数据都塞进一个大 key，测试同学不知道如何设计并发用例，运维同学看到内存上涨却不知道是过期策略、复制缓冲区还是大对象导致的。更麻烦的是，一旦线上出现慢请求，大家会第一时间怀疑 Redis “单线程扛不住”，却没有真正理解 Redis 的请求处理链路。

本章的目标不是背命令，而是先建立一张 Redis 地图：Redis 由客户端、网络层、命令解析与执行层、数据结构层、过期与淘汰机制、持久化、复制、集群和模块扩展共同组成。只有先知道每个术语在系统里所处的位置，后续学习 String、Hash、Stream、Cluster、源码时才不会碎片化。

本章实战会用 Docker 启动一个 Redis，执行最基础的 `PING`、`SET`、`GET`、`INFO`，再用一张文本架构图把“一条命令从客户端到 Redis 再返回”的过程串起来。

## 2. 项目设计

小胖先开口：“Redis 不就是个会过期的 Map 吗？我 `set user:1 张三`，然后 `get user:1`，这和 Java 里的 HashMap 有啥区别？”

小白摇头：“如果只是 HashMap，那它怎么支持网络访问、持久化、主从复制、集群分片？还有，大家都说 Redis 是单线程，单线程为什么能扛那么高的 QPS？”

大师把白板分成四层：“你们可以把 Redis 想象成一家高效奶茶店。客户端是顾客，网络层是点单窗口，命令执行层是操作台，数据结构是各种原料架。Redis 快，不是因为它无限多线程，而是因为窗口排队、点单、制作流程极短，且大部分操作都在内存里完成。”

技术映射：客户端通过 TCP 连接 Redis，Redis 使用事件循环监听连接读写事件，解析 RESP 协议后查找命令表，再调用对应数据结构的实现函数。

小胖追问：“那单线程不会堵车吗？要是一个顾客点了一百杯奶茶，后面不都等着？”

小白补充：“这对应 Redis 的慢命令和大 key 问题吧？比如 `KEYS *`、一次性返回超大 List、删除巨大 Hash，都可能阻塞事件循环。”

大师点头：“没错。Redis 的主执行路径像一个很快的单窗口，一般命令只花微秒级。但如果你让它做大范围扫描、巨大对象删除、复杂脚本，它就会让后面的请求排队。所以 Redis 的性能秘诀不是‘单线程万能’，而是数据结构紧凑、内存访问快、IO 多路复用减少线程切换，再加上使用者避免阻塞命令。”

技术映射：Redis 的事件循环在 `ae.c`，网络请求处理在 `networking.c`，命令执行入口大致会进入 `processCommand` 和 `call`。

小胖又问：“那 Redis 到底是缓存还是数据库？”

大师回答：“它可以做缓存，也可以做某些实时数据存储，但不能偷换概念。缓存强调命中率、TTL 和失效策略；数据库强调事务、查询能力和数据可靠性。Redis 有 RDB、AOF、复制和集群，但它的优势仍然是内存数据结构和高性能访问。设计时要先问：这份数据丢了能不能恢复？是否需要强一致？是否需要复杂查询？”

技术映射：Redis 的定位取决于业务约束。同一个 Redis，在验证码场景是临时缓存，在 Stream 场景是轻量消息流，在排行榜场景是实时排序存储。

## 3. 项目实战

### 3.1 环境准备

本专栏默认使用 Docker 启动 Redis，便于所有章节保持一致环境。先准备一个最小命令：

```bash
docker run --name redis-lab-01 -p 6379:6379 -d redis:8.6
```

验证容器状态：

```bash
docker ps --filter name=redis-lab-01
```

进入命令行：

```bash
docker exec -it redis-lab-01 redis-cli
```

### 3.2 执行第一组命令

在 `redis-cli` 中输入：

```bash
PING
SET user:1 "xiaopang"
GET user:1
INFO server
INFO clients
INFO memory
```

预期现象：
- `PING` 返回 `PONG`，说明客户端与 Redis 服务端通信正常。
- `SET` 返回 `OK`，说明写入成功。
- `GET user:1` 返回 `xiaopang`。
- `INFO` 可以看到 Redis 版本、连接数、内存使用量等运行指标。

这组命令虽简单，却已经覆盖了 Redis 的完整请求链路：客户端发送 RESP 协议数据，服务端读入查询缓冲区，解析命令和参数，查找命令表，操作内存中的 key-value 数据结构，再把响应写回客户端。

### 3.3 画出第一张架构图

可以先用文本图表达 Redis 的核心链路：

```text
Client
  |
  | TCP / RESP
  v
Redis Server
  |
  +-- 网络事件层：连接、读取、写回
  |
  +-- 命令执行层：PING / SET / GET / INFO
  |
  +-- 数据结构层：String / Hash / List / Set / ZSet / Stream
  |
  +-- 生命周期层：Expire / Eviction
  |
  +-- 可靠性层：RDB / AOF / Replication / Cluster
  |
  +-- 扩展层：Module / JSON / Query / Vector
```

### 3.4 常见坑

第一，Redis 启动成功不代表业务连接成功。生产环境通常还会涉及密码、ACL、TLS、网络白名单和客户端连接池。

第二，`INFO` 输出很多，不要一上来全部背诵。初学阶段优先关注 `connected_clients`、`used_memory_human`、`total_commands_processed`、`instantaneous_ops_per_sec`、`keyspace_hits`、`keyspace_misses`。

第三，不要用 `KEYS *` 当日常排查命令。学习环境可以观察效果，生产环境应使用 `SCAN`。

## 4. 项目总结

本章建立了 Redis 的第一层认知：Redis 不是简单的 HashMap，也不是传统关系数据库，而是一套以内存数据结构为核心、围绕高性能访问构建的服务端系统。

优点：
- 命令模型简单，上手快。
- 数据结构丰富，能直接贴近业务建模。
- 内存访问和事件驱动模型让常规命令延迟很低。
- 支持持久化、复制、集群和模块，能覆盖多种工程场景。

缺点：
- 单线程执行路径害怕慢命令和大 key。
- 内存成本高，需要容量规划。
- 复杂查询和强事务不是 Redis 的传统优势。
- 错误的 key 设计和 TTL 策略会快速放大线上风险。

适用场景包括热点缓存、验证码、计数器、排行榜、分布式锁、轻量消息流、实时状态存储。不适合把 Redis 当成无限容量的主数据库，也不适合承载复杂关联查询。

思考题：
1. 为什么 Redis 单线程仍然可以很快？哪些命令会破坏这个优势？
2. 如果商品详情缓存丢失，业务能恢复；如果订单支付状态丢失，业务不能接受。两者在 Redis 设计上应该有什么差异？

推广建议：开发团队需要先统一术语和 key 设计意识；测试团队应从命令正确性扩展到并发和故障场景；运维团队需要从第一天开始关注内存、连接数、慢查询和持久化配置。
