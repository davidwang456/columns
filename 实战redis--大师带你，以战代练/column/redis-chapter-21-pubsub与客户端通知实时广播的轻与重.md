# 第21章：Pub/Sub 与客户端通知：实时广播的轻与重

## 1. 项目背景

商城后台有一个“全站配置中心”：运营同学会调整首页活动开关、优惠券文案、风控阈值和缓存清理策略。过去做法是每台应用每 30 秒轮询数据库，虽然简单，但配置生效慢，数据库还会被无意义查询打扰。另一个需求来自订单团队：超过 15 分钟未支付的订单需要提醒业务服务关闭订单，大家想用 Redis 的过期事件顺手做一个轻量通知。

这两个需求看起来都像“消息广播”：有人发布，很多服务接收。小系统里上 Kafka 显得重，用数据库轮询又显得笨。Redis Pub/Sub 和 Keyspace Notification 正好站在中间：命令简单、延迟低、适合实时通知。但它们也有清晰边界：消息不持久、不重放、没有消费者确认，客户端断线期间发布的消息就像喊在空气里。

本章实战会完成两个 Demo：一个是配置变更广播，多个服务订阅同一个频道；另一个是订单过期事件监听。重点不是把 Pub/Sub 当消息队列，而是学会判断“轻广播”什么时候足够，什么时候必须换 Stream、MQ 或数据库补偿。

## 2. 项目设计

小胖先说：“这不就是微信群通知吗？运营一改配置，往 Redis 频道里喊一声，所有服务都刷新缓存，多省事。”

小白追问：“微信群还有历史消息，Pub/Sub 有吗？服务重启 10 秒，刚好错过配置变更怎么办？订单过期事件如果没收到，订单是不是永远不关？”

大师点头：“Pub/Sub 像现场广播，不像快递签收。它的优点是轻、快、实时；缺点是没有投递确认、没有积压、没有重放。配置变更可以用它做通知，但真实配置必须仍然落在数据库或配置表里，服务收到消息后再主动拉取最新版本。”

技术映射：`PUBLISH` 只把消息推给当前在线订阅者；`SUBSCRIBE` 建立订阅连接；消息可靠性需要业务侧通过版本号、定时校验或 Stream 补偿。

小胖接着问：“那过期订单不是更方便？给 `order:timeout:1001` 设置 15 分钟 TTL，到期 Redis 自动通知我。”

小白摇头：“Redis 过期删除不是精确闹钟，Keyspace Notification 默认也没开。主从、重启、网络断开时，事件能不能保证？”

大师解释：“过期事件适合提醒，不适合当唯一事实来源。订单关闭必须以订单表状态为准。Redis 过期事件可以触发一次扫描或关闭尝试，没收到也要靠定时任务补扫。你们可以把它理解成门铃：响了就去看门，没响也不能假设没人来。”

技术映射：开启 `notify-keyspace-events Ex` 后，Redis 会在 key 过期时向 `__keyevent@0__:expired` 发布事件，但事件不是可靠队列。

小胖又问：“那 Pub/Sub 和 Stream 到底怎么选？”

大师回答：“如果只是告诉在线服务‘缓存失效了、配置有新版本了’，Pub/Sub 很合适。如果要求每条消息都被消费、失败可重试、消费者可扩容，就用 Stream。配置广播可以轻，订单状态变更要重；两者可以组合：Pub/Sub 负责快通知，数据库或 Stream 负责可追溯。”

## 3. 项目实战

### 3.1 环境准备

启动一个实验 Redis：

```bash
docker run --name redis-lab-21 -p 6379:6379 -d redis:8.6
docker exec -it redis-lab-21 redis-cli
```

如果要监听过期事件，需要开启通知。学习环境可以直接执行：

```bash
CONFIG SET notify-keyspace-events Ex
CONFIG GET notify-keyspace-events
```

生产环境建议写入配置文件，避免重启丢失：

```conf
notify-keyspace-events Ex
```

### 3.2 配置变更广播

业务流程如下：

```text
运营后台修改配置
  -> 写入数据库并生成 version
  -> 更新 Redis 中的配置缓存
  -> PUBLISH config:changed {"key":"coupon.switch","version":12}
  -> 各应用收到消息后拉取最新配置
```

打开第一个终端作为应用 A：

```bash
SUBSCRIBE config:changed
```

打开第二个终端作为应用 B：

```bash
SUBSCRIBE config:changed
```

第三个终端模拟运营发布：

```bash
SET config:coupon.switch "{\"enabled\":true,\"version\":12}"
PUBLISH config:changed "{\"key\":\"coupon.switch\",\"version\":12}"
GET config:coupon.switch
```

订阅端会看到消息。真实代码里不要只相信消息体，要用 `key` 和 `version` 再查一次权威配置，避免旧消息覆盖新配置。

### 3.3 过期订单提醒

先启动订阅：

```bash
PSUBSCRIBE __keyevent@0__:expired
```

再设置一个 10 秒后过期的订单提醒 key：

```bash
SET order:timeout:1001 pending EX 10
TTL order:timeout:1001
```

订阅端预期收到：

```text
pmessage
__keyevent@0__:expired
__keyevent@0__:expired
order:timeout:1001
```

业务服务收到 `order:timeout:1001` 后，应解析订单号 `1001`，再访问订单数据库：

```text
查询订单状态
  -> 仍为待支付：执行关闭订单，释放库存
  -> 已支付或已关闭：忽略事件
  -> 查询失败：写入补偿任务，稍后重试
```

### 3.4 常见坑

第一，Pub/Sub 订阅连接进入订阅模式后，不能像普通连接一样随意执行业务命令。客户端通常要为订阅单独准备连接。

第二，`PUBLISH` 返回的是当前收到消息的订阅者数量，不是业务处理成功数量。返回 3 只能说明 3 个订阅连接在线。

第三，Keyspace Notification 默认关闭，且过期事件不是精确计时器。Redis 的删除策略会影响事件触发时机，不能用它做严格定时调度。

第四，广播消息要带版本号。服务收到版本 `12` 时，如果本地已经是 `13`，应该忽略旧消息。

第五，集群环境下要特别注意订阅连接接入哪个节点。普通 Pub/Sub 会在集群节点间传播，但客户端路由、网络隔离和代理行为都要实测。

## 4. 项目总结

Pub/Sub 的价值是轻量实时广播：配置刷新、缓存失效、在线状态通知、简单聊天室都可以用它快速落地。它的短板同样明显：消息不落盘、订阅者离线就丢、没有 ACK、没有消费进度，也不适合承载核心交易链路。

本章推荐的工程原则是“通知与事实分离”。Pub/Sub 只负责告诉应用“可能有变化”，事实仍然放在数据库、配置中心或 Redis 中可查询的 key 里。过期事件也只做触发器，订单关闭必须二次校验数据库状态，并配合定时补扫。

适用场景：配置变更广播、缓存失效通知、在线调试频道、低成本实时提示。不适用场景：支付状态流转、账务消息、要求重试和审计的业务事件。

思考题：
1. 如果配置变更消息丢失，如何用版本号和定时校验保证最终一致？
2. 如果订单过期事件没有收到，你会设计什么补偿任务，扫描范围和频率如何控制？
