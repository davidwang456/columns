# 第40章：【高级篇综合实战】从零设计企业级 Redis 平台

## 1. 项目背景

集团内部有几十个业务线都在使用 Redis：交易团队用它做库存和订单缓存，会员团队用它做登录态，内容团队用它做排行榜和推荐特征，客服团队开始尝试语义缓存。起初每个团队各自申请机器、自己部署 Redis、自己写监控，短期很快，长期却出现了大量平台级问题。

第一，规格混乱。有人把 2GB 缓存跑到 90% 内存，有人申请 64GB 只存几万个 key。第二，运维标准不统一。不同团队的持久化、备份、密码、ACL、慢查询阈值、告警规则都不一样。第三，故障演练缺失。主从切换、Cluster failover、AOF rewrite、热点 key、过期风暴都没有标准预案。第四，新能力接入无章法。JSON、Query、Vector、Module 有人想用，但安全、容量、版本和支持边界都没定义。

本章的综合实战目标是从零设计一套企业级 Redis 平台方案。它不是简单画一张架构图，而是要覆盖需求拆解、架构设计、平台能力、SRE 落地和验收标准，最终能交付给开发、测试、运维、安全和架构委员会评审。

## 2. 项目设计

小胖先说：“我们是不是做个网页，让大家点一下就创建 Redis？像点外卖一样，选小杯中杯大杯，付款后自动送到。”

小白提醒：“如果只是创建实例，那叫资源入口，不叫平台。平台要回答：规格怎么评估？谁能访问？慢查询谁治理？备份能不能恢复？故障时谁值班？跨业务线能不能隔离？新模块谁审核？”

大师把需求拆成五类：“第一是资源生命周期：申请、审批、创建、扩容、缩容、下线。第二是运行治理：监控、告警、慢查询、大 key、热 key、无 TTL key。第三是高可用与恢复：Sentinel、Cluster、备份、演练和回滚。第四是安全合规：ACL、TLS、审计、租户隔离。第五是能力扩展：JSON、Query、Vector、Module 和语义缓存支持。”

技术映射：企业级平台不是 Redis 单点能力，而是 Redis 内核、部署编排、客户端规范、观测系统、SRE 流程和组织协作的组合。

小胖问：“架构上到底用 Sentinel 还是 Cluster？要不要代理层？”

小白说：“小实例用 Sentinel 可能更简单，大容量和高吞吐要 Cluster。代理层能统一入口，但也可能增加延迟和故障点。”

大师回答：“平台要提供分层规格。标准缓存、小容量、强依赖主从切换的场景可以用 Sentinel 版；大容量、可按 key 分片的场景用 Cluster 版；跨语言客户端治理困难的团队可以接代理层，但代理层必须纳入 SLO。不要用一种架构覆盖所有需求。”

技术映射：Sentinel 版关注主从复制和故障转移；Cluster 版关注槽位、reshard、failover；代理层关注路由、连接复用、限流和可观测性。

小胖继续说：“SRE 是不是就是写几个告警？”

小白摇头：“SRE 要有 SLO、错误预算、值班手册、故障演练、复盘和持续改进。只有告警没有动作，最后会变成噪音。”

大师总结：“平台设计的验收标准必须可验证。比如实例创建 10 分钟内完成，P99 延迟小于 5 毫秒，主库故障 30 秒内完成切换，备份恢复演练每月至少一次，大 key 巡检每天出报告。这些指标比口号更重要。”

## 3. 项目实战

### 3.1 需求拆解

先把平台目标写成可评审需求：

1. 多租户：按业务线、应用、环境隔离实例、账号和权限。
2. 多规格：提供单机开发版、Sentinel 高可用版、Cluster 分片版、Redis Stack 扩展版。
3. 自助申请：用户提交容量、QPS、命令类型、可用性等级、数据重要性和模块需求。
4. 容量评估：根据 key 数、value 大小、TTL、写入峰值、复制副本和索引开销计算规格。
5. 可观测：统一采集连接数、QPS、延迟、内存、碎片率、命中率、慢查询、复制延迟、槽位状态。
6. 治理：自动发现大 key、热 key、无 TTL key、危险命令、过期风暴风险。
7. 安全：ACL、TLS、密码轮换、审计日志、网络白名单。
8. 恢复：RDB/AOF 策略、备份保留、恢复演练、跨可用区部署。
9. 扩展：Module、JSON、Query、Vector、语义缓存通过白名单和容量模型接入。

### 3.2 架构设计

平台建议分为六层：

```text
用户入口层：控制台 / OpenAPI / 工单
编排层：规格评估 / 审批 / 实例创建 / 扩缩容 / 下线
接入层：客户端规范 / 可选代理 / ACL / TLS / 白名单
Redis 运行层：Standalone / Sentinel / Cluster / Stack
观测治理层：Exporter / 日志 / 慢查询 / 大 key / 热 key / 告警
SRE 流程层：SLO / 值班 / 演练 / 复盘 / 容量评审
```

Sentinel 版适合中小容量、高可用缓存。标准配置是一主两从三哨兵，开启 AOF 或 RDB 取决于数据可恢复性，客户端必须通过 Sentinel 获取主库地址。

Cluster 版适合大容量和高吞吐。标准配置至少三主三从，槽位均衡，客户端启用 cluster mode。扩容和缩容必须走平台编排，迁移前检查客户端是否支持 `MOVED` 和 `ASK`。

Redis Stack 扩展版用于 JSON、Query、Vector 和语义缓存。它必须单独做容量模型：JSON 文档大小、索引字段数量、向量维度、候选集规模、更新频率都会影响内存和延迟。

### 3.3 平台能力

实例申请表单不只问“要几 GB”，还要问：
- 业务等级：核心链路、重要链路、普通链路。
- 数据类型：缓存、会话、排行榜、消息流、搜索索引、语义缓存。
- 数据可恢复性：可回源、可补偿、不可丢。
- 峰值 QPS、读写比例、value 大小、key 数、TTL 分布。
- 是否需要 Cluster、TLS、ACL、模块能力。

自动化治理任务包括：

```bash
redis-cli --bigkeys
redis-cli --hotkeys
redis-cli slowlog get 128
redis-cli info memory
redis-cli info replication
redis-cli cluster info
```

生产平台不能直接依赖人工运行这些命令，而是要封装成巡检任务，输出责任人、风险等级、建议动作和治理截止时间。例如发现 50MB 大 key，要给出拆分建议；发现大量无 TTL 缓存 key，要通知业务补生命周期；发现复制延迟持续升高，要检查网络、从库负载和 backlog。

### 3.4 SRE 落地

先定义 SLO：
- 可用性：核心实例月可用性不低于 99.95%。
- 延迟：核心缓存 P99 小于 5 毫秒，普通缓存 P99 小于 10 毫秒。
- 恢复：Sentinel 主库故障 30 秒内恢复写入，Cluster 单主故障 60 秒内恢复槽位可用。
- 数据：可恢复缓存允许短暂丢失，不可丢数据必须有数据库、消息或日志兜底。

再设计错误预算。若某业务线一个月 Redis 相关故障耗尽预算，则冻结非必要变更，优先治理大 key、慢命令、容量和客户端连接池。

值班手册至少包含：
1. 连接数打满：定位客户端、临时扩连接、回收空闲连接、推动连接池修复。
2. 内存逼近上限：识别增长来源、扩容或清理、检查淘汰策略。
3. 慢查询升高：查看 `SLOWLOG`，拦截危险命令，拆分大对象。
4. 主从延迟：检查写入峰值、网络、从库 CPU、backlog。
5. Cluster 槽位异常：查看 `cluster info`，暂停变更，执行修复或回滚。

故障演练每月至少覆盖：主库宕机、从库重建、Cluster failover、AOF rewrite 压力、备份恢复、热点 key、过期风暴、语义缓存错误命中回滚。

### 3.5 验收标准

平台上线前必须通过以下验收：

功能验收：
- 能创建 Sentinel、Cluster、Stack 三类实例。
- 能完成扩容、缩容、备份、恢复和下线。
- 能自动生成 ACL 账号、连接串和客户端示例。

可靠性验收：
- 主库宕机演练达到恢复时间目标。
- Cluster reshard 期间业务无明显错误峰值。
- 备份文件可在隔离环境恢复并校验关键 key。

治理验收：
- 大 key、热 key、慢查询、无 TTL key 每日出报表。
- 高危命令有禁用或审批机制。
- 告警能关联实例、业务线、责任人和处理手册。

安全验收：
- 生产实例默认开启认证，敏感业务启用 TLS。
- ACL 最小权限，账号可轮换。
- 管理操作有审计日志。

业务验收：
- 接入方完成压测报告。
- 客户端支持超时、重试、连接池、熔断和降级。
- 关键写路径有 Redis 不可用时的业务预案。

## 4. 项目总结

企业级 Redis 平台的核心不是“把 Redis 部署得更多”，而是把资源、架构、治理、安全、可靠性和组织流程统一起来。前 39 章学到的单线程、数据结构、缓存设计、复制、Cluster、模块和语义缓存，最终都要在平台里落成规范、工具和验收标准。

优点：
- 降低各业务线重复建设成本。
- 统一高可用、安全、监控和备份标准。
- 通过巡检和 SRE 流程提前发现风险。
- 为 JSON、Query、Vector 等新能力提供受控入口。

缺点：
- 平台建设需要持续投入，不是一次性交付。
- 标准化可能牺牲部分团队的自由度。
- 自动化编排和代理层本身也会成为新的运维对象。
- 容量模型和 SLO 需要随着业务变化持续校准。

推广建议：先选择两三个典型业务试点，一个 Sentinel 缓存场景、一个 Cluster 大容量场景、一个语义缓存场景；试点稳定后再推广到集团。开发团队负责接入规范和降级预案，测试团队负责压测与故障注入，运维/SRE 负责平台运行与演练，架构团队负责规格、模块和新能力准入。最终验收不是控制台有多漂亮，而是当 Redis 出问题时，平台能让团队更快定位、更稳恢复、更少复发。
