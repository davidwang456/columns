# 第8章 定时器与节奏控制：吞吐、并发、Think Time的区别

> 级别：基础篇  
> 目标读者：开发 / 测试 / 运维  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
“并发 1000 就是高压”是压测中的常见误解。真实系统承压不仅取决于并发线程，还取决于请求节奏、用户停顿、到达模式和上游限流策略。某金融业务在预发环境做压测时，采用“0 think time + 瞬时拉满线程”方式，结果 QPS 远高于线上峰值，误判系统可承载能力。上线后真实用户有明显停顿与页面跳转，导致请求节奏与压测模型完全不同，最终在促销高峰时触发网关限流抖动。

定时器的作用，是把“请求频率”从线程模型里解耦出来。你需要清楚：
- 并发（多少用户同时在线）
- 吞吐（单位时间处理多少请求）
- Think Time（用户操作间隔）
三者不是同一个概念。若混淆，会导致压力模型偏离现实。

本章将通过 Constant Timer、Uniform Random Timer、Constant Throughput Timer 等组件，构建更接近真实行为的节奏模型。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：
“我通常把线程数调大，QPS 不够就再加线程。这样不是最直接吗？”

小白：
“这种方式常常把客户端打满，而不是服务端。你看到的瓶颈可能来自压测机，不是系统。”

大师：
“正确。线程数决定潜在并发，定时器决定请求节奏。两者必须一起设计。没有节奏控制，就像让所有用户同时点击同一个按钮，现实中几乎不会发生。”

技术映射：
- 演唱会检票全体同时冲门 -> 无节奏高峰
- 分批入场 -> 可控到达率

小胖：
“Think Time 真的重要吗？我担心加了会把压力降太低。”

小白：
“不加会高估写放大和连接竞争，不像真实用户。”

大师：
“Think Time 不是降压，而是还原真实交互。你可以设置多个场景：
- 峰值压测：较小 think time；
- 真实回放：按用户行为分布；
- 稳定性：中等节奏长时间运行。”

小胖：
“Constant Throughput Timer 是不是等于精准控 QPS？”

小白：
“它是目标值，不保证绝对精确，还受响应耗时和线程能力影响。”

大师：
“对。控吞吐是闭环行为：设置目标 + 观测实际 + 调整线程和定时器参数。”

### 2.2 关键设计决策
- 方案A：只调线程，不设定时器。
- 方案B：线程、吞吐、think time 联合建模，分场景调参。
- 选型结论与理由：采用方案B，模型更接近真实用户行为。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK 17
- JMeter 5.6.x
- 目标接口：浏览、详情、下单

### 3.2 分步实现
1. 步骤目标：建立无定时器基线（对照组）。  
   - 命令：
```bash
jmeter -n -t plans/ch08-no-timer.jmx -Jthreads=500 -Jrampup=30 -Jduration=600 -l reports/ch08-no-timer.jtl
```
   - 输出示例：
```text
throughput=3200/s p95=980ms err=3.2%
```
   - 结论：高吞吐但抖动明显，难代表真实流量。

2. 步骤目标：引入 Constant Timer（固定停顿）。  
   - 配置：每次请求前 300ms 停顿。
   - 命令：
```bash
jmeter -n -t plans/ch08-constant-timer.jmx -Jthreads=500 -Jduration=600 -l reports/ch08-constant.jtl
```
   - 输出示例：
```text
throughput=2100/s p95=520ms err=0.8%
```
   - 常见坑与解决：
     - 坑：把定时器挂在错误层级导致仅部分请求生效。  
       解：用 Test Plan 树检查作用域。

3. 步骤目标：引入随机停顿模拟真实用户。  
   - 组件：Uniform Random Timer（基础 200ms，增量 400ms）。
   - 结果：延迟分布更平滑，峰值更接近线上行为。

4. 步骤目标：使用 Constant Throughput Timer 控制目标吞吐。  
   - 配置：目标 1800 req/min（按线程组共享）。
   - 命令：
```bash
jmeter -n -t plans/ch08-ctt.jmx -Jthreads=300 -Jrampup=60 -Jduration=1200 -l reports/ch08-ctt.jtl -e -o reports/ch08-ctt-html
```
   - 输出示例：
```text
target=1800/min actual=1768/min deviation=-1.8%
```
   - 常见坑与解决：
     - 坑：线程太少导致目标吞吐达不到。  
       解：先估算单线程能力，再反推线程池规模。

5. 步骤目标：构建三种节奏场景模板。  
   - 模板A：峰值冲刺（低 think time，高并发，短时）；
   - 模板B：真实回放（随机 think time，中并发，中时长）；
   - 模板C：稳定耐力（中等吞吐，长时）。

### 3.3 测试验证
- 验证项1：同接口在三种节奏模板下的 P95/P99 差异。
- 验证项2：实际吞吐偏差是否在 ±5% 内。
- 验证项3：压测机资源是否成为瓶颈（CPU/网络）。

## 5. 项目总结（约500-800字）
- 优点：
  - 压力模型更贴近用户行为；
  - 降低“客户端瓶颈误判服务端”的风险；
  - 支持不同测试目标的节奏模板化。
- 缺点：
  - 参数调优需要多轮试验；
  - 初学者容易误配定时器作用域；
  - 吞吐控制并非绝对精确。
- 适用场景：活动峰值评估、稳定性测试、回归门禁。
- 不适用场景：纯连通性检查。
- 注意事项：
  - 先确定目标指标再调节奏；
  - 定时器层级要清晰；
  - 客户端资源监控必须同步采集。
- 真实踩坑案例（3个）：
  1. 无 think time 导致写冲突虚高；
  2. 误把并发当吞吐，结论错误；
  3. 目标吞吐达不到却未识别线程不足。
- 思考题（2道）：
  1. 如何把线上访问日志转换为可执行的节奏分布？
  2. 当吞吐稳定但 P99 恶化时，你会优先排查哪些系统层面因素？
- 跨部门协作细则：
  - 产品：提供用户操作停留分布；
  - 开发：提供限流策略与降级逻辑；
  - 测试：维护节奏模板与参数基线；
  - 运维：验证网关和主机资源保护策略。


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
