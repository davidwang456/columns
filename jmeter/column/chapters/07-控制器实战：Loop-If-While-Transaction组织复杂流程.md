# 第7章 控制器实战：Loop/If/While/Transaction组织复杂流程

> 级别：基础篇  
> 目标读者：开发 / 测试 / 运维  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
在真实业务压测中，接口几乎不会孤立执行。用户行为通常是链路式的：登录 -> 浏览 -> 加购 -> 下单 -> 支付查询。很多团队把这些步骤简单线性拼接，忽略分支条件与重试路径，导致脚本“看起来完整”，实际却没有覆盖关键异常分支。某出行平台在版本上线前压测“下单链路”时，只覆盖了成功支付路径，线上却在“余额不足->换支付方式->重试”分支出现严重慢响应，直接触发投诉。

控制器的意义，就是把“业务流程图”准确映射成“压测执行图”。Loop 控制重复行为，If/While 负责分支和循环条件，Transaction 用于汇总业务事务耗时。若控制器设计错误，压测结论会完全失真：要么高估容量，要么漏测风险。

本章目标是让你掌握流程编排的工程方法：先建业务状态机，再映射到 JMeter 控制器，最后用事务统计验证链路级 SLA。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：
“我一般把请求按顺序排好就行，控制器感觉是可选项。脚本能跑通最重要。”

小白：
“能跑通不等于覆盖真实行为。比如用户浏览商品会循环，支付失败会重试，库存不足会走降级分支。你不建分支，结果只会‘好看但没用’。”

大师：
“流程建模是压测可信度的核心。你需要从业务视角定义状态转换，再从脚本视角选择控制器：
- Loop：固定次数重复；
- While：条件驱动循环；
- If：分支决策；
- Transaction：按业务事务统计端到端耗时。”

技术映射：
- 地铁换乘图 -> 业务状态机
- 自动闸机规则 -> 控制器执行条件

小胖：
“Transaction Controller 是不是只是显示好看？”

小白：
“不只是展示。它能帮助你看链路耗时，而不是只看单接口。”

大师：
“补充：事务统计能直接映射业务 SLA，例如‘下单全流程 P95 <= 800ms’。这比看单点接口更接近用户体验。”

小胖：
“While 我总怕死循环。”

小白：
“所以要加退出条件和最大重试次数。”

大师：
“工程上必须有护栏：超时退出、计数上限、异常打标。否则脚本会把压测机拖死，甚至污染测试结论。”

### 2.2 关键设计决策
- 方案A：线性脚本，不建分支，不计事务。
- 方案B：按业务状态机设计控制器，并建立重试护栏和事务统计。
- 选型结论与理由：采用方案B，覆盖真实路径并保证可观测。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK 17
- JMeter 5.6.x
- 业务接口：登录、列表、加购、下单、支付查询

### 3.2 分步实现
1. 步骤目标：构建基础链路 + 事务统计。  
   - 结构：
     - Transaction Controller: `order_flow`
       - login
       - query_item
       - add_cart
       - create_order
   - 命令：
```bash
jmeter -n -t plans/ch07-transaction-basic.jmx -Jthreads=300 -Jrampup=60 -Jduration=900 -l reports/ch07-tx.jtl -e -o reports/ch07-tx-html
```
   - 输出示例：
```text
order_flow   p95=720ms  err=0.35%
create_order p95=280ms  err=0.12%
```
   - 常见坑与解决：
     - 坑：Transaction 默认设置不当导致子采样统计混淆。  
       解：明确是否“Generate parent sample”。

2. 步骤目标：引入 If 分支（库存不足降级）。  
   - 条件：`${stock_ok}`
   - 分支A：正常下单
   - 分支B：替代 SKU 重试
   - 命令：
```bash
jmeter -n -t plans/ch07-if-branch.jmx -Jthreads=350 -Jduration=900 -l reports/ch07-if.jtl
```
   - 输出示例：
```text
branch_normal count=210000 err=0.20%
branch_fallback count=35000 err=0.95%
```
   - 常见坑与解决：
     - 坑：If 条件字符串比较写错，导致分支全部命中。  
       解：用 Debug Sampler 验证条件表达式。

3. 步骤目标：引入 While 循环（支付轮询）并设置护栏。  
   - 条件：`${pay_status}` != SUCCESS && `${retry_count}` < 5
   - 命令：
```bash
jmeter -n -t plans/ch07-while-polling.jmx -Jthreads=200 -Jduration=1200 -l reports/ch07-while.jtl
```
   - 输出示例：
```text
pay_polling avg_loop=2.3 max_loop=5 timeout_ratio=1.4%
```
   - 常见坑与解决：
     - 坑：忘记更新 `retry_count` 导致死循环。  
       解：每次循环显式 `vars.put("retry_count", ...)`。

4. 步骤目标：Loop 模拟“浏览-点击”行为。  
   - 配置：Loop Controller 循环 5 次浏览后进入下单。
   - 结果：更接近真实用户“先看再买”行为，缓存命中结构更真实。

### 3.3 测试验证
- 验证项1：正常链路、降级链路、超时链路占比可解释。
- 验证项2：事务级 P95 与单接口 P95 对比，确认链路瓶颈位置。
- 验证项3：异常分支触发时脚本不死循环。

## 5. 项目总结（约500-800字）
- 优点：
  - 将业务流程真实映射到压测模型；
  - 能覆盖关键分支与异常路径；
  - Transaction 提供链路级 SLA 视角。
- 缺点：
  - 脚本复杂度显著提升；
  - 分支条件维护成本高；
  - 对调试和日志要求更高。
- 适用场景：电商交易、支付流程、审批流、预约流程。
- 不适用场景：单接口探活、极简健康检查。
- 注意事项：
  - While 必须设退出护栏；
  - If 条件先调试后施压；
  - Transaction 统计口径要提前约定。
- 真实踩坑案例（3个）：
  1. 分支条件失效导致全量走主路径，漏测降级风险；
  2. 轮询死循环拖垮压测机；
  3. 只看单接口不看事务，错判用户体验。
- 思考题（2道）：
  1. 如何把“业务状态机”自动转换成可复用的控制器模板？
  2. 当事务耗时劣化但单接口稳定时，最可能的系统问题在哪里？
- 跨部门协作细则：
  - 产品/业务：提供关键用户路径与异常分支；
  - 开发：定义状态字段与重试上限；
  - 测试：维护流程图与脚本一致性；
  - 运维：提供链路观测与告警联动。


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
