# 第3章 Test Plan骨架：线程组、Sampler、Listener的最小闭环

> 级别：基础篇  
> 目标读者：开发 / 测试  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
很多初学者会把 JMeter 当作“组件拼装器”，在树上不断添加节点，最终脚本看起来很热闹，但没人能解释执行顺序和压测结论。某团队脚本中加入十几个监听器和多个线程组，结果是压测机先被拖慢，报告波动巨大，甚至同参数重复执行得到完全不同趋势。

这背后反映的是心智模型缺失：不清楚线程组控制压力、不清楚 Sampler 代表业务动作、不清楚 Listener 只是观测工具而非越多越好。要解决这个问题，最有效的方法是先掌握“最小闭环”。

最小闭环只有三件事：Thread Group（怎么压）、Sampler（压什么）、Listener（怎么看）。一旦闭环建立，后续再扩展控制器、定时器、提取器，你会非常清楚每个新增组件的作用与代价。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：“Sampler 才是核心吧？Thread Group 不就是填个并发数，Listener 多加几个图才直观。”

小白：“并发数只是线程组的一部分，ramp-up、循环次数、持续时间都会影响压力形态。监听器太多还会干扰结果。”

大师：“Thread Group 决定流量模型，Sampler 决定业务语义，Listener 决定观测视角。三者是系统，不是单点。”

技术映射：
- 发车计划 -> 线程调度
- 乘客上车动作 -> Sampler 请求
- 监控摄像头 -> Listener 观察

小胖：“同级两个 Sampler 是并行跑吗？”

小白：“同一线程内是顺序执行，不同线程之间并发执行。”

大师：“非常关键。线程模型理解错，所有吞吐判断都会错。”

小胖：“那为什么我把线程数翻倍，吞吐不一定翻倍？”

小白：“可能受服务端瓶颈、网络、连接池限制，也可能是你加了定时器。”

大师：“吞吐是系统共同结果，不是线程参数单变量函数。压测要看全局：客户端发压能力 + 服务端处理能力 + 依赖链路能力。”

### 2.2 关键设计决策
- 方案A：先堆功能组件，再回头修结构。
- 方案B：先做最小闭环并验证，再逐步扩展。
- 结论：采用方案B，保证每一步新增都可解释、可回滚。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK 17
- JMeter 5.6.x
- 一个简单 HTTP 接口（/health、/profile）

### 3.2 分步实现
1. 创建最小闭环脚本：
   - Thread Group：20线程，ramp-up 10 秒，loops 5
   - Sampler：GET /health
   - Listener：Summary Report

2. 验证执行顺序：
   - 新增 Sampler A（/login）与 B（/profile）
   - 观察单线程顺序 A->B
   - 多线程并发下呈现多组 A->B

3. 对比监听器开销：
   - 场景1：仅 Summary Report
   - 场景2：加 View Results Tree
   - 对比压测机 CPU、内存与吞吐差异

4. CLI 执行规范：
```bash
jmeter -n -t plans/mini-loop.jmx -l reports/mini-loop.jtl -e -o reports/mini-loop-html
```

5. 结果归档：
   - 保存 jmx、jtl、html 报告；
   - 记录线程参数与执行时间窗口。

### 3.3 测试验证
- 改变线程数 20/50/100，观察响应曲线变化；
- 重复执行三次，验证结果稳定性；
- 去掉重监听器后比较差异。

## 5. 项目总结（约500-800字）
- 优点：
  - 结构清晰，便于排错和培训；
  - 结果解释一致性提升；
  - 为复杂场景扩展打底。
- 缺点：
  - 对复杂链路表达力有限；
  - 仍需配合可观测平台定位根因。
- 注意事项：线程模型优先、监听器轻量化、结果可追溯。
- 思考题：
  1. 如何在最小闭环上平滑演进到“登录-查询-下单”链路？
  2. 当吞吐上不去时，你会优先排查客户端还是服务端？


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
