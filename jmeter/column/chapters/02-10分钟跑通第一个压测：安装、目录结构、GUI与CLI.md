# 第2章 10分钟跑通第一个压测：安装、目录结构、GUI与CLI

> 级别：基础篇  
> 目标读者：开发 / 测试  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
新成员接手性能测试时，最常见的问题不是“不会写复杂脚本”，而是“第一步就被环境绊住”：JDK 版本不匹配、插件冲突、路径含中文导致脚本报错、GUI 可以跑 CLI 却失败。某团队新人一周内提交了 4 份压测报告，结论彼此冲突，最后发现是每个人本地 JMeter/JDK 版本不同，导致协议行为与默认参数不一致。

因此，建立统一、可复现的“10 分钟首跑流程”非常关键。它的目标不是做高强度压力，而是保证每个人都能在同样目录、同样命令、同样输出格式下跑出第一份结果。只有这个基础统一了，后续脚本复用、结果对比、流水线集成才有意义。

在企业里，最难的往往不是技术，而是协作。一份脚本能在你电脑上跑，不能在团队环境跑，等于没有价值。今天我们把“第一次跑通”设计成一个可传播、可训练、可审计的流程：新人拿到仓库后，不需要额外口头指导，照着固定步骤就能得到一致报告。这个能力，是后续构建压测平台和自动化门禁的起点。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：
“我本地点开 JMeter 能跑就行了吧？搞那么多目录规范是不是形式主义？”

小白：
“你说的是‘个人可用’，不是‘团队可用’。你本地能跑，不代表 CI 节点能跑；你自己看得懂，不代表下一位同事看得懂。”

大师：
“性能测试属于工程活动，不是个人秀。第一阶段的目标不是把压测做深，而是把流程做稳。环境一致、输入一致、输出一致，这是最低标准。”

技术映射：
- 家里炒菜凭感觉 -> 个人脚本，可跑不可复现
- 连锁餐厅标准出品 -> 团队脚本，可复核可交接

小胖：
“那 GUI 和 CLI 怎么分？网上有人说 GUI 千万别用。”

小白：
“那是简化说法。GUI 对建模调试很有帮助，CLI 对正式执行更稳定。”

大师：
“准确讲，GUI 是设计器，CLI 是执行器。设计阶段用 GUI 快速验证请求结构，执行阶段必须 CLI 保障资源可控、结果可追溯。两者不是替代关系，是分工关系。”

技术映射：
- 白板画流程 -> GUI 快速试错
- 自动化产线 -> CLI 稳定执行

小胖：
“目录规范有必要这么死吗？我觉得随便放也能找到。”

小白：
“短期你能找到，长期团队找不到。尤其 CSV、证书、结果文件用相对路径时，目录一乱就全崩。”

大师：
“目录规范本质是依赖管理。脚本依赖什么、结果输出到哪里、如何归档，必须可预测。否则根本无法做自动回归和历史对比。”

技术映射：
- 仓库货架编码 -> 目录规范
- 出入库记录 -> 结果归档与版本追踪

### 2.2 关键设计决策
- 方案A：每人本地自由安装、自由目录、自由命令。优点是快，缺点是不可复现。
- 方案B：统一 JDK/JMeter 版本、统一工程目录、统一 CLI 执行入口。优点是稳定、可审计。
- 选型结论与理由：采用方案B。虽然初始约束更强，但能显著降低后续协作成本。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK版本：17（与团队标准保持一致）
- JMeter版本：5.6.x
- 依赖与工具：Git、PowerShell/Bash、一个可访问的 Demo API

推荐目录结构：
- perf-tests/
  - plans/
  - data/
  - reports/
  - scripts/

### 3.2 分步实现
1. 步骤目标：完成环境一致性检查。  
   - 代码/脚本片段：
```bash
java -version
jmeter -v
```
   - 运行结果：输出符合团队版本要求。  
   - 常见坑与解决：
     - 坑：系统装了多个 JDK，命令行调用到旧版本。  
       解：固定 `JAVA_HOME`，并在脚本启动时打印版本。

2. 步骤目标：用 GUI 创建最小可执行 Test Plan。  
   - 操作说明：
     - 新建 Thread Group（1线程，1次循环）；
     - 新建 HTTP Request（GET 一个健康检查接口）；
     - 新建 View Results Tree（仅调试阶段使用）；
     - 保存为 `plans/hello.jmx`。
   - 运行结果：GUI 可看到响应成功。  
   - 常见坑与解决：
     - 坑：脚本没保存直接执行，修改丢失。  
       解：养成“先保存再运行”习惯。

3. 步骤目标：CLI 标准执行并产出报告。  
   - 代码/脚本片段：
```bash
jmeter -n -t plans/hello.jmx -l reports/hello.jtl -e -o reports/hello-html
```
   - 运行结果：得到 `hello.jtl` 与 HTML 报告。  
   - 常见坑与解决：
     - 坑：输出目录已存在导致命令失败。  
       解：执行前清理目录或使用新目录。

4. 步骤目标：把并发参数改为命令行可控。  
   - 代码/脚本片段：
```bash
jmeter -n -t plans/hello.jmx -Jthreads=20 -Jrampup=10 -Jloops=5 -l reports/hello-param.jtl
```
   - 运行结果：同一脚本可在不同强度下复用。  
   - 常见坑与解决：
     - 坑：脚本内部未用 `${__P(threads,1)}` 读取参数。  
       解：线程组参数全部改成属性驱动。

5. 步骤目标：封装统一启动脚本。  
   - 代码/脚本片段：
```powershell
$threads=50
$rampup=20
$loops=10
jmeter -n -t plans/hello.jmx -Jthreads=$threads -Jrampup=$rampup -Jloops=$loops -l reports/hello.jtl
```
   - 运行结果：新人只改变量即可执行。  
   - 常见坑与解决：
     - 坑：参数散落在文档，容易漏改。  
       解：统一收敛到启动脚本。

### 3.3 测试验证
- 验证1：同一脚本在两台机器执行，确认都能产出报告。
- 验证2：调整 `threads`，观察吞吐和响应变化趋势。
- 验证3：故意改错路径，确认错误提示可读并可快速定位。

期望结果：
- 任何新成员在 10 分钟内可完成首跑；
- 输出产物结构一致；
- 参数化执行可复用。

## 5. 项目总结（约500-800字）
- 优点：
  - 低门槛建立团队统一执行标准；
  - GUI/CLI 分工明确，兼顾效率与稳定；
  - 为后续自动化回归打下基础。
- 缺点：
  - 规范初期会增加约束感；
  - 仅完成首跑并不能得出容量结论；
  - 需要团队共同维护版本与脚本模板。
- 适用场景：新人培训、环境验收、脚本模板初始化。
- 不适用场景：直接用于复杂链路容量评估。
- 注意事项：正式执行只用 CLI；保留 jtl 原始数据；目录结构不可随意漂移。
- 真实踩坑案例（3个）：
  1. 版本不一致导致 TLS 行为差异；
  2. 路径含空格导致批处理脚本失败；
  3. 只发截图不保留原始数据，无法复盘。
- 思考题（2道）：
  1. 你会如何设计一个跨平台（Windows/Linux）通用启动脚本？
  2. 如果新人经常把调试监听器带入正式压测，如何从流程上预防？
- 推广计划：
  - 开发提供可稳定调用的演示接口；
  - 测试维护模板脚本与运行说明；
  - 运维提供统一压测节点镜像与执行权限。


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
