# 第4章 HTTP接口压测入门：请求方法、参数化、断言

> 级别：基础篇  
> 目标读者：开发 / 测试  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
HTTP 压测最常见误区是“接口 200 就算成功”。真实项目里，200 可能只代表网关受理，不代表业务成功。某团队对支付下单接口做压测，报告显示成功率 99.8%，上线后却大量卡单。复盘发现响应中的业务字段 `tradeStatus` 大量为 `PENDING`，只是传输层成功、业务层未完成。

另一个高频误区是“参数写死”。写死参数导致缓存命中异常高、幂等接口重复命中，结果明显偏离真实流量。第三个误区是请求语义不对，比如把本应 GET 的查询改成 POST，触发不同网关策略与缓存路径，结论自然失真。

本章核心目标是建立可信 HTTP 脚本：方法正确、参数真实、断言有效。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：“HTTP 不就填 URL？能返回 200 就行。”

小白：“200 只是第一层。还要看业务码和关键字段。否则你可能把失败当成功。”

大师：“压测成功要分层定义：网络层、协议层、业务层。三层都满足才是有效样本。”

小胖：“参数化是不是把 userId 换成变量就结束？”

小白：“还要考虑数据分布、唯一性和边界值。全量重复数据会让缓存命中过高。”

大师：“参数化本质是数据建模，不是字符串替换。你在模拟用户群体，不是在替换文本。”

小胖：“请求方法真有那么重要？”

小白：“当然。方法不同会触发不同后端逻辑和网关行为。”

大师：“尊重接口契约，是压测可信的前提。”

### 2.2 关键设计决策
- 方案A：只做状态码断言。
- 方案B：状态码 + 业务码 + 关键字段断言。
- 结论：采用方案B，避免假成功。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK 17
- JMeter 5.6.x
- 测试接口（登录、下单、查询）
- 数据文件 `data/users.csv`

### 3.2 分步实现
1. 规范基础请求配置
   - HTTP Request Defaults：协议、域名、端口
   - Header Manager：Content-Type 与鉴权头

2. 构建请求方法与请求体
   - GET 用于查询、POST 用于创建
   - JSON Body 与表单请求分开建模

3. 参数化数据
```text
CSV Data Set Config
vars: user_id,sku_id,amount
```
   - 配置回收策略，避免数据耗尽或重复失真

4. 设置断言
   - 状态码断言：200
   - JSON 断言：`$.code == 0`
   - 业务断言：`$.data.status == "SUCCESS"`

5. CLI 执行并生成报告
```bash
jmeter -n -t plans/http-order.jmx -l reports/http-order.jtl -e -o reports/http-order-html
```

### 3.3 测试验证
- 正常数据集执行，预期全部通过；
- 异常数据集执行，预期断言精准失败；
- 对比“仅状态码断言”和“三层断言”差异。

## 5. 项目总结（约500-800字）
- 优点：压测结果更贴近真实业务成功率。
- 缺点：断言与数据维护成本提升。
- 适用：REST API、网关性能回归、活动前验证。
- 注意事项：参数化要结合业务分布，断言要覆盖关键字段。
- 思考题：
  1. 幂等接口如何设计参数以避免误判吞吐？
  2. 当业务码成功但耗时明显上升时，你会怎样分层定位？


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
