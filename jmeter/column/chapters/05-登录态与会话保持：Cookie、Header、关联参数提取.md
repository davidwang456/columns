# 第5章 登录态与会话保持：Cookie、Header、关联参数提取

> 级别：基础篇  
> 目标读者：开发 / 测试  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
在真实业务系统中，用户请求很少是无状态的。大多数接口调用都依赖登录态、租户标识、签名字段和会话上下文。压测脚本如果没有正确模拟这些信息，即使请求发出去了，也无法代表真实业务压力。某内容平台曾在发布前进行“高并发发文”压测，结果失败率高达 60%。团队最初怀疑数据库写入能力不足，排查两天后才发现脚本里使用了固定 token，30 分钟后统一过期，后半场请求全部被鉴权拒绝。

这类问题说明：会话建模不只是功能正确性问题，更直接影响性能结论。若大量失败来自鉴权误配，你看到的“高错误率”并不反映系统容量，而是脚本构造错误。要解决这个问题，必须系统掌握三件事：Cookie 生命周期管理、Header 透传策略、关联参数动态提取。

本章把“能跑”升级为“能持续稳定跑”：不靠写死字段，不靠手工改 token，而是让脚本在长时间场景下自动维持会话。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：“我直接把抓包 token 填进 Header，不就搞定了吗？”

小白：“短时间调试可以，正式压测不行。token 会过期，401 会越来越多。”

大师：“性能脚本要模拟真实用户生命周期：登录、拿会话、访问业务、会话过期、刷新或重登。写死 token 属于静态快照，无法代表动态行为。”

技术映射：
- 一次性门票 -> 短周期 token
- 可续费会员卡 -> 可刷新会话机制

小胖：“Cookie 和 Header 都能传身份信息，差别大吗？”

小白：“Cookie 常由服务端下发并自动回传，Header 常用于显式鉴权。实际系统经常混用。”

大师：“所以脚本必须对齐接口契约。不要靠猜测。是用 Session Cookie、JWT Header、还是签名 Header，都要从接口文档和实际响应确认。”

小胖：“关联提取我总不稳定，正则老失效。”

小白：“能结构化提取就别用正则。JSON 用 JSON Extractor，HTML 用 CSS/XPath。”

大师：“稳定性优先原则：结构化 > 文本匹配。正则灵活但脆弱，字段一改就崩。”

### 2.2 关键设计决策
- 方案A：写死 token，快速跑通。
- 方案B：动态登录并提取 token/session，自动透传并处理过期。
- 结论：采用方案B，保障长时间场景真实性与稳定性。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK 17
- JMeter 5.6.x
- 登录接口、业务接口、测试账号池

### 3.2 分步实现
1. 登录并提取会话参数
   - 登录 Sampler：`/auth/login`
   - JSON Extractor：`token = $.data.token`
   - 若有 sessionId 一并提取

2. 透传会话参数
```text
Header Manager
Authorization: Bearer ${token}
X-Tenant-Id: ${tenant_id}
```
   - 公共 Header 放在线程组上层，避免作用域缺失

3. 启用 Cookie Manager
   - 根据场景决定是否“每次迭代清理 Cookie”
   - 保证一线程一用户的会话隔离

4. 处理过期与重登
   - If Controller 判断 401/403
   - 触发重登并刷新 token
   - 重试次数限制为 1~2 次，避免雪崩

5. 长稳执行验证
```bash
jmeter -n -t plans/auth-flow.jmx -Jduration=3600 -l reports/auth-flow.jtl -e -o reports/auth-flow-html
```

### 3.3 测试验证
- 观察 60 分钟内 401 比例变化；
- 统计登录请求占比，确认重登策略合理；
- 校验账号是否串用、会话是否泄漏。

## 5. 项目总结（约500-800字）
- 优点：
  - 会话行为更贴近真实用户；
  - 显著降低鉴权误伤导致的假失败；
  - 支撑长稳与链路场景压测。
- 缺点：
  - 脚本复杂度和维护成本上升；
  - 依赖接口契约稳定性。
- 适用：登录后业务流、多租户系统、权限敏感系统。
- 不适用：匿名只读探活场景。
- 注意事项：作用域清晰、提取器容错、重试有上限。
- 真实踩坑案例（3个）：
  1. token 写死导致后半场全 401；
  2. Cookie 未隔离引发用户串号；
  3. 正则提取脆弱，字段改名后整体失效。
- 思考题（2道）：
  1. JWT + Session 混合系统如何建模？
  2. 当鉴权服务抖动时，如何区分业务失败和认证失败？
- 推广计划：
  - 开发提供完整鉴权契约；
  - 测试维护账号池与提取规则；
  - 运维建立鉴权链路专项监控。


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
