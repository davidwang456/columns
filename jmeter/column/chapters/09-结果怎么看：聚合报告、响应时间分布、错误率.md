# 第9章 结果怎么看：聚合报告、响应时间分布、错误率

> 级别：基础篇  
> 目标读者：开发 / 测试 / 运维  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
压测结果误读，是比“脚本跑不起来”更危险的问题。很多团队在评审会上只看两项指标：平均响应时间和总 QPS。结果是上线后用户投诉不断，因为平均值掩盖了长尾延迟，QPS 掩盖了错误类型。某社交业务发布前报告显示“平均 120ms，吞吐 2500/s”，上线后却出现大面积接口超时。复盘发现：P99 达到 3.6s，且错误集中在某个依赖调用，平均值完全掩盖了风险。

结果分析不是“看报表”，而是“做诊断”。你要回答：
1) 系统在目标流量下是否满足 SLA？
2) 失败主要来自哪类错误？
3) 长尾是否可接受？
4) 回归相比基线是否退化？

本章给出一套可执行的分析框架：聚合报告读法、分位数解释、错误分层、对比基线和结论模板。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：
“报告里平均 180ms，感觉很好啊。我们可以过发布门禁了吧？”

小白：
“平均值不够。你看 P95、P99 没有？错误率按类型拆了吗？如果 1% 的请求 5 秒以上，用户照样骂。”

大师：
“结果分析要从‘总体好看’转为‘风险可控’。至少看四组指标：
- 延迟分位数（P50/P95/P99）；
- 错误率与错误类型；
- 吞吐与并发关系；
- 资源侧证据（CPU、GC、连接池、下游）。
单看客户端统计，会误判很多问题。”

技术映射：
- 班级平均分高 -> 不代表没有不及格学生
- 体检总分达标 -> 仍可能有单项高风险

小胖：
“错误率 0.5% 不算高吧？”

小白：
“要看错误是什么。超时、业务断言失败、连接拒绝，风险等级不一样。”

大师：
“对，错误必须分层：网络层、协议层、业务层。否则‘0.5%’没有决策价值。”

小胖：
“那发布结论怎么写才专业？”

小白：
“不能只写‘通过/不通过’，要写原因、边界、风险和改进建议。”

大师：
“结论模板要标准化：
- 场景说明；
- 指标结果；
- 与基线对比；
- 风险等级；
- 发布建议与回滚条件。”

### 2.2 关键设计决策
- 方案A：以平均值和总吞吐为主。
- 方案B：分位数 + 错误分层 + 基线对比 + 证据链。
- 选型结论与理由：采用方案B，结论更具决策价值。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JMeter 结果文件：`reports/ch09.jtl`
- HTML 报告：`reports/ch09-html`
- 监控面板：CPU、JVM、DB、网关

### 3.2 分步实现
1. 步骤目标：读取聚合报告核心字段。  
   - 关注指标：Samples、Average、Min/Max、错误率、Throughput。
   - 命令：
```bash
jmeter -g reports/ch09.jtl -o reports/ch09-html
```
   - 输出示例：
```text
Samples=540000 Avg=210ms 错误率=0.62 Throughput=900/s
```

2. 步骤目标：分析分位数与长尾。  
   - 关注：P90/P95/P99。
   - 示例结论：
```text
P95=420ms, P99=1850ms -> 长尾明显，需要定位下游抖动
```
   - 常见坑与解决：
     - 坑：只看 P95 忽略 P99。  
       解：发布门禁至少包含 P95 + P99 双阈值。

3. 步骤目标：错误分层归因。  
   - 分类：
     - 网络错误（connect/reset）
     - 协议错误（4xx/5xx）
     - 业务错误（断言失败）
   - 示例输出：
```text
timeout=0.21% 5xx=0.18% assertion_fail=0.23%
```
   - 常见坑与解决：
     - 坑：把所有失败合并成一个 错误率。  
       解：按错误标签输出明细并关联日志。

4. 步骤目标：做基线对比。  
   - 对比项：当前版本 vs 上版本
```text
P95: 360ms -> 420ms (+16.7%)
P99: 1300ms -> 1850ms (+42.3%)
错误率: 0.31% -> 0.62% (+0.31pp)
```
   - 结论：平均值接近，但长尾与错误显著劣化，建议暂缓发布。

5. 步骤目标：形成标准结论模板。  
   - 模板字段：场景、参数、结果、风险、建议、后续动作。

### 3.3 测试验证
- 验证项1：同一场景重复执行，关键指标波动在可接受范围。
- 验证项2：错误分类与应用日志一致。
- 验证项3：结论模板可被开发/运维共同认可。


## 4. 本章交付物清单
- 可执行脚本：`plans/ch09.jmx`（按本章替换编号）
- 执行命令模板：CLI 无界面命令 1 份
- 结果产物：`jtl` 原始数据 + HTML 报告
- 结论文档：场景参数、关键指标、风险结论、改进建议
- 回归资产：阈值基线与复盘记录各 1 份

## 5. 项目总结（约500-800字）
- 优点：
  - 从“看数字”升级为“做诊断”；
  - 显著提高发布评审质量；
  - 降低误判和争议成本。
- 缺点：
  - 分析链路更复杂；
  - 需要监控与日志协同支持；
  - 对报告标准化要求高。
- 适用场景：发布门禁、容量评估、故障复盘。
- 不适用场景：临时探活或极短冒烟。
- 注意事项：
  - 平均值不可单独用于结论；
  - 错误必须分层；
  - 对比必须基于同口径基线。
- 真实踩坑案例（3个）：
  1. 平均值好看但 P99 崩盘；
  2. 错误率 低但全部是关键交易失败；
  3. 报告与日志口径不一致导致误判。
- 思考题（2道）：
  1. 如何定义你的团队“发布通过”最低指标集合？
  2. 如何在结果波动较大时判断是否为环境噪声？
- 跨部门协作细则：
  - 开发：输出关键接口 SLA 与错误码语义；
  - 测试：维护分析模板与基线仓库；
  - 运维：提供监控证据链与容量视图；
  - 管理：将结论模板纳入发布评审流程。


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
