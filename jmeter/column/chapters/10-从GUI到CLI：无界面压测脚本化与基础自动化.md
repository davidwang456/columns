# 第10章 从GUI到CLI：无界面压测脚本化与基础自动化

> 级别：基础篇  
> 目标读者：开发 / 测试 / 运维  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
很多团队在 GUI 中能把脚本“点出来”，但一到发布回归就卡住：没人记得当时点了什么、参数写在哪、输出目录在哪里、如何复现同一场景。GUI 适合调试，不适合规模执行和流水线集成。某互联网团队曾在发版前临时做压测，工程师在 GUI 中跑出“通过”结论，第二天另一位同事用相同脚本重跑却失败，原因是 GUI 本地环境里手动改过代理和 Header，未沉淀到脚本和命令。

CLI化的价值是把“人的操作”转成“机器可执行说明书”：输入参数明确、执行命令固定、产物可归档、流程可审计。只有 CLI化，才能进入 CI/CD，实现版本级性能回归。

本章目标是完成从 GUI 到 CLI 的工程跃迁：参数外置、命令模板化、报告自动生成、失败门禁化。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：
“GUI 可视化很方便，为什么一定要 CLI？感觉命令行很麻烦。”

小白：
“GUI 适合你一个人调试，不适合团队复现。你点一次我点一次，细节差异就会放大。”

大师：
“本质是可重复性。CLI 把执行细节写成可版本化的命令，任何人、任何节点都能按同样方式运行。性能测试要走向工程化，CLI 是必经之路。”

技术映射：
- 手工记账 -> GUI 临时操作
- 自动记账系统 -> CLI 标准流程

小胖：
“那 GUI 以后就不用了？”

小白：
“不是。GUI 负责建模调试，CLI 负责执行回归。”

大师：
“对。建议工作流：GUI 构建 -> 保存 jmx -> CLI 执行 -> 报告归档 -> 流水线门禁。分工清晰，效率最高。”

小胖：
“门禁怎么做？失败就全拦吗？”

小白：
“要按阈值策略，不是简单成功失败。”

大师：
“门禁要有层级：
- 强门禁：关键交易错误率、P99 超阈值；
- 弱门禁：次要接口轻微波动，给出预警但不阻断。”

### 2.2 关键设计决策
- 方案A：GUI 手工执行，报告人工截图。
- 方案B：CLI 标准执行 + 报告自动生成 + 阈值门禁。
- 选型结论与理由：采用方案B，支撑持续性能回归。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK 17
- JMeter 5.6.x
- Git 仓库结构：plans/data/reports/scripts
- CI 环境：Jenkins/GitLab CI/GitHub Actions（任选）

### 3.2 分步实现
1. 步骤目标：将 GUI 脚本参数外置。  
   - 线程组参数改为 `${__P(threads,100)}` 等属性读取。
   - 目标地址改为 `${__P(host,example.com)}`。

2. 步骤目标：建立 CLI 执行模板。  
   - 命令：
```bash
jmeter -n -t plans/ch10-flow.jmx -Jthreads=300 -Jrampup=60 -Jduration=1200 -Jhost=perf.api.local -l reports/ch10.jtl -e -o reports/ch10-html
```
   - 输出示例：
```text
Created the tree successfully using plans/ch10-flow.jmx
summary = 216000 in 00:20:00 = 180.0/s Avg:238 Min:41 Max:2890 Err:0.42%
```

3. 步骤目标：封装脚本化入口（Windows 示例）。  
   - `scripts/run-ch10.ps1`：读取参数、创建报告目录、执行命令、返回码透传。
   - 关键点：失败时 `exit 1`，让 CI 能识别。

4. 步骤目标：自动生成报告并归档。  
   - 命令：
```bash
jmeter -g reports/ch10.jtl -o reports/ch10-html
```
   - 归档产物：jmx、jtl、html、执行日志、参数快照。

5. 步骤目标：接入基础门禁。  
   - 示例阈值：
```text
P95 <= 500ms
P99 <= 1200ms
错误率 <= 0.5%
```
   - 若超阈值：流水线标红并附失败原因。

### 3.3 测试验证
- 验证项1：同脚本在本地与 CI 节点执行结果趋势一致。
- 验证项2：故意设置低阈值，确认门禁触发。
- 验证项3：产物归档可追溯到 commit 与参数。


## 4. 本章交付物清单
- 可执行脚本：`plans/ch10.jmx`（按本章替换编号）
- 执行命令模板：CLI 无界面命令 1 份
- 结果产物：`jtl` 原始数据 + HTML 报告
- 结论文档：场景参数、关键指标、风险结论、改进建议
- 回归资产：阈值基线与复盘记录各 1 份

## 5. 项目总结（约500-800字）
- 优点：
  - 执行过程标准化，复现成本低；
  - 报告可追溯，支持版本对比；
  - 可直接嵌入 CI/CD，形成性能守门。
- 缺点：
  - 前期需要改造脚本参数化；
  - 需要维护运行脚本和门禁逻辑；
  - CI 环境资源不稳定会引入噪声。
- 适用场景：版本回归、活动前演练、夜间定时压测。
- 不适用场景：一次性临时调试。
- 注意事项：
  - GUI 只用于调试，正式执行统一 CLI；
  - 参数与产物必须版本化；
  - 门禁阈值要按业务分级设定。
- 真实踩坑案例（3个）：
  1. GUI 手工修改未入库导致无法复现；
  2. 命令参数遗漏导致环境差异误判；
  3. 报告未归档导致回归对比失效。
- 思考题（2道）：
  1. 如何设计“强门禁+弱门禁”策略，兼顾质量与交付效率？
  2. 如何在 CI 资源抖动下提升性能回归稳定性？
- 跨部门协作细则：
  - 开发：提供性能预算与关键阈值；
  - 测试：维护脚本参数化与门禁规则；
  - 运维：提供稳定执行节点和归档存储；
  - 发布管理：将性能门禁纳入上线准入标准。


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
