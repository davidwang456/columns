# 第11章 典型业务场景一：电商下单链路压测（登录-加购-下单）

> 级别：基础篇  
> 目标读者：开发 / 测试 / 运维  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
本章以真实业务压力验证为主线，解决“脚本能跑但结论不可信”的常见问题。典型痛点包括：场景覆盖不完整、指标口径不一致、执行环境不可复现、跨团队职责边界模糊。通过引入明确业务目标、可审计参数、标准报告与复盘机制，本章将把一次性压测升级为可持续性能工程能力。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：
“我们先跑一轮看看数字，后面再补细节，应该也可以吧？”

小白：
“先跑不等于先对。没有模型、没有口径、没有分层，数字只能制造幻觉。”

大师：
“压测设计要先定义‘要回答什么业务问题’，再设计场景与指标，最后落地脚本。否则测试很忙，决策无效。”

技术映射：
- 先开车再看导航 -> 方向可能完全错
- 先规划路线再出发 -> 到达可预期

### 2.2 关键设计决策
- 方案A：一次性执行、人工解释。
- 方案B：参数化执行、标准化报告、证据链闭环。
- 选型结论与理由：采用方案B，保证结果可复核与可持续改进。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK 17
- JMeter 5.6.x
- 目录规范：plans/data/reports/scripts
- 观测面板：应用、数据库、网关、主机

### 3.2 分步实现
1. 步骤目标：建立本章核心场景并参数化。  
   - 命令：
```bash
jmeter -n -t plans/ch11.jmx -Jthreads=300 -Jrampup=60 -Jduration=1200 -l reports/ch11.jtl -e -o reports/ch11-html
```
   - 运行结果示例：
```text
summary = 216000 in 00:20:00 = 180.0/s Avg:245 Min:42 Max:3220 Err:0.58%
```
   - 常见坑与解决：参数散落、目录冲突、报告不可追溯。

2. 步骤目标：加入业务案例与异常分支验证。  
   - 案例：正常路径、降级路径、超时路径。
   - 结果：形成分支级错误分类和耗时分位数。
   - 常见坑与解决：仅统计总体错误，无法定位问题。

3. 步骤目标：沉淀命令输出与复盘模板。  
   - 输出模板字段：场景、参数、结果、风险、建议、回归计划。
   - 常见坑与解决：结论缺少边界条件，导致发布争议。

### 3.3 测试验证
- 验证同参数重复 3 次，观察波动；
- 验证错误分类与日志一致；
- 验证报告可追溯到脚本版本与参数快照。


## 4. 本章交付物清单
- 可执行脚本：`plans/ch11.jmx`（按本章替换编号）
- 执行命令模板：CLI 无界面命令 1 份
- 结果产物：`jtl` 原始数据 + HTML 报告
- 结论文档：场景参数、关键指标、风险结论、改进建议
- 回归资产：阈值基线与复盘记录各 1 份

## 5. 项目总结（约500-800字）
- 优点：
  - 形成可执行、可复核、可回归的性能实践；
  - 缩短定位与决策时间；
  - 强化跨团队协同效率。
- 缺点：
  - 前期规范建设成本较高；
  - 需要持续维护基线与模板。
- 适用场景：版本回归、活动演练、容量评估、生产问题复现。
- 不适用场景：无稳定环境、无明确业务指标的临时测试。
- 注意事项：统一口径、保留原始数据、建立定期复盘。
- 真实踩坑案例（3个）：
  1. 指标口径不一致导致评审冲突；
  2. 测试覆盖缺失导致上线后分支故障；
  3. 结果无法追溯导致回归失真。
- 思考题（2道）：
  1. 如何把本章实践沉淀为组织级标准资产？
  2. 如何平衡严格门禁与迭代效率？
- 跨部门协作细则：
  - 开发：定义接口 SLA、错误语义、性能预算；
  - 测试：维护场景脚本、基线与复盘模板；
  - 运维：提供稳定环境、观测看板和容量视图；
  - 产品/业务：确认关键链路和优先级边界。


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
