# 第6章 数据驱动测试：CSV Data Set Config与变量作用域

> 级别：基础篇  
> 目标读者：开发 / 测试 / 运维  
> 建议字数：3000-5000字

## 1. 项目背景（约500字）
很多团队做压测时，脚本看起来“跑得很快”，但结果一旦上线就失真。最常见的根因不是线程数不够，而是数据模型错误：请求参数固定、账号重复、商品 ID 集中、地域分布单一。某零售业务在活动前压测“秒杀下单接口”，压测报告显示系统可承载 2 万 QPS，线上开卖 5 分钟后却出现库存锁冲突和数据库热点。复盘后发现，压测脚本只用了 100 个 SKU 循环，导致缓存和数据库访问高度集中，和真实活动“数万 SKU 分散访问”的模式完全不一致。

数据驱动测试的价值，不是“把参数换成变量”，而是把真实业务数据分布映射进压测行为。JMeter 的 `CSV Data Set Config` 只是入口，真正难点在于变量作用域、数据读取策略、线程隔离、EOF 处理和数据治理。比如：一个线程是否对应一个用户？文件读到末尾后是循环还是停止？变量在同级组件和子组件中的可见性怎样？这些问题如果处理不当，得到的结论会系统性偏差。

本章目标是建立一套“可复现的数据驱动标准”：数据可管理、变量可追踪、结果可解释。你会看到从“演示级变量替换”走向“工程级数据建模”的完整路径。

## 2. 项目设计（约1200字）
### 2.1 剧本式交锋对话
小胖：
“CSV 不就是一行一行读吗？我把 userId、skuId 写进去就完了。只要接口能回 200，不就说明数据驱动成功了？”

小白：
“你这个说法太乐观了。先不说 200 不等于业务成功，就算都 200，也可能是重复数据把缓存打热了，根本不是生产流量分布。你有没有考虑线程共享同一批数据导致撞号？”

大师：
“对，数据驱动有三个层级。第一层是变量替换，解决‘能跑’；第二层是数据分布，解决‘像不像真实业务’；第三层是数据治理，解决‘能不能长期复用’。大多数团队只做了第一层，就急着给容量结论。”

技术映射：
- 只换变量名 -> 把标签贴到空瓶子上
- 建模数据分布 -> 按人群特征真实备货
- 数据治理 -> 建立可追溯供应链

小胖：
“那变量作用域到底怎么理解？我有时候在后置处理器里能取到变量，有时候取不到。”

小白：
“你得区分 JMeter 变量、属性，以及组件层级。变量通常在同一线程上下文可见，跨线程要用属性或者外部存储。还有控制器嵌套会影响你以为的‘就近可见’。”

大师：
“补一条：不要滥用全局属性代替变量。属性适合全局常量，不适合高并发下存线程私有数据，否则会发生覆盖和串值。线程安全问题在压测场景尤其隐蔽。”

技术映射：
- 家庭冰箱私人物品 -> 线程私有变量
- 小区公共公告栏 -> 全局属性

小胖：
“CSV 文件读完后我一般直接循环，省事。”

小白：
“循环不一定安全。比如账号密码读完再循环，可能同账号多点登录互踢，结果全是鉴权失败。”

大师：
“所以要按场景选择 EOF 策略：
1) 账号唯一场景：EOF 后停止线程；
2) 可重复数据场景：允许循环；
3) 高并发活动场景：分层数据池 + 随机/权重抽样。不是一刀切。”

### 2.2 关键设计决策
- 方案A：简单 CSV 替换，默认循环，变量随用随写。
- 方案B：数据分层建模（账号池/商品池/地域池）、变量作用域规范、EOF 策略分场景配置。
- 选型结论与理由：采用方案B。虽然准备成本更高，但能显著降低压测结论偏差和线上翻车风险。

## 3. 项目实战（约1500-2000字）
### 3.1 环境准备
- JDK 17
- JMeter 5.6.x
- 数据文件：
  - `data/users.csv`（uid,token,region）
  - `data/sku_hot.csv`（热门商品）
  - `data/sku_longtail.csv`（长尾商品）
- 目标接口：登录、查询库存、下单

### 3.2 分步实现
1. 步骤目标：建立基础 CSV 数据驱动脚本。  
   - 配置：
     - `CSV Data Set Config`
     - Filename: `data/users.csv`
     - Variable Names: `uid,token,region`
     - Recycle on EOF: `False`
     - Stop thread on EOF: `True`
   - 命令：
```bash
jmeter -n -t plans/ch06-csv-basic.jmx -Jthreads=200 -Jrampup=30 -Jduration=600 -l reports/ch06-basic.jtl -e -o reports/ch06-basic-html
```
   - 运行结果示例：
```text
summary +  120000 in 00:10:00 = 200.0/s Avg:145 Min:38 Max:1280 Err:0 (0.00%)
summary =  120000 in 00:10:00 = 200.0/s Avg:145 Min:38 Max:1280 Err:0 (0.00%)
```
   - 常见坑与解决：
     - 坑：文件路径写相对路径但执行目录变化导致找不到文件。  
       解：在运行脚本中固定工作目录，或使用绝对路径参数 `-Jdata.dir`。

2. 步骤目标：构建“热门+长尾”混合数据分布。  
   - 实现思路：
     - 80% 请求来自热门 SKU，20% 来自长尾 SKU；
     - 使用两个 CSV + Throughput Controller 控制占比。
   - 命令：
```bash
jmeter -n -t plans/ch06-sku-distribution.jmx -Jthreads=400 -Jrampup=60 -Jduration=900 -l reports/ch06-distribution.jtl -e -o reports/ch06-distribution-html
```
   - 运行结果示例：
```text
hot_sku_sampler    count=432000 p95=260ms err=0.12%
longtail_sampler   count=108000 p95=340ms err=0.08%
```
   - 常见坑与解决：
     - 坑：热门和长尾用同一数据池，统计无法分层。  
       解：按业务语义分开采样器和标签，便于后续归因。

3. 步骤目标：验证变量作用域与线程隔离。  
   - 验证方式：
     - 使用 Debug Sampler 打印 `${uid}`、`${token}`；
     - 增加 BeanShell/JSR223 记录线程名与变量值，确认无串值。
   - 示例输出：
```text
Thread Group 1-23 uid=u_000231 token=t_xxx region=华东
Thread Group 1-24 uid=u_000232 token=t_xxx region=华南
```
   - 常见坑与解决：
     - 坑：用 `props.put()` 写线程私有 token，导致跨线程覆盖。  
       解：线程私有数据一律用 `vars.put()`。

4. 步骤目标：处理 EOF 与异常数据回收。  
   - 场景策略：
     - 唯一账号：`Stop thread on EOF=True`
     - 可重复 SKU：`Recycle on EOF=True`
   - 命令：
```bash
jmeter -n -t plans/ch06-eof-policy.jmx -Jthreads=800 -Jrampup=120 -Jduration=1200 -l reports/ch06-eof.jtl
```
   - 结果观察：线程是否出现大量提前退出、错误是否激增。
   - 常见坑与解决：
     - 坑：全部线程因 EOF 停止，误判系统稳定。  
       解：在报告中追加“活跃线程数曲线”审查。

### 3.3 测试验证
- 验证项1：同一数据集重复执行 3 次，关键指标偏差 < 8%。
- 验证项2：替换为“固定参数脚本”对照，比较 P95 与错误率差异。
- 验证项3：模拟坏数据（空 token、非法 region）确认断言拦截有效。

## 5. 项目总结（约500-800字）
- 优点：
  - 让压测输入更接近真实业务分布；
  - 显著降低“缓存虚高”“数据撞号”导致的误判；
  - 脚本可持续复用，支持回归与对比。
- 缺点：
  - 数据准备与治理成本上升；
  - 需要明确变量作用域规范；
  - 数据安全（脱敏）要求更高。
- 适用场景：电商、支付、内容平台、账号体系复杂业务。
- 不适用场景：纯匿名读接口、超轻量探活。
- 注意事项：
  - 线程私有变量用 `vars`，全局常量才用 `props`；
  - EOF 策略按场景配置，不要默认循环；
  - 数据文件需版本化并建立变更审计。
- 真实踩坑案例（3个）：
  1. 账号池过小导致互踢登录，错误率虚高；
  2. 固定 SKU 导致缓存过热，容量结论虚高；
  3. 变量覆盖造成跨用户串单，问题难复现。
- 思考题（2道）：
  1. 如何在不暴露真实用户信息的前提下构造高保真压测数据？
  2. 当数据分布变化导致性能回归失败时，如何区分“代码退化”与“流量结构变化”？
- 跨部门协作细则：
  - 开发：提供关键业务字段与幂等规则；
  - 测试：维护数据字典、CSV 版本、分布策略；
  - 运维：保障数据文件分发、执行节点权限和审计日志；
  - 安全：审核脱敏策略与数据存储合规性。


### 统一指标口径建议
- 延迟指标：默认关注 P95、P99 与最大值，不仅看平均值。
- 稳定性指标：统一使用“错误率”并按网络/协议/业务三层分类。
- 对比基线：同环境、同参数、同时间窗下进行版本对比。
- 结论输出：必须包含风险等级、影响范围、建议动作与回归计划。
