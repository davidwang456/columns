# 第9章：密码策略与暴力破解防护

## 1 项目背景

某社交平台"年度安全审计报告"的封面赫然印着三个大字——"撞库案"。事情的起因很简单：攻击者从暗网获取了一批其他网站泄露的邮箱密码组合，编了个脚本对着该平台的登录接口轮番尝试。平台部署了三台登录服务器，但没做任何登录失败限制——既没有失败次数上限，也没有验证码，更没有异常IP检测。攻击者在24小时内以每秒约200次的速度发起请求，累计尝试超过1700万次，最终约3000个用户账号被成功撞库。

事后复盘发现的问题令人震惊：平台允许用户设置6位纯数字密码，大量用户使用"123456""password""qwerty"等Top 100弱密码，而系统对此从未拦截，甚至连密码长度下限都没有设置。暴力破解的防护完全缺失——攻击者可以对着同一个用户名连续尝试数万次密码而不受任何惩罚。密码历史记录不存在，不少用户在被撞库后重置密码，写下的竟然还是原来的密码。更令人后怕的是，平台没有多因素认证——只要密码对了，账号即刻沦陷。最终，该平台因"未采取有效措施保护用户数据"被监管部门罚款580万元，用户信任度暴跌，DAU在一个月内下滑了23%。

这个案例揭示了一个残酷的事实：认证安全不是靠一个环节撑起来的，而是一条环环相扣的链条——密码复杂度决定了破解难度门槛，失败次数限制决定了攻击者的尝试空间，密码历史防止了旧凭证的循环复用，多因素认证则是最后一道防线。任何一环的缺失，都会让整个认证体系形同虚设。本章将聚焦Keycloak在这一领域的三大核心能力：密码策略体系、Brute Force Detection机制、OTP多因素认证入门，构建一套"进不来、猜不到、偷不走"的认证防线。

## 2 项目设计——剧本式交锋对话

**小胖**（把银行卡拍在桌上）：大师，我昨天去ATM取钱，输错三次密码卡就被吞了，银行让我拿身份证去柜台解锁。这不就是暴力破解防护吗？但我记得卡被锁是永久的，得人工解。网站为啥不也这样搞——输错三次直接永久封号？

**大师**：问得好。ATM锁卡之所以敢"永久"，是因为你有身份证可以去柜台验证身份后解锁，这是一个可靠的恢复通道。但网站没有柜台——如果把用户永久锁定，攻击者可以用一个他根本不想登录的账号反复输错密码，导致真实用户被锁在外面，这叫"拒绝服务攻击"。所以Keycloak默认采用**临时锁定（Temporary Lockout）**而非永久锁定：失败达到阈值后，账户暂停一段时间（比如15分钟），超时自动恢复。如果想永久锁定，需要管理员手动操作，适合安全等级极高的内部系统。

**小胖**：哦，所以银行敢永久是因为有柜台兜底。网站没柜台，锁死反而不安全。

**大师**：正是。这叫"安全性和可用性的平衡"——你不能因为怕被偷就把门焊死。

**小白**（端着咖啡杯走过来）：我接着问。这失败计数存在哪里？单机没问题，但如果是集群部署——节点A收到了3次失败，节点B收到了2次，两个节点各自只记了3和2，都不满足"5次失败锁定"的阈值，攻击者等于拥有了无限尝试的机会？

**大师**：好问题，直击要害。Keycloak将失败计数存储在**Infinispan分布式缓存**中。在集群模式下，Infinispan的Distributed Cache会自动将数据分片到多个节点，每个key（在这里就是用户ID对应的失败计数）分配一个primary owner和若干backup owner。无论请求打到哪个节点，都会路由到primary owner读写数据，保证计数的全局一致性。

但这有个前提——集群节点必须通过Infinispan正确组网。如果网络分区导致两个节点各自独立工作，就会出现你说的"计数分裂"问题。生产环境的解决方案是：设置合适的Infinispan owner数量（通常为2-3），配合JGroups的FD（Failure Detection）协议快速检测节点宕机，确保故障转移在秒级完成。

**小白**：也就是说，失败计数不是存在数据库，而是在缓存里。那集群重启计数不就全丢了？

**大师**：对，这也是一个需要正视的固有限制。Infinispan是内存缓存，虽然可以配置持久化到磁盘（Single File Store），但Keycloak默认不开启。重启后失败计数确实会被清零。所以如果你的安全策略要求"绝对不能容忍计数丢失"，有两个增强方案：一是开启Infinispan的持久化配置；二是在网络层（WAF/API网关）额外维护一套基于IP的失败计数，形成纵深防御。

**小白**：还有一个密码学的核心问题——三种主流哈希算法，到底怎么选？

**大师**：这个问题值得深聊。三者区别如下：

- **PBKDF2**：基于HMAC的密钥派生函数，迭代次数可调（Keycloak默认27500次）。CPU密集型，对GPU/ASIC加速有一定抗性。特点是标准化早（RFC 2898，2000年），兼容性最好，FIPS认证。劣势是内存消耗固定——攻击者可以用大量并行硬件加速破解。

- **bcrypt**：基于Blowfish分组密码，自带盐值。特点是内置的"cost factor"控制迭代轮数（2^cost次），随硬件发展可调。对GPU有一定抗性（因为Blowfish结构对GPU不友好），但不抗FPGA/ASIC。实际使用广泛，社区积累深厚。

- **argon2**：2015年密码哈希竞赛冠军。最大特点是**内存硬**（Memory-Hard）——算法设计使得每次哈希计算必须消耗指定量的内存（默认1MB），且攻击者无法用更少内存完成计算。这意味着攻击者无法用ASIC芯片大量并行破解，因为每增加一条计算流水线就要额外分配一块内存。argon2有三个变体：argon2d（抗GPU）、argon2i（抗侧信道）、argon2id（混合，Keycloak使用）。推荐新系统直接选argon2。

**大师总结技术映射**：

- ATM吞卡 → Temporary Lockout：有限时间自动解禁，平衡安全与可用
- 分布式计数器 → Infinispan Distributed Cache：跨节点共享失败状态
- 密码哈希 → PBKDF2/bcrypt/argon2：单向派生，破解成本随算法升级指数增长
- 三把钥匙开一个保险柜 → MFA：任一因素泄露不影响整体安全

**小胖**（嚼着薯片）：那我再问一个实用的。密码策略设得太严，用户抱怨记不住；设得太松，安全部门不答应。到底怎么找平衡？

**大师**：这个问题所有安全架构师都为难过。我的建议参考NIST SP 800-63B最新指南：

1. **长度优先于复杂度**：12位以上，不强求四种字符组合。`correct-horse-battery-staple`这种长口令比`P@ssw0rd!`更安全也更易记。
2. **取消定期强制改密**：研究表明强制定期改密反而导致用户选择弱密码或用序列密码（Password1→Password2→Password3）。NIST建议只在怀疑泄露时才要求改密。
3. **用密码黑名单替代复杂规则**：禁止Top 10000弱密码比要求"大写+小写+数字+特殊字符"更有效。

**小白**：说到MFA推广，我们公司上次推OTP，员工反对声音很大，都说"登个系统还要掏手机扫二维码，效率太低"。

**大师**：推广MFA的核心不是技术，是心理学。几个关键策略：先对高管和IT团队强制启用，形成示范效应；用"条件OTP"（Conditional OTP）降低打扰——同一设备同一IP第二次登录时跳过OTP；初期给一个30天的过渡期，期间可以选"30天内不再验证"。另外，WebAuthn/FIDO2值得关注——用指纹或Windows Hello解锁手机完成认证，体验远优于输入6位数字验证码，这是未来趋势。

**大师总结技术映射**：

- 密码黑名单 → Password Blacklist Policy：从根上禁止弱密码
- NIST指南 → 用户体验驱动安全：拉长不改的密码比频繁改的短密码更安全
- 条件OTP → 自适应认证：低风险场景降低认证强度
- WebAuthn/FIDO2 → 无密码认证：生物特征+设备密钥替代传统密码

### 步骤1：配置企业级密码策略

**目标**：在Realm级别配置全局密码策略，确保所有新用户密码满足企业安全基线。

操作路径：`Admin Console → demo-realm → Realm Settings → Authentication → Password Policy`

点击**Add Policy**，依次添加以下策略：

| 策略 | 参数 | 说明 |
|------|------|------|
| Minimum Length | 12 | 最小密码长度 |
| Uppercase Characters | 1 | 至少1个大写字母 |
| Lowercase Characters | 1 | 至少1个小写字母 |
| Digits | 1 | 至少1个数字 |
| Special Characters | 1 | 至少1个特殊字符 |
| Not Username | - | 密码不得包含用户名（不区分大小写） |
| Not Email | - | 密码不得包含邮箱本地部分 |
| Hashing Algorithm | argon2 | 密码哈希算法，内存硬抗GPU/ASIC |
| Password History | 5 | 不能重复使用最近5次密码 |
| Password Expiration | 90 | 90天必须修改密码 |

配置完成后点击**Save**。策略立即对新创建/修改密码的用户生效。

**验证——尝试创建弱密码用户**：

```bash
# 1. 获取管理员Token
ADMIN_TOKEN=$(curl -s -X POST http://localhost:8080/realms/master/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=admin-cli" \
  -d "username=admin" \
  -d "password=admin" \
  -d "grant_type=password" | jq -r '.access_token')

# 2. 尝试创建弱密码用户
curl -X POST http://localhost:8080/admin/realms/demo-realm/users \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "username": "weakuser",
    "email": "weak@test.com",
    "credentials": [{
      "type": "password",
      "value": "123",
      "temporary": false
    }],
    "enabled": true
  }'
```

**运行结果**——返回400 Bad Request，响应体：

```json
{
  "error": "invalidPasswordMinLengthMessage",
  "error_description": "Invalid password: minimum length 12.",
  "params": ["12"]
}
```

再尝试密码为`weakuser123`（包含用户名），返回：

```json
{
  "error": "invalidPasswordNotUsernameMessage",
  "error_description": "Invalid password: must not be equal to the username."
}
```

只有当密码满足所有策略（如`Keycloak@2026!`）时，用户创建才能返回201 Created。

### 步骤2：配置Brute Force Detection

**目标**：配置暴力破解检测机制，防止攻击者通过反复尝试密码来攻破账号。

操作路径：`Admin Console → demo-realm → Realm Settings → Security Defenses → Brute Force Detection`

配置参数：

| 参数 | 值 | 说明 |
|------|---|------|
| Enabled | ON | 启用暴力破解检测 |
| Permanent Lockout | OFF | 临时锁定（推荐），超时后自动解禁 |
| Max Login Failures | 5 | 连续失败5次后触发锁定 |
| Wait Increment | 1 minute | 每次锁定后等待时间递增1分钟 |
| Quick Login Check MilliSeconds | 1000 | 1秒内的快速连续登录记为失败 |
| Minimum Quick Login Wait | 1 minute | 快速登录失败后的最短等待时间 |
| Max Failure Wait | 15 minutes | 最长锁定时间上限 |
| Failure Reset Time | 12 hours | 12小时后失败计数器自动清零 |

**锁定逻辑解析**：用户连续5次登录失败后，账户进入临时锁定状态。第一次锁定的等待时长为`Wait Increment`（1分钟），此后每次再次触发锁定，等待时间递增1分钟，直到达到`Max Failure Wait`（15分钟）的上限。12小时内没有新的失败尝试，计数器自动归零。`Quick Login Check`机制防止攻击者编写脚本在1秒内注入多次登录请求。

### 步骤3：Python脚本模拟暴力破解

**目标**：用Python模拟攻击者反复尝试密码，观察锁定机制的实际行为。

```python
import requests
import time

TOKEN_URL = "http://localhost:8080/realms/demo-realm/protocol/openid-connect/token"

for i in range(10):
    resp = requests.post(TOKEN_URL, data={
        "client_id": "oms-frontend",
        "username": "targetuser",
        "password": f"wrongpassword{i}",
        "grant_type": "password"
    })

    status = resp.status_code
    error_desc = ""
    if resp.headers.get("content-type", "").startswith("application/json"):
        error_desc = resp.json().get("error_description", "")

    print(f"Attempt {i+1}: status={status}, error={error_desc}")
    time.sleep(0.5)

print("\n--- 等待2分钟后重试，验证锁定是否解除 ---")
time.sleep(120)

resp = requests.post(TOKEN_URL, data={
    "client_id": "oms-frontend",
    "username": "targetuser",
    "password": "correct_password",
    "grant_type": "password"
})
print(f"After lock period: status={resp.status_code}")
if resp.status_code == 200:
    print("Login successful! Lock cleared.")
else:
    print(f"Still locked: {resp.json().get('error_description', '')}")
```

**运行结果**：

```
Attempt 1: status=401, error=Invalid user credentials
Attempt 2: status=401, error=Invalid user credentials
Attempt 3: status=401, error=Invalid user credentials
Attempt 4: status=401, error=Invalid user credentials
Attempt 5: status=401, error=Invalid user credentials
Attempt 6: status=401, error=Account is temporarily locked. Contact admin.
Attempt 7: status=401, error=Account is temporarily locked. Contact admin.
Attempt 8: status=401, error=Account is temporarily locked. Contact admin.
Attempt 9: status=401, error=Account is temporarily locked. Contact admin.
Attempt 10: status=401, error=Account is temporarily locked. Contact admin.

--- 等待2分钟后重试，验证锁定是否解除 ---
After lock period: status=200
Login successful! Lock cleared.
```

可以看到：前5次返回"Invalid user credentials"，第6次开始返回"Account is temporarily locked"。等待锁定周期后，用正确密码登录成功。

### 步骤6：配置OTP（一次性密码）作为第二因素

**目标**：为用户启用TOTP（基于时间的一次性密码），实现双因素认证。

操作路径：`Admin Console → demo-realm → Realm Settings → Authentication → OTP Policy`

配置：

| 参数 | 值 | 说明 |
|------|---|------|
| OTP Type | Time Based (TOTP) | 基于时间，每30秒生成新码 |
| OTP Hash Algorithm | SHA1 | TOTP标准哈希算法 |
| Number of Digits | 6 | 6位验证码 |
| Look Ahead Window | 1 | 允许前后1个时间窗口的偏差 |
| OTP Token Period | 30s | 每30秒刷新一次 |

为用户启用OTP：

1. Admin Console → Users → targetuser → Credentials
2. 点击"Configure OTP"按钮
3. 系统生成二维码（QR Code）和手动密钥
4. 用户打开手机Authenticator App（Google Authenticator / Microsoft Authenticator），扫描二维码或输入密钥
5. 在App中看到6位动态验证码后，在Keycloak界面输入验证码确认绑定

**验证双因素认证流程**：

```bash
# 第一步：使用密码获取Token（无OTP）——预期失败
curl -s -X POST http://localhost:8080/realms/demo-realm/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=oms-frontend" \
  -d "username=targetuser" \
  -d "password=Keycloak@2026!" \
  -d "grant_type=password" | jq .

# 返回：
# { "error": "invalid_grant", "error_description": "Invalid user credentials" }
# 注意：用户已绑定OTP后，仅凭密码无法通过认证

# 第二步：使用密码+OTP获取Token
TOTP_CODE="123456"  # 从Authenticator App获取当前6位码
curl -s -X POST http://localhost:8080/realms/demo-realm/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=oms-frontend" \
  -d "username=targetuser" \
  -d "password=Keycloak@2026!" \
  -d "otp=$TOTP_CODE" \
  -d "grant_type=password" | jq .
```

**运行结果**：输入正确的密码和OTP后，返回access_token、refresh_token等信息，认证成功。

### 可能遇到的坑

1. **暴力破解计数存在Infinispan缓存中**：集群重启、缓存回收、节点故障都可能导致失败计数丢失。如果你需要持久化计数，可以配置Infinispan的`Single File Store`将缓存写入磁盘，或在网关层独立维护一套计数逻辑。

2. **Temporary Lockout vs Permanent Lockout的区别**：Temporary Lockout（临时锁定）在锁定周期后自动解禁；Permanent Lockout（永久锁定）必须由管理员手动操作才能解除。生产环境推荐Temporary Lockout，只有银行、金融等极端安全敏感场景才考虑Permanent Lockout。

3. **密码策略修改后，已有用户不受影响**：密码策略只在用户**下一次修改密码**时被校验。已存在的不合规密码不会自动失效。如果需要强制所有用户更新密码，可以结合`Password Expiration`设置为0（T+0天过期），配合全局`Required Action`批量推送到所有用户。

4. **OTP的Look Ahead Window设置过大会降低安全性**：Look Ahead Window允许验证码在时间窗口前后N个周期内有效。Window=1时，验证码在±30秒内有效（共90秒窗口）。如果设备时间偏差较大，可以适当增大，但每增大1个窗口就多出1/30的暴力破解可能性。推荐值：1或2。

5. **Quick Login Check可能导致合法用户被误锁**：如果用户浏览器插件或移动端App自动重试登录，在1秒内连续发送多个请求，可能被判定为快速登录失败而触发锁定。排查方法：在Brute Force Detection页面观察`numFailures`的增长模式，确认是否有自动化工具在重试。

## 4 项目总结

### 优点与缺点

| 维度 | Keycloak内置防护 | 自建防护系统 | WAF/网关层防护 |
|------|---------------|------------|--------------|
| 部署复杂度 | ✅ 开箱即用，勾选启用 | ❌ 需完整开发 | ⚠️ 依赖网关能力（如NGINX rate limiting） |
| 防护粒度 | ✅ 用户级失败计数+IP关联 | ✅ 完全自定义 | ⚠️ 通常仅IP维度，不感知用户 |
| 密码复杂度 | ✅ 丰富的策略项+哈希算法选择 | ❌ 手写正则+哈希逻辑 | ❌ 不涉及密码层面 |
| 多因素认证 | ✅ 内置TOTP/HOTP/WebAuthn | ❌ 集成第三方SDK | ❌ 不支持 |
| 集群支持 | ⚠️ 依赖Infinispan缓存共享 | ⚠️ 需自建分布式计数（Redis等） | ✅ 网关层天然支持 |
| 持久化可靠性 | ⚠️ 缓存重启丢失计数 | ✅ 完全可控（DB存储） | ✅ 网关配置持久化 |
| 管理界面 | ✅ Admin Console可视化 | ❌ 需自建后台 | ⚠️ 依赖网关管理面板 |

### 适用场景

1. **Web应用登录防护**：面向普通用户的B2C/B2B登录表单，防止撞库和暴力破解。
2. **API认证保护**：OAuth2 Resource Owner Password Grant场景下的端点保护。
3. **敏感系统多因素认证**：财务系统、管理后台、运维平台等高风险系统的强制OTP。
4. **合规审计要求**：等保三级、SOC2等标准明确要求密码复杂度策略和失败登录限制。
5. **企业内部身份管理**：统一员工密码策略，避免"一套系统一套规则"的混乱局面。

**不适用场景**：纯API Key认证的服务间调用（使用Client Credentials Grant，不涉及用户密码）；需要IP-用户关联深度分析的复杂风控场景（建议结合SIEM/UEBA系统）。

### 注意事项

- **密码策略不宜过度复杂**：NIST SP 800-63B明确指出，强制四种字符组合+频繁更换密码反而降低安全性。建议优先确保最小长度12位以上，搭配密码黑名单策略，弱化复杂字符组合要求。
- **TOTP依赖设备时间同步**：手机时间偏差超过Look Ahead Window时，用户输入正确验证码也会被拒绝。建议在用户绑定OTP后的界面上提示"请确保手机时间与网络时间同步"。
- **OTP密钥备份至关重要**：用户手机损坏或丢失后，OTP密钥也随之消失。管理员需要具备重置用户OTP凭证的权限（通过Admin Console或API删除凭证），并建立工单流程验证用户身份。
- **Brute Force Detection并非万能的DDoS防护**：如果攻击者使用分布式IP池进行慢速攻击，Keycloak的用户级锁定无法识别。建议在Keycloak前端部署WAF或API网关，在IP维度实现rate limiting，形成双层防护。
- **密码过期后用户可能被迫设置弱密码**：如果用户在密码即将过期时被迫修改，可能选择`OldPassword1→OldPassword2`这样的序列。建议在密码过期前几天通过邮件提醒用户，给予充分时间选择强密码。

### 常见踩坑经验

1. **问题**：三节点集群中某用户被锁定后，其他节点仍显示未锁定。**根因**：Infinispan的JGroups发现机制使用PING协议（基于UDP），在云环境中UDP组播被禁用导致节点发现失败，各节点维护独立的失败计数。**解决**：将JGroups切换为`TCPPING`或`JDBC_PING`协议，明确指定各节点IP地址。

2. **问题**：用户投诉"明明输入了正确密码，系统却说账户被锁定"。**根因**：用户的手机App自动填充了旧密码，导致后台默默累积失败计数。**解决**：在登录页面上显示剩余尝试次数（可通过Keycloak主题模板定制），并在锁定后提供明确的"联系管理员"指引。

3. **问题**：管理员删除用户OTP凭证后，用户重新绑定OTP时仍然失败。**根因**：仅删除Credentials中的OTP记录，但用户的Required Actions中仍保留"Configure OTP"标记，导致登录流程异常。**解决**：删除OTP凭证的同时，检查并清理用户的Required Actions列表。
