# 第4章：客户端管理——从注册到信任

## 1 项目背景

某中型互联网公司正在构建一套内部运营管理系统（OMS），技术栈选用Vue3作为前端框架、Spring Boot作为后端服务，并决定引入Keycloak作为统一身份认证平台。系统架构包含三个核心组件：面向员工的Web控制台（Vue3 SPA）、处理业务逻辑的REST API服务（Spring Boot）、以及独立运行的夜间数据同步定时任务服务。这三者需要与Keycloak对接完成统一认证与授权。

在Keycloak的客户端模型中，上述三个组件正好对应三种不同的客户端类型。Vue3前端是运行在浏览器中的单页应用，无法安全存储密钥，必须注册为Public客户端；Spring Boot后端部署在服务器端，可以安全保管Client Secret，应注册为Confidential客户端；定时任务服务没有人机交互，仅需服务间可信通信，应配置为Bearer-only客户端。此外，公司还有两套遗留系统使用SAML 2.0协议，也需要纳入统一管理。

在实际集成过程中，团队暴露出一系列典型问题：有人将前端应用配置成了Confidential客户端并暴露了Client Secret，导致任意调用者均可通过client_credentials流获取具有高权限的access_token；有人为Bearer-only客户端开启了Standard Flow，试图在浏览器中登录却发现始终报错；还有人配置Redirect URI时不加末尾斜杠，导致OAuth2回调验证失败，用户登录后页面白屏。SAML和OIDC客户端的配置项差异较大，混用时容易将SAML的Assertion Consumer Service URL误填到OIDC的Redirect URI字段。

这些问题的根源在于：对OAuth2/OIDC客户端类型的分层设计缺乏理解，对Redirect URI的严格校验机制不熟悉，对Client Secret的生命周期管理没有预案。本章将从三种客户端类型的本质区别出发，结合对话、实战和总结，帮助读者建立起"从注册到信任"的完整认知。

## 2 项目设计——剧本式交锋对话

**小胖**（挠头盯着Keycloak管理后台的Clients列表）：大师，我有点懵。为啥创建客户端的时候有三个选项没得选——Public、Confidential、Bearer-only？我看网上教程都是随便选一个，然后填Redirect URI就行了。

**大师**（微笑）：你用"不同身份拜访大楼"来理解。一栋写字楼有三种访客通道——

- **公共客户端（Public Client）**就像去大堂前台访客登记。你没工卡，保安只能核对你填的访客信息。谁都可以声称自己是来访者，因此前台不会给你任何机密凭证。浏览器里的SPA应用就是这样，代码全暴露在客户端，无法安全保管任何密钥。
- **机密客户端（Confidential Client）**就像正式员工刷卡。你有工卡（Client Secret），服务器验证通过后才放行。Spring Boot后端服务部署在我们控制的服务器上，可以安全存储Secret。
- **仅承载客户端（Bearer-only）**就像快递员扫码进闸。快递员不和前台交互，只凭系统下发的二维码通行。定时任务服务就是这种情况——没有人在浏览器里点"登录"，它只需要出示Service Account的Token即可。

**小白**（追问）：那为啥Public客户端不能存Client Secret？如果硬塞一个Secret进去会怎样？还有，Redirect URI为什么非要严格校验，少一个斜杠都不行？

**大师**：好问题。Public客户端不能存Secret是OAuth2规范（RFC 6749 §2.1）明确规定的——因为无法保证Secret的机密性。浏览器里的JavaScript代码任何人都能查看，Secret一旦写入前端代码就等于公开。攻击者拿到Secret后可以直接调用token端点，绕过所有用户认证。至于Redirect URI的严格校验——这是防御"开放重定向攻击"（Open Redirect Attack）的关键。假设攻击者构造如下链接：

```
https://sso.company.com/auth?redirect_uri=https://evil.com/phishing
```

如果Keycloak不严格匹配Redirect URI（包括路径、端口、scheme），授权码就会发给攻击者的服务器。末尾斜杠问题尤其隐蔽：`http://app.com/callback`和`http://app.com/callback/`在HTTP规范中是两个不同的URI，OAuth2要求精确匹配（RFC 3986），所以少一个斜杠真的会失败。

**小胖**（若有所思）：那我理解了三种类型的区别。但OIDC Client配置里还有一大堆选项——Client ID和Secret我懂，但Standard Flow、Implicit Flow、Direct Access Grants、Service Accounts Roles这些开关各自对应什么场景？

**大师**（展开白板画图）：我来做个完整的映射。

| 配置项 | 对应OAuth2流 | 适用场景 | 注意事项 |
|--------|-------------|----------|----------|
| Standard Flow | Authorization Code Grant（含PKCE） | SPA、传统Web应用 | 当前推荐的主流流程，Implicit Flow已被OAuth 2.1废弃 |
| Implicit Flow | Implicit Grant | 已废弃 | Token直接返回在URL Fragment中，安全风险高，建议仅维持旧系统兼容 |
| Direct Access Grants | Resource Owner Password Credentials | 受信任的一方应用、测试脚本 | 直接暴露用户凭证给客户端，生产环境慎用 |
| Service Accounts Roles | Client Credentials Grant | 服务间通信 | 用于Confidential客户端的机器认证 |
| Client Authentication | 客户端身份验证开关 | 区分Public/Confidential | On=Confidential（需Secret/JWT/X.509）；Off=Public |

**认证方式**方面，Client Secret只是最基础的方式。生产环境推荐使用Signed JWT或X.509证书进行客户端认证——前者使用非对称密钥签名JWT断言，后者使用TLS双向认证，都避免了Client Secret在网络上传输的风险。

**Client Scopes**是另一个关键概念。Scope决定access_token中携带的权限声明（claims）。默认的`openid`、`profile`、`email`是OIDC标准Scope。业务Scope如`order:read`需要通过Client Scopes→Mappers绑定到具体客户端，并在Token的`scope`字段中体现。

**SAML客户端**的配置则完全不同——没有Redirect URI，取而代之的是Assertion Consumer Service URL（ACS URL）。SAML断言通过HTTP POST发送到ACS URL，而非OAuth2的回调模式。混用时务必区分这两套术语体系。

——第一轮大师总结：客户端类型本质是"信任边界的划分"。浏览器端不可信→Public；服务器端可信→Confidential；纯服务无交互→Bearer-only。Redirect URI是OAuth2安全模型的第一道防线，必须精确配置。

---

**小胖**（第二轮）：大师，刚才提到Implicit Flow已被废弃，但我看很多旧教程还在用，它到底有什么问题？

**大师**：Implicit Flow的设计背景是早期的浏览器不支持跨域请求（CORS），所以Authorization Code Flow无法在SPA中实施——因为token端点不允许跨域调用。于是OAuth2工作组设计了Implicit Flow，让授权服务器直接在回调的URL Fragment中返回access_token，绕过token端点的跨域限制。

但它的致命缺陷是：access_token直接暴露在浏览器URL中。攻击者可以通过Referer头泄露、浏览器历史记录、JavaScript读取URL hash等方式窃取Token。而且Implicit Flow不返回refresh_token，导致用户体验也很差——Token过期就必须重新登录。OAuth 2.0安全最佳实践（BCP，RFC 8252 §8.3）明确建议废弃Implicit Flow，改用Authorization Code Flow + PKCE。PKCE（Proof Key for Code Exchange）通过在authorization_code交换阶段引入code_verifier/code_challenge对，彻底防止了授权码拦截攻击——即使授权码在回调URL中被截获，没有code_verifier也无法换取Token。

**小白**：那Consent Required这个开关呢？什么时候该打开？

**大师**：Consent Required控制是否在用户首次授权时展示"同意页面"——即列出该客户端请求的所有Scope，让用户显式确认。对于企业内部的SSO系统，通常关闭Consent（管理员已预授权）。但对于面向第三方开发者的平台（如Google OAuth），Consent Required是合规与透明度的重要保障。Fine Grain配置则允许对单个Client Scope设置独立的Consent策略，实现更精细的权限控制。

——第二轮大师总结：Implicit Flow→Authorization Code + PKCE是SPA安全的关键演进。Consent机制是"知情同意"原则的技术实现，企业内部可关闭，对外平台必须开启。

## 3 项目实战

### 环境准备

- 延续第2-3章搭建的Keycloak环境（Keycloak 25.x，Realm: demo-realm）
- Postman（用于API测试）
- curl命令行工具

### 步骤1：创建Confidential客户端（Spring Boot后端）

**操作路径**：Keycloak管理控制台 → 左侧菜单 Clients → Create client

填写配置如下：

| 字段 | 值 | 说明 |
|------|-----|------|
| Client ID | oms-backend | 后端服务标识 |
| Client Type | OpenID Connect | 使用OIDC协议 |
| Name | OMS Backend Service | 显示名称 |
| Client authentication | **On** | 启用客户端认证，即Confidential客户端 |
| Authorization | **On** | 启用细粒度授权服务 |

点击Next，继续配置：

| 字段 | 值 |
|------|-----|
| Root URL | http://localhost:8081 |
| Home URL | http://localhost:8081 |
| Valid redirect URIs | http://localhost:8081/* |
| Web Origins | http://localhost:8081 |

（截图描述：Client配置界面中，Client authentication开关为On状态，Authorization开关为On状态，Standard Flow保持On，Implicit Flow和Direct Access Grants保持Off，Service accounts roles自动选中。）

点击Save完成创建。

### 步骤2：获取并验证Client Secret

**操作路径**：oms-backend → Credentials 标签页 → Client Secret 区域 → 点击 Copy 图标

验证Client Credentials流程：

```bash
curl -X POST http://localhost:8080/realms/demo-realm/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=oms-backend" \
  -d "client_secret=YOUR_CLIENT_SECRET" \
  -d "grant_type=client_credentials"
```

运行结果示例：

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCIgOiAiSldUIiwia2lkIiA6IC...",
  "expires_in": 300,
  "refresh_expires_in": 0,
  "token_type": "Bearer",
  "not-before-policy": 0,
  "scope": "email profile"
}
```

注意：client_credentials模式返回的Token中`sub`字段对应的是Client ID（而不是用户ID），因为此流程不涉及用户认证。

### 步骤3：创建Public客户端（Vue3前端）

**操作路径**：Clients → Create client

| 字段 | 值 | 说明 |
|------|-----|------|
| Client ID | oms-frontend | SPA前端标识 |
| Client Type | OpenID Connect | |
| Client authentication | **Off** | Public客户端，不启用认证 |
| Authorization | Off | SPA无需授权服务 |

进入配置页：

| 字段 | 值 |
|------|-----|
| Standard Flow | **On** |
| Implicit Flow | **Off**（已废弃） |
| Direct Access Grants | **Off** |
| Valid redirect URIs | http://localhost:3000/* , http://localhost:3000/silent-check-sso.html |
| Valid post logout redirect URIs | http://localhost:3000 |
| Web Origins | http://localhost:3000 |

（截图描述：Public客户端配置中，Client authentication为Off，Standard Flow为On，Implicit Flow为Off。Redirect URIs包含通配符*匹配所有子路径，以及专门的silent-check-sso.html用于静默SSO检查。）

**关键说明**：
- `silent-check-sso.html`是keycloak-js适配器用来在iframe中静默刷新Token的专用页面，如果没有配置此URI，静默刷新将失败。
- Web Origins配置了`http://localhost:3000`以确保CORS允许该域的请求。漏配Web Origins将导致浏览器端的CORS预检请求（OPTIONS）被拒绝。

### 步骤4：创建Bearer-only客户端（定时任务服务）

Keycloak新版UI不再显式提供"Bearer-only"选项，正确做法是创建一个Confidential客户端，但关闭所有交互式认证流程，仅开启Service Accounts Roles。

| 字段 | 值 |
|------|-----|
| Client ID | oms-scheduler |
| Client Type | OpenID Connect |
| Client authentication | **On** |
| Authorization | Off |

进入配置页：

| 字段 | 值 |
|------|-----|
| Standard Flow | **Off** |
| Implicit Flow | **Off** |
| Direct Access Grants | **Off** |
| Service accounts roles | **On** |

Save后，该客户端将只能通过client_credentials方式获取Token——这正是"仅承载"（Bearer-only）的语义。

### 步骤5：配置Client Scopes

业务需要为OMS系统定义细粒度的订单操作权限。

**操作路径**：左侧菜单 Client Scopes → Create client scope

创建两个自定义Scope：

| Name | Type | Protocol | Display On Consent Screen |
|------|------|----------|---------------------------|
| order:read | Default | OpenID Connect | On |
| order:write | Optional | OpenID Connect | On |

接下来为Scope配置Mapper，使其能在Token中体现：

进入 `order:read` → Mappers → Add mapper → By Configuration → User Attribute

| 字段 | 值 |
|------|-----|
| Name | order-read-claim |
| User Attribute | order_read |
| Token Claim Name | order_read |
| Claim JSON Type | boolean |
| Add to ID token | Off |
| Add to access token | On |
| Add to userinfo | Off |

类似地配置`order:write`的Mapper。

最后将这两个Scope分配给客户端：oms-backend → Client Scopes → Add client scope → 选择order:read和order:write，分别设置为Default和Optional。

### 步骤6：验证三种客户端的认证行为差异

**测试1：Public客户端（Authorization Code + PKCE）**

SPA客户端只能通过浏览器重定向流程获取Token，不能直接调用token端点：

```bash
# 此操作会失败——Public客户端不能使用client_secret
curl -X POST http://localhost:8080/realms/demo-realm/protocol/openid-connect/token \
  -d "client_id=oms-frontend" \
  -d "grant_type=client_credentials"
# 返回: {"error":"unauthorized_client","error_description":"Public client not allowed to retrieve service account"}
```

SPA正确流程是：打开浏览器访问：
```
http://localhost:8080/realms/demo-realm/protocol/openid-connect/auth?client_id=oms-frontend&redirect_uri=http://localhost:3000/&response_type=code&scope=openid
```
用户完成登录后，浏览器携带authorization code回调到Redirect URI，前端再用code向token端点换取Token（此请求不需要Client Secret——Public客户端的token端点调用不验证Secret）。

**测试2：Confidential客户端（Client Credentials）**

```bash
curl -X POST http://localhost:8080/realms/demo-realm/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=oms-backend" \
  -d "client_secret=YOUR_CLIENT_SECRET" \
  -d "grant_type=client_credentials"

# 返回 access_token，scope包含 "order:read order:write email profile"
```

**测试3：Bearer-only客户端（Service Account）**

```bash
curl -X POST http://localhost:8080/realms/demo-realm/protocol/openid-connect/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "client_id=oms-scheduler" \
  -d "client_secret=YOUR_CLIENT_SECRET" \
  -d "grant_type=client_credentials"

# 同样返回access_token，但scope不包含用户相关的claims
# 该Token用于服务间API调用：Authorization: Bearer <token>
```

### 可能遇到的坑

**坑1：Redirect URI末尾斜杠**

配置`http://localhost:3000/callback`，但实际回调URL是`http://localhost:3000/callback/`（前端路由自动追加了斜杠）。Keycloak校验时严格匹配，导致`error=invalid_redirect_uri`。解决方案：要么配置两组URI（带斜杠和不带斜杠），要么使用通配符`http://localhost:3000/*`。

**坑2：Web Origins遗漏导致CORS错误**

浏览器控制台报错：`Access to XMLHttpRequest at '...' from origin 'http://localhost:3000' has been blocked by CORS policy`。原因是Keycloak不允许未配置的Origin发起跨域请求。检查Client设置的Web Origins字段，添加前端应用的完整Origin。

**坑3：Client Secret泄露应急处理**

如果Client Secret被提交到代码仓库或泄露：
1. 立即进入Client → Credentials → 点击"Regenerate"按钮使旧Secret失效
2. 更新所有使用该Secret的服务配置
3. 审计Token发放记录（Events → Admin events）排查异常访问
4. 考虑迁移到JWT签名认证或X.509证书认证

**坑4：生产环境Client Secret硬编码**

前端代码中绝不应出现Client Secret。Confidential客户端的Secret应通过环境变量、Kubernetes Secret或Vault注入。对于无法避免在前端使用Secret的场景，该应用应注册为Public客户端并通过PKCE保证安全性。

### 测试验证完整流程

利用Postman的Token获取功能验证权限边界：

1. **oms-frontend（Public）**：使用Authorization Code（With PKCE）流程，需用户交互，获取的Token包含用户身份信息。
2. **oms-backend（Confidential）**：使用Client Credentials流程，无需用户交互，Token中的`sub`为Client ID。
3. **oms-scheduler（Bearer-only）**：同样使用Client Credentials流程，但Token的scope受限于分配给Service Account的角色。

三者权限边界清晰：Public客户端的Token代表"哪个用户在用哪个应用"；Confidential客户端的Token代表"哪个应用在请求"；Bearer-only客户端的Token仅用于机器间API调用，与用户无关。

## 4 项目总结

### 客户端类型对比

| 特性 | Public | Confidential | Bearer-only |
|------|--------|-------------|-------------|
| 适用架构 | SPA、移动App、桌面应用 | Web后端、API网关 | 微服务间调用、定时任务 |
| 密钥保管 | 不存储Secret | 安全存储Secret/JWT Key | 存储Secret/Kerberos Keytab |
| 认证方式 | PKCE强制 | Client Secret / JWT / X.509 | Client Secret / JWT |
| 用户交互 | 必须有（授权码流） | 可选（也可用client_credentials） | 无（仅Service Account） |
| Token类型 | 用户委托Token | 用户委托或应用Token | 应用Token |
| 安全核心 | PKCE防授权码拦截 | 密钥保管+轮换 | 密钥保管+最小权限 |

### 适用场景

- **SPA + API**：前端Public（PKCE），后端Confidential（Client Secret），各司其职。
- **Server-to-Server**：两个后端服务间通信，双方均为Confidential客户端，使用client_credentials流。
- **移动App**：应注册为Public客户端并使用PKCE。虽然App可以比浏览器更安全地存储数据（Keychain/Keystore），但OAuth2最佳实践（RFC 8252）仍将其视为Public，因为无法确保所有安装环境的安全。
- **IoT设备**：设备本身为Public客户端，通过Device Authorization Grant（RFC 8628）在另一设备上完成用户授权。

### 注意事项

1. **Client Secret轮换策略**：建议每90天轮换一次Client Secret。轮换过程中注意新旧Secret并行窗口——Keycloak的Regenerate操作会立即失效旧Secret，建议在业务低峰期操作。
2. **PKCE已是Public客户端的强制要求**：Keycloak 17+版本默认要求Public客户端使用PKCE。不要在非HTTPS环境下禁用PKCE。
3. **Consent页面用户体验**：如果开启Consent Required，自定义Consent页面文案和样式以降低用户困惑。

### 常见踩坑

- **Redirect URI配置错误导致登录后白屏**：最常见的问题是scheme不匹配（http vs https）、端口遗漏、路径末尾斜杠。排查方法：查看Keycloak日志中的`invalid_redirect_uri`错误，对比配置值与实际回调URL的每个字符。
- **Web Origins遗漏导致CORS拦截**：浏览器开发者工具→Network面板→查看OPTIONS预检请求的响应头中是否缺少`Access-Control-Allow-Origin`。
- **授权码被多次使用**：授权码（code）是一次性的，重复使用会触发`invalid_grant`错误。前端不要在失败后直接用旧code重试。

### 思考题

**为什么Confidential客户端同时支持authorization_code和client_credentials两种授权方式？各适用于什么场景？**

authorization_code用于"用户委托授权"场景——后端代表某个已登录用户执行操作。例如用户通过前端SPA登录后，后端API收到带有用户Token的请求，Token中的`sub`是该用户的ID。client_credentials用于"应用自主授权"场景——后端应用以自己的身份执行操作，不涉及任何用户。例如定时任务从消息队列消费数据、后台数据迁移脚本。一个Confidential客户端可以同时在这两种模式下工作，取决于调用token端点时传入的grant_type参数。

**移动端App应该注册为Public还是Confidential客户端？为什么？**

应当注册为Public客户端。核心原因是一个技术现实：移动App的安装环境不可控。即使我们将Client Secret加密存储在App内部，攻击者可以通过逆向工程、越狱/Root设备、代理工具（如Frida、Burp Suite）提取出Secret。OAuth 2.0 for Native Apps（RFC 8252）明确规定原生应用必须作为Public客户端，并通过PKCE保护Authorization Code Flow。此外，移动端应使用系统的安全浏览器（Chrome Custom Tabs / SafariViewController）完成认证，而非App内嵌的WebView——前者可以访问系统级Cookie存储，避免用户重复登录。
