# 第28章：微服务架构深度集成

## 1 项目背景

某电商中台经过两年的技术演进，从单体架构裂变为50+微服务的分布式体系，技术栈统一为Spring Cloud Gateway + Spring Boot 3 + Keycloak。当前架构中，Keycloak作为统一认证中心负责签发JWT，但认证鉴权的执行层面呈现"战国割据"态势——每个微服务各自为战：各自依赖`spring-boot-starter-oauth2-resource-server`独立校验JWT签名，各自配置`JwtAuthenticationConverter`提取角色权限，各自通过`NimbusJwtDecoder`向Keycloak的JWKS端点拉取公钥。

表面上看，每个微服务都做到了"自包含安全校验"，架构似乎健壮。但随着微服务数量从10个增长到50+，四个核心痛点逐渐浮出水面。

**痛点一：JWKS端点成为瓶颈。** 每个微服务在启动时和每5分钟定期向`/realms/demo-realm/protocol/openid-connect/certs`端点拉取公钥集。50个微服务意味着每5分钟产生250次JWKS请求（加上启动时的一次性请求高峰），在生产环境中这已经导致Keycloak的CPU间歇性飙高。当Keycloak自身也需要横向扩展时，JWKS缓存策略的一致性又成了新问题——Pod A拉取的公钥和Pod B拉取的公钥在密钥轮换时可能出现短暂不一致。

**痛点二：安全配置不一致。** 由于每个微服务由不同的小团队维护，Token校验的严格程度参差不齐。有的服务在校验时只验证了签名却遗漏了`exp`过期检查（因为依赖的库版本差异导致默认行为不同）；有的服务配置了`aud`校验而有的没有；更有甚者，某团队为了"方便调试"在开发环境关掉了签名校验，代码却不小心合入了生产分支。安全边界模糊——攻击者只要攻破校验最弱的那个微服务，就能在整个内网横向移动。

**痛点三：全链路追踪断裂。** 当用户请求流经Gateway → UserService → OrderService → InventoryService时，每跳都需要在日志中关联同一个TraceID。但当前实现中，TraceID在微服务边界会"断掉"——Gateway生成的TraceID通过HTTP Header传递给UserService，UserService调用OrderService时使用的Feign客户端默认不携带任何自定义Header，导致OrderService拿不到TraceID。运维排查跨服务问题时，需要在四个不同的日志系统中手动拼接时间线。

**痛点四：服务间认证方案混乱。** 当OrderService需要调用NotificationService发送订单确认时，用什么凭证？团队A用的是client_credentials模式——OrderService以"自己"的身份获取一个Service Token然后调用；团队B直接在配置文件中写死了一个共享API Key并通过HTTP Header传递；团队C则把前端的用户Token透传到了下游——这带来了"Token权限放大"的严重安全风险——前端用户Token中包含的`platform_admin`角色权限被传播到了用户本不该访问的服务。

这些痛点的根源在于一个架构层面的模糊地带：**API Gateway与微服务之间的认证职责该如何划分？谁负责校验Token合法性？谁负责判断角色权限？服务间调用时Token该以什么形式传递？**

## 2 项目设计——剧本式交锋对话

**小胖**（抱着笔记本冲进茶水间，手里还捏着一包薯片）：大师！我昨天突然想到一个问题——咱公司架构里，API Gateway不是已经做了JWT校验了吗？那为啥每个微服务还要再做一遍？这不是重复劳动吗？这就像我们小区——大门口有保安查门禁卡，进了小区之后每个单元楼还有自己的门禁！那我要是小区业主，我都过了大门口保安那关了，单元楼凭啥不直接信任我？

**大师**（放下咖啡）：小胖你"小区门禁"这个比喻非常到位，但回答你这个问题，恰恰要把你比喻里的逻辑漏洞揪出来。你说的"小区大门保安"和"单元楼门禁"职责不同——保安只验证你的身份（你是不是这个小区的业主），单元楼门禁验证你的权限（你是否有权限进这栋楼）。对应到技术术语上，**Gateway做的是"认证"（Authentication）——你是谁；微服务做的是"鉴权"（Authorization）——你能做什么**。

**小白**（合上手中的《Designing Data-Intensive Applications》）：大师，但Spring Cloud Gateway本身也有`hasRole("admin")`这种鉴权能力啊，Gateway也能做部分鉴权吧？如果我把所有鉴权逻辑都放在Gateway，微服务关闭Security配置，只信任Gateway传递的X-Auth-User和X-Auth-Roles头，这样做有什么问题？

**大师**：小白的方案正是业界讨论最多的三种微服务认证架构之一，我展开对比。

**方案A：Gateway统一认证+微服务无鉴权（Trust Gateway模式）。** Gateway校验JWT后，将用户信息（username、roles）以明文Header注入到下游请求中。微服务关闭Security，直接用`request.getHeader("X-Auth-User")`获取用户身份，无条件信任Gateway提供的信息。优点是微服务实现极简——零安全依赖、零Token校验开销。但**致命缺陷是内网信任边界坍塌**——如果某个微服务或某个中间件服务（如Kong、Envoy）被人入侵，攻击者可以直接构造`X-Auth-User: admin`和`X-Auth-Roles: platform_admin`的请求绕过所有安全校验。在零信任架构中，这是不可接受的。

**方案B：Gateway认证+微服务独立鉴权（本章核心方案）。** Gateway校验JWT签名和有效期——这是"入口认证"，快速拒绝非法Token，避免无效请求流入内网。Token本身通过`TokenRelay`过滤器透传给下游微服务。微服务仍然独立校验JWT签名，不做Trust Gateway假设——但此时微服务的校验开销大幅降低（公钥已缓存，本地验签仅微秒级）。这种方式既保持了零信任安全模型，又通过Gateway统一了入口校验策略——特别是角色权限的Gatekeeping（如admin接口在Gateway层即被拦截）。

**方案C：BFF（Backend for Frontend）认证。** 为每种前端类型（Web、Mobile、IoT）部署独立的BFF服务，BFF持有Token、管理Session，下游微服务只接受BFF签发的内部Token。适用于前端类型多变的场景，但增加了架构复杂度。

> **大师技术映射**：方案A → 小区大门有保安，进楼不需要门禁卡——方便但不安全。方案B → 大门保安初步核验，每栋楼也刷卡——安全但需协调。方案C → 主人在门口迎接客人，带客人进入每栋楼——体验好但主人要全程陪同。

---

**小胖**（把薯片袋子推到一边）：好，那选方案B。但还有个大问题——前端用户登录后拿到Token，调用`/api/orders`走到Gateway，Gateway转发到OrderService，OrderService需要调用InventoryService查库存。这个链路上Token怎么传递？前端Token能直接传给InventoryService吗？

**大师**：这是微服务安全中最经典的Token Relay问题。前端→Gateway→Service A→Service B这条链路上，有两种传递模型。

**模型一：Token Relay（用户令牌透传）。** 前端的access_token沿调用链一路透传——Gateway通过`TokenRelay`过滤器传递给OrderService，OrderService通过Feign拦截器传递给InventoryService。优势是下游服务可以获得**完整的用户上下文**（userId、roles、attributes），适合需要用户上下文的业务操作（如"查询我的订单"——InventoryService需要知道是哪个用户）。但风险是你提到过的**"上帝Token"问题**——前端Token中可能包含比当前操作所需更大的权限集（如Token里既有`order:read`也有`admin:delete-all`），透传到下游时权限被放大。

**模型二：Client Credentials（服务间独立凭证）。** OrderService不以用户Token调用下游，而是用自己的Client ID/Secret去Keycloak获取一个新的Service Token（grant_type=client_credentials），该Token的scope被精确限定为本次服务间调用所需的最小权限（如仅含`inventory:read`）。这符合**最小权限原则**——即使下游服务被攻破，攻击者拿到的Service Token只拥有一组极小的scope。

**小白**：两种模型怎么选择？

**大师**：判断标准只有一个——**下游服务是否需要用户上下文**。如果InventoryService需要知道"谁在查库存"（如做用户画像、风控分析），用Token Relay；如果NotificationService只需要"发送一封邮件给某用户"（用户ID可以作为方法参数传入），用Client Credentials。实际生产中，一条调用链上可能混合两种模式——比如OrderService→InventoryService用Token Relay（需要用户上下文判断VIP用户预留库存），OrderService→NotificationService用Client Credentials（发邮件不需要用户Token的读写权限）。

> **大师技术映射**：Token Relay → 你把工牌借给同事去帮你刷卡吃饭——食堂知道你同事是替你来的，同事能进你所有能进的门（风险大）。Client Credentials → 公司给保洁部门统一办了一张"工作区域通行卡"，只能进规定的楼层（权限最小化）。

---

**小胖**（第二轮，从裤兜里掏出保温杯）：大师，Gateway那个全局Filter的`getOrder()`返回值影响什么？我有一次把TraceID Filter的顺序写成了`Ordered.HIGHEST_PRECEDENCE + 100`，结果Authorization头在到达下游时是空的！换个顺序就好了——这是什么玄学？

**大师**：这不是玄学，这是Spring Cloud Gateway的**Filter执行链顺序**。Gateway的Filter执行分为三个阶段。`Ordered.HIGHEST_PRECEDENCE`（值为Integer.MIN_VALUE + 某个偏移）是最先执行的。`TokenRelay`过滤器（由`TokenRelayGatewayFilterFactory`生成）在请求发送到下游之前从当前SecurityContext中提取Token并添加到HTTP Header。如果你的TraceID Filter的order值大于`TokenRelay`的order，它会在`TokenRelay`之前执行——那时SecurityContext可能还没被Spring Security的认证Filter填充完毕，你通过`exchange.getPrincipal()`拿到的Authentication为null，下游自然收到空的Authorization头。

Gateway的Filter Chain中关键组件顺序如下：`NettyRoutingFilter`（实际发送HTTP请求到下游，order=`LOWEST_PRECEDENCE`）→ `TokenRelay`（order=？在ReactiveSecurityContextHolder写入后执行）→ 你的自定义Filter。规范做法是：`TraceIDFilter`应该设置order ≈ `Ordered.HIGHEST_PRECEDENCE + 10`，确保在Spring Security认证Filter之后、但在`TokenRelay`之前执行，这样既能拿到用户信息又能确保TraceID先于业务Header注入。

> **大师技术映射**：Filter Chain的顺序 → 就像快递分拣流水线——贴面单（TraceID）必须在称重（TokenRelay）和打包（路由转发）之前完成。如果把贴面单的机器放在打包机之后，快递就已经出厂了。

**大师**（在白板上画了一张图）：最后总结今天的架构核心。在Spring Cloud Gateway + Keycloak的微服务认证体系中，四层安全防线缺一不可：第一层，Gateway入口校验JWT签名+全局角色Gatekeeping；第二层，网关Token Relay透传用户上下文；第三层，微服务独立验签+本地鉴权（零信任）；第四层，服务间调用按需选择Token Relay或Client Credentials。每一层各司其职，通过W3C TraceContext实现跨服务追踪——TraceID从Gateway注入后，随Token一起沿调用链流动直到最终响应返回。

## 3 项目实战

### 环境准备

- JDK 17+
- Spring Boot 3.2.x + Spring Cloud 2023.0.x
- Keycloak 24.x（已配置Realm `demo-realm`，含客户端`api-gateway`（Client Secret模式）、`order-service`、`user-service`、`notification-service-client`）
- Redis（限流过滤器需要）
- 启动顺序：Keycloak → Nacos/Consul（服务注册）→ Gateway → 下游微服务

### 步骤1：Spring Cloud Gateway配置

**目标**：配置Gateway路由规则，启用TokenRelay将用户Token透传给下游，并为关键路由配置限流。

```yaml
# gateway/src/main/resources/application.yml
server:
  port: 8080

spring:
  application:
    name: api-gateway
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: lb://user-service
          predicates:
            - Path=/api/users/**
          filters:
            - TokenRelay=
            - StripPrefix=1
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - TokenRelay=
            - StripPrefix=1
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 10
                redis-rate-limiter.burstCapacity: 20
                key-resolver: "#{@principalKeyResolver}"
        - id: public-resources
          uri: lb://user-service
          predicates:
            - Path=/api/public/**
          filters:
            - StripPrefix=1
      default-filters:
        - DedupeResponseHeader=Access-Control-Allow-Credentials Access-Control-Allow-Origin
  security:
    oauth2:
      client:
        registration:
          keycloak:
            client-id: api-gateway
            client-secret: ${GATEWAY_CLIENT_SECRET}
            authorization-grant-type: authorization_code
            scope: openid,profile,email
        provider:
          keycloak:
            issuer-uri: http://localhost:8081/realms/demo-realm
      resourceserver:
        jwt:
          issuer-uri: http://localhost:8081/realms/demo-realm
          jwk-set-uri: http://localhost:8081/realms/demo-realm/protocol/openid-connect/certs
```

> **说明**：当启用TokenRelay时，Gateway需要一个`oauth2.client.registration`（即使它主要作为Resource Server），因为TokenRelay内部使用`OAuth2AuthorizedClient`机制管理Token透传。`jwk-set-uri`显式指定可以绕过OIDC Discovery，减少一次网络往返。

### 步骤2：Gateway安全配置——JWT校验+角色鉴权

**目标**：在Gateway层统一校验JWT合法性并执行粗粒度角色Gatekeeping，拒绝无效请求进入内网。

```java
// gateway/src/main/java/com/ecommerce/gateway/security/GatewaySecurityConfig.java
package com.ecommerce.gateway.security;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.core.convert.converter.Converter;
import org.springframework.security.authentication.AbstractAuthenticationToken;
import org.springframework.security.config.annotation.web.reactive.EnableWebFluxSecurity;
import org.springframework.security.config.web.server.ServerHttpSecurity;
import org.springframework.security.oauth2.jwt.Jwt;
import org.springframework.security.oauth2.jwt.ReactiveJwtDecoder;
import org.springframework.security.oauth2.jwt.NimbusReactiveJwtDecoder;
import org.springframework.security.oauth2.server.resource.authentication.JwtAuthenticationConverter;
import org.springframework.security.oauth2.server.resource.authentication.JwtGrantedAuthoritiesConverter;
import org.springframework.security.oauth2.server.resource.authentication.ReactiveJwtAuthenticationConverterAdapter;
import org.springframework.security.web.server.SecurityWebFilterChain;
import org.springframework.security.config.Customizer;

@Configuration
@EnableWebFluxSecurity
public class GatewaySecurityConfig {

    private final String jwkSetUri;

    public GatewaySecurityConfig(
            @org.springframework.beans.factory.annotation.Value(
                "${spring.security.oauth2.resourceserver.jwt.jwk-set-uri}")
            String jwkSetUri) {
        this.jwkSetUri = jwkSetUri;
    }

    @Bean
    public SecurityWebFilterChain securityFilterChain(ServerHttpSecurity http) {
        http
            .authorizeExchange(exchanges -> exchanges
                .pathMatchers("/api/public/**").permitAll()
                .pathMatchers("/api/admin/**").hasRole("platform_admin")
                .pathMatchers("/api/orders/**").hasAnyRole("user", "admin")
                .anyExchange().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt
                    .jwtAuthenticationConverter(jwtAuthConverter())
                    .jwtDecoder(reactiveJwtDecoder())
                )
            )
            .oauth2Login(Customizer.withDefaults())
            .csrf(ServerHttpSecurity.CsrfSpec::disable);

        return http.build();
    }

    @Bean
    public ReactiveJwtDecoder reactiveJwtDecoder() {
        return NimbusReactiveJwtDecoder.withJwkSetUri(jwkSetUri).build();
    }

    @Bean
    public Converter<Jwt, AbstractAuthenticationToken> jwtAuthConverter() {
        JwtGrantedAuthoritiesConverter grantedAuthoritiesConverter =
            new JwtGrantedAuthoritiesConverter();
        grantedAuthoritiesConverter.setAuthorityPrefix("ROLE_");
        grantedAuthoritiesConverter.setAuthoritiesClaimName("realm_access.roles");

        JwtAuthenticationConverter jwtAuthConverter = new JwtAuthenticationConverter();
        jwtAuthConverter.setJwtGrantedAuthoritiesConverter(grantedAuthoritiesConverter);

        return new ReactiveJwtAuthenticationConverterAdapter(jwtAuthConverter);
    }
}
```

### 步骤3：网关注入TraceID到下游请求

**目标**：在Gateway层从上游请求提取或生成TraceID，注入到下游请求头和Gateway日志MDC，确保全链路追踪标识一致。

```java
// gateway/src/main/java/com/ecommerce/gateway/filter/TraceIDFilter.java
package com.ecommerce.gateway.filter;

import org.slf4j.MDC;
import org.springframework.cloud.gateway.filter.GatewayFilterChain;
import org.springframework.cloud.gateway.filter.GlobalFilter;
import org.springframework.core.Ordered;
import org.springframework.http.server.reactive.ServerHttpRequest;
import org.springframework.stereotype.Component;
import org.springframework.web.server.ServerWebExchange;
import reactor.core.publisher.Mono;

import java.security.Principal;
import java.util.Optional;
import java.util.UUID;

@Component
public class TraceIDFilter implements GlobalFilter, Ordered {

    private static final String TRACE_ID_HEADER = "X-Trace-Id";
    private static final String AUTH_USER_HEADER = "X-Auth-User";

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 1. 从上游提取或生成TraceID
        String traceId = exchange.getRequest().getHeaders()
            .getFirst(TRACE_ID_HEADER);
        if (traceId == null || traceId.isBlank()) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }

        // 2. 提取当前认证用户的用户名
        String username = exchange.getPrincipal()
            .map(Principal::getName)
            .orElse("anonymous");

        // 3. 构造增强的下游请求
        ServerHttpRequest mutatedRequest = exchange.getRequest().mutate()
            .header(TRACE_ID_HEADER, traceId)
            .header(AUTH_USER_HEADER, username)
            .build();

        // 4. 注入到MDC（用于Gateway自身的日志）
        MDC.put("traceId", traceId);

        // 5. 将TraceID加入响应头（便于前端或调用方提取）
        exchange.getResponse().getHeaders()
            .add(TRACE_ID_HEADER, traceId);

        return chain.filter(
            exchange.mutate().request(mutatedRequest).build()
        ).doFinally(signalType -> MDC.clear());
    }

    @Override
    public int getOrder() {
        // 必须在ReactiveSecurityContextHolder填充之后执行
        // 确保能通过exchange.getPrincipal()获取用户信息
        return Ordered.HIGHEST_PRECEDENCE + 10;
    }
}
```

### 步骤4：下游微服务安全配置（信任Gateway传递的Token）

**目标**：下游微服务作为独立Resource Server，本地验签JWT并提取角色，同时捕获Gateway注入的TraceID。

```java
// order-service/src/main/java/com/ecommerce/order/security/ServiceSecurityConfig.java
package com.ecommerce.order.security;

import jakarta.servlet.Filter;
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.ServletRequest;
import jakarta.servlet.ServletResponse;
import jakarta.servlet.http.HttpServletRequest;
import org.slf4j.MDC;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.security.config.annotation.method.configuration.EnableMethodSecurity;
import org.springframework.security.config.annotation.web.builders.HttpSecurity;
import org.springframework.security.config.annotation.web.configuration.EnableWebSecurity;
import org.springframework.security.config.http.SessionCreationPolicy;
import org.springframework.security.oauth2.server.resource.authentication.JwtAuthenticationConverter;
import org.springframework.security.oauth2.server.resource.authentication.JwtGrantedAuthoritiesConverter;
import org.springframework.security.web.SecurityFilterChain;

import java.io.IOException;

@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class ServiceSecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/actuator/health").permitAll()
                .requestMatchers("/api/**/admin/**").hasRole("platform_admin")
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt
                    .jwtAuthenticationConverter(jwtAuthConverter())
                )
            )
            // 如果配置了OAuth2 Client，显式指定Resource Server（JWT），
            // 避免Spring Security决策冲突
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            );

        return http.build();
    }

    @Bean
    public JwtAuthenticationConverter jwtAuthConverter() {
        JwtGrantedAuthoritiesConverter grantedConverter =
            new JwtGrantedAuthoritiesConverter();
        grantedConverter.setAuthorityPrefix("ROLE_");
        grantedConverter.setAuthoritiesClaimName("realm_access.roles");

        JwtAuthenticationConverter converter = new JwtAuthenticationConverter();
        converter.setJwtGrantedAuthoritiesConverter(grantedConverter);
        return converter;
    }

    @Bean
    public Filter traceIdFilter() {
        return new Filter() {
            @Override
            public void doFilter(ServletRequest request,
                                 ServletResponse response,
                                 FilterChain chain)
                    throws IOException, ServletException {
                String traceId = ((HttpServletRequest) request)
                    .getHeader("X-Trace-Id");
                if (traceId != null) {
                    MDC.put("traceId", traceId);
                }
                try {
                    chain.doFilter(request, response);
                } finally {
                    MDC.remove("traceId");
                }
            }
        };
    }
}
```

### 步骤5：服务间调用——Feign Token Relay拦截器

**目标**：当微服务需要以"用户身份"调用下游时（需要用户上下文），通过Feign RequestInterceptor自动从一个微服务提取JWT并透传给下游。

```java
// order-service/src/main/java/com/ecommerce/order/client/OAuth2FeignInterceptor.java
package com.ecommerce.order.client;

import feign.RequestInterceptor;
import feign.RequestTemplate;
import org.slf4j.MDC;
import org.springframework.security.core.Authentication;
import org.springframework.security.core.context.SecurityContextHolder;
import org.springframework.security.oauth2.jwt.Jwt;
import org.springframework.security.oauth2.server.resource.authentication.JwtAuthenticationToken;
import org.springframework.stereotype.Component;

@Component
public class OAuth2FeignInterceptor implements RequestInterceptor {

    @Override
    public void apply(RequestTemplate template) {
        // 1. 从Spring Security上下文获取当前用户Token
        Authentication auth = SecurityContextHolder.getContext()
            .getAuthentication();
        if (auth instanceof JwtAuthenticationToken jwtAuth) {
            Jwt jwt = jwtAuth.getToken();
            template.header("Authorization",
                "Bearer " + jwt.getTokenValue());
        }

        // 2. 透传TraceID（保持全链路追踪延续）
        String traceId = MDC.get("traceId");
        if (traceId != null) {
            template.header("X-Trace-Id", traceId);
        }
    }
}
```

### 步骤6：服务间调用——Client Credentials模式

**目标**：当微服务不需要用户上下文时，使用Keycloak的Client Credentials Grant获取最小权限的Service Token。

```java
// notification-service/src/main/java/com/ecommerce/notification/security/ClientCredentialsTokenProvider.java
package com.ecommerce.notification.security;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.security.oauth2.client.AuthorizedClientServiceOAuth2AuthorizedClientManager;
import org.springframework.security.oauth2.client.OAuth2AuthorizeRequest;
import org.springframework.security.oauth2.client.OAuth2AuthorizedClient;
import org.springframework.security.oauth2.client.OAuth2AuthorizedClientManager;
import org.springframework.security.oauth2.client.OAuth2AuthorizedClientProvider;
import org.springframework.security.oauth2.client.OAuth2AuthorizedClientProviderBuilder;
import org.springframework.security.oauth2.client.registration.ClientRegistrationRepository;
import org.springframework.security.oauth2.client.web.DefaultOAuth2AuthorizedClientManager;
import org.springframework.security.oauth2.client.web.OAuth2AuthorizedClientRepository;
import org.springframework.stereotype.Component;

@Component
public class ClientCredentialsTokenProvider {

    private final OAuth2AuthorizedClientManager authorizedClientManager;

    public ClientCredentialsTokenProvider(
            ClientRegistrationRepository clientRegistrationRepository,
            OAuth2AuthorizedClientRepository authorizedClientRepository) {

        // 构建Client Credentials模式的AuthorizedClientManager
        DefaultOAuth2AuthorizedClientManager clientManager =
            new DefaultOAuth2AuthorizedClientManager(
                clientRegistrationRepository,
                authorizedClientRepository
            );

        OAuth2AuthorizedClientProvider authorizedClientProvider =
            OAuth2AuthorizedClientProviderBuilder.builder()
                .clientCredentials()
                .build();

        clientManager.setAuthorizedClientProvider(authorizedClientProvider);
        this.authorizedClientManager = clientManager;
    }

    public String getServiceToken() {
        OAuth2AuthorizeRequest authorizeRequest = OAuth2AuthorizeRequest
            .withClientRegistrationId("notification-client")
            .principal("service-account")
            .build();

        OAuth2AuthorizedClient authorizedClient =
            authorizedClientManager.authorize(authorizeRequest);

        if (authorizedClient == null ||
            authorizedClient.getAccessToken() == null) {
            throw new IllegalStateException(
                "Failed to obtain client_credentials token");
        }

        return authorizedClient.getAccessToken().getTokenValue();
    }
}
```

### 可能遇到的坑

**1. TokenRelay过滤器顺序错误导致下游收到空的Authorization头。** 根本原因是`TraceIDFilter`或自定义GlobalFilter的`getOrder()`返回值不合理——如果优先级低于`TokenRelayGatewayFilterFactory`的安排，SecurityContext中的Authentication尚未被Spring Security写入，导致Token透传失败。解决方案：将自定义Filter的order设为`Ordered.HIGHEST_PRECEDENCE + 10`。

**2. 微服务间循环调用导致Token过期。** 如果OrderService→InventoryService→PromotionService→OrderService形成循环调用链，Token在链路上消耗的时间可能超过其`exp`有效期。Keycloak支持`Token Exchange`（RFC 8693）来在传播Token时刷新其有效期，但建议在架构层面避免过深的同步调用链——深调用链考虑用异步消息（Kafka/RabbitMQ）解耦。

**3. Gateway与微服务JWT校验时间不一致。** Gateway的NimbusReactiveJwtDecoder和微服务的NimbusJwtDecoder是各自独立的时间源（默认取JVM系统时间）。如果Gateway和微服务未做NTP时间同步，可能出现Gateway认为Token有效而微服务因`exp`不满足而拒绝的情况。解决方案：所有服器启用NTP同步，并统一配置`NimbusJwtDecoder`的`jwtValidator`允许60秒以内的时钟偏差（clockSkew）。

**4. Feign拦截器中SecurityContext在异步线程中丢失。** Spring Security的`SecurityContextHolder`默认使用`ThreadLocal`存储，Feign如果通过Hystrix/Resilience4j的线程池隔离执行，子线程无法访问主线程的SecurityContext。解决方案：在主线程中将Token提取为final变量传递给Feign拦截器构造方法；或在拦截器中显式设置`SecurityContextHolder.setStrategyName(SecurityContextHolder.MODE_INHERITABLETHREADLOCAL)`——注意后者可能导致线程池线程之间的上下文污染。

**5. Gateway路由优先级与Security路径匹配冲突。** Gateway的路由匹配和Security的`pathMatchers`是两个独立的规则引擎。如果Gateway路由配了`Path=/api/**`通配但Security的`authorizeExchange`中未显式声明该路径，Spring Security会应用`anyExchange().authenticated()`，导致公开端点意外被保护。建议Gateway和Security的路径规则保持一致，用同一份YAML锚点别名管理。

## 4 项目总结

### 方案对比

| 维度 | Gateway统一认证（本章） | 各服务独立认证 | Sidecar代理认证（Istio） |
|------|----------------------|--------------|------------------------|
| 安全模型 | 入口校验+微服务零信任验签 | 纯零信任，每服独立 | Sidecar拦截，微服务无感 |
| 性能开销 | 中（Gateway验签+每服本地验签） | 高（每服Nimbus轮询JWKS） | 低（Envoy本地验签） |
| 架构复杂度 | 中（需理解Gateway Filter顺序） | 低（每服独立但配置散乱） | 高（需Service Mesh体系） |
| 运维成本 | 中（Gateway集中管理+各服自管） | 高（50+份配置需统一） | 低（声明式配置，一键下发） |
| Token Relay | 原生支持（TokenRelay过滤器） | 需手动Feign拦截器 | 自动（Sidecar透明代理） |
| 技术栈耦合 | 紧耦合Spring Cloud | 仅依赖Spring Security | 与Spring无关，语言无关 |
| 故障隔离 | Gateway单点风险（需高可用） | 无单点 | Sidecar故障隔离 |

### 适用场景

1. **Spring Cloud微服务体系**：Gateway作为统一入口，各服务是Resource Server，适合20~100个服务的中等规模集群。
2. **前后端分离+API Gateway架构**：前端Token经Gateway透传，下游微服务验签执行鉴权。
3. **需要全链路追踪的分布式系统**：TraceID从Gateway注入后沿调用链传播，适合排查跨服务故障。
4. **渐进式迁移遗留系统**：Gateway先做统一认证，下游服务逐步从"Trust Gateway"升级到"独立验签"。

### 不适用场景

- **极致性能要求（P99 < 1ms）**：Gateway层JWT验签+微服务再次验签的双重开销对超低延迟场景是浪费，应考虑Sidecar方案（Istio Envoy）在L7层卸除Token校验。
- **不基于HTTP的内部通信**：如果微服务间使用gRPC或消息队列通信，Gateway的TokenRelay机制无法覆盖——gRPC需通过Interceptor实现类似的Token传播，消息队列则需要将Token或用户Claims编码到消息元数据中。

### 注意事项

1. **Gateway Token校验性能**：本地JWT验签（使用缓存的公钥）仅需微秒级，不会成为性能瓶颈。但如果错误配置为Introspection模式（每次请求调Keycloak），延迟将增加20-50ms，且Keycloak在加载下可能不可用。
2. **Token过期时差**：access_token建议设置为5-15分钟，确保在最长调用链时间（例如500ms × 5跳 = 2.5秒）内Token不会过期。Gateway和各微服务的`NimbusJwtDecoder`可以配置clockSkew来容忍轻微时钟偏差。
3. **Client Credentials的Scope粒度**：每个服务间调用Client的Scope应遵循最小权限原则——如`notification-client`只有`notification:send`，`report-client`只有`report:read`。在Keycloak中通过Authorization Services为此类Service Account Client的Scope做精细化授权。
4. **Gateway高可用**：Gateway成为单点后必须配置多实例+负载均衡（至少2个实例），并使用Redis等分布式存储共享限流计数器。

### 常见踩坑经验

1. **TokenRelay过滤器在Spring Cloud 2021.x与2023.x版本中初始化时机不同**：旧版本中`TokenRelayGatewayFilterFactory`的order在`NettyRoutingFilter.ORDER`之前，新版本中可能后移。排查时在Gateway日志中打开`reactor.netty.http.client`的DEBUG日志，观察发给下游的请求头中`Authorization`是否包含Bearer Token。

2. **OAuth2 Client与Resource Server的角色冲突导致Gateway行为异常**：当Gateway同时配置了`oauth2.client`和`oauth2.resourceserver`时，Spring Security的Reactive Filter Chain中会同时存在`OAuth2Login`和`BearerToken`两种认证机制的Filter。如果请求中携带Authorization头，走Resource Server路线；如果不携带，可能被OAuth2 Client拦截并重定向到Keycloak登录页。排查方法是查看Gateway的`/actuator/beans`中SecurityWebFilterChain的filter列表，确认BearerTokenAuthenticationWebFilter总是在OAuth2Login之前。

3. **Feign拦截器在异步调用场景中SecurityContext丢失**：当使用`@Async`或`CompletableFuture`时，主线程的`SecurityContext`不会自动传递到子线程。解决方案有：(a) 使用`DelegatingSecurityContextAsyncTaskExecutor`包装异步执行器；(b) 在拦截器中改为从HTTP请求Header解析Token而非依赖`SecurityContextHolder`；(c) 将Token作为方法参数显式传递给Feign接口。方案(b)最为健壮——直接从ServletRequest的Authorization头提取Token，不依赖ThreadLocal。

### 思考题

1. **在零信任架构中，即使内网微服务之间也需要严格认证。如何用Keycloak + SPIFFE/SPIRE实现微服务间的mTLS认证？** 请描述：(1) SPIFFE（Secure Production Identity Framework for Everyone）如何为每个微服务签发唯一身份标识SPIFFE ID（如`spiffe://company.com/order-service`）；(2) SPIRE Agent如何管理微服务Pod的X.509-SVID证书颁发和轮换；(3) Keycloak如何作为SPIFFE Federation的OIDC Provider，将OAuth2 Token中的用户身份映射到SPIFFE verifiable identity；(4) mTLS握手时，调用方使用SPIFFE签发的证书替代传统OAuth2 Bearer Token——这对Tokn Relay模型有何影响？

2. **Service Mesh（如Istio）接管了部分认证职责后，Keycloak的角色应该是什么？** 请分析：(1) Istio的RequestAuthentication和AuthorizationPolicy分别能在Sidecar层实现哪些Keycloak原生功能（JWT验签、Role-Based路由、mTLS）？(2)哪些场景仍必须在应用层保留Keycloak集成（如基于用户属性的动态授权、Token Exchange、UMA细粒度授权）？(3) 混合方案——Istio在L4/L7层做粗粒度安全控制（连接级mTLS、入口JWT验签），Keycloak在应用层做细粒度授权——各自的边界如何界定？是否存在"双保险导致双漏洞"的问题？
