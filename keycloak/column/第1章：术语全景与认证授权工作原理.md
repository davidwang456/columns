# 第1章：术语全景与认证授权工作原理

## 1 项目背景

### 1.1 业务场景

一家300人的中型SaaS公司，内部有CRM、ERP、OA三个管理系统，外加一个面向客户的客户门户。每个系统都是不同时期、由不同团队搭建的——CRM用Spring Security自己写了套登录逻辑，ERP是购买的商业软件带内置认证模块，OA用的是LDAP目录服务，客户门户则接入了微信扫码登录。

看似每个系统都能正常工作，但IT部门的张工每天焦头烂额：

- **入职流程噩梦**：新员工第一天，IT需要在4个系统分别开设账号、配置权限。其中ERP必须在服务器上用命令行操作，OA要通过Web管理后台逐条录入，CRM倒是支持HR系统同步但从来没调通过。一个员工入职，张工至少忙活2小时。

- **离职漏注销**：上个月离职的销售总监，他的CRM账号一直到离职后第15天才被关闭——这期间他仍能通过VPN访问CRM中的客户数据和报价单。类似的事故每年至少发生3次。

- **密码工单灾难**：IT部门每天平均接到15个"密码忘了""登录不上去"的工单，占全部工单量的40%以上。最离谱的是，同一个员工可能上午重置了CRM密码，下午又来问ERP的密码怎么不同。

- **审计困境**：去年一次安全审查，需要提供"所有系统的用户访问日志"。张工花了整整一周，从4个系统的不同位置——数据库表、日志文件、审计模块——手工汇总了一份148页的Excel报告，格式七拼八凑，被安全审计团队直接打了回来。

CTO在季度复盘会上拍板："不管花多大代价，必须搞一个统一的认证平台。"

## 2 项目设计——剧本式交锋对话

> 人物：**小胖**（两年经验，爱用生活比喻理解技术，不求甚解）、**小白**（应届生，喜欢刨根问底）、**大师**（技术Leader，10年老兵）
>
> 场景：周一晨会后的技术方案讨论，白板上画着"统一认证平台"几个大字。

---

**小胖**（指着白板）：这不就相当于给整栋写字楼装一个大门禁嘛！以前每个房间自己装锁，现在所有房间都用同一张门禁卡。这有什么难的，搞个数据库放用户密码，每个系统都来查一下不就完了？

**大师**（放下咖啡杯）：胖子的比喻方向是对的，但方案太简化了。你说"每个系统都来查一下"，那就意味着每个系统都能拿到用户密码——一个系统被攻破，所有用户的密码全泄露。而且，如果门禁系统挂了，整栋楼都进不去——这就是所谓"单点故障"。一个好的统一认证方案需要解决的是：如何在不暴露密码的前提下，让各个应用系统"相信"你已经登录了。

> **技术映射**：这就是"认证中心"和"信任传递"两个核心概念。密码只交给认证中心验证，认证中心签发凭证（Token），应用系统验证凭证而不接触密码。

---

**小白**（推了推眼镜）：我有个疑问。如果我们搞了统一认证，假设我登录了CRM，再打开ERP，是不是就不用再输入密码了？这就是SSO对吧？但那个Token是什么东西，为什么ERP看到这个Token就相信我是我了？

**大师**：问得好。SSO——单点登录，本质是"会话共享"。你用银行卡在ATM取钱时，插卡输入密码，银行验证之后给你一个"会话"，接下来的转账、查余额都不需要重新输密码，直到你退卡或者超时。Web的SSO也是类似：你登录一次认证中心，认证中心给你发一个Token——你可以理解为一张盖了公章的"临时通行证"。各个应用系统看到这个通行证时，不是去验证"你是谁"，而是去验证"这张通行证是不是我们信任的机关签发的"。验证方式就是检查签名。

**小白**：那Token具体长什么样？

**大师**：最常见的JWT——JSON Web Token，由三部分组成，用点号分隔：Header、Payload、Signature。Header声明算法，Payload里放用户ID、过期时间等信息，Signature是用认证中心的私钥对前两部分算出来的签名。任何人拿到Token都能读到Payload，但只有持有私钥的认证中心才能签发新Token，也只有拥有公钥的人才能验证签名是否被篡改。

```
JWT结构示意：
eyJhbGciOiJSUzI1NiJ9        ← Header（Base64编码的JSON）
.eyJzdWIiOiJ1c2VyMTIzIn0    ← Payload（Base64编码的JSON）
.SflKxwRJSMeKKF2QT4fwpMe...  ← Signature（签名）
```

> **技术映射**：Token = 数字通行证。三个特征——**(1)自包含**（用户信息在Token里，应用不用再去查数据库）、**(2)可验证**（任何持有公钥的系统都能独立验证真伪）、**(3)有时效**（过期自动失效，短生命周期降低被盗风险）。

---

**小胖**（突然插话）：等等，我又想到一个比喻！Realm是不是就像写字楼里的不同公司？A公司在12楼，B公司在13楼，虽然是同一栋楼（同一个Keycloak实例），但门禁系统里A公司看到的员工和B公司看到的员工是完全隔离的，A公司的HR不可能看到B公司的员工名单。Client是不是就是每家公司里的不同房间——财务室、技术部、会议室？

**大师**（露出欣慰的表情）：小胖今天开窍了。你说的完全正确。详细展开：

| Keycloak术语 | 写字楼比喻 | 技术含义 |
|---|---|---|
| **Realm**（领域） | 一栋楼里的某家公司 | 租户隔离单元，拥有独立的用户、角色、客户端。Master Realm是"物业管理处"，管理所有公司 |
| **Client**（客户端） | 公司里的一个房间/部门 | 需要认证保护的应用系统（CRM是Client、ERP是Client） |
| **User**（用户） | 某公司的员工 | 属于某个Realm的认证实体 |
| **Role**（角色） | 员工工牌上的权限级别 | 权限标签，如"sales-admin"、"finance-readonly" |
| **Session**（会话） | 员工当前是否在楼内 | 用户登录后的活动状态，包含Token、登录时间、IP等 |
| **Group**（组） | 部门（销售部、技术部） | 用户的组织归属，可批量赋予角色 |

> **技术映射**：Realm是Keycloak最核心的隔离概念。多租户SaaS产品中，每个客户通常对应一个Realm。Master Realm是Keycloak自带的管理专用Realm，可以管理其他所有Realm但不能用于业务应用。

---

**小胖**（又插话）：哎大师，我看网上有人提OAuth 2.0，又有人提OIDC，还有人提SAML，这三个到底是啥关系？是不是都是"认证协议"？

**大师**：这是最容易混淆的地方，很多人用了一两年Keycloak都没完全搞懂。

- **OAuth 2.0** 严格来说是一个**授权协议**，不是认证协议。它解决的核心问题是："应用A如何在不获取用户密码的情况下，访问用户在应用B中的数据？"典型场景：你用微信账号登录一个第三方App，第三方App不会拿到你的微信密码，而是拿到一个Access Token，只能访问你授权的那部分微信资料。OAuth 2.0描述的是一种"委托授权"机制。

- **OpenID Connect（OIDC）** 是在OAuth 2.0之上的**认证层**。它在OAuth 2.0的Access Token之外，增加了一个ID Token（JWT格式），专门用来告诉应用"这个用户到底是谁"。你可以这样记：OAuth 2.0说"这个Token可以访问什么资源"，OIDC说"这个Token证明了谁在操作"。所有OIDC一定是OAuth 2.0的扩展，但反过来不成立。

- **SAML 2.0** 是比OAuth/OIDC更早的协议，基于XML和SOAP，在企业级应用中根深蒂固（很多老牌企业SSO产品还在用SAML）。它用XML断言来描述用户身份，比OIDC笨重，但成熟度和安全性通过了多年验证。在Keycloak中，SAML通过"IDP Broker"机制支持。

```
  ┌─────────────────────────────────┐
  │         OpenID Connect          │
  │   (认证层：ID Token证明你是谁)     │
  │  ┌───────────────────────────┐  │
  │  │       OAuth 2.0           │  │
  │  │  (授权层：Access Token     │  │
  │  │   描述你能访问什么资源)      │  │
  │  └───────────────────────────┘  │
  └─────────────────────────────────┘

  SAML 2.0 ← 独立的老牌企业SSO协议（XML格式）
```

> **技术映射**：简单记法——要认证优先选OIDC（现代化的JSON/JWT系协议，适合SPA/移动端），要和老系统对接看SAML支持情况，要纯授权（如API网关）用OAuth 2.0就好。Keycloak对三者都提供完整支持。

---

**小白**（一直在记笔记，终于抬起头）：大师，你说Token有时效，那用户登录后过一会儿Token过期了，难道要重新登录？SSO到底是怎么记住"我已经登录过了"这件事的？

**大师**：这才是SSO的灵魂问题。Keycloak的会话管理是**三级缓存机制**：

```
第一级：浏览器Cookie（AUTH_SESSION_ID / KEYCLOAK_SESSION）
   ↓ 丢失后查第二级
第二级：Keycloak服务端Session（内存/Infispan缓存）
   ↓ 过期后查第三级
第三级：Refresh Token（有效期通常30分钟到数小时）
   ↓ 过期后只能用
最终手段：重新登录 → 生成新的用户Session
```

具体流程：

1. 用户访问CRM → CRM重定向到Keycloak登录页（`/realms/company/protocol/openid-connect/auth`）
2. 用户输入用户名密码 → Keycloak验证通过，在服务端创建Session，在浏览器写入Cookie `KEYCLOAK_SESSION`
3. Keycloak签发Code → CRM用Code换取Access Token + ID Token + Refresh Token
4. 用户再访问ERP → ERP同样重定向到Keycloak → Keycloak发现Cookie `KEYCLOAK_SESSION`存在且有效 → **直接跳过登录页，返回Code给ERP**
5. Access Token过期（默认5分钟）→ CRM用Refresh Token静默换取新Token，用户无感知
6. Refresh Token也过期 → Cookie `KEYCLOAK_SESSION`仍有效 → 重新走OIDC流程但不显示登录页
7. Cookie也过期或Session被管理员吊销 → 需要重新登录

这就是SSO不需要反复输入密码的秘密——只要浏览器里那个Cookie还在，Token过期了可以悄悄续。

> **技术映射**：SSO的关键不是"永远不过期"，而是**策略性地让重新认证过程对用户透明**。Cookie维持认证状态，Refresh Token承担续期职责，只有在两者都失效时才需要交互式登录。

## 3 项目实战

### 3.1 架构全景图

在动手之前，先建立Keycloak的全局架构概念：

```
                        ┌─────────────────────────────┐
                        │       Keycloak Server        │
                        │  ┌─────────┐  ┌───────────┐ │
  浏览器 ───────────────▶│  │ 认证引擎  │  │  管理后台   │ │◀─────── 管理员
  (登录/SSO)             │  │ (AuthN)  │  │ (Admin UI) │ │
                        │  └────┬────┘  └───────────┘ │
                        │       │                      │
                        │  ┌────▼──────────────────┐   │
                        │  │   协议适配层            │   │
                        │  │  OIDC │ SAML │ Custom  │   │
                        │  └────┬──────────────────┘   │
                        │       │                      │
                        │  ┌────▼────┐  ┌───────────┐  │
                        │  │ 用户存储  │  │  Session   │  │
                        │  │ Provider │  │  Infispan  │  │
                        │  │ (DB/LDAP)│  │  (分布式)  │  │
                        │  └─────────┘  └───────────┘  │
                        └──────────────┬──────────────┘
                                       │
                    ┌──────────────────┼───────────────────┐
                    ▼                  ▼                    ▼
              ┌──────────┐     ┌──────────┐        ┌──────────┐
              │  CRM App │     │  ERP App │        │ API网关   │
              │(OIDC Client)│   │(SAML Client)│     │(Bearer Token)│
              └──────────┘     └──────────┘        └──────────┘
```

**组件说明**：
- **认证引擎**：处理登录表单、多因素认证、密码策略等
- **协议适配层**：将认证结果包装成不同协议格式（OIDC的JWT、SAML的XML断言）
- **用户存储Provider**：默认使用内置数据库，可对接LDAP、AD或自定义用户存储
- **Session Infispan**：Quarkus版Keycloak使用Infispan缓存用户会话，支持集群部署时跨节点共享
- **管理后台/API**：提供Realm、Client、User的CRUD操作，支持`kcadm`命令行工具和REST API

### 3.2 Keycloak术语与OAuth 2.0标准术语映射

OAuth 2.0定义了一套标准角色和端点，Keycloak在实现时使用自己的术语。以下是对照表：

| OAuth 2.0 / OIDC标准术语 | Keycloak对应术语 | 说明 |
|---|---|---|
| Authorization Server（授权服务器） | Keycloak Server | 签发Token的服务器 |
| Resource Server（资源服务器） | 不直接对应，由Client App或API网关充当 | 受保护的资源/API |
| Client（客户端） | Client | 请求认证/授权的应用，*两者术语一致* |
| Client ID / Client Secret | Client ID / Client Secret | 客户端标识与密钥，*两者术语一致* |
| Authorization Endpoint | `/realms/{realm}/protocol/openid-connect/auth` | 认证端点，用户在此页输入密码 |
| Token Endpoint | `/realms/{realm}/protocol/openid-connect/token` | 用Code/Refresh Token换取Token |
| UserInfo Endpoint | `/realms/{realm}/protocol/openid-connect/userinfo` | 获取当前用户身份信息 |
| JWKS (JWK Set) | `/realms/{realm}/protocol/openid-connect/certs` | 验证JWT签名所需的公钥集合 |
| End Session Endpoint | `/realms/{realm}/protocol/openid-connect/logout` | 登出端点 |
| Grant Type: authorization_code | Standard Flow（标准流程） | 客户端使用Code换取Token |
| Grant Type: client_credentials | Service Account（服务账号） | 服务间通信，无用户参与 |
| ID Token | ID Token（JWT格式） | 证明用户身份 |
| Access Token | Access Token（JWT格式） | 访问资源的许可 |
| Refresh Token | Refresh Token | 续期Access Token |
| Scopes | Client Scopes | 定义Token中应包含哪些用户属性/权限 |
| Claims | Mappers（映射器） | 将用户属性写入Token的规则 |
| Resource Owner（资源所有者） | User（用户） | 最终用户，资源的拥有者 |

热知识：Keycloak旧版（WildFly版）的API路径为`/auth/realms/...`，Quarkus版（17+）移除了`/auth`前缀，直接使用`/realms/...`。本文以Quarkus版（26.x）为准。

### 3.3 实战：用curl验证OIDC Discovery端点

#### 步骤1：启动Keycloak

确保Docker Desktop已运行，执行以下命令启动Keycloak 26.1：

```bash
docker run -d --name keycloak-dev \
  -p 8080:8080 \
  -e KC_BOOTSTRAP_ADMIN_USERNAME=admin \
  -e KC_BOOTSTRAP_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:26.1 start-dev
```

参数说明：
- `-d`：后台运行容器
- `-p 8080:8080`：将容器的8080端口映射到宿主机
- `-e`：设置环境变量，`KC_BOOTSTRAP_ADMIN_*`用于创建初始管理员
- `start-dev`：开发模式（不推荐生产使用，但启动最快，H2内存数据库）

首次拉取镜像可能需要1-3分钟，启动后约30秒Keycloak即可就绪。

#### 步骤2：验证启动状态

```bash
# 检查Keycloak是否启动完成（等待日志出现 "Listening on"）
docker logs keycloak-dev 2>&1 | Select-String "Listening on"
```

你也可以直接访问 `http://localhost:8080`，看到Keycloak欢迎页即表示启动成功。

#### 步骤3：访问OIDC Discovery端点

OIDC协议定义了Discovery机制，让客户端应用能够自动发现认证服务器的所有端点配置，无需硬编码。Keycloak为每个Realm生成独立的发现文档：

```bash
# 方式一：直接curl（Windows PowerShell中jq可替换为python json解析）
curl http://localhost:8080/realms/master/.well-known/openid-configuration

# 方式二（如有jq）：格式化输出
curl -s http://localhost:8080/realms/master/.well-known/openid-configuration | jq .
```

#### 步骤4：解读返回结果

Discovery端点返回的JSON文档包含以下关键字段（节选）：

```json
{
  "issuer": "http://localhost:8080/realms/master",
  "authorization_endpoint": "http://localhost:8080/realms/master/protocol/openid-connect/auth",
  "token_endpoint": "http://localhost:8080/realms/master/protocol/openid-connect/token",
  "userinfo_endpoint": "http://localhost:8080/realms/master/protocol/openid-connect/userinfo",
  "end_session_endpoint": "http://localhost:8080/realms/master/protocol/openid-connect/logout",
  "jwks_uri": "http://localhost:8080/realms/master/protocol/openid-connect/certs",
  "scopes_supported": ["openid", "profile", "email", ...],
  "response_types_supported": ["code", "token", "id_token", ...],
  "grant_types_supported": ["authorization_code", "client_credentials", "refresh_token", ...],
  "token_endpoint_auth_methods_supported": ["client_secret_basic", "client_secret_post", ...],
  "id_token_signing_alg_values_supported": ["RS256", "PS256", "ES256", "HS256"]
}
```

**各字段含义**：

| 字段 | 作用 | 使用场景 |
|---|---|---|
| `issuer` | 签发者标识 | Token中的`iss`字段必须与此一致，防止Token被重放到其他Realm |
| `authorization_endpoint` | 用户登录页 | 应用重定向用户到此URL完成认证 |
| `token_endpoint` | Token颁发端点 | 服务端用Code或Refresh Token换取Access/ID Token |
| `jwks_uri` | 公钥集合 | 资源服务器从此获取公钥验证JWT签名 |
| `end_session_endpoint` | 登出端点 | 清除SSO会话 |
| `scopes_supported` | 支持的Scope | `openid`是OIDC必须的Scope，`profile`、`email`等可选 |

你可以选任意OIDC客户端库（如`openid-client`、AppAuth SDK），只需传入Discovery URL，库会自动拉取所有端点配置，无需手动设置。

#### 步骤5：验证JWKS端点

```bash
# 获取用于验证JWT签名的公钥
curl -s http://localhost:8080/realms/master/protocol/openid-connect/certs | jq .
```

返回的JSON包含一个`keys`数组，每个Key包含`kid`（Key ID）、`kty`（Key Type，通常为RSA）、`n`、`e`等公钥参数。资源服务器用`kid`匹配JWT Header中的`kid`，找到对应公钥验证签名。

### 3.4 完整命令清单

以下是本章所有实操命令的汇总，可一键执行：

  -e KC_BOOTSTRAP_ADMIN_USERNAME=admin ^
  -e KC_BOOTSTRAP_ADMIN_PASSWORD=admin ^

# 2. 查看启动日志

### 3.5 可能遇到的坑

| 问题 | 原因 | 解决方案 |
|---|---|---|
| **端口8080已被占用** | 其他服务（如Jenkins、Tomcat）占用了8080 | 改用其他端口映射：`-p 9090:8080`，然后访问`http://localhost:9090` |
| **Docker内存不足** | Keycloak Quarkus版默认堆内存需求较高 | Docker Desktop → Settings → Resources → 将内存调至至少4GB |

| **Windows下curl命令404** | 使用了PowerShell的`curl`别名（实际是`Invoke-WebRequest`） | 使用`curl.exe`而非`curl`，或安装原生curl |

## 4 项目总结

### 4.1 Keycloak vs 自建 vs Auth0

| 维度 | Keycloak（开源） | 自建认证系统 | Auth0（商业SaaS） |
|---|---|---|---|
| **开发成本** | 低——开箱即用，Docker一键部署 | 高——需开发认证、授权、Token管理、管理后台、协议适配等完整功能 | 低——注册即用，SDK集成简单 |
| **运维成本** | 中——需自行维护服务器、数据库、升级版本 | 极高——全部自己承担 | 低——Okta托管，按需付费 |
| **定制灵活性** | 高——开源，SPI扩展点丰富，可自定义主题、用户存储Provider、认证流程SPI | 最高——完全可控 | 中——受限于SaaS平台提供的定制选项 |
| **协议支持** | OIDC + SAML + OAuth 2.0 + LDAP | 取决于开发投入 | OIDC + SAML + OAuth 2.0 |
| **多租户能力** | 原生Realm支持，天然多租户 | 需自行实现 | 通过"品牌域名"和"组织"功能支持 |
| **社区/支持** | 社区活跃，红帽官方提供RHBK商业版 | 无外部支持 | Okta官方支持，响应快 |
| **合规认证** | FIPS 140-2需额外配置 | 需自行达标 | SOC2、GDPR、HIPAA等已认证 |
| **费用** | 免费（基础设施费用另计） | 人力成本 | 按月活用户计费，量大不便宜 |
| **适用规模** | 中大型企业（万级~十万级用户） | 小团队起步或高度定制场景 | 初创到大型均可，量越大越贵 |

### 4.2 适用场景

- **企业内部SSO**：最经典场景。10个、20个内部系统统一用一个Keycloak，员工一次登录全通。
- **微服务认证**：API Gateway + Keycloak，每个微服务独立验证JWT，认证逻辑从业务代码中完全剥离。
- **社交登录聚合**：通过Keycloak Identity Broker对接Google、GitHub、微信等外部IDP，为应用提供统一登录入口。
- **B2B多租户认证**：一个Keycloak实例，每个客户企业一个Realm，数据完全隔离，管理员各自管理自己的用户。
- **零信任架构的Token服务**：Keycloak充当STS（Secure Token Service），签发短效Token供服务间调用。

### 4.3 不适用场景

- **极简应用（如个人博客）**：Keycloak的内存占用通常在512MB以上，对于只需要简单Basic Auth的极简应用而言开销过大，建议直接用轻量方案。
- **完全无状态的Serverless场景**：Keycloak依赖数据库和会话缓存，在Lambda这类无状态、无法保持长连接的环境中部署复杂。可考虑用Auth0或Clerk等托管方案。
- **超大规模（亿级用户）C端场景**：Keycloak并非为亿级C端用户设计，虽然可通过集群扩展，但在这种规模下更常见的是自研方案或使用专门的大规模身份平台。

### 4.4 版本兼容注意事项

- **WildFly版（<17.x）vs Quarkus版（≥17.x）**：Keycloak 17进行了架构级迁移，从WildFly应用服务器迁移到Quarkus框架。API路径从`/auth/realms/...`变为`/realms/...`，部署方式从解压war包变为`kc.sh start`命令。**本文使用Quarkus 26.x版本，命令和陈旧教程不兼容**，遇到教程时请注意版本号。
- **旧版环境变量**：Quarkus版不再使用`KEYCLOAK_USER`/`KEYCLOAK_PASSWORD`（17-18版曾用过），改用`KC_BOOTSTRAP_ADMIN_USERNAME`/`KC_BOOTSTRAP_ADMIN_PASSWORD`。
- **安全边界**：Keycloak的默认配置运行在HTTP模式下，生产部署必须配置HTTPS（TLS终止可在Keycloak自身或前置Nginx进行）。`start-dev`模式严格禁止用于生产环境。

### 4.5 常见踩坑经验

1. **端口冲突导致Keycloak启动失败**：同事机器上Jenkins占用了8080，Docker `-p 8080:8080`发现端口冲突，错误信息藏在海量日志里不容易找到。快速定位方法：`docker logs keycloak-dev 2>&1 | Select-String "Address already in use"`。解决：换端口或停掉占用进程。

2. **数据库连接失败**：生产环境使用外部PostgreSQL时，最常见的问题是Keycloak容器先于数据库容器启动，或JDBC URL格式错误。Quarkus版的数据库配置通过`KC_DB=postgres`、`KC_DB_URL`、`KC_DB_USERNAME`、`KC_DB_PASSWORD`环境变量设置，格式与WildFly版完全不同。

3. **Realm导入失败**：使用`--import-realm`启动参数导入JSON格式的Realm时，如果JSON中缺少`realm`顶层字段或`enabled`字段为`false`，Keycloak会静默跳过该Realm，不会报错，日志中也仅有INFO级别提示。排查时需要检查JSON的顶层key是否为`"realm"`而非根对象本身。

### 4.6 思考题

1. **OIDC和OAuth 2.0的本质区别是什么？** 提示：不要停留在"一个管认证一个管授权"的表面表述，请从Token类型（ID Token vs Access Token）、协议角色（RP vs Client）、端点差异三个维度给出技术层面的精确回答。

2. **Keycloak如何实现跨域SSO？** 假设A系统部署在`a.company.com`，B系统部署在`b.company.com`，Keycloak部署在`sso.company.com`。用户登录A后再访问B如何保持登录状态？请从Cookie Domain、SameSite属性、Token传递三个层面分析技术实现。
