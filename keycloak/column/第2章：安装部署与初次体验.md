# 第2章：安装部署与初次体验

## 1 项目背景

某电商公司的架构师小李接到CTO的紧急任务——公司业务线快速扩张，订单系统、客服系统、运营后台三个业务系统各自维护了一套登录逻辑。用户需要在不同系统间反复登录，体验极差；安全部门也频频告警：某系统使用明文存储密码、另一个系统Session机制存在CSRF漏洞。CTO要求在一周内搭建统一的身份认证平台，实现SSO单点登录，且必须与公司现有的PostgreSQL数据库集成。

小李此前对Keycloak有所耳闻——知道它是Red Hat开源的身份认证解决方案，但从未实际部署过。时间紧、任务重，他需要从零开始，在一周内完成选型、部署、基础配置和三系统对接。

**痛点放大**：传统自建认证系统的开发工作量极其庞大。仅基础功能就包括：用户注册与激活、登录与登出、密码找回与重置、Session生命周期管理、角色权限模型、OAuth2/OIDC协议实现、Token签发与校验等。以中等团队的经验估算，从设计到上线至少需要2-3个月的开发周期。此外，安全防护更是一个无底洞——CSRF跨站请求伪造、XSS跨站脚本攻击、暴力破解防护、密码哈希加盐、Token防重放……每一项都需要专门的安全专家介入。

而选用Keycloak，可以将这些通用能力全部交给成熟的开源中间件处理，团队只需专注于业务对接和定制化开发。小李评估：部署Keycloak最多1天，基础配置1天，业务系统对接3天，总共5天即可完成核心交付——这与CTO的要求高度吻合。

## 2 项目设计——剧本式交锋对话

办公室下午三点，小李正对着Keycloak官方文档发愁。小胖端着一杯奶茶晃过来，看到屏幕上密密麻麻的英文，好奇地凑了过来。

**小胖**："哥，这玩意儿是不是就跟装个新游戏一样？Steam上点一下就开始下载，哪有这么复杂？"

**小白**放下手中的书，推了推眼镜："Keycloak的安装方式有好几种——Docker、裸机手动、Kubernetes。你说'简单'，那到底选哪种？Docker虽然快，但生产环境能有保障吗？"

**大师**刚好路过，听到这话停下了脚步："你们在讨论部署方式？这个问题问得好。我先给一个总体原则——**开发学习用Docker，生产环境视规模而定**。"

"具体来说，"大师拉过一把椅子坐下，"Docker方式最适合快速体验和学习，一条命令就能跑起来，环境完全隔离，不污染宿主机。但Docker容器重启后默认的数据会丢失，需要挂载数据卷才能持久化。"

**小胖**恍然大悟："就像我玩Switch游戏，存档要存到卡带里而不是主机内存里！"

**大师**笑了笑："差不多这个意思。裸机部署适合需要精细控制JVM参数、做深度定制的场景。Kubernetes部署则适用于需要高可用、弹性伸缩的生产集群。三种方式由简到繁，对应不同的组织成熟度。"

**小白**继续追问："我注意到Keycloak 26.x改用Quarkus了，之前不是跑在WildFly上的吗？Quarkus到底是什么？"

**大师**点头赞许："问到了关键。Keycloak从17版本开始迁移到Quarkus框架，到26.x版本已经完全基于Quarkus。Quarkus是Red Hat推出的云原生Java框架，专为容器化环境优化——启动速度快、内存占用小、支持GraalVM原生编译。"

"在Quarkus架构下，启动命令也变了，"大师在白板上写起来：

- `start-dev`：开发模式启动。自动启用热重载、调试日志、H2内存数据库，配置文件变更无需重启。**严禁用于生产环境**。
- `start`：生产模式启动。需要先执行`build`编译优化，所有配置在构建时固化，启动后不可变。
- `build`：编译优化阶段。将所有Quarkus扩展和配置编译为优化后的字节码，是生产启动的前置步骤。
- `optimize`：进一步优化——可以在构建阶段生成Native镜像（需要GraalVM），获得亚秒级启动。

**小胖**："那数据库选哪个好？我听说有好几种选项。"

**大师**继续写道：

| 模式 | 数据库 | 适用场景 |
|------|--------|----------|
| `dev-file` | H2文件数据库（`data/h2/keycloak.mv.db`） | 开发测试，数据可持久化到磁盘 |
| `dev-mem` | H2内存数据库 | 纯实验学习，重启后数据全丢 |
| `postgresql` / `mysql` / `mssql` | 各主流关系型数据库 | 生产环境唯一选择 |

"26.x版本的默认开发数据库是`dev-file`。如果你在开发环境想避免数据丢失，用默认的`dev-file`就够。生产环境必须对接PostgreSQL、MySQL等外部数据库——H2不支持高并发、集群也不可用。"

**小白**："管理控制台有什么功能？我们怎么确认安装成功了？"

**大师**："访问管理控制台就是最好的验证。Keycloak分两块界面——Admin Console是管理员用的，管理Realm、用户、客户端、角色等配置；Account Console是终端用户用的，用户可以自己修改密码、管理OTP设备、查看登录会话。初次启动后会有一个Bootstrap引导流程——创建管理员用户名和密码，这是Keycloak 24+版本引入的安全性增强。"

**大师总结技术映射**：Docker/裸机/K8s是三种载体选择，本质是环境隔离与运维成本的权衡。`start-dev` vs `start` vs `build`是Quarkus框架的生命周期——从快速迭代到编译优化到原生编译，就像汽车从"试驾模式"到"日常驾驶"再到"赛车调校"。数据库选型取决于数据是否需要持久、是否需要高并发、是否需要集群部署。

"操作Realm很简单，"大师演示道，"左上角有个下拉菜单，默认是Master——这是Keycloak内置的管理域，专门用来管理Keycloak自身。点击下拉，选择'Create Realm'，输入一个名称就能创建一个全新的独立域。每个Realm之间的用户、角色、客户端完全隔离，就像写字楼里不同公司的独立门禁系统。"

**小白**追问："那Account Console和Admin Console到底什么区别？普通用户也能登录Admin Console吗？"

**大师**："Account Console面向终端用户，URL是`/realms/{realm}/account`，用户可以在这里自助修改密码、管理双因素认证、查看设备登录记录。Admin Console面向管理员，URL是`/admin`，需要有管理权限的用户才能访问。两者安全边界完全不同——绝不要给普通用户分配admin角色。"

**大师总结技术映射**：Admin Console是"后台管理系统"，操作Keycloak的一切配置；Account Console是"用户自助中心"，让用户自己管理自己的身份数据。Master Realm是Keycloak自身的"特区"，不应承载业务用户——为每个业务线创建独立的Realm才是最核心的多租户隔离原则。

## 3 项目实战

### 环境准备

| 工具 | 版本要求 | 用途 |
|------|---------|------|
| Docker Desktop | 20.10+ | 容器运行时 |
| curl | 7.0+ | HTTP请求测试（Windows可用PowerShell内置`Invoke-RestMethod`替代） |
| jq | 1.6+ | JSON格式化展示（可选） |

### 步骤1：Docker快速启动（开发模式）

将以下命令粘贴到终端执行：

```bash
docker run -d --name keycloak-dev -p 8080:8080 \
  -e KC_BOOTSTRAP_ADMIN_USERNAME=admin \
  -e KC_BOOTSTRAP_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:26.1 start-dev
```

**参数说明**：
- `-d`：后台运行容器
- `--name keycloak-dev`：容器命名，便于后续管理
- `-p 8080:8080`：将容器8080端口映射到宿主机8080端口
- `-e KC_BOOTSTRAP_ADMIN_USERNAME=admin`：设置初始管理员用户名
- `-e KC_BOOTSTRAP_ADMIN_PASSWORD=admin`：设置初始管理员密码
- `quay.io/keycloak/keycloak:26.1`：从Quay.io镜像仓库拉取Keycloak 26.1版本
- `start-dev`：以开发模式启动

**运行结果**：首次启动会从Quay.io拉取镜像层，耗时取决于网络速度。拉取完成后，控制台会输出启动日志，关键行如下：

```
Running the server in development mode.
Profile dev activated and persisted.
Listening on: http://0.0.0.0:8080
```

可以通过以下命令实时查看启动日志：

```bash
docker logs -f keycloak-dev
```

当看到`Listening on`字样时，表示Keycloak已成功启动，大约需要30秒到1分钟（包含镜像下载时间）。

### 步骤2：理解Bootstrap Admin概念

在上述Docker命令中，我们通过环境变量指定了初始管理员账号。这是Keycloak 24+版本引入的**Bootstrap Admin（引导管理员）**机制。

**传统方式 vs Bootstrap方式**：
- **传统方式**（已废弃）：首次访问管理控制台时在Web页面填写管理员信息，容易被未授权的访问者抢占。
- **Bootstrap方式**：通过环境变量（`KC_BOOTSTRAP_ADMIN_USERNAME`和`KC_BOOTSTRAP_ADMIN_PASSWORD`）在启动前就锁定了管理员身份。启动完成后，网页端不再提供注册页面，避免了"先到先得"的安全风险。

Bootstrap Admin创建的账户会被自动分配到Master Realm，并拥有最高的`admin`角色。这是唯一可以管理其他Realm的超级管理员账户。

### 步骤3：访问管理控制台

打开浏览器，访问 `http://localhost:8080/admin`（注意必须用**localhost**而非127.0.0.1，否则浏览器可能触发混合内容安全策略）。

使用步骤1中设置的用户名`admin`和密码`admin`登录。

**登录后界面总览**：

左侧导航分为两大区域：

**Manage（管理）分组**：
- **Clients**：管理OAuth2/OIDC/SAML客户端
- **Client Scopes**：管理客户端作用域（声明集合）
- **Realm Roles**：管理Realm级别的角色
- **Users**：管理所有用户
- **Groups**：管理用户组层次结构
- **Sessions**：查看活跃的用户和客户端会话
- **Events**：查看登录、登出、授权等审计事件

**Configure（配置）分组**：
- **Realm Settings**：Realm全局配置（会话超时、令牌有效期、安全策略等）
- **Authentication**：配置认证流程（密码、OTP、WebAuthn、条件编排）
- **Identity Providers**：配置社交登录和外部IdP
- **User Federation**：集成LDAP/AD等外部用户存储

左上角的Realm下拉选择器当前显示为`Master`——这是Keycloak的内置管理Realm。

### 步骤4：创建第一个Realm

**操作流程**：

1. 点击左侧导航栏顶部显示"Master"的下拉菜单
2. 点击**Create Realm**
3. 在"Realm name"字段中输入`demo-realm`
4. 保持"Enabled"开关为开启状态
5. 点击**Create**按钮

**运行结果**：页面标题变为"demo-realm"，左侧导航菜单内容与Master Realm完全一致——这表示一个全新的独立域已创建成功。demo-realm拥有自己独立的用户、角色、客户端、会话，与Master Realm完全隔离。

也可以通过REST API直接创建Realm（先获取Master Realm的Token）：

```bash
# 获取管理员Token
ADMIN_TOKEN=$(curl -s -X POST http://localhost:8080/realms/master/protocol/openid-connect/token \
  -d "client_id=admin-cli" \
  -d "username=admin" \
  -d "password=admin" \
  -d "grant_type=password" | jq -r '.access_token')

# 创建Realm
curl -X POST http://localhost:8080/admin/realms \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"realm":"demo-realm2","enabled":true}'
```

### 步骤5：创建第一个用户

**操作流程**：

1. 在左侧导航中点击**Users**
2. 点击页面右侧的**Create new user**按钮
3. 在弹出的表单中填写：
   - **Username**：`testuser`（必填）
   - **Email**：`testuser@example.com`（可选）
   - **First Name**：`Test`（可选）
   - **Last Name**：`User`（可选）
4. 点击**Create**按钮

用户创建完成后，页面自动跳转至用户详情页。接下来设置密码：

5. 在用户详情页，点击顶部的**Credentials**标签
6. 点击**Set Password**按钮
7. 输入密码`test123`和确认密码`test123`
8. **将"Temporary"开关关闭**（关闭后用户首次登录不需要强制修改密码）
9. 点击**Save**按钮

**运行结果**：Credentials标签下出现一个密码凭证记录，类型为"password"，创建时间记录为当前时刻。这表示密码设置成功。

### 步骤6：用curl验证用户认证

打开终端，执行以下命令测试密码授权流程（Resource Owner Password Credentials Grant）：

```bash
curl -X POST http://localhost:8080/realms/demo-realm/protocol/openid-connect/token \
  -d "client_id=admin-cli" \
  -d "username=testuser" \
  -d "password=test123" \
  -d "grant_type=password"
```

**运行结果**：服务端返回JSON格式的Token响应，关键字段如下：

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "expires_in": 300,
  "refresh_expires_in": 1800,
  "refresh_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "not-before-policy": 0,
  "session_state": "a1b2c3d4-5678-90ab-cdef-1234567890ab",
  "scope": "email profile"
}
```

**字段解读**：
- `access_token`：JWT格式的访问令牌，默认有效期5分钟
- `refresh_token`：刷新令牌，可用于获取新的Access Token而无需重新输入密码
- `expires_in`：Access Token剩余有效时间（秒）
- `refresh_expires_in`：Refresh Token剩余有效时间（秒）

**测试验证**：成功返回`access_token`即表示Keycloak安装、Realm配置、用户创建和认证流程全部正常。可以进一步通过解析Access Token来验证用户信息：

```bash
# 使用jq解析Token（跳过签名校验，仅查看Payload）
echo "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9..." | cut -d'.' -f2 | base64 -d 2>/dev/null | jq .
```

从Payload中可以看到`sub`（subject）字段为用户的UUID，`preferred_username`为`testuser`。

### 可能遇到的坑

**坑1：Docker端口冲突**

如果8080端口已被其他程序占用（如Jenkins、Tomcat），Docker容器启动时会报错。解决方案：改用其他端口映射，例如`-p 9090:8080`。

```bash
docker run -d --name keycloak-dev -p 9090:8080 \
  -e KC_BOOTSTRAP_ADMIN_USERNAME=admin \
  -e KC_BOOTSTRAP_ADMIN_PASSWORD=admin \
  quay.io/keycloak/keycloak:26.1 start-dev
```

访问时使用`http://localhost:9090/admin`。

**坑2：Windows WSL2内存不足**

在Windows上Docker Desktop默认依赖WSL2后端。如果`.wslconfig`中分配的内存过小（如1GB），Keycloak启动时会OOM崩溃。在用户目录`%USERPROFILE%\.wslconfig`中调整配置：

```ini
[wsl2]
memory=4GB
processors=2
```

修改后执行`wsl --shutdown`使配置生效，再重启Docker Desktop。

**坑3：启动后无法访问**

容器状态显示Running但浏览器无法打开页面。原因通常是Keycloak内部仍在初始化。使用`docker logs -f keycloak-dev`观察日志，等待`Listening on`字样出现后再访问。正常初始化需要15-30秒。

**坑4：Chrome安全策略阻止HTTP**

某些Chrome版本会对`127.0.0.1`启用HTTPS升级策略。解决方案：始终使用`http://localhost:8080`而非`http://127.0.0.1:8080`。

## 4 项目总结

### 优点与缺点对比

| 维度 | Keycloak | 自建认证系统 | Auth0 | Okta |
|------|----------|-------------|-------|------|
| 部署方式 | Docker/裸机/K8s自主托管 | 自主开发部署 | SaaS云服务 | SaaS云服务 |
| 成本 | 免费开源（Apache 2.0） | 人力成本2-3月开发 | 免费额度后按MAU计费 | 按用户数订阅 |
| OAuth2/OIDC支持 | 完整支持，开箱即用 | 需自行实现 | 完整支持 | 完整支持（行业标杆） |
| 社交登录 | 内置20+平台适配器 | 需逐一对接 | 内置支持 | 内置支持 |
| 自定义程度 | 极高（SPI扩展机制） | 完全可控 | 受平台限制 | 受平台限制 |
| 运维成本 | 需要自运维数据库和集群 | 需要长期维护 | 零运维 | 零运维 |
| 数据控制 | 完全自主 | 完全自主 | 数据在第三方 | 数据在第三方 |

### 适用场景

1. **快速搭建企业认证中心**：中小型企业需要统一的SSO认证平台，Keycloak能在数小时内完成部署和基础配置
2. **学习OAuth2/OIDC**：Keycloak是学习开放标准的绝佳工具，所有协议细节都可通过实际操作直观理解
3. **微服务认证底座**：作为服务网格的认证基础设施，为所有微服务提供统一的Token签发和校验能力
4. **多租户SaaS平台**：Realm机制天然支持多租户隔离，每个租户拥有独立的用户和角色体系
5. **自建IAM中台**：对数据合规性有严格要求的企业（金融、医疗），通过Keycloak构建完全自主可控的身份管理中台

**不适用场景**：极小型单系统（引入Keycloak反而增加架构复杂度）；对延迟极其敏感的超高频Token校验场景（微秒级需求需自行实现轻量校验层）。

### 注意事项

1. **开发模式禁用于生产**：`start-dev`模式下关闭了多项安全检查，使用H2文件数据库，绝对不可暴露在公网
2. **Bootstrap Admin密码必须立即修改**：生产环境中环境变量可能被日志记录或错误暴露，部署后需立即在Admin Console中修改密码
3. **数据库选择**：生产环境必须使用PostgreSQL/MySQL等外部数据库。H2不支持集群部署，并发能力弱，崩溃后恢复困难
4. **HTTPS强制**：生产环境必须在反向代理后配置TLS终止（Nginx/Caddy），Keycloak不应直接暴露HTTP端口

### 常见踩坑经验

| 故障现象 | 根因分析 | 解决方案 |
|---------|---------|---------|
| 容器不断重启，日志显示`OOM Killer` | JVM堆内存默认可能超过容器限制 | 增加容器内存限制：`-m 2g`；或设置JVM参数：`-e JAVA_OPTS_APPEND="-Xms512m -Xmx512m"` |
| 登录后立即跳回登录页 | 未配置HTTPS导致Cookie被浏览器拒绝 | 生产环境必须启用TLS；开发环境可设置`--proxy-headers forwarded`并通过Nginx代理 |
| 密码重置后无法登录 | Temporary开关未关闭，用户被要求修改密码但跳过了Required Action页面 | 创建测试用户时务必关闭Temporary开关，或通过Admin Console手动完成Required Action |
