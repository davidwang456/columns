# 第21章：数据库选型调优与多数据中心架构

## 1 项目背景

某跨境电商平台在全球8个区域运营，核心认证系统采用Keycloak。2024年"黑五"零点，QPS从平日的800飙升到12000——支付接口、购物车刷新、优惠券领取等场景同时触发大量Token创建和鉴权请求。不到30秒，运维告警平台炸锅：Nginx upstream大量报502/504，数据库连接池耗尽，用户登录页面空白，订单转化率断崖式下跌。

事后排查发现，问题出在"数据库"这一层。初期团队图省事，直接用Keycloak自带的H2文件数据库上线——测试环境跑得欢，生产用户量突破1万后，频繁出现**锁等待**和**OOM**。H2是嵌入式进程内数据库，所有请求共享一个`.mv.db`文件读写锁，并发超过50就开始排队。紧急迁移到MySQL 5.7后，又踩了新坑：默认字符集为`latin1`，Realm配置中存储的中文名称直接变成一连串问号。这一路折腾，团队终于下定决心迁移到PostgreSQL，并同时响应CEO的要求——在新加坡和法兰克福两个数据中心各自部署Keycloak集群，实现**用户就近认证**。

但挑战远不止换一个数据库引擎这么简单。核心痛点包括：

**痛点一：数据库选型的"银色子弹"不存在。** MySQL和PostgreSQL各有优劣，不是简单看两个Benchmark就能定论。MySQL在简单读场景的吞吐量更高（轻量级线程模型），而PostgreSQL的写场景和复杂查询更稳定（MVCC实现更健壮）。Keycloak官方文档将PostgreSQL列为推荐数据库，但很多企业已有MySQL DBA团队和运维体系，硬切数据库意味着人力成本翻倍。

**痛点二：大表查询性能坍塌。** `EVENT_ENTITY`表记录每次登录、注册、Token刷新事件，日增百万行。不做优化的情况下，一个`SELECT COUNT(*) FROM event_entity WHERE realm_id = ?`就能把数据库CPU打到90%以上。

**痛点三：连接池配置不当。** Keycloak默认HikariCP连接池最大100连接，在12000 QPS面前瞬间耗尽。简单的调大数字不能解决问题——PostgreSQL每个连接消耗2-4MB共享内存，连接数翻倍意味着`shared_buffers`被挤占，反而降低整体吞吐。

**痛点四：跨数据中心数据一致性。** 新加坡Primary数据库写入后，法兰克福Replica需要通过广域网异步复制——延迟200-400ms。在这段窗口内，用户在新加坡登录后立即访问法兰克福节点，可能看到"用户不存在"。这本质上是CAP定理在Keycloak场景下的现实映射。

**痛点五：备份策略的两难。** 全量备份耗时长（EVENT_ENTITY几十GB），增量备份依赖WAL归档，任何归档中断都可能丢失数据。

## 2 项目设计——剧本式交锋对话

**小胖**（端着刚冲好的速溶咖啡，屏幕上还开着昨晚的故障报告）：大师，我这回是真糊涂了。不都是"存数据"吗？H2、MySQL、PostgreSQL，不就跟买车一样——能跑就行呗？代步用H2、家用轿车用MySQL、越野车用PostgreSQL？不都是四个轱辘嘛，为啥不随便选一个完事儿？

**大师**（把他的咖啡杯挪到一边，在白板上画了三列）：你这个比喻方向没错，但细节差太远。H2是电单车——一个人骑着上下班（本地开发）可以，上高速（生产环境）违法。MySQL像丰田凯美瑞——好开、省油、保有量大，全家出行（常规OLTP）绰绰有余，但你要拖个房车（复杂分析查询、JSON操作）就费劲。PostgreSQL是路虎卫士——带差速锁、空气悬挂、底盘升降，越野调教让你处理JSONB用户属性、全文搜索、复杂事务都游刃有余。

> **大师技术映射**：H2 → 嵌入式单文件FS锁，并发超50即瓶颈。MySQL → 轻量级连接线程+fdatasync写入策略，读密集场景优秀。PostgreSQL → 进程模型+MVCC+丰富索引类型（GiST/GIN/BRIN）+JSONB，适配Keycloak自定义属性场景。

---

**小白**（把他那一本磨旧的《数据库系统实现》合上）：大师，我查了Keycloak官方文档——PostgreSQL被标记为"推荐数据库"。但具体好在哪里？我看到一个关键点：Keycloak的自定义用户属性在数据库里用`user_attribute`表存EAV（实体-属性-值）模型，这跟JSONB什么关系？还有，连接池——我们配了HikariCP，为什么还要加pgBouncer？

**大师**：问得好，分三个问题答。

**第一，JSONB的价值。** Keycloak的用户、组、角色都支持自定义属性，传统做法是EAV模式——`user_attribute(name, value, user_id)`，查"所有department=Engineering且age>30的用户"要三表JOIN，SQL写得又长又慢。如果你用PostgreSQL，可以在UserRepresentation的`attributes`字段映射到JSONB列，一个简单索引就能搞定：`CREATE INDEX idx_user_attrs ON user_entity USING GIN (attributes jsonb_path_ops)`。而且，JSONB支持原生SQL查询——`SELECT * FROM user_entity WHERE attributes @> '{"department": "Engineering"}'`，不用写复杂的EAV自连接。

**第二，双层连接池架构。** HikariCP在Keycloak这一侧做**数据库连接复用**——Keycloak的请求线程从连接池借走连接，用完还回去。但PostgreSQL的进程模型意味着每建一个新连接都要fork一个进程，代价很大。pgBouncer在PostgreSQL前面充当**连接多路复用器**——成百上千的客户端连接被聚合成少量PostgreSQL后端连接，把进程fork的开销降到最低。HikariCP是Keycloak侧的"本地管家"，pgBouncer是数据库侧的"统一前台"。两层之间的分工：HikariCP池大小设为20-40（应用侧快速复用），pgBouncer的`default_pool_size=25`、`max_db_connections=50`（数据库侧限制后端连接数）。

**第三，PostgreSQL的其他专有优势。** 全文搜索（`tsvector`类型）让Keycloak用户搜索更快，强事务支持（SSI可序列化隔离级别）保障Realm配置变更的原子性，以及最关键的——PostgreSQL的流复制比MySQL的异步复制更成熟、监控更到位。

> **大师技术映射**：HikariCP → 公司内部每个部门有自己的助理，帮忙预约会议室（连接）。pgBouncer → 公司大楼前台的统一接线员，200个外来电话被汇聚到20条内部线路。双层池化 → 助理先用自己部门的会议室配额，再由前台统一调配整栋楼的会议室资源。

---

**小胖**（吞下最后一口饼干，指着白板上的"多数据中心"四个字）：等一下，新加坡和法兰克福两个DC，各放一套Keycloak集群，那数据库怎么办？两套独立数据库各管各的？那用户在新加坡注册了，去法兰克福出差一登录——提示"用户不存在"，这不炸了吗？

**小白**（抢先回答）：这正是CAP定理的现实场景。两个数据中心之间网络延迟200ms+，如果我们选择CP（强一致性+分区容错），那么当新加坡写入后，法兰克福必须同步确认才能返回——这200ms就会卡在每次登录请求里，用户体验极差。而Keycloak的认证场景对一致性的要求其实不高——用户属性变更、密码修改这些操作频率远低于登录验证。

**大师**：小白说到了关键。Keycloak的场景优先选**AP（可用性+分区容错，牺牲强一致性）**。具体架构：新加坡部署Primary PostgreSQL + 3节点Keycloak集群（读写），法兰克福部署Standby PostgreSQL（流复制只读副本）+ 2节点Keycloak集群（读写，但指向法兰克福本地只读库）。DNS使用Geo-Routing——`auth.company.com`在亚太解析到新加坡LB，欧洲解析到法兰克福LB。关键机制：用户注册/密码变更写操作全部路由到新加坡Primary，认证/Token刷新读操作就近读取本地只读副本。跨站点Session同步由Infinispan的Cross-Site Replication实现（`backup site="fra" strategy="ASYNC"`）——新加坡的登录Session异步同步到法兰克福，用户跨区域飞过去也能无缝认证。

> **大师技术映射**：CAP取舍 → 银行异地取款。你想在纽约开户、香港取钱——AP模型允许你立即取款（基于香港分行的缓存数据），但可能账户余额有几分钟的延迟。CP模型要求每一笔操作都实时同步确认——那到了香港就得等总行上班才能取。

---

**小胖**（第二轮，看着昨晚的备份日志发愁）：大师，那个EVENT_ENTITY表都快100GB了，每天全量备份5个小时还没跑完，增量备份的WAL文件又怕丢，这备份策略到底怎么定？

**大师**：备份要分层设计。EVENT_ENTITY这类日志大表，第一要务是**分区归档**——按月分区，超过6个月的分区可以直接移到归档表或冷存储（S3），在active表上只保留最近数据。全量备份只做`--exclude-table=event_entity_archive_*`——这意味着只备份"热数据"。增量备份依赖PostgreSQL的WAL归档（Continuous Archiving）：`archive_command = 'cp %p /backups/wal/%f'`实现PITR（时间点恢复），即使全量备份是周日的，你可以恢复到周三下午14:32的任意状态。每周日全量 + 每日WAL归档 + 每月归档清理，三管齐下。

**小白**（追问）：故障切换怎么演练？如果新加坡机房真挂了，法兰克福能自动接替吗？

**大师**：PostgreSQL的流复制天然支持Failover——配合Patroni/Stolon这类HA工具，当Primary失联超过`failover_timeout`（建议30秒）后，自动将法兰克福的Standby提升为Primary。但Keycloak层面需要注意：切换后KC_DB_URL需要指向新Primary（或通过HAProxy hold一个虚拟IP）。生产环境建议做**每季度一次的故障切换演练**——在非业务高峰时段手动终止新加坡PostgreSQL进程，观测法兰克福Standby的升主时间和Keycloak的重连恢复时间，目标RTO ≤ 60秒。

> **大师技术映射**：备份分层 → 家里重要文件分类存放——户口本（Realm配置）放保险柜（全量备份+WAL），水电费账单（EVENT_ENTITY老数据）可放储物间（冷归档），丢了能补办、不致命。故障切换演练 → 消防演习，平时不走一遍，真着火时肯定手忙脚乱。

### 步骤1：PostgreSQL性能配置优化

在PostgreSQL容器的`/etc/postgresql/postgresql.conf`（或通过volume挂载自定义配置）中覆盖以下参数：

```ini
# postgresql.conf —— 生产推荐配置（假设物理内存8GB）
max_connections = 200                 # 总连接数上限，注意与pgBouncer联动
shared_buffers = 2GB                  # 物理内存的25%
effective_cache_size = 6GB            # 物理内存的75%，引导规划器选择索引扫描
work_mem = 64MB                       # 单查询排序/哈希操作内存
maintenance_work_mem = 512MB          # VACUUM/CREATE INDEX等维护操作内存
wal_buffers = 64MB                    # WAL写入缓冲区，高并发写入建议64MB
wal_level = replica                   # 必须 ≥ replica 才能启用流复制
max_wal_senders = 5                   # 允许的最大流复制连接数
wal_keep_size = 1024                  # 保留1GB WAL文件防止复制槽断连后丢失

# 查询优化（SSD环境）
random_page_cost = 1.1                # 默认4.0是HDD设定，SSD降为1.0-1.5
effective_io_concurrency = 200        # SSD的并发IO能力
enable_partitionwise_aggregate = on   # 分区表并行聚合
enable_partitionwise_join = on        # 分区表并行JOIN

# 日志（生产排障的关键）
log_statement = 'ddl'                 # 记录所有DDL变更（CREATE/ALTER/DROP）
log_min_duration_statement = 1000     # 记录超过1秒的慢查询
log_line_prefix = '%t [%p-%l] %q%u@%d '   # 时间戳+进程ID+用户+数据库
log_checkpoints = on                  # 记录检查点信息（判断写入压力）
log_autovacuum_min_duration = 1000    # 记录超过1秒的自动VACUUM操作
```

**Docker Compose挂载配置**：

```yaml
postgres:
  image: postgres:16-alpine
  volumes:
    - ./configs/postgresql.conf:/etc/postgresql/postgresql.conf:ro
    - pgdata:/var/lib/postgresql/data
  command: >
    -c config_file=/etc/postgresql/postgresql.conf
```

### 步骤2：数据库表优化——创建复合索引

连接到Keycloak数据库，执行以下索引创建语句：

```sql
-- 加速用户搜索（按邮箱/用户名 + Realm维度快速定位用户）
CREATE INDEX IF NOT EXISTS idx_user_email
    ON user_entity(email, realm_id);
CREATE INDEX IF NOT EXISTS idx_user_username
    ON user_entity(username, realm_id);

-- 加速用户属性查询（name + value 联合筛选，比如查所有 department=Engineering）
CREATE INDEX IF NOT EXISTS idx_user_attr
    ON user_attribute(name, value, user_id);

-- 加速客户端查找（client_id 是OAuth2 client_credentials流程的热点查询）
CREATE INDEX IF NOT EXISTS idx_client_client_id
    ON client(client_id, realm_id);

-- 加速事件查询 —— 大表优化的核心！按 Realm 分区 + 按时间倒序排序
CREATE INDEX IF NOT EXISTS idx_event_time
    ON event_entity(realm_id, event_time DESC);
CREATE INDEX IF NOT EXISTS idx_event_type
    ON event_entity(realm_id, type, event_time DESC);
CREATE INDEX IF NOT EXISTS idx_event_user
    ON event_entity(realm_id, user_id, event_time DESC);

-- 加速会话查询（client_session 表在 Token 刷新时高频读取）
CREATE INDEX IF NOT EXISTS idx_client_session
    ON client_session(client_id, user_session_id);
```

**注意**：索引不是越多越好。每条索引都会拖慢INSERT/UPDATE的速度。建议在生产环境对`event_entity`这类高写入表，先用`EXPLAIN ANALYZE`验证慢查询路径，再有针对性地创建索引。可以使用`pg_stat_user_indexes`视图监控索引使用率，对从未被扫描的索引果断删除。

### 步骤3：分区表——EVENT_ENTITY按月分区

`event_entity`表日增百万行，是全表扫描的重灾区。采用PostgreSQL原生声明式分区，按月切割。

```sql
-- Step 1: 创建分区主表（结构与原表一致）
CREATE TABLE event_entity_partitioned (
    id              VARCHAR(36)  NOT NULL,
    client_id       VARCHAR(255),
    details_json    VARCHAR(2550),
    error           VARCHAR(255),
    ip_address      VARCHAR(255),
    realm_id        VARCHAR(255),
    session_id      VARCHAR(255),
    event_time      BIGINT       NOT NULL,
    type            VARCHAR(255),
    user_id         VARCHAR(255)
) PARTITION BY RANGE (event_time);

-- Step 2: 将 event_time（毫秒时间戳）转换为日期范围，按月创建分区
-- PostgreSQL分区键需要能映射到日期范围，因此先把 event_time 转化为 epoch 秒
-- 每月范围: 2024-01-01 00:00:00 = 1704067200 秒 → event_time = 1704067200000 毫秒

CREATE TABLE event_entity_2024_01 PARTITION OF event_entity_partitioned
    FOR VALUES FROM (1704067200000) TO (1706745600000);   -- 2024-01-01 到 2024-02-01
CREATE TABLE event_entity_2024_02 PARTITION OF event_entity_partitioned
    FOR VALUES FROM (1706745600000) TO (1709251200000);
CREATE TABLE event_entity_2024_03 PARTITION OF event_entity_partitioned
    FOR VALUES FROM (1709251200000) TO (1711929600000);
-- ... 以相同模式继续创建后续月份

-- Step 3: 在分区主表上创建索引（会自动继承到子分区）
CREATE INDEX ON event_entity_partitioned(realm_id, event_time DESC);
CREATE INDEX ON event_entity_partitioned(realm_id, type, event_time DESC);

-- Step 4: 数据迁移——将原表数据插入分区表
INSERT INTO event_entity_partitioned
SELECT * FROM event_entity
WHERE event_time >= 1704067200000;    -- 从2024年1月开始的数据

-- Step 5: 原子切换表名（建议在维护窗口执行）
BEGIN;
ALTER TABLE event_entity RENAME TO event_entity_old;
ALTER TABLE event_entity_partitioned RENAME TO event_entity;
-- 验证后再删除旧表
-- DROP TABLE event_entity_old;
COMMIT;
```

**自动化脚本——按月创建新分区**（crontab每月1日执行）：

```bash
#!/bin/bash
# add-monthly-partition.sh
NEXT_MONTH=$(date -d "$(date +%Y-%m-01) +1 month" +%s)000
TWO_MONTHS=$(date -d "$(date +%Y-%m-01) +2 month" +%s)000
LABEL=$(date -d "$(date +%Y-%m-01) +1 month" +%Y_%m)

psql -h $DB_HOST -U $DB_USER -d keycloak -c "
  CREATE TABLE IF NOT EXISTS event_entity_${LABEL}
    PARTITION OF event_entity
    FOR VALUES FROM ($NEXT_MONTH) TO ($TWO_MONTHS);
"
echo "Partition event_entity_${LABEL} created."
```

### 步骤4：pgBouncer连接池配置

```ini
# pgbouncer.ini
[databases]
keycloak = host=postgres port=5432 dbname=keycloak

[pgbouncer]
listen_addr = 0.0.0.0
listen_port = 6432
auth_type = scram-sha-256             # PostgreSQL 14+ 推荐 SCRAM 认证
auth_file = /etc/pgbouncer/userlist.txt

# 连接池模式 —— transaction 级别复用（每个事务结束后归还连接）
pool_mode = transaction
max_client_conn = 300                 # 最大接受来自 Keycloak 的客户端连接
default_pool_size = 25                # 每个用户/数据库的默认后端连接数
max_db_connections = 50               # 后端 PostgreSQL 的最大连接数（≤ max_connections）
reserve_pool_size = 5                 # 预留池，应对突发流量
reserve_pool_timeout = 3              # 预留池激活的超时时间（秒）

# 超时配置
server_idle_timeout = 600             # 后端连接空闲10分钟回收
client_idle_timeout = 600             # 客户端连接空闲10分钟断开
query_timeout = 30                    # 单条查询超过30秒自动取消
server_lifetime = 3600                # 后端连接最长存活1小时（防止内存泄漏）
server_connect_timeout = 15           # 建立后端连接的超时时间

# 管理界面
admin_users = admin
stats_users = admin

# 日志（生产排障）
log_connections = 1
log_disconnections = 1
```

**userlist.txt**（SCRAM密码通过`pgbouncer -v`生成）：

```
"admin" "SCRAM-SHA-256$4096:..."
"keycloak" "SCRAM-SHA-256$4096:..."
```

**Keycloak配置指向pgBouncer**：

```yaml
keycloak:
  environment:
    KC_DB: postgres
    KC_DB_URL: jdbc:postgresql://pgbouncer:6432/keycloak
    KC_DB_USERNAME: keycloak
    KC_DB_PASSWORD: keycloak123
    KC_DB_POOL_MAX_SIZE: 40           # HikariCP 应用侧连接池
    KC_DB_POOL_MIN_SIZE: 5
    KC_DB_POOL_INITIAL_SIZE: 5
```

**连接池总量验算公式**：

```
Keycloak节点数 × KC_DB_POOL_MAX_SIZE ≤ pgBouncer max_client_conn
pgBouncer max_db_connections ≤ PostgreSQL max_connections
假设3节点 × 40 = 120 ≤ 300（pgBouncer客户端连接） ✓
pgBouncer后端连接50 ≤ PostgreSQL max_connections=200 ✓
```

### 步骤6：自动备份脚本

```bash
#!/bin/bash
# daily-backup.sh —— 全量+增量分层备份
set -euo pipefail

DB_HOST="${DB_HOST:-postgres}"
DB_USER="${DB_USER:-keycloak}"
DB_PASSWORD="${DB_PASSWORD:-keycloak123}"
BACKUP_BASE="/backups/keycloak"
DATE_DIR="$(date +%Y%m%d)"
BACKUP_DIR="${BACKUP_BASE}/${DATE_DIR}"
RETENTION_DAYS=30

mkdir -p "$BACKUP_DIR"

# 全量备份（每周日凌晨2点执行，通过crontab控制）
if [ "$(date +%u)" -eq 7 ]; then
    echo "[$(date)] Starting full backup..."
    PGPASSWORD="$DB_PASSWORD" pg_dump \
        -h "$DB_HOST" -U "$DB_USER" -d keycloak \
        -Fc --clean --no-owner --verbose \
        --exclude-table-data='event_entity_*_archive' \
        -f "$BACKUP_DIR/keycloak-full-${DATE_DIR}.dump"
    echo "[$(date)] Full backup completed."
fi

# WAL归档（应在postgresql.conf中配置 archive_command）
# archive_command = 'cp %p /backups/wal/%f'
# 无需脚本干预，PostgreSQL自动将已完成的WAL段复制到归档目录
# 但需要定期清理过期WAL：

# 清理30天前的备份和归档
find "$BACKUP_BASE" -type f -mtime +"$RETENTION_DAYS" -delete
find "/backups/wal" -type f -mtime +7 -delete   # WAL保留7天即可（有全量备份兜底）

echo "[$(date)] Backup rotation completed."
```

**crontab配置**：

```
0 2 * * 0 /scripts/daily-backup.sh    # 每周日凌晨2点执行全量备份
0 3 * * * /scripts/daily-backup.sh    # 每天凌晨3点执行WAL轮转清理
```

### 可能遇到的坑

**坑1：pgBouncer的`pool_mode=transaction`与Prepared Statement冲突。** Keycloak的JDBC驱动默认可能使用Prepared Statement，而pgBouncer在transaction模式下不保留Prepared Statement的状态。解决方案有两个：一是在Keycloak JDBC URL中添加`?prepareThreshold=0`禁用服务端预处理；二是将pgBouncer的`pool_mode`改为`session`（牺牲部分连接复用效率换取兼容性）。

**坑2：分区表建立后，旧数据需手动迁移。** 如果原`event_entity`表已有数亿行数据，直接`INSERT INTO ... SELECT`会在一个长事务中锁住源表。建议分批迁移：每次迁移10万行，`VACUUM`后再继续。

**坑3：跨数据中心复制延迟导致"幽灵数据"。** 新加坡写入后，法兰克福读副本有200-400ms延迟。如果用户在新加坡注册后立刻访问法兰克福节点，可能看到"用户不存在"。解决方案：写操作（注册、修改密码）始终路由到新加坡Primary，法兰克福节点仅处理读操作（认证、Token验证）。

**坑4：WAL归档撑满磁盘。** 如果`archive_command`返回失败（比如NFS挂载掉了），PostgreSQL会一直保留WAL段不删除，直到`pg_wal`目录撑满磁盘，数据库进入只读模式。务必监控WAL目录使用率并设置告警阈值（>80%立即通知）。

## 4 项目总结

### 优点与缺点对比：Keycloak数据库选型

| 维度 | PostgreSQL | MySQL | Oracle | MSSQL |
|------|-----------|-------|--------|-------|
| Keycloak官方支持 | **首选推荐** | 支持 | 支持 | 支持 |
| JSON/自定义属性查询 | JSONB + GIN索引，原生SQL | JSON类型（5.7+），无索引加速 | JSON支持（12c+） | JSON支持（2016+） |
| 全文搜索 | `tsvector`内建 | 需外部引擎（Sphinx/ES） | Oracle Text | 全文索引 |
| 事务隔离 | SSI可序列化（无锁） | 间隙锁+MVCC | 强一致+锁升级 | 快照隔离 |
| 流复制成熟度 | 物理/逻辑流复制，成熟 | 异步/半同步，需额外配置 | Data Guard（企业版$$$） | Always On（企业版$$$） |
| 分区表 | 声明式分区（PG 10+） | 需手动分区或8.0+ | 分区选项丰富 | 分区表 |
| 运维成本 | 免费开源，社区活跃 | 免费开源，社区最大 | 企业授权费昂贵 | 企业授权费昂贵 |
| 连接模型 | 进程模型（需pgBouncer） | 线程模型（轻量级） | 进程+线程混合 | 线程模型 |
| DBA技能门槛 | 中 | 低（普及度高） | 高（需Oracle认证） | 中 |

### 适用场景

- **PostgreSQL（通用推荐）**：Keycloak生产环境首选。JSONB适配自定义属性，流复制+逻辑复制支持多DC。无需外部授权费的商业化友好数据库。
- **MySQL**：企业已有MySQL DBA团队和运维体系的场景。读密集场景表现优秀，但复杂查询和JSON操作弱于PostgreSQL。
- **Oracle**：金融、政府等合规要求严格的环境（等保三级可能要求使用商业数据库）。Oracle Data Guard的灾备方案经过大量金融级验证。
- **MSSQL**：Azure + .NET技术栈企业，Keycloak与AAD集成时天然选MSSQL。
- **不适用场景**：H2绝对不能用于生产环境。SQLite不支持并发写入，不适用于任何服务端场景。

### 注意事项

1. **连接池上限验算。** 部署前必须用公式 `Keycloak节点数 × KC_DB_POOL_MAX_SIZE ≤ pgBouncer max_client_conn ≤ PostgreSQL max_connections` 验算，确保不会出现连接泄漏撑爆数据库。
2. **分区表运维自动化。** 每月必须运行`add-monthly-partition.sh`添加新分区，同时归档/删除超过保留期的旧分区。忘记加分区会导致INSERT失败——数据写入找不到对应分区。
3. **索引监控。** 使用`pg_stat_user_indexes`和`pg_stat_user_tables`定期检查索引使用率，删除冗余索引（每条索引拖慢写入速度约5-15%）。
4. **WAL归档监控。** `pg_wal`目录绝对不能超过磁盘容量的50%，设置告警并及时转储或清理。

### 常见踩坑经验

**故障一：连接池泄漏。** 运维发现PostgreSQL连接数持续增长，最终触达`max_connections`。排查发现原因是HikariCP的`connectionTimeout`设置过长（30000ms），网络瞬断时连接未及时回收，新的请求不断申请新连接。修复：将`connectionTimeout`降为10000ms，配置`leakDetectionThreshold=10000`启用泄漏检测。

**故障二：索引选择性太低导致全表扫描。** 在`event_entity.type`上建了单列索引，但由于该列取值只有有限的几种（LOGIN、REGISTER、TOKEN_REFRESH等），选择性极低。查询优化器直接忽略索引走Seq Scan。修复：改为复合索引`(realm_id, type, event_time)`，先按Realm大幅缩小范围，再按类型和时间过滤。

**故障三：备份恢复遇到版本兼容问题。** 本地开发环境用PostgreSQL 17的`pg_dump`导出，生产环境PostgreSQL 16的`pg_restore`恢复时报语法错误。修复：始终使用目标数据库版本的`pg_dump`和`pg_restore`工具，或在CI中统一PostgreSQL版本。

### 思考题

1. **如果PostgreSQL主从复制延迟达到5秒，Keycloak用户可能遇到什么问题？如何检测和处理这种延迟？**
   - 提示：考虑用户在新加坡重置密码后，立即使用法兰克福节点登录的场景。检测手段包括`pg_stat_replication`视图的`write_lag`和`replay_lag`字段。处理方案包括在Keycloak端对写后读操作强制路由到Primary。

2. **100个Realm的SaaS场景下，是共用一个大型PostgreSQL集群，还是按Realm分库？**
   - 提示：权衡数据库连接管理开销（共享集群：一组连接池服务所有Realm vs 分库：每个Realm独立连接池）、备份恢复粒度（分库可单Realm恢复）、资源隔离（一个Realm的慢查询不应影响其他租户）、DBA运维成本（100个库的Schema升级工作量巨大）。
