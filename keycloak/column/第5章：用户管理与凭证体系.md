# 第5章：用户管理与凭证体系

## 1 项目背景

某公司完成Keycloak初步部署后，IT部门面临一个棘手任务：将现有的1000多名员工用户数据迁移到Keycloak。这些员工分布在研发部、市场部、财务部等不同部门，每个部门对用户属性有不同要求——研发部需要关联GitHub账号，财务部需要录入身份证号，市场部则需要记录负责区域。与此同时，公司管理层提出了明确要求：新员工入职时，IT通过脚本批量创建账号并自动分配部门角色；员工离职后24小时内自动禁用账号；所有用户首次登录时必须强制修改初始密码并绑定手机号。

手动创建1000多个用户不仅耗时而且极易出错。想象一下，IT管理员对着Excel表格逐行复制粘贴用户名、邮箱，再逐个配置密码策略和部门归属——哪怕每天处理50个用户，也需要连续工作三周，期间任何拼写错误都可能导致员工无法正常登录。更严重的是，用户属性散落在HR系统、OA系统、门禁系统等各个业务线中，数据一致性完全无法保证：一个员工在HR系统改了名字，但Keycloak中的firstName仍然保持旧值，导致邮件地址与显示名称不匹配。

用户生命周期管理的缺失则带来了安全隐患。员工离职后，HR在内部系统标记了"已离职"，但Keycloak中的账号依旧处于启用状态。这些"幽灵账号"成为攻击者眼中的薄弱环节——只要猜中一个弱密码，就能以合法身份进入企业系统。去年某安全审计报告就指出，该公司至少有37个离职三个月以上但仍处于活跃状态的账号，其中两个还在离职后被登录过。密码策略方面更是各自为战：研发系统要求8位以上含特殊字符，OA系统只检查6位长度，财务系统甚至允许纯数字密码。用户在三个系统间疲于记忆不同规则，最终选择了"Password123!"这样的"万能密码"，使所有策略形同虚设。

本章将围绕Keycloak的用户管理与凭证体系展开，解决上述痛点：如何建立统一的用户目录、如何定义自定义属性体系、如何利用Group实现组织架构映射、如何通过Required Actions确保首次登录安全、以及如何配置全局密码策略杜绝弱密码。最终目标是在Keycloak中构建一套从入职到离职的全生命周期用户管理方案。

## 2 项目设计——剧本式交锋对话

**小胖**（抱着一袋薯片推开会议室门）：大师，小白，我昨天去档案馆查资料，突然想到一个比喻——Keycloak的用户管理，不就相当于学校的学生档案室嘛！所有学生的信息统一存放在档案室，学号是唯一标识，班级归属决定了你能进哪间教室。每个学生的档案里既有固定字段（姓名、性别、出生日期），也可以自己附加材料（获奖证书、体检报告）。这不就是Keycloak的用户管理吗？

**大师**（放下咖啡杯）：小胖这个比喻出乎意料地准确。Keycloak中的每个用户就像一份档案，username是学号，email是联系方式，Group是班级归属，Role是图书证或实验室钥匙，而attributes就是那些额外的附加材料。学校档案室的索引卡系统对应Keycloak的用户搜索机制，档案的借阅归还对应会话管理。

**小白**（在白板上画着）：那我有三个问题。第一，自定义属性到底能存什么？能不能存数组或者JSON？第二，Group和Role到底有什么区别——我昨天把Group当成了Role用，结果权限乱套了。第三，Required Actions的执行时机是怎么控制的？比如我想让用户先改密码、再验证邮箱、最后才配置OTP，这个顺序能保证吗？

**大师**：问得好，逐一解答。

首先是自定义属性体系。Keycloak的用户属性本质上是一个`Map<String, List<String>>`结构——每个属性名对应一个字符串列表。这意味着单个属性可以存多个值，比如`department: ["研发部", "AI实验室"]`表示一个跨部门员工。属性值支持任意字符串，你可以存储JSON字符串来实现复杂结构，比如`metadata: {"github": "zhangsan-dev", "employeeId": "EMP001", "入职日期": "2023-06-15"}`。但注意，属性不支持嵌套索引查询——你不能直接查"所有GitHub账号为zhangsan-dev的用户"，需要读取后自行解析。

第二是Group与Role的核心区别。Group是"组织结构"，代表"谁是谁的下属、谁和谁在一个部门"。它支持树形层级——公司总部→研发部→后端组，子组可以继承父组的属性（比如研发部所有人自动拥有`department: 研发部`属性）。Role是"权限集合"，代表"能干什么、不能干什么"。比如`developer`角色可以访问代码仓库，`viewer`角色只能查看。一个经典判断：如果你在画组织架构图，用Group；如果你在画权限矩阵，用Role。一个用户既可以属于"研发部/后端组"的Group，又同时拥有`developer`和`gitlab-user`两个Role。

**小胖**：那我能不能直接把Group映射成Role？比如"后端组"的人自动获得`developer`角色？

**大师**：当然可以，这正是Keycloak推荐的做法。在Group设置中配置Default Roles，加入该组的用户自动获得角色。这叫"组织驱动权限"——你不需要给每个新人单独分配角色，只要把他扔进正确的Group就行。

现在说第三点，Required Actions的执行时机。当用户登录时，Keycloak会检查该用户是否还有未完成的Required Actions。如果有，认证流程会"暂停"，按配置顺序依次弹出相应的操作页面。顺序由你添加时的先后决定：先添加"Update Password"再添加"Configure OTP"，用户就会先看到改密码页面，改完后才跳到OTP配置页面。所有Required Actions完成后，用户才能到达目标应用。关键细节：`temporary`密码标记——创建用户时将密码标记为temporary=true，Keycloak会自动将"Update Password"加入Required Actions列表，无需手动配置。

**小白**：明白了。那用户禁用和删除的策略怎么选？还有批量导入时会不会有并发问题？

**大师**：用户禁用和删除是两种不同哲学。禁用（enabled=false）是"软操作"：用户无法登录，但所有数据（角色、组、会话历史）完整保留，随时可以恢复。适用于离职交接期、暂时停职等场景。删除（DELETE）是"硬操作"：数据不可逆地消失，但出于数据库引用完整性考虑，Keycloak并不会物理删除用户记录，而是在数据库中标记删除。生产环境的最佳实践是：员工离职→立即禁用→90天后确认无遗留业务→再执行删除。

批量导入方面，Keycloak Admin REST API支持单用户创建端点`POST /admin/realms/{realm}/users`。对于1000+用户的批量导入，核心策略是控制并发——建议每批10-20个用户，批次间间隔500ms，避免触发Brute Force Detection误判和数据库连接池耗尽。另一个容易被忽视的细节：自定义属性的值必须是数组格式`["value"]`而非字符串`"value"`，否则API会返回400错误。

**大师总结技术映射**：
- 学生档案室 → Keycloak User Management：统一用户目录
- 档案的年级班级字段 → Group：用户组织归属
- 图书证借阅权限 → Role：权限集合
- 档案附加材料 → Attributes：自定义属性，支持单值/多值/JSON
- 转学前归还图书 → Required Actions：登录前必须完成的强制操作

## 3 项目实战

### 步骤2：配置密码策略

**目标**：在Realm级别配置全局密码策略，确保所有用户密码满足安全基线。

操作路径：`Admin Console → demo-realm → Realm Settings → Authentication → Password Policy`

点击**Add Policy**，依次添加以下策略：

| 策略 | 参数 | 说明 |
|------|------|------|
| Hashing Algorithm | pbkdf2-sha256 | 密码哈希算法，比默认pbkdf2-sha512性能均衡 |
| Minimum Length | 8 | 最小密码长度 |
| Uppercase Characters | 1 | 至少1个大写字母 |
| Lowercase Characters | 1 | 至少1个小写字母 |
| Digits | 1 | 至少1个数字 |
| Special Characters | 1 | 至少1个特殊字符 |
| Not Username | - | 密码不得包含用户名 |
| Password History | 3 | 密码历史保留，防止复用最近3个密码 |

**验证效果**：尝试为用户zhangsan设置弱密码`123456`。

在用户详情页点击Credentials标签 → Set Password，输入`123456`两次，Temporary开关保持On。点击Save，页面顶部出现红色错误提示：

```
Invalid password: must contain at least 1 uppercase characters, 1 lowercase characters, 
1 digits, 1 special characters, and minimum length 8.
```

再尝试设置`Zhangsan123!`——密码包含用户名"zhangsan"→触发Not Username策略拦截。最终设置`Welcome@2024`——符合所有规则，成功保存。

### 步骤3：创建组织架构Group

**目标**：利用Admin CLI创建层级化Group树，映射企业组织架构。

```bash
kcadm.sh create groups/公司总部/children/研发部/children -r demo-realm -s name=前端组

# 创建其他部门
kcadm.sh create groups/公司总部/children -r demo-realm -s name=市场部
```

**运行结果**：通过Admin Console的Groups菜单查看，树形结构如下：


### 步骤4：通过Admin REST API批量导入用户

**目标**：编写Python脚本从CSV文件批量创建用户，设置临时密码和自定义属性。

**CSV文件** (`users.csv`)：

```csv
username,email,firstName,lastName,department,employeeId,password
lisi,lisi@company.com,四,李,研发部,EMP002,Welcome@2024
wangwu,wangwu@company.com,五,王,市场部,EMP003,Welcome@2024
zhaoliu,zhaoliu@company.com,六,赵,财务部,EMP004,Welcome@2024
```

**Python脚本** (`batch_import.py`)：

```python
import requests
import json
import csv
import time

# ==================== 1. 获取Admin Token ====================
token_url = "http://localhost:8080/realms/master/protocol/openid-connect/token"
token_resp = requests.post(token_url, data={
    "client_id": "admin-cli",
    "username": "admin",
    "password": "admin",
    "grant_type": "password"
})
token_resp.raise_for_status()
admin_token = token_resp.json()["access_token"]

headers = {
    "Authorization": f"Bearer {admin_token}",
    "Content-Type": "application/json"
}

# ==================== 2. 从CSV批量创建用户 ====================
admin_url = "http://localhost:8080/admin/realms/demo-realm/users"

with open("users.csv", encoding="utf-8") as f:
    reader = csv.DictReader(f)
    for idx, row in enumerate(reader):
        user_data = {
            "username": row["username"],
            "email": row["email"],
            "firstName": row["firstName"],
            "lastName": row["lastName"],
            "enabled": True,
            "emailVerified": True,
            "attributes": {
                "department": [row["department"]],
                "employeeId": [row["employeeId"]]
            },
            "credentials": [{
                "type": "password",
                "value": row["password"],
                "temporary": True
            }]
        }

        resp = requests.post(admin_url, json=user_data, headers=headers)
        if resp.status_code == 201:
            print(f"[{idx+1}] Created user: {row['username']}")
        elif resp.status_code == 409:
            print(f"[{idx+1}] SKIP - already exists: {row['username']}")
        else:
            print(f"[{idx+1}] ERROR {resp.status_code}: {row['username']} - {resp.text}")

        # 避免API限流：每10个用户暂停0.5秒
        if (idx + 1) % 10 == 0:
            print("--- batch pause 500ms ---")
            time.sleep(0.5)

print("Done.")
```

**运行结果**：

```
[1] Created user: lisi
[2] Created user: wangwu
[3] Created user: zhaoliu
Done.
```

通过Admin Console → Users确认三个新用户已出现，且属性中`department`和`employeeId`正确显示。每个用户的状态为"Enabled"，点击Credentials可看到临时密码。

### 步骤5：配置Required Actions

**目标**：配置新用户的强制操作流程，确保首次登录安全。

方案一：针对特定用户配置。在Admin Console中进入用户zhangsan → Details → Required User Actions，从下拉列表选择并添加：

1. Update Password
2. Configure OTP
3. Verify Email

方案二：全局默认Required Actions（对所有新用户生效）。操作路径：`Realm Settings → Authentication → Required Actions`，将上述三项启用为Default Action。此方案适合批量场景——步骤4导入的所有用户将自动继承这些Required Actions。

**验证流程**：
1. 打开浏览器无痕模式，访问`http://localhost:8080/realms/demo-realm/account`
2. 使用`zhangsan / Welcome@2024`登录
3. 系统自动跳转到"Update Password"页面，提示输入新旧密码
4. 修改密码后，自动进入"Configure OTP"页面，提示扫描二维码
5. 用手机Authenticator App扫描、输入验证码确认
6. 最后进入"Verify Email"页面（如果邮件服务未配置，可在Admin Console中手动标记为已验证跳过）
7. 完成后跳转到账户管理首页

至此，用户zhangsan完成了完整的首次登录安全流程。

### 步骤6：用户加入Group并分配角色

**目标**：将用户zhangsan加入"后端组"，并自动继承该组的角色权限。

```bash
# 将用户zhangsan加入"后端组"
kcadm.sh update users/zhangsan/groups/后端组 -r demo-realm \
  -s realmRoles='["developer"]'

# 验证用户所属Group
kcadm.sh get users/zhangsan/groups -r demo-realm | jq '.[].path'
# 输出："/公司总部/研发部/后端组"

# 验证用户角色
kcadm.sh get users/zhangsan/role-mappings/realm -r demo-realm | jq '.[].name'
# 输出："developer"
```

**Group角色继承机制**：如果为"后端组"配置了Default Roles，则加入该组的任何用户都会自动获得这些角色，无需手动分配。配置方式：在Admin Console中进入后端组的Role Mappings标签页，将`developer`角色分配到该组。

### 可能遇到的坑

1. **API限流问题**：批量导入时如果连续发送请求超过50个/秒，Keycloak可能返回429或503错误。解决方案：在脚本中加入`time.sleep(0.5)`，每10个批次暂停一下，或者用并发线程池控制速率。

2. **自定义属性值格式错误**：Admin REST API要求attributes的值必须是字符串数组`["研发部"]`，如果误写为字符串`"研发部"`，API返回400 Bad Request且错误信息不直观。排查技巧：先用Admin Console创建一个正确用户，再通过`kcadm.sh get users/{id}`查看标准JSON格式。

3. **Group名称不可重命名**：创建后Group的name字段无法通过API直接修改。计划好命名规范（如"公司总部/研发部/后端组"），如需改名只能删除重建并迁移用户。

4. **temporary密码标记理解偏差**：`temporary: true`的真实含义是"此密码需要用户在下一次登录时修改"，而不是"此密码有时效性"。如果你想让密码在24小时后过期，需要额外实现自定义SPI或定时任务。

5. **Required Actions与Client Scope冲突**：如果Client配置了"Bypass Required Actions"（如CLI工具），用户将跳过强制操作直接登录。适用于服务账号等非交互场景。

## 4 项目总结

### 优点与缺点

| 维度 | Keycloak用户管理 | 自建用户系统 | LDAP原生管理 |
|------|---------------|------------|------------|
| 开箱即用 | ✅ Admin Console + REST API，零代码 | ❌ 需完整开发前后端 | ⚠️ 需额外Web管理工具（如phpLDAPadmin） |
| 密码策略 | ✅ 可视化配置，支持哈希算法/历史/复杂度 | ❌ 手写正则+哈希逻辑 | ⚠️ 依赖ppolicy overlay，配置复杂 |

### 适用场景

1. **企业员工身份管理**：100-10000人规模，需要统一登录入口、自定义属性（工号/部门/职级）、入职/离职自动化。
2. **多级组织架构**：集团-子公司-部门-小组的树形组织，需要Group层次结构和属性继承。
3. **密码安全合规**：等保三级/ISO 27001要求强制密码复杂度、定期更换、历史密码不可复用。
4. **批量用户迁移**：从Excel/HR系统/旧LDAP批量导入用户到Keycloak，一次性或增量同步。
5. **无密码化演进**：通过WebAuthn/FIDO2逐步替代传统密码（进阶使用，本章仅涉及OTP）。

**不适用场景**：纯消费者级C端应用（千万级用户建议自研轻量方案）；需要复杂用户关系图谱的场景（如社交网络好友关系）。

### 注意事项

- **密码策略不宜过严**：要求8位+四种字符确实安全，但每周强制更换会导致用户在便利贴上记录密码。建议：8-12位+三种字符+90天更换，配合OTP即可达到实际安全水平。
- **Group深度有隐性上限**：过深的Group层级（>5层）不仅Admin Console展示混乱，还会影响用户权限计算性能——每次登录都需要递归解析所有父组的角色继承。建议层级不超过4层。
- **自定义属性不宜存大文本**：虽然支持任意字符串，但每个属性的值存储在数据库的`USER_ATTRIBUTE`表中，大量大文本属性会拖慢用户查询。需要存储大段JSON时考虑拆分到独立系统。
- **临时密码创建后用户未及时登录**：临时密码被创建2小时后用户才尝试登录，此时管理员可能已忘记密码。建议临时密码规范化为初始密码（如`Welcome@{工号后4位}`），并在用户文档中说明。

### 常见踩坑经验

1. **问题**：批量导入500个用户后Admin Console加载极慢。**根因**：未配置数据库索引，用户表全表扫描。**解决**：在PostgreSQL中为`USER_ENTITY`表的`EMAIL`字段创建索引，并为`USER_ATTRIBUTE`表的`NAME`+`VALUE`创建联合索引。

2. **问题**：用户设置OTP后手机丢失，无法登录。**根因**：Keycloak未配置备用恢复方案。**解决**：为管理员开启"删除用户OTP凭证"的能力（通过Admin Console→用户Credentials标签直接操作），并建立工单流程要求用户身份二次确认。

3. **问题**：批量导入脚本中，部分用户返回201成功但实际未在Users列表中显示。**根因**：同一批次中用户名与已存在用户冲突，409 Conflict被try...catch吞掉后继续循环。**解决**：按步骤4示例代码中区分处理201/409/其他状态码，并在最终打印汇总报告。
