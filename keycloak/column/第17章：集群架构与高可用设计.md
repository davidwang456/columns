# 第17章：集群架构与高可用设计

## 1 项目背景

某电商平台的Keycloak认证服务在"双11"零点遭遇了灾难性崩溃。零点钟声敲响的瞬间，10万+用户同时涌入——抢优惠券、刷新购物车、提交订单，登录QPS瞬间飙升至5000。单节点Keycloak的CPU从平稳的25%跳变到100%，内存耗尽（16GB JVM堆直接被吃满），数据库连接池（默认100连接）瞬时饱和。用户在浏览器端看到"504 Gateway Timeout"后，投诉电话打爆客服热线。运维紧急重启Keycloak服务，但启动过程需要加载所有Realm配置、初始化缓存、预热数据库连接池，耗时超过3分钟——这3分钟里，所有登录、注册、Token刷新功能全部瘫痪，直接经济损失超过200万元。

事后复盘时，团队发现了一个残酷的事实：这个架构只有一个Keycloak节点。单节点意味着单点故障——没有热备、没有故障转移、没有负载分担。当流量超出单节点处理能力时，纵向扩容（升级到32核CPU + 64GB内存）确实能应付一时，但存在物理天花板，且硬件成本呈指数级增长（一台高配服务器价格是普通服务器的5-8倍）。更致命的是，纵向扩容从不解决高可用问题——服务器电源故障、网卡损坏、机房交换机宕机，任何一种硬件故障都会让整个认证系统瘫痪。

除此之外，单节点架构还暴露了四个深层痛点：

**痛点一：无法水平扩展。** Keycloak默认依赖本地内存缓存——Session存储在单个JVM堆中，Token缓存、认证Session、登录失败计数全在进程内部。这意味着即使加了第二个Keycloak节点，两个节点的内存不共享，用户在节点A登录后，下一次请求被负载均衡器分到节点B时，Session丢失，用户被踢回登录页。运维被迫在Nginx上配置`ip_hash`（客户端IP哈希做会话保持），但这样做又引入了新问题：某个节点宕机后，对应IP段的所有用户全部掉线。

**痛点二：数据库连接池瓶颈。** 10万并发登录请求意味着至少10万个数据库连接同时建立和释放。PostgreSQL默认的`max_connections=100`很快被打满，后续请求排队等待连接释放，等待超过30秒后触发超时，用户看到504。虽然可以上调`max_connections`，但数据库的连接数并非无上限——每个连接占用2-4MB内存，1000个连接就是2-4GB，数据库很快会因内存不足而OOM。

**痛点三：无法实现滚动发布。** 单节点意味着任何一次版本升级、配置变更都需要重启服务，而重启必然导致服务中断。电商平台不可能接受"凌晨3点到5点停机升级"这种方案——大促期间每一分钟都在产生收入。

**痛点四：本地内存Session无法跨节点共享。** 这是集群化改造面临的核心技术挑战。Keycloak的认证流程涉及多步跳转（Authorization Code Flow），用户的认证状态通过`AUTH_SESSION_ID` Cookie在浏览器和服务器间传递，而实际的认证会话对象存储在服务器内存中。如果不解决Session共享，负载均衡器必须在TCP/IP层强制会话粘性（Sticky Session），这与"无状态、可水平扩展"的云原生理念背道而驰。

## 2 项目设计——剧本式交锋对话

**小胖**（一边嚼着薯片一边翻着监控面板）：大师！我有一个绝妙的比喻——这Keycloak集群不就是银行营业厅吗？你看，一个柜台排长队（单节点高负载），最好的办法就是多开几个窗口（加节点），每个窗口的柜员都能办同样的业务。那我直接把Keycloak的Docker容器多起几个，前端用Nginx一分流，不就完事儿了？为啥还要搞什么Infinispan缓存共享、JDBC_PING发现这些麻烦玩意儿？

**大师**（把小胖的薯片抽走）：你这个银行窗口的比喻方向是对的，但有一个关键遗漏——**银行柜员背后有一个共享金库**。假设3个柜员各自有自己的小保险箱，客户在A窗口存了钱，走到B窗口取钱时，柜员B在自己保险箱里找不到这笔钱——这就是单节点Keycloak用本地内存存Session的后果。Infinispan分布式缓存扮演的就是那个"共享金库"的角色：所有柜员（Keycloak节点）看到的金库内容是一致的，客户在哪存、在哪取都一样。

> **大师技术映射**：银行窗口（Keycloak节点）+ 共享金库（Infinispan缓存）→ 负载均衡分发流量 + 分布式缓存共享会话状态。`ip_hash`相当于"客户绑定特定柜员"——能解决问题但牺牲了弹性。

---

**小白**（皱着眉头合上笔记本）：大师，我查了一下Infinispan的集群发现机制，发现有好几种协议——DNS_PING、JDBC_PING、TCPPING、KUBE_PING……这几个名字看得我头晕。它们到底有什么区别？我们的部署环境是Docker Compose + 自建服务器，该选哪个？

**大师**：集群发现本质上是解决一个问题——**新节点加入集群时，如何找到集群中已有的其他节点**。四种发现协议分别对应四种部署场景：

- **JDBC_PING**：所有节点共享同一个数据库，利用数据库中的一张发现表（`JGROUPSPING`）来注册和发现节点。每个节点启动时向这张表写入自己的IP和端口，同时读取其他节点的注册信息。优点是零额外依赖（已有数据库），适合虚拟机/裸金属部署；缺点是数据库一旦挂掉，集群发现也失效，存在一定耦合。

- **DNS_PING / DNSPING**：通过DNS查询集群成员的A记录或SRV记录。在Kubernetes中，一个Headless Service的DNS会返回所有匹配Pod的IP——这正是K8s环境的标准做法。优点是天然适配容器编排平台，无需额外存储；缺点是在非K8s环境需要手动维护DNS记录。

- **TCPPING**：静态配置所有节点IP——在配置文件中写死`node1:7800, node2:7800, node3:7800`。最简单粗暴，但每增删一个节点都要修改所有节点的配置并重启，完全不优雅。

- **KUBE_PING**：专为Kubernetes设计，通过调用K8s API获取Pods列表来发现集群成员，比DNS_PING更主动（不需要等DNS缓存刷新）。

我们这个Docker Compose场景，三个Keycloak容器共享一个PostgreSQL，JDBC_PING是最简单且可行的选择——只需一个`KC_CACHE_STACK=jdbc-ping`环境变量即可启用。

> **大师技术映射**：集群发现 → 新员工入职时如何找到自己的同事。JDBC_PING → 公司花名册（数据库），大家翻同一本名册找联系方式。DNS_PING → 公司DNS域名，问"技术部有哪些人"就能拿到列表。TCPPING → 入职时HR给的一张纸，上面写着所有同事的分机号——改人就得重新打印。

---

**小胖**（举手打断）：不对不对，回到刚才银行窗口的例子——如果我用Nginx的`ip_hash`做Sticky Session，同一个用户的请求永远打到同一个节点，那不就不需要共享缓存了吗？

**小白**（立即跟进）：`ip_hash`确实能让Session固定在某节点，但小胖你想过三个致命问题吗？第一，某个节点宕机后，该节点上的所有用户Session全部丢失，而且ip_hash会把它的流量重新分配到其他节点，用户突然变成"未登录"状态。第二，如果客户端IP发生变化——比如用户从WiFi切换到4G、或者经过多层NAT代理——ip_hash的哈希值就变了，请求打到另一个节点，Session同样丢失。第三，Sticky Session本质上是将"无状态"的认证服务变成了"有状态"的服务，这与云原生的弹性扩缩容理念背道而驰。

**大师**（赞许地点头）：小白分析得透彻。Keycloak在集群模式下，浏览器会将`AUTH_SESSION_ID`写入Cookie——这是一个随机的Session ID，由Keycloak在用户开始认证时生成。即使关闭了Sticky Session，Keycloak也能通过这个ID从Infinispan分布式缓存中找回认证会话，无论请求被哪个节点处理。

**小白**：那Infinispan的分布式缓存会不会有脑裂问题？比如三个节点之间的网络断了，每个节点都以为自己还活着，各自独立处理请求，导致数据不一致？

**大师**：这正是分布式缓存的经典挑战。Infinispan通过JGroups管理集群拓扑，结合**分区处理策略（Partition Handling）**来应对脑裂：

- Inifinispan会持续监控集群成员的存活状态。当一个网络分区发生时，JGroups先检测到成员的离开，然后Infinispan评估"我所在的分区是否拥有大于总数一半的节点"。
- 如果分区中的节点数 > n/2（拥有主分区），该分区继续提供服务。
- 如果分区中的节点数 ≤ n/2（沦为次分区），该分区的所有缓存操作变为**只读模式**（DENY_READ_WRITES），直到网络恢复。

这样确保只有一个分区（主分区）能写入，避免了数据冲突。但代价是次分区上的节点虽然进程还活着，实际上已经无法处理新的登录请求——这引出了一个重要结论：**集群节点数应为奇数（3、5、7），最小生产部署为3节点**。

> **大师技术映射**：脑裂 → 公司三个合伙人意见不合分成两派，各执一词。Infinispan分区策略 → 公司章程规定"重大决策需要>50%的股份支持"——没拿到多数票的一方只能等下次会议的记录（只读模式）。

---

**小胖**（第二轮）：那负载均衡器怎么选？团队里有说Nginx的、有说HAProxy的，还有人说"上K8s还用啥LB，直接用Service不就行了"。

**大师**：负载均衡器选型取决于部署环境和性能要求，我给出三个维度的对比：

| 维度 | Nginx | HAProxy | K8s Service |
|------|-------|---------|-------------|
| 层级 | 七层（HTTP/HTTPS）为主，也支持四层 | 七层和四层都强 | 四层（TCP/UDP） |
| 会话保持 | ip_hash、sticky cookie（需plus版） | cookie、stick-table | ClientIP（Session Affinity） |
| 健康检查 | 被动（proxy_next_upstream） | 主动+被动，丰富检测方式 | 被动+Endpoint探活 |
| 配置复杂度 | 低，语法直观 | 中，配置项丰富 | 极低，声明式YAML |
| 性能 | 高（C语言编写） | 极高（事件驱动，零拷贝） | 依赖kube-proxy，iptables/IPVS转发 |
| 扩展性 | Lua/OpenResty脚本扩展 | 内置ACL和Lua | 依赖CNI插件 |

对于非K8s环境，**Nginx是我们的推荐选型**——配置简洁、社区资源丰富、与Keycloak的Cookie（`AUTH_SESSION_ID`）配合良好。关键配置在于健康检查端点——Keycloak在26.x版本中内置了三个健康端点：

- `/health/ready`：就绪检查，返回200表示该节点可以接收请求（包括数据库连接、缓存已初始化）。
- `/health/live`：存活检查，返回200表示进程没有挂，但不保证能处理请求。
- `/health/started`：启动完成检查，返回200表示节点已完成启动预热。

Nginx的`upstream`应配置`/health/ready`作为健康检查端点，确保只有完全就绪的节点才接收流量。健康检查间隔建议设为2-3秒——太频会增加Keycloak负载，太疏会导致宕机后10秒以上才发现。

> **大师技术映射**：负载均衡器 → 医院分诊台，根据每个科室（节点）的空闲情况和状态（健康检查）把病人（请求）分配到最合适的窗口。`/health/ready` → 挂号护士确认"医生已到岗、设备已开机"，而不是只确认"诊室门开着"。

## 3 项目实战

### 环境准备

- **Docker Desktop**（Windows/Mac）或Docker Engine（Linux）
- **Docker Compose** v2.x
- **Keycloak 26.1**（`quay.io/keycloak/keycloak:26.1`）
- **PostgreSQL 16**作为共享数据库

### 步骤1：设计3节点集群架构图

```
                       │   Nginx LB      │
                       │   (port 8080)   │
                       │   ip_hash/      │
                       │   sticky cookie │
                       └────────┬────────┘
                ┌───────────────┼───────────────┐
                ▼               ▼               ▼
          ┌──────────┐   ┌──────────┐   ┌──────────┐
          │Keycloak-1│   │Keycloak-2│   │Keycloak-3│
          │ :8080    │   │ :8080    │   │ :8080    │
          │  Node1   │◄──│  Node2   │──►│  Node3   │
          └─────┬────┘   └─────┬────┘   └─────┬────┘
                │               │               │
                └───────────────┼───────────────┘
                                │
                                ▼
                       ┌─────────────────┐
                       │  PostgreSQL 16  │
                       │  (共享数据库)    │
                       │  keycloak db    │
                       └─────────────────┘
```

**架构说明**：
- 三个Keycloak节点通过**JDBC_PING**发现彼此，将集群拓扑写入数据库的`JGROUPSPING`表。
- Infinispan分布式缓存在三个节点间同步`sessions`、`authenticationSessions`、`clientSessions`、`loginFailures`、`work`等缓存域，确保任意节点都能处理任意用户的请求。
- Nginx作为七层反向代理，对`/health/ready`做被动健康检查，自动剔除故障节点。
- PostgreSQL作为共享持久层——所有节点的Realm配置、用户数据、客户端注册信息最终落盘于此。

### 步骤3：Nginx负载均衡配置

创建 `nginx.conf`，配置反向代理与健康检查：

```nginx
# nginx.conf
worker_processes auto;

events {
    worker_connections 1024;
}

http {
    upstream keycloak_cluster {
        ip_hash;  # 客户端IP哈希——简单的会话保持策略
        server keycloak-1:8080 weight=1 max_fails=2 fail_timeout=10s;
        server keycloak-2:8080 weight=1 max_fails=2 fail_timeout=10s;
        server keycloak-3:8080 weight=1 max_fails=2 fail_timeout=10s;
    }

    server {
        listen 80;
        server_name localhost;

        # 健康检查端点
        location /health/ready {
            proxy_pass http://keycloak_cluster;
            proxy_http_version 1.1;
            proxy_set_header Connection "";
            access_log off;
        }

        # 主代理
        location / {
            proxy_pass http://keycloak_cluster;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_set_header X-Forwarded-Host $host;
            proxy_set_header X-Forwarded-Port $server_port;

            # 长连接优化
            proxy_http_version 1.1;
            proxy_set_header Connection "";

            # 超时设置
            proxy_connect_timeout 10s;
            proxy_read_timeout 60s;
            proxy_send_timeout 60s;

            # 缓冲设置
            proxy_buffer_size 16k;
            proxy_buffers 4 32k;
            proxy_busy_buffers_size 64k;

            # 关键：透传Keycloak的AUTH_SESSION_ID Cookie
            proxy_cookie_path / "/; HTTPOnly; Secure";
        }
    }
}
```

**配置要点解析**：

- `ip_hash`作为会话保持策略，确保同一客户端的请求优先落到同一节点。即使集群已启用了Infinispan会话共享，`ip_hash`仍可减少跨节点Session查找带来的网络开销。
- `max_fails=2 fail_timeout=10s`：被动健康检查——连续2次请求失败后，将该节点标记为不可用，10秒后再重新尝试。
- `X-Forwarded-*`请求头透传：Keycloak需要这些头来正确生成重定向URL。缺少`X-Forwarded-Proto`和`X-Forwarded-Port`会导致登录回调URL变成`http`而非`https`，或端口号丢失。

### 步骤4：部署与验证

**4.1 启动集群**

```bash
# 启动所有服务（后台运行）
docker compose -f docker-compose-cluster.yml up -d

docker logs keycloak-1 2>&1 | Select-String "CLUSTER"
```

预期输出（日志中出现的Infinispan集群视图）：

### 可能遇到的坑

**坑1：容器时间不同步导致Token验证失败。** Keycloak的JWT包含`iat`（签发时间）和`exp`（过期时间）claim，如果集群节点间的系统时间偏差超过30秒，已经签发的Token可能被另一个节点判定为"未生效"或"已过期"。解决方案：在Docker Compose中统一TZ环境变量，或在宿主机上配置NTP时间同步服务。

**坑2：JDBC_PING在数据库连接断开时的故障表现。** JDBC_PING依赖数据库的心跳检测来维护集群视图。当PostgreSQL重启或网络抖动导致连接断开时，Keycloak节点可能误判其他节点已离开集群，触发脑裂。表现为短时间内两个分区各自独立处理请求。解决方案：适当增大`jgroups.jdbc.ping.remove_old_members_on_startup`的检测间隔，配合PostgreSQL的连接池重试策略。

**坑3：Docker网络模式下的容器间通信。** 默认的bridge网络下，容器间通过服务名（如`keycloak-1`）通信时，JDBC_PING写入数据库的是容器IP（如`172.17.0.5`），而其他节点尝试用这个IP直接连接时可能因不在同一bridge子网而失败。使用Compose默认网络（同一个network内所有容器可互访）通常能避免此问题。

**坑4：ip_hash在NAT/代理场景下的失效。** 如果用户通过公司出口代理访问（所有员工共享同一个出口IP），`ip_hash`会将所有用户分配到同一个节点，导致负载严重不均。生产环境建议配合`sticky cookie`策略或完全移除`ip_hash`，依赖Infinispan的Session共享能力。

**坑5：首次启动时`JGROUPSPING`表不存在。** Keycloak 26.x的JDBC_PING模块会在首次启动时自动创建`JGROUPSPING`表，但如果数据库用户缺少DDL权限，集群发现会失败。确保数据库用户拥有CREATE TABLE权限，或者预先生成DDL语句手动建表。

## 4 项目总结

### 优点与缺点

| 维度 | 单节点 | 三节点集群 | 多数据中心 |
|------|--------|-----------|-----------|
| 可用性 | 无冗余，宕机即不可用 | 任意单节点故障不影响服务 | 单数据中心整体故障仍可用 |
| 性能 | 单机CPU/内存上限 | 水平扩展，线性提升QPS | 就近接入，全球低延迟 |
| 运维复杂度 | 极低 | 中等（需管理LB、缓存发现） | 高（跨数据中心复制、DNS智能解析） |
| 部署成本 | 1台服务器 | 4-5台（3节点+LB+DB） | 6台以上（每中心至少2节点） |
| 数据一致性 | 不适用 | 分布式缓存，Partition处理 | 最终一致性（异地延迟） |
| Session共享 | 本地内存 | Infinispan分布式缓存 | Infinispan跨站点复制（x-site） |

**适用场景**：生产环境高可用（99.9% SLA）、大流量电商/社交平台、金融级别可靠性要求（需审计日志不丢失）、需要滚动发布的持续部署流程。

**不适用场景**：个人开发/测试环境（单节点docker-compose足够）、企业内部小流量应用（<100 QPS）、对延迟极度敏感且可接受Sticky Session的场景。

### 注意事项

1. **时间同步是硬性要求。** 所有Keycloak节点之间、Keycloak与数据库之间、Keycloak与前端应用之间的时间偏差必须控制在5秒以内。建议所有服务器配置NTP或Chrony客户端，在启动脚本中增加`ntpdate -u ntp.aliyun.com`作为前置检查。

2. **数据库连接池总上限需重新评估。** 单节点Keycloak默认连接池为100个连接，3节点就是300个——确认PostgreSQL的`max_connections`已上调到至少400（预留100给其他连接）。同时注意数据库服务器的内存容量（每个连接约2-4MB）。

3. **负载均衡健康检查间隔不宜过大。** 推荐2-3秒的检查间隔——既保证故障节点能在5秒内被剔除，又不给Keycloak增加过多健康检查请求。Nginx的`fail_timeout`建议设为健康检查间隔的3-5倍。

4. **Infinispan缓存域的角色理解**：`sessions`（用户登录Session）、`authenticationSessions`（登录流程中的临时状态）、`clientSessions`（客户端级别的会话）、`loginFailures`（暴力破解计数器）、`work`（集群间的任务分发）。在集群日志中留意这些缓存域的操作耗时，可以帮助定位性能问题。

5. **灰度发布时注意版本兼容。** Keycloak不同大版本间的集群通信协议可能不兼容——不要试图混合v25和v26节点组成集群。滚动升级时，先停一个节点→升级→启动并加入集群→验证→继续下一个节点，始终保持集群中至少2个节点在运行。

### 常见踩坑经验

**案例一：JDBC_PING表的并发写入冲突。** 某生产环境3节点频繁重启，每次重启都在`JGROUPSPING`表中留下`ping_data`记录。当表积累到1000+条记录后，新节点加入时因序列化旧记录的延迟而超时，集群永远无法形成完整视图。根因分析：缺少定期清理过期记录的机制。解决：在Keycloak启动参数中配置`jgroups.jdbc.ping.remove_all_data_on_view_change=true`，每次集群视图变更时清理旧记录。

**案例二：Nginx `ip_hash`在企业代理环境下全部失效。** 某公司所有员工通过同一个Squid代理服务器访问外网，`ip_hash`将所有请求分配到同一个Keycloak节点，其他两个节点几乎零负载。根因分析：`ip_hash`基于`$remote_addr`哈希，而代理环境下`$remote_addr`始终是代理服务器的IP。解决：改为`hash $http_x_forwarded_for`对真实客户端IP做哈希，或直接移除会话粘性策略。

**案例三：滚动升级到一半发现版本不兼容。** 三节点集群从Keycloak 25升级到26，先停了node3升级后启动，node3无法加入由node1/node2（仍运行v25）组成的集群，因为v25和v26的JGroups消息格式不兼容。根因分析：大版本升级应按照"两阶段升级"操作——先升级所有节点到v25的最新补丁版本（确保数据格式兼容），再将所有节点一起滚动到v26。

### 思考题

1. **在Kubernetes环境下，为什么推荐使用DNS_PING而不是JDBC_PING？** 提示：从K8s Pod生命周期（IP不固定、随时重建）、Headless Service的DNS自动更新、以及数据库耦合性三个角度分析。

2. **如果3个节点中的2个同时宕机（例如交换机故障导致整个机架断电），剩下的1个节点能继续提供服务吗？为什么？** 提示：回顾Infinispan的分区处理策略——当节点数 ≤ 集群总数的一半时，该分区的缓存进入什么模式？
