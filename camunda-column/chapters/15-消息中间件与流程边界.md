## 1. 项目背景

企业里常有 **Kafka/RabbitMQ** 作为系统间事件总线；流程引擎则管理**有状态、可等待**的实例。两者混用若边界不清，会出现「每条消息都起一个流程」或「关联键对不上实例」。本章要解决的**一条主线问题**是：给出**事件驱动 vs 命令式**启动的选型框架，以及 **consumer 内 correlate 消息** 时的幂等与重放策略。

---

## 2. 项目设计（三角色对话）

*Confluent 控制台与 Camunda Cockpit 并排，大家在吵「到底谁主谁宾」。*
**小胖**：支付成功 topic 一响，我 `startProcessInstance`，**一行消费者**结束战斗？  
**小白**：支付回调**重复投递**三次，你也起三个流程实例？**幂等键**、去重表、`businessKey` 查重——先想清楚再写 `@KafkaListener`。  
**大师**：分清 **Command**（起流程）和 **Event**（唤醒等待中的消息捕获）：后者走 `correlateMessage`，体里带 `orderId` 绑实例。**死信**里烂掉的消息谁捞、业务怎么补偿，**BPMN 第二十单**讲；Kafka 不管替你擦屁股。  
**小白**：多分区乱序怎么破？  
**大师**：跨分区本就不保全局顺序；要强顺序就 **key 分区**或牺牲吞吐单分区——别又想吃并行又想吃线序。命名也暴露心智：Command topic 和 Event topic **分开**，消费者别写成**上帝类**一锅炖。

## 3. 项目实战

### 3.1 环境前提

- 本地或容器 Kafka；Spring Kafka 消费者。

### 3.2 步骤说明

1. 定义 topic `payment.completed` 消息体 schema。
2. 消费者解析后调用 `runtimeService.createMessageCorrelation(...).correlate()`。
3. 用重复消息验证**不会**重复推进（依赖幂等设计：如已完成则 ack 并跳过）。
4. 记录监控：消费 lag、 correlate 失败率、DLQ 重放流程（运维手册）。
5. 文档化「禁止 consumer 直接写引擎库表」。

### 3.3 源码与说明

**为什么禁止绕开 API 写库**：破坏引擎不变量，升级必炸。

**幂等示例思路**：`businessKey` 查实例；或业务表记录「流程已起」状态位。

### 3.4 验证

- 重放同一条消息不造成重复业务副作用；Cockpit 轨迹正确。

### 3.5 监控面板建议

- Consumer **lag**、**重试次数**、**DLQ 深度**  
- `correlate` **失败率**（消息名/租户/版本错误）  
- 流程实例 **堆积**（特定定义 key）

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 解耦、削峰、与事件驱动架构一致。 |
| 缺点 / 代价 | 幂等、顺序、观测复杂。 |
| 适用场景 | 异步回调、跨系统事件触发。 |
| 不适用场景 | 强同步人机混合可优先 REST。 |
| 注意事项 | 关联键、DLQ、监控、schema 演进。 |
| 常见踩坑 | 重复消费；消息体无版本；消费者拖慢导致 lag 爆炸。 |

**延伸阅读**：第 8、20 章；第 25 章架构边界。

## 5. 附录：与第 14 章 External Task 的关系

**Kafka** 管「事件投递语义」；**External Task** 管「引擎侧待办工作项」。可以并存：事件进 Kafka，消费者里既更新业务库又 `complete` external task——关键是**幂等键**与**状态机**不要分裂在两处各写一套。

## 7. 章末提要（面向推广口播）

消息集成的关键词是 **幂等与关联**：Kafka 保证「投过至少一次」并不代表「流程只起一次」。给架构评审的材料里，应同时包括 **消息 schema 版本**、**幂等键**、**DLQ 回放审批** 三页纸。**没有这三样，别谈事件驱动上线。**

## 8. 深度追问（写给半年后的自己）

1. **乱序消息**到达时，业务是否定义「以谁为准」？  
2. 消费者 **batch** 与流程 **单条**语义是否一致？  
3. 若消息体增大，**分区**策略是否要调？  
4. **死信**里是否含足够上下文排障？  
5. 跨团队 **topic 所有权**与 **SLA**是否写入合同？  
6. **重放**演练是否每半年做一次？  

**补白**：消息与流程的边界是组织边界；扯皮多半来自责权不清。
