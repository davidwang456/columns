## 1. 项目背景

想象你在深夜点外卖：下单、支付、商家接单、骑手取餐、送达、评价。表面上是一个 App 按钮，背后却是一条**多人、多系统、多分支**的协作链。只要其中一环变慢、失败或需要人工介入，整条链路的状态就会变得复杂：钱扣了但订单没创建怎么办？骑手拒单谁重派？用户取消时厨房已经出餐又怎么算？

许多企业信息化系统里，类似的链路不叫外卖，而叫「采购审批」「客户开户」「理赔」「上线发布」。传统做法是把规则写死在代码里：`if (金额 > x) 找经理`，再叠一层又一层的 `switch` 与定时任务。需求一改，就要改代码、联调、回归测试，跨团队沟通成本极高。本章要解决的**一条主线问题**是：**当业务协作规则频繁变化、参与角色众多、还需要可观测与可追溯时，为什么业界会引入「流程引擎」，以及 Camunda 在这类问题里扮演什么角色。**

## 2. 项目设计（三角色对话）

*深夜便利店门口，三人拿「外卖订单卡餐」当假想事故在吵。默认规则：**谁主张「加段 if 就行」，谁就要说出状态散在哪几张表**。*
**小胖**：（晃手机）我不就支付调完调物流吗？上 Camunda 是不是**过度设计**？我们一共三个人诶。  
**小白**：等等。你告诉我：骑手**拒单重派**的 `if` 写在 Payment 服务还是 Order 服务？三个月后加「到店自取」，你准备打几个 hotfix？  
**小胖**：……那再包一层嘛。  
**小白**：包完你发现「用户取消」要同时撩支付、厨房、骑手、券——半夜三点你敢改哪个仓库？  
**大师**：别吵定义先吵现象：**引擎先解决「协作+演化」，不是 TPS。** 三个人、规则永远不变，可以硬编码；一旦**多角色、常改、要留痕**，就该把**控制流**抬到 **BPMN**，让撕逼发生在**图**上，而不是发生在某人的私有分支上。  
**小胖**：画个图不就是 **if-else 挪到图上**换皮肤？  
**大师**：烂团队才是。**网关只管谁先谁后**；**DMN 或外部服务**才扛复杂规则。图丑是**过程病**，不是 BPMN 必然失败。  
**小白**：那跟 Spring StateMachine、Temporal、**Zeebe** 咋切？  
**大师**：内部小状态机够用就别上大炮；要 **BPMN+人+DMN+现成 Cockpit**，嵌入式 **Camunda 7** 在企业里最常见。**Camunda 8** 换的是运维模型，专栏**第 34 章**单独对齐，别在第二章就跟 7 混着讲。  
**小胖**：超时、补偿、审计谁管？总不能监听器里写满 `try` 吧？  
**大师**：引擎给你**跑道**：作业、历史、事件语义在上面跑；**重试怎么退避、补偿是不是幂等**还是你领域的事——想「监听器万能」的，晚上值班见。  
*白板写上：隐式流程 → 先画到纸上，再决定动不动引擎。*  
**小白**：最后一嘴：**DB 压力、版本迁移、历史膨胀**都在账上，谁跟财务说？  
**大师**：所以叫「流程即代码」——**评审、测试、CI**得跟上；换的是沟通成本，不是魔法。

## 3. 项目实战

本章以**概念实战**为主：用「外卖」故事走一遍最小要素，并给出你可在第 2 章落地的对照清单。不要求本地已安装 Camunda。

### 3.1 环境前提

- 无强制软件版本；阅读即可。
- 若你愿意手绘或打开任意 BPMN 建模工具，可跟着画草图。

### 3.2 步骤说明：把外卖拆成「可被引擎理解」的片段

1. **列出参与者与系统**：用户、商户系统、支付、配送、客服。引擎里对应「泳道」思维（不一定第一章就画泳道图，但心里要有角色）。
2. **标出自动化与人机节点**：支付回调通常是**服务任务**；用户取消是**用户任务**或消息相关的事件；超时提醒往往是**定时器边界事件**。
3. **标出分支条件**：例如「是否已出餐」决定取消策略——这是**排他网关**要表达的语义，而不是散落在三个接口里各写一遍。
4. **标出需要留痕的审计点**：在哪些节点要记录业务快照——为后续在 Cockpit 里追踪实例埋线索（具体实现见后续章节）。

### 3.3 源码与说明

本章不绑定具体仓库代码，给出一个**对照表**，便于你把「生活语言」映射到 Camunda 术语；第 2 章起会对应到真实配置与 API。

| 生活概念 | BPMN/Camunda 概念（入门理解） |
|----------|-------------------------------|
| 用户点按钮、填表 | 用户任务（User Task） |
| 系统自动扣款、通知配送 | 服务任务（Service Task）或外部任务（External Task） |
| 等支付结果、等回调 | 消息事件、接收任务（视建模风格而定） |
| 超时未支付自动关单 | 定时器边界事件 |
| 配送失败改派 | 错误边界、补偿或重试策略（中级展开） |

**为什么先给映射表**：避免初学者一上来就钻 XML，却在脑子里没有「节点类型」的锚点。

### 3.4 验证

本章的「验证」是自查：

- 你能用自己的话解释：**流程引擎主要解决的是协作与演化，而不是替代你的业务算法吗？**
- 你能列出本组织若上 Camunda，**至少三条业务侧收益**和**两条工程侧代价**吗？

若两条都能写清楚，即可进入第 2 章动手跑通最小工程。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 把协作结构从隐式代码变为显式模型；利于跨角色沟通与版本管理；Camunda 7 嵌入式与 Java 生态结合紧，上手曲线相对可控。 |
| 缺点 / 代价 | 引入存储与运维复杂度；建模质量决定成败；错误使用会导致「图很炫、系统更难懂」。 |
| 适用场景 | 多角色审批与协作；规则常变；需要实例级追踪与审计；Java/Spring 技术栈为主。 |
| 不适用场景 | 极简单、极少变更的后台任务；超高吞吐纯事件流且无人机协作（需另评 Camunda 8 或其他技术）。 |
| 注意事项 | 7 与 8 运维模型差异大，勿混讲；从第一天就规划历史数据与清理策略（高级章节展开）。 |
| 常见踩坑 | 把引擎当万能队列；网关上堆满业务规则；忽略流程版本升级对运行实例的影响。 |
