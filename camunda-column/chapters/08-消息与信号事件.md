## 1. 项目背景

有些步骤不能一直轮询数据库「好了没」，而是**外部系统回调成功**后再往下走；有时需要**唤醒**某个等待中的流程实例。BPMN 用**消息事件**表达点对点关联，用**信号**表达广播。本章要解决的**一条主线问题**是：理解 **correlation（关联键）** 在消息投递中的作用，并区分消息与信号的适用边界，避免「一条消息唤醒错实例」的生产事故。

---

## 2. 项目设计（三角色对话）

*白板左边写着 Kafka topic，右边画着 BPMN 消息捕获——像两条铁轨没接上。*
**小胖**：我们**Kafka 都有了**，还搞 BPMN 消息事件？重复建设吧？  
**小白**：Kafka 只管**字节运到**；BPMN 消息管**哪条流程实例在哪个耳朵上等着**——语义层两码事。典型打法：消费者收到消息后调 `correlateMessage`，把「消息到了」翻译成**令牌前进**。  
**小胖**：那信号呢？是不是群@全员？  
**大师**：信号偏**广播**，能一次唤醒一批订阅者；用不好就是**误伤全场**。业务键不熟别乱玩广播。  
**小白**：防串单靠啥？消息名和订单号怎么绑？边界消息和中间捕获又在哪停？  
**大师**：`businessKey` / 关联变量把**业务主键**钉死；边界挂在活动上——活动还活着时插进来；中间捕获在**主路径**上站岗。别把消息名当口号，**契约里写清**：消息名、`payload` 字段、流程 `key`，升级先双读再切——否则深夜工单教你做人。

## 3. 项目实战

### 3.1 环境前提

- 流程含**消息中间捕获事件**或**接收任务 + 消息**模式（按建模习惯二选一）。
- 会调用 Runtime API 发消息。

### 3.2 步骤说明

1. 建模：流程在「等待支付回调」处等待消息 `PaymentReceived`。
2. 启动实例，设置 `businessKey` 为订单号。
3. 使用 REST `message` 端点或 Java API，携带 `correlationKeys`（如订单号）触发关联。
4. 验证实例前进到下一节点。
5. 再启动第二单实例，故意用错关联键，确认**不会错误唤醒**。

### 3.3 源码与说明

Java 关联示意：

```java
runtimeService.createMessageCorrelation("PaymentReceived")
  .processInstanceBusinessKey(orderId)
  .setVariable("paid", true)
  .correlate();
```

**为什么用 businessKey**：简单场景下易与业务订单对齐；复杂场景可用 `correlateMessage` 的其他重载。

**风险**：关联键重复、消息名拼写错误、多版本流程定义并存时需指定 `tenantId` 或 `processDefinitionId`（进阶）。

### 3.4 验证

- Cockpit 显示实例从未等待态推进；日志无关联失败。
- 单测模拟消息投递（第 18 章）。

### 3.5 与信号对比的小表

| 机制 | 语义 | 典型风险 |
|------|------|----------|
| Message | 点对点关联 | 关联键错 |
| Signal | 广播 | 误触发多实例 |

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 与外部系统异步衔接自然；语义在图上可见。 |
| 缺点 / 代价 | 关联设计错误难排查；多实例环境要谨慎。 |
| 适用场景 | 支付回调、审批结果回传、跨系统触发。 |
| 不适用场景 | 简单同步 HTTP 可直接用服务任务。 |
| 注意事项 | 幂等、重放、消息风暴。 |
| 常见踩坑 | 关联键不唯一；版本升级后消息名不一致。 |

## 5. 附录：REST 发消息 vs Java API

二者本质相同；REST 便于跨语言集成测试。生产需 **认证** 与 **审计**（谁 correlate 了哪条业务键）。

## 7. 章末提要（面向推广口播）

消息章节是集成事故高发区：**名字错、键错、版本错** 三错必占其一。对外宣讲时强调「**一条业务消息 = 一组关联键 + 一个消息名 + 可选 payload 版本**」。信号广播更要谨慎：**能唤醒多少实例**先在心里算清。**把 correlate 失败率做成告警，比在群里艾特全体成员有用。**
