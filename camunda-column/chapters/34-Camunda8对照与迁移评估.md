## 1. 项目背景

团队听说 **Camunda 8**「云原生、高吞吐」，想评估是否从 **Platform 7 嵌入式** 迁移。两栈 **运维模型、客户端、BPMN 支持程度** 均有差异。本章要解决的**一条主线问题**是：用**决策表**从业务、团队、运维、合规维度评估迁移，并明确 **本专栏为何以 7 为主线**（避免全文概念混用）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「8 数字更大，是不是更好？」

不一定。8 的 **Zeebe** 架构适合特定工作负载与运维能力；7 嵌入式在 **Java 单体、现有 Camunda 技能栈** 上仍大量存在。

### 2.2 小白追问

小白问：「第一，**BPMN 特性**差异？第二，**Tasklist/Operate** 与 7 控制台差异？第三，**迁移成本**主要在哪？**（以官方当前文档为准）**」

### 2.3 大师定调

大师归纳：**迁移**是产品级项目，不是升版本号。需 PoC、压测、双跑、培训与回滚方案。若仅为「品牌升级」而动机不足，建议暂缓。

### 2.4 组织问题往往比技术大

小胖问：「技术上能迁，为啥还拖？」大师答：**运维模型**、**预算**、**供应商合同**、**团队技能**任一卡住都会搁浅。评估报告里应单列「组织准备度」。

---

## 3. 项目实战

### 3.1 环境前提

- 阅读 Camunda 8 官方架构与兼容性说明；不强制搭建在线集群。

### 3.2 步骤说明

1. **列需求**：吞吐、RPO/RTO、多租户、人机协作比例、外部任务占比。
2. **对照表**（团队自行打分 1～5）：

| 维度 | Camunda 7 嵌入式 | Camunda 8（概念） |
|------|------------------|-------------------|
| 运维模型 | 自建 DB + 应用 | 集群 broker 等（以官方为准） |
| 客户端 | Java/REST 成熟 | Zeebe gRPC 客户端 |
| 团队技能 | 现有 Java | 需学习新运维面 |
| 迁移成本 | 低（保持） | 高（重评模型与集成） |

3. 选 **2～3 条**代表性流程做 PoC 迁移评估（时间盒）。
4. 与法务/安全确认数据驻留与合规。
5. 输出决策：**维持 7 / 双栈 / 迁移**，附理由与时间表。

### 3.3 源码与说明

**为什么本章不写具体代码**：8 的 API 与部署与 7 不同步；代码示例易快速过时，**以官方示例为准**。

### 3.4 验证

- 决策文档经架构评审；风险与成本量化。

### 3.5 PoC 成功标准（示例）

- **功能**：选定流程端到端跑通；人机任务与外部任务覆盖。  
- **性能**：在目标 TPS 下 P95 达标；无异常作业积压。  
- **运维**：备份恢复演练通过；告警规则可落地。  
- **团队**：≥2 名核心开发完成培训考核。

## 7. 章末提要（面向推广口播）

迁移评估最怕「**为了新而新**」。本章给出的表格意在迫使团队写出**可量化的否**：若运维能力、预算、时间表任一不及格，就应坦诚写「**暂缓**」。双跑与回滚不是口号，要进项目计划与预算。**记住：7 与 8 不是升级关系，是产品族分叉。**

## 8. 深度追问（写给半年后的自己）

1. 业务方是否能接受 **运维模型**的根本变化？  
2. 是否有 **双跑预算**（两套监控、两套 on-call）？  
3. 若迁移暂停，**技能投资**如何转为通用能力（BPMN/DMN）？  
4. 合同与 **license**条款是否重新审视？  
5. 是否有「**永不迁移**」场景（例如强嵌入式）被正式记录？  
6. 决策若「暂缓」，**复盘周期**设为多久再评估？  

**补白**：架构结论要可证伪；最怕的是没有结论的长期悬置消耗士气。
