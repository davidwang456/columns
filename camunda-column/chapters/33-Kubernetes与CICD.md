## 1. 项目背景

同样一个 Spring Boot + Camunda 镜像，在笔记本与 K8s 上行为可能不同：**配置外置**、**探针**、**滚动发布**、**文件系统只读** 都会影响启动与作业调度。本章要解决的**一条主线问题**是：给出 **12-factor** 对齐清单（配置、日志 stdout、无本地状态），并定义 **liveness/readiness** 探针避免「未就绪就接流量」与**误杀**长作业进程。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我把 jar 扔镜像里跑，和 VM 有啥区别？」

区别在**重启策略**与**网络**：Pod 随时重建，本地磁盘不可靠；配置应来自 **ConfigMap/Secret** 或外部配置中心。

### 2.2 小白追问

小白问：「第一，**readiness** 应检查什么？第二，**HPA** 与作业线程池关系？第三，**CI** 里部署 BPMN 的流水线步骤？」

### 2.3 大师定调

大师归纳：

- **readiness**：依赖 DB 可达、迁移完成、引擎启动完毕；**liveness**：死锁才重启，勿把慢查询当死。
- **HPA**：基于 CPU/自定义指标；External Task Worker 可横向扩。
- **CI**：构建镜像 → 镜像扫描 → 部署到环境 → **冒烟**（启动实例、完成一任务）。

### 2.4 配置外置：12-factor

小胖问：「镜像里能写死数据库密码吗？」大师答：**禁止**。Secret 走 K8s Secret/外部密钥管理；配置通过环境变量或挂载；镜像**可复现**于 dev/stage/prod。

## 3. 项目实战

### 3.1 环境前提

- 可访问 K8s 集群；Helm 或原生 YAML。

### 3.2 步骤说明

1. 将 `application.yaml` 敏感项迁到 Secret；非敏感到 ConfigMap。
2. 配置 `readinessProbe` HTTP 指向 Actuator health（若启用）。
3. 滚动发布：观察旧 Pod 终止前作业是否 drain（按架构）。
4. 在流水线加 **API 冒烟**：`POST` 启动流程并 `complete`。
5. 记录资源限制：CPU/内存与 GC 日志。

### 3.3 源码与说明

**为什么 Actuator**：统一健康检查端点；勿自定义随意端口。

**注意**：Camunda 迁移脚本在启动时执行——readiness 需等迁移完成（版本依赖）。

### 3.4 验证

- 新 Pod 就绪前无流量；发布零失败或可控回滚（第 19 章）。

### 3.5 探针建议（概念）

- **readiness**：`GET /actuator/health/readiness`（若启用）或等价端点；包含 DB 检查。  
- **liveness**：仅进程存活；避免与 DB 强耦合导致全量重启。  
- **startup**（若启用）：长启动应用避免被 kubelet 过早杀。
