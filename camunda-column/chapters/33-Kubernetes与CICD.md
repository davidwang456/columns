## 1. 项目背景

同样一个 Spring Boot + Camunda 镜像，在笔记本与 K8s 上行为可能不同：**配置外置**、**探针**、**滚动发布**、**文件系统只读** 都会影响启动与作业调度。本章要解决的**一条主线问题**是：给出 **12-factor** 对齐清单（配置、日志 stdout、无本地状态），并定义 **liveness/readiness** 探针避免「未就绪就接流量」与**误杀**长作业进程。

---

## 2. 项目设计（三角色对话）

*Staging 集群滚动到一半，旧 Pod 还在写库。*
**小胖**：Fat JAR 塞镜像，跟以前 VM 跑，**差别**在哪？  
**小白**：Pod **说没就没**，本地盘别当硬盘；网络名会变，**服务发现**重写。配置还写死在镜像里？12-factor 先重修。  
**大师**：**readiness** 等 **DB、迁移、引擎起完**；**liveness** 别把慢查询当尸体——误杀比慢更惨（Actuator Health 对齐版本）。**HPA** 弹副本，**External Worker** 也能弹——线程池和连接池别跟副本数**线性幻想**。CI：**build → 镜像扫描 → deploy → API 冒烟**（起流程 + complete），别只靠「Pod 绿了」。  
**小胖**：镜像里 `spring.datasource.password=123456`？  
**大师**：**禁止**。Secret / 外置密钥；同一镜像跑 dev/stage/prod，换环境不换 Dockerfile。

## 3. 项目实战

### 3.1 环境前提

- 可访问 K8s 集群；Helm 或原生 YAML。

### 3.2 步骤说明

1. 将 `application.yaml` 敏感项迁到 Secret；非敏感到 ConfigMap。
2. 配置 `readinessProbe` HTTP 指向 Actuator health（若启用）。
3. 滚动发布：观察旧 Pod 终止前作业是否 drain（按架构）。
4. 在流水线加 **API 冒烟**：`POST` 启动流程并 `complete`。
5. 记录资源限制：CPU/内存与 GC 日志。

### 3.3 源码与说明

**为什么 Actuator**：统一健康检查端点；勿自定义随意端口。

**注意**：Camunda 迁移脚本在启动时执行——readiness 需等迁移完成（版本依赖）。

### 3.4 验证

- 新 Pod 就绪前无流量；发布零失败或可控回滚（第 19 章）。

### 3.5 探针建议（概念）

- **readiness**：`GET /actuator/health/readiness`（若启用）或等价端点；包含 DB 检查。  
- **liveness**：仅进程存活；避免与 DB 强耦合导致全量重启。  
- **startup**（若启用）：长启动应用避免被 kubelet 过早杀。

## 7. 章末提要（面向推广口播）

容器化后最大的坑是 **配置与密钥外置** 与 **探针语义**。给团队一句话：**readiness 不等于 liveness**；给流水线一句话：**镜像同源、配置分环境、密钥不入库**。Camunda 带 DB 迁移的应用，务必让 readiness **等待迁移完成**，否则流量打到半初始化实例。**K8s 不会替你理解 BPMN，只会替你重启 Pod。**

## 8. 深度追问（写给半年后的自己）

1. **Pod 资源 limit**是否压测验证过 GC 与线程开销？  
2. 多环境 **同名 Secret** 轮换策略是否一致？  
3. 是否在流水线做 **镜像漏洞**阻断发布？  
4. **HPA** 依据自定义指标时，是否验证过冷启动？  
5. **网络策略**是否限制仅ingress到引擎REST？  
6. **回滚**演练是否包含「DB 已向前迁移」场景？  

**补白**：平台工程与业务工程要合唱；单方背锅的发布夜往往源于缺口清单。
