## 1. 项目背景

默认 Java 序列化或 Spin JSON 无法满足**版本演进**或**跨语言**需求时，需要 **SPI 自定义序列化** 或统一 **DTO 映射**。本章要解决的**一条主线问题**是：在**不破坏历史实例反序列化**的前提下演进变量结构，并给出**兼容矩阵**（字段新增、重命名、类型变更）。

### 1.1 场景延伸：「订单快照」要不要进变量？

常见需求是希望在流程图上一眼看到当时的商品行项目与金额，于是把整个 `OrderDTO` 嵌进变量。短期 PoC 最省事，但一旦价格、税率、优惠规则随活动变化，**历史实例里的快照**与**业务库真实订单**可能不一致：客服截图用的是流程里旧值，财务报表用的是库里新值。更稳妥的策略是：变量里只存 **`orderId` + `snapshotVersion` + 关键展示字段**；完整明细需要时，用 `snapshotVersion` 去查**不可变**的历史快照表（或事件溯源投影）。若必须存富对象，就要本章所述的 **schema 版本**、迁移与监控，而不是依赖「反正能序列化」的侥幸。

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我把变量全改成 JSON 字符串不就完了？」

可行，但要 schema 治理；否则「字符串里是什么」无人知晓。

### 2.2 小白追问

小白问：「第一，SPI 注册与类加载器问题？第二，重命名字段老实例怎么办？第三，与 REST 类型声明一致性？」

### 2.3 大师定调

大师归纳：

- **优先瘦变量**；复杂对象分层版本号。
- **迁移**：必要时写批处理把旧变量升级到新 schema（维护窗口）。
- **测试**：老实例回放任务必须覆盖。

### 2.4 与「配置中心」的边界

小白追问：「规则都在 Nacos，变量里还要存吗？」大师答：若只是**开关**与**阈值**，配置中心更合适；若审计要求**当时生效的是哪版规则**，流程变量或历史注释里要有**可追溯引用**（规则 id + 版本）。两者不是替代关系，是**分层**：配置管「系统默认」，流程管「实例语境」。

## 3. 项目实战

### 3.1 环境前提

- 可插拔序列化 SPI（按官方扩展点）；测试库含旧实例。

### 3.2 步骤说明

1. 定义 `v1`/`v2` DTO 与转换器。
2. 读取变量时若缺字段则应用默认值或回填。
3. 部署后验证旧实例仍可 `complete`。
4. 文档化 breaking change 流程。
5. 与第 24 章「变量坑」交叉引用。

### 3.3 源码与说明

**为什么版本号**：比靠 `instanceId` 时间推断可靠。

### 3.4 验证

- 新旧实例并存无异常；监控反序列化错误率。

### 3.5 扩展：演进策略清单

1. **仅新增可选字段**：老实例缺省走默认分支；新实例写满字段。
2. **重命名**：保留旧 key 一段时间双写，或批处理迁移旧变量。
3. **类型变更**（如 String→Number）：禁止静默强转；显式转换失败时落人工处理节点。
4. **跨语言**：统一 JSON schema（Draft 版本固定），CI 校验示例消息。
5. **监控**：日志关键字 `Serialization`/`Spin`/`Variable` 告警阈值。

## 5. 附录：与第 5 章对照

第 5 章强调「变量要瘦」；本章解决「不得不复杂时如何演进」。二者同时遵守：**禁止在变量中存放唯一业务真相的大对象**——真相仍在业务库，变量只存键与版本化快照。

## 6. 课后作业（可选）

1. 选一个真实 DTO，写出 **v1→v2** 字段变更的兼容矩阵（新增/重命名/废弃）。  
2. 给出一组 **单元测试**：旧实例变量仍能反序列化；新实例写入 v2。  
3. 若使用 JSON：固定 **schema 版本号** 与 CI 校验命令。  
4. 讨论：什么时候应放弃变量对象，改为 **只存 id** 并在服务任务查询。  
5. 记录一次「失败演练」：故意改类全名，观察异常并写回 **第 24 章** 条目。  

## 7. 兼容策略的「三条铁律」（团队可贴墙）

**铁律一：能瘦则瘦。** 任何新需求先问：能否改为仅传 `id` 与展示用短字段；对象层能否下沉到领域服务。**铁律二：能显式则显式。** 版本号、`schemaVersion`、`payloadType` 等元数据必须可读、可 grep、可告警；禁止「魔法字符串」。**铁律三：能测则测。** 至少三类自动化：**老实例只读变量**、**新实例写新 schema**、**混合版本并发**（预发）。违反任一条，生产上多半会经历一次连夜回补与发版。

## 8. Spin JSON 与「手写字符串」的折中

若团队统一使用 **Spin JSON**（或等价）管理变量，可获得 XPath/路径访问便利，但仍需在代码层限制**深度与大小**；若完全手写字符串拼接 JSON，则必须在 CI 用 **jsonschema** 校验样例，避免括号错误导致运行时才发现。两类方案都要配合 **错误监控**：特定异常类型计数突增时，优先怀疑**部署错位**（新旧代码并存）或**类路径冲突**。

## 9. 章末提要（面向推广口播）

自定义序列化是 **高级补丁**：能救命，也能在半夜要你命。上线前问自己三句：**老实例还能打开吗？新实例写得对吗？发版回滚怎么办？** 若答不上，先加测试再加功能。与业务沟通时强调：**变量不是另一个业务库**，别指望在变量里解决全部报表难题。**克制是最高级的架构能力。**

## 10. 深度追问（写给半年后的自己）

1. **SPI**实现是否在独立模块，避免与业务代码 classpath 污染？  
2. 反序列化失败时，是否 **自动**开 incident 还是仅日志？  
3. 若使用 `Kryo`/protobuf 等，**跨语言**团队是否知情？  
4. **灰度发布**时新旧反序列化器并存时长上限？  
5. **监控**：`ClassNotFound` 与 `Conversion` 异常是否分渠道告警？  
6. **文档**：演进矩阵是否附在每次发版说明？  

**补白**：对象映射是跨团队合约；悄悄改包名等于撕毁合约。
