## 1. 项目背景

默认 demo 用户不能进生产。**授权**要回答：谁能部署流程、谁能读 Cockpit、谁能处理某租户的任务。**多租户** SaaS 场景还要隔离数据与定义。本章要解决的**一条主线问题**是：建立 **用户—组—权限—租户** 的入门心智，并强调**最小权限**与**与组织 IAM 集成**的路径（深入实现因项目而异）。

---

## 2. 项目设计（三角色对话）

*安全评审表上「网关已鉴权」打了勾，红线还在往下问。*
**小胖**：API 网关都验过 JWT 了，**引擎里再来一遍**？重复劳动吧？  
**小白**：网关防**进院**；引擎还要防**随便起流程、改别人任务、偷看租户 B 的实例**——粒度根本不是一个量级。  
**大师**：身份源优先 **LDAP/IdP**，Camunda 做映射；`tenantId` 穿在定义和实例上，**查询 API 不带租户过滤等于裸奔**。Cockpit **写权限**生产锁死，别让救火变纵火。  
**小白**：CI 机器人要部署 BPMN，密钥跟我的个人账号共用？  
**大师**：**服务账户**只给 deployment 最小权限， secret 进 Vault/CI，**定期轮换**——别把机器人当共享网盘。人账户与机账户混用，出事追责会像打地鼠。

## 3. 项目实战

### 3.1 环境前提

- Admin 可访问；测试用户若干。

### 3.2 步骤说明

1. 创建两组：`approvers`、`ops`；分配用户。
2. 为流程定义配置候选组与授权规则（按引擎支持方式）。
3. 验证非组成员无法 claim；无部署权限者无法上传 BPMN。
4. （概念）配置两租户 A/B，验证 A 用户看不到 B 实例。
5. 文档化生产：禁用 Admin 外网、审计变更。

### 3.3 源码与说明

REST 调用使用服务账户时，**最小权限**：仅授予完成任务所需 scope。

**为什么租户**：SaaS 与集团多子公司场景必备；漏租户等于数据泄露。

### 3.4 验证

- 越权操作返回 403；审计日志记录管理操作（第 21 章）。

### 3.5 多租户检查表

- [ ] 查询任务/实例 API 是否默认带 `tenantId` 过滤  
- [ ] 管理端创建流程定义是否写入正确租户  
- [ ] 报表 SQL 是否禁止跨租户 `SELECT *`  
- [ ] 日志是否打印租户上下文（脱敏）  
- [ ] 灾备恢复演练是否按租户抽样验证

## 5. 附录：与 API 网关协同

网关负责 **认证**（谁）与 **粗粒度路由**；引擎侧负责 **授权**（能对哪条流程/任务做什么）。二者职责清晰后，**403** 问题可在两层分别定位：网关 JWT 无效 vs 引擎资源权限不足。

## 6. 课后作业（可选）

1. 画出 **RBAC 矩阵**：角色 × 资源（部署/读取实例/完成任务）。  
2. 为 **服务账户**写一页运维手册：创建、轮换、吊销。  
3. 设计租户 **数据泄露**演练：A 用户能否通过改参数访问 B 租户 task。  
4. 对齐 IAM：**组**从何处同步、冲突如何处理。  
5. 与审计对齐：**谁在何时**改了授权策略（日志字段）。  

## 7. 章末提要（面向推广口播）

授权的本质是「**谁能对哪类流程资源做什么**」，而不是「能登录即可」。多租户场景下，还要叠加「**数据属于哪个组织**」这道护栏。推广培训时建议演示一次典型越权：用 B 租户 token 调 A 租户实例接口，应稳定失败并留下审计。对开发强调：REST 网关鉴权通过后，引擎侧仍需最小权限的服务账户；对运维强调：Admin 面板的网络暴露面最小化；对测试强调：租户边界与角色矩阵必须表驱动用例。若团队能输出一页 **RBAC + Tenant 矩阵** 并获得安全签字的「可上线」，本章目标达成。

## 8. 深度追问（写给半年后的自己）

1. **租户 id** 来自 JWT 还是数据库映射？被伪造时第二道校验是什么？  
2. 「超级管理员」名单是否 **双人复核**与**定期回收**？  
3. 服务账户 **密钥轮换**频率与失败回滚？  
4. 跨租户报表需求是否存在？若存在，审批链是什么？  
5. **数据导出**权限是否单独分级？  
6. 若并购导致租户合并，**实例迁移**策略是否已有预案？
