## 1. 项目背景

需要在**任务创建/分配/完成**或**执行流转**时写审计、同步外部系统、动态改 assignee——常用 **TaskListener、ExecutionListener** 与 **BpmnParseListener** 等扩展。本章要解决的**一条主线问题**是：选对监听点、控制**副作用与事务**，避免监听器里长调用拖死线程或导致**部分提交**（与第 23 章事务一致）。

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我在每个 Delegate 头尾复制粘贴日志不行吗？」

行，但横切关注点应集中；监听器减少重复，但要警惕**顺序**与**异常传播**。

### 2.2 小白追问

小白问：「第一，listener 抛错会怎样？第二，与 Spring 事件区别？第三，性能采样怎么做？」

### 2.3 大师定调

大师归纳：

- **异常**：可能回滚整个事务；需区分业务可恢复错误。
- **Spring 事件**：解耦但脱离流程事务语义时要小心。
- **性能**：避免 N+1 外部 HTTP；批量或异步。

### 2.4 何时用 Listener，何时用 Delegate？

小胖问：「那我把所有业务都写进 TaskListener 行不行？」大师答：**Delegate/服务任务**表达「完成一步业务动作」；**Listener**更适合**旁路审计、补数据、技术钩子**。若 listener 里写满业务规则，图上看不见，测试也难覆盖——这是典型**坏味道**。

## 3. 项目实战

### 3.1 环境前提

- 可在 BPMN 上配置 listener 或使用解析监听器。

### 3.2 步骤说明

1. 在任务 create 事件写审计日志（脱敏）。
2. 在 complete 前校验业务规则，不通过则抛异常阻止完成。
3. 压测对比加监听前后延迟。
4. 异常路径演练：审计写失败是否阻断流程（按需求）。
5. 文档化 listener 顺序与 owner。

### 3.3 源码与说明

**为什么慎用重逻辑 listener**：与主路径同线程，易放大延迟。

### 3.4 验证

- 审计完整；无意外双写；性能在预算内。

### 3.5 常见监听器清单（入门）

| 钩子 | 典型用途 | 风险 |
|------|----------|------|
| TaskListener create | 写入审计、计算候选人 | 远程调用超时 |
| TaskListener assignment | 通知、IM 推送 | 重复通知 |
| TaskListener complete | 汇总表单、校验 | 与事务边界耦合 |
| ExecutionListener start/end | 技术埋点 | 异常阻断流转 |

## 5. 附录：与拦截器（CommandInterceptor）分工

**Listener**偏业务生命周期；**CommandInterceptor**偏引擎命令层横切（如统一日志、租户校验）。平台型团队才会触及拦截器；业务项目优先 listener + 规范治理，避免深耦合内核升级路径。
