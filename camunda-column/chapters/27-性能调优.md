## 1. 项目背景

流程量上来后，**历史表**膨胀、查询变慢、备份窗口拉长。**History level** 决定记多少细节；**批量**与**索引**影响写入与查询；**清理**策略决定磁盘寿命。本章要解决的**一条主线问题**是：从配置到容量规划给出**调优清单**，并明确「审计要求」与「性能」的 trade-off。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「历史全开最保险吧？」

全量审计最完整，但存储与写入成本最高；合规允许时可降级别或异步归档。

### 2.2 小白追问

小白问：「第一，哪些表最热？第二，**异步历史**（若启用）注意点？第三，索引与 PG `VACUUM`？」

### 2.3 大师定调

大师归纳：

- **分级**：开发 `full`、生产按合规与容量选 `activity`/`audit`/`none`（具体枚举以版本文档为准）。
- **批量**：大批量完成用户任务时调批量大小减少 round-trip。
- **DBA**：定期统计、索引维护、分区（大表）。

### 2.4 再追问：「慢」在哪一端？

小白追问：「用户体感慢，是 **应用** 还是 **数据库**？」大师建议分层验证：先看 **应用 CPU/GC** 与 **REST P95**；再看 **DB active sessions、锁等待、缓冲池命中率**；最后看 **作业队列深度**。没有数据不要同时调大三处线程池——容易把 DB 打爆。

## 3. 项目实战

### 3.1 环境前提

- 压测环境；可改 history level 的许可。

### 3.2 步骤说明

1. 基线压测：固定 TPS 下记录 P95 延迟与 DB CPU。
2. 调整 history level，对比存储增长速率。
3. 打开批量相关配置（按文档）复测。
4. 分析慢查询与缺失索引。
5. 制定历史清理或归档策略与窗口（第 28 章）。

### 3.3 源码与说明

**为什么勿在生产盲试**：先在预发压测；变更需审批。

### 3.4 验证

- 指标达标；审计要求仍满足。

### 3.5 调优清单（可打印）

| 项 | 检查 | 备注 |
|----|------|------|
| history level | 是否与合规一致 | 变更需业务签批 |
| 批量大小 | 完成/删除是否分批 | 防长事务 |
| 连接池 | maxPoolSize 与节点数×实例 | 避免 DB 连接风暴 |
| 慢查询 | Top N SQL | 与 DBA 周会 |
| 作业线程 | 队列长度与处理时间 | 与第 22 章联动 |
| 磁盘 | 表空间增长率 | 触发归档（第 28 章） |

## 5. 附录：历史粒度与审计的折中话术（给业务）

可用三句话对齐期望：（1）**全量细节**最利于纠纷复盘，但成本最高；（2）**活动级**通常能支撑流程合规与大部分报表；（3）**无历史**仅适合纯自动、无审计要求的后台任务。任何降级都应有**书面确认**与**回滚方案**。

## 6. 课后作业（可选）

1. 在预发对 **history level** 做一次 A/B：记录 **磁盘日增量** 与 **查询 P95**。  
2. 抓一条 **Top 慢 SQL**（脱敏）并与 DBA 共评索引。  
3. 计算：若实例数 × 字段历史 = X，是否触发 **分区/归档**（概念即可）。  
4. 列出三项「**不能降历史**」的合规理由（若有）与替代（导出归档）。  
5. 产出一份一页纸 **容量预估**：明年 Q4 的吞吐与存储假设。  

## 7. 章末提要（面向推广口播）

性能工作永远是 **三角：审计完整、写入成本、查询体验**。没有业务签字就降历史级别，容易在事故后变成责任拉锯。对内可固定一句「**先量再调**」：用压测与慢 SQL 说话，而不是先拍脑袋加机器。清理与归档是性能与成本的终局战，应在第 28 章形成闭环。**记住：优化的第一行日志是慢查询，不是 JVM 参数。**

## 8. 深度追问（写给半年后的自己）

1. 你们 **P95** 目标与 **存储年增量**是否写在同一页 SLO？  
2. 若 DBA 要求拆库拆表，流程历史是否跟着分区策略走？  
3. 大批量完成任务的接口是否做过 **批处理窗口**与 **限流**？  
4. 历史清理后 BI 报表 **口径变更**谁来通知业务？  
5. 若降低 history level，**法务抽样**用什么替代证据？  
6. 压测数据是否与生产 **数据分布**一致（租户、流程 key）？
