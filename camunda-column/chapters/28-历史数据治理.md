## 1. 项目背景

三年前的流程实例是否必须在线可查？法务要求**留存七年**但可离线？**删除**与**归档**影响备份大小与合规审计。本章要解决的**一条主线问题**是：制定 **保留策略**（按业务类型、租户）、**批处理删除或导出**流程，并与法务对齐**免责声明**（以本组织政策为准）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「删表我还不会吗？」

随便删会破坏外键与审计链；需**分批、可回滚、可证明**的治理流程。

### 2.2 小白追问

小白问：「第一，删除运行中实例？第二，归档到对象存储的格式？第三，GDPR/个人信息删除？」

### 2.3 大师定调

大师归纳：

- **仅历史完结**且满足保留期才可清理。
- **归档**：导出到冷存储，元数据索引单独维护。
- **合规**：个人数据脱敏与删除流程与业务库联动。

---

## 3. 项目实战

### 3.1 环境前提

- 非生产演练库；完整备份。

### 3.2 步骤说明

1. 梳理法规与业务保留年限。
2. 选择官方 **History Cleanup** 或自定义批处理（按版本能力）。
3. 小批量试运行，验证 Cockpit 与报表仍满足需求。
4. 建立变更窗口与回滚（从备份恢复演练）。
5. 文档化「谁批准删除」与审计日志。

### 3.3 源码与说明

**为什么分批**：长事务锁表与大回滚风险。

### 3.4 验证

- 空间下降；无业务投诉；审计抽检通过。

### 3.5 扩展：批处理与窗口

- **窗口**：优先选业务低峰；与备份错开；提前公告影响范围（只读报表可能短暂不一致）。
- **分批**：按 `END_TIME` 或 `INSTANCE_ID` 范围分段删除/归档；每批记录行数与耗时，失败可断点续跑。
- **对账**：删除前后抽样核对「业务侧完结单」与「引擎历史」数量级；异常则暂停并回滚策略。
- **冷存储**：对象存储加密、访问审计、保留周期与法务一致；索引单独维护以便按 `businessKey` 检索。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 控成本；满足合规。 |
| 缺点 / 代价 | 流程重；误删难恢复。 |
| 适用场景 | 长周期生产、强合规行业。 |
| 不适用场景 | 无保留约束的小型系统。 |
| 注意事项 | 法务审批、备份、演练。 |
| 常见踩坑 | 一次删太多；无归档先删。 |

## 6. 课后作业（可选）

1. 写出你们行业对流程轨迹的**最短留存年限**与**可删除条件**（若不确定，标注「待法务确认」）。  
2. 设计一条 **批处理伪代码**：按实例结束时间滚动删除，包含**失败重试**与**暂停开关**。  
3. 与备份同事确认：**删库前**是否需要额外 **逻辑导出**（CSV/Parquet）到冷存储。  
4. 列出三类风险：**误删**、**删了还要查**、**删了影响报表**——各给一个缓解手段。  
5. 复盘：若某监管检查要求提供三年前的审批记录，你能否在 **SLA 时间**内从归档中取回？若不能，差距在哪？  

以上作业用于团队内部演练，**不构成法律意见**。
