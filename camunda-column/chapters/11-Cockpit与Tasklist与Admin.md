## 1. 项目背景

Camunda 默认提供多个 Web 应用：**Cockpit** 看运行与历史实例；**Tasklist** 处理待办；**Admin** 管理用户、组、租户（视版本与授权）。新人常把三者混为一谈，运维不知道备份与权限该盯哪里。本章要解决的**一条主线问题**是：说清**各应用职责**与**生产环境禁用默认账户、接入真实 IAM** 的注意事项（授权深化见第 17 章）。

---

## 2. 项目设计（三角色对话）

*浏览器里开了三个标签：Cockpit、Tasklist、Admin——像三顶帽子轮流戴。*
**小胖**：不都是一个 Camunda 登录吗？菜单多点而已，**分仨 app** 给谁加戏？  
**小白**：分工是**风险控制**：Cockpit **上帝视角**盯实例和部署；Tasklist **搬砖**认领待办；Admin **发账号**——你让产品经理开 Admin，等于给他_root。  
**大师**：生产常见策略：运维 Cockpit **尽量只读**；业务走 Tasklist 或自研门户嵌表单；高危操作（跳节点、改实例变量）走**审批 + 留痕**，第二十一是审计专场。  
**小胖**：我们自研了门户，还要 Tasklist 吗？  
**大师**：常保留引擎当后端，自建 UI；Tasklist 当备胎或仅内网救火——别把两套待办同时推给同一拨人**重复劳动**。多租户？Admin 里配 **tenant + membership**，实例/定义带 `tenantId`，租户间别串台。  
**小白**：大促夜**随手一改变量**，算.hero 还是算事故预备役？  
**大师**：算后者。双人复核、事后复盘，别靠英雄主义给 CMDB 填坑。

## 3. 项目实战

### 3.1 环境前提

- 第 2 章工程已启动；默认账户仅 dev。

### 3.2 步骤说明

1. 登录 **Cockpit**：找到流程定义、发起测试实例、查看 token 与变量（脱敏）。
2. 登录 **Tasklist**：认领并完成用户任务，观察列表筛选。
3. 登录 **Admin**：新建组、用户，分配 membership，回到 Tasklist 验证候选组生效。
4. 记录各 URL 路径与反向代理配置要点（第 33 章衔接）。
5. 列出生产 checklist：HTTPS、禁用 demo、审计日志（第 21 章）。

### 3.3 源码与说明

`application.yaml` 中 `camunda.bpm` 相关配置控制 Webapp 是否启用；生产可关闭部分 app 或置于内网。

**为什么拆分权限**：减少误操作与越权查看敏感流程变量。

### 3.4 验证

- 三应用可访问；组权限与任务可见性一致；非授权用户无法完成他人任务。

### 3.5 反向代理要点（Nginx 示例思路）

- 统一前缀 `/camunda` 或拆分三应用子路径；**HTTPS** 终止在网关。  
- **WebSocket**（如使用）需配置 upgrade 头。  
- **Admin** 仅内网 IP 白名单；公网暴露需 WAF。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 开箱即用；排障与学习成本低。 |

## 7. 章末提要（面向推广口播）

三件套用一句话：**Cockpit 看全局，Tasklist 干活，Admin 管钥匙**。培训非开发同事时，强调「**看到 ≠ 有权改**」；培训运维时强调「**内网 ≠ 安全**」。把默认账户从入职材料里删掉，改成「**申请—审批—开通**」。**截图教程尽量带水印与脱敏实例 id。**

## 8. 深度追问（写给半年后的自己）

1. 三应用的 **URL** 是否统一走 SSO？若有例外，原因是否备案？  
2. Cockpit 的 **危险操作**是否二次确认或审批流？  
3. Tasklist 是否限制 **批量操作**防误触？  
4. Admin 变更是否进入 **SIEM**？  
5. 是否每季度做一次 **默认账户扫描**？  
6. 截图与水印规范是否写入 **品牌/安全**文档？  

**补白**：界面再好用，也不能替代**权限与审计**这两根龙骨。
