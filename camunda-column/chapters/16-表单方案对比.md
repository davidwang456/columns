## 1. 项目背景

Tasklist 自带表单能力，但企业往往已有 **Vue/React 门户**。表单方案影响**安全（CSRF、XSS）、体验（移动端）、与流程变量的绑定方式**。本章要解决的**一条主线问题**是：对比 **嵌入式表单、外部表单 URL、前后端分离自研** 三种路径的取舍，并给出安全与维护性 checklist（第 4 章已讲 form key 基础）。

---

## 2. 项目设计（三角色对话）

*产品经理打开 Tasklist，隔壁团队打开自家 React——两台笔记本互相翻白眼。*
**小胖**：**全用 Tasklist** 最快吧？内网够用啊。  
**小白**：PoC 够用。要上**品牌色、复杂控件、跟主站 SSO**，你让业务在俩 UI 之间切？多数最后是 **自研前端 + REST complete**。  
**大师**：embedded 表单跟引擎**黏在一起**，快但难独立发版；`formKey` 指外部 URL 时，带上**签名或短时效 token**，别让改 `taskId` 当万能钥匙。校验：**前端体验，后端定罪**——绕过浏览器直接调 API 的一抓一把。  
**小白**：移动端和无障碍谁背锅？  
**大师**：自研反而好控；Tasklist 偏桌面内网。公众场景把 **a11y/i18n** 打进选型分，别上线后靠投诉驱动迭代。  

---

## 3. 项目实战

### 3.1 环境前提

- 至少一种表单跑通；自研前端可调 REST。

### 3.2 步骤说明

1. 在 Tasklist 使用 embedded 表单完成一任务（教程路径）。
2. 配置 `formKey` 为 `http://portal/task?tid=...`（示意），验证跳转。
3. 自研页面：`GET /task` 拉任务，`POST /complete` 提交变量。
4. 安全：HTTPS、任务查询权限校验、CSRF token。
5. 对比表记录：开发成本、运维成本、用户体验。

### 3.3 源码与说明

**为什么后端校验**：前端不可信；与权限模型一致（第 17 章）。

### 3.4 验证

- 非任务处理人无法 complete；变量注入测试失败。

### 3.5 对比表（示例）

| 方案 | 上线速度 | 品牌/体验 | 安全耦合 | 维护成本 |
|------|----------|-----------|----------|----------|
| 嵌入式 | 快 | 低 | 中 | 中 |
| 外部 URL | 中 | 中 | 需签名 | 中 |
| 自研 | 慢 | 高 | 可控 | 高 |

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 选型清晰可减少返工。 |
| 缺点 / 代价 | 自研成本高；嵌入式难品牌化。 |
| 适用场景 | PoC 用嵌入式；生产常自研或深度定制。 |
| 不适用场景 | 无 UI 的纯自动流程。 |
| 注意事项 | SSO、校验、审计、移动端。 |
| 常见踩坑 | taskId 泄露；HTTP 明文传敏感字段。 |

**延伸阅读**：第 4、13、17 章。

## 5. 附录：安全基线（最小集）

- HTTPS 全站；HSTS（按组织规范）。
- 任务查询与完成必须**鉴权**；禁止仅凭 taskId 盲调。
- 敏感字段**脱敏展示**；日志禁止打印完整身份证/卡号。
- CSRF：浏览器会话场景启用；纯 API token 场景按网关策略。

## 7. 章末提要（面向推广口播）

表单方案没有银弹：**快**与**好看**与**安全**三者时常互相打架。对内训可总结为三段话：第一段讲 Tasklist 能快速验证流程；第二段讲生产门户几乎必然自研或深度集成 SSO；第三段讲后端校验与权限不可绕过。测试团队应把「篡改 taskId」「重放提交」列入必修负面用例。读完本章若能在架构图上标出「浏览器—网关—引擎—表单服务」的边界与信任关系，就算真正理解。

## 8. 深度追问（写给半年后的自己）

1. 自研表单的 **CSRF** 策略与 **移动端**策略是否一致？  
2. 附件上传是否走 **独立对象存储**？病毒扫描谁负责？  
3. 「只读字段」在前后端是否双重实现？  
4. 表单 schema 变更是否需要 **流程版本**联动发布？  
5. 无障碍（若要求）责任在哪个角色？  
6. 国际化文案谁拥有翻译 keys 的最终审批？  

**补白**：表单是用户唯一可见的「流程」；体验差会被记到流程引擎头上，哪怕根因是前端。
