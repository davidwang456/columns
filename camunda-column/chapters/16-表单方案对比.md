## 1. 项目背景

Tasklist 自带表单能力，但企业往往已有 **Vue/React 门户**。表单方案影响**安全（CSRF、XSS）、体验（移动端）、与流程变量的绑定方式**。本章要解决的**一条主线问题**是：对比 **嵌入式表单、外部表单 URL、前后端分离自研** 三种路径的取舍，并给出安全与维护性 checklist（第 4 章已讲 form key 基础）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「全用 Tasklist 最省事吧？」

省事适合 PoC；品牌化、复杂 UI、与主站 SSO 统一时，多数团队**自研前端 + REST 完成任务**。

### 2.2 小白追问

小白问：「第一，**embedded** 表单部署在引擎 war 里吗？第二，外部 URL 如何传 taskId 防篡改？第三，变量校验放前端还是后端？」

### 2.3 大师定调

大师归纳：

- **嵌入式**：快速；与引擎耦合紧。
- **外部 URL**：跳转门户；需签名校验或短期 token。
- **前后端分离**：体验最佳；后端必须重复校验（防绕过）。

### 2.4 移动端与无障碍

小白追问：「移动端表单谁做？」大师答：**自研前端**更易控体验；Tasklist 适合桌面内网。若面向公众，**无障碍**与**多语言**也应在选型中打分，而不是上线后补洞。

---

## 3. 项目实战

### 3.1 环境前提

- 至少一种表单跑通；自研前端可调 REST。

### 3.2 步骤说明

1. 在 Tasklist 使用 embedded 表单完成一任务（教程路径）。
2. 配置 `formKey` 为 `http://portal/task?tid=...`（示意），验证跳转。
3. 自研页面：`GET /task` 拉任务，`POST /complete` 提交变量。
4. 安全：HTTPS、任务查询权限校验、CSRF token。
5. 对比表记录：开发成本、运维成本、用户体验。

### 3.3 源码与说明

**为什么后端校验**：前端不可信；与权限模型一致（第 17 章）。

### 3.4 验证

- 非任务处理人无法 complete；变量注入测试失败。

### 3.5 对比表（示例）

| 方案 | 上线速度 | 品牌/体验 | 安全耦合 | 维护成本 |
|------|----------|-----------|----------|----------|
| 嵌入式 | 快 | 低 | 中 | 中 |
| 外部 URL | 中 | 中 | 需签名 | 中 |
| 自研 | 慢 | 高 | 可控 | 高 |

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 选型清晰可减少返工。 |
| 缺点 / 代价 | 自研成本高；嵌入式难品牌化。 |
| 适用场景 | PoC 用嵌入式；生产常自研或深度定制。 |
| 不适用场景 | 无 UI 的纯自动流程。 |
| 注意事项 | SSO、校验、审计、移动端。 |
| 常见踩坑 | taskId 泄露；HTTP 明文传敏感字段。 |

**延伸阅读**：第 4、13、17 章。

## 5. 附录：安全基线（最小集）

- HTTPS 全站；HSTS（按组织规范）。
- 任务查询与完成必须**鉴权**；禁止仅凭 taskId 盲调。
- 敏感字段**脱敏展示**；日志禁止打印完整身份证/卡号。
- CSRF：浏览器会话场景启用；纯 API token 场景按网关策略。
