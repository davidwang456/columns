## 1. 项目背景

移动端、中台、合作伙伴系统往往通过 **HTTP** 与流程引擎交互：发起实例、查询待办、完成任务。与 Java API 相比，REST **语言无关**、易走 API 网关，但要处理**认证、幂等、分页与错误码**。本章要解决的**一条主线问题**是：用 REST **安全地**完成「启动—查询—完成」闭环，并列出与测试自动化对接的要点（与第 18 章衔接）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我直接调 Java API 不更快吗？」

同进程内当然优先 Java API；跨服务、跨语言、跨团队边界时 REST 更常见。注意：**事务边界**不再共享，需用业务幂等与重试设计。

### 2.2 小白追问

小白问：「第一，**Basic** 与 **OAuth2/Bearer** 怎么选？第二，重复 `POST` 启动会不会多实例？第三，`firstResult`/`maxResults` 分页漏数据怎么防？」

### 2.3 大师定调

大师归纳：

- **认证**：生产禁止明文；对接组织统一 IAM 或 API 网关。
- **幂等**：用 `businessKey` + 业务侧去重，或接口幂等键；Camunda 本身不替你防「重复下单」。
- **分页**：按排序字段稳定分页；大批量用异步导出或流式方案。

### 2.4 429 与限流

小白追问：「大促时被打爆怎么办？」大师答：网关 **rate limit** + 引擎连接池上限 + **队列化**峰值请求；别指望单靠 Camunda 扛住全站流量——它是**编排中枢**，不是边缘 CDN。

## 3. 项目实战

### 3.1 环境前提

- 引擎 REST 启用；有测试用户；curl 或 Postman。

### 3.2 步骤说明

1. `POST /process-definition/key/{key}/start` 启动，body 带 variables。
2. `GET /task?assignee=...` 或 candidate 查询待办。
3. `POST /task/{id}/complete` 完成任务。
4. 用错误 `businessKey` 再启动一条，验证两条实例隔离。
5. 编写集合测试：401/403/404 与引擎错误响应解析。

### 3.3 源码与说明

启动 body 示例（节选）：

```json
{
  "businessKey": "ORD-2026-0001",
  "variables": {
    "amount": {"value": 99, "type": "Integer"}
  }
}
```

**为什么带 businessKey**：运维与业务追踪友好；消息关联常用。

**安全**：HTTPS、最小权限账户、敏感变量脱敏日志（第 21 章）。

### 3.4 验证

- Postman 集合可重复运行；与 CI 集成的 smoke 测试通过。

### 3.5 常用端点记忆卡（入门）

| 目的 | 思路 |
|------|------|
| 启流程 | `POST .../start` |
| 查待办 | `GET /task` + 过滤条件 |
| 完成 | `POST /task/{id}/complete` |
| 查实例 | `GET /process-instance` |

具体路径以官方 **OpenAPI/文档**为准，版本升级时核对。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 解耦、易集成、易测试。 |
| 缺点 / 代价 | 网络故障与幂等复杂；需网关治理。 |
| 适用场景 | 跨系统编排、前后端分离。 |
| 不适用场景 | 同 JVM 高频同步调用可优先 Java API。 |
| 注意事项 | 认证、分页、限流、幂等。 |
| 常见踩坑 | 变量类型遗漏；权限不足 403；大结果集一次拉爆。 |

**延伸阅读**：第 18 章自动化测试；官方 REST 参考。

## 5. 附录：与 Java API 选型

| 场景 | 倾向 |
|------|------|
| 同服务内 | `RuntimeService`/`TaskService` |
| 跨语言/跨团队 | REST |
| 极高频同步 | 评估是否应直接 Java API + 合并部署 |
