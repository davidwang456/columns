## 1. 项目背景

折扣阶梯、风控阈值、请假天数对应审批层级——规则若全写在 Java `if-else` 或 BPMN 网关表达式里，业务想调就要开发改代码、发版。**DMN** 把规则表格式化，命中策略清晰，可单独版本管理与评审。本章要解决的**一条主线问题**是：用一张最小决策表算出**是否需要 HR 备案**（或类似输出），并在 BPMN 中**调用决策任务或委派**，对比「全写网关条件」的边界。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「Excel 里也能维护规则啊。」

Excel 缺**正式语义**与**引擎执行路径**；DMN 1.x 是标准，Camunda 可直接执行并记录**命中规则**（视历史级别与配置）。业务与 IT 对「优先级、命中策略」有共同语言。

### 2.2 小白追问

小白问：「第一，**FIRST** 与 **COLLECT** 命中策略差别？第二，DMN 与数据库配置表方案谁更易审计？第三，规则爆炸时怎么治理？」

### 2.3 大师定调

大师归纳：

- **命中策略**决定多行匹配时取哪条或如何聚合；选错会导致与业务预期相反。
- **审计**：DMN 决策可随流程版本管理；大表配置需另建治理流程。
- **治理**：按域拆分多张表、限制列数、建立评审与仿真机制。

### 2.4 业务与 IT 如何一起评审？

小胖问：「规则表业务能看懂吗？」大师答：DMN 的价值就在于**共同语言**——列是输入输出，行是案例。评审会议应带 **边界样例**（null、极大值、边界日期）与 **命中策略**一页说明，避免「以为 FIRST 其实是 PRIORITY」。

## 3. 项目实战

### 3.1 环境前提

- Camunda Modeler 支持 DMN；流程引擎启用 DMN 引擎。

### 3.2 步骤说明

1. 新建 `leave-tier.dmn`，输入 `days`（number），输出 `needHr`（boolean）。
2. 规则示例：`days > 3` → `true`；否则 `false`。
3. 在 BPMN 中用 **Business Rule Task** 绑定 DMN，或将结果写入变量供网关使用。
4. 部署后发起多组变量，核对输出。
5. 与业务一起评审表头含义与命中策略文档化。

### 3.3 源码与说明

BPMN 中规则任务（概念）引用 DMN 决策 key；执行后变量 `needHr` 可用于排他网关。

**为什么用 DMN**：规则tabular，易 diff；比网关长表达式可读。

**边界**：极动态规则（每分钟变）仍可能更适合配置中心 + 服务任务。

### 3.4 验证

- Cockpit/日志可见决策结果；单元测试对 DMN 输入输出表驱动（第 18 章）。

### 3.5 小练习（可当作业）

输入：`customerTier=GOLD`，`amount=120`；期望输出：`discount=0.15`。让业务写三行规则，开发补 **默认行** 与 **命中策略** 说明；双方在 Modeler 里跑一遍 **Evaluate**（若工具支持）。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 规则可视化；业务可读；易版本化。 |
| 缺点 / 代价 | 学习命中策略；复杂规则可能不如代码清晰。 |
| 适用场景 | 阶梯、评级、资格判定。 |
| 不适用场景 | 强过程依赖、需多步推理的规则引擎。 |
| 注意事项 | 命中策略、数据类型、空值。 |
| 常见踩坑 | 类型不一致；策略选错；表与流程版本不匹配。 |

**延伸阅读**：第 12 章与请假联动；官方 DMN 引擎文档。

## 5. 附录：何时不要用 DMN

当规则需要 **多轮对话**、**机器学习评分**、或 **实时特征工程**，DMN 可能不合适；此时用 **服务任务** 调专用风控服务，流程只负责编排与留痕。
