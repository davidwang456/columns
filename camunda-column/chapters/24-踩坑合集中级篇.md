## 1. 项目背景

生产问题往往不是单一 bug，而是**组合拳**：变量塞了整个 JSON 报表导致 DB 膨胀；多节点部署下重复订阅消息；并发更新触发 **OptimisticLockingException**。本章以**条目化手册**形式给出**现象—原因—对策—预防**，便于测试设计负面用例、运维快速检索。

### 3.2 步骤说明

### 3.5 条目详解（现象 / 原因 / 对策 / 预防）

**A. 变量过大**

- **现象**：DB 体积暴涨、完成用户任务变慢、备份时间拉长；偶发 OOM 或序列化超时。
- **原因**：把报表、附件 Base64、整包 JSON 塞进变量；历史与运行表同存大对象。
- **对策**：改存 `reportId`/`fileId`；大对象走对象存储；必要时历史降级或异步归档（第 27、28 章）。
- **预防**：变量大小上限评审；CI 对「变量快照」做抽样检查。

**B. 乐观锁（OptimisticLockingException）**

- **现象**：并发完成同一任务、并发写同一变量时抛错；重试后可能「已成功一次」。
- **原因**：多线程/多节点同时更新同一执行或变量行。
- **对策**：业务层串行化关键操作；重试带**幂等**；对 UI 使用「乐观 UI + 二次确认」。
- **预防**：测试覆盖并发；关键 API 限流。

**C. 重复订阅 / 重复消息**

- **现象**：同一业务事件启动多条流程实例；或 correlate 多次导致重复副作用。
- **原因**：消息中间件 at-least-once；消费者无幂等；业务键未对齐。
- **对策**：`businessKey` 去重表；启动前查询是否已存在实例；下游接口幂等键（第 15 章）。
- **预防**：契约测试 + 重放演练 + DLQ 监控。

**D. 作业积压「假死」**

- **现象**：流程不前移、定时器不响、异步点卡住。
- **原因**：Job 线程池过小、DB 慢、长事务占锁。
- **对策**：调线程池与连接池、查慢 SQL、拆分长事务（第 22 章）。
- **预防**：作业延迟指标告警；压测。
