## 1. 项目背景

单节点扛不住或要消除 SPOF，需要**多实例**部署 Camunda 应用。此时 **Job Executor**、**会话**、**数据库** 成为焦点：作业会不会重复执行？Sticky session 要不要？本章要解决的**一条主线问题**是：理解 Camunda 7 集群的**基本假设**（共享数据库、作业锁），并列出**会话亲和**与**无状态 REST** 的运维取舍。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我 K8s 起三个 Pod，负载均衡一开，不就高可用了？」

若 Web 会话落在内存，用户可能被踢到不同 Pod 丢会话；需 **session replication** 或 **无状态 + token**。引擎侧多节点依赖**共享 DB** 协调作业。

### 2.2 小白追问

小白问：「第一，Job 重复执行可能吗？第二，DB 成为瓶颈怎么办？第三，滚动发布时注意什么？」

### 2.3 大师定调

大师归纳：

- **共享数据库**是多数集群模式基石；确保连接池与锁竞争可接受。
- **作业**：通过锁避免双执行；业务 Delegate 仍要幂等。
- **发布**：滚动时关注长事务与作业积压；健康检查要区分**存活**与**就绪**（第 33 章）。

### 2.4 会话：有状态 Web 的坑

小白追问：「Tasklist 登录后负载均衡到另一台会怎样？」大师答：要么 **Sticky Session**，要么 **会话外置（Redis）**，要么 **OAuth 无状态**。Camunda 应用本身是否有会话取决于集成方式；**最省心**是前置 API 网关统一认证，业务 UI 自建无状态调用 REST。

## 3. 项目实战

### 3.1 环境前提

- 至少两实例共享同一 Camunda schema；负载均衡器。

### 3.2 步骤说明

1. 部署双实例，验证 Tasklist 登录会话策略（按架构选型）。
2. 压测作业执行，观察是否单点积压。
3. 停一实例，验证另一实例接管作业与请求。
4. 记录 DB 连接总数与最大会话。
5. 演练滚动重启与版本升级窗口。

### 3.3 源码与说明

**为什么幂等仍重要**：极端情况下仍可能出现重复副作用业务调用。

### 3.4 验证

- SLA 下无大量失败；无会话错乱报告。

### 3.5 部署拓扑草图（文字版）

- **LB** → **N 个 Camunda App** → **同一 DB 集群**（主从/HA 以 DBA 方案为准）。
- **External Task Worker** 可独立 Deployment，横向扩展；与引擎进程解耦。
- **只读报表**走从库或异步投影，避免拖慢主事务。

## 5. 附录：RPO/RTO 与流程引擎

若业务要求 **RPO≈0**，仅靠多应用节点不够，**数据库**与**备份**策略才是瓶颈；引擎多节点解决的是**应用层高可用**，不是磁盘损坏时的数据魔法恢复。与业务方对齐「可接受丢失窗口」非常关键。
