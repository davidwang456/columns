## 1. 项目背景

员工提交请假：填写天数与事由，直属领导审批；若天数超过阈值则再经 HR 备案；全程希望**可追溯**：谁提交的、当前卡在哪、历史审批意见是什么。许多团队第一次接触 Camunda，会卡在「模型画好了，实例跑不起来」或「Tasklist 里看不到任务」。本章主线问题是：**从 BPMN 模型、流程变量、用户任务分配到最终在 Tasklist 完成审批，端到端最小闭环如何落地**，并明确测试与运维各看一眼就能懂的验证点。

## 2. 项目设计（三角色对话）

*会议室里摊着 HR 的纸质请假单和一张 BPMN——两代人互相瞪眼。*
**小胖**：请假不就领导点头吗？表里两行状态搞定，**上引擎是不是装**？还有「候选人」「受托人」，**像相亲节目**，人话版有没有？  
**小白**：你先说说加「抄送全员、并行总监、自动扣年假」第几版开始删库跑路？表驱动状态机在**分支爆炸**面前只有墓志铭写得好看。  
**小胖**：……行。那接力棒到底给谁？`assignee` 和候选组同时写，会不会俩人互相指「以为对方签」？  
**大师**：把实例想成**接力棒**：`assignee` 是棒在**张三手上**；候选组是棒在**经理池**里，谁 `claim` 谁接力。评审里写死规则，**禁止同任务既点名又撒网却不定义优先级**——那不是灵活，是扯皮接口。  
**小白**：变量里塞整份**员工档案**呢？序列化哪天炸？  
**大师**：放 **id + 请假天数** 等决策字段就够；胖数据让 HR 服务查。反序列化雷常在**翻历史**或 **类改版**时响——约定 **JSON + schema** 或版本化 DTO，别靠「当时能跑」。  
**小胖**：端到端验收我_assert 数据库行数行不行？  
**大师**：分三件事验：**（1）模型能 deploy；（2）任务出现在该在的节点、候选人没串；（3）走完能 **history** 复盘。**测试盯路径**，运维盯版本，开发盯监听器别越权**偷改业务表**——第十八单再把自动化拧紧。

## 3. 项目实战

### 3.1 环境前提

- JDK 17+，Maven 3.9+，Camunda 7.20+ 与 Spring Boot 版本以 `VERSIONS.md` 为准。
- 示例目录（规划）：`examples/camunda-column-examples/part1-basic/ch12-leave-request/`。若仓库中尚未提交 runnable 工程，可按下列**文件清单**自建，与正文一致即可验证。

### 3.2 步骤说明

1. **建模**：在 Camunda Modeler 新建 BPMN，流程 id 建议 `leave-request`。节点包含：开始事件 → 用户任务「直属审批」→ 排他网关（条件：天数是否大于阈值）→ 用户任务「HR 备案」或跳过 → 结束事件。网关上定义条件表达式，例如 `${days > 3}`（具体属性名与变量一致）。
2. **部署**：将 BPMN 放入 `src/main/resources/processes/`，启用 Camunda Spring Boot starter 的自动部署（第 2 章模式）。启动应用后，在 Cockpit 确认流程定义已出现。
3. **启动实例**：通过 REST 或 Java API 传入变量：`starter`（申请人）、`days`、`reason`。注意变量名与网关表达式一致。
4. **任务处理**：打开 Tasklist（或自定义 UI），以候选人组/用户登录，认领并完成「直属审批」，在表单或 API 中写入 `approved` 等变量；继续驱动令牌前进。
5. **断言结束**：在 Cockpit 查看该实例是否到达结束事件；历史表中是否有完整审计轨迹（入门可先以界面为准）。

### 3.3 源码与说明

最小工程通常包含：

- `pom.xml`：`camunda-bpm-spring-boot-starter`、`spring-boot-starter-web` 等依赖由父 BOM 管理。
- `application.yaml`：`camunda.bpm.admin-user` 等内置用户（仅开发环境），生产需换认证体系。
- `processes/leave-request.bpmn`：流程定义源文件。
- （可选）`Delegate` 类：若「年假额度校验」用服务任务实现，避免把复杂规则塞进网关表达式。

**为什么用户任务优先用组**：企业组织常变动，写死 `assignee` 不利于岗位替换；组 + 认领是常见模式。若必须直达个人，可在启动监听器里根据组织数据解析为 `assignee`。

**为什么阈值规则可部分用 DMN**：当规则表格化维护更合适时，第 10 章的 DMN 可与本章联动；本章保持 BPMN 网关以降低概念数量。

### 3.4 验证

- **开发自测**：启动后 `POST` 启动流程（可用 Camunda REST），返回实例 id；查询活动任务列表应出现「直属审批」。
- **测试同学**：设计用例——（1）短假仅直属；（2）长假经 HR；（3）拒绝路径是否结束或退回（依你模型）。
- **运维同学**：确认应用日志无部署异常；数据库连接与表自动创建是否符合环境策略（H2 仅开发）。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 端到端故事把「模型—实例—任务—历史」串成一条线，便于全员建立共同语言。 |
| 缺点 / 代价 | 模型过简会误导「上线即简单」；模型过复杂会在第一章综合课就劝退——需控制第一节端到端的分支数量。 |
| 适用场景 | 培训破冰、PoC 验收、跨部门演示。 |
| 不适用场景 | 已熟练团队可直接跳入集成与性能专题。 |
| 注意事项 | 变量命名与表达式统一编码规范；生产环境禁用默认管理员口令。 |
| 常见踩坑 | 网关条件写错导致**死实例**；候选人组拼写与身份系统不一致；表单字段与变量不同步。 |

**延伸阅读**：用户任务表单（第 4 章）、REST 驱动（第 13 章）、测试策略（第 18 章）。本章是基础篇合拢点，建议读者完成后用自己的业务场景替换「请假」，复述一遍端到端步骤，以检验是否真正理解。
