## 1. 项目背景

BPMN 改了网关、加了节点——新版本部署后，**正在跑的实例**仍按旧定义执行，还是自动变新？迁移工具要不要上？CI/CD 流水线如何做到**可回滚**？本章要解决的**一条主线问题**是：讲清 **Process Definition 版本**语义、**运行中实例**与**新定义**的关系，并给出升级前检查清单（完整迁移见官方 migration 工具与专项评估）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我覆盖同名文件重新 deploy，线上正在审批的单子会变吗？」

通常**不会自动跳到新图**：运行实例绑定**部署 id**。新部署产生**新版本号**；新启动的实例用新版本。混用两版并存是常态，直到旧实例结束。

### 2.2 小白追问

小白问：「第一，**tenant** 与版本关系？第二，改 Java Delegate 类名算不算流程变更？第三，回滚部署怎么做？」

### 2.3 大师定调

大师归纳：

- **版本**：key 相同、version 递增；API 启动时可指定 version 或 latest。
- **Delegate 改名**：属代码变更，需保证旧部署仍能找到 Bean（或保持兼容别名）。
- **回滚**：重新部署旧 BPMN（生成新 version 号）+ 新实例指向旧逻辑；已运行实例仍按原 deployment 绑定，除非做迁移。

### 2.4 灰度：谁该用新版本？

小胖问：「新版本一发布，所有新单都走新图吗？」大师答：**默认**是 latest 部署；若需灰度，应通过 **API 网关路由**、**feature flag** 或 **显式 version** 启动，配合业务侧试点名单。与产品/运营对齐「灰度窗口」与「回滚条件」。

## 3. 项目实战

### 3.1 环境前提

- 可重复部署；Cockpit 可见多版本。

### 3.2 步骤说明

1. 部署 v1 流程，启动实例 A 停在用户任务。
2. 修改 BPMN（加节点）部署 v2，启动实例 B。
3. 观察 A 仍走旧结构，B 走新结构。
4. 记录运维策略：何时清理旧版本定义、文档化「兼容窗口」。
5. （阅读）官方 migration 场景：大改模型需迁移运行实例时的工具链。

### 3.3 源码与说明

启动时指定版本（概念）：

```java
runtimeService.startProcessInstanceByKeyAndVersion("leave", 2, vars);
```

**为什么显式 version**：灰度时控制新流程仅部分流量。

### 3.4 验证

- Cockpit 显示两版本；实例绑定正确；无意外升级。

### 3.5 CI/CD 建议步骤

1. **构建**：单元测试 + BPMN XML schema 校验（如有）。  
2. **镜像**：`docker build` + 漏洞扫描。  
3. **部署**：Helm upgrade；**DB 迁移**作为前置 Job。  
4. **冒烟**：启动实例 + 完成一任务 + 查询历史。  
5. **回滚**：保留上一镜像 tag；必要时 **冻结**新流程定义部署。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 版本隔离保护运行实例；支持灰度。 |
| 缺点 / 代价 | 多版本并存增加治理成本；大改需迁移。 |
| 适用场景 | 任何长期演进系统。 |
| 不适用场景 | 无（只要会改模型就要懂版本）。 |
| 注意事项 | 与业务沟通停机窗口；监控旧实例数量。 |
| 常见踩坑 | 误以为覆盖文件会改运行实例；Delegate 不兼容。 |

**延伸阅读**：官方 migration；第 33 章 CI/CD。

## 5. 附录：与 Git 协作

BPMN/DMN **必须**进 Git；评审用 **Pull Request**；**禁止**手工在服务器上改模型不回流。版本标签建议与发布说明（Release Notes）关联，便于事后追溯。

## 7. 章末提要（面向推广口播）

版本化的核心是 **不要把「部署」理解成覆盖文件**：运行中实例绑定的是某次部署快照。推广时一句话：**新单走新图，老单走完老图，除非做迁移项目**。CI 里加一段 **BPMN 校验** 往往能拦截 80% 的低级错误。**发版前在预发点一遍「旧实例仍在跑」的冒烟，是成熟团队的肌肉记忆。**

## 8. 深度追问（写给半年后的自己）

1. **Delegate 兼容层**如何通过代码审查强制执行？  
2. 若业务坚持「**全部实例立刻升级**」，迁移项目边界与预算？  
3. 版本号是否进入 **客服工单**查询体系？  
4. **回滚**后新实例启动策略是否冻结？  
5. 多租户下版本是否 **分租户灰度**？  
6. Git tag 与 **DB schema**版本是否在发布单双向绑定？
