## 1. 项目背景

「希望业务订单表和流程**同一事务提交**」——在**单库**嵌入式场景有时可行；**跨库跨服务**则不行。乐观锁、隔离级别、长事务都会导致「流程已前进、业务回滚」或反之。本章要解决的**一条主线问题**是：讲清 **Delegate 与 Spring `@Transactional`** 的常见组合、**跨库**时的最终一致策略，以及 **OptimisticLockingException** 类问题如何重试（与第 25 章架构一致）。

---

## 2. 项目设计（三角色对话）

*数据架构评审，有人在白板上画了三个圈：引擎库、订单库、钱包——只有一个打了勾。*
**小胖**：`Delegate` 里 `@Transactional` 顺手写订单表，**不是跟流程一坨提交**吗？  
**小白**：**同库同事务管理器**才可能一家亲；一分库、一 RPC，边界就裂。**XA 不是银弹**，别拿来当心理按摩。  
**大师**：PoC 同事务省事；生产要看**长事务**和连接占用。库分离就 **Saga/补偿/消息** 上岗，重试**必幂等**；只读副本查运行实例？**延迟**和一致性先签合同。并发改同一实例变量——**乐观锁冲突**自己设计重试。  
**小胖**：订单表一份状态，变量里再 mirror 一份，**双保险**？  
**大师**：双写容易双谎：**真相源**只能有一个——业务主数据权威，变量存**引用键 + 结论**；需要快照就版本号拍清楚。报表和客服各念各的经，那叫企业文化事故。

## 3. 项目实战

### 3.1 环境前提

- Spring 管理事务；可选两数据源模拟。

### 3.2 步骤说明

1. 在 Delegate 同一事务写业务表 + 引擎变量（单数据源）。
2. 模拟下游失败，观察流程与业务是否一致回滚。
3. 切换为双数据源：业务提交成功、流程失败时的对账脚本思路。
4. 并发 complete 同一任务（负面），观察异常与处理。
5. 文档化「真相源」：订单状态以业务库为准还是以流程为准（需架构决策）。

### 3.3 源码与说明

**为什么明确真相源**：避免两库各说各话。

### 3.4 验证

- 无脏数据；对账任务可发现不一致。

### 3.5 对账脚本思路（文字）

每日批处理比对「订单完结」与「流程实例结束」数量；**差异**进入人工队列。字段级对账可抽样 `businessKey` 对齐。若使用 **双库**，对账必须**异步**且**幂等**，避免与业务高峰争用连接。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 正确事务边界减少资金类事故。 |
| 缺点 / 代价 | 分布式一致性成本高。 |
| 适用场景 | 金融、订单、库存。 |
| 不适用场景 | 纯查询报表。 |
| 注意事项 | 幂等、重试、对账、监控。 |
| 常见踩坑 | 长事务锁表；忽略乐观锁。 |

**延伸阅读**：第 20、25、18 章。

## 5. 附录：给 DBA 的三句话

（1）引擎表与业务表**同库**时，关注**长事务**与**索引**；（2）**分库**时，关注**跨库一致性**与**对账**；（3）**备份恢复**演练要包含「流程实例进行中」场景，避免只验空库。

## 6. 课后作业（可选）

1. 画一张**时序图**：下单成功 → 启动流程 → Delegate 写业务表 → 任一步失败时的回滚边界。  
2. 写出你们系统的**真相源**声明：订单状态以哪张表为准。  
3. 设计一条**对账 SQL/脚本**（可伪代码）：发现「流程结束但订单未付」的记录。  
4. 讨论：**消息外发**与引擎事务同事务的风险，以及替代方案（Outbox，若采用）。  
5. 压测一个「并发 complete」用例，记录 **OptimisticLockingException** 率与重试策略。  

## 7. 章末提要（面向推广口播）

本章反复出现的词是 **真相源** 与 **边界**：同事务能省事，但难扩展；分库能解耦，但要 **补偿与对账**。给架构评审的结论是：先写下「**订单权威在哪个库**」，再谈「流程变量该不该复制订单状态」。给开发的具体约束是：Delegate 中禁止隐式长事务与无界远程调用；给 DBA 的诉求是：关注锁等待与慢查询高峰是否与流程高峰重合。若团队在复盘会上能共识「我们不指望分布式 XA 包治百病」，就已经比多数第一次上流程引擎的项目前进一大步。

## 8. 深度追问（写给半年后的自己）

1. 「同事务」的边界是否画在 **单服务**内？若否，请画出时序图并标红风险。  
2. Outbox（若引入）与流程实例的 **发布顺序**如何保证？  
3. 业务回滚成功但流程已前进，**人工介入**节点是否已在 BPMN 设计？  
4. 跨库对账是 **T+1** 还是 **实时**？差异如何向财务解释？  
5. 你们是否为 **幂等键**建立了独立表或唯一索引？  
6. 若使用读写分离，**读从历史**是否会读到不一致？如何兜底？
