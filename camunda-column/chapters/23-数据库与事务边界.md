## 1. 项目背景

「希望业务订单表和流程**同一事务提交**」——在**单库**嵌入式场景有时可行；**跨库跨服务**则不行。乐观锁、隔离级别、长事务都会导致「流程已前进、业务回滚」或反之。本章要解决的**一条主线问题**是：讲清 **Delegate 与 Spring `@Transactional`** 的常见组合、**跨库**时的最终一致策略，以及 **OptimisticLockingException** 类问题如何重试（与第 25 章架构一致）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我在 Delegate 里 `@Transactional` 写订单表，不就和流程一起提交了？」

同数据源且同一事务管理器时可能成立；一旦分库或远程调用，事务边界分裂。

### 2.2 小白追问

小白问：「第一，引擎库与业务库**物理分离**推荐模式？第二，**重试**与业务幂等？第三，只读副本能否查运行实例？」

### 2.3 大师定调

大师归纳：

- **同事务**：PoC 可；生产要评估连接占用与长事务风险。
- **分离**：流程库与业务库用 **Saga/补偿** 或可靠消息；避免幻想分布式 XA 万能。
- **乐观锁**：并发更新同一实例变量可能冲突，需重试策略。

### 2.4 典型反模式：「双写双真相」

小胖问：「订单状态在订单表，流程变量也存一份，算不算双保险？」大师答：若**不一致**，谁对外？若未定义**真相源**，报表与客服口径会分裂。建议：**业务主数据**为权威，流程变量仅存**引用键**与**审批结论**；需要快照时显式版本号。

## 3. 项目实战

### 3.1 环境前提

- Spring 管理事务；可选两数据源模拟。

### 3.2 步骤说明

1. 在 Delegate 同一事务写业务表 + 引擎变量（单数据源）。
2. 模拟下游失败，观察流程与业务是否一致回滚。
3. 切换为双数据源：业务提交成功、流程失败时的对账脚本思路。
4. 并发 complete 同一任务（负面），观察异常与处理。
5. 文档化「真相源」：订单状态以业务库为准还是以流程为准（需架构决策）。

### 3.3 源码与说明

**为什么明确真相源**：避免两库各说各话。

### 3.4 验证

- 无脏数据；对账任务可发现不一致。

### 3.5 对账脚本思路（文字）

每日批处理比对「订单完结」与「流程实例结束」数量；**差异**进入人工队列。字段级对账可抽样 `businessKey` 对齐。若使用 **双库**，对账必须**异步**且**幂等**，避免与业务高峰争用连接。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 正确事务边界减少资金类事故。 |
| 缺点 / 代价 | 分布式一致性成本高。 |
| 适用场景 | 金融、订单、库存。 |
| 不适用场景 | 纯查询报表。 |
| 注意事项 | 幂等、重试、对账、监控。 |
| 常见踩坑 | 长事务锁表；忽略乐观锁。 |

**延伸阅读**：第 20、25、18 章。

## 5. 附录：给 DBA 的三句话

（1）引擎表与业务表**同库**时，关注**长事务**与**索引**；（2）**分库**时，关注**跨库一致性**与**对账**；（3）**备份恢复**演练要包含「流程实例进行中」场景，避免只验空库。

## 6. 课后作业（可选）

1. 画一张**时序图**：下单成功 → 启动流程 → Delegate 写业务表 → 任一步失败时的回滚边界。  
2. 写出你们系统的**真相源**声明：订单状态以哪张表为准。  
3. 设计一条**对账 SQL/脚本**（可伪代码）：发现「流程结束但订单未付」的记录。  
4. 讨论：**消息外发**与引擎事务同事务的风险，以及替代方案（Outbox，若采用）。  
5. 压测一个「并发 complete」用例，记录 **OptimisticLockingException** 率与重试策略。
