## 1. 项目背景

并行网关、子流程、多实例会让**执行树**变复杂。理解 **PVM（Process Virtual Machine）** 或等价执行语义，有助于解释「为什么 token 卡在这」。本章要解决的**一条主线问题**是：用**直觉**描述 **Execution** 与 **Activity** 关系，并指向源码中**推进、分裂、合并**的关键路径（不替代官方文档）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我就想看 BPMN XML，不想看执行树。」

XML 是静态；**运行态**是树/图上的实例。疑难 bug 往往在「执行树状态」与「预期」不一致。

### 2.2 小白追问

小白问：「第一，**scope** 与并行实例？第二，**incident** 与错误事件？第三，调试 API 有哪些？」

### 2.3 大师定调

大师归纳：

- **ExecutionQuery** 可查当前执行点；Cockpit 图形化。
- **Incident**：技术失败与业务异常入口；与运维告警联动（第 21 章）。
- **深入源码**：从 `PvmExecutionImpl`（类名以版本为准）入手，**谨慎**，升级会变。

### 2.4 并行为什么「数量对不上」？

小胖看到三条 execution 懵了。大师解释：**并行分叉**会产生多条执行路径；**子流程**可能再嵌套 scope。入门只需建立直觉：**汇合前 execution 数量与分叉语义一致**；若不一致，优先怀疑**模型未汇合**或**某分支 incident**。

## 3. 项目实战

### 3.1 环境前提

- 本地可调试；示例流程含并行网关。

### 3.2 步骤说明

1. 在并行分叉处打断点（若有自定义扩展）或打日志。
2. 查看 DB `ACT_RU_EXECUTION` 行数与预期。
3. 阅读一版 `Pvm*` 相关源码走读笔记（团队内）。
4. 对照 BPMN 活动 id 与 execution 的 `activityId`。
5. 形成「卡死」排查清单：网关、事件、作业、外部任务。

### 3.3 源码与说明

**为什么结合 DB**：运行态真相在库表与引擎内存一致。

### 3.4 验证

- 能解释一次真实并行实例的 execution 数量与原因。

### 3.5 建议阅读顺序（源码）

1. `ProcessEngineConfiguration` 初始化参数。
2. `ExecutionEntity` / 运行表映射关系（概念）。
3. `AtomicOperation` 跳转（不必一次读完）。
4. 结合 **DB 表** `ACT_RU_EXECUTION` 对照实例。

## 5. 附录：调试口诀

「**先看 BPMN 活动 id，再看 execution 的 activityId，再看 job/external task，最后才怀疑内核 bug**」。多数问题出在模型与集成，而非引擎本身。
