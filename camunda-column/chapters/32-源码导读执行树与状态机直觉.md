## 1. 项目背景

并行网关、子流程、多实例会让**执行树**变复杂。理解 **PVM（Process Virtual Machine）** 或等价执行语义，有助于解释「为什么 token 卡在这」。本章要解决的**一条主线问题**是：用**直觉**描述 **Execution** 与 **Activity** 关系，并指向源码中**推进、分裂、合并**的关键路径（不替代官方文档）。

---

## 2. 项目设计（三角色对话）

*Cockpit 上三条 execution，小胖数手指对不上。*
**小胖**：XML 我看懂了，**执行树**不是给引擎自嗨用的吗？  
**小白**：XML 是**静态图**；运行态是**遗嘱执行现场**。卡 bug 多半是 execution 状态和你脑补不一致。  
**大师**：`ExecutionQuery` / `ACT_RU_EXECUTION` 对齐；**Incident** 接告警（第二十一单说过了）。啃 `PvmExecutionImpl` 可以，**升级会搬家**，别抄类名当圣经。  
**小胖**：并行咋多出一条 execution？  
**大师**：分叉 token、子流程 **scope** 都会加戏——**汇合前数量要和分叉语义对得上**；对不上先想**网关没收口**或某分支 **incident**，别先怪数据库心情不好。

## 3. 项目实战

### 3.1 环境前提

- 本地可调试；示例流程含并行网关。

### 3.2 步骤说明

1. 在并行分叉处打断点（若有自定义扩展）或打日志。
2. 查看 DB `ACT_RU_EXECUTION` 行数与预期。
3. 阅读一版 `Pvm*` 相关源码走读笔记（团队内）。
4. 对照 BPMN 活动 id 与 execution 的 `activityId`。
5. 形成「卡死」排查清单：网关、事件、作业、外部任务。

### 3.3 源码与说明

**为什么结合 DB**：运行态真相在库表与引擎内存一致。

### 3.4 验证

- 能解释一次真实并行实例的 execution 数量与原因。

### 3.5 建议阅读顺序（源码）

1. `ProcessEngineConfiguration` 初始化参数。
2. `ExecutionEntity` / 运行表映射关系（概念）。
3. `AtomicOperation` 跳转（不必一次读完）。
4. 结合 **DB 表** `ACT_RU_EXECUTION` 对照实例。

## 7. 章末提要（面向推广口播）

源码导读的目的不是背类名，而是建立「**现象—机制—证据**」的排障链路：卡住时先对齐 BPMN activity 与 execution，再决定该不该下钻 `Pvm*`。向团队强调：并行、子流程、多实例才是复杂度的来源；大部分线上问题仍是建模与集成。**建议把本章当作地图**：第一次看不懂正常，第二次排障翻回来看笔记才有用。

## 8. 深度追问（写给半年后的自己）

1. 并行汇合「少一路」时，你能否在 **Cockpit + DB** 上复现并写出一页 RCA？
