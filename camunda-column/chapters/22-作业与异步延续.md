## 1. 项目背景

实例「停」在某节点不动，可能不是卡业务，而是**作业未执行**：定时器、异步延续、失败重试都在**作业表**里排队。线程池配小了会积压；配大了可能打满数据库。本章要解决的**一条主线问题**是：理解 **Job Executor** 如何拉取与执行作业，并给出**调优与排查**入口（与第 27 章性能衔接）。

---

## 2. 项目设计（三角色对话）

*监控里 JVM 很闲，用户却说「流程卡成 PPT」。*
**小胖**：CPU 空转，凭什么还慢？  
**小白**：线程可能在**等 DB、抢作业锁、排队 async 点**——空 CPU 常见，别装看不见。  
**大师**：引擎**作业**是内置调度；**External Task** 是外部 Worker 抢锁——负载和故障模式不一样。**exclusive** 让同一实例的作业串行，少写冲突，读懂文档再调并发。线程池、队列、索引、批量，全要**压测数字**说话，别拍脑袋加线程。  
**小白**：Tasklist 有「待办」，Cockpit 也叫 pending，**同一个 pending 吗**？  
**大师**：**人待办** ≠ **token 等事件** ≠ **作业没跑**。排障：**先 user task** → 再中间事件/接收 → 再 job/external job → 最后才怀疑 DB/升级事故——**别遇事就重启**，那是掩耳盗铃 Pro。

## 3. 项目实战

### 3.1 环境前提

- 可查看引擎库作业表或 Cockpit 作业视图。

### 3.2 步骤说明

1. 在建模中勾选异步延续 before/after 某任务（按教程）。
2. 发起实例，观察短暂时延与作业记录。
3. 调低线程池故意制造积压，记录恢复手段。
4. 监控作业执行耗时分布。
5. 文档化与 DB 连接池配比关系。

### 3.3 源码与说明

**为什么谨慎滥用异步**：调试难度上升；分布式追踪必配。

### 3.4 验证

- 积压可告警；调参后延迟下降。

### 3.5 运维排障速查

1. **作业线程是否饱和**：线程池队列是否持续增长。
2. **DB 是否慢**：`ACT_RU_JOB` 或等价视图是否有大量锁定等待（以实际库表为准）。
3. **时钟**：多节点时间漂移导致定时器异常（少见但致命）。
4. **长事务**：Delegate 中同步 HTTP 超时占用连接。
5. **集群重复执行错觉**：实际是业务回调重试，需幂等。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 解耦长步骤；提高吞吐潜力。 |
| 缺点 / 代价 | 排查难；依赖线程与 DB。 |
| 适用场景 | 高负载、定时器多、异步延续明确。 |
| 不适用场景 | 极简流程盲目开异步。 |
| 注意事项 | 监控、线程池、DB 容量。 |
| 常见踩坑 | 作业线程池过小；死锁与长事务。 |

**延伸阅读**：第 9、27 章。

## 5. 附录：与 External Task 的选型一句话

**作业**：引擎进程内调度，适合短步骤、与事务紧密衔接。**External Task**：适合长耗时、异构语言、独立扩缩。二者不要混用同一业务语义的两套实现，否则排障时不知道卡在内置作业还是外部队列。

## 7. 章末提要（面向推广口播）

可以把 **Job** 想成引擎内部的「**待办调度**」：线程池吃紧时，不是 CPU 图表好看就能解释一切，而要去看**队列**与 **DB 等锁**。给运维的 take-away 是：先拉作业延迟与慢 SQL，再动线程池配置；给开发的 take-away 是：异步延续越多，排障越依赖追踪与日志。**若「pending」成为热词，先查本章再开会**。

## 8. 深度追问（写给半年后的自己）

1. 作业线程池与 **业务线程池**是否隔离？若不隔离，争用如何量化？  
2. 定时器密集的场景，是否评估过 **DB 写入放大**？  
3. `exclusive` 作业对吞吐的影响能否用压测复现？  
4. 你能否解释：为何「异步延续」不等于「异步业务成功」？  
5. Job 失败重试与 **Delegate 异常**如何映射到同一告警面板？  
6. 若迁移到多节点，作业重复执行的 **业务幂等**是否全覆盖？
