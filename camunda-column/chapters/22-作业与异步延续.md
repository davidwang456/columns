## 1. 项目背景

实例「停」在某节点不动，可能不是卡业务，而是**作业未执行**：定时器、异步延续、失败重试都在**作业表**里排队。线程池配小了会积压；配大了可能打满数据库。本章要解决的**一条主线问题**是：理解 **Job Executor** 如何拉取与执行作业，并给出**调优与排查**入口（与第 27 章性能衔接）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「我 CPU 很空，为啥流程还慢？」

可能作业线程池在等待 DB、或作业锁竞争、或**异步点**太多导致排队。

### 2.2 小白追问

小白问：「第一，**异步延续**与 External Task 差别？第二，`exclusive` 作业含义？第三，如何安全提高并发？」

### 2.3 大师定调

大师归纳：

- **作业**：引擎内置调度；External Task：外部 Worker。
- **exclusive**：同一实例作业串行避免并发写冲突（概念需对照文档）。
- **调优**：线程数、队列、DB 索引、批量大小——需压测数据支撑。

### 2.4 小白：「pending」到底是谁的 pending？

小白追问：Tasklist 里待办是**人**的任务；Cockpit 里「卡住」可能是 **token 等待** 或 **作业未执行**。大师建议排障顺序：**先看是否有 user task** → 无则看 **中间事件/接收** → 再看 **作业/外部任务** → 最后才怀疑 **DB 锁或版本 bug**。避免一上来就重启应用掩盖根因。

## 3. 项目实战

### 3.1 环境前提

- 可查看引擎库作业表或 Cockpit 作业视图。

### 3.2 步骤说明

1. 在建模中勾选异步延续 before/after 某任务（按教程）。
2. 发起实例，观察短暂时延与作业记录。
3. 调低线程池故意制造积压，记录恢复手段。
4. 监控作业执行耗时分布。
5. 文档化与 DB 连接池配比关系。

### 3.3 源码与说明

**为什么谨慎滥用异步**：调试难度上升；分布式追踪必配。

### 3.4 验证

- 积压可告警；调参后延迟下降。

### 3.5 运维排障速查

1. **作业线程是否饱和**：线程池队列是否持续增长。
2. **DB 是否慢**：`ACT_RU_JOB` 或等价视图是否有大量锁定等待（以实际库表为准）。
3. **时钟**：多节点时间漂移导致定时器异常（少见但致命）。
4. **长事务**：Delegate 中同步 HTTP 超时占用连接。
5. **集群重复执行错觉**：实际是业务回调重试，需幂等。

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 解耦长步骤；提高吞吐潜力。 |
| 缺点 / 代价 | 排查难；依赖线程与 DB。 |
| 适用场景 | 高负载、定时器多、异步延续明确。 |
| 不适用场景 | 极简流程盲目开异步。 |
| 注意事项 | 监控、线程池、DB 容量。 |
| 常见踩坑 | 作业线程池过小；死锁与长事务。 |

**延伸阅读**：第 9、27 章。

## 5. 附录：与 External Task 的选型一句话

**作业**：引擎进程内调度，适合短步骤、与事务紧密衔接。**External Task**：适合长耗时、异构语言、独立扩缩。二者不要混用同一业务语义的两套实现，否则排障时不知道卡在内置作业还是外部队列。
