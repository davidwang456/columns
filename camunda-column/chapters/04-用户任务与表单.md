## 1. 项目背景

流程「卡」在用户任务上：屏幕上要出现谁的待办？是固定张三，还是「部门经理池里认领」？表单上的字段如何写回流程变量、再被网关读取？很多缺陷来自**分配策略**与**数据绑定**不清晰。本章要解决的**一条主线问题**是：讲清 **assignee / candidateUsers / candidateGroups** 的语义与典型组合，并说明 **form key** 与内嵌表单、外部表单在入门阶段如何取舍（深度对比见第 16 章）。

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖笑：「候选人、候选组，听着像选秀节目。我就想让经理批，写死经理工号不行吗？」

行，但经理请假、调岗时你就要**改模型或改数据**。**候选组**把「岗位角色」与「具体的人」解耦：权限系统里把人放进 `managers` 组，流程只认组。

### 2.2 小白追问

小白问：「第一，同时配了 assignee 和 candidateGroups，到底听谁的？第二，**委托**与**转办**在引擎层是什么 API？第三，测试环境没有 LDAP，怎么模拟组？」

### 2.3 大师定调

大师归纳：

- **assignee**：任务直接属于某人，池子里不再「待认领」（除非业务再改）。
- **candidateUsers / candidateGroups**：任务进入**候选列表**，需 `claim` 再 `complete`。适合职责共享、负载分担。
- **优先级**：具体配置以引擎行为为准；团队规范应写清「禁止同时混用导致歧义」。
- **表单**：`formKey` 指向表单定义或外部 URL；变量通过提交表单写回流程。前后端分离项目常自定义 UI，仅把 Tasklist 当参考实现。
- **测试**：用 Camunda 自带用户/组 API 或初始化脚本建组，不必上完整 LDAP。

### 2.4 委托、转办与「谁负责」

小白追问：「代理人请假了怎么办？」大师答：模型层可用 **委托（delegate）** 或组织数据驱动 **重新分配**；产品上要区分「临时委托」与「永久调岗」。无论哪种，**审计日志**应记录实际操作人（第 21 章）。

## 3. 项目实战

### 3.1 环境前提

- 第 2 章最小工程；BPMN 中至少一个用户任务。
- Tasklist 可访问。

### 3.2 步骤说明

1. 在 BPMN 用户任务上设置 `candidateGroups`，例如 `managers`。
2. 启动流程实例，变量带业务单号。
3. 以属于 `managers` 的用户登录 Tasklist，应能看到待办；执行认领与完成，并提交表单字段（如 `approved`）。
4. 在网关条件中使用 `${approved == true}` 之类表达式（与第 7 章联动）。
5. 编写一条**负面用例**：非组成员看不到任务（测试同学可据此写断言）。

### 3.3 源码与说明

BPMN 片段（示意）：

```xml
<bpmn:userTask id="approveTask" name="经理审批"
  camunda:candidateGroups="managers"
  camunda:formKey="embedded:app:forms/approve.form" />
```

**为什么用 candidateGroups**：组与组织身份对齐，模型不随人事变动频繁改。

Java 侧也可在启动监听器里动态设置 assignee（灵活但需单测）：

```java
// 监听器中（概念示意）
delegateTask.setAssignee("userId");
```

**为什么谨慎动态分配**：规则复杂时建议集中在一个监听器或委派服务，避免散落在多个网关。

### 3.4 验证

- Tasklist 任务列表与预期一致；完成后 token 前进。
- 自动化：用 REST 查询 task、complete 并断言流程变量（第 13、18 章深化）。

### 3.5 测试同学：负面用例清单

- 非候选人尝试 `claim` / `complete`  
- 表单缺必填字段（若校验在引擎侧/后端侧）  
- 并发认领（若业务允许抢单）

## 4. 项目总结

| 维度 | 内容 |
|------|------|
| 优点 | 人机协作语义清晰；组模型贴近企业权限。 |
| 缺点 / 代价 | 与组织主数据不同步时「看得见任务但不该看见」会扯皮。 |
| 适用场景 | 审批、复核、任何需要人输入的节点。 |
| 不适用场景 | 纯自动流水线（用服务任务/外部任务）。 |
| 注意事项 | 生产身份集成与最小权限（第 17 章）。 |
| 常见踩坑 | 组名拼写错误；忘记 claim；表单字段名与变量不一致。 |

**延伸阅读**：第 12 章端到端；第 16 章表单方案大对比。

## 5. 附录：与组织主数据同步

组名应与 **HR/IdM** 系统一致；定期对账「经理组成员」与流程候选人是否匹配。人走了、组还在，是常见事故源。

## 6. 课后作业（可选）

1. 画 **泳道图**草图：业务方如何对应 `candidateGroups`。  
2. 建两个测试用户：验证 **认领-完成**与**非认领直接完成**（若允许）差异。  
3. 导出 **formKey** 与前端字段映射表。  
4. 写一条 **权限缺陷**用户故事：看见不该见的任务。  
5. 与产品确认：**加签/转办**是否在本期范围（若否，记入 roadmap）。  

## 7. 章末提要（面向推广口播）

用户任务连接 **组织身份** 与 **流程**：候选人/受托人不是玄学，是权限模型。对内培训用生活例子「**泳池里认领一条泳道**」往往比念文档有效。**若 Tasklist 里任务总不对，先查组，再查建模，最后才怀疑引擎 bug。**

## 8. 深度追问（写给半年后的自己）

1. **代理人**规则是否与 HR 政策同步（请假/离职）？  
2. 候选组与 **组织架构**部门是否一一映射？例外如何处理？  
3. 表单字段与 **审计要求**是否逐项对齐？  
4. **批量审批**（若上）权限模型是否单独评审？  
5. **移动端**认领体验是否纳入验收？  
6. **多语言**任务标题数据源是否统一？  

**补白**：人机协作的摩擦，多半来自组织数据，不是来自 XML。
