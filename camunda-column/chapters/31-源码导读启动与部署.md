## 1. 项目背景

排障「为什么没部署成功」「Bean 没注入」时，**翻源码**比猜配置快。本章要解决的**一条主线问题**是：沿着 **Spring Boot Starter** 初始化 **ProcessEngineConfiguration**、**部署扫描**、**JobExecutor** 启动的路径，标出**关键类名**（版本以 7.20.x 为例，升级时自行 diff）。

---

## 2. 项目设计（三角色对话）

### 2.1 小胖开球

小胖说：「源码几万行，从哪看起？」

从 **启动入口** 与 **自动配置类** 起，跟到 `ProcessEngine` bean 创建；不要一上来读执行树。

### 2.2 小白追问

小白问：「第一，与纯 `camunda.cfg.xml` 差异？第二，哪些 Bean 可覆盖？第三，升级小版本关注点？」

### 2.3 大师定调

大师归纳：

- **Spring Boot** 路径：`CamundaBpmAutoConfiguration`（类名以实际版本为准）。
- **扩展**：通过 `ProcessEnginePlugin` 注入自定义行为。
- **升级**：看 release notes 中 **database schema** 与 **配置项** 变更。

### 2.4 小胖：「Bean 都有了为啥还部署失败？」

大师答：常见原因包括 **BPMN XML 不合法**、**资源未进类路径**、**租户/权限**限制、**数据库迁移未执行**。源码阅读时优先搜 **Deployment** 相关异常栈，再回退到资源扫描配置。

## 3. 项目实战

### 3.1 环境前提

- IDE 可下载 sources；依赖与专栏 `VERSIONS.md` 一致。

### 3.2 步骤说明

1. 在 IDE 中导航 `camunda-bpm-spring-boot-starter` 自动配置。
2. 跟踪创建 `ProcessEngine` 的工厂方法。
3. 找到 `deployments` 相关调用，对应自动部署。
4. 记录 `JobExecutor` 启动位置与默认线程配置。
5. 画一张**启动时序草图**（类名列表）团队内分享。

### 3.3 源码与说明

**为什么标注版本**：同名类在不同版本可能有包调整。

**勿抄类名进生产文档而不写版本**：易误导后人。

### 3.4 验证

- 能解释一次「部署失败」日志从哪抛出；能找到覆盖点。

## 7. 章末提要（面向推广口播）

启动链阅读的价值在于：**下次报 `deployment failed` 时你能定位到「资源没进来」还是「DB 迁移没跑」**。把类名写进 wiki 时请带 **引擎小版本**。给新人一句劝：**先看自动配置与插件点，不要先跳实现细节**。升版时对照 release notes 的 **DB 变更**是第一要务。

## 8. 深度追问（写给半年后的自己）

1. 你能否口述从 **Spring Boot main** 到 **首个 BPMN 出现在 RepositoryService** 的七步？  
2. 若启用多引擎（极少数场景），Bean 命名冲突如何解？  
3. `ProcessEngineConfiguration` 哪些项你们团队**禁止在 YAML 改**（必须走代码）？  
4. 若 CI 偶发「库未迁完就探针通过」，根因可能在何处分层？  
5. 升级失败后**回滚数据库**与**回滚应用**的顺序如何写 Runbook？  
6. 你是否保存了每个环境的 **schema 版本号**查询方式（SQL 或 API）？
