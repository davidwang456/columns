# 第 22 章：访问日志：格式、字段与脱敏

**受众提示**：运维管日志采集与存储；安全审脱敏；开发少打重复日志。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

Envoy **Access Log** 记录每次请求的关键字段，可对接 **ELK/Loki**。网格层日志与**应用日志**互补：前者**统一**，后者含**业务语义**。推广时重点：**字段标准**、**PII 脱敏**、**与 trace id 关联**。

---

## 2. 项目设计：大师与小白的对话

**小白**：有了 Prometheus 还要 access log 吗？

**大师**：指标看**趋势**；日志看**单次请求细节**与**审计**。

**小白**：默认格式够吗？

**大师**：常要加 **%REQ(x-request-id)%** 等；注意 **性能**。

**小白**：日志里的 Authorization 怎么办？

**大师**：**脱敏**或 **不打**；合规团队签字。

**要点清单**

1. **结构化**（JSON）便于检索；
2. **采样**或**限流**防刷屏；
3. **保留期**与 **GDPR**。

---

## 3. 项目实战

### 3.1 Telemetry API / EnvoyFilter（视版本）

优先使用 **Telemetry** 资源配置访问日志（推荐路径以官方为准）；教学环境可临时用网关 access log。

### 3.2 查看 Sidecar 日志

```bash
kubectl logs deploy/productpage -c istio-proxy --tail=50
```

### 3.3 与 trace 关联

在日志格式中包含 **trace_id**（若 Envoy 与 OTel 集成允许）。

---

## 4. 项目总结

**优点**

- **审计**与**合规**证据；
- **个案**排障。

**缺点**

- **体量**大、**成本**高；
- 配置错误易**泄露**敏感信息。

**适用场景**

- 金融、**安全审计**、**客户投诉**定位。

**注意事项**

- **日志注入**攻击防护；
- **集群级**与 **网关级**日志分层。

**常见踩坑**

1. **现象**：磁盘满。**原因**：**全量**访问日志到节点。**处理**：集中采集 + 轮转。
2. **现象**：字段为空。**原因**：TLS 终止位置导致 **某些头**不可见。**处理**：调整日志模板。
3. **现象**：性能下降。**原因**：同步写日志。**处理**：异步 sink / 采样。

---

### 4.1 再谈一个场景：日志与指标双写

同一事件既打 metrics 又打全量 access log，可能造成**成本叠加**。建议分层：**默认采样/聚合**，个案排障再拉全量。

### 4.2 合规

日志出境与留存周期要符合公司政策；**脱敏规则**需安全团队确认。

---

### 4.3 日志与指标字段对齐

同一概念（如 `user_id` 哈希、`tenant`）在日志与指标里的**命名**应一致，减少 join 成本。安全团队对字段白名单的变更要走**评审**。

### 4.4 成本：存储与查询

访问日志全量进 ES 可能非常贵；采用**采样**、**热冷分层**或**仅错误全量**等策略，并在 FinOps 例会上回顾。
