# 第 15 章：金丝雀发布：从 5% 到 100% 的运维剧本

**受众提示**：运维主导 Runbook；开发提供健康检查与指标；测试定义准入与回滚判据。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

技术对象（VS 权重、Header）已在前章覆盖；本章讲**组织流程**：如何把金丝雀从「能配」变成「敢上生产」。包括：**前置检查**、**阶段表（5%→20%→50%→100%）**、**每步观测指标**、**一键回滚**与**沟通机制**。Istio 是工具，**SRE 实践**才是成败关键。

---

## 2. 项目设计：大师与小白的对话

**小白**：5% 谁定的？

**大师**：来自**风险承受度**与**统计显著性**粗算——不是数学最优，是**团队共识**。

**小白**：每步等多久？

**大师**：至少覆盖一个**业务高峰**周期；并看 **ERROR BUDGET**（第 25 章）。

**小白**：回滚是改权重还是滚 Deployment？

**大师**：**优先**改权重（快）；若新版本**本身**有毒，要**滚镜像**。

**小白**：CI/CD 怎么接？

**大师**：流水线改 **Git 中的 VS/Helm values**，合并触发 **渐进 PR** 或 **GitOps Reconcile**。

**要点清单**

1. **准入**：健康检查、SLO、无 P0 缺陷。
2. **观测**：RED + 业务 KPI 同屏。
3. **权限**：谁可改生产路由 RBAC。

## 3. 项目实战

### 3.1 Runbook 表格（模板）

| 阶段 | 权重 | 最短观察 | 通过条件 | 回滚 |
|------|------|----------|----------|------|
| T0 | 0% / 镜像预发 | 1h | 预发指标正常 | 保持旧版 100% |
| T1 | 5% | 2h | 5xx < 0.1%, P99 不劣化 > 10% | 权重回 0% |
| T2 | 20% | 4h | 同上 + 业务 KPI 无异常 | 权重回 T1 |
| T3 | 50% | 8h | 同上 | 权重回 T2 |
| T4 | 100% | 24h 观察 | 无告警升级 | 紧急滚 Deployment |

### 3.2 权重 YAML 片段（示例）

```yaml
  http:
  - route:
    - destination:
        host: api
        subset: stable
      weight: 95
    - destination:
        host: api
        subset: canary
      weight: 5
```

### 3.3 观测命令（节选）

```bash
kubectl get pods -l app=api
istioctl proxy-status
# Grafana：按 destination workload 过滤 5xx、latency
```

## 4. 项目总结

**优点**

- **分钟级**流量切换，比纯 Deployment 更贴近用户体验风险。
- 与 **GitOps** 兼容。

**缺点**

- 流程若不制度化，易变成**个人英雄主义**操作。
- **数据库 schema** 变更需配套策略（双写/兼容期）。

**适用场景**

- 高频发布、**互联网 ToC**、**SLA 敏感**业务。

**注意事项**

- **会话粘性**与多版本 API 兼容性（第 16 章）。
- **移动端强制升级**与后端不同步时的沟通。

**常见踩坑**

1. **现象**：指标正常但客诉爆。**原因**：**业务 KPI** 未纳入。**处理**：补齐漏斗指标。
2. **现象**：回滚后仍异常。**原因**：**数据**已写坏或**缓存**未失效。**处理**：数据层预案。
3. **现象**：权重改了不生效。**原因**：**缓存网关**或 **CDN**。**处理**：全链路排查。

---

### 4.1 再谈一个场景：数据库与契约

金丝雀只切流量不够：若新版本引入**不兼容 schema**，小比例用户也可能触发**写路径**故障。Runbook 中应包含：**向后兼容期**、**双写/双读**、**回滚后数据修复**条目。

### 4.2 与组织沟通

设定「**谁有权推进下一阶段**」：建议由 oncall + 业务负责人共同签字，而不是单人拍脑袋。

### 4.3 「人」与「权限」

Runbook 里写清：**谁有权推进下一阶段**、**谁必须在线观察**、**谁可以拍板回滚**。避免「大家都在群里但没人负责」。与 **PagerDuty** 或值班表集成，演练时也要按真实角色到场。

### 4.4 与数据库迁移的耦合

若金丝雀包含**schema 变更**，Runbook 必须引用**迁移工单**与**回滚脚本**位置。只切流量不改数据，往往在回滚时出现「代码回去了，数据回不去」。

---

### 4.5 本章小结：把知识变成「可执行」

推广 Istio 时，最容易失败的不是「不会配」，而是「配了没人敢动」。建议每章学完后，在团队看板增加一张卡片：**谁能在生产执行与本章相关的变更**、**需要哪些审批**、**回滚命令是什么**。技术文档若不能映射到流程，就仍是幻灯片。

### 4.6 与第 24、31、36 章的联动提示

- **排障**：把 `istioctl analyze` 与 `proxy-config` 写进 oncall 第一步（第 24 章）。
- **交付**：把安装与 values 纳入 GitOps（第 31 章）。
- **治理**：把试点范围、预算与里程碑写成公开路线图（第 36 章）。

### 写给不同角色的一句话

- **开发**：把「业务语义」与「网格语义」对齐，避免把幂等/鉴权全推给平台。
- **运维**：把变更纳入窗口与 GitOps，任何手工 kubectl 都要留痕。
- **测试**：把策略生效写成可重复断言，而不是一次性手工点点点。

### 常见误解澄清（防抬杠小抄）

- 「上了网格就不用关心网络」：错误；网络/CNI/防火墙仍是底座。
- 「网格能自动让系统高可用」：错误；高可用需要架构、容量、数据与演练。
- 「配置在 YAML 里就是真相」：不完整；真相是 **集群实际生效配置**（见 istioctl proxy-config）。
