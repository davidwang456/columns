# 第22章 核心能力篇复盘：从对象模型到运维闭环

## 22.1 项目背景

**碎片化知识需要一张“总图”**

第二部分涉及 VirtualService、DestinationRule、Gateway、ServiceEntry、安全策略、可观测性与出站治理。单章掌握容易，但生产故障往往跨越多个对象——例如证书问题表现为路由失败，配额问题表现为 Sidecar 启动失败。

**从功能列表走向平台思维**

团队真正需要的是：**标准模板**（可复用的 YAML 与参数基线）、**变更门禁**（analyze + CI）、**观测基线**（黄金信号 + 访问日志字段）三位一体的闭环。

## 22.2 项目设计：小胖、小白与大师的「对象协作图」

**场景设定**：小白要给新人十分钟讲清「请求从入口到出站」；小胖嫌对象太多想记「一张图走天下」。

**第一轮**

> **小胖**：Gateway、VirtualService、DestinationRule……背完我就忘了，能再短点吗？
>
> **小白**：证书问题为啥会表现成 404？
>
> **大师**：记三条线：**入口**（Gateway→VirtualService→Service）、**安全**（PeerAuthentication / RequestAuthentication→AuthorizationPolicy）、**出口**（ServiceEntry→可选 Egress→外网）。DR 管连接/子集/熔断，Telemetry 管观测。证书与 hosts 不一致时，**表象可能是 404/连接重置**，要串起来想。
>
> **大师 · 技术映射**：**对象协作 = 分层责任；排障 = analyze → proxy-config → 网络。**

**第二轮**

> **小白**：排查顺序？
>
> **大师**：**先 `istioctl analyze`，再 `proxy-config`，最后才怀疑 CNI/内核**——多数问题前两步能收口。

**类比**：乐队里 Gateway 定拍，VirtualService 指挥，DR 是力度，安全是准入，Telemetry 是录音。

## 22.3 项目实战：复盘清单

**步骤 1：对照表自检**

| 主题 | 自检问题 |
|:---|:---|
| 入口 | Gateway `hosts` 与证书 SAN 是否一致 |
| 路由 | VirtualService 匹配顺序是否由严到宽 |
| 子集 | subset 标签是否与 Pod 标签一致 |
| 安全 | STRICT 与未注入 Pod 是否冲突 |
| 出站 | REGISTRY_ONLY 是否遗漏 ServiceEntry |
| 观测 | Telemetry 是否覆盖关键命名空间 |

**步骤 2：一键体检**

```bash
istioctl analyze --all-namespaces
istioctl proxy-status
```

## 22.4 项目总结

**优点与缺点**

| 维度 | 体系化复盘 | 单点学章节 |
|:---|:---|:---|
| 排障 | 有地图 | 易碎片化 |

**适用场景**：培训；事故复盘；架构评审。

**不适用场景**：无（复盘为元能力）。

**典型故障**：升级默认行为变化；只改配置不看指标。

**思考题（参考答案见第23章或附录）**

1. 写出你从「用户报 503」到「定位到具体 CRD」的三步最小路径。
2. DestinationRule 与 VirtualService 同时参与时，若 subset 不存在，通常先查哪一类资源？

**推广与协作**：入职培训必修；变更单附 analyze 截图；值周工程师维护清单版本。
