# 第16章 熔断与降级：韧性设计的核心

## 16.1 项目背景

**业务场景（拟真）：下游支付抖动，全链路排队到超时**

下游 **连续 5xx** 或 **慢响应** 时，若上游仍无限重试、无限建连，会把线程与连接池占满，引发 **级联故障**。**熔断**（outlier detection：驱逐不健康端点）与 **连接池限制**（防止打爆下游）在 DestinationRule 中声明；**降级**多在业务层返回缓存/默认值——网格负责「快速失败 + 少连接」，业务负责「给用户看什么」。

**痛点放大**

- **只加重试不加池**：重试风暴放大下游故障（与第8章联动）。
- **误驱逐**：阈值过严把健康实例踢光 → `UH`。
- **无监控**：看不到 ejection 与池耗尽。

```mermaid
flowchart LR
  DR[DestinationRule\n连接池+outlier] --> E[Envoy]
  E --> X[快速失败/换端点]
```

## 16.2 项目设计：小胖、小白与大师的韧性防线

**第一轮**

> **小胖**：熔断不就是报错吗？用户体验还不是一样差。
>
> **小白**：outlierDetection 和「业务熔断」啥区别？`maxEjectionPercent` 设大了会怎样？
>
> **大师**：网格侧 **outlier** 是 **端点级**：连续失败则暂时不往该 Pod 发新连接。业务熔断常是 **功能开关**（返回兜底）。`maxEjectionPercent` 过大可能 **可用容量骤降**。
>
> **大师 · 技术映射**：**connectionPool ↔ 背压；outlierDetection ↔ eject unhealthy endpoints。**

**第二轮**

> **大师**：降级策略要 **可观测**：熔断触发时告警与 Runbook 对齐，否则只是「静默失败」。

## 16.3 项目实战：全链路韧性配置

**步骤 1：DestinationRule 连接池 + 异常检测**

```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: resilience-config
spec:
  host: downstream-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        http1MaxPendingRequests: 50
        http2MaxRequests: 1000
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
    loadBalancer:
      simple: LEAST_REQUEST
```

**测试验证**：对单 Pod 注入 503；`istioctl proxy-config endpoint` 观察端点被移除与恢复。

## 16.4 项目总结

**优点与缺点**

| 维度 | DR 熔断+连接池 | 无网格策略 |
|:---|:---|:---|
| 雪崩防护 | 有 | 弱 |
| 调参 | 需压测与混沌 | 无 |

**适用场景**：高可用；不可靠依赖；峰值保护。

**不适用场景**：极低延迟路径且不能容忍任何排队/驱逐抖动（需专项评估）。

**典型故障**：误驱逐；连接池过小 `UO`；与重试叠加放大负载。

**思考题（参考答案见第17章或附录）**

1. `outlierDetection` 驱逐的端点会在什么条件下重新被接受？
2. 同步 HTTP 链路的「快速失败」与消息队列削峰，适用边界有何不同？

**推广与协作**：SRE 定阈值与告警；开发实现业务降级；测试混沌注入验证。
