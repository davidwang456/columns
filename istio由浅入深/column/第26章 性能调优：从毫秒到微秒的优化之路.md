# 第26章 性能调优：从毫秒到微秒的优化之路

## 26.1 项目背景

**业务场景（拟真）：上 mesh 后 P99 从 5ms 变 15ms**

Sidecar 增加一跳、TLS 握手与过滤器链带来 **CPU/延迟**；高并发下还有 **连接池、线程数（concurrency）**、**遥测采样** 等叠加。优化目标不是「玄学调参」，而是 **基线压测 → 找瓶颈（Envoy vs 应用 vs 网络）→ 渐进变更 → 回归监控**。

**痛点放大**

- **无基线**：无法区分「mesh 税」与业务退化。
- **过度优化**：调低 telemetry 导致排障盲飞。

```mermaid
flowchart LR
  B[压测基线] --> P[Profile]
  P --> T[调 concurrency/池/采样]
  T --> R[回归]
```

## 26.2 项目设计：小胖、小白与大师的「压榨微秒」

**第一轮**

> **小胖**：15ms 用户也感觉不到吧，折腾啥？
>
> **小白**：先调 `concurrency` 还是先调连接池？eBPF 啥时候上？
>
> **大师**：先 **profile**：Sidecar CPU、upstream_time、应用自身。`concurrency` 影响 worker 与上下文切换；连接池解决建连开销；eBPF/Ambient 是更大架构决策，不是第一旋钮。
>
> **大师 · 技术映射**：**数据面延迟 = f(过滤器链, TLS, 池, 遥测, 网络)。**

## 26.3 项目实战：系统化性能优化

**步骤 1：措施表（按环境验证）**

| 优化维度 | 具体措施 | 预期收益 |
|:---|:---|:---|
| CPU亲和性 | `proxy.istio.io/config: '{"concurrency": 2}'` | 减少上下文切换 |
| 连接池调优 | 增大maxConnections、启用keepalive | 减少连接建立开销 |
| TLS会话复用 | 启用sessionTickets、调整sessionTimeout | 减少TLS握手 |
| eBPF加速 | Cilium集成、sockops启用 | 绕过iptables，降低延迟 |
| 专用硬件 | SmartNIC、DPU卸载 | 极致性能场景 |

**步骤 2**：`fortio`/`hey` 压测，记录 P50/P99 与 Sidecar CPU；变更单条回滚。

## 26.4 项目总结

**优点与缺点**

| 维度 | 系统调优 | 不调参 |
|:---|:---|:---|
| 成本/体验 | 可能显著改善 | 维持默认税 |

**适用场景**：延迟敏感；高吞吐；FinOps。

**不适用场景**：未建立监控与基线（禁止乱调）。

**典型故障**：并发过低排队；池过小建连风暴；采样过低无观测。

**思考题（参考答案见第27章或附录）**

1. 为何说「先 profile 再调参」比「直接改 concurrency」更安全？
2. 降低访问日志采样率对排障有何副作用？

**推广与协作**：性能组主导；SRE 守门；变更分批发布。
