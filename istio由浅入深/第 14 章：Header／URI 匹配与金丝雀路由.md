# 第 14 章：Header/URI 匹配与金丝雀路由

**受众提示**：开发定义 Header 契约；运维配置灰度名单；测试覆盖匹配边界。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

## 1. 项目背景

按**权重**金丝雀是基线；按 **Header、URI、方法** 分流能支持「**内部用户先新版本**」「**移动端与 Web 分流**」。`VirtualService` 的 `http.match` 支持多种条件，**顺序**决定优先级。本章与第 15 章（运维剧本）衔接：技术对象在此，流程在彼。

## 2. 项目设计：大师与小白的对话

**小白**：`headers` 匹配大小写敏感吗？

**大师**：依配置与 Envoy 行为，**不要依赖模糊约定**；统一用**规范头名**并在文档写明。

**小白**：多个 match 谁赢？

**大师**：**自上而下第一个命中**；把**最具体**的放前面。

**小白**：能按 JWT claim 路由吗？

**大师**：通常先把 claim **传播为 Header**（网关或应用），再匹配；纯网格层能力随版本演进，要以官方为准。

**要点清单**

1. **显式**列出灰度 Header 与取值规范。
2. 与 **API 网关** 协作避免重复剥离/注入。
3. 记录**回滚**路径：去掉 match 或改权重。

## 3. 项目实战

### 3.1 按 Header 走 v2，其余 v1

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews-canary-header
spec:
  hosts:
  - reviews
  http:
  - match:
    - headers:
        end-user:
          exact: jason
    route:
    - destination:
        host: reviews
        subset: v2
  - route:
    - destination:
        host: reviews
        subset: v1
```

Bookinfo 页面登录用户 **jason** 可走 v2（按官方示例）。

### 3.2 URI 前缀匹配示例

```yaml
  http:
  - match:
    - uri:
        prefix: /api/v2
    route:
    - destination:
        host: backend
        subset: v2
  - route:
    - destination:
        host: backend
        subset: v1
```

### 3.3 验证

```bash
curl -H "end-user: jason" http://.../productpage
istioctl proxy-config route deploy/productpage -n default
```

## 4. 项目总结

**优点**

- **精细化**灰度与 A/B；
- 与 **用户识别** 体系对接灵活。

**缺点**

- 规则复杂 → **运维成本**与**审计**难度上升。
- Header **伪造**风险需网关层鉴权（第 27 章）。

**适用场景**

- 内部 dogfood、**大客**体验版本、**API 版本**共存。

**注意事项**

- **CDN/缓存**可能剥离 Header，要在全链路验证。
- **大小**与 **重复 Header** 行为要测。

**常见踩坑**

1. **现象**：match 永远不命中。**原因**：实际 Header 名不一致（如 `X-User` vs `x-user`）。**处理**：统一规范与大小写策略。
2. **现象**：部分客户端异常。**原因**：老版本不发送该 Header。**处理**：默认路由兜底。
3. **现象**：与 **Canary 权重** 混用混乱。**原因**：团队未定义优先级。**处理**：文档化「Header 优先于权重」或反之。

---

### 4.1 再谈一个场景：CDN 与缓存

若静态资源命中 CDN，而动态 API 走网格，可能出现「版本不一致」错觉。A/B 与金丝雀要联合前端发布策略，明确**缓存键**与**版本号**。

### 4.2 与安全

Header 路由若依赖客户端可伪造字段，必须配合网关鉴权或 mTLS 内网访问模型。

### 4.3 规则评审：「为什么需要第 N 条」

每增加一条 `match`，都要求作者回答：若删除它，**哪类用户**会受影响？若答不上来，说明规则可能来自**临时救火**，应合并或下沉到应用。长期堆积的匹配规则会让**新人**与**oncall**陷入恐惧：不敢改、只能加。

### 4.4 与 API 版本化

URI 前缀路由常与 **/v1 /v2** 共存；要确保**弃用策略**（deprecation header、文档、客户端升级窗口）同步，否则网格层「能路由」也救不了**契约撕裂**。

---

### 4.5 本章小结：把知识变成「可执行」

推广 Istio 时，最容易失败的不是「不会配」，而是「配了没人敢动」。建议每章学完后，在团队看板增加一张卡片：**谁能在生产执行与本章相关的变更**、**需要哪些审批**、**回滚命令是什么**。技术文档若不能映射到流程，就仍是幻灯片。

### 常见误解澄清（防抬杠小抄）

- 「上了网格就不用关心网络」：错误；网络/CNI/防火墙仍是底座。
- 「网格能自动让系统高可用」：错误；高可用需要架构、容量、数据与演练。
- 「配置在 YAML 里就是真相」：不完整；真相是 **集群实际生效配置**（见 istioctl proxy-config）。
