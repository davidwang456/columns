# 第 16 章：A/B 与基于用户的粘性路由

**受众提示**：产品/开发定义实验；运维配置路由；测试验证样本偏差。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

**A/B 测试**关注**统计效力**：用户应**稳定**落在同一版本，否则体验飘忽、指标失真。网格层可通过 **Header/cookie** 识别用户并路由；更完整的实验平台常在外部（Feature Flag）。本章讲清：**网格能做什么**、**不能做什么**（实验分析、分层抽样）。

---

## 2. 项目设计：大师与小白的对话

**小白**：粘性路由是不是会话保持（session affinity）？

**大师**：相关但不等同。**会话保持**多指 L4/L7 到同一 Pod；这里强调**同一用户多次请求**命中同一**版本**。

**小白**：用户 ID 从哪来？

**大师**：登录态、网关注入 Header、或**哈希**匿名 ID——注意 **GDPR**。

**小白**：比例 50/50 怎么保证？

**大师**：可用**确定性哈希**（对用户 ID）或外部实验服务；网格层常用 **match + 权重** 组合。

**要点清单**

1. **隐私**：最小化采集；**脱敏**。
2. **偏差**：运营活动、地域可能影响 A/B，需要**分层**。
3. **回滚**：实验结束**清理**路由规则。

## 3. 项目实战

### 3.1 基于 Header 的用户路由（简化）

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: ab-demo
spec:
  hosts:
  - api
  http:
  - match:
    - headers:
        x-ab-group:
          exact: B
    route:
    - destination:
        host: api
        subset: version-b
  - route:
    - destination:
        host: api
        subset: version-a
```

网关或 BFF 负责根据用户计算 `x-ab-group`。

### 3.2 与 DestinationRule 子集配合

`version-a` / `version-b` 对应不同 labels。

### 3.3 验证

用两个不同 Header 的客户端压测，检查**指标隔离**。

## 4. 项目总结

**优点**

- 低侵入实现**用户级**路由；
- 与网格观测打通。

**缺点**

- **不是**完整实验平台；统计分析需外部工具。
- Header **伪造**需防护。

**适用场景**

- 功能开关试点、**UX** 改版、**API 行为**对比。

**注意事项**

- **爬虫**与**未登录**用户路径单独处理。
- **长事务**跨版本一致性。

**常见踩坑**

1. **现象**：指标交叉污染。**原因**：用户分组不稳定。**处理**：稳定哈希或登录态。
2. **现象**：法务风险。**原因**：未告知用户实验。**处理**：合规流程。
3. **现象**：CDN 缓存导致版本错乱。**处理**：缓存键含版本或绕开缓存。

---

### 4.1 再谈一个场景：实验伦理与通知

对用户可见体验变化的实验，涉及产品告知与法务合规。技术团队要把「路由实验」纳入**实验治理**，而不是仅当作配置变更。

### 4.2 指标污染

若实验组与对照组在用户结构上差异大（例如新用户偏多），指标对比会失真；需要分层或外部实验平台支持。

---

### 4.3 分桶算法与「可解释性」

当产品问「为什么 A 组转化率低」时，技术团队要能说清：**分桶依据**、**样本量**、**置信区间**粗算。网格层只保证路由，不负责统计显著性；需要 **BI** 或 **数据科学**同事参与实验设计。

### 4.4 与缓存一致性

实验版本若改变 HTML/JS 资源 hash，CDN 缓存可能导致**用户看到混合版本**。要与前端发布流程对齐，必要时**版本号**写入 URL。

---

### 4.5 本章小结：把知识变成「可执行」

推广 Istio 时，最容易失败的不是「不会配」，而是「配了没人敢动」。建议每章学完后，在团队看板增加一张卡片：**谁能在生产执行与本章相关的变更**、**需要哪些审批**、**回滚命令是什么**。技术文档若不能映射到流程，就仍是幻灯片。

### 常见误解澄清（防抬杠小抄）

- 「上了网格就不用关心网络」：错误；网络/CNI/防火墙仍是底座。
- 「网格能自动让系统高可用」：错误；高可用需要架构、容量、数据与演练。
- 「配置在 YAML 里就是真相」：不完整；真相是 **集群实际生效配置**（见 istioctl proxy-config）。
