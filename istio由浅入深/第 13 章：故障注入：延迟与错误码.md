# 第 13 章：故障注入：延迟与错误码

**受众提示**：测试主导场景设计；开发确认接口契约；运维观察告警与自愈。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

`VirtualService` 支持 **fault** 注入：**延迟**与**中止（abort）**。用于混沌演练、验证超时/重试、以及培训排障。与生产事故不同，注入是**可控、可回滚**的。推广时强调：**仅非生产或隔离命名空间**默认开启；注入范围要**最小化**（特定 subset、特定 header）。

---

## 2. 项目设计：大师与小白的对话

**小白**：这和杀 Pod 有什么区别？

**大师**：杀 Pod 测的是**Kubernetes 恢复**；fault 测的是**应用与网格策略**在异常响应下的行为。

**小白**：延迟 7s 为啥总超时？

**大师**：可能你 **VS timeout** 更短——这是**故意**用来教「预算」概念的。

**小白**：能只给测试用户注入吗？

**大师**：用 **match**（Header、cookie）缩小范围；或单独**测试集群**。

**要点清单**

1. **abort** 前后端要一致处理 HTTP/gRPC 码。
2. 与 **监控告警** 联动，避免演练变「狼来了」。
3. 演练结束**必须**删除 fault 配置。

## 3. 项目实战

### 3.1 对 ratings 注入 503（示例）

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: ratings-fault
spec:
  hosts:
  - ratings
  http:
  - fault:
      abort:
        percentage:
          value: 50
        httpStatus: 503
    route:
    - destination:
        host: ratings
        subset: v1
```

### 3.2 延迟示例

```yaml
  http:
  - fault:
      delay:
        percentage:
          value: 100
        fixedDelay: 5s
    route:
    - destination:
        host: reviews
        subset: v2
```

### 3.3 验证

访问 Bookinfo，观察页面错误与 Prometheus 指标；用 `istioctl proxy-config route` 核对。

```bash
kubectl apply -f ratings-fault.yaml
```

## 4. 项目总结

**优点**

- **低成本**演练依赖故障；
- 可**精确**到路由级。

**缺点**

- 误配影响真实用户；
- 与 **缓存**、**重试** 组合可能放大困惑。

**适用场景**

- 预发混沌、**oncall 培训**、重试策略验收。

**注意事项**

- **变更窗口**与**公告**；
- gRPC **status** 与 HTTP **abort** 映射要测。

**常见踩坑**

1. **现象**：注入不生效。**原因**：更高优先级路由或 **subset** 不匹配。**处理**：检查 VS 顺序。
2. **现象**：全站挂。**原因**：percentage 过大且命中主路径。**处理**：缩小范围。
3. **现象**：监控无报警。**原因**：告警未覆盖新错误码。**处理**：更新告警规则（第 25 章）。

---

### 4.1 再谈一个场景：演练与真实事故混淆

若故障注入未登记窗口，可能与真实事故难以区分。建议：**演练工单号**写入 Header 或 **Annotation**，并在告警规则中临时抑制已知演练标签（谨慎使用，避免误抑制真实故障）。

### 4.2 gRPC 与 HTTP 差异

abort 的语义在 gRPC 上映射为 **status code**；演练前确认客户端是否正确处理 **DEADLINE_EXCEEDED** 等。

## 附：自测与练习

### 自测题

### 动手作业

**延伸阅读**：Istio *Fault Injection*；混沌工程原则。

---

### 4.3 与告警的「演练标签」

若告警系统支持 **annotation**，可在演练窗口给相关告警加 `drill=true` 标签，复盘时过滤误报。注意不要长期依赖「全局静默」，避免掩盖真实事故。

### 4.4 客户端行为清单

浏览器、移动端、服务端 SDK 对超时与重试的默认行为不同；fault 演练要覆盖**主要客户端**。特别是移动弱网场景，**延迟**可能被本地重试放大，需要与产品、客户端负责人一起评估体验。

### 常见误解澄清（防抬杠小抄）

- 「上了网格就不用关心网络」：错误；网络/CNI/防火墙仍是底座。
- 「网格能自动让系统高可用」：错误；高可用需要架构、容量、数据与演练。
- 「配置在 YAML 里就是真相」：不完整；真相是 **集群实际生效配置**（见 istioctl proxy-config）。
