# 第 28 章：AuthorizationPolicy：命名空间到工作负载

**受众提示**：安全架构师设计规则；开发提供操作语义；测试做矩阵与渗透补充。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

**AuthorizationPolicy** 实现 **L7 RBAC**：按 **source**（命名空间、服务账户、JWT claims）与 **operation**（方法、路径）允许/拒绝。与 **PeerAuthentication**、**RequestAuthentication** 形成 **零信任闭环**。规则错误会导致**静默拒绝**或**过度放行**。

---

## 2. 项目设计：大师与小白的对话

**小白**：ALLOW 和 DENY 谁先？

**大师**：Istio 有**固定评估顺序**（以官方为准）：通常 **DENY** 优先于 **ALLOW**；**显式**写清，别靠记忆。

**小白**：`CUSTOM` 干啥？

**大师**：对接 **外部授权**（如 OPA）；复杂企业常用。

**小白**：能和 K8s RBAC 混吗？

**大师**：**不同层**：K8s RBAC 管 **API Server**；AuthorizationPolicy 管 **服务流量**。

**要点清单**

1. **默认拒绝**还是**默认允许**策略要全公司统一。
2. **渐进**：先 **audit** 模式（若支持）或**预发**全量验证。
3. **最小权限**原则。

---

## 3. 项目实战

### 3.1 仅允许同命名空间

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: ns-isolation
  namespace: foo
spec:
  action: ALLOW
  rules:
  - from:
    - source:
        namespaces: ["foo"]
```

### 3.2 路径级（示例）

```yaml
spec:
  action: ALLOW
  rules:
  - to:
    - operation:
        methods: ["GET"]
        paths: ["/public/*"]
```

### 3.3 验证

从 **其他 namespace** `curl` **拒绝**；从 **foo** **允许**。

## 4. 项目总结

**优点**

- **细粒度**、**可审计**；
- 与 **身份**模型一致。

**缺点**

- 规则爆炸 → **难维护**；
- **调试**需要 **RBAC** 思维。

**适用场景**

- **多租户**、**合规分区**、**内部 API**。

**注意事项**

- **健康检查**路径放行；
- **mesh 外**客户端识别。

**常见踩坑**

1. **现象**：全 403。**原因**：**默认拒绝**过严。**处理**：分层放开。
2. **现象**：以为安全其实放行。**原因**：**宽规则**在后。**处理：**规则评审工具。
3. **现象**：与 **NetworkPolicy** 混淆。**处理**：**两层**文档说明。

---

### 4.1 再谈一个场景：健康检查与探测

readiness/liveness 探针路径若被 AuthorizationPolicy 误伤，会造成**滚动发布雪崩**。规则里必须**显式放行**探针来源或路径。

### 4.2 与 CUSTOM 授权

引入 OPA 等外部系统时，要把**超时**与**失败策略**（fail-open vs fail-closed）写成明文。

### 常见误解澄清（防抬杠小抄）

- 「上了网格就不用关心网络」：错误；网络/CNI/防火墙仍是底座。
- 「网格能自动让系统高可用」：错误；高可用需要架构、容量、数据与演练。
- 「配置在 YAML 里就是真相」：不完整；真相是 **集群实际生效配置**（见 istioctl proxy-config）。
