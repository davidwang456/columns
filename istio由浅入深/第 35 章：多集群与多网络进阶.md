## 2. 项目设计：大师与小白的对话

**小白**：我们三个集群，要不要「一个大 istiod」？

**大师**：**单控制面**运维集中，但**爆炸半径**大；**多控制面**复杂但**隔离**好。

**小白**：DNS 怎么设计？

**工程师**：**CoreDNS** 插件、**外部** DNS、或 **.global** 虚拟服务（视方案）。

**小白**：出事了先查谁？

**大师**：**连通性** → **证书** → **服务发现** → **路由**。

**要点清单**

1. **文档化**拓扑与**owner**；
2. **混沌**定期演练（第 13、25 章）；
3. **成本**核算（网关、带宽、人力）。

## 3. 项目实战

### 3.1 决策树（文字版）

```
是否必须跨集群调用？
 否 → 保持单集群网格
 是 → Pod IP 是否直连？
   是 → 同网络多集群简化路径
   否 → 多网络 + east-west + mTLS
```

### 3.2 健康检查脚本思路

定期从 **集群 A** `curl` **集群 B** 的 **服务入口**；**失败**告警到 **平台**。

### 3.3 与 GitOps

每个集群 **IstioOperator** 与 **信任 bundle** 同步 **Git** 管理。

## 4. 项目总结

**优点**

- **全局**策略与**容灾**；
- **统一**身份。

**缺点**

- **最高**复杂度；
- **排障**需 **跨团队**。

**适用场景**

- **多区域**、**并购**整合、**混合云**。

**注意事项**

- **数据主权**与 **合规**；
- **费用**（跨区流量）。

**常见踩坑**

1. **现象**：证书链复杂导致 **间歇失败**。**原因**：**中间 CA** 不一致。**处理**：**统一** PKI。
2. **现象**：**DNS** 缓存。**原因：**TTL** 过长。**处理**：**切换**演练。
3. **现象**：**单点** east-west gateway。**处理**：**HA** 与 **限流**。

### 4.3 「先能解释失败，再追求成功」

多集群排障培训应从**典型失败**开始：证书错误、服务发现缺失、路由黑洞。成功路径反而太理想化。

### 4.4 与费用对账

跨区流量费用可能出现在**云账单**多个条目；要与网络团队一起**归因**到业务服务。

---

### 4.5 本章小结：把知识变成「可执行」

推广 Istio 时，最容易失败的不是「不会配」，而是「配了没人敢动」。建议每章学完后，在团队看板增加一张卡片：**谁能在生产执行与本章相关的变更**、**需要哪些审批**、**回滚命令是什么**。技术文档若不能映射到流程，就仍是幻灯片。

### 常见误解澄清（防抬杠小抄）

- 「上了网格就不用关心网络」：错误；网络/CNI/防火墙仍是底座。
- 「网格能自动让系统高可用」：错误；高可用需要架构、容量、数据与演练。
- 「配置在 YAML 里就是真相」：不完整；真相是 **集群实际生效配置**（见 istioctl proxy-config）。
