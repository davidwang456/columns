# 第 26 章：mTLS：PeerAuthentication 与渐进式开启

**受众提示**：安全与平台主导；开发理解应用侧 TLS 与网格 mTLS 关系；测试做连通性矩阵。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

**双向 TLS（mTLS）** 提供服务间强身份与加密。Istio 通过 **PeerAuthentication** 控制模式：**PERMISSIVE**（兼容明文）→ **STRICT**（强制 mTLS）。推广应采用**渐进式**：先 PERMISSIVE 观测，再按命名空间/工作负载收紧。

---

## 2. 项目设计：大师与小白的对话

**小白**：应用自己 HTTPS 了，还要 mTLS 吗？

**大师**：这是**两层**：应用 TLS 终端用户；**mTLS** 保护**服务到服务**。可并存，注意 **CPU**。

**小白**：PERMISSIVE 是不是不安全？

**大师**：是**过渡态**；长期应 **STRICT**，并配合 **AuthorizationPolicy**。

**小白**：老系统非 mesh 怎么办？

**大师**：**WorkloadEntry**、**网关**、或 **例外** ——要显式建模，不能假装看不见。

**要点清单**

1. **兼容性矩阵**先行。
2. **证书**轮转监控。
3. 与 **DestinationRule** TLS 设置一致（第 8 章）。

---

## 3. 项目实战

### 3.1 命名空间级 PERMISSIVE（示例）

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: foo
spec:
  mtls:
    mode: PERMISSIVE
```

### 3.2 收紧到 STRICT

```yaml
spec:
  mtls:
    mode: STRICT
```

### 3.3 验证

```bash
istioctl authn tls-check $POD.foo
kubectl logs -n istio-system -l app=istiod --tail=20
```

观察 **503** 是否因非 mesh 客户端。

## 4. 项目总结

**优点**

- **零信任**基础能力；
- **自动**证书生命周期（视实现）。

**缺点**

- **老客户端**改造压力；
- **调试**复杂度上升。

**适用场景**

- **合规**、**多租户**、**东西向**敏感数据。

**注意事项**

- **headless**、**Job**、**外部**调用需逐一验证。
- **性能**：握手成本（第 33 章）。

**常见踩坑**

1. **现象**：突然全 503。**原因**：**STRICT** 但一侧无 Sidecar。**处理**：回退 PERMISSIVE 或补注入。
2. **现象**：证书过期。**原因**：**istiod** 或 **中间 CA** 故障。**处理**：监控 cert **TTL**。
3. **现象**：双重 TLS。**原因**：应用与网格**重复**。**处理**：分层设计。

---

### 4.1 再谈一个场景：渐进式背后的组织成本

PERMISSIVE 不是「半安全」，而是给团队时间完成 Sidecar 注入与客户端改造。要设**截止日期**与**例外清单**，否则永远停在过渡态。

### 4.2 与存量系统

对无法注入的工作负载，用 **WorkloadEntry**、**网关**或**明确例外**记录，不要「假装在网格里」。

**延伸阅读**：Istio *Security*；第 27、28 章。

---

### 4.3 例外清单与到期

所有无法 STRICT 的工作负载登记：**原因**、**责任人**、**到期日**、**补救计划**。季度复盘，避免「永久例外」。

### 4.4 与客户端证书

若某些客户端使用**自定义证书**访问，需确认与 Istio **信任链**关系，避免「双栈证书」混乱。
