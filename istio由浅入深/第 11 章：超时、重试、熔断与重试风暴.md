# 第 11 章：超时、重试、熔断与重试风暴

**受众提示**：开发定义幂等与重试安全边界；运维设默认与告警；测试构造超时与下游故障用例。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

韧性参数散落在 `VirtualService`（超时、重试）与 `DestinationRule`（连接池、异常检测）中。推广时最危险的是**重试风暴**：上游重试 × 下游重试 × 多副本，放大流量把脆弱服务打挂。本章用**可复现实验**建立直觉，并给出**团队规范**（如只允许 idempotent 方法重试）。

---

## 2. 项目设计：大师与小白的对话

**小白**：默认重试不是越多越好吗？

**大师**：**错**。POST 非幂等重试可能**重复扣款**。GET 也未必安全，要看业务。

**小白**：超时设多少？

**大师**：从 **SLO** 反推：端到端预算分给每一跳；没有 SLO 就先从**压测 P99** 起步。

**小白**：熔断触发后谁负责「半开」恢复？

**大师**：Envoy **outlierDetection** 有驱逐与恢复节奏；业务层仍要有**降级**。

**小白**：Istio 和 Hystrix 同时配行吗？

**大师**：**容易打架**。尽量**单点治理**，否则计数与超时重复。

**要点清单**

1. **重试**必须配 **超时** 与 **重试预算**（retryOn 谨慎）。
2. 与 **并发**、**队列** 联动评估（第 33 章）。
3. 变更后盯 **5xx、延迟、下游 QPS** 三角。

## 3. 项目实战

### 3.1 VirtualService：超时与有限重试（示例）

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: reviews-timeout-retry
spec:
  hosts:
  - reviews
  http:
  - route:
    - destination:
        host: reviews
        subset: v2
    timeout: 2s
    retries:
      attempts: 2
      perTryTimeout: 1s
      retryOn: connect-failure,refused-stream,503
```

### 3.2 DestinationRule：异常检测（接第 8 章）

确保 `outlierDetection` 与重试策略**一起评审**。

### 3.3 故障注入配合（预告第 13 章）

对下游注入延迟，观察重试是否放大延迟与负载。

```bash
kubectl apply -f vs-reviews-timeout-retry.yaml
istioctl proxy-config route deploy/productpage -n default
```

## 4. 项目总结

**优点**

- **平台级**统一超时/重试，减少各语言库分裂。
- 与指标联动，可**量化**重试率。

**缺点**

- 配置错误可导致**雪崩**；
- 与**应用线程池**超时需对齐。

**适用场景**

- 读多写少、可幂等接口；**同步链路**有明确 SLA。

**注意事项**

- **POST/PUT** 默认不重试或仅限特定错误码。
- **gRPC** 与 HTTP/2 的 retry 语义要查版本说明。

**常见踩坑**

1. **现象**：延迟飙升。**原因**：`perTryTimeout` > `timeout` 或重试次数过多。**处理**：修正预算。
2. **现象**：下游 QPS 翻倍。**原因**：多服务同时重试叠加。**处理**：限制 attempts、使用**全局限流**（扩展）。
3. **现象**：熔断频繁抖动。**原因**：阈值过严或下游**冷启动**。**处理**：调参 + 预热。

---

### 4.1 再谈一个场景：重试风暴的量化

在故障演练中记录：**下游 QPS**、**上游重试倍数**、**p99 延迟**。若重试倍数明显高于 1，需要检查多服务是否同时重试、以及是否存在「超时大于重试窗口」等配置错误。

### 4.2 与业务幂等表

建议平台提供「幂等接口清单」模板，网格策略变更必须引用该清单作为附件。

---

### 4.3 超时预算：从用户 SLA 反推

把端到端 SLA 拆成「网关—服务 A—服务 B—外部 API」的**时间片**，任何一截拉长都要显式记录原因。网格层的 `timeout` 不应大于业务可接受的用户等待；`perTryTimeout` 与 `attempts` 的乘积关系要在白板上演算一遍，写进变更说明。对批处理与异步链路，超时策略应与**队列**一致，避免出现「HTTP 已超时但后台仍在跑」的双世界。

### 4.4 演练记录模板

一次完整演练至少包含：**注入类型**（延迟/错误）、**范围**（路由/百分比）、**开始/结束时间**、**观测截图**（RED）、**结论**（参数调整或回滚）。模板进入 Confluence，后续审计与复盘才有抓手。
