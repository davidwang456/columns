# 第 18 章：多集群流量：locality 与 failover

**受众提示**：运维与网络工程师主导 locality 标签与拓扑；开发了解调用是否跨地域；测试验证容灾切换。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

在**多集群/多可用区**部署同一服务时，希望**优先本地域**流量以降低延迟与出口成本，故障时再**故障转移**。`DestinationRule` 的 `trafficPolicy.loadBalancer.localityLbSetting` 与 **outlierDetection** 配合可表达这一意图。注意：**拓扑标签**（region/zone）需与 K8s 节点 labels 一致。

本章强调**概念与参数方向**；真实多集群需第 17 章前提。

---

## 2. 项目设计：大师与小白的对话

**小白**：有了 K8s，kube-scheduler 不是已经就近了吗？

**大师**：调度解决**Pod 放哪**；locality lb 解决**客户端侧**优先选哪组 endpoint。

**小白**：failover 会不会把流量冲到另一个**满负载**集群？

**大师**：需要 **容量规划** 与 **限流**；网格不是自动「扩容」。

**小白**：和 DNS 地理位置解析冲突吗？

**大师**：可能。要**统一入口策略**，避免两层打架。

2. **分阶段**启用 failover，观察 **RED**。

## 3. 项目实战

### 3.1 DestinationRule 片段（示例意图）

```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: reviews-locality
spec:
  host: reviews
  trafficPolicy:
    loadBalancer:
      localityLbSetting:
        enabled: true
        failover:
        - from: regionA
          to: regionB
```

> 具体字段以当前 Istio API 为准；启用前在**测试集群**验证。

### 3.2 验证思路

- 人为** cordon** 一个 zone 的 endpoints，观察流量是否迁移。
- 用 **metrics** 看跨区流量比例。

## 4. 项目总结

**优点**

- **降低跨区**流量成本与延迟；
- **容灾**路径可声明式管理。

**缺点**

- 标签与**实际网络**不一致时行为诡异；
- **脑裂**场景下需额外治理。

**适用场景**

- **多 AZ**、**双活**、**异地多活**初期。

**注意事项**

- **数据层**复制延迟与**读己之写**语义。
- **合规**：数据不得离境时，failover 可能**不可用**。

**常见踩坑**

1. **现象**：仍跨区。**原因**：locality 标签缺失。**处理**：规范化节点 label。
2. **现象**：切换风暴。**原因**：outlier 阈值过松。**处理**：调参 + 冷却。
3. **现象**：单点网关瓶颈。**原因**：east-west **容量**不足。**处理**：横向扩展。

---

### 4.1 再谈一个场景：脑裂与误切换

当网络分区时，locality failover 可能把流量切到**并非真正健康**的区域。需要结合**外部权威**（例如全局负载）与**人工介入**流程。

### 4.2 与数据一致性

跨区 failover 若读到**旧副本**，业务上可能比「503」更糟。读己之写、复制延迟要在应用层有策略。

---

### 4.3 观测：跨区域流量比

为 failover 建立专门面板：**同区流量占比**、**跨区 RTT**、**跨区字节数**。没有基线就无法判断「 failover 是否过度」。

### 4.4 与业务「一致性级别」

强一致系统往往**不能**随意切换读区；最终一致系统要定义**用户可接受的延迟窗口**。locality 策略必须服从业务一致性，而不是反过来。
