# 跨部门评审清单（Review Loop）

每章合并前至少经过 **开发、运维、测试各一人** 轮审（可同一人兼多角色，但需标明）。评审在 PR 或内部文档库评论中完成。

## 1. 开发评审（Dev）

- [ ] **架构与概念**：术语与 Istio/K8s 官方含义一致，无误导性简化。
- [ ] **与代码边界**：JWT、业务 Header、重试语义等是否说清「应用 vs 网格」职责。
- [ ] **可扩展性**：高级章是否指向合理延伸阅读（文档/源码入口），避免过时 API 名。

## 2. 运维评审（Ops）

- [ ] **命令可复现**：按 LAB-KIT 基线从零执行能否跑通；路径、命名空间、版本是否写明。
- [ ] **生产警觉**：`demo` profile、宽松 mTLS、全局 EnvoyFilter 等是否标注风险。
- [ ] **升级与回滚**：涉及安装的章节是否与修订版、金丝雀升级叙事一致。

## 3. 测试评审（QA）

- [ ] **验收点**：实战步骤是否有明确「预期现象」（HTTP 码、指标、拓扑变化）。
- [ ] **可测性**：金丝雀、故障注入、镜像流量等是否可被自动化或半自动化验证。
- [ ] **负面用例**：是否提到常见失败形态（如重试风暴、策略不生效）。

## 4. 通用（全员）

- [ ] **字数**：正文约 **3000～5000 字**（中文），四段式结构完整。
- [ ] **趣味性**：对话段落是否承担纠偏，而非单纯段子。
- [ ] **踩坑**：总结中含「现象 → 原因 → 处理」至少 2 条（技术章建议更多）。
