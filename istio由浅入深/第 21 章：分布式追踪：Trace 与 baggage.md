# 第 21 章：分布式追踪：Trace 与 baggage

**受众提示**：开发传递 trace 头；运维部署 Jaeger/Zipkin；测试用 trace 断言调用链。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

Istio 自动处理 **Envoy 生成的 span** 与常见 **trace header**（如 `x-b3-*`、`traceparent`）传播，但**应用层**仍需正确转发或生成上下文，否则链路断裂。`baggage` 可携带**业务上下文**（慎用，隐私与体积）。本章讲**端到端**打通与**采样**策略。

---

## 2. 项目设计：大师与小白的对话

**小白**：装了 Jaeger 为啥还是断的？

**大师**：可能 **应用** 没转发头，或 **采样率** 太低看不到。

**小白**：100% 采样行吗？

**大师**：**高流量**下存储与性能扛不住；生产常用 **头部采样**或 **尾部采样**（外部系统）。

**小白**：baggage 和 Header 啥区别？

**大师**：baggage 是 **W3C baggage** 概念，可传播键值；**不要**塞大对象。

**要点清单**

1. 统一 **propagation** 格式（B3 vs W3C）。
2. **PII** 禁止进 trace。
3. 与 **日志** 关联 trace id（第 22 章）。

## 4. 项目总结

**优点**

- **跨服务**定位延迟与错误根因；
- 与 **指标**互补（RED 告诉你「慢」，trace 告诉你「慢在哪」）。

**缺点**

- **成本**与 **隐私合规**；
- 异步消息、批处理链路需**额外**建模。

**适用场景**

- **复杂调用图**、**性能优化**、**oncall** 排障。

**注意事项**

- **时钟同步**；
- **批处理**与 **kafka** 消费的 trace 模型。

**常见踩坑**

1. **现象**：只看到一跳。**原因**：Sidecar 未注入或 **mTLS** 中断。**处理**：分层验证。
2. **现象**：trace 爆炸。**原因**：**采样**过高。**处理**：调采样与保留期。
3. **现象**：头被 **API 网关** 剥离。**处理**：网关配置放行。

---

### 4.1 再谈一个场景：异步与消息

Kafka/Rabbit 消费的 trace 需要额外 instrument；否则链路在队列处断裂。推广叙事要诚实：**网格不是全自动魔法**。

### 4.2 采样与成本

生产常用低采样；排障时要有 **动态提升采样** 或 **基于 trace id 的补采** 机制（依平台能力）。

### 动手作业

**延伸阅读**：OpenTelemetry；Istio *Distributed Tracing*。

---

### 4.3 采样策略的业务沟通

向业务经理解释「为什么不能 100% 采样」时，用**成本—覆盖**曲线比用术语更有效。给出两档：**日常**与**事故响应**（临时提高采样）的操作步骤。

### 4.4 跨系统 trace

消息队列、批处理、定时任务需要 **manual span** 或 **links**；要在团队规范里写明**责任服务**谁来埋点。
