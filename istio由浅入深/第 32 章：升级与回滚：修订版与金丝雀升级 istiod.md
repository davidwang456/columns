# 第 32 章：升级与回滚：修订版与金丝雀升级 istiod

**受众提示**：运维主导；开发感知 CRD 变更；测试跑回归套件。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

Istio **频繁发版**；升级 **istiod** 与 **数据面** 需 **revision** 策略：**并行**安装新版本控制面 → **迁移**命名空间/workload → **下线**旧版。回滚包括：**流量回切**与 **控制面**降级。

---

## 2. 项目设计：大师与小白的对话

**小白**：能原地覆盖升级吗？

**工程师**：**小版本**有时可以，但生产推荐 **canary control plane** 降低爆炸半径。

**小白**：CRD 升级会破坏吗？

**大师**：**大版本**可能有 **breaking**；要读 **upgrade notes** 与 **兼容性**。

**小白**：数据面先还是控制面先？

**大师**：以**官方升级文档**为准；一般 **先控制面**再 **滚动数据面**。

**要点清单**

1. **备份**：`kubectl get istiooperators -A -o yaml` 等；
2. **兼容性矩阵**：k8s / istio / addon；
3. **回滚演练**每季度一次。

## 3. 项目实战

### 3.2 金丝雀思路（概念）

安装 `istiod-canary`，将试点 namespace 标 `istio.io/rev=canary`，验证后全量迁移。


## 4. 项目总结

**优点**

- **风险可控**；
- **渐进**验证。

**缺点**

- **操作步骤**多；
- **临时**双控制面成本。

**适用场景**

- **生产**、**大集群**。

**注意事项**

- **Webhook** 指向；
- **Sidecar** 版本 **skew** 窗口尽量短。

**常见踩坑**

1. **现象**：新 Pod 未注入。**原因**：**label** 与 **revision** 不匹配。**处理**：对照注入规则。
2. **现象**：配置不生效。**原因**：连到**旧 istiod**。**处理**：`proxy-status`。
3. **现象**：升级后 **503**。**原因**：**CRD** 行为变化。**处理**：**预发**全量用例。

---

### 4.1 再谈一个场景：Webhook 故障

升级过程中 Mutating Webhook 异常会导致**新 Pod 无法创建**。要在 Runbook 中写明 **紧急恢复**步骤与 **影响面**评估。

### 4.2 与业务窗口

大版本升级应避开业务高峰；并与 **变更委员会**对齐。

- 「上了网格就不用关心网络」：错误；网络/CNI/防火墙仍是底座。
