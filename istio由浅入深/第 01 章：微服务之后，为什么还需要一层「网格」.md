# 第 1 章：微服务之后，为什么还需要一层「网格」

**受众提示**：本章适合全员首读；开发关注与业务库/SDK 的边界，运维关注平台职责划分，测试关注可测性与发布策略的前置概念。

**实验基线**：概念章无强制命令；后续实战请统一参考 [LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

团队在 Kubernetes 上拆分微服务后，普遍遇到三类矛盾：**流量治理与业务代码耦合**、**安全与可观测性重复建设**、**多语言栈下行为不一致**。传统做法是在每个服务里引入熔断、重试、限流、认证客户端库，或在「API 网关」上集中一部分能力。前者导致版本碎片化与「依赖地狱」，后者则常把东西向（服务间）流量能力挤到南北向网关上，治理粒度与身份模型容易错位。

服务网格（Service Mesh）把上述横切能力下沉到**基础设施层**：通过 Sidecar 代理（如 Envoy）拦截进出 Pod 的流量，由控制面（如 Istio）统一下发策略。类比城市交通：Kubernetes 负责「划街区与门牌」，服务网格负责「红绿灯、限速、监控摄像头与通行证」——路还是那条路，但规则与观测从司机脑子里挪到了市政系统。

本章不展开 Istio 安装细节，只回答「为什么值得推广」：为后续章节建立共同语言（数据面/控制面、策略声明式配置、与 GitOps 的天然契合）。

## 2. 项目设计：大师与小白的对话

**小白**：我们已经有 K8s Service 和 Ingress 了，再加一层 Istio，是不是「为了技术而技术」？

**大师**：Service 解决的是**发现与负载均衡**；Ingress 多数是**集群入口**。服务间调用里的 mTLS、按百分比金丝雀、按 Header 分流、出站访问外部 SaaS 的策略，这些不是 Ingress 的强项。网格把「服务间」也变成一等公民。

**小白**：那我们在 Spring Cloud / Dubbo 里已经做了熔断和重试呀。

**大师**：可以，但代价是**语言绑定与版本同步**。Java 升级、Go 新服务、Python 脚本各写一套，运维排查时要猜「是框架 bug 还是网络」。网格在 L7/L4 代理上统一行为，业务只关心业务语义。

**小白**：Sidecar 不是多占 CPU/内存吗？

**大师**：是成本，而且是**可计量、可优化的平台成本**。对比的是：每个团队自己造轮子 + 线上不一致的隐性成本。成熟团队会设资源上限、选择性启用特性，甚至评估 Ambient 等新模式——这些在后续章节谈。

**小白**：测试同学能得到什么？

**大师**：同一套流量策略可以在预发**镜像、注入故障、按用户粘性路由**，用可观测性断言「策略真的生效」。这比「只在集成环境手工点一点」更贴近生产。

**要点清单**

1. 网格解决的是**横切能力平台化**，不是替代 K8s。
2. 与「每个服务自带库」相比，网格强调**一致性与运维可见性**。
3. Sidecar 有开销，应与 SLO、容量规划一起评估。
4. 推广时避免「万能药」叙事：不适合的场景也要诚实写进 POC（见第 36 章）。

## 3. 项目实战

本章以**思辨 + 最小对照**为主，主「代码」是团队内部讨论用的检查清单与表格，便于落地到评审会。

### 3.1 现状盘点表（可复制到 Confluence）

| 维度 | 问题举例 | 是否适合用网格承接 |
|------|----------|----------------------|
| 东西向 mTLS | 各服务自签证书、过期无人知 | 高 |
| 金丝雀 | 仅靠 Deployment 副本比例，无法按用户/Header | 高 |
| 可观测性 | Prometheus 端口约定混乱 | 中高（需规范） |
| 强业务路由 | 与订单状态机强耦合的规则 | 低（应保留在应用） |

### 3.2 「反模式」一句话（用于架构评审）

- 把**业务幂等键**完全交给网格重试而不改应用——易出现重复下单。
- 在网格里堆**复杂 Lua/WASM** 替代缺失的领域服务——维护成本会反噬。

### 3.3 与后续章节衔接

安装与 Profile 见第 4 章；第一次摸流量对象从第 7 章 `VirtualService` 开始。若管理层要 ROI，可把「减少重复客户端升级人天」「故障定位时间」列为试点指标。

## 4. 项目总结

**优点**

- 将流量、安全、观测从应用中**解耦**，利于多语言与多团队。
- 策略**声明式**、可 Git 审计，契合平台工程方向。

**缺点**

- 引入控制面与数据面**运维面**；学习曲线客观存在。
- Sidecar（或新模式）带来**资源与延迟**成本，需基线测试。

**适用场景**

- 微服务数量多、迭代快，需要**统一治理与渐进发布**。
- 合规要求**服务间加密与访问控制**可审计。

**注意事项**

- 先**试点命名空间**，再全集群；与现有 Ingress/网关方案**边界画清**。
- 开发需理解：网格**增强**而非**替代**应用层幂等与业务规则。

**常见踩坑**

1. **现象**：试点「啥也没配却变慢了」。**原因**：默认 mTLS、遥测、访问日志全开。**处理**：按阶段启用，结合第 20、26 章做分级。
2. **现象**：团队以为「上了网格就不用改代码」。**原因**：超时、重试语义仍要与业务一致。**处理**：联合评审重试策略（第 11 章）。
3. **现象**：测试环境数据与生产策略不一致。**原因**：配置漂移。**处理**：GitOps + `istioctl analyze` 进 CI（见第 24、31 章）。

---

### 4.1 再谈一个场景：支付与幂等

某团队在支付回调链路上启用了网格重试，结果在下游短暂 503 时出现了「重复通知」。根因并不是 Istio「错了」，而是**业务接口在语义上并非幂等**，而网格按可配置策略进行了重试。推广 Istio 时，这类故事比口号更有说服力：平台提供能力，**应用与架构要共同定义边界**。建议在评审模板中增加一行：「该路由是否允许重试？若允许，幂等键在哪里？」

### 4.2 跨角色协作清单（简版）

- **开发**：列出对外部系统调用的超时、重试期望；标注非幂等接口。
- **运维**：给出集群与 Istio 版本、Profile、注入策略；提供变更窗口。
- **测试**：把「策略生效」写成可执行用例（HTTP 断言 + 指标/日志断言）。
