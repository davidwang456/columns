# 第 30 章：安全运维：策略变更与审计

**受众提示**：安全与合规主导审计；运维管变更窗口；全员理解责任分界。

**实验基线**：[LAB-KIT.md](LAB-KIT.md)。

---

## 1. 项目背景

网格安全策略（**PeerAuthentication**、**AuthorizationPolicy**、**RequestAuthentication**）应纳入 **变更管理**：**评审**、**双人复核**、**回滚包**、**审计日志**。推广不仅是技术，更是 **流程**。本章给**可操作**清单，与第 36 章治理衔接。

---

## 2. 项目设计：大师与小白的对话

**小白**：Git 里 YAML 算不算审计？

**大师**：算**一部分**；还要 **K8s audit log**、**工单**、**审批**链。

**小白**：谁有权改生产 AuthorizationPolicy？

**工程师**：**最小**权限 RBAC；敏感规则要 **security** 签字。

**小白**：演练算变更吗？

**大师**：算；**故障注入**（第 13 章）也要**登记**，避免与真实事故混淆。

**要点清单**

1. **变更模板**：动机、影响面、回滚。
2. **定期**访问评审（**quarterly**）。
3. **例外**登记与到期时间。

---

## 3. 项目实战

### 3.1 变更单模板（粘贴用）

```
标题：[Istio] foo 命名空间 STRICT mTLS
动机：合规要求
影响：东西向调用方列表（附件）
验证：预发矩阵 + istioctl analyze
回滚：PeerAuthentication PERMISSIVE + 工单号
审批：@security @platform
```

### 3.2 审计命令

```bash
kubectl get authorizationpolicy -A -o yaml > snapshot-$(date +%F).yaml
```

### 3.3 与 CI 集成

`istioctl analyze` **门禁**；**OPA/Gatekeeper** 约束 **禁止** `hosts: "*"` 在生产网关。

## 4. 项目总结

**优点**

- **可追溯**、**可问责**；
- 降低 **「某人 kubectl 了一下」** 风险。

**缺点**

- **流程**摩擦；
- 小团队可能 **过重**，需裁剪。

**适用场景**

- **金融**、**医疗**、**强合规**行业。

**注意事项**

- **秘密**不进 Git；
- **break-glass** 账号管控。

**常见踩坑**

1. **现象**：审计不全。**原因**：**直连**集群绕过 Git。**处理**：**禁止**本地 apply 生产。
2. **现象**：回滚失败。**原因**：**数据**已变。**处理**：**业务**补偿预案。
3. **现象**：多环境漂移。**原因**：**复制粘贴**。**处理**：**Kustomize/Helm** 单一来源。

---

### 4.1 再谈一个场景：紧急放行

break-glass 账号若可绕过审计，会把合规努力归零。要有**双人授权**、**自动过期**与**事后审计**。

### 4.2 与供应链

Istio 相关镜像与 Helm chart 的签名、来源校验应纳入**供应链安全**流程。

## 附：自测与练习

### 动手作业

为你们团队写一条 AuthorizationPolicy 变更模板（动机/影响/验证/回滚），并发给安全同事征求意见。

**延伸阅读**：Kubernetes *Auditing*；企业变更管理实践。

---

### 4.3 证据链：从工单到 Git 到集群

一次合规审计要能串起：**变更工单** → **PR** → **合并记录** → **集群 apply 记录**（或 GitOps sync）。缺任何一环都要补流程，而不是补「说明材料」。

### 4.4 与供应链

Istio 相关镜像的 **digest pin**、**签名验证**纳入安全基线。

---

### 4.5 本章小结：把知识变成「可执行」

推广 Istio 时，最容易失败的不是「不会配」，而是「配了没人敢动」。建议每章学完后，在团队看板增加一张卡片：**谁能在生产执行与本章相关的变更**、**需要哪些审批**、**回滚命令是什么**。技术文档若不能映射到流程，就仍是幻灯片。

### 写给不同角色的一句话

- **开发**：把「业务语义」与「网格语义」对齐，避免把幂等/鉴权全推给平台。
- **运维**：把变更纳入窗口与 GitOps，任何手工 kubectl 都要留痕。
- **测试**：把策略生效写成可重复断言，而不是一次性手工点点点。

### 常见误解澄清（防抬杠小抄）

- 「上了网格就不用关心网络」：错误；网络/CNI/防火墙仍是底座。
- 「网格能自动让系统高可用」：错误；高可用需要架构、容量、数据与演练。
- 「配置在 YAML 里就是真相」：不完整；真相是 **集群实际生效配置**（见 istioctl proxy-config）。
