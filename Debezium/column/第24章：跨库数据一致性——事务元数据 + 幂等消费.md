# 第24章：跨库数据一致性——事务元数据 + 幂等消费

## 1. 项目背景

某金融支付平台的分布式核心交易链路涉及 3 个 MySQL 数据库：`order_db`（订单写入）、`inventory_db`（库存扣减）、`payment_db`（支付记录）。用户下单时，系统以 SAGA 模式执行分布式事务——先在 order_db INSERT 订单，再在 inventory_db UPDATE 库存，最后在 payment_db INSERT 支付记录。

某天 CDC 管道网络中断 10 分钟，导致一个噩梦般的数据不一致：order_db 的订单 INSERT 事件已被下游消费（订单已同步到数据仓库），但 inventory_db 的库存 UPDATE 事件因网络中断而延迟，下游收到的顺序变成"先有订单，再有库存扣减"——但中间隔了 10 分钟。数据仓库中出现了 10 分钟的"挂账"状态：订单存在但库存未扣减，风控系统基于这份数据做出了错误判断。

跨库数据一致性是 CDC 最难的问题之一。Debezium 不是分布式事务协调器，但可以通过 **GTID 全局时钟 + 事务元数据原子标记 + 幂等消费去重** 的组合拳，在下游消费端构建最终一致性保障。

### 痛点放大

| 问题类型 | 具体表现 | 根本原因 |
|---------|---------|---------|
| 跨库时序错乱 | A 库的 UPDATE 在 B 库的 UPDATE 之前到达，但实际相反 | 无全局时钟排序 |
| 部分消费 | 分布式事务的 3 个操作中只消费了 2 个 | 无事务边界标记 |
| 重复消费 | 消费者崩溃恢复后事件被重复处理 | 无幂等去重机制 |
| 核对困难 | 不知道源库和目标库的数据是否一致 | 无自动化校验 |

## 2. 项目设计——三人对话

**（周五下午，运维群突然弹出一条报警）**

**小胖**："大师救命！风控系统报警说有一批订单的库存数据对不上——订单表里有 50 条新订单，但库存表中对应的 SKU 没变化。我查了 MySQL binlog，三个操作确实都在同一个分布式事务里执行成功了。为什么 CDC 消费端只收到了订单，没收到库存扣减？"

**大师**："你先看看这三个操作分别来自哪个数据库？"

**小胖**："订单在 order_db，库存扣减在 inventory_db，支付记录在 payment_db。三个不同的 MySQL 实例，各自有独立的 Connector。"

**大师**："这就对了。三个 Connector 各自独立消费各自的 binlog——它们之间没有任何协调机制。order_db 的 Connector 消费了订单 INSERT 事件后，inventory_db 的 Connector 可能因为网络抖动还没消费到库存 UPDATE 事件。下游消费者收到订单事件后直接处理了，10 分钟后才收到库存事件——但为时已晚。"

**小白**："那 GTID 能解决这个问题吗？GTID 不是全局事务 ID 吗？"

**大师**："GTID 是**单个 MySQL 实例内部**的全局事务 ID，不同实例的 GTID 是**各自独立**的——`db1:f3b3c7e4:1-100` 和 `db2:a1b2c3d4:1-50` 是两个不同的 GTID 空间。你不能用 GTID 做跨库关联。"

**小胖**："那跨库数据一致性到底怎么保证？"

**大师**："三段式方案——**用业务关联键做跨库关联、用事务元数据做原子性标记、用幂等消费做最终一致性**。"

```
三段式方案：

第一段：业务关联键
orders.order_id = inventory_log.order_id ← 用业务字段关联不同表的事件

第二段：事务元数据
提供 provide.transaction.metadata=true → 每个库的事务内事件共享 tx_id
单库内保证原子性（一个事务的 2 条 UPDATE 要么全消费，要么全不消费）

第三段：幂等消费
基于 source.db + source.table + after.id + source.gtid 的复合幂等键
确保每一条事件最多被消费一次
```

**大师**："技术映射：GTID = 每个城市自己的时钟（不同城市时钟不同，不能直接比较）。业务关联键（order_id）= 你的身份证号（不管你在哪个城市办事，身份证号是统一的）。幂等消费 = 每办完一件事就在护照上盖章，重复办事被拒绝。"

**小胖**："那我的那个问题具体怎么解？"

**大师**："在你下游消费端的代码里，不是收到订单事件就立刻处理——而是先缓存到 `pending_orders` 列表，然后等着收到对应的 inventory_log 事件（通过 order_id 关联）——收到之后批量提交。同时设置一个 60 秒的超时——超时后触发告警而不是静默丢弃。"

### 步骤2：实现跨库幂等消费端

```python
#!/usr/bin/env python3
"""跨库 CDC 消费者——基于 order_id 的事务关联 + GTID 幂等去重"""
import json, redis
from kafka import KafkaConsumer

r = redis.Redis(decode_responses=True)

# 消费两个 Topic
consumer = KafkaConsumer(
    'cross1.inventory.orders', 'cross2.warehouse.inventory_log',
    bootstrap_servers='localhost:9092',
    value_deserializer=lambda m: json.loads(m.decode('utf-8')),
    group_id='cross-db-consumer', auto_offset_reset='earliest'
)

pending_orders = {}  # key: order_id, value: {order_event, inventory_events}

def process_order_complete(order_id, data):
    """订单 + 库存事件凑齐后批量提交"""
    print(f"✅ 提交订单 {order_id}: order={data.get('order')}, inventory={len(data.get('inventory',[]))} events")
    # 实际业务：写入下游数据仓库
    r.delete(f"pending:{order_id}")  # 清理缓存

for msg in consumer:
    payload = msg.value.get("payload", msg.value)
    source = payload.get("source", {})
    gtid = source.get("gtid")
    db = source.get("db", "")
    table = source.get("table", "")
    after = payload.get("after", {})
    order_id = after.get("order_id") or after.get("id")
    
    # Step 1: 幂等检查（基于 GTID 或 复合键）
    dedup_key = f"cdc:{db}.{table}:{order_id}:{gtid}" if gtid else f"cdc:{db}.{table}:{order_id}"
    if r.exists(dedup_key):
        print(f"⏭ 跳过重复事件: {dedup_key}")
        continue
    
    # Step 2: 缓存事件，等待关联
    if order_id not in pending_orders:
        pending_orders[order_id] = {"order": None, "inventory": [], "started_at": payload.get("ts_ms")}
    
    if table == "orders":
        pending_orders[order_id]["order"] = after
    elif table == "inventory_log":
        pending_orders[order_id]["inventory"].append(after)
    
    # Step 3: 判断是否凑齐（订单 + 至少 1 条库存记录）
    data = pending_orders[order_id]
    if data["order"] and len(data["inventory"]) >= 1:
        process_order_complete(order_id, data)
        del pending_orders[order_id]
    
    # Step 4: 标记已处理（幂等去重）
    r.setex(dedup_key, 86400, "1")
```

### 步骤3：超时清理机制

```python
import threading, time

def cleanup_stale_orders():
    """每 30 秒清理超过 60 秒未凑齐的订单——触发告警"""
    while True:
        now = int(time.time() * 1000)
        for order_id, data in list(pending_orders.items()):
            if now - data["started_at"] > 60000:  # 60 秒超时
                print(f"⚠️ 超时订单告警: order_id={order_id}, has_order={data['order'] is not None}, "
                      f"inventory_count={len(data['inventory'])}")
                # 发送 P1 告警 → 人工介入排查
                del pending_orders[order_id]
        time.sleep(30)

threading.Thread(target=cleanup_stale_orders, daemon=True).start()
```

### 可能遇到的坑及解决方法

| 坑 | 现象 | 根因 | 解决 |
|----|------|------|------|
| 库存事件永远不到 | pending 缓存堆积 OOM | 数据库写入失败或 Connector 故障 | 60s 超时清理 + 告警 |
| GTID 重置导致重复消费 | 幂等去重失效 | MySQL RESET MASTER 归零 GTID | 复合去重键含 server_uuid |
| 跨库时序颠倒 | 库存事件先于订单事件到达 | 网络延迟差异 | pending 缓存双向等待 |

## 4. 项目总结

### 优点 & 缺点

| 方案 | 一致性级别 | 复杂度 | 适用场景 |
|------|-----------|--------|---------|
| GTID 去重 | At-least-once → Exactly-once | ★★ | 单数据库 |
| 事务元数据 + 批量 Upsert | 事务级原子性 | ★★★ | 多表事务（同库） |
| 业务关联键 + pending 缓存 | 最终一致性 | ★★★★ | 跨库关联 |
| 行级校验和对比 | 最终一致性校验 | ★★ | 自动化对账 |
| Kafka 事务 | Exactly-once（Kafka端） | ★★★★ | Kafka 到下游的精确一次 |

### 适用场景

1. **分布式事务的 CDC 同步**：订单+库存+支付的跨库关联
2. **多数据中心数据核对**：主备中心独立 CDC，通过校验和对比发现不一致
3. **合规审计**：DML 操作全量记录 + transaction.id 追溯完整操作链
4. **CDC 管道故障恢复**：消费者崩溃后通过复合幂等键精确恢复

### 不适用场景

1. **强一致性要求的实时转账**：CDC 的最终一致性不满足金融交易的强一致性要求
2. **极高吞吐 + 极低延迟**：pending 缓存机制引入额外延迟，不适合 < 50ms 场景

### 注意事项

- **GTID 在 MySQL Failover 后会变（UUID 部分变）**：去重键必须包含 `server_uuid`，不仅仅是 gtid 字符串
- **pending 缓存必须设置 TTL**：永远凑不齐的事务会 OOM
- **跨库关联字段的一致性**：确保 order_id 在两个库中类型一致（INT vs BIGINT）

### 常见踩坑经验

1. **"GTID 去重在 MySQL MHA 切换后失效"**——切换后 GTID 的 UUID 部分变了，去重键包含 UUID，导致"新"GTID 被误认为未处理。解决：用 `server_uuid + gtid_sequence` 作为复合键。
2. **"pending 缓存中大量超时订单，每个都是真实的"**——排查后发现是 inventory_log 的 Connector 因为 server.id 冲突而静默 FAILED。监控必须同时检查两个 Connector 的 Task 状态。
3. **"跨库 Join 时发现 target 表比 source 表多了几行"**——根因是手工在 target 表插入了数据。需要通过校验和对比发现差异后触发修复。

### 思考题

1. 如果 MySQL 的 GTID 被 `RESET MASTER` 手动重置归零，所有 GTID 从头开始计数（如 `uuid:1, uuid:2, ...`）。下游幂等去重的 Redis 缓存中已存在 `uuid:1` 的去重标记，新事务的 GTID 也是 `uuid:1`，导致误判为"重复"。如何设计更健壮的幂等去重方案来避免这个问题？

2. 某金融系统的订单同步链路为：order_db → Kafka Topic A → Flink → ClickHouse。如果要实现全链路 Exactly-Once（精确一次），需要在哪些环节引入事务或幂等机制？请画出全链路的幂等保证边界。

**（第23章思考题答案）**

1. 增量快照读到第 300 个 Chunk 时崩溃，恢复后从第 300 个 Chunk 继续——条件：① offset 中记录了 `snapshot_completed=false` 和当前的 `chunk_id`；② Schema History 中记录了快照执行进度。如果 offset 丢失（如 `connect-offsets` Topic 被清理），则快照从头开始。

2. `insert_insert` 策略使用 INSERT 型的水印事件来标记快照边界（MySQL binlog 支持 INSERT watermark），当 Connector 从 binlog 中读取到该水印事件时，知道"水印之后的数据属于 streaming 阶段"。`insert_delete` 策略使用 DELETE + INSERT 组合做水印标记（适用于 PG 的 WAL），因为 PG 的逻辑复制不支持 INSERT-only 的水印。
