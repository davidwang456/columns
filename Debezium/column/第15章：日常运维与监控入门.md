# 第15章：日常运维与监控入门

## 1. 项目背景

"大师，我们的 Debezium 已经跑了一个月，Connector 越来越多，但我不确定它到底稳不稳——Log 什么时候该看？Topic 什么时候该清理？offset 什么时候会丢？上周 Kafka 磁盘满了，Connector 不声不响地挂了 3 个小时我们才从业务方那里知道..."

从"能跑"到"稳跑"之间，隔着一整套日常运维体系。Debezium 涉及 MySQL/PostgreSQL/MongoDB（数据源层）、Kafka Connect（引擎层）、Kafka Broker（消息层）、Schema Registry（Schema 层）共 4 层技术栈，每一层都有各自的运维要点。本章将从最关键的三个方面建立日常运维体系——**巡检 Checklist**（每天/每周/每月该查什么）、**监控指标**（JMX + REST API 暴露的关键数字）、**应急 SOP**（三大最紧急故障的处理流程）。

### 痛点放大

- **"没问题"的 Connector 可能一直在 "silent fail"**：Connector 状态 RUNNING 但 Task 已经因为内存不足而静默停止消费
- **binlog 和 WAL 的过期清理可能误杀 Connector**：MySQL 的 `PURGE BINARY LOGS` 如果清理了 Connector 还没消费完的 binlog，Connector 永久 FAILED
- **Kafka Topic 的数据保留策略与 CDC 消息量不匹配**：7 天保留期的 Topic 每天写入 500GB，最终磁盘写满导致 ISR 收缩

## 2. 项目设计——三人对话

**（周一早会后的运维讨论）**

**小胖**："大师，上个项目我们 CDC 跑得好好的，突然有一天订单表的数据不更新了。查了半天才发现 Kafka 那个 Topic 的分区数据过期了，Consumer 的 offset 没了..."

**大师**："这就是为什么需要一套系统化的运维 Checklist。来，跟着我在白板上画一张运维地图——"

```
CDC 运维四层模型：

┌──────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────┐
│ 数据源层  │──▶│  Connector 层 │──▶│  Kafka 总线层  │──▶│ 消费端   │
│          │   │              │   │              │   │          │
│ 巡检项：  │   │ 巡检项：      │   │ 巡检项：      │   │ 巡检项：  │
│ binlog✅ │   │ 状态🔍       │   │ 磁盘使用率📊   │   │ Lag⚠    │
│ WAL✅   │   │ Lag📉        │   │ Topic 保留🕐   │   │ 数据一致✅│
│ Slot✅  │   │ 吞吐量📈      │   │ ISR 健康🔴     │   │ 幂等去重🔄│
│ 权限🔐  │   │ offset📌      │   │ 网路延迟🌐     │   │          │
└──────────┘   └──────────────┘   └──────────────┘   └──────────┘
```

**小胖**："分四层看得更清楚了。但我最关心的是——每天上班第一件事该查什么？"

**大师**："记住'日三八'——**

```
每日巡检（5 分钟）：
1. curl /connectors → 所有 Connector 列表，有无 FAILED
2. curl /connectors/{name}/status → 每个 Connector 和 Task 的状态
3. Kafka Topic Lag → kafka-consumer-groups 查看滞后的 offset

每周巡检（15 分钟）：
1. Kafka 磁盘使用率 → df -h
2. 数据库的 binlog/WAL 堆积情况
3. Connector 日志中 ERROR/WARN 关键词统计

每月巡检（30 分钟）：
1. 清理过期的 binlog（已确认被 Connector 消费过的）
2. Topic 数据保留策略审查（过期时间是否合理）
3. Schema Registry 中的 Subject 清理（不用的旧版本）
4. 压测演练（确保 CDC 管道的吞吐量达标）
```

**小白**："说到 Monitoring，Debezium 怎么暴露性能指标？比如每秒处理了多少事件、Lag 是多少？"

**大师**："通过 JMX。Debezium 暴露了丰富的 MBean（Management Bean），关键的有——"

| JMX MBean | 指标名 | 含义 | 告警阈值 |
|-----------|-------|------|---------|
| `debezium.mysql` | `MilliSecondsBehindSource` | 数据延迟（ms） | > 60000（1 分钟） |
| `debezium.mysql` | `TotalNumberOfEventsSeen` | 累计事件数 | 不设告警 |
| `debezium.mysql` | `SnapshotCompleted` | 快照是否完成 | false = 快照卡住了 |
| `debezium.mysql` | `Connected` | 是否连接到数据库 | false = 断连 |

**小胖**："这些 JMX 指标怎么看？我总不能每次都 JConsole 吧？"

**大师**："用 JMX Prometheus Java Agent——在 Kafka Connect 的 JVM 启动参数上加一行 `-javaagent:jmx_prometheus_javaagent.jar=8080:config.yml`，Prometheus 就可以直接拉指标数据了。Grafana 大盘我们下一章中级篇详细搭建。"

## 3. 项目实战

### 环境准备

确认 Kafka Connect 已运行，且至少部署了 2 个 Connector。

```bash
curl http://localhost:8083/connectors | python3 -m json.tool
```

### 步骤1：编写每日巡检脚本

**目标**：一键执行所有 Connector 的健康检查。

```bash
cat > daily_check.sh << 'EOF'
#!/bin/bash

CONNECT_URL="http://localhost:8083"

echo "========== Connector 巡检 $(date +%Y-%m-%d\ %H:%M:%S) =========="
echo ""

# 1 列出所有 Connector
echo "--- Connector List ---"
CONNECTORS=$(curl -s "${CONNECT_URL}/connectors" | python3 -c "import sys,json; print('\n'.join(json.load(sys.stdin)))")
echo "$CONNECTORS"
echo ""

# 2 检查每个 Connector 状态
echo "--- Connector Status ---"
for name in $(curl -s "${CONNECT_URL}/connectors" | python3 -c "import sys,json; [print(x) for x in json.load(sys.stdin)]"); do
    STATUS=$(curl -s "${CONNECT_URL}/connectors/${name}/status")
    CONN_STATE=$(echo "$STATUS" | python3 -c "import sys,json; print(json.load(sys.stdin).get('connector',{}).get('state','UNKNOWN'))")
    
    for task in $(echo "$STATUS" | python3 -c "import sys,json; [print(t['state']) for t in json.load(sys.stdin).get('tasks',[])]"); do
        if [ "$task" = "FAILED" ]; then
            echo "  ❌ [${name}] Task FAILED"
        fi
    done
    
    echo "  [${name}] Connector: ${CONN_STATE}"
done
echo ""

# 3 检查重要 Topic 是否存在
echo "--- Kafka Topic Check ---"
REQUIRED_TOPICS="connect-configs connect-offsets connect-statuses"
docker exec kafka kafka-topics --bootstrap-server localhost:9092 --list > /tmp/topics.txt
for t in $REQUIRED_TOPICS; do
    if grep -q "$t" /tmp/topics.txt; then
        echo "  ✅ $t"
    else
        echo "  ❌ $t MISSING!"
    fi
done
echo ""

echo "========== 巡检完成 =========="
EOF

chmod +x daily_check.sh
bash daily_check.sh
```

### 步骤2：监控 Kafka Topic 的 Lag

**目标**：确认 Connector Consumer 能否跟上数据产生速度。

```bash
# 查看 Connector 对应的 Consumer Group
docker exec kafka kafka-consumer-groups \
  --bootstrap-server localhost:9092 \
  --list | grep connect

# 查看 Lag（假设 group 名为 connect-debezium-lab）
# 实际 group 名由 CONNECT_GROUP_ID 参数决定
docker exec kafka kafka-consumer-groups \
  --bootstrap-server localhost:9092 \
  --group debezium-lab \
  --describe

# 预期输出：
# GROUP           TOPIC                PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
# debezium-lab    production-orders... 0          12345           12345           0
```

**关键指标解读**：
- `LAG=0`：消费无延迟，正常
- `LAG>1000`：消费有积压，需要关注
- `LAG` 持续增长：Connector 消费速度落后于数据写入速度

### 步骤3：JMX 指标暴露与采集

**目标**：通过 JMX 暴露 Debezium 指标，使用 jmx_prometheus_javaagent 采集。

```bash
# 在 docker-compose.yml 的 connect service 中添加 JMX 配置
# 如果已有 connect service，修改它的 environment 和 volumes：

# 方式1：使用 Confluent 镜像内置的 JMX 支持
# 在 connect service 的 environment 中添加：
#   CONNECT_METRIC_REPORTERS: io.confluent.metrics.reporter.ConfluentMetricsReporter
#   CONNECT_METRICS_ENABLE: "true"

# 方式2：使用 JMX Prometheus Agent（更灵活）
# 下载 jmx_prometheus_javaagent
wget https://repo1.maven.org/maven2/io/prometheus/jmx/jmx_prometheus_javaagent/0.20.0/jmx_prometheus_javaagent-0.20.0.jar -P ~/debezium-lab/

# 创建 jmx 配置
cat > ~/debezium-lab/jmx-config.yml << 'EOF'
rules:
- pattern: "debezium.*"
  name: "debezium_$1"
  labels: {}
- pattern: "kafka.connect.*"
  name: "kafka_connect_$1"
  labels: {}
EOF

# 测试直接访问 Connect 的 JMX 端口（默认不对外暴露）
# 在 connect service 的 environment 中添加：
#   KAFKA_JMX_PORT: 9999
#   KAFKA_JMX_HOSTNAME: connect
# 然后重启容器
```

### 步骤4：日志巡检与关键字告警

**目标**：编写脚本定期扫描 Connect 日志中的异常关键字。

```bash
cat > log_check.sh << 'EOF'
#!/bin/bash
LOG_FILE="/tmp/connect-recent-errors.log"

# 提取最近 30 分钟的日志
docker logs connect --since 30m 2>&1 > "$LOG_FILE"

# 关键字扫描
echo "--- 错误统计（30 分钟内） ---"
echo "ERROR count:   $(grep -c 'ERROR' "$LOG_FILE" 2>/dev/null || echo 0)"
echo "WARN count:    $(grep -c 'WARN' "$LOG_FILE" 2>/dev/null || echo 0)"
echo "Exception:     $(grep -c 'Exception' "$LOG_FILE" 2>/dev/null || echo 0)"
echo "Timeout:       $(grep -c 'Timeout' "$LOG_FILE" 2>/dev/null || echo 0)"
echo "Connect reset: $(grep -c 'Connection reset' "$LOG_FILE" 2>/dev/null || echo 0)"
echo ""

# 如果 ERROR 超过阈值（如 5 条），触发告警
ERROR_COUNT=$(grep -c 'ERROR' "$LOG_FILE" 2>/dev/null || echo 0)
if [ "$ERROR_COUNT" -gt 5 ]; then
    echo "⚠️  告警：最近 30 分钟 ERROR 日志超过 5 条！"
fi
EOF

chmod +x log_check.sh
bash log_check.sh
```

### 步骤5：Connector 配置备份与恢复

**目标**：定期备份所有 Connector 配置，防止意外丢失。

```bash
cat > backup_connectors.sh << 'EOF'
#!/bin/bash

BACKUP_DIR="./connector-backups/$(date +%Y%m%d_%H%M%S)"
mkdir -p "$BACKUP_DIR"

CONNECTORS=$(curl -s http://localhost:8083/connectors | python3 -c "import sys,json; print('\n'.join(json.load(sys.stdin)))")

for name in $CONNECTORS; do
    echo "Backing up: $name"
    curl -s "http://localhost:8083/connectors/${name}/config" | python3 -m json.tool > "${BACKUP_DIR}/${name}.json"
done

echo "Backup completed: ${BACKUP_DIR}"
ls -la "${BACKUP_DIR}"
EOF

chmod +x backup_connectors.sh
bash backup_connectors.sh
```

### 可能遇到的坑及解决方法

| 坑 | 现象 | 解决 |
|----|------|------|
| Connector 看起来正常但数据停止更新 | LAG=0 但没新数据 | 检查数据库端——binlog 是否正常写入、Replication Slot 是否 active |
| `kafka-consumer-groups` 找不到 Connect Group | 命令返回空 | Group ID 由 `CONNECT_GROUP_ID` 设置，确认拼写正确 |
| 磁盘使用率持续上涨 | 监控曲线一直向右上 | Topic 的 `retention.ms` 设置太长或 `retention.bytes` 太小。检查 Topic 配置 |

## 4. 项目总结

### 优点 & 缺点

| 方案 | 优点 | 缺点 |
|------|------|------|
| Shell 脚本 | 简单、零依赖、可定制 | 缺乏可视化、趋势分析 |
| JMX + Prometheus | 标准化、微秒级采集 | 需要额外的 Prometheus 基础设施 |
| Debezium UI | 可视化、易上手 | 功能有限、不可编程报警 |

### 适用场景

1. **日巡检**：Shell 脚本 + crontab 定时执行，失败发邮件/webhook
2. **实时监控**：JMX + Prometheus + Grafana（第28章详讲）
3. **配置管理**：定期备份 Connector JSON 到 Git
4. **合规审计**：脚本输出作为运维日志保存
5. **故障排查**：本章的 5 个步骤作为新人培训材料

### 注意事项

- **`kafka-consumer-groups --reset-offsets` 是危险命令**：错误使用会导致数据丢失或重复。禁止在生产环境使用 `--to-earliest`
- **binlog 清理与 Connector 的协调**：在 MySQL 执行 `PURGE BINARY LOGS TO 'xxx'` 前，确认所有 Connector 的 offset 都已经超过该文件
- **备份文件的安全存储**：Connector 配置包含数据库密码，备份文件需加密存储或放在权限受控的目录

### 思考题

1. 如果你的巡检脚本检测到 Connector FAILED，你会如何设计自动恢复策略？需要考虑哪些边界条件（如：连续重启 3 次仍 FAILED 则停止自动恢复并发送告警）？

2. Kafka Connect 的 offset 存储在 `connect-offsets` Topic 中，如何确保这个 Topic 的数据不会因为 `retention.ms` 过期而被清理？如果清理了会发生什么？

**（第14章思考题答案）**

1. 三种幂等方案：① 基于业务主键的 Upsert——利用 `source.db + source.table + after.id` 组合键，下游执行 `INSERT ... ON DUPLICATE KEY UPDATE` 或 Redis `SET key value NX`（仅不存在时写入）；② 基于事件位置的去重表——维护一张消费记录表 `(connector_name, binlog_file, binlog_pos)`，每次消费前检查是否已处理过（和数据库事务绑定）；③ 基于事务 ID 的去重——使用 `source.gtid`（MySQL）或 `source.lsn`（PG）作为去重键。

2. 防止 offset 丢失的方案：① 使用 Redis 作为 offset store（配置 `debezium.source.offset.storage=io.debezium.server.redis.RedisOffsetBackingStore`），Redis 独立于容器生命周期；② 使用 Docker volume 挂载 offset 文件目录到宿主机；③ 使用 K8s PersistentVolume + StatefulSet 确保 pods 重启后数据不丢失；④ 定期备份 offset 文件到对象存储（S3/OSS）。
