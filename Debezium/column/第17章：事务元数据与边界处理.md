# 第17章：事务元数据与边界处理

## 1. 项目背景

凌晨 3:15，某电商平台的运维监控群炸了——"库存扣减成功但物流单没生成！"运维组长紧急排查后发现，3:15:32 到 3:15:45 之间，一个分布式事务跨 3 张表执行了关键操作：`UPDATE inventory SET stock=stock-1 WHERE sku='PHONE15'` → `INSERT INTO shipments (order_id, tracking_no) VALUES (...)` → `UPDATE orders SET status='shipped' WHERE id=98765`。但下游的 Flink 任务在消费完第一条 UPDATE 后因为 OOM 崩溃重启了 5 分钟，导致后面两条事件在恢复时被当作"新事件"重复消费——物流单重复生成、订单状态被重复更新，最终库存扣了 1 件却发了 2 件货。

这不是个例。在未开启事务元数据之前，Debezium 产出的每条 Change Event 都是"孤立"的——下游消费者无从判断"A、B、C 三条事件属于同一个数据库事务"。如果中间崩溃，A 已经被消费了但 B 和 C 丢失，数据就处于"部分提交"的不一致状态。更糟糕的是，当多张表的数据混在同一个 Topic 时，消费者无法按事务维度做批量提交（要么全成功，要么全回滚）。

**痛点总结**：
- **无事务边界**：消费者无法判断哪些事件属于同一个数据库事务，一致性无法保证
- **崩溃风险**：消费者在事务中间崩溃，已消费的事件无法回滚
- **跨表关联困难**：多表事务中，下游无法将分散的 Change Event 关联为一次原子操作

开启事务元数据后，每条 Change Event 会增加 `transaction` 字段，包含事务 ID（基于 GTID/LSN）、事务内序号、总事件数。下游消费者可以用事务 ID 做幂等去重，收到 BEGIN 开始缓存、收到 END 批量提交。

Debezium 从 1.5 版本开始支持事务元数据（Transaction Metadata）。本章将带你从理解原理到实战部署，最终掌握基于事务元数据的"精确一次"消费模式。

## 2. 项目设计——三人对话

**（周二下午，小胖端着一杯奶茶走进会议室）**

**小胖**："大师，我上周部署的 CDC 管道出了个诡异的问题——下游物流系统说它收到了两次同一条订单的'shipped'通知，发了两个快递出去。我看 MySQL binlog 里明明只 UPDATE 了一次，Kafka 里怎么会有两条消息？"

**大师**："这大概率是消费者在事务中间崩溃导致的。我问你——你那个 `UPDATE orders SET status='shipped'` 是不是和别的 SQL 在一个事务里？"

**小胖**："是啊，一个事务里先 UPDATE inventory 扣库存，再 INSERT shipments 生成物流单，最后 UPDATE orders 改状态。这是标准的'一单三操作'。"

**大师**："那问题就很清晰了。你的 Flink 消费者在消费完第一条 UPDATE 后崩溃了，重启后从 Kafka offset 恢复——但恢复点刚好是第一条 UPDATE 之后。于是第一条 UPDATE 被消费了两次。"

**小白**（放下手中的《设计数据密集型应用》）："等等，Kafka 不是有 Consumer Group 的 offset 机制吗？崩溃恢复应该从最后提交的 offset 开始才对，怎么会重复？"

**大师**："因为 Flink 在做 checkpoint 时，第一条 UPDATE 已经处理完了但还未提交到下游物流系统，checkpoint 记录的就是第一条之后的 offset。恢复时 Flink 从 checkpoint 恢复，重新消费了第一条 UPDATE，导致物流系统收到了两次通知。"

**小胖**："那怎么办？难道要我自己在代码里写去重逻辑？"

**大师**："对，而且 Debezium 已经帮你做好了 80% 的工作——事务元数据。开启 `provide.transaction.metadata=true` 后，每条 Change Event 的 `transaction` 字段会包含事务 ID 和状态标记。"

```json
// 开启事务元数据后的事件结构
{
  "payload": {
    "before": null,
    "after": { "id": 98765, "status": "shipped", ... },
    "source": { "gtid": "f3b3c7e4:1-50", ... },
    "op": "u",
    "transaction": {
      "id": "f3b3c7e4:1-50",        // 事务 ID（基于 GTID）
      "total_order": 3,              // 事务内总共 3 条事件
      "data_collection_order": 2     // 这是第 2 条（1-based）
    }
  }
}
```

**小白**："那 transaction.status 字段呢？我怎么区分 BEGIN 和 END？"

**大师**："好问题。`status` 字段不是每条事件都有——它只出现在事务的**第一条**和**最后一条**事件的消息**外层**（在 `ExtractNewRecordState` 拍平之前）。如果不用 SMT 拍平，结构是这样的——"

```
事务内 3 条事件的消息信封：
Event 1: { "status": "BEGIN",  "payload": { "transaction": { "id":"gtid:50", "total_order":3, "data_collection_order":1 } } }
Event 2: { "status": null,     "payload": { "transaction": { "id":"gtid:50", "total_order":3, "data_collection_order":2 } } }
Event 3: { "status": "END",    "payload": { "transaction": { "id":"gtid:50", "total_order":3, "data_collection_order":3 } } }
```

**技术映射**：事务元数据 ≈ 快递的三连包——快递单号（`transaction.id`）= 同一批包裹的唯一标记；`total_order` = 这批一共几件；`data_collection_order` = 这是第几件；`status=BEGIN/END` = 包裹清单的开头和结尾标记。

**大师**："消费者收到 BEGIN 后开始缓存事件放入 `pending_tx` 字典，收到中间事件继续追加，收到 END 后把整个事务的所有事件一次性批量提交。如果没收到 END（Connector 崩溃），缓存的 `pending_tx` 会被丢弃，不会出现'部分提交'。"

**小白**："但如果订单事务有 10 万行变更（比如批量更新库存），缓存 10 万条事件在内存里不会 OOM 吗？"

**大师**："这正是事务元数据的一个经典陷阱。对于长事务（>1000 行），建议配合 `max.queue.size` 和分批消费策略。简单说——不要等 END 才提交，而是在收到 `data_collection_order % 1000 == 0` 时做一次'分段提交'，同时记录 progress marker（进度标记），确保即使中间崩溃也能从最后一段恢复。"

**小胖**："我懂了！但还有一个问题——如果用 ExtractNewRecordState SMT 拍平了消息，那些 BEGIN/END status 还在吗？"

**大师**（摇头）："这就是很多人踩的坑。`ExtractNewRecordState` 会去掉外层的 `schema` + `payload` 信封，而 `status` 字段刚好在信封层——所以拍平后 `status` 就丢了。如果要用事务元数据做一致性保障，**要么不拍平**（保留完整事件结构），**要么用 `transaction.id`** 做幂等去重（不需要 status）。"

### 步骤3：实现基于 transaction.id 的幂等消费（Python）

**目标**：编写一个消费端程序，利用 `transaction.id` 实现事务级幂等去重。

```python
#!/usr/bin/env python3
"""基于事务 ID 的幂等 CDC 消费者"""

import json
from kafka import KafkaConsumer
import redis

consumer = KafkaConsumer(
    'tx_demo.inventory.orders',
    bootstrap_servers='localhost:9092',
    value_deserializer=lambda m: json.loads(m.decode('utf-8')),
    group_id='tx-consumer-group',
    auto_offset_reset='earliest'
)

# Redis 用于幂等存储（24 小时去重窗口）
redis_client = redis.Redis(host='localhost', port=6379, decode_responses=True)

# 内存缓存——收集同一事务的事件
pending_transactions = {}

def process_transaction(tx_id, events):
    """批量处理一个完整事务的所有事件"""
    print(f"Processing transaction {tx_id} with {len(events)} events")
    
    # 按 data_collection_order 排序
    events.sort(key=lambda e: e.get("data_collection_order", 0))
    
    for event in events:
        op = event.get("op")
        after = event.get("after", {})
        
        if op == "c":
            # INSERT → 写入下游
            print(f"  INSERT: {after}")
        elif op == "u":
            # UPDATE → 更新下游
            print(f"  UPDATE: {after}")
        elif op == "d":
            # DELETE → 删除下游（使用 before 中的主键）
            before = event.get("before", {})
            print(f"  DELETE: {before}")
    print(f"  Transaction {tx_id} committed ✅")

for msg in consumer:
    payload = msg.value.get("payload", msg.value)
    tx_info = payload.get("transaction")
    
    if not tx_info:
        # 非事务事件——直接处理（开启事务元数据后这种情况少见）
        print(f"Single event (no tx): {payload.get('op')}")
        continue
    
    tx_id = tx_info["id"]
    total = tx_info["total_order"]
    current = tx_info["data_collection_order"]
    
    # 幂等检查——如果这个事务 ID 已经处理过，跳过
    dedup_key = f"cdc:tx:{tx_id}"
    if redis_client.exists(dedup_key):
        print(f"Skipping duplicate tx: {tx_id}")
        continue
    
    # 收集事件
    if tx_id not in pending_transactions:
        pending_transactions[tx_id] = {"events": [], "total": total}
    pending_transactions[tx_id]["events"].append(payload)
    
    # 判断事务是否完整（所有事件已到齐）
    if len(pending_transactions[tx_id]["events"]) >= pending_transactions[tx_id]["total"]:
        # 事务完整——批量处理
        process_transaction(tx_id, pending_transactions[tx_id]["events"])
        # 标记为已处理（24 小时去重窗口）
        redis_client.setex(dedup_key, 86400, "1")
        del pending_transactions[tx_id]
```

### 可能遇到的坑及解决方法

| 坑 | 现象 | 根因 | 解决 |
|----|------|------|------|
| 拍平后 status 丢失 | 看不到 BEGIN/END 标记 | ExtractNewRecordState 去掉了外层信封 | 不拍平或改用 transaction.id 去重 |
| 长事务 OOM | 消费者 GC 频繁 / 内存溢出 | pending_tx 中缓存了 > 10 万条事件 | 实现分段提交 + 超时清理（60s 内没收齐则丢弃） |
| GTID 重复导致误判 | 幂等去重误跳过了新事务 | MySQL 手工 RESET MASTER 后 GTID 归零 | 使用复合去重键：`gtid + server_id + timestamp` |
| transaction 字段为 null | 部分事件没有 transaction 信息 | 快照阶段的事件不包含事务元数据 | 只对 `source.snapshot=false` 的事件做事务幂等 |

## 4. 项目总结

### 优点 & 缺点

| 维度 | 开启事务元数据 | 不开启 | Canal 事务支持 |
|------|--------------|--------|---------------|
| 事务边界感知 | ★★★★★ 完整 BEGIN/END | ★☆☆☆☆ 无法感知 | ★★★☆☆ 部分支持 |
| 下游幂等实现 | ★★★★★ 基于 tx_id 天然去重 | ★★☆☆☆ 需业务主键 | ★★★★☆ 基于 GTID |
| 跨表一致性 | ★★★★★ 多表同 tx_id | ★☆☆☆☆ 不可关联 | ★★★☆☆ 需额外关联 |
| 消息体积 | ★★★★☆ 增加 ~10% | ★★★★★ 最小 | ★★★★☆ 增加 ~15% |
| 长事务内存压力 | ★★★☆☆ 需分段提交 | ★★★★★ 无 | ★★★☆☆ 同样问题 |

### 适用场景

1. **金融交易系统**：一笔交易涉及扣款 + 记账 + 风控的多表写入，必须原子提交到下游
2. **订单-物流-库存联动**：订单创建后需要库存+物流同时更新，不允许部分成功
3. **跨系统数据一致性校验**：利用 tx_id 关联不同 Topic 的事件做数据核对
4. **审计溯源**：通过 tx_id 回溯一次用户操作引发的全部数据库变更
5. **CDC 管道故障恢复**：消费者崩溃后利用 tx_id 精确恢复，避免重复或遗漏

### 不适用场景

1. **纯单表单行操作**：每次只有一条 INSERT/UPDATE，事务元数据的开销大于收益
2. **低延迟极致场景**：事务元数据的 BEGIN/END 判断增加了延迟（需等事务提交）

### 注意事项

- **`provide.transaction.metadata` 默认关闭**：生产环境建议**必须**手动开启
- **快照阶段无事务元数据**：只有 streaming 阶段的事件才包含 transaction 字段
- **需要 GTID 或 binlog file+pos 支持**：MySQL 的 GTID 是最佳实践，PG 使用 LSN

### 常见踩坑经验

1. **"开启事务元数据后下游消息量翻倍"**——根因是 `provide.transaction.metadata` 误读了参数，实际没有翻倍，只是每条消息多了 ~100 bytes 的 transaction 字段。排查：确认消费代码没有将 transaction 字段误当作业务字段处理。

2. **"pending_tx 内存泄漏导致 OOM"**——根因是事务不完整（BEGIN 已到但 END 永不到达）。原因可能是 Connector 在事务中间崩溃，重建后该事务的剩余事件不会再被捕获。排查：为 pending_tx 中的事务设置 60 秒超时（`expireAfterWrite`），过期自动丢弃并告警。

3. **"基于 GTID 的去重键在 MySQL failover 后失效"**——根因是 GTID 的 UUID 部分在 MySQL 主从切换后改变。解决：使用 `server_uuid + gtid_sequence` 作为复合去重键，而不是单一的 gtid 字符串。

### 思考题

1. 如果开启 `provide.transaction.metadata=true` 同时也开启 `ExtractNewRecordState` SMT，拍平后的消息中 transaction 字段是否还存在？如果在，它位于消息的哪个层级？消费者如何读取它？

2. 一个跨 3 张表的事务在 CDC 中被分到了 3 个不同的 Kafka Topic（按表路由）。下游如何在一个消费组中同时消费这 3 个 Topic 并实现事务级原子提交？（提示：考虑使用 Kafka Streams 的 `foreign-key join` 或 Flink 的 interval join）

**（第16章思考题答案）**

1. 三个 Connector 共用同一套 Connect 集群时，单个 Connector OOM 在 Java 层面只会影响该 Connector 的 Task 线程（每个 Connector Task 是独立线程），不会直接 OOM kill 其他 Connector。但极端情况下（如共享内存池耗尽），可能影响同一 JVM 内的其他 Task。资源隔离方案：① 使用不同的 `CONNECT_GROUP_ID` 将高风险 Connector 拆分到独立 Worker 组；② 在 K8s 中为每个 Connector 分配独立的 `resources.limits`；③ 使用 JVM 的 `-XX:MaxRAMPercentage` 限制单个 Worker 进程的内存上限。

2. Flink 方案：使用 Flink CDC 的 `mysql-cdc` connector 消费两个 Topic，通过 `interval join`（时间窗口 JOIN）关联订单和用户画像，需注意状态 TTL 和 watermark 策略。ksqlDB 方案：`CREATE STREAM order_stream AS SELECT ... FROM orders_stream o LEFT JOIN users_table u ON o.user_id = u.id`，ksqlDB 自动维护 join 状态。
