# 第25章：性能调优实战——吞吐量翻倍的 10 个参数

## 1. 项目背景

"运营大屏上的 GMV 数字和真实成交差了一天！就因为我们昨天上午系统 CPU 跑满了!"——电商平台的运营总监在周会上拍了桌子。排查发现：Debezium 全部使用默认配置运行，在促销活动期间订单写入量飙升到 15000 QPS，而 `max.batch.size=2048` 和 `max.queue.size=8192` 这两个默认值严重不足——内存队列满之后，binlog 消费线程被阻塞，形成"背压死循环"：队列满 → 停止消费 binlog → binlog 积压 → 恢复后大量消费 → 队列又满。

更糟的是，排查过程中发现有 5 个关键参数从未被调整过，而这些参数正是 CDC 管道从"能用"到"好用"的分水岭。运维团队因为不了解这些参数的联动关系——比如改了 `max.batch.size` 但忘了调 Kafka Broker 的 `max.message.bytes`——导致消息被 Kafka 拒绝，数据严重延迟。

**痛点放大**：默认配置在低流量场景（< 1000 QPS）下表现正常，但当流量增长 10 倍以上时会出现三个连锁问题：

1. **队列背压死循环**：`max.queue.size=8192` 很快被填满 → 暂停消费 → binlog 积压 → 恢复后队列瞬间再满 → 再次暂停。延迟从秒级退化到分钟级。
2. **Kafka 发送碎片化**：`max.batch.size=2048` 意味着每 2048 条事件就要发起一次 Kafka 网络请求——在 15000 QPS 下每秒 7 次请求，每次只传几百 KB，TCP 连接效率极低。
3. **单分区瓶颈**：默认 `topic.creation.default.partitions=1`，下游消费者只能单线程消费，即使 Connector 吞吐再高，消费端也追不上。

本章将带你通过系统性实验，把 Connector 吞吐量从 5000 事件/秒提升到 20000+ 事件/秒，并理解每个参数背后的权衡代价。

### 痛点流程图

```
高峰期 15000 QPS
      │
      ▼
max.queue.size=8192 (默认)
      │
      ▼
队列 0.5 秒内填满 ──→ Connector 暂停消费 binlog
      │                      │
      ▼                      ▼
队列排空 (0.3s)         binlog 积压增加
      │                      │
      └──────────────────────┘
              循环
      │
      ▼
端到端延迟: 2s → 15min+
```

## 2. 项目设计——三人对话

**（周三下午，运维小张盯着 Grafana 上那条几乎垂直向上的 Lag 曲线，满头大汗）**

**小胖**（拿着一包薯片走进来）："小张你怎么了？看起来像是刚跑完马拉松。"

**小张**："比马拉松累多了——促销期间 CDC 延迟都 15 分钟了！我改了 `max.batch.size` 从 2048 到 8192，结果不但没有好转，Connector 还开始报 Kafka 发送错误。我是不是改错了？"

**大师**（放下手中的咖啡杯）："你没有改错方向——但你没改全。改 `max.batch.size` 时，如果 Kafka Broker 的 `max.message.bytes` 还是默认的 1MB，一批 8192 条 Avro 消息（约 1.2MB）就会超过 Broker 的接收上限被拒绝。这是一个典型的'联动参数'问题。"

**小白**："等等，我怎么知道哪个参数先调、哪个后调？这么多参数之间有没有一个优先级顺序？"

**大师**："好问题。记住 **BQNJ 调优漏斗**——从外到内、从上到下、从大到小：

```
第一优先：Topic 分区数
    └── 决定了并行消费的天花板——分区不扩，后面全白做

第二优先：队列大小（max.queue.size）
    └── 决定了背压容忍度——队列太小，流量稍有波动就被打回

第三优先：批次大小（max.batch.size）
    └── 决定了吞吐效率——批次太小，网络请求太碎

第四优先：JVM 参数
    └── 决定了前三者的上限——heap 不够，队列和批次调大只会 OOM
```

**小胖**："这不就像我们学校食堂——先决定开几个打饭窗口（分区），再决定每个窗口前面的排队区能站多少人（队列），再决定阿姨每次打多少菜（批次），最后确保阿姨有体力不累趴（JVM）！"

**大师**："小胖你这次比喻很精准！技术映射：分区数 = 食堂窗口数，队列大小 = 排队区容量，批次大小 = 一勺菜的量，JVM = 阿姨的体力。四个都对了，才能高效出餐。"

**小白**："那快照阶段的 `snapshot.fetch.size` 和 streaming 阶段的 `max.batch.size` 有什么区别？前面是读数据库的批次，后者是写 Kafka 的批次？"

**大师**："精确！快照阶段是 Connector 从 MySQL 拉数据的节奏——`snapshot.fetch.size=2000` 意味着每次 `SELECT * FROM orders WHERE id BETWEEN 1 AND 2000`。Streaming 阶段是 Connector 向 Kafka 推数据的节奏——`max.batch.size=2048` 意味着攒够 2048 条事件才调用一次 `KafkaProducer.send()`。"

**小张**："那 compression.type 用 snappy 还是 zstd？网上说 zstd 压缩率最高但 CPU 开销也高。"

**大师**："这个话题需要结合硬件条件来选——

| 压缩类型 | 压缩率 | CPU 开销 | 适用场景 |
|---------|--------|---------|---------|
| `none` | 0% | 0% | 内网带宽充足 |
| `snappy` | 50-60% | 3-5% | 平衡推荐 |
| `zstd` | 70-80% | 8-12% | 极限压缩 |
| `gzip` | 60-70% | 15-20% | 不推荐（CPU太高） |
| `lz4` | 40-50% | 1-2% | 低延迟场景 |

如果 Connector CPU 余量 > 30%，用 zstd 获得最佳压缩；如果 CPU 已经 > 70%，用 snappy 平衡一下；如果跨机房（带宽是瓶颈），无脑上 zstd。"

**小张**："最后一个问题——调优完怎么验证效果？就看那个 Grafana 面板吗？"

**大师**："建议三步验证法：① 快照阶段——对比同一张 50 万行表的快照耗时；② Streaming 阶段——用 Sysbench 持续写入，观察 `MilliSecondsBehindSource` 的 P99；③ 消息体积——对比 Kafka Topic 的 `LogEndOffset` 增长速率和磁盘占用。如果三个指标都有 ≥ 3 倍改善，说明调优有效。"

### 步骤1：建立性能基线——默认配置的快照耗时

**目标**：使用全默认配置对一张 50 万行表执行快照，记录耗时作为基线。

    DECLARE i INT DEFAULT 1;

DELIMITER ;
CALL fill_perf_data();

### 步骤4：联动参数——Kafka Broker 限制

**目标**：验证如果不调大 `max.message.bytes`，大批次会被 Broker 拒绝。

```bash
# 当前 Kafka Broker 的默认 max.message.bytes = 1MB
docker exec kafka kafka-configs --bootstrap-server localhost:9092 \
  --entity-type brokers --entity-name 1 --describe | grep max.message.bytes

# 计算：16384 条 × 120 bytes/条(Avro) ≈ 1.97MB
# 如果没有调大 Broker 的 max.message.bytes，消息会被拒绝！

# 解决方案——联动调整 Kafka Broker 配置：
docker exec kafka kafka-configs --bootstrap-server localhost:9092 \
  --entity-type brokers --entity-name 1 --alter \
  --add-config max.message.bytes=10485760  # 10MB
```

### 可能遇到的坑及解决方法

| 坑 | 现象 | 根因 | 解决方法 |
|----|------|------|---------|
| 调大批次后 Kafka 拒绝 | Connector 日志 `RecordTooLargeException` | `max.batch.size × 单条大小 > max.message.bytes` | 调大 Broker 的 `max.message.bytes` 至少为 `batch.size × 2 bytes` |
| 调大分区后 offset 丢失 | 老 Consumer Group 从 offset 0 开始 | 分区数变大后，新分区的 offset 默认为 0 | 使用 `kafka-consumer-groups --reset-offsets --to-latest` |
| `snapshot.fetch.size` 过大导致 OOM | Connector JVM GC 频繁 | 单次从 MySQL 读取过多数据 | 20000 是安全上限，超过 50000 风险高 |
| 压缩后 CPU 飙升 | Connect Worker CPU 从 30% → 85% | snappy/zstd 压缩消耗 CPU | 如果 CPU 基线 > 70%，先用 `none` 确认瓶颈在其他地方 |
| 10 个参数一起改不知道哪个生效 | 性能提升无法归因 | 一次改了太多参数 | **每次只改 1-2 个参数，测完再改下一个** |

## 4. 项目总结

### 优点 & 缺点（性能调优 vs 默认配置 vs 极限调优）

| 维度 | 默认配置 | 本章调优 | 极限调优（百万 TPS） |
|------|---------|---------|-------------------|
| 吞吐量 | ★★ 5000 events/s | ★★★★ 18000 events/s | ★★★★★ 85000 events/s |
| 延迟 (P99) | ★★ < 10s | ★★★★★ < 0.5s | ★★★★★ < 0.1s |
| CPU 使用率 | ★★★★★ 30% | ★★★★ 55% | ★★★ 75% |
| 内存占用 | ★★★★★ 512MB | ★★★★ 2GB | ★★★ 8GB |
| 运维复杂度 | ★★★★★ 零配置 | ★★★★ 调 10 个参数 | ★★★ 需 OS+Kafka+JVM 联动 |

### 适用场景

1. **促销/大促等高流量场景**：业务流量是日常的 5-20 倍，必须提前调优
2. **新业务上线全量快照**：大表（> 500 万行）的快照时间从小时级降到分钟级
3. **跨机房数据同步**：带宽是瓶颈，压缩参数（snappy/zstd）能省 70% 传输量
4. **延迟敏感系统**：实时风控、实时大屏要求 < 1s 的数据新鲜度
5. **多团队共享 Kafka 集群**：压缩后节省磁盘，避免 Topic 快速写满 Broker 磁盘

### 不适用场景

1. **日常低流量场景**（< 1000 QPS）：默认配置已足够，调优收益不明显
2. **资源极度受限的 IoT/边缘设备**：内存 < 1GB 时，调大 queue.size 会导致 OOM

### 注意事项

- **参数联动陷阱**：`max.batch.size ↑ → 消息批变大 → 必须联动调 Kafka `max.message.bytes`**
- **分区数是"硬上限"**：Topic 只有 1 个分区，Connector 参数调到天上也无法并行消费
- **Compaction 后 CPU 增加了**：snappy/zstd 消耗 CPU，如果 CPU 已 > 80%，只用 `none`

### 常见踩坑经验

1. **"我一口气把 10 个参数全改了，性能反而更差"**——根因是 `compression.type=zstd` 在 CPU 已经 > 85% 的情况下雪上加霜。正确做法：每次只改 1-2 个参数，梯度测试。
2. **"快照快了 3 倍但 streaming 阶段经常 OOM"**——根因是 `snapshot.fetch.size=50000` 过大，单次 SELECT 返回了太多数据到内存。最大安全值为 20000。
3. **"调大分区后老消费者数据重复"**——根因是 Kafka 分区数变大后，新分区的 offset 默认从 earliest 开始。消费者需要手动 `--reset-offsets`。

### 思考题

1. 如果 `max.batch.size=65536`，Avro 序列化后单条约 120 bytes，一批数据约 8MB。Kafka Broker 的 `max.message.bytes=1MB`（默认值）时，这批数据会发生什么？除了调大 Broker 的 `max.message.bytes` 外，还有哪些参数需要联动调整？（提示：Connector Producer 端的 `max.request.size`）

2. 为什么调优后 Full GC 频率从每 3 分钟一次降到了几乎没有？队列变大了（8192 → 65536）不是意味着更多内存占用吗？（提示：考虑 GC 的触发原因是"频繁创建临时对象"还是"长期占用大对象"）

**（第24章思考题答案）**

1. `RESET MASTER` 后 GTID 归零，旧缓存中的去重键 `cdc:gtid:{uuid:1}` 会和新事务的 GTID `uuid:1` 冲突。健壮方案：复合去重键 `{server_uuid}:{gtid_sequence}:{source.db}:{source.table}:{after.id}`。即使 GTID 归零，`server_uuid` 不变（除非重建了 MySQL 实例的 datadir）。
2. 全链路 Exactly-Once 的保证边界：① Kafka Connect → Kafka：依赖 Kafka 的幂等 Producer（`enable.idempotence=true`）确保不重复投递；② Kafka → Flink：Flink 的 Two-Phase Commit（Checkpoint + Kafka 事务）确保 checkpoint 内的消息精确一次；③ Flink → ClickHouse：Flink 的 JDBC Sink 配合 ClickHouse 的 `ReplacingMergeTree` + 幂等写入实现最终一致。每个边界都需要独立的事务/幂等机制。

> **推广提示**：将性能参数模板化为三个 Profile 文件——`profile-small.json`（< 1000 QPS）、`profile-medium.json`（1000-10000 QPS）、`profile-large.json`（> 10000 QPS）。运维人员在部署新 Connector 时根据业务规模选择对应 Profile，避免凭感觉调参。测试团队应将这三个 Profile 作为压测场景的 baseline，在新版本上线前执行性能回归对比。
