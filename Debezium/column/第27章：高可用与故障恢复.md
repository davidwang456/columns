# 第27章：高可用与故障恢复

## 1. 项目背景

某金融支付平台的核心 CDC 管道平稳运行半年后，遭遇了第一次严重的 HA 事故——核心交换机因光模块故障导致 MySQL 与 Kafka Connect Worker 之间的网络中断 15 分钟。网络恢复后，运维发现一个灾难性现象：MySQL 的 binlog 文件已被自动清理脚本（`PURGE BINARY LOGS BEFORE NOW() - INTERVAL 24 HOUR`）删除了异常期间的 binlog，而 Connector 的 offset 仍然指向 15 分钟前的中断点。offset 指向的 binlog 文件已不存在——Connector 状态 PERMANENT FAILED。

唯一的恢复手段是全量快照重建——但这张表有 2 亿行，快照需要 5 小时。下游 6 个核心业务系统（风控、对账、报表、清算...）全线瘫痪 5 小时，造成了严重的事故定级。

**高可用不是"不挂"，而是"挂了能快速恢复且数据不丢"。** 本章将构建三层 HA 防线——数据库层（binlog/WAL 保留策略）、Connect 层（多 Worker + offset 冗余）、恢复层（GTID/LSN 增量恢复），并通过灾害模拟演练验证每层的恢复能力。

### 痛点放大

| 故障层级 | 典型故障 | 未防护后果 | 防护手段 |
|---------|---------|-----------|---------|
| 数据库层 | binlog 过期清理 | offset 失效 → 永久 FAILED → 全量快照 | 保留至少 72h |
| 数据库层 | PG WAL 膨胀 | Replication Slot 不消费 → 磁盘满 | Slot 监控 + 告警 |
| Connect 层 | Worker 宕机 | 所有 Task 停止 | 多 Worker + Rebalance |
| Connect 层 | offset Topic 被删 | 所有 Connector 从零开始 | replication factor ≥ 3 |
| 恢复层 | GTID 对应事务已过期 | binlog 不再包含该事务 | binlog 保留 ≥ offset 回溯窗口 |

## 2. 项目设计——三人对话

**（凌晨 3 点，运维小张从被窝里爬起来紧急处理事故。第二天一早就直奔大师的工位）**

**小张**："大师，昨晚的事故复盘结果出来了——根因是 binlog 只保留了 24 小时，而网络中断了 15 分钟，但我们没及时发现。等我们发现时，MySQL 的自动清理脚本已经把故障点附近的 binlog 全删了。我有个疑问——为什么 MySQL 要默认只保留 24 小时的 binlog？多保留几天不行吗？"

**大师**："这确实是最容易被忽视的配置。MySQL 的默认 `binlog_expire_logs_seconds = 2592000`（30 天）是从 8.0.26 版本才开始改的——之前是 `expire_logs_days = 0`（永不删除），但很多人为了省磁盘空间会设成 1-2 天。对于 CDC 管道来说，binlog 保留时长至少应该是故障最大恢复时间（MTTR） × 2。建议至少 72 小时——给运维团队和自动化恢复足够的时间窗口。"

**小胖**："那如果 binlog 已经被清理了，是不是就只能重做全量快照这一条路了？"

**大师**："几乎是这样——但有一个后门，叫 **GTID 增量恢复**。如果你开启了 GTID，并且 GTID 所对应的事务数据还在 binlog 中（没有被 PURGE），Connector 可以用 GTID 直接跳到指定的位点。但前提是——那个 GTID 对应的 binlog 还存在。如果 binlog 已经被清理了...那就只能重做快照。"

**小白**："那 offset 是怎么保持高可用的？如果 connect-offsets Topic 所在的 Kafka Broker 挂了，offset 不就丢了吗？"

**大师**："这就要靠 Kafka 自身的机制——`connect-offsets` Topic 的 `replication factor` 至少设为 3，`min.insync.replicas` 至少设为 2。这样即使一台 Broker 宕机，另外两台 Broker 仍有完整的 offset 副本。Connector 重启时从新的 Leader Broker 读取 offset，完全无感。"

**小白**："但还有一个风险——如果整个 Kafka 集群都挂了怎么办？Connect Worker 内部的 offset 缓存还在吗？"

**大师**："Connect Worker 有自己的内存 offset 缓存，也有定期的 `offset.flush()` 操作。但如果 Kafka 集群完全宕机（3 个 Broker 全挂），Worker 中的未提交 offset 确实会丢失。不过这种情况已经属于'数据中心级故障'了——需要通过跨机房灾备来解决，这是第 31 章的话题。"

**技术映射**：binlog 保留期 = 监控摄像头的录像保存天数。offset = 录像中"看到哪一秒"的书签。如果录像（binlog）被删了，书签（offset）就失去参照物了。

**小胖**："那我们怎么验证 HA 真的可用？总不能等到故障发生才验证吧？"

**大师**："每季度执行一次 **DR Drill（灾难恢复演练）**——在预定的维护窗口内，模拟网络中断 5 分钟 → 自动切换备用 Worker → 自动恢复消费 → 验证数据一致性 → 记录 RPO（Recovery Point Objective，数据丢失多少）和 RTO（Recovery Time Objective，多久恢复）。"

## 3. 项目实战

### 环境准备

```bash
docker ps --filter "name=mysql" --filter "name=connect" --filter "name=kafka"
```

### 步骤1：数据库层防护——binlog 保留策略调整

**目标**：确保 MySQL binlog 至少保留 72 小时，PG 监控 Replication Slot 防 WAL 膨胀。

```bash
# MySQL: 设置 binlog 过期时间为 72 小时（259200 秒）
docker exec mysql mysql -uroot -proot1234 -e "
  SHOW VARIABLES LIKE 'binlog_expire_logs_seconds';
  SET GLOBAL binlog_expire_logs_seconds = 259200;  -- 3 天
  SHOW VARIABLES LIKE 'binlog_expire_logs_seconds';
"
# 预期：259200

# 查看当前 binlog 文件列表
docker exec mysql mysql -uroot -proot1234 -e "SHOW BINARY LOGS;"
# 预期输出：
# +------------------+-----------+-----------+
# | Log_name         | File_size | Encrypted |
# +------------------+-----------+-----------+
# | mysql-bin.000001 |    1234   | No        |
# | mysql-bin.000002 |  567890   | No        |
# ...

# PostgreSQL: 监控 Replication Slot 状态（防 WAL 膨胀）
docker exec postgres psql -U postgres -d user_profile -c "
  SELECT slot_name, active,
         pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS wal_lag
  FROM pg_replication_slots;
"
# 如果 wal_lag > 500MB 且 active = false → 说明 Connector 停止消费 → WAL 堆积风险！
```

### 步骤2：Kafka Connect Worker HA 配置

**目标**：确保内部 Topic 的高可用——replication factor ≥ 3。

```yaml
# docker-compose.yml 中 Connect Worker 的关键配置
environment:
  # 三个内部 Topic 的 HA 配置
  CONNECT_CONFIG_STORAGE_REPLICATION_FACTOR: 3
  CONNECT_OFFSET_STORAGE_REPLICATION_FACTOR: 3
  CONNECT_STATUS_STORAGE_REPLICATION_FACTOR: 3
  
  # Worker 会话超时（不宜太小，防止 GC 误判）
  CONNECT_SESSION_TIMEOUT_MS: 30000
  CONNECT_HEARTBEAT_INTERVAL_MS: 10000
  
  # 新 Worker 加入后的延迟（减少抖动）
  CONNECT_SCHEDULED_REBALANCE_MAX_DELAY_MS: 300000  # 5 分钟
```

```bash
# 验证三个内部 Topic 的复制因子
docker exec kafka kafka-topics --bootstrap-server localhost:9092 --describe --topic connect-configs | head -3
docker exec kafka kafka-topics --bootstrap-server localhost:9092 --describe --topic connect-offsets | head -3
docker exec kafka kafka-topics --bootstrap-server localhost:9092 --describe --topic connect-statuses | head -3
```

### 步骤4：GTID 增量恢复原理实验

**目标**：理解 GTID 在 Connector 恢复中的作用。

```bash
# 查看当前 Connector 的 GTID offset
docker exec kafka kafka-console-consumer --bootstrap-server localhost:9092 \
  --topic connect-offsets --from-beginning --max-messages 5 \
  --property print.key=true 2>/dev/null | grep ha-test

# MySQL 中查看当前 GTID 执行情况
docker exec mysql mysql -uroot -proot1234 -e "SHOW MASTER STATUS\G" | grep -E "File|Position|Executed_Gtid_Set"
# 输出示例：
# File: mysql-bin.000003
# Position: 12345
# Executed_Gtid_Set: f3b3c7e4-1234-5678-9abc-def012345678:1-150

# 恢复原理：如果 offset 丢失，Connector 可以用 Executed_Gtid_Set 作为起点
# 配置 database.initial.statements: "SET @gtid_purged='f3b3c7e4-...:1-150'"
```

### 可能遇到的坑及解决方法

| 坑 | 现象 | 根因 | 解决方法 |
|----|------|------|---------|
| 网络恢复后 Connector 未自动重连 | 状态一直 FAILED | `connect.timeout.ms` 太大或重试次数耗尽 | 手动 POST /connectors/{name}/restart |
| binlog 保留期设了但重启 MySQL 后失效 | `binlog_expire_logs_seconds` 回到默认 | 改了 GLOBAL 但没写入 my.cnf | 在 my.cnf 中永久配置 `binlog_expire_logs_seconds=259200` |
| 多 Worker 环境下 Rebalance 后 Task 未恢复 | Task 显示 UNASSIGNED | Worker 的 Group ID 不一致 | 确保所有 Worker 的 CONNECT_GROUP_ID 相同 |
| PG 的 Slot 堆积导致 WAL 磁盘 100% | PG 实例无法写入 | Connector 停止消费 → Slot 卡住不动 | 删除死 Slot: `SELECT pg_drop_replication_slot('slot_name')` |

## 4. 项目总结

### 优点 & 缺点

| 方案 | RPO（数据丢失） | RTO（恢复时间） | 自动化程度 |
|------|----------------|----------------|-----------|
| 自动重连（短断连） | 0 | < 30s | ✅ 自动 |
| Worker Rebalance | 0 | < 60s | ✅ 自动 |
| GTID 增量恢复 | 0（如果 binlog 还在） | < 5min | ⚠️ 半自动 |
| 全量快照恢复 | > 0（快照期间增量不可用） | 数小时 | ❌ 手动 |

### 适用场景

1. **金融/支付系统**：任何数据丢失或长时间不可用都可能带来财务损失
2. **7×24 业务系统**：电商、游戏、在线教育等不能停机的系统
3. **合规要求**：SOC2/ISO27001 要求系统有经过验证的 DR 方案
4. **跨机房部署**：主备机房自动切换，保障区域级故障的可用性
5. **关键基础设施**：作为企业数据中台的底座，CDC 的可用性直接影响所有下游

### 不适用场景

1. **开发/测试环境**：可接受数分钟到数小时的中断
2. **非核心业务**：如内部报表系统，允许 T+1 恢复

### 注意事项

- **binlog 保留时长必须 ≥ 最大故障恢复时间 × 2**（留出安全缓冲）
- **offset Topic 的 `cleanup.policy` 必须是 `compact`，绝不能是 `delete`**
- **PG 的 Replication Slot 是最隐蔽的炸弹**——Connector 停 1 天，WAL 可能膨胀到 50GB+

### 常见踩坑经验

1. **"我以为 GTID 可以解决一切恢复问题，结果恢复时还是重做了快照"**——GTID 能跳到指定位点的前提是 binlog 还存在。如果 binlog 已被 PURGE，GTID 也无能为力。
2. **"网络恢复后 Connector 一直卡在 RECONNECTING 状态，最后超时"**——根因是 MySQL 的 `wait_timeout` 设得太短（如 60s），Connector 在重连后需要重新认证，认证超时。调大 `wait_timeout` 至少为 300s。
3. **"Worker Rebalance 后 Task 状态成了 UNASSIGNED，等了 10 分钟都没分配"**——根因是 Leader Worker 在 Rebalance 时也宕机了，没有新 Leader 分配 Task。确保至少 3 个 Worker，并且 Leader 选举的 `zookeeper.session.timeout.ms` 合理。

### 思考题

1. MySQL PITR（Point-in-Time Recovery）将数据库恢复到 2 小时前的某个时间点后，Debezium Connector 的 offset 还在 2 小时后——这意味着什么？如何让 Connector 在 PITR 后免于重做全量快照？（提示：利用 GTID 和 `binlog_expire_logs_seconds` 的协调）

2. 一个 Connect Worker 因为 Full GC 导致心跳超时 35 秒。Leader 判定它挂了（`session.timeout.ms=30000`），触发 Rebalance 将 Task 分配给其他 Worker。30 秒后 Full GC 结束，该 Worker 恢复响应——此时集群中出现"两个 Worker 声称负责同一个 Task"的冲突。Kafka Connect 的 Group 协议如何解决这个冲突？消费者端的 offset 会乱吗？

**（第26章思考题答案）**

1. 理论最大 10000 Task/Worker，但实际推荐 < 1000。500 Connector × 2 Task = 1000 Task。建议 3 个 Worker（每个 ~333 Task）可实现 HA（1 个 Worker 宕机后剩余 2 个承载 1000 Task，每 Worker 500 Task 仍在安全区）。VIP 租户需要 99.99% 可用性，至少 3 个 Worker（如跨 3 个可用区部署），确保任一 AZ 故障后仍有 2 个 Worker 可用。

2. 500 个 Connector 意味着三个内部 Topic 各自产生大量消息：`connect-configs`（500 条不变，compact 后无压力）、`connect-offsets`（1000 Task × 定期 flush，压力中等）、`connect-statuses`（1000 Task × 心跳上报，压力最大）。建议：三个 Topic 各设 10 个分区，`connect-statuses` 的 `retention.ms` 设 1 天（状态信息时效性短），`connect-offsets` 必须 `cleanup.policy=compact`。
