# 10-粘包半包与拆包策略

## 1. 项目背景
TCP 是字节流，不保消息边界。发送端连续写两条消息，接收端可能一次收全（粘包），也可能只收一半（半包）。很多业务同学第一次写 Netty 协议时，直接在 `channelRead` 里按“每次读就是一条消息”处理，结果线上出现随机解析失败、JSON 截断、命令串包。这个问题不是偶发 bug，而是 TCP 的基础特性。

Netty 提供了成熟拆包器：`LineBasedFrameDecoder`、`DelimiterBasedFrameDecoder`、`LengthFieldBasedFrameDecoder` 等，源码都在 `codec/src/main/java/io/netty/handler/codec`。关键不在“选一个 decoder”，而在“协议先定义边界”。你只有先确定帧结构（定长、分隔符、长度字段），才能写出稳定链路。

## 2. 项目设计（剧本式交锋对话）
小胖：客户端每次都 `writeAndFlush`，服务端不就该一条条收到吗？  
小白：不保证。TCP 只保证字节顺序，不保证业务消息边界。  
大师：这就是粘包半包本质。技术映射：快递车只负责按顺序送箱子，不管箱里几件货。

小胖：那我用换行分隔最简单，所有协议都这么干行不行？  
小白：文本协议可以，二进制协议用换行容易冲突。  
大师：拆包策略要和协议形态匹配。技术映射：液体、固体、易碎品，包装标准不一样。

小胖：`LengthFieldBasedFrameDecoder` 参数太多，容易配错。  
小白：是，但它最通用。只要明确长度字段偏移、长度和补偿，就能稳定解析。  
大师：参数要写成表格并配样例包。技术映射：地图坐标要标清起点、偏移和目的地。

## 3. 项目实战
### 3.1 环境准备
- JDK 17，Maven 3.9+
- 关键源码：`codec/LengthFieldBasedFrameDecoder.java`
- 示例路径：`example/src/main/java/io/netty/example/factorial`

### 3.2 分步实现
**步骤目标：实现“2 字节长度字段 + 变长消息体”的拆包链路。**

1. 协议定义：前 2 字节无符号长度，后续为 UTF-8 消息体。  
2. pipeline 添加：
   - `new LengthFieldBasedFrameDecoder(65535, 0, 2, 0, 2)`
   - `new LengthFieldPrepender(2)`
   - 自定义业务 handler  
3. 客户端连续发送三条消息（短、长、短），验证不会粘连。  
4. 人为拆分发送（先发长度再延迟发体），验证半包可重组。

命令示例：
```bash
mvn -pl example -DskipTests compile
mvn -pl example exec:java -Dexec.mainClass=io.netty.example.factorial.FactorialServer
```

可编写小客户端连续写入多帧，服务端日志应逐帧解析，且消息边界稳定。

#### 项目实战补强：拆包策略验收与故障注入
拆包正确性必须靠“边界样本 + 故障样本”共同验证，建议固定以下口径：

1. **边界样本**：空消息、最小消息、最大合法消息、接近最大长度消息。  
2. **故障样本**：长度字段伪造、长度大于上限、分隔符缺失、乱码混入。  
3. **网络样本**：随机拆分发送、随机合并发送、抖动延迟发送。  
4. **结果口径**：合法帧全部解析成功，非法帧可预期失败并正确关闭或隔离连接。

可执行验证建议（示例）：

命令示例：
```bash
python .\bench\frame_fuzzer.py --mode mixed --rounds 3000 --host 127.0.0.1 --port 8007
```

运行后应输出三项核心指标：解析成功率、协议异常率、连接恢复时间。

#### 常见坑
- **长度字段偏移配错**：解析错位，后续全部报文报错。  
- **最大帧长设置过小**：合法大包被拒绝触发 `TooLongFrameException`。  
- **忽略异常清理**：拆包失败后连接不关闭，状态污染。  
- **文本与二进制策略混用**：分隔符在二进制体中误命中。

#### 故障案例（协议演进期高发）
- **案例 A：升级后旧客户端大量报错**  
  根因：新协议引入长度字段，旧客户端仍按分隔符发送，服务端未做兼容分流。  
  修复：握手阶段先识别版本，再路由到对应 decoder。  

- **案例 B：偶发超大包触发连锁断开**  
  根因：最大帧长配置过小且异常策略设置为全局断链，导致正常流量被牵连。  
  修复：区分单连接异常与全局异常，按连接粒度隔离处理。

## 4. 项目总结
粘包半包不是“Netty 偶发问题”，而是 TCP 世界的常态。正确姿势是：先协议建模，再 decoder 落地，再用命令和压测反复验证边界。只要拆包策略正确，业务处理层就能稳定拿到“完整消息帧”，不会再陷入随机解析异常。

推荐团队把拆包参数文档化：帧格式图、字段偏移、最大长度、异常策略。每次协议演进都要更新文档和回归样例，这比上线后追查“为什么某次消息被截断”便宜得多。

补充一条实践建议：协议变更发布必须附带“新旧版本互通报告”，明确哪些客户端组合可兼容、哪些会被拒绝，以及拒绝时的错误码与提示语。这能显著降低升级窗口的灰度风险。

另外，建议把拆包异常纳入运维值班手册：当出现 `TooLongFrameException`、帧错位、解码失败暴增时，分别对应哪些排查步骤、回滚动作和临时限流策略。把处置流程标准化后，协议问题就不会在夜间发布时演变成大面积故障。

团队还可以定期做一次“协议兼容演练”：随机抽取旧版本客户端与新服务端组合进行互通测试，提前识别灰度窗口里的边界冲突。把这类演练固化为月度动作，比事后救火更省成本。

实践证明，提前做兼容演练能明显缩短发布窗口。
