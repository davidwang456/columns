# 附录-思考题答案索引

## 第01章

题目1答案：连接数代表“潜在并发”，CPU 消耗更受“活跃连接比例”和“单次事件处理成本”影响。大量空闲长连接在事件驱动模型下几乎不占执行时间。

题目2答案：优先在业务线程池隔离阻塞调用，EventLoop 只负责网络事件与轻量编排，避免同 loop 的其他连接被拖慢。

## 第05章

题目1答案：`ctx.writeAndFlush` 从当前 handler 节点向前传播出站事件；`channel.writeAndFlush` 通常从 pipeline 尾部出发。两者命中的出站 handler 集合可能不同。

题目2答案：热插拔要在 EventLoop 线程内执行（或提交到对应 EventLoop），并在边界节点统一变更，避免并发修改引起顺序错乱。

## 第15章

题目1答案：可在路由层先按 URI 或 Header 分流，小包接口走聚合 pipeline，大文件接口走流式 pipeline，避免一套策略覆盖全部流量。

题目2答案：chunked 传输下应谨慎使用全量聚合，优先流式消费；若必须聚合，要设置更严格阈值、超时和限流策略。

## 第29章

题目1答案：将阻塞或重 CPU 任务下沉到独立线程池，并通过异步回调回到 EventLoop；同时控制单任务执行时间与队列长度。

题目2答案：先查未完成任务队列、未完成 Promise、周期任务是否持续重投递，再看是否有阻塞调用卡住 EventLoop 线程。
