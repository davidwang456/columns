# 07-Future与Promise异步编排

## 1. 项目背景
同步式代码写起来直观，但在高并发网络服务里代价极高：一旦线程阻塞等待下游结果，IO 线程就被占住，吞吐和延迟一起崩。Netty 选择用 `Future` / `Promise` 描述异步结果，把“发起动作”和“处理结果”拆开。问题是，很多团队只会 `sync()`，等于把异步 API 又用回同步风格，性能优势直接丢掉。

在 `transport/src/main/java/io/netty/util/concurrent` 下，`Future`、`Promise`、`DefaultPromise`、`GenericFutureListener` 构成了 Netty 异步编排核心。官方示例里几乎每个 bind、connect、write 都返回 `ChannelFuture`。真正关键在于：你如何用监听器、聚合器和超时机制，把多步网络动作编排成可失败、可取消、可追踪的流程。

## 2. 项目设计（剧本式交锋对话）
小胖：我最省事的方法是每次 `writeAndFlush(...).sync()`，这样保证成功再往下走。  
小白：这会阻塞线程，尤其在 EventLoop 上调用会直接拖垮整条链。  
大师：异步编排的第一原则：不要在 IO 线程阻塞等待。技术映射：点餐后站柜台等全部菜会堵住后面所有人。

小胖：那不用 `sync()` 怎么知道写成功没？  
小白：加 `addListener`，在回调里处理成功/失败分支。  
大师：对，而且要把失败路径写完整，不然只会“成功日志很多，失败都失踪”。技术映射：外卖到没到看通知，不是每分钟打电话催。

小胖：多个异步步骤要串起来怎么办？  
小白：可以先 A 成功再发 B，或用 `PromiseCombiner` 聚合多任务。  
大师：并行、串行、超时、取消都要设计。技术映射：旅行不是只买一张票，要同时盯机票、酒店、接驳车。

## 3. 项目实战
### 3.1 环境准备
- JDK 17，Maven 3.9+
- 关键源码：`util/concurrent/DefaultPromise.java`
- 示例路径：`example/src/main/java/io/netty/example/http/snoop`

### 3.2 分步实现
**步骤目标：实现“异步写入 + 失败回滚 + 聚合完成通知”。**

1. 发起两次 `writeAndFlush`，分别拿到 `ChannelFuture f1/f2`。  
2. 使用监听器处理结果：成功计数，失败记录异常并关闭连接。  
3. 用 `PromiseCombiner` 聚合 `f1/f2` 到一个 `Promise<Void>`，仅在全部成功时标记完成。  
4. 对聚合 Promise 增加超时保护（定时任务），超时则 fail 并回收上下文。

命令示例：
```bash
mvn -pl example -DskipTests compile
mvn -pl example exec:java -Dexec.mainClass=io.netty.example.http.snoop.HttpSnoopServer
curl -v http://127.0.0.1:8080/
```

可通过模拟下游异常（手工抛错）验证失败回调是否命中。

#### 项目实战加强：编排模板与验证口径
为了避免“功能可用但异常不可控”，建议把 Future/Promise 编排沉淀为统一模板：

1. **发起阶段**：记录请求 ID、开始时间、调用链标签。  
2. **回调阶段**：成功路径只做轻逻辑，重逻辑转移到业务线程池。  
3. **失败阶段**：统一映射异常码，明确是否重试、是否关闭连接。  
4. **结束阶段**：输出耗时、结果状态、失败原因，便于事后审计。

可执行验证命令建议：

命令示例：
```bash
mvn -pl example -Dtest=PromiseOrchestrationTest test
```

验收口径建议固定三条：

- 串行编排失败时，后续步骤不得继续执行；  
- 并行编排任一失败时，聚合结果必须在阈值时间内完成失败返回；  
- 超时取消后，不得再出现“晚到成功回调改写状态”的竞态。

#### 常见坑
- **在 EventLoop 里 `sync()`**：阻塞 IO，导致级联超时。  
- **只处理成功不处理失败**：异常静默，问题延后爆炸。  
- **监听器里做重活**：回调执行线程被拖慢。  
- **未做超时和取消**：请求悬挂，占资源不释放。

#### 故障案例（异步链路最常见）
- **案例 A：线上线程池被打满**  
  根因：监听器里直接做 JSON 解析和 DB 写入，导致 EventLoop 回调长时间占用。  
  修复：监听器只做状态流转，重逻辑 `submit` 到业务执行器。  

- **案例 B：请求状态“成功后又失败”**  
  根因：超时后未取消下游回调，晚到结果再次写状态。  
  修复：引入幂等状态机（INIT/SUCCESS/FAILED/TIMEOUT），只允许单向终态迁移。

## 4. 项目总结
Future/Promise 让 Netty 具备“高并发下仍可组合”的能力，但前提是你按异步范式写代码：监听器驱动、失败优先、超时可控。只要你回到 `sync()` 思维，系统马上退化为阻塞模型。工程上，建议把常用编排模式抽成工具：串行链、并行聚合、超时包装、统一异常映射。

此外要注意线程语义：回调通常运行在触发该 Future 的线程上，重逻辑应转移到业务执行器。这样才能既保住 IO 延迟，又保住业务复杂度的可维护性。

在项目管理层面，建议把“异步失败路径覆盖率”纳入发布门槛：不仅测成功链路，还要测超时、取消、重试、并发失败合并。只要失败路径可预测，异步系统才真正可运营。

实践中还可增加“编排观测看板”：按接口维度展示成功率、超时率、取消率与平均回调耗时，并按版本对比。这样当某次发布引入异步回归时，你能在分钟级定位是编排设计问题、线程池问题，还是下游依赖问题。

这也是异步工程化的关键一环。
