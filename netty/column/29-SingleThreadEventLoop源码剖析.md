# 29-SingleThreadEventLoop源码剖析

## 1. 项目背景

某实时推送服务在峰值期间出现“平均延迟正常但 P99 突刺”的问题。监控显示网络带宽没有打满、GC 也不高，但某些连接会突然慢 1-2 秒。团队最初怀疑是内核网络抖动，后来通过线程栈发现：事件循环线程上堆积了大量任务，执行节奏被打乱。这个问题在业务扩展后很常见，因为很多人理解了“Netty 是多线程”，却没真正理解“一个 Channel 绑定一个 EventLoop，且 EventLoop 常常是单线程执行事件与任务”。

高级篇必须讲清楚这个事实：SingleThreadEventLoop 不只是“一个线程读写 socket”，它还承担任务队列、定时任务、关闭流程协调等职责。理解其 run loop 行为，才能解释为什么阻塞调用会引发全链路长尾，为什么某些定时任务会延后触发，为什么优雅停机要给充足窗口。

## 2. 项目设计（剧本式交锋对话）

| 角色 | 性格标签 | 职责 | 话风示例 |
| --- | --- | --- | --- |
| 小胖 | 爱吃爱玩、不求甚解 | 把复杂问题转成直观疑问 | “一个线程要干这么多活，不会堵车吗？” |
| 小白 | 喜静、喜深入 | 追问源码细节 | “任务队列和 IO 事件谁优先？定时任务怎么插队？” |
| 大师 | 资深技术 Leader | 串联源码与故障现象 | “EventLoop 像机场塔台，调度失衡就会排队。” |

### 第一轮

小胖：既然是异步框架，为啥一个 loop 还是单线程？  
小白：单线程是不是性能瓶颈？  
大师：单线程是为了避免同一 Channel 并发访问导致锁竞争和状态错乱。它用“串行执行”换“线程安全与低开销”。

技术映射：单航道按序放行，减少冲突开销。

### 第二轮

小胖：那任务很多时，IO 会不会饿死？  
小白：`run()` 循环里是先取任务还是先处理 IO？  
大师：不同实现细节略有差异，但本质是“在 IO 事件和任务队列间做时间片平衡”。若你在 loop 里塞阻塞任务，任何平衡策略都救不了。

技术映射：塔台调度再好，也扛不住跑道被一架飞机长时间占用。

### 第三轮

小胖：优雅关机不就是停线程吗？  
小白：那在飞的任务和未刷盘的数据怎么办？  
大师：关机是状态机，不是 `stop`。要经历拒绝新任务、处理存量任务、触发关闭钩子、确认退出。

技术映射：机场停运要先清空航班，不是直接关灯。

## 3. 项目实战

### 3.1 环境准备

- 核心源码入口：`transport/src/main/java/io/netty/channel/DefaultEventLoop.java`
- 建议联读：`SingleThreadEventLoop`、`SingleThreadEventExecutor`
- 示例项目：`example` 任意 TCP 服务均可用于复现实验

### 3.2 分步实现

**步骤目标 1：阅读并标注 run loop。**

以 `DefaultEventLoop` 为起点，关注三件事：
1. `takeTask()` 如何取任务  
2. `runTask(task)` 如何执行  
3. `confirmShutdown()` 如何决定退出

**步骤目标 2：构造阻塞任务复现长尾。**

在业务 handler 中加入 300ms 阻塞，压测并观察：
- 同 EventLoop 绑定的连接延迟同时上升  
- 不同 EventLoop 的影响相对独立

**步骤目标 3：验证关机语义。**

调用 `shutdownGracefully`，记录从发起到终止的时间与未完成任务数量，确认是否符合预期窗口。

可能遇到的坑与解决：
- 误把 CPU 密集任务放 EventLoop：迁移到独立业务线程池。  
- 定时任务漂移：检查是否被长任务阻塞。  
- 关机超时：排查未完成 Promise 或循环重试任务。

### 3.3 完整代码清单

- `transport/src/main/java/io/netty/channel/DefaultEventLoop.java`
- `transport` 下 `SingleThreadEventLoop` / `SingleThreadEventExecutor` 相关类
- 示例服务：`example/src/main/java/io/netty/example/echo/EchoServer.java`

### 3.4 测试验证

- 压测前后对比 P95/P99。  
- 记录 EventLoop 任务队列长度变化。  
- 执行优雅关机，验证连接 drain 和任务收敛时间。

补充一个更贴近生产的验证动作：在压测中途注入一次“短时阻塞 + 关机指令”组合场景。先让某个 handler 人为阻塞 200~300ms 持续 10 秒，再立即触发 `shutdownGracefully()`，同时采集 `pendingTasks`、连接关闭耗时、未完成任务数三组指标。若模型理解正确，你会看到阻塞窗口内同 loop 连接延迟同步抬升，但进入关机阶段后队列应逐步回落且不会无限增长。这组组合实验能同时验证 run loop 调度、公平性边界与关机状态机行为，比只做单点压测更能暴露真实风险。

#### 验证口径

- [ ] **功能**：核心用例可复现，关键输入输出与预期一致。
- [ ] **稳定性**：连续压测或重复执行无异常抖动、无明显长尾退化。
- [ ] **可观测性**：日志、指标与关键错误信号可定位并支持问题回溯。

## 4. 项目总结

### 4.1 优点&缺点

| 维度 | 优点 | 缺点 |
| --- | --- | --- |
| 线程安全 | 同 Channel 串行执行，锁少 | 误用阻塞任务会放大影响 |
| 可预测性 | 事件顺序稳定，易推理 | 长任务导致队列积压 |
| 运维可控 | 关机语义清晰可治理 | 需要额外监控队列与长任务 |

### 4.2 适用场景

- 高并发长连接系统
- 需要严格事件顺序的协议处理
- 可把重任务外移的网关系统

不适用场景：
- 大量阻塞 IO 难以改造的旧业务
- 线程模型不受控的插件生态系统

### 4.3 注意事项

- 事件循环线程只做轻量、短时任务。
- 所有潜在阻塞点必须可观测。
- 优雅关机要有超时和降级策略。

### 4.4 常见踩坑经验

1. 把数据库调用直接写进 handler，造成整批连接延迟尖刺。  
2. 定时任务与普通任务混跑，长任务导致心跳超时误判。  
3. 关机窗口过短，消息未刷出就被强停。
