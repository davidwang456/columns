# 01-从连接风暴到事件驱动

## 1. 项目背景

你所在团队要做一个“物流轨迹推送网关”：仓库扫码枪、司机 App、商家后台都要通过 TCP 长连接实时收发状态。试点阶段只有 500 台设备，传统“一个连接一个线程”的 BIO 服务跑得还行；一旦业务扩到全国，连接数从 500 涨到 3 万，CPU 开始飙高，线程栈内存吃满，Full GC 频率上升，响应延迟从 20ms 拉长到 2s 以上。最痛苦的是，问题不稳定：白天高峰慢，夜间恢复，日志里看起来没有明显报错。

这类系统的核心矛盾不是“业务逻辑复杂”，而是“连接数量巨大且生命周期很长”。如果继续沿用线程阻塞模型，运维会被线程数、上下文切换、内核调度拖死，开发会被回调地狱与异常吞没困住，测试会发现同一用例在小并发稳定、上压测就随机失败。Netty 解决的正是这个维度：它不是一个“HTTP 框架”，而是异步事件驱动网络编程底座，用少量线程处理大量连接，再通过 pipeline 把协议与业务解耦。

本章目标不是上来就讲源码细节，而是先建立一套统一认知：为什么要从阻塞模型迁移到事件驱动模型；Netty 在系统里承担什么角色；后续章节为什么都围绕 EventLoop、Pipeline、ByteBuf 展开。

## 2. 项目设计（剧本式交锋对话）

| 角色 | 性格标签 | 职责 | 话风示例 |
| --- | --- | --- | --- |
| 小胖 | 爱吃爱玩、不求甚解 | 用生活化比喻抛问题 | “这不就像食堂排队嘛，干嘛搞这么多线程池？” |
| 小白 | 喜静、喜深入 | 追问原理、边界和风险 | “那队头阻塞时后面的连接怎么保证不被拖死？” |
| 大师 | 资深技术 Leader | 讲透约束与选型 | “把 EventLoop 想成地铁调度台，不是每位乘客配一名司机。” |

### 第一轮：为什么 BIO 会在连接风暴里失效

小胖：我们以前 Java Socket 一连接一线程，代码很直观，读写阻塞就阻塞呗，业务照样跑。  
小白：直观不代表可扩展。3 万连接是不是就 3 万线程？线程上下文切换和栈内存怎么扛？  
大师：核心是“并发连接数”和“活跃连接数”不一致。很多连接大部分时间在等数据，如果每个都占着线程，就像每个乘客都包一辆出租车，城市肯定堵死。

技术映射：**阻塞 IO = 连接绑定线程；事件驱动 IO = 事件绑定线程**。

### 第二轮：事件驱动到底解决了什么

小胖：那 Netty 是不是魔法？  
小白：我更关心它如何让少量线程服务大量连接。  
大师：不是魔法，是复用。内核先告诉你“哪些连接可读可写”，用户态线程只处理就绪事件。Netty 再把这个模型工程化，给你 Channel、Pipeline、Future、ByteBuf 这套组合件。

技术映射：**Selector/epoll 负责“发现可处理事件”，EventLoop 负责“执行事件回调”**。

### 第三轮：迁移策略如何落地

小胖：听起来得重写全部网络层？  
小白：我们能不能先替换接入层，不动核心业务？  
大师：可以。第一阶段把“连接接入 + 协议编解码”迁到 Netty，业务逻辑先沿用；第二阶段再把同步调用拆成异步链路。这样风险最小，收益最快。

技术映射：**分层迁移 = 先替换 IO 模型，再治理业务线程与异步编排**。

## 3. 项目实战

### 3.1 环境准备

- JDK 17（建议）
- Maven 3.9+
- Windows / Linux 均可
- 参考本仓库示例：`example/src/main/java/io/netty/example/echo`

最小依赖示例（独立小项目可用）：

```xml
<dependency>
  <groupId>io.netty</groupId>
  <artifactId>netty-all</artifactId>
  <version>4.1.108.Final</version>
</dependency>
```

### 3.2 分步实现

**步骤目标 1：跑通最小 Echo 服务，验证事件驱动模型可用。**

使用仓库中的 `EchoServer` / `EchoClient`，先不改源码，直接运行并建立连接。

命令示例：
```bash
mvn -pl example -DskipTests compile
mvn -pl example -DskipTests exec:java -Dexec.mainClass=io.netty.example.echo.EchoServer
```

预期输出（文字描述）：服务端打印启动日志，客户端发送字符串后立即回显。

可能踩坑与解决：
- 端口被占用：改 `-Dport=9001`。
- Windows 防火墙拦截：临时放行 Java 进程或改本地回环测试。

**步骤目标 2：模拟阻塞业务，观察线程模型差异。**

在业务 handler 中临时加 `Thread.sleep(200)`，压测后观察延迟。你会发现 EventLoop 上阻塞会影响该线程绑定的一组 Channel，这就是后续“线程隔离章节”要解决的问题。

**步骤目标 3：定位代码结构，建立后续阅读地图。**

- `transport`：网络事件主干
- `buffer`：内存与字节容器
- `handler`：可插拔能力
- `codec-*`：协议编解码

### 3.3 完整代码清单

- 现成示例入口：`example/src/main/java/io/netty/example/echo/EchoServer.java`
- 现成示例客户端：`example/src/main/java/io/netty/example/echo/EchoClient.java`
- 本章建议不新增代码，重点是跑通与观察。

### 3.4 测试验证

**功能验证**
- 启动服务端后，客户端发送 100 条短消息，回包内容应一致。

**稳定性验证**
- 并发 500 连接、每连接每秒 5 条消息，观察 CPU 和内存曲线应平稳。

**验证口径**
- 若出现明显长尾，检查是否在 IO 线程做了阻塞操作（sleep、DB、HTTP 调用）。

#### 验证口径

- [ ] **功能**：核心用例可复现，关键输入输出与预期一致。
- [ ] **稳定性**：连续压测或重复执行无异常抖动、无明显长尾退化。
- [ ] **可观测性**：日志、指标与关键错误信号可定位并支持问题回溯。

## 4. 项目总结

### 4.1 优点&缺点

| 维度 | Netty 事件驱动 | 传统阻塞模型 |
| --- | --- | --- |
| 连接扩展性 | 强，少线程可支撑大量连接 | 弱，线程数随连接增长 |
| 延迟稳定性 | 在无阻塞前提下较稳 | 受线程竞争影响大 |
| 开发心智 | 需理解异步与事件流 | 上手直观 |
| 故障排查 | 需掌握 pipeline/线程边界 | 调试路径短但瓶颈明显 |

### 4.2 适用场景

- 长连接网关（IM、车联网、IoT）
- 高并发二进制协议服务
- 需要自定义协议编解码的接入层
- 代理、隧道、推送系统

不适用场景：
- 低并发后台管理系统
- 团队暂时无法接受异步编程范式改造

### 4.3 注意事项

- 不要在 事件循环线程做阻塞调用。
- 先统一协议边界，再谈业务并发优化。
- 版本升级优先参考仓库模块关系，避免一次引入过多实验特性。

### 4.4 常见踩坑经验

1. **连接数上升但吞吐不涨**：根因是业务 handler 阻塞 IO 线程。  
2. **偶发内存抖动**：根因是 ByteBuf 使用不当或对象分配过多。  
3. **压测和线上表现不一致**：根因是线上有慢客户端与网络抖动，压测环境过于理想。
