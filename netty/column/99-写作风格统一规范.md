# 99-写作风格统一规范

面向本专栏 `01-38` 章节（重点 `02-38`）的统一写作规则。目标只有两个：**术语不混用**、**实战节可复用**。

## 1. 术语统一词典

### 1.1 强制替换规则（写作前先自查）

- 固定写 `事件传播`，禁用 `链路传播`、`事件传递`（泛指场景）。
- 固定写 `责任链`，禁用 `处理链`、`调用链`（指 `ChannelPipeline` 时）。
- 固定写 `入站/出站`，禁用 `上行/下行`、`输入/输出方向`（指 pipeline 事件方向时）。
- 固定写 `阻塞调用`，禁用 `同步阻塞`、`卡线程`（正文主叙述用词）。
- 固定写 `线程隔离`，禁用 `线程池隔离`、`业务池隔离`（章节正文统一简称）。
- 固定写 `异步编排`，禁用 `异步链式调用`、`回调编排`（`Future/Promise` 场景）。
- 固定写 `连接生命周期`，禁用 `连接状态流`、`连接阶段流转`（章节主叙述）。
- 固定写 `引用计数`，禁用 `引用数`、`计数引用`。
- 固定写 `泄漏检测`，禁用 `内存泄漏检测`（标题内可简写）。
- 固定写 `拆包`，禁用 `分包解析`、`半包处理`（统称时）。
- 固定写 `心跳保活`，禁用 `保活心跳`、`长连接保活`（统称时）。
- 固定写 `背压`，禁用 `反压`（全专栏统一为 `背压`）。
- 固定写 `优雅关机`，禁用 `平滑停机`、`平滑下线`（章节主题词）。
- 固定写 `零拷贝`，禁用 `零复制`。

### 1.2 API/类名写法规则

- 类名、方法名、配置项一律保留官方大小写：如 `EventLoop`、`ChannelPipeline`、`writeAndFlush`、`AUTO_READ`。
- 中文段落首次出现术语时，使用“中文 + 英文标识”格式：例如 `线程隔离（EventExecutorGroup）`；同段后续可只用中文。
- `HTTP/2`、`HTTP/3` 统一写法带斜杠；文件名可沿用已有命名，不反向修改历史文件。
- `NIO`/`epoll`/`kqueue` 并列时统一格式：`NIO、epoll、kqueue`。

### 1.3 易混词对照（出现即改）

- `channel.writeAndFlush`（禁）-> `ctx.writeAndFlush`（默认推荐，除非明确要从尾节点发起）
- `fireRead`（禁）-> `fireChannelRead`
- `写刷`、`刷写`（禁）-> `writeAndFlush`
- `粘包/半包问题`（可）-> 统一章节内先写 `粘包半包`，后文可简写 `拆包问题`

## 2. 实战节格式规范（强制模板）

所有章节 `## 3. 项目实战` 固定为 4 个小节，不增删编号：

### 3.1 环境准备

固定包含 3 类信息，顺序不可变：

1. 运行环境（JDK/Maven/Netty 版本）
2. 示例或演示路径（`example/...`）
3. 核心源码路径（2-3 个，避免堆列表）

### 3.2 分步实现

固定结构：

1. `步骤目标：...`（一句话）
2. 4-6 条编号步骤（每条只表达一个动作）
3. `命令块-构建与启动`
4. `命令块-交互或压测验证`
5. `预期结果：...`（必须可观测，禁止“应正常”）
6. `常见坑：`（2-4 条，和本章主题强相关）

### 3.3 完整代码清单

固定 3-6 条路径，必须是“本章实际会看的文件”，禁止泛化占位（如 `server/handler/*`）：

- 先列示例入口
- 再列核心源码
- 最后列辅助文件（可选）

### 3.4 测试验证

固定四段，不得缺项：

1. `功能验证`：输入、预期输出、失败判据
2. `稳定性验证`：并发/时长/观测指标
3. `故障注入（可选）`：触发方式与预期恢复
4. `验收清单`：统一 checklist 语法

验收清单必须使用如下格式：

```md
- [ ] 功能结果符合预期（给出关键日志或回包特征）
- [ ] 性能指标未劣化（给出阈值，如 P99、CPU、pendingTasks）
- [ ] 异常路径可收敛（超时/异常/断连后有明确收口）
```

## 3. 命令块与输出描述规范

### 3.1 命令块标题（统一命名）

命令块前必须有标题行，限定以下四类：

- `命令块-构建`
- `命令块-启动`
- `命令块-功能验证`
- `命令块-压测/回归`

不要使用：`命令行示例`、`运行命令`、`可执行验证建议`（改为上面四类之一）。

### 3.2 输出描述写法

- 使用 `预期结果：` 起句，后接“可直接观察”的结果。
- 禁用模糊词：`正常`、`无异常`、`看起来可以`。
- 推荐模板：
  - `预期结果：服务端日志出现 <关键字>，客户端收到 <响应特征>。`
  - `预期结果：压测 5 分钟内 P99 <= <阈值>，无 <关键错误>。`

## 4. 章节内一致性约束（针对当前专栏）

- `02-38` 章统一保留四段主结构：背景 -> 剧本对话 -> 项目实战 -> 项目总结。
- 若新增扩展段，统一放在 `## 4. 项目总结` 下，命名为 `### 补充实战手册` 或 `### 补充复盘案例`，不要新增其他同级命名。
- `3.4 测试验证` 禁止出现模板占位符（例如 `{cfg['tests']}`）；必须写成可执行内容。
- 每章至少出现一次“边界条件或失败路径”的可验证描述，避免只有 happy path。

## 5. 执行方式（作者自检 1 分钟）

发稿前逐章检查：

1. 先扫术语：是否出现禁用词（按 1.1/1.3 替换）。
2. 再扫实战：`3.1-3.4` 是否完整且顺序正确。
3. 再扫命令块：标题是否为四种标准名之一。
4. 最后扫验收：是否有 checklist，且包含“功能/性能/异常收敛”三项。

未通过任一项，不进入下一章写作。
