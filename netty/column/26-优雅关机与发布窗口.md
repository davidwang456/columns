# 26-优雅关机与发布窗口

## 1. 项目背景

粗暴停机会导致在途请求丢失和重试风暴。优雅关机要求摘流、drain、超时兜底、资源回收一体化。Netty 的 `shutdownGracefully()` 是基础动作，但不是完整流程。

## 2. 项目设计（剧本式交锋对话）

| 角色 | 定位 |
| --- | --- |
| 小胖 | 业务视角提出诉求与约束 |
| 小白 | 工程实现与落地执行 |
| 大师 | 架构把关与取舍决策 |

### 第一轮

小胖：发布不就是重启进程？

小白：在途请求会被中断。

大师：发布是流量迁移过程，不是简单重启。

### 第二轮

小胖：先停监听端口行吗？

小白：已有连接还在，怎么办？

大师：先摘流，再 drain，在途处理完再关。

### 第三轮

小胖：排不空怎么办？

小白：无限等会拖慢发布。

大师：设置硬超时和强制关闭兜底。

## 3. 项目实战

### 3.1 环境准备

- 源码锚点：`common/src/main/java/io/netty/util/concurrent/SingleThreadEventExecutor.java`
- 示例锚点：`example/src/main/java/io/netty/example/echo/EchoServer.java`

命令示例：
```bash
mvn -DskipTests compile
```

### 3.2 分步实现

1) 从 LB/注册中心摘流；
2) 标记 draining，拒绝新请求；
3) 调用 `shutdownGracefully()`；
4) 超时后强制关闭并记录残留请求。

预期输出：发布期间成功率平稳，无大规模连接重置。

### 3.3 完整代码清单

- 各 `example` 服务启动与关闭逻辑
- 运维编排脚本（如 K8s preStop）配套验证

### 3.4 测试验证

滚动发布压测，观察成功率、RST 比例、单实例退出时长。

#### 验证口径

- [ ] **功能**：核心用例可复现，关键输入输出与预期一致。
- [ ] **稳定性**：连续压测或重复执行无异常抖动、无明显长尾退化。
- [ ] **可观测性**：日志、指标与关键错误信号可定位并支持问题回溯。

## 4. 项目总结

优雅关机保障发布稳定。优点：降低中断、支持灰度和回滚。缺点：流程复杂，需要开发运维测试协同。常见踩坑：未摘流先停机、无硬超时、只看进程不看请求完成率。


优雅关机能否成功，关键在“是否把状态从代码层传递到流量层”。很多团队只在进程内调用 `shutdownGracefully()`，却忘了先从负载均衡摘流，结果新请求仍持续进入，drain 永远结束不了。推荐采用五阶段流程：准备、摘流、drain、超时兜底、验收。每阶段都要有可观测指标和超时阈值。

实操上建议增加发布状态机：`RUNNING -> DRAINING -> STOPPING -> STOPPED`。当进入 `DRAINING` 时，健康检查改为“不可接新流量但仍存活”，并对新请求返回可重试语义（如 503 + Retry-After）。这样上游可快速切换实例而不触发错误重试风暴。对于长连接业务，还应广播“服务即将重启”信号，促使客户端提前重连到新实例。

本仓库 `example` 模块的服务样例可用于验证关闭行为。建议在压测环境执行滚动发布：
1) 先摘掉 10% 实例；
2) 观察成功率、RST 比例、平均退出时长；
3) 逐步扩大到 30%、50%、100%。
预期输出：在途请求完成率高，连接重置率低，单实例退出时间稳定在窗口内。

命令参考：
`mvn -pl example -DskipTests compile`
配合你的部署脚本执行分批重启。

踩坑补充：
1) preStop 时间小于应用 drain 时间，导致系统提前杀进程；
2) 关闭顺序错误，先关 worker 后关 boss 导致异常噪声；
3) 无法区分“主动关闭”与“故障退出”，误触发告警。

协作建议：开发提供发布状态机与指标，运维固化摘流与回滚脚本，测试演练正常发布与中途回滚。每次大版本前做一次“强制中断演练”，确保兜底路径在最坏情况下可控。


### 补充实战手册

围绕优雅关机发布窗口主题，建议团队在预发布环境执行一轮完整走查。走查不应只看功能通过，还要验证失败路径、回退路径、告警路径是否同样可用。下面给出可直接复用的执行模板：

- 步骤A：准备阶段，确认配置项、证书、白名单、超时参数、线程池参数与线上目标值一致。
- 步骤B：冒烟阶段，先跑单实例验证，记录首次成功耗时与失败错误码分布。
- 步骤C：压力阶段，逐步提高并发，观察 p50/p95/p99、错误率、连接数、队列深度。
- 步骤D：故障注入，注入延迟、丢包、异常响应、依赖不可达，检查系统是否按预期降级。
- 步骤E：回退演练，强制回退到上一版本，验证配置与协议兼容性。

排障顺序建议遵循“先链路、再线程、后业务”的原则：

3) 业务：错误码分布、降级命中、重试行为。

补充踩坑清单：
- 坑1：只验证成功路径，忽略异常路径，导致上线后第一波异常直接打穿。
- 坑2：没有统一日志字段，跨层排障时无法串联同一请求。
- 坑3：告警阈值照搬历史值，业务规模变化后出现大量误报或漏报。
- 坑4：回退脚本未演练，真实故障时“会回退”变成“回不去”。

团队协作建议：开发负责实现与埋点，测试负责场景覆盖和回归基线，运维负责容量与发布节奏。每次变更后进行 30 分钟复盘，沉淀参数变化、故障样本、修复动作，为下一次发布建立可复制经验。


### 补充复盘案例

案例一：高峰期突发抖动。现象是请求成功率先下降后恢复，但 p99 长时间居高不下。复盘发现并非单点故障，而是参数组合在高并发下触发了连锁效应。处理策略是先冻结变更、保留现场日志、按时间线回放关键指标，再逐项恢复配置并观察回归。

案例二：灰度阶段指标正常，全量后异常。根因通常是流量结构变化导致边界条件被放大。建议灰度不仅按流量比例，还要按请求类型和客户端版本分层抽样。

案例三：回退后问题仍在。说明故障可能不在版本本身，而在运行时状态、缓存、连接或外部依赖。回退策略应包含状态清理步骤，并验证依赖侧是否同步恢复。

执行建议：每次演练后沉淀“触发条件-检测信号-处置动作-恢复判据”四元组，形成章节主题的长期知识库，避免同类问题重复踩坑。
