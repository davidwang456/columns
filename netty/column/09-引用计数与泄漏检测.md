# 09-引用计数与泄漏检测

## 1. 项目背景
Netty 用直接内存和池化换性能，但这也带来一个现实问题：JVM GC 不再自动兜底所有内存。`ByteBuf`、`ByteBufHolder` 等对象普遍实现 `ReferenceCounted`，依赖引用计数控制释放。一旦你多 retain 少 release，内存会悄悄上涨；一旦你提前 release，又会触发非法访问。引用计数相关故障往往不是立刻爆炸，而是上线几天后才显现，最难定位。

为此 Netty 提供了 `ResourceLeakDetector`，源码在 `common/src/main/java/io/netty/util/ResourceLeakDetector.java`。它通过采样跟踪对象分配与访问轨迹，在泄漏时输出创建栈。示例代码里常见 `ReferenceCountUtil.release(msg)` 就是基本防线。掌握“谁拥有释放责任”，是写稳定 Netty 代码的硬门槛。

## 2. 项目设计（剧本式交锋对话）
小胖：我看框架很多地方会自动释放，那我是不是不用管 `release()` 了？  
小白：只有遵循特定 handler 约定时才会自动释放，自定义转发很容易打破这个假设。  
大师：引用计数必须有明确 owner。技术映射：公用工具没人负责归还，迟早丢。

小胖：那我每个地方都 `release()` 一次，肯定安全吧？  
小白：会过度释放，后续访问直接报 `IllegalReferenceCountException`。  
大师：多释放和不释放一样危险。技术映射：账本不能“宁可多扣”，必须精准对账。

小胖：泄漏检测开到最严会不会拖慢服务？  
小白：`PARANOID` 开销高，适合测试和灰度，不建议全量长期开。  
大师：环境分级：开发 `PARANOID`，预发 `ADVANCED`，生产 `SIMPLE` 或按窗口排查。技术映射：安检强度随场景切换。

## 3. 项目实战
### 3.1 环境准备
- JDK 17，Maven 3.9+
- 关键源码：`util/ReferenceCountUtil.java`、`util/ResourceLeakDetector.java`
- 示例路径：`example/src/main/java/io/netty/example/echo`

### 3.2 分步实现
**步骤目标：复现实例泄漏并用 LeakDetector 定位与修复。**

1. 在自定义 handler 中故意截断消息不释放，制造泄漏。  
2. 启动参数设置：
   - `-Dio.netty.leakDetection.level=paranoid`
   - `-Dio.netty.leakDetection.targetRecords=32`
3. 发送压测流量，观察日志中的 `LEAK: ByteBuf.release() was not called`。  
4. 按日志栈定位代码后补齐 `try/finally` 释放，或改用 `SimpleChannelInboundHandler` 让框架托管释放。

命令示例：
```bash
mvn -pl example -DskipTests compile
mvn -pl example exec:java -Dexec.mainClass=io.netty.example.echo.EchoServer -Dio.netty.leakDetection.level=paranoid
for /l %i in (1,1,2000) do @echo leak-%i|nc 127.0.0.1 8007
```

修复后重复压测，泄漏日志应消失。

#### 项目实战补充：泄漏治理闭环
建议把“发现泄漏 -> 定位 -> 修复 -> 回归”做成固定闭环，而不是一次性排障：

1. **发现阶段**：开发环境默认 `paranoid`，每日构建跑一轮泄漏冒烟。  
2. **定位阶段**：保留泄漏日志中的创建栈和最近访问栈，关联到具体 PR。  
3. **修复阶段**：明确对象所有权，补齐 `retain/release` 责任边界。  
4. **回归阶段**：同等流量重放 2~3 轮，确认无新增 leak 日志。

建议增加验证脚本（示例）：

命令示例：
```bash
python .\bench\leak_replay.py --messages 50000 --concurrency 100
```

验收口径建议：

- 压测窗口内无 `LEAK:` 关键字；  
- 直接内存曲线在压测后 5 分钟内回落到稳定区间；  
- 修复代码包含至少一条异常分支释放测试。

#### 常见坑
- **跨线程传递 ByteBuf 未 retain**：接收侧访问已失效对象。  
- **异常分支遗漏 release**：只在 happy path 释放。  
- **对同一对象重复 release**：计数归零后仍访问。  
- **生产常开 paranoid**：排障方便但吞吐明显下降。

#### 故障案例（最具迷惑性）
- **案例 A：只在高峰期出现 OOM**  
  根因：某异常分支漏 release，低流量下不明显，高流量下持续积累。  
  修复：统一 `try/finally` 释放模板，异常路径与正常路径共用同一收口。  

- **案例 B：偶发非法引用计数**  
  根因：消息被多个异步任务共享但未做 retain，先释放方提前归零。  
  修复：共享前显式 retain，消费方各自 release，并建立引用计数审计日志。

## 4. 项目总结
引用计数管理是 Netty 与“普通 Java 服务”最大的思维差异之一。你需要把内存对象当作“需要显式结算的资源”，而不是“等 GC 回收的垃圾”。工程落地建议是：定义所有权规则、统一释放模板、把 leak 检测纳入 CI 冒烟，避免等到线上内存报警才补锅。

最实用的经验是“先保正确，再谈性能”：先让引用计数绝对正确，再逐步引入切片、零拷贝等优化。如果基础不稳，任何性能技巧都会放大风险。

从团队协作角度看，引用计数问题最适合“模板化治理”：统一工具方法、统一日志字段、统一测试样例。只要所有人使用同一套释放模式，泄漏类问题会从“偶发玄学”变成“可工程化治理”。
