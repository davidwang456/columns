# 37-BlockHound阻塞检测实践

## 1. 项目背景

某实时通知服务偶发“全链路雪崩”：CPU 没打满，线程数也稳定，但请求在几十秒内大量超时。最终定位到一个看似无害的代码：在 事件循环线程里调用了阻塞式配置读取。问题在压测中难复现，因为只有配置中心抖动时才触发阻塞。此类故障的共性是：触发概率低、影响半径大、日志线索弱。

BlockHound 的价值就在这里。它通过字节码增强拦截阻塞调用，在不该阻塞的线程上直接报错，把“潜伏风险”前置到开发与测试阶段。对于 Netty/Reactor 栈，它尤其适合防止误用 `Thread.sleep`、阻塞 IO、同步锁等待等操作污染事件循环线程。

但 BlockHound 不是“装上就完事”。白名单配置、误报控制、第三方库兼容、CI 接入策略都要设计，否则要么噪声太多没人看，要么规则太弱抓不到真问题。

### 第一轮

小胖：我代码里没写 `sleep`，还会阻塞吗？  
小白：很多 SDK 内部会不会偷偷做阻塞调用？  
大师：正是难点。你不写，不代表依赖不写。BlockHound 的意义是把“隐式阻塞”显性化。

技术映射：你没在路口闯红灯，不代表外卖车不会逆行。

### 第二轮

小胖：检测这么严，会不会误伤正常逻辑？  
小白：比如启动阶段、管理线程是不是允许阻塞？  
大师：要按线程与场景分层。EventLoop 严格禁止，初始化线程可适度放行，规则要可审计可追踪。

技术映射：厂区安检不是一刀切，生产线和办公区标准不同。

### 第三轮

小胖：上线环境也开 BlockHound 吗？  
小白：性能开销可接受吗？  
大师：常见做法是测试/预发全开，生产按采样或短期开关用于排障。关键是形成“发现-修复-回归”闭环。

技术映射：体检不是每天做 CT，但高风险时段要专项筛查。

## 3. 项目实战

### 3.1 环境准备

- JDK 17、Netty 4.1.x、BlockHound 1.0+
- 测试框架：JUnit 5
- CI 任务：阻塞检测专用测试集
- 观测：阻塞调用次数、触发线程、调用栈 TopN

### 3.2 分步实现

**步骤目标 1：启动时安装 BlockHound。**

```java
public final class BlockHoundBootstrap {
    public static void install() {
        BlockHound.builder()
            .allowBlockingCallsInside("com.example.bootstrap.LegacyInit", "load")
            .install();
    }
}
```

在应用入口最早阶段调用 `install()`。

**步骤目标 2：构造 EventLoop 阻塞用例。**

```java
ctx.executor().execute(() -> {
    try {
        Thread.sleep(50);
    } catch (InterruptedException ignored) {
    }
});
```

期望：测试环境抛出阻塞异常并包含调用栈。

**步骤目标 3：接入单测防回归。**

```java
@Test
void shouldFailWhenBlockingOnEventLoop() {
    assertThrows(RuntimeException.class, () -> service.runBlockingCallInLoop());
}
```

命令：

命令示例：
```bash
mvn -q test -Dtest=*BlockHound*
```

**步骤目标 4：建立白名单审计。**

- 每条 `allowBlockingCallsInside` 必须写原因和失效日期。
- 每周扫描白名单增长趋势，防止“越放越松”。

可能遇到的坑：

1. 安装时机过晚，部分类已加载导致拦截不完整。  
2. 白名单滥用，把真实风险“合法化”。  
3. CI 只跑 happy path，阻塞路径未覆盖。

## 4. 项目总结

### 4.1 优点&缺点

| 维度 | 优点 | 缺点 |
| --- | --- | --- |
| 风险前置 | 可提前发现事件循环阻塞 | 初期误报多，需要治理 |
| 质量保障 | 可纳入 CI，防止回归 | 依赖团队执行纪律 |
| 运行排障 | 可在灰度期快速定位阻塞源 | 生产全开存在额外开销 |

### 4.2 适用场景

适用：

1. Netty/Reactor 事件驱动服务
2. 第三方 SDK 较多、阻塞风险高的系统
3. 对长尾抖动敏感的核心链路

不适用：

1. 纯同步阻塞模型应用（检测意义有限）
2. 无法维护规则与白名单治理流程的团队

### 4.3 注意事项

- BlockHound 要与线程命名规范配合，便于快速判断风险线程。
- 白名单必须最小化并定期回收。
- 发现阻塞后优先改异步方案，其次才是隔离线程池兜底。

### 4.4 常见踩坑经验

1. **故障案例：检测规则形同虚设**  
   根因：大量 `allowBlockingCallsInside` 无审计，真实问题被掩盖。
2. **故障案例：预发通过、线上仍抖动**  
   根因：线上依赖版本与预发不一致，新增阻塞路径未覆盖。
3. **故障案例：修复后吞吐下降**  
   根因：阻塞调用改异步后线程池参数未调整，产生新瓶颈。
