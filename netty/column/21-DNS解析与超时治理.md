# 21-DNS解析与超时治理

## 1. 项目背景

DNS 是链路里最常被忽略的隐形延迟源。压测时稳定、生产高峰抖动，很多都是解析波动引发。Netty 的 `resolver-dns` 模块让解析可异步、可观测、可缓存治理，核心位于 `resolver-dns/src/main/java/io/netty/resolver/dns`，示例位于 `example/src/main/java/io/netty/example/dns/udp/DnsClient.java` 与 `example/src/main/java/io/netty/example/dns/dot/DoTClient.java`。

常见问题：只设连接超时不设解析超时、失败重试放大故障、TTL 失配导致流量黏在坏节点。本章目标是建立“解析预算”观念：把总超时拆为 DNS、连接、业务三段，并独立监控。

## 4. 项目总结

DNS 治理把“隐形依赖”变成一等指标。优点是耗时可拆分、告警精准、策略可调；缺点是参数多、协同成本高。适用多下游和跨地域场景；不适用于极简单机工具。踩坑集中在 TTL 配错、全量重试、口径混淆。建议开发先接分段打点，运维做可达巡检，测试做抖动注入。


为了让 DNS 治理真正可落地，这里补一套可执行的排障闭环。第一步是建立“解析画像”：按域名维度统计查询量、超时率、命中缓存比例、CNAME 链长度、单次解析耗时分布。很多团队只看总体成功率，忽略了热点域名的集中故障。建议至少按核心域名维护 TopN 看板。第二步是建立解析分层日志：在发起查询时记录 query id、nameserver、record type；收到响应时记录 rcode、answer 数量、ttl；超时时记录 elapsed、重试次数、最终降级动作。这样当线上出现“偶发超时”时，你能快速区分是权威 DNS 抖动、网络抖动、还是本地缓存策略不当。

在本仓库路径中，`resolver-dns/src/main/java/io/netty/resolver/dns/DnsQueryContext.java` 和 `DnsResolveContext.java` 体现了查询上下文推进与失败处理机制；`DefaultDnsCache.java` 则是缓存策略核心。实操中建议先在测试环境做三类注入：一是单个 nameserver 200~500ms 延迟抖动；二是权威返回 `SERVFAIL`；三是部分域名返回超短 TTL（例如 1~5 秒）。每类注入都要记录“应用总超时是否被拉高”“是否触发重试风暴”“下游连接超时占比是否被误导上升”。

命令层面可以这样执行：
`mvn -pl resolver-dns -Dtest=DnsNameResolverBuilderTest test`，重点观察 builder 参数变更对行为的影响；
`mvn -pl resolver-dns -Dtest=DnsNameResolverTest test`，关注超时、缓存、重试路径是否符合预期。
预期输出应为测试通过且无随机失败。若出现偶发失败，优先排查是否依赖外部网络环境。建议将网络相关测试拆为“本地可重复单测”和“集成环境连通测试”，避免 CI 抖动污染结论。

常见踩坑再补三条：
1) 仅配置 JVM 级 DNS 缓存，忽略 Netty 侧缓存，导致口径不一致；
2) 多租户共享同一解析器实例却没有隔离查询配额，单租户抖动会拖垮整体；
3) 失败重试没有幂等保护，和上游请求重试叠加形成放大回路。

推广协作建议：开发负责分段埋点与重试策略，运维维护 nameserver 健康与路由可达，测试负责注入场景回归。三方每次发布前进行 15 分钟“解析健康巡检”，形成固定 checklist：核心域名可达、缓存命中率在合理区间、超时率无异常抬升、重试量未超过阈值。


### 补充实战手册

围绕DNS解析主题，建议团队在预发布环境执行一轮完整走查。走查不应只看功能通过，还要验证失败路径、回退路径、告警路径是否同样可用。下面给出可直接复用的执行模板：

- 步骤A：准备阶段，确认配置项、证书、白名单、超时参数、线程池参数与线上目标值一致。
- 步骤B：冒烟阶段，先跑单实例验证，记录首次成功耗时与失败错误码分布。
- 步骤C：压力阶段，逐步提高并发，观察 p50/p95/p99、错误率、连接数、队列深度。
- 步骤D：故障注入，注入延迟、丢包、异常响应、依赖不可达，检查系统是否按预期降级。
- 步骤E：回退演练，强制回退到上一版本，验证配置与协议兼容性。

排障顺序建议遵循“先链路、再线程、后业务”的原则：
1) 链路：端口可达、协议头合法、握手/解析/连接成功；

补充踩坑清单：
- 坑1：只验证成功路径，忽略异常路径，导致上线后第一波异常直接打穿。
- 坑2：没有统一日志字段，跨层排障时无法串联同一请求。
- 坑3：告警阈值照搬历史值，业务规模变化后出现大量误报或漏报。
- 坑4：回退脚本未演练，真实故障时“会回退”变成“回不去”。

团队协作建议：开发负责实现与埋点，测试负责场景覆盖和回归基线，运维负责容量与发布节奏。每次变更后进行 30 分钟复盘，沉淀参数变化、故障样本、修复动作，为下一次发布建立可复制经验。
