# 23-端口统一与协议嗅探

## 1. 项目背景

随着协议演进，同一服务可能要同时接 TLS、明文 HTTP 和自定义协议。多端口方案简单但运维成本高。端口统一通过首包嗅探动态组装 pipeline，是迁移与灰度的常用手段。参考 `example/src/main/java/io/netty/example/portunification`。

## 4. 项目总结

优点：入口收敛、灰度灵活、兼容旧客户端。缺点：嗅探复杂、攻击面集中。踩坑主要是窗口过大、顺序错误、无超时保护。建议先小流量验证再逐步放量。


端口统一落地时，最难的不是识别协议，而是处理“不确定状态”。现实网络里首包可能分片、乱序到达、甚至被恶意拖慢。为了避免事件循环被慢连接占满，建议把嗅探阶段设计为独立状态机：`INIT -> SNIFFING -> DECIDED -> FAILED`。在 `SNIFFING` 状态设置最大等待时长和最大探测字节，一旦超限立即失败。这样即便遭遇慢速攻击，也能快速释放连接资源。

本仓库 `example/src/main/java/io/netty/example/portunification/PortUnificationServerHandler.java` 的关键价值在于“判定后替换 pipeline”。你可以延伸为多级判定：
- 第一级：TLS/非 TLS；
- 第二级：HTTP/自定义二进制；
- 第三级：按版本或客户端标识分流旧新协议。
每一级都要打点：判定耗时、命中分支、失败原因。没有打点就无法做灰度控制。

推荐实操演练：
1) 模拟首包完整与首包分片，验证判定正确率；
2) 模拟慢连接每秒只发 1 字节，验证超时是否生效；
3) 模拟错误魔数，验证是否快速失败而非进入业务层。
命令可使用示例服务配合 `openssl` 和 `curl`，同时开压测工具观察事件循环延迟曲线。预期结果是：在混合流量下协议命中准确、慢连接被快速清理、业务线程无异常积压。

踩坑补充：
1) 判定逻辑里做复杂字符串解析，导致首包路径 CPU 升高；
2) 动态替换 pipeline 后未移除嗅探 handler，导致重复判定；
3) 灰度策略与客户端版本映射表不同步，引发误路由。

推广建议：端口统一的收益来自运维收敛和迁移平滑。发布前要和网络团队确认安全策略、健康检查、端口暴露规则是否同步调整；测试侧要建立协议回归矩阵，至少覆盖旧协议、混合协议、新协议三条路径，确保回退可用。


### 补充实战手册

围绕端口统一协议嗅探主题，建议团队在预发布环境执行一轮完整走查。走查不应只看功能通过，还要验证失败路径、回退路径、告警路径是否同样可用。下面给出可直接复用的执行模板：

- 步骤A：准备阶段，确认配置项、证书、白名单、超时参数、线程池参数与线上目标值一致。
- 步骤B：冒烟阶段，先跑单实例验证，记录首次成功耗时与失败错误码分布。
- 步骤C：压力阶段，逐步提高并发，观察 p50/p95/p99、错误率、连接数、队列深度。
- 步骤D：故障注入，注入延迟、丢包、异常响应、依赖不可达，检查系统是否按预期降级。
- 步骤E：回退演练，强制回退到上一版本，验证配置与协议兼容性。

补充踩坑清单：
- 坑1：只验证成功路径，忽略异常路径，导致上线后第一波异常直接打穿。
- 坑2：没有统一日志字段，跨层排障时无法串联同一请求。
- 坑3：告警阈值照搬历史值，业务规模变化后出现大量误报或漏报。
- 坑4：回退脚本未演练，真实故障时“会回退”变成“回不去”。

团队协作建议：开发负责实现与埋点，测试负责场景覆盖和回归基线，运维负责容量与发布节奏。每次变更后进行 30 分钟复盘，沉淀参数变化、故障样本、修复动作，为下一次发布建立可复制经验。
