# 28-CORS与安全中间件拼装

## 1. 项目背景

CORS 是浏览器场景下的重要安全边界。配置过松会泄露数据，配置过严会阻断合法调用。Netty 示例位于 `example/src/main/java/io/netty/example/http/cors`，可用于验证预检与跨域策略。

命令示例：
```bash

curl -i -X OPTIONS http://127.0.0.1:8080/ -H "Origin: https://admin.example.com" -H "Access-Control-Request-Method: POST"
```

预期输出：返回 `Access-Control-Allow-Origin/Methods/Headers`。

步骤建议：
1) 维护分环境白名单；
2) 预检快速返回；
3) 凭证模式禁止 `*`；
4) 审计日志记录 `Origin` 与结果。

## 4. 项目总结

CORS 是安全治理的一部分，不是前端兼容补丁。优点：策略清晰、可审计。缺点：环境差异和浏览器细节多。常见踩坑：`*` 与凭证共存、预检被提前拦截、网关与应用策略冲突。建议统一策略来源并做自动化回归。


CORS 落地失败常见于“策略分散”：网关一套、应用一套、业务中间件又一套，最终谁生效取决于链路偶然顺序。正确做法是建立单一策略源，再按环境编译成运行配置。对浏览器请求，必须明确区分“简单请求”和“预检请求”：预检应轻量快速返回，不进入重鉴权路径；实际请求再按认证授权与业务规则处理。

本仓库 `example/src/main/java/io/netty/example/http/cors/HttpCorsServerInitializer.java` 可作为中间件顺序参考。建议在安全拼装时把以下能力放到统一 pipeline：
1) CORS 白名单与方法控制；
2) 安全响应头（如 `X-Content-Type-Options`、`X-Frame-Options`）；
3) 认证鉴权；
4) 访问频控；
5) 审计日志；
6) 异常归一化。
顺序推荐先 CORS 后鉴权，避免预检被误拦。

命令回归建议：
`mvn -pl example -DskipTests exec:java -Dexec.mainClass=io.netty.example.http.cors.HttpCorsServer`
然后分别执行合法源、非法源、带凭证请求、预检请求四组 curl。
预期输出：合法源通过，非法源拒绝，凭证模式返回具体 Origin 而非 `*`，预检返回允许头和方法。

踩坑补充：
1) 允许任意 Origin + Allow-Credentials=true，浏览器会拒绝且存在安全隐患；
2) 预检响应未带 `Access-Control-Max-Age`，导致浏览器频繁预检放大延迟；
3) 只在应用层配 CORS，网关提前拦截导致策略失效。

推广建议：开发维护策略模板与灰度开关，运维同步网关和边缘配置，测试用真实浏览器自动化脚本验证。发布窗口内重点观察跨域失败率、预检比例和被拒 Origin TopN，确保策略变更可解释、可回滚。


### 补充实战手册

围绕CORS安全中间件主题，建议团队在预发布环境执行一轮完整走查。走查不应只看功能通过，还要验证失败路径、回退路径、告警路径是否同样可用。下面给出可直接复用的执行模板：

- 步骤A：准备阶段，确认配置项、证书、白名单、超时参数、线程池参数与线上目标值一致。
- 步骤B：冒烟阶段，先跑单实例验证，记录首次成功耗时与失败错误码分布。
- 步骤C：压力阶段，逐步提高并发，观察 p50/p95/p99、错误率、连接数、队列深度。
- 步骤D：故障注入，注入延迟、丢包、异常响应、依赖不可达，检查系统是否按预期降级。
- 步骤E：回退演练，强制回退到上一版本，验证配置与协议兼容性。

推荐命令模板（按需替换模块名）：

命令示例：
```bash
mvn -pl example -DskipTests compile
mvn -pl example -DskipTests exec:java -Dexec.mainClass=<YourMainClass>
mvn -pl <module> -Dtest=<YourTestClass> test
```

预期输出模板：

```text

启动日志无 ERROR
关键指标在阈值内
```

排障顺序建议遵循“先链路、再线程、后业务”的原则：
1) 链路：端口可达、协议头合法、握手/解析/连接成功；
2) 线程：事件循环延迟、业务池排队、拒绝率；
3) 业务：错误码分布、降级命中、重试行为。

补充踩坑清单：
- 坑1：只验证成功路径，忽略异常路径，导致上线后第一波异常直接打穿。
- 坑2：没有统一日志字段，跨层排障时无法串联同一请求。
- 坑3：告警阈值照搬历史值，业务规模变化后出现大量误报或漏报。
- 坑4：回退脚本未演练，真实故障时“会回退”变成“回不去”。

团队协作建议：开发负责实现与埋点，测试负责场景覆盖和回归基线，运维负责容量与发布节奏。每次变更后进行 30 分钟复盘，沉淀参数变化、故障样本、修复动作，为下一次发布建立可复制经验。
