# 14-基础篇综合实战迷你网关

## 1. 项目背景

经历了前面十三章，你已经掌握连接、编解码、心跳、日志等单点能力，但真实生产不会按知识点拆题。业务方现在要你两周内上线一个“迷你网关”：统一接入三类内部服务，支持限流、鉴权、路由转发和统一错误码，同时要求可观测、可灰度。看似是“做个转发器”，实际上是把前面所有基础能力拼成一个可运行系统。

很多团队第一次做网关都会犯两个错误：一是把业务逻辑塞进单个 handler，后期不可维护；二是只关注功能跑通，忽略连接生命周期、背压、异常与超时，导致压测漂亮、上线翻车。本章目标不是复制企业级网关，而是用 Netty examples 和核心组件搭一个可发布的最小闭环，让你形成“按 pipeline 分层”的工程习惯。

## 2. 项目设计（剧本式交锋对话）

| 角色 | 性格标签 | 职责 | 话风示例 |
| --- | --- | --- | --- |
| 小胖 | 爱吃爱玩、不求甚解 | 用生活化比喻抛出问题，引发讨论 | “这不就跟食堂打饭排队一样吗？” |
| 小白 | 喜静、喜深入 | 追问原理、边界、风险和备选方案 | “那极端情况下会不会反噬系统？” |
| 大师 | 资深技术 Leader | 讲透业务约束、机制原理和落地取舍 | “先分层，再定边界，最后看成本。” |

小胖：我理解这章主题了，但能不能给个最省事做法？上线时间很紧，我只想先跑起来。  
小白：你这个“先跑起来”每次都容易埋雷。我们得先明确边界：消息生命周期、资源上限、错误语义、回收策略。  
大师：两位都对。工程不是只看最优理论，也不是只求最快上线。先把目标拆成三个层次：业务目标、系统约束、应急兜底。业务目标决定我们追求什么，系统约束决定我们能做到什么，应急兜底决定故障来时系统是否可控。  
小胖：听起来像做菜先定菜单、再看冰箱、最后准备泡面。  
大师：这个比喻可以。技术映射就是：菜单=需求范围，冰箱=资源预算，泡面=降级与回退。

小白：那我们如何把这章能力落在 pipeline 上？我担心 handler 之间职责不清，后续一改就连锁反应。  
大师：按“入口治理、核心处理、收尾治理”三段切。入口治理负责校验和分流；核心处理只做业务主路径；收尾治理统一日志、指标和错误。这样每段都能独立压测和替换。  
小胖：是不是跟商场动线一样，先安检、再购物、最后结账开发票？  
大师：正是。技术映射：安检=鉴权限流，购物=业务执行，结账=响应封装和可观测。  
小白：如果某一段出错，比如核心处理超时，怎么保证用户侧有确定性结果？  
大师：必须定义错误语义分级：可重试、不可重试、降级成功。并且每个分级要有统一状态码、日志字段和指标标签。否则运维看不懂，客户端也无法正确重试。

小胖：我还有个担心，线上流量波动时，平时看着没问题的策略会不会突然失效？  
小白：对，尤其是高峰时段，队列、窗口、连接都可能被打满。  
大师：所以我们要把“正常路径”与“拥塞路径”都设计出来。正常路径追求效率，拥塞路径追求可控。比如达到阈值后优先降级非核心功能，而不是让所有请求一起超时。  
小胖：就像电梯高峰期先保障上班楼层，外卖员慢一点也能接受。  
大师：技术映射：优先级调度与背压控制。  
小白：最后一个问题，怎么证明方案真的可发布，而不是 PPT 可行？  
大师：三类验证必须齐全：功能正确性、压力稳定性、故障可恢复性。每类都要有可重复脚本和可观察指标。只有能反复验证，才叫工程能力沉淀。  
小胖：懂了，能跑只是开始，能稳、能查、能恢复才是上线标准。

## 3. 项目实战

### 3.1 环境准备

- JDK 17、Maven 3.9+、Netty 4.1.x。
- 建议以 `example` 模块作为对照阅读，先跑通官方样例再迁移到业务工程。
- 关键参考路径：
  - `example/src/main/java/io/netty/example/http/snoop`
  - `example/src/main/java/io/netty/example/proxy/HexDumpProxyFrontendHandler.java`
  - `codec-http/src/main/java/io/netty/handler/codec/http/HttpServerCodec.java`

### 3.2 分步实现

实战按“入口治理 -> 路由转发 -> 回包统一”三段推进。
步骤一：入口层 pipeline：`HttpServerCodec` + `HttpObjectAggregator` + `AuthHandler` + `RateLimitHandler`。`AuthHandler` 校验 token，失败直接返回 401；`RateLimitHandler` 用令牌桶按 clientId 限流，超限返回 429。
步骤二：路由层 `RouteDispatchHandler` 按 URI 前缀映射后端地址，复用 `Bootstrap` 建立到后端的短连接或连接池连接。参考 proxy example 的前后端 channel 绑定方式，把前端请求写到后端 channel。
步骤三：回包层 `UnifiedResponseHandler` 统一错误码、透传 traceId，并在异常时包装标准 JSON。
步骤四：治理能力补齐：超时控制（后端 2s）、熔断阈值、日志采样、优雅关机钩子。

运行后你将得到一个可用迷你网关：`/api/user/*` 路由到用户服务，`/api/order/*` 路由到订单服务，鉴权/限流/错误语义一致。常见坑：前后端 channel 生命周期未绑定造成泄漏；后端失败未及时回写导致前端悬挂；统一异常处理遗漏业务异常类型。

### 3.3 完整代码清单

- `server/BootstrapConfig.java`：启动参数、线程模型、通用选项。
- `server/initializer/ServerChannelInitializer.java`：pipeline 装配与顺序定义。
- `server/handler/*`：按职责拆分的业务与治理 handler。
- `server/metrics/*`：指标上报与 trace 关联。
- 上述官方路径用于对照实现细节与行为语义。

## 4. 项目总结

### 4.1 优点&缺点

| 维度 | 优点 | 缺点 |
| --- | --- | --- |
| 开发效率 | 使用 Netty 标准组件，接口清晰，落地速度快 | 前期需要花时间明确 handler 边界和顺序 |
| 运行稳定性 | 通过超时、限流、背压、回收策略提升可控性 | 参数调优依赖压测和监控，经验门槛较高 |
| 可观测性 | 事件、日志、指标可形成完整排障链路 | 指标维度过多时需要治理，避免噪音 |
| 演进能力 | 分层设计便于替换单个 handler 或策略 | 若缺少契约约束，后续改造仍可能耦合 |

### 4.2 适用场景

- 需要高并发 I/O 处理且对时延敏感的网关/接入层系统。
- 协议处理链路清晰、可按 handler 分层治理的中间件服务。
- 需要“可灰度、可回滚、可压测复现”的团队协作场景。
- 对连接生命周期、资源回收和异常语义有严格要求的系统。

不适用场景：
- 单机低并发、生命周期短且无需复杂治理的脚本型服务。
- 团队缺少基本监控与压测能力、无法支撑参数化调优的项目。

### 4.3 注意事项

- 不要把“示例可跑”当作“生产可用”，必须补齐超时、限流、错误语义。
- 关键阈值（窗口、聚合长度、队列长度）必须以压测数据为依据。
- 对跨线程对象传递、引用计数、连接关闭顺序要有明确约定。
- 任何降级策略都要可观测，避免静默失败掩盖真实故障。

### 4.4 常见踩坑经验

1. 只优化主路径，不设计拥塞与故障路径，导致高峰期雪崩。  
2. handler 职责混杂，后续需求迭代时出现不可预期副作用。  
3. 指标和日志未统一字段，跨团队排障时无法快速对齐事实。
