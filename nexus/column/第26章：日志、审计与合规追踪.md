# 第26章：日志、审计与合规追踪

## 1. 项目背景

云鲸科技的年度 ISO 27001 审计即将来临，安全团队提出了一个让炮哥冷汗直冒的问题："请提供过去 12 个月内所有对生产仓库的制品删除操作的完整记录，包括操作人、时间、IP 和删除的制品坐标。"炮哥的第一反应是翻 `nexus.log`——结果发现 `nexus.log` 是按大小滚动的，已经只剩下最近 7 天的内容。他又去翻 `request.log`——里面有 HTTP 请求记录，但 DELETE 请求只记录了请求路径和状态码，没有记录被删除的具体制品名称和坐标。

更尴尬的是，当安全团队追问"上周谁修改了 maven-releases 仓库的 writePolicy 从 ALLOW_ONCE 改为 ALLOW"时，炮哥翻遍了所有日志也找不到答案——配置变更在 `nexus.log` 中确实有一行 INFO 日志，但被淹没在每天几十万行的日志海洋中，没人注意到，也无法快速检索。

Nexus 不是没有审计能力——它内置了一套结构化的审计日志系统（`audit.log`），记录了大量关键操作的 AuditData（操作域、类型、上下文、属性）。问题在于：**大多数团队根本不知道 audit.log 的存在，或者知道但没有结构化采集和分析它**。本章将揭开 Nexus 审计日志的全貌，建立从采集到 Elasticsearch 到 Kibana 看板的完整审计链路，让"谁在什么时候做了什么事"从不可知变成一目了然。

## 2. 项目设计

安全审计师老刘把一份 200 页的合规检查清单放在桌上，炮哥面色发白。

**老刘**："我需要审计三类事件——制品操作（上传/删除/下载）、权限变更（角色/用户/权限的创建/修改/删除）、配置变更（仓库设置/系统设置修改）。Nexus 有记录吗？"

**大师**："有，但你需要知道去哪找。Nexus 有五类日志文件，各有分工。`nexus.log` 是应用主日志——框架层的 INFO/WARN/ERROR，适合排障但不适合审计。`request.log` 是 HTTP 访问日志——每行一个请求，适合流量分析。`audit.log` 是审计日志——记录所有操作的领域、类型、上下文和属性，是审计的核心数据源。`task.log` 记录调度任务的执行情况。`jvm.log` 记录 GC 和内存信息。"

> **技术映射**：Nexus 日志分层——request.log（流量层）、nexus.log（应用层）、audit.log（审计层）、task.log（任务层）、jvm.log（系统层）。审计合规主要使用 audit.log。

**小胖**："audit.log 里面长什么样？不会是乱码吧？"

**大师**："audit.log 中每条记录是一个 JSON 对象，包含六个核心字段——`domain`（操作域，如 repository、security）、`type`（操作类型，如 component、user、privilege）、`context`（操作上下文，如操作发生的仓库名或组件 ID）、`attributes`（操作详情，如 GAV 坐标、文件路径）、`timestamp`（精确时间戳）、`initiator`（操作者信息）。"

**小白**："那 Nexus 自己怎么查询这些审计数据？Web UI 里有审计页面吗？"

**大师**："Nexus OSS 版的审计能力比较基础——它记录但不提供高级搜索和可视化。PRO 版有 `Audit` 页面可以按时间、用户、领域筛选。OSS 版的最佳实践是把 audit.log 采集到 Elasticsearch/OpenSearch 中，用 Kibana 做检索和看板。采集方案可以是 Filebeat 直接读日志文件推送到 Elasticsearch，或者通过 Syslog/Logstash 中转。"

> **技术映射**：审计数据流 = audit.log（生成）→ Filebeat/Syslog（采集）→ Elasticsearch（存储）→ Kibana（可视化）。这是 ELK Stack 的标准用法。

**老刘**："日志要保留多久？我们审计要求至少 12 个月。"

**大师**："Nexus 默认按大小滚动（每 10MB 一个文件，保留 90 天）。你需要调整 `logback.xml` 配置改为按时间滚动。但更关键的是——日志文件存在 `/nexus-data/log/` 下，属于 Nexus 的数据目录，备份策略中需要包含。Elasticsearch 侧的保留策略则独立管理。"

**炮哥**："那高风险操作——比如删除生产仓库的组件、修改 admin 角色——能实时告警吗？"

**大师**："audit.log 是文件级日志，有写入延迟。实时告警需要走 Webhook——第 27 章会详细讲。简单说，Nexus 在关键操作发生时通过 Webhook 实时推送事件，比定期扫描日志快得多。两者的关系是：Webhook 做实时告警，日志做长期审计，互补不对立。"

#### 步骤二：配置 Filebeat 采集 Nexus 审计日志

**目标**：用 Filebeat 将 audit.log 结构化采集到 Elasticsearch。

```yaml
# filebeat.yml — Filebeat 配置文件
filebeat.inputs:
  - type: log
    enabled: true
    paths:
      - /nexus-data/log/audit/audit*.log
    json.keys_under_root: true
    json.add_error_key: true
    json.overwrite_keys: true
    fields:
      log_type: nexus-audit
    fields_under_root: true

  - type: log
    enabled: true
    paths:
      - /nexus-data/log/request.log
    fields:
      log_type: nexus-request
    fields_under_root: true

output.elasticsearch:
  hosts: ["elasticsearch.internal:9200"]
  index: "nexus-audit-%{+yyyy.MM.dd}"
  # 如果 ES 需要认证：
  # username: "elastic"
  # password: "${ES_PASSWORD}"

setup.template.name: "nexus-audit"
setup.template.pattern: "nexus-audit-*"
setup.template.overwrite: true

# 多行 JSON 支持
multiline:
  pattern: '^{'
  negate: true
  match: after
```

**部署 Filebeat**：

#### 步骤四：编写高风险操作日志查询脚本

**目标**：用 curl 直接查询 ES 或通过脚本检索关键审计事件。

```bash
#!/bin/bash
# audit-search.sh：检索特定类型的审计事件
ES_HOST="${ES_HOST:-localhost:9200}"
DAYS="${1:-7}"  # 查询最近 N 天
DOMAIN="${2:-repository}"  # 审计领域
ACTION="${3:-DELETED}"   # 操作类型

echo "=== Nexus 审计查询 ==="
echo "查询: 最近 ${DAYS} 天的 ${DOMAIN} 域 ${ACTION} 操作"
echo ""

# 查询 Elasticsearch
curl -s -X POST "${ES_HOST}/nexus-audit-*/_search" \
  -H "Content-Type: application/json" \
  -d "{
    \"query\": {
      \"bool\": {
        \"must\": [
          {\"match\": {\"domain\": \"${DOMAIN}\"}},
          {\"match\": {\"attributes.action\": \"${ACTION}\"}}
        ],
        \"filter\": [
          {\"range\": {\"timestamp\": {\"gte\": \"now-${DAYS}d\"}}}
        ]
      }
    },
    \"sort\": [{\"timestamp\": \"desc\"}],
    \"size\": 50
  }" | jq '.hits.hits[]._source | {time: .timestamp, user: .initiator, domain: .domain, context: .context, action: .attributes.action}'
```

```bash
chmod +x audit-search.sh

# 查询最近 7 天所有删除操作
./audit-search.sh 7 repository DELETED

# 查询最近 30 天所有权限变更
./audit-search.sh 30 security
```

#### 步骤五：配置审计日志的保留期

**目标**：在 Elasticsearch 中设置索引生命周期策略，满足合规保留要求。

```bash
# 在 Kibana Dev Tools 中执行（或通过 curl）

# 1. 创建 ILM Policy（保留 12 个月）
curl -X PUT "${ES_HOST}/_ilm/policy/nexus-audit-retention" \
  -H "Content-Type: application/json" \
  -d '{
    "policy": {
      "phases": {
        "hot": {
          "min_age": "0ms",
          "actions": {
            "rollover": {"max_age": "1d", "max_size": "10gb"}
          }
        },
        "delete": {
          "min_age": "365d",
          "actions": {"delete": {}}
        }
      }
    }
  }'

# 2. 应用到索引模板
curl -X PUT "${ES_HOST}/_template/nexus-audit-template" \
  -H "Content-Type: application/json" \
  -d '{
    "index_patterns": ["nexus-audit-*"],
    "settings": {
      "index.lifecycle.name": "nexus-audit-retention",
      "index.lifecycle.rollover_alias": "nexus-audit"
    }
  }'

echo "ILM 策略已配置：索引保留 365 天后自动删除"
```

#### 步骤六：合规审计报告生成工具

**目标**：一键生成指定时间范围内的合规审计报告。

```bash
#!/bin/bash
# compliance-report.sh：生成合规审计报告
ES_HOST="${ES_HOST:-localhost:9200}"
MONTH="${1:-$(date +%Y-%m)}"  # 如 2025-06

REPORT="nexus-compliance-report-${MONTH}.md"

cat > "$REPORT" << HEADER
# Nexus 合规审计报告
## 报告周期：${MONTH}

### 1. 制品删除汇总
HEADER

curl -s -X POST "${ES_HOST}/nexus-audit-*/_search" \
  -H "Content-Type: application/json" \
  -d "{
    \"query\": {
      \"bool\": {
        \"must\": [
          {\"match\": {\"attributes.action\": \"DELETED\"}},
          {\"match\": {\"domain\": \"repository\"}}
        ],
        \"filter\": [{\"range\": {\"timestamp\": {\"gte\": \"${MONTH}-01\", \"lt\": \"${MONTH}-32\"}}}]
      }
    },
    \"size\": 0,
    \"aggs\": {
      \"by_user\": {\"terms\": {\"field\": \"initiator.keyword\", \"size\": 20}},
      \"by_repo\": {\"terms\": {\"field\": \"context.keyword\", \"size\": 20}}
    }
  }" | jq -r '
  "| 操作者 | 删除次数 |",
  "|--------|---------|",
  (.aggregations.by_user.buckets[] | "| \(.key) | \(.doc_count) |"),
  "",
  "| 仓库 | 删除次数 |",
  "|------|---------|",
  (.aggregations.by_repo.buckets[] | "| \(.key) | \(.doc_count) |")
' >> "$REPORT"

echo "" >> "$REPORT"
echo "### 2. 权限变更记录" >> "$REPORT"

curl -s -X POST "${ES_HOST}/nexus-audit-*/_search" \
  -H "Content-Type: application/json" \
  -d "{
    \"query\": {
      \"bool\": {
        \"must\": [{\"match\": {\"domain\": \"security\"}}],
        \"filter\": [{\"range\": {\"timestamp\": {\"gte\": \"${MONTH}-01\", \"lt\": \"${MONTH}-32\"}}}]
      }
    },
    \"size\": 50,
    \"sort\": [{\"timestamp\": \"desc\"}]
  }" | jq -r '
  "| 时间 | 操作者 | 操作类型 | 上下文 |",
  "|------|--------|---------|--------|",
  (.hits.hits[]._source | "| \(.timestamp) | \(.initiator) | \(.type) | \(.context) |")
' >> "$REPORT"

echo ""
echo "报告已生成: $REPORT"
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| audit.log 文件过大 | 单个 log 文件超过 1GB，Filebeat 采集性能下降 | 调整 `logback.xml` 中 audit logger 的滚动策略 |
| JSON 多行被拆散 | Elasticsearch 中一条审计记录被拆成多行 | Filebeat 中配置 `multiline` 匹配 JSON 起始符 `{` |
| 时区不一致 | audit.log 用 +08:00，Elasticsearch 用 UTC | 在 Filebeat 或 Logstash 中统一转换为 UTC |
| 非 JSON 日志混入 audit.log | 偶尔出现纯文本异常堆栈混入 JSON 流 | Filebeat 中 `json.add_error_key: true` 标记解析失败的记录 |

## 4. 项目总结

### 4.1 Nexus 日志分类速查

| 日志文件 | 路径 | 内容 | 用途 | 保留策略 |
|---------|------|------|------|---------|
| `nexus.log` | `/nexus-data/log/` | 应用 INFO/ERROR 日志 | 排障、性能分析 | 保留 30 天 |
| `request.log` | `/nexus-data/log/` | HTTP 请求一行一个 | 流量分析、4xx/5xx 统计 | 保留 30 天 |
| `audit.log` | `/nexus-data/log/audit/` | JSON 格式审计记录 | 合规审计、安全追溯 | 保留 12 个月 |
| `task.log` | `/nexus-data/log/tasks/` | 任务执行日志 | 任务失败排查 | 保留 30 天 |
| `jvm.log` | `/nexus-data/log/` | JVM GC 日志 | 内存调优 | 保留 7 天 |

### 4.2 适用场景

1. **合规审计**：提供 ISO 27001/SOX 要求的操作追溯能力
2. **安全事故溯源**：追踪"谁删了生产仓库的包"等安全事件
3. **操作行为分析**：分析团队使用 Nexus 的模式（谁用了哪些仓库、上传频率等）
4. **变更管理**：追踪所有配置变更操作及其操作者
5. **容量使用审计**：分析哪些用户/团队产生了最多的制品

**不适用场景**：
1. 实时告警——用 Webhook 代替（第 27 章）
2. 性能监控——用 JMX/Prometheus 指标代替（第 28 章）

### 4.3 注意事项

- **audit.log 默认按大小滚动（10MB）**：高负载环境可能需要调整为按时间滚动
- **Elasticsearch 索引需要 ILM 策略**：避免磁盘被审计日志填满
- **访问审计日志本身也是敏感操作**：ensure audit.log 的读取权限仅限管理员
- **Filebeat 资源消耗**：Nexus 大量审计事件时 Filebeat 可能消耗 200-500MB 内存

### 4.4 常见踩坑经验

**故障一：audit.log 滚动后老文件未压缩**

运维发现 `/nexus-data/log/audit/` 下堆积了几百个未压缩的 `audit-*.log` 文件，占用 8GB。根因：Nexus 的 `logback.xml` 中未启用压缩。解决：修改 `logback.xml` 中 `fileNamePattern` 为 `audit-%d{yyyy-MM-dd}.%i.log.gz`。

**故障二：Elasticsearch 索引爆炸**

按天创建的 `nexus-audit-*` 索引 6 个月后达到了 180 个分片，集群变慢。根因：索引模板中 `number_of_shards: 3`，每个索引 3 个分片，180 天就是 540 个分片。解决：审计日志量不大时，`number_of_shards: 1` + Rollover `max_size: 5gb`。

**故障三：时区导致审计时间线混乱**

审计报告显示"某用户在凌晨 3 点删除了制品"，但该用户声称当时在睡觉。根因：Nexus 记录的时间戳带时区（`+08:00`），但 Kibana 默认按浏览器时区显示，审计报告却按 UTC 显示。解决：Elasticsearch 将所有时间统一转为 UTC，Kibana 中使用 `dateFormat:tz` 设置。

### 4.5 思考题

1. 安全团队要求对所有包含 `production` 关键词的仓库执行删除操作时，必须触发二级审批（发送通知并等待确认）。基于 audit.log 和 Elasticsearch，如何设计这个"删除操作实时审批"方案？
2. audit.log 中记录了 `READ` 操作（组件下载），但下载量极大（每天几十万次），写入 audit.log 会导致严重的 IO 压力。如何在保留下载审计能力的同时减少日志量？

（第25章思考题答案：1. 配置同步扩展为事件驱动方案：在每个 Nexus 实例中配置 Webhook ——当源实例的仓库/权限配置变更时，Webhook 推送变更事件到配置同步服务。同步服务将事件转译为 API 调用，在目标实例上执行相同操作。使用"最终一致性"模型——允许两个实例短暂不一致（< 5 分钟），但不能超过。通过版本号或时间戳对比解决冲突。2. 配置回滚方案：在 Git 仓库中保留 YAML 配置文件的完整历史（每个 merge 产生一个 commit）。回滚时，`git checkout <last-good-commit>` 恢复 YAML，执行 `DRY_RUN=true ./nexus-as-code.sh` 预览差异，确认后 `DRY_RUN=false` 执行调和。同时保留执行前的配置快照（GET 当前所有资源到 backup.yaml），如果调和失败可以手动恢复。）

### 4.6 推广计划提示

- **安全团队**：将合规审计报告生成脚本加入季度审计流程，确保 ELK 中审计数据保留 12 个月
- **运维部门**：将 ELK 采集管道的健康状态纳入监控，确保审计日志采集不中断
- **开发部门**：了解审计日志的存在——当你"手滑"删了一个制品时，审计日志里有你删的证据
