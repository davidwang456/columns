# 第8章：用户、角色、权限与团队隔离

## 1. 项目背景

云鲸科技的 Nexus 已经跑了一个月，仓库里堆满了 Java 的 jar、前端的 tgz、Docker 的镜像和运维的脚本。一切看起来井井有条，直到事故发生——

周二上午，Java 组的实习生小陈在 Nexus Web UI 里"手滑"删除了 `common-security:2.3.1` 这个版本，导致依赖它的 7 个微服务在 CI 流水线中全部构建失败。更糟糕的是，前端组的阿玲发现她的团队成员用 CI 机器人账号登录后，居然能看到（并下载）Java 组的内部 jar 包——这些包中有一些尚未正式发布的内部 API。

炮哥调出审计日志后发现：目前整个 Nexus 只有两个用户——`admin`（所有权限）和 `anonymous`（被禁用了）。所有人共享 `admin` 账号的密码，"谁在什么时候做了什么"完全不可追溯。"一人犯错、全员遭殃"的根本原因是**权限模型缺失**——没有用户隔离、没有角色拆分、没有最小权限原则，整个 Nexus 就是一座没有门卫、没有门禁卡、没有分区钥匙的开放仓库。

本章将彻底解决这个问题：建立用户、角色、权限的完整体系，为 Java、前端、Docker 和运维四个团队配置最小化的读写权限，并创建独立的 CI 机器人账号——确保每个人的操作可追溯，每个团队的制品可隔离。这是从"能用"到"安全可控"的关键一步。

## 2. 项目设计

炮哥把审计日志投到大屏幕上，大师召集了所有组长开会。

**大师**："我先把概念厘清。Nexus 的权限体系分四层：**Privilege（权限）→ Role（角色）→ User（用户）→ Content Selector（内容选择器）**。你们在共享 admin 账号这件事上已经踩了红线，今天必须拆分。"

**小胖**："四层？我只用过用户和密码，角色和权限有什么区别？"

**大师**："打个比方。Privilege 是一把具体的**钥匙**——比如"打开 maven-releases 仓库的门"、"往 npm-hosted 里放东西"。Role 是一个**钥匙串**——你把多把钥匙串在一起，比如'Java 开发者'这个角色，他需要能读 maven-releases、读 maven-snapshots，但不能删。User 是**拿着钥匙串的人**。Content Selector 是**仓库内部的围栏**——同一个仓库里，Java 组只能访问 `/com/cloudwhale/java/` 路径下的东西。"

> **技术映射**：Privilege = 原子权限（对某仓库/某资源的某操作），Role = 权限集合（一组 Privilege 的聚合），User = 身份实体（可以拥有多个 Role），Content Selector = 仓库内路径级隔离（CSEL 表达式匹配路径/格式/坐标）。

**小白**："那 Nexus 内置的那些'Built-in'角色和权限呢？我能直接用吗？"

**大师**："Nexus 出厂自带一批角色，比如 `nx-admin`（管理员）、`nx-anonymous`（匿名用户）。`nx-admin` 是上帝角色，拥有所有权限——不要把它分配给你团队中的任何人。实践中最常用的内置权限是 `nx-repository-view-<format>-<repo>-<action>` 这种模式，比如 `nx-repository-view-maven2-maven-releases-read` 就是只读 maven-releases 仓库的权限。"

**炮哥**："那我给 Java 组长创建一个用户，然后把 nx-admin 角色赋给他不就行了？"

**大师**："绝对不行。这是最常见的反模式——'为了方便给所有人管理员权限'。正确的做法是为每个团队创建自定义角色。举个例子，Java 团队的开发者角色 `role-java-developer` 应该包含："
- `nx-repository-view-maven2-maven-releases-read`（读 release 仓库）
- `nx-repository-view-maven2-maven-releases-add`（上传到 release 仓库）
- `nx-repository-view-maven2-maven-snapshots-read`（读 snapshot 仓库）
- `nx-repository-view-maven2-maven-snapshots-add`（上传到 snapshot 仓库）
- `nx-repository-view-maven2-maven-public-read`（通过 group 读）
- **不包含**：任何 `*-delete` 或 `*-edit` 权限

> **技术映射**：最小权限原则（Least Privilege）——每个角色只包含完成工作所需的最小权限集合，不多给一个。

**小胖**："那前端团队想看 Java 仓库的包行不行？有些公共组件可能前端也需要知道。"

**大师**："可以，这就是'只读'角色存在的意义。创建一个 `role-cross-team-reader` 角色，只包含所有仓库的 `*-read` 权限，赋给需要跨团队浏览的人。但不能给 `role-cross-team-reader` 任何写入权限。"

**小白**："CI 机器人账号呢？它的权限模型应该和人类用户不一样吧？"

**大师**："对。CI 机器人有三个特点：第一，它只需要**极少数**仓库的写入权限（通常是它负责构建的包的 hosted 仓库）；第二，它**不需要**浏览权限和 UI 访问；第三，它应该是**专用的、一人一账号**——每个 CI 项目独立一个机器人用户，这样审计日志里才知道是哪个流水线做了什么事。"

**炮哥**："那我怎么排查权限问题？比如用户说它上传失败，我不知道是他没权限还是仓库配置有问题。"

**大师**："Nexus 的 API 提供了权限检查工具——你可以用 `POST /service/rest/v1/security/privileges/verify` 来测试某个用户是否有某个具体权限。另外，审计日志里记录的是实际发生了什么，而权限排查需要反问'为什么没发生'——这种时候审计日志帮不上忙，需要看 `request.log` 中的 403 详情。"

## 3. 项目实战

### 3.1 环境准备

### 3.2 分步实战

#### 步骤一：创建自定义角色（Role）

**目标**：为 Java 开发者、前端开发者、Docker 运维、CI 机器人各创建一个最小权限角色。

```bash
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

# ---- 1. Java 开发者角色 (读写 Maven, 只读其他) ----
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/security/roles" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "role-java-developer",
    "name": "Java Developer",
    "description": "Java 开发者：可读写 Maven 仓库，只读其他仓库",
    "privileges": [
      "nx-repository-view-maven2-maven-releases-read",
      "nx-repository-view-maven2-maven-releases-add",
      "nx-repository-view-maven2-maven-snapshots-read",
      "nx-repository-view-maven2-maven-snapshots-add",
      "nx-repository-view-maven2-maven-public-read"
    ],
    "roles": []
  }'

# ---- 2. 前端开发者角色 (读写 npm, 只读其他) ----
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/security/roles" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "role-frontend-developer",
    "name": "Frontend Developer",
    "description": "前端开发者：可读写 npm 仓库",
    "privileges": [
      "nx-repository-view-npm-npm-hosted-read",
      "nx-repository-view-npm-npm-hosted-add",
      "nx-repository-view-npm-npm-public-read"
    ],
    "roles": []
  }'

# ---- 3. Docker 运维角色 (读写 Docker, 只读其他) ----
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/security/roles" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "role-docker-ops",
    "name": "Docker Ops",
    "description": "Docker 运维：可读写 Docker 仓库",
    "privileges": [
      "nx-repository-view-docker-docker-hosted-read",
      "nx-repository-view-docker-docker-hosted-add",
      "nx-repository-view-docker-docker-public-read"
    ],
    "roles": []
  }'

# ---- 4. 全局只读角色 (跨团队浏览) ----
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/security/roles" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "role-global-reader",
    "name": "Global Reader",
    "description": "全局只读：可浏览所有仓库但不能修改",
    "privileges": [
      "nx-repository-view-maven2-maven-public-read",
      "nx-repository-view-npm-npm-public-read",
      "nx-repository-view-docker-docker-public-read",
      "nx-repository-view-raw-raw-public-read"
    ],
    "roles": []
  }'

# ---- 5. CI 机器人角色 (只写特定 hosted 仓库) ----
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/security/roles" \
  -H "Content-Type: application/json" \
  -d '{
    "id": "role-ci-java",
    "name": "CI Java Bot",
    "description": "CI 机器人：可发布 Java 制品，不可删除",
    "privileges": [
      "nx-repository-view-maven2-maven-releases-add",
      "nx-repository-view-maven2-maven-snapshots-add"
    ],
    "roles": []
  }'
```

**运行结果**：五个自定义角色创建成功。在 Web UI `Security → Roles` 中可以看到新增的角色及其包含的权限列表。

#### 步骤四：编写权限检查脚本

**目标**：提供一个快速检查用户权限的脚本，用于排障。

```bash
#!/bin/bash
# check-permission.sh：检查某用户是否有指定权限
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

USER_ID="$1"
PRIVILEGE="$2"

if [ -z "$USER_ID" ] || [ -z "$PRIVILEGE" ]; then
    echo "用法: $0 <userId> <privilege-name>"
    echo "示例: $0 zhangsan nx-repository-view-maven2-maven-releases-delete"
    exit 1
fi

# 查询用户的角色
echo "=== 用户 $USER_ID 的角色 ==="
curl -u $AUTH "$NEXUS/service/rest/v1/security/users?userId=$USER_ID" | \
  jq '.roles[]'

# 查询指定角色的权限
echo ""
echo "=== 各角色的权限详情 ==="
for ROLE_ID in $(curl -u $AUTH "$NEXUS/service/rest/v1/security/users?userId=$USER_ID" | jq -r '.roles[]'); do
    echo "--- 角色: $ROLE_ID ---"
    curl -u $AUTH "$NEXUS/service/rest/v1/security/roles/$ROLE_ID" | \
      jq '.privileges[]'
done
```

```bash
chmod +x check-permission.sh
./check-permission.sh zhangsan nx-repository-view-maven2-maven-releases-delete
```

#### 步骤五：禁用匿名访问并验证

**目标**：彻底关闭匿名访问，确认未认证的请求全部返回 401。

```bash
# 1. 查看当前匿名配置
curl -u $AUTH "$NEXUS/service/rest/v1/security/anonymous" | jq .

# 2. 禁用匿名访问
curl -u $AUTH -X PUT "$NEXUS/service/rest/v1/security/anonymous" \
  -H "Content-Type: application/json" \
  -d '{"enabled": false, "userId": "anonymous", "realmName": "NexusAuthorizingRealm"}'

# 3. 验证：不带认证信息访问
curl -s -o /dev/null -w "%{http_code}" \
  "$NEXUS/repository/maven-public/com/google/guava/guava/31.1-jre/guava-31.1-jre.jar"

# 预期输出: 401
```

**运行结果**：匿名访问已关闭，所有未认证请求返回 401。

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| Privilege 命名模式记错 | 创建的角色不生效 | 格式严格遵循 `nx-repository-view-<format>-<repo>-<action>`，format 名从 Web UI 创建仓库页面的下拉框中确认 |
| 用户有角色但操作被拒 | 403 但检查角色权限正确 | 检查是否有其他角色覆盖了权限（Nexus 没有显式 DENY 机制，如果任一角色包含所需权限即可放行） |
| 内置 Privilege 不够用 | 需要的某个粒度权限不存在 | 使用 `nx-repository-view-*-*` 通配符，或创建 Repository View 类型的自定义 Privilege |
| CI 用户密码泄露 | 密码在日志/代码中明文 | 使用 Nexus User Token（而非密码），Token 可独立吊销，不泄露主密码 |
| 更改角色后不生效 | 用户权限似乎有"延迟" | Nexus 的权限决策是实时的，不存在缓存延迟。如果改了角色后用户仍报 403，优先检查角色是否正确保存 |

## 4. 项目总结

### 4.1 权限体系速查表

| 概念 | 含义 | 粒度 | 示例 |
|------|------|------|------|
| Privilege | 对某资源的单一操作权限 | 仓库 + 操作 | `nx-repository-view-maven2-maven-releases-read` |
| Role | 一组 Privilege 的集合 | 多仓库 + 多操作 | `role-java-developer` |
| User | 一个登录身份 | 单个实体 | `zhangsan` |
| Content Selector | 仓库内部的路径级过滤 | 路径/格式/坐标表达式 | `format == "maven2" && path =^ "/com/cloudwhale/java/"` |

### 4.2 适用场景

1. **多团队制品隔离**：Java、前端、Docker、运维各管各的仓库，互不越权
2. **CI/CD 安全**：每个流水线使用独立的机器人账号，权限最小化，审计可追溯
3. **生产环境保护**：只有指定角色能删除/修改生产仓库制品，其他用户只读
4. **外包/实习生管控**：只读浏览 + 沙箱仓库上传，避免误删生产制品
5. **合规审计**：每个用户的操作都绑定到审计日志，满足 SOX/等保要求

**不适用场景**：
1. 个人单机开发环境——admin 账号直接用更省事
2. 团队 < 5 人的初创阶段——过度权限拆分反而降低效率

### 4.3 注意事项

- **nx-admin 角色不要用于日常操作**：保留给管理员进行系统级配置修改，日常仓库操作应使用自定义角色
- **每用户一个账号**：不要共享账号，共享 = 不可审计
- **定期审查角色成员**：每月审查一次角色中的用户列表，删除离职用户
- **Content Selector 仅 PRO 版完整支持**：OSS 版的 Content Selector 功能有限，需要确认是否满足需求
- **权限变更应有审批流程**：生产仓库的删除权限应经过审批

### 4.4 常见踩坑经验

**故障一：用户被分配了多个角色但权限不叠加**

某用户同时拥有 `role-java-developer` 和 `role-frontend-developer`，预期能同时读写 Maven 和 npm，但 npm 操作返回 403。排查发现：`role-frontend-developer` 创建时拼写错误导致权限列表为空。解决：通过 API 检查角色详情，修正后问题消失。

**故障二：修改角色后一批用户同时失去权限**

管理员修改了 `role-java-developer` 角色的权限列表，移除了 `maven-releases-add` 权限，导致 6 个 Java 开发者同时无法发布。根因：直接修改共享角色未做变更通知。解决：建立角色变更流程——新增角色而非修改已有角色，让用户逐步迁移。

**故障三：使用了错误的 repository format 名称**

`nx-repository-view-maven2-maven-releases-read` vs `nx-repository-view-maven-maven-releases-read`——后者不存在但创建角色时不会报错，导致权限检查时始终不匹配。根因：format 名称不是用户自定义的，而是 Nexus 内部固定的（如 `maven2`、`npm`、`docker`、`raw`）。解决：从已有 Privilege 列表中复制准确名称，不要手打。

### 4.5 思考题

1. 如果需要在同一个 Maven hosted 仓库中实现"Java 组 A 只能上传到 `/com/cloudwhale/team-a/` 路径下，Java 组 B 只能上传到 `/com/cloudwhale/team-b/` 路径下"，用基础的 Role + Privilege 能否实现？如果不能，需要什么额外机制？
2. 一个离职员工的账号需要如何处理？直接删除账号和使用 `status: "disabled"` 禁用账号，对审计日志的可追溯性有什么影响？

（第7章思考题答案：1. 路径结构建议为：`/configs/<hostname>/<app>/config.yml`。上传脚本接收主机名和应用名作为参数，自动拼接路径。下载脚本根据 `$HOSTNAME` 环境变量动态构造 URL。2. 在路径规范中固定"版本号"的位置（如 `agent/linux/amd64/<version>/agent-install.sh`），通过搜索 API 查询该路径前缀下的所有文件，对结果按路径字符串排序取最大者（如果版本号格式可排序）；或者维护一个单独的 `latest-version.txt` 元数据文件指向最新版本路径。）

### 4.6 推广计划提示

- **安全部门**：审查角色矩阵和最小权限原则是否落实，确认审计日志的可追溯性
- **开发/测试/运维各团队**：每位成员独立申请自己的 Nexus 账号，由团队 Lead 审批角色分配
- **CI/CD 团队**：创建独立机器人账号，凭据存入 CI 平台的 Secret Manager，不硬编码
