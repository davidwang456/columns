# 第29章：性能调优：下载慢、上传慢与高并发访问

## 1. 项目背景

周一上午 9:30 是云鲸科技的"开工高峰"——200 个开发者同时 `mvn compile`、`npm install`、`docker pull`，Nexus 服务器的 CPU 飙到 90%，HTTP 请求延迟从平时的 50ms 飙升到 4.2 秒。Java 组的老王发现一个 2MB 的 jar 包下载了 35 秒，而直接在 Nexus 宿主机上 `wget` 同一个文件只要 0.3 秒。Docker 组的浩子反映推一个 200MB 的镜像到 `docker-hosted`，速度只有 8MB/s——而宿主机磁盘的写入速度是 300MB/s。

同一个下午，前端组的吴凡投诉 `npm install` 经常中断："我们的前端项目有 937 个依赖包，npm 会对每个包单独发 HTTP GET 请求——900 多个小请求把 Nexus 的线程池打满了，后面的请求排队超时。"而 Maven 的传输就快得多——因为它对一个 artifact 的多个文件（jar/pom/sources）使用持久连接。

这些都是**性能瓶颈分散在不同层次**的典型案例——瓶颈可能在客户端参数（Maven 的并发下载设置）、在网络（带宽限制/TCP 窗口）、在 Nexus JVM（堆/GC/线程池）、在 BlobStore（磁盘 IO）、在远程仓库（Maven Central 延迟）。本章将建立从客户端到存储的端到端性能分析框架，针对三种典型负载（Maven 下载、Docker 拉取、npm 小文件风暴）输出调优方案，并用压测工具验证效果。

## 2. 项目设计

浩子、老王和吴凡各自描述了性能问题，大师一一拆解瓶颈。

**大师**："先说 Java 组的问题——'近距离下载慢'。一个 2MB 的 jar 从宿主机 wget 只要 0.3 秒，但 Maven 下载要 35 秒——差距 100 倍。可能的原因有三个。第一，Maven 客户端默认只用单线程下载，且有内部的 Wagon 实现约束。第二，如果 Maven 走的是 HTTPS 而你的证书校验有延迟——虽然 Nexus 内部用 HTTP，但如果你给客户端配置了 HTTPS 反向代理，SSL 握手的开销很大。第三，也可能是最可能的原因——你的 `settings.xml` 中 mirror url 配错了，Maven 实际走了远程 Maven Central 而非本地 Nexus。"

**老王**："如果真的是 Maven 走了远程——那 35 秒也是有道理的。但怎么快速验证？"

**大师**："三个命令。第一，在 Maven 编译时加 `-X` 参数看 DEBUG 日志——日志中会明确显示 `Downloading from <repository-id>: <full-url>`，一眼就能看出是从 Nexus 还是从 Maven Central 下载。第二，查看 Nexus 的 `request.log`——如果日志中有对应请求，说明到了 Nexus；如果没有，说明根本没到。第三，在用户机器上 `curl -w '%{time_total}'` 测试直连 Nexus 的延迟。"

> **技术映射**：性能诊断铁三角——客户端日志（谁在哪慢的）、Nexus request.log（请求是否到达 + 处理耗时）、网络层面（ping/curl 基准延迟）。三者交叉定位瓶颈。

**浩子**："Docker push 慢呢？宿主机磁盘 300MB/s，push 速度只有 8MB/s。"

**大师**："Docker push 的瓶颈往往不在存储层，而在**网络层 + 协议层**。Docker 客户端和 Nexus 之间是 HTTP/1.1，大文件上传走 chunked transfer encoding——每个 chunk 需要一次 TCP ACK 确认。如果客户端和 Nexus 之间有防火墙或负载均衡，可能引入了额外的延迟。此外，如果用了 `insecure-registries` 而非 HTTPS，HTTP 本身的性能是 OK 的。最后，确保 Nexus 的 `MaxDirectMemorySize` 足够大——它用 Direct ByteBuffer 做大文件 IO，如果直接内存不足会回退到堆内 IO，性能下降 3-5 倍。"

> **技术映射**：Docker push 性能 = 磁盘 IO（最小影响）+ TCP 窗口大小（中等影响）+ 直接内存大小（大文件最大影响）+ 中间网络设备（防火墙/负载均衡的缓冲限制）。

**吴凡**："npm install 937 个包的问题呢？每个小包都要一个 HTTP 请求，把 Nexus 打满了。"

**大师**："npm 的'小文件风暴'是制品仓库的经典性能挑战——937 个请求意味着 937 次 TCP 握手、937 次权限校验、937 次 BlobStore 查询。优化方向有三个。第一，**客户端侧**——npm 7+ 支持 `prefer-offline` 和 `prefer-online` 策略，合理配置减少请求数。第二，**Nexus 侧**——增加 HTTP 线程池大小（`${jetty.threadPool.maxThreads}` 默认 200，可调到 400）、确保频繁访问的小文件被 OS 页缓存命中。第三，**架构侧**——将 npm proxy 的 `metadataMaxAge` 调到 60 分钟以上，这样 npm install 的 metadata 查询全部命中本地缓存，只有首次 miss 时才请求远程。"

## 3. 项目实战

### 3.1 环境准备

- 已部署 Nexus 实例
- JMeter 或 wrk（HTTP 压测工具）
- Apache Bench（ab）或 vegeta
- Maven 3.8+、Docker CLI

### 3.2 分步实战

#### 步骤一：Nexus JVM 调优

**目标**：优化 Nexus 的 JVM 启动参数以匹配实际负载。

```bash
# 当前 JVM 参数检查
docker compose exec nexus java -XX:+PrintFlagsFinal -version 2>&1 | grep -E "(MaxHeapSize|MaxDirectMemorySize|ThreadStackSize)"

# 推荐生产环境 JVM 参数（在 docker-compose.yml 中配置）
# INSTALL4J_ADD_VM_PARAMS=\
#   -Xms2048m \                          # 初始堆大小
#   -Xmx4096m \                          # 最大堆大小（物理内存的 50-70%）
#   -XX:MaxDirectMemorySize=1024m \      # 直接内存（至少为最大上传文件大小的 2 倍）
#   -XX:+UseG1GC \                       # 使用 G1 垃圾收集器（大堆低延迟首选）
#   -XX:MaxGCPauseMillis=200 \           # G1 期望最大 GC 暂停时间
#   -XX:ParallelGCThreads=4 \            # GC 并行线程数
#   -XX:ConcGCThreads=2 \                # 并发 GC 线程数
#   -Djava.util.concurrent.ForkJoinPool.common.parallelism=8 \
#   -Dnexus.threads.jetty.max=400        # HTTP 线程池最大线程数
```

**应用 JVM 参数**：

```bash
# 编辑 docker-compose.yml
# 在 environment 中更新 INSTALL4J_ADD_VM_PARAMS 为上述推荐值
docker compose down && docker compose up -d

# 验证新参数
docker compose exec nexus java -XX:+PrintFlagsFinal -version 2>&1 | grep MaxHeapSize
```

#### 步骤二：诊断 Maven 下载慢

**目标**：用系统化方法定位 Maven 下载的瓶颈层次。

```bash
#!/bin/bash
# maven-perf-diag.sh：Maven 下载性能诊断
NEXUS="http://localhost:8081"
AUTH="admin:admin123"
TEST_ARTIFACT="/repository/maven-public/com/google/guava/guava/31.1-jre/guava-31.1-jre.jar"

echo "=== Maven 下载性能诊断 ==="

# 1. 基准测试——curl 直连 Nexus
echo "[1] curl 直连延迟测试 (3 次取平均):"
for i in 1 2 3; do
    curl -s -o /dev/null -w "%{time_total}\n" -u "$AUTH" "$NEXUS$TEST_ARTIFACT"
done | awk '{sum+=$1; count++} END {printf "  平均: %.2f 秒\n", sum/count}'

# 2. 检查 Maven 实际连接的仓库来源
echo "[2] 检查 Maven 编译日志:"
echo "  执行 mvn compile -X 2>&1 | grep 'Downloading from'"
echo "  确认 URL 前缀是否为 Nexus 地址"
echo ""

# 3. 检查 Nexus request.log 中对应请求的耗时
echo "[3] 在 Nexus 中检查该请求的耗时:"
docker compose exec nexus grep "guava-31.1-jre.jar" /nexus-data/log/request.log | tail -3
echo ""

# 4. 检查是否有 GC 影响
echo "[4] 最近 GC 情况:"
docker compose exec nexus tail -30 /nexus-data/log/jvm.log 2>/dev/null | grep -E "GC|Full GC" | tail -5
echo ""

echo "=== 诊断完成 ==="
echo "建议:"
echo "  - curl 慢 + Nexus request 耗时短 → 网络/客户端问题"
echo "  - curl 快 + Maven 慢 → 检查 mirror 和 settings.xml 配置"
echo "  - 两者都慢 + GC 频繁 → JVM 堆不足，考虑扩大 -Xmx"
```

```bash
chmod +x maven-perf-diag.sh && ./maven-perf-diag.sh
```

#### 步骤三：HTTP 线程池与连接数调优

**目标**：增大 Nexus HTTP 线程池以应对高并发 npm 和 Docker 请求。

```bash
# 查看当前线程池配置
docker compose exec nexus cat /nexus-data/etc/nexus.properties | grep -i thread

# 在 nexus.properties 中追加调优参数
docker compose exec nexus bash -c 'cat >> /nexus-data/etc/nexus.properties << EOF

# HTTP 线程池调优
nexus.threads.jetty.min=20
nexus.threads.jetty.max=400
nexus.threads.jetty.idleTimeout=30000

# 连接池调优
nexus.httpclient.maxConnections=200
nexus.httpclient.maxConnectionsPerRoute=50
EOF'

# 重启生效
docker compose restart nexus
```

#### 步骤五：Docker 大文件上传性能对比

**目标**：测试不同 `MaxDirectMemorySize` 对 Docker push 速度的影响。

```bash
#!/bin/bash
# docker-perf-test.sh：对比不同直接内存配置的 push 性能
NEXUS_DOCKER="localhost:5000"

echo "=== Docker Push 性能测试 ==="

# 生成 500MB 测试镜像
dd if=/dev/urandom of=/tmp/bigfile.bin bs=1M count=500 2>/dev/null
cat > /tmp/Dockerfile.perf << 'DOCKER'
FROM alpine:3.19
COPY bigfile.bin /data/bigfile.bin
DOCKER

cd /tmp
docker build -t ${NEXUS_DOCKER}/perf-test:big -f Dockerfile.perf .

# 测试 1：默认直接内存
echo "[测试1] MaxDirectMemorySize=512m (默认)"
docker compose restart nexus
sleep 30
time docker push ${NEXUS_DOCKER}/perf-test:big 2>&1 | tail -5

# 测试 2：加大直接内存
echo "[测试2] MaxDirectMemorySize=2048m"
# 修改 docker-compose.yml 中 MaxDirectMemorySize=2048m
docker compose down && docker compose up -d
sleep 30
docker rmi ${NEXUS_DOCKER}/perf-test:big
docker pull ${NEXUS_DOCKER}/perf-test:big  # 先拉回再测push
docker tag ${NEXUS_DOCKER}/perf-test:big ${NEXUS_DOCKER}/perf-test:big2
time docker push ${NEXUS_DOCKER}/perf-test:big2 2>&1 | tail -5

echo "=== 完成：对比两次 push 的 real time 差异 ==="
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| `FileDescriptor` 耗尽 | `lsof -p <pid>` 显示 fds 近 65535 | `ulimits.nofile` 从默认的 1024 调整为 65536 |
| HTTP 连接泄漏 | `request.log` 中大量等待 > 30s | 检查 Nginx 反向代理的 `proxy_read_timeout` 和 `keepalive_timeout` |
| GC 频繁导致请求超时 | 每次 GC 耗时 1-2 秒 | 改用 G1GC + 调大堆 + 调低 `MaxGCPauseMillis` |
| Docker push 极慢但有大量小文件 | 每个文件一次 HTTP PUT + TCP 开销 | 换用 `docker save` + Raw 仓库上传 tar（单次大文件传输） |

## 4. 项目总结

### 4.1 Nexus 性能参数速查

| 参数 | 默认值 | 推荐值（100+ 并发） | 说明 |
|------|--------|-------------------|------|
| `-Xmx` | 2703m | 4096m | JVM 最大堆 |
| `MaxDirectMemorySize` | 随 -Xmx | 1024-2048m | 大文件 IO 直接内存 |
| `jetty.maxThreads` | 200 | 400 | HTTP 处理线程数 |
| `httpclient.maxConnections` | 200 | 200-400 | 远程代理连接池 |
| `ulimit -n` | 1024 | 65536 | 文件描述符上限 |

### 4.2 适用场景

1. **团队规模增长**：从 30 人到 200 人，Nexus 的默认配置需要跟随调优
2. **大文件场景**：Docker 镜像 > 500MB 或 Raw 大文件上传需要调直接内存
3. **小文件风暴**：npm 依赖多、Maven 插件多时需要增大线程池
4. **跨国团队**：远程拉取 Maven Central 延迟高，需要优化 proxy 缓存策略

**不适用场景**：
1. 团队 < 20 人——默认配置完全够用
2. 性能问题 100% 在客户端或网络——调 Nexus 只是缘木求鱼

### 4.3 常见踩坑经验

**故障一：Nginx 反向代理 buffer 太小导致大文件下载极慢**

前端团队下载一个 200MB 的 tgz 包时速度只有 2MB/s，但直接 curl 访问 Nexus 8081 端口速度有 80MB/s。根因：Nginx 反向代理的 `proxy_buffer_size` 和 `proxy_buffers` 使用默认值（8K），大文件响应被拆成大量小块逐块传输，TCP 窗口效率极低。解决：在 Nginx 的 `location /repository/` 块中添加 `proxy_buffering off;` 或在超大文件场景使用 `proxy_max_temp_file_size 0;` 禁用缓冲。

**故障二：OS 页缓存被 Blob 读写频繁驱逐**

某团队发现在执行 Compact BlobStore 任务时，正在进行的 Maven 下载速度从 50ms 飙升到 800ms。根因：Compact 任务扫描和删除大量 blob 文件导致操作系统的页缓存（Page Cache）被频繁更新，之前缓存在内存中的热门 jar 文件被驱逐出缓存，需要重新从磁盘读取。解决：将 Compact 任务严格限制在凌晨业务低峰期执行；为 Nexus 部署独立的 SSD 存放热门制品；调整 Linux 的 `vm.vfs_cache_pressure` 参数降低缓存回收激进程度。

**故障三：Docker 大 layer 推送失败但无错误日志**

CI 推送 2GB 的 Docker layer 到 Nexus，在 95% 进度时连接断开，Docker 客户端报 `EOF`，Nexus 日志中无对应错误。根因：Nginx 的 `client_max_body_size` 默认 1MB——但实际上 Docker push 走的是 chunked transfer encoding，理论上不受此限制。真正的原因是 Nginx 的 `proxy_read_timeout`（默认 60s）在大文件传输中触发超时。解决：增大 `proxy_read_timeout 600s` 和 `proxy_send_timeout 600s`；在 `location` 块中添加 `client_body_timeout 600s`。

### 4.4 注意事项

- **调优前先建立基线**：用本章压测方法记录调优前的 QPS 和延迟，调优后对比
- **不要一次调太多参数**：一次只改一个变量，观察效果再继续
- **`MaxDirectMemorySize` 不要超过物理内存的 50%**：直接内存不受 JVM GC 管理但占用物理内存
- **高并发时关注 `request.log` 的最后列（响应耗时）**：找出慢请求的模式

### 4.5 思考题

1. 压测发现 Nexus 在 100 并发时 QPS 正常，但升到 200 并发时 QPS 反而下降。从 JVM、HTTP 线程池、文件描述符三个维度分析可能的瓶颈。
2. 某团队发现上午 9:30-10:00 的下载延迟异常高，但 Nexus CPU 和内存都正常。基于本章的性能层次模型，列出可能的非 Nexus 瓶颈点并一一排查方法。

（第28章思考题答案：1. Groovy 脚本：`import org.sonatype.nexus.repository.manager.RepositoryManager; import org.sonatype.nexus.repository.Repository; def repoManager = container.lookup(RepositoryManager.class.name); repoManager.browse().each { repo -> if (repo.type.value == 'proxy') { def status = repo.status.health; def healthy = status.healthy ? 1 : 0; log.info("nexus_proxy_health_status{repository=\"${repo.name}\"} ${healthy}"); } }` 将此脚本注册为定时任务，通过 JMX Exporter 暴露为 Prometheus 指标。2. SLI 计算：`sum(rate(nexus_http_requests_total{status!~"5.."}[28d])) / sum(rate(nexus_http_requests_total[28d]))` 得到过去 28 天的成功率。SLO：成功率 > 99.9%。在 Prometheus 中用 `(1 - SLI) > 0.001` 设置告警。）

### 4.5 推广计划提示

- **运维部门**：将性能诊断脚本（maven-perf-diag.sh）和压测方案纳入定期巡检
- **开发团队**：当报告"下载慢"时，先运行诊断脚本定位问题层级再决定谁来修
- **架构组**：审查 JVM 配置是否与硬件资源匹配（内存/CPU 核数）
