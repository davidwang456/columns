# 第12章：清理策略与过期制品治理

## 1. 项目背景

云鲸科技的 Nexus 磁盘从 68GB 涨到 94GB 只用了三周。炮哥在第 11 章学会了不直接摸 Blob 文件，但他面对的问题是：如果不手动删文件，那"正规途径"的清理怎么操作？

Java 组的 `maven-snapshots` 仓库里积压了 240 个历史 SNAPSHOT 版本（有些从 6 个月前就没人用过）；前端组的 `npm-hosted` 里有 37 个 `alpha`、`beta`、`rc` 包永远停留在了实验阶段；Docker 组的 `docker-hosted` 里 60% 的标签是 CI 每次构建自动生成的 commit-hash 临时标签。测试组的阿玲每周跑一次冒烟测试，每次都往 `maven-snapshots` 丢一批依赖，测试完就走人，从不清理。

更严重的是，炮哥在周一晨会上提出"做个清理"，老王立刻反对："别乱删！上周有个线上用户反馈问题，我们翻出了一个三个月前的 SNAPSHOT 版本来复现。你要是全清了，我们没法做历史回溯。"浩子也附和："Docker 那个 `latest` 标签不能删，生产环境回滚全靠它。"

清理困境的本质是：**不清理，磁盘会满；清理，可能误删有用的历史版本；不同团队的清理需求完全不同，没有一个统一的规则能覆盖所有场景**。Nexus 的 Cleanup Policy + Cleanup Task 体系正是解决这一困局的机制——它为每种制品类型、每个仓库、每个版本生命周期阶段定义独立的保留与淘汰规则。

## 2. 项目设计

炮哥在白板上列出了四个组的清理需求，大师逐条分析。

**炮哥**："Java 组的 maven-snapshots：保留最近 7 天发布的版本，删除 7 天前的。前端组的 npm-hosted：删除所有标签含 `alpha`、`beta`、`rc` 且 30 天未下载的包。Docker 组的 docker-hosted：只保留每个仓库最近 5 个 tag 的镜像，删除 commit-hash 临时标签。这三个需求，Nexus 能同时满足吗？"

**大师**："能，但不是一条策略全搞定。Nexus 的 Cleanup Policy 是基于**匹配条件 + 时间窗口**来筛选待删除的组件，然后通过**Cleanup Task**来执行。关键来了一一你要为不同的仓库创建不同的 Policy，然后关联到对应的仓库。"

**小胖**："Cleanup Policy 能按什么条件匹配？能按'最近有没下载过'来决定删不删吗？"

**大师**："Cleanup Policy 支持四类条件。第一，**格式匹配**——只对 maven2、npm、docker、raw 中的某一种生效。第二，**版本匹配**——通过正则表达式匹配 version 字段，比如 `.*-SNAPSHOT` 匹配所有快照版本。第三，**时间条件**——'发布于多少天前'（age）或'距离上次下载已过多少天'（last downloaded）。第四，**资产名称正则**——按 asset 文件名匹配。组合使用这些条件，能覆盖大部分场景。"

> **技术映射**：Cleanup Policy = 筛选条件（WHERE 子句），Cleanup Task = 执行器（DELETE FROM ... WHERE ...），关联仓库 = 作用域（ON repository）。

**小白**："那些被 Cleanup Task 删除的组件，它们的 Blob 是立即物理删除还是软删除？"

**大师**："Cleanup Task 执行的是组件软删除——标记 `deleted=true`，组件从搜索结果中移除。但 Blob 文件不会立即释放，需要后续的 `Admin - Compact blob store` 任务来物理回收。所以有效清理的一条完整链路是：**Cleanup Policy → Cleanup Task → Compact BlobStore → 磁盘释放**。"

**小胖**："这跟食堂收碗一样——阿姨先收碗（Cleanup Task），放回收车里（软删除），洗碗工再洗（Compact BlobStore），碗才能再次使用（磁盘释放）。"

**大师**："精准。"

**浩子**："Docker 的清理有什么特殊之处？我之前手动删了一个 tag，空间一点没少。"

**大师**："Docker 的清理最复杂，因为有**layer 共享**。一个 Docker base layer 可能被 10 个 tag 引用，你随便删一个 tag，这些 layer 不能释放——因为其他 tag 还在用。正确的 Docker 清理流程分两步：第一步，**Cleanup Policy 删除 tag（manifest）**；第二步，执行 `Delete unused manifests and images` 任务，它扫描所有 Docker 仓库中的 manifest，找出没有被任何 tag 引用的'悬空'manifest 和只有它引用的 layer，然后删除。第三步才是 Compact BlobStore。"

> **技术映射**：Docker 清理 = tag 删除（Cleanup Policy）→ 悬空 manifest/layer 删除（专用 Task）→ 物理回收（Compact BlobStore）。三层瀑布式清理，缺一不可。

**老王**："如果我想在执行清理前先预览——'看看删了哪些东西但先不真删'——能做吗？"

**大师**："这正是 Nexus Cleanup 的'Preview 模式'。你在创建 Cleanup Task 时用 preview 模式执行，它不会真的删除任何东西，而是输出一份清单——列出所有匹配条件的组件名称、版本、仓库。你审核无误后，再正式切换为 delete 模式执行。"

## 3. 项目实战

### 3.1 环境准备

- 已部署 Nexus 实例
- 有已填充的 Maven、npm、Docker 仓库
- curl、jq

### 3.2 分步实战

#### 步骤一：为 Maven SNAPSHOT 创建清理策略

**目标**：创建一条保留最近 7 天、删除过期 SNAPSHOT 的策略。

```bash
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

# 创建 Cleanup Policy（Maven SNAPSHOT 清理）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/cleanup-policies" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "cleanup-maven-snapshots",
    "format": "maven2",
    "notes": "清除 7 天前的 Maven SNAPSHOT 版本",
    "criteriaLastBlobUpdated": 7,
    "criteriaLastDownloaded": 0,
    "criteriaReleaseType": "SNAPSHOT",
    "criteriaAssetRegex": ".*\\\\.jar$"
  }'
```

**参数说明**：
- `criteriaLastBlobUpdated: 7`：Blob 最后一次更新距今超过 7 天
- `criteriaLastDownloaded: 0`：不检查最后下载时间（0 = 不检查）
- `criteriaReleaseType: SNAPSHOT`：只匹配 SNAPSHOT 版本
- `criteriaAssetRegex`：只清理 jar 文件，保留 pom 和 sources

#### 步骤二：为 npm beta 包创建清理策略

**目标**：清理 30 天未下载的 pre-release 包。

```bash
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/cleanup-policies" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "cleanup-npm-prerelease",
    "format": "npm",
    "notes": "清除 30 天未下载且版本名含 alpha/beta/rc 的 npm 包",
    "criteriaLastBlobUpdated": 0,
    "criteriaLastDownloaded": 30,
    "criteriaReleaseType": null,
    "criteriaAssetRegex": ".*-(alpha|beta|rc).*\\\\.tgz$"
  }'
```

#### 步骤三：为 Docker 临时镜像创建清理策略

**目标**：删除超过 14 天且按名称匹配临时标签的 Docker 镜像。

```bash
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/cleanup-policies" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "cleanup-docker-temp",
    "format": "docker",
    "notes": "清除 14 天前的 Docker 临时标签（commit-hash、dev、test）",
    "criteriaLastBlobUpdated": 14,
    "criteriaLastDownloaded": 0,
    "criteriaReleaseType": null,
    "criteriaAssetRegex": ".*"
  }'
```

**运行结果**：三条 Cleanup Policy 创建成功。在 Web UI `Administration → Repository → Cleanup Policies` 中可见新增策略。

#### 步骤四：将策略关联到仓库

**目标**：将清理策略绑定到具体仓库。

```bash
# 将 maven SNAPSHOT 清理策略关联到 maven-snapshots 仓库
curl -u $AUTH -X PUT "$NEXUS/service/rest/v1/repositories/maven/hosted/maven-snapshots" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "maven-snapshots",
    "online": true,
    "storage": {
      "blobStoreName": "default",
      "strictContentTypeValidation": true,
      "writePolicy": "ALLOW"
    },
    "cleanup": {
      "policyNames": ["cleanup-maven-snapshots"]
    },
    "maven": {
      "versionPolicy": "SNAPSHOT",
      "layoutPolicy": "STRICT"
    }
  }'

# 将 npm pre-release 清理策略关联到 npm-hosted
curl -u $AUTH -X PUT "$NEXUS/service/rest/v1/repositories/npm/hosted/npm-hosted" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "npm-hosted",
    "online": true,
    "storage": {
      "blobStoreName": "default",
      "strictContentTypeValidation": true,
      "writePolicy": "ALLOW_ONCE"
    },
    "cleanup": {
      "policyNames": ["cleanup-npm-prerelease"]
    }
  }'

# 将 Docker 临时镜像策略关联到 docker-hosted
curl -u $AUTH -X PUT "$NEXUS/service/rest/v1/repositories/docker/hosted/docker-hosted" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "docker-hosted",
    "online": true,
    "storage": {
      "blobStoreName": "default",
      "strictContentTypeValidation": true,
      "writePolicy": "ALLOW"
    },
    "cleanup": {
      "policyNames": ["cleanup-docker-temp"]
    },
    "docker": {
      "v1Enabled": false,
      "forceBasicAuth": true,
      "httpPort": 5000
    }
  }'
```

#### 步骤五：Preview 模式预演清理

**目标**：用 Preview 模式查看哪些组件将被清理，确认无误后再正式执行。

```bash
# 创建 Preview 模式的 Cleanup Task
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "repository.cleanup",
    "name": "预览清理 maven-snapshots",
    "typeId": "repository.cleanup",
    "schedule": "manual",
    "properties": {
      "repositoryName": "maven-snapshots",
      "preview": "true"
    }
  }'
```

在 Web UI `System → Tasks` 中找到刚创建的任务，点击 `Run`。任务执行完毕后，查看日志：

```bash
# 获取任务执行记录
TASK_ID=$(curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/tasks" | jq -r '.items[] | select(.name=="预览清理 maven-snapshots") | .id')

echo "请在 Web UI System → Tasks → 点击任务 → 查看执行日志"
echo "Preview 日志会列出所有将被清理的组件（GAV + 仓库）"
```

**执行 Preview 后确认无误**，再创建正式版本的 Cleanup Task：

```bash
# 创建正式的 Cleanup Task（非 preview）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "repository.cleanup",
    "name": "执行清理 maven-snapshots（正式）",
    "typeId": "repository.cleanup",
    "schedule": "weekly",
    "properties": {
      "repositoryName": "maven-snapshots",
      "preview": "false"
    },
    "alertEmail": "ops@cloudwhale.com"
  }'
```

#### 步骤六：配置 Docker 专项清理链路

**目标**：配置完整的 Docker 清理链路——Cleanup + Delete unused manifests + Compact。

```bash
# 1. Cleanup Policy 已在步骤三创建并关联

# 2. 创建 Delete unused manifests 任务
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "docker.delete-unused",
    "name": "清理无引用的 Docker manifest",
    "typeId": "docker.delete-unused",
    "schedule": "weekly",
    "properties": {
      "repositoryName": "*"
    }
  }'
# 注意: repositoryName 为 "*" 表示扫描所有 Docker 仓库
```

**运行结果**：完整的 Docker 清理链路配置完成。执行顺序：每周先执行 Cleanup Task（删除过期 tag），再执行 `Delete unused manifests`（清理悬空 manifest 和 layer），最后 Compact BlobStore（物理回收）。

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| Policy 创建但不关联仓库 | 任务执行后无任何效果 | 在仓库设置中关联 `cleanup.policyNames`，或在 Web UI 的仓库编辑页面选择策略 |
| Preview 和 Delete 用同一个 Task | 每次执行前都要手动确认 | 创建两个独立 Task——一个 permanent preview，一个 weekly delete |
| `criteriaLastDownloaded` 设置不当 | 刚发布就被删除 / 永远删不掉 | 生产包设 0（不检查下载时间），测试包可设 14~30 天 |
| Docker cleanup 只删 tag 不删 manifest | 磁盘空间无变化 | 必须部署 `Delete unused manifests` 任务 + `Compact BlobStore` 任务 |
| 正则表达式未转义 | 策略不生效 | JSON 中 `\\.` 需要转义为 `\\\\.` |

## 4. 项目总结

### 4.1 清理体系速查

| 组件 | 作用 | 时机 |
|------|------|------|
| Cleanup Policy | 定义"哪些组件该被删"的条件规则 | 按需配置 |
| Cleanup Task (Preview) | 预览匹配的组件清单 | 正式清理前 |
| Cleanup Task (Delete) | 执行软删除 | 定时 / 手动 |
| Delete unused manifests (Docker) | 清理无 tag 引用的 Docker manifest/layer | 每周 |
| Compact BlobStore | 物理回收已删除 blob 的磁盘空间 | 每周 |

### 4.2 各格式推荐的清理策略模板

| 格式/仓库 | 发布时间 | 最后下载 | 版本类型 | 保留策略 |
|-----------|---------|---------|---------|---------|
| Maven SNAPSHOT | > 7 天 | 不检查 | SNAPSHOT | 删除 |
| Maven RELEASE | 不检查 | 不检查 | RELEASE | 永不清除 |
| npm pre-release | > 30 天 | > 30 天 | alpha/beta/rc | 删除 |
| npm RELEASE | 不检查 | 不检查 | 正式版本 | 永不清除 |
| Docker commit tag | > 14 天 | 不检查 | 临时 tag | 删除 |
| Docker semantic tag | 不检查 | 不检查 | 正式版本 | 保留最近 5 个 |
| Raw 文件 | > 90 天 | > 90 天 | 所有 | 删除 |

### 4.3 注意事项

- **Preview 永远是第一步**：在生产仓库上执行清理前，先用 Preview 模式确认匹配的组件是否正确
- **Release 版本基本不删**：一旦发布到 RELEASE 仓库，应视为永久保存，被删除的 release 可能导致下游构建失败
- **Docker 清理三部曲不可少**：tag 删除 → 悬空 manifest 清理 → Compact BlobStore，跳步骤会导致空间无法释放
- **清理任务应在低峰期执行**：大规模删除会产生大量数据库 IO，可能影响正常上传下载性能

### 4.4 常见踩坑经验

**故障一：Cleanup Policy 误删正在使用的 SNAPSHOT**

某开发团队正在联调期间，`cleanup-maven-snapshots` 策略按计划执行，把所有超过 7 天的 SNAPSHOT 全删了，包括一个仍在使用的 `common-data:2.0-SNAPSHOT`。根因：策略中 `criteriaLastDownloaded` 设为 0，只要发布时间超 7 天就删，不检查是否仍在下载。解决：将 `criteriaLastDownloaded` 改为 7，确保最近 7 天有人用过的 SNAPSHOT 保留。

**故障二：npm scope 包的正则匹配失败**

`cleanup-npm-prerelease` 策略的 `criteriaAssetRegex` 是 `.*-alpha.*\\.tgz$`，但 `@scope/pkg` 格式的 scope 包路径中的 `@` 和 `/` 导致正则不匹配。根因：npm scope 包的 asset path 是 `@scope/pkg/-/pkg-1.0.0-alpha.tgz`，路径中包含 `/`。解决：调整正则适配实际路径格式。

**故障三：Docker 清理后磁盘反而增长**

执行完 Docker Cleanup Task 和 Delete unused manifests 后，磁盘空间非但没有减少反而增加了。根因：Delete unused manifests 任务在清理过程中需要重建 Docker 仓库的索引，期间写入的临时数据超过了删除释放的空间。解决：确保 Compact BlobStore 排在 Delete unused manifests 之后执行，待所有回收完成后观察最终结果。

### 4.5 思考题

1. 如果一条 Cleanup Policy 同时设置了 `criteriaLastBlobUpdated: 7` 和 `criteriaLastDownloaded: 7`，两者的逻辑关系是 AND 还是 OR？一个 3 天前发布、但从未被下载的组件会被清理吗？
2. 你所在的团队需要确保"所有已发布到 `maven-releases` 的 GA（正式发布）版本永远不会被清理，但 RC（候选发布）版本在 GA 发布后自动清理"。如何用 Cleanup Policy 的组合实现这一需求？

（第11章思考题答案：1. Blob 本身没有"引用计数"——Nexus 使用的是软删除机制。当一个 Component 被删除时，它引用的所有 Asset Blob 被标记为 soft-deleted。如果其他 Component 仍引用同一个 Blob，该 Blob 会保留（因为还有 active 的引用）。只有当所有引用该 Blob 的 Component 都被删除后，Compact BlobStore 才会物理回收该 Blob。所以 Docker 的 shared base layer 不会被误删——只要还有至少一个 tag 引用它。2. 零停机迁移方案：① 在 HDD 上创建新的 BlobStore（如 `default-hdd`）；② 在 new BlobStore 上创建新仓库，绑定相同类型的仓库配置；③ 通过 Nexus REST API 遍历 `default` 仓库的组件，使用 `GET /components/{id}/assets` 下载 asset，再 `PUT` 到新仓库；④ 更新 group 仓库成员列表，将新仓库加入并移除旧仓库；⑤ 旧仓库和 BlobStore 保留 1 周观察期后删除。风险点：大文件迁移可能耗尽带宽，需限速；迁移期间有新组件上传到旧仓库，导致数据不一致；Docker 镜像迁移后 digest 可能变化。）

### 4.6 推广计划提示

- **开发/测试部门**：确认各自仓库的清理策略是否合理，重点关注"正在联调的 SNAPSHOT 是否会被误删"
- **运维部门**：配置每周的 Cleanup → Delete unused manifests → Compact BlobStore 三段式任务链，确保任务按正确顺序执行
- **安全部门**：关注清理策略的 `criteriaReleaseType`，确保 RELEASE 版本的清理经过审批
