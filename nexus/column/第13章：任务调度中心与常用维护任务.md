# 第13章：任务调度中心与常用维护任务

## 1. 项目背景

云鲸科技的 Nexus 已经运行了三个多月，炮哥在第 12 章配置好了三段式清理链路，但他面临一个新问题——全公司有 6 个人的团队都在用 `admin` 账号，有人觉得某个任务"看着没用"就随手点了 `Run`，有人不小心 `Stop` 了一个正在执行的任务，还有人把 Cleanup Task 的执行频率从 `weekly` 改成了 `manual` 导致清理中断了三周而无人察觉。

更严重的是，上周四凌晨运维组发现 Nexus 响应极慢——Maven 下载一个 2MB 的 jar 花了 45 秒。排查后发现，两个重 IO 的维护任务——`Rebuild Maven repository metadata` 和 `Compact blob store`——被错误地安排在了同一个周六凌晨窗口执行，CPU 和 IO 同时打满。

"我们到底有哪些任务在跑？什么时间跑？谁创建的？跑的结果是成功还是失败？上次谁把它停掉了？"——炮哥在周会上连珠炮似的问了一串问题，没有一个能立刻回答。Nexus 的任务调度系统就像一间没有值班表的锅炉房——每个人都在加煤，没人知道火力有多大、会不会爆炸、下次检修是什么时候。

本章将建立完整的任务管理视图，掌握所有内置任务类型的用途和风险，编排一套经过验证的生产环境排期方案，让维护任务从"随手点一下"变成"有计划的自动化"。

## 2. 项目设计

炮哥把所有任务列表投屏，上面密密麻麻 40 多条，很多是手抖创建的重复任务。

**炮哥**："大师，这任务列表跟垃圾堆一样。哪些必须有？哪些可以删？"

**大师**："Nexus 的任务分四大类。第一，**数据修复类**——`Rebuild repository index`（重建搜索索引）、`Repair - Rebuild Maven repository metadata`（重建 maven-metadata.xml）。第二，**清理回收类**——`Admin - Cleanup repositories using their associated policies`（清理任务）、`Admin - Compact blob store`（压实 BlobStore）、`Docker - Delete unused manifests and images`。第三，**元数据类**——`Publish Maven indexes`、`Reconcile component attributes`。第四，**自定义类**——Groovy 脚本任务。你现在的 40 多条里，至少有 20 条是重复创建的 Cleanup Task。"

> **技术映射**：Nexus 任务类型按用途分为数据修复（索引/元数据）、存储维护（清理/压实）、格式特定（Maven/npm/Docker/Raw 专项）、自定义脚本四大类。

**小胖**："那我能不能把 Cleanup 和 Compact 两个任务排在一起连续执行？"

**大师**："可以，但它们必须**串行而非并行**。Cleanup Task 执行软删除后，Compact BlobStore 才能回收——如果两者同时跑，Compact 可能在 Cleanup 删除完之前就错过了大量可回收的 blob。建议 Cleanup 每周三跑，Compact 每周六跑，中间留两天让软删除标记充分传播。"

**小白**："Rebuild Maven metadata 这个任务什么时候需要执行？"

**大师**："常规情况下**不需要**——Nexus 在每次 deploy 时自动更新 metadata。但三种情况需要手动触发：第一，metadata 文件损坏导致 SNAPSHOT 不更新（常见）；第二，从外部大量导入 jar 后 metadata 缺失；第三，怀疑 metadata 与实际 blob 不一致时。这个任务的执行成本不低——对于有 10 万个组件的仓库，可能需要 30~60 分钟，期间 CPU 密集。"

> **技术映射**：Rebuild metadata 是重 IO + 重 CPU 操作，不应排在高频执行，建议每月一次或按需触发。

**炮哥**："任务执行失败了怎么发现？我现在都不知道哪些任务失败过。"

**大师**："每个任务都有执行日志——在 Web UI 点击任务名，切到 `Log` 标签页可以查看最近一次执行的标准输出。但问题是，这个日志只在有人主动去看的时候才有用。更自动化的方案是：配置 `alertEmail` 属性，任务失败时自动发邮件通知；或者通过 API 轮询任务状态，集成到你们的监控系统。"

**小胖**："那 Groovy 脚本任务是什么？能写什么？"

**大师**："Groovy 脚本任务是 Nexus 的'任意门'——你可以在任务中写 Groovy 代码，调用 Nexus 内部 API 做几乎任何事。比如编写一个定时巡检脚本，检查所有 proxy 仓库的远程连接状态；或者写一个自定义的清理逻辑，比 Cleanup Policy 更灵活。但要小心——Groovy 脚本运行在 Nexus JVM 内部，脚本写错了可能拖垮整个服务。"

> **技术映射**：Groovy Script Task = Nexus 内部的"超级管理员"脚本，可以访问所有内部服务和存储，权限极高、风险也极高。

**炮哥**："那运维的最佳实践是什么？排期怎么定？"

**大师**："一个成熟的生产环境排期应该是这样的——**每日轻量**：健康检查脚本（Groovy，检查 proxy 连通性和磁盘水位，1 分钟内完成）。**每周中量**：Cleanup Task（清理过期制品）、Delete unused Docker manifests。**每月重量**：Rebuild metadata（如有需要）、Compact BlobStore（压实回收）。所有重量任务排在周六凌晨 3:00-6:00，不同任务间错开至少 1 小时。"

## 3. 项目实战

### 3.1 环境准备

### 3.2 分步实战

#### 步骤一：查看和管理所有任务

**目标**：学会通过 API 获取任务列表、查看详情、管理生命周期。

```bash
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

# 1. 获取所有任务列表
curl -s -u $AUTH "$NEXUS/service/rest/v1/tasks" | jq '.items[] | {id, name, type: .type, schedule, enabled: .enabled}'

# 2. 查看某个任务的详细配置
TASK_ID=$(curl -s -u $AUTH "$NEXUS/service/rest/v1/tasks" | jq -r '.items[0].id')
curl -s -u $AUTH "$NEXUS/service/rest/v1/tasks/$TASK_ID" | jq '{name, type, schedule, status, properties}'

# 3. 查看任务类型列表（所有可用的任务 action）
curl -s -u $AUTH "$NEXUS/service/rest/v1/tasks/types" | jq '.[].typeId'
```

#### 步骤二：创建每周维护任务套件

**目标**：创建一套标准的生产维护任务——索引重建、元数据修复、清理、压实。

```bash
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

# 1. 重建 Maven 仓库索引（每两周周日凌晨 2:00）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "coreui_Task",
    "name": "重建 maven-releases 索引",
    "typeId": "repository.maven.rebuild-index",
    "schedule": "biweekly",
    "properties": {
      "repositoryName": "maven-releases"
    },
    "alertEmail": "ops@cloudwhale.com"
  }'

# 2. 修复 Maven 元数据（每月第一个周六凌晨 3:00）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "coreui_Task",
    "name": "修复 maven-releases 元数据",
    "typeId": "repository.maven.rebuild-metadata",
    "schedule": "monthly",
    "properties": {
      "repositoryName": "maven-releases"
    },
    "alertEmail": "ops@cloudwhale.com"
  }'

# 3. 执行清理策略关联仓库的清理（每周六凌晨 4:00）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "repository.cleanup",
    "name": "每周清理过期制品",
    "typeId": "repository.cleanup",
    "schedule": "weekly",
    "properties": {
      "repositoryName": "*",
      "preview": "false"
    },
    "alertEmail": "ops@cloudwhale.com"
  }'

# 注意: repositoryName 为 "*" 意味着应用所有关联了 Cleanup Policy 的仓库

# 4. 压缩 BlobStore（每周日凌晨 5:00）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "blobstore.compact",
    "name": "每周压缩所有 BlobStore",
    "typeId": "blobstore.compact",
    "schedule": "weekly",
    "properties": {
      "blobstoreName": "*"
    },
    "alertEmail": "ops@cloudwhale.com"
  }'

# 5. Docker 无用 manifest 清理（每周六凌晨 3:30）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "docker.delete-unused",
    "name": "清理无引用 Docker manifest",
    "typeId": "docker.delete-unused",
    "schedule": "weekly",
    "properties": {
      "repositoryName": "*"
    },
    "alertEmail": "ops@cloudwhale.com"
```

#### 步骤三：编写 Groovy 健康检查脚本任务

**目标**：创建一个定时执行的 Nexus 内部健康检查脚本。

```bash
# 创建 Groovy 脚本任务——检查 proxy 仓库远程连通性
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "script_Task",
    "name": "每日健康检查：proxy 连通性",
    "typeId": "script",
    "schedule": "daily",
    "properties": {
      "source": "import org.sonatype.nexus.repository.manager.RepositoryManager\nimport org.sonatype.nexus.repository.Repository\n\ndef repoManager = container.lookup(RepositoryManager.class.name)\n\ndef results = []\nrepoManager.browse().each { repo ->\n  if (repo.type.value == \"proxy\") {\n    def status = repo.status\n    results.add(\"${repo.name}: ${status}\")\n  }\n}\n\nlog.info(\"=== Proxy 仓库健康检查: ${new Date()} ===\")\nresults.each { log.info(it) }\nreturn results.join(\"\\n\")"
    },
    "alertEmail": "ops@cloudwhale.com"
  }'
```

#### 步骤四：编写磁盘水位监控 Groovy 脚本

**目标**：每日检查 BlobStore 使用情况并输出日志。

```bash
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/tasks" \
  -H "Content-Type: application/json" \
  -d '{
    "action": "script_Task",
    "name": "每日 BlobStore 容量报告",
    "typeId": "script",
    "schedule": "daily",
    "properties": {
      "source": "import org.sonatype.nexus.blobstore.api.BlobStoreManager\n\ndef blobStoreManager = container.lookup(BlobStoreManager.class.name)\n\ndef report = []\nblobStoreManager.browse().each { bs ->\n  def metrics = bs.metrics\n  def totalSizeMB = (metrics.totalSize / 1024 / 1024).round(2)\n  def blobCount = metrics.blobCount\n  report.add(\"${bs.blobStoreConfiguration.name}: ${totalSizeMB} MB, ${blobCount} blobs\")\n}\n\nlog.info(\"=== BlobStore 容量报告: ${new Date()} ===\")\nreport.each { log.info(it) }\nreturn report.join(\"\\n\")"
    },
    "alertEmail": "ops@cloudwhale.com"
  }'
```

#### 步骤五：任务状态监控脚本

**目标**：编写外部脚本，通过 API 检查所有定时任务是否按计划执行。

```bash
#!/bin/bash
# monitor-tasks.sh：检查 Nexus 定时任务是否正常执行
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

echo "=== Nexus 任务状态检查 ==="
echo "时间: $(date '+%Y-%m-%d %H:%M:%S')"
echo ""

ALERT=0

# 获取所有任务
curl -s -u $AUTH "$NEXUS/service/rest/v1/tasks" | jq -r '.items[] | "\(.id)|\(.name)|\(.type)|\(.schedule)"' | while IFS='|' read -r TID TNAME TTYPE TSCHED; do

    # 跳过 manual 任务（手动触发的不需要检查）
    if [ "$TSCHED" = "manual" ]; then
        continue
    fi

    # 获取任务详情
    DETAIL=$(curl -s -u $AUTH "$NEXUS/service/rest/v1/tasks/$TID")

    # 检查上次执行状态
    LAST_STATUS=$(echo "$DETAIL" | jq -r '.lastRunResult // "never_run"')
    LAST_RUN=$(echo "$DETAIL" | jq -r '.lastRun // "从未执行"')

    if [ "$LAST_STATUS" = "never_run" ]; then
        echo "⚠️  $TNAME: 从未执行过"
        ALERT=$((ALERT + 1))
    elif echo "$LAST_STATUS" | grep -qi "error\|exception\|fail"; then
        echo "❌ $TNAME: 上次执行失败 ($LAST_RUN)"
        ALERT=$((ALERT + 1))
    else
        echo "✅ $TNAME: 正常 (上次: $LAST_RUN)"
    fi
done
```

```bash
chmod +x monitor-tasks.sh && ./monitor-tasks.sh
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| 多个重 IO 任务同时执行 | Nexus 响应极慢甚至超时 | 错峰排期，重 IO 任务至少间隔 1 小时 |
| Groovy 脚本语法错误 | 任务日志显示 `groovy.lang.MissingMethodException` | 先在 Nexus 的 `System → Scripts` 中创建脚本，本地调试通过后再创建 Task |
| Cleanup Task 与 Compact 同时触发 | 压缩回收效率低，大量 blob 未被回收 | Cleanup 先行，Compact 延后 1~2 天 |
| `schedule: weekly` 但不确定哪一天 | 任务在一个奇怪的时间点执行 | 通过 Cron 表达式精确控制执行时间（`configuration.repository.maven.rebuild-index` 等支持 cron） |
| 任务执行不产生库存数据 | Rebuild Index 日志显示 "success" 但搜索仍无结果 | 先检查仓库是否有内容，索引重建仅对已有制品有效 |

## 4. 项目总结

### 4.1 生产环境推荐排期

| 频率 | 时间窗口 | 任务 | 预计耗时 | 风险 |
|------|---------|------|---------|------|
| 每日 | 凌晨 2:00 | BlobStore 容量报告（Groovy） | < 1 min | 低 |
| 每日 | 凌晨 2:30 | Proxy 健康检查（Groovy） | < 1 min | 低 |
| 每周 | 周六 3:00 | Cleanup Task（清理过期组件） | 10~30 min | 中 |
| 每周 | 周六 3:30 | Delete unused Docker manifests | 5~15 min | 中 |
| 每周 | 周日 4:00 | Compact BlobStore | 10~60 min | 高 |
| 每月 | 第1个周日 2:00 | Rebuild Maven metadata | 30~60 min | 高 |
| 按需 | - | Rebuild Index | 10~120 min | 高 |

### 4.2 适用场景

1. **自动化清理**：定时清理过期 SNAPSHOT、临时 Docker 标签和废弃的 npm pre-release
2. **索引与元数据维护**：在制品大规模变更后重建索引
3. **存储健康检查**：每日自动生成 BlobStore 容量报告和 proxy 连通性报告
4. **自定义巡检**：通过 Groovy 脚本实现企业特有的监控逻辑
5. **故障恢复**：在元数据损坏时按需触发修复任务

**不适用场景**：
1. 需要外部系统调度的跨系统工作流——Nexus 任务间没有依赖编排，建议用 Jenkins/Argo Workflow 等外部调度器
2. 频率低于 1 分钟的任务——Nexus 任务的最小调度单位是分钟级

### 4.3 注意事项

- **Cron 表达式优先**：`weekly` / `daily` 等预设 schedule 不够精确时，使用 cron 表达式
- **Groovy 脚本的安全边界**：脚本运行在 Nexus JVM 内，有权限访问所有内部组件和数据库，务必 Code Review
- **任务并发度**：多个任务可以并行执行，但重 IO 任务不建议并行
- **"*" 通配符谨慎使用**：`repositoryName: "*"` 会应用到所有仓库，确保策略已正确配置

### 4.4 常见踩坑经验

**故障一：Compact BlobStore 执行了 3 个小时还没完成**

某周六凌晨 Compact 任务一直跑到早上 9 点，期间所有 Nexus 上传下载超时。根因：三个月未执行过 Compact，被软删除的 blob 累积了上万个。删除大量文件是重 IO 操作，在 HDD 磁盘上尤其慢。解决：将 Compact 频率从每月改为每周，避免一次性处理太多文件；将 Docker BlobStore 迁移到 SSD。

**故障二：Rebuild metadata 后所有 SNAPSHOT 下载失败**

执行 Rebuild metadata 任务后，下游项目 `mvn -U compile` 报 SNAPSHOT 解析错误。根因：Rebuild 过程中元数据重建不完整——某些时间戳版本的 jar 存在但 pom 缺失。解决：先删除问题组件再重新 deploy，而不是依赖 metadata 重建。

**故障三：Groovy 脚本任务导致 Nexus OOM**

某团队写了一个 Groovy 脚本遍历所有 20 万个组件做统计，运行时 Nexus 占用的堆内存从 2GB 飙到 4GB 以上触发 OOM。根因：脚本中一次性加载所有组件到 List 而非流式处理。解决：使用游标分页遍历，每次只加载一页数据。

### 4.5 思考题

1. 如果 Compact BlobStore 任务与 Cleanup Task 被设置在了同一时间执行，两者在 BlobStore 层面会产生怎样的竞争？会导致数据不一致吗？
2. 你需要设计一个"灰度清理"方案——每天的清理任务只清理总量不超过 10% 的过期组件，避免一次清理过多导致下游连锁故障。在不修改 Cleanup Policy 的情况下，如何通过 Task 编排实现？

（第12章思考题答案：1. 两者是 AND 关系——组件必须同时满足"发布时间 > 7 天"和"距离最后下载 > 7 天"才会被清理。所以 3 天前发布但从未下载的组件不会被清理（第一个条件不满足）。如果想去掉时间条件中的某一个，将其值设为 0（表示不检查）。2. 创建两条 Cleanup Policy：Policy-RELEASE 设置 `criteriaReleaseType: "RELEASE"` 且所有时间条件为 0（永不清理），关联到 `maven-releases` 仓库。Policy-RC 设置 `criteriaAssetRegex: ".*-RC.*"` 和 `criteriaLastBlobUpdated: 7`，关联到同一仓库。当 RC 版本对应的 GA 版本发布后，约定 7 天内清理 RC 包。通过版本命名规范（如 `1.0-RC1`、`1.0`）和正则匹配实现区分。）
