# 第21章：代理缓存策略与远程仓库稳定性

## 1. 项目背景

某周四凌晨 2:17，云鲸科技的值班运维被 PagerDuty 叫醒：Nexus 的所有 proxy 仓库全部进入 "Auto blocked" 状态，前端 CI 构建失败、Java 组 `mvn compile` 超时、Docker 拉取卡死——整个研发环境处于半瘫痪状态。紧急排查后发现根因：Maven Central 在凌晨时段进行了 CDN 切换维护，所有请求超时 90 秒以上。Nexus 的 `autoBlock` 机制检测到连续失败后，自动冻结了 `maven-central-proxy` 仓库，将其状态从 "Online" 变为 "Online - Remote Unavailable"。本该是一个合理的保护机制——但运维错误地将 `npm-proxy` 和 `docker-hub-proxy` 的 `autoBlock` 也设置为了 `true`，导致三个 proxy 仓库连锁冻结。

更糟糕的是，之前的运维为了"加快下载速度"，把所有 proxy 仓库的 `metadataMaxAge` 设为了 `0`（每次请求都向远程源重新验证）。结果 Nexus 对每个依赖请求都向远程源发起条件请求，当远程源不可用时，**连本地缓存中有完美副本的包也无法返回**——因为 Nexus 试图验证元数据时连接超时了。

这个凌晨事故暴露出一个关键认知盲区：**Nexus 的 proxy 仓库不是"透明缓存代理"，而是一个有独立缓存策略、独立健康检查、独立熔断机制的缓存层**。本章将深入 proxy 仓库的三大缓存参数（metadata max age、content max age、negative cache）、两种防护机制（autoBlock、health check）、以及缓存预热与污染防控策略，确保 Nexus 在公网波动时仍能稳定服务内部团队。

## 2. 项目设计

事故复盘会上，炮哥把凌晨的告警时间线投影出来，大师逐条分析。

**炮哥**："大师，proxy 仓库的 `metadataMaxAge: 0` 是我设的——我以为设成 0 就能获取最新 metadata，为什么反而导致缓存失效？"

**大师**："`metadataMaxAge: 0` 的意思是'不信任本地缓存的 metadata，每次请求都要向远程服务器确认'。正常情况下，Nexus 会发一个条件请求（`If-Modified-Since` 或 `If-None-Match`），如果远程的 metadata 没变化，远程返回 `304 Not Modified`，Nexus 就继续用本地缓存——总耗时大概 200ms。但凌晨 Maven Central 维护时，这个条件请求超时了 90 秒，Nexus 拿不到 `304` 响应，不敢返回本地缓存，就只能报错。设成 `1440`（24 小时）反而更好——24 小时内 Nexus 直接返回本地缓存，根本不请求远程。"

> **技术映射**：`metadataMaxAge` 是缓存新鲜度窗口。0 = 每次都验证（最安全但最脆弱），1440 = 24 小时内信任缓存（最快最稳）。生产环境推荐按格式区分：SNAPSHOT 代理设为 5-15 分钟，RELEASE 代理设为 1440 分钟。

**小白**："那 `autoBlock` 机制呢？它是在什么条件下触发的？为什么一触发三个仓库全被封？"

**大师**："`autoBlock` 内置一个故障计数器——当向远程仓库的连续请求失败达到一定阈值（通常是 5 次），Nexus 自动将 proxy 仓库标记为 'Remote Unavailable'，后续所有请求直接返回缓存的本地副本而不尝试远程连接。这是一个**熔断机制**——防止雪崩。但凌晨的问题恰恰相反——`metadataMaxAge: 0` 意味着本地没有有效缓存，熔断之后没有副本可以返回，仓库等于完全不可用。"

**小胖**："那负缓存（negative cache）又是什么？听起来像是不好的东西。"

**大师**："负缓存是很好的东西。正常缓存——'这个文件存在，我缓存它 24 小时'。负缓存——'这个文件 404，我记住它不存在，24 小时内不再尝试'。没有负缓存的话，你对不存在的路径每次请求都要去远程确认一次，浪费了大量连接。但风险是——如果远程刚好在此期间发布了那个包，你的 Nexus 在负缓存过期之前不会去拉取。"

> **技术映射**：Negative Cache = 404 响应的本地记忆，避免重复请求不存在的资源。`negativeCache.enabled: true` + `timeToLive` 合理设置是关键。

**浩子**："Docker Hub 的 proxy 和 Maven Central 的 proxy 行为有什么不同？我发现 Docker Hub 的限流经常导致 autoBlock。"

**大师**："最大的差异在认证和限流。Maven Central 是**完全开放**的——没有认证、没有限流。npm registry 也是开放的但有软性的速率限制。Docker Hub 对匿名用户有严格的**拉取次数限制**（每 6 小时 100-200 次）。如果你的 `docker-hub-proxy` 没有配置 Docker Hub 账户认证，Nexus 以匿名身份拉取，整个团队的流量都算在一个匿名 IP 上——很容易触发限流。触发限流后 Docker Hub 返回 `429 Too Many Requests`，Nexus 将其视为失败，连续失败触发 autoBlock。"

**炮哥**："有没有办法在远程仓库恢复后自动恢复 proxy？还是要手动去点？"

**大师**："`autoBlock` 会自动检测恢复——Nexus 会定期对 auto-blocked 的仓库发送探测请求，如果远程恢复正常就自动解冻。间隔时间大约 5~15 分钟，取决于配置。你也可以手动在 Web UI 中点击 'Unblock' 立即解冻。"

#### 步骤一：配置各格式 proxy 的推荐缓存参数

**目标**：为 Maven、npm、Docker 三种 proxy 设置适合生产环境的缓存参数。

```bash
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

# === Maven Central Proxy（RELEASE 为主，缓存周期长） ===
curl -u $AUTH -X PUT "$NEXUS/service/rest/v1/repositories/maven/proxy/maven-central" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "maven-central",
    "online": true,
    "proxy": {
      "remoteUrl": "https://repo1.maven.org/maven2/",
      "contentMaxAge": -1,
      "metadataMaxAge": 1440
    },
    "negativeCache": {
      "enabled": true,
      "timeToLive": 1440
    },
    "httpClient": {
      "blocked": false,
      "autoBlock": true,
      "connection": {
        "retries": 3,
        "timeout": 60,
        "userAgentSuffix": "Nexus-CloudWhale"
      }
    }
  }'

# === npm Proxy（metadata 变化频繁，缓存周期适中） ===
```

#### 步骤二：模拟远程仓库不可用 + 缓存兜底验证

**目标**：验证当远程仓库不可用时，Nexus 本地缓存仍能正常服务。

```bash
# === 实验：验证远程不可用时缓存兜底 ===

echo "   缓存兜底: HTTP $HTTP (预期 200——即使远程不通，本地缓存仍可用)"
```

#### 步骤三：缓存预热策略脚本

**目标**：编写脚本在新环境部署后预热 proxy 缓存，减少首次请求延迟。

```bash
#!/bin/bash
# cache-warmup.sh：proxy 缓存预热——预下载常用依赖
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

# 定义常用组件列表（Maven 坐标）
WARMUP_LIST=(
  "com/google/guava/guava/31.1-jre/guava-31.1-jre.jar"
  "com/fasterxml/jackson/core/jackson-databind/2.15.0/jackson-databind-2.15.0.jar"
  "org/slf4j/slf4j-api/2.0.7/slf4j-api-2.0.7.jar"
  "junit/junit/4.13.2/junit-4.13.2.jar"
  "org/projectlombok/lombok/1.18.28/lombok-1.18.28.jar"
)

echo "=== Nexus Maven Proxy 缓存预热 ==="
WARMED=0
FAILED=0

for path in "${WARMUP_LIST[@]}"; do
    HTTP=$(curl -s -o /dev/null -w "%{http_code}" \
      -u "$AUTH" "$NEXUS/repository/maven-central/$path")
    if [ "$HTTP" = "200" ]; then
        echo "  ✅ $(basename "$path")"
        ((WARMED++))
    else
        echo "  ⚠️  $(basename "$path") 返回 HTTP $HTTP"
        ((FAILED++))
    fi
done

echo ""
echo "预热完成: 成功 $WARMED / 失败 $FAILED"
echo "提示：预热后这些依赖将直接从 Nexus 本地缓存返回，无需远程拉取"
```

```bash
chmod +x cache-warmup.sh && ./cache-warmup.sh
```

#### 步骤五：排查缓存污染与手动清理

**目标**：当某个包的缓存版本有问题时，如何手动清理 proxy 缓存中的特定资产。

```bash
#!/bin/bash
# purge-proxy-cache.sh：清理 proxy 中指定包的缓存
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

REPO="${1:-maven-central}"       # proxy 仓库名
GROUP_PATH="${2:-}"              # group 路径（如 com/google/guava/guava）

if [ -z "$GROUP_PATH" ]; then
    echo "用法: $0 <proxy-repo> <group-path>"
    echo "示例: $0 maven-central com/google/guava/guava"
    echo "      这将删除 maven-central 中所有 guava 的缓存版本"
    exit 1
fi

echo "=== 清理 proxy 缓存 ==="
echo "仓库: $REPO"
echo "路径: $GROUP_PATH"
echo ""

# 1. 搜索该路径下的所有组件
echo "[1] 搜索 $GROUP_PATH 下的组件..."
COMPONENTS=$(curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/search?repository=$REPO&group=$GROUP_PATH")

COUNT=$(echo "$COMPONENTS" | jq '.items | length')
echo "  找到 $COUNT 个组件"

if [ "$COUNT" = "0" ]; then
    echo "  无组件需要清理"
    exit 0
fi

# 2. 列出所有组件（确认范围）
echo "[2] 待清理的组件:"
echo "$COMPONENTS" | jq -r '.items[] | "  \(.name):\(.version)"'

# 3. 确认后执行删除
read -p "确认删除以上组件？(yes/no): " CONFIRM
if [ "$CONFIRM" != "yes" ]; then
    echo "已取消"
    exit 0
fi

echo "[3] 执行删除..."
DELETED=0
echo "$COMPONENTS" | jq -r '.items[].id' | while read -r CID; do
    HTTP=$(curl -s -o /dev/null -w "%{http_code}" -u $AUTH -X DELETE \
      "$NEXUS/service/rest/v1/components/$CID")
    echo "  $CID → HTTP $HTTP"
done
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| `metadataMaxAge: 0` + 远程不可用 | 所有请求失败，缓存完全无效 | 设置为合理值（SNAPSHOT 5-15min，RELEASE 1440min） |
| 未配置 `autoBlock` | 远程故障时所有请求堆积超时，Nexus 线程池被打满 | 启用 `autoBlock: true` 作为熔断保护 |
| negative cache 时间过长 | 新发布的包 24 小时内不可用 | 适当调低 `negativeCache.timeToLive`，或提供手动刷新入口 |
| proxy 未配置企业网络代理 | 内网 Nexus 无法访问外网远程仓库 | 在 `Administration → System → HTTP` 中配置 proxy 代理 |

## 4. 项目总结

### 4.1 缓存参数速查表

| 参数 | 含义 | Maven RELEASE 推荐 | Maven SNAPSHOT 推荐 | npm 推荐 | Docker 推荐 |
|------|------|-------------------|--------------------|---------|------------|
| `metadataMaxAge` | metadata 缓存有效期（分钟） | 1440 | 5-15 | 60 | 60 |
| `contentMaxAge` | 文件内容缓存有效期（分钟） | -1（永不过期） | -1 | -1 | 1440 |
| `negativeCache.timeToLive` | 404 记忆有效期（分钟） | 1440 | 60 | 1440 | 1440 |
| `autoBlock` | 连续失败自动熔断 | ✅ true | ✅ true | ✅ true | ✅ true |
| `connection.timeout` | 远程请求超时（秒） | 60 | 60 | 30 | 300 |
| `connection.retries` | 重试次数 | 3 | 3 | 3 | 3 |

### 4.2 适用场景

1. **外网不稳定环境**：通过合理的缓存策略让 Nexus 在公网波动时仍能稳定服务
2. **多地域团队**：在靠近公网出口的机房部署 Nexus 作为一级 proxy 缓存
3. **离线/弱网环境**：在有网环境预热缓存后导入内网 Nexus
4. **安全隔离**：通过 Nexus proxy 控制外部依赖的来源，统一管理远程 URL 白名单

**不适用场景**：
1. 内网带宽充足且远程仓库 SLA 99.99% 以上——缓存收益不明显
2. 所有依赖均已内部化（无外网依赖）——不需要 proxy

### 4.3 注意事项

- **`contentMaxAge: -1` 不是"永远不更新"**：Nexus 会通过条件请求（`If-Modified-Since`）在 remote 有变化时更新
- **autoBlock 的恢复探测有延迟**：5~15 分钟不等，如果远程短时间内恢复需要手动 Unblock
- **企业网络代理需要额外配置**：Nexus 需要能访问远程仓库，如果有企业防火墙/代理，在 `System → HTTP` 中配置

### 4.4 常见踩坑经验

**故障一：npm proxy 缓存了错误的 tarball（checksum 不匹配）**

某个 npm 包的 tarball 在远程传输中被截断，Nexus 缓存了不完整的 tarball。后续所有 `npm install` 都因 integrity 校验失败而中断。根因：网络中断导致 tarball 下载不完整且 Nexus 未校验。解决：手动删除 proxy 中该包的缓存目录（通过 Browse 页面或 API 删除对应 component），触发重新下载。

**故障二：Maven proxy 缓存了 SNAPSHOT 但 metadata 已是旧版**

release 包发布 24 小时后 Java 组仍拉不到。根因：`metadataMaxAge: 1440` 已过期，但 `contentMaxAge: -1` 的前提下，如果 metadata 中的 last-modified 没有变化，条件请求返回 304，Nexus 继续用旧缓存。原因是远程 CDN 节点的时间戳不一致。解决：手动 `DELETE` 缓存组件的 metadata 文件触发强制更新。

**故障三：autoBlock 导致"假死"——proxy 实际可用但 Nexus 认为不可用**

阿里云 Maven 镜像夜间维护返回 502，触发 autoBlock。但 nexus 探测恢复时仍用同一 CDN 节点（该节点尚未恢复），导致持续 blocked。其他 CDN 节点已正常。解决：手动 Unblock + 调整 `autoBlock` 的探测间隔。

### 4.5 思考题

1. 如果你的企业同时有 Maven Central、Aliyun Maven Mirror、和腾讯云 Maven Mirror 三个远程代理可用。如何设计 group + proxy 的冗余方案，使得其中一个远程故障时能自动切换到另一个，同时尽可能减少重复缓存？
2. 设计一个"缓存预热即服务"方案——新项目启动时，通过分析其 `pom.xml` 或 `package.json` 中声明的依赖列表，自动预热 Nexus proxy 缓存中缺失的包。需要考虑大规模预热的限速和失败重试。

（第20章思考题答案：1. 利用 Docker Registry v2 API 遍历所有镜像的 manifest→config→layer 引用链，构建一张"layer→引用它的 tag"的映射表。对于每个旧版 base image，查询映射表确定是否有活跃 tag 仍在引用。如果某个 layer 只被已删除或天数的 tag 引用，可以安全清理。可用 `docker manifest inspect` 或 Nexus search API 实现。2. 方案：创建 Cleanup Policy A（`criteriaAssetRegex: ".*-dev-.*"` + `criteriaLastBlobUpdated: 30`），Policy B（`criteriaLastBlobUpdated: 14` + 只保留最近 5 个版本），Policy C（排除 `release-*` 标签——方法是不创建匹配 release 标签的 policy）。将全部 Policy 关联到同一仓库，三个 Task 串行执行。关键技巧：Nexus Cleanup Policy 的逻辑是"满足条件的会被删除"，不存在"排除"机制——需要用"不匹配则不删"的思路设计 regex。对于 `release-*` 标签，不做任何 policy 匹配即可保护。）

### 4.6 推广计划提示

- **运维部门**：按推荐参数表调整所有 proxy 仓库配置；将 proxy 健康检查脚本加入每日巡检
- **开发部门**：当遇到"依赖下载失败但昨天还能下"时，第一步先检查 proxy 仓库是否 auto-blocked
- **架构组**：评估是否引入二级 proxy 方案（核心数据中心 → 区域机房代理链）
