# 第35章：下载链路源码：路由、权限、缓存与响应

## 1. 项目背景

云鲸科技的 Java 组经常遇到一个奇怪的下载行为：通过 `maven-public`（group 仓库）下载 `common-utils:1.0.0`，有时候下载到的是 `maven-releases` 中的正式版本，有时候却下载到了 `maven-snapshots` 中的一个旧版本。组内排查发现——`maven-public` group 的成员顺序是 `maven-releases → maven-central-proxy → maven-snapshots`，但 Maven 下载时似乎没有严格按照这个顺序查找。

更怪异的是，同样一个 group 仓库，Maven 3.8 和 Maven 3.9 下载同一个包时走了不同的仓库成员路径——3.8 命中 hosted 仓库，3.9 却命中了 proxy 缓存的版本。架构组怀疑这与 `maven-metadata.xml` 的 `lastUpdated` 字段以及不同 Maven 版本对 group 仓库 metadata 合并算法的差异有关，但需要从 Nexus 源码层面验证。

下载链路比上传链路更复杂——因为它涉及**四种不同路径**：hosted 直接命中、proxy 本地缓存命中、proxy 远程拉取、group 仓库的顺序查找。每一步都可能遇到权限被拒、缓存过期、远程连接失败等问题。本章将追踪一个 Maven 下载请求从 HTTP 入口到 Blob 响应的完整过程，区分四种路径，标注出权限校验点、缓存检查点和代理拉取点。

## 2. 项目设计

小李追踪了一整天，发现下载链路的入口不是一个类，而是一个"HTTP Bridge + Repository View"的组合机制。

**小李**："大师，上传链路我知道从 `UploadResource` 开始。下载链路呢？Maven 的 GET 请求 `repository/maven-public/com/.../guava.jar` 根本没有经过任何 `@Path` 注解的 Resource 类。"

**大师**："因为下载请求不走 JAX-RS（REST API），而是走 **Nexus 的 HTTP Bridge**。Nexus 有两套 HTTP 处理机制——`/service/rest/` 前缀的走 JAX-RS（REST 端点），`/repository/` 前缀的走 **Repository View**——这是 Nexus 自己的一个扩展 Servlet，它将 URL 路径映射为仓库操作。当你 GET `/repository/maven-public/com/.../guava.jar` 时，HTTP Bridge 解析出仓库名 `maven-public` 和 `asset path`，找到对应的 Repository 对象，然后触发下载流程。"

> **技术映射**：Nexus HTTP 分层 = JAX-RS（`/service/rest/` REST API）+ Repository View（`/repository/` 制品访问）。前者用于管理操作（CRUD），后者用于制品传输（上传下载）。两者的权限校验和事件发布是统一的。

**小胖**："那 Request 到了 Repository View 之后呢？怎么决定是从 hosted 取、从 proxy 缓存取、还是从远程拉？"

**大师**："这是一个责任链（Chain of Responsibility）模式。Request 进入 Repository View 后，Nexus 创建一组 **Handler**，按顺序执行——每个 Handler 决定'我能处理吗？能就处理并返回，不能就交给下一个 Handler'。对于 group 仓库，Handler 链的第一个是 `GroupHandler`——它遍历 group 的成员仓库列表，对每个成员仓库递归调用下载流程，第一个返回成功的就使用。对于 hosted 仓库，`HostedHandler` 直接查 BlobStore。对于 proxy 仓库，`ProxyHandler` 先查本地缓存，缓存 miss 时向远程发起请求并更新缓存。"

**小李**："那权限检查在哪？是在 Handler 链的最前面统一检查，还是每个 Handler 各自检查？"

**大师**："Nexus 在请求进入 Handler 链之前，通过一个 **SecurityFilter** 统一做权限校验。这个 Filter 从 `SecurityFacet` 中获取当前用户对目标仓库的权限。权限检查用的不是 Handler 模式，而是 Filter/Interceptor 模式——它是在整个处理链的最外层执行的，确保不管后续走哪个 Handler，权限检查都会先发生。"

> **技术映射**：下载链路的两种设计模式——Handler Chain（责任链，按仓库类型依次处理）+ Security Filter（拦截器，在处理前统一校验）。前者处理业务逻辑（怎么做），后者处理横切关注点（能不能做）。

**小李**："group 仓库中 maven-metadata.xml 的合并是什么时候发生的？这是什么逻辑？"

**大师**："这是 group 仓库最复杂的部分——**元数据合并**（Metadata Merging）。当客户端请求 group 仓库中的 `maven-metadata.xml` 时，Nexus 不是简单地从第一个成员仓库返回——它先分别请求所有成员仓库的 metadata，然后合并成一个合并后的 metadata 返回。合并逻辑由 `MavenGroupFacet` 负责——它按 group 的成员顺序合并 `<versions>` 列表（去重），`<latest>` 取第一个成员的值（按 group 成员顺序），`<release>` 取第一个成员的值。这就是为什么 group 成员顺序决定了'哪个版本的 RELEASE 被认为是最新'。"

### 3.2 分步实战

#### 步骤一：追踪请求从 HTTP Bridge 到 Repository View

**目标**：理解 `GET /repository/<repo>/<path>` 的入口处理。

```bash
# 在 Nexus 容器中执行下载请求并观察 request.log
curl -s -o /dev/null -u admin:admin123 \
  "http://localhost:8081/repository/maven-public/com/google/guava/guava/31.1-jre/guava-31.1-jre.jar" \
  -w "\nHTTP %{http_code}  Size: %{size_download}  Time: %{time_total}s"

# 查看 request.log
docker compose exec nexus tail -3 /nexus-data/log/request.log
```

**下载请求的 URL 路由机制**：

```
GET /repository/maven-public/com/google/guava/guava/31.1-jre/guava-31.1-jre.jar

URL 解析：
  /repository/    → 触发 Repository View Servlet
  maven-public/   → 仓库名称
  com/google/.../guava-31.1-jre.jar → 制品路径（asset path）

关键类调用链：
  NexusServlet        → 入口 Servlet（触发 Repository View）
    → RepositoryViewServlet  → 查找 Repository + 创建 Request
      → SecurityFilter       → 权限检查（@PreAuthorize 或 SecurityFacet）
        → HandlerChain         → 按仓库类型分发处理
```

#### 步骤二：区分四种下载路径

**目标**：通过对比实验理解 hosted 命中、proxy 缓存命中、proxy 远程拉取、group 顺序查找的区别。

```bash
#!/bin/bash
# download-paths.sh：四种下载路径验证
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

echo "=== 下载路径实验 ==="

# 路径 1：hosted 直接命中
echo "[路径1] hosted 直接命中"
# 上传一个测试文件到 hosted 仓库后下载
curl -u $AUTH -X PUT "$NEXUS/repository/maven-releases/com/test/dl/1.0/dl-1.0.jar" \
  -d "hosted-download-test"
HTTP=$(curl -s -o /dev/null -w "%{http_code}" -u $AUTH \
  "$NEXUS/repository/maven-releases/com/test/dl/1.0/dl-1.0.jar")
echo "  maven-releases (hosted) → HTTP $HTTP — BlobStore 直接返回"

# 路径 2：proxy 缓存命中
echo "[路径2] proxy 缓存命中"
HTTP=$(curl -s -o /dev/null -w "%{http_code}" -u $AUTH \
  "$NEXUS/repository/maven-central/com/google/guava/guava/31.1-jre/guava-31.1-jre.jar")
echo "  maven-central (proxy) → HTTP $HTTP (200=缓存命中，首次可能 202 Accept 等待远程)"

# 路径 3：proxy 远程拉取（首次请求未缓存的包）
echo "[路径3] proxy 远程拉取"
HTTP=$(curl -s -o /dev/null -w "%{http_code}" -u $AUTH \
  "$NEXUS/repository/maven-central/org/apache/commons/commons-lang3/3.14.0/commons-lang3-3.14.0.jar")
echo "  maven-central (proxy) → HTTP $HTTP (首次请求需等待远程响应)"

# 路径 4：group 仓库按成员顺序查找
echo "[路径4] group 顺序查找"
# 创建或确认一个 group 仓库
curl -s -u $AUTH "$NEXUS/service/rest/v1/repositories/maven/group/maven-public" | \
  jq '.group.memberNames'
echo "  maven-public (group) → 按成员顺序 [maven-releases, maven-snapshots, maven-central] 依次查找"
echo "  第一个返回非 404 的结果被使用"
```

#### 步骤五：编写缓存命中率统计脚本

**目标**：通过解析 request.log 统计 proxy 仓库的缓存命中率。

```bash
#!/bin/bash
# cache-hit-ratio.sh：proxy 缓存命中率统计
REPO="${1:-maven-central}"
LOG_FILE="${2:-}"  # 如果空则从 Nexus 容器中取

echo "=== Proxy 缓存命中率统计 ($REPO) ==="

# 统计各状态码
TOTAL=$(grep "/repository/$REPO/" /nexus-data/log/request.log 2>/dev/null | wc -l)
HIT=$(grep "/repository/$REPO/" /nexus-data/log/request.log 2>/dev/null | awk '$9==200 || $9==304' | wc -l)
MISS=$(grep "/repository/$REPO/" /nexus-data/log/request.log 2>/dev/null | awk '$9==404' | wc -l)
ERROR=$(grep "/repository/$REPO/" /nexus-data/log/request.log 2>/dev/null | awk '$9>=500' | wc -l)

if [ "$TOTAL" -gt 0 ]; then
    HIT_RATE=$(echo "scale=2; $HIT * 100 / $TOTAL" | bc)
    echo "总请求: $TOTAL"
    echo "命中 (200/304): $HIT (${HIT_RATE}%)"
    echo "未命中 (404): $MISS"
    echo "错误 (5xx): $ERROR"
else
    echo "无 $REPO 的请求记录"
fi
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| Group 成员顺序导致错误版本 | 下载到同一 GAV 但不同仓库的版本 | 检查 group 成员的排列顺序——hosted 必须排在 proxy 前面 |
| Proxy 返回 304 Not Modified | 客户端报错但本地缓存存在 | `304` 是条件请求的合法响应，客户端应使用本地缓存 |
| Handler 链中断 | 中间某个 Handler 抛异常导致后续 Handler 未执行 | 检查异常是 checked 还是 runtime——只有 runtime 异常会中断链 |
| 大文件下载中途断开 | 客户端报 `Connection reset` 但 Nexus 无异常 | Nginx 的 `proxy_read_timeout` 或 `proxy_buffering` 设置问题 |

## 4. 项目总结

### 4.1 四种下载路径速查

| 路径 | 仓库类型 | 流程 | 关键类 | 耗时 |
|------|---------|------|--------|------|
| Hosted 直接命中 | hosted | BlobStore 直接读 | `HostedHandler` | < 10ms |
| Proxy 缓存命中 | proxy | 本地 BlobStore 命中 | `ProxyHandler` (缓存分支) | < 10ms |
| Proxy 远程拉取 | proxy | 远程下载 → 写缓存 → 返回 | `ProxyHandler` (远程分支) | 1-60s |
| Group 顺序查找 | group | 依次尝试成员仓库 | `GroupHandler` | 取决于第一个命中的成员 |

### 4.2 适用场景

1. **性能优化**：分析 group 仓库的成员顺序，减少不必要的查找
2. **缓存策略设计**：理解 proxy 的缓存命中/未命中机制，优化 `metadataMaxAge`
3. **故障排查**：当"下载到错误版本"时定位 group 元数据合并的根因
4. **安全审计**：确认所有下载路径都执行了权限检查
5. **自定义下载处理**：新增格式时实现自定义的 Handler 替换默认行为

### 4.3 常见踩坑经验

**故障一：Group 仓库请求大版本列表时 OOM**

某团队在开发中使用了 Maven 的版本范围解析（如 `[1.0,2.0)`），Maven 向 Nexus 发送了一个不带 artifactId 的 group metadata 请求——Nexus 需要合并该 group 下所有成员仓库中该 groupId 对应的所有 artifact 的所有版本。如果 group 中有 5 个成员仓库、该 groupId 下积累了 2000+ 个版本——合并后的 metadata XML 可能达到 8MB+。Nexus 在内存中构建这个 XML 时，如果堆内存不足，会导致频繁 Full GC 甚至 OOM。解决：限制 group 成员仓库数量；在 proxy 仓库中适当设置 `metadataMaxAge`，减少元数据合并的频率；避免在 pom.xml 中使用过宽泛的版本范围声明。

**故障二：Handler 链中的磁盘检查导致下载中断**

开发报告"某个特定 jar 文件始终下载失败"，其他 jar 正常。排查发现该 jar 对应的 Blob 的 `.bytes` 文件所在的磁盘分区有坏道，操作系统的读操作返回 `EIO` 错误。但 Nexus 的代码中 `StorageFacet.get()` 将这个 IOException 包装后抛给了 Handler 链——Handler 链没有优雅降级，直接将异常透传为 HTTP 500 错误。日志中只显示 `Internal Server Error` 没有指明是磁盘 IO 错误。解决：在 Handler 链的顶层增加了 IOException → 友好错误信息的映射；同时修复了坏道磁盘。

**故障三：Group 中某个成员离线导致整体延迟飙升**

运维将 `maven-public` group 的一个成员 `maven-jcenter-proxy` 下线（因为 Bintray/JCenter 已停止服务），但忘记从 group 成员列表中移除。结果所有通过 group 的请求都会先尝试访问这个已下线的 proxy 仓库——每次尝试超时 60 秒后才 fallback 到下一个成员。原本 50ms 的下载延迟飙升到 60 秒。排查路径：在 request.log 中发现大量 60 秒级的请求耗时 → 检查 group 成员列表 → 发现下线的 proxy 成员。解决：从 group 中移除已下线的成员；考虑启用 `autoBlock` 让 proxy 在多次超时后自动熔断。

### 4.4 注意事项

- **group 元数据合并是最复杂的部分**：不同 Maven 版本对合并结果的解析行为有细微差异
- **Proxy 的条件请求**：Nexus 用 `If-Modified-Since` 和 `If-None-Match` 实现缓存更新
- **Download 流是边读边写的**：Nexus 不把文件全部读入内存，而是 stream copy
- **权限检查在请求最前**：如果返回 403，说明在 Handler 链执行前就被拦截了

### 4.5 思考题

1. 如果 `maven-public` group 中有 `maven-releases`（hosted）、`maven-snapshots`（hosted）、`maven-central`（proxy）三个成员。当请求一个同时存在于三个仓库中的 `maven-metadata.xml` 时，最终的 `<versions>` 列表是否包含来自 `maven-snapshots` 的版本？为什么？
2. Proxy 仓库返回 `304 Not Modified` 时，响应体为空。但客户端（Maven/npm）期望的是一个完整的文件。Nexus 如何处理这个矛盾？`304` 在哪种场景下被正确使用？

（第34章思考题答案：1. 最佳注入点是 `SecurityFacet` 或创建一个新的 `ValidationFacet`——原因：① HTTP Filter 只知道 HTTP 层面的信息（URL、header），无法访问 Nexus 的内部模型（Component、Asset、Repository）；② `UploadHandler` 是格式相关的，新检查需要分别注入到每种格式的 Handler，代码重复；③ `StorageFacet` 在所有格式中都一致，且在 Blob 写入前调用，是放置通用校验逻辑的最佳位置。如果新增 `ValidationFacet`，在所有上传 Recipe 中统一 attach，可以配置化注入多种校验规则。2. Docker push 不走 `UploadResource`，而是直接通过 Docker Registry v2 API（`PUT /v2/<name>/blobs/` 和 `PUT /v2/<name>/manifests/`）。入口在 `DockerHostedFacet` 或 Docker 专用的 Servlet——Docker Registry 协议是完全独立于 Nexus REST API 的另一套 HTTP 端点。Docker 格式在仓库创建时注册了一个独立的 Servlet Filter，拦截 `/v2/` 路径下的所有请求，映射到 Docker 格式的 Facet 处理。）
