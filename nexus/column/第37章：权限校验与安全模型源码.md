# 第37章：权限校验与安全模型源码

## 1. 项目背景

云鲸科技的安全团队在一次渗透测试中发现了一个令人不安的问题——一个只拥有 `role-global-reader` 角色的用户（理论上只能浏览制品），竟然通过直接调用 REST API 成功修改了自己的用户属性（First Name 从 "User" 改成了 "Admin"）。虽然这个操作没有影响权限分配，但它暴露了一个安全隐患：**Nexus 的权限校验在不同 API 端点之间存在不一致**。

安全团队深入排查后发现——修改用户信息的 API（`PUT /service/rest/v1/security/users/{userId}`）的权限校验逻辑与仓库操作的权限校验逻辑走的是不同的代码路径。仓库操作通过 `SecurityFacet` 的 `ensurePermitted()` 方法检查，而用户管理 API 走的是 Shiro 的注解式权限检查（`@RequiresPermissions`）。更复杂的是，匿名访问的权限检查、Bearer Token 认证、Basic Auth 认证——三种认证方式的入口和校验点也分布在不同的类中。

"我们只知道'配置了什么角色有什么权限'，但不知道为什么一个看似正确的配置在某个特定 API 上没有生效。"安全团队在周会上总结道。本章将深入 Shiro 在 Nexus 中的认证和授权实现，追踪一次 403 响应的完整源码链路，建立从用户请求到权限判决的完整心智模型，并输出一份可在开发阶段自查的权限排查 checklist。

## 2. 项目设计

安全团队的小孙把渗透测试报告投影出来，大师逐条对照源码。

**小孙**："大师，同一个 `role-global-reader` 角色——为什么在仓库下载时权限有效（没权限下载就 403），但在用户 API 端点上权限无效（没权限修改但仍然成功了）？"

**大师**："因为 Nexus 的权限体系有两条并行的检查路径。**路径 A——Repository View 权限检查**：针对 `/repository/` 路径下的请求（制品上传下载）。走的是 `SecurityFacet.ensurePermitted()` → `NexusAuthorizingRealm.isPermitted()` → 数据库查询角色和 Privilege。**路径 B——REST API 权限检查**：针对 `/service/rest/` 路径下的请求（管理操作）。部分端点用 Shiro 的 `@RequiresPermissions` 注解，部分端点在方法体中显式调用 `SecurityHelper.isPermitted()`。你们发现的用户 API 漏洞恰恰是因为**该端点既没有 `@RequiresPermissions` 注解，也没有显式权限检查**——它是一个权限检查遗漏的 bug。"

> **技术映射**：Nexus 权限检查双轨——Repository View 路径（SecurityFacet，覆盖率 100%）+ REST API 路径（注解或代码调用，覆盖率依赖开发者自觉）。后者容易出现遗漏。

**小胖**："Shiro 在 Nexus 里是怎么工作的？我只知道它是个安全框架，但具体流程呢？"

**大师**："Shiro 的流程分两步。**第一步——认证（Authentication）**：当请求到达时，Nexus 的 `NexusAuthenticatingRealm` 从 HTTP 请求中提取凭据——优先检查 `Authorization: Bearer xxx`（User Token），其次检查 `Authorization: Basic xxx`（用户名密码），最后检查是否匿名访问。**第二步——授权（Authorization）**：认证通过后，`NexusAuthorizingRealm` 根据用户拥有的 Role 查询其拥有的所有 Privilege，然后逐条匹配当前请求需要什么 Privilege——如果用户拥有的 Privilege 覆盖了请求需要的 Privilege，就放行。"

> **技术映射**：Shiro 认证-授权两阶段 = Realm 提取凭据（你是谁）→ Realm 匹配权限（你能干什么）。Nexus 有两个 Realm——`NexusAuthenticatingRealm` 负责认证，`NexusAuthorizingRealm` 负责授权。

**小白**："匿名访问的权限是怎么处理的？匿名用户也有 Role 吗？"

**大师**："Nexus 给匿名用户分配了一个特殊的 `nx-anonymous` 角色。如果管理员在全局设置中启用了匿名访问，那么所有未携带凭据的请求都会被当作 `anonymous` 用户——它能访问的仓库完全取决于 `nx-anonymous` 角色被赋予了哪些 Privilege。如果禁用了匿名访问，未认证的请求在认证阶段就直接返回 401。"

**小孙**："那 '越权' 风险呢？比如一个普通用户通过 API 修改 admin 的密码？"

**大师**："修改用户密码的 API 需要 `nx-userschangepw` 这个 Privilege。操作别人的密码需要 `nx-userschangepw` + 目标用户不是自己时还需要审核。这是 OSS 版的限制——PRO 版有更细粒度的 `nx-users-manage`。但核心原则不变——**如果 API 端点忘了加权限检查，任何认证过的用户都能执行该操作**。这就是为什么权限排查 checklist 的第一条永远是——'这个 API 有权限检查吗？'"

### 3.2 分步实战

#### 步骤一：追踪 Shiro 认证入口和 Realm 注册

**目标**：找到 Nexus 中 Shiro 的配置和认证 Realm 的实现。

```bash
# 搜索 Shiro 配置和 Realm 类
grep -r "AuthenticatingRealm\|AuthorizingRealm\|ShiroFilter" --include="*.java" components/nexus-security/ | head -10

# 关键类：
# NexusAuthenticatingRealm — 认证 Realm（从 token/basic auth 提取身份）
# NexusAuthorizingRealm — 授权 Realm（从 role/privilege 判断权限）
# SecurityModule — Shiro 的 Guice 配置模块
```

**NexusAuthenticatingRealm 认证流程简化示意**：

```java
// 认证流程：从 HTTP 请求中提取用户身份
public class NexusAuthenticatingRealm extends AuthenticatingRealm {
    
    @Override
    protected AuthenticationInfo doGetAuthenticationInfo(AuthenticationToken token) {
        // 1. 判断认证类型
        if (token instanceof BearerToken) {
            // User Token 认证：从数据库查询 token 对应的用户
            String userToken = ((BearerToken) token).getToken();
            User user = userManager.findByToken(userToken);
            return new SimpleAuthenticationInfo(user.getUserId(), userToken, getName());
        }
        
        if (token instanceof UsernamePasswordToken) {
            // Basic Auth 认证：验证用户名密码
            UsernamePasswordToken upToken = (UsernamePasswordToken) token;
            User user = userManager.authenticate(upToken.getUsername(), upToken.getPassword());
            return new SimpleAuthenticationInfo(user, upToken.getPassword(), getName());
        }
        
        // 匿名访问：使用 anonymous 用户身份
        if (anonymousAccessEnabled) {
            return new SimpleAuthenticationInfo("anonymous", null, getName());
        }
        
        throw new AuthenticationException("No valid credentials");
    }
}
```

#### 步骤二：追踪 403 响应的完整源码链路

**目标**：从一个仓库下载请求被拒开始，追踪完整的权限判断流程。

```bash
# 模拟无权限用户的下载请求
# 创建一个只读 raw 用户（无 Maven 权限）
curl -s -o /dev/null -w "HTTP %{http_code}" \
  -u raw-reader:Password123! \
  "http://localhost:8081/repository/maven-releases/com/test/foo/1.0/foo-1.0.jar"
# 预期: HTTP 403

# 在 request.log 中确认
docker compose exec nexus tail -5 /nexus-data/log/request.log
```

**403 请求的源码调用链**：

```
HTTP Request: GET /repository/maven-releases/... 
  ↓
SecurityFilter (NexusServlet 的最外层 Filter)
  ↓
NexusAuthorizingRealm.isPermitted(principal, permission)
  ↓
  1. 获取用户的所有 Role
     SELECT * FROM user_role WHERE userId = 'raw-reader'
  ↓
  2. 获取所有 Role 的所有 Privilege
     SELECT * FROM role_privilege WHERE roleId IN (...)
     → 结果: [nx-repository-view-raw-*-read, nx-repository-view-raw-*-browse]
  ↓
  3. 匹配当前请求需要什么 Privilege
     当前请求: GET maven-releases → 需要 nx-repository-view-maven2-maven-releases-read
  ↓
  4. 用户拥有的 Privilege 中无匹配项 → 权限拒绝
  ↓
  5. Shiro 抛 UnauthorizedException
  ↓
  6. Nexus 转换为 HTTP 403 Forbidden 响应
```

#### 步骤四：Content Selector 的源码级原理

**目标**：理解 Content Selector 权限检查的底层实现。

```java
// NexusAuthorizingRealm 中 Content Selector 权限匹配简化示意
@Override
protected AuthorizationInfo doGetAuthorizationInfo(PrincipalCollection principals) {
    // 获取用户的所有权限（包括 Content Selector 类型）
    Set<String> privileges = permissionManager.getPrivileges(userId);
    
    // 对于 Content Selector 类型的权限：
    // privilegeName = "nx-repository-view-raw-raw-hosted-{contentSelector}-{action}"
    // 解析出 contentSelector 名称 → 查询 CSEL 表达式 → 评估表达式
}

// 权限判断时：如果是 Content Selector 类型
public boolean isPermitted(Permission permission) {
    if (permission instanceof ContentSelectorPermission) {
        ContentSelectorPermission csPerm = (ContentSelectorPermission) permission;
        // 获取 Content Selector 的 CSEL 表达式
        String csel = contentSelectorManager.get(csPerm.getSelectorName()).getExpression();
        // 评估表达式是否匹配当前请求的 asset path / format / group
        return cselEvaluator.evaluate(csel, request.getAssetPath(), request.getFormat());
    }
    // 非 Content Selector: 简单名称匹配
    return privileges.contains(permission.getName());
}
```

#### 步骤五：输出权限排查 checklist 脚本

**目标**：编写一个脚本，一键检查指定用户对指定资源是否有权限。

```bash
#!/bin/bash
# perm-check.sh：权限排查工具
NEXUS="http://localhost:8081"
AUTH="admin:admin123"
USER_ID="$1"
RESOURCE="$2"  # 仓库名 或 API 端点

if [ -z "$USER_ID" ] || [ -z "$RESOURCE" ]; then
    echo "用法: $0 <userId> <repository|api-endpoint>"
    echo "示例: $0 zhangsan maven-releases"
    echo "      $0 zhangsan /service/rest/v1/security/users"
    exit 1
fi

echo "=== 权限排查: $USER_ID → $RESOURCE ==="

# 1. 列出用户所有角色
echo "--- 用户角色 ---"
curl -s -u $AUTH "$NEXUS/service/rest/v1/security/users?userId=$USER_ID" | \
  jq -r '.roles[]' | while read role; do
    echo "  $role"
    # 列出该角色的权限
    curl -s -u $AUTH "$NEXUS/service/rest/v1/security/roles/$role" | \
      jq -r '.privileges[]' | while read priv; do
        echo "    → $priv"
    done
done

echo ""
echo "--- 权限匹配检查 ---"
# 检查是否有匹配 RESOURCE 的权限
HAS_PERM=$(curl -s -u $AUTH "$NEXUS/service/rest/v1/security/users?userId=$USER_ID" | \
  jq -r '.roles[]' | while read role; do
    curl -s -u $AUTH "$NEXUS/service/rest/v1/security/roles/$role" | \
      jq -r --arg res "$RESOURCE" '.privileges[] | select(contains($res))'
done)

if [ -z "$HAS_PERM" ]; then
    echo "❌ 无匹配权限：$USER_ID 对 $RESOURCE 没有任何授权"
else
    echo "✅ 匹配的权限："
    echo "$HAS_PERM" | while read perm; do echo "   $perm"; done
fi
```

```bash
chmod +x perm-check.sh
./perm-check.sh zhangsan maven-releases
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| REST API 端点缺少权限注解 | 任何认证用户都能调用该 API | 在端点方法上加 `@RequiresPermissions("nexus:xxx")` |
| Content Selector 表达式不生效 | CSEL 正确但权限检查通过 | 检查 Privilege 的 `format` 是否与请求的 format 一致 |
| Token 认证绕过了角色检查 | User Token 持有者权限比预期大 | Token 的权限 = 创建 Token 时用户拥有的权限，定期回收旧 Token |
| Shiro 缓存导致权限变更延迟 | 修改 Role 后立即测试仍返回 403 | Shiro 的权限缓存默认有 1-5 分钟存活期，修改后需等缓存过期 |

## 4. 项目总结

### 4.1 Nexus 权限模型层级

| 层级 | 概念 | 数据库存储 | 源码关键类 |
|------|------|-----------|-----------|
| L1 认证 | User + Password/Token | `user` 表 | `NexusAuthenticatingRealm` |
| L2 角色 | Role → Privilege 集合 | `role` + `role_privilege` 表 | `NexusAuthorizingRealm` |
| L3 权限 | Privilege (仓库+操作) | `privilege` 表 | `PermissionManager` |
| L4 细粒度 | Content Selector (CSEL) | `content_selector` 表 | `ContentSelectorManager` |

### 4.2 适用场景

1. **安全审计**：验证所有 REST API 端点都有适当的权限检查
2. **权限故障排查**：403 错误时快速定位是缺失哪个 Privilege
3. **插件开发**：在自定义 REST 端点中添加权限注解
4. **合规审查**：确认最小权限原则在源码层面得到执行
5. **渗透测试修复**：针对发现的越权漏洞定位修复点

### 4.3 注意事项

- **不要给匿名用户 `nx-repository-view-*-*-read`**：这会让任何人看到你的所有内部包
- **Token 的权限快照机制**：Token 创建时捕获用户当前权限，后续角色变更不影响已有 Token
- **权限检查双轨制是风险点**：审计时应重点检查 REST API 端点是否有遗漏的权限检查
- **Content Selector 性能**：每个请求都需评估 CSEL 表达式，避免使用复杂的正则表达式

### 4.4 常见踩坑经验

**故障一：给用户分配了角色但权限不生效**

某运维给新用户分配了 `role-java-developer` 角色，但用户仍报 403。排查发现：该角色的 Privilege 列表中有且只有一个拼写错误的权限名——`nx-repository-view-maven2-maven-releases-add`（正确是 `*-add`，但实际仓库名是 `maven-trade-releases`）。Nexus 角色保存时不做权限名校验，导致这个角色实际上是"空角色"。解决：建立角色创建后的验证步骤——用实际权限名赋值，不要手工输入。

**故障二：Shiro 权限缓存导致安全变更延迟**

安全团队紧急回收了一个离职用户的 Token，但该用户仍能访问 Nexus 长达 5 分钟。根因：Shiro 的 `AuthorizationInfo` 被缓存了（默认 `cachingEnabled=true`），Token 回收后缓存未失效。解决：回收 Token 时同时调用 `SecurityUtils.getSubject().logout()` 主动清除缓存；或缩短 `securityCacheTTL` 参数。

**故障三：匿名访问关闭后 Maven 插件下载 401 但依赖下载 200**

Maven 的 mirror 请求走了 settings.xml 中的 server 认证（正常），但 plugin 下载走了不同的路径——Maven 插件仓库默认不需要认证。根因：Nexus 的 group 仓库中，proxy 成员仓库可能允许匿名读取（取决于 proxy 仓库自身的权限配置）。解决：确保所有仓库（包括 proxy）都拒绝匿名访问；或统一配置 settings.xml 中的 `<pluginRepositories>` 也使用 Nexus 地址。

### 4.5 思考题

1. 设计一个"API 权限覆盖率审计"方案——自动扫描 Nexus 源码中所有 JAX-RS 端点，检查是否有对应的权限保护，并输出覆盖率报告。
2. Nexus 的权限模型中，一个用户可能拥有多个角色，每个角色包含多个 Privilege。当两个 Privilege 冲突时（一个允许 READ，另一个允许 DELETE），Nexus 如何决策？从源码角度分析"权限叠加"的规则。

（第36章思考题答案：1. 异步审计日志方案：使用 `LinkedBlockingQueue` 作为内存缓冲，`AuditRecorder` 的 `@Subscribe` 方法改为将事件放入队列后立即返回（不阻塞业务线程）。一个后台线程从队列中 batch 读取事件并批量刷盘。边界情况处理：① Nexus 正常关闭时——JVM shutdown hook 中 flush 队列中剩余事件；② Nexus 崩溃——队列中未刷盘的事件丢失，但 audit.log 文件本身已持久化的部分安全；③ 队列满——阻塞 `offer` 或丢弃最旧事件并记录放弃数量到 metrics。此方案将 P99 上传延迟从 200ms 降到 2ms。2. 外部差异化重试方案：在 Nexus 和 Webhook 接收端之间引入一个"Webhook 代理网关"（如自建的 `webhook-relay` 服务）——Nexus 只向代理发送 Webhook（固定重试策略），代理接收后再按规则转发到不同的目标——Audit Webhook 目标配置 10 次重试、Repository Webhook 配置 3 次。代理基于 Webhook payload 中的 `type` 字段（或自定义 header）路由到不同的重试策略。这样 Nexus 不用修改，灵活性在外部代理层。）

### 4.6 推广计划提示

- **安全团队**：将 `perm-check.sh` 和 API 权限覆盖率审计方案纳入定期安全审计流程
- **核心开发**：在提交新 REST 端点时，Code Review 必须确认是否包含适当的权限注解
- **运维团队**：在任何权限配置变更后，执行 `perm-check.sh` 验证目标用户的权限达到预期
