# 第15章：基础故障排查：401、403、404、502 与上传失败

## 1. 项目背景

"Nexus 又挂了！"——这是云鲸科技 IM 群里每周至少出现三次的消息。每次出现，开发组长们轮流@炮哥，炮哥再手动排查。"301 redirect?" "那明明是 401 Unauthorized。""不对，我明明配了 password，是 403 Forbidden。""别吵了，我这边是 404，包找不到。"——四个不同的错误码，表示四个完全不同的问题，却都被笼统地称为"挂了"。

更棘手的是，周一早上 9:30 是故障高发期——周末 CI 累积了大量构建任务，周一早上一拥而上，Nexus 不堪重负。Java 组报 settings.xml 配错了，前端组说 npm install 卡住不动，Docker 组反映 push 超时。炮哥需要在 10 分钟内做出判断：这是环境问题（需要重启？）、配置问题（需要改客户端？）、还是上游问题（远程仓库挂了？）。然而他翻遍了 `nexus.log` 也找不到有效线索——日志里充斥着 `DEBUG` 级别的框架日志，真正的错误淹没在噪音中。

本章将建立一线排障的标准化方法论：解析每个 HTTP 状态码在 Nexus 场景下的真正含义，学习锁定关键日志文件的方法，编写一份可交给三线支持的"信息采集清单"，并模拟 8 个经典故障场景——让每个开发、测试、运维人员都能在 5 分钟内定位根因、10 分钟内完成修复或升级。

## 2. 项目设计

大师在白板上列出了 6 个 HTTP 状态码，炮哥、老王、阿玲、浩子围坐在前。

**大师**："Nexus 排障的第一步，是读懂 HTTP 状态码。你们每个人报的'挂了下'的原因都不一样。"

**小胖**："401 和 403 到底什么区别？我老搞混。"

**大师**："401 Unauthorized = '你谁啊？我不认识你'。通常原因有两个：认证信息没带（settings.xml 没配 server、.npmrc 没配 _authToken），或者认证信息错了（密码过期、token 失效）。403 Forbidden = '我认识你，但你没资格进这扇门'——用户认证通过了，但权限不足。比如你配了 server 但账号对 `maven-releases` 只有 read 权限，deploy 就会 403。"

> **技术映射**：401 = 认证层失败（你是谁？），403 = 授权层失败（你能干什么？）。排查 401 看认证配置，排查 403 看角色权限。

**老王**："那 404 呢？包确实在仓库里，但 Maven 就是报 404。"

**大师**："404 在 Nexus 里有三种含义。第一，路径错误——你的 `mirror url` 或者 `repository url` 写错了。第二，格式不匹配——你在 Maven 仓库里请求 npm 路径，Nexus 返回 404。第三，文件真的不存在——Component 的元数据在但 Asset 的 Blob 被误删了。"

**浩子**："502 Bad Gateway 呢？Docker push 到一半突然 502。"

**大师**："502/503/504 都是服务端不可用的信号。502 通常意味着 Nexus 前面的 Nginx 收到了但后端 Nexus 连接拒绝；503 是 Nexus 的只读模式（Freeze）或者线程池打满；504 是后端处理超时。三者最快的诊断方法是直接 `curl /service/rest/v1/status`——如果超时或无响应，问题在 Nexus 进程层；如果返回 `frozen: true`，就是只读模式。"

> **技术映射**：502 = Nginx → Nexus 连接失败。503 = Nexus 主动拒绝（只读/过载）。504 = 处理超时（上传大文件/远程拉取超时）。

**小白**："Nexus 有哪些日志文件？我每次都在 nexus.log 里大海捞针。"

**大师**："Nexus 有五类关键日志。第一，`nexus.log`——应用主日志，INFO 级别记录关键操作，DEBUG 级别非常详细但噪音大。第二，`request.log`——每个 HTTP 请求一行，含 HTTP 方法、URL、状态码、耗时，排障第一站。第三，`jvm.log`——JVM GC 日志和崩溃信息。第四，`karaf.log`——Karaf 容器日志，Nexus 启动和插件加载问题看这里。第五，`audit.log`——审计日志，记录谁什么时间做了什么。排障的黄金顺序是：先看 request.log 定位请求状态码，再看 nexus.log 找异常堆栈，最后查 audit.log 确认操作者。"

**炮哥**："那我收到一个'npm install 失败'的报告，从哪里开始？"

**大师**："建立排障 SOP。第一步，**复现并确认错误码**——让用户把完整的客户端输出（包括 HTTP 状态码）发给你，不是'装不上了'。第二步，**检查 request.log**——`grep '/npm-public/' request.log | tail -20` 找到对应的请求和返回的状态码。第三步，如果是 4xx，**检查客户端配置**——.npmrc 的 registry 和 authToken；如果是 5xx，**检查 Nexus 自身状态**——`curl /service/rest/v1/status`、磁盘空间、JVM 内存。第四步，定位到问题后**输出一份简短的根因报告**——什么出了问题、谁受影响、怎么修、怎么防止下次再发生。"

### 3.2 分步实战

#### 步骤一：定位并阅读关键日志文件

**目标**：学会快速定位 Nexus 日志文件并提取关键信息。

```bash
# 进入 Nexus 容器
docker compose exec nexus bash

# 1. 日志目录结构
ls -la /nexus-data/log/

# 预期输出：
# nexus.log          # 应用主日志
# request.log        # HTTP 请求日志 ★ 排障第一站
# jvm.log            # JVM GC 日志
# karaf.log          # Karaf 容器日志
# audit/             # 审计日志目录
# tasks/             # 任务执行日志

# 2. 查看最近 20 条 HTTP 请求（含状态码和耗时）
tail -20 /nexus-data/log/request.log

# 预期输出格式（示例）：
# 192.168.1.100 - admin [15/Jan/2025:10:30:15 +0800] "GET /repository/maven-public/com/google/guava/guava/31.1-jre/guava-31.1-jre.jar HTTP/1.1" 200 2746200 234

# request.log 字段含义：
# IP - 用户 [时间] "请求方法和URL" 状态码 响应大小(字节) 耗时(ms)

# 3. 统计各 HTTP 状态码的请求量分布
cat /nexus-data/log/request.log | awk '{print $9}' | sort | uniq -c | sort -rn | head -10

# 4. 查找所有 4xx 和 5xx 请求
grep -E ' (40[134]|50[0234]) ' /nexus-data/log/request.log | tail -20

# 5. 查找最近 10 分钟的异常堆栈
grep -A 5 "ERROR\|Exception\|Caused by" /nexus-data/log/nexus.log | tail -40

# 退出容器
exit
```

#### 步骤二：模拟并排查 401 未授权错误

**目标**：模拟 Maven 认证失败场景，完整走通排查链路。

```bash
NEXUS="http://localhost:8081"

echo "=== 故障模拟：401 Unauthorized ==="

echo "上传（无认证）→ HTTP $HTTP_CODE"
echo ""

# 排查步骤
echo "=== 排查 SOP ==="

echo "   Maven: 检查 settings.xml 中 <server><id> 是否与 distributionManagement 一致"
echo "   npm: 检查 .npmrc 中 _auth / _authToken 配置"
echo "   Docker: 是否执行了 docker login"
echo "   curl: 是否带了 -u 参数"

echo ""
echo "2. 检查 request.log 中的请求记录:"
echo "   docker compose exec nexus grep '401' /nexus-data/log/request.log | tail -5"
```

**排查修复**：

```bash
# 如果是 settings.xml 问题：检查 server id 匹配
# <server>
#   <id>nexus-releases</id>           ← 这个 id
#   <username>admin</username>
#   <password>admin123</password>
# </server>

# pom.xml 中的
# <distributionManagement>
#   <repository>
#     <id>nexus-releases</id>          ← 必须和上面的 id 完全一致
#     <url>http://localhost:8081/repository/maven-releases/</url>
#   </repository>
# </distributionManagement>

# 验证修复：用 curl 模拟带认证的上传
curl -u $AUTH -X PUT \
  "http://localhost:8081/repository/maven-releases/com/test/foo/1.0/foo-1.0.jar" \
  --data-binary @/tmp/test-401.txt
# 预期输出: HTTP 201
```

#### 步骤六：编写统一排障信息采集脚本

**目标**：将所有排障所需的系统信息一键采集，方便升级给二线支持。

```bash
#!/bin/bash
# nexus-diag.sh：Nexus 诊断信息一键采集
OUTFILE="nexus-diag-$(date '+%Y%m%d_%H%M%S').txt"
NEXUS_URL="${NEXUS_URL:-http://localhost:8081}"
NEXUS_AUTH="${NEXUS_AUTH:-admin:admin123}"

{
    echo "=== Nexus 诊断报告 ==="
    echo "生成时间: $(date '+%Y-%m-%d %H:%M:%S %Z')"
    echo ""

    echo "--- 1. 服务状态 ---"
    curl -s -u "$NEXUS_AUTH" "$NEXUS_URL/service/rest/v1/status" | jq '.'
    echo ""

    echo "--- 2. 最近 20 条 HTTP 错误 (request.log) ---"
    docker compose exec -T nexus cat /nexus-data/log/request.log 2>/dev/null | \
      grep -E ' (40[134]|50[0234]) ' | tail -20
    echo ""

    echo "--- 3. 最近 50 行应用错误 (nexus.log) ---"
    docker compose exec -T nexus tail -50 /nexus-data/log/nexus.log 2>/dev/null
    echo ""

    echo "--- 4. 磁盘使用 ---"
    docker compose exec -T nexus df -h /nexus-data 2>/dev/null
    echo ""

    echo "--- 5. 容器资源 ---"
    docker stats nexus --no-stream 2>/dev/null
    echo ""

    echo "--- 6. 仓库状态 ---"
    curl -s -u "$NEXUS_AUTH" "$NEXUS_URL/service/rest/v1/repositories" | \
      jq '.[] | {name, online, type, format}'
    echo ""

    echo "--- 7. 正在执行的任务 ---"
    curl -s -u "$NEXUS_AUTH" "$NEXUS_URL/service/rest/v1/tasks" | \
      jq '.items[] | select(.status == "running") | {name, type, currentState}'
    echo ""

    echo "--- 8. BlobStore 容量 ---"
    curl -s -u "$NEXUS_AUTH" "$NEXUS_URL/service/rest/v1/blobstores" | \
      jq '.[] | {name, totalSize, blobCount}'
    echo ""

} > "$OUTFILE"

echo "✅ 诊断报告已生成: $OUTFILE"
echo "请将此文件发送给二线运维支持团队。"
```

```bash
chmod +x nexus-diag.sh && ./nexus-diag.sh
```

### 3.3 常见故障速查表

| HTTP 状态码 | Nexus 场景含义 | 常见原因 TOP3 | 首要检查 |
|------------|---------------|-------------|---------|
| 401 | 未认证 | ① 客户端没配认证 ② 密码错误/token 过期 ③ Realm 未启用 | 客户端配置文件 |
| 403 | 无权操作 | ① 角色缺少对应 privilege ② 仓库无写权限 ③ Content Selector 限制 | 用户角色列表 |
| 404 | 资源不存在 | ① URL 路径错误 ② 仓库名错误 ③ Blob 被误删 | request.log 中的完整 URL |
| 502 | 网关错误 | ① Nginx 到 Nexus 连接失败 ② Nexus 进程挂了 | docker compose ps |
| 503 | 服务不可用 | ① 只读模式 ② 线程池满 ③ 磁盘满 | /status/writable |
| 504 | 网关超时 | ① 大文件处理超时 ② 远程仓库响应慢 | proxy connection timeout |

## 4. 项目总结

### 4.1 故障响应分级

| 等级 | 影响范围 | 响应时间 | 典型场景 | 值班角色 |
|------|---------|---------|---------|---------|
| P0 | 全公司不可用 | 5 分钟 | 磁盘满导致写拒绝 | 运维 + 开发 |
| P1 | 单个团队不可用 | 15 分钟 | 仓库被误删 | 运维 |
| P2 | 部分功能异常 | 1 小时 | proxy 缓存过期 | 开发 |
| P3 | 单个用户/项目 | 4 小时 | 401/403 配置问题 | 开发 |

### 4.2 适用场景

1. **日常开发排障**：`mvn compile` 失败、`npm install` 报错、`docker pull` 超时
2. **CI/CD 构建失败**：流水线发布步骤报错，快速区分是代码问题还是 Nexus 问题
3. **运维值班**：收到监控告警后 5 分钟内定位根因
4. **故障复盘**：通过 request.log 和 audit.log 还原事故时间线
5. **新人培训**：作为 Nexus 排障的标准化培训材料

### 4.3 注意事项

- **永远从 request.log 开始**：它记录了每一个请求的状态码和耗时，是最直接的故障证据
- **不要直接改生产 Nexus 的日志级别为 DEBUG**：DEBUG 日志量巨大，可能在几分钟内写满磁盘
- **权限问题是人因问题**：大部分 401/403 不是 Nexus 的 bug，而是配置被改了或者用户角色变了
- **先止血，后根因分析**：生产故障第一目标是恢复服务（重启/解冻/切换），根因分析在事后进行

### 4.4 常见踩坑经验

**故障一：request.log 滚动后被覆盖**

某次排障需要看 3 小时前的请求日志，但 request.log 已被滚动压缩（`.log.1.gz`）。解决：Nexus 默认保留 request.log 的滚动文件，用 `zcat request.log.1.gz | grep "401"` 搜索压缩的历史日志。

**故障二：Nginx 反向代理返回 502 但 Nexus 正常**

Nginx 日志显示 `upstream prematurely closed connection`，但 Nexus 进程存活。根因：Nginx 的 `proxy_read_timeout` 只有 30 秒，Docker pull 大 layer 超过 30 秒就给客户端返回 502，但 Nexus 实际仍在处理。解决：将 Nginx `proxy_read_timeout` 调整为 600 秒。

**故障三：错误修改角色的权限导致全员 403**

管理员在修改 `role-global-reader` 时误删了 `*-read` 权限，导致所有拥有此角色的用户无法下载任何制品。排查路径：request.log 全部 403 → 检查用户角色 → 角色权限列表为空。修复：快速回滚权限变更（通过审计日志找到变更前的权限列表）。

### 4.5 思考题

1. 一个 Maven 项目报告 `Could not find artifact com.cloudwhale:common-data:1.5.1`，但你在 Nexus Web UI 的 Browse 页面确认文件存在。请按排障 SOP 逐步列出可能的根因和验证方法。
2. 设计一个自动化的"健康检查 × 故障自愈"系统：当 Nexus 进入只读模式（frozen）时，系统自动执行诊断脚本，如果磁盘使用率 > 95% 则触发自动清理和 Compact，如果一切正常但 frozen 持续超过 10 分钟则发 P0 告警。

（第14章思考题答案：1. 跨主机备份方案：① 备份脚本执行结束后计算备份目录的 sha256sum 摘要文件；② 使用 MinIO 客户端 `mc mirror` 同步到远程 bucket，`--retry 3` 做重试；③ 上传后执行 `mc stat` 确认远程文件存在且大小匹配；④ 本地保留 7 天（`find -mtime +7 -delete`），远程通过 MinIO Bucket Lifecycle 策略保留 30 天；⑤ 网络中断时脚本记录失败状态，通过 Webhook 告警，并在下一轮执行时自动追补。2. 恢复后 '即用' 方案：① 恢复完成后立即执行一个初始化脚本，生成客户端的认证配置文件模板（settings.xml / .npmrc / docker config.json）；② 脚本通过 API 为每个团队创建新的 API Token；③ 将配置文件通过公司内部的配置下发系统（如 Consul/Ansible）推送到所有开发机和 CI 节点；④ 或者使用 Nginx 反向代理做认证剥离——客户端用统一 Token 访问 Nginx，Nginx 用 Nexus API Token 转发，Token 变更只需改 Nginx 配置。）
