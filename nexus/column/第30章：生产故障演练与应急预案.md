# 第30章：生产故障演练与应急预案

## 1. 项目背景

云鲸科技的 CTO 在季度稳定性复盘会上说了一句令全场沉默的话："我们不知道 Nexus 挂掉之后要怎么恢复——因为从来没练过。"这句话的背景是上周三的一次真实故障——`/nexus-data` 所在磁盘被一个 50GB 的备份日志文件意外写满，Nexus 进入只读模式（frozen）。从磁盘满到发现花了 45 分钟，从发现到恢复又花了 90 分钟——其中 60 分钟花在"要不要重启"、"重启会不会丢数据"、"谁来做"的内部争论上。

"演练是检验应急预案的唯一标准"这句话在运维圈人人会背，但真正落实的团队寥寥无几。对于 Nexus 这种"平时没事、出事就是大事"的关键基础设施，缺乏故障演练的代价不是在故障时多花几十分钟——而是整个研发团队的生产力瘫痪。由 200 个开发者每小时的开发生效率损失 × 2 小时的故障恢复时间，换算成经济成本可能高达数十万元。

本章将从云鲸科技的实际故障经验出发，设计 6 个真实故障演练剧本（磁盘满、数据库损坏、BlobStore 误删、权限被清空、远程仓库不可用、任务卡死），每个演练包含触发条件、预期症状、诊断步骤、止血方案和恢复流程，最终形成一份可在团队内季度执行的应急手册。

## 2. 项目设计

大师拿着上周故障的复盘报告，在白板上画了一条时间线。

**大师**："上周的故障从发生到完全恢复花了 135 分钟。我把这条时间线拆开——前 45 分钟是 MTTD（平均检测时间），后 90 分钟是 MTTR（平均恢复时间）。45 分钟的检测时间完全是监控缺失造成的——如果磁盘使用率超过 85% 时有告警，根本不会等到 100% 才发现。"

**炮哥**："那恢复的 90 分钟呢？为什么这么久？"

**大师**："因为没人知道 Nexus frozen 了该怎么处理。我看了那天运维群里的聊天记录——有人说要重启容器、有人说要先备份再重启、有人说 frozen 只是不能写但不影响下载可以等到凌晨再处理。花了 60 分钟争论，实际动手只用了 30 分钟。归根结底——没有预案，也没有演练过。"

> **技术映射**：MTTD（故障检测时间）依赖监控和告警，MTTR（故障恢复时间）依赖预案和演练。两者都能通过本章的故障演练剧本实质性降低。

**小胖**："那我们设计几个演练？每次执行演练会不会真的把 Nexus 搞挂？"

**大师**："演练分两种——**桌面推演**（Tabletop）和**实战演练**（Live Drill）。桌面推演是大家围坐在一起，拿着故障剧本一步步讨论'如果你是值班运维，看到这个告警你第一步做什么'——零风险。实战演练是在一个隔离的测试 Nexus 实例上真的触发故障然后按照预案恢复——有一定风险但训练效果最好。建议季度一次实战演练，月度一次桌面推演。"

**小白**："每个故障的应急预案应该包含什么？"

**大师**："标准七段式——**触发条件**（什么告警/现象表示这个故障发生了）、**影响范围**（哪些人/团队/仓库受影响）、**诊断步骤**（执行什么命令确认根因）、**止血方案**（先让服务恢复可用，哪怕不是最优方案）、**根因修复**（彻底解决问题）、**验证方法**（确认修复后服务恢复正常）、**复盘模板**（记录关键时间点和改进项）。"

**炮哥**："磁盘满、数据库异常、BlobStore 损坏、权限误删、远程仓库不可用、任务卡死——这六个够覆盖吗？"

**大师**："这六个覆盖了 Nexus 故障的四大类——**存储故障**（磁盘满、BlobStore 损坏、数据库异常）、**安全故障**（权限误删）、**外部依赖故障**（远程仓库不可用）、**内部服务故障**（任务卡死）。基本够用。每半年可以根据实际发生的故障增补新的演练剧本。"

## 3. 项目实战

### 3.1 环境准备

- 已部署 Nexus 实例（生产环境 + 独立测试环境）

### 3.2 分步实战

#### 步骤一：演练 1 —— 磁盘满导致 Nexus 进入只读模式

**目标**：训练团队在磁盘满时的快速恢复流程。

```bash
# === 演练剧本 1：磁盘满 → frozen ===

# 触发条件 :
# - Grafana 告警: Nexus BlobStore 使用率 > 90% → P1
# - 现象: 上传操作返回 HTTP 503, /service/rest/v1/status 返回 frozen: true

# 影响范围 :
# - 所有格式的 deploy/push/publish 操作不可用
# - 下载/拉取仍然可用
# - CI 流水线中发布步骤全部失败

# 诊断步骤 (5 分钟) :
# 1. 确认 frozen 状态
curl -s "http://nexus:8081/service/rest/v1/status" | jq '{status, frozen}'
# 2. 检查磁盘使用率
df -h /nexus-data
# 3. 找出占用最大的目录
du -sh /nexus-data/blobs/*/
du -sh /nexus-data/log/

# 止血方案 (10 分钟) :
# 方案 A（快速）：清理旧日志文件
find /nexus-data/log -name "*.log" -mtime +7 -delete
find /nexus-data/log -name "*.gz" -delete
# 方案 B（如 A 不够）：执行 Admin - Compact blob store 任务回收空间
# 方案 C（如 A+B 不够）：手动触发 Cleanup Task 删除过期组件
curl -u admin:admin123 -X POST \
  "http://nexus:8081/service/rest/v1/status/unfreeze"

# 根因修复 (事后) :
# - 调整 BlobStore 配额（softQuota.limit）
# - 增加清理策略的覆盖范围或提高执行频率
# - 扩容磁盘或迁移到更大存储

# 验证方法 :
curl -s "http://nexus:8081/service/rest/v1/status" | jq '{frozen, writable}'
curl -u admin:admin123 -X PUT "http://nexus:8081/repository/raw-hosted/test-drill.txt" -d "test"
# 预期: HTTP 201 Created
```

#### 步骤二：演练 2 和 3 —— 数据库异常 + BlobStore 误删

**目标**：训练数据库损坏和 BlobStore 数据丢失的恢复流程。

```bash
# === 演练剧本 2：数据库损坏导致 Nexus 启动失败 ===

# 触发条件 :
# - 告警: NexusDown (Prometheus up{job="nexus"} == 0)
# - 现象: Nexus 容器不断重启, 日志中报 "Database corrupted"

# 诊断:
docker compose logs nexus | grep -i "database\|corrupt\|orientdb"
# 确认数据库文件路径: /nexus-data/db/

# 止血方案: 从最近一次备份恢复数据库
# 1. 停止 Nexus
docker compose stop nexus
# 2. 备份当前的损坏数据（取证用）
mv /nexus-data/db /nexus-data/db.corrupted.$(date +%Y%m%d_%H%M%S)
# 3. 从备份恢复
cp -r /backup/nexus-backup-<latest>/db /nexus-data/db
chown -R 200:200 /nexus-data/db
# 4. 重启
docker compose start nexus

# 恢复时间目标 (RTO): < 30 分钟
# 数据恢复点目标 (RPO): 最后一次备份的时间点

# === 演练剧本 3：误删 BlobStore 文件导致制品下载 404 ===

# 触发条件:
# - 用户报告: 之前能下载的包现在返回 404
# - request.log 中对应请求返回 404

# 诊断:
curl "http://nexus:8081/repository/maven-releases/com/example/lib/1.0/lib-1.0.jar"
# 确认: HTTP 404
# 在 Nexus Web UI Browse 中确认组件存在但 asset 不可下载

# 止血方案:
# Blob 文件一旦从磁盘删除无法恢复（除非有备份）
# 1. 如果能重新 build → 重新 deploy 该版本
# 2. 从备份中提取单个 blob 文件 → 这是高级操作
#    从备份的 blobs/ 目录中找到对应的 .bytes 文件（通过 API 获取 blob ID）
#    拷贝到当前 blobs/ 目录的对应路径
#    再执行 Rebuild repository index 任务

# 预防措施:
# - 严格权限管理: 只有 admin 可访问 /nexus-data
# - 定期备份 BlobStore
# - 严禁手工操作 blobs/ 目录
```

#### 步骤三：演练 4 —— 权限被清空

**目标**：训练权限批量丢失后的快速恢复。

```bash
# === 演练剧本 4：误操作导致角色权限被清空 ===

# 触发条件:
# - 大量用户同时报告 403 Forbidden
# - request.log 中大量 403

# 诊断:
# 1.检查角色是否正常（以 role-trade-developer 为例）
curl -u admin:admin123 \
  "http://nexus:8081/service/rest/v1/security/roles/role-trade-developer" | \
  jq '{id, privileges}'
# 如果 privileges 为空 → 确认权限丢失

# 2. 检查审计日志中最近的 security 域变更
# ES 查询: domain: "security" AND type: "role" AND action: "UPDATED"

# 止血方案:
# 如果在 30 分钟内发现 → 从第 25 章的 YAML 配置重新执行 nexus-as-code.sh
# ./nexus-as-code.sh configs/roles-and-permissions.yaml

# 如果没有 YAML → 在另一个环境（staging）上通过 API 导出角色再导入
curl -u admin:admin123 \
  "http://nexus-staging:8081/service/rest/v1/security/roles" | \
  jq '.' > roles-backup.json

# 根因修复：
# - 限制 nx-admin 角色的使用
# - 权限变更操作通过 PR 审批
# - 配置 Webhook 告警 security 域的变更

# 预防措施:
# - 使用第 25 章 nexus-as-code.sh 管理所有权限
# - Git 中保留配置 YAML，变更可追溯和回滚
```

#### 步骤四：演练 5 和 6 —— 远程仓库不可用 + 任务卡死

**目标**：训练外部依赖故障和内部任务异常的应对。

```bash
# === 演练剧本 5：远程仓库不可用（如 Maven Central 故障） ===

# 触发条件:
# - Grafana 告警: NexusProxyUnavailable → P2
# - 现象: 下载未缓存的公共包时返回 503

# 诊断:
# 1. 检查 proxy 仓库状态
curl -u admin:admin123 \
  "http://nexus:8081/service/rest/v1/repositories/maven-central" | \
  jq '{name, online, blocked: .httpClient.blocked}'
# 如果 blocked: true → proxy 被 auto-block

# 2. 验证远程是否真的不可用
curl -I --connect-timeout 5 https://repo1.maven.org/maven2/

# 止血方案:
# - 已缓存的包不受影响，正常服务
# - 未缓存的包 → 如果 autoBlock 已启动:
#   curl -u admin:admin123 -X POST \
#     "http://nexus:8081/service/rest/v1/repositories/maven/proxy/maven-central/health-check"
#   等待 auto-block 自动恢复（5-15 分钟）
#   - 或手动 unblock:
#   curl -u admin:admin123 -X PUT \
#     "http://nexus:8081/service/rest/v1/repositories/maven/proxy/maven-central" \
#     -H "Content-Type: application/json" \
#     -d '{"httpClient":{"blocked":false}}'

# 长期方案:
# - 配置多 proxy 冗余（Maven Central + Aliyun Mirror 双 proxy）
# - group 中成员顺序排列，一个不可用时自动 fallback

# === 演练剧本 6：任务卡死导致后续任务不执行 ===

# 触发条件:
# - 告警: 任务失败率升高
# - 现象: 某个任务状态一直是 "running" 超过预期执行时间

# 诊断:
curl -u admin:admin123 \
  "http://nexus:8081/service/rest/v1/tasks" | \
  jq '.items[] | select(.status=="running") | {name, type, lastRun}'

# 止血方案:
# 1. 在 Web UI System → Tasks → 选中卡死任务 → Stop
# 2. 或通过 API 停止:
# TASK_ID="<卡死任务 ID>"
# curl -u admin:admin123 -X POST \
#   "http://nexus:8081/service/rest/v1/tasks/$TASK_ID/stop"

# 根因修复:
# - 分析卡死任务日志
# - 如果是 Cleanup 任务卡死 → 仓库组件太多需要分批
# - 如果是 Compact 任务卡死 → BlobStore 太多待删除 blob
# - 给重 IO 任务设置资源上限（Groovy 脚本中可以限制处理数量）
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| 演练后发现备份文件过旧 | 恢复测试时发现上一次备份是 2 周前 | 将备份脚本健康检查加入演练 checklist |
| 桌面推演流于形式 | 每个人都知道答案但实际故障时仍手忙脚乱 | 桌面推演中要求每个人写出具体的命令，而不是口述 |
| 实战演练影响了生产 | 测试环境的清理任务误关联到生产仓库 | 实战演练必须在完全隔离的测试 Nexus 上执行 |
| 应急手册从未更新 | 第一次演练后手册内容过时 | 每次演练后强制更新手册并 commit 到 Git |

## 4. 项目总结

### 4.1 故障剧本速查表

| 编号 | 故障名 | 典型告警 | 影响 | RTO | 第一命令 |
|------|--------|---------|------|-----|---------|
| E1 | 磁盘满 | BlobStoreHighUsage + 上传 503 | 写操作不可用 | 15min | `curl /status` + `df -h` |
| E2 | 数据库损坏 | NexusDown | 完全不可用 | 30min | 从备份恢复 db/ |
| E3 | Blob 丢失 | 下载 404 | 部分包不可用 | 30min | 重新 deploy |
| E4 | 权限清空 | 大量 403 | 大部分操作不可用 | 10min | re-run nexus-as-code |
| E5 | Remote Unavailable | ProxyUnavailable | 新包不可下载 | 15min | manual unblock |
| E6 | 任务卡死 | 后续任务不执行 | 维护动作中断 | 5min | Stop task |

### 4.2 适用场景

1. **新人入职培训**：用故障演练作为 Nexus 运维的"实战考试"
2. **季度稳定性审查**：验证 RTO/RPO 是否达标
3. **架构变更验证**：在新增/移除核心组件后演练相关故障
4. **合规审计**：向审计方证明团队具备应急响应能力

### 4.3 注意事项

- **演练在隔离环境**：实战演练绝不可以在生产 Nexus 上进行
- **记录每个步骤的耗时**：用于优化应急预案中的时间估计
- **演练后 48 小时内完成复盘**：趁记忆新鲜记录改进项
- **演练不是惩罚**：目标是指出流程缺陷而非追究个人责任

### 4.4 常见踩坑经验

**故障 E1 变体：日志写满而非 Blob 写满**

运维以为磁盘满一定是 Blob 太多，反复执行 Compact 但无效。根因：GC 日志开启了 DEBUG 级别，每天产生 3GB+ 日志文件。解决：检查 `du -sh /nexus-data/log`，调整日志级别。

**故障 E4 变体：某角色被部分修改**

一个角色的 privileges 列表被误改为空数组导致大批用户 403。根因：API PUT 是完整替换而非增量更新。解决：恢复角色配置时用之前导出的 JSON 备份。

**故障 E5 变体：autoBlock 持续不解冻**

proxy 已恢复正常但 autoBlock 仍然 Blocked。根因：autoBlock 的探测请求所用的 URL 被防火墙拦截了。解决：检查 `httpClient.connection` 的超时和重试设置，确保探测 URL 可达。

### 4.5 思考题

1. 设计一个"混沌工程"实验：在测试 Nexus 上随机注入网络延迟（100ms→500ms 随机波动），观察客户端（Maven/npm/Docker）的容错能力和 Nexus 的自我保护机制。哪些参数可以控制 Nexus 在恶劣网络下的行为？
2. 如何将应急手册中的 6 个故障剧本转化为 Prometheus Alertmanager 的告警路由规则？设计一套"告警 → 自动诊断 → 推送处置建议"的自愈流程雏形。
