# 第14章：备份、恢复与升级前演练

## 1. 项目背景

"炮哥不好了！服务器磁盘阵列掉电，Nexus 数据全丢了！"——周五晚上 11 点，这条消息在运维群里炸开了锅。

炮哥赶到机房，确认了残酷的事实：RAID 卡故障导致两块盘同时离线，`/nexus-data` 所在的分区数据全部不可读。更要命的是，由于 Nexus"一直很稳定"，谁也没想过做备份——"Docker 容器嘛，坏了重建一个不就行了？"但重建的 Nexus 是一个空壳——三个月的 Maven 缓存、两周前发布的 12 个正式版本、前端组引以为傲的 `@cloudwhale/ui-components` 组件库、运维组所有 Agent 安装包的历史版本——全部消失。

那天晚上，整个研发团队被迫回退到最原始的"共享目录"模式：老王贡献了自己本地 `~/.m2/repository` 里的 jar 包，阿玲紧急在 npmjs 上恢复了一些私有包的备份，浩子重新从 Dockerfile 构建基础镜像。但那些只在 Nexus 中存在过的内部发布包——永久丢失。后续两周，研发效率下降约 40%，因为每个项目都要重新发布一次基础依赖。

事后复盘时，安全总监冷冷地问了一句："如果这不是硬件故障，而是勒索病毒呢？你们能恢复到哪个时间点？"

Nexus 不是无状态服务——它持有的是整个企业软件开发的生命线数据。将 Nexus 当作关键基础设施保护，建立备份、恢复和升级演练体系，是本章要解决的核心命题。

## 2. 项目设计

事故复盘会上，大师在白板上画了一个简单的图。

**大师**："Nexus 的数据分四层。最底层是**数据库**（OrientDB，存在 `/nexus-data/db/`），存所有配置——仓库定义、用户、角色、权限、任务、清理策略。上一层是**BlobStore**（`/nexus-data/blobs/`），存所有制品的二进制文件。再往上是**配置目录**（`/nexus-data/etc/`），存 nexus.properties、SSL keystore 等。最上层是**日志**（`/nexus-data/log/`），用于审计和排障。"

**小胖**："所以备份就是 `cp -r /nexus-data /backup/` 吗？"

**大师**："方向对，细节错。一个关键的约束是：备份时必须**暂停 Nexus 的写操作**，否则备份出的数据库可能处于中间状态——恢复后无法启动。Nexus 提供了两种模式：**只读模式（Freeze）**和**完全停止服务**。只读模式适合热备——暂停所有写入（上传、删除、配置修改），但下载仍然可用。完全停止服务适合冷备——最安全，但需要维护窗口。"

> **技术映射**：Nexus 备份的黄金法则 = Freeze（暂停写）→ rsync（拷贝数据）→ Release（恢复写）。类似数据库的 flush + lock tables + dump 流程。

**小白**："那恢复的时候呢？直接把备份文件覆盖回 `/nexus-data` 然后重启就行？"

**大师**："理论上是。但有三个陷阱。第一，**数据库版本兼容性**——如果备份时 Nexus 版本是 3.60，恢复时用的是 3.70，数据库可能无法直接挂载。第二，**BlobStore 路径一致性**——备份时用的绝对路径必须在恢复环境中完全一样。第三，**认证信息**——admin 密码、user token 全部在备份中，恢复后自动生效。"

**炮哥**："升级前的备份有什么特殊要求？"

**大师**："升级前的备份是'后悔药'——如果升级后出现严重 bug 或不兼容，你需要回退到旧版本。回退流程是：保持旧版本安装包 → 恢复升级前备份 → 重启旧版本。注意：Nexus 3.x 不支持版本降级——你不能直接把 3.70 的数据目录给 3.60 用。只能通过备份恢复的方式实现回退。"

> **技术映射**：升级前备份 = 全量快照。恢复 = 数据回到快照时刻 + 软件回到旧版本。

**小胖**："那我们多长时间备份一次？每天备份太占存储了吧？"

**大师**："分层策略。**配置数据**（db/ + etc/）小但关键——每天备份，保留 30 天。**BlobStore** 中 release 部分——每周备份，保留 4 周。**BlobStore** 中 proxy 缓存——不备份，因为可以重新从远程拉取。**BlobStore** 中 Docker layer——根据变更频率，每日或每周。总体原则：备份成本 > 数据丢失成本的就备份；反之不备份。"

**小白**："备份和快照什么区别？我们公司存储有定时快照功能，还用单独做备份吗？"

**大师**："快照是存储层的 crash-consistent 拷贝——它不知道 Nexus 数据库内部是否一致。如果在数据库写入中途做快照，恢复后数据库可能损坏。所以，**快照不能替代应用级备份**——必须在快照前先 Freeze Nexus，确保数据库处于一致状态。"

## 3. 项目实战

### 3.1 环境准备

- 已部署 Nexus 实例
- 具有 sudo 权限的宿主机访问
- rsync 工具（通常已预装）

### 3.2 分步实战

#### 步骤一：Freeze/Unfreeze — 暂停与恢复写入

**目标**：掌握只读模式的开关，为安全备份做准备。

```bash
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

# 1. 检查当前状态
curl -s -u $AUTH "$NEXUS/service/rest/v1/status" | jq '{frozen, writable}'

# 预期输出：{"frozen": false, "writable": true}

# 2. 启用只读模式（Freeze）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/status/freeze" \
  -H "Content-Type: application/json"

# 预期输出：HTTP 200（无 body）

# 3. 验证只读模式
curl -s -u $AUTH "$NEXUS/service/rest/v1/status" | jq '{frozen, writable}'
# 预期输出：{"frozen": true, "writable": false}

# 4. 试上传文件（应被拒绝）
echo "test" > /tmp/test-freeze.txt
curl -u $AUTH -X PUT \
  "http://localhost:8081/repository/raw-agents/test-freeze.txt" \
  --data-binary @/tmp/test-freeze.txt

# 预期输出：HTTP 503 Service Unavailable

# 5. 备份完成后恢复写入（Unfreeze）
curl -u $AUTH -X POST "$NEXUS/service/rest/v1/status/unfreeze" \
  -H "Content-Type: application/json"
```

#### 步骤二：编写全量备份脚本

**目标**：编写生产级备份脚本，支持 Freeze → 备份 → Unfreeze 流程。

```bash
#!/bin/bash
# nexus-backup.sh：Nexus 全量备份脚本
set -e

NEXUS_URL="${NEXUS_URL:-http://localhost:8081}"
NEXUS_AUTH="${NEXUS_AUTH:-admin:admin123}"
NEXUS_DATA="${NEXUS_DATA:-$HOME/nexus-local/data}"
BACKUP_DIR="${BACKUP_DIR:-$HOME/nexus-backups}"
RETENTION_DAYS="${RETENTION_DAYS:-30}"

TIMESTAMP=$(date '+%Y%m%d_%H%M%S')
BACKUP_PATH="$BACKUP_DIR/nexus-backup-$TIMESTAMP"

echo "=== Nexus 备份开始 ==="
echo "时间: $(date)"
echo "备份路径: $BACKUP_PATH"

# 1. 检查 Nexus 是否可访问
echo "[1/5] 检查 Nexus 状态..."
STATUS=$(curl -sf -u "$NEXUS_AUTH" "$NEXUS_URL/service/rest/v1/status" | jq -r '.status')
if [ "$STATUS" != "running" ]; then
    echo "错误: Nexus 不处于运行状态"
    exit 1
fi
echo "  Nexus 状态: $STATUS"

# 2. 启用只读模式
echo "[2/5] 启用只读模式..."
RESP=$(curl -s -w "%{http_code}" -o /dev/null -u "$NEXUS_AUTH" \
  -X POST "$NEXUS_URL/service/rest/v1/status/freeze" \
  -H "Content-Type: application/json")

if [ "$RESP" != "200" ] && [ "$RESP" != "204" ]; then
    echo "错误: Freeze 失败 (HTTP $RESP)"
    exit 1
fi
echo "  Nexus 已进入只读模式"

# 3. 执行备份
echo "[3/5] 执行数据备份..."
mkdir -p "$BACKUP_PATH"

# 备份关键目录
rsync -av --delete "$NEXUS_DATA/db/" "$BACKUP_PATH/db/" 2>&1 | tail -1
rsync -av --delete "$NEXUS_DATA/etc/" "$BACKUP_PATH/etc/" 2>&1 | tail -1
rsync -av --delete "$NEXUS_DATA/blobs/" "$BACKUP_PATH/blobs/" 2>&1 | tail -1
rsync -av --delete "$NEXUS_DATA/log/" "$BACKUP_PATH/log/" 2>&1 | tail -1

# 保存备份版本信息
echo "备份时间: $TIMESTAMP" > "$BACKUP_PATH/BACKUP_INFO.txt"
echo "Nexus 版本: $(curl -sf -u "$NEXUS_AUTH" "$NEXUS_URL/service/rest/v1/status" | jq -r '.version')" >> "$BACKUP_PATH/BACKUP_INFO.txt"
echo "备份主机: $(hostname)" >> "$BACKUP_PATH/BACKUP_INFO.txt"

BACKUP_SIZE=$(du -sh "$BACKUP_PATH" | awk '{print $1}')
echo "  备份大小: $BACKUP_SIZE"

# 4. 恢复读写模式
echo "[4/5] 恢复读写模式..."
curl -s -u "$NEXUS_AUTH" -X POST "$NEXUS_URL/service/rest/v1/status/unfreeze" \
  -H "Content-Type: application/json"
echo "  Nexus 已恢复读写模式"

# 5. 清理过期备份
echo "[5/5] 清理 $RETENTION_DAYS 天前的旧备份..."
find "$BACKUP_DIR" -maxdepth 1 -type d -name "nexus-backup-*" -mtime +$RETENTION_DAYS -exec rm -rf {} \; 2>/dev/null
echo "  过期备份已清理"

echo "=== 备份完成 ==="
echo "备份路径: $BACKUP_PATH"
echo "备份大小: $BACKUP_SIZE"
```

```bash
chmod +x nexus-backup.sh
# 执行备份
sudo ./nexus-backup.sh
```

#### 步骤三：执行恢复演练

**目标**：从备份文件恢复 Nexus 到全新的 Docker 容器，验证数据完整性。

```bash
#!/bin/bash
# nexus-restore-drill.sh：恢复演练脚本
set -e

BACKUP_PATH="${1:-}"
RESTORE_DATA="${2:-$HOME/nexus-restore/data}"

if [ -z "$BACKUP_PATH" ]; then
    echo "用法: $0 <backup-path> [restore-data-dir]"
    echo "示例: $0 ~/nexus-backups/nexus-backup-20250115_030000"
    exit 1
fi

if [ ! -f "$BACKUP_PATH/BACKUP_INFO.txt" ]; then
    echo "错误: 备份目录无效（缺少 BACKUP_INFO.txt）"
    exit 1
fi

echo "=== Nexus 恢复演练 ==="
echo "备份来源: $BACKUP_PATH"
echo "恢复目标: $RESTORE_DATA"
echo ""

# 1. 准备空的恢复目录
echo "[1/5] 准备恢复目录..."
if [ -d "$RESTORE_DATA" ]; then
    echo "  警告: $RESTORE_DATA 已存在，将清空"
    rm -rf "$RESTORE_DATA"
fi
mkdir -p "$RESTORE_DATA"
chmod 777 "$RESTORE_DATA"  # Docker 需要写权限

# 2. 拷贝备份数据
echo "[2/5] 拷贝备份数据..."
cp -r "$BACKUP_PATH/db" "$RESTORE_DATA/"
cp -r "$BACKUP_PATH/etc" "$RESTORE_DATA/"
cp -r "$BACKUP_PATH/blobs" "$RESTORE_DATA/"
cp -r "$BACKUP_PATH/log" "$RESTORE_DATA/" 2>/dev/null || true
echo "  数据拷贝完成"

# 3. 修正目录权限（Nexus 容器内以 UID 200 运行）
echo "[3/5] 修正目录权限..."
chown -R 200:200 "$RESTORE_DATA"
chmod -R 755 "$RESTORE_DATA"
echo "  权限修正完成"

# 4. 创建恢复 Compose 文件
echo "[4/5] 创建恢复环境..."
RESTORE_COMPOSE="$HOME/nexus-restore/docker-compose.yml"
mkdir -p "$(dirname "$RESTORE_COMPOSE")"

cat > "$RESTORE_COMPOSE" << COMPOSE
version: "3.8"
services:
  nexus-restore:
    image: sonatype/nexus3:latest
    container_name: nexus-restore
    restart: unless-stopped
    ports:
      - "9081:8081"  # 使用不同端口避免冲突
    volumes:
      - $RESTORE_DATA:/nexus-data
    environment:
      - INSTALL4J_ADD_VM_PARAMS=-Xms512m -Xmx1024m
COMPOSE

# 5. 启动恢复实例并验证
echo "[5/5] 启动恢复实例..."
cd "$(dirname "$RESTORE_COMPOSE")"
docker compose up -d

echo ""
echo "=== 等待 Nexus 启动（最多 5 分钟） ==="
for i in $(seq 1 60); do
    if curl -sf http://localhost:9081/service/rest/v1/status > /dev/null 2>&1; then
        echo "✅ Nexus 恢复实例已启动！"
        break
    fi
    echo "  等待中... ($i/60)"
    sleep 5
done

echo ""
echo "=== 恢复验证 ==="
echo "验证项目:"
echo "1. 打开浏览器访问 http://localhost:9081"
echo "2. 使用备份时的 admin 账号登录"
echo "3. 检查仓库是否存在，制品是否可下载"
echo "4. 执行验证命令:"
echo "   curl -u admin:<password> http://localhost:9081/service/rest/v1/repositories | jq '.[].name'"

# 显示备份信息
echo ""
echo "备份信息:"
cat "$BACKUP_PATH/BACKUP_INFO.txt"

echo ""
echo "恢复完成后，请手动清理："
echo "  cd $(dirname "$RESTORE_COMPOSE") && docker compose down"
echo "  rm -rf $RESTORE_DATA"
```

```bash
chmod +x nexus-restore-drill.sh
# 执行恢复演练
./nexus-restore-drill.sh ~/nexus-backups/nexus-backup-20250115_030000
```

#### 步骤四：编写升级前检查清单脚本

**目标**：在升级前自动完成数据备份、插件兼容性检查、BlobStore 完整性校验。

```bash
#!/bin/bash
# pre-upgrade-check.sh：Nexus 升级前准备脚本
NEXUS_URL="http://localhost:8081"
AUTH="admin:admin123"

echo "=== Nexus 升级前检查 ==="
echo "当前时间: $(date)"
echo ""

# 1. 获取当前版本
echo "[检查1] 当前版本信息"
CURRENT_VERSION=$(curl -s -u $AUTH "$NEXUS_URL/service/rest/v1/status" | jq -r '.version')
echo "  当前版本: $CURRENT_VERSION"
echo ""

# 2. 检查磁盘空间（升级需要额外空间）
echo "[检查2] 磁盘空间"
DATA_DIR="$HOME/nexus-local/data"
USAGE=$(df -h "$DATA_DIR" | tail -1 | awk '{print $5}')
AVAILABLE=$(df -h "$DATA_DIR" | tail -1 | awk '{print $4}')
echo "  数据目录: $DATA_DIR"
echo "  已使用: $USAGE"
echo "  可用空间: $AVAILABLE"
echo ""

# 3. 检查所有仓库是否在线
echo "[检查3] 仓库在线状态"
curl -s -u $AUTH "$NEXUS_URL/service/rest/v1/repositories" | \
  jq -r '.[] | "  \(.name): \(if .online then "在线" else "离线⚠️" end)"'
echo ""

# 4. 检查是否有正在执行的任务
echo "[检查4] 正在执行的任务"
RUNNING_TASKS=$(curl -s -u $AUTH "$NEXUS_URL/service/rest/v1/tasks" | \
  jq -r '.items[] | select(.status == "running") | "  \(.name)"')
if [ -z "$RUNNING_TASKS" ]; then
    echo "  无正在执行的任务 ✅"
else
    echo "$RUNNING_TASKS"
    echo "  ⚠️ 请等待任务完成后再升级"
fi
echo ""

# 5. 检查数据库健康
echo "[检查5] 数据库检查"
echo "  （OrientDB 一致性检查依赖运维手动执行，详见 Wiki）"
echo "  ⚠️ 升级前务必执行: docker compose exec nexus java -jar ... check-database"
echo ""

# 6. 提示备份
echo "========================================="
echo "⚠️  升级前必须完成的步骤："
echo "  1. 执行完整备份: ./nexus-backup.sh"
echo "  2. 保存当前版本的 Docker 镜像 tag"
echo "  3. 阅读 Release Notes: https://help.sonatype.com/"
echo "  4. 在测试环境先做一次升级演练"
echo "  5. 确保有回退计划（旧版本镜像 + 备份数据）"
echo "========================================="
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| Freeze 后备份仍不完整 | 数据库启动失败 | Freeze 后需等 30 秒让所有进行中的事务完成，`curl status` 确保 `frozen: true` 后再备份 |
| 恢复后仓库显示空白 | 组件存在但 Blob 404 | BlobStore 的路径和配置恢复不全，检查 `nexus.properties` 和 BlobStore 配置文件 |
| 权限问题 | 恢复后启动失败 Permission denied | 恢复后 `chown -R 200:200` 整个 `/nexus-data` 目录 |
| 端口冲突 | 恢复演练和原 Nexus 端口冲突 | 恢复演练容器使用不同端口映射（9081:8081） |
| 备份过程中磁盘满 | rsync 失败、备份文件不完整 | 备份前检查目标磁盘空间，确保为 `/nexus-data` 当前大小的 1.5 倍以上 |

## 4. 项目总结

### 4.1 备份策略速查

| 数据类型 | 路径 | 备份频率 | 保留期 | 备份方式 |
|---------|------|---------|--------|---------|
| 数据库 | `/nexus-data/db/` | 每日 | 30 天 | Freeze → rsync |
| 配置 | `/nexus-data/etc/` | 每日 | 30 天 | Freeze → rsync |
| Release Blob | `/nexus-data/blobs/blob-*-release/` | 每周 | 30 天 | rsync |
| Proxy 缓存 | `/nexus-data/blobs/default/` (proxy) | 不备份 | — | 重新拉取 |
| Docker layer | `/nexus-data/blobs/blob-docker/` | 每周 | 30 天 | rsync |
| 日志 | `/nexus-data/log/` | 按需 | 审计要求 | rsync |

### 4.2 适用场景

1. **灾难恢复**：硬件故障、勒索病毒、误删仓库等场景的完整数据恢复
2. **版本回退**：升级失败后回退到旧版本
3. **环境克隆**：从生产备份恢复一个测试环境用于调试
4. **合规审计**：按监管要求保留历史数据快照（如 GDPR"被遗忘权"的删除前快照）
5. **迁移准备**：从单机迁移到 K8s 之前做全量数据备份

**不适用场景**：
1. Cloud 对象存储中的 Blob——S3/Azure Blob 服务自身提供版本控制和跨区域复制
2. 高频实时备份（秒级 RPO）——Nexus 备份方案是分钟到小时级的

### 4.3 注意事项

- **Freeze 是必要条件**：不 Freeze 的备份 = 数据库可能损坏的备份 = 不可恢复的备份
- **备份定期验证**：每季度至少执行一次恢复演练，确保备份文件实际可用
- **升级前必须全量备份**：Nexus 大版本升级（如 3.60 → 3.70）涉及数据库 schema 变更，升级路径不可逆
- **备份文件加密**：备份数据中含制品二进制和用户认证信息，传输和存储时应加密

### 4.4 常见踩坑经验

**故障一：恢复后 admin 密码无效**

团队从备份恢复 Nexus 后，使用备份前的 admin 密码登录失败。根因：备份文件中的数据库确实包含旧密码，但恢复过程触发了 Nexus 的"首次启动检测"，重置了 admin 密码到 `admin.password` 文件。解决：检查 `admin.password` 文件使用随机密码登录，然后改回旧密码。

**故障二：Docker BlobStore 恢复后镜像拉取失败**

恢复后的 Docker 仓库中 tag 列表正确，但 `docker pull` 报 `manifest unknown`。根因：Docker 的 BlobStore 中 `content` 目录下的 `.bytes` 文件被复制时权限不正确（root:root 而非 200:200），导致 Nexus 无法读取。解决：恢复后执行 `chown -R 200:200` 并重启容器。

**故障三：备份脚本在周末假期无人关注时失败**

某个周末停电导致备份失败，但无告警通知。直到两周后需要回滚时才发现上一个有效备份是 18 天前。根因：备份脚本没有错误通知机制。解决：在 cron 中配置 `MAILTO=ops@cloudwhale.com`，脚本失败时 stderr 输出自动发送邮件；或在脚本中添加 Webhook 通知（如企业微信/钉钉）。

### 4.5 思考题

1. 设计一个"跨主机备份"方案：每天凌晨将 Nexus 备份自动上传到远程的 MinIO 对象存储，并保留 7 天的本地副本和 30 天的远程副本。需要考虑网络中断时的重试机制和备份文件的完整性校验。
2. 恢复到新环境后，Docker 客户端仍然使用旧的 `docker login` 凭据连接恢复后的 Nexus——但恢复后的 Nexus 中用户 Token 已经变了。如何设计一个"恢复即用"的客户端重连方案？

（第13章思考题答案：1. Compact 任务和 Cleanup Task 同时执行不会导致数据不一致，但会产生竞争：Cleanup 刚删除的组件，Compact 可能因为启动时间早于 Cleanup 而没有扫描到，导致本次 Compact 回收空间不足。解决：确保两者先后执行——Cleanup 完成至少 1 小时后才启动 Compact。2. '灰度清理'可通过创建多个小范围 Cleanup Task 实现：每天只对 ~10% 的仓库执行清理（第一天清理 maven-snapshots，第二天清理 npm-hosted，etc.），循环执行。每个 Task 手动触发或按单独的 cron 调度，确保不会一次性清理所有仓库。配合 Preview 模式确认每天的清理量是否在预期范围内。）

### 4.6 推广计划提示

- **运维部门**：本章是运维团队的最高优先级任务。必须在 72 小时内完成首次备份 + 恢复演练，并将备份脚本纳入监控
- **开发部门**：了解备份对下载服务的短暂影响（Freeze 期间不可上传），如遇 `503` 错误需查看当前是否正在备份
- **安全部门**：审查备份策略的 RPO/RTO 是否满足业务连续性要求（建议 RPO < 24h，RTO < 4h）
