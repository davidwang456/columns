# 第32章：源码工程结构与启动链路

## 1. 项目背景

云鲸科技的架构组决定参与 Nexus 社区——他们需要为 PyPI 格式写一个 Nexus 插件，同时修复一个 group 仓库在特定顺序下元数据合并的 bug。架构师老周带着两个核心开发开了三天的会，GitHub 上 `sonatype/nexus-public` 仓库的代码也 clone 下来了，但面对 200 多个 Maven 模块、上万个 Java 文件，完全不知道从哪开始看。

"这代码量比我们整个业务系统加起来还大。"老周指着 IDE 里的项目树说道，"我连入口类在哪都找不到。"更困难的是，Nexus 不是一个标准的 Spring Boot 应用——它基于 OSGi/Karaf 容器、用 Sisu 做 DI、用 Guice 管理生命周期、用 Pax Web 处理 HTTP。这些技术栈在云鲸科技内部几乎无人了解。

"我们只是想修一个 group 仓库的元数据合并逻辑，"核心开发小李说，"但改了 `MavenGroupFacet` 之后 Nexus 直接启动不了。日志里只报 `NoClassDefFoundError`，也没有指向具体哪个 bundle 没加载。"

从使用者到源码贡献者，中间隔着的不仅仅是 Java 代码量，更是一整套陌生的架构体系和模块组装方式。本章将作为高级篇的开篇，从源码仓库的模块划分开始，梳理 OSGi/Karaf 容器、Sisu/Guice 依赖注入、组件扫描和插件加载的完整启动过程，建立一套"先入口、再接口、最后实现"的源码阅读方法论。

## 2. 项目设计

老周、小李和大师在会议室里，三台显示器上都是 IntelliJ IDEA 的窗口，各自跳到不同的类。

**老周**："大师，Nexus 源码 200 多个模块，`components`、`plugins`、`assemblies`、`buildsupport`——这四个目录有什么区别？"

**大师**："`components/` 是核心基础设施层——`nexus-core`、`nexus-repository`、`nexus-security`、`nexus-webhooks` 都在这里。`plugins/` 是功能插件层——`nexus-coreui-plugin`（Web UI 后端接口）、`nexus-audit-plugin`（审计实现）、`nexus-repository-maven`（Maven 格式处理器）。`assemblies/` 是打包层——把 components 和 plugins 组装成可发布的 tar.gz/zip。`buildsupport/` 是构建辅助——代码风格检查、license 头生成、Idea 配置模板。"

> **技术映射**：Nexus 源码分层 = components（基础设施库）→ plugins（功能扩展）→ assemblies（发布打包）。这是典型的"内核 + 插件"架构——内核提供接口和抽象，插件提供具体实现。

**小胖**："OSGi 是什么？为什么 Nexus 不用 Spring Boot 而用 Karaf？"

**大师**："Nexus 最初在 2008 年启动，当时 Spring Boot 还不存在。选择 OSGi 是因为它能提供**严格的模块隔离**——每个 bundle（OSGi 中的"模块"）有独立的 ClassLoader，不同 bundle 之间通过 `Export-Package` 和 `Import-Package` 声明公开接口。这带来了两个好处：第一，不会出现 Jar Hell（同一个类的不同版本冲突）；第二，bundle 可以热加载和热卸载——Nexus 的插件系统就依赖这个能力。"

**小李**："那 Nexus 的入口在哪？我找了三天没找到 `main` 方法。"

**大师**："Nexus 不自己写 `main`——它委托给 Apache Karaf 容器。启动链路是这样的——`bin/nexus` 脚本启动 JVM，JVM 加载 Karaf 的 `Main` 类。Karaf 读取 `etc/org.apache.karaf.features.cfg` 中的 features 列表，逐个解析和安装 feature。每个 feature 是一个 feature.xml 文件（在 Nexus 源码中由 `src/main/feature/feature.xml` 定义），描述了这个 feature 包含哪些 bundle、依赖哪些其他 feature。`nexus-core-feature` 是第一个被安装的 feature，它启动了 Sisu/Guice 容器，扫描 `@Named`、`@Inject`、`@Singleton` 注解并完成依赖注入。"

> **技术映射**：Nexus 启动链 = bin/nexus → Karaf Main → featuresBoot → nexus-core-feature → Sisu/Guice 容器 → 组件扫描 → 服务就绪。没有传统的 `main` 方法，入口分散在 Karaf 配置和 feature 声明中。

**小李**："那我要怎么找到'上传一个 jar 到 Maven 仓库'这个功能对应的后端类？从 REST API 端点反查？"

**大师**："三步法。第一步，打开浏览器的开发者工具（F12），在 Nexus Web UI 中执行上传操作，Network 面板中捕获到的请求 URL 就是入口——比如 `POST /service/rest/v1/components?repository=maven-releases`。第二步，在源码中全局搜索这个 URL 片段——通常是 `@Path` 注解中的值，找到对应的 JAX-RS Resource 类。第三步，在这个 Resource 类中跟踪 `@Inject` 注入的 Service 类，一路追到 BlobStore 写入。这就是源码阅读的'反向追踪法'——从 HTTP 端点反推调用链。"

**老周**："那 Nexus 的 DI 是怎么回事？我看到代码里有 `@Named`、`@Inject`、`@Singleton`，还有 `@Named("${nexus.threads.jetty.max:-200}")` 这种带默认值的——这是什么框架？"

**大师**："这是一个混合 DI 方案。核心是 **Sisu**，它是 Google Guice 的一个 JSR-330 扩展——让 Guice 能自动扫描 classpath 中的 `@Named` 组件。`@Inject` 来自 JSR-330（`javax.inject`），`@Singleton` 来自 Guice。`@Named("${...:-default}")` 这种带默认值的语法是 Sisu 的扩展——它允许从 `nexus.properties` 配置文件中读取参数值，如果配置文件中不存在则使用默认值。这个机制让 Nexus 可以在不重新编译的情况下通过配置文件调整参数。"

## 3. 项目实战

### 3.1 环境准备

- JDK 17+
- Git（clone Nexus 源码）
- IntelliJ IDEA（推荐，社区版即可）
- Maven 3.8+

### 3.2 分步实战

#### 步骤一：Clone 源码并导入 IDE

**目标**：获取 Nexus OSS 源码，成功导入 IntelliJ IDEA。

```bash
# Clone Nexus 源码（OSS 版本）
git clone https://github.com/sonatype/nexus-public.git
cd nexus-public

# 切换到最新稳定版 tag（以 3.70.0 为例）
git checkout release-3.70.0-04

# 查看模块结构
find . -maxdepth 2 -name pom.xml | sed 's|./||;s|/pom.xml||' | sort | head -30

# 预期输出（示例）：
# assemblies/nexus-base-template
# buildsupport
# components/nexus-blobstore
# components/nexus-blobstore-api
# components/nexus-core
# components/nexus-repository
# components/nexus-security
# components/nexus-webhooks
# plugins/nexus-audit-plugin
# plugins/nexus-coreui-plugin
# plugins/nexus-repository-maven
# ...

echo "模块总数: $(find . -maxdepth 2 -name pom.xml | wc -l)"
```

**导入 IntelliJ IDEA**：

```
1. File → Open → 选择 nexus-public 目录
2. IDE 自动识别为 Maven 项目，点击 "Load" 加载所有模块
3. 等待 Maven 下载依赖（首次可能 10-15 分钟）
4. 确认 Project Structure 中 JDK 设为 17
```

#### 步骤二：定位启动入口与核心 feature

**目标**：找到 Karaf 容器入口和 `nexus-core-feature`。

```bash
# 1. 查找 Karaf features 启动配置文件（在 assemblies/nexus-base-template 中）
cd nexus-public
find . -name "org.apache.karaf.features.cfg" -path "*/nexus-base-template/*"

# 打开该文件，查看 featuresBoot 列表：
# featuresBoot = \
#   nexus-elasticsearch, \
#   nexus-core-feature, \
#   nexus-coreui-plugin, \
#   nexus-repository-httpbridge, \
#   ...

# 2. 找到 nexus-core-feature 的 feature.xml
find . -name "feature.xml" -path "*nexus-core*" | head -3

# 3. 在 feature.xml 中查看 bundle 列表
# 关键 bundle：nexus-core、nexus-security、nexus-repository 等
# <bundle>mvn:org.sonatype.nexus/nexus-core/${project.version}</bundle>

# 4. 找到 Sisu 容器的启动引导类
# 搜索 "SisuMain" 或 "org.eclipse.sisu" 相关类
grep -r "SisuMain\|ContainerBoot" --include="*.java" components/nexus-core/ | head -5
```

#### 步骤三：绘制模块依赖图

**目标**：理解核心模块之间的依赖关系。

```bash
# 使用 Maven dependency:tree 查看 nexus-core 的依赖
cd nexus-public/components/nexus-core
mvn dependency:tree -Dincludes=org.sonatype.nexus:* | head -40

# 简化版模块依赖关系图（手工绘制）：
cat << 'EOF'
# Nexus 核心模块依赖关系（依赖方向：上依赖下）

assemblies/nexus-base-template (打包入口)
  ├── plugins/nexus-coreui-plugin (Web UI REST 接口)
  │     ├── components/nexus-repository (仓库抽象层)
  │     └── components/nexus-security (认证授权)
  ├── plugins/nexus-repository-maven (Maven 格式)
  │     └── components/nexus-repository
  ├── plugins/nexus-audit-plugin (审计日志)
  │     └── components/nexus-core
  └── components/nexus-core (核心基础设施)
        ├── components/nexus-blobstore (Blob 存储)
        ├── components/nexus-security
        └── components/nexus-webhooks (事件通知)
EOF
```

#### 步骤四：追踪一个 REST 端点从 HTTP 到后端实现

**目标**：用"反向追踪法"找到 UI 上传接口的后端实现类。

```bash
# 问题：Web UI 中上传 jar 到 maven-releases 的 REST 端点是什么？
# 答案：POST /service/rest/v1/components?repository=maven-releases

# 步骤1：在源码中搜索这个 URL 路径
grep -r "service/rest/v1/components" --include="*.java" plugins/nexus-coreui-plugin/ | head -5

# 预期找到：
# plugins/nexus-coreui-plugin/src/main/java/org/sonatype/nexus/coreui/UploadResource.java
# @Path("/v1/components")

# 步骤2：打开 UploadResource.java，找到上传方法
# @POST
# @Consumes(MediaType.MULTIPART_FORM_DATA)
# public ComponentXO upload(...)

# 步骤3：跟踪 upload 方法中注入的 Service
# @Inject
# private UploadManager uploadManager;

# 步骤4：找到 UploadManager 的实现
grep -r "class.*UploadManager" --include="*.java" components/ | head -3
# 预期：components/nexus-repository/src/main/java/.../UploadManagerImpl.java
```

#### 步骤五：理解 Sisu/Guice 的组件注册机制

**目标**：通过日志和配置理解组件如何被扫描和注入。

```bash
# 在 IDE 中搜索关键 DI 注解的使用
# 1. @Named 注解——声明一个组件
grep -r "@Named" --include="*.java" components/nexus-repository/ | head -10
# 示例: @Named("default") （AbstractMavenFormat）

# 2. @Singleton 注解——单例作用域
grep -r "@Singleton" --include="*.java" components/nexus-core/src/main/java/org/sonatype/nexus/internal/ | head -10

# 3. @Inject 注解——依赖注入
# 在 UploadResource 中：
# @Inject
# private UploadManager uploadManager;
# Sisu 自动查找所有实现了 UploadManager 接口且标注了 @Named 的类进行注入

# 4. 启动时查看组件扫描日志
# 在 nexus.log 中搜索 "sisu" 或 "Guice" 关键词
docker compose exec nexus grep -i "sisu\|guice\|binding" /nexus-data/log/nexus.log | head -20
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| Maven 依赖下载失败 | `mvn compile` 时报 `Could not resolve dependencies` | 确认 Maven `settings.xml` 中 mirror 配置正确，部分依赖需从 Sonatype 仓库拉取 |
| IDE 中模块未识别 | IntelliJ 中大量类报红 | File → Project Structure → Modules → 检查 source root 是否正确标记 |
| `feature.xml` 修改不生效 | 改了 feature.xml 但重新打包后 bundle 未加载 | 确认 assemblies 中引用的是修改后的 feature 版本 |
| Karaf 启动报 `BundleException` | bundle 加载失败但日志不明确 | 检查 bundle 的 `MANIFEST.MF` 中 `Import-Package` 声明是否完整 |

## 4. 项目总结

### 4.1 Nexus 源码关键路径

| 你想做什么 | 从哪看起 | 关键类 |
|-----------|---------|--------|
| 理解启动流程 | `assemblies/nexus-base-template/` → `bin/nexus` → Karaf Main | `NexusMain`, `SisuMain` |
| 追踪 REST API | F12 找 URL → `@Path` 搜索 → Resource 类 | `UploadResource`, `RepositoryResource` |
| 理解仓库模型 | `components/nexus-repository/` → Repository, Format, Recipe, Facet | `RepositoryImpl`, `MavenFormat` |
| 理解存储层 | `components/nexus-blobstore/` → BlobStore, Blob | `FileBlobStore`, `S3BlobStore` |
| 理解安全模型 | `components/nexus-security/` → Shiro, Realm | `SecurityModule`, `NexusAuthorizingRealm` |
| 理解事件机制 | `components/nexus-core/` → EventBus, Webhook | `WebhookServiceImpl` |

### 4.2 适用场景

1. **定制插件开发**：需要理解模块加载机制和 DI 容器
2. **Bug 修复贡献**：需要快速定位到相关模块和实现类
3. **架构学习**：通过 Nexus 学习 OSGi/Karaf 在企业级 Java 应用中的实践
4. **性能优化**：理解启动链路中的瓶颈点（如 feature 解析、bundle 加载）
5. **安全审计**：追踪关键操作在源码中的完整调用链

**不适用场景**：
1. 仅日常使用 Nexus 的开发者——不需要读源码
2. 想快速增加一个新格式支持——第 33 章的 Format/Recipe/Facet 模式是更好的入口

### 4.3 注意事项

- **`release-3.70.0-04` 等 tag 间的差异可能较大**：源码阅读和插件开发应固定在一个版本上
- **不要改 framework 层的核心类**：`components/nexus-core` 和 `nexus-security` 中的变更会影响所有插件
- **Karaf shell 是一个强大的调试工具**：`ssh -p 8202 admin@localhost` 连接到 Nexus 的 Karaf 控制台，可以查看 bundle 状态、执行诊断命令

### 4.4 常见踩坑经验

**故障一：IDE 编译通过但打包后 bundle 加载失败**

某开发者在 `nexus-repository-maven` 中新增了一个类，IDE 编译通过，但 `mvn clean package` 后 Nexus 启动报 `ClassNotFoundException`。根因：新增类未被 bundle 的 `Export-Package` 导出。Maven bundle plugin（`maven-bundle-plugin`）的 `Private-Package` 配置导致该类对 OSGi 运行时不可见。解决：将新的 package 路径加入 `Export-Package` 或 `Private-Package` 声明。

**故障二：DI 注入失败返回 null**

在 Resource 类中使用 `@Inject` 注入自定义 Service，运行时 `NullPointerException`。根因：自定义 Service 类缺少 `@Named` 注解，Sisu 在 classpath 扫描时跳过了它——即使类实现了正确的接口。解决：在实现类上添加 `@Named` 注解。

**故障三：修改 nexus.properties 不生效**

修改了 `etc/nexus.properties` 中的线程池参数但重启无效。根因：Docker Compose 的 volume 挂载只映射了 `/nexus-data` 目录，而 `etc/` 在容器启动时从内置路径加载。解决：通过 `INSTALL4J_ADD_VM_PARAMS` 环境变量传入 JVM 参数，或修改镜像内的配置并重新打包。

### 4.5 思考题

1. 在 Nexus 源码中，`@Named` 注解的组件是如何被 Sisu 发现并注册到 Guice 容器的？描述从 `META-INF/sisu/javax.inject.Named` 文件到组件实例化的完整流程。
2. Nexus 支持热部署插件——在不重启整个 Karaf 容器的情况下加载/卸载一个插件。从源码角度分析，热部署依赖 OSGi 的哪些特性？存在哪些限制？

（第31章思考题答案：1. PRO 版升级后的本质性改进：① HA——用多个 Nexus 节点 + 共享 BlobStore（S3）消去单点故障，架构图增加 Load Balancer + 多 Pod；② Repository Replication——跨地域仓库自动复制，省去手工迁移；③ Staging Suite——在 dev→prod 之间增加 staging 审批环节，用 PRO 的 Staging 套件管理而非手工 tag；④ FIPS——合规场景下强制加密和哈希算法。2. 制品治理成熟度模型：L1 初始级——Nexus 仅作为静态文件存储，无规范无监控；L2 可重复级——有基本的仓库类型划分和用户管理，团队各自为政；L3 已定义级——命名规范、权限矩阵、清理策略形成制度文档，部分操作脚本化；L4 已管理级——全面的自动化治理（IaC + CI/CD + 监控告警 + 审计日志），KPI 可量化；L5 优化级——基于数据和故障复盘持续优化治理策略，混沌工程验证，治理平台成为内部产品。云鲸当前对标 L3→L4。）
