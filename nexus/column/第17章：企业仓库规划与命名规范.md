# 第17章：企业仓库规划与命名规范

## 1. 项目背景

云鲸科技从 30 人扩张到 200 人，Nexus 从"天枢平台专用"变成了全公司的制品中枢。Java 组有 5 个子团队，前端有 3 个产品线，DevOps 管理着 8 个项目的 CI/CD，还有新组建的数据中台团队（需要 PyPI 和 Raw 两种格式）。

当前 Nexus 里仓库命名混乱至极：有人建了 `maven-releases`，有人建了 `maven-releases-new`，有一个叫 `maven-hosted-release`，还有一个叫 `maven-release`（少了 s）。group 仓库的成员排列也没人说得清——`maven-public` 里有 `maven-central`、`maven-alibaba`、`maven-jcenter` 三个 proxy，顺序在不同环境中还不一致，导致测试环境下载的版本和生产环境不同。最致命的是一次安全审计时，安全团队要求"列出所有生产环境可用的仓库"——炮哥花了半天才凑齐一份不完整的清单。

**企业级仓库规划的核心不是技术，而是约定**。当只有 3 个仓库时，命名不重要；但当仓库数量增长到 30 个以上时，没有规范的命名体系就意味着无法自动化管理、无法快速定位、无法安全审查。本章将从仓库生命周期（dev→test→stage→prod）出发，建立一套可扩展的仓库架构和命名规范，涵盖命名模式、group 组合策略、多租户隔离边界和制品晋级路径。

## 2. 项目设计

大师在白板上画完第三版仓库矩阵图后，炮哥终于忍不住了。

**炮哥**："大师，你画了三个小时了。这么多仓库，到底怎么命名才不会乱？"

**大师**："建立命名规范的核心原则只有一条——**从名字就能回答三个问题：什么格式？谁在用？什么用途？** 统一的命名模式是 `<format>-<team>-<lifecycle>`。比如 `maven-tianshu-snapshots` 代表'天枢团队的 Maven SNAPSHOT 制品'，`npm-tianshu-hosted` 是'天枢团队的前端私有包'。"

**小胖**："那 `maven-releases` 这种不加团队名的呢？"

**大师**："不加团队名的仓库就是**共享仓库**——所有团队都能用。比如 `maven-releases-shared` 是公司的公共基础组件发布区，`npm-proxy-central` 是所有前端团队共享的 npm 公共源缓存。共享 vs 专用——这是多团队仓库规划的第一个决策点。"

> **技术映射**：命名模式 = `<format>-<scope>-<lifecycle>`。format 决定格式处理器，scope 决定可见性范围（shared/team-xxx），lifecycle 决定制品生命周期状态（dev/test/stage/prod）。

**小白**："那 lifecycle 里的 dev、test、snapshot、release、archive 分别对应什么？每个都需要一个独立仓库吗？"

**大师**："不需要每一个都建仓库。制品生命周期一般分四个阶段：**开发期**（SNAPSHOT，频繁变更）、**候选期**（candidate/RC，待验证）、**发布期**（RELEASE，锁定不变）、**归档期**（archive，保留不动）。对应的仓库策略是——开发期用 snapshot 仓库（`ALLOW` 写策略），候选和发布期用 release 仓库（`ALLOW_ONCE` 写策略），归档期为 release 仓库加只读锁或迁移到专门归档仓库。"

**浩子**："Docker 的生命周期更复杂——我们有 dev 镜像、test 镜像、staging 镜像、prod 镜像，而且每个环境的 tag 策略不一样。"

**大师**："Docker 的仓库策略和 Maven 不同——不建议按环境建多个 hosted 仓库（dev-docker、prod-docker），而是用**tag 命名规范**区分环境。比如 `my-app:1.2.3-dev-abc1234`（开发快照）、`my-app:1.2.3-rc1`（候选）、`my-app:1.2.3`（生产）。同一个 `docker-hosted` 仓库中，由清理策略按 tag 正则分别处理。唯一的例外是生产镜像——可以考虑单独建一个 `docker-prod` 仓库，只有特定角色能上传。"

> **技术映射**：制品晋级 = 开发仓库（ALLOW 覆盖）→ 候选标签 → 生产仓库（ALLOW_ONCE 锁定）。晋级路径的核心是**写策略的收敛**——从可覆盖到不可覆盖。

**炮哥**："那 group 仓库怎么设计？每个团队一个 group 还是全公司一个大 group？"

**大师**："这是多团队策略的核心——**洋葱模型**。最内层是 `maven-public-shared`，聚合所有 shared 仓库（hosted + proxy），所有团队的基础依赖都从这里拉。中间层是 `maven-public-<team>`，聚合 shared 仓库 + 本团队的 hosted 仓库 + proxy。最外层是专用仓库 URL——每个团队还有自己 release 仓库的写入 URL。这样，跨团队依赖只能通过 shared 仓库（经过审核），团队内部依赖走自己的 group，互不干扰。"

### 3.2 分步实战

#### 步骤一：制定企业命名规范文档

**目标**：输出一份所有团队共同遵循的仓库命名规范。

```markdown
# 云鲸科技 Nexus 仓库命名规范 v2.0

## 1. 命名模式
<format>-<scope>-<lifecycle>

## 2. Format 取值
| 值 | 含义 |
|----|------|
| maven  | Maven2 格式 |
| npm    | npm 格式 |
| docker | Docker 格式 |
| raw    | Raw 格式 |
| pypi   | PyPI 格式 |

## 3. Scope 取值
| 值 | 含义 |
|----|------|
| shared | 全公司共享 |
| tianshu | 天枢平台 |
| trade | 交易中台 |
| data | 数据中台 |

## 4. Lifecycle 取值
| 值 | 含义 | 写策略 |
|----|------|--------|
| dev / snapshots | 开发期，可覆盖 | ALLOW |
| releases | 发布期，不可覆盖 | ALLOW_ONCE |
| archive | 归档期，只读 | DENY |
| proxy | 代理远程仓库 | — |
| group / public | 组合入口 | — |

## 5. 完整示例
- maven-shared-releases       → 全公司共享 Maven 正式版
- maven-tianshu-snapshots     → 天枢团队 Maven 快照
- npm-shared-proxy            → 全公司共享 npm 公共源缓存
- docker-tianshu-hosted       → 天枢团队 Docker 仓库
- raw-trade-scripts           → 交易中台脚本仓库
```

#### 步骤二：实现仓库矩阵自动化创建

**目标**：根据命名规范，用脚本创建完整的仓库矩阵。

```bash
#!/bin/bash
# create-repo-matrix.sh：按命名规范创建仓库矩阵
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

declare -A TEAMS=(["shared"]=1 ["tianshu"]=1 ["trade"]=1 ["data"]=1)
declare -A FORMATS=(["maven"]=1 ["npm"]=1 ["docker"]=1 ["raw"]=1)

echo "=== 创建企业仓库矩阵 ==="

# 遍历每个团队的每种格式
for TEAM in "${!TEAMS[@]}"; do
  for FORMAT in "${!FORMATS[@]}"; do
    BLOB="blob-${FORMAT}"

    # 1. Hosted（snapshots/dev + releases）
    if [ "$TEAM" != "shared" ]; then
      # 开发仓库
      SNAP_NAME="${FORMAT}-${TEAM}-snapshots"
      echo "  创建: $SNAP_NAME"
      curl -s -u $AUTH -X POST "$NEXUS/service/rest/v1/repositories/${FORMAT}/hosted" \
        -H "Content-Type: application/json" \
        -d "{\"name\":\"${SNAP_NAME}\",\"online\":true,\"storage\":{\"blobStoreName\":\"${BLOB}\",\"writePolicy\":\"ALLOW\"},\"${FORMAT}\":{\"versionPolicy\":\"SNAPSHOT\",\"layoutPolicy\":\"STRICT\"}}" 2>/dev/null
    fi

    # Release 仓库
    if [ "$TEAM" = "shared" ]; then
      REL_NAME="${FORMAT}-shared-releases"
    else
      REL_NAME="${FORMAT}-${TEAM}-releases"
    fi
    echo "  创建: $REL_NAME"
    curl -s -u $AUTH -X POST "$NEXUS/service/rest/v1/repositories/${FORMAT}/hosted" \
      -H "Content-Type: application/json" \
      -d "{\"name\":\"${REL_NAME}\",\"online\":true,\"storage\":{\"blobStoreName\":\"${BLOB}\",\"writePolicy\":\"ALLOW_ONCE\"},\"${FORMAT}\":{\"versionPolicy\":\"RELEASE\",\"layoutPolicy\":\"STRICT\"}}" 2>/dev/null

    # 2. Proxy（仅 shared 需要）
    if [ "$TEAM" = "shared" ]; then
      PROXY_NAME="${FORMAT}-shared-proxy"
      echo "  创建: $PROXY_NAME"
      curl -s -u $AUTH -X POST "$NEXUS/service/rest/v1/repositories/${FORMAT}/proxy" \
        -H "Content-Type: application/json" \
        -d "{\"name\":\"${PROXY_NAME}\",\"online\":true,\"storage\":{\"blobStoreName\":\"${BLOB}\"},\"proxy\":{\"remoteUrl\":\"https://default-remote\",\"contentMaxAge\":1440,\"metadataMaxAge\":1440},\"httpClient\":{\"blocked\":false,\"autoBlock\":true,\"connection\":{\"retries\":3,\"timeout\":60}},\"negativeCache\":{\"enabled\":true,\"timeToLive\":1440}}" 2>/dev/null
    fi

    # 3. Group（每个团队一个，含 shared Release + 自己 hosted + Proxy）
    GROUP_NAME="${FORMAT}-${TEAM}-public"
    if [ "$TEAM" = "shared" ]; then
      MEMBERS="[\"${FORMAT}-shared-releases\""
    else
      MEMBERS="[\"${FORMAT}-shared-releases\",\"${FORMAT}-${TEAM}-releases\",\"${FORMAT}-${TEAM}-snapshots\""
    fi
    MEMBERS="${MEMBERS},\"${FORMAT}-shared-proxy\"]"

    echo "  创建: $GROUP_NAME (成员: $MEMBERS)"
    curl -s -u $AUTH -X POST "$NEXUS/service/rest/v1/repositories/${FORMAT}/group" \
      -H "Content-Type: application/json" \
      -d "{\"name\":\"${GROUP_NAME}\",\"online\":true,\"storage\":{\"blobStoreName\":\"${BLOB}\"},\"group\":{\"memberNames\":${MEMBERS}}}" 2>/dev/null

  done
done

echo "=== 仓库矩阵创建完成 ==="
echo "各组 group 地址:"
echo "  天枢: http://nexus:8081/repository/maven-tianshu-public/"
echo "  交易: http://nexus:8081/repository/maven-trade-public/"
echo "  数据: http://nexus:8081/repository/maven-data-public/"
echo "  (npm/docker 同理)"
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| 命名不一致 | 同类仓库有 3 种命名，无法编写通用脚本 | 严格遵循 `<format>-<scope>-<lifecycle>` 模式，新仓库命名审批 |
| Group 成员顺序混乱 | 开发环境下载的内部包是 proxy 缓存的旧版本 | 规范化 group 成员顺序：shared-release → team-release → team-snap → proxy |
| Scope 颗粒度过细 | 每个子团队都建了全套仓库，总数爆炸 | "共享优先"原则——先判断是否可以放 shared 仓库，不行再建 team 仓库 |
| 生命周期仓库过多 | dev/test/stage/prod 各有独立仓库，维护负担重 | 用 tag/版本策略而非仓库数区分生命周期 |

## 4. 项目总结

### 4.1 推荐仓库矩阵

| 格式 | Shared 仓库 | Team 仓库 | Group 入口 |
|------|------------|----------|-----------|
| Maven | `maven-shared-releases`, `maven-shared-proxy` | `maven-<team>-snapshots`, `maven-<team>-releases` | `maven-<team>-public` |
| npm | `npm-shared-proxy` | `npm-<team>-hosted` | `npm-<team>-public` |
| Docker | `docker-shared-proxy` | `docker-<team>-hosted` | `docker-<team>-public` |
| Raw | — | `raw-<team>-hosted` | `raw-<team>-public` |

### 4.2 适用场景

1. **多团队共用 Nexus**：需要命名规范来区分团队、隔离权限
2. **多环境制品晋级**：dev→test→stage→prod 需要清晰的生命周期定义
3. **自动化仓库管理**：规范的命名使脚本化批量操作成为可能
4. **安全合规审查**：审计时能快速回答"所有生产仓库有哪些"
5. **新团队扩展**：新团队接入时按模板直接生成仓库矩阵

**不适用场景**：
1. 单团队、10 人以下——仓库总数 < 10 个，命名规范带来的收益小于维护成本
2. 有现成的仓库自动生成系统——直接用系统默认命名即可

### 4.3 注意事项

- **命名即约定，不可中途更改**：一旦规范发布，不允许改名的例外——改名会导致所有客户端配置文件失效
- **Shared 仓库的审批流程**：发布到 shared 仓库应经过 Code Review 和架构评审
- **Proxy 仓库永远放 group 最后**：这是制品的"兜底"来源
- **定期审查废弃仓库**：每季度检查哪些仓库 90 天无活动，标记废弃或删除

### 4.4 思考题

1. 交易中台团队内部有 3 个子团队（订单、支付、结算），它们需要共享一部分基础组件，但也需要各自独立发布业务组件。如何用命名规范 + group 组合在同一个 `trade` scope 下实现？提示：是否需要引入子 scope？
2. 如果公司决定使用 GitLab CI 的 Package Registry 代替 Nexus 管理 npm 私有包，但 Maven 和 Docker 继续用 Nexus。从仓库命名规范的角度，这种混合方案会带来什么问题？如何统一管理？

### 4.5 推广计划提示

- **架构组**：审批命名规范和仓库矩阵，确保与公司技术路线图一致
- **运维部门**：将命名规范固化为自动化脚本，新团队接入时只需执行脚本即可生成全部仓库
- **开发部门**：客户端配置文件中的 URL 必须与仓库命名规范一致，变更时全员同步
