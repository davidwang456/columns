# 第38章：自定义插件与扩展开发实战

## 1. 项目背景

云鲸科技的安全团队在所有内部组件中强制要求引入 SBOM（Software Bill of Materials，软件物料清单）——每个发布到 Nexus 的制品必须附带一个 `sbom.json` 文件，记录该制品的所有直接和传递依赖。但问题来了：Nexus 原生不支持 SBOM 的上传、存储和关联查询。开发团队不得不在 CI 流水线中生成 SBOM 后，用一个独立的 S3 Bucket 存储，再维护一份"制品 → SBOM"的映射表。这种割裂的方案导致每次安全审计时都要在两个系统间来回对比。

"我们不能修改 Nexus 的源码（因为升级时会被覆盖），但能不能写一个 Nexus 插件来实现这个功能？"——架构组的老周提出了这个想法。这正是 Nexus 插件系统（Plugin System）的用武之地：**在不修改 Nexus 核心代码的前提下，通过独立的 bundle 扩展新能力**——新增 REST API 端点、注册定时任务、接入新的事件订阅者、甚至新增一种 Capability 来管理自定义功能。

本章将从零开始，开发一个最小可用的 Nexus 插件——`nexus-sbom-plugin`，实现：提供 REST API 上传 SBOM 文件并与组件关联、注册定时任务统计缺失 SBOM 的组件数量、将统计结果输出到日志。通过这个实战，完整走通"插件工程结构 → 组件注入 → 生命周期管理 → 与 Nexus UI 集成"的全流程。

## 2. 项目设计

老周把插件需求列在白板上，小李小胖围坐观摩大师示范如何创建一个 Nexus 插件。

**老周**："大师，Nexus 插件的工程结构和普通 Maven 项目有什么不同？"

**大师**："一个 Nexus 插件本质上是一个 OSGi bundle——它需要遵循 OSGi 的打包规范。具体来说，`pom.xml` 中的 `<packaging>` 是 `bundle`（而不是 `jar`），需要 `maven-bundle-plugin` 来生成正确的 `MANIFEST.MF`。父 POM 继承自 `nexus-plugins`，这个 POM 提供了所有 Nexus 核心 API 的依赖管理。另外，`feature.xml` 是插件的'安装说明书'——告诉 Karaf 这个 bundle 叫什么名字、依赖哪些其他 feature。"

**小胖**："那插件怎么被 Sisu 发现和注入呢？普通的 `@Named` 注解就行？"

**大师**："对——因为 Nexus 的 Sisu 扫描是 classpath 级别的。只要你的插件 bundle 被 Karaf 加载到 classpath 上，Sisu 就会扫描其中的 `@Named`、`@Inject`、`@Singleton` 注解，自动注册到 Guice 容器。但有一个前提——你的 bundle 的 `MANIFEST.MF` 中必须声明正确的 `Import-Package` 和 `Export-Package`，否则 OSGi 的 ClassLoader 隔离会导致 `ClassNotFoundException`。"

> **技术映射**：Nexus 插件 = OSGi Bundle（MANIFEST.MF）+ Sisu 组件扫描（@Named/@Inject）+ Karaf Feature（feature.xml）。三者协同，插件无需修改 Nexus 源码就能注入新功能。

**小李**："那怎么给插件加一个 REST API 端点？这需要 JAX-RS 吧？但 JAX-RS 的 Resource 类怎么注册到 Jersey？"

**大师**："你的插件中的 JAX-RS Resource 类只需标注 `@Named` + `@Singleton` + `@Path`，Nexus 会自动发现并注册到 Jersey。不需要任何额外的配置——Sisu 的 Jersey 集成会自动收集所有 `@Path` 标注的类。你的 REST API 端点自然就以 `/service/rest/v1/sbom/...` 的路径暴露出来了。"

**老周**："定时任务呢？怎么新增一个 Task 类型？"

**大师**："实现 `TaskSupport` 抽象类，重写 `execute()` 方法。然后在 `feature.xml` 中注册一个 TaskDescriptor——它告诉 Nexus 的 Task 管理系统'有一个新的任务类型，它的名称是 xxx，需要什么参数'。用户的 UI 中就会自动出现这个新的任务类型。"

> **技术映射**：插件能力矩阵 = JAX-RS Resource（REST API）+ TaskSupport（定时任务）+ @Subscribe Event Listener（事件处理）+ Capability（功能开关管理）。

**老周**："测试呢？怎么在不需要启动完整 Nexus 的情况下测试插件？"

**大师**："Nexus 提供了 `nexus-testbase` 依赖——它包含了 Pax Exam（OSGi 集成测试框架），可以在测试中启动一个完整的 Nexus 容器来运行集成测试。但开发阶段的快速验证可以用单元测试 + Mock——只需要 mock `BlobStore` 和 `RepositoryManager` 这两个核心接口。"

#### 步骤二：实现 SBOM 上传 REST API

**目标**：创建 `SbomResource` 类，提供 SBOM 文件上传和查询 API。

```java
// SbomResource.java — REST API 端点
package com.cloudwhale.nexus.sbom.rest;

import com.cloudwhale.nexus.sbom.internal.SbomStore;
import javax.inject.Inject;
import javax.inject.Named;
import javax.inject.Singleton;
import javax.ws.rs.*;
import javax.ws.rs.core.MediaType;
import javax.ws.rs.core.Response;
import java.io.InputStream;

@Named                    // Sisu 组件扫描
@Singleton               // 单例
@Path("/v1/sbom")        // REST API 路径前缀
public class SbomResource {

    @Inject
    private SbomStore sbomStore;  // SBOM 存储服务

    /**
     * 上传 SBOM 文件并与组件关联
     * POST /service/rest/v1/sbom/upload
     */
    @POST
    @Path("/upload")
    @Consumes(MediaType.MULTIPART_FORM_DATA)
    @Produces(MediaType.APPLICATION_JSON)
    public Response uploadSbom(
        @QueryParam("repository") String repositoryName,
        @QueryParam("componentId") String componentId,
        @FormDataParam("sbom") InputStream sbomStream) {

        try {
            // 1. 验证组件存在
            if (!sbomStore.componentExists(componentId)) {
                return Response.status(404)
                    .entity("{\"error\":\"Component not found: " + componentId + "\"}")
                    .build();
            }

            // 2. 存储 SBOM 文件到 BlobStore
            String sbomId = sbomStore.store(repositoryName, componentId, sbomStream);

            // 3. 返回成功
            return Response.status(201)
                .entity("{\"sbomId\":\"" + sbomId + "\", \"componentId\":\"" + componentId + "\"}")
                .build();

        } catch (Exception e) {
            return Response.serverError()
                .entity("{\"error\":\"" + e.getMessage() + "\"}")
                .build();
        }
    }

    /**
     * 查询组件是否有关联 SBOM
     * GET /service/rest/v1/sbom/{componentId}
     */
    @GET
    @Path("/{componentId}")
    @Produces(MediaType.APPLICATION_JSON)
    public Response getSbomStatus(@PathParam("componentId") String componentId) {
        boolean hasSbom = sbomStore.hasSbom(componentId);
        return Response.ok("{\"componentId\":\"" + componentId + "\", \"hasSbom\":" + hasSbom + "}").build();
    }
}
```

**SbomStore 实现（核心存储逻辑）**：

```java

import org.sonatype.nexus.blobstore.api.BlobStore;

@Named
@Singleton
public class SbomStore {

    @Inject
    private BlobStoreManager blobStoreManager;

    @Inject
    private RepositoryManager repositoryManager;

    /**
     * 存储 SBOM 文件到指定 BlobStore
     */
    public String store(String repositoryName, String componentId, InputStream sbomStream) {
        BlobStore blobStore = blobStoreManager.get("blob-sbom");  // 专用 BlobStore
        // BlobStore 写入逻辑...
        return "sbom-" + componentId;  // 简化返回
    }

    public boolean componentExists(String componentId) {
        // 检查组件是否存在
        return true;  // 简化
    }

    public boolean hasSbom(String componentId) {
        // 检查是否已有 SBOM
        return false;  // 简化
    }
}
```

#### 步骤三：实现定时任务——SBOM 缺失统计

**目标**：创建 `SbomAuditTask`，定时统计缺少 SBOM 的组件。

```java
// SbomAuditTask.java — 定时任务
package com.cloudwhale.nexus.sbom.task;

import com.cloudwhale.nexus.sbom.internal.SbomStore;
import org.sonatype.nexus.scheduling.TaskSupport;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import javax.inject.Inject;
import javax.inject.Named;

@Named
public class SbomAuditTask extends TaskSupport {

    private static final Logger log = LoggerFactory.getLogger(SbomAuditTask.class);

    @Inject
    private SbomStore sbomStore;

    @Override
    protected String execute() throws Exception {
        // 获取仓库范围参数（从 task properties 中读取）
        String repositoryName = getConfiguration().getString("repositoryName", "*");

        log.info("=== SBOM 审计任务开始 ===");
        log.info("范围: repository={}", repositoryName);

        // 扫描指定仓库中的所有组件
        int totalComponents = 0;
        int missingSbom = 0;

        // 分页遍历组件（简化示意）
        // ComponentSearch search = repositoryManager.search(...);
        // while (search.hasNext()) { check sbomStore.hasSbom(component.getId()); }
        totalComponents = 150;   // 模拟
        missingSbom = 23;        // 模拟

        double coverage = ((double)(totalComponents - missingSbom) / totalComponents) * 100;
        String result = String.format(
            "SBOM 覆盖率: %.1f%% (%d/%d 个组件缺少 SBOM)",
            coverage, missingSbom, totalComponents
        );

        log.info(result);
        log.info("=== SBOM 审计任务完成 ===");

        return result;
    }

    @Override
    protected String getMessage() {
        return "统计 SBOM 覆盖率";
    }
}
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| `ClassNotFoundException` | Nexus 启动时 bundle 加载失败 | 检查 `Import-Package` 是否声明了所有依赖的 package |
| REST 端点 404 | `/service/rest/v1/sbom` 返回 404 | 确保 Resource 类标注了 `@Named` + `@Singleton` + `@Path` |
| DI 注入失败（NPE） | `@Inject` 的字段为 null | 确保被注入的类也被 `@Named` 标注并在同一 bundle 或前置 bundle 中 |
| 插件 bundle 升级后旧 bundle 未卸载 | 新旧两个版本同时存在 | 先 `feature:uninstall` 旧版再 `feature:install` 新版 |

## 4. 项目总结

### 4.1 插件能力矩阵

| 能力类型 | 实现方式 | 关键注解/类 | 用户可见入口 |
|---------|---------|-----------|------------|
| REST API | JAX-RS Resource | `@Path` + `@Named` | Web UI / curl |
| 定时任务 | TaskSupport 子类 | `@Named` | System → Tasks |
| 事件监听 | EventBus Subscriber | `@Subscribe` | 后台静默 |
| Capability（功能开关） | Capability 接口实现 | `@Named` | System → Capabilities |
| BlobStore 扩展 | BlobStore 实现类 | `@Named` | Repository → Blob Stores |

### 4.2 适用场景

1. **企业内部合规定制**：SBOM、许可证扫描、自定义文件格式校验
2. **运维自动化扩展**：自定义巡检任务、容量预测、自动清理规则
3. **多系统集成**：自定义 Webhook 接收端、外部事件转发
4. **UI 扩展**：在 Nexus 控制台增加自定义页面（React 组件 + REST API）
5. **A/B 测试新功能**：通过插件灰度发布新能力，不影响核心版本升级

**不适用场景**：
1. 修改 Nexus 的核心行为（如 group 元数据合并算法）——应提交 PR 到上游
2. 简单的运维脚本（如定时备份）——用 Groovy Script Task 即可

### 4.3 注意事项

- **插件版本必须与 Nexus 版本匹配**：`nexus-plugins` 父 POM 的版本需要与目标 Nexus 实例一致
- **依赖 scope 必须为 `provided`**：Nexus 运行时提供所有核心 API，打包时不能包含它们
- **`MANIFEST.MF` 是 OSGi 的生命线**：错误的 `Import-Package` 导致 bundle 安装成功但类不可见
- **插件升级是卸载→安装**：没有热升级机制，需要重启 Nexus

### 4.4 思考题

1. 如果需要在插件中增加一个 UI 页面（在 Nexus 左侧导航栏增加一个 "SBOM" 菜单项），需要修改哪些文件？前端 React 组件如何与后端 REST API 交互？
2. 插件中的定时任务如果执行时间过长（如扫描 10 万个组件的 SBOM 覆盖情况），会不会阻塞 Nexus 的其他操作？Nexus 的任务调度框架是如何处理任务并发的？

（第37章思考题答案：1. API 权限覆盖率审计方案：① 用 JAX-RS 的 `Application` 扫描机制或 classpath 扫描找出所有 `@Path` 注解的类；② 对每个类中的每个 `@GET`/`@POST`/`@PUT`/`@DELETE` 方法，检查是否有 `@RequiresPermissions`、`@RequiresAuthentication`、或方法体内调用 `SecurityHelper.isPermitted()`；③ 没有保护的方法记为"遗漏"，输出覆盖率报告（受保护数 / 总端点数）。方案可以用 Java 反射 + 自定义 Maven 插件实现，或集成到 SonarQube 自定义规则中。2. Nexus 的权限叠加规则是**任意匹配即放行（OR 逻辑）**。源码中 `NexusAuthorizingRealm.isPermitted()` 遍历用户所有角色包含的所有 Privilege，只要其中**任何一个**匹配当前请求所需的权限，就返回 true。不存在"负面权限"（DENY）——不能通过添加一个"禁止操作"的权限来覆盖已有的"允许操作"。这意味着如果一个用户同时拥有 `role-readonly` 和 `role-full` 两个角色，最终结果是拥有 `role-full` 的全部权限。这是基于"最小权限原则"的设计——防止管理员疏忽赋予的权限被意外限制。）
