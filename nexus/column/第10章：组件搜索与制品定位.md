# 第10章：组件搜索与制品定位

## 1. 项目背景

某周三下午，云鲸科技的线上监控突然告警：订单服务的响应时间从 50ms 飙到了 2.3 秒。老王排查后发现，最近一次发版时 `common-data:1.5.2` 依赖了一个新版本的 JDBC 驱动，该驱动在特定 SQL 模式下不走索引。修复方案很简单：回退到 `common-data:1.5.1`。然而——

"1.5.1 还在 Nexus 上吗？"、"谁发布的 1.5.2？有没有经过代码 review？"、"1.5.1 依赖了哪些传递依赖？它们的版本分别是什么？"——老王在 Nexus Web UI 的搜索框里输入 `common-data`，返回了 47 个结果，分布在 `maven-releases`、`maven-snapshots`、`maven-central-proxy` 三个仓库中。他想按版本号筛选、按上传时间排序、看每个版本有哪些具体文件（jar、pom、sources），却发现 Web UI 的搜索功能远不如他期望灵活。

更令测试组困扰的是：他们每周要编译一份"各仓库中所有 SNAPSHOT 组件清单"用于版本审计，但手动在 Web UI 里翻页、复制、粘贴，两个小时才能搞完一个仓库。"包到底在哪？谁上传的？能不能删？还依赖了哪些东西？"——这四个问题是制品仓库运维中最频繁的日常问询，而它们的答案都指向同一个能力：**制品搜索与定位**。

本章将从 Component 与 Asset 的本质区别出发，详细拆解 Nexus 的搜索 API、Web UI 搜索、Browse 页面的工作机制及适用场景，并建立一份"依赖找不到"的标准化排障 SOP——让你从"满仓库找包"升级为"5 分钟定位根因"。

## 2. 项目设计

老王和测试组长阿玲把 Nexus 搜索页面投屏，两人已经跟"找不到的包"较劲了一下午。

**老王**："大师，我在搜索框输入 `common-data 1.5.1`，为什么搜出来一堆乱七八糟的结果？有些是 1.5.1 之前的版本，有些根本不是 common-data。"

**大师**："因为 Web UI 的搜索框默认做的是**关键词全文匹配**，不是坐标精确查询。你输入 `common-data 1.5.1`，它会把 `common-data` 和 `1.5.1` 当成两个独立的关键词去匹配组件名、组名、版本号和路径——所以只要版本号里包含 `1.5.1` 的都会出来，比如 `1.5.10`、`1.5.11`。"

> **技术映射**：Nexus Web UI 搜索默认用 `keyword` 参数做模糊匹配，等价于 API 的 `search?name=keyword`。精确查询需使用 `search?name=common-data&version=1.5.1`。

**小胖**："那 Component 和 Asset 到底什么区别？我经常在 Browse 页面看到一个 jar 下面还有 pom、sources.jar——它们算一个东西还是多个？"

**大师**："Component 是一个逻辑实体——用坐标唯一标识。比如 `com.cloudwhale:common-data:1.5.1`。Asset 是这个 Component 下面的每个具体二进制文件——`common-data-1.5.1.jar`、`common-data-1.5.1.pom`、`common-data-1.5.1-sources.jar`。打比方：Component 是一个**人员档案**（姓名+身份证号），Asset 是这个人的**每张证件照**（一寸照、两寸照、全身照）。删 Component = 删掉整个人员档案和全部证件照；删 Asset = 只删一张证件照。"

**小白**："那 Browse 页面里的目录树结构呢？Browse 显示的是文件系统路径，search 显示的是坐标。对 Maven 包来说，Browse 里的路径 `/com/cloudwhale/common-data/1.5.1/common-data-1.5.1.jar` 和坐标 `com.cloudwhale:common-data:1.5.1` 是一一对应的吧？"

**大师**："对 Maven 来说是——Maven 的坐标天然映射为目录路径。但 npm 就不一样了：`@cloudwhale/utils@1.0.0` 在 Browse 中的路径是 `@cloudwhale/utils/-/utils-1.0.0.tgz`，而不是 `/cloudwhale/utils/1.0.0/`。Docker 更特殊——它的 Browse 视图下是 `v2/<name>/blobs/` 和 `v2/<name>/manifests/` 这种 registry 协议路径。所以，**Browse 适合按格式理解目录结构，Search 适合跨格式按坐标查询**。"

> **技术映射**：Browse = 仓库内的物理路径视图（格式相关），Search = 跨仓库的逻辑坐标视图（格式无关）。

**老王**："我现在需要查：'谁在什么时候把 `common-data:1.5.2` 上传到 `maven-releases` 的？' 搜索结果里看不到上传者信息啊？"

**大师**："上传者信息不在搜索结果中，在审计日志里。Nexus 的搜索 API 只管'有什么'，审计日志管'谁干的、什么时候干的'。两者的关联是`组件ID`——你从搜索结果中拿到 component ID，再到审计日志中按 `componentId` 查询对应的事件。"

**阿玲**："那如果我需要查一个 jar 的直接依赖和传递依赖呢？比如我想知道 `common-data:1.5.1` 依赖了哪些第三方包，万一其中一个有安全漏洞。"

**大师**："Nexus 不解析依赖关系——它只是一个仓库，不是依赖分析器。解析依赖是 Maven/Gradle/npm 这类构建工具的工作。但你可以通过 pom.xml 文件间接查看——Maven 的每个发布包里都有 `xxx.pom` 文件，pom 中声明了直接依赖。对于传递依赖，需要借助 `mvn dependency:tree`，或者企业级的依赖分析工具（如 OWASP Dependency-Check、Snyk）。Nexus 的价值在于：你能通过搜索快速定位到 pom 文件，下载后查看声明依赖。"

**小胖**："那如果我压根就不确定包在哪个仓库里呢？比如只知道大概叫 `common-data`，但不知道在 maven-releases 还是 maven-snapshots。"

**大师**："搜索 API 支持不带 `repository` 参数的全局搜索——它会查所有仓库。但注意两点：第一，全局搜索更慢；第二，结果中会标明每个组件来自哪个仓库。如果你确定知道是 Maven 格式，加 `format=maven2` 参数可以加快搜索。"

## 3. 项目实战

### 3.1 环境准备

- 已部署 Nexus 实例，有已填充的 Maven、npm、Docker 仓库
- curl、jq
- 第 9 章的 `nexus-api-lib.sh`（可选但推荐）

### 3.2 分步实战

#### 步骤一：掌握 Component 与 Asset 的差异

**目标**：通过 API 深入理解 Component 和 Asset 的区别。

```bash
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

# 1. 搜索一个特定组件（Component 视角）
COMPS=$(curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/search?repository=maven-releases")

# 提取第一个组件的 ID
COMP_ID=$(echo "$COMPS" | jq -r '.items[0].id')
COMP_NAME=$(echo "$COMPS" | jq -r '.items[0].name')
COMP_VERSION=$(echo "$COMPS" | jq -r '.items[0].version')
COMP_FORMAT=$(echo "$COMPS" | jq -r '.items[0].format')

echo "=== Component 信息 ==="
echo "ID:      $COMP_ID"
echo "名称:    $COMP_NAME"
echo "版本:    $COMP_VERSION"
echo "格式:    $COMP_FORMAT"

# 2. 查看这个组件的所有 Assets
echo ""
echo "=== 该 Component 的 Assets ==="
curl -s -u $AUTH "$NEXUS/service/rest/v1/components/$COMP_ID" | \
  jq '.assets[] | {id, path, contentType, fileSize: (.fileSize / 1024 | floor | tostring + " KB")}'

# 预期输出（示例）：
# {"id": "...", "path": "com/cloudwhale/common-data/1.5.1/common-data-1.5.1.jar", "contentType": "application/java-archive", "fileSize": "245 KB"}
# {"id": "...", "path": "com/cloudwhale/common-data/1.5.1/common-data-1.5.1.pom", "contentType": "application/xml", "fileSize": "2 KB"}
# {"id": "...", "path": "com/cloudwhale/common-data/1.5.1/common-data-1.5.1-sources.jar", "contentType": "application/java-archive", "fileSize": "180 KB"}
```

**运行结果**：一个 Maven Component 下面通常有 jar、pom 两个 Asset（如果配置了 source plugin，还有 sources.jar 和 javadoc.jar）。

#### 步骤二：利用搜索 API 做精确坐标查询

**目标**：比较关键词搜索和精确坐标搜索的差异。

```bash
# ---- 对比 1：关键词搜索（模糊匹配） ----
echo "=== 关键词搜索 'common-data' ==="
curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/search?name=common-data" | \
  jq '{total: (.items | length), items: [.items[].name]}'

# 预期输出：可能返回多个包含 "common-data" 的组件

# ---- 对比 2：精确坐标搜索 ----
echo ""
echo "=== 精确搜索 group + name + version ==="
curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/search?group=com.cloudwhale&name=common-data&version=1.5.1" | \
  jq '{total: (.items | length), component: .items[0]}'

# 预期输出：基本只有一条精确匹配结果

# ---- 对比 3：按仓库过滤 ----
echo ""
echo "=== 限定仓库搜索 ==="
curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/search?repository=maven-releases&name=common-data" | \
  jq '.items[] | {name, version, repository: "maven-releases"}'

# ---- 对比 4：跨仓库全局搜索 ----
echo ""
echo "=== 全局搜索（不指定仓库） ==="
curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/search?name=common-data" | \
  jq '.items[] | {name, version, repository}'
```

**运行结果**：展示了四种搜索模式的差异。精确搜索最快最准，全局搜索最全面但最慢。

#### 步骤三：构建制品信息查询工具

**目标**：封装一个"制品查询"命令，一次性输出制品的完整画像。

```bash
#!/bin/bash
# artifact-info.sh：给定 Maven 坐标，输出完整制品信息
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

GROUP="$1"
ARTIFACT="$2"
VERSION="$3"

if [ -z "$GROUP" ] || [ -z "$ARTIFACT" ]; then
    echo "用法: $0 <groupId> <artifactId> [version]"
    echo "示例: $0 com.cloudwhale common-data 1.5.1"
    exit 1
fi

echo "=== 制品画像: $GROUP:$ARTIFACT${VERSION:+:$VERSION} ==="

# 构造查询 URL（精确匹配）
URL="$NEXUS/service/rest/v1/search?group=$GROUP&name=$ARTIFACT"
[ -n "$VERSION" ] && URL="$URL&version=$VERSION"

RESULT=$(curl -s -u $AUTH "$URL")
TOTAL=$(echo "$RESULT" | jq '.items | length')

if [ "$TOTAL" = "0" ]; then
    echo "❌ 未找到匹配组件"
    exit 1
fi

echo "找到 $TOTAL 个匹配组件"
echo ""

echo "$RESULT" | jq -r '.items[] | "
📦 组件
   ID:          \(.id)
   坐标:        \(.group):\(.name):\(.version)
   格式:        \(.format)
   仓库:        \(.repository)
   资产数量:    \(.assets | length)

   📎 资产列表:
   \(.assets[] | "   - \(.path) (\(.contentType // "N/A"))")
---
"'

# 额外：如果找到单个组件，显示 checksum 信息
if [ "$TOTAL" = "1" ]; then
    COMP_ID=$(echo "$RESULT" | jq -r '.items[0].id')
    echo "🔒 校验信息:"
    curl -s -u $AUTH "$NEXUS/service/rest/v1/components/$COMP_ID" | \
      jq '.assets[] | {path, sha1: .checksum.sha1, sha256: .checksum.sha256, md5: .checksum.md5}'
fi
```

```bash
chmod +x artifact-info.sh
./artifact-info.sh com.cloudwhale common-data 1.5.1
# 或不带版本号查看所有版本
./artifact-info.sh com.cloudwhale common-data
```

#### 步骤四：透视 Browse 页面 vs Search API

**目标**：学会在两种视图间切换，找到制品在 BlobStore 中的物理位置。

```bash
# Browse 视图等价于直接访问仓库路径
# 示例：在 Nexus Web UI 中 Browse maven-releases
# 路径: com/cloudwhale/common-data/1.5.1/

# 等效的 curl 请求（Browse 本质是 GET 仓库路径）：
curl -u $AUTH http://localhost:8081/repository/maven-releases/com/cloudwhale/common-data/1.5.1/

# 预期输出：返回目录下文件列表（HTML 页面或 JSON，取决于 Accept 头）

# 而搜索 API 返回的是有结构的 Component/Asset 对象：
curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/search?repository=maven-releases&group=com.cloudwhale&name=common-data&version=1.5.1" | \
  jq '.items[0].assets[].path'

# 两个路径应该一致
```

#### 步骤五：编写"依赖找不到"标准化排障 SOP

**目标**：形成一套快速定位制品问题的排查步骤。

```bash
#!/bin/bash
# troubleshoot-missing-artifact.sh：依赖找不到时的排障工具
NEXUS="http://localhost:8081"
AUTH="admin:admin123"

GROUP="$1"
ARTIFACT="$2"
VERSION="${3:-}"
FORMAT="${4:-maven2}"

echo "=== Nexus 制品排障工具 ==="
echo "目标: $GROUP:$ARTIFACT${VERSION:+:$VERSION} (format: $FORMAT)"
echo ""

# Step 1：全局搜索确认组件是否存在
echo "[Step 1] 全局搜索组件..."
RESULT=$(curl -s -u $AUTH \
  "$NEXUS/service/rest/v1/search?name=$ARTIFACT&group=$GROUP&format=$FORMAT")
COUNT=$(echo "$RESULT" | jq '.items | length')

if [ "$COUNT" = "0" ]; then
    echo "  ❌ 未找到任何匹配组件"
    echo "  → 检查: 组件是否已上传？坐标是否拼写正确？仓库类型是否正确？"
    echo ""

    # 尝试模糊搜索
    echo "  [模糊搜索 '$ARTIFACT']"
    FUZZY=$(curl -s -u $AUTH "$NEXUS/service/rest/v1/search?name=$ARTIFACT&format=$FORMAT")
    FUZZY_COUNT=$(echo "$FUZZY" | jq '.items | length')
    echo "  模糊匹配到 $FUZZY_COUNT 个结果:"
    echo "$FUZZY" | jq -r '.items[] | "    \(.group):\(.name):\(.version)  [仓库: \(.repository)]"' | head -20
    exit 1
fi

echo "  ✅ 找到 $COUNT 个匹配组件"

# Step 2：列出所有匹配版本及其仓库
echo ""
echo "[Step 2] 版本与仓库分布:"
echo "$RESULT" | jq -r '.items[] | "  \(.version) (仓库: \(.repository), ID: \(.id))"'

# Step 3：检查组件是否在线（未被删除）
echo ""
echo "[Step 3] 组件状态检查:"
echo "$RESULT" | jq -r '.items[] | "  \(.version) → 资产数: \(.assets | length), 仓库在线: \(.repository)"'

# Step 4：如果指定了 version，检查该版本是否可下载
if [ -n "$VERSION" ]; then
    echo ""
    echo "[Step 4] 下载链接验证:"
    DOWNLOAD_URLS=$(echo "$RESULT" | jq -r ".items[] | select(.version == \"$VERSION\") | .assets[].downloadUrl")
    if [ -z "$DOWNLOAD_URLS" ]; then
        echo "  ❌ 版本 $VERSION 没有可下载的 asset"
    else
        echo "$DOWNLOAD_URLS" | while read -r URL; do
            HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" -u $AUTH "$URL")
            FILENAME=$(basename "$URL")
            echo "  [$HTTP_CODE] $FILENAME"
        done
    fi
fi

# Step 5：代理仓库特别检查
echo ""
echo "[Step 5] 代理仓库缓存检查:"
echo "$RESULT" | jq -r '.items[0].repository as $repo | if $repo | test("proxy") then "  ⚠️  组件来自 proxy 仓库，检查远程连接状态和缓存策略" else "  ✅ 组件来自 hosted 仓库" end'

# Step 6：审计日志查询（最近相关操作）
echo ""
echo "[Step 6] 检查仓库是否在线:"
REPO_NAME=$(echo "$RESULT" | jq -r '.items[0].repository')
curl -s -u $AUTH "$NEXUS/service/rest/v1/repositories/$REPO_NAME" | jq '{name, online, type, format}'

echo ""
echo "=== 排障完成 ==="
```

```bash
chmod +x troubleshoot-missing-artifact.sh

# 测试：模拟排障一个不存在的包
./troubleshoot-missing-artifact.sh com.cloudwhale common-data 9.9.9

# 测试：排障一个存在的包
./troubleshoot-missing-artifact.sh com.cloudwhale common-data 1.5.1
```

### 3.3 常见坑点

| 坑点 | 现象 | 解决方法 |
|------|------|----------|
| 搜索无结果但 Browse 能看到 | `search?name=xxx` 返回 0，但 Browse 中有文件 | 检查 `name` 参数是否与 Component 的 `name` 字段（而非文件名）一致；npm 包的 name 含 `@scope/` |
| version 参数无效 | `search?version=1.0.0` 返回所有版本 | API 对 version 参数的匹配更严格，需与组件存储的 version 完全一致；Raw 仓库无 version 概念 |
| Docker 搜索没有 name | Docker 组件在搜索结果中 name 为 `null` | Docker 制品的 `name` 字段可能是 null，需用 Browse 或 v2 API 查询 |
| 分页遗漏结果 | 搜索大量结果时只拿到前 50 条 | 使用 `continuationToken` 翻页，每次请求响应中检查是否有下一页 token |
| 刚上传搜不到 | 上传后 3 秒搜索无结果 | 索引更新存在 1~5 秒延迟，等待后重试；大批量上传后建议执行 `Rebuild repository index` 任务 |

## 4. 项目总结

### 4.1 搜索功能矩阵

| 查找方式 | 适用场景 | 精确度 | 速度 | 是否需要认证 |
|---------|---------|--------|------|-------------|
| Web UI 搜索框 | 快速浏览、模糊查找 | 低 | 快 | 是 |
| API `search?name=` | 按名称模糊匹配 | 中 | 快 | 是 |
| API `search?group=&name=&version=` | 精确坐标匹配 | 高 | 最快 | 是 |
| Browse 页面 | 浏览仓库目录结构、查看文件元数据 | 高 | 中 | 是 |
| API `GET /repository/<repo>/<path>` | 按路径直接下载或列表 | 最高 | 最快 | 是 |
| 审计日志查询 | 查找"谁做了什么操作" | 高 | 慢 | 是 (admin) |

### 4.2 适用场景

1. **日常开发："包在哪？"** — 用精确坐标搜索快速定位，然后下载或引用
2. **依赖审计："生产依赖了哪些版本？"** — 全局搜索 + 版本汇总，生成依赖清单
3. **故障排查："这个 jar 谁传的？"** — 搜索定位 component → 获取 ID → 审计日志反查
4. **安全扫描："有哪些有漏洞的版本？"** — 搜索指定 artifact → 版本列表 → 与 CVE 清单交叉比对
5. **制品迁移："maven-releases 里有多少个组件？"** — 分页遍历搜索 → 统计 → 批量迁移

**不适用场景**：
1. 依赖关系分析（传递依赖、循环依赖）—— 应使用 Maven/Gradle 插件
2. 代码内容搜索（如"哪个 jar 里有一段特定代码"）—— 需反编译工具

### 4.3 注意事项

- **搜索结果是 Component 而非 Asset**：搜索结果列表中的每一项是 Component，不是文件。下载安装需进一步获取 Asset
- **npm scope 包的搜索名称**：`@cloudwhale/utils` 在搜索 API 中的 `name` 参数应传 `@cloudwhale/utils`（保持 scope 前缀）
- **Docker 制品的 version 映射**：Docker tag 映射为 Nexus Component 的 `version` 字段
- **Browse 不反映权限全貌**：在 Browse 页面能看到的目录结构不代表当前用户有权下载其中的所有文件

### 4.4 常见踩坑经验

**故障一：搜索 Web UI 正常但 API 返回空**

某用户的脚本用 `search?name=common-data` 查不到结果，但 Web UI 能搜到。根因 URL 中空格未编码——`common data` 被拆成两个参数。解决：URL 中空格用 `%20` 或 `+` 编码。

**故障二：误删了 Component 而只删 Asset**

前端开发想删除 `@cloudwhale/utils:0.9-beta` 的 tarball 但保留包信息。他通过 Browse 页面删除了 `.tgz` 文件（Asset），却发现整个包（Component）也从搜索结果中消失了。根因：Nexus 在最后一个 Asset 被删除时会级联删除 Component。解决：如果想"用新包替换旧包"但不改版本号，应重新 `npm publish` 覆盖。

**故障三：`continuationToken` 在脚本中被截断**

Token 通过环境变量传递时被 shell 截断了末尾的 `=` 号。解决：令牌用 `jq -r` 读出后直接在同一个管道中使用，不要存入变量后再展开。如必须存变量，用 `printf '%s' "$TOKEN"` 而非 `echo "$TOKEN"`。

### 4.5 思考题

1. 如果一个 Maven 组件 `com.cloudwhale:common-data:1.5.1` 在 `maven-releases`（hosted）和 `maven-central-proxy`（proxy、缓存了同坐标的公共版本）中同时存在，通过 `search?name=common-data` 会返回几个结果？每个结果的 `repository` 字段分别是什么？
2. 生产环境中，你需要在 5 分钟内定位到"哪个服务使用了存在 CVE-2025-XXXXX 漏洞的 `logback-core:1.4.x`"。设计一套搜索 + 关联方案，从 Nexus 搜索出发，最终输出受影响的服务清单。

（第9章思考题答案：1. 迁移脚本的核心挑战是幂等性——目标仓库可能已有同名组件（不允许覆盖），需要先检查再跳过；网络中断时重复请求可能导致同一组件被上传多次（虽然 RELEASE 仓库的 ALLOW_ONCE 策略会拒绝重复，但 SNAPSHOT 允许多次上传会产生重复时间戳版本）。方案：迁移前先用 search API 建立"源仓库组件清单"和"目标仓库已有组件清单"，只迁移缺失的条目；上传前用 HEAD 请求检查目标路径是否存在；每个组件迁移后记录状态到日志文件，断点续传。2. 凌晨 3:00-3:05 返回空结果，最可能的原因是 Nexus 在执行定时任务（如 Compact BlobStore、Cleanup Policy 执行），这些任务会短暂锁定索引或暂停部分服务。排查方法：查看 Nexus Tasks 页面，确认凌晨的定时任务排期；查看 search API 的响应是否返回 HTTP 503（服务不可用）而非 200 空结果；在函数中添加重试逻辑（间隔 30s 重试 3 次）。）
