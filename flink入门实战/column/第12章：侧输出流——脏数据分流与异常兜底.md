# 第12章：侧输出流——脏数据分流与异常兜底

---

## 1. 项目背景

某支付公司用Flink处理交易流水。每天处理约500万笔交易，其中约0.3%的交易数据存在各种问题：字段缺失、格式错误、金额为负、重复交易ID——这些"脏数据"如果混在正常流中一并处理，会导致聚合统计结果失真。

运营要求：**脏数据不能丢，但也不能影响正常数据的处理**。脏数据需要单独记入一个"异常流水表"，供数据团队次日排查修复。

最简单的做法——在flatMap里用if-else判断，脏数据写到不同的地方：

```java
if (isValid) {
    out.collect(validData);  // 正常流
} else {
    writeToErrorTable(invalidData);  // 脏数据 → 错误表
}
```

但问题是：`writeToErrorTable` 如果直接在flatMap里调用，它是在Flink的主处理线程中同步执行的——对脏数据的任何处理都会**阻塞**正常数据的处理。而且如果脏数据越来越多（如异常流量攻击），写入错误表的延迟会直接反压到整个Flink作业。

Flink的 **Side Output（侧输出流）** 给出了更优雅的解法：在同一个算子中分出多个输出支流，脏数据走SideOutput，正常数据走主流。两个流可以各自指定不同的Sink、并行度、Backpressure处理策略，互不干扰。

---

## 2. 项目设计

> 场景：运营反馈今日的交易统计大屏上，成交额比期望值低了约2%。排查发现一批金额异常的"脏数据"被静默丢弃了。

**小胖**：我在flatMap里写了个try-catch，解析失败的数据直接return——不发射它就不影响下游了。然后扔日志里，回头查日志呗。

**大师**：try-catch吞异常是隐蔽的大坑——你今天吞了0.3%的脏数据，明天可能吞了3%，后天10%。运营发现数据趋势不对时已经过了好几天，根本没法追溯。

**小白**：那SideOutput就是能给同一个算子开"多个出口"？正常数据走正门，脏数据走后门？

**大师**：比喻很准确。ProcessFunction比Map/FlatMap多了一个`Context`参数——通过`ctx.output(tag, value)`可以发射到侧输出流。主流的`out.collect()`不变。

**技术映射：SideOutput = 在DataStream中开一个命名的"副管道"。每个副管道用OutputTag标记类型，可以与主流完全不同的数据类型。**

**小胖**：那SideOutput的数据怎么消费？后面再加一个流来读吗？

**大师**：对。通过`DataStream.getSideOutput(outputTag)`获取侧输出流——它就是一个普通的DataStream，可以接map/sink/keyBy/window等所有操作。**技术映射：sideOutput返回的DataStream和主流地位完全平等，唯一的区别是数据的来源不同。**

**小白**：主流和侧输出流在Checkpoint上怎么处理？

**大师**：主流和侧输出流共用同一个Checkpoint——它们的数据都在同一个算子实例的状态中。所以Checkpoint一致性对侧输出流一样有效。如果主流做了Exactly-Once，侧输出流也是Exactly-Once。

## 3. 项目实战

### 分步实现

#### 步骤1：ProcessFunction + SideOutput——交易流水脏数据分流

**目标**：用ProcessFunction实现交易流水的校验，正常数据走主流，三种类型的脏数据分别走三个侧输出流。

```java
package com.flink.column.chapter12;

import org.apache.flink.api.common.functions.MapFunction;
import org.apache.flink.api.java.tuple.Tuple3;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.api.datastream.SingleOutputStreamOperator;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.streaming.api.functions.ProcessFunction;
import org.apache.flink.util.Collector;
import org.apache.flink.util.OutputTag;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

/**
 * 交易流水脏数据分流：使用SideOutput将不同类型的错误数据分流到不同Sink
 * 输入: <transactionId>,<userId>,<amount>,<timestamp>
 */
public class SideOutputDemo {

    private static final Logger LOG = LoggerFactory.getLogger(SideOutputDemo.class);

    // ============== 定义三种侧输出流的Tag ==============
    // Tag需要是匿名内部类（保留泛型信息）
    private static final OutputTag<String> TAG_NEGATIVE_AMOUNT =
            new OutputTag<String>("negative-amount") {};
    private static final OutputTag<String> TAG_MISSING_FIELD =
            new OutputTag<String>("missing-field") {};
    private static final OutputTag<String> TAG_DUPLICATE_ID =
            new OutputTag<String>("duplicate-id") {};

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        env.setParallelism(2);

        DataStream<String> source = env.socketTextStream("localhost", 9999);

        // ProcessFunction：同时发射主流+多个侧流
        SingleOutputStreamOperator<String> mainStream = source
                .process(new TransactionValidator())
                .name("tx-validator");

        // ============== 主流：有效交易 ==============
        mainStream
                .map(t -> "[有效交易] " + t)
                .print()
                .name("valid-tx-output");

        // ============== 侧流1：金额为负 ==============
        DataStream<String> negativeStream = mainStream.getSideOutput(TAG_NEGATIVE_AMOUNT);
        negativeStream
                .map(t -> "[错误-金额异常] " + t)
                .print()
                .name("negative-amount-output");

        // ============== 侧流2：字段缺失 ==============
        DataStream<String> missingStream = mainStream.getSideOutput(TAG_MISSING_FIELD);
        missingStream
                .map(t -> "[错误-字段缺失] " + t)
                .print()
                .name("missing-field-output");

        // ============== 侧流3：重复交易ID ==============
        DataStream<String> duplicateStream = mainStream.getSideOutput(TAG_DUPLICATE_ID);
        duplicateStream
                .map(t -> "[错误-重复交易ID] " + t)
                .print()
                .name("duplicate-id-output");

        env.execute("Chapter12-SideOutputDemo");
    }

    /**
     * 交易校验器：解析、校验、分类
     */
    public static class TransactionValidator
            extends ProcessFunction<String, String> {

        private transient java.util.Set<String> seenIds;

        @Override
        public void open(org.apache.flink.configuration.Configuration parameters) {
            seenIds = new java.util.HashSet<>();
        }

        @Override
        public void processElement(String line, Context ctx, Collector<String> out) {
            String[] parts = line.split(",");

            // ----- 校验1: 字段缺失 -----
            if (parts.length < 4) {
                LOG.warn("字段缺失: {}", line);
                ctx.output(TAG_MISSING_FIELD, line);
                return;
            }

            String txId = parts[0];
            String amountStr = parts[2];

            // ----- 校验2: 金额为负或为0 -----
            double amount;
            try {
                amount = Double.parseDouble(amountStr);
            } catch (NumberFormatException e) {
                ctx.output(TAG_MISSING_FIELD, line);
                return;
            }

            if (amount <= 0) {
                LOG.warn("金额异常: {} <= 0", amount);
                ctx.output(TAG_NEGATIVE_AMOUNT, line);
                return;
            }

            // ----- 校验3: 重复交易ID -----
            if (seenIds.contains(txId)) {
                LOG.warn("重复交易ID: {}", txId);
                ctx.output(TAG_DUPLICATE_ID, line);
                return;
            }
            seenIds.add(txId);

            // ----- 通过所有校验：发射到主流 -----
            out.collect(line);
        }
    }
}
```

**测试数据**：

```
TX001,user1,100.00,1714293932000      # 正常
TX002,user2,200.00,1714293933000      # 正常
TX003,user3,-50.00,1714293934000      # 金额为负→侧流
TX004,user4                           # 字段缺失→侧流
TX001,user1,300.00,1714293935000      # 重复ID→侧流
TX005,user4,abc,1714293936000         # 金额非数字→字段缺失
TX006,user5,500.00,1714293937000      # 正常
```

**预期输出**：

```

[有效交易] TX002,user2,200.00,1714293933000
[错误-金额异常] TX003,user3,-50.00,1714293934000
[错误-字段缺失] TX004,user4
[错误-重复交易ID] TX001,user1,300.00,1714293935000
[错误-字段缺失] TX005,user4,abc,1714293936000
[有效交易] TX006,user5,500.00,1714293937000
```

#### 步骤2：SideOutput与不同的Sink对接

**目标**：为每个侧流配置不同的Sink——主流→Kafka，错误流→MySQL异常表。

```java
// 生产环境的典型配置：
// 主流 → Kafka 成功交易Topic
// 金额异常流 → MySQL异常交易表（人工排查）
// 字段缺失流 → 单独日志文件 + 告警
// 重复ID流 → Redis缓存（方便快速查询）

mainStream.addSink(kafkaSink("valid-tx-topic"));
mainStream.getSideOutput(TAG_NEGATIVE_AMOUNT)
    .addSink(jdbcSink("INSERT INTO error_tx(type, data) VALUES ('NEGATIVE', ?)"));
mainStream.getSideOutput(TAG_MISSING_FIELD)
    .addSink(jdbcSink("INSERT INTO error_tx(type, data) VALUES ('MISSING', ?)"));
mainStream.getSideOutput(TAG_DUPLICATE_ID)
    .addSink(new PrintSinkFunction<>());  // 开发期间直接打印
```

#### 步骤3：在窗口中使用SideOutput——捕获迟到数据

**目标**：在EventTime窗口中用SideOutput捕获Watermark之后的迟到数据。

```java
// 重复第7章的迟到数据处理模式：
SingleOutputStreamOperator<...> windowed = stream
    .keyBy(...)

    .sideOutputLateData(new OutputTag<...>("late-events"){})
    .reduce(...);

DataStream<...> lateStream = windowed.getSideOutput(LATE_TAG);
lateStream.addSink(new LateDataSink());  // 迟到数据单独落到HDFS
```

### 可能遇到的坑

1. **SideOutput的OutputTag定义必须用匿名内部类（保留泛型信息）**
   ```java
   // ✅ 正确
   private static final OutputTag<String> tag = new OutputTag<String>("name") {};
   // ❌ 错误——编译能过但运行时类型推断失败
   private static final OutputTag<String> tag = new OutputTag<>("name");
   ```

2. **一个算子可以定义多个OutputTag，但每个tag必须在类级别定义**
   - 不能在processElement方法内动态创建OutputTag——Flink需要在作业图构建时就确定所有输出类型

3. **getSideOutput(tag)只能调用一次——如果多次调用，会创建多个重复的流**
   - 这不会报错但会导致数据重复处理

## 4. 项目总结

### SideOutput的典型使用场景

| 场景 | 主流 | 侧流 |
|------|------|------|
| 数据校验 | 有效数据 | 缺失字段/格式错误/非法值 |
| 异常检测 | 正常事件 | 异常告警事件 |
| 迟到数据 | 准时到达的数据 | Watermark之后到达的数据 |
| 分级处理 | 高优先级数据 | 低优先级数据（降级处理） |
| 事件分类 | 全部事件 | 特定类型事件（如仅提取登录事件） |

### SideOutput vs 分流（Split）

| 维度 | SideOutput（推荐） | Split（已废弃） |
|------|-------------------|----------------|
| API状态 | 活跃维护 | Flink 1.12+已经标记为废弃 |
| 类型安全 | 是（OutputTag带类型） | 否（Selector返回字符串标签） |
| 不同数据类型 | 支持（侧流可以与主流不同类型） | 不支持（流内元素类型必须相同） |
| 多级分类 | 支持多个OutputTag | 支持多标签但无法控制路径 |

### 注意事项
- OutputTag必须在类的静态域中定义（或作为成员变量），使用匿名内部类语法`new OutputTag<T>(name){}`
- 一个算子可以发射到任意数量的侧输出流，但这不会影响该算子的并行度
- SideOutput流同样在Checkpoint管理范围内——恢复作业时侧流数据不会丢失

### 常见踩坑经验

**案例1：SideOutput流数据量远大于主流，导致Sink压力集中到侧流**
- 根因：正常数据少、脏数据多时（如爬虫攻击），侧流Sink被冲垮
- 解方：给侧流Sink设置独立的并行度和反压控制；或增加脏数据采样比例

**案例2：getSideOutput(tag)返回的是DataStream，但tag定义在另一个类中**
- 根因：OutputTag的泛型信息在跨类访问时丢失
- 解方：将OutputTag定义为public static final，或者通过公共常量类管理

**案例3：SideOutput数据在算子链优化后被串到了错误的算子**
- 根因：算子链优化合并了多个算子，SideOutput可能从合并后的Task中提取
- 解方：在ProcessFunction后调用`disableChaining()`或`startNewChain()`确保SideOutput的边界清晰

### 优点 & 缺点

| | SideOutput侧输出流 | Filter分流（多条流水线） |
|------|-----------|-----------|
| **优点1** | 一个ProcessFunction支持多个输出，数据只解析一次 | 每条流水线独立解析数据，重复计算 |
| **优点2** | 侧流可与主流数据类型不同，灵活性高 | 分流后流类型必须一致，无法独立定义 |
| **优点3** | 侧流和主流共用Checkpoint，Exactly-Once一致 | 多条流水线需各自独立管理Checkpoint |
| **优点4** | 侧流可设置独立的Sink和并行度，互不干扰 | 所有分流共享Source并行度和反压 |
| **缺点1** | OutputTag必须提前定义，不够灵活 | 运行时可通过条件动态分流 |
| **缺点2** | 侧流数据无法被主流聚合引用 | 多流可用CoGroup/Connect等显式关联 |

### 适用场景

**典型场景**：
1. 数据校验分流——有效数据走主流，脏数据走侧流（字段缺失/格式错误/金额异常等）
2. 迟到数据捕获——EventTime窗口中使用sideOutputLateData捕获迟到事件
3. 事件分类处理——按事件类型分流到不同Sink（日志/告警/归档）
4. 分级降级处理——低优先级数据走侧流做降级处理，不影响主流吞吐

**不适用场景**：
1. 需要跨流Join的场景——SideOutput流之间不能直接关联，需用Connect/Union合并
2. 运行时动态添加新的分类——OutputTag必须在作业构建时定义，无法运行时新增
