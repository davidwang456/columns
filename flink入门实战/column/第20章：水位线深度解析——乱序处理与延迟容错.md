# 第20章：水位线深度解析——乱序处理与延迟容错

---

## 1. 项目背景

第7章我们学习了EventTime和Watermark的基础：`forBoundedOutOfOrderness(Duration.ofSeconds(10))`，设置了最大乱序容忍度。

但在生产中，Watermark的细节远比想象中复杂。某电商公司的Flink作业用EventTime做交易统计，遇到了这些问题：

- 一个Kafka Topic有8个分区，其中1个分区因为网络抖动数据延迟了30秒——结果所有窗口都等这个慢分区，整体处理延迟从毫秒级变成了30秒级
- 部分数据晚到了20秒，但Watermark容忍度只设了5秒——这些数据被直接丢弃，统计结果偏差0.3%
- 窗口触发了之后又来了一个晚到数据，Flink虽然允许处理但这个数据没能进入已经闭合的窗口

这些问题都指向同一个核心技术：**Watermark的生成、传播和多分区协调**。

---

## 2. 项目设计

> 场景：大促期间，交易数据全链路延迟突然从2秒飙到45秒，运营反馈大屏数据慢了。

**大师**：我们先看看Watermark的情况。你设置了maxOutOfOrderness=10秒，但实际延迟怎么可能到45秒？

**小胖**：我看了WebUI，JobManager的Watermark = 14:00:50，但当前时间已经是14:01:35——差了45秒！

**大师**：这说明某个Source分区的Watermark卡住了。Flink的全局Watermark取所有分区Watermark的**最小值**——如果某个分区没有新数据，它的Watermark停留在初始值（Long.MIN_VALUE），全局Watermark也卡死。

**技术映射：多分区Watermark = min(分区1的Watermark, 分区2的Watermark, ...)。一个慢分区拖慢全局。**

**小白**：那把慢分区标记为"空闲"不就好了？空闲分区不参与最小值运算？

**大师**：对，这就是`withIdleness(Duration)`的作用。如果一个分区超过指定时间（如2分钟）没有数据进来，Flink将其标记为空闲——它的Watermark不再参与全局最小值计算。

**技术映射：空闲分区的Watermark会被"忽略"——不再阻塞全局Watermark的推进。但要注意：空闲分区恢复后有数据时，Flink会重新将其纳入计算。**

**小胖**：那如果数据迟到超过了Watermark容忍范围，但又不想丢，怎么处理？

**大师**：三个级别的兜底：

1. **maxOutOfOrderness**：第一道防线。Watermark = maxEventTime - 容忍度。在这个范围内的乱序数据正常处理。
2. **AllowedLateness**：第二道防线。Watermark通过窗口endTime后，窗口不立即销毁，而是等待额外的时间。在这段时间内到达的迟到数据，可以触发窗口重新计算。
3. **SideOutput**：最终兜底。超过AllowedLateness的数据被发送到侧输出流，记录到错误Topic，供离线补算。

**技术映射：乱序处理三级体系——容忍（保大部分）→ 等待（保临界）→ 兜底（完全不丢）。**

## 3. 项目实战

### 分步实现

#### 步骤1：多分区Watermark推进可视化

**目标**：打印每个分区的水位线，观察慢分区的影响。

```java
package com.flink.column.chapter20;

import org.apache.flink.api.common.eventtime.WatermarkStrategy;
import org.apache.flink.api.java.tuple.Tuple3;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.streaming.api.functions.ProcessFunction;
import org.apache.flink.util.Collector;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import java.time.Duration;

/**
 * 多分区Watermark推进观察——模拟慢分区
 * 输入: <partitionId>,<eventTime>
 */
public class MultiPartitionWatermark {

    private static final Logger LOG = LoggerFactory.getLogger(MultiPartitionWatermark.class);

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        env.setParallelism(1);

        DataStream<String> text = env.socketTextStream("localhost", 9999);

        text.map(line -> {
            String[] p = line.split(",");
            return Tuple3.of(p[0], Long.parseLong(p[1]), 0L);
        })
        .returns(Types.TUPLE(Types.STRING, Types.LONG, Types.LONG))

        .assignTimestampsAndWatermarks(
                WatermarkStrategy.<Tuple3<String, Long, Long>>forBoundedOutOfOrderness(
                                Duration.ofSeconds(10))
                        .withTimestampAssigner((event, ts) -> event.f1)
                        // 关键：设置空闲超时2分钟——超过2分钟无数据的partition自动隔离
                        .withIdleness(Duration.ofMinutes(2))
        )

        .keyBy(t -> t.f0)  // 按分区ID分组，模拟多分区场景

        .process(new ProcessFunction<Tuple3<String, Long, Long>,
                                    Tuple3<String, Long, Long>>() {
            @Override
            public void processElement(
                    Tuple3<String, Long, Long> event,
                    Context ctx,
                    Collector<Tuple3<String, Long, Long>> out) {

                long watermark = ctx.timerService().currentWatermark();
                event.f2 = watermark;

                boolean isLate = event.f1 < watermark;
                LOG.info("[分区{}] eventTime={}, watermark={}, diff={}ms, {}",
                        event.f0, event.f1, watermark,
                        event.f1 - watermark,
                        isLate ? "[已过期] !!!!" : "[正常]");

                if (!isLate) {
                    out.collect(event);
                }
            }
        })
        .print();

        env.execute("Chapter20-MultiPartitionWatermark");
    }
}
```

**测试数据**：

```
A,1000
A,2000
B,1500
A,3000
B,2500
# 分区A在此停顿（模拟慢分区）
C,10000  # 新分区C的数据，但A已经很久没来了
A,4000   # A恢复
```

#### 步骤2：AllowedLateness——窗口等待迟到数据

**目标**：窗口触发后继续等待迟到数据，不立即关闭。

```java
package com.flink.column.chapter20;

import org.apache.flink.api.common.eventtime.WatermarkStrategy;
import org.apache.flink.api.java.tuple.Tuple2;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.api.datastream.SingleOutputStreamOperator;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.streaming.api.functions.windowing.ProcessWindowFunction;
import org.apache.flink.streaming.api.windowing.assigners.TumblingEventTimeWindows;
import org.apache.flink.streaming.api.windowing.time.Time;
import org.apache.flink.streaming.api.windowing.windows.TimeWindow;
import org.apache.flink.util.Collector;
import org.apache.flink.util.OutputTag;
import java.time.Duration;

/**
 * AllowedLateness + SideOutput 迟到数据三级处理
 */
public class AllowedLatenessDemo {

    private static final OutputTag<Tuple2<String, Long>> LATE_TAG =
            new OutputTag<Tuple2<String, Long>>("late-data") {};

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        env.setParallelism(1);

        DataStream<String> text = env.socketTextStream("localhost", 9999);

        DataStream<Tuple2<String, Long>> data = text
                .map(line -> {
                    String[] p = line.split(",");
                    return Tuple2.of(p[0], Long.parseLong(p[1]));
                })
                .returns(Types.TUPLE(Types.STRING, Types.LONG))
                .assignTimestampsAndWatermarks(
                        WatermarkStrategy.<Tuple2<String, Long>>forBoundedOutOfOrderness(
                                        Duration.ofSeconds(5))
                                .withTimestampAssigner((event, ts) -> event.f1)
                );

        SingleOutputStreamOperator<String> windowed = data
                .keyBy(t -> t.f0)
                .window(TumblingEventTimeWindows.of(Time.seconds(10)))
                // AllowedLateness: 窗口触发后额外等2秒的迟到数据
                .allowedLateness(Time.seconds(2))
                .sideOutputLateData(LATE_TAG)
                .process(new ProcessWindowFunction<
                        Tuple2<String, Long>, String, String, TimeWindow>() {

                    @Override
                    public void process(String key, Context ctx,
                                        Iterable<Tuple2<String, Long>> elements,
                                        Collector<String> out) {

                        long count = 0;
                        for (Tuple2<String, Long> e : elements) {
                            count++;
                        }

                        String trigger = ctx.currentProcessingTime() > ctx.window().getEnd()
                                ? "[迟到触发]"
                                : "[首次触发]";

                        out.collect(String.format(
                                "%s key=%s, count=%d, window=[%d, %d)",
                                trigger, key, count,
                                ctx.window().getStart(), ctx.window().getEnd()));
                    }
                });

        // 主流结果
        windowed.print("Window");

        // 侧输出流——超过AllowedLateness的终极迟到数据
        windowed.getSideOutput(LATE_TAG)
                .map(t -> String.format("[终极迟到] key=%s, time=%d", t.f0, t.f1))
                .print("Late");

        env.execute("Chapter20-AllowedLateness");
    }
}
```

**测试数据**（注意时间戳是相对值）：

```
A,1000
A,5000
A,12000    # Watermark = 12000-5000=7000, 不超过10000, 正常进入[0,10000)窗口
# 等几秒
A,8000     # 迟到的数据。Watermark已超过8000，但还在AllowedLateness(2秒)范围内→触发二次计算
# 再等几秒
A,3000     # 超过了AllowedLateness→侧输出流
```

**预期输出**：

```
Window> [首次触发] key=A, count=3, window=[0, 10000)     ← 前3条
Window> [迟到触发] key=A, count=4, window=[0, 10000)    ← 第4条(8000)来后重算
Late>   [终极迟到] key=A, time=3000                      ← 第5条(3000)进侧流
```

#### 步骤3：自定义WatermarkStrategy——更精细的控制

**目标**：实现自定义Watermark生成策略，打破"固定延迟"的限制。

```java
// 场景：数据来源有多个，有的分区严格有序，有的分区乱序大
// 不能用统一的maxOutOfOrderness——为有序分区浪费了不必要的延迟

// 自定义：仅追加大于之前最大eventTime时推进Watermark
// 等价于 forMonotonousTimestamps()
WatermarkStrategy<String> monotonic =
    WatermarkStrategy.<String>forMonotonousTimestamps()
        .withTimestampAssigner((event, ts) -> Long.parseLong(event));

// 自定义：基于punctuated watermark——只在满足条件时推进
WatermarkStrategy<String> punctuated =
    WatermarkStrategy.<String>forGenerator(context -> new MyPunctuatedGenerator())
        .withTimestampAssigner((event, ts) -> Long.parseLong(event));

// 自定义WatermarkGenerator示例
public static class MyPunctuatedGenerator implements WatermarkGenerator<String> {
    private long maxTs = Long.MIN_VALUE;

    @Override
    public void onEvent(String event, long eventTimestamp, WatermarkOutput output) {
        maxTs = Math.max(maxTs, eventTimestamp);
        // 只在特定数据到达时发射Watermark（如"end_of_batch"标记）
        if (event.contains("flush")) {
            output.emitWatermark(new Watermark(maxTs));
        }
    }

    @Override
    public void onPeriodicEmit(WatermarkOutput output) {
        // 不做周期性发射
    }
}
```

### 可能遇到的坑

1. **withIdleness设置太短导致频繁的idle/active切换**
   - 根因：某些分区天生稀疏（如低频事件Topic的分区），每5分钟才有几条数据。如果idle设为1分钟，这些分区频繁在idle/active状态切换，Watermark反复跳变
   - 解决：将idle设为最大合理间隔的2-3倍（如10-15分钟）

2. **AllowedLateness导致同一个窗口被触发多次**
   - 根因：每次迟到数据到达都会触发窗口重新计算，下游Sink会看到同一窗口的多次输出
   - 解方：使用幂等Sink或窗口聚合的"最终结果"标识

3. **Periodic Watermark的发射间隔（默认200ms）过高**
   - 根因：高频发射Watermark增加不必要的CPU开销
   - 解方：`ExecutionConfig.setAutoWatermarkInterval(1000)`（设置为1秒）

## 4. 项目总结

### Watermark参数调优速查

| 参数 | 默认值 | 含义 | 推荐值 |
|------|--------|------|--------|
| maxOutOfOrderness | 0 | 最大乱序容忍度 | 5-60秒（视网络质量） |
| withIdleness | 关闭 | 空闲超时 | 2-15分钟 |
| allowedLateness | 0 | 窗口等待迟到数据时间 | 0-60秒 |
| autoWatermarkInterval | 200ms | Watermark发射间隔 | 500-2000ms |

### 三级延迟处理体系

```
数据到来
    │
    ├── eventTime ≥ Watermark - maxOutOfOrderness
    │   └── 正常处理（第一级）
    │
    ├── eventTime < Watermark - maxOutOfOrderness
    │   ├── eventTime ≥ 窗口endTime + allowedLateness
    │   │   └── 窗口重新计算（第二级）
    │   │
    │   └── eventTime < 窗口endTime + allowedLateness
    │       └── 侧输出流（第三级）
    │
    └── 兜底：离线补算（完全超出Flink处理范围）
```

### 注意事项
- withIdleness对多分区Kafka Source尤其重要——一个分区无数据不会拖垮整个作业
- allowedLateness要谨慎使用——太大会延迟结果输出，太小会丢失数据
- Watermark是估计值，不是精确值——它基于"当前看到的最大EventTime"推测后续不会再有序时间的数据

### 常见踩坑经验

**案例1：作业启动后很长时间窗口不触发，但数据已经源源不断到来**
- 根因：Watermark没有数据源。如果Source是`fromElements()`或Socket，需要显式设置`assignTimestampsAndWatermarks`
- 解方：确认DataStream调用了`assignTimestampsAndWatermarks`方法

**案例2：所有分区的Watermark都正常推进，但窗口就是不触发**
- 根因：窗口的类型是EventTime窗口，但Watermark一直小于窗口endTime
- 解方：检查`maxOutOfOrderness`是否正确——如果设得太大（如1小时），Watermark = maxEventTime - 1小时，窗口endTime需要比当前maxEventTime小1小时才能触发

**案例3：空闲分区恢复后，Watermark突然大幅"后退"**
- 根因：空闲分区恢复后，它的初始Watermark=Long.MIN_VALUE（或当前第一条数据的eventTime），导致全局Watermark最小值瞬间下降
- 解方：升级到Flink 1.16+（修复了此问题）；或使用`withIdleness`配合`forBoundedOutOfOrderness`的正确组合

### 优点 & 缺点

| | Watermark多级迟到处理（容忍+等待+兜底） | 窗口到期即关闭，迟到数据丢弃 |
|------|-----------|-----------|
| **优点1** | 三级兜底体系，数据零丢失 | 实现最简单，无需额外配置 |
| **优点2** | withIdleness自动隔离慢分区，不阻塞全局 | 无需处理分区空闲问题 |
| **优点3** | AllowedLateness平衡延迟与完整性 | 无 |
| **缺点1** | 配置复杂——maxOutOfOrderness/Idleness/AllowedLateness需配合调参 | 零配置 |
| **缺点2** | AllowedLateness导致窗口多次触发，下游需处理重复结果 | 触发一次，结果确定 |
| **缺点3** | 空闲分区恢复时Watermark可能回退 | 无此问题 |

### 适用场景

**典型场景**：
1. 多分区Kafka消费——各分区数据延迟不均匀，需withIdleness隔离慢分区
2. 容忍网络抖动的实时统计——maxOutOfOrderness吸收短暂延迟
3. 数据完整性的关键场景——AllowedLateness+SideOutput确保数据零丢失
4. 广告/交易等需准确窗口归因的场景——EventTime+Watermark精确归因

**不适用场景**：
1. 数据严格有序的单分区场景——使用forMonotonousTimestamps即可，无需复杂配置
2. 对延迟极度敏感且容忍少量丢数——ProcessingTime更简单高效

### 思考题

1. 一个Kafka Topic有3个分区，分别承载了60%、30%、10%的数据。分区3因为下游消费慢产生了反压，导致数据延迟45秒。此时全局Watermark是多少？Watermark的推进速度取决于哪个分区？

2. AllowedLateness = 10秒，窗口大小 = 1分钟。一条迟到数据在窗口首次触发后第5秒到达——它会触发窗口重新计算吗？在第15秒到达呢？窗口一共会被触发几次？（提示：AllowedLateness从Watermark超过endTime时开始计算，不是从首次触发开始）
