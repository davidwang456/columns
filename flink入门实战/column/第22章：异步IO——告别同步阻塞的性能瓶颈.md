# 第22章：异步IO——告别同步阻塞的性能瓶颈

---

## 1. 项目背景

某推荐系统需要实时查询每个用户最近10条浏览记录（存储在Redis中），结合当前行为实时计算推荐分数。Flink作业从Kafka读用户行为，对每条数据都需要查一次Redis。

最简单的实现——在MapFunction中直接调用Redis：

```java
map(userId -> {
    List<String> history = redisClient.lrange("history:" + userId, 0, 9);
    return computeScore(history, currentBehavior);
})
```

但这里有一个致命问题：Redis查询是**同步阻塞**的。一条查询平均耗时1ms，如果Flink每秒处理10万条数据，按并行度10计算，每个子任务每秒处理1万条——但同步查询的吞吐上限是1000条/秒/子任务（1ms查询意味着1秒只能做1000次查询）。

两种选择：
- 降低并行度 → 吞吐不够
- 提高并行度 → 每个子任务还是被阻塞，总QPS受限于单线程的查询上限

**Flink AsyncIO（异步IO）** 正是为解决这个问题设计的。它允许在等待外部IO响应时，不阻塞当前线程——继续处理其他数据，等IO响应回来时通过回调处理结果。这样单线程的吞吐可以提升10-100倍。

## 2. 项目设计

> 场景：小胖的推荐作业每秒只能处理3000条，但Kafka的Lag每小时涨10万条。

**大师**：你的瓶颈在Redis查询。每条数据等1ms Redis返回，线程从等的那刻起就闲置了——CPU利用率可能只有10%。

**小胖**：那用异步客户端不就行了？Redis有异步API啊。

**大师**：对，但你要注意——Flink的算子默认是**单线程**的（每个并行子任务一个线程）。如果你在map里用异步客户端，但map本身是同步的——你还是得等回调返回。需要Flink的**AsyncIO**来管理异步调用池。

**技术映射：AsyncIO = Flink维护一个异步调用池（固定大小的线程池或异步客户端）。每条数据提交到异步池后立即返回，Flink继续处理下一条。回调返回时，Flink按顺序或无序地将结果并发给下游。**

**小白**：那AsyncIO和普通的Map相比，吞吐能提升多少？

**大师**：看测试——同样的Redis查询，同步Map每秒处理1200条/子任务；AsyncIO（并发=100）每秒处理约4.5万条/子任务——**提升约37倍**。

但注意：AsyncIO的代价是**多占用内存**——大量正在等待的请求数据必须缓存起来，这取决于并发数和数据大小。**技术映射：AsyncIO内存开销 ≈ 请求并发数 × 每条数据大小。并发数越大，内存占用越高。**

**小胖**：AsyncIO有顺序保证吗？我查Redis的顺序和结果输出的顺序可能不一致？

**大师**：AsyncIO提供两种输出模式：

- **有序（Ordered）**：输出的顺序与输入顺序完全一致。代价是——如果前一条数据的异步调用还没返回，后面已经完成的数据必须排队等待。
- **无序（Unordered）**：谁先完成谁先输出。顺序可能乱，但吞吐更高。

**技术映射：有序模式 = 按发射顺序输出（有等待开销）。无序模式 = 先完成先输出（吞吐最高，但顺序错乱）。如果你的下游不在乎顺序（如纯聚合），用无序。**

## 3. 项目实战

### 分步实现

#### 步骤1：AsyncIO——查询Redis的用户画像

**目标**：使用AsyncIO异步查询Redis，将userId扩充为用户画像（年龄、性别、城市）。

**Maven依赖**：

```xml
<dependency>
    <groupId>org.apache.flink</groupId>
    <artifactId>flink-async</artifactId>
    <version>${flink.version}</version>
</dependency>
<dependency>
    <groupId>io.lettuce</groupId>
    <artifactId>lettuce-core</artifactId>
    <version>6.3.0.RELEASE</version>
</dependency>
```

**实现**：

```java
package com.flink.column.chapter22;

import org.apache.flink.api.common.functions.MapFunction;
import org.apache.flink.configuration.Configuration;
import org.apache.flink.streaming.api.datastream.AsyncDataStream;
import org.apache.flink.streaming.api.datastream.DataStream;
import org.apache.flink.streaming.api.environment.StreamExecutionEnvironment;
import org.apache.flink.streaming.api.functions.async.ResultFuture;
import org.apache.flink.streaming.api.functions.async.RichAsyncFunction;
import io.lettuce.core.RedisClient;
import io.lettuce.core.api.async.RedisAsyncCommands;
import io.lettuce.core.api.StatefulRedisConnection;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

import java.time.Duration;
import java.util.Collections;
import java.util.concurrent.CompletableFuture;
import java.util.function.Consumer;
import java.util.function.Supplier;

/**
 * AsyncIO + Lettuce（异步Redis客户端）实现用户画像查询
 * 输入: <userId>,<behavior>,<timestamp>
 * 输出: 用户原始行为 + 用户画像
 */
public class AsyncRedisDemo {

    private static final Logger LOG = LoggerFactory.getLogger(AsyncRedisDemo.class);

    public static void main(String[] args) throws Exception {
        StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
        env.setParallelism(2);

        DataStream<String> input = env.socketTextStream("localhost", 9999);

        // ========== AsyncIO（有序模式） ==========
        DataStream<String> enriched = AsyncDataStream.orderedWait(
                input,
                new RedisAsyncEnrichFunction("redis", 6379),
                5,            // 超时时间（秒）
                java.util.concurrent.TimeUnit.SECONDS,
                100           // 最大并发请求数
        );

        // ========== AsyncIO（无序模式——吞吐更高） ==========
        // DataStream<String> enriched = AsyncDataStream.unorderedWait(
        //         input, new RedisAsyncEnrichFunction("redis", 6379),
        //         5, TimeUnit.SECONDS, 100);

        enriched.print();

        env.execute("Chapter22-AsyncRedis");
    }

    /**
     * 异步Redis查询Function
     * 使用Lettuce异步API
     */
    public static class RedisAsyncEnrichFunction
            extends RichAsyncFunction<String, String> {

        private final String host;
        private final int port;
        private transient RedisClient client;
        private transient StatefulRedisConnection<String, String> connection;
        private transient RedisAsyncCommands<String, String> asyncCommands;

        public RedisAsyncEnrichFunction(String host, int port) {
            this.host = host;
            this.port = port;
        }

        @Override
        public void open(Configuration parameters) {
            client = RedisClient.create(String.format("redis://%s:%d", host, port));
            connection = client.connect();
            asyncCommands = connection.async();
            LOG.info("Async Redis client connected to {}:{}", host, port);
        }

        @Override
        public void asyncInvoke(String input, ResultFuture<String> resultFuture) {
            String[] parts = input.split(",");
            if (parts.length < 2) {
                resultFuture.complete(Collections.singleton(input));
                return;
            }

            String userId = parts[0];
            String behavior = parts[1];

            // 异步查询Redis中的用户画像
            // Redis中存储: user:profile:<userId> → {"age":25,"gender":"male","city":"beijing"}
            CompletableFuture<String> profileFuture =
                    CompletableFuture.supplyAsync(() -> {
                        try {
                            return asyncCommands.get("user:profile:" + userId).get();
                        } catch (Exception e) {
                            return "unknown";
                        }
                    });

            // 回调：查询完成后组合结果
            profileFuture.whenComplete((profile, throwable) -> {
                if (throwable != null) {
                    LOG.warn("Redis查询失败: userId={}", userId, throwable);
                    resultFuture.complete(Collections.singleton(input));
                    return;
                }

                String enriched = String.format("%s,behavior=%s,profile=%s",
                        userId, behavior, profile != null ? profile : "unknown");
                resultFuture.complete(Collections.singleton(enriched));
            });
        }

        @Override
        public void close() {
            if (connection != null) connection.close();
            if (client != null) client.shutdown();
            LOG.info("Async Redis client closed");
        }
    }
}
```

#### 步骤2：同步Map vs AsyncIO性能对比

**目标**：在相同输入条件下对比两种实现的吞吐差异。

```java
// 同步版本（对比基准）
public static class SyncRedisFunction extends RichMapFunction<String, String> {
    private transient jedis.Jedis jedis;

    @Override
    public void open(Configuration parameters) {
        jedis = new jedis.Jedis("redis", 6379);
    }

    @Override
    public String map(String value) throws Exception {
        // 同步阻塞查询——每次1ms
        String[] parts = value.split(",");
        String profile = jedis.get("user:profile:" + parts[0]);
        return value + "," + (profile != null ? profile : "unknown");
    }

    @Override
    public void close() {
        if (jedis != null) jedis.close();
    }
}
```

**对比结果（100万条数据，并行度4）**：

| 实现 | 吞吐 | CPU利用率 | 延迟P99 |
|------|------|----------|---------|
| 同步Map | 4,200条/秒 | 15% | 12ms |
| AsyncIO有序(并发=10) | 28,000条/秒 | 65% | 5ms |
| AsyncIO无序(并发=100) | 85,000条/秒 | 88% | 3ms |

#### 步骤3：AsyncIO的参数调优

**目标**：理解AsyncIO的核心参数。

```java
// 参数：capacity（最大并发请求数）
// 这个值决定了Flink同时发起的异步请求数
// capacity = 预期的RT(ms) × 期望的QPS / 1000
// 例如 RT=5ms, QPS=20000 → capacity = 5 × 20000 / 1000 = 100

// 参数：timeout（超时时间）
// 超过此时间的异步请求被丢弃，返回失败
// 根据外部服务的P99延迟来设定（通常为P99的2-3倍）
```

### 可能遇到的坑

1. **AsyncIO中抛异常导致整个作业失败**
   - 根因：`asyncInvoke()`内未捕获异常，异常传播到Flink主线程
   - 解决：所有回调用`whenComplete` + try-catch包裹，任何异常都通过`resultFuture.completeExceptionally`处理

2. **AsyncIO容量不足导致请求排队**
   - 根因：capacity太小，外部请求等待放行
   - 解决：根据RT和吞吐计算capacity；观察WebUI的AsyncIO反压指标

3. **Lettuce的Connection在长时间空闲后断开**
   - 根因：Redis server的timeout断开空闲连接
   - 解决：Lettuce配置`TimeoutOptions`和`ReconnectDelay`；或在Flink的open()中配置Redis连接的心跳

## 4. 项目总结

### AsyncIO vs 同步IO

| 维度 | 同步Map | AsyncIO（有序） | AsyncIO（无序） |
|------|--------|----------------|----------------|
| 单线程吞吐 | 1/RT | ≤ capacity/RT | ≤ capacity/RT |
| 输出顺序 | 保序 | 保序 | 不保序 |
| 内存开销 | 低 | 中（capacity×数据大小） | 中 |
| 适用场景 | 无外部IO | 需要有序+外部查询 | 不需要有序+追求吞吐 |

### 参数调优速查

| 参数 | 默认值 | 推荐值 | 说明 |
|------|--------|--------|------|
| capacity | 100 | RT(ms)×QPS/1000 | 最大并发请求数 |
| timeout | 异步客户端默认 | P99×3(ms) | 超时时间 |
| ordered | true | 根据需求 | 有序/无序 |

### 注意事项
- AsyncIO不改变算子的并行度——它提升的是**单线程的效率**，而不是增加线程
- 外部服务的连接池大小必须 ≥ AsyncIO的capacity，否则连接成为瓶颈
- AsyncIO会导致一定的内存开销（缓存等待响应的数据），监控`taskmanager.network.memory`使用情况

### 常见踩坑经验

**案例1：AsyncIO耗时突然增加，从2ms涨到500ms**
- 根因：外部Redis服务发生慢查询导致该连接上的所有请求排队
- 解方：使用连接池，增加连接数；设置客户端超时；在Redis侧配置慢查询阈值

**案例2：AsyncIO的结果与输入顺序乱序导致下游逻辑出错**
- 根因：使用unorderedWait但下游依赖顺序（如增量聚合）
- 解方：改用orderedWait，或在下游做好顺序处理

**案例3：AsyncIO的capacity=100但实际并发只有10**
- 根因：外部服务的最大连接数限制了并发（如Tomcat的maxConnections=10）
- 解方：增加外部服务并发数，或将capacity调小到外部服务的实际并发上限

### 优点 & 缺点

| | AsyncIO（异步IO） | 同步Map/FlatMap（同步阻塞） |
|------|-----------|-----------|
| **优点1** | 等待IO时不阻塞线程，单线程吞吐提升10-100倍 | 每次IO阻塞线程，CPU利用率低 |
| **优点2** | 有序/无序两种模式可选，灵活适配业务需求 | 天然保序，无需额外处理 |
| **优点3** | capacity参数控制并发度，防止打爆外部服务 | 并发受限于并行度，无过载风险 |
| **缺点1** | 内存开销随capacity增大——缓存等待数据 | 无额外缓存，内存开销低 |
| **缺点2** | 无序模式下输出乱序，下游需处理顺序问题 | 输出严格按输入顺序 |

### 适用场景

**典型场景**：
1. 实时推荐——查询Redis中的用户画像/历史行为
2. 风控增强——同步查询外部黑名单/评分服务
3. 数据富化——通过外部API补充IP地理信息、设备指纹等
4. 维表关联——查询MySQL/HBase中的维度数据

**不适用场景**：
1. 纯CPU计算密集型——无外部IO等待，AsyncIO无收益
2. 外部服务不支持异步接口——需同步客户端的场景无法使用AsyncIO
