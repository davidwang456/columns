# 第5章：知识库与 RAG 基础——文档问答从此不求人

## 1. 项目背景

"我们的产品文档有 200 页，客服每天被重复问题问到崩溃。"产品总监在周会上拍着桌子说，"能不能让 AI 直接读我们的文档，用户问啥它答啥？"

这不是科幻。这就是 RAG（Retrieval-Augmented Generation，检索增强生成）要做的事。但很多技术团队在落地 RAG 时，遇到了一系列让人抓狂的问题：PDF 里表格的数据提取出来全乱了、长文档切分不知道切多长合适、向量检索的结果和用户问题驴唇不对马嘴、换了 Embedding 模型后向量维度对不上导致数据库报错……

这些问题的根源在于：**RAG 不是简单的"上传文档→问问题→得答案"，而是一个多阶段的流水线**。Dify 把这个流水线封装成了可视化的操作：上传文件（支持 PDF/Word/Excel/Markdown/HTML/Notion 等 20+ 格式）→ 选择分段策略 → 选择 Embedding 模型 → 索引到向量数据库 → 配置检索模式 → 在 App 中使用。每一环都影响最终效果。

本章将带你从文档上传到检索调优，完整走通 Dify 的 RAG 管线。你会理解为什么"好的分段"是 RAG 成功的一半，为什么"混合检索"比纯向量检索更靠谱，以及如何通过 Re-rank 来挽救低质量的检索结果。学完这一章，你就能自信地回答产品总监的那句话："能，而且比你想的更好。"

## 2. 项目设计——剧本式交锋对话

**小胖**：（抱着一本打印出来的产品手册）"大师，我把这本 200 页的 PDF 上传到 Dify，建了个知识库，然后问了一句'产品保修期多久'，结果 AI 回复说'未找到相关信息'。明明第 15 页上就写了保修期是 2 年啊！"

**大师**："你的 PDF 是扫描件还是文字版？"

**小胖**："呃……好像是我们市场部用 Word 排版后导出的。"

**大师**："那问题大概率出在分段（Chunking）上。Dify 不是把你的 200 页 PDF 整本'喂'给 AI 的——那样 Token 直接就爆了。它会把文档切成很多小段（Chunks），比如每 500 个字一段。用户提问时，先在向量数据库里检索最相关的几个小段，再把这些小段的内容拼接后送给 LLM。如果你的'保修期 2 年'刚好被切成两半——上一段的结尾是'产品保修'，下一段的开头是'期为 2 年'——那检索时就没有一段完整包含'产品保修期为 2 年'，自然就搜不到。"

**技术映射**：文本分段策略直接影响检索命中率，过细则语义割裂，过粗则噪声过多。

**小白**：（认真记笔记）"所以分段策略是 RAG 的第一个关键决策点。Dify 支持哪几种分段方式？"

**大师**："三种模式：
1. **自动分段**：Dify 自动判断文档结构，按段落切分。适合结构清晰的 Markdown 和 HTML。
2. **自定义分段**：你设定分段长度（默认 500 tokens）和重叠长度（默认 50 tokens）。重叠就是相邻两段之间共享的那段文本——比如上一段是'产品保修'，下一段是'产品保修期为 2 年'，有 4 个字的重叠。重叠越大，信息丢失越少，但索引也越大。
3. **父子分段**：先按大块分（如 2000 tokens 的'父段'），再在每个大块内按小块分（如 500 tokens 的'子段'）。检索时先匹配子段，返回结果时把整个父段带上。这样既保证了检索精度（小块匹配），又保留了完整上下文（大块返回）。"

**技术映射**：父子分段 = 小粒度索引 + 大粒度返回，平衡检索精度和上下文完整性。

**小胖**："那 Embedding 模型又是什么？我选了一个叫 text-embedding-3-small 的，它会把我的文档变成啥？"

**大师**："Embedding 模型把文本转成向量——一个浮点数组，比如 `[0.023, -0.145, 0.891, ...]`。语义相近的文本，其向量在空间中的距离也近。'苹果很好吃'和'这个苹果真甜'的向量夹角很小，但和'苹果发布了 iPhone 16'的向量夹角就很大。Dify 用这个向量来计算用户问题和文档段落之间的语义相似度，得分最高的就被认为是'最相关的'。"

**小白**："那为啥有时候向量检索不准？比如我问'怎么退换货'，结果给我返回了'如何下单'。"

**大师**："因为向量检索只看语义相似度，不看精确关键词。'退换货'和'下单'在语义上都是和'购物流程'相关的，但显然不是同一个问题。这就是为什么 Dify 还提供了**混合检索**——向量检索（语义）+ 关键词检索（BM25，精确匹配），两种结果加权融合。你可以通过调整权重来决定偏重哪个。"

**小胖**："还有 Re-rank 呢？"

**大师**："Re-rank 是'二次筛选'。假设检索返回了 10 个段落，前 3 个最相关。但有时候第 4-10 名里也有好东西，被不准确的向量检索给排到后面了。Re-rank 模型会把这 10 个段落都和问题重新对比一遍，用更强的语义模型重新打分排序，然后只返回 Top 3。代价是需要额外调用一次 Re-rank 模型（增加约 200ms 延迟）。"

**技术映射**：检索流程 = 粗筛（向量/关键词）→ 精排（Re-rank）→ 注入 Prompt（`{{#context#}}`）。

## 3. 项目实战

### 环境准备

| 条件 | 说明 |
|------|------|
| Dify 已部署 | `docker ps` 确认 api、worker、db、redis 运行中 |
| Embedding Provider 已配置 | 第 3 章完成，至少配一个 Embedding 模型（如 OpenAI text-embedding-3-small 或 Ollama BGE-M3） |
| 向量数据库 | Docker Compose 默认带 Weaviate，无需额外配置 |
| 测试文档 | 准备 2-3 个不同类型的文档（PDF、Markdown、TXT） |

### 分步实现

#### 步骤1：创建知识库并上传文档（目标：建立第一个知识库）

1. Dify 控制台 → **知识库** → **创建知识库**
2. 填写信息：
   - 名称：`产品知识库`
   - 描述：`包含产品手册、FAQ、技术规格`
3. **上传文档**：拖拽或选择文件，支持批量上传

```bash
# 准备测试文档（一个简单的 Markdown 文件）
@'
# Acme 产品知识库

## 产品保修政策
Acme 所有硬件产品享受 2 年免费保修服务。
保修范围包括：制造缺陷、材料缺陷、正常使用下的故障。

## 退换货流程
1. 在购买后 30 天内可申请退换货
2. 联系客服获取 RMA 编号
3. 将产品寄回指定地址
4. 退款在收到产品后 5 个工作日内处理

## 常见问题
Q: 产品支持哪些操作系统？
A: 支持 Windows 10+, macOS 12+, Ubuntu 20.04+
'@ | Set-Content -Path "product_faq.md" -Encoding UTF8
```

#### 步骤2：选择分段策略并观察效果（目标：理解分段对检索的影响）

**策略 A：自定义分段（500 tokens，50 重叠）**

在文档列表中点击文档名称 → 分段设置：

```
分段方式：自定义
分段最大长度：500 tokens
分段重叠长度：50 tokens
```

点击**预览分段**，你将看到文档被拆分后的样子：

```
Chunk 1: "Acme 产品知识库 / 产品保修政策 / Acme 所有硬件产品享受 2 年免费保修服务..."
Chunk 2: "...保修服务。保修范围包括：制造缺陷、材料缺陷、正常使用下的故障。退换货流程..."
Chunk 3: "...退换货流程 / 1. 在购买后 30 天内可申请退换货 / 2. 联系客服获取 RMA 编号..."
```

**策略 B：父子分段（大段 2000，小段 400）**

切换分段方式为**父子分段**：

```
父段最大长度：2000 tokens
子段最大长度：400 tokens
```

此时每个子段更小（检索更精准），但返回结果时会带上整个父段的完整上下文。

**关键对比**：

| 分段策略 | 检索精度 | 上下文完整性 | 索引大小 | 适用场景 |
|---------|---------|------------|---------|---------|
| 自定义 500 | 中 | 中 | 中 | 通用场景 |
| 自定义 1000 | 低 | 高 | 小 | 长文档综述 |
| 父子分段 | 高 | 高 | 大 | 精细问答 |
| 自动分段 | 取决于文档结构 | 取决于文档结构 | 取决于文档 | 结构化文档 |

#### 步骤3：配置 Embedding 模型并索引（目标：将文档向量化存储）

1. 在知识库设置中选择 Embedding 模型
2. 确认向量数据库连接正常（Docker Compose 默认已配好 Weaviate）
3. 点击**保存并处理**，Dify 会启动 Celery 任务执行索引

```bash
# 监控索引进度
docker logs docker-worker-1 --tail 20 -f

# 预期日志：
# [INFO] Start indexing document: product_faq.md
# [INFO] Splitting document into 5 chunks
# [INFO] Generating embeddings for 5 chunks
# [INFO] Storing 5 embeddings to Weaviate
# [INFO] Document indexing completed: product_faq.md
```

**常见坑**：
- 索引一直 pending：Celery Worker 未启动，`docker ps` 确认 worker 状态
- "Too many requests"：Embedding API 的 RPM 限制，等待自动重试
- "Dimension mismatch"：Embedding 模型输出维度与向量数据库配置不匹配

#### 步骤4：配置检索策略并测试（目标：找到最佳检索模式组合）

在知识库设置 → **检索设置** 中配置：

```
检索方式：混合检索
权重设置：向量检索 0.6 | 关键词检索 0.4
Top-K：3（每次返回最相关的 3 个段落）
Score 阈值：0.5（低于 0.5 的结果丢弃）

Re-rank：开启
Re-rank 模型：（需额外配置 Cohere 或本地 Rerank 模型）
```

**召回测试**：在知识库详情页的**召回测试**面板中输入查询，观察返回结果：

```
查询："保修期多久"
─────────────────────────
结果 1 [Score: 0.92]: "Acme 所有硬件产品享受 2 年免费保修服务。保修范围包括..."
结果 2 [Score: 0.78]: "退换货流程 / 1. 在购买后 30 天内可申请退换货..."
结果 3 [Score: 0.45]: "常见问题 Q: 产品支持哪些操作系统？"  ← 低于阈值，会被丢弃
```

**调优口诀**：
- 检索结果不相关 → 降低 Score 阈值或切换 Embedding 模型
- 检索结果漏掉 → 增加 Top-K 或尝试混合检索
- 检索结果太泛 → 开启 Re-rank
- Token 消耗太高 → 减小分段长度或减少 Top-K

#### 步骤5：在 Chat App 中使用知识库（目标：端到端验证 RAG 效果）

回到第 4 章创建的 Chat App（或新建一个）：

1. **上下文** → **添加知识库** → 选择"产品知识库"
2. 编辑 System Prompt：

```text
你是 Acme 公司的产品顾问，请基于知识库中的信息回答问题。

## 参考信息
{{#context#}}

## 回答规则
- 优先使用知识库信息，如果没有找到相关信息，诚实告知
- 如需引用文档中的具体内容，请注明出处
- 回答简洁专业，控制在 150 字以内
```

3. 在预览窗口测试：

```
用户：保修期多久？
AI：根据产品知识库，Acme 所有硬件产品享受 2 年免费保修服务，
保修范围包括制造缺陷、材料缺陷和正常使用情况下的故障。

用户：能送货到拉萨吗？
AI：知识库中未找到关于配送范围的明确信息。建议您联系客服或查阅
官网的配送政策页面获取最新信息。
```

## 4. 项目总结

### 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **格式支持** | 支持 PDF/Word/Excel/Markdown/HTML/Notion/CSV 等 20+ 格式 | 扫描版 PDF（图片型）需 OCR 处理，Dify 本身不包含 OCR 引擎 |
| **分段灵活性** | 自动/自定义/父子三种模式，覆盖多数场景 | 分段参数调优依赖经验，缺乏自动最优参数推荐 |
| **检索多样性** | 关键词、向量、混合三种检索 + Re-rank 精排 | 混合检索的权重是经验值，缺乏 A/B 测试工具 |
| **向量数据库** | 支持 Weaviate/Qdrant/Milvus/pgvector/Elasticsearch 等 30+ 方案 | 切换向量数据库需要重建索引，成本较高 |
| **RAG 管线可视化** | 能在控制台直观看到分段、检索、重排的效果 | 缺乏端到端的 RAG 评估指标（如 Hit Rate、MRR），需要自行评估 |

### 适用场景

| 场景 | 说明 |
|------|------|
| **产品文档问答** | 将产品手册、API 文档上传，构建智能技术支持助手 |
| **企业合规查询** | 法规/制度/政策文档，需要精确引用原文 |
| **客服知识库** | FAQ、操作指南、排错手册的智能检索 |
| **技术文档搜索** | 开发者文档、Release Notes 的自然语言查询 |
| **教育内容问答** | 教材、课件、考试大纲的智能问答 |

**不适用场景**：
- 实时数据的问答（如"当前股价"），因为知识库是静态的
- 需要 SQL 精确查询的结构化数据（如"今年销售额 TOP 10 的产品"），请用 HTTP 节点 + 数据库 API

### 注意事项

1. **Embedding 模型选择**：不要只盯价格——OpenAI text-embedding-3-small 每 1000 tokens 仅 $0.00002，但 BGE-M3 在中文场景效果更好。建议做 A/B 测试
2. **向量数据库容量**：100 万条向量 × 1536 维 × 4 字节 ≈ 6GB 存储，大规模知识库需做好容量规划
3. **分段长陷阱**：如果文档包含表格，表头和内容可能被切到不同段中，建议对表格密集型文档用更长的分段长度
4. **检索权重平衡**：法律/合规场景应偏重关键词检索（权重 0.6），避免语义检索"近似匹配"导致的误解

### 常见踩坑经验

1. **坑：上传 PDF 后文本全乱码** → 根因：PDF 是扫描件/图片，没有嵌入文本层。解决：先用 OCR 工具（如 Tesseract）提取文字为 Markdown 后再上传
2. **坑：检索结果总是不相关** → 根因排查步骤：检查分段是否合理（预览分段结果）→ 检查 Embedding 模型是否支持中文（如果文档是中文）→ 开启混合检索并调整关键词权重 → 尝试 Re-rank
3. **坑：知识库更新后问答结果没变** → 根因：文档重新上传后需要手动触发重新索引，系统不会自动感知旧文档的变更。解决：在知识库中点击文档的"重新索引"

### 思考题

1. **进阶题**：在 Dify 的 RAG 管线中，如果要实现"实时数据问答"（比如查询最新的天气、股价），你应该在哪个环节接入外部数据？（提示：思考 RAG 和 Function Calling 的分工）

2. **进阶题**：假设你的知识库有 100 万条文档，单次检索延迟 500ms，用户感受太慢。你能想到哪些优化方案？（提示：从分段策略、索引结构、缓存、异步预取等角度思考）
