# 第16章：【基础篇综合实战】搭建企业级智能客服系统

## 1. 项目背景

宏达电商是一家年销售额 50 亿的中型电商企业，每天客服团队接到约 2000 通客户咨询。这 2000 通咨询里，有 60% 是高频重复问题——"我的订单到哪了？""怎么退换货？""这款产品有优惠吗？"——客服人员每天回答同样的内容上百次，离职率高达 40%。CTO 拍板：用 Dify 搭建一套智能客服系统，覆盖售前咨询、订单查询、售后投诉三个核心场景。

但需求方提的要求比想象中复杂："FAQ 问答要基于我们的产品知识库，不能瞎编""用户问订单状态要能实时查询，不能只说'请稍等'""退款投诉这种复杂场景要能自动生成工单""聊天界面要嵌在我们官网里，不能跳到第三方页面"。如果只用 Chat App，无法查订单；只用 Workflow，无法处理开放式投诉——需要用 Dify 的三种应用模式组合完成。

更关键的是，从知识库架构设计到 Prompt 调优，从 Workflow 流程编排到 Agent 策略配置，需要把基础篇 15 章的知识全部用上。这不再是一个"Hello World"教程，而是一个真正可上线的、给客户用的智能客服系统。

## 2. 项目设计——剧本式交锋对话

**小胖**：（看着需求文档发愁）"大师，这次真不是小打小闹了——售前 FAQ、订单查询、投诉处理，三个场景三种要求，我得用三个不同的 App？那用户不是得在三个聊天窗口之间跳来跳去？"

**大师**："你说到点子上了。所以你需要一个'路由层'——用户进入聊天窗口后，先通过一个意图识别 Workflow 判断用户想问什么：问产品参数 → 转给 Chat App 处理；问订单状态 → 转给订单查询 Workflow；投诉退款 → 转给投诉处理 Agent。说白了，你需要一个总入口。"

**技术映射**：多 App 协同 = 意图识别（路由）+ 子 App 调用（分发），实现"一个入口，多个能力"。

**小白**：（在白板上画知识库架构）"知识库怎么规划？是把所有文档堆到一个库，还是分多个库？我担心全堆一起检索精度会很差，分开又怕用户问的问题跨了知识域。"

**大师**："一个总库就够了，但要分层管理内容。我的建议是：

1. **产品知识层**：所有在售商品的参数、价格、库存说明——这是回答'这款手机内存多大'的基础
2. **政策文档层**：退换货政策、保修条款、配送规则——这是回答'能退货吗'的依据
3. **FAQ 高频层**：前人总结的 TOP 100 常见问答——这是最快的匹配库
4. **操作指南层**：如何下单、如何查物流、如何开发票——这是'怎么操作'的答案"

"但关键不在分多少层，而在分段策略。政策文档建议用父子分段——大段保证上下文完整（比如退换货政策的一大段被完整保留），小段保证检索精度。"

**技术映射**：知识库架构 = 内容分层 + 分段策略定制，不同内容类型用不同策略。

**小胖**："那订单查询场景，客户报个订单号，我去查公司内部的订单 API。但我担心 API 挂了怎么办？或者用户报的订单号格式不对？"

**大师**："问得好。这就是 Workflow 比 Chat App 强的地方——它有 IF/ELSE 节点做错误分支。你的订单查询 Workflow 应该是这样的：

开始（输入订单号）→ 格式校验（Code 节点检查是不是纯数字）→ 不合法？→ LLM 提示'请确认订单号格式'。合法？→ HTTP 请求调订单 API → 成功(200)？→ LLM 把 JSON 转成人类语言 → 结束。失败？→ IF/ELSE 按错误码分支：401 是认证问题（报修），404 是订单不存在（引导用户核对），500 是系统故障（道歉+转人工）。"

**小白**："那 Agent 处理投诉呢？我担心它'过度自由'——搜了一圈没找到答案就开始编造。"

**大师**："Agent 约束三件套：
1. **严格的系统提示词**：明确告诉它'如果 3 步内没有进展，必须建议转人工'，不要让它无限循环
2. **工具结果校验**：在 Agent 调用订单查询工具后，用 Code 节点检查返回结果是否有效，无效则终止 Agent 思考
3. **最大迭代限制**：投诉场景设 8 步，第 7 步时强制引导它输出最终答案"

**技术映射**：Agent 约束 = 迭代上限 + 强制转向 + 结果校验，防止"自由过了火"。

**小胖**："还有个问题——用户在这个统一的聊天窗口里，能不能跨场景？比如先问了产品参数，接着又问'我的订单到哪了'，这时上下文不会乱吧？"

**大师**："这是会话级变量的用场。当 Chat App 回答完 FAQ 后，如果用户追问订单问题，意图识别 Workflow 可以读取上一个对话的 conversation 变量——比如 '上一次识别到的用户意图'——来做更准确的二次判断。但要注意，Chat App 和 Agent 的 conversation_id 是可能不同的，你需要在中间加一个 conversation 桥接层。"

**技术映射**：跨场景会话 = conversation_id 传递 + 对话变量共享，多 App 间的上下文保持。

#### 步骤1：构建商品知识库（目标：建立 RAG 数据基础）

```bash

│   └── 配送说明.md          # 覆盖区域、运费、时效
├── faq/
│   └── top100_faq.md        # 前人总结的高频问答

在 Dify 控制台中：
1. 知识库 → 创建 → 命名"宏达电商知识库"
2. 上传以上所有文档
3. 分段策略配置：

```

#### 步骤3：Workflow——订单查询（目标：实时查询订单状态）

创建 Workflow → 命名"订单查询流水线"：

```
开始（输入变量：order_id, api_token）
    ↓
Code_格式校验
    Python代码：
    def main(order_id: str):
        import re
        if re.match(r'^\d{10}$', order_id):
            return {"valid": True, "order_id": order_id}
        return {"valid": False, "message": "订单号应为10位数字"}
    
    ↓
IF/ELSE（条件：{{#code_valid.valid#}} == true）
    ├── [True] → HTTP_查询订单
    │   URL: https://order-api.internal.acme.com/api/v1/orders/{{#code_valid.order_id#}}
    │   Headers: Authorization: Bearer {{#start.api_token#}}
    │   超时: 15秒
    │   
    │   ↓
    │   IF/ELSE（条件：{{#http_order.status_code#}}）
    │   ├── 200 → LLM_格式化订单信息
    │   │   System Prompt:
    │   │   将以下订单 JSON 转换为用户友好的信息：
    │   │   {{#http_order.body#}}
    │   │   提取：订单号、下单时间、商品名称、金额、物流状态、物流单号
    │   │   格式：📦 订单XXX ｜⏰ 时间 ｜💰 金额 ｜🚚 状态
    │   │   
    │   ├── 404 → LLM_订单不存在
    │   │   "未找到订单{{#code_valid.order_id#}}，
    │   │    请核对订单号后重新输入。"
    │   │   
    │   └── ELSE → LLM_系统异常
    │       "系统繁忙，请稍后重试或拨打 400-888-9999 转人工。"
    │
    └── [False] → LLM_格式错误
        "订单号应为 10 位数字，请核对后重新输入。示例：20260511001"
    ↓
结束（分支汇聚，输出最终回复）
```

#### 步骤4：Agent——投诉处理（目标：自动生成售后工单）

创建 Agent App → 命名"宏达投诉处理专员"：

```text
System Prompt：
你是宏达电商的高级售后专员。面对客户的投诉，你需要冷静、共情、高效地处理。

## 核心流程
1. 先安抚情绪：表达歉意，让客户感到被重视
2. 获取信息：询问订单号，调用 get_order_info 工具
3. 诊断问题：根据工具返回的订单信息和 check_warranty 结果判断
4. 给出方案：
   - 质量问题且在保修期 → 生成退换货工单（create_ticket）
   - 物流问题 → 提供物流查询链接并致歉
   - 价格争议 → 检查是否在价保期内
5. 明确下一步：告诉客户接下来会发生什么（工单号、处理时长、联系方式）

## 工具清单
- get_order_info: 输入订单号，返回订单详情
- check_warranty: 输入订单号+商品ID，返回保修状态
- create_ticket: 输入工单类型+描述，返回工单号
- search_knowledge: 搜索退换货政策

## 约束
- 最大迭代：8 步
- 连续 2 步没有进展 → 主动建议转接人工客服
- 最终回复包含：工单号 + 预计处理时间 + 客服热线
- 回复语气：真诚道歉 + 不推卸责任 + 给出具体行动
```

工具配置：添加 get_order_info（自定义 API 工具）、search_knowledge（知识库检索工具）、create_ticket（自定义 API 工具）

#### 步骤5：前端集成与发布（目标：嵌入官网）

1. 为三个 App 分别创建 API Key
2. 编写前端中间层——意图识别路由：

```python
# intent_router.py（简化后端路由服务）
from flask import Flask, request
import requests

app = Flask(__name__)
DIFY_FAQ_KEY = "app-faq-xxx"
DIFY_ORDER_KEY = "app-order-xxx"
DIFY_AGENT_KEY = "app-agent-xxx"

def detect_intent(query: str) -> str:
    """简易意图识别：基于关键词规则"""
    order_keywords = ['订单', '物流', '到哪了', '发货', '快递']
    complaint_keywords = ['投诉', '退款', '退货', '坏了', '假的', '差评']
    
    if any(k in query for k in complaint_keywords):
        return 'agent'
    elif any(k in query for k in order_keywords):
        return 'order'
    else:
        return 'faq'

@app.route('/chat', methods=['POST'])
def chat():
    query = request.json['query']
    intent = detect_intent(query)
    
    key_map = {'faq': DIFY_FAQ_KEY, 'order': DIFY_ORDER_KEY, 'agent': DIFY_AGENT_KEY}
    
    resp = requests.post(f"http://localhost/v1/chat-messages",
        headers={"Authorization": f"Bearer {key_map[intent]}"},
        json={"query": query, "user": request.json['user'], "response_mode": "blocking"}
    )
    return resp.json()
```

## 4. 项目总结

### 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **场景覆盖** | Chat+Workflow+Agent 三模式组合，覆盖 90% 客服场景 | 三种模式需要维护三套配置，运营成本较高 |
| **知识库** | 分层管理 + 分段策略定制，检索精度高 | 文档更新后需手动触发重新索引，不够自动化 |
| **错误处理** | Workflow 中 IF/ELSE 全覆盖了各 HTTP 状态码场景 | 复杂错误分支增多后画布可读性下降 |
| **Agent 约束** | 迭代上限 + 强制转向防循环 | 刚性约束可能影响 Agent 解决复杂问题的灵活性 |
| **前端集成** | iframe + 意图路由实现统一入口 | 意图识别基于关键词规则，复杂语义可能误判 |

### 适用场景

| 场景 | 推荐方案 |
|------|---------|
| **纯售前咨询** | Chat App + 知识库，快速上线 |
| **订单物流查询** | Workflow + HTTP 节点，实时数据 |
| **售后投诉** | Agent + 多工具，自动化处理 |
| **混合场景** | 意图路由 + 三个 App 组合 |
| **内部员工帮助台** | Chat App + 内部文档知识库 |

**不适用场景**：
- 涉及支付的敏感操作（需要支付密码/验证码，安全合规问题）
- 紧急求助（如安全事故），必须直连人工

### 注意事项

1. **意图路由精度**：关键词匹配在复杂场景下容易误判，建议后续升级为 LLM 意图识别
2. **多 App Token 成本**：一次投诉可能先经 Agent、再调 API、再创工单，Token 消耗不可忽视
3. **会话延续**：不同 App 的 conversation_id 不同，需要额外维护用户级会话映射

### 常见踩坑经验

1. **坑：意图路由总是误判为 FAQ** → 根因：关键词列表不完整，用户说"订单有问题"触发不了 order 关键词。解决：增加模糊匹配或升级为 LLM 分类
2. **坑：Agent 生成工单后仍继续思考** → 根因：没有在 System Prompt 中写"工单创建成功后立即结束任务"。解决：在 create_ticket 工具的 description 中加入"完成此操作后直接输出工单号并结束"
3. **坑：用户投诉时情绪激动但 Agent 回复冷淡** → 根因：System Prompt 中强调了"效率"而忽略了"共情"。解决：重新平衡 Prompt，加示例

### 思考题

1. **进阶题**：如果每月新增 1000 份产品文档，知识库的维护成本如何控制？请设计一套"文档变更自动同步"的方案。（提示：考虑 Webhook + Celery 定时任务 + 增量索引）

2. **进阶题**：如果意图路由的准确率从当前的 85% 提升到 98%，你会在哪些环节做改进？（提示：从关键词规则 → LLM 分类 → 上下文辅助判断）
