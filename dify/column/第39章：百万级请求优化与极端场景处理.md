# 第39章：百万级请求优化与极端场景处理

## 1. 项目背景

第 29 章我们从单机 20 QPS 优化到了 500 QPS。但如果你的 Dify 需要支撑日活 100 万用户、峰值 10000 QPS——单机优化已经到天花板了。你需要从**架构层面**重新设计。

本章围绕五个极端场景给出生产级解决方案：

- **场景一：Message 表 5000 万行**，查询越来越慢，单表扛不住了
- **场景二：知识库 100 万篇文档**，全量索引耗时 3 天，增量更新怎么做？
- **场景三：10000 QPS 并发**，单实例 500 QPS 不够用
- **场景四：LLM Provider 全量限流**，所有请求返回 429
- **场景五：某个 AZ 机房断电**，服务不能中断

每个场景不仅有代码方案，更有背后的架构思想。

## 2. 项目设计——剧本式交锋对话

**小胖**：（看着数据库慢查询日志）"大师！Message 表突破 5000 万行，`SELECT * FROM messages WHERE conversation_id = 'xxx' ORDER BY created_at DESC LIMIT 20` 跑了 8 秒。索引已经加了，还是慢。单表扛不住了。"

**大师**："索引能优化查询，但不能缩小扫描范围。你要的是**表分区**。PostgreSQL 的声明式分区——按 `created_at` 每月一个分区。查询"最近 7 天消息"时，PostgreSQL 的查询规划器会自动识别——只需要扫描当前月分区，可能还有上个月的（如果跨月）。其他 48 个分区完全不碰。扫描量缩小 50 倍，查询时间从 8 秒降到 100ms。"

**技术映射**：表分区 = 按业务规则把大表拆成多个物理小表，但保持逻辑上的单一表名。

**小白**："那 100 万篇知识库文档呢？全量重建索引要 3 天。"

**大师**："**增量索引**。加一个 `last_modified_at` 字段。每次重建索引时，不是把所有文档重新处理——而是只处理 `last_modified_at > last_indexed_at` 的文档。Dify 原生的 Document 模型有这个字段，但你需要在定时任务中编写增量索引的逻辑。"

**小胖**："10000 QPS 怎么办？单实例撑死 500 QPS。"

**大师**："**水平扩展 + 消除共享瓶颈**。API 和 Worker 是无状态的——直接加 Pod。瓶颈会转移到 PostgreSQL（所有 Pod 共享同一个 DB）和 Redis（所有 Pod 共享同一个 Redis）。PostgreSQL 用读写分离（Pgpool-II 或云厂商的 Read Replica），Redis 用 Cluster 模式分散 Key。"

**小白**："LLM 全量 429——所有 Key 都被限流了，用户怎么办？"

**大师**："**熔断器（Circuit Breaker）+ 优雅降级**。设连续 5 次 429 后熔断 60 秒——请求不再发给 LLM API，直接返回'系统繁忙，请稍后重试'。60 秒后半开状态尝试恢复。同时启动**降级策略**——切换到更便宜的模型（GPT-3.5 代替 GPT-4）或使用本地缓存的热门回答。"

-- 创建分区表（保留原表名，应用层无感知）
CREATE TABLE messages (
    id UUID,
    conversation_id UUID,
    query TEXT,
    answer TEXT,
    created_at TIMESTAMP NOT NULL
) PARTITION BY RANGE (created_at);

-- 创建 2026 年 12 个月的分区
CREATE TABLE messages_2026_01 PARTITION OF messages
    FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');
CREATE TABLE messages_2026_02 PARTITION OF messages
    FOR VALUES FROM ('2026-02-01') TO ('2026-03-01');
-- ... 到 messages_2026_12

-- 创建默认分区（处理分区外的数据，防止插入失败）
CREATE TABLE messages_default PARTITION OF messages DEFAULT;

-- ★ 查询自动路由
EXPLAIN SELECT * FROM messages WHERE created_at BETWEEN '2026-01-15' AND '2026-01-16';
-- 输出: Append → Subplans Removed: 11
-- 只扫描 messages_2026_01 分区

-- 每月自动创建下月分区（定时任务）
-- crontab: 0 0 25 * * 每月 25 号运行
INSERT INTO partition_jobs VALUES (date_trunc('month', now() + interval '2 month'));

### 场景 2：LLM 熔断器

```python
from enum import Enum
import time

class CircuitState(Enum):
    CLOSED = "closed"          # 正常通行
    OPEN = "open"              # 熔断中，拒绝请求
    HALF_OPEN = "half_open"   # 试探恢复

class LLMCircuitBreaker:
    """
    调用方式: cb = LLMCircuitBreaker()
             cb.call(lambda: model.invoke(prompt))
    """
    def __init__(self, failure_threshold=5, recovery_timeout=60):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout  # 熔断后 60 秒尝试恢复
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.last_failure_time = 0
    
    def call(self, fn, *args, **kwargs):
        # === 如果熔断中 ===
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN  # 尝试恢复
                print("熔断器进入半开状态，尝试恢复...")
            else:
                raise CircuitBreakerOpenError(
                    f"LLM 调用已熔断，请 {int(self.recovery_timeout - (time.time() - self.last_failure_time))} 秒后重试"
                )
        
        # === 执行调用 ===
        try:
            result = fn(*args, **kwargs)
            
            # 半开状态下恢复成功 → 关闭熔断
            if self.state == CircuitState.HALF_OPEN:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
                print("熔断器恢复成功，已关闭")
            
            return result
        
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()
            
            if self.failure_count >= self.failure_threshold:
                self.state = CircuitState.OPEN
                print(f"熔断器触发！连续 {self.failure_count} 次失败")
            
            raise e  # 仍然抛出原始异常

# 使用示例
circuit_breaker = LLMCircuitBreaker(failure_threshold=5, recovery_timeout=60)

def call_llm(prompt):
    return circuit_breaker.call(
        lambda: requests.post("https://api.openai.com/v1/chat/completions", ...)
    )
```

### 场景 3：多级缓存

```python
from functools import lru_cache
import redis
import json

class MultiLevelCache:
    """
    L1: 本地 Python LRU（最快，但 Pod 间不共享）
    L2: Redis（快，跨 Pod 共享）
    L3: PostgreSQL（慢，最终数据源）
    """
    def __init__(self):
        self.redis = redis.Redis()
    
    @lru_cache(maxsize=1000)
    def _get_from_db(self, key: str) -> dict:
        """L3: 数据库查询（慢但最终正确）"""
        return db_query(key)
    
    def get(self, key: str) -> dict:
        # L1: 进程内内存缓存
        local = self._get_from_local(key)
        if local:
            return local
        
        # L2: Redis
        cached = self.redis.get(key)
        if cached:
            data = json.loads(cached)
            self._set_to_local(key, data)  # 回填 L1
            return data
        
        # L3: PostgreSQL（miss 才查）
        data = self._get_from_db(key)
        self.redis.setex(key, 300, json.dumps(data))  # 回填 L2, TTL 5min
        self._set_to_local(key, data)                 # 回填 L1
        return data
    
    def invalidate(self, key: str):
        """数据更新后，主动使所有缓存失效"""
        self._del_from_local(key)
        self.redis.delete(key)
```

### 场景 4：跨 AZ 容灾

```python
class MultiRegionRedis:
    """
    多区域 Redis：主节点故障自动切换到备节点
    """
    def __init__(self, primary_url: str, standby_urls: list[str]):
        self.primary = redis.Redis.from_url(primary_url)
        self.standbys = [redis.Redis.from_url(u) for u in standby_urls]
        self.current = self.primary
    
    def execute(self, command: str, *args, **kwargs):
        """执行 Redis 命令，主节点故障自动切换"""
        try:
            return getattr(self.current, command)(*args, **kwargs)
        except redis.ConnectionError:
            # 尝试切换到备节点
            for standby in self.standbys:
                try:
                    standby.ping()
                    self.current = standby
                    print(f"Redis 主节点故障，已切换到备节点")
                    return getattr(self.current, command)(*args, **kwargs)
                except redis.ConnectionError:
                    continue
            
            raise AllRedisNodesFailedError("所有 Redis 节点不可用")
```

## 4. 项目总结

| 极端场景 | 技术方案 | 核心思想 |
|---------|---------|---------|
| 大表 5000 万行 | PostgreSQL 表分区 | 缩小扫描范围 |
| 知识库 100 万篇 | 增量索引 | 只处理变更 |
| 10000 QPS | K8s HPA + 读写分离 | 水平扩展消除瓶颈 |
| LLM 全量 429 | 熔断器 + 降级 | 快速失败 + 优雅降级 |
| AZ 断电 | 多 AZ 部署 + 主备切换 | 单点故障无影响 |
