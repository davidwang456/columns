# 第26章：工具系统全解析——从内置工具到自定义插件

## 1. 项目背景

Agent 的"超能力"不是推理能力，而是**工具调用能力**——没有工具的 Agent 只是一个能说会道的 Chat。Dify 的工具生态从简单到复杂分为五个层次：内置工具（50+ 个，直接可用）、自定义 API 工具（写 YAML 即可）、插件工具（Marketplace 安装）、MCP 工具（Model Context Protocol 标准）、Workflow-as-Tool（将一个 Workflow 发布为工具）。

但本章的价值不在于"有哪些工具可用"，而在于**工具是如何被注册、发现和调用的**。当 Agent 决定"我需要搜索"时，Dify 在后台做了什么？工具管理器（ToolManager）如何收集所有可用的工具？工具的 `description` 如何影响 Agent 的选择准确度？工具调用的错误如何反馈给 Agent 让它自我修正？

理解这套机制有两个实际价值：**封装内部 API 时知道应该写 OpenAPI YAML 还是开发插件**；**Agent 调错工具时知道是工具描述写得太模糊还是参数定义有问题**。

## 2. 项目设计——剧本式交锋对话

**小胖**：（看着 Dify 的工具面板）"大师！Agent 可用的工具有 50 多个——搜索、计算、画图、读网页、操作 Jira……但我想加一个自己公司的 CRM 查询工具，怎么搞？这 50 个没有一个是能查我们内部系统的。"

**大师**："三种方式，从简到难排序。**① OpenAPI Schema 工具**——写一个 YAML 描述你 CRM API 的 URL、参数、认证方式，粘贴到 Dify 里，自动生成工具。5 分钟搞定，适合大多数 REST API。**② Workflow-as-Tool**——如果你已经有一个'客户查询'的 Workflow，在画布上点'发布为工具'，Agent 就能把它当成一个工具来调。**③ 插件开发**——如果你需要复杂的认证逻辑（如 OAuth2）、或需要自定义代码逻辑，开发一个完整的插件。"

**技术映射**：工具集成方式 = OpenAPI YAML（HTTP 层）< Workflow-as-Tool（编排层）< Plugin（代码层）。复杂度依次递增。

**小白**："OpenAPI YAML 里最关键的几个配置是什么？"

**大师**："三个字段决定工具好不好用：
- **`operationId`**（如 `searchCustomer`）：Agent 用它来区分"我可以调用 searchCustomer 这个操作"。如果 Agent 想'查客户'，它会扫描所有工具的 operationId 和 description，找最匹配的。
- **`description`**（工具描述）：这是 Agent '决定用不用这个工具'的主要依据。如果你写'查询客户信息'——太模糊，Agent 不确定什么时候用。如果你写'根据客户姓名或手机号查询该客户的会员等级、购买历史和优惠券余额'——精确，Agent 能准确判断'这个工具可以解答用户关于会员和优惠券的问题'。
- **`parameters`**（参数定义）：Agent 会尝试从用户对话中提取参数值。如果你的参数叫 `cust_name` 但 description 写的是'客户标识'——Agent 不知道从'查张三的信息'里提取'张三'。如果 description 写的是'客户姓名，如 张三'——Agent 知道了。"

**技术映射**：工具选择准确度 = description 清晰度。描述越精确，Agent 越能正确选择工具。

**小胖**："那工具调用的错误怎么处理？比如 CRM API 挂了返回 500，Agent 会怎么办？"

**大师**："取决于错误类型和 Agent 的心智。Dify 的工具管理器会捕获异常，包装成自然语言返回给 Agent。比如 API 返回 500，Agent 会收到'工具执行失败：HTTP 500 Internal Server Error'。然后 Agent 根据 System Prompt 中的错误处理规则来决定——'向用户道歉并建议稍后重试'或'换个方式再试'或'转人工'。"

## 3. 项目实战

### 环境准备

| 条件 | 说明 |
|------|------|
| Dify 已部署 | Agent App 可运行 |
| 至少一个可测试的 REST API | 用于自定义工具 |

### 分步实现

#### 步骤1：工具基类解剖（目标：理解所有工具的公共接口）

```python
# api/core/tools/__base/base_tool.py（简化）
from abc import ABC, abstractmethod
from pydantic import BaseModel

class ToolParameter(BaseModel):
    """工具输入参数的定义"""
    name: str           # 参数名（如 "city"）
    type: str           # string / number / boolean / select
    required: bool = False
    description: str    # ★ Agent 从对话中提取参数时的依据
    options: list[dict] = []  # select 类型的选项列表

class BaseTool(ABC):
    name: str                    # ★ operationId，全局唯一的工具名
    description: str             # ★ Agent 用这个判断是否调用此工具
    
    @abstractmethod
    def _validate_credentials(self, credentials: dict) -> bool:
        """校验凭据（如 API Key 格式是否正确）"""
        pass
    
    @abstractmethod
    def _invoke(self, user_id: str, tool_parameters: dict) -> str:
        """★ 核心执行逻辑。返回自然语言字符串而非 JSON"""
        pass
    
    @abstractmethod
    def get_parameters(self) -> list[ToolParameter]:
        """定义工具的输入参数 Schema"""
        pass
```

**关键设计**：`_invoke` 方法的返回值是**自然语言字符串**而非 JSON——因为 Agent 要能读懂工具返回了什么。返回原始 JSON 会给 Agent 增加额外的解析负担。

#### 步骤2：创建自定义 API 工具（目标：封装公司 CRM 查询接口）

在 Dify 控制台 → **工具** → **创建自定义工具** → 粘贴以下 OpenAPI YAML：

```yaml
openapi: 3.0.0
info:
  title: 宏达 CRM 客户查询工具
  version: 1.0.0
  description: 用于查询宏达电商平台客户的会员信息
servers:
  - url: https://crm.internal.acme.com/api
paths:
  /v1/customers/search:
    get:
      operationId: searchCustomer
      summary: 根据姓名或手机号查询客户信息
      description: |
        查询客户的会员等级、购买历史、优惠券余额。
        至少填写 name 或 phone 其中一个。
        返回的 customers 字段是匹配的客户列表。
      parameters:
        - name: name
          in: query
          required: false
          schema:
            type: string
          description: 客户姓名（支持模糊匹配），如"张三"
        - name: phone
          in: query
          required: false
          schema:
            type: string
          description: 客户手机号（精确匹配），如"13812345678"
        - name: api_token
          in: header
          required: true
          schema:
            type: string
          description: CRM API 认证 Token（平台统一配置，无需从对话提取）
      responses:
        '200':
          description: 客户信息列表
          content:
            application/json:
              schema:
                type: object
                properties:
                  customers:
                    type: array
                    items:
                      type: object
                      properties:
                        id:
                          type: string
                          description: 客户ID
                        name:
                          type: string
                          description: 客户姓名
                        phone:
                          type: string
                          description: 手机号
                        level:
                          type: string
                          description: 会员等级（普通/银卡/金卡/钻石）
                        total_purchases:
                          type: number
                          description: 累计消费金额（元）
                        coupon_balance:
                          type: number
                          description: 可用优惠券余额
```

**关键配置细节**：
- `description` 中写了详细的业务含义——帮助 Agent 判断"用户问会员等级时该调用这个工具"
- `parameters` 中每个字段的 `description` 都给出了示例值——帮助 Agent 从对话中正确提取参数
- `responses.schema` 中每个字段都有 `description`——帮助 Agent 理解返回数据的含义

#### 步骤3：Workflow-as-Tool（目标：复用已有流程）

在 Dify 控制台：
1. 打开已有的"订单查询" Workflow
2. 右上角 **...** → **发布为工具**
3. 填写工具名称：`order_lookup`
4. 填写工具描述：`查询宏达电商平台订单的详情，包括订单状态、商品信息、物流单号。输入为 10 位订单号。`
5. 发布后，在 Agent App 的工具面板中即可勾选此 Workflow 作为工具

```text
调用链：Agent 决定查订单 → Dify 启动 Workflow 引擎执行"订单查询"画布 → 
        Workflow 结果返回给 Agent → Agent 基于结果生成最终回复
```

**关键注意**：Workflow-as-Tool 的输入变量必须和 Workflow 开始节点定义的输入变量一致。否则 Agent 传入的参数名对不上，Workflow 接收不到。

## 4. 项目总结

### 工具系统的五层架构

| 层次 | 方式 | 复杂度 | 灵活性 | 适用场景 |
|------|------|-------|--------|---------|
| 1 | 内置工具（50+） | 零配置 | 低 | 通用场景 |
| 2 | OpenAPI Schema YAML | 低 | 中 | 封装 REST API |
| 3 | Workflow-as-Tool | 低 | 中 | 复用已有流程 |
| 4 | Plugin（Marketplace） | 中 | 高 | 复杂逻辑/第三方贡献 |
| 5 | MCP Tool | 中 | 高 | Model Context Protocol 标准 |

### 工具描述最佳实践

| 字段 | 不好的写法 | 好的写法 |
|------|----------|---------|
| `description` | "查询信息" | "根据客户姓名或手机号查询其会员等级、累计消费金额和可用优惠券余额" |
| 参数 `description` | "参数1" | "客户姓名，支持模糊匹配，如'张三'" |
| 返回 `description` | "返回数据" | "客户的会员等级（普通/银卡/金卡/钻石）" |

### 注意事项

1. **工具描述的长度和精准度**：太短 Agent 不知道怎么用，太长占用 System Prompt 的 Token 配额
2. **工具返回值格式**：返回自然语言字符串，而不是原始 JSON——Agent 更容易理解
3. **工具错误处理**：工具执行失败时，返回有意义的错误信息（如"CRM 系统暂时不可用，请 5 分钟后重试"），而不是抛 TypeError

### 常见踩坑经验

1. **坑：Agent 从不调用我写的工具** → 根因：工具的 description 太笼统，Agent 判断"不相关"。解决：精确描述工具的输入/输出/业务含义
2. **坑：Agent 调用工具时传参错误** → 根因：参数的 description 没有给示例值，Agent 不知如何从对话中提取。解决：每个参数加 `description: "...，如'张三'"`
3. **坑：工具调用成功后 Agent 说"没有找到信息"** → 根因：工具返回的是原始 JSON，Agent 不理解 JSON 结构。解决：在工具服务的返回值中自己先用 LLM 把 JSON 转成自然语言

### 思考题

1. **进阶题**：如果 Agent 同时拥有 5 个功能相近的工具（如"查订单""查物流""查退款"，它们的 description 都很像），Agent 如何精确选择？（提示：考虑在 System Prompt 中加入工具选择的决策树）
