# 第4章：Chat App 入门——构建第一个对话助手

## 1. 项目背景

小周是公司的产品运营，最近被安排了一项任务：搭建一个"新人入职指南"问答助手，让新员工可以通过自然语言询问公司制度、福利、流程等问题，而不是每次都去翻 50 多页的 PDF 员工手册。之前小周试图用 ChatGPT 的 GPTs 功能来做，但发现几个致命问题：不能上传多个 PDF（单文件限制）、不能精确控制检索范围（GPTs 会"编造"不存在的内容）、不能嵌入公司内部的飞书/企业微信、领导不满意数据放在 OpenAI 的服务器上。

Dify 的 Chat App 模式恰好能解决这些痛点。它不像 Completion 模式那样"一锤子买卖"——问一句答一句，没有记忆。Chat 模式自带对话上下文管理，LLM 会记得你们刚才聊了什么。更重要的是，Chat App 可以挂载知识库——用户在对话框中提问时，Dify 会先去你指定的知识库中检索相关内容，拼接到 Prompt 里再送给 LLM，这样回复就有了"事实依据"。

本章的目标是带你从零构建一个真正可用的 Chat App——不只是能聊天，而是能基于知识库回答问题、有多轮对话记忆、能在对话过程中引用变量、发布为公开链接供团队使用。通过这个实战项目，你会理解 System Prompt 和 User Prompt 的角色分工、上下文窗口的工作原理、以及一个 Chat 请求从发出到返回的完整链路。

## 2. 项目设计——剧本式交锋对话

**小胖**：（从工位跳起来）"大师大师！我刚才用 Dify 的 Chat App 和 AI 聊了 10 分钟，它居然记得我前面说过什么！我问'我叫小胖'，过了 5 轮再问'我叫什么'，它还能答出来！这也太神奇了，它是怎么记住的？"

**大师**："你在微信上和别人聊天，怎么知道上下文？翻聊天记录对吧。Dify 也一样，每次你发新消息之前，它会把最近 N 轮的历史对话拼到 Prompt 里一起发给 LLM。LLM 看到了'我叫小胖'这条记录，自然就能回答你的名字。"

**小白**：（皱眉）"但这就带来两个问题：第一，对话太长的话，上下文窗口装不下怎么办？第二，如果对话历史很长，每次请求都重新传一遍，Token 消耗会不会爆炸？"

**大师**："两个问题都问到点子上了。先说窗口限制——假设 GPT-4 的上下文窗口是 128K tokens，如果对话历史 + 知识库检索结果 + System Prompt 总共超过了这个限制，Dify 会自动截断。它保留最近的消息，丢掉最早的。这个策略叫'滑动窗口'。配置里有一个'上下文轮数'参数，调小它就能控制 Token 消耗。"

**技术映射**：对话记忆 = 将历史对话拼入 Prompt，受模型的上下文窗口（Context Window）限制，超出时通过滑动窗口截断。

**小胖**："那 System Prompt 和 User Prompt 又是咋回事？我看 Chat App 配置页里有两个输入框。"

**大师**："System Prompt 是用来定义 AI 的'人设'和规则。比如你可以写：'你是公司的新人入职助手，回答请使用礼貌、专业的语气。如果遇到你不知道的问题，请诚实地引导用户联系 HR。'这段 Prompt 在每次对话中都会被放在最前面，LLM 会把它当作'最高指导思想'来遵守。"

"User Prompt 则是用户的当前问题。Dify 会自动把你在对话框里输入的内容填到 User Prompt 的位置。你可以通过 `{{#query#}}` 变量引用用户的输入，用 `{{#context#}}` 引用知识库检索到的内容。"

**小白**："也就是说，最终送给 LLM 的 Prompt 是：System Prompt + 知识库检索结果 + 历史对话 + 当前用户问题？"

**大师**："完全正确。这个拼接逻辑在 `api/core/app/apps/chat/` 中可以找到源码实现。理解了这个组装过程，你就掌控了 Chat App 的灵魂。"

**技术映射**：Chat Prompt 组装 = System Prompt（角色定义）+ Context（知识库结果）+ History（滑动窗口截取）+ Query（当前问题）。

**小胖**："发布功能是啥意思？我点了'发布'，生成了一个链接，别人点开就能用我的助手了？"

**大师**："对。Dify 为每个 Chat App 提供了一个公开的 WebApp URL——你用 Dify 控制台配好助手，点发布生成链接，别人不需要注册 Dify 账号就能和你助手的 WebApp 页面聊天。而且 Dify 还支持 iframe 嵌入——你可以把这个聊天窗口嵌到你们公司官网的某个角落里。"

**小白**："那这个 Chat App 和 Agent 模式有什么区别？Agent 不是也能聊天吗？"

**大师**："核心区别：Agent 可以主动调用工具（搜索、计算、API），Chat 不会。Chat 是被动回答——你问啥它答啥，最多从知识库里检索一下。Agent 是主动执行——你说'帮我查一下昨天的销售额并生成图表'，它会自己去调用内部的数据 API，拿到数据，调用画图工具生成图表，最后给你结果。Chat 是图书馆管理员，Agent 是你的私人助理。基础篇我们先搞定 Chat，到了第 8 章再玩 Agent。"

## 3. 项目实战

### 环境准备

| 条件 | 说明 |
|------|------|
| Dify 已部署 | 第 2 章完成 |
| LLM Provider 已配置 | 第 3 章完成，至少配一个 LLM（如 GPT-3.5） |
| 测试知识库（可选） | 准备一个 PDF 或 Markdown 文档作为知识源 |

### 分步实现

#### 步骤1：创建 Chat App 并配置基础 Prompt（目标：搭建最基本的对话框架）

1. Dify 首页 → **创建应用** → 选择 **Chat** 模式 → 输入名称"新人入职指南助手"
2. 进入 App 编辑页，左上角选定模型（如 GPT-3.5-turbo）
3. 在 **系统提示词** 输入框中编写：

```text
你是 Acme 公司的新人入职指南助手，名叫"小A"。

## 你的职责
- 回答关于公司制度、福利、流程、文化等问题
- 帮助新员工快速了解公司

## 回答规则
- 使用友好、热情的语气，像一位热心的老员工
- 如果问题超出你的知识范围，请诚实告知并建议联系 HR（hr@acme.com）
- 回复控制在 200 字以内，简洁清晰
- 不要编造不存在的信息

## 公司基本信息
- 公司名称：Acme 科技有限公司
- 成立时间：2018 年
- 办公地址：北京市海淀区中关村软件园
- 工作时间：9:00-18:00，弹性半小时
```

4. 在右侧预览窗口测试：

```
用户：公司几点上班？
小A：我们公司的标准工作时间为 9:00-18:00，提供半小时的弹性打卡时间哦！
可以在 8:30-9:30 之间到岗，只要保证每天工作满 8 小时即可~

用户：能报销出租车吗？
小A：这个问题涉及到具体的报销制度，我的知识库中暂时没有相关信息。
建议您查看《员工报销手册》或直接联系 HR 邮箱：hr@acme.com，他们会给您准确答复！
```

**关键观察**：
- 小A 回答了工作时间问题（因为写在 System Prompt 里）
- 小A 诚实地说不知道报销信息（遵守了"不要编造"的规则）
- 每次对话都保持了友好热心的语气（遵守了"人设"要求）

#### 步骤2：添加上下文变量（目标：让 System Prompt 使用外部变量）

在 System Prompt 中添加用户名称变量：

```text
...（保留原有内容）

当前对话对象：新员工 {{user_name}}
```

在右侧预览窗口的**变量**面板中，填写 `user_name` = "张三"，再次测试：

```
用户：你认识我吗？
小A：当然认识！你是新加入的张三同学，欢迎来到 Acme 大家庭！
我是你的入职指南小助手小A，有什么问题随时问我~
```

**背后的机制**：Dify 使用 Jinja2 模板引擎渲染 Prompt，`{{user_name}}` 是占位符，运行时会被替换为变量面板中的实际值。这样同一个 App 可以服务不同用户。

#### 步骤3：配置对话上下文（目标：实现多轮对话记忆）

在 **对话设置** 中：

```
上下文轮数：5  （保留最近 5 轮对话）
上下文窗口 Token 上限：2000 （历史对话超过 2000 tokens 时自动截断）
```

测试多轮记忆：

```
第 1 轮：我叫张三，工号 12345
第 2 轮：我是什么部门的？ → 小A：你还没告诉我呢
第 3 轮：我在研发部    → 小A：好的记住了
第 4 轮：我叫什么？    → 小A：你是张三
第 5 轮：我工号多少？  → 小A：工号 12345
```

第 6 轮时，由于上下文轮数设置为 5，第 1 轮的历史可能已经被截断。但这不影响工号记忆，因为第 1 轮的信息在更近的对话中被"复述"过。

**配置建议**：
- 上下文轮数：客服场景建议 3-5 轮；教育辅导场景建议 8-10 轮
- Token 上限：根据模型能力选择，GPT-3.5 建议 2000-3000，GPT-4 可以设到 8000

#### 步骤4：挂载知识库（目标：让 Chat 基于真实文档回答）

1. 先在**知识库**页面创建一个知识库（第 5 章会详细讲解，此处先快速操作）：
   - 进入**知识库** → **创建知识库** → 上传一份员工手册 PDF
   - 使用默认的分段策略，等待索引完成

2. 回到 Chat App 编辑页 → **上下文** → **添加知识库** → 选择刚创建的知识库

3. 修改 System Prompt，添加知识库占位符：

```text
...（保留前面的内容）

## 知识库上下文
以下是公司知识库中与用户问题相关的信息，请优先基于这些信息回答：
{{#context#}}

如果知识库中没有相关信息，请结合你的已有知识回答，但务必告知用户信息来源的可靠程度。
```

4. 测试知识库问答：

```
用户：我们公司的年假怎么算？
小A：根据《员工手册》第三章第2节，Acme 公司的年假制度如下：
- 入职满 1 年：5 天年假
- 入职满 3 年：8 天年假
- 入职满 5 年：10 天年假
以上信息来自公司的《员工手册》，具体以 HR 最新通知为准哦！

用户：加班有加班费吗？
小A：[知识库中未找到相关信息]
我没有在知识库中找到加班费的具体规定。这个问题建议您查阅《薪酬制度》文档或咨询 HR。
```

#### 步骤5：发布与嵌入（目标：让团队可以使用你的助手）

1. 点击右上角 **发布** → 生成公开 URL
2. 在 **发布** 页面可以：
   - 复制公开链接，分享给团队
   - 配置密码保护（可选）
   - 获取 iframe 嵌入代码
   - 获取 API 调用凭据

3. 通过 API 调用你的助手：

```python
import requests
import json

url = "http://localhost/v1/chat-messages"
headers = {
    "Authorization": "Bearer app-xxxxxxxxxxxxx",  # 从发布页获取
    "Content-Type": "application/json"
}

data = {
    "inputs": {"user_name": "李四"},
    "query": "公司年假怎么算？",
    "response_mode": "streaming",
    "conversation_id": "",  # 首次对话留空，后续传入以保持上下文
    "user": "lisi"
}

# 流式读取回复
response = requests.post(url, headers=headers, json=data, stream=True)
for line in response.iter_lines():
    if line:
        line_str = line.decode('utf-8')
        if line_str.startswith('data: '):
            event_data = json.loads(line_str[6:])
            if event_data.get('event') == 'message':
                print(event_data.get('answer', ''), end='', flush=True)
```

**运行结果描述**：执行上述 Python 脚本，你将在控制台看到流式输出的回复文本。设置 `response_mode` 为 `blocking` 则可以获取一次性完整回复。

## 4. 项目总结

### 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **上手速度** | 5 分钟即可创建第一个能用的 Chat App | Prompt 调优需要反复实验，没有"最佳实践"模板 |
| **知识库集成** | 一键挂载知识库，`{{#context#}}` 变量开箱即用 | 检索质量依赖知识库构建质量，复杂的跨文档问答效果波动 |
| **多轮对话** | 滑动窗口自动管理上下文，无需手动拼接历史 | Token 消耗随对话长度线性增长，长对话成本高 |
| **发布分发** | 支持公开 URL、iframe 嵌入、API 调用三种发布方式 | WebApp 定制能力有限（仅 Logo/配色），深度定制需 Fork 前端 |
| **变量系统** | Jinja2 模板灵活注入各种变量 | 复杂逻辑（条件判断、循环）需要 Workflow，Chat App 做不了 |

### 适用场景

| 场景 | 说明 |
|------|------|
| **智能客服** | FAQ 问答、售后支持、操作指南 |
| **内部知识助手** | HR 问答、IT 帮助台、培训辅导 |
| **产品导购** | 基于产品知识库的选型推荐和配置咨询 |
| **文档问答** | 技术文档、API 手册的自然语言问答 |
| **教育辅导** | 基于课程材料的学情答疑 |

**不适用场景**：
- 需要调用外部 API 获取实时数据（如订单状态查询），请使用 Workflow
- 需要高度自定义的 UI 和交互，建议直接使用 Dify API + 自建前端

### 注意事项

1. **Prompt 长度控制**：System Prompt 越长，留给对话历史的空间越少，精简 System Prompt 是关键优化点
2. **知识库检索阈值**：Score 阈值设太高会导致"查不到"，设太低会导致"乱回复"，建议从 0.5 起步调优
3. **会话并发**：同一个 conversation_id 的并发请求可能导致上下文错乱，注意锁或排队处理
4. **个人信息脱敏**：如果 System Prompt 包含了 `{{user_name}}` 等敏感变量，确保 WebApp 层做了访问控制

### 常见踩坑经验

1. **坑：知识库挂载了但回答中不包含检索内容** → 根因：System Prompt 中没写 `{{#context#}}` 占位符，知识库内容没有被注入到 LLM 的 Prompt 中。解决：在 Prompt 适当位置添加 `{{#context#}}`
2. **坑：对话几轮之后 AI 就"失忆"了** → 根因：上下文轮数设置太小（如设为 1），或者 Token 上限触发了截断。解决：增加上下文轮数和 Token 上限，但要关注成本
3. **坑：发布后公开链接打不开** → 根因：`service_api_url` 环境变量未配置或配置为 `localhost`（只有本机能访问）。生产环境需改成实际的公网域名

### 思考题

1. **进阶题**：如果用户上传了一张图片问"这是什么"，Chat App 能回答吗？如果不能，需要怎样扩展？（提示：思考 Dify 的多模态支持和 Vision 模型）

2. **进阶题**：当知识库检索返回了 10 条结果，但其中只有 3 条真正相关，如何让 LLM 更好地利用这些信息而不是被无关内容干扰？（提示：思考 Rerank 和 Prompt 设计）
