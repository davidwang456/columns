# 第32章：请求生命周期完整源码链路

## 1. 项目背景

前 31 章你一直在"用" Dify——创建 App、写 Prompt、调 API。从本章开始进入高级篇，你要"懂"每一行关键代码。本章是高级篇的"热身"和"导航图"——用一个 `POST /v1/chat-messages` 请求串起 Dify 后端的完整 10 步请求生命周期。

为什么要理解这 10 步？三个场景：

**场景一：排查"500 Internal Server Error"**。如果你不知道请求经过了 Controller → Service → Core → ModelManager → LLM，你只能逐文件 grep "Error"，大海捞针。但如果你知道链路——先检查 Controller 层的 Pydantic 校验是否通过（通常是 400 而非 500）、再看 Service 层的数据库查询是否超时、再看 Core 层的 Prompt 组装是否有异常——排查效率提升 10 倍。

**场景二：添加新功能**。你需要在"每次 Chat 请求前检查用户是否被封禁"。这个检查应该放在哪里？Controller 的 `before_request` 钩子（最外层，拦截所有请求）还是 Service 层的 `generate()` 方法（业务编排层）？理解了每一层的职责边界，你才能把代码放在正确的位置。

**场景三：修改 Workflow 引擎行为**。你想让 Workflow 执行时推送进度百分比。你知道要改 `api/core/workflow/workflow_entry.py` 的 `run()` 方法，而不是去改 Controller 或 Service。

## 2. 项目设计——剧本式交锋对话

**小胖**：（在 VS Code 中跳来跳去）"大师！用户发消息返回 500，我在 Controller 的 `post()` 方法里加了 try-except 打日志，但 500 还是可能发生在 Service、Core、ModelManager 任何一层。有没有一张'路线图'告诉我每一步经过哪些文件？"

**大师**："当然。一个 Chat API 请求的完整 10 步链路：

```
浏览器 POST /v1/chat-messages
  ↓
[1] Nginx: docker/nginx/conf.d/dify.conf
    location /api/ → proxy_pass http://api:5001
  ↓
[2] Flask before_request: api/extensions/ext_login.py
    @login_manager.request_loader → 从 Session/Token 恢复用户
  ↓
[3] Blueprint 路由: api/controllers/web/app.py
    @web_blueprint.route('/chat-messages')
  ↓
[4] Controller 校验: ChatMessageApi.post()
    → Pydantic ChatMessageRequest 校验输入参数
  ↓
[5] Service 编排: api/services/app_generate_service.py
    AppGenerateService.generate() → 查 App + 获取/创建 Conversation
  ↓
[6] Core 组装: api/core/app/apps/chat/chat_app_runner.py
    ChatAppRunner.run() → 组装 Prompt + 知识库检索
  ↓
[7] ModelManager: api/core/model_manager.py
    get_instance() → ProviderManager 获取配置 → LoadBalancer 选 Key
  ↓
[8] LLM 调用: Provider specific (如 openai.py)
    invoke_llm_stream() → httpx.post → SSE chunk yield
  ↓
[9] 保存消息: api/models/model.py
    Message(conversation_id, query, answer, tokens).save()
  ↓
[10] SSE 响应: api/controllers/web/completion.py
    generate_sse_response() → yield "data: {...}\n\n"
```

每一步都有精确的源文件和函数名。这就是你的'排查地图'——以后任何一次 500，你都能快速缩小排查范围。"

**技术映射**：请求生命周期 = Nginx 转发 → Flask 拦截器 → Blueprint 路由 → Controller → Service → Core → ModelManager → Provider → Persistence → Response。10 步贯穿约 7-8 个源文件。

**小白**："Celery Worker 里执行 Workflow 时，`current_user` 还存在吗？"

**大师**："不存在了！这是最容易踩的一个坑。Worker 是独立进程（甚至独立容器），它没有 HTTP 请求上下文。`current_user` 是从浏览器发来的 Session Cookie 中恢复的——Worker 根本没有 Cookie。

但 Worker 需要知道'当前是哪个租户'才能正确调用 ModelManager。Dify 是怎么解决的？两层方案：
1. **FlaskTask 包装器**：`api/extensions/ext_celery.py` 中定义了自定义 Task 基类，在 `__call__()` 中用 `app.app_context()` 给每个 Celery 任务注入 Flask 应用上下文——这样 Worker 能访问 `current_app.config`、能用数据库连接池。
2. **显式传参**：Workflow 异步任务创建时，把 `tenant_id`、`app_id` 等必要信息作为任务参数传给 Worker，Worker 代码通过任务参数获取租户信息，而不是依赖 `current_user`。"

**技术映射**：Celery Worker 中的用户上下文 = FlaskTask（注入 app context）+ 显式传参（tenant_id）。永远不要依赖 `current_user`，它在 Worker 中就是 None。

## 3. 项目实战

### 完整 10 步链路（源码文件 + 关键代码）

```text
[Step 1] Nginx
  docker/nginx/conf.d/dify.conf:
    location /api/ {
        proxy_pass http://api:5001;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_buffering off;           # ★ SSE 必须关 buffer
        proxy_read_timeout 300s;       # ★ LLM 可能耗时
    }

[Step 2] Flask before_request
  api/extensions/ext_login.py:
    @login_manager.request_loader
    def load_user_from_request(request):
        token = request.headers.get('Authorization', '').replace('Bearer ', '')
        if token.startswith('app-'):
            return load_user_from_service_api_token(token)  # Service API
        return load_user_from_session(request)              # Console API

[Step 3] Blueprint 路由
  api/controllers/web/__init__.py:
    web_blueprint = Blueprint('web', __name__)

[Step 4] Controller
  api/controllers/web/app.py:
    class ChatMessageApi(Resource):
        def post(self):
            # ① Pydantic 自动校验请求体
            data = ChatMessageRequest.model_validate(request.get_json())
            # ② 委托给 Service（Controller 不写业务逻辑）
            return AppGenerateService.generate(
                app_id=data.app_id, user=current_user, query=data.query
            )

[Step 5] Service
  api/services/app_generate_service.py:
    class AppGenerateService:
        @classmethod
        def generate(cls, app_id, user, query, ...):
            app = App.query.get(app_id)                # 查数据库
            conversation = cls._get_or_create(...)      # 获取/创建会话
            runner = ChatAppRunner(app, user, conversation)
            return runner.run(query)                     # 委托给 Core

[Step 6] Core
  api/core/app/apps/chat/chat_app_runner.py:
    class ChatAppRunner:
        def run(self, query):
            messages = self._build_messages(query)      # 组装 Prompt
            # 如果挂载了知识库，先检索
            if self.app.dataset:
                context = Retriever.search(query)
                messages[0].content += f"\n\n参考资料：{context}"
            # 获取模型实例并调用
            instance = ModelManager.get_instance(...)
            return instance.invoke_llm_stream(messages)

[Step 7] ModelManager
  api/core/model_manager.py:
    @staticmethod
    def get_instance(tenant_id, provider, model_type, model):
        config = ProviderManager.get_config(tenant_id, provider)
        credentials = config.get_credentials(model_type, model)
        selected = LoadBalancer.select(tenant_id, provider, model_type, model, credentials)
        return ModelInstance(provider, model, selected)

[Step 8] LLM 调用 (OpenAI 示例)
  api/core/model_runtime/model_providers/openai/llm/llm.py:
    def invoke_llm_stream(self, messages):
        client = httpx.Client(timeout=60)
        with client.stream('POST', 'https://api.openai.com/v1/chat/completions', json={
            'model': self.model, 'messages': messages, 'stream': True
        }) as resp:
            for line in resp.iter_lines():
                if line.startswith('data: '):
                    chunk = json.loads(line[6:])
                    yield chunk['choices'][0]['delta']['content']

[Step 9] 保存消息
  api/models/model.py:
    message = Message(
        conversation_id=conv_id, query=query,
        answer=full_response, message_tokens=total_tokens
    )
    db.session.add(message)
    db.session.commit()

[Step 10] SSE 响应
  api/controllers/web/completion.py:
    def generate_sse_response(generator):
        def gen():
            for event in generator:
                yield f"data: {json.dumps(event.to_dict())}\n\n"
            yield "data: [DONE]\n\n"
        return Response(stream_with_context(gen()), mimetype='text/event-stream')
```

## 4. 项目总结

### 10 步链路速查表

| 步 | 层 | 源文件 | 关键函数/类 |
|----|-----|-------|-----------|
| 1 | Nginx | `docker/nginx/conf.d/dify.conf` | `proxy_pass` |
| 2 | Flask | `extensions/ext_login.py` | `load_user_from_request` |
| 3 | Blueprint | `controllers/web/__init__.py` | `route('/chat-messages')` |
| 4 | Controller | `controllers/web/app.py` | `ChatMessageApi.post()` |
| 5 | Service | `services/app_generate_service.py` | `AppGenerateService.generate()` |
| 6 | Core | `core/app/apps/chat/chat_app_runner.py` | `ChatAppRunner.run()` |
| 7 | ModelManager | `core/model_manager.py` | `get_instance()` |
| 8 | Provider | `core/model_runtime/.../llm.py` | `invoke_llm_stream()` |
| 9 | Persistence | `models/model.py` | `Message.save()` |
| 10 | Response | `controllers/web/completion.py` | `generate_sse_response()` |

**追踪口诀**：Nginx → Flask → Blueprint → Controller → Service → Core → ModelManager → Provider → DB → SSE。这就是 Dify 数据的"脊椎"。
