# 第36章：多租户架构与权限系统源码剖析

## 1. 项目背景

"我们 Dify SaaS 版服务了 200 家企业客户。上周一个客户截图说在控制台看到了另一家公司的 App 名称。"这是 SaaS 服务商最怕的事故。Dify 从架构设计之初就考虑了多租户，但"设计支持"≠"部署即安全"——你需要理解**三层隔离机制的实现细节**和**潜在绕过风险**。

三层隔离：**SQL 查询层**的 tenant_id 自动注入（每次数据库查询自动加上 `WHERE tenant_id = current_tenant_id`）、**Redis Key 层**的租户前缀隔离（`tenant_a:model_lb:...` vs `tenant_b:model_lb:...`）、**对象存储层**的目录前缀（`tenants/{tenant_id}/documents/...`）。任何一层有疏漏都可能导致跨租户数据泄露。

同时 Dify 实现了 RBAC 权限系统——Owner、Admin、Editor、Member 四角色。理解权限装饰器 `require_permission()` 和角色权限矩阵，能让你在不改源码的情况下审计越权操作、为 SaaS 版扩展自定义角色。

## 2. 项目设计——剧本式交锋对话

**小胖**：（看着生产事故报告）"大师！租户 A 的客服 App 回复里出现了租户 B 公司的产品信息。我查了代码——Dify 不是自动按 tenant_id 过滤吗？怎么会泄露？"

**大师**："自动过滤的前提是'你用的是 Dify 的 ORM 查询'。如果你在自定义代码中直接写了 `SELECT * FROM apps`（原生 SQL），或不带 `filter_by(tenant_id=...)` 的 ORM 查询——那就绕过了自动过滤。SQL 层隔离的可靠性 = 所有查询都经过 ORM 过滤。原生 SQL 是绕过点。"

**技术映射**：SQL 层隔离的脆弱点 = 原生 SQL 查询。只要有一条没加 tenant_id 过滤，就可能泄漏。

**小白**："Redis 呢？如果 Key 格式写错了——忘了加 tenant_id 前缀——是不是也能读到别的租户的数据？"

**大师**："对。Dify 所有 Redis Key 都用 `{tenant_id}:{resource}:{id}` 格式。但如果某段代码不遵循这个规范——比如 `SET model_cache:gpt4 "key123"` 没加租户前缀——那所有租户读到的是同一份缓存。这就是为什么代码审查时要特别关注 Redis Key 的构建。"

**小胖**："权限呢？我们有个场景——只让编辑者改 Prompt 但不能改模型配置。原生四角色不够细。"

**大师**："需要扩展权限矩阵。一种做法：在 `ROLE_PERMISSIONS` 中增加 `'prompt_editor': {'app:edit_prompt', 'app:view'}` 新角色。但这需要改后端代码。另一种做法：不改源码，通过 Workflow 的权限控制——把模型配置做成环境变量，只有 Admin 能改环境变量。"

### 权限系统

```python
ROLE_PERMISSIONS = {
    'owner':  {'app:create','app:edit','app:delete','app:view',
               'dataset:create','dataset:edit','dataset:delete','dataset:view',
               'member:manage','workspace:manage','billing:manage'},
    'admin':  {'app:create','app:edit','app:view',
               'dataset:create','dataset:edit','dataset:view','member:view'},
    'editor': {'app:edit','app:view','dataset:edit','dataset:view'},
    'member': {'app:view','dataset:view'},
}

from functools import wraps

def require_permission(permission: str):
    def decorator(f):
        @wraps(f)
        def wrapper(*args, **kwargs):
            role = current_user.get_role_in_current_tenant()
            if permission not in ROLE_PERMISSIONS.get(role, set()):
                raise ForbiddenError(f"权限不足。需要: {permission}，当前角色: {role}")
            return f(*args, **kwargs)
        return wrapper
    return decorator

# 使用
class AppApi(Resource):
    @require_permission('app:delete')
    def delete(self, app_id):
        App.query.filter_by(id=app_id).delete()
```

class Tenant(Base):
    __tablename__ = 'tenants'

## 4. 项目总结

| 隔离层 | 策略 | 绕过风险 |
|--------|------|---------|
| SQL | ORM 自动 filter_by(tenant_id=...) | 原生 SQL 会绕过 |
| Redis | `{tenant}:{type}:{id}` 前缀 | 自定义代码忘加前缀 |
| 存储 | `tenants/{tenant_id}/...` 目录 | 直接访问底层存储 |
| 权限 | `require_permission` 装饰器 | 不写装饰器的 API |
