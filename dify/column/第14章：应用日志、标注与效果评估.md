# 第14章：应用日志、标注与效果评估

## 1. 项目背景

"上线一个月了，AI 客服到底好不好用？有没有帮我们省人力？有没有说错话得罪客户？"这是每个部署了 AI 应用的团队都会问的问题。Dify 的应用日志页提供了基础的数据面板——活跃用户数、消息量、Token 消耗——但这些只是"量"的统计，不是"质"的评估。

更棘手的是，LLM 的应用质量是"主观的"——同样一个回答，对 A 用户是"太好了"，对 B 用户可能是"答非所问"。如何量化评估？Dify 提供了**标注（Annotation）**功能：你可以人工标记优质的问答对，构建"黄金数据集"。这些标注数据有两大用途：一是用于 Few-shot Prompt 优化（把标注的优质问答作为示例），二是用于定期评估模型表现。

但标注只是"质"评估的起点。真正的效果评估体系需要结合：用户反馈（点赞/点踩）、业务指标（如客服场景的"问题解决率"）、自动评估（用另一个 LLM 给回复打分）。本章将带你建立一套从"看日志"到"做标注"再到"量化评估"的完整流程，帮助你用数据驱动 AI 应用的持续优化。

## 2. 项目设计——剧本式交锋对话

**小胖**：（拿着运营给的周报）"大师，你看这个数据：上周 AI 客服处理了 500 次对话，Token 花了 2 万。但这能说明啥？是好还是坏？我完全没概念。"

**大师**："你说到点子上了。日志页给的是'运营数据'（Operational Metrics），不是'质量数据'（Quality Metrics）。运营数据告诉你'用了多少'——消息量、Token 消耗、活跃用户。质量数据告诉你'用得好不好'——回答准确率、用户满意度、问题解决率。"

**技术映射**：运营指标（消息量、Token）= 回答了"用了多少"，质量指标（标注、反馈）= 回答了"用得好不好"。

**小白**："那怎么衡量质量？用户点踩就算了，很多人懒得点。"

**大师**："多层评估体系：
1. **显式反馈**：用户主动点赞/点踩（信号明确但样本少）
2. **隐式反馈**：用户是否追问（追问通常意味着不满意）、是否重新提问、是否切换了人工客服
3. **人工标注**：从日志中抽样，人工逐条打分（最可靠但成本高）
4. **自动评估**：用另一个 LLM 给回复打分（成本低但不够精准）"

**小胖**："标注具体怎么操作？我在 Dify 里看到了'标注'页面。"

**大师**："标注页让你做三件事：
1. **标记优质问答**：从对话日志中找到回答得好的，勾选'加入标注'
2. **编辑修正**：如果 AI 的回答大体对但细节有误，你可以手动改写答案
3. **导出数据集**：把标注数据导出为 CSV/JSON，用于优化 Prompt 或训练评估模型"

**技术映射**：标注（Annotation）= 人工构造的高质量问答对，用于模型评估和 Few-shot Prompt 优化。

**小白**："标注数据怎么反馈到 App 优化上？"

**大师**："流程是这样的：

1. 从日志中筛选 100 条对话（抽样策略：随机 + 分层，覆盖不同场景）
2. 人工标注——给每条回答打分（满意/基本满意/不满意）+ 修正不满意回答
3. 把标注中"满意的问答对"整理成 Few-shot 示例，加到 System Prompt 里
4. 对比接入 Few-shot 前后，用户点踩率是否下降"

**小胖**："那如果我标注了 50 条，怎么确保新标注不会泄露给其他租户？"

**大师**："Dify 的标注数据是租户隔离的——每个 Workspace 的标注数据独立存储，其他租户看不到。数据库里 Annotation 表有 tenant_id 字段，查询时自动过滤。"

## 3. 项目实战

### 分步实现

#### 步骤1：解读应用日志页（目标：读懂关键指标）

登录 Dify 控制台 → 进入 Chat App → **日志与标注**：

**关键指标解读**：

| 指标 | 含义 | 健康基线 | 告警阈值 |
|------|------|---------|---------|
| **活跃用户数** | 当天至少发了一条消息的用户数 | — | 突降 > 50%（可能服务挂了） |
| **消息量** | 当天发送的对话消息总数 | — | 突增 > 300%（可能被刷量） |
| **Token 消耗** | 当天消耗的 Token 总量 | — | 日消耗 > 预算的 80% |
| **平均响应时间** | 从用户发送到 AI 回复的平均秒数 | < 5s | > 10s（可能模型超时） |
| **对话轮数** | 每会话的平均问答轮数 | 3-8 轮 | > 20 轮（可能 Agent 陷入循环） |

**API 读取日志数据**：

```python
import requests
from datetime import datetime, timedelta

API_KEY = "app-xxx"
BASE = "http://localhost/v1"

# 获取最近 7 天的对话列表
resp = requests.get(
    f"{BASE}/conversations",
    headers={"Authorization": f"Bearer {API_KEY}"},
    params={
        "user": "all",
        "limit": 100,
        "sort_by": "-created_at"
    }
)

conversations = resp.json()["data"]

# 分析对话质量
total = len(conversations)
long_conv = [c for c in conversations if c.get("message_count", 0) > 20]
short_conv = [c for c in conversations if c.get("message_count", 0) == 1]

print(f"总对话数: {total}")
print(f"长对话（>20轮）: {len(long_conv)} ({len(long_conv)/total*100:.1f}%)")
print(f"单轮对话: {len(short_conv)} ({len(short_conv)/total*100:.1f}%)")
print(f"长对话的 ID: {[c['id'][:8] for c in long_conv]}")
```

**关键发现**：
- 长对话比例高（> 30%）：可能 Agent 陷入循环，或知识库不够导致用户反复追问
- 单轮对话比例高（> 70%）：可能是简单问答场景，也可能是用户第一轮就不满意放弃

#### 步骤4：LLM 自动评估——用 AI 给 AI 打分（目标：降低评估成本）

```python
def auto_evaluate(user_query, ai_answer, expected_keywords=None):
    """使用另一个 LLM 对回答进行自动评分"""
    
    eval_prompt = f"""
你是一个 AI 质量评审专家。请评估以下客服回答的质量。

用户问题：{user_query}
客服回答：{ai_answer}

请按以下 4 个维度打分（每个维度 1-5 分）：
1. 相关性：回答是否直接回应了用户问题（无关 = 1，精准命中 = 5）
2. 准确性：回答中的事实是否正确、有无编造（大量编造 = 1，完全正确 = 5）
3. 完整性：是否覆盖了问题的所有方面（严重遗漏 = 1，全面完整 = 5）
4. 语气：语气是否专业、友好（生硬冷漠 = 1，温暖专业 = 5）

输出格式（仅 JSON，不要其他内容）：
{{"relevance": 5, "accuracy": 4, "completeness": 3, "tone": 5, "total": 17, "passed": true}}

如果总分 >= 15，passed 为 true，否则为 false。
"""
    
    # 调用一个廉价模型做评估
    resp = requests.post(
        "http://localhost/v1/completion-messages",
        headers={"Authorization": f"Bearer app-xxx-evaluator"},
        json={"query": eval_prompt, "user": "evaluator", "response_mode": "blocking"}
    )
    
    eval_result = resp.json().get("answer", "{}")
    try:
        return json.loads(eval_result)
    except:
        return {"error": "评估失败"}
```

## 4. 项目总结

### 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **日志面板** | 开箱即用，核心指标一目了然 | 缺少自定义时间对比、同比环比、趋势图不够丰富 |
| **标注系统** | 支持人工标注和修正，导出入参方便 | 标注功能比较基础，缺少标注任务分配、审核流程 |
| **反馈收集** | WebApp 内置点赞/点踩，数据自动记录 | 反馈数据分散，缺少聚合分析工具 |
| **自动评估** | 无需额外接入即可实现 | Dify 本身不提供自动评估功能，需要外挂 LLM |

### 适用场景

| 场景 | 推荐评估方式 |
|------|------------|
| **客服场景** | 日志面板 + 用户反馈 + 人工抽样标注 |
| **内容生成** | Few-shot 标注 + A/B 测试 + LLM 自动评估 |
| **数据分析** | 业务指标对比（准确率、召回率）+ 标注 |
| **教育辅导** | 专家标注 + 学习者满意度调查 |
| **内部工具** | 使用频率 + 追问率 + 定期人工巡检 |

### 注意事项

1. **标注数据的代表性**：不要只挑"最完美"的回答标注——也要标注中等和差回答，否则 Few-shot 会让模型过度自信
2. **反馈偏差**：主动点赞/点踩的用户通常是"极端满意"或"极端不满"的用户，中间大多数是沉默的
3. **评估成本**：人工标注成本高（约 2-5 元/条），建议先跑 LLM 自动评估筛选后再人工复核

### 常见踩坑经验

1. **坑：标注数据用于 Few-shot 后效果反而变差** → 根因：标注数据太少（< 10 条）或太单一（都是同类型问题）。解决：至少标注 30 条，覆盖 5 种以上不同意图
2. **坑：Token 消耗突然暴涨** → 根因：检查日志，可能是某个用户陷入了"追问循环"或 Agent 无限迭代。解决：在日志中发现异常 conversation_id，单独排查
3. **坑：自动评估的 LLM 打分和人工打分差异很大** → 根因：评估 Prompt 太模糊或评估模型本身不够强。解决：完善评估 Prompt（加详细评分标准），用更好的模型做评估
