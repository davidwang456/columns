# 第11章：对话管理与会话持久化

## 1. 项目背景

"我昨天跟你们的 AI 客服聊了 20 分钟，今天再点进去，它完全不记得我了！"这是用户投诉的最高频词之一。Chat App 的多轮对话记忆默认只在同一个"会话"内生效，一旦用户关闭浏览器、会话过期、或者 conversation_id 丢失，对话上下文就归零了。

这背后涉及一个核心问题：Dify 如何管理会话（Conversation）和消息（Message）的生命周期？会话什么时候创建？历史消息存哪里？什么时候截断？跨天的对话能恢复吗？在 Workflow 和 Agent 中，对话变量（conversation 级变量）和用户变量（user 级变量）又有什么区别？

这些问题不仅在 Chat App 中重要，当你需要用 API 编程调用 Dify 时更关键——你需要手动管理 conversation_id，确保多轮请求归属同一个会话；你需要理解消息的存储和检索机制，避免上下文的"丢失"和"错乱"。本章将深入 Dify 的会话模型，从数据库表结构到 API 用法，从 Chat App 的滑动窗口到 Workflow 的 conversation 变量，帮你真正"掌控对话"。

## 2. 项目设计——剧本式交锋对话

**小胖**：（满脸委屈）"大师，我写的聊天机器人有个严重 bug——用户说'我叫小胖'，过了几轮能记住。但是用户关掉浏览器，第二天再打开，AI 就像失忆了一样问'你好，请问有什么可以帮你？'这咋整？"

**大师**："你碰到的是'会话持久化'问题。Dify 的对话记忆不是存在客户端浏览器的 Cookie 里，而是存在服务端的数据库里。每次对话对应一个 conversation_id，关掉浏览器再打开，只要你能找回之前的 conversation_id，就能恢复对话。但如果你每次打开都创建一个新会话，AI 当然从头开始。"

**小白**：（在白板上画数据模型）"所以数据库里是 Conversation 表和 Message 表的关系？一个 Conversation 可以有多个 Message？"

**大师**："对，大致是这样的结构：

```
Conversation（会话）
├── id: "conv-abc-123"
├── app_id: 应用 ID
├── user: "user-xiaopang"
├── status: "normal"
├── created_at: 2026-05-11 09:00:00
│
└── Message（消息列表）
    ├── id: "msg-001"
    │   query: "我叫小胖"
    │   answer: "好的小胖，我记住了！"
    │   created_at: 09:00:05
    │
    ├── id: "msg-002"
    │   query: "我叫什么"
    │   answer: "你叫小胖呀！"
    │   created_at: 09:00:30
    │
    └── id: "msg-003"
        ...
```

**技术映射**：Conversation（1:N）Message，Hibernate 式的 ORM 关系映射。

**小胖**："那 Dify 是怎么决定往 Prompt 里塞多少历史消息的？"

**大师**："你在 Chat App 设置里配的'上下文轮数'和'Token 上限'决定了两件事：
1. **上下文轮数**：比如设为 5，意味着只保留最近的 5 轮问答（Q1+A1, Q2+A2, ..., Q5+A5）拼到 Prompt 里。
2. **Token 上限**：如果最近 5 轮的历史消息加起来超过了 Token 上限（比如你设的 3000 tokens），Dify 会从最早的消息开始截断，直到总 Token 数不超过上限。

这两个参数构成了一个'滑动窗口'——窗口随着新消息到来向前滑动，老消息被丢弃。"

**技术映射**：滑动窗口 = 最近 N 轮 + Token 截断，在上下文长度和成本之间取得平衡。

**小白**："对话变量是什么？和用户变量有什么区别？"

**大师**：
- **对话变量（Conversation Variable）**：作用域是整个会话。比如 '用户情绪' = '焦急'，这个值在整个会话中保持不变，Agent 每一步都可以读取。
- **用户变量（User Variable）**：作用域是一个用户，跨会话保持。比如 'VIP 等级' = '金卡'，这个用户不管开启多少个会话，这个值都生效。

在 Dify 控制台，你可以在 App 的'变量'面板中分别配置。源码里，它们是通过不同的前缀作用的：`conversation.情绪` vs `user.等级`。"

**小胖**："那如果我想让用户下次打开浏览器时自动恢复上一次的会话怎么办？"

**大师**："你需要在自己的应用中实现：
1. 后端存储 `user_id → conversation_id` 的映射
2. 用户打开聊天窗口时，查询他是否有未完结的会话
3. 如果有，拿到 conversation_id，加载历史消息列表展示
4. 用户发新消息时，带上这个 conversation_id"

## 3. 项目实战

### 分步实现

#### 步骤1：通过 API 管理会话生命周期（目标：掌握 conversation_id 的创建/查询/恢复）

**创建新会话并发送消息**：

```python
import requests

API_BASE = "http://localhost/v1"
API_KEY = "app-xxxxxxxxxxxx"

headers = {
    "Authorization": f"Bearer {API_KEY}",
    "Content-Type": "application/json"
}

# 1. 首次对话（不传 conversation_id）
resp = requests.post(f"{API_BASE}/chat-messages", headers=headers, json={
    "query": "你好，我叫张三，工号 10001",
    "user": "zhangsan",
    "response_mode": "blocking"
})

data = resp.json()
conversation_id = data["conversation_id"]
message_id = data["message_id"]
answer = data["answer"]
print(f"会话ID: {conversation_id}")
print(f"消息ID: {message_id}")
print(f"回复: {answer}")
```

**在同一会话中发第二条消息**：

```python
# 2. 继续对话（传入 conversation_id）
resp2 = requests.post(f"{API_BASE}/chat-messages", headers=headers, json={
    "query": "我刚才说我叫什么名字？",
    "user": "zhangsan",
    "response_mode": "blocking",
    "conversation_id": conversation_id  # 关键：带上会话 ID
})

print(f"回复: {resp2.json()['answer']}")
# 预期：张三，工号 10001（AI 记得上一轮的上下文）
```

**不同用户创建不同会话**：


#### 步骤3：理解滑动窗口的 Token 计算（目标：优化对话成本）

在 Chat App 设置中，实际测试不同配置对 Token 消耗的影响：

```bash
# 测试脚本：观察不同轮数的 Token 消耗
import requests

API_KEY = "app-xxx"
headers = {"Authorization": f"Bearer {API_KEY}"}

# 发送 10 轮对话，观察每轮的 Token 消耗
conv_id = ""
for i in range(10):
    resp = requests.post("http://localhost/v1/chat-messages", 
        headers=headers,
        json={
            "query": f"这是第{i+1}次对话，请记住数字{i+1}",
            "user": "test",
            "response_mode": "blocking",
            "conversation_id": conv_id
        }
    )
    data = resp.json()
    conv_id = data["conversation_id"]
    tokens = data.get("metadata", {}).get("usage", {}).get("total_tokens", 0)
    print(f"第{i+1}轮 | Token: {tokens} | conversation_id: {conv_id[:8]}...")
```

**观察结果**：随着轮数增加，Token 消耗应该先增长（因为历史消息变多），然后趋于平稳（到达配置的上限后开始截断）。

#### 步骤4：实现"会话恢复"功能（目标：用户下次回来能继续对话）

```python
import requests
import json

class DifyChatSession:
    def __init__(self, api_key, api_base="http://localhost/v1"):
        self.api_key = api_key
        self.api_base = api_base
        self.headers = {"Authorization": f"Bearer {api_key}", "Content-Type": "application/json"}
    
    def get_or_create_conversation(self, user_id):
        """为用户获取最新会话，如果没有则创建"""
        # 查询最近的会话
        resp = requests.get(
            f"{self.api_base}/conversations",
            headers=self.headers,
            params={"user": user_id, "limit": 1, "sort_by": "-created_at"}
        )
        convs = resp.json().get("data", [])
        
        if convs:
            return convs[0]["id"]
        return None  # 返回 None，调用方首次发消息时会自动创建
    
    def send_message(self, user_id, query, conversation_id=None):
        """发送消息，返回回复"""
        payload = {
            "query": query,
            "user": user_id,
            "response_mode": "blocking"
        }
        if conversation_id:
            payload["conversation_id"] = conversation_id
        
        resp = requests.post(f"{self.api_base}/chat-messages", 
            headers=self.headers, json=payload)
        return resp.json()
    
    def get_history(self, user_id, conversation_id, limit=20):
        """获取会话历史消息"""
        resp = requests.get(
            f"{self.api_base}/messages",
            headers=self.headers,
            params={"conversation_id": conversation_id, "user": user_id, "limit": limit}
        )
        return resp.json().get("data", [])

# 使用示例
session = DifyChatSession("app-xxx")
user = "zhangsan"

# 恢复上次会话
conv_id = session.get_or_create_conversation(user)
if conv_id:
    history = session.get_history(user, conv_id, limit=10)
    print(f"恢复了 {len(history)} 条历史消息")
    
    # 继续对话
    result = session.send_message(user, "刚才我们聊到哪了？", conv_id)
    print(result.get("answer", ""))
else:
    # 首次对话
    result = session.send_message(user, "你好！")
    conv_id = result["conversation_id"]
    print(f"新建会话: {conv_id}")
```

## 4. 项目总结

### 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **会话模型** | Conversation-Message 一对多关系清晰，支持多用户隔离 | 不支持会话分支（从某条消息分叉出两个子对话） |
| **滑动窗口** | 自动管理上下文长度，无需手动处理 Token | 截断策略简单（丢弃最早），可能丢掉重要上下文 |
| **对话变量** | 支持会话级和用户级两级作用域 | 对话变量的读写 API 不够完善（部分操作需 Workflow 节点） |
| **API 设计** | conversation_id 直观易用，RESTful 风格 | 前端 WebApp 的会话恢复体验较弱（URL 参数方式不够优雅） |

### 适用场景

| 场景 | 会话管理策略 |
|------|------------|
| **在线客服** | 会话级变量追踪用户情绪和问题状态，问题解决后标记 closed |
| **AI 辅导** | 长对话上下文 10+ 轮，定期总结关键信息压缩存储 |
| **购物助手** | 对话变量存储购物车、偏好，跨会话保持 |
| **医疗问诊** | 严格的会话隔离（一个患者一个会话），加密存储 |
| **内部工具** | 简单的会话管理，定时清理过期对话 |

### 注意事项

1. **conversation_id 安全**：不要在 URL 中明文传递 conversation_id，可能导致会话劫持
2. **会话清理**：定期清理过期会话和消息表，避免数据库膨胀（社区版无自动清理配置）
3. **上下文污染**：用户在对话中说了大量无关信息后，AI 的回复质量可能下降——建议提供"清除上下文"或"开启新会话"按钮

### 常见踩坑经验

1. **坑：API 调用时不传 conversation_id，每次都新建会话** → 根因：前端或后端代码没有保存和传递 conversation_id。解决：将 conversation_id 存到 sessionStorage/数据库/Redis 中
2. **坑：切换模型后对话记忆丢失** → 根因：Dify 的对话记忆存的是 query 和 answer 文本，换模型后历史消息仍在，但不同模型对同一历史的"理解"可能不同，导致回复风裂变
3. **坑：workflow 节点的 conversation 变量读不到** → 根因：Workflow 独立运行时没有对话上下文，只有 Chat App 中内嵌 Workflow 时才有 conversation 变量。详见第 22 章
