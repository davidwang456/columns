# 第34章：变量系统源码——多级作用域与类型推导

## 1. 项目背景

`{{#llm_node.text#}}`——这个 Workflow 中最常用的变量引用语法，背后是一套完整的**多级作用域机制**和**8 种类型推导系统**。很多开发者用变量时靠"试"——写了 `{{#text#}}` 不行，改成 `{{#llm.text#}}` 也不行，最后改成 `{{#sys.query#}}` 才对了。但下次遇到另一个变量又得从头试。

这种"试错法"低效的原因在于不理解变量的三个核心属性：

**①作用域（Scope）**——这个变量从哪来？`sys.query` 是引擎注入的系统变量（Workflow 启动时自动创建），`env.API_BASE` 是部署时的环境变量（docker-compose.yaml 里配的），`conversation.user_emotion` 是跨消息的会话变量（Agent 在某轮对话中写入的），`node_id.text` 是某节点执行后的输出。引用时不写前缀或写错前缀，就会报"变量未找到"。

**②注册时机（Registration Timing）**——变量不是一开始就全在池子里的。节点输出变量在该节点**执行成功**后才注册到变量池。如果你在节点 B 的 Prompt 中引用了节点 A 的 `text`，但节点 A 还没执行完（或执行失败抛异常了），变量池里就没有这个变量。这是"变量未找到"最常见的时序问题。

**③类型（Type）**——下游节点引用变量时，Dify 会校验类型匹配。LLM 节点的 Prompt 期望 string，如果你传了 number 或 array，Jinja2 渲染会失败。

理解这套系统的实际价值：排查"变量未找到"时从作用域、时机、类型三个维度定位；开发自定义节点时正确定义输入输出 Schema 让下游能引用。

## 2. 项目设计——剧本式交锋对话

**小胖**："大师！我写的 `{{#llm_node.text#}}` 提示变量未找到，但我打开日志明明 LLM 节点执行成功了。ID 我对着画布抄了三遍，确认没写错。这是什么玄学 bug？"

**大师**："三步排查法。**①检查 ID**——你抄的是画布上显示的标题'LLM1'，还是节点实际的 ID？标题是可编辑的（你随时可以改），但节点 ID 是一串自动生成的数字（如 `1734567890`）。变量引用必须用 ID。打开 DSL 导出的 JSON，找到这个 LLM 节点的 `id` 字段。

**②检查执行状态**——你说'日志显示执行成功了'——但你确认日志里写的是 `NodeFinished` 而不是 `NodeFailed` 吗？如果节点执行时抛了异常，`add_node_output` 不会被调用——变量池里没有注册。

**③检查作用域**——`sys.query` 和 `query` 是完全不同的变量。如果不写前缀，解析器会从变量池**顶层**开始找——先找系统变量区（`sys`），再找环境变量区（`env`），再找节点输出区（以 node_id 为 key）。你确认这个变量在正确的层级吗？"

**技术映射**：变量未找到 = 三要素排查——ID 是否正确、节点是否真的成功执行、作用域是否匹配。

**小白**："作用域分层的设计哲学是什么？为什么不能把所有变量放一个平面字典里？"

**大师**："三个原因：

**①命名冲突隔离**——假设你的 Workflow 里有个节点叫 `query`，系统变量也有个 `sys.query`。如果都放平面字典里，`query` 会被覆盖。作用域前缀把它们隔离到不同命名空间。

**②读写权限控制**——`sys.*` 和 `env.*` 是**只读的**——你不能在 Workflow 中修改 `sys.query` 或 `env.API_BASE`。`conversation.*` 是**可读写的**——Agent 可以在运行中更新用户情绪等会话状态。节点输出是**自动注册的**（节点不能直接写其他节点的输出，只能通过 VariableAssigner）。

**③生命周期管理**——`sys.*` 的生命周期是一次 Workflow 执行；`conversation.*` 的生命周期是整个会话（跨多轮对话持久化）。生命周期不同，存储位置也不同（`sys` 在内存，`conversation` 可能持久化到 Redis/DB）。"

**技术映射**：作用域 = 命名空间（防冲突）+ ACL（读写权限）+ TTL（生命周期）。三层需求决定了不能是平面设计。

**小胖**："VariableAssigner 节点写变量的时候，是写到 `conversation.*` 还是写到节点输出层？"

**大师**："默认写到全局层——不带前缀。如果你运行一个 VariableAssigner 节点，设置 `user_emotion = '焦虑'`，下游节点引用 `{{#user_emotion#}}` 就能读到。如果你想写到 `conversation.*` 作用域，需要用完整路径 `{{#conversation.user_emotion#}}`。两种写法都在变量池中生效，区别是作用域前缀不同。"

## 3. 项目实战

### 变量池核心源码

```python
# api/core/workflow/variable_pool_initializer.py（核心逻辑，带详尽注释）
import os
from typing import Any

class VariablePool:
    """
    Workflow 变量池——三层作用域 + 节点输出层
    
    数据模型：
    {
        'sys': {           # ★ 系统作用域（引擎注入，节点只读）
            'query': '用户输入的问题',
            'workflow_id': 'wf-xxx',
            'conversation_id': 'conv-yyy',
        },
        'env': {           # ★ 环境作用域（部署配置，节点只读）
            'API_BASE_URL': 'https://api.example.com',
        },
        'conversation': {  # ★ 会话作用域（Agent/Workflow 可读写）
            'user_emotion': '焦虑',
        },
        # ★ 节点输出（无前缀，以 node_id 为 key）
        'node_1734567890': {'text': 'AI 回复的内容', 'usage': {...}},
        'node_1734567891': {'body': {...}, 'status_code': 200},
    }
    """
    
    SCOPE_PREFIXES = ['sys', 'env', 'conversation']
    
    def __init__(self, system_variables=None, user_inputs=None):
        self._pool = {
            'sys': system_variables or {},
            'env': dict(os.environ),      # 环境变量只读快照
            'conversation': {},            # 会话变量（可读写）
        }
        if user_inputs:
            self._pool.update(user_inputs)  # 用户输入也放顶层
    
    def resolve(self, reference: str) -> Any:
        """
        ★ 核心方法：解析 {{#scope.path[...].path#}} 引用
        
        例：{{#sys.query#}}              → _pool['sys']['query']
        例：{{#conversation.user_info.name#}} → _pool['conversation']['user_info']['name']
        例：{{#node_123.text#}}          → _pool['node_123']['text']
        例：{{#node_456.items[0].name#}} → _pool['node_456']['items'][0]['name']
        """
        clean = reference.strip('{{#').strip('#}}').strip()
        parts = clean.split('.')
        current = self._pool
        
        # Step 1: 检查作用域前缀
        if parts[0] in self.SCOPE_PREFIXES:
            scope = parts[0]
            current = current[scope]
            parts = parts[1:]
        
        # Step 2: 逐级取值
        for i, part in enumerate(parts):
            if isinstance(current, dict):
                if part not in current:
                    raise VariableNotFoundError(
                        f"变量 '{reference}' 未找到。\n"
                        f"当前路径: {'.'.join(parts[:i])}\n"
                        f"当前层级可用字段: {list(current.keys())[:10]}"
                    )
                current = current[part]
            elif isinstance(current, list):
                # ★ 支持数组索引
                try:
                    current = current[int(part)]
                except (ValueError, IndexError):
                    raise VariableNotFoundError(f"数组索引无效: {part}")
            else:
                raise VariableNotFoundError(
                    f"无法从 {type(current).__name__} 类型获取字段 '{part}'"
                )
        
        return current
    
    def add_node_output(self, node_id: str, output: dict):
        """
        ★ 节点执行成功后调用——注册输出到变量池顶层
        注意：这会覆盖同 ID 节点的旧输出（如果 Workflow 重新执行同一节点）
        """
        self._pool[node_id] = output
```

### 8 种变量类型 + 校验器

```python
from enum import Enum

class VariableType(str, Enum):
    STRING = "string"
    NUMBER = "number"
    BOOLEAN = "boolean"
    OBJECT = "object"
    ARRAY_STRING = "array[string]"
    ARRAY_NUMBER = "array[number]"
    ARRAY_OBJECT = "array[object]"
    FILE = "file"  # 特殊类型：文件引用 {url, name, size, mime_type}

TYPE_VALIDATORS = {
    VariableType.STRING:        lambda v: isinstance(v, str),
    VariableType.NUMBER:        lambda v: isinstance(v, (int, float)) and not isinstance(v, bool),
    VariableType.BOOLEAN:       lambda v: isinstance(v, bool),
    VariableType.OBJECT:        lambda v: isinstance(v, dict),
    VariableType.ARRAY_STRING:  lambda v: isinstance(v, list) and all(isinstance(i, str) for i in v),
    VariableType.ARRAY_NUMBER:  lambda v: isinstance(v, list) and all(isinstance(i, (int, float)) for i in v),
    VariableType.ARRAY_OBJECT:  lambda v: isinstance(v, list) and all(isinstance(i, dict) for i in v),
    VariableType.FILE:          lambda v: isinstance(v, dict) and 'url' in v,
}

def validate_variable(value, expected_type: VariableType) -> bool:
    validator = TYPE_VALIDATORS.get(expected_type)
    return validator(value) if validator else True
```

### 调试变量池快照

```python
import json

def debug_pool(pool: VariablePool) -> str:
    """导出变量池完整快照（脱敏版，用于调试）"""
    snapshot = {}
    for scope_name, scope_vars in pool._pool.items():
        if isinstance(scope_vars, dict):
            snapshot[scope_name] = {
                k: (type(v).__name__, str(v)[:100])
                for k, v in scope_vars.items()
            }
    return json.dumps(snapshot, indent=2, ensure_ascii=False, default=str)
```

## 4. 项目总结

| 变量类型 | 前缀 | 作用域 | 读写 | 生命周期 | 注册时机 |
|---------|------|--------|------|---------|---------|
| 系统变量 | `sys.` | 全局 | 只读 | Workflow 执行 | 引擎启动时注入 |
| 环境变量 | `env.` | 全局 | 只读 | 部署配置 | 应用启动时快照 |
| 会话变量 | `conversation.` | 会话级 | 读写 | 会话 | VariableAssigner 写入 |
| 节点输出 | `node_id.` | 图内 | 只读 | 图执行 | 节点成功后注册 |

### 常见"变量未找到"排查口诀

1. **ID 对不对？** → 对着 DSL JSON 里的 `id` 字段，不是画布上的标题
2. **节点执行成功没？** → 日志里确认是 `NodeFinished` 不是 `NodeFailed`
3. **前缀写对没？** → 是 `sys.query` 还是 `query`？是不写前缀默认顶层查找
4. **类型匹配吗？** → LLM 节点期望 string，传了 array 会渲染失败
