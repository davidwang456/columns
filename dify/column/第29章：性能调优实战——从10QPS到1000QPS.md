# 第29章：性能调优实战——从 10 QPS 到 1000 QPS

## 1. 项目背景

"618 大促第一分钟，AI 客服就崩了——用户消息发出去 25 秒才收到回复，Nginx 日志里 502 堆积如山。运营总监在工作群里 @我 八次。"小陈事后复盘，发现问题根源不是"机器不够"（4 核 16GB 的服务器 CPU 才用了 15%），而是"不知道瓶颈在哪"。

Dify 是典型的 **I/O 密集型应用**——大部分时间不是在算东西，而是在等：等 LLM API 返回（网络 I/O，2-30 秒）、等数据库查询（磁盘 I/O，5-500ms）、等 Redis（内存 I/O，0.3ms）。CPU 只有 15% 说明 CPU 不是瓶颈——问题一定在某个 I/O 层。

小陈的排查路径是经典的"试错法"：先加了 2 台机器——QPS 没变；然后调大了 Gunicorn Worker——还是没变；最后偶然发现 Message 表 200 万行没有索引，每次查询全表扫描耗时 5 秒——加上复合索引后 QPS 从 22 飙到 180。如果他一上来就用火焰图定位，这个优化只需要 30 分钟，而不是 3 天。

本章不是"给出指令你照做"，而是一套**数据驱动的性能优化方法论**：**测量（火焰图 + 慢日志 + 压测）→ 定位（找到真正的瓶颈层）→ 优化（对症下药，一层一层改）→ 验证（重新压测对比）。**

## 2. 项目设计——剧本式交锋对话

**小胖**：（把 wrk 压测报告往桌上一拍）"大师！wrk 显示 QPS 才 22，P99 延迟 18 秒！用户等 18 秒才收到'你好'两个字，不投诉才怪。我已经试了加机器、加 Worker、加 Redis 内存——全都没用。这 Dify 是不是性能不行？"

**大师**："别急着下结论。你做的三件事——加机器（水平扩展）、加 Worker（增加并发处理能力）、加 Redis 内存（缓存层扩量）——全是'猜测式优化'。真正的问题是：你知道瓶颈在哪吗？不知道瓶颈就动手，就像不知道哪里漏水就到处砸墙。"

**技术映射**：性能优化第一步永远是测量（Profiling），不是调参数。找到瓶颈→对症下药。

**小白**："火焰图我看过——py-spy 生成的一个 SVG，全是一层层横条。怎么从中判断瓶颈？"

**大师**："火焰图的铁律：**最宽的横条 = CPU 消耗最多的函数**。三种最常见的瓶颈模式：
- **`psycopg2` 相关函数占了 40% 宽度** → 数据库是瓶颈（连接池太小、查询没命中索引、或表太大需要分区）
- **`httpx` / `requests` 相关函数占了 60% 宽度** → LLM API 调用是瓶颈（网络延迟、Provider 限流、或模型排队）
- **`json.loads` / `json.encoder` 占了 30% 宽度** → 应用层在大量解析 JSON（可能是 LLM 返回了超大 JSON 响应或 Workflow 输出太大）"

**技术映射**：火焰图解读 = 看最宽的函数在哪个模块——数据库连接层？网络 I/O 层？序列化层？

**小胖**："那慢 SQL 呢？我 200 万条 Message，翻一页要 5 秒。加了索引不管用。"

**大师**："不是所有索引都管用。`EXPLAIN ANALYZE` 是你最好的朋友。跑一下实际查询，看 PostgreSQL 选择的是 Seq Scan（全表扫描——说明索引没覆盖查询条件）还是 Index Scan（用了索引但仍然慢——可能是统计信息过期，PostgreSQL 误判了）。你的 Messages 表大概率需要 `(conversation_id, created_at DESC)` 复合索引——两个字段放一起，一个 B-Tree 查找就能定位到所有对话的消息并按时间排序。"

**小白**："那 LLM 限流怎么办？GPT-4 的 RPM（每分钟请求限制）就 500，大促期间轻松突破。加 Key 能解决吗？"

**大师**："加 Key 能治标，但治本需要**背压机制（Backpressure）**。当检测到 429 的频率上升时，不要继续硬发请求——而是在 API 层做排队（queue），让请求等 5-10 秒再发。用户看到的是'正在排队，预计等待 8 秒'——比直接报错好得多。这个需要自定义中间件实现。"

## 3. 项目实战

### 环境准备

| 工具 | 安装 | 用途 |
|------|------|------|
| wrk | `brew install wrk` | HTTP 压测 |
| py-spy | `pip install py-spy` | CPU 火焰图 |
| EXPLAIN ANALYZE | PostgreSQL 内置 | SQL 查询计划 |
| redis-cli | Docker 内置 | Redis 延迟 |

### 分步实现

#### 步骤1：压测基线——找到当前性能上限

```bash
# === 纯 Web 层压测（无 DB、无 LLM） ===
wrk -t4 -c100 -d30s http://localhost/health
# 预期 QPS: ~850（纯 Python + Gevent 的 Web 层上限）

# === 全链路压测（Chat API） ===
# post.lua:
# wrk.method = "POST"
# wrk.body = '{"query":"hi","user":"test-{{$request}}","response_mode":"blocking"}'
# wrk.headers["Content-Type"] = "application/json"
# wrk.headers["Authorization"] = "Bearer app-xxx"

wrk -t4 -c20 -d60s -s post.lua http://localhost/v1/chat-messages
# 典型结果（调优前）:
#   QPS: 22
#   P50: 8s, P99: 18s, P999: 25s
#   Non-2xx: 15% (502 + 504)
```

#### 步骤2：火焰图定位 CPU 热点

```bash
API_PID=$(docker top docker-api-1 | grep gunicorn | head -1 | awk '{print $2}')
py-spy record -o flamegraph_before.svg --pid $API_PID --duration 30

# 在浏览器打开 flamegraph_before.svg
# 按 Ctrl+F 搜索关键词：
# - "psycopg2" → 数据库瓶颈
# - "httpx" / "requests" / "send" → LLM API 瓶颈
# - "json.loads" / "json.dumps" → 序列化瓶颈
# - "gevent" / "hub" / "switch" → 正常 I/O 等待
```

#### 步骤3：数据库优化

```sql
-- Step 3.1：定位慢查询
SELECT pid, now() - query_start AS duration, left(query, 150)
FROM pg_stat_activity
WHERE state = 'active' AND now() - query_start > interval '1 second';

-- Step 3.2：分析查询计划
EXPLAIN (ANALYZE, BUFFERS, TIMING)
SELECT m.* FROM messages m
JOIN conversations c ON m.conversation_id = c.id
WHERE c.app_id = 'xxx-app-id'
ORDER BY m.created_at DESC LIMIT 20;

-- 无索引: Seq Scan on messages → cost=0..45234 → actual time=5234ms
-- 加索引后: Index Scan using idx_messages_conv_created → time=12ms

-- Step 3.3：添加复合索引
CREATE INDEX CONCURRENTLY idx_messages_conv_created 
    ON messages (conversation_id, created_at DESC);
CREATE INDEX CONCURRENTLY idx_conversations_app 
    ON conversations (app_id, created_at DESC);

-- Step 3.4：连接池调优
-- .env 中:
SQLALCHEMY_POOL_SIZE=30
SQLALCHEMY_POOL_RECYCLE=1800
SQLALCHEMY_POOL_PRE_PING=true
```

#### 步骤4：Web 层 + Nginx 调优

```python
# gunicorn.conf.py
workers = 4                      # 4 核 × 1 Gevent Worker
worker_class = 'gevent'          # 协程模型（I/O 密集）
worker_connections = 1000        # ★ 每 Worker 最大并发连接
keepalive = 5                    # Keep-Alive 减少 TCP 握手
timeout = 300                    # LLM 最坏情况 5 分钟
max_requests = 50000             # 防 Python 内存泄漏
max_requests_jitter = 5000       # 随机偏移防同时重启
graceful_timeout = 60
preload_app = True               # 预加载减少内存
```

```nginx
# docker/nginx/conf.d/dify.conf 调优关键
upstream dify_api {
    server api:5001;
    keepalive 32;                # 到后端的连接池
}

server {
    keepalive_timeout 65;
    keepalive_requests 1000;
    
    gzip on;
    gzip_types application/json text/plain text/css application/javascript;
    gzip_min_length 1000;
    
    location /api/ {
        proxy_pass http://dify_api;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_buffering off;      # ★ SSE 必须关 buffer
        proxy_read_timeout 300s;
        proxy_connect_timeout 10s;
    }
}
```

#### 步骤5：验证 + 对比

```bash
# 重新压测 /health
wrk -t4 -c100 -d30s http://localhost/health
# 调优前: 850 QPS  →  调优后: 3200 QPS (3.8×)

# 重新压测 Chat API
wrk -t4 -c20 -d60s -s post.lua http://localhost/v1/chat-messages
# 调优前: 22 QPS, P99=18s, Err=15%
# 调优后: 180 QPS, P99=3s, Err=2%

# 生成调优后火焰图对比
py-spy record -o flamegraph_after.svg --pid $API_PID --duration 30
```

## 4. 项目总结

### 优化效果矩阵

| 优化层 | 手段 | 预期提升 | 风险 |
|--------|------|---------|------|
| **数据库** | 复合索引 | 5-100× | 写入略慢 + 磁盘占用 |
| **数据库** | 连接池调优 (30→50) | 2-3× | 需配合 PostgreSQL max_connections |
| **Web 层** | Workers ×4 + Gevent | 3-5× | 内存占用 × Workers 数 |
| **Nginx** | Gzip + KeepAlive + Buffering Off | 1.5-2× | 几乎无风险 |
| **应用层** | Redis 多级缓存 | 2-3× | 缓存一致性风险 |

### 适用场景

| QPS 级别 | 推荐方案 |
|---------|---------|
| < 100 | 默认配置 + 关键索引 |
| 100-500 | Workers ×4 + 复合索引 + Nginx Gzip + 连接池 |
| 500-2000 | K8s HPA + Redis 缓存 + 读写分离 |
| > 2000 | 架构级重构（表分区 + 熔断器 + 多 AZ） |

### 常见踩坑经验

1. **坑：加了索引但 EXPLAIN 还是 Seq Scan** → 根因：`ANALYZE` 没跑，统计信息过期。解决：`ANALYZE messages;` 强制更新统计
2. **坑：加了 Worker 但 QPS 没提升** → 根因：瓶颈在数据库或外部 API，不在 Web 层
3. **坑：`proxy_buffering off` 后 Nginx 内存持续增长** → 根因：SSE 长连接过多。解决：限制单 IP 并发连接数

### 思考题

1. **进阶题**：火焰图中 `json.loads` 占用 30% CPU——这说明了 LLM 返回的 JSON 太大。如何在不改代码的情况下优化？（提示：限制 LLM 的 max_tokens 输出；或用 orjson 替代标准库）

2. **进阶题**：单机优化到 500 QPS 后，下一步做到 5000 QPS 必须水平扩展。水平扩展时最大的瓶颈是什么？（提示：所有 Pod 共享同一个 PostgreSQL——读写分离）
