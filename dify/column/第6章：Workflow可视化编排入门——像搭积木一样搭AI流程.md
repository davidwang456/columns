# 第6章：Workflow 可视化编排入门——像搭积木一样搭 AI 流程

## 1. 项目背景

小陈是公司的内容运营。每天早上的流程是：打开 5 个行业资讯网站，浏览标题，找出和自己公司业务相关的新闻，用 ChatGPT 写 200 字的摘要，再发到公司飞书群里。小陈算了算，这套动作每天要花 40 分钟。他想："能不能让 AI 自动完成？RSS 抓新闻 → 筛选相关的 → 生成摘要 → 推送到飞书？"

如果是直接写代码，这需要：写一个 RSS 解析脚本 → 调用 LLM 做筛选 → 再调 LLM 写摘要 → 调飞书 API 发送。步骤不复杂，但涉及网络请求、数据格式转换、条件判断、错误处理……小陈只是个运营，不是程序员。

这就是 Dify Workflow 的用武之地。如果用 Chat App 去实现——你只能说一句话，等 AI 回答，再说下一句。但 Workflow 让你像画流程图一样，把"抓新闻→筛选→摘要→推送"的每一步都拖拽到画布上，用连线串联起来。设定好"每天早上 9 点自动执行"，小陈再也不用手动刷新闻了。

Workflow 是 Dify 区别于其他 LLM 工具平台的核心竞争力。它不是简单的"对话链"（LangChain），而是一个**有向无环图（DAG）执行引擎**——节点之间通过变量池传递数据，支持条件分支、循环、并行，甚至能在沙箱中运行你写的 Python 代码。本章从最基础的 Start→LLM→End 三节点流水线开始，带你进入 Workflow 的世界。

## 2. 项目设计——剧本式交锋对话

**小胖**：（盯着 Dify 的 Workflow 画布，满脸困惑）"大师！我打开了这个 Workflow 页面，好大一片空白画布，左边一大堆节点——开始了、LLM、结束、IF/ELSE、迭代、代码……这跟 Chat App 比，完全不是一个东西啊！"

**大师**："你可以把 Chat App 想象成外卖点餐——你告诉骑手你要啥，骑手（LLM）给你送到了。Workflow 更像是你自己设计一条自动化流水线——先洗菜（开始节点）→ 切菜（LLM 节点生成大纲）→ 炒菜（第二个 LLM 扩写）→ 装盘（结束节点输出），每一步的顺序和逻辑都由你决定。"

**小白**：（已经拖了几个节点到画布上）"我看到节点之间有连线，这是传递数据的吗？如果我第一个 LLM 节点输出了一段文本，第二个节点怎么拿到它？"

**大师**："好问题。Dify 的 Workflow 有一个**全局变量池**。每个节点跑完后，它的输出会自动存入变量池，命名规则是 `{{节点ID.输出字段名}}`。比如你的第一个 LLM 节点 id 是 `llm_outline`，它的输出字段叫 `text`，那第二个节点里写 `{{llm_outline.text}}` 就能拿到那段文本。"

**技术映射**：变量池（VariablePool）= Workflow 的"共享内存"，节点通过引用语法 `{{node_id.field}}` 读取上游数据。

**小胖**："那系统变量是什么？我看到有个 `sys.query`，这又是啥？"

**大师**："系统变量是 Workflow 引擎自动注入的，不需要你手动创建。常用的有三个：
- `sys.query`：用户在调用 Workflow 时输入的文本（类似 Chat App 里的 `{{#query#}}`）
- `sys.conversation_id`：如果 Workflow 被嵌在对话中使用，这就是对话的 ID
- `sys.workflow_id`：当前 Workflow 的 ID

它们以 `sys.` 开头，和你手动创建的变量区分开。"

**技术映射**：系统变量 = 引擎注入的元信息，用户变量 = 节点输出或手动定义的变量。

**小白**："节点是什么时候执行的？是按我连线的顺序吗？"

**大师**："不完全是。Workflow 按**拓扑顺序**执行——不是看几何位置，而是看依赖关系。如果节点 A 的输出是 B 的输入，那 A 一定先于 B 执行。但如果 C 和 D 互相不依赖，它们可以**并行执行**。这个底层逻辑我们到中级篇第 22 章会深入源码，现在你只需要记住：连线的方向决定执行顺序。"

**小胖**：（画了一个三节点的流程图）"所以最简单的 Workflow 就是：开始 → LLM → 结束。那调试模式怎么用？我配了 Prompt 之后直接跑就行吗？"

**大师**："调试模式有两个级别：单节点运行和全图运行。单节点运行让你测试每个节点的输入输出是否符合预期，而不依赖上游节点。全图运行则从头到尾执行一遍。调试日志会告诉你每个节点消耗了多少 Token、执行了多长时间、输出了什么。这就像盖楼，你可以先单独检查每块砖的质量，再测试整面墙的稳固性。"

## 3. 项目实战

### 环境准备

| 条件 | 说明 |
|------|------|
| Dify 已部署 | 第 2 章完成 |
| LLM Provider 已配置 | 第 3 章完成 |
| 基本理解变量概念 | 第 4 章 Chat App 中的 `{{variable}}` |

### 分步实现

#### 步骤1：创建 Workflow 并搭建三节点流水线（目标：理解画布基本操作）

1. Dify 首页 → **创建应用** → 选择 **Workflow** → 输入名称"新闻摘要助手"
2. 进入画布编辑器，默认已有**开始节点**

3. 从左侧节点面板拖入 **LLM 节点**，点击该节点配置：

```yaml
模型：GPT-3.5-turbo（或你配置的任意 LLM）
上下文：System
提示词内容：
  你是一个专业的新闻编辑。请根据用户提供的新闻标题和链接，
  判断这条新闻是否与"人工智能、科技创新"相关。
  
  如果相关，请回复"相关｜[新闻标题]"。
  如果不相关，请回复"不相关"。
  
  用户输入：{{#sys.query#}}
  
  请严格按照上述格式回复，不要添加额外内容。
```

4. 从开始节点拖一条连线到 LLM 节点（表示数据流从开始 → LLM）

5. 拖入**结束节点**，从 LLM 节点连线到结束节点。在结束节点的输出变量中，选择 `{{llm_node.text}}`

6. 点击右上角**运行** → 输入测试内容 `"OpenAI 发布 GPT-5，性能提升 10 倍"` → 点击运行

**预期输出**：
```
相关｜OpenAI 发布 GPT-5，性能提升 10 倍
```

再测试一条不相关的：输入 `"欧洲杯足球赛开幕"` → 预期输出：
```
不相关
```

#### 步骤2：添加第二个 LLM 节点实现摘要生成（目标：理解节点间的数据传递）

在第一个 LLM 节点之后，创建分支逻辑前，先测试多节点数据传递：

1. 再拖入一个 **LLM 节点**（LLM2）
2. 连线：开始 → LLM1（筛选）→ LLM2（生成摘要）→ 结束
3. LLM2 配置：

```yaml
模型：GPT-3.5-turbo
上下文：System
提示词内容：
  请为以下新闻撰写一段 100 字以内的摘要，要求简洁、客观、无主观评价。
  
  新闻标题：{{#llm1.text#}}
  
  摘要：
```

**关键细节**：`{{#llm1.text#}}` 引用了上游 LLM1 节点的 `text` 输出。你需要确认 LLM1 的节点 ID（在节点标题下方可见，如 `1723456789`），然后使用 `{{#1723456789.text#}}` 格式。

4. 运行测试，观察 LLM2 是否能拿到 LLM1 的输出

**遇到问题**：如果 LLM2 输出"请提供新闻标题"，说明变量引用失败，检查节点 ID 是否写对。

#### 步骤3：理解变量作用域（目标：掌握输入/输出变量的配置）

在 Workflow 编辑器中，每个节点底部都有**输入变量**和**输出变量**面板：

- **开始节点**：
  - 输出变量（可添加自定义输入字段）：`query`（文本）、可添加 `category`（文本）、`max_length`（数字）
  
- **LLM 节点**：
  - 输出变量（固定）：`text`（LLM 回复的完整文本）、`usage`（Token 用量信息）

点击开始节点，添加一个自定义输入变量：

```
变量名：category
变量类型：文本
默认值：人工智能
```

然后修改 LLM1 的提示词：

```text
请判断新闻是否与"{{#sys.query#}}"相关，
重点关注领域：{{#start.category#}}

如果相关，回复"相关｜[新闻标题]"。
...
```

**运行说明**：在运行面板中，你将看到"category"这个输入框，可以动态填写。

#### 步骤4：导出 Flow 配置（目标：理解 Workflow 背后的 JSON DSL）

点击画布左上角的 **...** 菜单 → **导出 DSL**，你将看到一段 JSON：

```json
{
  "graph": {
    "nodes": [
      {
        "id": "1723456789",
        "type": "start",
        "data": {
          "title": "开始",
          "variables": [
            {"variable": "query", "type": "text", "required": true}
          ]
        }
      },
      {
        "id": "1723456790",
        "type": "llm",
        "data": {
          "title": "LLM1",
          "model": {"provider": "openai", "name": "gpt-3.5-turbo"},
          "prompt_template": [
            {"role": "system", "text": "你是一个专业的新闻编辑..."}
          ]
        }
      },
      {
        "id": "1723456791",
        "type": "end",
        "data": {
          "title": "结束",
          "outputs": [{"variable": "text", "value_selector": ["1723456790", "text"]}]
        }
      }
    ],
    "edges": [
      {"source": "1723456789", "target": "1723456790"},
      {"source": "1723456790", "target": "1723456791"}
    ]
  }
}
```

**关键发现**：Workflow 的可视化画布本质上是在编辑一份 JSON 配置文件，`nodes` 定义了节点列表，`edges` 定义了数据流向（谁连接到谁）。这份 JSON 就是"Workflow DSL"——Dify 的图执行引擎的输入。

## 4. 项目总结

### 优点与缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **可视化** | 拖拽式编排，产品经理也能上手 | 复杂工作流（20+ 节点）时画布会显得杂乱，缺少分组/折叠功能 |
| **变量系统** | 直观的 `{{node.field}}` 引用语法 | 节点 ID 是数字，可读性差，需在节点标题中手动备注 |
| **调试能力** | 支持单节点运行和全图运行，Token 消耗可见 | 缺少断点调试和变量实时监视（需到中级篇自行添加） |
| **执行模型** | 拓扑排序自动处理依赖，支持隐式并行 | 并行策略不可控，无法手动指定并行度 |
| **导出导入** | DSL 是标准 JSON，可版本管理、团队协作 | JSON 手动编辑容易出错，缺少 Schema 校验 |

### 适用场景

| 场景 | Workflow 设计 |
|------|-------------|
| **内容审核流水线** | 开始 → LLM（敏感词检测）→ IF/ELSE（是否违规）→ [违规] LLM（生成整改建议） |
| **数据分析报告生成** | 开始 → HTTP（拉取数据）→ LLM（分析趋势）→ LLM（生成报告）→ 结束 |
| **多语言翻译** | 开始 → LLM（翻译为英文）→ LLM（翻译为日文）→ LLM（翻译为法文）→ 结束 |
| **邮件自动分类回复** | 开始 → LLM（意图识别）→ IF/ELSE（按意图分支）→ [投诉] LLM（安抚回复） / [咨询] LLM（标准回复） |
| **新闻聚合简报** | 开始 → HTTP（RSS 抓取）→ 迭代（逐条处理）→ LLM（筛选+摘要）→ 结束 |

**不适用场景**：
- 简单的单轮问答（直接用 Chat App，Workflow 是大材小用）
- 需要复杂的人机交互（Workflow 是批量执行，不适合需要频繁暂停的流程——用 Agent 更好）

### 注意事项

1. **变量名大小写敏感**：`{{llm.text}}` ≠ `{{LLM.text}}`，节点 ID 必须精确匹配
2. **节点输出字段固定**：LLM 节点输出 `text`，HTTP 节点输出 `body`、`status_code`、`headers`，不同节点输出字段不同
3. **循环引用禁止**：连线不能形成环路（A→B→C→A），否则拓扑排序会失败
4. **Token 实时查看**：调用一次 Workflow 的 Token 消耗 = 所有 LLM 节点消耗之和，调试时注意全局运行的成本

### 常见踩坑经验

1. **坑：连线后节点不执行** → 根因：开始节点没有连线到该节点，Workflow 从开始节点出发沿连线追溯可执行节点，孤立节点会被跳过
2. **坑：变量引用报错"变量不存在"** → 根因：引用的节点 ID 与实际不匹配。解决：在变量选择器中点击节点导航选择，而不是手打 ID
3. **坑：Workflow 运行超时** → 根因：单个 LLM 节点超时或 HTTP 节点未配置超时时间。解决：在节点设置的"超时时间"中配置合理值（LLM 建议 60s，HTTP 建议 30s）

### 思考题

1. **进阶题**：Workflow 中的并行节点是如何被 Dify 调度的？如果两个并行节点都调用了同一个 LLM Provider 的同一个 Key，会不会触发限流？（提示：思考 ModelManager 的负载均衡机制）

2. **进阶题**：如果你需要让 Workflow 中的某个节点在执行到一半时暂停，等待外部人工审批后再继续，Dify 目前有节点支持这个功能吗？（提示：查找"Human Input"节点）
